# 目录结构与模块职责

在深入 KrKr2 的源代码之前，理解项目的物理布局和逻辑划分至关重要。KrKr2 是一个高度模块化的跨平台游戏引擎，其目录结构不仅反映了代码的组织方式，还暗示了各个组件之间的层次关系。

## 1. 完整目录结构树

以下是 KrKr2 项目的顶层目录结构及其核心模块的分布情况：

```text
krkr2/
├── cpp/                # 核心 C++ 源代码
│   ├── core/           # 引擎核心（由 9 个子模块组成的 INTERFACE 库）
│   │   ├── base/       # 归档格式（XP3/ZIP/TAR/7z）、流处理、事件系统、KAG 解析器
│   │   ├── common/     # 共享头文件（如 Defer.h 等基础工具）
│   │   ├── environ/    # 平台抽象层：Cocos2d 桥接、各平台（Win/Linux/Android/SDL）适配、UI 表单
│   │   ├── extension/  # 引擎扩展 API 接口
│   │   ├── movie/      # 视频播放，内置完整的 ffmpeg 媒体播放栈
│   │   ├── plugin/     # 插件加载系统（ncbind 绑定机制）
│   │   ├── sound/      # 音频系统：含 Vorbis/FFmpeg 解码器、DSP 效果、波形管理
│   │   ├── tjs2/       # TJS2 脚本引擎：词法/语法分析（含 bison 文件）、虚拟机、字节码执行
│   │   ├── utils/      # 通用工具：定时器、多线程、剪贴板、输入设备处理、随机数
│   │   └── visual/     # 渲染引擎：图层管理、位图处理、多种图片格式解析（TLG/WEBP等）、OpenGL 渲染
│   ├── external/       # 嵌入的第三方库，如 libbpg、minizip 等
│   └── plugins/        # 独立的功能插件（编译为静态库并链接至主程序）
│       ├── psdfile/    # PSD 格式解析支持
│       ├── psbfile/    # PSB (E-mote) 资源支持
│       ├── motionplayer/# E-mote 动画播放组件
│       └── ...         # 其他如 csvParser, xp3filter, saveStruct 等散落插件
├── platforms/          # 平台特定的入口代码
│   ├── android/        # Android Gradle 项目、JNI 桥接、SDL2 与 Cocos2dx 集成
│   ├── windows/        # Windows WinMain 入口与资源文件 (.rc)
│   ├── linux/          # Linux main.cpp 入口
│   └── apple/          # macOS 与 iOS 适配代码
├── tests/              # 基于 Catch2 v3 的单元测试集
├── tools/              # 开发辅助工具，如 xp3 归档打包/提取工具
├── ui/                 # UI 资源文件（Cocos Studio 导出的 .csb 等）
├── cmake/              # CMake 构建脚本与辅助宏
├── scripts/            # 一键编译脚本（.sh / .bat）
├── dockers/            # 用于构建环境一致性的 Docker 配置文件
├── doc/                # 开发者文档、插件说明、FAQ
└── vcpkg/              # 自定义 vcpkg port 与编译环境配置
```

## 2. 顶层目录职责说明

- **cpp/**: 这是引擎的灵魂。所有的核心逻辑、脚本引擎实现、渲染逻辑和音频处理都位于此。
- **platforms/**: 虽然核心逻辑是跨平台的，但每个操作系统启动应用程序的方式都不同。该目录包含了引导引擎进入各个平台环境的“胶水代码”。
- **tests/**: 保证代码质量的关键。通过 Catch2 框架，开发者可以验证 TJS2 语法解析、归档读取等核心逻辑的正确性。
- **tools/**: 提供了命令行工具，用于处理游戏资源（如 XP3 包），方便内容创作者和开发者。
- **vcpkg/**: KrKr2 依赖许多第三方库（如 FFmpeg, libpng 等）。这里存放了自定义的编译配置，确保在不同平台上能拉取并编译正确的库版本。

## 3. Core 子模块职责

`cpp/core` 目录下的 9 个核心模块共同构成了 KrKr2 的引擎主体。以下是这些子模块的具体职能：

- **tjs2**: TJS2 是吉里吉里 2 的专属脚本语言。该模块实现了从脚本文件的读取、词法与语法分析，到生成 TJS2 字节码并在虚拟机（VM）中执行的全过程。由于是引擎的大脑，它是最复杂的模块。
- **base**: 提供了强大的文件系统抽象，支持将 XP3、ZIP、TAR、7z 等格式透明地挂载为虚拟文件系统。同时，它还包含了 KAG 脚本解析器和基础的流处理接口。
- **visual**: 渲染核心。负责管理图层、处理复杂的图像混合（Alpha 混合、过渡效果）、位图存储和多格式图像解码（如 TLG, PNG, WebP）。它最终将这些元素交给 OpenGL 进行硬件加速渲染。
- **environ**: 这是一个中间层，将底层的操作系统差异（如 Win32 窗口句柄、Android Activity）抽象为统一的 Cocos2d 接口。同时处理了跨平台的 UI 表单展示和配置管理（ConfigManager）。
- **sound**: 音频驱动。集成了 FFmpeg 和 Vorbis 等解码器。它不仅负责播放音轨，还支持 DSP 音效处理和实时的波形数据管理。
- **movie**: 视频播放引擎。内置了基于 FFmpeg 的完整播放栈，确保在移动端和桌面端都能平滑地播放各种格式的过场动画。
- **utils**: 就像引擎的瑞士军刀。提供了跨平台的多线程实现、剪贴板访问、手柄输入适配、定时器以及随机数生成等基础服务。
- **plugin**: 实现了插件的热插拔功能。利用 ncbind 机制，使得开发者可以用 C++ 编写插件并直接暴露类和方法给 TJS2 脚本。
- **extension**: 定义了扩展 API，允许在不修改核心逻辑的情况下增强引擎的功能。

## 4. 模块间的依赖关系

KrKr2 的架构遵循“自下而上”的层级结构：

1. **底层工具 (utils/common)**: 几乎被所有模块引用。
2. **抽象基础 (base)**: 为上层提供文件和数据流。
3. **功能组件 (tjs2/visual/sound/movie)**: 依赖 base 提供的资源，是引擎的核心功能实现。
4. **插件与扩展 (plugin/extension)**: 基于核心功能，为外部提供交互接口。
5. **平台桥接 (environ)**: 将以上所有逻辑通过 Cocos2d 与各个操作系统连接起来。

这种松耦合的设计使得 KrKr2 能够轻松地在不同渲染引擎（虽然目前主推 Cocos2d）和平台之间进行移植。
