# 常见问题排查

由于 KR2 是一个大型的跨平台 C++ 项目，且引入了庞大的第三方依赖库，因此在第一次编译和运行过程中，你可能会遇到各种各样的问题。本节列出了最常见的问题及其解决方案。

## vcpkg 相关问题

vcpkg 是 KR2 赖以生存的依赖管理器，绝大多数编译失败都与它有关。

### 1. VCPKG_ROOT 未设置或路径错误
如果在 CMake 配置阶段报错提示找不到 vcpkg，请检查环境变量 `VCPKG_ROOT`。
- **Windows 用户**：请务必使用反斜杠 `\`（如 `C:\vcpkg`），或者在 CMakeLists.txt 中统一使用正斜杠 `/`。路径中严禁包含中文字符或空格。
- **验证方式**：在终端输入 `vcpkg --version`，确保命令能被正确识别。

### 2. 依赖下载失败
由于某些依赖库（如 boost, ffmpeg）的源码存储在国外服务器，国内开发者常会遇到下载超时。
- **解决方案**：为终端配置 HTTP 代理。
```cmd
set HTTP_PROXY=http://127.0.0.1:7890
set HTTPS_PROXY=http://127.0.0.1:7890
```
- **手动下载**：如果代理失效，可根据错误日志中的 URL 手动下载 `.tar.gz` 文件，并将其放置在 `VCPKG_ROOT/downloads` 目录下，再重新编译。

## 工具链相关问题

### 1. bison/winflexbison 找不到
TJS2 脚本引擎的语法解析器依赖于 `bison`。
- **错误表现**：CMake 提示 `BISON_EXECUTABLE-NOTFOUND`。
- **解决方案**：确保你已经按照 [Ch3](../03-环境搭建/README.md) 的说明安装了 `winflexbison`（Windows）或 `bison`（Linux/macOS），并将其所在目录添加到了系统的 `PATH` 环境变量中。
- **版本要求**：必须是 **3.8.2** 或更高版本。

### 2. CMake 版本过低
KR2 深度使用了 CMake Presets 特性。
- **最低要求**：CMake 3.28+
- **推荐版本**：CMake 3.31.1+
- **检查方式**：执行 `cmake --version`。如果版本过低，请前往 [cmake.org](https://cmake.org) 下载最新版本。

## Android 特定问题

Android 编译是最容易出错的部分，特别是涉及 `glib` 的安装。

### 1. glib 安装失败（路径拼接 Bug）
这是 `vcpkg` 及其配套的 `meson` 构建工具在处理 Android 交叉编译时的已知 Bug，表现为在安装 `glib` 时报错。

**解决方案：修改 meson 源码**
1. 找到 `VCPKG_ROOT/downloads/tools/meson-1.6.1-哈希值/mesonbuild/minstall.py` 文件。
2. 搜索 `def install_data(self):` 函数。
3. 检查代码逻辑中是否存在路径中包含 `"/./"` 的情况。你需要将其替换为正确的路径拼接逻辑。

**修改后的核心逻辑参考：**
```python
def install_data(self, d):
    # ... 原有代码 ...
    out_file = os.path.join(self.destdir, d.install_path, os.path.basename(d.fname))
    # 修复逻辑：确保路径中没有 "/./"
    out_file = out_file.replace('/./', '/')
    # ... 原有代码 ...
```

### 2. NDK 版本不匹配
KR2 的 CMake 配置对 NDK 版本有严格限制。
- **强制要求**：NDK 版本必须是 `28.0.13004108`。
- **检查方式**：查看 `ANDROID_NDK_HOME` 目录下的 `source.properties` 文件。
- **解决方案**：在 Android Studio 的 SDK Manager 中精准下载此版本，不要使用默认的最新版本。

## 运行时排查

### 1. 游戏文件路径错误
- **表现**：启动后卡在黑屏，或者弹窗报错“找不到文件”。
- **原因**：KR2 启动后会扫描所选目录下的 `data.xp3` 或 `config.ksc`。如果目录结构不符合 KiriKiri 引擎的标准，则无法加载。
- **解决方案**：请核对 [02-运行游戏](02-运行游戏.md) 中提到的典型文件结构。

### 2. Linux 缺少 libfmod.so
- **表现**：终端提示 `error while loading shared libraries: libfmod.so.13: cannot open shared object file`。
- **解决方案**：再次运行 `./scripts/build-linux.sh`。或者手动将 `cpp/3rdparty/fmod/lib/linux/x86_64/libfmod.so` 软链接到 `/usr/lib/`。

## 排查方法论

如果你遇到了本手册未涵盖的问题，请按照以下步骤自行排查：

1. **阅读 CMake 错误日志**：查看 `out/build/Windows Debug Config/CMakeFiles/CMakeError.log`。
2. **定位 vcpkg 构建日志**：`VCPKG_ROOT/buildtrees/<库名>/install-x64-windows-dbg-out.log`。这里通常会记录具体的 C++ 编译报错详情。
3. **启用详细输出**：在 CMake 命令中加入 `--verbose` 参数，或者在构建命令中加入 `-v`。

<details>
<summary>查看答案</summary>

1. 应该前往 `vcpkg/buildtrees/<失败的库名>/` 目录下查找以 `.log` 结尾的日志文件。
