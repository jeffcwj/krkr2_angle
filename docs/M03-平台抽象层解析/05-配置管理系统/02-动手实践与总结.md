## 常见错误及解决方案

### 错误 1：修改 XML 配置文件后重启应用配置未生效

**现象**：手动编辑 `GlobalPreference.xml` 修改了 `renderer` 的值为 `opengl`，重启应用后发现仍然使用 `software` 渲染器。

**原因**：`GlobalConfigManager` 在初始化时从 XML 文件读取配置到内存 map（映射表，即键值对的内存存储结构）中。但 `IndividualConfigManager`（游戏级别的配置管理器）的读取链中，Individual 配置会**覆盖** Global 配置。如果 `Kirikiroid2Preference.xml` 中也定义了 `renderer=software`，即使修改了 Global 文件也不会生效——Individual 的值优先级更高。

**解决**：先确认你要修改的配置项在哪一级生效，检查是否被更高优先级的配置覆盖：

```cpp
// 调试配置优先级：分别查看各层级的值
auto* g = GlobalConfigManager::GetInstance();
auto* i = IndividualConfigManager::GetInstance();

// 直接读 Global 层（跳过 Individual 覆盖）
std::string globalVal = g->GetValue<std::string>("renderer", "default");
SPDLOG_INFO("Global renderer = {}", globalVal);

// 读 Individual 层（带覆盖链）
std::string indVal = i->GetValue<std::string>("renderer", "default");
SPDLOG_INFO("Individual renderer = {}", indVal);

// 如果两者不同，说明 Individual 覆盖了 Global
```

> 对照 `IndividualConfigManager.cpp` 第 58-84 行，可以看到覆盖优先级的实现逻辑。

### 错误 2：配置持久化时 XML 文件为空或内容丢失

**现象**：调用 `SaveToFile()` 后打开 XML 文件发现内容为空，或者只保存了部分配置项。

**原因**：KrKr2 的配置持久化由**脏标记**（dirty flag，标记内存中的配置是否被修改过的布尔值）驱动。只有通过 `SetValue()` 修改过的配置项才会标记为"脏"，而 `SaveToFile()` 只会写入脏数据。如果你直接修改了内存中的 map 而没有经过 `SetValue()` 接口，脏标记不会被设置，`SaveToFile()` 就认为没有数据需要写入。

**解决**：始终通过 `SetValue()` 接口修改配置，不要直接操作底层数据结构：

```cpp
// 正确：通过 SetValue 修改，自动标记脏
auto* g = GlobalConfigManager::GetInstance();
g->SetValue("bgm_volume", std::string("50"));  // 脏标记被设置
g->SaveToFile();  // 写入 XML

// 错误：直接修改底层 map（假设你拿到了内部引用）
// 脏标记未设置，SaveToFile() 不会写入任何内容
```

> 对照 `GlobalConfigManager.cpp` 第 87-121 行的写回流程，可以看到脏标记检查逻辑。

### 错误 3：LocaleConfigManager 加载语言文件失败但无报错

**现象**：设置了 `user_language=ja`（日语），但界面文本仍然显示英文，没有任何错误日志。

**原因**：`LocaleConfigManager`（区域化配置管理器）的语言文件回退机制会静默降级。当指定的语言文件 `locale/ja.xml` 不存在时，它会按回退链（fallback chain）依次尝试：`ja.xml` → `en.xml` → 硬编码默认值。整个过程不抛异常，只会在 debug 级别日志中记录回退信息。

**解决**：确保语言文件存在且路径正确，并开启 debug 级别日志来排查回退行为：

```cpp
// 检查语言文件是否存在
// 文件应放在 ui/cocos-studio/locale/ 目录下
// 命名规则：{语言代码}.xml，如 zh_cn.xml、ja.xml、en.xml

// 开启 debug 日志查看回退过程
spdlog::set_level(spdlog::level::debug);
// 重新初始化 LocaleConfigManager，观察日志输出：
// [debug] Trying locale file: locale/ja.xml — not found
// [debug] Falling back to: locale/en.xml — loaded
```

> 对照 `LocaleConfigManager.cpp` 第 10-20 行的语言文件回退逻辑和第 36-61 行的初始化与加载流程。

## 本节小结

1. KrKr2 的三级配置系统解决了“全局默认、游戏覆盖、文本本地化”三类需求。
2. `iSysConfigManager` 提供统一的 XML 读写、字符串存储与模板化访问入口。
3. `GlobalConfigManager` 管理全局默认值；`IndividualConfigManager` 管理每个游戏的覆盖值。
4. `LocaleConfigManager` 管理区域化文本，语言选择优先用户设置。
5. 配置读取链和回退链清晰，便于定位问题。
6. 持久化由脏标记驱动，减少无效写盘。
7. 平台差异通过 PAL 接口封装，不污染配置层逻辑。

### 题目 1
已知 Global 中 `renderer=software`，Individual 中 `renderer=opengl`，调用 `IndividualConfigManager::GetValue<std::string>("renderer", "software")`，结果是什么？

<details>
<summary>查看答案</summary>

返回 `opengl`。因为 Individual 特化会先拿 Global 结果作为默认链，再查 Individual；命中时 Individual 覆盖 Global。

</details>

### 题目 2
为什么当前实现里 `value="true"` 可能被读成 false？应如何修复？

<details>
<summary>查看答案</summary>

`GetValue<bool>` 调用了 `GetValue<int>`，而 `GetValue<int>` 用 `atoi`，`atoi("true")` 结果是 0。修复方式：要么约定布尔只写 `0/1`，要么升级 bool 特化以支持 `true/false/yes/no/on/off`。

</details>

### 题目 3
`CreatePreferenceAt` 与 `UsePreferenceAt` 的区别是什么？

<details>
<summary>查看答案</summary>

`CreatePreferenceAt` 只设置路径上下文，不读取文件；`UsePreferenceAt` 会先检查文件存在，存在时设置路径并调用 `Initialize` 读取。

</details>
