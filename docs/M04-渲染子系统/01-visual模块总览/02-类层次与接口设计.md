## 1. Intf/Impl 设计模式

### 1.1 什么是 Intf/Impl 模式

KrKr2 渲染子系统采用 **接口-实现分离**（Interface-Implementation Separation）模式，这是一种经典的面向对象设计技巧：

```
┌─────────────────────────────────────────────────────────────────┐
│                       Intf/Impl 模式架构                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌──────────────────┐      ┌──────────────────┐                │
│  │   LayerIntf.h    │      │   LayerImpl.h    │                │
│  │  (接口 + 基类)    │ ───▶ │  (平台实现)       │                │
│  │  tTJSNI_BaseLayer│      │  tTJSNI_Layer    │                │
│  └──────────────────┘      └──────────────────┘                │
│           │                         │                           │
│           │ include                 │ platform-specific         │
│           ▼                         ▼                           │
│  ┌──────────────────┐      ┌──────────────────┐                │
│  │ LayerBitmapIntf.h│      │LayerBitmapImpl.h │                │
│  │ iTVPBaseBitmap   │ ───▶ │tTVPNativeBaseBitmap               │
│  └──────────────────┘      └──────────────────┘                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**核心思想**：

- **Intf 文件**（`*Intf.h`）：定义纯虚接口和与平台无关的基类实现
- **Impl 文件**（`*Impl.h`）：包含平台相关的具体实现

### 1.2 为什么使用 Intf/Impl 模式

| 优势 | 说明 |
|------|------|
| **平台隔离** | 所有平台共享同一套接口，具体实现由 `impl/` 目录下的平台代码提供 |
| **编译依赖最小化** | 修改平台实现时，不需要重新编译使用接口的上层代码 |
| **测试友好** | 可以为接口编写 Mock 实现，方便单元测试 |
| **扩展性强** | 新增平台支持只需实现 Impl，不改动 Intf |

### 1.3 代码示例：LayerIntf.h 中的接口定义

```cpp
// 文件: cpp/core/visual/LayerIntf.h（第 165-186 行）

class tTJSNI_BaseLayer : public tTJSNativeInstance,
                         public tTVPDrawable,
                         public tTVPCompactEventCallbackIntf {
    // 多重继承：
    // - tTJSNativeInstance: TJS2 脚本引擎原生对象基类
    // - tTVPDrawable: 可绘制对象接口
    // - tTVPCompactEventCallbackIntf: 内存紧凑事件回调

    friend class tTVPLayerManager;  // 允许 LayerManager 访问私有成员

protected:
    iTJSDispatch2 *Owner;           // 拥有此图层的 TJS 对象
    tTJSVariantClosure ActionOwner; // 事件处理器闭包

public:
    tTJSNI_BaseLayer();
    ~tTJSNI_BaseLayer() override;
    
    // TJS 对象生命周期
    tjs_error Construct(tjs_int numparams, tTJSVariant **param,
                        iTJSDispatch2 *tjs_obj) override;
    void Invalidate() override;
    
    // 所有权访问
    iTJSDispatch2 *GetOwnerNoAddRef() const { return Owner; }
};
```

**关键点解析**：

1. `tTJSNI_BaseLayer` 是所有图层的**基类**，定义在 `LayerIntf.h` 中
2. 它继承自三个接口，分别处理脚本绑定、渲染和内存管理
3. 友元声明 `friend class tTVPLayerManager` 允许管理器直接操作私有状态

## 3. tTVPDrawable 接口详解

### 3.1 接口定义

`tTVPDrawable` 是渲染管线中最核心的接口之一，定义了"可绘制对象"的标准协议：

```cpp
// 文件: cpp/core/visual/drawable.h（第 79-96 行）

class tTVPDrawable {
public:
    // 获取绘制目标位图
    // rect: 请求的绘制区域
    // cliprect: 返回实际可绘制的裁剪区域
    virtual tTVPBaseTexture *GetDrawTargetBitmap(const tTVPRect &rect,
                                                 tTVPRect &cliprect) = 0;

    // 获取目标图层类型（决定如何混合）
    virtual tTVPLayerType GetTargetLayerType() = 0;

    // 绘制完成回调
    // destrect: 目标矩形
    // bmp: 包含绘制结果的位图
    // cliprect: 实际绘制的区域
    // type: 图层类型
    // opacity: 不透明度 (0-255)
    virtual void DrawCompleted(const tTVPRect &destrect, tTVPBaseTexture *bmp,
                               const tTVPRect &cliprect, tTVPLayerType type,
                               tjs_int opacity) = 0;
};
```

### 3.2 接口方法用途

| 方法 | 职责 | 调用时机 |
|------|------|----------|
| `GetDrawTargetBitmap` | 为渲染器提供一个可写入的位图缓冲区 | 渲染开始前 |
| `GetTargetLayerType` | 告诉渲染器应使用何种混合模式 | 计算像素混合时 |
| `DrawCompleted` | 通知对象绘制已完成，可进行后处理 | 渲染完成后 |

### 5.1 图层创建到渲染的方法调用链

```cpp
// 示例：从 TJS 脚本创建图层到最终渲染

// 1. TJS 脚本调用
// var layer = new Layer(window, parent);

// 2. C++ 层构造
tTJSNI_BaseLayer::Construct(numparams, param, tjs_obj) {
    // 绑定到 TJS 对象
    Owner = tjs_obj;
    
    // 注册内存紧凑事件
    RegisterCompactEventHook();
    
    // 连接到父图层
    if (parent) Join(parent);
    
    // 分配默认图像
    AllocateDefaultImage();
}

// 3. 加入图层树
void tTJSNI_BaseLayer::Join(tTJSNI_BaseLayer *parent) {
    Part();  // 先从旧父节点脱离
    Parent = parent;
    parent->AddChild(this);
    Manager = parent->Manager;  // 共享同一个 LayerManager
}

// 4. 更新请求传播
void tTJSNI_BaseLayer::Update(const tTVPRect &rect, bool tempupdate) {
    // 将脏区域加入更新列表
    Manager->AddUpdateRegion(rect);
}

// 5. 图层管理器执行绘制
void tTVPLayerManager::UpdateToDrawDevice() {
    // 获取绘制目标
    tTVPBaseTexture *buffer = GetOrCreateDrawBuffer();
    
    // 遍历所有可见图层
    for (auto *layer : GetAllNodes()) {
        if (layer->IsSeen()) {
            // 获取图层类型
            tTVPLayerType type = layer->GetType();
            
            // 执行绘制
            layer->Draw(this, updateRect);
        }
    }
    
    // 通知设备完成
    DrawCompleted(destRect, buffer, clipRect, type, opacity);
}

// 6. 单个图层绘制
void tTJSNI_BaseLayer::Draw(tTVPDrawable *target, const tTVPRect &r) {
    // 获取绘制目标位图
    tTVPRect cliprect;
    tTVPBaseTexture *bmp = target->GetDrawTargetBitmap(r, cliprect);
    
    // 复制自身图像到目标
    DrawSelf(target, parentRect, clipRect);
    
    // 递归绘制子图层
    for (auto *child : Children) {
        child->Draw(target, childRect);
    }
    
    // 通知绘制完成
    target->DrawCompleted(destrect, bmp, cliprect, DisplayType, Opacity);
}
```

## 本节小结

- **Intf/Impl 模式**：接口定义在 `*Intf.h`，平台实现在 `impl/` 目录下的 `*Impl.h`
- **核心类层次**：
  - 图层：`tTJSNI_BaseLayer` → `tTJSNI_Layer`（平台实现）
  - 位图：`tTVPNativeBaseBitmap` → `iTVPBaseBitmap` → `tTVPBaseBitmap`/`tTVPBaseTexture`
  - 管理器：`iTVPLayerManager`（接口） → `tTVPLayerManager`（实现）
  - 渲染器：`iTVPRenderManager` → `RenderManager_ogl`/`RenderManager_software`
- **tTVPDrawable 接口**：定义可绘制对象的标准协议，核心三个方法：
  - `GetDrawTargetBitmap()` — 获取绘制目标
  - `GetTargetLayerType()` — 获取混合类型
  - `DrawCompleted()` — 绘制完成回调
- **关键枚举**：
  - `tTVPLayerType` — 图层混合类型（29 种）
  - `tTVPDrawFace` — 绘制面类型（主图/蒙版/省份图）
  - `tTVPBlendOperationMode` — 混合操作模式
  - `tTVPBBBltMethod` — 位块传输方法
  - `tTVPBBStretchType` — 缩放插值类型

### 题目 1：Intf/Impl 模式辨析

以下哪个说法**正确**描述了 KrKr2 visual 模块中的 Intf/Impl 模式？

A. Intf 文件只包含纯虚接口，不包含任何实现代码  
B. Impl 文件中的类直接继承自 Intf 文件中的接口类  
C. 每个平台都有独立的 Intf 文件  
D. Intf 文件通过 `#include` 包含对应的 Impl 文件

<details>
<summary>查看答案</summary>

**正确答案：D**

解析：
- A 错误：Intf 文件不仅包含接口，还包含平台无关的基类实现
- B 部分正确但不完整：Impl 通常是 Intf 类的特化或扩展
- C 错误：Intf 文件是共享的，只有 Impl 文件是平台特定的
- D 正确：例如 `LayerIntf.h` 第 1112 行 `#include "LayerImpl.h"`
