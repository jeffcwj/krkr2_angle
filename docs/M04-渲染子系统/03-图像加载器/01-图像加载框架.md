## 1. 回调驱动架构总览

KrKr2 的图像加载系统采用了**回调驱动架构**（Callback-Driven Architecture），而非常见的"加载函数返回一个位图对象"模式。这种设计的核心思想是：加载器（Loader）不直接创建位图，而是通过回调函数将解码后的像素数据逐行写入调用方提供的缓冲区。

为什么采用回调驱动？有三个核心原因：

1. **内存效率**：调用方可以按需分配缓冲区，避免解码器内部额外分配大块内存。对于 4K 分辨率（3840×2160）的 32 位图像，这意味着避免分配 31MB 的临时缓冲区。
2. **流式处理**：某些格式（如隔行扫描的 PNG）可以逐行解码并立即显示，实现渐进式加载。
3. **格式无关性**：所有格式的加载器共享同一个回调接口，调用方无需关心底层格式差异。

```
┌─────────────────────────────────────────────────────────┐
│                   TVPLoadGraphic()                       │
│                   (加载入口函数)                          │
└───────────────────────┬─────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│              TVPLoadGraphicRouter()                      │
│          读取前16字节 → 魔数匹配 → 分发                   │
└───────────────────────┬─────────────────────────────────┘
                        │
           ┌────────────┼────────────────┐
           ▼            ▼                ▼
     TVPLoadPNG    TVPLoadTLG      TVPLoadJPEG ...
           │            │                │
           └────────────┼────────────────┘
                        │
                        ▼
          ┌──────────────────────────┐
          │   sizecallback(w, h, fmt) │ ← 通知尺寸和像素格式
          └──────────┬───────────────┘
                     │
                     ▼
          ┌──────────────────────────┐
          │   scanlinecallback(y)     │ ← 逐行获取写入缓冲区
          │   scanlinecallback(-1)    │ ← 通知当前行写入完成
          └──────────────────────────┘
```

### 1.1 与传统加载模式的对比

| 特性 | 传统模式（返回位图） | KrKr2 回调模式 |
|:---|:---|:---|
| 内存分配 | 解码器内部分配 | 调用方通过回调提供缓冲区 |
| 流式处理 | 不支持，必须全部解码完成 | 支持，逐行回调 |
| 格式扩展 | 需要修改加载函数签名 | 只需实现统一回调接口 |
| 插件兼容 | 需要共享内存分配器 | 无此限制，缓冲区由宿主管理 |
| 错误处理 | 返回空指针或错误码 | 抛出异常，由调用方 catch |

## 2. 核心回调类型详解

### 2.1 尺寸回调：tTVPGraphicSizeCallback

```cpp
// 定义在 GraphicsLoaderIntf.h 第 28-29 行
typedef int (*tTVPGraphicSizeCallback)(
    void *callbackdata,          // 调用方传入的上下文数据
    tjs_uint w,                  // 图像宽度（像素）
    tjs_uint h,                  // 图像高度（像素）
    tTVPGraphicPixelFormat fmt   // 像素格式枚举
);
// 返回值：行跨距（line pitch，即每行字节数）
```

尺寸回调（Size Callback）是加载过程中第一个被调用的回调函数。它的职责是告知调用方图像的尺寸和像素格式，让调用方据此分配缓冲区。返回值为行跨距（line pitch），即每行像素数据占用的字节数（可能因内存对齐大于 `width × bytesPerPixel`）。

像素格式枚举 `tTVPGraphicPixelFormat` 定义了四种基本格式：

```cpp
// 定义在 GraphicsLoaderIntf.h 第 22 行
enum tTVPGraphicPixelFormat {
    gpfLuminance,   // 灰度（8位亮度通道）
    gpfPalette,     // 调色板（8位索引 + 外部颜色表）
    gpfRGB,         // RGB（24位，无 Alpha）
    gpfRGBA         // RGBA（32位，含 Alpha 通道）
};
```

### 2.2 扫描行回调：tTVPGraphicScanLineCallback

```cpp
// 定义在 GraphicsLoaderIntf.h 第 35 行
typedef void *(*tTVPGraphicScanLineCallback)(
    void *callbackdata,   // 调用方传入的上下文数据
    tjs_int y             // 行号（0 起始），或 -1 表示写入完成通知
);
// 返回值：指向该行缓冲区的指针（用于写入解码后的像素数据）
//         返回 nullptr 可以中止加载过程
```

扫描行回调（Scan Line Callback）是逐行回调的核心。每次调用时，加载器传入当前要写入的行号 `y`，回调返回一个指向该行缓冲区的指针。加载器将解码后的像素数据写入该缓冲区后，再次调用回调并传入 `y = -1`，表示当前行的数据已经写入完毕。

这个"写入完成通知"（y = -1）机制非常关键：

1. **同步点**：调用方可以在收到 -1 通知时立即处理该行数据（如纹理上传、显示更新）。
2. **流水线**：允许调用方在下一行解码的同时处理上一行数据。
3. **双缓冲**：调用方可以在两个缓冲区之间交替，避免读写冲突。

### 2.3 元信息回调：tTVPMetaInfoPushCallback

```cpp
// 定义在 GraphicsLoaderIntf.h 第 49-50 行
typedef void (*tTVPMetaInfoPushCallback)(
    void *callbackdata,       // 上下文数据
    const ttstr &name,        // 元信息键名（如 "offs_x"、"vpag_w"）
    const ttstr &value        // 元信息值（字符串形式）
);
```

元信息回调用于传递图像中嵌入的元数据。这个回调可以为 `nullptr`（表示调用方不关心元信息）。典型的元信息包括：

| 键名 | 含义 | 来源格式 |
|:---|:---|:---|
| `offs_x` / `offs_y` | 图像偏移量 | PNG (oFFs chunk) |
| `reso_x` / `reso_y` | 分辨率 | PNG (pHYs chunk) |
| `vpag_w` / `vpag_h` | 虚拟页面尺寸 | PNG (vpAg chunk) |
| TLG SDS tags | 任意键值对 | TLG (SDS 结构化数据) |

### 2.4 完整的回调交互时序

下面用代码演示一个完整的回调交互过程：

```cpp
#include "GraphicsLoaderIntf.h"
#include <cstdio>
#include <cstdlib>
#include <cstring>

// 用于存储加载上下文的结构体
struct MyLoadContext {
    tjs_uint width;       // 图像宽度
    tjs_uint height;      // 图像高度
    tjs_uint8 *pixels;    // 像素缓冲区
    int pitch;            // 行跨距
};

// 尺寸回调实现
static int MySizeCallback(void *callbackdata, tjs_uint w, tjs_uint h,
                           tTVPGraphicPixelFormat fmt) {
    MyLoadContext *ctx = (MyLoadContext *)callbackdata;
    ctx->width = w;
    ctx->height = h;

    // 根据像素格式计算行跨距，确保 16 字节对齐
    int bpp = (fmt == gpfRGBA) ? 4 : (fmt == gpfRGB) ? 3 : 1;
    ctx->pitch = (w * bpp + 15) & ~15; // 16 字节对齐

    // 分配像素缓冲区
    ctx->pixels = (tjs_uint8 *)malloc(ctx->pitch * h);
    if (!ctx->pixels) {
        // 内存不足，返回 0 表示错误
        return 0;
    }
    memset(ctx->pixels, 0, ctx->pitch * h);

    printf("图像尺寸: %u x %u, 格式: %d, 行跨距: %d\n", w, h, fmt, ctx->pitch);
    return ctx->pitch; // 返回行跨距
}

// 扫描行回调实现
static void *MyScanLineCallback(void *callbackdata, tjs_int y) {
    MyLoadContext *ctx = (MyLoadContext *)callbackdata;

    if (y == -1) {
        // 写入完成通知 — 必须返回 nullptr
        return nullptr;
    }

    if (y < 0 || (tjs_uint)y >= ctx->height) {
        return nullptr; // 越界保护
    }

    // 返回第 y 行缓冲区的起始地址
    return ctx->pixels + y * ctx->pitch;
}

// 元信息回调实现
static void MyMetaInfoCallback(void *callbackdata,
                                const ttstr &name, const ttstr &value) {
    // 打印接收到的元信息
    printf("元信息: %ls = %ls\n", name.c_str(), value.c_str());
}
```

## 3. 格式路由器：自动格式识别

### 3.1 魔数检测机制

KrKr2 不依赖文件扩展名来确定图像格式，而是通过读取文件头的前 16 个字节（魔数，Magic Number）来自动识别。这种方式更加可靠，因为文件扩展名可能被修改或缺失。

路由器函数 `TVPLoadGraphicRouter` 定义在 `GraphicsLoaderIntf.cpp` 第 50-91 行：

```cpp
// 来源：krkr2/cpp/core/visual/GraphicsLoaderIntf.cpp 第 50-91 行
static void TVPLoadGraphicRouter(
    void *formatdata, void *callbackdata,
    tTVPGraphicSizeCallback sizecallback,
    tTVPGraphicScanLineCallback scanlinecallback,
    tTVPMetaInfoPushCallback metainfopushcallback,
    tTJSBinaryStream *src, tjs_int keyidx,
    tTVPGraphicLoadMode mode)
{
    uint8_t header[16];
    tjs_uint64 origSrcPos = src->GetPosition(); // 保存流位置
    if (src->Read(header, sizeof(header)) == sizeof(header)) {
        src->SetPosition(origSrcPos);            // 恢复流位置（不消耗数据）

        // 按优先级逐个匹配魔数
        if (!memcmp(header, "BM", 2))             return TVPLoadBMP(...);
        if (!memcmp(header, "\x89PNG", 4))        return TVPLoadPNG(...);
        if (!memcmp(header, "TLG", 3))            return TVPLoadTLG(...);
        if (!memcmp(header, "\xFF\xD8\xFF", 3)
            && header[3] >= 0xE0
            && header[3] <= 0xEF)                 return TVPLoadJPEG(...);
        if (!memcmp(header, "BPG", 3))            return TVPLoadBPG(...);
        if (!memcmp(header, "RIFF", 4)
            && !memcmp(header + 8, "WEBPVP8", 7)) return TVPLoadWEBP(...);
        if (!memcmp(header, "\x49\x49\xbc\x01", 4)) return TVPLoadJXR(...);
        if (!memcmp(header, "PVR\3", 4))          return TVPLoadPVRv3(...);
    }
    TVPThrowExceptionMessage(TVPImageLoadError, TJS_W("Invalid image"));
}
```

### 3.2 支持的格式与魔数对照表

| 格式 | 魔数（十六进制） | 匹配字节数 | 备注 |
|:---|:---|:---|:---|
| BMP | `42 4D` ("BM") | 2 | Windows 位图 |
| PNG | `89 50 4E 47` ("\x89PNG") | 4 | 含 CRC 校验 |
| TLG | `54 4C 47` ("TLG") | 3 | KiriKiri 原生格式 |
| JPEG | `FF D8 FF E0-EF` | 4 | JFIF/EXIF 标记 |
| BPG | `42 50 47` ("BPG") | 3 | Better Portable Graphics |
| WebP | `52 49 46 46` + `57 45 42 50 56 50 38` | 11 | RIFF 容器 |
| JPEG XR | `49 49 BC 01` | 4 | Little-endian TIFF 变体 |
| PVRv3 | `50 56 52 03` ("PVR\3") | 4 | PowerVR 纹理 |

### 3.3 同样的路由也用于头信息加载

```cpp
// 来源：GraphicsLoaderIntf.cpp 第 93-128 行
static void TVPLoadHeaderRouter(void *formatdata,
                                 tTJSBinaryStream *src,
                                 iTJSDispatch2 **dic) {
    uint8_t header[16];
    tjs_uint64 origSrcPos = src->GetPosition();
    if (src->Read(header, sizeof(header)) == sizeof(header)) {
        src->SetPosition(origSrcPos);
        // 与加载路由完全相同的魔数匹配逻辑
        if (!memcmp(header, "BM", 2))    return TVPLoadHeaderBMP(...);
        if (!memcmp(header, "\x89PNG", 4)) return TVPLoadHeaderPNG(...);
        // ... 其余格式同上
    }
}
```

头信息加载（Header Loading）用于快速读取图像的基本属性（宽、高、BPP）而不解码像素数据。返回的字典对象包含 `width`、`height`、`bpp` 三个字段。

## 4. 格式处理器注册表

### 4.1 tTVPGraphicHandlerType 结构

每种图像格式在注册表中对应一个 `tTVPGraphicHandlerType` 实例，它封装了该格式的所有处理函数：

```cpp
// 简化展示，实际定义更复杂
struct tTVPGraphicHandlerType {
    ttstr Extension;                          // 文件扩展名（如 ".png"）
    bool IsPlugin;                            // 是否来自插件

    // 原生处理函数（使用 tTJSBinaryStream）
    tTVPGraphicLoadingHandler NativeLoading;
    tTVPGraphicHeaderLoadingHandler NativeHeader;
    tTVPGraphicSaveHandler NativeSave;
    tTVPGraphicAcceptSaveHandler NativeAcceptSave;

    // 插件处理函数（使用 IStream）
    tTVPGraphicLoadingHandlerForPlugin PluginLoading;
    tTVPGraphicHeaderLoadingHandlerForPlugin PluginHeader;
    tTVPGraphicSaveHandlerForPlugin PluginSave;

    void *FormatData;                         // 格式特定数据

    // 统一调用接口
    void Load(tTJSBinaryStream *src, ...);    // 根据 IsPlugin 选择调用
    void Header(tTJSBinaryStream *src, ...);
    void Save(const ttstr &name, ...);
    bool AcceptSave(const ttstr &type, ...);
};
```

### 4.2 注册表管理类：tTVPGraphicType

```cpp
// 来源：GraphicsLoaderIntf.cpp 第 160-247 行
class tTVPGraphicType {
public:
    // 哈希表，按扩展名快速查找
    tTJSHashTable<ttstr, tTVPGraphicHandlerType> Hash;
    // 有序列表，保留注册顺序
    std::vector<tTVPGraphicHandlerType> Handlers;

    static bool Avail; // 全局可用标志

    tTVPGraphicType() {
        // 构造时注册所有内置格式
        Handlers.push_back(tTVPGraphicHandlerType(
            TJS_W(".pvr"), TVPLoadGraphicRouter, TVPLoadHeaderRouter,
            nullptr, nullptr, nullptr));
        Handlers.push_back(tTVPGraphicHandlerType(
            TJS_W(".bmp"), TVPLoadGraphicRouter, TVPLoadHeaderRouter,
            TVPSaveAsBMP, TVPAcceptSaveAsBMP, nullptr));
        Handlers.push_back(tTVPGraphicHandlerType(
            TJS_W(".png"), TVPLoadGraphicRouter, TVPLoadHeaderRouter,
            TVPSaveAsPNG, TVPAcceptSaveAsPNG, nullptr));
        // ... 注册 .jpg, .jpeg, .jif, .tlg, .tlg5, .tlg6, .webp, .jxr, .bpg
        ReCreateHash(); // 重建哈希索引
        Avail = true;
    }

    void ReCreateHash() {
        // 遍历 Handlers 列表，将所有扩展名插入哈希表
        for (auto i = Handlers.begin(); i != Handlers.end(); i++) {
            Hash.Add(i->Extension, *i);
        }
    }

    void Register(const tTVPGraphicHandlerType &handler) {
        Handlers.push_back(handler);
        ReCreateHash(); // 注册后重建索引
    }

    void Unregister(const tTVPGraphicHandlerType &handler) {
        // 从列表中移除匹配项
        for (auto i = Handlers.begin(); i != Handlers.end(); i++) {
            if (handler == *i) { Handlers.erase(i); break; }
        }
        ReCreateHash();
    }
} static TVPGraphicType; // 全局单例
```

## 5. 加载模式与颜色键

### 5.1 三种加载模式

```cpp
// 定义在 GraphicsLoaderIntf.h 第 56-60 行
enum tTVPGraphicLoadMode {
    glmNormal,      // 普通模式：输出 32 位 ARGB
    glmPalettized,  // 调色板模式：输出 8 位索引（用于过渡规则图）
    glmGrayscale    // 灰度模式：输出 8 位灰度（用于蒙版/遮罩）
};
```

- **glmNormal**（默认）：所有格式都支持，输出标准 32 位 ARGB 像素。
- **glmPalettized**：仅用于加载"过渡规则图"（Province Image）。像素颜色不重要，关键是保留颜色索引值。部分格式（如 TLG5）不支持此模式，会抛出异常。
- **glmGrayscale**：用于加载遮罩和蒙版图像。输出 8 位灰度值。

### 5.2 颜色键（Color Key）

颜色键 `keyidx` 参数用于将指定颜色索引设置为完全透明。这是一种传统的透明度实现方式，常见于 8 位或更低色深的图像：

```cpp
// 在 BMP 加载器中的颜色键处理
// 来源：GraphicsLoaderIntf.cpp 第 457-460 行
if (keyidx != -1) {
    // 将指定索引的调色板颜色的 Alpha 设为 0（完全透明）
    palette[keyidx & 0xff] &= 0x00ffffff;
}
```

当 `keyidx` 为 -1 时表示不使用颜色键。颜色键不会覆盖图像自身的 Alpha 通道——如果图像本身就有 Alpha 信息，优先使用图像自身的 Alpha。

## 6. 图像缓存系统

KrKr2 内置了一个图像缓存系统，避免重复加载相同的图像资源：

```cpp
// 缓存相关 API（声明在 GraphicsLoaderIntf.h）
extern tjs_uint64 TVPGraphicCacheSystemLimit; // 缓存上限（字节）

void TVPSetGraphicCacheLimit(tjs_uint64 limit);  // 设置缓存上限
void TVPClearGraphicCache();                     // 清空所有缓存

// 缓存操作
void TVPPushGraphicCache(const ttstr &name,
                          /* 缓存数据 */);       // 将图像推入缓存
bool TVPCheckImageCache(const ttstr &name,
                         /* 输出参数 */);         // 检查缓存是否命中
bool TVPHasImageCache(const ttstr &name);        // 快速检查缓存是否存在
void TVPTouchImages(const ttstr &pattern);       // 预加载匹配的图像
```

缓存系统的工作流程：

```
加载请求 → 检查缓存(TVPCheckImageCache)
    ├─ 命中 → 直接返回缓存的纹理数据
    └─ 未命中 → 执行完整加载流程
         └─ 加载完成 → 推入缓存(TVPPushGraphicCache)
              └─ 若超出限制 → LRU 淘汰旧条目
```

## 动手实践

### 练习 1：模拟回调加载流程

创建一个测试程序，模拟 KrKr2 的回调加载过程：

```cpp
#include <cstdio>
#include <cstdlib>
#include <cstring>
#include <cstdint>

// 模拟 KrKr2 的类型定义
typedef unsigned int tjs_uint;
typedef int tjs_int;
typedef uint8_t tjs_uint8;
enum tTVPGraphicPixelFormat { gpfLuminance, gpfPalette, gpfRGB, gpfRGBA };

struct SimpleImage {
    tjs_uint width, height;
    int pitch;
    tjs_uint8 *data;
};

// 尺寸回调
static int OnSize(void *ctx, tjs_uint w, tjs_uint h,
                   tTVPGraphicPixelFormat fmt) {
    SimpleImage *img = (SimpleImage *)ctx;
    img->width = w;
    img->height = h;
    int bpp = (fmt == gpfRGBA) ? 4 : (fmt == gpfRGB) ? 3 : 1;
    img->pitch = (w * bpp + 15) & ~15;
    img->data = (tjs_uint8 *)calloc(img->pitch * h, 1);
    printf("[SizeCallback] %ux%u, bpp=%d, pitch=%d\n", w, h, bpp, img->pitch);
    return img->pitch;
}

// 扫描行回调
static void *OnScanLine(void *ctx, tjs_int y) {
    SimpleImage *img = (SimpleImage *)ctx;
    if (y == -1) {
        printf("[ScanLine] 行写入完成通知\n");
        return nullptr;
    }
    printf("[ScanLine] 请求第 %d 行缓冲区\n", y);
    return img->data + y * img->pitch;
}

// 模拟一个 4x2 的 RGBA 图像加载器
static void FakeLoader(void *ctx,
                        int (*sizecb)(void *, tjs_uint, tjs_uint, tTVPGraphicPixelFormat),
                        void *(*scanlinecb)(void *, tjs_int)) {
    // 1. 通知尺寸
    sizecb(ctx, 4, 2, gpfRGBA);

    // 2. 逐行写入像素数据
    for (int y = 0; y < 2; y++) {
        tjs_uint8 *line = (tjs_uint8 *)scanlinecb(ctx, y);
        if (!line) break;
        // 填充红色 (BGRA 格式，KrKr2 内部使用 BGRA)
        for (int x = 0; x < 4; x++) {
            line[x * 4 + 0] = 0;     // B
            line[x * 4 + 1] = 0;     // G
            line[x * 4 + 2] = 255;   // R
            line[x * 4 + 3] = 255;   // A
        }
        scanlinecb(ctx, -1); // 通知写入完成
    }
}

int main() {
    SimpleImage img = {};
    FakeLoader(&img, OnSize, OnScanLine);

    // 验证结果
    printf("\n加载完成: %ux%u\n", img.width, img.height);
    printf("第一个像素 BGRA: %02X %02X %02X %02X\n",
           img.data[0], img.data[1], img.data[2], img.data[3]);

    free(img.data);
    return 0;
}
```

## 常见错误及解决方案

### 错误 1：扫描行回调中忘记处理 y = -1

```
症状：图像加载后出现花屏或数据错乱
原因：y = -1 是写入完成通知，必须返回 nullptr
修复：在回调函数开头检查 y == -1
```

```cpp
// ❌ 错误写法
void *MyScanLine(void *ctx, int y) {
    return ((MyCtx*)ctx)->buffer + y * pitch; // y=-1 时会越界！
}

// ✅ 正确写法
void *MyScanLine(void *ctx, int y) {
    if (y == -1) return nullptr;  // 写入完成通知
    return ((MyCtx*)ctx)->buffer + y * pitch;
}
```

### 错误 2：注册格式时扩展名未小写

```
症状：自定义格式无法被识别
原因：注册表使用小写扩展名进行哈希查找
修复：确保扩展名全部小写，以点号开头
```

```cpp
// ❌ 错误写法
TVPRegisterGraphicLoadingHandler(TJS_W(".AVIF"), ...); // 大写！

// ✅ 正确写法
TVPRegisterGraphicLoadingHandler(TJS_W(".avif"), ...); // 小写
```

### 错误 3：路由器中未恢复流位置

```
症状：所有格式加载都失败，报 "Invalid image"
原因：读取魔数后未调用 SetPosition 恢复流位置
修复：魔数检测后必须恢复到原始位置
```

## 本节小结

- KrKr2 图像加载系统采用**回调驱动架构**，加载器通过 `sizecallback` 和 `scanlinecallback` 与调用方交互。
- `scanlinecallback(y=-1)` 是**写入完成通知**，必须返回 `nullptr`。
- 格式路由器通过读取文件头前 16 字节的**魔数**自动识别格式，支持 8 种内置格式。
- `tTVPGraphicType` 全局单例管理所有注册的格式处理器，支持运行时动态注册/注销。
- 三种加载模式（Normal / Palettized / Grayscale）适配不同的使用场景。
- 图像缓存系统通过 LRU 策略避免重复加载。

## 练习题与答案

### 题目 1：为什么 KrKr2 选择回调驱动而非返回位图对象的加载模式？列出至少三个技术原因。

<details>
<summary>查看答案</summary>

1. **内存效率**：回调模式下，调用方控制缓冲区分配。解码器不需要内部分配临时缓冲区，避免了大图像解码时的双倍内存消耗。
2. **流式处理**：逐行回调允许渐进式显示——调用方可以在解码过程中就开始显示已解码的行，而不必等待整张图像解码完成。
3. **格式无关性**：所有格式加载器共享完全相同的回调接口。调用方（如 `tTVPBaseBitmap`）只需实现一套回调，就能接收任何格式的像素数据。
4. **插件兼容**：回调模式避免了解码器和调用方共享内存分配器的需求。插件可以使用不同的运行时（如不同版本的 MSVC CRT），只要回调接口一致即可。

</details>

### 题目 2：如果要让 KrKr2 支持 AVIF 格式，需要在哪些地方做修改？写出注册代码。

<details>
<summary>查看答案</summary>

需要修改两个地方：

1. **创建加载器文件** `LoadAVIF.cpp`，实现 `TVPLoadAVIF` 和 `TVPLoadHeaderAVIF` 函数。
2. **在路由器中添加魔数匹配**（如果不使用扩展名匹配的话）。
3. **注册格式处理器**。

注册代码：

```cpp
// 在 GraphicsLoaderIntf.cpp 的 tTVPGraphicType 构造函数中添加
Handlers.push_back(tTVPGraphicHandlerType(
    TJS_W(".avif"),         // 扩展名
    TVPLoadGraphicRouter,   // 使用路由器（需要在路由器中添加 AVIF 魔数检测）
    TVPLoadHeaderRouter,    // 头信息路由器
    nullptr,                // 暂不支持保存
    nullptr,                // 暂不支持保存格式判断
    nullptr                 // 无格式特定数据
));

// 或者，如果不使用路由器，直接注册专用加载函数：
Handlers.push_back(tTVPGraphicHandlerType(
    TJS_W(".avif"),
    TVPLoadAVIF,            // 专用加载函数
    TVPLoadHeaderAVIF,      // 专用头信息加载函数
    nullptr, nullptr, nullptr));
```

AVIF 格式的魔数检测（添加到路由器中）：

```cpp
// AVIF 使用 ISOBMFF 容器，ftypavif 标记
// 文件头: XX XX XX XX 66 74 79 70 61 76 69 66
// 偏移4开始为 "ftypavif"
if (!memcmp(header + 4, "ftypavif", 8))
    return CALL_LOAD_FUNC(TVPLoadAVIF);
```

</details>

### 题目 3：编写一个回调函数，将加载的 RGBA 图像实时转换为灰度显示。

<details>
<summary>查看答案</summary>

```cpp
#include <cstdint>
#include <cstdio>

struct GrayConvertContext {
    uint32_t width, height;
    int pitch;
    uint8_t *rgbaBuffer;   // 原始 RGBA 数据
    uint8_t *grayBuffer;   // 灰度输出
    uint32_t currentLine;
};

static int GraySizeCallback(void *ctx, unsigned int w, unsigned int h,
                             int fmt) {
    GrayConvertContext *gc = (GrayConvertContext *)ctx;
    gc->width = w;
    gc->height = h;
    gc->pitch = (w * 4 + 15) & ~15;
    gc->rgbaBuffer = new uint8_t[gc->pitch * h];
    gc->grayBuffer = new uint8_t[w * h];
    gc->currentLine = 0;
    return gc->pitch;
}

static void *GrayScanLineCallback(void *ctx, int y) {
    GrayConvertContext *gc = (GrayConvertContext *)ctx;

    if (y == -1) {
        // 写入完成通知 — 立即将当前行转换为灰度
        uint32_t line = gc->currentLine;
        uint8_t *src = gc->rgbaBuffer + line * gc->pitch;
        uint8_t *dst = gc->grayBuffer + line * gc->width;
        for (uint32_t x = 0; x < gc->width; x++) {
            uint8_t b = src[x * 4 + 0];
            uint8_t g = src[x * 4 + 1];
            uint8_t r = src[x * 4 + 2];
            // ITU-R BT.601 亮度公式
            dst[x] = (uint8_t)(0.299 * r + 0.587 * g + 0.114 * b);
        }
        gc->currentLine++;
        return nullptr;
    }

    return gc->rgbaBuffer + y * gc->pitch;
}

// 使用示例：
// GrayConvertContext ctx = {};
// TVPLoadPNG(nullptr, &ctx, GraySizeCallback,
//            GrayScanLineCallback, nullptr, stream, -1, glmNormal);
// 此时 ctx.grayBuffer 中包含灰度数据
```

</details>
