## 4.3.1 着色器系统架构

KrKr2 的 OGL 后端使用一套基于字符串拼接的 GLSL 着色器系统。所有混合效果（Alpha 混合、加色混合、颜色映射等）都通过动态生成的 GLSL 片段着色器实现。

### tTVPOGLRenderMethod 基类

`tTVPOGLRenderMethod` 封装了一个 GL 着色器程序和混合状态：

```cpp
// 源码位置：RenderManager_ogl.cpp:1821-1870（简化）
class tTVPOGLRenderMethod : public iTVPRenderMethod {
protected:
    GLuint program;           // 链接好的着色器程序
    GLint posAttr;            // a_position 属性位置
    GLint texCoordAttr[4];    // a_texCoord0..3 属性位置
    GLenum blendSrcRGB, blendDstRGB;    // 颜色混合因子
    GLenum blendSrcA, blendDstA;        // Alpha 混合因子
    GLenum blendEquation;               // 混合方程（通常 GL_FUNC_ADD）
    bool tar_as_src;          // 是否需要读取目标纹理作为源

public:
    // 自定义处理回调（用于特殊渲染路径）
    typedef bool (*CustomProcFunc)(
        tTVPOGLRenderMethod*, tTVPOGLTexture2D* tar,
        tTVPOGLTexture2D* ref, const tTVPRect*,
        const tRenderTexRectArray&);
    CustomProcFunc CustomProc;

    void Apply() {
        // 激活着色器程序并设置混合状态
        cocos2d::GL::useProgram(program);
        glBlendFuncSeparate(blendSrcRGB, blendDstRGB, blendSrcA, blendDstA);
        glBlendEquation(blendEquation);
    }

    void SetBlendFuncSeparate(GLenum eq,
        GLenum srcRGB, GLenum dstRGB,
        GLenum srcA, GLenum dstA)
    {
        blendEquation = eq;
        blendSrcRGB = srcRGB; blendDstRGB = dstRGB;
        blendSrcA = srcA;     blendDstA = dstA;
    }
};
```

### tTVPOGLRenderMethod_Script：脚本化着色器

`tTVPOGLRenderMethod_Script` 是最常用的渲染方法子类。它通过字符串拼接生成完整的 GLSL 着色器：

```cpp
// 源码位置：RenderManager_ogl.cpp:1880-1960（简化）
class tTVPOGLRenderMethod_Script : public tTVPOGLRenderMethod {
private:
    // 顶点着色器缓存（按纹理数量索引）
    static std::map<int, GLuint> VertShaderCache;

public:
    void init(const char* body, int nTextures, const char* prefix = nullptr) {
        // 1. 构建片段着色器源码
        std::string fragSrc;

        // 精度声明（移动端必须）
        fragSrc += "precision mediump float;\n";

        // 可选前缀（framebuffer fetch 扩展声明等）
        if (prefix) fragSrc += prefix;

        // 纹理采样器声明
        for (int i = 0; i < nTextures; ++i) {
            fragSrc += "uniform sampler2D tex" + std::to_string(i) + ";\n";
        }

        // varying 变量声明（从顶点着色器接收纹理坐标）
        for (int i = 0; i < nTextures; ++i) {
            fragSrc += "varying vec2 v_texCoord" + std::to_string(i) + ";\n";
        }

        // 用户提供的着色器主体
        fragSrc += body;

        // 2. 编译片段着色器
        GLuint fragShader = CompileShader(GL_FRAGMENT_SHADER, fragSrc.c_str());

        // 3. 获取或创建顶点着色器（按纹理数量缓存）
        GLuint vertShader = GetVertShader(nTextures);

        // 4. 链接程序
        program = CombineProgram(vertShader, fragShader);

        // 5. 获取属性位置
        posAttr = glGetAttribLocation(program, "a_position");
        for (int i = 0; i < nTextures; ++i) {
            std::string name = "a_texCoord" + std::to_string(i);
            texCoordAttr[i] = glGetAttribLocation(program, name.c_str());
        }
    }

    // 顶点着色器生成（按纹理数量缓存）
    static GLuint GetVertShader(int nTextures) {
        auto it = VertShaderCache.find(nTextures);
        if (it != VertShaderCache.end()) return it->second;

        std::string src = "attribute vec2 a_position;\n";
        for (int i = 0; i < nTextures; ++i) {
            src += "attribute vec2 a_texCoord" + std::to_string(i) + ";\n";
            src += "varying vec2 v_texCoord" + std::to_string(i) + ";\n";
        }
        src += "void main() {\n";
        src += "    gl_Position = vec4(a_position, 0.0, 1.0);\n";
        for (int i = 0; i < nTextures; ++i) {
            src += "    v_texCoord" + std::to_string(i) +
                   " = a_texCoord" + std::to_string(i) + ";\n";
        }
        src += "}\n";

        GLuint shader = CompileShader(GL_VERTEX_SHADER, src.c_str());
        VertShaderCache[nTextures] = shader;
        return shader;
    }
};
```

## 4.3.2 Framebuffer Fetch 优化

这是 KrKr2 在移动端的关键优化。传统的混合操作需要两张纹理——源纹理和目标纹理的拷贝（因为 GPU 不能同时读写同一张纹理）。`GL_EXT_shader_framebuffer_fetch` 扩展允许片段着色器直接读取帧缓冲区中的当前值。

```
传统路径（无 framebuffer fetch）：
  1. 将目标纹理拷贝到临时纹理（GetTempTexture2D）
  2. 绑定 2 张纹理：源 + 临时目标拷贝
  3. 着色器读取两张纹理做混合
  4. 输出到目标纹理
  额外开销：一次纹理拷贝 + 一个纹理单元

优化路径（有 framebuffer fetch）：
  1. 绑定 1 张纹理：仅源纹理
  2. 着色器通过 gl_LastFragColor 读取当前帧缓冲值
  3. 直接做混合输出
  节省：无拷贝 + 少用一个纹理单元
```

引擎在编译混合方法时自动选择路径：

```cpp
// 源码位置：RenderManager_ogl.cpp:2421-2500（简化）
void CompileAndRegRegularBlendMethod(
    const char* name,
    const std::string& prefix,
    const char* blendBody)
{
    if (GL_CHECK_shader_framebuffer_fetch) {
        // 路径A：framebuffer fetch（1 纹理）
        std::string shader = prefix +
            "void main(){\n"
            "    vec4 s = texture2D(tex0, v_texCoord0);\n"
            "    vec4 d = gl_LastFragColor;\n"  // 直接读取帧缓冲
            + blendBody;

        CompileAndRegScript<tTVPOGLRenderMethod_Script>(
            name, shader, 1,
            common_shader_framebuffer_fetch_prefix);
            // prefix 声明: #extension GL_EXT_shader_framebuffer_fetch : require
    } else {
        // 路径B：target-as-source（2 纹理）
        std::string shader = prefix +
            "void main(){\n"
            "    vec4 s = texture2D(tex0, v_texCoord0);\n"
            "    vec4 d = texture2D(tex1, v_texCoord1);\n"  // 从拷贝读取
            + blendBody;

        auto* method = CompileAndRegScript<tTVPOGLRenderMethod_Script>(
            name, shader, 2);
        method->SetTargetAsSrc();  // 标记需要目标纹理拷贝
    }
}
```

## 4.3.3 OperateRect 完整流程

`OperateRect` 是最常用的绘制入口，完成一次"源纹理到目标纹理"的矩形绘制。以下是完整执行流程：

```cpp
// 源码位置：RenderManager_ogl.cpp:4498-4636（简化注释版）
void OperateRect(iTVPRenderMethod* _method, iTVPTexture2D* _tar,
                 iTVPTexture2D* reftar, const tTVPRect& rctar,
                 const tRenderTexRectArray& textures)
{
    ++_drawCount;  // 统计 draw call 数

    tTVPOGLRenderMethod* method = (tTVPOGLRenderMethod*)_method;
    tTVPOGLTexture2D* tar = (tTVPOGLTexture2D*)_tar;

    // 步骤1：检查自定义处理回调
    if (method->CustomProc) {
        if (method->CustomProc(method, tar, reftar, &rctar, textures))
            return;  // 自定义处理已完成
    }

    // 步骤2：准备纹理坐标
    std::vector<GLVertexInfo> texlist;
    // 如果方法需要 tar_as_src，额外分配一个纹理槽位
    texlist.resize(textures.size() + (method->tar_as_src ? 1 : 0));

    for (int i = 0; i < textures.size(); ++i) {
        auto* tex = (tTVPOGLTexture2D*)textures[i].first;
        tex->SyncPixel();  // 确保 CPU 修改已上传到 GPU

        // 特殊情况：源纹理与目标纹理相同
        if (tex == tar) {
            // 需要拷贝到临时纹理避免读写冲突
            auto* newtex = GetTempTexture2D(tex, textures[i].second);
            newtex->ApplyVertex(texlist[i], ...);
        } else {
            tex->ApplyVertex(texlist[i], textures[i].second);
        }
    }

    // 步骤3：处理 tar_as_src（无 framebuffer fetch 时）
    if (method->tar_as_src) {
        auto* newtex = GetTempTexture2D(tar, rctar);
        newtex->ApplyVertex(texlist.back(), ...);
    }

    // 步骤4：设置渲染状态
    // NDC 全屏四边形（用 glViewport 控制实际绘制区域）
    static const GLfloat vertices[12] = {
        -1, -1,   1, -1,  -1,  1,   // 三角形 1
         1, -1,  -1,  1,   1,  1    // 三角形 2
    };

    method->Apply();           // 激活着色器 + 混合状态
    tar->AsTarget();           // 将目标纹理绑定到 FBO

    // 设置 viewport 到目标区域（考虑纹理缩放因子）
    glViewport(
        rctar.left * tar->_scaleW,
        rctar.top * tar->_scaleH,
        rctar.get_width() * tar->_scaleW,
        rctar.get_height() * tar->_scaleH);

    // 步骤5：绑定顶点属性
    int VA_flag = 1 << method->GetPosAttr();
    for (int i = 0; i < texlist.size(); ++i) {
        VA_flag |= 1 << method->GetTexCoordAttr(i);
    }
    cocos2d::GL::enableVertexAttribs(VA_flag);
    glVertexAttribPointer(method->GetPosAttr(), 2, GL_FLOAT, GL_FALSE,
                          0, vertices);

    // 步骤6：绑定纹理并设置纹理坐标
    for (int i = 0; i < texlist.size(); ++i) {
        method->ApplyTexture(i, texlist[i]);
    }

    // 步骤7：绘制
    glDrawArrays(GL_TRIANGLES, 0, 6);  // 2 个三角形 = 1 个四边形

    method->onFinish();
    CHECK_GL_ERROR_DEBUG();
}
```

这个流程可以用时序图表示：

```
OperateRect 调用
  │
  ├── 1. CustomProc 检查 → 有自定义处理则直接返回
  │
  ├── 2. 源纹理坐标准备
  │   ├── SyncPixel() → CPU→GPU 同步
  │   └── 源==目标？ → 拷贝到临时纹理
  │
  ├── 3. tar_as_src 处理
  │   └── 拷贝目标纹理到临时纹理
  │
  ├── 4. 渲染状态设置
  │   ├── method->Apply() → useProgram + blendFunc
  │   ├── tar->AsTarget() → FBO 绑定
  │   └── glViewport → 目标区域
  │
  ├── 5. 顶点属性绑定
  │   ├── enableVertexAttribs
  │   └── glVertexAttribPointer × N
  │
  ├── 6. 纹理绑定
  │   └── ApplyTexture × N → glActiveTexture + glBindTexture
  │
  └── 7. glDrawArrays(GL_TRIANGLES, 0, 6)
```

## 4.3.4 模板缓冲（Stencil Buffer）

模板缓冲用于实现图层裁剪遮罩。当一个图层被设置为另一个图层的遮罩时，引擎先将遮罩图层写入模板缓冲，然后在绘制被遮罩图层时启用模板测试。

```cpp
// BeginStencil/EndStencil 的使用流程
void RenderWithMask(iTVPRenderManager* mgr,
                    iTVPTexture2D* target,
                    iTVPTexture2D* maskLayer,
                    iTVPTexture2D* contentLayer)
{
    // 1. 开始模板写入
    mgr->BeginStencil(target);
    // 内部：glEnable(GL_STENCIL_TEST)
    //        glStencilFunc(GL_ALWAYS, 1, 0xFF)
    //        glStencilOp(GL_KEEP, GL_KEEP, GL_REPLACE)
    //        glColorMask(0, 0, 0, 0)  // 禁止颜色写入

    // 2. 绘制遮罩形状（只写入模板值）
    mgr->OperateRect(target, nullptr, ..., maskLayer, copyMethod);

    // 3. 结束模板写入，开始模板测试
    mgr->EndStencil(target, false);
    // 内部：glStencilFunc(GL_EQUAL, 1, 0xFF)
    //        glStencilOp(GL_KEEP, GL_KEEP, GL_KEEP)
    //        glColorMask(1, 1, 1, 1)  // 恢复颜色写入

    // 4. 绘制内容（只有模板值为 1 的区域通过测试）
    mgr->OperateRect(target, nullptr, ..., contentLayer, blendMethod);

    // 5. 清理模板状态
    glDisable(GL_STENCIL_TEST);
}
```

## 4.3.5 GL 上下文丢失恢复

Android 平台上，当应用进入后台或 GL 上下文被系统回收时，所有 GL 资源（纹理、着色器、FBO）都会失效。引擎通过 Cocos2d-x 的事件系统监听上下文重建：

```cpp
// 源码位置：RenderManager_ogl.cpp:2953-2966
cocos2d::EventListenerCustom* listener =
    cocos2d::EventListenerCustom::create(
        EVENT_RENDERER_RECREATED,
        [this](cocos2d::EventCustom*) {
            // 1. 清除着色器缓存
            tTVPOGLRenderMethod_Script::ClearCache();

            // 2. 重新编译所有渲染方法
            for (auto it : AllMethods) {
                tTVPOGLRenderMethod* method =
                    static_cast<tTVPOGLRenderMethod*>(it.second);
                method->Rebuild();  // 重新编译着色器 + 链接程序
            }
        });
cocos2d::Director::getInstance()
    ->getEventDispatcher()
    ->addEventListenerWithFixedPriority(listener, 1);
```

**常见错误1：上下文丢失后使用旧的 GL 对象名称**

```cpp
// ❌ 错误：保存了 GL 名称但未监听上下文重建
GLuint myTexture;
glGenTextures(1, &myTexture);
// ... 应用进入后台 → 上下文丢失 → 上下文重建
glBindTexture(GL_TEXTURE_2D, myTexture);  // 旧名称无效！

// ✅ 正确：注册重建回调
void OnContextRecreated() {
    glGenTextures(1, &myTexture);    // 重新创建
    UploadTextureData(myTexture);    // 重新上传数据
}
```

**常见错误2：混合方程不匹配导致渲染异常**

```cpp
// ❌ 错误：只设置了 RGB 混合因子，忽略了 Alpha 通道
glBlendFunc(GL_SRC_ALPHA, GL_ONE_MINUS_SRC_ALPHA);
// Alpha 通道也被同样处理，可能导致累积 alpha 错误

// ✅ 正确：使用 glBlendFuncSeparate 分别控制 RGB 和 Alpha
glBlendFuncSeparate(
    GL_SRC_ALPHA, GL_ONE_MINUS_SRC_ALPHA,  // RGB 混合
    GL_ZERO, GL_ONE);                       // Alpha 保持目标值
```

**常见错误3：性能分析时忽略 GPU 流水线延迟**

```cpp
// ❌ 错误：用 CPU 计时评估 GPU 渲染性能
auto start = std::chrono::high_resolution_clock::now();
glDrawArrays(GL_TRIANGLES, 0, 6);
auto end = std::chrono::high_resolution_clock::now();
// 这只测量了 CPU 提交命令的时间，不是 GPU 执行时间

// ✅ 正确：使用 GL_TIME_ELAPSED 查询或 glFinish
glFinish();  // 等待 GPU 完成所有命令
auto start = std::chrono::high_resolution_clock::now();
glDrawArrays(GL_TRIANGLES, 0, 6);
glFinish();  // 等待本次绘制完成
auto end = std::chrono::high_resolution_clock::now();
```

// 模拟 tTVPOGLRenderMethod_Script::init 的着色器拼接过程
#include <cstdio>
#include <string>

std::string GenerateFragShader(const char* body, int nTextures,
                                const char* prefix = nullptr)
{
    std::string src;
    src += "precision mediump float;\n";
    if (prefix) { src += prefix; src += "\n"; }

    for (int i = 0; i < nTextures; ++i) {
        src += "uniform sampler2D tex" + std::to_string(i) + ";\n";
    }
    for (int i = 0; i < nTextures; ++i) {
        src += "varying vec2 v_texCoord" + std::to_string(i) + ";\n";
    }
    src += body;
    return src;
}

std::string GenerateVertShader(int nTextures) {
    std::string src;
    src += "attribute vec2 a_position;\n";
    for (int i = 0; i < nTextures; ++i) {
        src += "attribute vec2 a_texCoord" + std::to_string(i) + ";\n";
        src += "varying vec2 v_texCoord" + std::to_string(i) + ";\n";
    }
    src += "void main() {\n";
    src += "    gl_Position = vec4(a_position, 0.0, 1.0);\n";
    for (int i = 0; i < nTextures; ++i) {
        src += "    v_texCoord" + std::to_string(i) +
               " = a_texCoord" + std::to_string(i) + ";\n";
    }
    src += "}\n";
    return src;
}

int main() {
    // 生成 AlphaBlend 着色器（2 纹理版本）
    const char* blendBody =
        "void main(){\n"
        "    vec4 s = texture2D(tex0, v_texCoord0);\n"
        "    vec4 d = texture2D(tex1, v_texCoord1);\n"
        "    gl_FragColor = s + d * (1.0 - s.a);\n"
        "}";

    printf("=== 片段着色器 ===\n%s\n",
           GenerateFragShader(blendBody, 2).c_str());
    printf("=== 顶点着色器 ===\n%s\n",
           GenerateVertShader(2).c_str());
    return 0;
}

## 本节小结

1. 着色器通过字符串拼接动态生成，顶点着色器按纹理数量缓存，片段着色器由混合逻辑决定
2. `OperateRect` 使用 NDC 全屏四边形 + `glViewport` 控制绘制区域，避免了顶点坐标变换的复杂性
3. framebuffer fetch 扩展将混合操作从 2 纹理优化为 1 纹理，是移动端性能的关键
4. 源纹理与目标纹理相同时，引擎通过 `GetTempTexture2D` 拷贝到临时纹理避免读写冲突
5. Android 上下文丢失时，所有着色器和纹理需要重建，引擎通过 `EVENT_RENDERER_RECREATED` 事件自动处理

## 练习题与答案

### 练习1：着色器优化分析

**问题**：为什么顶点着色器按纹理数量缓存，而不是每个渲染方法独立编译顶点着色器？

<details>
<summary>参考答案</summary>

因为不同混合方法的顶点着色器结构完全相同——都是"传递顶点坐标和纹理坐标到片段着色器"。区别只在于纹理数量（决定了有多少组 `a_texCoordN → v_texCoordN` 传递）。

缓存的好处：
1. **减少编译次数**：KrKr2 有数十种混合方法，但纹理数量只有 0-4 种，缓存后只编译 4 个顶点着色器而不是数十个
2. **减少 GL 对象**：每个顶点着色器对象占用 GPU 内存
3. **加速链接**：同一个顶点着色器对象可以与不同的片段着色器链接，驱动可能对此做优化

这是典型的"分离变化点"设计模式——变化的是片段着色器（混合逻辑），不变的是顶点着色器（坐标传递），把不变的部分抽出来复用。

</details>

### 练习2：OperateRect 的 NDC 四边形

**问题**：`OperateRect` 中的顶点坐标固定为 `(-1,-1)` 到 `(1,1)` 的 NDC 全屏四边形，那它如何实现"只绘制目标纹理的一小部分区域"？

<details>
<summary>参考答案</summary>

通过 `glViewport` 实现。`glViewport(x, y, width, height)` 定义了 NDC 空间到帧缓冲像素空间的映射：

```
NDC (-1,-1)~(1,1)  →  像素 (x,y)~(x+width, y+height)
```

例如，要在 1024×1024 的目标纹理上只绘制 (100,200)~(300,400) 的区域：
```cpp
glViewport(100, 200, 200, 200);
glDrawArrays(GL_TRIANGLES, 0, 6);  // 四边形只覆盖 viewport 区域
```

这比修改顶点坐标更高效，因为：
1. 顶点数据是 `static const`，可以放在只读内存中
2. 每次绘制只需一次 `glViewport` 调用（比更新顶点缓冲便宜得多）
3. 不需要计算目标纹理坐标到 NDC 的变换

纹理坐标则通过 `ApplyVertex` 计算，将源矩形映射到 `[0,1]` 的纹理空间。

</details>

### 练习3：framebuffer fetch 兼容性

**问题**：在一台不支持 `GL_EXT_shader_framebuffer_fetch` 的 Android 设备上运行 KrKr2，Alpha 混合的性能瓶颈在哪里？如何缓解？

<details>
<summary>参考答案</summary>

**瓶颈**：每次 `OperateRect` 调用如果使用了需要读取目标纹理的混合方法（如 Alpha Blend with destination），都需要通过 `GetTempTexture2D` 将目标纹理拷贝到临时纹理。这意味着：

1. 每次混合操作额外产生一次 `glCopyTexSubImage2D` 或 FBO+Draw 拷贝
2. 临时纹理需要额外显存
3. 拷贝操作本身消耗 GPU 带宽

在一个有 20 个图层的场景中，传统合成需要 19 次混合，每次额外拷贝 → 19 次额外的纹理拷贝。

**缓解方法**：

1. **减少图层数量**：合并不需要独立动画的图层
2. **使用简单混合模式**：不需要读取目标的混合方法（如 `GL_SRC_ALPHA, GL_ONE_MINUS_SRC_ALPHA` 硬件混合）不触发拷贝
3. **启用纹理压缩**：减少拷贝的数据量
4. **降低分辨率**：使用 `half` 纹理策略减少像素数
5. **复用临时纹理**：引擎的 `GetTempTexture2D` 已经实现了纹理池复用

</details>
