## 回调驱动的图像加载架构

### 为什么使用回调而不是返回值？

KrKr2 的图像加载框架采用**回调驱动**（callback-driven）设计，而不是让加载函数直接返回位图对象。这种设计有三个优势：

1. **内存零拷贝**：解码器直接写入引擎分配的扫描线缓冲区，无需中间 Bitmap 对象
2. **逐行解码**：支持边解码边显示（如 PNG 隔行扫描），不必等整张图加载完
3. **格式无关**：引擎不关心解码细节，只关心尺寸和像素数据

```
图像文件 (PNG/BMP/TLG/...)
       │
       ▼
  加载函数 TVPLoad{Format}()
       │
       ├─→ SizeCallback(w, h, fmt)     ← 通知引擎图像尺寸
       │          │
       │          ▼ 引擎分配缓冲区
       │
       ├─→ ScanLineCallback(y=0)       ← 请求第 0 行缓冲区
       │          │
       │          ▼ 返回指针，解码器写入
       │
       ├─→ ScanLineCallback(y=-1)      ← 通知第 0 行写入完成
       │
       ├─→ ScanLineCallback(y=1)       ← 请求第 1 行...
       │   ...重复...
       │
       └─→ MetaInfoCallback(name, val) ← 可选：推送元数据
```

### 三个核心回调类型

所有回调类型定义在 `GraphicsLoaderIntf.h` 第 28-54 行：

```cpp
// 回调 1：尺寸通知回调
// 源码: GraphicsLoaderIntf.h 第 28-33 行
typedef int (*tTVPGraphicSizeCallback)(
    void *callbackdata,        // 引擎传入的上下文指针
    tjs_uint w,                // 图像宽度（像素）
    tjs_uint h,                // 图像高度（像素）
    tTVPGraphicPixelFormat fmt // 像素格式枚举
);
// 返回值：行间距（pitch），单位字节
// 引擎据此分配内存。必须在 ScanLineCallback 之前调用一次

// 像素格式枚举
// 源码: GraphicsLoaderIntf.h 第 22 行
enum tTVPGraphicPixelFormat {
    gpfLuminance,  // 8bit 灰度（亮度通道）
    gpfPalette,    // 8bit 调色板索引
    gpfRGB,        // 24bit RGB（无 Alpha）
    gpfRGBA        // 32bit RGBA（含 Alpha）
};
```

```cpp
// 回调 2：扫描线请求回调
// 源码: GraphicsLoaderIntf.h 第 35-44 行
typedef void *(*tTVPGraphicScanLineCallback)(
    void *callbackdata,  // 引擎上下文
    tjs_int y            // 行号（≥0）或 -1（写入完成通知）
);
// y ≥ 0: 请求第 y 行的缓冲区指针
//   返回值: 指向扫描线缓冲区的指针
//   返回 nullptr 可中止加载
// y == -1: 通知引擎上一次返回的缓冲区已写入完成
//   此时必须返回 nullptr
```

```cpp
// 回调 3：元数据推送回调
// 源码: GraphicsLoaderIntf.h 第 49-54 行
typedef void (*tTVPMetaInfoPushCallback)(
    void *callbackdata,  // 引擎上下文
    const ttstr &name,   // 元数据键名
    const ttstr &value   // 元数据值
);
// 可选回调，可能为 nullptr
// 用于推送偏移量、分辨率等元信息
// 如 PNG 的 oFFs chunk: name="offs_x", value="10"
```

### 加载模式枚举

加载函数还接收一个模式参数，决定输出像素格式：

```cpp
// 源码: GraphicsLoaderIntf.h 第 56-61 行
enum tTVPGraphicLoadMode {
    glmNormal,      // 标准模式：输出 32bit ARGB
    glmPalettized,  // 调色板模式：输出 8bit 索引（省份图使用）
    glmGrayscale    // 灰度模式：输出 8bit 灰度
};
```

这三种模式的使用场景：

| 模式 | 输出位深 | 使用场景 |
|------|----------|----------|
| `glmNormal` | 32bit BGRA | 普通图像显示、角色立绘、背景 |
| `glmPalettized` | 8bit 索引 | 省份图（province image）用于碰撞检测 |
| `glmGrayscale` | 8bit 灰度 | 遮罩图（mask _m 后缀）、rule 图 |

## 加载函数签名

每个图像格式需要实现一个符合以下签名的加载函数：

```cpp
// 源码: GraphicsLoaderIntf.h 第 63-82 行
typedef void (*tTVPGraphicLoadingHandler)(
    void *formatdata,      // 格式特定数据（注册时传入）
    void *callbackdata,    // 引擎上下文（传给回调函数）
    tTVPGraphicSizeCallback sizecallback,       // 尺寸回调
    tTVPGraphicScanLineCallback scanlinecallback,// 扫描线回调
    tTVPMetaInfoPushCallback metainfopushcallback,// 元数据回调
    tTJSBinaryStream *src, // 源数据流
    tjs_int32 keyidx,      // 透明色键索引
    tTVPGraphicLoadMode mode // 加载模式
);
```

参数解释：

- **`formatdata`**：在注册时传入的自定义数据指针。大多数格式不使用，传 `nullptr`
- **`callbackdata`**：引擎内部的上下文指针，原样传给回调函数即可
- **`src`**：`tTJSBinaryStream` 流对象，提供 `ReadBuffer`、`GetPosition`、`SetPosition`、`GetSize` 等方法
- **`keyidx`**：颜色键索引。值为 -1 表示无透明色。用于 8bit 调色板图像指定哪个索引是透明色
- **`mode`**：加载模式，决定解码器应输出什么格式的像素数据

## tTVPGraphicHandlerType 结构体

### 结构体定义

`tTVPGraphicHandlerType` 是注册到引擎的核心数据结构，封装了一个格式的所有处理函数。定义在 `GraphicsLoaderIntf.h` 第 371-453 行：

```cpp
// 源码: GraphicsLoaderIntf.h 第 371-453 行
struct tTVPGraphicHandlerType {
    bool IsPlugin;        // 是否为插件注册（影响函数指针类型）
    ttstr Extension;      // 扩展名（含点号，如 ".png"）

    // 加载函数（联合体区分内部/插件版本）
    union {
        tTVPGraphicLoadingHandler LoadHandler;
        tTVPGraphicLoadingHandlerForPlugin LoadHandlerPlugin;
    };

    // 头部读取函数
    union {
        tTVPGraphicHeaderLoadingHandler HeaderHandler;
        tTVPGraphicHeaderLoadingHandlerForPlugin HeaderHandlerPlugin;
    };

    // 保存函数
    union {
        tTVPGraphicSaveHandler SaveHandler;
        tTVPGraphicSaveHandlerForPlugin SaveHandlerPlugin;
    };

    // 保存格式检查函数
    tTVPGraphicAcceptSaveHandler AcceptHandler;

    // 格式自定义数据
    void *FormatData;

    // 内部构造函数
    tTVPGraphicHandlerType(
        const ttstr &ext,
        tTVPGraphicLoadingHandler loading,
        tTVPGraphicHeaderLoadingHandler header,
        tTVPGraphicSaveHandler save,
        tTVPGraphicAcceptSaveHandler accept,
        void *data
    );

    // 插件构造函数
    tTVPGraphicHandlerType(
        const ttstr &ext,
        tTVPGraphicLoadingHandlerForPlugin loading,
        tTVPGraphicHeaderLoadingHandlerForPlugin header,
        tTVPGraphicSaveHandlerForPlugin save,
        tTVPGraphicAcceptSaveHandler accept,
        void *data
    );

    // 统一加载入口
    void Load(void *formatdata, void *callbackdata,
              tTVPGraphicSizeCallback sizecallback,
              tTVPGraphicScanLineCallback scanlinecallback,
              tTVPMetaInfoPushCallback metainfopushcallback,
              tTJSBinaryStream *src, tjs_int32 keyidx,
              tTVPGraphicLoadMode mode);

    // 统一保存入口
    void Save(const ttstr &storagename, const ttstr &mode,
              const iTVPBaseBitmap *image, iTJSDispatch2 *meta);

    // 统一头部读取入口
    void Header(tTJSBinaryStream *src, iTJSDispatch2 **dic);
};
```

### 内部版 vs 插件版

引擎区分两种注册路径，因为插件使用 COM 风格的 `IStream` 接口而非内部的 `tTJSBinaryStream`：

```cpp
// 内部版（引擎自带格式使用）
typedef void (*tTVPGraphicLoadingHandler)(
    void *formatdata, void *callbackdata,
    tTVPGraphicSizeCallback sizecallback,
    tTVPGraphicScanLineCallback scanlinecallback,
    tTVPMetaInfoPushCallback metainfopushcallback,
    tTJSBinaryStream *src,   // ← 内部流接口
    tjs_int32 keyidx,
    tTVPGraphicLoadMode mode
);

// 插件版（第三方插件使用）
typedef void (*tTVPGraphicLoadingHandlerForPlugin)(
    void *formatdata, void *callbackdata,
    tTVPGraphicSizeCallback sizecallback,
    tTVPGraphicScanLineCallback scanlinecallback,
    tTVPMetaInfoPushCallback metainfopushcallback,
    struct IStream *src,     // ← COM IStream 接口
    tjs_int32 keyidx,
    tTVPGraphicLoadMode mode
);
```

在本项目（KrKr2 模拟器）中，我们始终使用**内部版**注册。

### TVPRegisterGraphicLoadingHandler

注册函数有两个重载版本。内部版定义在 `GraphicsLoaderIntf.h` 第 103-108 行：

### 格式查找与回退机制

当引擎需要加载图像时，通过 `TVPFindGraphicLoadHandler` 查找对应的 Handler。该函数定义在 `GraphicsLoaderIntf.cpp` 第 1575-1686 行，实现了完整的查找与回退逻辑：

```
加载请求: "background.png"
       │
       ▼
  提取扩展名: ".png"
       │
       ▼
  Hash.Find(".png")  ─→ 找到 PNG Handler ─→ 返回
       │
       ▼ (未找到)
  回退到 BMP Handler
       │
       ▼ (仍未找到)
  抛出异常: TVPUnknownGraphicFormat

加载请求: "background" (无扩展名)
       │
       ▼
  遍历所有注册扩展名
       │ 尝试 "background.png"、"background.bmp"...
       ▼
  TVPIsExistentStorage(name + ext)
       │ 第一个存在的文件 → 返回对应 Handler
       ▼ (全部不存在)
  抛出异常: 无法推测扩展名
```

关键代码（`GraphicsLoaderIntf.cpp` 第 1590-1618 行）：

```cpp
if(ext.IsEmpty()) {
    // 无扩展名：遍历注册表，尝试每个扩展名
    tTJSHashTable<ttstr, tTVPGraphicHandlerType>::tIterator i;
    for(i = TVPGraphicType.Hash.GetFirst();
        !i.IsNull(); /*i++*/)
    {
        ttstr newname = name + i.GetKey();
        if(TVPIsExistentStorage(newname)) {
            name = newname;
            break;
        }
        i++;
    }
    if(i.IsNull()) {
        // 未找到任何匹配文件
        TVPThrowExceptionMessage(...);
    }
    handler = &i.GetValue();
} else {
    // 有扩展名：直接哈希查找
    handler = TVPGraphicType.Hash.Find(ext);
    if(!handler) {
        // 未知扩展名，回退到 BMP
        static const ttstr ext_bmp(TJS_W(".bmp"));
        handler = TVPGraphicType.Hash.Find(ext_bmp);
    }
}
```

## 常见错误及解决方案

### 错误 1：SizeCallback 返回值被忽略导致像素错位

```
症状：图像显示为斜线状的彩色噪声
原因：加载函数忽略了 SizeCallback 的返回值（行间距）
```

`SizeCallback` 返回的是实际内存行间距（pitch），可能大于 `width * bytesPerPixel`（因为内存对齐）。解码器必须使用返回的 pitch 而不是自己计算：

```cpp
// 错误做法
sizecallback(callbackdata, width, height, gpfRGBA);
// 然后假设 pitch = width * 4

// 正确做法
int pitch = sizecallback(callbackdata, width, height, gpfRGBA);
// pitch 可能是 width*4 向上对齐到 16 字节
```

### 错误 2：未在写入完成后调用 ScanLineCallback(-1)

```
症状：图像显示为空白或只有部分行
原因：解码器写入扫描线后未通知引擎
```

每次写入扫描线缓冲区后，**必须**调用 `scanlinecallback(callbackdata, -1)` 通知引擎处理该行：

```cpp
for(int y = 0; y < height; y++) {
    void *line = scanlinecallback(callbackdata, y);
    if(!line) break;  // 引擎要求中止

    // 写入像素数据到 line
    decode_row(src, line, width);

    // 必须通知写入完成！
    scanlinecallback(callbackdata, -1);
}
```

### 错误 3：MetaInfoCallback 为 nullptr 时崩溃

```
症状：加载不含元数据的图像时崩溃
原因：未检查 metainfopushcallback 是否为 nullptr
```

`metainfopushcallback` 参数可能为 `nullptr`（引擎不需要元数据时），必须检查：

```cpp
// 正确做法
if(metainfopushcallback) {
    metainfopushcallback(callbackdata,
        TJS_W("offs_x"), ttstr(offset_x));
}
```

## 本节小结

- KrKr2 图像加载使用**回调驱动**架构：SizeCallback 通知尺寸、ScanLineCallback 逐行传递像素、MetaInfoCallback 推送元数据
- 每个格式通过 `tTVPGraphicHandlerType` 结构体封装 Load/Header/Save/Accept 四个函数
- 注册 API 有两个版本：内部版（`tTJSBinaryStream`）和插件版（`IStream`），本项目使用内部版
- 格式查找支持**扩展名直接匹配**和**无扩展名遍历探测**两种模式
- 未知扩展名回退到 BMP Handler，仍未找到则抛异常

## 练习题与答案

### 题目 1：为什么 ScanLineCallback 设计为 `y=-1` 表示写入完成，而不是增加一个单独的 CommitLineCallback？

<details>
<summary>查看答案</summary>

这是一个经典的 C 风格 API 设计选择，主要基于以下考虑：

1. **减少函数指针数量**：在 C 风格 API 中，每增加一个回调类型就要增加一个 typedef 和参数。复用同一个回调减少了 `tTVPGraphicLoadingHandler` 的参数数量（已经有 8 个参数了）
2. **保证配对调用**：一个回调处理请求和提交两个操作，在内部实现中可以用状态机确保每次 `y≥0` 调用后必定跟着 `y=-1` 调用
3. **简化格式插件**：插件开发者只需记住一个回调的用法。如果是两个回调，可能忘记调用 Commit

缺点是语义不够清晰，新开发者容易忘记 `y=-1` 调用。现代 API 设计可能会使用 RAII 模式（如返回一个 ScopedLine 对象，析构时自动提交）。

</details>

### 题目 3：`tTVPGraphicHandlerType` 为什么使用 `union` 区分内部版和插件版函数指针？为什么不使用继承或模板？

<details>
<summary>查看答案</summary>

三个原因：

1. **历史兼容**：KiriKiri2 原始代码是 C++98/03 风格，广泛使用 C 风格 union 和函数指针。`union` 是最自然的选择
2. **内存效率**：`union` 让两种函数指针共享同一块内存。每个 Handler 节省 24 字节（3 个指针×8 字节），在注册大量格式时有意义
3. **ABI 简单**：`union` + `bool IsPlugin` 标记的方式在 DLL 边界传递是安全的。继承涉及虚表指针，跨 DLL 有兼容问题。模板会导致代码膨胀

`Load` 方法内部根据 `IsPlugin` 标记选择调用哪个版本：

```cpp
void tTVPGraphicHandlerType::Load(...) {
    if(IsPlugin) {
        // 将 tTJSBinaryStream 包装为 IStream
        // 然后调用 LoadHandlerPlugin
        IStream *istream = WrapStream(src);
        LoadHandlerPlugin(formatdata, callbackdata,
            sizecallback, scanlinecallback,
            metainfopushcallback, istream,
            keyidx, mode);
    } else {
        // 直接调用内部版本
        LoadHandler(formatdata, callbackdata,
            sizecallback, scanlinecallback,
            metainfopushcallback, src,
            keyidx, mode);
    }
}
```

</details>
