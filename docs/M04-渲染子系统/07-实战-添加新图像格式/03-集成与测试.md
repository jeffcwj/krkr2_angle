## 7.3.1 注册 TGA 格式处理器

### 注册 API 回顾

框架通过 `TVPRegisterGraphicLoadingHandler` 注册新格式。每种格式需要提供一个 `tTVPGraphicHandlerType` 结构体，包含扩展名、加载函数、保存函数和头部解析函数。

```cpp
// 示例 1：TGA 格式注册（完整代码）
// 文件：LoadTGA.cpp 末尾追加

// 声明注册/注销函数
static tTVPGraphicHandlerType gTGAHandlerType;

void TVPRegisterTGAFormat()
{
    // 填充处理器描述结构
    gTGAHandlerType.Extension    = TJS_W(".tga");
    gTGAHandlerType.Load         = TVPLoadTGA;
    gTGAHandlerType.Save         = TVPSaveAsTGA;
    gTGAHandlerType.Header       = TVPLoadHeaderTGA;
    gTGAHandlerType.FormatData   = nullptr;  // 无额外格式数据

    // 配置保存模式名称（用于 TJS 脚本的 saveas 参数）
    gTGAHandlerType.SaveFormatName.push_back(TJS_W("tga"));
    gTGAHandlerType.SaveFormatName.push_back(TJS_W("tga32"));
    gTGAHandlerType.SaveFormatName.push_back(TJS_W("tga24"));
    gTGAHandlerType.SaveFormatName.push_back(TJS_W("tga8"));

    // 注册到全局格式列表
    TVPRegisterGraphicLoadingHandler(gTGAHandlerType);
}

void TVPUnregisterTGAFormat()
{
    TVPUnregisterGraphicLoadingHandler(gTGAHandlerType);
}
```

> **概念解释**：`SaveFormatName` 是一个字符串数组，列出本格式支持的所有保存模式名。当 TJS 脚本调用 `Layer.saveLayerImage("output", "tga32")` 时，框架会遍历所有已注册格式，查找 `SaveFormatName` 包含 `"tga32"` 的处理器，然后调用其 `Save` 函数。一个格式可以注册多个保存模式，例如 PNG 注册了 `"png"` 和 `"png24"` 两种模式。

> **概念解释**：`FormatData` 指针允许向加载/保存函数传递额外的格式配置数据。例如，如果同一个加载函数要处理多种变体格式，可以通过 `FormatData` 区分。对于 TGA 这种单一格式，设为 `nullptr` 即可。该指针会原样传递给 `TVPLoadTGA` 的 `formatdata` 参数。

### 注册调用时机

注册函数应在程序初始化阶段调用，与 PNG、TLG 等内置格式的注册同时进行：

```cpp
// 示例 2：在初始化函数中注册 TGA
// 文件：GraphicsLoaderIntf.cpp 中的初始化块
// 查找 TVPRegisterBMPFormat() 调用位置，在其附近添加：

void TVPInitGraphicFormats()
{
    // 内置格式
    TVPRegisterBMPFormat();   // BMP（硬编码的内部加载）
    TVPRegisterPNGFormat();   // PNG（LoadPNG.cpp）
    TVPRegisterTLGFormat();   // TLG（LoadTLG.cpp）
    TVPRegisterJPEGFormat();  // JPEG（LoadJPEG.cpp）

    // 新增格式
    TVPRegisterTGAFormat();   // TGA（LoadTGA.cpp）
}
```

### 格式查找流程验证

注册后，框架通过 `TVPFindGraphicLoadHandler` 查找处理器。流程如下：

```
用户加载 "bg.tga"
    ↓
TVPLoadGraphic("bg.tga", ...)
    ↓
TVPFindGraphicLoadHandler(".tga")
    ↓
哈希表查找 → 找到 gTGAHandlerType
    ↓
调用 gTGAHandlerType.Load（即 TVPLoadTGA）
    ↓
返回位图数据
```

```cpp
// 示例 3：验证格式注册的测试代码
// 在 TJS 脚本中测试：
var layer = new Layer(window, window.primaryLayer);
layer.loadImages("test_image"); // 框架自动尝试 .tga 扩展名

// 无扩展名时的查找流程：
// TVPFindGraphicLoadHandler 遍历所有已注册扩展名
// 对每个扩展名调用 TVPIsExistentStorage("test_image.tga")
// 找到存在的文件后使用对应的处理器
```

> **概念解释**：无扩展名查找（`TVPFindGraphicLoadHandler` 的 fallback 分支）是 KrKr2 的重要特性。游戏脚本中通常不写扩展名（如 `loadImages("bg")`），框架会自动在存储系统中搜索 `bg.bmp`、`bg.png`、`bg.tlg`、`bg.tga` 等所有已注册扩展名，返回第一个存在的文件。注册顺序决定了搜索优先级——后注册的格式先被搜索到。

## 7.3.2 图像缓存系统交互

KrKr2 的图像缓存系统会自动缓存已加载的图像，避免重复解码。新格式无需额外代码即可享受缓存。

### 缓存工作流

```
┌───────────────────────────────────────────────────┐
│  TVPLoadGraphic("bg.tga") 调用流程                 │
├───────────────────────────────────────────────────┤
│  ① TVPCheckImageCache("bg.tga", hash)             │
│     → 命中：直接返回缓存的位图，跳过解码           │
│     → 未命中：继续 ②                               │
│  ② TVPFindGraphicLoadHandler(".tga")              │
│     → 找到 TGA 处理器                              │
│  ③ TVPInternalLoadBitmap(TVPLoadTGA, ...)         │
│     → 解码 TGA → 生成 tTVPBitmap                  │
│  ④ TVPPushGraphicCache("bg.tga", bitmap, size)    │
│     → 存入 LRU 缓存                                │
│  ⑤ 返回位图给调用方                                │
└───────────────────────────────────────────────────┘
```

### 缓存键与命中条件

```cpp
// 示例 4：缓存键的组成
// 缓存键 = 文件名 + 加载选项（色键、加载模式等）
// 源码位置：GraphicsLoaderIntf.cpp TVPCheckImageCache()

struct tTVPGraphicCacheKey {
    ttstr Name;          // 文件路径（如 "bg.tga"）
    tjs_uint32 Hash;     // 文件名的哈希值
    tjs_int KeyIdx;      // 色键值（-1 表示无色键）
    tTVPGraphicLoadMode Mode;  // 加载模式（Normal/Grayscale/Palettized）
};

// 同一文件、不同色键 → 不同缓存条目
// 例如：
//   loadImages("bg", 0x00FF00)  → 缓存键 = ("bg.tga", hash, 0x00FF00, glmNormal)
//   loadImages("bg")            → 缓存键 = ("bg.tga", hash, -1, glmNormal)
// 两者不会互相命中
```

### 缓存大小与淘汰策略

缓存使用 LRU（最近最少使用）淘汰策略：

- **默认上限**：`TVPGraphicCacheLimit = 0`（无限制，直到内存不足）
- **可配置**：通过 `-graphiccachelimit` 命令行参数设置（单位：字节）
- **淘汰触发**：`TVPGraphicCacheTotalBytes > TVPGraphicCacheLimit` 时，从链表尾部开始淘汰
- **单条目上限**：超过 `TVPGraphicCacheLimit / 2` 的单张图像不会被缓存

```cpp
// 缓存淘汰代码片段（GraphicsLoaderIntf.cpp 1473-1522 行）
// TVPPushGraphicCache 内部逻辑：
if (imageSize > TVPGraphicCacheLimit / 2) {
    return; // 单张图太大，不缓存
}
while (TVPGraphicCacheTotalBytes + imageSize > TVPGraphicCacheLimit) {
    // 淘汰 LRU 链表末尾的条目
    ChopLast();
}
// 插入新条目到链表头部
```

## 7.3.3 遮罩图与省份图约定

KrKr2 有两种特殊的辅助图像加载约定，新格式需要正确支持它们。

### 遮罩图（Mask Image）— `_m` 后缀

遮罩图用于为不含 Alpha 通道的图像提供透明度信息。框架自动查找并加载。

```cpp
// 示例 5：遮罩图加载流程
// 当加载 "character.tga"（24 位无 Alpha）时：
// 框架检查 "character_m.tga" 是否存在
// 若存在 → 加载为灰度 → 绑定到主图的 Alpha 通道

// 源码位置：GraphicsLoaderIntf.cpp TVPInternalLoadBitmap()
// 关键调用：TVPBindMaskToMain(mainBitmap, maskBitmap)
// 遮罩图的白色(255) = 完全不透明，黑色(0) = 完全透明

// TJS 脚本验证：
var layer = new Layer(window, window.primaryLayer);
layer.loadImages("character");
// 自动加载 character.tga + character_m.tga
// 结果：layer 的像素包含完整 BGRA 数据
```

### 省份图（Province Image）— `_p` 后缀

省份图用于区域点击检测（hit testing），将图像划分为可点击区域。

```cpp
// 省份图加载流程：
// 加载 "map.tga" 时，框架检查 "map_p.tga" 是否存在
// 若存在 → 以 glmPalettized 模式加载为 8 位索引图
// 每个像素值代表一个"省份"编号（0-255）
// TJS 脚本通过 Layer.getProvinceAtPoint(x, y) 查询

// 省份图的 TGA 建议：
// - 使用灰度 TGA（imageType=3, 8bit）
// - 每个灰度值对应一个可点击区域
// - 灰度值 0 通常表示"不可点击"
```

### 对 TGA 解码器的要求

遮罩图和省份图的加载由框架自动处理，但解码器必须正确支持所有三种 `tTVPGraphicLoadMode`：

| 加载模式 | 用途 | 解码器职责 |
|:--------:|:----:|:----------:|
| `glmNormal` | 主图像 | 输出 32 位 BGRA 行数据 |
| `glmGrayscale` | 遮罩图 | 输出 8 位灰度行数据 |
| `glmPalettized` | 省份图 | 输出 8 位索引行数据 |

### 平台差异注意事项

| 差异项 | Windows | Linux | macOS | Android |
|:------:|:-------:|:-----:|:-----:|:-------:|
| 字节序 | 小端 | 小端 | 小端（x86/ARM） | 小端（ARM） |
| `#pragma pack` | MSVC 支持 | GCC 支持 | Clang 支持 | NDK Clang 支持 |
| 文件路径 | `\` 分隔 | `/` 分隔 | `/` 分隔 | `/` 分隔 |
| 存储访问 | 文件系统直读 | 文件系统直读 | 文件系统直读 | AAssetManager |

> **概念解释**：字节序（Endianness）决定多字节数据的存储顺序。TGA 使用小端序（低字节在前），而所有 KrKr2 支持的平台恰好也是小端序，因此 `ReadBuffer` 读入的 `tjs_uint16` 字段可以直接使用。如果未来移植到大端平台（如某些 PowerPC 设备），需要添加字节交换宏 `SWAP16(x)` 和 `SWAP32(x)` 处理头部和像素数据。

```python
# 使用 Python Pillow 库生成测试 TGA 文件
from PIL import Image
import struct

# 生成 4x4 纯红色 32 位 TGA
def make_test_tga(filename, w=4, h=4):
    with open(filename, 'wb') as f:
        # TGA 头部（18 字节）
        header = struct.pack('<BBBHHBHHHHBB',
            0,    # idLength
            0,    # colorMapType
            2,    # imageType (真彩色)
            0, 0, # colorMap origin, length
            0,    # colorMap depth
            0, 0, # x/y origin
            w, h, # width, height
            32,   # bitsPerPixel
            0x28  # descriptor: top-down + 8bit alpha
        )
        f.write(header)
        # 像素数据：BGRA 格式
        for y in range(h):
            for x in range(w):
                f.write(struct.pack('BBBB',
                    0x00,  # B
                    0x00,  # G
                    0xFF,  # R
                    0xFF   # A
                ))

make_test_tga('test_rgba.tga')

# 生成灰度遮罩图
def make_mask_tga(filename, w=4, h=4):
    with open(filename, 'wb') as f:
        header = struct.pack('<BBBHHBHHHHBB',
            0, 0, 3,  # imageType=3 (灰度)
            0, 0, 0,
            0, 0, w, h,
            8, 0x20    # 8bpp, top-down
        )
        f.write(header)
        for y in range(h):
            for x in range(w):
                # 左半透明，右半不透明
                f.write(struct.pack('B', 0 if x < w//2 else 255))

make_mask_tga('test_rgba_m.tga')
```

## 本节小结

- 注册新格式只需填充 `tTVPGraphicHandlerType` 并调用 `TVPRegisterGraphicLoadingHandler`
- 缓存系统自动工作——解码器无需关心缓存逻辑
- 缓存键包含文件名 + 色键 + 加载模式，同一文件不同参数产生不同缓存条目
- 遮罩图（`_m`）和省份图（`_p`）由框架自动查找加载，解码器只需正确支持三种 `glmXxx` 模式
- 四平台编译差异主要在构建系统（CMake vs Gradle）和存储访问层，TGA 解码器本身无平台差异
- 调试优先：在回调链关键点插入 `TVPAddLog`，比盲目断点更高效

### 练习 3：支持 TGA 2.0 Footer

TGA 2.0 在文件末尾有 26 字节的 Footer，包含扩展区和开发者区偏移。如何检测并解析？

<details>
<summary>参考答案</summary>

```cpp
// TGA 2.0 Footer 位于文件末尾 26 字节
struct tTGAFooter {
    tjs_uint32 extensionOffset;   // 扩展区偏移
    tjs_uint32 developerOffset;   // 开发者区偏移
    char       signature[18];     // "TRUEVISION-XFILE.\0"
};

void CheckTGA2Footer(tTJSBinaryStream* src)
{
    tjs_uint64 fileSize = src->GetSize();
    if (fileSize < 26 + 18) return; // 文件太小

    src->SetPosition(fileSize - 26);
    tTGAFooter footer;
    src->ReadBuffer(&footer, 26);

    if (memcmp(footer.signature,
               "TRUEVISION-XFILE.", 18) == 0) {
        // 这是 TGA 2.0 文件
```

## 常见错误及解决方案

### 错误 1：注册顺序导致格式覆盖

**症状**：TGA 文件被当作 BMP 解析，输出乱码。

**原因**：BMP 是内置硬编码的 fallback 格式。如果 TGA 注册失败（如函数未被调用），框架回退到 BMP 解析。

```cpp
// ❌ 错误：忘记调用注册函数
void TVPInitGraphicFormats()
{
    TVPRegisterBMPFormat();
    TVPRegisterPNGFormat();
    // 漏掉了 TVPRegisterTGAFormat()!
}

// ✅ 正确：确保注册函数被调用，并验证
void TVPInitGraphicFormats()
{
    TVPRegisterBMPFormat();
    TVPRegisterPNGFormat();
    TVPRegisterTGAFormat(); // 不要遗漏！

    #ifdef _DEBUG
    // 验证注册成功
    auto handler = TVPFindGraphicLoadHandler(TJS_W(".tga"));
    assert(handler != nullptr);
    #endif
}
```

### 错误 2：保存模式名称冲突

**症状**：调用 `saveLayerImage("out", "tga")` 时抛出"unsupported format"异常。

**原因**：`SaveFormatName` 未正确填充，或与其他格式的模式名冲突。

```cpp
// ❌ 错误：SaveFormatName 为空
gTGAHandlerType.Save = TVPSaveAsTGA;
// 忘记设置 SaveFormatName → 框架找不到保存处理器

// ✅ 正确：必须设置 SaveFormatName
gTGAHandlerType.SaveFormatName.push_back(TJS_W("tga"));
gTGAHandlerType.SaveFormatName.push_back(TJS_W("tga32"));
```

### 错误 3：缓存中残留旧版本图像

**症状**：修改 TGA 文件后重新运行游戏，显示的仍是旧图像。

**原因**：图像缓存的键不包含文件修改时间，如果文件名和加载参数相同则命中旧缓存。

```cpp
// 解决方案 1：开发阶段禁用缓存
// 命令行添加：-graphiccachelimit=0

// 解决方案 2：代码中手动清除缓存
TVPClearGraphicCache(); // 清除所有图像缓存

// 解决方案 3：给文件名添加版本号
layer.loadImages("bg_v2"); // 不同文件名 → 不同缓存键
```
