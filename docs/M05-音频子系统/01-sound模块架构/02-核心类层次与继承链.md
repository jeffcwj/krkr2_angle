## 继承体系总览

KrKr2 音频子系统的类设计遵循**分层继承**（Layered Inheritance）模式，每一层只负责一个关注点。完整继承链如下：

```
tTJSNativeInstance                          ← TJS2 原生实例基类
  └── tTJSNI_BaseSoundBuffer                ← 声音缓冲区接口层（状态机、淡入淡出、事件）
      └── tTJSNI_SoundBuffer                ← 平台定时器层（心跳调度）
          └── tTJSNI_BaseWaveSoundBuffer    ← 波形播放接口层（解码器、滤波器链、循环管理）
              └── tTJSNI_WaveSoundBuffer    ← 完整播放器实现（L1/L2 缓冲、OpenAL 输出）
```

每一层都定义在不同的文件中，职责严格分离：

| 层级 | 类名 | 文件 | 职责 |
|------|------|------|------|
| 第 0 层 | `tTJSNativeInstance` | `tjs2/tjsNative.h` | TJS2 原生对象生命周期 |
| 第 1 层 | `tTJSNI_BaseSoundBuffer` | `SoundBufferBaseIntf.h` | 状态机、音量、淡入淡出 |
| 第 2 层 | `tTJSNI_SoundBuffer` | `win32/SoundBufferBaseImpl.h` | 平台定时器注册 |
| 第 3 层 | `tTJSNI_BaseWaveSoundBuffer` | `WaveIntf.h` | 解码器接口、滤波器链 |
| 第 4 层 | `tTJSNI_WaveSoundBuffer` | `win32/WaveImpl.h` | 双层缓冲、OpenAL 输出 |

### 为什么需要这么多层？

这不是过度设计（Over-engineering），而是原版 KiriKiri 引擎的有意设计：

1. **接口与实现分离**：`Intf` 后缀的文件定义平台无关的接口，`Impl` 后缀的文件提供平台特定实现。这让原版 KiriKiri 可以针对不同 Windows 版本提供不同的音频后端（DirectSound / WASAPI）
2. **关注点分离**：状态管理（第 1 层）、定时器调度（第 2 层）、解码逻辑（第 3 层）、播放输出（第 4 层）各自独立，修改某一层不影响其他层
3. **TJS2 绑定需求**：第 0 层是 TJS2 脚本系统的要求，所有暴露给脚本的对象必须继承 `tTJSNativeInstance`

## 第 0 层：tTJSNativeInstance（TJS2 基类）

`tTJSNativeInstance` 是 TJS2 脚本引擎定义的所有原生实例的基类。任何需要暴露给 TJS2 脚本使用的 C++ 对象都必须继承它：

```cpp
// 文件：cpp/core/tjs2/tjsNative.h（简化）
class tTJSNativeInstance {
public:
    virtual tjs_error Construct(tjs_int numparams, tTJSVariant **param,
                                iTJSDispatch2 *tjs_obj);
    virtual void Invalidate();      // 析构前的清理
    virtual ~tTJSNativeInstance();
};
```

关键方法：
- `Construct()` —— TJS2 脚本中 `new WaveSoundBuffer(window)` 时调用，负责初始化。`param` 是 TJS2 传入的参数，`tjs_obj` 是 TJS2 对象的 dispatch 接口
- `Invalidate()` —— 对象被 TJS2 垃圾回收前调用，负责释放资源。在音频子系统中，这里停止播放、释放缓冲区

TJS2 脚本中使用音频的典型代码：

```javascript
// TJS2 脚本
var bgm = new WaveSoundBuffer(window);  // 调用 Construct()
bgm.open("bgm01.ogg");                 // 调用 tTJSNI_WaveSoundBuffer::Open()
bgm.volume = 80000;                    // 调用 SetVolume()
bgm.play();                            // 调用 Play()
// ... 当不再使用时，GC 触发 Invalidate()
```

```cpp
void tTJSNI_BaseSoundBuffer::SetStatus(tTVPSoundStatus s) {
    if(Status != s) {
        Status = s;
        if(Owner) {
            TVPCancelSourceEvents(Owner);  // 取消旧的未投递事件
            if(CanDeliverEvents) {
                tTJSVariant param(GetStatusString());  // "unload"/"stop"/"play"
                static ttstr eventname(TJS_W("onStatusChanged"));
                TVPPostEvent(Owner, Owner, eventname, 0,
                             TVP_EPT_IMMEDIATE, 1, &param);
            }
        }
    }
}
```

### 淡入淡出机制

淡入淡出是通过定时器心跳实现的离散逼近算法：

```cpp
// 文件：cpp/core/sound/SoundBufferBaseIntf.cpp 第 149-171 行
void tTJSNI_BaseSoundBuffer::Fade(tjs_int to, tjs_int time, tjs_int blanktime) {
    if(time <= 0 || blanktime < 0)
        TVPThrowExceptionMessage(TVPInvalidParam);

    if(InFading) StopFade(false, false);   // 停止正在进行的淡入淡出

    // 计算每步音量变化：(目标-当前) * 间隔 / 总时间
    DeltaVolume = (to - GetVolume()) * TVP_SB_BEAT_INTERVAL / time;
    TargetVolume = to;
    FadeCount = time / TVP_SB_BEAT_INTERVAL;  // 总步数
    BlankLeft = blanktime;                     // 开始前的等待时间
    InFading = true;
}
```

其中 `TVP_SB_BEAT_INTERVAL` 定义为 60（毫秒），即淡入淡出的精度约为 60ms 一步。例如 1000ms 的淡入淡出约分为 16 步（1000/60≈16）。

`TimerBeatHandler()` 每 60ms 被调用一次：

```cpp
// 文件：cpp/core/sound/SoundBufferBaseIntf.cpp 第 118-146 行
void tTJSNI_BaseSoundBuffer::TimerBeatHandler() {
    if(!Owner) return;
    if(!InFading) return;

    if(BlankLeft) {
        BlankLeft -= TVP_SB_BEAT_INTERVAL;  // 等待空白时间
        if(BlankLeft < 0) BlankLeft = 0;
    } else if(FadeCount) {
        if(FadeCount == 1) {
            StopFade(true, true);  // 最后一步：精确设置目标音量
        } else {
            FadeCount--;
            tjs_int v = GetVolume();
            v += DeltaVolume;           // 逐步逼近目标
            if(v < 0) v = 0;
            if(v > 100000) v = 100000;  // 音量范围 0-100000
            SetVolume(v);
        }
    }
}
```

**音量范围说明：** KrKr2 使用 0 到 100000 的整数范围表示音量（而非 0.0-1.0 浮点数），这是 KiriKiri 引擎的约定。0 = 静音，100000 = 最大音量。

## 第 2 层：tTJSNI_SoundBuffer（平台定时器）

这一层非常薄，只负责将缓冲区注册到全局定时器调度器：

```cpp
// 文件：cpp/core/sound/win32/SoundBufferBaseImpl.h 第 20-33 行
#define TVP_SB_BEAT_INTERVAL 60  // 心跳间隔 60ms

class tTJSNI_SoundBuffer : public tTJSNI_BaseSoundBuffer {
    typedef tTJSNI_BaseSoundBuffer inherited;
public:
    tTJSNI_SoundBuffer();
    tjs_error Construct(tjs_int numparams, tTJSVariant **param,
                        iTJSDispatch2 *tjs_obj) override;
    void Invalidate() override;
};
```

实现中的核心逻辑是全局缓冲区向量和定时器管理：

```cpp
// 文件：cpp/core/sound/win32/SoundBufferBaseImpl.cpp 第 23-69 行
static class TVPTimer *TVPSoundBufferTimer = nullptr;
static std::vector<tTJSNI_SoundBuffer *> TVPSoundBufferVector;

// 定时器回调分发器——每 60ms 遍历所有缓冲区
class tTVPSoundBufferTimerDispatcher {
public:
    void Handler() {
        for(auto i = TVPSoundBufferVector.begin();
            i != TVPSoundBufferVector.end(); i++) {
            (*i)->TimerBeatHandler();  // 调用每个缓冲区的心跳
        }
        TVPWaveSoundBufferCommitSettings();  // 提交延迟设置
    }
} static TVPSoundBufferTimerDispatcher;

// 注册缓冲区——第一个缓冲区创建时启动定时器
void TVPAddSoundBuffer(tTJSNI_SoundBuffer *buf) {
    if(TVPSoundBufferVector.empty()) {
        TVPSoundBufferTimer = new TVPTimer();
        TVPSoundBufferTimer->SetInterval(TVP_SB_BEAT_INTERVAL);
        TVPSoundBufferTimer->SetOnTimerHandler(
            &TVPSoundBufferTimerDispatcher,
            &tTVPSoundBufferTimerDispatcher::Handler);
    }
    TVPSoundBufferVector.push_back(buf);
}

// 注销缓冲区——最后一个缓冲区销毁时停止定时器
void TVPRemoveSoundBuffer(tTJSNI_SoundBuffer *buf) {
    // ... 从向量中移除 ...
    if(TVPSoundBufferVector.empty()) {
        delete TVPSoundBufferTimer;  // 无缓冲区则停止定时器
    }
}
```

设计要点：
- **懒初始化**：没有音频缓冲区时不创建定时器，节省资源
- **全局调度**：所有缓冲区共享一个定时器，避免每个缓冲区各自创建定时器的开销
- **60ms 间隔**：足以满足淡入淡出的平滑度（人耳对 50ms 以下的音量变化不敏感）

### 滤波器链架构

滤波器链是一个**责任链**（Chain of Responsibility）模式的实现。数据从 `LoopManager` 出发，经过每个滤波器处理后到达最终输出：

```
LoopManager → Filter[0].Recreate() → Filter[1].Recreate() → ... → FilterOutput
     ↑                                                                  │
   解码器                                                          最终 PCM 输出
```

每个滤波器实现 `iTVPBasicWaveFilter` 接口：

```cpp
// 文件：cpp/core/sound/WaveIntf.h 第 89-100 行
class iTVPBasicWaveFilter {
public:
    virtual tTVPSampleAndLabelSource *
    Recreate(tTVPSampleAndLabelSource *source) = 0;  // 重建：设置输入源
    virtual void Clear() = 0;    // 清除：断开连接
    virtual void Update() = 0;   // 更新：每个解码周期前调用
    virtual void Reset() = 0;    // 重置：seek 后重置状态
};
```

滤波器链重建过程（每次 `open()` 新文件时调用）：

```cpp
// 文件：cpp/core/sound/WaveIntf.cpp 第 920-980 行
```

## tTVPWaveFormat 结构体

`tTVPWaveFormat` 是贯穿整个音频子系统的核心数据结构，描述 PCM 音频的格式：

```cpp
// 文件：cpp/core/sound/WaveIntf.h 第 44-58 行
struct tTVPWaveFormat {
    tjs_uint SamplesPerSec;    // 采样率，如 44100, 48000
    tjs_uint Channels;         // 声道数，如 1(单声道), 2(立体声)
    tjs_uint BitsPerSample;    // 有效位深，如 16, 24, 32
    tjs_uint BytesPerSample;   // 每采样字节数（BitsPerSample / 8）
    tjs_uint64 TotalSamples;   // 总采样数
    tjs_uint64 TotalTime;      // 总时长（毫秒）
    tjs_uint32 SpeakerConfig;  // 声道布局（Speaker Configuration）
    bool IsFloat;              // 是否为 IEEE 754 浮点格式
    bool Seekable;             // 是否支持随机定位
};
```

字段详解：

| 字段 | 类型 | 典型值 | 说明 |
|------|------|--------|------|
| `SamplesPerSec` | `tjs_uint` | 44100, 48000 | 每秒采样数（Hz）|
| `Channels` | `tjs_uint` | 1, 2, 6 | 声道数。1=单声道，2=立体声，6=5.1 环绕 |
| `BitsPerSample` | `tjs_uint` | 8, 16, 24, 32 | 每个采样的有效位数 |
| `BytesPerSample` | `tjs_uint` | 1, 2, 3, 4 | 每个采样占用的字节数。通常 = BitsPerSample/8 |
| `TotalSamples` | `tjs_uint64` | 可达 2^63 | 音频文件的总采样数 |
| `TotalTime` | `tjs_uint64` | 毫秒 | 总时长。计算方式：`TotalSamples * 1000 / SamplesPerSec` |
| `SpeakerConfig` | `tjs_uint32` | 位掩码 | 声道到扬声器的映射。使用 Windows `SPEAKER_*` 常量 |
| `IsFloat` | `bool` | true/false | true 表示 IEEE 754 32位浮点 PCM |
| `Seekable` | `bool` | true/false | 是否支持 `SetPosition()` 随机定位 |

### SpeakerConfig 位掩码

声道布局使用位掩码定义，与 Windows `WAVEFORMATEXTENSIBLE` 标准一致：

```cpp
// 文件：cpp/core/sound/WaveIntf.cpp 第 42-59 行
#define SPEAKER_FRONT_LEFT            0x1     // 前左
#define SPEAKER_FRONT_RIGHT           0x2     // 前右
```

- 立体声：`SPEAKER_FRONT_LEFT | SPEAKER_FRONT_RIGHT = 0x3`

## 常见错误与解决方案

### 错误 1：混淆 SetStatus 和 SetStatusAsync

```cpp
// 错误：在解码线程中调用同步版本
void DecodeThread::Run() {
    // ...
    buffer->SetStatus(ssStop);  // 危险！同步投递事件到主线程
}
```

**原因：** `SetStatus()` 使用 `TVP_EPT_IMMEDIATE` 立即投递事件，可能导致线程安全问题。

**解决：** 在非主线程中必须使用 `SetStatusAsync()`，它使用 `TVP_EPT_POST` 将事件异步投递到主线程事件队列：

```cpp
void DecodeThread::Run() {
    // ...
    buffer->SetStatusAsync(ssStop);  // 安全：异步投递
}
```

### 错误 2：音量值超出范围

```javascript
// TJS2 脚本
bgm.volume = 200;  // 期望 200%，但实际范围是 0-100000
```

**原因：** KrKr2 的音量范围是 0-100000，不是 0-100。200 对应极小的音量。

**解决：** 使用正确的范围。100% 音量 = 100000，50% 音量 = 50000。

### 错误 3：忘记在 Invalidate 中清理资源

如果自定义声音缓冲区子类忘记在 `Invalidate()` 中调用父类：

```cpp
// 错误
void MyBuffer::Invalidate() {
    // 只清理自己的资源，忘记调用父类
    delete myResource;
}

// 正确
void MyBuffer::Invalidate() {
    delete myResource;
    inherited::Invalidate();  // 必须调用！否则定时器不会注销
}
```

## 本节小结

- 音频子系统使用 4 层继承结构，每层职责单一：TJS2 基类 → 状态/淡入淡出 → 定时器 → 解码/滤波 → 播放输出
- `tTVPWaveFormat` 是核心数据结构，贯穿解码器、滤波器和播放器
- 音量范围 0-100000（非 0-100），淡入淡出精度 60ms
- 滤波器链末尾自动添加格式转换，确保输出为 16bit PCM
- TJS2 通过 `tTJSNC_WaveSoundBuffer` 绑定 17 个属性和 6 个方法

### 题目 1：如果游戏需要一个不支持淡入淡出但支持波形可视化的简化声音缓冲区，应该从哪一层开始继承？为什么？

<details>
<summary>查看答案</summary>

应该从 **第 2 层 `tTJSNI_SoundBuffer`** 继承，而不是从第 3 层开始。

原因：
1. 必须继承第 0-2 层，因为 TJS2 绑定（第 0 层）、状态机和淡入淡出（第 1 层）、定时器调度（第 2 层）都是所有声音缓冲区的基础功能
2. 淡入淡出功能在第 1 层，但即使不使用淡入淡出也不能跳过它，因为第 2 层直接继承第 1 层
3. 如果不需要循环管理和滤波器链（第 3 层），可以跳过第 3 层直接实现自己的播放逻辑
4. 波形可视化可以作为第 4 层的自定义实现

```cpp
class tTJSNI_SimpleVisBuffer : public tTJSNI_SoundBuffer {
    // 跳过 tTJSNI_BaseWaveSoundBuffer（不需要滤波器链）
    // 直接实现简化的播放逻辑
    tTVPWaveDecoder *Decoder;
    iTVPSoundBuffer *Output;
    float *VisBuffer;  // 波形可视化缓冲区
    // ...
};
```

注意：实际操作中，如果需要循环播放功能，还是应该从第 3 层继承，因为 `LoopManager` 的实现很复杂。

### 题目 2：写出 `tTVPWaveFormat` 初始化代码，描述一个 48kHz 立体声 16bit PCM 的 3 分钟音频文件。

<details>
<summary>查看答案</summary>

```cpp
#include <cstdint>

// 模拟 KrKr2 类型
using tjs_uint = unsigned int;
using tjs_uint32 = uint32_t;
using tjs_uint64 = uint64_t;

struct tTVPWaveFormat {
    tjs_uint SamplesPerSec;
    tjs_uint Channels;
    tjs_uint BitsPerSample;
    tjs_uint BytesPerSample;
    tjs_uint64 TotalSamples;
    tjs_uint64 TotalTime;
    tjs_uint32 SpeakerConfig;
    bool IsFloat;
    bool Seekable;
};

int main() {
    tTVPWaveFormat format;
    format.SamplesPerSec = 48000;       // 48kHz
    format.Channels = 2;                 // 立体声
    format.BitsPerSample = 16;           // 16 位有效位深
    format.BytesPerSample = 2;           // 16 / 8 = 2 字节
    format.TotalSamples = 48000 * 180;   // 3 分钟 = 180 秒 = 8,640,000 采样
    format.TotalTime = 180000;           // 180 秒 = 180,000 毫秒
    format.SpeakerConfig = 0x3;          // SPEAKER_FRONT_LEFT | SPEAKER_FRONT_RIGHT
    format.IsFloat = false;              // 整数 PCM
    format.Seekable = true;              // 支持 seek

    // 验证
    // 每帧大小 = BytesPerSample * Channels = 2 * 2 = 4 字节
    // 总大小 = TotalSamples * 4 = 8,640,000 * 4 = 34,560,000 字节 ≈ 33 MB
    // TotalTime 验证 = TotalSamples * 1000 / SamplesPerSec
    //                = 8,640,000 * 1000 / 48000 = 180,000 ms ✓
    return 0;
}
```

</details>

### 题目 3：解释为什么滤波器链末尾要自动添加 PCM 格式转换滤波器，如果去掉会有什么后果？

<details>
<summary>查看答案</summary>

**为什么需要自动添加格式转换滤波器：**

1. **解码器输出格式不统一**：不同解码器可能输出不同格式的 PCM
   - Vorbis 解码器输出 32bit Float PCM
   - FFmpeg 解码器可能输出 16/24/32bit Integer 或 Float PCM
   - RIFF WAV 解码器输出与文件一致的格式（8/16/24/32bit）

2. **用户滤波器可能改变格式**：PhaseVocoder（时间拉伸）等 DSP 滤波器内部以 Float 格式处理

3. **OpenAL 输出层期望固定格式**：最终的 `iTVPSoundBuffer`（OpenAL 后端）只接受 16bit PCM（`AL_FORMAT_MONO16` 或 `AL_FORMAT_STEREO16`）

**如果去掉会发生什么：**

1. **Vorbis 文件播放错误**：Vorbis 解码器输出 Float32，直接传给 OpenAL 会被当作 16bit PCM 解释，产生严重的噪声和失真
2. **24bit WAV 数据截断**：24bit PCM 的字节对齐与 16bit 不同，OpenAL 会读取错误的采样边界
3. **不同文件表现不同**：16bit WAV 碰巧正常，但切换到 Ogg 文件就会出问题，增加调试难度

这种"管道末端格式归一化"是音频系统的常见设计模式（类似于视频渲染管线末端的色彩空间转换）。

</details>
