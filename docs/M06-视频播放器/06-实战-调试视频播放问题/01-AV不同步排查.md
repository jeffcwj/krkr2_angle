## 6.1.1 什么是 A/V 不同步

音视频不同步（A/V desync）是视频播放器中最常见也最令人困扰的问题。
表现为：

- **音频超前**：画面滞后于声音，角色嘴型还没动就已经有声音
- **音频落后**：声音滞后于画面，画面中的动作已完成但声音延迟到达
- **渐进漂移**：开始时同步正常，随时间推移越来越不同步

在 KrKr2 的视觉小说场景中，A/V 不同步主要出现在以下场景：

1. **OP/ED 动画**：配合歌曲的开场/结束动画，时长 1.5-3 分钟
2. **过场视频**：剧情过场 CG 动画，时长 10-60 秒
3. **Seek 后恢复**：跳转到视频中间位置后重新同步

```
正常同步状态：
Audio PTS: ─────────────────────────────────▶ 时间
Video PTS: ─────────────────────────────────▶ 时间
            │完全对齐│

音频超前状态：
Audio PTS: ──────────────────────────────────▶
Video PTS:    ──────────────────────────────────▶
              │←── 偏差 ──→│

音频落后状态：
Audio PTS:    ──────────────────────────────────▶
Video PTS: ──────────────────────────────────▶
              │←── 偏差 ──→│
```

## 6.1.2 CDVDClock 主时钟机制

CDVDClock 是整个 A/V 同步系统的核心。它维护一个单调递增的时钟，
所有线程通过读取这个时钟来判断当前应该播放到哪个时间点：

```cpp
// 源码：Clock.h/cpp（关键接口）
class CDVDClock
{
public:
    // 获取当前时钟值（DVD 时间单位）
    double GetClock() {
        std::unique_lock<std::recursive_mutex> lock(m_section);

        // 当前系统时间
        int64_t now = std::chrono::duration_cast<
            std::chrono::microseconds>(
            std::chrono::steady_clock::now()
                .time_since_epoch()
        ).count();

        // 时钟值 = 上次设置点 + 已流逝的时间 * 速度系数
        double clock = m_startClock +
            (now - m_systemUsed) * m_speedFactor;

        return clock;
    }

    // 设置时钟不连续点（Seek 或流开始时调用）
    void Discontinuity(double clock) {
        std::unique_lock<std::recursive_mutex> lock(m_section);

        m_startClock = clock;
        m_systemUsed = GetSystemTime();
        m_bReset = true;
    }

    // 设置播放速度
    void SetSpeed(int speed) {
        std::unique_lock<std::recursive_mutex> lock(m_section);

        double current = GetClock();
        m_speedFactor = (double)speed / DVD_PLAYSPEED_NORMAL;
        Discontinuity(current);  // 重设基准点
    }

    // 获取时钟速度系数
    double GetSpeedFactor() {
        return m_speedFactor;
    }

private:
    double    m_startClock;    // 基准时钟值
    int64_t   m_systemUsed;    // 基准系统时间
    double    m_speedFactor;   // 速度系数（1.0 = 正常）
    bool      m_bReset;        // 时钟重置标志
    CCriticalSection m_section;
};
```

### 时钟漂移的产生原理

时钟漂移发生在系统时钟与媒体时钟之间：

```
理想状态（无漂移）：
系统时间: 0ms → 100ms → 200ms → 300ms → ...
媒体时钟: 0ms → 100ms → 200ms → 300ms → ...
偏差:      0      0       0       0

实际状态（有漂移）：
系统时间: 0ms → 100ms → 200ms → 300ms → ...
音频 PTS: 0ms →  98ms → 198ms → 296ms → ...  ← 音频解码器时间
视频 PTS: 0ms → 102ms → 201ms → 305ms → ...  ← 视频解码器时间
A-V 偏差:  0     +4ms    +3ms    +9ms         ← 逐渐累积！
```

### 调试：添加时钟监控日志

```cpp
// 在 VideoPlayerVideo.cpp 的 Process() 循环中添加：
void CVideoPlayerVideo::DebugClockDrift() {
    double clockTime = m_pClock->GetClock() / DVD_TIME_BASE;
    double videoPTS  = m_lastPTS / DVD_TIME_BASE;

               drift * 1000.0, frameCount);
    }

    // 漂移超过 100ms 时告警
    if (std::abs(drift) > 0.1) {
        printf("[WARN] Clock drift %.1fms exceeds 100ms!\n",
               drift * 1000.0);
    }
}
```

## 6.1.3 ErrorAdjust 阈值分析

KrKr2 使用固定的不对称阈值来判断何时需要校正 A/V 同步：

```cpp
// 源码：VideoPlayerVideo.cpp
// 这些阈值控制"同步追赶"的触发条件

const double ERROR_AHEAD  = 0.020;   // 20ms：音频超前阈值
const double ERROR_BEHIND = 0.027;   // 27ms：音频落后阈值

void CVideoPlayerVideo::CalcDropRequirement(double pts)
{
    // 计算视频与音频时钟的偏差
    double error = m_pClock->GetClock() / DVD_TIME_BASE - pts;

    // error > 0 : 视频落后于时钟（需要追赶）
    // error < 0 : 视频超前于时钟（需要等待）

    if (error > ERROR_BEHIND) {
        // ===== 视频严重落后 =====
        // 音频已经播放到 27ms 之后了，视频需要追赶
        m_lateframes++;

        // m_lateframes 累积越多，丢帧越激进
        if (m_lateframes > 10) {
            // 连续 10 帧落后 → 可能是系统性问题
            printf("[SYNC] Video severely behind: "
                   "%.1fms, lateframes=%d\n",
                   error * 1000.0, m_lateframes);
        }
    }
    else if (error < -ERROR_AHEAD) {
        // ===== 视频严重超前 =====
        // 视频比音频快了 20ms 以上，需要等待
        // 通过 Sleep 减速
        int waitMs = (int)(-error * 1000.0);
        waitMs = std::min(waitMs, 50);  // 最多等 50ms
        Sleep(waitMs);
    }
    else {
        // ===== 在容忍范围内 =====
        m_lateframes = 0;  // 重置计数器
    }
}
```

### 为什么阈值是不对称的？

```
←─── 超前阈值（20ms）───→ ←── 正常区间 ──→ ←─── 落后阈值（27ms）───→

-50ms  -30ms  -20ms  -10ms   0ms   10ms   20ms   27ms   40ms   50ms
  │      │      │      │      │      │      │      │      │      │
  ▼      ▼      ▼      ▼      ▼      ▼      ▼      ▼      ▼      ▼
  等待   等待   阈值   正常   完美   正常   正常   阈值   追赶   追赶
```

不对称的原因：

1. **人耳对提前更敏感**：声音比画面早 20ms 就能察觉
2. **人眼对延迟更宽容**：画面比声音晚 27ms 通常不易察觉
3. **追赶比等待代价高**：丢帧会造成画面跳跃，等待只是稍微延迟

### 常见错误场景及诊断

```cpp
// === 场景 1：持续的小幅漂移 ===
// 症状：error 值逐渐增大，从 0 → 10ms → 20ms → 30ms
// 原因：音频采样率与系统时钟的微小差异累积
// 诊断：
void DiagnoseGradualDrift(double error, int frameCount) {
    static double firstError = 0;
    if (frameCount == 1) firstError = error;

    double driftRate = (error - firstError) /
                       (frameCount / 30.0);  // 假设 30fps
    printf("[DIAG] Drift rate: %.3f ms/sec\n",
           driftRate * 1000.0);

    // 如果 drift rate > 1ms/sec，可能是采样率问题
    if (std::abs(driftRate) > 0.001) {
        printf("[DIAG] WARNING: Significant drift rate. "
               "Check audio sample rate mismatch.\n");
    }
}

// === 场景 2：Seek 后突然跳变 ===
// 症状：Seek 后 error 突然变为 ±200ms
// 原因：Seek 后时钟 Discontinuity 但解码器还在输出旧帧
// 诊断：
void DiagnoseSeekDesync(double error, bool wasSeek) {
    if (wasSeek && std::abs(error) > 0.1) {
        printf("[DIAG] Post-seek desync: %.1fms\n",
               error * 1000.0);
        printf("[DIAG] Check: Was Flush sent to both "
               "video and audio queues?\n");
        printf("[DIAG] Check: Was Clock.Discontinuity() "
               "called with correct PTS?\n");
    }
}

// === 场景 3：间歇性大幅跳动 ===
// 症状：error 大部分时间正常，偶尔突然跳到 ±100ms
// 原因：DemuxPacket 的 PTS 不连续（编码问题或容器损坏）
// 诊断：
void DiagnosePTSJump(double currentPTS, double lastPTS) {
    double gap = currentPTS - lastPTS;
    double expectedGap = 1.0 / 30.0;  // 假设 30fps

    if (std::abs(gap - expectedGap) > 0.05) {  // 50ms 偏差
        printf("[DIAG] PTS jump detected: gap=%.1fms "
               "(expected %.1fms)\n",
               gap * 1000.0, expectedGap * 1000.0);
    }
}
```

## 6.1.4 SYNC_DISCON 与 SYNC_RESAMPLE 校正策略

CDVDClock 支持两种同步校正策略：

```cpp
// 源码：Clock.h
enum SyncMode {
    SYNC_DISCON,     // 不连续校正：直接跳转时钟
    SYNC_RESAMPLE    // 重采样校正：渐进式调整时钟速率
};
```

### SYNC_DISCON（不连续校正）

当偏差超过一定阈值时，直接将时钟跳转到正确值：

```cpp
// SYNC_DISCON 的工作原理
void CDVDClock::SyncDiscontinuity(double targetPTS) {
    double currentClock = GetClock();
    double error = currentClock - targetPTS;

    if (std::abs(error) > SYNC_DISCON_THRESHOLD) {
        // 偏差太大，直接跳转
        Discontinuity(targetPTS);

        printf("[SYNC] Discontinuity correction: "
               "jumped %.1fms\n", error * 1000.0);
    }
}
```

**优点**：立即修正，不会累积偏差
**缺点**：可能导致画面/音频跳跃（glitch）

### SYNC_RESAMPLE（重采样校正）

通过微调音频的播放速率来渐进式地消除偏差：

```cpp
// SYNC_RESAMPLE 的工作原理
double CDVDClock::GetResampleAdjust() {
    double error = GetClock() - m_audioClock;

    // 将偏差转换为速率调整量
    // 偏差为正（音频落后）→ 加速音频
    // 偏差为负（音频超前）→ 减速音频
    double adjust = error * RESAMPLE_GAIN;

    // 限制调整范围：±5%
    adjust = std::clamp(adjust, -0.05, 0.05);

    return 1.0 + adjust;
    // 返回值：0.95 ~ 1.05
    // 1.0 = 正常速率
    // 1.05 = 加速 5%
    // 0.95 = 减速 5%
}
```

**优点**：平滑过渡，用户几乎感知不到
**缺点**：修正速度慢，大偏差时不够用

### 策略选择逻辑

```cpp
// KrKr2 中的同步策略选择
void CVideoPlayer::SyncCorrection(double error) {
    if (std::abs(error) > 0.5) {
        // 偏差 > 500ms → 使用 DISCON（直接跳转）
        // 大偏差通常发生在 Seek 之后
        m_clock.Discontinuity(m_audioClock);
        printf("[SYNC] Large desync (%.0fms), "
               "using DISCON\n", error * 1000.0);
    }
    else if (std::abs(error) > ERROR_BEHIND) {
        // 偏差 27-500ms → 使用 RESAMPLE（渐进调整）
        double rate = m_clock.GetResampleAdjust();
        m_audioOutput.SetRate(rate);
        printf("[SYNC] Moderate desync (%.1fms), "
               "resample rate=%.4f\n",
               error * 1000.0, rate);
    }
    else {
        // 偏差 < 27ms → 正常，无需校正
        m_audioOutput.SetRate(1.0);
    }
}
```

## 6.1.5 m_lateframes 丢帧分析

m_lateframes 是诊断 A/V 同步问题的关键指标。它记录了连续多少帧
视频落后于音频：

```cpp
// m_lateframes 的状态机
//
// 正常状态: m_lateframes = 0
//     │
//     ▼ error > ERROR_BEHIND (27ms)
// 追赶状态: m_lateframes = 1, 2, 3, ...
//     │
//     ├─ error 回到正常范围 → m_lateframes = 0 (恢复)
//     │
//     └─ m_lateframes > 阈值 → 触发丢帧 (注意: KrKr2 中实际不丢帧！)

// 监控 m_lateframes 的诊断代码
void DebugLateFrames(int m_lateframes, double error) {
    // 级别 1：偶尔落后（1-3 帧）
    if (m_lateframes >= 1 && m_lateframes <= 3) {
        printf("[SYNC] Minor: lateframes=%d error=%.1fms "
               "(probably transient)\n",
               m_lateframes, error * 1000.0);
    }

    // 级别 2：持续落后（4-10 帧）
    if (m_lateframes >= 4 && m_lateframes <= 10) {
        printf("[SYNC] Moderate: lateframes=%d error=%.1fms "
               "(possible decoder stall)\n",
               m_lateframes, error * 1000.0);
    }

    // 级别 3：严重落后（>10 帧）
    if (m_lateframes > 10) {
        printf("[SYNC] SEVERE: lateframes=%d error=%.1fms "
               "(system overloaded or clock error)\n",
               m_lateframes, error * 1000.0);
    }
}
```

### KrKr2 特殊情况：m_bAllowDrop = false

在 KrKr2 中，由于 CalcFrameRate() 被禁用，m_bAllowDrop 始终为 false。
这意味着即使 m_lateframes 累积很高，播放器也不会主动丢帧：

```cpp
// 源码：VideoPlayerVideo.cpp
if (m_bAllowDrop && m_lateframes > 3) {
    // 这段代码在 KrKr2 中永远不执行！
    // m_bAllowDrop 始终为 false
    DropFrame();  // 死代码
}

// 实际行为：
// 1. 视频落后时 m_lateframes 持续增加
// 2. 但不会丢帧，只是解码速度跟不上
// 3. 最终表现为视频卡顿或慢速播放
// 4. 直到解码器追上或 error 回到正常范围
```

> **调试技巧**：如果发现 m_lateframes 持续增加但画面不丢帧，
> 这是 KrKr2 的正常行为。问题的根因通常在 CPU 性能不足或
> 解码器配置不当，而非同步逻辑本身。

## 6.1.6 系统性排查清单

当遇到 A/V 不同步问题时，按以下清单逐步排查：

### 第一步：确认问题类型

```cpp
// 添加到 VideoPlayerVideo::Process() 循环中
void DiagnoseDesyncType(double error) {
    static double maxError = 0;
    static double minError = 0;
    static int sampleCount = 0;
    static double sumError = 0;

    maxError = std::max(maxError, error);
    minError = std::min(minError, error);
    sumError += error;
    sampleCount++;

    // 每 300 帧（约 10 秒）输出一次统计
    if (sampleCount % 300 == 0) {
        double avgError = sumError / sampleCount;
        printf("\n=== A/V Sync Report (last %d frames) ===\n",
               sampleCount);
        printf("  Average error: %.1f ms\n",
               avgError * 1000.0);
        printf("  Max ahead:     %.1f ms\n",
               -minError * 1000.0);
        printf("  Max behind:    %.1f ms\n",
               maxError * 1000.0);
        printf("  Error range:   %.1f ms\n",
               (maxError - minError) * 1000.0);

        // 判断类型
        if (avgError > 0.02) {
            printf("  Type: PERSISTENT LAG "
                   "(video consistently behind)\n");
        } else if (avgError < -0.02) {
            printf("  Type: PERSISTENT LEAD "
                   "(video consistently ahead)\n");
        } else if ((maxError - minError) > 0.1) {
            printf("  Type: JITTER "
                   "(large fluctuations)\n");
        } else {
            printf("  Type: NORMAL (within tolerance)\n");
        }
        printf("=======================================\n");
    }
}
```

### 第二步：检查时钟状态

```cpp
// 添加时钟状态快照
struct ClockSnapshot {
    double clockTime;
    double audioTime;
    double videoTime;
    double speedFactor;
    int    lateframes;
    int    videoQueueLevel;
    int    audioQueueLevel;
};

ClockSnapshot TakeClockSnapshot(CVideoPlayer* player) {
    ClockSnapshot s;
    s.clockTime  = player->m_clock.GetClock() / DVD_TIME_BASE;
    s.audioTime  = player->m_audioTime / DVD_TIME_BASE;
    s.videoTime  = player->m_videoTime / DVD_TIME_BASE;
    s.speedFactor = player->m_clock.GetSpeedFactor();
    s.lateframes = player->m_lateframes;
    s.videoQueueLevel = player->m_CurrentVideo
                               .dvdNavResult.GetLevel();
    s.audioQueueLevel = player->m_CurrentAudio
                               .dvdNavResult.GetLevel();
    return s;
}

void PrintClockSnapshot(const ClockSnapshot& s) {
    printf("[CLOCK] time=%.3fs  audio=%.3fs  video=%.3fs  "
           "speed=%.2f  late=%d  vq=%d%%  aq=%d%%\n",
           s.clockTime, s.audioTime, s.videoTime,
           s.speedFactor, s.lateframes,
           s.videoQueueLevel, s.audioQueueLevel);
}
```

### 第三步：按症状定位根因

| 症状 | 可能原因 | 检查点 | 修复方案 |
|------|---------|--------|---------|
| 持续音频超前 | 视频解码太慢 | CPU 使用率，视频编码复杂度 | 降低分辨率或使用硬件加速 |
| 持续音频落后 | 音频缓冲区underrun | 音频队列 level，音频 callback 频率 | 增大音频缓冲区 |
| Seek 后大偏差 | Flush 不完整 | GENERAL_FLUSH 是否到达两个队列 | 检查 FlushBuffers() |
| 渐进漂移 | 采样率不匹配 | 音频 sample rate，系统时钟精度 | 启用 RESAMPLE 校正 |
| 间歇性跳动 | PTS 不连续 | DemuxPacket 的 PTS 序列 | 检查容器文件完整性 |
| 暂停恢复后偏差 | 时钟未正确恢复 | SetSpeed(PAUSE) 和 SetSpeed(NORMAL) | 检查 Discontinuity 调用 |

### 跨平台排查差异

| 排查步骤 | Windows | Linux | macOS | Android |
|---------|---------|-------|-------|---------|
| CPU 使用率 | 任务管理器 | top/htop | Activity Monitor | adb shell top |
| 线程状态 | VS Thread Window | gdb info threads | Xcode Debug Navigator | Android Studio |
| 音频延迟 | WASAPI latency | ALSA/PulseAudio | CoreAudio | AudioTrack |
| 典型音频延迟 | 10-30ms | 20-50ms | 5-15ms | 50-200ms |
| 常见问题 | DirectSound 延迟 | PulseAudio 缓冲 | 通常最稳定 | **最易出现同步问题** |

> **Android 特别注意**：Android 的音频延迟（AudioTrack latency）
> 通常在 50-200ms 之间，远高于其他平台。这是 Android 平台 A/V
> 不同步的首要原因。解决方案是在时钟中预补偿音频延迟。

## 动手实践

### 实践 1：构建 A/V 同步监控仪表盘

将以上所有诊断代码整合为一个统一的监控类：

```cpp
#include <chrono>
#include <cstdio>
#include <cmath>
#include <algorithm>

class AVSyncMonitor {
public:
    AVSyncMonitor()
        : m_frameCount(0)
        , m_maxError(0), m_minError(0), m_sumError(0)
        , m_lateCount(0), m_aheadCount(0)
        , m_maxLateframes(0)
    {}

    // 每帧调用
    void OnFrame(double clockTime, double videoPTS,
                 double audioPTS, int lateframes)
    {
        m_frameCount++;

        double error = clockTime - videoPTS;
        double avDiff = videoPTS - audioPTS;

        m_maxError = std::max(m_maxError, error);
        m_minError = std::min(m_minError, error);
        m_sumError += error;
        m_maxLateframes = std::max(m_maxLateframes,
                                    lateframes);

        if (error > 0.027) m_lateCount++;
        if (error < -0.020) m_aheadCount++;

        // 每 5 秒输出报告
        if (m_frameCount % 150 == 0) {
            PrintReport();
        }

        // 严重偏差立即告警
        if (std::abs(error) > 0.2) {
            printf("[!!! ALERT !!!] Severe desync: "
                   "%.0fms (frame %d)\n",
                   error * 1000.0, m_frameCount);
        }
    }

    void PrintReport() {
        double avgError = m_sumError / m_frameCount;
        double lateRate = 100.0 * m_lateCount / m_frameCount;
        double aheadRate = 100.0 * m_aheadCount / m_frameCount;

        printf("\n┌─── A/V Sync Report ─────────────────┐\n");
        printf("│ Frames analyzed: %8d            │\n",
               m_frameCount);
        printf("│ Avg error:      %+7.1f ms           │\n",
               avgError * 1000.0);
        printf("│ Max behind:      %6.1f ms           │\n",
               m_maxError * 1000.0);
        printf("│ Max ahead:       %6.1f ms           │\n",
               -m_minError * 1000.0);
        printf("│ Late frame rate:  %5.1f%%             │\n",
               lateRate);
        printf("│ Ahead frame rate: %5.1f%%             │\n",
               aheadRate);
        printf("│ Max consecutive late: %4d           │\n",
               m_maxLateframes);
        printf("│ Verdict: %-27s │\n", GetVerdict());
        printf("└─────────────────────────────────────┘\n");
    }

private:
    const char* GetVerdict() {
        double avgError = m_sumError / m_frameCount;
        double lateRate = 100.0 * m_lateCount / m_frameCount;

        if (lateRate > 20.0)
            return "POOR - Frequent late frames";
        if (std::abs(avgError) > 0.05)
            return "BAD - Persistent drift";
        if (m_maxLateframes > 30)
            return "WARN - Severe stalls detected";
        if (lateRate > 5.0)
            return "FAIR - Occasional sync issues";
        return "GOOD - Sync within tolerance";
    }

    int    m_frameCount;
    double m_maxError, m_minError, m_sumError;
    int    m_lateCount, m_aheadCount;
    int    m_maxLateframes;
};
```

## 本节小结

- CDVDClock 通过"基准值 + 已流逝时间 × 速度系数"计算当前时钟
- ErrorAdjust 使用不对称阈值（超前 20ms / 落后 27ms），因为人耳对提前更敏感
- SYNC_DISCON 直接跳转时钟（适合大偏差），SYNC_RESAMPLE 渐进调整速率（适合小偏差）
- m_lateframes 记录连续落后帧数，但在 KrKr2 中因 m_bAllowDrop=false 不会触发丢帧
- A/V 不同步排查需要区分：持续漂移、Seek 后跳变、间歇性跳动三种类型
- Android 平台因 AudioTrack 高延迟（50-200ms）是最容易出现同步问题的平台
- 系统性排查应从症状分类开始，逐步检查时钟状态、队列填充度、PTS 连续性

## 练习题与答案

### 练习 1：阈值调整影响

**题目**：如果将 ERROR_AHEAD 从 20ms 改为 5ms，ERROR_BEHIND 从 27ms 改为 50ms，
会对播放效果产生什么影响？

**答案**：

| 参数变化 | 效果 |
|---------|------|
| ERROR_AHEAD: 20ms → 5ms | 视频超前 5ms 就开始等待，导致频繁的微小延迟。画面可能出现轻微卡顿感，因为正常的 ±5ms 抖动就会触发等待。实际上 5ms 的超前人眼根本感知不到。 |
| ERROR_BEHIND: 27ms → 50ms | 视频落后 50ms 才开始追赶。音画不同步的容忍度大幅提高，用户可能明显感知到"嘴型不对"。好处是减少了不必要的追赶操作，系统更稳定。 |

总结：ERROR_AHEAD 过小会导致过度敏感（频繁等待），ERROR_BEHIND 过大会导致
用户可感知的音画不同步。原始的 20ms/27ms 是经过 Kodi 多年实践验证的平衡值。

### 练习 2：时钟漂移计算

**题目**：音频采样率标称 44100Hz，但实际硬件以 44098Hz 输出。
播放 3 分钟后，累积的时钟漂移是多少毫秒？

**答案**：

```
采样率偏差 = (44100 - 44098) / 44100 = 0.00004535 (约 0.0045%)

3 分钟 = 180 秒

累积漂移 = 180 × 0.00004535 = 0.00816 秒 ≈ 8.16 毫秒
```

8.16ms 落在 ERROR_AHEAD (20ms) 和 ERROR_BEHIND (27ms) 的正常范围内，
所以 3 分钟以内不会触发同步校正。但如果播放一个 10 分钟的视频：

```
累积漂移 = 600 × 0.00004535 = 0.02721 秒 ≈ 27.2 毫秒
```

刚好超过 ERROR_BEHIND 阈值，触发追赶！这就是为什么长视频更容易
出现 A/V 不同步——微小的采样率偏差在长时间播放中被放大。

### 练习 3：Seek 后同步恢复时间估算

**题目**：Seek 后视频 PTS 比音频 PTS 落后 200ms。假设视频解码器
每帧比正常速度快 5ms 进行追赶（正常帧间隔 33ms，追赶时 28ms），
需要多少帧才能恢复到正常范围（error < 27ms）？

**答案**：

```
初始偏差：200ms
目标偏差：< 27ms
需要消除的偏差：200 - 27 = 173ms

每帧追赶量：33ms - 28ms = 5ms

所需帧数：173 / 5 = 34.6 → 35 帧

在 30fps 下的时间：35 / 30 ≈ 1.17 秒
```

Seek 后约 1.2 秒才能恢复同步。在这 1.2 秒内，用户会感受到
轻微的画面加速。如果使用 SYNC_DISCON 策略直接跳转，则可以
立即恢复，但会有一帧画面跳跃。
