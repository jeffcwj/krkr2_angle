# 02-解码失败分析

## 本节目标

视频播放中最常见的故障之一就是"解码失败"——画面黑屏、花屏、或者播放器直接报错退出。与音视频不同步这类"能播但不对"的问题不同，解码失败往往意味着整个解码管线在某个环节彻底中断。本节将带你深入 KR2 的解码器创建与初始化流程，掌握从编解码器查找、硬件解码器协商、到像素格式验证的完整排查方法。

学完本节后，你将能够：

1. 理解 `FactoryCodec::CreateVideoCodec()` 的编解码器工厂链——硬件优先、FFmpeg 兜底的设计
2. 掌握 `CDVDVideoCodecFFmpeg::Open()` 中 extradata 传递、`avcodec_find_decoder()` 查找、`avcodec_open2()` 初始化的关键步骤
3. 分析 `GetFormat()` 回调中硬件解码器协商与 `STATE_HW_FAILED` 降级的机制
4. 识别 `Decode()` 方法返回的各种错误码（`VC_ERROR`、`VC_BUFFER`、`VC_REOPEN`）并定位根因
5. 排查像素格式不匹配导致的花屏和 `sws_scale` 转换失败

---

## 一、编解码器工厂链：谁来负责解码？

### 1.1 FactoryCodec 的设计哲学

KR2 的视频解码器选择并非"直接用 FFmpeg"这么简单。`CFactoryCodec::CreateVideoCodec()` 实现了一条优先级链——先尝试硬件加速解码器，全部失败后才回退到纯软件 FFmpeg 解码：

```cpp
// FactoryCodec.cpp — CreateVideoCodec() 核心逻辑
CDVDVideoCodec* CFactoryCodec::CreateVideoCodec(
    CDVDStreamInfo &hint,
    CProcessInfo &processInfo,
    const CRenderInfo &info)
{
    CDVDVideoCodec* pCodec = nullptr;
    
    // ========== 阶段1: 尝试所有硬件解码器 ==========
    // Amlogic 硬解（Android TV 盒子）
    pCodec = new CDVDVideoCodecAmlogic(processInfo);
    if (pCodec->Open(hint, info)) return pCodec;
    delete pCodec;
    
    // Android MediaCodec 硬解
    pCodec = new CDVDVideoCodecAndroidMediaCodec(processInfo);
    if (pCodec->Open(hint, info)) return pCodec;
    delete pCodec;
    
    // OpenMax 硬解（嵌入式平台）
    // MMAL 硬解（树莓派）
    // ... 依次尝试 ...
    
    // ========== 阶段2: FFmpeg 软解兜底 ==========
    pCodec = new CDVDVideoCodecFFmpeg(processInfo);
    if (pCodec->Open(hint, info)) return pCodec;
    delete pCodec;
    
    return nullptr;  // 所有解码器都失败了
}
```

这个设计的关键特征：

- **逐一尝试**：每个解码器被 `new` 出来后立即调用 `Open()`，失败就 `delete` 并尝试下一个
- **硬件优先**：硬件解码器排在前面，因为它们通常能提供更好的性能和更低的功耗
- **FFmpeg 兜底**：作为最后的选择，FFmpeg 软解支持几乎所有编码格式
- **nullptr 终止**：如果连 FFmpeg 都打不开，返回 `nullptr`，上层会报错

### 1.2 工厂链的完整流程图

```
┌─────────────────────────────────────────────────┐
│         CFactoryCodec::CreateVideoCodec()       │
│                                                 │
│  CDVDStreamInfo hint (codec_id, extradata, ...)  │
└──────────────────────┬──────────────────────────┘
                       │
          ┌────────────▼────────────┐
          │  Amlogic 硬解 Open()?   │──── 成功 ──→ 返回
          └────────────┬────────────┘
                  失败 │ delete
          ┌────────────▼────────────┐
          │  MediaCodec 硬解 Open()?│──── 成功 ──→ 返回
          └────────────┬────────────┘
                  失败 │ delete
          ┌────────────▼────────────┐
          │  OpenMax 硬解 Open()?   │──── 成功 ──→ 返回
          └────────────┬────────────┘
                  失败 │ delete
          ┌────────────▼────────────┐
          │  MMAL 硬解 Open()?     │──── 成功 ──→ 返回
          └────────────┬────────────┘
                  失败 │ delete
          ┌────────────▼────────────┐
          │  FFmpeg 软解 Open()?    │──── 成功 ──→ 返回
          └────────────┬────────────┘
                  失败 │ delete
                       │
                 返回 nullptr
                 （全部失败）
```

### 1.3 调试技巧：确定哪个解码器被选中

在实际调试中，你需要知道最终使用的是哪个解码器。最直接的方法是在每个 `Open()` 调用前后加日志：

```cpp
// 调试用：在 FactoryCodec.cpp 中添加日志
CLog::Log(LOGDEBUG, "CFactoryCodec: trying FFmpeg decoder for codec_id=%d",
           hint.codec);

pCodec = new CDVDVideoCodecFFmpeg(processInfo);
if (pCodec->Open(hint, info))
{
    CLog::Log(LOGINFO, "CFactoryCodec: using FFmpeg decoder for codec_id=%d",
               hint.codec);
    return pCodec;
}
CLog::Log(LOGDEBUG, "CFactoryCodec: FFmpeg decoder failed for codec_id=%d",
           hint.codec);
delete pCodec;
```

> **跨平台说明**：
> - **Windows**：通常只有 FFmpeg 软解可用（除非安装了 DXVA2/D3D11VA 硬解扩展）
> - **Linux**：可能有 VAAPI/VDPAU 硬解，取决于 GPU 驱动
> - **macOS**：VideoToolbox 硬解可用于 H.264/HEVC
> - **Android**：MediaCodec 硬解是首选，几乎所有设备都支持

## 二、CDVDVideoCodecFFmpeg::Open() 深度解析

当工厂链走到 FFmpeg 软解时，`Open()` 方法执行以下关键步骤。任何一步失败都会导致解码器初始化失败。

### 2.1 第一步：查找解码器

```cpp
// VideoCodecFFmpeg.cpp — Open() 中查找解码器
AVCodec* pCodec = nullptr;

// 优先使用 hint 中指定的编解码器名称
if (!hint.codec_name.empty())
{
    pCodec = avcodec_find_decoder_by_name(hint.codec_name.c_str());
    if (pCodec && pCodec->id != hint.codec)
    {
        // 名称匹配但 codec_id 不匹配，这个解码器不能用
        CLog::Log(LOGDEBUG, "CFactoryCodec: codec name %s doesn't match codec_id %d",
                   hint.codec_name.c_str(), hint.codec);
        pCodec = nullptr;
    }
}

// 按 codec_id 查找（常规路径）
if (!pCodec)
    pCodec = avcodec_find_decoder(hint.codec);

if (!pCodec)
{
    CLog::Log(LOGERROR, "CDVDVideoCodecFFmpeg: Unable to find codec %d",
               hint.codec);
    return false;  // ❌ 查找失败，Open() 返回 false
}
```

**常见失败原因**：

| 错误场景 | 原因 | 解决方法 |
|---------|------|---------|
| `codec_id = 0` | `CDVDStreamInfo` 未正确解析容器格式 | 检查 demuxer 是否正确识别了视频流 |
| 找不到解码器 | FFmpeg 编译时未包含该解码器 | 确认 FFmpeg 配置包含 `--enable-decoder=xxx` |
| 名称与 ID 不匹配 | hint 中的 `codec_name` 与 `codec` 字段不一致 | 检查 `CDVDStreamInfo` 的填充逻辑 |

### 2.2 第二步：传递 Extradata

Extradata（也叫 codec-specific data 或 sequence header）是解码器初始化必需的关键数据。对于 H.264 来说就是 SPS/PPS，对于 HEVC 就是 VPS/SPS/PPS。缺少 extradata 通常会导致解码器无法正确解析第一帧：

```cpp
// VideoCodecFFmpeg.cpp — extradata 传递逻辑
m_pCodecContext = avcodec_alloc_context3(pCodec);

if (hint.extrasize > 0 && hint.extradata != nullptr)
{
    // 分配并拷贝 extradata
    m_pCodecContext->extradata_size = hint.extrasize;
    m_pCodecContext->extradata = (uint8_t*)av_mallocz(
        hint.extrasize + AV_INPUT_BUFFER_PADDING_SIZE);
    memcpy(m_pCodecContext->extradata, hint.extradata, hint.extrasize);
}
```

注意这里的 `AV_INPUT_BUFFER_PADDING_SIZE`（通常是 64 字节）——这是 FFmpeg 要求的尾部填充，防止某些 SIMD 优化的解码器越界读取。如果分配时不加这个填充，可能会导致难以复现的内存访问错误。

**Extradata 的来源链**：

```
容器文件 (MP4/MKV/AVI)
    │
    ▼ demuxer 解析
CDVDStreamInfo.extradata / extrasize
    │
    ▼ Open() 拷贝
AVCodecContext.extradata / extradata_size
    │
    ▼ avcodec_open2()
解码器内部使用（解析 SPS/PPS 等）
```

### 2.3 第三步：配置解码线程

KR2 根据 CPU 核心数自动配置解码线程数，这是一个经常被忽略但对性能有重大影响的参数：

```cpp
// VideoCodecFFmpeg.cpp — 线程数配置
int num_threads = TVPGetProcessorNum() * 3 / 2;
if (num_threads < 1)  num_threads = 1;
if (num_threads > 16) num_threads = 16;

m_pCodecContext->thread_count = num_threads;
m_pCodecContext->thread_type  = FF_THREAD_FRAME | FF_THREAD_SLICE;
```

**解读**：
- `TVPGetProcessorNum()` 返回 CPU 逻辑核心数
- 乘以 `3/2` 是因为解码线程不会 100% 利用 CPU（有 I/O 等待）
- 上限 16 防止线程过多导致调度开销
- `FF_THREAD_FRAME`：帧级并行——同时解码多个帧（需要参考帧独立）
- `FF_THREAD_SLICE`：切片级并行——单帧内的切片并行解码

| CPU 核心数 | 计算的线程数 | 实际线程数 |
|-----------|------------|-----------|
| 1 | 1 | 1 |
| 2 | 3 | 3 |
| 4 | 6 | 6 |
| 8 | 12 | 12 |
| 12 | 18 | 16 (上限) |

### 2.4 第四步：打开解码器

```cpp
// VideoCodecFFmpeg.cpp — avcodec_open2() 调用
if (avcodec_open2(m_pCodecContext, pCodec, nullptr) < 0)
{
    CLog::Log(LOGDEBUG, "CDVDVideoCodecFFmpeg: avcodec_open2 failed");
    
    // 尝试降低分辨率重试（lowres 机制）
    if (m_pCodecContext->lowres > 0)
    {
        m_pCodecContext->lowres--;
        // 重新尝试 avcodec_open2() ...
    }
    
    return false;  // ❌ 最终失败
}
```

`avcodec_open2()` 是解码器初始化的最后一步。它会：

1. 验证 `AVCodecContext` 中所有参数的合法性
2. 分配解码器内部缓冲区
3. 解析 extradata（如果有的话）
4. 初始化解码器状态机

**常见失败原因**：

```cpp
// 调试示例：打印 avcodec_open2 的详细错误
int ret = avcodec_open2(m_pCodecContext, pCodec, nullptr);
if (ret < 0)
{
    char errbuf[256];
    av_strerror(ret, errbuf, sizeof(errbuf));
    CLog::Log(LOGERROR, 
        "CDVDVideoCodecFFmpeg: avcodec_open2 failed: %s (ret=%d)\n"
        "  codec_id=%d, width=%d, height=%d, extradata_size=%d",
        errbuf, ret,
        m_pCodecContext->codec_id,
        m_pCodecContext->width,
        m_pCodecContext->height,
        m_pCodecContext->extradata_size);
}
```

## 三、GetFormat() 回调：硬件解码器协商

### 3.1 回调机制

FFmpeg 在解码器初始化或遇到格式变化时，会通过 `GetFormat()` 回调询问应用程序："你支持哪种像素格式？"。这是硬件解码器协商的核心入口：

```cpp
// VideoCodecFFmpeg.cpp — GetFormat() 回调注册
m_pCodecContext->get_format = GetFormat;
m_pCodecContext->opaque     = this;  // 传递 this 指针以便回调中访问成员
```

### 3.2 GetFormat() 内部逻辑

```cpp
// VideoCodecFFmpeg.cpp — GetFormat() 回调实现（简化）
enum AVPixelFormat CDVDVideoCodecFFmpeg::GetFormat(
    struct AVCodecContext* avctx,
    const enum AVPixelFormat* fmt)
{
    CDVDVideoCodecFFmpeg* ctx = 
        static_cast<CDVDVideoCodecFFmpeg*>(avctx->opaque);
    
    // 遍历 FFmpeg 提供的候选格式列表
    for (int i = 0; fmt[i] != AV_PIX_FMT_NONE; i++)
    {
        // ========== 尝试硬件格式 ==========
        if (fmt[i] == AV_PIX_FMT_VDPAU   ||
            fmt[i] == AV_PIX_FMT_VAAPI    ||
            fmt[i] == AV_PIX_FMT_D3D11VA_VLD ||
            fmt[i] == AV_PIX_FMT_DXVA2_VLD)
        {
            // 尝试创建对应的硬件解码器
            IHardwareDecoder* hwDecoder = 
                CFactoryCodec::CreateHWAccel(avctx, fmt[i]);
            
            if (hwDecoder)
            {
                ctx->m_pHardware = hwDecoder;
                ctx->m_iHwState = HW_STATE_OK;
                return fmt[i];  // 使用硬件格式
            }
            
            // 硬件解码器创建失败
            ctx->m_iHwState = HW_STATE_FAILED;
            CLog::Log(LOGWARNING, 
                "CDVDVideoCodecFFmpeg: HW accel for %s failed, falling back",
                av_get_pix_fmt_name(fmt[i]));
        }
    }
    
    // ========== 回退到软件格式 ==========
    // 从候选列表中选择第一个软件格式
    for (int i = 0; fmt[i] != AV_PIX_FMT_NONE; i++)
    {
        if (fmt[i] == AV_PIX_FMT_YUV420P  ||
            fmt[i] == AV_PIX_FMT_YUV420P10 ||
            fmt[i] == AV_PIX_FMT_NV12)
        {
            return fmt[i];
        }
    }
    
    return fmt[0];  // 实在没有合适的，用第一个
}
```

### 3.3 硬件解码器状态机

`m_iHwState` 跟踪硬件解码器的状态，控制降级行为：

```
     ┌──────────────┐
     │  HW_STATE_   │
     │  FIND        │ ← 初始状态：还未尝试硬解
     └──────┬───────┘
            │ GetFormat() 被调用
            ▼
   ┌────────────────────┐
   │ 尝试创建 HW 解码器 │
   └────────┬───────────┘
      成功  │        │ 失败
            ▼        ▼
   ┌────────────┐  ┌──────────────┐
   │ HW_STATE_  │  │ HW_STATE_    │
   │ OK         │  │ FAILED       │
   └──────┬─────┘  └──────┬───────┘
          │               │
     正常解码         回退 FFmpeg 软解
          │               │
   解码出错时:             │
   ┌──────▼─────┐         │
   │ HW_STATE_  │         │
   │ FAILED     │─────────┘
   └────────────┘
   触发 VC_REOPEN → 重新走 Open()
```

当硬件解码器在运行中出错（比如 GPU 驱动崩溃），`Decode()` 方法会返回 `VC_REOPEN`，上层会销毁当前解码器并重新调用 `CreateVideoCodec()`。由于 `m_iHwState` 已经被标记为 `FAILED`，下次 `GetFormat()` 会跳过硬件格式，直接使用软件解码。

## 四、Decode() 返回码与错误处理

### 4.1 返回码含义

`CDVDVideoCodecFFmpeg::Decode()` 的返回值是一个位掩码，可以组合多个状态：

```cpp
// DVDCodecs/Video/DVDVideoCodec.h — 返回码定义
#define VC_ERROR    0x00000001  // 解码出错
#define VC_BUFFER   0x00000002  // 需要更多输入数据
#define VC_PICTURE  0x00000004  // 有解码完成的图像可取
#define VC_FLUSHED  0x00000008  // 解码器已被刷新
#define VC_REOPEN   0x00000010  // 需要重新打开解码器
#define VC_EOF      0x00000020  // 输入流结束
```

### 4.2 Decode() 核心流程

```cpp
// VideoCodecFFmpeg.cpp — Decode() 核心逻辑（简化）
int CDVDVideoCodecFFmpeg::Decode(uint8_t* pData, int iSize, 
                                  double dts, double pts)
{
    // 构造输入 packet
    AVPacket avpkt;
    av_init_packet(&avpkt);
    avpkt.data = pData;
    avpkt.size = iSize;
    avpkt.dts  = (dts == DVD_NOPTS_VALUE) ? AV_NOPTS_VALUE : (int64_t)(dts / DVD_TIME_BASE * AV_TIME_BASE);
    avpkt.pts  = (pts == DVD_NOPTS_VALUE) ? AV_NOPTS_VALUE : (int64_t)(pts / DVD_TIME_BASE * AV_TIME_BASE);
    
    // 发送 packet 给解码器
    int ret = avcodec_send_packet(m_pCodecContext, &avpkt);
    if (ret == AVERROR(EAGAIN))
    {
        // 解码器内部缓冲区满，需要先取出已解码的帧
        return VC_BUFFER;
    }
    else if (ret < 0 && ret != AVERROR_EOF)
    {
        CLog::Log(LOGERROR, "CDVDVideoCodecFFmpeg: send_packet error %d", ret);
        return VC_ERROR;
    }
    
    // 尝试接收解码后的帧
    ret = avcodec_receive_frame(m_pCodecContext, m_pFrame);
    if (ret == 0)
    {
        // 成功解码一帧
        m_iLastKeyframe++;
        return VC_PICTURE | VC_BUFFER;
    }
    else if (ret == AVERROR(EAGAIN))
    {
        // 解码器需要更多数据才能输出帧
        return VC_BUFFER;
    }
    else if (ret == AVERROR_EOF)
    {
        return VC_EOF;
    }
    else
    {
        CLog::Log(LOGERROR, "CDVDVideoCodecFFmpeg: receive_frame error %d", ret);
        return VC_ERROR;
    }
}
```

### 4.3 返回码组合与上层响应

上层 `CVideoPlayerVideo::Process()` 根据返回码决定下一步操作：

```cpp
// 上层处理逻辑（VideoPlayerVideo.cpp Process() 中）
int iDecoderState = m_pVideoCodec->Decode(pData, iSize, dts, pts);

if (iDecoderState & VC_ERROR)
{
    // 解码错误：记录日志，尝试继续（跳过当前帧）
    // 连续多次 VC_ERROR 会触发视频流重置
    m_iConsecutiveErrors++;
    if (m_iConsecutiveErrors > 100)
    {
        CLog::Log(LOGERROR, "CDVDPlayerVideo: too many consecutive errors");
        // 触发播放停止或流切换
    }
}

if (iDecoderState & VC_BUFFER)
{
    // 解码器需要更多数据：从 demuxer 读取下一个 packet
    // 这是正常状态，特别是在帧级多线程模式下
    // 解码器需要缓冲几帧才能输出第一帧
}

if (iDecoderState & VC_PICTURE)
{
    // 有解码完成的画面：调用 GetPicture() 取出
    DVDVideoPicture picture;
    if (m_pVideoCodec->GetPicture(&picture))
    {
        OutputPicture(&picture);
    }
}

if (iDecoderState & VC_REOPEN)
{
    // 需要重新打开解码器（通常是硬件解码器崩溃后降级）
    // 销毁当前解码器，重新走 FactoryCodec 工厂链
    CloseCodec();
    OpenCodec();
}
```

### 4.4 常见解码错误场景排查表

| 症状 | 可能的返回码 | 根因分析 | 排查方法 |
|------|-------------|---------|---------|
| 黑屏，无画面 | 持续 `VC_BUFFER` | extradata 缺失，解码器等待 SPS/PPS | 检查 `m_pCodecContext->extradata_size` |
| 花屏、马赛克 | `VC_PICTURE` 但画面异常 | 像素格式不匹配或 YUV 排列错误 | 用 `av_get_pix_fmt_name()` 打印实际格式 |
| 播放几秒后崩溃 | `VC_ERROR` 累积 | 参考帧损坏或内存泄漏 | 检查 `avcodec_receive_frame()` 返回值 |
| 硬解突然转软解 | `VC_REOPEN` | GPU 驱动不稳定或内存不足 | 检查 `m_iHwState` 的变化 |
| 画面卡住不动 | `VC_BUFFER` 不停 | 解码器死锁或输入数据格式错误 | 检查 `avcodec_send_packet()` 是否返回 `AVERROR(EINVAL)` |

## 五、像素格式验证与转换失败

### 5.1 g_format_map 像素格式映射表

KR2 维护了一张 FFmpeg 像素格式到内部渲染格式的映射表。如果解码器输出的像素格式不在这张表中，就需要通过 `sws_scale` 进行格式转换，而转换失败则会导致花屏或黑屏：

```cpp
// CodecUtils.cpp — 像素格式映射表
static const EFormatMap g_format_map[] = {
    { AV_PIX_FMT_YUV420P,     RENDER_FMT_YUV420P    },
    { AV_PIX_FMT_YUV420P10,   RENDER_FMT_YUV420P10  },
    { AV_PIX_FMT_YUV420P16,   RENDER_FMT_YUV420P16  },
    { AV_PIX_FMT_NV12,        RENDER_FMT_NV12       },
    { AV_PIX_FMT_YUYV422,     RENDER_FMT_YUYV422    },
    { AV_PIX_FMT_UYVY422,     RENDER_FMT_UYVY422    },
    { AV_PIX_FMT_NONE,        RENDER_FMT_NONE       }  // 终止符
};
```

### 5.2 格式转换的触发条件

当解码器输出的格式不在 `g_format_map` 中时，KR2 会通过 `sws_getContext()` + `sws_scale()` 进行格式转换：

```cpp
// CodecUtils.cpp — ConvertToYUV422PackedPicture() 转换逻辑
bool CDVDCodecUtils::ConvertToYUV422PackedPicture(
    DVDVideoPicture* pPicture, 
    struct SwsContext*& pSwsContext,
    int srcWidth, int srcHeight, 
    enum AVPixelFormat srcFormat)
{
    // 创建或更新 sws 上下文
    pSwsContext = sws_getCachedContext(
        pSwsContext,
        srcWidth, srcHeight, srcFormat,          // 输入：解码器输出格式
        srcWidth, srcHeight, AV_PIX_FMT_YUYV422, // 输出：目标格式
        SWS_FAST_BILINEAR,                        // 算法：快速双线性
        nullptr, nullptr, nullptr);
    
    if (!pSwsContext)
    {
        CLog::Log(LOGERROR, 
            "CDVDCodecUtils: sws_getCachedContext failed for %s -> YUYV422",
            av_get_pix_fmt_name(srcFormat));
        return false;  // ❌ 无法创建转换上下文
    }
    
    // 执行转换
    int ret = sws_scale(
        pSwsContext,
        pPicture->data, pPicture->iLineSize,     // 输入平面
        0, srcHeight,                              // 转换行范围
        pPicture->data, pPicture->iLineSize);      // 输出平面
    
    if (ret < 0)
    {
        CLog::Log(LOGERROR, "CDVDCodecUtils: sws_scale failed (ret=%d)", ret);
        return false;  // ❌ 转换失败
    }
    
    return true;
}
```

### 5.4 常见像素格式问题

| 源格式 | 目标格式 | 问题 | 解决方案 |
|--------|---------|------|---------|
| `YUV444P` | `YUV420P` | 色度下采样导致画质损失 | 使用 `SWS_LANCZOS` 算法提升质量 |
| `YUV420P10LE` | `YUV420P` | 10-bit 到 8-bit 截断 | 确认渲染器是否支持 10-bit |
| `NV21`(Android) | `NV12` | UV 平面交错顺序不同 | 需要 U/V 字节交换 |
| `VAAPI`/`DXVA2` | 任何软件格式 | 硬件表面不能直接访问 | 通过硬件解码器的 `transfer_data_from` 下载 |

> **跨平台说明**：
> - **Windows (DXVA2/D3D11VA)**：硬解输出 `AV_PIX_FMT_DXVA2_VLD` 或 `AV_PIX_FMT_D3D11`，需要通过 D3D 表面拷贝到系统内存才能进行软件转换
> - **Linux (VAAPI/VDPAU)**：`AV_PIX_FMT_VAAPI` 表面需要 `vaDeriveImage` + `vaMapBuffer` 才能获取 YUV 数据
> - **macOS (VideoToolbox)**：`AV_PIX_FMT_VIDEOTOOLBOX` 通过 `CVPixelBufferLockBaseAddress` 访问
> - **Android (MediaCodec)**：`AV_PIX_FMT_MEDIACODEC` 表面通常直接渲染到 SurfaceTexture，不需要下载

## 六、Filter 系统与解码后处理

### 6.1 Filter 链的作用

KR2 在解码之后、渲染之前还有一个 filter 链，用于图像后处理（反交错、缩放、旋转等）。filter 链的故障也会导致画面异常：

```cpp
// VideoCodecFFmpeg.cpp — FilterOpen() 初始化 filter 链
bool CDVDVideoCodecFFmpeg::FilterOpen(
    const std::string& filters, bool scale)
{
    // 创建 filter graph
    m_pFilterGraph = avfilter_graph_alloc();
    if (!m_pFilterGraph)
    {
        CLog::Log(LOGERROR, "CDVDVideoCodecFFmpeg: Failed to alloc filter graph");
        return false;
    }
    
    // 配置 buffer source（输入）
    char args[512];
    snprintf(args, sizeof(args),
        "video_size=%dx%d:pix_fmt=%d:time_base=%d/%d:pixel_aspect=%d/%d",
        m_pCodecContext->width, m_pCodecContext->height,
        m_pCodecContext->pix_fmt,
        m_pCodecContext->time_base.num, m_pCodecContext->time_base.den,
        m_pCodecContext->sample_aspect_ratio.num,
        m_pCodecContext->sample_aspect_ratio.den);
    
    const AVFilter* buffersrc = avfilter_get_by_name("buffer");
    int ret = avfilter_graph_create_filter(
        &m_pFilterIn, buffersrc, "in", args, nullptr, m_pFilterGraph);
    
    if (ret < 0)
    {
        CLog::Log(LOGERROR, "CDVDVideoCodecFFmpeg: Failed to create buffer source: %d", ret);
        FilterClose();
        return false;
    }
    
    // 配置 buffer sink（输出）
    const AVFilter* buffersink = avfilter_get_by_name("buffersink");
    ret = avfilter_graph_create_filter(
        &m_pFilterOut, buffersink, "out", nullptr, nullptr, m_pFilterGraph);
    
    if (ret < 0)
    {
        CLog::Log(LOGERROR, "CDVDVideoCodecFFmpeg: Failed to create buffer sink: %d", ret);
        FilterClose();
        return false;
    }
    
    // 解析并插入用户指定的 filter 字符串
    // 例如 "yadif=0:-1:0" (反交错) 或 "scale=1280:720" (缩放)
    AVFilterInOut* outputs = avfilter_inout_alloc();
    AVFilterInOut* inputs  = avfilter_inout_alloc();
    
    outputs->name       = av_strdup("in");
    outputs->filter_ctx = m_pFilterIn;
    outputs->pad_idx    = 0;
    outputs->next       = nullptr;
    
    inputs->name       = av_strdup("out");
    inputs->filter_ctx = m_pFilterOut;
    inputs->pad_idx    = 0;
    inputs->next       = nullptr;
    
    ret = avfilter_graph_parse_ptr(
        m_pFilterGraph, filters.c_str(), &inputs, &outputs, nullptr);
    
    if (ret < 0)
    {
        CLog::Log(LOGERROR, "CDVDVideoCodecFFmpeg: avfilter_graph_parse_ptr "
                  "failed for '%s': %d", filters.c_str(), ret);
        avfilter_inout_free(&inputs);
        avfilter_inout_free(&outputs);
        FilterClose();
        return false;
    }
    
    ret = avfilter_graph_config(m_pFilterGraph, nullptr);
    if (ret < 0)
    {
        CLog::Log(LOGERROR, "CDVDVideoCodecFFmpeg: avfilter_graph_config failed: %d", ret);
        FilterClose();
        return false;
    }
    
    avfilter_inout_free(&inputs);
    avfilter_inout_free(&outputs);
    return true;
}
```

## 常见错误及解决方案

### 错误 1：extradata 分配时忘记 AV_INPUT_BUFFER_PADDING_SIZE 导致 SIMD 越界

**现象**：播放某些 H.264/H.265 视频时，解码器在第一帧就崩溃（SIGSEGV/Access Violation），崩溃栈位于 FFmpeg 内部的 SIMD 优化代码（如 `ff_h264_decode_seq_parameter_set`）中。

**原因**：解码器初始化时需要将容器中的 `extradata`（SPS/PPS 等参数集）拷贝给 `AVCodecContext`。FFmpeg 的 SIMD 优化代码（SSE/NEON）会以 32 字节对齐读取内存，可能越过 `extradata` 的实际长度读取后续几个字节。FFmpeg 要求 `extradata` 的分配大小必须是 `extradata_size + AV_INPUT_BUFFER_PADDING_SIZE`（目前为 64 字节），多出的空间用零填充。

**解决**：

```cpp
// ❌ 错误：只分配了实际大小，没有 padding
ctx->extradata = (uint8_t*)av_malloc(extradata_size);
memcpy(ctx->extradata, src, extradata_size);

// ✅ 正确：加上 padding 并清零
ctx->extradata = (uint8_t*)av_mallocz(
    extradata_size + AV_INPUT_BUFFER_PADDING_SIZE);
if (!ctx->extradata) {
    // 处理内存分配失败
    return false;
}
memcpy(ctx->extradata, src, extradata_size);
ctx->extradata_size = extradata_size;
// av_mallocz 已经把 padding 区域清零，无需额外 memset
```

> 💡 `AV_INPUT_BUFFER_PADDING_SIZE` 在 FFmpeg 4.x 是 64，早期版本是 32。使用宏而非硬编码数字可以保证跨版本兼容。

### 错误 2：硬件解码器 HW_STATE_FAILED 后未正确重走工厂链

**现象**：在某些 Android 设备上，硬件解码器（MediaCodec）首次初始化失败后，播放器直接报"视频无法播放"，没有尝试回退到软件解码器（libavcodec）。

**原因**：`FactoryCodec` 按优先级尝试创建解码器：硬件 → 软件。当硬件解码器返回 `HW_STATE_FAILED` 时，代码应该释放失败的解码器并继续遍历工厂链中的下一个候选。但如果错误处理代码直接 `return nullptr` 而没有 `continue`，就跳过了后续的软件解码器。

**解决**：

```cpp
CDVDVideoCodec* FactoryCodec::CreateVideoCodec(/* ... */) {
    for (auto& creator : m_codecFactories) {
        CDVDVideoCodec* codec = creator.Create(hints);
        if (!codec) continue;

        // 尝试打开解码器
        if (!codec->Open(hints, options)) {
            spdlog::warn("解码器 {} 打开失败，尝试下一个",
                         codec->GetName());
            delete codec;  // 【关键】必须释放失败的实例
            continue;      // 【关键】继续尝试下一个，不能 return
        }

        // 检查硬件状态
        if (codec->GetHWState() == HW_STATE_FAILED) {
            spdlog::warn("硬件解码器 {} 状态异常，降级到下一个",
                         codec->GetName());
            delete codec;
            continue;  // ← 关键：不是 return nullptr
        }

        return codec;  // 成功
    }
    return nullptr;  // 所有解码器都失败了
}
```

### 错误 3：GetFormat 回调中选择了解码器不支持的像素格式

**现象**：硬件解码路径初始化成功，但第一帧解码时返回 `AVERROR_INVALIDDATA` 或 `AVERROR_EXTERNAL`。日志中可能出现 `unsupported pixel format` 字样。

**原因**：FFmpeg 通过 `AVCodecContext::get_format` 回调让应用选择输出像素格式。回调传入一个候选格式列表（如 `AV_PIX_FMT_VAAPI, AV_PIX_FMT_YUV420P, AV_PIX_FMT_NONE`），应用应从中选一个自己能处理的。如果回调返回了列表中不存在的格式（比如无条件返回 `AV_PIX_FMT_CUDA`，但候选列表里没有它），解码器会按 `AV_PIX_FMT_NONE` 处理，导致后续解码全部失败。

**解决**：

```cpp
// get_format 回调的安全实现
AVPixelFormat GetFormat(AVCodecContext* ctx,
                        const AVPixelFormat* fmt) {
    // fmt 是以 AV_PIX_FMT_NONE 结尾的候选列表
    // 优先选硬件格式，但必须确认在列表中
    for (const AVPixelFormat* p = fmt; *p != AV_PIX_FMT_NONE; p++) {
        if (*p == AV_PIX_FMT_VAAPI ||
            *p == AV_PIX_FMT_MEDIACODEC ||
            *p == AV_PIX_FMT_VIDEOTOOLBOX) {
            spdlog::info("选择硬件像素格式: {}", av_get_pix_fmt_name(*p));
            return *p;
        }
    }

    // 没有硬件格式，选第一个软件格式（通常是 YUV420P）
    spdlog::info("回退到软件像素格式: {}", av_get_pix_fmt_name(fmt[0]));
    return fmt[0];  // ← 安全：列表中的第一个一定是合法的
}
```

> ⚠️ 永远不要在回调中返回一个不在 `fmt` 列表中的格式。如果需要特定格式但列表中没有，应该返回 `fmt[0]` 并在上层做像素格式转换（`sws_scale`）。

bool validateH264Extradata(const uint8_t* data, int size)

    uint8_t profile = data[1];
    printf("  Profile: %d (", profile);
    switch (profile)
    {
        case 66:  printf("Baseline");  break;
        case 77:  printf("Main");      break;
        case 100: printf("High");      break;
        case 110: printf("High 10");   break;
        case 122: printf("High 4:2:2"); break;

## 本节小结

- **编解码器工厂链** (`CFactoryCodec::CreateVideoCodec()`) 实现了硬件优先、FFmpeg 兜底的解码器选择策略。每个解码器被 `new` 后立即 `Open()`，失败就 `delete` 并尝试下一个
- **`CDVDVideoCodecFFmpeg::Open()`** 包含四个关键步骤：查找解码器 → 传递 extradata → 配置线程 → `avcodec_open2()`。任何一步失败都会导致 `Open()` 返回 false
- **Extradata** 是解码器初始化的核心数据（H.264 的 SPS/PPS、HEVC 的 VPS/SPS/PPS），缺失会导致黑屏。分配时必须加 `AV_INPUT_BUFFER_PADDING_SIZE` 填充
- **`GetFormat()` 回调** 是硬件解码器协商的入口，通过 `m_iHwState` 状态机控制降级行为
- **`Decode()` 返回码** 是位掩码组合，`VC_ERROR` 表示错误，`VC_BUFFER` 表示需要更多数据，`VC_PICTURE` 表示有帧可取，`VC_REOPEN` 触发解码器重建
- **像素格式** 通过 `g_format_map` 映射，不在映射表中的格式需要 `sws_scale` 转换，转换失败会导致花屏

## 练习题与答案

### 题目 1：解码器工厂链分析

假设在 Android 平台上，`CFactoryCodec::CreateVideoCodec()` 被调用来创建 H.264 解码器。Amlogic 硬解的 `Open()` 返回 false（设备不是 Amlogic SoC），但 MediaCodec 硬解的 `Open()` 返回 true。请回答：

1. FFmpeg 软解会被尝试吗？为什么？
2. 如果 MediaCodec 在解码过程中崩溃，会发生什么？
3. 如果我们想强制使用 FFmpeg 软解，应该怎么做？

<details>
<summary>查看答案</summary>

1. **不会**。工厂链是"第一个成功就返回"的策略。MediaCodec 的 `Open()` 返回 true 后，`CreateVideoCodec()` 直接返回 MediaCodec 解码器实例，不会继续尝试 FFmpeg。

2. **会触发 `VC_REOPEN` 流程**。MediaCodec 崩溃后，`Decode()` 返回 `VC_ERROR`，上层检测到连续错误后设置 `m_iHwState = HW_STATE_FAILED`，然后返回 `VC_REOPEN`。上层调用 `CloseCodec()` + `OpenCodec()`，重新走工厂链。由于 `HW_STATE_FAILED` 已被标记，`GetFormat()` 会跳过硬件格式，最终使用 FFmpeg 软解。

3. **方法有多种**：
   - 最简单：在 `CreateVideoCodec()` 中注释掉所有硬件解码器的尝试代码，只保留 FFmpeg
   - 更优雅：添加一个配置项（如 `video.hwdecoding=false`），在 `CreateVideoCodec()` 开头检查该配置，为 false 时直接跳到 FFmpeg
   - 运行时：设置 `CDVDStreamInfo.hint` 中的某个标志位，让硬件解码器的 `Open()` 自动拒绝

```cpp
// 方法2：配置项控制
CDVDVideoCodec* CFactoryCodec::CreateVideoCodec(
    CDVDStreamInfo &hint,
    CProcessInfo &processInfo,
    const CRenderInfo &info)
{
    bool useHwDecoding = ConfigManager::GetBool("video.hwdecoding", true);
    
    if (useHwDecoding)
    {
        // 尝试硬件解码器...
    }
    
    // FFmpeg 软解兜底
    CDVDVideoCodec* pCodec = new CDVDVideoCodecFFmpeg(processInfo);
    if (pCodec->Open(hint, info)) return pCodec;
    delete pCodec;
    
    return nullptr;
}
```

</details>

### 题目 2：Extradata 缺失排查

你在调试一个 H.264 视频播放问题，发现画面全黑但音频正常播放。日志显示 `avcodec_open2()` 成功，`Decode()` 持续返回 `VC_BUFFER` 但从不返回 `VC_PICTURE`。请分析可能的原因并给出排查步骤。

<details>
<summary>查看答案</summary>

**最可能的原因**：extradata（SPS/PPS）缺失或损坏。

**分析**：
- `avcodec_open2()` 成功说明解码器已创建，但不代表 extradata 有效（某些版本的 FFmpeg 在没有 extradata 的情况下也能打开解码器）
- 持续 `VC_BUFFER` 说明解码器在等待参考信息——没有 SPS/PPS，解码器无法解析 NAL 单元
- 音频正常说明 demuxer 工作正常，问题出在视频解码链路

**排查步骤**：

```cpp
// 步骤1: 检查 extradata 是否存在
void debugExtradata(AVCodecContext* ctx)
{
    printf("extradata_size = %d\n", ctx->extradata_size);
    if (ctx->extradata_size > 0 && ctx->extradata)
    {
        printf("extradata first 16 bytes: ");
        for (int i = 0; i < 16 && i < ctx->extradata_size; i++)
            printf("%02X ", ctx->extradata[i]);
        printf("\n");
        
        // 检查是否是有效的 AVC 配置
        if (ctx->extradata[0] == 0x01)
            printf("→ AVC Decoder Configuration Record (有效)\n");
        else if (ctx->extradata[0] == 0x00 && 
                 ctx->extradata[1] == 0x00 &&
                 ctx->extradata[2] == 0x00 &&
                 ctx->extradata[3] == 0x01)
            printf("→ Annex B 起始码格式 (有效)\n");
        else
            printf("→ ⚠️ 未识别的 extradata 格式\n");
    }
    else
    {
        printf("⚠️ extradata 为空！这是黑屏的根因\n");
        printf("→ 检查 CDVDStreamInfo 的填充逻辑\n");
        printf("→ 检查 demuxer 是否正确解析了视频流头部\n");
    }
}

// 步骤2: 检查 send_packet 的输入数据
void debugPacket(AVPacket* pkt)
{
    printf("packet: size=%d, pts=%lld, dts=%lld, flags=%d\n",
           pkt->size, pkt->pts, pkt->dts, pkt->flags);
    if (pkt->size > 4)
    {
        printf("  first 4 bytes: %02X %02X %02X %02X\n",
               pkt->data[0], pkt->data[1], pkt->data[2], pkt->data[3]);
        // H.264 IDR 帧应以 00 00 00 01 65 或 NALU 长度开头
    }
}
```

**修复方向**：
1. 如果 extradata 为空 → 检查 `CDVDStreamInfo` 的构造过程，确认 demuxer 正确设置了 `extradata` 和 `extrasize`
2. 如果 extradata 格式不正确 → 可能需要 Annex B ↔ AVCC 格式转换
3. 如果 extradata 有效但仍然黑屏 → 检查输入 packet 是否包含有效的 NAL 单元

</details>

### 题目 3：GetFormat() 降级机制

阅读以下 `GetFormat()` 的调用序列日志，分析发生了什么：

```
[DEBUG] GetFormat: candidate formats: VAAPI, YUV420P, NV12
[INFO]  GetFormat: trying HW accel for VAAPI...
[ERROR] GetFormat: VAAPI hw accel creation failed
[INFO]  GetFormat: falling back to software format
[DEBUG] GetFormat: selected YUV420P
[DEBUG] Decode: VC_PICTURE returned
[DEBUG] GetPicture: format=yuv420p, 1920x1080
```

请回答：
1. 为什么 `GetFormat` 最终选择了 `YUV420P` 而不是 `NV12`？
2. 如果这台机器的 GPU 驱动修复后，下次播放会怎样？
3. 在 `GetFormat` 选择 `YUV420P` 之后，渲染管线需要做什么额外处理？

<details>
<summary>查看答案</summary>

1. **`GetFormat()` 在软件格式候选中按顺序选择第一个匹配的**。VAAPI 是硬件格式，创建失败后被跳过。在剩余的软件格式中，`YUV420P` 排在 `NV12` 前面，所以被优先选中。代码中的匹配顺序是 `YUV420P → YUV420P10 → NV12`。

2. **GPU 驱动修复后，下次播放会使用 VAAPI 硬解**。因为 `m_iHwState` 只在当前解码器实例的生命周期内保持 `HW_STATE_FAILED`。新的播放会创建新的 `CDVDVideoCodecFFmpeg` 实例，`m_iHwState` 重置为 `HW_STATE_FIND`，`GetFormat()` 会重新尝试 VAAPI。

3. **`YUV420P` 在 `g_format_map` 中有直接映射**，所以渲染管线不需要额外的 `sws_scale` 转换。`EFormatFromPixfmt(AV_PIX_FMT_YUV420P)` 返回 `RENDER_FMT_YUV420P`，渲染器可以直接使用 Y/U/V 三个平面进行 OpenGL 纹理上传。具体来说，渲染器会创建三个纹理（Y 全分辨率、U 和 V 各半分辨率），在 fragment shader 中做 YUV→RGB 转换。

</details>
