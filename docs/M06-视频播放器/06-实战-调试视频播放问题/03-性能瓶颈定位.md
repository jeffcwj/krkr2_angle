# 03-性能瓶颈定位

## 本节目标

在前两节中，我们解决了"能不能播"和"能不能解码"的问题。但在实际场景中，视频播放器面临的更常见挑战是——**能播，但不流畅**。1080p 视频在低端设备上卡顿、4K 视频解码跟不上播放速度、高帧率视频出现周期性丢帧——这些都是性能瓶颈的表现。本节将带你深入 KR2 的性能监控与自适应调节机制，掌握从 CPU 消耗分析到智能丢帧的完整排查与优化方法。

学完本节后，你将能够：

1. 分析 `sws_scale` 像素格式转换的 CPU 开销，理解为什么"不需要转换"是最优策略
2. 掌握 `CRenderManager` 渲染队列的三级缓冲设计（`m_free` / `m_queued` / `m_discard`）及其流量控制
3. 理解 `PrepareNextRender()` 如何检测延迟帧并触发队列跳跃
4. 深入 `CalcDropRequirement()` 的自适应丢帧算法——`EOS_VERYLATE`、`EOS_BUFFER_LEVEL`、`EOS_DROPPED` 三个信号
5. 掌握 `CDropControl` 状态机如何在解码端主动降级（跳过 B 帧、简化环路滤波）

## 一、sws_scale 像素格式转换：最大的 CPU 陷阱

### 1.1 为什么格式转换是性能杀手

在整个视频播放管线中，像素格式转换（通常由 `sws_scale` 执行）是仅次于解码本身的最大 CPU 消耗点。原因很简单——它需要对每一帧的**每一个像素**进行数学运算：

```
解码输出 YUV420P10 (10-bit)
    │
    ▼ sws_scale (逐像素转换)
渲染输入 YUV420P (8-bit)
    │
    ▼ 纹理上传
GPU 渲染
```

对于 1920×1080 的视频，每帧有 `1920 × 1080 × 1.5 = 3,110,400` 字节的 YUV 数据需要处理（YUV420P 中 U 和 V 各为 Y 的 1/4）。以 30fps 计算，每秒需要处理约 **89 MB** 的像素数据。

### 1.2 KR2 中的 sws_scale 使用

```cpp
// CodecUtils.cpp — ConvertToYUV422PackedPicture()
// 这是 KR2 中最主要的格式转换路径
bool CDVDCodecUtils::ConvertToYUV422PackedPicture(
    DVDVideoPicture* pPicture,
    struct SwsContext*& context,
    int srcWidth, int srcHeight,
    enum AVPixelFormat srcFormat)
{
    // sws_getCachedContext 会缓存已创建的上下文
    // 如果参数相同，直接复用；不同则重新创建
    context = sws_getCachedContext(
        context,
        srcWidth, srcHeight, srcFormat,           // 输入参数
        srcWidth, srcHeight, AV_PIX_FMT_YUYV422,  // 输出参数
        SWS_FAST_BILINEAR,                         // 缩放算法
        nullptr, nullptr, nullptr);
    
    if (!context)
        return false;
    
    // sws_scale 是 CPU 密集型操作
    // 对每个像素执行：色彩空间转换 + 色度重采样 + 位深转换
    int ret = sws_scale(
        context,
        pPicture->data, pPicture->iLineSize,  // 输入平面指针和行宽
        0, srcHeight,                           // 转换的行范围
        pPicture->data, pPicture->iLineSize);   // 输出
    
    return (ret > 0);
}
```

### 1.3 不同转换场景的性能对比

| 转换类型 | 示例 | CPU 开销 | 说明 |
|---------|------|---------|------|
| 无转换（直通） | YUV420P → YUV420P | 零 | 理想情况，格式直接匹配 |
| 色度重采样 | YUV420P → YUV422P | 中 | 只需要对 U/V 平面插值 |
| 位深转换 | YUV420P10 → YUV420P | 中 | 每像素右移 2 位 + 截断 |
| 打包转换 | YUV420P → YUYV422 | 高 | 平面格式→交错格式，需要重排内存 |
| 全转换 | RGB24 → YUV420P | 非常高 | 每像素 3 次乘法 + 色度下采样 |
| 缩放 + 转换 | 4K YUV420P → 1080p YUV422 | 极高 | 缩放算法 + 格式转换双重开销 |

### 1.4 性能测量方法

```cpp
// perf_sws_scale.cpp — 测量 sws_scale 的实际耗时
#include <cstdio>
#include <cstdint>
#include <cstdlib>
#include <cstring>
#include <chrono>

// 模拟 sws_scale 的性能测量
// 实际项目中应使用 FFmpeg 的 sws 库
void measureConversionCost(int width, int height, int iterations)
{
    // 分配模拟的 YUV420P 缓冲区
    int ySize  = width * height;
    int uvSize = (width / 2) * (height / 2);
    int totalSize = ySize + 2 * uvSize;  // Y + U + V
    
    uint8_t* srcBuffer = (uint8_t*)malloc(totalSize);
    uint8_t* dstBuffer = (uint8_t*)malloc(width * height * 2);  // YUYV422
    
    // 填充随机数据
    for (int i = 0; i < totalSize; i++)
        srcBuffer[i] = rand() & 0xFF;
    
    auto start = std::chrono::high_resolution_clock::now();
    
    // 模拟逐像素转换（YUV420P → YUYV422）
    for (int iter = 0; iter < iterations; iter++)
    {
        for (int y = 0; y < height; y++)
        {
            for (int x = 0; x < width; x += 2)
            {
                // YUYV422: 每两个像素共享一组 UV
                int srcY0 = srcBuffer[y * width + x];
                int srcY1 = srcBuffer[y * width + x + 1];
                int srcU  = srcBuffer[ySize + (y/2) * (width/2) + x/2];
                int srcV  = srcBuffer[ySize + uvSize + (y/2) * (width/2) + x/2];
                
                int dstIdx = (y * width + x) * 2;
                dstBuffer[dstIdx + 0] = srcY0;  // Y0
                dstBuffer[dstIdx + 1] = srcU;   // U
                dstBuffer[dstIdx + 2] = srcY1;  // Y1
                dstBuffer[dstIdx + 3] = srcV;   // V
            }
        }
    }
    
    auto end = std::chrono::high_resolution_clock::now();
    double ms = std::chrono::duration<double, std::milli>(end - start).count();
    
    printf("%dx%d YUV420P→YUYV422: %.2f ms/帧 (共 %d 帧, 总耗时 %.1f ms)\n",
           width, height, ms / iterations, iterations, ms);
    printf("  数据量: %.1f MB/帧, 吞吐量: %.1f MB/s\n",
           totalSize / (1024.0 * 1024.0),
           (totalSize * iterations) / (ms / 1000.0) / (1024.0 * 1024.0));
    
    free(srcBuffer);
    free(dstBuffer);
}

int main()
{
    printf("=== sws_scale 性能模拟 ===\n\n");
    
    measureConversionCost(640, 480, 1000);     // 480p
    measureConversionCost(1280, 720, 500);     // 720p
    measureConversionCost(1920, 1080, 200);    // 1080p
    measureConversionCost(3840, 2160, 50);     // 4K
    
    return 0;
}
```

> **跨平台性能差异**：
> - **Windows**：FFmpeg 的 sws_scale 会自动使用 SSE2/AVX2 SIMD 指令，性能通常较好
> - **Linux**：与 Windows 类似，取决于编译时的 SIMD 支持
> - **macOS (ARM)**：FFmpeg 使用 NEON 指令集，Apple Silicon 上性能优异
> - **Android (ARM)**：NEON 加速可用，但低端设备的内存带宽可能成为瓶颈

## 二、CRenderManager 渲染队列：缓冲策略与流量控制

### 2.1 三级缓冲设计

`CRenderManager` 使用三个 `std::deque` 管理帧缓冲区，实现了生产者-消费者模型：

```cpp
// VideoRenderer.cpp — CRenderManager 核心成员
class CRenderManager {
    // 三级缓冲队列
    std::deque<int> m_free;      // 空闲缓冲区索引
    std::deque<int> m_queued;    // 已填充、等待渲染的缓冲区
    std::deque<int> m_discard;   // 已渲染完成、等待回收的缓冲区
    
    // 缓冲区数组
    static const int NUM_BUFFERS = 6;  // 总共 6 个缓冲区
    CRenderBuffer m_buffers[NUM_BUFFERS];
    
    // 队列配置
    int m_QueueSize;     // m_queued 的最大长度（动态调整）
    int m_QueueSkip;     // 需要跳过的帧数（延迟恢复用）
    
    // 延迟检测
    int m_lateframes;    // 连续延迟帧计数
    double m_presentpts; // 当前呈现的 PTS
};
```

### 2.2 缓冲区生命周期

```
┌──────────────────────────────────────────────────────────────┐
│                   缓冲区生命周期                               │
│                                                              │
│  ┌────────┐   FlipPage()    ┌──────────┐   Render()     ┌──────────┐
│  │ m_free │ ──────────────→ │ m_queued │ ─────────────→ │ m_discard│
│  │ (空闲) │                 │ (待渲染) │                │ (已渲染) │
│  └────┬───┘                 └──────────┘                └─────┬────┘
│       │                                                        │
│       └────────────────── ReleaseBuffer() ────────────────────┘
│            (回收到空闲队列，循环使用)
│                                                              │
│  初始状态: m_free = {0,1,2,3,4,5}, m_queued = {}, m_discard = {}  │
└──────────────────────────────────────────────────────────────┘
```

### 2.3 FlipPage()：生产者端的流量控制

```cpp
// VideoRenderer.cpp — FlipPage() 简化版
void CRenderManager::FlipPage(
    volatile std::atomic_bool& bStop,
    double pts, int source, int flags)
{
    std::unique_lock<std::mutex> lock(m_mutex);
    
    // 等待空闲缓冲区可用
    while (m_free.empty() && !bStop)
    {
        // ⚠️ 性能关键点：如果 m_free 为空，解码线程会被阻塞
        // 这意味着渲染跟不上解码速度
        m_flipped.wait(lock, std::chrono::milliseconds(50));
    }
    
    if (bStop || m_free.empty())
        return;
    
    // 从空闲队列取出一个缓冲区
    int idx = m_free.front();
    m_free.pop_front();
    
    // 将帧数据填入缓冲区（实际是在 Configure 中已经上传纹理）
    m_buffers[idx].pts = pts;
    m_buffers[idx].source = source;
    
    // 放入待渲染队列
    m_queued.push_back(idx);
    
    // 队列溢出保护：如果 m_queued 超过 m_QueueSize，丢弃最旧的帧
    while ((int)m_queued.size() > m_QueueSize)
    {
        int discardIdx = m_queued.front();
        m_queued.pop_front();
        m_discard.push_back(discardIdx);
        CLog::Log(LOGDEBUG, "CRenderManager: queue overflow, "
                  "discarding buffer %d (queued=%zu, max=%d)",
                  discardIdx, m_queued.size(), m_QueueSize);
    }
}
```

## 三、PrepareNextRender()：延迟帧检测与队列跳跃

### 3.1 核心职责

`PrepareNextRender()` 是渲染线程在每个垂直同步（VSync）周期调用的方法，它负责从 `m_queued` 中选择**应该在当前时刻显示的帧**。它的核心挑战是处理"帧来晚了"的情况：

```cpp
// VideoRenderer.cpp — PrepareNextRender() 核心逻辑（简化）
bool CRenderManager::PrepareNextRender()
{
    std::unique_lock<std::mutex> lock(m_mutex);
    
    if (m_queued.empty())
        return false;
    
    // 获取当前时钟
    double clockPts = GetPresentPts();
    
    // 查找应该显示的帧
    // 策略：跳过所有 PTS 早于当前时钟的帧（它们已经"迟到"了）
    int lateCount = 0;
    while (m_queued.size() > 1)
    {
        int frontIdx = m_queued.front();
        double framePts = m_buffers[frontIdx].pts;
        
        // 如果队列前端的帧 PTS 已经过期
        if (framePts < clockPts - m_maxLateTime)
        {
            // 丢弃这一帧
            m_queued.pop_front();
            m_discard.push_back(frontIdx);
            lateCount++;
            
            CLog::Log(LOGDEBUG, 
                "PrepareNextRender: dropping late frame PTS=%.3f "
                "(clock=%.3f, late by %.1fms)",
                framePts, clockPts, 
                (clockPts - framePts) * 1000.0);
        }
        else
        {
            break;  // 找到了还没过期的帧
        }
    }
    
    // 更新延迟帧统计
    if (lateCount > 0)
    {
        m_lateframes += lateCount;
        CLog::Log(LOGDEBUG, 
            "PrepareNextRender: dropped %d late frames "
            "(total late: %d)", lateCount, m_lateframes);
    }
    else
    {
        // 没有延迟帧，重置计数
        m_lateframes = 0;
    }
    
    // 取出要渲染的帧
    m_renderIdx = m_queued.front();
    m_queued.pop_front();
    m_presentpts = m_buffers[m_renderIdx].pts;
    
    return true;
}
```

## 四、CalcDropRequirement()：自适应丢帧算法

### 4.1 算法概述

`CVideoPlayerVideo::CalcDropRequirement()` 是 KR2 视频播放器的核心调度算法。它综合多个信号来决定是否应该丢帧，以及丢帧的程度。返回值是一个位掩码，可以包含以下标志：

```cpp
// VideoPlayerVideo.cpp — 丢帧信号定义
#define EOS_ABORT       1   // 放弃：严重延迟，需要 seek 或跳跃
#define EOS_DROPPED     2   // 当前帧被丢弃
#define EOS_VERYLATE    4   // 非常延迟：需要跳过非关键帧
#define EOS_BUFFER_LEVEL 8  // 缓冲区水位过低：需要加速解码
```

### 4.2 算法实现

```cpp
// VideoPlayerVideo.cpp — CalcDropRequirement() 核心逻辑（简化）
int CVideoPlayerVideo::CalcDropRequirement(double pts)
{
    int result = 0;
    
    // ===== 信号1: 绝对延迟检测 =====
    double iClock = m_pClock->GetClock();
    double iDelay = iClock - pts;  // 正值=延迟，负值=提前
    
    // 延迟超过 200ms → 非常延迟
    if (iDelay > 0.2)
    {
        result |= EOS_VERYLATE;
        CLog::Log(LOGDEBUG, 
            "CalcDropRequirement: VERYLATE delay=%.1fms", iDelay * 1000);
    }
    
    // 延迟超过 2 秒 → 放弃追赶，需要 seek
    if (iDelay > 2.0)
    {
        result |= EOS_ABORT;
        CLog::Log(LOGWARNING, 
            "CalcDropRequirement: ABORT delay=%.1fs", iDelay);
    }
    
    // ===== 信号2: 缓冲区水位检测 =====
    int queuedBuffers = m_renderManager.GetQueuedCount();
    int freeBuffers   = m_renderManager.GetFreeCount();
    
    // 空闲缓冲区 ≤ 1 且有排队的帧 → 缓冲区紧张
    if (freeBuffers <= 1 && queuedBuffers > 0)
    {
        result |= EOS_BUFFER_LEVEL;
        CLog::Log(LOGDEBUG, 
            "CalcDropRequirement: BUFFER_LEVEL free=%d, queued=%d",
            freeBuffers, queuedBuffers);
    }
    
    // ===== 信号3: 丢帧统计反馈 =====
    // 如果连续丢帧过多，说明解码速度远不够
    if (m_droppingStats.m_gain > 0.5)
    {
        result |= EOS_DROPPED;
    }
    
    return result;
}
```

### 4.3 CDroppingStats：丢帧增益跟踪

`CDroppingStats` 是一个统计类，用于跟踪丢帧是否"有效"——即丢帧之后延迟有没有减少：

```cpp
// VideoPlayerVideo.h — CDroppingStats 定义
class CDroppingStats {
public:
    double m_gain;          // 丢帧增益：正值=丢帧有效，负值=丢帧无效
    int    m_totalDropped;  // 总丢帧数
    int    m_totalDecoded;  // 总解码数
    double m_lastPts;       // 上一帧 PTS
    
    void AddOutputPicture(double pts)
    {
        // 计算当前帧与上一帧的时间差
        if (m_lastPts != DVD_NOPTS_VALUE)
        {
            double diff = pts - m_lastPts;
            // 如果时间差明显大于期望的帧间隔，
            // 说明中间有帧被丢掉了
            // m_gain 衡量丢帧是否在追回延迟
        }
        m_lastPts = pts;
        m_totalDecoded++;
    }
    
    void AddDroppedPicture()
    {
        m_totalDropped++;
    }
    
    double GetDropRatio() const
    {
        if (m_totalDecoded + m_totalDropped == 0) return 0;
        return (double)m_totalDropped / (m_totalDecoded + m_totalDropped);
    }
};
```

### 4.4 各信号的处理策略

| 信号 | 含义 | 解码端响应 | 渲染端响应 |
|------|------|----------|----------|
| 无信号 | 正常状态 | 正常解码所有帧 | 正常渲染 |
| `EOS_BUFFER_LEVEL` | 缓冲区紧张 | 加速解码（减少后处理） | 减小 QueueSize |
| `EOS_VERYLATE` | 严重延迟 | 跳过 B 帧解码 | 丢弃过期帧 |
| `EOS_DROPPED` | 丢帧过多 | 降低解码质量 | 调整播放速度 |
| `EOS_ABORT` | 无法追赶 | 停止解码，等待 seek | 清空队列 |
| `EOS_VERYLATE + EOS_BUFFER_LEVEL` | 双重压力 | 最大程度跳帧 + 降质 | 丢弃 + 跳跃 |

## 五、CDropControl 状态机：解码端的智能降级

### 5.1 状态机设计

`CDropControl` 是嵌入在 `CDVDVideoCodecFFmpeg` 中的帧间隔跟踪器，它通过分析解码帧的 PTS 间隔来判断是否需要启用 FFmpeg 的丢帧机制：

```cpp
// VideoCodecFFmpeg.cpp — CDropControl 类
class CDropControl {
public:
    enum State {
        INIT,   // 初始化：正在收集 PTS 样本
        VALID   // 有效：已建立帧率模型，可以做丢帧决策
    };
    
    State  m_state;
    double m_diffSum;      // PTS 差值累积和
    int    m_diffCount;    // PTS 差值样本数
    double m_avgDiff;      // 平均帧间隔
    int    m_invalidCount; // 连续异常帧计数
    
    void Reset()
    {
        m_state = INIT;
        m_diffSum = 0;
        m_diffCount = 0;
        m_avgDiff = 0;
        m_invalidCount = 0;
    }
    
    void Process(double pts, bool
    )
    {
        if (m_state == INIT)
        {
            // 收集前 10 个 PTS 样本建立帧率模型
            if (m_lastPts != DVD_NOPTS_VALUE && pts != DVD_NOPTS_VALUE)
            {
                double diff = pts - m_lastPts;
                if (diff > 0 && diff < 1.0)  // 合理范围：0~1秒
                {
                    m_diffSum += diff;
                    m_diffCount++;
                }
            }
            m_lastPts = pts;
            
            // 收集够 10 个样本后切换到 VALID 状态
            if (m_diffCount >= 10)
            {
                m_avgDiff = m_diffSum / m_diffCount;
                if (m_avgDiff > 0)
                {
                    m_state = VALID;
                    CLog::Log(LOGDEBUG, 
                        "CDropControl: INIT→VALID, avgDiff=%.3fms (%.1f fps)",
                        m_avgDiff * 1000, 1.0 / m_avgDiff);
                }
                else
                {
                    // 平均间隔 ≤ 0，重置
                    Reset();
                }
            }
        }
        else  // VALID 状态
        {
            if (m_lastPts != DVD_NOPTS_VALUE && pts != DVD_NOPTS_VALUE)
            {
                double diff = pts - m_lastPts;
                double deviation = (diff - m_avgDiff) / m_avgDiff;
                
                // 如果 PTS 偏差超过 20%（说明帧率不稳定或有跳变）
                if (deviation > 0.2 || deviation < -0.2)
                {
                    m_invalidCount++;
                    // 连续 5 个异常帧 → 重置模型
                    if (m_invalidCount >= 5)
                    {
                        CLog::Log(LOGDEBUG, 
                            "CDropControl: VALID→INIT (5 consecutive invalid frames)");
                        Reset();
                    }
                }
                else
                {
                    m_invalidCount = 0;
                    // 更新滑动平均
                    m_avgDiff = m_avgDiff * 0.9 + diff * 0.1;
                }
            }
            m_lastPts = pts;
        }
    }
    
private:
    double m_lastPts = DVD_NOPTS_VALUE;
};
```

### 5.2 SetDropState()：FFmpeg 丢帧控制

当 `CalcDropRequirement()` 返回 `EOS_VERYLATE` 时，`SetDropState()` 会通知 FFmpeg 解码器跳过不重要的帧：

```cpp
// VideoCodecFFmpeg.cpp — SetDropState()
void CDVDVideoCodecFFmpeg::SetDropState(bool drop)
{
    if (drop)
    {
        // 启用丢帧模式：
        // skip_frame = AVDISCARD_NONREF → 跳过非参考帧（B 帧）
        // skip_idct  = AVDISCARD_NONREF → 跳过非参考帧的 IDCT 变换
        // skip_loop_filter = AVDISCARD_NONREF → 跳过非参考帧的去块效应滤波
        m_pCodecContext->skip_frame       = AVDISCARD_NONREF;
        m_pCodecContext->skip_idct        = AVDISCARD_NONREF;
        m_pCodecContext->skip_loop_filter = AVDISCARD_NONREF;
        
        CLog::Log(LOGDEBUG, "CDVDVideoCodecFFmpeg: DROP mode ON "
                  "(skip B-frames and loop filter)");
    }
    else
    {
        // 恢复正常模式：解码所有帧
        m_pCodecContext->skip_frame       = AVDISCARD_DEFAULT;
        m_pCodecContext->skip_idct        = AVDISCARD_DEFAULT;
        m_pCodecContext->skip_loop_filter = AVDISCARD_DEFAULT;
        
        CLog::Log(LOGDEBUG, "CDVDVideoCodecFFmpeg: DROP mode OFF");
    }
}
```

### 5.3 丢帧级别详解

FFmpeg 提供了多级丢帧控制，KR2 使用的是 `AVDISCARD_NONREF`，但了解所有级别有助于调优：

```cpp
// FFmpeg 丢帧级别（从保守到激进）
enum AVDiscard {
    AVDISCARD_NONE     = -16,  // 不丢弃任何帧
    AVDISCARD_DEFAULT  = 0,    // 默认：由解码器决定
    AVDISCARD_NONREF   = 8,    // 丢弃非参考帧（B 帧）
    AVDISCARD_BIDIR    = 16,   // 丢弃双向预测帧
    AVDISCARD_NONINTRA = 24,   // 丢弃非 I 帧（只保留关键帧）
    AVDISCARD_NONKEY   = 32,   // 丢弃非关键帧
    AVDISCARD_ALL      = 48    // 丢弃所有帧（只做解复用）
};
```

| 级别 | 跳过的帧 | CPU 节省 | 画面影响 |
|------|---------|---------|---------|
| `AVDISCARD_NONE` | 无 | 0% | 无 |
| `AVDISCARD_NONREF` | B 帧 | ~30-50% | 帧率降低但画面连贯 |
| `AVDISCARD_BIDIR` | 双向预测帧 | ~40-60% | 帧率明显降低 |
| `AVDISCARD_NONINTRA` | P+B 帧 | ~70-85% | 只剩关键帧，画面跳跃 |
| `AVDISCARD_NONKEY` | 非关键帧 | ~90% | 极度跳跃，几乎无法观看 |

## 六、OutputPicture() 中的丢帧逻辑

### 6.1 速度调整与最小间隔

`OutputPicture()` 是解码帧输出到渲染队列之前的最后一个关卡。它会根据播放速度和当前延迟来决定是否真的输出这一帧：

```cpp
// VideoPlayerVideo.cpp — OutputPicture() 丢帧逻辑（简化）
int CVideoPlayerVideo::OutputPicture(DVDVideoPicture* pPicture)
{
    double pts = pPicture->pts;
    double iClock = m_pClock->GetClock();
    double iDelay = iClock - pts;
    
    // 获取播放速度（1.0 = 正常，2.0 = 两倍速）
    double speed = m_pClock->GetSpeed();
    
    // 计算速度调整后的最小帧间隔
    // 正常速度下 mindiff = 0（不丢帧）
    // 加速播放时 mindiff > 0（需要丢帧来匹配速度）
    double mindiff = 0;
    if (speed > 1.0)
    {
        // 两倍速时，每隔一帧丢一帧
        mindiff = (1.0 - 1.0/speed) * m_dropControl.m_avgDiff;
    }
    
    // 检查缓冲区水位
    int freeBuffers = m_renderManager.GetFreeCount();
    int queuedBuffers = m_renderManager.GetQueuedCount();
    
    // 丢帧决策
    bool bDrop = false;
    
    // 条件1：严重延迟 + 非关键帧
    if (iDelay > 0.5 && !pPicture->iFlags & DVP_FLAG_KEY)
    {
        bDrop = true;
        CLog::Log(LOGDEBUG, "OutputPicture: dropping (delay=%.1fms, non-key)",
                   iDelay * 1000);
    }
    
    // 条件2：缓冲区已满
    if (freeBuffers == 0 && queuedBuffers >= m_renderManager.GetQueueSize())
    {
        bDrop = true;
        CLog::Log(LOGDEBUG, "OutputPicture: dropping (buffer full)");
    }
    
    // 条件3：帧间隔小于速度调整后的最小值
    if (m_dropControl.m_state == CDropControl::VALID)
    {
        double diff = pts - m_lastOutputPts;
        if (diff < mindiff && diff > 0)
        {
            bDrop = true;
            CLog::Log(LOGDEBUG, 
                "OutputPicture: dropping (diff=%.1fms < mindiff=%.1fms)",
                diff * 1000, mindiff * 1000);
        }
    }
    
    if (bDrop)
    {
        m_droppingStats.AddDroppedPicture();
        return 0;  // 帧被丢弃
    }
    
    // 正常输出
    m_lastOutputPts = pts;
    m_droppingStats.AddOutputPicture(pts);
    m_renderManager.FlipPage(m_bStop, pts, 0, 0);
    
    return 1;  // 帧已输出
}
```

## 常见错误及解决方案

### 错误 1：NUM_BUFFERS 过小导致解码线程频繁阻塞

**现象**：视频播放帧率不稳定，CPU 占用率不高（解码能力充足），但画面仍然出现周期性卡顿。日志显示解码线程频繁进入 `WaitForBuffer` 等待状态。

**原因**：渲染队列的缓冲区数量（`NUM_BUFFERS`）设置过小。解码线程将解码好的帧放入缓冲区，渲染线程从缓冲区取帧显示。如果缓冲区只有 2 个，解码线程在放入第 3 帧时必须等渲染线程消费掉一帧才能继续——任何一帧渲染稍慢，解码线程就被阻塞，形成"解码等渲染"的瓶颈。

**解决**：

```cpp
// ❌ 缓冲区太小：解码线程频繁阻塞
static const int NUM_BUFFERS = 2;  // 仅双缓冲，几乎没有容错余量

// ✅ 三级缓冲是最低推荐值，视频播放通常用 4-6 个
static const int NUM_BUFFERS = 4;  // 四缓冲：解码线程有 2 帧余量

// 验证方法：添加缓冲区使用率监控
void LogBufferUsage() {
    int freeBuffers = CountFreeBuffers();
    if (freeBuffers <= 1) {
        spdlog::warn("渲染队列接近满载: 空闲缓冲={}/{}",
                     freeBuffers, NUM_BUFFERS);
    }
}
```

> 💡 增大 `NUM_BUFFERS` 会增加内存占用（每个缓冲约 `width × height × 4` 字节，1080p ≈ 8MB），需在内存和流畅度之间权衡。Android 低端设备建议 3，桌面端建议 4-6。

### 错误 2：CDropControl 在 seek 后误判帧率导致不必要的丢帧

**现象**：seek（拖动进度条）后的 1-2 秒内画面异常不流畅，日志显示 `CDropControl` 持续丢帧（drop），即使此时系统负载很低，解码速度完全跟得上。

**原因**：`CDropControl`（丢帧控制器）通过统计最近 N 帧的解码耗时来判断是否需要丢帧。seek 操作后，第一帧通常是关键帧（I 帧），解码耗时比普通帧（P/B 帧）长 3-5 倍。`CDropControl` 把这个异常值计入统计，误以为解码速度不足，连续触发丢帧策略，直到统计窗口中的异常值被冲刷掉。

**解决**：seek 后重置丢帧控制器的统计数据：

```cpp
void CDVDPlayer::HandleSeek() {
    // ... seek 操作 ...

    // 【关键】重置丢帧统计，避免 seek 后的 I 帧耗时污染统计
    m_dropControl.Reset();

    // 同时重置帧计数器
    m_renderManager.ResetFrameCount();
}

// CDropControl::Reset() 的实现
void CDropControl::Reset() {
    m_totalDecodeTime = 0;
    m_frameCount = 0;
    m_dropLevel = NONE;
    // 清空历史记录环形缓冲
    std::fill(m_history.begin(), m_history.end(), 0.0);
}
```

### 错误 3：SetDropState 启用 AVDISCARD_NONREF 后忘记恢复导致永久跳过 B 帧

**现象**：播放器在高负载时主动降质（丢 B 帧）后，即使负载恢复正常，画面质量也不会恢复——动态场景出现明显的"跳帧感"，类似帧率从 24fps 降到 8fps 的效果。

**原因**：性能不足时调用 `SetDropState(AVDISCARD_NONREF)` 通知解码器跳过所有非参考帧（即 B 帧）。这个状态是持久的——设置后解码器会一直跳过 B 帧，直到显式恢复为 `AVDISCARD_DEFAULT`。如果负载恢复正常但没有调用恢复代码，B 帧就永久丢失了。

**解决**：

```cpp
void CDVDPlayerVideo::ProcessDropState() {
    double decodeRatio = GetDecodeTimeRatio();  // 解码耗时 / 帧间隔

    if (decodeRatio > 1.2) {
        // 解码跟不上：启用丢帧
        if (m_currentDiscard != AVDISCARD_NONREF) {
            spdlog::warn("性能不足 (ratio={:.2f})，启用 B 帧丢弃",
                         decodeRatio);
            SetDropState(AVDISCARD_NONREF);
            m_currentDiscard = AVDISCARD_NONREF;
        }
    } else if (decodeRatio < 0.8) {
        // 【关键】性能恢复：必须恢复正常解码
        if (m_currentDiscard != AVDISCARD_DEFAULT) {
            spdlog::info("性能恢复 (ratio={:.2f})，恢复完整解码",
                         decodeRatio);
            SetDropState(AVDISCARD_DEFAULT);
            m_currentDiscard = AVDISCARD_DEFAULT;  // ← 必须更新状态
        }
    }
    // 0.8-1.2 之间：保持当前状态，避免频繁切换
}
```

> ⚠️ 使用滞后阈值（启用 >1.2，恢复 <0.8）而非单一阈值，可以避免在临界状态下频繁切换丢帧/恢复，导致画面忽好忽差。

## 动手实践

### 实践一：模拟渲染队列的缓冲与丢帧

```cpp
// render_queue_sim.cpp — 模拟三级缓冲区管理
#include <cstdio>
#include <deque>
#include <cstdlib>
#include <cmath>

static const int NUM_BUFFERS = 6;

struct FrameBuffer {
    double pts;
    bool   used;
};

class SimpleRenderQueue {
public:
    std::deque<int> m_free;
    std::deque<int> m_queued;
    std::deque<int> m_discard;
    FrameBuffer     m_buffers[NUM_BUFFERS];
    int             m_queueSize;
    int             m_lateFrames;
    int             m_totalDropped;
    
    SimpleRenderQueue() : m_queueSize(3), m_lateFrames(0), m_totalDropped(0)
    {
        for (int i = 0; i < NUM_BUFFERS; i++)
        {
            m_free.push_back(i);
            m_buffers[i].pts = 0;
            m_buffers[i].used = false;
        }
    }
    
    // 生产者：解码线程调用
    bool flipPage(double pts)
    {
        // 先回收 m_discard 中的缓冲区
        while (!m_discard.empty())
        {
            int idx = m_discard.front();
            m_discard.pop_front();
            m_free.push_back(idx);
        }
        
        if (m_free.empty())
        {
            printf("  [FlipPage] ⚠️ 无空闲缓冲区，解码阻塞!\n");
            return false;
        }
        
        int idx = m_free.front();
        m_free.pop_front();
        m_buffers[idx].pts = pts;
        m_queued.push_back(idx);
        
        // 溢出保护
        while ((int)m_queued.size() > m_queueSize)
        {
            int discardIdx = m_queued.front();
            m_queued.pop_front();
            m_discard.push_back(discardIdx);
            m_totalDropped++;
            printf("  [FlipPage] 队列溢出，丢弃帧 PTS=%.3f\n",
                   m_buffers[discardIdx].pts);
        }
        
        return true;
    }
    
    // 消费者：渲染线程调用
    bool prepareNextRender(double clockPts)
    {
        if (m_queued.empty())
            return false;
        
        int lateCount = 0;
        
        // 跳过过期帧
        while (m_queued.size() > 1)
        {
            int frontIdx = m_queued.front();
            if (m_buffers[frontIdx].pts < clockPts - 0.05)  // 50ms 容忍
            {
                m_queued.pop_front();
                m_discard.push_back(frontIdx);
                lateCount++;
                m_totalDropped++;
            }
            else
            {
                break;
            }
        }
        
        if (lateCount > 0)
        {
            m_lateFrames += lateCount;
            printf("  [Render] 丢弃 %d 个延迟帧 (累计延迟帧: %d)\n",
                   lateCount, m_lateFrames);
        }
        else
        {
            m_lateFrames = 0;
        }
        
        int renderIdx = m_queued.front();
        m_queued.pop_front();
        m_discard.push_back(renderIdx);
        printf("  [Render] 渲染帧 PTS=%.3f (clock=%.3f)\n",
               m_buffers[renderIdx].pts, clockPts);
        
        return true;
    }
    
    void printState() const
    {
        printf("  状态: free=%zu, queued=%zu, discard=%zu, "
               "总丢帧=%d\n",
               m_free.size(), m_queued.size(), m_discard.size(),
               m_totalDropped);
    }
};

int main()
{
    SimpleRenderQueue queue;
    
    printf("=== 场景1: 正常播放 (30fps) ===\n");
    for (int i = 0; i < 10; i++)
    {
        double pts = i * (1.0 / 30);
        double clockPts = pts + 0.005;  // 时钟略微领先
        
        printf("帧 %d:\n", i);
        queue.flipPage(pts);
        queue.prepareNextRender(clockPts);
        queue.printState();
    }
    
    printf("\n=== 场景2: 解码卡顿 (连续3帧延迟) ===\n");
    SimpleRenderQueue queue2;
    
    // 先正常送入 5 帧
    for (int i = 0; i < 5; i++)
        queue2.flipPage(i * (1.0 / 30));
    
    // 模拟时钟已经走到第 8 帧的位置（解码卡了 3 帧）
    double clockPts = 8 * (1.0 / 30);
    printf("时钟=%.3f, 但队列中只有帧 0-4:\n", clockPts);
    queue2.prepareNextRender(clockPts);
    queue2.printState();
    
    printf("\n=== 场景3: 缓冲区耗尽 ===\n");
    SimpleRenderQueue queue3;
    queue3.m_queueSize = 4;
    
    // 快速送入 8 帧（超过 NUM_BUFFERS=6）
    printf("快速送入 8 帧（缓冲区上限 %d）:\n", NUM_BUFFERS);
    for (int i = 0; i < 8; i++)
    {
        bool ok = queue3.flipPage(i * (1.0 / 30));
        if (!ok) printf("  帧 %d: 送入失败\n", i);
    }
    queue3.printState();
    
    return 0;
}
```

### 实践二：模拟 CDropControl 状态机

```cpp
// drop_control_sim.cpp — 模拟 CDropControl 的 INIT→VALID 转换
#include <cstdio>
#include <cmath>

static const double DVD_NOPTS_VALUE = -1.0;

class DropControlSim {
public:
    enum State { INIT, VALID };
    
    State  m_state;
    double m_diffSum;
    int    m_diffCount;
    double m_avgDiff;
    int    m_invalidCount;
    double m_lastPts;
    
    DropControlSim() { reset(); }
    
    void reset()
    {
        m_state = INIT;
        m_diffSum = 0;
        m_diffCount = 0;
        m_avgDiff = 0;
        m_invalidCount = 0;
        m_lastPts = DVD_NOPTS_VALUE;
    }
    
    void process(double pts)
    {
        if (m_state == INIT)
        {
            if (m_lastPts != DVD_NOPTS_VALUE && pts != DVD_NOPTS_VALUE)
            {
                double diff = pts - m_lastPts;
                if (diff > 0 && diff < 1.0)
                {
                    m_diffSum += diff;
                    m_diffCount++;
                    printf("  [INIT] 样本 %d: diff=%.3fms\n",
                           m_diffCount, diff * 1000);
                }
            }
            m_lastPts = pts;
            
            if (m_diffCount >= 10)
            {
                m_avgDiff = m_diffSum / m_diffCount;
                if (m_avgDiff > 0)
                {
                    m_state = VALID;
                    printf("  ✓ INIT→VALID: avgDiff=%.3fms (%.1f fps)\n",
                           m_avgDiff * 1000, 1.0 / m_avgDiff);
                }
            }
        }
        else
        {
            if (m_lastPts != DVD_NOPTS_VALUE && pts != DVD_NOPTS_VALUE)
            {
                double diff = pts - m_lastPts;
                double deviation = fabs((diff - m_avgDiff) / m_avgDiff);
                
                if (deviation > 0.2)
                {
                    m_invalidCount++;
                    printf("  [VALID] ⚠️ 异常帧: diff=%.3fms, 偏差=%.1f%%, "
                           "连续异常=%d\n",
                           diff * 1000, deviation * 100, m_invalidCount);
                    
                    if (m_invalidCount >= 5)
                    {
                        printf("  ✗ VALID→INIT (5 次连续异常)\n");
                        reset();
                        m_lastPts = pts;
                        return;
                    }
                }
                else
                {
                    m_invalidCount = 0;
                    double oldAvg = m_avgDiff;
                    m_avgDiff = m_avgDiff * 0.9 + diff * 0.1;
                    printf("  [VALID] 正常帧: diff=%.3fms, "
                           "avgDiff=%.3f→%.3fms\n",
                           diff * 1000, oldAvg * 1000, m_avgDiff * 1000);
                }
            }
            m_lastPts = pts;
        }
    }
};

int main()
{
    DropControlSim dc;
    
    printf("=== 场景1: 稳定 30fps ===\n");
    for (int i = 0; i < 15; i++)
    {
        double pts = i * (1.0 / 30);
        printf("帧 %d (PTS=%.3f):\n", i, pts);
        dc.process(pts);
    }
    
    printf("\n=== 场景2: 帧率突变 (30→60fps) ===\n");
    // 继续上面的状态，突然帧率加倍
    double basePts = 15.0 / 30;
    for (int i = 0; i < 8; i++)
    {
        double pts = basePts + i * (1.0 / 60);
        printf("帧 %d (PTS=%.3f):\n", i + 15, pts);
        dc.process(pts);
    }
    
    printf("\n=== 场景3: PTS 跳变（seek 后） ===\n");
    DropControlSim dc2;
    // 先建立稳定模型
    for (int i = 0; i < 12; i++)
        dc2.process(i * (1.0 / 24));
    
    // 模拟 seek 跳到 10 秒处
    printf("模拟 seek 到 10.0s:\n");
    dc2.process(10.0);
    dc2.process(10.0 + 1.0/24);
    dc2.process(10.0 + 2.0/24);
    
    return 0;
}
```

## 本节小结

- **sws_scale 是最大的 CPU 陷阱**：像素格式转换对每个像素进行数学运算，1080p@30fps 每秒处理约 89MB 数据。最佳优化是让渲染器直接支持解码器输出格式，完全避免 `sws_scale`
- **渲染队列的三级缓冲**（`m_free` → `m_queued` → `m_discard`）实现了生产者-消费者模型，总共 6 个缓冲区循环使用。`m_free` 为空时解码线程阻塞，`m_queued` 溢出时丢弃最旧的帧
- **`PrepareNextRender()`** 在每个 VSync 周期检测并跳过过期帧。`m_lateframes` 超过阈值时触发 `m_QueueSkip`，通知解码端加速
- **`CalcDropRequirement()`** 综合三个信号：绝对延迟（`EOS_VERYLATE`）、缓冲区水位（`EOS_BUFFER_LEVEL`）、丢帧增益（`EOS_DROPPED`），返回位掩码指导解码端行为
- **`CDropControl` 状态机** 通过收集 10 个 PTS 样本建立帧率模型（INIT→VALID），在 VALID 状态下检测帧率偏差，连续 5 个异常帧触发模型重建
- **`SetDropState()`** 通过 FFmpeg 的 `AVDISCARD_NONREF` 跳过 B 帧和环路滤波，可节省约 30-50% 的 CPU 开销

### 题目 1：渲染队列容量分析

假设 KR2 使用 `NUM_BUFFERS = 6`、`m_QueueSize = 3`。在以下场景中，分析各队列的状态：

1. 解码线程连续生产了 5 帧，但渲染线程还没来得及消费
2. 渲染线程开始消费，连续渲染了 2 帧
3. 解码线程又生产了 2 帧

请画出每个步骤后 `m_free`、`m_queued`、`m_discard` 的状态。

<details>
<summary>查看答案</summary>

**初始状态**：
- `m_free = {0,1,2,3,4,5}`, `m_queued = {}`, `m_discard = {}`

**步骤1：解码线程连续生产 5 帧**

生产帧 0: `m_free={1,2,3,4,5}`, `m_queued={0}`, `m_discard={}`
生产帧 1: `m_free={2,3,4,5}`, `m_queued={0,1}`, `m_discard={}`
生产帧 2: `m_free={3,4,5}`, `m_queued={0,1,2}`, `m_discard={}`
生产帧 3: `m_free={4,5}`, `m_queued={1,2,3}`, `m_discard={}` ← 帧0被溢出丢弃到哪里了？

注意：`m_QueueSize = 3`，当 `m_queued` 超过3时触发溢出保护。

实际流程：
- 生产帧3时，`m_queued = {0,1,2,3}` 超过 `m_QueueSize=3`
- 帧0被移到 `m_discard`: `m_queued={1,2,3}`, `m_discard={0}`

生产帧 4: `m_free={5}`, `m_queued={2,3,4}`, `m_discard={0,1}` ← 帧1也被溢出丢弃

**步骤2：渲染线程消费 2 帧**

渲染帧2: `m_free={5}`, `m_queued={3,4}`, `m_discard={0,1,2}`
渲染帧3: `m_free={5}`, `m_queued={4}`, `m_discard={0,1,2,3}`

**步骤3：解码线程生产 2 帧**

先回收 `m_discard`（在 `FlipPage` 中）:
`m_free={5,0,1,2,3}`, `m_queued={4}`, `m_discard={}`

生产帧5: `m_free={0,1,2,3}`, `m_queued={4,5}`, `m_discard={}`
生产帧6: `m_free={1,2,3}`, `m_queued={4,5,0}`, `m_discard={}`

最终状态：`m_free={1,2,3}`, `m_queued={4,5,0}`, `m_discard={}`

**关键结论**：在步骤1中，由于 `m_QueueSize=3` 的限制，有 2 帧因溢出被丢弃。这说明当渲染线程来不及消费时，旧帧会被丢弃以避免延迟累积。

### 题目 3：CDropControl 帧率模型

一个视频以 24fps 播放，`CDropControl` 已经处于 VALID 状态，`avgDiff = 41.67ms`。如果用户执行了 seek 操作，PTS 从 10.0s 跳到 50.0s，请分析 `CDropControl` 会如何响应？需要几帧才能恢复到 VALID 状态？

<details>
<summary>查看答案</summary>

**Seek 后第一帧**（PTS=50.0s, lastPts=10.0+k/24）：
- `diff = 50.0 - lastPts ≈ 40秒`
- `deviation = (40000 - 41.67) / 41.67 ≈ 959.6`（偏差96000%）
- 远超20%阈值 → `m_invalidCount = 1`

**后续帧**（PTS=50.0+1/24, 50.0+2/24, ...）：
- 第2帧：`diff = 41.67ms`, `deviation ≈ 0%` → 正常，`m_invalidCount = 0`

**关键分析**：seek 后只会产生**1个异常帧**（跳变帧），之后立即恢复正常帧间隔。由于阈值是"连续5个异常帧"才重置，所以 `CDropControl` **不会**从 VALID 退回 INIT。

这是一个优秀的设计——单次跳变（如 seek）不会打断帧率模型，只有持续的帧率变化（如视频切换到不同帧率）才会触发模型重建。

**但如果 seek 后的视频帧率改变了**（比如从 24fps 变成 30fps）：
- 每帧 `diff = 33.33ms` vs `avgDiff = 41.67ms`
- `deviation = (33.33 - 41.67) / 41.67 = -0.2 = -20%`
- 刚好在20%阈值边界上
- 如果实际偏差略大于20%（由于时间抖动），连续5帧后会触发 `VALID → INIT`
- 然后需要10个新样本才能回到 VALID 状态
- 总共需要约 **1 + 5 + 10 = 16帧** 才能建立新的帧率模型

</details>
