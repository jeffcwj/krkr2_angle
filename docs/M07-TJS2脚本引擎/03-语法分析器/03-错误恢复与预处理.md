# 第三章 第三节：错误恢复与预处理器

## 术语预览

| 术语 | 英文 | 含义 |
|------|------|------|
| 错误恢复 | Error Recovery | 语法分析器在遇到语法错误后尝试继续解析的机制，而非立即终止 |
| 移进/归约冲突 | Shift/Reduce Conflict | Bison 文法中同一状态下既可移进又可归约时产生的歧义，需要优先级或 `%expect` 解决 |
| 悬挂 else | Dangling Else | `if-else` 嵌套时 `else` 应归属哪个 `if` 的经典文法歧义问题 |
| 预处理器 | Preprocessor | 编译前对源码进行条件编译、宏替换等文本处理的系统 |
| 条件编译 | Conditional Compilation | 根据编译时条件选择性地包含或排除代码段的技术 |
| PP 变量 | Preprocessor Variable | 预处理阶段使用的变量，存储在引擎全局哈希表中，仅支持整数值 |
| `%expect` | — | Bison 指令，声明预期的移进/归约冲突数量，超出则报错 |
| `error` 令牌 | Error Token | Bison 内置的特殊令牌，用于在文法规则中标记错误恢复点 |
| 哈希表 | Hash Table | 以键值对方式存储数据的数据结构，TJS2 中用 `tTJSHashTable` 实现 |
| IfLevel | — | 词法分析器中跟踪 `@if`/`@endif` 嵌套深度的计数器 |

## §1 引言：TJS2 为什么没有错误恢复？

大多数教科书在讲解 Bison/Yacc 语法分析器时，都会花大量篇幅介绍**错误恢复**（Error Recovery）机制。Bison 提供了一个内置的 `error` 特殊令牌，允许文法规则在遇到语法错误时跳过一些输入，然后恢复到一个可以继续解析的状态。例如典型的用法：

```yacc
/* 典型的 Bison 错误恢复规则（TJS2 中并未使用！） */
statement_list
    : statement_list statement
    | statement_list error ';'   /* 遇到错误时跳到下一个分号 */
    | /* empty */
    ;
```

然而，当你打开 TJS2 的 `tjs.y` 文件搜索 `error` 令牌时，你会发现——**一个都找不到**。TJS2 的 862 行 Bison 文法中，没有任何一条规则使用了 `error` 令牌。

这是一个刻意的设计选择，而非遗漏。要理解这个选择，我们需要思考 TJS2 的使用场景：

1. **脚本完整性要求高**：TJS2 脚本控制视觉小说的核心逻辑（场景跳转、变量管理、事件处理）。一个"部分编译"的脚本运行起来可能产生难以调试的状态不一致问题。
2. **即时反馈优于部分结果**：对于脚本作者来说，立即报错并指出错误位置，比编译器默默跳过错误继续编译更有帮助。
3. **错误恢复的复杂性**：正确的错误恢复需要在文法中精心设计同步点（synchronization points），这增加了文法复杂度，而 TJS2 的文法已经有 862 行。

TJS2 选择了一条更简洁的路线：**遇到语法错误立即抛出 C++ 异常，终止编译**。

## §2 Bison 的 `%expect` 指令与移进/归约冲突

### 2.1 什么是移进/归约冲突？

在 LALR(1) 分析中，**移进/归约冲突**（Shift/Reduce Conflict）发生在解析器的某个状态下，面对当前的前看令牌（lookahead token），既可以选择"移进"（shift，读取下一个令牌），也可以选择"归约"（reduce，将栈顶的符号按规则收缩）。

最经典的例子是**悬挂 else**（Dangling Else）问题：

```javascript
// 这段代码中的 else 属于哪个 if？
if (a)
    if (b)
        doSomething();
    else          // 属于外层 if(a) 还是内层 if(b)？
        doOther();
```

从文法角度看，`else` 既可以归约内层 `if(b) doSomething()` 为一个完整的 `if` 语句（然后 `else` 与外层配对），也可以移进 `else` 让它与内层 `if(b)` 配对。Bison 的默认行为是**选择移进**（shift），即 `else` 与最近的 `if` 配对——这正是几乎所有编程语言的约定。

### 2.2 TJS2 中的 `%expect 1`

打开 `tjs.y`，第 3 行就是：

```yacc
/* tjs.y 第 3 行 */
%expect 1
```

这条指令告诉 Bison："我预期整个文法恰好有 **1 个**移进/归约冲突。如果冲突数量不是 1，就报告错误。"

这是一个**安全网**——文法的作者知道悬挂 else 会产生 1 个不可避免的冲突，并且 Bison 默认的移进优先策略正好是期望的行为。`%expect 1` 确保如果文法修改引入了新的意外冲突，Bison 会立即发出警告。

### 2.3 TJS2 的 if-else 文法规则

让我们看 `tjs.y` 中的相关规则：

```yacc
/* tjs.y 中 if 语句的文法规则 */
if_statement
    : T_IF '(' expr_no_comma ')' block_or_statement
      {
          /* if 没有 else */
          cc->ExitIfCode();
      }
    | T_IF '(' expr_no_comma ')' block_or_statement T_ELSE block_or_statement
      {
          /* if-else */
          cc->ExitElseCode();
      }
    ;
```

当解析器看到：

```
if (a) if (b) doSomething();
```

然后遇到 `else` 令牌时：
- **移进**（shift）：把 `else` 移入栈中，让它与内层 `if(b)` 匹配 → 产生 `if(b)...else...`
- **归约**（reduce）：先把 `if(b) doSomething();` 归约为完整的 `if_statement`（第一条规则），然后 `else` 与外层 `if(a)` 匹配

Bison 选择移进，因此 `else` 总是与最近的 `if` 配对。`%expect 1` 声明了这个预期冲突。

### 2.4 为什么不用 `%nonassoc` 消除冲突？

有些语言用 `%nonassoc T_ELSE` 或更复杂的文法改写来彻底消除悬挂 else 冲突。TJS2 选择保留冲突并用 `%expect 1` 标记，原因是：

1. **文法更简洁**：不需要引入 `matched_statement` / `unmatched_statement` 等辅助规则
2. **行为完全正确**：Bison 默认的移进行为就是期望行为
3. **维护更安全**：`%expect 1` 作为断言，任何引入新冲突的修改都会被立即发现

```
开发者的心声：
    "我知道有 1 个冲突，这是故意的。
     如果你看到 2 个冲突，说明有人改坏了文法。"
                                    —— %expect 的设计意图
```

## §3 TJS2 的错误处理机制：C++ 异常替代 Bison error

### 3.1 Bison error 令牌回顾

在标准 Bison 用法中，`error` 是一个特殊令牌。当解析器遇到语法错误（当前令牌无法被任何文法规则接受），它会：

1. 将栈中的符号逐个弹出，直到找到一个能接受 `error` 令牌的状态
2. "移进" `error` 令牌
3. 丢弃输入中的令牌，直到找到一个可以继续解析的令牌
4. 恢复正常解析

这个过程叫做**恐慌模式恢复**（Panic Mode Recovery）。例如：

```yacc
/* 某些编译器的典型错误恢复规则 */
statement
    : expression_statement
    | compound_statement
    | error ';'    /* 遇到错误，跳到下一个分号，然后继续 */
    ;
```

### 3.2 TJS2 的替代方案：C++ 异常

TJS2 完全没有使用 `error` 令牌。取而代之的是 C++ 异常机制。当解析器（Bison 生成的代码）调用 `yyerror()` 报告语法错误时，控制流转入 TJS2 的 C++ 错误处理系统。

让我们追踪错误处理的完整调用链：

```
                 Bison 生成代码
                      │
            遇到语法错误，调用 yyerror()
                      │
                      ▼
        ┌──────────────────────────┐
        │ parser::error()          │
        │ tjsCompileControl.cpp:31 │
        │ spdlog 记录错误信息      │
        └──────────────────────────┘
                      │
                      ▼
        ┌──────────────────────────┐
        │ TJS_eTJSCompileError()   │
        │ 抛出 C++ 异常            │
        │ 附带错误位置和消息       │
        └──────────────────────────┘
                      │
            C++ 异常展开调用栈
                      │
                      ▼
        ┌──────────────────────────┐
        │ 上层 try/catch           │
        │ (ScriptMgnIntf 或调用者) │
        │ 显示错误对话框           │
        └──────────────────────────┘
```

### 3.3 源码解读：error 回调函数

在 `tjsCompileControl.cpp` 中，Bison 的错误回调函数非常简洁：

```cpp
/* tjsCompileControl.cpp 第 31-33 行 */
void parser::error(const location_type& loc, const std::string& msg)
{
    spdlog::error("Bison Parser Error: {}", msg);
}
```

这个函数**只做日志记录**。它没有抛出异常——因为 Bison 生成的代码在调用 `yyerror()` 后会返回错误状态码，而 TJS2 的编译控制逻辑会检查这个状态码并决定是否抛出异常。

### 3.4 编译异常类：eTJSCompileError

TJS2 定义了专用的编译错误异常类。当编译过程中的任何阶段（词法分析、语法分析、代码生成）遇到不可恢复的错误时，都会抛出此异常：

```cpp
/* TJS2 编译错误的典型抛出方式 */
TJS_eTJSCompileError(
    TJSPPExprEvalError,     // 错误消息字符串
    lexer,                   // 词法分析器实例（用于获取位置信息）
    lexer->GetCurrentPosition()  // 源码位置
);
```

异常被抛出后，C++ 的栈展开机制会自动清理 Bison 解析栈、临时变量等资源。这比 Bison 自带的 `error` 恢复机制更彻底——不会留下任何"半解析"的状态。

### 3.5 错误信息的生成

TJS2 在多个层次生成错误信息：

```cpp
/* 词法分析阶段的错误（tjsLex.cpp 中的各种位置） */
// 未终止的字符串
TJS_eTJSCompileError(TJSStringParseError, this, Current);

// 未终止的注释
TJS_eTJSCompileError(TJSCommentNotClosed, this, Current);

// 无效的字符
TJS_eTJSCompileError(TJSInvalidChar, this, Current);

/* 预处理阶段的错误（tjsLex.cpp ProcessPPStatement 中） */
// @endif 多余
TJS_eTJSCompileError(TJSPPError, this, Current);

// @if/@endif 不匹配（文件结束时 IfLevel > 0）
TJS_eTJSCompileError(TJSPPError, this, Current);

/* 代码生成阶段的错误（tjsInterCodeGen.cpp 中） */
// 不可赋值的目标
TJS_eTJSCompileError(TJSCannotAssignError, block, node->GetPosition());

// break/continue 出现在循环外
TJS_eTJSCompileError(TJSMisplacedBreakContinue, block, node->GetPosition());
```

### 3.6 与 Bison 错误恢复的对比

| 特性 | Bison `error` 令牌 | TJS2 C++ 异常 |
|------|---------------------|---------------|
| 错误后行为 | 尝试恢复并继续解析 | 立即终止编译 |
| 错误报告数量 | 可报告多个错误 | 报告第一个错误即停止 |
| 实现复杂度 | 需要在文法中仔细设计恢复规则 | 仅需 try/catch |
| 误报风险 | 恢复后可能产生级联假错误 | 无假错误（因为只报第一个） |
| 资源清理 | 需要手动清理语义值 | C++ 析构函数自动清理 |
| 适用场景 | IDE 需要多错误提示时 | 命令行/引擎嵌入编译时 |
| 编译产物 | 可能产生不完整的编译结果 | 要么完整编译，要么完全失败 |

对于 TJS2 这样的嵌入式脚本引擎来说，"全有或全无"（all-or-nothing）的编译策略是更安全的选择。一个视觉小说引擎加载了"半编译"的脚本，可能在运行几十分钟后才因为缺失的函数或变量而崩溃——这比编译时直接报错要糟糕得多。

## §4 错误定位与诊断信息

### 4.1 位置跟踪机制

TJS2 通过词法分析器维护精确的源码位置信息。每个令牌都携带其在源文件中的位置：

```cpp
/* tjsLex.h 中的位置相关成员 */
class tTJSLexicalAnalyzer {
    tjs_int PrevPos;       // 前一个令牌的位置
    tjs_int PrevToken;     // 前一个令牌的类型
    
    // GetCurrentPosition() 返回当前扫描位置
    tjs_int GetCurrentPosition() { return (tjs_int)(Current - Script); }
};
```

位置信息用指针偏移量表示——`Current - Script` 得到的是当前位置相对于脚本起始的字节偏移。这个偏移量会被嵌入到抛出的异常中，用于事后计算行号和列号。

### 4.2 从字节偏移到行列号

当异常被捕获并需要显示给用户时，TJS2 通过遍历源码来计算行列号：

```cpp
/* 简化的行号计算逻辑 */
int line = 1;
int column = 1;
for (int i = 0; i < errorPosition; i++) {
    if (script[i] == '\n') {
        line++;
        column = 1;
    } else {
        column++;
    }
}
// 现在 line 和 column 就是错误位置的行列号
```

这种"惰性计算"策略很高效：正常编译时不需要为每个令牌计算行号（节省内存和时间），只有在出错时才进行一次遍历。对于 TJS2 的典型使用场景——脚本文件通常只有几百到几千行——遍历的开销完全可以忽略。

### 4.3 错误信息格式

TJS2 向用户展示的错误信息通常包含：

```
Script exception : 位置 line(42) --> 语法错误：意外的令牌 "}"
  脚本: scenario/first.ks
```

这个格式清晰地告诉脚本作者：
- **在哪个文件**：`scenario/first.ks`
- **在第几行**：`line(42)`
- **什么问题**：语法错误的具体描述

### 4.4 多脚本编译的错误隔离

在吉里吉里引擎中，通常会加载多个 TJS2 脚本文件。每个脚本的编译是独立的：

```
┌─────────────────────────────────────────────┐
│  引擎加载流程                                │
│                                              │
│  脚本1.tjs ──→ 编译 ──→ 成功 ✓              │
│  脚本2.tjs ──→ 编译 ──→ 失败 ✗ ──→ 报错     │
│  脚本3.tjs ──→ 编译 ──→ 跳过（依赖脚本2）   │
│                                              │
│  结果：脚本1可用，脚本2和3不可用             │
└─────────────────────────────────────────────┘
```

每个脚本的编译被独立的 try/catch 块包裹。一个脚本的编译失败不会影响其他脚本的编译（除非存在依赖关系）。这种隔离是 C++ 异常机制天然提供的——每个编译调用有自己的 catch 处理器。

### 4.5 实际错误示例

让我们看几个 TJS2 中常见的编译错误场景：

```javascript
/* 错误1：缺少分号 */
var x = 10
var y = 20;    // ← 解析器在这里报错，因为 "var" 不是表达式的一部分

/* 错误2：括号不匹配 */
if (x > 0 {   // ← 期望 ')' 但遇到了 '{'
    doSomething();
}

/* 错误3：无效的赋值目标 */
10 = x;        // ← 代码生成阶段报错：不可赋值

/* 错误4：break 在循环外 */
function test() {
    break;     // ← 代码生成阶段报错：break/continue 不在循环中
}

/* 错误5：预处理错误 */
@endif         // ← 没有匹配的 @if
```

注意错误 3 和 4 不是在语法分析阶段捕获的，而是在**代码生成阶段**（`tjsInterCodeGen.cpp`）。TJS2 的文法允许这些结构通过语法检查，但在生成字节码时检测到语义错误。这是一种常见的设计模式：让文法保持简洁，把复杂的语义检查推迟到后续阶段。

## §5 TJS2 预处理器架构总览

### 5.1 预处理器的设计定位

TJS2 内嵌了一个轻量级的**预处理器**（Preprocessor），支持条件编译功能。与 C/C++ 的预处理器不同，TJS2 的预处理器：

- **不支持宏替换**（`#define`）：没有文本替换功能
- **不支持文件包含**（`#include`）：不能在预处理阶段引入其他文件
- **只支持条件编译**：`@if`、`@set`、`@endif` 三个指令
- **仅操作整数值**：PP 变量只能存储 `tjs_int32`（32 位整数），不支持字符串或浮点数

这个设计非常克制——它只解决一个问题：**根据运行时条件选择性地编译代码段**。

### 5.2 预处理器不是独立阶段

**关键设计决策**：TJS2 的预处理器**嵌入在词法分析器中**，而非作为独立的预处理阶段。

```
传统编译器流水线（如 C/C++）:
    源码 → 预处理器 → 词法分析器 → 语法分析器 → ...
              ↑
          独立阶段，先执行

TJS2 的流水线:
    源码 → 词法分析器（内嵌预处理器）→ 语法分析器 → ...
              ↑
          预处理在词法分析过程中按需执行
```

这意味着：
1. 词法分析器在逐字符扫描源码时，遇到行首的 `@` 就切换到预处理模式
2. 预处理指令的解析和执行发生在 `GetToken()` 的调用过程中
3. 对语法分析器来说，预处理指令是**完全透明的**——语法分析器永远看不到 `@if` 或 `@set`

### 5.3 预处理器的核心组件

## §6 PP 变量存储：PPValues 哈希表

### 6.1 PPValues 的初始化

PP 变量存储在 TJS2 引擎的全局对象 `tTJS` 中：

```cpp
/* tjs.h 第 110 行附近 */

    tTJSPPValues *PPValues;  // 预处理变量存储
    // ...
};
```

`tTJSPPValues` 内部使用 TJS2 自己的哈希表实现来存储键值对。在引擎初始化时，会预设一个名为 `version` 的 PP 变量：


`TJSVersionHex` 是一个编译时常量，格式为 `0xMMNNPPPP`：
- `MM` = 主版本号
- `NN` = 次版本号  
- `PPPP` = 补丁版本号

例如 TJS2 版本 2.4.28 对应 `0x0204001C`。

### 6.2 SetPPValue 与 GetPPValue

```cpp

tjs_int32 tTJS::GetPPValue(const tjs_char *name, tjs_int32 default_value) {
    tjs_int32 *f = PPValues->Values.Find(ttstr(name));

}
```

注意 `GetPPValue` 的安全设计：
- 查找不到的变量**不会报错**，而是返回 `default_value`（通常为 0）

## §7 @set 指令的完整执行流程

### 7.1 入口：ProcessPPStatement

当词法分析器的 `GetToken()` 方法在行首遇到 `@` 字符时，它会调用 `ProcessPPStatement()` 来处理预处理指令。让我们逐步追踪 `@set(debug = 1)` 的执行过程：

```cpp
/* tjsLex.cpp 第 1123-1233 行（简化） */
void tTJSLexicalAnalyzer::ProcessPPStatement() {
    // 跳过 '@' 后的空白
    SkipSpace();
    
    // 读取指令名称（"set"、"if"、"endif"）
    tjs_char *start = Current;
    while (isalpha(*Current)) Current++;
    ttstr directive(start, Current - start);
    
    if (directive == TJS_W("set")) {
        // === @set 处理 ===
        // 跳过 '('
        SkipSpace();
        if (*Current != '(') /* 报错 */;
        Current++;
        
        // 调用表达式解析器
        tjs_int32 result = ParsePPExpression();
        
        // 跳过 ')'
        SkipSpace();
        if (*Current != ')') /* 报错 */;
        Current++;
    }
    else if (directive == TJS_W("if")) {
        // === @if 处理（见 §8） ===
        // ...
    }
    else if (directive == TJS_W("endif")) {
        // === @endif 处理 ===
        IfLevel--;
        if (IfLevel < 0) {
            TJS_eTJSCompileError(TJSPPError, this, Current);
        }
    }
}
```

### 7.2 ParsePPExpression：创建 PP 解析器

`ParsePPExpression()` 是连接词法分析器和 PP 表达式 Bison 解析器的桥梁函数：

```cpp
/* tjsLex.cpp 第 1578-1604 行 */
tjs_int32 tTJSLexicalAnalyzer::ParsePPExpression() {
    // 1. 创建 PP 表达式解析器实例
    tTJSPPExprParser *parser = new tTJSPPExprParser(
        TJS,           // TJS2 引擎实例（用于访问 PPValues）
        Script,        // 当前脚本文本
        Current        // 当前扫描位置
    );
    
    // 2. 调用 Bison 生成的解析器
    tjs_int32 result = 0;
    try {
        result = parser->Parse();  // 执行解析和求值
    } catch (...) {
        delete parser;
        throw;
    }
    
    // 3. 更新词法分析器的扫描位置
    //    （PP 解析器已经消耗了表达式中的字符）
    Current = parser->GetCurrent();
    
    delete parser;
    return result;
}
```

这里有一个精妙的设计：**PP 解析器和主词法分析器共享同一个源码缓冲区**。PP 解析器从 `Current` 开始解析表达式，完成后通过 `GetCurrent()` 返回新的位置。主词法分析器更新自己的 `Current` 指针，从 PP 表达式之后继续扫描。

### 7.3 tTJSPPExprParser 的词法分析

PP 解析器有自己独立的 `yylex()` 函数（区别于主解析器的 `yylex()`），用来为 `tjspp.y` 提供令牌：

```cpp
/* tjsCompileControl.cpp 中 PP 的 yylex（简化） */
int tTJSPPExprParser::GetNext(YYSTYPE *value) {
    SkipSpace();
    
    tjs_char c = *Current;
    
    // 整数字面量
    if (isdigit(c)) {
        *value = ParseInteger();  // 解析为 tjs_int32
        return PT_NUM;
    }
    
    // 符号（PP 变量名）
    if (isalpha(c) || c == '_') {
        ttstr name = ParseIdentifier();
        *value = GetTJS()->GetPPValue(
            name.c_str(), 0  // 默认值为 0
        );
        return PT_SYMBOL;  // 注意：符号立即求值为整数
    }
    
    // 运算符
    switch (c) {
        case '+': Current++; return '+';
        case '-': Current++; return '-';
        case '*': Current++; return '*';
        case '/': Current++; return '/';
        case '(': Current++; return '(';
        case ')': return ')';  // 注意：不消耗 ')'，留给调用者
        // ... 更多运算符
        case '=':
            if (*(Current+1) == '=') {
                Current += 2; return PT_EQUALEQUAL;
            }
            Current++; return '=';
        // ...
    }
    
    return 0;  // 结束
}
```

**关键观察**：`PT_SYMBOL`（变量名）在 `yylex` 阶段就**立即求值**为整数值。Bison 文法看到的不是变量名，而是已经替换好的整数值。这极大地简化了文法——文法只需要处理整数运算。

### 7.4 tjspp.y 文法详解

`tjspp.y` 是一个完整的 Bison 文法文件，定义了 PP 表达式的语法。虽然只有 116 行，但支持丰富的运算符：

```yacc
/* bison/tjspp.y（关键部分） */
%token PT_SYMBOL PT_NUM
%token PT_LPARENTHESIS PT_RPARENTHESIS
%token PT_EQUALEQUAL PT_EXCRAMATIONEQUAL
%token PT_AMPAMP PT_PIPEPIPE
%token PT_VERTLINE PT_CHEVRON PT_AMPERSAND
%token PT_PLUS PT_MINUS PT_ASTERISK PT_SLASH PT_PERCENT
%token PT_NOTEQUAL PT_LOGICALNOT
%token PT_LESS PT_GREATER PT_LESSEQUAL PT_GREATEREQUAL
%token PT_LSHIFT PT_RSHIFT

/* 优先级声明（从低到高） */
%right '='               /* 赋值（最低优先级） */
%left PT_PIPEPIPE        /* ||  逻辑或 */
%left PT_AMPAMP          /* &&  逻辑与 */
%left PT_VERTLINE        /* |   按位或 */
%left PT_CHEVRON         /* ^   按位异或 */
%left PT_AMPERSAND       /* &   按位与 */
%left PT_EQUALEQUAL PT_EXCRAMATIONEQUAL  /* ==  != */
%left PT_LESS PT_GREATER PT_LESSEQUAL PT_GREATEREQUAL  /* < > <= >= */
%left PT_LSHIFT PT_RSHIFT  /* <<  >> */
%left PT_PLUS PT_MINUS     /* +  - */
%left PT_ASTERISK PT_SLASH PT_PERCENT  /* *  /  % */
%right PT_LOGICALNOT       /* !  逻辑非（最高优先级） */

%%

/* 文法规则 */
input
    : expr    { ptr->Result = $1; }  /* 表达式结果存入解析器 */
    ;

expr
    : PT_NUM                   { $$ = $1; }
    | PT_SYMBOL                { $$ = $1; }  /* 已经在 yylex 中求值 */
    | PT_SYMBOL '=' expr       {
        /* 赋值：设置 PP 变量 */
        ptr->SetValue($1_name, $3);
        $$ = $3;
      }
    | expr PT_EQUALEQUAL expr  { $$ = ($1 == $3) ? 1 : 0; }
    | expr PT_EXCRAMATIONEQUAL expr  { $$ = ($1 != $3) ? 1 : 0; }
    | expr PT_AMPAMP expr      { $$ = ($1 && $3) ? 1 : 0; }
    | expr PT_PIPEPIPE expr    { $$ = ($1 || $3) ? 1 : 0; }
    | expr PT_VERTLINE expr    { $$ = $1 | $3; }
    | expr PT_CHEVRON expr     { $$ = $1 ^ $3; }
    | expr PT_AMPERSAND expr   { $$ = $1 & $3; }
    | expr PT_LESS expr        { $$ = ($1 < $3) ? 1 : 0; }
    | expr PT_GREATER expr     { $$ = ($1 > $3) ? 1 : 0; }
    | expr PT_LESSEQUAL expr   { $$ = ($1 <= $3) ? 1 : 0; }
    | expr PT_GREATEREQUAL expr  { $$ = ($1 >= $3) ? 1 : 0; }
    | expr PT_LSHIFT expr     { $$ = $1 << $3; }
    | expr PT_RSHIFT expr     { $$ = $1 >> $3; }
    | expr PT_PLUS expr       { $$ = $1 + $3; }
    | expr PT_MINUS expr      { $$ = $1 - $3; }
    | expr PT_ASTERISK expr   { $$ = $1 * $3; }
    | expr PT_SLASH expr      {
        if ($3 == 0) {
            /* 除零检查 */
            yyerror("division by zero in preprocessor expression");
            $$ = 0;
        } else {
            $$ = $1 / $3;
        }
      }
    | expr PT_PERCENT expr    {
        if ($3 == 0) {
            yyerror("modulo by zero in preprocessor expression");
            $$ = 0;
        } else {
            $$ = $1 % $3;
        }
      }
    | PT_LOGICALNOT expr      { $$ = (!$2) ? 1 : 0; }
    | PT_MINUS expr %prec PT_LOGICALNOT  { $$ = -$2; }  /* 一元负号 */
    | '(' expr ')'            { $$ = $2; }
    ;
```

### 7.5 赋值操作的特殊处理

在 `@set(debug = 1)` 中，`debug = 1` 的赋值操作需要知道变量名（不仅仅是值）。但前面说过 `PT_SYMBOL` 在 yylex 阶段就被求值为整数了。那赋值时如何知道变量名呢？

解决方案是 `tTJSPPExprParser` 在 yylex 时**同时保存变量名**：

```cpp
/* 简化的变量名记录机制 */
// 在 GetNext() 中：
if (isalpha(c)) {
    ttstr name = ParseIdentifier();
    last_symbol_name = name;        // 记录变量名
    *value = GetTJS()->GetPPValue(name.c_str(), 0);
    return PT_SYMBOL;
}

// 在 Bison 动作中：
// PT_SYMBOL '=' expr
//   → 使用 last_symbol_name 调用 SetPPValue
ptr->GetTJS()->SetPPValue(last_symbol_name.c_str(), $3);
```

### 7.6 完整执行追踪：`@set(debug = 1)`

让我们完整追踪这条指令的执行过程：

```
源码: @set(debug = 1)
       ^
       词法分析器在行首遇到 '@'

步骤 1: ProcessPPStatement() 被调用
步骤 2: 读取指令名 "set"
步骤 3: 跳过 '('
步骤 4: ParsePPExpression() 被调用
```

## §8 @if/@endif 条件编译与 SkipUntil_endif

### 8.1 @if 指令的执行逻辑

`@if` 指令评估一个 PP 表达式。如果结果为非零（真），则正常继续编译后续代码；如果结果为零（假），则跳过代码直到遇到匹配的 `@endif`。

```cpp
/* tjsLex.cpp ProcessPPStatement() 中 @if 的处理（简化） */
if (directive == TJS_W("if")) {
    SkipSpace();
    if (*Current != '(') /* 报错 */;
    Current++;
    
    // 评估条件表达式
    tjs_int32 result = ParsePPExpression();
    
    SkipSpace();
    if (*Current != ')') /* 报错 */;
    Current++;
    
    if (result) {
        // 条件为真：递增嵌套层级，正常继续词法分析
        IfLevel++;
    } else {
        // 条件为假：跳过源码直到匹配的 @endif
        SkipUntil_endif();
    }
}
```

### 8.2 SkipUntil_endif：字符级跳过

当 `@if` 条件为假时，`SkipUntil_endif()` 负责跳过所有源码直到找到匹配的 `@endif`。这个函数直接在字符级别操作，**不经过正常的词法分析**。

```cpp
/* tjsLex.cpp 第 1042-1120 行（简化） */
void tTJSLexicalAnalyzer::SkipUntil_endif() {
    tjs_int nest_level = 0;  // 本地嵌套计数器
    
    while (true) {
        // 跳到下一行的开头
        while (*Current && *Current != '\n') {
            // 跳过注释（避免误识别注释中的 @if/@endif）
            if (*Current == '/' && *(Current+1) == '/') {
                // 行注释：跳到行尾
                while (*Current && *Current != '\n') Current++;
                break;
            }
            if (*Current == '/' && *(Current+1) == '*') {
                // 块注释：跳到 */
                Current += 2;
                while (*Current) {
                    if (*Current == '*' && *(Current+1) == '/') {
                        Current += 2;
                        break;
                    }
                    Current++;
                }
                continue;
            }
            Current++;
        }
        
        if (!*Current) {
            // 到达文件末尾但没找到 @endif → 报错
            TJS_eTJSCompileError(TJSPPError, this, Current);
        }
        
        Current++;  // 跳过 '\n'
        
        // 检查新行是否以 '@' 开头
        SkipSpace();
        if (*Current == '@') {
            Current++;
            ttstr directive = ReadIdentifier();
            
            if (directive == TJS_W("if")) {
                // 嵌套的 @if：增加本地嵌套计数
                nest_level++;
            }
            else if (directive == TJS_W("endif")) {
                if (nest_level == 0) {
                    // 找到匹配的 @endif：退出跳过模式
                    return;
                }
                nest_level--;  // 这是内层 @if 的 @endif
            }
            // 注意：跳过模式中的 @set 被忽略（不执行）
        }
    }
}
```

### 8.3 嵌套处理的精妙之处

`SkipUntil_endif()` 使用一个**独立的本地** `nest_level` 来跟踪嵌套，而不是使用词法分析器的 `IfLevel`。这很重要：

```javascript
/* 嵌套的条件编译示例 */
@if(platform == 1)         // IfLevel: 0 → 条件为假
    // 整个块被跳过
    @if(version >= 0x0200)  // nest_level: 0 → 1
        // 也被跳过
    @endif                  // nest_level: 1 → 0
    // 继续跳过
    doWindowsStuff();
@endif                      // nest_level: 0 → 找到匹配！返回

@if(platform == 2)         // IfLevel: 0 → 条件为真
    // 正常编译
    IfLevel++;              // IfLevel: 1
    doCrossPlatformStuff();
@endif                      // IfLevel: 1 → 0
```

跳过追踪（当 `platform == 2` 时）：

```
行  1: @if(platform == 1)  → 条件为假 → 进入 SkipUntil_endif()
行  3:     @if(...)         → nest_level: 0 → 1（仅计数，不求值）
行  5:     @endif           → nest_level: 1 → 0（内层结束）
行  7: @endif               → nest_level == 0 → 匹配！返回
行  9: @if(platform == 2)  → 条件为真 → IfLevel++，正常编译
行 11: @endif               → IfLevel--
```

注意在跳过模式中：
- **不求值**嵌套的 `@if` 表达式（只计数）
- **不执行**跳过区域中的 `@set`（变量不被修改）
- **跳过注释**但不解析（避免注释中的 `@if` 误触发）

### 8.4 IfLevel 的作用域

`IfLevel` 是词法分析器的成员变量，跟踪**正常模式**（非跳过模式）下 `@if` 的嵌套深度：

```
IfLevel 在正常模式下的变化：

@if(condA)          // condA 为真 → IfLevel: 0 → 1
    @if(condB)      // condB 为真 → IfLevel: 1 → 2
        code();     // IfLevel == 2
    @endif          // IfLevel: 2 → 1
@endif              // IfLevel: 1 → 0

// 如果 condA 为真但 condB 为假：
@if(condA)          // condA 为真 → IfLevel: 0 → 1
    @if(condB)      // condB 为假 → 调用 SkipUntil_endif()
                    //（IfLevel 不变，跳过使用本地 nest_level）
    // ...跳过...
    @endif          // 由 SkipUntil_endif() 处理
@endif              // IfLevel: 1 → 0
```

### 8.5 边界情况：文件末尾的 @if 检查

当词法分析器到达文件末尾时，它会检查 `IfLevel` 是否为 0。如果不为 0，说明有未匹配的 `@if`，这是一个编译错误：

```cpp
/* 文件扫描结束时的检查 */
if (IfLevel > 0) {
    TJS_eTJSCompileError(TJSPPError, this, Current);
    // 错误消息："预处理指令不匹配"
}
```

这个检查确保了每个 `@if` 都有对应的 `@endif`。注意它只检查**正常模式**下的嵌套——跳过模式中的未匹配情况由 `SkipUntil_endif()` 自己处理（到达文件末尾时报错）。

### 8.6 @if 不支持 @else

一个值得注意的限制是：TJS2 的预处理器**不支持 `@else`**。如果需要"非此即彼"的条件编译，必须这样写：

```javascript
/* TJS2 中的 "if-else" 条件编译 */

// 不能这样写（不支持 @else）：
// @if(debug)
//     debugCode();
// @else
//     releaseCode();
// @endif

// 必须这样写：
@if(debug)
    debugCode();
@endif
@if(!debug)
    releaseCode();
@endif
```

这是一个有意的简化设计。`@else` 的实现需要更复杂的状态管理（跳过模式中遇到 `@else` 要切换回正常模式），而 `@if(!expr)` 的写法虽然稍显冗长，但实现简单且不容易出 bug。

## §9 两个 Bison 解析器的对比

TJS2 中同时存在**两个**独立的 Bison 解析器，它们有着截然不同的设计目标和实现特征。理解这种对比有助于深入理解编译器设计中"一个工具多种用法"的思想。

### 9.1 架构对比

```
┌───────────────────────────────────────────────────────┐
│              TJS2 的双 Bison 解析器架构                │
│                                                       │
│  ┌─────────────────────┐  ┌─────────────────────────┐ │
│  │   主解析器 (tjs.y)  │  │  PP解析器 (tjspp.y)     │ │
│  │                     │  │                         │ │
│  │  862 行文法         │  │  116 行文法             │ │
│  │  ~50 种令牌类型     │  │  ~20 种令牌类型         │ │
│  │  处理完整 TJS2 语法 │  │  仅处理整数表达式       │ │
│  │  输出 AST 节点      │  │  直接计算整数值         │ │
│  │  由主词法分析器喂养 │  │  有自己的微型词法分析器 │ │
│  │  生命周期=整个编译  │  │  生命周期=单个 @set/@if │ │
│  │  C++ 命名空间: TJS  │  │  C++ 命名空间: TJSPP   │ │
│  └─────────────────────┘  └─────────────────────────┘ │
│                                                       │
│  共性：都使用 Bison/Yacc 生成，LALR(1) 分析算法      │
└───────────────────────────────────────────────────────┘
```

### 9.2 详细对比表

| 特性 | 主解析器 `tjs.y` | PP 解析器 `tjspp.y` |
|------|-------------------|---------------------|
| **文件大小** | 862 行 | 116 行 |
| **命名空间** | `TJS` | `TJSPP` |
| **输入令牌来源** | `tTJSLexicalAnalyzer::GetNext()` | `tTJSPPExprParser::GetNext()` |
| **语义值类型** | `tjs_int`（AST 节点索引） | `tjs_int32`（整数值） |
| **输出** | AST 节点树 | 单个整数结果 |
| **文法复杂度** | 完整编程语言（类、函数、循环、异常等） | 仅算术/逻辑/比较表达式 |
| **生命周期** | 整个脚本编译过程 | 单条 `@set` 或 `@if` 指令 |
| **错误恢复** | 无（C++ 异常） | 无（直接报错） |
| **移进/归约冲突** | 1（`%expect 1`，悬挂 else） | 0 |
| **赋值支持** | 完整赋值表达式系统 | 仅 `symbol = expr` 形式 |
| **数据类型** | 整数、浮点、字符串、对象等 | 仅 `tjs_int32` 整数 |
| **运算符数量** | 40+（含属性访问、函数调用等） | 20（纯算术/逻辑） |
| **对应的编译控制类** | `tTJSScriptBlock` 包裹 | `tTJSPPExprParser` 包裹 |

### 9.3 为什么需要两个解析器？

一个自然的疑问是：为什么不让主解析器也处理 PP 表达式？

答案涉及几个设计考量：

1. **执行时机不同**：PP 表达式在**词法分析阶段**求值，而主解析器在词法分析完成后才工作。如果让主解析器处理 PP 表达式，就需要先完成整个脚本的词法分析（包括被条件排除的代码），这违背了条件编译"跳过不需要的代码"的目的。

2. **类型系统不同**：主解析器处理 TJS2 的完整类型系统（tTJSVariant），而 PP 表达式只需要整数。使用完整类型系统来处理简单整数运算是浪费。

3. **关注点分离**：PP 解析器的文法不需要处理函数定义、类声明、异常处理等复杂结构，保持它的简洁性有助于维护。

4. **共存的优雅性**：两个 Bison 文法各自生成独立的解析器代码，通过 C++ 命名空间隔离（`TJS::parser` vs `TJSPP::parser`），互不干扰。

```cpp
/* 两个解析器的命名空间隔离 */

// tjs.y 生成的代码在 TJS 命名空间中
namespace TJS {
    class parser { /* ... */ };
}

// tjspp.y 生成的代码在 TJSPP 命名空间中
namespace TJSPP {
    class parser { /* ... */ };
}

// 两者可以在同一个编译单元中共存
```

### 实验1：模拟 PP 表达式求值器

我们来实现一个简化版的 PP 表达式求值器，体验 TJS2 预处理器的核心逻辑：

```cpp
/* pp_eval_demo.cpp — PP 表达式求值器模拟 */
#include <iostream>
#include <string>
#include <unordered_map>
#include <cctype>
#include <cstdint>

// PP 变量存储（模拟 PPValues 哈希表）
class PPVariableStore {
    std::unordered_map<std::string, int32_t> values_;
public:
    PPVariableStore() {
        // 预设 version 变量（模拟引擎初始化）

    }

        auto it = values_.find(name);
        if (it != values_.end()) {
            std::cout << "  [PPValues] 读取 " << name << " = " << it->second << std::endl;
            return it->second;

        std::cout << "  [PPValues] 未找到 " << name << "，返回默认值 " << default_val << std::endl;
        return default_val;
    }
    
    void dump() const {
        std::cout << "  [PPValues 状态]" << std::endl;
        for (const auto& [k, v] : values_) {
            std::cout << "    " << k << " = " << v;
            if (k == "version") {
                // 格式化版本号显示
                std::cout << " (0x" << std::hex << v << std::dec << ")";
            }
            std::cout << std::endl;
        }
    }

// 递归下降表达式求值器（模拟 tjspp.y 的功能）
class PPExprEvaluator {
    const char* pos_;          // 当前扫描位置（模拟 Current 指针）
    PPVariableStore& store_;   // PP 变量存储的引用
    std::string last_symbol_;  // 最近读取的符号名（用于赋值）
    
    void skipSpaces() {
        while (*pos_ == ' ' || *pos_ == '\t') pos_++;
    }
    
    // 解析整数字面量
    int32_t parseNumber() {
        int32_t result = 0;
        if (*pos_ == '0' && (*(pos_+1) == 'x' || *(pos_+1) == 'X')) {
            // 十六进制
            pos_ += 2;
            while (std::isxdigit(*pos_)) {
                char c = *pos_++;
                int digit = (c >= '0' && c <= '9') ? c - '0' :
                           (c >= 'a' && c <= 'f') ? c - 'a' + 10 :
                           c - 'A' + 10;
                result = result * 16 + digit;
            }
        } else {
            // 十进制
            while (std::isdigit(*pos_)) {
                result = result * 10 + (*pos_++ - '0');
            }
        }
        return result;
    }
    
    // 解析标识符
    std::string parseIdentifier() {
        std::string name;
        while (std::isalnum(*pos_) || *pos_ == '_') {
            name += *pos_++;
        }
        return name;
    }
    
    // 基本因子：数字、变量、括号、一元运算符
    int32_t parsePrimary() {
        skipSpaces();
        
        if (*pos_ == '(') {
            pos_++;  // 跳过 '('
            int32_t result = parseExpr();
            skipSpaces();
            if (*pos_ == ')') pos_++;  // 跳过 ')'
            return result;
        }
        
        if (*pos_ == '!') {
            pos_++;
            return parsePrimary() ? 0 : 1;
        }
        
        if (*pos_ == '-' && !std::isdigit(*(pos_-1))) {
            pos_++;
            return -parsePrimary();
        }
        
        if (std::isdigit(*pos_)) {
            return parseNumber();
        }
        
        if (std::isalpha(*pos_) || *pos_ == '_') {
            std::string name = parseIdentifier();
            last_symbol_ = name;  // 记录变量名（用于赋值）
            
            skipSpaces();
            if (*pos_ == '=') {
                // 赋值表达式
                pos_++;  // 跳过 '='
                int32_t value = parseExpr();
                store_.set(name, value);
                return value;
            }
            
            return store_.get(name);
        }
        
        throw std::runtime_error("意外的字符: " + std::string(1, *pos_));
    }
    
    // 乘除模
    int32_t parseMulDiv() {
        int32_t left = parsePrimary();
        skipSpaces();
        while (*pos_ == '*' || *pos_ == '/' || *pos_ == '%') {
            char op = *pos_++;
            int32_t right = parsePrimary();
            if (op == '*') left *= right;
            else if (op == '/') {
                if (right == 0) throw std::runtime_error("除零错误");
                left /= right;
            } else {
                if (right == 0) throw std::runtime_error("模零错误");
                left %= right;
            }
            skipSpaces();
        }
        return left;
    }
    
    // 加减
    int32_t parseAddSub() {
        int32_t left = parseMulDiv();
        skipSpaces();
        while (*pos_ == '+' || (*pos_ == '-' && std::isdigit(*(pos_+1)))) {
            char op = *pos_++;
            int32_t right = parseMulDiv();
            if (op == '+') left += right;
            else left -= right;
            skipSpaces();
        }
        return left;
    }
    
    // 比较运算符
    int32_t parseComparison() {
        int32_t left = parseAddSub();
        skipSpaces();
        if (*pos_ == '>' && *(pos_+1) == '=') { pos_ += 2; return (left >= parseAddSub()) ? 1 : 0; }
        if (*pos_ == '<' && *(pos_+1) == '=') { pos_ += 2; return (left <= parseAddSub()) ? 1 : 0; }
        if (*pos_ == '=' && *(pos_+1) == '=') { pos_ += 2; return (left == parseAddSub()) ? 1 : 0; }
        if (*pos_ == '!' && *(pos_+1) == '=') { pos_ += 2; return (left != parseAddSub()) ? 1 : 0; }
        if (*pos_ == '>') { pos_++; return (left > parseAddSub()) ? 1 : 0; }
        if (*pos_ == '<') { pos_++; return (left < parseAddSub()) ? 1 : 0; }
        return left;
    }
    
    // 逻辑与
    int32_t parseLogicalAnd() {
        int32_t left = parseComparison();
        skipSpaces();
        while (*pos_ == '&' && *(pos_+1) == '&') {
            pos_ += 2;
            int32_t right = parseComparison();
            left = (left && right) ? 1 : 0;
            skipSpaces();
        }
        return left;
    }
    
    // 顶层：逻辑或
    int32_t parseExpr() {
        int32_t left = parseLogicalAnd();
        skipSpaces();
        while (*pos_ == '|' && *(pos_+1) == '|') {
            pos_ += 2;
            int32_t right = parseLogicalAnd();
            left = (left || right) ? 1 : 0;
            skipSpaces();
        }
        return left;
    }
    
public:
    PPExprEvaluator(PPVariableStore& store) : store_(store), pos_(nullptr) {}
    
    int32_t evaluate(const std::string& expr) {
        pos_ = expr.c_str();
        return parseExpr();
    }
};

int main() {
    PPVariableStore store;
    PPExprEvaluator eval(store);
    
    std::cout << "=== 实验1：模拟 TJS2 预处理器表达式求值 ===" << std::endl;
    
    // 模拟 @set(debug = 1)
    std::cout << "\n--- @set(debug = 1) ---" << std::endl;
    int32_t r1 = eval.evaluate("debug = 1");
    std::cout << "  结果: " << r1 << std::endl;
    
    // 模拟 @set(target = 2)
    std::cout << "\n--- @set(target = 2) ---" << std::endl;
    int32_t r2 = eval.evaluate("target = 2");
    std::cout << "  结果: " << r2 << std::endl;
    
    // 模拟 @if(debug && target == 2)
    std::cout << "\n--- @if(debug && target == 2) ---" << std::endl;
    int32_t r3 = eval.evaluate("debug && target == 2");
    std::cout << "  条件结果: " << r3 << (r3 ? " (真 → 编译)" : " (假 → 跳过)") << std::endl;
    
    // 模拟 @if(version >= 0x02040000)
    std::cout << "\n--- @if(version >= 0x02040000) ---" << std::endl;
    int32_t r4 = eval.evaluate("version >= 0x02040000");
    std::cout << "  条件结果: " << r4 << (r4 ? " (真 → 编译)" : " (假 → 跳过)") << std::endl;
    
    // 模拟 @if(!debug)
    std::cout << "\n--- @if(!debug) ---" << std::endl;
    int32_t r5 = eval.evaluate("!debug");
    std::cout << "  条件结果: " << r5 << (r5 ? " (真 → 编译)" : " (假 → 跳过)") << std::endl;
    
    // 复杂表达式
    std::cout << "\n--- @if(debug && (target >= 1) && version >= 0x02000000) ---" << std::endl;
    int32_t r6 = eval.evaluate("debug && (target >= 1) && version >= 0x02000000");
    std::cout << "  条件结果: " << r6 << (r6 ? " (真 → 编译)" : " (假 → 跳过)") << std::endl;
    
    // 显示最终变量状态
    std::cout << "\n--- 最终 PPValues 状态 ---" << std::endl;
    store.dump();
    
    return 0;
}
```

### 实验2：模拟条件编译的代码跳过

这个实验模拟 `SkipUntil_endif()` 的嵌套跟踪逻辑：

```cpp
/* skip_demo.cpp — 条件编译跳过逻辑模拟 */
#include <iostream>
#include <string>
#include <vector>
#include <sstream>

// 模拟 SkipUntil_endif() 的行为
class ConditionalCompiler {
    int if_level_ = 0;  // 模拟 IfLevel（正常模式嵌套）
    
public:
    // 处理一个脚本，返回被编译的行
    std::vector<std::string> process(
        const std::string& script,
        bool debug_mode,  // PP 变量 debug 的值
        int platform       // PP 变量 platform 的值
    ) {
        std::vector<std::string> compiled_lines;  // 实际编译的行
        std::istringstream stream(script);
        std::string line;
        int line_num = 0;
        bool skipping = false;
        int skip_nest = 0;  // SkipUntil_endif 的本地嵌套计数
        
        while (std::getline(stream, line)) {
            line_num++;
            
            // 去除前导空白
            size_t start = line.find_first_not_of(" \t");
            std::string trimmed = (start != std::string::npos) ? line.substr(start) : "";
            
            if (trimmed.substr(0, 1) == "@") {
                // 预处理指令
                std::string directive = trimmed.substr(1);
                
                if (skipping) {
                    // 在跳过模式中：只跟踪 @if/@endif 嵌套
                    if (directive.substr(0, 2) == "if") {
                        skip_nest++;
                        std::cout << "  行 " << line_num << ": [跳过模式] 嵌套 @if，skip_nest: "
                                  << (skip_nest-1) << " → " << skip_nest << std::endl;
                    } else if (directive == "endif") {
                        if (skip_nest == 0) {
                            // 找到匹配的 @endif
                            skipping = false;
                            std::cout << "  行 " << line_num << ": [跳过模式] 找到匹配 @endif，恢复正常模式" << std::endl;
                        } else {
                            skip_nest--;
                            std::cout << "  行 " << line_num << ": [跳过模式] 内层 @endif，skip_nest: "
                                      << (skip_nest+1) << " → " << skip_nest << std::endl;
                        }
                    } else {
                        std::cout << "  行 " << line_num << ": [跳过] " << trimmed << std::endl;
                    }
                    continue;
                }
                
                // 正常模式处理
                if (directive.find("if(debug)") != std::string::npos) {
                    if (debug_mode) {
                        if_level_++;
                        std::cout << "  行 " << line_num << ": @if(debug) → 真，IfLevel: "
                                  << (if_level_-1) << " → " << if_level_ << std::endl;
                    } else {
                        skipping = true;
                        skip_nest = 0;
                        std::cout << "  行 " << line_num << ": @if(debug) → 假，进入跳过模式" << std::endl;
                    }
                } else if (directive.find("if(!debug)") != std::string::npos) {
                    if (!debug_mode) {
                        if_level_++;
                        std::cout << "  行 " << line_num << ": @if(!debug) → 真，IfLevel: "
                                  << (if_level_-1) << " → " << if_level_ << std::endl;
                    } else {
                        skipping = true;
                        skip_nest = 0;
                        std::cout << "  行 " << line_num << ": @if(!debug) → 假，进入跳过模式" << std::endl;
                    }
                } else if (directive.find("if(platform==") != std::string::npos) {
                    int target = directive[directive.find("==")+2] - '0';
                    if (platform == target) {
                        if_level_++;
                        std::cout << "  行 " << line_num << ": @if(platform==" << target
                                  << ") → 真，IfLevel: " << (if_level_-1) << " → " << if_level_ << std::endl;
                    } else {
                        skipping = true;
                        skip_nest = 0;
                        std::cout << "  行 " << line_num << ": @if(platform==" << target
                                  << ") → 假，进入跳过模式" << std::endl;
                    }
                } else if (directive == "endif") {
                    if_level_--;
                    std::cout << "  行 " << line_num << ": @endif，IfLevel: "
                              << (if_level_+1) << " → " << if_level_ << std::endl;
                }
            } else if (!skipping && !trimmed.empty()) {
                // 正常代码行：编译
                compiled_lines.push_back(trimmed);
                std::cout << "  行 " << line_num << ": [编译] " << trimmed << std::endl;
            }
        }
        
        if (if_level_ != 0) {
            std::cout << "  错误：IfLevel = " << if_level_ << "，存在未匹配的 @if！" << std::endl;
        }
        
        return compiled_lines;
    }
};

int main() {
    std::string script = R"(
var common = "共用代码";
@if(debug)
    System.inform("调试模式");
    @if(platform==1)
        System.inform("Windows 调试");
    @endif
    @if(platform==2)
        System.inform("Linux 调试");
    @endif
@endif
@if(!debug)
    // 发布模式的优化代码
    optimizeAll();
@endif
var end = "结束";
)";

    std::cout << "=== 实验2：条件编译模拟 ===" << std::endl;
    
    // 场景1：debug=true, platform=Linux(2)
    std::cout << "\n--- 场景1: debug=true, platform=2(Linux) ---" << std::endl;
    ConditionalCompiler c1;
    auto lines1 = c1.process(script, true, 2);
    std::cout << "\n  实际编译的代码行：" << std::endl;
    for (const auto& l : lines1) std::cout << "    " << l << std::endl;
    
    // 场景2：debug=false, platform=1(Windows)
    std::cout << "\n--- 场景2: debug=false, platform=1(Windows) ---" << std::endl;
    ConditionalCompiler c2;
    auto lines2 = c2.process(script, false, 1);
    std::cout << "\n  实际编译的代码行：" << std::endl;
    for (const auto& l : lines2) std::cout << "    " << l << std::endl;
    
    return 0;
}
```

### 实验3：手动模拟 Bison 错误处理流程

这个实验帮助你理解 TJS2 为什么选择 C++ 异常而非 Bison error 令牌：

```cpp
/* error_demo.cpp — 对比两种错误处理策略 */
#include <iostream>
#include <string>
#include <stdexcept>
#include <vector>

// 策略1：模拟 Bison error 令牌的恢复机制
class RecoveryParser {
public:
    struct ParseError {
        int line;
        std::string message;
    };
    
    // 尝试解析多条语句，遇到错误时跳到下一个分号恢复
    std::vector<ParseError> parseWithRecovery(const std::vector<std::string>& statements) {
        std::vector<ParseError> errors;
        
        for (int i = 0; i < (int)statements.size(); i++) {
            const auto& stmt = statements[i];
            std::cout << "  解析行 " << (i+1) << ": " << stmt << std::endl;
            
            // 模拟语法检查
            if (stmt.find("???") != std::string::npos) {
                errors.push_back({i+1, "语法错误：无效令牌 '???'"});
                std::cout << "    → 错误！跳过此语句，继续解析..." << std::endl;
                // 恢复：跳到下一条语句继续
                continue;
            }
            
            // 模拟级联假错误（错误恢复的典型问题）
            if (i > 0 && !errors.empty() && stmt.find("= x +") != std::string::npos) {
                // 前面的错误可能导致变量 x 未被正确解析，产生假错误
                errors.push_back({i+1, "可能的假错误：变量 'x' 未定义（可能由前面的错误导致）"});
                std::cout << "    → 警告：可能是级联错误" << std::endl;
            }
            
            std::cout << "    → 解析成功" << std::endl;
        }
        
        return errors;
    }
};

// 策略2：模拟 TJS2 的 C++ 异常策略
class ExceptionParser {
public:
    struct CompileError : std::runtime_error {
        int line;
        CompileError(int l, const std::string& msg)
            : std::runtime_error(msg), line(l) {}
    };
    
    // 遇到第一个错误就立即停止
    void parseWithException(const std::vector<std::string>& statements) {
        for (int i = 0; i < (int)statements.size(); i++) {
            const auto& stmt = statements[i];
            std::cout << "  解析行 " << (i+1) << ": " << stmt << std::endl;
            
            if (stmt.find("???") != std::string::npos) {
                throw CompileError(i+1, "语法错误：无效令牌 '???'");
            }
            
            std::cout << "    → 解析成功" << std::endl;
        }
    }
};

int main() {
    std::vector<std::string> code = {
        "var x = 10;",          // 正确
        "var y = ??? + 5;",     // 错误
        "var z = x + y;",       // 正确（但 y 未正确定义）
        "var w = z * 2;",       // 正确
    };
    
    std::cout << "=== 策略1：Bison error 恢复（尝试报告所有错误） ===" << std::endl;
    RecoveryParser rp;
    auto errors = rp.parseWithRecovery(code);
    std::cout << "\n  总共 " << errors.size() << " 个错误：" << std::endl;
    for (const auto& e : errors) {
        std::cout << "    行 " << e.line << ": " << e.message << std::endl;
    }
    std::cout << "  编译结果：部分编译（可能不可靠）" << std::endl;
    
    std::cout << "\n=== 策略2：TJS2 C++ 异常（遇到第一个错误即停止） ===" << std::endl;
    ExceptionParser ep;
    try {
        ep.parseWithException(code);
        std::cout << "  编译结果：成功" << std::endl;
    } catch (const ExceptionParser::CompileError& e) {
        std::cout << "\n  编译失败！" << std::endl;
        std::cout << "    行 " << e.line << ": " << e.what() << std::endl;
        std::cout << "  编译结果：失败（无产物，需修复后重新编译）" << std::endl;
    }
    
    std::cout << "\n=== 结论 ===" << std::endl;
    std::cout << "  Bison error 恢复：报告了 " << errors.size() << " 个错误（可能含假错误）" << std::endl;
    std::cout << "  TJS2 异常策略：只报告第 1 个真实错误，避免了级联假错误" << std::endl;
    std::cout << "  对视觉小说引擎而言，异常策略更安全——不会产生半成品编译结果" << std::endl;
    
    return 0;
}
```

编译和运行：

```bash
g++ -std=c++17 -o error_demo error_demo.cpp && ./error_demo
```

## §12 本节小结

1. **TJS2 没有 Bison 错误恢复**：862 行文法中没有任何 `error` 令牌。这是刻意的设计选择，而非遗漏。

2. **`%expect 1` 声明了唯一的移进/归约冲突**：来自悬挂 else 问题。Bison 的默认移进行为就是期望行为（`else` 与最近的 `if` 配对）。

3. **C++ 异常替代错误恢复**：所有编译错误通过 `TJS_eTJSCompileError()` 抛出 C++ 异常，实现"全有或全无"的编译策略。

4. **多层错误检测**：词法阶段检测字符错误、语法阶段检测结构错误、代码生成阶段检测语义错误，三层互补。

5. **预处理器嵌入词法分析器**：不是独立阶段，而是在 `GetToken()` 过程中按需处理 `@` 指令。

6. **三个 PP 指令**：`@set(expr)` 设置变量、`@if(expr)` 条件编译、`@endif` 结束条件块。不支持 `@else`。

7. **tjspp.y 是独立的 Bison 解析器**：仅处理整数表达式，116 行文法，支持 20 种运算符。与主解析器通过 C++ 命名空间隔离。

8. **SkipUntil_endif 用独立的嵌套计数**：不复用词法分析器的 `IfLevel`，使用本地 `nest_level` 变量跟踪跳过区域中的嵌套。

9. **PP 变量仅支持整数**：存储在 `tTJS` 对象的 `PPValues` 哈希表中，预设 `version` 变量。`GetPPValue` 对未定义变量返回默认值 0。

10. **两个 Bison 解析器各司其职**：主解析器处理完整语言，PP 解析器只处理整数表达式。分离的原因是执行时机和类型系统不同。

## §13 练习题与答案

### 题目 1：为什么 TJS2 选择 C++ 异常而不是 Bison 的 error 令牌？

请从至少三个角度分析这个设计决策的原因和影响。

<details>
<summary>查看答案</summary>

TJS2 选择 C++ 异常而非 Bison `error` 令牌的原因：

**1. 使用场景决定**：TJS2 是视觉小说引擎的嵌入式脚本语言。脚本控制场景跳转、变量管理、事件处理等核心逻辑。一个"部分编译"的脚本运行起来可能产生难以调试的状态不一致。"全有或全无"的策略更安全。

**2. 避免级联假错误**：Bison 的错误恢复在跳过令牌后可能导致后续的正确代码被误报为错误（级联错误）。例如，一个变量声明被跳过后，后续使用该变量的所有语句都会报"未定义变量"。只报告第一个错误避免了这个问题。

**3. 资源清理更简洁**：C++ 异常的栈展开机制自动调用析构函数，清理 AST 节点、临时变量等资源。而 Bison 的 `error` 恢复需要手动处理语义值的清理（通过 `%destructor` 指令），增加了维护负担。

**4. 文法复杂度**：正确的错误恢复需要在文法中精心设计同步点（通常在每个语句类规则中添加 `error ';'`）。TJS2 的文法已有 862 行，添加错误恢复规则会进一步增加复杂度。

**影响**：
- 优点：实现简单、无假错误、资源管理安全
- 缺点：每次只能报告一个错误，修复效率稍低（需要多次编译-修复循环）

</details>

### 题目 2：解释以下 TJS2 脚本的预处理行为

```javascript
@set(a = 3)
@set(b = a * 2 + 1)
@if(b > 5)
    var x = "条件为真";
    @if(a == 3)
        var y = "嵌套条件";
    @endif
@endif
@if(b <= 5)
    var z = "条件为假分支";
@endif
```

请回答：(1) 执行完两条 `@set` 后，PPValues 中 `a` 和 `b` 的值分别是多少？(2) 哪些 `var` 语句会被实际编译？(3) `IfLevel` 在各个 `@if`/`@endif` 处的变化是什么？

<details>
<summary>查看答案</summary>

**(1) PPValues 的值：**
- `@set(a = 3)`：`a = 3`
- `@set(b = a * 2 + 1)`：先读取 `a`（值为 3），计算 `3 * 2 + 1 = 7`，所以 `b = 7`

PPValues 状态：`{"version": 0x02040000, "a": 3, "b": 7}`

**(2) 实际编译的语句：**
- `@if(b > 5)`：`7 > 5` 为真 → 编译内部代码
  - `var x = "条件为真";` → **编译** ✓
  - `@if(a == 3)`：`3 == 3` 为真 → 编译内部代码
    - `var y = "嵌套条件";` → **编译** ✓
  - `@endif`
- `@endif`
- `@if(b <= 5)`：`7 <= 5` 为假 → 跳过内部代码
  - `var z = "条件为假分支";` → **跳过** ✗
- `@endif`

最终编译的语句：`var x = "条件为真";` 和 `var y = "嵌套条件";`

**(3) IfLevel 的变化：**

```
@set(a = 3)             // IfLevel: 0（预处理指令不影响 IfLevel）
@set(b = a * 2 + 1)     // IfLevel: 0
@if(b > 5)              // 条件为真 → IfLevel: 0 → 1
    var x = ...          // IfLevel: 1
    @if(a == 3)          // 条件为真 → IfLevel: 1 → 2
        var y = ...      // IfLevel: 2
    @endif               // IfLevel: 2 → 1
@endif                   // IfLevel: 1 → 0
@if(b <= 5)             // 条件为假 → 调用 SkipUntil_endif()
                         //（IfLevel 不变，保持 0）
    var z = ...          // 被跳过
@endif                   // 由 SkipUntil_endif() 处理
                         // IfLevel: 0（不变）
```

</details>

### 题目 3：如果 TJS2 要支持 `@else` 指令，需要修改哪些代码？

请描述需要修改的文件和函数，以及实现的基本思路。

<details>
<summary>查看答案</summary>

要支持 `@else` 指令，需要修改以下部分：

**1. `tjsLex.cpp` 中的 `ProcessPPStatement()`（第 1123-1233 行）：**

添加对 `@else` 的识别：

```cpp
else if (directive == TJS_W("else")) {
    if (IfLevel <= 0) {
        // 没有匹配的 @if
        TJS_eTJSCompileError(TJSPPError, this, Current);
    }
    // 当前 @if 条件为真，遇到 @else 需要跳过 else 分支
    IfLevel--;  // 退出当前 @if 的层级
    SkipUntil_endif();  // 跳到 @endif
}
```

**2. `tjsLex.cpp` 中的 `SkipUntil_endif()`（第 1042-1120 行）：**

需要修改为同时能被 `@else` 停止（当前只被 `@endif` 停止）。有两种方案：

方案 A：添加一个参数控制停止条件：
```cpp
void SkipUntil_endif(bool stop_at_else = false) {
    // ... 现有逻辑 ...
    if (directive == TJS_W("else") && stop_at_else && nest_level == 0) {
        IfLevel++;  // 进入 else 分支的层级
        return;
    }
}
```

方案 B：新增 `SkipUntil_else_or_endif()` 函数。

**3. `@if` 条件为假时的处理修改：**

当 `@if` 条件为假时，不再直接调用 `SkipUntil_endif()`，而是调用 `SkipUntil_else_or_endif()`。如果停在了 `@else`，则进入正常编译模式（`IfLevel++`）；如果停在了 `@endif`，则不增加 `IfLevel`。

**4. 嵌套处理的复杂性：**

这是最棘手的部分。在跳过模式中遇到嵌套的 `@if...@else...@endif`，需要正确处理嵌套计数。`@else` 不增减嵌套计数（它属于当前层级的 `@if`），只有 `@if` 和 `@endif` 才影响嵌套。

**5. 文件末尾检查不需要修改：**

`IfLevel` 检查仍然有效，因为 `@if...@else...@endif` 结构中 `IfLevel` 的增减与 `@if...@endif` 相同。

这个修改的总工作量不大（约 30-50 行代码），但需要仔细处理嵌套场景的边界情况，这也解释了为什么原作者选择了更简单的 `@if(!expr)` 方案。

</details>
