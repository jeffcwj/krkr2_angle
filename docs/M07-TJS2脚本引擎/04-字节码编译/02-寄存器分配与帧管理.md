## §1 寄存器数组的完整内存布局

TJS2 虚拟机是一台**基于寄存器的虚拟机**（Register-based VM，与 Java/Python 等基于栈的虚拟机不同，操作数直接通过寄存器地址寻址，无需频繁 push/pop）。每次函数调用时，VM 会分配一块连续的 `tTJSVariant` 数组作为该函数的"寄存器文件"。理解这块内存的布局，是理解 TJS2 编译器寄存器分配策略的基础。

### 1.1 运行时分配公式

在 `tjsInterCodeExec.cpp` 第 640-648 行，`ExecuteAsFunction` 函数负责为每次函数调用分配寄存器数组：

```cpp
// tjsInterCodeExec.cpp 第 640-648 行
// 计算需要分配的 tTJSVariant 数量
tjs_int num_alloc =
    MaxVariableCount + VariableReserveCount + 1 + MaxFrameCount;

// 从 VariantArrayStack（变体数组栈，一种内存池，避免频繁 new/delete）中分配
tTJSVariant *regs = TJSVariantArrayStack->Allocate(num_alloc);

// ra 指向寄存器数组的"零号寄存器"位置
// 向左（负方向）是局部变量和保留寄存器
// 向右（正方向）是帧临时寄存器
tTJSVariant *ra = regs + MaxVariableCount + VariableReserveCount;
```

这里出现了四个编译期确定的常量。编译器在编译一个函数体时会计算出它们的值，VM 执行时直接使用。

### 1.2 内存布局图

下图展示了一个典型函数的寄存器数组布局：

```
分配的连续内存 (regs)
┌──────────────────────────────────────────────────────────────────────┐
│  局部变量区          │ 保留区        │ this │ 帧临时寄存器区          │
│  MaxVariableCount 个 │ VarReserve 个 │  1个 │ MaxFrameCount 个       │
├──────────────────────┼───────────────┼──────┼────────────────────────┤
│ var[n-1] ... var[0]  │ res[1] res[0] │  %0  │ %1  %2  %3 ... %max   │
│                      │ (%-2)  (%-1)  │      │                        │
└──────────────────────┴───────────────┴──────┴────────────────────────┘
                                       ↑
                                       ra 指针（零号寄存器）
                                       
◄────── 负方向地址（局部变量）──────────┤──── 正方向地址（临时寄存器）────►

寄存器地址编码：
  %0       = ra[0]            → this 对象
  %1, %2   = ra[1], ra[2]    → 帧临时寄存器（FrameBase 起始）
  %-1      = ra[-1]           → 保留寄存器 0（objthis 引用）
  %-2      = ra[-2]           → 保留寄存器 1（objthis-proxy）
  %-3      = ra[-3]           → 第一个函数参数 / 第一个局部变量
  %-4      = ra[-4]           → 第二个函数参数 / 第二个局部变量
  ...
```

### 1.3 各区域的具体用途

| 区域 | 大小（tTJSVariant 个数） | 寄存器地址 | 用途 |
|------|--------------------------|-----------|------|
| 局部变量区 | `MaxVariableCount` | `-(VariableReserveCount+1)` 到 `-(VariableReserveCount+MaxVariableCount)` | 函数参数、var 声明的局部变量 |
| 保留寄存器区 | `VariableReserveCount` | `-1` 到 `-VariableReserveCount` | 系统预留，`%-1` = objthis 引用，`%-2` = objthis-proxy |
| this 寄存器 | 1 | `%0` | 结果返回寄存器 / this 指针 |
| 帧临时区 | `MaxFrameCount` | `%1` 到 `%MaxFrameCount` | 表达式求值过程中的临时值 |

> **关键理解**：编译器中所有寄存器地址都以 `%0`（this 寄存器）为原点。负方向存放"有名字的变量"——先是保留区、然后是局部变量；正方向存放"无名字的临时值"——表达式编译过程中的中间结果。

## §2 VariableReserveCount — 上下文类型与保留寄存器

保留寄存器（Variable Reserve，编译器预留的特殊用途寄存器，数量由上下文类型决定）是寄存器数组中最容易被忽略的一块区域。它在 `%0` 的负方向紧邻 this 寄存器，为 VM 运行时提供两个关键引用。

### 2.1 各上下文类型的 VariableReserveCount

在 `tjsInterCodeGen.cpp` 第 227-254 行的构造函数中，编译器根据当前编译的代码上下文类型（Context Type，指编译器正在编译的代码块种类——是函数体、类体、属性访问器还是顶层脚本）设置 `VariableReserveCount`：

```cpp
// tjsInterCodeGen.cpp 第 227-254 行
switch(Type) {
case ctTopLevel:           VariableReserveCount = 2; break;  // 顶层脚本
case ctFunction:           VariableReserveCount = 2; break;  // 普通函数
case ctExprFunction:       VariableReserveCount = 2; break;  // 匿名函数表达式
case ctClass:              VariableReserveCount = 2; break;  // 类体
case ctPropertySetter:     VariableReserveCount = 2; break;  // 属性 setter
case ctPropertyGetter:     VariableReserveCount = 2; break;  // 属性 getter
case ctSuperClassGetter:   VariableReserveCount = 2; break;  // 父类 getter
case ctProperty:           VariableReserveCount = 0; break;  // 属性声明上下文
}
```

**几乎所有上下文类型都是 2**——唯一例外是 `ctProperty`，它的 `VariableReserveCount` 为 0。

### 2.2 为什么 ctProperty 是 0？

要理解这个例外，需要区分"属性声明上下文"和"属性 getter/setter 上下文"。在 TJS2 中，一个属性定义包含三层结构：

```tjs
class MyClass {
    var _name = "default";    // 这里在 ctClass 上下文中编译
    
    property name {           // 这里进入 ctProperty 上下文
        getter() {            // 这里进入 ctPropertyGetter 上下文（独立函数）
            return _name;
        }
        setter(v) {           // 这里进入 ctPropertySetter 上下文（独立函数）
            _name = v;
        }
    }
}
```

`ctProperty` 上下文本身**不包含任何可执行代码**——它只是 getter 和 setter 的容器。getter 和 setter 各自作为独立函数编译，拥有自己的寄存器数组。因此 `ctProperty` 不需要保留寄存器：它根本不会被 VM 执行。

### 2.3 保留寄存器的具体内容

当 `VariableReserveCount = 2` 时，两个保留寄存器的用途如下：

| 寄存器地址 | 变量名 | 类型 | 用途 |
|-----------|--------|------|------|
| `%-1` (`ra[-1]`) | objthis | `iTJSDispatch2*` | 当前对象的 `this` 引用——方法调用时指向调用者对象 |
| `%-2` (`ra[-2]`) | objthis-proxy | `tTJSObjectProxy*` | 双重派发代理对象——先查 objthis，查不到再查全局对象 |

**objthis-proxy**（双重派发代理，Dual-dispatch Proxy）是 TJS2 的一个独特设计。它包装了 `objthis`（当前对象）和全局对象（Global Object，TJS2 运行时中存放所有顶层声明的根对象），当代码中使用一个名字时：

1. 先在 `objthis` 上查找该属性
2. 找不到则回退到全局对象查找
3. 都找不到才报错

这个机制使得 TJS2 函数中可以直接使用全局变量而无需任何前缀，同时又能被同名的对象属性"遮蔽"——行为类似 JavaScript 的作用域链查找。

```cpp
// tjsInterCodeExec.cpp 中的 tTJSObjectProxy 类（简化）
// tTJSObjectProxy 继承自 tTJSDispatch，实现了 iTJSDispatch2 接口
class tTJSObjectProxy : public tTJSDispatch {
    iTJSDispatch2 *ObjThis;   // 当前对象引用
    iTJSDispatch2 *Global;    // 全局对象引用
public:
    // PropGet 的实现：先查 ObjThis，失败则查 Global
    tjs_error PropGet(tjs_uint32 flag, const tjs_char *membername,
                      tjs_uint32 *hint, tTJSVariant *result,
                      iTJSDispatch2 *objthis) override {
        tjs_error hr = ObjThis->PropGet(flag, membername, hint, result, ObjThis);
        if(TJS_FAILED(hr)) {
            // ObjThis 上找不到，回退到 Global
            hr = Global->PropGet(flag, membername, hint, result, objthis);
        }
        return hr;
    }
};
```

### 2.4 运行时如何设置保留寄存器

在 `ExecuteAsFunction`（`tjsInterCodeExec.cpp` 第 640-700 行）中，分配完寄存器数组后，VM 立即设置保留寄存器：

```cpp
// tjsInterCodeExec.cpp 第 660-680 行（简化）
// ra 已指向寄存器数组的零号位置
ra[-1] = objthis;                          // %-1 = 当前对象引用
ra[-2] = tTJSObjectProxy(objthis, global); // %-2 = 双重派发代理
```

这两个寄存器在函数执行期间保持不变——它们是"只读"的基础设施寄存器，编译器生成的字节码通过 `%-1` 和 `%-2` 来引用当前对象和进行名字查找。

> **关键理解**：`VariableReserveCount` 决定了保留区的大小，从而影响所有局部变量的地址计算。当 `VariableReserveCount = 2` 时，第一个局部变量的地址是 `-(0 + 2 + 1) = -3`，即 `%-3`。如果某天某个上下文类型需要 3 个保留寄存器，那么第一个局部变量就会变成 `%-4`。这就是为什么编译器不硬编码 `-3`，而是使用公式 `-n - VariableReserveCount - 1`。

## §3 FrameBase 与 frame 变量 — 临时寄存器的分配策略

### 3.1 两个容易混淆的概念

在阅读 TJS2 编译器代码时，`FrameBase` 和 `frame` 是最容易混淆的两个名字。它们虽然都和"帧"有关，但性质完全不同：

| 属性 | `FrameBase` | `frame` |
|------|------------|---------|
| 存储位置 | `tTJSInterCodeContext` 的成员变量 | `GenNodeCode` 函数的局部变量 / 引用参数 |
| 生命周期 | 整个编译上下文存在期间 | 每次 `GenNodeCode` 调用期间 |
| 初始值 | 固定为 1（`tjsInterCodeGen.cpp` 第 190 行和第 324 行） | 等于调用者传入的 `frame` 值（通常从 `FrameBase` 开始） |
| 变化规律 | **永远不变** | 随表达式编译过程**单调递增** |
| 含义 | 临时寄存器区的起始位置——"帧的底部在哪里" | 当前可用的下一个临时寄存器——"帧的顶部到哪里了" |

### 3.2 FrameBase 为什么是 1？

```cpp
// tjsInterCodeGen.cpp 第 190 行
FrameBase = 1;

// 第 324 行（CreateExprCode 中重新赋值）
FrameBase = 1;
```

`FrameBase = 1` 意味着临时寄存器从 `%1` 开始分配。`%0` 被跳过，因为它有特殊用途：

- 在函数体内，`%0` 是 `this` 指针
- 在表达式求值完成后，`%0` 用于存放返回值（`VM_SRV %0` 指令将 `%0` 的值设为函数返回值）

因此 `%0` 既不是"空闲的临时寄存器"，也不是"普通的局部变量"——它是 VM 的约定寄存器（Convention Register，虚拟机和编译器之间约定好用途的特殊寄存器）。

### 3.3 frame 变量的"只增不减"模式

`GenNodeCode`（代码生成的核心递归函数，详见上一节）的第一个参数是 `tjs_int &frame`——一个**引用参数**。调用者传入当前可用的帧位置，`GenNodeCode` 在编译子表达式时会递增 `frame` 来分配临时寄存器：

```cpp
// 编译 a + b 的伪代码流程
void GenNodeCode(tjs_int &frame, tTJSExprNode *node, ...) {
    // 进入时 frame = 1（FrameBase）
    
    // 编译左操作数 a → 结果存到 %1（当前 frame）
    tjs_int left = frame;  // left = 1
    GenNodeCode(frame, node->left, ...);
    // frame 可能增加到 2（如果 a 是复杂表达式需要临时值）
    
    // 编译右操作数 b → 结果存到 %2（当前 frame）
    tjs_int right = frame; // right = 2
    frame++;               // frame = 3，预留一个槽位
    GenNodeCode(frame, node->right, ...);
    
    // 生成 ADD %1, %2 → 结果存到 %1
    PutCode(VM_ADD);
    PutCode(left);   // %1
    PutCode(right);  // %2
    
    // 注意：frame 现在是 3，但 %2、%3 已经"用完"了
    // 不会回退 frame——只增不减
}
```

**为什么只增不减？** 因为编译器在第一遍编译时需要确定寄存器数组的最大尺寸。如果 `frame` 可以回退，那么编译器需要追踪"同时活跃的最大临时变量数"——这在存在条件分支、短路求值等情况下会变得复杂。TJS2 采用了一种简单策略：

1. `frame` 只增不减，记录"到目前为止使用过的最大位置"
2. 在每个语句结束后，调用 `ClearFrame` 将 `frame` 重置回 `FrameBase`
3. `ClearFrame` 同时更新 `MaxFrameCount`（记录整个函数编译过程中 `frame` 达到的最大值）

这意味着**每条语句都从 `%1` 开始重新分配临时寄存器**，而 `MaxFrameCount` 记录了所有语句中"最费寄存器的那条语句"需要的数量。

### 3.4 MaxFrameCount 的维护

`MaxFrameCount`（最大帧计数，编译完成后记录的临时寄存器区最大使用量）在编译开始时初始化为 0：

```cpp
// tjsInterCodeGen.cpp 第 194 行
MaxFrameCount = 0;
```

每次 `ClearFrame` 被调用时（详见 §5），它会检查当前 `frame` 是否超过了之前的记录：

```cpp
if(frame - 1 > MaxFrameCount) MaxFrameCount = frame - 1;
```

注意这里是 `frame - 1` 而不是 `frame`，因为 `frame` 指向"下一个可用位置"，而 `MaxFrameCount` 记录的是"已使用的最大索引"。例如，如果某条语句编译后 `frame = 4`，说明使用了 `%1`、`%2`、`%3` 三个临时寄存器，`MaxFrameCount` 被更新为 3。

对于 `ctClass` 上下文，编译器还有一个特殊保证：

```cpp
// tjsInterCodeGen.cpp 第 267 行
if(MaxFrameCount < 1) MaxFrameCount = 1;
```

这确保类体至少有一个临时寄存器可用，即使类体内没有任何表达式。

> **一个类比**：想象你在一张长条纸上从左到右写字。`FrameBase` 是你每行开始写字的位置（总是从最左边开始），`frame` 是笔尖当前位置（只向右移动），`MaxFrameCount` 是你写过的所有行中最长那行的长度。每写完一行（一条语句），你把笔尖拉回最左边重新开始，但记住了最长的那行有多长——最终你需要这个长度来决定纸条应该裁多宽。

## §4 局部变量的负地址编码

### 4.1 地址计算公式

当编译器遇到一个 `var` 声明时，它通过 `AddLocalVariable` 函数（`tjsInterCodeGen.cpp` 第 2845-2883 行）将变量名注册到命名空间栈（Namespace Stack）中，并获得一个**从 0 开始的索引 `n`**。这个索引通过以下公式转换为寄存器地址：

```
寄存器地址 = -(n + VariableReserveCount + 1)
```

在代码中，这个公式通过 `TJS_TO_VM_REG_ADDR` 宏实现：

```cpp
// tjsInterCodeGen.h 第 30-34 行
#define TJS_TO_VM_REG_ADDR(x)  ((tjs_int)(x))
#define TJS_FROM_VM_REG_ADDR(x) ((tjs_int)(x))

// AddLocalVariable 中的使用（第 2865-2870 行）
tjs_int n = Namespace.Add(tTJSString(name));  // 获取命名空间索引
// 带初始值的局部变量：
PutCode(VM_CP);                                         // Copy 指令
PutCode(TJS_TO_VM_REG_ADDR(-n - VariableReserveCount - 1));  // 目标地址
PutCode(TJS_TO_VM_REG_ADDR(init));                      // 初始值来源寄存器
```

让我们用具体例子展示地址计算过程（假设 `VariableReserveCount = 2`）：

```tjs
function example(a, b) {      // 函数声明
    var x = 10;                // 第三个局部变量
    var y = x + 1;             // 第四个局部变量
    {
        var z = 20;            // 块内第五个变量
    }
}
```

| 变量 | Namespace 索引 `n` | 公式 `-n - 2 - 1` | 寄存器地址 | 物理位置 |
|------|--------------------|--------------------|-----------|---------|
| `a`（第 1 个参数） | 0 | `-(0 + 2 + 1)` | `%-3` | `ra[-3]` |
| `b`（第 2 个参数） | 1 | `-(1 + 2 + 1)` | `%-4` | `ra[-4]` |
| `x` | 2 | `-(2 + 2 + 1)` | `%-5` | `ra[-5]` |
| `y` | 3 | `-(3 + 2 + 1)` | `%-6` | `ra[-6]` |
| `z` | 4 | `-(4 + 2 + 1)` | `%-7` | `ra[-7]` |

### 4.2 Namespace.Find 的累积索引

当编译器在代码中遇到一个变量名引用（例如使用 `x`）时，需要通过 `Namespace.Find(name)` 查找该变量的索引。`Find` 的实现在 `tjsNamespace.cpp` 第 130-160 行：

```cpp
// tjsNamespace.cpp 第 130-160 行（简化）
tjs_int tTJSLocalNamespace::Find(const tTJSString &name) {
    // 从最内层作用域向外搜索
    for(tjs_int i = Levels.size() - 1; i >= 0; i--) {
        tTJSLocalSymbolList *list = Levels[i];
        tjs_int lidx = list->Find(name);      // 在当前层级中查找
        if(lidx != -1) {
            // 找到了！返回 累积索引 = 层内索引 + 该层起始偏移
            return lidx + list->GetLocalCountStart();
        }
    }
    return -1;  // 未找到
}
```

**关键细节**：`Find` 返回的是**累积索引**（Cumulative Index，将多层作用域中的变量统一编号的索引），不是某一层内的相对索引。`GetLocalCountStart()` 返回该层以下所有层级的变量总数。

为什么需要累积索引？因为寄存器数组是一块**扁平的连续内存**——不存在"层级"的概念。编译器需要把嵌套作用域中的所有变量映射到同一个一维数组中。累积索引保证了无论变量在哪一层声明，它们在寄存器数组中都有唯一且不冲突的位置。

```
命名空间栈（编译期）          寄存器数组（运行时）
┌─────────────────┐
│ Level 2: {z}    │  z: lidx=0, start=4  → n=4  →  ra[-7]
├─────────────────┤
│ Level 1: {x, y} │  x: lidx=0, start=2  → n=2  →  ra[-5]
│                 │  y: lidx=1, start=2  → n=3  →  ra[-6]
├─────────────────┤
│ Level 0: {a, b} │  a: lidx=0, start=0  → n=0  →  ra[-3]
│                 │  b: lidx=1, start=0  → n=1  →  ra[-4]
└─────────────────┘
```

### 4.3 AddLocalVariable 的完整流程

`AddLocalVariable`（`tjsInterCodeGen.cpp` 第 2845-2883 行）不仅仅是注册变量名，它还区分**局部变量**和**全局变量**的创建方式：

```cpp
// tjsInterCodeGen.cpp 第 2845-2883 行（简化后的核心逻辑）
void tTJSInterCodeContext::AddLocalVariable(const tjs_char *name, tjs_int init) {
    tjs_int base = (Type == ctClass) ? 2 : 1;
    // base 表示"第几层级以上才算局部变量"
    // ctClass 上下文中 Level 0 和 Level 1 是类体本身，Level 2+ 才是局部块
    
    if(Namespace.GetLevel() >= base) {
        // ===== 局部变量路径 =====
        tjs_int n = Namespace.Add(tTJSString(name));  // 注册到命名空间栈
        
        if(init != 0) {
            // 有初始值：生成 VM_CP（Copy，复制指令）
            // 将初始值从临时寄存器复制到局部变量寄存器
            PutCode(VM_CP);
            PutCode(TJS_TO_VM_REG_ADDR(-n - VariableReserveCount - 1));
            PutCode(TJS_TO_VM_REG_ADDR(init));
        } else {
            // 无初始值：生成 VM_CL（Clear，清空指令）
            // 将局部变量寄存器设为 void（TJS2 的 undefined）
            PutCode(VM_CL);
            PutCode(TJS_TO_VM_REG_ADDR(-n - VariableReserveCount - 1));
        }
    } else {
        // ===== 全局/类属性路径 =====
        // 不在命名空间栈中注册，而是通过属性设置
        // VM_SPDS = Set Property Direct by String（通过字符串直接设置属性）
        PutCode(VM_SPDS);
        PutCode(TJS_TO_VM_REG_ADDR(-1));  // %-1 = objthis
        PutData(tTJSVariant(name));        // 属性名
        PutCode(TJS_TO_VM_REG_ADDR(init)); // 值
    }
}
```

**两条路径的区别**：

| 特征 | 局部变量路径 | 全局/属性路径 |
|------|------------|-------------|
| 触发条件 | `Namespace.GetLevel() >= base` | `Namespace.GetLevel() < base` |
| 存储位置 | 寄存器数组（负地址） | 对象属性（通过 `%-1` 访问） |
| 字节码 | `VM_CP` 或 `VM_CL` | `VM_SPDS` |
| 性能 | 极快（数组索引） | 较慢（哈希表查找） |
| 典型场景 | 函数内的 `var x = 10` | 顶层脚本的 `var globalConfig = {}` |

### 4.4 InitLocalVariable — 带初始表达式的完整流程

当 `var x = someExpression` 中的初始值不是简单常量而是一个需要编译的表达式时，编译器使用 `InitLocalVariable`（`tjsInterCodeGen.cpp` 第 2886-2895 行）：

```cpp
// tjsInterCodeGen.cpp 第 2886-2895 行
void tTJSInterCodeContext::InitLocalVariable(const tjs_char *name,
                                              tTJSExprNode *node) {
    tjs_int fr = FrameBase;  // 从帧基址开始（通常是 1）
    
    // 第 1 步：编译初始化表达式，结果存到临时寄存器
    tjs_int resaddr = GenNodeCode(fr, node, TJS_RT_NEEDED, 0, tSubParam());
    
    // 第 2 步：将结果注册为局部变量
    AddLocalVariable(name, resaddr);
    
    // 第 3 步：清除临时寄存器
    ClearFrame(fr);
}
```

这个三步流程揭示了 TJS2 编译器的一个核心模式——**"编译-注册-清除"**：

1. **编译**：调用 `GenNodeCode` 将表达式编译为字节码，结果存在临时寄存器 `resaddr`
2. **注册**：`AddLocalVariable` 生成 `VM_CP` 指令将结果复制到局部变量的负地址寄存器
3. **清除**：`ClearFrame` 重置 `frame` 回 `FrameBase`，释放临时寄存器空间

## §5 ClearFrame 详解 — "什么都不清除"的清除函数

### 5.1 源码分析

`ClearFrame` 是编译器中调用频率最高的函数之一——几乎每条语句编译完成后都会调用。但它的实现可能会让你大吃一惊：

```cpp
// tjsInterCodeGen.cpp 第 2814-2841 行
void tTJSInterCodeContext::ClearFrame(tjs_int &frame, tjs_int base) {
    if(base == -1) base = FrameBase;  // 默认参数处理
    
    // 更新 MaxFrameCount（记录临时寄存器的高水位线）
    if(frame - 1 > MaxFrameCount) MaxFrameCount = frame - 1;
    
    // 根据需要清除的寄存器数量选择策略
    if(frame - base >= 3) {
        // 批量清除路径（3 个以上）
#if 0
        // 【已禁用】生成批量清除指令
        PutCode(VM_CCL);                      // CCL = Clear Continuous Local
        PutCode(TJS_TO_VM_REG_ADDR(base));    // 起始寄存器
        PutCode(frame - base);                // 数量
#endif
        frame = base;  // 仅重置 frame 计数器
    } else {
        // 逐个清除路径（1-2 个）
        while(frame > base) {
            frame--;
#if 0
            // 【已禁用】生成逐个清除指令
            PutCode(VM_CL);                         // CL = Clear Local
            PutCode(TJS_TO_VM_REG_ADDR(frame));     // 要清除的寄存器
#endif
        }
    }
}
```

### 5.2 为什么所有清除字节码都被 `#if 0` 禁用了？

这是 TJS2 的一个重要工程决策。在早期版本中，`ClearFrame` 会为每个不再使用的临时寄存器生成清除指令（将寄存器设为 `void`）。但后来开发者发现这些清除指令是**不必要的开销**：

1. **临时寄存器会被覆写**：下一条语句编译时，`frame` 重新从 `FrameBase` 开始，相同的寄存器位置会被新的值覆写。旧值自然被替换，无需显式清除。

2. **函数退出时统一回收**：当函数返回时，整个寄存器数组被归还给 `VariantArrayStack`（变体数组栈，一种预分配的内存池，见 §1.1）。`tTJSVariant` 的析构函数会处理引用计数和内存释放。

3. **性能收益明显**：去掉这些清除指令后，字节码变短了，执行速度也更快。对于包含大量表达式的函数，减少的指令数量相当可观。

因此，当前版本的 `ClearFrame` **实际只做两件事**：

| 操作 | 代码 | 作用 |
|------|------|------|
| 更新高水位线 | `if(frame - 1 > MaxFrameCount) MaxFrameCount = frame - 1;` | 记录临时寄存器区域需要的最大尺寸 |
| 重置帧指针 | `frame = base;` | 让下一条语句从 `FrameBase` 重新开始分配 |

### 5.3 ClearLocalVariable 同样被禁用

与 `ClearFrame` 类似，`ClearLocalVariable`（`tjsInterCodeGen.cpp` 第 2793-2811 行）也将所有字节码生成代码用 `#if 0` 禁用了：

```cpp
// tjsInterCodeGen.cpp 第 2793-2811 行
void tTJSInterCodeContext::ClearLocalVariable(tjs_int top, tjs_int base) {
    // 当离开一个块作用域时调用
    // top = 当前层级的变量总数，base = 上一层级的变量总数
    if(top - base >= 3) {
#if 0
        PutCode(VM_CCL);
        PutCode(TJS_TO_VM_REG_ADDR(-base - VariableReserveCount - 1));
        PutCode(top - base);
#endif
    } else {
        tjs_int c = top;
        while(c > base) {
            c--;
#if 0
            PutCode(VM_CL);
            PutCode(TJS_TO_VM_REG_ADDR(-c - VariableReserveCount - 1));
#endif
        }
    }
}
```

这意味着**离开块作用域时也不会生成任何清除字节码**。块作用域内声明的局部变量在寄存器中"残留"到函数退出，由 VM 的内存回收机制统一处理。

> **工程启示**：这种"延迟回收"策略在很多虚拟机中都能看到。例如 V8 引擎对局部变量也采用类似策略——在函数退出前不单独清理局部变量。这是"用少量内存浪费换取执行速度"的典型权衡。

## §6 tTJSLocalNamespace 层级栈 — 块作用域的编译期管理

### 6.1 整体架构

`tTJSLocalNamespace`（局部命名空间，编译期管理局部变量名到索引映射的层级结构，定义在 `tjsNamespace.h` 第 60-152 行）是编译器中管理变量作用域的核心组件。它使用一个**层级栈**（Level Stack）来模拟代码中的块嵌套结构：

```cpp
// tjsNamespace.h 第 60-90 行（简化）
class tTJSLocalNamespace {
    std::vector<tTJSLocalSymbolList*> Levels;  // 层级栈——每个元素是一个作用域
    tjs_int CurrentCount;   // 当前所有层级的变量总数
    tjs_int MaxCount;       // 历史最大变量总数（高水位线）
    
public:
    void Push();            // 进入新的块作用域
    void Pop();             // 离开块作用域
    tjs_int Find(const tTJSString &name);   // 查找变量
    tjs_int Add(const tTJSString &name);    // 添加变量
    void Commit();          // 提交当前计数到高水位线
    tjs_int GetLevel() const { return Levels.size(); }
    tjs_int GetMaxCount() const { return MaxCount; }
};
```

### 6.2 Push 和 Pop — 作用域的进出

```cpp
// tjsNamespace.cpp 第 40-65 行
void tTJSLocalNamespace::Push() {
    // 创建新的符号列表，起始偏移 = 当前变量总数
    tTJSLocalSymbolList *list = new tTJSLocalSymbolList(CurrentCount);
    Levels.push_back(list);
}

void tTJSLocalNamespace::Pop() {
    Commit();  // 先提交——更新 MaxCount
    
    // 恢复 CurrentCount 到进入此层之前的值
    tTJSLocalSymbolList *list = Levels.back();
    CurrentCount = list->GetLocalCountStart();
    
    delete list;
    Levels.pop_back();
}
```

`Push` 时记录当前变量总数作为新层的起始偏移（`LocalCountStart`）。`Pop` 时恢复到进入前的状态。这意味着同一个索引可以在不同块中被**重复使用**——例如前一个块的 `z`（索引 4）和后一个块的 `w`（也是索引 4）可以共用同一个寄存器槽位。

### 6.3 Commit 与 MaxCount

```cpp
// tjsNamespace.cpp 第 100-115 行
void tTJSLocalNamespace::Commit() {
    tjs_int count = 0;
    // 遍历所有层级，计算总变量数
    for(auto *level : Levels) {
        count += level->GetCount();
    }
    if(count > MaxCount) MaxCount = count;
}
```

`Commit()` 统计当前所有层级中的变量总数，如果超过历史最大值则更新 `MaxCount`。这个值最终在 `tTJSInterCodeContext::Commit()`（第 994 行）中通过 `MaxVariableCount = Namespace.GetMaxCount()` 传递给运行时，决定寄存器数组中局部变量区的大小。

**为什么需要 MaxCount？** 因为不同块作用域可以复用寄存器槽位。编译器需要知道"最多有多少个局部变量同时存在"，而不是"总共声明了多少个变量"。例如：

```tjs
function test() {
    var a = 1;          // n=0, MaxCount=1
    {
        var b = 2;      // n=1, MaxCount=2
        var c = 3;      // n=2, MaxCount=3
    }                   // Pop: CurrentCount 回退到 1
    {
        var d = 4;      // n=1（复用了 b 的槽位！）, MaxCount 仍为 3
        var e = 5;      // n=2（复用了 c 的槽位）
        var f = 6;      // n=3, MaxCount=4!
    }                   // Pop: CurrentCount 回退到 1
}
// 最终 MaxVariableCount = 4（不是 6）
```

### 6.4 Add 的空洞填充机制

`Namespace.Add` 在当前层级中注册变量时，会优先填充已被 `Remove()` 释放的空洞：

```cpp
// tjsNamespace.cpp 第 170-195 行（简化）
tjs_int tTJSLocalSymbolList::Add(const tTJSString &name) {
    // 先查找空洞（值为 nullptr 的槽位）
    for(tjs_int i = 0; i < List.size(); i++) {
        if(List[i] == nullptr) {
            List[i] = new tTJSString(name);  // 填充空洞
            return i;
        }
    }
    // 没有空洞，追加到末尾
    List.push_back(new tTJSString(name));
    CurrentCount++;  // 更新全局计数
    return List.size() - 1;
}
```

空洞填充保证了**索引的稳定性**——已分配的变量不会因为其他变量被删除而改变位置。这在嵌套作用域中至关重要：内层块的变量被移除后，外层变量的索引不受影响。

### 6.5 EnterBlock / ExitBlock — 编译器的作用域管理

编译器在遇到 `{ }` 块时调用 `EnterBlock` 和 `ExitBlock`（`tjsInterCodeGen.cpp` 第 3653-3680 行）：

```cpp
// tjsInterCodeGen.cpp 第 3653-3680 行
void tTJSInterCodeContext::EnterBlock() {
    Namespace.Push();  // 推入新的作用域层级
    
    tjs_int varcount = Namespace.GetCount();  // 记录进入前的变量数
    
    // 将当前状态压入嵌套栈
    NestVector.push_back(tNestData());
    NestVector.back().Type = ntBlock;         // 块类型
    NestVector.back().VariableCount = varcount;
}

void tTJSInterCodeContext::ExitBlock() {
    tjs_int varcount = NestVector.back().VariableCount;  // 恢复进入前的变量数
    NestVector.pop_back();
    
    tjs_int curcount = Namespace.GetCount();   // 当前变量数
    
    Namespace.Pop();  // 弹出作用域层级
    
    // 清除该块中声明的局部变量（但实际不生成字节码——#if 0）
    ClearLocalVariable(curcount, varcount);
}
```

注意 `ExitBlock` 调用了 `ClearLocalVariable`，但正如 §5.3 所述，该函数当前不生成任何字节码。块作用域退出时，Namespace 层级被弹出（`Pop()`），变量名不再可访问，但对应的寄存器槽位中的值仍然存在——直到被复用或函数退出。

## §7 运行时寄存器使用 — 从调用到执行

### 7.1 函数调用的参数传递

当 VM 执行一个函数调用时，参数通过寄存器数组的负地址区域传递。`ExecuteAsFunction`（`tjsInterCodeExec.cpp` 第 700-728 行）负责将参数从调用者传递到被调用函数的寄存器数组中：

```cpp
// tjsInterCodeExec.cpp 第 700-728 行（简化）
// param = 调用者传入的参数数组
// numparams = 实际传入的参数数量
// ra = 被调用函数的寄存器数组零号位置

if(numparams >= FuncDeclArgCount) {
    // 参数数量 >= 声明的参数数量
    // 复制所有声明的参数
    for(tjs_int i = 0; i < FuncDeclArgCount; i++) {
        ra[-3 - i] = *param[i];  // 第 i 个参数 → ra[-(3+i)]
    }
} else {
    // 参数数量 < 声明的参数数量
    // 复制已传入的参数
    for(tjs_int i = 0; i < numparams; i++) {
        ra[-3 - i] = *param[i];
    }
    // 未传入的参数清空为 void
    for(tjs_int i = numparams; i < FuncDeclArgCount; i++) {
        ra[-3 - i].Clear();     // 设为 void（TJS2 中的 undefined）
    }
}
```

**为什么从 `-3` 开始？** 回顾地址公式：第一个参数的 Namespace 索引是 0，地址 = `-(0 + 2 + 1) = -3`。这里的 2 就是 `VariableReserveCount`（保留寄存器数量），1 是 `%0`（this 寄存器）占据的位置。

### 7.2 可变参数的折叠（Collapse）

TJS2 支持函数声明中使用 `*` 语法收集多余参数（类似 JavaScript 的 rest 参数 `...args`）：

```tjs
function example(a, b, *rest) {
    // rest 是一个数组，包含第三个及之后的所有参数
}
```

VM 在 `ExecuteAsFunction` 中处理参数折叠（Argument Collapse，将多余的位置参数收集到一个数组对象中的过程）：

```cpp
// tjsInterCodeExec.cpp 第 731-744 行（简化）
if(FuncDeclCollapseBase >= 0) {
    // FuncDeclCollapseBase = 带 * 的参数在声明中的位置（0-based）
    // 创建数组对象来存放多余参数
    tjs_int collapse_count = numparams - FuncDeclCollapseBase;
    if(collapse_count < 0) collapse_count = 0;
    
    iTJSDispatch2 *arr = TJSCreateArrayObject();
    for(tjs_int i = 0; i < collapse_count; i++) {
        // 将多余参数添加到数组中
        tTJSVariant val = *param[FuncDeclCollapseBase + i];
        arr->PropSetByNum(TJS_MEMBERENSURE, i, &val, arr);
    }
    
    // 将数组存到折叠参数的寄存器位置
    ra[-3 - FuncDeclCollapseBase] = tTJSVariant(arr, arr);
    arr->Release();
}
```

### 7.3 ExecuteCode 调度循环

参数设置完毕后，`ExecuteAsFunction` 调用 `ExecuteCode`（`tjsInterCodeExec.cpp` 第 780 行起）进入字节码执行循环。此时寄存器数组的布局如下：

```
执行开始时的寄存器状态
┌──────────────────────┬──────────────┬──────┬──────────────┐
│ 参数已就位            │ 保留已设置    │  %0  │ 帧区全 void   │
│ ra[-3]=a, ra[-4]=b   │ ra[-1]=this  │  =0  │ ra[1]...     │
│ 其余 var 槽位=void    │ ra[-2]=proxy │      │ =void        │
└──────────────────────┴──────────────┴──────┴──────────────┘
```

在执行循环中，字节码指令通过寄存器地址直接读写这块内存：

```cpp
// 执行循环伪代码
while(true) {
    tjs_int32 code = *CodeArea++;  // 读取操作码
    switch(code) {
        case VM_CP: {  // Copy: 复制寄存器
            tjs_int dst = *CodeArea++;
            tjs_int src = *CodeArea++;
            ra[dst] = ra[src];     // 通过地址直接索引
            break;
        }
        case VM_ADD: {  // 加法
            tjs_int dst = *CodeArea++;
            tjs_int src = *CodeArea++;
            ra[dst] += ra[src];    // 直接在寄存器上操作
            break;
        }
        case VM_SRV: {  // Set Return Value：设置返回值
            tjs_int src = *CodeArea++;
            ra[0] = ra[src];       // 返回值存到 %0
            break;
        }
        // ... 更多指令
    }
}
```

**关键观察**：所有寄存器访问都是 `ra[offset]` 形式——正数访问帧临时区，负数访问局部变量区和保留区。这就是 §1 中"以 `ra` 为原点"设计的实际体现。

### 7.4 VariantArrayStack — 高效的寄存器数组分配器

每次函数调用都需要分配一个寄存器数组，频繁的 `new`/`delete` 会造成严重的性能问题。TJS2 使用 `tTJSVariantArrayStack`（变体数组栈，一种预分配的内存池，用于避免频繁的堆内存分配）来优化这个过程：

```cpp
// tjsInterCodeExec.cpp 第 578-613 行（简化）
class tTJSVariantArrayStack {
    struct Block {
        tTJSVariant *Array;    // 预分配的大块 tTJSVariant 数组
        tjs_int Used;          // 已使用的元素数
    };
    std::vector<Block*> Blocks;
    
    static const tjs_int TJS_VA_ONE_ALLOC_MAX = 1024;  // 每块最大元素数
    
    tTJSVariant* Allocate(tjs_int num) {
        // 在当前块中分配 num 个 tTJSVariant
        Block *cur = Blocks.back();
        if(cur->Used + num <= TJS_VA_ONE_ALLOC_MAX) {
            tTJSVariant *ret = cur->Array + cur->Used;
            cur->Used += num;
            return ret;
        }
        // 当前块空间不足，分配新块
        Block *newBlock = new Block();
        newBlock->Array = new tTJSVariant[TJS_VA_ONE_ALLOC_MAX];
        newBlock->Used = num;
        Blocks.push_back(newBlock);
        return newBlock->Array;
    }
    
    void Deallocate(tjs_int num) {
        // 简单地回退 Used 指针
        Blocks.back()->Used -= num;
    }
};
```

这是一个**栈式分配器**（Stack Allocator，按后进先出顺序分配和释放的内存管理器）。它利用了函数调用天然的栈结构——最后被调用的函数最先返回。分配只需移动指针，释放也只需回退指针，两者都是 O(1) 操作。

## §8 动手实践

### 练习 1：手动推演寄存器布局

给定以下 TJS2 函数：

```tjs
function calc(x, y) {
    var sum = x + y;
    var diff = x - y;
    return sum * diff;
}
```

**步骤 1**：确定编译期常量。

```
上下文类型：ctFunction
VariableReserveCount = 2

局部变量（按声明顺序）：
  x: Namespace 索引 0  →  地址 -(0+2+1) = -3
  y: Namespace 索引 1  →  地址 -(1+2+1) = -4
  sum: Namespace 索引 2  →  地址 -(2+2+1) = -5
  diff: Namespace 索引 3  →  地址 -(3+2+1) = -6

MaxVariableCount = 4（最多 4 个局部变量同时存在）
FrameBase = 1
```

**步骤 2**：推演字节码生成过程中 `frame` 的变化。

```
// 编译 var sum = x + y
InitLocalVariable("sum", node_add(x, y)):
  fr = FrameBase = 1
  GenNodeCode(fr, add_node):
    编译左操作数 x → 结果在 %-3（直接引用，不消耗 frame）
    编译右操作数 y → 结果在 %-4（直接引用）
    生成: VM_ADD %1, %-3, %-4（结果存到帧临时 %1）
    frame = 2
  resaddr = 1（%1）
  AddLocalVariable("sum", 1):
    生成: VM_CP %-5, %1   （将 %1 复制到 sum 的寄存器）
  ClearFrame(fr):
    MaxFrameCount = max(0, 2-1) = 1
    frame = 1（重置）

// 编译 var diff = x - y（流程类似）
  frame 变化: 1 → 2 → 重置到 1
  MaxFrameCount = max(1, 1) = 1

// 编译 return sum * diff
  GenNodeCode(fr, mul_node):
    编译 sum → %-5
    编译 diff → %-6
    生成: VM_MUL %1, %-5, %-6
    frame = 2
  生成: VM_SRV %0, %1（返回值存到 %0）
  ClearFrame: MaxFrameCount = max(1, 1) = 1

最终: MaxFrameCount = 1
```

**步骤 3**：计算运行时分配大小。

```
num_alloc = MaxVariableCount + VariableReserveCount + 1 + MaxFrameCount
         = 4 + 2 + 1 + 1
         = 8

分配 8 个 tTJSVariant：
  [var3][var2][var1][var0][res1][res0][this][tmp0]
    ↑diff  ↑sum   ↑y    ↑x   ↑proxy ↑obj  ↑%0   ↑%1
    %-6    %-5    %-4   %-3   %-2    %-1         
```

### 练习 5：完整的函数调用与参数传递流程

```tjs

}

```

**VM 执行流程**：

```
1. 调用者准备参数：
   param[0] = "World"
   param[1] = "Hello"
   numparams = 2

2. ExecuteAsFunction 为 greet 分配寄存器数组：
   num_alloc = MaxVariableCount(2) + VariableReserveCount(2) + 1 + MaxFrameCount(N)
   
3. 设置保留寄存器：
   ra[-1] = objthis（调用者对象）
   ra[-2] = tTJSObjectProxy(objthis, global)

4. 传递参数（因为 numparams == FuncDeclArgCount == 2）：
   ra[-3] = "World"    // name 参数
   ra[-4] = "Hello"    // greeting 参数

5. 执行字节码：
   // greeting + ", " → 临时结果存 %1
   VM_ADD %1, %-4, #", "
   // %1 + name → 结果存 %1
   VM_ADD %1, %1, %-3
   // %1 + "!" → 结果存 %1
   VM_ADD %1, %1, #"!"
   // 设置返回值
   VM_SRV %0, %1
   // 返回
   VM_RET

6. 函数返回后：
   调用者从返回值中获取 "Hello, World!"
   寄存器数组归还给 VariantArrayStack
```

## §10 本节小结

- **寄存器数组**是一块连续内存，以 `ra`（零号寄存器）为原点，负方向存放局部变量和保留寄存器，正方向存放临时寄存器。总大小 = `MaxVariableCount + VariableReserveCount + 1 + MaxFrameCount`
- **VariableReserveCount** 几乎总是 2（唯一例外是 `ctProperty` 为 0），保留 `%-1`（objthis）和 `%-2`（objthis-proxy 双重派发代理）
- **FrameBase** 固定为 1（跳过 `%0` 约定寄存器），`frame` 变量在编译每条语句时单调递增，语句结束后由 `ClearFrame` 重置
- **MaxFrameCount** 是 `frame` 的高水位线，记录所有语句中最大的临时寄存器使用量
- **局部变量地址** = `-(n + VariableReserveCount + 1)`，其中 `n` 是 Namespace 的累积索引
- **ClearFrame 和 ClearLocalVariable 当前不生成任何字节码**——所有 `VM_CL`/`VM_CCL` 代码被 `#if 0` 禁用，VM 依赖函数退出时统一回收
- **tTJSLocalNamespace** 使用层级栈管理块作用域，支持 Push/Pop/Find/Add，通过 MaxCount 高水位线决定局部变量区大小
- **VariantArrayStack** 是栈式内存池，O(1) 分配和释放寄存器数组

## §11 练习题与答案

### 题目 1：寄存器地址计算

给定以下代码，在 `ctFunction` 上下文中编译：

```tjs
function process(input, config) {
    var result = "";
    {
        var temp = input + config;
        result = temp;
    }
    return result;
}
```

**问题**：
1. 每个变量的寄存器地址是什么？
2. `MaxVariableCount` 是多少？
3. 块退出后 `temp` 的寄存器槽位会发生什么？

<details>
<summary>查看答案</summary>

**1. 变量寄存器地址**（VariableReserveCount = 2）：

| 变量 | Namespace 索引 | 地址公式 | 寄存器地址 |
|------|---------------|---------|-----------|
| `input` | 0 | -(0+2+1) | `%-3` |
| `config` | 1 | -(1+2+1) | `%-4` |
| `result` | 2 | -(2+2+1) | `%-5` |
| `temp` | 3 | -(3+2+1) | `%-6` |

**2. MaxVariableCount = 4**

Namespace 操作序列：
- Push Level 0 (start=0)
- Add "input" → CurrentCount=1
- Add "config" → CurrentCount=2
- Add "result" → CurrentCount=3
- Push Level 1 (start=3)
- Add "temp" → CurrentCount=4
- Pop Level 1 → Commit: MaxCount=max(0,4)=4, CurrentCount=3
- Pop Level 0 → Commit: MaxCount=max(4,3)=4

虽然最后只有 3 个变量活跃（input, config, result），但由于 temp 存在过，MaxVariableCount = 4。

**3. 块退出后 temp 的寄存器槽位**

`ExitBlock` 调用 `ClearLocalVariable(4, 3)`，但由于所有清除字节码被 `#if 0` 禁用，`ra[-6]` 中仍然保留着 `temp` 最后的值（即 `input + config` 的结果）。这个值会一直存在直到函数返回，寄存器数组被 `VariantArrayStack` 回收时才被清除。

在编译层面，`Namespace.Pop()` 之后 `temp` 这个名字不再可查找（`Find("temp")` 返回 -1），所以不可能生成引用 `%-6` 的字节码——变量虽然"残留"在寄存器中，但在语义上已经不可访问。

</details>

### 题目 2：ClearFrame 的 MaxFrameCount 更新

考虑以下编译过程中 `frame` 的变化序列：

```
语句 1: frame 从 1 增长到 3，然后 ClearFrame
语句 2: frame 从 1 增长到 6，然后 ClearFrame
语句 3: frame 从 1 增长到 2，然后 ClearFrame
语句 4: frame 从 1 增长到 4，然后 ClearFrame
```

**问题**：
1. 每次 ClearFrame 后 MaxFrameCount 是多少？
2. 最终运行时帧临时区需要多少个 tTJSVariant？
3. 为什么不在每条语句后"收缩" MaxFrameCount？

<details>
<summary>查看答案</summary>

**1. 每次 ClearFrame 后的 MaxFrameCount**：

| 时刻 | frame 到达 | ClearFrame 计算 frame-1 | MaxFrameCount 更新 |
|------|-----------|------------------------|-------------------|
| 语句 1 后 | 3 | 3-1=2 | max(0, 2) = **2** |
| 语句 2 后 | 6 | 6-1=5 | max(2, 5) = **5** |
| 语句 3 后 | 2 | 2-1=1 | max(5, 1) = **5**（不变） |
| 语句 4 后 | 4 | 4-1=3 | max(5, 3) = **5**（不变） |

**2. 帧临时区需要 5 个 tTJSVariant**（%1 到 %5）。

**3. 为什么不收缩？**

因为 `MaxFrameCount` 的目的是决定**运行时一次性分配多大的寄存器数组**。寄存器数组在函数进入时分配、退出时释放，中间不会调整大小。所以编译器必须记录"最坏情况"（高水位线），确保最复杂的那条语句也有足够的临时空间。

如果允许收缩，那么运行时分配的数组可能不够大——当执行到语句 2 时就会发生数组越界。

</details>

### 题目 3：设计一个简化版寄存器分配器

请用 C++ 实现一个简化版的寄存器分配模拟器，模拟 TJS2 的 FrameBase/frame/ClearFrame 机制：

```cpp
#include <iostream>
#include <string>
#include <vector>
#include <cassert>

// 简化版寄存器分配器
// 模拟 TJS2 的帧临时寄存器分配策略
class SimpleRegAllocator {
    int FrameBase;       // 临时寄存器区的起始位置（固定为 1）
    int frame;           // 当前可用的下一个临时寄存器
    int MaxFrameCount;   // 历史最大使用量（高水位线）
    
    // 编译日志——记录每次分配和清除操作
    std::vector<std::string> log;
    
public:
    SimpleRegAllocator() : FrameBase(1), frame(1), MaxFrameCount(0) {
        log.push_back("初始化: FrameBase=1, frame=1, MaxFrameCount=0");
    }
    
    // 分配一个临时寄存器（模拟 GenNodeCode 中 frame++ 的操作）
    int AllocTemp() {
        int reg = frame;
        frame++;
        log.push_back("分配临时寄存器 %" + std::to_string(reg)
                       + ", frame 递增到 " + std::to_string(frame));
        return reg;
    }
    
    // 清除帧（模拟 ClearFrame）
    void ClearFrame() {
        if(frame - 1 > MaxFrameCount) {
            MaxFrameCount = frame - 1;
        }
        log.push_back("ClearFrame: frame 从 " + std::to_string(frame)
                       + " 重置到 " + std::to_string(FrameBase)
                       + ", MaxFrameCount=" + std::to_string(MaxFrameCount));
        frame = FrameBase;
    }
    
    int GetMaxFrameCount() const { return MaxFrameCount; }
    int GetCurrentFrame() const { return frame; }
    
    // 打印编译日志
    void PrintLog() const {
        std::cout << "=== 寄存器分配日志 ===" << std::endl;
        for(const auto &entry : log) {
            std::cout << "  " << entry << std::endl;
        }
        std::cout << "最终 MaxFrameCount = " << MaxFrameCount << std::endl;
    }
};

int main() {
    SimpleRegAllocator alloc;
    
    // 模拟编译语句 1: var x = a + b
    std::cout << "--- 编译: var x = a + b ---" << std::endl;
    int t1 = alloc.AllocTemp();  // %1: 存放 a + b 的结果
    // 生成 VM_ADD %1, %-3, %-4（此处只模拟分配，不生成真正字节码）
    // 生成 VM_CP %-5, %1（将结果复制到局部变量 x）
    alloc.ClearFrame();
    
    // 模拟编译语句 2: var y = (a * b) + (a - b) + c
    std::cout << "--- 编译: var y = (a * b) + (a - b) + c ---" << std::endl;
    int t2 = alloc.AllocTemp();  // %1: a * b
    int t3 = alloc.AllocTemp();  // %2: a - b
    int t4 = alloc.AllocTemp();  // %3: %1 + %2
    int t5 = alloc.AllocTemp();  // %4: %3 + c（最终结果）
    alloc.ClearFrame();
    
    // 模拟编译语句 3: return x
    std::cout << "--- 编译: return x ---" << std::endl;
    int t6 = alloc.AllocTemp();  // %1: 加载 x
    // 生成 VM_SRV %0, %1
    alloc.ClearFrame();
    
    alloc.PrintLog();
    
    // 验证结果
    assert(alloc.GetMaxFrameCount() == 4);
    std::cout << "\n验证通过: MaxFrameCount = 4（由语句 2 决定）" << std::endl;
    std::cout << "运行时需要分配 4 个临时寄存器槽位（%1 到 %4）" << std::endl;
    
    return 0;
}
```

**编译并运行**：

```bash
# Linux / macOS
g++ -std=c++17 -o reg_alloc reg_alloc.cpp && ./reg_alloc

# Windows (MSVC)
cl /std:c++17 /EHsc reg_alloc.cpp && reg_alloc.exe
```

**预期输出**：

```
--- 编译: var x = a + b ---
--- 编译: var y = (a * b) + (a - b) + c ---
--- 编译: return x ---
=== 寄存器分配日志 ===
  初始化: FrameBase=1, frame=1, MaxFrameCount=0
  分配临时寄存器 %1, frame 递增到 2
  ClearFrame: frame 从 2 重置到 1, MaxFrameCount=1
  分配临时寄存器 %1, frame 递增到 2
  分配临时寄存器 %2, frame 递增到 3
  分配临时寄存器 %3, frame 递增到 4
  分配临时寄存器 %4, frame 递增到 5
  ClearFrame: frame 从 5 重置到 1, MaxFrameCount=4
  分配临时寄存器 %1, frame 递增到 2
  ClearFrame: frame 从 2 重置到 1, MaxFrameCount=4
最终 MaxFrameCount = 4

验证通过: MaxFrameCount = 4（由语句 2 决定）
运行时需要分配 4 个临时寄存器槽位（%1 到 %4）
```

<details>
<summary>查看扩展思考：如何改进这个分配器？</summary>

TJS2 的"只增不减"策略虽然简单高效，但会浪费一些临时寄存器空间。例如在上面的语句 2 中，`%1`（a*b 的结果）在被加到 `%3` 后就不再需要了，理论上可以回收。

更精细的寄存器分配算法（如线性扫描分配、图着色分配）可以识别出寄存器的"活跃区间"（Live Range，变量从定义到最后使用之间的代码范围），在变量不再活跃时立即回收其寄存器。但这需要额外的编译期分析，增加了编译器的复杂度。

TJS2 选择简单策略的原因是：
1. TJS2 是脚本语言，编译速度比运行时的极致优化更重要
2. 大多数语句的临时寄存器使用量都不大（通常 1-5 个）
3. 寄存器数组从内存池分配，浪费几个槽位的代价很小

</details>
