## §1 数据区（Data Area）的结构与 PutData

### 1.1 为什么需要数据区？

TJS2 的字节码是纯整数流——`CodeArea` 是一个 `tjs_int32` 数组。但程序中不可避免地要使用字符串、浮点数、函数对象等非整数类型的常量值。数据区（Data Area，编译期收集的所有常量值的数组）就是为解决这个问题而设计的：

- 常量值存放在数据区 `_DataArea`（编译期）/ `DataArea`（运行时）中
- 字节码通过**索引**引用数据区中的常量——例如 `VM_CONST %1, #3` 表示"将数据区第 3 项的值加载到寄存器 %1"

这种设计使得字节码流保持简洁的整数格式，同时又能引用任意类型的常量值。

### 1.2 编译期数据区 _DataArea

编译期的数据区是一个**指针数组**，每个元素指向一个堆上分配的 `tTJSVariant`：

```cpp
// tjsInterCodeGen.h 中的成员变量
tTJSVariant **_DataArea;      // 指针数组（编译期使用）
tjs_int _DataAreaSize;        // 当前元素数量
tjs_int _DataAreaCapa;        // 数组容量

// 运行时数据区（Commit 后使用）
tTJSVariant *DataArea;        // 值数组（连续内存）
tjs_int DataAreaSize;         // 元素数量
```

为什么编译期使用指针数组而不是直接的值数组？因为编译过程中数据区需要频繁扩容（`TJS_realloc`），如果存储值本身，`realloc` 会涉及 `tTJSVariant` 对象的内存移动（memcpy），但 `tTJSVariant` 可能持有引用计数的对象指针——简单的 memcpy 不会调用拷贝构造函数，可能导致引用计数错误。使用指针数组，扩容时只移动指针，不触及实际的 `tTJSVariant` 对象。

### 1.3 PutData 函数详解

`PutData`（`tjsInterCodeGen.cpp` 第 638-682 行）是编译器向数据区添加常量的核心函数：

```cpp
// tjsInterCodeGen.cpp 第 638-682 行
tjs_int tTJSInterCodeContext::PutData(const tTJSVariant &val) {
    // 第 1 步：检查并扩容
    if(_DataAreaSize >= _DataAreaCapa) {
        _DataArea = (tTJSVariant **)TJS_realloc(
            _DataArea,
            sizeof(tTJSVariant *) * (_DataAreaCapa + TJS_INC_SIZE));
        if(!_DataArea)
            TJS_eTJSScriptError(TJSInsufficientMem, Block, LEX_POS);
        _DataAreaCapa += TJS_INC_SIZE;  // TJS_INC_SIZE = 256
    }

    // 第 2 步：去重——在已有数据中查找相同值
    if(_DataAreaSize) {
        tTJSVariant **ptr = _DataArea + _DataAreaSize - 1;  // 从末尾开始
        tjs_int count = 0;
        while(count < 20) {       // 最多回溯 20 项
            if((*ptr)->DiscernCompareStrictReal(val)) {
                return (tjs_int)(ptr - _DataArea);  // 找到相同值，返回已有索引
            }
            count++;
            if(ptr == _DataArea) break;  // 到达数组头部
            ptr--;
        }
    }

    // 第 3 步：创建新的常量值
    tTJSVariant *v;
    if(val.Type() == tvtString) {
        // 字符串常量使用全局字符串映射表共享
        v = new tTJSVariant(TJSMapGlobalStringMap(val));
    } else {
        v = new tTJSVariant(val);
    }
    _DataArea[_DataAreaSize] = v;

    return _DataAreaSize++;
}
```

### 1.4 去重策略的工程权衡

PutData 的去重算法有一个显眼的限制——**最多回溯 20 项**。源码中的注释 `is waste of time if it exceeds 20 limit?` 透露了设计者的思考：

| 策略 | 优点 | 缺点 |
|------|------|------|
| 不去重 | 最简单，PutData 为 O(1) | 数据区膨胀，浪费内存 |
| 全量去重（遍历整个数据区） | 数据区最小 | PutData 为 O(n)，大脚本编译缓慢 |
| **回溯 20 项**（TJS2 实际方案） | **近似 O(1)，局部去重** | **极远距离的重复常量无法去重** |
| 哈希表去重 | O(1) 查找，完美去重 | 额外内存开销，实现更复杂 |

为什么"回溯 20 项"是一个好的折中？因为在实际代码中，同一个常量的重复使用通常集中在相邻的代码区域。例如：

```tjs
// 同一个函数中多次使用 "name" 字符串
obj.name = "default";      // PutData("name"), PutData("default")
var n = obj.name;          // PutData("name") → 回溯找到，复用！
System.inform(obj.name);   // PutData("name") → 回溯找到，复用！
```

20 项的回溯窗口足以覆盖绝大多数局部重复。而真正相隔很远的重复常量（例如文件开头和结尾各出现一次的字符串），它们的重复存储开销在实践中可以忽略。

### 1.5 DiscernCompareStrictReal — 严格值比较

去重比较使用的是 `DiscernCompareStrictReal` 方法，它与普通的 `==` 比较有重要区别：

- **类型必须相同**：整数 `1` 和浮点数 `1.0` 被视为**不同的常量**
- **字符串按内容比较**：不是比较指针，而是逐字符比较
- **对象按引用比较**：两个不同的函数对象即使代码相同也被视为不同

这种严格比较确保了数据区的索引在 VM 执行时语义正确——不会因为去重而混淆类型不同但值"相近"的常量。

### 1.6 字符串的全局共享优化

当常量值是字符串时，PutData 使用 `TJSMapGlobalStringMap` 进行全局字符串池化（String Interning，将相同内容的字符串映射到同一个内存地址，节省内存并加速比较）：

```cpp
if(val.Type() == tvtString) {
    v = new tTJSVariant(TJSMapGlobalStringMap(val));
} else {
    v = new tTJSVariant(val);
}
```

`TJSMapGlobalStringMap` 在全局字符串映射表（Global String Map，整个 TJS2 引擎共享的字符串去重表）中查找是否已有相同内容的字符串。如果有，则共享同一份底层内存；如果没有，则注册并共享。这意味着即使两个不同函数中都使用了字符串 `"hello"`，底层只有一份 `"hello"` 的字符内存。

## §2 PutCode — 字节码的写入

### 2.1 代码区 CodeArea

与数据区类似，代码区也是一个动态扩容的数组：

```cpp
// tjsInterCodeGen.h 中的成员变量
tjs_int32 *CodeArea;       // 字节码数组
tjs_int CodeAreaSize;      // 当前字节码数量
tjs_int CodeAreaCapa;      // 数组容量
```

### 2.2 PutCode 函数

`PutCode`（`tjsInterCodeGen.cpp` 第 584-632 行）负责将一个 32 位整数写入代码区。除了基本的写入，它还维护**源码位置映射**（Source Position Mapping，将字节码地址映射到源码行号的对应关系，用于错误报告和调试）：

```cpp
// tjsInterCodeGen.cpp 第 584-632 行（简化）
tjs_int tTJSInterCodeContext::PutCode(tjs_int32 num, tjs_int32 pos) {
    // 第 1 步：扩容检查
    if(CodeAreaSize >= CodeAreaCapa) {
        CodeArea = (tjs_int32 *)TJS_realloc(
            CodeArea, sizeof(tjs_int32) * (CodeAreaCapa + TJS_INC_SIZE));
        if(!CodeArea)
            TJS_eTJSScriptError(TJSInsufficientMem, Block, pos);
        CodeAreaCapa += TJS_INC_SIZE;  // 每次增长 256 个 int32
    }

    // 第 2 步：记录源码位置映射
    if(pos != -1) {
        if(PrevSourcePos != pos) {
            PrevSourcePos = pos;
            SourcePosArraySorted = false;
            // 记录：字节码地址 CodeAreaSize → 源码位置 pos
            SourcePosArray[SourcePosArraySize].CodePos = CodeAreaSize;
            SourcePosArray[SourcePosArraySize].SourcePos = pos;
            SourcePosArraySize++;
        }
    }

    // 第 3 步：写入字节码
    CodeArea[CodeAreaSize] = num;
    return CodeAreaSize++;
}
```

**源码位置映射的用途**：当 VM 执行出错时（例如类型错误、空引用），需要告诉用户"错误发生在源码第 XX 行"。`SourcePosArray` 将每个字节码地址关联到对应的源码位置，使得 VM 可以从当前执行的字节码地址反查源码行号。

### 2.3 PutCode 与 PutData 的协作

在编译过程中，`PutCode` 和 `PutData` 配合工作。以 `VM_CONST`（加载常量到寄存器）指令为例：

```cpp
// 编译表达式 var x = "hello"
// 第 1 步：将字符串 "hello" 存入数据区
tTJSVariant val("hello");
tjs_int dataIdx = PutData(val);  // 返回数据区索引，比如 5

// 第 2 步：生成字节码
PutCode(VM_CONST);                          // 操作码
PutCode(TJS_TO_VM_REG_ADDR(frame));         // 目标寄存器（如 %1）
PutCode(TJS_TO_VM_REG_ADDR(dataIdx));       // 数据区索引（5）
```

VM 执行 `VM_CONST %1, #5` 时，从 `DataArea[5]` 读取 `"hello"` 并存到寄存器 `%1`。

## §3 常量折叠（Constant Folding）

### 3.1 什么是常量折叠？

常量折叠（Constant Folding，编译期直接计算常量表达式的值，将结果替换原表达式，减少运行时计算开销）是编译器优化中最基础也最有效的技术之一。例如：

```tjs
var x = 2 + 3;        // 编译器直接计算为 var x = 5
var y = "hello" + " world";  // 编译器直接计算为 var y = "hello world"
var z = !true;         // 编译器直接计算为 var z = false
```

### 3.2 MakeNP1 — 一元运算的常量折叠

`MakeNP1`（Make Node with 1 Parameter，创建一元运算 AST 节点的工厂函数，`tjsInterCodeGen.cpp` 第 3812-3885 行）在创建 AST 节点时检查操作数是否为常量。如果是，则在编译期直接计算结果：

```cpp
// tjsInterCodeGen.cpp 第 3812-3885 行（简化核心逻辑）
tTJSExprNode *tTJSInterCodeContext::MakeNP1(tjs_int opecode,
                                             tTJSExprNode *node1) {
#ifndef TJS_NO_CONSTANT_FOLDING
    // 检查：操作数是常量值节点吗？
    if(node1 && node1->GetOpcode() == T_CONSTVAL) {
        tTJSExprNode *ret = nullptr;
        
        switch(opecode) {
            case T_EXCRAMATION:    // ! 逻辑取反
                ret = MakeConstValNode(!node1->GetValue());
                break;
            case T_TILDE:          // ~ 按位取反
                ret = MakeConstValNode(~node1->GetValue());
                break;
            case T_SHARP:          // # 字符编码转换（字符→数值）
            {
                tTJSVariant val(node1->GetValue());
                CharacterCodeOf(val);
                ret = MakeConstValNode(val);
            } break;
            case T_DOLLAR:         // $ 数值转字符
            {
                tTJSVariant val(node1->GetValue());
                CharacterCodeFrom(val);
                ret = MakeConstValNode(val);
            } break;
            case T_UPLUS:          // +x 一元正号（转数值）
            {
                tTJSVariant val(node1->GetValue());
                val.tonumber();
                ret = MakeConstValNode(val);
            } break;
            case T_UMINUS:         // -x 一元负号（取反）
            {
                tTJSVariant val(node1->GetValue());
                val.changesign();
                ret = MakeConstValNode(val);
            } break;
            case T_INT:            // int(x) 转整数
            {
                tTJSVariant val(node1->GetValue());
                val.ToInteger();
                ret = MakeConstValNode(val);
            } break;
            case T_REAL:           // real(x) 转浮点数
            {
                tTJSVariant val(node1->GetValue());
                val.ToReal();
                ret = MakeConstValNode(val);
            } break;
            case T_STRING:         // string(x) 转字符串
            {
                tTJSVariant val(node1->GetValue());
                val.ToString();
                ret = MakeConstValNode(val);
            } break;
            case T_OCTET:          // (octet) 转字节序列
            {
                tTJSVariant val(node1->GetValue());
                val.ToOctet();
                ret = MakeConstValNode(val);
            } break;
        }
        
        if(ret) {
            node1->Clear();  // 释放原节点持有的数据
            return ret;       // 返回常量值节点
        }
    }
#endif
    // 不能折叠——创建正常的运算节点
    tTJSExprNode *n = new tTJSExprNode;
    n->SetOpcode(opecode);
    n->Add(node1);
    return n;
}
```

### 3.3 MakeNP2 — 二元运算的常量折叠

`MakeNP2`（Make Node with 2 Parameters，`tjsInterCodeGen.cpp` 第 3888-3980 行）处理二元运算的常量折叠。触发条件是**两个操作数都是常量值**：

```cpp
// tjsInterCodeGen.cpp 第 3888-3980 行（简化核心逻辑）
tTJSExprNode *tTJSInterCodeContext::MakeNP2(tjs_int opecode,
                                             tTJSExprNode *node1,
                                             tTJSExprNode *node2) {
#ifndef TJS_NO_CONSTANT_FOLDING
    if(node1 && node1->GetOpcode() == T_CONSTVAL &&
       node2 && node2->GetOpcode() == T_CONSTVAL) {
        switch(opecode) {
            case T_COMMA:        return MakeConstValNode(node2->GetValue());
            case T_LOGICALOR:    return MakeConstValNode(v1 || v2);
            case T_LOGICALAND:   return MakeConstValNode(v1 && v2);
            case T_VERTLINE:     return MakeConstValNode(v1 | v2);   // 按位或
            case T_CHEVRON:      return MakeConstValNode(v1 ^ v2);   // 按位异或
            case T_AMPERSAND:    return MakeConstValNode(v1 & v2);   // 按位与
            case T_NOTEQUAL:     return MakeConstValNode(v1 != v2);
            case T_EQUALEQUAL:   return MakeConstValNode(v1 == v2);
            case T_DISCNOTEQUAL: return MakeConstValNode(!v1.DiscernCompare(v2));
            case T_DISCEQUAL:    return MakeConstValNode(v1.DiscernCompare(v2));
            case T_LT:           return MakeConstValNode(v1 < v2);
            case T_GT:           return MakeConstValNode(v1 > v2);
            case T_LTOREQUAL:    return MakeConstValNode(v1 <= v2);
            case T_GTOREQUAL:    return MakeConstValNode(v1 >= v2);
            case T_RARITHSHIFT:  return MakeConstValNode(v1 >> v2);
            case T_LARITHSHIFT:  return MakeConstValNode(v1 << v2);
            case T_RBITSHIFT:    return MakeConstValNode(v1.rbitshift(v2));
            case T_PLUS:         return MakeConstValNode(v1 + v2);
            case T_MINUS:        return MakeConstValNode(v1 - v2);
            case T_PERCENT:      return MakeConstValNode(v1 % v2);
            case T_SLASH:        return MakeConstValNode(v1 / v2);
            case T_BACKSLASH:    return MakeConstValNode(v1.idiv(v2));  // 整除
            case T_ASTERISK:     return MakeConstValNode(v1 * v2);
        }
    }
#endif
    // 不能折叠——创建正常的二元运算节点
    tTJSExprNode *n = new tTJSExprNode;
    n->SetOpcode(opecode);
    n->Add(node1);
    n->Add(node2);
    return n;
}
```

### 3.4 支持的运算符总览

| 类别 | 运算符 | MakeNP1 | MakeNP2 |
|------|--------|---------|---------|
| 逻辑运算 | `!`, `\|\|`, `&&` | ✅ `!` | ✅ `\|\|` `&&` |
| 位运算 | `~`, `\|`, `^`, `&`, `>>`, `<<`, `>>>` | ✅ `~` | ✅ 全部 |
| 算术运算 | `+`, `-`（一元/二元）, `*`, `/`, `\`, `%` | ✅ `+x` `-x` | ✅ 全部 |
| 比较运算 | `==`, `!=`, `===`, `!==`, `<`, `>`, `<=`, `>=` | — | ✅ 全部 |
| 类型转换 | `int()`, `real()`, `string()`, `(octet)` | ✅ 全部 | — |
| 特殊 | `#`（字符转码）, `$`（码转字符）, `,`（逗号） | ✅ `#` `$` | ✅ `,` |

### 3.5 常量折叠的限制

常量折叠**不能**处理以下场景：

1. **任一操作数不是常量**：`var x = a + 1` 中 `a` 是变量，无法折叠
2. **三元运算**：`MakeNP3`（用于条件运算 `? :`）**没有常量折叠逻辑**
3. **函数调用**：即使参数全部是常量，编译器也不会在编译期执行函数
4. **可能抛出异常的运算**：除以 0 等操作，编译器选择保留到运行时处理

```tjs
// 可以折叠的例子
var a = 2 + 3;            // → var a = 5
var b = "hello" + "!";    // → var b = "hello!"
var c = !false;           // → var c = true
var d = int(3.14);        // → var d = 3

// 不能折叠的例子
var e = x + 1;            // x 是变量，不能折叠
var f = condition ? 1 : 2; // MakeNP3 不支持折叠
var g = Math.sqrt(4);      // 函数调用不折叠
```

### 3.6 TJS_NO_CONSTANT_FOLDING 编译开关

常量折叠可以通过预处理宏 `TJS_NO_CONSTANT_FOLDING` 禁用。这个开关主要用于调试编译器本身——禁用折叠后可以观察编译器对每个运算生成的完整字节码，便于检验代码生成逻辑的正确性。在正式构建中该宏不会定义，常量折叠始终启用。

## §4 FixCode — 编译后的代码修补

`FixCode`（`tjsInterCodeGen.cpp` 第 702-878 行）是编译完成后对字节码进行后处理的核心函数。它完成三项关键工作：代码插入、跳转地址修正、冗余跳转消除。

### 4.1 阶段 1：类成员注册代码的生成与插入

当编译的是类体（`ctClass`）时，`FixCode` 首先在 `FunctionRegisterCodePoint`（函数注册代码插入点，编译器在类体编译开始时记录的一个字节码地址，标记"在这里插入成员注册代码"）处插入 `VM_REGMEMBER` 指令：

```cpp
// tjsInterCodeGen.cpp 第 702-716 行
if(ContextType == ctClass) {
    tjs_int32 *code = new tjs_int32[1];
    code[0] = VM_REGMEMBER;  // 注册类成员到新创建的对象
    
    // 在 FunctionRegisterCodePoint 处插入 1 条指令
    FixList.emplace_back(FunctionRegisterCodePoint, 0, 1, code, true);
}
```

`VM_REGMEMBER`（Register Member，注册类成员指令）告诉 VM："现在把这个类的所有方法和属性注册到当前正在构造的对象上"。这条指令必须在类体的初始化代码之前执行，确保成员函数在类体代码使用它们之前就已经可用。

### 4.2 阶段 1b：非局部函数声明的前置代码

如果编译上下文中有**前向引用的函数声明**（Non-Local Function Declaration，在使用位置之前就需要可用的函数，例如顶层脚本中的全局函数），`FixCode` 生成一段代码将这些函数注册到 `objthis`：

```cpp
// tjsInterCodeGen.cpp 第 720-771 行（简化）
if(NonLocalFunctionDeclVector.size() >= 1) {
    if(MaxFrameCount < 1) MaxFrameCount = 1;  // 确保至少有一个临时寄存器
    
    // 为每个非局部函数生成注册代码：
    for(auto &func : NonLocalFunctionDeclVector) {
        // VM_CONST %1, #funcdata    — 加载函数对象到临时寄存器
        *(codep++) = VM_CONST;
        *(codep++) = TJS_TO_VM_REG_ADDR(1);
        *(codep++) = TJS_TO_VM_REG_ADDR(func.DataPos);
        
        if(func.ChangeThis) {
            // VM_CHGTHIS %1, %-1    — 将函数的 this 绑定到当前对象
            *(codep++) = VM_CHGTHIS;
            *(codep++) = TJS_TO_VM_REG_ADDR(1);
            *(codep++) = TJS_TO_VM_REG_ADDR(-1);
        }
        
        // VM_SPDS %-1, #funcname, %1  — 将函数注册为 objthis 的属性
        *(codep++) = VM_SPDS;
        *(codep++) = TJS_TO_VM_REG_ADDR(-1);
        *(codep++) = TJS_TO_VM_REG_ADDR(func.NameDataPos);
        *(codep++) = TJS_TO_VM_REG_ADDR(1);
    }
    
    // VM_CL %1  — 清除临时寄存器
    *(codep++) = VM_CL;
    *(codep++) = TJS_TO_VM_REG_ADDR(1);
}
```

这段代码使得**函数提升**（Function Hoisting，函数声明在其所在作用域的任何位置都可以被调用，不受声明位置限制）成为可能——类似于 JavaScript 中的函数声明提升。

### 4.3 阶段 2：跳转地址修正

代码插入会改变字节码的位置，所有跳转指令的目标地址都需要相应调整：

```cpp
// tjsInterCodeGen.cpp 第 778-835 行（简化逻辑）
for(auto &fix : FixList) {
    // 遍历所有跳转指令
    for(auto jmp = JumpList.begin(); jmp != JumpList.end(); jmp++) {
        tjs_int jmptarget = CodeArea[*jmp + 1] + *jmp;  // 计算跳转目标
        
        if(跳转指令在被移动的代码区域内) {
            // 删除该跳转记录（代码已被替换）
            jmp = JumpList.erase(jmp);
        } else if(跳转跨越了插入点) {
            // 调整跳转偏移量
            CodeArea[*jmp + 1] += fix.NewSize - fix.Size;
        }
        
        if(跳转指令在插入点之后) {
            // 调整跳转指令本身的位置
            *jmp += fix.NewSize - fix.Size;
        }
    }
    
    // 移动代码
    memmove(CodeArea + fix.StartIP + fix.NewSize,
            CodeArea + fix.StartIP + fix.Size,
            sizeof(tjs_int32) * (CodeAreaSize - fix.StartIP - fix.Size));
    
    // 复制新代码到插入点
    memcpy(CodeArea + fix.StartIP, fix.Code,
           sizeof(tjs_int32) * fix.NewSize);
    
    // 更新源码位置映射
    for(auto &sp : SourcePosArray) {
        if(sp.CodePos >= fix.StartIP + fix.Size)
            sp.CodePos += fix.NewSize - fix.Size;
    }
    
    CodeAreaSize += fix.NewSize - fix.Size;
}
```

### 4.4 阶段 3：冗余跳转消除

这是 `FixCode` 中最精巧的优化。当一个跳转指令的目标是另一个跳转指令时，编译器直接将第一个跳转的目标修改为最终目标——消除跳转链：

```cpp
// tjsInterCodeGen.cpp 第 837-868 行
for(auto jmp = JumpList.begin(); jmp != JumpList.end(); jmp++) {
    tjs_int32 jumptarget = CodeArea[*jmp + 1] + *jmp;
    tjs_int32 jumpcode = CodeArea[*jmp];
    tjs_int addr = *jmp + CodeArea[*jmp + 1];
    
    for(;;) {
        if(CodeArea[addr] == VM_JMP ||           // 无条件跳转
           (CodeArea[addr] == jumpcode &&
            (jumpcode == VM_JF || jumpcode == VM_JNF))) {
            // 场景 1：JMP → JMP 或 JF → JF 或 JNF → JNF
            // 直接跳到最终目标
            jumptarget = CodeArea[addr + 1] + addr;
            addr += CodeArea[addr + 1];
        } else if((CodeArea[addr] == VM_JF && jumpcode == VM_JNF) ||
                  (CodeArea[addr] == VM_JNF && jumpcode == VM_JF)) {
            // 场景 2：JF → JNF 或 JNF → JF（相反条件）
            // 跳到第二个跳转之后（因为条件互斥，第二个一定不跳）
            jumptarget = addr + 2;
            addr += 2;
        } else {
            break;  // 不是跳转指令，停止追踪
        }
    }
    
    // 更新原始跳转的目标
    CodeArea[*jmp + 1] = jumptarget - *jmp;
}
```

**三种跳转消除场景**：

```
场景 1：JMP → JMP（或同类型条件跳转）
  优化前：JMP A → ... → A: JMP B → ... → B: 实际代码
  优化后：JMP B → ... → A: JMP B → ... → B: 实际代码
  效果：跳过中间的跳转链

场景 2：JF → JNF（相反条件跳转）
  优化前：JF A → ... → A: JNF B → ...
  优化后：JF (A+2) → ... → A: JNF B → ...
  原理：如果 JF 跳到了 A，说明条件为 false；
        此时 A 处的 JNF（Jump if Not False = Jump if True）一定不跳
        所以直接跳到 JNF 后面的指令

场景 3：目标不是跳转指令
  不做任何优化，保持原样
```

### 4.5 阶段 4：跳转地址转换

最后，所有跳转地址从编译期的相对字节偏移转换为 VM 使用的代码地址格式：

```cpp
// tjsInterCodeGen.cpp 第 870-876 行
for(auto jmp = JumpList.begin(); jmp != JumpList.end(); jmp++) {
    CodeArea[*jmp + 1] = TJS_TO_VM_CODE_ADDR(CodeArea[*jmp + 1]);
}
JumpList.clear();
FixList.clear();
```

## §5 Commit 函数 — 编译的最终收尾

`Commit`（`tjsInterCodeGen.cpp` 第 952-1038 行）是每个编译上下文完成编译后必须调用的收尾函数。它完成一系列最终处理，将编译期的临时数据结构转化为运行时可用的最终形态。

### 5.1 Commit 的整体流程

Commit 按照严格的顺序执行以下 8 个步骤：

```
┌─────────────────────────────────────────────────────┐
│                   Commit() 流程                      │
├─────────────────────────────────────────────────────┤
│ 1. 递归提交 SuperClassGetter（仅 ctClass）           │
│ 2. 生成函数返回指令 VM_SRV + VM_RET                  │
│ 3. RegisterFunction — 注册到父上下文                 │
│ 4. FixCode — 后处理字节码                            │
│ 5. _DataArea → DataArea 转换                        │
│ 6. 设置 MaxVariableCount                            │
│ 7. 内存压缩（SourcePosArray + CodeArea）             │
│ 8. RE_CREATE — 释放编译期容器的内部存储               │
└─────────────────────────────────────────────────────┘
```

### 5.2 步骤 1：递归提交 SuperClassGetter

如果当前上下文是类体（`ctClass`）并且有父类获取器（SuperClassGetter，用于在类实例化时计算父类对象的特殊上下文），需要先递归提交它：

```cpp
// tjsInterCodeGen.cpp 第 958-962 行
if(ContextType == ctClass) {
    if(SuperClassGetter)
        SuperClassGetter->Commit();  // 先提交子上下文
}
```

这确保了父类获取器的字节码在主类体之前就已经完成所有后处理。SuperClassGetter 是编译 `class Derived extends Base { ... }` 时为 `Base` 表达式创建的一个独立上下文——它负责在运行时计算父类对象。

### 5.3 步骤 2：生成函数返回指令

对于大多数上下文类型，Commit 在字节码末尾追加一组返回指令（Return Epilogue，函数返回序言——确保函数执行到末尾时能正确返回 `void` 值）：

```cpp
// tjsInterCodeGen.cpp 第 964-968 行
if(ContextType != ctProperty && ContextType != ctSuperClassGetter) {
    PutCode(VM_SRV, LEX_POS);       // Set Return Value — 设置返回值
    PutCode(TJS_TO_VM_REG_ADDR(0)); // 参数：寄存器 %0（void 值）
    PutCode(VM_RET);                 // Return — 返回
}
```

**`VM_SRV %0`** 将寄存器 %0 的值设置为函数的返回值。由于 %0 在初始化时被清空（值为 `void`），这等价于 `return void`。`VM_RET` 则触发实际的返回操作。

**为什么 ctProperty 和 ctSuperClassGetter 不需要返回指令？**

- **ctProperty**：属性上下文不生成独立的字节码流——它的 getter/setter 是独立的子上下文，各自有自己的 Commit
- **ctSuperClassGetter**：父类获取器的代码在类构造过程中内联执行，不需要独立的返回逻辑

### 5.4 步骤 3：RegisterFunction — 注册到父上下文

`RegisterFunction`（`tjsInterCodeGen.cpp` 第 881-946 行）将当前编译完成的上下文注册到父上下文中。注册方式取决于上下文类型和父上下文类型的组合：

```cpp
// tjsInterCodeGen.cpp 第 970 行
RegisterFunction();
```

注册路径一览：

| 当前类型 | 父类型 | 注册方式 | 说明 |
|---------|--------|---------|------|
| ctPropertySetter | ctProperty | `Parent->PropSetter = this` | 属性写入器注册为父属性的 setter |
| ctPropertyGetter | ctProperty | `Parent->PropGetter = this` | 属性读取器注册为父属性的 getter |
| ctSuperClassGetter | ctClass | _(已在创建时注册)_ | 直接返回，无需额外操作 |
| ctFunction/ctProperty/ctClass | ctTopLevel | `NonLocalFunctionDeclVector` | 加入非局部函数声明列表，FixCode 会生成前置注册代码 |
| ctFunction | ctFunction | `InitLocalFunction` | 注册为父函数的局部变量 |
| ctFunction/ctClass | ctFunction/ctClass | `PropSet` | 注册为父对象的属性成员 |

```cpp
// tjsInterCodeGen.cpp 第 881-946 行（核心注册逻辑）
void tTJSInterCodeContext::RegisterFunction() {
    if(!Parent) return;  // 顶层上下文没有父级

    // 属性 setter/getter 的注册——简单的指针赋值
    if(ContextType == ctPropertySetter) {
        Parent->PropSetter = this;
        AddRef();  // 增加引用计数，防止提前析构
        return;
    }
    if(ContextType == ctPropertyGetter) {
        Parent->PropGetter = this;
        AddRef();
        return;
    }

    // 顶层上下文中的函数/类——加入 NonLocalFunctionDeclVector
    if(Parent->ContextType == ctTopLevel) {
        tTJSVariant val;
        val = this;                              // 函数对象存入数据区
        int data = Parent->PutData(val);
        val = Name;                              // 函数名也存入数据区
        int name = Parent->PutData(val);
        bool changethis =
            ContextType == ctFunction || ContextType == ctProperty;
        Parent->NonLocalFunctionDeclVector.emplace_back(data, name, changethis);
    }

    // 函数中的局部函数——注册为父函数的局部变量
    if(ContextType == ctFunction && Parent->ContextType == ctFunction) {
        Parent->InitLocalFunction(Name, data);
    }

    // 函数/类中的子函数/子类——注册为父对象的属性
    if(Parent->ContextType == ctFunction || Parent->ContextType == ctClass) {
        tTJSVariant val = this;
        Parent->PropSet(TJS_MEMBERENSURE | TJS_IGNOREPROP, Name, nullptr,
                        &val, Parent);
    }
}
```

**changethis 标志的含义**：当函数被注册到顶层上下文时，如果它是 `ctFunction` 或 `ctProperty`，需要在注册时绑定 `this` 对象（通过 `VM_CHGTHIS` 指令），确保函数在被调用时 `this` 指向正确的对象。

### 5.5 步骤 4：FixCode 调用

```cpp
// tjsInterCodeGen.cpp 第 972-973 行
if(ContextType != ctProperty && ContextType != ctSuperClassGetter)
    FixCode();
```

FixCode 的详细流程已在 §4 中讲解。跳过条件与步骤 2 相同——ctProperty 和 ctSuperClassGetter 不产生独立字节码流。

### 5.6 步骤 5：_DataArea → DataArea 转换

这是 Commit 中最关键的数据转换步骤——将编译期的指针数组转换为运行时的值数组：

```cpp
// tjsInterCodeGen.cpp 第 975-989 行
if(!DataArea) {
    // 步骤 1：创建运行时值数组
    DataArea = new tTJSVariant[_DataAreaSize];
    DataAreaSize = _DataAreaSize;

    // 步骤 2：将指针数组中的值逐个复制到值数组
    for(tjs_int i = 0; i < _DataAreaSize; i++) {
        DataArea[i].CopyRef(*_DataArea[i]);  // CopyRef：引用计数安全的拷贝
    }

    // 步骤 3：释放编译期指针数组
    if(_DataArea) {
        for(tjs_int i = 0; i < _DataAreaSize; i++)
            delete _DataArea[i];       // 释放每个堆上的 tTJSVariant
        TJS_free(_DataArea);           // 释放指针数组本身
        _DataArea = nullptr;
    }
}
```

**为什么用 CopyRef 而不是赋值？**

`CopyRef` 是 `tTJSVariant` 的引用计数安全拷贝方法。与普通赋值运算符的区别在于：

| 操作 | 行为 | 适用场景 |
|------|------|---------|
| `operator=` | 深拷贝值，字符串和对象创建新副本 | 需要独立副本时 |
| `CopyRef` | 共享底层存储，仅增加引用计数 | 从临时数组转移到永久数组时 |

使用 `CopyRef` 避免了不必要的深拷贝开销——编译期数据区中的值直接"移动"到运行时数据区，底层字符串和对象存储共享。

**转换的时序图**：

```
编译期:
  _DataArea: [ ptr0 ] [ ptr1 ] [ ptr2 ] ...    ← 指针数组（可安全 realloc）
                |        |        |
                v        v        v
              [val0]   [val1]   [val2]          ← 堆上独立 tTJSVariant

Commit 后:
  DataArea:  [ val0 ] [ val1 ] [ val2 ] ...    ← 连续值数组（运行时高效访问）
  _DataArea: nullptr                            ← 已释放
```

### 5.7 步骤 6：设置 MaxVariableCount

```cpp
// tjsInterCodeGen.cpp 第 991-994 行
if(ContextType == ctSuperClassGetter)
    MaxVariableCount = 2;  // 父类获取器固定为 2
else
    MaxVariableCount = Namespace.GetMaxCount();
```

`MaxVariableCount`（最大变量计数，运行时需要为此上下文分配的局部变量槽数量）来自命名空间（Namespace）的 `GetMaxCount()`——这是编译期通过 `tTJSLocalNamespace` 的高水位标记（High-Water Mark，一种只增不减的计数方式，记录编译过程中同时使用的最大变量数量）计算出的最大并发变量数。

SuperClassGetter 的特殊处理是因为它只需要两个固定变量槽（`this` 和一个临时变量），无需通过命名空间计算。

### 5.8 步骤 7：内存压缩

编译过程中，`SourcePosArray` 和 `CodeArea` 都按 256 增量分配容量。Commit 将它们压缩到实际大小：

```cpp
// tjsInterCodeGen.cpp 第 1001-1017 行

// 压缩源码位置数组
if(SourcePosArraySize && SourcePosArray) {
    SourcePosArray = (tSourcePos *)TJS_realloc(
        SourcePosArray, SourcePosArraySize * sizeof(tSourcePos));
    if(!SourcePosArray)
        TJS_eTJSScriptError(TJSInsufficientMem, Block, 0);
    SourcePosArrayCapa = SourcePosArraySize;  // 容量 = 实际大小
}

// 压缩代码区数组
if(CodeAreaSize && CodeArea) {
    CodeArea = (tjs_int32 *)TJS_realloc(
        CodeArea, sizeof(tjs_int32) * CodeAreaSize);
    if(!CodeArea)
        TJS_eTJSScriptError(TJSInsufficientMem, Block, 0);
    CodeAreaCapa = CodeAreaSize;  // 容量 = 实际大小
}
```

**压缩的意义**：假设一个函数编译后生成了 100 条字节码。编译期 CodeArea 的容量可能是 256（最近一次扩容后的大小），浪费了 `(256 - 100) × 4 = 624` 字节。对于包含数百个函数的大型脚本，这种浪费是可观的。`realloc` 到实际大小可以释放多余的内存。

### 5.9 步骤 8：RE_CREATE — 释放编译期容器

Commit 的最后一步是释放编译过程中使用的各种临时容器：

```cpp
// tjsInterCodeGen.cpp 第 1025-1038 行

// RE_CREATE 宏：先析构，再原地重新构造空容器
#define RE_CREATE(place, type, classname)   \
    (&place)->~classname();                 \
    new(&place) type()

RE_CREATE(NodeToDeleteVector, std::vector<tTJSExprNode *>, vector);
RE_CREATE(CurrentNodeVector, std::vector<tTJSExprNode *>, vector);
RE_CREATE(FuncArgStack, std::stack<tFuncArg>, stack);
RE_CREATE(ArrayArgStack, std::stack<tArrayArg>, stack);
RE_CREATE(NestVector, std::vector<tNestData>, vector);
RE_CREATE(JumpList, std::list<tjs_int>, list);
RE_CREATE(FixList, std::list<tFixData>, list);
RE_CREATE(NonLocalFunctionDeclVector, std::vector<tNonLocalFunctionDecl>, vector);
```

**为什么不直接调用 `clear()`？**

源码中的注释说明了原因：*"we do thus nasty thing because the std::vector does not free its storage even we call 'clear' method"*。

C++ 标准规定 `std::vector::clear()` 只销毁元素但不释放底层内存（`capacity()` 不变）。编译完成后这些容器不再使用，但如果只 `clear()`，它们的内存仍然被占用。RE_CREATE 宏通过**原地析构 + 原地构造**（Placement Destruction + Placement New，在对象原有内存位置上先调用析构函数释放资源，再调用构造函数重新初始化）强制释放底层存储：

1. `(&place)->~classname()` — 调用析构函数，释放容器内部的堆内存
2. `new(&place) type()` — 在原地址构造一个新的空容器（容量为 0）

这是一种"黑科技"（Nasty Trick），在现代 C++ 中可以用 `shrink_to_fit()` 替代，但在 TJS2 编写的年代这种方法是有效的。

## §6 ExportByteCode — 字节码序列化

编译完成后，字节码可以通过 `ExportByteCode` 导出为二进制格式，也可以通过 `tTJSByteCodeLoader` 从二进制格式反序列化恢复。这种机制使得脚本可以"预编译"——避免每次启动时重新编译。

### 6.1 二进制文件的整体格式

TJS2 字节码文件由三大段组成：

```
┌────────────────────────────────────────────────────┐
│  文件头（File Header）— 12 字节                      │
│    [4 bytes] FILE_TAG: "TJS2" (0x324A5354)         │
│    [4 bytes] VER_TAG:  "100\0" (0x00303031)        │
│    [4 bytes] FileSize: 文件总字节数                  │
├────────────────────────────────────────────────────┤
│  DATA 段 — 共享常量池                                │
│    [4 bytes] DATA_TAG: "DATA" (0x41544144)         │
│    [4 bytes] DataSize: 本段字节数                    │
│    [n bytes] ByteArray: 字节常量数组                 │
│    [n bytes] ShortArray: 短整数常量数组              │
│    [n bytes] LongArray: 32位整数常量数组             │
│    [n bytes] LongLongArray: 64位整数常量数组         │
│    [n bytes] DoubleArray: 浮点数常量数组             │
│    [n bytes] StringArray: 字符串常量数组             │
│    [n bytes] OctetArray: 字节序列常量数组            │
├────────────────────────────────────────────────────┤
│  OBJS 段 — 编译上下文对象数组                        │
│    [4 bytes] OBJ_TAG: "OBJS" (0x534A424F)          │
│    [4 bytes] ObjSize: 本段字节数                    │
│    [4 bytes] TopLevel: 顶层上下文索引                │
│    [4 bytes] ObjCount: 上下文对象数量                │
│    [n bytes] Object[0] ... Object[N-1]             │
└────────────────────────────────────────────────────┘
```

**文件标签的定义**（小端序编码）：

```cpp
// tjsByteCodeLoader.h 第 25-28 行
static const tjs_uint32 FILE_TAG_LE =
    ('T') | ('J' << 8) | ('S' << 16) | ('2' << 24);   // "TJS2"
static const tjs_uint32 VER_TAG_LE =
    ('1') | ('0' << 8) | ('0' << 16) | (0 << 24);      // "100\0"
```

### 6.2 DATA 段 — 共享常量池

DATA 段存储所有编译上下文共享的常量值。`tjsConstArrayData` 类负责收集和去重——它按类型分类存储：

```cpp
// tjsByteCodeLoader.h 第 73-84 行 — 类型标记
static const tjs_int32 TYPE_VOID     = 0;   // void 值
static const tjs_int32 TYPE_OBJECT   = 1;   // null 对象
static const tjs_int32 TYPE_INTER_OBJECT  = 2;   // 内部对象引用（编译上下文）
```

反序列化时，`ReadDataArea` 依次读取这 7 种类型的数组：

```cpp
// tjsByteCodeLoader.cpp 第 74-158 行（简化）
void tTJSByteCodeLoader::ReadDataArea(const tjs_uint8 *buff,
                                      int offset, size_t size) {
    // 1. ByteArray — 字节常量
    int count = read4byte(&(buff[offset]));
    ByteArray.set((tjs_int8 *)&buff[offset + 4], count);

    // 2. ShortArray — 短整数常量
    count = read4byte(&(buff[offset]));
    for(int i = 0; i < count; i++)
        ShortArray.push_back(read2byte(&(buff[offset])));

    // 3. LongArray — 32 位整数常量
    count = read4byte(&(buff[offset]));
    for(int i = 0; i < count; i++)
        LongArray.push_back(read4byte(&(buff[offset])));

    // 4. LongLongArray — 64 位整数常量
    count = read4byte(&(buff[offset]));
    for(int i = 0; i < count; i++)
        LongLongArray.push_back(read8byte(&(buff[offset])));

    // 5. DoubleArray — 浮点数常量
    count = read4byte(&(buff[offset]));
    for(int i = 0; i < count; i++) {
        tjs_uint64 tmp = read8byte(&(buff[offset]));
        DoubleArray.push_back(*(double *)&tmp);  // 位模式转换
    }

    // 6. StringArray — 字符串常量
    count = read4byte(&(buff[offset]));
    for(int i = 0; i < count; i++) {
        int len = read4byte(&(buff[offset]));
        // 逐字符读取 UTF-16 编码的字符串
        std::vector<tjs_uint16> ch(len + 1);
        for(int j = 0; j < len; j++)
            ch[j] = read2byte(&(buff[offset]));
        StringArray.push_back(
            TJSMapGlobalStringMap((const tjs_char *)&(ch[0])));
    }

    // 7. OctetArray — 字节序列常量
    count = read4byte(&(buff[offset]));
    for(int i = 0; i < count; i++) {
        int len = read4byte(&(buff[offset]));
        auto *octet = new tTJSVariantOctet(&(buff[offset]), len);
        OctetArray.push_back(octet);
    }
}
```

注意字符串加载时也使用了 `TJSMapGlobalStringMap` 进行全局字符串池化——与编译期的优化一致。

### 6.3 ExportByteCode — 单个上下文的序列化

每个编译上下文通过 `ExportByteCode`（`tjsInterCodeGen.cpp` 第 4005-4116 行）序列化为二进制数据。序列化包含 12 个头字段和 5 个数据段：


**头字段详解**：

| 字段 | 大小 | 说明 |
|------|------|------|
| parent | 4 bytes | 父上下文在对象数组中的索引，-1 表示无父级 |
| name | 4 bytes | 函数名在共享常量池中的索引 |
| ContextType | 4 bytes | 上下文类型（0=TopLevel, 1=Function, ...） |
| MaxVariableCount | 4 bytes | 运行时局部变量槽数量 |
| VariableReserveCount | 4 bytes | 保留寄存器数（通常为 2） |
| MaxFrameCount | 4 bytes | 临时寄存器的最大数量 |
| FuncDeclArgCount | 4 bytes | 函数声明的形参数量 |
| FuncDeclUnnamedArgArrayBase | 4 bytes | `*` 参数（可变参数）的起始索引 |
| FuncDeclCollapseBase | 4 bytes | 参数折叠基址 |
| propSetter | 4 bytes | 属性 setter 的上下文索引，-1 表示无 |
| propGetter | 4 bytes | 属性 getter 的上下文索引，-1 表示无 |
| superClassGetter | 4 bytes | 父类获取器的上下文索引，-1 表示无 |

紧随头部之后是 5 个数据段：

```cpp
// 数据段 1：源码位置映射（仅调试模式）

    Add4ByteToVector(result, SourcePosArray[i].CodePos);   // 字节码地址
for(int i = 0; i < count; i++)
    Add4ByteToVector(result, SourcePosArray[i].SourcePos); // 源码位置

// 数据段 2：代码区（字节码指令流）
Add4ByteToVector(result, CodeAreaSize);
block->TranslateCodeAddress(CodeArea, CodeAreaSize);  // 地址格式转换
for(int i = 0; i < CodeAreaSize; i++)
    Add2ByteToVector(result, CodeArea[i]);  // 注意：每条指令压缩为 16 位
if((CodeAreaSize % 2) == 1)
    Add2ByteToVector(result, 0);  // 对齐填充

// 数据段 3：数据区（常量引用）
Add4ByteToVector(result, DataAreaSize);
for(int i = 0; i < DataAreaSize; i++) {
    tjs_int16 type = constarray.GetType(DataArea[i], block);
    tjs_int16 v = (tjs_int16)constarray.PutVariant(DataArea[i], block);
    Add2ByteToVector(result, type);   // 类型标记（2 bytes）
    Add2ByteToVector(result, v);      // 共享常量池索引（2 bytes）
}

// 数据段 4：父类获取器指针数组
Add4ByteToVector(result, SuperClassGetterPointer.size());
for(auto v : SuperClassGetterPointer)
    Add4ByteToVector(result, v);

// 数据段 5：属性列表
if(Properties != nullptr) {
    Add4ByteToVector(result, Properties->size());
    for(auto prop : *Properties) {
        Add4ByteToVector(result, constarray.PutString(prop->Name));  // 属性名
        Add4ByteToVector(result, block->GetCodeIndex(prop->Value));  // 属性值上下文索引
    }
}
```

**代码区压缩**：注意 `CodeArea` 中每个 `tjs_int32`（32 位）在导出时被压缩为 `tjs_int16`（16 位）。这是因为绝大多数操作码和寄存器地址的值范围在 int16 内。这使得字节码文件大小减半。

**数据区引用格式**：每个数据区条目被序列化为 `(type, index)` 对（各 2 字节），其中 type 表示常量的类型标记，index 指向共享常量池（DATA 段）中对应类型数组的索引。这避免了在每个上下文中重复存储相同的常量值。

### 练习 2：常量折叠效果验证

编写一个简化的常量折叠演示，展示编译器如何在编译期计算表达式：

```cpp
#include <iostream>
#include <string>
#include <variant>
#include <functional>

// 简化的 AST 节点类型
enum class NodeType { ConstVal, BinaryOp, UnaryOp };

struct ASTNode {
    NodeType type;
    std::variant<int, double, std::string> value;   // 常量值
    int opcode = 0;                                  // 运算符
    ASTNode* left = nullptr;
    ASTNode* right = nullptr;

    ~ASTNode() { delete left; delete right; }
};

// 运算符编码（简化）
enum Op { OP_ADD = 1, OP_SUB, OP_MUL, OP_DIV, OP_NEG, OP_NOT };

// 模拟 MakeNP1 — 一元运算常量折叠
ASTNode* MakeNP1(int op, ASTNode* node) {
    if(node && node->type == NodeType::ConstVal) {
        int val = std::get<int>(node->value);
        ASTNode* result = new ASTNode{NodeType::ConstVal, {}, 0, nullptr, nullptr};

        switch(op) {
            case OP_NEG:
                result->value = -val;
                std::cout << "  [折叠] -(" << val << ") → " << -val << "\n";
                delete node;
                return result;
            case OP_NOT:
                result->value = val ? 0 : 1;
                std::cout << "  [折叠] !(" << val << ") → " << (val ? 0 : 1) << "\n";
                delete node;
                return result;
        }
        delete result;
    }
    // 无法折叠——创建普通运算节点
    ASTNode* n = new ASTNode{NodeType::UnaryOp, {}, op, node, nullptr};
    std::cout << "  [保留] 一元运算，操作数不是常量\n";
    return n;
}

// 模拟 MakeNP2 — 二元运算常量折叠
ASTNode* MakeNP2(int op, ASTNode* left, ASTNode* right) {
    if(left && left->type == NodeType::ConstVal &&
       right && right->type == NodeType::ConstVal) {
        int lv = std::get<int>(left->value);
        int rv = std::get<int>(right->value);
        ASTNode* result = new ASTNode{NodeType::ConstVal, {}, 0, nullptr, nullptr};

        switch(op) {
            case OP_ADD:
                result->value = lv + rv;
                std::cout << "  [折叠] " << lv << " + " << rv
                          << " → " << lv + rv << "\n";
                break;
            case OP_SUB:
                result->value = lv - rv;
                std::cout << "  [折叠] " << lv << " - " << rv
                          << " → " << lv - rv << "\n";
                break;
            case OP_MUL:
                result->value = lv * rv;
                std::cout << "  [折叠] " << lv << " * " << rv
                          << " → " << lv * rv << "\n";
                break;
            case OP_DIV:
                if(rv != 0) {
                    result->value = lv / rv;
                    std::cout << "  [折叠] " << lv << " / " << rv
                              << " → " << lv / rv << "\n";
                } else {
                    // 除以 0 不折叠——保留到运行时处理
                    std::cout << "  [保留] 除以 0，不折叠\n";
                    delete result;
                    result = new ASTNode{NodeType::BinaryOp, {}, op, left, right};
                    return result;
                }
                break;
        }
        delete left;
        delete right;
        return result;
    }

    // 无法折叠
    ASTNode* n = new ASTNode{NodeType::BinaryOp, {}, op, left, right};
    std::cout << "  [保留] 二元运算，操作数不全是常量\n";
    return n;
}

    auto* n3 = new ASTNode{NodeType::ConstVal, 3};
    auto* r1 = MakeNP2(OP_ADD, n2, n3);

    std::cout << "表达式: -(10)\n";
    auto* n10 = new ASTNode{NodeType::ConstVal, 10};
    auto* r2 = MakeNP1(OP_NEG, n10);
    std::cout << "\n";

    // 表达式 3：(2 + 3) * 4 → 5 * 4 → 20（嵌套折叠）
    std::cout << "表达式: (2 + 3) * 4\n";
    auto* na = new ASTNode{NodeType::ConstVal, 2};
    auto* nb = new ASTNode{NodeType::ConstVal, 3};
    auto* sum = MakeNP2(OP_ADD, na, nb);    // 先折叠 2+3 → 5
    auto* nc = new ASTNode{NodeType::ConstVal, 4};
    auto* r3 = MakeNP2(OP_MUL, sum, nc);    // 再折叠 5*4 → 20
    std::cout << "\n";

    // 表达式 4：x + 1（x 是变量，不能折叠）
    std::cout << "表达式: x + 1（x 是变量）\n";
    auto* var_x = new ASTNode{NodeType::BinaryOp, {}, 0};  // 非常量节点
    auto* n1 = new ASTNode{NodeType::ConstVal, 1};
    auto* r4 = MakeNP2(OP_ADD, var_x, n1);

    // 清理
    delete r1; delete r2; delete r3; delete r4;

}
```

### 练习 3：模拟冗余跳转消除

```cpp
#include <iostream>
#include <vector>
#include <string>

// 简化的指令类型
enum Opcode { OP_NOP, OP_JMP, OP_JF, OP_JNF, OP_CODE };

struct Instruction {
    Opcode op;
    int target;  // 跳转目标地址（仅跳转指令使用）
    std::string label;
};

void PrintCode(const std::vector<Instruction>& code) {
    for(size_t i = 0; i < code.size(); i++) {
        std::cout << "  [" << i << "] ";
        switch(code[i].op) {
            case OP_JMP:  std::cout << "JMP  → " << code[i].target; break;
            case OP_JF:   std::cout << "JF   → " << code[i].target; break;
            case OP_JNF:  std::cout << "JNF  → " << code[i].target; break;
            case OP_CODE: std::cout << "CODE (" << code[i].label << ")"; break;
            default:      std::cout << "NOP"; break;
        }
        std::cout << "\n";
    }
}

// 模拟 FixCode 的冗余跳转消除
void EliminateRedundantJumps(std::vector<Instruction>& code) {
    for(size_t i = 0; i < code.size(); i++) {
        if(code[i].op != OP_JMP && code[i].op != OP_JF && code[i].op != OP_JNF)
            continue;

        int target = code[i].target;
        Opcode jumpCode = code[i].op;
        bool changed = false;

        // 追踪跳转链
        for(;;) {
            if(target < 0 || target >= (int)code.size()) break;

            if(code[target].op == OP_JMP) {
                // 场景 1：任何跳转 → JMP：直接跳到最终目标
                std::cout << "  场景1: [" << i << "] " << target
                          << " → " << code[target].target << "\n";
                target = code[target].target;
                changed = true;
            } else if(code[target].op == jumpCode &&
                      (jumpCode == OP_JF || jumpCode == OP_JNF)) {
                // 场景 1b：JF → JF 或 JNF → JNF（同类型）
                std::cout << "  场景1b: [" << i << "] 同类型跳转链\n";
                target = code[target].target;
                changed = true;
            } else if((code[target].op == OP_JF && jumpCode == OP_JNF) ||
                      (code[target].op == OP_JNF && jumpCode == OP_JF)) {
                // 场景 2：JF → JNF 或 JNF → JF（相反条件）
                std::cout << "  场景2: [" << i << "] 条件互斥，跳到 "
                          << target + 1 << "\n";
                target = target + 1;
                changed = true;
            } else {
                break;  // 目标不是跳转指令
            }
        }

        if(changed) {
            code[i].target = target;
        }
    }
}

int main() {
```

```cpp
#include <cstring>
#include <vector>

// 手动解析 TJS2 字节码文件头
void ParseTJS2Header(const uint8_t* data, size_t size) {
    if(size < 12) {
        std::cout << "错误：数据不足 12 字节\n";
        return;
    }

    // 读取文件标签（小端序）
    uint32_t fileTag = data[0] | (data[1] << 8) | (data[2] << 16) | (data[3] << 24);
    uint32_t expectedTag = 'T' | ('J' << 8) | ('S' << 16) | ('2' << 24);

    std::cout << "文件标签: 0x" << std::hex << fileTag;
    if(fileTag == expectedTag) {
        std::cout << " ✓ (TJS2)\n";
    } else {
        std::cout << " ✗ 不是 TJS2 字节码文件\n";
        return;
    }

    // 版本标签
    uint32_t verTag = data[4] | (data[5] << 8) | (data[6] << 16) | (data[7] << 24);
    uint32_t expectedVer = '1' | ('0' << 8) | ('0' << 16) | (0 << 24);
    std::cout << "版本标签: 0x" << std::hex << verTag;
    std::cout << (verTag == expectedVer ? " ✓ (100)" : " ✗ 版本不匹配") << "\n";

    // 文件大小
    uint32_t fileSize = data[8] | (data[9] << 8) | (data[10] << 16) | (data[11] << 24);
    std::cout << "文件大小: " << std::dec << fileSize << " 字节";
    std::cout << (fileSize == size ? " ✓" : " ✗ 大小不匹配") << "\n";

    // DATA 段标签
    if(size >= 20) {
        uint32_t dataTag = data[12] | (data[13] << 8) | (data[14] << 16) | (data[15] << 24);
        uint32_t expectedData = 'D' | ('A' << 8) | ('T' << 16) | ('A' << 24);
        std::cout << "DATA 段: 0x" << std::hex << dataTag;
        std::cout << (dataTag == expectedData ? " ✓ (DATA)" : " ✗ 无效") << "\n";

        uint32_t dataSize = data[16] | (data[17] << 8) | (data[18] << 16) | (data[19] << 24);
        std::cout << "DATA 大小: " << std::dec << dataSize << " 字节\n";
    }
}

int main() {
    // 构造一个最小的 TJS2 字节码文件头

        0x00, 0x00, 0x00, 0x00,  // LongArray count = 0
```

### 练习 5：模拟 Commit 的内存压缩效果

```cpp
#include <iostream>
#include <vector>
#include <cstdlib>

// 模拟编译器的动态数组管理
struct CodeAreaSimulator {
    int* data = nullptr;
    int size = 0;
    int capacity = 0;
    static const int INC_SIZE = 256;  // 每次扩容增量

    void PutCode(int value) {
        if(size >= capacity) {
            capacity += INC_SIZE;
            data = (int*)realloc(data, sizeof(int) * capacity);
            std::cout << "  扩容: capacity = " << capacity << "\n";
        }
        data[size++] = value;
    }

    void Compact() {
        if(size > 0 && data) {
            int oldCap = capacity;
            data = (int*)realloc(data, sizeof(int) * size);
            capacity = size;
            std::cout << "  压缩: " << oldCap << " → " << capacity
                      << "（节省 " << (oldCap - capacity) * (int)sizeof(int)
                      << " 字节）\n";
        }
    }

    ~CodeAreaSimulator() { free(data); }
};

int main() {
    std::cout << "=== 内存压缩演示 ===\n\n";

    CodeAreaSimulator code;

    // 模拟编译 100 条字节码
    std::cout << "编译 100 条字节码：\n";
    for(int i = 0; i < 100; i++) {
        code.PutCode(i);
    }
    std::cout << "\n编译后状态:\n";
    std::cout << "  size = " << code.size << "\n";
    std::cout << "  capacity = " << code.capacity << "\n";
    std::cout << "  浪费 = " << (code.capacity - code.size) * (int)sizeof(int)
              << " 字节\n\n";

    // 模拟 Commit 的内存压缩
    std::cout << "执行 Commit 压缩：\n";
    code.Compact();

    std::cout << "\n压缩后状态:\n";
    std::cout << "  size = " << code.size << "\n";
    std::cout << "  capacity = " << code.capacity << "\n";
    std::cout << "  浪费 = " << (code.capacity - code.size) * (int)sizeof(int)
              << " 字节\n";

    return 0;
}
```

## §9 本节小结

- **PutData 使用"回溯 20 项"的局部去重策略**：在编译速度和数据区大小之间取得工程平衡。局部重复（同一函数内的相同常量）几乎总能被去重；全局重复的少量冗余存储可以接受
- **编译期使用指针数组 _DataArea，Commit 后转换为值数组 DataArea**：指针数组支持安全的 realloc 扩容（不触及 tTJSVariant 对象本身），值数组则提供运行时的连续内存高效访问
- **常量折叠在 AST 节点构建阶段执行**：MakeNP1 支持 10 种一元运算符，MakeNP2 支持 22 种二元运算符。MakeNP3（三元运算）不支持折叠
- **FixCode 执行 4 个阶段的字节码后处理**：类成员注册代码插入 → 跳转地址修正 → 冗余跳转消除（3 种场景）→ 跳转地址格式转换
- **Commit 函数按 8 个步骤完成编译收尾**：递归提交子上下文 → 生成返回指令 → 注册到父上下文 → FixCode → 数据区转换 → 设置 MaxVariableCount → 内存压缩 → 释放临时容器
- **ExportByteCode 将上下文序列化为 12 字段头 + 5 个数据段**：代码区压缩为 16 位，数据区使用 (type, index) 对引用共享常量池
- **字节码文件格式由三段组成**：文件头（TJS2 + 100\0 + FileSize）、DATA 段（共享常量池）、OBJS 段（编译上下文数组）。反序列化时通过延迟引用解决对象间的循环依赖

### 题目 1：PutData 去重行为分析

给定以下 TJS2 代码：

```tjs
function test() {
    var a = "hello";
    var b = 42;
    var c = "world";
    var d = 42;
    var e = "hello";
}
```

假设编译器按从上到下的顺序处理每个语句，每条 `var x = val` 语句会调用 `PutData(变量名)` 和 `PutData(常量值)`。

请回答：
1. PutData 总共被调用多少次？
2. 数据区最终有多少个条目？
3. 哪些调用会触发去重复用？

<details>
<summary>查看答案</summary>

**PutData 调用次数**：10 次（每条 var 语句 2 次调用：变量名 + 常量值）

**调用顺序与去重分析**：

| 调用序号 | 参数 | 结果 | 数据区 |
|---------|------|------|--------|
| 1 | "a" | 新增 → 索引 0 | ["a"] |
| 2 | "hello" | 新增 → 索引 1 | ["a", "hello"] |
| 3 | "b" | 新增 → 索引 2 | ["a", "hello", "b"] |
| 4 | 42 | 新增 → 索引 3 | ["a", "hello", "b", 42] |
| 5 | "c" | 新增 → 索引 4 | ["a", "hello", "b", 42, "c"] |
| 6 | "world" | 新增 → 索引 5 | ["a", "hello", "b", 42, "c", "world"] |
| 7 | "d" | 新增 → 索引 6 | [..., "d"] |
| 8 | 42 | **去重** → 索引 3 | 不变（7 个条目） |
| 9 | "e" | 新增 → 索引 7 | [..., "e"] |
| 10 | "hello" | **去重** → 索引 1 | 不变（8 个条目） |

**答案**：
1. PutData 总共被调用 **10 次**
2. 数据区最终有 **8 个条目**（"a", "hello", "b", 42, "c", "world", "d", "e"）
3. 第 8 次调用（42）和第 10 次调用（"hello"）会触发去重。两者都在回溯 20 项的窗口内

</details>

### 题目 2：Commit 执行顺序推理

以下代码片段改变了 Commit 中 FixCode 和 RegisterFunction 的调用顺序：

```cpp
// 假设的修改版 Commit
void tTJSInterCodeContext::ModifiedCommit() {
    // ... 省略前面的步骤 ...
    
    // 原版：先 RegisterFunction()，后 FixCode()
    // 修改版：先 FixCode()，后 RegisterFunction()
    if(ContextType != ctProperty && ContextType != ctSuperClassGetter)
        FixCode();
    RegisterFunction();
    
    // ... 省略后面的步骤 ...
}
```

这个修改会导致什么问题？请从 `RegisterFunction` 中涉及 `PutData` 调用的角度分析。

<details>
<summary>查看答案</summary>

**会导致严重问题**。

`RegisterFunction` 中存在对 `Parent->PutData()` 的调用：

```cpp
// RegisterFunction 中的关键代码
if(Parent->ContextType == ctTopLevel) {
    tTJSVariant val;
    val = this;
    data = Parent->PutData(val);      // ← 修改父上下文的数据区！
    val = Name;
    tjs_int name = Parent->PutData(val);  // ← 再次修改！
    Parent->NonLocalFunctionDeclVector.emplace_back(data, name, changethis);
}
```

如果先执行 FixCode 后执行 RegisterFunction：

1. **FixCode 使用 `NonLocalFunctionDeclVector` 生成前置注册代码**。如果 RegisterFunction 还没有把当前函数加入这个列表，FixCode 就会遗漏这个函数的注册代码。

2. **RegisterFunction 调用 `Parent->PutData()`**。对于父上下文已经执行过 Commit 的情况，`_DataArea` 已经被转换为 `DataArea` 并释放了 `_DataArea`。此时再调用 `PutData` 会访问已释放的 `_DataArea` 指针——**产生未定义行为**（通常是崩溃）。

3. 但在 TJS2 的实际编译流程中，**子上下文的 Commit 在父上下文的 Commit 之前调用**（因为子上下文在编译完成时立即 Commit，而父上下文还在继续编译）。所以步骤 2 的问题实际上不会发生——但步骤 1 的 FixCode 时机问题是真实存在的。

**结论**：RegisterFunction 必须在 FixCode 之前执行，因为 RegisterFunction 可能向父上下文的 `NonLocalFunctionDeclVector` 添加条目，而 FixCode 需要使用这些条目来生成函数前置注册代码。

</details>

### 题目 3：字节码序列化与反序列化

给定以下 TJS2 函数：

```tjs
function add(a, b) {
    return a + b;
}
```

假设编译后的上下文有以下属性：

- parent = 0（顶层上下文索引）
- name = "add"（常量池索引 3）
- ContextType = 1（ctFunction）
- MaxVariableCount = 4
- VariableReserveCount = 2
- MaxFrameCount = 1
- FuncDeclArgCount = 2
- FuncDeclUnnamedArgArrayBase = 0
- FuncDeclCollapseBase = 0
- propSetter = -1, propGetter = -1, superClassGetter = -1

请回答：
1. ExportByteCode 序列化这 12 个头字段需要多少字节？
2. 写出这 12 个字段在二进制文件中的十六进制表示（小端序）
3. ReadObjects 反序列化时如何知道一个上下文的数据在哪里结束？

<details>
<summary>查看答案</summary>

**1. 头字段大小**：12 × 4 = **48 字节**

**2. 十六进制表示（小端序）**：

```
偏移   十六进制             字段                    值
+00    00 00 00 00         parent                  0
+04    03 00 00 00         name                    3
+08    01 00 00 00         ContextType             1 (ctFunction)
+0C    04 00 00 00         MaxVariableCount        4
+10    02 00 00 00         VariableReserveCount    2
+14    01 00 00 00         MaxFrameCount           1
+18    02 00 00 00         FuncDeclArgCount        2
+1C    00 00 00 00         FuncDeclUnnamedArgArrayBase  0
+20    00 00 00 00         FuncDeclCollapseBase    0
+24    FF FF FF FF         propSetter              -1 (0xFFFFFFFF)
+28    FF FF FF FF         propGetter              -1
+2C    FF FF FF FF         superClassGetter        -1
```

注意 -1 在 32 位有符号整数的补码表示中是 `0xFFFFFFFF`，小端序写出为 `FF FF FF FF`。

**3. 反序列化边界判定**：

ReadObjects 通过**自描述格式**确定每个上下文的数据边界：

- 读完 12 个固定长度头字段后
- 每个变长数据段都以一个 4 字节的 `count` 值开头，指明该段有多少个元素
- 依次读取：源码位置映射（count × 8 字节）→ 代码区（count × 2 字节，按 4 字节对齐）→ 数据区（count × 4 字节）→ 父类获取器指针（count × 4 字节）→ 属性列表（count × 8 字节）

每个段读完后，偏移量自然前进到下一个段的起始位置。通过这种方式，反序列化器不需要知道整个上下文的总字节数就能正确解析。

实际上 ExportByteCode 确实计算了总大小 `size = 12 * 4 + srcpossize + codesize + datasize + scgpsize + propsize + 4 * 4`，但反序列化端不依赖这个值——它通过逐段解析来确定边界。

</details>
