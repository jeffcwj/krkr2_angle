## 术语预览

本节将涉及以下术语，先有个印象，正文中会逐一详细讲解：

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| 操作码 | Opcode | 指令的第一个 32 位字，决定 VM 执行哪种操作 |
| 操作数 | Operand | 紧跟操作码之后的 32 位字，指定寄存器地址或常量池索引 |
| 字节偏移 | Byte Offset | 操作数中寄存器地址以字节为单位编码，需要除以 `sizeof(tTJSVariant)` 才得到逻辑寄存器号 |
| 条件标志 | Condition Flag | VM 内部的单个布尔变量 `flag`，由比较/测试指令写入，由跳转指令读取 |
| 属性访问器变体 | Property Accessor Variant | 一条算术指令通过宏自动生成 4 个版本：基础（Base）、直接属性（PD）、间接属性（PI）、无名属性（P） |
| 闭包 | Closure | `tTJSVariantClosure` 结构，封装了对象指针 `Object` 和 this 指针 `ObjThis` |
| 成员确保标志 | MEMBERENSURE Flag | `TJS_MEMBERENSURE` 位标志，设置后属性赋值时若成员不存在则自动创建 |

## 1 指令编码格式总览

### 1.1 32 位定长编码

TJS2 虚拟机采用**定长 32 位字**（`tjs_int32`）编码。每条指令由 1 个操作码字 + 0~N 个操作数字组成，所有字对齐在 4 字节边界上。这种设计的好处是：

- **解码简单**：`switch(*code)` 直接取第一个字即可
- **指令推进确定**：每种操作码对应固定的字数，`code += N` 即可跳到下一条指令
- **地址计算快**：跳转偏移以字节为单位，通过 `TJS_ADD_VM_CODE_ADDR` 宏直接做指针算术

源码位置：`cpp/core/tjs2/tjsInterCodeExec.cpp` 第 869-1302 行——整个 `switch(*code)` 块。

### 1.2 操作数的两种地址空间

操作数中出现的地址分属两个地址空间：

| 地址空间 | 基址寄存器 | 访问宏 | 含义 |
|----------|-----------|--------|------|
| 寄存器空间 | `ra`（Register Array，寄存器数组的起始指针） | `TJS_GET_VM_REG(ra, offset)` | 当前函数帧的局部寄存器，偏移量是 **字节偏移** |
| 常量池空间 | `da`（Data Area，常量池数组的起始指针） | `TJS_GET_VM_REG(da, offset)` | 编译期确定的常量（字符串字面量、整数、浮点数等），偏移量同样是 **字节偏移** |

> **关键陷阱**：操作数中的寄存器编号不是"第 N 个寄存器"，而是 `N * sizeof(tTJSVariant)` 的字节偏移。在 64 位平台上 `sizeof(tTJSVariant)` 通常为 24 或 32 字节。因此操作数值 `48` 在 `sizeof(tTJSVariant)==24` 的平台上表示逻辑寄存器 %2。

```
┌──────────────────────────────────────────────────────────┐
│ 指令流 (code)          │ 常量池 (da)                      │
│ ┌─────┬─────┬─────┐   │ ┌─────────────────────────────┐  │
│ │OP   │OP1  │OP2  │   │ │ da[0]: "hello"              │  │
│ ├─────┼─────┼─────┤   │ │ da[1]: 42                   │  │
│ │OP   │OP1  │     │   │ │ da[2]: 3.14                  │  │
│ └─────┴─────┴─────┘   │ └─────────────────────────────┘  │
│                        │                                  │
│ 寄存器帧 (ra)          │                                  │
│ ┌─────────────────┐   │                                  │
│ │ ra[-2]: proxy   │   │  ← 对象代理 (SuperClassProxy)    │
│ │ ra[-1]: objthis │   │  ← 当前 this 对象               │
│ │ ra[ 0]: %0 返回值│   │                                  │
│ │ ra[ 1]: %1      │   │                                  │
│ │ ra[ 2]: %2      │   │                                  │
│ │ ...             │   │                                  │
│ └─────────────────┘   │                                  │
└──────────────────────────────────────────────────────────┘
```

### 1.3 指令长度一览

不同指令占用的 32 位字数各不相同。以下是按字数分组的统计：

| 字数 | 指令 | 操作数含义 |
|------|------|-----------|
| 1 字 | `VM_NOP`, `VM_NF`, `VM_REGMEMBER`, `VM_DEBUGGER`, `VM_RET`, `VM_EXTRY` | 无操作数 |
| 2 字 | `VM_CL`, `VM_TT`, `VM_TF`, `VM_SETF`, `VM_SETNF`, `VM_LNOT`, `VM_JF`, `VM_JNF`, `VM_JMP`, `VM_INC`, `VM_DEC`, `VM_BNOT`, `VM_ASC`, `VM_CHR`, `VM_NUM`, `VM_CHS`, `VM_INV`, `VM_CHKINV`, `VM_INT`, `VM_REAL`, `VM_STR`, `VM_OCTET`, `VM_TYPEOF`, `VM_EVAL`, `VM_EEXP`, `VM_SRV`, `VM_THROW`, `VM_GLOBAL` | 1 个寄存器地址或跳转偏移 |
| 3 字 | `VM_CONST`, `VM_CP`, `VM_CCL`, `VM_CEQ`, `VM_CDEQ`, `VM_CLT`, `VM_CGT`, `VM_CHKINS`, `VM_GETP`, `VM_SETP`, `VM_CHGTHIS`, `VM_ADDCI`, 以及 14 条基础算术/逻辑指令（`VM_LOR`~`VM_MUL`） | 2 个操作数 |
| 4 字 | `VM_ENTRY`, `VM_GPD`, `VM_SPD`, `VM_SPDE`, `VM_SPDEH`, `VM_GPDS`, `VM_SPDS`, `VM_GPI`, `VM_SPI`, `VM_SPIE`, `VM_GPIS`, `VM_SPIS`, `VM_DELD`, `VM_DELI`, `VM_TYPEOFD`, `VM_TYPEOFI`, 以及 `VM_INCPD/PI/P`, `VM_DECPD/PI/P` 等属性操作变体 | 3 个操作数 |
| 5 字 | 14 条算术/逻辑的 PD/PI 变体（`VM_LORPD`~`VM_MULPD`, `VM_LORPI`~`VM_MULPI`） | 4 个操作数 |
| 4 字 | 14 条算术/逻辑的 P 变体（`VM_LORP`~`VM_MULP`） | 3 个操作数 |
| 可变长 | `VM_CALL`, `VM_NEW`, `VM_CALLD`, `VM_CALLI` | 3~4 个固定操作数 + 可变长参数列表 |

## 3 数据搬运指令

数据搬运指令负责在寄存器之间、常量池到寄存器之间移动数据。它们是最基础、出现频率最高的指令。

### 3.1 VM_NOP — 空操作

```
格式: [VM_NOP]
字数: 1
语义: 什么也不做，code 指针前进 1 字
```

用途：编译器在某些优化场景下（如删除死代码后填充）会留下 NOP。在字节码导出时，NOP 通常被优化器移除。

源码（第 872-874 行）：
```cpp
case VM_NOP:
    code++;  // 仅推进指令指针
    break;
```

### 3.2 VM_CONST — 加载常量

```
格式: [VM_CONST] [dest_reg] [data_offset]
字数: 3
语义: ra[dest_reg] = da[data_offset]  (从常量池拷贝到寄存器)
```

这是最常用的指令之一。每当 TJS2 脚本中出现字面量（字符串、整数、浮点数），编译器都会生成一条 `VM_CONST` 把常量池中的值加载到寄存器。

源码（第 876-880 行）：
```cpp
case VM_CONST:
    // 把常量池 da 中偏移 code[2] 处的值拷贝到寄存器 ra 中偏移 code[1] 处
    TJS_GET_VM_REG(ra, code[1]).CopyRef(TJS_GET_VM_REG(da, code[2]));
    code += 3;
    break;
```

> **注意**：`CopyRef` 是引用拷贝（Reference Copy），对于字符串等引用类型不会深拷贝内容，而是增加引用计数。这避免了字符串常量每次使用都产生内存分配。

**TJS2 脚本对照示例**：

```javascript
// TJS2 脚本
var x = "hello";
var y = 42;
```

编译后的字节码（伪反汇编）：
```
0000: VM_CONST  %3, *0    // %3 = da[0] = "hello"
0003: VM_CONST  %4, *1    // %4 = da[1] = 42
```

### 3.3 VM_CP — 寄存器拷贝

```
格式: [VM_CP] [dest_reg] [src_reg]
字数: 3
语义: ra[dest_reg] = ra[src_reg]  (寄存器间拷贝)
```

用于变量赋值、参数传递等场景。

源码（第 882-886 行）：
```cpp
case VM_CP:
    TJS_GET_VM_REG(ra, code[1]).CopyRef(TJS_GET_VM_REG(ra, code[2]));
    code += 3;
    break;
```

**TJS2 脚本对照**：
```javascript
var a = 10;
var b = a;  // 生成 VM_CP %b, %a
```

### 3.4 VM_CL — 清除寄存器

```
格式: [VM_CL] [reg]
字数: 2
语义: ra[reg].Clear()  (将寄存器置为 void/空)
```

将寄存器重置为 `void` 类型。这对于 GC（垃圾回收，Garbage Collection）很重要——清除寄存器中的对象引用可以让对象尽早被回收。

源码（第 888-891 行）：
```cpp
case VM_CL:
    TJS_GET_VM_REG(ra, code[1]).Clear();
    code += 2;
    break;
```

> **历史注记**：编译器中有一个 `ClearFrame` 函数（`tjsInterCodeGen.cpp`），其内部的 `VM_CL` / `VM_CCL` 生成代码全部被 `#if 0` 注释掉了，意味着当前版本不会在函数入口自动清除帧。寄存器的初始化完全依赖各指令本身。

### 3.5 VM_CCL — 连续清除

```
格式: [VM_CCL] [start_reg] [count]
字数: 3
语义: 从 ra[start_reg] 开始，连续清除 count 个寄存器
特殊: code[2] 是元素个数，不是字节偏移！
```

这是 TJS2 指令集中少有的**操作数不是字节偏移**的例外。`code[2]` 直接表示要清除的 `tTJSVariant` 个数。

源码（第 893-896 行 + 第 1391-1397 行）：
```cpp
case VM_CCL:
    ContinuousClear(ra, code);
    code += 3;
    break;

// ContinuousClear 实现
void tTJSInterCodeContext::ContinuousClear(
    tTJSVariant *ra, const tjs_int32 *code) {
    tTJSVariant *r  = TJS_GET_VM_REG_ADDR(ra, code[1]);  // 起始地址
    tTJSVariant *rl = r + code[2];  // code[2] 是个数，不是字节偏移!
    while(r < rl)
        (r++)->Clear();  // 逐个调用 Clear()
}
```

> **常见错误**：如果误把 `code[2]` 当做字节偏移（像其他操作数一样除以 `sizeof(tTJSVariant)`），就会少清除很多寄存器。这是阅读源码时容易掉进的陷阱。

### 3.6 VM_GLOBAL — 获取全局对象

```
格式: [VM_GLOBAL] [dest_reg]
字数: 2
语义: ra[dest_reg] = TJS 全局对象
```

将 TJS2 引擎的全局命名空间对象（`tTJS::GetGlobalNoAddRef()` 返回值）写入寄存器。之后可以通过属性访问指令在全局对象上读写全局变量。

源码（第 1279-1283 行）：
```cpp
case VM_GLOBAL:
    TJS_GET_VM_REG(ra, code[1]) = Block->GetTJS()->GetGlobalNoAddRef();
    code += 2;
    break;
```

**TJS2 脚本对照**：
```javascript
// 在函数内访问全局变量
function test() {
    global.someGlobalVar = 100;
    //  ^^^^^^ 编译器先生成 VM_GLOBAL 把全局对象加载到一个临时寄存器
    //         然后用 VM_SPDE 在该对象上设置属性 "someGlobalVar"
}
```

## 4 条件标志指令

TJS2 VM 使用一个独立的布尔变量 `flag` 来存储比较结果，而不是把布尔值写入寄存器。这种设计减少了寄存器压力，因为条件判断通常紧跟跳转指令使用，无需长期保存。

### 4.1 VM_TT — 测试为真

```
格式: [VM_TT] [reg]
字数: 2
语义: flag = (bool)(ra[reg])
```

将寄存器的值转换为布尔类型并写入 `flag`。`tTJSVariant::operator bool()` 的转换规则：
- `void` → `false`
- 整数 0 → `false`，非零 → `true`
- 空字符串 → `false`，非空 → `true`
- 对象 `nullptr` → `false`，非空 → `true`

源码（第 898-901 行）：
```cpp
case VM_TT:
    flag = TJS_GET_VM_REG(ra, code[1]).operator bool();
    code += 2;
    break;
```

### 4.2 VM_TF — 测试为假

```
格式: [VM_TF] [reg]
字数: 2
语义: flag = !(bool)(ra[reg])
```

与 `VM_TT` 相反，测试结果取反。这条指令的存在是为了优化 `if(!condition)` 模式——不需要先 `VM_TT` 再 `VM_NF`，一条指令搞定。

源码（第 903-906 行）：
```cpp
case VM_TF:
    flag = !(TJS_GET_VM_REG(ra, code[1]).operator bool());
    code += 2;
    break;
```

### 4.3 VM_CEQ — 相等比较

```
格式: [VM_CEQ] [reg1] [reg2]
字数: 3
语义: flag = (ra[reg1] == ra[reg2])  (宽松比较)
```

调用 `NormalCompare` 进行**类型转换后比较**（类似 JavaScript 的 `==`）。例如整数 1 和字符串 "1" 比较结果为 `true`。

源码（第 908-912 行）：
```cpp
case VM_CEQ:
    flag = TJS_GET_VM_REG(ra, code[1])
               .NormalCompare(TJS_GET_VM_REG(ra, code[2]));
    code += 3;
    break;
```

### 4.4 VM_CDEQ — 严格相等比较

```
格式: [VM_CDEQ] [reg1] [reg2]
字数: 3
语义: flag = (ra[reg1] === ra[reg2])  (严格比较，不做类型转换)
```

调用 `DiscernCompare` 进行**类型敏感比较**（类似 JavaScript 的 `===`）。类型不同直接返回 `false`。

源码（第 914-918 行）：
```cpp
case VM_CDEQ:
    flag = TJS_GET_VM_REG(ra, code[1])
               .DiscernCompare(TJS_GET_VM_REG(ra, code[2]));
    code += 3;
    break;
```

### 4.5 VM_CLT — 小于比较（注意操作数反转！）

```
格式: [VM_CLT] [reg1] [reg2]
字数: 3
语义: flag = (ra[reg1] < ra[reg2])
实现: flag = ra[reg1].GreaterThan(ra[reg2])  ← 注意调用的是 GreaterThan!
```

这是 TJS2 源码中一个著名的**阅读陷阱**：`VM_CLT`（Compare Less Than）调用的是 `GreaterThan` 方法。原因是 `GreaterThan(other)` 的语义是 `other > this`，即 `this < other`，恰好就是"小于"。

源码（第 920-924 行）：
```cpp
case VM_CLT:
    flag = TJS_GET_VM_REG(ra, code[1])
               .GreaterThan(TJS_GET_VM_REG(ra, code[2]));
    code += 3;
    break;
```

### 4.6 VM_CGT — 大于比较（同样反转！）

```
格式: [VM_CGT] [reg1] [reg2]
字数: 3
语义: flag = (ra[reg1] > ra[reg2])
实现: flag = ra[reg1].LittlerThan(ra[reg2])  ← 调用的是 LittlerThan!
```

与 `VM_CLT` 对称，`VM_CGT`（Compare Greater Than）调用 `LittlerThan`（注意原始代码中 "Littler" 是拼写错误，正确应为 "Lesser"，但这是原项目的命名）。

源码（第 926-930 行）：
```cpp
case VM_CGT:
    flag = TJS_GET_VM_REG(ra, code[1])
               .LittlerThan(TJS_GET_VM_REG(ra, code[2]));
    code += 3;
    break;
```

> **为什么 CLT 调用 GreaterThan？** 因为 `tTJSVariant::GreaterThan(rhs)` 的实现语义是 "rhs > this"。当我们写 `a.GreaterThan(b)` 时实际测试的是 `b > a`，即 `a < b`。这种"操作数反转"在许多早期脚本引擎中都存在。

### 4.7 VM_SETF / VM_SETNF — 将 flag 写入寄存器

```
VM_SETF:
  格式: [VM_SETF] [reg]
  字数: 2
  语义: ra[reg] = flag

VM_SETNF:
  格式: [VM_SETNF] [reg]
  字数: 2
  语义: ra[reg] = !flag
```

当需要把比较结果保存到变量中（而非立刻跳转）时使用。例如 `var result = (a == b);`。

源码（第 932-940 行）：
```cpp
case VM_SETF:
    TJS_GET_VM_REG(ra, code[1]) = flag;  // tTJSVariant 会转为整数 0/1
    code += 2;
    break;

case VM_SETNF:
    TJS_GET_VM_REG(ra, code[1]) = !flag;
    code += 2;
    break;
```

### 4.8 VM_LNOT — 逻辑非

```
格式: [VM_LNOT] [reg]
字数: 2
语义: ra[reg] = !ra[reg]  (逻辑非，结果为 0 或 1)
```

注意 `VM_LNOT` 操作的是**寄存器内容**（就地修改），而非 `flag`。对应 TJS2 脚本中的 `!expr` 运算符。

源码（第 942-945 行）：
```cpp
case VM_LNOT:
    TJS_GET_VM_REG(ra, code[1]).logicalnot();
    code += 2;
    break;
```

### 4.9 VM_NF — 标志取反

```
格式: [VM_NF]
字数: 1
语义: flag = !flag
```

无操作数的单字指令，直接对 `flag` 取反。用于优化 `if(!(a==b))` 等模式。

源码（第 947-950 行）：
```cpp
case VM_NF:
    flag = !flag;
    code++;
    break;
```

## 5 跳转与流程控制指令

### 5.1 VM_JF — 条件跳转（flag 为 true 时跳）

```
格式: [VM_JF] [offset]
字数: 2 (但跳转后 code 指针变化)
语义: if(flag) goto (code + offset_bytes); else code += 2;
```

`code[1]` 是以**字节**为单位的相对偏移量。跳转使用 `TJS_ADD_VM_CODE_ADDR` 宏，它直接在 `code` 指针上做字节级加法。

源码（第 952-957 行）：
```cpp
case VM_JF:
    if(flag)
        TJS_ADD_VM_CODE_ADDR(code, code[1]);  // 字节偏移跳转
    else
        code += 2;  // 不跳转，前进到下一条指令
    break;
```

> **名称容易混淆**：`VM_JF` 是 "Jump if Flag"（flag 为 true 时跳），不是 "Jump if False"。想表达"为假时跳"要用 `VM_JNF`。

### 5.2 VM_JNF — 条件跳转（flag 为 false 时跳）

```
格式: [VM_JNF] [offset]
字数: 2
语义: if(!flag) goto (code + offset_bytes); else code += 2;
```

源码（第 959-964 行）：
```cpp
case VM_JNF:
    if(!flag)
        TJS_ADD_VM_CODE_ADDR(code, code[1]);
    else
        code += 2;
    break;
```

### 5.3 VM_JMP — 无条件跳转

```
格式: [VM_JMP] [offset]
字数: 2
语义: goto (code + offset_bytes)
```

无条件跳转。`while` 循环的回跳、`break` 的前跳都用这条指令。

源码（第 966-968 行）：
```cpp
case VM_JMP:
    TJS_ADD_VM_CODE_ADDR(code, code[1]);
    break;
```

### 5.4 跳转偏移计算详解

跳转偏移是相对于**当前指令起始位置**的字节偏移。计算方式：

```
目标地址 = 当前 code 指针 + offset 字节

由于 code 是 tjs_int32* 类型，TJS_ADD_VM_CODE_ADDR 宏
先把 code 转为 char*，加上 offset 字节，再转回 tjs_int32*。
```

**完整的条件分支流程**：

```
TJS2 脚本:  if (x > 0) { y = 1; } else { y = 2; }

字节码:
0000: VM_CONST  %3, *0     // %3 = 0
0003: VM_CGT    %2, %3     // flag = (%2 > %3)，即 x > 0
0006: VM_JNF    +20        // if(!flag) 跳到 else 分支 (偏移 20 字节 = 5 个 int32)
0008: VM_CONST  %4, *1     // %4 = 1  (then 分支)
0011: VM_CP     %5, %4     // y = 1
0014: VM_JMP    +16        // 跳过 else 分支
0016: VM_CONST  %4, *2     // %4 = 2  (else 分支)
0019: VM_CP     %5, %4     // y = 2
0022: ...                   // 后续代码
```

## 6 算术与逻辑运算指令

这是 TJS2 指令集中数量最多的一类，通过 `TJS_NORMAL_AND_PROPERTY_ACCESSER` 宏和 `TJS_DEF_VM_P` 宏，14 条二元运算指令各自展开为 4 个变体，加上自增/自减的 4 个变体和 3 条一元运算，总共 67 条指令。

### 6.1 宏展开机制

#### TJS_NORMAL_AND_PROPERTY_ACCESSER 宏

定义在 `tjsInterCodeGen.h` 第 42 行：

```cpp
#define TJS_NORMAL_AND_PROPERTY_ACCESSER(x)  x, x##PD, x##PI, x##P
```

这个宏在枚举定义中使用，将一个操作码名 `x` 展开为 4 个枚举值：
- `x` — 基础版本，直接操作寄存器
- `x##PD` — Property Direct，通过常量池中的名字访问对象属性
- `x##PI` — Property Indirect，通过寄存器中的名字访问对象属性
- `x##P` — Property (无名)，对属性对象本身进行操作

#### TJS_DEF_VM_P 宏

定义在 `tjsInterCodeExec.cpp` 第 1010-1026 行，用于在 `switch` 语句中生成 4 个 `case` 分支：

```cpp
#define TJS_DEF_VM_P(vmcode, rope)                          \
    case VM_##vmcode:                                       \
        TJS_GET_VM_REG(ra, code[1]).rope(                   \
            TJS_GET_VM_REG(ra, code[2]));                   \
        code += 3;                                          \
        break;                                              \
    case VM_##vmcode##PD:                                   \
        OperatePropertyDirect(ra, code, TJS_OP_##vmcode);   \
        code += 5;                                          \
        break;                                              \
    case VM_##vmcode##PI:                                   \
        OperatePropertyIndirect(ra, code, TJS_OP_##vmcode); \
        code += 5;                                          \
        break;                                              \
    case VM_##vmcode##P:                                    \
        OperateProperty(ra, code, TJS_OP_##vmcode);         \
        code += 4;                                          \
        break
```

**4 个变体的操作数格式与语义对比**：

| 变体 | 格式 | 字数 | 语义 | 调用函数 |
|------|------|------|------|---------|
| Base | `[OP] [dest] [src]` | 3 | `ra[dest] OP= ra[src]` | 直接调用 `tTJSVariant` 的运算方法 |
| PD | `[OP_PD] [result] [obj] [name] [param]` | 5 | `ra[result] = ra[obj].da[name] OP ra[param]` | `OperatePropertyDirect` |
| PI | `[OP_PI] [result] [obj] [key] [param]` | 5 | `ra[result] = ra[obj][ra[key]] OP ra[param]` | `OperatePropertyIndirect` |
| P | `[OP_P] [result] [obj] [param]` | 4 | `ra[result] = *ra[obj] OP ra[param]` | `OperateProperty` |

### 6.2 自增/自减指令

`VM_INC` 和 `VM_DEC` 的 4 个变体单独实现（不通过 `TJS_DEF_VM_P`），因为它们是一元操作，无第二操作数。

```
VM_INC:   [VM_INC] [reg]           → ra[reg]++        (2 字)
VM_INCPD: [VM_INCPD] [r] [obj] [name]  → ra[obj].da[name]++  (4 字)
VM_INCPI: [VM_INCPI] [r] [obj] [key]   → ra[obj][ra[key]]++   (4 字)
VM_INCP:  [VM_INCP] [r] [obj]          → (*ra[obj])++          (3 字)

VM_DEC 同理，把 ++ 换成 --
```

源码（第 970-1008 行）：
```cpp

    code += 2;
    break;

case VM_INCPD:
    OperatePropertyDirect0(ra, code, TJS_OP_INC);  // 0 = 无参数版

    break;

case VM_INCPI:
    OperatePropertyIndirect0(ra, code, TJS_OP_INC);

### 6.5 OperatePropertyDirect 系列函数详解

属性上的复合运算（如 `obj.x += 1`）通过 `OperatePropertyDirect` / `OperatePropertyIndirect` / `OperateProperty` 三个函数实现。它们的核心是调用 `iTJSDispatch2::Operation` 方法，该方法在对象内部原子地完成"读取→运算→写回"操作。

**OperatePropertyDirect** 源码（第 1607-1630 行）：

```cpp
void tTJSInterCodeContext::OperatePropertyDirect(
    tTJSVariant *ra, const tjs_int32 *code, tjs_uint32 ope) const {
    // 语义: ra[code[1]] = ope(ra[code[2]][DataArea[code[3]]], ra[code[4]])
    //
    // code[1] = 结果寄存器（0 表示丢弃结果）
    // code[2] = 对象寄存器
    // code[3] = 属性名（常量池索引）
    // code[4] = 参数寄存器

    tTJSVariantClosure clo = TJS_GET_VM_REG(ra, code[2]).AsObjectClosure();
    tjs_error hr;
    try {
        tTJSVariant *name = TJS_GET_VM_REG_ADDR(DataArea, code[3]);
        hr = clo.Operation(
            ope,                    // 操作类型 (TJS_OP_ADD 等)
            name->GetString(),      // 属性名字符串
            name->GetHint(),        // 哈希提示 (加速查找)
            code[1] ? TJS_GET_VM_REG_ADDR(ra, code[1]) : nullptr,  // 结果
            TJS_GET_VM_REG_ADDR(ra, code[4]),  // 运算参数
            clo.ObjThis ? clo.ObjThis : ra[-1].AsObjectNoAddRef());
    } catch(...) {
        clo.Release();
        throw;
    }
    clo.Release();
    if(TJS_FAILED(hr))
        TJSThrowFrom_tjs_error(hr, ...);
}
```

> **code[1] 为 0 的特殊含义**：当 `code[1]` 为 0 时，传入 `nullptr` 作为结果指针，表示"丢弃运算结果"。这对应 TJS2 脚本中 `obj.x += 1;` 且不使用返回值的场景。

## 8 属性访问指令

属性访问是 TJS2 脚本最常用的操作之一——`obj.name`、`obj[key]`、`delete obj.name` 等。TJS2 VM 提供了 16 条属性访问指令，按 **三个维度** 组合：

1. **方向**：Get（读取）vs Set（写入）vs Delete（删除）
2. **键的来源**：Direct（常量池中的字符串名）vs Indirect（寄存器中的动态键）vs 无名（对属性对象本身操作）
3. **标志位**：是否带 `MEMBERENSURE`（成员确保，不存在时自动创建）、`HIDDENMEMBER`（隐藏成员）、`IGNOREPROP`（忽略属性触发器）

### 8.1 直接属性读取（GPD 系列）

"直接"指属性名来自常量池（编译期确定的字符串字面量）。

```
VM_GPD:   [VM_GPD]  [dest] [obj] [name]   → ra[dest] = ra[obj].da[name]       flags=0
VM_GPDS:  [VM_GPDS] [dest] [obj] [name]   → ra[dest] = ra[obj].da[name]       flags=IGNOREPROP
字数: 4
```

- `VM_GPD`：普通属性读取。如果属性是 property（带 getter），会触发 getter 调用
- `VM_GPDS`：带 `TJS_IGNOREPROP` 标志，绕过 property getter，直接读取底层值

源码（第 1166-1174 行）：
```cpp
case VM_GPD:
    GetPropertyDirect(ra, code, 0);             // flags = 0
    code += 4;
    break;

case VM_GPDS:
    GetPropertyDirect(ra, code, TJS_IGNOREPROP); // 绕过 getter
    code += 4;
    break;
```

**GetPropertyDirect 函数内部流程**（第 1400-1427 行）：

```
1. 检查 ra[code[2]] 的类型
   ├─ tvtString → 调用 GetStringProperty (字符串内置属性如 .length)
   ├─ tvtOctet  → 调用 GetOctetProperty  (字节流内置属性)
   └─ 其他      → 获取闭包，调用 clo.PropGet(flags, name, hint, dest, objthis)
                   如果失败(TJS_FAILED)，抛出异常
```

**特殊的字符串/字节流处理**：当对字符串或字节流调用属性访问时，TJS2 不会把它们当做对象，而是直接调用内置的字符串/字节流属性处理函数。例如 `"hello".length` 不需要创建字符串对象。

### 8.2 直接属性写入（SPD 系列）

```
VM_SPD:   [VM_SPD]  [obj] [name] [val]   → ra[obj].da[name] = ra[val]   flags=0
VM_SPDE:  [VM_SPDE] [obj] [name] [val]   → 同上    flags=MEMBERENSURE
VM_SPDEH: [VM_SPDEH][obj] [name] [val]   → 同上    flags=MEMBERENSURE|HIDDENMEMBER
VM_SPDS:  [VM_SPDS] [obj] [name] [val]   → 同上    flags=MEMBERENSURE|IGNOREPROP
字数: 4
```

四种变体的标志位含义：

| 指令 | 标志 | 用途 |
|------|------|------|
| `VM_SPD` | 无 | 普通赋值，成员必须已存在，会触发 setter |
| `VM_SPDE` | `MEMBERENSURE` | 赋值时若成员不存在则自动创建（用于 `var x = ...` 或 `obj.newProp = ...`） |
| `VM_SPDEH` | `MEMBERENSURE \| HIDDENMEMBER` | 创建隐藏成员（不会被 `for...in` 枚举到） |
| `VM_SPDS` | `MEMBERENSURE \| IGNOREPROP` | 创建成员并绕过 setter（直接写底层值） |

源码（第 1176-1196 行）：
```cpp
case VM_SPD:
    SetPropertyDirect(ra, code, 0);
    code += 4;
    break;

case VM_SPDE:
    SetPropertyDirect(ra, code, TJS_MEMBERENSURE);
    code += 4;
    break;

case VM_SPDEH:
    SetPropertyDirect(ra, code, TJS_MEMBERENSURE | TJS_HIDDENMEMBER);
    code += 4;
    break;

case VM_SPDS:
    SetPropertyDirect(ra, code, TJS_MEMBERENSURE | TJS_IGNOREPROP);
    code += 4;
    break;
```

**SetPropertyDirect 的双重尝试策略**（第 1430-1461 行）：

```cpp
// 先尝试 PropSetByVS (VS = VariantString, 更高效的内部接口)
tjs_error hr = clo.PropSetByVS(flags, name->AsStringNoAddRef(), ...);

if(hr == TJS_E_NOTIMPL)
    // 如果对象不支持 PropSetByVS，退回到通用 PropSet
    hr = clo.PropSet(flags, name->GetString(), name->GetHint(), ...);
```

这种"先快后慢"的策略是性能优化：`PropSetByVS` 使用 `tTJSVariantString*` 直接传递，避免了字符串哈希计算；如果对象不支持这个接口（返回 `TJS_E_NOTIMPL`），再退回到标准的 `PropSet`。

### 8.3 间接属性访问（GPI / SPI 系列）

"间接"指属性键来自寄存器（运行时动态确定的值），支持字符串键和整数键。

```
VM_GPI:   [VM_GPI]  [dest] [obj] [key]   → ra[dest] = ra[obj][ra[key]]   flags=0
VM_GPIS:  [VM_GPIS] [dest] [obj] [key]   → 同上    flags=IGNOREPROP
VM_SPI:   [VM_SPI]  [obj]  [key] [val]   → ra[obj][ra[key]] = ra[val]    flags=0
VM_SPIE:  [VM_SPIE] [obj]  [key] [val]   → 同上    flags=MEMBERENSURE
VM_SPIS:  [VM_SPIS] [obj]  [key] [val]   → 同上    flags=MEMBERENSURE|IGNOREPROP
字数: 4
```

**间接访问的整数键优化**（`GetPropertyIndirect`，第 1490-1536 行）：

```cpp
tTJSVariant *ra_code3 = TJS_GET_VM_REG_ADDR(ra, code[3]);
if(ra_code3->Type() != tvtInteger) {
    // 字符串键: 调用 clo.PropGet(flags, str, nullptr, dest, objthis)
    tTJSVariantString *str = ra_code3->AsString();
    hr = clo.PropGet(flags, *str, nullptr, dest, objthis);
} else {
    // 整数键: 调用 clo.PropGetByNum(flags, index, dest, objthis)
    hr = clo.PropGetByNum(flags, (tjs_int)ra_code3->AsInteger(), dest, objthis);
}
```

当键是整数时，使用 `PropGetByNum` 而非 `PropGet`。这对数组访问（`arr[0]`, `arr[1]`）是重要的优化，因为 `PropGetByNum` 可以直接用整数索引而无需字符串哈希查找。

### 8.4 无名属性访问（GETP / SETP）

"无名"指对属性对象本身进行读写，不指定属性名。用于 property 对象的直接取值/赋值。

```
VM_GETP:  [VM_GETP] [dest] [prop_obj]   → ra[dest] = *ra[prop_obj]    (3 字)
VM_SETP:  [VM_SETP] [prop_obj] [val]     → *ra[prop_obj] = ra[val]     (3 字)
```

### 8.5 成员删除（DELD / DELI）

```
VM_DELD:  [VM_DELD] [result] [obj] [name]   → ra[result] = delete ra[obj].da[name]   (4 字)
VM_DELI:  [VM_DELI] [result] [obj] [key]    → ra[result] = delete ra[obj][ra[key]]    (4 字)
```

删除操作的返回值是布尔值：`true` 表示删除成功，`false` 表示成员不存在或删除失败。当 `code[1]`（result 寄存器）为 0 时，丢弃返回值。

源码（`DeleteMemberDirect`，第 1806-1827 行）：
```cpp
void tTJSInterCodeContext::DeleteMemberDirect(
    tTJSVariant *ra, const tjs_int32 *code) {
    tTJSVariantClosure clo = TJS_GET_VM_REG(ra, code[2]).AsObjectClosure();
    tjs_error hr;
    try {
        tTJSVariant *name = TJS_GET_VM_REG_ADDR(DataArea, code[3]);

                              ra[-1].AsObjectNoAddRef());

        throw;

    clo.Release();

    if(code[1]) {  // code[1] 非 0 时才写结果
        if(TJS_FAILED(hr))
            TJS_GET_VM_REG(ra, code[1]) = false;
        else
            TJS_GET_VM_REG(ra, code[1]) = true;
    }
}
```

## 9 函数调用指令

TJS2 有 4 条函数调用指令，它们共享一个复杂的参数传递宏 `TJS_BEGIN_FUNC_CALL_ARGS`。

### 9.1 四条调用指令一览

| 操作码 | 用途 | 固定操作数 | 语义 |
|--------|------|-----------|------|
| `VM_CALL` | 普通函数调用 | `[VM_CALL] [result] [func] [args...]` | `ra[result] = ra[func](args...)` |
| `VM_NEW` | 创建新对象 | `[VM_NEW] [result] [class] [args...]` | `ra[result] = new ra[class](args...)` |
| `VM_CALLD` | 直接成员函数调用 | `[VM_CALLD] [result] [obj] [name] [args...]` | `ra[result] = ra[obj].da[name](args...)` |
| `VM_CALLI` | 间接成员函数调用 | `[VM_CALLI] [result] [obj] [key] [args...]` | `ra[result] = ra[obj][ra[key]](args...)` |

> **VM_CALL 和 VM_NEW 共用同一个 case 入口**（第 1153-1156 行）：
> ```cpp
> case VM_CALL:
> case VM_NEW:
>     code += CallFunction(ra, code, args, numargs);
>     break;
> ```
> 在 `CallFunction` 内部通过检查 `code[0] == VM_CALL` 还是 `VM_NEW` 来分支。

### 9.2 参数传递的三种模式

函数调用指令后面的参数列表由 `TJS_BEGIN_FUNC_CALL_ARGS` 宏处理。参数计数字段（`code[0]` 在参数区域的开头）决定了三种模式：

#### 模式 1：直接参数（count ≥ 0）

最常见的模式。参数计数 ≥ 0 时，后面跟着 `count` 个寄存器偏移，每个指向一个参数值。

```
参数区布局: [count] [arg0_reg] [arg1_reg] ... [argN_reg]
code_size = count + 1
```

源码（第 2072-2082 行）：
```cpp
} else if(pass_args_count <= TJS_PASS_ARGS_PREPARED_ARRAY_COUNT) {
    code_size = pass_args_count + 1;
    pass_args = pass_args_p;  // 使用栈上预分配的 20 元素数组
    for(tjs_int i = 0; i < pass_args_count; i++)
        pass_args[i] = TJS_GET_VM_REG_ADDR(ra, (_code)[1 + i]);
} else {
    // 参数超过 20 个时，堆分配
    pass_args = new tTJSVariant *[pass_args_count];
    alloc_args = true;
    ...
}
```

> **性能优化**：参数指针数组在栈上预分配了 20 个位置（`TJS_PASS_ARGS_PREPARED_ARRAY_COUNT = 20`）。只有当参数超过 20 个时才动态分配内存。大多数函数调用参数不超过 5 个，所以几乎从不需要堆分配。

#### 模式 2：透传参数（count = -1）

当 count 为 -1 时，直接把当前函数接收到的 `args` 和 `numargs` 传递给被调用函数，不做任何复制。这用于 `...`（省略号）参数传递。

```
参数区布局: [-1]
code_size = 1
```

源码（第 1999-2004 行）：
```cpp
if(pass_args_count == -1) {
    // 省略参数: 直接把调用者的参数原封不动传递
    pass_args = args;
    pass_args_count = numargs;
    code_size = 1;
}
```

#### 模式 3：数组展开参数（count = -2）

最复杂的模式，支持三种参数类型的混合：

```
参数区布局: [-2] [arg_written_count]
            [type0] [reg0] [type1] [reg1] ... 
code_size = arg_written_count * 2 + 2
```

每个参数前有一个类型标签（`fatNormal`/`fatExpand`/`fatUnnamedExpand`）：

| 类型标签 | 值 | 含义 | TJS2 脚本语法 |
|----------|---|------|-------------|
| `fatNormal` | 0 | 普通参数 | `func(a, b)` 中的 `a`、`b` |
| `fatExpand` | 1 | 数组展开 | `func(arr*)` — 将数组的每个元素作为独立参数 |
| `fatUnnamedExpand` | 2 | 未命名参数展开 | `func(...)` — 将当前函数的剩余参数传递 |

源码（第 2005-2071 行）展示了两遍扫描算法：
1. **第一遍**：遍历所有标签，计算总参数数量（包括展开后的数组元素数）
2. **第二遍**：遍历所有标签，把参数指针填入 `pass_args` 数组

```cpp
// 第一遍: 计数
for(tjs_int i = 0; i < arg_written_count; i++) {
    switch((_code)[i * 2 + 2]) {
        case fatNormal:   pass_args_count++; break;
        case fatExpand:
            args_v_count += TJSGetArrayElementCount(...);  // 获取数组长度
            break;
        case fatUnnamedExpand:
            pass_args_count += (numargs > FuncDeclUnnamedArgArrayBase)
                ? (numargs - FuncDeclUnnamedArgArrayBase) : 0;
            break;
    }
}

// 第二遍: 填充
for(tjs_int i = 0; i < arg_written_count; i++) {
    switch((_code)[i * 2 + 2]) {
        case fatNormal:
            pass_args[pass_args_count++] = TJS_GET_VM_REG_ADDR(ra, ...);
            break;
        case fatExpand: {
            tjs_int count = TJSCopyArrayElementTo(...);
            for(tjs_int j = 0; j < count; j++)
                pass_args[pass_args_count++] = pass_args_v + j + args_v_count;
            break;
        }
        case fatUnnamedExpand: {
            tjs_int count = ...;
            for(tjs_int j = 0; j < count; j++)
                pass_args[pass_args_count++] = args[FuncDeclUnnamedArgArrayBase + j];
            break;
        }
    }
}
```

### 9.3 VM_CALL / VM_NEW 详解

`CallFunction`（第 2100-2143 行）处理 `VM_CALL` 和 `VM_NEW` 两条指令：

```
指令布局: [VM_CALL/VM_NEW] [result_reg] [func_reg] [args_section...]
           code[0]          code[1]      code[2]    code[3] 开始
```

**VM_CALL 路径**：
```cpp
if(code[0] == VM_CALL) {
    hr = clo.FuncCall(
        0,                     // flags
        nullptr,               // 函数名 (无名调用)
        nullptr,               // 哈希提示
        code[1] ? TJS_GET_VM_REG_ADDR(ra, code[1]) : nullptr,  // 结果
        pass_args_count,       // 参数数量
        pass_args,             // 参数指针数组
        clo.ObjThis ? clo.ObjThis : ra[-1].AsObjectNoAddRef()); // this
}
```

**VM_NEW 路径**：
```cpp
else {
    iTJSDispatch2 *dsp;
    hr = clo.CreateNew(
        0, nullptr, nullptr,
        &dsp,                  // 输出: 新创建的对象
        pass_args_count, pass_args,
        clo.ObjThis ? clo.ObjThis : ra[-1].AsObjectNoAddRef());
    if(TJS_SUCCEEDED(hr)) {
        if(dsp) {
            if(code[1])
                TJS_GET_VM_REG(ra, code[1]) = tTJSVariant(dsp, dsp);
            dsp->Release();    // 减少引用计数 (tTJSVariant 内部已加)
        }
    }
}
```

> **VM_NEW 的 this 绑定**：`tTJSVariant(dsp, dsp)` 创建的闭包，`Object` 和 `ObjThis` 都指向新建对象本身。

### 9.4 VM_CALLD 的字符串/字节流特殊处理

`CallFunctionDirect`（第 2148-2193 行）在调用成员函数之前检查对象类型：

```
指令布局: [VM_CALLD] [result_reg] [obj_reg] [name_data] [args_section...]
```

```cpp
tTJSVariantType type = TJS_GET_VM_REG(ra, code[2]).Type();
if(type == tvtString) {
    // 字符串对象: 调用内置字符串方法
    ProcessStringFunction(name, target_string, pass_args, ...);
} else if(type == tvtOctet) {
    // 字节流对象: 调用内置字节流方法
    ProcessOctetFunction(name, target_octet, pass_args, ...);
} else {
    // 普通对象: 调用 clo.FuncCall(0, name, hint, result, ...)
}
```

**ProcessStringFunction** 支持 13 种内置字符串方法，使用**哈希+字符串比较**的两步匹配：

```
charAt, indexOf, toUpperCase, toLowerCase, substring, substr,
sprintf, replace, escape, split, trim, reverse, repeat
```

**ProcessOctetFunction** 仅支持 1 种方法：`unpack`。

### 9.5 调用指令的返回值与 code 推进

所有调用函数都返回一个 `tjs_int` 值，表示本条指令占用的字数。调度循环用 `code += 返回值` 推进到下一条指令。

```
VM_CALL / VM_NEW:  返回 code_size + 3  (3 个固定操作数 + 参数区大小)
VM_CALLD:          返回 code_size + 4  (4 个固定操作数 + 参数区大小)
VM_CALLI:          返回 code_size + 4
```

## 10 异常与返回指令

### 10.1 VM_SRV — 设置返回值

```
格式: [VM_SRV] [reg]
字数: 2
语义: 如果调用者提供了 result 指针，则 *result = ra[reg]
```

`VM_SRV`（Set Return Value）不会终止函数执行——它只是把值拷贝到调用者传入的 `result` 指针中。通常紧跟 `VM_RET`。

源码（第 1244-1248 行）：
```cpp
case VM_SRV:
    if(result)  // result 可能为 nullptr (调用者不关心返回值)

    break;
```

### 10.2 VM_RET — 函数返回

```
格式: [VM_RET]
字数: 1
语义: 终止当前 ExecuteCode 调用，返回下一条指令的 IP
```

`VM_RET` 是最简洁的指令之一——直接 `return`，不修改任何寄存器。

源码（第 1250-1251 行）：
```cpp
case VM_RET:
    return code + 1 - CodeArea;  // 返回下一条指令的偏移量
```

> **返回值是 IP 偏移**，不是布尔值。`ExecuteCode` 的返回类型是 `tjs_int`，表示"执行到了 CodeArea 中的第几个字"。

### 10.3 VM_ENTRY — 进入 try 块

```
格式: [VM_ENTRY] [catch_offset] [exception_reg]
字数: 3
语义: 执行 try 块，如果抛出异常则跳到 catch 块
```

操作数含义：
- `code[1]`：catch 块的**代码偏移**（字节偏移，需要通过 `TJS_FROM_VM_CODE_ADDR` 转换为字索引）
- `code[2]`：异常对象存放的**寄存器偏移**（字节偏移，转换为寄存器索引）

源码（第 1253-1261 行）：
```cpp
case VM_ENTRY:
    code = CodeArea + ExecuteCodeInTryBlock(
        ra,
        code - CodeArea + 3,                    // try 块的起始 IP (ENTRY 之后第 3 个字)
        args, numargs, result,
        TJS_FROM_VM_CODE_ADDR(code[1]) + code - CodeArea,  // catch IP
        TJS_FROM_VM_REG_ADDR(code[2]));          // 异常对象寄存器索引
    break;
```

**执行流程**：
```
VM_ENTRY [catch_off] [exc_reg]
  │
  ├─ 正常执行 try 块内代码
  │   └─ 遇到 VM_EXTRY → 返回 EXTRY 的下一条 IP → 继续执行 try 块之后的代码
  │
  └─ 抛出异常
      ├─ eTJSScriptException → 将异常值存入 ra[exc_reg]，跳到 catch IP
      ├─ eTJSScriptError → 构造异常对象(message+trace)，存入 ra[exc_reg]，跳到 catch IP
      ├─ eTJS → 构造异常对象(message)，存入 ra[exc_reg]，跳到 catch IP
      ├─ std::exception → 同上
      └─ ... (未知异常) → 清除 ra[exc_reg]，跳到 catch IP
```

### 10.4 VM_EXTRY — 退出 try 块

```
格式: [VM_EXTRY]
字数: 1
语义: 与 VM_RET 完全相同——返回下一条指令的 IP
```

`VM_EXTRY` 在功能上等同于 `VM_RET`。它的存在是为了**语义清晰**：`VM_RET` 表示"函数返回"，`VM_EXTRY` 表示"正常离开 try 块"。

源码（第 1263-1264 行）：
```cpp
case VM_EXTRY:
    return code + 1 - CodeArea;  // 与 VM_RET 完全相同
```

### 10.5 VM_THROW — 抛出异常

```
格式: [VM_THROW] [reg]
字数: 2
语义: 抛出 ra[reg] 作为脚本异常
```

源码（第 1266-1270 行）：
```cpp
case VM_THROW:
    ThrowScriptException(
        TJS_GET_VM_REG(ra, code[1]),   // 异常值
        Block,                          // 当前脚本块 (用于生成错误位置信息)
        CodePosToSrcPos(code - CodeArea));  // 字节码位置 → 源码行号

```

> **注释中的 "actually here not proceed..."**：`ThrowScriptException` 会抛出 C++ 异常，所以 `code += 2` 永远不会被执行。保留这行代码只是为了形式上的完整性。

这用于 `instanceof` 检查机制。当创建类实例时，编译器会生成 `VM_ADDCI` 把类对象注册到实例的类信息列表中。

源码（第 1285-1288 行 + `AddClassInstanceInfo`，第 2245-2255 行）：
```cpp
case VM_ADDCI:
    AddClassInstanceInfo(ra, code);
    code += 3;
    break;

void tTJSInterCodeContext::AddClassInstanceInfo(
    tTJSVariant *ra, const tjs_int32 *code) {
    iTJSDispatch2 *dsp = TJS_GET_VM_REG(ra, code[1]).AsObjectNoAddRef();
    if(dsp) {
        dsp->ClassInstanceInfo(
            TJS_CII_ADD,   // 操作: 添加
            0,             // flags
            TJS_GET_VM_REG_ADDR(ra, code[2]));  // 类信息
    }
}
```

### 11.3 VM_REGMEMBER — 注册对象成员

```
格式: [VM_REGMEMBER]
字数: 1
语义: 将当前上下文中定义的所有成员注册到 ra[-1] (objthis) 对象中
```

这是类体（class body）编译的关键指令。当类体执行完毕后（所有成员变量和方法已通过 `VM_SPDE` 写入当前上下文），`VM_REGMEMBER` 把这些成员一次性注册到目标对象中。

源码（第 1290-1293 行）：
```cpp

    RegisterObjectMember(ra[-1].AsObjectNoAddRef());
    code++;
    break;
```

`RegisterObjectMember`（第 2708-2753 行）遍历当前上下文的超类列表，将继承的成员也注册到对象中。

### 练习 2：实现属性访问模拟器

模拟 TJS2 的直接属性读写指令（`VM_GPD` / `VM_SPDE`）：

```cpp
#include <cstdint>
#include <cstdio>
#include <cstring>
#include <map>
#include <string>

// 模拟 TJS2 对象系统
struct TJSObject {
    std::map<std::string, int64_t> props;  // 属性表

    bool propGet(const char* name, int64_t* result) const {
        auto it = props.find(name);
        if(it == props.end()) return false;
        *result = it->second;
        return true;
    }

    bool propSet(const char* name, int64_t value, bool memberEnsure) {
        auto it = props.find(name);
        if(it == props.end()) {
            if(!memberEnsure) return false;  // 成员不存在且未设 MEMBERENSURE
            props[name] = value;  // 自动创建
        } else {
            it->second = value;
        }
        return true;
    }

    void dump() const {
        printf("对象属性:\n");
        for(auto& [k, v] : props)
            printf("  .%s = %lld\n", k.c_str(), (long long)v);
    }
};

// 操作码
enum Op {
    OP_CONST = 0,   // [dest_reg] [data_idx]
    OP_SPDE  = 1,   // [obj_reg] [name_idx] [val_reg]  (带 MEMBERENSURE)
    OP_GPD   = 2,   // [dest_reg] [obj_reg] [name_idx]
    OP_RET   = 3,
};

int main() {

    const char* strData[] = {"x", "y", "z"};   // 属性名
    int64_t numData[] = {100, 200, 300};        // 数值常量

    // 寄存器
    int64_t regs[10] = {};
    TJSObject obj;       // 模拟对象
    int objReg = 1;      // %1 存放对象

    // 字节码: 创建对象并设置属性, 然后读回

    int32_t code[] = {
        // 设置属性
        OP_CONST, 2, 0,       // %2 = 100
        OP_SPDE,  1, 0, 2,    // obj.x = %2 = 100  (name_idx=0 → "x")
        OP_CONST, 3, 1,       // %3 = 200
        OP_SPDE,  1, 1, 3,    // obj.y = %3 = 200  (name_idx=1 → "y")

        // 读取属性
        OP_GPD,   4, 1, 0,    // %4 = obj.x  (name_idx=0 → "x")
        OP_GPD,   5, 1, 1,    // %5 = obj.y  (name_idx=1 → "y")

        OP_RET,
    };

    int32_t* ip = code;
    bool running = true;

    while(running) {
        switch(*ip) {

            case OP_GPD: {

            }
            case OP_RET:
                running = false;
                break;
        }
    }

    printf("\n");
    obj.dump();
    printf("\n读回结果: %%4=%lld, %%5=%lld\n",

}

### 练习 3：条件分支字节码模拟

```cpp
#include <cstdint>

// 操作码
enum Op {
    OP_CONST = 0,  // [dest] [value]  (简化: 直接嵌入值)
    OP_CGT   = 1,  // [reg1] [reg2]   → flag = (reg[reg1] > reg[reg2])
    OP_JNF   = 2,  // [offset]        → if(!flag) ip += offset
    OP_JMP   = 3,  // [offset]        → ip += offset
    OP_CP    = 4,  // [dest] [src]    → reg[dest] = reg[src]
    OP_RET   = 5,
};

int main() {
    int64_t regs[10] = {};
    bool flag = false;

    // 测试值: 尝试修改 regs[1] 看不同分支

    //        x = %1,  0 = %2,  y = %3

    int32_t code[] = {
        //  0: 设置初始值
        OP_CONST, 1, 5,      // %1 = x = 5
        OP_CONST, 2, 0,      // %2 = 0

        //  6: 比较
        OP_CGT,  1, 2,       // flag = (%1 > %2), 即 x > 0

        //  9: 条件跳转

                printf("CONST %%%d = %lld\n", ip[1], (long long)ip[2]);
                ip += 3;
                break;
            case OP_CGT:
                flag = regs[ip[1]] > regs[ip[2]];

                ip += 3;

                break;
            case OP_RET:
                printf("RET\n");
                running = false;
                break;
        }
    }

    printf("\n结果: x=%lld, y=%lld\n",

### 练习 5：完整 TJS2 反汇编器

实现一个能反汇编 TJS2 主要指令的反汇编器：

```cpp
#include <cstdint>
#include <cstdio>
#include <cstring>

// 操作码定义 (复制 TJS2 的枚举值，用数字编号简化)
enum VMCode {
    VM_NOP = 0, VM_CONST, VM_CP, VM_CL, VM_CCL,
    VM_TT, VM_TF, VM_CEQ, VM_CDEQ, VM_CLT, VM_CGT,
    VM_SETF, VM_SETNF, VM_LNOT, VM_NF,
    VM_JF, VM_JNF, VM_JMP,
    VM_INC, VM_DEC,
    VM_BNOT, VM_CHS,
    VM_ADD, VM_SUB, VM_MUL, VM_DIV,
    VM_SRV, VM_RET,
};

// 操作码名称表
const char* opName(int op) {
    static const char* names[] = {
        "NOP", "CONST", "CP", "CL", "CCL",
        "TT", "TF", "CEQ", "CDEQ", "CLT", "CGT",
        "SETF", "SETNF", "LNOT", "NF",
        "JF", "JNF", "JMP",
        "INC", "DEC",
        "BNOT", "CHS",
        "ADD", "SUB", "MUL", "DIV",
        "SRV", "RET",
    };
    if(op >= 0 && op < (int)(sizeof(names)/sizeof(names[0])))
        return names[op];
    return "???";
}

// 每条指令的字数
int opSize(int op) {
    switch(op) {
        case VM_NOP: case VM_NF: case VM_RET:
            return 1;
        case VM_CL: case VM_TT: case VM_TF:
        case VM_SETF: case VM_SETNF: case VM_LNOT:
        case VM_JF: case VM_JNF: case VM_JMP:
        case VM_INC: case VM_DEC:
        case VM_BNOT: case VM_CHS:
        case VM_SRV:
            return 2;
        case VM_CONST: case VM_CP: case VM_CCL:
        case VM_CEQ: case VM_CDEQ: case VM_CLT: case VM_CGT:
        case VM_ADD: case VM_SUB: case VM_MUL: case VM_DIV:
            return 3;
        default: return 1;
    }
}

void disassemble(const int32_t* code, int length) {
    int ip = 0;
    while(ip < length) {
        int op = code[ip];
        int sz = opSize(op);

        printf("%04d: %-8s", ip, opName(op));

        switch(op) {
            // 无操作数
            case VM_NOP: case VM_NF: case VM_RET:
                break;
            // 1 寄存器
            case VM_CL: case VM_TT: case VM_TF:
            case VM_SETF: case VM_SETNF: case VM_LNOT:
            case VM_INC: case VM_DEC:
            case VM_BNOT: case VM_CHS:
            case VM_SRV:
                printf("%%%d", code[ip+1]);
                break;
            // 跳转
            case VM_JF: case VM_JNF: case VM_JMP:
                printf("+%d → IP %d", code[ip+1], ip + code[ip+1]);
                break;
            // 2 寄存器
            case VM_CP:
            case VM_CEQ: case VM_CDEQ: case VM_CLT: case VM_CGT:
            case VM_ADD: case VM_SUB: case VM_MUL: case VM_DIV:
                printf("%%%d, %%%d", code[ip+1], code[ip+2]);
                break;
            // 常量加载
            case VM_CONST:
                printf("%%%d, *%d", code[ip+1], code[ip+2]);
                break;
            // 连续清除
            case VM_CCL:
                printf("%%%d, count=%d", code[ip+1], code[ip+2]);
                break;
        }

        printf("\n");
        ip += sz;
    }
}

int main() {
    printf("=== TJS2 字节码反汇编器 ===\n\n");

    // 示例字节码: 计算 abs(x)
    // if (x < 0) { x = -x; }
    // x 在 %1, 0 在 %2
    int32_t code[] = {
        VM_CONST, 1, 0,    // %1 = x (从常量 *0 加载)
        VM_CONST, 2, 1,    // %2 = 0 (从常量 *1 加载)
        VM_CLT,   1, 2,    // flag = (%1 < %2)
        VM_JNF,   4,       // if(!flag) 跳过 4 字 (到 SRV)
        VM_CHS,   1,       // %1 = -%1
        VM_SRV,   1,       // result = %1
        VM_RET,            // 返回
    };

    int length = sizeof(code) / sizeof(code[0]);
    disassemble(code, length);

    printf("\n注意: 逻辑寄存器号在真实 TJS2 中是字节偏移,\n");
    printf("      这里为简化直接使用整数索引.\n");

    return 0;
}
```

**预期输出**：
```

0000: CONST    %1, *0
0003: CONST    %2, *1
0006: CLT      %1, %2
0009: JNF      +4 → IP 13
0011: CHS      %1
0013: SRV      %1
0015: RET

注意: 逻辑寄存器号在真实 TJS2 中是字节偏移,
      这里为简化直接使用整数索引.

## 14 本节小结

1. **定长 32 位编码**：TJS2 VM 所有指令使用 32 位字定长编码，操作数中的寄存器地址是**字节偏移**而非逻辑寄存器号
2. **100+ 条指令分 9 大类**：数据搬运、条件标志、跳转、算术逻辑（占 67 条）、类型转换、属性访问（16 条）、函数调用、异常返回、对象调试
3. **宏展开生成 56 条指令**：`TJS_NORMAL_AND_PROPERTY_ACCESSER` 和 `TJS_DEF_VM_P` 两个宏将 14 条运算指令各展开为 4 个变体（Base/PD/PI/P）
4. **flag 机制**：比较指令写入独立的 `bool flag` 变量，跳转指令读取 flag——不占用寄存器
5. **操作数反转陷阱**：`VM_CLT` 调用 `GreaterThan`、`VM_CGT` 调用 `LittlerThan`——名称与实现方法相反
6. **属性访问三维组合**：16 条属性指令按 Get/Set/Delete × Direct/Indirect/Unnamed × 各种标志位组合
7. **函数调用三种参数模式**：直接（count ≥ 0）、透传（count = -1）、数组展开（count = -2），参数指针数组预分配 20 个位置避免堆分配
8. **try/catch 编译模式**：`VM_ENTRY` 启动 try 块并记录 catch IP 和异常寄存器，`VM_EXTRY` 正常退出等同于 `VM_RET`

### 题目 1：指令分类

以下字节码片段中每条指令属于哪个分类？各占几个 32 位字？

```
VM_CONST  %2, *0
VM_TT     %2
VM_JNF    +12
VM_ADDPD  %0, %1, *1, %3
VM_RET
```

<details>
<summary>查看答案</summary>

| 指令 | 分类 | 字数 | 说明 |
|------|------|------|------|
| `VM_CONST %2, *0` | 数据搬运 | 3 | 从常量池加载到寄存器 |
| `VM_TT %2` | 条件标志 | 2 | 测试寄存器为真，结果写入 flag |
| `VM_JNF +12` | 跳转与流程 | 2 | flag 为假时跳转 12 字节 |
| `VM_ADDPD %0, %1, *1, %3` | 算术与逻辑（PD 变体） | 5 | 对象属性的复合加法赋值 |
| `VM_RET` | 异常与返回 | 1 | 函数返回 |

总计：3 + 2 + 2 + 5 + 1 = **13 个 32 位字**。

### 题目 2：属性访问标志位

请解释以下四条属性写入指令的标志位差异，并给出每条指令在 TJS2 脚本中的典型使用场景：

```
VM_SPD   %1, *0, %2
VM_SPDE  %1, *0, %2
VM_SPDEH %1, *0, %2
VM_SPDS  %1, *0, %2
```

<details>
<summary>查看答案</summary>

| 指令 | 标志位 | TJS2 场景 | 行为 |
|------|--------|----------|------|
| `VM_SPD` | 无标志（0） | `obj.existingProp = value;` | 赋值已存在的属性，如果属性有 setter 则触发。成员不存在时抛出异常 |
| `VM_SPDE` | `TJS_MEMBERENSURE` | `var obj.newProp = value;` 或 `obj.newProp = value;`（在允许创建新成员的上下文中） | 赋值时若成员不存在则自动创建。这是最常见的变体 |
| `VM_SPDEH` | `TJS_MEMBERENSURE \| TJS_HIDDENMEMBER` | 类的内部实现成员、系统隐藏属性 | 同 SPDE，但创建的成员被标记为"隐藏"，不会被 `for(var k in obj)` 枚举到 |
| `VM_SPDS` | `TJS_MEMBERENSURE \| TJS_IGNOREPROP` | 直接设置 property 的底层值（绕过 setter） | 同 SPDE，但绕过 property setter，直接写入底层存储。用于初始化或内部赋值 |

关键区别：
- `MEMBERENSURE`：控制是否允许自动创建不存在的成员
- `HIDDENMEMBER`：控制成员可见性
- `IGNOREPROP`：控制是否触发 property setter

</details>

### 题目 3：参数传递模式判断

阅读以下 TJS2 脚本，判断每个函数调用的参数传递模式（直接/透传/数组展开），并说明原因：

```javascript
function a(x, y) { return x + y; }
function b(...) { return a(...); }
function c(arr) { return a(arr*); }
function d(x, y, z) { return a(x, y); }
```

<details>
<summary>查看答案</summary>

1. **`a(x, y)`**——**直接模式**（count = 2）
   - 普通函数调用，2 个确定参数
   - 参数区：`[2] [%x] [%y]`

2. **`a(...)` 在函数 b 中**——**透传模式**（count = -1）
   - `b` 的参数声明是 `...`（接收所有参数），调用 `a(...)` 时原封不动转发
   - 参数区：`[-1]`
   - `pass_args = args; pass_args_count = numargs;` 直接复用调用者的参数指针

3. **`a(arr*)` 在函数 c 中**——**数组展开模式**（count = -2）
   - `arr*` 语法将数组展开为独立参数
   - 参数区：`[-2] [1] [fatExpand] [%arr]`
   - 运行时 `TJSCopyArrayElementTo` 提取数组元素

4. **`a(x, y)` 在函数 d 中**——**直接模式**（count = 2）
   - 虽然 `d` 有 3 个参数，但调用 `a` 时只传了 `x` 和 `y`，是普通的 2 参数直接调用
   - 参数区：`[2] [%x] [%y]`
   - `z` 不参与调用，不出现在参数区

</details>
