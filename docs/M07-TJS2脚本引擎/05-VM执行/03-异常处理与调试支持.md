## §1 异常处理的整体架构

TJS2 的异常处理分为**三个层次**，从底层到顶层依次是：

1. **C++ 层异常**（`eTJS` 继承链）：引擎内部使用 C++ throw/catch 机制传播错误
2. **VM 层异常处理**（`ExecuteCode` + `ExecuteCodeInTryBlock`）：VM 在执行字节码时捕获 C++ 异常，附加脚本位置信息后重新抛出
3. **TJS2 脚本层异常**（`try/catch` 语法）：TJS2 脚本中的 `try { } catch(e) { }` 被编译为 `VM_ENTRY`/`VM_EXTRY` 指令，由 `ExecuteCodeInTryBlock` 在 C++ 层实现

这三层的关系可以用下图表示：

```
┌─────────────────────────────────────────────────────────┐
│                TJS2 脚本层                                │
│   try {                                                  │
│       someFunc();  // 可能抛出异常                         │
│   } catch(e) {                                           │
│       // e 是 Exception 对象                              │
│   }                                                      │
├──────────────────────┬──────────────────────────────────┤
│   编译为字节码        │   VM_ENTRY → 执行 → VM_EXTRY     │
├──────────────────────┼──────────────────────────────────┤
│   VM 层              │   ExecuteCodeInTryBlock()         │
│                      │   ├─ 调用 ExecuteCode(tryCatch=true)│
│                      │   ├─ catch(eTJSScriptException)   │
│                      │   │   → 取出 Value 放入寄存器       │
│                      │   ├─ catch(eTJSScriptError)        │
│                      │   │   → 构造 Exception 对象         │
│                      │   ├─ catch(eTJS)                   │
│                      │   │   → 构造 Exception 对象         │
│                      │   └─ catch(...)                    │
│                      │       → 清空异常寄存器              │
├──────────────────────┼──────────────────────────────────┤
│   C++ 层             │   ExecuteCode() 外层 try/catch    │
│                      │   ├─ catch(eTJSSilent) → 重抛      │
│                      │   ├─ catch(eTJSScriptError)        │
│                      │   │   → AddTrace 后重抛            │
│                      │   ├─ catch(eTJS)                   │
│                      │   │   → 转为 eTJSScriptError 抛出  │
│                      │   ├─ catch(exception)              │
│                      │   │   → 转为 eTJSScriptError 抛出  │
│                      │   └─ catch(const char*)            │
│                      │       → 转为 eTJSScriptError 抛出  │
└──────────────────────┴──────────────────────────────────┘
```

### 关键设计原则

TJS2 异常处理的核心设计原则有三条：

**原则一：C++ 异常统一包装。** 无论底层抛出什么类型的 C++ 异常（`eTJS` 系列、`std::exception`、裸 `const char*`），`ExecuteCode` 的外层 catch 都会将其包装为 `eTJSScriptError`，附加脚本文件名和行号信息。这样上层调用者只需要处理 `eTJSScriptError` 及其子类。

**原则二：调用栈逐层累积。** 每当异常穿过一层 `ExecuteCode` 调用（即穿过一次函数调用边界），`AddTrace` 方法会将当前函数的位置信息追加到异常对象的 `Trace` 字段中。最终用户看到的调用栈是一条类似 `scriptA.tjs(10) <-- scriptB.tjs(20) <-- scriptC.tjs(30)` 的字符串。

**原则三：静默异常优先重抛。** `eTJSSilent` 是一种特殊的"中断执行流"信号，不携带任何错误信息。所有 catch 块都必须在第一个分支中检查并重抛 `eTJSSilent`，确保它能一路传播到最顶层而不被任何中间层拦截。

## §2 C++ 异常类层次结构

TJS2 定义了一条清晰的异常类继承链，每种异常类型有明确的职责分工。完整的继承关系如下：

```
                        eTJSSilent（独立类，无继承关系）
                        │
                        │（不参与下面的继承链）
                        │
                        eTJS（基类）
                        │
                        ├── eTJSError（+ std::exception）
                        │   │
                        │   ├── eTJSVariantError
                        │   │
                        │   ├── eTJSScriptError
                        │   │   │
                        │   │   ├── eTJSScriptException
                        │   │   │
                        │   │   └── eTJSCompileError
                        │   │
                        │   └──（其他可能的子类）
                        │
                        └──（无其他直接子类）
```

下面逐一讲解每个类的定义和用途。

### 2.1 eTJSSilent — 静默异常

```cpp
// 源码：cpp/core/tjs2/tjsError.h 第 60-62 行
class eTJSSilent {
    // silent exception — 静默异常，不携带任何数据
};
```

`eTJSSilent` 是一个空类，不继承任何基类，不携带任何信息。它的唯一用途是**中断执行流**——当引擎需要立刻终止当前脚本的执行（例如用户关闭窗口、引擎销毁），但不希望输出任何错误信息时，就抛出 `eTJSSilent`。

**为什么它不继承 `eTJS`？** 因为如果继承了 `eTJS`，那么 `catch(eTJS &)` 就会捕获到它，导致静默异常被当作普通错误处理。将它设计为独立类，确保只有 `catch(eTJSSilent &)` 能捕获它。

**在所有 catch 链中的位置：** 总是排在第一位。例如：

```cpp
// 所有异常处理代码都遵循这个模式
try {
    // ... 执行代码 ...
} catch(eTJSSilent &) {
    throw;  // 永远重抛，绝不拦截
} catch(eTJSScriptError &e) {
    // 处理脚本错误 ...
} catch(eTJS &e) {
    // 处理一般 TJS 错误 ...
}
```

### 2.2 eTJS — 异常基类

```cpp
// 源码：cpp/core/tjs2/tjsError.h 第 65-78 行
class eTJS {
public:
    eTJS() = default;
    eTJS(const eTJS &) = default;
    eTJS &operator=(const eTJS &e) = default;
    virtual ~eTJS() = default;

    // 返回异常消息；基类返回固定字符串 "No-named exception"
    [[nodiscard]] virtual const ttstr &GetMessage() const {
        return TJSNonamedException;  // 全局变量，值为 "No-named exception"
    }
};
```

`eTJS` 是所有"有信息"异常的基类（`eTJSSilent` 除外）。它提供了一个虚方法 `GetMessage()`，子类通过覆盖这个方法来返回具体的错误消息。

`eTJS` 本身很少直接被抛出。它主要作为 catch 链中的"兜底"类型——当某个函数不确定会抛出什么具体异常时，可以用 `catch(eTJS &e)` 捕获所有 TJS2 相关异常。

### 2.3 eTJSError — 带消息的错误

```cpp
// 源码：cpp/core/tjs2/tjsError.h 第 84-99 行
class eTJSError : public eTJS, public std::exception {
public:
    eTJSError(const ttstr &Msg) : Message(Msg) {}

    // 实现 std::exception 接口，方便与标准 C++ 互操作
    const char *what() const noexcept override {
        cachedStr = Message.AsStdString();  // 惰性转换
        return cachedStr.c_str();
    }

    // 覆盖 eTJS 的虚方法
    const ttstr &GetMessage() const override { return Message; }

    // 允许追加消息文本
    void AppendMessage(const ttstr &msg) { Message += msg; }

private:
    ttstr Message;                // TJS2 宽字符串格式的错误消息
    mutable std::string cachedStr; // 窄字符串缓存，供 what() 使用
};
```

`eTJSError` 是第一个"实用"异常类。它**同时继承了 `eTJS` 和 `std::exception`**（多重继承），这意味着它既能被 `catch(eTJS &)` 捕获，也能被 `catch(std::exception &)` 捕获。

`what()` 方法使用 `mutable` 缓存实现惰性转换——只有在实际调用 `what()` 时才将宽字符串转为窄字符串。这种设计避免了构造异常对象时不必要的字符串转换开销。

典型用法：当引擎内部检测到错误条件时，通过辅助函数抛出：

```cpp
// 辅助函数 —— 避免在调用点写 throw 语句
void TJS_eTJSError(const ttstr &msg) {
    throw eTJSError(msg);  // 直接构造并抛出
}

// 使用示例
if(TJS_FAILED(hr))
    TJS_eTJSError(TJSExceptionNotFound);
    // TJSExceptionNotFound 是预定义的消息常量
```

### 2.4 eTJSVariantError — 类型转换错误

```cpp
// 源码：cpp/core/tjs2/tjsError.h 第 107-112 行
class eTJSVariantError : public eTJSError {
public:
    eTJSVariantError(const ttstr &Msg) : eTJSError(Msg) {}
    eTJSVariantError(const eTJSVariantError &ref) = default;
};
```

`eTJSVariantError` 是 `eTJSError` 的子类，专门用于**类型转换失败**的场景。例如，当脚本试图将一个对象转换为整数时，会抛出此异常。它没有添加任何新字段，只是通过类型区分来标识错误类别。

### 2.5 eTJSScriptError — 带脚本位置信息的错误（核心类）

```cpp
// 源码：cpp/core/tjs2/tjsError.h 第 124-160 行
class eTJSScriptError : public eTJSError {
    // 内部类：持有脚本块的引用，自动管理引用计数
    class tScriptBlockHolder {
    public:
        tScriptBlockHolder(tTJSScriptBlock *block);  // AddRef
        ~tScriptBlockHolder();                         // Release
        tScriptBlockHolder(const tScriptBlockHolder &holder); // 拷贝时 AddRef
        tTJSScriptBlock *Block;
    } Block;

    tjs_int Position;  // 源码中的字符偏移位置
    ttstr Trace;       // 调用栈追踪字符串

public:
    // 获取脚本块指针（不增加引用计数）
    tTJSScriptBlock *GetBlockNoAddRef() const { return Block.Block; }

    // 获取源码字符偏移
    tjs_int GetPosition() const { return Position; }

    // 将字符偏移转换为行号（1-based）
    tjs_int GetSourceLine() const;

    // 获取脚本文件名
    const tjs_char *GetBlockName() const;

    // 获取调用栈追踪字符串
    const ttstr &GetTrace() const { return Trace; }

    // 追加调用栈信息（三种重载）
    bool AddTrace(tTJSScriptBlock *block, tjs_int srcpos);
    bool AddTrace(tTJSInterCodeContext *context, tjs_int codepos);
    bool AddTrace(const ttstr &data);

    // 构造函数
    eTJSScriptError(const ttstr &Msg, tTJSScriptBlock *block, tjs_int pos);
    eTJSScriptError(const eTJSScriptError &ref) = default;
};
```

`eTJSScriptError` 是 TJS2 异常处理的**核心类**。相比 `eTJSError`，它增加了三个关键字段：

1. **`Block`**（脚本块引用）：指向异常发生时正在执行的脚本文件/代码块。通过 `tScriptBlockHolder` 内部类自动管理引用计数，确保异常对象持有脚本块引用期间不会被释放。

2. **`Position`**（源码位置）：异常发生点在源码中的**字符偏移量**（不是行号）。可以通过 `GetSourceLine()` 转换为行号：

```cpp
// 源码：cpp/core/tjs2/tjsError.cpp 第 84-86 行
tjs_int eTJSScriptError::GetSourceLine() const {
    return Block.Block->SrcPosToLine(Position) + 1;
    // SrcPosToLine 返回 0-based 行号，+1 转为 1-based
}
```

3. **`Trace`**（调用栈）：一个字符串，记录异常从抛出点到当前捕获点经过的所有函数调用位置。每经过一层函数调用边界，`AddTrace` 方法就追加一条位置信息。

### AddTrace 的三种重载

`AddTrace` 有三种重载形式，适用于不同的上下文：

```cpp
// 重载 1：通过脚本块 + 源码位置追加
// 源码：cpp/core/tjs2/tjsError.cpp 第 95-105 行
bool eTJSScriptError::AddTrace(tTJSScriptBlock *block, tjs_int srcpos) {
    tjs_int len = Trace.GetLen();
    if(len >= TJS_MAX_TRACE_TEXT_LEN)  // 限制最大 1500 字符
        return false;
    if(len != 0)
        Trace += TJS_W("\n<-- ");  // 用换行 + 箭头分隔
    Trace += block->GetLineDescriptionString(srcpos);
    // GetLineDescriptionString 返回 "文件名(行号)" 格式的字符串
    return true;
}

// 重载 2：通过 InterCodeContext + 字节码位置追加
// 源码：cpp/core/tjs2/tjsError.cpp 第 108-119 行
bool eTJSScriptError::AddTrace(tTJSInterCodeContext *context,
                                tjs_int codepos) {
    tjs_int len = Trace.GetLen();
    if(len >= TJS_MAX_TRACE_TEXT_LEN)
        return false;
    if(len != 0)
        Trace += TJS_W(" <-- ");  // 用空格 + 箭头分隔（注意与重载1不同）
    Trace += context->GetPositionDescriptionString(codepos);
    // GetPositionDescriptionString 将 VM 字节码位置转为源码位置描述
    return true;
}

// 重载 3：直接追加字符串
// 源码：cpp/core/tjs2/tjsError.cpp 第 122-130 行
bool eTJSScriptError::AddTrace(const ttstr &data) {
    tjs_int len = Trace.GetLen();
    if(len >= TJS_MAX_TRACE_TEXT_LEN)
        return false;
    if(len != 0)
        Trace += TJS_W(" <-- ");
    Trace += data;
    return true;
}
```

注意 `TJS_MAX_TRACE_TEXT_LEN` 限制为 **1500 字符**。当调用栈过深时，超出限制的部分会被截断，`AddTrace` 返回 `false`。

### 2.6 eTJSScriptException — 携带脚本值的异常

```cpp
// 源码：cpp/core/tjs2/tjsError.h 第 176-190 行
class eTJSScriptException : public eTJSScriptError {
    tTJSVariant Value;  // 脚本层 throw 抛出的值

public:
    tTJSVariant &GetValue() { return Value; }

    eTJSScriptException(const ttstr &Msg, tTJSScriptBlock *block,
                        tjs_int pos, tTJSVariant &val)
        : eTJSScriptError(Msg, block, pos), Value(val) {}

    eTJSScriptException(const eTJSScriptException &ref)
        : eTJSScriptError(ref), Value(ref.Value) {}
};
```

`eTJSScriptException` 是 `eTJSScriptError` 的子类，额外携带一个 `tTJSVariant Value` 字段。这个 `Value` 就是 TJS2 脚本中 `throw expr;` 语句抛出的值。

当 TJS2 脚本执行 `throw "something went wrong";` 时：
1. 编译器生成 `VM_THROW` 指令
2. VM 执行 `VM_THROW` 时，取出寄存器中的值
3. 调用 `ThrowScriptException`，将该值包装为 `eTJSScriptException` 并抛出
4. 在 `ExecuteCodeInTryBlock` 的 `catch(eTJSScriptException &e)` 中，通过 `e.GetValue()` 取出这个值，赋给 catch 块的异常变量寄存器

### 2.7 eTJSCompileError — 编译错误

```cpp
// 源码：cpp/core/tjs2/tjsError.h 第 208-214 行
class eTJSCompileError : public eTJSScriptError {
public:
    eTJSCompileError(const ttstr &Msg, tTJSScriptBlock *block,
                     tjs_int pos)
        : eTJSScriptError(Msg, block, pos) {}
    eTJSCompileError(const eTJSCompileError &ref)
        : eTJSScriptError(ref) {}
};
```

`eTJSCompileError` 用于**编译阶段**的错误（语法错误、未识别的标识符等），继承自 `eTJSScriptError`，没有添加新字段。它与运行时的 `eTJSScriptError` 共用相同的位置追踪能力，但通过类型区分可以让调用者知道"这是编译错误，不是运行时错误"。

### 2.8 异常类层次的 catch 顺序

由于 C++ 异常匹配是按 catch 块的**声明顺序**进行的（不是按"最具体子类优先"），TJS2 源码中所有 catch 链都严格遵循从子类到基类的顺序：

```
catch(eTJSSilent &)         → 重抛（永远第一位）
catch(eTJSScriptException &) → 只在 ExecuteCodeInTryBlock 中出现
catch(eTJSScriptError &)     → 处理带位置信息的错误
catch(eTJS &)                → 兜底所有 TJS 异常
catch(std::exception &)      → 兜底标准 C++ 异常
catch(const char *)          → 兜底裸字符串异常
catch(...)                   → 终极兜底（只在 ExecuteCodeInTryBlock 中）
```

如果把 `catch(eTJS &)` 放在 `catch(eTJSScriptError &)` 前面，那么 `eTJSScriptError` 永远不会被匹配到——它会被 `catch(eTJS &)` 捕获，因为 `eTJSScriptError` 是 `eTJS` 的子类。这是 C++ 异常处理的基本规则。

## §3 ExecuteCode 的异常捕获机制

`ExecuteCode` 是 VM 的主调度循环函数（详见 [01-ExecuteCode调度循环](./01-ExecuteCode调度循环.md)）。它的外层包裹了一个 try/catch 块，负责将执行过程中抛出的任何异常转化为带脚本位置信息的 `eTJSScriptError`。

### 3.1 完整的 catch 链

```cpp
// 源码：cpp/core/tjs2/tjsInterCodeExec.cpp 第 852-1333 行（简化）
tjs_int tTJSInterCodeContext::ExecuteCode(
    tTJSVariant *ra_org, tjs_int startip,
    tTJSVariant **args, tjs_int numargs,
    tTJSVariant *result, bool tryCatch)
{
    tjs_int32 *codesave;
    try {
        tjs_int32 *code = codesave = CodeArea + startip;

        // 设置调用栈追踪器的代码指针
        if(TJSStackTracerEnabled())
            TJSStackTracerSetCodePointer(CodeArea, &codesave);

        tTJSVariant *ra = ra_org;
        tTJSVariant *da = DataArea;
        bool flag = false;

        while(true) {
            codesave = code;  // 保存当前指令位置（异常时用于定位）
            switch(*code) {
                // ... 几百行的指令分发 ...
            }
        }

    // ========== 异常处理部分 ==========

    } catch(eTJSSilent &) {
        throw;  // 分支 1：静默异常 → 原样重抛

    } catch(eTJSScriptError &e) {
        e.AddTrace(this, codesave - CodeArea);  // 分支 2：追加当前位置到调用栈
        throw;                                   // 然后重抛

    } catch(eTJS &e) {
        if(tryCatch) {
            // 分支 3a：在 try 块内 → 只记录日志
            spdlog::get("tjs2")->debug(e.GetMessage().AsStdString());
        } else {
            // 分支 3b：不在 try 块内 → 输出异常现场
            DisplayExceptionGeneratedCode(codesave - CodeArea, ra_org);
        }
        // 转化为 eTJSScriptError 并抛出
        TJS_eTJSScriptError(e.GetMessage(), this, codesave - CodeArea);

    } catch(exception &e) {
        if(tryCatch) {
            spdlog::get("tjs2")->debug(e.what());
        } else {
            DisplayExceptionGeneratedCode(codesave - CodeArea, ra_org);
        }
        // 将 std::exception 转化为 eTJSScriptError
        TJS_eTJSScriptError(e.what(), this, codesave - CodeArea);

    } catch(const char *text) {
        if(tryCatch) {
            spdlog::get("tjs2")->debug(text);
        } else {
            DisplayExceptionGeneratedCode(codesave - CodeArea, ra_org);
        }
        // 将裸字符串转化为 eTJSScriptError
        TJS_eTJSScriptError(text, this, codesave - CodeArea);
    }

    return codesave - CodeArea;  // 返回异常发生时的指令偏移
}
```

### 3.2 各 catch 分支的行为分析

**分支 1 — `catch(eTJSSilent &)`：** 无条件重抛。`eTJSSilent` 是不可拦截的"执行流中断"信号，必须一路传播到最顶层。

**分支 2 — `catch(eTJSScriptError &e)`：** 调用 `e.AddTrace(this, codesave - CodeArea)`，将当前函数的位置信息追加到异常的调用栈追踪中。`this` 是当前 `tTJSInterCodeContext` 对象（包含函数名和所属脚本块信息），`codesave - CodeArea` 是异常发生时的 VM 字节码偏移量。追加完毕后重抛异常，让上层继续处理。

**分支 3 — `catch(eTJS &e)`：** 这里处理的是**还不是 `eTJSScriptError` 的一般性 TJS 异常**。根据 `tryCatch` 参数（是否在 try 块内执行）采取不同策略：
- `tryCatch == true`：仅通过 spdlog 记录调试日志，不输出异常现场
- `tryCatch == false`：调用 `DisplayExceptionGeneratedCode` 输出完整的异常现场信息（反汇编 + 寄存器转储）

然后调用 `TJS_eTJSScriptError` 辅助函数，将 `eTJS` 异常**升级**为 `eTJSScriptError`（附加脚本位置信息）并抛出。

**分支 4/5 — `catch(exception &e)` 和 `catch(const char *text)`：** 与分支 3 逻辑相同，处理非 TJS 系的异常（标准 C++ 异常和裸字符串），统一转化为 `eTJSScriptError`。

### 3.3 codesave 变量的作用

```cpp
tjs_int32 *codesave;
// ...
while(true) {
    codesave = code;  // 每次循环开始时保存当前指令地址
    switch(*code) {
        // ...
    }
}
```

`codesave` 在每次指令分发循环的开头被赋值为当前的 `code` 指针。当异常发生时（无论是在 switch 的哪个 case 中抛出），catch 块通过 `codesave - CodeArea` 计算出**异常发生时正在执行的指令偏移量**。

这个偏移量有两个用途：
1. 传递给 `AddTrace`，记录到调用栈追踪信息中
2. 传递给 `DisplayExceptionGeneratedCode`，用于反汇编异常发生点附近的指令

### 3.4 tryCatch 参数

`ExecuteCode` 的最后一个参数 `tryCatch` 默认为 `false`。只有 `ExecuteCodeInTryBlock` 在调用 `ExecuteCode` 时才会传入 `true`：

```cpp
// ExecuteCodeInTryBlock 中的调用
ret = ExecuteCode(ra, startip, args, numargs, result, true);
//                                                     ^^^^ tryCatch = true
```

当 `tryCatch == true` 时，`ExecuteCode` 的 catch 块**不输出异常现场信息**（因为异常将被外层的 try/catch 捕获处理），只记录调试日志。这避免了在脚本代码正常使用 `try/catch` 处理异常时产生大量无用的错误输出。

## §4 ExecuteCodeInTryBlock 详解

`ExecuteCodeInTryBlock` 是实现 TJS2 脚本层 `try { } catch(e) { }` 语义的核心函数。当 VM 执行到 `VM_ENTRY` 指令时，会调用此函数来执行 try 块中的代码。

### 4.1 函数签名

```cpp
// 源码：cpp/core/tjs2/tjsInterCodeExec.cpp 第 1336-1388 行
tjs_int tTJSInterCodeContext::ExecuteCodeInTryBlock(
    tTJSVariant *ra,       // 寄存器基址
    tjs_int startip,       // try 块起始 IP（VM_ENTRY 指令之后的位置）
    tTJSVariant **args,    // 函数参数
    tjs_int numargs,       // 参数个数
    tTJSVariant *result,   // 返回值
    tjs_int catchip,       // catch 块的起始 IP（从 VM_ENTRY 指令的 code[1] 获取）
    tjs_int exobjreg)      // 异常对象存放的寄存器号（从 VM_ENTRY 的 code[2] 获取）
```

参数 `catchip` 和 `exobjreg` 来自 `VM_ENTRY` 指令的操作数：
- `code[1]`：catch 块在当前函数字节码中的偏移量
- `code[2]`：异常对象赋值给的寄存器编号（如果 `catch(e)` 有变量名 `e`）

### 4.2 完整的执行流程

```cpp
tjs_int tTJSInterCodeContext::ExecuteCodeInTryBlock(
    tTJSVariant *ra, tjs_int startip, tTJSVariant **args, tjs_int numargs,
    tTJSVariant *result, tjs_int catchip, tjs_int exobjreg)
{
    try {
        // 步骤 1：推入调用栈追踪器（标记为 in_try = true）
        if(TJSStackTracerEnabled())
            TJSStackTracerPush(this, true);

        tjs_int ret;
        try {
            // 步骤 2：执行 try 块的字节码
            ret = ExecuteCode(ra, startip, args, numargs, result, true);
            //                                                     ^^^
            //                                      tryCatch = true
        } catch(...) {
            // 步骤 2a：任何异常都先弹出追踪栈，再重抛
            if(TJSStackTracerEnabled())
                TJSStackTracerPop();
            throw;
        }

        // 步骤 3：正常执行完毕，弹出追踪栈
        if(TJSStackTracerEnabled())
            TJSStackTracerPop();
        return ret;  // 返回正常的执行结果

    // ========== 异常捕获分支 ==========

    } catch(eTJSSilent &) {
        throw;  // 静默异常：永远重抛

    } catch(eTJSScriptException &e) {
        // TJS2 脚本 throw 抛出的异常
        if(exobjreg)
            *(ra + exobjreg) = e.GetValue();
            // 直接取出脚本值，赋给异常寄存器
        return catchip;  // 跳转到 catch 块

    } catch(eTJSScriptError &e) {
        // C++ 层产生的脚本错误（非 throw 语句产生）
        if(exobjreg) {
            tTJSVariant msg(e.GetMessage());
            tTJSVariant trace(e.GetTrace());
            TJSGetExceptionObject(
                Block->GetTJS(), ra + exobjreg,
                msg, &trace);
            // 构造 Exception 对象，包含 message 和 trace
        }
        return catchip;

    } catch(eTJS &e) {
        // 一般 TJS 异常（无位置信息）
        if(exobjreg) {
            tTJSVariant msg(e.GetMessage());
            TJSGetExceptionObject(
                Block->GetTJS(), ra + exobjreg,
                msg, nullptr);  // 没有 trace 信息
        }
        return catchip;

    } catch(std::exception &e) {
        // 标准 C++ 异常
        if(exobjreg) {
            tTJSVariant msg(e.what());
            TJSGetExceptionObject(
                Block->GetTJS(), ra + exobjreg,
                msg, nullptr);
        }
        return catchip;

    } catch(...) {
        // 未知异常类型
        if(exobjreg)
            (ra + exobjreg)->Clear();  // 清空异常寄存器
        return catchip;
    }
}
```

### 4.3 关键行为分析

**返回值语义：** `ExecuteCodeInTryBlock` 的返回值是**下一条要执行的指令偏移量**。正常执行时返回 `ExecuteCode` 的返回值（通常是 `VM_EXTRY` 或 `VM_RET` 指令的偏移）。异常发生时返回 `catchip`，即 catch 块的入口偏移。调用者（即触发 `VM_ENTRY` 指令的 `ExecuteCode` 循环）收到返回值后，将 `code` 指针设置为 `CodeArea + 返回值`，继续执行 catch 块中的代码。

**eTJSScriptException vs eTJSScriptError 的差异处理：**

这是一个关键的设计细节——两种异常的处理方式截然不同：

| 异常类型 | 异常值来源 | 处理方式 |
|----------|-----------|---------|
| `eTJSScriptException` | TJS2 的 `throw expr` | 直接取 `e.GetValue()`，赋给寄存器 |
| `eTJSScriptError` | C++ 内部错误 | 构造新的 `Exception` 对象，包含 `message` 和 `trace` |
| `eTJS` | C++ 内部错误 | 构造 `Exception` 对象，只有 `message`，无 `trace` |
| `std::exception` | 标准 C++ 异常 | 构造 `Exception` 对象，只有 `message`，无 `trace` |
| `...`（未知） | 完全未知 | 清空异常寄存器 |

当 TJS2 脚本执行 `throw someValue;` 时，`someValue` 直接从 `eTJSScriptException::Value` 中取出赋给 catch 变量——**不经过任何包装**。这意味着脚本层可以 `throw` 任何类型的值（整数、字符串、对象），catch 到的就是那个原始值。

而当异常来自 C++ 内部错误时（比如访问 null 对象、类型转换失败），则通过 `TJSGetExceptionObject` 构造一个标准的 `Exception` 对象，这样 catch 到的值是一个带有 `message` 和 `trace` 属性的对象。

**调用栈追踪的 Push/Pop：**

注意 `ExecuteCodeInTryBlock` 中有一个嵌套的 try/catch 结构：

```cpp
try {
    if(TJSStackTracerEnabled())
        TJSStackTracerPush(this, true);  // 推入追踪栈，in_try = true
    try {
        ret = ExecuteCode(...);
    } catch(...) {
        TJSStackTracerPop();  // 异常路径：弹出追踪栈
        throw;                // 然后重抛
    }
    TJSStackTracerPop();      // 正常路径：弹出追踪栈
    return ret;
} catch(...) {
    // 外层 catch 处理各种异常类型
}
```

这个嵌套结构确保**无论 `ExecuteCode` 是正常返回还是抛出异常**，追踪栈都会被正确弹出。内层 `catch(...)` 先 Pop 再 throw，保证追踪栈的平衡。

`in_try = true` 参数的作用：在 `GetTraceString` 中，连续的 `InTry == true` 的栈帧会被跳过，因为 try 块在 TJS2 中被实现为"子函数调用"，在调用栈中会产生额外的帧。跳过这些帧可以让调用栈追踪输出更简洁。

### 4.4 从 TJS2 脚本到字节码的完整流程

以下是一段 TJS2 脚本及其完整的执行流程：

```javascript
// TJS2 脚本
try {
    var x = 1 / 0;  // 除零错误
} catch(e) {
    System.inform(e.message);
}
```

编译产生的字节码（简化）：

```
0000: VM_ENTRY    catch=0020, exobj=%5
        // 进入 try 块：catch 块在偏移 0020，异常变量在寄存器 %5
0003: VM_CONST    %2, *1     // %2 = 1
0006: VM_CONST    %3, *0     // %3 = 0
0009: VM_DIV      %2, %3     // %2 = %2 / %3 → 除零异常！
0012: VM_CP       %1, %2     // var x = %2（永远执行不到）
0015: VM_EXTRY                // try 块正常结束
        // VM_EXTRY 返回，跳过 catch 块
0016: VM_JMP      0030       // 跳过 catch 块
0020: // === catch 块起始 ===
0020: VM_GPDS     %6, %5, "message"  // %6 = e.message
0024: VM_CALLD    %0, %7, "inform"   // System.inform(%6)
0029: // === catch 块结束 ===
0030: // 继续执行后续代码
```

执行流程：

1. VM 执行 `VM_ENTRY`，调用 `ExecuteCodeInTryBlock(ra, 0003, ..., 0020, 5)`
2. `ExecuteCodeInTryBlock` 调用 `ExecuteCode(ra, 0003, ..., true)` 执行 try 块
3. `ExecuteCode` 执行到 `VM_DIV` 时，底层除零操作抛出异常
4. `ExecuteCode` 的 catch 链将异常转化为 `eTJSScriptError`（附加位置信息）
5. 异常传播到 `ExecuteCodeInTryBlock` 的 `catch(eTJSScriptError &e)`
6. `TJSGetExceptionObject` 构造 `Exception` 对象（含 `message` 和 `trace`）
7. 对象赋值给 `*(ra + 5)` 即寄存器 `%5`
8. 返回 `catchip = 0020`
9. 外层 `ExecuteCode` 将 `code` 设置为 `CodeArea + 0020`，继续执行 catch 块

## §5 TJS_eTJSScriptError 辅助函数

异常的抛出不是直接写 `throw` 语句，而是通过一系列辅助函数来完成。这些辅助函数负责在抛出异常之前**报告异常信息到控制台**（如果调试模式开启）。

### 5.1 报告与抛出分离

```cpp
// 源码：cpp/core/tjs2/tjsError.cpp 第 137-146 行
static void TJSReportExceptionSource(
    const ttstr &msg,
    const tTJSScriptBlock *block,
    tjs_int srcpos)
{
    if(TJSEnableDebugMode) {
        tTJS *tjs = block->GetTJS();
        tjs->OutputExceptionToConsole(
            (msg + TJS_W(" at ") +
             block->GetLineDescriptionString(srcpos)).c_str());
        // 输出格式："错误消息 at 文件名(行号)"
    }
}

// 重载版本：接受 InterCodeContext + 字节码位置
// 源码：cpp/core/tjs2/tjsError.cpp 第 149-159 行
static void TJSReportExceptionSource(
    const ttstr &msg,
    const tTJSInterCodeContext *context,
    tjs_int codepos)
{
    if(TJSEnableDebugMode) {
        tTJS *tjs = context->GetBlock()->GetTJS();
        tjs->OutputExceptionToConsole(
            (msg + TJS_W(" at ") +
             context->GetPositionDescriptionString(codepos)).c_str());
    }
}
```

`TJSReportExceptionSource` 是一个 `static`（文件内局部）函数。只有在 `TJSEnableDebugMode` 为 `true` 时才输出异常信息到控制台。这个函数不抛出异常——它只负责"报告"。

### 5.2 各辅助抛出函数

```cpp
// 源码：cpp/core/tjs2/tjsError.cpp 第 162-236 行

// 最基础的抛出
void TJS_eTJS() { throw eTJS(); }
void TJS_eTJSError(const ttstr &msg) { throw eTJSError(msg); }
void TJS_eTJSVariantError(const ttstr &msg) { throw eTJSVariantError(msg); }

// 带脚本位置的抛出（通过脚本块）
void TJS_eTJSScriptError(const ttstr &msg,
                          tTJSScriptBlock *block, tjs_int srcpos) {
    TJSReportExceptionSource(msg, block, srcpos);  // 先报告
    throw eTJSScriptError(msg, block, srcpos);       // 再抛出
}

// 带脚本位置的抛出（通过 InterCodeContext）
void TJS_eTJSScriptError(const ttstr &msg,
                          tTJSInterCodeContext *context, tjs_int codepos) {
    TJSReportExceptionSource(msg, context, codepos);
    throw eTJSScriptError(
        msg,
        context->GetBlock(),                   // 从 context 获取脚本块
        context->CodePosToSrcPos(codepos));    // 将 VM 偏移转为源码偏移
}

// 脚本异常（携带值）
void TJS_eTJSScriptException(const ttstr &msg,
                               tTJSInterCodeContext *context,
                               tjs_int codepos, tTJSVariant &val) {
    TJSReportExceptionSource(msg, context, codepos);
    throw eTJSScriptException(
        msg,
        context->GetBlock(),
        context->CodePosToSrcPos(codepos),
        val);  // 携带脚本层的值
}

// 编译错误
void TJS_eTJSCompileError(const ttstr &msg,
                            tTJSScriptBlock *block, tjs_int srcpos) {
    TJSReportExceptionSource(msg, block, srcpos);
    throw eTJSCompileError(msg, block, srcpos);
}
```

### 5.3 TJSThrowFrom_tjs_error — 错误码转异常

```cpp
// 源码：cpp/core/tjs2/tjsError.cpp 第 254-289 行
void TJSThrowFrom_tjs_error(tjs_error hr, const tjs_char *name) {
    // 将 tjs_error 错误码转化为 C++ 异常
    switch(hr) {
        case TJS_E_MEMBERNOTFOUND: {
            if(name) {
                ttstr str(TJSMemberNotFound);  // "Member \"%1\" does not exist"
                str.Replace(TJS_W("%1"), name);
                TJS_eTJSError(str);
            } else {
                TJS_eTJSError(TJSMemberNotFoundNoNameGiven);
            }
        }
        case TJS_E_NOTIMPL:
            TJS_eTJSError(TJSNotImplemented);
        case TJS_E_INVALIDPARAM:
            TJS_eTJSError(TJSInvalidParam);
        case TJS_E_BADPARAMCOUNT:
            TJS_eTJSError(TJSBadParamCount);
        case TJS_E_INVALIDTYPE:
            TJS_eTJSError(TJSInvalidType);
        case TJS_E_ACCESSDENYED:
            TJS_eTJSError(TJSAccessDenyed);
        case TJS_E_INVALIDOBJECT:
            TJS_eTJSError(TJSInvalidObject);
        case TJS_E_NATIVECLASSCRASH:
            TJS_eTJSError(TJSNativeClassCrash);
        default:
            if(TJS_FAILED(hr)) {
                TJS_eTJSError(fmt::sprintf(
                    ttstr{TJSUnknownFailure}.AsNarrowStdString(), hr));
            }
    }
}
```

注意这个 switch 语句**没有 break**——这是有意为之的，因为每个 case 都调用 `TJS_eTJSError` 抛出异常（永远不会 fall through 到下一个 case）。

`TJS_THROW_IF_ERROR` 宏将此函数与错误码检查结合使用：

```cpp
// 源码：cpp/core/tjs2/tjsError.h 第 226-232 行
#define TJS_THROW_IF_ERROR(x)                   \
    {                                            \
        tjs_error ____er;                        \
        ____er = (x);                            \
        if(TJS_FAILED(____er))                   \
            TJSThrowFrom_tjs_error(____er);      \
    }

// 使用示例
TJS_THROW_IF_ERROR(obj->PropGet(0, name, hint, &result, objthis));
// 如果 PropGet 返回错误码，自动转为异常抛出
```

### 5.4 预定义错误消息

TJS2 使用 `TJS_MSG_DECL` 宏在 `tjsError.h` 文件末尾定义了大量预定义错误消息：

```cpp
// 源码：cpp/core/tjs2/tjsError.h 第 238-347 行

    inline TJS::tTJSMessageHolder name(TJS_W(#name), msg);

TJS_MSG_DECL(TJSInvalidParam,      TJS_W("Invalid argument"))
TJS_MSG_DECL(TJSBadParamCount,     TJS_W("Invalid argument count"))
TJS_MSG_DECL(TJSInvalidType,       TJS_W("Not a function or invalid method/property type"))
TJS_MSG_DECL(TJSDivideByZero,      TJS_W("Division by zero"))
TJS_MSG_DECL(TJSInvalidOpecode,    TJS_W("Invalid VM code"))
TJS_MSG_DECL(TJSRangeError,        TJS_W("The value is out of the range"))
TJS_MSG_DECL(TJSExceptionNotFound, TJS_W("Cannot convert exception because \"Exception\" does not exist"))

```

`TJS_MSG_DECL` 展开后创建一个 `tTJSMessageHolder` 类型的全局 `inline` 变量。这些消息是**可替换的**——理论上可以通过替换 `tTJSMessageHolder` 的内容来实现错误消息的本地化（国际化）。


## §6 TJSGetExceptionObject — 构造 TJS2 异常对象

当 C++ 层的异常被 `ExecuteCodeInTryBlock` 捕获后，需要将其转化为 TJS2 脚本层能使用的对象。这就是 `TJSGetExceptionObject` 的职责——它在 TJS2 全局作用域中查找 `Exception` 类，创建一个实例，设置 `message` 和 `trace` 属性，然后赋值给 catch 变量对应的寄存器。

> **术语提示：** `Exception` 类是 TJS2 内置的异常类（在 TJS2 运行时初始化时注册到全局作用域），与 C++ 的 `std::exception` 无继承关系。它是一个普通的 TJS2 类，通过 `CreateNew` 创建实例。

### 6.1 函数完整源码

```cpp
// 源码：cpp/core/tjs2/tjsError.cpp 第 26-54 行
void TJSGetExceptionObject(tTJS *tjs, tTJSVariant *dest,
                            tTJSVariant &msg, tTJSVariant *trace)
{
    if(!tjs) return;  // 安全检查：引擎实例不存在则什么都不做

    // 第 1 步：在全局作用域中查找 "Exception" 类
    iTJSDispatch2 *global = tjs->GetGlobalNoAddRef();
    // GetGlobalNoAddRef：获取全局对象但不增加引用计数
    // （因为全局对象的生命周期与引擎相同，不需要额外引用）

    tTJSVariant val;
    static ttstr Exception_name(TJS_W("Exception"));
    // static 局部变量：只初始化一次，避免每次调用都构造字符串

    if(TJS_SUCCEEDED(global->PropGet(0, Exception_name.c_str(),
                     Exception_name.GetHint(), &val, global)))
    {
        // PropGet 成功：全局作用域中存在 "Exception"
        // 第 2 步：取出 Exception 类对象
        iTJSDispatch2 *excpobj = val.AsObjectNoAddRef();
        // AsObjectNoAddRef：从 tTJSVariant 中取出对象指针

        // 第 3 步：调用 CreateNew 创建实例
        tTJSVariant *pmsg = &msg;
        iTJSDispatch2 *obj = nullptr;
        if(TJS_SUCCEEDED(excpobj->CreateNew(0, nullptr, nullptr,
                         &obj, 1, &pmsg, excpobj)))
        {
            // CreateNew 参数说明：
            //   0          — flags（无特殊标志）
            //   nullptr    — membername（调用默认构造器）
            //   nullptr    — hint
            //   &obj       — 输出：新创建的对象
            //   1          — 参数个数
            //   &pmsg      — 参数数组（只有一个参数：message）
            //   excpobj    — objthis（构造器内的 this）

            // 第 4 步：如果有 trace 信息，设置 trace 属性
            if(trace) {
                static ttstr trace_name(TJS_W("trace"));
                obj->PropSet(TJS_MEMBERENSURE, trace_name.c_str(),
                            trace_name.GetHint(), trace, obj);
                // TJS_MEMBERENSURE：如果属性不存在则创建
            }

            // 第 5 步：将新对象赋值给目标 variant
            *dest = tTJSVariant(obj, obj);
            // tTJSVariant(obj, obj)：创建一个 object 类型的 variant
            //   第一个 obj = 对象指针
            //   第二个 obj = objthis（对象自身作为 this）

            obj->Release();
            // 释放 CreateNew 增加的引用计数
            // dest 中的 variant 已持有自己的引用
        } else {
            *dest = msg;
            // CreateNew 失败：退回到直接使用 message 字符串
        }
    } else {
        *dest = msg;
        // 查找 "Exception" 失败：退回到直接使用 message 字符串
        // 这种情况很少见，通常意味着 TJS2 运行时未正确初始化
    }
}
```

### 6.2 调用时机与参数分析

`TJSGetExceptionObject` 在 `ExecuteCodeInTryBlock` 中被三处调用：

```cpp
// 调用 1：eTJSScriptError（C++ 内部产生的脚本错误）
catch(eTJSScriptError &e) {
    tTJSVariant msg(e.GetMessage());   // 错误消息
    tTJSVariant trace(e.GetTrace());   // 调用栈追踪字符串
    TJSGetExceptionObject(Block->GetTJS(), ra + exobjreg, msg, &trace);
    //                     ^^^^^^^^^^^^    ^^^^^^^^^^^^^  ^^^  ^^^^^^
    //                     引擎实例        目标寄存器     消息  有trace
}

// 调用 2：eTJS（一般 TJS 异常，无位置信息）
catch(eTJS &e) {
    tTJSVariant msg(e.GetMessage());
    TJSGetExceptionObject(Block->GetTJS(), ra + exobjreg, msg, nullptr);
    //                                                         ^^^^^^^
    //                                                         无trace
}

// 调用 3：std::exception（标准 C++ 异常）
catch(std::exception &e) {
    tTJSVariant msg(e.what());
    TJSGetExceptionObject(Block->GetTJS(), ra + exobjreg, msg, nullptr);
}
```

关键区别：
- **eTJSScriptError** 携带 `Trace` 信息（调用栈追踪），构造的 `Exception` 对象有 `trace` 属性
- **eTJS** 和 **std::exception** 没有 `Trace` 信息，构造的对象只有 `message`

### 6.3 降级策略

注意函数中有两处 `*dest = msg` 的降级处理：

1. **全局对象中找不到 "Exception" 类**：这通常意味着 TJS2 运行时尚未完成初始化（或已经被销毁）。此时 catch 变量拿到的只是一个字符串。
2. **CreateNew 失败**：Exception 类的构造器抛出了异常。同样降级为字符串。

这个降级策略确保即使在异常的异常情况下（异常处理本身出错），程序也不会崩溃——catch 块至少能拿到一个字符串形式的错误消息。

### 6.4 引用计数的精确管理

```
GetGlobalNoAddRef()    → 不增加引用计数（无需 Release）
AsObjectNoAddRef()     → 不增加引用计数（无需 Release）
CreateNew() 的 &obj   → 增加引用计数（需要 Release）
*dest = tTJSVariant(obj, obj) → variant 持有自己的引用
obj->Release()         → 释放 CreateNew 的引用
```

每一个 `NoAddRef` 后缀的函数都明确表示"不增加引用计数"，这样调用者不需要配对 `Release`。只有 `CreateNew` 产生的新对象需要显式释放。

### 6.5 GetHint 优化

```cpp
static ttstr Exception_name(TJS_W("Exception"));
global->PropGet(0, Exception_name.c_str(), Exception_name.GetHint(), ...);
```

`GetHint()` 返回一个指向内部缓存的哈希值（`tjs_uint32*`）的指针。TJS2 的属性查找使用字符串名称，但会缓存哈希值以加速后续查找。`static` 变量确保字符串和哈希只初始化一次——因为异常处理可能在性能敏感的路径上被频繁调用。

## §7 ThrowScriptException — VM_THROW 的实现

当 TJS2 脚本执行 `throw expr;` 语句时，编译器生成 `VM_THROW` 指令。`ExecuteCode` 执行该指令时，调用 `ThrowScriptException` 来构建并抛出 C++ 异常。

### 7.1 完整源码

```cpp
// 源码：cpp/core/tjs2/tjsInterCodeExec.cpp 第 823-849 行
void tTJSInterCodeContext::ThrowScriptException(
    tTJSVariant &val,     // throw 的值（来自 VM_THROW 指令的操作数寄存器）
    tTJSScriptBlock *block,
    tjs_int srcpos)       // 源码位置
{
    ttstr msg;

    if(val.Type() == tvtObject) {
        // 如果 throw 的值是对象类型，尝试读取 .message 属性
        try {
            tTJSVariant v2;
            iTJSDispatch2 *obj = val.AsObjectNoAddRef();
            iTJSDispatch2 *objthis = val.AsObjectThisNoAddRef();

            static ttstr message_name(TJS_W("message"));
            tjs_error hr = obj->PropGet(0,
                message_name.c_str(), message_name.GetHint(),
                &v2, objthis);

            if(TJS_SUCCEEDED(hr)) {
                msg = v2;  // 成功读取到 .message 属性
            }
        } catch(...) {
            // 读取 .message 失败（对象不支持 PropGet 等）
            // 忽略错误，msg 保持为空
        }
    }

    if(msg.IsEmpty()) {
        msg = TJS_W("script exception");
        // 没有获取到 message：使用默认消息
    } else {
        msg = TJS_W("script exception : ") + msg;
        // 有 message：拼接前缀
    }

    // 抛出 eTJSScriptException
    TJS_eTJSScriptException(msg, this, srcpos, val);
    // 注意：val 作为最后一个参数传入，会被存储在
    // eTJSScriptException::Value 中
}
```

### 7.2 执行流程分析

**步骤 1：检查值类型。** 如果 throw 的值是对象类型（`tvtObject`），尝试从中获取 `.message` 属性。这意味着：
```javascript
// TJS2 脚本
throw new Exception("出错了");
// ThrowScriptException 会读取 Exception 对象的 .message 属性
// msg = "出错了"
// 最终异常消息 = "script exception : 出错了"
```

**步骤 2：构造消息字符串。** 有两种情况：
- 值是对象且有 `.message`：消息格式为 `"script exception : {message}"`
- 值不是对象，或对象没有 `.message`：消息为 `"script exception"`

**步骤 3：调用 `TJS_eTJSScriptException`。** 这个辅助函数会：
1. 调用 `TJSReportExceptionSource` 输出到控制台（如果调试模式开启）
2. 抛出 `eTJSScriptException` 异常，其中 `Value` 字段保存原始的 throw 值

### 7.3 与 ExecuteCode 中 VM_THROW 的关联

```cpp
// 源码：cpp/core/tjs2/tjsInterCodeExec.cpp（ExecuteCode 函数内部）
case VM_THROW: {
    ThrowScriptException(
        *(ra + code[1]),           // throw 的值（寄存器内容）
        Block,                     // 当前脚本块
        CodePosToSrcPos(code - CodeArea));  // 将 VM 偏移转为源码位置
    break;  // 永远执行不到（ThrowScriptException 总是抛异常）
}
```

### 7.4 脚本到 C++ 再到脚本的完整闭环

以下是一个完整的异常生命周期：

```javascript
// TJS2 脚本
function riskyOperation() {
    throw new Exception("文件不存在");
}

try {
    riskyOperation();
} catch(e) {
    System.inform(e.message);
    // e.message == "文件不存在"
}
```

执行流程：

```
1. riskyOperation() 中的 throw 语句
   → 编译为 VM_THROW 指令
   → ExecuteCode 执行 VM_THROW

2. ThrowScriptException(val, block, srcpos)
   → val 是 Exception 对象
   → 读取 val.message = "文件不存在"
   → msg = "script exception : 文件不存在"
   → 调用 TJS_eTJSScriptException(msg, context, srcpos, val)
   → 抛出 C++ 异常 eTJSScriptException

3. eTJSScriptException 向上传播
   → 穿过 riskyOperation 的 ExecuteCode
   → 穿过 ExecuteCodeInTryBlock 的内层 try（Pop 追踪栈后重抛）
   → 被外层 catch(eTJSScriptException &e) 捕获

4. ExecuteCodeInTryBlock 的 catch 分支
   → *(ra + exobjreg) = e.GetValue()
   → 直接取出原始 Exception 对象
   → 赋给 catch(e) 的变量 e

5. 返回 catchip → 执行 catch 块代码
   → e.message 仍然是 "文件不存在"（原始对象未经包装）
```

**关键点：** `eTJSScriptException` 的 catch 分支直接用 `e.GetValue()` 取出原始值——不经过 `TJSGetExceptionObject` 包装。这就是为什么 TJS2 脚本的 `throw` 可以抛出任意值（整数、字符串、对象），catch 到的就是那个原始值。

## §8 TJS_CONVERT_TO_TJS_EXCEPTION 宏

在 §3 中我们看到，`ExecuteCode` 的 catch 链需要按特定顺序捕获多种异常类型，并为每种类型添加位置信息。这个 catch 链在多处重复出现——不仅在 `ExecuteCode` 中，还在其他需要"捕获所有异常并转化为 TJS2 异常"的地方。`TJS_CONVERT_TO_TJS_EXCEPTION` 宏正是为了消除这种重复而设计的。

### 8.1 宏的完整定义

```cpp
// 源码：cpp/core/tjs2/tjsError.h 第 24-48 行
#define TJS_CONVERT_TO_TJS_EXCEPTION                               \
    catch(eTJSSilent &) {                                          \
        throw;                                                      \
    } catch(eTJSScriptException &) {                               \
        throw;                                                      \
    } catch(eTJSScriptError &) {                                   \
        throw;                                                      \
    } catch(eTJS &e) {                                             \
        TJS_eTJSScriptError(e.GetMessage(), Block,                 \
                            CodePosToSrcPos(code - CodeArea));     \
    } catch(std::exception &e) {                                   \
        TJS_eTJSScriptError(ttstr(e.what()), Block,                \
                            CodePosToSrcPos(code - CodeArea));     \
    } catch(const char *text) {                                    \
        TJS_eTJSScriptError(ttstr(text), Block,                   \
                            CodePosToSrcPos(code - CodeArea));     \
    } catch(const wchar_t *text) {                                 \
        TJS_eTJSScriptError(ttstr(text), Block,                   \
                            CodePosToSrcPos(code - CodeArea));     \
    } catch(...) {                                                 \
        TJS_eTJSScriptError(TJSUnknownError, Block,               \
                            CodePosToSrcPos(code - CodeArea));     \
    }
```

### 8.2 各分支行为

| 异常类型 | 行为 | 理由 |
|----------|------|------|
| `eTJSSilent` | 直接重抛 | 静默异常不应被处理 |
| `eTJSScriptException` | 直接重抛 | 已经是脚本异常，无需转化 |
| `eTJSScriptError` | 直接重抛 | 已经携带位置信息 |
| `eTJS` | **转化**为 `eTJSScriptError` | 添加位置信息 |
| `std::exception` | **转化**为 `eTJSScriptError` | 添加位置信息 |
| `const char*` | **转化**为 `eTJSScriptError` | C 风格字符串异常 |
| `const wchar_t*` | **转化**为 `eTJSScriptError` | 宽字符串异常 |
| `...` | **转化**为 `eTJSScriptError` | 未知异常 |

**核心逻辑**：已经是脚本异常（`eTJSScriptException`/`eTJSScriptError`）的直接重抛（它们已经携带了位置信息）；其他类型的异常都被"升级"为 `eTJSScriptError`，附加当前的代码位置信息。

### 8.3 宏的使用前提

这个宏内部使用了三个未限定的标识符：
- `Block`：当前的 `tTJSScriptBlock*`
- `code`：当前的指令指针
- `CodeArea`：字节码数组起始地址

因此，这个宏**只能在 `tTJSInterCodeContext` 的成员函数中使用**——只有那里这三个变量才有意义。这是一种隐式的"接口约定"：使用此宏的函数必须有这些局部变量/成员可用。

### 8.4 使用场景

```cpp
// 典型使用方式（在 ExecuteCode 内部）
try {
    // 执行某个操作，如调用对象方法
    hr = objthis->FuncCall(0, memberName, hint, result, numargs, args, objthis);
}
TJS_CONVERT_TO_TJS_EXCEPTION
// 等价于：
// catch(eTJSSilent &) { throw; }
// catch(eTJSScriptException &) { throw; }
// catch(eTJSScriptError &) { throw; }
// catch(eTJS &e) { TJS_eTJSScriptError(e.GetMessage(), Block, ...); }
// catch(std::exception &e) { TJS_eTJSScriptError(ttstr(e.what()), Block, ...); }
// ...
```

这种模式确保了任何从 `FuncCall` 中抛出的异常都能被正确转化为带位置信息的 TJS2 脚本异常，使得调用栈追踪能够包含准确的源码位置。

### 8.5 TJS_CONVERT_TO_TJS_EXCEPTION 与 ExecuteCodeInTryBlock 的区别

这是一个容易混淆的地方——两者都是"catch 链"，但它们的职责完全不同：

| | TJS_CONVERT_TO_TJS_EXCEPTION | ExecuteCodeInTryBlock 的 catch |
|---|---|---|
| **目的** | 将任意异常**转化为** `eTJSScriptError` | 将 C++ 异常**交付给** TJS2 catch 块 |
| **产出** | 重新抛出（throw） | 返回 catchip（跳转到 catch 块） |
| **位置** | VM 指令执行中 | `try { } catch(e) { }` 语法实现中 |
| **`eTJSScriptException`** | 直接重抛（不处理） | 取出 Value 赋给寄存器 |
| **调用 TJSGetExceptionObject?** | 否 | 是（对非 ScriptException 类型） |

## §9 tTJSStackTracer — 调用栈追踪系统

调用栈追踪是调试的基石——当异常发生时，开发者需要知道"程序是怎样到达这里的"。TJS2 通过 `tTJSStackTracer` 类维护一个手工管理的调用栈（与 C++ 自身的调用栈独立），记录 TJS2 脚本的函数调用链。

> **术语提示：** 调用栈追踪（Stack Trace）是一个调用链的文本表示，类似于 Java 的 `e.printStackTrace()` 或 Python 的 `traceback`。TJS2 的调用栈追踪格式为 `"位置1 <-- 位置2 <-- 位置3"`。

### 9.1 tTJSStackRecord — 栈帧记录

调用栈的每一帧由 `tTJSStackRecord` 结构表示：

```cpp
// 源码：cpp/core/tjs2/tjsDebug.cpp 第 524-561 行
class tTJSStackRecord {
public:
    tTJSInterCodeContext *Context;  // 执行上下文（函数/方法）
    const tjs_int32 *CodeBase;     // 字节码数组基址
    const tjs_int32 *CodePtr;      // 当前执行到的指令指针
    bool InTry;                    // 是否在 try 块内

    // 构造函数：引用计数管理
    tTJSStackRecord(tTJSInterCodeContext *context, bool in_try)
        : Context(context), InTry(in_try),
          CodeBase(nullptr), CodePtr(nullptr)
    {
        Context->AddRef();  // 增加引用计数，防止上下文被提前释放
    }

    // 析构函数
    ~tTJSStackRecord() {
        Context->Release();  // 释放引用
    }

    // 复制构造函数
    tTJSStackRecord(const tTJSStackRecord &other)
        : Context(other.Context), CodeBase(other.CodeBase),
          CodePtr(other.CodePtr), InTry(other.InTry)
    {
        Context->AddRef();  // 复制时也要增加引用
    }
};
```

**四个字段的含义：**
- `Context`：指向一个 `tTJSInterCodeContext`，代表一个 TJS2 函数/方法/属性访问器的执行上下文
- `CodeBase`：该上下文的字节码数组起始地址（对应 `CodeArea`）
- `CodePtr`：当前执行到的指令位置（在 `ExecuteCode` 主循环中随指令执行不断更新）
- `InTry`：标记此帧是否来自 `ExecuteCodeInTryBlock`（try 块在 TJS2 中被实现为"子函数调用"）

### 9.2 tTJSStackTracer 类

```cpp
// 源码：cpp/core/tjs2/tjsDebug.cpp 第 564-657 行
class tTJSStackTracer {
    std::vector<tTJSStackRecord> Stack;  // 栈帧列表
    tjs_int RefCount;                     // 引用计数

public:
    tTJSStackTracer() : RefCount(1) {}

    void AddRef() { RefCount++; }
    void Release() {
        if(RefCount == 1) delete this;
        else RefCount--;
    }

    // 压入一个新栈帧
    void Push(tTJSInterCodeContext *context, bool in_try) {
        Stack.push_back(tTJSStackRecord(context, in_try));
    }

    // 弹出栈顶帧
    void Pop() {
        Stack.pop_back();
    }

    // 更新栈顶帧的代码指针（每执行一条指令都调用）
    void SetCodePointer(const tjs_int32 *codebase,
                        const tjs_int32 *codeptr) {
        if(!Stack.empty()) {
            Stack.back().CodeBase = codebase;
            Stack.back().CodePtr = codeptr;
        }
    }

    // 生成调用栈追踪字符串
    ttstr GetTraceString(tjs_int limit = 0,
                         const tjs_char *delimiter = nullptr);
};
```

### 9.3 GetTraceString 的实现细节

```cpp
// 源码：cpp/core/tjs2/tjsDebug.cpp 第 610-657 行
ttstr tTJSStackTracer::GetTraceString(
    tjs_int limit, const tjs_char *delimiter)
{
    if(!delimiter) delimiter = TJS_W(" <-- ");
    // 默认分隔符："<--"，从右到左表示调用链方向

    ttstr result;
    tjs_int count = 0;
    bool first = true;

    // 从栈顶（最近的调用）向栈底（最早的调用）遍历
    for(tjs_int i = (tjs_int)Stack.size() - 1; i >= 0; i--) {
        tTJSStackRecord &rec = Stack[i];

        // 跳过连续的 InTry 帧
        // try 块被实现为子函数调用，会在栈中产生额外帧
        // 跳过这些帧使追踪输出更清晰
        if(rec.InTry) {
            // 如果当前帧和下一帧都是 InTry，跳过当前帧
            if(i > 0 && Stack[i-1].InTry) continue;
            // 如果只有当前帧是 InTry，保留（它是 try 块的入口帧）
        }

        if(rec.CodeBase && rec.CodePtr) {
            // 获取位置描述字符串
            ttstr pos = rec.Context->GetPositionDescriptionString(
                (tjs_int)(rec.CodePtr - rec.CodeBase));
            // GetPositionDescriptionString 返回格式：
            //   "script.tjs(15)" 或 "global(23)"

            if(!first) result += delimiter;
            result += pos;
            first = false;
            count++;

            // 检查限制
            if(limit > 0 && count >= limit) break;
        }
    }

    return result;
}
```

**遍历方向**：从栈顶到栈底（从最近的调用到最早的调用），输出格式如：

```
script.tjs(15) <-- main.tjs(42) <-- global(3)
```

表示：全局代码第 3 行调用了 main.tjs 第 42 行的函数，该函数又调用了 script.tjs 第 15 行的函数，异常在那里发生。

**InTry 帧跳过逻辑**：

```
栈（从底到顶）：
  [0] global()           InTry=false   ← 保留
  [1] main.tjs(42)       InTry=false   ← 保留
  [2] try-block-entry    InTry=true    ← 连续InTry的第一个，跳过
  [3] try-block-inner    InTry=true    ← 连续InTry的最后一个，保留
  [4] script.tjs(15)     InTry=false   ← 保留
```

遍历从 `[4]` 到 `[0]`：
- `[4]` InTry=false → 输出
- `[3]` InTry=true，检查 `[2]` 也是 InTry → 保留（因为是连续序列的最后一个）
- `[2]` InTry=true，检查 `[1]` 不是 InTry → 跳过
- `[1]` InTry=false → 输出
- `[0]` InTry=false → 输出

等等——这里的逻辑需要仔细看。实际源码中 `i > 0 && Stack[i-1].InTry` 检查的是**下方**的帧（更早的调用），如果下方也是 InTry 则跳过当前帧。这样的效果是：连续的多个 InTry 帧中，只有最下方的那个会被输出。

### 9.4 全局辅助函数

为了方便从 VM 的各处调用，TJS2 提供了一组全局函数来操作线程局部的 `tTJSStackTracer` 实例：

```cpp
// 源码：cpp/core/tjs2/tjsDebug.cpp 第 660-691 行

// 每个线程有自己的 StackTracer 实例
static thread_local tTJSStackTracer *TJSStackTracer = nullptr;

bool TJSStackTracerEnabled() {
    return TJSStackTracer != nullptr;
}

void TJSStackTracerPush(tTJSInterCodeContext *context, bool in_try) {
    if(TJSStackTracer)
        TJSStackTracer->Push(context, in_try);
}

void TJSStackTracerPop() {
    if(TJSStackTracer)
        TJSStackTracer->Pop();
}

void TJSStackTracerSetCodePointer(
    const tjs_int32 *codebase, const tjs_int32 *codeptr) {
    if(TJSStackTracer)
        TJSStackTracer->SetCodePointer(codebase, codeptr);
}

ttstr TJSGetStackTraceString(tjs_int limit,
                              const tjs_char *delimiter) {
    if(TJSStackTracer)
        return TJSStackTracer->GetTraceString(limit, delimiter);
    return ttstr();
}
```

**线程安全：** 使用 `thread_local` 确保每个线程有独立的调用栈追踪器。TJS2 不支持多线程脚本执行，但 C++ 宿主程序可能在多个线程中运行不同的 TJS2 引擎实例。

### 9.5 在 ExecuteCode 中的集成

调用栈追踪器通过三个调用点与 VM 执行循环集成：

```cpp
// 源码：cpp/core/tjs2/tjsInterCodeExec.cpp

// 1. 进入函数时 Push
tjs_int tTJSInterCodeContext::ExecuteCode(...) {
    if(TJSStackTracerEnabled())
        TJSStackTracerPush(this, false);  // 普通函数调用
    // ...

// 2. 每执行一条指令时更新代码指针
    while(true) {
        if(TJSStackTracerEnabled())
            TJSStackTracerSetCodePointer(CodeArea, code);
        // 这样在异常发生时，调用栈中每一帧的位置都是最新的

        switch(*code) {
            // ... 指令执行 ...
        }
    }

// 3. 退出函数时 Pop
    if(TJSStackTracerEnabled())
        TJSStackTracerPop();
    return ret;
}
```

**性能影响：** `SetCodePointer` 在每条指令执行时都被调用。虽然它只是更新两个指针（O(1) 操作），但在紧密的指令循环中仍有可测量的开销。因此 `TJSStackTracerEnabled()` 检查使得追踪可以完全禁用，避免在生产运行中的性能损失。

### 9.6 调用栈追踪字符串在异常中的传播

```
异常发生
  ↓
ExecuteCode 的 catch 链捕获异常
  ↓
构造 eTJSScriptError，调用 TJSGetStackTraceString() 获取追踪字符串
  ↓
追踪字符串存储在 eTJSScriptError::Trace 中
  ↓
异常向上传播，每层 AddTrace() 追加更多位置信息
  ↓
到达 ExecuteCodeInTryBlock
  ↓
TJSGetExceptionObject 将 trace 设置为 Exception 对象的属性
  ↓
TJS2 脚本的 catch(e) 可以通过 e.trace 获取完整追踪字符串
```

## §10 tTJSObjectHashMap — 对象生命周期追踪

`tTJSObjectHashMap` 是 TJS2 的**调试辅助设施**——一个全局的对象注册表，记录所有通过 `iTJSDispatch2` 接口创建的对象。它的主要功能是**检测内存泄漏**：在程序退出时列出所有未释放的对象。

> **术语提示：** `iTJSDispatch2` 是 TJS2 中所有可调用对象的接口（类似于 COM 的 IDispatch），所有 TJS2 对象（普通对象、数组、字典、函数等）都实现此接口。对象通过引用计数管理生命周期。

### 10.1 记录结构

```cpp
// 源码：cpp/core/tjs2/tjsDebug.h 第 94-118 行
class tTJSObjectHashMapRecord {
public:
    ttstr History;    // 对象创建时的调用栈追踪（限制 4 层）
    ttstr Where;      // 创建位置（限制 1 层，最近的调用点）
    ttstr Type;       // 对象类型名称（如 "Array"、"Dictionary"）
    tjs_uint32 Flags; // 状态标志

    tTJSObjectHashMapRecord()
        : Flags(TJS_OHMF_EXIST) {}  // 初始状态：存在
};
```

**四个字段的用途：**

| 字段 | 内容 | 来源 |
|------|------|------|
| `History` | `"func1(10) <-- main(5) <-- global(1) <-- ..."` | `TJSGetStackTraceString(4)` |
| `Where` | `"func1(10)"` | `TJSGetStackTraceString(1)` |
| `Type` | `"Array"` 或 `"Dictionary"` 等 | `SetType()` 设置 |
| `Flags` | 位标志组合 | `SetFlag()` 设置 |

### 10.2 标志位定义

```cpp
// 源码：cpp/core/tjs2/tjsDebug.h 第 68-72 行
#define TJS_OHMF_EXIST       1  // 对象存在（已注册未删除）
#define TJS_OHMF_INVALIDATED 2  // 对象已失效（invalidate 已调用）
#define TJS_OHMF_DELETING    4  // 对象正在删除中
```

**生命周期状态转换：**

```
创建 → Flags = TJS_OHMF_EXIST (1)
   ↓
失效 → Flags = TJS_OHMF_EXIST | TJS_OHMF_INVALIDATED (3)
   ↓
删除中 → Flags = TJS_OHMF_EXIST | TJS_OHMF_INVALIDATED | TJS_OHMF_DELETING (7)
   ↓
删除完成 → 从 HashMap 中移除
```

`TJS_OHMF_DELETING` 标志有一个特殊用途——`WarnIfObjectIsDeleting` 函数会检查此标志：

```cpp
// 伪代码
void WarnIfObjectIsDeleting(void *obj) {
    auto rec = Find(obj);
    if(rec && (rec->Flags & TJS_OHMF_DELETING)) {
        // 警告：正在被删除的对象上执行了操作
        OutputWarning("Operating on a deleting object!");
    }
}
```

这能捕获一类常见的 bug——在对象的析构过程中，其他代码仍然持有指向该对象的指针并试图操作它。

### 10.3 哈希表实现

```cpp
// 源码：cpp/core/tjs2/tjsDebug.cpp 第 144-388 行
class tTJSObjectHashMap {
    // 开放地址哈希表（不使用链表）
    struct tHashEntry {
        void *Ptr;                      // 对象指针（作为 key）
        tTJSObjectHashMapRecord *Rec;   // 记录（作为 value）
    };

    tHashEntry *Table;     // 哈希表数组
    tjs_int TableSize;     // 表大小（总是 2 的幂）
    tjs_int Count;         // 当前元素个数

    // 哈希函数：指针值右移 3 位后取模
    tjs_int HashFunc(void *ptr) {
        return ((tjs_int)(tjs_intptr_t)ptr >> 3) & (TableSize - 1);
    }
    // 右移 3 位：因为对象通常按 8 字节对齐，低 3 位总是 0
    // & (TableSize - 1)：等价于 % TableSize（因为 TableSize 是 2 的幂）
};
```

**为什么不用 `std::unordered_map`？** 这个类最初编写时（约 2000 年），C++ 标准库的哈希表尚未广泛可用。此外，自定义实现允许精确控制内存分配和扩容策略，在调试场景下性能更可预测。

### 10.4 Add 和 Remove 操作

```cpp
// 简化后的 Add 操作
void tTJSObjectHashMap::Add(void *ptr) {
    if(Count * 2 >= TableSize) Expand();  // 负载因子 >= 0.5 时扩容

    tjs_int idx = HashFunc(ptr);
    while(Table[idx].Ptr != nullptr) {
        idx = (idx + 1) & (TableSize - 1);  // 线性探测
    }

    Table[idx].Ptr = ptr;
    Table[idx].Rec = new tTJSObjectHashMapRecord();
    Count++;
}

// 简化后的 Remove 操作
void tTJSObjectHashMap::Remove(void *ptr) {
    tjs_int idx = InternalFind(ptr);
    if(idx >= 0) {
        delete Table[idx].Rec;
        Table[idx].Ptr = nullptr;
        Table[idx].Rec = nullptr;
        Count--;
        // 注意：开放地址法删除后需要 rehash 后续元素
        // 实际代码处理了这个问题
    }
}
```

### 10.5 TJSAddObjectHashRecord — 创建时的信息采集

```cpp
// 源码：cpp/core/tjs2/tjsDebug.cpp 第 397-424 行
void TJSAddObjectHashRecord(void *obj) {
    if(!TJSObjectHashMap) return;

    TJSObjectHashMap->Add(obj);

    // 采集创建时的调用栈信息
    tTJSObjectHashMapRecord *rec = TJSObjectHashMap->GetRecord(obj);
    if(rec) {
        // History：完整的创建调用栈（限制 4 层深度）
        rec->History = TJSGetStackTraceString(4);
        // Where：最近的创建位置（限制 1 层）
        rec->Where = TJSGetStackTraceString(1);
    }
}
```

这个函数在每个 `iTJSDispatch2` 对象创建时被调用。它利用 `tTJSStackTracer`（§9）来获取对象创建时的调用栈信息。`limit=4` 表示只记录最近 4 层调用，`limit=1` 只记录最近 1 层——这是在信息丰富度和内存占用之间的折中。

### 10.6 ReportAllUnfreedObjects — 泄漏报告

程序退出时，调用此函数列出所有未释放的对象：

```cpp
// 简化后的泄漏报告逻辑
void tTJSObjectHashMap::ReportAllUnfreedObjects() {
    // 第一遍：按 (Type + History) 分组计数
    std::map<std::pair<ttstr,ttstr>, int> groups;
    for(int i = 0; i < TableSize; i++) {
        if(Table[i].Ptr) {
            auto key = std::make_pair(
                Table[i].Rec->Type,
                Table[i].Rec->History);
            groups[key]++;
        }
    }

    // 输出分组报告
    OutputToConsole("== Unfreed objects report ==");
    for(auto &[key, count] : groups) {
        OutputToConsole(fmt::format(
            "  Type: {}, Count: {}, History: {}",
            key.first, count, key.second));
    }

    // 第二遍：按 Type 汇总
    std::map<ttstr, int> typeCount;
    for(int i = 0; i < TableSize; i++) {
        if(Table[i].Ptr) {
            typeCount[Table[i].Rec->Type]++;
        }
    }

    OutputToConsole("== Summary by type ==");
    for(auto &[type, count] : typeCount) {
        OutputToConsole(fmt::format(
            "  {}: {} objects", type, count));
    }
}
```

输出示例：

```
== Unfreed objects report ==
  Type: Array, Count: 3, History: loadData(15) <-- main(42) <-- global(1)
  Type: Dictionary, Count: 1, History: createConfig(8) <-- init(20)
  Type: (unknown), Count: 2, History: (no trace)

== Summary by type ==
  Array: 3 objects
  Dictionary: 1 objects
  (unknown): 2 objects

Total unfreed objects: 6
```

### 10.7 二进制日志系统

`tTJSObjectHashMap` 还支持将所有操作记录到二进制日志文件，以便事后分析：

```cpp
enum tTJSObjectHashMapLogItemId {
```

每个日志条目的二进制格式：

### 10.8 ObjectHashMap 与引用计数的关系

TJS2 对象使用引用计数管理生命周期。`tTJSObjectHashMap` 是引用计数的**外部验证机制**：

```
对象生命周期：
  new → AddRef → AddRef → Release → Release → delete
  
ObjectHashMap 追踪：
  new → Add()                    // 进入 HashMap
  ...                            // 期间可以 SetType, SetFlag
  delete → Remove()              // 离开 HashMap
  
程序退出时：
  HashMap 中仍有记录 → 泄漏！   // ReportAllUnfreedObjects
  HashMap 为空 → 一切正常
```

它不干预引用计数的运作，只是在旁边做记录。这种设计确保了调试设施不会影响正常的对象生命周期管理。

## §11 DisplayExceptionGeneratedCode — 异常现场信息

当 `ExecuteCode` 的 catch 链捕获到异常时（且 `tryCatch == false`，即不在脚本 try 块内），会调用 `DisplayExceptionGeneratedCode` 输出详细的异常现场信息，包括字节码反汇编和寄存器转储。这是 TJS2 最强大的调试辅助功能之一。

### 11.1 完整源码

```cpp
// 源码：cpp/core/tjs2/tjsInterCodeExec.cpp 第 780-820 行
void tTJSInterCodeContext::DisplayExceptionGeneratedCode(
    tjs_int codepos,      // 异常发生时的字节码偏移量
    const tTJSVariant *ra) // 寄存器基址
{
    tTJS *tjs = Block->GetTJS();
    if(!tjs) return;

    // 第 1 步：输出头部信息
    tjs->OutputToConsole(
        (ttstr(
            TJS_W("==== An exception occurred")
            TJS_W(" at ")) +
         GetPositionDescriptionString(codepos) +
         TJS_W(", VM ip = ") +
         ttstr(codepos) +
         TJS_W(" ====")).c_str());
    // 输出格式：
    //   "==== An exception occurred at script.tjs(15), VM ip = 42 ===="
    //   GetPositionDescriptionString：返回 "文件名(行号)"
    //   codepos：字节码偏移量（VM instruction pointer）

    // 第 2 步：反汇编异常发生处的字节码
    DisassembleSrcLine(codepos);
    // 输出异常所在行对应的所有字节码指令
    // 格式类似：
    //   0042: VM_DIV      %2, %3
    //   0045: VM_CP       %1, %2

    // 第 3 步：转储所有寄存器
    tjs->OutputToConsoleSeparator(TJS_W("-- Registers --"), false);

    tjs_int start = -(MaxVariableCount + VariableReserveCount);
    // 起始寄存器编号：负数区域是局部变量和保留寄存器
    //   MaxVariableCount：函数声明的最大变量数
    //   VariableReserveCount：保留寄存器数（通常为 2）

    tjs_int end = MaxFrameCount;
    // 结束寄存器编号：正数区域是栈帧空间（临时值）

    for(tjs_int i = start; i < end; i++) {
        tjs->OutputToConsole(
            (ttstr(TJS_W("%")) +
             ttstr(i) +
             TJS_W("=") +
             TJSVariantToReadableString(*(ra + i))).c_str());
    }
    // 输出格式（每个寄存器一行）：
    //   %-4=(int)0
    //   %-3=(string)"hello"
    //   %-2=(void)
    //   %-1=(void)
    //   %0=(object)(0x7fff1234)
    //   %1=(int)42
    //   %2=(real)3.14
    //   ...
}
```

### 11.2 输出解读

一个完整的异常现场输出示例：

```
==== An exception occurred at script.tjs(15), VM ip = 42 ====

  0039: VM_CONST    %2, *1         // 加载常量 1
  0042: VM_CONST    %3, *0         // 加载常量 0
  0045: VM_DIV      %2, %3         // 除法操作 ← 异常在这里

-- Registers --
%-4=(string)"x"
%-3=(int)100
%-2=(void)
%-1=(object)(0x7fff1234)
%0=(object)(0x7fff5678)
%1=(void)
%2=(int)1
%3=(int)0
%4=(void)
```

**如何使用这些信息调试：**
1. **头部行**告诉你异常发生在哪个文件的哪一行（`script.tjs(15)`），以及对应的字节码偏移（`VM ip = 42`）
2. **反汇编输出**显示异常所在行的所有字节码指令，你可以看到哪条指令触发了异常
3. **寄存器转储**显示异常发生时所有变量和临时值的状态，帮助你理解为什么会出错

### 11.3 寄存器编号含义

```
负数区域（局部变量）：
  -(MaxVariableCount + VariableReserveCount) 到 -1
  
  例如 MaxVariableCount=3, VariableReserveCount=2：
    %-5, %-4, %-3  → 局部变量（var x, y, z）
    %-2, %-1       → 保留寄存器（this, arguments 等）

零和正数区域（栈帧）：
  0 到 MaxFrameCount-1
  
    %0   → 通常是 objthis（当前对象）
    %1+  → 临时值、中间计算结果、函数参数
```

### 11.4 DisassembleSrcLine 的工作方式

`DisassembleSrcLine` 不只反汇编一条指令——它反汇编**同一源码行**对应的所有指令。一行 TJS2 源码可能编译为多条字节码指令，全部显示出来能帮助开发者理解编译器产生了什么代码。

```cpp
// 伪代码
void DisassembleSrcLine(tjs_int codepos) {
    // 找到 codepos 对应的源码行号
    tjs_int line = SrcPosToLine(CodePosToSrcPos(codepos));
    
    // 遍历所有指令，输出同一行的指令
    for(tjs_int i = 0; i < CodeAreaSize; ) {
        if(SrcPosToLine(CodePosToSrcPos(i)) == line) {
            OutputDisassembly(i);  // 输出这条指令的反汇编
        }
        i += GetInstructionSize(CodeArea[i]);  // 跳到下一条指令
    }
}
```

### 11.5 调用时机

```cpp
// 在 ExecuteCode 的 catch 链中
} catch(eTJSScriptError &e) {
    if(!tryCatch) {
        // 不在脚本 try 块内 → 输出异常现场
        DisplayExceptionGeneratedCode(code - CodeArea, ra);
    }
    // ...
}
```

`tryCatch` 参数控制是否输出异常信息：
- `false`（默认）：输出异常现场。这是"顶层"的未处理异常。
- `true`：不输出。异常将被外层的 `ExecuteCodeInTryBlock` 捕获，属于正常的异常处理流程。

## §12 动手实践

### 练习 1：观察异常类层次结构

编写一个 C++ 程序，模拟 TJS2 的异常类层次结构，验证 catch 链的顺序匹配行为：

```cpp
#include <iostream>
#include <string>
#include <stdexcept>

// 模拟 TJS2 异常类层次结构

// 静默异常——不继承任何基类
class eTJSSilent {};

// TJS 基础异常
class eTJS {
public:
    virtual ~eTJS() = default;
    virtual std::string GetMessage() const { return ""; }
};

// TJS 错误——同时继承 eTJS 和 std::exception
class eTJSError : public eTJS, public std::exception {
    std::string message_;
public:
    explicit eTJSError(const std::string &msg) : message_(msg) {}
    std::string GetMessage() const override { return message_; }
    const char* what() const noexcept override {
        return message_.c_str();
    }
};

// 脚本错误——携带位置信息
class eTJSScriptError : public eTJSError {
    std::string trace_;
    int position_;
public:
    eTJSScriptError(const std::string &msg, int pos)
        : eTJSError(msg), position_(pos) {}
    int GetPosition() const { return position_; }
    std::string GetTrace() const { return trace_; }
    void AddTrace(const std::string &t) { trace_ += t; }
};

// 脚本异常——携带脚本值
class eTJSScriptException : public eTJSScriptError {
    std::string value_;  // 简化：用字符串代替 tTJSVariant
public:
    eTJSScriptException(const std::string &msg, int pos,
                         const std::string &val)
        : eTJSScriptError(msg, pos), value_(val) {}
    std::string GetValue() const { return value_; }
};

// 测试 catch 链顺序
void testCatchOrder(int exceptionType) {
    try {
        switch(exceptionType) {
            case 0: throw eTJSSilent();
            case 1: throw eTJSScriptException("脚本异常", 15, "错误值");
            case 2: throw eTJSScriptError("脚本错误", 42);
            case 3: throw eTJSError("一般错误");
            case 4: throw eTJS();
            case 5: throw std::runtime_error("C++运行时错误");
            case 6: throw 42;  // 未知类型
        }
    }
```

### 练习 2：实现简化的 StackTracer

```cpp
#include <iostream>
#include <vector>
#include <string>

// 简化的栈帧记录
struct StackRecord {
    std::string functionName;  // 函数名（代替 tTJSInterCodeContext*）
    int lineNumber;            // 当前行号（代替 CodePtr）
    bool inTry;                // 是否在 try 块内

    StackRecord(const std::string &name, int line, bool try_block)
        : functionName(name), lineNumber(line), inTry(try_block) {}
};

// 简化的调用栈追踪器
class SimpleStackTracer {
    std::vector<StackRecord> stack_;

public:
    void push(const std::string &funcName, bool inTry = false) {
        stack_.emplace_back(funcName, 0, inTry);
        std::cout << "[Push] " << funcName
                  << (inTry ? " (try)" : "") << "\n";
    }

    void pop() {
        if(!stack_.empty()) {
            std::cout << "[Pop] " << stack_.back().functionName << "\n";
            stack_.pop_back();
        }
    }

    void setLineNumber(int line) {
        if(!stack_.empty()) {
            stack_.back().lineNumber = line;
        }
    }

    // 生成调用栈字符串（模拟 TJS2 的 GetTraceString）
    std::string getTraceString(int limit = 0) const {
        std::string result;
        int count = 0;
        bool first = true;

        // 从栈顶向栈底遍历
        for(int i = (int)stack_.size() - 1; i >= 0; i--) {
            const auto &rec = stack_[i];

            // 跳过连续 InTry 帧的逻辑
            if(rec.inTry && i > 0 && stack_[i-1].inTry) {
                continue;  // 跳过
            }

            if(!first) result += " <-- ";
            result += rec.functionName + "(" +
                      std::to_string(rec.lineNumber) + ")";
            first = false;
            count++;

            if(limit > 0 && count >= limit) break;
        }

        return result;
    }

    void printStack() const {
        std::cout << "\n调用栈（从顶到底）：\n";
        for(int i = (int)stack_.size() - 1; i >= 0; i--) {
            std::cout << "  [" << i << "] "
                      << stack_[i].functionName
                      << "(" << stack_[i].lineNumber << ")"
                      << (stack_[i].inTry ? " [InTry]" : "")
                      << "\n";
        }
    }
};

// 模拟 TJS2 脚本执行
void simulateExecution() {
    SimpleStackTracer tracer;

    // 模拟：global → main() → tryBlock → riskyFunc()
    tracer.push("global");
    tracer.setLineNumber(1);

    tracer.push("main");
    tracer.setLineNumber(10);

    // 进入 try 块（在 TJS2 中 try 块是子函数调用）
    tracer.push("main_try", true);
    tracer.setLineNumber(15);

    // 嵌套 try
    tracer.push("main_try_inner", true);
    tracer.setLineNumber(16);

    tracer.push("riskyFunc");
    tracer.setLineNumber(42);

    // 输出当前调用栈
    tracer.printStack();

    // 生成追踪字符串
    std::cout << "\n完整追踪: " << tracer.getTraceString() << "\n";
    std::cout << "限制2层:  " << tracer.getTraceString(2) << "\n";

    // 模拟异常退栈
    std::cout << "\n--- 异常发生，开始退栈 ---\n";
    tracer.pop();  // riskyFunc
    tracer.pop();  // main_try_inner
    tracer.pop();  // main_try
    // main 的 catch 块处理异常
    std::cout << "\n异常被 main 的 catch 块捕获\n";
}

int main() {
    std::cout << "=== SimpleStackTracer 演示 ===\n\n";
    simulateExecution();
    return 0;
}
// 编译: g++ -std=c++17 -o stack_tracer stack_tracer.cpp
// 预期输出：
//   完整追踪: riskyFunc(42) <-- main_try_inner(16) <-- main(10) <-- global(1)
//   注意: main_try(15) 被跳过（连续 InTry 帧的非末尾帧）
```

### 练习 3：实现简化的 ObjectHashMap

```cpp
#include <iostream>
#include <unordered_map>
#include <string>
#include <vector>

// 对象生命周期标志
constexpr int OHMF_EXIST       = 1;
constexpr int OHMF_INVALIDATED = 2;
constexpr int OHMF_DELETING    = 4;

// 对象记录
struct ObjectRecord {
    std::string type;     // 对象类型
    std::string where;    // 创建位置
    int flags;            // 状态标志

    ObjectRecord() : flags(OHMF_EXIST) {}

    std::string flagsToString() const {
        std::string s;
        if(flags & OHMF_EXIST)       s += "EXIST ";
        if(flags & OHMF_INVALIDATED) s += "INVALIDATED ";
        if(flags & OHMF_DELETING)    s += "DELETING ";
        return s.empty() ? "(none)" : s;
    }
};

// 简化的 ObjectHashMap
class ObjectHashMap {
    std::unordered_map<void*, ObjectRecord> map_;

public:
    void add(void *obj, const std::string &where) {
        ObjectRecord rec;
        rec.where = where;
        map_[obj] = rec;
        std::cout << "[Add] 对象 " << obj
                  << " 创建于 " << where << "\n";
    }

    void remove(void *obj) {
        auto it = map_.find(obj);
        if(it != map_.end()) {
            std::cout << "[Remove] 对象 " << obj
                      << " (类型: " << it->second.type << ")\n";
            map_.erase(it);
        }
    }

    void setType(void *obj, const std::string &type) {
        auto it = map_.find(obj);
        if(it != map_.end()) {
            it->second.type = type;
        }
    }

    void setFlag(void *obj, int flag) {
        auto it = map_.find(obj);
        if(it != map_.end()) {
            it->second.flags |= flag;
        }
    }

    bool warnIfDeleting(void *obj) {
        auto it = map_.find(obj);
        if(it != map_.end() && (it->second.flags & OHMF_DELETING)) {
            std::cerr << "[警告] 正在操作一个正被删除的对象: "
                      << obj << " (类型: " << it->second.type << ")\n";
            return true;
        }
        return false;
    }

    void reportUnfreed() {
        std::cout << "\n=== 未释放对象报告 ===\n";
        if(map_.empty()) {
            std::cout << "  (无泄漏，所有对象已释放)\n";
            return;
        }

        // 按类型统计
        std::unordered_map<std::string, int> typeCount;
        for(auto &[ptr, rec] : map_) {
            typeCount[rec.type.empty() ? "(unknown)" : rec.type]++;
            std::cout << "  对象 " << ptr
                      << " | 类型: " << (rec.type.empty() ? "(unknown)" : rec.type)
                      << " | 创建于: " << rec.where
                      << " | 标志: " << rec.flagsToString() << "\n";
        }

        std::cout << "\n--- 按类型汇总 ---\n";
        for(auto &[type, count] : typeCount) {
            std::cout << "  " << type << ": " << count << " 个对象\n";
        }
        std::cout << "总计: " << map_.size() << " 个未释放对象\n";
    }
};

// 模拟对象管理
int main() {
    ObjectHashMap map;

    // 模拟创建对象
    int obj1, obj2, obj3, obj4, obj5;

    map.add(&obj1, "loadData(15)");
    map.setType(&obj1, "Array");

    map.add(&obj2, "createConfig(8)");
    map.setType(&obj2, "Dictionary");

    map.add(&obj3, "main(42)");
    map.setType(&obj3, "Array");

    map.add(&obj4, "init(5)");
    map.setType(&obj4, "Function");

    map.add(&obj5, "handler(20)");
    map.setType(&obj5, "String");

    // 正常释放一些对象
    std::cout << "\n--- 正常释放 ---\n";
    map.remove(&obj4);  // Function 对象正常释放
    map.remove(&obj5);  // String 对象正常释放

    // 模拟在删除过程中操作对象
    std::cout << "\n--- 模拟删除中操作 ---\n";
    map.setFlag(&obj3, OHMF_INVALIDATED);  // 标记为已失效
    map.setFlag(&obj3, OHMF_DELETING);     // 标记为正在删除
    map.warnIfDeleting(&obj3);             // 应该输出警告

    // 程序退出时的泄漏报告
    // obj1, obj2, obj3 未被 remove → 泄漏
    map.reportUnfreed();

    return 0;
}
// 编译: g++ -std=c++17 -o object_hashmap object_hashmap.cpp
// 预期输出：3 个未释放对象（Array×2, Dictionary×1）
```

```cpp
// 将 Variant 转为可读字符串
std::string variantToString(const Variant &v) {
    if(std::holds_alternative<std::monostate>(v))
        return "(void)";
    if(std::holds_alternative<int>(v))
        return "(int)" + std::to_string(std::get<int>(v));
    if(std::holds_alternative<double>(v)) {
        std::ostringstream oss;
        oss << "(real)" << std::get<double>(v);
        return oss.str();
    }
    if(std::holds_alternative<std::string>(v))
        return "(string)\"" + std::get<std::string>(v) + "\"";
    return "(unknown)";
}
```

## §14 本节小结

- **异常类层次结构**：TJS2 使用 5 层 C++ 异常类层次（`eTJSSilent` → `eTJS` → `eTJSError` → `eTJSScriptError` → `eTJSScriptException`），每层增加更多上下文信息。`eTJSSilent` 是独立的，不继承任何基类
- **catch 链顺序**：必须从最具体（`eTJSSilent`）到最通用（`...`），否则子类异常会被父类 catch 错误捕获
- **两种 catch 链的区别**：`TJS_CONVERT_TO_TJS_EXCEPTION` 将异常"升级"（附加位置信息后重新抛出）；`ExecuteCodeInTryBlock` 的 catch 链将异常"交付"（转化为 TJS2 值赋给 catch 变量）
- **TJSGetExceptionObject**：在全局作用域查找 `Exception` 类，调用 `CreateNew` 创建实例，设置 `message` 和 `trace` 属性。有两级降级策略（找不到类/创建失败时退回字符串）
- **ThrowScriptException**：`VM_THROW` 指令的实现，尝试从 throw 值中读取 `.message` 属性，构造 `eTJSScriptException`
- **tTJSStackTracer**：手工管理的调用栈追踪器，每个线程独立（`thread_local`）。Push/Pop 与函数调用配对，SetCodePointer 每条指令更新。InTry 帧在输出时被合并
- **tTJSObjectHashMap**：对象生命周期追踪系统，记录创建位置、类型、状态标志。程序退出时报告所有未释放的对象，支持二进制日志的录制和回放
- **DisplayExceptionGeneratedCode**：异常现场信息输出，包含源码位置、字节码反汇编、完整寄存器转储
- **设计哲学**：报告与抛出分离（`TJSReportExceptionSource` + `throw`）、可选的调试设施（`TJSEnableDebugMode`/`TJSStackTracerEnabled()`）、降级而非崩溃（`TJSGetExceptionObject` 的 fallback）

## §15 练习题与答案

### 题目 1：异常类层次结构与 catch 顺序

以下代码有一个隐蔽的 bug，请找出问题并说明原因：

```cpp
try {
    // 执行某些 TJS2 操作
    someOperation();
} catch(eTJSScriptError &e) {
    // 处理脚本错误
    handleScriptError(e);
} catch(eTJSScriptException &e) {
    // 处理脚本异常
    handleScriptException(e);
} catch(eTJS &e) {
    // 处理一般 TJS 异常
    handleTJSError(e);
}
```

<details>
<summary>查看答案</summary>

**问题**：`eTJSScriptException` 的 catch 分支永远不会被执行。

**原因**：`eTJSScriptException` 继承自 `eTJSScriptError`。在 C++ 中，catch 链是按顺序匹配的——当抛出 `eTJSScriptException` 时，它首先匹配到 `catch(eTJSScriptError &e)`（因为子类引用可以绑定到父类引用），永远不会到达 `catch(eTJSScriptException &e)`。

**修正**：必须把更具体的子类放在前面：

```cpp
try {
    someOperation();
} catch(eTJSScriptException &e) {
    // 先捕获子类（eTJSScriptException）
    handleScriptException(e);
} catch(eTJSScriptError &e) {
    // 再捕获父类（eTJSScriptError）
    handleScriptError(e);
} catch(eTJS &e) {
    handleTJSError(e);
}
```

这正是 TJS2 源码中所有 catch 链的写法——`eTJSSilent` 最前（虽然不在层次结构中，但永远重抛），然后 `eTJSScriptException`，然后 `eTJSScriptError`，最后 `eTJS`、`std::exception`、`...`。

</details>

### 题目 2：TJSGetExceptionObject 的降级行为

假设 TJS2 引擎正在关闭过程中，全局对象已被部分销毁。此时一个 C++ 异常被 `ExecuteCodeInTryBlock` 捕获。请描述 `TJSGetExceptionObject` 在以下三种情况下分别如何处理：

1. 全局对象中找不到 `"Exception"` 类
2. 找到 `"Exception"` 类但 `CreateNew` 失败
3. `tjs` 参数为 `nullptr`

<details>
<summary>查看答案</summary>

**情况 1：找不到 "Exception" 类**

```cpp
if(TJS_SUCCEEDED(global->PropGet(0, Exception_name.c_str(), ...))) {
    // 不会进入这里
} else {
    *dest = msg;  // ← 执行这里：目标寄存器直接赋值为消息字符串
}
```

catch 变量拿到的是一个**字符串**而非 Exception 对象。TJS2 脚本中 `catch(e)` 的 `e` 将是字符串类型，`e.message` 访问会失败（字符串没有 `.message` 属性），但 `e` 本身可以被输出为错误消息。

**情况 2：CreateNew 失败**

```cpp
if(TJS_SUCCEEDED(excpobj->CreateNew(0, nullptr, nullptr, &obj, 1, &pmsg, excpobj))) {
    // 不会进入这里
} else {
    *dest = msg;  // ← 执行这里：同样降级为字符串
}
```

与情况 1 相同，catch 变量拿到字符串。CreateNew 失败可能的原因：Exception 类的构造器本身抛出了异常（被 TJS_SUCCEEDED 宏检测到的错误码），或者内存不足。

**情况 3：tjs 为 nullptr**

```cpp
void TJSGetExceptionObject(tTJS *tjs, tTJSVariant *dest, ...) {
    if(!tjs) return;  // ← 直接返回，什么都不做
    // ...
}
```

函数直接返回，`dest` 寄存器**不被修改**。这意味着 catch 变量保持之前的值（可能是 `void` 或之前的值）。这是最极端的情况——引擎实例已经不存在了，只能做到"不崩溃"。

**设计思想**：异常处理本身不应该产生新的异常。三级降级策略（正常对象 → 字符串 → 什么都不做）确保了程序在任何状态下都能安全地处理异常。

</details>

### 题目 3：tTJSStackTracer 的 InTry 帧过滤

给定以下调用栈状态（从底到顶）：

```
[0] global()          InTry=false  line=1
[1] main()            InTry=false  line=10
[2] processData()     InTry=false  line=25
[3] try_block_1       InTry=true   line=30
[4] try_block_2       InTry=true   line=31
[5] try_block_3       InTry=true   line=32
[6] innerFunc()       InTry=false  line=50
[7] try_block_4       InTry=true   line=55
[8] deepFunc()        InTry=false  line=70
```

1. `GetTraceString(0)` 的输出是什么？（`limit=0` 表示无限制）
2. `GetTraceString(3)` 的输出是什么？
3. 哪些帧会被跳过？为什么？

<details>
<summary>查看答案</summary>

**分析 InTry 跳过逻辑**：从栈顶向栈底遍历（i 从 8 到 0），对于每个 InTry=true 的帧，检查 `i > 0 && Stack[i-1].InTry`——如果下方帧也是 InTry，则跳过当前帧。

逐帧分析：
- `[8]` deepFunc, InTry=false → **输出**
- `[7]` try_block_4, InTry=true, `Stack[6].InTry=false` → **输出**（下方不是 InTry）
- `[6]` innerFunc, InTry=false → **输出**
- `[5]` try_block_3, InTry=true, `Stack[4].InTry=true` → **跳过**（下方也是 InTry）
- `[4]` try_block_2, InTry=true, `Stack[3].InTry=true` → **跳过**（下方也是 InTry）
- `[3]` try_block_1, InTry=true, `Stack[2].InTry=false` → **输出**（下方不是 InTry）
- `[2]` processData, InTry=false → **输出**
- `[1]` main, InTry=false → **输出**
- `[0]` global, InTry=false → **输出**

**1. `GetTraceString(0)` 输出：**
```
deepFunc(70) <-- try_block_4(55) <-- innerFunc(50) <-- try_block_1(30) <-- processData(25) <-- main(10) <-- global(1)
```
共 7 个位置（跳过了 `[4]` 和 `[5]`）。

**2. `GetTraceString(3)` 输出：**
```
deepFunc(70) <-- try_block_4(55) <-- innerFunc(50)
```
只输出前 3 个。

**3. 被跳过的帧：**
- `[5]` try_block_3——因为下方 `[4]` 也是 InTry
- `[4]` try_block_2——因为下方 `[3]` 也是 InTry

**为什么跳过**：在 TJS2 中，try 块被实现为"子函数调用"，每层 try 都会在调用栈中增加一个帧。连续嵌套的 try 块会产生多个连续的 InTry 帧。保留最底部的一个帧（代表整个 try 区域的入口），跳过上面的冗余帧，使输出更简洁。

</details>
