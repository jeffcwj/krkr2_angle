本节将涉及以下术语，先有个印象，正文中会逐一详细讲解：

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| 原生实例 | Native Instance (NI) | C++ 端存放对象真实数据的结构体，如 Array 的 `std::vector` 就住在原生实例里 |
| 原生类 | Native Class (NC) | 负责注册 TJS2 方法/属性、创建原生实例的工厂类，相当于"类的定义" |

| 数值索引路由 | Numeric Index Routing | Array 把形如 `"3"` 的字符串成员名识别为数字 3，然后调用 ByNum 系列方法——实现 `arr["3"]` 和 `arr[3]` 等价 |

| 结构化数据 | Structured Data | TJS2 的序列化格式——Array 输出 `[ ... ]`、Dictionary 输出 `%[ "key" => value ]`，可以直接被 TJS2 解析器重新读入 |

| Xorshift128 | Xorshift128 | 一种快速的伪随机数生成算法，用 4 个 32 位状态值通过位移和异或运算产生随机数，比 Mersenne Twister 更快但周期更短 |
| Y2038 问题 | Year 2038 Problem | 32 位 `time_t` 在 2038 年 1 月 19 日溢出——Date 类使用 `time_t` 存储时间，在 32 位系统上会遇到这个问题 |

## §1 内置对象的三层架构模式

### 1.1 统一的架构范式

TJS2 的内置对象（Array、Dictionary、RegExp、Date）都遵循同一个三层架构：

```
┌─────────────────────────────────────────────────┐
│                TJS2 脚本层                       │
│   var arr = new Array();                        │
│   arr[0] = "hello";                             │
│   arr.push("world");                            │
└──────────────────┬──────────────────────────────┘
                   │ Dispatch 调用
┌──────────────────▼──────────────────────────────┐
│  第 3 层：Object Shell（对象壳）                  │
│  tTJSArrayObject : tTJSCustomObject             │
│  ─ 重写 PropGetByNum / PropSetByNum 等方法       │
│  ─ 把 ByNum 调用路由到 NI 的 Items[num]          │
│  ─ 处理闭包追踪（CheckObjectClosureAdd/Remove）  │
└──────────────────┬──────────────────────────────┘
                   │ 直接访问
┌──────────────────▼──────────────────────────────┐
│  第 2 层：Native Instance（原生实例）             │
│  tTJSArrayNI                                    │
│  ─ std::vector<tTJSVariant> Items  // 真实数据   │
│  ─ Assign / SaveStructuredData 等数据操作方法     │
└──────────────────┬──────────────────────────────┘
                   │ 创建
┌──────────────────▼──────────────────────────────┐
│  第 1 层：Native Class（原生类）                  │
│  tTJSArrayClass : tTJSNativeClass               │
│  ─ 注册 TJS2 方法（push/pop/sort/join...）       │
│  ─ 注册 TJS2 属性（count/length）                │
│  ─ CreateNativeInstance() → new tTJSArrayNI      │
│  ─ CreateBaseTJSObject() → new tTJSArrayObject   │
└─────────────────────────────────────────────────┘
```

**为什么需要三层？** 职责分离：

- **Native Class** 只负责"定义"——注册有哪些方法和属性，以及如何创建实例。它是全局单例（singleton），整个引擎只有一个 `tTJSArrayClass` 对象
- **Native Instance** 只负责"存数据"——Array 的 vector、Dictionary 的 Owner 指针、RegExp 的编译后正则表达式。它不了解 Dispatch 协议
- **Object Shell** 负责"桥接"——把 iTJSDispatch2 的标准化调用翻译成对 Native Instance 数据的操作

Math 是个特例——它没有原生实例（因为 Math 上的方法都是静态的，不需要对象级别的数据），所以 Math 只有两层。

### 1.2 工厂函数与单例模式

每个内置对象都通过一个全局工厂函数创建，例如：

```cpp
// tjsArray.cpp 第 1826 行 — Array 的工厂函数
iTJSDispatch2 *TJSCreateArrayObject(iTJSDispatch2 **classout) {
    // 静态单例持有者——tTJSArrayClass 只创建一次
    struct tHolder {
        tTJSArrayClass *Obj;
        tHolder() {
            Obj = new tTJSArrayClass();  // 创建类对象
        }
        ~tHolder() {
            Obj->Release();  // 引擎关闭时释放
        }
    };
    static tHolder holder;  // 懒初始化——首次调用时创建

    // 如果调用方需要类对象的引用，也返回给它
    if (classout) {
        *classout = holder.Obj;
        holder.Obj->AddRef();
    }

    // 调用类对象的 CreateNew 方法创建一个新的 Array 实例
    iTJSDispatch2 *out;
    tTJSVariant val;
    // CreateNew 内部调用 CreateBaseTJSObject() 得到 tTJSArrayObject
    // 再调用 CreateNativeInstance() 得到 tTJSArrayNI
    // 把 NI 绑定到 Object Shell 上
    holder.Obj->CreateNew(0, NULL, NULL, &out, 0, NULL, holder.Obj);
    return out;  // 返回可以直接使用的 Array 对象
}
```

这个模式在 Dictionary、RegExp 等类中完全一样。使用静态局部变量（static holder）实现了线程安全的懒初始化（C++11 保证静态局部变量初始化的线程安全性）。`ClassID` 是关键——每个内置类型都有一个唯一的 ClassID，VM 通过它来识别"这个对象是不是 Array"或"是不是 Dictionary"。

## §2 Array——动态数组

### 2.1 数据结构：std::vector 作为后端存储

Array 的核心数据就是一个 `std::vector<tTJSVariant>`：

```cpp
// tjsArray.h 第 89-137 行 — Array 原生实例
class tTJSArrayNI : public tTJSNativeInstance {
public:
    std::vector<tTJSVariant> Items;  // 核心数据——所有元素存在这里

    // 数据操作方法
    void Assign(iTJSDispatch2 *source);           // 从另一个对象复制
    void SaveStructuredData(/* ... */);            // 序列化为文本
    void SaveStructuredBinary(/* ... */);           // 序列化为二进制
    void AssignStructure(/* ... */);                // 深拷贝（处理循环引用）
};
```

**为什么用 `std::vector` 而不是 `std::deque`？** 源码中有一个残留的 `#include <deque>` 和一行引用了 deque 迭代器的注释（第 1675 行），说明早期可能考虑过 deque。最终选择 vector 的原因是：
- 连续内存布局，缓存友好
- 通过索引随机访问是 O(1)，这是数组最常见的操作
- `tTJSVariant` 的大小固定（约 16 字节），vector 的内存重分配成本可控

### 2.2 数值索引路由——IsNumber() 辅助函数

TJS2 脚本中 `arr[3]` 和 `arr["3"]` 的效果完全一样，这是通过 `IsNumber()` 辅助函数实现的：

```cpp
// tjsArray.cpp 第 58-93 行 — 判断成员名是否为数字
static bool IsNumber(const tjs_char *name, tjs_int &num) {
    // 跳过前导空格
    while (*name && *name == ' ') name++;

    // 检查可选的负号
    bool sign = false;
    if (*name == '-') { sign = true; name++; }

    // 检查是否全是数字字符
    const tjs_char *start = name;
    while (*name >= '0' && *name <= '9') name++;

    // 跳过尾部空格
    while (*name && *name == ' ') name++;

    // 如果消耗完了整个字符串且至少有一个数字
    if (*name == 0 && name != start) {
        // 用 TJS_atoi 把字符串转为整数
        num = TJS_atoi(start);
        if (sign) num = -num;
        return true;   // 是数字索引
    }
    return false;      // 不是数字，当作普通成员名
}
```

这个函数被 `tTJSArrayObject` 的非 ByNum 方法调用。比如当 TJS2 脚本执行 `arr["3"] = value` 时：

1. VM 调用 `PropSet(0, "3", NULL, &value, objthis)`
2. `tTJSArrayObject::PropSet` 调用 `IsNumber("3", num)` → 返回 true，num = 3
3. 内部转发到 `PropSetByNum(0, 3, &value, objthis)` —— 写入 `Items[3]`

如果成员名不是数字（比如 `"count"` 或 `"push"`），IsNumber 返回 false，就走 `tTJSCustomObject` 的常规哈希表查找路径。

### 2.3 自动扩容——PropSetByNum 的行为

Array 在索引赋值时会自动扩展：

```cpp
// tjsArray.cpp 第 1612 行附近 — PropSetByNum
tjs_error tTJSArrayObject::PropSetByNum(tjs_uint32 flag,
    tjs_int num, const tTJSVariant *param,
    iTJSDispatch2 *objthis)
{
    // 获取原生实例
    tTJSArrayNI *ni;
    ARRAY_GET_NI(ni);

    // 支持负索引：-1 表示最后一个元素
    if (num < 0) num += (tjs_int)ni->Items.size();

    // 关键：如果索引超出当前大小，自动扩容
    if (num >= (tjs_int)ni->Items.size()) {
        ni->Items.resize(num + 1);  // 扩展到 num+1
        // 新增的元素自动初始化为 void（tTJSVariant 默认构造）
    }

    // 闭包追踪：先移除旧值的闭包关联，再设置新值
    CheckObjectClosureRemove(ni->Items[num]);
    ni->Items[num] = *param;
    CheckObjectClosureAdd(ni->Items[num]);

    return TJS_S_OK;
}
```

**关键行为：**
- **负索引支持**：`arr[-1]` 等价于 `arr[arr.count - 1]`，和 Python 类似
- **自动扩容**：赋值 `arr[100] = "x"` 时，如果数组只有 3 个元素，会扩展到 101 个元素（索引 3-99 的值为 void）
- **闭包追踪**：每次存入新值前先 `CheckObjectClosureRemove` 旧值，存入后 `CheckObjectClosureAdd` 新值。这确保了如果值是一个闭包对象，其上下文（`this` 指向）被正确管理

### 2.4 小哈希表优化

```cpp
// tjsArray.cpp 第 19 行
#define TJS_ARRAY_BASE_HASH_BITS 3  // 只有 2^3 = 8 个桶

// tTJSArrayObject 构造函数
tTJSArrayObject::tTJSArrayObject()
    : tTJSCustomObject(TJS_ARRAY_BASE_HASH_BITS)  // 传入哈希位数
{
    CallFinalize = false;  // Array 析构时不调用 TJS2 层的 finalize
}
```

普通 `tTJSCustomObject` 默认用较大的哈希表（`TJS_OBJECT_HASH_BITS_DEFAULT = 5`，即 32 个桶），因为普通对象可能有几十个成员。但 Array 的命名成员（named member）很少——只有 `count`、`length` 和几个方法（`push`、`pop` 等都注册在类对象上，不在实例上）。大部分访问走 ByNum 路径，根本不经过哈希表。所以用 3 位（8 个桶）的小哈希表就够了，节省内存。

### 2.5 排序系统——7 种比较器

Array 的 `sort` 方法支持多种排序模式，通过不同的比较器类实现：

```cpp
// tjsArray.cpp 第 135-217 行 — 排序比较器定义

// 1. 普通升序（默认）——使用 tTJSVariant 的 < 运算符
class tSortCompare_NormalAscending {
public:
    bool operator()(const tTJSVariant &lhs, const tTJSVariant &rhs) const {
        return lhs < rhs;  // tTJSVariant 支持跨类型比较
    }
};

// 2. 普通降序——反转比较方向
class tSortCompare_NormalDescending {
public:
    bool operator()(const tTJSVariant &lhs, const tTJSVariant &rhs) const {
        return lhs > rhs;
    }
};

// 3. 数值升序——强制把两边转为数值（Real）再比较
class tSortCompare_NumericAscending {
public:
    bool operator()(const tTJSVariant &lhs, const tTJSVariant &rhs) const {
        return lhs.AsReal() < rhs.AsReal();
    }
};

// 4. 数值降序
class tSortCompare_NumericDescending { /* 类似 */ };

// 5. 字符串升序——强制把两边转为字符串再比较
class tSortCompare_StringAscending {
public:
    bool operator()(const tTJSVariant &lhs, const tTJSVariant &rhs) const {
        return lhs.AsString() < rhs.AsString();
    }
};

// 6. 字符串降序
class tSortCompare_StringDescending { /* 类似 */ };

// 7. 函数回调——用户传入一个闭包作为比较函数
class tSortCompare_Functional {
    iTJSDispatch2 *FuncObj;  // 用户的比较函数
public:
    tSortCompare_Functional(iTJSDispatch2 *func) : FuncObj(func) {}
    bool operator()(const tTJSVariant &lhs, const tTJSVariant &rhs) const {
        tTJSVariant result;
        tTJSVariant args[2] = {lhs, rhs};
        tTJSVariant *pargs[2] = {&args[0], &args[1]};
        // 调用用户函数，传入两个待比较的元素
        FuncObj->FuncCall(0, NULL, NULL, &result, 2, pargs, NULL);
        // 如果返回值 < 0 表示 lhs < rhs
        return result.AsInteger() < 0;
    }
};
```

TJS2 脚本中的使用示例：

```javascript
// TJS2 脚本 — Array 排序用法

var arr = [30, 10, 20, 5, 15];

// 默认排序（普通升序）
arr.sort();  // → [5, 10, 15, 20, 30]

// 使用排序模式字符串
arr.sort("+");   // 普通升序（同默认）
arr.sort("-");   // 普通降序 → [30, 20, 15, 10, 5]
arr.sort("0+");  // 数值升序
arr.sort("0-");  // 数值降序
arr.sort("a+");  // 字符串升序（按字典序）
arr.sort("a-");  // 字符串降序

// 使用自定义比较函数
arr.sort(function(a, b) {
    // 按个位数排序
    return (a % 10) - (b % 10);
});
// → [10, 20, 30, 5, 15]
```

底层实现使用 `std::stable_sort`（稳定排序，保持相等元素的相对顺序），而不是 `std::sort`（不稳定排序）。这对于游戏引擎很重要——比如按优先级排序图层时，相同优先级的图层需要保持它们添加的先后顺序。

**count/length 属性**——读写均支持：

```cpp
// tjsArray.cpp — count 属性注册
```

### 2.7 结构化数据序列化

Array 可以序列化为人类可读的 TJS2 表达式格式：

```cpp
// tjsArray.cpp 第 1114 行附近 — SaveStructuredData
void tTJSArrayNI::SaveStructuredData(
    std::vector<iTJSDispatch2 *> &stack,  // 递归检测栈
    ttstr &result,                         // 输出字符串
    const ttstr &indentstr)                // 缩进前缀
{
    // 检查循环引用——如果当前对象已在 stack 中，说明出现了递归
    for (auto &item : stack) {
        if (item == Owner) {
            result += TJS_W("null /* (递归引用) */");
            return;
        }
    }
    stack.push_back(Owner);  // 入栈

    result += TJS_W("(const) [\n");  // 开头

    ttstr nextindent = indentstr + TJS_W("\t");  // 子元素缩进

    for (tjs_uint i = 0; i < Items.size(); i++) {
        result += nextindent;

        tTJSVariantType type = Items[i].Type();
        if (type == tvtObject) {
            // 如果元素是对象（Array 或 Dictionary），递归序列化
            iTJSDispatch2 *obj = Items[i].AsObjectNoAddRef();
            tTJSArrayNI *arrayNI = NULL;
            tTJSDictionaryNI *dictNI = NULL;

            if (TJS_SUCCEEDED(obj->NativeInstanceSupport(
                    TJS_NIS_GETINSTANCE, tTJSArrayClass::ClassID,
                    (iTJSNativeInstance**)&arrayNI))) {
                // 元素是 Array → 递归
                arrayNI->SaveStructuredData(stack, result, nextindent);
            } else if (TJS_SUCCEEDED(obj->NativeInstanceSupport(
                    TJS_NIS_GETINSTANCE, tTJSDictionaryClass::ClassID,
                    (iTJSNativeInstance**)&dictNI))) {
                // 元素是 Dictionary → 递归
                dictNI->SaveStructuredData(stack, result, nextindent);
            } else {
                // 其他对象类型→写成 void
                result += TJS_W("void");
            }
        } else {
            // 基本类型（整数、实数、字符串、void）→ 直接输出字面量
            result += TJSVariantToExpressionString(Items[i]);
        }

        if (i < Items.size() - 1)
            result += TJS_W(",\n");  // 逗号分隔
        else
            result += TJS_W("\n");
    }

    result += indentstr + TJS_W("]");  // 结尾
    stack.pop_back();  // 出栈
}
```

输出格式示例：

```javascript
// TJS2 — 序列化输出示例
// 以下是 saveStruct 的输出，它本身就是合法的 TJS2 表达式

(const) [
    "hello",
    42,
    3.14,
    (const) [
        "嵌套数组",
        100
    ],
    (const) %[
        "name" => "Alice",
        "age" => 25
    ]
]
```

`(const)` 前缀表示这是一个常量表达式。这种格式的巧妙之处在于——输出的文本可以直接被 TJS2 的词法/语法分析器解析，用作数据序列化/反序列化，而不需要单独的序列化格式（如 JSON）。

### 2.8 Clear 的安全释放模式

```cpp
// tjsArray.cpp 第 1407 行 — Clear 方法
void tTJSArrayObject::Clear() {
    tTJSArrayNI *ni;
    if (TJS_FAILED(GetNativeInstance(/* ... */, &ni))) return;

    // 第 1 步：收集所有包含对象的元素
    std::vector<tTJSVariant> temp;
    for (auto &item : ni->Items) {
        if (item.Type() == tvtObject) {
            temp.push_back(item);  // 先复制到临时数组
        }
    }

    // 第 2 步：清空 Items
    ni->Items.clear();  // 现在 Items 为空

    // 第 3 步：释放对象引用
    for (auto &item : temp) {
        CheckObjectClosureRemove(item);
        item.Clear();  // Release 引用
    }
    // temp 析构时，tTJSVariant 的析构函数会再次释放
}
```

**为什么不能直接 `Items.clear()`？** 因为 `tTJSVariant` 存储的对象可能有析构函数（finalize），这些析构函数可能反过来访问当前 Array。如果直接清空，析构过程中访问 Array 就会看到不一致的状态。"先收集、后清空、再释放"的三步策略确保了：清空时没有析构发生，析构时 Array 已经是一致的空状态。

这个模式在 `tTJSCustomObject::DeleteAllMembers` 中也有使用——是 TJS2 对象生命周期管理的一个核心安全模式。

## §3 Dictionary——关联数组

### 3.1 架构：依赖 tTJSCustomObject 的哈希表

与 Array 不同，Dictionary 没有自己的存储容器——它直接复用 `tTJSCustomObject` 内置的哈希表：

```
┌──────────────────────────────────────┐
│  tTJSDictionaryObject               │
│  ───────────────────                │
│  extends tTJSCustomObject           │
│  ┌─────────────────────────────┐    │
│  │ HashTable (继承自父类)       │    │
│  │  "name" → tTJSVariant("A")  │    │
│  │  "age"  → tTJSVariant(25)   │    │
│  │  "list" → tTJSVariant(obj)  │    │
│  └─────────────────────────────┘    │
│                                      │
│  tTJSDictionaryNI (原生实例)         │
│  ┌─────────────────────────────┐    │
│  │ Owner → 指向外层的          │    │
│  │         tTJSDictionaryObject│    │
│  │ // 没有自己的数据存储！       │    │
│  │ // 通过 Owner 访问哈希表     │    │
│  └─────────────────────────────┘    │
└──────────────────────────────────────┘
```

```cpp
// tjsDictionary.h 第 51-153 行
class tTJSDictionaryNI : public tTJSNativeInstance {
public:
    tTJSCustomObject *Owner;  // 指向包含自己的 tTJSDictionaryObject

    // Construct 时建立关联
    tjs_error Construct(tjs_int numparams, tTJSVariant **param,
                        iTJSDispatch2 *tjsobj) {
        // 关键：保存 Owner 指针
        Owner = static_cast<tTJSCustomObject *>(tjsobj);
        return TJS_S_OK;
    }
};
```

**为什么 Dictionary 的 NI 不存数据？** 因为 `tTJSCustomObject` 的哈希表就是天然的键值存储。Dictionary 只需要修改少量行为（PropGet 对缺失成员的处理、Operation 的自动创建），不需要额外的数据结构。这种设计非常优雅——最大化代码复用。

### 3.2 核心语义：缺失成员返回 void

Dictionary 和普通对象的关键区别在于：访问不存在的成员时，Dictionary 返回 void 而不是报错：

```cpp
// tjsDictionary.cpp 第 732 行 — PropGet 重写
tjs_error tTJSDictionaryObject::PropGet(
    tjs_uint32 flag, const tjs_char *membername,
    tjs_uint32 *hint, tTJSVariant *result,
    iTJSDispatch2 *objthis)
{
    // 先尝试普通查找
    tjs_error hr = inherited::PropGet(flag, membername, hint,
                                       result, objthis);

    if (hr == TJS_E_MEMBERNOTFOUND &&
        !(flag & TJS_MEMBERMUSTEXIST))
    {
        // 成员不存在，但调用方没有要求"必须存在"
        // → 返回 void 而不是错误
        if (result) result->Clear();  // 设为 void
        return TJS_S_OK;              // 返回成功
    }

    return hr;
}
```

这意味着在 TJS2 脚本中：

```javascript
// TJS2 脚本 — Dictionary 的 void 语义
var dic = %["name" => "Alice", "age" => 25];

var name = dic.name;       // → "Alice"
var hobby = dic.hobby;     // → void（不报错！）

// 利用 void 语义做存在性检查
if (dic.hobby === void) {
    // 键不存在
    System.inform("hobby 未设置");
}

// 对比普通对象：
var obj = new Object();
var x = obj.nonexistent;   // → 抛出异常 TJS_E_MEMBERNOTFOUND
```

**注意 `TJS_MEMBERMUSTEXIST` 标志**——如果调用方明确要求成员必须存在，Dictionary 也会返回 `TJS_E_MEMBERNOTFOUND`。这个标志通常在 `typeof` 运算符等需要区分"成员是 void"和"成员不存在"的场景中使用。

### 3.3 自动创建语义——Operation 方法

当对 Dictionary 的不存在成员执行操作（如 `+=`、`++`）时，Dictionary 会自动创建该成员：

```cpp
// tjsDictionary.cpp 第 759 行 — Operation 重写
tjs_error tTJSDictionaryObject::Operation(
    tjs_uint32 flag, const tjs_char *membername,
    tjs_uint32 *hint, tTJSVariant *result,
    const tTJSVariant *param, iTJSDispatch2 *objthis)
{
    // 先尝试普通操作
    tjs_error hr = inherited::Operation(flag, membername, hint,
                                         result, param, objthis);

    if (hr == TJS_E_MEMBERNOTFOUND) {
        // 成员不存在→先创建一个 void 成员
        tTJSVariant voidVal;  // 默认构造 = void
        hr = inherited::PropSet(
            TJS_MEMBERENSURE,  // 确保创建
            membername, hint, &voidVal, objthis);
        if (TJS_FAILED(hr)) return hr;

        // 再次尝试操作——现在成员存在了
        hr = inherited::Operation(flag, membername, hint,
                                   result, param, objthis);
    }
    return hr;
}
```

实际效果：

```javascript
// TJS2 脚本 — 自动创建成员
var dic = %[];

// 对不存在的成员执行 += 操作
dic.count += 1;
// 内部流程：
// 1. Operation("count", +=, 1) → E_MEMBERNOTFOUND
// 2. 自动创建 dic.count = void
// 3. 重试 Operation → void + 1 = 1（void 在数值上下文中等于 0）
// 结果：dic.count == 1

dic.count += 5;
// → dic.count == 6

// 这种行为让 Dictionary 非常适合做计数器
var wordCount = %[];
var words = ["hello", "world", "hello", "tjs", "world", "hello"];
for (var i = 0; i < words.count; i++) {
    wordCount[words[i]] += 1;
}
// wordCount == %["hello" => 3, "world" => 2, "tjs" => 1]
```

### 3.4 自定义哈希表大小——CreateNew 优化

当你知道 Dictionary 将存储大量键值对时，可以在创建时预设哈希表大小，避免频繁的动态扩容（rehash）：

```cpp
// tjsDictionary.cpp 第 262-316 行 — CreateNew 重写
tjs_error tTJSDictionaryClass::CreateNew(
    tjs_uint32 flag, const tjs_char *membername,
    tjs_uint32 *hint, iTJSDispatch2 **result,
    tjs_int numparams, tTJSVariant **param,
    iTJSDispatch2 *objthis)
{
    // 检查第一个参数是否为整数且 >= 8
    if (numparams >= 1 && param[0]->Type() == tvtInteger) {
        tjs_int count = param[0]->AsInteger();
        if (count >= 8) {
            // 计算最优 hashbits
            // 目标：桶数 ≈ count，减少冲突
            tjs_int v = count;
            tjs_int r = 0;

            // 找到最高位的位置（类似 log2）
            if (v & 0xFFFF0000) { v >>= 16; r += 16; }
            if (v & 0x0000FF00) { v >>= 8;  r += 8;  }
            if (v & 0x000000F0) { v >>= 4;  r += 4;  }

            // 微调：查表决定是否需要多加 1-2 位
            tjs_int hashbits = r +
                ((0xffffaa50 >> (v * 2)) & 0x03) + 2;

            // 用计算出的 hashbits 创建对象
            *result = new tTJSDictionaryObject(hashbits);
            // ... 后续初始化
        }
    }
    // 默认路径：使用标准哈希位数
    // ...
}
```

TJS2 脚本中的使用：

```javascript
// TJS2 脚本 — 预分配 Dictionary
// 当你知道会有大量键值对时，传入预期数量
var bigDic = new Dictionary(10000);
// 内部会计算 hashbits ≈ 14（2^14 = 16384 个桶）
// 这样插入 10000 个键时几乎不需要 rehash

// 对比：不预分配
var smallDic = %[];  // 默认 hashbits = 5（32 个桶）
// 插入 10000 个键时会触发多次 rehash：
// 32→64→128→256→512→1024→2048→4096→8192→16384
// 每次 rehash 都要重新计算所有键的桶位置——非常昂贵
```

### 3.5 Assign 的预计数优化

Dictionary 在从另一个对象复制成员时，会先计算源对象的成员数量，然后预分配哈希表：

```cpp
// tjsDictionary.cpp 第 344 行附近 — Assign 方法

// 第 1 步：先计算源对象有多少个成员
struct tSaveMemberCountCallback : tTJSDispatch {
    tjs_int Count = 0;
    tjs_error FuncCall(/* ... */) override {
        Count++;  // 每枚举到一个成员就加 1
        return TJS_S_OK;
    }
};

tSaveMemberCountCallback counter;
source->EnumMembers(TJS_IGNOREPROP, &counter, source);
// 现在 counter.Count 就是源对象的成员数

// 第 2 步：用成员数重建哈希表
Owner->RebuildHash(counter.Count);
// RebuildHash 内部计算最优 hashbits 并重新分配桶数组

// 第 3 步：逐个复制成员
struct tAssignCallback : tTJSDispatch {
    tTJSCustomObject *Dest;
    tjs_error FuncCall(/* ... */) override {
        // 把每个成员的 name → value 对写入目标
        Dest->PropSet(TJS_MEMBERENSURE, name, NULL, value, Dest);
        return TJS_S_OK;
    }
};

tAssignCallback copier;
copier.Dest = Owner;
source->EnumMembers(TJS_IGNOREPROP, &copier, source);
```

这个两遍扫描（先计数、再复制）的模式看起来浪费了一次遍历，但实际上非常值得——避免了复制过程中的多次 rehash。对于几千个键的 Dictionary，这可以节省数倍的时间。

Dictionary 的序列化格式使用 `%[ ... ]` 语法：


## §4 RegExp——正则表达式

### 4.1 oniguruma 引擎集成

TJS2 的正则表达式不是自己实现的，而是集成了 **oniguruma**（鬼車，一个用 C 编写的高性能正则引擎）。oniguruma 支持多种字符编码和正则语法风格，TJS2 配置为：

- **编码**：`ONIG_ENCODING_UTF16_LE`（因为 TJS2 内部使用 UTF-16LE 字符串）
- **语法**：`ONIG_SYNTAX_PERL`（Perl 兼容正则语法，支持 `\d`、`\w`、`(?:)` 等）

```cpp
// tjsRegExp.cpp — 编译正则表达式
void tTJSNI_RegExp::Compile(const tjs_char *pattern,
                             tjs_uint32 flags)
{
    OnigErrorInfo einfo;
    int r = onig_new(
        &RegEx,                          // 输出：编译后的正则对象
        (const OnigUChar *)pattern,       // 正则表达式文本
        (const OnigUChar *)(pattern + TJS_strlen(pattern)),
        flags & ~tjsflagsmask,            // oniguruma 的标志位
        ONIG_ENCODING_UTF16_LE,          // 编码：UTF-16LE
        ONIG_SYNTAX_PERL,               // 语法：Perl 兼容
        &einfo                           // 错误信息
    );

    if (r != ONIG_NORMAL) {
        // 编译失败——构造错误信息并抛出 TJS2 异常
        char errbuf[ONIG_MAX_ERROR_MESSAGE_LEN];
        onig_error_code_to_str((OnigUChar *)errbuf, r, &einfo);
        TJS_eTJSError(/* 格式化错误信息 */);
    }
}
```

### 4.2 标志系统——TJS 标志与 oniguruma 标志的分离

正则表达式的标志（如 `g`、`i`）存储在一个 32 位整数中，高 8 位是 TJS2 自己的标志，低 24 位是 oniguruma 的标志：

```
┌─────────────────────┬────────────────────────────┐
│  TJS2 标志 (高8位)    │  oniguruma 标志 (低24位)    │
│  bit 31: 全局搜索(g)  │  ONIG_OPTION_IGNORECASE(i) │
│  bit 30-24: 保留      │  ONIG_OPTION_CAPTURE_GROUP │
│                      │  ONIG_OPTION_FIND_NOT_EMPTY│
└─────────────────────┴────────────────────────────┘
```

```cpp
// tjsRegExp.cpp — 标志定义
static const tjs_uint32 globalsearch = (tjs_uint32)(1 << 31);
    // 全局搜索标志——bit 31
static const tjs_uint32 tjsflagsmask = 0xff000000;
    // TJS 标志区域——高 8 位

// 标志字符到标志值的转换
static tjs_uint32 TJSRegExpFlagToValue(tjs_char ch, tjs_uint32 prev) {
    switch (ch) {
        case 'g':  return prev | globalsearch;
            // 全局搜索——匹配所有出现位置
        case 'i':  return prev | ONIG_OPTION_IGNORECASE;
            // 忽略大小写
        default:
            return prev | ONIG_OPTION_CAPTURE_GROUP
                        | ONIG_OPTION_FIND_NOT_EMPTY;
            // 默认：启用捕获组 + 不匹配空字符串
    }
}
```

**为什么要分离？** 因为 `g`（全局搜索）不是 oniguruma 的概念——oniguruma 每次只做单次匹配，全局搜索是 TJS2 自己在循环中反复调用 oniguruma 实现的。把全局标志放在高位，传给 oniguruma 前用 `flags & ~tjsflagsmask` 屏蔽掉，就不会干扰 oniguruma 的正常工作。

### 4.3 Exec 与状态追踪

RegExp 对象维护了一套丰富的匹配状态，每次调用 `exec` 或 `match` 后会更新：

```cpp
// tjsRegExp.h 第 25-47 行 — RegExp 原生实例
class tTJSNI_RegExp : public tTJSNativeInstance {
public:
    regex_t *RegEx;       // oniguruma 编译后的正则对象
    tjs_uint32 Flags;     // 标志位（含全局搜索标志）
    tjs_int Start;        // 下次搜索的起始位置

    // 匹配结果
    iTJSDispatch2 *Array; // 匹配结果数组（包含捕获组）
    tjs_int Index;        // 匹配位置（在原字符串中的索引）
    ttstr Input;          // 原始输入字符串
    tjs_int LastIndex;    // 匹配结束位置

    // RegExp 静态属性（Perl 风格）
    ttstr LastMatch;      // 最后一次匹配的文本（$&）
    ttstr LastParen;      // 最后一个捕获组的文本（$+）
    ttstr LeftContext;    // 匹配位置左边的文本（$`）
    ttstr RightContext;   // 匹配位置右边的文本（$'）
};
```

```cpp
// tjsRegExp.cpp 第 681 行附近 — Exec 方法的状态更新
void tTJSNC_RegExp::Exec(/* ... */) {
    OnigRegion *region = onig_region_new();

    int r = onig_search(
        ni->RegEx,
        (const OnigUChar *)input_str,
        (const OnigUChar *)(input_str + input_len),
        (const OnigUChar *)(input_str + ni->Start),  // 起始位置
        (const OnigUChar *)(input_str + input_len),
        region, ONIG_OPTION_NONE
    );

    if (r >= 0) {
        // 匹配成功——更新所有状态
        ni->Index = region->beg[0] / sizeof(tjs_char);
        ni->LastIndex = region->end[0] / sizeof(tjs_char);

        // 提取匹配文本和上下文
        ni->LastMatch = ttstr(input_str + ni->Index,
                              ni->LastIndex - ni->Index);
        ni->LeftContext = ttstr(input_str, ni->Index);
        ni->RightContext = ttstr(input_str + ni->LastIndex);

        // 如果有捕获组，提取最后一个
        if (region->num_regs > 1) {
            int last = region->num_regs - 1;
            ni->LastParen = ttstr(
                input_str + region->beg[last] / sizeof(tjs_char),
                (region->end[last] - region->beg[last]) / sizeof(tjs_char));
        }

        // 构建匹配结果数组
        // Array[0] = 整体匹配
        // Array[1] = 第一个捕获组
        // Array[2] = 第二个捕获组 ...

        // 如果是全局搜索，推进 Start 到 LastIndex
        if (ni->Flags & globalsearch) {
            ni->Start = ni->LastIndex;
        }
    }

    onig_region_free(region, 1);
}
```

### 4.4 replace 与 split

RegExp 的 `replace` 方法支持字符串替换和函数回调两种模式：

```cpp
// tjsRegExp.cpp 第 71-139 行 — replace_regex
static void replace_regex(/* ... */) {
    // 循环匹配（如果是全局搜索）
    while (/* 还有匹配 */) {
        int r = onig_search(/* ... */);
        if (r < 0) break;

        // 把匹配位置之前的文本追加到结果
        result += ttstr(input + lastend, match_start - lastend);

        if (func != NULL) {
            // 函数回调模式：调用用户函数获取替换文本
            tTJSVariant args[1] = { /* 匹配文本 */ };
            tTJSVariant *pargs[1] = {&args[0]};
            tTJSVariant ret;
            func->FuncCall(0, NULL, NULL, &ret, 1, pargs, NULL);
            result += ret.AsString();
        } else {
            // 字符串替换模式：直接使用替换字符串
            result += replacestr;
        }

        lastend = match_end;
        if (!(flags & globalsearch)) break;  // 非全局→只替换一次
    }

    // 追加最后一次匹配之后的文本
    result += ttstr(input + lastend);
}
```

### 4.5 _compile 方法——正则字面量的内部表示

TJS2 的正则字面量（`/pattern/flags`）在词法分析阶段被转换成 `//flags/pattern` 的内部格式，然后通过 `_compile` 方法解析：

```cpp
// tjsRegExp.cpp — _compile 方法（内部使用）
// 输入格式：//gi/pattern
// 第一对 // 后面是标志字符，第二个 / 后面是正则表达式
void tTJSNC_RegExp::InternalCompile(const ttstr &pattern) {
    const tjs_char *p = pattern.c_str();

    // 跳过开头的 //
    if (*p != '/') TJS_eTJSError(/* ... */);
    p++;
    if (*p != '/') TJS_eTJSError(/* ... */);
    p++;

    // 解析标志字符直到遇到 /
    tjs_uint32 flags = 0;
    while (*p && *p != '/') {
        flags = TJSRegExpFlagToValue(*p, flags);
        p++;
    }

    // 跳过分隔符 /
    if (*p == '/') p++;

    // 剩下的就是正则表达式本体
    Compile(p, flags);
}
```

## §5 Math——静态数学工具类

### 5.1 无实例模式

Math 是 TJS2 中唯一不需要 `new` 的内置对象——所有方法和属性都是静态的：

```cpp
// tjsMath.cpp — 构造函数使用"无实例"声明
TJS_BEGIN_NATIVE_CONSTRUCTOR_DECL_NO_INSTANCE(
    /* typename */ Math)
{
    return TJS_S_OK;  // 不创建原生实例
}
TJS_END_NATIVE_CONSTRUCTOR_DECL(/* typename */ Math)
```

这意味着你永远不会写 `var m = new Math()`，而是直接使用 `Math.sin(x)` 或 `Math.PI`。从 C++ 角度看，Math 只有 Native Class（注册方法和属性）和 Class Object（单例），没有 Native Instance。

### 5.2 Xorshift128 随机数生成器

`Math.random()` 使用 Xorshift128 算法实现，而不是更常见的 Mersenne Twister（MT19937）：

```cpp
// tjsMath.cpp 第 49-78 行 — 随机数生成器
class tTJSXorshift {
    tjs_uint32 x, y, z, w;  // 4 个 32 位状态变量

public:
    tTJSXorshift() {
        // 用当前时间作为种子
        x = (tjs_uint32)time(NULL);
        y = x * 1812433253 + 1;
        z = y * 1812433253 + 1;
        w = z * 1812433253 + 1;
    }

    // 生成一个 32 位随机整数
    tjs_uint32 random_uint32() {
        tjs_uint32 t = x ^ (x << 11);  // 左移异或
        x = y;                          // 状态轮转
        y = z;
        z = w;
        w = (w ^ (w >> 19)) ^ (t ^ (t >> 8));
            // 右移异或 + 混合
        return w;
    }

    // 生成 [0, 1) 范围的浮点数
    double random_real() {
        return random_uint32() * (1.0 / 4294967296.0);
            // 除以 2^32
    }
};

// 全局单例
static tTJSXorshift xorshift;
```

**Xorshift128 vs Mersenne Twister 对比：**

| 特性 | Xorshift128 | MT19937 |
|------|-------------|---------|
| 周期 | 2^128 - 1 | 2^19937 - 1 |
| 状态大小 | 16 字节 | 2.5 KB |
| 速度 | 极快（4 次移位 + 4 次异或） | 中等 |
| 质量 | 通过大部分统计测试 | 通过所有统计测试 |
| 适用场景 | 游戏、一般用途 | 科学计算、密码学 |

对于视觉小说引擎，Xorshift128 的质量完全够用（随机选择对话分支、随机效果等），而且速度更快、内存占用更少。

### 5.3 IEEE 754 精确处理——max 和 min

Math.max 和 Math.min 的实现需要特殊处理两个边界情况：NaN 传播和正负零的区分：

```cpp
// tjsMath.cpp 第 238 行 — Math.max 实现
TJS_BEGIN_NATIVE_METHOD_DECL(max) {
    if (numparams < 2) return TJS_E_BADPARAMCOUNT;

    tjs_real r = *param[0];  // 第一个参数

    for (int i = 1; i < numparams; i++) {
        tjs_real v = *param[i];

        // 检查 NaN——NaN 与任何值比较都返回 false
        // 如果任一参数是 NaN，结果也是 NaN
        if (TJS_IEEE_D_IsNaN(v) || TJS_IEEE_D_IsNaN(r)) {
            TJS_IEEE_D_SetNaN(r);  // 传播 NaN
            break;  // 不需要继续比较
        }

        if (v > r) {
            r = v;
        } else if (v == r) {
            // 两个值相等——但可能一个是 +0.0，一个是 -0.0
            // IEEE 754 规定 +0.0 == -0.0，但 max 应该返回 +0.0
            tjs_uint64 *vi = (tjs_uint64 *)&v;
            tjs_uint64 *ri = (tjs_uint64 *)&r;
            // 检查符号位（最高位）
            // +0.0 的符号位 = 0，-0.0 的符号位 = 1
            // max 取符号位为 0 的（正零）
            if ((*vi & TJS_IEEE_D_SIGN_MASK) == 0) {
                r = v;  // v 是 +0.0，选择它
            }
        }
    }

    if (result) *result = r;
    return TJS_S_OK;
}
```

**为什么需要区分 +0.0 和 -0.0？** 在大多数编程中这无所谓，但在数值计算中，`1.0 / +0.0 = +Infinity` 而 `1.0 / -0.0 = -Infinity`。正确处理零的符号可以避免下游计算出现意外的负无穷。

## §6 Date——日期时间

### 6.1 基于 time_t 的存储模型

Date 类使用 C 标准库的 `time_t` 存储时间，这是从 1970 年 1 月 1 日 00:00:00 UTC 开始的秒数：

```cpp
// tjsDate.h 第 20-27 行
class tTJSNI_Date : public tTJSNativeInstance {
public:
    time_t DateTime;  // 核心数据——Unix 时间戳（秒级精度）
};
```

**Y2038 问题**：在 32 位系统上，`time_t` 是 `int32_t`，最大值为 2,147,483,647，对应 2038 年 1 月 19 日 03:14:07 UTC。超过这个时间就会溢出。源码中有注释承认了这个问题：

```cpp
// tjsDate.cpp — 注释
// time_t may cause Y2038 problem
// → 在 64 位系统上 time_t 是 64 位，不存在此问题
// → 在 32 位 Android/Windows 上仍有风险
```

### 6.2 构造函数——三种创建方式

```cpp
// tjsDate.cpp 第 53-92 行 — 构造函数
TJS_BEGIN_NATIVE_CONSTRUCTOR_DECL(Date) {
    if (numparams == 0) {
        // 方式 1：无参数 → 当前时间
        ni->DateTime = time(NULL);
    }
    else if (numparams >= 1 && param[0]->Type() == tvtString) {
        // 方式 2：字符串 → 解析日期
        ni->DateTime = TJSParseDateString(param[0]->GetString());
        // TJSParseDateString 使用 bison 生成的解析器
    }
    else if (numparams >= 3) {
        // 方式 3：数值参数 → 年/月/日/时/分/秒
        struct tm t;
        memset(&t, 0, sizeof(t));
        t.tm_year = (int)*param[0] - 1900;  // tm_year 从 1900 算起
        t.tm_mon  = (int)*param[1];          // tm_mon 从 0 算起
        t.tm_mday = (int)*param[2];
        if (numparams >= 4) t.tm_hour = (int)*param[3];
        if (numparams >= 5) t.tm_min  = (int)*param[4];
        if (numparams >= 6) t.tm_sec  = (int)*param[5];
        ni->DateTime = mktime(&t);  // 转换为 time_t
    }
    return TJS_S_OK;
}
```

TJS2 脚本示例：

```javascript

// 方式 1：当前时间

// 方式 2：解析日期字符串

// 内部使用 bison 生成的 tTJSDateParser 解析

// 方式 3：数值参数（年、月、日、时、分、秒）

// 注意：月份从 0 开始（0=一月，11=十二月）

// 读取各部分

System.inform("月: " + d2.getMonth());    // → 0（一月）
System.inform("日: " + d2.getDate());     // → 15

System.inform("分: " + d2.getMinutes());  // → 30
System.inform("秒: " + d2.getSeconds());  // → 0

```

### 6.3 set*/get* 方法的 localtime/mktime 模式

所有 set 方法都使用相同的 `localtime → 修改 → mktime` 模式：

```cpp
// tjsDate.cpp — setYear 方法（其他 set* 方法结构完全一样）
TJS_BEGIN_NATIVE_METHOD_DECL(setYear) {
    TJS_GET_NATIVE_INSTANCE(ni, tTJSNI_Date);

    // 第 1 步：把 time_t 转换为本地时间结构体
    struct tm *t = localtime(&ni->DateTime);
    struct tm tmp;
    memcpy(&tmp, t, sizeof(tmp));
    // 为什么要 memcpy？因为 localtime 返回的是静态内部缓冲区的指针
    // 如果不拷贝，下次调用 localtime 时数据就会被覆盖

    // 第 2 步：修改目标字段
    tmp.tm_year = (int)*param[0] - 1900;
    // tm_year 是从 1900 年算起的偏移量

    // 第 3 步：转换回 time_t
    ni->DateTime = mktime(&tmp);
    // mktime 会自动处理溢出——比如设置 tm_mday = 32
    // 会自动进位到下个月

    return TJS_S_OK;
}
```

**mktime 的自动进位特性**很方便：

```javascript
// TJS2 脚本 — mktime 的自动进位
var d = new Date(2024, 0, 31);  // 1月31日
d.setMonth(1);                   // 设置为2月
// 2024年2月没有31日
// mktime 自动调整为 3月2日（2024是闰年，2月有29天）
System.inform(d.getMonth());     // → 2（三月）
System.inform(d.getDate());      // → 2
```

### 6.4 毫秒级 API 与秒级存储

Date 对外暴露毫秒级的时间 API（与 JavaScript 兼容），但内部存储是秒级的 `time_t`：

```cpp
// tjsDate.cpp — getTime 返回毫秒
TJS_BEGIN_NATIVE_METHOD_DECL(getTime) {
    *result = (tjs_real)ni->DateTime * 1000.0;
    // time_t（秒）× 1000 → 毫秒
    return TJS_S_OK;
}

// setTime 接收毫秒
TJS_BEGIN_NATIVE_METHOD_DECL(setTime) {
    ni->DateTime = (time_t)((tjs_real)*param[0] / 1000.0);
    // 毫秒 ÷ 1000 → time_t（秒），小数部分被截断
    return TJS_S_OK;
}
```

这意味着 Date 的精度实际上只有秒级——毫秒信息在存储时丢失。对于视觉小说引擎来说，秒级精度对于存档时间戳、日期显示等场景完全足够。

### 6.5 bison 生成的日期解析器

`TJSParseDateString` 函数使用从 `bison/tjsdate.y` 生成的解析器来解析各种日期格式：

```
源文件：bison/tjsdate.y
         ↓ bison 编译
生成文件：tjsDateParser.h / tjsDateParser.cpp
         ↓ 被 TJSParseDateString 调用
输入："2024-01-15 14:30:00"
输出：time_t 值
```

支持的格式包括但不限于：
- `"2024-01-15"` — ISO 日期
- `"2024/01/15"` — 斜杠分隔
- `"Jan 15, 2024"` — 英文月份名
- `"2024-01-15 14:30:00"` — 日期+时间
- `"Mon Jan 15 2024 14:30:00"` — 带星期

## 本节小结

- TJS2 的五大内置对象都遵循**三层架构**（Native Class → Native Instance → Object Shell），Math 除外（无实例）
- **Array** 使用 `std::vector<tTJSVariant>` 存储，支持负索引、自动扩容、7 种排序模式。通过 `IsNumber()` 实现字符串索引到数值索引的路由
- **Dictionary** 复用 `tTJSCustomObject` 的哈希表作为存储，两大核心语义：缺失成员返回 void、Operation 自动创建。支持预设哈希表大小和 Assign 的预计数优化
- **RegExp** 集成 oniguruma 正则引擎（UTF-16LE + Perl 语法），标志系统高 8 位是 TJS2 的（全局搜索），低 24 位是 oniguruma 的。Exec 维护丰富的匹配状态
- **Math** 是纯静态工具类，使用 Xorshift128 做随机数。max/min 正确处理 NaN 传播和正负零
- **Date** 基于 `time_t` 存储（秒级精度），所有 set* 方法用 `localtime → 修改 → mktime` 模式。对外暴露毫秒 API 兼容 JavaScript
- **结构化数据**序列化输出合法的 TJS2 表达式，支持循环引用检测
- **安全释放**模式（收集-清空-释放）防止析构过程中的数据竞争

```
   - arr[5] = 99（覆盖了刚才的 50）
```

### 题目 3：理解 RegExp 标志系统

解释以下代码为什么 `globalsearch` 标志要放在 bit 31 而不是低位（如 bit 0）：

```cpp
static const tjs_uint32 globalsearch = (tjs_uint32)(1 << 31);
static const tjs_uint32 tjsflagsmask = 0xff000000;
```

<details>
<summary>查看答案</summary>

```
原因：oniguruma 库使用低 24 位（0x00FFFFFF）存储自己的选项标志，
如 ONIG_OPTION_IGNORECASE、ONIG_OPTION_MULTILINE 等。

TJS2 把自己的标志（如全局搜索 g）放在高 8 位（0xFF000000），
这样就不会和 oniguruma 的标志冲突。

传给 oniguruma 时用位掩码屏蔽：
  onig_flags = flags & ~tjsflagsmask;  // 清除高 8 位
  // 只保留 oniguruma 认识的标志

检查全局搜索时直接用位与：
  if (flags & globalsearch) { /* 全局搜索模式 */ }

这是一种经典的"标志位分区"设计模式——多个子系统共享同一个
标志字段，各自使用不同的位区域，互不干扰。

如果把 globalsearch 放在 bit 0：
- 可能和 oniguruma 的某个标志冲突
- 传给 oniguruma 前需要逐个检查和移除 TJS2 标志
- 容易出错，维护困难
```

</details>
