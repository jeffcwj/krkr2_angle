# 01 - 需求分析与类设计

## 术语预览

| 术语 | 含义 |
|------|------|
| **内置类（Built-in Class）** | 编译进引擎核心的TJS2类，使用低级宏方式（TJS_BEGIN_NATIVE_MEMBERS）注册，如Array、Date、Math等 |
| **StringBuffer** | 本章实战项目——一个高效的字符串拼接类，解决TJS2中字符串连接操作效率低下的问题 |
| **NI类（Native Instance）** | 继承自tTJSNativeInstance的C++类，每个TJS2对象实例持有一个NI，负责管理C++侧的状态和资源 |
| **NC类（Native Class）** | 继承自tTJSNativeClass的C++类，代表TJS2中的一个类（不是实例），负责成员注册和实例工厂 |
| **不可变字符串（Immutable String）** | TJS2中ttstr的特性——每次修改都会创建新字符串对象，原字符串不变。这导致大量拼接操作时产生大量临时对象 |
| **ClassID** | 每个原生类的唯一标识符（tjs_int32），由TJSRegisterNativeClass分配，用于实例关联和类型检查 |
| **API设计契约（API Contract）** | 类的公开接口规范，定义了方法签名、参数约束、返回值语义和错误处理策略，是实现之前必须确定的 |

## 1. 问题识别：TJS2字符串拼接的性能瓶颈

### 1.1 TJS2字符串的不可变性

TJS2中的字符串类型（`ttstr`）是**不可变的（immutable）**。这意味着每次字符串操作（拼接、替换、截取）都会创建一个新的字符串对象。这个设计在多数场景下没有问题，但在需要大量拼接的场景中会产生严重的性能问题：

```javascript
// TJS2脚本 - 字符串拼接的性能问题演示

// 场景1：循环拼接（常见于日志收集、HTML生成等）
var result = "";
for (var i = 0; i < 10000; i++) {
    result += "行 " + i + ": 这是一行测试数据\n";
    // 每次 += 操作：
    // 1. 分配新字符串（长度 = 旧字符串 + 新内容）
    // 2. 复制旧字符串内容到新位置
    // 3. 追加新内容
    // 4. 释放旧字符串
    // 总复制量：O(n²)，其中n是最终字符串长度
}

// 场景2：对话文本处理（视觉小说引擎的典型场景）
var dialogue = "";
for (var i = 0; i < lines.count; i++) {
    dialogue += processLine(lines[i]);
    dialogue += "\n";
    // 每行对话产生2次字符串分配和完整复制
}

// 场景3：CSV/配置文件生成
var csv = "名前,年齢,性別\n";
for (var i = 0; i < characters.count; i++) {
    csv += characters[i].name + ",";
    csv += characters[i].age + ",";
    csv += characters[i].gender + "\n";
    // 每个角色产生6次分配（3个 + 和3个 +=）
}
```

### 1.2 性能分析：O(n²) vs O(n)

不可变字符串拼接的核心问题是**二次时间复杂度**。以下图表展示了拼接次数与总复制量的关系：

```
拼接次数 vs 总内存复制量（假设每次追加100字节）

拼接次数    不可变字符串(累计复制)    StringBuffer(累计复制)
   10         5,500 bytes               1,000 bytes
  100       505,000 bytes              10,000 bytes
 1000    50,050,000 bytes             100,000 bytes
10000  5,000,500,000 bytes          1,000,000 bytes

可视化（对数刻度）：

不可变字符串:  ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓ 5.0 GB
StringBuffer:  ▓▓                            1.0 MB
                                          差异: 5000倍
```

> **为什么是O(n²)**：假设每次追加`k`字节。第`i`次追加时，已有`i*k`字节的内容需要复制到新位置。总复制量 = Σ(i*k, i=1..n) = k*n*(n+1)/2 ≈ O(kn²)。StringBuffer使用预分配的缓冲区，只在缓冲区满时才重新分配（通常2倍扩容），总复制量约为O(kn)。

### 1.3 现有解决方案的不足

在当前KrKr2/TJS2生态中，开发者面对字符串拼接问题有以下选项：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 直接`+=`拼接 | 简单直观 | O(n²)性能，大量临时对象 |
| Array + join | 比直接拼接快 | TJS2的Array.join()仍需遍历和分配 |
| 写入文件再读回 | 避免内存问题 | I/O开销大，需要临时文件 |
| C++层面处理 | 最高效 | 需要修改引擎代码，不够通用 |

**我们的方案**：创建一个`StringBuffer`内置类，在C++层面使用`std::wstring`的预分配缓冲区，暴露简洁的TJS2接口，让脚本开发者可以像使用Array一样自然地进行高效字符串拼接。

## 2. 需求规格

### 2.1 功能需求

基于KrKr2作为视觉小说引擎的使用场景，StringBuffer需要支持以下功能：

**核心功能（必须实现）**：
1. **追加（append）**：追加字符串、数字、其他类型到缓冲区末尾
2. **插入（insert）**：在指定位置插入内容
3. **删除（remove）**：删除指定范围的内容
4. **替换（replace）**：替换指定范围或匹配的内容
5. **转字符串（toString）**：将缓冲区内容转为TJS2字符串
6. **清空（clear）**：清空缓冲区但保留分配的内存
7. **长度查询（length属性）**：获取当前内容长度

**扩展功能（建议实现）**：
8. **容量管理（capacity/reserve）**：预分配缓冲区大小
9. **子串查找（indexOf）**：查找子串位置
10. **子串提取（substring）**：提取指定范围的内容
11. **链式调用**：append等方法返回自身，支持`sb.append("a").append("b")`
12. **格式化（format）**：类似printf的格式化追加

### 2.2 非功能需求

| 维度 | 要求 | 理由 |
|------|------|------|
| **性能** | append操作为均摊O(1) | 解决O(n²)问题是核心目标 |
| **内存** | 支持预分配，2倍扩容策略 | 减少重分配次数 |
| **线程安全** | 不要求 | TJS2是单线程的 |
| **编码** | 内部使用宽字符（tjs_char） | 与TJS2字符串类型一致 |
| **兼容性** | 四平台编译通过 | KrKr2支持Win/Linux/macOS/Android |
| **一致性** | API风格与现有TJS2内置类一致 | Array的push/count模式 |

### 2.3 API设计约束

根据KrKr2现有内置类的设计模式，StringBuffer需要遵循以下约束：

```
1. 使用低级宏方式（TJS_BEGIN_NATIVE_MEMBERS）注册
   理由：作为内置类，与Array/Date保持一致

2. 提供finalize方法（TJS_DECL_EMPTY_FINALIZE_METHOD）
   理由：所有原生类必须提供

3. 类名在全局命名空间
   理由：与Array、Date、Math等一致

4. 属性使用TJS2风格（count而非size/length）
   理由：Array使用count，保持一致

5. 方法返回有意义的值
   理由：Array.push返回新长度，我们也应该返回有用的信息
```

### 3.2 方法设计

```javascript
// === 核心方法 ===

// append(value) → StringBuffer自身（支持链式调用）
// 将任意类型的值转为字符串并追加到末尾
// value: 字符串、整数、实数、void均可
sb.append("hello");
sb.append(42);
sb.append(3.14);
sb.append("a").append("b").append("c");  // 链式

// appendLine(value) → StringBuffer自身
// 追加内容并在末尾加换行符
sb.appendLine("第一行");
sb.appendLine("第二行");

// insert(pos, value) → StringBuffer自身
// 在指定位置插入内容
// pos: 0-based索引，超出范围则追加到末尾
sb.insert(0, "前缀: ");  // 在开头插入
sb.insert(5, "插入");     // 在位置5插入

// remove(start, count) → StringBuffer自身
// 删除从start开始的count个字符
// start: 0-based索引
// count: 要删除的字符数，省略则删除到末尾
sb.remove(0, 3);   // 删除前3个字符
sb.remove(5);      // 删除从位置5到末尾的所有内容

// replace(oldStr, newStr) → 替换次数(整数)
// 替换所有匹配的子串
var count = sb.replace("foo", "bar");

// clear() → void
// 清空内容但保留已分配的内存
sb.clear();

// toString() → String
// 将缓冲区内容转为TJS2字符串
// 也会在字符串上下文中自动调用
var str = sb.toString();
var str2 = "" + sb;  // 自动调用toString


// === 查询方法 ===

// indexOf(searchStr, fromIndex) → 整数
// 查找子串首次出现的位置，未找到返回-1
// fromIndex: 可选，开始搜索的位置（默认0）
var pos = sb.indexOf("hello");
var pos2 = sb.indexOf("hello", 10);  // 从位置10开始找

// substring(start, end) → String
// 提取子串，不修改缓冲区
// end: 可选，省略则到末尾
var sub = sb.substring(0, 5);   // 前5个字符
var sub2 = sb.substring(10);    // 从位置10到末尾

// charAt(index) → String
// 获取指定位置的单个字符
var ch = sb.charAt(0);  // 第一个字符


// === 容量管理 ===

// reserve(capacity) → void
// 预分配至少capacity个字符的缓冲区
// 如果当前容量已足够，不做任何事
sb.reserve(10000);  // 预分配10000字符空间

// shrink() → void
// 缩小缓冲区到刚好容纳当前内容
sb.shrink();
```

## 4. 内部数据结构设计

### 4.1 C++类设计

StringBuffer的C++实现使用`std::wstring`作为底层缓冲区。`std::wstring`自带动态扩容机制，但我们需要在其上层添加一些TJS2特定的功能：

```cpp
// tjs2StringBuffer.h - StringBuffer的C++核心类
// 设计原则：
// 1. 不依赖TJS2头文件（纯C++）
// 2. 使用wchar_t兼容TJS2的tjs_char
// 3. 提供清晰的公有接口

#ifndef __TJS2_STRING_BUFFER_H__
#define __TJS2_STRING_BUFFER_H__

#include <string>
#include <cstddef>
#include <algorithm>

class tTJS2StringBuffer {
private:
    std::wstring buffer_;  // 底层缓冲区

public:
    // === 构造 ===

    // 默认构造：预分配defaultCap个字符
    explicit tTJS2StringBuffer(size_t defaultCap = 64) {
        buffer_.reserve(defaultCap);
    }

    // 字符串构造：使用初始内容
    explicit tTJS2StringBuffer(const wchar_t* initial)
        : buffer_(initial ? initial : L"") {}

    // === 修改操作 ===

    // 追加宽字符串
    void append(const wchar_t* str) {
        if (str) buffer_.append(str);
    }

    // 追加宽字符串（带长度）
    void append(const wchar_t* str, size_t len) {
        if (str) buffer_.append(str, len);
    }

    // 追加整数
    void appendInt(long long val) {
        buffer_.append(std::to_wstring(val));
    }

    // 追加浮点数
    void appendReal(double val) {
        // 移除末尾多余的零
        std::wstring s = std::to_wstring(val);
        size_t dot = s.find(L'.');
        if (dot != std::wstring::npos) {
            size_t last = s.find_last_not_of(L'0');
            if (last == dot) last++;  // 保留至少一位小数
            s.erase(last + 1);
        }
        buffer_.append(s);
    }

    // 追加换行符
    void appendLine(const wchar_t* str) {
        if (str) buffer_.append(str);
        buffer_.push_back(L'\n');
    }

    // 插入
    void insert(size_t pos, const wchar_t* str) {
        if (!str) return;
        if (pos > buffer_.size()) pos = buffer_.size();
        buffer_.insert(pos, str);
    }

    // 删除
    void remove(size_t start, size_t count) {
        if (start >= buffer_.size()) return;
        if (count == (size_t)-1 ||
            start + count > buffer_.size())
            count = buffer_.size() - start;
        buffer_.erase(start, count);
    }

    // 替换所有匹配
    int replace(const wchar_t* oldStr,
                const wchar_t* newStr) {
        if (!oldStr || !newStr) return 0;
        std::wstring old(oldStr);
        std::wstring rep(newStr);
        if (old.empty()) return 0;

        int count = 0;
        size_t pos = 0;
        while ((pos = buffer_.find(old, pos))
               != std::wstring::npos) {
            buffer_.replace(pos, old.size(), rep);
            pos += rep.size();
            count++;
        }
        return count;
    }

    // 清空
    void clear() {
        buffer_.clear();
        // 注意：clear()不释放内存，保留capacity
    }

    // === 查询操作 ===

    size_t length() const { return buffer_.size(); }
    size_t capacity() const { return buffer_.capacity(); }
    bool empty() const { return buffer_.empty(); }

    const wchar_t* c_str() const {
        return buffer_.c_str();
    }

    // 查找子串
    int indexOf(const wchar_t* str,
                size_t fromIndex = 0) const {
        if (!str) return -1;
        size_t pos = buffer_.find(str, fromIndex);
        return (pos == std::wstring::npos) ?
            -1 : (int)pos;
    }

    // 提取子串
    std::wstring substring(size_t start,
                           size_t end = (size_t)-1) const {
        if (start >= buffer_.size())
            return std::wstring();
        if (end == (size_t)-1 || end > buffer_.size())
            end = buffer_.size();
        if (start >= end)
            return std::wstring();
        return buffer_.substr(start, end - start);
    }

    // 获取单个字符
    wchar_t charAt(size_t index) const {
        if (index >= buffer_.size()) return 0;
        return buffer_[index];
    }

    // === 容量管理 ===

    void reserve(size_t cap) {
        buffer_.reserve(cap);
    }

    void shrink() {
        buffer_.shrink_to_fit();
    }
};

#endif
```

### 4.2 数据结构选择的理由

为什么选择`std::wstring`而不是其他方案？

| 方案 | 优点 | 缺点 | 是否采用 |
|------|------|------|---------|
| `std::wstring` | 标准库、动态扩容、丰富API | wchar_t平台差异 | ✅ 采用 |
| `std::vector<wchar_t>` | 更底层控制 | 缺少字符串操作 | ❌ |
| `手动管理char*` | 最大灵活性 | 复杂、容易出错 | ❌ |
| `ttstr`（TJS2字符串） | 无需转换 | 不可变，无法追加 | ❌ |
| `rope数据结构` | 理论最优 | 过度工程、实现复杂 | ❌ |

`std::wstring`的扩容策略（通常2倍扩容）保证了`append`操作的均摊O(1)复杂度。在KrKr2的使用场景中（视觉小说脚本，字符串长度通常不超过几MB），这是最合适的选择。

### 4.3 NI类设计

```cpp
// tjs2StringBufferNI.h - NI类设计
// NI类是TJS2对象和C++对象之间的桥梁

#include "tjsNative.h"
#include "tjs2StringBuffer.h"

class tTJS2StringBufferNI : public tTJSNativeInstance {
public:
    tTJS2StringBuffer* sb_;  // C++层的StringBuffer

    tTJS2StringBufferNI() : sb_(nullptr) {}

    // Construct: TJS2 new 操作时调用
    // 处理三种构造参数模式：
    //   无参数 → 默认容量
    //   整数   → 预分配容量
    //   字符串 → 初始内容
    tjs_error TJS_INTF_METHOD Construct(
        tjs_int numparams,
        tTJSVariant **param,
        iTJSDispatch2 *tjs) override
    {
        if (numparams == 0) {
            // 无参数：默认容量
            sb_ = new tTJS2StringBuffer();
        } else if (param[0]->Type() == tvtInteger) {
            // 整数参数：预分配容量
            tjs_int cap = (tjs_int)*param[0];
            if (cap < 0) cap = 0;
            sb_ = new tTJS2StringBuffer((size_t)cap);
        } else if (param[0]->Type() == tvtString) {
            // 字符串参数：初始内容
            ttstr str(*param[0]);
            sb_ = new tTJS2StringBuffer(str.c_str());
        } else {
            // 其他类型：尝试转为字符串
            ttstr str(*param[0]);
            sb_ = new tTJS2StringBuffer(str.c_str());
        }
        return TJS_S_OK;
    }

    // Invalidate: 释放C++资源
    void TJS_INTF_METHOD Invalidate() override {
        if (sb_) {
            delete sb_;
            sb_ = nullptr;
        }
    }
};
```

## 6. 与现有内置类的对比参照

### 6.1 StringBuffer vs Array

StringBuffer的设计参考了Array类的模式，但针对字符串场景做了调整：

| 维度 | Array | StringBuffer | 差异原因 |
|------|-------|-------------|---------|
| NI持有数据 | `vector<tTJSVariant>` | `tTJS2StringBuffer*` | 数组元素vs字符串 |
| 追加方法 | `push(elem)` | `append(str)` | 命名与用途不同 |
| 长度属性 | `count` | `length` | 字符串用length更自然 |
| 索引访问 | `[i]`（operator[] 重载） | `charAt(i)` | 字符串不需要方括号语法 |
| 链式调用 | 不支持 | 支持（append返回自身） | 字符串拼接的常见需求 |
| 容量管理 | 无暴露 | `reserve`/`capacity` | StringBuffer的核心价值 |

### 6.2 StringBuffer vs Math

Math是纯静态类的典范，StringBuffer是实例类——两者在注册模式上有本质区别：

```
Math:
  - TJS_BEGIN_NATIVE_CONSTRUCTOR_DECL_NO_INSTANCE
  - 所有方法用 TJS_END_NATIVE_STATIC_METHOD_DECL
  - 没有NI类

StringBuffer:
  - TJS_BEGIN_NATIVE_CONSTRUCTOR_DECL（有实例）
  - 大多数方法用 TJS_END_NATIVE_METHOD_DECL（实例方法）
  - 需要NI类（tTJS2StringBufferNI）
  - 需要 TJS_GET_NATIVE_INSTANCE 获取NI
```

## 本节小结

- **问题明确**：TJS2字符串的不可变性导致大量拼接时产生O(n²)复杂度和大量临时对象

## 练习题与答案

### 题目1：为什么StringBuffer选择低级宏方式而非ncbind？

分析在KrKr2代码库中，StringBuffer应该使用哪种绑定方式，并给出至少3个理由。

<details>
<summary>查看答案</summary>

StringBuffer应该使用**低级宏方式**（TJS_BEGIN_NATIVE_MEMBERS），理由如下：

1. **一致性**：KrKr2中所有7个内置类（Array、Date、Dictionary、Exception、Math、RandomGenerator、RegExp）都使用低级宏方式。StringBuffer作为内置类应保持一致。

2. **注册时机**：内置类在引擎初始化时注册，不经过插件加载的V2Link流程。ncbind的自动注册机制依赖V2Link入口，不适用于内置类。

3. **性能考虑**：StringBuffer的核心价值是性能优化。低级宏方式跳过ncbind的模板间接层，直接操作tTJSVariant，在高频调用（如循环内的append）场景下避免了不必要的开销。

4. **可变参数支持**：append方法需要接受任意类型的参数（字符串、整数、实数等）。低级宏方式可以直接通过`param[i]->Type()`判断参数类型并分别处理，而ncbind的自动类型推导需要额外的RawCallback支持。

5. **隐藏成员**：低级宏方式支持`TJS_END_NATIVE_HIDDEN_METHOD_DECL`，可以隐藏内部辅助方法不让TJS2脚本直接调用。ncbind不直接支持此功能。

</details>

### 题目2：分析以下StringBuffer API设计的问题

以下API设计有什么问题？请找出至少3个设计缺陷并提出修正方案。

```javascript
// 有问题的API设计
var sb = new StringBuffer();
sb.add("hello");          // 问题1
sb.length = 10;           // 问题2
sb.delete(0, 5);          // 问题3
var s = sb.string;        // 问题4
```

<details>
<summary>查看答案</summary>

**问题分析与修正**：

1. **`add`方法命名不当**
   - 问题：`add`在TJS2中通常用于集合操作（如Array），字符串操作应使用`append`
   - 修正：改为`sb.append("hello")`
   - 理由：与Java的StringBuilder.append()、C#的StringBuilder.Append()命名一致

2. **`length`属性可写**
   - 问题：允许设置length可能导致缓冲区内容被截断或包含未初始化的数据
   - 修正：`length`设为只读属性（TJS_DENY_NATIVE_PROP_SETTER）
   - 理由：如需截断使用`remove(n)`方法更明确，Array的`count`可写是因为数组元素有默认值（void），但字符串截断应该是显式操作

3. **`delete`是TJS2保留字**
   - 问题：`delete`是TJS2（和JavaScript）的关键字，不能用作方法名
   - 修正：改为`sb.remove(0, 5)`
   - 理由：避免与语言关键字冲突导致语法错误

4. **`string`属性语义不明**
   - 问题：`string`作为属性名不够清晰，且与TJS2的String类型名冲突
   - 修正：使用`sb.toString()`方法
   - 理由：`toString()`是TJS2对象的标准转换方法，在字符串上下文中会自动调用，且名称明确表示"转换为字符串"的操作语义

</details>

### 题目3：设计一个StringBuffer的扩展方法

如果要给StringBuffer添加一个`format`方法，支持类似printf的格式化追加，请设计其TJS2接口（参数、返回值、使用示例），并分析实现的技术难点。

<details>
<summary>查看答案</summary>

**接口设计**：

```javascript
// format(formatStr, arg1, arg2, ...) → StringBuffer自身
// 格式化追加，支持以下占位符：
//   %d - 整数
//   %f - 浮点数
//   %s - 字符串
//   %% - 字面%
var sb = new StringBuffer();
sb.format("姓名: %s, 年龄: %d, 身高: %f", "太郎", 25, 175.5);
// 结果: "姓名: 太郎, 年龄: 25, 身高: 175.5"

// 支持链式调用
sb.format("第%d行: %s\n", 1, "你好")
  .format("第%d行: %s\n", 2, "世界");
```

**C++实现签名**：

```cpp
TJS_BEGIN_NATIVE_METHOD_DECL(format)
{
    TJS_GET_NATIVE_INSTANCE(_this, tTJS2StringBufferNI);

    if (numparams < 1) return TJS_E_BADPARAMCOUNT;
    if (param[0]->Type() != tvtString)
        return TJS_E_INVALIDPARAM;

    ttstr fmt(*param[0]);
    const tjs_char* p = fmt.c_str();
    int argIdx = 1;  // 从param[1]开始取参数

    while (*p) {
        if (*p == L'%' && *(p+1)) {
            p++;
            switch (*p) {
                case L'd':
                    if (argIdx < numparams)
                        _this->sb_->appendInt(
                            (tjs_int)*param[argIdx++]);
                    break;
                case L'f':
                    if (argIdx < numparams)
                        _this->sb_->appendReal(
                            (tjs_real)*param[argIdx++]);
                    break;
                case L's':
                    if (argIdx < numparams)
                        _this->sb_->append(
                            ttstr(*param[argIdx++]).c_str());
                    break;
                case L'%':
                    _this->sb_->append(L"%", 1);
                    break;
            }
        } else {
            _this->sb_->append(p, 1);
        }
        p++;
    }

    // 返回自身，支持链式调用
    if (result) *result = tTJSVariant(objthis, objthis);
    return TJS_S_OK;
}
TJS_END_NATIVE_METHOD_DECL(format)
```

**技术难点分析**：

1. **类型安全**：TJS2是动态类型语言，参数类型可能与格式化占位符不匹配（如`%d`传入字符串），需要做类型检查或隐式转换
2. **参数数量验证**：占位符数量可能多于实际参数，需要优雅处理（跳过或输出原始占位符）
3. **宽字符处理**：格式化字符串的遍历需要考虑宽字符编码，特别是在不同平台上wchar_t大小不同
4. **精度控制**：浮点数的精度（如`%0.2f`）需要解析格式化修饰符，增加了实现复杂度
5. **性能**：逐字符遍历格式化字符串可能较慢，但由于format通常不在热循环中调用，可接受

</details>
