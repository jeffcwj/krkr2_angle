## 1. 分段压缩的设计哲学

### 1.1 为什么不整体压缩？

在传统归档格式（如 ZIP）中，每个文件被作为一个整体进行压缩。这种方式有一个致命缺陷：**随机访问**（Random Access）性能极差。要读取压缩文件中间的一个字节，必须先解压从文件开头到该字节之前的所有数据。

对于游戏引擎来说，这是不可接受的。设想一个 10MB 的背景音乐文件被整体压缩后存储在 XP3 归档中。当玩家从游戏中途加载存档时，音频引擎需要 seek 到歌曲中间某个位置开始播放。如果是整体压缩，引擎必须先解压前 5MB 的数据才能读到目标位置——这会造成明显的加载卡顿。

### 1.2 分段压缩的核心思想

XP3 格式采用了一种精妙的折中方案——**分段压缩**（Segmented Compression）。其核心思想是：

1. **切分**：将一个文件切成若干个独立的**段**（Segment），每段通常为几十 KB 到几百 KB
2. **独立压缩**：每个段使用 zlib 独立压缩，段与段之间没有依赖关系
3. **随机访问**：要读取文件中某个位置的数据，只需找到并解压该位置所在的段，而非整个文件
4. **混合模式**：每个段可以独立选择压缩或不压缩（RAW），小段或难以压缩的段可以保持原始状态

这种设计在压缩比（Compression Ratio，压缩后大小与原始大小的比值）和随机访问性能之间取得了良好的平衡：

```
┌─────────────────────────────────────────────────────┐
│ 整体压缩 vs 分段压缩 对比                           │
├─────────────────────────────────────────────────────┤
│                                                     │
│  整体压缩（ZIP 风格）：                             │
│  ┌──────────────────────────────────────────────┐   │
│  │  zlib(整个文件数据)                           │   │
│  └──────────────────────────────────────────────┘   │
│  seek(偏移5MB) → 必须解压 0~5MB 的全部数据          │
│                                                     │
│  分段压缩（XP3 风格）：                             │
│  ┌─────┐┌─────┐┌─────┐┌─────┐┌─────┐┌─────┐       │
│  │seg 0││seg 1││seg 2││seg 3││seg 4││seg 5│       │
│  │zlib ││zlib ││ RAW ││zlib ││zlib ││zlib │       │
│  └─────┘└─────┘└─────┘└─────┘└─────┘└─────┘       │
│  seek(偏移5MB) → 只需解压 seg 3 一个段              │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 1.3 性能权衡分析

分段压缩并非没有代价。以下是详细的权衡分析：

| 维度 | 整体压缩 | 分段压缩 | 说明 |
|------|----------|----------|------|
| 压缩比 | ★★★★★ | ★★★★ | 分段压缩的每个段独立压缩，失去了段间的冗余消除机会，压缩比略低 |
| 顺序读取速度 | ★★★★ | ★★★★ | 两者差异不大，分段压缩有少量段切换开销 |
| 随机读取速度 | ★ | ★★★★★ | 分段压缩只需解压目标段，整体压缩需从头解压 |
| 内存占用 | ★★★ | ★★★★★ | 分段压缩只需缓存单个段，整体压缩可能需要缓存整个解压结果 |
| 索引开销 | ★★★★★ | ★★★ | 分段压缩需要为每个段存储描述信息，增加了索引大小 |
| 实现复杂度 | ★★★★★ | ★★★ | 分段压缩的流实现需要处理段切换、缓存管理等额外逻辑 |

## 2. 段描述符结构详解

### 2.1 tTVPXP3ArchiveSegment 结构体

每个段在内存中用 `tTVPXP3ArchiveSegment` 结构体表示。这个结构体是从 XP3 索引区的 `segm` 子块（sub-chunk）解析而来的：

> **源码位置：** `cpp/core/base/XP3Archive.h` 第 86-93 行

```cpp
// XP3 归档中单个段的描述信息
struct tTVPXP3ArchiveSegment {
    tjs_uint64 Start;        // 该段在归档文件中的绝对起始位置（字节偏移）
    tjs_uint64 Offset;       // 该段在解压后文件中的逻辑偏移（累加值）
    tjs_uint64 OrgSize;      // 段的原始大小（解压后大小）
    tjs_uint64 ArcSize;      // 段在归档中的大小（压缩后大小，若未压缩则等于 OrgSize）
    bool IsCompressed;       // 该段是否经过 zlib 压缩
};
```

每个字段的详细含义：

**`Start`（归档内起始位置）**：这是该段的压缩数据在 XP3 归档文件中的绝对字节偏移。读取时，引擎会调用 `Stream->SetPosition(CurSegment->Start)` 直接跳转到这个位置。

**`Offset`（逻辑偏移）**：这是该段在解压后完整文件中的起始位置。例如，一个 1MB 的文件被分成 4 个各 256KB 的段，则四个段的 `Offset` 分别为 0、262144（256K）、524288（512K）、786432（768K）。这个值在 `SeekToPosition()` 的二分搜索中用于定位。

**`OrgSize`（原始大小）**：段解压后的数据长度。对于未压缩段，`OrgSize == ArcSize`。

**`ArcSize`（归档大小）**：段在归档中实际占用的字节数。如果段被 zlib 压缩，`ArcSize < OrgSize`；如果未压缩，`ArcSize == OrgSize`。

**`IsCompressed`（压缩标记）**：布尔值，指示该段是否使用了 zlib 压缩。引擎根据此标记决定是直接读取还是先解压。

### 2.2 段描述符的解析过程

在索引解析阶段（`tTVPXP3Archive::Init()` 函数），每个文件的 `segm` 子块被逐一解析为 `tTVPXP3ArchiveSegment` 数组。解析过程中会计算累加偏移：

```cpp
// 段描述符的解析伪代码（基于 XP3Archive.cpp Init() 中的逻辑）
// segm 子块中每个段占 28 字节
void ParseSegments(const tjs_uint8 *segmData, tjs_uint segmSize,
                   std::vector<tTVPXP3ArchiveSegment> &segments) {
    tjs_uint64 offset = 0;  // 逻辑偏移累加器

    // 每个段描述占 28 字节：4(flags) + 8(Start) + 8(OrgSize) + 8(ArcSize)
    tjs_uint segCount = segmSize / 28;

    for (tjs_uint i = 0; i < segCount; i++) {
        tTVPXP3ArchiveSegment seg;
        const tjs_uint8 *p = segmData + i * 28;

        // 前 4 字节是压缩标记（flags）
        tjs_uint32 flags = ReadI32FromMem(p);
        // 低 3 位表示压缩方式：0=RAW, 1=zlib
        seg.IsCompressed =
            (flags & TVP_XP3_SEGM_ENCODE_METHOD_MASK) == TVP_XP3_SEGM_ENCODE_ZLIB;

        // 接下来 8 字节是段在归档中的起始位置
        seg.Start = ReadI64FromMem(p + 4);

        // 再接下来 8 字节是原始大小
        seg.OrgSize = ReadI64FromMem(p + 12);

        // 最后 8 字节是归档大小
        seg.ArcSize = ReadI64FromMem(p + 20);

        // 逻辑偏移由前面所有段的 OrgSize 累加得到
        seg.Offset = offset;
        offset += seg.OrgSize;

        segments.push_back(seg);
    }
}
```

> **关键细节**：`Offset` 字段并不存储在 XP3 文件中，而是在解析时动态计算的。这是一个重要的设计决策——它减少了索引数据的大小（每段少 8 字节），同时通过简单的累加即可恢复。

## 3. tTVPXP3ArchiveStream 流类详解

`tTVPXP3ArchiveStream` 是 XP3 归档系统的核心类之一。它继承自 `tTJSBinaryStream`（TJS2 二进制流基类），对外表现为一个普通的可读、可 seek 的字节流，对内则管理着复杂的分段读取、解压缩和缓存逻辑。

### 3.1 类的成员变量

> **源码位置：** `cpp/core/base/XP3Archive.h` 第 162-185 行

```cpp
class tTVPXP3ArchiveStream : public tTJSBinaryStream {
    // --- 归属关系 ---
    tTVPXP3Archive *Owner;    // 所属的 XP3 归档对象（引用计数管理）

    tjs_int StorageIndex;     // 该文件在归档 ItemVector 中的索引号

    // --- 段信息 ---
    std::vector<tTVPXP3ArchiveSegment> *Segments;  // 指向段描述符数组（不拥有所有权）
    tTJSBinaryStream *Stream;   // 底层归档文件流（来自句柄缓存池）
    tjs_uint64 OrgSize;         // 文件的完整原始大小（所有段 OrgSize 之和）

    // --- 当前段状态 ---
    tjs_int CurSegmentNum;              // 当前段编号（从 0 开始）
    tTVPXP3ArchiveSegment *CurSegment;  // 当前段的描述符指针
    tjs_int LastOpenedSegmentNum;       // 上次打开过的段编号（-1 表示还未打开任何段）

    // --- 读取位置 ---
    tjs_uint64 CurPos;          // 当前逻辑读取位置（在解压后完整文件中的位置）
    tjs_uint64 SegmentRemain;   // 当前段剩余可读字节数
    tjs_uint64 SegmentPos;      // 在当前段内的偏移

    // --- 缓存 ---
    tTVPSegmentData *SegmentData;  // 已解压的段数据缓冲区（引用计数管理）
    bool SegmentOpened;            // 当前段是否已"打开"（数据已就绪）

    // --- 过滤器 ---
    tTJSVariant FilterContext;     // 提取过滤器的上下文变量（由内容过滤器设置）
};
```

这些成员变量共同构成了一个**状态机**（State Machine），管理着"当前正在读取哪个段的哪个位置"的完整状态。

### 3.2 构造函数：初始化状态机

> **源码位置：** `cpp/core/base/XP3Archive.cpp` 第 808-828 行

```cpp
tTVPXP3ArchiveStream::tTVPXP3ArchiveStream(
    tTVPXP3Archive *owner, tjs_int storageindex,
    std::vector<tTVPXP3ArchiveSegment> *segments,
    tTJSBinaryStream *stream, tjs_uint64 orgsize) {

    StorageIndex = storageindex;   // 记录文件在归档中的索引
    Segments = segments;           // 指向段描述符数组
    SegmentData = nullptr;         // 初始无缓存数据
    CurSegmentNum = 0;             // 从第 0 段开始
    CurSegment = &(Segments->operator[](0));  // 指向第 0 段描述符
    SegmentPos = 0;                // 段内偏移为 0
    SegmentRemain = CurSegment->OrgSize;  // 剩余字节 = 第 0 段的完整大小
    SegmentOpened = false;         // 段尚未打开（数据未就绪）
    CurPos = 0;                    // 逻辑位置从 0 开始

    LastOpenedSegmentNum = -1;     // 标记：从未打开过任何段

    Owner = owner;
    Owner->AddRef();               // 增加归档对象的引用计数
    Stream = stream;               // 接管底层文件流
    OrgSize = orgsize;             // 记录文件总大小
}
```

构造完成后，状态机处于"指向第 0 段开头，但尚未加载数据"的初始状态。数据的实际加载被推迟到第一次 `Read()` 调用时——这就是**惰性初始化**（Lazy Initialization）的设计模式。

### 3.3 析构函数：清理资源

> **源码位置：** `cpp/core/base/XP3Archive.cpp` 第 831-836 行

```cpp
tTVPXP3ArchiveStream::~tTVPXP3ArchiveStream() {
    // 将底层文件流归还到句柄缓存池（而非直接 delete）
    TVPReleaseCachedArchiveHandle(Owner, Stream);

    // 释放归档对象的引用
    Owner->Release();

    // 释放已缓存的段数据
    if (SegmentData)
        SegmentData->Release();
}
```

注意析构函数中的 `TVPReleaseCachedArchiveHandle`——它并不关闭底层文件流，而是将其归还到**归档句柄缓存池**中，供后续打开同一归档的其他文件时复用。这是一个重要的性能优化，我们在第 5 节详细讨论。

## 4. 分段读取核心逻辑

### 4.1 EnsureSegment()：按需加载段数据

`EnsureSegment()` 是整个分段读取机制的核心。每次 `Read()` 之前都会调用它，确保当前段的数据已经准备好：

> **源码位置：** `cpp/core/base/XP3Archive.cpp` 第 839-893 行

```cpp
void tTVPXP3ArchiveStream::EnsureSegment() {
    // 如果当前段已打开，直接返回（最常见的路径）
    if (SegmentOpened)
        return;

    // 如果是同一个段的重新打开（seek 后回到同一段）
    if (LastOpenedSegmentNum == CurSegmentNum) {
        if (!CurSegment->IsCompressed)
            // 未压缩段：直接 seek 底层流到正确位置
            Stream->SetPosition(CurSegment->Start + SegmentPos);
        // 压缩段：SegmentData 仍在内存中，无需重新解压
        return;
    }

    // --- 需要打开一个新段 ---

    // 释放旧段的缓存数据
    if (SegmentData)
        SegmentData->Release(), SegmentData = nullptr;

    if (CurSegment->IsCompressed) {
        // ===== 压缩段处理 =====

        if (CurSegment->OrgSize >= TVP_SEGCACHE_ONE_LIMIT) {
            // 段太大（≥1MB），不放入缓存，直接解压到临时对象
            Stream->SetPosition(CurSegment->Start);
            SegmentData = new tTVPSegmentData;
            SegmentData->SetData(
                (tjs_uint)CurSegment->OrgSize,  // 解压后大小
                Stream,                          // 从此流读取压缩数据
                (tjs_uint)CurSegment->ArcSize    // 压缩数据大小
            );
        } else {
            // 段大小可接受，尝试从缓存查找
            tTVPSegmentCacheSearchData sdata;
            sdata.Name = Owner->GetName();        // 归档文件名
            sdata.StorageIndex = StorageIndex;    // 文件在归档中的索引
            sdata.SegmentIndex = CurSegmentNum;   // 段编号

            tjs_uint32 hash =
                tTVPSegmentCacheSearchHashFunc::Make(sdata);

            // 在全局段缓存中查找
            SegmentData = TVPSearchFromSegmentCache(sdata, hash);

            if (!SegmentData) {
                // 缓存未命中 → 从归档读取并解压
                Stream->SetPosition(CurSegment->Start);
                SegmentData = new tTVPSegmentData;
                SegmentData->SetData(
                    (tjs_uint)CurSegment->OrgSize,
                    Stream,
                    (tjs_uint)CurSegment->ArcSize
                );
                // 解压后放入缓存
                TVPPushToSegmentCache(sdata, hash, SegmentData);
            }
        }
    } else {
        // ===== 未压缩段处理 =====
        // 直接定位到归档中的正确位置
        Stream->SetPosition(CurSegment->Start + SegmentPos);
    }

    SegmentOpened = true;
    LastOpenedSegmentNum = CurSegmentNum;
}
```

这个函数的决策流程可以用以下流程图表示：

```
EnsureSegment() 调用
       │
       ▼
  SegmentOpened? ──是──→ 直接返回
       │否
       ▼
  同一段重开? ──是──→ 未压缩？seek底层流 → 返回
       │否               压缩？数据还在 → 返回
       ▼
  释放旧 SegmentData
       │
       ▼
  IsCompressed? ──否──→ seek 底层流到 Start+SegmentPos → 完成
       │是
       ▼
  OrgSize ≥ 1MB? ──是──→ 直接解压（不缓存）→ 完成
       │否
       ▼
  查缓存 ──命中──→ 使用缓存数据 → 完成
       │未命中
       ▼
  读取+解压+放入缓存 → 完成
```

### 4.2 SeekToPosition()：二分搜索定位段

当调用者 seek 到文件中的任意位置时，引擎需要快速确定该位置落在哪个段中。由于段的 `Offset` 字段是单调递增的，可以使用**二分搜索**（Binary Search）算法高效定位：

> **源码位置：** `cpp/core/base/XP3Archive.cpp` 第 897-927 行

```cpp
void tTVPXP3ArchiveStream::SeekToPosition(tjs_uint64 pos) {
    // pos 必须在 [0, OrgSize] 范围内
    if (CurPos == pos)
        return;  // 位置未变，无需操作

    // 二分搜索确定 pos 所在的段编号
    tjs_int st = 0;                          // 搜索下界
    tjs_int et = (tjs_int)Segments->size();  // 搜索上界（不含）
    tjs_int seg_num;

    while (true) {
        if (et - st <= 1) {
            // 搜索范围缩小到只剩一个元素
            seg_num = st;
            break;
        }
        tjs_int m = st + (et - st) / 2;  // 中点（避免溢出的写法）
        if (Segments->operator[](m).Offset > pos)
            et = m;   // 目标在左半部分
        else
            st = m;   // 目标在右半部分（或就是 m）
    }

    // 更新状态机
    CurSegmentNum = seg_num;
    CurSegment = &(Segments->operator[](CurSegmentNum));
    SegmentOpened = false;  // 标记段需要重新打开

    // 计算段内偏移和剩余字节
    SegmentPos = pos - CurSegment->Offset;
    SegmentRemain = CurSegment->OrgSize - SegmentPos;
    CurPos = pos;
}
```

**二分搜索的工作原理**：

假设一个文件有 4 个段，逻辑偏移分别为 0、1000、2500、4000。现在要 seek 到位置 2700：

```
段编号:  [0]       [1]       [2]       [3]
Offset:  0         1000      2500      4000
         ├─────────┼─────────┼─────────┼─────────┤
                                 ↑
                              pos=2700

第 1 轮: st=0, et=4, m=2 → Offset[2]=2500 ≤ 2700 → st=2
第 2 轮: st=2, et=4, m=3 → Offset[3]=4000 > 2700 → et=3
第 3 轮: et-st=1 → seg_num=2

结果: 段 2, SegmentPos = 2700 - 2500 = 200, SegmentRemain = (4000-2500) - 200
```

### 4.3 OpenNextSegment()：切换到下一段

当当前段读完后，需要切换到下一个段继续读取：

> **源码位置：** `cpp/core/base/XP3Archive.cpp` 第 930-942 行

```cpp
bool tTVPXP3ArchiveStream::OpenNextSegment() {
    // 检查是否已是最后一段
    if (CurSegmentNum == (tjs_int)(Segments->size() - 1))
        return false;  // 没有更多段了，返回 false

    CurSegmentNum++;   // 前进到下一段
    CurSegment = &(Segments->operator[](CurSegmentNum));
    SegmentOpened = false;   // 新段需要打开
    SegmentPos = 0;          // 从段开头开始
    SegmentRemain = CurSegment->OrgSize;  // 剩余 = 整个段
    CurPos = CurSegment->Offset;          // 更新逻辑位置
    EnsureSegment();         // 立即加载新段的数据
    return true;
}
```

### 4.4 Read()：核心读取函数

`Read()` 是对外暴露的读取接口。它需要处理以下复杂情况：
- 请求的数据可能跨越多个段
- 每个段可能是压缩的也可能是未压缩的
- 读取后需要应用提取过滤器（如果已注册）

> **源码位置：** `cpp/core/base/XP3Archive.cpp` 第 973-1015 行

```cpp
tjs_uint tTVPXP3ArchiveStream::Read(void *buffer, tjs_uint read_size) {
    EnsureSegment();  // 确保当前段已就绪

    tjs_uint write_size = 0;  // 已写入调用者缓冲区的字节数

    while (read_size) {
        // 如果当前段已读完，切换到下一段
        while (SegmentRemain == 0) {
            if (!OpenNextSegment())
                return write_size;  // 没有更多段 → 返回已读字节数
        }

        // 计算本次从当前段能读取的字节数
        tjs_uint one_size =
            read_size > SegmentRemain
                ? (tjs_uint)SegmentRemain  // 只能读到段尾
                : read_size;               // 可以满足全部请求

        if (CurSegment->IsCompressed) {
            // 压缩段：从已解压的内存缓冲区复制数据
            memcpy(
                (tjs_uint8 *)buffer + write_size,      // 目标：调用者缓冲区
                SegmentData->GetData() + (tjs_uint)SegmentPos,  // 源：解压数据
                one_size
            );
        } else {
            // 未压缩段：直接从底层文件流读取
            Stream->ReadBuffer(
                (tjs_uint8 *)buffer + write_size,
                one_size
            );
        }

        // 应用提取过滤器（用于解密等后处理）
        if (TVPXP3ArchiveExtractionFilter) {
            tTVPXP3ExtractionFilterInfo info(
                CurPos,                                // 当前逻辑位置
                (tjs_uint8 *)buffer + write_size,      // 刚读取的数据
                one_size,                              // 数据大小
                Owner->GetFileHash(StorageIndex),      // 文件哈希（adlr 块）
                Owner->GetName(StorageIndex)           // 文件名
            );
            TVPXP3ArchiveExtractionFilter(
                (tTVPXP3ExtractionFilterInfo *)&info,
                &FilterContext
            );
        }

        // 更新状态
        SegmentPos += one_size;
        CurPos += one_size;
        SegmentRemain -= one_size;
        read_size -= one_size;
        write_size += one_size;
    }

    return write_size;
}
```

### 4.5 跨段读取示例

假设文件有 3 个段，每段 1000 字节。调用者请求读取 2500 字节：

```
调用: Read(buffer, 2500)

┌────────────┐┌────────────┐┌────────────┐
│  段 0      ││  段 1      ││  段 2      │
│  1000 B    ││  1000 B    ││  1000 B    │
└────────────┘└────────────┘└────────────┘

第 1 轮: 从段 0 读取 1000 字节（one_size=1000, SegmentRemain=0）
         → buffer[0..999] = 段 0 数据
第 2 轮: SegmentRemain==0 → OpenNextSegment() → 切换到段 1
         从段 1 读取 1000 字节（one_size=1000, SegmentRemain=0）
         → buffer[1000..1999] = 段 1 数据
第 3 轮: SegmentRemain==0 → OpenNextSegment() → 切换到段 2
         从段 2 读取 500 字节（one_size=500, SegmentRemain=500）
         → buffer[2000..2499] = 段 2 前 500 字节
循环结束: read_size == 0, 返回 write_size = 2500
```

### 4.6 Seek()：统一的定位接口

`Seek()` 是 `tTJSBinaryStream` 基类定义的虚函数，提供标准的文件定位操作：

> **源码位置：** `cpp/core/base/XP3Archive.cpp` 第 945-970 行

```cpp
tjs_uint64 tTVPXP3ArchiveStream::Seek(tjs_int64 offset, tjs_int whence) {
    tjs_int64 newpos;
    switch (whence) {
        case TJS_BS_SEEK_SET:  // 从文件开头计算
            newpos = offset;
            break;
        case TJS_BS_SEEK_CUR:  // 从当前位置计算
            newpos = offset + CurPos;
            break;
        case TJS_BS_SEEK_END:  // 从文件末尾计算
            newpos = offset + OrgSize;
            break;
    }
    // 边界检查：newpos 必须在 [0, OrgSize] 范围内
    if (newpos >= 0 && newpos <= static_cast<tjs_int64>(OrgSize)) {
        SeekToPosition(newpos);  // 执行二分搜索定位
    }
    return CurPos;
}
```

三种 seek 模式与标准 C 的 `fseek()` 完全对应：
- `TJS_BS_SEEK_SET` ↔ `SEEK_SET`：绝对定位
- `TJS_BS_SEEK_CUR` ↔ `SEEK_CUR`：相对当前位置
`TVP_COMPACT_LEVEL_DEACTIVATE` 是 KrKr2 事件系统中定义的紧凑级别（Compact Level）。当系统内存紧张或应用切换到后台时，事件系统会广播紧凑事件，各子系统根据级别决定释放哪些资源。这对于 Android 等移动平台尤为重要——后台应用随时可能被系统杀死，提前释放内存可以延长应用的存活时间。

## 6. 归档句柄缓存

除了段数据缓存，XP3 系统还有一个独立的**归档句柄缓存**（Archive Handle Cache）。它缓存的不是解压后的数据，而是打开的归档文件流对象（`tTJSBinaryStream*`）。

### 6.1 为什么需要句柄缓存？

考虑以下场景：一个 XP3 归档中包含 1000 个文件。游戏运行时，脚本引擎频繁打开和关闭这些文件：

```
// 游戏运行时的典型文件访问模式
打开 bgm/title.ogg → 读取 → 关闭
打开 image/bg01.png → 读取 → 关闭
打开 scenario/main.ks → 读取 → 关闭
打开 image/chara01.png → 读取 → 关闭
// ... 每秒可能打开/关闭数十次
```

如果每次打开文件都重新打开归档文件（调用操作系统的文件 open 系统调用），会产生显著的性能开销——文件 open 是一个需要内核态切换的**系统调用**（System Call），涉及路径解析、权限检查、文件句柄分配等步骤。

归档句柄缓存的解决方案是：当一个 `tTVPXP3ArchiveStream` 被销毁时，不关闭底层的归档文件流，而是将其放入缓存池。下次需要打开同一归档时，直接从池中取出已有的流对象，跳过文件 open 操作。

### 6.2 缓存池结构

> **源码位置：** `cpp/core/base/XP3Archive.cpp` 第 45-56 行

```cpp
// 最大缓存句柄数（池的容量）
#define TVP_MAX_ARCHIVE_HANDLE_CACHE 8

// 全局年龄计数器（每次归还操作 +1）
static tjs_uint TVPArchiveHandleCacheAge = 0;

// 缓存条目结构
struct tTVPArchiveHandleCacheItem {
    void *Pointer;           // 归档对象指针（作为匹配键）
    tTJSBinaryStream *Stream; // 缓存的文件流对象
    tjs_uint Age;            // 放入缓存时的年龄（用于 LRU 淘汰）
};

// 缓存池数组（固定大小 8）
static tTVPArchiveHandleCacheItem *TVPArchiveHandleCachePool = nullptr;

// 初始化和关闭标志
static bool TVPArchiveHandleCacheInit = false;
static bool TVPArchiveHandleCacheShutdown = false;

// 线程安全保护
static tTJSCriticalSection TVPArchiveHandleCacheCS;
```

缓存池使用固定大小的数组（8 个槽位）而非动态容器，原因是：
1. 8 个槽位对于线性搜索来说足够快
2. 避免动态分配的额外开销
3. 现实中同时活跃的归档文件通常不超过 8 个

### 6.3 获取缓存句柄

> **源码位置：** `cpp/core/base/XP3Archive.cpp` 第 59-95 行

```cpp
tTJSBinaryStream *TVPGetCachedArchiveHandle(void *pointer, const ttstr &name) {
    // 如果缓存系统已关闭（程序退出阶段），直接创建新流
    if (TVPArchiveHandleCacheShutdown)
        return TVPCreateStream(name);

    tTJSCriticalSectionHolder cs_holder(TVPArchiveHandleCacheCS);

    // 延迟初始化：首次调用时分配缓存池
    if (!TVPArchiveHandleCacheInit) {
        TVPArchiveHandleCachePool =
            new tTVPArchiveHandleCacheItem[TVP_MAX_ARCHIVE_HANDLE_CACHE];
        for (tjs_int i = 0; i < TVP_MAX_ARCHIVE_HANDLE_CACHE; i++) {
            TVPArchiveHandleCachePool[i].Pointer = nullptr;
            TVPArchiveHandleCachePool[i].Stream = nullptr;
            TVPArchiveHandleCachePool[i].Age = 0;
        }
        TVPArchiveHandleCacheInit = true;
    }

    // 线性搜索：查找匹配的缓存条目
    for (tjs_int i = 0; i < TVP_MAX_ARCHIVE_HANDLE_CACHE; i++) {
        tTVPArchiveHandleCacheItem *item = TVPArchiveHandleCachePool + i;
        if (item->Stream && item->Pointer == pointer) {
            // 命中！取出流对象
            tTJSBinaryStream *stream = item->Stream;
            item->Stream = nullptr;  // 标记槽位为空
            return stream;
        }
    }

    // 缓存未命中 → 创建新流
    return TVPCreateStream(name);
}
```

### 6.4 归还句柄到缓存

> **源码位置：** `cpp/core/base/XP3Archive.cpp` 第 97-139 行

```cpp
void TVPReleaseCachedArchiveHandle(void *pointer, tTJSBinaryStream *stream) {
    if (TVPArchiveHandleCacheShutdown) return;
    if (!TVPArchiveHandleCacheInit) return;

    tTJSCriticalSectionHolder cs_holder(TVPArchiveHandleCacheCS);

    // 寻找空闲槽位 + 记录最老条目
    tjs_uint oldest_age = 0;
    tjs_int oldest = 0;

    for (tjs_int i = 0; i < TVP_MAX_ARCHIVE_HANDLE_CACHE; i++) {
        tTVPArchiveHandleCacheItem *item = TVPArchiveHandleCachePool + i;
        if (item->Stream == nullptr) {
            // 找到空槽 → 直接放入
            item->Pointer = pointer;
            item->Stream = stream;
            item->Age = ++TVPArchiveHandleCacheAge;
            return;
        }

        // 记录最老的条目（年龄最小的）
        if (i == 0 || oldest_age > item->Age) {
            oldest_age = item->Age;
            oldest = i;
        }
    }

    // 没有空槽 → 淘汰最老的条目
    tTVPArchiveHandleCacheItem *oldest_item =
        TVPArchiveHandleCachePool + oldest;
    delete oldest_item->Stream;           // 关闭旧流
    oldest_item->Pointer = pointer;       // 替换为新条目
    oldest_item->Stream = stream;
    oldest_item->Age = ++TVPArchiveHandleCacheAge;
}
```

句柄缓存的淘汰策略比段缓存简单得多——它使用**年龄计数器**（Age Counter）而非链表来实现 LRU。每次归还操作都递增全局年龄，淘汰时选择年龄最小的条目。

### 6.5 两层缓存的协作关系

段缓存和句柄缓存构成了一个**两层缓存架构**：


## 7. 动手实践：实现简化版分段流

下面我们实现一个简化版的分段压缩流，帮助理解核心逻辑。这个实现省略了缓存和过滤器，但保留了分段读取和 seek 的核心机制：

```cpp
#include <cstdint>
#include <cstring>
#include <vector>
#include <stdexcept>
#include <iostream>
#include <sstream>
#include <zlib.h>

// 段描述符
struct Segment {
    uint64_t start;        // 归档中的起始位置
    uint64_t offset;       // 逻辑偏移
    uint64_t orgSize;      // 原始大小
    uint64_t arcSize;      // 归档大小
    bool isCompressed;     // 是否压缩
};

// 简化版分段流（从内存缓冲区读取，模拟归档文件）
class SegmentedStream {
    const uint8_t *archiveData_;   // 模拟的归档数据
    size_t archiveSize_;           // 归档数据总大小
    std::vector<Segment> segments_; // 段描述符数组
    uint64_t orgSize_;             // 文件原始总大小

    // 状态机
    int curSegNum_;           // 当前段编号
    uint64_t curPos_;         // 当前逻辑位置
    uint64_t segPos_;         // 段内偏移
    uint64_t segRemain_;      // 段内剩余

    // 解压缓冲区
    std::vector<uint8_t> decompBuf_;

public:
    SegmentedStream(const uint8_t *data, size_t size,
                    const std::vector<Segment> &segs, uint64_t orgSize)
        : archiveData_(data), archiveSize_(size),
          segments_(segs), orgSize_(orgSize),
          curSegNum_(0), curPos_(0), segPos_(0) {
        if (!segments_.empty()) {
            segRemain_ = segments_[0].orgSize;
        }
    }

    // 二分搜索定位段
    void seekToPosition(uint64_t pos) {
        if (pos == curPos_) return;
        if (pos > orgSize_) pos = orgSize_;

        // 二分搜索
        int st = 0, et = static_cast<int>(segments_.size());
        while (et - st > 1) {
            int m = st + (et - st) / 2;
            if (segments_[m].offset > pos)
                et = m;
            else
                st = m;
        }

        curSegNum_ = st;
        segPos_ = pos - segments_[st].offset;
        segRemain_ = segments_[st].orgSize - segPos_;
        curPos_ = pos;
        decompBuf_.clear();  // 强制重新加载段数据
    }

    // 确保当前段数据已就绪
    void ensureSegment() {
        const Segment &seg = segments_[curSegNum_];
        if (seg.isCompressed && decompBuf_.empty()) {
            // 需要解压
            decompBuf_.resize(seg.orgSize);
            uLongf destLen = static_cast<uLongf>(seg.orgSize);
            int ret = uncompress(
                decompBuf_.data(), &destLen,
                archiveData_ + seg.start,
                static_cast<uLong>(seg.arcSize)
            );
            if (ret != Z_OK) {
                throw std::runtime_error("zlib 解压失败, 错误码: "
                                         + std::to_string(ret));
            }
        }
    }

    // 读取数据（可能跨段）
    size_t read(uint8_t *buffer, size_t readSize) {
        size_t written = 0;

        while (readSize > 0) {
            // 当前段已读完？切换到下一段
            while (segRemain_ == 0) {
                if (curSegNum_ >= static_cast<int>(segments_.size()) - 1)
                    return written;  // 没有更多段
                curSegNum_++;
                segPos_ = 0;
                segRemain_ = segments_[curSegNum_].orgSize;
                curPos_ = segments_[curSegNum_].offset;
                decompBuf_.clear();  // 新段需要重新加载
            }

            ensureSegment();

            // 本次可读字节数
            size_t oneSize = (readSize > segRemain_)
                ? static_cast<size_t>(segRemain_) : readSize;

            const Segment &seg = segments_[curSegNum_];
            if (seg.isCompressed) {
                // 从解压缓冲区复制
                std::memcpy(buffer + written,
                            decompBuf_.data() + segPos_, oneSize);
            } else {
                // 直接从归档数据复制
                std::memcpy(buffer + written,
                            archiveData_ + seg.start + segPos_, oneSize);
            }

            segPos_ += oneSize;
            curPos_ += oneSize;
            segRemain_ -= oneSize;
            readSize -= oneSize;
            written += oneSize;
        }
        return written;
    }

    uint64_t getPosition() const { return curPos_; }
    uint64_t getSize() const { return orgSize_; }
};
```

## 9. 常见错误与排查

### 错误 1: 解压失败（uncompress 返回非 Z_OK）

**症状**：`TVPThrowExceptionMessage(TVPUncompressionFailed)` 异常

**可能原因**：
1. XP3 文件损坏（归档或网络传输时数据被截断/篡改）
2. 段描述符中的 `ArcSize` 或 `OrgSize` 不正确
3. 归档被加密但未正确设置提取过滤器

**排查步骤**：
```cpp
// 在 SetData() 中添加诊断信息
void SetData(unsigned long outsize, tTJSBinaryStream *instream,
             unsigned long insize) {
    tjs_uint8 *indata = new tjs_uint8[insize];
    try {
        instream->Read(indata, insize);
        Data = new tjs_uint8[outsize];
        unsigned long destlen = outsize;
        int result = uncompress(
            (unsigned char *)Data, &outsize,
            (unsigned char *)indata, insize
        );

        // 添加诊断输出
        if (result != Z_OK) {
            std::cerr << "[段缓存] 解压失败:"
                      << " result=" << result      // Z_DATA_ERROR=-3 表示数据损坏
                      << " insize=" << insize
                      << " outsize=" << outsize
                      << " destlen=" << destlen
                      << std::endl;
        }

        if (result != Z_OK || destlen != outsize)
            TVPThrowExceptionMessage(TVPUncompressionFailed);
        Size = outsize;
    } catch (...) {
        delete[] indata;
        throw;
    }
    delete[] indata;
}
```

### 错误 2: 段缓存导致内存持续增长

**症状**：程序运行一段时间后内存占用不断增加，即使切换场景也不释放

**排查方法**：
```cpp
// 添加缓存统计函数
void TVPDumpSegmentCacheStats() {
    tTJSCriticalSectionHolder cs_holder(TVPSegmentCacheCS);
    std::cout << "[段缓存统计]"
              << " 总大小=" << TVPSegmentCacheTotalBytes << " 字节"
              << " 上限=" << TVPSegmentCacheLimit << " 字节"
              << " 条目数=" << TVPSegmentCache.GetCount()
              << std::endl;
}
```

**常见原因**：`TVPSegmentCacheLimit` 被错误设置为过大的值，或 Compact 事件未正确注册。

### 错误 3: seek 后读取数据错乱

**症状**：`Seek()` 到某个位置后，`Read()` 返回的数据与预期不符

**排查要点**：
1. 检查 `SeekToPosition()` 的二分搜索是否正确定位了段
2. 确认 `SegmentPos` 计算是否正确（`pos - CurSegment->Offset`）
3. 对于未压缩段，确认底层流的 `SetPosition()` 是否调用了正确的位置

## 10. 本节小结

- XP3 采用**分段压缩**设计，将文件切成多个独立段分别压缩，在压缩比和随机访问性能之间取得平衡
- 每个段由 `tTVPXP3ArchiveSegment` 描述，包含归档位置（Start）、逻辑偏移（Offset）、原始大小（OrgSize）、归档大小（ArcSize）和压缩标记（IsCompressed）五个关键字段
- `tTVPXP3ArchiveStream` 是一个状态机，维护着当前段编号、段内偏移、剩余字节等状态，对外表现为普通的可 seek 字节流
- **段缓存**使用哈希表 + LRU 淘汰策略，将已解压的段数据缓存在内存中，默认总容量 1MB
- **归档句柄缓存**使用固定大小池（8 槽位）+ 年龄计数器淘汰，复用打开的归档文件流
- 两层缓存协作：句柄缓存避免文件重复打开，段缓存避免数据重复解压
- 所有缓存操作通过临界区保证线程安全，使用 RAII 模式管理锁

### 题目 1：段缓存容量计算

假设 `TVPSegmentCacheLimit` 设置为 2MB（2097152 字节）。当前缓存中已有以下段数据：

| 段 | 解压后大小 | 放入缓存的顺序 |
|----|-----------|----------------|
| A  | 512 KB    | 第 1 个        |
| B  | 768 KB    | 第 2 个        |
| C  | 256 KB    | 第 3 个        |

现在要放入一个新的段 D（解压后大小 800 KB）。请问：
1. 放入 D 之前缓存总大小是多少？
2. 放入 D 之后是否会触发淘汰？如果是，哪些段会被淘汰？
3. 淘汰后缓存中还有哪些段？总大小是多少？

<details>
<summary>查看答案</summary>

1. **放入前总大小**：512 + 768 + 256 = 1536 KB = 1,572,864 字节

2. **放入 D（800 KB）后总大小**：1536 + 800 = 2336 KB = 2,392,064 字节 > 2,097,152 字节（2MB）
   → **会触发淘汰**

3. **淘汰过程**：
   - 缓存使用 LRU，最久未访问的在链表尾部
   - 假设 A、B、C 依次放入且中间没有被访问过，则 A 在尾部
   - 淘汰 A（512 KB）→ 总大小 = 2336 - 512 = 1824 KB = 1,867,776 字节
   - 1,867,776 < 2,097,152 → 停止淘汰
   - 缓存中剩余：**B（768 KB）、C（256 KB）、D（800 KB）**
   - 总大小：**1824 KB = 1,867,776 字节**

注意：如果在放入 D 之前，段 A 曾被再次访问（通过 `FindAndTouchWithHash`），它会被移到链表头部。此时尾部会变成 B，淘汰的就是 B 而非 A。

</details>

### 题目 3：实现带缓存的分段流

在第 7 节的 `SegmentedStream` 基础上，添加一个简单的段缓存功能。要求：
1. 使用 `std::unordered_map` 实现缓存
2. 缓存键为段编号（`int`）
3. 缓存总大小上限为 `maxCacheSize` 字节
4. 超过上限时，淘汰最早放入的段（FIFO 策略即可，不要求 LRU）

<details>
<summary>查看答案</summary>

```cpp
#include <cstdint>
#include <cstring>
#include <vector>
#include <unordered_map>
#include <queue>
#include <stdexcept>
#include <iostream>
#include <zlib.h>

struct Segment {
    uint64_t start;
    uint64_t offset;
    uint64_t orgSize;
    uint64_t arcSize;
    bool isCompressed;
};

// 带缓存的分段流
class CachedSegmentedStream {
    const uint8_t *archiveData_;
    size_t archiveSize_;
    std::vector<Segment> segments_;
    uint64_t orgSize_;

    int curSegNum_;
    uint64_t curPos_;
    uint64_t segPos_;
    uint64_t segRemain_;

    // === 缓存相关 ===
    struct CacheEntry {
        std::vector<uint8_t> data;  // 解压后的数据
    };
    std::unordered_map<int, CacheEntry> cache_;  // 段编号 → 解压数据
    std::queue<int> cacheOrder_;                  // FIFO 淘汰顺序
    size_t cacheBytes_ = 0;                       // 当前缓存总大小
    size_t maxCacheSize_;                          // 缓存上限

    // 当前活跃的段数据指针（指向缓存或临时缓冲区）
    const uint8_t *activeData_ = nullptr;
    std::vector<uint8_t> tempBuf_;  // 不进缓存时的临时缓冲区

    void evictIfNeeded() {
        while (cacheBytes_ > maxCacheSize_ && !cacheOrder_.empty()) {
            int oldest = cacheOrder_.front();
            cacheOrder_.pop();
            auto it = cache_.find(oldest);
            if (it != cache_.end()) {
                cacheBytes_ -= it->second.data.size();
                cache_.erase(it);
            }
        }
    }

    void loadSegment() {
        const Segment &seg = segments_[curSegNum_];
        if (!seg.isCompressed) {
            // 未压缩段：直接指向归档数据
            activeData_ = archiveData_ + seg.start;
            return;
        }

        // 检查缓存
        auto it = cache_.find(curSegNum_);
        if (it != cache_.end()) {
            // 缓存命中
            activeData_ = it->second.data.data();
            return;
        }

        // 缓存未命中：解压
        std::vector<uint8_t> decompressed(seg.orgSize);
        uLongf destLen = static_cast<uLongf>(seg.orgSize);
        int ret = uncompress(
            decompressed.data(), &destLen,
            archiveData_ + seg.start,
            static_cast<uLong>(seg.arcSize)
        );
        if (ret != Z_OK)
            throw std::runtime_error("解压失败");

        // 决定是否放入缓存
        if (seg.orgSize <= maxCacheSize_) {
            // 放入缓存
            cacheBytes_ += decompressed.size();
            cache_[curSegNum_] = {std::move(decompressed)};
            cacheOrder_.push(curSegNum_);
            evictIfNeeded();
            activeData_ = cache_[curSegNum_].data.data();
        } else {
            // 段太大，不缓存
            tempBuf_ = std::move(decompressed);
            activeData_ = tempBuf_.data();
        }
    }

public:
    CachedSegmentedStream(const uint8_t *data, size_t size,
                          const std::vector<Segment> &segs,
                          uint64_t orgSize, size_t maxCache = 1024 * 1024)
        : archiveData_(data), archiveSize_(size),
          segments_(segs), orgSize_(orgSize),
          curSegNum_(0), curPos_(0), segPos_(0),
          maxCacheSize_(maxCache) {
        if (!segments_.empty())
            segRemain_ = segments_[0].orgSize;
    }

    void seekToPosition(uint64_t pos) {
        if (pos == curPos_) return;
        int st = 0, et = static_cast<int>(segments_.size());
        while (et - st > 1) {
            int m = st + (et - st) / 2;
            if (segments_[m].offset > pos) et = m;
            else st = m;
        }
        curSegNum_ = st;
        segPos_ = pos - segments_[st].offset;
        segRemain_ = segments_[st].orgSize - segPos_;
        curPos_ = pos;
        activeData_ = nullptr;  // 需要重新加载
    }

    size_t read(uint8_t *buffer, size_t readSize) {
        size_t written = 0;
        while (readSize > 0) {
            while (segRemain_ == 0) {
                if (curSegNum_ >= static_cast<int>(segments_.size()) - 1)
                    return written;
                curSegNum_++;
                segPos_ = 0;
                segRemain_ = segments_[curSegNum_].orgSize;
                curPos_ = segments_[curSegNum_].offset;
                activeData_ = nullptr;
            }
            if (!activeData_) loadSegment();
            size_t oneSize = std::min(readSize,
                                      static_cast<size_t>(segRemain_));
            std::memcpy(buffer + written,
                        activeData_ + segPos_, oneSize);
            segPos_ += oneSize;
            curPos_ += oneSize;
            segRemain_ -= oneSize;
            readSize -= oneSize;
            written += oneSize;
        }
        return written;
    }

    // 获取缓存统计信息
    void printCacheStats() const {
        std::cout << "[缓存统计] 条目数=" << cache_.size()
                  << " 总大小=" << cacheBytes_
                  << " 上限=" << maxCacheSize_
                  << std::endl;
    }
};

int main() {
    // 创建测试数据
    std::string raw1(500, 'A');  // 500 字节全 A
    std::string raw2(500, 'B');  // 500 字节全 B

    // 压缩两个段
    std::vector<uint8_t> comp1(compressBound(raw1.size()));
    uLongf len1 = comp1.size();
    compress(comp1.data(), &len1,
             reinterpret_cast<const uint8_t*>(raw1.data()), raw1.size());
    comp1.resize(len1);

    std::vector<uint8_t> comp2(compressBound(raw2.size()));
    uLongf len2 = comp2.size();
    compress(comp2.data(), &len2,
             reinterpret_cast<const uint8_t*>(raw2.data()), raw2.size());
    comp2.resize(len2);

    // 构建归档
    std::vector<uint8_t> archive;
    size_t off1 = 0;
    archive.insert(archive.end(), comp1.begin(), comp1.end());
    size_t off2 = archive.size();
    archive.insert(archive.end(), comp2.begin(), comp2.end());

    std::vector<Segment> segs = {
        {off1, 0,   500, len1, true},
        {off2, 500, 500, len2, true}
    };

    // 缓存上限设为 600 字节（只能缓存一个段）
    CachedSegmentedStream stream(archive.data(), archive.size(),
                                 segs, 1000, 600);

    uint8_t buf[100];

    // 读段 0
    stream.read(buf, 10);
    stream.printCacheStats();
    // 输出: [缓存统计] 条目数=1 总大小=500 上限=600

    // 读段 1 → 缓存超限，淘汰段 0
    stream.seekToPosition(500);
    stream.read(buf, 10);
    stream.printCacheStats();
    // 输出: [缓存统计] 条目数=1 总大小=500 上限=600

    // 回到段 0 → 缓存未命中，重新解压
    stream.seekToPosition(0);
    stream.read(buf, 10);
    stream.printCacheStats();
    // 输出: [缓存统计] 条目数=1 总大小=500 上限=600

    std::cout << "所有测试通过!" << std::endl;
    return 0;
}
```

## 5. 段缓存系统详解

段缓存（Segment Cache）是 XP3 归档系统的关键性能优化。它将已解压的段数据缓存在内存中，当同一段被再次访问时可以直接使用，避免重复的磁盘 IO 和 zlib 解压开销。

### 5.1 缓存容量配置

> **源码位置：** `cpp/core/base/XP3Archive.cpp` 第 638-640 行

```cpp
// 单个段的最大可缓存大小（超过此大小的段不会被放入缓存）
#define TVP_SEGCACHE_ONE_LIMIT (1024 * 1024)   // 1MB

// 缓存总容量上限（所有缓存段的解压后数据总和不超过此值）
#define TVP_SEGCACHE_TOTAL_LIMIT (1024 * 1024)  // 1MB

// 运行时可调整的缓存上限变量
tjs_uint TVPSegmentCacheLimit = TVP_SEGCACHE_TOTAL_LIMIT;
```

两个限制条件：
1. **单段限制**（`TVP_SEGCACHE_ONE_LIMIT = 1MB`）：解压后大小 ≥ 1MB 的段不会被放入缓存。这些大段通常是大型资源文件（如背景音乐），缓存它们的性价比很低——它们可能只被顺序读取一次就不再访问。
2. **总量限制**（`TVPSegmentCacheLimit = 1MB`）：所有缓存段的解压后数据总和不超过 1MB。超过时触发 LRU 淘汰。

> **设计思考**：1MB 的总缓存限制在现代标准下显得很保守。这是因为 KiriKiri2 最初设计于 2000 年代初，当时 PC 的典型内存为 256MB-512MB。在现代设备上，可以通过修改 `TVPSegmentCacheLimit` 变量来增大缓存（例如设为 16MB 或 32MB），以获得更好的性能。

### 5.2 缓存键设计

段缓存使用**三元组**作为缓存键（Cache Key），唯一标识一个段：

> **源码位置：** `cpp/core/base/XP3Archive.cpp` 第 643-652 行

```cpp
struct tTVPSegmentCacheSearchData {
    ttstr Name;             // 归档文件名（如 "data.xp3"）
    tjs_int StorageIndex;   // 文件在归档中的索引号
    tjs_int SegmentIndex;   // 段在文件中的编号

    // 相等性比较（用于哈希表冲突解决时的精确匹配）
    bool operator==(const tTVPSegmentCacheSearchData &rhs) const {
        return Name == rhs.Name &&
               StorageIndex == rhs.StorageIndex &&
               SegmentIndex == rhs.SegmentIndex;
    }
};
```

**为什么需要三个字段？**

- **Name（归档名）**：因为系统可能同时打开多个 XP3 归档（如 `data.xp3`、`patch.xp3`），不同归档中的段必须区分开
- **StorageIndex（文件索引）**：同一归档内的不同文件有各自的段
- **SegmentIndex（段编号）**：同一文件的不同段

### 5.3 哈希函数

缓存使用哈希表（Hash Table）实现，需要一个将三元组映射为 32 位整数的哈希函数：

> **源码位置：** `cpp/core/base/XP3Archive.cpp` 第 655-664 行

```cpp
class tTVPSegmentCacheSearchHashFunc {
public:
    static tjs_uint32 Make(const tTVPSegmentCacheSearchData &val) {
        // 先对归档名计算字符串哈希
        tjs_uint32 v = tTJSHashFunc<ttstr>::Make(val.Name);

        // 异或文件索引
        v ^= val.StorageIndex;

        // 异或段编号（左移 2 位以减少低位冲突）
        v ^= (val.SegmentIndex << 2);

        return v;
    }
};
```

哈希函数的设计要点：
1. 以归档名的字符串哈希为基础——这通常已经是一个分布良好的值
2. 通过 XOR（异或）运算混入文件索引和段编号
3. 段编号左移 2 位（`<< 2`）是为了减少连续段编号在低位上的冲突。例如，段 0 和段 1 的差异如果不左移只在最低位，左移后差异扩展到 bit2 和 bit3

### 5.4 段数据对象

缓存中存储的是 `tTVPSegmentData` 对象，它封装了解压后的数据缓冲区，并使用引用计数管理生命周期：

> **源码位置：** `cpp/core/base/XP3Archive.cpp` 第 667-718 行

```cpp
class tTVPSegmentData {
    tjs_int RefCount;    // 引用计数
    tjs_uint Size;       // 解压后数据大小
    tjs_uint8 *Data;     // 解压后数据缓冲区

public:
    tTVPSegmentData() {
        RefCount = 1;     // 初始引用计数为 1
        Size = 0;
        Data = nullptr;
    }

    ~tTVPSegmentData() {
        if (Data) delete[] Data;
    }

    // 从流中读取压缩数据并解压
    void SetData(unsigned long outsize,        // 期望的解压后大小
                 tTJSBinaryStream *instream,   // 输入流（已定位到压缩数据起始处）
                 unsigned long insize) {       // 压缩数据大小
        // 分配临时缓冲区读取压缩数据
        tjs_uint8 *indata = new tjs_uint8[insize];
        try {
            instream->Read(indata, insize);   // 读取压缩数据

            // 分配输出缓冲区并解压
            Data = new tjs_uint8[outsize];
            unsigned long destlen = outsize;
            int result = uncompress(
                (unsigned char *)Data, &outsize,    // 输出
                (unsigned char *)indata, insize     // 输入
            );
            // 验证解压结果
            if (result != Z_OK || destlen != outsize)
                TVPThrowExceptionMessage(TVPUncompressionFailed);
            Size = outsize;
        } catch (...) {
            delete[] indata;  // 异常安全：确保释放临时缓冲区
            throw;
        }
        delete[] indata;
    }

    const tjs_uint8 *GetData() const { return Data; }
    tjs_uint GetSize() const { return Size; }

    void AddRef() { RefCount++; }

    void Release() {
        if (RefCount == 1) {
            delete this;       // 最后一个引用 → 销毁对象
        } else {
            RefCount--;        // 还有其他引用 → 减少计数
        }
    }
};
```

**引用计数的使用场景**：

```
场景 1: 段被放入缓存后，流对象也持有引用
  → RefCount = 2（缓存持有 1 + 流持有 1）
  → 流释放后 RefCount = 1（缓存仍持有）
  → 下次命中缓存时 RefCount = 2

场景 2: 段太大不进缓存，只有流持有引用
  → RefCount = 1（只有流持有）
  → 流释放后 RefCount = 0 → 对象销毁
```

### 5.5 LRU 淘汰机制

当缓存总大小超过限制时，需要淘汰旧条目。KrKr2 使用 `tTJSHashTable` 的内置 LRU 特性实现：

> **源码位置：** `cpp/core/base/XP3Archive.cpp` 第 732-747 行

```cpp
static void TVPCheckSegmentCacheLimit() {
    tTJSCriticalSectionHolder cs_holder(TVPSegmentCacheCS);

    // 循环淘汰，直到总大小降到限制以下
    while (TVPSegmentCacheTotalBytes > TVPSegmentCacheLimit) {
        // 获取最久未访问的条目（链表尾部）
        tTVPSegmentCache::tIterator i;
        i = TVPSegmentCache.GetLast();

        if (!i.IsNull()) {
            // 计算该条目占用的内存
            tjs_uint size = i.GetValue().GetObjectNoAddRef()->GetSize();
            TVPSegmentCacheTotalBytes -= size;

            // 从哈希表中移除最后一个条目
            TVPSegmentCache.ChopLast(1);
        } else {
            break;  // 缓存已空
        }
    }
}
```

`tTJSHashTable` 内部维护了一个**访问顺序链表**（Access-Order Linked List）。每次查找命中时，被访问的条目会被移到链表头部（通过 `FindAndTouchWithHash`）。因此链表尾部的条目就是最久未被访问的——这正是 LRU 策略要淘汰的目标。

LRU 淘汰的完整流程：

```
1. 新段数据放入缓存
     │
     ▼
2. TVPSegmentCacheTotalBytes += 新段大小
     │
     ▼
3. 调用 TVPCheckSegmentCacheLimit()
     │
     ▼
4. while (总大小 > 限制)
     │    ├── 取链表尾部条目（最久未访问）
     │    ├── 总大小 -= 该条目大小
     │    └── ChopLast(1) 移除该条目
     ▼
5. 缓存大小恢复到限制以内
```

### 5.6 线程安全

段缓存通过临界区（Critical Section）保护，确保多线程安全：

```cpp
// 全局临界区对象
static tTJSCriticalSection TVPSegmentCacheCS;

// 所有缓存操作都在临界区内执行
static tTVPSegmentData *
TVPSearchFromSegmentCache(const tTVPSegmentCacheSearchData &sdata,
                          tjs_uint32 hash) {
    tTJSCriticalSectionHolder cs_holder(TVPSegmentCacheCS);  // 加锁
    // ... 查找逻辑 ...
}   // cs_holder 析构时自动解锁（RAII 模式）

static void TVPPushToSegmentCache(const tTVPSegmentCacheSearchData &sdata,
                                  tjs_uint32 hash, tTVPSegmentData *data) {
    // 延迟注册 Compact 事件回调（首次使用时注册）
    if (!TVPClearSegmentCacheCallbackInit) {
        TVPAddCompactEventHook(&TVPClearSegmentCacheCallback);
        TVPClearSegmentCacheCallbackInit = true;
    }

    tTJSCriticalSectionHolder cs_holder(TVPSegmentCacheCS);  // 加锁
    // ... 插入逻辑 + LRU 淘汰 ...
}
```

**RAII**（Resource Acquisition Is Initialization，资源获取即初始化）是一种 C++ 编程范式——通过将资源的获取放在构造函数中、释放放在析构函数中，利用作用域结束时的自动析构来确保资源一定被释放。这里 `tTJSCriticalSectionHolder` 在构造时加锁、析构时解锁，即使中间发生异常也不会死锁。

### 5.7 应用失活时清空缓存

当应用程序切换到后台（失活）时，KrKr2 会清空段缓存以释放内存：

> **源码位置：** `cpp/core/base/XP3Archive.cpp` 第 758-767 行

```cpp
struct tTVPClearSegmentCacheCallback : public tTVPCompactEventCallbackIntf {
    void OnCompact(tjs_int level) override {
        if (level >= TVP_COMPACT_LEVEL_DEACTIVATE) {
            // 应用失活 → 清空段缓存
            TVPClearXP3SegmentCache();
            // 同时释放归档句柄池
            TVPFreeArchiveHandlePool();
        }
    }
} static TVPClearSegmentCacheCallback;
```

`TVP_COMPACT_LEVEL_DEACTIVATE` 是 KrKr2 事件系统中定义的紧凑级别（Compact Level）。当系统内存紧张或应用切换到后台时，事件系统会广播紧凑事件，各子系统根据级别决定释放哪些资源。这对于 Android 等移动平台尤为重要——后台应用随时可能被系统杀死，提前释放内存可以延长应用的存活时间。
