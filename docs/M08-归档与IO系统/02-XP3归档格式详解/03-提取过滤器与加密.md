## 1. 过滤器系统概述

### 1.1 为什么需要过滤器？

XP3 归档格式是 KiriKiri 引擎的标准资源打包格式。由于格式规范是公开的，任何人都可以编写工具提取归档中的文件（图片、音频、脚本等）。这对游戏发行商来说是不可接受的——他们需要保护自己的游戏资源不被轻易提取和盗用。

然而，KiriKiri 的设计者 W.Dee 选择了一种优雅的方案：**不修改归档格式本身**，而是提供一个**可插拔的过滤器接口**。游戏开发者可以编写自己的加密/解密逻辑，以回调函数的形式注册到引擎中。这样：

1. XP3 格式保持不变——工具链（打包器、检查器等）不需要修改
2. 每个游戏可以使用不同的加密方案——一个工具无法通吃所有游戏
3. 加密逻辑可以用 TJS2 脚本编写——无需重新编译引擎

### 1.2 两种过滤器的分工

XP3 归档系统提供了两种过滤器，各自在不同的时间点介入：

```
文件打开流程:
    CreateStreamByIndex(idx)
        │
        ├──① 创建 tTVPXP3ArchiveStream
        │
        ├──② 内容过滤器（Content Filter）介入
        │     │
        │     ├── 返回 0: 使用正常的分段读取模式
        │     └── 返回 1: 全量读取模式（整个文件一次性读入内存）
        │
        └──③ 每次 Read() 时
              │
              └── 提取过滤器（Extraction Filter）介入
                    对刚读取的数据块执行解密操作
```

| 特性 | 提取过滤器 | 内容过滤器 |
|------|-----------|-----------|
| 调用时机 | 每次 Read() 后 | 文件打开时（一次性） |
| 作用 | 逐块解密数据 | 决定是否全量读取 |
| 参数 | 数据块、偏移、文件哈希、文件名 | 文件路径、归档名、文件大小 |
| 返回值 | void（直接修改缓冲区） | int（0=正常, 1=全量读取） |
| 典型用途 | XOR/自定义加密解密 | 需要整体操作的复杂加密 |

## 2. 提取过滤器详解

### 2.1 tTVPXP3ExtractionFilterInfo 结构体

提取过滤器被调用时，引擎会传入一个包含当前数据块完整上下文的结构体：

> **源码位置：** `cpp/core/base/XP3Archive.h` 第 23-41 行

```cpp
#pragma pack(push, 4)  // 强制 4 字节对齐

struct tTVPXP3ExtractionFilterInfo {
    // 结构体自身的大小（用于版本兼容性检查）
    const tjs_uint SizeOfSelf;

    // 当前数据块在解压后文件中的偏移位置
    // 例如：文件总大小 10000 字节，本次读取的是第 5000~5100 字节
    // 则 Offset = 5000
    const tjs_uint64 Offset;

    // 指向待处理数据的缓冲区（可读可写）
    // 过滤器需要直接修改这块内存中的数据来完成解密
    void *Buffer;

    // 缓冲区大小（字节数）
    const tjs_uint BufferSize;

    // 文件哈希值（来自 XP3 索引中的 adlr 块）
    // 通常用作加密密钥的种子——不同文件有不同的哈希，实现"一文件一密"
    const tjs_uint32 FileHash;

    // 文件名（如 "image/bg01.png"）
    // 某些加密方案会根据文件名选择不同的加密策略
    const ttstr &FileName;

    // 构造函数
    tTVPXP3ExtractionFilterInfo(
        tjs_uint64 offset, void *buffer,
        tjs_uint buffersize, tjs_uint32 filehash,
        const ttstr &filename)
        : Offset(offset), Buffer(buffer), BufferSize(buffersize),
          FileHash(filehash), FileName(filename),
          SizeOfSelf(sizeof(tTVPXP3ExtractionFilterInfo)) {}
};

#pragma pack(pop)
```

**关键设计点**：

1. **`SizeOfSelf` 字段**：用于前向兼容。如果未来版本在结构体末尾添加新字段，旧的过滤器插件可以通过检查 `SizeOfSelf` 来判断自己是否能处理新版结构。这是 Windows COM 接口中常见的版本兼容手法。

2. **`#pragma pack(push, 4)`**：强制结构体按 4 字节对齐。这确保在不同编译器、不同编译选项下，结构体的内存布局保持一致——这对于跨 DLL 边界传递结构体至关重要。

3. **`const` 修饰符**：除了 `Buffer` 之外的所有字段都是 `const`，表示过滤器**不应该**修改偏移、大小等元数据——它只应该修改 `Buffer` 指向的数据内容。

### 2.2 过滤器函数签名

> **源码位置：** `cpp/core/base/XP3Archive.h` 第 49-51 行

```cpp
// 调用约定宏（在非 Win32 平台上为空）
#define TVP_tTVPXP3ArchiveExtractionFilter_CONVENTION
// 在 Win32 上，这个宏定义为 _stdcall（标准调用约定）
// 使用 _stdcall 是为了与旧版 KiriKiri 插件保持二进制兼容

// 提取过滤器的函数指针类型
typedef void (TVP_tTVPXP3ArchiveExtractionFilter_CONVENTION
              *tTVPXP3ArchiveExtractionFilter)(
    tTVPXP3ExtractionFilterInfo *info,  // 数据块信息
    tTJSVariant *ctx                     // 过滤器上下文（由内容过滤器设置）
);
```

**调用约定**（Calling Convention）是一个底层概念——它规定了函数调用时参数如何压栈、谁负责清理栈帧等细节。`_stdcall` 是 Windows 平台上的标准调用约定，由被调用者负责清理栈。KrKr2 在非 Windows 平台上使用默认调用约定（`_cdecl`），因此将宏定义为空。

### 2.3 过滤器的注册

过滤器通过全局函数注册，整个系统同一时间只能有一个活跃的提取过滤器：

> **源码位置：** `cpp/core/base/XP3Archive.cpp` 第 29-33 行

```cpp
// 全局过滤器函数指针（nullptr 表示未注册过滤器）
tTVPXP3ArchiveExtractionFilter TVPXP3ArchiveExtractionFilter = nullptr;

// 注册函数——直接替换全局函数指针
void TVPSetXP3ArchiveExtractionFilter(
    tTVPXP3ArchiveExtractionFilter filter) {
    TVPXP3ArchiveExtractionFilter = filter;
}
```

### 2.4 过滤器的调用位置

提取过滤器在 `tTVPXP3ArchiveStream::Read()` 函数中、每次数据块读取完成后被调用：

> **源码位置：** `cpp/core/base/XP3Archive.cpp` 第 997-1004 行

```cpp
// 在 Read() 函数的循环中，每读取一块数据后：
if (TVPXP3ArchiveExtractionFilter) {
    // 构造过滤器信息
    tTVPXP3ExtractionFilterInfo info(
        CurPos,                            // 数据在文件中的逻辑偏移
        (tjs_uint8 *)buffer + write_size,  // 指向刚读取的数据
        one_size,                          // 本次读取的字节数
        Owner->GetFileHash(StorageIndex),  // 从 adlr 块获取的文件哈希
        Owner->GetName(StorageIndex)       // 文件名
    );
    // 调用过滤器——过滤器直接修改 buffer 中的数据
    TVPXP3ArchiveExtractionFilter(
        (tTVPXP3ExtractionFilterInfo *)&info,
        &FilterContext  // 上下文变量（由内容过滤器在文件打开时设置）
    );
}
```

**注意**：过滤器在 `memcpy`/`ReadBuffer` 之**后**调用，即它处理的是已解压的数据。因此加密流程是：

```
打包时: 原始数据 → 加密 → zlib压缩 → 写入XP3
读取时: XP3读取 → zlib解压 → 过滤器解密 → 原始数据
```

## 3. 内容过滤器详解

### 3.1 内容过滤器的函数签名

> **源码位置：** `cpp/core/base/XP3Archive.h` 第 53-56 行

```cpp
typedef tjs_int (TVP_tTVPXP3ArchiveExtractionFilter_CONVENTION
                 *tTVPXP3ArchiveContentFilter)(
    const ttstr &filepath,      // 文件路径（如 "image/bg01.png"）
    const ttstr &archivename,   // 归档名（如 "data.xp3"）
    tjs_uint64 filesize,        // 文件原始大小
    tTJSVariant *ctx            // 上下文变量（可设置，后续传给提取过滤器）
);
```

内容过滤器的返回值决定了文件的读取模式：
- **返回 0**：使用正常的分段读取模式（`tTVPXP3ArchiveStream`）
- **返回 1**（`XP3_CONTENT_FILTER_FETCH_FULLDATA`）：全量读取模式

### 3.2 全量读取模式的实现

当内容过滤器返回 1 时，引擎会将整个文件一次性读入内存流：

> **源码位置：** `cpp/core/base/XP3Archive.cpp` 第 563-574 行

```cpp
if (TVPXP3ArchiveContentFilter) {
    tjs_int result = TVPXP3ArchiveContentFilter(
        item.Name,           // 文件路径
        ArchiveName,         // 归档名
        item.OrgSize,        // 原始大小
        &out->GetFilterContext()  // 上下文——过滤器可以在这里存储数据
    );

    #define XP3_CONTENT_FILTER_FETCH_FULLDATA 1
    if (result == XP3_CONTENT_FILTER_FETCH_FULLDATA) {
        // 创建内存流，预分配完整大小
        tTVPMemoryStream *memstr = new tTVPMemoryStream();
        memstr->SetSize(item.OrgSize);

        // 通过 XP3ArchiveStream 读取全部数据（触发所有段的解压+过滤器）
        out->ReadBuffer(memstr->GetInternalBuffer(), item.OrgSize);

        // 销毁 XP3ArchiveStream，返回内存流
        delete out;
        return memstr;
    }
}
```

**为什么需要全量读取模式？**

某些加密方案需要知道文件的全局信息才能解密——例如，密钥可能是文件最后 4 个字节，或者解密某个位置的数据需要参考其他位置的数据。分段读取模式下，过滤器只能看到当前块的数据，无法实现这类全局依赖的解密。全量读取模式先将整个文件读入内存（同时触发每个块的提取过滤器），然后返回一个包含完整解密数据的内存流。

### 3.3 上下文变量的传递

内容过滤器可以通过 `ctx` 参数设置上下文信息，这些信息随后会传递给每次 `Read()` 调用的提取过滤器：

```
内容过滤器调用:
    TVPXP3ArchiveContentFilter(filepath, archivename, filesize, &ctx)
        │
        └── 过滤器可以在 ctx 中存储任意 TJS2 值
            例如：解密密钥、加密模式标志等
                │
                ▼
提取过滤器调用（每次 Read）:
    TVPXP3ArchiveExtractionFilter(info, &ctx)
        │
        └── 可以读取内容过滤器设置的上下文
            例如：根据 ctx 中的密钥解密当前数据块
```

这种两阶段设计允许复杂的加密方案：
1. 内容过滤器在文件打开时做一次性的初始化（计算密钥、判断加密类型等）
2. 提取过滤器在每次读取时高效地执行解密（使用预计算的密钥）

## 4. xp3filter.tjs 脚本化过滤器系统

KrKr2 允许游戏开发者用 TJS2 脚本编写过滤器逻辑，而不必编写 C++ 代码。这是通过 `xp3filter` 插件实现的。

### 4.1 脚本加载流程

> **源码位置：** `cpp/plugins/xp3filter.cpp` 第 550-566 行

```cpp
static void PostRegistCallback() {
    // 构建脚本路径：<应用目录>/xp3filter.tjs
    ttstr path = TVPGetAppPath() + TJS_W("xp3filter.tjs");

    // 检查脚本文件是否存在
    if (TVPIsExistentStorageNoSearch(path)) {
        // 读取脚本内容
        iTJSTextReadStream *stream = TVPCreateTextStreamForRead(path, "");
        try {
            stream->Read(sXP3FilterScript, 0);
        } catch (...) {
            stream->Destruct();
            throw;
        }
        stream->Destruct();

        // 注册 C++ 包装函数为全局过滤器
        TVPSetXP3ArchiveExtractionFilter(
            TVPXP3ArchiveExtractionFilterWrapper);
        TVPSetXP3ArchiveContentFilter(
            TVPXP3ArchiveContentFilterWrapper);
    }
}

// 在插件系统初始化后自动调用
NCB_POST_REGIST_CALLBACK(PostRegistCallback);
```

流程说明：
1. 引擎启动时，插件系统自动调用 `PostRegistCallback`
2. 检查游戏目录下是否存在 `xp3filter.tjs` 脚本
3. 如果存在，读取脚本内容，注册 C++ 包装函数作为全局过滤器
4. 当过滤器被调用时，C++ 包装函数会转发到 TJS2 脚本中定义的解密函数

### 4.2 TJS2 脚本引擎的创建

每个线程拥有独立的 TJS2 脚本引擎实例，以确保线程安全：

> **源码位置：** `cpp/plugins/xp3filter.cpp` 第 301-310 行，394-420 行

```cpp
// 过滤器解码器——封装一个独立的 TJS2 脚本引擎
struct XP3FilterDecoder {
    tTJS *ScriptEngine = new tTJS();      // 独立的 TJS2 引擎实例
    tTJSVariantClosure ManagedDecoder;    // TJS2 中注册的解密回调
    tTJSVariantClosure ManagedFilter;     // TJS2 中注册的内容过滤回调

    XP3FilterDecoder()
        : ManagedDecoder(nullptr), ManagedFilter(nullptr) {}
    ~XP3FilterDecoder() {
        if (ScriptEngine) ScriptEngine->Release();
    }
};

// 创建并初始化过滤器解码器
static XP3FilterDecoder *AddXP3Decoder() {
    XP3FilterDecoder *decoder = new XP3FilterDecoder;
    iTJSDispatch2 *global = decoder->ScriptEngine->GetGlobalNoAddRef();

    // 注册 Debug、System、Storages 等内置类到脚本引擎
    // 使脚本可以调用 Storages.setXP3ArchiveExtractionFilter()
    REGISTER_OBJECT(Debug, TVPCreateNativeClass_Debug());
    REGISTER_OBJECT(System, TVPCreateNativeClass_System());

    tTJSNativeClass *cls = TVPCreateNativeClass_Storages();
    // 注册自定义的 setXP3ArchiveExtractionFilter 方法
    TJSNativeClassRegisterNCM(cls,
        TJS_W("setXP3ArchiveExtractionFilter"),
        new XP3FilterRegister(decoder),  // 回调会存储到 decoder
        cls->GetClassName().c_str(), nitMethod, TJS_STATICMEMBER);
    // 注册自定义的 setXP3ArchiveContentFilter 方法
    TJSNativeClassRegisterNCM(cls,
        TJS_W("setXP3ArchiveContentFilter"),
        new XP3ContentFilterRegister(decoder),
        cls->GetClassName().c_str(), nitMethod, TJS_STATICMEMBER);
    REGISTER_OBJECT(Storages, cls);

    // 执行过滤器脚本——脚本中会调用上述注册方法
    decoder->ScriptEngine->ExecScript(sXP3FilterScript);
    return decoder;
}
```

### 4.3 线程本地解码器管理

为了支持多线程环境（如音频线程、IO 线程同时读取 XP3 文件），系统为每个线程维护独立的解码器实例：

> **源码位置：** `cpp/plugins/xp3filter.cpp` 第 423-456 行

```cpp
// 线程 ID → 解码器实例的映射
static std::map<std::thread::id, XP3FilterDecoder*> _thread_decoders;
static std::mutex _decoders_mtx;
static std::vector<XP3FilterDecoder*> _cached_decoders;  // 回收池

static XP3FilterDecoder *FetchXP3Decoder() {
    std::lock_guard<std::mutex> lk(_decoders_mtx);

    // 检查当前线程是否已有解码器
    auto it = _thread_decoders.find(std::this_thread::get_id());
    if (it != _thread_decoders.end())
        return it->second;

    // 首次使用时注册线程退出回调
    static bool Inited = false;
    if (!Inited) {
        Inited = true;
        TVPAddOnThreadExitEvent([]() {
            std::lock_guard<std::mutex> lk(_decoders_mtx);
            auto it = _thread_decoders.find(std::this_thread::get_id());
            if (it != _thread_decoders.end()) {
                // 线程退出时，将解码器放入回收池（而非销毁）
                _cached_decoders.emplace_back(it->second);
                _thread_decoders.erase(it);
            }
        });
    }

    // 优先从回收池取已有解码器（避免重新创建 TJS2 引擎）
    XP3FilterDecoder *ret;
    if (!_cached_decoders.empty()) {
        ret = _cached_decoders.back();
        _cached_decoders.pop_back();
    } else {
        ret = AddXP3Decoder();  // 回收池空 → 创建新实例
    }

    _thread_decoders[std::this_thread::get_id()] = ret;
    return ret;
}
```

这个设计类似于**线程池**（Thread Pool）中的对象复用——TJS2 脚本引擎的创建和脚本编译是昂贵的操作，通过回收池避免反复创建。

### 4.4 提取过滤器的 C++/TJS2 桥接

当 C++ 引擎调用提取过滤器时，包装函数将参数转换为 TJS2 变量，然后调用脚本中定义的解密函数：

> **源码位置：** `cpp/plugins/xp3filter.cpp` 第 493-531 行

```cpp
void TVPXP3ArchiveExtractionFilterWrapper(
    tTVPXP3ExtractionFilterInfo *info,
    tTJSVariant *ctx) {

    // 版本兼容性检查
    if (info->SizeOfSelf != sizeof(tTVPXP3ExtractionFilterInfo))
        TVPThrowExceptionMessage(
            TJS_W("Incompatible tTVPXP3ExtractionFilterInfo size"));

    // 获取当前线程的解码器
    XP3FilterDecoder *decoder = FetchXP3Decoder();

    if (decoder->ManagedDecoder.Object) {
        // 将 C++ 参数转换为 TJS2 变量
        tTJSVariant FileHash = (tjs_int64)info->FileHash;
        tTJSVariant Offset = (tjs_int64)info->Offset;

        // 创建二进制访问器——将 C++ 缓冲区暴露给 TJS2
        CBinaryAccessor *buf = new CBinaryAccessor(
            (unsigned char *)info->Buffer, info->BufferSize);
        tTJSVariant Buffer(buf);
        buf->Release();

        tTJSVariant BufferSize((tjs_int64)info->BufferSize);
        tTJSVariant FileName(info->FileName);

        // 构造参数数组
        tTJSVariant *vars[] = {
            &FileHash,    // 参数 0: 文件哈希（32位）
            &Offset,      // 参数 1: 数据在文件中的偏移
            &Buffer,      // 参数 2: CBinaryAccessor 对象
            &BufferSize,  // 参数 3: 缓冲区大小
            &FileName,    // 参数 4: 文件名
            ctx           // 参数 5: 上下文变量
        };

        // 调用 TJS2 脚本中定义的解密函数
        decoder->ManagedDecoder.FuncCall(
            0, nullptr, nullptr, nullptr,
            sizeof(vars) / sizeof(vars[0]),
            vars, nullptr
        );
    }
}
```

### 4.5 CBinaryAccessor：字节级操作桥接

`CBinaryAccessor` 是一个特殊的 TJS2 对象，它允许脚本直接操作 C++ 内存缓冲区中的字节：

> **源码位置：** `cpp/plugins/xp3filter.h` 第 4-101 行，`cpp/plugins/xp3filter.cpp` 第 244-296 行

```cpp
class CBinaryAccessor : public tTJSDispatch {
    unsigned int m_length;    // 缓冲区长度
    unsigned int m_curPos;    // 当前指针位置
    unsigned char *m_buff;    // 底层缓冲区（不拥有所有权）

    // ... TJS2 接口实现 ...
};
```

CBinaryAccessor 向 TJS2 脚本暴露以下功能：

| 脚本操作 | C++ 实现 | 说明 |
|----------|----------|------|
| `buf[i]` 读取 | `PropGetByNum` | 返回 `m_buff[m_curPos + i]` |
| `buf[i] = v` 写入 | `PropSetByNum` | 设置 `m_buff[m_curPos + i] = v` |
| `buf[i] ^= v` XOR | `OperationByNum` | 执行 `m_buff[m_curPos + i] ^= v` |
| `buf.ptr` 读写 | `PropGet/Set` | 获取/设置当前指针位置 |
| `buf.count` | `PropGet` | 返回缓冲区总长度 |
| `buf.xor(off, len, val)` | `FuncXor` | 批量 XOR 操作（有 SIMD 优化） |

**批量 XOR 函数的 SIMD 优化**：

> **源码位置：** `cpp/plugins/xp3filter.cpp` 第 266-296 行

```cpp
tjs_error CBinaryAccessor::FuncXor(tjs_int numparams, tTJSVariant **param) {
    if (numparams < 3) return TJS_E_BADPARAMCOUNT;

    int bufoff = param[0]->AsInteger();  // 起始偏移
    int len = param[1]->AsInteger();     // 长度
    unsigned char xorval = param[2]->AsInteger();  // XOR 值

    unsigned char *buf = m_buff + m_curPos + bufoff;
    unsigned char *pend = buf + len;

    if (len > 32) {
        // 数据量较大时，使用 8 字节对齐的批量处理
        // 先处理未对齐的前导字节
        int PreFragLen =
            (unsigned char *)((((intptr_t)buf) + 7) & ~7) - buf;
        for (int i = 0; i < PreFragLen; i++)
            *(buf++) ^= xorval;

        // 将单字节 XOR 值扩展为 8 字节
        uint64_t k = xorval;
        k |= k << 8;   // 0x00XX → 0xXXXX
        k |= k << 16;  // → 0xXXXXXXXX
        k |= k << 32;  // → 0xXXXXXXXXXXXXXXXX

        // 每次处理 8 字节
        unsigned char *pVecEnd =
            (unsigned char *)(((intptr_t)pend) & ~7) - 7;
        while (buf < pVecEnd) {
            *(uint64_t *)(buf) ^= k;
            buf += 8;
        }
    }
    // 处理剩余的尾部字节
    while (buf < pend)
        *(buf++) ^= xorval;

    return TJS_S_OK;
}
```

这段代码展示了一个经典的**SIMD 风格优化**——通过将 1 字节的 XOR 值复制为 8 字节，然后一次处理 8 个字节，将循环迭代次数减少到原来的 1/8。虽然没有使用真正的 SIMD 指令（如 SSE/NEON），但利用 64 位整数运算达到了类似的效果。

### 5.2 基于偏移的变密钥加密

更安全的方案——密钥随偏移位置变化：

```javascript
// xp3filter.tjs —— 基于偏移的变密钥加密
Storages.setXP3ArchiveExtractionFilter(
    function(fileHash, offset, buffer, bufferSize, fileName, ctx) {
        // 将文件哈希分解为 4 个字节
        var k0 = (fileHash      ) & 0xFF;
        var k1 = (fileHash >>  8) & 0xFF;
        var k2 = (fileHash >> 16) & 0xFF;
        var k3 = (fileHash >> 24) & 0xFF;

        // 每个字节的密钥 = 哈希字节 XOR 偏移位置
        for (var i = 0; i < bufferSize; i++) {
            var pos = offset + i;       // 在文件中的绝对位置
            var keyByte;
            switch (pos % 4) {          // 4 字节密钥循环
                case 0: keyByte = k0 ^ ((pos >> 4) & 0xFF); break;
                case 1: keyByte = k1 ^ ((pos >> 8) & 0xFF); break;
                case 2: keyByte = k2 ^ ((pos >> 12) & 0xFF); break;
                case 3: keyByte = k3 ^ ((pos >> 16) & 0xFF); break;
            }
            buffer[i] ^= keyByte;
        }
    }
);
```

### 5.3 带内容过滤器的完整方案

```javascript
// xp3filter.tjs —— 完整的内容过滤器 + 提取过滤器方案

// 内容过滤器：决定哪些文件需要全量读取
Storages.setXP3ArchiveContentFilter(
    function(filePath, archiveName, fileSize) {
        // 脚本文件需要全量读取（因为解密算法需要全局信息）
        if (filePath.indexOf(".ks") >= 0 ||
            filePath.indexOf(".tjs") >= 0) {
            // 返回数组：[模式, 上下文]
            // 模式 1 = 全量读取
            return [1, %["mode" => "script_decrypt"]];
        }
        // 其他文件使用正常的分段读取
        return [0, %["mode" => "normal"]];
    }
);

// 提取过滤器：实际的解密逻辑
Storages.setXP3ArchiveExtractionFilter(
    function(fileHash, offset, buffer, bufferSize, fileName, ctx) {
        // ctx 是内容过滤器设置的上下文
        if (ctx !== void && ctx.mode == "script_decrypt") {
            // 脚本文件使用复杂的解密
            var seed = fileHash;
            for (var i = 0; i < bufferSize; i++) {
                seed = (seed * 1103515245 + 12345) & 0x7FFFFFFF;
                buffer[i] ^= (seed >> 16) & 0xFF;
            }
        } else {
            // 其他文件使用简单 XOR
            buffer.xor(0, bufferSize, fileHash & 0xFF);
        }
    }
);
```

### 5.4 不使用过滤器的场景

某些游戏不加密资源文件。此时 `xp3filter.tjs` 文件不存在，引擎中 `TVPXP3ArchiveExtractionFilter` 保持为 `nullptr`。`Read()` 函数中的 `if (TVPXP3ArchiveExtractionFilter)` 检查直接跳过，没有任何性能开销。

## 6. 动手实践：实现自定义加密/解密

下面我们实现一个完整的自定义加密/解密系统，包括加密工具（模拟打包器）和解密过滤器：

```cpp
#include <cstdint>
#include <cstring>
#include <vector>
#include <string>
#include <iostream>
#include <functional>
#include <cassert>

// ========== 加密/解密算法 ==========

// 用文件哈希生成密钥流
static uint8_t generateKeyByte(uint32_t fileHash, uint64_t offset) {
    // 简单的伪随机密钥生成：
    // 将文件哈希和偏移混合，得到每个位置的密钥字节
    uint32_t seed = fileHash ^ static_cast<uint32_t>(offset);
    seed = seed * 2654435761u;  // Knuth 乘法哈希
    seed ^= (seed >> 16);
    return static_cast<uint8_t>(seed & 0xFF);
}

// 加密一块数据（用于打包器）
void encryptBlock(uint8_t *data, size_t size,
                  uint32_t fileHash, uint64_t offset) {
    for (size_t i = 0; i < size; i++) {
        data[i] ^= generateKeyByte(fileHash, offset + i);
    }
}

// 解密一块数据（用于过滤器——与加密完全相同，因为 XOR 是自反的）
void decryptBlock(uint8_t *data, size_t size,
                  uint32_t fileHash, uint64_t offset) {
    // XOR 加密的特性：加密和解密是同一个操作
    encryptBlock(data, size, fileHash, offset);
}

// ========== 模拟过滤器信息结构 ==========
struct FilterInfo {
    uint64_t offset;       // 数据偏移
    uint8_t *buffer;       // 数据缓冲区
    uint32_t bufferSize;   // 缓冲区大小
    uint32_t fileHash;     // 文件哈希
    std::string fileName;  // 文件名
};

// 过滤器回调类型
using ExtractionFilter = std::function<void(FilterInfo *info)>;

// ========== 测试 ==========
int main() {
    // 模拟原始文件数据
    std::string originalText = "这是一段需要加密保护的游戏脚本内容。"
                               "KiriKiri2 引擎使用 XP3 归档格式存储游戏资源。";

    std::vector<uint8_t> originalData(originalText.begin(), originalText.end());
    uint32_t fileHash = 0xDEADBEEF;  // 模拟 adlr 块中的文件哈希

    // ===== 步骤 1: 加密（模拟打包器行为）=====
    std::vector<uint8_t> encryptedData = originalData;
    encryptBlock(encryptedData.data(), encryptedData.size(), fileHash, 0);

    std::cout << "原始数据前 16 字节: ";
    for (int i = 0; i < 16 && i < originalData.size(); i++)
        printf("%02X ", originalData[i]);
    std::cout << std::endl;

    std::cout << "加密后前 16 字节: ";
    for (int i = 0; i < 16 && i < encryptedData.size(); i++)
        printf("%02X ", encryptedData[i]);
    std::cout << std::endl;

    // 验证加密确实改变了数据
    assert(originalData != encryptedData);
    std::cout << "✓ 加密成功（数据已变化）" << std::endl;

    // ===== 步骤 2: 设置解密过滤器 =====
    ExtractionFilter myFilter = [](FilterInfo *info) {
        decryptBlock(
            info->buffer,
            info->bufferSize,
            info->fileHash,
            info->offset
        );
    };

    // ===== 步骤 3: 模拟分段读取 + 过滤器解密 =====
    std::vector<uint8_t> decryptedData;
    size_t chunkSize = 20;  // 模拟每次读取 20 字节

    for (size_t offset = 0; offset < encryptedData.size(); offset += chunkSize) {
        size_t readSize = std::min(chunkSize,
                                   encryptedData.size() - offset);

        // 模拟 Read(): 从"归档"复制数据
        std::vector<uint8_t> chunk(readSize);
        std::memcpy(chunk.data(),
                    encryptedData.data() + offset, readSize);

        // 模拟 Read() 后调用过滤器
        FilterInfo info;
        info.offset = offset;
        info.buffer = chunk.data();
        info.bufferSize = static_cast<uint32_t>(readSize);
        info.fileHash = fileHash;
        info.fileName = "scenario/main.ks";

        myFilter(&info);

        // 收集解密后的数据
        decryptedData.insert(decryptedData.end(),
                             chunk.begin(), chunk.end());
    }

    // ===== 步骤 4: 验证解密结果 =====
    assert(decryptedData == originalData);
    std::string decryptedText(decryptedData.begin(), decryptedData.end());
    std::cout << "✓ 解密成功！内容: " << decryptedText << std::endl;

    // ===== 步骤 5: 测试偏移一致性 =====
    // 无论 chunkSize 是多少，解密结果必须相同
    for (size_t testChunk : {1, 7, 50, 100, 999}) {
        std::vector<uint8_t> testDecrypted;
        for (size_t off = 0; off < encryptedData.size(); off += testChunk) {
            size_t rs = std::min(testChunk,
                                 encryptedData.size() - off);
            std::vector<uint8_t> c(rs);
            std::memcpy(c.data(), encryptedData.data() + off, rs);

            FilterInfo ti;
            ti.offset = off;
            ti.buffer = c.data();
            ti.bufferSize = static_cast<uint32_t>(rs);
            ti.fileHash = fileHash;
            myFilter(&ti);

            testDecrypted.insert(testDecrypted.end(), c.begin(), c.end());
        }
        assert(testDecrypted == originalData);
        std::cout << "✓ chunkSize=" << testChunk << " 解密一致" << std::endl;
    }

    return 0;
}
```

## 8. 常见错误与排查

### 错误 1: "Incompatible tTVPXP3ExtractionFilterInfo size"

**症状**：过滤器回调抛出此异常

**原因**：过滤器插件编译时的 `tTVPXP3ExtractionFilterInfo` 结构体大小与引擎运行时的不一致。常见于：
- 使用了不同版本的头文件编译插件
- 32 位插件加载到 64 位引擎中（指针大小不同导致结构体大小变化）

**解决方案**：确保插件和引擎使用相同版本的 `XP3Archive.h`，并使用相同的目标平台编译。

### 错误 2: 过滤器脚本未生效

**症状**：`xp3filter.tjs` 存在但加密文件读取后仍是乱码

**排查步骤**：
```javascript
// 在 xp3filter.tjs 中添加调试输出
Storages.setXP3ArchiveExtractionFilter(
    function(fileHash, offset, buffer, bufferSize, fileName, ctx) {
        // 添加调试日志
        Debug.message("过滤器被调用: " + fileName +
                      " offset=" + offset +
                      " size=" + bufferSize +
                      " hash=" + fileHash);

        // ... 解密逻辑 ...
    }
);
```

**常见原因**：
1. `xp3filter.tjs` 编码不正确（应使用 UTF-8 或 UTF-16LE with BOM）
2. 脚本有语法错误导致 `ExecScript` 失败
3. 脚本中调用 `setXP3ArchiveExtractionFilter` 的名称拼写错误

### 错误 3: 多线程环境下解密结果随机出错

**症状**：大部分文件解密正确，但偶尔出现部分文件解密失败

**原因**：可能是 TJS2 脚本中使用了共享的全局变量，导致线程间数据竞争

**解决方案**：确保解密脚本中不使用全局可变状态。所有临时变量都应在函数内部声明。

## 9. 本节小结

- XP3 过滤器系统采用**可插拔回调**设计，不修改归档格式本身即可实现加密保护
- **提取过滤器**在每次 `Read()` 后被调用，接收数据块、偏移、文件哈希等信息，直接修改缓冲区实现解密
- **内容过滤器**在文件打开时被调用，可以决定是否全量读取，并设置传递给提取过滤器的上下文
- `xp3filter` 插件实现了 C++/TJS2 桥接，允许用脚本编写过滤器逻辑
- `CBinaryAccessor` 将 C++ 内存缓冲区暴露给 TJS2 脚本，支持按字节读写和批量 XOR 操作
- 线程安全通过**每线程独立的 TJS2 引擎实例**和**解码器回收池**保证
- XOR 加密是自反运算（`a ^ k ^ k == a`），因此加密和解密使用同一算法
- 过滤器的性能关键：`TVPXP3ArchiveExtractionFilter == nullptr` 时无开销（零成本抽象）

### 题目 1：分析过滤器调用时序

假设一个 XP3 中的文件有 3 个段，每段 100 字节。调用者请求 `Read(buffer, 250)`。已注册提取过滤器。

请问：
1. 过滤器会被调用几次？
2. 每次调用时，`Offset` 和 `BufferSize` 分别是多少？
3. 如果中间某段是未压缩段，是否影响过滤器的调用？

<details>
<summary>查看答案</summary>

1. **过滤器调用 3 次**（每读取一个段的数据就调用一次）

2. 三次调用的参数：
   ```
   第 1 次: Offset=0,   BufferSize=100  （段 0 全部 100 字节）
   第 2 次: Offset=100, BufferSize=100  （段 1 全部 100 字节）
   第 3 次: Offset=200, BufferSize=50   （段 2 的前 50 字节）
   ```

3. **不影响**。过滤器在数据读取之后调用，无论数据来自压缩段（memcpy from SegmentData）还是未压缩段（ReadBuffer from Stream），过滤器都会对读到的数据进行处理。

补充：`CurPos` 在过滤器调用时的值就是 `Offset`，它是在数据复制/读取之前的值（因为 `CurPos += one_size` 在过滤器调用之后执行）。

</details>

### 题目 2：XOR 加密的安全性分析

考虑以下简单的 XOR 加密方案：

```javascript
Storages.setXP3ArchiveExtractionFilter(
    function(fileHash, offset, buffer, bufferSize, fileName, ctx) {
        buffer.xor(0, bufferSize, fileHash & 0xFF);
    }
);
```

请分析这种加密方案的安全弱点（至少列出 3 个），并说明如何改进。

<details>
<summary>查看答案</summary>

**安全弱点**：

1. **密钥空间太小**：`fileHash & 0xFF` 只有 256 种可能的密钥值。攻击者只需尝试 256 次即可暴力破解。

2. **已知明文攻击**：PNG 文件以固定的魔数 `89 50 4E 47` 开头，如果攻击者知道某个文件是 PNG 格式，可以用第一个字节 `0x89` XOR 加密后的第一个字节得到密钥。

3. **所有位置使用相同密钥**：单字节 XOR 意味着文件中每隔 1 个字节就重复使用相同的密钥。频率分析（Frequency Analysis）可以轻松破解。

4. **同一文件始终相同**：相同文件的 fileHash 不变，加密结果永远相同——不具备语义安全（Semantic Security）。

5. **文件哈希可提取**：fileHash 存储在 XP3 索引的 `adlr` 块中，索引本身通常不加密，攻击者可以直接读取。

**改进方案**：

```javascript
Storages.setXP3ArchiveExtractionFilter(
    function(fileHash, offset, buffer, bufferSize, fileName, ctx) {
        // 改进 1: 使用全部 32 位哈希
        // 改进 2: 密钥随偏移变化
        // 改进 3: 加入额外的固定密钥（需要逆向才能获得）
        var SECRET = 0x5A3C7B91;  // 编译到脚本中的密钥
        var seed = fileHash ^ SECRET;

        for (var i = 0; i < bufferSize; i++) {
            // 使用 LCG 伪随机数生成器
            seed = (seed * 1103515245 + 12345) & 0x7FFFFFFF;
            buffer[i] ^= (seed >> 16) & 0xFF;
        }
    }
);
```

这种改进方案的密钥空间为 2^32（全部 fileHash 位），密钥随偏移变化（LCG 产生不同值），并且额外混入了编译时固定的密钥。虽然仍非密码学安全（LCG 可预测），但足以阻止简单的提取工具。

</details>

### 题目 3：实现内容过滤器

编写一个 C++ 程序，模拟内容过滤器和提取过滤器的协作。要求：
1. 内容过滤器根据文件扩展名决定加密模式（`.ks` 文件使用强加密，其他文件使用简单加密）
2. 将加密模式通过上下文变量传递给提取过滤器
3. 提取过滤器根据上下文中的模式选择不同的解密算法

<summary>查看答案</summary>

```cpp
#include <cstdint>
#include <cstring>
#include <vector>
#include <string>
#include <iostream>
#include <functional>
#include <cassert>
#include <variant>

// 上下文类型（模拟 tTJSVariant）
struct FilterContext {
    std::string mode;     // "strong" 或 "simple"
    uint32_t extraKey;    // 额外密钥（仅强加密模式使用）
};

// 内容过滤器类型
// 返回: 0=正常读取, 1=全量读取
// ctx: 输出参数，设置上下文
using ContentFilter = std::function<int(
    const std::string &filePath,
    const std::string &archiveName,
    uint64_t fileSize,
    FilterContext &ctx
)>;

// 提取过滤器类型
using ExtractionFilter = std::function<void(
    uint8_t *buffer, uint32_t size,
    uint64_t offset, uint32_t fileHash,
    const std::string &fileName,
    const FilterContext &ctx
)>;

// 简单加密：单字节 XOR
void simpleEncrypt(uint8_t *data, size_t size,
                   uint32_t fileHash, uint64_t offset) {
    uint8_t key = fileHash & 0xFF;
    for (size_t i = 0; i < size; i++)
        data[i] ^= key;
}

// 强加密：LCG 伪随机流
void strongEncrypt(uint8_t *data, size_t size,
                   uint32_t fileHash, uint64_t offset,
                   uint32_t extraKey) {
    uint32_t seed = fileHash ^ extraKey;
    // 快进到正确的偏移位置
    for (uint64_t i = 0; i < offset; i++)
        seed = (seed * 1103515245 + 12345) & 0x7FFFFFFF;
    // 加密当前块
    for (size_t i = 0; i < size; i++) {
        seed = (seed * 1103515245 + 12345) & 0x7FFFFFFF;
        data[i] ^= (seed >> 16) & 0xFF;
    }
}

int main() {
    // 设置内容过滤器
    ContentFilter contentFilter = [](
        const std::string &filePath,
        const std::string &archiveName,
        uint64_t fileSize,
        FilterContext &ctx) -> int {

        // 检查文件扩展名
        if (filePath.size() >= 3 &&
            filePath.substr(filePath.size() - 3) == ".ks") {
            ctx.mode = "strong";
            ctx.extraKey = 0x12345678;
            std::cout << "[内容过滤器] " << filePath
                      << " → 强加密模式" << std::endl;
            return 0;  // 正常读取（分段）
        } else {
            ctx.mode = "simple";
            ctx.extraKey = 0;
            std::cout << "[内容过滤器] " << filePath
                      << " → 简单加密模式" << std::endl;
            return 0;
        }
    };

    // 设置提取过滤器
    ExtractionFilter extractionFilter = [](
        uint8_t *buffer, uint32_t size,
        uint64_t offset, uint32_t fileHash,
        const std::string &fileName,
        const FilterContext &ctx) {

        if (ctx.mode == "strong") {
            strongEncrypt(buffer, size, fileHash, offset, ctx.extraKey);
        } else {
            simpleEncrypt(buffer, size, fileHash, offset);
        }
    };

    // 测试文件
    struct TestFile {
        std::string name;
        std::string content;
        uint32_t hash;
    };

    std::vector<TestFile> files = {
        {"scenario/main.ks", "这是脚本文件内容", 0xAABBCCDD},
        {"image/bg01.png",   "PNG图片数据模拟", 0x11223344}
    };

    for (auto &file : files) {
        std::vector<uint8_t> original(file.content.begin(),
                                       file.content.end());

        // 1. 内容过滤器设置上下文
        FilterContext ctx;
        int mode = contentFilter(file.name, "data.xp3",
                                  original.size(), ctx);

        // 2. 加密（模拟打包器）
        std::vector<uint8_t> encrypted = original;
        if (ctx.mode == "strong") {
            strongEncrypt(encrypted.data(), encrypted.size(),
                         file.hash, 0, ctx.extraKey);
        } else {
            simpleEncrypt(encrypted.data(), encrypted.size(),
                         file.hash, 0);
        }

        // 3. 解密（模拟引擎读取 + 过滤器）
        std::vector<uint8_t> decrypted = encrypted;
        extractionFilter(decrypted.data(), decrypted.size(),
                        0, file.hash, file.name, ctx);

        // 4. 验证
        assert(decrypted == original);
        std::string result(decrypted.begin(), decrypted.end());
        std::cout << "✓ " << file.name << " 解密正确: "
                  << result << std::endl;
    }

    return 0;
}
```
