## 一、7z 格式概述——高压缩比的现代归档格式

### 1.1 7z 的设计哲学

7z 格式由 Igor Pavlov 于 1999 年随 7-Zip 压缩工具一同发布。与 TAR（无压缩）和 ZIP（逐文件压缩）不同，7z 的核心设计理念是**固实压缩**（Solid Compression）：

- **固实归档**：将多个文件的数据合并为一个连续流后统一压缩，而不是逐个文件单独压缩。这使得相似文件之间的冗余数据可以被字典压缩算法利用，压缩率远高于 ZIP
- **多编码器链**：每个文件夹可以使用多个编码器串联——例如先用 BCJ2（x86 分支跳转预处理器，一种针对可执行文件的预处理滤镜，将跳转地址转换为相对偏移以提高压缩率）预处理，再用 LZMA 压缩
- **插件式算法**：7z 容器不绑定特定压缩算法，支持 LZMA、LZMA2、PPMD、BZip2、Deflate 等多种算法

```

│                    7z 文件结构                             │

│                                                          │
│  ┌─────────────────────────────┐                         │
│  │  Signature Header          │ ← 32 字节固定头部        │
```

### 1.2 为什么 KrKr2 需要支持 7z？

1. **高压缩比**：对于大量文本资源（脚本、配置文件），LZMA 的压缩率比 zlib 高 30-50%，可以显著减小游戏包体积
2. **社区需求**：某些游戏 MOD 和资源包使用 7z 格式分发
3. **现成 SDK**：LZMA SDK 是公有领域代码（Public Domain），没有任何许可证限制
4. **实现可控**：利用 LZMA SDK 的高层 API（`SzArEx_Open`/`SzArEx_Extract`），260 行代码即可完成完整的 7z 读取支持

### 1.3 7z vs ZIP vs TAR 的关键区别

| 特性 | 7z | ZIP | TAR |
|------|-----|-----|-----|
| 压缩粒度 | **文件夹级**（多文件合并压缩） | 文件级（逐文件压缩） | 无压缩 |
| 随机访问 | ⚠️ 需解压整个文件夹 | ✅ 直接定位文件 | ❌ 顺序扫描 |
| 解压性能 | 较慢（CPU 密集） | 快（zlib 高效） | 最快（零拷贝） |
| 适用场景 | 最终分发（高压缩） | 通用归档 | 开发调试 |

## 二、LZMA SDK 的 C 语言接口

### 2.1 为什么是 C 接口而不是 C++？

LZMA SDK 同时提供 C 和 C++ 接口。KrKr2 选择使用 C 接口有几个原因：

1. **更轻量**：C 接口不需要 C++ 运行时库（RTTI、异常处理等）
2. **更稳定**：C ABI 跨编译器兼容性更好
3. **更可控**：通过函数指针回调实现多态，避免 C++ 虚表的开销

KrKr2 使用 `extern "C"` 块来包含 LZMA SDK 的 C 头文件：

```cpp
// 来源：cpp/core/base/7zArchive.cpp 第 6-10 行
extern "C" {
#include <7zip/C/7z.h>      // 7z 解压核心 API
#include <7zip/C/7zFile.h>  // 文件 I/O 接口
#include <7zip/C/7zCrc.h>   // CRC32 校验
}
```

### 2.2 ISzAlloc——内存分配器接口

LZMA SDK 不直接调用 `malloc/free`，而是通过 `ISzAlloc` 接口让调用者控制内存分配策略：

```cpp
// 来源：cpp/core/base/7zArchive.cpp 第 14-17 行
static ISzAlloc allocImp = {
    // Alloc 函数指针：分配内存
    [](ISzAllocPtr p, size_t size) -> void * {
        return malloc(size);
    },
    // Free 函数指针：释放内存
    [](ISzAllocPtr p, void *addr) {
        free(addr);
    }
};
```

KrKr2 使用 C++ lambda 表达式（会被编译器转换为普通函数指针，因为这两个 lambda 不捕获任何变量）来实现这个接口，直接转发到标准的 `malloc/free`。

在生产环境中，你可以替换为自定义的内存池分配器以提高性能。例如：

```cpp
// 示例：使用内存池的自定义分配器
class MemoryPool {
    static constexpr size_t POOL_SIZE = 64 * 1024 * 1024; // 64MB
    char pool[POOL_SIZE];
    size_t offset = 0;
public:
    void* alloc(size_t size) {
        size = (size + 7) & ~7; // 8 字节对齐
        if (offset + size > POOL_SIZE) return nullptr;
        void* ptr = pool + offset;
        offset += size;
        return ptr;
    }
    void reset() { offset = 0; }
};
```

### 2.3 ISeekInStream——可定位输入流接口

LZMA SDK 的核心 I/O 接口是 `ISeekInStream`，它定义了两个函数指针：

```c
// LZMA SDK 定义（简化）
typedef struct ISeekInStream {
    // 读取数据：从当前位置读取 *size 字节到 buf
    // 实际读取的字节数写回 *size
    SRes (*Read)(ISeekInStreamPtr p, void *buf, size_t *size);

    // 定位：将流位置设置为 *pos（相对于 origin）
    // 新位置写回 *pos
    SRes (*Seek)(ISeekInStreamPtr p, Int64 *pos, ESzSeek origin);
} ISeekInStream;
```

`ESzSeek` 枚举与标准 C 的 `SEEK_SET/SEEK_CUR/SEEK_END` 对应：

```c
typedef enum {
    SZ_SEEK_SET = 0,  // 从文件开头定位
    SZ_SEEK_CUR = 1,  // 从当前位置定位
    SZ_SEEK_END = 2   // 从文件末尾定位
} ESzSeek;
```

### 2.4 CLookToRead2——缓冲读取层

直接使用 `ISeekInStream` 每次只读取少量字节效率很低，因为每次 `Read` 调用都可能触发系统调用（文件 I/O 或网络 I/O）。LZMA SDK 提供了 `CLookToRead2` 作为缓冲层：

```
┌─────────────────────────────────────────────────────┐
│              I/O 层次结构                            │
├─────────────────────────────────────────────────────┤
│                                                     │
│  ┌───────────────────┐                              │
│  │  SzArEx_Open()    │  ← LZMA SDK 高层 API        │
│  │  SzArEx_Extract() │                              │
│  └────────┬──────────┘                              │
│           │ 调用                                    │
│           ▼                                         │
│  ┌───────────────────┐                              │
│  │  CLookToRead2     │  ← 缓冲层（16KB 缓冲区）    │
│  │  .vt (ILookInStream 虚表)                       │
│  │  .realStream      │  → 指向 ISeekInStream       │
│  │  .buf[16384]      │  预读缓冲                   │
│  └────────┬──────────┘                              │
│           │ 转发读取/定位请求                        │
│           ▼                                         │
│  ┌───────────────────┐                              │
│  │  CSeekInStream    │  ← KrKr2 的 ISeekInStream   │
│  │  .Read → lambda   │    实现（函数指针回调）      │
│  │  .Seek → lambda   │                              │
│  │  .Host → wrap     │                              │
│  └────────┬──────────┘                              │
│           │ 委托到 Host                             │
│           ▼                                         │
│  ┌───────────────────┐                              │
│  │  tTJSBinaryStream │  ← KrKr2 的流基类           │
│  │  .Read()          │                              │
│  │  .Seek()          │                              │
│  └───────────────────┘                              │
│                                                     │
└─────────────────────────────────────────────────────┘
```

`CLookToRead2` 维护一个内部缓冲区（默认 16KB），将小的、频繁的读取请求合并为较大的块读取，显著减少底层 I/O 调用次数。

## 三、SevenZipStreamWrap 适配器——核心桥接层

### 3.1 设计动机

KrKr2 的流系统使用 C++ 类 `tTJSBinaryStream`，而 LZMA SDK 使用 C 语言的函数指针回调（`ISeekInStream`）。`SevenZipStreamWrap` 类的作用就是**在两者之间建立桥梁**——将 `tTJSBinaryStream` 的方法包装为 LZMA SDK 期望的 C 回调函数。

这种设计模式在前一节的 [ZIP 归档实现](./01-ZIP归档实现.md) 中也出现过——ZIP 的 minizip 库同样使用 C 回调，KrKr2 用类似的方式桥接。

### 3.2 完整代码分析

```cpp
// 来源：cpp/core/base/7zArchive.cpp 第 19-75 行
class SevenZipStreamWrap {
public:
    CSzArEx db;               // 7z 归档数据库——存储所有元数据
    tTJSBinaryStream *_stream;// KrKr2 的底层流
    CLookToRead2 lookStream;  // LZMA SDK 的缓冲读取层

    // 内嵌结构体：继承 ISeekInStream，添加 Host 指针
    struct CSeekInStream : public ISeekInStream {
        SevenZipStreamWrap *Host;
    } archiveStream;

public:
    SevenZipStreamWrap(tTJSBinaryStream *st) : _stream(st) {
        // 1. 保存 this 指针到内嵌结构体
        archiveStream.Host = this;

        // 2. 设置 Read 回调函数
        archiveStream.Read = [](ISeekInStreamPtr p, void *buf,
                                size_t *size) -> SRes {
            // 关键：通过 CSeekInStream* 找到 Host，再调用 StreamRead
            return ((CSeekInStream *)p)->Host->StreamRead(buf, size);
        };

        // 3. 设置 Seek 回调函数
        archiveStream.Seek = [](ISeekInStreamPtr p, Int64 *pos,
                                ESzSeek origin) -> SRes {
            return ((CSeekInStream *)p)->Host->StreamSeek(pos, origin);
        };

        // 4. 初始化 CLookToRead2 缓冲层
        LookToRead2_CreateVTable(&lookStream, false);
        // false 表示不分配内部缓冲区（使用默认静态缓冲）
        lookStream.realStream = &archiveStream;
        // 将缓冲层连接到我们的 ISeekInStream 实现

        // 5. 初始化 7z 数据库
        SzArEx_Init(&db);

        // 6. 初始化 CRC 表（全局操作，只需执行一次）
        if (!g_CrcTable[1])
            CrcGenerateTable();
    }

    ~SevenZipStreamWrap() {
        SzArEx_Free(&db, &allocImp);  // 释放 7z 数据库
        delete _stream;               // 释放底层流
    }

    // 读取实现：转发到 tTJSBinaryStream::Read
    SRes StreamRead(void *buf, size_t *size) {
        *size = _stream->Read(buf, *size);
        return SZ_OK;
    }

    // 定位实现：转发到 tTJSBinaryStream::Seek
    SRes StreamSeek(Int64 *pos, ESzSeek origin) {
        tjs_int whence = TJS_BS_SEEK_SET;
        switch (origin) {
            case SZ_SEEK_CUR: whence = TJS_BS_SEEK_CUR; break;
            case SZ_SEEK_END: whence = TJS_BS_SEEK_END; break;
            case SZ_SEEK_SET: whence = TJS_BS_SEEK_SET; break;
            default: break;
        }
        *pos = _stream->Seek(*pos, whence);
        return SZ_OK;
    }
};
```

### 3.3 桥接机制的关键——CSeekInStream 技巧

`CSeekInStream` 继承自 `ISeekInStream`（一个只含函数指针的 C 结构体），并在末尾添加了一个 `Host` 指针。LZMA SDK 调用 `Read`/`Seek` 时传入的第一个参数 `p` 实际上就是 `archiveStream` 的地址。通过 `(CSeekInStream *)p` 强制类型转换，就能访问到追加的 `Host` 字段，从而拿到 `SevenZipStreamWrap` 实例。

这与 ZIP 归档中将 `ZipArchive* this` 作为"文件名"传递给 minizip 回调的技巧异曲同工——都是在 C 语言的函数指针回调体系中传递 C++ 对象指针的经典方法。

```
内存布局：
┌──────────────────────────────┐
│  CSeekInStream               │
│  ┌────────────────────────┐  │
│  │ ISeekInStream 部分     │  │
│  │  .Read  = lambda ptr   │  │  ← LZMA SDK 只看到这部分
│  │  .Seek  = lambda ptr   │  │
│  └────────────────────────┘  │
│  .Host = this 指针           │  ← CSeekInStream 追加的字段
└──────────────────────────────┘
```

### 3.4 CRC 表初始化

```cpp
if (!g_CrcTable[1])
    CrcGenerateTable();
```

LZMA SDK 使用 CRC32 校验来验证 7z 头部和数据的完整性。`CrcGenerateTable()` 会填充一个全局查找表 `g_CrcTable`（256 个 `uint32_t` 值），之后 CRC32 计算只需要查表即可。

`g_CrcTable[1]` 被用作初始化标志——未初始化时为 0（全局数组默认初始化），初始化后为非零值。虽然这不是线程安全的初始化方式，但在 KrKr2 中，归档系统在主线程初始化，不存在竞争条件。

## 四、SevenZipArchive 类——归档读取实现

### 4.1 类继承结构

`SevenZipArchive` 同时继承了两个类：

```cpp
// 来源：cpp/core/base/7zArchive.cpp 第 77 行
class SevenZipArchive : public tTVPArchive, public SevenZipStreamWrap {
    std::vector<std::pair<ttstr, tjs_uint>> filelist;
    // filelist 存储 {文件名, 原始索引} 对
    // 原始索引是文件在 CSzArEx 数据库中的编号
    // 排序后需要保留原始索引才能正确调用 SzArEx_Extract
    ...
};
```

- **tTVPArchive**：KrKr2 的归档基类——提供文件名规范化、哈希索引等基础功能
- **SevenZipStreamWrap**：LZMA SDK 的 I/O 桥接层——持有 `CSzArEx` 数据库和缓冲流

这种多重继承的设计使得 `SevenZipArchive` 既是一个标准的 KrKr2 归档，又自带完整的 LZMA SDK I/O 基础设施。

### 4.2 Open() 方法——归档初始化

```cpp
// 来源：cpp/core/base/7zArchive.cpp 第 141-170 行
bool Open(bool normalizeFileName) {
    // 1. 解析 7z 文件头部和数据库
    SRes res = SzArEx_Open(&db, &lookStream.vt, &allocImp, &allocImp);
    if (res != SZ_OK) {
        _stream = nullptr;
        return false;
    }

    // 2. 遍历所有文件条目
    for (int i = 0; i < db.NumFiles; i++) {
        bool isDir = SzArEx_IsDir(&db, i);
        if (isDir) continue;  // 跳过目录条目

        // 3. 获取文件名（UTF-16 编码）
        size_t len = SzArEx_GetFileNameUtf16(&db, i, nullptr);
        // 第一次调用：len 包含 null 终止符
        ttstr filename;
        SzArEx_GetFileNameUtf16(&db, i,
                                (UInt16 *)filename.AllocBuffer(len));
        // 第二次调用：直接写入 ttstr 的缓冲区
        filename.FixLen();

        // 4. 文件名规范化
        if (normalizeFileName)
            NormalizeInArchiveStorageName(filename);

        // 5. 保存 {文件名, 原始索引} 对
        filelist.emplace_back(filename, i);
    }

    // 6. 排序以支持二分查找
    if (normalizeFileName) {
        std::sort(filelist.begin(), filelist.end(),
                  [](const auto &a, const auto &b) {
                      return a.first < b.first;
                  });
    }
    return true;
}
```

关键点：

- **`SzArEx_Open`** 是 LZMA SDK 的高层 API，它会读取 7z 文件的签名头部、解析数据库（可能本身也是压缩的）、填充 `CSzArEx` 结构体中的所有元数据
- **两次调用 `SzArEx_GetFileNameUtf16`**：第一次传 `nullptr` 获取长度，第二次传缓冲区指针获取实际数据——这是 Windows API 风格的"先查长度再读数据"模式
- **`filelist` 存储 `pair<ttstr, uint>`**：因为排序会改变元素顺序，但 `SzArEx_Extract` 需要原始的文件索引（`i`），所以必须保存映射关系

### 4.3 CreateStreamByIndex——双策略文件读取

这是 7z 归档实现中最复杂也最精妙的部分。`CreateStreamByIndex` 使用两种策略来读取文件：

**策略 1：零拷贝直接读取**（适用于单编码器 + 无压缩的情况）

```cpp
// 来源：cpp/core/base/7zArchive.cpp 第 90-120 行
tTJSBinaryStream *CreateStreamByIndex(tjs_uint idx) override {
    tjs_uint fileIndex = filelist[idx].second;  // 获取原始索引
    UInt64 fileSize = SzArEx_GetFileSize(&db, fileIndex);
    if (fileSize <= 0)
        return new tTVPMemoryStream();  // 空文件

    UInt32 folderIndex = db.FileToFolder[fileIndex];
    if (folderIndex == (UInt32)-1)
        return nullptr;  // 文件不属于任何文件夹（异常情况）

    // 读取文件夹的编码器信息
    const CSzAr *p = &db.db;
    CSzFolder folder;
    CSzData sd;
    const Byte *data = p->CodersData + p->FoCodersOffsets[folderIndex];
    sd.Data = data;
    sd.Size = p->FoCodersOffsets[folderIndex + 1] -
              p->FoCodersOffsets[folderIndex];

    if (SzGetNextFolderItem(&folder, &sd) != SZ_OK)
        return nullptr;

    // 关键优化：单编码器 + Copy 方法 + 大小一致
    if (folder.NumCoders == 1) {
        UInt64 startPos = db.dataPos;
        const UInt64 *packPositions =
            p->PackPositions + p->FoStartPackStreamIndex[folderIndex];
        UInt64 offset = packPositions[0];
        UInt64 inSize = packPositions[1] - offset;

        #define k_Copy 0  // MethodID 0 = Copy（无压缩）
        if (folder.Coders[0].MethodID == k_Copy && inSize == fileSize) {
            // 文件未压缩！直接返回偏移量包装流
            return new TArchiveStream(this, startPos + offset, inSize);
        }
    }
    // ... 策略 2 继续
```

**为什么要做这个优化？** 如果一个文件在 7z 归档中没有被压缩（MethodID = 0，即 Copy），那么它的原始数据就原封不动地存在归档文件中。这时候我们不需要解压——只需要知道数据在归档文件中的偏移和大小，用 `TArchiveStream` 包装即可。这与 TAR 归档的处理方式完全一样：零拷贝、零内存分配。

**策略 2：完整解压到内存**（适用于压缩过的文件）

```cpp
    // 来源：cpp/core/base/7zArchive.cpp 第 122-138 行
    UInt32 blockIndex;
    Byte *outBuffer = nullptr;
    size_t outBufferSize;
    size_t offset, outSizeProcessed;

    // 调用 LZMA SDK 解压整个文件夹
    SRes res = SzArEx_Extract(&db, &lookStream.vt, fileIndex,
                               &blockIndex, &outBuffer, &outBufferSize,
                               &offset, &outSizeProcessed,
                               &allocImp, &allocImp);

    tTVPMemoryStream *mem;
    if (offset == 0 && fileSize <= outBufferSize) {
        // 优化：文件是文件夹中唯一的文件（或第一个文件）
        // 直接接管 outBuffer 的所有权，避免复制
        mem = new tTVPMemoryStream(outBuffer, outBufferSize);
    } else {
        // 文件是文件夹中的某个子文件，需要从解压缓冲区中复制
        Byte *buf = new Byte[fileSize];
        memcpy(buf, outBuffer + offset, fileSize);
        mem = new tTVPMemoryStream(buf, fileSize);
        delete outBuffer;  // 释放 LZMA SDK 分配的缓冲区
    }
    return mem;
}
```

**`SzArEx_Extract` 的工作机制**：

7z 的固实压缩意味着一个文件夹（Folder）中的所有文件被合并成一个数据流后统一压缩。要读取文件夹中的任何一个文件，必须解压整个文件夹。`SzArEx_Extract` 负责这个过程：

1. 读取文件夹的压缩数据
2. 使用对应的编码器链（LZMA、BCJ2 等）解压
3. 将完整的解压结果存入 `outBuffer`
4. 通过 `offset` 和 `outSizeProcessed` 告诉调用者目标文件在 `outBuffer` 中的位置和大小

```
7z 文件夹（Folder）的解压过程：

归档文件：
[...压缩数据块（包含 File_A + File_B + File_C 的合并数据）...]

解压后的 outBuffer：
┌──────────────────────────────────────────────────────┐
│  File_A 数据  │  File_B 数据  │  File_C 数据         │
│  (10KB)       │  (20KB)       │  (15KB)              │
└──────────────────────────────────────────────────────┘
                 ↑
                 offset（如果请求 File_B）
```

### 4.4 工厂函数——签名检测

```cpp
// 来源：cpp/core/base/7zArchive.cpp 第 173-186 行
tTVPArchive *TVPOpen7ZArchive(const ttstr &name,
                               tTJSBinaryStream *st,
                               bool normalizeFileName) {
    tjs_uint64 pos = st->GetPosition();
    bool checkZIP = st->ReadI16LE() == 0x7A37; // '7z' 的小端序
    // 0x37 = '7', 0x7A = 'z' → 小端序读取 = 0x7A37
    st->SetPosition(pos);

    if (!checkZIP) return nullptr;
    // 前两字节不是 '7z'，不是 7z 文件

    SevenZipArchive *arc = new SevenZipArchive(name, st);
    if (!arc->Open(normalizeFileName)) {
        delete arc;
        return nullptr;
    }
    return arc;
}
```

7z 文件的完整签名是 6 字节：`{'7', 'z', 0xBC, 0xAF, 0x27, 0x1C}`。但 KrKr2 只检查前两字节 `'7z'`（0x37, 0x7A）作为快速筛选。完整的签名验证由 `SzArEx_Open` 内部完成——如果签名不完全匹配，`SzArEx_Open` 会返回错误。

注意变量名 `checkZIP` 是一个小笔误——实际检查的是 7z 签名，不是 ZIP。

### 5.2 模拟 SevenZipStreamWrap 适配器

以下示例展示如何自己实现 `ISeekInStream` 适配器，将标准 `FILE*` 包装为 LZMA SDK 接口：

```cpp
// custom_stream_wrap.cpp — 自定义 ISeekInStream 适配器
#include <cstdio>
#include <cstdint>
#include <cstdlib>

// 模拟 LZMA SDK 的接口定义
typedef int SRes;
#define SZ_OK 0
typedef void* ISzAllocPtr;
typedef int64_t Int64;

enum ESzSeek { SZ_SEEK_SET = 0, SZ_SEEK_CUR = 1, SZ_SEEK_END = 2 };

// ISeekInStream 接口（简化版）
struct ISeekInStream;
typedef ISeekInStream* ISeekInStreamPtr;

struct ISeekInStream {
    SRes (*Read)(ISeekInStreamPtr p, void* buf, size_t* size);
    SRes (*Seek)(ISeekInStreamPtr p, Int64* pos, ESzSeek origin);
};

// 自定义流包装器——将 FILE* 适配为 ISeekInStream
class FileStreamWrap {
    FILE* _fp;

public:
    // 继承 ISeekInStream 并追加 Host 指针
    struct MyStream : public ISeekInStream {
        FileStreamWrap* Host;
    } stream;

    FileStreamWrap(const char* path) {
        _fp = fopen(path, "rb");
        stream.Host = this;

        // 使用 lambda 设置函数指针
        stream.Read = [](ISeekInStreamPtr p, void* buf,
                         size_t* size) -> SRes {
            auto* self = ((MyStream*)p)->Host;
            *size = fread(buf, 1, *size, self->_fp);
            return SZ_OK;
        };

        stream.Seek = [](ISeekInStreamPtr p, Int64* pos,
                         ESzSeek origin) -> SRes {
            auto* self = ((MyStream*)p)->Host;
            int whence = SEEK_SET;
            switch (origin) {
                case SZ_SEEK_CUR: whence = SEEK_CUR; break;
                case SZ_SEEK_END: whence = SEEK_END; break;
                default: break;
            }
            fseek(self->_fp, (long)*pos, whence);
            *pos = ftell(self->_fp);
            return SZ_OK;
        };
    }

    ~FileStreamWrap() {
        if (_fp) fclose(_fp);
    }

    bool isOpen() const { return _fp != nullptr; }

    // 获取 ISeekInStream 指针——传递给 LZMA SDK
    ISeekInStream* getStream() { return &stream; }
};

int main() {
    // 使用示例
    FileStreamWrap wrap("test.7z");
    if (!wrap.isOpen()) {
        printf("无法打开文件\n");
        return 1;
    }

    // 通过 ISeekInStream 接口读取前 6 字节（7z 签名）
    ISeekInStream* stream = wrap.getStream();
    char signature[7] = {0};
    size_t readSize = 6;
    stream->Read(stream, signature, &readSize);

    printf("读取了 %zu 字节\n", readSize);
    printf("签名: ");
    for (int i = 0; i < 6; i++)
        printf("%02X ", (unsigned char)signature[i]);
    printf("\n");

    // 7z 签名应该是: 37 7A BC AF 27 1C
    if (signature[0] == '7' && signature[1] == 'z') {
        printf("这是一个 7z 文件！\n");
    } else {
        printf("不是 7z 文件\n");
    }

    // 测试 Seek
    Int64 seekPos = 0;
    stream->Seek(stream, &seekPos, SZ_SEEK_END);
    printf("文件大小: %lld 字节\n", (long long)seekPos);

    return 0;
}
```

## 六、常见错误与排查

### 6.1 SzArEx_Open 返回错误

**症状**：`SzArEx_Open` 返回非零错误码。

**常见错误码**：
| 错误码 | 宏名称 | 含义 |
|--------|--------|------|
| 1 | `SZ_ERROR_DATA` | 数据损坏（CRC 校验失败） |
| 2 | `SZ_ERROR_MEM` | 内存分配失败 |
| 3 | `SZ_ERROR_CRC` | CRC 校验不匹配 |
| 6 | `SZ_ERROR_INPUT_EOF` | 意外的文件结束（文件不完整） |
| 8 | `SZ_ERROR_UNSUPPORTED` | 不支持的压缩方法或加密 |
| 17 | `SZ_ERROR_ARCHIVE` | 无效的 7z 归档结构 |

**排查步骤**：
```bash
# 使用 7z 命令行工具验证归档完整性
7z t archive.7z

# 检查签名
hexdump -C archive.7z | head -1
# 应显示：37 7a bc af 27 1c ...
```

### 6.2 解压后数据为空

**症状**：`SzArEx_Extract` 返回 `SZ_OK` 但 `outSizeProcessed` 为 0。

**原因**：文件大小为 0 或 `FileToFolder` 映射无效。

**解决方案**：提前检查 `SzArEx_GetFileSize` 和 `FileToFolder[i]`，如 KrKr2 源码中所做的那样。

### 6.3 CRC 表未初始化

**症状**：`SzArEx_Open` 崩溃或返回随机错误。

**原因**：忘记调用 `CrcGenerateTable()`。

**解决方案**：确保在任何 7z 操作之前调用一次 `CrcGenerateTable()`。KrKr2 用 `if (!g_CrcTable[1])` 做守卫检查。

## 本节小结

- 7z 是高压缩比的现代归档格式，核心概念是文件夹（Folder）固实压缩和多编码器链
- KrKr2 通过 `SevenZipStreamWrap` 适配器将 `tTJSBinaryStream` 桥接到 LZMA SDK 的 C 回调接口
- 桥接的关键技巧是 `CSeekInStream` 继承 `ISeekInStream` 并追加 `Host` 指针——与 ZIP 归档的 I/O 回调技巧如出一辙
- `CLookToRead2` 缓冲层在 `ISeekInStream` 和高层 API 之间提供 16KB 预读缓冲
- `CreateStreamByIndex` 使用双策略优化：单编码器无压缩文件夹→零拷贝 `TArchiveStream`，其他→`SzArEx_Extract` 完整解压到 `tTVPMemoryStream`
- 工厂函数只检查前两字节 `'7z'`（0x37, 0x7A）作为快速筛选，完整验证由 `SzArEx_Open` 内部完成

### 题目 1：分析 SevenZipStreamWrap 的内存生命周期

阅读以下代码，回答问题：

```cpp
}
```

问题：当 `delete arc` 执行时，`tTJSBinaryStream *st` 会被释放吗？如果会，在哪个析构函数中释放？如果多重继承的析构顺序不对，可能导致什么问题？

<details>
<summary>查看答案</summary>

**会被释放。** `tTJSBinaryStream *st` 存储在 `SevenZipStreamWrap::_stream` 中。

析构顺序（C++ 多重继承析构顺序是声明的逆序）：

1. `SevenZipArchive::~SevenZipArchive()` —— 空实现（第 84 行）
2. `SevenZipStreamWrap::~SevenZipStreamWrap()` —— 调用 `SzArEx_Free(&db, &allocImp)` 释放 7z 数据库，然后 `delete _stream` 释放底层流
3. `tTVPArchive::~tTVPArchive()` —— 基类析构

关键点：`SzArEx_Free` 必须在 `delete _stream` 之前调用——因为 `SzArEx_Free` 可能还需要读取流中的数据来完成清理。由于它们在同一个析构函数中按顺序执行，这是安全的。

但如果代码写成这样就有问题：

```cpp
// 错误示例：如果 _stream 的释放在 db 清理之前
~SevenZipStreamWrap() {
    delete _stream;           // ❌ 先释放流
    SzArEx_Free(&db, &allocImp); // 此时 db 可能引用已释放的流
}
```

验证代码：

```cpp
#include <cstdio>

struct Base1 {
    ~Base1() { printf("~Base1\n"); }
};

struct Base2 {
    ~Base2() { printf("~Base2\n"); }
};

struct Derived : public Base1, public Base2 {
    ~Derived() { printf("~Derived\n"); }
};

int main() {
    Derived* d = new Derived();
    delete d;
    // 输出：
    // ~Derived
    // ~Base2（声明的逆序：先 Base2）
    // ~Base1
    return 0;
}
```

</details>

### 题目 2：实现一个可复用的 ISeekInStream 适配器模板

将 `SevenZipStreamWrap` 中的适配器技巧抽象为一个通用的模板类，使得任何具有 `Read(void*, size_t)` 和 `Seek(int64_t, int)` 方法的流对象都可以被适配为 `ISeekInStream`。

<details>
<summary>查看答案</summary>

```cpp
#include <cstdio>
#include <cstdint>
#include <cstring>

// 模拟 LZMA SDK 接口
typedef int SRes;
#define SZ_OK 0
typedef int64_t Int64;
enum ESzSeek { SZ_SEEK_SET = 0, SZ_SEEK_CUR = 1, SZ_SEEK_END = 2 };

struct ISeekInStream;
typedef ISeekInStream* ISeekInStreamPtr;
struct ISeekInStream {
    SRes (*Read)(ISeekInStreamPtr p, void* buf, size_t* size);
    SRes (*Seek)(ISeekInStreamPtr p, Int64* pos, ESzSeek origin);
};

// 通用适配器模板
// StreamT 必须提供：
//   size_t Read(void* buf, size_t size)
//   int64_t Seek(int64_t offset, int whence)
template <typename StreamT>
class SeekInStreamAdapter {
    StreamT* _stream;

    // 内嵌结构体，追加 adapter 指针
    struct AdaptedStream : public ISeekInStream {
        SeekInStreamAdapter* adapter;
    } _adapted;

public:
    explicit SeekInStreamAdapter(StreamT* stream) : _stream(stream) {
        _adapted.adapter = this;

        _adapted.Read = [](ISeekInStreamPtr p, void* buf,
                           size_t* size) -> SRes {
            auto* self = ((AdaptedStream*)p)->adapter;
            *size = self->_stream->Read(buf, *size);
            return SZ_OK;
        };

        _adapted.Seek = [](ISeekInStreamPtr p, Int64* pos,
                           ESzSeek origin) -> SRes {
            auto* self = ((AdaptedStream*)p)->adapter;
            int whence = 0; // SEEK_SET
            switch (origin) {
                case SZ_SEEK_CUR: whence = 1; break;
                case SZ_SEEK_END: whence = 2; break;
                default: break;
            }
            *pos = self->_stream->Seek(*pos, whence);
            return SZ_OK;
        };
    }

    // 获取 ISeekInStream 指针
    ISeekInStream* getInterface() { return &_adapted; }
};

// 测试用的流类——模拟内存中的数据
class MemoryStream {
    const char* _data;
    size_t _size;
    size_t _pos = 0;

public:
    MemoryStream(const char* data, size_t size)
        : _data(data), _size(size) {}

    size_t Read(void* buf, size_t size) {
        if (_pos + size > _size)
            size = _size - _pos;
        memcpy(buf, _data + _pos, size);
        _pos += size;
        return size;
    }

    int64_t Seek(int64_t offset, int whence) {
        switch (whence) {
            case 0: _pos = (size_t)offset; break;        // SET
            case 1: _pos += (size_t)offset; break;        // CUR
            case 2: _pos = _size + (size_t)offset; break; // END
        }
        if (_pos > _size) _pos = _size;
        return (int64_t)_pos;
    }
};

int main() {
    // 准备测试数据
    const char testData[] = "Hello, 7z World! This is a test stream.";
    MemoryStream memStream(testData, sizeof(testData) - 1);

    // 使用模板适配器
    SeekInStreamAdapter<MemoryStream> adapter(&memStream);
    ISeekInStream* istream = adapter.getInterface();

    // 通过 ISeekInStream 接口读取
    char buf[16] = {0};
    size_t readSize = 5;
    istream->Read(istream, buf, &readSize);
    printf("读取 %zu 字节: '%s'\n", readSize, buf);
    // 输出: 读取 5 字节: 'Hello'

    // 通过 ISeekInStream 接口 Seek
    Int64 pos = 7;
    istream->Seek(istream, &pos, SZ_SEEK_SET);
    printf("Seek 到位置 %lld\n", (long long)pos);

    readSize = 9;
    memset(buf, 0, sizeof(buf));
    istream->Read(istream, buf, &readSize);
    printf("读取 %zu 字节: '%s'\n", readSize, buf);
    // 输出: 读取 9 字节: '7z World!'

    return 0;
}
```

**要点**：模板参数 `StreamT` 使用鸭子类型（Duck Typing）——只要提供 `Read` 和 `Seek` 方法即可，不需要继承特定基类。这比 LZMA SDK 的 C 函数指针方式更类型安全，也更易于复用。

</details>

### 题目 3：设计零拷贝优化的判断逻辑

KrKr2 的 `CreateStreamByIndex` 在文件夹只有一个编码器且 MethodID 为 0（Copy）时使用零拷贝路径。但还有一种情况也可以优化：文件夹使用 LZMA 压缩但只包含一个文件。在这种情况下，虽然需要解压，但 `outBuffer` 中的数据就是目标文件的全部数据（`offset == 0`），可以直接接管 `outBuffer` 而不需要复制。请分析 KrKr2 源码中的 `offset == 0` 分支是否已经处理了这种情况。

<details>
<summary>查看答案</summary>

**是的，KrKr2 已经处理了这种情况。**

```cpp
if (offset == 0 && fileSize <= outBufferSize) {
    mem = new tTVPMemoryStream(outBuffer, outBufferSize);
} else {
    Byte *buf = new Byte[fileSize];
    memcpy(buf, outBuffer + offset, fileSize);
    mem = new tTVPMemoryStream(buf, fileSize);
    delete outBuffer;
}
```

当文件是文件夹中的第一个（或唯一一个）文件时，`offset` 为 0。此时直接将 `outBuffer` 的所有权转移给 `tTVPMemoryStream`，避免了一次内存复制（`memcpy`）。

但这里有一个细微问题：即使 `offset == 0`，`outBufferSize` 可能大于 `fileSize`（如果文件夹包含多个文件但目标文件是第一个）。这意味着 `tTVPMemoryStream` 持有的缓冲区可能比实际文件数据大。这通常不是问题，因为流的有效数据由文件大小控制，但会浪费一些内存。

更保守的做法是总是复制精确大小的数据：

```cpp
// 更精确但有额外复制开销
Byte *buf = new Byte[fileSize];
memcpy(buf, outBuffer + offset, fileSize);
mem = new tTVPMemoryStream(buf, fileSize);
free(outBuffer);
```

KrKr2 选择了性能优先的方案——对于单文件文件夹的常见情况，避免复制比节省少量内存更有价值。

</details>
