## 一、文本流系统架构

### 1.1 为什么需要文本流

KrKr2 引擎使用 TJS2 脚本语言驱动游戏逻辑，脚本文件（`.tjs`）和场景文件（`.ks`）本质上都是文本文件。然而，这些文本文件面临多重复杂性：

```
┌─────────────────────────────────────────────────────────┐
│                    文本文件的复杂性                        │
├─────────────────────────────────────────────────────────┤
│ ① 编码多样：UTF-8 / UTF-16LE / UTF-16BE / Shift_JIS     │
│ ② 可能加密：防止玩家直接查看脚本源码                       │
│ ③ 可能压缩：减小文件体积                                  │
│ ④ BOM 可选：有些文件有 BOM，有些没有                       │
│ ⑤ 平台差异：Windows 用 UTF-16LE，其他平台可能用 UTF-8     │
└─────────────────────────────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│              tTVPTextReadStream                          │
│                                                         │
│  输入：任意编码/加密/压缩的文本文件                        │
│  输出：统一的 UTF-16 字符串（std::u16string）             │
│                                                         │
│  流程：                                                  │
│  文件 → 二进制读取 → 检测加密 → 解密/解压                  │
│       → 检测编码 → 转换为 UTF-16 → 存入内部缓冲区         │
└─────────────────────────────────────────────────────────┘
```

### 1.2 接口层次

文本流建立在二进制流之上，形成两层抽象：

```
    应用层（TJS2 脚本引擎）
         │
         │  调用 Read(tTJSString&, size)
         ▼
    iTJSTextReadStream / iTJSTextWriteStream  ← 文本流接口
         │
         │  内部使用
         ▼
    tTJSBinaryStream                          ← 二进制流接口
         │
         │  实际读写
         ▼
    文件系统 / 归档 / 网络
```

文本流接口定义在 TJS2 核心头文件中（不在 `TextStream.h`），只有两个方法：

```cpp
// TJS2 文本读取流接口
class iTJSTextReadStream {
public:
    virtual tjs_uint Read(tTJSString &targ, tjs_uint size) = 0;
    virtual void Destruct() = 0;  // 代替 delete（允许自定义销毁）
    virtual ~iTJSTextReadStream() = default;
};

// TJS2 文本写入流接口
class iTJSTextWriteStream {
public:
    virtual void Write(const ttstr &targ) = 0;
    virtual void Destruct() = 0;
    virtual ~iTJSTextWriteStream() = default;
};
```

工厂函数在 `TextStream.h` 中声明：

```cpp
// 源码：cpp/core/base/TextStream.h 第 4-7 行
iTJSTextReadStream *TVPCreateTextStreamForRead(const ttstr &name,
                                               const ttstr &mode);
iTJSTextWriteStream *TVPCreateTextStreamForWrite(const ttstr &name,
                                                 const ttstr &mode);
```

## 二、编码检测：checkTextEncoding

### 2.1 BOM 检测

BOM（Byte Order Mark，字节序标记）是文件开头的特殊字节序列，用于标识文本的编码方式。checkTextEncoding 函数首先检查 BOM：

```cpp
// 源码：cpp/core/base/TextStream.cpp 第 19-61 行
std::string checkTextEncoding(const void *buf, size_t size,
                              std::uint8_t &bomSize) {
    auto raw = static_cast<const unsigned char *>(buf);
    std::string encoding;

    // --- BOM 检测（按字节序列匹配）---
    if(size >= 2 && raw[0] == 0xFF && raw[1] == 0xFE) {
        bomSize = 2;
        encoding = "UTF-16LE";       // FF FE → 小端序 UTF-16
    } else if(size >= 2 && raw[0] == 0xFE && raw[1] == 0xFF) {
        bomSize = 2;
        encoding = "UTF-16BE";       // FE FF → 大端序 UTF-16
    } else if(size >= 3 && raw[0] == 0xEF && raw[1] == 0xBB && raw[2] == 0xBF) {
        bomSize = 3;
        encoding = "UTF-8";          // EF BB BF → UTF-8 BOM
    } else if(size >= 4 && raw[0] == 0xFF && raw[1] == 0xFE
              && raw[2] == 0x00 && raw[3] == 0x00) {
        bomSize = 4;
        encoding = "UTF-32LE";       // FF FE 00 00 → 小端序 UTF-32
    } else if(size >= 4 && raw[0] == 0x00 && raw[1] == 0x00
              && raw[2] == 0xFE && raw[3] == 0xFF) {
        bomSize = 4;
        encoding = "UTF-32BE";       // 00 00 FE FF → 大端序 UTF-32
    }
    // ...
}
```

**BOM 字节序列速查表：**

| 编码 | BOM 字节 | 大小 | 说明 |
|------|----------|------|------|
| UTF-8 | `EF BB BF` | 3 字节 | 可选，很多 UTF-8 文件不带 BOM |
| UTF-16LE | `FF FE` | 2 字节 | Windows 默认的 Unicode 编码 |
| UTF-16BE | `FE FF` | 2 字节 | 网络字节序的 Unicode |
| UTF-32LE | `FF FE 00 00` | 4 字节 | 罕见，但标准中定义了 |
| UTF-32BE | `00 00 FE FF` | 4 字节 | 罕见 |

**注意检测顺序的微妙之处**：UTF-32LE 的 BOM 以 `FF FE` 开头，与 UTF-16LE 的 BOM 完全相同。源码中先检查 2 字节的 UTF-16LE，后检查 4 字节的 UTF-32LE。这意味着如果一个 UTF-32LE 文件的前 4 字节恰好是 `FF FE 00 00`，它会被**错误地**识别为 UTF-16LE（bomSize=2），因为 2 字节检查先匹配了。不过实际上这种情况极少发生，因为 KrKr2 游戏脚本几乎不使用 UTF-32 编码。

### 2.2 uchardet 自动检测

当文件没有 BOM 时，使用 uchardet 库进行统计分析：

```cpp
// 源码：cpp/core/base/TextStream.cpp 第 46-58 行（续）
    } else {
        // 无 BOM：使用 uchardet 自动检测编码
        uchardet_t ud = uchardet_new();       // 创建检测器实例
        uchardet_handle_data(ud,              // 输入待检测数据
            reinterpret_cast<const char *>(raw), size);
        uchardet_data_end(ud);                // 通知检测器数据输入完毕
        encoding = uchardet_get_charset(ud);  // 获取检测结果
        uchardet_delete(ud);                  // 释放检测器

        // 编码名称标准化
        if(encoding == "SHIFT_JIS") {
            encoding = "cp932";       // Shift_JIS → Windows Code Page 932
        } else if(encoding == "WINDOWS-1252") {
            encoding = "ASCII";       // 西欧编码当作 ASCII 处理
        }
    }
```

**uchardet 工作原理**：uchardet（Universal Charset Detector）源自 Mozilla Firefox 的编码检测代码。它通过分析字节序列的统计特征来推断编码：

1. **字节频率分析**：不同编码的文本文件中，各字节值出现的频率不同。例如，Shift_JIS 的日文文本中 `0x82-0x9F` 和 `0xE0-0xEF` 范围的字节频率很高
2. **序列模型**：相邻字节的组合模式在不同编码中有不同的统计分布
3. **置信度评分**：uchardet 为每种候选编码计算置信度分数，返回最高分的编码

**SHIFT_JIS → cp932 映射**：Shift_JIS 是日文编码标准（JIS X 0208），而 cp932 是微软对 Shift_JIS 的扩展版本，包含了 NEC 特殊字符和 IBM 扩展字符。KrKr2 作为 Windows 原生引擎，使用 cp932 是更准确的选择。

### 2.3 默认编码回退

```cpp
// 源码：cpp/core/base/TextStream.cpp 第 17 行
static std::string G_DefaultReadEncoding = "UTF-8";  // 全局默认编码

// 第 417-429 行：设置默认编码
void TVPSetDefaultReadEncoding(const ttstr &encoding) {
    ttstr codestr = encoding;
    codestr.ToLowerCase();
    if(codestr == TJS_W("sjis") || codestr == TJS_W("shiftjis") ||
       codestr == TJS_W("shift_jis") || codestr == TJS_W("shift-jis")) {
        G_DefaultReadEncoding = "cp932";      // 各种 Shift_JIS 变体名 → cp932
    } else if(codestr == TJS_W("utf8") || codestr == TJS_W("utf-8")) {
        G_DefaultReadEncoding = "UTF-8";
    } else {
        G_DefaultReadEncoding = encoding.AsStdString();  // 直接使用
    }
}
```

当 uchardet 返回空字符串（无法识别编码）时，使用 `G_DefaultReadEncoding` 作为回退。游戏开发者可以通过 TJS2 脚本调用 `TVPSetDefaultReadEncoding` 来设置默认编码——这在日文游戏中通常设为 `"sjis"`。

## 三、tTVPTextReadStream 详解

### 3.1 构造流程总览

tTVPTextReadStream 的构造函数是整个文本读取系统的核心——它在构造时就完成了所有解码工作，将结果存入内部 `_buffer`（`std::u16string`）。之后的 Read 调用只是从缓冲区中取数据。

```
构造函数执行流程：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
① 打开二进制流，读取全部原始字节 → raw[]
                    │
② 检查前 2 字节是否为 0xFE 0xFE （加密标记）
       │                    │
    是加密               不是加密
       │                    │
③ 读取加密模式字节 m        ⑥ checkTextEncoding()
       │                         检测编码
  ┌────┼────┐                     │
  m=0  m=1  m=2             ┌─────┼─────┐─────┐
  │    │    │              UTF-8  UTF-16  其他编码
  │    │    │               │      │       │
④ XOR  位交换 zlib解压   boost::locale 转换
  解密  解密    │            │      │       │
  │    │    │              ▼      ▼       ▼
  └────┴────┴──────→  _buffer (std::u16string)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 3.2 加密检测与解密

KrKr2 的文本加密通过文件头部的 magic bytes 标识：

```
加密文件格式：
┌──────────────┬─────┬──────┬──────────────────────┐
│ 0xFE  0xFE  │  m  │ 0x00 │  UTF-16LE 密文数据    │
│  (2字节)     │(1B) │(1B)  │  (size-4)/2 个字符    │
└──────────────┴─────┴──────┴──────────────────────┘
    magic         mode  pad     加密后的 UTF-16 数据
```

#### 模式 0：XOR 加密

```cpp
// 源码：cpp/core/base/TextStream.cpp 第 93-95 行
if(m == 0) {
    if(ch >= 0x20)  // 只加密可打印字符（跳过控制字符）
        ch ^= (((ch & 0xfe) << 8) ^ 1);
}
```

**算法分析**：
- `ch & 0xfe`：取字符的低 8 位中除最低位外的所有位
- `<< 8`：左移 8 位到高字节
- `^ 1`：翻转最低位
- 最终：`ch ^= result`——用计算出的掩码对字符进行 XOR

这个加密是**自反**的（self-reciprocal）：对同一字符执行两次相同操作会得到原始值，因为 XOR 运算满足 `a ^ b ^ b = a`。

```
示例：字符 'A' (0x0041)
  ch & 0xfe     = 0x0040
  << 8          = 0x4000
  ^ 1           = 0x4001
  ch ^ 0x4001   = 0x0041 ^ 0x4001 = 0x4040  ← 加密结果

验证解密：
  0x4040 & 0xfe = 0x0040
  << 8          = 0x4000
  ^ 1           = 0x4001
  0x4040 ^ 0x4001 = 0x0041 = 'A'  ✓ 恢复原文
```

#### 模式 1：位交换加密

```cpp
// 源码：cpp/core/base/TextStream.cpp 第 96-98 行
if(m == 1) {
    ch = ((ch & 0xaaaaaaaa) >> 1) | ((ch & 0x55555555) << 1);
}
```

**算法分析**：
- `0xAAAAAAAA` 的二进制是 `1010 1010 ...`——选出所有偶数位
- `0x55555555` 的二进制是 `0101 0101 ...`——选出所有奇数位
- `>> 1` 和 `<< 1`：将偶数位右移、奇数位左移——实际效果是相邻位互换

```
示例：字符 'H' (0x0048 = 0000 0000 0100 1000)

  偶数位 (& 0xAAAA): 0000 0000 0000 1000  → >> 1: 0000 0000 0000 0100
  奇数位 (& 0x5555): 0000 0000 0100 0000  → << 1: 0000 0000 1000 0000
  OR 合并:                                         0000 0000 1000 0100 = 0x0084

验证解密（同一操作再执行一次）：
  偶数位 (& 0xAAAA): 0000 0000 1000 0000  → >> 1: 0000 0000 0100 0000
  奇数位 (& 0x5555): 0000 0000 0000 0100  → << 1: 0000 0000 0000 1000
  OR 合并:                                         0000 0000 0100 1000 = 0x0048 = 'H' ✓
```

位交换加密同样是**自反**的——执行两次位交换恢复原始值。

#### 模式 2：zlib 压缩

```cpp
// 源码：cpp/core/base/TextStream.cpp 第 104-133 行
if(m == 2) {
    // 压缩格式头部：
    // [0xFE][0xFE][0x02][0x00][BOM]
    // [compressed_size: uint64][uncompressed_size: uint64]
    // [compressed_data...]

    if(size < 3 + 2 + 16)  // 最小：3(magic) + 2(BOM) + 16(两个uint64)
        TVPThrowExceptionMessage(TVPUnsupportedCipherMode, name);

    std::uint8_t *ptr = raw.data() + 5;  // 跳过 magic(3) + BOM(2)
    std::uint64_t compressed = *reinterpret_cast<std::uint64_t *>(ptr);
    ptr += 8;
    std::uint64_t uncompressed = *reinterpret_cast<std::uint64_t *>(ptr);
    ptr += 8;

    // zlib 解压
    std::vector<std::uint8_t> compBuf(compressed);
    memcpy(compBuf.data(), ptr, compressed);
    std::vector<std::uint8_t> uncompBuf(uncompressed);
    auto destLen = static_cast<unsigned long>(uncompressed);
    int ret = uncompress(uncompBuf.data(), &destLen,
                         compBuf.data(), static_cast<unsigned long>(compressed));
    if(ret != Z_OK || destLen != uncompressed)
        TVPThrowExceptionMessage(TVPUnsupportedCipherMode, name);

    // 解压后的数据是 UTF-16 文本
    _buffer.assign(
        reinterpret_cast<char16_t *>(uncompBuf.data()),
        reinterpret_cast<char16_t *>(uncompBuf.data() + uncompressed));
}
```

**压缩文件格式详解：**

```
偏移    大小    内容
─────────────────────────────────────────
 0      1      0xFE（magic 第 1 字节）
 1      1      0xFE（magic 第 2 字节）
 2      1      0x02（mode = 压缩）
 3      1      0xFF（BOM 第 1 字节）
 4      1      0xFE（BOM 第 2 字节）
 5      8      compressed_size（小端序 uint64）
13      8      uncompressed_size（小端序 uint64）
21      N      zlib 压缩数据（N = compressed_size）
─────────────────────────────────────────
```

### 3.3 编码转换分支

解密/解压处理完毕后（或者文件本身未加密），构造函数进入编码转换阶段。首先调用 `checkTextEncoding` 检测编码，然后根据检测结果选择不同的转换路径：

```cpp
// 源码：cpp/core/base/TextStream.cpp 第 136-193 行
std::uint8_t bomSize = 0;
std::string encoding = checkTextEncoding(raw.data(), size, bomSize);
raw.erase(raw.begin(), raw.begin() + bomSize);  // 跳过 BOM 字节

if(encoding.empty())
    encoding = G_DefaultReadEncoding;  // 空字符串 → 使用默认编码
```

#### ASCII 分支

最简单的情况——每个字节直接赋值给 `char16_t`（零扩展，即高 8 位补零）：

```cpp
// 源码：cpp/core/base/TextStream.cpp 第 143-146 行
if(encoding == "ASCII") {
    _buffer.assign(raw.data(), raw.data() + size);
    // std::u16string::assign 接受 uint8_t 迭代器
    // 每个 uint8_t 隐式转换为 char16_t（高位补零）
    return;
}
```

**为什么可以直接赋值**：ASCII 码值范围 0x00-0x7F，与 UTF-16 的 BMP（基本多语言平面，Basic Multilingual Plane，Unicode 前 65536 个码点所在的区域）中 0x0000-0x007F 完全一致，无需任何转换。

#### UTF-8 分支

使用 boost::locale::conv 库将 UTF-8 字节序列转换为 UTF-16：

```cpp
// 源码：cpp/core/base/TextStream.cpp 第 148-153 行
if(encoding == "UTF-8") {
    _buffer = boost::locale::conv::utf_to_utf<char16_t>(
        reinterpret_cast<const char *>(raw.data()),
        reinterpret_cast<const char *>(raw.data() + size));
    return;
}
```

**boost::locale::conv::utf_to_utf** 是一个模板函数，它在不同 UTF 编码之间转换。`<char16_t>` 模板参数指定目标类型为 UTF-16，输入参数 `const char *` 表示源数据是 UTF-8（char 即 8 位）。

#### UTF-16 分支

UTF-16LE 可以直接解释为 `char16_t` 数组（小端序机器上字节序一致），UTF-16BE 需要交换每个字符的高低字节：

```cpp
// 源码：cpp/core/base/TextStream.cpp 第 155-172 行
if(encoding == "UTF-16" || encoding == "UTF-16LE" ||
   encoding == "UTF-16BE") {
    _buffer.assign(
        reinterpret_cast<const char16_t *>(raw.data()),
        reinterpret_cast<const char16_t *>(raw.data() + raw.size()));

    if(encoding == "UTF-16BE") {
        // 大端序 → 小端序：交换每个 char16_t 的高低字节
        size_t len = raw.size() / 2;
        _buffer.resize(len);
        auto src = reinterpret_cast<const char16_t *>(raw.data());
        for(size_t i = 0; i < len; i++) {
            char16_t ch = src[i];
            _buffer[i] = (ch >> 8) | (ch << 8);  // 字节序交换
        }
    }
    return;
}
```

**字节序交换示例**：字符 'A' 在 UTF-16BE 中存储为 `00 41`，在 UTF-16LE 中存储为 `41 00`。`(ch >> 8) | (ch << 8)` 将 `0x0041` 变为 `(0x00) | (0x4100)` → 需要注意这里有个隐含的 16 位截断。实际上 `char16_t` 是 16 位类型，所以 `ch << 8` 会自动截断高位，结果正确。

#### UTF-32 分支

UTF-32 每个码点占 4 字节，需要转换为 UTF-16（BMP 范围内码点直接截断为 16 位，辅助平面码点需要代理对）：

```cpp
// 源码：cpp/core/base/TextStream.cpp 第 174-180 行
if(encoding == "UTF-32" || encoding == "UTF-32LE" ||
   encoding == "UTF-32BE") {
    _buffer = boost::locale::conv::utf_to_utf<char16_t>(
        reinterpret_cast<const char32_t *>(raw.data()),
        reinterpret_cast<const char32_t *>(raw.data() + size));
    return;
}
```

#### 其他编码分支（cp932、EUC-JP 等）

对于非 Unicode 编码，需要两步转换：先转为 `wchar_t`（宽字符），再转为 `char16_t`：

```cpp
// 源码：cpp/core/base/TextStream.cpp 第 182-193 行
try {
    // 第一步：编码 → wchar_t（通过 boost::locale 的 ICU 后端）
    std::wstring wide = boost::locale::conv::to_utf<wchar_t>(
        reinterpret_cast<const char *>(raw.data()),
        reinterpret_cast<const char *>(raw.data() + raw.size()),
        encoding);
    // 第二步：wchar_t → char16_t
    _buffer = boost::locale::conv::utf_to_utf<char16_t>(wide);
} catch(const std::exception &e) {
    spdlog::error(e.what());
    TVPThrowExceptionMessage(TJSNarrowToWideConversionError);
}
```

**为什么需要两步**：`boost::locale::conv::to_utf` 只能转换到 `wchar_t`/`char`/`char16_t`/`char32_t` 这几种标准字符类型，且从任意编码名称到 UTF 的转换只支持 `wchar_t` 作为中间表示。`wchar_t` 在 Windows 上是 16 位（等同于 UTF-16），在 Linux/macOS 上是 32 位（等同于 UTF-32），所以第二步 `utf_to_utf<char16_t>` 在跨平台场景下是必要的。

### 3.4 Read 方法

构造函数完成后，所有文本已经在 `_buffer`（`std::u16string`）中。Read 方法只需从缓冲区中切片返回：

```cpp
// 源码：cpp/core/base/TextStream.cpp 第 197-212 行
tjs_uint Read(tTJSString &targ, tjs_uint size) override {
    static_assert(sizeof(tjs_char) == sizeof(char16_t),
                  "Char size mismatch");
    if(_pos >= _buffer.size()) {
        targ.Clear();     // 已读完 → 返回空字符串
        return 0;
    }
    size_t remain = _buffer.size() - _pos;
    size_t n = size ? size : remain;  // size=0 → 读取全部剩余
    tjs_char *buf = targ.AllocBuffer(n);  // 分配目标缓冲区
    std::copy_n(_buffer.data() + _pos, n, buf);  // 复制数据
    buf[n] = 0;           // null 终止
    _pos += n;            // 前进读取位置
    targ.FixLen();        // 修正字符串长度
    return n;
}
```

**关键设计**：
- `static_assert` 确保编译时 `tjs_char` 与 `char16_t` 大小一致（均为 2 字节），否则内存复制会出错
- `size=0` 的语义是"读取全部剩余内容"，这是 TJS2 脚本引擎的惯例
- `AllocBuffer` 分配的缓冲区大小是 `n+1`（包含 null 终止符），所以 `buf[n] = 0` 不会越界

## 四、tTVPTextWriteStream 详解

tTVPTextWriteStream 是文本写入流，支持明文、位交换加密（mode 1）和 zlib 压缩（mode 2）三种写入模式。

### 4.1 类成员与常量

```cpp
// 源码：cpp/core/base/TextStream.cpp 第 218-236 行
class tTVPTextWriteStream : public iTJSTextWriteStream {
    static constexpr size_t COMPRESSION_BUFFER_SIZE = 1024 * 1024;  // 1MB 压缩缓冲区

    std::unique_ptr<tTJSBinaryStream> _stream{};   // 底层二进制流
    tjs_int _cryptMode{};                            // 加密模式：-1/1/2
    int _compressionLevel{};                         // zlib 压缩级别
    std::unique_ptr<z_stream_s> _zStream{};          // zlib 压缩流状态
    tjs_uint _compressionSizePosition{ 0 };          // 压缩大小字段的文件偏移
    std::vector<Bytef> _compressionBuffer =
        std::vector<Bytef>(COMPRESSION_BUFFER_SIZE); // 压缩输出缓冲区
    bool _compressionFailed{ false };                // 压缩是否已失败
};
```

**`_cryptMode` 取值说明**：
| 值 | 含义 | 源码注释 |
|----|------|----------|
| -1 | 不加密（明文写入） | `no-crypt` |
| 0  | 未使用 | `old buggy crypt mode`（旧版有 bug 的加密，已废弃） |
| 1  | 位交换加密 | `simple crypt` |
| 2  | zlib 压缩 | `compressed` |

### 4.2 构造函数：模式解析

构造函数通过 `parseModeNumber` 解析模式字符串中的各种标记：

```cpp
// 源码：cpp/core/base/TextStream.cpp 第 238-303 行
tTVPTextWriteStream(const ttstr &name, const ttstr &mode) {
    // 模式字符串支持：
    // cN: 加密模式 N（当前 N 被忽略，固定使用 mode 1）
    // zN: 压缩模式，N 是压缩级别
    // oN: 从字节偏移 N 处开始写入

    // 解析 'c' 模式（加密）
    _cryptMode = parseModeNumber(mode.c_str(), TJS_W('c'), 1, -1)
                     .value_or(1);
    // parseModeNumber 返回 optional<int>
    // - 模式字符串中没有 'c' → 返回 nullopt，value_or(1) → _cryptMode=1？
    //   不对——让我们仔细看：
    //   parseModeNumber(mode, 'c', maxDigits=1, defaultValue=-1)
    //   如果 'c' 不存在：返回 optional(defaultValue=-1)
    //   如果 'c' 存在但无数字：返回 optional(defaultValue=-1)?
    //   .value_or(1) 只在 nullopt 时生效

    // 解析 'z' 模式（压缩）
    if(auto z = parseModeNumber(mode.c_str(), TJS_W('z'), 1,
                                Z_DEFAULT_COMPRESSION)) {
        _compressionLevel = z.value();
    } else {
        _cryptMode = 2;  // 有 'z' 标记但无值 → 启用压缩模式
    }

    // 合法性检查
    if(_cryptMode != -1 && _cryptMode != 1 && _cryptMode != 2)
        TVPThrowExceptionMessage(TVPUnsupportedModeString,
                                 TJS_W("unsupported cipher mode"));
```

**模式字符串示例**：

| 模式字符串 | 效果 |
|-----------|------|
| `""` | 默认：_cryptMode=-1（明文写入） |
| `"c"` | 启用位交换加密（mode 1） |
| `"z6"` | 启用 zlib 压缩，级别 6 |
| `"z"` | 启用 zlib 压缩，使用默认压缩级别 |
| `"o100"` | 从文件偏移 100 字节处开始写入 |

### 4.3 文件头部写入

```cpp
    // 偏移模式：从指定位置开始写入（UPDATE 模式）
    int ofs = parseModeNumber(mode.c_str(), TJS_W('o'), 255, 0).value();
    if(ofs != 0) {
        _stream.reset(TVPCreateStream(name, TJS_BS_UPDATE));
        _stream->SetPosition(ofs);
    } else {
        _stream.reset(TVPCreateStream(name, TJS_BS_WRITE));
    }

    // 写入加密标记（3 字节：0xFE 0xFE mode）

        crypt_mode_sig[0] = crypt_mode_sig[1] = 0xfe;  // magic
        crypt_mode_sig[2] = static_cast<tjs_uint8>(_cryptMode);
        crypt_mode_sig[3] = 0;  // 预留
        _stream->WriteBuffer(crypt_mode_sig, 3);  // 注意：只写 3 字节
    }

    // 写入 BOM（UTF-16LE 标记）
    static tjs_uint8 bommark[2] = { 0xff, 0xfe };
    _stream->WriteBuffer(bommark, 2);
```

**写入的文件头结构**：

```
明文模式（_cryptMode == -1）：
┌──────────────────────────────┐
│ 0xFF 0xFE (BOM)  │ UTF-16数据 │
└──────────────────────────────┘

加密模式（_cryptMode == 1 或 2）：
┌──────────────────────────────────────────────┐
│ 0xFE 0xFE mode │ 0xFF 0xFE (BOM) │ 加密数据  │
└──────────────────────────────────────────────┘
```

### 4.4 zlib 流初始化

压缩模式（mode 2）下，构造函数还需要初始化 zlib 压缩流并预留大小字段：

```cpp
    if(_cryptMode == 2) {
        _zStream.reset(new z_stream_s());  // 分配 zlib 状态结构
        _zStream->zalloc = Z_NULL;
        _zStream->zfree = Z_NULL;
        _zStream->opaque = Z_NULL;
        if(deflateInit(_zStream.get(), _compressionLevel) != Z_OK) {
            _compressionFailed = true;
            TVPThrowExceptionMessage(TVPCompressionFailed);
        }

        _zStream->next_in = nullptr;
        _zStream->avail_in = 0;
        _zStream->next_out = _compressionBuffer.data();
        _zStream->avail_out = COMPRESSION_BUFFER_SIZE;

        // 预留两个 uint64 字段的位置（后续析构时回填）
        _compressionSizePosition =
            static_cast<tjs_uint>(_stream->GetPosition());

        WriteI64LE(0);   // uncompressed_size 占位
    }
```

**关键设计**：压缩大小和未压缩大小只有在所有数据写完后才能确定。因此构造函数先写两个 0 占位，析构函数中回填实际值。这种"先占位后回填"的模式在二进制文件格式中非常常见。

### 4.5 Write 方法：加密写入

```cpp
// 源码：cpp/core/base/TextStream.cpp 第 353-378 行
void Write(const ttstr &targ) override {
    tjs_int len = targ.GetLen();
    auto buf = std::make_unique<tjs_uint16[]>(len + 1);
    const tjs_char *src = targ.c_str();
    tjs_int i;
    // 将 tjs_char（可能是 16 位或 32 位）转换为 uint16
    for(i = 0; i < len; i++) {
        buf[i] = src[i];
    }
    buf[i] = 0;

    if(_cryptMode == 1) {
        // 位交换加密（与读取时相同的算法——自反操作）
        if(tjs_uint16 *p = buf.get()) {
            while(*p) {
                tjs_char ch = *p;
                ch = (ch & 0xaaaaaaaa) >> 1 | (ch & 0x55555555) << 1;
                *p = ch;
                p++;
            }
        }
        WriteRawData(buf.get(), len * sizeof(tjs_uint16));
    } else {
        WriteRawData(buf.get(), len * sizeof(tjs_uint16));
    }
}
```

**注意**：Write 方法中只实现了 mode 1（位交换）加密。mode 0（XOR 加密）**不在写入端实现**——这与 `_cryptMode` 成员的注释一致：`0: (unused) (old buggy crypt mode)`。也就是说，mode 0 只存在于旧版 KiriKiri 生成的文件中，KrKr2 的写入端已不再支持生成 mode 0 加密文件。

### 4.6 WriteRawData：直接写入与压缩写入

```cpp
// 源码：cpp/core/base/TextStream.cpp 第 380-402 行
void WriteRawData(void *ptr, size_t size) {
    if(_cryptMode == 2) {
        // zlib 压缩路径
        _zStream->next_in = static_cast<Bytef *>(ptr);
        _zStream->avail_in = size;

        while(_zStream->avail_in > 0) {
            int result = deflate(_zStream.get(), Z_NO_FLUSH);
            if(result != Z_OK) {
                _compressionFailed = true;
                TVPThrowExceptionMessage(TVPCompressionFailed);
            }
            if(_zStream->avail_out == 0) {
                // 输出缓冲区满 → 刷新到文件
                _stream->WriteBuffer(_compressionBuffer.data(),
                                     COMPRESSION_BUFFER_SIZE);
                _zStream->next_out = _compressionBuffer.data();
                _zStream->avail_out = COMPRESSION_BUFFER_SIZE;
            }
        }
    } else {
        _stream->WriteBuffer(ptr, size);  // 直接写入（明文或已加密数据）
    }
}
```

**zlib 压缩流程图**：

```
┌─────────────────────────────────────────────────────┐
│             WriteRawData 压缩路径                    │
├─────────────────────────────────────────────────────┤
│                                                     │
│   输入数据 → next_in/avail_in                       │
│        │                                            │
│        ▼                                            │
│   ┌──────────┐                                      │
│   │ deflate  │ ← Z_NO_FLUSH（不强制刷新）            │
│   │ (压缩)    │                                      │
│   └────┬─────┘                                      │
│        │ 压缩后数据 → compressionBuffer              │
│        │                                            │
│        ├── avail_out > 0 → 继续（缓冲区未满）        │
│        │                                            │
│        └── avail_out == 0 → 缓冲区满                │
│                │                                    │
│                ▼                                    │
│         WriteBuffer 到文件                           │
│         重置 next_out/avail_out                      │
│                │                                    │
│                └── 继续 while 循环                   │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 4.7 析构函数：压缩流收尾

析构函数是 tTVPTextWriteStream 中最复杂的部分——它负责完成 zlib 压缩流的最终化并回填文件头中的大小字段：

```cpp
// 源码：cpp/core/base/TextStream.cpp 第 306-341 行
~tTVPTextWriteStream() override {
    if(_cryptMode == 2) {
        if(!_compressionFailed) {
            try {
                // 1. 用 Z_FINISH 标志完成压缩流
                int result = 0;
                do {
                    result = deflate(_zStream.get(), Z_FINISH);
                    if(result != Z_OK && result != Z_STREAM_END)
                        TVPThrowExceptionMessage(TVPCompressionFailed);

                    // 刷新剩余压缩数据到文件
                    _stream->WriteBuffer(_compressionBuffer.data(),
                        COMPRESSION_BUFFER_SIZE - _zStream->avail_out);
                    _zStream->next_out = _compressionBuffer.data();
                    _zStream->avail_out = COMPRESSION_BUFFER_SIZE;
                } while(result != Z_STREAM_END);

                // 2. 回填压缩/未压缩大小
                _stream->SetPosition(_compressionSizePosition);
                WriteI64LE(_zStream->total_out);  // 实际压缩后大小
                WriteI64LE(_zStream->total_in);   // 原始未压缩大小
            } catch(...) {
                if(_zStream) deflateEnd(_zStream.get());
                throw;
            }
        }
        if(_zStream) deflateEnd(_zStream.get());  // 释放 zlib 资源
    }
}
```

**析构函数中的关键操作**：

1. **Z_FINISH 循环**：调用 `deflate(Z_FINISH)` 直到返回 `Z_STREAM_END`，确保所有缓冲的压缩数据都被输出
2. **回填大小字段**：`SetPosition` 跳回构造函数中预留的位置，写入实际的 `total_out`（压缩后大小）和 `total_in`（压缩前大小）
3. **异常安全**：即使回填过程抛出异常，也会在 catch 块中调用 `deflateEnd` 释放 zlib 资源

**WriteI64LE 辅助方法**：

```cpp
// 源码：cpp/core/base/TextStream.cpp 第 344-351 行
void WriteI64LE(tjs_uint64 v) {
    tjs_uint8 buf[8];
    for(int i = 0; i < 8; i++) {
        buf[i] = static_cast<tjs_uint8>(v >> (i * 8));
    }
    _stream->WriteBuffer(buf, 8);
}
```

手工将 64 位整数分解为 8 个字节，按小端序（Little-Endian，低字节在前的存储方式）写入——不依赖平台字节序，保证跨平台兼容。

## 五、CharacterSet 模块

CharacterSet 模块提供手工实现的 UTF-8 ↔ 宽字符（`tjs_char`）转换函数。这些函数没有使用任何第三方库，而是直接按 UTF-8 编码规则逐字节处理。

### 5.1 UTF-8 编码原理

UTF-8（8-bit Unicode Transformation Format，一种将 Unicode 码点编码为 1-4 字节变长序列的方式）的编码规则：

```
码点范围              字节数    编码格式
──────────────────────────────────────────────
U+0000 - U+007F      1字节    0xxxxxxx
U+0080 - U+07FF      2字节    110xxxxx 10xxxxxx
U+0800 - U+FFFF      3字节    1110xxxx 10xxxxxx 10xxxxxx
U+10000 - U+1FFFFF   4字节    11110xxx 10xxxxxx 10xxxxxx 10xxxxxx
U+200000 - U+3FFFFFF 5字节    111110xx 10xxxxxx × 4    ← 非标准
U+4000000 - U+7FFFFFFF 6字节  1111110x 10xxxxxx × 5   ← 非标准
```

**注意**：UTF-8 标准（RFC 3629）只定义了 1-4 字节编码，覆盖到 U+10FFFF。5-6 字节序列是原始 UTF-8 提案（RFC 2279）中的内容，后来被废弃。但 KrKr2 的解码器仍然支持 5-6 字节序列，这是为了兼容性。

### 5.2 TVPWideCharToUtf8：宽字符→UTF-8

```cpp
// 源码：cpp/core/base/CharacterSet.cpp 第 17-75 行
static tjs_int inline TVPWideCharToUtf8(tjs_char in, char *out) {
    if(in < (1 << 7)) {         // U+0000 - U+007F → 1字节
        if(out) out[0] = (char)in;
        return 1;
    } else if(in < (1 << 11)) { // U+0080 - U+07FF → 2字节
        if(out) {
            out[0] = (char)(0xc0 | (in >> 6));    // 110xxxxx
            out[1] = (char)(0x80 | (in & 0x3f));  // 10xxxxxx
        }
        return 2;
    } else if(in < (1 << 16)) { // U+0800 - U+FFFF → 3字节
        if(out) {
            out[0] = (char)(0xe0 | (in >> 12));         // 1110xxxx
            out[1] = (char)(0x80 | ((in >> 6) & 0x3f)); // 10xxxxxx
            out[2] = (char)(0x80 | (in & 0x3f));        // 10xxxxxx
        }
        return 3;
    }
#if 1
    else {
        TVPThrowExceptionMessage(
            TJS_W("Out of UTF-16 range conversion from UTF-8 code"));
    }
#else
    // 以下オリジナルのコードだけど、通らないはず。
    // （日文注释："以下是原始代码，但应该不会走到这里"）
    // ... 4-6 字节编码路径（被 #if 0 禁用）
#endif
    return -1;
}
```

**关键限制**：这个函数只支持 BMP（U+0000 到 U+FFFF），即最多 3 字节 UTF-8。超出 BMP 范围（如 emoji）会抛出异常。4-6 字节的编码路径被 `#if 0` 注释掉了，日文注释说"这是原始代码但应该走不到这里"。

**`out` 为 nullptr 的设计**：当 `out` 为 `nullptr` 时，函数只返回编码后的字节数而不实际写入数据。这种"空运行"（dry run）设计允许调用者先计算需要的缓冲区大小，再分配内存并实际转换。

### 5.3 TVPWideCharToUtf8String：字符串转换

```cpp
// 源码：cpp/core/base/CharacterSet.cpp 第 78-100 行
tjs_int TVPWideCharToUtf8String(const tjs_char *in, char *out) {
    int count = 0;
    while(*in) {
        tjs_int n;
        if(out) {
            n = TVPWideCharToUtf8(*in, out);
            out += n;  // 推进输出指针
        } else {
            n = TVPWideCharToUtf8(*in, nullptr);
            // 空运行：编译器内联后，TVPWideCharToUtf8 中的 nullptr
            // 检查分支会被完全优化掉（死代码消除）
        }
        if(n == -1) return -1;  // 遇到无效字符
        count += n;
        in++;  // 下一个宽字符
    }
    return count;  // 返回总字节数（不含 null 终止符）
}
```

**编译器优化注释**：源码中有一段有趣的注释："in this situation, the compiler's inliner will collapse all nullptr check parts in TVPWideCharToUtf8"。这说明开发者期望编译器在内联 `TVPWideCharToUtf8` 后，会将 `if(out)` 分支中 `out==nullptr` 的路径完全消除，只保留返回字节数的逻辑。

### 5.4 TVPUtf8ToWideChar：UTF-8→宽字符

```cpp
// 源码：cpp/core/base/CharacterSet.cpp 第 103-180 行
static bool inline TVPUtf8ToWideChar(const char *&in, tjs_char *out) {
    const auto &p = (const unsigned char *&)in;

    if(p[0] < 0x80) {                    // 0xxxxxxx → 1字节 ASCII
        if(out) *out = (tjs_char)in[0];
        in++;
        return true;
    } else if(p[0] < 0xc2) {             // 0x80-0xC1 → 无效起始字节
        return false;                      // （0x80-0xBF 是续延字节，0xC0-0xC1 是过长编码）
    } else if(p[0] < 0xe0) {             // 110xxxxx → 2字节
        if((p[1] & 0xc0) != 0x80) return false;  // 验证续延字节
        if(out) *out = ((p[0] & 0x1f) << 6) + (p[1] & 0x3f);
        in += 2;
        return true;
    } else if(p[0] < 0xf0) {             // 1110xxxx → 3字节
        if((p[1] & 0xc0) != 0x80) return false;
        if((p[2] & 0xc0) != 0x80) return false;
        if(out) *out = ((p[0] & 0x1f) << 12) + ((p[1] & 0x3f) << 6)
                       + (p[2] & 0x3f);
        in += 3;
        return true;
    } else if(p[0] < 0xf8) {             // 11110xxx → 4字节
        // ... 续延字节验证 + 解码（支持 U+10000-U+1FFFFF）
        in += 4; return true;
    } else if(p[0] < 0xfc) {             // 111110xx → 5字节（非标准）
        in += 5; return true;
    } else if(p[0] < 0xfe) {             // 1111110x → 6字节（非标准）
        in += 6; return true;
    }
    return false;
}
```

**与 TVPWideCharToUtf8 的不对称性**：编码函数（宽→UTF-8）只支持 BMP（3 字节），而解码函数（UTF-8→宽）支持 1-6 字节全部序列。这种不对称是有意的：

- **编码端限制**：KrKr2 内部使用 16 位 `tjs_char`，无法表示 BMP 以外的码点（不使用代理对），所以编码时限制到 3 字节是合理的
- **解码端宽容**：解码时可能遇到外部文件中的超 BMP 字符，支持更宽的范围可以避免崩溃（虽然高位数据会丢失到 16 位截断中）

**0xC0-0xC1 被拒绝的原因**：以 `0xC0` 或 `0xC1` 开头的 2 字节序列能编码的范围是 U+0000-U+007F——这些码点应该用 1 字节编码。2 字节编码它们属于"过长编码"（overlong encoding），被 UTF-8 标准禁止。

### 5.5 TVPUtf8ToWideCharString：两个重载

```cpp
// 重载 1：null 终止的字符串（源码第 183-199 行）
tjs_int TVPUtf8ToWideCharString(const char *in, tjs_char *out) {
    int count = 0;
    while(*in) {  // 遇到 '\0' 停止
        tjs_char c;
        if(out) {
            if(!TVPUtf8ToWideChar(in, &c)) return -1;
            *out++ = c;
        } else {
            if(!TVPUtf8ToWideChar(in, nullptr)) return -1;
        }
        count++;
    }
    return count;
}

// 重载 2：指定长度的字符串（源码第 202-247 行）
tjs_int TVPUtf8ToWideCharString(const char *in, tjs_uint length,
                                tjs_char *out) {
    int count = 0;
    const char *end = in + length;
    while(*in && in < end) {
        // 边界安全检查：当剩余字节 < 6 时，检查 UTF-8 序列是否会越界
        if(in + 6 > end) {
            const unsigned char ch = *(const unsigned char *)in;
            if(ch >= 0x80) {
                tjs_uint len = 0;
                // 根据首字节确定序列长度
                if(ch < 0xc2) return -1;
                else if(ch < 0xe0) len = 2;
                else if(ch < 0xf0) len = 3;
                // ... 以此类推到 6 字节
                if(in + len > end) return -1;  // 序列会越界 → 报错
            }
        }
        // 正常解码
        tjs_char c;
        if(out) {
            if(!TVPUtf8ToWideChar(in, &c)) return -1;
            *out++ = c;
        } else {
            if(!TVPUtf8ToWideChar(in, nullptr)) return -1;
        }
        count++;
    }
    return count;
}
```

**重载 2 的边界安全**：当剩余可读字节少于 6 时（`in + 6 > end`），函数先检查当前 UTF-8 序列的完整性。如果序列需要 N 个字节但只剩不到 N 个字节，直接返回 -1（错误）。当剩余字节 ≥ 6 时，跳过这个检查，因为即使是最长的 6 字节序列也不会越界。

```cpp
#include <cstdint>
#include <cstdio>
#include <vector>

// KrKr2 mode 0 XOR 加密算法（TextStream.cpp 第 93-95 行）
void xorCipher(char16_t *data, size_t len) {
    for(size_t i = 0; i < len; i++) {
        char16_t ch = data[i];
        if(ch >= 0x20) {  // 只处理可打印字符
            ch ^= (((ch & 0xfe) << 8) ^ 1);
        }
        data[i] = ch;
    }
}

int main() {
    // 测试文本："Hello世界"
    char16_t original[] = {
        u'H', u'e', u'l', u'l', u'o',     // ASCII 部分
        u'\u4e16', u'\u754c',              // "世界"
        u'\n',                              // 控制字符（不加密）
        0                                   // null 终止
    };
    size_t len = 8;  // 不含 null 终止符

    // 显示原文
    printf("原文 (hex): ");
    for(size_t i = 0; i < len; i++) printf("%04X ", original[i]);
    printf("\n");

    // 加密
    std::vector<char16_t> encrypted(original, original + len + 1);
    xorCipher(encrypted.data(), len);
    printf("密文 (hex): ");
    for(size_t i = 0; i < len; i++) printf("%04X ", encrypted[i]);
    printf("\n");

    // 解密（同一算法再执行一次）
    xorCipher(encrypted.data(), len);
    printf("解密 (hex): ");
    for(size_t i = 0; i < len; i++) printf("%04X ", encrypted[i]);
    printf("\n");

    // 验证
    bool match = true;
    for(size_t i = 0; i < len; i++) {
        if(encrypted[i] != original[i]) { match = false; break; }
    }
    printf("解密验证: %s\n", match ? "✓ 与原文一致" : "✗ 不一致!");

    // 控制字符不变
    printf("\n换行符 (0x000A): 加密前=%04X → ", u'\n');
    char16_t nl = u'\n';
    xorCipher(&nl, 1);
    printf("加密后=%04X %s\n", nl, nl == u'\n' ? "(未变化 ✓)" : "(已变化 ✗)");

    return 0;
}
```

### 示例 3：位交换加密/解密器（mode 1）

```cpp
#include <cstdint>
#include <cstdio>
#include <cstring>

// KrKr2 mode 1 位交换算法（TextStream.cpp 第 96-98 行）
void bitSwapCipher(char16_t *data, size_t len) {
    for(size_t i = 0; i < len; i++) {
        uint32_t ch = data[i];
        // 奇偶位互换：偶数位右移 1 位 | 奇数位左移 1 位
        ch = ((ch & 0xAAAAAAAA) >> 1) | ((ch & 0x55555555) << 1);
        data[i] = static_cast<char16_t>(ch);
    }
}

// 辅助函数：打印二进制表示
void printBinary16(uint16_t val) {
    for(int i = 15; i >= 0; i--) {
        putchar((val >> i) & 1 ? '1' : '0');
        if(i % 4 == 0) putchar(' ');  // 每 4 位加空格
    }
}

int main() {
    // 逐字符演示位交换过程
    char16_t test_chars[] = {u'A', u'H', u'\u3042'};  // A, H, あ
    const char *names[] = {"'A' (U+0041)", "'H' (U+0048)", "'あ' (U+3042)"};

    for(int i = 0; i < 3; i++) {
        char16_t original = test_chars[i];
        char16_t encrypted = original;
        bitSwapCipher(&encrypted, 1);
        char16_t decrypted = encrypted;
        bitSwapCipher(&decrypted, 1);

        printf("  原文: 0x%04X  = ", original);
        printBinary16(original); printf("\n");
        printf("  密文: 0x%04X  = ", encrypted);
        printBinary16(encrypted); printf("\n");
        printf("  解密: 0x%04X  = ", decrypted);
        printBinary16(decrypted);
        printf("  %s\n\n", decrypted == original ? "✓" : "✗");
```

```cpp
int wideCharToUtf8(uint16_t in, char *out) {
    if(in < 0x80) {
        if(out) out[0] = static_cast<char>(in);
        return 1;
    } else if(in < 0x800) {
        if(out) {
            out[0] = static_cast<char>(0xC0 | (in >> 6));
            out[1] = static_cast<char>(0x80 | (in & 0x3F));
        }
        return 2;
    } else {
        if(out) {
            out[0] = static_cast<char>(0xE0 | (in >> 12));
            out[1] = static_cast<char>(0x80 | ((in >> 6) & 0x3F));
            out[2] = static_cast<char>(0x80 | (in & 0x3F));
        }
        return 3;
    }
}

// 简化版 UTF-8 解码器（基于 CharacterSet.cpp 第 103-180 行）
bool utf8ToWideChar(const unsigned char *&p, uint16_t *out) {
    if(p[0] < 0x80) {
        if(out) *out = p[0];
        p += 1;
        return true;
    } else if(p[0] < 0xC2) {
        return false;  // 无效起始字节
    } else if(p[0] < 0xE0) {
        if((p[1] & 0xC0) != 0x80) return false;
        if(out) *out = ((p[0] & 0x1F) << 6) | (p[1] & 0x3F);
        p += 2;
        return true;
    } else if(p[0] < 0xF0) {
        if((p[1] & 0xC0) != 0x80 || (p[2] & 0xC0) != 0x80) return false;
        if(out) *out = ((p[0] & 0x0F) << 12) | ((p[1] & 0x3F) << 6)
                     | (p[2] & 0x3F);
        p += 3;
        return true;
    }
    return false;  // 4+ 字节序列不支持
}
```

```cpp
    // 检查加密标记
```

## 八、常见错误与解决方案

### 错误 1：UTF-32LE 文件被误识别为 UTF-16LE

**症状**：打开 UTF-32LE 编码的文件后，文本显示乱码。

**原因**：`checkTextEncoding` 中 UTF-16LE（`FF FE`）的 BOM 检查在 UTF-32LE（`FF FE 00 00`）之前。由于 UTF-32LE 的前 2 字节与 UTF-16LE 的 BOM 完全相同，UTF-32LE 文件会被错误识别为 UTF-16LE。

**解决方案**：调整 BOM 检测顺序，先检查 4 字节 BOM（UTF-32），再检查 2 字节 BOM（UTF-16）。本节示例 1 中已展示了正确的检测顺序。

### 错误 2：加密文件的 magic bytes 与 UTF-16BE BOM 冲突

**症状**：未加密的 UTF-16BE 文件（BOM 为 `FE FF`）被误判为加密文件。

**原因**：加密标记 `0xFE 0xFE` 与 UTF-16BE 的 BOM `0xFE 0xFF` 仅第二个字节不同。如果文件以 `0xFE` 开头但第二个字节不是 `0xFE`，代码能正确区分。但如果有其他程序错误地写入了 `0xFE 0xFE` 开头的非加密文件，会被误判为加密。

**解决方案**：KrKr2 引擎通过约定保证不会发生冲突——普通文本文件不会以 `0xFE 0xFE` 开头。如果需要额外安全，可以在加密标记后增加校验字段。

### 错误 3：Emoji 或辅助平面字符导致 TVPWideCharToUtf8 抛出异常

**症状**：包含 emoji（如 😊 U+1F60A）的文本在转换为 UTF-8 时抛出 "Out of UTF-16 range" 异常。

**原因**：`TVPWideCharToUtf8` 只支持 BMP（U+0000-U+FFFF），emoji 位于辅助平面（U+10000 以上），需要 4 字节 UTF-8 编码，但这个路径被 `#if 0` 禁用了。

**解决方案**：KrKr2 设计上不支持辅助平面字符（内部 `tjs_char` 是 16 位），所以这是已知限制。如果需要支持 emoji，需要同时修改 `tjs_char` 为 32 位并启用 4-6 字节编码路径。

## 本节小结

- tTVPTextReadStream **在构造时完成所有工作**：加密检测 → 解密/解压 → 编码检测 → 转换为 UTF-16 内部缓冲区
- KrKr2 支持三种文本加密模式：mode 0（XOR，只读不写）、mode 1（位交换，读写都支持）、mode 2（zlib 压缩，读写都支持）
- 编码检测优先级：BOM → uchardet 统计分析 → 默认编码回退
- tTVPTextWriteStream 的析构函数负责 zlib 压缩流的收尾工作（Z_FINISH + 回填大小字段）
- CharacterSet 模块的编码器（宽→UTF-8）只支持 BMP，解码器（UTF-8→宽）支持 1-6 字节全部序列
- `checkTextEncoding` 中存在 BOM 检测顺序 bug：UTF-32LE 可能被误判为 UTF-16LE

### 题目 1：分析 XOR 加密的自反性

KrKr2 mode 0 XOR 加密公式为 `ch ^= (((ch & 0xfe) << 8) ^ 1)`。请手动计算字符 '日' (U+65E5) 的加密结果，并验证对加密结果再执行一次同样操作能恢复原文。

<details>
<summary>查看答案</summary>

```
原始字符：ch = 0x65E5

第一步：计算掩码
  ch & 0xfe     = 0x65E5 & 0x00FE = 0x00E4
  << 8          = 0xE400
  ^ 1           = 0xE401

第二步：加密
  ch ^ 0xE401   = 0x65E5 ^ 0xE401 = 0x81E4

加密结果：0x81E4

验证解密（对 0x81E4 执行同一操作）：
  ch & 0xfe     = 0x81E4 & 0x00FE = 0x00E4
  << 8          = 0xE400
  ^ 1           = 0xE401
  ch ^ 0xE401   = 0x81E4 ^ 0xE401 = 0x65E5 ✓

结论：0x65E5 → 加密 → 0x81E4 → 解密 → 0x65E5，自反性成立。

原理：XOR 运算的自反性保证 a ^ b ^ b = a。虽然掩码 b 是从 ch 本身
计算出来的，但加密和解密时的 ch 不同（加密时是原文 0x65E5，解密时是
密文 0x81E4），为什么还能自反？

关键在于掩码只使用了 ch 的低 8 位中除最低位外的 7 位（ch & 0xfe），
而这 7 位在 XOR 操作后不会改变：
  0x65E5 & 0x00FE = 0x00E4
  0x81E4 & 0x00FE = 0x00E4  ← 相同！

这是因为 (ch & 0xfe) << 8 产生的掩码高字节来自 ch 的低字节，
XOR 后只影响高字节，不影响低字节。所以低字节中的 & 0xfe 部分
在加密前后保持不变，掩码也保持不变。
```

</details>

### 题目 2：修复 BOM 检测顺序

下面的 BOM 检测代码存在一个与 KrKr2 源码相同的 bug。请指出 bug 所在并给出修复后的代码：

```cpp
std::string detectEncoding(const uint8_t *data, size_t size) {
    if(size >= 2 && data[0] == 0xFF && data[1] == 0xFE)
        return "UTF-16LE";
    if(size >= 2 && data[0] == 0xFE && data[1] == 0xFF)
        return "UTF-16BE";
    if(size >= 3 && data[0] == 0xEF && data[1] == 0xBB && data[2] == 0xBF)
        return "UTF-8";
    if(size >= 4 && data[0] == 0xFF && data[1] == 0xFE
                 && data[2] == 0x00 && data[3] == 0x00)
        return "UTF-32LE";
    return "unknown";
}
```

<details>
<summary>查看答案</summary>

**Bug**：UTF-32LE 的 BOM 是 `FF FE 00 00`（4 字节），其前 2 字节 `FF FE` 与 UTF-16LE 的 BOM 完全相同。由于 UTF-16LE 的检查在前面，UTF-32LE 文件会被错误识别为 UTF-16LE——第 4 个 `if` 永远不会匹配到。

**修复**：将长 BOM 的检查放在短 BOM 之前（4 字节优先于 2 字节）：

```cpp
std::string detectEncoding(const uint8_t *data, size_t size) {
    // 先检查 4 字节 BOM（优先级高于 2 字节）
    if(size >= 4 && data[0] == 0xFF && data[1] == 0xFE
                 && data[2] == 0x00 && data[3] == 0x00)
        return "UTF-32LE";
    if(size >= 4 && data[0] == 0x00 && data[1] == 0x00
                 && data[2] == 0xFE && data[3] == 0xFF)
        return "UTF-32BE";
    // 然后检查 3 字节 BOM
    if(size >= 3 && data[0] == 0xEF && data[1] == 0xBB && data[2] == 0xBF)
        return "UTF-8";
    // 最后检查 2 字节 BOM
    if(size >= 2 && data[0] == 0xFF && data[1] == 0xFE)
        return "UTF-16LE";
    if(size >= 2 && data[0] == 0xFE && data[1] == 0xFF)
        return "UTF-16BE";
    return "unknown";
}
```

注意：同时补上了 UTF-32BE 的检测，虽然它的 BOM `00 00 FE FF` 不会与其他编码冲突，但为完整性应一并处理。

</details>

### 题目 3：实现简化版 TVPUtf8ToWideChar

请实现一个函数 `int utf8Decode(const uint8_t *input, size_t inputLen, uint16_t *output)`，将 UTF-8 编码的字节序列转换为 UTF-16（仅 BMP）。要求：
1. 支持 1-3 字节 UTF-8 序列（不需要支持 4 字节以上）
2. 拒绝过长编码（如 2 字节编码 ASCII 范围的字符）
3. 验证续延字节（`10xxxxxx`）
4. 返回转换后的字符数，出错返回 -1

<details>
<summary>查看答案</summary>

```cpp
#include <cstdint>
#include <cstdio>

int utf8Decode(const uint8_t *input, size_t inputLen,
               uint16_t *output) {
    int count = 0;
    size_t i = 0;

    while(i < inputLen && input[i] != 0) {
        uint8_t b0 = input[i];

        if(b0 < 0x80) {
            // 1 字节：0xxxxxxx → U+0000-U+007F
            if(output) output[count] = b0;
            count++;
            i += 1;
        } else if(b0 < 0xC2) {
            // 0x80-0xBF：续延字节不能作为起始
            // 0xC0-0xC1：过长编码（2 字节编码 U+0000-U+007F）
            return -1;
        } else if(b0 < 0xE0) {
            // 2 字节：110xxxxx 10xxxxxx → U+0080-U+07FF
            if(i + 1 >= inputLen) return -1;  // 数据不完整
            uint8_t b1 = input[i + 1];
            if((b1 & 0xC0) != 0x80) return -1;  // 续延字节验证

            uint16_t cp = ((b0 & 0x1F) << 6) | (b1 & 0x3F);
            // 过长编码检查：2 字节序列的码点必须 >= 0x80
            // （由 b0 >= 0xC2 已保证，这里不需要额外检查）
            if(output) output[count] = cp;
            count++;
            i += 2;
        } else if(b0 < 0xF0) {
            // 3 字节：1110xxxx 10xxxxxx 10xxxxxx → U+0800-U+FFFF
            if(i + 2 >= inputLen) return -1;  // 数据不完整
            uint8_t b1 = input[i + 1];
            uint8_t b2 = input[i + 2];
            if((b1 & 0xC0) != 0x80) return -1;
            if((b2 & 0xC0) != 0x80) return -1;

            uint16_t cp = ((b0 & 0x0F) << 12) | ((b1 & 0x3F) << 6)
                        | (b2 & 0x3F);
            // 过长编码检查：3 字节序列的码点必须 >= 0x0800
            if(cp < 0x0800) return -1;
            if(output) output[count] = cp;
            count++;
            i += 3;
        } else {
            // 4 字节以上：不支持
            return -1;
        }
    }

    return count;
}

int main() {
    // 测试数据："Aé世" 的 UTF-8 编码
    // A = 0x41 (1字节)
    // é = 0xC3 0xA9 (2字节)
    // 世 = 0xE4 0xB8 0x96 (3字节)
    uint8_t input[] = {0x41, 0xC3, 0xA9, 0xE4, 0xB8, 0x96, 0x00};

    // 先计算需要的缓冲区大小
    int charCount = utf8Decode(input, sizeof(input), nullptr);
    printf("字符数: %d\n", charCount);

    // 实际解码
    uint16_t output[10];
    int result = utf8Decode(input, sizeof(input), output);
    printf("解码结果: ");
    for(int i = 0; i < result; i++) {
        printf("U+%04X ", output[i]);
    }
    printf("\n");
    // 期望输出: U+0041 U+00E9 U+4E16

    // 测试过长编码拒绝
    uint8_t overlong[] = {0xC0, 0x80};  // 2 字节编码 U+0000（应被拒绝）
    int r = utf8Decode(overlong, 2, nullptr);
    printf("过长编码测试: %s\n", r == -1 ? "✓ 已拒绝" : "✗ 未拒绝");

    // 测试无效续延字节
    uint8_t invalid[] = {0xE4, 0x28, 0x96};  // 第 2 字节不是 10xxxxxx
    r = utf8Decode(invalid, 3, nullptr);
    printf("无效续延字节: %s\n", r == -1 ? "✓ 已拒绝" : "✗ 未拒绝");

    return 0;
}
```

</details>
