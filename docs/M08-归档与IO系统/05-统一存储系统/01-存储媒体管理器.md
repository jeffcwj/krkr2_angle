## 一、为什么需要存储媒体管理器？

### 1.1 问题场景

一个视觉小说引擎需要从各种不同的"地方"读取资源：

```
┌─────────────────────────────────────────────────────────────┐
│                      脚本层 (TJS2)                          │
│                                                             │
│   Storages.addAutoPath("data/");                            │
│   var stream = Storages.open("bgm01.ogg");  ← 从哪里读？   │
│                                                             │
└─────────────────────────┬───────────────────────────────────┘
                          │
            ┌─────────────▼─────────────────┐
            │    存储媒体管理器              │
            │  tTVPStorageMediaManager       │
            │                               │
            │   "我来帮你找到正确的数据源"    │
            └─────────────┬─────────────────┘
          ┌───────────────┼───────────────────┐
          ▼               ▼                   ▼
   ┌─────────────┐ ┌──────────────┐ ┌──────────────────┐
   │  file://     │ │  http://      │ │  memory://        │
   │  本地磁盘    │ │  远程服务器   │ │  内存中的数据     │
   │              │ │  （可扩展）   │ │  （可扩展）       │
   └─────────────┘ └──────────────┘ └──────────────────┘
```

如果引擎只支持本地文件系统，代码里到处写 `fopen()` 就够了。但 KrKr2 的存储系统需要支持：

- **本地文件系统**：游戏安装目录下的散文件
- **归档文件内**：XP3/ZIP/TAR/7z 包中的虚拟文件（通过前面章节学习的 `tTVPArchive` 体系）
- **可扩展的自定义来源**：插件可以注册自己的存储媒体（如网络存储、加密存储等）

存储媒体管理器（`tTVPStorageMediaManager`）就是这个"统一调度中心"——它维护一张存储媒体的注册表，根据路径中的协议名（如 `file:`）找到对应的存储媒体，再委托给它完成实际的文件操作。

### 1.2 设计目标

存储媒体管理器需要满足以下设计目标：

1. **协议透明**：上层代码不需要知道文件来自磁盘还是归档，只需给出统一格式的路径
2. **可扩展**：新的存储来源可以通过 `Register` 注册，无需修改管理器本身
3. **路径标准化**：自动处理各种不规范的路径写法（`.`、`..`、混用 `\` 和 `/`、重复分隔符）
4. **当前目录感知**：像操作系统一样维护"当前工作目录"，支持相对路径解析

## 二、iTVPStorageMedia 接口——存储媒体的抽象契约

### 2.1 接口定义

`iTVPStorageMedia` 是所有存储媒体必须实现的纯虚接口。它定义在 `StorageIntf.h` 中，包含 7 个方法：

```cpp
// 源码位置：cpp/core/base/StorageIntf.h
// iTVPStorageMedia —— 存储媒体抽象接口
class iTVPStorageMedia {
public:
    virtual ~iTVPStorageMedia() {}  // 虚析构函数，保证子类析构时调用正确的析构器

    // ---- 身份标识 ----
    // 返回此媒体的协议名（如 "file"），用于在 "file://..." 路径中匹配
    virtual void TJS_INTF_METHOD GetName(ttstr &name) = 0;

    // ---- 路径标准化 ----
    // 标准化域名部分（如将 "C:" 统一为 "c:"）
    // 参数 name 是域名字符串，由管理器传入，媒体实现负责就地修改
    virtual void TJS_INTF_METHOD NormalizeDomainName(ttstr &name) = 0;

    // 标准化路径部分（如将 "\" 替换为 "/"，处理 ".." 等）
    virtual void TJS_INTF_METHOD NormalizePathName(ttstr &name) = 0;

    // ---- 文件操作 ----
    // 检查指定路径的文件是否存在
    // name 格式为已标准化的完整路径
    virtual bool TJS_INTF_METHOD CheckExistentStorage(const ttstr &name) = 0;

    // 打开文件并返回一个 tTJSBinaryStream 指针
    // flags 为 TJS_BS_READ / TJS_BS_WRITE 等模式标志
    virtual tTJSBinaryStream* TJS_INTF_METHOD Open(
        const ttstr &name, tjs_uint32 flags) = 0;

    // ---- 目录列举 ----
    // 列出指定目录下的所有文件/子目录
    // lister 是回调接口，每发现一个条目就调用 lister->Add(filename)
    virtual void TJS_INTF_METHOD GetListAt(
        const ttstr &name, iTVPStorageLister *lister) = 0;

    // ---- 本地路径转换 ----
    // 如果此媒体的文件可以直接通过操作系统路径访问，
    // 返回其本地文件系统路径（用于传给第三方库如 FFmpeg）
    // 不可本地访问时返回空字符串
    virtual void TJS_INTF_METHOD GetLocallyAccessibleName(ttstr &name) = 0;
};
```

### 2.2 各方法的设计意图

| 方法 | 调用时机 | 典型实现 |
|------|----------|----------|
| `GetName` | 管理器注册时，用于建立"协议名→媒体实例"的映射 | `name = TJS_W("file");` |
| `NormalizeDomainName` | 路径解析阶段，标准化 `//` 后的域名部分 | Windows：盘符小写化 `"C:" → "c:"` |
| `NormalizePathName` | 路径解析阶段，标准化 `/` 后的路径部分 | 全小写化、`\`→`/`、处理 `..` |
| `CheckExistentStorage` | 脚本调用 `Storages.isExistentStorage()` 时 | 底层调 `stat()` 或 `access()` |
| `Open` | 脚本/引擎需要读写文件时 | 底层调 `fopen()` / `CreateFile()` |
| `GetListAt` | 自动搜索路径系统构建目录索引时 | 底层调 `opendir()` / `FindFirstFile()` |
| `GetLocallyAccessibleName` | FFmpeg 等第三方库需要操作系统级路径时 | 返回 `C:\game\data\bgm.ogg` 格式路径 |

### 2.3 iTVPStorageLister 回调接口

`GetListAt` 使用的回调接口非常简单，只有一个方法：

```cpp
// 源码位置：cpp/core/base/StorageIntf.h
class iTVPStorageLister {
public:
    // 每发现一个文件或子目录时调用
    // file 参数只包含文件名，不含路径前缀
    virtual void TJS_INTF_METHOD Add(const ttstr &file) = 0;
};
```

这种"回调接口"（Callback Interface）模式在 C++ 中很常见：调用方提供一个实现了 `Add` 方法的对象，被调用方遍历目录时逐个回调。相比返回一个完整列表，回调模式的优势是：

1. **内存友好**：不需要一次性分配存储所有文件名的容器
2. **可中断**：理论上回调方可以在找到目标后提前终止（虽然 KrKr2 的实现没有用到这一点）
3. **解耦**：调用方可以自由选择如何存储结果——数组、哈希表、或者直接处理

### 2.4 为什么用接口而不是函数指针？

C 语言风格的做法是传一个函数指针（`void (*callback)(const char*)`），但 C++ 接口有两个显著优势：

```cpp
// C 风格 —— 函数指针回调
// 问题：如果回调需要额外上下文（如写入哪个容器），只能用全局变量或 void* userdata
void list_callback(const char *filename, void *userdata) {
    auto *vec = (std::vector<std::string>*)userdata;  // 类型不安全的强制转换
    vec->push_back(filename);
}

// C++ 风格 —— 接口回调
// 优势：回调对象可以携带任意状态，且类型完全安全
class MyLister : public iTVPStorageLister {
    std::vector<ttstr> &results_;   // 引用外部容器
public:
    MyLister(std::vector<ttstr> &r) : results_(r) {}
    void TJS_INTF_METHOD Add(const ttstr &file) override {
        results_.push_back(file);   // 类型安全，无需强制转换
    }
};
```

## 三、tTVPStorageMediaManager——管理器内部结构

### 3.1 整体架构

`tTVPStorageMediaManager` 是整个存储系统的核心管理器。它作为**全局单例**存在（并非使用设计模式中的 Singleton，而是直接定义为静态全局变量）：

```cpp
// 源码位置：cpp/core/base/StorageIntf.cpp 第 85 行附近
static tTVPStorageMediaManager TVPStorageMediaManager;
```

管理器内部使用**哈希表**来维护"协议名 → 媒体实例"的映射关系。为了让哈希表正确工作，它定义了三个辅助内部类：

```
tTVPStorageMediaManager
├── tMediaNameString    —— 键类型：支持冒号感知的字符串比较
├── tHashFunc           —— 哈希函数：只对冒号之前的部分计算哈希值
├── tMediaRecord        —— 值类型：存储媒体实例 + 当前域名/路径
└── HashTable           —— tTJSHashTable<tMediaNameString, tMediaRecord, tHashFunc, 16>
```

### 3.2 tMediaNameString——冒号感知的字符串

为什么需要自定义字符串类型？看一个例子：

```
路径:  file://C:/game/data/image.png
       ^^^^
       协议名 = "file"

但在内部查找时，传入的可能是 "file:" 或 "file"——带不带冒号都要能匹配。
```

`tMediaNameString` 的核心在于其 `operator==`：

```cpp
// 概念性实现（基于 StorageIntf.cpp 源码分析）
class tMediaNameString {
    ttstr String;  // 存储的协议名字符串
public:
    tMediaNameString(const tjs_char *str) : String(str) {}
    tMediaNameString(const ttstr &str) : String(str) {}

    // 冒号感知的相等比较：
    // "file" == "file:" → true
    // "file" == "file"  → true
    // "file:" == "file:" → true
    // 比较时忽略末尾的冒号
    bool operator==(const tMediaNameString &other) const {
        const tjs_char *s1 = String.c_str();
        const tjs_char *s2 = other.String.c_str();
        // 逐字符比较，直到某一方遇到 ':' 或字符串结束
        while (*s1 && *s2 && *s1 != TJS_W(':') && *s2 != TJS_W(':')) {
            if (*s1 != *s2) return false;
            s1++; s2++;
        }
        // 两者同时到达 ':' 或末尾则相等
        return (*s1 == 0 || *s1 == TJS_W(':')) &&
               (*s2 == 0 || *s2 == TJS_W(':'));
    }

    const ttstr &GetString() const { return String; }
};
```

### 3.3 tHashFunc——只哈希冒号之前的部分

哈希函数必须与相等比较保持一致（即 `a == b` 则 `hash(a) == hash(b)`）。由于 `tMediaNameString` 在比较时忽略冒号，哈希函数也必须只对冒号之前的字符计算哈希值：

```cpp
// 概念性实现（基于 StorageIntf.cpp 源码分析）
class tHashFunc {
public:
    static tjs_uint32 Make(const tMediaNameString &key) {
        const tjs_char *str = key.GetString().c_str();
        tjs_uint32 hash = 0;
        while (*str && *str != TJS_W(':')) {
            hash = hash * 37 + *str;  // 简单的乘法哈希
            str++;
        }
        return hash;
    }
};
```

这里使用的 **乘法哈希**（Multiplicative Hash）是一种经典的字符串哈希算法。乘以质数 37 使得不同字符串产生的哈希值分布更均匀，减少哈希冲突（两个不同的键映射到同一个桶的情况）。

### 3.4 tMediaRecord——媒体记录

每个注册的存储媒体在哈希表中对应一条 `tMediaRecord`：

```cpp
// 概念性实现（基于 StorageIntf.cpp 源码分析）
class tMediaRecord {
public:
    iTVPStorageMedia *MediaIntf;  // 存储媒体接口指针
    ttstr CurrentDomain;           // 此媒体当前的活动域名
    ttstr CurrentPath;             // 此媒体当前的工作路径

    tMediaRecord()
        : MediaIntf(nullptr) {}

    tMediaRecord(iTVPStorageMedia *intf)
        : MediaIntf(intf) {}
};
```

这三个字段的含义：
- **`MediaIntf`**：指向实际的存储媒体实例（如 `file:` 媒体实例），管理器通过它调用 `Open`、`CheckExistentStorage` 等方法
- **`CurrentDomain`**：类似于操作系统中的"当前盘符"概念——在 Windows 上可以是 `"c:"`，在其他媒体上可以是主机名
- **`CurrentPath`**：类似于操作系统中的"当前目录"——如 `"/game/data/"`

### 3.5 哈希表的模板参数

管理器的核心数据结构是一个 TJS2 引擎提供的哈希表模板：

```cpp
// 哈希表声明
tTJSHashTable<tMediaNameString, tMediaRecord, tHashFunc, 16> HashTable;
//             ^键类型            ^值类型       ^哈希函数   ^桶数量
```

4 个模板参数的含义：

| 参数 | 类型 | 说明 |
|------|------|------|
| `tMediaNameString` | 键类型 | 协议名（冒号感知比较） |
| `tMediaRecord` | 值类型 | 媒体实例 + 当前域/路径 |
| `tHashFunc` | 哈希函数类 | 只对冒号前部分哈希 |
| `16` | 桶数量 | 16 个桶——注册的媒体类型通常很少（1-5 个），16 桶已绰绰有余 |

为什么只用 16 个桶？因为一个游戏引擎通常只注册 1-3 种存储媒体（`file:`、可能的 `http:`、可能的自定义媒体）。桶数量过多反而浪费内存。16 个桶在这种场景下几乎不会发生哈希冲突。

## 四、Register / Unregister——媒体注册与注销

### 4.1 构造函数——自动注册默认媒体

管理器的构造函数不是空的——它会自动创建并注册一个默认的文件系统媒体：

```cpp
// 源码位置：cpp/core/base/StorageIntf.cpp 构造函数
tTVPStorageMediaManager::tTVPStorageMediaManager() {
    // TVPCreateFileMedia() 是一个工厂函数，
    // 创建平台对应的本地文件系统媒体实例
    // Windows 上返回 Windows 文件系统实现
    // Linux/macOS 上返回 POSIX 文件系统实现
    // Android 上返回 Android 资产文件系统实现
    Register(TVPCreateFileMedia());
}
```

这意味着引擎启动后，`file:` 协议就已经可用了——不需要任何手动配置。`TVPCreateFileMedia()` 是一个平台相关的工厂函数，每个平台提供各自的实现：

```
┌─────────────────────────────────────────────┐
│           TVPCreateFileMedia()               │
├─────────────────────────────────────────────┤
│  Windows:  cpp/core/environ/win32/          │
│            → 使用 CreateFile/FindFirstFile   │
│                                             │
│  Linux:    cpp/core/environ/linux/           │
│            → 使用 open/opendir (POSIX)       │
│                                             │
│  macOS:    cpp/core/environ/apple/           │
│            → 使用 POSIX + NSBundle           │
│                                             │
│  Android:  cpp/core/environ/android/         │
│            → 使用 AAssetManager + POSIX      │
└─────────────────────────────────────────────┘
```

### 4.2 Register——注册存储媒体

```cpp
// 源码位置：cpp/core/base/StorageIntf.cpp
void tTVPStorageMediaManager::Register(iTVPStorageMedia *media) {
    // 第一步：获取媒体的协议名
    ttstr medianame;
    media->GetName(medianame);  // 例如：medianame = "file"

    // 第二步：检查是否已经注册了同名的媒体
    tMediaRecord *rec = HashTable.Find(tMediaNameString(medianame));
    if (rec) {
        // 如果已存在，抛出异常——不允许重复注册
        TVPThrowExceptionMessage(
            TVPMediaAlreadyRegistered, medianame);
    }

    // 第三步：创建新的媒体记录并插入哈希表
    tMediaRecord new_rec(media);
    HashTable.Add(tMediaNameString(medianame), new_rec);
}
```

注册流程的设计考量：
1. **唯一性检查**：同一个协议名只能注册一次，防止冲突。如果插件尝试注册已有的 `file:` 协议，会收到明确的错误信息
2. **无 AddRef/Release**：注意管理器并不对 `media` 指针做引用计数管理。这意味着注册方必须保证媒体实例的生命周期长于管理器——通常是全局静态变量或引擎生命周期内的对象

### 4.3 Unregister——注销存储媒体

```cpp
// 源码位置：cpp/core/base/StorageIntf.cpp
void tTVPStorageMediaManager::Unregister(iTVPStorageMedia *media) {
    // 获取媒体协议名
    ttstr medianame;
    media->GetName(medianame);

    // 从哈希表中删除
    // 如果不存在，Delete 操作不会抛异常——静默忽略
    HashTable.Delete(tMediaNameString(medianame));
}
```

`Unregister` 通常在插件卸载时调用。注意它不会检查要删除的媒体是否确实是当初注册的那个实例——只根据协议名删除。在实际使用中，插件卸载的顺序通常由引擎控制，不会出现"删错"的情况。

### 4.4 注册流程的完整时序

```
引擎启动
    │
    ▼
tTVPStorageMediaManager 构造
    │
    ├── TVPCreateFileMedia() 创建平台文件系统媒体
    │       │
    │       ▼
    ├── Register("file") → 插入哈希表
    │
    │   [此时 file:// 协议已可用]
    │
    ▼
插件加载阶段
    │
    ├── 插件 A: Register("http")  → 插入哈希表
    ├── 插件 B: Register("mem")   → 插入哈希表
    │
    │   [现在可以使用 file://, http://, mem:// 三种协议]
    │
    ▼
引擎运行中...
    │
    ▼
插件卸载阶段
    │
    ├── 插件 B: Unregister("mem") → 从哈希表删除
    ├── 插件 A: Unregister("http") → 从哈希表删除
    │
    ▼
引擎关闭
```

## 五、NormalizeStorageName——路径标准化的核心

### 5.1 为什么需要路径标准化？

用户（TJS2 脚本开发者）可能以各种格式书写路径：

```tjs
// 以下路径都应该指向同一个文件：
Storages.open("file://C:/game/data/image.png");
Storages.open("file://c:/game/data/image.png");   // 盘符大小写不同
Storages.open("file://C:/game/./data/image.png");  // 包含 "."
Storages.open("file://C:/game/temp/../data/image.png"); // 包含 ".."
Storages.open("C:\\game\\data\\image.png");         // Windows 反斜杠
Storages.open("data/image.png");                    // 相对路径
Storages.open("data//image.png");                   // 重复斜杠
```

`NormalizeStorageName` 的职责就是将所有这些变体统一为一个标准格式。这是整个存储系统最复杂的单个方法，在源码中占据近 200 行。

### 5.2 路径格式解析

KrKr2 的路径格式借鉴了 URL 的结构：

```
media://domain/path/to/file
^^^^^   ^^^^^^ ^^^^^^^^^^^^^
  │       │        │
  │       │        └── 路径部分：文件在该域中的位置
  │       └────────── 域名部分：逻辑分组（盘符、主机名等）
  └────────────────── 媒体名：使用哪种存储媒体
```

但路径中可以省略各个部分，`NormalizeStorageName` 需要处理所有省略情况：

| 输入格式 | 含义 | 标准化结果（假设当前目录为 file://c:/game/data/） |
|----------|------|------|
| `file://c:/game/data/image.png` | 完整路径 | `file://c:/game/data/image.png` |
| `///game/data/image.png` | 省略媒体名，省略域名 | `file://c:/game/data/image.png` |
| `//d:/other/file.txt` | 省略媒体名，指定域名 | `file://d:/other/file.txt` |
| `/absolute/path/file.txt` | 省略媒体名和域名，绝对路径 | `file://c:/absolute/path/file.txt` |
| `relative/file.txt` | 相对路径 | `file://c:/game/data/relative/file.txt` |

### 5.3 完整解析流程

下面是 `NormalizeStorageName` 的完整分步解析流程（基于 StorageIntf.cpp 第 130-350 行分析）：

```
输入: name = "file://C:/game/./temp/../data//image.png"

步骤 1: 分隔符统一
    将所有 '\' 替换为 '/'
    → "file://C:/game/./temp/../data//image.png"（本例无变化）

步骤 2: 归档分隔符处理
    如果包含 '>'，将归档路径部分单独标记
    archivedelim = 未找到 → 处理整个路径
    
    示例: "archive.xp3>data/image.png"
    → 归档路径 = "archive.xp3"
    → 归档内路径 = "data/image.png"（归档内路径单独标准化）

步骤 3: 提取媒体名
    查找 "://" 序列
    → 媒体名 = "file"
    → 剩余 = "//C:/game/./temp/../data//image.png"
    
    如果没有 "://"：使用 TVPCurrentMedia（当前活动媒体）

步骤 4: 在哈希表中查找媒体记录
    HashTable.Find("file")
    → 找到 tMediaRecord，其中 CurrentDomain="c:", CurrentPath="/game/data/"

步骤 5: 提取域名
    检查路径开头的 "//" 后面部分
    "//C:/game/./temp/../data//image.png"
       ^^
    → 域名 = "C:"
    → 路径 = "/game/./temp/../data//image.png"
    
    如果路径以 "///" 开头（域名为空）：使用 CurrentDomain
    如果路径以 "/" 开头（无域名标记）：使用 CurrentDomain
    如果是相对路径：使用 CurrentDomain + CurrentPath 拼接

步骤 6: 调用媒体的标准化方法
    media->NormalizeDomainName(domain);  → "C:" → "c:"
    media->NormalizePathName(path);      → 平台相关处理

步骤 7: 路径压缩
    处理 "."  → 当前目录，直接删除
    处理 ".." → 上级目录，删除前一级路径段
    处理 "//" → 重复斜杠，合并为单个 "/"
    
    "/game/./temp/../data//image.png"
    → "/game/temp/../data//image.png"   （删除 "."）
    → "/game/data//image.png"           （处理 ".."，删除 "temp"）
    → "/game/data/image.png"            （合并 "//"）

步骤 8: 组装最终路径
    → "file://c:/game/data/image.png"
```

### 5.4 路径压缩的 C++ 实现细节

路径压缩部分是最容易出错的地方。KrKr2 的实现使用**原地修改**（in-place modification）策略，用两个指针遍历路径字符串：

```cpp
// 概念性实现（基于 StorageIntf.cpp 路径压缩逻辑）
void CompressPath(ttstr &path) {
    tjs_char *p = path.Independ();  // 获取可写缓冲区
    tjs_char *out = p;               // 写入指针
    const tjs_char *in = p;          // 读取指针

    while (*in) {
        if (in[0] == TJS_W('/') && in[1] == TJS_W('/')) {
            // 情况 1: 重复斜杠 "//"
            // 跳过多余的斜杠
            in++;
            continue;
        }

        if (in[0] == TJS_W('/') && in[1] == TJS_W('.')) {
            if (in[2] == TJS_W('/') || in[2] == 0) {
                // 情况 2: "/./" 或 "/." 在末尾
                // "当前目录"——跳过 "/."
                in += 2;
                continue;
            }
            if (in[1] == TJS_W('.') && in[2] == TJS_W('.') &&
                (in[3] == TJS_W('/') || in[3] == 0)) {
                // 情况 3: "/../" 或 "/.." 在末尾
                // "上级目录"——回退 out 指针到前一个 '/'
                in += 3;
                if (out > p) {
                    out--;  // 跳过当前 '/'
                    while (out > p && *out != TJS_W('/')) out--;
                    // out 现在指向上级目录的 '/'
                }
                continue;
            }
        }

        // 普通字符：直接复制
        *out++ = *in++;
    }
    *out = 0;  // 终止符

    // 更新字符串长度
    path.FixLen();
}
```

这种双指针（read pointer + write pointer）的原地压缩技巧在系统编程中非常常见。它的优势是不需要额外分配内存，时间复杂度为 O(n)。

### 5.5 归档分隔符的特殊处理

当路径中包含归档分隔符 `>` 时，`NormalizeStorageName` 会将路径分成两部分分别标准化：

```
输入: "file://c:/game/data.xp3>images/bg01.png"

分割:
    归档路径:    "file://c:/game/data.xp3"    → 按正常路径标准化
    归档内路径:  "images/bg01.png"             → 小写化 + 斜杠标准化

组装:
    → "file://c:/game/data.xp3>images/bg01.png"
```

归档内路径的标准化比较简单，主要是小写化和统一斜杠方向（由 `tTVPArchive::NormalizeInArchiveStorageName` 完成，我们在前面章节已经学习过）。

## 七、Open 与 CheckExistentStorage——文件操作的转发

### 7.1 Open 方法

管理器的 `Open` 方法是一个简单的转发器——它不自己做任何 IO 操作，而是找到对应的媒体实例，委托给它：

```cpp
// 源码位置：cpp/core/base/StorageIntf.cpp
tTJSBinaryStream* tTVPStorageMediaManager::Open(
    const ttstr &name, tjs_uint32 flags)
{
    // 第一步：标准化路径名
    ttstr normalized = name;
    NormalizeStorageName(normalized);

    // 第二步：提取媒体名，查找媒体记录
    ttstr medianame;
    // ... 解析 ...
    tMediaRecord *rec = HashTable.Find(tMediaNameString(medianame));
    if (!rec) {
        TVPThrowExceptionMessage(TVPUnknownMedia, medianame);
    }

    // 第三步：委托给媒体实例
    return rec->MediaIntf->Open(normalized, flags);
}
```

### 7.2 CheckExistentStorage 方法

与 `Open` 的结构完全类似：

```cpp
bool tTVPStorageMediaManager::CheckExistentStorage(const ttstr &name) {
    ttstr normalized = name;
    NormalizeStorageName(normalized);

    ttstr medianame;
    // ... 解析 ...
    tMediaRecord *rec = HashTable.Find(tMediaNameString(medianame));
    if (!rec) return false;  // 未知媒体 → 文件不存在

    return rec->MediaIntf->CheckExistentStorage(normalized);
}
```

注意两个方法在"媒体未找到"时的行为差异：
- `Open`：抛出异常（因为调用方期望获得一个流，无法返回"不存在"）
- `CheckExistentStorage`：返回 `false`（因为调用方本来就是在检查存在性）

```cpp
    std::vector<char> data_;  // 内存中的文件内容
    size_t pos_ = 0;          // 当前读写位置
public:
    // 从已有数据构造（用于预填充文件内容）
    MemoryStream(const std::vector<char> &data) : data_(data) {}
    MemoryStream() = default;

    tjs_uint32 Read(void *buf, tjs_uint32 size) override {
        size_t avail = data_.size() - pos_;          // 剩余可读字节数
        size_t toRead = std::min((size_t)size, avail);
        std::memcpy(buf, data_.data() + pos_, toRead);
        pos_ += toRead;
        return (tjs_uint32)toRead;
    }

    tjs_uint32 Write(const void *buf, tjs_uint32 size) override {
        if (pos_ + size > data_.size()) {
            data_.resize(pos_ + size);  // 自动扩容
        }
        std::memcpy(data_.data() + pos_, buf, size);
        pos_ += size;
        return size;
    }

    tjs_uint64 Seek(tjs_uint64 offset, int whence) override {
        switch (whence) {
            case 0: pos_ = (size_t)offset; break;             // SEEK_SET
            case 1: pos_ += (size_t)offset; break;            // SEEK_CUR
            case 2: pos_ = data_.size() + (size_t)offset; break; // SEEK_END
        }
        return pos_;
    }

    tjs_uint64 GetSize() override { return data_.size(); }
};
```

### 示例 2：路径标准化函数

```cpp
#include <string>
#include <iostream>
#include <vector>
#include <algorithm>

// 路径标准化：处理 '.', '..', 重复斜杠
std::string NormalizePath(const std::string &input) {
    // 第一步：将 '\' 替换为 '/'
    std::string path = input;
    for (auto &c : path) {
        if (c == '\\') c = '/';
    }

    // 第二步：按 '/' 分割路径
    std::vector<std::string> segments;  // 存储路径各段
    std::string segment;
    for (size_t i = 0; i < path.size(); i++) {
        if (path[i] == '/') {
            if (!segment.empty()) {
                if (segment == ".") {
                    // "." 表示当前目录，跳过
                } else if (segment == "..") {
                    // ".." 表示上级目录，弹出最后一段
                    if (!segments.empty()) {
                        segments.pop_back();
                    }
                } else {
                    segments.push_back(segment);
                }
                segment.clear();
            }
            // 跳过连续的 '/'（重复斜杠处理）
        } else {
            segment += path[i];
        }
    }
    // 处理最后一段（路径末尾没有 '/'）
    if (!segment.empty()) {
        if (segment == ".") {
            // 跳过
        } else if (segment == "..") {
            if (!segments.empty()) segments.pop_back();
        } else {
            segments.push_back(segment);
        }
    }

    // 第三步：重新组装路径
    std::string result;
    if (!path.empty() && path[0] == '/') {
        result = "/";  // 保留开头的 '/'（绝对路径）
    }
    for (size_t i = 0; i < segments.size(); i++) {
        if (i > 0) result += '/';
        result += segments[i];
    }

    // 保留末尾的 '/'（如果输入以 '/' 结尾）
    if (path.size() > 1 && path.back() == '/' && result.back() != '/') {
        result += '/';
    }

    return result;
}

int main() {
    // 测试各种路径格式
    struct TestCase {
        std::string input;
        std::string expected;

    std::vector<TestCase> tests = {
        {"/game/./data/image.png",           "/game/data/image.png"},
        {"/game/temp/../data/image.png",     "/game/data/image.png"},
        {"/game//data///image.png",          "/game/data/image.png"},
        {"C:\\game\\data\\image.png",        "C:/game/data/image.png"},

        {"/game/data/",                      "/game/data/"},
        {"relative/path/file.txt",           "relative/path/file.txt"},
```

```cpp
class StorageMediaManager {
    // 协议名 → 媒体实例
    std::map<std::string, IStorageMedia*> registry_;
    std::string currentMedia_ = "file";     // 当前活动媒体
    std::string currentPath_ = "/game/data/"; // 当前工作路径

public:
    // 注册媒体
    void Register(IStorageMedia *media) {
        std::string name = media->GetName();
        if (registry_.count(name)) {
            throw std::runtime_error(
                "媒体 '" + name + "' 已注册，不可重复");
        }
        registry_[name] = media;
        std::cout << "[注册] 协议 '" << name << "://' 已就绪" << std::endl;
    }

    // 注销媒体
    void Unregister(const std::string &name) {
        registry_.erase(name);
        std::cout << "[注销] 协议 '" << name << "://' 已移除" << std::endl;
    }

    // 解析路径并找到对应的媒体
    struct ResolvedPath {
        IStorageMedia *media;   // 目标媒体
        std::string path;       // 标准化后的路径
    };

    ResolvedPath Resolve(const std::string &rawPath) {
        // 查找 "://" 分隔符
        auto pos = rawPath.find("://");
        if (pos != std::string::npos) {
            // 完整路径：media://path
            std::string mediaName = rawPath.substr(0, pos);
            std::string path = rawPath.substr(pos + 2); // 保留 "//"
            auto it = registry_.find(mediaName);
            if (it == registry_.end())
                throw std::runtime_error("未知媒体: " + mediaName);
            return {it->second, path};
        } else {
            // 相对路径：使用当前媒体 + 当前路径
            auto it = registry_.find(currentMedia_);
            if (it == registry_.end())
                throw std::runtime_error("当前媒体未注册");
            return {it->second, currentPath_ + rawPath};
        }
    }

    // 检查文件是否存在
    bool Exists(const std::string &path) {
        auto resolved = Resolve(path);
        return resolved.media->Exists(resolved.path);
    }

    // 读取文件
    std::string Read(const std::string &path) {
        auto resolved = Resolve(path);
        return resolved.media->Read(resolved.path);
    }
};
```

```cpp
// 冒号感知的字符串比较
// 在比较时忽略末尾的冒号，使 "file" 和 "file:" 被视为相等
class ColonAwareString {
    std::string str_;
public:
    ColonAwareString(const std::string &s) : str_(s) {}
    ColonAwareString(const char *s) : str_(s) {}

    // 核心比较逻辑：逐字符比较，直到遇到 ':' 或字符串结束
    bool operator==(const ColonAwareString &other) const {
        const char *s1 = str_.c_str();
        const char *s2 = other.str_.c_str();

        // 逐字符比较，在遇到 ':' 时停止
        while (*s1 && *s2 && *s1 != ':' && *s2 != ':') {
            if (*s1 != *s2) return false;
            s1++; s2++;
        }
        // 两者都到达了 ':' 或末尾 → 相等
        return (*s1 == 0 || *s1 == ':') &&
               (*s2 == 0 || *s2 == ':');
    }

    bool operator!=(const ColonAwareString &o) const {
        return !(*this == o);
    }

    // 哈希函数：只对冒号之前的部分计算
    size_t hash() const {
        size_t h = 0;
        for (const char *p = str_.c_str(); *p && *p != ':'; p++) {
            h = h * 37 + (unsigned char)*p;  // 乘法哈希
        }
        return h;
    }

    const std::string &str() const { return str_; }
};
```

```cpp
#include <iostream>
#include <string>
#include <optional>

// 解析 "media://domain/path" 格式的路径
struct ParsedPath {
    std::string media;    // 媒体协议名（如 "file"）
    std::string domain;   // 域名部分（如 "c:"）
    std::string path;     // 路径部分（如 "/game/data/image.png"）
};

// 路径解析器
// 支持以下格式：
//   "media://domain/path"  → 完整路径
//   "///path"              → 省略媒体和域名
//   "//domain/path"        → 省略媒体
//   "/path"                → 省略媒体和域名，绝对路径
//   "relative/path"        → 相对路径
std::optional<ParsedPath> ParseStoragePath(
    const std::string &input,
    const std::string &defaultMedia = "file",
    const std::string &defaultDomain = "c:",
    const std::string &currentPath = "/")
{
    ParsedPath result;
    std::string remaining = input;

    // 第一步：提取媒体名
    auto schemePos = remaining.find("://");
    if (schemePos != std::string::npos) {
        result.media = remaining.substr(0, schemePos);
        remaining = remaining.substr(schemePos + 1); // 保留 "//"
    } else {
        result.media = defaultMedia;
    }

    // 第二步：处理 "//" 前缀和域名
    if (remaining.size() >= 3 && remaining[0] == '/' &&
        remaining[1] == '/' && remaining[2] == '/') {
        // "///" → 省略域名
        result.domain = defaultDomain;
        result.path = remaining.substr(2); // 从第三个 '/' 开始
    } else if (remaining.size() >= 2 &&
               remaining[0] == '/' && remaining[1] == '/') {
        // "//domain/path" → 提取域名
        size_t domainEnd = remaining.find('/', 2);
        if (domainEnd == std::string::npos) {
            result.domain = remaining.substr(2);
            result.path = "/";
        } else {
            result.domain = remaining.substr(2, domainEnd - 2);
            result.path = remaining.substr(domainEnd);
        }
    } else if (!remaining.empty() && remaining[0] == '/') {
        // "/path" → 绝对路径，使用默认域名
        result.domain = defaultDomain;
        result.path = remaining;
    } else {
        // "relative/path" → 相对路径
        result.domain = defaultDomain;
        result.path = currentPath + remaining;
    }

    return result;
}

        {"images/bg01.png",                 "相对路径"},
        {"http://example.com/api/data",     "HTTP 协议"},
    };

            std::cout << "  域名:  " << parsed->domain << std::endl;
            std::cout << "  路径:  " << parsed->path << std::endl;
            std::cout << "  完整:  " << parsed->media << "://"

        }
    }
```

<details>
<summary>查看答案</summary>

```cpp
#include <iostream>
#include <map>
#include <vector>
#include <string>
#include <algorithm>

// 简化的流基类
class SimpleStream {
public:
    virtual ~SimpleStream() = default;
    virtual int Read(char *buf, int size) = 0;
    virtual int Write(const char *buf, int size) = 0;
};

// XOR 加密/解密流
// 在 Read 和 Write 时自动进行 XOR 运算
class XorCipherStream : public SimpleStream {
    std::vector<char> &storage_;  // 引用底层存储
    size_t pos_ = 0;
    char key_;

public:
    XorCipherStream(std::vector<char> &storage, char key)
        : storage_(storage), key_(key) {}

    int Read(char *buf, int size) override {
        int avail = (int)(storage_.size() - pos_);
        int toRead = std::min(size, avail);
        for (int i = 0; i < toRead; i++) {
            buf[i] = storage_[pos_ + i] ^ key_;  // 读取时解密
        }
        pos_ += toRead;
        return toRead;
    }

    int Write(const char *buf, int size) override {
        for (int i = 0; i < size; i++) {
            if (pos_ + i >= storage_.size()) {
                storage_.resize(pos_ + i + 1);
            }
            storage_[pos_ + i] = buf[i] ^ key_;  // 写入时加密
        }
        pos_ += size;
        return size;
    }
};

// 加密存储媒体
class EncryptedStorageMedia {
    std::map<std::string, std::vector<char>> files_;
    static constexpr char XOR_KEY = 0x42;

public:
    std::string GetName() { return "enc"; }

    bool CheckExistentStorage(const std::string &path) {
        return files_.count(path) > 0;
    }

    // 打开文件（如果不存在则创建）
    SimpleStream* Open(const std::string &path, bool create = false) {
        if (!create && files_.find(path) == files_.end()) {
            throw std::runtime_error("加密文件不存在: " + path);
        }
        return new XorCipherStream(files_[path], XOR_KEY);
    }
};

int main() {
    EncryptedStorageMedia enc;

    // 写入加密文件
    auto *ws = enc.Open("/secret/message.dat", true);
    const char *plaintext = "Hello, KrKr2!";
    ws->Write(plaintext, (int)strlen(plaintext));
    delete ws;

    std::cout << "文件已加密写入" << std::endl;

    // 读取并解密
    auto *rs = enc.Open("/secret/message.dat");
    char buf[64] = {};
    int n = rs->Read(buf, sizeof(buf));
    delete rs;

    std::cout << "解密读取 " << n << " 字节: " << buf << std::endl;
    // 输出: 解密读取 13 字节: Hello, KrKr2!

    // 验证加密确实生效
    std::cout << "文件存在检查: "
              << enc.CheckExistentStorage("/secret/message.dat")
              << std::endl;  // 输出: 1

    return 0;
}
```

### 题目 3：分析 NormalizeStorageName 的边界情况

以下路径会被 `NormalizeStorageName` 如何处理？说明理由：

1. `""`（空字符串）
2. `"file://"`（只有协议名，没有路径）
3. `"/../../above_root.txt"`（`..` 超出根目录）
4. `"data.xp3>"`（归档分隔符后面为空）

<details>
<summary>查看答案</summary>

1. **空字符串 `""`**
   - 被视为相对路径，拼接当前目录
   - 结果：当前目录路径本身（如 `file://c:/game/data/`）
   - 实际上，空字符串在业务逻辑层通常会被提前拒绝

2. **`"file://"`**（只有协议名）
   - 提取媒体名 `"file"`，剩余路径为空
   - 空路径被视为相对路径，使用当前域名和当前路径
   - 结果：`file://c:/game/data/`（即当前目录）

3. **`"/../../above_root.txt"`**（`..` 超出根目录）
   - 第一个 `..`：从 `/` 回退——但已经是根目录，无法再回退
   - 在 KrKr2 的实现中，当 `out > p` 的条件不满足时，`..` 被忽略
   - 结果：`file://c:/above_root.txt`（`..` 被静默忽略，不会跳出根目录）
   - 这是一种安全措施，防止路径遍历攻击（Path Traversal Attack，一种通过构造特殊路径来访问不应访问的文件的攻击手段）

4. **`"data.xp3>"`**（归档分隔符后面为空）
   - 归档路径 = `"data.xp3"`，归档内路径 = `""`（空）
   - 归档内空路径不指向任何文件
   - 在后续的 `CreateStream` 调用中，空的归档内文件名会导致查找失败并抛出异常

</details>
