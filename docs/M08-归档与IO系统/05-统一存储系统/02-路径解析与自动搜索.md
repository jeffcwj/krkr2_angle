## 一、路径工具函数

### 1.1 为什么需要路径工具函数？

在游戏引擎中，操作文件路径是家常便饭：

- 加载图片时需要知道文件扩展名（`.png`?`.jpg`?）来选择解码器
- 保存文件时需要提取目录路径来确保目录存在
- 脚本中经常需要获取文件名（不含路径）来做匹配

KrKr2 提供了 4 个路径工具函数来完成这些操作。它们定义在 `StorageIntf.cpp` 第 775-887 行。

### 1.2 四个函数一览

```
路径示例:  "file://c:/game/data/images/hero.png"

TVPExtractStorageExt  → ".png"           提取扩展名
TVPExtractStorageName → "hero.png"       提取文件名
TVPExtractStoragePath → "file://c:/game/data/images/"  提取目录路径
TVPChopStorageExt     → "file://c:/game/data/images/hero"  去掉扩展名
```

### 1.3 TVPExtractStorageExt——提取扩展名

```cpp
// 源码位置：cpp/core/base/StorageIntf.cpp 第 775 行附近
// 功能：从路径中提取文件扩展名（包含 '.'）
// 返回：如 ".png"、".ogg"；无扩展名时返回空字符串
void TVPExtractStorageExt(const ttstr &name, ttstr &ext) {
    // 从字符串末尾向前搜索 '.'
    const tjs_char *p = name.c_str();
    tjs_int len = name.GetLen();
    tjs_int i = len - 1;

    // 向前扫描，遇到路径分隔符就停止
    // 分隔符包括 '/'、'\' 和归档分隔符 '>'
    while (i >= 0) {
        if (p[i] == TJS_W('.')) {
            // 找到 '.'，从这里到末尾就是扩展名
            ext = ttstr(p + i);
            return;
        }
        if (p[i] == TJS_W('/') ||
            p[i] == TJS_W('\\') ||
            p[i] == TJS_W('>')) {
            // 遇到分隔符——说明文件名中没有 '.'
            break;
        }
        i--;
    }
    // 没找到扩展名
    ext = TJS_W("");
}
```

**关键细节**：为什么要在遇到 `/`、`\`、`>` 时停止搜索？考虑这个路径：

```
"file://c:/game/data.v2/images/hero"
                  ^^^^
                  这是目录名中的 '.'，不是文件扩展名！
```

如果不在分隔符处停止，就会错误地把 `.v2/images/hero` 当成扩展名。归档分隔符 `>` 也需要同样处理：

```
"archive.xp3>images/hero"
        ^^^^
        这是归档文件的扩展名，不是归档内文件的扩展名
```

### 1.4 TVPExtractStorageName——提取文件名

```cpp
// 源码位置：cpp/core/base/StorageIntf.cpp 第 810 行附近
// 功能：提取路径中最后一个分隔符之后的部分（文件名）
void TVPExtractStorageName(const ttstr &name, ttstr &sname) {
    const tjs_char *p = name.c_str();
    tjs_int len = name.GetLen();
    tjs_int i = len - 1;

    // 从末尾向前找第一个分隔符
    while (i >= 0) {
        if (p[i] == TJS_W('/') ||
            p[i] == TJS_W('\\') ||
            p[i] == TJS_W('>')) {
            // 分隔符后面的部分就是文件名
            sname = ttstr(p + i + 1);
            return;
        }
        i--;
    }
    // 没有分隔符——整个字符串就是文件名
    sname = name;
}
```

### 1.5 TVPExtractStoragePath——提取目录路径

```cpp
// 源码位置：cpp/core/base/StorageIntf.cpp 第 840 行附近
// 功能：提取路径中最后一个分隔符及之前的部分（目录路径）
// 注意：结果包含末尾的分隔符
void TVPExtractStoragePath(const ttstr &name, ttstr &spath) {
    const tjs_char *p = name.c_str();
    tjs_int len = name.GetLen();
    tjs_int i = len - 1;

    while (i >= 0) {
        if (p[i] == TJS_W('/') ||
            p[i] == TJS_W('\\') ||
            p[i] == TJS_W('>')) {
            // 包含分隔符本身
            spath = ttstr(p, i + 1);
            return;
        }
        i--;
    }
    // 没有分隔符——没有目录部分
    spath = TJS_W("");
}
```

### 1.6 TVPChopStorageExt——去掉扩展名

```cpp
// 源码位置：cpp/core/base/StorageIntf.cpp 第 870 行附近
// 功能：返回去掉扩展名后的路径
// "hero.png" → "hero"
// "archive.xp3>data/hero.png" → "archive.xp3>data/hero"
void TVPChopStorageExt(const ttstr &name, ttstr &cname) {
    const tjs_char *p = name.c_str();
    tjs_int len = name.GetLen();
    tjs_int i = len - 1;

    while (i >= 0) {
        if (p[i] == TJS_W('.')) {
            cname = ttstr(p, i);  // 不包含 '.'
            return;
        }
        if (p[i] == TJS_W('/') ||
            p[i] == TJS_W('\\') ||
            p[i] == TJS_W('>')) {
            break;  // 文件名中没有 '.'
        }
        i--;
    }
    // 没有扩展名，返回原始路径
    cname = name;
}
```

### 1.7 四个函数的对比总结

| 函数 | 输入 | 输出 | 搜索方向 | 搜索目标 |
|------|------|------|----------|----------|
| `ExtractStorageExt` | `"dir/file.png"` | `".png"` | 从末尾向前 | 找 `.` |
| `ExtractStorageName` | `"dir/file.png"` | `"file.png"` | 从末尾向前 | 找分隔符 |
| `ExtractStoragePath` | `"dir/file.png"` | `"dir/"` | 从末尾向前 | 找分隔符 |
| `ChopStorageExt` | `"dir/file.png"` | `"dir/file"` | 从末尾向前 | 找 `.` |

四个函数都采用**从末尾向前扫描**的策略，都在遇到分隔符时停止（防止匹配到目录名中的 `.`）。

## 二、自动搜索路径系统

### 2.1 问题背景——为什么需要自动搜索？

游戏开发中，资源文件通常分散在多个目录和归档包中：

```
game/
├── data/
│   ├── images/
│   │   ├── bg/          ← 背景图
│   │   ├── character/   ← 人物立绘
│   │   └── ui/          ← UI 素材
│   ├── sound/
│   │   ├── bgm/         ← 背景音乐
│   │   └── se/          ← 音效
│   └── scenario/        ← 剧本文件
├── patch.xp3            ← 补丁包（覆盖部分资源）
└── extra.xp3>images/    ← DLC 归档包中的图片目录
```

脚本中想加载一张图片时，如果每次都要写完整路径 `"data/images/character/hero.png"`，那既麻烦又难以维护——特别是当资源在补丁包和 DLC 包之间移动时。

KrKr2 的解决方案是**自动搜索路径（Auto Search Path）**系统：脚本只需写文件名 `"hero.png"`，引擎会自动在预注册的目录列表中查找这个文件的完整路径。

### 2.2 三层数据结构

自动搜索路径系统由三个核心数据结构协作完成，定义在 `StorageIntf.cpp` 第 892-896 行：

```cpp
// 源码位置：cpp/core/base/StorageIntf.cpp 第 892-897 行

// 第一层：搜索路径列表（用户注册的目录）
std::vector<ttstr> TVPAutoPathList;

// 第二层：路径缓存（最近查询的 LRU 缓存）
tTJSHashCache<ttstr, ttstr> TVPAutoPathCache(TVP_DEFAULT_AUTOPATH_CACHE_NUM);

// 第三层：路径表（文件名→所在目录的映射）
tTJSHashTable<ttstr, ttstr, tTJSHashFunc<ttstr>, TVP_AUTO_PATH_HASH_SIZE>
    TVPAutoPathTable;   // TVP_AUTO_PATH_HASH_SIZE = 1024

// 惰性重建标志
bool AutoPathTableInit = false;
```

三者的协作关系可以用下图表示：

```
脚本请求: TVPGetPlacedPath("hero.png")
         │
         ▼
  ┌─────────────────┐
  │ TVPAutoPathCache │ ← 第一步：查 LRU 缓存
  │ (LRU, 最大 N 条) │    命中？直接返回
  └────────┬────────┘
           │ 未命中
           ▼
  ┌─────────────────┐
  │ 当前目录检查     │ ← 第二步：在当前目录中查找
  │ (TVPNormalize +  │    文件存在？缓存并返回
  │  ExistentCheck)  │
  └────────┬────────┘
           │ 不存在
           ▼
  ┌─────────────────────┐
  │ TVPAutoPathTable     │ ← 第三步：查路径表
  │ (hash表, 1024 桶)    │    文件名→目录映射
  │                     │
  │ 惰性重建：          │
  │ TVPRebuildAutoPath  │
  │ Table() 从          │
  │ TVPAutoPathList     │ ← 数据源：用户注册的搜索目录
  │ 构建映射            │
  └─────────────────────┘
```

让我们逐一深入了解每个组件。

#### TVPAutoPathList——搜索目录注册表

`TVPAutoPathList` 是一个 `std::vector<ttstr>`，存储所有用户通过 `TVPAddAutoPath` 注册的搜索目录。每个条目都是一个**规范化后的目录路径**，且必须以路径分隔符（`/`、`\` 或 `>`）结尾。

典型内容示例：

```
TVPAutoPathList = [
    "file://./data/images/bg/",
    "file://./data/images/character/",
    "file://./data/sound/bgm/",
    "patch.xp3>images/",
    "extra.xp3>sound/"
]
```

注意列表中既有普通文件系统目录（以 `/` 结尾），也有归档内目录（以 `>` 后的子路径表示）。这是 KrKr2 统一存储系统的一大特点——归档内路径和文件系统路径可以混合使用。

#### TVPAutoPathTable——文件名到目录的映射

`TVPAutoPathTable` 是一个 `tTJSHashTable`，使用 1024 个桶（bucket）的哈希表。它的键是**纯文件名**（不含路径），值是**文件所在的目录路径**。

例如：

```
TVPAutoPathTable 的内容：
  "hero.png"      → "file://./data/images/character/"
  "villain.png"   → "file://./data/images/character/"
  "sky.png"       → "file://./data/images/bg/"
  "battle.ogg"    → "patch.xp3>sound/"
  "title.ogg"     → "file://./data/sound/bgm/"
```

查找时只需用文件名作为键查表，然后拼接 `目录 + 文件名` 就得到完整路径。

#### TVPAutoPathCache——查询结果的 LRU 缓存

`TVPAutoPathCache` 是一个 `tTJSHashCache`，它继承自 `tTJSHashTable` 并添加了 **LRU（Least Recently Used，最近最少使用）淘汰策略**。LRU 缓存的核心思想是：当缓存满时，淘汰最久没有被访问过的条目。

`tTJSHashCache` 的实现很精巧（定义在 `tjsHashSearch.h` 第 574-599 行）：

```cpp
// 源码位置：cpp/core/tjs2/tjsHashSearch.h 第 574 行
template <typename KeyT, typename ValueT,
          typename HashFuncT = tTJSHashFunc<KeyT>,
          tjs_int HashSize = TJS_HS_DEFAULT_HASH_SIZE>
class tTJSHashCache
    : public tTJSHashTable<KeyT, ValueT, HashFuncT, HashSize> {
    typedef tTJSHashTable<KeyT, ValueT, HashFuncT, HashSize> inherited;

    tjs_uint MaxCount;  // 最大条目数

public:
    tTJSHashCache(tjs_uint maxcount) { MaxCount = maxcount; }

    void Add(const KeyT &key, const ValueT &value) {
        inherited::Add(key, value);
        // 超过上限时，砍掉最旧的条目
        if(this->GetCount() > MaxCount) {
            this->ChopLast(this->GetCount() - MaxCount);
        }
    }
};
```

**关键机制**：`tTJSHashTable` 内部维护了一个**双向链表**（通过 `NPrev`/`NNext` 指针），按访问时间排序。`FindAndTouch` 方法在查找到条目后会把它**移动到链表头部**（`CheckUpdateElementOrder`），而 `ChopLast` 会从**链表尾部**（最旧的）开始删除。

这个缓存存储的是 `原始文件名 → 完整规范化路径` 的映射，用于避免重复进行昂贵的路径查找操作。

### 2.3 注册与移除搜索路径

#### TVPAddAutoPath——添加搜索路径

```cpp
// 源码位置：cpp/core/base/StorageIntf.cpp 第 921 行
void TVPAddAutoPath(const ttstr &name) {
    tTJSCriticalSectionHolder cs_holder(TVPCreateStreamCS);  // 线程安全

    // 检查路径必须以分隔符结尾
    tjs_char lastchar = name.GetLastChar();
    if(lastchar != TVPArchiveDelimiter &&   // '>'
       lastchar != TJS_W('/') &&
       lastchar != TJS_W('\\'))
        TVPThrowExceptionMessage(TVPMissingPathDelimiterAtLast);
            // 异常："路径末尾缺少分隔符"

    // 规范化路径
    ttstr normalized = TVPNormalizeStorageName(name);

    // 查重——已存在则跳过
    auto i = std::find(TVPAutoPathList.begin(),
                       TVPAutoPathList.end(), normalized);
    if(i == TVPAutoPathList.end())
        TVPAutoPathList.push_back(normalized);

    // 清除缓存（路径列表变了，旧缓存可能失效）
    TVPClearAutoPathCache();
}
```

几个重要的设计决策：

1. **强制路径分隔符结尾**：如果传入 `"data/images"` 而不是 `"data/images/"`，会抛出异常。这个约定确保了路径拼接时不会出错——`目录 + 文件名` 总是正确的。

2. **去重**：使用线性查找 `std::find` 来去重。由于搜索路径列表通常只有十几个条目，线性查找的性能完全可以接受。

3. **清除缓存**：每次添加路径后都会清除 `TVPAutoPathCache` 和 `TVPAutoPathTable`，并重置 `AutoPathTableInit = false`。这意味着下次查找时会触发路径表的惰性重建。

#### TVPRemoveAutoPath——移除搜索路径

```cpp
// 源码位置：cpp/core/base/StorageIntf.cpp 第 940 行
void TVPRemoveAutoPath(const ttstr &name) {
    tTJSCriticalSectionHolder cs_holder(TVPCreateStreamCS);

    // 同样检查末尾分隔符
    tjs_char lastchar = name.GetLastChar();
    if(lastchar != TVPArchiveDelimiter &&
       lastchar != TJS_W('/') &&
       lastchar != TJS_W('\\'))
        TVPThrowExceptionMessage(TVPMissingPathDelimiterAtLast);

    ttstr normalized = TVPNormalizeStorageName(name);

    // 查找并删除
    auto i = std::find(TVPAutoPathList.begin(),
                       TVPAutoPathList.end(), normalized);
    if(i != TVPAutoPathList.end())
        TVPAutoPathList.erase(i);

    // 同样清除缓存
    TVPClearAutoPathCache();
}
```

注意移除一个不存在的路径不会报错——只是静默忽略。这在运行时动态调整搜索路径时很有用。

#### TVPClearAutoPathCache——缓存清除

```cpp
// 源码位置：cpp/core/base/StorageIntf.cpp 第 900 行
static void TVPClearAutoPathCache() {
    TVPAutoPathCache.Clear();   // 清空 LRU 缓存
    TVPAutoPathTable.Clear();   // 清空路径映射表
    AutoPathTableInit = false;  // 标记需要重建
}
```

这个函数同时清除了缓存和路径表。看起来有点"激进"——为什么不只清缓存、保留路径表呢？原因是路径列表变化后，路径表中的映射可能已经过时（比如新加的目录中有同名文件覆盖了旧路径），所以必须完全重建。

### 2.4 路径表的惰性重建

`TVPRebuildAutoPathTable` 是整个自动搜索系统中最复杂的函数。它遍历 `TVPAutoPathList` 中的每个目录，列出其中所有文件，然后建立 `文件名 → 目录` 的映射表。

这个函数采用**惰性重建（Lazy Rebuild）策略**：只有在 `AutoPathTableInit == false` 时才真正执行重建。正常查找流程中，只有当缓存被清除后（例如添加/移除了搜索路径），下一次查询才会触发重建。

```cpp
// 源码位置：cpp/core/base/StorageIntf.cpp 第 959 行
static tjs_uint TVPRebuildAutoPathTable() {
    // 惰性检查——已初始化则直接返回
    if(AutoPathTableInit)
        return 0;

    tTJSCriticalSectionHolder cs_holder(TVPCreateStreamCS);

    TVPAutoPathTable.Clear();

    tjs_uint64 tick = TVPGetTickCount();        // 计时开始
    TVPAddLog(TVPInfoRebuildingAutoPath);        // 日志："正在重建自动搜索路径表"

    tjs_uint totalcount = 0;

    // 遍历每个注册的搜索路径
    for(auto it = TVPAutoPathList.begin();
        it != TVPAutoPathList.end(); it++) {
        const ttstr &path = *it;
        tjs_uint count = 0;

        // 检查路径中是否包含归档分隔符 '>'
        const tjs_char *sharp_pos =
            TJS_strchr(path.c_str(), TVPArchiveDelimiter);

        if(sharp_pos) {
            // === 归档内路径 ===
            // 例如 "game.xp3>images/"
            // ...（下方详解）
        } else {
            // === 普通文件夹路径 ===
            // 例如 "file://./data/images/"
            // ...（下方详解）
        }
        totalcount += count;
    }

    tjs_uint64 endtick = TVPGetTickCount();     // 计时结束
    // 日志："共 N 个文件，M 个已激活（耗时 Xms）"

    AutoPathTableInit = true;                   // 标记已初始化
    return totalcount;
}
```

这个函数的核心逻辑分为两个分支：处理归档内路径和处理普通文件夹路径。

#### 归档内路径的处理

当搜索路径包含 `>` 分隔符时，说明它指向一个归档包内的目录：

```cpp
// path = "game.xp3>images/"
//                  ^ sharp_pos 指向这里

// 分离归档名和归档内路径
ttstr arcname(path, (int)(sharp_pos - path.c_str()));
    // arcname = "game.xp3"
ttstr in_arc_name(sharp_pos + 1);
    // in_arc_name = "images/"

tTVPArchive::NormalizeInArchiveStorageName(in_arc_name);
tjs_int in_arc_name_len = in_arc_name.GetLen();

// 通过归档缓存获取归档对象（下一节详解）
tTVPArchive *arc = TVPArchiveCache.Get(arcname);

try {
    tjs_uint storagecount = arc->GetCount();

    // 利用归档文件列表有序的特性，快速定位
    tjs_int i = arc->GetFirstIndexStartsWith(in_arc_name);
    if(i != -1) {
        for(; i < (tjs_int)storagecount; i++) {
            ttstr name = arc->GetName(i);

            if(name.StartsWith(in_arc_name)) {
                // 只取直接子文件（不含子目录中的文件）
                if(!TJS_strchr(name.c_str() + in_arc_name_len,
                               TJS_W('/'))) {
                    ttstr sname = TVPExtractStorageName(name);
                    TVPAutoPathTable.Add(sname, path);
                    count++;
                }
            } else {
                // 列表有序，后续不可能再有匹配项
                break;
            }
        }
    }
} catch(...) {
    arc->Release();  // 异常安全：释放引用
    throw;
}
arc->Release();
```

这段代码有几个值得注意的性能优化：

1. **`GetFirstIndexStartsWith`**：利用归档文件列表按名称排序的特性，用二分查找快速定位第一个以 `in_arc_name` 开头的条目，而不是从头遍历。

2. **提前退出**：由于列表有序，一旦发现当前条目不再以 `in_arc_name` 开头，就可以 `break` 退出循环。

3. **只取直接子文件**：通过检查 `in_arc_name` 之后是否还有 `/`，过滤掉子目录中的文件。例如对于 `in_arc_name = "images/"`，`"images/hero.png"` 会被收录，但 `"images/bg/sky.png"` 不会——因为 `bg/sky.png` 中间还有一个 `/`。

#### 普通文件夹路径的处理

当搜索路径不包含 `>` 时，说明它是一个普通的文件系统目录：

```cpp
// 普通文件夹路径处理
// path = "file://./data/images/character/"

// 定义一个临时文件列举器
class tLister : public iTVPStorageLister {
public:
    std::vector<ttstr> list;
    void Add(const ttstr &file) override {
        list.push_back(file);
    }
} lister;

// 通过存储媒体管理器列举目录内容
TVPStorageMediaManager.GetListAt(path, &lister);

// 将每个文件名映射到其所在目录
for(auto &i : lister.list) {
    TVPAutoPathTable.Add(i, path);
    count++;
}
```

这里用到了上一节学过的 `TVPStorageMediaManager.GetListAt`——它会找到对应的存储媒体实现，调用其 `GetListAt` 方法列出指定目录下的所有文件。`iTVPStorageLister` 是一个简单的回调接口（callback interface），只有一个 `Add` 方法。

**对比两种处理方式**：

| | 归档内路径 | 普通文件夹路径 |
|---|---|---|
| 文件列举方式 | 通过 `tTVPArchive` 的有序列表 | 通过 `iTVPStorageMedia::GetListAt` |
| 子目录过滤 | 手动检查 `/` 分隔符 | 由存储媒体实现自行过滤 |
| 性能特点 | 二分查找定位 + 顺序扫描 | 取决于操作系统的目录列举 |
| 缓存优势 | 归档头只解析一次（`TVPArchiveCache`） | 每次重建都需要重新列举 |

### 2.5 核心查找函数——TVPGetPlacedPath

`TVPGetPlacedPath` 是自动搜索系统的入口函数，也是游戏引擎中调用最频繁的路径查找函数。它实现了三级查找策略：

```cpp
// 源码位置：cpp/core/base/StorageIntf.cpp 第 1061 行
ttstr TVPGetPlacedPath(const ttstr &name) {
    // === 第一级：查缓存 ===
    ttstr *incache = TVPAutoPathCache.FindAndTouch(name);
    if(incache)
        return *incache;  // 缓存命中，直接返回

    tTJSCriticalSectionHolder cs_holder(TVPCreateStreamCS);

    // === 第二级：在当前目录中查找 ===
    ttstr normalized(TVPNormalizeStorageName(name));
    bool found = TVPIsExistentStorageNoSearchNoNormalize(normalized);
    if(found) {
        // 当前目录中存在该文件
        TVPAutoPathCache.Add(name, normalized);  // 缓存结果
        return normalized;
    }

    // === 第三级：在自动搜索路径表中查找 ===
    ttstr storagename = TVPExtractStorageName(normalized);

    TVPRebuildAutoPathTable();  // 确保路径表已构建（惰性重建）
    ttstr *result = TVPAutoPathTable.Find(storagename);
    if(result) {
        ttstr found = *result + storagename;  // 目录 + 文件名
        TVPAutoPathCache.Add(name, found);    // 缓存结果
        return found;
    }

    // 未找到，返回空字符串
    return {};
}
```

**三级查找策略的设计思路**：

| 级别 | 查找范围 | 代价 | 缓存效果 |
|------|----------|------|----------|
| 第一级 | `TVPAutoPathCache` | O(1) 哈希查找 | 命中率最高（热点文件反复访问） |
| 第二级 | 当前工作目录 | 一次 `ExistentCheck` | 命中后加入缓存 |
| 第三级 | `TVPAutoPathTable` | 可能触发重建（遍历所有目录） | 命中后加入缓存 |

注意一个细节：**第一级缓存使用原始文件名作为键**（未规范化），而**第二级和第三级的结果存入缓存时也使用原始文件名作为键**。这意味着 `"Hero.PNG"` 和 `"hero.png"` 在缓存中是不同的条目。在大小写不敏感的文件系统上（如 Windows），两者最终会找到同一个文件，但会各占一个缓存位置。

还有一个有趣的注释：

```cpp
// not found
// TVPAutoPathCache.Add(name, ttstr()); // do not cache now
return {};
```

作者注释掉了"缓存未命中结果"的代码。这是一个深思熟虑的决定——如果缓存了"文件不存在"的结果，那么后来文件被动态添加时（比如下载 DLC 后），就可能错误地报告文件不存在。

### 2.6 搜索相关的辅助函数

基于 `TVPGetPlacedPath`，引擎还提供了两个便捷函数：

```cpp
// 源码位置：cpp/core/base/StorageIntf.cpp 第 1111 行
// 功能：查找文件完整路径，找不到则抛异常
ttstr TVPSearchPlacedPath(const ttstr &name) {
    ttstr place = TVPGetPlacedPath(name);
    if(place.IsEmpty())
        TVPThrowExceptionMessage(TVPCannotFindStorage, name);
    return place;
}

// 源码位置：cpp/core/base/StorageIntf.cpp 第 1122 行
```

两者的区别很简单：
- `TVPSearchPlacedPath`：找不到就**抛异常**（适合必须存在的资源，如剧本文件）
- `TVPIsExistentStorage`：找不到就返回 `false`（适合可选资源，如补丁包中的覆盖文件）

## 三、归档缓存（tTVPArchiveCache）

### 3.1 为什么需要归档缓存？

上一节 `TVPRebuildAutoPathTable` 中出现了一行关键代码：

```cpp
tTVPArchive *arc = TVPArchiveCache.Get(arcname);
```

打开一个归档文件是一个**昂贵的操作**——需要读取归档头、解析文件索引、可能还要解压索引数据。以 XP3 格式为例，一个包含数千个文件的 XP3 归档包，其文件索引可能有数十 KB 甚至数 MB。每次需要访问归档内文件时都重新打开和解析归档头，显然是不可接受的浪费。

`tTVPArchiveCache` 就是解决这个问题的：它缓存已经打开并解析好的 `tTVPArchive` 对象，后续访问同一个归档包时直接复用。

### 3.2 实现源码分析

`tTVPArchiveCache` 定义在 `StorageIntf.cpp` 第 686-726 行：

```cpp
// 源码位置：cpp/core/base/StorageIntf.cpp 第 686 行
class tTVPArchiveCache {
    // tTJSRefHolder 是一个智能引用持有者——
    // 构造时调用 AddRef()，析构时调用 Release()
    typedef tTJSRefHolder<tTVPArchive> tHolder;

    // 内部使用 tTJSHashCache 实现 LRU 缓存
    tTJSHashCache<ttstr, tHolder> ArchiveCache;

    // 线程安全
    tTJSCriticalSection CS;

public:
    // 默认容量由 TVP_DEFAULT_ARCHIVE_CACHE_NUM 控制
    tTVPArchiveCache()
        : ArchiveCache(TVP_DEFAULT_ARCHIVE_CACHE_NUM) {}

    ~tTVPArchiveCache() = default;

    // 动态调整缓存上限
    void SetMaxCount(tjs_int maxcount) {
        ArchiveCache.SetMaxCount(maxcount);
    }

    // 清空所有缓存（释放所有归档对象的引用）
    void Clear() {
        ArchiveCache.Clear();
    }

    // 获取归档对象——核心方法
    tTVPArchive *Get(ttstr name) {
        name = TVPNormalizeStorageName(name);   // 规范化路径
        tTJSCSH csh(CS);                        // 加锁

        // 计算哈希值
        tjs_uint32 hash =
            tTJSHashCache<ttstr, tHolder>::MakeHash(name);

        // 在缓存中查找并更新 LRU 顺序
        tHolder *ptr =
            ArchiveCache.FindAndTouchWithHash(name, hash);
        if(ptr) {
            // 缓存命中——直接返回
            return ptr->GetObject();
        }

        // 缓存未命中——检查归档文件是否存在
        if(!TVPIsExistentStorageNoSearch(name)) {
            TVPThrowExceptionMessage(TVPCannotFindStorage, name);
        }

        // 打开归档文件（解析归档头、建立索引）
        tTVPArchive *arc = TVPOpenArchive(name, true);
        tHolder holder(arc);

        // 加入缓存
        ArchiveCache.AddWithHash(name, hash, holder);
        return arc;
    }
} TVPArchiveCache;  // 全局唯一实例
```

**设计要点分析**：

1. **引用计数管理**：`tTJSRefHolder<tTVPArchive>` 在构造时调用 `arc->AddRef()`，析构时调用 `arc->Release()`。当 LRU 缓存淘汰旧条目时，`tHolder` 的析构函数会自动减少引用计数。如果引用计数归零，归档对象就会被自动销毁。

2. **缓存未命中的代价**：`TVPOpenArchive(name, true)` 会根据文件扩展名或文件头来判断归档类型（XP3、ZIP、TAR、7z），然后创建对应的 `tTVPArchive` 子类对象。这个过程需要读取磁盘、解析归档头、可能还需要解压索引——代价很高。

3. **全局唯一实例**：`TVPArchiveCache` 是一个全局静态对象，不是动态分配的。程序退出时通过 `tTVPAtExit` 注册的回调清空缓存。

### 3.3 Compact 事件——内存紧张时的应对

当应用程序进入后台（例如 Android 上用户切换到其他应用）或系统内存紧张时，KrKr2 的事件系统会发出 **Compact 事件（紧凑化事件）**。自动搜索路径系统注册了一个回调来响应这个事件：

```cpp
// 源码位置：cpp/core/base/StorageIntf.cpp 第 907 行
struct tTVPClearAutoPathCacheCallback
    : public tTVPCompactEventCallbackIntf {

    void OnCompact(tjs_int level) override {
        if(level >= TVP_COMPACT_LEVEL_DEACTIVATE) {
            // 应用进入后台时，清除自动搜索路径缓存
            tTJSCriticalSectionHolder cs_holder(TVPCreateStreamCS);
            TVPClearAutoPathCache();
        }
    }
} static TVPClearAutoPathCacheCallback;
```

`TVP_COMPACT_LEVEL_DEACTIVATE` 表示应用失去焦点（被切到后台）。此时清除路径缓存和路径表的理由是：

- **释放内存**：路径表可能包含数千个映射条目，占用不少内存
- **应对外部变化**：应用在后台时，用户可能安装了补丁或 DLC，文件系统内容可能已经改变
- **下次恢复时重建**：当应用回到前台，下一次文件查找会触发惰性重建，自然会扫描到最新的文件

注意归档缓存（`TVPArchiveCache`）有自己独立的清理机制——通过程序退出时的 `tTVPAtExit` 回调清空：

```cpp
// 源码位置：cpp/core/base/StorageIntf.cpp 第 728-731 行
static void TVPClearArchiveCache() { TVPArchiveCache.Clear(); }

static tTVPAtExit TVPClearArchiveCacheAtExit(
    TVP_ATEXIT_PRI_SHUTDOWN, TVPClearArchiveCache);
```

`TVP_ATEXIT_PRI_SHUTDOWN` 是退出优先级常量，确保归档缓存在其他组件之前被清理（因为其他组件的清理过程可能还需要访问归档）。

### 示例 1：简化版路径提取工具

下面是一个独立可编译的路径工具函数实现，演示了 KrKr2 路径解析的核心逻辑：

```cpp
// path_utils.cpp
// 编译：g++ -std=c++17 -o path_utils path_utils.cpp
// 或：cl /std:c++17 /EHsc path_utils.cpp

#include <iostream>
#include <string>

// 路径分隔符集合
// 在 KrKr2 中分隔符包括 '/'、'\' 和归档分隔符 '>'
bool isPathDelimiter(char ch) {
    return ch == '/' || ch == '\\' || ch == '>';
}

// 提取扩展名（模拟 TVPExtractStorageExt）
std::string extractExt(const std::string &path) {
    for (int i = static_cast<int>(path.size()) - 1; i >= 0; i--) {
        if (path[i] == '.') {
            return path.substr(i);  // 包含 '.'
        }
        if (isPathDelimiter(path[i])) {
            break;  // 遇到分隔符，文件名中没有 '.'
        }
    }
    return "";
}

// 提取文件名（模拟 TVPExtractStorageName）
std::string extractName(const std::string &path) {
    for (int i = static_cast<int>(path.size()) - 1; i >= 0; i--) {
        if (isPathDelimiter(path[i])) {
            return path.substr(i + 1);
        }
    }
    return path;  // 没有分隔符，整个就是文件名
}

// 提取目录路径（模拟 TVPExtractStoragePath）
std::string extractPath(const std::string &path) {
    for (int i = static_cast<int>(path.size()) - 1; i >= 0; i--) {
        if (isPathDelimiter(path[i])) {
            return path.substr(0, i + 1);  // 包含分隔符
        }
    }
    return "";
}

// 去掉扩展名（模拟 TVPChopStorageExt）
std::string chopExt(const std::string &path) {
    for (int i = static_cast<int>(path.size()) - 1; i >= 0; i--) {
        if (path[i] == '.') {
            return path.substr(0, i);
        }
        if (isPathDelimiter(path[i])) {
            break;
        }
    }
    return path;
}

int main() {
    // 测试用例
    struct TestCase {
        std::string path;
        std::string expectedExt;
        std::string expectedName;
        std::string expectedPath;
        std::string expectedChop;
    };

    TestCase tests[] = {
        // 普通路径
        {"data/images/hero.png",
         ".png", "hero.png", "data/images/", "data/images/hero"},

        // 归档内路径
        {"game.xp3>images/hero.png",
         ".png", "hero.png", "game.xp3>images/", "game.xp3>images/hero"},

        // 目录名含点号
        {"data.v2/images/hero",
         "", "hero", "data.v2/images/", "data.v2/images/hero"},

        // 无路径
        {"readme.txt",
         ".txt", "readme.txt", "", "readme"},

        // 多层归档
        {"outer.xp3>inner/deep/file.ogg",
         ".ogg", "file.ogg", "outer.xp3>inner/deep/",
         "outer.xp3>inner/deep/file"},
    };

    std::cout << "=== 路径工具函数测试 ===" << std::endl;
    int passed = 0, total = 0;

    for (auto &t : tests) {
        total += 4;

        auto ext = extractExt(t.path);
        auto name = extractName(t.path);
        auto path = extractPath(t.path);
        auto chop = chopExt(t.path);

        auto check = [&](const std::string &label,
                         const std::string &got,
                         const std::string &expected) {
            bool ok = (got == expected);
            if (ok) {
                passed++;
            } else {
                std::cout << "FAIL: " << label << "(\""
                          << t.path << "\")\n"
                          << "  期望: \"" << expected << "\"\n"
                          << "  实际: \"" << got << "\"\n";
            }
        };

        check("extractExt",  ext,  t.expectedExt);
        check("extractName", name, t.expectedName);
        check("extractPath", path, t.expectedPath);
        check("chopExt",     chop, t.expectedChop);
    }

    std::cout << passed << "/" << total << " 测试通过" << std::endl;
    return (passed == total) ? 0 : 1;
}
```

```

```

### 示例 2：简化版自动搜索路径系统

这个示例实现了 KrKr2 自动搜索路径系统的核心逻辑——注册搜索目录、构建文件名映射表、按名称查找文件：

```cpp
// auto_path.cpp
// 编译：g++ -std=c++17 -o auto_path auto_path.cpp
// 或：cl /std:c++17 /EHsc auto_path.cpp

#include <iostream>
#include <string>
#include <vector>
#include <unordered_map>
#include <algorithm>

// 模拟 KrKr2 的自动搜索路径系统（简化版）
class AutoPathSystem {
    // 对应 TVPAutoPathList：注册的搜索路径列表
    std::vector<std::string> pathList_;

    // 对应 TVPAutoPathTable：文件名→目录 映射
    std::unordered_map<std::string, std::string> pathTable_;

    // 对应 AutoPathTableInit：惰性重建标志
    bool tableInit_ = false;

    // 模拟的文件系统内容（真实场景中由 OS 或归档提供）
    // 键=目录路径，值=该目录下的文件名列表
    std::unordered_map<std::string, std::vector<std::string>> fakeFS_;

public:
    // 注册模拟文件系统内容（仅供测试）
    void registerDirectory(const std::string &dir,
                          const std::vector<std::string> &files) {
        fakeFS_[dir] = files;
    }

    // 对应 TVPAddAutoPath
    void addAutoPath(const std::string &path) {
        // 检查末尾分隔符
        if (path.empty() ||
            (path.back() != '/' && path.back() != '\\')) {
            throw std::runtime_error(
                "路径末尾缺少分隔符: " + path);
        }

        // 去重
        auto it = std::find(pathList_.begin(),
                           pathList_.end(), path);
        if (it == pathList_.end()) {
            pathList_.push_back(path);
        }

        // 清除缓存——路径列表变了，旧的映射可能失效
        clearCache();
    }

    // 对应 TVPRemoveAutoPath
    void removeAutoPath(const std::string &path) {
        auto it = std::find(pathList_.begin(),
                           pathList_.end(), path);
        if (it != pathList_.end()) {
            pathList_.erase(it);
        }
        clearCache();
    }

    // 对应 TVPGetPlacedPath（简化版，省略了缓存层）
    std::string getPlacedPath(const std::string &filename) {
        // 确保路径表已构建
        rebuildTable();

        // 在路径表中查找
        auto it = pathTable_.find(filename);
        if (it != pathTable_.end()) {
            return it->second + filename;  // 目录 + 文件名
        }
        return "";  // 未找到
    }

    // 打印当前状态（调试用）
    void dump() const {
        std::cout << "--- 搜索路径列表 ("
                  << pathList_.size() << " 条) ---\n";
        for (auto &p : pathList_) {
            std::cout << "  " << p << "\n";
        }
        std::cout << "--- 路径表 ("
                  << pathTable_.size() << " 条映射) ---\n";
        for (auto &[name, dir] : pathTable_) {
            std::cout << "  " << name << " → " << dir << "\n";
        }
    }

private:
    // 对应 TVPClearAutoPathCache
    void clearCache() {
        pathTable_.clear();
        tableInit_ = false;
    }

    // 对应 TVPRebuildAutoPathTable（惰性重建）
    void rebuildTable() {
        if (tableInit_) return;  // 已初始化，跳过

        pathTable_.clear();

        for (auto &path : pathList_) {
            // 在模拟文件系统中查找该目录的文件
            auto fsIt = fakeFS_.find(path);
            if (fsIt != fakeFS_.end()) {
                for (auto &file : fsIt->second) {
                    pathTable_[file] = path;
                }
            }
        }

        tableInit_ = true;
        std::cout << "[重建] 路径表已构建，共 "
                  << pathTable_.size() << " 条映射\n";
    }
};

int main() {
    AutoPathSystem sys;

    // 注册模拟文件系统
    sys.registerDirectory("data/images/bg/",
        {"sky.png", "forest.png", "city.png"});
    sys.registerDirectory("data/images/character/",
        {"hero.png", "villain.png"});
    sys.registerDirectory("data/sound/bgm/",
        {"title.ogg", "battle.ogg"});

    // 添加搜索路径
    std::cout << "=== 注册搜索路径 ===\n";
    sys.addAutoPath("data/images/bg/");
    sys.addAutoPath("data/images/character/");
    sys.addAutoPath("data/sound/bgm/");

    // 按文件名查找——只需文件名，不需要完整路径
    std::cout << "\n=== 按文件名查找 ===\n";
    auto test = [&](const std::string &name) {
        auto result = sys.getPlacedPath(name);
        if (!result.empty()) {
            std::cout << "  \"" << name << "\" → \""
                      << result << "\"\n";
        } else {
            std::cout << "  \"" << name << "\" → 未找到\n";
        }
    };

    test("hero.png");       // 期望：data/images/character/hero.png
```

### 示例 3：LRU 缓存实现

下面实现一个通用的 LRU 缓存，模拟 `tTJSHashCache` 的核心机制——哈希表 + 双向链表：

```cpp
// lru_cache.cpp
// 编译：g++ -std=c++17 -o lru_cache lru_cache.cpp
// 或：cl /std:c++17 /EHsc lru_cache.cpp

#include <iostream>
#include <string>
#include <unordered_map>
#include <list>
#include <optional>

// 通用 LRU 缓存
// 模拟 KrKr2 的 tTJSHashCache 行为
template<typename Key, typename Value>
class LRUCache {
    size_t maxCount_;

    // 双向链表：存储 key-value 对，最新的在前面
    // 对应 tTJSHashTable 的 NPrev/NNext 链表
    using ListEntry = std::pair<Key, Value>;
    std::list<ListEntry> order_;

    // 哈希表：key → 链表迭代器，用于 O(1) 查找
    // 对应 tTJSHashTable 的哈希桶
    std::unordered_map<Key, typename std::list<ListEntry>::iterator>
        index_;

public:
    explicit LRUCache(size_t maxCount)
        : maxCount_(maxCount) {}

    // 对应 tTJSHashCache::Add
    void add(const Key &key, const Value &value) {
        // 如果 key 已存在，先删除旧条目
        auto it = index_.find(key);
        if (it != index_.end()) {
            order_.erase(it->second);
            index_.erase(it);
        }

        // 插入到链表头部（最新）
        order_.push_front({key, value});
        index_[key] = order_.begin();

        // 超过上限时，删除最旧的条目（链表尾部）
        // 对应 ChopLast
        while (order_.size() > maxCount_) {
            auto &last = order_.back();
            std::cout << "  [淘汰] \"" << last.first << "\"\n";
            index_.erase(last.first);
            order_.pop_back();
        }
    }

    // 对应 FindAndTouch：查找并移到最前面
    std::optional<Value> findAndTouch(const Key &key) {
        auto it = index_.find(key);
        if (it == index_.end()) {
            return std::nullopt;  // 未命中
        }

        // 移到链表头部（更新 LRU 顺序）
        // 对应 CheckUpdateElementOrder
        order_.splice(order_.begin(), order_, it->second);
        return it->second->second;
    }

    // 清空缓存
    void clear() {
        order_.clear();
        index_.clear();
    }

    size_t size() const { return order_.size(); }

    // 打印缓存内容（从最新到最旧）
    void dump() const {
        std::cout << "缓存内容 (" << order_.size()
                  << "/" << maxCount_ << "):\n";
        int i = 0;
        for (auto &[k, v] : order_) {
            std::cout << "  [" << i++ << "] \""
                      << k << "\" → \"" << v << "\"\n";
        }
    }
};

int main() {
    // 创建容量为 3 的 LRU 缓存
    // （KrKr2 的 TVPAutoPathCache 默认容量更大）
    LRUCache<std::string, std::string> cache(3);

    std::cout << "=== 添加条目 ===\n";
    cache.add("hero.png", "images/character/hero.png");
    cache.add("sky.png", "images/bg/sky.png");
    cache.add("battle.ogg", "sound/bgm/battle.ogg");
    cache.dump();

    // 添加第 4 个条目——最旧的 hero.png 被淘汰
    std::cout << "\n=== 添加第 4 个（触发淘汰）===\n";
    cache.add("title.ogg", "sound/bgm/title.ogg");
    cache.dump();

    // 访问 sky.png——它被移到最前面
    std::cout << "\n=== 访问 sky.png ===\n";
    auto result = cache.findAndTouch("sky.png");
    if (result) {
        std::cout << "  命中: " << *result << "\n";
    }
    cache.dump();

    // 再添加一个——最旧的 battle.ogg 被淘汰（sky 刚被访问过）
    std::cout << "\n=== 再添加一个 ===\n";
    cache.add("forest.png", "images/bg/forest.png");
    cache.dump();

    return 0;
}
```

### 示例 4：引用计数归档缓存

下面模拟 `tTVPArchiveCache` 的核心设计——引用计数对象 + LRU 缓存，展示当缓存淘汰条目时引用计数如何自动管理内存：

```cpp
// archive_cache.cpp
// 编译：g++ -std=c++17 -o archive_cache archive_cache.cpp
// 或：cl /std:c++17 /EHsc archive_cache.cpp

#include <iostream>
#include <string>
#include <unordered_map>
#include <list>
#include <vector>

// 模拟引用计数的归档对象
class Archive {
    std::string name_;
    int refCount_ = 1;  // 初始引用计数为 1
    std::vector<std::string> files_;  // 归档内文件列表

public:
    explicit Archive(const std::string &name,
                    std::vector<std::string> files)
        : name_(name), files_(std::move(files)) {
        std::cout << "  [创建] Archive(\"" << name_
                  << "\") - 解析归档头...\n";
    }

    ~Archive() {
        std::cout << "  [销毁] Archive(\"" << name_ << "\")\n";
    }

    void addRef() {
        refCount_++;
        std::cout << "  [AddRef] \"" << name_
                  << "\" refCount=" << refCount_ << "\n";
    }

    void release() {
        refCount_--;
        std::cout << "  [Release] \"" << name_
                  << "\" refCount=" << refCount_ << "\n";
        if (refCount_ <= 0) {
            delete this;
        }
    }

    const std::string &getName() const { return name_; }
    size_t getFileCount() const { return files_.size(); }
};

// 引用持有者（模拟 tTJSRefHolder）
// 构造时 AddRef，析构时 Release
class RefHolder {
    Archive *obj_ = nullptr;

public:
    RefHolder() = default;

    explicit RefHolder(Archive *obj) : obj_(obj) {
        if (obj_) obj_->addRef();
    }

    RefHolder(const RefHolder &other) : obj_(other.obj_) {
        if (obj_) obj_->addRef();
    }

    RefHolder &operator=(const RefHolder &other) {
        if (this != &other) {
            if (obj_) obj_->release();
            obj_ = other.obj_;
            if (obj_) obj_->addRef();
        }
        return *this;
    }

    ~RefHolder() {
        if (obj_) obj_->release();
    }

    Archive *get() const { return obj_; }
};

// 简化版归档缓存（模拟 tTVPArchiveCache）
class ArchiveCache {
    size_t maxCount_;

    using Entry = std::pair<std::string, RefHolder>;
    std::list<Entry> order_;
    std::unordered_map<std::string,
        std::list<Entry>::iterator> index_;

public:
    explicit ArchiveCache(size_t maxCount)
        : maxCount_(maxCount) {}

    // 对应 tTVPArchiveCache::Get
    Archive *get(const std::string &name) {
        auto it = index_.find(name);
        if (it != index_.end()) {
            // 缓存命中——移到前面
            std::cout << "  [缓存命中] \"" << name << "\"\n";
            order_.splice(order_.begin(), order_, it->second);
            return it->second->second.get();
        }

        // 缓存未命中——创建新的归档对象
        std::cout << "  [缓存未命中] \"" << name << "\"\n";
        Archive *arc = new Archive(name, {"file1.txt", "file2.png"});
        RefHolder holder(arc);

        order_.push_front({name, holder});
        index_[name] = order_.begin();

        // 淘汰超出上限的条目
        while (order_.size() > maxCount_) {
            auto &last = order_.back();
            std::cout << "  [淘汰] \"" << last.first << "\"\n";
            index_.erase(last.first);
            order_.pop_back();
            // RefHolder 析构 → Release → 可能 delete
        }

        return arc;
    }

    void clear() {
        order_.clear();
        index_.clear();
    }

    size_t size() const { return order_.size(); }
};

int main() {
    // 缓存容量为 2（便于观察淘汰行为）
    ArchiveCache cache(2);

    std::cout << "=== 首次访问三个归档 ===\n";
    auto *a1 = cache.get("game.xp3");
    std::cout << "\n";
    auto *a2 = cache.get("patch.xp3");
    std::cout << "\n";
    auto *a3 = cache.get("dlc.xp3");
    // game.xp3 应该被淘汰

    std::cout << "\n=== 再次访问 patch.xp3（缓存命中）===\n";
    auto *a4 = cache.get("patch.xp3");

    std::cout << "\n=== 清空缓存 ===\n";
    cache.clear();
    std::cout << "缓存已清空\n";

    return 0;
}
```

运行结果（注意引用计数的变化）：

```

  [缓存未命中] "game.xp3"
  [创建] Archive("game.xp3") - 解析归档头...
  [AddRef] "game.xp3" refCount=2

  [缓存未命中] "patch.xp3"
```

### 示例 5：Compact 事件模拟——内存紧张时的缓存清理

这个示例模拟了 KrKr2 的 Compact 事件机制——当应用进入后台或内存紧张时自动清理缓存：

```cpp
// compact_event.cpp
// 编译：g++ -std=c++17 -o compact_event compact_event.cpp
// 或：cl /std:c++17 /EHsc compact_event.cpp

#include <iostream>
#include <string>
#include <vector>
#include <functional>
#include <unordered_map>

// Compact 级别常量（对应 KrKr2 的 EventIntf.h）
enum CompactLevel {
    COMPACT_LEVEL_IDLE = 5,        // 空闲时
    COMPACT_LEVEL_DEACTIVATE = 50, // 应用进入后台
    COMPACT_LEVEL_MINIMIZE = 80,   // 窗口最小化
    COMPACT_LEVEL_MAX = 100        // 内存极度紧张
};

// Compact 事件回调接口
// 对应 tTVPCompactEventCallbackIntf
class CompactCallback {
public:
    virtual ~CompactCallback() = default;
    virtual void onCompact(int level) = 0;
};

// 全局事件管理器
class EventManager {
    std::vector<CompactCallback *> callbacks_;

public:
    void addCompactHook(CompactCallback *cb) {
        callbacks_.push_back(cb);
    }

    // 触发 compact 事件
    void fireCompact(int level) {
        std::string levelName;
        if (level >= COMPACT_LEVEL_MAX) levelName = "MAX";
        else if (level >= COMPACT_LEVEL_MINIMIZE)
            levelName = "MINIMIZE";
        else if (level >= COMPACT_LEVEL_DEACTIVATE)
            levelName = "DEACTIVATE";
        else levelName = "IDLE";

        std::cout << "\n>>> Compact 事件触发 (level="
                  << level << ", " << levelName << ")\n";

        for (auto *cb : callbacks_) {
            cb->onCompact(level);
        }
    }
};

// 全局事件管理器实例
EventManager gEventManager;

// 简化的路径缓存
class PathCache {
    std::unordered_map<std::string, std::string> cache_;

public:
    void add(const std::string &key, const std::string &val) {
        cache_[key] = val;
    }

    bool find(const std::string &key) const {
        return cache_.find(key) != cache_.end();
    }

    void clear() {
        size_t count = cache_.size();
        cache_.clear();
        std::cout << "  [PathCache] 已清除 " << count
                  << " 条缓存\n";
    }

    size_t size() const { return cache_.size(); }
};

// 路径缓存的 Compact 回调
// 对应 tTVPClearAutoPathCacheCallback
class PathCacheCompactCallback : public CompactCallback {
    PathCache &cache_;

public:
    explicit PathCacheCompactCallback(PathCache &cache)
        : cache_(cache) {}

    void onCompact(int level) override {
        if (level >= COMPACT_LEVEL_DEACTIVATE) {
            // 只在应用失去焦点及以上级别时清理
            std::cout << "  [Compact] 清理路径缓存\n";
            cache_.clear();
        } else {
            std::cout << "  [Compact] 级别不足，跳过路径缓存清理"
                      << " (当前:" << level
                      << ", 需要:" << COMPACT_LEVEL_DEACTIVATE
                      << ")\n";
        }
    }
};

int main() {
    PathCache pathCache;

    // 注册 Compact 回调
    PathCacheCompactCallback cacheCallback(pathCache);
    gEventManager.addCompactHook(&cacheCallback);

    // 模拟正常使用——填充缓存
    std::cout << "=== 模拟正常使用 ===\n";
    pathCache.add("hero.png", "images/character/hero.png");
    pathCache.add("sky.png", "images/bg/sky.png");
    pathCache.add("battle.ogg", "sound/bgm/battle.ogg");
    std::cout << "路径缓存大小: " << pathCache.size() << "\n";

    // 场景 1：空闲 Compact（级别不够，不清理）
    gEventManager.fireCompact(COMPACT_LEVEL_IDLE);
    std::cout << "路径缓存大小: " << pathCache.size() << "\n";

    // 场景 2：应用进入后台（级别足够，清理）
    gEventManager.fireCompact(COMPACT_LEVEL_DEACTIVATE);
    std::cout << "路径缓存大小: " << pathCache.size() << "\n";

    // 场景 3：恢复后重新填充
    std::cout << "\n=== 应用恢复，重新使用 ===\n";
    pathCache.add("hero.png", "images/character/hero.png");
    std::cout << "路径缓存大小: " << pathCache.size() << "\n";

    // 场景 4：内存极度紧张
    gEventManager.fireCompact(COMPACT_LEVEL_MAX);
    std::cout << "路径缓存大小: " << pathCache.size() << "\n";

    return 0;
}
```

## 本节小结

- KrKr2 提供 4 个路径工具函数，都采用**从末尾向前扫描、遇分隔符停止**的策略
- 自动搜索路径系统由三层数据结构组成：**搜索路径列表**（用户注册）→ **路径映射表**（文件名→目录）→ **查询缓存**（LRU）
- 路径表采用**惰性重建**策略——只在路径列表变更后的首次查询时重建
- 重建时同时处理**归档内路径**（通过 `tTVPArchive` 枚举）和**文件系统路径**（通过 `GetListAt` 列举）
- 归档缓存（`tTVPArchiveCache`）使用引用计数 + LRU 避免重复打开归档
- Compact 事件在应用进入后台时清理路径缓存，释放内存并应对外部文件变化

## 常见错误与解决方案

### 错误 1：添加搜索路径时忘记末尾分隔符

```
// 错误：缺少末尾 '/'
Storages.addAutoPath("data/images");
// 抛出异常：TVPMissingPathDelimiterAtLast

// 正确：
Storages.addAutoPath("data/images/");
```

**原因**：KrKr2 强制要求搜索路径以 `/`、`\` 或 `>` 结尾。这不是随意的限制——当引擎用 `目录 + 文件名` 拼接完整路径时，如果目录末尾没有分隔符，就会得到 `data/imageshero.png` 而不是 `data/images/hero.png`。

### 错误 2：同名文件被意外覆盖

```
// 注册搜索路径
Storages.addAutoPath("data/images/");
Storages.addAutoPath("patch/images/");

// 两个目录都有 hero.png，查找时只返回一个
var path = Storages.getPlacedPath("hero.png");
// 结果取决于 TVPAutoPathTable 的 Add 顺序
```

**解决方案**：了解搜索路径的优先级规则——后注册的路径中的文件会覆盖先注册的（因为 `tTJSHashTable::Add` 会覆盖已有键）。利用这个特性可以实现补丁系统：先注册基础资源路径，再注册补丁路径。

### 错误 3：文件明明存在却找不到

```
// 添加了搜索路径
Storages.addAutoPath("data/images/bg/");

// 但查找子目录中的文件失败
var path = Storages.getPlacedPath("special/rare.png");
// 返回空字符串
```

**原因**：`TVPRebuildAutoPathTable` 在处理归档内路径时，会**过滤掉子目录中的文件**——只索引直接子文件。如果 `data/images/bg/` 下有 `special/` 子目录，其中的文件不会被自动搜索发现。

**解决方案**：需要单独注册子目录为搜索路径：`Storages.addAutoPath("data/images/bg/special/")`。

### 题目 1：路径工具函数的边界情况

分析以下路径，写出 4 个路径工具函数的返回值：

```
路径 A: "file://./game.data/archive.xp3>res/img.v2/hero.portrait.png"
路径 B: "noext"
路径 C: ".hidden"
```

<details>
<summary>查看答案</summary>

**路径 A**: `"file://./game.data/archive.xp3>res/img.v2/hero.portrait.png"`

- `ExtractStorageExt` → `".png"`
  - 从末尾向前扫描，第一个 `.` 在 `portrait` 和 `png` 之间
- `ExtractStorageName` → `"hero.portrait.png"`
  - 从末尾向前扫描，第一个分隔符是 `/`（在 `img.v2/` 处）
- `ExtractStoragePath` → `"file://./game.data/archive.xp3>res/img.v2/"`
  - 同上，包含分隔符
- `ChopStorageExt` → `"file://./game.data/archive.xp3>res/img.v2/hero.portrait"`
  - 去掉最后一个 `.png`

注意 `hero.portrait.png` 中有两个 `.`，但函数只匹配**最后一个** `.` 作为扩展名分隔符——这符合直觉（扩展名是 `.png`，不是 `.portrait.png`）。

**路径 B**: `"noext"`

- `ExtractStorageExt` → `""`（无扩展名）
- `ExtractStorageName` → `"noext"`（无分隔符，整个就是文件名）
- `ExtractStoragePath` → `""`（无目录部分）
- `ChopStorageExt` → `"noext"`（无扩展名可去）

**路径 C**: `".hidden"`

- `ExtractStorageExt` → `".hidden"`
  - 从末尾向前扫描找到 `.`，返回 `".hidden"`
  - ⚠️ 注意：Unix 上 `.hidden` 通常是隐藏文件名而非扩展名，但 KrKr2 不做这种区分
- `ExtractStorageName` → `".hidden"`
- `ExtractStoragePath` → `""`
- `ChopStorageExt` → `""`（去掉 `.hidden` 后为空字符串）

</details>

### 题目 2：设计一个支持优先级的自动搜索路径系统

KrKr2 的自动搜索路径系统中，后注册的路径优先级更高。请设计一个改进版本，支持显式指定优先级（0 最高），让用户可以控制"当多个目录包含同名文件时，使用哪个目录的文件"。

要求：
1. `addAutoPath(path, priority)` 支持优先级参数
2. `getPlacedPath(filename)` 返回优先级最高的匹配
3. 给出完整可运行代码

<details>
<summary>查看答案</summary>

```cpp
// priority_path.cpp
// 编译：g++ -std=c++17 -o priority_path priority_path.cpp

#include <iostream>
#include <string>
#include <vector>
#include <unordered_map>
#include <algorithm>

class PriorityAutoPath {
    struct PathEntry {
        std::string path;
        int priority;  // 0 = 最高优先级
    };

    std::vector<PathEntry> pathList_;

    // 文件名 → {目录, 优先级}
    struct FileMapping {
        std::string dir;
        int priority;
    };
    std::unordered_map<std::string, FileMapping> pathTable_;
    bool tableInit_ = false;

    // 模拟文件系统
    std::unordered_map<std::string,
        std::vector<std::string>> fakeFS_;

public:
    void registerDirectory(const std::string &dir,
                          const std::vector<std::string> &files) {
        fakeFS_[dir] = files;
    }

    void addAutoPath(const std::string &path, int priority = 100) {
        pathList_.push_back({path, priority});
        clearCache();
    }

    std::string getPlacedPath(const std::string &filename) {
        rebuildTable();
        auto it = pathTable_.find(filename);
        if (it != pathTable_.end()) {
            return it->second.dir + filename;
        }
        return "";
    }

private:
    void clearCache() {
        pathTable_.clear();
        tableInit_ = false;
    }

    void rebuildTable() {
        if (tableInit_) return;
        pathTable_.clear();

        for (auto &entry : pathList_) {
            auto fsIt = fakeFS_.find(entry.path);
            if (fsIt == fakeFS_.end()) continue;

            for (auto &file : fsIt->second) {
                auto existing = pathTable_.find(file);
                if (existing == pathTable_.end() ||
                    entry.priority < existing->second.priority) {
                    // 新条目，或优先级更高（数字更小）
                    pathTable_[file] = {entry.path, entry.priority};
                }
            }
        }
        tableInit_ = true;
    }
};

int main() {
    PriorityAutoPath sys;

    sys.registerDirectory("base/images/",
        {"hero.png", "bg.png"});
    sys.registerDirectory("patch/images/",
        {"hero.png"});  // 补丁版 hero
    sys.registerDirectory("mod/images/",
        {"hero.png", "extra.png"});  // MOD 版 hero

    // 基础路径优先级 100，补丁优先级 50，MOD 优先级 10
    sys.addAutoPath("base/images/", 100);
    sys.addAutoPath("patch/images/", 50);
    sys.addAutoPath("mod/images/", 10);  // MOD 优先级最高

    std::cout << "hero.png → "
              << sys.getPlacedPath("hero.png") << "\n";
    // 期望：mod/images/hero.png（优先级 10 最高）

    std::cout << "bg.png → "
              << sys.getPlacedPath("bg.png") << "\n";
    // 期望：base/images/bg.png（只有 base 有）

    std::cout << "extra.png → "
              << sys.getPlacedPath("extra.png") << "\n";
    // 期望：mod/images/extra.png

    return 0;
}
```

输出：
```
hero.png → mod/images/hero.png
bg.png → base/images/bg.png
extra.png → mod/images/extra.png
```

</details>

### 题目 3：分析 TVPRebuildAutoPathTable 的性能瓶颈

假设一个游戏有：
- 10 个搜索路径（5 个归档内路径 + 5 个本地目录）
- 每个路径下平均 500 个文件
- 总共 5000 个文件

回答以下问题：
1. `TVPRebuildAutoPathTable` 的时间复杂度是多少？
2. 什么情况下重建会被频繁触发？如何避免？
3. 如果哈希表的桶数量从 1024 改为 64，会有什么影响？

<details>
<summary>查看答案</summary>

**1. 时间复杂度分析**

重建过程遍历每个搜索路径，对每个路径列举所有文件：

- 归档内路径：`GetFirstIndexStartsWith` 用二分查找 O(log N)，然后顺序遍历匹配的文件 O(M)
- 本地目录：`GetListAt` 的代价取决于操作系统的目录列举，通常 O(M)
- 每个文件插入哈希表：O(1)（均摊）

总时间复杂度：**O(P × M)**，其中 P 是路径数量，M 是每个路径下的平均文件数。

对于 10 个路径 × 500 个文件 = 5000 次哈希表插入，实际执行时间通常在几毫秒到几十毫秒之间。

**2. 频繁触发重建的场景**

- 在循环中反复调用 `addAutoPath` / `removeAutoPath`——每次调用都会清除缓存
- 应用频繁在前台/后台切换——Compact 事件会清除缓存

避免方法：
- 在游戏启动时一次性注册所有搜索路径，避免运行时频繁修改
- 对于 Android 应用，可以考虑只在 `COMPACT_LEVEL_MAX` 时才清除路径缓存（而不是 `COMPACT_LEVEL_DEACTIVATE`）

**3. 桶数量从 1024 改为 64 的影响**

- 平均链长从 5000/1024 ≈ 4.9 增加到 5000/64 ≈ 78.1
- 哈希表查找从接近 O(1) 退化为接近 O(N/64)
- 对于 `TVPGetPlacedPath` 这种高频调用的函数，性能退化会很明显
- 内存占用略有减少（少了 960 个桶指针），但这点内存节省不值得

结论：1024 个桶对于几千个文件来说是合理的选择，保持了低冲突率和快速查找。

</details>
