## 术语预览

本节将涉及以下术语，先有个印象，正文中会逐一详细讲解：

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| 魔数 | Magic Number | 文件开头的固定字节序列，用于标识文件格式（如 XP3 以 `XP3\r\n\x1a\x8b\x67\x01` 开头） |
| 文件索引 | File Index / Directory | 归档内所有文件的元数据列表（文件名、偏移量、大小、校验和等） |
| 数据区 | Data Area | 归档中存放文件实际内容（原始字节或压缩后字节）的连续区域 |
| 尾部索引 | Trailing Index | 将索引放在文件末尾而非开头的布局方式——允许流式写入，先写数据再回填索引 |
| 对齐填充 | Alignment Padding | 在数据块之间插入填充字节，使每个块的起始偏移量对齐到特定边界（如 16 字节），提高磁盘和内存访问效率 |
| Adler-32 | Adler-32 Checksum | 一种快速校验和算法，比 CRC32 计算速度更快但冲突率略高，适合用于完整性快速验证 |

## 一、为什么需要自定义归档格式

### 1.1 现有格式的局限性

KrKr2 已经支持 XP3、ZIP、TAR、7z 四种归档格式，每种都有成熟的实现。那为什么还需要自定义格式？

在实际项目中，自定义归档格式的需求通常源于以下几类场景：

**场景一：资源保护与混淆**

XP3 虽然支持提取过滤器（Extraction Filter，见 [02-03 提取过滤器与加密](../02-XP3归档格式详解/03-提取过滤器与加密.md)），但其格式结构是公开的——任何了解 XP3 规范的人都可以编写解包工具。如果你的游戏需要更强的资源保护，使用私有格式可以增加逆向难度（注意：这不是"安全"的加密，只是提高了门槛）。

**场景二：特定性能需求**

XP3 的分段压缩虽然灵活，但每个段需要独立的 zlib 解压。如果你的游戏场景需要极高频率的小文件随机读取（例如粒子系统每帧加载几十个小纹理），XP3 的段缓存 + 解压开销可能成为瓶颈。自定义格式可以针对性地优化——例如不压缩小文件、使用更快的压缩算法（如 LZ4）、或预计算偏移量消除索引查找开销。

**场景三：嵌入额外元数据**

标准归档格式的索引通常只存储文件名、大小、偏移量等基本信息。如果需要在归档级别存储自定义元数据——如资源标签（"UI纹理"、"背景音乐"）、加载优先级、预加载提示等——通用格式很难扩展，自定义格式可以原生支持。

**场景四：学习与教学**

对于本教程的读者而言，从零实现一个归档格式是理解 KrKr2 存储系统最有效的方式。你将亲手触及 `tTVPArchive` 基类的每一个虚函数，理解为什么索引要这样设计、流创建要那样实现。

### 1.2 本实战的设计目标

我们将设计一个名为 **KAR**（KrKr2 Archive，KrKr2 自定义归档）的归档格式。设计目标：

| 目标 | 优先级 | 说明 |
|------|--------|------|
| 简单可理解 | 最高 | 格式结构清晰，便于学习和调试 |
| 完整集成 | 高 | 能无缝接入 KrKr2 的 `TVPCreateStream` → `tTVPArchive` 体系 |
| 可选压缩 | 中 | 每个文件可独立选择是否压缩（使用 zlib deflate） |
| 完整性校验 | 中 | 每个文件携带 Adler-32 校验和，读取时可验证 |
| 性能合理 | 中 | 索引放在文件尾部，支持快速加载；对齐填充提升读取效率 |
| 扩展性 | 低 | 预留版本号字段，未来可扩展格式 |

**明确的非目标**（不在本实战范围内）：

- ❌ 加密功能——增加复杂度但不增加教学价值
- ❌ 归档写入/打包工具——只实现读取器（打包工具用 Python 脚本替代）
- ❌ 超大文件支持（>4GB）——使用 32 位偏移量，简化实现
- ❌ 增量更新——不支持修改已打包的归档

## 二、从 tTVPArchive 接口反推格式需求

在设计格式之前，我们必须先搞清楚 KrKr2 引擎对归档格式**要求什么**。答案在 `tTVPArchive` 基类的虚函数接口中（详见 [01-02 tTVPArchive基类设计](../01-归档系统概述/02-tTVPArchive基类设计.md)）。

### 2.1 必须实现的虚函数

回顾 `tTVPArchive` 基类（`StorageIntf.h` 第 26-80 行），子类必须实现以下纯虚函数：

```cpp
// StorageIntf.h:26-80（简化）
class tTVPArchive {
public:
    // 构造函数：接收归档文件路径
    tTVPArchive(const ttstr &name);
    
    // 获取归档中的文件总数
    virtual tjs_uint GetCount() = 0;
    
    // 获取指定索引位置的文件名
    virtual ttstr GetName(tjs_uint idx) = 0;
    
    // 根据文件索引创建流对象
    virtual tTJSBinaryStream *CreateStreamByIndex(tjs_uint idx) = 0;
    
    // 析构函数
    virtual ~tTVPArchive();
    
    // 以下是基类已实现的方法（子类通常不需要重写）：
    // GetName2Index() — 通过文件名查找索引（使用哈希表加速）
    // CreateStream(name) — 通过文件名创建流（调用 GetName2Index + CreateStreamByIndex）
    // Normalize() — 规范化所有文件名
    // GetArchiveName() — 获取归档文件自身的路径
};
```

这个接口告诉我们，格式设计必须满足：

1. **文件可枚举**——需要维护一个有序的文件列表，能按索引访问（支持 `GetCount()` 和 `GetName(idx)`）
2. **文件可通过索引创建流**——给定索引号，能快速定位到该文件的数据并创建一个 `tTJSBinaryStream`（支持 `CreateStreamByIndex(idx)`）
3. **文件名可查找**——基类的 `GetName2Index()` 方法使用哈希表实现文件名到索引的映射，这个哈希表在基类构造函数中通过遍历 `GetName()` 自动建立，我们不需要额外处理

### 2.2 接口要求转化为格式需求

| 接口要求 | 格式设计影响 |
|----------|-------------|
| `GetCount()` 返回文件总数 | 索引中需要存储文件数量，或者索引结构本身可以推算出数量 |
| `GetName(idx)` 按索引返回文件名 | 索引必须是有序数组（而非无序散列），每个条目包含文件名 |
| `CreateStreamByIndex(idx)` 创建流 | 每个索引条目必须包含文件在归档中的数据偏移量和数据大小 |
| 基类构造函数调用 `Normalize()` | 文件名会被规范化（转小写、统一分隔符），格式本身无需特殊处理 |

### 2.3 参考现有格式的设计经验

在定义具体字段之前，让我们对比 KrKr2 支持的四种格式，吸取它们的设计经验：

```
┌─────────────────────────────────────────────────────────────────┐
│                     各格式索引位置对比                           │
├─────────────┬───────────────────────────────────────────────────┤
│   XP3       │  [魔数头] [索引偏移量] [...数据...] [压缩索引]    │
│             │   索引在尾部，头部只存偏移量指针                    │
├─────────────┼───────────────────────────────────────────────────┤
│   ZIP       │  [...数据+本地头...] [中央目录] [EOCD]            │
│             │   中央目录在尾部，EOCD 记录目录位置                 │
├─────────────┼───────────────────────────────────────────────────┤
│   TAR       │  [头1][数据1][头2][数据2]...[EOF标记]             │
│             │   每个文件的元数据紧跟在数据前，无全局索引          │
├─────────────┼───────────────────────────────────────────────────┤
│   7z        │  [签名头] [...数据...] [编码头] [结束头]           │
│             │   类似 XP3，索引（编码头）在尾部                   │
└─────────────┴───────────────────────────────────────────────────┘
```

**观察与决策：**

1. **索引位置**：XP3、ZIP、7z 都把索引放在尾部。好处是打包时可以先流式写出所有数据，最后一次性生成索引写在末尾。TAR 的交错布局（每个文件前放头部）虽然简单，但不支持快速跳转——读第 N 个文件必须线性扫描前 N-1 个头部。**KAR 决策：索引放在尾部。**

2. **压缩策略**：XP3 支持逐段压缩（一个文件可以有多个压缩段），ZIP 是逐文件压缩，TAR 不压缩（通常外层套 gzip），7z 支持固实压缩（多个文件压成一个块）。**KAR 决策：逐文件压缩（简单但足够实用）。**

3. **校验和**：XP3 使用 Adler-32（`adlr` 子块），ZIP 使用 CRC32，TAR 使用头部八进制校验和，7z 使用 CRC32。**KAR 决策：使用 Adler-32（速度快，已有 zlib 提供实现）。**

4. **文件名编码**：XP3 存储 UTF-16（宽字符），ZIP 历史上用 CP437/系统编码但现代实现用 UTF-8，TAR 用 ASCII/UTF-8。**KAR 决策：UTF-8 编码，兼容性最好。**

## 三、KAR 格式规范定义

基于以上分析，我们定义 KAR（KrKr2 Archive）格式的完整二进制规范。

### 3.1 总体布局

```
┌──────────────────────────────────────────────────────────────┐
│                       KAR 文件总体布局                        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌─────────────────────────┐                                │
│  │      文件头 (32 字节)     │  魔数 + 版本 + 文件数量 +      │
│  │      File Header         │  索引偏移量 + 标志位            │
│  └─────────────────────────┘                                │
│                                                              │
│  ┌─────────────────────────┐                                │
│  │      数据区              │  所有文件的实际内容              │
│  │      Data Area           │  （可能经过 zlib 压缩）         │
│  │                         │  每个文件按 16 字节对齐          │
│  │      [文件数据 1]        │                                │
│  │      [对齐填充]          │                                │
│  │      [文件数据 2]        │                                │
│  │      [对齐填充]          │                                │
│  │      ...                 │                                │
│  └─────────────────────────┘                                │
│                                                              │
│  ┌─────────────────────────┐                                │
│  │      文件索引区          │  每个文件一个索引条目            │
│  │      Index Area          │  （固定部分 + 变长文件名）       │
│  │                         │                                │
│  │      [索引条目 1]        │                                │
│  │      [索引条目 2]        │                                │
│  │      ...                 │                                │
│  └─────────────────────────┘                                │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 文件头结构（32 字节固定大小）

```cpp
// KAR 文件头结构定义
struct KARHeader {
    char     magic[8];           // 偏移 0:  魔数 "KAR\r\n\x1a\x00\x00"
    uint16_t version_major;      // 偏移 8:  主版本号（当前 = 1）
    uint16_t version_minor;      // 偏移 10: 次版本号（当前 = 0）
    uint32_t flags;              // 偏移 12: 全局标志位
    uint32_t file_count;         // 偏移 16: 归档中的文件总数
    uint32_t index_offset;       // 偏移 20: 索引区相对文件起始的字节偏移量
    uint32_t index_size;         // 偏移 24: 索引区总字节数
    uint32_t reserved;           // 偏移 28: 保留字段（填充为 0）
};
// sizeof(KARHeader) == 32
```

**各字段设计理由：**

**`magic[8]`**：8 字节魔数 `KAR\r\n\x1a\x00\x00`。设计参考了 XP3 的魔数设计哲学：
- `KAR` — 人类可读的格式名称（在文本编辑器中打开能看到）
- `\r\n` — 检测行尾转换（如果文件被 FTP 以文本模式传输，`\r\n` 会变成 `\n`，读取时可检测到损坏）
- `\x1a` — DOS 的 EOF 标记（防止 `type` 命令输出乱码）
- `\x00\x00` — 填充到 8 字节，同时确保不会被误识别为纯文本文件

```cpp
// 魔数定义
static const char KAR_MAGIC[8] = {
    'K', 'A', 'R',       // 格式标识
    '\r', '\n',           // 行尾转换检测
    '\x1a',               // DOS EOF 标记
    '\x00', '\x00'        // 填充
};
```

**`version_major` / `version_minor`**：版本号拆分为主版本和次版本。主版本变更表示不兼容的格式变化（读取器需要拒绝不认识的主版本），次版本变更表示向后兼容的扩展（读取器可以忽略不认识的次版本特性）。

**`flags`**：32 位标志位，当前定义：

```cpp
enum KARFlags : uint32_t {
    KAR_FLAG_NONE        = 0x00000000,  // 无特殊标志
    KAR_FLAG_COMPRESSED  = 0x00000001,  // 索引区本身经过 zlib 压缩
    // 位 1-31 保留，必须为 0
};
```

**`file_count`**：直接存储文件数量，而非让读取器通过扫描索引计算。好处是 `GetCount()` 只需返回这个值，O(1) 时间复杂度。

**`index_offset` / `index_size`**：索引区的位置和大小。读取器先读 32 字节头部，从中获取 `index_offset`，然后 `fseek` 到该位置读取 `index_size` 字节的索引数据。这就是"尾部索引"设计的核心——头部包含指向尾部的指针。

### 3.3 数据区布局

数据区紧跟在文件头之后，存放所有文件的实际内容。每个文件的数据按 16 字节边界对齐：

```
偏移 32:      [文件 0 的数据（原始或压缩后的字节）]
              [0-15 字节的对齐填充（值为 0x00）]
偏移 32+N:    [文件 1 的数据]
              [0-15 字节的对齐填充]
              ...
```

**对齐填充的意义：**

1. **磁盘读取效率**——磁盘的最小读取单位通常是扇区（512 字节或 4KB），16 字节对齐虽然不能对齐到扇区，但可以保证数据不会跨缓存行（CPU 缓存行通常为 64 字节），减少缓存未命中
2. **内存映射友好**——如果将来扩展为支持 `mmap()` 直接映射归档文件，对齐的数据偏移量可以避免跨页边界访问
3. **调试方便**——在十六进制编辑器中查看时，对齐的偏移量更容易识别各块的边界

```cpp
// 对齐计算辅助函数
inline uint32_t AlignUp(uint32_t value, uint32_t alignment) {
    // 将 value 向上对齐到 alignment 的倍数
    // 例如：AlignUp(37, 16) = 48, AlignUp(32, 16) = 32
    return (value + alignment - 1) & ~(alignment - 1);
}

// 使用示例
uint32_t data_size = 1000;            // 文件实际数据大小
uint32_t aligned_size = AlignUp(data_size, 16);  // = 1008
uint32_t padding = aligned_size - data_size;      // = 8 字节填充
```

### 3.4 索引条目结构

每个文件在索引区中有一个索引条目。条目分为固定部分和变长部分：

```cpp
// KAR 索引条目（固定部分，24 字节）
struct KARIndexEntry {
    uint32_t data_offset;        // 偏移 0:  文件数据在归档中的字节偏移量
    uint32_t compressed_size;    // 偏移 4:  压缩后的数据大小（如果未压缩，等于 original_size）
    uint32_t original_size;      // 偏移 8:  文件原始大小（未压缩时的字节数）
    uint32_t adler32;            // 偏移 12: 原始数据的 Adler-32 校验和
    uint16_t flags;              // 偏移 16: 文件级标志位
    uint16_t name_length;        // 偏移 18: 文件名长度（UTF-8 字节数，不含结尾 \0）
    uint32_t reserved;           // 偏移 20: 保留字段（填充为 0）
};
// sizeof(KARIndexEntry) == 24

// 紧跟在固定部分之后的是变长文件名：
// char name[name_length];       // UTF-8 编码的文件名，不含结尾 \0
```

**字段说明：**

**`data_offset`**：从归档文件起始计算的绝对偏移量。读取器创建流时，通过 `fseek(file, data_offset, SEEK_SET)` 定位到数据起始位置。使用绝对偏移量（而非相对于数据区起始的偏移量）可以简化读取逻辑——不需要再加上头部大小。

**`compressed_size` vs `original_size`**：当文件被压缩时，`compressed_size < original_size`；未压缩时两者相等。读取器需要读取 `compressed_size` 字节的原始/压缩数据，如果检测到压缩标志则解压为 `original_size` 字节。

**`adler32`**：对**原始数据**（未压缩的）计算 Adler-32 校验和。选择 Adler-32 而非 CRC32 的原因：
- Adler-32 计算速度比 CRC32 快约 2-3 倍
- zlib 库已经提供了 `adler32()` 函数，不需要额外依赖
- 对于完整性验证（检测传输/存储损坏）而言，Adler-32 足够可靠

```cpp
// Adler-32 校验和计算示例
#include <zlib.h>

uint32_t CalculateAdler32(const void *data, size_t size) {
    // zlib 的 adler32 函数
    // 第一个参数是初始值，通常传 adler32(0, Z_NULL, 0) 获取
    uLong checksum = adler32(0L, Z_NULL, 0);
    checksum = adler32(checksum, 
                       static_cast<const Bytef *>(data), 
                       static_cast<uInt>(size));
    return static_cast<uint32_t>(checksum);
}
```

**`flags`**：文件级标志位，当前定义：

```cpp
enum KAREntryFlags : uint16_t {
    KAR_ENTRY_FLAG_NONE       = 0x0000,  // 无特殊标志（未压缩）
    KAR_ENTRY_FLAG_COMPRESSED = 0x0001,  // 数据经过 zlib deflate 压缩
    // 位 1-15 保留，必须为 0
};
```

**`name_length`**：文件名的 UTF-8 编码字节数。使用 `uint16_t` 表示，最大支持 65535 字节长的文件名——对于任何合理的文件路径都绰绰有余。

### 3.5 索引区整体结构

索引区由 `file_count` 个索引条目连续排列组成，每个条目的大小为 `24 + name_length` 字节：

```
索引区起始（偏移 = index_offset）：
  [条目 0 固定部分 (24 字节)] [条目 0 文件名 (name_length 字节)]
  [条目 1 固定部分 (24 字节)] [条目 1 文件名 (name_length 字节)]
  ...
  [条目 N-1 固定部分 (24 字节)] [条目 N-1 文件名 (name_length 字节)]
索引区结束（偏移 = index_offset + index_size）
```

索引条目之间**不对齐**——因为索引在加载时会一次性读入内存并解析为 C++ 结构体，不需要磁盘级别的对齐优化。

## 四、格式设计的关键决策分析

### 4.1 索引位置：头部 vs 尾部

这是归档格式设计中最基础的决策。两种方案各有优劣：

| 维度 | 头部索引 | 尾部索引 |
|------|---------|---------|
| 打包流程 | 必须预知所有文件大小，先生成索引再写数据；或两遍扫描 | 流式写入：先写数据，记录偏移量，最后生成索引 |
| 读取速度 | 打开归档后立即可读索引，无需 seek 到尾部 | 需要先 seek 到头部读取 `index_offset`，再 seek 到尾部读取索引 |
| 嵌入 EXE | 困难——EXE 文件开头有 PE/ELF 头，归档头部会冲突 | 简单——XP3 正是用尾部索引实现嵌入 EXE 的 |
| 流式传输 | 可以边下载边解析索引（HTTP Range 请求） | 必须先获取文件尾部才能解析索引 |

**KAR 选择尾部索引的原因：**

1. 打包工具实现更简单——用 Python 脚本流式写出数据，最后追加索引
2. 与 KrKr2 现有的 XP3、ZIP、7z 格式保持一致的设计风格
3. 尾部索引的"读取时多一次 seek"在实际应用中可以忽略——归档通常在启动时一次性加载索引

### 4.2 压缩策略：逐文件 vs 分段 vs 固实

```
逐文件压缩（KAR 采用）：
  每个文件独立压缩，可单独解压任意文件
  ┌────────┐ ┌────────┐ ┌────────┐
  │ 文件A  │ │ 文件B  │ │ 文件C  │
  │ (压缩) │ │ (未压) │ │ (压缩) │
  └────────┘ └────────┘ └────────┘

分段压缩（XP3 采用）：
  每个文件拆成多个段，每段独立压缩
  ┌──────┬──────┬──────┐
  │ 段1  │ 段2  │ 段3  │  ← 文件A的三个压缩段
  │(压缩)│(压缩)│(压缩)│
  └──────┴──────┴──────┘

固实压缩（7z 采用）：
  多个文件拼接后整体压缩
  ┌──────────────────────┐
  │ 文件A+B+C 整体压缩   │  解压文件C需要先解压A和B
  └──────────────────────┘
```

KAR 选择逐文件压缩：
- **优点**：实现最简单；随机访问任意文件无需解压其他文件；压缩/解压可以并行化
- **缺点**：压缩率不如固实压缩（无法利用文件间的冗余）；不如分段压缩的随机访问粒度细（大文件需要完整解压）
- **权衡**：对于游戏资源（图片、音频已经是压缩格式），逐文件压缩的压缩率劣势不明显

### 4.3 字节序选择

多字节整数（`uint16_t`、`uint32_t`）在文件中的存储需要确定字节序（Byte Order，即一个多字节值在内存/文件中是高位字节在前还是低位字节在前）：

- **小端序（Little-Endian）**：低位字节在前。x86/x64、ARM（默认模式）使用小端序
- **大端序（Big-Endian）**：高位字节在前。网络协议、Java class 文件使用大端序

**KAR 选择小端序**（Little-Endian），原因：

1. KrKr2 的目标平台（x86_64 Windows/Linux、ARM64 Android/macOS）全部使用小端序
2. XP3 格式使用小端序
3. ZIP 格式使用小端序
4. 在小端平台上，小端格式可以直接 `memcpy` 读写，不需要字节序转换

```cpp
// 小端序读取辅助函数（在大端平台上需要翻转字节）
inline uint32_t ReadLE32(const uint8_t *data) {
    return static_cast<uint32_t>(data[0])       |
           static_cast<uint32_t>(data[1]) << 8  |
           static_cast<uint32_t>(data[2]) << 16 |
           static_cast<uint32_t>(data[3]) << 24;
}

inline uint16_t ReadLE16(const uint8_t *data) {
    return static_cast<uint16_t>(data[0])       |
           static_cast<uint16_t>(data[1]) << 8;
}

// 小端序写入辅助函数
inline void WriteLE32(uint8_t *data, uint32_t value) {
    data[0] = static_cast<uint8_t>(value);
    data[1] = static_cast<uint8_t>(value >> 8);
    data[2] = static_cast<uint8_t>(value >> 16);
    data[3] = static_cast<uint8_t>(value >> 24);
}

inline void WriteLE16(uint8_t *data, uint16_t value) {
    data[0] = static_cast<uint8_t>(value);
    data[1] = static_cast<uint8_t>(value >> 8);
}
```

### 4.4 文件名规范化

KrKr2 的 `tTVPArchive` 基类在构造时会调用 `Normalize()` 方法，对所有文件名进行规范化处理：

- 路径分隔符统一为 `/`（Windows 的 `\` 转换为 `/`）
- 文件名转为小写（KrKr2 对文件名不区分大小写）
- 去除前导 `./`

因此，KAR 格式的打包工具在存储文件名时应该**预先规范化**——虽然读取器会再次规范化，但预处理可以减少不一致带来的哈希冲突。

```python
# Python 打包工具中的文件名规范化
def normalize_kar_name(path: str) -> str:
    """将文件路径规范化为 KAR 格式标准形式"""
    # 统一路径分隔符
    path = path.replace('\\', '/')
    # 去除前导 ./
    while path.startswith('./'):
        path = path[2:]
    # 转小写
    path = path.lower()
    return path
```

### 5.1 KAR 格式规范 v1.0

2. 字符编码：UTF-8（文件名）
3. 最大文件大小：4GB（32 位偏移量限制）
4. 最大文件数量：2^32 - 1（实际受索引大小限制）

  [文件头 (32 字节)]

  位 0:  数据已压缩 (zlib deflate)
  位 1-15: 保留 (必须为 0)

## 六、Python 打包工具原型

为了在下一节实现 C++ 读取器之前能够生成测试用的 KAR 文件，我们先编写一个简单的 Python 打包工具。

### 6.1 完整打包脚本

```python
#!/usr/bin/env python3
"""
kar_pack.py — KAR 归档打包工具
用法: python kar_pack.py <输出文件> <输入目录>
示例: python kar_pack.py game.kar ./data/
"""

import os
import sys
import struct
import zlib

# ========== 常量定义 ==========
KAR_MAGIC = b'KAR\r\n\x1a\x00\x00'   # 8 字节魔数
KAR_VERSION_MAJOR = 1
KAR_VERSION_MINOR = 0
KAR_HEADER_SIZE = 32                    # 文件头固定 32 字节
KAR_ALIGNMENT = 16                      # 数据对齐边界
KAR_ENTRY_FLAG_COMPRESSED = 0x0001      # 文件级压缩标志

# 不压缩的文件扩展名（这些格式已经是压缩的）
NO_COMPRESS_EXTS = {'.png', '.jpg', '.jpeg', '.ogg', '.mp3',
                    '.mp4', '.webm', '.webp', '.flac'}

def align_up(value: int, alignment: int) -> int:
    """将 value 向上对齐到 alignment 的倍数"""
    return (value + alignment - 1) & ~(alignment - 1)

def normalize_name(path: str) -> str:
    """规范化文件名：统一分隔符、去前导 ./、转小写"""
    path = path.replace('\\', '/')
    while path.startswith('./'):
        path = path[2:]
    return path.lower()

def pack_kar(output_path: str, input_dir: str):
    """将 input_dir 下的所有文件打包为 KAR 归档"""
    
    # 1. 收集所有文件
    files = []
    for root, dirs, filenames in os.walk(input_dir):
        for fname in sorted(filenames):
            full_path = os.path.join(root, fname)
            rel_path = os.path.relpath(full_path, input_dir)
            files.append((normalize_name(rel_path), full_path))
    
    if not files:
        print(f"错误：目录 {input_dir} 中没有文件")
        sys.exit(1)
    
    print(f"打包 {len(files)} 个文件到 {output_path}")
    
    # 2. 准备输出文件
    with open(output_path, 'wb') as out:
        # 写入占位头部（稍后回填）
        out.write(b'\x00' * KAR_HEADER_SIZE)
        
        # 3. 逐文件写入数据区
        entries = []  # 存储每个文件的索引信息
        
        for name, full_path in files:
            # 读取原始数据
            with open(full_path, 'rb') as f:
                raw_data = f.read()
            
            original_size = len(raw_data)
            checksum = zlib.adler32(raw_data) & 0xFFFFFFFF
            
            # 决定是否压缩
            ext = os.path.splitext(name)[1].lower()
            should_compress = ext not in NO_COMPRESS_EXTS
            
            if should_compress and original_size > 64:
                # 尝试压缩
                compressed = zlib.compress(raw_data, 6)
                # 只有压缩后确实更小才使用压缩版本
                if len(compressed) < original_size * 0.95:
                    write_data = compressed
                    flags = KAR_ENTRY_FLAG_COMPRESSED
                else:
                    write_data = raw_data
                    flags = 0
            else:
                write_data = raw_data
                flags = 0
            
            # 记录当前偏移量
            data_offset = out.tell()
            
            # 写入数据
            out.write(write_data)
            
            # 对齐填充
            aligned_pos = align_up(out.tell(), KAR_ALIGNMENT)
            padding = aligned_pos - out.tell()
            if padding > 0:
                out.write(b'\x00' * padding)
            
            # 记录索引信息
            entries.append({
                'data_offset': data_offset,
                'compressed_size': len(write_data),
                'original_size': original_size,
                'adler32': checksum,
                'flags': flags,
                'name': name.encode('utf-8'),
            })
            
            status = "压缩" if flags else "存储"
            ratio = len(write_data) / original_size * 100 if original_size > 0 else 100
            print(f"  {status} {name} ({original_size} -> "
                  f"{len(write_data)} 字节, {ratio:.1f}%)")
        
        # 4. 写入索引区
        index_offset = out.tell()
        
        for entry in entries:
            name_bytes = entry['name']
            # 固定部分 (24 字节)
            out.write(struct.pack('<IIIIHHI',
                entry['data_offset'],
                entry['compressed_size'],
                entry['original_size'],
                entry['adler32'],
                entry['flags'],
                len(name_bytes),
                0  # reserved
            ))
            # 变长文件名
            out.write(name_bytes)
        
        index_size = out.tell() - index_offset
        
        # 5. 回填文件头
        out.seek(0)
        out.write(KAR_MAGIC)                               # magic (8)
        out.write(struct.pack('<HH', 
                  KAR_VERSION_MAJOR, KAR_VERSION_MINOR))   # version (4)
        out.write(struct.pack('<I', 0))                    # flags (4)
        out.write(struct.pack('<I', len(entries)))          # file_count (4)
        out.write(struct.pack('<I', index_offset))         # index_offset (4)
        out.write(struct.pack('<I', index_size))           # index_size (4)
        out.write(struct.pack('<I', 0))                    # reserved (4)
    
    total_size = os.path.getsize(output_path)
    print(f"\n完成！归档大小: {total_size} 字节, "
          f"文件数: {len(entries)}, "
          f"索引大小: {index_size} 字节")

if __name__ == '__main__':
    if len(sys.argv) != 3:
        print(f"用法: python {sys.argv[0]} <输出文件.kar> <输入目录>")
        sys.exit(1)
    pack_kar(sys.argv[1], sys.argv[2])
```

### 6.2 验证脚本——读取并检验 KAR 头部

```python
#!/usr/bin/env python3
"""
kar_verify.py — KAR 归档验证工具
用法: python kar_verify.py <归档文件.kar>
"""

import struct
import sys
import zlib

KAR_MAGIC = b'KAR\r\n\x1a\x00\x00'

def verify_kar(path: str):
    """读取 KAR 归档并验证完整性"""
    with open(path, 'rb') as f:
        # 读取文件头
        header = f.read(32)
        if len(header) < 32:
            print("错误：文件太小，不是有效的 KAR 归档")
            return False
        
        magic = header[0:8]
        if magic != KAR_MAGIC:
            print(f"错误：魔数不匹配 (期望 {KAR_MAGIC!r}, "
                  f"实际 {magic!r})")
            return False
        
        ver_major, ver_minor = struct.unpack_from('<HH', header, 8)
        flags, = struct.unpack_from('<I', header, 12)
        file_count, = struct.unpack_from('<I', header, 16)
        index_offset, = struct.unpack_from('<I', header, 20)
        index_size, = struct.unpack_from('<I', header, 24)
        
        print(f"KAR 归档: {path}")
        print(f"  版本: {ver_major}.{ver_minor}")
        print(f"  标志: 0x{flags:08X}")
        print(f"  文件数: {file_count}")
        print(f"  索引偏移: 0x{index_offset:08X}")
        print(f"  索引大小: {index_size} 字节")
        print()
        
        # 读取索引
        f.seek(index_offset)
        index_data = f.read(index_size)
        
        offset = 0
        for i in range(file_count):
            # 解析固定部分 (24 字节)
            entry = struct.unpack_from('<IIIIHHI', index_data, offset)
            data_off, comp_size, orig_size, checksum, \
                entry_flags, name_len, reserved = entry
            offset += 24
            
            # 解析文件名
            name = index_data[offset:offset + name_len].decode('utf-8')
            offset += name_len
            
            compressed = bool(entry_flags & 0x0001)
            
            # 读取并验证数据
            f.seek(data_off)
            file_data = f.read(comp_size)
            
            if compressed:
                raw_data = zlib.decompress(file_data)
            else:
                raw_data = file_data
            
            actual_checksum = zlib.adler32(raw_data) & 0xFFFFFFFF
            ok = actual_checksum == checksum
            
            status = "✓" if ok else "✗"
            comp_str = "压缩" if compressed else "存储"
            print(f"  [{status}] {name}")
            print(f"       {comp_str} | 原始:{orig_size} "
                  f"压缩后:{comp_size} | "
                  f"Adler32: 0x{checksum:08X}")
            
            if not ok:
                print(f"       校验失败！期望 0x{checksum:08X}, "
                      f"实际 0x{actual_checksum:08X}")
        
        print(f"\n验证完成: {file_count} 个文件")
        return True

if __name__ == '__main__':
    if len(sys.argv) != 2:
        print(f"用法: python {sys.argv[0]} <归档文件.kar>")
        sys.exit(1)
    verify_kar(sys.argv[1])
```

### 步骤 1：手动构造一个最小 KAR 文件

不用打包工具，用 Python 直接构造一个只包含 1 个文件的 KAR 归档：

```python
import struct
import zlib

# 创建一个包含 "Hello, KAR!" 的归档
content = b"Hello, KAR!"
checksum = zlib.adler32(content) & 0xFFFFFFFF

with open("minimal.kar", "wb") as f:
    # === 文件头 ===
    f.write(b'KAR\r\n\x1a\x00\x00')      # magic
    f.write(struct.pack('<HH', 1, 0))       # version 1.0
    f.write(struct.pack('<I', 0))           # flags
    f.write(struct.pack('<I', 1))           # file_count = 1
    # index_offset 和 index_size 稍后回填
    f.write(struct.pack('<I', 0))           # index_offset (占位)
    f.write(struct.pack('<I', 0))           # index_size (占位)
    f.write(struct.pack('<I', 0))           # reserved
    
    # === 数据区 ===
    data_offset = f.tell()  # = 32
    f.write(content)
    # 对齐到 16 字节
    while f.tell() % 16 != 0:
        f.write(b'\x00')
    
    # === 索引区 ===
    index_offset = f.tell()
    name = b"hello.txt"
    f.write(struct.pack('<IIIIHHI',
        data_offset,            # data_offset
        len(content),           # compressed_size (未压缩)
        len(content),           # original_size
        checksum,               # adler32
        0,                      # flags (未压缩)
        len(name),              # name_length
        0                       # reserved
    ))
    f.write(name)
    index_size = f.tell() - index_offset
    
    # === 回填头部 ===
    f.seek(20)
    f.write(struct.pack('<II', index_offset, index_size))

print(f"创建 minimal.kar: {data_offset=}, {index_offset=}, "
      f"{index_size=}, checksum=0x{checksum:08X}")

    hex_str = ' '.join(f'{b:02X}' for b in data[i:i+16])
    ascii_str = ''.join(chr(b) if 32 <= b < 127 else '.' for b in data[i:i+16])
    print(f'{i:04X}  {hex_str:<48s}  {ascii_str}')
"
```

## 常见错误与解决方案

### 错误 1：忘记对齐填充导致数据偏移量错误

**症状：** 打包工具生成的归档中，第二个及之后的文件读取出乱码或崩溃。

**原因：** 数据区中文件之间没有进行 16 字节对齐填充，导致索引条目中记录的 `data_offset` 与实际数据位置不匹配。

**解决：** 确保每个文件数据写入后，用 `align_up()` 计算对齐后的位置，并写入相应的 padding 字节。在记录下一个文件的 `data_offset` 之前，用 `f.tell()` 获取当前位置。

### 错误 2：struct.pack 格式字符串与字段顺序不匹配

**症状：** 读取归档时字段值明显异常（如 `file_count` 为几百万，`version` 为 0 等）。

**原因：** Python `struct.pack`/`struct.unpack` 的格式字符串必须严格匹配字段的类型和顺序。常见错误是把 `uint16_t` 写成 `I`（4 字节）而非 `H`（2 字节），导致后续所有字段的偏移量错位。

**解决：** 对照结构体定义，逐字段核对格式字符：
- `uint8_t` → `B`
- `uint16_t` → `H`
- `uint32_t` → `I`
- 前缀 `<` 表示小端序

### 错误 3：文件名编码问题——中文路径乱码

**症状：** 包含中文文件名的归档在读取时文件名显示为乱码或找不到文件。

**原因：** 打包时使用了系统默认编码（如 GBK）而非 UTF-8 编码文件名。

**解决：** 始终使用 `name.encode('utf-8')` 编码文件名，读取时使用 `name_bytes.decode('utf-8')` 解码。在 C++ 读取器中，需要将 UTF-8 转换为 KrKr2 内部使用的宽字符（`wchar_t`/`tjs_char`），使用 `TVPUtf8ToWideChar()`。

## 本节小结

- **自定义归档格式**的典型场景包括资源保护、性能优化、嵌入元数据和教学实践
- **KAR 格式**采用 32 字节固定头部 + 数据区（16 字节对齐）+ 尾部索引的三段布局
- **格式设计必须从接口出发**——`tTVPArchive` 基类要求归档能按索引枚举文件名和创建流，这决定了索引必须是有序数组且包含数据偏移量
- **设计决策的取舍**：尾部索引（简化打包）、逐文件压缩（简化随机访问）、小端序（匹配目标平台）、Adler-32（速度优先）
- **Python 打包工具**可以在几分钟内生成测试用的 KAR 归档，为下一节的 C++ 读取器提供测试数据

## 练习题与答案

### 题目 1：格式布局分析

假设一个 KAR 归档包含 3 个文件：
- `a.txt`（原始 100 字节，不压缩）
- `b.dat`（原始 200 字节，压缩后 150 字节）
- `c.tjs`（原始 50 字节，不压缩）

请计算：
1. 文件头之后，文件 `a.txt` 的数据起始偏移量
2. 文件 `b.dat` 的数据起始偏移量
3. 文件 `c.tjs` 的数据起始偏移量
4. 索引区的起始偏移量

<details>
<summary>查看答案</summary>

1. **`a.txt` 数据起始偏移量 = 32**（即 `KAR_HEADER_SIZE`，文件头之后立即开始）

2. **`b.dat` 数据起始偏移量**：
   - `a.txt` 数据占 100 字节，起始于偏移 32
   - 32 + 100 = 132，对齐到 16 字节：`AlignUp(132, 16) = 144`（即 0x0090）
   - 所以 `b.dat` 起始于偏移 **144**

3. **`c.tjs` 数据起始偏移量**：
   - `b.dat` 压缩后数据占 150 字节，起始于偏移 144
   - 144 + 150 = 294，对齐到 16 字节：`AlignUp(294, 16) = 304`（即 0x0130）
   - 所以 `c.tjs` 起始于偏移 **304**

4. **索引区起始偏移量**：
   - `c.tjs` 数据占 50 字节，起始于偏移 304
   - 304 + 50 = 354，对齐到 16 字节：`AlignUp(354, 16) = 368`（即 0x0170）
   - 索引区起始于偏移 **368**

注意：数据区中每个文件之后都有对齐填充，但索引区条目之间不需要对齐。

</details>

### 题目 2：修改 KAR 格式以支持 64 位偏移量

当前 KAR 格式使用 `uint32_t` 存储偏移量，最大支持 4GB 的归档。如果需要支持超过 4GB 的大型归档，你需要修改哪些字段？修改后的头部结构体大小是多少？还需要注意什么兼容性问题？

<details>
<summary>查看答案</summary>

需要修改以下字段从 `uint32_t` 扩展到 `uint64_t`：

```cpp
struct KARHeader64 {
    char     magic[8];           // 不变 (8 字节)
    uint16_t version_major;      // 改为 2（不兼容变更）
    uint16_t version_minor;      // 不变 (2 字节)
    uint32_t flags;              // 不变 (4 字节)
    uint32_t file_count;         // 不变——文件数量用 32 位足够 (4 字节)
    uint64_t index_offset;       // 扩展！4→8 字节
    uint64_t index_size;         // 扩展！4→8 字节
    uint32_t reserved;           // 不变 (4 字节)
};
// sizeof(KARHeader64) == 40（从 32 扩展到 40）
```

索引条目也需要修改：

```cpp
struct KARIndexEntry64 {
    uint64_t data_offset;        // 扩展！4→8 字节
    uint64_t compressed_size;    // 扩展！4→8 字节
    uint64_t original_size;      // 扩展！4→8 字节
    uint32_t adler32;            // 不变
    uint16_t flags;              // 不变
    uint16_t name_length;        // 不变
    uint32_t reserved;           // 不变
};
// sizeof(KARIndexEntry64) == 36（从 24 扩展到 36）
```

**兼容性注意事项：**

1. **版本号必须变更**：`version_major` 改为 2，因为头部大小和索引条目大小都变了，v1 读取器无法正确解析 v2 格式
2. **v2 读取器应同时支持 v1**：读取时先检查 `version_major`，如果是 1 则按 32 字节头部 / 24 字节条目解析，如果是 2 则按 40 字节头部 / 36 字节条目解析
3. **Python 打包工具中的 struct 格式字符**：`uint64_t` 对应 `Q`（`struct.pack('<Q', value)`）

</details>

### 题目 3：设计一个带元数据标签的索引扩展

假设你需要在 KAR 格式中为每个文件添加一个"资源类型标签"（如 `"texture"`、`"script"`、`"audio"`），允许引擎按类型批量查询文件。请设计扩展方案，要求：
1. 与现有 v1.0 格式向后兼容
2. 标签为 UTF-8 字符串，最大 32 字节
3. 给出修改后的索引条目结构体

<details>
<summary>查看答案</summary>

**方案：利用 `reserved` 字段和次版本号实现向后兼容扩展**

```cpp
// KAR v1.1 索引条目（向后兼容 v1.0）
struct KARIndexEntryV11 {
    // === v1.0 部分（24 字节，完全不变）===
    uint32_t data_offset;
    uint32_t compressed_size;
    uint32_t original_size;
    uint32_t adler32;
    uint16_t flags;
    uint16_t name_length;
    uint32_t tag_length;        // 原来的 reserved 字段，现在用于存储标签长度
    
    // === v1.0 变长文件名 ===
    // char name[name_length];
    
    // === v1.1 新增：变长标签 ===
    // char tag[tag_length];     // UTF-8 编码的标签字符串
};
```

**向后兼容性分析：**

- **v1.0 读取器读 v1.1 归档**：`reserved` 字段（现在是 `tag_length`）会被 v1.0 读取器忽略。但是，v1.0 读取器在解析索引时，会在 24 字节固定部分之后读取 `name_length` 字节的文件名，然后期望下一个条目紧随其后。如果 v1.1 在文件名之后追加了 `tag_length` 字节的标签，v1.0 读取器的偏移量就会错位。

  **解决方法**：不修改原有 `reserved` 的语义。改为在文件级 `flags` 中添加标志位，在文件名之后附加标签数据：

```cpp
enum KAREntryFlags : uint16_t {
    KAR_ENTRY_FLAG_COMPRESSED  = 0x0001,
    KAR_ENTRY_FLAG_HAS_TAG     = 0x0002,  // v1.1 新增
};

// 索引条目布局（v1.1）：
// [固定部分 24 字节]
// [文件名 name_length 字节]
// 如果 flags & KAR_ENTRY_FLAG_HAS_TAG：
//   [uint16_t tag_length]  (2 字节)
//   [char tag[tag_length]]
```

这样 v1.0 读取器不认识 `KAR_ENTRY_FLAG_HAS_TAG` 标志位，但 `reserved` 字段仍为 0，不影响偏移量计算。不过 v1.0 读取器仍然会因为不知道标签部分的存在而偏移错位。

**真正的向后兼容方案**：将版本号改为 v1.1（次版本号变更），并要求 v1.0 读取器在遇到 `version_minor > 0` 时，使用 `index_size` 和 `file_count` 来计算每个条目的平均大小，或者直接拒绝打开。更实际的做法是：**在 `reserved` 字段中存储单个条目的完整大小（含变长部分后的扩展部分大小），让旧版读取器可以正确跳过不认识的数据**。

</details>
