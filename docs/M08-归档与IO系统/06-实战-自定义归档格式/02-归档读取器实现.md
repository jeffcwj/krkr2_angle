## 一、实现思路总览

### 1.1 从 XP3 实现中学到的模式

在动手写代码之前，让我们先回顾 XP3 归档读取器的架构，它是我们实现 KAR 读取器的"参考模板"。

XP3 实现由两个核心类构成：

```
tTVPXP3Archive（归档类）
├── 构造函数：打开 .xp3 文件 → 校验魔数 → 定位索引 → 解析索引到内存
├── GetCount()：返回文件总数
├── GetName(idx)：返回第 idx 个文件的名称
├── CreateStreamByIndex(idx)：创建第 idx 个文件的读取流
└── ItemVector：存储所有文件的元数据（名称、哈希、分段信息）

tTVPXP3ArchiveStream（流类）
├── 构造函数：接收归档引用 + 文件索引
├── Read()：从当前位置读取数据（自动处理分段 + 解压）
├── Seek()：在文件内移动读取位置
└── GetSize()：返回文件原始大小
```

我们的 KAR 实现将沿用完全相同的双类架构：

```
tTVPKARArchive ←→ tTVPXP3Archive 对应
tTVPKARArchiveStream ←→ tTVPXP3ArchiveStream 对应
```

### 1.2 KAR 与 XP3 的关键差异

虽然架构相同，但 KAR 的实现比 XP3 简单得多——这正是教学设计的目标：

| 维度 | XP3 | KAR |
|------|-----|-----|
| 文件结构 | 多段（Segment）——每个文件可拆分为多个压缩段 | 单段——每个文件只有一整块数据 |
| 索引位置 | 头部记录索引偏移量（11 字节后） | 头部 `index_offset` 字段直接指向尾部索引 |
| 索引格式 | 嵌套 chunk（`File` → `info` + `segm` + `adlr`） | 扁平结构——固定 24 字节头 + 变长文件名 |
| 校验和 | Adler-32（存在 `adlr` chunk 中） | Adler-32（存在索引条目中） |
| 提取过滤器 | 支持（`tTVPXP3ArchiveExtractionFilter`） | 不支持（简化设计） |
| 句柄缓存 | 有（`TVPArchiveHandleCachePool`） | 不需要（每次新建流） |

### 1.3 实现路线图

我们将按以下顺序逐步实现：

```
步骤 1：定义头文件 KARArchive.h ── 声明两个类
步骤 2：实现魔数校验 TVPIsKARArchive() ── 格式识别
步骤 3：实现构造函数 ── 头部解析 + 索引加载
步骤 4：实现三个虚函数 ── GetCount / GetName / CreateStreamByIndex
步骤 5：实现 tTVPKARArchiveStream ── 数据读取 + 解压
步骤 6：注册到 TVPOpenArchive ── 接入引擎
```

## 二、头文件定义：KARArchive.h

### 2.1 完整头文件

先给出完整的头文件，然后逐段分析每个设计决策：

```cpp
// KARArchive.h — KAR 自定义归档格式读取器
// 文件路径：cpp/core/base/KARArchive.h

#ifndef KARArchiveH
#define KARArchiveH

#include "StorageIntf.h"  // tTVPArchive 基类定义
#include <vector>
#include <cstdint>

// ─────────────────────────────────────────────
// KAR 格式常量
// ─────────────────────────────────────────────

// 魔数：8 字节，用于在文件开头快速识别 KAR 格式
// "KAR\r\n\x1a\x00\x00"
// 设计理由：
//   - "KAR" 三个可读字符，方便 hex dump 时肉眼识别
//   - \r\n 用于检测 FTP 传输时的换行符篡改（与 XP3 相同技巧）
//   - \x1a 是 DOS EOF 标记，防止 type 命令输出乱码
//   - \x00\x00 填充到 8 字节对齐
static const uint8_t KAR_MAGIC[8] = {
    'K', 'A', 'R', '\r', '\n', '\x1a', '\0', '\0'
};

static const uint16_t KAR_VERSION_MAJOR = 1;  // 主版本号
static const uint16_t KAR_VERSION_MINOR = 0;  // 次版本号

// 头部 flags 位定义
static const uint32_t KAR_FLAG_INDEX_COMPRESSED = 0x01;
// bit 0：索引区是否经过 zlib 压缩

// 索引条目 flags 位定义
static const uint16_t KAR_ENTRY_FLAG_COMPRESSED = 0x01;
// bit 0：该文件的数据是否经过 zlib deflate 压缩

// ─────────────────────────────────────────────
// KAR 文件头结构（32 字节）
// ─────────────────────────────────────────────
#pragma pack(push, 1)

struct KARHeader {
    uint8_t  magic[8];         // "KAR\r\n\x1a\x00\x00"
    uint16_t version_major;    // 主版本号（当前 = 1）
    uint16_t version_minor;    // 次版本号（当前 = 0）
    uint32_t flags;            // 头部标志位
    uint32_t file_count;       // 归档内文件总数
    uint32_t index_offset;     // 索引区在文件中的绝对偏移量
    uint32_t index_size;       // 索引区大小（字节）
    uint32_t reserved;         // 保留字段，必须为 0
};
// sizeof(KARHeader) == 32

#pragma pack(pop)

// ─────────────────────────────────────────────
// KAR 索引条目（内存表示）
// ─────────────────────────────────────────────
struct tTVPKARIndexEntry {
    ttstr    Name;              // 文件名（UTF-8 → ttstr 转换后）
    uint32_t DataOffset;        // 数据在归档文件中的绝对偏移量
    uint32_t CompressedSize;    // 压缩后大小（未压缩时 = OriginalSize）
    uint32_t OriginalSize;      // 原始大小
    uint32_t Adler32;           // 原始数据的 Adler-32 校验和
    uint16_t Flags;             // 条目标志位（bit 0: 是否压缩）

    // 判断数据是否经过压缩
    bool IsCompressed() const {
        return (Flags & KAR_ENTRY_FLAG_COMPRESSED) != 0;
    }
};

// ─────────────────────────────────────────────
// 格式识别函数
// ─────────────────────────────────────────────
extern bool TVPIsKARArchive(const ttstr &name);

// ─────────────────────────────────────────────
// tTVPKARArchive：KAR 归档读取器
// ─────────────────────────────────────────────
class tTVPKARArchive : public tTVPArchive {
public:
    // 构造函数：打开归档文件，解析头部和索引
    // name: 归档文件的完整路径
    tTVPKARArchive(const ttstr &name);

    ~tTVPKARArchive() override;

    // ── tTVPArchive 纯虚函数实现 ──
    tjs_uint GetCount() override;
    ttstr GetName(tjs_uint idx) override;
    tTJSBinaryStream *CreateStreamByIndex(tjs_uint idx) override;

private:
    // 索引条目列表——构造时一次性加载到内存
    std::vector<tTVPKARIndexEntry> Items;
};

// ─────────────────────────────────────────────
// tTVPKARArchiveStream：KAR 归档内文件流
// ─────────────────────────────────────────────
class tTVPKARArchiveStream : public tTJSBinaryStream {
public:
    // owner: 所属归档（用于获取归档文件名以打开底层流）
    // entry: 要读取的文件的索引条目
    tTVPKARArchiveStream(tTVPKARArchive *owner,
                         const tTVPKARIndexEntry &entry);

    ~tTVPKARArchiveStream() override;

    // ── tTJSBinaryStream 虚函数实现 ──
    tjs_uint64 Seek(tjs_int64 offset, tjs_int whence) override;
    tjs_uint Read(void *buffer, tjs_uint read_size) override;
    tjs_uint Write(const void *buffer, tjs_uint write_size) override;
    tjs_uint64 GetSize() override;

private:
    tTVPKARArchive *Owner;           // 所属归档（弱引用）
    tTVPKARIndexEntry Entry;         // 文件元数据的副本
    std::vector<uint8_t> Data;       // 解压后的完整数据
    tjs_uint64 CurPos;               // 当前读取位置
};

#endif // KARArchiveH
```

### 2.2 设计决策详解

**为什么 `tTVPKARIndexEntry` 不用 `#pragma pack`？**

磁盘上的索引条目（24 字节 + 变长文件名）和内存中的 `tTVPKARIndexEntry` 是**不同的表示**。磁盘格式用 `#pragma pack(push, 1)` 消除填充以节省空间；内存结构不需要，因为：

1. 内存结构包含 `ttstr`（KrKr2 的字符串类），它内部是指针 + 长度，无法 pack
2. 内存对齐的结构访问更快
3. 解析时我们逐字段读取磁盘数据并填入内存结构，而非直接 `memcpy`

**为什么 `tTVPKARArchiveStream` 一次性加载全部数据到 `Data` 向量？**

这是一个有意识的简化设计：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 一次性加载（我们的方案） | 实现简单，`Read`/`Seek` 只需操作内存 | 大文件占内存 |
| 按需读取（XP3 的方案） | 内存效率高 | 需要管理分段、缓存、文件句柄 |

对于教学目的和典型视觉小说资源（单文件 < 10MB），一次性加载完全可行。生产环境中如果需要支持大文件，可以参考 XP3 的 `tTVPXP3ArchiveStream` 的分段按需读取设计。

**为什么 `Owner` 是弱引用（裸指针）而非智能指针？**

`tTVPArchive` 使用 KrKr2 自带的引用计数机制（`AddRef` / `Release`），不使用 `std::shared_ptr`。流对象的生命周期短于归档对象——归档在所有流关闭后才会被释放，因此裸指针在此处是安全的。这与 `tTVPXP3ArchiveStream` 中 `Owner` 的设计完全一致（参见 `XP3Archive.h` 第 163 行）。

## 三、格式识别：TVPIsKARArchive

### 3.1 为什么需要格式识别函数

KrKr2 的 `TVPOpenArchive` 函数（位于 `StorageIntf.cpp`）在打开归档文件时，需要**依次尝试**每种已知格式——先试 XP3，再试 ZIP、TAR、7z，最终都不匹配则报错。我们的 KAR 格式也需要加入这个"识别链"。

格式识别函数的职责很简单：读文件前 8 字节，判断是否匹配 KAR 魔数。

### 3.2 实现代码

```cpp
// KARArchive.cpp — KAR 归档读取器实现
// 文件路径：cpp/core/base/KARArchive.cpp

#include "tjsCommHead.h"    // TJS2 公共头文件（必须第一个 include）
#include "KARArchive.h"
#include "MsgIntf.h"         // TVPThrowExceptionMessage 错误报告
#include "DebugIntf.h"       // TVPAddLog 日志输出
#include "UtilStreams.h"     // tTVPStreamHolder 流持有器

#include <zlib.h>            // uncompress, adler32
#include <cstring>           // memcmp, memcpy
#include <algorithm>         // std::min

// ─────────────────────────────────────────────
// 格式识别
// ─────────────────────────────────────────────
bool TVPIsKARArchive(const ttstr &name) {
    // 尝试打开文件并检查魔数
    try {
        tTVPStreamHolder holder(name);
        // tTVPStreamHolder 是 RAII 包装器，
        // 构造时调用 TVPCreateStream 打开文件，
        // 析构时自动关闭

        tTJSBinaryStream *st = holder.Get();

        // 读取前 8 字节
        uint8_t magic[8] = {};
        st->ReadBuffer(magic, 8);
        // ReadBuffer 在读取不足时会抛异常，
        // 所以文件小于 8 字节会被 catch 捕获

        // 比较魔数
        return memcmp(magic, KAR_MAGIC, 8) == 0;
    } catch (...) {
        // 任何异常（文件不存在、读取失败、太短）都返回 false
        return false;
    }
}
```

**代码解析——为什么用 `try-catch` 包裹全部逻辑？**

格式识别函数会被 `TVPOpenArchive` 对**每个可能的归档文件**调用。如果文件根本不是归档（比如是普通的 `.png` 图片），读取可能会失败。使用 `try-catch` 确保任何异常都被安全吞掉，返回 `false` 表示"不是 KAR 格式"。这与 `TVPIsXP3Archive`（`XP3Archive.cpp` 第 309-317 行）完全相同的模式。

## 四、构造函数：头部解析与索引加载

### 4.1 构造函数整体流程

构造函数是归档读取器最复杂的部分。它需要完成以下步骤：

```
打开归档文件
    │
    ▼
读取并校验 32 字节头部（KARHeader）
    │
    ▼
校验魔数 → 失败则抛异常
    │
    ▼
校验版本号 → 不兼容则抛异常
    │
    ▼
定位索引区（seek 到 index_offset）
    │
    ▼
读取索引数据（如果 flags 指示压缩，先解压）
    │
    ▼
逐条解析索引条目，填充 Items 向量
    │
    ▼
构建名称哈希表（tTVPArchive 基类功能）
```

### 4.2 完整构造函数实现

```cpp
// ─────────────────────────────────────────────
// tTVPKARArchive 构造函数
// ─────────────────────────────────────────────
tTVPKARArchive::tTVPKARArchive(const ttstr &name)
    : tTVPArchive(name)  // 调用基类构造函数，保存归档文件名
{
    TVPAddLog(TJS_W("KAR: 正在打开归档: ") + name);

    // ── 步骤 1：打开归档文件 ──
    tTVPStreamHolder holder(name);
    tTJSBinaryStream *st = holder.Get();

    // ── 步骤 2：读取 32 字节头部 ──
    KARHeader header;
    // 使用 ReadBuffer 精确读取，不足 32 字节会抛异常
    st->ReadBuffer(&header, sizeof(KARHeader));

    // ── 步骤 3：校验魔数 ──
    if (memcmp(header.magic, KAR_MAGIC, 8) != 0) {
        TVPThrowExceptionMessage(
            TJS_W("KAR: 无效的归档文件，魔数不匹配: %1"),
            name);
    }

    // ── 步骤 4：校验版本号 ──
    // 当前只支持 1.x 版本
    if (header.version_major != KAR_VERSION_MAJOR) {
        TVPThrowExceptionMessage(
            TJS_W("KAR: 不支持的版本 %1.%2，当前仅支持 1.x"),
            ttstr((tjs_int)header.version_major),
            ttstr((tjs_int)header.version_minor));
    }

    TVPAddLog(TJS_W("KAR: 版本 ") +
              ttstr((tjs_int)header.version_major) + TJS_W(".") +
              ttstr((tjs_int)header.version_minor) +
              TJS_W(", 文件数: ") +
              ttstr((tjs_int)header.file_count));

    // ── 步骤 5：读取索引区 ──
    st->SetPosition(header.index_offset);
    // 定位到索引区起始位置

    // 读取索引原始数据
    std::vector<uint8_t> index_raw(header.index_size);
    st->ReadBuffer(index_raw.data(), header.index_size);

    // 如果索引区被压缩，需要先解压
    std::vector<uint8_t> index_data;
    if (header.flags & KAR_FLAG_INDEX_COMPRESSED) {
        // 索引被 zlib 压缩
        // 压缩格式：前 4 字节是解压后大小（uint32_t LE）
        // 后面是压缩数据
        if (header.index_size < 4) {
            TVPThrowExceptionMessage(
                TJS_W("KAR: 压缩索引数据太短"));
        }

        // 读取解压后大小
        uint32_t uncompressed_size = 0;
        memcpy(&uncompressed_size, index_raw.data(), 4);

        // 分配解压缓冲区
        index_data.resize(uncompressed_size);

        // 调用 zlib uncompress
        unsigned long dest_len = uncompressed_size;
        int zret = uncompress(
            index_data.data(), &dest_len,
            index_raw.data() + 4,          // 跳过前 4 字节
            (unsigned long)(header.index_size - 4));

        if (zret != Z_OK || dest_len != uncompressed_size) {
            TVPThrowExceptionMessage(
                TJS_W("KAR: 索引解压失败，zlib 返回 %1"),
                ttstr((tjs_int)zret));
        }

        TVPAddLog(TJS_W("KAR: 索引解压完成，") +
                  ttstr((tjs_int)header.index_size) +
                  TJS_W(" → ") +
                  ttstr((tjs_int)uncompressed_size) +
                  TJS_W(" 字节"));
    } else {
        // 索引未压缩，直接使用
        index_data = std::move(index_raw);
    }

    // ── 步骤 6：解析索引条目 ──
    Items.reserve(header.file_count);
    const uint8_t *ptr = index_data.data();
    const uint8_t *end = ptr + index_data.size();

    for (uint32_t i = 0; i < header.file_count; i++) {
        // 每个条目的磁盘布局（共 24 字节固定部分）：
        //   [0..3]   data_offset      (uint32_t)
        //   [4..7]   compressed_size   (uint32_t)
        //   [8..11]  original_size     (uint32_t)
        //   [12..15] adler32           (uint32_t)
        //   [16..17] flags             (uint16_t)
        //   [18..19] name_length       (uint16_t)
        //   [20..23] reserved          (uint32_t)
        //   [24..]   name              (UTF-8, name_length 字节)

        if (ptr + 24 > end) {
            TVPThrowExceptionMessage(
                TJS_W("KAR: 索引数据截断，在第 %1 个条目处"),
                ttstr((tjs_int)i));
        }

        tTVPKARIndexEntry entry;

        // 使用 memcpy 读取各字段（避免对齐问题）
        memcpy(&entry.DataOffset,     ptr + 0,  4);
        memcpy(&entry.CompressedSize, ptr + 4,  4);
        memcpy(&entry.OriginalSize,   ptr + 8,  4);
        memcpy(&entry.Adler32,        ptr + 12, 4);
        memcpy(&entry.Flags,          ptr + 16, 2);

        uint16_t name_length = 0;
        memcpy(&name_length, ptr + 18, 2);

        ptr += 24;  // 跳过固定部分

        // 读取文件名（UTF-8 编码）
        if (ptr + name_length > end) {
            TVPThrowExceptionMessage(
                TJS_W("KAR: 文件名数据截断，在第 %1 个条目处"),
                ttstr((tjs_int)i));
        }

        // UTF-8 → ttstr 转换
        // ttstr 内部使用 UTF-16（Windows）或 UTF-32（其他平台）
        std::string utf8_name(
            reinterpret_cast<const char*>(ptr), name_length);
        entry.Name = ttstr(utf8_name.c_str());

        // 规范化文件名（统一大小写、路径分隔符）
        // tTVPArchive 基类提供了此功能
        tTVPArchive::NormalizeInArchiveStorageName(entry.Name);

        ptr += name_length;

        Items.push_back(std::move(entry));
    }

    // ── 步骤 7：构建名称哈希表 ──
    // tTVPArchive 基类的哈希表在 GetCount/GetName 首次调用时惰性构建
    // 我们不需要手动操作

    TVPAddLog(TJS_W("KAR: 归档加载完成，共 ") +
              ttstr((tjs_int)Items.size()) +
              TJS_W(" 个文件"));
}
```

### 4.3 关键细节解析

**`#pragma pack(push, 1)` 与 `memcpy` 的取舍**

我们对 `KARHeader` 使用了 `#pragma pack(push, 1)`，这意味着可以直接 `ReadBuffer(&header, 32)` 一次性读入。但对索引条目，我们选择逐字段 `memcpy`。为什么？

```cpp
// 方案 A：pack + 整体读取（KARHeader 使用此方案）
#pragma pack(push, 1)
struct OnDiskEntry { uint32_t a; uint16_t b; ... };
#pragma pack(pop)
st->ReadBuffer(&entry, sizeof(OnDiskEntry));
// 优点：代码简洁
// 缺点：结构体内含 ttstr 等复杂类型时不可用

// 方案 B：逐字段 memcpy（索引条目使用此方案）
memcpy(&entry.DataOffset, ptr + 0, 4);
memcpy(&entry.Flags, ptr + 16, 2);
// 优点：不受对齐和内存布局限制
// 缺点：代码稍显冗长
```

经验法则：磁盘结构用 packed struct 直接读；内存结构用 `memcpy` 逐字段填充。

**为什么用 `std::vector<uint8_t>` 而非 `new uint8_t[]`？**

XP3 的实现大量使用 `new/delete[]`（这是 2000 年代的 C++ 风格）。我们的 KAR 实现使用现代 C++17 风格——`std::vector` 自动管理内存，即使抛出异常也不会泄漏。对比：

```cpp
// XP3 风格（需要手动清理）
tjs_uint8 *indexdata = nullptr;
try {
    indexdata = new tjs_uint8[index_size];
    // ... 使用 indexdata ...
} catch (...) {
    delete[] indexdata;  // 必须手动清理
    throw;
}
delete[] indexdata;

// KAR 风格（RAII 自动清理）
std::vector<uint8_t> index_data(index_size);
// ... 使用 index_data ...
// 超出作用域自动释放，无需 try-catch
```

**`NormalizeInArchiveStorageName` 的作用**

这是 `tTVPArchive` 基类提供的静态方法（位于 `StorageIntf.cpp`），它会：

1. 将路径分隔符统一为 `/`
2. 将文件名转为小写（KrKr2 的文件名查找是大小写不敏感的）
3. 去除前导 `./`

这样当脚本层请求 `Storages.open("data>BGM/Title.ogg")` 时，引擎会将 `BGM/Title.ogg` 规范化为 `bgm/title.ogg`，然后在归档的哈希表中查找匹配。

## 六、流类实现：tTVPKARArchiveStream

### 6.1 构造函数——数据加载与解压

流类的构造函数负责将文件数据从归档中读出并（如需要）解压到内存：

```cpp
// ─────────────────────────────────────────────
// tTVPKARArchiveStream 实现
// ─────────────────────────────────────────────

tTVPKARArchiveStream::tTVPKARArchiveStream(
    tTVPKARArchive *owner,
    const tTVPKARIndexEntry &entry)
    : Owner(owner)
    , Entry(entry)
    , CurPos(0)
{
    // ── 步骤 1：打开归档文件 ──
    // 每次创建流都重新打开归档文件
    // （生产环境可优化为文件句柄缓存，参考 XP3 的做法）
    tTVPStreamHolder holder(owner->GetArchiveName());
    tTJSBinaryStream *st = holder.Get();

    // ── 步骤 2：定位到文件数据区 ──
    st->SetPosition(entry.DataOffset);

    // ── 步骤 3：读取原始数据 ──
    std::vector<uint8_t> raw(entry.CompressedSize);
    st->ReadBuffer(raw.data(), entry.CompressedSize);

    if (entry.IsCompressed()) {
        // ── 步骤 4a：解压数据 ──
        Data.resize(entry.OriginalSize);

        unsigned long dest_len = entry.OriginalSize;
        int zret = uncompress(
            Data.data(), &dest_len,
            raw.data(), (unsigned long)entry.CompressedSize);

        if (zret != Z_OK) {
            TVPThrowExceptionMessage(
                TJS_W("KAR: 文件解压失败: %1, zlib=%2"),
                entry.Name,
                ttstr((tjs_int)zret));
        }

        if (dest_len != entry.OriginalSize) {
            TVPThrowExceptionMessage(
                TJS_W("KAR: 解压后大小不匹配: 期望 %1, 实际 %2"),
                ttstr((tjs_int)entry.OriginalSize),
                ttstr((tjs_int)dest_len));
        }
    } else {
        // ── 步骤 4b：未压缩，直接使用 ──
        Data = std::move(raw);
    }

    // ── 步骤 5：校验 Adler-32 ──
    uint32_t computed = adler32(
        adler32(0L, Z_NULL, 0),  // 初始化 Adler-32
        Data.data(),
        (uInt)Data.size());

    if (computed != entry.Adler32) {
        TVPThrowExceptionMessage(
            TJS_W("KAR: Adler-32 校验失败: %1 (期望 0x%2, 计算 0x%3)"),
            entry.Name,
            ttstr((tjs_int)entry.Adler32),
            ttstr((tjs_int)computed));
    }
}
```

**Adler-32 校验的工作原理**

Adler-32 是 zlib 库提供的一种快速校验和算法（由 Mark Adler 发明，也是 zlib 的作者之一）。它的特点是：

1. **计算极快**——比 CRC32 快约 5-10 倍，因为它只用加法和取模运算
2. **碰撞率稍高**——32 位校验和的碰撞概率约为 1/2³²，但 Adler-32 在特定模式下碰撞率高于 CRC32
3. **适合完整性快速验证**——我们用它来检测传输损坏或存储错误，不用于安全场景

使用两步调用：

```cpp
// 第一步：初始化（返回初始值 1）
uint32_t checksum = adler32(0L, Z_NULL, 0);

// 第二步：计算数据的校验和
checksum = adler32(checksum, data_ptr, data_size);

// 也可以分块计算（流式处理）：
checksum = adler32(checksum, chunk1, chunk1_size);
checksum = adler32(checksum, chunk2, chunk2_size);
// 结果与一次性计算相同
```

### 6.2 Read — 读取数据

```cpp
tjs_uint tTVPKARArchiveStream::Read(
    void *buffer, tjs_uint read_size)
{
    // 计算实际可读取的字节数
    tjs_uint64 remain = Data.size() - CurPos;
    tjs_uint actual = static_cast<tjs_uint>(
        std::min(static_cast<tjs_uint64>(read_size), remain));

    if (actual > 0) {
        // 从内存缓冲区复制数据
        memcpy(buffer, Data.data() + CurPos, actual);
        CurPos += actual;
    }

    // 返回实际读取的字节数
    // 如果到达文件末尾，返回值 < read_size
    return actual;
}
```

因为我们在构造时就将整个文件加载到了 `Data` 向量中，`Read` 的实现极其简单——就是一个带边界检查的 `memcpy`。对比 XP3 的 `Read` 实现（`XP3Archive.cpp` 第 700-800 行），需要处理跨段读取、按需解压、缓存管理等复杂逻辑。

### 6.3 Seek — 移动读取位置

```cpp
tjs_uint64 tTVPKARArchiveStream::Seek(
    tjs_int64 offset, tjs_int whence)
{
    // whence 参数含义（与 C 标准库 fseek 相同）：
    //   TJS_BS_SEEK_SET (0) — 从文件开头计算
    //   TJS_BS_SEEK_CUR (1) — 从当前位置计算
    //   TJS_BS_SEEK_END (2) — 从文件末尾计算

    tjs_int64 new_pos;

    switch (whence) {
    case TJS_BS_SEEK_SET:
        new_pos = offset;
        break;
    case TJS_BS_SEEK_CUR:
        new_pos = static_cast<tjs_int64>(CurPos) + offset;
        break;
    case TJS_BS_SEEK_END:
        new_pos = static_cast<tjs_int64>(Data.size()) + offset;
        break;
    default:
        new_pos = static_cast<tjs_int64>(CurPos);
        break;
    }

    // 边界钳位（clamp）——防止越界
    if (new_pos < 0) {
        new_pos = 0;
    }
    if (new_pos > static_cast<tjs_int64>(Data.size())) {
        new_pos = static_cast<tjs_int64>(Data.size());
    }

    CurPos = static_cast<tjs_uint64>(new_pos);
    return CurPos;
}
```

**为什么 `Seek` 不抛异常而是钳位？**

这是 KrKr2 流接口的约定——`Seek` 超出范围时进行钳位而非报错。调用方通过检查返回值来判断实际位置。这与 POSIX 的 `lseek` 行为一致——`lseek` 允许 seek 到文件末尾之后（用于创建稀疏文件），但 KrKr2 的只读流将其钳位到 `[0, size]` 范围。

### 6.4 Write 与 GetSize

## 九、常见错误与解决方案

### 错误 1：字节序问题导致魔数校验失败

**症状：** 在大端序（Big-Endian）平台上，`KARHeader` 的多字节字段值异常。

**原因：** KAR 格式规定使用小端序（Little-Endian），但我们直接用 `memcpy` 读取字段，在大端序平台上读出的值会字节反转。

**解决方案：**

```cpp
// 添加字节序转换工具函数
inline uint32_t kar_le32(uint32_t val) {
#if defined(__BYTE_ORDER__) && __BYTE_ORDER__ == __ORDER_BIG_ENDIAN__
    // 大端序平台：反转字节序
    return ((val & 0xFF) << 24) |
           ((val & 0xFF00) << 8) |
           ((val & 0xFF0000) >> 8) |
           ((val & 0xFF000000) >> 24);
#else
    // 小端序平台（x86/ARM）：无需转换
    return val;
#endif
}

inline uint16_t kar_le16(uint16_t val) {
#if defined(__BYTE_ORDER__) && __BYTE_ORDER__ == __ORDER_BIG_ENDIAN__
    return ((val & 0xFF) << 8) | ((val & 0xFF00) >> 8);
#else
    return val;
#endif
}

// 在读取头部后使用：
header.file_count = kar_le32(header.file_count);
header.index_offset = kar_le32(header.index_offset);
header.index_size = kar_le32(header.index_size);
```

**实际影响：** KrKr2 支持的四个平台（Windows x64、Linux x64、macOS ARM64、Android ARM64）全部是小端序，所以这个问题目前不会触发。但作为防御性编程，加上转换函数是好习惯。

### 错误 2：UTF-8 文件名包含 BOM 或非法序列

**症状：** 文件名在日志中显示为乱码，或者 `Storages.open()` 找不到文件。

**原因：** 打包工具可能在 UTF-8 文件名前加了 BOM（`EF BB BF`），或文件名包含非法的 UTF-8 序列。

**解决方案：**

```cpp
// 在 UTF-8 → ttstr 转换前，跳过 BOM
const char *name_start =
    reinterpret_cast<const char*>(ptr);
size_t name_len = name_length;

// 跳过 UTF-8 BOM（如果存在）
if (name_len >= 3 &&
    (uint8_t)name_start[0] == 0xEF &&
    (uint8_t)name_start[1] == 0xBB &&
    (uint8_t)name_start[2] == 0xBF) {
    name_start += 3;
    name_len -= 3;
}

std::string utf8_name(name_start, name_len);
entry.Name = ttstr(utf8_name.c_str());
```

### 错误 3：索引区大小为 0 但 file_count > 0

**症状：** 构造函数在解析第一个索引条目时因 `ptr + 24 > end` 抛异常。

**原因：** 归档文件被截断，或头部的 `index_size` 字段损坏。

**解决方案：** 在开始解析索引前添加一致性检查：

```cpp
// 一致性检查：索引区大小至少能容纳所有条目的固定部分
// 每个条目至少 24 字节（固定部分）+ 0 字节（空文件名）
uint32_t min_index_size = header.file_count * 24;
if (index_data.size() < min_index_size) {
    TVPThrowExceptionMessage(
        TJS_W("KAR: 索引区太小 (%1 字节)，")
        TJS_W("无法容纳 %2 个条目（最少需要 %3 字节）"),
        ttstr((tjs_int)index_data.size()),
        ttstr((tjs_int)header.file_count),
        ttstr((tjs_int)min_index_size));
}
```

### 错误 4：解压后 Adler-32 不匹配

**症状：** `tTVPKARArchiveStream` 构造时抛出"Adler-32 校验失败"。

**原因：** 可能是打包工具的 bug——计算校验和时使用了压缩后数据而非原始数据。

**解决方案：** 首先确认打包工具（`kar_pack.py`，见上一节）的校验和计算逻辑：

```python
# 正确：对原始数据计算 Adler-32
checksum = zlib.adler32(original_data) & 0xFFFFFFFF

# 错误：对压缩后数据计算
# checksum = zlib.adler32(compressed_data) & 0xFFFFFFFF
```

如果问题出在已打包的归档文件中，可以临时跳过校验：

```cpp
// 临时调试用——发布前必须移除
#ifdef KAR_SKIP_CHECKSUM
    // 跳过 Adler-32 校验
#else
    if (computed != entry.Adler32) { ... }
#endif
```

### 题目 1：为 KAR 格式添加加密支持

假设你需要为 KAR 格式添加简单的 XOR 加密。请修改 `tTVPKARArchiveStream` 构造函数，在解压之后、Adler-32 校验之前，对数据执行 XOR 解密。加密密钥为 4 字节，存储在 `KARHeader` 的 `reserved` 字段中。

<details>
<summary>查看答案</summary>

```cpp
// 修改 KARHeader：reserved 字段改为 encrypt_key
struct KARHeader {
    // ... 其他字段不变 ...
    uint32_t encrypt_key;  // XOR 加密密钥（0 = 未加密）
};

// 修改 tTVPKARArchiveStream 构造函数
// 在 Adler-32 校验之前添加：

// ── 步骤 4.5：XOR 解密（如果有密钥）──
// 注意：需要从归档对象获取密钥
// 方案：在 tTVPKARArchive 中保存 header.encrypt_key，
//       并通过 getter 暴露给 stream
uint32_t key = Owner->GetEncryptKey();
if (key != 0) {
    // 将 4 字节密钥展开为字节数组
    uint8_t key_bytes[4] = {
        static_cast<uint8_t>(key & 0xFF),
        static_cast<uint8_t>((key >> 8) & 0xFF),
        static_cast<uint8_t>((key >> 16) & 0xFF),
        static_cast<uint8_t>((key >> 24) & 0xFF)
    };

    // 对解压后的数据执行 XOR
    for (size_t i = 0; i < Data.size(); i++) {
        Data[i] ^= key_bytes[i % 4];
    }
}

// 然后执行 Adler-32 校验...
// 注意：Adler-32 应该在解密之后计算，
// 即打包时的顺序是：原始数据 → 计算校验和 → 加密 → 压缩
// 读取时的顺序是：解压 → 解密 → 校验
```

**关键设计决策**：校验和应该基于解密后的原始数据计算。如果基于加密后的数据计算，那么修改密钥不需要重新计算校验和，这削弱了完整性保护。

### 题目 2：实现文件句柄缓存

XP3 实现了 8 个文件句柄的缓存池（`TVPArchiveHandleCachePool`），避免每次 `CreateStreamByIndex` 都重新打开归档文件。请为 `tTVPKARArchive` 设计并实现类似的缓存机制。

<details>
<summary>查看答案</summary>

```cpp
// 在 KARArchive.h 中添加缓存管理

// 缓存池大小
static const int KAR_HANDLE_CACHE_SIZE = 4;

class tTVPKARArchive : public tTVPArchive {
public:
    // ... 现有接口不变 ...

    // 获取缓存的文件句柄（如果有的话）
    tTJSBinaryStream *GetCachedHandle();

    // 归还文件句柄到缓存池
    void ReleaseCachedHandle(tTJSBinaryStream *stream);

private:
    // 缓存池：简单的数组 + 互斥锁
    struct CacheSlot {
        tTJSBinaryStream *Stream = nullptr;
        uint32_t Age = 0;
    };

    CacheSlot HandleCache[KAR_HANDLE_CACHE_SIZE];
    uint32_t CacheAge = 0;
    tTJSCriticalSection CacheCS;  // 线程安全
};

// 在 KARArchive.cpp 中实现

tTJSBinaryStream *tTVPKARArchive::GetCachedHandle() {
    tTJSCriticalSectionHolder cs(CacheCS);

    // 查找可用的缓存句柄
    for (int i = 0; i < KAR_HANDLE_CACHE_SIZE; i++) {
        if (HandleCache[i].Stream != nullptr) {
            tTJSBinaryStream *st = HandleCache[i].Stream;
            HandleCache[i].Stream = nullptr;
            return st;  // 返回缓存的句柄
        }
    }

    // 缓存未命中，创建新句柄
    return TVPCreateStream(ArchiveName);
}

void tTVPKARArchive::ReleaseCachedHandle(
    tTJSBinaryStream *stream)
{
    tTJSCriticalSectionHolder cs(CacheCS);

    // 找空槽位
    for (int i = 0; i < KAR_HANDLE_CACHE_SIZE; i++) {
        if (HandleCache[i].Stream == nullptr) {
            HandleCache[i].Stream = stream;
            HandleCache[i].Age = ++CacheAge;
            return;
        }
    }

    // 缓存已满，淘汰最老的
    int oldest_idx = 0;
    uint32_t oldest_age = UINT32_MAX;
    for (int i = 0; i < KAR_HANDLE_CACHE_SIZE; i++) {
        if (HandleCache[i].Age < oldest_age) {
            oldest_age = HandleCache[i].Age;
            oldest_idx = i;
        }
    }

    delete HandleCache[oldest_idx].Stream;
    HandleCache[oldest_idx].Stream = stream;
    HandleCache[oldest_idx].Age = ++CacheAge;
}

// 修改 tTVPKARArchiveStream 构造函数：
// 替换 tTVPStreamHolder holder(owner->GetArchiveName());
// 为：
tTJSBinaryStream *st = Owner->GetCachedHandle();
// ... 使用 st ...
// 在析构函数中归还：
// Owner->ReleaseCachedHandle(st);
// 注意：需要在 stream 对象中保存 st 指针
```

</details>

### 题目 3：支持增量索引更新

假设你需要支持"热更新"——在不重启引擎的情况下，用新的归档文件替换旧的。请描述需要修改哪些组件，并给出 `tTVPKARArchive` 的 `Reload()` 方法实现。

<details>
<summary>查看答案</summary>

需要修改的组件：

1. **`tTVPKARArchive`**：添加 `Reload()` 方法
2. **`StorageIntf.cpp`**：添加归档刷新接口
3. **流管理**：确保旧流对象不受影响（引用计数保护）

```cpp
// 在 KARArchive.h 中添加
class tTVPKARArchive : public tTVPArchive {
public:
    // 重新加载归档（热更新）
    void Reload();

private:
    // 将索引解析逻辑从构造函数提取为独立方法
    void ParseIndex(tTJSBinaryStream *st);

    // 读写锁保护 Items（允许并发读取）
    tTJSCriticalSection IndexCS;
};

// 在 KARArchive.cpp 中实现
void tTVPKARArchive::Reload() {
    TVPAddLog(TJS_W("KAR: 重新加载归档: ") +
              ArchiveName);

    // 打开归档文件
    tTVPStreamHolder holder(ArchiveName);
    tTJSBinaryStream *st = holder.Get();

    // 加锁——防止其他线程在重载期间读取
    tTJSCriticalSectionHolder cs(IndexCS);

    // 保存旧索引（以便回滚）
    auto old_items = std::move(Items);
    Items.clear();

    try {
        // 重新解析
        ParseIndex(st);
        TVPAddLog(TJS_W("KAR: 重载完成，新文件数: ") +
                  ttstr((tjs_int)Items.size()));
    } catch (...) {
        // 解析失败，回滚到旧索引
        Items = std::move(old_items);
        TVPAddLog(
            TJS_W("KAR: 重载失败，已回滚"));
        throw;
    }

    // 注意：已打开的 tTVPKARArchiveStream 对象
    // 持有 Entry 的副本，不受 Items 更新影响。
    // 它们的 Data 缓冲区也已经独立于归档文件。
    // 因此热更新对已打开的流是安全的。
}
```

**关键设计要点**：

1. `tTVPKARArchiveStream` 在构造时**复制**了 `tTVPKARIndexEntry`（而非保存引用），所以 `Items` 向量的更新不会影响已打开的流
2. 使用 `tTJSCriticalSection` 保证线程安全——在重载期间，其他线程对 `GetCount`/`GetName` 的调用会被阻塞
3. 异常安全：如果新索引解析失败，自动回滚到旧索引

</details>
