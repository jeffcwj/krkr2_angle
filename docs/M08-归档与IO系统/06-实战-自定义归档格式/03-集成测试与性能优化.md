## 术语预览

本节将涉及以下术语，先有个印象，正文中会逐一详细讲解：

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| 单元测试 | Unit Test | 对代码中最小可测试单元（函数/类）进行独立验证的自动化测试方法 |
| 测试夹具 | Test Fixture | 测试前预先准备好的数据或环境——这里指预生成的 .kar 归档文件 |

## 一、测试环境搭建

### 1.1 KrKr2 项目的测试框架

KrKr2 使用 **Catch2 v3** 作为测试框架。Catch2（C++ Automated Test Cases in a Header，C++ 自动化测试用例框架）是一个仅头文件的现代 C++ 测试框架，语法简洁、无需注册测试——用宏 `TEST_CASE` 定义测试即可自动发现。

项目中测试代码位于 `tests/` 目录，结构如下：

```
tests/
├── unit-tests/
│   ├── core/
│   │   ├── movie/          # 视频模块测试
│   │   ├── tjs2/           # TJS2 脚本引擎测试
│   │   └── visual/         # 渲染模块测试
│   └── plugins/            # 插件测试
├── test_files/             # 测试夹具文件（.tjs 脚本、.pimg/.psb 二进制文件等）
└── CMakeLists.txt          # 测试构建配置
```

每个测试目标都有自定义 `main()` 函数，负责初始化 spdlog 日志并启动 `Catch::Session`。测试夹具文件路径通过 CMake 的 `configure_file` 机制注入 `TEST_FILES_PATH` 宏。

### 1.2 为 KAR 创建测试目录

首先在测试目录中创建 KAR 专用的测试结构：

```bash
# 在项目根目录执行
mkdir -p tests/unit-tests/core/base
```

然后创建测试源文件 `tests/unit-tests/core/base/test_kar_archive.cpp`：

```cpp
// tests/unit-tests/core/base/test_kar_archive.cpp
// KAR 归档格式的单元测试套件

#define CATCH_CONFIG_MAIN  // 让 Catch2 生成 main() 函数
#include <catch2/catch_all.hpp>

#include <cstdint>
#include <cstring>
#include <fstream>
#include <string>
#include <vector>

// KAR 格式常量（与 KARArchive.h 中定义一致）
static constexpr char KAR_MAGIC[8] = {
    'K', 'A', 'R', '\r', '\n', '\x1a', '\0', '\0'
};
static constexpr uint16_t KAR_VERSION_MAJOR = 1;
static constexpr uint16_t KAR_VERSION_MINOR = 0;
```

### 1.3 CMake 配置

在 `tests/CMakeLists.txt` 中添加 KAR 测试目标：

```cmake
# tests/CMakeLists.txt（追加以下内容）

# KAR 归档格式测试
add_executable(test_kar_archive
    unit-tests/core/base/test_kar_archive.cpp
)

target_link_libraries(test_kar_archive PRIVATE
    Catch2::Catch2WithMain    # Catch2 v3 自带 main()
    core_base_module          # 链接归档模块以使用 tTVPArchive 等类
    ZLIB::ZLIB                # zlib 用于压缩/解压测试数据
)

# 将测试夹具目录路径注入为宏
# TEST_FILES_PATH 会在代码中展开为字符串字面量，指向夹具文件所在目录
target_compile_definitions(test_kar_archive PRIVATE
    TEST_FILES_PATH="${CMAKE_CURRENT_SOURCE_DIR}/test_files"
)

# 注册到 CTest，这样 ctest 命令可以自动发现并运行
include(Catch)
catch_discover_tests(test_kar_archive)
```

> **跨平台注意事项：** `TEST_FILES_PATH` 在 Windows 上展开为反斜杠路径（如 `C:\...\test_files`），在 Linux/macOS 上为正斜杠。KrKr2 的 `TVPNormalizeStorageName` 会统一处理路径分隔符，但在纯 C++ 标准库中使用时建议用 `std::filesystem::path` 处理。

## 二、KAR 测试夹具生成器

### 2.1 为什么需要夹具生成器

测试归档读取器需要合法的 `.kar` 文件作为输入。手动创建二进制文件既容易出错又难以维护。更好的做法是：在测试代码中编写一个 **夹具生成器**（Fixture Generator），它能以编程方式创建各种测试场景的 `.kar` 文件——正常文件、空归档、损坏文件、大文件等。

这种方式有三个优势：
1. **自文档化**：生成代码本身就说明了 .kar 文件的结构
2. **可变参数**：可以轻松生成不同大小、不同压缩配置的测试文件
3. **不依赖外部工具**：测试套件完全自包含，任何人 clone 后直接运行

### 2.2 KAR 文件写入器

以下是一个最小的 KAR 写入器，仅用于测试目的（不是生产级打包工具）：

```cpp
// 测试辅助：KAR 归档写入器
// 仅用于生成测试夹具，不处理边界情况

#include <zlib.h>    // adler32(), compress()
#include <cstring>   // memcpy()
#include <fstream>   // ofstream
#include <vector>    // vector

// KAR 头部结构——32 字节，与 KARArchive.h 中定义一致
struct KARHeader {
    char     magic[8];         // "KAR\r\n\x1a\x00\x00"
    uint16_t version_major;    // 主版本号：1
    uint16_t version_minor;    // 次版本号：0
    uint32_t flags;            // 标志位（bit 0：索引是否压缩）
    uint32_t file_count;       // 归档内文件总数
    uint32_t index_offset;     // 索引区在文件中的偏移量
    uint32_t index_size;       // 索引区的字节大小
    uint32_t reserved;         // 保留字段，填 0
};

// 单个文件的打包参数
struct KARFileEntry {
    std::string name;          // 文件名（UTF-8 编码）
    std::vector<uint8_t> data; // 文件原始内容
    bool compress;             // 是否对此文件启用 zlib 压缩
};

// 将 uint32_t 以小端序写入缓冲区
// 小端序（Little-Endian）：低字节在前，高字节在后
// 例如 0x12345678 写入后在内存中为 78 56 34 12
static void WriteLE32(std::vector<uint8_t>& buf, uint32_t val) {
    buf.push_back(static_cast<uint8_t>(val & 0xFF));
    buf.push_back(static_cast<uint8_t>((val >> 8) & 0xFF));
    buf.push_back(static_cast<uint8_t>((val >> 16) & 0xFF));
    buf.push_back(static_cast<uint8_t>((val >> 24) & 0xFF));
}

// 将 uint16_t 以小端序写入缓冲区
static void WriteLE16(std::vector<uint8_t>& buf, uint16_t val) {
    buf.push_back(static_cast<uint8_t>(val & 0xFF));
    buf.push_back(static_cast<uint8_t>((val >> 8) & 0xFF));
}

// 生成 .kar 归档文件
// 参数：
//   output_path — 输出文件路径
//   entries     — 要打包的文件列表
// 返回：写入是否成功
bool CreateKARArchive(const std::string& output_path,
                      const std::vector<KARFileEntry>& entries) {
    // 第一阶段：压缩数据 + 计算偏移量
    struct ProcessedEntry {
        std::string name;
        std::vector<uint8_t> compressed_data; // 压缩后（或原始）数据
        uint32_t original_size;               // 原始大小
        uint32_t compressed_size;             // 压缩后大小
        uint32_t adler32;                     // 原始数据的 Adler-32
        uint16_t flags;                       // bit 0: 是否压缩
        uint32_t data_offset;                 // 数据区内偏移量
    };

    std::vector<ProcessedEntry> processed;
    processed.reserve(entries.size());

    // 数据区从头部之后开始（头部固定 32 字节）
    uint32_t current_offset = 32;

    for (const auto& entry : entries) {
        ProcessedEntry pe;
        pe.name = entry.name;
        pe.original_size = static_cast<uint32_t>(entry.data.size());
        // 计算原始数据的 Adler-32 校验和
        pe.adler32 = static_cast<uint32_t>(
            adler32(adler32(0L, Z_NULL, 0),
                    entry.data.data(),
                    static_cast<uInt>(entry.data.size())));

        if (entry.compress && !entry.data.empty()) {
            // 使用 zlib compress() 压缩
            uLongf comp_size = compressBound(
                static_cast<uLong>(entry.data.size()));
            pe.compressed_data.resize(comp_size);
            int ret = compress(
                pe.compressed_data.data(), &comp_size,
                entry.data.data(),
                static_cast<uLong>(entry.data.size()));
            if (ret != Z_OK) return false; // 压缩失败
            pe.compressed_data.resize(comp_size);
            pe.compressed_size = static_cast<uint32_t>(comp_size);
            pe.flags = 1; // bit 0 = 已压缩
        } else {
            // 不压缩，直接拷贝原始数据
            pe.compressed_data = entry.data;
            pe.compressed_size = pe.original_size;
            pe.flags = 0;
        }

        // 16 字节对齐——确保每个文件数据的起始地址对齐
        // 对齐的好处：CPU 从对齐地址读取数据更快，尤其在 ARM 平台上
        if (current_offset % 16 != 0) {
            current_offset += 16 - (current_offset % 16);
        }
        pe.data_offset = current_offset;
        current_offset += pe.compressed_size;
        processed.push_back(std::move(pe));
    }

    // 第二阶段：构建索引区
    std::vector<uint8_t> index_buf;
    for (const auto& pe : processed) {
        WriteLE32(index_buf, pe.data_offset);
        WriteLE32(index_buf, pe.compressed_size);
        WriteLE32(index_buf, pe.original_size);
        WriteLE32(index_buf, pe.adler32);
        WriteLE16(index_buf, pe.flags);
        WriteLE16(index_buf, static_cast<uint16_t>(pe.name.size()));
        WriteLE32(index_buf, 0); // reserved
        // 写入文件名（UTF-8，无 null 终止符）
        index_buf.insert(index_buf.end(),
                         pe.name.begin(), pe.name.end());
    }

    // 索引紧跟数据区之后
    uint32_t index_offset = current_offset;
    uint32_t index_size = static_cast<uint32_t>(index_buf.size());

    // 第三阶段：写入文件
    std::ofstream out(output_path, std::ios::binary);
    if (!out) return false;

    // 写头部（32 字节）
    KARHeader header{};
    std::memcpy(header.magic, KAR_MAGIC, 8);
    header.version_major = KAR_VERSION_MAJOR;
    header.version_minor = KAR_VERSION_MINOR;
    header.flags = 0;
    header.file_count = static_cast<uint32_t>(entries.size());
    header.index_offset = index_offset;
    header.index_size = index_size;
    header.reserved = 0;
    out.write(reinterpret_cast<const char*>(&header),
              sizeof(header));

    // 写数据区（含对齐填充）
    for (const auto& pe : processed) {
        // 填充到对齐位置
        auto pos = static_cast<uint32_t>(out.tellp());
        while (pos < pe.data_offset) {
            out.put(0);
            pos++;
        }
        out.write(reinterpret_cast<const char*>(
                      pe.compressed_data.data()),
                  pe.compressed_data.size());
    }

    // 写索引区
    out.write(reinterpret_cast<const char*>(index_buf.data()),
              index_buf.size());
    out.close();
    return out.good();
}
```

这个写入器实现了 KAR 格式规范中的全部要素：32 字节头部、16 字节对齐的数据区、尾部索引。它不处理索引压缩（`flags` bit 0 = 0），因为大多数测试场景不需要。

## 三、核心单元测试

### 3.1 测试策略

KAR 读取器的测试分为四个层次：

```
┌──────────────────────────────────────────┐
│  第 4 层：端到端测试（E2E）              │
│  TJS2 脚本 → TVPCreateStream → KAR 读取  │
├──────────────────────────────────────────┤
│  第 3 层：集成测试                        │
│  TVPOpenArchive → tTVPKARArchive 注册     │
├──────────────────────────────────────────┤
│  第 2 层：流操作测试                      │
│  tTVPKARArchiveStream 的 Read/Seek/Size  │
├──────────────────────────────────────────┤
│  第 1 层：格式解析测试                    │
│  魔数校验、头部解析、索引解析            │
└──────────────────────────────────────────┘
```

我们从底层开始逐层编写测试。

### 3.2 第 1 层：格式解析测试

```cpp
// 测试辅助：创建一个包含指定文件的临时 .kar 归档
// 使用 RAII 模式——构造时创建文件，析构时删除
class TempKARArchive {
    std::string path_;
public:
    TempKARArchive(const std::string& name,
                   const std::vector<KARFileEntry>& entries)
        : path_(std::string(TEST_FILES_PATH) + "/" + name) {
        REQUIRE(CreateKARArchive(path_, entries));
    }
    ~TempKARArchive() {
        std::remove(path_.c_str()); // 测试结束后清理
    }
    const std::string& path() const { return path_; }
};

// ===== 魔数校验测试 =====

TEST_CASE("KAR: 正确的魔数可以通过 TVPIsKARArchive 检测",
          "[kar][format]") {
    // 创建一个包含单文件的合法归档
    TempKARArchive archive("test_magic.kar", {
        {"hello.txt", {'H', 'e', 'l', 'l', 'o'}, false}
    });

    // TVPIsKARArchive() 读取文件前 8 字节，与 KAR_MAGIC 比对
    // 返回 true 表示这是一个 KAR 格式文件
    REQUIRE(TVPIsKARArchive(archive.path()));
}

TEST_CASE("KAR: 错误的魔数被 TVPIsKARArchive 拒绝",
          "[kar][format]") {
    // 创建一个普通文本文件，魔数不匹配
    std::string path = std::string(TEST_FILES_PATH)
                       + "/not_kar.bin";
    {
        std::ofstream out(path, std::ios::binary);
        out << "This is not a KAR file";
    }
    REQUIRE_FALSE(TVPIsKARArchive(path));
    std::remove(path.c_str());
}

TEST_CASE("KAR: 空文件被安全拒绝", "[kar][format]") {
    std::string path = std::string(TEST_FILES_PATH)
                       + "/empty.kar";
    {
        std::ofstream out(path, std::ios::binary);
        // 不写任何内容——0 字节文件
    }
    // 空文件小于 8 字节，无法读取完整魔数
    REQUIRE_FALSE(TVPIsKARArchive(path));
    std::remove(path.c_str());
}

// ===== 头部解析测试 =====

TEST_CASE("KAR: 头部正确解析文件数量和索引位置",
          "[kar][header]") {
    // 创建包含 3 个文件的归档
    TempKARArchive archive("test_header.kar", {
        {"a.txt", {'A'}, false},
        {"b.txt", {'B', 'B'}, false},
        {"c.txt", {'C', 'C', 'C'}, true} // 第三个文件启用压缩
    });

    // 构造归档对象——构造函数会解析头部和索引
    tTVPKARArchive kar(archive.path());

    // GetCount() 应返回 3
    REQUIRE(kar.GetCount() == 3);

    // 验证文件名
    REQUIRE(kar.GetName(0) == "a.txt");
    REQUIRE(kar.GetName(1) == "b.txt");
    REQUIRE(kar.GetName(2) == "c.txt");
}
```

**代码解析：**

- `TempKARArchive` 使用 RAII（Resource Acquisition Is Initialization，资源获取即初始化）模式：构造时生成 `.kar` 文件，析构时自动删除。这确保即使测试失败也不会留下临时文件。
- `REQUIRE` 是 Catch2 的断言宏。与 `CHECK` 不同，`REQUIRE` 失败时立即终止当前测试用例，避免后续代码访问无效对象。
- 每个 `TEST_CASE` 的第二个参数是标签（Tag），用于分类运行：`ctest -R "[kar]"` 只运行 KAR 相关测试。

### 3.3 第 2 层：流操作测试

流操作测试验证 `tTVPKARArchiveStream` 的 `Read`、`Seek`、`GetSize` 三个核心方法。

```cpp
// ===== 未压缩文件的读取 =====

TEST_CASE("KAR: 读取未压缩文件的完整内容",
          "[kar][stream]") {
    // 准备 256 字节的测试数据（0x00 到 0xFF）
    std::vector<uint8_t> test_data(256);
    for (int i = 0; i < 256; i++) {
        test_data[i] = static_cast<uint8_t>(i);
    }

    TempKARArchive archive("test_read_raw.kar", {
        {"data.bin", test_data, false} // 不压缩
    });

    tTVPKARArchive kar(archive.path());

    // 创建流对象
    auto* stream = kar.CreateStreamByIndex(0);
    REQUIRE(stream != nullptr);

    // 验证文件大小
    REQUIRE(stream->GetSize() == 256);

    // 读取全部内容
    std::vector<uint8_t> buf(256);
    tjs_uint read_count = stream->Read(buf.data(), 256);
    REQUIRE(read_count == 256);

    // 逐字节验证
    for (int i = 0; i < 256; i++) {
        REQUIRE(buf[i] == static_cast<uint8_t>(i));
    }

    stream->Release(); // 释放流对象（引用计数 -1）
}

// ===== 压缩文件的读取 =====

TEST_CASE("KAR: 读取 zlib 压缩文件并自动解压",
          "[kar][stream][compress]") {
    // 准备高度可压缩的测试数据——重复的 "ABCD" 模式
    // zlib 对重复数据有极高的压缩率
    std::vector<uint8_t> test_data(4096);
    for (size_t i = 0; i < test_data.size(); i++) {
        test_data[i] = "ABCD"[i % 4];
    }

    TempKARArchive archive("test_read_compressed.kar", {
        {"pattern.bin", test_data, true} // 启用压缩
    });

    tTVPKARArchive kar(archive.path());
    auto* stream = kar.CreateStreamByIndex(0);
    REQUIRE(stream != nullptr);

    // 解压后大小应等于原始大小
    REQUIRE(stream->GetSize() == 4096);

    // 分块读取（模拟实际使用中的小块读取）
    std::vector<uint8_t> buf(4096);
    size_t total_read = 0;
    // 每次读取 512 字节，共读 8 次
    while (total_read < 4096) {
        tjs_uint n = stream->Read(
            buf.data() + total_read, 512);
        REQUIRE(n > 0); // 每次至少读到 1 字节
        total_read += n;
    }

    // 验证解压后数据与原始数据完全一致
    REQUIRE(std::memcmp(buf.data(), test_data.data(),
                        4096) == 0);
    stream->Release();
}
```

## 四、性能基准测试

### 4.1 为什么需要基准测试

单元测试验证的是"正确性"——读取器是否返回正确的数据。但在游戏运行时，性能同样关键。一个场景切换可能需要同时加载几十个资源文件，如果单文件读取耗时 1ms，100 个文件就是 100ms——这会导致可感知的卡顿。

基准测试（Benchmark）通过重复执行操作并测量耗时，帮助我们量化性能表现，找到瓶颈所在。

### 4.2 使用 Catch2 内置基准测试

Catch2 v3 内置了基准测试功能，无需额外库。使用 `BENCHMARK` 宏即可定义基准测试：

```cpp
// tests/unit-tests/core/base/bench_kar_archive.cpp
// KAR 归档格式性能基准测试

#include <catch2/catch_all.hpp>
#include <catch2/benchmark/catch_benchmark.hpp>

#include <fstream>
#include <random>
#include <vector>
```

## 五、性能优化策略

### 5.1 优化一：内存映射 I/O

**问题：** 当前实现使用 `std::ifstream` 读取文件，每次 `Read` 调用都会触发系统调用（System Call），系统调用需要从用户态切换到内核态再切换回来，开销约 100-500 纳秒。高频小文件读取时，系统调用累计开销可观。

**解决方案：** 使用内存映射 I/O（Memory-Mapped I/O）将整个归档文件映射到进程的虚拟地址空间。映射之后，读取文件数据就是普通的内存访问，不需要系统调用。操作系统会按需将磁盘数据加载到内存页（称为"按需分页"，Demand Paging）。

```cpp
// 内存映射 I/O 的跨平台封装
// Windows 使用 CreateFileMapping + MapViewOfFile
// POSIX（Linux/macOS）使用 mmap

#ifdef _WIN32
#include <windows.h>

class MemoryMappedFile {
    HANDLE file_handle_ = INVALID_HANDLE_VALUE;
    HANDLE mapping_ = nullptr;
    void* data_ = nullptr;    // 映射后的内存起始地址
    size_t size_ = 0;         // 文件大小

public:
    // 打开文件并创建内存映射
    bool Open(const std::string& path) {
        // 以只读方式打开文件
        file_handle_ = CreateFileA(
            path.c_str(),
            GENERIC_READ,         // 只读访问
            FILE_SHARE_READ,      // 允许其他进程同时读取
            nullptr,
            OPEN_EXISTING,        // 文件必须已存在
            FILE_ATTRIBUTE_NORMAL,
            nullptr);
        if (file_handle_ == INVALID_HANDLE_VALUE)
            return false;

        // 获取文件大小
        LARGE_INTEGER li;
        GetFileSizeEx(file_handle_, &li);
        size_ = static_cast<size_t>(li.QuadPart);

        // 创建文件映射对象
        // PAGE_READONLY 表示映射为只读
        mapping_ = CreateFileMappingA(
            file_handle_, nullptr,
            PAGE_READONLY,
            0, 0,     // 映射整个文件
            nullptr);
        if (!mapping_) { Close(); return false; }

        // 将映射视图映射到进程地址空间
        data_ = MapViewOfFile(
            mapping_,
            FILE_MAP_READ,   // 只读映射
            0, 0,            // 从文件开头开始
            0);              // 映射整个文件
        if (!data_) { Close(); return false; }

        return true;
    }

    void Close() {
        if (data_) { UnmapViewOfFile(data_); data_ = nullptr; }
        if (mapping_) { CloseHandle(mapping_); mapping_ = nullptr; }
        if (file_handle_ != INVALID_HANDLE_VALUE) {
            CloseHandle(file_handle_);
            file_handle_ = INVALID_HANDLE_VALUE;
        }
    }

    ~MemoryMappedFile() { Close(); }

    const uint8_t* Data() const {
        return static_cast<const uint8_t*>(data_);
    }
    size_t Size() const { return size_; }
};

#else // POSIX (Linux / macOS)
#include <fcntl.h>
#include <sys/mman.h>
#include <sys/stat.h>
#include <unistd.h>

class MemoryMappedFile {
    void* data_ = MAP_FAILED; // 映射后的内存地址
    size_t size_ = 0;
    int fd_ = -1;             // 文件描述符

public:
    bool Open(const std::string& path) {
        fd_ = open(path.c_str(), O_RDONLY);
        if (fd_ < 0) return false;

        // 通过 stat 获取文件大小
        struct stat st;
        if (fstat(fd_, &st) != 0) { Close(); return false; }
        size_ = static_cast<size_t>(st.st_size);

        // mmap：将文件映射到内存
        // PROT_READ   — 只读保护
        // MAP_PRIVATE — 私有映射（写入不影响原文件）
        data_ = mmap(nullptr, size_,
                     PROT_READ, MAP_PRIVATE, fd_, 0);
        if (data_ == MAP_FAILED) { Close(); return false; }

        // 提示操作系统：我们可能顺序访问这块内存
        // 这让 OS 可以提前预读后续页面（Read-ahead）
        madvise(data_, size_, MADV_SEQUENTIAL);
        return true;
    }

    void Close() {
        if (data_ != MAP_FAILED) {
            munmap(data_, size_);
            data_ = MAP_FAILED;
        }
        if (fd_ >= 0) { close(fd_); fd_ = -1; }
    }

    ~MemoryMappedFile() { Close(); }

    const uint8_t* Data() const {
        return static_cast<const uint8_t*>(data_);
    }
    size_t Size() const { return size_; }
};
#endif
```

使用内存映射后，`tTVPKARArchiveStream::Read` 可以直接 `memcpy` 数据，无需文件 I/O：

```cpp
// 优化后的 Read 实现（使用内存映射）
tjs_uint tTVPKARArchiveStream::Read(void* buffer,
                                     tjs_uint count) {
    if (current_pos_ >= data_size_) return 0;
    tjs_uint to_read = std::min(count,
                                data_size_ - current_pos_);
    // 直接从映射内存拷贝——无系统调用
    std::memcpy(buffer,
                archive_->MappedData() + data_offset_
                    + current_pos_,
                to_read);
    current_pos_ += to_read;
    return to_read;
}
```

> **Android 注意事项：** Android 的 APK 中的 assets 不支持 mmap（因为 assets 实际存储在 ZIP 格式的 APK 内）。如果归档文件放在 assets 目录，需要先解压到外部存储再映射。或者使用 Android NDK 的 `AAsset_getBuffer()`，它在某些情况下可以返回直接指针。

### 5.2 优化二：LZ4 替换 zlib

**问题：** zlib 的 deflate 算法虽然压缩率高（通常 60-70%），但解压速度相对较慢。在游戏运行时，我们只做解压——压缩率不如解压速度重要。

**解决方案：** LZ4 是一种专为速度优化的无损压缩算法。它的解压速度可达 zlib 的 5-10 倍，代价是压缩率稍低（通常比 zlib 大 10-20%）。

| 指标 | zlib (deflate) | LZ4 | LZ4 HC |
|------|-------|-----|--------|
| 压缩速度 | ~100 MB/s | ~700 MB/s | ~30 MB/s |
| 解压速度 | ~300 MB/s | ~3000 MB/s | ~3000 MB/s |
| 压缩率（典型） | 60-70% | 50-60% | 55-65% |
| 适用场景 | 存储空间敏感 | 运行时解压 | 打包时压缩 |

> **LZ4 HC**（High Compression，高压缩模式）是 LZ4 的变体：压缩更慢但压缩率接近 zlib，解压速度与标准 LZ4 相同。适合"打包时慢一点没关系，运行时要快"的场景。

要在 KAR 中支持 LZ4，需要扩展 `flags` 字段：

```cpp
// KARIndexEntry.flags 的位定义扩展
// bit 0: 是否压缩
// bit 1-2: 压缩算法选择
//   00 = zlib deflate（默认，向后兼容）
//   01 = LZ4
//   10 = LZ4 HC
//   11 = 保留（未来扩展）

enum KARCompressionMethod : uint16_t {
    KAR_COMPRESS_NONE  = 0x0000, // 无压缩
    KAR_COMPRESS_ZLIB  = 0x0001, // zlib deflate
    KAR_COMPRESS_LZ4   = 0x0003, // LZ4（bit0=1, bit1=1）
    KAR_COMPRESS_LZ4HC = 0x0005, // LZ4 HC（bit0=1, bit2=1）
};

// LZ4 解压实现
#include <lz4.h>

bool DecompressLZ4(const uint8_t* src,
                   uint32_t compressed_size,
                   uint8_t* dst,
                   uint32_t original_size) {
    // LZ4_decompress_safe 是安全版本——
    // 会检查输入边界，防止缓冲区溢出
    int result = LZ4_decompress_safe(
        reinterpret_cast<const char*>(src),
        reinterpret_cast<char*>(dst),
        static_cast<int>(compressed_size),
        static_cast<int>(original_size));

    // result == original_size 表示解压成功
    // result < 0 表示输入数据损坏
    return result == static_cast<int>(original_size);
}
```

### 5.3 优化三：并行解压

**问题：** 场景切换时可能需要同时加载几十个压缩文件。串行解压意味着总耗时 = 所有文件解压时间之和。

**解决方案：** 使用 C++17 的 `<thread>` 或线程池并行解压多个文件。KrKr2 项目中已有线程工具（`cpp/core/utils/` 下的线程相关代码），可以复用。

```cpp
#include <future>   // std::async, std::future
#include <vector>

// 批量预加载：并行解压多个归档文件
// 适用于场景切换时的资源预加载
std::vector<std::vector<uint8_t>> PreloadFiles(
    tTVPKARArchive& archive,
    const std::vector<int>& indices) {

    // 为每个文件创建一个异步任务
    std::vector<std::future<std::vector<uint8_t>>> futures;
    futures.reserve(indices.size());

    for (int idx : indices) {
        // std::async 自动在线程池中执行 lambda
        // std::launch::async 保证立即启动新线程
        futures.push_back(std::async(
            std::launch::async,
            [&archive, idx]() -> std::vector<uint8_t> {
                auto* stream =
                    archive.CreateStreamByIndex(idx);
                if (!stream) return {};

                size_t size = stream->GetSize();
                std::vector<uint8_t> buf(size);
                stream->Read(buf.data(), size);
                stream->Release();
                return buf;
            }
        ));
    }

    // 等待所有任务完成并收集结果
    std::vector<std::vector<uint8_t>> results;
    results.reserve(futures.size());
    for (auto& f : futures) {
        results.push_back(f.get()); // 阻塞直到该任务完成
    }
    return results;
}
```

> **线程安全注意：** 上述代码要求 `CreateStreamByIndex` 是线程安全的。在我们的 KAR 实现中，由于每个流独立持有自己的文件句柄和读取位置，多个流可以并行使用。但如果改为内存映射实现（共享同一个映射），则天然线程安全——只读映射不需要加锁。

并行解压的效果取决于 CPU 核心数和 I/O 带宽：

```
单线程解压 20 个 64KB 文件（zlib）:  ~4.2 ms
4 线程并行解压:                      ~1.3 ms (加速比 3.2x)
8 线程并行解压:                      ~0.8 ms (加速比 5.3x)
```

> 加速比不是线性的，因为 I/O 带宽有限——当 CPU 解压速度超过磁盘读取速度时，增加线程无法进一步加速。这也是为什么内存映射 + 并行解压的组合效果最佳——消除 I/O 瓶颈后，多线程才能充分发挥。

### 错误 1：测试夹具路径在 CI 环境中找不到

**症状：** 本地测试通过，但 CI（GitHub Actions）中 `TEST_FILES_PATH` 指向错误路径，所有需要创建临时文件的测试失败。

**原因：** CI 的构建目录和源码目录布局与本地不同。`CMAKE_CURRENT_SOURCE_DIR` 在 CI 中指向 checkout 目录。

**解决方案：** 使用 `configure_file` 将路径烘焙到头文件中：

```cmake
# 生成包含正确路径的头文件
configure_file(
    "${CMAKE_CURRENT_SOURCE_DIR}/test_config.h.in"
    "${CMAKE_CURRENT_BINARY_DIR}/test_config.h"
)
# test_config.h.in 内容：
# #define TEST_FILES_PATH "@CMAKE_CURRENT_SOURCE_DIR@/test_files"
```

### 错误 2：基准测试结果波动过大

**症状：** 同一基准测试每次运行结果相差 2-3 倍，无法得出有意义的结论。

**原因：** 操作系统的文件缓存（Page Cache）导致第一次读取是"冷"读取（从磁盘），后续是"热"读取（从内存缓存）。另外后台进程也会影响结果。

**解决方案：**

```cpp
// 在基准测试前清除文件缓存（仅限 Linux）
// Windows 需要用 RAMMap 工具或 SysInternals
BENCHMARK("清除缓存后的冷读取") {
    // Linux: sync && echo 3 > /proc/sys/vm/drop_caches
    // 注意：需要 root 权限
    system("sync && echo 3 > /proc/sys/vm/drop_caches");
    tTVPKARArchive kar(path);
    return kar.GetCount();
};
```

更实用的做法是：运行足够多的迭代让 Catch2 自动计算标准差，标准差超过平均值 10% 时视为不可靠。

### 错误 3：内存映射在 32 位系统上失败

**症状：** 归档文件大于 2GB 时，`MapViewOfFile` / `mmap` 返回失败。

**原因：** 32 位进程的虚拟地址空间只有 4GB（实际可用约 2-3GB），无法映射超大文件。

**解决方案：** 对大文件使用分段映射：

```cpp
// 只映射当前需要读取的区域（窗口映射）
// 例如每次映射 64MB 的滑动窗口
const size_t WINDOW_SIZE = 64 * 1024 * 1024; // 64MB
size_t window_offset = (data_offset / WINDOW_SIZE)
                       * WINDOW_SIZE;
size_t map_size = std::min(WINDOW_SIZE,
                           file_size - window_offset);

#ifdef _WIN32
void* view = MapViewOfFile(mapping, FILE_MAP_READ,
    static_cast<DWORD>(window_offset >> 32),
    static_cast<DWORD>(window_offset & 0xFFFFFFFF),
    map_size);
#else
void* view = mmap(nullptr, map_size, PROT_READ,
                  MAP_PRIVATE, fd, window_offset);
#endif
```

### 错误 4：并行解压时出现数据竞争

**症状：** 多线程读取同一归档时偶发读取到错误数据，或程序崩溃（SIGSEGV）。

**原因：** 多个 `tTVPKARArchiveStream` 共享同一个底层文件句柄，`Seek` + `Read` 不是原子操作——线程 A `Seek` 后、`Read` 前，线程 B 可能已经改变了文件指针位置。

**解决方案：** 每个流持有独立的文件句柄（如当前实现），或使用内存映射（只读映射天然线程安全）。如果必须共享句柄，用互斥锁保护：

```cpp
// 方案 A：每个流独立打开文件（当前 KAR 实现）
// 简单但文件句柄数量多
auto* stream = new tTVPKARArchiveStream(
    archive_path, entry); // 构造函数内部 open() 新句柄

// 方案 B：内存映射（推荐）
// 所有流共享同一个映射，只读访问无需加锁
const uint8_t* data = mmap_file.Data()
                      + entry.data_offset;
std::memcpy(buf, data + pos, count);

// 方案 C：互斥锁保护（不推荐——串行化导致性能下降）
std::lock_guard<std::mutex> lock(file_mutex_);
fseek(shared_fp_, offset, SEEK_SET);
fread(buf, 1, count, shared_fp_);
```

## 本节小结

- **测试分层策略**：KAR 读取器的测试分四层——格式解析、流操作、集成测试、端到端测试，从底层到顶层逐步验证
- **夹具生成器**：在测试代码中以编程方式生成 `.kar` 文件，避免依赖外部工具，实现测试自包含
- **Catch2 基准测试**：使用 `BENCHMARK` 宏进行性能测量，量化 KAR 与其他格式的读取性能差异
- **内存映射 I/O**：通过 `mmap`（POSIX）或 `MapViewOfFile`（Windows）消除系统调用开销，适合高频读取场景
- **LZ4 替换 zlib**：解压速度提升 5-10 倍，代价是压缩率略低，适合运行时解压场景
- **并行解压**：利用 `std::async` 或线程池并行处理多个文件，需确保线程安全——独立文件句柄或内存映射是首选方案
- **跨平台测试**：四平台（Windows/Linux/macOS/Android）构建和运行命令各不相同，注意路径分隔符和文件系统差异

### 题目 1：为 KAR 测试添加"损坏文件检测"测试用例

编写一个 Catch2 测试用例，验证 KAR 读取器在遇到以下损坏情况时能正确报错（而不是崩溃）：
1. 头部的 `file_count` 字段值为 0xFFFFFFFF（异常大值）
2. 索引中某个文件的 `data_offset` 超出归档文件大小
3. 压缩文件的 Adler-32 校验和被篡改

<details>
<summary>查看答案</summary>

```cpp
TEST_CASE("KAR: 损坏的文件数量字段被安全处理",
          "[kar][corrupt]") {
    // 先创建合法归档
    std::string path = std::string(TEST_FILES_PATH)
                       + "/corrupt_count.kar";
    CreateKARArchive(path, {
        {"test.txt", {'A'}, false}
    });

    // 篡改 file_count 字段（偏移量 20，4 字节）
    {
        std::fstream file(path,
            std::ios::in | std::ios::out | std::ios::binary);
        file.seekp(20); // KARHeader.file_count 的偏移量
        uint32_t bad_count = 0xFFFFFFFF;
        file.write(reinterpret_cast<const char*>(&bad_count),
                   4);
    }

    // 构造函数应该检测到异常并抛出异常
    // 而不是尝试分配 4GB 内存导致 OOM 崩溃
    REQUIRE_THROWS_AS(
        tTVPKARArchive(path),
        eTJSError); // KrKr2 的标准异常类型

    std::remove(path.c_str());
}

TEST_CASE("KAR: 超出范围的 data_offset 被检测",
          "[kar][corrupt]") {
    std::string path = std::string(TEST_FILES_PATH)
                       + "/corrupt_offset.kar";
    CreateKARArchive(path, {
        {"test.txt", {'A', 'B', 'C'}, false}
    });

    // 获取文件大小
    std::ifstream measure(path,
        std::ios::ate | std::ios::binary);
    size_t file_size = measure.tellg();
    measure.close();

    // 篡改索引中第一个条目的 data_offset
    // 索引区的第一个字段就是 data_offset（4 字节）
    {
        // 先读取 index_offset（头部偏移量 24）
        std::ifstream reader(path, std::ios::binary);
        reader.seekg(24);
        uint32_t index_offset;
        reader.read(reinterpret_cast<char*>(&index_offset), 4);
        reader.close();

        // 将 data_offset 改为超出文件大小的值
        std::fstream file(path,
            std::ios::in | std::ios::out | std::ios::binary);
        file.seekp(index_offset);
        uint32_t bad_offset = static_cast<uint32_t>(
            file_size + 10000);
        file.write(reinterpret_cast<const char*>(&bad_offset),
                   4);
    }

    tTVPKARArchive kar(path);
    // CreateStreamByIndex 应检测到偏移量越界
    REQUIRE_THROWS_AS(
        kar.CreateStreamByIndex(0),
        eTJSError);

    std::remove(path.c_str());
}

TEST_CASE("KAR: Adler-32 校验和不匹配时报错",
          "[kar][corrupt]") {
    std::string path = std::string(TEST_FILES_PATH)
                       + "/corrupt_checksum.kar";
    std::vector<uint8_t> data(1024, 0x42); // 1KB 全 0x42
    CreateKARArchive(path, {
        {"data.bin", data, true} // 压缩
    });

    // 篡改索引中的 adler32 字段
    // adler32 在索引条目中的偏移量是 12（前面有
    // data_offset(4) + compressed_size(4) + original_size(4)）
    {
        std::ifstream reader(path, std::ios::binary);
        reader.seekg(24); // 头部的 index_offset 字段
        uint32_t index_offset;
        reader.read(reinterpret_cast<char*>(&index_offset), 4);
        reader.close();

        std::fstream file(path,
            std::ios::in | std::ios::out | std::ios::binary);
        file.seekp(index_offset + 12); // adler32 偏移
        uint32_t bad_checksum = 0xDEADBEEF;
        file.write(
            reinterpret_cast<const char*>(&bad_checksum), 4);
    }

    tTVPKARArchive kar(path);
    auto* stream = kar.CreateStreamByIndex(0);
    // 读取时解压并校验——校验和不匹配应抛出异常
    std::vector<uint8_t> buf(1024);
    REQUIRE_THROWS_AS(
        stream->Read(buf.data(), 1024),
        eTJSError);

    stream->Release();
    std::remove(path.c_str());
}
```

- 通过直接修改二进制文件来模拟损坏——比手动构造损坏文件更可靠
- 每个测试结束后清理临时文件

### 题目 2：实现 KAR 归档的缓存预热策略

游戏启动时需要预加载 UI 资源（约 50 个小文件，总计 2MB）。请设计并实现一个 `WarmupCache` 函数，在后台线程中预读这些文件到内存缓存。要求：
1. 使用 `std::thread` 在后台执行
2. 提供 `IsReady()` 方法查询预热是否完成
3. 缓存使用 `std::unordered_map<std::string, std::vector<uint8_t>>` 存储

<details>
<summary>查看答案</summary>

```cpp
#include <atomic>
#include <mutex>
#include <string>
#include <thread>
#include <unordered_map>
#include <vector>

class KARCacheWarmer {
    // 缓存：文件名 → 文件内容
    std::unordered_map<std::string,
                       std::vector<uint8_t>> cache_;
    std::mutex cache_mutex_;        // 保护 cache_ 的互斥锁
    std::atomic<bool> ready_{false}; // 原子布尔——线程安全的完成标志
    std::thread worker_;            // 后台工作线程

public:
    // 启动预热——传入归档路径和要预加载的文件名列表
    void WarmupCache(const std::string& archive_path,
                     const std::vector<std::string>& files) {
        ready_ = false;

        // 在后台线程中执行预加载
        worker_ = std::thread([this, archive_path, files]() {
            tTVPKARArchive archive(archive_path);

            for (const auto& name : files) {
                // 在归档中查找文件索引
                int idx = -1;
                for (int i = 0; i < archive.GetCount(); i++) {
                    if (archive.GetName(i) == name) {
                        idx = i;
                        break;
                    }
                }
                if (idx < 0) continue; // 文件不存在，跳过

                // 读取文件内容
                auto* stream =
                    archive.CreateStreamByIndex(idx);
                if (!stream) continue;

                size_t size = stream->GetSize();
                std::vector<uint8_t> data(size);
                stream->Read(data.data(), size);
                stream->Release();

                // 写入缓存（加锁保护）
                {
                    std::lock_guard<std::mutex> lock(
                        cache_mutex_);
                    cache_[name] = std::move(data);
                }
            }

            // 标记预热完成
            ready_.store(true,
                         std::memory_order_release);
        });
    }

    // 查询预热是否完成——可在主线程中无阻塞调用
    bool IsReady() const {
        return ready_.load(std::memory_order_acquire);
    }

    // 从缓存中获取文件内容
    // 返回空 vector 表示文件不在缓存中
    std::vector<uint8_t> Get(const std::string& name) {
        std::lock_guard<std::mutex> lock(cache_mutex_);
        auto it = cache_.find(name);
        if (it != cache_.end()) return it->second;
        return {};
    }

    ~KARCacheWarmer() {
        if (worker_.joinable()) worker_.join();
    }
};

// 使用示例
int main() {
    KARCacheWarmer warmer;

    // 游戏启动时立即开始预热
    warmer.WarmupCache("game_data.kar", {
        "ui/title.png",
        "ui/button_start.png",
        "ui/button_load.png",
        "ui/background.jpg",
        // ... 更多 UI 资源
    });

    // 继续执行其他初始化（不阻塞）
    InitAudioSystem();
    InitInputSystem();

    // 需要 UI 资源时检查缓存是否就绪
    while (!warmer.IsReady()) {
        // 显示加载动画
        ShowLoadingSpinner();
        std::this_thread::sleep_for(
            std::chrono::milliseconds(16)); // 约 60fps
    }

    // 从缓存中获取资源
    auto title_data = warmer.Get("ui/title.png");
    // 使用 title_data 创建纹理...

    return 0;
}
```

**要点：**
- `std::atomic<bool>` 提供无锁的线程间通信
- `std::memory_order_release` / `acquire` 确保数据可见性——`ready_` 变为 true 时，所有缓存写入对读取线程可见
- 析构函数中 `join()` 等待后台线程完成，防止悬空线程

</details>

### 题目 3：编写比较 mmap 与 ifstream 性能的基准测试

编写一个 Catch2 基准测试，对同一个 10MB 的 KAR 归档文件，分别使用标准 `ifstream` 和内存映射方式读取全部文件内容，比较两者的性能差异。要求覆盖 Windows 和 Linux 两种内存映射 API。

<details>
<summary>查看答案</summary>

```cpp
#include <catch2/catch_all.hpp>
#include <catch2/benchmark/catch_benchmark.hpp>
#include <fstream>
#include <vector>

// 包含之前实现的 MemoryMappedFile 类
#include "memory_mapped_file.h"
#include "kar_test_helpers.h"

TEST_CASE("Benchmark: mmap vs ifstream 读取 10MB KAR",
          "[kar][benchmark][mmap]") {
    // 生成 10MB 的测试归档
    // 100 个文件，每个 100KB
    std::vector<KARFileEntry> entries;
    for (int i = 0; i < 100; i++) {
        std::string name = "big_" + std::to_string(i);
        std::vector<uint8_t> data(102400); // 100KB
        std::mt19937 rng(i);
        for (auto& b : data) b = rng() & 0xFF;
        entries.push_back({name, data, false}); // 无压缩
    }

    std::string path = std::string(TEST_FILES_PATH)
                       + "/bench_mmap.kar";
    CreateKARArchive(path, entries);

    // 方案 A：ifstream 逐文件读取
    BENCHMARK("ifstream: 读取 100 x 100KB") {
        tTVPKARArchive kar(path);
        std::vector<uint8_t> buf(102400);
        for (int i = 0; i < 100; i++) {
            auto* s = kar.CreateStreamByIndex(i);
            s->Read(buf.data(), 102400);
            s->Release();
        }
    };

    // 方案 B：内存映射全文件后逐文件提取
    BENCHMARK("mmap: 读取 100 x 100KB") {
        MemoryMappedFile mmf;
        REQUIRE(mmf.Open(path));

        // 解析头部
        const auto* header =
            reinterpret_cast<const KARHeader*>(mmf.Data());

        // 解析索引
        const uint8_t* index_ptr =
            mmf.Data() + header->index_offset;

        std::vector<uint8_t> buf(102400);
        for (uint32_t i = 0; i < header->file_count; i++) {
            // 读取索引条目
            uint32_t data_offset;
            std::memcpy(&data_offset, index_ptr, 4);
            uint32_t compressed_size;
            std::memcpy(&compressed_size, index_ptr + 4, 4);

            // 直接从映射内存拷贝
            std::memcpy(buf.data(),
                        mmf.Data() + data_offset,
                        compressed_size);

            // 跳到下一个索引条目
            uint16_t name_len;
            std::memcpy(&name_len, index_ptr + 18, 2);
            index_ptr += 24 + name_len;
        }
    };

    std::remove(path.c_str());
}
```

典型输出：

```
benchmark name                     samples  iterations  mean
ifstream: 读取 100 x 100KB          100        1       3.42 ms
mmap: 读取 100 x 100KB              100        1       0.87 ms
```

内存映射通常快 3-5 倍，因为：
1. 消除了 `open`/`seek`/`read`/`close` 系统调用
2. 数据直接在用户空间通过 `memcpy` 访问
3. 操作系统的页面缓存和预读机制自动优化 I/O

</details>
