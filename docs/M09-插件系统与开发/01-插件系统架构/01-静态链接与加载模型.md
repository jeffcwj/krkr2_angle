## 原版 KiriKiri2 的动态加载模型

### 什么是动态加载？

在原版 KiriKiri2（吉里吉里2）引擎中，插件采用**动态加载**（Dynamic Loading）模式。所谓动态加载，就是程序在运行时才去磁盘上找到并加载插件文件，而不是在编译时就把插件代码包含进来。

具体来说，原版引擎的插件工作流程如下：

```
游戏启动 → TJS2 脚本执行 Plugins.link("xxx.dll")
         → 引擎调用 Windows API LoadLibrary("xxx.dll")
         → 操作系统将 DLL 文件加载进进程内存
         → 引擎调用 GetProcAddress 获取 V2Link 函数地址
         → 调用 V2Link(iTVPFunctionExporter*)
         → 插件在 V2Link 中注册自己的类/函数到 TJS2 引擎
         → 插件加载完成，TJS2 脚本可以使用新注册的类/函数
```

### V2Link 和 V2Unlink 接口

每个原版 KiriKiri 插件都必须导出两个 C 风格函数：

```cpp
// 插件加载时调用 —— 在这里注册类、函数到 TJS2 引擎
// exporter 参数提供了引擎的功能导出表，插件通过它访问引擎内部功能
extern "C" __declspec(dllexport)
HRESULT V2Link(iTVPFunctionExporter* exporter) {
    // 1. 保存 exporter 指针，后续通过它调用引擎功能
    TVPInitImportStub(exporter);

    // 2. 注册 TJS2 类/函数
    // （具体注册方式取决于使用 ncbind 还是手动注册）

    return S_OK;  // 返回 S_OK 表示加载成功
}

// 插件卸载时调用 —— 在这里清理资源、注销已注册的类
extern "C" __declspec(dllexport)
HRESULT V2Unlink() {
    // 注销所有已注册的 TJS2 类
    // 释放插件分配的资源
    return S_OK;
}
```

这里有几个关键概念：

1. **`iTVPFunctionExporter`**（函数导出器接口）：这是引擎暴露给插件的一组函数指针。因为插件是独立的 DLL，它无法直接调用主程序中的函数，所以引擎通过这个"导出器"把插件需要的函数（如内存分配、TJS2 对象创建等）打包传递给插件
2. **`__declspec(dllexport)`**（DLL 导出声明）：Windows 特有的声明方式，告诉编译器"这个函数要从 DLL 中导出，让外部程序能找到它"
3. **`HRESULT`**（返回码类型）：Windows COM 编程中的标准返回类型，`S_OK`（值为 0）表示成功

### .tpm 文件格式

在 KiriKiri 游戏发行时，插件文件的扩展名通常从 `.dll` 改为 `.tpm`（TJS Plugin Module）。这只是文件名变化，文件内容和 `.dll` 完全一样。游戏脚本中这样使用：

```javascript
// data.xp3 中的 startup.tjs
Plugins.link("csvParser.dll");       // 直接指定 DLL 名
// 或者
Plugins.link("extrans/csvParser.tpm"); // 从子目录加载 .tpm 文件
```

引擎在加载 `.tpm` 文件时，会自动将扩展名替换为 `.dll` 进行处理。

## KrKr2 为什么选择静态链接？

### 跨平台的根本挑战

原版 KiriKiri2 的插件系统完全依赖 Windows API：

| 功能 | Windows API | 其他平台等价物 |
|------|------------|---------------|
| 加载动态库 | `LoadLibrary()` | Linux: `dlopen()`，macOS: `dlopen()`，Android: `dlopen()` |
| 获取函数地址 | `GetProcAddress()` | `dlsym()` |
| 卸载动态库 | `FreeLibrary()` | `dlclose()` |
| 导出声明 | `__declspec(dllexport)` | `__attribute__((visibility("default")))` |

虽然各平台都有动态库加载机制，但存在以下问题：

1. **Android 限制**：Android 的安全模型限制了从应用包外加载 `.so` 文件。用户安装的游戏数据（`.xp3` 归档中的 `.tpm` 文件）无法直接作为 `.so` 加载
2. **ABI 兼容性**：不同平台、不同编译器版本的 C++ ABI（Application Binary Interface，应用二进制接口——规定了函数调用约定、对象内存布局等底层细节）不兼容。原版 Windows 上编译的 `.dll` 文件无法在 Linux/macOS/Android 上使用
3. **分发复杂度**：如果每个插件都是独立的动态库，就需要为 4 个目标平台分别编译、分发和管理这些文件
4. **性能开销**：动态库加载涉及磁盘 IO、符号解析、内存映射等操作，在移动设备上可能造成可感知的延迟

### 静态链接方案

KrKr2 的解决方案很直接：**把所有插件在编译时就链接进主程序**。

```
原版 KiriKiri2：              KrKr2 模拟器：
┌─────────────┐              ┌─────────────────────────────┐
│  主程序 .exe │              │       主程序 (单一二进制)      │
└──────┬──────┘              │  ┌─────────┐ ┌───────────┐  │
       │ 运行时加载            │  │ 引擎核心  │ │ TJS2 引擎  │  │
┌──────▼──────┐              │  └─────────┘ └───────────┘  │
│ csvParser.dll│              │  ┌─────────┐ ┌───────────┐  │
│ scriptsEx.dll│              │  │csvParser │ │ scriptsEx │  │
│ windowEx.dll │              │  │(静态链接) │ │ (静态链接) │  │
│  ...更多 DLL  │              │  └─────────┘ └───────────┘  │
└─────────────┘              │  ┌─────────────────────────┐ │
                             │  │ ...所有插件都在主程序内部  │ │
                             │  └─────────────────────────┘ │
                             └─────────────────────────────┘
```

在 CMake 构建系统中，所有插件被编译为一个名为 `krkr2plugin` 的**静态库**（Static Library），然后链接到主程序：

```cmake
# cpp/plugins/CMakeLists.txt — 插件静态库构建配置
add_library(krkr2plugin STATIC
    # 各插件源文件
    scriptsEx.cpp        # 脚本扩展插件
    csvParser.cpp        # CSV 解析器插件
    dirlist.cpp          # 目录列表插件
    varfile.cpp          # 变量文件读写插件
    saveStruct.cpp       # 结构化存档插件
    windowEx.cpp         # 窗口扩展插件
    addFont.cpp          # 字体加载插件
    wutcwf.cpp           # 文件写入工具插件
    getSample.cpp        # 音频采样获取插件
    getabout.cpp         # 关于信息插件
    fftgraph.cpp         # FFT 频谱图插件
    xp3filter.cpp        # XP3 归档过滤器插件
    layerExBase.cpp      # 基础层扩展插件
    layerExMovie.cpp     # 视频层扩展插件
    layerExPerspective.cpp  # 透视层扩展插件
)

# 子目录中的更多插件
add_subdirectory(psdfile)       # PSD 文件解析器
add_subdirectory(psbfile)       # PSB/E-mote 文件解析器
add_subdirectory(motionplayer)  # E-mote 动画播放器
add_subdirectory(layerex_draw)  # 扩展绘图插件
add_subdirectory(fstat)         # 文件状态查询插件
```

> 📂 **对照源码：** `cpp/plugins/CMakeLists.txt` 第 1-74 行

### 静态链接 vs 动态加载对比

| 维度 | 动态加载（原版 KiriKiri） | 静态链接（KrKr2） |
|------|------------------------|-------------------|
| **加载时机** | 运行时按需加载 | 编译时全部链接 |
| **文件形式** | 独立的 `.dll` / `.tpm` 文件 | 编译进主程序二进制 |
| **跨平台** | 仅 Windows | 全平台（Win/Linux/macOS/Android） |
| **性能** | 有动态加载开销 | 无加载开销，直接调用 |
| **分发** | 需要额外分发插件文件 | 单一二进制即包含全部插件 |
| **灵活性** | 可运行时添加新插件 | 添加新插件需重新编译 |
| **体积** | 主程序较小，但总体可能更大 | 链接器可移除未使用代码（Dead Code Elimination，链接器自动删除没有被调用的函数代码） |
| **兼容性** | 需要模拟的 API 多 | 不需要模拟 Windows DLL 加载 |

### "虚拟加载"——让游戏脚本无感切换

虽然 KrKr2 采用静态链接，但游戏脚本中仍然会执行 `Plugins.link("xxx.dll")`。为了保持兼容性，KrKr2 实现了一套"虚拟加载"机制：

```
游戏脚本：Plugins.link("csvParser.dll")
           │
           ▼
KrKr2 引擎：TVPLoadPlugin("csvParser.dll")
           │
           ├─ 1. 去掉路径前缀，提取文件名
           ├─ 2. 将 .tpm 替换为 .dll（如有）
           ├─ 3. 在内部注册表中查找 "csvParser.dll"
           ├─ 4. 找到 → 执行对应的静态注册函数
           └─ 5. 未找到 → 静默忽略（不报错）
```

这样设计的好处是：**游戏脚本完全不需要修改**。不管原版游戏脚本怎么调用 `Plugins.link()`，KrKr2 都能正确处理。

## 本节小结

- 原版 KiriKiri2 使用 **动态加载** 模型——插件以 `.dll` / `.tpm` 文件形式在运行时加载，通过 `V2Link` / `V2Unlink` 接口与引擎交互
- KrKr2 改用 **静态链接** 模型——所有插件在编译时链接进主程序的 `krkr2plugin` 静态库，消除了对平台特定动态库 API 的依赖
- 静态链接的主要原因是 **跨平台兼容性**（Android 安全限制、ABI 不兼容）、**分发简便性** 和 **性能**
- KrKr2 通过 **"虚拟加载"机制** 保持了对原版游戏脚本的完全兼容——`Plugins.link()` 调用被透明地路由到静态注册表查找
- 关键源码位于 `cpp/core/plugin/PluginImpl.cpp`（加载逻辑）和 `cpp/plugins/CMakeLists.txt`（构建配置）

### 题目 1：静态链接 vs 动态加载的权衡

请回答以下问题：如果你需要开发一个**仅在 Windows 上运行**的 KiriKiri 兼容引擎，你会选择动态加载还是静态链接？给出你的理由（至少 3 点）。

<details>
<summary>查看答案</summary>

如果仅在 Windows 上运行，**动态加载可能是更好的选择**，理由如下：

1. **完全兼容原版**：可以直接加载原版游戏附带的 `.dll` / `.tpm` 插件文件，无需重新编译，最大程度保持兼容性
2. **灵活性高**：用户或开发者可以在不重新编译主程序的情况下添加、替换、更新插件
3. **减少编译时间**：修改一个插件只需重新编译该插件的 DLL，不需要重新链接整个主程序
4. **社区生态**：原版 KiriKiri 有大量第三方插件以 DLL 形式分发，动态加载可以直接利用这些现有资源

但如果需要**跨平台**（如 KrKr2 的目标），静态链接是更务实的选择，因为 Android 的安全模型使得从游戏数据中加载动态库变得极其困难。

</details>

### 题目 2：编写一个插件名称映射函数

请编写一个 C++ 函数 `mapPluginName`，模拟 KrKr2 的插件名称处理逻辑：
- 输入：原始插件路径（如 `"extrans/csvParser.tpm"`）
- 输出：映射后的 DLL 名称（如 `"csvParser.dll"`）
- 要求：去掉路径前缀，将 `.tpm` 替换为 `.dll`，处理 `emoteplayer.dll` → `motionplayer.dll` 的特殊映射

<details>
<summary>查看答案</summary>

```cpp
#include <iostream>
#include <string>
#include <algorithm>

// 模拟 KrKr2 的插件名称映射逻辑
// 输入：原始插件路径（如 "extrans/csvParser.tpm"）
// 输出：映射后的内部 DLL 名称（如 "csvParser.dll"）
std::string mapPluginName(const std::string& rawPath) {
    std::string name = rawPath;

    // 第一步：去掉路径前缀，只保留文件名
    // 查找最后一个路径分隔符（支持 / 和 \）
    auto lastSlash = name.find_last_of("/\\");
    if (lastSlash != std::string::npos) {
        name = name.substr(lastSlash + 1);  // 取分隔符后面的部分
    }

    // 第二步：将 .tpm 扩展名替换为 .dll
    // 查找最后一个 '.' 的位置
    auto dotPos = name.rfind('.');
    if (dotPos != std::string::npos) {
        std::string ext = name.substr(dotPos);
        // 转换为小写进行比较（不区分大小写）
        std::transform(ext.begin(), ext.end(), ext.begin(), ::tolower);
        if (ext == ".tpm") {
            name = name.substr(0, dotPos) + ".dll";
        }
    }

    // 第三步：处理特殊映射
    // emoteplayer.dll → motionplayer.dll
    if (name == "emoteplayer.dll") {
        name = "motionplayer.dll";
    }

    return name;
}

int main() {
    // 测试用例
    std::cout << mapPluginName("extrans/csvParser.tpm")
              << std::endl;  // 期望输出: csvParser.dll

    std::cout << mapPluginName("csvParser.dll")
              << std::endl;  // 期望输出: csvParser.dll

    std::cout << mapPluginName("plugin/emoteplayer.tpm")
              << std::endl;  // 期望输出: motionplayer.dll

    std::cout << mapPluginName("scriptsEx.dll")
              << std::endl;  // 期望输出: scriptsEx.dll

    std::cout << mapPluginName("sub/dir/windowEx.tpm")
              << std::endl;  // 期望输出: windowEx.dll

    return 0;
}
```

编译运行：

```bash
# Windows (MSVC)
cl /EHsc /std:c++17 mapPluginName.cpp /Fe:mapPluginName.exe
mapPluginName.exe

# Linux/macOS (GCC/Clang)
g++ -std=c++17 -o mapPluginName mapPluginName.cpp
./mapPluginName
```

```

csvParser.dll

scriptsEx.dll
windowEx.dll
```

</details>

### 题目 3：分析静态链接对体积的影响

假设 KrKr2 项目中有 20 个插件，每个插件平均 500 行 C++ 代码（编译后约 50KB 目标代码）。请回答：

1. 使用静态链接时，主程序二进制大约增加多少体积？
2. 如果某个游戏只使用了其中 5 个插件，剩余 15 个插件的代码会被包含进最终二进制吗？
3. 有什么 CMake/链接器选项可以优化这种情况？

<details>
<summary>查看答案</summary>

1. **静态链接增加的体积**：20 × 50KB = 约 1MB。但实际情况取决于编译器优化级别——开启 `-O2` 或 `/O2` 优化后，内联和代码折叠可能减少 20-30% 的体积。

2. **未使用的插件代码是否被包含**：这取决于链接方式：
   - **默认情况下会被包含**。因为 ncbind 使用静态变量的构造函数来注册插件（后续章节详讲），链接器认为这些静态变量有"副作用"（Side Effect，即构造函数会执行代码），因此不会移除它们
   - 具体来说，`ncbAutoRegister` 的构造函数在 `main()` 之前执行，将插件注册到内部映射表。这个过程对所有编译进来的插件都会发生，即使游戏脚本从未调用 `Plugins.link()` 加载它们

3. **优化选项**：
   - **链接时优化（LTO，Link-Time Optimization）**：使用 `-flto`（GCC/Clang）或 `/GL` + `/LTCG`（MSVC），让链接器在链接阶段进行全局优化，移除确实未被引用的代码
   - **CMake 中启用 LTO**：
     ```cmake
     # 在 CMakeLists.txt 中启用链接时优化
     set(CMAKE_INTERPROCEDURAL_OPTIMIZATION TRUE)
     ```
   - **注意**：由于 ncbind 的静态注册机制，即使启用 LTO，注册代码本身也不会被移除（因为有副作用）。真正未被调用的插件内部函数可能会被优化掉
   - **更激进的方案**：按需编译——修改 `CMakeLists.txt`，根据目标游戏所需的插件列表选择性地包含源文件。但这会增加构建配置的复杂度

</details>
