## 术语预览

本节将涉及以下术语，先有个印象，正文中会逐一详细讲解：

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| `TVPLoadPlugin` | — | KrKr2 加载插件的顶层入口函数，处理路径解析和名称转换 |
| `TVPLoadInternalPlugin` | — | 实际执行插件注册的内部函数，查找内部注册表并调用注册器 |
| `TVPLoadInternalPlugins` | — | 引擎启动时调用的批量初始化函数，构建注册表并加载默认插件 |
| `_internal_plugins` | — | 内部插件映射表（`std::map`），键为 DLL 名称，值为注册器链表 |
| `TVPRegisteredPlugins` | — | 已注册插件名称的集合（`std::set`），用于防止重复加载 |
| `ncbAutoRegister` | — | ncbind 的自动注册器类，每个宏展开都会创建一个静态实例 |
| `AllRegist` | — | `ncbAutoRegister` 的静态方法，遍历所有静态注册器构建映射表 |
| `LoadModule` | — | `ncbAutoRegister` 的静态方法，按模块名查找并执行注册 |
| `inline variable` | 内联变量 | C++17 特性，允许在头文件中定义全局变量而不会重复定义 |

## 全局调用链概览

当游戏脚本执行 `Plugins.link("csvParser.dll")` 时，会触发以下调用链：

```
TJS2 脚本: Plugins.link("csvParser.dll")
    │
    ▼
tTJSNC_Plugins::link() 方法     ← PluginIntf.cpp
    │
    ▼
TVPLoadPlugin("csvParser.dll")  ← PluginImpl.cpp（顶层入口）
    │
    ├─ 1. 去掉路径前缀，提取文件名
    ├─ 2. .tpm → .dll 扩展名替换
    ├─ 3. emoteplayer.dll → motionplayer.dll 特殊映射
    │
    ▼
TVPLoadInternalPlugin("csvParser.dll")  ← PluginImpl.cpp（内部执行）
    │
    ├─ 4. 检查 TVPRegisteredPlugins：是否已加载？
    │      ├─ 已加载 → 直接返回（跳过）
    │      └─ 未加载 → 继续
    │
    ├─ 5. 在 _internal_plugins 映射表中查找 "csvParser.dll"
    │      ├─ 找到 → 取出注册器链表
    │      └─ 未找到 → 静默返回（不报错）
    │
    ├─ 6. 调用 ncbAutoRegister::LoadModule("csvParser.dll")
    │      │
    │      ├─ 遍历该模块的注册器链表
    │      ├─ 对每个注册器调用 Regist()
    │      └─ 注册 C++ 类/函数到 TJS2 全局作用域
    │
    └─ 7. 将 "csvParser.dll" 加入 TVPRegisteredPlugins
```

## TVPLoadPlugin —— 顶层入口函数

### 源码逐行解读

以下是 `TVPLoadPlugin()` 函数的核心逻辑（简化自 `cpp/core/plugin/PluginImpl.cpp`）：

```cpp
// cpp/core/plugin/PluginImpl.cpp
// TVPLoadPlugin —— 从 TJS2 脚本 Plugins.link() 调用到达这里
// 参数 name：插件的原始路径字符串，如 "extrans/csvParser.tpm"
void TVPLoadPlugin(const ttstr& name) {
    // ttstr 是 TJS2 引擎的字符串类型（TJS Text String），
    // 内部使用宽字符（wchar_t / tjs_char）存储 Unicode 文本

    ttstr pluginName = name;

    // ========== 第一步：路径规范化 ==========

    // 去掉路径中的目录前缀，只保留文件名部分
    // 例如 "extrans/csvParser.tpm" → "csvParser.tpm"
    tjs_int pos = pluginName.GetLen() - 1;  // 从字符串末尾开始
    while (pos >= 0) {
        tjs_char ch = pluginName[pos];
        if (ch == TJS_W('/') || ch == TJS_W('\\')) {
            // 找到路径分隔符，取后面的部分
            pluginName = ttstr(pluginName.c_str() + pos + 1);
            break;
        }
        pos--;
    }

    // ========== 第二步：扩展名映射 ==========

    // 将 .tpm 替换为 .dll
    // KiriKiri 游戏中的插件文件通常使用 .tpm 扩展名
    // 但内部注册表使用 .dll 作为键名
    ttstr ext = TVPExtractStorageExt(pluginName);  // 提取扩展名
    if (ext == TJS_W(".tpm")) {
        // 去掉 .tpm，加上 .dll
        pluginName = TVPChangeStorageExt(pluginName, TJS_W(".dll"));
    }

    // ========== 第三步：特殊名称映射 ==========

    // emoteplayer.dll → motionplayer.dll
    // 原版游戏中的 E-mote 动画插件叫 emoteplayer，
    // 但 KrKr2 项目中实现时改名为 motionplayer
    if (pluginName == TJS_W("emoteplayer.dll")) {
        pluginName = TJS_W("motionplayer.dll");
    }

    // ========== 第四步：调用内部加载函数 ==========

    TVPLoadInternalPlugin(pluginName);
}
```

> 📂 **对照源码：** `cpp/core/plugin/PluginImpl.cpp` 第 50-120 行（实际代码更紧凑，这里展开了注释）

### 关键细节

1. **`ttstr` 类型**：TJS2 引擎自定义的字符串类，类似 `std::wstring` 但支持引用计数。在 M07（TJS2 脚本引擎）模块中有详细讲解
2. **`TJS_W()` 宏**：将 ASCII 字符串字面量转换为 TJS2 宽字符格式，类似 Windows 的 `L""` 前缀
3. **路径分隔符处理**：同时处理 `/` 和 `\` 两种分隔符，兼容不同操作系统的路径风格

## TVPLoadInternalPlugin —— 内部注册执行

### 源码逐行解读

```cpp
// cpp/core/plugin/PluginImpl.cpp
// TVPLoadInternalPlugin —— 在内部注册表中查找并执行插件注册
// 参数 name：规范化后的 DLL 名称，如 "csvParser.dll"
static void TVPLoadInternalPlugin(const ttstr& name) {
    // ========== 去重检查 ==========

    // TVPRegisteredPlugins 是一个 std::set<ttstr>（有序集合），
    // 记录所有已经成功加载的插件名称。
    // 使用 C++17 的 inline variable 声明，可以在头文件中定义
    // 而不会导致多重定义链接错误。
    if (TVPRegisteredPlugins.count(name) > 0) {
        // 这个插件已经加载过了，直接跳过
        // 不报错——同一插件被 link 多次是正常的
        // （多个脚本文件可能各自 link 同一个插件）
        return;
    }

    // ========== 查找内部注册表 ==========

    // 调用 ncbAutoRegister::LoadModule 在静态注册表中查找
    // 如果找到对应模块，执行注册
    bool loaded = ncbAutoRegister::LoadModule(name);

    if (loaded) {
        // 注册成功，将插件名加入已注册集合
        TVPRegisteredPlugins.insert(name);

        // 记录日志
        TVPAddLog(ttstr(TJS_W("Plugin loaded: ")) + name);
    }
    // 如果未找到，静默忽略——不是所有原版插件在 KrKr2 中都有实现
}
```

> 📂 **对照源码：** `cpp/core/plugin/PluginImpl.cpp` 第 120-160 行

### TVPRegisteredPlugins 的去重机制

```cpp
// cpp/core/plugin/PluginImpl.h 第 ~18 行
// C++17 inline variable —— 全局唯一的已注册插件名称集合
inline std::set<ttstr> TVPRegisteredPlugins;
```

**为什么需要去重？** 在 KiriKiri 游戏中，多个 TJS2 脚本文件可能都包含 `Plugins.link("scriptsEx.dll")` 这样的语句。如果不做去重检查，同一个插件的类和函数会被重复注册到 TJS2 引擎，导致内存泄漏甚至崩溃。

**`inline` 关键字的作用**：在 C++17 之前，全局变量如果定义在头文件中会导致"多重定义"（Multiple Definition）链接错误——每个包含该头文件的 `.cpp` 文件都会创建一份副本。`inline` 关键字告诉链接器"这个变量只需要一份实例"，即使它被多个编译单元包含。

## TVPLoadInternalPlugins —— 启动时批量初始化

### 初始化时机

引擎启动时（在 `TVPAppDelegate::applicationDidFinishLaunching()` 之后、游戏脚本执行之前），会调用 `TVPLoadInternalPlugins()` 进行一次性初始化：

```cpp
// cpp/core/plugin/PluginImpl.cpp
void TVPLoadInternalPlugins() {
    // ========== 阶段 1：构建内部注册表 ==========

    // AllRegist() 遍历所有通过 ncbind 宏注册的静态对象，
    // 按模块名（如 "csvParser.dll"）分组，
    // 构建 _internal_plugins 映射表。
    // 这个过程只执行一次。
    ncbAutoRegister::AllRegist();

    // ========== 阶段 2：加载默认插件 ==========

    // 某些插件需要在游戏脚本执行前就加载好
    // 例如 xp3filter.dll 用于 XP3 归档解密
    TVPLoadInternalPlugin(TJS_W("xp3filter.dll"));
}
```

> 📂 **对照源码：** `cpp/core/plugin/PluginImpl.cpp` 第 160-180 行

### _internal_plugins 映射表

```
_internal_plugins（std::map 结构）:
┌──────────────────────┬──────────────────────────────────────┐
│ 键 (DLL 名称)         │ 值 (注册器链表)                        │
├──────────────────────┼──────────────────────────────────────┤
│ "ScriptsEx.dll"      │ → [scriptsEx注册器1] → [注册器2] → ...│
│ "csvParser.dll"      │ → [csvParser注册器1] → ...            │
│ "windowEx.dll"       │ → [windowEx注册器1] → [注册器2] → ... │
│ "dirlist.dll"        │ → [dirlist注册器1] → ...              │
│ "saveStruct.dll"     │ → [saveStruct注册器1] → ...           │
│ "xp3filter.dll"      │ → [xp3filter注册器1] → ...            │
│ "motionplayer.dll"   │ → [motion注册器1] → [注册器2] → ...   │
│ ... (更多插件)        │                                      │
└──────────────────────┴──────────────────────────────────────┘
```

每个"注册器"（`ncbAutoRegister` 实例）对应一个 ncbind 宏展开。例如，`scriptsEx.cpp` 中可能有多个 `NCB_ATTACH_CLASS`、`NCB_ATTACH_FUNCTION` 宏，每个宏都会创建一个注册器实例。这些实例通过链表串联，属于同一个模块（DLL 名称）。

### 练习 3：模拟 _internal_plugins 映射表

编写一个简化版的内部插件注册表模拟程序：

```cpp
#include <iostream>
#include <map>
#include <string>
#include <vector>
#include <functional>
#include <set>

// 模拟 ncbAutoRegister 注册器
struct PluginRegistrar {
    std::string description;                // 描述信息
    std::function<void()> registerFunc;     // 注册函数

    // 执行注册
    void regist() const {
        std::cout << "  注册: " << description << std::endl;
        registerFunc();
    }
};

// 模拟 _internal_plugins 映射表
// 键: DLL 名称, 值: 该模块的所有注册器
std::map<std::string, std::vector<PluginRegistrar>> internalPlugins;

// 模拟 TVPRegisteredPlugins 已加载集合
std::set<std::string> registeredPlugins;

// 模拟 AllRegist() —— 构建注册表
void allRegist() {
    // 模拟 scriptsEx.dll 的注册器
    internalPlugins["ScriptsEx.dll"] = {
        {"Scripts.exec 方法", []() {
            std::cout << "    → Scripts.exec() 已注册到 TJS2" << std::endl;
        }},
        {"Scripts.eval 方法", []() {
            std::cout << "    → Scripts.eval() 已注册到 TJS2" << std::endl;
        }},
    };

    // 模拟 csvParser.dll 的注册器
    internalPlugins["csvParser.dll"] = {
        {"CSVParser 类", []() {
            std::cout << "    → CSVParser 类已注册到 TJS2 全局" << std::endl;
        }},
    };

    // 模拟 windowEx.dll 的注册器
    internalPlugins["windowEx.dll"] = {
        {"Window 类扩展方法", []() {
            std::cout << "    → Window.bringToFront() 等方法已注册" << std::endl;
        }},
    };

    std::cout << "AllRegist 完成: 共 " << internalPlugins.size()
              << " 个模块注册" << std::endl;
}

// 模拟 LoadModule() —— 按模块名查找并执行注册
bool loadModule(const std::string& name) {
    auto it = internalPlugins.find(name);
    if (it == internalPlugins.end()) {
        std::cout << "模块 " << name << " 未找到（跳过）" << std::endl;
        return false;
    }

    std::cout << "加载模块: " << name << std::endl;
    for (const auto& reg : it->second) {
        reg.regist();
    }
    return true;
}

// 模拟 TVPLoadInternalPlugin()
void loadInternalPlugin(const std::string& name) {
    // 去重检查
    if (registeredPlugins.count(name) > 0) {
        std::cout << "[跳过] " << name << " 已加载" << std::endl;
        return;
    }

    // 查找并注册
    if (loadModule(name)) {
        registeredPlugins.insert(name);
        std::cout << "[完成] " << name << " 加载成功" << std::endl;
    }
}

int main() {
    std::cout << "===== 引擎启动 =====" << std::endl;
    allRegist();

    std::cout << "\n===== 游戏脚本执行 =====" << std::endl;

    // 模拟游戏脚本调用
    loadInternalPlugin("ScriptsEx.dll");
    std::cout << std::endl;

    loadInternalPlugin("csvParser.dll");
    std::cout << std::endl;

    // 模拟重复加载
    loadInternalPlugin("ScriptsEx.dll");
    std::cout << std::endl;

    // 模拟不存在的插件
    loadInternalPlugin("nonExist.dll");

    return 0;
}
```

## 常见错误及解决方案

### 错误 1：插件名称大小写不匹配

**现象**：游戏脚本中写了 `Plugins.link("ScriptsEx.dll")`，但内部注册表的键是 `"scriptsex.dll"`（全小写），导致查找失败。

**原因**：KrKr2 的 `_internal_plugins` 映射表使用 `ttstr` 作为键，而 `ttstr` 的比较是区分大小写的。

**解决方案**：确保 `NCB_MODULE_NAME` 宏定义的模块名与游戏脚本中 `Plugins.link()` 使用的名称大小写一致。在项目中，约定使用与原版 KiriKiri 游戏相同的大小写格式。

### 错误 2：插件注册后 TJS2 找不到类

**现象**：`TVPLoadInternalPlugin` 返回成功，但 TJS2 脚本中 `new CSVParser()` 报错"类未定义"。

**原因**：可能的原因包括：
1. 注册器的 `Regist()` 方法执行了但内部出错（例如类名冲突）
2. 注册阶段错误——应该在 `ClassRegist` 阶段注册的类放在了 `PreRegist` 阶段

**排查方法**：在 `ncbAutoRegister::LoadModule()` 中添加日志，打印每个注册器的类型和结果。

### 错误 3：重复注册导致崩溃

**现象**：引擎在加载某个插件时崩溃，堆栈指向 TJS2 的类注册代码。

**原因**：绕过了 `TVPRegisteredPlugins` 的去重检查（例如直接调用了 `ncbAutoRegister::LoadModule()` 而没有先检查去重集合），导致同一个类被注册两次。

**解决方案**：始终通过 `TVPLoadPlugin()` 或 `TVPLoadInternalPlugin()` 加载插件，不要直接调用底层注册函数。

## 本节小结

- **`TVPLoadPlugin()`** 是插件加载的顶层入口，负责路径规范化（去目录前缀）、扩展名映射（`.tpm` → `.dll`）和特殊名称处理（`emoteplayer` → `motionplayer`）
- **`TVPLoadInternalPlugin()`** 执行实际的注册工作：先检查 `TVPRegisteredPlugins` 集合避免重复加载，再调用 `ncbAutoRegister::LoadModule()` 查找并执行注册器
- **`TVPLoadInternalPlugins()`** 在引擎启动时被调用一次，先执行 `AllRegist()` 构建 `_internal_plugins` 映射表，再预加载 `xp3filter.dll` 等必需插件
- **`_internal_plugins`** 是一个 `std::map<ttstr, 注册器链表>`，在 `AllRegist()` 中由静态注册器自动构建
- **`TVPRegisteredPlugins`** 是一个 `std::set<ttstr>`，防止同一插件被重复注册

### 题目 2：解释 inline variable 的必要性

请回答：如果 `TVPRegisteredPlugins` 的声明从 `inline std::set<ttstr>` 改为普通的 `std::set<ttstr>`（去掉 `inline`），会发生什么问题？为什么？

<details>
<summary>查看答案</summary>

**会发生"多重定义"（Multiple Definition）链接错误。**

原因分析：

1. `TVPRegisteredPlugins` 定义在头文件 `PluginImpl.h` 中
2. `PluginImpl.h` 被多个 `.cpp` 文件包含（例如 `PluginImpl.cpp`、`PluginIntf.cpp` 等）
3. 每个 `.cpp` 文件编译为一个**编译单元**（Translation Unit，即 `.o` / `.obj` 目标文件）
4. 如果不加 `inline`，每个编译单元都会创建一个独立的 `TVPRegisteredPlugins` 实例
5. 链接时，链接器发现多个编译单元都定义了同名的全局变量，报出 "multiple definition of `TVPRegisteredPlugins`" 错误

`inline` 的作用：
- 告诉链接器"这个变量虽然在多个编译单元中出现，但只需要保留**一份实例**"
- 所有编译单元共享同一个 `TVPRegisteredPlugins` 对象
- 这是 C++17 引入的特性，之前需要在头文件中 `extern` 声明、在一个 `.cpp` 文件中定义

替代方案（C++17 之前的做法）：
```cpp
// PluginImpl.h — 只声明，不定义
extern std::set<ttstr> TVPRegisteredPlugins;

// PluginImpl.cpp — 在一个编译单元中定义
std::set<ttstr> TVPRegisteredPlugins;
```

</details>

### 题目 3：设计一个带优先级的插件加载器

请编写一个扩展版的 `TVPLoadInternalPlugin`，增加以下功能：
- 支持插件依赖关系（插件 A 依赖插件 B，加载 A 时自动先加载 B）
- 加载失败时抛出异常而非静默忽略

<details>
<summary>查看答案</summary>

```cpp
#include <iostream>
#include <map>
#include <set>
#include <string>
#include <vector>
#include <stdexcept>
#include <functional>

// 插件注册器（简化版）
struct PluginRegistrar {
    std::string name;
    std::function<void()> registerFunc;
};

// 插件依赖关系表
// 键: 插件名, 值: 该插件依赖的其他插件列表
std::map<std::string, std::vector<std::string>> pluginDependencies = {
    {"layerExMovie.dll", {"layerExBase.dll"}},  // 视频层依赖基础层
    {"layerExPerspective.dll", {"layerExBase.dll"}},  // 透视层依赖基础层
    {"motionplayer.dll", {"psbfile.dll"}},  // 动画播放依赖 PSB 解析
};

// 内部注册表
std::map<std::string, std::vector<PluginRegistrar>> internalPlugins;

// 已加载集合
std::set<std::string> registeredPlugins;

// 正在加载的集合（用于检测循环依赖）
std::set<std::string> loadingPlugins;

// 初始化模拟注册表
void initRegistry() {
    for (const auto& name : {"layerExBase.dll", "layerExMovie.dll",
                              "layerExPerspective.dll", "psbfile.dll",
                              "motionplayer.dll", "csvParser.dll"}) {
        internalPlugins[name] = {{name, [name]() {
            std::cout << "    → " << name << " 注册完成" << std::endl;
        }}};
    }
}

// 带依赖解析的插件加载器
void loadInternalPluginWithDeps(const std::string& name) {
    // 1. 已加载则跳过
    if (registeredPlugins.count(name) > 0) {
        std::cout << "[跳过] " << name << " 已加载" << std::endl;
        return;
    }

    // 2. 检测循环依赖
    if (loadingPlugins.count(name) > 0) {
        throw std::runtime_error(
            "检测到循环依赖: " + name + " 正在加载中");
    }

    // 3. 检查插件是否存在
    if (internalPlugins.find(name) == internalPlugins.end()) {
        throw std::runtime_error(
            "插件未找到: " + name + " (不在内部注册表中)");
    }

    // 4. 标记为正在加载
    loadingPlugins.insert(name);

    // 5. 递归加载依赖项
    auto depIt = pluginDependencies.find(name);
    if (depIt != pluginDependencies.end()) {
        for (const auto& dep : depIt->second) {
            std::cout << "[依赖] " << name << " 需要 "
                      << dep << std::endl;
            loadInternalPluginWithDeps(dep);  // 递归加载
        }
    }

    // 6. 执行本插件的注册
    std::cout << "[加载] " << name << std::endl;
    for (const auto& reg : internalPlugins[name]) {
        reg.registerFunc();
    }

    // 7. 标记为已加载，从正在加载集合中移除
    registeredPlugins.insert(name);
    loadingPlugins.erase(name);
    std::cout << "[完成] " << name << std::endl;
}

int main() {
    initRegistry();

    std::cout << "===== 测试 1: 带依赖的加载 =====" << std::endl;
    try {
        // layerExMovie 依赖 layerExBase
        loadInternalPluginWithDeps("layerExMovie.dll");
    } catch (const std::exception& e) {
        std::cerr << "错误: " << e.what() << std::endl;
    }

    std::cout << "\n===== 测试 2: 不存在的插件 =====" << std::endl;
    try {
        loadInternalPluginWithDeps("nonExist.dll");
    } catch (const std::exception& e) {
        std::cerr << "错误: " << e.what() << std::endl;
    }

    std::cout << "\n===== 测试 3: 无依赖的简单加载 =====" << std::endl;
    try {
        loadInternalPluginWithDeps("csvParser.dll");
    } catch (const std::exception& e) {
        std::cerr << "错误: " << e.what() << std::endl;
    }

    return 0;
}
```
