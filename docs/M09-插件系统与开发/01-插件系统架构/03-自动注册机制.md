## C++ 自注册模式原理

### 问题：如何在不修改主程序的情况下注册新功能？

假设你有一个插件框架，每次添加新插件都要在主程序的某个地方写一行 `registerPlugin(new MyPlugin())`。这意味着：

1. 主程序必须知道每个插件的存在——**紧耦合**
2. 添加新插件需要修改主程序——**违反开闭原则**（Open-Closed Principle，对扩展开放、对修改关闭）
3. 忘记写注册代码就会导致插件不工作——**容易出错**

### 解决方案：利用静态变量的构造函数

C++ 有一个特性：**全局变量和静态变量的构造函数在 `main()` 执行之前自动调用**。我们可以利用这个特性实现自注册：

```cpp
#include <iostream>
#include <vector>
#include <string>

// ========== 注册表框架 ==========

// 全局注册表——存放所有已注册的插件名称
// 使用函数内静态变量避免"静态初始化顺序问题"
// （Static Initialization Order Fiasco，不同编译单元中
//   静态变量的初始化顺序是不确定的，如果 A 依赖 B，
//   但 B 在 A 之后才初始化，就会崩溃）
std::vector<std::string>& getRegistry() {
    static std::vector<std::string> registry;
    return registry;
}

// 自注册基类——构造时自动将名称写入注册表
struct AutoRegister {
    AutoRegister(const std::string& name) {
        getRegistry().push_back(name);
        std::cout << "[自注册] " << name << " 在 main() 之前注册"
                  << std::endl;
    }
};

// ========== 插件 A（独立文件 pluginA.cpp）==========
// 只需要定义一个静态变量，插件 A 就会自动注册
static AutoRegister pluginA_reg("pluginA");

// ========== 插件 B（独立文件 pluginB.cpp）==========
static AutoRegister pluginB_reg("pluginB");

// ========== 插件 C（独立文件 pluginC.cpp）==========
static AutoRegister pluginC_reg("pluginC");

// ========== 主程序 ==========
int main() {
    std::cout << "\n===== main() 开始 =====" << std::endl;
    std::cout << "已注册的插件：" << std::endl;
    for (const auto& name : getRegistry()) {
        std::cout << "  - " << name << std::endl;
    }
    return 0;
}
```

编译运行：

```bash
# Windows
cl /EHsc /std:c++17 auto_register_demo.cpp /Fe:auto_register_demo.exe
auto_register_demo.exe

# Linux/macOS
g++ -std=c++17 -o auto_register_demo auto_register_demo.cpp
./auto_register_demo
```

输出（注意注册发生在 `main()` 之前）：

```
[自注册] pluginA 在 main() 之前注册
[自注册] pluginB 在 main() 之前注册
[自注册] pluginC 在 main() 之前注册

===== main() 开始 =====
已注册的插件：
  - pluginA
  - pluginB
  - pluginC
```

关键要点：
- **每个插件只需定义一个静态变量**，不需要修改主程序
- **添加新插件只需在 CMakeLists.txt 中添加源文件**——编译器和链接器自动完成剩余工作
- 这正是 ncbind 使用的核心机制

## ncbAutoRegister 的链表结构

### 类定义概览

`ncbAutoRegister` 是所有 ncbind 注册器的基类，定义在 `cpp/core/plugin/ncbind.hpp` 中：

```cpp
// cpp/core/plugin/ncbind.hpp（简化版，保留核心结构）

class ncbAutoRegister {
public:
    // 注册阶段枚举
    enum LineT {
        PreRegist   = 0,  // 阶段 0：预注册（初始化回调）
        ClassRegist = 1,  // 阶段 1：类注册（核心）
        PostRegist  = 2,  // 阶段 2：后注册（清理回调）
    };

    // 子类必须实现的纯虚函数
    virtual void Regist() = 0;     // 执行注册
    virtual void Unregist() = 0;   // 执行注销

    // 获取该注册器所属的模块名（如 "ScriptsEx.dll"）
    virtual const tjs_char* GetModuleName() const = 0;

    // ---------- 静态成员 ----------

    // 三阶段链表的头指针数组
    // _top[0] 指向 PreRegist 阶段的第一个注册器
    // _top[1] 指向 ClassRegist 阶段的第一个注册器
    // _top[2] 指向 PostRegist 阶段的第一个注册器
    static ncbAutoRegister* _top[3];

    // 构建内部映射表并执行 PreRegist 阶段
    static void AllRegist();

    // 按模块名查找并执行 ClassRegist + PostRegist 阶段
    static bool LoadModule(const ttstr& name);

protected:
    // 构造函数——将自己插入对应阶段的链表头部
    ncbAutoRegister(LineT line) : _line(line) {
        // 头插法：新节点的 _next 指向当前头节点
        _next = _top[line];
        // 更新头节点为新节点
        _top[line] = this;
    }

private:
    LineT _line;                   // 所属阶段
    ncbAutoRegister* _next;        // 链表下一个节点
};

// 静态成员初始化
ncbAutoRegister* ncbAutoRegister::_top[3] = {nullptr, nullptr, nullptr};
```

> 📂 **对照源码：** `cpp/core/plugin/ncbind.hpp` 第 60-200 行

### 链表构建过程图解

每当一个 ncbind 宏（如 `NCB_REGISTER_CLASS`）被展开，就会创建一个 `ncbAutoRegister` 的子类静态实例。该实例在构造时自动插入对应阶段的链表：

```
假设有以下宏展开（按编译单元链接顺序）：

1. scriptsEx.cpp: NCB_ATTACH_CLASS(ScriptsClass, Scripts)  → ClassRegist 阶段
2. scriptsEx.cpp: NCB_ATTACH_FUNCTION(...)                 → ClassRegist 阶段
3. csvParser.cpp: NCB_PRE_REGIST_CALLBACK(InitCSV)         → PreRegist 阶段
4. csvParser.cpp: NCB_REGISTER_CLASS(CSVParser)             → ClassRegist 阶段
5. windowEx.cpp:  NCB_ATTACH_CLASS(WindowExClass, Window)   → ClassRegist 阶段

构建后的链表结构（头插法，所以顺序与定义顺序相反）：

_top[0] (PreRegist):
    ┌──────────────────────┐
    │ InitCSV 回调 (csv)    │ → nullptr
    └──────────────────────┘

_top[1] (ClassRegist):
    ┌──────────────────────┐     ┌──────────────────────┐
    │ WindowExClass (winEx) │ ──→ │ CSVParser (csv)       │ ──→
    └──────────────────────┘     └──────────────────────┘
    ┌──────────────────────┐     ┌──────────────────────┐
    │ AttachFunc (scripts)  │ ──→ │ ScriptsClass (scripts)│ → nullptr
    └──────────────────────┘     └──────────────────────┘

_top[2] (PostRegist):
    nullptr（没有后注册回调）
```

### NCB_MODULE_NAME 的作用

每个插件源文件都会在开头定义 `NCB_MODULE_NAME` 宏，告诉 ncbind 这个文件中的注册器属于哪个"虚拟 DLL"：

```cpp
// cpp/plugins/scriptsEx.cpp 开头
#define NCB_MODULE_NAME TJS_W("ScriptsEx.dll")

// 后续的所有 ncbind 宏展开的注册器，
// 其 GetModuleName() 都会返回 "ScriptsEx.dll"
NCB_ATTACH_CLASS(ScriptsExClass, Scripts) {
    NCB_METHOD(exec);
    NCB_METHOD(eval);
}
```

```cpp
// cpp/plugins/csvParser.cpp 开头
#define NCB_MODULE_NAME TJS_W("csvParser.dll")

NCB_PRE_REGIST_CALLBACK(InitPlugin_CSVParser) {
    // 在类注册之前执行的初始化代码
}
```

`NCB_MODULE_NAME` 的值就是游戏脚本 `Plugins.link()` 时使用的名称。当 `AllRegist()` 遍历链表时，会通过 `GetModuleName()` 获取每个注册器的模块名，将其分组到 `_internal_plugins` 映射表中。

## 三阶段注册详解

### 为什么需要三个阶段？

插件注册不能一步完成，因为存在**依赖关系**：

1. **PreRegist（预注册）**：执行前置初始化——例如创建 TJS2 字典对象、注册回调函数。这些操作必须在类注册之前完成
2. **ClassRegist（类注册）**：注册 C++ 类到 TJS2——这是核心阶段，将 C++ 的类、方法、属性暴露给脚本
3. **PostRegist（后注册）**：执行后置清理——例如将已注册的类对象添加到特定的全局字典中，或执行其他需要所有类都注册完毕后才能进行的操作

```
                    ┌─────────────────────────────────────────┐
                    │           AllRegist() 执行流程            │
                    └─────────────────┬───────────────────────┘
                                      │
                    ┌─────────────────▼───────────────────────┐
                    │     阶段 0: PreRegist                     │
                    │     遍历 _top[0] 链表                     │
                    │     执行所有 PreRegist 注册器的 Regist()   │
                    │     例: InitPlugin_CSVParser()             │
                    │     例: 创建全局字典对象                    │
                    └─────────────────┬───────────────────────┘
                                      │
                    ┌─────────────────▼───────────────────────┐
                    │     构建 _internal_plugins 映射表         │
                    │     遍历 _top[1] (ClassRegist) 链表       │
                    │     按 GetModuleName() 分组               │
                    │     （此时不执行 Regist()，只分组）        │
                    └─────────────────┬───────────────────────┘
                                      │
                                      │  （ClassRegist 和 PostRegist
                                      │   延迟到 LoadModule() 时执行）
                                      ▼
          后续 LoadModule("xxx.dll") 时：
                    ┌─────────────────────────────────────────┐
                    │     阶段 1: ClassRegist                   │
                    │     从映射表取出该模块的注册器链表          │
                    │     执行每个注册器的 Regist()              │
                    │     例: 注册 CSVParser 类到 TJS2           │
                    │     例: 给 Scripts 类附加 exec 方法        │
                    └─────────────────┬───────────────────────┘
                                      │
                    ┌─────────────────▼───────────────────────┐
                    │     阶段 2: PostRegist                    │
                    │     执行该模块的 PostRegist 注册器         │
                    │     例: 后置清理、全局对象注册              │
                    └─────────────────────────────────────────┘
```

### AllRegist 源码解读

```cpp
// cpp/core/plugin/ncbind.cpp（简化版）
void ncbAutoRegister::AllRegist() {
    // ========== 第一步：执行所有 PreRegist 阶段注册器 ==========
    // PreRegist 在这里立即执行，因为它们通常是全局初始化，
    // 不绑定到特定模块
    for (auto* p = _top[PreRegist]; p != nullptr; p = p->_next) {
        p->Regist();  // 执行预注册回调
    }

    // ========== 第二步：将 ClassRegist 注册器分组到映射表 ==========
    // 不立即执行 Regist()，而是按模块名分组存储
    // 等到 LoadModule() 时才真正执行
    for (auto* p = _top[ClassRegist]; p != nullptr; p = p->_next) {
        ttstr moduleName = p->GetModuleName();
        _internal_plugins[moduleName].push_back(p);
    }

    // PostRegist 注册器也类似分组，但这里省略了细节
    // 它们在 LoadModule() 中、ClassRegist 之后执行
}
```

> 📂 **对照源码：** `cpp/core/plugin/ncbind.cpp` 第 10-29 行

### LoadModule 源码解读

```cpp
// cpp/core/plugin/ncbind.cpp
bool ncbAutoRegister::LoadModule(const ttstr& name) {
    // 在映射表中查找模块
    auto it = _internal_plugins.find(name);
    if (it == _internal_plugins.end()) {
        return false;  // 模块未找到
    }

    // 取出该模块的注册器列表
    auto& registrars = it->second;

    // 执行所有 ClassRegist 阶段注册器
    for (auto* reg : registrars) {
        reg->Regist();
    }

    // 执行该模块的 PostRegist 阶段注册器（如有）
    // （实际代码通过遍历 _top[PostRegist] 链表，
    //   找到模块名匹配的注册器并执行）

    return true;
}
```

## ncbind 宏如何创建注册器

### NCB_REGISTER_CLASS 宏展开

当你写下 `NCB_REGISTER_CLASS(MyClass) { ... }` 时，宏展开后大致等价于：

```cpp
// NCB_REGISTER_CLASS(MyClass) 宏展开（简化）

// 1. 定义一个 ncbAutoRegister 的子类
class _ncbAutoRegister_MyClass : public ncbAutoRegister {
public:
    _ncbAutoRegister_MyClass()
        : ncbAutoRegister(ClassRegist)  // 注册到 ClassRegist 阶段
    {}

    void Regist() override {
        // 创建 TJS2 原生类对象
        // 注册方法、属性、构造函数
        // 将类注册到 TJS2 全局作用域
    }

    void Unregist() override {
        // 注销类
    }

    const tjs_char* GetModuleName() const override {
        return NCB_MODULE_NAME;  // 返回当前文件定义的模块名
    }
};

// 2. 创建一个静态实例——触发自注册
static _ncbAutoRegister_MyClass _ncbAutoRegInst_MyClass;
```

关键流程：
1. 编译器看到静态变量 `_ncbAutoRegInst_MyClass`
2. 链接器将其放入"需要初始化"的列表
3. 程序启动时（`main()` 之前），C++ 运行时调用其构造函数
4. 构造函数调用基类 `ncbAutoRegister(ClassRegist)`
5. 基类构造函数将 `this` 插入 `_top[ClassRegist]` 链表头部
6. 自注册完成——主程序不需要任何显式代码

### NCB_PRE_REGIST_CALLBACK 宏展开

```cpp
// NCB_PRE_REGIST_CALLBACK(MyCallback) 宏展开（简化）

// 1. 定义回调注册器
class _ncbPreRegist_MyCallback : public ncbAutoRegister {
public:
    _ncbPreRegist_MyCallback()
        : ncbAutoRegister(PreRegist)  // 注册到 PreRegist 阶段
    {}

    void Regist() override {
        MyCallback();  // 执行用户定义的回调函数
    }

    void Unregist() override {}  // 预注册通常没有注销操作

    const tjs_char* GetModuleName() const override {
        return NCB_MODULE_NAME;
    }
};

// 2. 静态实例
static _ncbPreRegist_MyCallback _ncbPreRegInst_MyCallback;
```

## 动手实践

### 练习：实现一个简化版的三阶段自注册框架

```cpp
#include <iostream>
#include <map>
#include <string>
#include <vector>
#include <functional>

// ========== 自注册框架 ==========

class AutoRegister {
public:
    // 注册阶段
    enum Phase {
        PreRegist   = 0,  // 预注册
        ClassRegist = 1,  // 类注册
        PostRegist  = 2,  // 后注册
    };

    virtual ~AutoRegister() = default;
    virtual void regist() = 0;                     // 执行注册
    virtual std::string getModuleName() const = 0;  // 获取模块名

    // 链表头指针数组（3 个阶段）
    static AutoRegister* top[3];

    // 内部映射表
    static std::map<std::string, std::vector<AutoRegister*>>
        internalPlugins;

    // AllRegist —— 构建映射表
    static void allRegist() {
        // 阶段 0：执行所有 PreRegist 注册器
        std::cout << "--- 阶段 0: PreRegist ---" << std::endl;
        for (auto* p = top[PreRegist]; p; p = p->next_) {
            std::cout << "  执行预注册: " << p->getModuleName()
                      << std::endl;
            p->regist();
        }

        // 阶段 1：分组 ClassRegist 注册器
        std::cout << "--- 构建映射表 ---" << std::endl;
        for (auto* p = top[ClassRegist]; p; p = p->next_) {
            internalPlugins[p->getModuleName()].push_back(p);
        }
        for (const auto& [name, regs] : internalPlugins) {
            std::cout << "  模块 " << name << ": "
                      << regs.size() << " 个注册器" << std::endl;
        }
    }

    // LoadModule —— 按模块名执行注册
    static bool loadModule(const std::string& name) {
        auto it = internalPlugins.find(name);
        if (it == internalPlugins.end()) return false;

        std::cout << "加载模块: " << name << std::endl;
        for (auto* reg : it->second) {
            reg->regist();
        }

        // 执行 PostRegist（遍历阶段 2 链表找匹配模块）
        for (auto* p = top[PostRegist]; p; p = p->next_) {
            if (p->getModuleName() == name) {
                std::cout << "  执行后注册: " << name << std::endl;
                p->regist();
            }
        }
        return true;
    }

protected:
    // 构造函数——头插法插入链表
    AutoRegister(Phase phase) : phase_(phase), next_(nullptr) {
        next_ = top[phase];
        top[phase] = this;
    }

private:
    Phase phase_;
    AutoRegister* next_;
};

// 静态成员初始化
AutoRegister* AutoRegister::top[3] = {nullptr, nullptr, nullptr};
std::map<std::string, std::vector<AutoRegister*>>
    AutoRegister::internalPlugins;

// ========== 宏定义（模拟 ncbind 宏）==========

// 当前模块名（每个"插件文件"定义一次）
#define MODULE_NAME ""  // 默认空

// 预注册回调宏
#define PRE_REGIST_CALLBACK(name, moduleName)                \
    void name();                                             \
    struct _PreReg_##name : public AutoRegister {             \
        _PreReg_##name() : AutoRegister(PreRegist) {}        \
        void regist() override { name(); }                   \
        std::string getModuleName() const override {          \
            return moduleName;                                \
        }                                                    \
    };                                                       \
    static _PreReg_##name _preRegInst_##name;                \
    void name()

// 类注册宏
#define REGISTER_CLASS(className, moduleName)                 \
    struct _ClassReg_##className : public AutoRegister {      \
        _ClassReg_##className() : AutoRegister(ClassRegist) {}\
        void regist() override {                             \
            std::cout << "  注册类: " #className << std::endl;\
        }                                                    \
        std::string getModuleName() const override {          \
            return moduleName;                                \
        }                                                    \
    };                                                       \
    static _ClassReg_##className _classRegInst_##className

// 后注册回调宏
#define POST_REGIST_CALLBACK(name, moduleName)                \
    void name();                                             \
    struct _PostReg_##name : public AutoRegister {            \
        _PostReg_##name() : AutoRegister(PostRegist) {}      \
        void regist() override { name(); }                   \
        std::string getModuleName() const override {          \
            return moduleName;                                \
        }                                                    \
    };                                                       \
    static _PostReg_##name _postRegInst_##name;              \
    void name()

// ========== 模拟插件文件 ==========

// --- scriptsEx.dll ---
REGISTER_CLASS(ScriptsEx, "ScriptsEx.dll");

// --- csvParser.dll ---
PRE_REGIST_CALLBACK(initCSVParser, "csvParser.dll") {
    std::cout << "  [PreRegist] CSV 解析器初始化完成" << std::endl;
}
REGISTER_CLASS(CSVParser, "csvParser.dll");
POST_REGIST_CALLBACK(postCSVParser, "csvParser.dll") {
    std::cout << "  [PostRegist] CSV 解析器后置清理完成" << std::endl;
}

// --- windowEx.dll ---
REGISTER_CLASS(WindowEx, "windowEx.dll");

// ========== 主程序 ==========
int main() {
```

### 链表结构在源码中的体现

```cpp
// ncbind.hpp 中 ncbAutoRegister 构造函数（实际代码）
ncbAutoRegister(LineT line) {
    _next = _top[(int)line];    // 新节点的 next 指向当前头节点
    _top[(int)line] = this;     // 头节点更新为新节点（头插法）
}
```

这段代码就是链表头插法的经典实现。每次创建新的静态注册器实例，它都会把自己插到对应阶段链表的最前面。

## 常见错误及解决方案

### 错误 1：忘记定义 NCB_MODULE_NAME

**现象**：编译错误 `NCB_MODULE_NAME was not declared`。

**原因**：在插件源文件中使用了 ncbind 宏，但没有在文件开头定义 `NCB_MODULE_NAME`。

**解决方案**：
```cpp
// 必须在所有 ncbind 宏之前定义
#define NCB_MODULE_NAME TJS_W("myPlugin.dll")
```

### 错误 2：静态初始化顺序问题

**现象**：程序启动时崩溃，堆栈指向 `ncbAutoRegister` 构造函数中的 `_top` 访问。

**原因**：在极少数情况下，`_top` 数组的初始化可能晚于某个 `ncbAutoRegister` 子类的构造函数调用。这是 C++ 的"静态初始化顺序问题"（Static Initialization Order Fiasco）。

**解决方案**：ncbind 通过将 `_top` 定义为 POD 类型数组（`ncbAutoRegister* _top[3]`）来避免此问题——POD 类型的静态变量在"零初始化"（Zero Initialization）阶段就完成了，这比任何动态初始化都早。

### 错误 3：链接时插件注册器被优化掉

**现象**：插件代码编译了，但运行时 `_internal_plugins` 中找不到对应模块。

**原因**：链接器可能将"看似未使用"的静态变量优化掉。尤其是当插件代码在一个独立的静态库中时，链接器默认只包含被引用的目标文件。

**解决方案**：
- **方案 A**（推荐）：在 CMake 中使用 `target_link_libraries` 时加上 `WHOLE_ARCHIVE` 选项：
  ```cmake
  # Linux/macOS
  target_link_options(krkr2 PRIVATE
      -Wl,--whole-archive $<TARGET_FILE:krkr2plugin> -Wl,--no-whole-archive)

  # Windows (MSVC)
  target_link_options(krkr2 PRIVATE
      /WHOLEARCHIVE:krkr2plugin)
  ```
- **方案 B**：将插件源文件直接编译进主目标（而非静态库），这是 KrKr2 项目对部分插件采用的做法

## 本节小结

- **自注册模式** 利用 C++ 静态变量在 `main()` 之前构造的特性，实现了"添加新插件无需修改主程序"的解耦设计
- **`ncbAutoRegister`** 使用**链表 + 头插法**管理所有注册器，通过 `_top[3]` 数组维护三个阶段的链表
- **三阶段注册**：PreRegist（前置初始化）→ ClassRegist（类注册，按模块分组延迟执行）→ PostRegist（后置清理）
- **`NCB_MODULE_NAME`** 宏决定了注册器属于哪个"虚拟 DLL"，是连接 ncbind 宏和插件加载流程的桥梁
- **`AllRegist()`** 在引擎启动时执行一次，立即运行所有 PreRegist 注册器，然后将 ClassRegist 注册器按模块名分组到 `_internal_plugins` 映射表
- **`LoadModule()`** 在 `Plugins.link()` 时被调用，从映射表取出对应模块的注册器并执行

## 练习题与答案

### 题目 1：预测链表顺序

假设有以下三个插件文件，按字母顺序被链接器处理：

```cpp
// a_plugin.cpp
#define NCB_MODULE_NAME TJS_W("a.dll")
NCB_REGISTER_CLASS(ClassA) { ... }  // ClassRegist

// b_plugin.cpp
#define NCB_MODULE_NAME TJS_W("b.dll")
NCB_PRE_REGIST_CALLBACK(InitB) { ... }  // PreRegist
NCB_REGISTER_CLASS(ClassB) { ... }      // ClassRegist

// c_plugin.cpp
#define NCB_MODULE_NAME TJS_W("c.dll")
NCB_REGISTER_CLASS(ClassC) { ... }  // ClassRegist
```

请画出 `main()` 开始时 `_top[0]`（PreRegist）和 `_top[1]`（ClassRegist）的链表结构。

<details>
<summary>查看答案</summary>

由于使用头插法，后链接的文件中的注册器在链表前面。假设链接顺序为 a → b → c：

```
_top[0] (PreRegist):
    InitB(b.dll) → nullptr
    （只有 b_plugin.cpp 有 PreRegist 注册器）

_top[1] (ClassRegist):
    ClassC(c.dll) → ClassB(b.dll) → ClassA(a.dll) → nullptr
    （头插法：最后链接的 c 在最前面）
```

**重要注意**：链接器处理静态库中目标文件的顺序取决于它们在 `add_library` 中的列出顺序，但同一编译单元内的静态变量按定义顺序初始化。不同编译单元之间的初始化顺序是**未指定的**（Unspecified），所以上面的链表顺序在不同编译器/平台上可能不同。ncbind 的设计不依赖特定顺序——它只关心分组是否正确。

</details>

### 题目 2：为什么 _top 用 POD 数组而不是 std::vector？

请解释为什么 `ncbAutoRegister::_top` 使用原始指针数组 `ncbAutoRegister* _top[3]` 而不是 `std::vector<ncbAutoRegister*>`。

<details>
<summary>查看答案</summary>

原因是**避免静态初始化顺序问题**（Static Initialization Order Fiasco）。

1. **`ncbAutoRegister* _top[3]`** 是 POD 类型（Plain Old Data，即没有构造函数的简单数据类型）的数组。POD 类型的全局变量在程序启动的**零初始化阶段**（Zero Initialization Phase）就被设置为 `{nullptr, nullptr, nullptr}`。零初始化在任何动态初始化之前发生。

2. **`std::vector<ncbAutoRegister*>`** 是非 POD 类型，它有构造函数。作为全局变量，它的构造函数在**动态初始化阶段**（Dynamic Initialization Phase）被调用。问题是：不同编译单元中的动态初始化顺序是不确定的。

3. 如果某个 `ncbAutoRegister` 子类的静态实例在 `_top` 的 `vector` 构造之前被初始化，它会尝试向一个**尚未构造的 vector** 写入数据——导致未定义行为（通常是崩溃）。

4. 使用 POD 数组就没有这个问题：不管哪个 `ncbAutoRegister` 子类先初始化，`_top` 数组早已是 `{nullptr, nullptr, nullptr}`，可以安全地进行链表头插操作。

总结：POD 数组的零初始化保证在所有动态初始化之前完成，消除了初始化顺序依赖。

</details>

### 题目 3：扩展自注册框架支持注销

在前面的动手实践代码基础上，添加"模块卸载"功能：实现一个 `unloadModule(name)` 函数，调用该模块所有注册器的 `unregist()` 方法（按注册的反序执行），并从映射表中移除。

<details>
<summary>查看答案</summary>

```cpp
#include <iostream>
#include <map>
#include <string>
#include <vector>
#include <algorithm>

class AutoRegister {
public:
    enum Phase { PreRegist = 0, ClassRegist = 1, PostRegist = 2 };

    virtual ~AutoRegister() = default;
    virtual void regist() = 0;
    virtual void unregist() = 0;  // 注销接口
    virtual std::string getModuleName() const = 0;

    static AutoRegister* top[3];
    static std::map<std::string, std::vector<AutoRegister*>>
        internalPlugins;

    static void allRegist() {
        for (auto* p = top[PreRegist]; p; p = p->next_) {
            p->regist();
        }
        for (auto* p = top[ClassRegist]; p; p = p->next_) {
            internalPlugins[p->getModuleName()].push_back(p);
        }
    }

    static bool loadModule(const std::string& name) {
        auto it = internalPlugins.find(name);
        if (it == internalPlugins.end()) return false;
        std::cout << "[加载] " << name << std::endl;
        for (auto* reg : it->second) {
            reg->regist();
        }
        return true;
    }

    // 新增：卸载模块（反序执行注销）
    static bool unloadModule(const std::string& name) {
        auto it = internalPlugins.find(name);
        if (it == internalPlugins.end()) return false;

        std::cout << "[卸载] " << name << std::endl;

        // 反序遍历注册器列表，执行注销
        auto& regs = it->second;
        for (auto rit = regs.rbegin(); rit != regs.rend(); ++rit) {
            (*rit)->unregist();
        }

        // 从映射表中移除
        internalPlugins.erase(it);
        std::cout << "[完成] " << name << " 已卸载" << std::endl;
        return true;
    }

protected:
    AutoRegister(Phase phase) : next_(nullptr) {
        next_ = top[phase];
        top[phase] = this;
    }

private:
    AutoRegister* next_;
};

AutoRegister* AutoRegister::top[3] = {};
std::map<std::string, std::vector<AutoRegister*>>
    AutoRegister::internalPlugins;

// 模拟插件
struct PluginA : AutoRegister {
    PluginA() : AutoRegister(ClassRegist) {}
    void regist() override {
        std::cout << "  ClassA 已注册" << std::endl;
    }
    void unregist() override {
        std::cout << "  ClassA 已注销" << std::endl;
    }
    std::string getModuleName() const override {
        return "pluginA.dll";
    }
};
static PluginA pluginA;

struct PluginA2 : AutoRegister {
    PluginA2() : AutoRegister(ClassRegist) {}
    void regist() override {
        std::cout << "  ClassA2 已注册" << std::endl;
    }
    void unregist() override {
        std::cout << "  ClassA2 已注销" << std::endl;
    }
    std::string getModuleName() const override {
        return "pluginA.dll";
    }
};
static PluginA2 pluginA2;

int main() {
    AutoRegister::allRegist();

    std::cout << "===== 加载 =====" << std::endl;
    AutoRegister::loadModule("pluginA.dll");

    std::cout << "\n===== 卸载（反序注销）=====" << std::endl;
    AutoRegister::unloadModule("pluginA.dll");

    std::cout << "\n===== 再次加载（应失败）=====" << std::endl;
    bool ok = AutoRegister::loadModule("pluginA.dll");
    if (!ok) std::cout << "模块已被移除，无法加载" << std::endl;

    return 0;
}
```

预期输出：

```
===== 加载 =====
[加载] pluginA.dll
  ClassA 已注册
  ClassA2 已注册

===== 卸载（反序注销）=====
[卸载] pluginA.dll
  ClassA2 已注销
  ClassA 已注销
[完成] pluginA.dll 已卸载

===== 再次加载（应失败）=====
模块已被移除，无法加载
```

反序注销确保了后注册的类先被注销，避免依赖关系导致的问题。

</details>
