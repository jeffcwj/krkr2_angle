## 术语预览

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| 类对象 | Class Object | TJS2 中表示一个"类型"的 `iTJSDispatch2` 对象，类似 JavaScript 的构造函数对象 |

| 实例适配器 | Instance Adaptor | `ncbInstanceAdaptor<T>` 模板，在 TJS2 对象内部存储 C++ 原生实例指针的桥梁 |
| 延迟创建 | Delay Create | 附着模式下，首次访问时才创建 C++ 原生实例的策略 |

## ncbRegistNativeClass —— 类对象的生命周期管理

### 什么是 ncbRegistNativeClass

`ncbRegistNativeClass<T>` 是 ncbind 框架中负责**创建和管理 TJS2 类对象**的核心模板类。当你用 `NCB_REGISTER_CLASS(MyClass)` 宏注册一个 C++ 类时，底层就是通过 `ncbRegistNativeClass<MyClass>` 来完成以下全部工作的：

1. 调用 TJS2 引擎 API 创建一个新的类对象（Class Object，即 `tTJSNativeClassForPlugin` 类型的 `iTJSDispatch2` 派生对象）
2. 为该类生成唯一的类 ID（通过 `TJSRegisterNativeClass` 函数获取一个整数标识符）
3. 将类名、类 ID、类对象指针三元组存储到 `ncbClassInfo<T>` 静态信息中
4. 注册一个空的 `finalize` 方法（TJS2 垃圾回收时调用的清理回调）
5. 将所有方法、属性、常量挂载到类对象上
6. 最终将类对象挂载到 TJS2 全局作用域，使脚本端可以通过 `new MyClass()` 创建实例

### 类的定义结构

`ncbRegistNativeClass` 继承自 `ncbRegistNativeClassBase`，后者定义了注册/反注册的虚函数接口：

```cpp
// 源码位置：cpp/core/plugin/ncbind.hpp 第 1819-1841 行
struct ncbRegistNativeClassBase {
    typedef ncbNativeClassMethodBase::NameT      NameT;   // const tjs_char*
    typedef ncbNativeClassMethodBase::iMethodT   ItemT;   // ncbIMethodObject 指针
    typedef ncbNativeClassMethodBase::FlagsT     FlagsT;  // tjs_uint32
    typedef tTJSString                           StringT; // TJS 字符串类型

    ncbRegistNativeClassBase(NameT name) : _className(name) {}

    // 注册生命周期回调 —— 子类重写这些方法实现具体逻辑
    virtual void RegistBegin()   {}       // 注册开始：创建类对象
    virtual void RegistItem(NameT, ItemT) {} // 注册单个成员（方法/属性）
    virtual void RegistEnd()     {}       // 注册结束：挂载到全局作用域
    virtual void UnregistBegin() {}       // 反注册开始
    virtual void UnregistItem(NameT) {}   // 移除单个成员
    virtual void UnregistEnd()   {}       // 反注册结束：从全局作用域移除
    virtual void RegistVariant(NameT, tTJSVariant const &, FlagsT) {}  // 注册常量值

    NameT GetName() const { return _className; }  // 获取类名

protected:
    NameT const _className;  // 类名（宽字符指针）
};
```

这个基类采用**模板方法模式**（Template Method Pattern，一种设计模式，在基类中定义算法骨架——注册开始→注册成员→注册结束，而把每一步的具体实现推迟到子类）。所有注册流程的"骨架"已经固定，子类只需填充每个步骤的具体逻辑。

### RegistBegin —— 类对象的诞生

`ncbRegistNativeClass<T>::RegistBegin()` 是注册流程的第一步，它完成了类对象从无到有的全部创建工作。让我们逐行分析：

```cpp
// 源码位置：cpp/core/plugin/ncbind.hpp 第 1855-1873 行
void RegistBegin() override {
    // 第 1 步：创建类对象
    // TJSCreateNativeClassForPlugin 是 TJS2 引擎提供的 API，
    // 它创建一个 tTJSNativeClassForPlugin 对象（继承自 iTJSDispatch2），
    // 第二个参数 AdaptorT::CreateEmptyAdaptor 是一个函数指针，
    // 当 TJS2 脚本执行 "new MyClass(void)" 时会调用此函数创建空的适配器
    _classobj = TJSCreateNativeClassForPlugin(
        _className,                      // 类名，如 "MyClass"
        AdaptorT::CreateEmptyAdaptor     // 空适配器工厂函数
    );

    // 第 2 步：生成类 ID
    // TJSRegisterNativeClass 向 TJS2 引擎注册类名，返回唯一整数 ID
    // 后续可用此 ID 快速判断 TJS2 对象是否属于该类
    IdentT id = TJSRegisterNativeClass(_className);

    // 第 3 步：将三元组（类名、类 ID、类对象指针）存入 ncbClassInfo
    // ncbClassInfo<T> 是一个静态信息持有者（上一节已讲），
    // Set 返回 false 表示该类已被注册过，此时抛出异常防止重复注册
    if (!ClassInfoT::Set(_className, id, _classobj))
        TVPThrowExceptionMessage(TJS_W("Already registerd class."));

    // 第 4 步：将类 ID 设置到类对象上
    // 这使得 TJS2 引擎内部能通过类对象反查出类 ID
    TJSNativeClassSetClassID(_classobj, id);

    // 第 5 步：注册空的 finalize 方法
    // TJS2 的垃圾回收机制要求每个原生类都有 finalize 方法，
    // 即使什么都不做也必须注册，否则 GC 时会报错
    TJSNativeClassRegisterNCM(
        _classobj,                              // 目标类对象
        TJS_W("finalize"),                      // 方法名
        TJSCreateNativeClassMethod(EmptyCallback), // 空回调
        _className,                             // 所属类名
        nitMethod                               // 成员类型：方法
    );
}

// 空回调的实现 —— 什么都不做，直接返回成功
static tjs_error EmptyCallback(
    tTJSVariant *, tjs_int, tTJSVariant **, iTJSDispatch2 *)
{
    return TJS_S_OK;
}
```

用流程图表示：

```
RegistBegin() 执行流程
┌──────────────────────────────────────────┐
│ 1. TJSCreateNativeClassForPlugin(name)   │
│    → 创建 TJS2 类对象 _classobj          │
├──────────────────────────────────────────┤
│ 2. TJSRegisterNativeClass(name)          │
│    → 获取唯一整数 类 ID                  │
├──────────────────────────────────────────┤
│ 3. ncbClassInfo<T>::Set(name, id, obj)   │
│    → 存储到静态信息表                    │
├──────────────────────────────────────────┤
│ 4. TJSNativeClassSetClassID(obj, id)     │
│    → 类对象绑定 ID                       │
├──────────────────────────────────────────┤
│ 5. 注册空 finalize 回调                  │
│    → 满足 GC 要求                        │
└──────────────────────────────────────────┘
```

### RegistItem —— 成员的挂载

每个方法、属性、常量的注册都通过 `RegistItem` 完成：

```cpp
// 源码位置：cpp/core/plugin/ncbind.hpp 第 1879-1888 行
void RegistItem(NameT name, ItemT item) override {
    // 检测是否为构造函数（成员名 == 类名 则视为构造函数）
    if (name == _className) {
        if (_hasCtor)  // 已经注册过构造函数了？
            TVPAddLog(tTJSString(TJS_W("Multiple constructors.("))
                      + _className + TJS_W(")"));
        _hasCtor = true;  // 标记已有构造函数
    }

    if (item) {
        // 调用 TJS2 引擎 API 将成员注册到类对象上
        // item->GetDispatch() 获取方法/属性的 iTJSDispatch2 对象
        // item->GetType()     获取类型（nitMethod / nitProperty）
        // item->GetFlags()    获取标志（如 TJS_STATICMEMBER）
        TJSNativeClassRegisterNCM(
            _classobj,              // 目标类对象
            name,                   // 成员名（宽字符）
            item->GetDispatch(),    // 方法/属性的 Dispatch 对象
            _className,             // 所属类名
            item->GetType(),        // nitMethod 或 nitProperty
            item->GetFlags()        // 如 TJS_STATICMEMBER（静态成员标志）
        );
        item->Release();  // 释放 iMethodObject 的引用
    }
}
```

这里有一个重要细节：**构造函数的判定方式是成员名等于类名**。这与 TJS2 的约定一致——在 TJS2 中，`MyClass.MyClass()` 就是构造函数。如果你注册了多个同名方法（即多个构造函数），ncbind 会打印警告日志但不会阻止注册（最后注册的会覆盖之前的）。

### RegistEnd —— 类的全局挂载

`RegistEnd()` 是注册流程的收尾步骤，它将创建好的类对象挂载到 TJS2 全局作用域：

```cpp
// 源码位置：cpp/core/plugin/ncbind.hpp 第 1890-1925 行
void RegistEnd() override {
    // 如果没有注册任何构造函数，添加一个"未实现"的默认构造函数
    // 这样脚本端执行 new MyClass() 时会得到明确的错误信息
    _AddDummyConstructor();

    // 获取 TJS2 全局对象（相当于 JavaScript 的 window / globalThis）
    iTJSDispatch2 *global = TVPGetScriptDispatch();
    if (!global) {
        NCB_WARN_W("No Global Dispatch, Regist failed.");
        return;  // 全局对象还没准备好，注册失败
    }

    // 将类对象包装为 tTJSVariant（TJS2 的通用值类型）
    tTJSVariant val(static_cast<iTJSDispatch2*>(_classobj));

    // val 现在持有 _classobj 的引用，所以可以释放一次
    _classobj->Release();

    // 将类对象挂载到全局作用域
    // TJS_MEMBERENSURE 标志表示"如果该成员不存在就创建"
    global->PropSet(
        TJS_MEMBERENSURE,  // 确保成员存在
        _className,        // 成员名（即类名，如 "MyClass"）
        0,                 // hint 参数（哈希值，0 表示自动计算）
        &val,              // 要设置的值（类对象）
        global             // 上下文对象
    );

    global->Release();  // 释放全局对象引用

    // val 在局部作用域结束时自动析构，会释放其持有的引用
}
```

执行完 `RegistEnd()` 后，TJS2 脚本端就可以这样使用该类了：

```javascript
// TJS2 脚本
var obj = new MyClass();    // 调用 C++ 端注册的构造函数
obj.someMethod();           // 调用 C++ 端注册的方法
var v = obj.someProperty;   // 读取 C++ 端注册的属性
```

### UnregistEnd —— 类的清理

当插件被卸载时，需要从全局作用域移除已注册的类：

```cpp
// 源码位置：cpp/core/plugin/ncbind.hpp 第 1928-1935 行
void UnregistEnd() override {
    // 获取全局对象
    iTJSDispatch2 *global = TVPGetScriptDispatch();
    if (global) {
        // 从全局作用域中删除该类名对应的成员
        global->DeleteMember(0, _className, 0, global);
        global->Release();
    }
    // 清除 ncbClassInfo 中保存的静态信息
    _RemoveClassInfo();
}
```

注册与反注册是严格对称的——`RegistEnd` 添加全局成员，`UnregistEnd` 删除全局成员并清除静态信息。这保证了插件卸载后不会留下悬空引用（Dangling Reference，指向已释放内存的指针/引用，访问它会导致未定义行为）。

## ncbAutoRegister —— 自动注册链表

### 链表结构与三阶段注册

前一节（[自动注册机制](../01-插件系统架构/03-自动注册机制.md)）介绍了 `ncbAutoRegister` 的基本原理。这里我们深入分析其与类注册的配合机制。

`ncbAutoRegister` 定义了三个注册阶段：

```cpp
// 源码位置：cpp/core/plugin/ncbind.hpp 第 2096-2141 行
struct ncbAutoRegister {
    // 三个注册阶段
    enum LineT {
        PreRegist = 0,   // 前置阶段：执行初始化回调
        ClassRegist,     // 类注册阶段：创建和注册类对象
        PostRegist,      // 后置阶段：执行收尾回调
        LINE_COUNT       // 阶段总数（= 3）
    };

    NameT modulename;  // 模块名（如 "scriptsEx"）

    // 构造函数：将自身插入到对应阶段的链表头部
    ncbAutoRegister(NameT name, LineT line)
        : modulename(name),
          _next(_top[line])   // 当前链表头变为自己的 next
    {
        _top[line] = this;    // 自己成为新的链表头
    }

    // 将所有注册器按模块名分组到 _internal_plugins 映射表
    static void AllRegist(LineT line) {
        for (ThisClassT const* p = _top[line]; p; p = p->_next) {
            ttstr name = p->modulename;
            name.ToLowerCase();  // 模块名转为小写，确保大小写无关
            _internal_plugins[name].lists[line].push_back(p);
        }
    }

private:
    ThisClassT const* _next;  // 链表的下一个节点

    // 三个阶段各维护一个链表头指针
    static ThisClassT const* _top[LINE_COUNT];

    // 内部插件映射表：模块名 → 三个阶段的注册器列表
    struct INTERNAL_PLUGIN_LISTS {
        std::list<ThisClassT const*> lists[LINE_COUNT];
    };
    static std::map<ttstr, INTERNAL_PLUGIN_LISTS> _internal_plugins;
};
```

整个注册收集过程如下图：

```
C++ 静态初始化阶段（程序启动时，main() 之前）
═══════════════════════════════════════════════

  静态对象 A (ClassRegist, "scriptsEx")
       ↓ 构造函数：_top[ClassRegist] = A
  静态对象 B (ClassRegist, "csvParser")
       ↓ 构造函数：B._next = A; _top[ClassRegist] = B
  静态对象 C (PreRegist, "csvParser")
       ↓ 构造函数：_top[PreRegist] = C

最终链表结构：
  _top[PreRegist]  → C → null
  _top[ClassRegist] → B → A → null
  _top[PostRegist]  → null

AllRegist() 执行后的 _internal_plugins：
  "scriptsex" → { lists[ClassRegist] = [A] }
  "csvparser" → { lists[PreRegist] = [C],
                   lists[ClassRegist] = [B] }
```

### ncbNativeClassAutoRegister —— 类注册的自动触发器

`ncbNativeClassAutoRegister<T>` 是 `ncbAutoRegister` 的子类，专门负责类注册/反注册的自动触发：

```cpp
// 源码位置：cpp/core/plugin/ncbind.hpp 第 2144-2155 行
template <class T>
struct ncbNativeClassAutoRegister : public ncbAutoRegister {
    // 构造函数：在 ClassRegist 阶段注册，记录模块名和类名
    ncbNativeClassAutoRegister(NameT modulename, NameT n)
        : ncbAutoRegister(modulename, ClassRegist),  // 父类构造：加入链表
          _name(n) {}

    typedef ncbRegistNativeClass<T> DelegateT;  // 实际的注册执行器
    typedef ncbRegistClass<DelegateT> RegistT;  // 注册 DSL 适配器

private:
    NameT const _name;  // 类名

protected:
    // 注册：创建 DelegateT 和 RegistT，执行注册 DSL
    void Regist() const override {
        DelegateT d(_name);
        { RegistT r(d, true);  r.Regist(); }  // true = 注册模式
    }

    // 反注册：同样的结构，但 false = 反注册模式
    void Unregist() const override {
        DelegateT d(_name);
        { RegistT r(d, false); r.Regist(); }  // false = 反注册模式
    }
};
```

注册和反注册使用**完全相同的代码路径**，只通过一个 `bool` 参数区分。`RegistT r(d, true)` 时调用 `RegistBegin/RegistItem/RegistEnd`；`RegistT r(d, false)` 时调用 `UnregistBegin/UnregistItem/UnregistEnd`。这种设计保证了注册和反注册的对称性——你注册了什么，反注册就会精确地撤销什么。

## NCB_REGISTER_CLASS 宏 —— 一行代码完成类注册

### 宏展开全过程

`NCB_REGISTER_CLASS` 是开发者最常使用的宏。它只需要一行代码就能完成类注册的全部准备工作。让我们逐层展开：

```cpp
// 第 1 层：NCB_REGISTER_CLASS(cls)
#define NCB_REGISTER_CLASS(cls) NCB_REGISTER_CLASS_DIFFER(cls, cls)

// 第 2 层：NCB_REGISTER_CLASS_DIFFER(name, cls)
// name = 类在 TJS2 端的名字
// cls  = C++ 类名（两者通常相同）
#define NCB_REGISTER_CLASS_DIFFER(name, cls)          \
    NCB_TYPECONV_BOXING(cls);                         \
    NCB_REGISTER_CLASS_COMMON(cls,                    \
        ncbNativeClassAutoRegister,                   \
        ncbRegistNativeClass,                         \
        (NCB_MODULE_NAME, TJS_W(# name)))

// 第 3 层：NCB_REGISTER_CLASS_COMMON(cls, tmpl, nt, init)
#define NCB_REGISTER_CLASS_COMMON(cls, tmpl, nt, init)    \
    template struct ncbClassInfo<cls>;                    \
    static tmpl<cls> tmpl ## _ ## cls init;               \
    template <> void ncbRegistClass<nt<cls>>::Regist()
```

以 `NCB_REGISTER_CLASS(MyPlugin)` 为例，最终展开为：

```cpp
// 1. 注册 MyPlugin 的 Boxing/Unboxing 类型转换
NCB_TYPECONV_BOXING(MyPlugin);
// 展开为：在 ncbTypeConvertor 中注册 MyPlugin 的双向转换

// 2. 显式实例化 ncbClassInfo<MyPlugin>
template struct ncbClassInfo<MyPlugin>;
// 确保静态成员 _info 被分配存储空间

// 3. 创建静态自动注册器对象
static ncbNativeClassAutoRegister<MyPlugin>
    ncbNativeClassAutoRegister_MyPlugin(
        NCB_MODULE_NAME,        // 模块名（通过 NCB_MODULE_NAME 宏定义）
        TJS_W("MyPlugin")      // TJS2 端的类名
    );
// 这个静态对象的构造函数会在程序启动时自动执行，
// 将自身加入 ncbAutoRegister 的 ClassRegist 链表

// 4. 开始注册 DSL 的函数体
template <> void ncbRegistClass<ncbRegistNativeClass<MyPlugin>>::Regist()
// 后面跟花括号 {} 就是注册 DSL 的内容
```

### NCB_MODULE_NAME 的作用

每个插件源文件顶部都需要定义 `NCB_MODULE_NAME`，它指定了该插件在 `_internal_plugins` 映射表中的键名：

```cpp
// 示例：scriptsEx.cpp 顶部
#define NCB_MODULE_NAME TJS_W("scriptsEx.dll")
```

这个名字**必须与游戏期望加载的 DLL 名匹配**。原版 KiriKiri2 是动态加载 DLL 的，游戏脚本中会写 `Plugins.link("scriptsEx.dll")`。KrKr2 模拟器将所有插件静态链接到主程序中，但仍通过模块名来做映射查找。当 TJS2 脚本调用 `Plugins.link("scriptsEx.dll")` 时，引擎会在 `_internal_plugins` 映射表中查找 `"scriptsex.dll"`（小写化后），找到后按顺序执行该模块的 PreRegist → ClassRegist → PostRegist 三个阶段的所有注册器。

### 完整的注册示例

下面是一个完整的、可以在 KrKr2 项目中编译的插件注册代码：

```cpp
// 文件：cpp/plugins/my_counter.cpp
#include "ncbind.hpp"

// 第 1 步：定义模块名
#define NCB_MODULE_NAME TJS_W("my_counter.dll")

// 第 2 步：定义 C++ 类
class MyCounter {
public:
    // 构造函数 —— 接受一个整数初始值
    MyCounter(int initial) : count_(initial) {
        // 可以在构造时打印日志
        TVPAddLog(ttstr(TJS_W("MyCounter created with initial="))
                  + ttstr(initial));
    }

    // 析构函数 —— ncbInstanceAdaptor 会在 TJS2 对象被 GC 时调用
    ~MyCounter() {
        TVPAddLog(ttstr(TJS_W("MyCounter destroyed, final count="))
                  + ttstr(count_));
    }

    // 公开方法：递增计数
    void Increment() { ++count_; }

    // 公开方法：递减计数
    void Decrement() { --count_; }

    // 属性 getter：获取当前值
    int GetValue() const { return count_; }

    // 属性 setter：设置当前值
    void SetValue(int v) { count_ = v; }

    // 静态方法：创建一个零初始值的计数器
    static MyCounter* CreateZero() { return new MyCounter(0); }

private:
    int count_;
};

// 第 3 步：使用 NCB_REGISTER_CLASS 注册
NCB_REGISTER_CLASS(MyCounter) {
    // 注册带参数的构造函数
    // NCB_CONSTRUCTOR 接受参数类型列表
    NCB_CONSTRUCTOR((int));  // 对应 MyCounter(int)

    // 注册方法
    NCB_METHOD(Increment);   // 对应 MyCounter::Increment()
    NCB_METHOD(Decrement);   // 对应 MyCounter::Decrement()

    // 注册读写属性
    // 第一个参数是 TJS2 端的属性名
    // 第二个参数是 getter 方法
    // 第三个参数是 setter 方法
    NCB_PROPERTY(value, GetValue, SetValue);

    // 注册只读属性（使用 NCB_PROPERTY_RO）
    NCB_PROPERTY_RO(count, GetValue);
}
```

对应的 TJS2 脚本使用方式：


## NCB_ATTACH_CLASS —— 向已有类添加功能

### 与 NCB_REGISTER_CLASS 的区别

`NCB_REGISTER_CLASS` 创建一个**全新的类**，而 `NCB_ATTACH_CLASS` 向**已有的 TJS2 类**添加方法和属性。这是两种完全不同的使用场景：

| 对比项 | NCB_REGISTER_CLASS | NCB_ATTACH_CLASS |
|--------|-------------------|------------------|
| 用途 | 创建新的 TJS2 类 | 向已有类追加成员 |
| 类对象来源 | 新建 `tTJSNativeClassForPlugin` | 从全局作用域获取已有类对象 |
| 构造函数 | 可以注册 | **禁止注册**（已有类已有构造函数） |
| 典型使用场景 | 定义新插件类（如 MyCounter） | 扩展 Scripts、Window 等内建类 |
| C++ 实例创建时机 | 构造函数调用时 | 延迟创建（首次访问时） |
| 底层实现 | `ncbRegistNativeClass` | `ncbAttachTJS2Class` |

### ncbAttachTJS2Class 的实现

`ncbAttachTJS2Class<T>` 的核心区别在于 `RegistBegin` —— 它不创建新类，而是获取已有类：

```cpp
// 源码位置：cpp/core/plugin/ncbind.hpp 第 2006-2088 行
template <class CLASS>
struct ncbAttachTJS2Class : public ncbRegistNativeClassBase {
    // 两个类名：C++ 侧的原生类名 和 TJS2 侧的目标类名
    ncbAttachTJS2Class(NameT nativeClass, NameT tjs2Class)
        : BaseT(nativeClass), _tjs2ClassName(tjs2Class) {}

    void RegistBegin() override {
        // 获取 TJS2 全局对象
        _global = TVPGetScriptDispatch();

        // 从全局作用域获取目标 TJS2 类对象
        // 例如获取内建的 "Scripts" 类
        _tjs2ClassObj = GetGlobalObject(_tjs2ClassName, _global);

        // 注册类 ID（用于原生实例的类型识别）
        IdentT id = TJSRegisterNativeClass(_className);

        // 注册到 ncbClassInfo（注意第三个参数是 0，不是类对象指针）
        // 因为附着模式下不拥有类对象的所有权
        if (!ClassInfoT::Set(_className, id, 0))
            TVPThrowExceptionMessage(
                TJS_W("Already registerd class:"), ttstr(_className));
    }

    void RegistItem(NameT name, ItemT item) override {
        if (!item) return;
        // 禁止注册构造函数 —— 已有类的构造函数不能被覆盖
        if (name == _className) {
            TVPThrowExceptionMessage(
                TJS_W("Constructor attached: "), ttstr(_className));
        }

        // 将成员作为属性设置到目标类对象上
        iTJSDispatch2 *dsp = item->GetDispatch();
        tTJSVariant val(dsp);
        dsp->Release();
        FlagsT flg = item->GetFlags();
        RegistVariant(name, val, flg);  // 通过 PropSet 添加
        item->Release();
    }

    void RegistEnd() override {
        if (_global) _global->Release();
        _global = nullptr;
        // _tjs2ClassObj 是 NoAddRef 获取的，不需要 Release
    }
};
```

关键差异总结：

1. **不创建类对象**——从全局作用域获取已有的类对象（如 `Scripts`、`Window`）
2. **禁止构造函数**——如果尝试注册与类名同名的方法，会直接抛出异常
3. **成员通过 PropSet 添加**——不使用 `TJSNativeClassRegisterNCM`，而是像设置属性一样添加方法

### NCB_ATTACH_CLASS 宏展开

```cpp
// 宏定义：
#define NCB_ATTACH_CLASS(cls, attach)                         \
    NCB_ATTACHED_INSTANCE_DELAY_CREATE(cls);                  \
    NCB_ATTACH_CLASS_WITH_HOOK(cls, attach)

// NCB_ATTACHED_INSTANCE_DELAY_CREATE 展开为：
// 定义一个特化的 nativeInstanceGetter，实现延迟创建
NCB_GET_INSTANCE_HOOK(cls) {
    NCB_INSTANCE_GETTER(objthis) {
        ClassT* obj = GetNativeInstance(objthis);
        if (!obj) {
            obj = new ClassT();           // 首次访问时才创建 C++ 实例
            SetNativeInstance(objthis, obj);  // 绑定到 TJS2 对象
        }
        return obj;
    }
};

// NCB_ATTACH_CLASS_WITH_HOOK 展开为自动注册器的创建
```

**延迟创建**（Delay Create）是附着模式的一个核心机制：由于 `NCB_ATTACH_CLASS` 不创建新的 TJS2 类，而是扩展已有类，所以 C++ 原生实例不能在构造函数中创建（已有类有自己的构造函数）。取而代之的方案是——当脚本端**第一次调用扩展方法/访问扩展属性**时，通过 `nativeInstanceGetter` 的延迟创建逻辑自动 `new` 一个 C++ 实例并绑定。

### NCB_ATTACH_CLASS 实战示例

项目中 `scriptsEx.cpp` 就是 `NCB_ATTACH_CLASS` 的典型使用：

```cpp
// 源码位置：cpp/plugins/scriptsEx.cpp（简化版）
#define NCB_MODULE_NAME TJS_W("scriptsEx.dll")

#include "ncbind.hpp"
#include <fstream>

// 定义一个 C++ 类来持有扩展 Scripts 类需要的状态
class ScriptsEx {
public:
    ScriptsEx() = default;  // 默认构造（延迟创建时调用）

    // 将要绑定到 Scripts 类的静态方法
    // 注意：附着模式下，方法通常声明为 static
    static tjs_error evalStorage(
        tTJSVariant *result, tjs_int numparams,
        tTJSVariant **param, ScriptsEx *nativeInstance)
    {
        // 从 XP3 存档中读取 .tjs 脚本文件并执行
        if (numparams < 1) return TJS_E_BADPARAMCOUNT;
        ttstr filename = *param[0];
        // ... 实际的文件读取和执行逻辑
        return TJS_S_OK;
    }
};

// 将 ScriptsEx 的方法附着到 TJS2 内建的 Scripts 类上
NCB_ATTACH_CLASS(ScriptsEx, Scripts) {
    // 使用 RawCallback 注册静态方法
    // TJS_STATICMEMBER 标志表示这是一个静态方法
    NCB_METHOD_RAW_CALLBACK(evalStorage,
                            ScriptsEx::evalStorage,
                            TJS_STATICMEMBER);
}
```

注册完成后，TJS2 脚本端可以直接调用：

```javascript
// TJS2 脚本 —— Scripts 类已经有了新方法
Scripts.evalStorage("data/config.tjs");  // 调用 C++ 端的 evalStorage
```

## NCB_ATTACH_FUNCTION —— 注册独立函数

### 宏定义与展开

有时候你只想向某个 TJS2 对象添加一个**独立函数**，而不需要定义整个类。`NCB_ATTACH_FUNCTION` 就是为此设计的：

```cpp
// 宏定义：
#define NCB_ATTACH_FUNCTION(name, attach, function) \
    NCB_REGISTER_FUNCTION_COMMON(                   \
        name,                                       \
        attach ## _ ## name,                        \
        TJS_W(# attach),                            \
        function)

// NCB_REGISTER_FUNCTION_COMMON 展开为：
// 创建一个匿名 tag 结构体和对应的自动注册器
struct ncbFunctionTag_attach_name {};
template <>
struct ncbNativeFunctionAutoRegisterTempl<ncbFunctionTag_attach_name>
    : public ncbNativeFunctionAutoRegister
{
    ncbNativeFunctionAutoRegisterTempl()
        : ncbNativeFunctionAutoRegister(NCB_MODULE_NAME) {}
    void Regist() const {
        RegistFunction(TJS_W("name"), TJS_W("attach"), &function);
    }
    void Unregist() const {
        UnregistFunction(TJS_W("name"), TJS_W("attach"));
    }
};
static ncbNativeFunctionAutoRegisterTempl<ncbFunctionTag_attach_name>
    ncbFunctionAutoRegister_attach_name;
```

### 独立函数注册示例

```cpp
// 定义一个全局 C++ 函数
static int myGlobalAdd(int a, int b) {
    return a + b;
}

// 将函数注册到 TJS2 全局作用域
// 第一个参数：TJS2 端的函数名
// 第二个参数：挂载目标（0 = 全局作用域）
// 第三个参数：C++ 函数
NCB_ATTACH_FUNCTION(myAdd, Scripts, myGlobalAdd);
// 注册后 TJS2 端可以这样调用：
// var result = Scripts.myAdd(3, 5);  // result = 8
```

也可以挂载到全局作用域：

```cpp
// 挂载到全局（不指定附着对象）
NCB_REGISTER_FUNCTION(myAdd, myGlobalAdd);
// TJS2 端：var result = myAdd(3, 5);
```

## ncbRegistClass —— 注册 DSL 适配器

### DSL 的工作原理

`ncbRegistClass<IMPL>` 是一个模板类，它将 C++ 类型信息翻译为 TJS2 注册调用。当你在 `NCB_REGISTER_CLASS(MyClass) { ... }` 的花括号内写 `NCB_METHOD(Foo)` 时，实际上是在 `ncbRegistClass::Regist()` 方法体内调用成员函数。

```cpp
// 源码位置：cpp/core/plugin/ncbind.hpp 第 1664-1815 行（简化）
template <class IMPL>
struct ncbRegistClass : public ncbNativeClassMethodBase::InvokeType {
    typedef IMPL _ImplementT;                    // 底层实现（如 ncbRegistNativeClass）
    typedef typename _ImplementT::NativeClassT _ClassT;  // C++ 类

    // 构造时调用 Begin()，析构时调用 End()
    // 利用 RAII 保证 RegistBegin/RegistEnd 一定成对调用
    ncbRegistClass(_ImplementT &d, bool r)
        : _impl(d), _isRegist(r) { Begin(); }
    ~ncbRegistClass() { End(); }

    // 注册一个方法
    template <typename NAME, typename MethodT, class IVT>
    void Method(NAME n, MethodT m, IVT const&) {
        // 创建 ncbNativeClassMethod 对象并注册
        DoItem(GetName(n),
               ncbNativeClassMethod<
                   InvokeCommand<ClassT, MethodT, IVT>
               >::Create(m, _isRegist));
    }

    // 注册一个属性（getter + setter）
    template <typename NAME, typename GetterT, typename SetterT, class IVT>
    void Property(NAME n, GetterT g, SetterT s, IVT const&) {
        DoItem(GetName(n),
               ncbNativeClassProperty<
                   PropertyCommand<ClassT, GetterT, SetterT, IVT>
               >::Create(g, s, _isRegist));
    }

    // 注册构造函数
    template <typename MethodT>
    void Constructor(TypeWrap<MethodT>) {
        DoItem(GetName(),  // 构造函数名 = 类名
               ncbNativeClassConstructor<
                   InvokeCommand<ClassT, MethodT, ivtCtor>
               >::Create(0, _isRegist));
    }

    // 注册常量值
    template <typename NAME, typename VALUE>
    void Variant(NAME n, VALUE const v, FlagsT flag = TJS_STATICMEMBER) {
        // 通过类型转换器将 C++ 值转为 tTJSVariant
        // 然后调用 RegistVariant 挂载到类对象上
    }

    // 实际的 DSL 入口 —— 由宏展开的模板特化提供
    void Regist();
};
```

整个 DSL 的调用链：

```
NCB_REGISTER_CLASS(MyClass) {     // 宏展开为 ncbRegistClass::Regist() 方法体
    NCB_METHOD(Foo);               // 展开为 Method(TJS_W("Foo"), &Class::Foo)
    NCB_PROPERTY(bar, GetBar, SetBar);  // 展开为 Property(...)
}
// ncbRegistClass 析构 → 调用 End() → RegistEnd()
```

```
调用链详细流程：
NCB_METHOD(Foo)
  → Method(TJS_W("Foo"), &MyClass::Foo, Normal)
    → DoItem("Foo", ncbNativeClassMethod<...>::Create(&MyClass::Foo, true))
      → _impl.RegistItem("Foo", iMethodObject)
        → TJSNativeClassRegisterNCM(_classobj, "Foo", dispatch, ...)
```

## 动手实践

> 本练习带你从零完成一个 `StringStack` 插件类的注册，巩固 `NCB_REGISTER_CLASS` 和 `NCB_ATTACH_CLASS` 两种模式的使用。

### 步骤 1：创建插件源文件

在 `cpp/plugins/` 目录下新建 `string_stack.cpp`：

```cpp
// 文件：cpp/plugins/string_stack.cpp
#include "ncbind.hpp"
#include <vector>
#include <string>

// 定义模块名 —— 模拟游戏脚本中 Plugins.link("string_stack.dll")
#define NCB_MODULE_NAME TJS_W("string_stack.dll")

// C++ 实现类：一个简单的字符串栈
class StringStack {
public:
    // 构造函数：可指定最大容量（默认 100）
    StringStack(int maxSize = 100) : maxSize_(maxSize) {
        TVPAddLog(ttstr(TJS_W("StringStack created, maxSize="))
                  + ttstr(maxSize));
    }

    ~StringStack() {
        TVPAddLog(ttstr(TJS_W("StringStack destroyed, had "))
                  + ttstr((tjs_int)stack_.size()) + TJS_W(" items"));
    }

    // 入栈
    void Push(const ttstr& str) {
        if ((int)stack_.size() >= maxSize_) {
            TVPThrowExceptionMessage(TJS_W("Stack overflow"));
        }
        stack_.push_back(str);
    }

    // 出栈
    ttstr Pop() {
        if (stack_.empty()) {
            TVPThrowExceptionMessage(TJS_W("Stack underflow"));
        }
        ttstr top = stack_.back();
        stack_.pop_back();
        return top;
    }

    // 查看栈顶（不弹出）
    ttstr Peek() const {
        if (stack_.empty()) return TJS_W("");
        return stack_.back();
    }

    // 属性：获取当前元素数量
    int GetSize() const { return (int)stack_.size(); }

    // 属性：判断是否为空
    bool GetEmpty() const { return stack_.empty(); }

    // 清空栈
    void Clear() { stack_.clear(); }

private:
    std::vector<ttstr> stack_;  // 底层存储
    int maxSize_;               // 最大容量
};
```

### 步骤 2：使用 NCB_REGISTER_CLASS 注册

在同一文件末尾添加注册代码：

```cpp
// 注册 StringStack 为新的 TJS2 类
NCB_REGISTER_CLASS(StringStack) {
    // 注册构造函数 —— 接受一个 int 参数（最大容量）
    NCB_CONSTRUCTOR((int));

    // 注册方法
    NCB_METHOD(Push);    // void Push(ttstr)
    NCB_METHOD(Pop);     // ttstr Pop()
    NCB_METHOD(Peek);    // ttstr Peek()
    NCB_METHOD(Clear);   // void Clear()

    // 注册只读属性
    NCB_PROPERTY_RO(size, GetSize);    // .size 只读
    NCB_PROPERTY_RO(empty, GetEmpty);  // .empty 只读
}
```

### 步骤 3：再用 NCB_ATTACH_CLASS 扩展 Scripts 类

在同一文件中继续添加——向 TJS2 内建 `Scripts` 类附着一个 `reverseString` 工具方法：

```cpp
// 用于附着到 Scripts 类的辅助类
class ScriptsStackHelper {
public:
    ScriptsStackHelper() = default;  // 延迟创建需要默认构造函数

    // 静态方法：反转字符串
    static tjs_error reverseString(
        tTJSVariant *result,        // 返回值
        tjs_int numparams,          // 参数个数
        tTJSVariant **param,        // 参数数组
        ScriptsStackHelper *)       // 原生实例（这里不使用）
    {
        if (numparams < 1) return TJS_E_BADPARAMCOUNT;

        ttstr input = *param[0];           // 取第一个参数
        tjs_int len = input.GetLen();      // 获取字符串长度
        ttstr output;
        // 逆序拼接
        for (tjs_int i = len - 1; i >= 0; --i) {
            output += input[i];
        }
        if (result) *result = output;      // 设置返回值
        return TJS_S_OK;
    }
};

// 将 reverseString 附着到 Scripts 类
NCB_ATTACH_CLASS(ScriptsStackHelper, Scripts) {
    NCB_METHOD_RAW_CALLBACK(reverseString,
                            ScriptsStackHelper::reverseString,
                            TJS_STATICMEMBER);
}
```

编辑 `cpp/plugins/CMakeLists.txt`，在插件列表中添加新文件：

csvParser 是一个特殊案例——它使用 `NCB_PRE_REGIST_CALLBACK` 在 PreRegist 阶段执行初始化，然后用传统 TJS2 宏（`TJS_BEGIN_NATIVE_MEMBERS`）而非 `NCB_REGISTER_CLASS` 来注册类成员。这说明 ncbind 的三阶段机制（Pre/Class/Post）可以与传统 TJS2 注册 API 混用。

## 常见错误与排查

### 错误 1：忘记定义 NCB_MODULE_NAME

**症状：** 编译错误，提示 `NCB_MODULE_NAME` 未定义。

```
error: 'NCB_MODULE_NAME' was not declared in this scope
```

**原因：** 每个使用 ncbind 宏的 `.cpp` 文件都**必须**在 `#include "ncbind.hpp"` 之前定义 `NCB_MODULE_NAME`。这个宏被展开为自动注册器构造函数的第一个参数。

**修复：**

```cpp
// ✅ 正确：在 include 之前定义
#define NCB_MODULE_NAME TJS_W("my_plugin.dll")
#include "ncbind.hpp"

// ❌ 错误：漏掉了 NCB_MODULE_NAME
#include "ncbind.hpp"
NCB_REGISTER_CLASS(MyClass) { ... }  // 编译失败
```

**注意事项：** 模块名必须与游戏脚本中 `Plugins.link(...)` 的参数一致（含 `.dll` 后缀），因为引擎在查找时会将模块名转为小写后比对。

### 错误 2：在 NCB_ATTACH_CLASS 中注册构造函数

**症状：** 运行时抛出异常 `"Constructor attached: ClassName"`。

**原因：** `ncbAttachTJS2Class::RegistItem` 内部有显式检测——如果成员名与类名相同（即构造函数），会直接抛出 `TVPThrowExceptionMessage`。附着模式下，目标类（如 Scripts、Window）已经有自己的构造函数，不允许被覆盖。

**修复：** 使用普通方法名代替构造函数，或改用 `NCB_REGISTER_CLASS` 创建新类。

```cpp
// ❌ 错误：在 ATTACH 模式下注册构造函数
NCB_ATTACH_CLASS(MyHelper, Scripts) {
    NCB_CONSTRUCTOR(());  // 运行时异常！
}

// ✅ 正确：ATTACH 模式只注册方法和属性
NCB_ATTACH_CLASS(MyHelper, Scripts) {
    NCB_METHOD(DoSomething);
    NCB_PROPERTY_RO(value, GetValue);
}
```

### 错误 3：重复注册同一个类

**症状：** 运行时抛出异常 `"Already registerd class."`（注意源码中的拼写错误是原版代码的特征）。

**原因：** `ncbClassInfo<T>::Set()` 在设置类信息时会检查是否已存在。如果同一个 C++ 类被 `NCB_REGISTER_CLASS` 注册了两次（例如在两个不同的 `.cpp` 文件中），就会在第二次注册时触发此错误。

**修复方案：**

```cpp
// ❌ 错误：在两个文件中都注册同一个类
// file_a.cpp
NCB_REGISTER_CLASS(MyClass) { NCB_METHOD(Foo); }

// file_b.cpp
NCB_REGISTER_CLASS(MyClass) { NCB_METHOD(Bar); }
// 链接后运行 → "Already registerd class."

// ✅ 正确：所有注册放在一个 NCB_REGISTER_CLASS 块中
// my_class_plugin.cpp
NCB_REGISTER_CLASS(MyClass) {
    NCB_METHOD(Foo);
    NCB_METHOD(Bar);
}
```

### 错误 4：NCB_ATTACH_CLASS 的目标类不存在

**症状：** 运行时找不到目标类对象，注册静默失败，脚本端调用时报 `member not found`。

**原因：** `ncbAttachTJS2Class::RegistBegin` 中通过 `GetGlobalObject` 从全局作用域获取目标类。如果该类尚未创建（例如拼写错误、或目标类在当前插件之后才注册），获取会失败。

**修复：** 确认目标类名拼写正确，并确保目标类已经在 TJS2 全局作用域中存在。内建类（Scripts、Window、System 等）在引擎初始化时就已创建，可以安全附着。

```cpp
// ❌ 错误：目标类名拼写错误
NCB_ATTACH_CLASS(MyHelper, Scirpts) { ... }  // "Scirpts" 拼错

// ✅ 正确
NCB_ATTACH_CLASS(MyHelper, Scripts) { ... }
```

## 本节小结

- `ncbRegistNativeClass<T>` 是类注册的核心执行器，负责创建类对象、生成类 ID、注册 finalize、挂载到全局作用域
- 注册生命周期分为 `RegistBegin`（创建类对象）→ `RegistItem`（挂载成员）→ `RegistEnd`（全局挂载）三步，反注册严格对称
- `ncbAutoRegister` 通过链表收集所有静态注册器，分三阶段（PreRegist / ClassRegist / PostRegist）按模块名分组
- `NCB_REGISTER_CLASS` 宏经过三层展开，最终创建静态 `ncbNativeClassAutoRegister<T>` 对象并定义注册 DSL 函数体
- `NCB_ATTACH_CLASS` 不创建新类，而是向已有 TJS2 类追加方法/属性，使用延迟创建（Delay Create）在首次访问时才实例化 C++ 对象
- `NCB_ATTACH_FUNCTION` 用于注册独立函数到指定对象，每个函数通过匿名 tag 结构体避免 ODR 冲突
- `ncbRegistClass` 利用 RAII 保证 Begin/End 配对，其内部 DSL（Method/Property/Constructor/Variant）将 C++ 类型信息翻译为 TJS2 注册调用
- `NCB_MODULE_NAME` 决定模块在 `_internal_plugins` 中的键名，必须与游戏脚本中 `Plugins.link(...)` 的参数一致

### 题目 1：分析宏展开结果

给定以下代码：

```cpp
#define NCB_MODULE_NAME TJS_W("example.dll")
#include "ncbind.hpp"

class Greeter {
public:
    Greeter(const ttstr& name) : name_(name) {}
    ttstr SayHello() { return TJS_W("Hello, ") + name_; }
    ttstr GetName() const { return name_; }
private:
    ttstr name_;
};

NCB_REGISTER_CLASS(Greeter) {
    NCB_CONSTRUCTOR((ttstr));
    NCB_METHOD(SayHello);
    NCB_PROPERTY_RO(name, GetName);
}
```

请回答：
1. `NCB_REGISTER_CLASS(Greeter)` 展开后会创建什么名字的静态变量？
2. 该静态变量的构造函数接受哪两个参数？
3. 在 TJS2 中，`new Greeter("World")` 会触发哪些 ncbind 内部调用？

<details>
<summary>查看答案</summary>

1. 静态变量名为 `ncbNativeClassAutoRegister_Greeter`，类型为 `ncbNativeClassAutoRegister<Greeter>`。

2. 构造函数接受两个参数：
   - 第一个参数：`NCB_MODULE_NAME`，即 `TJS_W("example.dll")`，用作模块名
   - 第二个参数：`TJS_W("Greeter")`，用作 TJS2 端的类名

3. 调用链如下：
   ```
   TJS2: new Greeter("World")
     → TJS2 引擎查找全局作用域中的 "Greeter" 类对象
     → 类对象是 tTJSNativeClassForPlugin 类型
     → 调用注册的构造函数（成员名 "Greeter"）
     → ncbNativeClassConstructor<...>::Callback 被触发
     → ncbInvoker 将 tTJSVariant 参数转换为 ttstr
     → 调用 C++ 的 Greeter("World") 构造函数
     → ncbInstanceAdaptor<Greeter> 将 C++ 实例绑定到 TJS2 对象
     → 返回 TJS2 对象给脚本端
   ```

</details>

需求：你要为 KrKr2 添加一个 JSON 解析功能。提供两种使用方式：

<summary>查看答案</summary>

需要同时使用 `NCB_REGISTER_CLASS` 和 `NCB_ATTACH_CLASS` 两种模式：

```cpp

#include "ncbind.hpp"

// ==========================================
// 方案一：NCB_REGISTER_CLASS 创建独立类
// ==========================================
class JSONParser {
public:
    JSONParser() : root_(nullptr) {}
    ~JSONParser() { /* 释放 root_ */ }

    // 解析 JSON 字符串
    void Parse(const ttstr& jsonStr) {
        // ... 解析逻辑，结果存入 root_
    }

    // 获取解析结果中的字段
    tTJSVariant Get(const ttstr& key) {
        // ... 根据 key 返回对应值
        return tTJSVariant();
    }

    // 属性：是否解析成功
    bool GetValid() const { return root_ != nullptr; }

private:
    void* root_;  // JSON 根节点
};

NCB_REGISTER_CLASS(JSONParser) {
    NCB_CONSTRUCTOR(());             // 无参构造
    NCB_METHOD(Parse);               // 解析方法
    NCB_METHOD(Get);                 // 取值方法
    NCB_PROPERTY_RO(valid, GetValid); // 只读属性
}

// ==========================================
// 方案二：NCB_ATTACH_CLASS 扩展 Scripts
// ==========================================
class ScriptsJSONHelper {
public:
    ScriptsJSONHelper() = default;

    static tjs_error parseJSON(
        tTJSVariant *result, tjs_int numparams,
        tTJSVariant **param, ScriptsJSONHelper *)
    {
        if (numparams < 1) return TJS_E_BADPARAMCOUNT;
        ttstr jsonStr = *param[0];
        // ... 解析并将结果设置到 *result
        return TJS_S_OK;
    }
};

NCB_ATTACH_CLASS(ScriptsJSONHelper, Scripts) {
    RawCallback(TJS_W("parseJSON"),
                &ScriptsJSONHelper::parseJSON,
                TJS_STATICMEMBER);
}
```
