| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| 方法分发器 | Method Dispatcher (`ncbNativeClassMethod`) | 将 TJS2 的 `FuncCall` 调用转发到 C++ 成员方法的模板类 |
| 属性分发器 | Property Dispatcher (`ncbNativeClassProperty`) | 将 TJS2 的 `PropGet`/`PropSet` 转发到 C++ getter/setter 的模板类 |

| 调用模式 | Invoke Select (`ivsMethod` / `ivsProxy` / `ivsConstructor` / `ivsFactory`) | `doInvokeBase` 内部通过标签分发选择的四种不同调用路径 |
| 原始回调 | Raw Callback (`ncbRawCallbackMethod`) | 绕过自动类型转换、直接操作 `tTJSVariant` 参数的回调注册方式 |

| 代理模式 | Proxy Mode | 将原生实例指针作为第一个参数传给自由函数（而非成员方法）的调用方式 |
| 桥接模式 | Bridge Mode | 在获取原生实例后通过类型转换函子转为另一种类型，再调用其方法 |
| 访问拒绝 | Access Denied (`AccessDenied`) | 对只读/只写属性的缺失方向返回 `TJS_E_ACCESSDENYED` 的哑结构体 |
| 子类注册 | SubClass Registration (`ncbSubClassItem`) | 将一个 C++ 类注册为另一个 TJS2 类的嵌套子类 |
| 常量注册 | Variant Registration | 通过类型转换器将 C++ 常量值注册为 TJS2 类的静态成员 |

## 概述：从宏到模板类的完整链路

上一节我们学习了 `NCB_REGISTER_CLASS` 和 `NCB_ATTACH_CLASS` 两大宏如何注册整个类。本节将深入这些注册块 **内部** 的成员注册机制——当你写下 `NCB_METHOD(doSomething)` 时，ncbind 框架在编译期和运行期分别做了什么。

整体流程可以用一张图概括：

```
NCB_METHOD(myFunc)                        ← 用户写的宏
    ↓ 宏展开
Method(TJS_W("myFunc"), &Class::myFunc)   ← ncbRegistClass 的 DSL 方法
    ↓ 模板推导
ncbNativeClassMethod<                     ← 方法分发器模板类
    InvokeCommand<ClassT, MethodT, ivtNormal>
>::Create(m, _isRegist)
    ↓ 运行期调用
FuncCall(flag, name, hint,                ← TJS2 虚拟机调用入口
         result, numparams, param, objthis)
    ↓ 内部转发
doInvoke<CommandT>(_method, result,       ← 构造 doInvokeX 调用器
                   numparams, param, objthis)
    ↓ 参数提取
paramsFunctor<METHOD>(result, n, param)   ← 逐个转换参数类型
    ↓ 最终调用
CallerT::Invoke(functor, method, inst)    ← 调用 C++ 成员方法
```

这条链路贯穿了宏展开→模板实例化→类型推导→参数转换→方法调用的全过程。接下来我们逐层拆解。

## 一、NCB_METHOD —— 方法注册

### 1.1 宏展开

`NCB_METHOD` 是最常用的方法注册宏。它的定义极其简洁（`ncbind.hpp` 第 2256 行）：

```cpp
// ncbind.hpp 第 2256 行
#define NCB_METHOD(method) \
    NCB_METHOD_DIFFER(method, method)

// NCB_METHOD_DIFFER 的定义（第 2255 行）
#define NCB_METHOD_DIFFER(name, method) \
    Method(TJS_W(# name), &Class::method)
```

`NCB_METHOD(doSomething)` 展开后变成 `Method(TJS_W("doSomething"), &Class::doSomething)`。这里的 `Class` 是 `ncbRegistClass` 内部的类型别名（typedef），在 `NCB_REGISTER_CLASS` 注册块中指向你正在注册的 C++ 类。`NCB_METHOD_DIFFER` 允许 TJS2 侧的方法名与 C++ 侧不同，例如 `NCB_METHOD_DIFFER(jsName, cppMethod)` 会将 C++ 的 `cppMethod` 暴露为 TJS2 的 `jsName`。

### 1.2 ncbRegistClass::Method() —— DSL 入口

`Method()` 是 `ncbRegistClass` 模板类的成员方法（第 1716 行），它是整个方法注册的 DSL（Domain-Specific Language，领域专用语言——一种让注册代码读起来像声明而非命令的设计风格）入口：

```cpp
// ncbind.hpp 第 1715-1718 行
template <typename NAME, typename MethodT, class IVT>
void Method(NAME n, MethodT m, IVT const&) {
    DoItem(GetName(n),
           ncbNativeClassMethod<
               InvokeCommand<ClassT, MethodT, IVT>
           >::Create(m, _isRegist));
}

// 省略 IVT 时默认使用 Normal（即 ivtNormal）
// ncbind.hpp 第 1731 行
template <typename NAME, typename MethodT>
void Method(NAME n, MethodT m) {
    Method(n, m, Normal);  // Normal 是 ivtNormal 类型的 const 成员
}
```

这段代码做了三件事：

1. **`GetName(n)`**：将宏传入的 `TJS_W("doSomething")` 转成内部名称类型
2. **`ncbNativeClassMethod<...>::Create(m, _isRegist)`**：创建一个方法分发器（Method Dispatcher）对象，它实现了 TJS2 的 `FuncCall` 接口
3. **`DoItem(name, item)`**：将该分发器注册到 TJS2 类对象上（注册时）或从中移除（注销时）

### 1.3 InvokeCommand —— 编译期调用描述

`InvokeCommand` 是一个纯编译期结构体，它将三个信息打包在一起（定义位于 `ncbNativeClassMethodBase::InvokeType` 内部）：

```cpp
// 概念性展开（实际代码分布在多处）
template <class ClassT, typename MethodT, class InvokeTag>
struct InvokeCommand {
    typedef ClassT  ClassT;   // 原生 C++ 类类型
    typedef MethodT MethodT;  // 成员方法指针类型（如 int (MyClass::*)(const tjs_char*)）
    typedef InvokeTag IVT;    // 调用方式标签

    // Flags：决定注册到 TJS2 时的标志位
    static const FlagsT Flags = /* 根据 MethodT 是否为 static 推导 */;

    // InvokeSelect：选择 doInvokeBase 中的哪条调用路径
    static const int InvokeSelect = /* ivsMethod / ivsProxy / ... */;
};
```

其中 `InvokeTag`（调用方式标签）有以下几种：

| 标签类型 | 含义 | 使用场景 |
|---------|------|---------|
| `ivtNormal` | 普通成员方法调用 | `NCB_METHOD`（默认） |
| `ivtProxy<ivtNormal>` | 代理调用（自由函数，第一参数为实例指针） | `NCB_METHOD_PROXY` |
| `ivtCtor` | 构造函数调用 | `NCB_CONSTRUCTOR` |
| `ivtFactory` | 工厂函数调用 | `Factory()` |
| `ivtBridge<METHOD>` | 桥接调用（类型转换后再调方法） | `Bridge<B>` |

### 1.4 ncbNativeClassMethod —— 方法分发器

`ncbNativeClassMethod` 是真正实现 TJS2 `FuncCall` 接口的模板类（第 1326-1358 行）：

```cpp
// ncbind.hpp 第 1326-1358 行（简化注释）
template <class CommandT>
struct ncbNativeClassMethod : public ncbNativeClassMethodBase {
    typedef typename CommandT::MethodT MethodT;

    // 构造函数：保存方法指针
    ncbNativeClassMethod(MethodT m)
        : ncbNativeClassMethodBase(nitMethod), _method(m) {
        if (!_method)
            TVPThrowExceptionMessage(
                TJS_W("No method pointer."));
    }

    // TJS2 调用入口
    tjs_error FuncCall(
        tjs_uint32 flag,
        const tjs_char* membername,
        tjs_uint32* hint,
        tTJSVariant* result,
        tjs_int numparams,
        tTJSVariant** param,
        iTJSDispatch2* objthis) override
    {
        // 如果 membername 不为空，说明是通过成员名
        // 间接调用（如 obj.method()），委托给基类处理
        if (membername)
            return BaseT::FuncCall(
                flag, membername, hint,
                result, numparams, param, objthis);

        // objthis 为空说明原生实例已崩溃
        if (!objthis)
            return TJS_E_NATIVECLASSCRASH;

        // 核心：构造调用器并执行
        return (doInvoke<CommandT>(
            _method, result, numparams,
            param, objthis)).Invoke();
    }

protected:
    MethodT const _method;  // 保存的 C++ 方法指针
};
```

关键点：

- **`nitMethod`**：告诉 TJS2 这是一个"函数"类型的成员（与 `nitProperty` 相对）
- **`membername` 检查**：TJS2 的调用协议中，当 `membername` 不为 `nullptr` 时，表示调用者在访问某个成员。如果这个分发器不是目标成员，就把请求转发给基类（类似于责任链模式，Chain of Responsibility——多个处理器按链式结构依次判断是否处理请求的设计模式）
- **`doInvoke<CommandT>`**：这是连接分发器和参数提取的桥梁，下面详细讲

### 1.5 doInvokeBase —— 四种调用模式

`doInvokeBase` 是所有调用路径的基类（第 1087-1139+ 行），它内部定义了四个枚举值，对应四种完全不同的调用路径：

```cpp
// ncbind.hpp 第 1087-1090 行
struct doInvokeBase {
    typedef tjs_error ResultT;
    enum { Hook = true };
    // 四种调用模式
    enum {
        ivsMethod,       // 普通成员方法
        ivsProxy,        // 代理（自由函数，首参为实例指针）
        ivsConstructor,  // 构造函数（new 实例）
        ivsFactory       // 工厂函数（自定义创建逻辑）
    };
    // ... 参数保持 ...
};
```

四种模式通过 `DefsT::NumTag<N>` 进行编译期标签分发（Tag Dispatch——C++ 模板编程中根据编译期常量选择不同函数重载的技术），每种模式的 `CallInvoke` 实现如下：

**模式 1：ivsMethod（普通成员方法）**

```cpp
// ncbind.hpp 第 1105-1108 行
template <typename MethodT, class ClassT, class FunctorT>
ResultT CallInvoke(MethodT const& m, ClassT* inst,
    DefsT::Tag<FunctorT>,
    DefsT::NumTag<ivsMethod>) const
{
    // 用 paramsFunctor 提取参数，调用成员方法
    return CallerT::Invoke(
        FunctorT(_result, _numparams, _param),
        m,    // 方法指针
        inst  // C++ 实例指针
    ) ? TJS_S_OK : TJS_E_FAIL;
}
```

这是最常见的路径。`FunctorT` 就是 `paramsFunctor<METHOD>`，它负责从 `tTJSVariant` 数组中提取参数。`CallerT::Invoke` 是一个编译期生成的调用分发器（在 `ncb_invoke.hpp` 中通过预处理器循环生成），根据参数个数展开为 `inst->*m(arg1, arg2, ...)` 的直接调用。

**模式 2：ivsProxy（代理调用）**

```cpp
// ncbind.hpp 第 1110-1113 行
template <typename MethodT, class ClassT, class FunctorT>
ResultT CallInvoke(MethodT const& m, ClassT* inst,
    DefsT::Tag<FunctorT>,
    DefsT::NumTag<ivsProxy>) const
{
    // 使用 paramsFunctorWithInstance，
    // 将实例指针作为第一个参数传入
    return CallerT::Invoke(
        FunctorT(_result, _numparams, _param, inst),
        m,    // 自由函数指针
        inst  // （此处 inst 不用于成员调用，
              //   而是被 FunctorT 注入为首参数）
    ) ? TJS_S_OK : TJS_E_FAIL;
}
```

Proxy 模式用于注册"自由函数"而非成员方法。`FunctorT` 此时是 `paramsFunctorWithInstance`，它继承自 `paramsFunctor`，但在提取第 1 个参数时不从 `tTJSVariant` 数组取，而是直接返回原生实例指针。

**模式 3：ivsConstructor（构造函数）**

```cpp
// ncbind.hpp 第 1115-1132 行（简化）
template <typename MethodT, class ClassT, class FunctorT>
ResultT CallInvoke(MethodT const&, ClassT* inst,
    DefsT::Tag<FunctorT>,
    DefsT::NumTag<ivsConstructor>) const
{
    try {
        FunctorT fnct(_result, _numparams, _param);
        // CallerT::Factory 调用 new ClassT(args...)
        inst = CallerT::Factory(fnct,
            CallerT::tTypeTag<ClassT>(),
            CallerT::tTypeTag<
                typename FunctorT::TraitsT::ArgsT>());
        if (!inst) {
            TVPThrowExceptionMessage(
                TJS_W("NativeClassInstance creation faild."));
            return TJS_E_FAIL;
        }
        // 将 C++ 实例绑定到 TJS2 对象
        if (!ncbInstanceAdaptor<ClassT>
                ::SetNativeInstance(_objthis, inst)) {
            delete inst;
            return TJS_E_NATIVECLASSCRASH;
        }
    } catch (...) {
        delete inst;  // 异常安全：失败时清理
        throw;
    }
    return TJS_S_OK;
}
```

构造函数路径与普通方法不同之处在于：它调用 `CallerT::Factory` 通过 `new` 创建实例，然后通过 `ncbInstanceAdaptor::SetNativeInstance` 将其绑定到 TJS2 对象上。异常安全保证：如果绑定失败或发生异常，实例会被 `delete`。

**模式 4：ivsFactory（工厂函数）**

```cpp
// ncbind.hpp 第 1134-1139+ 行（简化）
template <typename MethodT, class ClassT, class FunctorT>
ResultT CallInvoke(MethodT const& m, ClassT* inst,
    DefsT::Tag<FunctorT>,
    DefsT::NumTag<ivsFactory>) const
{
    try {
        // 工厂函数有完全控制权
        inst = CallerT::Factory(
            FunctorT(_result, _numparams,
                     _param, _objthis),
            m);  // 方法指针即工厂函数
        // ... 同 ivsConstructor 的绑定和异常处理
    } catch (...) { ... }
}
```

工厂模式将创建逻辑完全交给一个自定义函数。这在需要复杂初始化逻辑（如条件选择子类、对象池管理等）时非常有用。

下面是四种模式的对比表：

| 调用模式 | 典型宏 | 方法签名要求 | 实例处理 | 使用场景 |
|---------|--------|------------|---------|---------|
| `ivsMethod` | `NCB_METHOD` | `RetT (ClassT::*)(args)` | 从 objthis 获取已有实例 | 普通成员方法 |
| `ivsProxy` | `NCB_METHOD_PROXY` | `RetT (*)(ClassT*, args)` | 实例作为首参注入 | 自由函数操作实例 |
| `ivsConstructor` | `NCB_CONSTRUCTOR` | `void (ClassT::*)(args)`（形式上） | `new ClassT(args)` 创建新实例 | 类构造函数 |
| `ivsFactory` | `Factory()` | `tjs_error (*)(ClassT**, ...)` | 工厂函数创建并返回实例 | 自定义创建逻辑 |

### 1.6 完整示例：注册一个带方法的类

下面是一个完整可编译的示例，展示 `NCB_METHOD` 的完整用法：

```cpp
// my_calculator.cpp
#include "ncbind.hpp"

#define NCB_MODULE_NAME TJS_W("calculator.dll")

// C++ 原生类
class Calculator {
public:
    Calculator() : _value(0.0) {}

    // 加法——将被注册为 TJS2 方法
    double add(double a, double b) {
        _value = a + b;       // 保存最后结果
        return _value;
    }

    // 减法
    double subtract(double a, double b) {
        _value = a - b;
        return _value;
    }

    // 获取最后结果
    double getLastResult() const {
        return _value;
    }

    // 带三个参数的方法
    double multiplyAdd(double a, double b, double c) {
        _value = a * b + c;   // a × b + c
        return _value;
    }

private:
    double _value;
};

// 注册到 TJS2
NCB_REGISTER_CLASS(Calculator) {
    // 默认构造函数
    NCB_CONSTRUCTOR(());

    // 注册四个方法
    NCB_METHOD(add);
    NCB_METHOD(subtract);
    NCB_METHOD(getLastResult);
    NCB_METHOD(multiplyAdd);
}
```

对应的 TJS2 脚本调用：

```javascript
// test_calculator.tjs
var calc = new Calculator();

// add 方法：ncbind 自动将 TJS2 的数值转为 double
var sum = calc.add(3.14, 2.86);
System.inform("3.14 + 2.86 = " + sum);  // 输出: 6.0

// multiplyAdd：三参数自动提取
var result = calc.multiplyAdd(2.0, 3.0, 1.0);
System.inform("2×3+1 = " + result);     // 输出: 7.0

// getLastResult：零参数方法
var last = calc.getLastResult();
System.inform("最后结果: " + last);       // 输出: 7.0
```

## 二、paramsFunctor —— 参数自动提取

### 2.1 整体结构

`paramsFunctor` 是整个 ncbind 类型转换机制的核心（第 928-1010 行）。它的职责是：

1. 在编译期为每个参数生成对应的类型转换器
2. 在运行期从 `tTJSVariant**` 数组中逐个提取参数并转换为 C++ 类型
3. 将 C++ 方法的返回值转换回 `tTJSVariant`

```cpp
// ncbind.hpp 第 928-1010 行（简化注释）
template <typename METHOD>
struct paramsFunctor {
    typedef traits<METHOD> TraitsT;
    typedef typename TraitsT::ResultT RES;   // 返回值类型
    typedef typename TraitsT::ArgsT   ARGS;  // 参数类型列表
    enum { ARGC = TraitsT::ArgsCount };      // 参数个数

    // 返回值转换器（C++ → tTJSVariant）
    typedef typename ConvT::SelectConvertorType<
        RES, VariantT>::Type ResultConvT;

    // 参数转换器集合（tTJSVariant → C++ 各参数类型）
    typedef CallerT::tArgsExtract<
        ArgsExtor, ARGS, ARGC> ArgsConvT;

    // 构造函数：保存参数数组的引用
    paramsFunctor(VariantT* r, tjs_int n,
                  VariantT const* const* p)
        : _numparams(n), _result(r), _param(p) {}

    // 第 N 个参数的提取（被 CallerT::Invoke 调用）
    template <int N, typename T>
    inline T operator()(
        CallerT::tNumTag<N> const&,
        CallerT::tTypeTag<T> const&)
    {
        typedef typename TypeWrap<T>::Type ParamT;
        ParamT ret;
        // 从 ArgsConvT 中取出第 N 个转换器
        (CallerT::tArgsSelect<ArgsConvT, N>
            ::Get(_aconv))(
            ret,
            // 参数不足时用默认 tTJSVariant（void）
            (_numparams >= N)
                ? *(_param[N - 1])
                : VariantT()
        );
        return TypeWrap<T>::Restore(ret);
    }

    // 返回值处理：void 返回
    inline bool operator()() const {
        return true;
    }

    // 返回值处理：有返回值
    template <typename ResultT>
    inline bool operator()(ResultT r,
        CallerT::tTypeTag<ResultT> const&)
    {
        if (_result)
            _rconv(*_result, r);  // C++ 值 → tTJSVariant
        return true;
    }

private:
    ArgsConvT   _aconv;   // 参数转换器集合
    ResultConvT _rconv;   // 返回值转换器
    tjs_int     _numparams;
    VariantT*   _result;
    VariantT const* const* _param;
};
```

### 2.2 ArgsExtor —— 参数转换器选择

`ArgsExtor` 模板通过 `ncbTypeConvertor::SelectConvertorType` 为每个参数类型选择正确的转换器：

```cpp
// ncbind.hpp 第 944 行
template <typename T>
struct ArgsExtor {
    typedef typename ConvT::SelectConvertorType<
        VariantT,                       // 源类型：tTJSVariant
        typename TypeWrap<T>::Type      // 目标类型：去除引用后的 C++ 类型
    >::Type Type;
};
```

当 `CallerT::tArgsExtract` 展开时，它会为参数列表中的每个类型生成一个对应的转换器实例。例如对于 `double add(double a, double b)`：

```
ArgsConvT = {
    ncbToValueConvertor<double> t1;  // 第 1 个参数的转换器
    ncbToValueConvertor<double> t2;  // 第 2 个参数的转换器
};
```

这个转换器集合在 `paramsFunctor` 构造时就已经准备好了（作为成员变量 `_aconv`），等到提取参数时直接调用。

### 2.3 TypeWrap —— 引用类型处理

C++ 中的参数可能是值类型（`int`）、引用（`int&`）或常量引用（`const int&`）。`TypeWrap` 用一组偏特化统一处理：

```cpp
// ncbind.hpp 第 940-942 行
template <typename T>
struct TypeWrap {
    typedef T Type;
    static inline T Restore(Type t) { return t; }
};

template <typename T>
struct TypeWrap<T&> {
    typedef T* Type;                    // 引用 → 指针
    static inline T& Restore(Type t) {
        return *t;                      // 指针 → 解引用还原
    }
};

template <typename T>
struct TypeWrap<T const&> {
    typedef T const* Type;              // const 引用 → const 指针
    static inline T const& Restore(Type t) {
        return *t;
    }
};
```

这个设计解决了一个实际问题：`ncbTypeConvertor` 的输出是值或指针，但 C++ 方法的参数签名可能要求引用。`TypeWrap` 在转换时将引用"降级"为指针存储，在最终传参时通过 `Restore` 还原为引用。

### 2.4 paramsFunctorWithInstance —— 代理模式的参数注入

Proxy 模式需要将原生实例指针作为第一个参数传递。`paramsFunctorWithInstance` 继承自 `paramsFunctor`，巧妙地改写了第 1 个参数的提取逻辑（第 1013-1026 行）：

```cpp
// ncbind.hpp 第 1013-1026 行
template <class CLASS, class FUNCT>
struct paramsFunctorWithInstance : public FUNCT {
    typedef CLASS ClassT;

    paramsFunctorWithInstance(
        VariantT* r, tjs_int n,
        VariantT const* const* p,
        ClassT* inst)
        : FUNCT(r, n + 1, p - 1),  // 偏移参数数组！
          _inst(inst) {}

    // 通用：第 N 个参数由基类处理
    template <int N, typename T>
    inline T operator()(
        CallerT::tNumTag<N> const& idx,
        CallerT::tTypeTag<T> const& tag) {
        return FUNCT::operator()(idx, tag);
    }

    // 特化：第 1 个参数返回实例指针
    template <typename T>
    inline T operator()(
        CallerT::tNumTag<1> const&,
        CallerT::tTypeTag<T> const&) {
        return _inst;  // 直接返回 C++ 实例
    }

private:
    ClassT* _inst;
};
```

关键技巧在构造函数中：`FUNCT(r, n + 1, p - 1)`。它把参数数量加 1、参数指针前移 1 位，这样当基类按 `_param[N-1]` 取第 N 个参数时，第 2 个参数实际上取的是原始数组的第 1 个元素——完美地为实例指针"腾出"了位置。

### 2.5 nativeInstanceGetter —— 获取原生实例

在调用成员方法之前，需要从 TJS2 对象（`objthis`）中提取 C++ 原生实例指针。这由 `nativeInstanceGetter` 完成（第 1032-1061 行）：

```cpp
// ncbind.hpp 第 1032-1061 行
template <class T>
struct nativeInstanceGetterBase {
    typedef ncbInstanceAdaptor<T> AdaptorT;

    inline T* GetNativeInstance(iTJSDispatch2* objthis) {
        T* r = AdaptorT::GetNativeInstance(objthis);
        if (!r) SetError(TJS_E_NATIVECLASSCRASH);
        return r;
    }

    // 默认 Get 方法
    inline T* Get(iTJSDispatch2* objthis) {
        return GetNativeInstance(objthis);
    }
};

// 默认实现：直接委托给 Base
template <class T>
struct nativeInstanceGetter
    : public nativeInstanceGetterBase<T> {};
```

对于 Bridge 模式（桥接），有一个特殊版本 `bridgeInstanceGetter`（第 1064-1073 行），它在获取原生实例后通过一个转换函子（Conversion Functor）将 `FROM*` 转为 `TO*`：

```cpp
// ncbind.hpp 第 1064-1073 行
template <class FROM, class TO, typename CONV>
struct bridgeInstanceGetter
    : public nativeInstanceGetterBase<FROM>
{
    inline TO* Get(iTJSDispatch2* objthis) {
        FROM* obj = BaseT::GetNativeInstance(objthis);
        return obj ? conv(obj) : nullptr;
    }
private:
    CONV conv;  // 类型转换函子
};
```

## 三、NCB_PROPERTY —— 属性注册

属性（Property）是 TJS2 中用 `obj.name` 访问而非 `obj.name()` 调用的成员。ncbind 通过 `NCB_PROPERTY` 系列宏将 C++ 的 getter/setter 方法对注册为 TJS2 属性。与方法不同，属性有两个方向——读（PropGet）和写（PropSet），而且允许只注册其中一个方向。

### 3.1 宏展开

`NCB_PROPERTY` 系列宏定义在 `ncbind.hpp` 第 2261-2263 行：

```cpp
// ncbind.hpp 第 2261 行
#define NCB_PROPERTY(name, get, set) \
    Property(TJS_W(# name), &Class::get, &Class::set)

// 第 2262 行：只读属性（没有 setter）
#define NCB_PROPERTY_RO(name, get) \
    Property(TJS_W(# name), &Class::get, (int)0)

// 第 2263 行：只写属性（没有 getter）
#define NCB_PROPERTY_WO(name, set) \
    Property(TJS_W(# name), (int)0, &Class::set)
```

注意三个宏的核心区别：

| 宏 | getter 参数 | setter 参数 | TJS2 行为 |
|---|---|---|---|
| `NCB_PROPERTY(name, get, set)` | `&Class::get` | `&Class::set` | 可读可写 |
| `NCB_PROPERTY_RO(name, get)` | `&Class::get` | `(int)0` | 只读，写入返回 `TJS_E_ACCESSDENYED` |
| `NCB_PROPERTY_WO(name, set)` | `(int)0` | `&Class::set` | 只写，读取返回 `TJS_E_ACCESSDENYED` |

这里的 `(int)0` 是一个**哨兵值**（Sentinel Value——一种特殊值，它的作用不是携带数据，而是触发编译器选择特定的函数重载）。它的类型是 `int`，而不是函数指针类型，因此编译器会选择 `ncbRegistClass::Property` 的不同重载版本。

### 3.2 ncbRegistClass::Property() —— 三个重载

`Property()` 方法有三个重载（第 1721-1728 行），分别对应可读可写、只读、只写：

```cpp
// ncbind.hpp 第 1721-1724 行：可读可写
template <typename NAME, typename GetT, typename SetT>
void Property(NAME n, GetT g, SetT s) {
    typedef ncbNativeClassProperty<
        PropertyCommand<ClassT, GetT, SetT>
    > PropT;
    DoItem(GetName(n), PropT::Create(g, s, _isRegist));
}

// 第 1726 行：只读（setter 是 int 类型的哨兵值）
template <typename NAME, typename GetT>
void Property(NAME n, GetT g, int) {
    typedef ncbNativeClassProperty<
        PropertyCommand<ClassT, GetT, void>  // void 表示无 setter
    > PropT;
    DoItem(GetName(n), PropT::Create(g, (void*)0, _isRegist));
}

// 第 1728 行：只写（getter 是 int 类型的哨兵值）
template <typename NAME, typename SetT>
void Property(NAME n, int, SetT s) {
    typedef ncbNativeClassProperty<
        PropertyCommand<ClassT, void, SetT>  // void 表示无 getter
    > PropT;
    DoItem(GetName(n), PropT::Create((void*)0, s, _isRegist));
}
```

重载选择的关键在于 `int` 参数。当你写 `NCB_PROPERTY_RO(value, getValue)` 时：
1. 宏展开为 `Property(TJS_W("value"), &Class::getValue, (int)0)`
2. 第三个参数类型是 `int`，不是函数指针
3. 编译器选择第二个重载 `Property(NAME, GetT, int)`
4. 模板参数 `SetT` 被填充为 `void`，表示没有 setter

### 3.3 PropertyCommand —— 属性命令打包

`PropertyCommand` 的作用类似于方法注册中的 `InvokeCommand`，它将 getter 和 setter 的类型信息打包在一起：

```cpp
// 概念性展开（实际分布在 ncbNativeClassMethodBase 内部）
template <class ClassT, typename GetT, typename SetT>
struct PropertyCommand {
    typedef ClassT ClassT;
    typedef GetT   GetMethodT;    // getter 方法指针类型
    typedef SetT   SetMethodT;    // setter 方法指针类型

    // 内嵌两个 InvokeCommand：一个用于 getter，一个用于 setter
    typedef InvokeCommand<ClassT, GetT, ivtNormal> GetCommandT;
    typedef InvokeCommand<ClassT, SetT, ivtNormal> SetCommandT;
};
```

当 `SetT` 为 `void` 时（只读属性），`SetCommandT` 的相关代码路径不会被编译器实例化（因为 `ncbNativeClassProperty` 在检测到空 setter 时直接返回错误码，不会进入参数提取流程）。

### 3.4 ncbNativeClassProperty —— 属性分发器

`ncbNativeClassProperty` 是实现 TJS2 `PropGet` 和 `PropSet` 接口的模板类（第 1441-1496 行）：

```cpp
// ncbind.hpp 第 1441-1496 行（简化注释）
template <class PropCommandT>
struct ncbNativeClassProperty : public ncbNativeClassMethodBase {
    typedef typename PropCommandT::GetCommandT GetCommandT;
    typedef typename PropCommandT::SetCommandT SetCommandT;
    typedef typename GetCommandT::MethodT GetMethodT;
    typedef typename SetCommandT::MethodT SetMethodT;

    // 构造函数：保存 getter 和 setter 方法指针
    ncbNativeClassProperty(GetMethodT g, SetMethodT s)
        : ncbNativeClassMethodBase(nitProperty),
          _getter(g), _setter(s) {}

    // TJS2 读属性入口
    tjs_error PropGet(
        tjs_uint32 flag,
        const tjs_char* membername,
        tjs_uint32* hint,
        tTJSVariant* result,
        iTJSDispatch2* objthis) override
    {
        if (membername)
            return BaseT::PropGet(
                flag, membername, hint, result, objthis);

        // 没有 getter → 访问被拒绝
        if (!_getter) return TJS_E_ACCESSDENYED;

        if (!objthis) return TJS_E_NATIVECLASSCRASH;

        // 调用 getter 方法，结果写入 result
        return (doInvoke<GetCommandT>(
            _getter, result, 0, nullptr, objthis
        )).Invoke();
    }

    // TJS2 写属性入口
    tjs_error PropSet(
        tjs_uint32 flag,
        const tjs_char* membername,
        tjs_uint32* hint,
        const tTJSVariant* param,
        iTJSDispatch2* objthis) override
    {
        if (membername)
            return BaseT::PropSet(
                flag, membername, hint, param, objthis);

        // 没有 setter → 访问被拒绝
        if (!_setter) return TJS_E_ACCESSDENYED;

        if (!objthis) return TJS_E_NATIVECLASSCRASH;

        // 将 param 作为唯一参数调用 setter 方法
        return (doInvoke<SetCommandT>(
            _setter, nullptr, 1,
            const_cast<tTJSVariant const* const*>(&param),
            objthis
        )).Invoke();
    }

protected:
    GetMethodT const _getter;  // getter 方法指针（可能为 nullptr）
    SetMethodT const _setter;  // setter 方法指针（可能为 nullptr）
};
```

理解这段代码的关键点：

1. **`nitProperty`**：告诉 TJS2 这是一个"属性"类型的成员。TJS2 对属性使用 `PropGet`/`PropSet` 协议，而非方法的 `FuncCall` 协议
2. **`!_getter` / `!_setter` 检查**：当 getter 或 setter 为 `nullptr` 时（由只读/只写宏传入），直接返回 `TJS_E_ACCESSDENYED`（TJS2 的"访问被拒绝"错误码，注意拼写是 `DENYED` 而非 `DENIED`，这是 KiriKiri 原始代码中的历史拼写）
3. **getter 调用**：`numparams` 为 0（属性读取不需要参数），`result` 指向接收返回值的 `tTJSVariant`
4. **setter 调用**：`numparams` 为 1（传入一个值），`result` 为 `nullptr`（属性写入不返回值）

### 3.5 完整示例：带属性的类

```cpp
// my_config.cpp
#include "ncbind.hpp"

#define NCB_MODULE_NAME TJS_W("config.dll")

class Config {
public:
    Config() : _width(800), _height(600),
               _title(TJS_W("默认标题")) {}

        if (w < 1) w = 1;        // 参数校验
        _width = w;
    }

    // 高度（可读可写）

    void setHeight(int h) {
        if (h < 1) h = 1;
        _height = h;
    }

    // 标题（可读可写）
    tTJSString getTitle() const { return _title; }
    void setTitle(const tjs_char* t) { _title = t; }

    // 像素总数（只读——由宽高动态计算）

    tTJSString _title;
    bool _debugMode = false;

NCB_REGISTER_CLASS(Config) {

    NCB_PROPERTY(title,  getTitle,  setTitle);

    // 只读属性——没有 setter
```

对应的 TJS2 脚本：

## 四、NCB_CONSTRUCTOR / Factory —— 构造函数注册

构造函数注册是 ncbind 中最特殊的注册类型：它不像方法和属性那样操作已有实例，而是**创建新实例**。ncbind 提供了两种创建方式——`NCB_CONSTRUCTOR`（自动构造）和 `Factory()`（自定义工厂函数）。

### 4.1 NCB_CONSTRUCTOR 宏展开

`NCB_CONSTRUCTOR` 的定义（第 2253 行）：

```cpp
// ncbind.hpp 第 2253 行
#define NCB_CONSTRUCTOR(cargs) \
    Constructor(TypeWrap<void (Class::*) cargs>())
```

使用时传入参数列表（用括号包裹）：

```cpp
// 无参数构造
NCB_CONSTRUCTOR(());
// 展开为: Constructor(TypeWrap<void (Class::*)(void)>())

// 两个参数构造
NCB_CONSTRUCTOR((int, const tjs_char*));
// 展开为: Constructor(TypeWrap<void (Class::*)(int, const tjs_char*)>())
```

这里的 `TypeWrap<void (Class::*) cargs>()` 是一个零大小的类型标签对象——它不携带任何运行时数据，仅用于在编译期将参数类型信息传递给 `Constructor()` 方法。`void` 返回值类型是形式上的：构造函数不返回值，但 ncbind 需要一个完整的方法指针类型来推导参数类型。

### 4.2 ncbRegistClass::Constructor()

`Constructor()` 方法（第 1734 行）创建一个 `ncbNativeClassConstructor` 对象：

```cpp
// ncbind.hpp 第 1734 行
template <typename MethodT>
void Constructor(CallerT::tTypeTag<MethodT>) {
    typedef ncbNativeClassConstructor<
        InvokeCommand<ClassT, MethodT, ivtCtor>
    > CtorT;
    _impl.RegistItem(
        CtorT::Create(0, _isRegist));  // 0 表示没有方法指针
}
```

注意 `ivtCtor` 标签——它触发 `doInvokeBase` 中的 `ivsConstructor` 路径（第 1.5 节已讲解），该路径调用 `CallerT::Factory` 通过 `new ClassT(args...)` 创建实例。

### 4.3 ncbNativeClassConstructor —— 构造函数分发器

`ncbNativeClassConstructor`（第 1362-1395 行）继承自 `ncbNativeClassMethodBase`，实现 `FuncCall` 接口：

```cpp
// ncbind.hpp 第 1362-1395 行（简化注释）
template <class CommandT>
struct ncbNativeClassConstructor
    : public ncbNativeClassMethodBase
{
    // 构造函数不保存方法指针（new 操作不需要）
    ncbNativeClassConstructor()
        : ncbNativeClassMethodBase(nitMethod) {}

    tjs_error FuncCall(
        tjs_uint32 flag,
        const tjs_char* membername,
        tjs_uint32* hint,
        tTJSVariant* result,
        tjs_int numparams,
        tTJSVariant** param,
        iTJSDispatch2* objthis) override
    {
        if (membername)
            return BaseT::FuncCall(
                flag, membername, hint,
                result, numparams, param, objthis);

        if (!objthis) return TJS_E_NATIVECLASSCRASH;

        // ★ void 参数快捷路径
        if (numparams == 1 &&
            param[0]->Type() == tvtVoid)
        {
            // TJS2 用 new Class(void) 表示
            // "创建空适配器，不创建原生实例"
            return TJS_S_OK;
        }

        // 正常路径：通过 ivsConstructor 创建实例
        return (doInvoke<CommandT>(
            (typename CommandT::MethodT)0,
            result, numparams, param, objthis
        )).Invoke();
    }
};
```

#### void 参数快捷路径

第 1382 行的检查是一个重要的设计细节：当 TJS2 传入单个 `void` 类型参数时，构造函数不创建原生 C++ 实例，直接返回 `TJS_S_OK`。这是 TJS2 内部创建"空壳"对象的机制——用于继承场景，子类先创建空的父类适配器，然后由子类构造函数负责创建真正的原生实例。

```javascript
// TJS2 中的继承场景
class MyWindow extends Window {
    // Window 的构造函数先被调用（传入 void）
    // 创建空适配器，不分配原生 Window 实例
    // 然后 MyWindow 的初始化代码执行，
    // 可能调用父类方法来完成原生实例的创建
}
```

### 4.4 ncbNativeClassFactory —— 工厂函数分发器

当默认的 `new ClassT(args...)` 不够灵活时（例如需要从对象池获取实例、需要根据参数选择不同子类、需要额外的初始化步骤），可以使用工厂函数模式。`ncbNativeClassFactory` 定义在第 1397-1436 行：

```cpp
// ncbind.hpp 第 1397-1436 行（简化注释）
template <class ClassT>
struct ncbNativeClassFactory
    : public ncbNativeClassMethodBase
{
    // 工厂函数签名
    typedef tjs_error (*FactoryFunc)(
        ClassT**,              // 输出：创建的实例
        tjs_int,               // 参数个数
        tTJSVariant**,         // 参数数组
        iTJSDispatch2*         // TJS2 对象
    );

    ncbNativeClassFactory(FactoryFunc f)
        : ncbNativeClassMethodBase(nitMethod),
          _factory(f) {}

    tjs_error FuncCall(
        tjs_uint32 flag,
        const tjs_char* membername,
        tjs_uint32* hint,
        tTJSVariant* result,
        tjs_int numparams,
        tTJSVariant** param,
        iTJSDispatch2* objthis) override
    {
        if (membername)
            return BaseT::FuncCall(
                flag, membername, hint,
                result, numparams, param, objthis);

        if (!objthis) return TJS_E_NATIVECLASSCRASH;

        ClassT* inst = nullptr;
        // 调用工厂函数
        tjs_error hr = _factory(
            &inst, numparams, param, objthis);

        if (TJS_SUCCEEDED(hr) && inst) {
            // 绑定到 TJS2 对象
            if (!ncbInstanceAdaptor<ClassT>
                    ::SetNativeInstance(objthis, inst)) {
                delete inst;
                return TJS_E_NATIVECLASSCRASH;
            }
        }
        return hr;
    }

protected:
    FactoryFunc const _factory;
};
```

工厂函数的使用方式：

```cpp
// 工厂函数——完全控制实例创建
static tjs_error CreateImageLoader(
    ImageLoader** out,         // 输出指针
    tjs_int numparams,         // 参数个数
    tTJSVariant** param,       // 参数数组
    iTJSDispatch2* objthis)    // TJS2 对象
{
    if (numparams < 1)
        return TJS_E_BADPARAMCOUNT;

    // 根据参数选择不同的子类
    tTJSString format =
        param[0]->AsStringNoAddRef();
    if (format == TJS_W("png")) {
        *out = new PngLoader();
    } else if (format == TJS_W("jpeg")) {
        *out = new JpegLoader();
    } else {
        return TJS_E_INVALIDPARAM;
    }
    return TJS_S_OK;
}

NCB_REGISTER_CLASS(ImageLoader) {
    // 使用工厂函数代替默认构造函数
    Factory(&CreateImageLoader);

    NCB_METHOD(load);
    NCB_METHOD(getWidth);
    NCB_METHOD(getHeight);
}
```

### 4.5 Constructor 与 Factory 对比

| 特性 | `NCB_CONSTRUCTOR` | `Factory()` |
|------|-------------------|-------------|
| 创建方式 | `new ClassT(args...)` | 自定义函数 |
| 参数处理 | 自动类型转换 | 手动从 `tTJSVariant` 提取 |
| 灵活性 | 低——只能调用构造函数 | 高——可选子类、对象池等 |
| void 快捷路径 | ✅ 支持 | ❌ 需自行处理 |
| 错误处理 | 自动（异常安全） | 手动返回错误码 |
| 典型场景 | 简单类注册 | 多态工厂、复杂初始化 |

## 五、RawCallback —— 原始回调注册

前面介绍的 `NCB_METHOD` 和 `NCB_PROPERTY` 都会自动进行参数类型转换（通过 `paramsFunctor`）。但有些场景需要**绕过自动转换**，直接操作 `tTJSVariant` 参数——例如参数个数不固定、参数类型需要动态判断、或者需要直接操作 TJS2 内部对象。这时就需要 RawCallback。

### 5.1 两种 ncbRawCallbackMethod 特化

`ncbRawCallbackMethod` 有两个版本——一个用于原生实例回调，另一个用于传统 TJS2 回调函数。

**特化 1：原生实例回调（主模板，第 1504-1554 行）**

```cpp
// ncbind.hpp 第 1504-1554 行（简化注释）
template <class T>
struct ncbRawCallbackMethod
    : public ncbNativeClassMethodBase
{
    // 回调函数签名：带原生实例指针
    typedef tjs_error (*CallbackFunc)(
        tTJSVariant* result,   // 返回值
        tjs_int numparams,     // 参数个数
        tTJSVariant** param,   // 参数数组
        T* nativeInstance      // C++ 原生实例
    );

    ncbRawCallbackMethod(CallbackFunc cb,
                         tjs_uint32 flags)
        : ncbNativeClassMethodBase(nitMethod, flags),
          _callback(cb) {}

    tjs_error FuncCall(
        tjs_uint32 flag,
        const tjs_char* membername,
        tjs_uint32* hint,
        tTJSVariant* result,
        tjs_int numparams,
        tTJSVariant** param,
        iTJSDispatch2* objthis) override
    {
        if (membername)
            return BaseT::FuncCall(
                flag, membername, hint,
                result, numparams, param, objthis);

        // 检查是否为静态方法
        if (flag & TJS_STATICMEMBER) {
            // 静态方法不需要原生实例
            return _callback(
                result, numparams, param, nullptr);
        }

        if (!objthis) return TJS_E_NATIVECLASSCRASH;

        // 获取原生实例
        T* inst = ncbInstanceAdaptor<T>
            ::GetNativeInstance(objthis);
        if (!inst) return TJS_E_NATIVECLASSCRASH;

        return _callback(
            result, numparams, param, inst);
    }

protected:
    CallbackFunc const _callback;
};
```

关键特点：回调函数的最后一个参数是 C++ 原生实例指针 `T*`。如果注册时带了 `TJS_STATICMEMBER` 标志，则跳过实例获取，传 `nullptr`。

**特化 2：传统 TJS2 回调（`tTJSNativeClassMethodCallback` 特化，第 1557-1583 行）**

```cpp
// ncbind.hpp 第 1557-1583 行
template <>
struct ncbRawCallbackMethod<tTJSNativeClassMethodCallback>
    : public ncbNativeClassMethodBase
{
    // 传统 TJS2 回调签名（不带原生实例指针）
    // tTJSNativeClassMethodCallback 是 TJS2 定义的类型：
    // tjs_error (*)(tTJSVariant* result,
    //              tjs_int numparams,
    //              tTJSVariant** param,
    //              iTJSDispatch2* objthis)

    ncbRawCallbackMethod(
        tTJSNativeClassMethodCallback cb,
        tjs_uint32 flags)
        : ncbNativeClassMethodBase(nitMethod, flags),
          _callback(cb) {}

    tjs_error FuncCall(
        tjs_uint32 flag,
        const tjs_char* membername,
        tjs_uint32* hint,
        tTJSVariant* result,
        tjs_int numparams,
        tTJSVariant** param,
        iTJSDispatch2* objthis) override
    {
        if (membername)
            return BaseT::FuncCall(
                flag, membername, hint,
                result, numparams, param, objthis);

        // 通过 TJSCreateNativeClassMethod 包装
        // 直接调用传统回调
        return _callback(
            result, numparams, param, objthis);
    }

protected:
    tTJSNativeClassMethodCallback const _callback;
};
```

两种特化的核心区别：

| 特性 | 主模板 `ncbRawCallbackMethod<T>` | 特化 `ncbRawCallbackMethod<tTJSNativeClassMethodCallback>` |
|------|---|---|
| 最后一个参数 | `T* nativeInstance`（C++ 实例） | `iTJSDispatch2* objthis`（TJS2 对象） |
| 实例获取 | 自动通过 `ncbInstanceAdaptor` | 不获取，传原始 `objthis` |
| 静态方法 | 检查 `TJS_STATICMEMBER` 跳过实例 | 不处理（传 `objthis`） |
| 使用场景 | 需要访问 C++ 原生数据 | 纯 TJS2 层面操作 |

### 5.2 ncbRawCallbackPropertySelector 与 AccessDenied

当使用 `NCB_PROPERTY_RAW_CALLBACK_RO` 注册只读的原始回调属性时，setter 方向需要一个"占位符"。`ncbRawCallbackPropertySelector<int>` 承担这个角色（第 1594-1605 行）：

```cpp
// ncbind.hpp 第 1594-1605 行
template <>
struct ncbRawCallbackPropertySelector<int> {
    // AccessDenied 是一个哑结构体
    struct AccessDenied {
        // FuncCall 永远返回"访问被拒绝"
        tjs_error TJS_INTF_METHOD FuncCall(
            tjs_uint32, const tjs_char*,
            tjs_uint32*, tTJSVariant*,
            tjs_int, tTJSVariant**,
            iTJSDispatch2*)
        {
            return TJS_E_ACCESSDENYED;
        }
    };
    typedef AccessDenied Type;

    // Create 返回一个 AccessDenied 实例
    static Type* Create(int, tjs_uint32, bool) {
        return new Type();
    }
};
```

`AccessDenied`（访问拒绝——一个什么都不做、只返回错误码的哑对象）是 ncbind 中处理"缺失方向"的通用模式。无论 TJS2 如何调用它的 `FuncCall`，它都返回 `TJS_E_ACCESSDENYED`。

### 5.3 ncbRawCallbackProperty —— 原始回调属性

`ncbRawCallbackProperty` 将两个 `ncbRawCallbackMethod`（或 `AccessDenied`）组合为一个属性（第 1608-1653 行）：

```cpp
// ncbind.hpp 第 1608-1653 行（简化注释）
template <class GETTER, class SETTER>
struct ncbRawCallbackProperty
    : public ncbNativeClassMethodBase
{
    ncbRawCallbackProperty(
        typename GETTER::Type* g,
        typename SETTER::Type* s)
        : ncbNativeClassMethodBase(nitProperty),
          _getter(g), _setter(s) {}

    // 读属性 → 委托给 getter 的 FuncCall
    tjs_error PropGet(
        tjs_uint32 flag,
        const tjs_char* membername,
        tjs_uint32* hint,
        tTJSVariant* result,
        iTJSDispatch2* objthis) override
    {
        if (membername)
            return BaseT::PropGet(
                flag, membername, hint,
                result, objthis);
        return _getter->FuncCall(
            flag, nullptr, nullptr,
            result, 0, nullptr, objthis);
    }

    // 写属性 → 委托给 setter 的 FuncCall
    tjs_error PropSet(
        tjs_uint32 flag,
        const tjs_char* membername,
        tjs_uint32* hint,
        const tTJSVariant* param,
        iTJSDispatch2* objthis) override
    {
        if (membername)
            return BaseT::PropSet(
                flag, membername, hint,
                param, objthis);
        return _setter->FuncCall(
            flag, nullptr, nullptr,
            nullptr, 1,
            const_cast<tTJSVariant**>(&param),
            objthis);
    }

    ~ncbRawCallbackProperty() {
        delete _getter;
        delete _setter;
    }

protected:
    typename GETTER::Type* _getter;
    typename SETTER::Type* _setter;
};
```

### 5.5 完整示例：RawCallback 注册

```cpp
// my_dynamic.cpp — 展示 RawCallback 的典型用法
#include "ncbind.hpp"

#define NCB_MODULE_NAME TJS_W("dynamic.dll")

class DynamicHandler {
public:
    DynamicHandler() : _callCount(0) {}

    // 常规方法（供对比）
    int getCallCount() const { return _callCount; }

private:
    int _callCount;

    // RawCallback 静态函数——可变参数的处理
    static tjs_error TJS_INTF_METHOD
    formatString(
        tTJSVariant* result,
        tjs_int numparams,
        tTJSVariant** param,
        DynamicHandler* inst)
    {
        // 至少需要一个格式字符串参数
        if (numparams < 1)
            return TJS_E_BADPARAMCOUNT;

        // 第一个参数作为格式字符串
        tTJSString fmt =
            param[0]->AsStringNoAddRef();
        tTJSString output = fmt;

        // 后续参数动态替换占位符 {0}, {1}, ...
        for (tjs_int i = 1; i < numparams; i++) {
            tTJSString placeholder =
                TJS_W("{") +
                ttstr(i - 1) +
                TJS_W("}");
            tTJSString value =
                param[i]->AsStringNoAddRef();
            // 简单替换（实际项目中会更复杂）
            output = output.Replace(
                placeholder, value);
        }

        if (result)
            *result = output;

        if (inst)  // 非静态调用时递增计数
            inst->_callCount++;

        return TJS_S_OK;
    }

    // 友元声明，允许注册块访问私有成员
    friend class ncbRegistClass<DynamicHandler>;
};

NCB_REGISTER_CLASS(DynamicHandler) {
    NCB_CONSTRUCTOR(());

    // RawCallback 方法：可接受任意数量参数
    NCB_METHOD_RAW_CALLBACK(
        format, DynamicHandler::formatString, 0);

    // 常规只读属性（对比用）
    NCB_PROPERTY_RO(callCount, getCallCount);
}
```

对应的 TJS2 脚本：

```javascript

System.inform(s1);  // 输出: 你好 世界！
```

## 六、高级注册模式 —— Proxy、Bridge、SubClass、Variant

除了前面介绍的基本注册模式外，ncbind 还提供了四种高级模式，用于更复杂的绑定场景。

### 6.1 NCB_METHOD_PROXY —— 代理方法

代理模式允许将**自由函数**（非成员方法）注册为类的方法，并自动将原生实例作为第一个参数传入。这在以下场景中特别有用：

- 你想为一个类添加方法，但不想修改类本身的代码
- 方法逻辑涉及多个类，不适合放在任何一个类中
- 需要注册 C 风格的函数到 TJS2 类

宏定义（第 2270 行）：

```cpp
// ncbind.hpp 第 2270 行
#define NCB_METHOD_PROXY(name, method) \
    Method(TJS_W(# name), &method, Proxy)
```

`Proxy` 是 `ncbRegistClass` 的一个 `const` 成员，类型为 `ivtProxy<ivtNormal>`。它作为标签告诉 `Method()` 使用 `ivsProxy` 调用路径（参见第 1.5 节），该路径使用 `paramsFunctorWithInstance` 将实例指针注入为第一个参数。

使用示例：

```cpp
#include "ncbind.hpp"

class Sprite {
public:
    Sprite() : _x(0), _y(0) {}
    int getX() const { return _x; }
    int getY() const { return _y; }
    void setX(int x) { _x = x; }
    void setY(int y) { _y = y; }
private:
    int _x, _y;
};

// 自由函数——不是 Sprite 的成员方法
static double distanceTo(
    Sprite* self,     // 原生实例（自动注入）
    int targetX,      // TJS2 传入的第 1 个参数
    int targetY)      // TJS2 传入的第 2 个参数
{
    double dx = self->getX() - targetX;
    double dy = self->getY() - targetY;
    return std::sqrt(dx * dx + dy * dy);
}

// 自由函数：移动到指定位置
static void moveTo(
    Sprite* self, int x, int y)
{
    self->setX(x);
    self->setY(y);
}

NCB_REGISTER_CLASS(Sprite) {
    NCB_CONSTRUCTOR(());
    NCB_PROPERTY_RO(x, getX);
    NCB_PROPERTY_RO(y, getY);

    // 代理方法——TJS2 中调用 sprite.distanceTo(x,y)
    // C++ 中实际调用 distanceTo(sprite*, x, y)
    NCB_METHOD_PROXY(distanceTo, distanceTo);
    NCB_METHOD_PROXY(moveTo, moveTo);
}
```

```javascript
// test_proxy.tjs
var s = new Sprite();
s.moveTo(100, 200);
var dist = s.distanceTo(0, 0);
System.inform("距离原点: " + dist);
// 输出: 距离原点: 223.6...
```

### 6.2 Bridge —— 桥接模式

桥接模式比代理更进一步：它不仅获取原生实例，还通过一个类型转换函子将实例转换为**另一种类型**，然后调用该类型的方法。

ncbind 内部定义了三个桥接相关的内嵌结构（第 1684-1687 行）：

```cpp
// ncbind.hpp 第 1684-1687 行（ncbRegistClass 内部）
template <class B>
struct Bridge {};        // 标签：桥接 + 普通调用

template <class B>
struct ProxyBridge {};   // 标签：代理 + 桥接

template <class B>
struct BridgeProxy {};   // 标签：桥接 + 代理

// 获取桥接标签的调用模式
template <class METHOD>
static ivtBridge<METHOD> getBridgeType() { return {}; }
```

使用示例：假设你的 `WindowImpl` 内部包含一个 `Renderer` 对象，你想将 `Renderer` 的方法直接暴露给 TJS2：

```cpp
class WindowImpl {
public:
    Renderer* getRenderer() { return &_renderer; }
private:
    Renderer _renderer;
};

// 桥接转换器：WindowImpl* → Renderer*
struct ToRenderer {
    Renderer* operator()(WindowImpl* w) {
        return w->getRenderer();
    }
};

NCB_REGISTER_CLASS(WindowImpl) {
    NCB_CONSTRUCTOR(());

    // Bridge<ToRenderer> 告诉 ncbind：
    // 1. 从 objthis 获取 WindowImpl* 实例
    // 2. 通过 ToRenderer 转换为 Renderer*
    // 3. 在 Renderer* 上调用 drawRect 方法
    Method(TJS_W("drawRect"),
           &Renderer::drawRect,
           Bridge<ToRenderer>());
}
```

桥接的内部流程：

```
TJS2: window.drawRect(10, 20, 100, 50)
  ↓ FuncCall
nativeInstanceGetter → 获取 WindowImpl* inst
  ↓ bridgeInstanceGetter
ToRenderer()(inst) → 获取 Renderer* renderer
  ↓ doInvoke
renderer->drawRect(10, 20, 100, 50)
```

### 6.3 SubClass —— 子类注册

`SubClass()` 将一个 C++ 类注册为另一个 TJS2 类的**嵌套子类**（第 1791-1797 行）：

```cpp
// ncbind.hpp 第 1791-1797 行
template <class SubClassT>
void SubClass(const tjs_char* name) {
    ncbSubClassItem<SubClassT>::Setup();
    DoItem(
        name,
        ncbSubClassItem<SubClassT>::Create(
            _isRegist));
}
```

`ncbSubClassItem` 在父类的 TJS2 对象上创建一个嵌套的类定义。使用场景：当你想将 `Graphics.Point` 这样的层级结构暴露给 TJS2 时。

```cpp
class Point {
public:
    Point(int x, int y) : _x(x), _y(y) {}
    int getX() const { return _x; }
    int getY() const { return _y; }
private:
    int _x, _y;
};

// 先注册 Point 为独立类

    NCB_CONSTRUCTOR((int, int));
    NCB_PROPERTY_RO(x, getX);
    NCB_PROPERTY_RO(y, getY);
}

class Graphics {
public:
    // ... 绘图方法 ...
};

NCB_REGISTER_CLASS(Graphics) {
    NCB_CONSTRUCTOR(());
    // 将 Point 注册为 Graphics 的子类
    // TJS2 中可用 new Graphics.Point(10, 20)
    SubClass<Point>(TJS_W("Point"));
}
```

```javascript

// 输出: 坐标: 10, 20
```

### 6.4 Variant —— 常量注册

`Variant()` 将 C++ 常量值注册为 TJS2 类的静态成员（第 1800-1812 行）：

```cpp
// ncbind.hpp 第 1800-1812 行
template <typename T>
void Variant(const tjs_char* name, T value) {
    tTJSVariant var;
    // 使用类型转换器将 C++ 值 → tTJSVariant
    ncbTypeConvertor::SelectConvertorType<
        T, tTJSVariant>::Type conv;
    conv(var, value);
    _impl.RegistVariant(name, var);
}
```

这是注册常量（如错误码、标志位、枚举值）的最简方式。

### 6.5 四种高级模式对比

| 模式 | 宏/DSL | 用途 | 典型场景 |
|------|--------|------|---------|
| Proxy | `NCB_METHOD_PROXY` | 自由函数作为实例方法 | 扩展类功能、C API 封装 |
| Bridge | `Bridge<B>()` | 类型转换后调用方法 | 内嵌对象方法暴露、适配器模式 |

## 动手实践

下面通过一个完整的实践项目，综合运用本节学到的所有注册模式。

### 目标：实现一个 TJS2 可用的文本编辑器类

**需求：**
1. `TextEditor` 类，支持文本内容的读写
2. `content` 属性（可读可写）——通过 `NCB_PROPERTY` 注册
3. `lineCount` 属性（只读）——通过 `NCB_PROPERTY_RO` 注册
4. `append` 方法——通过 `NCB_METHOD` 注册
5. `search` 方法（自由函数）——通过 `NCB_METHOD_PROXY` 注册
6. `MAX_LENGTH` 常量——通过 `Variant()` 注册

**步骤 1：编写 C++ 类**

```cpp
// text_editor.cpp
#include "ncbind.hpp"
#include <algorithm>  // for std::count

#define NCB_MODULE_NAME TJS_W("texteditor.dll")

class TextEditor {
public:
    TextEditor() : _content(TJS_W("")) {}

    // ---- 属性的 getter/setter ----

    tTJSString getContent() const {
        return _content;
    }

    void setContent(const tjs_char* text) {
        _content = text ? text : TJS_W("");
    }

    int getLineCount() const {
        if (_content.IsEmpty()) return 0;
        const tjs_char* p = _content.c_str();
        int count = 1;  // 至少一行
        while (*p) {
            if (*p == TJS_W('\n')) count++;
            p++;
        }
        return count;
    }

    // ---- 方法 ----

    void append(const tjs_char* text) {
        if (text) _content += text;
    }

    const tTJSString& getContentRef() const {
        return _content;
    }

private:
    tTJSString _content;
};

// Proxy 函数——在内容中搜索子串，返回位置
static int searchInContent(
    TextEditor* self,         // 实例注入
    const tjs_char* needle)   // TJS2 参数
{
    if (!needle) return -1;
    const tjs_char* hay = self->getContentRef().c_str();
    const tjs_char* found = wcsstr(hay, needle);
    return found ? (int)(found - hay) : -1;
}

NCB_REGISTER_CLASS(TextEditor) {
    NCB_CONSTRUCTOR(());

    // 可读可写属性
    NCB_PROPERTY(content, getContent, setContent);

    // 只读属性
    NCB_PROPERTY_RO(lineCount, getLineCount);

    // 成员方法
    NCB_METHOD(append);

    // 代理方法
    NCB_METHOD_PROXY(search, searchInContent);

    // 常量
    Variant(TJS_W("MAX_LENGTH"), (int)65536);
}
```

**步骤 2：编写 TJS2 测试脚本**

```javascript

// 属性读写

// 输出: 行数: 3

// 方法调用

System.inform("行数: " + editor.lineCount);
// 输出: 行数: 4

// Proxy 方法

System.inform("'第三行'位置: " + pos);
```

## 常见错误与排查

### 错误 1：属性 getter/setter 签名不匹配

**症状：** 编译时出现长串模板错误，提示类型推导失败。

**原因：** `NCB_PROPERTY` 要求 getter 返回值类型与 setter 参数类型一致。

```cpp
// ❌ 错误：getter 返回 int，setter 接受 double
int getSize() const;
void setSize(double s);
NCB_PROPERTY(size, getSize, setSize);
// 编译错误：PropertyCommand 内部的 GetCommandT
// 和 SetCommandT 类型不一致

// ✅ 正确：类型一致
int getSize() const;
void setSize(int s);
NCB_PROPERTY(size, getSize, setSize);
```

### 错误 2：RawCallback 函数签名错误

**症状：** 编译通过但运行时崩溃，或参数值不正确。

**原因：** `ncbRawCallbackMethod<T>` 的回调签名必须严格匹配：

```cpp
// ❌ 错误：缺少 instance 参数
static tjs_error myCallback(
    tTJSVariant* result,
    tjs_int numparams,
    tTJSVariant** param)
{
    return TJS_S_OK;
}

// ✅ 正确：包含 instance 参数
static tjs_error myCallback(
    tTJSVariant* result,
    tjs_int numparams,
    tTJSVariant** param,
    MyClass* inst)  // 必须有！
{
    return TJS_S_OK;
}
```

### 错误 3：Proxy 方法的首参数类型不匹配

**症状：** 编译时模板推导错误，或运行时实例指针为错误值。

**原因：** `NCB_METHOD_PROXY` 的自由函数的第一个参数必须是 `ClassT*`（注册块对应的类指针类型）。

```cpp
// ❌ 错误：首参是 const 指针
static void doStuff(const MyClass* self, int x);
NCB_METHOD_PROXY(doStuff, doStuff);
// paramsFunctorWithInstance 存的是 MyClass*
// const 不匹配

// ✅ 正确：非 const 指针
static void doStuff(MyClass* self, int x);
NCB_METHOD_PROXY(doStuff, doStuff);
```

### 错误 4：忘记注册构造函数

**症状：** TJS2 中 `new MyClass()` 不报错但实例无原生对象，调用方法时返回 `TJS_E_NATIVECLASSCRASH`。

**原因：** 未使用 `NCB_CONSTRUCTOR` 或 `Factory()` 注册构造函数。TJS2 会创建 JavaScript 对象，但 `ncbInstanceAdaptor::GetNativeInstance` 返回 `nullptr`。

```cpp
// ❌ 遗漏
NCB_REGISTER_CLASS(MyClass) {
    NCB_METHOD(doStuff);
    // 没有 NCB_CONSTRUCTOR！
}

// ✅ 完整
NCB_REGISTER_CLASS(MyClass) {
    NCB_CONSTRUCTOR(());     // 必须注册！
    NCB_METHOD(doStuff);
}
```

## 本节小结

本节深入解析了 ncbind 框架中**属性注册、构造函数/工厂、原始回调和高级注册模式**四大核心机制。以下是关键要点：

- **NCB_METHOD 的本质**：宏展开为 `Method(TJS_W("name"), Class, &Class::name, Bridge)`，通过 `ncbRegistClass::Method()` 创建 `ncbNativeClassMethod<>` 分发器对象，该分发器在 `FuncCall` 中完成 TJS2 Variant ↔ C++ 类型的自动转换
- **NCB_PROPERTY 三种访问模式**：读写属性用 `NCB_PROPERTY(name, getter, setter)`，只读用 `NCB_PROPERTY_RO(name, getter)`（setter 位传入 `(int)0` 哨兵值触发 `Property()` 只读重载），只写用 `NCB_PROPERTY_WO(name, setter)`。底层 `ncbNativeClassProperty` 通过 `!_getter` / `!_setter` 检查返回 `TJS_E_ACCESSDENYED`（注意 KiriKiri 源码中的历史拼写）
- **NCB_CONSTRUCTOR vs Factory**：`NCB_CONSTRUCTOR` 直接映射到 C++ 构造函数参数列表，适用于标准构造场景；`Factory()` 接受自由函数签名 `tjs_error (*)(ClassT**, tjs_int, tTJSVariant**, iTJSDispatch2*)`，适用于需要自定义创建逻辑（如对象池、延迟初始化）的场景
- **RawCallback 机制**：`ncbRawCallbackMethod` 有两种特化——传统 TJS 签名（4 参数）和原生实例签名（多一个 `ClassT*`）。配合 `ncbRawCallbackPropertySelector<int>`（始终返回 AccessDenied）可实现单向原始属性
- **四种高级模式**：Proxy（代理方法注入实例指针为首参数）、Bridge（桥接到内部成员的方法）、SubClass（注册内部类到 TJS 命名空间）、Variant（注册常量值）。各有适用场景，选择依据见对比表
- **注册流程的全局视角**：所有注册操作最终通过 `ncbRegistClass` 的成员方法将命令对象压入命令队列，在 `ncbClassInfo::TJS2NativeClass::Register()` 阶段批量执行，完成 TJS2 类的方法/属性/构造函数绑定

### 题目 1：为 AudioPlayer 类编写完整的属性注册

给定以下 C++ 类，请写出完整的 `NCB_REGISTER_CLASS` 注册块，要求：
- `volume` 为读写属性（getter: `getVolume`，setter: `setVolume`）
- `duration` 为只读属性（getter: `getDuration`）
- `muted` 为只写属性（setter: `setMuted`）
- `play` 和 `pause` 为普通方法
- 注册构造函数，接受一个 `ttstr` 参数（文件路径）

```cpp
class AudioPlayer {
public:
    AudioPlayer(const ttstr& filePath);
    
    int getVolume() const;
    void setVolume(int vol);
    
    double getDuration() const;
    
    void setMuted(bool m);
    
    void play();
    void pause();
};
```

<details>
<summary>查看答案</summary>

```cpp
// 注册块——注意宏的参数格式
NCB_REGISTER_CLASS(AudioPlayer) {
    // 1. 构造函数：接受一个 ttstr 参数
    NCB_CONSTRUCTOR((const ttstr&));
    
    // 2. 读写属性：volume
    //    展开为 Property(TJS_W("volume"),
    //            &AudioPlayer::getVolume,
    //            &AudioPlayer::setVolume, Bridge)
    NCB_PROPERTY(volume, getVolume, setVolume);
    
    // 3. 只读属性：duration
    //    展开为 Property(TJS_W("duration"),
    //            &AudioPlayer::getDuration,
    //            (int)0, Bridge)
    //    (int)0 哨兵值触发只读重载，setter 为 nullptr
    NCB_PROPERTY_RO(duration, getDuration);
    
    // 4. 只写属性：muted
    //    展开为 Property(TJS_W("muted"),
    //            (int)0,
    //            &AudioPlayer::setMuted, Bridge)
    //    (int)0 哨兵值触发只写重载，getter 为 nullptr
    NCB_PROPERTY_WO(muted, setMuted);
    
    // 5. 普通方法
    NCB_METHOD(play);
    NCB_METHOD(pause);
}
```

**关键点：**
- `NCB_PROPERTY_RO` 的 setter 位传入 `(int)0`，底层 `ncbNativeClassProperty` 的 `PropSet` 返回 `TJS_E_ACCESSDENYED`
- `NCB_PROPERTY_WO` 的 getter 位传入 `(int)0`，底层 `PropGet` 返回 `TJS_E_ACCESSDENYED`
- `NCB_CONSTRUCTOR` 的参数是**括号包裹的类型列表**，不是变量名

```javascript

System.inform(player.volume); // 调用 getVolume()，输出 80
System.inform(player.duration); // 调用 getDuration()
// player.duration = 10.0;    // 报错：TJS_E_ACCESSDENYED
player.muted = true;          // 调用 setMuted(true)
// var m = player.muted;      // 报错：TJS_E_ACCESSDENYED
player.play();
```

### 题目 2：解释 `(int)0` 哨兵值的完整流程

请详细说明：当 `NCB_PROPERTY_RO(duration, getDuration)` 宏展开后，`(int)0` 是如何一步步被处理，最终导致对 `duration` 属性的写操作返回 `TJS_E_ACCESSDENYED` 的？请列出涉及的每个函数/类。

<details>
<summary>查看答案</summary>

完整流程如下（共 5 步）：

**第 1 步：宏展开**

```cpp
NCB_PROPERTY_RO(duration, getDuration)
// 展开为：
NCB_PROPERTY(duration, getDuration, (int)0)
// 再展开为：
Property(TJS_W("duration"), Class, &Class::getDuration, (int)0, Bridge)
```

**第 2 步：`ncbRegistClass::Property()` 重载解析**

`Property()` 有三个重载（ncbind.hpp 第 1725-1728 行）：

```cpp
// 重载 1：完整读写（getter 和 setter 都是成员函数指针）
template <typename GF, typename SF>
void Property(... GF getter, SF setter, ...);

// 重载 2：只读（setter 参数类型为 int）
template <typename GF>
void Property(... GF getter, int, ...);  // ← (int)0 匹配这个！

// 重载 3：只写（getter 参数类型为 int）
template <typename SF>
void Property(... int, SF setter, ...);
```

`(int)0` 是 `int` 类型，所以编译器选择**重载 2**（只读版本）。

**第 3 步：只读重载的实现**

只读重载内部将 setter 指针设为 `nullptr`：

```cpp
template <typename GF>
void Property(const tjs_char* name, GF getter, int, BridgeT) {
    // 创建 PropertyCommand，setter 模板参数为 void(*)()（空函数类型）
    // setter 实际值为 nullptr
    _addCommand(new PropertyCommand<GF, NullSetter>(name, getter, nullptr));
}
```

**第 4 步：`ncbNativeClassProperty` 构造**

`PropertyCommand::Exec()` 创建 `ncbNativeClassProperty` 对象：

```cpp
ncbNativeClassProperty<ClassT, GetterWrap, SetterWrap> prop(getter, nullptr);
// 内部存储：
//   _getter = GetterWrap(getter)  // 有效
//   _setter = nullptr              // 无效
```

**第 5 步：PropSet 被调用时**

当 TJS2 脚本尝试写入 `obj.duration = 10.0` 时：

```cpp
tjs_error PropSet(tjs_uint32 flag, ...) {
    if (!_setter) {
        // setter 为 nullptr，直接返回访问拒绝
        return TJS_E_ACCESSDENYED;
    }
    return _setter->FuncCall(...);
}
```

**流程总结图：**

```
NCB_PROPERTY_RO(duration, getDuration)
    │
    ├─ 宏展开 → Property(..., &Class::getDuration, (int)0, ...)
    │
    ├─ 重载解析 → Property(... GF getter, int, ...) [只读版本]
    │
    ├─ 创建 PropertyCommand → setter = nullptr
    │
    ├─ Exec() → ncbNativeClassProperty(_getter=有效, _setter=nullptr)
    │
    └─ PropSet() → !_setter → return TJS_E_ACCESSDENYED
```

</details>

### 题目 3：编写一个 RawCallback 实现可变参数的 format 方法

要求：为 `Logger` 类实现一个 RawCallback 方法 `format`，该方法接受第一个参数作为格式字符串（`ttstr` 类型），后续参数作为格式化参数。方法需要：
1. 检查至少有 1 个参数（格式字符串），否则返回错误
2. 将所有参数拼接成 `"[格式字符串] arg1, arg2, ..."` 的形式
3. 通过 `result` 返回拼接后的字符串
4. 写出完整的注册块

<details>
<summary>查看答案</summary>

```cpp
#include <ncbind.hpp>
#include <string>

class Logger {
public:
    Logger(const ttstr& name) : name_(name) {}
    
    // RawCallback 签名：必须包含实例指针作为最后一个参数
    static tjs_error TJS_INTF_METHOD formatImpl(
        tTJSVariant* result,     // 返回值
        tjs_int numparams,       // 参数个数
        tTJSVariant** param,     // 参数数组
        Logger* inst)            // 原生实例指针（ncbind 自动注入）
    {
        // 检查至少 1 个参数（格式字符串）
        if (numparams < 1) {
            return TJS_E_BADPARAMCOUNT;
        }
        
        // 获取格式字符串（第一个参数）
        ttstr fmt = param[0]->AsStringNoAddRef();
        
        // 构建输出：[Logger名称] 格式字符串 | 参数列表
        ttstr output = TJS_W("[");
        output += inst->name_;   // 访问实例成员
        output += TJS_W("] ");
        output += fmt;
        
        // 拼接后续参数
        if (numparams > 1) {
            output += TJS_W(" | args: ");
            for (tjs_int i = 1; i < numparams; i++) {
                if (i > 1) {
                    output += TJS_W(", ");
                }
                // 将每个参数转为字符串
                output += param[i]->AsStringNoAddRef();
            }
        }
        
        // 设置返回值
        if (result) {
            *result = tTJSVariant(output);
        }
        
        return TJS_S_OK;
    }
    
    // 普通方法用于对比
    const ttstr& getName() const { return name_; }

private:
    ttstr name_;  // Logger 名称
};

// 注册块
NCB_REGISTER_CLASS(Logger) {
    // 构造函数：接受一个 ttstr 参数
    NCB_CONSTRUCTOR((const ttstr&));
    
    // RawCallback 方法：format
    // 使用 NCB_METHOD_RAW_CALLBACK 宏注册
    // 签名必须匹配 ncbRawCallbackMethod 的原生实例特化
    NCB_METHOD_RAW_CALLBACK(format, formatImpl, 0);
    
    // 普通只读属性用于对比
    NCB_PROPERTY_RO(name, getName);
}
```

TJS2 脚本测试：

```javascript
var log = new Logger("MyApp");

// 无额外参数
var s1 = log.format("启动完成");
// s1 == "[MyApp] 启动完成"

// 带额外参数
var s2 = log.format("加载文件", "data.xp3", 1024, true);
// s2 == "[MyApp] 加载文件 | args: data.xp3, 1024, true"

// 参数不足时（0 个参数）
// log.format();  // 返回 TJS_E_BADPARAMCOUNT
```

**关键点：**
- RawCallback 的函数签名最后一个参数是 `Logger*`（实例指针），由 `ncbRawCallbackMethod` 的原生实例特化自动注入
- `numparams` 不包含实例指针，只计算 TJS2 传入的显式参数
- `param[0]` 到 `param[numparams-1]` 是 TJS2 传入的参数
- 必须检查 `result` 非空再赋值，因为调用者可能不需要返回值（如 `log.format(...)` 不接收返回值时）
- 返回值必须是 `tjs_error` 类型，成功返回 `TJS_S_OK`

</details>
