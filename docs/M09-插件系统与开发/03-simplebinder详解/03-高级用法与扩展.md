## 一、UNROLL_FOREACH 宏展开机制深度解析

### 1.1 问题背景：为什么需要宏展开？

simplebinder 面临一个核心问题：如何为不同参数数量的函数生成对应的调用代码？

一个接受 2 个参数的函数需要这样的调用代码：

```cpp
// 2 参数版本
return func(result, param[0], param[1]);
```

而一个接受 5 个参数的函数需要：

```cpp
// 5 参数版本
return func(result, param[0], param[1], param[2], param[3], param[4]);
```

在 C++11/14 中（simplebinder 的设计年代），没有折叠表达式（Fold Expression，C++17 才引入的特性，允许对参数包进行统一的展开操作）。解决方案有两种：

1. **模板递归**——ncbind 的方式。通过 `paramsFunctor<N>` 递归展开，编译器逐层生成代码
2. **预处理器宏**——simplebinder 的方式。用宏一次性生成 0 到 8 参数的所有特化版本

simplebinder 选择了第二种，使用 `UNROLL_FOREACH` 宏系统。

### 1.2 UNROLL 宏族解析

simplebinder 定义了四组 UNROLL 宏，每组 9 个变体（0-8 参数），分别对应不同的参数列表格式：

```
┌──────────────────────────────────────────────────────────┐
│                    UNROLL 宏族总览                         │
├────────────┬─────────────────────────────────────────────┤
│ UNROLL_AM  │ (A, M(0), M(1), ...)                        │
│            │ 一个固定前缀 + N 个展开参数                  │
├────────────┼─────────────────────────────────────────────┤
│ UNROLL_ABM │ (A, B, M(0), M(1), ...)                     │
│            │ 两个固定前缀 + N 个展开参数                  │
├────────────┼─────────────────────────────────────────────┤
│ UNROLL_AMYZ│ (A, M(0), ..., Y, Z)                        │
│            │ 一个前缀 + N 个展开参数 + 两个后缀           │
├────────────┼─────────────────────────────────────────────┤
│ UNROLL_ABMYZ│(A, B, M(0), ..., Y, Z)                     │
│            │ 两个前缀 + N 个展开参数 + 两个后缀           │
└────────────┴─────────────────────────────────────────────┘
```

为什么要分四组？因为 simplebinder 中不同类型的调用器（Invoker，指负责实际调用 C++ 函数的模板类）需要不同的参数列表格式。例如：

- **MethodInvoker**：需要"返回值类型 + N 个参数类型"→ 使用 `UNROLL_AM`（A=返回值，M(i)=第 i 个参数类型）
- **FactoryInvoker**：需要"类类型 + N 个参数类型"→ 同样使用 `UNROLL_AM`
- **PropertyCallback**（属性回调，负责在 TJS2 属性读写时调用 C++ getter/setter 函数）：需要"类类型 + 属性类型 + 函数指针类型"→ 使用 `UNROLL_ABM` 的 0 参数特化

### 1.3 展开过程详解

我们以一个具体的 `UNROLL_FOREACH` 调用为例，跟踪它的完整展开过程。以下是 simplebinder.hpp 第 86-93 行中 `MethodInvokerBase` 的定义宏：

```cpp
// 源码位置：cpp/plugins/simplebinder/simplebinder.hpp 第 86-93 行
// 定义宏：用一个 UNROLL_AM 变体生成一个 MethodInvokerBase 模板特化
#define DEF_SMI(N)                                                      \
    template <bool Static, typename R, UNROLL_AM##N(typename, typename)> \
    struct MethodInvokerBase<Static, R(UNROLL_AM##N(, ))> {             \
        /* ... 省略函数体，后面详细讲 ... */                              \
    };

// 关键一步：用 UNROLL_FOREACH 一次性展开 0-8 共 9 个特化
UNROLL_FOREACH(DEF_SMI)
```

`UNROLL_FOREACH(DEF_SMI)` 展开后等价于：

```cpp
DEF_SMI(0)  // MethodInvokerBase<Static, R()>           — 0 参数
DEF_SMI(1)  // MethodInvokerBase<Static, R(A1)>         — 1 参数
DEF_SMI(2)  // MethodInvokerBase<Static, R(A1, A2)>     — 2 参数
DEF_SMI(3)  // MethodInvokerBase<Static, R(A1, A2, A3)> — 3 参数
DEF_SMI(4)  // ...
DEF_SMI(5)
DEF_SMI(6)
DEF_SMI(7)
DEF_SMI(8)  // MethodInvokerBase<Static, R(A1,...,A8)>  — 8 参数
```

每个 `DEF_SMI(N)` 内部又会展开 `UNROLL_AM##N`。以 `DEF_SMI(3)` 为例：

```cpp
// DEF_SMI(3) 展开第一步：
template <bool Static, typename R, UNROLL_AM3(typename, typename)>
struct MethodInvokerBase<Static, R(UNROLL_AM3(, ))> { ... };

// UNROLL_AM3(typename, typename) 展开为：
//   typename A, typename A1, typename A2, typename A3
// （A 是固定前缀，A1-A3 是 3 个参数类型名）

// UNROLL_AM3(, ) 展开为：
//   , A1, A2, A3
// （仅逗号分隔的参数类型，用在函数签名 R(A1, A2, A3) 中）

// 最终结果：
template <bool Static, typename R, typename A, typename A1, typename A2, typename A3>
struct MethodInvokerBase<Static, R(A1, A2, A3)> {
    // 这个特化处理 3 参数的函数
};
```

> **关键理解**：`UNROLL_AM` 的第一个参数 `A` 在模板参数列表中用作"前缀类型"，而 `M(0)..M(N)` 对应 `A1..AN`。在函数签名位置，`A` 被省略，只留 `A1..AN`，因为函数签名只需要参数类型，不需要额外的前缀。

下面是更复杂的 `UNROLL_ABMYZ` 变体的展开示例，它出现在类成员方法绑定（`DEF_SCM`）中：

```cpp
// 源码位置：cpp/plugins/simplebinder/simplebinder.hpp 第 173-197 行
// DEF_SCM 宏：为类成员函数生成 MethodInvoker 特化
#define DEF_SCM(N)                                                           \
    template <bool Static, typename C, typename R,                           \
              UNROLL_ABMYZ##N(typename, typename, typename, typename)>       \
    struct MethodInvoker<Static, C, R(C::*)(UNROLL_ABMYZ##N(,,,))> {        \
        typedef R(C::*MethodType)(UNROLL_ABMYZ##N(,,,));                     \
        static bool Invoke(MethodType func, tTJSVariant *result,            \
                           tjs_int numparams, tTJSVariant **param,          \
                           iTJSDispatch2 *objthis) {                        \
            C *self = ClassInstanceResolver<C>::GetInstance(objthis);        \
            if (!self) return false;                                         \
            /* 调用成员函数，参数逐个从 TJS2 variant 转换 */                  \
            MethodInvokerBase<Static, R(UNROLL_ABMYZ##N(,,,))>::Invoke(     \
                [self, func](auto&&... args) {                              \
                    return (self->*func)(std::forward<decltype(args)>(args)...); \
                }, result, numparams, param);                               \
            return true;                                                     \
        }                                                                    \
    };
UNROLL_FOREACH(DEF_SCM)
```

以 `DEF_SCM(2)` 展开为例：

```
UNROLL_ABMYZ2(typename, typename, typename, typename)
→ typename A, typename B, typename A1, typename A2, typename Y, typename Z
   ↑前缀A   ↑前缀B   ↑参数1    ↑参数2    ↑后缀Y   ↑后缀Z

UNROLL_ABMYZ2(,,,)
→ , , A1, A2, ,
  在函数签名中变为 C::*(A1, A2)
```

> **注意**：实际的 `DEF_SCM` 展开比上面的伪代码更复杂，因为它还需要处理 `const` 成员函数。simplebinder 通过 `DEF_SCM` 和一个单独的 `DEF_SCS`（静态函数绑定到类方法的变体）分别处理这两种情况。

### 1.4 代码生成量分析

simplebinder.hpp 中共有 **12 处** `UNROLL_FOREACH` 调用，每处生成 9 个特化（0-8 参数）。我们来详细统计：

```
┌─────────────────────────────────────┬──────────┬────────────────────────┐
│          UNROLL_FOREACH 调用         │ 源码行号  │      生成内容           │
├─────────────────────────────────────┼──────────┼────────────────────────┤
│ DEF_SMI (MethodInvokerBase)         │  86-93   │ 9 个基础调用器特化      │
│ DEF_SCF (FactoryInvoker)            │ 100-122  │ 9 个工厂函数特化        │
│ DEF_SMI_S (static MethodInvoker)    │ 131-142  │ 9 个静态函数绑定特化    │
│ DEF_SMI_C (objthis MethodInvoker)   │ 145-160  │ 9 个带上下文函数特化    │
│ DEF_SCM (class member)              │ 173-197  │ 9 个类成员方法特化      │
│ DEF_SCS (class static-style)        │ 200-222  │ 9 个类静态风格特化      │
│ DEF_SPCB (PropertyCallback setter)  │ 612-634  │ 9 个属性写回调特化      │
│ DEF_SPCB (PropertyCallback getter)  │ 636-652  │ 9 个属性读回调特化      │
│ DEF_SCCB (ClassEntry setter)        │ 830-852  │ 9 个类属性写回调特化    │
│ DEF_SCCB (ClassEntry getter)        │ 854-870  │ 9 个类属性读回调特化    │
│ DEF_SCEI (ClassEntryCallback)       │ 872-910  │ 9 个类方法回调特化      │
│ DEF_SCEI (ClassEntryCallback const) │ 912-930  │ 9 个 const 方法回调特化 │
├─────────────────────────────────────┼──────────┼────────────────────────┤
│ 合计                                 │          │ 108 个模板特化          │
└─────────────────────────────────────┴──────────┴────────────────────────┘
```

**108 个模板特化**——这就是 simplebinder 用 1237 行源码实现完整 TJS2 绑定能力的秘密。每个特化都是编译时生成的，运行时零开销（Zero-cost Abstraction，指编译期完全确定的模板代码在运行时不产生额外性能开销）。

与 ncbind 的对比：

| 维度 | simplebinder (UNROLL) | ncbind (递归模板) |
|------|----------------------|------------------|
| 展开方式 | 预处理器一次性生成 | 编译器逐层递归展开 |
| 代码膨胀 | 固定 108 个特化，可预测 | 每个绑定函数生成递归链，随绑定数量增长 |
| 编译时间 | 较快（预处理器阶段完成） | 较慢（深层模板实例化） |
| 最大参数数 | 硬编码 8 个 | 同样硬编码 8 个（`NCB_INNER_DEF_PARAMS_MAX`） |
| 可读性 | 宏展开难以调试 | 模板递归同样难以调试 |
| 扩展性 | 改宏定义可扩展到 16 参数 | 改递归终止条件可扩展 |

> **实用建议**：如果你需要支持超过 8 个参数的函数（这种情况极少见），可以在 simplebinder.hpp 顶部修改 `UNROLL_FOREACH` 的定义，添加 `X(9)`、`X(10)` 等。但超过 8 个参数的函数通常意味着设计有问题——应该考虑用结构体（struct）封装参数。

## 二、ClassStore 与 FactoryInvoker——真正的类注册

### 2.1 一个常见的误解

在前一节的对比分析中，我们提到 simplebinder 的 `BindUtil::Class("ClassName")` 看似只能创建"命名空间对象"（Namespace Object，一个仅作为容器存在的 TJS2 对象，挂载方法和属性但不能被 `new` 创建实例）。这个说法对于**不传工厂函数**的 `Class()` 调用确实成立——但 simplebinder 还有一个更强大的 `Class()` 重载：

```cpp
// 源码位置：cpp/plugins/simplebinder/simplebinder.hpp 第 1139-1171 行
// 带工厂函数的 Class() 重载——真正的类注册
template <typename C, typename F>
BindUtil& Class(const tjs_char *key, F factory) {
    // 创建完整的 TJS2 原生类（tTJSNativeClassForPlugin）
    auto *cls = ClassStore<C>::CreateClassObject(key, factory);
    // 将当前 store 切换到类对象上，后续的 Function/Property 挂到类上
    _store = rebase(cls);
    return *this;
}
```

这个带 `factory` 参数的重载，能创建一个支持 `new ClassName()` 的完整 TJS2 原生类。我们来深入分析它的实现链条。

### 2.2 ClassStore 内部架构

`ClassStore<C>` 是 simplebinder 中最复杂的组件，它管理 C++ 类 `C` 在 TJS2 中的完整生命周期。其核心结构如下：

```
┌─────────────────────────────────────────────────────────────┐
│                    ClassStore<C> 架构                         │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌──────────────────────┐    ┌───────────────────────────┐  │
│  │  _Singleton (static) │    │  ClassEntryInterface<C>   │  │
│  │  ─────────────────── │    │  ─────────────────────── │  │
│  │  指向唯一的           │───>│  createNew(): 调用工厂    │  │
│  │  ClassStore<C> 实例   │    │  invalidate(): 析构 C++  │  │
│  │                      │    │  实例                     │  │
│  └──────────────────────┘    └───────────────────────────┘  │
│           │                            │                     │
│           ▼                            ▼                     │
│  ┌──────────────────────┐    ┌───────────────────────────┐  │
│  │  CreateClassObject() │    │  InstanceWrapper<C>       │  │
│  │  ─────────────────── │    │  ─────────────────────── │  │
│  │  创建                │    │  实现 tTJSNativeInstance  │  │
│  │  tTJSNativeClassFor  │    │  Construct(): 创建 C++   │  │
│  │  Plugin 对象          │    │  Invalidate(): 销毁 C++ │  │
│  └──────────────────────┘    └───────────────────────────┘  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**关键设计：Singleton 模式**。`ClassStore<C>` 使用一个静态指针 `_Singleton` 确保每个 C++ 类型只能注册一次：

```cpp
// 源码位置：cpp/plugins/simplebinder/simplebinder.hpp 第 990-1005 行
template <typename C>
class ClassStore : public StoreUtil {
    static ClassStore<C> *_Singleton;  // 静态单例指针

public:
    template <typename F>
    static iTJSDispatch2* CreateClassObject(const tjs_char *key, F factory) {
        if (_Singleton) {
            // 已经注册过——打印错误，返回已有的类对象
            TVPAddLog(TJS_W("Class already created"));
            return _Singleton->GetObject();
        }
        // 首次注册：创建新的 ClassStore 实例
        _Singleton = new ClassStore<C>();
        // 创建 tTJSNativeClassForPlugin——TJS2 的原生类包装
        auto *cls = new tTJSNativeClassForPlugin(key);
        // 注册原生类 ID（NativeClassID），TJS2 用它识别类类型
        _Singleton->_classId = TJSRegisterNativeClass(key);
        // 设置构造器和析构器
        InstanceWrapper<C>::SetupClassObject(cls, _Singleton, factory);
        return cls;
    }
};

// 静态成员初始化
template <typename C>
ClassStore<C>* ClassStore<C>::_Singleton = nullptr;
```

> **为什么是 Singleton？** TJS2 的原生类系统要求每个 C++ 类型有唯一的 NativeClassID（原生类标识符，TJS2 用一个整数来识别不同的 C++ 类型）。如果同一个 C++ 类型注册两次，会产生 ID 冲突，导致实例解析失败。Singleton 模式从源头防止了这个问题。

### 2.3 FactoryInvoker——构造函数的桥梁

当 TJS2 脚本执行 `var obj = new MyClass(arg1, arg2)` 时，需要一个机制把 TJS2 参数转换为 C++ 构造函数调用。这就是 `FactoryInvoker` 的职责：

```cpp
// 源码位置：cpp/plugins/simplebinder/simplebinder.hpp 第 100-122 行
// 以 2 参数版本为例（由 UNROLL_FOREACH(DEF_SCF) 生成）
template <typename C, typename A1, typename A2>
struct FactoryInvoker<C(A1, A2)> {
    // FactoryType：工厂函数类型——接受 2 个参数，返回 C*
    typedef C* (*FactoryType)(A1, A2);

    static C* Create(FactoryType factory,
                     tjs_int numparams, tTJSVariant **param) {
        // 从 TJS2 参数数组中逐个转换类型
        // param[0] -> A1 类型，param[1] -> A2 类型
        return factory(
            static_cast<A1>(*param[0]),   // TJS2 variant 转 C++ A1 类型
            static_cast<A2>(*param[1])    // TJS2 variant 转 C++ A2 类型
        );
    }
};
```

使用方式——定义一个工厂函数，然后注册：

```cpp
// 示例：注册一个带构造参数的类
class MySprite {
    int _x, _y;
    tjs_char _name[64];
public:
    MySprite(int x, int y) : _x(x), _y(y) {
        _name[0] = 0;
    }
    ~MySprite() = default;

    int getX() const { return _x; }
    int getY() const { return _y; }
    void moveTo(int x, int y) { _x = x; _y = y; }
};

// 工厂函数：接受与构造函数相同的参数，返回 new 出来的对象
static MySprite* CreateMySprite(int x, int y) {
    return new MySprite(x, y);
}

// 注册——BindUtil::Class 会自动推导 FactoryInvoker<MySprite(int, int)>
SimpleBinder::BindUtil(TJS_W("MyPlugin"))
    .Class<MySprite>(TJS_W("MySprite"), &CreateMySprite)
        .Function(TJS_W("getX"), &MySprite::getX)
        .Function(TJS_W("getY"), &MySprite::getY)
        .Function(TJS_W("moveTo"), &MySprite::moveTo);

// TJS2 脚本中就可以这样用：
// var sprite = new MySprite(100, 200);
// sprite.moveTo(300, 400);
// System.inform(sprite.getX()); // 输出 300
```

> **注意工厂函数的签名**：返回值必须是 `C*`（裸指针），参数类型必须是 simplebinder 能从 `tTJSVariant` 转换的基本类型（`int`、`double`、`tjs_int`、`ttstr` 等）。不支持返回智能指针或引用。

### 2.4 InstanceWrapper——生命周期管理

`InstanceWrapper<C>` 实现了 TJS2 的 `tTJSNativeInstance` 接口，是 C++ 对象与 TJS2 对象之间的生命周期桥梁：

```cpp
// 源码位置：cpp/plugins/simplebinder/simplebinder.hpp 第 935-980 行
template <typename C>
class InstanceWrapper : public tTJSNativeInstance {
    C *_instance;  // 持有的 C++ 对象指针

public:
    InstanceWrapper() : _instance(nullptr) {}
    ~InstanceWrapper() override { delete _instance; }

    // TJS2 调用：new ClassName() 时触发
    tjs_error TJS_INTF_METHOD Construct(
        tjs_int numparams, tTJSVariant **param,
        iTJSDispatch2 *tjs_obj) override {
        // 通过 ClassStore 的接口调用工厂函数创建 C++ 实例
        _instance = ClassStore<C>::_Singleton->createNew(numparams, param);
        if (!_instance) return TJS_E_FAIL;
        return TJS_S_OK;
    }

    // TJS2 调用：对象被垃圾回收时触发
    void TJS_INTF_METHOD Invalidate() override {
        // 调用 ClassStore 的 invalidate（默认实现是 delete _instance）
        ClassStore<C>::_Singleton->invalidate(_instance);
        _instance = nullptr;
    }

    // 获取持有的 C++ 实例指针
    C* GetInstance() { return _instance; }

    // 设置类对象的构造器和析构器
    template <typename F>
    static void SetupClassObject(
        tTJSNativeClassForPlugin *cls,
        ClassStore<C> *store, F factory) {
        // 注册 NativeInstance 创建回调
        cls->RegisterNCM(
            TJS_W(""), /* 空名 = 构造函数 */
            TJS_STATICMETHOD(/* 构造函数处理逻辑 */));
        // 注册析构（finalize）回调
        cls->RegisterNCM(
            TJS_W("finalize"),
            TJS_STATICMETHOD(/* 析构函数处理逻辑 */));
    }
};
```

完整的对象创建流程：

```
TJS2 脚本: var obj = new MySprite(100, 200)
    │
    ▼
tTJSNativeClassForPlugin::CreateNew()
    │  创建 InstanceWrapper<MySprite>
    ▼
InstanceWrapper<MySprite>::Construct(2, [100, 200])
    │  调用 ClassStore<MySprite>::_Singleton->createNew()
    ▼
ClassEntryInterface<MySprite>::createNew()
    │  内部调用 FactoryInvoker<MySprite(int,int)>::Create()
    ▼
FactoryInvoker::Create(&CreateMySprite, 2, params)
    │  static_cast 转换参数，调用工厂函数
    ▼
CreateMySprite(100, 200)
    │  return new MySprite(100, 200)
    ▼
C++ 对象创建完成，指针存储在 InstanceWrapper::_instance 中
```

对象销毁流程是反向的：

```
TJS2 垃圾回收触发
    │
    ▼
InstanceWrapper<MySprite>::Invalidate()
    │  调用 ClassStore::invalidate(_instance)
    ▼
delete _instance  （默认行为：直接 delete）
    │
    ▼
~MySprite() 析构函数执行
```

### 2.5 与 ncbind 类注册的对比

| 维度 | simplebinder (ClassStore) | ncbind (NCB_REGISTER_CLASS) |
|------|--------------------------|----------------------------|
| 注册方式 | `BindUtil::Class<C>(name, factory)` | `NCB_REGISTER_CLASS(C)` 宏 |
| 构造函数 | 需要手写工厂函数 | 自动推导构造函数签名 |
| 析构管理 | 默认 `delete`，可自定义 `invalidate` | 自动调用析构函数 |
| 继承支持 | 无内置支持——需手动处理基类 | `NCB_REGISTER_SUBCLASS` 支持继承链 |
| 多构造函数 | 只能注册一个工厂函数 | 可通过 `Constructor` 指定不同签名 |
| Singleton 限制 | 每个 C++ 类型只能注册一次 | 无此限制（用不同名注册同一类型） |
| 方法/属性绑定 | `.Function()` / `.Property()` 链式调用 | `NCB_METHOD` / `NCB_PROPERTY` 宏 |

**结论**：simplebinder 的类注册能力**足够用于简单类**，但缺乏 ncbind 的继承支持和多构造函数能力。这也解释了为什么 KrKr2 项目中所有实际插件都选择了 ncbind——它们大多需要继承 TJS2 基类（如 `tTJSNativeInstance`）或支持多种构造方式。

## 三、ExceptionTrapper 异常捕获系统

### 3.1 C++ 异常与 TJS2 的鸿沟

C++ 和 TJS2 使用完全不同的异常机制。C++ 通过 `throw`/`catch` 抛出和捕获异常对象（如 `std::exception`），而 TJS2 使用自己的错误处理系统（`TVPThrowExceptionMessage`，将错误消息传递给 TJS2 脚本引擎的异常处理机制）。如果一个 C++ 异常从绑定函数中逃逸到 TJS2 层，整个程序会崩溃——因为 TJS2 虚拟机不知道如何处理 C++ 异常。

simplebinder 通过 `ExceptionTrapper` 系统解决这个问题：在每个绑定调用的边界捕获 C++ 异常，转换为 TJS2 错误消息。

### 3.2 ExceptionRethrowTrap 实现

```cpp
// 源码位置：cpp/plugins/simplebinder/simplebinder.hpp 第 304-380 行
class ExceptionRethrowTrap {
    // 注意：使用 static 变量存储错误消息——线程不安全！
    static ttstr error;  // ttstr = TJS2 的宽字符串类型

public:
    // 尝试执行一个函数调用，捕获所有 C++ 异常
    template <typename Func>
    static bool Try(Func&& func) {
        try {
            func();      // 执行实际的 C++ 函数调用
            return true; // 成功——没有异常
        }
        catch (const std::exception& e) {
            // 标准 C++ 异常：转换 what() 消息
            error = ttstr(e.what());
        }
        catch (const ttstr& msg) {
            // TJS2 字符串形式的异常
            error = msg;
        }
        catch (const tjs_char* msg) {
            // C 风格宽字符串异常
            error = ttstr(msg);
        }
        catch (...) {
            // 未知异常类型
            error = TJS_W("Unknown C++ exception");
        }
        // 到这里说明捕获了异常——转发给 TJS2
        TVPThrowExceptionMessage(error.c_str());
        return false;  // 实际上不会执行到这里，因为 TVPThrowExceptionMessage 会抛出 TJS2 异常
    }
};

// 静态成员定义
ttstr ExceptionRethrowTrap::error;
```

### 3.3 线程安全性分析

上面的代码有一个严重的隐患——`static ttstr error` 是一个**静态局部变量**：

```
线程 A:                          线程 B:
    catch(...) {                     catch(...) {
        error = "Error A";               error = "Error B";  ← 覆盖！
    }                                }
    TVPThrowExceptionMessage(        TVPThrowExceptionMessage(
        error.c_str()                    error.c_str()
    );                               );
    // 可能报告 "Error B"！           // 正确报告 "Error B"
```

**影响范围**：在单线程环境（KrKr2 的主线程 TJS2 执行通常是单线程的）下，这个问题不会触发。但如果将来 TJS2 支持多线程脚本执行，或者有其他线程调用 TJS2 API（如音频回调线程），就会出现竞争条件（Race Condition，多个线程同时读写共享数据导致结果不可预测的并发 bug）。

**修复方案**（如果需要）：

```cpp
// 方案 1：使用 thread_local（C++11 起支持）
static thread_local ttstr error;

// 方案 2：在栈上分配（完全避免共享）
template <typename Func>
static bool Try(Func&& func) {
    try {
        func();
        return true;
    } catch (const std::exception& e) {
        ttstr localError(e.what());  // 栈上变量，无竞争
        TVPThrowExceptionMessage(localError.c_str());
    }
    // ... 其他 catch 同理
    return false;
}
```

### 3.4 PassThroughInvoker——无异常保护的快速路径

simplebinder 还提供了一个**不捕获异常**的调用路径 `PassThroughInvoker`：

```cpp
// 源码位置：cpp/plugins/simplebinder/simplebinder.hpp 第 382-415 行
struct PassThroughInvoker {
    // 直接调用，不做任何 try/catch 包装
    template <typename Func>
    static bool Invoke(Func&& func) {
        func();  // 如果抛异常，直接传播到上层
        return true;
    }
};
```

**何时使用 PassThroughInvoker？**

1. **性能关键路径**：try/catch 在某些编译器/平台上有性能开销（主要是异常表的栈展开机制）。如果确认函数不会抛异常，可以跳过保护。
2. **上层已有保护**：如果调用方已经有 try/catch，避免双重捕获带来的混乱。
3. **调试阶段**：让异常直接传播有助于调试器定位问题源头。

在 simplebinder 中，`ExceptionTrapper` 和 `PassThroughInvoker` 通过模板参数选择：

```cpp
// 使用异常保护（默认）
template <typename Trap = ExceptionRethrowTrap>
struct FunctionCallback {
    static tjs_error TJS_INTF_METHOD Callback(...) {
        return Trap::Try([&]() {
            // 实际函数调用
        }) ? TJS_S_OK : TJS_E_FAIL;
    }
};

// 如果传入 PassThroughInvoker 作为 Trap，则跳过 try/catch
// FunctionCallback<PassThroughInvoker>::Callback(...)
```

## 四、SIMPLEBINDER_CUSTOM_CONTEXT_RESOLV 宏

### 4.1 默认的实例解析机制

当 TJS2 脚本调用一个类实例的方法时（如 `obj.moveTo(100, 200)`），simplebinder 需要从 TJS2 对象（`iTJSDispatch2 *objthis`）中获取对应的 C++ 实例指针。默认的解析过程通过 `ClassInstanceResolver` 模板完成：

```cpp
// 源码位置：cpp/plugins/simplebinder/simplebinder.hpp 第 1035-1042 行
// 默认实现：从 TJS2 对象中提取 InstanceWrapper，再获取 C++ 实例
template <typename C>
struct ClassInstanceResolver {
    static C* GetInstance(iTJSDispatch2 *objthis) {
        // 通过 NativeClassID 获取 InstanceWrapper
        auto *wrapper = static_cast<InstanceWrapper<C>*>(
            TJSGetNativeInstance(
                ClassStore<C>::_Singleton->_classId,
                objthis
            )
        );
        return wrapper ? wrapper->GetInstance() : nullptr;
    }
};
```

这个默认实现假设 C++ 实例是通过 `ClassStore::CreateClassObject` 注册的——即使用 simplebinder 自己的 `InstanceWrapper` 来持有实例。

### 4.2 自定义解析的需求场景

但有些情况下，C++ 类有自己的实例管理方式。例如：

1. **已有原生类包装器**：某个 C++ 类已经通过 ncbind 或手工代码注册为 TJS2 原生类，有自己的 `tTJSNativeInstance` 子类
2. **跨框架互操作**：C++ 实例存储在 TJS2 对象的自定义位置（如 `ClassID` 不同于 simplebinder 的默认值）
3. **代理模式**：TJS2 对象不直接持有 C++ 实例，而是通过间接方式（如全局表、弱引用）获取

对于这些场景，`SIMPLEBINDER_CUSTOM_CONTEXT_RESOLV` 宏提供了解析逻辑的自定义能力：

```cpp
// 源码位置：cpp/plugins/simplebinder/simplebinder.hpp 第 1226-1237 行
// 宏定义：特化 ClassInstanceResolver 以使用自定义的 GetInstance 实现
#define SIMPLEBINDER_CUSTOM_CONTEXT_RESOLV(C)                     \
    template <>                                                    \
    struct SimpleBinder::ClassInstanceResolver<C> {                \
        static C* GetInstance(iTJSDispatch2 *objthis) {           \
            return C::GetInstance(objthis);                        \
        }                                                          \
    };
```

### 4.3 使用示例

假设你有一个类 `LayerProxy`，它的实例存储方式与 simplebinder 默认不同：

```cpp
// 自定义类：有自己的实例获取方式
class LayerProxy {
    int _layerId;
    static std::unordered_map<iTJSDispatch2*, LayerProxy*> _instances;

public:
    LayerProxy(int id) : _layerId(id) {}

    // 自定义的实例解析：从全局映射表中查找
    static LayerProxy* GetInstance(iTJSDispatch2 *objthis) {
        auto it = _instances.find(objthis);
        return (it != _instances.end()) ? it->second : nullptr;
    }

    // 注册实例到映射表
    static void Register(iTJSDispatch2 *obj, LayerProxy *instance) {
        _instances[obj] = instance;
    }

    int getLayerId() const { return _layerId; }
    void setOpacity(double alpha) {
        // 设置图层透明度的实现...
    }
};

std::unordered_map<iTJSDispatch2*, LayerProxy*> LayerProxy::_instances;

// 关键：在任何使用 BindUtil 之前声明自定义解析
SIMPLEBINDER_CUSTOM_CONTEXT_RESOLV(LayerProxy)

// 然后正常注册方法——simplebinder 会自动使用 LayerProxy::GetInstance
// 而不是默认的 InstanceWrapper 解析
SimpleBinder::BindUtil(TJS_W("LayerPlugin"))
    .Class<LayerProxy>(TJS_W("LayerProxy"))  // 注意：无工厂函数，命名空间模式
        .Function(TJS_W("getLayerId"), &LayerProxy::getLayerId)
        .Function(TJS_W("setOpacity"), &LayerProxy::setOpacity);
```

> **注意**：`SIMPLEBINDER_CUSTOM_CONTEXT_RESOLV` 必须在**全局作用域**声明（不能在函数内部），因为它是一个模板特化声明。而且必须在使用 `BindUtil` 注册该类的方法之前声明，否则编译器会使用默认的 `ClassInstanceResolver`。

// 在 MethodInvokerBase 中，参数转换使用 static_cast：

template <typename T>

### 5.3 何时选择 simplebinder vs ncbind

| 场景 | 推荐 | 原因 |
|------|------|------|
| 简单全局函数/常量 | simplebinder | 代码更少，API 更直观 |
| 简单类（无继承） | 两者皆可 | simplebinder 代码更短，ncbind 功能更全 |
| 需要类继承 | ncbind | simplebinder 不支持 |

| 新项目/原型开发 | simplebinder | 更快上手，更少样板代码 |

> **KrKr2 项目的实际选择**：所有 14 个已实现插件无一例外使用 ncbind。simplebinder 虽然包含在项目中（`cpp/plugins/simplebinder/simplebinder.hpp`），但从未被任何 `.cpp` 文件引用。它更像是一个"备选方案"或"实验性实现"。

## 动手实践

### 实践 1：用 ClassStore 注册一个完整类

目标：使用 simplebinder 的完整类注册功能，创建一个可以在 TJS2 中 `new` 出来的计时器类。

**步骤 1**：编写 C++ 类和工厂函数

```cpp
// timer_plugin.cpp
#include "simplebinder/simplebinder.hpp"
#include <chrono>

// 简单计时器类
class SimpleTimer {
    std::chrono::steady_clock::time_point _start;
    ttstr _label;  // ttstr = TJS2 宽字符串类型

public:
    // 构造函数：接受一个标签名
    SimpleTimer(const tjs_char* label)
        : _start(std::chrono::steady_clock::now())
        , _label(label) {}

    ~SimpleTimer() = default;

    // 获取已过毫秒数
    double elapsed() const {
        auto now = std::chrono::steady_clock::now();
        auto ms = std::chrono::duration_cast<std::chrono::microseconds>(
            now - _start).count();
        return static_cast<double>(ms) / 1000.0;  // 微秒转毫秒
    }

    // 重置计时器
    void reset() {
        _start = std::chrono::steady_clock::now();
    }

    // 获取标签
    ttstr getLabel() const { return _label; }
};

// 工厂函数——simplebinder 要求返回裸指针
static SimpleTimer* CreateSimpleTimer(const tjs_char* label) {
    return new SimpleTimer(label);
}

// 注册入口
static void RegisterTimerPlugin() {
    SimpleBinder::BindUtil(TJS_W("TimerPlugin"))
        .Class<SimpleTimer>(TJS_W("SimpleTimer"), &CreateSimpleTimer)
            .Function(TJS_W("elapsed"), &SimpleTimer::elapsed)
            .Function(TJS_W("reset"), &SimpleTimer::reset)
            .Function(TJS_W("getLabel"), &SimpleTimer::getLabel);
}

// 使用 ncbind 的自动注册机制触发
NCB_PRE_REGIST_CALLBACK(RegisterTimerPlugin);
```

**步骤 2**：在 TJS2 脚本中使用

```javascript
// TJS2 脚本
var timer = new SimpleTimer("加载计时");
// ... 执行一些操作 ...
System.inform("耗时: " + timer.elapsed() + " 毫秒");
timer.reset();
// ... 更多操作 ...
System.inform(timer.getLabel() + " 第二段耗时: " + timer.elapsed() + " 毫秒");
```

**步骤 3**：验证结果

运行后应看到两个弹窗，分别显示两段操作的耗时。如果 `new SimpleTimer(...)` 失败（返回 `void` 而非对象），说明 ClassStore 注册有问题——检查工厂函数签名是否正确。

### 实践 2：测试 Singleton 限制

验证 ClassStore 的 Singleton 行为：

```cpp
// 尝试注册同一个 C++ 类型两次
static void TestSingletonLimit() {
    // 第一次注册——成功
    SimpleBinder::BindUtil(TJS_W("Test1"))
        .Class<SimpleTimer>(TJS_W("Timer1"), &CreateSimpleTimer)
            .Function(TJS_W("elapsed"), &SimpleTimer::elapsed);

    // 第二次注册同一类型——ClassStore 会打印 "Class already created"
    // 并返回第一次创建的类对象
    SimpleBinder::BindUtil(TJS_W("Test2"))
        .Class<SimpleTimer>(TJS_W("Timer2"), &CreateSimpleTimer)
            .Function(TJS_W("reset"), &SimpleTimer::reset);

    // 结果：TJS2 中只有 Timer1（或 Timer2 指向同一个类对象）
    // Timer2 上的 reset 方法可能不存在或行为异常
}
```

预期输出：日志中出现 `"Class already created"` 警告。

## 常见错误与排查

### 错误 1：工厂函数返回值类型不匹配

**现象**：编译错误，模板推导失败

```
error: no matching function for call to 'ClassStore<MyClass>::CreateClassObject'
```

**原因**：工厂函数的返回值不是 `C*`（裸指针）

```cpp
// ❌ 错误：返回 unique_ptr
static std::unique_ptr<MyClass> CreateMyClass() {
    return std::make_unique<MyClass>();
}

// ✅ 正确：返回裸指针
static MyClass* CreateMyClass() {
    return new MyClass();
}
```

### 错误 2：同一 C++ 类型注册两次

**现象**：日志中出现 `"Class already created"`，第二次注册的方法/属性丢失

**原因**：`ClassStore<C>` 的 Singleton 限制

**解决方案**：确保每个 C++ 类型只注册一次。如果需要以不同名字暴露同一类型，使用 wrapper 类：

```cpp
// 用 wrapper 类包装，使其成为不同的 C++ 类型
class TimerAlias : public SimpleTimer {
    using SimpleTimer::SimpleTimer;  // 继承构造函数
};

// 现在可以分别注册
SimpleBinder::BindUtil(TJS_W("P1"))
    .Class<SimpleTimer>(TJS_W("Timer"), &CreateSimpleTimer);
SimpleBinder::BindUtil(TJS_W("P2"))
    .Class<TimerAlias>(TJS_W("FastTimer"), &CreateTimerAlias);
```

### 错误 3：SIMPLEBINDER_CUSTOM_CONTEXT_RESOLV 声明位置错误

**现象**：编译通过，但运行时 `GetInstance` 返回 `nullptr`

**原因**：宏声明在函数内部或在 `BindUtil` 使用之后

```cpp
// ❌ 错误：在函数内部声明
void Register() {
    SIMPLEBINDER_CUSTOM_CONTEXT_RESOLV(MyClass)  // 不会生效！
    SimpleBinder::BindUtil(TJS_W("X"))
        .Class<MyClass>(TJS_W("MyClass"));
}

// ✅ 正确：在全局作用域，且在使用之前
SIMPLEBINDER_CUSTOM_CONTEXT_RESOLV(MyClass)

void Register() {
    SimpleBinder::BindUtil(TJS_W("X"))
        .Class<MyClass>(TJS_W("MyClass"));
}
```

## 本节小结

- **UNROLL_FOREACH** 是 simplebinder 的核心代码生成机制。它通过预处理器宏一次性生成 0-8 参数共 9 个模板特化，避免了 C++14 时代缺乏折叠表达式的限制。全文件 12 处调用产生 108 个模板特化
- **ClassStore + FactoryInvoker** 赋予 simplebinder 真正的类注册能力。通过工厂函数模式，simplebinder 可以创建支持 `new ClassName()` 的完整 TJS2 原生类。但 Singleton 限制意味着每个 C++ 类型只能注册一次
- **InstanceWrapper** 桥接 C++ 对象与 TJS2 对象的生命周期。它实现 `tTJSNativeInstance` 接口，在 TJS2 构造时调用工厂函数，在垃圾回收时调用析构逻辑
- **ExceptionRethrowTrap** 自动将 C++ 异常转换为 TJS2 错误消息，防止异常逃逸导致崩溃。但其 `static ttstr error` 设计存在线程安全隐患
- **PassThroughInvoker** 提供无异常保护的快速调用路径，适合性能关键场景或调试
- **SIMPLEBINDER_CUSTOM_CONTEXT_RESOLV** 宏允许自定义从 TJS2 对象到 C++ 实例的解析逻辑，支持与已有原生类系统的互操作
- simplebinder 虽然功能完整但在 KrKr2 中**从未被使用**——所有插件选择了功能更全面的 ncbind

### 题目 1：分析 UNROLL_FOREACH 的展开结果

给定以下宏定义和调用：

```cpp
#define MY_FUNC(N) \
    void process##N(UNROLL_AM##N(int, float));
UNROLL_FOREACH(MY_FUNC)
```

问：`MY_FUNC(2)` 展开后的最终结果是什么？`UNROLL_FOREACH` 总共生成几个函数声明？

<details>
<summary>查看答案</summary>

`MY_FUNC(2)` 展开为：

```cpp
void process2(UNROLL_AM2(int, float));
```

`UNROLL_AM2(int, float)` 展开为 `int A, float A1, float A2`（一个固定前缀类型 + 2 个展开参数类型）。

最终结果：

```cpp
void process2(int A, float A1, float A2);
```

`UNROLL_FOREACH(MY_FUNC)` 展开为 `MY_FUNC(0)` 到 `MY_FUNC(8)`，共 **9 个函数声明**：

```cpp
void process0(int A);                                          // 0 参数
void process1(int A, float A1);                                // 1 参数
void process2(int A, float A1, float A2);                      // 2 参数
void process3(int A, float A1, float A2, float A3);            // 3 参数
// ... 以此类推到 process8
void process8(int A, float A1, float A2, ..., float A8);      // 8 参数
```

注意：`A` 始终是第一个固定前缀参数，用 `int` 类型；`A1`-`AN` 是展开的参数，用 `float` 类型。

</details>

### 题目 2：解释 ClassStore Singleton 限制的技术原因

为什么 `ClassStore<C>` 限制每个 C++ 类型只能注册一次？如果去掉这个限制会发生什么？请从 TJS2 的 NativeClassID 机制角度分析。

<details>
<summary>查看答案</summary>

**限制原因**：

TJS2 使用 NativeClassID（一个整数标识符）来区分不同的原生类。`ClassStore::CreateClassObject()` 在创建类对象时调用 `TJSRegisterNativeClass(key)` 注册一个新的 NativeClassID。

当需要从 TJS2 对象中获取 C++ 实例时，`ClassInstanceResolver<C>::GetInstance()` 使用 `ClassStore<C>::_Singleton->_classId` 作为查找键。如果同一个 C++ 类型 `C` 注册两次：

1. 第二次 `TJSRegisterNativeClass()` 会生成一个**新的** NativeClassID
2. `_Singleton->_classId` 会被覆盖为新的 ID
3. 通过第一次注册创建的实例，其 NativeClassID 是旧 ID
4. 用新 ID 去查找旧 ID 的实例→**找不到**→返回 `nullptr`→方法调用失败

**如果去掉限制**：

```
第一次注册：ClassStore<Timer>::_classId = 42
    → new Timer("a") 时，TJS2 关联 NativeClassID=42

第二次注册：ClassStore<Timer>::_classId = 87  ← 覆盖！
    → new Timer("b") 时，TJS2 关联 NativeClassID=87

调用 timer_a.elapsed()：
    → GetInstance 用 _classId=87 查找
    → timer_a 的 NativeClassID=42 ≠ 87
    → 返回 nullptr → 方法调用失败！
```

所以 Singleton 限制不是任意设计，而是 TJS2 原生类系统的结构性约束。

</details>

### 题目 3：修复 ExceptionRethrowTrap 的线程安全问题

`ExceptionRethrowTrap` 使用 `static ttstr error` 存储错误消息，这在多线程环境下不安全。请写出修复后的完整 `Try` 方法，要求：(1) 线程安全，(2) 不改变外部接口，(3) 使用 C++17 特性。

<details>
<summary>查看答案</summary>

```cpp
class ExceptionRethrowTrap {
public:
    template <typename Func>
    static bool Try(Func&& func) {
        try {
            func();
            return true;
        }
        catch (const std::exception& e) {
            // 使用栈上局部变量——天然线程安全
            ttstr localError(e.what());
            TVPThrowExceptionMessage(localError.c_str());
        }
        catch (const ttstr& msg) {
            // 直接使用捕获的引用——不需要额外存储
            TVPThrowExceptionMessage(msg.c_str());
        }
        catch (const tjs_char* msg) {
            TVPThrowExceptionMessage(msg);
        }
        catch (...) {
            // 字符串字面量是全局常量，线程安全
            TVPThrowExceptionMessage(TJS_W("Unknown C++ exception"));
        }
        return false;
    }
    // 注意：移除了 static ttstr error 成员——完全不需要
};
```

**关键改动**：

1. **移除 `static ttstr error`**——这是线程不安全的根源
2. **在每个 catch 块内使用局部变量或直接传递**——栈上变量天然线程安全
3. **对于 `ttstr` 类型的异常**，直接使用 catch 捕获的引用的 `.c_str()`，因为在 catch 块内该引用有效
4. **对于 `const tjs_char*`**，直接传递指针——异常对象在 catch 块内有效

这个修改完全不改变外部接口——`Try` 函数的签名和返回值含义完全相同。C++17 特性体现在模板参数推导（虽然 C++14 也支持），以及可以使用 `if constexpr` 进一步优化（本例未使用）。

</details>
