## 一、实战目标——TextMetrics 插件

### 1.1 为什么选择 TextMetrics？

在完成了对 scriptsEx、csvParser、psdfile、psbfile、layerExDraw、fstat 六个插件的源码分析后，我们现在要**从零开始设计并实现一个真实可用的插件**。

我们选择实现 **TextMetrics 插件**，理由如下：

1. **真实需求**：游戏脚本经常需要计算文字渲染后的宽度和高度，以实现文本自动换行、对话框大小调整、文字居中等效果
2. **复杂度适中**：需要与系统字体 API 交互，有跨平台差异，但核心逻辑不复杂
3. **覆盖核心知识点**：涵盖 NCB_REGISTER_CLASS、Factory 构造、NCB_METHOD、跨平台条件编译等所有关键模式
4. **与已有插件互补**：layerExDraw 提供绘图能力，TextMetrics 提供排版计算能力，逻辑上是配套功能

### 1.2 功能需求清单

TextMetrics 插件需要提供以下 TJS API：

```javascript
// TJS 脚本使用示例

// 1. 创建 TextMetrics 实例，指定字体参数
var tm = new TextMetrics("MS Gothic", 24, TextMetrics.STYLE_BOLD);

// 2. 计算单行文字的宽度和高度
var w = tm.measureWidth("Hello, World!");     // 返回像素宽度
var h = tm.measureHeight("Hello, World!");    // 返回像素高度

// 3. 获取完整的度量信息（返回字典）
var info = tm.measureText("Hello, World!");
// info.width     → 文字宽度（像素）
// info.height    → 文字高度（像素）
// info.ascent    → 基线以上高度（Ascent，从基线到字符最高点的距离）
// info.descent   → 基线以下高度（Descent，从基线到字符最低点的距离）
// info.leading   → 行间距（Leading，两行文字基线之间除去 ascent+descent 的额外间距）

// 4. 计算多行文字在指定宽度内的自动换行结果
var lines = tm.wrapText("很长很长的一段话...", 300);
// lines 是数组，每个元素是一行文字

// 5. 修改字体参数
tm.setFont("SimSun", 18, TextMetrics.STYLE_NORMAL);

// 6. 获取当前字体信息
var fontName = tm.fontName;    // 只读属性
var fontSize = tm.fontSize;    // 只读属性
```

### 1.3 非功能需求

| 需求 | 说明 |
|------|------|
| **跨平台** | Windows、Linux、macOS、Android 四平台都必须工作 |
| **性能** | `measureWidth` 需要在 1ms 内返回（游戏中每帧可能调用多次） |
| **字体回退** | 指定字体不存在时，自动回退到平台默认字体而非崩溃 |
| **Unicode 支持** | 正确处理中日韩字符、emoji、混合脚本文本 |
| **无状态安全** | 多个 TextMetrics 实例互不影响 |

## 二、架构设计——注册策略选型

### 2.1 决策：新建类 vs 附加到已有类

回顾我们在 [两插件对比分析](../04-现有插件源码导读/03-layerExDraw与fstat.md) 中学到的选型决策树：

```
TextMetrics 需要扩展已有 TJS 类吗？
├─ 否 → 它是一个独立的功能模块
│       不需要附加到 Layer 或 Storages
│       → 选择 NCB_REGISTER_CLASS
```

TextMetrics 是**独立的新类**，有自己的实例状态（字体名、字号、样式、内部字体句柄），不需要附加到任何已有类。因此选择 `NCB_REGISTER_CLASS`。

### 2.2 决策：默认构造 vs Factory 构造

```
TextMetrics 的构造函数需要参数吗？
├─ 是 → 需要字体名、字号、样式
│       但这些都是 ncbind 可以自动转换的基本类型（string、int）
│       不需要 objthis
│       → 可以用默认的 Constructor(...) 宏
```

如果构造函数参数全是可自动转换的类型，不需要 `objthis`，就不必用 Factory。对比：

```cpp
// 方案 A：Constructor 宏（适合 TextMetrics）
NCB_REGISTER_CLASS(TextMetrics) {
    Constructor<const ttstr&, int, int>(0);  // (fontName, fontSize, style)
    // ...
};

// 方案 B：Factory 模式（适合需要 objthis 的类，如 PSD）
NCB_REGISTER_CLASS(TextMetrics) {
    Factory(&Class::factory);
    // factory 签名：static tjs_error factory(TextMetrics**, ...)
};
```

我们选择**方案 A**——更简洁，参数类型自动转换。

### 2.3 决策：属性实现方式

TextMetrics 有两个只读属性 `fontName` 和 `fontSize`。ncbind 提供两种方式：

```cpp
// 方式 1：Property 宏（getter + setter 函数指针）
Property(TJS_W("fontName"), &Class::getFontName, nullptr);  // setter=nullptr → 只读

// 方式 2：PROPERTY_RO 简写宏
NCB_PROPERTY_RO(fontName, getFontName);
```

两种方式效果相同，方式 2 更简洁。

### 2.4 完整的注册设计

```cpp
// TextMetrics 插件注册蓝图

// 常量定义
NCB_REGISTER_CLASS(TextMetrics) {
    // 样式常量——作为类级常量注册
    Variant(TJS_W("STYLE_NORMAL"),    (tjs_int)0);
    Variant(TJS_W("STYLE_BOLD"),      (tjs_int)1);
    Variant(TJS_W("STYLE_ITALIC"),    (tjs_int)2);
    Variant(TJS_W("STYLE_UNDERLINE"), (tjs_int)4);

    // 构造函数：(fontName, fontSize, style)
    Constructor<const ttstr&, int, int>(0);

    // 度量方法
    NCB_METHOD(measureWidth);    // float measureWidth(ttstr text)
    NCB_METHOD(measureHeight);   // float measureHeight(ttstr text)
    NCB_METHOD(measureText);     // tTJSVariant measureText(ttstr text)
    NCB_METHOD(wrapText);        // tTJSVariant wrapText(ttstr text, int maxWidth)

    // 字体管理
    NCB_METHOD(setFont);         // void setFont(ttstr name, int size, int style)

    // 只读属性
    NCB_PROPERTY_RO(fontName, getFontName);
    NCB_PROPERTY_RO(fontSize, getFontSize);
};
```

## 三、TJS 接口设计——API 契约

### 3.1 命名规范

分析 KrKr2 已有插件的命名惯例：

| 模式 | 示例 | 规则 |
|------|------|------|
| 方法名 | `measureWidth`、`drawLine`、`getTime` | camelCase，动词开头 |
| 属性名 | `fontName`、`fontSize`、`width` | camelCase，名词 |
| 常量名 | `STYLE_BOLD`、`FILE_ATTRIBUTE_READONLY` | UPPER_SNAKE_CASE |
| 类名 | `TextMetrics`、`LayerExDraw`、`PSD` | PascalCase |

我们的 API 设计完全遵循这些惯例。

### 3.2 参数与返回值规范

| 方法 | 参数 | 返回值 | 说明 |
|------|------|--------|------|
| `measureWidth(text)` | `ttstr` 文本 | `float` 像素宽度 | 最常用，性能优先 |
| `measureHeight(text)` | `ttstr` 文本 | `float` 像素高度 | 包含 ascent + descent + leading |
| `measureText(text)` | `ttstr` 文本 | TJS 字典 | 返回完整度量信息 |
| `wrapText(text, maxWidth)` | 文本 + 最大宽度 | TJS 数组 | 每元素是一行文字 |
| `setFont(name, size, style)` | 字体名 + 字号 + 样式 | void | 切换字体 |
| `getFontName()` | 无 | `ttstr` 字体名 | 只读属性 getter |
| `getFontSize()` | 无 | `int` 字号 | 只读属性 getter |

### 3.3 错误处理策略

分析已有插件的错误处理模式后，确定以下策略：

```cpp
// 策略 1：无效参数 → 返回默认值 + 日志警告（仿 fstat）
float TextMetrics::measureWidth(const ttstr &text) {
    if (text.IsEmpty()) {
        return 0.0f;  // 空字符串宽度为 0，不报错
    }
    if (!fontHandle) {
        spdlog::warn("TextMetrics: font not initialized");
        return 0.0f;  // 字体未初始化，返回 0 而非崩溃
    }
    // ... 正常计算
}

// 策略 2：构造函数中字体不存在 → 回退到默认字体 + 警告
TextMetrics::TextMetrics(const ttstr &name, int size, int style) {
    fontHandle = createFont(name, size, style);
    if (!fontHandle) {
        spdlog::warn("TextMetrics: font '{}' not found, "
                     "falling back to default", name.AsStdString());
        fontHandle = createFont(getDefaultFontName(), size, style);
    }
}
```

这遵循了 KrKr2 的一贯风格：**静默降级，不抛异常**。游戏引擎中，一个字体找不到不应该让整个游戏崩溃。

### 3.4 measureText 返回值格式

`measureText` 返回一个 TJS 字典，格式如下：

```javascript
// measureText("Hello") 返回的字典结构
{
    "width":   120.5,    // 文字总宽度（像素，float）
    "height":  28.0,     // 文字总高度（像素，float）
    "ascent":  22.0,     // 基线以上高度
    "descent": 6.0,      // 基线以下高度
    "leading": 2.0       // 行间距额外空间
}
```

C++ 实现中使用 `ncbDictionaryAccessor` 构建：

```cpp
tTJSVariant TextMetrics::measureText(const ttstr &text) {
    ncbDictionaryAccessor dict;

    // 获取平台相关的文字度量
    FontMetrics metrics = measureTextPlatform(text);

    dict.SetValue(TJS_W("width"),   (double)metrics.width);
    dict.SetValue(TJS_W("height"),  (double)metrics.height);
    dict.SetValue(TJS_W("ascent"),  (double)metrics.ascent);
    dict.SetValue(TJS_W("descent"), (double)metrics.descent);
    dict.SetValue(TJS_W("leading"), (double)metrics.leading);

    return tTJSVariant(dict.GetDispatch(), dict.GetDispatch());
}
```

## 四、跨平台策略设计

### 4.1 平台差异审计

TextMetrics 的核心操作是**计算文字的渲染尺寸**，这依赖平台的字体 API：

| 平台 | 字体 API | 度量函数 | 字体查找 |
|------|---------|---------|---------|
| Windows | GDI / DirectWrite | `GetTextExtentPoint32W` 或 `IDWriteTextLayout` | 系统字体目录 + 注册表 |
| Linux | FreeType + Fontconfig | `FT_Load_Char` + `FT_Get_Kerning` | `fc-match` 字体查找 |
| macOS | Core Text | `CTLineGetTypographicBounds` | `NSFontManager` |
| Android | Skia (通过 NDK) | `SkFont::measureText` 或 `Paint.measureText` (JNI) | 系统 `/system/fonts/` |

### 4.2 架构：平台抽象层

为了最小化平台条件编译的范围，我们设计一个**平台抽象层**：

```
TextMetrics（TJS 接口层）
    │
    ├── TextMetricsImpl.h        ← 平台无关的抽象接口
    │
    ├── TextMetricsImpl_Win.cpp  ← Windows: GDI
    ├── TextMetricsImpl_FT.cpp   ← Linux/Android: FreeType
    └── TextMetricsImpl_CT.cpp   ← macOS: Core Text
```

抽象接口定义：

```cpp
// TextMetricsImpl.h——平台无关接口
#pragma once
#include <string>

struct FontMetrics {
    float width;     // 文字总宽度
    float height;    // 文字总高度
    float ascent;    // 基线以上
    float descent;   // 基线以下
    float leading;   // 行间距
};

class TextMetricsImpl {
public:
    virtual ~TextMetricsImpl() = default;

    // 设置字体
    virtual bool setFont(const std::u16string &name,
                         int size, int style) = 0;

    // 获取完整度量
    virtual FontMetrics measure(const std::u16string &text) = 0;

    // 获取文字宽度（性能优化版，不需要完整度量）
    virtual float measureWidth(const std::u16string &text) = 0;

    // 获取默认字体名
    virtual std::u16string getDefaultFontName() = 0;

    // 工厂方法——根据平台创建实现
    static std::unique_ptr<TextMetricsImpl> create();
};
```

### 4.3 条件编译 CMake 设计

```cmake
# TextMetrics 的 CMakeLists.txt
cmake_minimum_required(VERSION 3.28)
project(textMetrics)

# 公共源码
set(TEXTMETRICS_SRC
    main.cpp                # ncbind 注册
    TextMetrics.cpp         # TJS 接口层
    TextMetricsImpl.h       # 平台抽象接口
)

# 平台特定源码
if(WINDOWS)
    list(APPEND TEXTMETRICS_SRC TextMetricsImpl_Win.cpp)
    # Windows 不需要额外依赖（GDI 是系统自带的）
elseif(MACOS)
    list(APPEND TEXTMETRICS_SRC TextMetricsImpl_CT.cpp)
    # Core Text 是 macOS 系统框架
elseif(LINUX OR ANDROID)
    list(APPEND TEXTMETRICS_SRC TextMetricsImpl_FT.cpp)
    # 需要 FreeType 和 Fontconfig
    find_package(Freetype REQUIRED)
    if(LINUX)
        find_package(Fontconfig REQUIRED)
    endif()
endif()

add_library(${PROJECT_NAME} STATIC ${TEXTMETRICS_SRC})

# 平台特定链接
if(WINDOWS)
    target_link_libraries(${PROJECT_NAME} PRIVATE gdi32)
elseif(MACOS)
    target_link_libraries(${PROJECT_NAME} PRIVATE
        "-framework CoreText"
        "-framework CoreFoundation"
        "-framework CoreGraphics"
    )
elseif(LINUX)
    target_link_libraries(${PROJECT_NAME} PRIVATE
        Freetype::Freetype
        Fontconfig::Fontconfig
    )
elseif(ANDROID)
    target_link_libraries(${PROJECT_NAME} PRIVATE
        Freetype::Freetype
    )
endif()
```

### 4.4 与 layerExDraw 条件编译的对比

| 方面 | layerExDraw | TextMetrics（我们的设计） |
|------|-------------|--------------------------|
| 条件粒度 | 整个源码目录切换 | 单个实现文件切换 |
| 接口统一 | 通过 C API（GDI+ Flat API） | 通过 C++ 虚函数（TextMetricsImpl） |
| 优点 | 无虚函数开销 | 接口清晰，新平台只需加一个文件 |
| 缺点 | 代码重复（两套目录） | 虚函数调用开销（微不足道） |
| 适合 | 平台差异极大 | 平台差异主要在实现细节 |

## 常见错误与排查

### 错误 1：Constructor 宏参数类型不匹配

**现象**：编译错误 `no matching function for call to Constructor`

**原因**：`Constructor<>` 模板参数必须与 C++ 构造函数签名精确匹配。

```cpp
// 错误：C++ 构造函数是 (const ttstr&, int, int)
// 但模板写成了 (ttstr, int, int)——缺少 const&
Constructor<ttstr, int, int>(0);  // ❌ 编译错误

// 正确：类型必须精确匹配
Constructor<const ttstr&, int, int>(0);  // ✅
```

### 错误 2：忘记在 plugins/CMakeLists.txt 注册

**现象**：插件编译通过，但 TJS 脚本中 `new TextMetrics(...)` 报 `member not found`

**原因**：插件的 .a/.lib 编译了，但没有链接到 krkr2plugin，所以 ncbind 的自动注册代码没有被链接进最终二进制。

**排查**：检查 `cpp/plugins/CMakeLists.txt` 是否有 `add_subdirectory(textMetrics)` 和 `target_link_libraries(krkr2plugin PUBLIC textMetrics)`。

### 错误 3：跨平台字体名不一致

**现象**：Windows 上 `new TextMetrics("MS Gothic", 24, 0)` 正常，Linux 上报字体找不到

**原因**：Linux 的 Fontconfig 字体名可能与 Windows 不同（如 `MS Gothic` → `MS PGothic` 或完全没有）。

**解决**：在 `TextMetricsImpl_FT.cpp` 中使用 Fontconfig 的 `FcFontMatch` 做模糊匹配，而非精确匹配。

## 本节小结

- **TextMetrics 插件**是一个典型的"独立新类"插件，选择 `NCB_REGISTER_CLASS` + `Constructor` 而非 Factory 或 ATTACH
- **API 设计遵循 KrKr2 惯例**：camelCase 方法名、UPPER_SNAKE_CASE 常量、TJS 字典返回复杂数据
- **错误处理策略是静默降级**：字体找不到就回退默认，返回 0 而非抛异常
- **跨平台设计采用抽象接口 + 条件编译实现**：比 layerExDraw 的"双目录"更细粒度，更易维护
- **依赖审计在设计阶段完成**：确认 GDI/FreeType/CoreText 在各平台的可用性和获取方式
- **构建集成需要两步**：`add_subdirectory` + `target_link_libraries`

**(1) 新增 API：**

```cpp
// 新增属性：字间距（像素）
NCB_PROPERTY(letterSpacing, getLetterSpacing, setLetterSpacing);

// C++ 实现
private:
    float letterSpacing_ = 0.0f;  // 默认无额外间距

public:
    float getLetterSpacing() const { return letterSpacing_; }
    void setLetterSpacing(float spacing) { letterSpacing_ = spacing; }
```

**(2) 需要修改的方法：**

- `measureWidth`：累加每个字符宽度时额外加上 `letterSpacing * (charCount - 1)`
- `measureText`：同上，影响 `width` 字段
- `wrapText`：自动换行计算时要考虑字间距

```cpp
float TextMetrics::measureWidth(const ttstr &text) {
    float totalWidth = impl->measureWidth(text.AsStdU16String());
    // 加上字间距：(字符数 - 1) × 间距
    int charCount = text.GetLen();
    if (charCount > 1 && letterSpacing_ != 0.0f) {
        totalWidth += letterSpacing_ * (charCount - 1);
    }
    return totalWidth;
}
```

**(3) 跨平台挑战：**

- **Windows GDI**：`SetTextCharacterExtra` 可以全局设置字间距，但只影响 `DrawText`，不影响 `GetTextExtentPoint32W` 的返回值——需要手动累加
- **FreeType**：FreeType 本身不支持字间距，纯粹是应用层逻辑——逐字符 `FT_Load_Char` 累加 advance + letterSpacing
- **Core Text**：`kCTKernAttributeName` 可以设置 kern 值，但这是字偶距（kerning）而非字间距（letter spacing），需要区分

因此最可靠的实现是**在平台无关层处理字间距**：先获取不含间距的宽度，再按字符数累加，避免依赖平台特定的间距 API。

### 题目 2：选择注册策略

以下场景各应使用哪种 ncbind 注册宏？请说明理由。

场景 A：一个 JSON 解析器，TJS 脚本通过 `JSON.parse(str)` 和 `JSON.stringify(obj)` 使用，所有方法都是静态的。

场景 B：一个粒子系统，需要在每个 Layer 上独立维护粒子状态（位置、速度、生命周期等）。

场景 C：一个加密工具，提供 `Cipher.encrypt(data, key)` 和 `Cipher.decrypt(data, key)` 两个静态方法。

<details>
<summary>查看答案</summary>

**场景 A（JSON 解析器）**：`NCB_REGISTER_CLASS(JSON)`

理由：JSON 是独立的功能类，不需要附加到已有类。虽然所有方法是静态的，但 TJS 中通过 `JSON.parse()` 调用需要 JSON 这个类名存在。使用 `NCB_REGISTER_CLASS` 创建类，方法用 `NCB_METHOD` 注册（ncbind 会自动处理静态方法）。

```cpp
NCB_REGISTER_CLASS(JSON) {
    NCB_METHOD(parse);
    NCB_METHOD(stringify);
};
```

**场景 B（粒子系统）**：`NCB_ATTACH_CLASS_WITH_HOOK(ParticleSystem, Layer)`

理由：粒子系统需要附加到已有的 Layer 类，且每个 Layer 有独立的粒子状态——这恰好是 layerExDraw 的模式。需要 Instance Hook 来惰性创建每个 Layer 的粒子系统实例。

```cpp
NCB_GET_INSTANCE_HOOK(ParticleSystem) {
    NCB_INSTANCE_GETTER(objthis) {
        ClassT *obj = GetNativeInstance(objthis);
        if (!obj) {
            obj = new ClassT(objthis);
            SetNativeInstance(objthis, obj);
        }
        return obj;
    }
};
NCB_ATTACH_CLASS_WITH_HOOK(ParticleSystem, Layer) {
    NCB_METHOD(emit);
    NCB_METHOD(update);
    NCB_METHOD(setGravity);
};
```

**场景 C（加密工具）**：`NCB_REGISTER_CLASS(Cipher)` 或 `NCB_ATTACH_CLASS(CipherExt, System)`

两种方案都可以。如果希望通过 `Cipher.encrypt()` 调用，用 `NCB_REGISTER_CLASS`。如果希望通过 `System.encrypt()` 调用（扩展已有类），用 `NCB_ATTACH_CLASS`。由于 Cipher 是独立概念，`NCB_REGISTER_CLASS` 更清晰。

</details>

### 题目 3：设计 wrapText 算法

编写 `wrapText` 方法的伪代码，输入一段文字和最大宽度，输出自动换行后的行数组。需要处理以下边界情况：
- 单个字符就超过最大宽度
- 文本中包含手动换行符 `\n`
- 中日韩字符可以在任意位置断行，但英文单词不应从中间断开

<details>
<summary>查看答案</summary>

```cpp
// wrapText 伪代码
std::vector<std::u16string> wrapText(
    const std::u16string &text, float maxWidth
) {
    std::vector<std::u16string> lines;
    std::u16string currentLine;
    float currentWidth = 0;

    size_t i = 0;
    while (i < text.size()) {
        // 处理手动换行符
        if (text[i] == u'\n') {
            lines.push_back(currentLine);
            currentLine.clear();
            currentWidth = 0;
            i++;
            continue;
        }

        // 判断字符类型
        if (isCJK(text[i])) {
            // CJK 字符：可以在任意位置断行
            float charWidth = measureCharWidth(text[i]);
            if (currentWidth + charWidth > maxWidth
                && !currentLine.empty()) {
                // 当前行放不下，换行
                lines.push_back(currentLine);
                currentLine.clear();
                currentWidth = 0;
            }
            // 单字符超过最大宽度：强制放入（避免死循环）
            currentLine += text[i];
            currentWidth += charWidth + letterSpacing_;
            i++;
        } else {
            // 非 CJK：提取整个英文单词
            std::u16string word;
            while (i < text.size()
                   && !isCJK(text[i])
                   && text[i] != u'\n'
                   && text[i] != u' ') {
                word += text[i++];
            }
            // 包含尾随空格
            if (i < text.size() && text[i] == u' ') {
                word += text[i++];
            }

            float wordWidth = measureStringWidth(word);
            if (currentWidth + wordWidth > maxWidth
                && !currentLine.empty()) {
                lines.push_back(currentLine);
                currentLine.clear();
                currentWidth = 0;
            }
            currentLine += word;
            currentWidth += wordWidth;
        }
    }

    // 最后一行
    if (!currentLine.empty()) {
        lines.push_back(currentLine);
    }

    return lines;
}

// CJK 字符判断
bool isCJK(char16_t ch) {
    return (ch >= 0x4E00 && ch <= 0x9FFF)    // CJK 统一表意文字
        || (ch >= 0x3000 && ch <= 0x303F)    // CJK 标点符号
        || (ch >= 0x3040 && ch <= 0x309F)    // 平假名
        || (ch >= 0x30A0 && ch <= 0x30FF)    // 片假名
        || (ch >= 0xFF00 && ch <= 0xFFEF);   // 全角字符
}
```

关键边界处理：
- **单字符超宽**：即使超过 maxWidth 也放入当前行（避免死循环），绝不跳过字符
- **手动换行符**：在循环最前面检查，保证 `\n` 始终触发换行
- **英文单词不断开**：提取整个 word 后再判断是否换行；如果单词本身就超过 maxWidth 则强制放入

</details>
