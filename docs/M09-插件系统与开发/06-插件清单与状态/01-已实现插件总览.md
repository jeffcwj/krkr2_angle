## 二、注册方式分类分析

从上表可以看到，KrKr2 的插件使用了三种主要的 ncbind 注册方式。理解这些分类有助于在开发新插件时选择正确的注册模式。

### 2.1 NCB_ATTACH_CLASS 类（扩展现有类）

**使用场景：** 为 TJS 中已存在的类（如 `Scripts`、`Storages`、`Window`）添加新方法，而不创建新类。

```cpp
// 典型模式 — scriptsEx.cpp
NCB_ATTACH_CLASS(ScriptsEx, Scripts) {
    NCB_METHOD(exec);
    NCB_METHOD(evalStorage);
    NCB_METHOD(execStorage);
}
```

**使用此方式的插件：**
- `scriptsEx` — 扩展 `Scripts` 类
- `dirlist` — 扩展 `Storages` 类
- `varfile` — 扩展 `Storages` 类
- `saveStruct` — 扩展 `Storages` 类
- `windowEx` — 扩展 `Window` 类
- `addFont` — 扩展 `Font` 相关功能
- `getSample` — 扩展 `WaveSoundBuffer` 类
- `getabout` — 扩展系统信息接口
- `layerExMovie` — 扩展 `Layer` 类
- `fstat` — 扩展 `Storages` 类

**共同特征：** 这些插件都是"功能增强型"——在已有的 TJS 类上添加新方法，而不引入全新的类型。对 TJS 脚本来说，新方法就像是原生类的一部分。

### 2.2 NCB_REGISTER_CLASS 类（注册新类）

**使用场景：** 在 TJS 中创建全新的类，脚本可以通过 `new ClassName()` 实例化。

```cpp
// 典型模式 — csvParser.cpp
NCB_REGISTER_CLASS(CSVParser) {
    Constructor<const ttstr&>(0);
    NCB_METHOD(parseLine);
    NCB_METHOD(parse);
    NCB_PROPERTY_RO(lineCount, getLineCount);
}
```

**使用此方式的插件：**
- `csvParser` — 注册 `CSVParser` 类
- `win32dialog` — 注册 `Win32Dialog` 类
- `fftgraph` — 注册 `FFTGraph` 类
- `layerExPerspective` — 注册透视变换类
- `psdfile` — 注册 `PSD` 类
- `psbfile` — 注册 `PSBFile` 类
- `motionplayer` — 注册 `EmotePlayer` 类

**共同特征：** 这些插件提供了全新的功能域——CSV 解析、PSD 文件操作等——在原版 TJS 中不存在对应的类。

### 2.3 NCB_PRE_REGIST_CALLBACK 类（底层回调注册）

**使用场景：** 不注册类或方法，而是在插件加载时执行一段初始化代码（如修改全局行为、注册过滤器）。

```cpp
// 典型模式 — xp3filter.cpp
static void PreRegistCallback() {
    // 注册 XP3 解密过滤器
    TVPSetXP3ArchiveExtractionFilter(myFilter);
}
NCB_PRE_REGIST_CALLBACK(PreRegistCallback);
```

**使用此方式的插件：**
- `xp3filter` — 注册 XP3 存档解密过滤器
- `wutcwf` — 注册字符宽度修正回调

**共同特征：** 这些插件修改引擎的全局行为，不暴露任何 TJS 可调用的接口。它们在引擎启动时自动生效，对 TJS 脚本完全透明。

### 2.4 NCB_ATTACH_CLASS_WITH_HOOK（带钩子的扩展）

**使用场景：** 扩展现有类，但需要为每个实例维护独立的原生状态（懒加载创建）。

```cpp
// 典型模式 — layerExDraw
NCB_GET_INSTANCE_HOOK(LayerExDraw) { /* 懒加载 */ }
NCB_ATTACH_CLASS_WITH_HOOK(LayerExDraw, Layer) {
    NCB_METHOD(drawLine);
    NCB_METHOD(drawRect);
    // ...
}
```

**使用此方式的插件：**
- `layerex_draw` — 扩展 `Layer` 类，每个 Layer 实例有独立的 GDI+/libgdiplus 绘图上下文

**特殊性：** 这是唯一使用 `WITH_HOOK` 的插件。它需要为每个 TJS Layer 对象自动创建一个关联的 `LayerExDraw` 实例（用于管理 GDI+ 绘图资源），这正是 `GET_INSTANCE_HOOK` 的用途。

### 2.5 注册方式选择决策树

```
开始
 │
 ├── 是否需要注册到 TJS？
 │   ├── 否 → NCB_PRE_REGIST_CALLBACK（全局初始化/过滤器）
 │   └── 是 ↓
 │
 ├── 是扩展现有类还是创建新类？
 │   ├── 创建新类 → NCB_REGISTER_CLASS + Constructor<>
 │   └── 扩展现有类 ↓
 │
 └── 是否需要为每个实例维护独立状态？
     ├── 否 → NCB_ATTACH_CLASS（无状态扩展）
     └── 是 → NCB_ATTACH_CLASS_WITH_HOOK + NCB_GET_INSTANCE_HOOK
```

### 3.2 适配模式深度分析

从上表可以归纳出 KrKr2 插件跨平台适配的 **四种典型模式**：

**模式 A：天然跨平台（14 个插件）**

大多数插件属于这一类——它们的核心逻辑是纯 C++ 运算（字符串解析、数学计算、像素操作、序列化/反序列化），不依赖任何操作系统 API。这类插件只需确保使用标准 C++17 特性即可在所有平台编译运行。

```cpp
// 示例：csvParser.cpp —— 纯 C++ 跨平台插件的典型结构
#include "ncbind.hpp"      // ncbind 框架本身已跨平台
#include <vector>           // 标准库容器
#include <string>           // 标准库字符串

// 所有逻辑都基于标准 C++ — 无需 #ifdef 条件编译
class CSVParser {
    std::vector<std::vector<ttstr>> data;  // 二维数组存储解析结果
public:
    void parseLine(const ttstr& line) {
        // 纯字符串分割逻辑，任何平台都能运行
    }
};
NCB_REGISTER_CLASS(CSVParser) { /* ... */ }
```

**模式 B：引擎抽象层委托（3 个插件）**

`addFont`、`dirlist`、`layerExMovie` 等插件不直接操作文件系统或平台 API，而是调用引擎内部的抽象层接口（如 `TVPEnumFontsProc()`、`TVPGetPlacedPath()`）。平台差异由引擎核心处理，插件层无需感知。

```cpp
// 示例：addFont.cpp —— 委托引擎抽象层
static tjs_error addFont(tTJSVariant *result, tjs_int numparams,
                         tTJSVariant **param, iTJSDispatch2 *objthis) {
    ttstr filename = TVPGetPlacedPath(*param[0]);  // 引擎抽象：跨平台路径解析
    int ret = TVPEnumFontsProc(filename);          // 引擎抽象：跨平台字体注册
    // 插件本身完全不关心当前是什么操作系统
    if(result) *result = (int)ret;
    return TJS_S_OK;
}
```

**模式 C：条件编译 + 替代实现（2 个插件）**

`windowEx` 和 `layerExDraw` 需要调用操作系统原生 API，因此使用 `#ifdef` 条件编译或 CMake 条件选择不同的源文件。

`windowEx` 的方法：在同一文件中用 `#if !defined(_WIN64) && !defined(_WIN32)` 区分平台，非 Windows 平台手动定义 `WM_*` 消息常量和 Win32 类型别名：

```cpp
// windowEx.cpp 第 9-15 行 — 条件编译示例
#if !defined(_WIN64) && !defined(_WIN32)
typedef unsigned long ULONG_PTR;    // 非 Windows 平台模拟 Win32 类型
#else
#include <windows.h>                // Windows 平台使用原生头文件
#include <WinDef.h>
#include <WinUser.h>
#endif

// 第 25-28 行 — 非 Windows 平台定义 WM_* 常量
#ifndef _WIN32
#define WM_NULL    0x0000
#define WM_CREATE  0x0001
#define WM_DESTROY 0x0002
// ... 数十个 WM_* 消息常量
#endif
```

`layerExDraw` 的方法：在 CMakeLists.txt 层面用 `if(WINDOWS)` 选择完全不同的源文件目录：

```cmake
# layerex_draw/CMakeLists.txt — 双目录条件编译
if(WINDOWS)
    target_sources(${PROJECT_NAME} PUBLIC
        windows/main.cpp           # Windows 原生 GDI+ 实现
        windows/LayerExDraw.cpp)
else()
    target_sources(${PROJECT_NAME} PUBLIC
        general/main.cpp           # 跨平台 libgdiplus 实现
        general/LayerExDraw.cpp
        general/DrawPath.cpp)      # 额外的路径绘制模块
    find_package(libgdiplus CONFIG REQUIRED)
    target_link_libraries(${PROJECT_NAME} PRIVATE libgdiplus::libgdiplus)
endif()
```

**模式 D：TJS 脚本层替代（1 个插件）**

`win32dialog` 完全绕过了原版的 Win32 对话框 API，改用 TJS 脚本在运行时动态创建一个兼容类：

```cpp
// win32dialog.cpp — 全部 12 行的完整实现
#include "ncbind.hpp"
#define NCB_MODULE_NAME TJS_W("win32dialog.dll")

static void InitPlugin_WIN32Dialog() {
    // 用 TJS 脚本创建 WIN32Dialog 类
    // messageBox() 方法直接调用引擎的 System.inform()
    TVPExecuteScript(TJS_W(
        "class WIN32Dialog {"
        "  function messageBox(message, caption, type) {"
        "    return !System.inform(message, caption, 2);"
        "  }"
        "}"));
}
NCB_PRE_REGIST_CALLBACK(InitPlugin_WIN32Dialog);
```

这是最激进的跨平台策略——不尝试移植原版 API，而是用最小可行实现（Minimum Viable Implementation）替代。代价是功能比原版大幅简化（原版支持复杂的模态对话框、控件布局等，这里只保留了 `messageBox` 单个方法）。

### 3.3 适配模式选择决策图

```
新插件需要跨平台吗？
 │
 ├── 核心逻辑是纯 C++ 运算？
 │   └── 是 → 模式 A：直接编写，标准 C++17 即可
 │
 ├── 需要文件/字体/音频等系统功能？
 │   └── 是 → 模式 B：查找引擎抽象层 API（TVP* 系列函数）
 │
 ├── 需要平台原生 API（图形绘制、窗口管理等）？
 │   └── 是 → 模式 C：条件编译 + 每平台独立实现
 │
 └── 原版功能在非 Windows 上无意义？
     └── 是 → 模式 D：TJS 脚本层最小替代实现
```

## 常见错误与排查

### 错误 1：新增插件编译通过但 TJS 脚本找不到类/方法

**症状：** 插件 `.cpp` 文件已添加到 `target_sources`，编译成功，但 TJS 脚本调用时报 `member "xxx" does not exist`。

**原因：** 忘记在插件源文件中定义 `NCB_MODULE_NAME` 宏。ncbind 系统依赖此宏来标识插件模块，缺失时注册逻辑不会执行。

**解决：** 在 `#include "ncbind.hpp"` 之后添加：

```cpp
#define NCB_MODULE_NAME TJS_W("yourPlugin.dll")
```

### 错误 2：子目录插件编译成功但功能不生效

**症状：** `add_subdirectory(myplugin)` 已添加，独立静态库编译成功，但插件的 TJS 接口不存在。

**原因：** 只添加了 `add_subdirectory()` 但忘记在 `target_link_libraries` 中链接。静态库虽然编译了，但没有链接进 `krkr2plugin`，其中的全局构造函数（负责 ncbind 注册）不会被执行。

**解决：** 在 `cpp/plugins/CMakeLists.txt` 中同时添加两行：

```cmake
add_subdirectory(myplugin)
# 在 target_link_libraries 块中添加
target_link_libraries(${PROJECT_NAME} PUBLIC myplugin)
```

### 错误 3：在非 Windows 平台编译 windowEx 类插件时出现 WM_* 未定义

**症状：** Linux/macOS/Android 构建时报 `error: use of undeclared identifier 'WM_LBUTTONDOWN'`。

**原因：** 插件代码中使用了 Windows 消息常量（Window Message，Windows 操作系统中用于进程间/窗口间通信的数值标识符）但没有为非 Windows 平台定义这些常量。

**解决：** 参考 `windowEx.cpp` 的做法，在 `#ifndef _WIN32` 块中定义所需的 `WM_*` 常量：

```cpp
#ifndef _WIN32
#define WM_LBUTTONDOWN 0x0201
#define WM_LBUTTONUP   0x0202
#define WM_RBUTTONDOWN 0x0204
// 只定义你实际使用的常量
#endif
```

<summary>查看答案</summary>

```cpp

class JSONParser {

public:

    JSONParser(const ttstr& jsonString) {
        // 解析 JSON 字符串...
        // 将结果存入 parsedData
    }
    
    // 获取解析结果
    tTJSVariant getRoot() const {
        return parsedData;
    }
    
    // 获取指定路径的值
    tTJSVariant query(const ttstr& path) const {
        // 按 JSONPath 查询...
        return tTJSVariant();

NCB_REGISTER_CLASS(JSONParser) {
    // Constructor<参数类型列表>(默认参数数量)
```

### 题目 2：跨平台适配策略分析

假设你需要实现一个 `clipboard` 插件，功能是读写系统剪贴板。请分析：
1. 这属于哪种跨平台适配模式？
2. 各平台应该使用什么 API？
3. 写出 CMakeLists.txt 的条件编译结构

<details>
<summary>查看答案</summary>

**1. 模式 C：条件编译 + 替代实现**——剪贴板是操作系统级功能，每个平台有不同的 API，无法用纯 C++ 实现。

**2. 各平台 API：**

| 平台 | 剪贴板 API |
|------|-----------|
| Windows | `OpenClipboard()` / `GetClipboardData()` / `SetClipboardData()` (Win32 API) |
| Linux | X11 的 `XSetSelectionOwner()` / `XGetSelectionOwner()`，或 SDL2 的 `SDL_GetClipboardText()` |
| macOS | `NSPasteboard` (Cocoa API)，或 SDL2 的 `SDL_GetClipboardText()` |
| Android | JNI 调用 `ClipboardManager.getText()` / `ClipboardManager.setText()` |

**3. CMakeLists.txt 结构：**

```cmake
cmake_minimum_required(VERSION 3.19)
project(clipboard LANGUAGES CXX)

add_library(${PROJECT_NAME} STATIC)
target_link_libraries(${PROJECT_NAME} PRIVATE krkr2plugin)

# 公共头文件（定义跨平台接口）
target_sources(${PROJECT_NAME} PUBLIC clipboard.h)

if(WINDOWS)
    # Windows: 使用 Win32 剪贴板 API
    target_sources(${PROJECT_NAME} PRIVATE
        windows/clipboard_impl.cpp)
elseif(ANDROID)
    # Android: 通过 JNI 调用 Java ClipboardManager
    target_sources(${PROJECT_NAME} PRIVATE
        android/clipboard_impl.cpp)
else()
    # Linux/macOS: 统一使用 SDL2 剪贴板接口
    target_sources(${PROJECT_NAME} PRIVATE
        sdl/clipboard_impl.cpp)
    # SDL2 已作为引擎依赖存在，无需额外 find_package
endif()
```

注意使用的是项目自定义的 `WINDOWS` / `ANDROID` 变量，而不是 CMake 内置的 `WIN32` / `UNIX`。
