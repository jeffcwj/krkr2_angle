# 无源码插件分析

> **所属模块：** M09-插件系统与开发
> **前置知识：** [已实现插件总览](./01-已实现插件总览.md)、[P09-逆向工程入门](../../P09-逆向工程入门/README.md)
> **预计阅读时间：** 25 分钟

## 本节目标

读完本节后，你将能够：
1. 列出原版 KiriKiri 插件清单中所有标记为 `nan`（无源码）的插件
2. 区分"已被 KrKr2 自行实现"和"尚未实现"的无源码插件
3. 分析每个无源码插件的功能定位和游戏兼容性影响
4. 理解逆向分析 DLL 插件接口的基本方法
5. 评估各无源码插件的实现优先级

## 术语预览

本节将涉及以下术语，先有个印象，正文中会逐一详细讲解：

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| 无源码插件 | Closed-source Plugin | 原版 KiriKiri 社区中未公开源代码的 `.dll` 插件，只能从二进制文件逆向分析其功能 |
| 导出表 | Export Table | PE 格式（Windows 可执行文件格式）DLL 中记录的函数名列表，标识了 DLL 对外提供哪些可调用的函数 |
| TJS 接口签名 | TJS Interface Signature | 插件注册到 TJS 引擎的类名、方法名和参数类型，游戏脚本通过这些签名调用插件功能 |
| 兼容性影响 | Compatibility Impact | 缺少某个插件时对游戏运行的影响程度——从完全无法启动到仅缺少次要功能 |
| stub 替代 | Stub Replacement | 为缺失插件创建空壳实现（函数签名正确但内部不做任何事），避免游戏因找不到插件而崩溃 |

## 二、KrKr2 已自行实现的无源码插件

虽然没有原版源码，KrKr2 通过逆向分析 DLL 接口或独立开发，成功实现了 3 个无源码插件。

### 2.1 getabout — 关于信息

**原版功能：** 提供 `System.getAbout()` 等方法返回引擎版本号、编译日期、操作系统信息等。

**KrKr2 实现方式：** 完全独立开发，不依赖原版代码。返回的信息反映 KrKr2 自身的版本和编译环境，而非原版 KiriKiri 的信息。

```cpp
// cpp/plugins/getabout.cpp — 简化示意
#include "ncbind.hpp"

#define NCB_MODULE_NAME TJS_W("getabout.dll")

struct GetAbout {
    // 返回引擎版本信息字符串
    static ttstr getAbout() {
        return ttstr(TJS_W("KrKr2 Emulator v2.x"));
    }
};

NCB_ATTACH_CLASS(GetAbout, System) {
    NCB_METHOD(getAbout);
}
```

**与原版差异：** 返回的版本字符串格式可能与原版不同。大多数游戏只是在"关于"对话框中显示此信息，不依赖具体格式，因此影响极小。

### 2.2 xp3filter — XP3 加密过滤器

**原版功能：** XP3 存档格式支持自定义加密。游戏开发者可以编写自己的 `xp3filter.dll` 来实现专有的加密/解密算法，保护游戏资源不被直接提取。

**KrKr2 实现方式：** 通过 `NCB_PRE_REGIST_CALLBACK` 在引擎启动时注册一个 XP3 解密过滤器函数。该函数在每次读取 XP3 存档中的文件时被调用，对加密数据进行解密。

```cpp
// cpp/plugins/xp3filter.cpp — 核心结构（简化示意）
#include "ncbind.hpp"

#define NCB_MODULE_NAME TJS_W("xp3filter.dll")

// XP3 解密过滤器回调函数
// hash: 文件在存档中的哈希标识
// offset: 当前数据块在文件中的偏移量
// buffer: 待解密的数据缓冲区
// bufferlen: 数据长度
static void XP3DecryptFilter(
    tTVPXP3ArchiveExtractionFilter_t hash,
    tjs_uint64 offset,
    void* buffer,
    tjs_uint bufferlen) {
    // 根据 hash 和 offset 对 buffer 中的数据进行异或解密
    // 不同游戏的加密算法不同——这里实现了常见的几种
    unsigned char* p = (unsigned char*)buffer;
    for(tjs_uint i = 0; i < bufferlen; i++) {
        p[i] ^= (unsigned char)(hash ^ (offset + i));
    }
}

static void PreRegistCallback() {
    TVPSetXP3ArchiveExtractionFilter(XP3DecryptFilter);
}
NCB_PRE_REGIST_CALLBACK(PreRegistCallback);
```

**与原版差异：** 每款游戏可能使用独特的 XP3 加密算法。KrKr2 当前实现覆盖了常见的加密方式（简单异或、基于文件名的密钥派生等），但不保证支持所有游戏。遇到不支持的加密时，需要通过逆向分析该游戏的 `xp3filter.dll` 来提取加密算法。

### 2.3 motionplayer — E-mote 动画播放

**原版功能：** 播放 M2 公司开发的 E-mote 格式动画（一种 2D 人物动态表现技术，可以让静态立绘做出眨眼、摇头、呼吸等动作）。

**KrKr2 实现方式：** 使用 M2 公司的 SDK（Software Development Kit，软件开发工具包）进行移植。`cpp/plugins/motionplayer/` 目录包含完整的 E-mote 运行时代码。

**注意：** 虽然 `doc/krkr2_plugins.md` 中 `motionplayer.dll` 标记为 `nan`，但 KrKr2 通过获取 M2 的 SDK 授权实现了该插件。这不是通过逆向工程实现的。

## 三、尚未实现的无源码插件逐个分析

### 3.1 TextRender.dll — 文本渲染增强（高优先级）

**功能推测：** 提供比引擎内置 `Font.drawText()` 更丰富的文本排版功能。可能包括：
- 文字阴影（Drop Shadow）和描边（Outline/Stroke）
- 渐变色填充（Gradient Fill）
- 纵向文字排版（Vertical Text Layout）
- 富文本标签（如 `<b>粗体</b>`、`<color=red>红色</color>`）

**为什么高优先级：** 文本显示是视觉小说的核心功能。如果游戏依赖 `TextRender.dll` 的特殊渲染效果，缺失该插件会导致文字显示异常——可能只是失去阴影效果（轻微），也可能导致整个对话框渲染失败（严重）。

**逆向分析思路：**

```
获取包含 TextRender.dll 的游戏
       ↓
用 PE 分析工具查看导出表
       ↓
提取 TJS 接口签名（类名 + 方法名）
       ↓
在 TJS 脚本中搜索调用方式
       ↓
通过动态调试观察输入输出行为
       ↓
实现兼容接口
```

**实现建议：** 可基于 `layerExDraw` 的绘图能力构建——`layerExDraw` 已经支持抗锯齿文本渲染（通过 GDI+/libgdiplus），`TextRender` 的功能很可能是在此基础上的高层封装。

### 3.2 PackinOne.dll — 资源打包（中优先级）

**功能推测：** 一种替代 XP3 的资源打包格式。名字 "PackinOne" 暗示其功能是"将所有资源打包为一个文件"。可能是某些游戏开发者自定义的存档格式。

**兼容性影响：** 使用 PackinOne 格式的游戏无法在 KrKr2 上读取资源文件，导致完全无法运行。但由于使用此格式的游戏数量有限，优先级为中等。

**实现建议：** 需要获取使用此格式的游戏样本，逆向分析打包文件的格式结构（文件头、索引表、数据区），然后在引擎的存档系统中添加新的存档格式处理器（参照 `XP3Archive`、`ZIPArchive` 的实现模式）。

### 3.3 layerExAlpha.dll — Alpha 通道扩展（中优先级）

**功能推测：** 提供高级的 Alpha 通道操作功能，可能包括：
- Alpha 通道的数学运算（加减乘除、阈值处理）
- Alpha 蒙版（Mask）的应用和混合
- Alpha 渐变生成（线性渐变、径向渐变的透明度过渡）
- Alpha 通道提取和注入（从 RGB 图像分离/合并 Alpha）

**兼容性影响：** 部分游戏使用 Alpha 扩展来实现特殊的视觉效果（如模糊边缘、光晕效果）。缺失时这些效果会消失，但通常不影响游戏核心流程。

**实现建议：** 基于 `LayerExBase.cpp` 的像素操作基础函数来构建。Alpha 操作本质上是逐像素的数学运算，实现难度不高，关键是确定原版的 TJS 接口签名。

### 3.4 lzfs.dll — LZ 虚拟文件系统（中优先级）

**功能推测：** 提供一种基于 LZ 系列压缩算法的虚拟文件系统。游戏资源文件经过 LZ 压缩后存储，`lzfs.dll` 在引擎层面透明地进行解压，让上层代码像读取普通文件一样读取压缩数据。

**兼容性影响：** 与 `PackinOne` 类似——使用此格式的游戏完全无法运行。但使用 LZFS 的游戏相对少见。

**实现建议：** LZ 压缩算法家族有成熟的开源实现（如 LZ4、LZMA），关键是确定具体使用的压缩变体。可以通过分析包含 `lzfs.dll` 的游戏的数据文件来识别压缩格式（文件头的 Magic Number，即文件格式标识字节序列）。

### 3.5 util_generic.dll / util_graph.dll / util_system.dll — 工具集三件套

这三个 DLL 是一组通用工具库，分别覆盖不同的功能域：

**util_generic.dll（通用工具）：**
- 字符串操作（正则表达式、格式化、编码转换）
- 数值工具（随机数增强、数学函数扩展）
- 日期时间处理（格式化、时区转换）
- 文件路径操作（拼接、规范化、扩展名提取）

**util_graph.dll（图形工具）：**
- 图像缩放（支持多种插值算法：最近邻、双线性、双三次）
- 色调/饱和度/亮度（HSL）调整
- 图像滤镜（模糊、锐化、边缘检测）
- 色彩空间转换（RGB↔HSL、RGB↔YUV）

**util_system.dll（系统工具）：**
- 系统信息查询（CPU 型号、内存大小、屏幕分辨率）
- 环境变量读取
- 注册表访问（Windows 特有）
- 进程管理（启动外部程序、获取进程 ID）

**兼容性影响：** 取决于游戏具体调用了哪些方法。如果只用了其中一两个函数（如 `getScreenWidth()`），可以通过 stub 快速解决；如果深度依赖了大量工具函数，则工作量较大。

**实现建议：** 优先采用 "按需 stub" 策略——先创建空壳插件注册所有已知方法名，遇到游戏调用时再逐个实现。工具函数通常是独立的，不需要一次性全部实现。

```cpp
// util_generic 的按需 stub 示例
#include "ncbind.hpp"

#define NCB_MODULE_NAME TJS_W("util_generic.dll")

struct UtilGeneric {
    // 已实现的方法
    static ttstr formatDate(const ttstr& format) {
        // 实际实现日期格式化...
        return ttstr();
    }
    
    // stub 方法 — 遇到游戏调用时再实现
    static tTJSVariant regexMatch(const ttstr& str,
                                  const ttstr& pattern) {
        TVPAddLog(TJS_W("WARNING: util_generic.regexMatch() "
                        "not implemented"));
        return tTJSVariant();  // 返回 void/undefined
    }
};

NCB_ATTACH_CLASS(UtilGeneric, System) {
    NCB_STATICMETHOD(formatDate);
    NCB_STATICMETHOD(regexMatch);
}
```

### 3.6 kirikiroid2.dll — KiriKiroid2 专属扩展（低优先级）

**功能推测：** KiriKiroid2 是另一个 KiriKiri 移动端移植项目。这个 DLL 可能包含 KiriKiroid2 特有的功能：
- 触摸手势识别（双指缩放、滑动翻页）
- 移动端特有的 UI 适配（虚拟按键、状态栏控制）
- KiriKiroid2 的配置管理接口

**兼容性影响：** 极低。KrKr2 有自己独立的移动端架构（基于 Cocos2d-x + SDL2），不需要 KiriKiroid2 的扩展。只有从 KiriKiroid2 迁移过来的游戏脚本中可能残留对此 DLL 的引用。

**实现建议：** 创建一个空壳 stub 即可——注册类名但所有方法返回默认值。如果有游戏依赖其特定方法，再按需实现。

### 3.7 multiimage.dll — 多图像处理（低优先级）

**功能推测：** 提供批量图像操作功能——同时对多个 Layer 执行相同的变换（批量缩放、批量色调调整、批量合成等）。

**兼容性影响：** 低——所有批量操作都可以通过 TJS 脚本循环调用单图像 API 来替代，只是性能可能较差。

**实现建议：** 除非发现有游戏严重依赖此插件的性能优化，否则用 TJS 脚本层面的 polyfill（功能填充代码，用脚本模拟原生插件的接口）即可。

## 四、逆向分析 DLL 插件的基本方法

当需要实现某个无源码插件时，第一步是确定该插件的 TJS 接口——即它注册了哪些类、方法和属性。以下是推荐的逆向分析流程。

### 4.1 工具准备

| 工具 | 用途 | 获取方式 |
|------|------|---------|
| PE Explorer / CFF Explorer | 查看 DLL 的导出表、导入表、资源 | 免费下载 |
| IDA Free / Ghidra | 反汇编和反编译 DLL 代码 | 免费（IDA Free 限 x86/x64） |
| x64dbg / WinDbg | 动态调试——在运行时观察函数调用 | 免费 |
| Dependency Walker / dumpbin | 查看 DLL 依赖关系和导出函数 | Visual Studio 自带 dumpbin |
| TJS 调试器 | 在 TJS 脚本层面测试插件接口 | KiriKiri 自带 |

### 4.2 静态分析流程

**第一步：提取导出表**

KiriKiri 插件 DLL 通常导出一个入口函数 `V2Link`（KiriKiri2 插件 API v2 的链接函数），通过该函数向引擎注册插件功能：

```bash
# 使用 Visual Studio 自带的 dumpbin 工具
dumpbin /exports TextRender.dll

# 输出示例：
#   ordinal  name
#          1  V2Link
#          2  V2Unlink
```

如果导出的是 `V2Link` / `V2Unlink` 这对函数，说明这是标准的 KiriKiri2 插件。

**第二步：反编译 V2Link 函数**

在 IDA/Ghidra 中打开 DLL，定位 `V2Link` 函数。该函数内部会调用 KiriKiri 的插件 API 来注册类和方法。关键是找到注册调用中的字符串参数——这些字符串就是 TJS 接口名：

```cpp
// V2Link 内部的典型模式（反编译伪代码）
int V2Link(iTVPFunctionExporter* exporter) {
    // 获取 TJS 引擎接口
    TVPGetFunctionExporter(exporter);
    
    // 注册 TJS 类 — 字符串 "TextRender" 就是类名
    TVPRegisterPlugin("TextRender", &TextRender_factory);
    
    // 注册方法 — 字符串 "drawText" 就是方法名
    // 参数和返回值类型需要从调用约定推断
    return 0;
}
```

**第三步：提取字符串常量**

在反汇编工具中搜索所有 Unicode 宽字符串（KiriKiri 使用 UTF-16 编码）。插件注册的类名、方法名、属性名都是字符串常量，可以批量提取：

```bash
# 用 strings 工具提取 DLL 中的字符串
strings -e l TextRender.dll | sort | uniq

# -e l 表示提取 UTF-16 Little-Endian 编码的字符串
# 输出中带有 TJS 接口风格的字符串（如 drawText、setFont、
# getWidth 等）就是目标
```

### 4.3 动态分析流程

静态分析可以确定接口签名，但无法确定参数的具体含义和返回值的格式。动态分析通过在运行时观察插件的行为来补充这些信息。

**方法 1：TJS 脚本探测**

在一个使用目标插件的游戏中，编写 TJS 测试脚本来探测接口行为：

```javascript
// 探测 TextRender 插件的接口
// 先尝试创建实例
try {
    var tr = new TextRender();
    System.inform("TextRender 类存在");
    
    // 枚举所有成员
    var members = [];
    for(var name in tr) {
        members.add(name);
    }
    System.inform("成员列表: " + members.join(", "));
    
    // 尝试调用已知方法
    try {
        tr.drawText("测试", 0, 0);
        System.inform("drawText(string, int, int) 调用成功");
    } catch(e) {
        System.inform("drawText 调用失败: " + e.message);
    }
} catch(e) {
    System.inform("TextRender 创建失败: " + e.message);
}
```

**方法 2：API Hook（API 钩子，在函数调用前/后插入自定义代码的调试技术）**

使用 x64dbg 在 `V2Link` 函数上设断点，单步跟踪注册过程：

```
1. 用 x64dbg 附加到 KiriKiri 引擎进程
2. 在 TextRender.dll 的 V2Link 上设断点
3. 运行游戏，断点命中后单步执行
4. 观察每次字符串参数传递——这些就是接口名
5. 记录参数数量和类型
```

## 动手实践

按以下步骤对一个无源码插件进行初步分析：

**第一步：获取样本**

找到一个包含 `util_generic.dll`（或任意无源码插件）的 KiriKiri 游戏。游戏目录中的 `plugin/` 文件夹通常包含所有插件 DLL。

**第二步：提取导出表**

```bash
# Windows — 使用 Visual Studio Developer Command Prompt
dumpbin /exports util_generic.dll

# 或使用 PowerShell + 自定义脚本
# 查看文件大小和基本信息
Get-Item util_generic.dll | Select-Object Name, Length, CreationTime
```

**第三步：提取字符串**

```bash
# Linux/macOS（如果在 Wine 环境中分析）
strings -e l util_generic.dll > strings_output.txt

# Windows（使用 Sysinternals strings 工具）
strings -n 4 util_generic.dll > strings_output.txt

# 过滤出可能的 TJS 接口名（通常是驼峰命名的英文单词）
# Linux:
grep -E "^[a-z][a-zA-Z]+$" strings_output.txt | sort | uniq

# Windows:
findstr /r "^[a-z][a-zA-Z]*$" strings_output.txt
```

**第四步：创建 stub 插件**

根据提取到的接口名，在 KrKr2 中创建空壳实现：

```cpp
// cpp/plugins/utilGeneric.cpp — stub 实现
#include "ncbind.hpp"
#include <spdlog/spdlog.h>

#define NCB_MODULE_NAME TJS_W("util_generic.dll")

struct UtilGeneric {
    // 每个 stub 方法都打印警告日志
    static tTJSVariant notImplemented(const ttstr& methodName) {
        spdlog::warn("util_generic.{} not implemented",
                     methodName.AsStdString());
        return tTJSVariant();
    }
    
    // 根据提取到的方法名逐个添加
    static ttstr formatString(const ttstr& fmt) {
        spdlog::warn("util_generic.formatString() stub called");
        return fmt;  // 直接返回原字符串作为最简替代
    }
    
    static tjs_int getTickCount() {
        // 这个方法可以直接实现
        auto now = std::chrono::steady_clock::now();
        auto ms = std::chrono::duration_cast<
            std::chrono::milliseconds>(
            now.time_since_epoch());
        return static_cast<tjs_int>(ms.count());
    }
};

NCB_ATTACH_CLASS(UtilGeneric, System) {
    NCB_STATICMETHOD(formatString);
    NCB_STATICMETHOD(getTickCount);
}
```

**第五步：注册并构建**

在 `cpp/plugins/CMakeLists.txt` 的 `target_sources` 中添加 `utilGeneric.cpp`，然后构建：

```bash
cmake --build --preset="Windows Debug Build" --target krkr2plugin
```

## 常见错误与排查

### 错误 1：从 DLL 提取的字符串包含大量噪声

**症状：** `strings` 命令输出了数百个无意义的字符串片段。

**原因：** `strings` 工具默认提取所有可打印字符序列，包括代码中的偶然匹配。

**解决：** 增大最小长度阈值，并专注于 TJS 风格的命名：

```bash
# 只提取长度 >= 6 的字符串，减少噪声
strings -n 6 -e l plugin.dll > strings_raw.txt

# 过滤：TJS 方法名通常是驼峰命名，以小写字母开头
# Linux:
grep -E "^[a-z][a-zA-Z]{4,}$" strings_raw.txt

# Windows:
findstr /r "^[a-z][a-zA-Z][a-zA-Z][a-zA-Z][a-zA-Z]" strings_raw.txt
```

### 错误 2：stub 插件注册后游戏仍然报错找不到方法

**症状：** 创建了 stub 插件，游戏不再报 "plugin not found"，但改为报 `member "xxx" does not exist`。

**原因：** 游戏调用了你没有注册的方法。stub 需要覆盖游戏实际使用的所有方法名。

**解决：** 在游戏的 TJS 脚本中搜索对该插件的调用：

```bash
# 在游戏脚本中搜索插件方法调用
# 如果游戏脚本未加密：
findstr /s /i "util_generic\|UtilGeneric" game_scripts/*.tjs

# 如果游戏脚本在 XP3 中：
# 先用 tools/xp3/ 工具提取，再搜索
```

### 错误 3：逆向分析时 IDA/Ghidra 无法解析插件的 C++ 符号

**症状：** 反编译结果只有 `sub_XXXX` 形式的函数名，没有可读符号。

**原因：** Release 版本的 DLL 通常去除了调试符号（Symbol Stripping，从二进制文件中删除调试信息以减小文件大小）。

**解决：** 
1. 利用 `V2Link` / `V2Unlink` 导出函数作为入口点开始分析
2. 在 `V2Link` 中寻找字符串引用——这些字符串通常是 TJS 接口名
3. 使用 Ghidra 的 "Defined Strings" 功能批量查看所有字符串常量
4. 通过交叉引用（Cross Reference，哪些代码引用了某个地址/函数）追踪字符串的使用位置

## 本节小结

- 原版 KiriKiri 有 **11 个**无源码插件（`doc/krkr2_plugins.md` 中标记为 `nan`）
- KrKr2 已自行实现了其中 **3 个**：`getabout`（独立开发）、`xp3filter`（逆向+重写）、`motionplayer`（M2 SDK 授权）
- 剩余 **8 个**未实现，按影响优先级：`TextRender`（高）> `PackinOne`/`layerExAlpha`/`lzfs`/`util_*`（中）> `kirikiroid2`/`multiimage`（低）
- 逆向分析的核心目标是确定 **TJS 接口签名**（类名 + 方法名 + 参数类型），可通过静态分析（导出表 + 字符串提取）和动态分析（TJS 探测 + API Hook）完成
- 推荐的实现策略是 **"按需 stub"**——先创建空壳注册所有已知方法，遇到游戏实际调用时再逐个实现
- `xp3filter` 是最特殊的无源码插件——每款游戏可能有独特的加密算法，需要逐游戏逆向

## 练习题与答案

### 题目 1：无源码插件优先级评估

假设你发现一款游戏在启动时报错 `Plugin "lzfs.dll" not found`，并且该游戏的所有资源文件都是 `.lzf` 扩展名。请回答：
1. 这属于哪种兼容性影响级别？
2. 仅创建 stub 能否解决问题？
3. 应该如何处理？

<details>
<summary>查看答案</summary>

**1. 高影响级别。** 虽然 `lzfs.dll` 整体被评为"中优先级"，但对于这款特定游戏来说是致命的——所有资源文件都使用 LZF 格式，没有该插件就无法读取任何资源。

**2. 不能。** stub 只能避免"插件未找到"的报错，但无法提供实际的文件解压功能。游戏在尝试读取 `.lzf` 文件时仍会失败（数据解压返回空或乱码）。

**3. 处理步骤：**

```
第 1 步：从其他渠道获取 lzfs.dll 样本
第 2 步：用 dumpbin/strings 提取接口签名
第 3 步：获取一个 .lzf 样本文件
第 4 步：分析 .lzf 文件头格式（Magic Number、压缩算法标识）
第 5 步：尝试已知的 LZ 算法解压（LZ4、LZMA、LZF/LibLZF）
第 6 步：如果已知算法不匹配，用 IDA/Ghidra 反编译 lzfs.dll 中的解压函数
第 7 步：在 KrKr2 的存档系统中添加 LZF 格式处理器
        （参照 cpp/core/base/XP3Archive.cpp 的模式）
第 8 步：验证游戏资源能否正确读取和显示
```

</details>

### 题目 3：xp3filter 加密算法分析

KrKr2 的 `xp3filter.cpp` 使用了简单的异或加密。请解释：
1. 为什么简单异或加密不够安全？
2. 如果某游戏使用了更复杂的加密（如 AES），xp3filter 的注册机制需要改变吗？
3. 如何确定一款游戏使用的具体 XP3 加密算法？

<details>
<summary>查看答案</summary>

**1. 简单异或加密不安全的原因：**
- **已知明文攻击**：如果知道文件的部分原始内容（如 PNG 文件头 `89 50 4E 47`），可以通过 `密文 XOR 明文 = 密钥` 直接恢复密钥
- **密钥重复**：异或加密的密钥是从 hash 和 offset 推导的，一旦密钥模式被识别，整个存档都可以解密
- **无扩散性**：修改一个字节的密钥只影响一个字节的密文，不像 AES 等块密码（Block Cipher，将数据分成固定大小的块进行加密的算法）有扩散效应

**2. 注册机制不需要改变。** XP3 过滤器的注册接口是固定的——`TVPSetXP3ArchiveExtractionFilter(callback)`。不管加密算法多复杂（异或、AES、自定义算法），都只需要提供一个回调函数，该函数接收 `(hash, offset, buffer, length)` 参数并在 `buffer` 上原地解密。变化的只是回调函数内部的解密逻辑。

**3. 确定游戏加密算法的方法：**

```
方法 A：静态分析（推荐）
  1. 提取游戏的 xp3filter.dll
  2. 用 IDA/Ghidra 反编译过滤器回调函数
  3. 识别算法特征（循环结构、常量表、API 调用）
  4. 常见模式：
     - 简单异或 → 看到单层 XOR 循环
     - 多轮异或 → 看到嵌套循环或查找表
     - AES → 看到 S-Box 常量表或 AES-NI 指令
     - 自定义 → 需要完整分析算法流程

方法 B：黑盒测试
  1. 从 XP3 存档中提取一个已知格式的文件（如 .png）
  2. 比较加密数据和预期的文件头
  3. 推断加密模式（异或密钥、偏移依赖等）
```

</details>
