### 2.4 格式化豁免区域

项目中有两个目录禁用了格式化（目录内放置了 `DisableFormat: true` 的 `.clang-format` 文件）：

| 豁免目录 | 原因 |
|----------|------|
| `cpp/external/` | 第三方供应商代码（libbpg、minizip），不应修改其格式 |
| `cpp/core/plugin/` | ncbind 框架代码使用大量宏嵌套，自动格式化会破坏可读性 |

**重要**：你的新插件代码放在 `cpp/plugins/` 目录下，**不在豁免范围内**，必须通过格式检查。

### 2.5 本地格式化命令

在提交代码前，务必在本地运行格式化检查：

```bash
# Windows（PowerShell）——格式化你的新插件文件
clang-format -i --verbose cpp/plugins/my_plugin.cpp

# Linux / macOS——批量格式化所有修改过的文件
git diff --name-only --diff-filter=ACMR '*.cpp' '*.h' '*.hpp' | \
    xargs clang-format -i --verbose

# 验证格式化结果（应该没有 diff 输出）
git diff
# 如果有输出，说明格式化改动了你的代码，需要重新 commit
```

## 三、插件文件结构规范

根据插件复杂度的不同，KrKr2 项目有两种组织方式。选择哪种取决于你的插件是否需要多文件和平台特定实现。

### 3.1 简单插件（单文件）

如果你的插件逻辑简单、没有平台差异、只需一个 `.cpp` 文件，直接放在 `cpp/plugins/` 根目录下：

```
cpp/plugins/
├── CMakeLists.txt          ← 在这里添加你的文件
├── my_plugin.cpp           ← 你的新插件
├── scriptsEx.cpp           ← 已有的简单插件示例
├── csvParser.cpp
└── ...
```

**注册步骤（仅需修改 1 个文件）**：

```cmake
# cpp/plugins/CMakeLists.txt
# 在 target_sources 块中添加你的文件（按字母序插入）
target_sources(${PROJECT_NAME} PUBLIC
    addFont.cpp
    csvParser.cpp
    dirlist.cpp
    # ... 已有文件 ...
    my_plugin.cpp              # ← 添加这一行
    # ... 已有文件 ...
    xp3filter.cpp
)
```

完整的单文件插件模板：

```cpp
// cpp/plugins/my_plugin.cpp
// KrKr2 插件：MyPlugin — [一句话功能描述]

#include "ncbind.hpp"          // ncbind 绑定框架头文件

#include <string>              // 标准库头文件放在项目头文件之后
#include <vector>

// ============================================================
// 第一部分：核心逻辑实现
// ============================================================

namespace {  // 匿名命名空间，防止符号泄漏到全局

    // 插件的核心数据结构
    struct PluginData {
        std::string name;      // 名称
        int value = 0;         // 值
    };

}  // namespace

// ============================================================
// 第二部分：TJS2 绑定类
// ============================================================

class MyPluginClass {
public:
    MyPluginClass() = default;  // 默认构造函数

    // TJS 可调用的方法
    static tjs_error TJS_INTF_METHOD process(
        tTJSVariant *result,   // 返回值
        tjs_int numparams,     // 参数个数
        tTJSVariant **param,   // 参数数组
        iTJSDispatch2 *objthis // this 对象
    ) {
        if(numparams < 1) {
            return TJS_E_BADPARAMCOUNT;  // 参数不足
        }
        // 处理逻辑
        ttstr input(*param[0]);  // 第一个参数转字符串
        // ... 核心处理 ...
        if(result) {
            *result = tTJSVariant(input);  // 设置返回值
        }
        return TJS_S_OK;  // 成功
    }
};

// ============================================================
// 第三部分：注册宏
// ============================================================

// NCB_PRE_REGIST_CALLBACK 在引擎启动时自动调用
NCB_PRE_REGIST_CALLBACK(registerMyPlugin) {
    // 向 TJS2 全局命名空间注册函数/类
    tTJSVariant val;
    TVPExecuteExpression(
        TJS_W("class MyPlugin {"
              "  function process(input) { /* native */ }"
              "}"),
        &val
    );
}
```

### 3.2 复杂插件（子目录）

如果你的插件需要多个源文件、头文件，或者有平台特定实现，应该创建子目录：

```
cpp/plugins/
├── CMakeLists.txt              ← 父级：add_subdirectory + link
└── my_complex_plugin/          ← 你的插件子目录
    ├── CMakeLists.txt          ← 插件自己的构建配置
    ├── my_complex_plugin.cpp   ← 主逻辑
    ├── my_complex_plugin.h     ← 头文件
    ├── windows/                ← Windows 平台特定代码
    │   └── platform_impl.cpp
    └── general/                ← 非 Windows 平台代码
        └── platform_impl.cpp
```

**子目录插件的 CMakeLists.txt 模板**：

```cmake
# cpp/plugins/my_complex_plugin/CMakeLists.txt
cmake_minimum_required(VERSION 3.19)
project(myComplexPlugin)

# 创建静态库目标
add_library(${PROJECT_NAME} STATIC)

# 公共源文件（所有平台都编译）
target_sources(${PROJECT_NAME} PUBLIC
    my_complex_plugin.cpp
    my_complex_plugin.h
)

# 平台条件编译（使用 KrKr2 自定义变量，不是 CMake 内置变量）
if(WINDOWS)
    # Windows 使用 Win32 API 实现
    target_sources(${PROJECT_NAME} PRIVATE
        windows/platform_impl.cpp
    )
else()
    # Linux/macOS/Android 使用通用实现
    target_sources(${PROJECT_NAME} PRIVATE
        general/platform_impl.cpp
    )
    # 如果通用实现需要第三方库
    # find_package(SomeLib CONFIG REQUIRED)
    # target_link_libraries(${PROJECT_NAME} PRIVATE SomeLib::SomeLib)
endif()

# 链接到核心库（获取 ncbind 头文件等）
target_link_libraries(${PROJECT_NAME} PRIVATE krkr2plugin)
```

**父级 CMakeLists.txt 的修改（两行缺一不可）**：

```cmake
# cpp/plugins/CMakeLists.txt — 在已有的 add_subdirectory 块中添加

# 第一行：添加子目录
add_subdirectory(my_complex_plugin)  # ← 必须添加

# 第二行：在 target_link_libraries 块中链接
target_link_libraries(${PROJECT_NAME} PUBLIC
        psdfile
        psbfile
        layerExDraw
        motionplayer
        fstat
        myComplexPlugin              # ← 必须添加（名称与子目录 project() 一致）
)
```

> **常见陷阱**：只加了 `add_subdirectory` 却忘了 `target_link_libraries`。结果：编译通过但插件不会被链接进主程序，运行时 TJS 脚本调用插件函数会报 "member not found" 错误，且**没有任何编译警告**。这是 KrKr2 插件贡献中最常见的"静默失败"。

### 3.3 平台变量对照表

KrKr2 使用**自定义** CMake 变量（在根 `CMakeLists.txt` 中设置），而不是 CMake 内置的 `WIN32`/`UNIX`/`APPLE`：

| KrKr2 变量 | CMake 内置等价物 | 适用平台 |
|------------|-----------------|----------|
| `WINDOWS` | `WIN32` | Windows x64 |
| `LINUX` | `UNIX AND NOT APPLE` | Linux x64 |
| `MACOS` | `APPLE` | macOS arm64 |
| `ANDROID` | `ANDROID` | Android arm64/x86_64 |

```cmake
# ✅ 正确：使用 KrKr2 自定义变量
if(WINDOWS)
    target_sources(${PROJECT_NAME} PRIVATE windows/impl.cpp)
elseif(ANDROID)
    target_sources(${PROJECT_NAME} PRIVATE android/impl.cpp)
else()
    target_sources(${PROJECT_NAME} PRIVATE general/impl.cpp)
endif()

# ❌ 错误：使用 CMake 内置变量（在 KrKr2 中可能未定义）
if(WIN32)
    # 这个变量在 KrKr2 构建系统中不一定被设置！
endif()
```

## 四、测试规范

每个新插件**必须**附带单元测试。KrKr2 使用 Catch2 v3 测试框架，测试代码统一放在 `tests/unit-tests/plugins/` 目录下。

### 4.1 测试文件结构

```
tests/
├── CMakeLists.txt              # 顶层测试配置
├── test_config.h.in            # 生成 TEST_FILES_PATH 宏
├── test_files/                 # 测试夹具（fixture）文件
│   └── plugins/                # 插件测试用的数据文件
│       └── my_plugin_data.bin  # 你的测试数据
└── unit-tests/
    └── plugins/
        ├── CMakeLists.txt      # 插件测试构建配置
        ├── main.cpp            # 自定义 main()（已有，不需修改）
        ├── psbfile-dll.cpp     # 已有的 PSB 插件测试
        └── my-plugin-test.cpp  # ← 你的新测试文件
```

### 4.2 测试文件模板

```cpp
// tests/unit-tests/plugins/my-plugin-test.cpp
// 单元测试：MyPlugin 功能验证

#include <catch2/catch_test_macros.hpp>  // Catch2 测试宏
#include "test_config.h"                 // TEST_FILES_PATH 宏

// 如果需要测试插件的内部函数（非 TJS 绑定的纯 C++ 函数）
// #include "my_plugin.h"

TEST_CASE("MyPlugin 基本功能测试", "[plugin][my-plugin]") {
    // [plugin] 和 [my-plugin] 是标签（tag），用于筛选运行特定测试
    // 例如：ctest -R "my-plugin" 只运行这个插件的测试

    SECTION("构造函数正确初始化") {
        // 测试插件对象能正确创建
        // MyPluginClass plugin;
        // REQUIRE(plugin.isValid());
        REQUIRE(true);  // 占位断言，替换为实际测试
    }

    SECTION("处理空输入返回默认值") {
        // 测试边界条件
        // auto result = plugin.process("");
        // REQUIRE(result.empty());
        REQUIRE(true);
    }
}

TEST_CASE("MyPlugin 跨平台行为一致性", "[plugin][my-plugin]") {
    SECTION("文件路径处理") {
        // 测试路径分隔符在不同平台上的处理
        std::string testPath = "data/test/file.txt";
        // 所有平台都应该能正确处理正斜杠路径
        REQUIRE(testPath.find('/') != std::string::npos);
    }
}

TEST_CASE("MyPlugin 使用测试夹具文件", "[plugin][my-plugin]") {
    // TEST_FILES_PATH 在编译时由 CMake 设置
    // 指向 tests/test_files/ 目录的绝对路径
    std::string fixturePath =
        std::string(TEST_FILES_PATH) + "/plugins/my_plugin_data.bin";

    SECTION("读取测试数据文件") {
        // 验证夹具文件存在且可读
        std::ifstream file(fixturePath, std::ios::binary);
        // 如果你的测试确实需要数据文件，取消下面的注释
        // REQUIRE(file.is_open());
        // 如果不需要数据文件，可以跳过此测试
        REQUIRE(true);
    }
}
```

### 4.3 注册新测试到 CMake

```cmake
# tests/unit-tests/plugins/CMakeLists.txt
# 在 SOURCES 列表中添加你的测试文件

set(SOURCES
        psbfile-dll.cpp
        my-plugin-test.cpp        # ← 添加这一行
)
```

CMake 的 `foreach` 循环会自动为每个 `.cpp` 文件创建独立的测试可执行文件、链接 Catch2 和 `krkr2plugin`，并通过 `catch_discover_tests()` 注册到 CTest。

### 4.4 自定义 main() 说明

KrKr2 测试**不使用** Catch2 内置的 `Catch2::Catch2WithMain`。每个测试目录有自己的 `main.cpp`，负责初始化 spdlog 日志器后再运行测试：

```cpp
// tests/unit-tests/plugins/main.cpp（已有，不需要修改）
#include <catch2/catch_session.hpp>
#include <spdlog/sinks/stdout_color_sinks.h>

int main(int argc, char *argv[]) {
    // 初始化三个 spdlog 日志器——与主程序相同
    // 这样插件代码中的 spdlog::get("plugin") 调用不会崩溃
    static auto core_logger =
        spdlog::stdout_color_mt("core");    // 核心引擎日志
    static auto tjs2_logger =
        spdlog::stdout_color_mt("tjs2");    // TJS2 脚本引擎日志
    static auto plugin_logger =
        spdlog::stdout_color_mt("plugin");  // 插件日志

    int result = Catch::Session().run(argc, argv);
    return result;
}
```

> **为什么不用 `Catch2::Catch2WithMain`？** 因为 KrKr2 的插件代码在运行时会调用 `spdlog::get("plugin")` 获取日志器。如果日志器未初始化，`get()` 返回 `nullptr`，后续的 `->info(...)` 调用会导致空指针崩溃。自定义 `main()` 确保日志器在任何测试代码执行前就已就绪。

### 4.5 运行测试

```bash

cmake --preset="Windows Debug Config"
cmake --build --preset="Windows Debug Build"
# 运行所有测试

# 只运行你的插件测试

cmake --build --preset="Linux Debug Build"
ctest --test-dir out/linux/debug --output-on-failure

# macOS

cmake --build --preset="MacOS Debug Build"
ctest --test-dir out/macos/debug --output-on-failure
ctest --test-dir out/macos/debug -R "my-plugin" --output-on-failure
```

## 五、CI 流水线与自动检查

当你将代码推送到 GitHub 并创建 PR 时，CI 会自动运行一系列检查。了解 CI 流程能帮助你在本地预先排查问题，避免反复推送修复。

### 5.1 CI 工作流总览

KrKr2 配置了 4 个 GitHub Actions 工作流（Workflow，即"自动化流水线"——在 GitHub 服务器上运行的脚本，代码提交后自动触发）：

| 工作流文件 | 触发条件 | 检查内容 |
|-----------|----------|----------|
| `code-format-check.yml` | push/PR 到 main（修改 cpp/platforms 等路径时） | clang-format 20 格式检查 |
| `build-windows.yml` | push/PR 到 main | Windows x64 构建 + 测试 |
| `build-linux.yml` | push/PR 到 main | Linux x64 构建 + 测试 |
| `build-android.yml` | push/PR 到 main | Android arm64/x86_64 构建 |

### 5.2 格式检查流水线详解

`code-format-check.yml` 是最容易触发失败的检查，因为它对格式要求非常严格：

```yaml
# .github/workflows/code-format-check.yml 核心配置
jobs:
  format-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4            # 检出代码
      - uses: DoozyX/clang-format-lint-action@v0.20
        with:
          source: 'cpp platforms'            # 检查范围：cpp/ 和 platforms/
          extensions: 'cpp,cc,h,hpp,inc'     # 检查的文件扩展名
          clangFormatVersion: 20             # 必须使用 clang-format 20
```

**关键细节**：
- **检查范围**：只检查 `cpp/` 和 `platforms/` 目录下的文件，`tests/`、`tools/` 等目录不检查
- **版本锁定**：必须使用 clang-format 20，其他版本的格式化结果可能不同
- **触发路径**：只有修改了 `cpp/`、`platforms/`、`cmake/`、`vcpkg/`、`.github/workflows/` 等路径时才触发

### 5.3 本地模拟 CI 检查清单

在创建 PR 前，按以下清单逐项确认：

```bash
# 1. 格式检查（最重要！）
clang-format --version  # 确认版本是 20.x.x
clang-format -i --verbose cpp/plugins/my_plugin.cpp
git diff  # 应该没有输出

# 2. Windows 构建
cmake --preset="Windows Debug Config"
cmake --build --preset="Windows Debug Build"

# 3. Linux 构建（可用 Docker）
docker build -f dockers/linux.Dockerfile -t linux-builder .
# 或直接在 Linux 环境中：
cmake --preset="Linux Debug Config"
cmake --build --preset="Linux Debug Build"

# 4. 测试通过
ctest --test-dir out/<platform>/debug --output-on-failure

# 5. Android 构建（可选，但强烈建议）
docker build -f dockers/android.Dockerfile -t android-builder .
```

## 六、PR 模板与提交规范

### 6.1 Git 提交消息规范

每个 commit（提交）的消息应该简洁、准确地描述改动内容。KrKr2 采用常见的前缀分类法：

```
<类型>: <简短描述>

[可选的详细说明]
```

常用类型前缀：

| 前缀 | 含义 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat: add TextRender plugin for custom font rendering` |
| `fix` | 修复 Bug | `fix: resolve crash in csvParser with empty input` |
| `refactor` | 重构（不改变功能） | `refactor: simplify windowEx platform detection logic` |
| `test` | 添加或修改测试 | `test: add unit tests for psdfile layer parsing` |
| `docs` | 文档变更 | `docs: update krkr2_plugins.md with TextRender status` |
| `chore` | 构建/配置变更 | `chore: add TextRender to plugins CMakeLists.txt` |

**完整的插件贡献通常需要 2-3 个 commit**：

```bash
# 第一个 commit：插件核心实现
git add cpp/plugins/textrender.cpp
git add cpp/plugins/CMakeLists.txt
git commit -m "feat: add TextRender plugin for custom font rendering

Implements the TextRender plugin that was previously only available
as a closed-source DLL in original KiriKiri. Supports:
- Custom font file loading via TVPEnumFontsProc
- Vertical text rendering for Japanese
- Shadow and outline effects

Closes #42"

# 第二个 commit：单元测试
git add tests/unit-tests/plugins/textrender-test.cpp
git add tests/unit-tests/plugins/CMakeLists.txt
git commit -m "test: add TextRender plugin unit tests"

# 第三个 commit：文档更新
git add doc/krkr2_plugins.md
```

## 动手实践

按以下步骤模拟一次完整的插件贡献流程（使用一个最小化的"Hello Plugin"作为练习）：

**步骤 1：创建插件源文件**

```cpp
// 创建文件：cpp/plugins/hello_plugin.cpp
// 练习用插件：向 TJS2 全局空间注册一个 helloFromPlugin() 函数

#include "ncbind.hpp"       // ncbind 绑定框架
#include <spdlog/spdlog.h>  // 日志库
#include <string>

// 核心逻辑：拼接问候语
static std::string buildGreeting(
    const std::string &name) {
    return "Hello from KrKr2 plugin, " + name + "!";
}

// TJS2 可调用的原生函数
static tjs_error TJS_INTF_METHOD tjs_helloFromPlugin(
    tTJSVariant *result,    // 返回值
    tjs_int numparams,      // 参数个数
    tTJSVariant **param,    // 参数数组
    iTJSDispatch2 *objthis  // this 对象
) {
    if(numparams < 1) {
        return TJS_E_BADPARAMCOUNT;  // 参数不足
    }
    // 将 TJS 参数转换为 C++ 字符串
    ttstr name(*param[0]);
    std::string nameUtf8 = name.AsNarrowStdString();
    // 调用核心逻辑
    std::string greeting = buildGreeting(nameUtf8);
    // 记录日志
    spdlog::get("plugin")->info(
        "helloFromPlugin called: {}", nameUtf8);
    // 设置返回值
    if(result) {
        *result = tTJSVariant(
            ttstr(greeting.c_str()));
    }
    return TJS_S_OK;
}

// 插件注册——引擎启动时自动调用
NCB_PRE_REGIST_CALLBACK(registerHelloPlugin) {
    tTJSVariant val;
    TVPExecuteExpression(
        TJS_W("function helloFromPlugin(name) {"
              "  return name;"
              "}"),
        &val);
    spdlog::get("plugin")->info(
        "Hello plugin registered");
}
```

**步骤 2：修改 CMakeLists.txt**

```cmake
# 编辑 cpp/plugins/CMakeLists.txt
# 在 target_sources 块中按字母序添加：
target_sources(${PROJECT_NAME} PUBLIC
    # ... 已有文件 ...
    hello_plugin.cpp          # ← 添加这一行
    # ... 已有文件 ...
)
```

**步骤 3：编写单元测试**

```cpp
// tests/unit-tests/plugins/hello-plugin-test.cpp
#include <catch2/catch_test_macros.hpp>

// 测试核心逻辑（不依赖 TJS 引擎）
TEST_CASE("Hello plugin 问候语拼接",
          "[plugin][hello-plugin]") {
    SECTION("正常输入") {
        std::string name = "World";
        std::string expected =
            "Hello from KrKr2 plugin, World!";
        std::string result =
            "Hello from KrKr2 plugin, " + name + "!";
        REQUIRE(result == expected);
    }

    SECTION("空字符串输入") {
        std::string name;
        std::string result =
            "Hello from KrKr2 plugin, " + name + "!";
        REQUIRE(result ==
                "Hello from KrKr2 plugin, !");
    }

    SECTION("Unicode 输入") {
        std::string name = "世界";
        std::string result =
            "Hello from KrKr2 plugin, " + name + "!";
        REQUIRE(result.find("世界") !=
                std::string::npos);
    }
}
```

## 常见错误与排查

### 错误 1：CI 格式检查失败

**现象**：PR 提交后 `Code Format Check` 工作流显示红叉，日志显示格式差异。

**原因**：

| 可能原因 | 排查方法 |
|----------|----------|
| 本地 clang-format 版本不是 20 | `clang-format --version` 检查版本号 |
| 提交前忘了运行格式化 | `clang-format -i <文件>` 后查看 `git diff` |
| 编辑器自动格式化覆盖了 clang-format 结果 | 检查编辑器（VS Code/CLion）的格式化配置，关闭或指向项目 `.clang-format` |

**解决方案**：

```bash
# 重新格式化所有修改过的文件
git diff --name-only HEAD~1 '*.cpp' '*.h' '*.hpp' | \
    xargs clang-format -i

# 追加提交格式修复
git add -u
git commit -m "style: fix clang-format issues"
git push
```

### 错误 2：插件注册了但 TJS 脚本找不到

**现象**：编译通过，运行时 TJS 脚本调用 `MyPlugin.method()` 报 `member "method" not found`。

**原因与排查**：

1. **子目录插件缺少 `target_link_libraries`**——最常见原因。检查 `cpp/plugins/CMakeLists.txt` 是否同时有 `add_subdirectory(xxx)` 和 `target_link_libraries(... xxx)`
2. **NCB 宏拼写错误**——`NCB_PRE_REGIST_CALLBACK` 不是 `NCB_PRE_REGISTER_CALLBACK`（少了一个 `E`）
3. **TJS 类名与 C++ 注册名不一致**——`NCB_REGISTER_CLASS(MyPlugin)` 注册的名字必须与 TJS 脚本中调用的名字完全一致（区分大小写）

```cpp
// ✅ 正确：名字完全一致
NCB_REGISTER_CLASS(MyPlugin) {
    // TJS: var obj = new MyPlugin();
}

// ❌ 错误：大小写不一致
NCB_REGISTER_CLASS(Myplugin) {
    // TJS: var obj = new MyPlugin(); ← 找不到！
}
```

### 错误 3：测试编译通过但运行时崩溃

**现象**：`ctest` 运行时测试可执行文件崩溃（segfault 或 abort），没有有用的错误信息。

**原因**：插件代码调用了 `spdlog::get("plugin")->info(...)` 但日志器未初始化。在正常运行时由引擎初始化，在测试中由 `main.cpp` 初始化。

**排查步骤**：

```bash
# 1. 确认测试链接了正确的 main.cpp
# 检查 tests/unit-tests/plugins/CMakeLists.txt
# foreach 中应该有：
#   add_executable(${name} ${name}.cpp main.cpp)
#                                      ^^^^^^^^ 确保包含

# 2. 单独运行崩溃的测试（获取更多信息）
# Windows:
out/windows/debug/my-plugin-test.exe --reporter console
# Linux:
./out/linux/debug/my-plugin-test --reporter console

# 3. 用调试器定位崩溃点
# Windows (Visual Studio):
devenv /debugexe out/windows/debug/my-plugin-test.exe
# Linux (GDB):
gdb ./out/linux/debug/my-plugin-test
# macOS (LLDB):
lldb ./out/macos/debug/my-plugin-test
```

## 本节小结

- KrKr2 插件贡献遵循标准的 GitHub Fork + PR 工作流，共 9 个步骤从 Fork 到合并
- 代码格式使用 clang-format 20（LLVM 风格），最显著特征是 `SpaceBeforeParens: Never`（括号前无空格）和 K&R 花括号风格
- `cpp/external/` 和 `cpp/core/plugin/` 两个目录豁免格式检查，其他所有 C++ 代码必须通过
- 简单插件（单文件）直接放在 `cpp/plugins/` 根目录，只需修改 `target_sources`
- 复杂插件（多文件/跨平台）创建子目录，需要同时添加 `add_subdirectory` 和 `target_link_libraries`（缺一不可）
- KrKr2 使用自定义平台变量 `WINDOWS`/`LINUX`/`MACOS`/`ANDROID`，不要用 CMake 内置的 `WIN32`/`UNIX`/`APPLE`
- 测试使用 Catch2 v3 + 自定义 `main()`（初始化 spdlog），测试文件放在 `tests/unit-tests/plugins/`
- CI 有 4 个工作流：格式检查 + Windows/Linux/Android 构建
- 提交消息使用前缀分类法（feat/fix/test/docs/chore），PR 使用 Squash Merge 策略
- 贡献插件后必须更新 `doc/krkr2_plugins.md` 和 `cpp/plugins/AGENTS.md`

### 题目 1：修复格式检查失败

你提交了一个新插件 `textrender.cpp`，CI 的 `Code Format Check` 工作流失败了。日志显示以下差异：

```diff
-if (text.empty()) {
+if(text.empty()) {
     return;
 }
```

请问：(a) 这个格式差异是哪条 clang-format 规则导致的？(b) 如何在本地修复并重新提交？

<details>
<summary>查看答案</summary>

**(a)** 这是 `SpaceBeforeParens: Never` 规则导致的。KrKr2 要求控制语句（if/for/while 等）的括号前**不加空格**，所以 `if (text.empty())` 应该写成 `if(text.empty())`。

**(b)** 本地修复步骤：

```bash
# 1. 用 clang-format 自动修复
clang-format -i cpp/plugins/textrender.cpp

# 2. 验证修复结果
git diff
# 应该看到空格被移除

# 3. 追加提交
git add cpp/plugins/textrender.cpp
git commit -m "style: fix SpaceBeforeParens format issue"

# 4. 推送更新
git push origin plugin/textrender
# CI 会自动重新运行检查
```

</details>

### 题目 2：子目录插件注册

你创建了一个子目录插件 `cpp/plugins/imgfilter/`，包含 `imgfilter.cpp`、`imgfilter.h` 和自己的 `CMakeLists.txt`（project 名为 `imgFilter`）。编译通过但运行时 TJS 脚本报 `member not found`。请找出原因并写出修复代码。

<details>
<summary>查看答案</summary>

**原因**：最可能的原因是在 `cpp/plugins/CMakeLists.txt` 中只添加了 `add_subdirectory(imgfilter)` 但没有添加 `target_link_libraries`。这导致插件虽然编译了，但没有被链接进主程序。

**修复**：在 `cpp/plugins/CMakeLists.txt` 中补充链接：

```cmake
# 确保两行都存在：

# 第一行（可能已有）：
add_subdirectory(imgfilter)

# 第二行（缺少的）——在 target_link_libraries 块中添加：
target_link_libraries(${PROJECT_NAME} PUBLIC
        psdfile
        psbfile
        layerExDraw
        motionplayer
        fstat
        imgFilter      # ← 添加这一行，名称必须与子目录 project() 一致
)
```

**注意**：`add_subdirectory` 的参数是目录名 `imgfilter`（小写），但 `target_link_libraries` 的参数是 CMake target 名 `imgFilter`（驼峰），两者可以不同。target 名由子目录 `CMakeLists.txt` 中的 `project(imgFilter)` 决定。

</details>
