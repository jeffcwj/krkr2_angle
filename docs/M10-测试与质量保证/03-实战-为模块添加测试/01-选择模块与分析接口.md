## 模块选择标准

不是每个模块都适合作为"第一个测试"的目标。选择不当会让你陷入依赖地狱——花大量时间搞环境配置而不是学习测试编写。好的测试目标应该满足以下标准：

### 五条选择标准

| 标准 | 说明 | 优先级 |
|------|------|--------|
| **低依赖** | 被测代码不需要初始化整个引擎（如 Cocos2d-x、TJS2 虚拟机等） | ★★★ 必须 |
| **纯逻辑** | 接口行为可以用简单的输入输出描述，不涉及 GPU、文件系统、网络等外部资源 | ★★★ 必须 |
| **已有基础设施** | 测试目录、CMakeLists.txt、main.cpp 已存在，只需添加测试源文件 | ★★ 强烈推荐 |
| **高价值** | 代码在项目中被广泛使用，测试能捕获影响面大的回归 | ★★ 强烈推荐 |
| **接口清晰** | 方法签名、返回值语义明确，不需要阅读大量上下文才能理解 | ★ 推荐 |

### 候选模块评估

让我们用这五条标准评估 KrKr2 测试框架中的四个模块：

```
tests/unit-tests/
├── core/
│   ├── movie/    ← 候选 A：视频模块
│   ├── tjs2/     ← 候选 B：脚本引擎
│   └── visual/   ← 候选 C：渲染/图形模块
└── plugins/      ← 候选 D：插件模块
```

| 候选 | 低依赖 | 纯逻辑 | 已有基础设施 | 高价值 | 接口清晰 | 现有测试数 | 总评 |
|------|--------|--------|-------------|--------|---------|-----------|------|
| A: core/movie | ❌ 需要 FFmpeg 初始化 | ❌ 依赖解码器 | ✅ 有目录和 CMake | ✅ 视频播放核心 | ❌ 复杂 API | 1 | ⛔ 不适合入门 |
| B: core/tjs2 | ❌ 需要 TJS VM 启动 | ❌ 脚本执行有副作用 | ✅ 有目录和 CMake | ✅ 引擎核心 | ❌ 复杂 | 2 | ⛔ 不适合入门 |
| C: core/visual | ✅ tTVPRect 无依赖 | ✅ 纯数学计算 | ✅ 有目录和 CMake | ✅ 图层系统基础 | ✅ 方法语义明确 | **0** | ✅ 最佳目标 |
| D: plugins | ❌ 需要 PSB 解析初始化 | ❌ 涉及二进制文件 I/O | ✅ 有目录和 CMake | ✅ 插件核心 | ✅ 较清晰 | 2 | ⚠️ 已有测试 |

**结论：core/visual 模块是最佳选择。** 原因：

1. **零测试覆盖**：`tests/unit-tests/core/visual/` 的 CMakeLists.txt 中 `SOURCES` 为空——这是项目中唯一一个有完整测试基础设施但零测试的模块
2. **存在纯逻辑类**：`tTVPRect`（矩形结构体）和 `tTVPPoint`（点结构体）是纯数据结构加数学方法，不需要任何引擎初始化
3. **高使用频率**：矩形运算是图层渲染、碰撞检测、裁剪计算的基础，遍布整个 visual 子系统

## tTVPRect 接口完整分析

确定了测试目标是 `core/visual` 模块中的 `tTVPRect` 结构体。下面逐一分析它的每个公开方法，为编写测试做准备。

### 源码位置

`tTVPRect` 定义在 `cpp/core/visual/ComplexRect.h` 第 45-169 行。它是一个简单的矩形结构体，使用左上角 (`left`, `top`) 和右下角 (`right`, `bottom`) 坐标表示矩形区域。

### 数据成员与内存布局

```cpp
// 来源：cpp/core/visual/ComplexRect.h 第 45-74 行
struct tTVPRect {
    union {
        struct {
            tjs_int left;    // 左边界 x 坐标
            tjs_int top;     // 上边界 y 坐标
            tjs_int right;   // 右边界 x 坐标
            tjs_int bottom;  // 下边界 y 坐标
        };
        struct {
            tjs_int Left, Top, Right, Bottom;  // 大写别名（兼容性）
        };
        struct {
            tTVPPoint upper_left;    // 左上角点
            tTVPPoint bottom_right;  // 右下角点
        };
        tjs_int array[4];  // 数组形式访问
    };
    // ...
};
```

> **union 的用途：** 同一块内存可以通过四种方式访问——小写字段名（`left`）、大写字段名（`Left`）、两个 `tTVPPoint` 结构体、或一个 4 元素数组。这是 C 风格的内存布局技巧，在游戏引擎中很常见，目的是兼容不同的 API 风格而不增加内存开销。

> **什么是 `tjs_int`？** 它是 TJS2 脚本引擎定义的整数类型别名（Type Alias），在所有平台上等同于 `int`（32 位有符号整数）。使用别名而非直接写 `int` 是为了方便未来统一修改整数宽度。

### 构造函数

```cpp
// 来源：cpp/core/visual/ComplexRect.h 第 46-50 行
tTVPRect(tjs_int l, tjs_int t, tjs_int r, tjs_int b) {
    left = l, top = t, right = r, bottom = b;
}
tTVPRect() {};  // 默认构造——不初始化成员（值未定义）
```

**测试要点：**
- 四参数构造函数：验证 `left`、`top`、`right`、`bottom` 是否正确赋值
- 默认构造函数：成员值未定义，不应该对其做任何假设（这是一个设计选择——性能优先于安全性）

### 尺寸方法

```cpp
// 来源：cpp/core/visual/ComplexRect.h 第 76-80 行
tjs_int get_width() const { return right - left; }
tjs_int get_height() const { return bottom - top; }
void set_width(tjs_int w) { right = left + w; }
void set_height(tjs_int h) { bottom = top + h; }
```

**测试要点：**
- 正常情况：`tTVPRect(10, 20, 110, 220)` → `get_width()` 应返回 100
- 零尺寸：`left == right` 时 `get_width()` 应返回 0
- 负尺寸：`left > right` 时 `get_width()` 返回负数——这不是错误，`tTVPRect` 允许"反转"矩形
- `set_width` 后 `get_width` 应返回设置的值
- `set_width` 不应改变 `left`，只改变 `right`

### 位置方法

```cpp
// 来源：cpp/core/visual/ComplexRect.h 第 82-96 行
void add_offsets(tjs_int x, tjs_int y) {
    left += x; right += x;
    top += y;  bottom += y;
}

void set_offsets(tjs_int x, tjs_int y) {
    tjs_int w = get_width();
    tjs_int h = get_height();
    left = x; top = y;
    right = x + w; bottom = y + h;
}
```

**`add_offsets` 与 `set_offsets` 的区别：**

- `add_offsets`：**相对移动**——在当前位置基础上偏移，矩形尺寸不变
- `set_offsets`：**绝对定位**——将左上角移到 (x, y)，保持宽高不变

```
add_offsets(5, 3):                    set_offsets(50, 30):
┌───────┐                            ┌───────┐
│ 原位置 │──(+5,+3)──► ┌───────┐     │ 原位置 │──(50,30)──► ┌───────┐
└───────┘              │ 新位置 │     └───────┘              │ 新位置 │
                       └───────┘                             └───────┘
  尺寸不变                                尺寸不变，左上角 = (50,30)
```

**测试要点：**
- `add_offsets` 后四个坐标都应偏移相应值
- `add_offsets` 后宽高不变
- `set_offsets` 后 `left == x`、`top == y`
- `set_offsets` 后 `get_width()` 和 `get_height()` 与移动前相同
- 负偏移量也应正常工作

### 重置与状态查询

```cpp
// 来源：cpp/core/visual/ComplexRect.h 第 98-105 行
void set_size(tjs_int w, tjs_int h) {
    right = left + w; bottom = top + h;
}

void clear() { left = top = right = bottom = 0; }

bool is_empty() const { return left >= right || top >= bottom; }
```

**`is_empty()` 的判定规则：**

矩形在以下任一条件成立时为"空"：
- `left >= right`（宽度为零或负）
- `top >= bottom`（高度为零或负）

```
非空矩形:         宽度为零:         反转矩形:
┌───────┐         │(left==right)    (left > right)
│       │         │                  ← 这算空
│       │         │
└───────┘
left < right      left >= right     left >= right
top < bottom      → is_empty()=true → is_empty()=true
→ is_empty()=false
```

**测试要点：**
- 正常矩形 → `is_empty()` 返回 `false`
- `clear()` 后 → `is_empty()` 返回 `true`（因为 `left == right == 0`）
- 零宽度 `tTVPRect(10, 10, 10, 20)` → `is_empty()` 返回 `true`
- 零高度 `tTVPRect(10, 10, 20, 10)` → `is_empty()` 返回 `true`
- 反转矩形 `tTVPRect(20, 20, 10, 10)` → `is_empty()` 返回 `true`

### 集合运算方法

这是 `tTVPRect` 最重要的功能——矩形间的几何运算，直接影响图层渲染的正确性。

```cpp
// 来源：cpp/core/visual/ComplexRect.h 第 107-151 行
bool do_union(const tTVPRect &ref) {
    if(ref.is_empty()) return false;
    if(left > ref.left)     left = ref.left;
    if(top > ref.top)       top = ref.top;
    if(right < ref.right)   right = ref.right;
    if(bottom < ref.bottom) bottom = ref.bottom;
    return true;
}

bool intersects_with(const tTVPRect &ref) const {
    if(ref.is_empty() || is_empty()) return false;
    return intersects_with_no_empty_check(ref);
}

bool intersects_with_no_empty_check(const tTVPRect &ref) const {
    return !(left >= ref.right || top >= ref.bottom ||
             right <= ref.left || bottom <= ref.top);
}

bool included_in(const tTVPRect &ref) const {
    if(ref.is_empty() || is_empty()) return false;
    return included_in_no_empty_check(ref);
}

bool included_in_no_empty_check(const tTVPRect &ref) const {
    return ref.left <= left && ref.top <= top &&
           ref.right >= right && ref.bottom >= bottom;
}
```

**方法语义图解：**

```
do_union（并集）:              intersects_with（相交判定）:
┌───────┐                    ┌───────┐
│   A   │                    │   A   │
│   ┌───┼───┐                │   ┌───┼───┐
└───┼───┘   │   → 结果:      └───┼───┘   │   → true
    │   B   │   ┌───────────┐    │   B   │
    └───────┘   │ A ∪ B     │    └───────┘
                └───────────┘

included_in（包含判定）:
┌─────────────┐
│  ref        │
│  ┌───────┐  │
│  │ this  │  │  → this.included_in(ref) = true
│  └───────┘  │
└─────────────┘
```

**测试要点（这是最关键的测试区域）：**

`do_union`:
- 两个重叠矩形的并集 → 结果应该是包围两个矩形的最小矩形
- 两个不相邻矩形的并集 → 结果包含中间的空白区域
- 与空矩形求并 → 返回 `false`，原矩形不变
- 与自身求并 → 返回 `true`，原矩形不变

`intersects_with`:
- 两个重叠矩形 → `true`
- 两个不重叠矩形 → `false`
- 仅边缘相邻（如 A 的右边 == B 的左边）→ `false`（因为 `right <= ref.left`）
- 空矩形与任何矩形 → `false`

`included_in`:
- 小矩形在大矩形内 → `true`
- 大矩形在小矩形内 → `false`
- 完全相同的矩形 → `true`（边界条件：`>=` 和 `<=` 包含等号）
- 空矩形 → `false`

### 比较运算符

`tTVPRect` 提供了四个比较运算符（Comparison Operator，用于比较两个对象大小或相等性的特殊方法），分为两类：排序运算符和相等性运算符。这些运算符在图层渲染中至关重要——当引擎需要按从上到下、从左到右的顺序绘制矩形区域时，就依赖排序运算符确定绘制顺序。

#### 排序运算符（operator< 和 operator>）

```cpp
// 来源：cpp/core/visual/ComplexRect.h 第 154-159 行
bool operator<(const tTVPRect &rhs) const {
    return top < rhs.top || (top == rhs.top && left < rhs.left);
}
bool operator>(const tTVPRect &rhs) const {
    return top > rhs.top || (top == rhs.top && left > rhs.left);
}
```

**排序规则**（字典序，Lexicographic Order——像查字典一样，先比较第一个字段，相同再比较第二个字段）：

1. **首先比较 `top`**（纵坐标优先）：`top` 小的矩形排在前面
2. **`top` 相同时比较 `left`**：`left` 小的排在前面

```
排序示意图：

   left →
   0    10    20    30
 0 ┌─A──┐
   │    │  ┌─B──┐
 5 └────┘  │    │
           └────┘
10         ┌─C──┐
           │    │
15         └────┘

排序结果：A < B < C
  A: top=0  → 最小
  B: top=3  → 居中
  C: top=10 → 最大

如果 A 和 B 的 top 相同（都是 0）：
  A: left=0  < B: left=10 → A < B
```

> **为什么只比较 `top` 和 `left`？** 排序运算符的目的是确定绘制顺序，而绘制顺序由矩形的起始位置（左上角）决定。矩形的尺寸（`right`、`bottom`）与绘制顺序无关——一个大矩形和一个小矩形如果左上角相同，就应该在同一个位置开始绘制。

**测试要点：**
- `top` 不同时：`top` 小的 `<` 返回 `true`
- `top` 相同、`left` 不同时：`left` 小的 `<` 返回 `true`
- `top` 和 `left` 都相同时：`<` 返回 `false`，`>` 也返回 `false`
- `operator<` 和 `operator>` 的对称性：`a < b` 为 `true` 时 `b > a` 也应为 `true`
- 自反性：`a < a` 应为 `false`，`a > a` 应为 `false`

#### 相等性运算符（operator== 和 operator!=）

```cpp
// 来源：cpp/core/visual/ComplexRect.h 第 162-168 行
bool operator==(const tTVPRect &rhs) const {
    return top == rhs.top && left == rhs.left &&
           right == rhs.right && bottom == rhs.bottom;
}
bool operator!=(const tTVPRect &rhs) const {
    return !this->operator==(rhs);
}
```

`operator==` 比较所有四个字段——两个矩形完全相同才返回 `true`。`operator!=` 直接取反 `operator==` 的结果，这是标准的实现模式，确保 `==` 和 `!=` 永远不会同时返回相同的值。

**测试要点：**
- 四个字段完全相同 → `==` 返回 `true`，`!=` 返回 `false`
- 任意一个字段不同 → `==` 返回 `false`，`!=` 返回 `true`
- 逐字段差异测试：只有 `left` 不同、只有 `top` 不同、只有 `right` 不同、只有 `bottom` 不同
- 对称性：`a == b` 与 `b == a` 结果相同
- 自反性：`a == a` 应为 `true`

| **排序** | operator< 完全相同 | A == B | `A < B == false` | 边界 |
| **排序** | 对称性 | `a < b` | `b > a` | 正常 |
| **相等** | operator== 相同 | 四字段相同 | `true` | 正常 |
| **相等** | operator== 任一不同 | 逐字段测试 | `false` | 正常 |
| **相等** | operator!= 一致性 | 任意矩形对 | `(a==b) != (a!=b)` | 正常 |

> **共计 32 个测试用例**，覆盖 7 个方法组。下一节我们将把这张清单转化为完整的 Catch2 测试代码。

## 常见错误与解决方案

### 错误 1：测试"反转矩形"时忘记 is_empty 检查

```cpp
// ❌ 错误：直接对反转矩形调用 do_union，期望正常结果
tTVPRect a(100, 100, 0, 0);  // 反转矩形：left > right
tTVPRect b(10, 10, 50, 50);
a.do_union(b);
// 此时 a 的值可能不是你预期的——因为 do_union 不检查 this 是否为空！
// do_union 只检查 ref.is_empty()，不检查 this.is_empty()

// ✅ 正确：先理解 do_union 的前置条件
// do_union 假设 this 是一个有效矩形（或者你接受未定义的合并结果）
// 如果 this 可能是空矩形，应该先初始化再合并
tTVPRect a(100, 100, 0, 0);  // 反转矩形
if(a.is_empty()) {
    a = b;  // 直接赋值，而不是合并
} else {
    a.do_union(b);
}
```

**根本原因**：`do_union` 的实现只检查参数 `ref` 是否为空，不检查 `this`。这是性能优化——在引擎内部，调用 `do_union` 前通常已经确保 `this` 是有效矩形。但在测试中，你需要显式测试这种"不对称"行为。

### 错误 2：混淆 intersects_with 的边缘判定

```cpp
// ❌ 错误假设：仅边缘相邻的矩形算"相交"
tTVPRect a(0, 0, 10, 10);    // 右边界 = 10
tTVPRect b(10, 0, 20, 10);   // 左边界 = 10
bool result = a.intersects_with(b);
// result 是 false，不是 true！
// 因为判定条件是 right <= ref.left，即 10 <= 10 为 true → 不相交
```

**根本原因**：`tTVPRect` 使用的是左闭右开区间（Left-Closed Right-Open Interval）——矩形包含 `left` 和 `top` 对应的像素，但不包含 `right` 和 `bottom` 对应的像素。这在像素渲染中很常见：一个 10×10 的矩形 `(0,0,10,10)` 实际覆盖像素 `(0,0)` 到 `(9,9)`。

### 错误 3：忽略 union 成员的内存别名效果

```cpp
// ❌ 错误：以为修改 left 不会影响 upper_left.x
tTVPRect rect(10, 20, 30, 40);
rect.left = 99;
// 此时 rect.upper_left.x 也变成了 99！
// 因为它们是同一块内存（union）
// 同理，rect.array[0] 也是 99

// ✅ 正确理解：union 的所有成员共享内存
// 修改任何一个成员，通过其他成员读取都能看到变化
assert(rect.left == rect.Left);              // 大写小写是同一个
assert(rect.left == rect.upper_left.x);      // Point 视图是同一个
assert(rect.left == rect.array[0]);          // 数组视图是同一个
```

## 本节小结

1. **选择测试目标有标准**：低依赖、纯逻辑、已有基础设施、高价值、接口清晰——五条标准帮你在大量代码中快速锁定最适合写测试的模块
2. **core/visual 是最佳起点**：零测试覆盖 + 纯数据结构 + 完整基础设施，满足所有五条标准
3. **tTVPRect 是核心测试目标**：一个包含 15+ 可测试方法的矩形结构体，覆盖构造、尺寸、位置、重置、状态查询、集合运算、比较运算七大方法组
4. **union 内存布局**：四种访问方式（小写字段、大写字段、Point 结构体、数组）共享同一块内存，这是 C 风格的零开销兼容技巧
5. **is_empty 使用左闭右开规则**：`left >= right || top >= bottom` 为空，等号情况也算空（零面积矩形）
6. **do_union 的不对称性**：只检查参数 `ref` 是否为空，不检查 `this`——这是有意的性能优化
7. **intersects_with 的边缘不算相交**：`right <= ref.left` 用 `<=` 而非 `<`，仅边缘相邻不算重叠
8. **排序运算符只看左上角**：先比 `top` 再比 `left`，不关心矩形大小——因为绘制顺序由起始位置决定
9. **相等运算符比较全部四字段**：`operator!=` 直接取反 `operator==`，保证逻辑一致性
10. **测试清单是施工图纸**：在写代码前，先把所有测试用例列成清单（32 个），下一节将逐一实现

## 练习题与答案

### 题目 1：分析 set_offsets 的实现细节

`set_offsets` 的实现如下：

```cpp
void set_offsets(tjs_int x, tjs_int y) {
    tjs_int w = get_width();
    tjs_int h = get_height();
    left = x; top = y;
    right = x + w; bottom = y + h;
}
```

问题：如果对一个"反转矩形"（`left > right`）调用 `set_offsets`，结果是什么？矩形还是反转的吗？请用具体数值说明。

<details>
<summary>查看答案</summary>

以 `tTVPRect(100, 80, 20, 30)` 为例（`left=100 > right=20`，反转矩形）：

```cpp
tTVPRect rect(100, 80, 20, 30);
// get_width() = right - left = 20 - 100 = -80（负值）
// get_height() = bottom - top = 30 - 80 = -50（负值）

rect.set_offsets(10, 10);
// w = -80, h = -50
// left = 10, top = 10
// right = 10 + (-80) = -70
// bottom = 10 + (-50) = -40

// 结果：rect = (10, 10, -70, -40)
// left(10) > right(-70)，仍然是反转的！
// set_offsets 保持宽高不变（包括负宽高），所以反转矩形移动后仍然反转
```

**关键认识**：`set_offsets` 忠实保留原有宽高——如果原来宽度是负的，移动后宽度还是负的。这意味着 `set_offsets` 不会"修复"反转矩形。如果你想让矩形变正常，需要先调用其他方法（如手动交换 `left`/`right`）或重新构造。

</details>

### 题目 2：预测 do_union 的结果

给定两个不相邻的矩形：

```cpp
tTVPRect a(0, 0, 10, 10);     // 左上角区域
tTVPRect b(100, 100, 200, 200); // 右下角区域（与 a 完全不重叠）
a.do_union(b);
```

问题：
1. `do_union` 的返回值是什么？
2. 调用后 `a` 的四个字段值分别是多少？
3. `a` 的面积（`get_width() * get_height()`）是多少？这个面积有什么含义？

<details>
<summary>查看答案</summary>

```cpp
tTVPRect a(0, 0, 10, 10);
tTVPRect b(100, 100, 200, 200);
bool result = a.do_union(b);

// 1. 返回值：true（因为 b 不是空矩形）

// 2. do_union 的逻辑——取各方向的极值：
//    if(left > ref.left)     left = ref.left;     → 0 > 100? 否，left 不变 = 0
//    if(top > ref.top)       top = ref.top;       → 0 > 100? 否，top 不变 = 0
//    if(right < ref.right)   right = ref.right;   → 10 < 200? 是，right = 200
//    if(bottom < ref.bottom) bottom = ref.bottom;  → 10 < 200? 是，bottom = 200
//
//    结果：a = (0, 0, 200, 200)

// 3. 面积：get_width() * get_height() = 200 * 200 = 40000
//    含义：这是能同时包围 a 和 b 的最小矩形的面积
//    但注意！这个矩形包含了大量"空白区域"——
//    原来 a 的面积是 100，b 的面积是 10000，两者之和才 10100
//    但并集矩形面积是 40000，中间有 29900 的空白
//    这说明 do_union 计算的是"包围盒"（Bounding Box），不是精确的几何并集
```

**关键认识**：`do_union` 不是精确的集合并集（Set Union），而是轴对齐包围盒（Axis-Aligned Bounding Box, AABB）。当两个矩形距离很远时，包围盒会包含大量无效区域。这在图层渲染中被称为"脏区域过估"（Dirty Region Over-estimation）——虽然不精确，但计算简单、速度快，对于逐帧渲染来说是合理的性能权衡。

</details>

### 题目 3：设计 clip 方法的测试策略

`tTVPRect` 还有一个 `clip` 方法（被 `#ifndef __TP_STUB_H__` 守卫）：

```cpp
bool clip(const tTVPRect &ref) {
    if(!TVPIntersectRect(this, this, &ref)) {
        clear();
        return false;
    }
    return true;
}
```

问题：为什么本节没有把 `clip` 列入测试用例清单？如果你一定要测试 `clip`，需要解决哪些技术问题？请列出至少 3 个。

<details>
<summary>查看答案</summary>

`clip` 没有列入测试清单的原因是它有**外部依赖**——调用了 `TVPIntersectRect` 函数，这个函数不是 `tTVPRect` 的内联方法，而是在其他编译单元中定义的外部函数。

如果一定要测试 `clip`，需要解决以下技术问题：

1. **链接依赖**：`TVPIntersectRect` 的定义不在 `ComplexRect.h` 中，而是在 visual 模块的某个 `.cpp` 文件里。要链接这个函数，可能需要链接整个 `core_visual_module` 目标，这会引入大量额外依赖（图片加载器、图层管理器等）

2. **预处理器守卫**：`clip` 被 `#ifndef __TP_STUB_H__` 包围。如果测试代码在某种条件下定义了 `__TP_STUB_H__`（比如间接引入了 stub 头文件），`clip` 方法将不可见，编译会报"找不到成员函数"的错误

3. **隔离性破坏**：我们选择 `tTVPRect` 的核心理由之一就是"纯逻辑、无依赖"。`clip` 打破了这个前提。测试 `clip` 就从单元测试（Unit Test，测试单个组件的独立行为）变成了集成测试（Integration Test，测试多个组件协同工作），测试的复杂度和失败排查难度都会显著上升

4. **可能的替代方案**：
   - 创建一个 Mock（模拟对象）版本的 `TVPIntersectRect`——但这需要修改头文件或使用链接替换技巧
   - 使用 `intersects_with` + 手动裁剪来间接验证 `clip` 的语义——但这不是真正在测试 `clip`
   - 在后续的集成测试章节中，链接完整的 visual 模块后再测试 `clip`

**结论**：遵循"先易后难"的原则，先测试所有无依赖的方法（本节的 32 个用例），建立信心后再考虑有外部依赖的 `clip`。

</details>
