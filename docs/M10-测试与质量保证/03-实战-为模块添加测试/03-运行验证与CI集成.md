## 术语预览

本节将涉及以下术语，先有个印象，正文中会逐一详细讲解：

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| CTest | CTest | CMake 自带的测试运行器，负责发现、调度、执行测试可执行文件并汇总结果 |
| 测试发现 | Test Discovery | CMake 在构建后自动扫描 Catch2 可执行文件中的 TEST_CASE，注册为 CTest 测试用例 |
| 工作流触发 | Workflow Trigger | GitHub Actions 中定义何时运行 CI 流水线的条件（如 push、pull_request、workflow_run） |
| 门禁检查 | Gate Check | CI 中的前置步骤（如格式检查），通过后才允许后续步骤（如构建）继续执行 |
| 二进制缓存 | Binary Caching | vcpkg 将已编译的包缓存到远端（NuGet），避免每次 CI 都从源码编译依赖 |
| 基准测试 | Benchmark | 对代码的执行性能进行量化测量，用于检测性能退化（Performance Regression） |
| 测试覆盖率 | Code Coverage | 衡量测试执行了多少比例的源代码行/分支，通常用 gcov/llvm-cov 工具生成报告 |

## 第一步：将测试文件注册到 CMake

在上一节中我们编写了 `rect_test.cpp`，但它还没有被 CMake 构建系统识别。要让 CMake "看到"这个文件，需要将它添加到对应模块的 `CMakeLists.txt` 的 `SOURCES` 变量中。

### 回顾 visual 模块的 CMake 配置

打开 `tests/unit-tests/core/visual/CMakeLists.txt`（项目根目录 27 行）：

```cmake
# tests/unit-tests/core/visual/CMakeLists.txt（完整文件）
cmake_minimum_required(VERSION 3.16)
project(TestVisualModule LANGUAGES CXX)

# 测试源文件列表——目前为空，我们要在这里添加
set(SOURCES
)

# 从 .cpp 文件名生成可执行文件名（去掉 .cpp 后缀）
# 例如 "rect_test.cpp" → "rect_test"
string(REPLACE ".cpp" "" BASENAMES_SOURCES "${SOURCES}")
set(TARGETS ${BASENAMES_SOURCES})

# 每个源文件生成一个独立的可执行文件
# 注意：每个可执行文件都会链接 main.cpp（自定义入口）
foreach(name ${TARGETS})
    add_executable(${name} ${name}.cpp main.cpp)
endforeach()

set(ALL_TARGETS
        ${TARGETS}
)

# 为每个测试目标配置链接和测试发现
foreach(name ${ALL_TARGETS})
    target_link_libraries(${name}
        PRIVATE
            Catch2::Catch2      # Catch2 测试框架（私有链接，不传播）
        PUBLIC
            krkr2core            # 引擎核心库（公开链接，获取所有头文件路径）
    )
    # 包含构建目录中生成的 test_config.h
    target_include_directories(${name} PRIVATE "${TEST_CONFIG_DIR}")
    # 自动发现 Catch2 TEST_CASE 并注册为 CTest 用例
    catch_discover_tests(${name})
endforeach()
```

关键点解读：

1. **一个源文件 = 一个可执行文件**：`string(REPLACE ".cpp" "" ...)` 把每个 `.cpp` 文件名转换为可执行文件名。这意味着 `rect_test.cpp` 会生成 `rect_test` 可执行文件
2. **自动链接 main.cpp**：`add_executable(${name} ${name}.cpp main.cpp)` 会把当前目录下的 `main.cpp`（包含 spdlog 初始化和 `Catch::Session().run()`）一起编译
3. **catch_discover_tests**：这是 Catch2 提供的 CMake 函数，它会在**构建后**运行测试可执行文件的 `--list-tests` 命令，自动将每个 `TEST_CASE` 注册为独立的 CTest 用例

### 注册 rect_test.cpp

修改 `SOURCES` 变量，添加我们的测试文件：

```cmake
set(SOURCES
    rect_test.cpp
)
```

仅此一行修改。CMake 的 foreach 循环会自动完成以下工作：

```
rect_test.cpp
    ↓ string(REPLACE ".cpp" "" ...)
rect_test
    ↓ add_executable(rect_test rect_test.cpp main.cpp)
rect_test 可执行文件
    ↓ target_link_libraries(rect_test ... Catch2::Catch2 ... krkr2core)
链接 Catch2 和引擎核心库
    ↓ catch_discover_tests(rect_test)
自动注册所有 TEST_CASE 为 CTest 用例
```

配置完成后，先运行 CMake 配置步骤（不构建），检查是否有错误：

**Windows（MSVC + Ninja）：**

```bash
# 在项目根目录执行
cmake --preset="Windows Debug Config"
```

**Linux：**

```bash
cmake --preset="Linux Debug Config"
```

**macOS：**

```bash

```

如果配置成功，你会看到类似输出：


```cmake
find_package(Catch2 3 REQUIRED)

```

## 第三步：使用 CTest 运行测试

### CTest 基础用法

CTest 是 CMake 自带的测试运行器（Test Runner）。当你在 CMakeLists.txt 中调用了 `enable_testing()` 和 `catch_discover_tests()`，CTest 就能自动发现并运行所有注册的测试。

运行所有测试的基本命令：

```bash
# 通用格式：指定构建目录
ctest --test-dir out/<platform>/debug --output-on-failure
```

各平台具体命令：

```bash
# Windows
ctest --test-dir out/windows/debug --output-on-failure

# Linux
ctest --test-dir out/linux/debug --output-on-failure

# macOS
ctest --test-dir out/macos/debug --output-on-failure
```

`--output-on-failure` 参数的作用是：只有当测试**失败**时才打印详细输出。如果测试全部通过，你看到的是简洁的汇总信息：

```
Test project /home/user/krkr2/out/linux/debug
      Start  1: tTVPRect - 构造函数/默认构造
 1/58 Test  #1: tTVPRect - 构造函数/默认构造 ........   Passed    0.01 sec
      Start  2: tTVPRect - 构造函数/四参数构造
 2/58 Test  #2: tTVPRect - 构造函数/四参数构造 ......   Passed    0.01 sec
      ...
58/58 Test #58: tTVPRect - union内存布局/...........   Passed    0.01 sec

100% tests passed, 0 tests failed out of 58

Total Test time (real) =   0.52 sec
```

### CTest 常用选项

CTest 提供了丰富的命令行选项来控制测试执行：

```bash
# 显示详细输出（即使通过的测试也打印详情）
ctest --test-dir out/linux/debug --verbose

# 极简输出（只显示通过/失败计数）
ctest --test-dir out/linux/debug --progress

# 并行运行测试（-j 后面是并行数）
# 每个 TEST_CASE 作为独立进程运行，互不干扰
ctest --test-dir out/linux/debug -j 8

# 只运行名称匹配正则的测试（-R = regex filter）
ctest --test-dir out/linux/debug -R "构造函数"

# 排除名称匹配正则的测试（-E = exclude regex）
ctest --test-dir out/linux/debug -E "union内存"

# 只运行上次失败的测试（快速重验）
ctest --test-dir out/linux/debug --rerun-failed

# 重复运行测试 N 次（检测偶发失败）
ctest --test-dir out/linux/debug --repeat-until-fail 5

# 设置单个测试超时（秒），防止死循环
ctest --test-dir out/linux/debug --timeout 10
```

### CTest 与 Catch2 的关系

理解 CTest 和 Catch2 的分工很重要：

```
┌─────────────────────────────────────────────────────┐
│                    CTest（调度层）                     │
│  - 发现测试（通过 catch_discover_tests 注册）          │
│  - 调度执行（并行、过滤、超时）                        │
│  - 汇总结果（通过/失败计数、耗时）                     │
│  - 生成报告（XML、JUnit 格式）                        │
└──────────────┬──────────────────────────────────────┘
               │ 对每个 TEST_CASE 执行：
               │ rect_test "tTVPRect - 构造函数/默认构造"
               ▼
┌─────────────────────────────────────────────────────┐
│               Catch2（执行层）                        │
│  - 运行具体的 TEST_CASE 代码                          │
│  - 执行 REQUIRE/CHECK 断言                           │
│  - 管理 SECTION 树的遍历                             │
│  - 报告失败位置（文件名 + 行号 + 表达式）              │
└─────────────────────────────────────────────────────┘
```

`catch_discover_tests()` 在**构建后**（Post-Build）阶段执行 `rect_test --list-tests`，获取所有 TEST_CASE 的全名，然后为每个 TEST_CASE 注册一条 CTest 记录。这就是为什么我们写了 10 个 TEST_CASE 但 CTest 报告 58 个测试——因为 `catch_discover_tests` 默认会将每个 SECTION 也展开为独立测试。

### 直接运行 Catch2 可执行文件

除了通过 CTest，你也可以直接运行测试可执行文件。Catch2 有自己的命令行参数系统，功能比 CTest 更丰富：

```bash
# 直接运行（Windows 示例，Linux/macOS 去掉 .exe）
./out/windows/debug/tests/unit-tests/core/visual/rect_test.exe

# 列出所有测试用例名
./rect_test --list-tests

# 列出所有标签
./rect_test --list-tags

# 只运行包含 "构造函数" 的测试
./rect_test "构造函数"

# 运行多个匹配模式（逗号分隔）
./rect_test "构造函数","尺寸方法"

# 排除某些测试（~ 前缀表示排除）
./rect_test "~union内存"

# 按标签过滤（如果 TEST_CASE 设置了标签）
./rect_test [rect][geometry]

# 输出 JUnit XML 格式（可被 CI 系统解析）
./rect_test -r junit -o test-results.xml

# 输出紧凑格式（适合日志）
./rect_test -r compact

# 显示每个 SECTION 的执行过程
./rect_test -s

# 失败时立即中断（不继续运行后续测试）
./rect_test -a
```

**CTest vs 直接运行的对比：**

| 场景 | 推荐方式 | 原因 |
|------|---------|------|
| CI 流水线 | CTest | 标准化接口，支持并行，生成 CTest 报告 |
| 本地快速验证 | CTest | 一条命令运行所有模块的测试 |
| 调试单个失败用例 | 直接运行 | 可以用 `-s` 看 SECTION 执行顺序，用 `--break` 触发调试器断点 |
| 生成 XML 报告 | 直接运行 | Catch2 的 `-r junit` 比 CTest 的 XML 输出更详细 |
| 性能基准测试 | 直接运行 | CTest 的进程启动开销会影响基准结果 |

## 第四步：跨平台构建与运行差异

不同平台在构建和运行测试时有一些需要注意的差异。

### 完整流程对比表

| 步骤 | Windows (MSVC) | Linux (GCC) | macOS (Clang) |
|------|----------------|-------------|---------------|
| 前置条件 | VS 2022 + vcpkg + winflexbison | gcc/g++ + vcpkg + bison | Xcode CLT + vcpkg |
| 配置 | `cmake --preset="Windows Debug Config"` | `cmake --preset="Linux Debug Config"` | `cmake --preset="MacOS Debug Config"` |
| 构建 | `cmake --build --preset="Windows Debug Build"` | `cmake --build --preset="Linux Debug Build"` | `cmake --build --preset="MacOS Debug Build"` |
| 运行测试 | `ctest --test-dir out\windows\debug` | `ctest --test-dir out/linux/debug` | `ctest --test-dir out/macos/debug` |
| 可执行文件后缀 | `.exe` | 无 | 无 |
| 路径分隔符 | `\`（CMD）或 `/`（PowerShell/Git Bash） | `/` | `/` |
| 编译器警告 | `/W4` | `-Wall -Wextra` | `-Wall -Wextra` |

### Windows 平台特殊注意事项

1. **环境变量**：必须设置 `VCPKG_ROOT` 指向 vcpkg 安装目录。MSVC 编译器需要通过 `vcvarsall.bat` 或 VS Developer Command Prompt 激活

2. **路径长度限制**：Windows 默认最大路径长度为 260 字符。如果项目路径过深，可能导致 vcpkg 安装失败。建议将项目放在短路径下，如 `C:\dev\krkr2\`

3. **winflexbison**：TJS2 模块的 bison 语法文件需要 `win_flex` 和 `win_bison` 工具。确保 `C:\winflexbison` 在 PATH 中

4. **MSVC 编译选项**：项目在 MSVC 下会自动添加 `/EHsc`（异常处理）、`/MP`（并行编译）、`/utf-8`（UTF-8 源文件编码）

### Linux 平台特殊注意事项

1. **系统依赖**：除了 vcpkg 管理的包，还需要系统级依赖：
   ```bash
   sudo apt-get install -y gcc g++ bison python3 nasm ninja-build \
       libxft-dev libxext-dev libxxf86vm-dev libx11-dev libglu1-mesa-dev \
       libgl2ps-dev libxi-dev libzip-dev libpng-dev libssl-dev libgtk-3-dev
   ```

2. **GCC 版本**：项目要求 C++17 支持，最低需要 GCC 7+（推荐 GCC 11+）

3. **OpenMP**：项目在 Linux 下启用了 OpenMP 并行加速，确保安装了 `libgomp`

### macOS 平台特殊注意事项

1. **Xcode Command Line Tools**：必须安装，执行 `xcode-select --install`

2. **编译器**：使用系统 Clang，CMakePresets.json 中指定了 `CMAKE_C_COMPILER=clang` 和 `CMAKE_CXX_COMPILER=clang++`

3. **OpenMP 不可用**：项目在 Apple 平台禁用了 OpenMP（Apple Clang 不支持）。如果测试代码使用了 OpenMP，需要用 `#ifdef _OPENMP` 条件编译

4. **arm64 架构**：M1/M2 Mac 使用 arm64 架构，vcpkg 会自动选择对应的 triplet

### Android 为什么不运行测试？

查看项目根目录 `CMakeLists.txt` 第 136 行：

```cpp
// CMakeLists.txt 第 136-139 行
if(ENABLE_TESTS AND NOT (IOS OR ANDROID))
    enable_testing()
    add_subdirectory(tests)
endif()
```

测试**只在桌面平台构建**，Android 和 iOS 被排除。原因：

1. **交叉编译限制**：Android 上的 Catch2 测试可执行文件是 ARM 二进制文件，无法在 x86 CI 主机上直接运行
2. **设备/模拟器需求**：要运行测试需要连接 Android 设备或启动模拟器（Android Emulator），这在 CI 环境中成本高且不稳定
3. **测试目标是纯 C++ 逻辑**：`tTVPRect` 等数据结构没有平台特定行为，在桌面平台测试已足够覆盖

## 第五步：理解项目 CI 流水线

KrKr2 项目使用 GitHub Actions 作为 CI/CD（持续集成/持续交付）平台。理解现有流水线的结构，是后续添加测试步骤的基础。

### 工作流概览

项目有 4 个工作流文件，位于 `.github/workflows/` 目录：

```
.github/workflows/
├── code-format-check.yml     # 代码格式检查（门禁）
├── build-windows.yml          # Windows 构建
├── build-linux.yml            # Linux 构建
└── build-android.yml          # Android 构建
```

它们之间的依赖关系：

```
                    push/PR 到 main
                         │
                         ▼
              ┌──────────────────┐
              │ Code Format Check │  ← 门禁：clang-format 20 检查
              │  (ubuntu-latest)  │
              └────────┬─────────┘
                       │ workflow_run (成功后触发)
           ┌───────────┼───────────┐
           ▼           ▼           ▼
   ┌──────────┐ ┌──────────┐ ┌───────────┐
   │  Build    │ │  Build   │ │   Build   │
   │ Windows   │ │  Linux   │ │  Android  │
   └──────────┘ └──────────┘ └───────────┘
```

### 门禁检查：code-format-check.yml

这是所有构建的前置条件。来看核心配置（`.github/workflows/code-format-check.yml`，40 行）：

```yaml
name: Code Format Check

on:
  # 手动触发
  workflow_dispatch:
  # push 到 main 分支时触发
  push:
    branches: [ main ]
    paths:  # 只有这些路径的文件变更才触发
      - 'cmake/**'
      - 'cpp/**'
      - 'platforms/**'
      - 'vcpkg/**'
      - 'CMakeLists.txt'
      - 'CMakePresets.json'
      - 'vcpkg.json'
      - 'vcpkg-configuration.json'
      - '.github/workflows/**'
  # PR 到 main 分支时也触发
  pull_request:
    branches: [ main ]
    paths: [...]  # 同上

jobs:
  format-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      # 使用 clang-format 20 检查 cpp/ 和 platforms/ 下的源文件
      - uses: DoozyX/clang-format-lint-action@v0.20
        with:
          source: 'cpp platforms'
          extensions: 'cpp,cc,h,hpp,inc'
          clangFormatVersion: 20
```

关键设计点：

1. **路径过滤**：只有 C++ 源码、CMake 配置、vcpkg 配置等变更才触发检查。修改 `docs/` 下的文档**不会**触发
2. **clang-format 版本固定**：使用 v20，与项目 `.clang-format` 配置文件匹配
3. **检查范围**：`cpp/` 和 `platforms/` 目录下的 `.cpp/.cc/.h/.hpp/.inc` 文件。注意 `tests/` 目录**不在检查范围内**

### 构建工作流：以 build-linux.yml 为例

Linux 构建工作流（`.github/workflows/build-linux.yml`，100 行）是三个构建工作流中最典型的。核心结构：

```yaml
name: Build Linux

on:
  workflow_dispatch:              # 手动触发
  workflow_run:                    # 由其他工作流触发
    workflows: ["Code Format Check"]
    types: [ "completed" ]         # 格式检查完成后触发

env:
  # vcpkg 二进制缓存：使用 GitHub Packages NuGet 源
  VCPKG_BINARY_SOURCES: "clear;nuget,https://nuget.pkg.github.com/.../index.json,readwrite"

jobs:
  build-linux:
    # 关键：只有格式检查成功才执行构建
    if: ${{ github.event.workflow_run.conclusion == 'success' }}
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # 1. 安装系统依赖（Mono、gcc、bison、各种 dev 库）
      - name: Install system dependencies
        run: sudo apt-get install -y ...

      # 2. 安装 CMake 3.31.1
      - name: Install CMake
        uses: lukka/get-cmake@latest

      # 3. 克隆并启动 vcpkg
      - name: Setup vcpkg
        run: |
          git clone https://github.com/microsoft/vcpkg.git $VCPKG_ROOT
          $VCPKG_ROOT/bootstrap-vcpkg.sh

      # 4. 配置 NuGet 二进制缓存源
      - name: Add NuGet sources
        run: mono `vcpkg fetch nuget` sources add ...

      # 5. 启用 ccache 加速编译
      - name: ccache
        uses: hendrikmuhs/ccache-action@v1.2.18

      # 6. 构建（注意：ENABLE_TESTS=OFF！）
      - name: Build Linux
        run: |
          cmake --preset="Linux Release Config" -DENABLE_TESTS=OFF
          cmake --build --preset="Linux Release Build"

      # 7. 上传构建产物
      - name: Upload artifacts
        uses: actions/upload-artifact@v4
```

### 关键发现：CI 目前不运行测试！

注意第 6 步的构建命令：

```bash
cmake --preset="Linux Release Config" -DENABLE_TESTS=OFF -DBUILD_TOOLS=OFF
```

`-DENABLE_TESTS=OFF` 表示**测试被显式禁用**。三个构建工作流（Windows、Linux、Android）全部如此。这意味着：

1. **测试不会被编译**：`add_subdirectory(tests)` 不会执行
2. **CTest 不可用**：`enable_testing()` 不会被调用
3. **没有测试结果**：CI 只验证"能编译通过"，不验证"功能是否正确"

这是项目当前的现状——CI 只做**格式检查 + 编译验证**，还没有集成自动化测试。这正是我们需要改进的地方。

### 设计测试 CI 工作流

基于项目现有的 CI 架构，我们可以设计一个新的测试工作流。以下是一个完整的示例：


## 第六步：测试覆盖率

测试覆盖率（Code Coverage）衡量测试执行了多少比例的源代码。它不能证明代码"正确"，但能告诉你"哪些代码从未被测试执行过"——这些未覆盖的代码就是潜在的质量盲区。

### gcov/llvm-cov 基础

在 Linux 上使用 GCC 的 `gcov` 工具链，在 macOS 上使用 LLVM 的 `llvm-cov` 工具链，可以生成覆盖率报告。

**原理**：编译时加入特殊标志，让编译器在每行代码中插入计数器。运行测试后，计数器数据会写入 `.gcda` 文件，然后用 `gcov` 或 `llvm-cov` 工具将其转换为可读报告。

#### 步骤 1：启用覆盖率编译选项

在 CMakeLists.txt 中添加覆盖率选项：

```cmake
# 在 tests/CMakeLists.txt 或根 CMakeLists.txt 中添加
option(ENABLE_COVERAGE "Enable code coverage" OFF)

if(ENABLE_COVERAGE)
    if(CMAKE_CXX_COMPILER_ID MATCHES "GNU")
        # GCC：使用 --coverage 标志（等价于 -fprofile-arcs -ftest-coverage）
        add_compile_options(--coverage)
        add_link_options(--coverage)
    elseif(CMAKE_CXX_COMPILER_ID MATCHES "Clang")
        # Clang：使用 -fprofile-instr-generate -fcoverage-mapping
        add_compile_options(-fprofile-instr-generate -fcoverage-mapping)
        add_link_options(-fprofile-instr-generate -fcoverage-mapping)
    endif()
endif()
```

#### 步骤 2：构建并运行

```bash
# Linux (GCC) 示例
cmake --preset="Linux Debug Config" -DENABLE_TESTS=ON -DENABLE_COVERAGE=ON
cmake --build --preset="Linux Debug Build"

# 运行测试（这会生成 .gcda 覆盖率数据文件）
ctest --test-dir out/linux/debug --output-on-failure
```

#### 步骤 3：生成报告

**使用 gcov + lcov（Linux/GCC）：**

```bash
# 安装 lcov（gcov 的前端，生成 HTML 报告）
sudo apt-get install -y lcov

# 收集覆盖率数据
lcov --capture \
     --directory out/linux/debug \
     --output-file coverage.info \
     --rc branch_coverage=1

# 过滤掉不需要的路径（系统头文件、测试代码本身、外部依赖）
lcov --remove coverage.info \
     '/usr/*' \
     '*/tests/*' \
     '*/external/*' \
     '*/vcpkg_installed/*' \
     --output-file coverage_filtered.info \
     --rc branch_coverage=1

# 生成 HTML 报告
genhtml coverage_filtered.info \
        --output-directory coverage_report \
        --branch-coverage \
        --title "KrKr2 Test Coverage"

# 用浏览器打开报告
# Linux: xdg-open coverage_report/index.html
# macOS: open coverage_report/index.html
```

**使用 llvm-cov（macOS/Clang）：**

```bash
# 合并原始覆盖率数据
llvm-profdata merge -sparse default.profraw -o coverage.profdata

# 生成报告
llvm-cov report \
    ./out/macos/debug/tests/unit-tests/core/visual/rect_test \
    -instr-profile=coverage.profdata \
    -ignore-filename-regex='(tests/|external/|vcpkg_installed/)'

# 生成 HTML 报告
llvm-cov show \
    ./out/macos/debug/tests/unit-tests/core/visual/rect_test \
    -instr-profile=coverage.profdata \
    -format=html \
    -output-dir=coverage_report
```

### 覆盖率指标解读

```
┌────────────────────────────────────────────┐
│           覆盖率报告示例                      │
├────────────────────────────────────────────┤
│ 文件: cpp/core/visual/ComplexRect.h        │
│                                            │
│ 行覆盖率:   92.3%  (120/130 行)             │
│ 分支覆盖率:  85.7%  (24/28 分支)            │
│ 函数覆盖率: 100.0%  (15/15 函数)            │
│                                            │
│ 未覆盖的行:                                 │
│   Line 142: clip() 方法体                   │
│   Line 143: (clip 依赖外部 TVPIntersectRect) │
│   ...                                       │
└────────────────────────────────────────────┘
```

三种覆盖率指标的含义：

| 指标 | 含义 | 达标建议 |
|------|------|---------|
| 行覆盖率（Line Coverage） | 被执行的代码行占总行数的比例 | ≥80% |
| 分支覆盖率（Branch Coverage） | 被执行的条件分支（if/else、三元运算等）占总分支的比例 | ≥70% |
| 函数覆盖率（Function Coverage） | 被调用的函数占总函数数的比例 | ≥90% |

> **注意**：100% 覆盖率不代表没有 Bug！覆盖率只保证代码"被执行过"，不保证"逻辑正确"。例如，一个测试可能执行了 `do_union` 的所有分支，但如果断言写错了（比如 `REQUIRE(result == wrong_value)`），覆盖率仍然是 100%。

## 第七步：Catch2 基准测试（Benchmark）

除了功能测试，Catch2 还内置了基准测试功能，可以测量代码的执行性能。这对于性能敏感的数据结构（如 `tTVPRect` 在渲染循环中被频繁调用）特别有用。

### 编写基准测试

```cpp
// bench_rect.cpp — 独立的基准测试文件
#include <catch2/catch_test_macros.hpp>
#include <catch2/benchmark/catch_benchmark.hpp>
#include "ComplexRect.h"

TEST_CASE("tTVPRect 性能基准", "[rect][benchmark]") {

    BENCHMARK("默认构造 1000 个 tTVPRect") {
        // BENCHMARK 宏会自动多次运行并取平均值
        std::vector<tTVPRect> rects(1000);
        return rects.size();  // 返回值防止编译器优化掉整个操作
    };

    BENCHMARK("do_union 连续合并 1000 次") {
        tTVPRect base(0, 0, 100, 100);
        for (int i = 1; i <= 1000; ++i) {
            tTVPRect other(i, i, i + 50, i + 50);
            base.do_union(other);
        }
        return base.get_width();  // 防止优化
    };

    BENCHMARK("intersects_with 判断 1000 次") {
        tTVPRect a(0, 0, 100, 100);
        tTVPRect b(50, 50, 150, 150);
        int count = 0;
        for (int i = 0; i < 1000; ++i) {
            if (a.intersects_with(b)) {
                ++count;
            }
        }
        return count;
    };

    BENCHMARK("operator< 比较排序 1000 个元素") {
        std::vector<tTVPRect> rects(1000);
        for (int i = 0; i < 1000; ++i) {
            rects[i] = tTVPRect(i % 100, i / 100,
                                i % 100 + 50, i / 100 + 50);
        }
        std::sort(rects.begin(), rects.end());
        return rects[0].left;
    };
}
```

### 运行基准测试

```bash
# 直接运行（不通过 CTest，避免进程启动开销）
./rect_test "[benchmark]" --benchmark-samples 100

# 输出示例：
# benchmark name                       samples       iterations    est run time
#                                      mean          low mean      high mean
#                                      std dev       low std dev   high std dev
# -----------------------------------------------------------------------
# 默认构造 1000 个 tTVPRect                100           1          1.2 ms
#                                      12.3 us       11.8 us       13.1 us
#                                      2.1 us        1.5 us        3.2 us
```

关键参数说明：

- `--benchmark-samples N`：每个 BENCHMARK 运行 N 次取平均（默认 100）
- `--benchmark-warmup-time T`：预热时间（毫秒），让 CPU 缓存预热
- `--benchmark-confidence-interval P`：置信区间（默认 0.95）
- `--benchmark-no-analysis`：跳过统计分析，只运行

> **提示**：基准测试应在 Release 构建下运行，Debug 构建的优化级别为 O0，性能数据没有参考价值。

## 常见错误与解决方案

### 错误 1：`catch_discover_tests` 命令未找到

```
CMake Error at tests/unit-tests/core/visual/CMakeLists.txt:26:
  Unknown CMake command "catch_discover_tests"
```

**原因**：`include(Catch)` 没有执行，通常是因为 Catch2 未正确安装。

**解决方案**：

```bash
# 检查 vcpkg 中 Catch2 是否安装
# Linux/macOS
ls $VCPKG_ROOT/installed/*/share/Catch2/

# Windows
dir %VCPKG_ROOT%\installed\x64-windows-static-md\share\Catch2\

# 如果目录不存在，手动安装
vcpkg install catch2:x64-windows-static-md  # Windows
vcpkg install catch2:x64-linux              # Linux
```

### 错误 2：测试编译通过但运行时段错误（Segmentation Fault）

**可能原因**：

1. `main.cpp` 中的 spdlog 初始化失败——检查 spdlog 是否正确链接
2. 测试代码中访问了未初始化的内存——检查 `tTVPRect` 的默认构造函数是否被正确调用
3. `union` 内存布局在不同编译器下行为不同——确保没有依赖未定义行为

**解决方案**：

```bash
# 使用 AddressSanitizer 检测内存问题
cmake --preset="Linux Debug Config" \
    -DCMAKE_CXX_FLAGS="-fsanitize=address -fno-omit-frame-pointer"
cmake --build --preset="Linux Debug Build"

# 运行，ASan 会报告详细的内存错误位置
./out/linux/debug/tests/unit-tests/core/visual/rect_test
```

### 错误 3：CTest 找不到任何测试

```
Test project /path/to/out/linux/debug
No tests were found!!!
```

**可能原因**：

1. `catch_discover_tests()` 在构建后阶段执行失败——测试可执行文件无法运行
2. `SOURCES` 列表为空——忘记添加 `rect_test.cpp`
3. CMake 缓存过期——删除构建目录重新配置

**解决方案**：

```bash
# 验证可执行文件是否存在
ls out/linux/debug/tests/unit-tests/core/visual/rect_test

# 手动运行 --list-tests 检查
./out/linux/debug/tests/unit-tests/core/visual/rect_test --list-tests

# 如果输出正常但 CTest 找不到，清理缓存重建
rm -rf out/linux/debug

cmake --build --preset="Linux Debug Build"
ctest --test-dir out/linux/debug --output-on-failure
```

## 本节小结

- **注册测试只需一行**：将 `rect_test.cpp` 添加到 `SOURCES` 变量，CMake 的 foreach 循环自动完成编译、链接、测试发现
- **CTest 是调度层**：负责发现、执行、汇总测试；Catch2 是执行层，负责运行具体的 TEST_CASE 和断言
- **`catch_discover_tests` 是桥梁**：在构建后阶段运行 `--list-tests`，将 Catch2 的 TEST_CASE 注册为 CTest 用例
- **三平台构建流程一致**：只是预设名称不同（Windows/Linux/MacOS Debug Config/Build）
- **CI 目前不运行测试**：三个构建工作流都使用 `ENABLE_TESTS=OFF`，只做编译验证
- **测试 CI 需要单独的工作流**：使用 Debug 构建 + `ENABLE_TESTS=ON` + `ctest --output-on-failure`
- **覆盖率需要额外编译标志**：GCC 用 `--coverage`，Clang 用 `-fprofile-instr-generate`，MSVC 用 OpenCppCoverage
- **基准测试用 Catch2 BENCHMARK 宏**：应在 Release 构建下运行，使用 `--benchmark-samples` 控制采样次数
- **Android/iOS 不编译测试**：`ENABLE_TESTS AND NOT (IOS OR ANDROID)` 条件排除了移动平台

## 练习题与答案

### 题目 1：如果你想在 CI 中运行测试，但又不想每次都从零编译 vcpkg 依赖，应该怎么做？

<details>
<summary>查看答案</summary>

利用项目已有的 **vcpkg 二进制缓存**机制。在工作流的 `env` 中配置：

```yaml
env:
  VCPKG_BINARY_SOURCES: "clear;nuget,https://nuget.pkg.github.com/${{ github.repository_owner }}/index.json,readwrite"
```

这会将编译好的 vcpkg 包上传到 GitHub Packages 的 NuGet 源。下次 CI 运行时，vcpkg 会先检查缓存中是否已有相同版本的包，如果有就直接下载二进制文件，跳过源码编译。

此外，项目还使用了 `ccache`（编译器缓存）来加速 C++ 源码编译：

```yaml
- name: ccache
  uses: hendrikmuhs/ccache-action@v1.2.18
  with:
    key: test-linux
```

两者配合可以将首次构建从 30+ 分钟缩短到后续构建的 5-10 分钟。

需要注意的是，vcpkg 二进制缓存需要 `NUGET_API_KEY` Secret，这需要在 GitHub 仓库的 Settings → Secrets and variables → Actions 中配置。

</details>

### 题目 2：编写一个 CTest 命令，要求：只运行名称中包含"集合运算"的测试，使用 4 个并行进程，单个测试超时 5 秒，如果失败则打印详细输出。

<details>
<summary>查看答案</summary>

```bash
ctest --test-dir out/linux/debug \
    -R "集合运算" \
    -j 4 \
    --timeout 5 \
    --output-on-failure
```

参数解读：
- `--test-dir out/linux/debug`：指定构建目录（根据平台替换）
- `-R "集合运算"`：正则匹配，只运行名称包含"集合运算"的测试
- `-j 4`：使用 4 个并行进程执行测试
- `--timeout 5`：每个测试最多运行 5 秒，超时视为失败
- `--output-on-failure`：只有失败的测试才打印详细输出

如果你还想把结果保存到文件，可以加上 `--output-log test_log.txt`。

如果想要反向过滤（排除"集合运算"，运行其他所有测试），把 `-R` 换成 `-E`：

```bash
ctest --test-dir out/linux/debug -E "集合运算" -j 4
```

</details>

### 题目 3：项目的 CI 工作流为什么使用 `workflow_run` 而不是直接在同一个工作流中先检查格式再构建？这种设计有什么优缺点？

<details>
<summary>查看答案</summary>

**为什么用 `workflow_run`：**

`workflow_run` 让格式检查和构建分成独立的工作流，通过事件触发串联。格式检查完成后，触发 `workflow_run` 事件，构建工作流检查 `github.event.workflow_run.conclusion == 'success'` 后才执行。

**对比两种设计：**

**方案 A：单工作流多 Job（用 `needs`）**

```yaml
jobs:
  format-check:
    runs-on: ubuntu-latest
    steps: [...]
  build-linux:
    needs: format-check    # 等待格式检查完成
    runs-on: ubuntu-latest
    steps: [...]
  build-windows:
    needs: format-check
    runs-on: windows-latest
    steps: [...]
```

优点：在同一个工作流中管理，逻辑清晰；共享同一次 `push` 事件上下文。

缺点：所有 Job 共用同一个 workflow run，无法单独重跑构建（不重跑格式检查）。

**方案 B：多工作流 + `workflow_run`（项目当前方案）**

```yaml
# code-format-check.yml
on: push/pull_request

# build-linux.yml
on:
  workflow_run:
    workflows: ["Code Format Check"]
    types: ["completed"]
```

优点：
1. 每个工作流独立管理，可以单独通过 `workflow_dispatch` 手动触发
2. 构建工作流可以独立重跑，不需要重跑格式检查
3. 触发条件更灵活——格式检查只对 C++ 文件变更触发，但构建可以有不同的触发条件

缺点：
1. 复杂度更高——需要理解 `workflow_run` 事件机制
2. `workflow_run` 在 fork 的 PR 上**不会触发**（GitHub 安全限制）
3. 状态检查不直观——GitHub UI 上显示为多个独立的 workflow run，而不是一个串联的流水线
4. 有延迟——`workflow_run` 事件是异步触发的，构建开始会比 `needs` 方案晚几秒

项目选择方案 B，主要是因为各平台构建相互独立，且需要支持手动触发单个构建的场景。

</details>
