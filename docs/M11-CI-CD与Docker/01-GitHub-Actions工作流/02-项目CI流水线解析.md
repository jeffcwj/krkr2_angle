## 术语预览

本节将涉及以下术语，先有个印象，正文中会逐一详细讲解：

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| 门禁工作流 | Gate Workflow | 作为前置检查的工作流——只有它通过，后续工作流才会真正执行 |
| 链式触发 | Chained Trigger | 一个工作流完成后自动触发另一个工作流，通过 `workflow_run` 事件实现 |
| NuGet 源 | NuGet Source | NuGet 包管理器的远程存储地址，vcpkg 用它来缓存预编译的 C++ 依赖包 |
| Mono | Mono | .NET 运行时的开源实现，Linux 上运行 NuGet CLI 需要 Mono（因为 NuGet CLI 是 .NET 程序） |
| ccache | ccache | C/C++ 编译缓存工具，记住之前编译过的文件，再次编译时直接返回缓存结果 |
| 构建产物 | Artifact | CI 构建过程中生成的文件（如可执行文件、APK），可以上传到 GitHub 供下载 |
| ACT | ACT | 一个在本地模拟运行 GitHub Actions 工作流的开源工具，用于调试 CI 配置 |

## 第一步：总览——四工作流架构

KrKr2 项目的 CI 由 4 个工作流文件组成，它们之间存在明确的依赖关系：

```
                    ┌─────────────────────────────────────┐
                    │     push 到 main / PR 到 main       │
                    │     或手动 workflow_dispatch         │
                    └──────────────┬──────────────────────┘
                                   │
                                   ▼
                    ┌─────────────────────────────────────┐
                    │  code-format-check.yml (40 行)      │
                    │  ─────────────────────────          │
                    │  Runner: ubuntu-latest               │
                    │  任务: clang-format 20 格式检查      │
                    │  检查范围: cpp/ 和 platforms/        │
                    └──────────────┬──────────────────────┘
                                   │ workflow_run: completed
                    ┌──────────────┼──────────────┐
                    │              │              │
                    ▼              ▼              ▼
         ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
         │build-linux   │ │build-windows │ │build-android │
         │.yml (100行)  │ │.yml (84行)   │ │.yml (169行)  │
         │──────────    │ │──────────    │ │──────────    │
         │Runner:       │ │Runner:       │ │Runner:       │
         │ubuntu-latest │ │windows-latest│ │ubuntu-latest │
         │              │ │              │ │              │
         │if: success   │ │if: success   │ │if: success   │
         └──────────────┘ └──────────────┘ └──────────────┘
```

**关键设计决策：**

1. **格式检查是门禁** — 格式不合规，三个构建工作流全部跳过（不浪费 CI 资源）
2. **三构建并行** — Linux、Windows、Android 同时开始构建，互不影响
3. **所有构建禁用测试** — 所有工作流都使用 `-DENABLE_TESTS=OFF`，CI 只验证编译通过

## 第二步：逐行解读 code-format-check.yml

这是 KrKr2 CI 的"入口"——所有后续构建都依赖它的成功。

```yaml
# .github/workflows/code-format-check.yml
# 完整文件共 40 行

name: Code Format Check              # 第 1 行：工作流名称
                                      # 显示在 GitHub Actions 页面和其他工作流的 workflow_run 引用中

on:                                   # 第 3 行：触发器定义
  workflow_dispatch:                  # 第 4 行：支持手动触发（GitHub 页面点击 Run workflow）

  push:                               # 第 6 行：push 触发
    branches: [ main ]               # 第 7 行：只在 main 分支的推送触发
    paths:                            # 第 8 行：路径过滤——只有以下文件有改动才触发
      - 'cmake/**'                   # CMake 辅助脚本
      - 'cpp/**'                     # C++ 源码（核心 + 外部 + 插件）
      - 'platforms/**'               # 平台入口代码
      - 'vcpkg/**'                   # vcpkg overlay ports
      - 'CMakeLists.txt'             # 根 CMake 文件
      - 'CMakePresets.json'          # CMake 预设
      - 'vcpkg.json'                 # vcpkg 依赖清单
      - 'vcpkg-configuration.json'   # vcpkg 配置
      - '.github/workflows/**'       # 工作流文件自身
  pull_request:                       # 第 18 行：PR 触发（相同的过滤条件）
    branches: [ main ]
    paths:                            # 第 20-29 行：与 push 完全相同的 paths
      - 'cmake/**'
      - 'cpp/**'
      - 'platforms/**'
      - 'vcpkg/**'
      - 'CMakeLists.txt'
      - 'CMakePresets.json'
      - 'vcpkg.json'
      - 'vcpkg-configuration.json'
      - '.github/workflows/**'

jobs:                                 # 第 31 行：作业定义
  format-check:                       # 第 32 行：作业 ID
    runs-on: ubuntu-latest           # 第 33 行：运行在 Ubuntu Runner 上
    steps:                            # 第 34 行：步骤列表
      - uses: actions/checkout@v4    # 第 35 行：检出代码
      - uses: DoozyX/clang-format-lint-action@v0.20  # 第 36 行：调用格式检查 Action
        with:                         # 第 37 行：Action 参数
          source: 'cpp platforms'    # 第 38 行：检查 cpp/ 和 platforms/ 目录
          extensions: 'cpp,cc,h,hpp,inc'  # 第 39 行：检查的文件扩展名
          clangFormatVersion: 20     # 第 40 行：使用 clang-format 版本 20
```

**逐行分析要点：**

**第 8-17 行的 `paths` 过滤意义重大：**

如果你只修改了 `docs/` 目录下的文档文件，或者只修改了 `README.md`，格式检查**不会触发**。这避免了文档修改也要等 CI 运行的尴尬。但注意 `.github/workflows/**` 也在监听列表中——修改工作流文件本身会触发检查，这是合理的（修改 CI 配置当然要验证）。

**第 36-40 行的 DoozyX/clang-format-lint-action：**

这个第三方 Action 的工作原理是：
1. 在 Runner 上安装 clang-format 20
2. 递归扫描 `source` 指定目录下的所有 `extensions` 匹配文件
3. 对每个文件运行 `clang-format --dry-run --Werror`（干运行 + 有格式差异就报错）
4. 如果任何文件格式不符合 `.clang-format` 配置，Action 返回非零退出码，步骤失败

**注意 `source: 'cpp platforms'` 只检查两个目录：**

- `cpp/` — 所有 C++ 核心代码和插件代码
- `platforms/` — 平台入口代码

**不检查的目录：** `tests/`（测试代码）、`tools/`（工具代码）。这意味着测试代码和工具代码的格式不在 CI 门禁范围内。

## 第三步：逐行解读 build-linux.yml

Linux 构建是三个构建工作流中最具代表性的，包含了完整的依赖安装、vcpkg 配置、NuGet 缓存和编译流程。

```yaml
# .github/workflows/build-linux.yml
# 完整文件共 100 行

name: Build Linux                     # 第 1 行：工作流名称

on:                                   # 第 3 行：触发器
  workflow_dispatch:                  # 第 4 行：手动触发

  workflow_run:                       # 第 6 行：链式触发
    workflows: ["Code Format Check"] # 第 7 行：监听格式检查工作流
    types: [ "completed" ]           # 第 8 行：当格式检查完成时触发

env:                                  # 第 10 行：工作流级环境变量
  VCPKG_BINARY_SOURCES: "clear;nuget,https://nuget.pkg.github.com/${{ github.repository_owner }}/index.json,readwrite"
  # 第 11 行：vcpkg 二进制缓存配置
  # clear          — 清除默认缓存源
  # nuget,...      — 使用 GitHub Packages 的 NuGet 作为缓存后端
  # readwrite      — 既读取缓存（加速），也写入缓存（供后续使用）

  VCPKG_INSTALL_OPTIONS: "--debug"   # 第 12 行：vcpkg 安装时输出调试信息
```

**第 14-20 行：作业定义**

```yaml
jobs:
  build-linux:                        # 第 15 行：作业 ID
    if: ${{ github.event.workflow_run.conclusion == 'success' }}
    # 第 16 行：关键条件！
    # workflow_run 的 completed 包括成功和失败
    # 这里只有格式检查"成功"时才真正执行构建
    # 如果格式检查失败，这个 job 会被跳过（Skip）

    runs-on: ubuntu-latest           # 第 17 行：Ubuntu Runner
    env:
      VCPKG_ROOT: ${{ github.workspace }}/.act/vcpkg
      # 第 19 行：vcpkg 安装路径
      # 注意 .act 子目录——这暗示开发者使用 ACT 工具在本地测试过
```

**第 21-52 行：环境准备步骤**

**第 54-79 行：工具链配置**

```yaml
      - name: Install CMake           # 第 54 行
        uses: lukka/get-cmake@latest
        with:
          cmakeVersion: 3.31.1        # 指定 CMake 3.31.1（与 CMakePresets.json 兼容）

      - name: Setup vcpkg             # 第 59 行
        run: |
          if [ ! -d "${{ env.VCPKG_ROOT }}" ]; then
            git clone https://github.com/microsoft/vcpkg.git ${{ env.VCPKG_ROOT }}
          fi
          ${{ env.VCPKG_ROOT }}/bootstrap-vcpkg.sh
          # 如果 vcpkg 目录不存在，克隆仓库
          # bootstrap-vcpkg.sh 编译 vcpkg 可执行文件
          # 注意：没有 pin 到特定 commit！
          # 这意味着每次 CI 运行可能使用不同版本的 vcpkg

      - name: Add NuGet sources       # 第 67 行：配置 NuGet 缓存源
        shell: bash
        run: |
          mono `${{ env.VCPKG_ROOT }}/vcpkg fetch nuget | tail -n 1` \
            sources add \
            -Source "https://nuget.pkg.github.com/${{ github.repository_owner }}/index.json" \
            -StorePasswordInClearText \
            -Name GitHubPackages \
            -UserName "${{ github.actor }}" \
            -Password "${{ secrets.NUGET_API_KEY }}"
          # 分解这行命令：
          # 1. vcpkg fetch nuget     — 让 vcpkg 下载 NuGet CLI
          # 2. | tail -n 1           — 取最后一行（NuGet 可执行文件路径）
          # 3. mono <nuget.exe>      — 用 Mono 运行 NuGet CLI
          # 4. sources add ...       — 添加 GitHub Packages 作为 NuGet 源
          # 5. -Password             — 使用仓库密钥 NUGET_API_KEY 认证

          mono `${{ env.VCPKG_ROOT }}/vcpkg fetch nuget | tail -n 1` \
            setapikey "${{ secrets.NUGET_API_KEY }}" \
            -Source "https://nuget.pkg.github.com/${{ github.repository_owner }}/index.json"
          # 设置 API Key，允许 vcpkg 向 GitHub Packages 推送（写入）缓存

      - name: ccache                   # 第 81 行：编译缓存
        uses: hendrikmuhs/ccache-action@v1.2.18
        with:
          key: ${{ github.job }}-${{ matrix.os }}
          # 缓存 key 包含作业名和操作系统
          # ccache 会缓存编译结果，二次构建时如果源文件没变，直接返回缓存
```

**第 86-100 行：构建和上传**


## 第四步：逐行解读 build-windows.yml

Windows 构建的主要差异在于 MSVC 编译器设置和 PowerShell 脚本语法。

```yaml
# .github/workflows/build-windows.yml 关键差异部分

jobs:
  build-windows:
    if: ${{ github.event.workflow_run.conclusion == 'success' }}
    runs-on: windows-latest          # Windows Runner
    env:
      VCPKG_ROOT: ${{ github.workspace }}/.act/vcpkg
    steps:
      - uses: actions/checkout@v4

      - name: Setup MSVC environment  # 第 23 行：MSVC 环境
        uses: ilammy/msvc-dev-cmd@v1
        with:
          arch: amd64                 # 64 位编译环境
        # 这个 Action 等价于运行 vcvarsall.bat amd64
        # 设置 cl.exe、link.exe、lib.exe 等 MSVC 工具链的 PATH

      - name: Normalize and export VCPKG_ROOT  # 第 28 行
        run: |
          $normalized = "${{ github.workspace }}/.act/vcpkg" -replace '\\', '/'
          echo "VCPKG_ROOT=$normalized" | Out-File -FilePath $env:GITHUB_ENV -Encoding utf8 -Append
        # Windows 路径使用反斜杠（C:\），但 vcpkg 和 CMake 更喜欢正斜杠
        # 这里将路径标准化为正斜杠格式

      - name: Install dependencies    # 第 33 行
        run: |
          Invoke-WebRequest -Uri "https://github.com/lexxmark/winflexbison/releases/download/v2.5.25/win_flex_bison-2.5.25.zip" -OutFile "$env:TEMP\winflexbison.zip"
          Expand-Archive -Path "$env:TEMP\winflexbison.zip" -DestinationPath "C:\winflexbison"
          echo "C:\winflexbison" | Out-File -Append -Encoding ascii $env:GITHUB_PATH
          # 安装 winflexbison（Windows 版的 flex + bison）
          # TJS2 的语法文件（.y）需要 bison 编译

          choco install nasm python3 -y
          # 使用 Chocolatey 包管理器安装 NASM 和 Python3
          # Chocolatey 预装在 GitHub Windows Runner 上
```

**Windows 与 Linux 的关键差异对比：**

| 对比项 | Linux (build-linux.yml) | Windows (build-windows.yml) |
|--------|------------------------|----------------------------|
| Runner | `ubuntu-latest` | `windows-latest` |
| 编译器 | gcc/g++（apt 安装） | MSVC（`ilammy/msvc-dev-cmd` Action） |
| Shell | bash（默认） | PowerShell（默认） |
| bison | `apt install bison` | winflexbison（手动下载 zip） |
| 包管理器 | apt-get | Chocolatey (choco) |
| NuGet 运行 | `mono nuget.exe`（需要 Mono） | 直接 `nuget.exe`（.NET 原生） |
| CMake Preset | "Linux Release Config" | "Windows Release Config" |
| 产物路径 | `out/linux/release/bin/krkr2/**` | `out/windows/release/bin/krkr2/**` |

## 第五步：逐行解读 build-android.yml

Android 构建是最复杂的工作流（169 行），因为需要额外安装 Android SDK、NDK 和 JDK。

```yaml
# .github/workflows/build-android.yml 关键部分

env:
  ANDROID_NDK_VERSION: "28.0.13004108"  # 固定 NDK 版本
  VCPKG_BINARY_SOURCES: "clear;nuget,..."
  VCPKG_INSTALL_OPTIONS: "--debug"

jobs:
  build-android:
    if: ${{ github.event.workflow_run.conclusion == 'success' }}
    runs-on: ubuntu-latest            # Android 构建也用 Ubuntu（交叉编译）
    env:
      VCPKG_ROOT: ${{ github.workspace }}/.act/vcpkg
      ANDROID_SDK_ROOT: ${{ github.workspace }}/.act/android-sdk
      ANDROID_NDK: ${{ github.workspace }}/.act/android-ndk
```

**Android 特有的步骤：**

```yaml
      - uses: actions/checkout@v4
        with:
          lfs: true                   # Android 构建需要 Git LFS 文件
          # 可能有大文件（预编译库、资源文件）存储在 LFS 中

      - name: Free disk space         # 释放磁盘空间（Android 构建需要更多空间）
        run: |
          sudo rm -rf /usr/share/dotnet         # ~6 GB
          sudo rm -rf /usr/local/lib/android    # ~10 GB（预装 Android SDK）
          sudo rm -rf /opt/ghc                  # ~2 GB
          sudo rm -rf /opt/hostedtoolcache      # ~5 GB
          sudo rm -rf /usr/local/share/boost    # ~1 GB
          # 总共释放约 24 GB 磁盘空间
          # Android 构建需要下载 SDK + NDK + vcpkg 依赖，非常吃空间

      - name: Setup JDK 17            # Android 构建必须有 JDK
        uses: actions/setup-java@v3
        with:
          java-version: '17'          # JDK 17（Gradle 8.x 要求）
          distribution: 'zulu'        # 使用 Azul Zulu 发行版
```

**Android SDK/NDK 安装流程：**


## 第六步：关键设计决策分析

### 决策 1：为什么所有构建都禁用测试？

三个构建工作流都使用 `-DENABLE_TESTS=OFF`：

```yaml
# build-linux.yml 第 88 行
cmake --preset="Linux Release Config" -DENABLE_TESTS=OFF -DBUILD_TOOLS=OFF

# build-windows.yml 第 75 行
cmake --preset="Windows Release Config" -DENABLE_TESTS=OFF -DBUILD_TOOLS=OFF
```

**可能的原因：**
1. **CI 时间限制** — 启用测试需要编译 Catch2 和测试代码，增加编译时间
2. **测试依赖** — 某些测试可能需要运行时环境（如图形界面、音频设备）
3. **尚未完善** — 项目的测试覆盖率较低，大部分测试目标的 SOURCES 为空（参见 M10）
4. **策略选择** — CI 只保证"能编译通过"，测试在本地开发环境运行

> **改进建议：** 可以新增一个独立的测试工作流，在 Linux 构建后运行单元测试。详见 M10 中设计的 `run-tests.yml` 方案。

### 决策 2：为什么使用 workflow_run 而不是单工作流多 job？

**方案 A（当前）：** 4 个独立工作流文件，通过 `workflow_run` 链接

```
code-format-check.yml → build-linux.yml
                       → build-windows.yml
                       → build-android.yml
```

**方案 B（替代）：** 1 个工作流文件，4 个 job 用 `needs` 链接

```yaml
jobs:
  format-check: { ... }
  build-linux: { needs: format-check, ... }
  build-windows: { needs: format-check, ... }
  build-android: { needs: format-check, ... }
```

| 对比维度 | 方案 A (workflow_run) | 方案 B (单文件 needs) |
|---------|----------------------|---------------------|
| 文件组织 | 4 个小文件，职责清晰 | 1 个大文件，可能数百行 |
| 独立重试 | 可以单独重跑某个平台 | 必须重跑整个工作流 |
| 权限隔离 | 每个工作流可以有独立权限 | 共享同一权限配置 |
| 触发灵活性 | 每个构建可以有独立触发条件 | 所有 job 共享触发条件 |
| 调试体验 | 各平台的日志独立查看 | 需要展开看特定 job |
| 缺点 | `workflow_run` 有延迟（排队）；代码有重复 | 文件过长；无法独立重试 |

### 决策 3：vcpkg 未固定版本

```yaml
# build-linux.yml 第 62 行
git clone https://github.com/microsoft/vcpkg.git ${{ env.VCPKG_ROOT }}
# 每次都 clone 最新版本！
```

这是一个**风险点**——vcpkg 主分支的更新可能导致某些端口编译失败。更安全的做法是固定到特定 commit：

```yaml
# 推荐的安全做法
git clone https://github.com/microsoft/vcpkg.git ${{ env.VCPKG_ROOT }}
cd ${{ env.VCPKG_ROOT }}
git checkout 2024.12.16  # 固定到已知可用的版本标签
./bootstrap-vcpkg.sh
```

## 常见错误与解决方案

### 错误 1：workflow_run 不触发

**症状：** 格式检查通过了，但构建工作流没有开始。
**原因：** `workflow_run` 的 `workflows` 字段必须与目标工作流的 `name` **完全匹配**（区分大小写和空格）。
**解决：** 确认 `workflows: ["Code Format Check"]` 与 `code-format-check.yml` 中的 `name: Code Format Check` 完全一致。

### 错误 2：NuGet 认证失败

```
error: Response status code does not indicate success: 401 (Unauthorized).
```

**原因：** `NUGET_API_KEY` 密钥过期、拼写错误或权限不足。
**解决：**
1. 在 GitHub 设置中重新生成 Personal Access Token（需要 `write:packages` 权限）
2. 更新仓库的 `NUGET_API_KEY` Secret
3. 确保 Token 的 scope 包含 `read:packages` 和 `write:packages`

### 错误 3：磁盘空间不足

```
Error: No space left on device
```

**原因：** vcpkg 编译大型依赖（FFmpeg、OpenCV）时消耗大量磁盘。
**解决：** 添加磁盘清理步骤（参考 build-android.yml 的 "Free disk space" 步骤），或使用更大磁盘的 Runner。

## 本节小结

- KrKr2 的 CI 由 **4 个工作流文件**组成：1 个格式检查（门禁）+ 3 个平台构建
- 使用 **`workflow_run` 链式触发**实现门禁机制——格式检查失败时，三个构建自动跳过
- Linux 构建需要安装 **Mono** 运行时来执行 NuGet CLI，用于 vcpkg 二进制缓存
- Windows 构建使用 **`ilammy/msvc-dev-cmd`** Action 设置 MSVC 环境，用 **Chocolatey** 安装依赖
- Android 构建最复杂（169 行），需要安装 SDK/NDK/JDK，并在 CI 中动态生成签名密钥
- **所有构建都禁用测试**（`-DENABLE_TESTS=OFF`），CI 只验证编译通过
- **vcpkg 未固定版本**是潜在风险点，建议在生产环境中固定到特定 commit
- **ccache** 用于加速重复编译，通过 `hendrikmuhs/ccache-action` 集成

### 题目 2：解释 NuGet 缓存配置

解释以下环境变量的含义，以及如果删除这个变量会发生什么：

```yaml
VCPKG_BINARY_SOURCES: "clear;nuget,https://nuget.pkg.github.com/jeffcwj/index.json,readwrite"
```

<details>
<summary>查看答案</summary>

**分段解读：**

1. **`clear`** — 清除 vcpkg 的所有默认缓存源（默认缓存源包括本地文件系统缓存 `~/.cache/vcpkg/archives`）。从一个干净的状态开始，确保只使用显式指定的缓存源。

2. **`nuget,https://nuget.pkg.github.com/jeffcwj/index.json,readwrite`** — 添加一个 NuGet 协议的缓存源：
   - `nuget` — 协议类型（使用 NuGet 协议存取包）
   - `https://nuget.pkg.github.com/jeffcwj/index.json` — NuGet 源地址（GitHub Packages 的 NuGet 端点）
   - `readwrite` — 权限模式。`read` = 只从缓存读取；`write` = 只向缓存写入；`readwrite` = 读写兼备

**完整工作流程：**

```
vcpkg 需要安装某个依赖（如 ffmpeg）
  │
  ├─ 计算该依赖的哈希值（包含版本 + 编译选项 + 平台）
  ├─ 向 GitHub Packages NuGet 源查询：有这个哈希的预编译包吗？
  │     ├─ 命中 → 直接下载预编译包，跳过编译（节省 5-30 分钟）
  │     └─ 未命中 → 从源码编译
  │           └─ 编译完成后，将产物打包推送到 GitHub Packages（供下次使用）
  └─ 继续安装下一个依赖
```

**如果删除这个变量：**

vcpkg 会使用默认的本地文件系统缓存（`~/.cache/vcpkg/archives`）。由于 GitHub Actions Runner 是临时虚拟机，每次运行结束后文件系统被销毁，本地缓存无法跨次复用。这意味着**每次 CI 运行都要从头编译所有依赖**，构建时间可能从 10 分钟暴增到 30-60 分钟。

</details>

### 题目 3：为什么 Android 构建需要释放磁盘空间？

Android 构建工作流在第 28-41 行大量删除预装工具。计算一下：GitHub Runner 默认有多少可用磁盘空间？Android 构建大约需要多少空间？

<details>
<summary>查看答案</summary>

**GitHub Runner 默认磁盘空间：** 大约 14 GB 可用空间（SSD 总容量约 84 GB，但预装了大量工具）。

**Android 构建所需空间估算：**

| 组件 | 大小（估算） |
|------|------------|
| Android SDK (platform-tools + platforms + build-tools) | ~2 GB |
| Android NDK 28 | ~3.5 GB |
| vcpkg (克隆 + 依赖编译中间文件) | ~5-10 GB |
| Gradle 缓存 + 构建中间文件 | ~2-3 GB |
| 源码 + 构建产物 | ~1-2 GB |
| **合计** | **~13-20 GB** |

**释放空间估算：**

| 删除目标 | 释放空间 |
|---------|---------|
| `/usr/share/dotnet` | ~6 GB |
| `/usr/local/lib/android` | ~10 GB |
| `/opt/ghc` | ~2 GB |
| `/opt/hostedtoolcache` | ~5 GB |
| `/usr/local/share/boost` | ~1 GB |
| **合计** | **~24 GB** |

清理后可用空间约为 14 + 24 = 38 GB，足够 Android 构建使用。不清理的话，14 GB 不够，构建中途会报 "No space left on device"。

</details>
