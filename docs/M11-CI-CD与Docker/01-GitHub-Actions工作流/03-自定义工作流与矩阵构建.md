## 术语预览

本节将涉及以下术语，先有个印象，正文中会逐一详细讲解：

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| 矩阵策略 | Matrix Strategy | 在一个 Job 中定义多个变量组合，GitHub Actions 自动为每个组合创建并行运行的实例 |
| 矩阵排除 | Matrix Exclude | 从自动生成的组合中移除特定的变量组合，避免不需要或不兼容的构建 |
| 矩阵包含 | Matrix Include | 向矩阵中额外添加特定的变量组合，或为已有组合附加额外变量 |
| 复合动作 | Composite Action | 将多个 `run` 步骤和 `uses` 步骤封装成一个可复用的自定义 Action |
| 可复用工作流 | Reusable Workflow | 一个完整的工作流文件，可被其他工作流通过 `workflow_call` 事件调用 |
| 快速失败 | Fail-fast | 矩阵策略的默认行为——当任何一个组合失败时，立即取消所有其他正在运行的组合 |
| 最大并行度 | Max-parallel | 控制矩阵中同时运行的 Job 数量上限，防止资源争抢 |

## 矩阵策略基础

### 什么是矩阵策略

矩阵策略（Matrix Strategy）是 GitHub Actions 中最强大的并行化机制。它允许你在一个 Job 定义中声明多个变量，GitHub Actions 会自动计算所有变量的笛卡尔积（Cartesian Product，即所有可能的组合），并为每个组合创建一个独立的 Job 实例并行运行。这对于需要在多个操作系统、多个编译器版本、多种构建配置下测试同一份代码的 C++ 项目来说，是不可或缺的功能。

举个最简单的例子——假设你要在 3 个操作系统和 2 种构建类型下测试代码，没有矩阵策略你需要手写 6 个几乎一样的 Job；而使用矩阵策略，只需定义两个变量即可：

```yaml
# 示例 1：最简矩阵——2 个变量自动组合为 6 个 Job
name: 矩阵构建示例

on:
  push:
    branches: [main]

jobs:
  build:
    # strategy.matrix 定义变量及其取值列表
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]  # 3 个操作系统
        build_type: [Debug, Release]                        # 2 种构建类型
    
    # ${{ matrix.os }} 引用当前组合中 os 变量的值
    runs-on: ${{ matrix.os }}
    
    steps:
      - uses: actions/checkout@v4
      
      - name: 配置 CMake
        # ${{ matrix.build_type }} 引用当前组合中的构建类型
        run: cmake -B build -DCMAKE_BUILD_TYPE=${{ matrix.build_type }}
      
      - name: 编译
        run: cmake --build build --config ${{ matrix.build_type }}
      
      - name: 测试
        run: ctest --test-dir build --output-on-failure -C ${{ matrix.build_type }}
```

上面的配置会自动生成以下 6 个并行 Job：

```
┌──────────────────────────────────────────────────────┐
│             矩阵展开（笛卡尔积）                      │
├──────────────────────┬───────────────────────────────┤
│ Job 实例             │ 变量值                         │
├──────────────────────┼───────────────────────────────┤
│ build (1/6)          │ os=ubuntu-latest,  type=Debug  │
│ build (2/6)          │ os=ubuntu-latest,  type=Release│
│ build (3/6)          │ os=windows-latest, type=Debug  │
│ build (4/6)          │ os=windows-latest, type=Release│
│ build (5/6)          │ os=macos-latest,   type=Debug  │
│ build (6/6)          │ os=macos-latest,   type=Release│
└──────────────────────┴───────────────────────────────┘
```

### 矩阵变量的引用方式

矩阵中定义的每个变量，都可以通过 `${{ matrix.<变量名> }}` 在 Job 的任何位置引用。常见的引用场景包括：

```yaml
jobs:
  build:
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest]
        compiler: [gcc, clang]
    
    # 1. 在 runs-on 中引用——决定运行在哪个操作系统上
    runs-on: ${{ matrix.os }}
    
    steps:
      # 2. 在 step 的 name 中引用——让日志更清晰
      - name: 使用 ${{ matrix.compiler }} 编译
        run: |
          # 3. 在 run 脚本中引用——控制编译器选择
          if [ "${{ matrix.compiler }}" = "gcc" ]; then
            export CC=gcc CXX=g++
          else
            export CC=clang CXX=clang++
          fi
          cmake -B build -DCMAKE_C_COMPILER=$CC -DCMAKE_CXX_COMPILER=$CXX
          cmake --build build
      
      # 4. 在 env 中引用——设置环境变量
      - name: 运行测试
        env:
          BUILD_COMPILER: ${{ matrix.compiler }}
        run: echo "当前编译器：$BUILD_COMPILER"
      
      # 5. 在 if 条件中引用——条件执行
      - name: 仅 GCC 执行的步骤
        if: matrix.compiler == 'gcc'
        run: echo "这一步只在 GCC 组合中运行"
```

### fail-fast 与 max-parallel

矩阵策略有两个重要的控制参数：

**`fail-fast`**（快速失败）：默认值为 `true`。当矩阵中任何一个 Job 失败时，GitHub Actions 会立即取消所有其他正在运行的 Job。这在 CI 中通常是合理的——如果 Linux + Debug 组合编译失败，其他组合大概率也会有问题，没必要继续浪费计算资源。但有时候你希望看到所有平台的完整结果（比如只是某一个平台有特定 Bug），这时需要关闭快速失败：

**`max-parallel`**（最大并行度）：控制同时运行的矩阵 Job 数量上限。默认情况下 GitHub Actions 会尽可能多地并行运行（受账户的并发 Runner 限制）。如果你的矩阵很大（比如 20 个组合）且步骤中会访问共享资源（如 NuGet 包源、Docker Registry），可以限制并行度避免竞争条件：

```yaml
# 示例 2：控制 fail-fast 和 max-parallel
jobs:
  comprehensive-test:
    strategy:
      fail-fast: false     # 关闭快速失败，所有组合都会运行到完成
      max-parallel: 4      # 最多同时运行 4 个 Job
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        build_type: [Debug, Release]
        compiler: [gcc, msvc, clang]
    
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - name: 构建 (${{ matrix.os }}, ${{ matrix.compiler }}, ${{ matrix.build_type }})
        run: echo "构建中..."
```

## 矩阵的高级用法

### exclude——排除特定组合

矩阵的笛卡尔积会生成所有可能的组合，但并非所有组合都是有意义的。比如，Windows 上通常不用 GCC，macOS 上通常不用 MSVC。`exclude` 关键字允许你移除不需要的组合：

```yaml
# 示例 3：排除不兼容的编译器-平台组合
jobs:
  build:
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        compiler: [gcc, msvc, clang]
        exclude:
          # Windows 不用 GCC（MinGW 场景除外，此处简化）
          - os: windows-latest
            compiler: gcc
          # macOS 不用 MSVC
          - os: macos-latest
            compiler: msvc
          # Linux 不用 MSVC
          - os: ubuntu-latest
            compiler: msvc
    
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - name: 编译 (${{ matrix.os }} + ${{ matrix.compiler }})
        run: echo "使用 ${{ matrix.compiler }} 在 ${{ matrix.os }} 上构建"
```

排除后，实际运行的组合从 9 个减少到 6 个：

```
排除前（9 个组合）：              排除后（6 个组合）：
ubuntu  + gcc    ✅               ubuntu  + gcc    ✅
ubuntu  + msvc   ❌ (excluded)    ubuntu  + clang  ✅
ubuntu  + clang  ✅               windows + msvc   ✅
windows + gcc    ❌ (excluded)    windows + clang  ✅
windows + msvc   ✅               macos   + gcc    ✅
windows + clang  ✅               macos   + clang  ✅
macos   + gcc    ✅
macos   + msvc   ❌ (excluded)
macos   + clang  ✅
```

### include——添加额外组合或附加变量

`include` 关键字有两个用途：

**用途一：为已有组合附加额外变量。** 当某个组合需要特殊的环境变量、编译选项或步骤参数时，用 `include` 为它"打补丁"：

```yaml
# 示例 4：include 附加变量
jobs:
  build:
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        build_type: [Debug, Release]
        include:
          # 为 ubuntu + Release 组合添加代码覆盖率标志
          - os: ubuntu-latest
            build_type: Release
            coverage: true            # 新增的附加变量
            cmake_extra: "-DCODE_COVERAGE=ON"
          
          # 为 windows 组合指定 MSVC 版本
          - os: windows-latest
            msvc_arch: x64            # 所有 windows 组合都会获得此变量
          
          # 为 macOS 指定部署目标
          - os: macos-latest
            macos_target: "11.0"
    
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      
      - name: 配置 CMake
        run: |
          cmake -B build \
            -DCMAKE_BUILD_TYPE=${{ matrix.build_type }} \
            ${{ matrix.cmake_extra }}   # 只有匹配的组合才有值
      
      - name: 上传覆盖率报告
        # 只有 coverage=true 的组合才执行此步骤
        if: matrix.coverage == true
        run: echo "上传覆盖率报告..."
```

**用途二：添加全新的组合。** 如果 `include` 条目中的变量值不匹配任何已有组合，它会被当作一个全新的组合添加进来：

```yaml
# 示例 5：include 添加全新组合
jobs:
  build:
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest]
        build_type: [Debug]
        include:
          # 这个组合（macos + Release）不在原始矩阵中，作为新组合添加
          - os: macos-latest
            build_type: Release
            special_flag: true
    
    # 最终运行 3 个 Job：
    #   ubuntu  + Debug    （原始矩阵）
    #   windows + Debug    （原始矩阵）
    #   macos   + Release  （include 新增）
    runs-on: ${{ matrix.os }}
    steps:
      - run: echo "${{ matrix.os }} / ${{ matrix.build_type }}"
```

### 动态矩阵——运行时生成矩阵变量

在某些高级场景中，矩阵的变量取值不是固定的，而是需要根据运行时条件动态生成。比如，只在特定分支上运行完整矩阵，在 PR 中只运行最小集合。GitHub Actions 支持通过 `fromJSON()` 函数将 JSON 字符串解析为矩阵变量：

```yaml
# 示例 6：动态矩阵——根据分支选择不同的平台集合
name: 动态矩阵

on:
  push:
    branches: [main, develop]
  pull_request:

jobs:
  # 第一个 Job：计算矩阵变量
  prepare:
    runs-on: ubuntu-latest
    outputs:
      matrix-os: ${{ steps.set-matrix.outputs.os }}
    steps:
      - id: set-matrix
        run: |
          # main 分支运行完整矩阵，PR 只跑 Linux
          if [ "${{ github.ref }}" = "refs/heads/main" ]; then
            echo 'os=["ubuntu-latest","windows-latest","macos-latest"]' >> $GITHUB_OUTPUT
          else
            echo 'os=["ubuntu-latest"]' >> $GITHUB_OUTPUT
          fi
  
  # 第二个 Job：使用动态矩阵
  build:
    needs: prepare
    strategy:
      matrix:
        # fromJSON() 将字符串解析为 JSON 数组
        os: ${{ fromJSON(needs.prepare.outputs.matrix-os) }}
        build_type: [Debug, Release]
    
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - name: 构建
        run: cmake -B build -DCMAKE_BUILD_TYPE=${{ matrix.build_type }} && cmake --build build
```

这个技巧在大型项目中非常实用——PR 阶段只在 Linux 上快速验证，合并到 main 后才运行完整的三平台矩阵，大幅节省 CI 时间和费用。

## 为 KrKr2 设计矩阵构建

### 当前项目的 CI 结构回顾

在前一节中我们分析过，KrKr2 项目目前使用 4 个独立的工作流文件实现多平台构建：

```
.github/workflows/
├── code-format-check.yml    # 格式检查门禁
├── build-linux.yml          # Linux 构建（workflow_run 触发）
├── build-windows.yml        # Windows 构建（workflow_run 触发）
└── build-android.yml        # Android 构建（workflow_run 触发）
```

这种"每平台一个文件"的方式简单直观，但存在以下问题：

| 问题 | 说明 |
|------|------|
| 代码重复 | 每个文件都重复了 checkout、vcpkg 安装、ccache 配置等步骤 |
| 维护成本高 | 修改 vcpkg 版本或 ccache 配置需要同时改 3 个文件 |
| 扩展困难 | 新增平台（如 macOS）需要复制整个文件 |
| 无法矩阵化 | 无法轻松添加 Debug/Release 等构建变体 |

### 矩阵化重构方案

下面我们设计一个使用矩阵策略的统一构建工作流，将 3 个平台文件合并为 1 个：

```yaml
# 示例 7：KrKr2 多平台矩阵构建方案（设计稿）
# 文件路径：.github/workflows/build-matrix.yml（设计参考，非当前项目实际使用）
name: Multi-Platform Build

on:
  workflow_run:
    workflows: ["Code Format Check"]
    types: [completed]

jobs:
  build:
    # 只在格式检查通过后运行
    if: ${{ github.event.workflow_run.conclusion == 'success' }}
    
    strategy:
      fail-fast: false    # 某平台失败不影响其他平台
      matrix:
        include:
          # Linux 构建配置
          - platform: linux
            os: ubuntu-latest
            cmake_preset_config: "Linux Release Config"
            cmake_preset_build: "Linux Release Build"
            vcpkg_triplet: x64-linux
            extra_packages: "mono-complete nuget libx11-dev libgl-dev"
          
          # Windows 构建配置
          - platform: windows
            os: windows-latest
            cmake_preset_config: "Windows Release Config"
            cmake_preset_build: "Windows Release Build"
            vcpkg_triplet: x64-windows-static
            extra_packages: ""
          
          # Android 构建配置（特殊处理）
          - platform: android
            os: ubuntu-latest
            cmake_preset_config: ""           # Android 用 Gradle，不直接用 CMake preset
            cmake_preset_build: ""
            vcpkg_triplet: arm64-android
            extra_packages: "dos2unix"
    
    runs-on: ${{ matrix.os }}
    
    env:
      VCPKG_BINARY_SOURCES: "clear;nuget,GitHub,readwrite"
    
    steps:
      # ===== 通用步骤（所有平台共享） =====
      - name: 检出代码
        uses: actions/checkout@v4
        with:
          submodules: recursive
      
      - name: 安装 vcpkg
        run: |
          git clone https://github.com/microsoft/vcpkg.git $HOME/vcpkg
          if [ "${{ matrix.platform }}" = "windows" ]; then
            $HOME/vcpkg/bootstrap-vcpkg.bat
          else
            $HOME/vcpkg/bootstrap-vcpkg.sh
          fi
        shell: bash
      
      - name: 配置 ccache
        uses: hendrikmuhs/ccache-action@v1.2.18
        with:
          key: ${{ matrix.platform }}-release
      
      # ===== 平台特定步骤（条件执行） =====
      - name: 安装 Linux 依赖
        if: matrix.platform == 'linux'
        run: |
          sudo apt-get update
          sudo apt-get install -y ${{ matrix.extra_packages }}
      
      - name: 配置 MSVC（仅 Windows）
        if: matrix.platform == 'windows'
        uses: ilammy/msvc-dev-cmd@v1
        with:
          arch: x64
      
      - name: 配置 Android SDK（仅 Android）
        if: matrix.platform == 'android'
        run: |
          # 安装 Android NDK r28
          sdkmanager --install "ndk;28.0.13004108"
          echo "ANDROID_NDK=$ANDROID_HOME/ndk/28.0.13004108" >> $GITHUB_ENV
      
      # ===== 构建步骤（按平台分支） =====
      - name: CMake 构建（Linux/Windows）
        if: matrix.platform != 'android'
        run: |
          cmake --preset="${{ matrix.cmake_preset_config }}"
          cmake --build --preset="${{ matrix.cmake_preset_build }}"
      
      - name: Gradle 构建（Android）
        if: matrix.platform == 'android'
        run: |
          dos2unix platforms/android/gradlew
          chmod +x platforms/android/gradlew
          ./platforms/android/gradlew -p ./platforms/android assembleRelease
```

> **注意：** 上面的工作流是一个设计参考方案。KrKr2 项目当前仍使用独立文件的方式（参见 `.github/workflows/build-linux.yml` 等），因为各平台的构建步骤差异较大（Linux 用 CMake preset、Windows 用 PowerShell、Android 用 Gradle），完全合并需要较多的条件判断。在实际项目中，选择哪种方式取决于平台间的共性程度。

## 自定义 Action（Custom Action）

### 为什么需要自定义 Action

在 KrKr2 的 CI 工作流中，有大量重复的操作模式。比如"克隆 vcpkg + 配置 NuGet 二进制缓存"这个步骤在 `build-linux.yml`、`build-windows.yml`、`build-android.yml` 三个文件中几乎一样。自定义 Action 允许你把这些重复步骤封装成一个可复用的"函数"，在多个工作流或多个 Job 中调用。

GitHub Actions 支持三种自定义 Action：

| 类型 | 运行方式 | 适用场景 | 示例 |
|------|----------|----------|------|
| JavaScript Action | 在 Node.js 环境中运行 | 需要复杂逻辑、调用 API | 发送通知、解析日志 |
| Docker Action | 在 Docker 容器中运行 | 需要特定工具链环境 | 运行特定版本的编译器 |
| 复合动作（Composite） | 直接在 Runner 上运行多个步骤 | 封装常见的 Shell 步骤组合 | vcpkg 安装、ccache 配置 |

对于 C++ 项目的 CI，**复合动作（Composite Action）** 是最常用的类型，因为我们主要需要封装 Shell 命令序列，不需要 Node.js 或 Docker 的额外开销。

### 创建复合动作（Composite Action）

复合动作由一个 `action.yml` 文件定义，通常放在仓库的 `.github/actions/<action名>/` 目录下。下面我们为 KrKr2 的 vcpkg 安装步骤创建一个复合动作：

```yaml
# 示例 8：复合动作——封装 vcpkg 安装与 NuGet 缓存配置
# 文件路径：.github/actions/setup-vcpkg/action.yml
name: 'Setup vcpkg with NuGet Cache'
description: '克隆 vcpkg、引导构建并配置 NuGet 二进制缓存'

# 定义输入参数——调用者可以传入值
inputs:
  vcpkg-root:
    description: 'vcpkg 安装目录'
    required: false
    default: '${{ github.workspace }}/vcpkg'
  nuget-api-key:
    description: 'GitHub Packages NuGet API Key'
    required: true
  use-mono:
    description: '是否使用 Mono 运行 NuGet（Linux 需要）'
    required: false
    default: 'false'

# 定义输出——让调用者获取结果
outputs:
  vcpkg-root:
    description: 'vcpkg 安装路径'
    value: ${{ inputs.vcpkg-root }}

# 复合动作的核心：runs.steps
runs:
  using: 'composite'    # 声明这是复合动作
  steps:
    - name: 克隆 vcpkg
      shell: bash
      run: |
        # 克隆 vcpkg 到指定目录
        git clone https://github.com/microsoft/vcpkg.git ${{ inputs.vcpkg-root }}
        echo "VCPKG_ROOT=${{ inputs.vcpkg-root }}" >> $GITHUB_ENV
    
    - name: 引导 vcpkg
      shell: bash
      run: |
        # 根据平台选择引导脚本
        if [ "$RUNNER_OS" = "Windows" ]; then
          ${{ inputs.vcpkg-root }}/bootstrap-vcpkg.bat -disableMetrics
        else
          ${{ inputs.vcpkg-root }}/bootstrap-vcpkg.sh -disableMetrics
        fi
    
    - name: 配置 NuGet 二进制缓存
      shell: bash
      run: |
        # 下载 NuGet CLI
        NUGET_DIR="${{ inputs.vcpkg-root }}/nuget"
        mkdir -p "$NUGET_DIR"
        curl -sSL -o "$NUGET_DIR/nuget.exe" \
          https://dist.nuget.org/win-x86-commandline/latest/nuget.exe
        
        # 添加 GitHub Packages 源
        NUGET_CMD="$NUGET_DIR/nuget.exe"
        if [ "${{ inputs.use-mono }}" = "true" ]; then
          NUGET_CMD="mono $NUGET_CMD"  # Linux 需要 Mono 运行 NuGet
        fi
        
        $NUGET_CMD sources add \
          -Source "https://nuget.pkg.github.com/${{ github.repository_owner }}/index.json" \
          -Name "GitHub" \
          -UserName "${{ github.repository_owner }}" \
          -Password "${{ inputs.nuget-api-key }}" \
          -StorePasswordInClearText
        
        # 设置 vcpkg 二进制缓存源
        echo "VCPKG_BINARY_SOURCES=clear;nuget,GitHub,readwrite" >> $GITHUB_ENV
    
    - name: 验证安装
      shell: bash
      run: |
        echo "vcpkg 版本：$(${{ inputs.vcpkg-root }}/vcpkg version | head -1)"
        echo "二进制缓存：$VCPKG_BINARY_SOURCES"
```

创建好复合动作后，在工作流中用 `uses: ./.github/actions/<action名>` 语法调用：

## 可复用工作流（Reusable Workflow）

### 复合动作 vs 可复用工作流

复合动作封装的是**步骤（Steps）**——它在一个 Job 内部工作，共享同一个 Runner 环境。而可复用工作流（Reusable Workflow）封装的是**完整的 Job**——它有自己独立的 Runner，可以包含自己的矩阵策略、环境变量和密钥引用。

两者的对比：

| 特性 | 复合动作（Composite Action） | 可复用工作流（Reusable Workflow） |
|------|------------------------------|----------------------------------|
| 封装粒度 | 步骤级别（Steps） | Job 级别（整个 Job） |
| Runner | 共享调用者的 Runner | 拥有独立的 Runner |
| 矩阵支持 | 不能定义自己的矩阵 | 可以定义自己的矩阵策略 |
| Secrets 访问 | 通过 inputs 传入 | 可通过 `secrets: inherit` 直接继承 |
| 跨仓库复用 | 需要发布到 Marketplace 或引用仓库路径 | 同组织内可直接引用 |
| 调用语法 | `uses: ./.github/actions/xxx` | `uses: ./.github/workflows/xxx.yml` |
| 嵌套层级 | 可被可复用工作流包含 | 最多嵌套 4 层 |

**选择建议：** 如果你只需要封装几个 Shell 命令（如安装 vcpkg），用复合动作。如果你需要封装一整套构建流程（从 checkout 到上传产物），用可复用工作流。

### 创建可复用工作流

可复用工作流就是一个普通的工作流文件，但它的触发事件包含 `workflow_call`：

```yaml
# 示例 10：可复用工作流——通用 C++ 构建流程
# 文件路径：.github/workflows/reusable-cpp-build.yml
name: Reusable C++ Build

# workflow_call 使这个工作流可以被其他工作流调用
on:
  workflow_call:
    # 定义调用者可以传入的参数
    inputs:
      cmake-preset-config:
        description: 'CMake 配置预设名称'
        required: true
        type: string
      cmake-preset-build:
        description: 'CMake 构建预设名称'
        required: true
        type: string
      runner-os:
        description: '运行器操作系统'
        required: true
        type: string
      enable-tests:
        description: '是否启用测试'
        required: false
        type: boolean
        default: false
      extra-cmake-args:
        description: '额外的 CMake 参数'
        required: false
        type: string
        default: ''
    
    # 定义需要的密钥
    secrets:
      NUGET_API_KEY:
        description: 'NuGet API Key for vcpkg binary caching'
        required: true
    
    # 定义输出（可选）
    outputs:
      build-artifact-name:
        description: '构建产物名称'
        value: ${{ jobs.build.outputs.artifact-name }}

jobs:
  build:
    runs-on: ${{ inputs.runner-os }}
    
    outputs:
      artifact-name: ${{ steps.upload.outputs.artifact-name }}
    
    env:
      VCPKG_BINARY_SOURCES: "clear;nuget,GitHub,readwrite"
    
    steps:
      - name: 检出代码
        uses: actions/checkout@v4
        with:
          submodules: recursive
      
      - name: 安装 vcpkg
        run: |
          git clone https://github.com/microsoft/vcpkg.git $HOME/vcpkg
          $HOME/vcpkg/bootstrap-vcpkg.sh -disableMetrics
          echo "VCPKG_ROOT=$HOME/vcpkg" >> $GITHUB_ENV
      
      - name: 配置 ccache
        uses: hendrikmuhs/ccache-action@v1.2.18
        with:
          key: ${{ inputs.runner-os }}-${{ inputs.cmake-preset-config }}
      
      - name: 配置 CMake
        run: |
          cmake --preset="${{ inputs.cmake-preset-config }}" \
            -DENABLE_TESTS=${{ inputs.enable-tests && 'ON' || 'OFF' }} \
            ${{ inputs.extra-cmake-args }}
      
      - name: 构建
        run: cmake --build --preset="${{ inputs.cmake-preset-build }}"
      
      - name: 运行测试
        if: inputs.enable-tests
        run: |
          ctest --test-dir out/build --output-on-failure
      
      - name: 上传构建产物
        id: upload
        uses: actions/upload-artifact@v4
        with:
          name: build-${{ inputs.runner-os }}
          path: out/build/
```

### 调用可复用工作流

在主工作流中，使用 `uses` 语法调用可复用工作流（注意：是在 `jobs` 级别的 `uses`，不是 `steps` 级别的）：

```yaml
# 示例 11：调用可复用工作流
# 文件路径：.github/workflows/ci.yml
name: CI Pipeline

on:
  workflow_run:
    workflows: ["Code Format Check"]
    types: [completed]

jobs:
  # 调用可复用工作流——Linux 构建
  build-linux:
    if: ${{ github.event.workflow_run.conclusion == 'success' }}
    uses: ./.github/workflows/reusable-cpp-build.yml    # 指向可复用工作流文件
    with:
      cmake-preset-config: "Linux Release Config"
      cmake-preset-build: "Linux Release Build"
      runner-os: ubuntu-latest
      enable-tests: true
    secrets:
      NUGET_API_KEY: ${{ secrets.NUGET_API_KEY }}

  build-windows:
    if: ${{ github.event.workflow_run.conclusion == 'success' }}
    uses: ./.github/workflows/reusable-cpp-build.yml
    with:
      cmake-preset-config: "Windows Release Config"
      cmake-preset-build: "Windows Release Build"
      runner-os: windows-latest
      enable-tests: false
      extra-cmake-args: "-DCMAKE_MSVC_RUNTIME_LIBRARY=MultiThreaded"
    secrets:
      NUGET_API_KEY: ${{ secrets.NUGET_API_KEY }}

  build-macos:

    with:
      cmake-preset-config: "MacOS Release Config"
      cmake-preset-build: "MacOS Release Build"
      runner-os: macos-latest
    secrets: inherit    # 继承调用者的所有 secrets
```

### 实践 2：创建一个 ccache 复合动作

将所有工作流中重复的 ccache 配置封装为复合动作：

**步骤 1：** 创建目录和文件

```bash
# 在项目根目录执行
mkdir -p .github/actions/setup-ccache
```

**步骤 2：** 编写 `action.yml`

```yaml
# 文件路径：.github/actions/setup-ccache/action.yml
name: 'Setup ccache'
description: '配置 ccache 编译缓存，加速 C++ 增量构建'

    required: false
```

## 对照项目源码

### 现有工作流中的"隐式矩阵"

虽然 KrKr2 目前没有使用 `strategy.matrix`，但三个独立的构建工作流本质上是一个"手动展开的矩阵"。对比如下：

| 矩阵维度 | build-linux.yml | build-windows.yml | build-android.yml |
|-----------|-----------------|-------------------|-------------------|
| **Runner** | `ubuntu-latest` | `windows-latest` | `ubuntu-latest` |
| **构建系统** | CMake Preset | CMake Preset (PowerShell) | Gradle |
| **编译器** | GCC（系统默认） | MSVC（`ilammy/msvc-dev-cmd`） | NDK clang |
| **vcpkg 引导** | `bootstrap-vcpkg.sh` | `bootstrap-vcpkg.bat` | `bootstrap-vcpkg.sh` |
| **NuGet CLI** | Mono + nuget.exe | 原生 nuget.exe | Mono + nuget.exe |
| **ccache key** | `linux-release` | `windows-release` | `android-release` |
| **测试** | `ENABLE_TESTS=OFF` | `ENABLE_TESTS=OFF` | 不支持（Android） |

相关文件：
- `.github/workflows/build-linux.yml` 第 1-100 行 — Linux 构建的完整工作流
- `.github/workflows/build-windows.yml` 第 1-84 行 — Windows 构建，使用 PowerShell 和 MSVC
- `.github/workflows/build-android.yml` 第 1-169 行 — Android 构建，包含 SDK/NDK 安装和磁盘清理
- `CMakePresets.json` 第 1-156 行 — 定义了各平台的 Configure/Build 预设，矩阵策略中引用的预设名来自此文件

### 重复模式分析

三个工作流中重复出现的模式（适合封装为复合动作）：

```
重复模式                          出现次数    封装建议
─────────────────────────────    ─────────   ────────────────────
vcpkg 克隆 + 引导                3 次        setup-vcpkg 复合动作
NuGet 源配置                     3 次        包含在 setup-vcpkg 中
ccache 配置                      3 次        setup-ccache 复合动作
VCPKG_BINARY_SOURCES 设置        3 次        包含在 setup-vcpkg 中
checkout + submodules            3 次        较简单，保持内联即可
```

### CMakePresets.json 与矩阵的关系

项目的 `CMakePresets.json`（第 1-156 行）为每个平台定义了 Configure 和 Build 预设：

```
CMakePresets.json 中的预设：
├── configurePresets:
│   ├── "Windows Debug Config"    →  矩阵变量 cmake_preset_config
│   ├── "Windows Release Config"
│   ├── "Linux Debug Config"
│   ├── "Linux Release Config"
│   ├── "MacOS Debug Config"
│   └── "MacOS Release Config"
└── buildPresets:
    ├── "Windows Debug Build"     →  矩阵变量 cmake_preset_build
    ├── "Windows Release Build"
    ├── "Linux Debug Build"
    ├── "Linux Release Build"
    ├── "MacOS Debug Build"
    └── "MacOS Release Build"
```

这些预设名称天然适合作为矩阵变量——每个 `(平台, 构建类型)` 组合对应一对预设。如果将来项目决定使用矩阵策略，这些预设名可以直接作为 `include` 中的附加变量。

## 常见错误与排查

### 错误 1：矩阵变量类型不匹配

```yaml
# ❌ 错误：布尔值在 YAML 中会被解析为 true/false，但在 shell 中变成字符串
strategy:
  matrix:
    enable_tests: [true, false]

steps:
  - run: |
      # 这里 ${{ matrix.enable_tests }} 的值是字符串 "true" 或 "false"
      # 不能直接用于 bash 的 if 判断
      if [ ${{ matrix.enable_tests }} ]; then  # ❌ 永远为 true！非空字符串在 bash 中都是 true
        echo "running tests"
      fi

# ✅ 正确做法 1：用字符串比较
steps:
  - run: |
      if [ "${{ matrix.enable_tests }}" = "true" ]; then
        echo "running tests"
      fi

# ✅ 正确做法 2：用 if 条件表达式（推荐）
steps:
  - name: 运行测试
    if: matrix.enable_tests == true    # GitHub Actions 表达式层面的布尔比较
    run: ctest --test-dir build --output-on-failure
```

### 错误 2：exclude 匹配不生效

```yaml
# ❌ 错误：exclude 中的值必须精确匹配，大小写、空格都算
strategy:
  matrix:
    os: [ubuntu-latest, Windows-Latest]    # 注意大写
    compiler: [gcc, msvc]
    exclude:
      - os: windows-latest                 # ❌ 小写的 "windows-latest" 不匹配 "Windows-Latest"
        compiler: gcc

# ✅ 正确：保持完全一致的字符串
strategy:
  matrix:
    os: [ubuntu-latest, windows-latest]    # 统一小写
    compiler: [gcc, msvc]
    exclude:
      - os: windows-latest                 # ✅ 精确匹配
        compiler: gcc
```

### 错误 3：可复用工作流的 secrets 未传递

```yaml
# ❌ 错误：调用可复用工作流时忘记传递 secrets
jobs:
  build:
    uses: ./.github/workflows/reusable-build.yml
    with:
      platform: linux
    # 缺少 secrets 块！可复用工作流中的 ${{ secrets.NUGET_API_KEY }} 将为空

# ✅ 正确做法 1：显式传递
jobs:
  build:
    uses: ./.github/workflows/reusable-build.yml
    with:
      platform: linux
    secrets:
      NUGET_API_KEY: ${{ secrets.NUGET_API_KEY }}

# ✅ 正确做法 2：继承所有 secrets（更简洁）
jobs:
  build:
    uses: ./.github/workflows/reusable-build.yml
    with:
      platform: linux
    secrets: inherit
```

### 错误 4：复合动作中缺少 shell 声明

```yaml
# ❌ 错误：复合动作的 run 步骤必须显式声明 shell
runs:
  using: 'composite'
  steps:
    - name: 安装依赖
      run: apt-get install -y curl    # ❌ 缺少 shell 字段，会报错

# ✅ 正确：每个 run 步骤都必须有 shell
runs:
  using: 'composite'
  steps:
    - name: 安装依赖
      shell: bash                     # ✅ 必须声明
      run: apt-get install -y curl
```

## 本节小结

- **矩阵策略**是 GitHub Actions 最强大的并行化工具，通过定义变量的笛卡尔积自动生成多个并行 Job
- **`exclude`** 排除不需要的组合，**`include`** 添加额外组合或为已有组合附加变量
- **`fail-fast`** 控制是否在某个组合失败时取消其他组合；**`max-parallel`** 限制并行数量
- **动态矩阵**通过 `fromJSON()` 实现运行时决定矩阵变量，适合按分支/PR 调整测试范围
- **复合动作（Composite Action）** 封装步骤级别的重复操作，适合 Shell 命令序列
- **可复用工作流（Reusable Workflow）** 封装 Job 级别的完整流程，支持矩阵、密钥继承
- KrKr2 的三个独立构建文件本质是"手动矩阵"，**`CMakePresets.json`** 中的预设名天然适合作为矩阵变量

## 练习题与答案

### 题目 1：设计一个三维矩阵

为一个需要在 Linux/Windows/macOS 上用 GCC/Clang/MSVC 编译 Debug/Release 的项目设计矩阵策略。要求：排除不兼容的平台-编译器组合（Windows 不用 GCC、Linux/macOS 不用 MSVC），并为每个 Windows 组合添加 `msvc_arch: x64` 额外变量。

<details>
<summary>查看答案</summary>

```yaml
# 完整的三维矩阵配置
jobs:
  build:
    strategy:
      fail-fast: false
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        compiler: [gcc, clang, msvc]
        build_type: [Debug, Release]
        
        # 排除不兼容的组合
        exclude:
          # Windows 不使用 GCC
          - os: windows-latest
            compiler: gcc
          # Linux 不使用 MSVC
          - os: ubuntu-latest
            compiler: msvc
          # macOS 不使用 MSVC
          - os: macos-latest
            compiler: msvc
        
        # 为 Windows 组合添加架构变量
        include:
          - os: windows-latest
            compiler: msvc
            build_type: Debug
            msvc_arch: x64
          - os: windows-latest
            compiler: msvc
            build_type: Release
            msvc_arch: x64
          - os: windows-latest
            compiler: clang
            build_type: Debug
            msvc_arch: x64
          - os: windows-latest
            compiler: clang
            build_type: Release
            msvc_arch: x64
    
    runs-on: ${{ matrix.os }}
    
    steps:
      - uses: actions/checkout@v4
      
      - name: 配置 MSVC（仅 Windows）
        if: matrix.msvc_arch != ''
        uses: ilammy/msvc-dev-cmd@v1
        with:
          arch: ${{ matrix.msvc_arch }}
      
      - name: 设置编译器
        shell: bash
        run: |
          case "${{ matrix.compiler }}" in
            gcc)   echo "CC=gcc"     >> $GITHUB_ENV; echo "CXX=g++"     >> $GITHUB_ENV ;;
            clang) echo "CC=clang"   >> $GITHUB_ENV; echo "CXX=clang++" >> $GITHUB_ENV ;;
            msvc)  echo "CC=cl"      >> $GITHUB_ENV; echo "CXX=cl"      >> $GITHUB_ENV ;;
          esac
      
      - name: 构建 (${{ matrix.os }}/${{ matrix.compiler }}/${{ matrix.build_type }})
        run: |
          cmake -B build -DCMAKE_BUILD_TYPE=${{ matrix.build_type }} \
            -DCMAKE_C_COMPILER=${{ env.CC }} \
            -DCMAKE_CXX_COMPILER=${{ env.CXX }}
          cmake --build build --config ${{ matrix.build_type }}

# 最终运行的组合数：
# 原始：3 OS × 3 编译器 × 2 构建类型 = 18
# 排除后：18 - 6 (gcc/win×2 + msvc/linux×2 + msvc/mac×2) = 12 个 Job
```

</details>

### 题目 2：将重复步骤封装为复合动作

观察以下两个工作流片段中的重复部分，将其封装为一个复合动作：

```yaml
# 工作流 A：
- name: 安装 vcpkg
  run: |
    git clone https://github.com/microsoft/vcpkg.git $HOME/vcpkg
    $HOME/vcpkg/bootstrap-vcpkg.sh -disableMetrics
    echo "VCPKG_ROOT=$HOME/vcpkg" >> $GITHUB_ENV

# 工作流 B（几乎一样）：
- name: 安装 vcpkg
  run: |
    git clone https://github.com/microsoft/vcpkg.git $HOME/vcpkg
    $HOME/vcpkg/bootstrap-vcpkg.sh -disableMetrics
    echo "VCPKG_ROOT=$HOME/vcpkg" >> $GITHUB_ENV
```

<details>
<summary>查看答案</summary>

```yaml
# 文件路径：.github/actions/install-vcpkg/action.yml
name: 'Install vcpkg'
description: '克隆并引导 vcpkg，设置 VCPKG_ROOT 环境变量'

inputs:
  install-dir:
    description: 'vcpkg 安装目录'
    required: false
    default: '$HOME/vcpkg'
  disable-metrics:
    description: '是否禁用 vcpkg 遥测'
    required: false
    default: 'true'

outputs:
  vcpkg-root:
    description: 'vcpkg 安装路径'
    value: ${{ inputs.install-dir }}

runs:
  using: 'composite'
  steps:
    - name: 克隆 vcpkg
      shell: bash
      run: |
        # 如果目录已存在则跳过克隆
        if [ ! -d "${{ inputs.install-dir }}" ]; then
          git clone https://github.com/microsoft/vcpkg.git "${{ inputs.install-dir }}"
        else
          echo "vcpkg 目录已存在，跳过克隆"
        fi
    
    - name: 引导 vcpkg
      shell: bash
      run: |
        BOOTSTRAP_ARGS=""
        if [ "${{ inputs.disable-metrics }}" = "true" ]; then
          BOOTSTRAP_ARGS="-disableMetrics"
        fi
        
        # 根据平台选择脚本
        if [ "$RUNNER_OS" = "Windows" ]; then
          "${{ inputs.install-dir }}/bootstrap-vcpkg.bat" $BOOTSTRAP_ARGS
        else
          "${{ inputs.install-dir }}/bootstrap-vcpkg.sh" $BOOTSTRAP_ARGS
        fi
    
    - name: 设置环境变量
      shell: bash
      run: |
        echo "VCPKG_ROOT=${{ inputs.install-dir }}" >> $GITHUB_ENV
        echo "vcpkg 已安装到 ${{ inputs.install-dir }}"
        "${{ inputs.install-dir }}/vcpkg" version | head -1
```

在工作流中使用：

```yaml
# 工作流 A 和 B 都改为：
- name: 安装 vcpkg
  uses: ./.github/actions/install-vcpkg
  with:
    install-dir: $HOME/vcpkg
```

</details>

### 题目 3：将动态矩阵应用于 KrKr2

为 KrKr2 设计一个动态矩阵方案：PR 时只构建 Linux Release，push 到 main 时构建 Linux + Windows + Android 的 Release。提示：使用 `fromJSON()` 和一个 `prepare` Job。

<details>
<summary>查看答案</summary>

```yaml
# 文件路径：.github/workflows/dynamic-matrix-build.yml
name: Dynamic Matrix Build

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  # 第一阶段：根据触发事件决定构建哪些平台
  prepare:
    runs-on: ubuntu-latest
    outputs:
      platforms: ${{ steps.set-platforms.outputs.platforms }}
    steps:
      - id: set-platforms
        run: |
          if [ "${{ github.event_name }}" = "push" ]; then
            # main 分支推送：完整构建
            echo 'platforms=[
              {"name":"linux","os":"ubuntu-latest","preset_config":"Linux Release Config","preset_build":"Linux Release Build","use_gradle":false},
              {"name":"windows","os":"windows-latest","preset_config":"Windows Release Config","preset_build":"Windows Release Build","use_gradle":false},
              {"name":"android","os":"ubuntu-latest","preset_config":"","preset_build":"","use_gradle":true}
            ]' | tr -d '\n' >> $GITHUB_OUTPUT
          else
            # PR：只构建 Linux
            echo 'platforms=[
              {"name":"linux","os":"ubuntu-latest","preset_config":"Linux Release Config","preset_build":"Linux Release Build","use_gradle":false}
            ]' | tr -d '\n' >> $GITHUB_OUTPUT
          fi
  
  # 第二阶段：动态矩阵构建
  build:
    needs: prepare
    strategy:
      fail-fast: false
      matrix:
        platform: ${{ fromJSON(needs.prepare.outputs.platforms) }}
    
    runs-on: ${{ matrix.platform.os }}
    
    steps:
      - uses: actions/checkout@v4
        with:
          submodules: recursive
      
      - name: 安装 vcpkg
        run: |
          git clone https://github.com/microsoft/vcpkg.git $HOME/vcpkg
          $HOME/vcpkg/bootstrap-vcpkg.sh -disableMetrics
          echo "VCPKG_ROOT=$HOME/vcpkg" >> $GITHUB_ENV
        shell: bash
      
      # CMake 构建路径（Linux/Windows）
      - name: CMake 构建
        if: matrix.platform.use_gradle == false
        shell: bash
        run: |
          cmake --preset="${{ matrix.platform.preset_config }}" -DENABLE_TESTS=OFF
          cmake --build --preset="${{ matrix.platform.preset_build }}"
      
      # Gradle 构建路径（Android）
      - name: Gradle 构建
        if: matrix.platform.use_gradle == true
        run: |
          sudo apt-get install -y dos2unix
          dos2unix platforms/android/gradlew
          chmod +x platforms/android/gradlew
          ./platforms/android/gradlew -p ./platforms/android assembleRelease
      
      - name: 构建完成
        run: echo "✅ ${{ matrix.platform.name }} 构建成功"
        shell: bash

# 运行结果：
# - PR 触发：只有 1 个 Job（linux）
# - push 到 main：有 3 个 Job（linux、windows、android）并行运行
```

</details>
