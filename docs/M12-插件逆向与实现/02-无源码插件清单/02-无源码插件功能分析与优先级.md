## 一、分析方法论

在逐个分析 C 类插件之前，先说明评估维度和数据来源。

### 1.1 评估维度

每个插件从以下 4 个维度评估：

```
┌─────────────────────────────────────────────────────────────┐
│                    插件逆向优先级评估模型                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  维度 1：使用频率（Impact）                                   │
│  ├── 高：>50% 的 KiriKiri 游戏使用                           │
│  ├── 中：10%-50% 的游戏使用                                  │
│  └── 低：<10% 的游戏使用                                     │
│                                                             │
│  维度 2：功能重要性（Criticality）                            │
│  ├── 关键：缺失会导致游戏完全无法运行                         │
│  ├── 重要：缺失会导致部分功能异常但可勉强运行                  │
│  └── 次要：缺失不影响核心游戏流程                             │
│                                                             │
│  维度 3：逆向难度（Difficulty）                               │
│  ├── 高：涉及复杂算法/GPU 渲染/大量状态管理                   │
│  ├── 中：算法明确但实现细节多                                 │
│  └── 低：功能简单、函数多但逻辑直白                           │
│                                                             │
│  维度 4：替代可行性（Alternatives）                           │
│  ├── 无替代：必须逆向原始实现                                 │
│  ├── 部分替代：可用其他方案覆盖核心功能                       │
│  └── 完全替代：可用现有库/功能替代                            │
│                                                             │
│  综合优先级 = Impact × Criticality / Difficulty              │
│  （替代可行性作为降权因子）                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 数据来源

- **使用频率**：基于 KiriKiri 游戏社区的 DLL 使用统计、GitHub Issues 中的兼容性报告
- **功能说明**：来源于原版 KiriKiri2 文档、插件作者的说明页、社区 wiki
- **逆向难度**：基于 DLL 文件大小、导出函数数量、算法复杂度的初步评估
- **替代方案**：基于 KrKr2 已有功能和可用的开源库

## 二、逐个分析 C 类插件

### 2.1 TextRender.dll — 高级文字渲染引擎

**功能概述：**

TextRender 是 KiriKiri 生态中使用最广泛的第三方文字渲染插件。原版 KiriKiri 自带的文字渲染功能比较基础——只能做简单的文字绘制、颜色设置和基本描边。TextRender 在此基础上提供了丰富的高级排版功能：

- **多层描边**（Outline）：支持多层不同颜色、不同粗细的文字描边效果，而非单层描边
- **阴影效果**（Shadow）：带偏移和模糊的文字阴影，模拟立体感
- **Ruby 注音**（Ruby Annotation）：在汉字上方标注假名读音，这是日语游戏的常见需求。例如"漢字"上方标注"かんじ"，让玩家能正确发音
- **渐变填充**（Gradient Fill）：文字颜色从上到下或从左到右渐变
- **缩放和间距控制**：精确调整字间距、行间距、字符缩放
- **特殊字符处理**：Emoji 支持、Unicode 合字（Ligature，多个字符组合显示为一个字形的排版技术）

**TJS 接口示例：**

```javascript
// TextRender 的典型 TJS 调用方式
// 这些是游戏脚本中常见的用法

// 创建 TextRender 对象
var tr = new TextRender();

// 设置基本属性
tr.font.face = "MS Gothic";    // 字体名称
tr.font.height = 24;           // 字号（像素）
tr.color = 0xFFFFFF;           // 文字颜色（白色）

// 多层描边设置
tr.outline1.color = 0x000000;  // 第一层描边颜色（黑色）
tr.outline1.width = 2;         // 第一层描边宽度
tr.outline2.color = 0x333333;  // 第二层描边颜色
tr.outline2.width = 4;         // 第二层描边宽度

// 阴影设置
tr.shadow.color = 0x000000;    // 阴影颜色
tr.shadow.offsetX = 2;         // 阴影水平偏移
tr.shadow.offsetY = 2;         // 阴影垂直偏移
tr.shadow.blur = 3;            // 阴影模糊半径

// 渲染到图层
tr.drawText(layer, 10, 10, "こんにちは世界");
```

**使用频率：** ⭐⭐⭐⭐⭐（高）——超过 60% 的日语视觉小说游戏使用 TextRender，因为原版文字渲染不支持 Ruby 注音

**功能重要性：** 🔴 关键——缺失导致游戏文字无法正常显示，游戏基本不可玩

**逆向难度：** ⭐⭐⭐⭐（高）
- DLL 文件较大（通常 200-500KB）
- 涉及字体光栅化（Rasterization）——需要处理 TrueType/OpenType 字体文件的矢量轮廓并转换为像素
- 多层混合渲染——描边、阴影、渐变的组合需要精确的像素级操作
- 跨平台适配困难——原版使用 Windows GDI/GDI+，KrKr2 需要使用 FreeType + OpenGL

**替代可行性：** 部分替代
- FreeType（开源字体库）可以处理字体光栅化
- HarfBuzz（开源文字塑形引擎）可以处理复杂排版
- 但 TextRender 的多层描边和特定 API 接口需要自行实现

**逆向策略：**
1. 用 PE 导出表（PE Export Table，Windows DLL 中记录对外公开函数的数据结构）分析导出函数，确定 TJS 绑定的方法列表
2. 在 IDA/Ghidra 中定位 V2Link 入口，追踪 iTJSDispatch2 注册过程
3. 重点逆向：`drawText` 核心渲染函数、描边算法、Ruby 注音排版逻辑
4. 用 FreeType + 自定义渲染管线重新实现，不需要还原原始算法

### 2.2 PackinOne.dll — 归档压缩工具

**功能概述：**

PackinOne 是一个归档/压缩插件，用于将多个文件打包成单个归档文件。它与 KiriKiri 原生的 XP3 归档格式不同——XP3 是 KiriKiri 的标准资源包格式，而 PackinOne 通常用于特定游戏的自定义打包需求。PackinOne 可能使用自定义的压缩算法或加密方式来保护游戏资源。

**TJS 接口推测：**

```javascript
// PackinOne 的 TJS 接口（基于社区文档推测）

// 打包操作
var packer = new PackinOne();
packer.addFile("data/script.ks");
packer.addFile("data/image.png");
packer.pack("output.pak");  // 打包为 .pak 文件

// 解包操作
var unpacker = new PackinOne();
unpacker.open("game.pak");
var data = unpacker.extract("script.ks");  // 提取单个文件

// 文件系统挂载
// 部分游戏将 .pak 挂载为虚拟文件系统
Storages.addAutoPath("game.pak");
```

**使用频率：** ⭐⭐⭐（中等）——约 20-30% 的游戏使用自定义打包格式

**功能重要性：** 🔴 关键——如果游戏资源都在 .pak 中，缺少 PackinOne 就无法读取任何资源

**逆向难度：** ⭐⭐⭐（中等）
- DLL 文件中等大小（100-200KB）
- 核心是压缩/解压缩算法——可能是 zlib、LZ4 或自定义变体
- 归档格式头结构需要逆向还原
- 没有 GPU 相关操作，纯 CPU 算法

**替代可行性：** 无替代——必须精确还原归档格式和压缩算法，否则无法读取游戏资源

**逆向策略：**
1. 导出表分析：识别 `pack`、`unpack`、`extract` 等关键函数
2. 重点逆向归档文件头结构（Magic Number、文件表偏移、压缩类型标记）
3. 识别压缩算法——对比已知算法（zlib、LZ4、LZMA）的特征码
4. 编写解包工具验证——能否正确提取已知内容的测试文件

### 2.3 util_generic.dll — 通用工具集

**功能概述：**

util_generic 是一个功能杂糅的工具插件，为 TJS 脚本提供各种通用辅助函数。这些函数在原版 TJS2 引擎中没有内置，但在游戏开发中经常需要。典型功能包括：

- **字符串工具**：正则表达式匹配、字符串格式化、编码转换（UTF-8 ↔ Shift_JIS）
- **数学函数**：三角函数、随机数生成器（比 TJS 内置的更高质量）、插值计算
- **数组/字典操作**：排序、过滤、映射（对数组每个元素执行相同操作）、查找
- **时间日期**：获取当前时间、格式化日期字符串、计算时间差

**TJS 接口示例：**

```javascript
// util_generic 提供的典型函数

// 字符串工具
var result = GenericUtil.regexMatch("hello123", "[0-9]+");
// result = "123"

// 编码转换
var sjis = GenericUtil.encodeShiftJIS("こんにちは");
var utf8 = GenericUtil.decodeShiftJIS(sjis);

// 数学函数
var angle = GenericUtil.atan2(y, x);   // 反正切
var lerped = GenericUtil.lerp(a, b, t); // 线性插值

// 数组操作
var sorted = GenericUtil.sort(arr, "ascending");
var filtered = GenericUtil.filter(arr, function(x) {
    return x > 0;
});
```

**使用频率：** ⭐⭐⭐⭐（高）——约 40% 的游戏使用，因为提供了 TJS 缺少的基础功能

**功能重要性：** 🟡 重要——缺失会导致部分功能异常，但核心剧情流程可能不受影响

**逆向难度：** ⭐⭐（低）
- 函数多（可能有 30-50 个导出函数）但每个函数的逻辑简单
- 大部分功能可以用 C++ 标准库直接实现
- 没有复杂的状态管理或算法

**替代可行性：** 完全替代——所有功能都可以用 C++ 标准库 + Boost 重新实现，不需要逆向原始算法

**逆向策略：**
1. 导出表分析：列出所有导出函数名——函数名本身就说明了功能
2. 不需要深入逆向每个函数的实现——直接根据函数名和参数类型重新编写
3. 使用游戏脚本作为测试用例——运行游戏并观察哪些函数被调用

### 2.4 util_graph.dll — 图形工具集

**功能概述：**

util_graph 提供像素级的图像处理功能，是 KiriKiri 图层系统（Layer，引擎中用于管理和渲染 2D 画面的基本单元）的扩展。它在 Layer 类上添加了专业的图像处理方法：

- **色彩空间转换**：RGB ↔ HSV（色相-饱和度-明度颜色模型）、RGB ↔ HSL
- **滤镜效果**：模糊（Blur）、锐化（Sharpen）、边缘检测（Edge Detection）
- **直方图**（Histogram，统计图像中各亮度值出现频率的分布图）：计算和均衡化
- **混合模式**：乘法混合（Multiply）、叠加混合（Overlay）、柔光混合（Soft Light）等
- **几何变换**：仿射变换（Affine Transform，包含平移、旋转、缩放、剪切的 2D 变换）

**TJS 接口示例：**

```javascript
// util_graph 提供的图像处理功能

// 色彩空间转换
var hsv = GraphUtil.rgbToHsv(r, g, b);
// hsv.h = 色相(0-360), hsv.s = 饱和度(0-1), hsv.v = 明度(0-1)

// 模糊滤镜
layer.graphBlur(radius);  // 高斯模糊，radius 为模糊半径

// 直方图均衡化（让图像对比度更均匀）
layer.graphHistogramEqualize();

// 混合模式
layer.graphBlend(srcLayer, "multiply");  // 乘法混合
layer.graphBlend(srcLayer, "overlay");   // 叠加混合

// 仿射变换
layer.graphAffine(
    a, b, c,   // 变换矩阵第一行
    d, e, f    // 变换矩阵第二行
);
```

**使用频率：** ⭐⭐⭐（中等）——约 20% 的游戏使用高级图像处理

**功能重要性：** 🟡 重要——缺失会导致特效和转场异常

**逆向难度：** ⭐⭐⭐（中等）
- 图像处理算法有标准实现可参考
- 需要处理像素格式和内存布局
- 部分算法可能有 SIMD 优化（使用 CPU 的并行指令加速计算）

**替代可行性：** 部分替代——OpenCV 可以覆盖大部分图像处理功能，但 TJS 绑定接口需要自行适配

**逆向策略：**
1. 导出函数分析：识别滤镜名称和参数
2. 对比 OpenCV 的标准算法实现
3. 重点逆向混合模式——KiriKiri 的像素格式是 ARGB（透明度-红-绿-蓝，32位每像素），需要确保与标准混合公式一致
4. 用已知图像做输入输出对比测试

### 2.5 util_system.dll — 系统工具集

**功能概述：**

util_system 提供系统级操作的 TJS 绑定，让游戏脚本能够查询和操作操作系统功能：

- **系统信息**：获取 OS 版本、CPU 信息、可用内存
- **文件操作**：文件复制/移动/删除（TJS 内置的文件操作有限）
- **进程管理**：启动外部程序、检测进程是否运行
- **注册表操作**（Windows 特有）：读写 Windows 注册表键值
- **环境变量**：读取和设置环境变量

**TJS 接口示例：**

```javascript
// util_system 的典型用法

// 系统信息
var os = SystemUtil.getOSVersion();    // "Windows 10 22H2"
var mem = SystemUtil.getFreeMemory();   // 可用内存（字节）
var cpu = SystemUtil.getCPUName();      // "Intel Core i7-12700"

// 文件操作
SystemUtil.copyFile("src.dat", "dst.dat");
SystemUtil.deleteFile("temp.dat");
var exists = SystemUtil.fileExists("save.dat");

// 启动外部程序
SystemUtil.shellExecute("notepad.exe", "readme.txt");

// 环境变量
var path = SystemUtil.getEnv("PATH");
```

**使用频率：** ⭐⭐⭐（中等）——约 25% 的游戏使用系统工具

**功能重要性：** 🟢 次要——大部分系统功能不影响核心游戏流程，缺失只影响存档管理等外围功能

**逆向难度：** ⭐⭐（低）
- 大部分是系统 API 的简单封装
- 跨平台适配是主要挑战（Windows 注册表在 Linux/Android 上无对应物）
- 没有复杂算法

**替代可行性：** 完全替代——C++ 标准库 + `<filesystem>` + 平台 API 即可实现

**逆向策略：**
1. 导出函数分析：函数名直接对应系统 API
2. 直接用跨平台 C++ 重写，不需要逆向原始实现
3. Windows 注册表操作在非 Windows 平台上提供空实现（返回空字符串/默认值）

### 2.6 layerExAlpha.dll — 图层透明度扩展

**功能概述：**

layerExAlpha 扩展了 KiriKiri Layer 类的 Alpha 混合能力。原版 Layer 只支持基本的 Alpha 混合（将前景图层按透明度叠加到背景上），layerExAlpha 增加了更专业的混合模式：

- **预乘 Alpha**（Premultiplied Alpha）：一种优化的 Alpha 混合方式。普通 Alpha 混合的公式是 `result = src * alpha + dst * (1 - alpha)`，需要 4 次乘法；预乘 Alpha 将颜色预先乘以透明度（`src_premul = src * alpha`），混合时只需 `result = src_premul + dst * (1 - alpha)`，减少为 3 次乘法。在大量像素混合时（如全屏特效），这种优化效果显著
- **Alpha 通道操作**：提取 Alpha 通道、设置区域 Alpha、Alpha 渐变
- **高级混合模式**：色相混合（Hue Blend）、饱和度混合（Saturation Blend）、亮度混合（Luminosity Blend）

**TJS 接口示例：**

```javascript
// layerExAlpha 的典型用法

// 预乘 Alpha 转换
layer.convertToPremultipliedAlpha();
layer.convertFromPremultipliedAlpha();

// Alpha 通道操作
layer.fillAlpha(128);              // 将整个图层 Alpha 设为 128（半透明）
layer.gradientAlpha(0, 255, "vertical");  // 垂直方向 Alpha 渐变

// 高级混合
layer.blendHue(srcLayer);          // 色相混合
layer.blendSaturation(srcLayer);   // 饱和度混合
layer.blendLuminosity(srcLayer);   // 亮度混合
```

**使用频率：** ⭐⭐（中等偏低）——约 15% 的游戏使用高级 Alpha 操作

**功能重要性：** 🟡 重要——缺失会导致图层混合效果不正确（颜色偏差或边缘锯齿）

**逆向难度：** ⭐⭐（低）
- 像素混合算法有标准数学公式
- 函数数量少（约 10-15 个）
- 每个函数的逻辑都是像素遍历 + 数学运算

**替代可行性：** 完全替代——所有混合模式都有标准定义（参考 W3C Compositing 规范），可以直接实现

**逆向策略：**
1. 不需要逆向——直接根据标准混合公式实现
2. 用 Photoshop/GIMP 的混合结果作为测试基准
3. 关注像素格式：KiriKiri 使用 BGRA 32位像素格式，需要确保字节顺序正确

### 2.7 lzfs.dll — LZ 压缩文件系统

**功能概述：**

lzfs 提供基于 LZ 系列压缩算法的文件系统访问。LZ 压缩（Lempel-Ziv Compression）是一类基于"滑动窗口"（Sliding Window，算法在已处理数据中维护的固定大小缓冲区，用于查找重复模式）的无损数据压缩算法。LZ77、LZ78、LZW（用于 GIF 格式）、LZMA（用于 7z 格式）等都属于这个家族。

lzfs 允许游戏将压缩文件作为透明的文件系统来访问——游戏脚本读取文件时自动解压，不需要显式调用解压函数。

**TJS 接口推测：**

```javascript
// lzfs 的可能用法

// 挂载压缩文件系统
LZFS.mount("data.lzfs", "/data/");

// 之后透明访问——看起来和普通文件一样
var file = Storages.open("/data/script.ks");
// LZFS 在底层自动解压

// 也可能提供直接的压缩/解压 API
var compressed = LZFS.compress(data);
var original = LZFS.decompress(compressed);
```

**使用频率：** ⭐（低）——不到 5% 的游戏使用（大多数游戏使用 XP3 格式）

**功能重要性：** 🔴 关键（对使用它的游戏而言）——如果游戏资源全部用 LZFS 格式存储，缺少此插件则无法读取任何资源

**逆向难度：** ⭐⭐⭐（中等）
- 需要识别具体使用的 LZ 变体（LZ4？LZMA？自定义？）
- 文件系统的元数据结构需要逆向
- 压缩算法本身有标准实现可参考

**替代可行性：** 部分替代——如果使用的是标准 LZ 变体，可以用现有库（zlib、lz4、lzma）；如果是自定义变体则必须逆向

**逆向策略：**
1. 分析 .lzfs 文件头的 Magic Number（文件开头的标识字节，用于识别文件格式）
2. 对比已知 LZ 算法的特征模式
3. 如果是标准算法——直接用现有库实现
4. 如果是自定义算法——逆向压缩/解压核心循环

### 2.8 multiimage.dll — 多图像处理

**功能概述：**

multiimage 提供同时操作多张图片的功能，是 KiriKiri Layer 系统的扩展。单个 Layer 只能持有一张图片，multiimage 允许在一个操作中处理多张图片：

- **图片合成**：将多张图片按指定位置和混合模式合成为一张
- **批量变换**：同时对多张图片执行相同的变换（缩放、旋转等）
- **图片比较**：像素级比较两张图片的差异
- **图集操作**（Atlas，将多个小图片打包到一张大图中的技术）：将多张小图打包成图集，或从图集中提取单张图

**使用频率：** ⭐（低）——约 5% 的游戏使用

**功能重要性：** 🟢 次要——大部分游戏不依赖此插件的核心功能

**逆向难度：** ⭐⭐⭐（中等）
- 图像操作算法有标准实现
- 需要处理多种像素格式的兼容
- 图集打包可能使用特定的排布算法

**替代可行性：** 完全替代——OpenCV 或 stb_image 可以覆盖所有功能

**逆向策略：**
1. 导出函数分析——确定支持哪些操作
2. 直接用 KrKr2 现有的图层系统 + OpenCV 重新实现
3. 低优先级——按需实现

### 2.9 kirikiroid2.dll — KiriKiriDroid2 核心运行时

**功能概述：**

这是一个特殊的 DLL。KiriKiriDroid2 是另一个 Android 平台的 KiriKiri 引擎模拟器（与 KrKr2 是竞品关系）。`kirikiroid2.dll` 是 KiriKiriDroid2 的核心运行时模块——它不是一个"插件"，而是一个完整的引擎核心。

**使用频率：** ⭐（极低）——只有经过 KiriKiriDroid2 修改的游戏

**功能重要性：** ⚪ 特殊——KrKr2 本身就是完整的 KiriKiri 模拟器，不需要 KiriKiriDroid2 的运行时

**逆向难度：** ⭐⭐⭐⭐⭐（极高）——这是一个完整的引擎模块，不是一个功能插件

**替代可行性：** 完全替代——创建一个空的占位插件（Stub Plugin）即可。KrKr2 自身已经实现了 KiriKiri 引擎的所有核心功能，不需要加载外部的引擎模块

**处理策略：**

### 示例 1：占位插件模板（最简实现）

这是处理 C 类插件最快速的方案——创建一个只注册模块名但不实现任何功能的空壳。当游戏脚本调用 `Plugins.link("xxx.dll")` 时，引擎能找到对应的模块名，不会报错退出。

### 示例 2：DLL 导出表分析脚本

在逆向 C 类插件之前，第一步是分析目标 DLL 的导出函数列表。以下 Python 脚本使用 `pefile` 库解析 PE 格式（Portable Executable，Windows 可执行文件的标准格式）的导出表：

```python
#!/usr/bin/env python3
"""
DLL 导出表分析工具
用法：python dll_exports.py TextRender.dll
依赖：pip install pefile
"""
import sys
import pefile  # PE 文件解析库

def analyze_dll(dll_path: str) -> None:
    """分析 DLL 的导出表，打印所有导出函数"""
    pe = pefile.PE(dll_path)  # 加载 PE 文件

    print(f"=== {dll_path} 导出表分析 ===")
    print(f"文件大小: {pe.OPTIONAL_HEADER.SizeOfImage} 字节")
    print(f"入口点 RVA: 0x{pe.OPTIONAL_HEADER.AddressOfEntryPoint:08X}")
    print()

    # KiriKiri 插件必须导出 V2Link 和 V2Unlink
    has_v2link = False
    has_v2unlink = False
    export_count = 0

    if hasattr(pe, 'DIRECTORY_ENTRY_EXPORT'):
        for exp in pe.DIRECTORY_ENTRY_EXPORT.symbols:
            name = exp.name.decode('ascii') if exp.name else f"Ordinal_{exp.ordinal}"
            rva = exp.address  # 相对虚拟地址（Relative Virtual Address）
            print(f"  [{export_count:3d}] {name:40s} RVA=0x{rva:08X}")
            export_count += 1

            if name == "V2Link":
                has_v2link = True
            elif name == "V2Unlink":
                has_v2unlink = True

    print(f"\n共 {export_count} 个导出函数")
    print(f"V2Link:   {'✅ 存在' if has_v2link else '❌ 缺失'}")
    print(f"V2Unlink: {'✅ 存在' if has_v2unlink else '❌ 缺失'}")

    # 判断是否为有效的 KiriKiri 插件
    if has_v2link:
        print("\n✅ 这是一个有效的 KiriKiri 插件 DLL")
    else:
        print("\n⚠️ 未找到 V2Link——可能不是 KiriKiri 插件")

if __name__ == "__main__":
    if len(sys.argv) != 2:
        print(f"用法: {sys.argv[0]} <dll文件路径>")
        sys.exit(1)
    analyze_dll(sys.argv[1])
```

### 示例 3：util_generic 替代实现（无需逆向）

对于"完全可替代"的插件，我们可以直接根据已知的 TJS 接口重新编写。以下是 util_generic.dll 的部分功能替代实现：

```cpp
// 文件：cpp/plugins/util_generic_impl.cpp
// 功能：替代实现 util_generic.dll 的核心功能
// 策略：不需要逆向原始 DLL——直接用 C++ 标准库实现

#include "ncbind.hpp"
#include <cmath>       // sin, cos, atan2, fmod
#include <algorithm>   // sort, find_if
#include <random>      // mt19937, uniform_int_distribution
#include <chrono>      // system_clock, time_point
#include <sstream>     // stringstream
#include <regex>       // regex_match, regex_search
#include <iomanip>     // put_time

#define NCB_MODULE_NAME TJS_W("util_generic.dll")

// ========== 数学函数 ==========

// 线性插值（Linear Interpolation）
// 在两个值 a 和 b 之间，根据比例 t (0.0-1.0) 计算中间值
// 例如 lerp(0, 100, 0.3) = 30
static tjs_real GenericLerp(tjs_real a, tjs_real b, tjs_real t) {
    return a + (b - a) * t;  // 标准线性插值公式
}

// 将角度限制在 0-360 范围内
static tjs_real GenericNormalizeAngle(tjs_real degrees) {
    degrees = fmod(degrees, 360.0);  // fmod: 浮点取余
    if (degrees < 0) degrees += 360.0;
    return degrees;
}

// 反正切函数——返回角度（度数），而非弧度
// atan2(y, x) 返回从正 X 轴到点 (x, y) 的角度
static tjs_real GenericAtan2Deg(tjs_real y, tjs_real x) {
    return atan2(y, x) * 180.0 / 3.14159265358979323846;
}

// ========== 随机数生成器 ==========

// 使用 Mersenne Twister（梅森旋转算法）——比 TJS 内置的
// 线性同余生成器质量更高，周期为 2^19937-1
static std::mt19937 g_rng(
    static_cast<unsigned>(
        std::chrono::system_clock::now()
            .time_since_epoch().count()));

// 生成 [min, max] 范围内的随机整数
static tjs_int GenericRandomInt(tjs_int min_val, tjs_int max_val) {
    std::uniform_int_distribution<tjs_int> dist(min_val, max_val);
    return dist(g_rng);
}

// ========== 注册到 TJS ==========

// NCB_ATTACH_FUNCTION 将 C++ 函数绑定到 TJS 全局命名空间
// 第一个参数是 TJS 中的函数名，第二个是 C++ 函数指针
NCB_ATTACH_FUNCTION(GenericUtil_lerp, GenericLerp);
NCB_ATTACH_FUNCTION(GenericUtil_normalizeAngle, GenericNormalizeAngle);
NCB_ATTACH_FUNCTION(GenericUtil_atan2Deg, GenericAtan2Deg);
NCB_ATTACH_FUNCTION(GenericUtil_randomInt, GenericRandomInt);

NCB_POST_REGIST_CALLBACK(init_util_generic) {
    TVPAddLog(TJS_W(
        "[util_generic.dll] Replacement implementation loaded. "
        "4 math functions registered."));
}
```

**关键点：** 这个文件完全不需要逆向任何 DLL。我们根据 util_generic 的已知接口文档，用 C++ 标准库直接实现了等效功能。这就是"完全可替代"插件的处理思路。

### 示例 4：layerExAlpha 预乘 Alpha 混合实现

layerExAlpha.dll 的核心是像素级的 Alpha 混合算法。这些算法有严格的数学定义（参考 W3C Compositing and Blending Level 1 规范），不需要逆向原始代码：

```cpp
// 文件：cpp/plugins/layer_ex_alpha_impl.cpp
// 功能：替代实现 layerExAlpha.dll 的预乘 Alpha 混合
// 参考：W3C Compositing and Blending Level 1
// https://www.w3.org/TR/compositing-1/

#include "ncbind.hpp"
#include <cstdint>    // uint8_t, uint32_t
#include <algorithm>  // std::min, std::max

#define NCB_MODULE_NAME TJS_W("layerExAlpha.dll")

// KiriKiri 的像素格式是 BGRA（蓝-绿-红-透明），32位每像素
// 每个通道 8 位，取值 0-255
struct Pixel {
    uint8_t b;  // 蓝色通道
    uint8_t g;  // 绿色通道
    uint8_t r;  // 红色通道
    uint8_t a;  // Alpha 通道（0=完全透明, 255=完全不透明）
};

// 将普通 Alpha 像素转换为预乘 Alpha（Premultiplied Alpha）
// 预乘 Alpha 的含义：每个颜色通道的值预先乘以透明度
// 优势：混合计算从 4 次乘法减少为 3 次，GPU 硬件原生支持
static void ConvertToPremultiplied(
    Pixel* pixels,       // 像素数组（原地修改）
    int width,           // 图像宽度
    int height           // 图像高度
) {
    const int total = width * height;
    for (int i = 0; i < total; ++i) {
        Pixel& p = pixels[i];
        // 预乘公式：color_premul = color * alpha / 255
        // 加 127 是为了四舍五入（避免截断误差）
        p.r = static_cast<uint8_t>((p.r * p.a + 127) / 255);
        p.g = static_cast<uint8_t>((p.g * p.a + 127) / 255);
        p.b = static_cast<uint8_t>((p.b * p.a + 127) / 255);
        // Alpha 通道本身不变
    }
}

// 从预乘 Alpha 转换回普通 Alpha
// 注意：这个转换会有精度损失（整数除法不可逆）
static void ConvertFromPremultiplied(
    Pixel* pixels,
    int width,
    int height
) {
    const int total = width * height;
    for (int i = 0; i < total; ++i) {
        Pixel& p = pixels[i];
        if (p.a == 0) {
            // Alpha 为 0 时，颜色值无意义，设为 0 避免除以零
            p.r = p.g = p.b = 0;
        } else {
            // 反预乘公式：color = color_premul * 255 / alpha
            p.r = static_cast<uint8_t>(
                std::min(255, (p.r * 255 + p.a / 2) / p.a));
            p.g = static_cast<uint8_t>(
                std::min(255, (p.g * 255 + p.a / 2) / p.a));
            p.b = static_cast<uint8_t>(
                std::min(255, (p.b * 255 + p.a / 2) / p.a));
        }
    }
}

// 预乘 Alpha 混合——这是 GPU 默认的混合模式
// result = src + dst * (1 - src.a)
// 比普通 Alpha 混合快，因为 src 的颜色已经包含了透明度信息
static void BlendPremultiplied(
    Pixel* dst,          // 目标像素（背景，会被修改）
    const Pixel* src,    // 源像素（前景）
    int count            // 像素数量
) {
    for (int i = 0; i < count; ++i) {
        const uint8_t inv_alpha = 255 - src[i].a;  // 1 - src.alpha
        // 只需 3 次乘法（普通混合需要 4 次）
        dst[i].r = src[i].r + ((dst[i].r * inv_alpha + 127) / 255);
        dst[i].g = src[i].g + ((dst[i].g * inv_alpha + 127) / 255);
        dst[i].b = src[i].b + ((dst[i].b * inv_alpha + 127) / 255);
        dst[i].a = src[i].a + ((dst[i].a * inv_alpha + 127) / 255);
    }
}

NCB_POST_REGIST_CALLBACK(init_layer_ex_alpha) {
    TVPAddLog(TJS_W(
        "[layerExAlpha.dll] Replacement implementation loaded. "
        "Premultiplied alpha blending available."));
}
```

**关键点：** 像素混合算法是纯数学运算，有标准规范可参考。代码中的 `+ 127` 技巧是整数四舍五入的常用手法——`(a * b + 127) / 255` 近似等于 `a * b / 255.0` 的浮点结果。

### 示例 5：插件优先级评估自动化工具

将前文的评估模型转化为可运行的 C++ 程序，自动计算每个插件的综合优先级得分：

```cpp
// 文件：tools/plugin_priority.cpp
// 功能：自动计算无源码插件的逆向优先级
// 编译：g++ -std=c++17 -o plugin_priority plugin_priority.cpp
// 运行：./plugin_priority

#include <iostream>    // cout, endl
#include <string>      // string
#include <vector>      // vector
#include <algorithm>   // sort
#include <iomanip>     // setw, setprecision, left

// 评估维度的量化定义
enum class Frequency { Low = 1, Medium = 2, High = 3 };
enum class Criticality { Minor = 1, Important = 2, Critical = 3 };
enum class Difficulty { Low = 1, Medium = 2, High = 3, VeryHigh = 4 };
enum class Alternative { None = 0, Partial = 1, Full = 2 };

// 插件评估数据结构
struct PluginEval {
    std::string name;          // DLL 名称
    Frequency frequency;       // 使用频率
    Criticality criticality;   // 功能重要性
    Difficulty difficulty;     // 逆向难度
    Alternative alternative;   // 替代可行性

    // 综合优先级计算公式：
    // score = (频率 × 重要性) / 难度 × 替代系数
    // 替代系数：完全替代=3（优先处理），部分替代=1.5，无替代=1
    double CalcScore() const {
        double base = static_cast<double>(frequency)
                    * static_cast<double>(criticality)
                    / static_cast<double>(difficulty);
        // 替代系数——可替代的插件优先处理（投入产出比高）
        double alt_factor = 1.0;
        if (alternative == Alternative::Full)
            alt_factor = 3.0;    // 完全可替代 → 最高优先
        else if (alternative == Alternative::Partial)
            alt_factor = 1.5;    // 部分可替代 → 中等优先
        return base * alt_factor;
    }
};

int main() {
    // 初始化 9 个 C 类插件的评估数据
    std::vector<PluginEval> plugins = {
        {"kirikiroid2.dll", Frequency::Low,    Criticality::Minor,
         Difficulty::VeryHigh, Alternative::Full},
        {"util_generic.dll", Frequency::High,  Criticality::Important,
         Difficulty::Low,      Alternative::Full},
        {"util_system.dll",  Frequency::Medium, Criticality::Minor,
         Difficulty::Low,      Alternative::Full},
        {"layerExAlpha.dll", Frequency::Medium, Criticality::Important,
         Difficulty::Low,      Alternative::Full},
        {"PackinOne.dll",    Frequency::Medium, Criticality::Critical,
         Difficulty::Medium,   Alternative::None},
        {"util_graph.dll",   Frequency::Medium, Criticality::Important,
         Difficulty::Medium,   Alternative::Partial},
        {"TextRender.dll",   Frequency::High,   Criticality::Critical,
         Difficulty::High,     Alternative::Partial},
        {"multiimage.dll",   Frequency::Low,    Criticality::Minor,
         Difficulty::Medium,   Alternative::Full},
        {"lzfs.dll",         Frequency::Low,    Criticality::Critical,
         Difficulty::Medium,   Alternative::Partial},
    };

    // 按优先级得分降序排序
    std::sort(plugins.begin(), plugins.end(),
        [](const PluginEval& a, const PluginEval& b) {
            return a.CalcScore() > b.CalcScore();
        });

    // 输出排序结果
    std::cout << "=== 无源码插件逆向优先级排序 ===" << std::endl;
    std::cout << std::left
              << std::setw(4)  << "排名"
              << std::setw(22) << "插件名称"
              << std::setw(8)  << "得分"
              << std::setw(12) << "建议方案" << std::endl;
    std::cout << std::string(46, '-') << std::endl;

    for (size_t i = 0; i < plugins.size(); ++i) {
        const auto& p = plugins[i];
        std::string strategy;
        if (p.alternative == Alternative::Full)
            strategy = "重写替代";
        else if (p.alternative == Alternative::Partial)
            strategy = "逆向+替代";
        else
            strategy = "必须逆向";

        std::cout << std::left
                  << std::setw(4)  << (i + 1)
                  << std::setw(22) << p.name
                  << std::setw(8)  << std::fixed
                  << std::setprecision(1) << p.CalcScore()
                  << strategy << std::endl;
    }

    return 0;
}
```

## 常见错误与排查

### 错误 1：占位插件注册后游戏仍然报错 "plugin not found"

**症状：** 已经创建了占位插件并编译成功，但游戏启动时仍然提示找不到 DLL。

**原因：** `NCB_MODULE_NAME` 中的模块名与游戏脚本中 `Plugins.link()` 的参数不一致。模块名是**区分大小写**的——`_internal_plugins` 使用 `std::map<ttstr, ...>`，而 `ttstr` 的比较是精确匹配。

**排查步骤：**

```cpp
// 错误写法——大小写不一致
#define NCB_MODULE_NAME TJS_W("textrender.dll")  // 全小写
// 游戏脚本中：Plugins.link("TextRender.dll")    // 首字母大写
// 结果：找不到模块！

// 正确写法——必须与游戏脚本完全一致
#define NCB_MODULE_NAME TJS_W("TextRender.dll")   // 与脚本一致
```

**解决方案：** 用文本搜索工具在游戏脚本中找到精确的 `Plugins.link` 调用，确保模块名字符串完全一致（包括大小写和 `.dll` 后缀）。

### 错误 2：替代实现的函数签名与游戏脚本调用不匹配

**症状：** 插件注册成功，但游戏运行到某处时报 TJS 错误：`member "xxx" does not exist` 或参数数量不对。

**原因：** 替代实现中注册的 TJS 函数名或参数数量与原始插件不同。例如原始 util_generic 中函数名是 `lerp`，但我们注册为 `GenericUtil_lerp`。

**排查步骤：**

```javascript
// 游戏脚本中的调用方式（假设）：
var result = GenericUtil.lerp(0, 100, 0.5);
// 这意味着 lerp 是 GenericUtil 对象的方法，不是全局函数

// 如果我们用 NCB_ATTACH_FUNCTION 注册为全局函数：
NCB_ATTACH_FUNCTION(lerp, GenericLerp);
// 游戏脚本调用 GenericUtil.lerp() 会报错——因为 lerp 不是 GenericUtil 的成员

// 正确做法：需要注册为类方法
// 先分析原始插件是注册了类还是全局函数
// 可以通过搜索游戏脚本中的调用模式来判断
```

**解决方案：**
1. 搜索游戏脚本中所有调用该插件功能的代码，确定是类方法调用还是全局函数调用
2. 如果是类方法，使用 `NCB_REGISTER_CLASS` 或 `NCB_ATTACH_CLASS` 注册
3. 如果是全局函数，使用 `NCB_ATTACH_FUNCTION` 注册
4. 函数名必须精确匹配游戏脚本中的调用名

### 错误 3：像素混合结果出现色偏或白边

**症状：** 实现了 layerExAlpha 的混合函数后，游戏画面中半透明图层的边缘出现白色光晕或颜色不正确。

**原因：** KiriKiri 的像素格式是 **BGRA**（蓝在低位），而代码中按 RGBA 顺序处理了像素通道。字节顺序搞反会导致红色和蓝色互换。

**排查步骤：**

```cpp
// 错误：假设像素格式是 RGBA
struct Pixel {
    uint8_t r, g, b, a;  // 错！KiriKiri 用 BGRA
};

// 正确：KiriKiri 使用 BGRA 格式
struct Pixel {
    uint8_t b, g, r, a;  // 蓝-绿-红-透明
};

// 验证方法：创建一个纯红色像素 (R=255, G=0, B=0)
// 如果显示为蓝色，说明 R 和 B 通道搞反了
```

**解决方案：**
1. 确认目标平台的像素字节顺序（KiriKiri 是 BGRA）
2. 如果使用整数操作像素（如 `uint32_t color = 0xAARRGGBB`），注意位移方向
3. 用纯色测试图像验证：纯红(255,0,0)、纯绿(0,255,0)、纯蓝(0,0,255)分别显示是否正确

```javascript
    %["name" => "util_generic.dll", "test" => "GenericUtil_lerp"],
    %["name" => "util_system.dll",  "test" => "SystemUtil_getOSVersion"],
    %["name" => "layerExAlpha.dll", "test" => ""],
    %["name" => "PackinOne.dll",    "test" => ""],
    %["name" => "util_graph.dll",   "test" => ""],
    %["name" => "TextRender.dll",   "test" => ""],
    %["name" => "multiimage.dll",   "test" => ""],
    %["name" => "lzfs.dll",         "test" => ""],
];

System.inform("=== C 类插件兼容性检测 ===");
var loaded = 0;     // 加载成功数
var functional = 0; // 功能可用数
var failed = 0;     // 加载失败数

for (var i = 0; i < plugins.count; i++) {
    var p = plugins[i];
    var status = "❌ 缺失";

    try {
        // 尝试加载插件
        Plugins.link(p.name);
        status = "✅ 已加载";
        loaded++;

        // 如果有特征函数，测试是否为真实实现
        if (p.test != "") {
            try {
                // 尝试调用特征函数
                var func = global[p.test];
                if (func !== void) {
                    status = "✅ 功能可用";
                    functional++;
                } else {
                    status = "⚠️ 占位（无功能）";
                }
            } catch(e) {
                status = "⚠️ 占位（调用失败）";
            }
        }
    } catch(e) {
        status = "❌ 缺失";
        failed++;
    }

    System.inform(p.name + " : " + status);
}

System.inform("");
System.inform("--- 检测结果 ---");
System.inform("加载成功: " + loaded + "/" + plugins.count);
System.inform("功能可用: " + functional + "/" + plugins.count);
System.inform("加载失败: " + failed + "/" + plugins.count);
System.inform(
    "兼容性评分: " +
    (loaded * 100 / plugins.count) + "%");
```

**脚本说明：**
- 使用 `try-catch` 包裹 `Plugins.link()` 调用，加载失败不会导致脚本中断
- 对有特征函数的插件，进一步检测该函数是否存在于全局命名空间
- 区分三种状态：功能可用（真实实现）、占位（能加载但无功能）、缺失（加载失败）
- 最终输出兼容性百分比
