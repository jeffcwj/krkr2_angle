## 术语预览

本节将涉及以下术语，先有个印象，正文中会逐一详细讲解：

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| 内存布局 | Memory Layout | 数据结构在内存中的字节排列方式——哪个字段在第几个偏移处 |
| 结构体对齐 | Structure Alignment/Padding | 编译器为了 CPU 访问效率，在结构体字段之间插入填充字节 |
| 虚函数表 | Virtual Function Table (vtable) | C++ 多态的实现机制——每个含虚函数的对象开头都有一个指向函数指针表的指针 |
| 成员偏移 | Member Offset | 结构体/类的某个字段相对于对象首地址的字节偏移量 |
| 控制流图 | Control Flow Graph (CFG) | 将函数的执行路径用图表示——节点是基本块，边是跳转关系 |
| 基本块 | Basic Block | 一段没有跳转的连续指令序列——从第一条执行到最后一条，中间不会分支 |
| 调用约定 | Calling Convention | 函数调用时参数如何传递（寄存器/栈）、谁清理栈的规则 |
| 符号恢复 | Symbol Recovery | 在反编译代码中给匿名变量和函数赋予有意义的名称 |

## 一、从反编译代码识别数据结构

### 1.1 结构体的内存布局识别

当反编译器（Decompiler，将机器码转换回高级语言伪代码的工具）输出代码时，结构体通常不会以 `struct { ... }` 的形式呈现，而是表现为对某个指针的偏移访问。识别这些偏移模式是推断数据结构的第一步。

**反编译代码中结构体访问的典型模式：**

```cpp
// Ghidra 反编译输出（未经整理）
// param_1 是某个未知结构体的指针
void FUN_10003000(int* param_1) {
    // 偏移 0x00: 直接解引用——第一个字段
    int field_0 = *param_1;

    // 偏移 0x04: 第二个 int 字段
    int field_4 = *(int*)((char*)param_1 + 4);

    // 偏移 0x08: 一个 char* 指针
    char* field_8 = *(char**)((char*)param_1 + 8);

    // 偏移 0x0C: 一个 float 字段
    // 注意：float 也是 4 字节，但反编译器可能显示为 int
    float field_c = *(float*)((char*)param_1 + 0xc);

    // 偏移 0x10: 一个 short 字段
    short field_10 = *(short*)((char*)param_1 + 0x10);

    // 偏移 0x14 (不是 0x12!)——说明 0x12-0x13 是填充字节
    int field_14 = *(int*)((char*)param_1 + 0x14);
}
```

**从上述偏移模式推断出的结构体：**

```cpp
// 推断结果——注意对齐和填充
struct UnknownStruct {
    int field_0;        // 偏移 0x00, 大小 4 字节
    int field_4;        // 偏移 0x04, 大小 4 字节
    char* field_8;      // 偏移 0x08, 大小 4 字节（32位指针）
    float field_c;      // 偏移 0x0C, 大小 4 字节
    short field_10;     // 偏移 0x10, 大小 2 字节
    // 2 字节填充      // 偏移 0x12, 编译器自动插入
    int field_14;       // 偏移 0x14, 大小 4 字节
};
// 总大小: 0x18 (24) 字节
```

**内存布局的 ASCII 图示：**

```
偏移:  0x00  0x04  0x08  0x0C  0x10 0x12 0x14
       ┌─────┬─────┬─────┬─────┬───┬───┬─────┐
字段:  │ int │ int │char*│float│s16│PAD│ int │
       └─────┴─────┴─────┴─────┴───┴───┴─────┘
大小:    4     4     4     4    2   2    4    = 24 字节
```

### 1.2 结构体对齐规则

结构体对齐（Structure Alignment）是编译器为了让 CPU 高效访问内存而自动插入的填充字节。不同编译器和平台的对齐规则略有差异，但 MSVC（Microsoft Visual C++ 编译器，KiriKiri 插件的主要编译工具）的默认规则如下：

| 数据类型 | 大小（字节） | 对齐要求 | 说明 |
|----------|------------|---------|------|
| `char` / `bool` | 1 | 1 字节对齐 | 可以放在任何地址 |
| `short` | 2 | 2 字节对齐 | 地址必须是 2 的倍数 |
| `int` / `float` | 4 | 4 字节对齐 | 地址必须是 4 的倍数 |
| `double` / `long long` | 8 | 8 字节对齐 | 地址必须是 8 的倍数 |
| 指针（32 位） | 4 | 4 字节对齐 | 32 位平台 |
| 指针（64 位） | 8 | 8 字节对齐 | 64 位平台 |

**验证对齐的 C++ 程序：**

```cpp
#include <cstdio>
#include <cstddef>  // offsetof 宏的头文件

// 模拟从反编译中推断出的结构体
struct ReversedStruct {
    int type;           // 偏移 0x00
    int flags;          // 偏移 0x04
    char* name;         // 偏移 0x08
    float opacity;      // 偏移 0x0C
    short layer_id;     // 偏移 0x10
    // 2 字节填充
    int parent_id;      // 偏移 0x14
};

int main() {
    // offsetof 宏返回字段在结构体中的字节偏移量
    // 用这个宏验证推断的偏移是否正确
    printf("type:      偏移=0x%02zX\n",
           offsetof(ReversedStruct, type));
    printf("flags:     偏移=0x%02zX\n",
           offsetof(ReversedStruct, flags));
    printf("name:      偏移=0x%02zX\n",
           offsetof(ReversedStruct, name));
    printf("opacity:   偏移=0x%02zX\n",
           offsetof(ReversedStruct, opacity));
    printf("layer_id:  偏移=0x%02zX\n",
           offsetof(ReversedStruct, layer_id));
    printf("parent_id: 偏移=0x%02zX\n",
           offsetof(ReversedStruct, parent_id));
    printf("总大小:    0x%02zX (%zu 字节)\n",
           sizeof(ReversedStruct),
           sizeof(ReversedStruct));
    return 0;
}
```

### 1.3 C++ 类与虚函数表识别

当反编译代码中出现对对象首地址（偏移 0x00）的二级指针解引用，再加上固定偏移的间接调用，这就是虚函数表（vtable）调用的特征。虚函数表是 C++ 实现多态的核心机制——编译器为每个含虚函数的类生成一张函数指针表，对象的第一个隐藏字段指向这张表。

```cpp
// 反编译中的虚函数调用模式
void FUN_10005000(int* obj) {
    // obj+0x00: vtable 指针（所有含虚函数的对象都有）
    int* vtable = *(int**)obj;

    // vtable[0] = 第一个虚函数（通常是析构函数）
    // vtable[1] = 第二个虚函数
    // vtable[2] = 第三个虚函数
    // ...

    // 调用 vtable[2]——第三个虚函数
    // 偏移 = 2 * 4 = 8（32位）或 2 * 8 = 16（64位）
    int result = (*(code*)(vtable[2]))(obj, 42);
    // ↑ 等价于: obj->virtual_method_2(42)
}

// 推断出的类定义
class ReversedClass {
public:
    // vtable 指针由编译器自动生成，不需要手动声明
    virtual ~ReversedClass();          // vtable[0]
    virtual void method_1();           // vtable[1]
    virtual int method_2(int param);   // vtable[2]

    // 非虚成员从 vtable 指针之后开始
    int field_04;    // 偏移 0x04（32位）
    float field_08;  // 偏移 0x08
};
```

**虚函数表的内存布局：**

```
对象内存:                     vtable 数组:
┌──────────────┐             ┌──────────────────────┐
│ vtable_ptr ──┼────────────→│ [0] ~ReversedClass() │
│              │             │ [1] method_1()       │
├──────────────┤             │ [2] method_2(int)    │
│ field_04     │             └──────────────────────┘
├──────────────┤
│ field_08     │
└──────────────┘

注意：vtable 指针占 4 字节（32位）或 8 字节（64位），
所以第一个用户字段的偏移是 0x04（32位）或 0x08（64位）。
```

**识别 iTJSDispatch2 的技巧：** 在 KiriKiri 插件逆向中，`iTJSDispatch2` 是最常见的虚函数类。它的 vtable 布局是固定的（定义在 `tp_stub.h` 中）。当你看到对某个对象偏移 0x28（32 位）处的虚函数调用时，这几乎一定是 `PropSet`。以下是 iTJSDispatch2 的部分 vtable 布局：

| vtable 偏移（32 位） | 偏移（64 位） | 方法名 | 功能 |
|---------------------|-------------|--------|------|
| 0x00 | 0x00 | `AddRef` | 增加引用计数 |
| 0x04 | 0x08 | `Release` | 减少引用计数 |
| 0x08 | 0x10 | `FuncCall` | 调用对象作为函数 |
| 0x0C | 0x18 | `FuncCallByNum` | 按序号调用函数 |
| 0x10 | 0x20 | `PropGet` | 获取属性值 |
| 0x14 | 0x28 | `PropGetByNum` | 按序号获取属性 |
| 0x18 | 0x30 | `PropSet` | 设置属性值 |
| 0x1C | 0x38 | `PropSetByNum` | 按序号设置属性 |
| 0x20 | 0x40 | `GetCount` | 获取成员数量 |
| 0x24 | 0x48 | `GetCountByNum` | 按序号获取数量 |
| 0x28 | 0x50 | `DeleteMember` | 删除成员 |

> **注意：** 上表的偏移可能因编译器版本和虚继承情况有所不同。实际分析时应以反编译结果为准，但此表可作为快速参考。

## 二、枚举与常量推断

### 2.1 从比较操作推断枚举值

反编译代码中的 `switch-case` 语句或连续 `if-else` 比较链，通常暗示原始代码使用了枚举类型。当一个变量被与多个整数常量比较时，这些常量很可能是枚举值。

```cpp
// 反编译输出——一个 switch 语句
void FUN_10006000(int* obj, int mode) {
    switch(mode) {
    case 0:
        // 调用某个处理函数
        FUN_10006100(obj);  // 处理模式 0
        break;
    case 1:
        FUN_10006200(obj);  // 处理模式 1
        break;
    case 2:
        FUN_10006300(obj);  // 处理模式 2
        break;
    case 3:
        FUN_10006400(obj);  // 处理模式 3
        break;
    default:
        // 设置错误标志
        *(int*)((char*)obj + 0x20) = -1;
        break;
    }
}

// 推断：mode 是一个枚举类型，有 4 个有效值（0-3）
// 结合上下文（如函数名中的线索、调用链中的字符串常量），
// 可以进一步推断每个值的含义
```

**推断出的枚举定义：**

```cpp
// 如果从上下文分析得知这是一个压缩模式选择器
enum CompressionMode {
    COMPRESS_NONE   = 0,  // 无压缩（存储模式）
    COMPRESS_ZLIB   = 1,  // zlib 压缩
    COMPRESS_LZ4    = 2,  // LZ4 快速压缩
    COMPRESS_LZMA   = 3,  // LZMA 高压缩比
};
// 超出范围时设置错误标志（-1）
```

### 2.2 从位操作推断标志位

标志位（Flags）是另一种常见的常量模式。当反编译代码中出现 `&`（按位与）和 `|`（按位或）操作，且操作数是 2 的幂时，几乎可以确定这是标志位操作。

```cpp
// 反编译输出——标志位操作
void FUN_10007000(int* obj) {
    int flags = *(int*)((char*)obj + 0x04);

    // 检查第 0 位
    if (flags & 1) {
        // ... 某个功能已启用
    }

    // 检查第 1 位
    if (flags & 2) {
        // ... 另一个功能已启用
    }

    // 设置第 2 位
    *(int*)((char*)obj + 0x04) = flags | 4;

    // 清除第 3 位
    *(int*)((char*)obj + 0x04) = flags & ~8;
}

// 推断出的标志位定义
enum RenderFlags {
    FLAG_VISIBLE    = 0x01,  // 第 0 位: 是否可见
    FLAG_ENABLED    = 0x02,  // 第 1 位: 是否启用
    FLAG_DIRTY      = 0x04,  // 第 2 位: 需要重绘
    FLAG_FOCUSED    = 0x08,  // 第 3 位: 是否拥有焦点
};
```

## 三、算法逻辑还原

### 3.1 循环结构识别

反编译器通常能很好地还原 `for`/`while`/`do-while` 循环，但有时会输出难以阅读的 `goto` 语句。识别循环的关键是找到计数器变量和边界条件。

```cpp
// 反编译输出——像素处理循环（典型的 KiriKiri 图像插件模式）
void FUN_10008000(int* src_pixels, int* dst_pixels,
                  int width, int height) {
    int i = 0;
    // 外层循环：遍历每一行
    while (i < height) {
        int j = 0;
        // 内层循环：遍历每行的每个像素
        while (j < width) {
            // 从源像素读取 32 位 BGRA 颜色值
            int pixel = src_pixels[i * width + j];

            // 提取颜色分量（位操作）
            int b = pixel & 0xFF;           // 蓝色（最低字节）
            int g = (pixel >> 8) & 0xFF;    // 绿色
            int r = (pixel >> 16) & 0xFF;   // 红色
            int a = (pixel >> 24) & 0xFF;   // Alpha

            // 预乘 Alpha 混合计算
            // (color * alpha + 127) / 255 是整数近似的 color * alpha / 255.0
            b = (b * a + 127) / 255;
            g = (g * a + 127) / 255;
            r = (r * a + 127) / 255;

            // 重新组合为 BGRA
            dst_pixels[i * width + j] =
                b | (g << 8) | (r << 16) | (a << 24);

            j = j + 1;
        }
        i = i + 1;
    }
}
```

**还原为整洁的 C++ 代码：**

```cpp
#include <cstdint>

// 预乘 Alpha 混合——将 BGRA 像素的 RGB 分量乘以 Alpha
// 用于 layerExAlpha 等 KiriKiri 图像处理插件
void premultiply_alpha(
    const uint32_t* src,  // 源像素数组（BGRA 格式）
    uint32_t* dst,        // 目标像素数组
    int width,            // 图像宽度（像素）
    int height            // 图像高度（像素）
) {
    const int total = width * height;  // 总像素数
    for (int i = 0; i < total; ++i) {
        uint32_t pixel = src[i];

        // 提取 BGRA 分量
        uint32_t b = (pixel >>  0) & 0xFF;
        uint32_t g = (pixel >>  8) & 0xFF;
        uint32_t r = (pixel >> 16) & 0xFF;
        uint32_t a = (pixel >> 24) & 0xFF;

        // 预乘 Alpha（整数近似公式）
        // 数学上等价于 color = color * alpha / 255.0
        // 加 127 是为了四舍五入而非截断
        b = (b * a + 127) / 255;
        g = (g * a + 127) / 255;
        r = (r * a + 127) / 255;

        // 重新打包为 BGRA
        dst[i] = (a << 24) | (r << 16) | (g << 8) | b;
    }
}
```

### 3.2 控制流图分析

当反编译器输出的代码包含复杂的 `goto` 语句或嵌套条件时，使用控制流图（Control Flow Graph, CFG——将函数拆分为基本块，用箭头连接它们的跳转关系）可以帮助理解程序逻辑。

**Ghidra 中查看控制流图：**

1. 在 Listing 窗口中定位到目标函数
2. Window → Function Graph
3. 每个节点是一个基本块（Basic Block——一段没有分支的连续指令）
4. 绿色箭头 = 条件为真的跳转
5. 红色箭头 = 条件为假的跳转
6. 蓝色箭头 = 无条件跳转


## 四、Ghidra 类型系统辅助分析

### 4.1 自定义数据类型

Ghidra 允许你在分析过程中定义自定义结构体和类型，一旦应用到变量上，反编译器会自动将偏移访问替换为有意义的字段名——极大地提升可读性。

**在 Ghidra 中创建自定义结构体：**

1. 打开 Data Type Manager（窗口左下角面板）
2. 右键项目名称 → New → Structure
3. 输入结构体名称（如 `PackinOneHeader`）
4. 添加字段：
   - 点击 "Add" 按钮
   - 设置字段名、数据类型、偏移量
5. 保存结构体定义

```cpp
// 在 Ghidra 中定义以下结构体后：
struct PackinOneHeader {
    char magic[4];       // 偏移 0x00: 文件魔数 "PKN1"
    uint32_t version;    // 偏移 0x04: 版本号
    uint32_t file_count; // 偏移 0x08: 包含的文件数量
    uint32_t index_offset;// 偏移 0x0C: 文件索引的偏移量
    uint32_t data_offset; // 偏移 0x10: 数据区域的偏移量
    uint32_t total_size;  // 偏移 0x14: 归档总大小
};

// 应用到变量前的反编译代码：
void FUN_10009000(int* param_1) {
    if (*(int*)param_1 == 0x314E4B50) {  // 魔数检查
        int count = *(int*)((char*)param_1 + 8);
        int offset = *(int*)((char*)param_1 + 0xC);
        // ...
    }
}

// 应用到变量后的反编译代码——可读性大幅提升：
void FUN_10009000(PackinOneHeader* header) {
    if (header->magic == "PKN1") {        // 魔数检查
        int count = header->file_count;
        int offset = header->index_offset;
        // ...
    }
}
```

### 4.2 导入 KiriKiri 头文件

Ghidra 支持导入 C 头文件来定义类型。对于 KiriKiri 插件分析，可以导入 `tp_stub.h`（TJS 插件 SDK 的核心头文件）来让反编译器自动识别 TJS 类型。

## 五、实战：从反编译代码还原压缩算法

### 5.1 完整还原示例

本小节以一个简化的 PackinOne 风格压缩算法为例，演示从反编译代码到可编译 C++ 代码的完整还原流程。PackinOne 是 KiriKiri 生态中的资源打包插件，它使用自定义的 LZ 变种压缩（LZ 压缩——一种基于"滑动窗口"的通用压缩算法，核心思想是用"回溯引用"替代重复出现的数据）来减小归档体积。

**步骤 1：识别函数签名和局部变量**

在 Ghidra 中定位到压缩/解压函数后，首先观察参数和局部变量：

```cpp
// Ghidra 反编译输出（原始，未修改）
// 函数地址: 0x10010000
// 参数分析:
//   param_1: 输出缓冲区指针（目标地址）
//   param_2: 输入缓冲区指针（压缩数据源）
//   param_3: 压缩数据长度（字节）
//   返回值: 解压后的数据长度
int FUN_10010000(char* param_1, char* param_2, int param_3) {
    char* local_4 = param_1;    // 输出位置指针（会随写入推进）
    char* local_8 = param_2;    // 输入位置指针（会随读取推进）
    char* local_c = param_2 + param_3;  // 输入数据的结束边界

    while (local_8 < local_c) {
        // 读取操作码（控制字节）
        unsigned char opcode = *local_8;
        local_8 = local_8 + 1;

        if ((opcode & 0x80) == 0) {
            // 最高位为 0: 字面量复制
            // 低 7 位表示要复制的字节数
            int count = (opcode & 0x7F) + 1;
            int i = 0;
            while (i < count) {
                *local_4 = *local_8;
                local_4 = local_4 + 1;
                local_8 = local_8 + 1;
                i = i + 1;
            }
        } else {
            // 最高位为 1: 回溯引用（匹配复制）
            // 读取第二个字节，组合成偏移和长度
            unsigned char byte2 = *local_8;
            local_8 = local_8 + 1;

            // 偏移量: opcode 的低 5 位 + byte2 的全部 8 位
            // 组合成 13 位偏移（最大 8191）
            int offset = ((opcode & 0x1F) << 8) | byte2;

            // 长度: opcode 的第 5-6 位（bit5, bit6）+ 3
            // 最小匹配长度为 3
            int length = ((opcode >> 5) & 0x03) + 3;

            // 从已输出数据中回溯复制
            char* src = local_4 - offset - 1;
            int j = 0;
            while (j < length) {
                *local_4 = src[j];
                local_4 = local_4 + 1;
                j = j + 1;
            }
        }
    }
    // 返回解压后的总字节数
    return (int)(local_4 - param_1);
}
```

**步骤 2：分析操作码编码格式**

从反编译代码中，我们可以推断出操作码的位级编码。这是逆向压缩算法最关键的一步——把"魔数般"的位操作翻译成清晰的协议格式：

```
操作码格式（1 字节）：

情况 A: 字面量复制（最高位 = 0）
┌─┬─────────────┐
│0│  count - 1   │  count: 1~128 字节的字面量数据
└─┴─────────────┘
 7  6           0

情况 B: 回溯引用（最高位 = 1）
┌─┬───┬─────────┐  ┌─────────────────┐
│1│len│ off_hi   │  │    off_lo        │
└─┴───┴─────────┘  └─────────────────┘
 7 6-5  4       0    7               0
  第一字节              第二字节

  len: (bit6-bit5) + 3  → 匹配长度 3~6
  offset: (off_hi << 8) | off_lo  → 回溯偏移 0~8191
  实际源地址: output_pos - offset - 1
```

**步骤 3：还原为整洁的 C++ 代码**

```cpp
#include <cstdint>
#include <cstddef>
#include <cstring>
#include <stdexcept>
#include <vector>

// PackinOne 风格 LZ 变种解压算法
// 从反编译代码还原——适用于 KiriKiri PackinOne 插件的资源归档解压
//
// 编码格式:
//   字面量: [0xxxxxxx] [data...]  — 复制 (x+1) 字节的原始数据
//   引用:   [1LL OOOOO] [OOOOOOOO] — 从已输出数据回溯复制
//           L = length-3 (2位, 匹配长度3-6)
//           O = offset (13位, 回溯距离0-8191)
size_t pkn_decompress(
    uint8_t* dst,           // 输出缓冲区（调用者需预分配足够空间）
    const uint8_t* src,     // 压缩数据
    size_t src_len          // 压缩数据长度
) {
    uint8_t* out = dst;             // 当前输出位置
    const uint8_t* in = src;        // 当前输入位置
    const uint8_t* in_end = src + src_len;  // 输入结束边界

    while (in < in_end) {
        uint8_t opcode = *in++;     // 读取操作码

        if ((opcode & 0x80) == 0) {
            // 字面量复制: 低 7 位 + 1 = 复制字节数
            int count = (opcode & 0x7F) + 1;
            if (in + count > in_end) {
                throw std::runtime_error(
                    "pkn_decompress: 字面量数据超出输入边界");
            }
            std::memcpy(out, in, count);
            out += count;
            in += count;
        } else {
            // 回溯引用: 需要第二个字节
            if (in >= in_end) {
                throw std::runtime_error(
                    "pkn_decompress: 引用操作码缺少第二字节");
            }
            uint8_t byte2 = *in++;

            // 解码偏移量（13 位）和长度（2 位 + 3）
            int offset = ((opcode & 0x1F) << 8) | byte2;
            int length = ((opcode >> 5) & 0x03) + 3;

            // 计算回溯源地址
            uint8_t* match_src = out - offset - 1;
            if (match_src < dst) {
                throw std::runtime_error(
                    "pkn_decompress: 回溯偏移超出输出缓冲区");
            }

            // 逐字节复制（不能用 memcpy，因为源和目标可能重叠）
            for (int i = 0; i < length; ++i) {
                *out++ = match_src[i];
            }
        }
    }

    return static_cast<size_t>(out - dst);  // 返回解压后大小
}
```

**步骤 4：编写验证程序**

还原算法后必须验证其正确性。编写一个配套的压缩函数，然后做"压缩→解压→比较"的往返测试（Round-trip Test——将数据压缩后再解压，验证结果与原始数据完全一致）：

```cpp
#include <cstdint>
#include <cstdio>
#include <cstring>
#include <vector>
#include <stdexcept>

// 前向声明（完整实现见上方）
size_t pkn_decompress(uint8_t* dst, const uint8_t* src, size_t src_len);

// 简易压缩器（仅用于验证，不追求压缩率）
// 策略: 对每个位置，在滑动窗口中查找最长匹配
std::vector<uint8_t> pkn_compress(const uint8_t* data, size_t data_len) {
    std::vector<uint8_t> out;
    out.reserve(data_len);  // 预分配（最坏情况不会比原始大很多）

    size_t pos = 0;
    std::vector<uint8_t> literal_buf;  // 待输出的字面量缓冲

    // 刷新字面量缓冲区的辅助 lambda
    auto flush_literals = [&]() {
        while (!literal_buf.empty()) {
            // 每次最多输出 128 字节字面量
            int chunk = (literal_buf.size() > 128) ?
                        128 : (int)literal_buf.size();
            out.push_back((uint8_t)(chunk - 1));  // 操作码: 0 + (count-1)
            out.insert(out.end(),
                       literal_buf.begin(),
                       literal_buf.begin() + chunk);
            literal_buf.erase(literal_buf.begin(),
                              literal_buf.begin() + chunk);
        }
    };

    while (pos < data_len) {
        int best_len = 0;
        int best_off = 0;

        // 在滑动窗口中搜索匹配（窗口大小最大 8192）
        int window_start = (pos > 8192) ? (int)(pos - 8192) : 0;
        for (int w = window_start; w < (int)pos; ++w) {
            int match_len = 0;
            int max_match = 6;  // 最大匹配长度为 6
            if ((int)(data_len - pos) < max_match) {
                max_match = (int)(data_len - pos);
            }
            while (match_len < max_match &&
                   data[w + match_len] == data[pos + match_len]) {
                ++match_len;
            }
            if (match_len >= 3 && match_len > best_len) {
                best_len = match_len;
                best_off = (int)pos - w - 1;
            }
        }

        if (best_len >= 3) {
            // 找到匹配——先刷新字面量，再输出引用
            flush_literals();
            uint8_t len_code = (uint8_t)(best_len - 3);  // 0~3
            uint8_t off_hi = (uint8_t)(best_off >> 8);    // 高 5 位
            uint8_t off_lo = (uint8_t)(best_off & 0xFF);  // 低 8 位
            out.push_back(0x80 | (len_code << 5) | off_hi);
            out.push_back(off_lo);
            pos += best_len;
        } else {
            // 无匹配——累积字面量
            literal_buf.push_back(data[pos]);
            ++pos;
        }
    }
    flush_literals();  // 别忘了刷新最后的字面量
    return out;
}

int main() {
    // 测试数据: 包含重复模式，便于验证压缩引用
    const char* test_str =
        "ABCABCABCDEFDEFDEFGHIGHIGHIJKLJKLJKL"
        "Hello World! Hello World! Hello World!";
    const uint8_t* test_data = (const uint8_t*)test_str;
    size_t test_len = strlen(test_str);

    printf("原始数据: %zu 字节\n", test_len);

    // 压缩
    std::vector<uint8_t> compressed = pkn_compress(test_data, test_len);
    printf("压缩后:   %zu 字节 (压缩率 %.1f%%)\n",
           compressed.size(),
           100.0 * compressed.size() / test_len);

    // 解压
    std::vector<uint8_t> decompressed(test_len * 2);  // 预留足够空间
    size_t decompressed_len = pkn_decompress(
        decompressed.data(),
        compressed.data(),
        compressed.size());

    printf("解压后:   %zu 字节\n", decompressed_len);

    // 验证
    if (decompressed_len == test_len &&
        memcmp(test_data, decompressed.data(), test_len) == 0) {
        printf("✅ 往返测试通过! 数据完全一致\n");
    } else {
        printf("❌ 往返测试失败!\n");
        printf("  期望 %zu 字节, 实际 %zu 字节\n",
               test_len, decompressed_len);
        return 1;
    }
    return 0;
}
```

### 实践 2：从 switch-case 推断枚举并还原处理函数

**目标：** 分析一段包含 switch-case 的反编译代码，推断枚举定义并将每个 case 还原为有意义的函数。

**步骤：**

1. 阅读以下反编译代码：

```cpp
// Ghidra 反编译输出——某个图像格式转换函数

    int result = 0;

    switch (mode) {
    case 0:
        // 直接复制，不做任何变换
        // param_3 是目标缓冲区
        // *(param_1 + 2) 是像素数量
        {
            int count = *(int*)((char*)param_1 + 8);
            int* src = *(int**)((char*)param_1 + 0x10);
            int i = 0;
            while (i < count) {
                param_3[i] = src[i];
                i = i + 1;
            }
            result = count;
        }
        break;
    case 1:
        // BGRA → RGBA 转换
        // 交换 B 和 R 通道
        {
            int count = *(int*)((char*)param_1 + 8);
            int* src = *(int**)((char*)param_1 + 0x10);

            while (i < count) {

                int b = pixel & 0xFF;
                int g = (pixel >> 8) & 0xFF;
                int r = (pixel >> 16) & 0xFF;
                int a = (pixel >> 24) & 0xFF;
                param_3[i] = r | (g << 8) | (b << 16) | (a << 24);
                i = i + 1;
            }

        }
        break;
    case 2:

            int* src = *(int**)((char*)param_1 + 0x10);
            int i = 0;
            while (i < count) {
                int pixel = src[i];
                int b = pixel & 0xFF;
                int g = (pixel >> 8) & 0xFF;
                int r = (pixel >> 16) & 0xFF;
                int a = (pixel >> 24) & 0xFF;
                b = (b * a + 127) / 255;
                g = (g * a + 127) / 255;
                r = (r * a + 127) / 255;
                param_3[i] = b | (g << 8) | (r << 16) | (a << 24);
                i = i + 1;
            }
            result = count;
        }
        break;
    case 3:
        // 灰度转换
        {
            int count = *(int*)((char*)param_1 + 8);
            int* src = *(int**)((char*)param_1 + 0x10);
            int i = 0;
            while (i < count) {
                int pixel = src[i];
                int b = pixel & 0xFF;
                int g = (pixel >> 8) & 0xFF;
                int r = (pixel >> 16) & 0xFF;
                int a = (pixel >> 24) & 0xFF;
                // 加权灰度公式: 0.299R + 0.587G + 0.114B
                // 整数近似: (77R + 150G + 29B) >> 8
                int gray = (77 * r + 150 * g + 29 * b) >> 8;
                param_3[i] = gray | (gray << 8) |
                             (gray << 16) | (a << 24);
                i = i + 1;
            }
            result = count;
        }
        break;
    default:
        result = -1;  // 不支持的模式
        break;
    }
    return result;
}
```

```cpp
#include <cstdio>
#include <cstdint>
#include <cstring>

enum PixelConvertMode {
    CONVERT_COPY        = 0,
    CONVERT_BGRA_RGBA   = 1,
    CONVERT_PREMULTIPLY = 2,
    CONVERT_GRAYSCALE   = 3,
};

// 像素转换函数（还原版本）
// 返回处理的像素数，失败返回 -1
int pixel_convert(
    const uint32_t* src,    // 源像素（BGRA 格式）
    uint32_t* dst,          // 目标像素
    int count,              // 像素数量
    PixelConvertMode mode   // 转换模式
) {
    switch (mode) {
    case CONVERT_COPY:
        memcpy(dst, src, count * sizeof(uint32_t));
        return count;

    case CONVERT_BGRA_RGBA:
        for (int i = 0; i < count; ++i) {
            uint32_t p = src[i];
            uint32_t b = (p >>  0) & 0xFF;
            uint32_t g = (p >>  8) & 0xFF;
            uint32_t r = (p >> 16) & 0xFF;
            uint32_t a = (p >> 24) & 0xFF;
            // 交换 R 和 B
            dst[i] = (r << 0) | (g << 8) | (b << 16) | (a << 24);
        }
        return count;

    case CONVERT_PREMULTIPLY:
        for (int i = 0; i < count; ++i) {
            uint32_t p = src[i];
            uint32_t b = (p >>  0) & 0xFF;
            uint32_t g = (p >>  8) & 0xFF;
            uint32_t r = (p >> 16) & 0xFF;
            uint32_t a = (p >> 24) & 0xFF;
            b = (b * a + 127) / 255;
            g = (g * a + 127) / 255;
            r = (r * a + 127) / 255;
            dst[i] = (b << 0) | (g << 8) | (r << 16) | (a << 24);
        }
        return count;

    case CONVERT_GRAYSCALE:
        for (int i = 0; i < count; ++i) {
            uint32_t p = src[i];
            uint32_t b = (p >>  0) & 0xFF;
            uint32_t g = (p >>  8) & 0xFF;
            uint32_t r = (p >> 16) & 0xFF;
            uint32_t a = (p >> 24) & 0xFF;
            uint32_t gray = (77 * r + 150 * g + 29 * b) >> 8;
            dst[i] = (gray << 0) | (gray << 8) |
                     (gray << 16) | (a << 24);
        }
        return count;

    default:
        return -1;
    }
}

int main() {
    // 测试像素: 半透明红色 (BGRA: B=0, G=0, R=255, A=128)
    uint32_t src_pixel = 0x80FF0000;  // BGRA 格式

    printf("  B=%d G=%d R=%d A=%d\n\n",
           (src_pixel >> 0) & 0xFF,
           (src_pixel >> 8) & 0xFF,
           (src_pixel >> 16) & 0xFF,
           (src_pixel >> 24) & 0xFF);

    // 测试每种模式
    const char* mode_names[] = {
        "COPY", "BGRA→RGBA", "PREMULTIPLY", "GRAYSCALE"
    };
    for (int m = 0; m <= 3; ++m) {
        pixel_convert(&src_pixel, &dst_pixel, 1,
                      (PixelConvertMode)m);
        printf("模式 %d (%s): 0x%08X\n", m, mode_names[m], dst_pixel);
        printf("  B=%d G=%d R=%d A=%d\n",
               (dst_pixel >> 0) & 0xFF,
               (dst_pixel >> 8) & 0xFF,
               (dst_pixel >> 16) & 0xFF,
               (dst_pixel >> 24) & 0xFF);
    }
    return 0;
}
```

## 常见错误与排查

### 错误 1：对齐假设错误——64 位偏移用于 32 位二进制

**症状：** 推断的结构体在验证程序中偏移不匹配，所有指针字段后面的偏移都偏大 4 字节。

**原因：** KiriKiri 原版插件是 32 位 DLL。如果你在 64 位系统上编译验证程序但没有指定 `-m32`，指针会占 8 字节而非 4 字节，导致后续所有字段的偏移都不同。

**解决方案：**

```bash
# 方案 1: 显式指定 32 位编译（推荐）
g++ -m32 -o check check.cpp

# 方案 2: 在结构体中用 uint32_t 代替指针
# 适用于不需要实际解引用的场景
struct FixedLayout {
    int32_t type;
    int32_t flags;
    uint32_t pixels_ptr;  // 用 uint32_t 代替 uint32_t*
    // ...
};

# 方案 3: 用 #pragma pack 强制对齐
#pragma pack(push, 4)  // 强制 4 字节对齐
struct PackedStruct {
    // ...
};
#pragma pack(pop)
```

**检测脚本：**

```cpp
#include <cstdio>

int main() {
    printf("当前环境:\n");
    printf("  sizeof(int)    = %zu\n", sizeof(int));
    printf("  sizeof(void*)  = %zu\n", sizeof(void*));
    printf("  sizeof(long)   = %zu\n", sizeof(long));
    printf("  sizeof(size_t) = %zu\n", sizeof(size_t));

    if (sizeof(void*) == 4) {
        printf("  ✅ 32 位模式——与 KiriKiri 插件匹配\n");
    } else {
        printf("  ⚠️ 64 位模式——指针偏移将与 32 位插件不同\n");
        printf("  建议: 使用 -m32 编译，或用 uint32_t 代替指针\n");
    }
    return 0;
}
```

### 错误 2：忽略虚函数表指针导致偏移偏差

**症状：** 类的第一个用户字段的偏移应该是 0x04（32 位）或 0x08（64 位），但推断时从 0x00 开始，导致所有字段偏移错位 4 字节。

**原因：** 含有虚函数的 C++ 类，编译器会在对象开头插入一个隐藏的 vtable 指针。如果在推断结构体时忘记预留这个指针的空间，所有后续字段的偏移都会错。

**判断方法：**

```
如何判断反编译中的对象是否含有 vtable？

1. 观察偏移 0x00 处是否被当作二级指针解引用：
   int** vtable = *(int***)obj;  // 有 vtable
   int field = *(int*)obj;        // 无 vtable，直接是数据

2. 观察是否有间接调用模式：
   (*(code*)(*(int**)obj)[N])(obj, ...);  // vtable 调用
   这等价于: obj->virtual_method_N(...);

3. 如果构造函数中有 "MOV [obj], <const_addr>" 模式：
   该常量地址就是 vtable 的地址——100% 确认含有 vtable
```

**修复方法：**

```cpp
// 错误: 忘记 vtable 指针
struct Wrong {
    int field_00;   // 实际偏移是 0x04!
    int field_04;   // 实际偏移是 0x08!
};

// 正确: 预留 vtable 指针
struct Correct {
    // vtable_ptr 由编译器自动生成，但推断时需要算入
    // 在非虚类中手动模拟:
    void* vtable_ptr;   // 偏移 0x00, 4 字节（32位）
    int field_04;       // 偏移 0x04
    int field_08;       // 偏移 0x08
};

// 或者直接用虚析构函数让编译器自动加 vtable:
struct CorrectV2 {
    virtual ~CorrectV2() = default;  // 自动添加 vtable_ptr
    int field_04;   // 偏移 0x04
    int field_08;   // 偏移 0x08
};
```

### 错误 3：回溯引用中源和目标重叠导致解压数据错误

**症状：** 解压输出的前几个字节正确，但后面出现垃圾数据或重复的错误模式。

**原因：** 在 LZ 类压缩算法中，回溯引用可能指向尚未完全写入的数据——源区域和目标区域重叠。如果用 `memcpy` 代替逐字节复制，`memcpy` 的行为在源和目标重叠时是未定义的（Undefined Behavior, UB——C/C++ 标准不保证结果，不同编译器/平台可能产生不同输出）。

**错误代码：**

```cpp
// ❌ 错误: memcpy 不允许源和目标重叠
char* match_src = out - offset - 1;
memcpy(out, match_src, length);  // UB! 重叠时结果不确定
out += length;
```

**正确代码：**

```cpp
// ✅ 正确: 逐字节复制（处理重叠情况）
char* match_src = out - offset - 1;
for (int i = 0; i < length; ++i) {
    *out++ = match_src[i];  // 每次写入 1 字节，读取时数据已更新
}

// ✅ 也可以用 memmove（它能正确处理重叠）
// 但对于 LZ 解压中的"自引用"模式，memmove 的语义不对
// 因为 LZ 的重叠复制是有意的：
//   "abc" + 引用(offset=2, length=6) → "abcabcabc"
// memmove 会把源数据先保存再复制，得到 "abcabcabc"（恰好正确）
// 但这只是巧合——当 length > offset 时:
//   memmove 复制的是"快照"，而 LZ 需要的是"逐步展开"
// 所以还是老老实实逐字节复制最安全
```

**验证重叠情况的测试：**

```cpp
#include <cstdio>
#include <cstdint>
#include <cstring>

int main() {
    // 模拟 LZ 解压中的重叠引用
    // 目标: "ABCABCABC"（从 "ABC" 通过 offset=3, length=6 展开）
    char buf[20] = "ABC";  // 已写入 3 字节
    char* out = buf + 3;   // 当前输出位置

    int offset = 2;   // 回溯 3 字节（offset + 1 = 3）
    int length = 6;   // 复制 6 字节

    char* match_src = out - offset - 1;  // = buf + 0 = "ABC..."

    // 正确的逐字节复制
    for (int i = 0; i < length; ++i) {
        *out++ = match_src[i];
    }

    buf[9] = '\0';
    printf("结果: %s\n", buf);           // 期望: ABCABCABC
    printf("正确: %s\n",
           strcmp(buf, "ABCABCABC") == 0 ? "✅" : "❌");
    return 0;
}
```

## 本节小结

- **偏移模式识别**是推断结构体的基础——从反编译代码中的 `*(type*)((char*)ptr + offset)` 模式可以逐字段推断出完整的结构体定义
- **结构体对齐**（Alignment）会在字段之间插入填充字节——偏移不连续的地方就是填充，用 `offsetof` 宏可以验证推断是否正确
- **虚函数表**（vtable）是识别 C++ 类的关键标志——对象首地址处的二级指针解引用 + 固定偏移的间接调用 = vtable 调用
- **iTJSDispatch2 的 vtable 偏移表**是分析 KiriKiri 插件的速查工具——背住几个常用偏移（FuncCall=0x08, PropGet=0x10, PropSet=0x18）可以快速理解反编译代码
- **switch-case 暗示枚举**——连续整数比较通常对应枚举类型，结合上下文可以推断每个值的语义
- **位操作暗示标志位**——`& 1`、`| 4`、`& ~8` 模式表明该字段是标志位集合
- **循环还原的关键**是找到计数器变量、边界条件和循环体操作——反编译的 `while` 可以还原为更清晰的 `for`
- **LZ 类压缩算法**的还原需要识别操作码格式（字面量 vs 回溯引用）、偏移和长度的编码方式
- **往返测试**（Round-trip Test）是验证还原算法正确性的黄金标准——压缩→解压→与原始数据比较
- **Ghidra 的自定义类型**可以将偏移访问转换为字段名访问，大幅提升反编译代码的可读性

```cpp
    int local_14 = (local_8 > local_c) ? local_8 : local_c;
    if (local_10 > local_14) local_14 = local_10;

    int local_18 = (local_8 < local_c) ? local_8 : local_c;
    if (local_10 < local_18) local_18 = local_10;

    int local_1c;
    if (local_14 == 0) {
        local_1c = 0;
    } else {
        local_1c = ((local_14 - local_18) * 255) / local_14;
    }

    return local_10 | (local_c << 8) |
           (local_1c << 16) | (local_14 << 24);
}
```

<details>
<summary>查看答案</summary>

这个函数将 BGRA 颜色转换为 HSV 色彩空间中的**饱和度（Saturation）**和**明度（Value）**，但跳过了色相（Hue）计算。具体来说，它计算 RGB 的最大值（明度 V）和饱和度 S，然后把结果打包回一个 32 位整数。

```cpp
#include <cstdint>
#include <cstdio>
#include <algorithm>

// 从 BGRA 像素计算饱和度和明度
// 输入: BGRA 格式的 32 位像素
// 输出: B | (G << 8) | (S << 16) | (V << 24)
//   其中 S = 饱和度 (0-255), V = 明度 (0-255)
uint32_t calc_saturation_value(uint32_t bgra_pixel) {
    // 提取分量（注意: KiriKiri 是 BGRA 格式）
    uint32_t alpha = (bgra_pixel >> 24) & 0xFF;  // 未使用
    uint32_t red   = (bgra_pixel >> 16) & 0xFF;
    uint32_t green = (bgra_pixel >>  8) & 0xFF;
    uint32_t blue  = (bgra_pixel >>  0) & 0xFF;

    // 计算 RGB 最大值（Value / 明度 V）
    uint32_t max_rgb = std::max({red, green, blue});

    // 计算 RGB 最小值
    uint32_t min_rgb = std::min({red, green, blue});

    // 计算饱和度 S = (max - min) / max * 255
    uint32_t saturation = 0;
    if (max_rgb > 0) {
        saturation = ((max_rgb - min_rgb) * 255) / max_rgb;
    }

    // 打包结果
    return blue | (green << 8) |
           (saturation << 16) | (max_rgb << 24);
}

int main() {
    // 测试: 纯红色 BGRA = 0xFF_FF_00_00
    uint32_t red = 0xFFFF0000;
    uint32_t result = calc_saturation_value(red);
    printf("纯红色: V=%d, S=%d\n",
           (result >> 24) & 0xFF,   // V = 255
           (result >> 16) & 0xFF);  // S = 255

    // 测试: 灰色 BGRA = 0xFF_80_80_80
    uint32_t gray = 0xFF808080;
    result = calc_saturation_value(gray);
    printf("灰色:   V=%d, S=%d\n",
           (result >> 24) & 0xFF,   // V = 128
           (result >> 16) & 0xFF);  // S = 0
    return 0;
}
```

**分析过程：**
1. `local_4` = alpha（最高字节），函数中未使用
2. `local_14` = R/G/B 三者最大值 → HSV 的 Value
3. `local_18` = R/G/B 三者最小值
4. `local_1c = (max - min) * 255 / max` → HSV 的 Saturation
5. 返回值重新打包，用 S 和 V 替换原来的 alpha 和 red 位

### 题目 3：修复解压算法中的边界错误

以下解压函数包含一个 Bug，会在特定输入下导致缓冲区越界访问。请找出 Bug 并修复。

```cpp
size_t buggy_decompress(
    uint8_t* dst, const uint8_t* src, size_t src_len
) {
    uint8_t* out = dst;
    const uint8_t* in = src;
    const uint8_t* in_end = src + src_len;

    while (in < in_end) {
        uint8_t opcode = *in++;

        if ((opcode & 0x80) == 0) {
            int count = (opcode & 0x7F) + 1;
            // BUG 在这里的某个地方...
            for (int i = 0; i < count; ++i) {
                *out++ = *in++;
            }
        } else {
            uint8_t byte2 = *in++;
            int offset = ((opcode & 0x1F) << 8) | byte2;
            int length = ((opcode >> 5) & 0x03) + 3;
            uint8_t* match = out - offset - 1;
            for (int i = 0; i < length; ++i) {
                *out++ = match[i];
            }
        }
    }
    return (size_t)(out - dst);
}
```

<details>
<summary>查看答案</summary>

**Bug：** 字面量复制分支中没有检查 `in + count` 是否超出 `in_end`。如果压缩数据被截断或损坏，`count` 可能大于剩余输入，导致读取越界。同样，回溯引用分支中没有检查 `match` 是否有效。

**修复版本：**

```cpp
#include <cstdint>
#include <cstddef>
#include <cstdio>
#include <cstring>
#include <stdexcept>

size_t fixed_decompress(
    uint8_t* dst,
    size_t dst_capacity,        // 新增: 输出缓冲区容量
    const uint8_t* src,
    size_t src_len
) {
    uint8_t* out = dst;
    uint8_t* out_end = dst + dst_capacity;  // 输出边界
    const uint8_t* in = src;
    const uint8_t* in_end = src + src_len;

    while (in < in_end) {
        uint8_t opcode = *in++;

        if ((opcode & 0x80) == 0) {
            int count = (opcode & 0x7F) + 1;

            // 修复 1: 检查输入边界
            if (in + count > in_end) {
                throw std::runtime_error(
                    "字面量数据超出输入缓冲区边界");
            }
            // 修复 2: 检查输出边界
            if (out + count > out_end) {
                throw std::runtime_error(
                    "字面量写入超出输出缓冲区容量");
            }
            memcpy(out, in, count);
            out += count;
            in += count;
        } else {
            // 修复 3: 检查第二字节是否可读
            if (in >= in_end) {
                throw std::runtime_error(
                    "引用操作码缺少第二字节");
            }
            uint8_t byte2 = *in++;
            int offset = ((opcode & 0x1F) << 8) | byte2;
            int length = ((opcode >> 5) & 0x03) + 3;

            // 修复 4: 检查回溯偏移是否有效
            uint8_t* match = out - offset - 1;
            if (match < dst) {
                throw std::runtime_error(
                    "回溯偏移超出已输出数据范围");
            }
            // 修复 5: 检查输出边界
            if (out + length > out_end) {
                throw std::runtime_error(
                    "引用写入超出输出缓冲区容量");
            }
            for (int i = 0; i < length; ++i) {
                *out++ = match[i];
            }
        }
    }
    return (size_t)(out - dst);
}

int main() {
    // 测试正常数据
    uint8_t compressed[] = {
        0x04, 'H', 'e', 'l', 'l', 'o',  // 字面量 5 字节
    };
    uint8_t output[256];
    size_t len = fixed_decompress(
        output, sizeof(output),
        compressed, sizeof(compressed));
    output[len] = '\0';
    printf("解压结果: %s (长度 %zu) ✅\n", (char*)output, len);

    // 测试截断数据
    uint8_t truncated[] = {
        0x09, 'H', 'e', 'l',  // 声称 10 字节，实际只有 3
    };
    try {
        fixed_decompress(output, sizeof(output),
                         truncated, sizeof(truncated));
        printf("❌ 应该抛出异常但没有\n");
    } catch (const std::runtime_error& e) {
        printf("截断检测: %s ✅\n", e.what());
    }
    return 0;
}
```

**五处修复总结：**
1. 字面量分支: 检查 `in + count <= in_end`（输入边界）
2. 字面量分支: 检查 `out + count <= out_end`（输出边界）
3. 引用分支: 检查第二字节可读 `in < in_end`
4. 引用分支: 检查回溯源有效 `match >= dst`
5. 引用分支: 检查输出不溢出 `out + length <= out_end`

**教训：** 逆向还原的代码必须添加原版可能没有的边界检查。原版 DLL 可能依赖"数据总是正确的"这一假设，但跨平台移植版必须防御性编程——任何来自文件的数据都不可信。

</details>
