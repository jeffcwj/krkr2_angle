## 术语预览

本节将涉及以下术语，先有个印象，正文中会逐一详细讲解：

| 术语 | 英文 | 一句话解释 |

| 回归测试 | Regression Test | 每次修改代码后重新运行所有测试，确保修改没有破坏已有功能 |
| 测试夹具 | Test Fixture | 测试运行前准备好的环境和数据——如预先生成的压缩文件、已知正确的输出等 |
| 断言 | Assertion | 代码中的检查点——"这个条件必须为真，否则说明有 Bug" |

## 一、为什么逆向代码需要特殊的验证策略

### 1.1 逆向代码的特殊风险

普通开发中，代码的"正确行为"由需求文档定义。但逆向还原的代码面临独特的挑战——它的"正确行为"由原始二进制文件定义，而我们无法阅读原始源代码。这意味着：

1. **没有规格说明（Specification）：** 我们不知道原始开发者的设计意图，只能观察行为
2. **反编译器引入的噪音：** Ghidra 等反编译器的输出不是 100% 准确——类型推断可能错误，优化代码可能被错误地还原
3. **平台差异放大错误：** 原版是 32 位 Windows DLL，而我们的目标是跨平台代码。整数溢出、字节序（Endianness——CPU 存储多字节数据时高位在前还是低位在前的规则）、对齐差异都可能导致细微的行为不一致
4. **边界行为不可知：** 原版在畸形输入下可能崩溃（或产生随机输出），但我们的还原版必须安全地处理这些情况

### 1.2 验证策略的四层防线

为了应对这些风险，我们采用四层递进的验证策略：

```
验证策略四层防线：

第 1 层: 往返测试（Round-trip Test）
  ├─ 数据 → 压缩 → 解压 → 比较原始数据
  └─ 验证: 编码/解码的对称性

第 2 层: 差分测试（Differential Testing）
  ├─ 相同输入 → 原版 DLL / 还原代码 → 比较输出
  └─ 验证: 与原版行为的一致性

第 3 层: 边界条件测试（Boundary Testing）
  ├─ 空数据、最大数据、畸形数据、随机数据
  └─ 验证: 鲁棒性和安全性

第 4 层: 模糊测试（Fuzz Testing）
  ├─ 自动生成海量随机输入
  └─ 验证: 无崩溃、无内存越界、无未定义行为
```

## 二、往返测试（Round-trip Test）

### 2.1 基本原理

往返测试是最简单也最有效的验证方法。如果你还原了一个压缩算法的解压函数，那么：

```
原始数据 → compress() → 压缩数据 → decompress() → 还原数据
                                                      ↓
                                              比较: 还原数据 == 原始数据 ?
```

如果两端数据完全一致，说明 compress 和 decompress 至少在这组输入上是互逆的。当然，往返测试无法保证与原版的兼容性——你的 compress 可能产生与原版不同的压缩数据（但解压结果相同）。这就是为什么需要配合差分测试。

### 2.2 完整的往返测试框架

以下是一个使用 Catch2（KrKr2 项目使用的测试框架——基于 C++ 的单头文件测试库，支持 BDD 风格和断言宏）的往返测试模板：

```cpp
// test_roundtrip.cpp
// 使用 Catch2 v3 编写的往返测试

#define CATCH_CONFIG_MAIN
#include <catch2/catch_test_macros.hpp>
#include <catch2/generators/catch_generators.hpp>
#include <cstdint>
#include <cstring>
#include <vector>
#include <random>

// 假设这些函数已在其他文件中实现
// （来自上一节的 pkn_decompress 和 pkn_compress）
extern size_t pkn_decompress(
    uint8_t* dst, const uint8_t* src, size_t src_len);
extern std::vector<uint8_t> pkn_compress(
    const uint8_t* data, size_t data_len);

// 辅助函数: 生成指定大小的随机数据
std::vector<uint8_t> random_data(size_t size, uint32_t seed = 42) {
    std::mt19937 rng(seed);  // 固定种子确保可重复
    std::uniform_int_distribution<int> dist(0, 255);
    std::vector<uint8_t> data(size);
    for (auto& b : data) {
        b = static_cast<uint8_t>(dist(rng));
    }
    return data;
}

// 辅助函数: 生成包含重复模式的数据（压缩率更高）
std::vector<uint8_t> pattern_data(size_t size) {
    std::vector<uint8_t> data(size);
    const char* pattern = "KiriKiri2 Plugin Test Data ";
    size_t plen = strlen(pattern);
    for (size_t i = 0; i < size; ++i) {
        data[i] = static_cast<uint8_t>(pattern[i % plen]);
    }
    return data;
}

TEST_CASE("往返测试: 空数据", "[roundtrip]") {
    std::vector<uint8_t> empty;
    auto compressed = pkn_compress(empty.data(), 0);
    // 空数据压缩后也应该是空的（或极小的头部）
    REQUIRE(compressed.size() <= 2);  // 允许最多 2 字节头部

    std::vector<uint8_t> output(1);  // 至少分配 1 字节防止空指针
    size_t len = pkn_decompress(
        output.data(), compressed.data(), compressed.size());
    REQUIRE(len == 0);  // 解压后应该是 0 字节
}

TEST_CASE("往返测试: 单字节", "[roundtrip]") {
    uint8_t data[] = {0x42};
    auto compressed = pkn_compress(data, 1);
    REQUIRE(!compressed.empty());

    std::vector<uint8_t> output(16);
    size_t len = pkn_decompress(
        output.data(), compressed.data(), compressed.size());
    REQUIRE(len == 1);
    REQUIRE(output[0] == 0x42);
}

TEST_CASE("往返测试: 全零数据（高压缩率）", "[roundtrip]") {
    std::vector<uint8_t> data(1024, 0x00);  // 1KB 全零
    auto compressed = pkn_compress(data.data(), data.size());

    // 全零数据应该压缩得很好
    INFO("压缩率: " << (100.0 * compressed.size() / data.size()) << "%");
    REQUIRE(compressed.size() < data.size());

    std::vector<uint8_t> output(data.size() * 2);
    size_t len = pkn_decompress(
        output.data(), compressed.data(), compressed.size());
    REQUIRE(len == data.size());
    REQUIRE(memcmp(output.data(), data.data(), len) == 0);
}

TEST_CASE("往返测试: 随机数据（低压缩率）", "[roundtrip]") {
    auto data = random_data(4096, 12345);  // 4KB 随机数据
    auto compressed = pkn_compress(data.data(), data.size());

    // 随机数据通常不可压缩，可能比原始还大
    INFO("压缩率: " << (100.0 * compressed.size() / data.size()) << "%");

    std::vector<uint8_t> output(data.size() * 2);
    size_t len = pkn_decompress(
        output.data(), compressed.data(), compressed.size());
    REQUIRE(len == data.size());
    REQUIRE(memcmp(output.data(), data.data(), len) == 0);
}

TEST_CASE("往返测试: 重复模式数据", "[roundtrip]") {
    auto data = pattern_data(8192);  // 8KB 重复模式
    auto compressed = pkn_compress(data.data(), data.size());

    INFO("压缩率: " << (100.0 * compressed.size() / data.size()) << "%");
    REQUIRE(compressed.size() < data.size());  // 重复模式应可压缩

    std::vector<uint8_t> output(data.size() * 2);
    size_t len = pkn_decompress(
        output.data(), compressed.data(), compressed.size());
    REQUIRE(len == data.size());
    REQUIRE(memcmp(output.data(), data.data(), len) == 0);
}

TEST_CASE("往返测试: 多种数据大小", "[roundtrip]") {
    // 测试从 1 字节到 64KB 的多种大小
    auto size = GENERATE(1, 2, 3, 127, 128, 129, 255, 256,
                         1023, 1024, 4096, 8191, 8192, 65535);

    auto data = random_data(size, static_cast<uint32_t>(size));
    auto compressed = pkn_compress(data.data(), data.size());

    std::vector<uint8_t> output(data.size() * 2);
    size_t len = pkn_decompress(
        output.data(), compressed.data(), compressed.size());

    INFO("数据大小: " << size << " 字节");
    REQUIRE(len == data.size());
    REQUIRE(memcmp(output.data(), data.data(), len) == 0);
}
```

## 三、差分测试（Differential Testing）

### 3.1 原理与适用场景

差分测试的核心思想：用**相同的输入**分别喂给**两个独立的实现**，然后比较它们的输出是否一致。在插件逆向的场景中，这两个实现分别是：

- **参考实现（Reference）：** 原始的 Windows DLL，通过 Wine 或 Windows 虚拟机运行
- **被测实现（Test）：** 我们逆向还原的跨平台 C++ 代码

```
差分测试流程：

输入数据 ──┬──→ 原版 DLL (Wine/VM) ──→ 输出 A
           │
           └──→ 还原代码 (本地)    ──→ 输出 B
                                          │
                                    比较: A == B ?
```

### 3.2 搭建差分测试环境

**方案 A：使用 Wine 加载原版 DLL（Linux/macOS）**

Wine 是一个在非 Windows 系统上运行 Windows 程序的兼容层。我们可以编写一个小型 Windows 程序来调用原版 DLL，然后通过 Wine 运行它：

```cpp
// diff_test_loader.cpp
// 这个程序在 Windows 或 Wine 下运行
// 加载原版 DLL，调用指定函数，输出结果到文件

#include <cstdio>
#include <cstdint>
#include <cstring>
#include <vector>

#ifdef _WIN32
#include <windows.h>
#else
// 在非 Windows 环境下模拟 (仅用于展示结构)
#error "此文件需要在 Windows 或 Wine 下编译"
#endif

// 原版 DLL 中解压函数的类型定义
// 从反编译中推断的函数签名
typedef int (*DecompressFunc)(
    char* dst, char* src, int src_len);

int main(int argc, char* argv[]) {
    if (argc < 4) {
        fprintf(stderr,
            "用法: diff_test_loader <dll路径> "
            "<输入文件> <输出文件>\n");
        return 1;
    }

    const char* dll_path = argv[1];
    const char* input_path = argv[2];
    const char* output_path = argv[3];

    // 加载 DLL
    HMODULE dll = LoadLibraryA(dll_path);
    if (!dll) {
        fprintf(stderr, "无法加载 DLL: %s (错误: %lu)\n",
                dll_path, GetLastError());
        return 1;
    }

    // 获取函数地址
    // 注意: 实际函数名需要从 DLL 导出表中获取
    DecompressFunc decompress =
        (DecompressFunc)GetProcAddress(dll, "Decompress");
    if (!decompress) {
        fprintf(stderr, "未找到 Decompress 导出函数\n");
        FreeLibrary(dll);
        return 1;
    }

    // 读取输入文件
    FILE* fin = fopen(input_path, "rb");
    if (!fin) {
        fprintf(stderr, "无法打开输入文件: %s\n", input_path);
        FreeLibrary(dll);
        return 1;
    }
    fseek(fin, 0, SEEK_END);
    long input_size = ftell(fin);
    fseek(fin, 0, SEEK_SET);
    std::vector<char> input(input_size);
    fread(input.data(), 1, input_size, fin);
    fclose(fin);

    // 调用原版解压函数
    // 预分配足够大的输出缓冲区（原始大小通常在文件头中）
    std::vector<char> output(input_size * 10);
    int result_len = decompress(
        output.data(), input.data(), input_size);

    if (result_len < 0) {
        fprintf(stderr, "解压失败 (返回值: %d)\n", result_len);
        FreeLibrary(dll);
        return 1;
    }

    // 写入输出文件
    FILE* fout = fopen(output_path, "wb");
    fwrite(output.data(), 1, result_len, fout);
    fclose(fout);

    printf("解压完成: %ld → %d 字节\n", input_size, result_len);
    FreeLibrary(dll);
    return 0;
}
```

**方案 B：使用测试夹具文件（推荐，无需 Wine）**

更实用的方法是提前生成一批"已知正确"的输入/输出对，保存为测试夹具（Test Fixture）文件：

```cpp
// generate_fixtures.cpp
// 在 Windows 上运行，生成差分测试夹具文件
// 输出: test_case_NNN.input + test_case_NNN.expected

#include <cstdio>
#include <cstdint>
#include <vector>
#include <random>

// 假设原版解压函数已链接
extern "C" int original_decompress(
    char* dst, char* src, int src_len);

void generate_fixture(int id, const uint8_t* data, size_t len) {
    char input_path[256], expected_path[256];
    snprintf(input_path, sizeof(input_path),
             "fixtures/test_case_%03d.input", id);
    snprintf(expected_path, sizeof(expected_path),
             "fixtures/test_case_%03d.expected", id);

    // 保存输入
    FILE* fin = fopen(input_path, "wb");
    fwrite(data, 1, len, fin);
    fclose(fin);

    // 用原版函数处理
    std::vector<char> output(len * 10);
    int result_len = original_decompress(
        output.data(), (char*)data, (int)len);

    // 保存期望输出
    FILE* fout = fopen(expected_path, "wb");
    if (result_len > 0) {
        fwrite(output.data(), 1, result_len, fout);
    }
    fclose(fout);

    printf("夹具 %03d: 输入 %zu 字节 → 输出 %d 字节\n",
           id, len, result_len);
}

int main() {
    // 从游戏的实际归档文件中提取测试数据
    // 这里用随机数据代替
    std::mt19937 rng(42);
    for (int i = 0; i < 100; ++i) {
        size_t len = (rng() % 10000) + 1;
        std::vector<uint8_t> data(len);
        for (auto& b : data) b = rng() & 0xFF;
        generate_fixture(i, data.data(), len);
    }
    return 0;
}
```

### 3.3 差分测试的自动化脚本

```bash
#!/bin/bash
# diff_test.sh — 差分测试自动化脚本
# 比较原版 DLL（通过 Wine）和还原代码的输出

FIXTURE_DIR="tests/test_files/pkn_fixtures"
REFERENCE_TOOL="wine diff_test_loader.exe"
TEST_TOOL="./build/pkn_decompress_test"
DIFF_FAILURES=0
TOTAL=0

echo "=== PackinOne 差分测试 ==="

for input_file in "$FIXTURE_DIR"/*.input; do
    TOTAL=$((TOTAL + 1))
    base=$(basename "$input_file" .input)
    expected="$FIXTURE_DIR/${base}.expected"
    actual="/tmp/${base}.actual"

    # 用还原代码处理
    $TEST_TOOL "$input_file" "$actual"

    # 比较输出
    if cmp -s "$expected" "$actual"; then
        echo "  ✅ $base"
    else
        echo "  ❌ $base — 输出不一致!"
        # 显示差异的前几个字节
        xxd "$expected" | head -5 > /tmp/exp.hex
        xxd "$actual" | head -5 > /tmp/act.hex
        diff /tmp/exp.hex /tmp/act.hex
        DIFF_FAILURES=$((DIFF_FAILURES + 1))
    fi
done

echo ""
echo "结果: $((TOTAL - DIFF_FAILURES))/$TOTAL 通过"
if [ $DIFF_FAILURES -gt 0 ]; then
    echo "❌ 有 $DIFF_FAILURES 个测试失败"
    exit 1
else
    echo "✅ 全部通过"
fi
```

}

## 四、边界条件测试

### 4.1 为什么逆向代码特别需要边界测试

普通代码的边界条件通常由 API 文档明确规定——"输入长度不得超过 MAX_PATH"。但逆向还原的代码没有任何文档，我们只能通过测试来探索边界行为。更棘手的是：

1. **原版可能没有边界检查：** 原始 DLL 开发者可能假设调用方永远传入合法数据，从不处理异常输入。但我们的跨平台还原版必须防御性编程，因为不同平台下同一个越界访问可能表现完全不同（Windows 下可能凑巧能跑，Linux 下直接 SIGSEGV 崩溃）
2. **整数溢出行为不可移植：** 原版 32 位 MSVC 编译的代码中，有符号整数溢出的行为可能"恰好正确"，但这在 C++ 标准中是未定义行为（Undefined Behavior，简称 UB——编译器可以做任何事，包括优化掉你的检查代码）。跨平台编译器（GCC、Clang）的优化策略不同，同样的溢出可能产生不同结果
3. **缓冲区边界差异：** Windows 的 `HeapAlloc` 通常会多分配几个字节的对齐填充，掩盖了轻微的越界写入。但 Linux 的 `malloc` 在启用 AddressSanitizer（一种内存错误检测工具，能在越界访问发生时立即报错）后会立刻捕获这些问题

### 4.2 边界测试用例设计矩阵

针对逆向还原的压缩/解压算法，我们需要覆盖以下边界：

```
边界测试矩阵（适用于 PackinOne/LZ 类压缩算法）：

┌──────────────────┬───────────────────────────────┬────────────────────┐
│ 边界类别         │ 测试输入                      │ 期望行为           │
├──────────────────┼───────────────────────────────┼────────────────────┤
│ 空输入           │ 长度 = 0 的缓冲区             │ 返回空或错误码     │
│ 单字节           │ 长度 = 1                      │ 正确处理           │
│ 最小压缩单元     │ 长度 = 算法最小块大小         │ 正确压缩/解压      │
│ 全零数据         │ 全部字节为 0x00               │ 高压缩比，正确还原 │
│ 全 0xFF 数据     │ 全部字节为 0xFF               │ 正确还原           │
│ 重复模式         │ 单字节/短串重复百万次         │ LZ 回引高效压缩    │
│ 完全随机         │ 每字节随机（不可压缩）        │ 压缩后可能更大     │
│ 恰好 64KB        │ 常见块大小边界                │ 无溢出             │
│ 64KB + 1         │ 跨块边界                      │ 分块正确           │
│ 接近 SIZE_MAX    │ 极大长度（仅检查不崩溃）      │ 拒绝或安全处理     │
│ 截断的压缩数据   │ 合法压缩流截掉最后 N 字节     │ 返回错误，不崩溃   │
│ 损坏的头部       │ 魔数/版本号被篡改             │ 检测到无效数据     │
│ 回引超出范围     │ LZ 偏移指向缓冲区之前        │ 拒绝，不越界读取   │
└──────────────────┴───────────────────────────────┴────────────────────┘
```

### 4.3 完整的边界条件测试代码

以下是针对逆向还原的 PackinOne 解压算法的 Catch2 边界测试。注意每个 `SECTION` 独立运行，互不影响：

```cpp
// tests/unit-tests/plugins/test_pkn_boundary.cpp
// PackinOne 解压算法的边界条件测试
#define CATCH_CONFIG_MAIN
#include <catch2/catch_test_macros.hpp>
#include <catch2/catch_approx.hpp>
#include <cstdint>
#include <cstring>
#include <vector>
#include <random>
#include <algorithm>
#include <numeric>

// 引入还原的 PackinOne 接口
// pkn_compress: 压缩数据，返回压缩后的缓冲区
// pkn_decompress: 解压数据，返回解压后的缓冲区
// 两个函数在失败时返回空 vector
#include "pkn_reversed.h"

// ============================================================
// 辅助函数
// ============================================================

// 生成指定大小的随机数据
static std::vector<uint8_t> make_random(size_t size, uint32_t seed = 42) {
    std::mt19937 rng(seed);                          // 固定种子保证可重现
    std::uniform_int_distribution<int> dist(0, 255);
    std::vector<uint8_t> data(size);
    for (auto& b : data) {
        b = static_cast<uint8_t>(dist(rng));
    }
    return data;
}

// 生成重复模式数据：pattern 重复 count 次
static std::vector<uint8_t> make_pattern(
    const std::vector<uint8_t>& pattern, size_t count
) {
    std::vector<uint8_t> data;
    data.reserve(pattern.size() * count);
    for (size_t i = 0; i < count; ++i) {
        data.insert(data.end(), pattern.begin(), pattern.end());
    }
    return data;
}

// 验证往返：压缩后解压，结果与原始一致
static bool verify_roundtrip(const std::vector<uint8_t>& original) {
    auto compressed = pkn_compress(original.data(), original.size());
    if (compressed.empty() && !original.empty()) return false;

    auto decompressed = pkn_decompress(compressed.data(), compressed.size());
    return decompressed == original;
}

// ============================================================
// 测试用例
// ============================================================

TEST_CASE("PackinOne 边界条件 — 空和最小输入", "[pkn][boundary]") {

    SECTION("空输入 — 长度为 0") {
        std::vector<uint8_t> empty;
        // 空输入压缩应该返回空或极小的头部
        auto compressed = pkn_compress(empty.data(), 0);
        // 无论压缩结果如何，解压回来必须是空的
        if (!compressed.empty()) {
            auto decompressed = pkn_decompress(
                compressed.data(), compressed.size()
            );
            REQUIRE(decompressed.empty());
        }
    }

    SECTION("单字节输入") {
        std::vector<uint8_t> single = {0x42};      // 任意单字节
        REQUIRE(verify_roundtrip(single));
    }

    SECTION("两字节输入 — 最小可能的回引") {
        std::vector<uint8_t> two = {0xAA, 0xBB};
        REQUIRE(verify_roundtrip(two));
    }

    SECTION("三字节输入 — LZ 最小匹配长度边界") {
        // 许多 LZ 变体的最小匹配长度是 3
        std::vector<uint8_t> three = {0x01, 0x02, 0x03};
        REQUIRE(verify_roundtrip(three));
    }
}

TEST_CASE("PackinOne 边界条件 — 特殊数据模式", "[pkn][boundary]") {

    SECTION("全零数据 — 最高压缩比场景") {
        // 全零数据对 LZ 算法来说是最佳输入，压缩比极高
        std::vector<uint8_t> zeros(65536, 0x00);    // 64KB 全零
        REQUIRE(verify_roundtrip(zeros));

        // 额外验证：压缩后的大小应远小于原始大小
        auto compressed = pkn_compress(zeros.data(), zeros.size());
        REQUIRE(compressed.size() < zeros.size() / 10);  // 至少 10:1 压缩比
    }

    SECTION("全 0xFF 数据") {
        std::vector<uint8_t> ones(32768, 0xFF);     // 32KB 全 0xFF
        REQUIRE(verify_roundtrip(ones));
    }

    SECTION("递增序列 — 每字节不同") {
        // 256 字节递增序列，无重复模式，LZ 难以压缩
        std::vector<uint8_t> seq(256);
        std::iota(seq.begin(), seq.end(), 0);        // 0, 1, 2, ..., 255
        REQUIRE(verify_roundtrip(seq));
    }

    SECTION("短模式大量重复 — LZ 回引极端场景") {
        // "ABC" 重复 100000 次 = 300KB
        // 这会产生大量 LZ 回引，测试回引编码的正确性
        auto data = make_pattern({0x41, 0x42, 0x43}, 100000);
        REQUIRE(data.size() == 300000);
        REQUIRE(verify_roundtrip(data));
    }

    SECTION("完全随机数据 — 不可压缩") {
        // 随机数据无法被有效压缩，压缩后大小可能更大
        auto data = make_random(65536);
        REQUIRE(verify_roundtrip(data));

        // 压缩后大小不应小于原始大小的 95%（随机数据几乎不可压缩）
        auto compressed = pkn_compress(data.data(), data.size());
        REQUIRE(compressed.size() >= data.size() * 95 / 100);
    }
}

TEST_CASE("PackinOne 边界条件 — 块大小边界", "[pkn][boundary]") {

    // PackinOne 可能使用 64KB 作为块大小
    constexpr size_t BLOCK_SIZE = 65536;

    SECTION("恰好一个块") {
        auto data = make_random(BLOCK_SIZE, 100);
        REQUIRE(verify_roundtrip(data));
    }

    SECTION("一个块 + 1 字节") {
        auto data = make_random(BLOCK_SIZE + 1, 200);
        REQUIRE(verify_roundtrip(data));
    }

    SECTION("一个块 - 1 字节") {
        auto data = make_random(BLOCK_SIZE - 1, 300);
        REQUIRE(verify_roundtrip(data));
    }

    SECTION("恰好两个块") {
        auto data = make_random(BLOCK_SIZE * 2, 400);
        REQUIRE(verify_roundtrip(data));
    }

    SECTION("两个块 + 1 字节 — 跨块边界") {
        auto data = make_random(BLOCK_SIZE * 2 + 1, 500);
        REQUIRE(verify_roundtrip(data));
    }

    SECTION("大数据 — 1MB") {
        auto data = make_random(1024 * 1024, 600);
        REQUIRE(verify_roundtrip(data));
    }
}

TEST_CASE("PackinOne 边界条件 — 损坏和截断数据", "[pkn][boundary]") {

    // 先生成一份合法的压缩数据
    auto original = make_random(10000, 777);
    auto compressed = pkn_compress(original.data(), original.size());
    REQUIRE(!compressed.empty());

    SECTION("截断 — 只保留头部前 4 字节") {
        std::vector<uint8_t> truncated(
            compressed.begin(), compressed.begin() + 4
        );
        // 解压必须返回空（检测到不完整的数据），不能崩溃
        auto result = pkn_decompress(truncated.data(), truncated.size());
        REQUIRE(result.empty());
    }
}

### 实践 1：为还原的解压函数编写完整测试套件

**目标：** 综合运用往返测试、边界测试和差分测试，为一个简单的 RLE（Run-Length Encoding，游程编码——将连续重复的字节压缩为"重复次数+字节值"的编码方式）压缩算法编写完整的测试套件。

**步骤：**

1. 实现一个简单的 RLE 编码/解码器：

```cpp
// rle.h — 简单的 RLE 编码器/解码器
#pragma once
#include <cstdint>
#include <vector>

// RLE 编码格式：
//   [count][byte] [count][byte] ...
//   count: 1-255，表示后面的 byte 重复 count 次
//   当遇到不重复的字节序列时，count=1

// 编码：将原始数据压缩为 RLE 格式
inline std::vector<uint8_t> rle_encode(
    const uint8_t* data, size_t size
) {
    std::vector<uint8_t> output;
    if (size == 0) return output;

    size_t i = 0;
    while (i < size) {
        uint8_t current = data[i];
        uint8_t count = 1;

        // 计算连续重复字节的数量（最多 255）
        while (i + count < size
               && data[i + count] == current
               && count < 255) {
            ++count;
        }

        output.push_back(count);    // 重复次数
        output.push_back(current);  // 字节值
        i += count;
    }
    return output;
}

// 解码：将 RLE 格式还原为原始数据
inline std::vector<uint8_t> rle_decode(
    const uint8_t* data, size_t size
) {
    std::vector<uint8_t> output;
    if (size == 0) return output;

    // RLE 数据必须是偶数长度（count+byte 成对出现）
    if (size % 2 != 0) return {};  // 格式错误

    for (size_t i = 0; i < size; i += 2) {
        uint8_t count = data[i];
        uint8_t byte = data[i + 1];

        if (count == 0) return {};  // count=0 无意义，视为格式错误

        // 防止解压炸弹（Decompression Bomb）：
        // 单次展开不超过 16MB
        if (output.size() + count > 16 * 1024 * 1024) {
            return {};  // 拒绝过大的输出
        }

        for (uint8_t j = 0; j < count; ++j) {
            output.push_back(byte);
        }
    }
    return output;
}
```

2. 为这个 RLE 编码器编写 Catch2 测试：

```cpp
// test_rle.cpp — RLE 编码器完整测试套件

#include <catch2/catch_test_macros.hpp>
#include <cstdint>
#include <vector>
#include <random>
#include <numeric>
#include "rle.h"

// 辅助：验证往返一致性
static bool verify_rle_roundtrip(const std::vector<uint8_t>& data) {
    auto encoded = rle_encode(data.data(), data.size());
    auto decoded = rle_decode(encoded.data(), encoded.size());
    return decoded == data;
}

TEST_CASE("RLE 往返测试", "[rle][roundtrip]") {
    SECTION("空数据") {

    }
    SECTION("单字节") {

        REQUIRE(verify_rle_roundtrip(same));
    }
    SECTION("全不同字节") {
        std::vector<uint8_t> diff(256);
        std::iota(diff.begin(), diff.end(), 0);
        REQUIRE(verify_rle_roundtrip(diff));
    }
    SECTION("超过 255 的连续重复") {
        // 300 个相同字节，需要分成 255+45 两段
        std::vector<uint8_t> long_run(300, 0xFF);
        REQUIRE(verify_rle_roundtrip(long_run));
    }
}

TEST_CASE("RLE 边界测试", "[rle][boundary]") {
    SECTION("解码奇数长度数据 — 格式错误") {
        std::vector<uint8_t> odd = {0x03, 0x41, 0x02};
        auto result = rle_decode(odd.data(), odd.size());
        REQUIRE(result.empty());
    }
    SECTION("解码 count=0 — 格式错误") {
        std::vector<uint8_t> zero_count = {0x00, 0x41};
        auto result = rle_decode(zero_count.data(), zero_count.size());

    }
    SECTION("解压炸弹防御") {

        for (int i = 0; i < 70000; ++i) {

            bomb.push_back(0x00);  // byte=0
        }
        // 255 * 70000 = 17,850,000 > 16MB

        REQUIRE(result.empty());  // 应该拒绝
    }
}

### 实践 2：用 libFuzzer 模糊测试 RLE 解码器

**目标：** 为上面的 RLE 解码器编写 Fuzz Target，运行 5 分钟，看能否发现 Bug。

```cpp
// fuzz_rle.cpp — RLE 解码器的 Fuzz Target
#include <cstdint>
#include <cstddef>
#include "rle.h"

extern "C" int LLVMFuzzerTestOneInput(
    const uint8_t* data,
    size_t size
) {
    // 测试 1：随机数据直接解码（不应崩溃）
    auto decoded = rle_decode(data, size);

    // 测试 2：如果解码成功，重新编码后解码应一致
    if (!decoded.empty()) {
        auto re_encoded = rle_encode(decoded.data(), decoded.size());
        auto re_decoded = rle_decode(
            re_encoded.data(), re_encoded.size()
        );
        if (re_decoded != decoded) {
            __builtin_trap();  // 往返不一致 = Bug
        }
    }

    // 测试 3：随机数据先编码再解码（往返测试）
    auto encoded = rle_encode(data, size);
    if (!encoded.empty()) {
        auto roundtrip = rle_decode(
            encoded.data(), encoded.size()
        );
        // 编码后解码必须与原始一致
        if (roundtrip.size() != size) {
            __builtin_trap();
        }
        for (size_t i = 0; i < size; ++i) {
            if (roundtrip[i] != data[i]) {
                __builtin_trap();
            }
        }
    }

    return 0;
}
```

## 常见错误与排查

### 错误 1：差分测试在 Wine 下加载 DLL 失败

**症状：** 在 Linux 上通过 Wine 加载原版 DLL 时，出现 `0x0000007e` 错误（模块未找到）。

**原因：** 原版 DLL 依赖其他 Windows 系统 DLL（如 `msvcrt.dll`、`kernel32.dll`），而 Wine 环境中缺少对应版本。

**解决方案：**

```bash
# 1. 检查 DLL 依赖
wine winedump -j import PackinOne.dll

# 2. 安装缺失的运行时
winetricks vcrun2010     # Visual C++ 2010 运行时
winetricks vcrun2015     # 或 2015 版本

# 3. 如果是 32 位 DLL，确保使用 32 位 Wine 前缀
WINEARCH=win32 WINEPREFIX=~/.wine32 wine wineboot
WINEPREFIX=~/.wine32 wine ./diff_test.exe
```

### 错误 2：ASan 报告 "stack-buffer-overflow" 但代码看起来正确

**症状：** 往返测试中，压缩函数在栈上分配了一个固定大小的临时缓冲区，ASan 报告越界写入。

**原因：** 逆向还原时，你可能把原版的 `alloca(n)` 还原为 `uint8_t buf[n]`，但 `n` 来自输入数据，不是编译时常量。当 `n` 很大时，栈缓冲区溢出。

**解决方案：**

```cpp
// 错误：用栈数组处理可变大小数据
void decompress(const uint8_t* src, size_t src_size) {
    // 🔴 如果 src_size 很大，栈直接爆掉
    uint8_t temp[src_size];  // VLA（Variable Length Array）——C++标准不支持！
    // ...
}

// 正确：用 vector 动态分配
void decompress(const uint8_t* src, size_t src_size) {
    // ✅ 堆分配，大小由运行时决定
    std::vector<uint8_t> temp(src_size);
    // 或者：
    auto temp2 = std::make_unique<uint8_t[]>(src_size);
    // ...
}
```

### 错误 3：模糊测试跑了几秒就因 OOM 被系统杀掉

**症状：** libFuzzer 运行后很快被操作系统的 OOM Killer 终止，或出现 "out-of-memory" 错误。

**原因：** Fuzz Target 中解压函数的输出大小由输入数据控制。攻击者可以构造一个小输入（几十字节）声称解压后大小为 4GB——即"解压炸弹"（Decompression Bomb）。

**解决方案：**

```cpp
// 在 Fuzz Target 中限制最大输入和输出大小
extern "C" int LLVMFuzzerTestOneInput(
    const uint8_t* data,
    size_t size
) {
    // 限制输入大小，避免处理过大的数据
    if (size > 1024 * 1024) return 0;  // 跳过 >1MB 的输入

    auto result = pkn_decompress(data, size);

    // 限制输出大小检查
    if (result.size() > 16 * 1024 * 1024) {
        // 如果解压后超过 16MB，说明缺少大小限制
        __builtin_trap();  // 报告为 Bug
    }

    return 0;
}

// 同时在 libFuzzer 命令行限制内存：
// ./fuzz_pkn corpus/ -rss_limit_mb=2048  # 限制 RSS 为 2GB
```

## 本节小结

- **逆向还原的代码面临独特的验证挑战**：没有规格说明、反编译器噪音、平台差异放大错误、边界行为不可知——因此需要比普通开发更严格的测试策略
- **四层防线**构成完整的验证体系：往返测试（内部一致性）→ 差分测试（与原版一致）→ 边界测试（极端输入安全）→ 模糊测试（自动发现未知 Bug）
- **往返测试**是最基础也最重要的验证：压缩 → 解压 → 比较，证明编码/解码的双向一致性。用 Catch2 的 `TEST_CASE` + `SECTION` 组织多种输入模式
- **差分测试**通过 Wine（Linux/macOS）或直接调用（Windows）加载原版 DLL，对比原版和还原版的输出，确保行为一致
- **边界测试**覆盖 13 类极端输入：空数据、单字节、全零、全 0xFF、重复模式、随机数据、块边界、截断数据、损坏头部、越界回引等
- **Sanitizer（ASan + UBSan）是逆向代码的必备工具**：能捕获肉眼不可见的内存越界和未定义行为。四个平台都有对应的配置方式
- **模糊测试**用 libFuzzer 或 AFL++ 自动生成数十亿种输入，发现人类想不到的崩溃。种子语料库的质量直接影响 fuzzer 的效率
- **崩溃输入必须转化为回归测试**：将 fuzzer 发现的 `crash-*` 文件最小化后，以字节数组形式写入 Catch2 测试，永久防止同类 Bug 复发
- **测试文件跟随项目规范组织**：放在 `tests/unit-tests/plugins/` 下，用 `TEST_FILES_PATH` 宏定位夹具文件
- **解压炸弹防御**是逆向压缩算法的必做项：在解压函数中添加输出大小上限检查，在 Fuzz Target 中限制 RSS 内存

        }
    }
    return true;
}

### 题目 2：修复一个边界条件 Bug

**场景：** 以下 LZ 解压函数在边界测试中崩溃了。ASan 报告 "heap-buffer-overflow"，崩溃输入是一个声称解压后大小为 100 字节但压缩数据只有 8 字节的流。请找出 Bug 并修复。

```cpp
// 有 Bug 的 LZ 解压函数
std::vector<uint8_t> lz_decompress(
    const uint8_t* src, size_t src_size
) {
    if (src_size < 4) return {};

    // 读取头部：前 4 字节是解压后大小（小端序）
    uint32_t output_size =
        src[0] | (src[1] << 8) | (src[2] << 16) | (src[3] << 24);

    std::vector<uint8_t> output(output_size);
    size_t src_pos = 4;
    size_t dst_pos = 0;

    while (dst_pos < output_size) {
        uint8_t flag = src[src_pos++];  // 🔴 Bug 在这里附近

        if (flag & 0x80) {
            // 回引：高 7 位是偏移，下一个字节是长度
            uint16_t offset = (flag & 0x7F) << 8 | src[src_pos++];
            uint8_t length = src[src_pos++];

            for (uint8_t i = 0; i < length; ++i) {
                output[dst_pos] = output[dst_pos - offset];
                ++dst_pos;
            }
        } else {
            // 字面量：flag 就是字节本身
            output[dst_pos++] = flag;
        }
    }

    return output;
}
```

<details>
<summary>查看答案</summary>

**Bug 分析：** 函数有 3 个越界访问风险：

1. **`src[src_pos++]` 没有检查 `src_pos < src_size`：** 当压缩数据提前结束时，继续读取会越界
2. **`output[dst_pos - offset]` 没有检查 `offset <= dst_pos`：** 偏移可能指向缓冲区之前
3. **`output_size` 可能是恶意构造的极大值：** 分配 4GB 内存会导致 OOM

**修复后的代码：**

```cpp
std::vector<uint8_t> lz_decompress(
    const uint8_t* src, size_t src_size
) {
    if (src_size < 4) return {};

    uint32_t output_size =
        src[0] | (src[1] << 8) | (src[2] << 16) | (src[3] << 24);

    // 修复 1：限制最大输出大小，防止解压炸弹
    constexpr uint32_t MAX_OUTPUT = 64 * 1024 * 1024;  // 64MB
    if (output_size > MAX_OUTPUT) return {};

    std::vector<uint8_t> output(output_size);
    size_t src_pos = 4;
    size_t dst_pos = 0;

    while (dst_pos < output_size) {
        // 修复 2：检查输入是否还有数据可读
        if (src_pos >= src_size) return {};  // 输入提前结束

        uint8_t flag = src[src_pos++];

        if (flag & 0x80) {
            // 修复 3：检查回引读取不越界
            if (src_pos + 1 >= src_size) return {};  // 需要再读 2 字节

            uint16_t offset = (flag & 0x7F) << 8 | src[src_pos++];
            uint8_t length = src[src_pos++];

            // 修复 4：检查回引偏移合法性
            if (offset == 0 || offset > dst_pos) return {};

            // 修复 5：检查输出不越界
            if (dst_pos + length > output_size) return {};

            for (uint8_t i = 0; i < length; ++i) {
                output[dst_pos] = output[dst_pos - offset];
                ++dst_pos;
            }
        } else {
            output[dst_pos++] = flag;
        }
    }

    return output;
}
```

共添加了 5 处边界检查，每一处都对应一种可能的攻击/错误输入。

</details>

### 题目 3：为 libFuzzer 编写压缩算法的 Fuzz Target

**问题：** 为一个同时支持压缩和解压的算法编写 Fuzz Target。要求：(a) 测试解压的鲁棒性（随机数据不崩溃）、(b) 测试往返一致性（压缩后解压等于原始）、(c) 设置合理的大小限制。请写出完整代码。

<details>
<summary>查看答案</summary>

```cpp
// fuzz_compress.cpp — 通用压缩算法 Fuzz Target
#include <cstdint>
#include <cstddef>
#include <cstring>
#include <vector>

// 假设被测接口
extern std::vector<uint8_t> my_compress(
    const uint8_t* data, size_t size
);
extern std::vector<uint8_t> my_decompress(
    const uint8_t* data, size_t size
);

extern "C" int LLVMFuzzerTestOneInput(
    const uint8_t* data,
    size_t size
) {
    // (c) 大小限制：跳过过大的输入，防止 OOM
    if (size > 256 * 1024) return 0;  // 最大 256KB

    // ====================================================
    // (a) 测试解压鲁棒性：随机数据当成压缩流解压
    // ====================================================
    {
        auto result = my_decompress(data, size);
        // 无论成功或失败，不崩溃就行
        // 如果成功，检查输出大小合理性
        if (result.size() > 64 * 1024 * 1024) {
            // 解压输出超过 64MB 说明缺少大小限制
            __builtin_trap();
        }
    }

    // ====================================================
    // (b) 测试往返一致性：原始数据 → 压缩 → 解压
    // ====================================================
    {
        auto compressed = my_compress(data, size);
        if (!compressed.empty()) {
            auto decompressed = my_decompress(
                compressed.data(), compressed.size()
            );
            // 解压后必须与原始数据完全一致
            if (decompressed.size() != size) {
                __builtin_trap();  // 大小不一致
            }
            if (size > 0 &&
                memcmp(decompressed.data(), data, size) != 0) {
                __builtin_trap();  // 内容不一致
            }
        }
    }

    return 0;
}
```

### 4.4 使用 Sanitizer 强化边界测试

光靠断言不够——有些越界访问不会立刻崩溃，而是悄悄写坏了相邻内存。Sanitizer（编译器内置的运行时检查工具）能在越界发生的瞬间报错。以下是四个平台的 Sanitizer 配置：

**CMake 配置（添加 Sanitizer 编译选项）：**

```cmake
# cmake/SanitizerOptions.cmake
# 为测试目标添加 Sanitizer 支持

option(ENABLE_ASAN "启用 AddressSanitizer（内存越界检测）" OFF)
option(ENABLE_UBSAN "启用 UndefinedBehaviorSanitizer（未定义行为检测）" OFF)

function(add_sanitizer_flags target)
    if(ENABLE_ASAN)
        if(MSVC)
            # MSVC 的 ASan 用法与 GCC/Clang 不同
            target_compile_options(${target} PRIVATE /fsanitize=address)
            # MSVC ASan 不需要额外链接选项
        else()
            # GCC / Clang（Linux / macOS）
            target_compile_options(${target} PRIVATE
                -fsanitize=address      # 检测堆/栈越界、use-after-free
                -fno-omit-frame-pointer # 保留栈帧指针，方便看调用栈
            )
            target_link_options(${target} PRIVATE -fsanitize=address)
        endif()
    endif()

    if(ENABLE_UBSAN)
        if(NOT MSVC)
            # MSVC 暂不支持 UBSan
            target_compile_options(${target} PRIVATE
                -fsanitize=undefined    # 检测整数溢出、空指针解引用等
                -fno-sanitize-recover=all  # 遇到 UB 立刻终止（而不是打印警告继续跑）
            )
            target_link_options(${target} PRIVATE -fsanitize=undefined)
        endif()
    endif()
endfunction()
```

## 五、模糊测试（Fuzz Testing）

### 5.1 什么是模糊测试

模糊测试（Fuzzing）是一种自动化测试技术：不是人工设计测试用例，而是让工具自动生成大量随机或半随机的输入数据，喂给被测程序，观察是否发生崩溃、断言失败或内存错误。模糊测试之所以对逆向还原代码特别有价值，有三个原因：

1. **覆盖人类想不到的输入：** 你可能想到了"空输入""全零"等边界，但不会想到"第 47 字节为 0xFE 且第 48 字节为 0x00 时触发的整数溢出"——模糊测试可以
2. **持续进化的输入：** 现代模糊测试工具（如 libFuzzer、AFL++）使用覆盖率反馈（Coverage-guided Fuzzing）——它们监测每个输入触发了哪些代码路径，然后变异那些触发了新路径的输入，逐步深入代码的各个分支
3. **7×24 小时运行：** 人写测试需要时间，但模糊测试可以在服务器上持续运行数天，探索数十亿种输入组合

### 5.2 libFuzzer 基础——编写第一个 Fuzz Target

libFuzzer 是 LLVM/Clang 内置的模糊测试引擎（不需要额外安装，只要有 Clang 编译器就行）。它的核心概念是 **Fuzz Target**——一个接收随机数据的函数：

```cpp
// tests/fuzz/fuzz_pkn_decompress.cpp
// PackinOne 解压算法的 Fuzz Target
//
// 编译方式（不需要 main 函数，libFuzzer 自带）：
//   clang++ -g -fsanitize=fuzzer,address -o fuzz_pkn fuzz_pkn_decompress.cpp pkn_reversed.cpp
//
// 运行方式：
//   ./fuzz_pkn corpus/          # corpus/ 是种子语料库目录
//   ./fuzz_pkn -max_total_time=3600   # 跑 1 小时

#include <cstdint>
#include <cstddef>
#include "pkn_reversed.h"

// libFuzzer 要求实现这个固定签名的函数
// data: libFuzzer 生成的随机数据
// size: 数据长度
// 返回值: 必须返回 0
extern "C" int LLVMFuzzerTestOneInput(
    const uint8_t* data,
    size_t size
) {
    // 直接把随机数据当成"压缩后的数据"喂给解压函数
    // 目标：无论输入多么奇怪，解压函数都不能崩溃
    auto result = pkn_decompress(data, size);

    // 如果解压成功，再压缩回去，检查往返一致性
    if (!result.empty()) {
        auto recompressed = pkn_compress(result.data(), result.size());
        if (!recompressed.empty()) {
            auto roundtrip = pkn_decompress(
                recompressed.data(), recompressed.size()
            );
            // 往返结果必须一致
            if (roundtrip != result) {
                __builtin_trap();  // 触发崩溃，让 fuzzer 保存这个输入
            }
        }
    }

    return 0;  // 必须返回 0
}
```

关键要点：

- **没有 `main()` 函数：** libFuzzer 自带 `main()`，它会反复调用 `LLVMFuzzerTestOneInput`
- **组合 ASan：** `-fsanitize=fuzzer,address` 同时启用模糊测试和内存检测，这样即使越界访问没有立刻崩溃，ASan 也会捕获
- **`__builtin_trap()`：** 这是手动触发崩溃的方式。当我们检测到逻辑错误（如往返不一致）但程序没有自然崩溃时，用它让 fuzzer 知道"这个输入有问题"

### 5.3 准备种子语料库（Seed Corpus）

模糊测试从"种子"开始变异。好的种子让 fuzzer 更快发现深层 Bug：

```cpp
// tools/generate_fuzz_corpus.cpp
// 生成 PackinOne 解压 fuzzer 的种子语料库
#include <cstdint>
#include <cstring>
#include <fstream>
#include <iostream>
#include <string>
#include <vector>
#include <random>
#include <filesystem>
#include "pkn_reversed.h"

namespace fs = std::filesystem;

// 将二进制数据写入文件
static void write_seed(
    const fs::path& dir,
    const std::string& name,
    const std::vector<uint8_t>& data
) {
    auto path = dir / name;
    std::ofstream ofs(path, std::ios::binary);
    ofs.write(
        reinterpret_cast<const char*>(data.data()),
        static_cast<std::streamsize>(data.size())
    );
    std::cout << "  生成: " << path.string()
              << " (" << data.size() << " 字节)\n";
}

int main(int argc, char* argv[]) {
    fs::path corpus_dir = (argc > 1) ? argv[1] : "corpus";
    fs::create_directories(corpus_dir);

    std::cout << "=== 生成 Fuzz 种子语料库 ===\n";

    // 种子 1：空数据的压缩结果
    {
        std::vector<uint8_t> empty;
        auto compressed = pkn_compress(empty.data(), 0);
        if (!compressed.empty()) {
            write_seed(corpus_dir, "seed_empty.pkn", compressed);
        }
    }

    // 种子 2：小数据（"Hello"）
    {
        const char* msg = "Hello";
        auto compressed = pkn_compress(
            reinterpret_cast<const uint8_t*>(msg), 5
        );
        write_seed(corpus_dir, "seed_hello.pkn", compressed);
    }

    // 种子 3：全零 1KB
    {
        std::vector<uint8_t> zeros(1024, 0x00);
        auto compressed = pkn_compress(zeros.data(), zeros.size());
        write_seed(corpus_dir, "seed_zeros_1k.pkn", compressed);
    }

    // 种子 4：随机数据 4KB
    {
        std::mt19937 rng(12345);
        std::vector<uint8_t> random_data(4096);
        for (auto& b : random_data) {
            b = static_cast<uint8_t>(rng() & 0xFF);
        }
        auto compressed = pkn_compress(
            random_data.data(), random_data.size()
        );
        write_seed(corpus_dir, "seed_random_4k.pkn", compressed);
    }

    // 种子 5：重复模式（高压缩比）
    {
        std::vector<uint8_t> pattern(8192, 0x00);
        for (size_t i = 0; i < pattern.size(); ++i) {
            pattern[i] = static_cast<uint8_t>(i % 7);  // 7 字节循环
        }
        auto compressed = pkn_compress(
            pattern.data(), pattern.size()
        );
        write_seed(corpus_dir, "seed_pattern_8k.pkn", compressed);
    }

    // 种子 6：恰好 64KB（块边界）
    {
        std::vector<uint8_t> block(65536);
        std::mt19937 rng(67890);
        for (auto& b : block) {
            b = static_cast<uint8_t>(rng() & 0xFF);
        }
        auto compressed = pkn_compress(block.data(), block.size());
        write_seed(corpus_dir, "seed_block_64k.pkn", compressed);
    }

    std::cout << "种子语料库生成完毕: " << corpus_dir.string() << "\n";
    return 0;
}
```

### 5.4 AFL++ 作为替代方案

AFL++（American Fuzzy Lop Plus Plus）是另一个强大的覆盖率引导模糊测试工具。与 libFuzzer 的主要区别：

| 特性 | libFuzzer | AFL++ |
|------|-----------|-------|
| 集成方式 | 库模式（编译进程序） | 进程模式（外部启动程序） |
| 平台支持 | Linux/macOS（Clang） | Linux（GCC/Clang） |
| Windows 支持 | 有限 | 通过 WinAFL 支持 |
| 并行化 | 多线程（`-jobs`） | 多进程（`-M`/`-S`） |
| 学习曲线 | 低（写一个函数） | 中（需要配置环境） |
| 适用场景 | 库函数、解析器 | 独立程序、文件处理 |

AFL++ 的 Fuzz Target 需要从标准输入或文件读取数据：

```cpp
// tests/fuzz/afl_pkn_decompress.cpp
// AFL++ 版本的 Fuzz Target（从 stdin 读取数据）
#include <cstdint>
#include <cstdio>
#include <cstdlib>
#include <vector>
#include "pkn_reversed.h"

int main() {
    // 从 stdin 读取全部数据
    std::vector<uint8_t> input;
    int ch;
    while ((ch = fgetc(stdin)) != EOF) {
        input.push_back(static_cast<uint8_t>(ch));
    }

    // 尝试解压
    auto result = pkn_decompress(input.data(), input.size());

    // 如果成功，验证往返
    if (!result.empty()) {
        auto recompressed = pkn_compress(
            result.data(), result.size()
        );
        if (!recompressed.empty()) {
            auto roundtrip = pkn_decompress(
                recompressed.data(), recompressed.size()
            );
            if (roundtrip != result) {
                abort();  // AFL 通过进程退出状态检测 Bug
            }
        }
    }

    return 0;
}
```

**编译和运行 AFL++：**

```bash
# 安装 AFL++（Ubuntu/Debian）
sudo apt install afl++

# 用 AFL++ 的编译器包装器编译
afl-clang-fast++ -std=c++17 -g -fsanitize=address \
    -o afl_pkn tests/fuzz/afl_pkn_decompress.cpp pkn_reversed.cpp

# 运行（-i 种子目录，-o 输出目录）
afl-fuzz -i corpus/ -o findings/ -- ./afl_pkn

# 多核并行运行（主节点 + 从节点）
afl-fuzz -i corpus/ -o findings/ -M main -- ./afl_pkn &
afl-fuzz -i corpus/ -o findings/ -S worker1 -- ./afl_pkn &
afl-fuzz -i corpus/ -o findings/ -S worker2 -- ./afl_pkn &
afl-fuzz -i corpus/ -o findings/ -S worker3 -- ./afl_pkn &
```

### 5.5 将模糊测试集成到 CMake 构建

将 Fuzz Target 作为正式的 CMake 目标管理，而不是手动编译：

```cmake
# tests/fuzz/CMakeLists.txt
# 模糊测试目标配置

# 只在 Clang 编译器下支持 libFuzzer
if(CMAKE_CXX_COMPILER_ID STREQUAL "Clang")
    option(ENABLE_FUZZING "启用模糊测试目标" OFF)

    if(ENABLE_FUZZING)
        message(STATUS "模糊测试已启用（仅 Clang 支持）")

        # Fuzz Target: PackinOne 解压
        add_executable(fuzz_pkn_decompress
            fuzz_pkn_decompress.cpp
        )
        target_link_libraries(fuzz_pkn_decompress PRIVATE
            pkn_reversed_lib    # 还原的 PackinOne 库
        )
        target_compile_options(fuzz_pkn_decompress PRIVATE
            -fsanitize=fuzzer,address,undefined
            -fno-omit-frame-pointer
            -g                  # 调试信息（方便定位崩溃）
        )
        target_link_options(fuzz_pkn_decompress PRIVATE
            -fsanitize=fuzzer,address,undefined
        )

        # 种子语料库生成工具
        add_executable(generate_fuzz_corpus
            ${CMAKE_SOURCE_DIR}/tools/generate_fuzz_corpus.cpp
        )
        target_link_libraries(generate_fuzz_corpus PRIVATE
            pkn_reversed_lib
        )
        target_compile_features(generate_fuzz_corpus PRIVATE cxx_std_17)

        # 自定义目标：一键生成种子 + 运行 fuzzer
        add_custom_target(fuzz_pkn
            COMMAND $<TARGET_FILE:generate_fuzz_corpus> ${CMAKE_BINARY_DIR}/fuzz_corpus
            COMMAND $<TARGET_FILE:fuzz_pkn_decompress> ${CMAKE_BINARY_DIR}/fuzz_corpus
                    -max_total_time=300
                    -print_final_stats=1
            DEPENDS fuzz_pkn_decompress generate_fuzz_corpus
            COMMENT "运行 PackinOne 解压模糊测试（5分钟）"
            WORKING_DIRECTORY ${CMAKE_BINARY_DIR}
        )
    endif()
endif()
```

使用方法：

```bash
# 配置（必须用 Clang）
cmake -B build-fuzz \
    -DCMAKE_CXX_COMPILER=clang++ \
    -DENABLE_FUZZING=ON \
    -DCMAKE_BUILD_TYPE=Debug

# 编译所有 fuzz 目标
cmake --build build-fuzz

# 运行 5 分钟模糊测试
cmake --build build-fuzz --target fuzz_pkn

# 或者手动运行更长时间
./build-fuzz/tests/fuzz/fuzz_pkn_decompress \
    build-fuzz/fuzz_corpus/ \
    -max_total_time=86400 \    # 24 小时
    -jobs=4 \
    -workers=4
```

### 5.6 分析模糊测试发现的崩溃

当 fuzzer 发现崩溃时，它会将导致崩溃的输入保存到 `crash-<hash>` 文件中。以下是分析流程：

```bash
# 1. 查看崩溃文件列表
ls -la crash-*
# 输出示例：
#   crash-da39a3ee5e6b4b0d3255bfef95601890afd80709
#   crash-e3b0c44298fc1c149afbf4c8996fb924

# 2. 用原始程序复现崩溃
./fuzz_pkn_decompress crash-da39a3ee5e6b4b0d3255bfef95601890afd80709
# ASan 会打印详细的错误信息和调用栈

# 3. 查看崩溃输入的十六进制内容
xxd crash-da39a3ee5e6b4b0d3255bfef95601890afd80709
# 输出示例：
#   00000000: 504b 4e00 ffff 0000 0100 0000           PKN.........
#                        ^^^^
#                        这里偏移字段为 0xFFFF，导致越界读取

# 4. 最小化崩溃输入（去掉不必要的字节）
./fuzz_pkn_decompress -minimize_crash=1 \
    -exact_artifact_path=crash_minimized \
    crash-da39a3ee5e6b4b0d3255bfef95601890afd80709
# minimized 版本通常只有几十字节，更容易分析

# 5. 将最小化后的输入转为 Catch2 测试用例
xxd -i crash_minimized
# 输出 C 数组，复制到测试代码中：
#   unsigned char crash_minimized[] = {
#     0x50, 0x4b, 0x4e, 0x00, 0xff, 0xff, ...
#   };
```

将发现的崩溃转化为回归测试，防止同类 Bug 再次出现：

```cpp
// tests/unit-tests/plugins/test_pkn_regression.cpp
// 由模糊测试发现的崩溃回归测试
#define CATCH_CONFIG_MAIN
#include <catch2/catch_test_macros.hpp>
#include <cstdint>
#include <vector>
#include "pkn_reversed.h"

TEST_CASE("回归测试 — fuzzer 发现的崩溃", "[pkn][regression]") {

    SECTION("crash-001: 偏移越界导致堆缓冲区溢出") {
        // 由 fuzz_pkn_decompress 在 2026-03-20 发现
        // 根因：pkn_decompress 未检查 LZ 回引偏移是否超出已解压数据长度
        // 修复：在 decompress_block() 第 87 行添加 offset < output_pos 检查
        const uint8_t crash_input[] = {
            0x50, 0x4b, 0x4e, 0x00,  // 魔数 "PKN\0"
            0x05, 0x00, 0x00, 0x00,  // 解压后大小 = 5
            0xff, 0xff,              // LZ 偏移 = 65535（远超已解压长度）
            0x03, 0x00,              // LZ 长度 = 3
        };
        auto result = pkn_decompress(crash_input, sizeof(crash_input));
        // 修复后应返回空（检测到无效偏移），而不是越界读取
        REQUIRE(result.empty());
    }

    SECTION("crash-002: 解压大小为 0 导致除零错误") {
        // 由 fuzz_pkn_decompress 在 2026-03-20 发现
        // 根因：块大小计算使用 total_size / num_blocks，当 total_size=0 时除零
        // 修复：在 init_blocks() 开头添加 total_size == 0 的提前返回
        const uint8_t crash_input[] = {
            0x50, 0x4b, 0x4e, 0x00,  // 魔数
            0x00, 0x00, 0x00, 0x00,  // 解压后大小 = 0
            0x01, 0x00, 0x00, 0x00,  // 块数 = 1
        };
        auto result = pkn_decompress(crash_input, sizeof(crash_input));
        REQUIRE(result.empty());
    }
}
