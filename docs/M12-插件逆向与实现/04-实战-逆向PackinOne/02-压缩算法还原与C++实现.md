## 术语预览

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| LZSS | Lempel-Ziv-Storer-Szymanski | LZ77 的改进变体——用标志位区分字面量和回引，比 LZ77 更紧凑 |

### 1.1 Ghidra 反编译输出的典型面貌

### 1.3 第二步：提取算法骨架

去掉 Ghidra 的噪音，算法的核心逻辑浮现：

```
LZSS 解压算法骨架：

1. 初始化：dst_pos = 0, flag_count = 0
2. 循环直到 dst_pos >= output_size：
   a. 如果 flag_count == 0：
      - 从输入读取一个字节作为 flag_byte
      - flag_count = 8（一个标志字节控制 8 个单元）
   b. 检查 flag_byte 的最低位：
      - 如果 == 0：字面量
        · 从输入读取一个字节，写入输出
      - 如果 == 1：回引
        · 从输入读取两个字节，解析为 (offset, length)
        · 从输出的 (dst_pos - offset) 处复制 length 个字节
   c. flag_byte 右移 1 位，flag_count 减 1
3. 结束
```

## 二、LZSS 解压算法的 C++ 实现

### 2.1 回引编码格式详解

在实现之前，我们需要完全理解两字节回引的位级编码。从反编译代码中提取的编码规则：

```
两字节回引编码（从压缩数据中读取的 byte0 和 byte1）：

       byte0              byte1
  ┌─────────────┐  ┌─────────────┐
  │ O₇ O₆ O₅ O₄│  │ O₁₁O₁₀O₉ O₈│
  │ O₃ O₂ O₁ O₀│  │ L₃ L₂ L₁ L₀│
  └─────────────┘  └─────────────┘

解析方式（从 Ghidra 代码翻译）：
  offset = byte0 | ((byte1 & 0xF0) << 4)
         = byte0 | ((byte1 >> 4) << 8)
         = 低 8 位来自 byte0，高 4 位来自 byte1 的高半字节
         → 12 位偏移，范围 0-4095

  length = (byte1 & 0x0F) + 3
         = byte1 的低 4 位 + 3
         → 4 位长度 + 3，范围 3-18

为什么长度要加 3？
  因为最小匹配长度是 3（低于 3 字节的匹配编码后反而更大），
  所以长度字段存储的是 (实际长度 - 3)，节省空间。
```

### 2.2 环形缓冲区解释

反编译代码中有一个关键操作：`(dst_pos - offset) & 0xFFF`。这是一个环形缓冲区（Ring Buffer）的取模运算：

```
环形缓冲区（大小 = 4096 = 0x1000）：

逻辑视图（无限长的输出流）：
  ... [已写入的字节] [当前位置 dst_pos]
                     ↑
                     这里

物理视图（4096 字节的环形数组）：
  ┌───────────────────────────────────────┐
  │ 位置 0    位置 1   ...   位置 4095    │
  │   ↑                                   │
  │   写入位置 = dst_pos % 4096           │
  └───────────────────────────────────────┘
     │                              ↑
     └──────────── 首尾相连 ─────────┘

当 dst_pos 超过 4096 时：
  实际位置 = dst_pos & 0xFFF   (等价于 dst_pos % 4096)

回引时：
  源位置 = (dst_pos - offset) & 0xFFF
  从源位置开始逐字节复制到当前位置
```

### 2.3 完整的解压函数实现

综合以上分析，写出干净的 C++ 解压函数：

```cpp
// pkn_decompress.h — PackinOne LZSS 解压算法
#pragma once
#include <cstdint>
#include <cstring>
#include <vector>

namespace pkn {

// LZSS 参数常量（从逆向分析确定）
constexpr size_t RING_SIZE = 4096;      // 滑动窗口大小 = 2^12
constexpr size_t RING_MASK = 0xFFF;     // 取模掩码 = RING_SIZE - 1
constexpr size_t MAX_MATCH = 18;        // 最大匹配长度 = 15 + 3
constexpr size_t MIN_MATCH = 3;         // 最小匹配长度
constexpr size_t FLAG_BITS = 8;         // 一个标志字节控制 8 个单元
constexpr uint8_t RING_FILL = 0x20;     // 环形缓冲区的初始填充值（空格）

/// 解压一个 LZSS 压缩块
/// @param input      压缩数据缓冲区
/// @param input_size 压缩数据长度
/// @param output_size 期望的解压后大小（从文件头读取）
/// @return 解压后的数据；如果输入无效则返回空 vector
inline std::vector<uint8_t> decompress_block(
    const uint8_t* input,
    size_t input_size,
    size_t output_size
) {
    // 安全检查：防止解压炸弹
    if (output_size > 64 * 1024 * 1024) {
        return {};  // 拒绝超过 64MB 的输出
    }

    std::vector<uint8_t> output(output_size);

    // 初始化环形缓冲区
    // 原版用 0x20（ASCII 空格）填充，这是 LZSS 的常见做法
    uint8_t ring[RING_SIZE];
    std::memset(ring, RING_FILL, sizeof(ring));

    size_t src_pos = 0;     // 输入读取位置
    size_t dst_pos = 0;     // 输出写入位置
    size_t ring_pos = RING_SIZE - MAX_MATCH;  // 环形缓冲区写入位置
    uint32_t flags = 0;     // 当前标志字节的值
    int flag_count = 0;     // 剩余标志位数量

    while (dst_pos < output_size) {
        // 每处理 8 个单元，读取一个新的标志字节
        if (flag_count == 0) {
            if (src_pos >= input_size) break;  // 输入耗尽
            flags = input[src_pos++];
            flag_count = FLAG_BITS;
        }

        if ((flags & 1) == 0) {
            // ── 字面量 ──
            // 标志位 = 0：下一个字节是原始数据
            if (src_pos >= input_size) break;

            uint8_t byte = input[src_pos++];
            output[dst_pos++] = byte;

            // 同时写入环形缓冲区
            ring[ring_pos] = byte;
            ring_pos = (ring_pos + 1) & RING_MASK;
        } else {
            // ── 回引 ──
            // 标志位 = 1：接下来两个字节编码 (offset, length)
            if (src_pos + 1 >= input_size) break;

            uint8_t byte0 = input[src_pos++];
            uint8_t byte1 = input[src_pos++];

            // 解析偏移和长度
            size_t offset = byte0 | ((byte1 & 0xF0) << 4);
            size_t length = (byte1 & 0x0F) + MIN_MATCH;

            // 从环形缓冲区复制
            for (size_t i = 0; i < length && dst_pos < output_size; ++i) {
                uint8_t byte = ring[(offset + i) & RING_MASK];
                output[dst_pos++] = byte;

                ring[ring_pos] = byte;
                ring_pos = (ring_pos + 1) & RING_MASK;
            }
        }

        // 右移标志字节，处理下一个标志位
        flags >>= 1;
        --flag_count;
    }

    return output;
}

} // namespace pkn
```

**代码中的关键决策和原因：**

| 代码行 | 决策 | 原因 |
|--------|------|------|
| `RING_FILL = 0x20` | 用空格填充 | 原版反编译代码中 `memset(ring, 0x20, ...)` 确认 |
| `ring_pos = RING_SIZE - MAX_MATCH` | 初始位置不是 0 | 原版代码中的初始化逻辑，保证前 18 字节的回引能正确落在填充区 |
| `(offset + i) & RING_MASK` | 回引用加法而不是减法 | 原版使用绝对位置而非相对偏移，offset 直接是环形缓冲区中的绝对索引 |
| `output_size > 64MB` 检查 | 拒绝过大输出 | 原版没有此检查，但我们必须防御解压炸弹 |

## 三、LZSS 压缩算法的 C++ 实现

### 3.1 为什么需要自己写压缩器？

你可能会想：我们是在做**解压**，为什么还要写**压缩**？三个原因：

1. **往返测试（Round-trip Testing）**：把数据压缩再解压，检查是否与原始数据完全一致，是验证解压器正确性的最可靠方法
2. **制作测试数据**：有了压缩器，可以为解压器生成各种特定模式的压缩数据，而不必依赖原版 DLL
3. **理解深化**：写压缩器能暴露你对算法的理解盲区——如果理解有误，压缩器和解压器的输出不会一致

> **注意**：我们的压缩器不需要与原版 DLL 的压缩输出逐字节一致（因为同一输入可以有多种合法压缩结果），但解压我们的压缩结果必须得到原始数据。

### 3.2 滑动窗口匹配搜索

LZSS 压缩的核心是**在滑动窗口中搜索最长匹配**。对于每个待压缩的字节位置，在之前的 4096 字节（窗口大小）中寻找与当前位置开始的数据最长的重复片段：

```
滑动窗口匹配搜索示意图：

输入数据：  ... A B C D E F A B C D X Y Z ...
                                ↑
                          当前位置 (pos)

搜索窗口（pos 之前的 4096 字节）：
  ┌──────────────────────────────────┐
  │ ... A B C D E F │ A B C D X Y Z │
  │     ↑           │ ↑             │
  │  匹配起点       │ 当前位置      │
  └──────────────────────────────────┘

找到匹配：从 6 字节前开始，长度 = 4 (A B C D)
编码为回引：(offset=6, length=4)
```

**朴素搜索算法**（O(n × window_size × max_match)）对于我们的场景足够快——PackinOne 处理的单个文件通常不超过几百 KB：

```cpp
// 在环形缓冲区中搜索最长匹配
// 返回 {最佳偏移, 最佳长度}，如果找不到 >= MIN_MATCH 的匹配则返回 {0, 0}
struct MatchResult {
    size_t offset;  // 环形缓冲区中的绝对位置
    size_t length;  // 匹配长度（0 表示未找到）
};

inline MatchResult find_longest_match(
    const uint8_t ring[RING_SIZE],  // 环形缓冲区
    size_t ring_pos,                // 当前写入位置
    const uint8_t* data,            // 待压缩的原始数据
    size_t data_pos,                // 当前处理位置
    size_t data_size                // 数据总长度
) {
    MatchResult best = {0, 0};

    // 计算本次最大可匹配长度
    size_t max_len = std::min(MAX_MATCH, data_size - data_pos);
    if (max_len < MIN_MATCH) {
        return best;  // 剩余数据不够最小匹配长度
    }

    // 遍历窗口中的每个位置
    for (size_t i = 1; i < RING_SIZE; ++i) {
        size_t candidate = (ring_pos - i) & RING_MASK;  // 候选起始位置
        size_t len = 0;

        // 计算从 candidate 开始能匹配多少字节
        while (len < max_len &&
               ring[(candidate + len) & RING_MASK] == data[data_pos + len]) {
            ++len;
        }

        // 更新最佳匹配
        if (len >= MIN_MATCH && len > best.length) {
            best.offset = candidate;
            best.length = len;
            if (len == max_len) break;  // 已达最大长度，无需继续
        }
    }

    return best;
}
```

**关键要点：**

- `candidate = (ring_pos - i) & RING_MASK`：从当前位置**往回**搜索，最近的匹配优先（虽然我们取最长的，但相同长度时更近的匹配可能更好）
- 搜索范围是整个环形缓冲区（4096 个位置），因为 LZSS 的偏移是绝对位置
- 匹配长度上限是 `MAX_MATCH`（18）或剩余数据长度，取较小值

### 3.3 标志字节编码机制

压缩输出时，每 8 个编码单元共享一个标志字节。这意味着我们不能边处理边输出——必须先攒够 8 个单元，才能确定标志字节的值，然后一起写入：

```
标志字节编码流程：

┌─ 标志字节 ─┐  ┌─ 单元 0 ─┐  ┌─ 单元 1 ─┐  ...  ┌─ 单元 7 ─┐
│ b7 b6 ... b0│  │ 1 或 2 字节│  │ 1 或 2 字节│       │ 1 或 2 字节│
└─────────────┘  └──────────┘  └──────────┘       └──────────┘

当 bi = 0 时：单元 i 是 1 字节的字面量
当 bi = 1 时：单元 i 是 2 字节的回引 (offset, length)

写入顺序：
1. 先在临时缓冲区中积累 8 个单元
2. 组装标志字节
3. 写入标志字节
4. 依次写入 8 个单元的数据
```

### 3.4 完整的压缩函数实现

```cpp
// pkn_compress.h — PackinOne LZSS 压缩算法
#pragma once
#include "pkn_decompress.h"  // 复用常量定义
#include <algorithm>

namespace pkn {

/// 压缩数据块（LZSS 算法）
/// @param data      待压缩的原始数据
/// @param data_size 数据长度
/// @return 压缩后的数据
inline std::vector<uint8_t> compress_block(
    const uint8_t* data,
    size_t data_size
) {
    // 压缩结果（预分配，最坏情况下压缩后比原始略大）
    std::vector<uint8_t> output;
    output.reserve(data_size + data_size / 8 + 16);

    // 初始化环形缓冲区（与解压器完全一致）
    uint8_t ring[RING_SIZE];
    std::memset(ring, RING_FILL, sizeof(ring));
    size_t ring_pos = RING_SIZE - MAX_MATCH;  // 初始写入位置 = 4078

    size_t pos = 0;  // 当前处理位置

    while (pos < data_size) {
        // ── 步骤 1：收集最多 8 个编码单元 ──
        uint8_t flag_byte = 0;        // 标志字节
        size_t unit_count = 0;        // 已收集的单元数
        // 临时缓冲区：最多 8 个单元，每个最多 2 字节
        uint8_t temp[8 * 2];
        size_t temp_len = 0;

        while (unit_count < FLAG_BITS && pos < data_size) {
            MatchResult match = find_longest_match(
                ring, ring_pos, data, pos, data_size
            );

            if (match.length >= MIN_MATCH) {
                // ── 回引：标志位 = 1 ──
                flag_byte |= (1 << unit_count);

                // 编码两字节：byte0 = offset 低 8 位
                //              byte1 高半字节 = offset 高 4 位
                //              byte1 低半字节 = length - 3
                uint8_t byte0 = match.offset & 0xFF;
                uint8_t byte1 = ((match.offset >> 4) & 0xF0)
                              | ((match.length - MIN_MATCH) & 0x0F);
                temp[temp_len++] = byte0;
                temp[temp_len++] = byte1;

                // 更新环形缓冲区
                for (size_t i = 0; i < match.length; ++i) {
                    ring[ring_pos] = data[pos + i];
                    ring_pos = (ring_pos + 1) & RING_MASK;
                }
                pos += match.length;
            } else {
                // ── 字面量：标志位 = 0 ──
                // flag_byte 对应位已经是 0，无需设置
                temp[temp_len++] = data[pos];

                ring[ring_pos] = data[pos];
                ring_pos = (ring_pos + 1) & RING_MASK;
                ++pos;
            }

            ++unit_count;
        }

        // ── 步骤 2：写入标志字节 + 单元数据 ──
        output.push_back(flag_byte);
        output.insert(output.end(), temp, temp + temp_len);
    }

    return output;
}

} // namespace pkn
```

**压缩器的正确性依赖于与解压器完全一致的状态管理：**

| 状态 | 压缩器 | 解压器 | 必须一致？ |
|------|--------|--------|-----------|
| 环形缓冲区初始填充 | `RING_FILL (0x20)` | `RING_FILL (0x20)` | ✅ 必须 |
| 初始 ring_pos | `RING_SIZE - MAX_MATCH` | `RING_SIZE - MAX_MATCH` | ✅ 必须 |
| 每次写入后 ring_pos 更新 | `(ring_pos + 1) & RING_MASK` | `(ring_pos + 1) & RING_MASK` | ✅ 必须 |
| 标志位含义 | 0=字面量, 1=回引 | 0=字面量, 1=回引 | ✅ 必须 |
| 偏移编码 | 绝对环形位置 | 绝对环形位置 | ✅ 必须 |

> **关键教训：** 任何一个初始化参数不一致（比如压缩器用 `ring_pos=0` 而解压器用 `ring_pos=4078`），解压结果就会完全错误。这就是为什么逆向分析时必须精确还原每一个初始值。

### 3.5 往返测试验证压缩-解压一致性

有了压缩器和解压器，我们可以写一个**往返测试（Round-trip Test）**来验证它们的正确性：

```cpp
// test_pkn_roundtrip.cpp — 往返测试

#include "pkn_decompress.h"
#include <cassert>
#include <cstdio>
#include <cstdlib>
#include <cstring>
#include <ctime>

// 测试用例 1：空数据
void test_empty() {
    std::vector<uint8_t> compressed = pkn::compress_block(nullptr, 0);
    // 空输入应该产生空输出（或只有一个标志字节）
    std::vector<uint8_t> decompressed = pkn::decompress_block(
        compressed.data(), compressed.size(), 0
    );
    assert(decompressed.size() == 0);
    std::printf("[PASS] test_empty\n");
}

// 测试用例 2：全相同字节（高压缩比场景）
void test_uniform() {
    std::vector<uint8_t> data(10000, 'A');
    std::vector<uint8_t> compressed = pkn::compress_block(
        data.data(), data.size()
    );
    std::vector<uint8_t> decompressed = pkn::decompress_block(
        compressed.data(), compressed.size(), data.size()
    );
    assert(decompressed == data);
    // 压缩后应该远小于原始大小
    std::printf("[PASS] test_uniform: %zu -> %zu bytes (%.1f%%)\n",
        data.size(), compressed.size(),
        100.0 * compressed.size() / data.size());
}

// 测试用例 3：随机数据（低压缩比场景）
void test_random() {
    std::srand(static_cast<unsigned>(std::time(nullptr)));
    std::vector<uint8_t> data(5000);
    for (auto& b : data) b = static_cast<uint8_t>(std::rand() % 256);

    std::vector<uint8_t> compressed = pkn::compress_block(
        data.data(), data.size()
    );
    std::vector<uint8_t> decompressed = pkn::decompress_block(
        compressed.data(), compressed.size(), data.size()
    );
    assert(decompressed == data);
    std::printf("[PASS] test_random: %zu -> %zu bytes (%.1f%%)\n",
        data.size(), compressed.size(),
        100.0 * compressed.size() / data.size());
}

// 测试用例 4：重复模式（中等压缩比）
void test_repeating_pattern() {
    // "ABCDEFGH" 重复 1000 次
    std::vector<uint8_t> data;
    const char* pattern = "ABCDEFGH";
    size_t pat_len = std::strlen(pattern);
    for (int i = 0; i < 1000; ++i) {
        data.insert(data.end(), pattern, pattern + pat_len);
    }

    std::vector<uint8_t> compressed = pkn::compress_block(
        data.data(), data.size()
    );
    std::vector<uint8_t> decompressed = pkn::decompress_block(
        compressed.data(), compressed.size(), data.size()
    );
    assert(decompressed == data);
    std::printf("[PASS] test_repeating_pattern: %zu -> %zu bytes (%.1f%%)\n",
        data.size(), compressed.size(),
        100.0 * compressed.size() / data.size());
}

// 测试用例 5：边界大小（1字节、2字节、3字节）
void test_tiny_sizes() {

}
```

## 四、PackinOne 归档文件的完整解析

### 4.1 PKN 归档格式结构

PackinOne 不仅是一个压缩库，它还定义了自己的归档格式（类似 ZIP 或 XP3，但更简单）。通过逆向 `getArchiveInfo` 和文件头解析函数，我们还原出了完整的文件格式：

```
PKN 归档文件整体布局：

偏移 0x00: ┌──────────────────────────────┐
           │       文件头 (PknHeader)      │  16 字节
           │  magic[4] = "PKN\0"          │
           │  version   = 1               │
           │  num_entries                  │
           │  data_offset                  │
偏移 0x10: ├──────────────────────────────┤
           │    条目表 (PknEntry × N)      │  每条 272 字节
           │  filename[256]               │
           │  original_size               │
           │  compressed_size             │
           │  crc32                        │
           ├──────────────────────────────┤
           │         ...更多条目...        │
           ├──────────────────────────────┤
data_offset:│    压缩数据区                │
           │  [entry0 的压缩数据]         │
           │  [entry1 的压缩数据]         │
           │  ...                          │
           └──────────────────────────────┘
```

### 4.2 C++ 数据结构定义

根据上面的格式分析，定义对应的 C++ 结构体：

```cpp
// pkn_archive.h — PackinOne 归档格式定义
#pragma once
#include "pkn_decompress.h"
#include <cstdint>
#include <cstring>
#include <fstream>
#include <string>
#include <vector>

namespace pkn {

// ── 文件头 ──
// 位于归档文件最开始的 16 字节
#pragma pack(push, 1)
struct PknHeader {
    char magic[4];        // "PKN\0"
    uint32_t version;     // 格式版本，目前只见过 1
    uint32_t num_entries; // 归档中的文件数量
    uint32_t data_offset; // 压缩数据区的起始偏移（从文件头开始算）
};
#pragma pack(pop)

// ── 条目头 ──
// 紧跟在文件头之后，每个条目 272 字节
#pragma pack(push, 1)
struct PknEntry {
    char filename[256];       // 文件名（Shift-JIS 编码，零填充）
    uint32_t original_size;   // 解压后大小
    uint32_t compressed_size; // 压缩后大小
    uint32_t crc32;           // 原始数据的 CRC32 校验和
};
#pragma pack(pop)

// 编译期验证结构体大小
static_assert(sizeof(PknHeader) == 16,
    "PknHeader must be exactly 16 bytes");
static_assert(sizeof(PknEntry) == 268,
    "PknEntry must be exactly 268 bytes");
    // 注意：256 + 4 + 4 + 4 = 268，不是 272
    // 实际文件中每条目可能有 4 字节对齐填充

} // namespace pkn
```

> **`#pragma pack(push, 1)` 的作用：** 告诉编译器不要在结构体成员之间插入填充字节（Padding）。这是读取二进制文件格式时的标准做法——确保内存布局与文件中的字节排列完全一致。`push` 保存当前对齐设置，`pop` 恢复，避免影响其他代码。

### 4.3 CRC32 校验实现

归档中每个条目都有 CRC32 校验和，用于验证解压后的数据完整性。CRC32 算法基于查表法实现：

```cpp
// pkn_crc32.h — CRC32 计算（IEEE 802.3 标准多项式）
#pragma once
#include <cstddef>
#include <cstdint>

namespace pkn {

// CRC32 查找表（编译期生成也行，这里用运行时初始化更清晰）
class CRC32 {
public:
    CRC32() {
        // 使用标准多项式 0xEDB88320（IEEE 802.3 反转形式）
        for (uint32_t i = 0; i < 256; ++i) {
            uint32_t crc = i;
            for (int j = 0; j < 8; ++j) {
                if (crc & 1)
                    crc = (crc >> 1) ^ 0xEDB88320u;
                else
                    crc >>= 1;
            }
            table_[i] = crc;
        }
    }

    /// 计算一块数据的 CRC32
    uint32_t compute(const uint8_t* data, size_t size) const {
        uint32_t crc = 0xFFFFFFFFu;  // 初始值全 1
        for (size_t i = 0; i < size; ++i) {
            crc = (crc >> 8) ^ table_[(crc ^ data[i]) & 0xFF];
        }
        return crc ^ 0xFFFFFFFFu;  // 最终取反
    }

private:
    uint32_t table_[256];
};

// 全局单例（线程安全的 C++11 静态初始化）
inline uint32_t crc32(const uint8_t* data, size_t size) {
    static const CRC32 instance;
    return instance.compute(data, size);
}

} // namespace pkn
```

**CRC32 的三个关键参数（决定兼容性）：**

| 参数 | 值 | 说明 |
|------|-----|------|
| 多项式 | `0xEDB88320` | IEEE 802.3 标准，反转位序（与 `zlib` 兼容） |
| 初始值 | `0xFFFFFFFF` | 全 1 初始化（避免前导零影响结果） |
| 最终异或 | `0xFFFFFFFF` | 对最终结果取反 |

> 如果你用错了多项式（比如用了 CRC-32C 的 `0x82F63B78`），校验和就对不上，文件会被判为损坏。

### 4.4 完整的归档解压工具

综合以上所有组件，写一个命令行工具，能解压 PKN 归档中的所有文件：

```cpp
// pkn_extract.cpp — PackinOne 归档解压工具
// 用法: pkn_extract <archive.pkn> [output_dir]
#include "pkn_archive.h"
#include "pkn_crc32.h"
#include "pkn_decompress.h"
#include <cstdio>
#include <cstdlib>
#include <cstring>
#include <filesystem>
#include <fstream>
#include <vector>

namespace fs = std::filesystem;

// 读取整个文件到 vector
std::vector<uint8_t> read_file(const fs::path& path) {
    std::ifstream f(path, std::ios::binary | std::ios::ate);
    if (!f.is_open()) {
        std::fprintf(stderr, "Error: cannot open '%s'\n",
            path.string().c_str());
        std::exit(1);
    }
    auto size = f.tellg();
    f.seekg(0);
    std::vector<uint8_t> data(static_cast<size_t>(size));
    f.read(reinterpret_cast<char*>(data.data()), size);
    return data;
}

int main(int argc, char* argv[]) {
    if (argc < 2) {
        std::printf("Usage: %s <archive.pkn> [output_dir]\n", argv[0]);
        return 1;
    }

    fs::path archive_path = argv[1];
    fs::path output_dir = (argc >= 3) ? argv[2] : ".";

    // ── 步骤 1：读取归档文件 ──
    auto file_data = read_file(archive_path);
    std::printf("Archive: %s (%zu bytes)\n",
        archive_path.string().c_str(), file_data.size());

    // ── 步骤 2：解析文件头 ──
    if (file_data.size() < sizeof(pkn::PknHeader)) {
        std::fprintf(stderr, "Error: file too small for PKN header\n");
        return 1;
    }

    pkn::PknHeader header;
    std::memcpy(&header, file_data.data(), sizeof(header));

    // 验证魔数
    if (std::memcmp(header.magic, "PKN", 3) != 0) {
        std::fprintf(stderr, "Error: invalid magic (expected PKN)\n");
        return 1;
    }

    std::printf("Version: %u, Entries: %u, Data offset: 0x%08X\n",
        header.version, header.num_entries, header.data_offset);

    // ── 步骤 3：解析条目表 ──
    size_t entry_offset = sizeof(pkn::PknHeader);
    size_t data_pos = header.data_offset;

    // 创建输出目录
    fs::create_directories(output_dir);

    int success_count = 0;
    int fail_count = 0;

    for (uint32_t i = 0; i < header.num_entries; ++i) {
        if (entry_offset + sizeof(pkn::PknEntry) > file_data.size()) {
            std::fprintf(stderr, "Error: truncated entry table at #%u\n", i);
            break;
        }

        pkn::PknEntry entry;
        std::memcpy(&entry, file_data.data() + entry_offset,
            sizeof(entry));
        entry_offset += sizeof(pkn::PknEntry);

        // 安全地获取文件名（确保零终止）
        char safe_name[257] = {};
        std::memcpy(safe_name, entry.filename,
            std::min<size_t>(256, sizeof(entry.filename)));

        std::printf("[%u/%u] %s (%u -> %u bytes) ",
            i + 1, header.num_entries, safe_name,
            entry.compressed_size, entry.original_size);

        // ── 步骤 4：解压每个条目 ──
        if (data_pos + entry.compressed_size > file_data.size()) {
            std::fprintf(stderr, "SKIP: compressed data out of bounds\n");
            ++fail_count;
            continue;
        }

        auto decompressed = pkn::decompress_block(
            file_data.data() + data_pos,
            entry.compressed_size,
            entry.original_size
        );
        data_pos += entry.compressed_size;

        // ── 步骤 5：CRC32 校验 ──
        uint32_t actual_crc = pkn::crc32(
            decompressed.data(), decompressed.size()
        );
        if (actual_crc != entry.crc32) {
            std::fprintf(stderr,
                "CRC MISMATCH (expected 0x%08X, got 0x%08X)\n",
                entry.crc32, actual_crc);
            ++fail_count;
            continue;
        }

        // ── 步骤 6：写入输出文件 ──
        fs::path out_path = output_dir / safe_name;
        fs::create_directories(out_path.parent_path());

        std::ofstream out(out_path, std::ios::binary);
        out.write(reinterpret_cast<const char*>(decompressed.data()),
            decompressed.size());
        out.close();

        std::printf("OK (CRC32: 0x%08X)\n", actual_crc);
        ++success_count;
    }

    std::printf("\nDone: %d extracted, %d failed, %u total\n",
        success_count, fail_count, header.num_entries);
    return fail_count > 0 ? 1 : 0;
}
```

## 五、差分测试验证

### 5.1 什么是差分测试

差分测试（Differential Testing）是验证逆向还原代码正确性的**黄金标准**。核心思想很简单：

```
             ┌──────────────────┐
             │   相同的输入数据   │
             └────────┬─────────┘
                ┌─────┴─────┐
                ↓           ↓
    ┌───────────────┐  ┌───────────────┐
    │ 原版 DLL      │  │ 我们的还原代码 │
    │ (Wine 运行)   │  │ (原生运行)     │
    └───────┬───────┘  └───────┬───────┘
            ↓                  ↓
    ┌───────────────┐  ┌───────────────┐
    │   输出 A       │  │   输出 B      │
    └───────┬───────┘  └───────┬───────┘
            └─────┬─────┘
                  ↓
          ┌──────────────┐
          │ A == B ?     │
          │ 逐字节比较    │
          └──────────────┘
```

**为什么不能只用往返测试？**
- 往返测试验证 `decompress(compress(data)) == data`，但这只能证明压缩器和解压器**自洽**
- 如果两者犯了同样的错误（比如偏移计算都差了 1），往返测试仍然通过
- 差分测试对比**原版实现**的输出，能发现这类系统性错误

### 5.2 搭建 Wine 差分测试环境

在 Linux/macOS 上通过 Wine 运行原版 DLL，是差分测试的典型方法。首先需要一个加载 DLL 并调用其解压函数的测试桩：

```cpp
// diff_test_stub.cpp — Wine 下调用原版 PackinOne.dll 的测试桩
// 编译: i686-w64-mingw32-g++ -std=c++17 -o diff_stub.exe diff_test_stub.cpp
#include <cstdint>
#include <cstdio>
#include <cstdlib>
#include <fstream>
#include <vector>
#include <windows.h>

// PackinOne.dll 导出的解压函数原型
// 从 dumpbin 导出表分析得到
typedef void(__cdecl* DecompressFunc)(
    uint8_t* output,
    int output_size,
    const uint8_t* input,
    int input_offset
);

int main(int argc, char* argv[]) {
    if (argc < 4) {
        std::printf(
            "Usage: %s <compressed_file> <output_size> <output_file>\n",
            argv[0]);
        return 1;
    }

    // 读取压缩数据
    std::ifstream fin(argv[1], std::ios::binary | std::ios::ate);
    if (!fin.is_open()) {
        std::fprintf(stderr, "Cannot open %s\n", argv[1]);
        return 1;
    }
    auto comp_size = fin.tellg();
    fin.seekg(0);
    std::vector<uint8_t> compressed(static_cast<size_t>(comp_size));
    fin.read(reinterpret_cast<char*>(compressed.data()), comp_size);
    fin.close();

    int output_size = std::atoi(argv[2]);

    // 加载原版 DLL
    HMODULE dll = LoadLibraryA("PackinOne.dll");
    if (!dll) {
        std::fprintf(stderr, "LoadLibrary failed: %lu\n", GetLastError());
        return 1;
    }

    // 获取解压函数地址
    // 注意：实际的导出名可能经过 C++ name mangling
    // 需要用 dumpbin 确认实际名称
    auto decompress = reinterpret_cast<DecompressFunc>(
        GetProcAddress(dll, "decompress_block")
    );
    if (!decompress) {
        std::fprintf(stderr, "GetProcAddress failed\n");
        FreeLibrary(dll);
        return 1;
    }

    // 调用解压
    std::vector<uint8_t> output(output_size);
    decompress(output.data(), output_size, compressed.data(), 0);

    // 写入结果
    std::ofstream fout(argv[3], std::ios::binary);
    fout.write(reinterpret_cast<const char*>(output.data()), output_size);
    fout.close();

    std::printf("Decompressed %zu -> %d bytes\n",
        compressed.size(), output_size);

    FreeLibrary(dll);
    return 0;
}
```

### 5.3 自动化差分测试脚本

用 Python 脚本自动化整个差分测试流程——生成测试数据、分别运行两个实现、比较输出：

```python
#!/usr/bin/env python3
"""
pkn_diff_test.py — PackinOne 差分测试自动化脚本

使用方法:
  python3 pkn_diff_test.py --dll PackinOne.dll --native ./pkn_extract

前提条件:
  - Wine 已安装 (Linux/macOS)
  - 或在 Windows 上直接运行
  - 原版 PackinOne.dll 在当前目录
  - 编译好的 diff_stub.exe (MinGW 交叉编译)
  - 编译好的原生 pkn_extract
"""
import hashlib
import os
import random
import struct
import subprocess
import sys
import tempfile

def generate_test_data(size: int, pattern: str = "random") -> bytes:
    """生成指定模式的测试数据"""
    if pattern == "random":
        return bytes(random.randint(0, 255) for _ in range(size))
    elif pattern == "zeros":
        return b'\x00' * size
    elif pattern == "text":
        # 模拟日文游戏脚本（高重复率的 Shift-JIS 文本）
        chars = "あいうえおかきくけこさしすせそたちつてと"
        text = "".join(random.choice(chars) for _ in range(size // 3))
        return text.encode("shift_jis")[:size]
    elif pattern == "binary":
        # 模拟二进制资源文件
        return struct.pack("<" + "I" * (size // 4),
            *[random.randint(0, 0xFFFF) for _ in range(size // 4)])
    else:
        raise ValueError(f"Unknown pattern: {pattern}")

def sha256(data: bytes) -> str:
    return hashlib.sha256(data).hexdigest()

def run_diff_test(
    test_data: bytes,
    native_cmd: str,
    wine_cmd: str
) -> bool:
    """运行一次差分测试，返回是否通过"""
    with tempfile.TemporaryDirectory() as tmp:
        # 写入测试数据
        input_path = os.path.join(tmp, "input.bin")
        with open(input_path, "wb") as f:
            f.write(test_data)

        native_out = os.path.join(tmp, "native_out.bin")
        wine_out = os.path.join(tmp, "wine_out.bin")

        # 运行原生实现
        subprocess.run(
            [native_cmd, input_path, str(len(test_data)), native_out],
            check=True, timeout=30
        )

        # 运行 Wine + 原版 DLL
        subprocess.run(
            ["wine", wine_cmd, input_path, str(len(test_data)), wine_out],
            check=True, timeout=30
        )

        # 比较输出
        with open(native_out, "rb") as f:
            native_data = f.read()
        with open(wine_out, "rb") as f:
            wine_data = f.read()

        if native_data == wine_data:
            return True
        else:
            print(f"  MISMATCH at size={len(test_data)}")
            print(f"  Native SHA256: {sha256(native_data)}")
            print(f"  Wine   SHA256: {sha256(wine_data)}")
            # 找到第一个不同的字节
            for i in range(min(len(native_data), len(wine_data))):
                if native_data[i] != wine_data[i]:
                    print(f"  First diff at byte {i}: "
                          f"native=0x{native_data[i]:02X} "
                          f"wine=0x{wine_data[i]:02X}")
                    break
            return False

def main():
    patterns = ["random", "zeros", "text", "binary"]
    sizes = [0, 1, 2, 3, 100, 1000, 4096, 8000, 65536]

    passed = 0
    failed = 0

    for pattern in patterns:
        for size in sizes:
            if size == 0 and pattern != "random":
                continue
            test_data = generate_test_data(max(1, size), pattern)
            ok = run_diff_test(
                test_data,
                native_cmd="./pkn_roundtrip",
                wine_cmd="diff_stub.exe"
            )
            status = "PASS" if ok else "FAIL"
            print(f"[{status}] pattern={pattern}, size={size}")
            if ok:
                passed += 1
            else:
                failed += 1

    print(f"\nResults: {passed} passed, {failed} failed")
    sys.exit(1 if failed > 0 else 0)

if __name__ == "__main__":
    main()
```

### 实践 1：实现一个简单的 PKN 归档创建工具

基于本节学到的压缩函数和归档格式，写一个 `pkn_create` 工具，能把多个文件打包成一个 PKN 归档：

```cpp
// pkn_create.cpp — PackinOne 归档创建工具
// 用法: pkn_create <output.pkn> <file1> [file2] ...
#include "pkn_archive.h"
#include "pkn_compress.h"
#include "pkn_crc32.h"
#include <cstdio>
#include <cstring>
#include <filesystem>
#include <fstream>
#include <vector>

namespace fs = std::filesystem;

std::vector<uint8_t> read_file(const fs::path& path) {
    std::ifstream f(path, std::ios::binary | std::ios::ate);
    if (!f) { std::fprintf(stderr, "Cannot open %s\n", path.string().c_str()); std::exit(1); }
    auto sz = f.tellg(); f.seekg(0);
    std::vector<uint8_t> d(static_cast<size_t>(sz));
    f.read(reinterpret_cast<char*>(d.data()), sz);
    return d;
}

int main(int argc, char* argv[]) {
    if (argc < 3) {
        std::printf("Usage: %s <output.pkn> <file1> [file2...]\n", argv[0]);
        return 1;
    }

    uint32_t num_files = argc - 2;
    // 预计算数据偏移
    uint32_t data_offset = sizeof(pkn::PknHeader)
                         + num_files * sizeof(pkn::PknEntry);

    // 收集所有文件的压缩数据
    struct FileInfo {
        std::string name;
        std::vector<uint8_t> compressed;
        uint32_t original_size;
        uint32_t crc;
    };
    std::vector<FileInfo> files;

    for (int i = 2; i < argc; ++i) {
        auto raw = read_file(argv[i]);
        FileInfo fi;
        fi.name = fs::path(argv[i]).filename().string();
        fi.original_size = static_cast<uint32_t>(raw.size());
        fi.crc = pkn::crc32(raw.data(), raw.size());
        fi.compressed = pkn::compress_block(raw.data(), raw.size());
        std::printf("  %s: %u -> %zu bytes\n",
            fi.name.c_str(), fi.original_size, fi.compressed.size());
        files.push_back(std::move(fi));
    }

    // 写入归档
    std::ofstream out(argv[1], std::ios::binary);

    // 文件头
    pkn::PknHeader hdr = {};
    std::memcpy(hdr.magic, "PKN", 4);  // 包含末尾 \0
    hdr.version = 1;
    hdr.num_entries = num_files;
    hdr.data_offset = data_offset;
    out.write(reinterpret_cast<const char*>(&hdr), sizeof(hdr));

    // 条目表
    for (const auto& fi : files) {
        pkn::PknEntry entry = {};
        std::strncpy(entry.filename, fi.name.c_str(),
            sizeof(entry.filename) - 1);
        entry.original_size = fi.original_size;
        entry.compressed_size =
            static_cast<uint32_t>(fi.compressed.size());
        entry.crc32 = fi.crc;
        out.write(reinterpret_cast<const char*>(&entry), sizeof(entry));
    }

    // 压缩数据
    for (const auto& fi : files) {
        out.write(reinterpret_cast<const char*>(fi.compressed.data()),
            fi.compressed.size());
    }

    out.close();
    std::printf("Created %s (%u files)\n", argv[1], num_files);
    return 0;
}
```

```cpp
void benchmark(size_t data_size) {
    // 生成测试数据（模拟游戏脚本：重复模式 + 随机噪声）
    std::vector<uint8_t> data(data_size);
    for (size_t i = 0; i < data_size; ++i) {

    // 解压计时
    auto t2 = std::chrono::high_resolution_clock::now();
    auto decompressed = pkn::decompress_block(
        compressed.data(), compressed.size(), data.size());
    auto t3 = std::chrono::high_resolution_clock::now();

    // 计算吞吐量
    double comp_ms = std::chrono::duration<double, std::milli>(t1 - t0).count();
    double decomp_ms = std::chrono::duration<double, std::milli>(t3 - t2).count();
    double comp_mbps = (data.size() / 1024.0 / 1024.0) / (comp_ms / 1000.0);
    double decomp_mbps = (data.size() / 1024.0 / 1024.0) / (decomp_ms / 1000.0);

    std::printf("Size: %7zu | Ratio: %5.1f%% | "
        "Compress: %6.1f ms (%5.1f MB/s) | "
        "Decompress: %6.1f ms (%5.1f MB/s)\n",
        data.size(),
        100.0 * compressed.size() / data.size(),
        comp_ms, comp_mbps,
        decomp_ms, decomp_mbps);

    assert(decompressed == data);
}
```

## 常见错误与排查

### 错误 1：环形缓冲区初始值不一致

**症状：** 解压后的数据前几个字节正确，但从第一个反向引用开始出现乱码。

```
预期: "Hello World! Hello W..."
实际: "Hello World!\x00\x00\x00\x00\x00..."
```

**原因：** 环形缓冲区初始化为 `0x00` 而非 `0x20`（ASCII 空格）。KiriKiri 的 LZSS 变体要求初始化为空格。

**修复：**

```cpp
// 错误 ❌
uint8_t ring[RING_SIZE] = {};  // 全零初始化

// 正确 ✅
uint8_t ring[RING_SIZE];
std::memset(ring, 0x20, sizeof(ring));  // 用 0x20 填充
```

### 错误 2：反向引用偏移解码位序搞反

**症状：** 解压数据完全是乱码，但大小正确。

**原因：** offset 的高 4 位和低 8 位在两个字节中的位置搞反了。

```cpp
// 错误 ❌ — 高低位搞反
uint16_t offset = (byte1 << 8) | (byte0 & 0xF0);
uint8_t  length = (byte0 & 0x0F) + 3;

// 正确 ✅ — byte0 是低 8 位，byte1 高半字节是高 4 位
uint16_t offset = byte0 | ((byte1 & 0xF0) << 4);
uint8_t  length = (byte1 & 0x0F) + 3;
```

### 错误 3：CRC32 校验始终失败

**症状：** 解压后的数据看起来正确，但 CRC32 校验报错。

**常见原因及修复：**

```cpp
// 原因 1：多项式用错了
// CRC32 IEEE 标准多项式是 0xEDB88320（反射形式）
// 不要用 0x04C11DB7（正常形式）

// 原因 2：初始值和最终异或
uint32_t crc = 0xFFFFFFFF;     // 初始值必须全 1
// ... 计算 ...
return crc ^ 0xFFFFFFFF;       // 最终要异或全 1

// 原因 3：对压缩数据而非原始数据做 CRC
// PKN 归档的 CRC 是原始数据的校验和
crc = pkn::crc32(original_data, original_size);  // ✅
// 不是：
// crc = pkn::crc32(compressed_data, compressed_size);  // ❌
```

## 本节小结

- **LZSS 变体参数**：KiriKiri 使用 12 位偏移（4096 字节窗口）、4 位长度（+3 = 3-18 范围）、8 位标志字节，环形缓冲区初始化为 `0x20`
- **解压算法**：逐位读取标志字节，1 = 字面量复制，0 = 反向引用（读取 2 字节解码偏移+长度），通过环形缓冲区输出
- **压缩算法**：滑动窗口朴素搜索最长匹配（≥3 字节），编码为反向引用或字面量，每 8 个决策生成一个标志字节
- **PKN 归档格式**：16 字节文件头 + 268 字节/条目的条目表 + 压缩数据区，使用 CRC32 校验原始数据完整性
- **IDA 逆向发现**：实际 PackinOne.dll 使用 LZ4（非纯 LZSS）作为主压缩算法，同时内嵌 CxImage 图像库和流式加密机制
- **差分测试**：通过 Wine/Windows 加载原版 DLL，对比原版与还原版的输出，是验证逆向正确性的金标准
- **往返测试**：compress → decompress → 对比原始数据，简单有效但无法发现"两端一致地错误"的 bug

<details>
<summary>查看答案</summary>

**1. 反向引用字节数**

13 位偏移 + 5 位长度 = 18 位，需要 3 个字节（向上取整到字节边界）。具体编码可以是：

```
byte0: offset 低 8 位
byte1: offset 高 5 位 (bit4-bit0) | length 低 3 位 (bit7-bit5)
byte2: length 高 2 位 (bit1-bit0) — 实际上 5 位长度在 byte1 和 byte2 中拆分
```

但更常见的做法是用整 3 字节：byte0 + byte1 = 16 位偏移（高 3 位填 0），byte2 = 长度。设计取舍取决于实际实现。

以 18 位紧凑编码为例：
- byte0: offset 低 8 位
- byte1: offset 中 5 位 (bit4-0) + length 高 3 位 (bit7-5)
- 如果 5 位长度够用，可以在 2.25 字节内编码，但实践中需要 3 字节。

**2. 压缩率对比**

- **更好的方面**：更大的窗口（8192 vs 4096）能找到更远处的匹配，更长的最大匹配（34 vs 18）能覆盖更多重复
- **更差的方面**：反向引用从 2 字节变为 3 字节，overhead 增加 50%
- **结论**：对重复率高的数据（如游戏脚本），更大窗口和更长匹配通常能补偿额外的引用开销，整体压缩率**略好**。对随机数据则更差（因为每个反向引用多花 1 字节）

**3. 解码代码**

```cpp
// 13-bit offset, 5-bit length 的反向引用解码
constexpr size_t RING_SIZE_V2  = 8192;  // 2^13
constexpr size_t MAX_MATCH_V2  = 34;    // 2^5 - 1 + 3
constexpr size_t MIN_MATCH     = 3;

// 反向引用：读 3 字节
uint8_t b0 = src[sp++]; // offset 低 8 位
uint8_t b1 = src[sp++]; // offset 高 5 位 + length 低 3 位
uint8_t b2 = src[sp++]; // length 高 2 位

uint16_t offset = b0 | ((b1 & 0x1F) << 8);         // 13 位偏移
uint8_t  length = ((b1 >> 5) | (b2 << 3)) + MIN_MATCH; // 5 位长度 + 3

for (uint8_t k = 0; k < length; ++k) {
    uint8_t c = ring[(offset + k) % RING_SIZE_V2];
    dst[dp++] = c;
    ring[rp] = c;
    rp = (rp + 1) % RING_SIZE_V2;
}
```

</details>

### 题目 2：PKN 归档安全性分析

一个恶意用户构造了一个 PKN 归档文件，其中某个条目的 `original_size` 字段设为 `0xFFFFFFFF`（约 4GB）。

1. 如果你的 `pkn_extract` 工具直接用 `original_size` 分配内存（`new uint8_t[original_size]`），会发生什么？
2. 请给出至少 3 个针对恶意 PKN 文件的防御措施，并写出对应的代码。
3. 为什么 CRC32 校验不能防止此类攻击？

<details>
<summary>查看答案</summary>

**1. 后果**

分配 4GB 内存几乎必然失败，`new` 抛出 `std::bad_alloc` 异常。如果代码使用 `malloc` 返回 `nullptr` 而没检查，后续写入会导致段错误（Segmentation Fault）或未定义行为。即使分配成功（64 位系统上可能），解压过程也会写入远超实际压缩数据的垃圾内容。

**2. 防御措施**

```cpp
// 防御 1：限制最大文件大小
constexpr uint32_t MAX_FILE_SIZE = 256 * 1024 * 1024; // 256MB 上限
if (entry.original_size > MAX_FILE_SIZE) {
    std::fprintf(stderr, "Entry '%s' size %u exceeds limit\n",
        entry.filename, entry.original_size);
    continue; // 跳过此条目
}

// 防御 2：校验压缩大小与原始大小的合理性
// LZSS/LZ4 压缩数据不可能比原始数据大太多
if (entry.compressed_size > entry.original_size + 1024) {
    std::fprintf(stderr, "Suspicious: compressed > original + margin\n");
    continue;
}

// 防御 3：校验偏移不越界
uint64_t end_pos = static_cast<uint64_t>(hdr.data_offset)
                 + entry_offset + entry.compressed_size;
if (end_pos > file_total_size) {
    std::fprintf(stderr, "Entry '%s' data extends beyond file\n",
        entry.filename);
    continue;
}

// 防御 4：文件名路径遍历检查
std::string fname(entry.filename);
if (fname.find("..") != std::string::npos ||
    fname.find('/') != std::string::npos ||
    fname.find('\\') != std::string::npos) {
    std::fprintf(stderr, "Path traversal in filename: %s\n",
        entry.filename);
    continue;
}
```

**3. CRC32 无法防御的原因**

CRC32 校验的是**解压后的数据**与**声明的原始大小**之间的一致性。但恶意文件可以：
- 让 `original_size` = 4GB，CRC 设为任意值——程序在计算 CRC 之前就会在内存分配阶段崩溃
- 即使不崩溃，CRC32 只有 32 位，可以被蓄意碰撞
- CRC32 不是密码学哈希，不能用于安全验证，只能检测意外损坏

CRC32 是**完整性检查**（防止传输/存储错误），不是**安全检查**（防止恶意篡改）。安全场景应使用 SHA-256 等密码学哈希。

</details>

### 题目 3：基于 IDA 数据的逆向推理

根据 IDA 分析发现 PackinOne.dll 中存在以下 RTTI 信息：

```
.?AVLZ4CompressStream@@
.?AVLZ4DecompressStream@@
.?AVLZ4StreamBase@@
.?AUStreamCryptReaderWriter@@
.?AUBufferedStreamCryptFilter@@
```

请回答：
1. 这些类名暗示了怎样的数据处理管线（pipeline）？请画出数据流向图。
2. 为什么 PackinOne 同时使用 LZ4 和流加密，而不单独使用其中一种？
3. 如果要在 KrKr2 仿真器中实现兼容的归档解压功能，最少需要实现哪些组件？

<details>
<summary>查看答案</summary>

**1. 数据处理管线**

```
写入（归档创建）流程：
┌──────────┐    ┌──────────────────┐    ┌───────────────────────┐    ┌──────────┐
│ 原始文件 │───▶│ LZ4CompressStream│───▶│ BufferedStreamCryptFilter│───▶│ PKN 文件 │
│ (明文)   │    │ (压缩)           │    │ (加密 + 缓冲)           │    │ (密文)   │
└──────────┘    └──────────────────┘    └───────────────────────┘    └──────────┘

读取（归档解压）流程：
┌──────────┐    ┌───────────────────────┐    ┌────────────────────┐    ┌──────────┐
│ PKN 文件 │───▶│ StreamCryptReaderWriter│───▶│ LZ4DecompressStream│───▶│ 原始文件 │
│ (密文)   │    │ (解密)                │    │ (解压)             │    │ (明文)   │
└──────────┘    └───────────────────────┘    └────────────────────┘    └──────────┘
```

继承关系：`LZ4CompressStream` 和 `LZ4DecompressStream` 继承 `LZ4StreamBase`。

**2. 同时使用 LZ4 和加密的原因**

- **LZ4 压缩**：减小归档体积，加速 IO（磁盘/网络读取量减少）。LZ4 解压极快（多 GB/s），几乎不增加运行时开销
- **流加密**：防止玩家直接解包游戏资源（脚本、图片、音频）。这是 galgame 行业的常见做法，用于保护剧情文本和 CG 图片不被轻易提取
- **为什么不单独用**：单独压缩无法保护资源；单独加密不能减小体积。先压缩再加密是标准做法（加密后的数据无法被压缩，所以顺序很重要）

**3. 仿真器最少需要的组件**

```
1. LZ4 解压器        — 可直接使用 lz4 开源库（vcpkg: lz4）
2. 流解密器          — 需要逆向还原加密算法和密钥派生逻辑
3. PKN 文件头解析器  — 读取条目表、定位压缩数据
4. IStream 适配器    — 将解压流包装为 KiriKiri 的 IStream 接口
5. 归档媒体注册      — 通过 TVPRegisterStorageMedia 注册
```

其中第 2 项（流解密器）是难度最大的部分，因为加密算法和密钥通常是每个游戏不同的。可以通过动态调试（hook `ReadFile`/`WriteFile`）来捕获解密后的数据流。

</details>
