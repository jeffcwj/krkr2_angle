## 1. PackinOne 压缩子系统概览

### 1.1 PackinOne 不是单纯的压缩插件

在逆向分析之前，许多资料将 PackinOne 描述为"数据压缩/归档插件"。
真实的 IDA 分析（`packinone_analysis.txt`）显示：PackinOne.dll 是一个**多功能工具集插件**，
压缩只是其中一个子系统。完整功能包括：

- **LZ4 压缩/解压**：核心压缩子系统（本节主题）
- **CxImage 图像处理**：PNG/JPEG 读写，含 iPhone CgBI 格式支持
- **进程管理**：`CreateProcess` + 管道通信
- **文件系统操作**：`StoragesFstat`、`TemporaryFiles`
- **TJS 扩展**：向 Layer、Font、Scripts、System 命名空间注册新方法
- **SHA-256 校验**：7519 字节的压缩函数，含 SIMD 加速

压缩子系统的代码规模（两个核心函数共 ~7200 字节）占整个 DLL 的显著比例，
说明是核心功能之一。

### 1.2 为什么选择 LZ4？

KiriKiri 引擎的游戏资源包（`.xp3` 格式）在载入时需要**实时解压**，
解压速度直接影响场景切换响应时间。LZ4 的优势恰好满足这个场景：

| 指标 | LZ4 | zlib/DEFLATE | LZMA |
|------|-----|--------------|------|
| 解压速度 | ~2 GB/s | ~300 MB/s | ~50 MB/s |
| 压缩速度 | ~500 MB/s | ~30 MB/s | ~10 MB/s |
| 压缩率 | 中等（~2:1） | 较好（~2.5:1） | 最好（~3:1） |
| 内存占用 | 极低 | 低 | 高 |

对于游戏资源实时加载，速度优先于压缩率，LZ4 是最合理的选择。

```
"Version 1.7.x"        ; 具体版本号从日志输出推断
```

## 2. LZ4 算法原理深度解析

### 2.1 LZ4 基本编码格式

LZ4 压缩流由若干 **sequence（序列）** 组成，每个 sequence 的结构如下：

```
┌─────────┬──────────────────┬─────────────────┬──────────┬──────────────────┐
│  token  │  extra literal   │    literals     │  offset  │  extra match     │
│ (1字节) │  (0-N字节)       │ (literal_len字节)│ (2字节LE)│  (0-N字节)       │
└─────────┴──────────────────┴─────────────────┴──────────┴──────────────────┘
```

**token 字节的位域**：

```
  7   6   5   4   3   2   1   0
┌───────────────┬───────────────┐
│  literal_len  │  match_len    │
│   (高4位)     │   (低4位)     │
└───────────────┴───────────────┘

literal_len = token >> 4          （0-14: 直接使用; 15: 需要读额外字节）
match_len   = (token & 0x0F) + 4  （最小匹配长度为4）
```

**extra length 编码**（当长度字段 = 15 时）：

```c
// 继续读取字节，每字节加到长度上，直到读到非 255 的字节
uint32_t read_extra_len(const uint8_t** src) {
    uint32_t len = 15;
    uint8_t b;
    do {
        b = *(*src)++;
        len += b;
    } while (b == 255);
    return len;
}
```

**一个完整 sequence 的伪代码解码**：

```c
// LZ4 解压核心伪代码（对应 sub_100553A0）
void lz4_decompress_sequence(const uint8_t* src, uint8_t* dst,
                              uint8_t** src_out, uint8_t** dst_out) {
    // 1. 读 token
    uint8_t token = *src++;
    
    // 2. 计算 literal 长度
    uint32_t lit_len = token >> 4;           // 高4位
    if (lit_len == 15) {
        lit_len = read_extra_len(&src);      // 读额外字节
    }
    
    // 3. 复制 literal
    memcpy(dst, src, lit_len);
    src += lit_len;
    dst += lit_len;
    
    // 4. 读 offset（小端2字节）
    uint16_t offset = src[0] | (src[1] << 8);
    src += 2;
    
    // 5. 计算 match 长度
    uint32_t mat_len = (token & 0x0F) + 4;  // 低4位 + 最小长度4
    if ((token & 0x0F) == 15) {
        mat_len = read_extra_len(&src) + 4;
    }
    
    // 6. 后向复制 match（注意：可能有重叠，不能用 memcpy）
    const uint8_t* match_src = dst - offset;
    for (uint32_t i = 0; i < mat_len; i++) {
        dst[i] = match_src[i];
    }
    dst += mat_len;
    
    *src_out = src;
    *dst_out = dst;
}
```

### 2.2 LZ4 压缩算法流程（哈希表匹配）

LZ4 压缩使用**哈希表**加速字符串匹配，而非 LZ77 的滑动窗口线性搜索：

```
输入字节流
    │
    ▼
┌─────────────────────────────────────┐
│  对当前位置取4字节，计算哈希         │
│  hash = (uint32)(input[pos:pos+4])  │
│       * 0x9E3779B1 >> 20            │
│  （Fibonacci 哈希，哈希表大小 4096） │
└────────────────┬────────────────────┘
                 │
    ┌────────────▼────────────┐
    │  查哈希表[hash]          │
    │  得到候选匹配位置 cand   │
    └────────────┬────────────┘
                 │
    ┌────────────▼────────────┐
    │  验证：input[pos:pos+4] │
    │      == input[cand:+4]? │
    │  且 pos - cand <= 65535?│
    └────────────┬────────────┘
                 │
         YES ◄───┴───► NO
          │              │
    记录 match        记录 literal
    (offset, len)     继续前进
          │              │
    更新哈希表[hash]←────┘
          │
    输出 sequence
```

**关键哈希常量来自 IDA 反汇编（`sub_10053F50` 第一个关键指令）**：

```asm
; sub_10053F50 +offset（LZ4 压缩核心，5142 字节）
; 从 packinone_analysis.txt 中的真实反汇编
mov  eax, [esi+ecx]         ; 读4字节输入
imul eax, 9E3779B1h         ; Fibonacci 哈希乘法
shr  eax, 14h               ; >> 20，得到 12 位索引（2^12 = 4096 个槽）
```

这个常量 `0x9E3779B1` 是 **Fibonacci 哈希常量**（`2^32 / φ`，其中 φ 是黄金比例），
LZ4 官方源码中完全一致，确认了这里使用的是标准 LZ4 算法。

### 2.3 LZ4 解压算法流程

解压比压缩简单得多，无需哈希表，只需顺序读取 sequence：

```
读 token
    │
    ├─► 高4位 → literal_len（+额外字节）
    │
    ├─► 复制 literal_len 个字节到输出
    │
    ├─► 读 2字节 offset
    │
    ├─► 低4位 → match_len（+4，+额外字节）
    │
    └─► 从 dst-offset 处复制 match_len 个字节到输出
            （允许重叠，即 RLE 编码情形）
```

## 3. IDA 反汇编分析：压缩函数

### 3.1 sub_10053F50 分析（LZ4 压缩核心，5142 字节）

**函数基本信息**：

| 属性 | 值 |
|------|-----|
| 地址 | `0x10053F50` |
| 大小 | 5142 字节（IDA 分析结果） |
| 调用约定 | `__thiscall`（`this` 在 ECX） |
| 参数 | `(void* src, size_t src_len, void* dst, size_t dst_capacity)` |

**关键反汇编片段解读**：

```asm
; ── 初始化阶段 ──────────────────────────────────────────────────────────────
; sub_10053F50 入口
push  ebp
mov   ebp, esp
push  ebx
push  esi
push  edi
sub   esp, 10h              ; 分配局部变量空间
mov   esi, ecx              ; this 指针（LZ4 上下文）

; ── 检查字典状态 ─────────────────────────────────────────────────────────────
cmp   dword ptr [esi+4000h], 0   ; ← 偏移 0x4000: 字典是否已初始化？
jnz   short loc_XXXXX            ; 已初始化则跳过初始化代码

; ── 窗口大小设置 ─────────────────────────────────────────────────────────────
cmp   ecx, 10000h           ; 当前数据 >= 64KB 滑动窗口上限？
jb    short loc_XXXXX
mov   dword ptr [esi+4010h], 10000h  ; 设置窗口上限 = 0x10000 (64KB)

; ── 核心哈希匹配 ─────────────────────────────────────────────────────────────
mov   eax, [esi+ecx]        ; 读4字节输入数据
imul  eax, 9E3779B1h        ; Fibonacci 哈希
shr   eax, 14h              ; >> 20 → 12位索引
; eax 现在是哈希表槽索引（0-4095）
```

**为什么哈希表有 4096 个槽？**

`shr eax, 14h` 即右移 20 位，保留高 12 位作为索引。
`2^12 = 4096`，这与 LZ4 官方实现的 `LZ4_HASHLOG = 12`、
`HASH_SIZE_U32 = (1U << 12)` 完全一致。

```c
// LZ4 官方源码对应代码（lz4.c）
#define LZ4_HASHLOG   12
#define HASHTABLESIZE (1 << LZ4_HASHLOG)  // = 4096

static uint32_t LZ4_hash4(uint32_t sequence) {
    return (sequence * 2654435761U) >> (32 - LZ4_HASHLOG);
    // 2654435761 = 0x9E3779B1（完全一致！）
}
```

### 3.2 sub_100553A0 分析（LZ4 解压核心，2052 字节）

**关键反汇编片段**：

```asm
; sub_100553A0 - LZ4 解压核心
; ── 读 token ────────────────────────────────────────────────────────────────
movzx eax, byte ptr [esi]   ; 读 token 字节
inc   esi                   ; src 前进
mov   [ebp+token], al       ; 保存 token

; ── 提取 literal 长度 ────────────────────────────────────────────────────────
mov   ecx, eax
shr   ecx, 4                ; ecx = token >> 4 = literal_len（高4位）

; ── 提取 match 长度 ──────────────────────────────────────────────────────────
and   eax, 0Fh              ; eax = token & 0x0F = match_len_field（低4位）
; 注意：实际 match_len = eax + 4（最小匹配为4字节）

; ── 检查是否需要额外字节（literal_len == 15）────────────────────────────────
cmp   ecx, 0Fh
jnz   short skip_extra_lit
; 进入 extra length 读取循环
extra_lit_loop:
    movzx edx, byte ptr [esi]
    inc   esi
    add   ecx, edx
    cmp   edx, 0FFh          ; 继续直到非 255
    jz    short extra_lit_loop

skip_extra_lit:
; ecx 现在是完整的 literal_len

; ── 复制 literal ─────────────────────────────────────────────────────────────
rep   movsb                  ; 从 esi 复制 ecx 字节到 edi（dst）
```

**关键指令 `and eax, 0Fh` 的意义**：

这行代码从 token 字节中提取低 4 位，即 match 长度字段。
这是 LZ4 格式的核心操作，也是 IDA 分析中最能确认"这是 LZ4"的特征之一：

```
token = 0b1010_0011
             ↑↑↑↑ ← and 0x0F → 0b0011 = 3 → match_len = 3 + 4 = 7
        ↑↑↑↑     ← shr 4    → 0b1010 = 10 → literal_len = 10
```

## 4. LZ4 流格式：ArchiveStreamBuf 的读写封装

### 4.1 LZ4 Frame Format 标准

LZ4 官方定义了 **LZ4 Frame Format**，这是 `lz4frame.h` 使用的格式（也是 `lz4` 命令行工具产生的格式）：

```
LZ4 Frame 结构：
┌──────────────┬──────────────┬──────────────────┬──────────┬──────────────┐
│ Magic Number │ Frame Descr. │  Data Blocks...  │ EndMark  │  C. Checksum │
│  (4 bytes)   │  (3-15 B)    │  (variable)      │ (4 bytes)│   (4 bytes)  │
└──────────────┴──────────────┴──────────────────┴──────────┴──────────────┘

Magic Number = 0x184D2204（小端）

Frame Descriptor：
┌─────┬─────┬────────────────┬──────────────────────────┬──────┐
│ FLG │ BD  │ Content Size   │        Reserved          │  HC  │
│(1B) │(1B) │  (0 or 8 B)    │  (如果 Content Size=0)   │ (1B) │
└─────┴─────┴────────────────┴──────────────────────────┴──────┘

FLG 字节：
  bit 7-6: 版本号（必须 = 01）
  bit 5:   B.Indep（块独立模式）= 1 则各块独立，= 0 则连续（有字典依赖）
  bit 4:   B.Checksum（块校验）= 1 则每块后有 xxHash32
  bit 3:   C.Size（内容大小）= 1 则帧描述符含 8 字节内容大小
  bit 2:   C.Checksum（内容校验）= 1 则帧末有 xxHash32
  bit 1:   保留（必须 = 0）
  bit 0:   DictID（字典ID）= 1 则帧描述符含 4 字节字典 ID

BD 字节：
  bit 6-4: 块最大大小
    010 = 64KB (0x40000 bytes)   ← PackinOne 使用（cmp ecx, 10000h）
    011 = 256KB
    100 = 1MB
    101 = 4MB

HC 字节：头部校验 = xxHash32(FLG + BD [+ Content Size] [+ DictID]) >> 8 & 0xFF

Data Block：
┌────────────────────────────┬─────────────────────────────┐
│    Block Size (4 bytes)    │     Block Data              │
│ bit31=1: 不压缩（raw）      │ (Block Size & 0x7FFFFFFF B) │
│ bit31=0: 已压缩             │                             │
└────────────────────────────┴─────────────────────────────┘

EndMark = 0x00000000（4字节零值）
```

### 4.2 PackinOne 中的流类层次

从 IDA RTTI 分析中还原的类继承关系：

```
std::streambuf
```

### 4.3 用标准 C++ 实现兼容 LZ4 流

以下代码使用 `lz4frame.h`（LZ4 Frame API）实现，与 PackinOne 使用的帧格式兼容：

```cpp
// lz4_stream.hpp — PackinOne 风格的 LZ4 流封装
#include <cstdint>
#include <vector>
#include <stdexcept>
#include <string>
#include <lz4frame.h>   // 需要 vcpkg install lz4

class LZ4CompressStream {
public:
    // block_size 对应 PackinOne 的 cmp ecx, 10000h（64KB 块）
    explicit LZ4CompressStream(size_t block_size = 64 * 1024)
        : block_size_(block_size)
    {
        LZ4F_errorCode_t err = LZ4F_createCompressionContext(&ctx_, LZ4F_VERSION);
        if (LZ4F_isError(err)) {
            throw std::runtime_error(
                std::string("LZ4F_createCompressionContext: ") + LZ4F_getErrorName(err));
        }
        
        // 配置帧参数（与 PackinOne 的 cmp ecx, 10000h 匹配）
        prefs_ = LZ4F_INIT_PREFERENCES;
        prefs_.frameInfo.blockSizeID   = LZ4F_max64KB;   // 64KB 块
        prefs_.frameInfo.blockMode     = LZ4F_blockLinked; // 块间有字典依赖（连续模式）
        prefs_.frameInfo.contentChecksumFlag = LZ4F_contentChecksumEnabled;
        prefs_.compressionLevel = 0;  // 默认速度优先（LZ4 fast）
    }
    
    ~LZ4CompressStream() {
        LZ4F_freeCompressionContext(ctx_);
    }
    
    // 压缩数据，返回压缩后的字节
    std::vector<uint8_t> compress(const void* src, size_t src_size) {
        // 最坏情况下的输出大小（LZ4F_compressBound 计算帧开销 + 数据）
        size_t dst_cap = LZ4F_compressFrameBound(src_size, &prefs_);
        std::vector<uint8_t> dst(dst_cap);
        
        // 一次性压缩整帧（最简单方式，包含帧头+数据块+帧尾）
        size_t written = LZ4F_compressFrame(
            dst.data(), dst.size(),
            src, src_size,
            &prefs_
        );
        
        if (LZ4F_isError(written)) {
            throw std::runtime_error(
                std::string("LZ4F_compressFrame: ") + LZ4F_getErrorName(written));
        }
        
        dst.resize(written);  // 收缩到实际大小
        return dst;
    }
    
private:
    LZ4F_compressionContext_t ctx_;
    LZ4F_preferences_t prefs_;
    size_t block_size_;
};

class LZ4DecompressStream {
public:
    LZ4DecompressStream() {
        LZ4F_errorCode_t err = LZ4F_createDecompressionContext(&ctx_, LZ4F_VERSION);
        if (LZ4F_isError(err)) {
            throw std::runtime_error(
                std::string("LZ4F_createDecompressionContext: ") + LZ4F_getErrorName(err));
        }
    }
    
    ~LZ4DecompressStream() {
        LZ4F_freeDecompressionContext(ctx_);
    }
    
    // 解压数据，dst_capacity 需足够大（通常是原始大小）
    std::vector<uint8_t> decompress(const void* src, size_t src_size,
                                     size_t dst_hint = 0) {
        // 如果没有提示，先用 src_size * 4 估算
        if (dst_hint == 0) dst_hint = src_size * 4;
        
        std::vector<uint8_t> dst(dst_hint);
        size_t dst_size = dst.size();
        size_t src_pos = 0;
        
        while (src_pos < src_size) {
            size_t dst_written = dst_size;
            size_t src_read = src_size - src_pos;
            
            size_t result = LZ4F_decompress(
                ctx_,
                dst.data() + (dst_hint - dst_size), &dst_written,
                (const char*)src + src_pos, &src_read,
                nullptr
            );
            
            if (LZ4F_isError(result)) {
                throw std::runtime_error(
                    std::string("LZ4F_decompress: ") + LZ4F_getErrorName(result));
            }
            
            src_pos  += src_read;
            dst_size -= dst_written;
            
            if (result == 0) break;  // 解压完成
        }
        
        dst.resize(dst_hint - dst_size);  // 收缩到实际写入量
        return dst;
    }

private:
    LZ4F_decompressionContext_t ctx_;
};
```

### 5.1 用原始块 API（lz4.h）实现 PackinOne 风格压缩

这是最接近 `sub_10053F50` / `sub_100553A0` 内部实现的方式（PackinOne 直接调用 LZ4 块级 API，
然后 `ArchiveStreamBuf` 负责帧封装）：

```cpp
// example1_lz4_block.cpp — 原始块 API，对应 PackinOne 内部行为
// 编译: g++ -std=c++17 -o example1 example1_lz4_block.cpp -llz4
#include <cstdint>
#include <cstring>
#include <iostream>
#include <vector>
#include <stdexcept>
#include <lz4.h>      // 原始块 API

// 模拟 PackinOne sub_10053F50 的调用方式
// 参数和内部逻辑参考 IDA 反汇编：5142 字节，64KB 滑动窗口
std::vector<uint8_t> packinone_compress_block(const uint8_t* src, int src_len) {
    // LZ4_COMPRESSBOUND 计算最坏情况输出大小
    // 公式：src_len + src_len/255 + 16
    int max_dst = LZ4_COMPRESSBOUND(src_len);
    std::vector<uint8_t> dst(max_dst);
    
    // LZ4_compress_default: 对应 PackinOne 压缩核心
    // 哈希表大小: HASHTABLESIZE = 4096 (hash >> 20 → 12位索引)
    // 滑动窗口: 64KB (cmp ecx, 10000h → mov [esi+4010h], 10000h)
    int compressed_size = LZ4_compress_default(
        (const char*)src,    // 输入数据
        (char*)dst.data(),   // 输出缓冲
        src_len,             // 输入大小
        max_dst              // 输出缓冲大小
    );
    
    if (compressed_size <= 0) {
        throw std::runtime_error("LZ4_compress_default 失败");
    }
    
    dst.resize(compressed_size);
    return dst;
}

// 模拟 PackinOne sub_100553A0 的调用方式
// 参数参考 IDA 反汇编：2052 字节，and eax, 0Fh 取 token 低4位
std::vector<uint8_t> packinone_decompress_block(const uint8_t* src, int src_len,
                                                  int original_size) {
    std::vector<uint8_t> dst(original_size);
    
    // LZ4_decompress_safe: 对应 PackinOne 解压核心
    // 内部即为 sub_100553A0 中 token 解析 + literal 复制 + match 复制的流程
    int decompressed_size = LZ4_decompress_safe(
        (const char*)src,    // 压缩数据
        (char*)dst.data(),   // 解压输出
        src_len,             // 压缩数据大小
        original_size        // 原始数据大小（用于边界检查）
    );
    
    if (decompressed_size < 0) {
        throw std::runtime_error("LZ4_decompress_safe 失败（数据损坏？）");
    }
    if (decompressed_size != original_size) {
        throw std::runtime_error("解压后大小与预期不符");
    }
    
    return dst;
}

int main() {
    // 测试数据（模拟游戏资源文本——重复内容有利于压缩）
    const char* test_data =
        "KiriKiri engine resource block\n"
        "KiriKiri engine resource block\n"
        "KiriKiri engine resource block\n"
        "PackinOne LZ4 compression test\n";
    int test_len = (int)strlen(test_data);
    
    std::cout << "原始大小: " << test_len << " 字节\n";
    
    // 压缩
    auto compressed = packinone_compress_block(
        (const uint8_t*)test_data, test_len);
    std::cout << "压缩后大小: " << compressed.size() << " 字节\n";
    std::cout << "压缩率: "
              << (100.0 * compressed.size() / test_len) << "%\n";
    
    // 解压
    auto decompressed = packinone_decompress_block(
        compressed.data(), (int)compressed.size(), test_len);

    std::cout << "解压验证: " << (ok ? "✓ 通过" : "✗ 失败") << "\n";
    
    return ok ? 0 : 1;
}
```

### 5.2 用帧 API（lz4frame.h）实现流式压缩

帧 API 处理帧头/数据块/帧尾的封装，对应 `ArchiveStreamBuf` 的流封装层：

```cpp
// example2_lz4_frame.cpp — 帧 API，对应 ArchiveStreamBuf 的封装层
// 编译: g++ -std=c++17 -o example2 example2_lz4_frame.cpp -llz4
#include <cstdio>
#include <cstring>
#include <iostream>
#include <vector>
#include <lz4frame.h>

// 流式压缩：模拟 LZ4CompressStream 写入逻辑
// 对应 PackinOne 中: ArchiveStreamBuf 写入时调用 sub_10053F50 + 帧封装
std::vector<uint8_t> stream_compress(const void* src, size_t src_size) {
    LZ4F_compressionContext_t ctx;
    LZ4F_createCompressionContext(&ctx, LZ4F_VERSION);
    
    // 帧配置（与 PackinOne 分析结果匹配）
    LZ4F_preferences_t prefs = LZ4F_INIT_PREFERENCES;
    prefs.frameInfo.blockSizeID          = LZ4F_max64KB;   // 64KB 块（与 0x10000 匹配）
    prefs.frameInfo.blockMode            = LZ4F_blockLinked; // 块间字典（连续流模式）
    prefs.frameInfo.contentChecksumFlag  = LZ4F_contentChecksumEnabled; // xxHash32 校验
    
    // 预分配输出缓冲
    std::vector<uint8_t> out_buf(LZ4F_compressBound(src_size, &prefs));
    size_t out_pos = 0;
    
    // 1. 写帧头（Magic + FLG + BD + HC）
    size_t header_size = LZ4F_compressBegin(ctx,
        out_buf.data() + out_pos,
        out_buf.size() - out_pos,
        &prefs);
    if (LZ4F_isError(header_size)) {
        LZ4F_freeCompressionContext(ctx);
        throw std::runtime_error(LZ4F_getErrorName(header_size));
    }
    out_pos += header_size;
    
    // 2. 按块写入数据（每块最多 64KB）
    const size_t CHUNK = 64 * 1024;  // 与 PackinOne 的 0x10000 匹配
    const uint8_t* in = (const uint8_t*)src;
    for (size_t in_pos = 0; in_pos < src_size; in_pos += CHUNK) {
        size_t chunk = std::min(CHUNK, src_size - in_pos);
        size_t written = LZ4F_compressUpdate(ctx,
            out_buf.data() + out_pos,
            out_buf.size() - out_pos,
            in + in_pos,
            chunk,
            nullptr);
        if (LZ4F_isError(written)) {
            LZ4F_freeCompressionContext(ctx);
            throw std::runtime_error(LZ4F_getErrorName(written));
        }
        out_pos += written;
    }
    
    // 3. 写帧尾（EndMark + Content Checksum）
    size_t tail_size = LZ4F_compressEnd(ctx,
        out_buf.data() + out_pos,
        out_buf.size() - out_pos,
        nullptr);
    if (LZ4F_isError(tail_size)) {
        LZ4F_freeCompressionContext(ctx);
        throw std::runtime_error(LZ4F_getErrorName(tail_size));
    }
    out_pos += tail_size;
    
    LZ4F_freeCompressionContext(ctx);
    out_buf.resize(out_pos);
    return out_buf;
}

// 解压帧格式数据（对应 LZ4DecompressStream 读取逻辑）
std::vector<uint8_t> stream_decompress(const void* src, size_t src_size) {
    LZ4F_decompressionContext_t ctx;
    LZ4F_createDecompressionContext(&ctx, LZ4F_VERSION);
    
    std::vector<uint8_t> out;
    out.reserve(src_size * 3);  // 预估3倍空间
    
    std::vector<uint8_t> tmp(64 * 1024);  // 64KB 临时缓冲（与块大小匹配）
    const uint8_t* in = (const uint8_t*)src;
    size_t in_pos = 0;
    
    while (in_pos < src_size) {
        size_t dst_size = tmp.size();
        size_t src_read = src_size - in_pos;
        
        size_t result = LZ4F_decompress(ctx,
            tmp.data(), &dst_size,
            in + in_pos, &src_read,
            nullptr);
        
        if (LZ4F_isError(result)) {
            LZ4F_freeDecompressionContext(ctx);
            throw std::runtime_error(LZ4F_getErrorName(result));
        }
        
        // 追加到输出
        out.insert(out.end(), tmp.begin(), tmp.begin() + dst_size);
        in_pos += src_read;
        
        if (result == 0) break;  // 帧结束标志
    }
    
    LZ4F_freeDecompressionContext(ctx);
    return out;
}

int main() {
    // 生成测试数据（2MB，含重复模式）
    std::vector<uint8_t> test_data(2 * 1024 * 1024);
    for (size_t i = 0; i < test_data.size(); i++) {
        // 模拟游戏资源中典型的重复模式（如纯色图像区域）
        test_data[i] = (uint8_t)(i % 256 ^ (i / 256 % 128));
    }
    
    std::cout << "测试数据大小: " << test_data.size() / 1024 << " KB\n";
    
    auto compressed = stream_compress(test_data.data(), test_data.size());
    std::cout << "压缩后大小: " << compressed.size() / 1024 << " KB\n";
    std::cout << "压缩率: "
              << (100.0 * compressed.size() / test_data.size()) << "%\n";
    
    // 验证帧格式魔数（小端 0x184D2204）
    uint32_t magic;
    memcpy(&magic, compressed.data(), 4);
    printf("帧魔数: 0x%08X (期望 0x184D2204: %s)\n",
           magic, (magic == 0x184D2204) ? "✓" : "✗");
    
    auto decompressed = stream_decompress(compressed.data(), compressed.size());
    bool ok = (decompressed == test_data);
    std::cout << "压缩-解压往返验证: " << (ok ? "✓ 通过" : "✗ 失败") << "\n";
    
    return ok ? 0 : 1;
}
```

## 6. SHA-256 在 PackinOne 中的作用

### 6.1 sub_10041B50 分析（7519 字节，SHA-256 压缩函数）

PackinOne 内嵌了完整的 SHA-256 实现。IDA 分析中这是**单个最大函数**（7519 字节），
远大于任何其他函数，说明是经过内联优化的版本。

**关键证据（来自 packinone_analysis.txt）**：

```asm
; sub_10041B50 - SHA-256 核心压缩函数（7519 字节）
; ── 初始哈希常量验证 ──────────────────────────────────────────────────────────
; SHA-256 标准初始哈希值 H[0..7]：
;   H[0] = 0x6A09E667
;   H[1] = 0xBB67AE85
;   H[2] = 0x3C6EF372
;   H[3] = 0xA54FF53A
;   H[4] = 0x510E527F    ← ← IDA 发现此常量，与标准完全一致
;   H[5] = 0x9B05688C
;   H[6] = 0x1F83D9AB
;   H[7] = 0x5BE0CD19

; IDA 发现的常量（直接引用）
mov  eax, 510E527Fh      ; H[4] 初始值，SHA-256 标准常量
; ...
; 16 个 DWORD 的消息调度数组 W[0..15]
; SIMD 指令：movdqa, punpcklwd（SSE2 加速并行处理）
```

SHA-256 的数学基础（简要）：

```
SHA-256 处理流程：
1. 消息填充：在消息末尾追加 1 bit + 若干 0 bit + 64位消息长度，使总长度 ≡ 448 (mod 512)
2. 分块：将填充后的消息按 512 bit（64字节）分块
3. 对每个 512-bit 块执行压缩函数：
   a. 消息调度：将 16 个 32-bit 字扩展为 64 个字（W[0..63]）
   b. 64 轮哈希计算（使用 Ch、Maj、Σ0、Σ1 四个辅助函数）
   c. 更新 H[0..7]
4. 最终将 H[0..7] 拼接为 256-bit（32字节）摘要
```

### 6.2 常量验证：0x510E527F 确认是 SHA-256

### 6.3 SIMD 优化特征

IDA 分析中在 `sub_10041B50` 内部发现了 SSE2 指令（`movdqa`、`punpcklwd`），
说明 PackinOne 使用了 **Intel SHA-NI 或手写 SSE2 向量化**来加速 SHA-256 计算。

```asm
; sub_10041B50 内部的 SIMD 片段
movdqa  xmm0, xmmword ptr [esi+eax*4]  ; 加载 4 个 DWORD（128位）到 XMM 寄存器
punpcklwd xmm1, xmm2                    ; 低字解包（处理 SHA-256 消息调度）
; 这对应 SHA-256 消息调度中的：
; W[i] = σ1(W[i-2]) + W[i-7] + σ0(W[i-15]) + W[i-16]
; 其中 σ0, σ1 是移位/旋转操作，可以向量化并行处理
```

### 6.4 SHA-256 在 PackinOne 中的可能用途

根据 IDA 分析和 PackinOne 的功能定位，SHA-256 最可能用于：

| 用途 | 可能性 | 依据 |
|------|--------|------|
| 压缩数据完整性校验 | ★★★★☆ | 压缩解压失败时需要检测数据损坏 |
| 加密密钥派生 | ★★★☆☆ | 配合 BufferedStreamCryptFilter |
| 文件哈希/去重 | ★★☆☆☆ | StoragesFstat 可能需要文件指纹 |
| 数字签名验证 | ★☆☆☆☆ | 游戏资源通常不需要签名 |

**结论**：最可能是压缩流的**内容完整性校验**（对应 LZ4 Frame Format 中的 Content Checksum 字段
通常是 xxHash32，但 PackinOne 可能选用安全性更强的 SHA-256）。

```cpp
// 演示如何用 SHA-256 为 LZ4 压缩数据添加完整性校验
// （这是对 PackinOne 内部行为的推断演示，非精确还原）
#include <openssl/sha.h>   // 或使用其他 SHA-256 实现

struct PackinOneCompressedBlock {
    uint32_t original_size;      // 原始大小
    uint32_t compressed_size;    // 压缩后大小
    uint8_t  sha256[32];         // 原始数据的 SHA-256（32字节）
    uint8_t  data[];             // 压缩数据（flexible array）
};

// 创建带校验的压缩块（推断的 PackinOne 行为）
void create_verified_block(const uint8_t* src, size_t src_size,
                            std::vector<uint8_t>& out) {
    // 1. 计算原始数据的 SHA-256
    uint8_t hash[32];
    SHA256(src, src_size, hash);
    
    // 2. LZ4 压缩
    int max_compressed = LZ4_COMPRESSBOUND((int)src_size);
    std::vector<uint8_t> compressed(max_compressed);
    int comp_size = LZ4_compress_default(
        (const char*)src, (char*)compressed.data(),
        (int)src_size, max_compressed);
    compressed.resize(comp_size);
    
    // 3. 打包：header + hash + compressed data
    out.resize(sizeof(uint32_t) * 2 + 32 + comp_size);
    uint8_t* p = out.data();
    memcpy(p, &src_size, 4);   p += 4;  // original_size
    memcpy(p, &comp_size, 4);  p += 4;  // compressed_size
    memcpy(p, hash, 32);       p += 32; // SHA-256
    memcpy(p, compressed.data(), comp_size);  // data
}
```

### 7.2 ncbind 风格的 C++ 插件实现

以下是用 KiriKiri ncbind 框架实现 LZ4 压缩 TJS 接口的完整代码：

```cpp
// lz4_plugin.cpp — PackinOne 风格的 LZ4 压缩插件（ncbind 版）
// 放置路径: cpp/plugins/lz4compress/lz4_plugin.cpp
#include <ncbind/ncbind.hpp>
#include <lz4.h>
#include <lz4frame.h>
#include <vector>
#include <cstring>
#include <stdexcept>

// ── LZ4 块压缩（对应 sub_10053F50 调用方式）────────────────────────────────
// TJS: var result = LZ4.compress(data)
// 参数: data — tTJSVariant (octet 类型)
// 返回: octet 字节数组（4字节原始大小 + N字节压缩数据）
tTJSVariant tjs_lz4_compress(tTJSVariant* data) {
    if (data->Type() != tvtOctet) {
        TVPThrowExceptionMessage(TJS_W("LZ4.compress: 参数必须是 octet 类型"));
    }
    
    const tTJSOctetBlock* octet = data->AsOctetNoAddRef();
    const uint8_t* src = octet->GetData();
    int src_len = (int)octet->GetLength();
    
    // LZ4 压缩（对应 PackinOne 内部调用流程）
    int max_dst = LZ4_COMPRESSBOUND(src_len);
    std::vector<uint8_t> buf(4 + max_dst);  // 头部4字节存原始大小
    
    // 小端序写入原始大小（解压时需要）
    buf[0] = (src_len >> 0)  & 0xFF;
    buf[1] = (src_len >> 8)  & 0xFF;
    buf[2] = (src_len >> 16) & 0xFF;
    buf[3] = (src_len >> 24) & 0xFF;
    
    int comp_size = LZ4_compress_default(
        (const char*)src,
        (char*)buf.data() + 4,
        src_len,
        max_dst
    );
    
    if (comp_size <= 0) {
        TVPThrowExceptionMessage(TJS_W("LZ4.compress: 压缩失败"));
    }
    
    buf.resize(4 + comp_size);
    
    // 返回 octet
    return tTJSVariant(buf.data(), buf.size());
}

// ── LZ4 块解压（对应 sub_100553A0 调用方式）────────────────────────────────
// TJS: var result = LZ4.decompress(data)
// 参数: data — tTJSVariant (octet 类型，头部含4字节原始大小)
// 返回: octet 字节数组（解压后的原始数据）
tTJSVariant tjs_lz4_decompress(tTJSVariant* data) {
    if (data->Type() != tvtOctet) {
        TVPThrowExceptionMessage(TJS_W("LZ4.decompress: 参数必须是 octet 类型"));
    }
    
    const tTJSOctetBlock* octet = data->AsOctetNoAddRef();
    const uint8_t* src = octet->GetData();
    int src_len = (int)octet->GetLength();
    
    if (src_len < 4) {
        TVPThrowExceptionMessage(TJS_W("LZ4.decompress: 数据太短（缺少头部）"));
    }
    
    // 读取原始大小（小端序）
    int original_size = src[0] | (src[1] << 8) | (src[2] << 16) | (src[3] << 24);
    
    if (original_size <= 0 || original_size > 256 * 1024 * 1024) {
        TVPThrowExceptionMessage(TJS_W("LZ4.decompress: 无效的原始大小"));
    }
    
    std::vector<uint8_t> dst(original_size);
    
    // LZ4 解压（对应 sub_100553A0 中 token 解析流程）
    int result = LZ4_decompress_safe(
        (const char*)(src + 4),   // 跳过4字节头部
        (char*)dst.data(),
        src_len - 4,
        original_size
    );
    
    if (result != original_size) {
        TVPThrowExceptionMessage(TJS_W("LZ4.decompress: 解压失败或大小不匹配"));
    }
    
    return tTJSVariant(dst.data(), dst.size());
}

// ── 注册到 TJS ────────────────────────────────────────────────────────────
// 使用 ncbind 将函数注册到 TJS "LZ4" 命名空间
NCB_REGISTER_FUNCTION(TJS_W("LZ4"), TJS_W("compress"),   tjs_lz4_compress);
NCB_REGISTER_FUNCTION(TJS_W("LZ4"), TJS_W("decompress"), tjs_lz4_decompress);
```

### 8.4 常见错误及解决方案

| 错误现象 | 原因 | 解决方案 |
|----------|------|----------|
| `LZ4_decompress_safe` 返回负值 | 压缩数据损坏或格式不正确 | 检查数据来源；确认使用 `LZ4_compress_default` 而非 frame API 压缩 |
| `LZ4F_isError(written)` 为真 | Frame API 缓冲区太小 | 使用 `LZ4F_compressBound()` 而非手动估算缓冲区大小 |
| 解压后大小不匹配 | 存储的原始大小错误 | 压缩前确保正确记录 `original_size`（小端序） |
| 帧魔数错误 | 误用 block API 数据调用 frame 解压 | block API 和 frame API 的格式不兼容，需统一使用一种 |
| 跨平台字节序问题 | 大端序机器上小端序读写错误 | 使用显式的字节序转换（`htole32`/`le32toh`） |

## 本节小结

- **PackinOne 使用 LZ4**（非 LZSS/zlib），由 IDA 分析中 `0x9E3779B1` 哈希常量、
  `shr eax, 14h` 指令、64KB 窗口（`0x10000`）三重证据确认
- **LZ4 核心格式**：sequence = token(1B) + extra_literal(0-NB) + literals + offset(2B) + extra_match(0-NB)
  - token 高4位 = literal 长度；低4位 = match 长度 - 4
  - 长度字段 = 15 时读额外字节（每字节加到长度，直到非 255）
- **压缩函数** `sub_10053F50`（5142 字节）使用 4096 槽哈希表 + 64KB 滑动窗口
- **解压函数** `sub_100553A0`（2052 字节）顺序解析 sequence，`and eax, 0Fh` 取 token 低4位
- **流封装**：`ArchiveStreamBuf → LZ4StreamBase → LZ4CompressStream/LZ4DecompressStream`
  对应 LZ4 Frame Format（魔数 `0x184D2204`，64KB 块，内容校验）
- **SHA-256**（`sub_10041B50`，7519 字节）由常量 `0x510E527F`（H[4]）确认，含 SSE2 优化，
  最可能用于压缩数据的完整性校验

### 题目 1：识别 LZ4 token

给定以下字节序列（LZ4 压缩数据的第一个 sequence）：

```
字节序列（十六进制）：
74 4B 69 72 69 4B 69 72 69 20 00 00 05 00

token = 0x74
随后若干字节...
offset（小端）= 0x0005
```

问题：
1. token `0x74` 表示 literal_len 和 match_len_field 各是多少？
2. 是否需要读取额外的长度字节？
3. literal 段有多少字节？match 最终长度是多少？

<details>
<summary>查看答案</summary>

**分析过程**：

```
token = 0x74 = 0b0111_0100

高4位 = 0x74 >> 4 = 0x7 = 7   → literal_len = 7（< 15，无需额外字节）
低4位 = 0x74 & 0F = 0x4 = 4   → match_len_field = 4（< 15，无需额外字节）
```

1. `literal_len = 7`，`match_len_field = 4`
2. 两者都 < 15，**不需要**读取额外字节
3. literal 段 = 7 字节（即 `4B 69 72 69 4B 69 72`，即 "KiriKir"）
4. match 最终长度 = match_len_field + 4 = 4 + 4 = **8 字节**
   - 从 `offset = 5` 处（即 dst - 5）复制 8 字节
   - 因为 8 > 5，会出现重叠复制（RLE 效果）：复制 "KiriKiri"

**验证代码**：

```python
token = 0x74
lit_len = token >> 4
mat_field = token & 0x0F
mat_len = mat_field + 4

print(f"literal_len = {lit_len}")        # 7
print(f"match_len   = {mat_len}")        # 8
print(f"需要额外字节: {lit_len == 15 or mat_field == 15}")  # False

# 读取 literal（token 之后的 7 字节）
literal = bytes([0x4B, 0x69, 0x72, 0x69, 0x4B, 0x69, 0x72])
print(f"literal = {literal}")            # b'KiriKir'

# 读取 offset（小端 2 字节）
offset = 0x0005
print(f"offset = {offset}")             # 5（从当前位置回退5字节）
```

### 题目 2：实现 LZ4 格式验证器

编写一个 C++ 函数，判断给定的字节数组是否是有效的 LZ4 Frame 格式数据
（即检查魔数 `0x184D2204` 和帧描述符的版本字段）。

<details>
<summary>查看答案</summary>

```cpp
// lz4_validate.cpp — LZ4 Frame 格式验证器
// 编译: g++ -std=c++17 -o validate lz4_validate.cpp
#include <cstdint>
#include <cstring>
#include <iostream>
#include <vector>

// LZ4 Frame 魔数
constexpr uint32_t LZ4F_MAGIC = 0x184D2204u;

// LZ4 Frame 验证结果
enum class LZ4ValidResult {
    OK,                    // 有效
    TOO_SHORT,             // 数据太短
    WRONG_MAGIC,           // 魔数不符
    WRONG_VERSION,         // FLG 版本字段不符（必须 = 01）
    RESERVED_BIT_SET,      // FLG 保留位被设置（必须 = 0）
};

LZ4ValidResult validate_lz4_frame(const uint8_t* data, size_t size) {
    // 至少需要 7 字节（4B魔数 + 1B FLG + 1B BD + 1B HC）
    if (size < 7) {
        return LZ4ValidResult::TOO_SHORT;
    }
    
    // 检查魔数（小端序）
    uint32_t magic;
    memcpy(&magic, data, 4);
    if (magic != LZ4F_MAGIC) {
        return LZ4ValidResult::WRONG_MAGIC;
    }
    
    // 检查 FLG 字节（data[4]）
    uint8_t flg = data[4];
    
    // bit 7-6: 版本（必须 = 01 = 0b01xxxxxx）
    uint8_t version = (flg >> 6) & 0x03;
    if (version != 1) {
        return LZ4ValidResult::WRONG_VERSION;
    }
    
    // bit 1: 保留位（必须 = 0）
    if (flg & 0x02) {
        return LZ4ValidResult::RESERVED_BIT_SET;
    }
    
    return LZ4ValidResult::OK;
}

int main() {
    // 测试 1：构造一个有效的 LZ4 帧头
    std::vector<uint8_t> valid_header = {
        0x04, 0x22, 0x4D, 0x18,  // 魔数 0x184D2204（小端）
        0x64,                     // FLG: version=01, B.Indep=1, B.Cksum=0, C.Size=0, C.Cksum=1
        0x40,                     // BD: blockSizeID=4 (64KB)
        0xA7,                     // HC: 头部校验
    };
    
    auto r1 = validate_lz4_frame(valid_header.data(), valid_header.size());
    std::cout << "有效帧头: " << (r1 == LZ4ValidResult::OK ? "✓ OK" : "✗ 无效") << "\n";
    
    // 测试 2：错误的魔数
    std::vector<uint8_t> wrong_magic = {
        0xFF, 0xFF, 0xFF, 0xFF,  // 错误魔数
        0x64, 0x40, 0xA7,
    };
    auto r2 = validate_lz4_frame(wrong_magic.data(), wrong_magic.size());
    std::cout << "错误魔数: " << (r2 == LZ4ValidResult::WRONG_MAGIC ? "✓ 检测到" : "✗ 漏检") << "\n";
    
    // 测试 3：用 lz4frame.h 生成真实帧，再验证
    // 此处需要链接 -llz4
    return 0;
}
```

### 题目 3：对照 IDA 分析，验证 Fibonacci 哈希常量

在 IDA 分析结果中发现 `imul eax, 9E3779B1h; shr eax, 14h`。

问题：
1. 为什么 `0x9E3779B1` 被称为 "Fibonacci 哈希常量"？它是怎么算出来的？
2. `shr eax, 14h` 右移 20 位后，哈希表有多少个槽？
3. 如果 PackinOne 使用的是 LZ4 HC（高压缩）模式，哈希表大小会有什么不同？

<details>
<summary>查看答案</summary>

**第1题：Fibonacci 哈希常量的来源**

```python
import math

phi = (1 + math.sqrt(5)) / 2   # 黄金比例 ≈ 1.6180339887
# 2^32 / phi ≈ 2654435769.497...
# 取近似整数: 2654435761 = 0x9E3779B1（奇数）
fibonacci_hash = int(2**32 / phi)
print(f"2^32 / φ ≈ {fibonacci_hash} = 0x{fibonacci_hash:08X}")
# 输出: 2654435761 = 0x9E3779B1

# 为什么是奇数？因为偶数乘以奇数的结果会损失低位信息
# 黄金比例保证哈希值的分布尽可能均匀（Knuth 的乘法哈希理论）
```

**第2题：哈希表槽数**

```
shr eax, 14h = 右移 20 位

保留 32 - 20 = 12 位作为索引
2^12 = 4096 个槽

对应 LZ4 源码中:
  #define LZ4_HASHLOG   12
  #define HASHTABLESIZE (1U << LZ4_HASHLOG)  // = 4096
```

**第3题：LZ4 HC 模式的差异**

LZ4 HC（High Compression）模式不只用哈希表，还用链表存储所有哈希碰撞位置，
找到最长匹配而非最近匹配：

```c
// LZ4 fast 模式（PackinOne 使用）:
//   HASHTABLESIZE = 4096，只存最近一个匹配位置
//   速度优先，可能错过更好的匹配

// LZ4 HC 模式（高压缩）:
//   HASHTABLESIZE 更大（默认 16384+）
//   每个槽是链表，存多个候选位置
//   搜索最长匹配，压缩率更好但速度更慢
//   对应 LZ4_compress_HC() 函数

// PackinOne 选用 fast 模式（shr 14h → 4096 槽）
// 说明它优先考虑压缩速度（游戏加载时间优先）
```

</details>
