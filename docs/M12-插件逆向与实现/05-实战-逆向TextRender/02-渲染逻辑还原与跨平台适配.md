## 一、核心渲染函数深度分析

### 1.1 函数调用层次

通过 IDA 的交叉引用（Cross Reference，跟踪函数之间的调用关系），我们还原了 textrender.dll 的渲染调用链：

```
sub_1000DFE0  (8,379 字节) — 顶层渲染入口（来自 None 调用者 0x1002B66C, 0x1002C5F4）
  ├── sub_100121A0 (4,402 字节) — 逐字符渲染循环（被调用 3 次：0x1000FB4E, 0x1000FF33, 0x1000FFE5）
  │     └── sub_1000D110 (?) — 单字符光栅化
  ├── sub_100111D0 (2,671 字节) — 排版预处理（被 5 处调用）
  │     └── 计算文本布局、换行点、总高度
  ├── sub_10010260 (2,773 字节) — 行级渲染
  │     ├── sub_100111D0 — 复用排版预处理
  │     └── 逐行写入像素到目标缓冲区
  ├── sub_100134E0 (1,225 字节) — 逐行绘制到缓冲区（被调用 2 次：0x1000F8C0, 0x10010029）
  └── sub_10005980 (3,310 字节) — ncbind 注册（非渲染）

sub_1000DA70  (1,062 字节) — 并列渲染入口（来自 None 调用者 0x1002B668, 0x1002C5F0）
  └── 渲染状态初始化 / 重置（设置坐标、标志位等字段归零）

sub_10011Ce0  (1,087 字节) — 布局节点递归构建（递归调用自身，0x10011FD0）
  ├── operator new — 分配布局树节点（0x10011D28）
  └── sub_10012120 (?) — 节点初始化辅助
```

### 1.2 顶层渲染函数 sub_1000DFE0 分析

这是 textrender.dll 中最大的函数（8,379 字节，约 1,900 条 x86 指令）。从反汇编中提取的关键行为：

```asm
; === 入口：参数解析 ===
0x1000E020: mov     eax, [ebp+arg_0]      ; 参数 1：目标缓冲区或上下文指针
0x1000E025: cmp     [ebp+arg_C], edi      ; 参数 4：行间距（0 表示使用默认值）
0x1000E028: mov     esi, ecx              ; this 指针（TextRender 实例）
0x1000E032: jle     short loc_1000E041
0x1000E034: cmp     [ebp+arg_8], edi      ; 参数 3：检查是否为 0
0x1000E039: fld     ds:flt_1002C728       ; 加载浮点常量（默认行间距）
0x1000E041: fild    [ebp+arg_8]           ; 否则把整数参数转为浮点

; === 虚函数调用：获取渲染参数 ===
0x1000E044: mov     edx, [esi]            ; 虚表指针
0x1000E04A: mov     eax, [edx+0Ch]        ; 虚表第 4 槽（index 3）
0x1000E04D: call    eax                    ; 调用虚函数获取渲染参数

; === 容器遍历：逐字符处理 ===
0x1000E05B: mov     edi, [esi+1D0h]       ; vector::_Mylast（end 指针）
0x1000E063: cmp     edi, [esi+1D4h]       ; vector::_Myend（capacity 末端指针）
0x1000E070: mov     ebx, [esi+1C4h]       ; vector::_Myfirst（begin/data 指针）
```

从这段反汇编中可以推断出：

1. **参数传递**：渲染函数接受至少 4 个参数（`arg_0` 到 `arg_C`），其中 `arg_C` 控制行间距
2. **浮点运算密集**：`fld`（浮点加载）、`fild`（整数转浮点）、`fstp`（浮点存储）指令频繁出现
3. **通过虚表调用获取参数**：说明渲染参数是通过虚函数接口获取的，支持子类覆写
4. **遍历 std::vector**：`[esi+1C4h]`/`[esi+1D0h]`/`[esi+1D4h]` 对应 MSVC `std::vector` 的三个内部指针（`_Myfirst`/`_Mylast`/`_Myend`，即 begin/end/capacity_end），注意 MSVC 存储的是三个**原始指针**而非"data + size + capacity"三个值；`size = (_Mylast - _Myfirst) / sizeof(element)`

### 1.3 逐字符渲染循环 sub_100121A0

这个 4,402 字节的函数是核心渲染循环，被顶层函数调用 3 次（可能对应：正文渲染、描边渲染、阴影渲染）。关键模式：

```asm
; === 字符码提取 ===
0x10012239: movzx   ecx, word ptr [esi]   ; 读取 2 字节 Unicode 字符码
0x1001223C: push    ecx                    ; 传给光栅化函数
0x1001223D: lea     edx, [esp+120h+var_80]
0x10012245: call    sub_1000D110           ; 调用字形光栅化

; === 浮点坐标计算 ===
0x10012251: fld     [esp+11Ch+var_F0]     ; 加载 Y 坐标
0x10012263: fld     dword ptr [ebp+58h]    ; 加载字体大小/缩放
0x1001226C: fmul    dword ptr [ebp+8]      ; 乘以缩放系数
```

这里可以清楚地看到：
- **宽字符处理**：`movzx ecx, word ptr [esi]` 读取的是 16 位 Unicode 字符（`wchar_t`）
- **字形光栅化**：`sub_1000D110` 是单字符光栅化函数，接受字符码和输出缓冲区
- **坐标使用浮点数**：支持子像素精度的文字定位
- **缩放系数**：字体大小通过浮点乘法应用，说明支持任意字号

### 1.4 排版预处理 sub_100111D0

这个 2,671 字节的函数被多处调用（5 次交叉引用），负责在实际渲染前计算文本布局：

```asm
; === 初始化 ===
0x100111D9: fldz                          ; 加载 0.0（初始偏移量）
0x100111DF: mov     ebx, [eax+1A0h]       ; 文本行数（从对象偏移 +1A0h 读取）
0x100111E7: lea     edi, [eax+184h]       ; 排版参数结构体（偏移 +184h）

; === 字形度量读取 ===
0x10011211: fld     dword ptr [ecx+24h]   ; 读取字形结构 +0x24 处的浮点度量值
0x10011214: add     ecx, 24h              ; 步进到下一字段（字形结构元素大小 ≥ 0x24）

; === 渲染参数比对 ===
0x10011238: fld     dword ptr [ecx+0C8h]  ; 读取 TextRender 对象 +0xC8 处的参数
0x10011253: fcom    dword ptr [ecx+94h]   ; 与 +0x94 处的浮点参数比较
0x10011260: fld     dword ptr [ecx+9Ch]   ; 读取 +0x9C 坐标值
0x10011277: cmp     byte ptr [ecx+177h], 0 ; 检查 +0x177 标志字节
0x1001127E: mov     byte ptr [ecx+17Dh], 1 ; 设置 +0x17D 标志字节为 1
```

偏移量 `+184h` 处的数据结构可能包含：行间距、首行缩进、最大宽度、对齐方式等排版参数。`+1A0h` 存储的是文本行数（从 `mov ebx, [eax+1A0h]` 的使用方式可判断）。字形结构偏移 `+0x24` 处存储的是某个浮点度量值（从 `fld dword ptr [ecx+24h]` 结合后续的浮点运算可推断为 advance width 或 bearing 类度量）。

### 1.5 并列渲染入口 sub_1000DA70（真实发现）

IDA 交叉引用显示存在第二个渲染入口 `sub_1000DA70`（1,062 字节），其调用者地址（`0x1002B668`、`0x1002C5F0`）与 `sub_1000DFE0` 的调用者（`0x1002B66C`、`0x1002C5F4`）紧密相邻——这说明在导出的渲染接口（`None` 类型调用，即从跳转表或函数指针调用）中，两个函数**成对出现**。

从反汇编特征分析，`sub_1000DA70` 的功能是**渲染状态初始化/重置**：

```asm
; sub_1000DA70 — 渲染状态重置（推断，来自析构/初始化模式）
; 该函数设置 TextRender 对象内多个浮点坐标字段为 0 或初始值
; 涉及偏移：+0x9Ch（X 坐标）、+0xA0h（Y 坐标）、+0x164h（当前绘制 X）、+0x168h（当前绘制 Y）

; 与 sub_1000DFE0 的关系（推断）：
;   sub_1000DA70  → 渲染前准备（设置坐标原点、清空状态）
;   sub_1000DFE0  → 实际渲染（使用 sub_1000DA70 设置好的状态）
```

在重新实现时，这两个函数对应"先初始化渲染上下文，再执行渲染"的两阶段调用模式。

### 1.6 布局节点递归构建 sub_10011Ce0（真实发现）

`sub_10011Ce0`（1,087 字节）有一个特殊行为：它**递归调用自身**（在 `0x10011FD0` 处有 `call sub_10011Ce0` 指令）。同时，函数内部有 `operator new`（`0x10011D28: call ??2@YAPAXI@Z`）调用，说明它在堆上动态分配新节点。

```asm
; sub_10011Ce0 — 布局树节点递归构建（推断）
; 地址: 0x10011Ce0

0x10011D28: call    ??2@YAPAXI@Z           ; operator new — 分配布局树节点
0x10011D52: cmp     dword ptr [eax+50h], 0 ; 检查节点标志（+0x50h: 0 = 无斜体等效果）
0x10011D5E: mov     ecx, [eax+1ACh]        ; 读取字形计数（+0x1ACh）
0x10011FD0: call    sub_10011Ce0           ; 递归调用自身（处理子节点）
```

偏移 `+0x50h` 处的标志位与渲染属性相关（0 表示无斜体等修饰效果）；`+0x1ACh` 存储字形计数。这个函数很可能是**文本布局树**（Layout Tree）的节点构建函数，通过递归处理嵌套的文字排版结构（如：包含换行、描边、颜色变化的富文本）。

### 1.7 逐行绘制函数 sub_100134E0（真实发现）

`sub_100134E0`（1,225 字节）被 `sub_1000DFE0` 调用**两次**（地址 `0x1000F8C0`、`0x10010029`），内部操作与 `sub_10010260` 相似，使用相同的 vector 偏移（`[ebp+1D0h]`、`[ebp+1C4h]`），处理的是同一个对象的同一批字形数据。两次调用说明一次完整的文本渲染会执行两轮逐行绘制（可能对应：第一轮绘制描边/阴影，第二轮绘制正文）。

```asm
; sub_100134E0 — 逐行绘制（两轮渲染模式推断）
; 调用者：sub_1000DFE0（0x1000F8C0 第一轮，0x10010029 第二轮）

; 读取行数和 vector 数据（与 sub_10010260 相同的偏移）
; [ebp+20Ch]: 当前处理的行数/行索引
; [ebp+21Ch]: 某个浮点参数（行高或偏移量）
; [ebp+1D0h]: vector::_Mylast（end 指针）
; [ebp+1C4h]: vector::_Myfirst（begin/data 指针）
```

## 二、字形度量还原

### 2.1 什么是字形度量

字形度量（Glyph Metrics）是排版系统的基础数据。每个字符被渲染成位图后，需要以下参数来正确定位：

```
        ┌─── bearingX ───┐
        │                 │
        ▼                 │
   ···········           │
   ·  ██████ ·  ↑        │
   · ██    ██·  │        │
   · ██    ██·  │ height │
   · ████████·  │        │
   · ██    ██·  │        │
   · ██    ██·  ↓        │
───·─────────·──── baseline（基线）
   ·         ·
   ·         ·
   ←─── advance width ──→

   bearingX: 字形左边缘到原点的水平距离
   bearingY: 字形顶部到基线的垂直距离
   width:    字形位图的像素宽度
   height:   字形位图的像素高度
   advance:  到下一个字符原点的距离
```

### 2.2 从 IDA 推断度量结构

在 `sub_10010260`（行级渲染函数）中，我们看到频繁访问的偏移量：

```asm
0x10010293: fld     dword ptr [ebp+9Ch]   ; 读取 float 值
0x10010299: mov     eax, [ebp+1A0h]       ; 读取整数值
0x1001029F: fstp    dword ptr [ebp+164h]  ; 存储到另一个位置
0x100102A5: fld     dword ptr [ebp+0A0h]  ; 读取另一个 float
0x100102AB: fstp    dword ptr [ebp+168h]  ; 存储
```

这对浮点赋值（从 `+9Ch`/`+A0h` 复制到 `+164h`/`+168h`）很可能是将**原始坐标**复制到**当前绘制坐标**——也就是文本绘制的起始位置 (x, y)。

结合 `sub_100121A0` 中的字形循环，推断的字形度量结构：

```cpp
// 推断的字形度量结构（根据反汇编偏移还原）
struct GlyphInfo {
    float bearingX;     // +0x00: 水平偏移
    float bearingY;     // +0x04: 垂直偏移（到基线）
    float advance;      // +0x08: 前进宽度
    int   width;        // +0x0C: 位图宽度（像素）
    int   height;       // +0x10: 位图高度（像素）
    int   pitch;        // +0x14: 位图行字节数
    // +0x18: 可能是 SSO 优化的字符串或指针
    uint8_t* bitmap;    // 字形位图数据
};

// 反汇编中的 0x10012231: mov esi, [esi+4]
// 和 0x1001222B: cmp dword ptr [esi+18h], 8
// 表明存在 SSO（Small String Optimization）检查：
// 当字符串长度 < 8 时数据内联存储，否则通过指针访问
```

### 2.3 像素写入模式

从反汇编中字节操作指令推断像素写入逻辑：

```asm
0x10012255: movzx   eax, byte ptr [ebp+4]  ; 读取颜色 R 分量
0x10012260: mov     cl, [ebp+5]            ; 读取颜色 G 分量
0x10012266: fld     dword ptr [ebp+58h]    ; 加载 alpha 或强度
0x1001226C: fmul    dword ptr [ebp+8]      ; 乘以透明度系数
0x1001226F: mov     byte ptr [esp+...], al ; 写入像素
0x10012284: mov     byte ptr [esp+...+1], cl
```

这段代码展示了**逐通道像素写入**：分别处理 R、G 通道，配合浮点运算的 Alpha 混合。KiriKiri 使用 BGRA 像素格式（Blue-Green-Red-Alpha，每通道 8 位，共 32 位/像素），因此偏移 `+4` 和 `+5` 分别对应一个像素结构的第 2、3 字节（G、R 通道）。

## 三、使用 FreeType 实现跨平台替代

### 3.1 FreeType 核心概念

FreeType 是一个开源的字体光栅化库（Font Rasterization Library，将矢量字体数据转换为位图的软件组件），支持 TrueType、OpenType、Type 1 等主流字体格式。它的核心 API 围绕三个对象：

```
FT_Library (库实例)
  └── FT_Face (字体面 — 代表一个字体文件)
        └── FT_GlyphSlot (字形槽 — 加载一个字符的渲染结果)
              ├── bitmap (像素数据)
              └── metrics (度量信息)
```

### 3.2 基础渲染实现

```cpp
// freetype_textrender.cpp — 使用 FreeType 的文本渲染器
// 编译: 需要 freetype 库（通过 vcpkg install freetype）
#include <ft2build.h>
#include FT_FREETYPE_H

#include <cstdio>
#include <cstdint>
#include <cstring>
#include <string>
#include <vector>

// BGRA 像素格式（与 KiriKiri 一致）
struct Pixel {
    uint8_t b, g, r, a;  // Blue-Green-Red-Alpha
};

// 简易画布
class Canvas {
public:
    Canvas(int w, int h)
        : width_(w), height_(h),
          pixels_(w * h, {0, 0, 0, 0}) {}

    // Alpha 混合写入单个像素
    void blendPixel(int x, int y,
                    uint8_t r, uint8_t g, uint8_t b,
                    uint8_t a) {
        if (x < 0 || x >= width_ ||
            y < 0 || y >= height_) return;

        Pixel& dst = pixels_[y * width_ + x];
        // 标准 Alpha 混合公式:
        //   out = src * srcA + dst * (1 - srcA)
        float srcA = a / 255.0f;
        float invA = 1.0f - srcA;
        dst.r = static_cast<uint8_t>(
            r * srcA + dst.r * invA);
        dst.g = static_cast<uint8_t>(
            g * srcA + dst.g * invA);
        dst.b = static_cast<uint8_t>(
            b * srcA + dst.b * invA);
        dst.a = static_cast<uint8_t>(
            a + dst.a * invA);  // 累加透明度
    }

    int width() const { return width_; }
    int height() const { return height_; }
    const Pixel* data() const {
        return pixels_.data();
    }

private:
    int width_, height_;
    std::vector<Pixel> pixels_;
};

// FreeType 文本渲染器
class FreeTypeTextRenderer {
public:
    FreeTypeTextRenderer() {
        // 初始化 FreeType 库
        FT_Error err = FT_Init_FreeType(&library_);
        if (err) {
            printf("[错误] FreeType 初始化失败: %d\n",
                   err);
        }
    }

    ~FreeTypeTextRenderer() {
        if (face_) FT_Done_Face(face_);
        if (library_) FT_Done_FreeType(library_);
    }

    // 加载字体文件
    bool loadFont(const char* path, int sizePx) {
        FT_Error err = FT_New_Face(
            library_, path, 0, &face_);
        if (err) {
            printf("[错误] 加载字体失败: %s (错误码 %d)\n",
                   path, err);
            return false;
        }

        // 设置字号（单位：1/64 像素）
        err = FT_Set_Pixel_Sizes(face_, 0, sizePx);
        if (err) {
            printf("[错误] 设置字号失败: %d\n", err);
            return false;
        }

        printf("[OK] 字体加载成功: %s (%s), %dpx\n",
               face_->family_name,
               face_->style_name, sizePx);
        return true;
    }

    // 渲染单行文本到画布
    void drawText(Canvas& canvas,
                  const wchar_t* text,
                  int startX, int startY,
                  uint8_t r, uint8_t g, uint8_t b) {
        if (!face_) return;

        int penX = startX;  // 当前绘制 X 坐标
        int penY = startY;  // 基线 Y 坐标

        for (const wchar_t* p = text; *p; ++p) {
            // 加载字形到 face->glyph
            FT_Error err = FT_Load_Char(
                face_,
                static_cast<FT_ULong>(*p),
                FT_LOAD_RENDER  // 直接光栅化为位图
            );
            if (err) continue;

            FT_GlyphSlot slot = face_->glyph;
            FT_Bitmap& bmp = slot->bitmap;

            // 计算字形在画布上的位置
            // bitmap_left: 字形左边缘相对于 pen 的偏移
            // bitmap_top:  字形顶部相对于基线的偏移
            int glyphX = penX + slot->bitmap_left;
            int glyphY = penY - slot->bitmap_top;

            // 逐像素 Alpha 混合
            for (unsigned int row = 0;
                 row < bmp.rows; ++row) {
                for (unsigned int col = 0;
                     col < bmp.width; ++col) {
                    // FreeType 8-bit 灰度位图
                    uint8_t alpha =
                        bmp.buffer[row * bmp.pitch + col];
                    if (alpha > 0) {
                        canvas.blendPixel(
                            glyphX + col,
                            glyphY + row,
                            r, g, b, alpha);
                    }
                }
            }

            // 前进到下一个字符位置
            // advance.x 单位是 1/64 像素
            penX += slot->advance.x >> 6;
        }
    }

    // 测量文本宽度（不渲染）
    int measureWidth(const wchar_t* text) {
        if (!face_) return 0;
        int width = 0;
        for (const wchar_t* p = text; *p; ++p) {
            FT_Load_Char(face_, *p, FT_LOAD_DEFAULT);
            width += face_->glyph->advance.x >> 6;
        }
        return width;
    }

private:
    FT_Library library_ = nullptr;
    FT_Face face_ = nullptr;
};

int main() {
    FreeTypeTextRenderer renderer;

    // 加载字体（路径因平台而异）
#ifdef _WIN32
    const char* fontPath =
        "C:\\Windows\\Fonts\\msgothic.ttc";
#elif defined(__linux__)
    const char* fontPath =
        "/usr/share/fonts/opentype/noto/"

        "/system/fonts/NotoSansCJK-Regular.ttc";
```

## 四、字形缓存策略

### 4.1 为什么需要字形缓存

FreeType 的 `FT_Load_Char` 每次调用都会执行完整的字形光栅化流程：解析轮廓 → 缩放 → 网格拟合 → 扫描线转换。对于 CJK 文字（数千个常用字符），如果每帧都重新光栅化所有可见字符，性能损耗巨大。

原版 textrender.dll 中 `[esi+1C4h]` 处的 `std::vector` 很可能就是字形缓存——存储已光栅化的字形位图，下次遇到相同字符直接复用。

### 4.2 LRU 缓存实现

```cpp
// glyph_cache.h — 字形 LRU 缓存
#pragma once

#include <unordered_map>
#include <list>
#include <vector>
#include <cstdint>

// 缓存键：字符码 + 字号（不同字号的同一字符位图不同）
struct GlyphKey {
    uint32_t charCode;  // Unicode 码点
    uint16_t fontSize;  // 字号（像素）
    bool bold;          // 粗体标志

    bool operator==(const GlyphKey& o) const {
        return charCode == o.charCode &&
               fontSize == o.fontSize &&
               bold == o.bold;
    }
};

// 自定义哈希（用于 unordered_map 的键）
struct GlyphKeyHash {
    size_t operator()(const GlyphKey& k) const {
        // 将三个字段组合成一个 64 位哈希值
        size_t h = k.charCode;
        h ^= static_cast<size_t>(k.fontSize) << 32;
        h ^= static_cast<size_t>(k.bold) << 48;
        return h;
    }
};

// 缓存的字形数据
struct CachedGlyph {
    int width;                    // 位图宽度
    int height;                   // 位图高度
    int bearingX;                 // 水平偏移
    int bearingY;                 // 垂直偏移
    int advance;                  // 前进宽度（像素）
    std::vector<uint8_t> bitmap;  // 8-bit 灰度位图
};

// LRU 字形缓存
class GlyphCache {
public:
    explicit GlyphCache(size_t maxEntries = 1024)
        : maxEntries_(maxEntries) {}

    // 查找缓存（命中返回指针，未命中返回 nullptr）
    const CachedGlyph* find(const GlyphKey& key) {
        auto it = map_.find(key);
        if (it == map_.end()) return nullptr;

        // 移到 LRU 链表头部（标记为"最近使用"）
        lru_.splice(lru_.begin(), lru_,
                    it->second.lruIter);
        return &it->second.glyph;
    }

    // 插入新条目（如果缓存满，淘汰最久未使用的）
    void insert(const GlyphKey& key,
                CachedGlyph glyph) {
        // 如果已存在，更新
        auto existing = map_.find(key);
        if (existing != map_.end()) {
            existing->second.glyph = std::move(glyph);
            lru_.splice(lru_.begin(), lru_,
                        existing->second.lruIter);
            return;
        }

        // 缓存满时淘汰最久未使用的
        while (map_.size() >= maxEntries_) {
            auto& oldest = lru_.back();
            map_.erase(oldest);
            lru_.pop_back();
        }

        // 插入新条目
        lru_.push_front(key);
        Entry entry;
        entry.glyph = std::move(glyph);
        entry.lruIter = lru_.begin();
        map_[key] = std::move(entry);
    }

    size_t size() const { return map_.size(); }

    void clear() {
        map_.clear();
        lru_.clear();
    }

private:
    struct Entry {
        CachedGlyph glyph;
        std::list<GlyphKey>::iterator lruIter;
    };

    size_t maxEntries_;
    std::list<GlyphKey> lru_;  // 最近使用排序
    std::unordered_map<GlyphKey, Entry,
                       GlyphKeyHash> map_;
};
```

### 4.3 集成缓存的渲染器

```cpp
// cached_renderer.cpp — 带字形缓存的渲染器（片段）
#include "glyph_cache.h"
#include <ft2build.h>
#include FT_FREETYPE_H

class CachedTextRenderer {
public:
    // ... 初始化和字体加载同前 ...

    void drawText(Canvas& canvas,
                  const wchar_t* text,
                  int startX, int startY,
                  uint8_t r, uint8_t g, uint8_t b) {
        int penX = startX;
        int penY = startY;

        for (const wchar_t* p = text; *p; ++p) {
            GlyphKey key{
                static_cast<uint32_t>(*p),
                fontSize_, bold_};

            // 先查缓存
            const CachedGlyph* cached =
                cache_.find(key);

            if (!cached) {
                // 缓存未命中：光栅化并存入缓存
                FT_Load_Char(face_, *p,
                             FT_LOAD_RENDER);
                FT_GlyphSlot slot = face_->glyph;
                FT_Bitmap& bmp = slot->bitmap;

                CachedGlyph g;
                g.width = bmp.width;
                g.height = bmp.rows;
                g.bearingX = slot->bitmap_left;
                g.bearingY = slot->bitmap_top;
                g.advance = slot->advance.x >> 6;
                g.bitmap.assign(
                    bmp.buffer,
                    bmp.buffer + bmp.rows * bmp.pitch);

                cache_.insert(key, std::move(g));
                cached = cache_.find(key);
            }

            // 使用缓存数据渲染
            int gx = penX + cached->bearingX;
            int gy = penY - cached->bearingY;
            for (int row = 0;
                 row < cached->height; ++row) {
                for (int col = 0;
                     col < cached->width; ++col) {
                    uint8_t alpha =
                        cached->bitmap[
                            row * cached->width + col];
                    if (alpha > 0) {
                        canvas.blendPixel(
                            gx + col, gy + row,
                            r, g, b, alpha);
                    }
                }
            }

            penX += cached->advance;
        }
    }

private:
    FT_Library library_ = nullptr;
    FT_Face face_ = nullptr;
    uint16_t fontSize_ = 24;
    bool bold_ = false;
    GlyphCache cache_{2048};  // 缓存 2048 个字形
};
```

```cpp
#include "font_finder.h"
#include <cstdio>

#ifdef _WIN32
#include <Windows.h>
#include <Shlobj.h>  // SHGetFolderPathW

std::string FontFinder::findFontWindows(
        const std::string& fontName) {
    // 从注册表查找字体文件名
    HKEY hKey;
    const char* regPath =
        "SOFTWARE\\Microsoft\\Windows NT\\"
        "CurrentVersion\\Fonts";

    if (RegOpenKeyExA(HKEY_LOCAL_MACHINE, regPath,
                      0, KEY_READ, &hKey) != ERROR_SUCCESS)
        return "";

    char valueName[256], valueData[256];
    DWORD nameLen, dataLen, type;

    for (DWORD i = 0; ; i++) {
        nameLen = sizeof(valueName);
        dataLen = sizeof(valueData);
        if (RegEnumValueA(hKey, i, valueName, &nameLen,
                          nullptr, &type,
                          reinterpret_cast<BYTE*>(valueData),
                          &dataLen) != ERROR_SUCCESS)
            break;

        // 检查值名是否包含目标字体名
        std::string vn(valueName);
        if (vn.find(fontName) != std::string::npos) {
            RegCloseKey(hKey);
            // 拼接完整路径
            return "C:\\Windows\\Fonts\\"
                   + std::string(valueData);
        }
    }

    RegCloseKey(hKey);
    return "";
}

        result += buf;

    if (!result.empty() && result[0] == '\'') {
        result = result.substr(1, result.size() - 2);

    return result;
```

## 常见错误与排查

### 错误 1：FreeType 字号单位混淆

**症状**：渲染出的文字特别小或特别大。

**原因**：FreeType 的 `FT_Set_Char_Size` 使用 1/64 点（point）为单位，而 `FT_Set_Pixel_Sizes` 使用像素为单位。两者容易混淆。

**解决方案**：
```cpp
// 方式 1：用像素设置字号（推荐，直观）
FT_Set_Pixel_Sizes(face, 0, 24);  // 24 像素

// 方式 2：用点设置字号（需要知道 DPI）
// 24pt at 96 DPI → 24 * 64 = 1536（1/64 点单位）
FT_Set_Char_Size(face, 0, 24 * 64, 96, 96);

// 常见错误：忘记乘 64
// FT_Set_Char_Size(face, 0, 24, 96, 96);  // 实际只有 24/64 ≈ 0.375 点！
```

### 错误 2：BGRA vs RGBA 像素顺序

**症状**：文字颜色错误（红蓝互换）。

**原因**：KiriKiri 使用 BGRA 格式，但很多图形库默认 RGBA。混合时如果不注意通道顺序，红色和蓝色会互换。

**解决方案**：
```cpp
// KiriKiri BGRA 像素写入
void writePixelBGRA(uint8_t* dst,
                    uint8_t r, uint8_t g,
                    uint8_t b, uint8_t a) {
    dst[0] = b;  // Blue  在最低地址
    dst[1] = g;  // Green
    dst[2] = r;  // Red
    dst[3] = a;  // Alpha 在最高地址
}

// 错误：按 RGBA 顺序写入
// dst[0] = r; dst[1] = g; dst[2] = b; dst[3] = a;
// 这会导致红蓝互换！
```

### 错误 3：字形缓存键不包含字号

**症状**：切换字号后文字大小不变，或显示为乱码。

**原因**：缓存键只用了字符码，没有包含字号。不同字号的同一字符光栅化结果完全不同。

**解决方案**：缓存键必须包含影响光栅化结果的所有参数：
```cpp
struct GlyphKey {
    uint32_t charCode;  // 字符码
    uint16_t fontSize;  // 字号
    bool bold;          // 粗体
    // 如果支持斜体、描边等，也要加入
};
```

## 本节小结

- textrender.dll 的核心渲染函数 `sub_1000DFE0`（8,379 字节）使用密集的浮点运算和 SEH 保护，通过虚函数获取渲染参数
- 渲染调用链包含多个并行/串联函数：顶层入口（`sub_1000DFE0`）→ 排版预处理（`sub_100111D0`）→ 逐行绘制（`sub_100134E0` × 2）→ 逐字符光栅化（`sub_100121A0` × 3）
- 同级别的 `sub_1000DA70`（1,062 字节）是并列渲染入口，负责渲染前状态初始化/重置，与 `sub_1000DFE0` 成对调用
- `sub_10011Ce0`（1,087 字节）通过递归调用自身构建布局节点树，内部使用 `operator new` 动态分配节点
- `sub_100134E0`（1,225 字节）被 `sub_1000DFE0` 调用两次，可能对应"先渲染描边/阴影，再渲染正文"的两轮绘制模式
- 逐字符循环中 `movzx ecx, word ptr [esi]` 读取 16 位 Unicode 字符，配合浮点坐标实现子像素精度定位
- 对象内部使用 `std::vector`（通过 MSVC 三指针布局 `_Myfirst`/`_Mylast`/`_Myend` 识别，对应偏移 `+1C4h`/`+1D0h`/`+1D4h`）作为字形缓存
- 像素写入采用逐通道 Alpha 混合，与 KiriKiri 的 BGRA 像素格式匹配
- FreeType 是推荐的跨平台替代方案：`FT_Load_Char` + `FT_LOAD_RENDER` 可直接获取 8 位灰度位图
- LRU 字形缓存（推荐 1024-2048 条目）可大幅减少重复光栅化开销
- 四平台字体路径差异需要统一的 `FontFinder` 抽象层处理

## 练习题与答案

### 题目 1：Alpha 混合公式推导

给定前景像素 (R=255, G=0, B=0, A=128) 和背景像素 (R=0, G=0, B=255, A=255)，计算 Alpha 混合后的最终像素值。

<details>
<summary>查看答案</summary>

Alpha 混合公式：`out = src × srcA + dst × (1 - srcA)`

```
srcA = 128 / 255 ≈ 0.502
invA = 1 - 0.502 = 0.498

R_out = 255 × 0.502 + 0 × 0.498 = 128
G_out = 0 × 0.502 + 0 × 0.498   = 0
B_out = 0 × 0.502 + 255 × 0.498 = 127
A_out = 128 + 255 × 0.498        = 255（饱和到 255）

最终像素: (R=128, G=0, B=127, A=255)
```

在 KiriKiri 的 BGRA 内存布局中，这个像素存储为：`[127, 0, 128, 255]`（B, G, R, A 字节顺序）。

</details>

### 题目 2：设计字形缓存淘汰策略

假设你的 TextRender 实现需要支持一个 1280×720 的游戏场景，屏幕上同时显示约 200 个不同字符（含汉字、假名、字母），字号范围 16-48px，有粗体/非粗体两种变体。

请计算：
1. 最坏情况下需要缓存多少个字形？
2. 每个缓存条目大约占多少内存（假设字号 32px）？
3. 推荐的缓存容量是多少？

<details>
<summary>查看答案</summary>

**1. 最坏情况字形数量：**
```
200 字符 × 可能的字号数 × 粗体变体
= 200 × (48 - 16 + 1) × 2
= 200 × 33 × 2
= 13,200 个字形
```
但实际上同时活跃的字号通常只有 2-4 种，所以实际需要：
`200 × 4 × 2 = 1,600 个字形`

**2. 单条目内存占用（32px 字号）：**
```
位图大小（CJK 字符）: 32 × 32 = 1,024 字节
GlyphKey: 8 字节
度量数据: 5 × 4 = 20 字节
std::vector 开销: 24 字节
LRU 链表节点: 48 字节（含指针）
unordered_map 节点: 64 字节

总计约: 1,024 + 8 + 20 + 24 + 48 + 64 ≈ 1,188 字节/条目
```

**3. 推荐缓存容量：**
```
实际需求: 1,600 条目
安全余量 ×1.5: 2,400 条目
内存占用: 2,400 × 1,188 ≈ 2.7 MB

推荐: 2,048 条目（2 的幂，约 2.4 MB）
```
这个内存占用在移动设备上也完全可接受。如果需要更大字号（如 48px），单字形位图约 2,304 字节，总内存约 4.5 MB，仍然合理。

</details>

### 题目 3：跨平台字体回退链

为 KrKr2 模拟器设计一个字体回退链（Font Fallback Chain）：当游戏指定的字体（如 "MS Gothic"）在 Linux 上不存在时，自动找到替代字体。请给出四平台的回退策略。

<details>
<summary>查看答案</summary>

```cpp
// 字体回退链设计
std::vector<std::string> getFontFallbackChain(
        const std::string& requestedFont) {
    std::vector<std::string> chain;

    // 1. 首先尝试用户请求的字体
    chain.push_back(requestedFont);

#ifdef _WIN32
    // Windows 回退链
    chain.push_back("MS Gothic");     // 日文默认
    chain.push_back("SimSun");        // 中文宋体
    chain.push_back("MingLiU");       // 繁体中文
    chain.push_back("Meiryo");        // 日文现代
    chain.push_back("Yu Gothic");     // Win10 日文
    chain.push_back("Arial");         // 拉丁字符兜底

#elif defined(__linux__)
    // Linux 回退链
    chain.push_back("Noto Sans CJK JP");
    chain.push_back("Noto Sans CJK SC");
    chain.push_back("WenQuanYi Micro Hei");
    chain.push_back("IPAGothic");
    chain.push_back("TakaoPGothic");
    chain.push_back("DejaVu Sans");

#elif defined(__APPLE__)
    // macOS 回退链
    chain.push_back("Hiragino Sans");
    chain.push_back("Hiragino Kaku Gothic ProN");
    chain.push_back("PingFang SC");
    chain.push_back("Apple SD Gothic Neo");
    chain.push_back("Helvetica Neue");

#elif defined(__ANDROID__)
    // Android 回退链（直接用文件路径）
    chain.push_back("NotoSansCJK-Regular");
    chain.push_back("DroidSansFallback");
    chain.push_back("NotoSansJP-Regular");
    chain.push_back("Roboto-Regular");
#endif

    return chain;
}
```

**设计原则：**
1. 日文优先：KiriKiri 游戏以日文 AVG 为主，日文字体放在回退链前面
2. CJK 覆盖：确保至少有一个字体能覆盖 CJK Unified Ideographs 范围
3. 拉丁兜底：最后总是有一个拉丁字体，确保英文/数字至少能显示
4. 平台适配：每个平台使用该平台上最可能存在的字体

</details>
