## 术语预览

本节将涉及以下术语，先有个印象，正文中会逐一详细讲解：

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| 静态库 | Static Library (.a / .lib) | 编译时链接到主程序的二进制代码集合，不需要运行时加载 DLL |
| ncbind | NativeClass Bind | KiriKiri 的 C++ → TJS2 绑定框架，通过宏自动注册 C++ 类到脚本引擎 |
| INTERFACE 库 | INTERFACE Library | CMake 中只传递编译参数/头文件路径但不生成二进制的虚拟目标 |
| target_sources | — | CMake 命令，向构建目标添加源文件 |
| NCB_REGISTER_CLASS | — | ncbind 宏，将一个 C++ 类注册为 TJS2 可访问的脚本类 |
| V2Link / V2Unlink | — | KiriKiri 插件的入口/退出函数，原版 DLL 导出，静态集成时改为内部调用 |

## 一、从逆向代码到可编译源文件

### 1.1 逆向成果整理

经过前几章的逆向工作，我们已经还原了两个无源码插件的核心逻辑：

- **PackinOne** — LZ4 压缩/解压流、CxImage 图像处理、多种 TJS 扩展类
- **TextRender** — 软件字体渲染器，不依赖 GDI，使用 FreeType 替代

逆向代码通常以"函数片段"形式存在——从 IDA 反编译得到的 C 代码、手工还原的 C++ 类。要把这些代码编译为可用的静态库，第一步是**规范化文件组织**。

### 1.2 目录结构规划

KrKr2 的插件代码统一放在 `cpp/plugins/` 下。每个有独立子目录的插件是一个单独的 CMake 目标（静态库），最终链接到 `krkr2plugin`。参考现有插件的目录结构：

```
cpp/plugins/
├── CMakeLists.txt          # 主 CMakeLists，定义 krkr2plugin 目标
├── psdfile/                # PSD 文件解析插件（有独立子目录）
│   ├── CMakeLists.txt
│   └── *.cpp / *.h
├── psbfile/                # PSB 文件解析插件
│   ├── CMakeLists.txt
│   └── *.cpp / *.h
├── layerex_draw/           # 图层扩展绘制
├── motionplayer/           # E-mote 动画播放
├── fstat/                  # 文件状态查询
├── scriptsEx.cpp           # 脚本扩展（单文件插件，直接加入 krkr2plugin）
├── csvParser.cpp           # CSV 解析
└── ... (其他单文件插件)
```

对于逆向还原的插件，我们遵循**同样的组织方式**。如果插件代码量大（多个 .cpp 文件），使用子目录 + 独立 CMake 目标；如果代码量小（单文件），直接作为源文件加入 `krkr2plugin`。

### 1.3 PackinOne 文件组织示例

PackinOne 是一个大型插件（原始 DLL 648 KB，1999 个函数），还原代码量较大，适合使用子目录：

```
cpp/plugins/packinone/
├── CMakeLists.txt              # 独立 CMake 目标
├── packinone.h                 # 公共头文件（类声明、常量定义）
├── packinone_register.cpp      # ncbind 注册入口
├── lz4_stream.cpp              # LZ4 压缩/解压流实现
├── lz4_stream.h                # LZ4 流类声明
├── temporary_files.cpp         # TemporaryFiles TJS 类实现
├── storage_fstat.cpp           # StoragesFstat TJS 类实现
└── scripts_add.cpp             # ScriptsAdd TJS 类实现
```

每个 `.cpp` 文件对应一个逻辑模块，头文件声明公共接口。注册入口（`packinone_register.cpp`）集中处理所有 ncbind 注册。

### 1.4 TextRender 文件组织示例

TextRender 相对紧凑（原始 DLL 222 KB，828 个函数），但渲染逻辑复杂，同样推荐子目录：

```
cpp/plugins/textrender/
├── CMakeLists.txt              # 独立 CMake 目标
├── textrender.h                # TextRender / TextRenderBase 类声明
├── textrender.cpp              # 核心渲染逻辑（还原自 sub_1000DFE0）
├── textrender_register.cpp     # ncbind 注册入口
├── glyph_cache.cpp             # 字形缓存（LRU 策略）
├── glyph_cache.h               # 缓存类声明
├── font_finder.cpp             # 跨平台字体查找
└── font_finder.h               # 字体查找接口
```

> **关键点：** 原始 textrender.dll 不导入 GDI32（Windows 图形 API），说明它使用内部软件渲染。我们的还原版使用 FreeType（一个开源字体渲染库，通过 vcpkg 安装）替代，保持了相同的"不依赖系统 GDI"特性。

## 二、CMake 构建配置

### 2.1 子目录插件的 CMakeLists.txt

以 TextRender 为例，编写独立的 CMake 构建文件：

```cmake
# cpp/plugins/textrender/CMakeLists.txt
cmake_minimum_required(VERSION 3.19)
project(textrender)

# 创建静态库目标
add_library(${PROJECT_NAME} STATIC)

# 添加源文件
target_sources(${PROJECT_NAME} PRIVATE
    textrender.cpp
    textrender_register.cpp
    glyph_cache.cpp
    font_finder.cpp
)

# 头文件搜索路径：自身目录 + 核心模块路径
target_include_directories(${PROJECT_NAME} PUBLIC
    ${CMAKE_CURRENT_SOURCE_DIR}
    ${KRKR2CORE_PATH}/
    ${KRKR2CORE_PATH}/tjs2
    ${KRKR2CORE_PATH}/plugin
    ${KRKR2CORE_PATH}/utils
)

# 依赖 FreeType（通过 vcpkg 安装）
find_package(Freetype REQUIRED)
target_link_libraries(${PROJECT_NAME} PRIVATE Freetype::Freetype)

# 依赖核心库（获取 TJS2 类型定义等）
target_link_libraries(${PROJECT_NAME} PUBLIC krkr2core)
```

**逐行解释：**

| 行 | 作用 |
|----|------|
| `add_library(... STATIC)` | 创建静态库，编译后生成 `.a`（Linux/macOS）或 `.lib`（Windows）文件 |
| `target_sources(... PRIVATE)` | 添加源文件，`PRIVATE` 表示这些文件只在本目标内编译 |
| `target_include_directories(... PUBLIC)` | `PUBLIC` 让链接此目标的其他目标也能找到这些头文件 |
| `find_package(Freetype REQUIRED)` | 通过 vcpkg 查找 FreeType 库，`REQUIRED` 表示找不到就报错 |
| `target_link_libraries(... PUBLIC krkr2core)` | 链接核心库，获取 TJS2 类型定义和 ncbind 头文件 |

### 2.2 将子目录插件集成到 krkr2plugin

在 `cpp/plugins/CMakeLists.txt` 中添加子目录和链接：

```cmake
# 在现有的 add_subdirectory 列表后面追加
add_subdirectory(textrender)
add_subdirectory(packinone)

# 在现有的 target_link_libraries 列表中追加
target_link_libraries(${PROJECT_NAME} PUBLIC
        psdfile
        psbfile
        layerExDraw
        motionplayer
        fstat
        textrender      # 新增：逆向还原的 TextRender 插件
        packinone       # 新增：逆向还原的 PackinOne 插件
)
```

这样，当编译 `krkr2plugin` 时，CMake 会自动编译并链接 textrender 和 packinone 两个子目标。

### 2.3 添加 FreeType 依赖到 vcpkg

TextRender 需要 FreeType 库。在项目根目录的 `vcpkg.json` 中添加依赖：

```json
{
  "dependencies": [
    "freetype",
    // ... 其他现有依赖
  ]
}
```

然后重新执行 CMake 配置，vcpkg 会自动下载并编译 FreeType：

```bash
# Windows
cmake --preset="Windows Debug Config"

# Linux
cmake --preset="Linux Debug Config"

# macOS
cmake --preset="MacOS Debug Config"

# Android（通过 Gradle 自动处理）
./platforms/android/gradlew -p ./platforms/android assembleDebug
```

### 2.4 PackinOne 的 CMakeLists.txt

PackinOne 使用 LZ4 压缩库。LZ4 有两种集成方式：

| 方式 | 优点 | 缺点 |
|------|------|------|
| vcpkg 安装 `lz4` | 标准化管理，自动处理跨平台 | 需要确保版本兼容 |
| 内嵌 LZ4 源码 | 完全控制，无外部依赖 | 需要手动维护源码 |

推荐使用 vcpkg 方式：

```cmake
# cpp/plugins/packinone/CMakeLists.txt
cmake_minimum_required(VERSION 3.19)
project(packinone)

add_library(${PROJECT_NAME} STATIC)

target_sources(${PROJECT_NAME} PRIVATE
    lz4_stream.cpp
    temporary_files.cpp
    storage_fstat.cpp
    scripts_add.cpp
    packinone_register.cpp
)

target_include_directories(${PROJECT_NAME} PUBLIC
    ${CMAKE_CURRENT_SOURCE_DIR}
    ${KRKR2CORE_PATH}/
    ${KRKR2CORE_PATH}/tjs2
    ${KRKR2CORE_PATH}/plugin
    ${KRKR2CORE_PATH}/utils
    ${KRKR2CORE_PATH}/base       # 归档/流相关头文件
)

# LZ4 压缩库
find_package(lz4 CONFIG REQUIRED)
target_link_libraries(${PROJECT_NAME} PRIVATE lz4::lz4)

target_link_libraries(${PROJECT_NAME} PUBLIC krkr2core)
```

## 三、ncbind 插件注册机制

### 3.1 原版 DLL 的注册方式回顾

原版 KiriKiri 插件以 DLL 形式加载，引擎通过 `LoadLibrary` + `GetProcAddress` 调用 DLL 的 `V2Link` 函数完成注册。流程如下：

```
引擎启动
  → 扫描 plugin/ 目录下的 .dll 文件
  → LoadLibrary("PackinOne.dll")
  → GetProcAddress("V2Link")
  → 调用 V2Link(ppVal) — 传入 iTJSDispatch2* 接口
  → DLL 内部通过 ncbind 宏注册 TJS 类
  → 引擎卸载时调用 V2Unlink()
```

### 3.2 静态集成的注册方式

KrKr2 采用**静态链接**而非动态加载。所有插件编译为静态库，链接到主程序。此时不存在 `LoadLibrary` 调用，注册改为**编译期自动注册**。

ncbind 框架的核心机制是 `NCB_PRE_REGIST_CALLBACK` 和 `NCB_REGISTER_CLASS` 宏。这些宏利用 C++ 的**静态初始化**（static initialization）特性——在 `main()` 函数执行之前，全局/静态对象的构造函数就会被调用。

```cpp
// ncbind 注册的本质：定义一个全局静态对象
// 该对象的构造函数在程序启动时自动执行，完成类注册
static ncbAutoRegister<MyClass> s_auto_register;
// 等价于：程序启动时自动调用 ncbAutoRegister<MyClass>::ncbAutoRegister()
// 该构造函数内部会把 MyClass 注册到 TJS2 引擎
```

这意味着：**只要你的 .cpp 文件被编译并链接进最终二进制，ncbind 注册就会自动生效**。不需要在任何地方手动调用注册函数。

### 3.3 编写注册文件：TextRender 示例

以下是 TextRender 插件的完整注册文件：

```cpp
// cpp/plugins/textrender/textrender_register.cpp
//
// TextRender 插件的 ncbind 注册入口
// 将 TextRenderBase C++ 类注册为 TJS2 脚本可访问的类

#include "ncbind.hpp"       // ncbind 框架头文件
#include "textrender.h"     // TextRender / TextRenderBase 类声明

// ============================================================
// NCB_REGISTER_CLASS: 注册 TextRenderBase 到 TJS2
// ============================================================
// 原版 textrender.dll 的 RTTI 显示注册目标是 TextRenderBase（不是 TextRender）
// IDA 分析: sub_10005980 注册了 "setOption", "setDefault", "setRenderSize"

NCB_REGISTER_CLASS(TextRenderBase) {
    // 构造函数：无参数构造
    NCB_CONSTRUCTOR(());

    // ---- 方法注册 ----
    // 每个 NCB_METHOD 将 C++ 成员函数暴露给 TJS2 脚本

    // 设置渲染选项（字体名、大小、颜色等）
    NCB_METHOD(setOption);

    // 恢复默认渲染参数
    NCB_METHOD(setDefault);

    // 设置渲染区域大小（宽度、高度，单位：像素）
    NCB_METHOD(setRenderSize);

    // ---- 属性注册 ----
    // NCB_PROPERTY 将 C++ 的 getter/setter 暴露为 TJS2 属性

    // 字体名（string 类型）
    // IDA RTTI: ncbind 模板实例化 Property<string>
    NCB_PROPERTY(face, getFace, setFace);

    // 字体大小（float 类型）
    NCB_PROPERTY(size, getSize, setSize);

    // 字体颜色（unsigned int 类型，BGRA 格式）
    NCB_PROPERTY(color, getColor, setColor);

    // 粗体标志（bool 类型）
    NCB_PROPERTY(bold, getBold, setBold);

    // 斜体标志（bool 类型）
    NCB_PROPERTY(italic, getItalic, setItalic);

    // 行间距（int 类型）
    NCB_PROPERTY(lineSpacing, getLineSpacing, setLineSpacing);
};
```

**代码解析：**

| 宏 | 功能 | 对应 IDA 发现 |
|----|------|--------------|
| `NCB_REGISTER_CLASS(TextRenderBase)` | 将 C++ 类注册为同名 TJS2 类 | RTTI 中 ncbind 模板以 TextRenderBase 实例化 |
| `NCB_CONSTRUCTOR(())` | 注册无参构造函数 | sub_10005980 的构造调用 |
| `NCB_METHOD(setOption)` | 暴露 setOption 方法 | IDA 字符串 "setOption" |
| `NCB_PROPERTY(face, ...)` | 暴露 face 属性（读写） | RTTI Property<string> 实例 |

### 3.4 编写注册文件：PackinOne 示例

PackinOne 注册更复杂，因为它包含多个 TJS 类：

```cpp
// cpp/plugins/packinone/packinone_register.cpp
//
// PackinOne 插件注册入口
// 注册多个 TJS2 类（来自 IDA 分析的 ncbind 注册列表）

#include "ncbind.hpp"
#include "packinone.h"
#include "lz4_stream.h"

// ============================================================
// 方式一：NCB_REGISTER_CLASS — 完整类注册
// ============================================================

// TemporaryFiles：临时文件管理
NCB_REGISTER_CLASS(TemporaryFiles) {
    NCB_CONSTRUCTOR(());
    NCB_METHOD(open);       // 打开临时文件
    NCB_METHOD(close);      // 关闭临时文件
    NCB_METHOD(remove);     // 删除临时文件
    NCB_METHOD(getPath);    // 获取临时文件路径
};

// StoragesFstat：文件状态查询扩展
NCB_REGISTER_CLASS(StoragesFstat) {
    NCB_CONSTRUCTOR(());
    NCB_METHOD(fstat);      // 获取文件状态信息
    NCB_METHOD(getSize);    // 获取文件大小
    NCB_METHOD(getMTime);   // 获取修改时间
};

// ============================================================
// 方式二：NCB_PRE_REGIST_CALLBACK — 函数式注册
// ============================================================
// 用于注册不依赖 C++ 类的全局 TJS 函数或简单扩展

// ScriptsAdd：向 TJS2 添加脚本扩展方法
static void PreRegistScriptsAdd() {
    // 此函数在引擎初始化阶段被自动调用
    // 可以在这里向全局对象添加方法
    TVPRegisteredPlugins.insert(TJS_W("PackinOne"));
}
NCB_PRE_REGIST_CALLBACK(PreRegistScriptsAdd);
```

### 3.5 NCB_REGISTER_CLASS 与 NCB_PRE_REGIST_CALLBACK 对比

| 特性 | NCB_REGISTER_CLASS | NCB_PRE_REGIST_CALLBACK |
|------|-------------------|------------------------|
| **用途** | 注册完整的 C++ 类 | 注册初始化回调函数 |
| **时机** | 引擎创建 TJS2 全局对象后 | 引擎初始化的更早阶段 |
| **参数** | 类名（必须与 C++ 类名一致） | 函数指针 |
| **典型场景** | TextRender、TemporaryFiles 等完整类 | 添加全局方法、修改引擎配置 |
| **IDA 识别** | RTTI 中有 ncbClassInfo 模板实例 | V2Link 内直接调用 |

## 四、跨平台编译处理

### 4.1 平台差异总览

逆向还原的插件代码需要在四个平台上编译。主要差异点：

| 差异项 | Windows | Linux | macOS | Android |
|--------|---------|-------|-------|---------|
| 静态库后缀 | `.lib` | `.a` | `.a` | `.a` |
| 字体路径 | `C:\Windows\Fonts\` | `/usr/share/fonts/` | `/Library/Fonts/` | `/system/fonts/` |
| FreeType | vcpkg 安装 | vcpkg 安装 | vcpkg 安装 | vcpkg 安装 |
| 字符编码 | UTF-16 (wchar_t) | UTF-8 (char) | UTF-8 (char) | UTF-8 (char) |
| 文件系统 | `\` 分隔符 | `/` 分隔符 | `/` 分隔符 | `/` 分隔符 |

### 4.2 条件编译处理

KrKr2 使用自定义平台宏（不是标准的 `CMAKE_SYSTEM_NAME`）：

```cpp
// cpp/plugins/textrender/font_finder.cpp
// 跨平台字体路径查找

#include "font_finder.h"
#include <filesystem>    // C++17 文件系统库
#include <vector>
#include <string>

namespace fs = std::filesystem;

// 获取系统字体目录列表
std::vector<fs::path> FontFinder::getSystemFontDirs() {
    std::vector<fs::path> dirs;

#if defined(_WIN32)
    // Windows: 系统字体目录 + 用户字体目录
    dirs.push_back("C:\\Windows\\Fonts");

    // 用户字体目录（Windows 10 1803+）
    const char* localAppData = std::getenv("LOCALAPPDATA");
    if (localAppData) {
        auto userFonts = fs::path(localAppData) / "Microsoft" / "Windows" / "Fonts";
        if (fs::exists(userFonts)) {
            dirs.push_back(userFonts);
        }
    }

#elif defined(__APPLE__)
    // macOS: 三级字体目录
    dirs.push_back("/System/Library/Fonts");    // 系统字体（受 SIP 保护）
    dirs.push_back("/Library/Fonts");           // 全局安装的字体
    const char* home = std::getenv("HOME");
    if (home) {
        dirs.push_back(fs::path(home) / "Library" / "Fonts"); // 用户字体
    }

#elif defined(__ANDROID__)
    // Android: 系统字体目录（只读）
    dirs.push_back("/system/fonts");
    dirs.push_back("/system/font");

#else
    // Linux: FreeDesktop 标准字体路径
    dirs.push_back("/usr/share/fonts");
    dirs.push_back("/usr/local/share/fonts");
    const char* home = std::getenv("HOME");
    if (home) {
        dirs.push_back(fs::path(home) / ".local" / "share" / "fonts");
        dirs.push_back(fs::path(home) / ".fonts");  // 旧式路径，仍被部分系统支持
    }
#endif

    return dirs;
}
```

### 4.3 字符编码转换

原版 textrender.dll 在 Windows 上使用 `wchar_t`（UTF-16）处理字体名称。跨平台版本需要统一为 UTF-8，再在必要时转换：

```cpp
// cpp/plugins/textrender/textrender.cpp
// 字符编码工具函数

#include "CharacterSet.h"  // KrKr2 自带的字符集转换工具

// 将 TJS 字符串（tjs_char，在 Windows 上是 wchar_t）转为 UTF-8
std::string tjs_to_utf8(const tjs_char* str) {
    if (!str) return "";

#if defined(_WIN32)
    // Windows: tjs_char = wchar_t (UTF-16)
    // 使用 WideCharToMultiByte 转换
    int len = WideCharToMultiByte(CP_UTF8, 0, str, -1, nullptr, 0, nullptr, nullptr);
    if (len <= 0) return "";
    std::string result(len - 1, '\0');  // len 包含 null 终止符
    WideCharToMultiByte(CP_UTF8, 0, str, -1, result.data(), len, nullptr, nullptr);
    return result;
#else
    // Linux/macOS/Android: tjs_char = char16_t 或已经是 UTF-8 兼容
    // 使用 KrKr2 内置的转换函数
    return TVPStringToUTF8(ttstr(str));
#endif
}
```

### 4.4 Android 特殊处理

Android 平台有两个特殊问题需要处理：

**1. 字体文件访问**

Android 的 `/system/fonts/` 目录在非 root 设备上只读。应用自带字体需要放在 `assets/` 中，通过 Android Asset Manager 读取：

```cpp
// Android 字体加载特殊路径
#ifdef __ANDROID__
#include <android/asset_manager.h>

// 从 Android assets 目录加载字体文件到内存
std::vector<uint8_t> loadFontFromAssets(
    AAssetManager* mgr,
    const char* fontName   // 如 "fonts/NotoSansCJK.ttc"
) {
    AAsset* asset = AAssetManager_open(mgr, fontName, AASSET_MODE_BUFFER);
    if (!asset) {
        spdlog::get("plugin")->error("无法从 assets 加载字体: {}", fontName);
        return {};
    }

    size_t size = AAsset_getLength(asset);  // 获取文件大小
    std::vector<uint8_t> data(size);
    AAsset_read(asset, data.data(), size);  // 读取全部内容
    AAsset_close(asset);

    return data;
}
#endif
```

**2. NDK 编译标志**

Android NDK 的 C++ 标准库使用 `c++_shared`。在 CMakeLists.txt 中需要确保链接正确：

```cmake
# 仅在 Android 平台添加的编译标志
if(ANDROID)
    target_compile_options(${PROJECT_NAME} PRIVATE
        -fPIC                   # 位置无关代码（共享库必需）
        -ffunction-sections     # 函数级别链接（减小体积）
        -fdata-sections         # 数据级别链接
    )
endif()
```

### 实践任务：创建一个最小化的逆向插件

下面我们从零创建一个简化版插件，模拟逆向还原代码的集成过程：

**目标：** 创建一个 `SimpleReversed` TJS2 类，有一个 `compute` 方法和一个 `result` 属性。

**步骤 1：创建文件**

```cpp
// cpp/plugins/simple_reversed.h
#ifndef SIMPLE_REVERSED_H
#define SIMPLE_REVERSED_H

#include "tjsNative.h"
#include <cstdint>

// 模拟从 IDA 还原的类
// 原始功能：对输入数据做简单的位运算变换
class SimpleReversed : public tTJSNativeInstance {
public:
    SimpleReversed() : result_(0) {}
    ~SimpleReversed() override = default;

    // 还原的核心算法
    // 原始反汇编：xor eax, 0xDEADBEEF; rol eax, 13; add eax, ecx
    void compute(int input) {
        uint32_t val = static_cast<uint32_t>(input);
        val ^= 0xDEADBEEF;                    // XOR 混淆
        val = (val << 13) | (val >> 19);       // 循环左移 13 位
        val += 0x12345678;                     // 加常量
        result_ = static_cast<int>(val);
    }

    int getResult() const { return result_; }

private:
    int result_;
};

#endif // SIMPLE_REVERSED_H
```

**步骤 2：编写注册文件**

```cpp
// cpp/plugins/simple_reversed_register.cpp
#include "ncbind.hpp"
#include "simple_reversed.h"

NCB_REGISTER_CLASS(SimpleReversed) {
    NCB_CONSTRUCTOR(());
    NCB_METHOD(compute);
    NCB_PROPERTY_RO(result, getResult);  // 只读属性
};
```

**步骤 3：添加到 krkr2plugin**

在 `cpp/plugins/CMakeLists.txt` 中添加：

```cmake
target_sources(${PROJECT_NAME} PUBLIC
    # ... 现有文件 ...
    simple_reversed_register.cpp
)
```

> **注意：** `simple_reversed.h` 是 header-only，不需要添加到 `target_sources`。

**步骤 4：编译并测试**


## 常见错误与排查

### 错误 1：链接时 "undefined reference to ncbClassInfo"

**症状：**

```
error: undefined reference to `ncbClassInfo<TextRenderBase>::GetName()`
```

**原因：** ncbind 的模板实例化需要看到完整的类定义。如果 `textrender.h` 中只有前向声明而没有完整类定义，模板无法实例化。

**解决：** 确保注册文件 `#include` 了包含完整类定义的头文件，而不是前向声明。

### 错误 2：运行时 TJS2 报 "member not found"

**症状：** TJS2 脚本调用 `tr.setOption(...)` 时报错。

**原因：** `NCB_METHOD(setOption)` 要求 C++ 类中有一个**完全匹配**的公有成员函数 `setOption`。如果函数签名不匹配（参数类型错误、const 修饰符缺失等），注册时不会报错，但运行时调用会失败。

**解决：** 检查 C++ 函数签名是否与 ncbind 的类型转换规则匹配。ncbind 支持的自动转换包括：

| TJS2 类型 | C++ 类型 |
|-----------|---------|
| integer | `int`, `tjs_int`, `tjs_int64` |
| real | `float`, `double` |
| string | `ttstr`, `const tjs_char*` |
| octet | `tTJSVariantOctet*` |
| object | `iTJSDispatch2*`, `tTJSVariant*` |

### 错误 3：Android 编译 "cannot find -lfreetype"

**症状：** Android NDK 编译时找不到 FreeType 库。

**原因：** vcpkg 的 Android triplet（如 `arm64-android`）可能没有正确配置，导致 FreeType 没有为 Android 架构编译。

**解决：**

```bash
# 检查 vcpkg 是否安装了 Android 版 FreeType
vcpkg list --triplet arm64-android | findstr freetype

# 如果没有，手动安装
vcpkg install freetype:arm64-android

# 或检查 vcpkg_android.cmake 中的 triplet 配置
# 文件位置：cmake/vcpkg_android.cmake
```

## 本节小结

- **文件组织**：逆向代码按模块拆分为 .cpp / .h 文件，大型插件使用子目录 + 独立 CMake 目标，小型插件直接作为源文件加入 krkr2plugin
- **CMake 集成**：子目录插件通过 `add_subdirectory` + `target_link_libraries` 集成；外部依赖（FreeType、LZ4）通过 vcpkg 管理
- **ncbind 注册**：`NCB_REGISTER_CLASS` 注册完整类，`NCB_PRE_REGIST_CALLBACK` 注册初始化回调；静态链接下利用 C++ 静态初始化自动完成注册
- **跨平台处理**：字体路径、字符编码、Android Asset 加载等需要条件编译处理
- **验证方法**：编译成功后用 TJS2 脚本测试类是否可创建和调用

### 题目 1：为 StoragesFstat 类编写完整的注册文件

假设 `StoragesFstat` 类有以下接口（从 IDA 还原）：
- 构造函数：无参数
- 方法：`fstat(path)`、`getSize(path)`、`getMTime(path)`
- 属性：`lastError`（只读，int 类型）

请编写完整的 ncbind 注册文件。

<details>
<summary>查看答案</summary>

```cpp
// cpp/plugins/packinone/storagefstat_register.cpp
#include "ncbind.hpp"
#include "storage_fstat.h"

NCB_REGISTER_CLASS(StoragesFstat) {
    // 无参构造函数
    NCB_CONSTRUCTOR(());

    // 方法注册
    NCB_METHOD(fstat);      // 获取文件状态
    NCB_METHOD(getSize);    // 获取文件大小
    NCB_METHOD(getMTime);   // 获取修改时间

    // 只读属性：使用 NCB_PROPERTY_RO 宏
    // 第一个参数是 TJS2 中的属性名
    // 第二个参数是 C++ getter 方法名
    NCB_PROPERTY_RO(lastError, getLastError);
};
```

**要点：**
- `NCB_PROPERTY_RO` 用于只读属性（只有 getter，没有 setter）
- `NCB_PROPERTY` 用于读写属性（同时需要 getter 和 setter）
- 方法和属性名必须与 C++ 类中的成员函数名完全一致

</details>

### 题目 2：分析现有插件的 CMake 结构

查看 `cpp/plugins/fstat/` 目录的 CMakeLists.txt，回答：
1. fstat 插件使用什么类型的库（STATIC / SHARED / INTERFACE）？
2. 它链接了哪些外部依赖？
3. 如果要为它添加一个新源文件 `fstat_ext.cpp`，需要修改哪些文件？

<details>
<summary>查看答案</summary>

1. **STATIC 库**。所有 KrKr2 插件都编译为静态库，链接到 `krkr2plugin` 目标。

2. **无外部依赖**。fstat 插件只依赖 `krkr2core`（通过父级 CMakeLists.txt 的 `target_link_libraries` 传递），不需要额外的第三方库。

不需要修改父级的 `cpp/plugins/CMakeLists.txt`，因为 `add_subdirectory(fstat)` 和 `target_link_libraries(... fstat)` 已经存在。

</details>

### 题目 3：排查注册失败问题

以下代码编译通过但运行时 TJS2 脚本无法找到 `MyPlugin` 类，请找出原因：

```cpp
// my_plugin.h
class MyPlugin : public tTJSNativeInstance {
public:
    void doSomething();
};

// my_plugin_register.cpp
#include "ncbind.hpp"
// 注意：没有 #include "my_plugin.h"

class MyPlugin;  // 前向声明

NCB_REGISTER_CLASS(MyPlugin) {
    NCB_METHOD(doSomething);
};
```

<details>
<summary>查看答案</summary>

**原因：** 注册文件使用了**前向声明**而非完整的类定义。

`NCB_REGISTER_CLASS` 宏内部需要实例化模板 `ncbClassInfo<MyPlugin>`，这要求编译器看到 `MyPlugin` 的完整定义（包括所有成员函数声明）。前向声明只告诉编译器"这个类存在"，不提供任何成员信息。

**修复方法：**

```cpp
// my_plugin_register.cpp
#include "ncbind.hpp"
#include "my_plugin.h"  // 必须包含完整类定义

NCB_REGISTER_CLASS(MyPlugin) {
    NCB_METHOD(doSomething);
};
```

**额外说明：** 这个错误在某些编译器上可能不会产生编译错误（因为模板实例化的延迟特性），但 ncbind 内部的类型推导会失败，导致注册的类信息不完整。运行时 TJS2 引擎找不到有效的类定义，所以报"类不存在"。

</details>
