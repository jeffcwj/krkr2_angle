## 术语预览

本节将涉及以下术语，先有个印象，正文中会逐一详细讲解：

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| 单元测试 | Unit Test | 针对单个函数/类的隔离测试，不依赖完整运行环境 |
| TEST_CASE | — | Catch2 的测试用例宏，类似 JUnit 中的 @Test |
| SECTION | — | Catch2 的子测试块，在同一 TEST_CASE 中隔离不同场景 |
| REQUIRE | — | Catch2 断言宏，失败时立即终止当前测试用例 |
| CHECK | — | Catch2 断言宏，失败时记录错误但继续执行 |
| 测试固件 | Test Fixture | 测试用到的资源文件（二进制文件、文本文件等） |
| catch_discover_tests | — | CMake 函数，将 Catch2 测试用例注册到 CTest |
| 行为等价验证 | Behavioral Equivalence | 验证还原代码与原版 DLL 在相同输入下产生相同输出 |

## 一、逆向还原测试策略

### 1.1 为什么逆向代码特别需要测试

普通代码的测试目标是验证"功能符合设计规格"。逆向还原代码的测试目标则更具体：**验证还原代码与原版 DLL 行为一致**。

这带来了几个独特挑战：

| 挑战 | 说明 |
|------|------|
| **无规格文档** | 没有原始需求文档，只有行为观察 |
| **算法不透明** | 内部算法通过反汇编推断，可能有误解 |
| **边界条件未知** | 原版 DLL 对非法输入的处理方式需要实验确认 |
| **平台差异** | 原版只在 Windows 运行，还原版需跨平台 |

### 1.2 测试层次设计

逆向还原的测试应分为三个层次：

```
层次 3：集成测试（Integration Test）
  └── 完整游戏脚本调用插件，验证端到端行为
  └── 最慢，但最接近真实使用

层次 2：接口测试（Interface Test）
  └── 通过 TJS2 脚本层调用插件方法
  └── 验证 ncbind 注册和参数传递正确

层次 1：单元测试（Unit Test）    ← 本节重点
  └── 直接调用 C++ 类的方法
  └── 最快，最精确，最容易定位问题
```

KrKr2 项目的测试基础设施主要支持**层次 1（单元测试）**，这也是我们的重点。

### 1.3 什么值得测试

对于逆向还原的插件，优先测试：

1. **算法核心逻辑**（LZ4 压缩/解压、文字渲染输出）
2. **输入验证**（空指针、零长度、边界值）
3. **跨平台行为一致性**（字体路径查找、字符编码转换）
4. **错误恢复**（损坏数据的处理）

不需要测试：
- ncbind 注册本身（框架已在 M09 验证）
- 平台无关的 getter/setter（太琐碎）
- 需要完整 TJS2 引擎的行为（归属集成测试）

## 二、项目测试基础设施回顾

### 2.1 测试目录结构

```
tests/
├── CMakeLists.txt              # 主 CMakeLists：find Catch2，configure_file
├── test_config.h.in            # 生成 test_config.h（含 TEST_FILES_PATH）
├── test_files/                 # 测试固件目录
│   ├── tjs2/                   # TJS2 脚本固件
│   └── emote/                  # PSB/PIMG 二进制固件
└── unit-tests/
    ├── core/
    │   ├── movie/              # ffmpeg 测试
    │   ├── tjs2/               # TJS2 引擎测试
    │   └── visual/             # 渲染测试
    └── plugins/                # 插件测试 ← 我们在这里添加
        ├── CMakeLists.txt
        ├── main.cpp            # 自定义入口（spdlog + Catch::Session）
        └── psbfile-dll.cpp     # 现有 psbfile 测试
```

### 2.2 tests/CMakeLists.txt 分析

```cmake
# tests/CMakeLists.txt（完整内容，共 18 行）
cmake_minimum_required(VERSION 3.16)
project(Tests LANGUAGES CXX)

find_package(Catch2 3 REQUIRED)          # 查找 Catch2 v3

include(CTest)                            # 启用 CTest 测试框架
include(Catch)                            # 引入 catch_discover_tests 函数

set(TEST_CONFIG_DIR "${CMAKE_CURRENT_BINARY_DIR}")

add_subdirectory(unit-tests/core/movie)
add_subdirectory(unit-tests/core/visual)
add_subdirectory(unit-tests/core/tjs2)
add_subdirectory(unit-tests/plugins)     # ← 插件测试在这里

set(TEST_FILES_PATH ${CMAKE_CURRENT_SOURCE_DIR}/test_files)
configure_file(test_config.h.in test_config.h)  # 生成含路径的头文件
```

**关键点：** `TEST_FILES_PATH` 宏通过 `configure_file` 嵌入到生成的 `test_config.h` 中。测试代码通过 `#include "test_config.h"` 获得这个路径，用于加载测试固件文件。

### 2.3 plugins 测试的 CMakeLists 模式

查看现有的 `tests/unit-tests/plugins/CMakeLists.txt`（推断其结构，基于 psbfile-dll.cpp 的模式）：

```cmake
# tests/unit-tests/plugins/CMakeLists.txt（参考现有模式）
set(SOURCES
    psbfile-dll.cpp
)

foreach(source ${SOURCES})
    string(REPLACE ".cpp" "" name "${source}")  # 去掉扩展名作为目标名
    add_executable(${name} main.cpp ${source})

    target_include_directories(${name} PRIVATE
        "${TEST_CONFIG_DIR}"                     # 找到生成的 test_config.h
        "${KRKR2CORE_PATH}/"
        "${KRKR2PLUGIN_PATH}/"
    )

    target_link_libraries(${name}
        PRIVATE Catch2::Catch2                   # Catch2 框架
        PUBLIC  krkr2plugin                      # 插件静态库（含所有插件）
    )

    catch_discover_tests(${name})                # 注册到 CTest
endforeach()
```

## 三、为 TextRender 编写测试

### 3.1 测试文件设计

TextRender 的核心功能是字体渲染。但由于它依赖 FreeType 和字体文件，完整的渲染测试需要运行环境支持。我们把测试分成两部分：

- **纯逻辑测试**（不需要字体文件）：字形缓存、参数验证、对象构造
- **渲染集成测试**（需要字体文件）：可选执行，用 `[.slow]` 标签排除

```cpp
// tests/unit-tests/plugins/textrender-test.cpp
//
// TextRender 插件单元测试
// 验证逆向还原的渲染逻辑与接口行为

#include <catch2/catch_test_macros.hpp>
#include <catch2/catch_approx.hpp>       // 浮点数近似比较

#include "textrender.h"
#include "glyph_cache.h"
#include "font_finder.h"
#include "test_config.h"

// ============================================================
// 测试组 1：TextRenderBase 基础属性
// ============================================================

TEST_CASE("TextRenderBase 属性读写", "[textrender][basic]") {
    TextRenderBase tr;

    SECTION("默认值验证") {
        // 验证构造后的默认状态
        // 这些默认值从 setDefault() 的 IDA 反汇编中还原
        REQUIRE(tr.getSize() == Catch::Approx(16.0f));         // 默认字号
        REQUIRE(tr.getColor() == 0xFFFFFFFFu);                  // 默认白色（BGRA）
        REQUIRE(tr.getBold() == false);
        REQUIRE(tr.getItalic() == false);
        REQUIRE(tr.getLineSpacing() == 0);
    }

    SECTION("字体大小范围验证") {
        // 有效字号：1.0 ~ 512.0
        tr.setSize(24.0f);
        REQUIRE(tr.getSize() == Catch::Approx(24.0f));

        tr.setSize(72.0f);
        REQUIRE(tr.getSize() == Catch::Approx(72.0f));

        // 边界值：极小字号
        tr.setSize(1.0f);
        REQUIRE(tr.getSize() == Catch::Approx(1.0f));
    }

    SECTION("颜色格式验证（BGRA）") {
        // KiriKiri 使用 BGRA 格式（不是 RGBA）
        // 纯红色在 BGRA 中是 0x000000FF（B=0, G=0, R=255, A=255）
        uint32_t red_bgra = 0xFF0000FFu;
        tr.setColor(red_bgra);
        REQUIRE(tr.getColor() == red_bgra);

        // 透明度测试
        uint32_t semi_transparent = 0x80FFFFFFu;  // 50% 透明白色
        tr.setColor(semi_transparent);
        REQUIRE(tr.getColor() == semi_transparent);
    }

    SECTION("渲染区域设置") {
        tr.setRenderSize(800, 600);
        // 内部状态通过渲染输出间接验证
        // 直接通过不抛出异常来确认调用成功
        REQUIRE_NOTHROW(tr.setRenderSize(1920, 1080));
        REQUIRE_NOTHROW(tr.setRenderSize(0, 0));   // 零尺寸应该被允许或优雅处理
    }
}

// ============================================================
// 测试组 2：字形缓存行为
// ============================================================

TEST_CASE("GlyphCache LRU 缓存", "[textrender][cache]") {
    // 创建容量为 3 的小缓存，便于测试驱逐行为
    GlyphCache cache(3);

    // 构造测试用字形数据
    GlyphCache::GlyphKey key1 = {U'A', 16.0f, false, false};
    GlyphCache::GlyphKey key2 = {U'B', 16.0f, false, false};
    GlyphCache::GlyphKey key3 = {U'C', 16.0f, false, false};
    GlyphCache::GlyphKey key4 = {U'D', 16.0f, false, false};  // 将触发驱逐

    // 创建简单的字形数据（5x5 像素，全白）
    GlyphCache::GlyphBitmap bitmap1(5, 5, 0xFFu);
    GlyphCache::GlyphBitmap bitmap2(5, 5, 0xAAu);
    GlyphCache::GlyphBitmap bitmap3(5, 5, 0x55u);
    GlyphCache::GlyphBitmap bitmap4(5, 5, 0x00u);

    SECTION("基本插入与查找") {
        REQUIRE(cache.get(key1) == nullptr);     // 缓存为空，查找失败

        cache.put(key1, bitmap1);
        auto result = cache.get(key1);
        REQUIRE(result != nullptr);
        REQUIRE(result->pixels[0] == 0xFFu);     // 验证像素数据一致
    }

    SECTION("LRU 驱逐策略") {
        // 填满缓存
        cache.put(key1, bitmap1);
        cache.put(key2, bitmap2);
        cache.put(key3, bitmap3);

        // 访问 key1（使其变为最近使用）
        cache.get(key1);

        // 插入 key4（应该驱逐 key2，因为 key2 是最久未使用的）
        cache.put(key4, bitmap4);

        REQUIRE(cache.get(key1) != nullptr);  // key1 最近访问，不被驱逐
        REQUIRE(cache.get(key2) == nullptr);  // key2 是 LRU，被驱逐
        REQUIRE(cache.get(key3) != nullptr);  // key3 还在
        REQUIRE(cache.get(key4) != nullptr);  // key4 刚插入，还在
    }

    SECTION("相同 key 不同字号不冲突") {
        // 同一字符但不同字号应该有不同的缓存条目
        GlyphCache::GlyphKey key_16 = {U'A', 16.0f, false, false};
        GlyphCache::GlyphKey key_24 = {U'A', 24.0f, false, false};

        GlyphCache::GlyphBitmap bmp_16(4, 4, 0x11u);
        GlyphCache::GlyphBitmap bmp_24(6, 6, 0x22u);

        cache.put(key_16, bmp_16);
        cache.put(key_24, bmp_24);

        auto r16 = cache.get(key_16);
        auto r24 = cache.get(key_24);

        REQUIRE(r16 != nullptr);
        REQUIRE(r24 != nullptr);
        REQUIRE(r16->pixels[0] != r24->pixels[0]);  // 像素数据不同
    }
}

// ============================================================
// 测试组 3：字体查找（跨平台）
// ============================================================

TEST_CASE("FontFinder 字体目录", "[textrender][fonts]") {
    SECTION("系统字体目录不为空") {
        auto dirs = FontFinder::getSystemFontDirs();
        // 每个平台都应该至少有一个字体目录
        REQUIRE(!dirs.empty());
    }

    SECTION("字体目录路径为绝对路径") {
        auto dirs = FontFinder::getSystemFontDirs();
        for(const auto& dir : dirs) {
            // std::filesystem::path::is_absolute() 跨平台判断
            REQUIRE(dir.is_absolute());
        }
    }
}

// ============================================================
// 测试组 4：渲染集成测试（需要字体文件，默认跳过）
// ============================================================
// "[.slow]" 标签：默认不执行，需要 --tags "[.slow]" 才运行

TEST_CASE("TextRender 实际渲染输出", "[textrender][render][.slow]") {
    TextRenderBase tr;

    // 寻找系统中存在的测试字体
    auto dirs = FontFinder::getSystemFontDirs();
    bool fontFound = false;
    for(const auto& dir : dirs) {
        if(std::filesystem::exists(dir)) {
            fontFound = true;
            break;
        }
    }

    if(!fontFound) {
        SKIP("未找到系统字体目录，跳过渲染测试");
    }

    SECTION("单字符渲染不为全零") {
        tr.setSize(16.0f);
        tr.setRenderSize(64, 64);

        // 渲染字符 'A'
        auto bitmap = tr.renderChar(U'A');
        REQUIRE(bitmap != nullptr);
        REQUIRE(bitmap->width > 0);
        REQUIRE(bitmap->height > 0);

        // 非空字符渲染结果不应该全为 0
        bool hasNonZero = false;
        for(size_t i = 0; i < bitmap->data.size(); i++) {
            if(bitmap->data[i] != 0) {
                hasNonZero = true;
                break;
            }
        }
        REQUIRE(hasNonZero);
    }
}
```

### 3.2 LZ4 压缩测试

PackinOne 的核心功能是 LZ4 压缩。测试策略是**往返验证**（round-trip）：压缩后再解压，得到原始数据。

```cpp
// tests/unit-tests/plugins/packinone-test.cpp
//
// PackinOne 插件单元测试
// 重点验证 LZ4 压缩/解压的往返正确性

#include <catch2/catch_test_macros.hpp>
#include <catch2/generators/catch_generators.hpp>  // 参数化测试

#include "lz4_stream.h"

// ============================================================
// 工具函数
// ============================================================

// 生成测试数据（简单的重复模式，高度可压缩）
static std::vector<uint8_t> makeTestData(size_t size, uint8_t pattern = 0xAB) {
    std::vector<uint8_t> data(size);
    for(size_t i = 0; i < size; i++) {
        data[i] = static_cast<uint8_t>((i * 137 + pattern) & 0xFF);
    }
    return data;
}

// ============================================================
// 测试组 1：LZ4 基础往返
// ============================================================

TEST_CASE("LZ4Stream 压缩解压往返", "[packinone][lz4]") {
    LZ4Stream lz4;

    SECTION("小数据（< 64 字节）") {
        auto original = makeTestData(32);
        auto compressed = lz4.compress(original);
        auto decompressed = lz4.decompress(compressed, original.size());

        REQUIRE(decompressed == original);
    }

    SECTION("中等数据（64 KB）") {
        auto original = makeTestData(64 * 1024);
        auto compressed = lz4.compress(original);
        auto decompressed = lz4.decompress(compressed, original.size());

        REQUIRE(decompressed == original);
        // 验证压缩实际有效（压缩率 < 100%）
        INFO("原始大小: " << original.size() << ", 压缩后: " << compressed.size());
        REQUIRE(compressed.size() < original.size());
    }

    SECTION("高度可压缩数据（全零）") {
        std::vector<uint8_t> zeros(1024 * 1024, 0x00);  // 1 MB 全零
        auto compressed = lz4.compress(zeros);
        auto decompressed = lz4.decompress(compressed, zeros.size());

        REQUIRE(decompressed == zeros);
        // 全零数据压缩率应该非常高（远小于 1%）
        REQUIRE(compressed.size() < zeros.size() / 10);
    }

    SECTION("不可压缩数据（伪随机）") {
        // 使用高熵数据，压缩后可能比原始更大（正常）
        std::vector<uint8_t> random_data;
        random_data.reserve(4096);
        uint32_t seed = 0x12345678;
        for(size_t i = 0; i < 4096; i++) {
            seed = seed * 1664525u + 1013904223u;  // LCG 伪随机
            random_data.push_back(static_cast<uint8_t>(seed >> 24));
        }

        auto compressed = lz4.compress(random_data);
        auto decompressed = lz4.decompress(compressed, random_data.size());
        REQUIRE(decompressed == random_data);
    }
}

// ============================================================
// 测试组 2：错误处理
// ============================================================

TEST_CASE("LZ4Stream 错误处理", "[packinone][lz4][error]") {
    LZ4Stream lz4;

    SECTION("解压空数据不崩溃") {
        std::vector<uint8_t> empty;
        // 应该返回空结果或抛出可捕获的异常，而不是崩溃
        REQUIRE_NOTHROW([&]() {
            try {
                auto result = lz4.decompress(empty, 0);
                REQUIRE(result.empty());
            } catch(const std::exception&) {
                // 抛出异常也可接受，只要不是段错误
            }
        }());
    }

    SECTION("解压损坏数据不崩溃") {
        std::vector<uint8_t> corrupted = {0xDE, 0xAD, 0xBE, 0xEF, 0x00, 0x00};
        REQUIRE_NOTHROW([&]() {
            try {
                auto result = lz4.decompress(corrupted, 1024);
            } catch(const std::exception&) {
                // 损坏数据抛出异常是正确行为
            }
        }());
    }
}

// ============================================================
// 测试组 3：参数化测试（不同数据大小）
// ============================================================

TEST_CASE("LZ4Stream 多种数据大小", "[packinone][lz4][parameterized]") {
    // 使用 Catch2 GENERATE 进行参数化测试
    auto size = GENERATE(1, 16, 256, 4096, 65536, 1024 * 1024);

    LZ4Stream lz4;
    auto original = makeTestData(size);
    auto compressed = lz4.compress(original);
    auto decompressed = lz4.decompress(compressed, original.size());

    REQUIRE(decompressed.size() == original.size());
    REQUIRE(decompressed == original);
}
```

## 四、集成到 CMake 构建系统

### 4.1 修改 tests/unit-tests/plugins/CMakeLists.txt

在现有插件测试的 CMakeLists.txt 中添加新测试文件：

```cmake
# tests/unit-tests/plugins/CMakeLists.txt
# 修改：在 SOURCES 中添加新测试文件

set(SOURCES
    psbfile-dll.cpp
    textrender-test.cpp      # 新增：TextRender 测试
    packinone-test.cpp       # 新增：PackinOne 测试
)

foreach(source ${SOURCES})
    string(REPLACE ".cpp" "" name "${source}")
    add_executable(${name} main.cpp ${source})

    target_include_directories(${name} PRIVATE
        "${TEST_CONFIG_DIR}"
        "${KRKR2CORE_PATH}/"
        "${KRKR2PLUGIN_PATH}/"
        # 添加逆向插件的头文件路径
        "${KRKR2PLUGIN_PATH}/textrender"
        "${KRKR2PLUGIN_PATH}/packinone"
    )

    target_link_libraries(${name}
        PRIVATE Catch2::Catch2
        PUBLIC  krkr2plugin
    )

    catch_discover_tests(${name})
endforeach()
```

### 4.2 添加测试固件文件

如果测试需要二进制固件文件（如渲染测试可能需要字体文件），放入 `tests/test_files/plugins/` 目录：

```bash
# 创建固件目录
```

### 4.3 运行测试

```bash
# 编译（包含测试）
cmake --preset="Windows Debug Config"
cmake --build --preset="Windows Debug Build"

# 运行所有测试
```

### 5.2 为逆向插件添加 CI 注意事项

逆向还原的插件测试有一个特殊性：**字体渲染测试在 CI 环境可能找不到系统字体**。

解决方案：

**方案 A：跳过需要字体的测试（推荐）**

用 `[.slow]` 标签标记，CI 不执行这类测试：

```yaml
- name: 运行测试（排除渲染测试）
  run: ctest --test-dir out/linux/debug --output-on-failure -E "slow"
```

**方案 B：在 CI 安装字体**

```yaml
- name: 安装测试字体
  run: |
    sudo apt-get install -y fonts-noto-cjk
    fc-cache -f  # 更新字体缓存
```

**方案 C：使用内嵌字体**

将一个极小的开源字体（如 Unifont 子集）放入 `tests/test_files/plugins/`，测试代码直接使用固件字体，不依赖系统安装。

```cpp
TEST_CASE("使用内嵌测试字体渲染", "[textrender][render]") {
    // 使用固件字体，不依赖系统字体安装
    std::string fontPath = std::string(TEST_FILES_PATH) + "/plugins/unifont_subset.ttf";
    REQUIRE(std::filesystem::exists(fontPath));

    TextRenderBase tr;
    tr.setFontFile(fontPath);  // 直接指定字体文件路径
    tr.setSize(16.0f);
    tr.setRenderSize(64, 64);

    auto bitmap = tr.renderChar(U'A');
    REQUIRE(bitmap != nullptr);
    REQUIRE(bitmap->width > 0);
}
```

## 动手实践

### 实践任务：为一个简单算法编写往返测试

假设你逆向了一个简单的 XOR 混淆算法（PackinOne 流加密中的一部分），实现如下：

```cpp
// cpp/plugins/packinone/stream_crypt.h
class StreamCrypt {
public:
    // XOR 流加密/解密（加密与解密使用同一算法）
    static std::vector<uint8_t> process(
        const std::vector<uint8_t>& data,
        uint32_t key
    ) {
        std::vector<uint8_t> result(data.size());
        uint32_t state = key;
        for(size_t i = 0; i < data.size(); i++) {
            state = state * 1664525u + 1013904223u;  // LCG 更新密钥流
            result[i] = data[i] ^ static_cast<uint8_t>(state >> 24);
        }
        return result;
    }
};
```

**要求：** 为 `StreamCrypt::process` 编写完整的 Catch2 测试，验证：
1. 加密后解密（使用相同 key）得到原始数据
2. 不同 key 产生不同结果
3. 空输入不崩溃

## 常见错误与排查

### 错误 1：链接错误 "undefined symbol: LZ4Stream::compress"

**症状：**

```
undefined reference to `LZ4Stream::compress(std::vector<unsigned char, ...>)'
```

**原因：** 测试的 CMakeLists.txt 链接了 `krkr2plugin`，但 `packinone` 子库还没有被添加到 `krkr2plugin` 的链接链中（`01-静态库编译与插件注册.md` 第2节的 `target_link_libraries` 没有加 `packinone`）。

**解决：** 确认 `cpp/plugins/CMakeLists.txt` 中已经添加了：
```cmake
add_subdirectory(packinone)
target_link_libraries(krkr2plugin PUBLIC packinone)
```

### 错误 2：`SKIP()` 宏找不到

**症状：**

```
error: 'SKIP' was not declared in this scope
```

**原因：** `SKIP()` 是 Catch2 v3.3+ 新增的宏，旧版本没有。

**解决：** 有两种替代方案：

```cpp
// 方案 1：使用 INFO + SUCCEED（在老版本 Catch2 中）
if(!condition) {
    INFO("跳过测试：条件不满足");
    SUCCEED("测试跳过");
    return;
}

// 方案 2：检查 Catch2 版本
#if CATCH_VERSION_MAJOR >= 3 && CATCH_VERSION_MINOR >= 3
    SKIP("原因说明");
#else
    SUCCEED("跳过：不支持 SKIP 宏");
    return;
#endif
```

### 错误 3：TEST_FILES_PATH 未定义

**症状：**

```
error: 'TEST_FILES_PATH' was not declared
```

**原因：** 测试文件忘记 `#include "test_config.h"`，或者 CMakeLists.txt 中没有添加 TEST_CONFIG_DIR 到 include 路径。

**解决：**

```cmake
# 在 CMakeLists.txt 中确认包含：
target_include_directories(${name} PRIVATE
    "${TEST_CONFIG_DIR}"   # 这一行是关键，TEST_CONFIG_DIR 在 tests/CMakeLists.txt 中定义
)
```

```cpp
// 在测试文件中确认包含：
#include "test_config.h"  // 必须在使用 TEST_FILES_PATH 之前
```

## 本节小结

- **测试策略**：逆向还原代码的测试目标是行为等价验证，分三层（单元/接口/集成），项目重点支持单元测试
- **往返测试**：压缩/加密等算法首选往返测试（压缩再解压得原始数据），简单可靠
- **参数化测试**：用 Catch2 的 `GENERATE` 对多种数据大小、边界值批量验证
- **测试标签**：需要系统资源（字体文件）的测试用 `[.slow]` 标签，CI 默认跳过
- **CMake 集成**：新测试文件加入 `SOURCES` 列表，CMake foreach 自动创建测试可执行文件
- **插件清单更新**：逆向完成后将 `doc/krkr2_plugins.md` 的 `nan` 改为实现路径和说明

### 题目 1：为 StreamCrypt 编写 Catch2 测试

使用动手实践中给出的 `StreamCrypt::process` 函数，编写完整测试文件，包含往返测试、不同 key 测试和空输入测试。

<details>
<summary>查看答案</summary>

```cpp
// tests/unit-tests/plugins/streamcrypt-test.cpp
#include <catch2/catch_test_macros.hpp>
#include <catch2/generators/catch_generators.hpp>

#include "stream_crypt.h"

TEST_CASE("StreamCrypt 往返加密", "[packinone][crypt]") {

    SECTION("基本往返验证") {
        std::vector<uint8_t> plaintext = {0x48, 0x65, 0x6C, 0x6C, 0x6F};  // "Hello"
        uint32_t key = 0xDEADBEEF;

        // 加密
        auto ciphertext = StreamCrypt::process(plaintext, key);
        REQUIRE(ciphertext != plaintext);  // 加密后应与原始不同

        // 解密（使用相同 key）
        auto decrypted = StreamCrypt::process(ciphertext, key);
        REQUIRE(decrypted == plaintext);   // 解密后应与原始相同
    }

    SECTION("不同 key 产生不同结果") {
        std::vector<uint8_t> data = {0x01, 0x02, 0x03, 0x04};

        auto result1 = StreamCrypt::process(data, 0x11111111);
        auto result2 = StreamCrypt::process(data, 0x22222222);

        REQUIRE(result1 != result2);
    }

    SECTION("空输入不崩溃") {
        std::vector<uint8_t> empty;
        REQUIRE_NOTHROW(StreamCrypt::process(empty, 0x12345678));
        auto result = StreamCrypt::process(empty, 0x12345678);
        REQUIRE(result.empty());
    }

    SECTION("多种数据大小的往返测试") {
        auto size = GENERATE(1, 16, 100, 1024, 65536);
        uint32_t key = 0xCAFEBABE;

        // 生成测试数据
        std::vector<uint8_t> original(size);
        for(size_t i = 0; i < size; i++) {
            original[i] = static_cast<uint8_t>(i & 0xFF);
        }

        auto encrypted = StreamCrypt::process(original, key);
        auto decrypted = StreamCrypt::process(encrypted, key);
        REQUIRE(decrypted == original);
    }
}
```

### 题目 2：分析 psbfile-dll.cpp 的测试结构

阅读 `tests/unit-tests/plugins/psbfile-dll.cpp`，回答：
1. 该文件有几个 TEST_CASE？分别测试什么？
2. 嵌套的 SECTION 有什么作用？
3. `CAPTURE` 宏的作用是什么？

<details>
<summary>查看答案</summary>

1. **两个 TEST_CASE：**
   - `"read psbfile ezsave.pimg"`：测试读取 .pimg 格式文件，验证图层数量、属性（宽高、名称、ID 等）和资源收集
   - `"read psbfile e-mote3.0 psb"`：测试读取加密的 .psb 格式文件，验证解密和格式识别

2. **嵌套 SECTION 的作用：**
   - `SECTION` 实现"独立的测试分支"。Catch2 会对每个叶子 SECTION 单独运行整个 TEST_CASE（从头开始，但只执行到那个 SECTION）
   - 在示例中，`"check width height"` 和 `"get layers"` 是两个独立的 SECTION，各自独立地从文件加载开始执行，互不影响
   - `"check layer properties"` 嵌套在 `"get layers"` 中，只有在 `"get layers"` 的 REQUIRE 通过后才会运行

3. **`CAPTURE` 宏的作用：**
   - `CAPTURE(header.version, f.getType())` 将这两个值添加到测试失败时的输出信息中
   - 如果后续的 REQUIRE 失败，错误报告中会显示 `header.version = X` 和 `f.getType() = Y`，方便诊断
   - 相当于给失败报告附上"当时的上下文变量值"

</details>

### 题目 3：解释测试固件机制

为什么项目用 `TEST_FILES_PATH` 宏而不是直接硬编码路径？`configure_file` 在这里做了什么？

<details>
<summary>查看答案</summary>

**为什么用宏而非硬编码路径：**

硬编码路径（如 `"/home/user/project/tests/test_files"`）的问题：
- 不同开发者的项目路径不同
- CI 服务器路径与本地不同
- Windows/Linux/macOS 路径分隔符不同

`TEST_FILES_PATH` 宏由 CMake 在配置阶段注入实际路径，每个环境都正确。

**`configure_file` 做了什么：**

`tests/test_config.h.in` 内容大致为：
```cpp
// 这是模板文件，@ 符号之间是 CMake 变量
#define TEST_FILES_PATH "@TEST_FILES_PATH@"
```

执行 `configure_file(test_config.h.in test_config.h)` 后，CMake 将 `@TEST_FILES_PATH@` 替换为实际路径：
```cpp
// 生成的 test_config.h（在 build 目录中）
#define TEST_FILES_PATH "C:/Users/dev/project/krkr2/tests/test_files"
// 或者在 Linux：
#define TEST_FILES_PATH "/home/dev/project/krkr2/tests/test_files"
```

这样测试代码就能用 `TEST_FILES_PATH "/emote/ezsave.pimg"` 构造正确的路径，跨平台、跨环境都能工作。

</details>
