## 本节目标

1. 理解 kirikiroid2 适配 Windows 插件的**三层策略**：常量补定义、类型 shim、方法 stub 化

## 术语预览

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| Stub | Stub | 用空函数体或固定返回值替代真实实现的占位代码，保持编译通过但功能不工作 |
| 条件编译 | Conditional Compilation | 用 `#ifdef`/`#ifndef` 根据预处理器宏决定是否编译某段代码 |
| 类型 shim | Type Shim | 用 `typedef` 或 `using` 为不存在的平台类型提供替代定义，让代码能在其他平台编译 |
| HWND | Window Handle | Windows 窗口句柄（不透明整数），跨平台版本定义为 `void*` 或自定义类型 |
| ncbPropAccessor | ncbPropAccessor | ncbind 框架中用于读写 TJS 对象属性的辅助类 |

## 1. 适配背景：为什么不能直接编译原版插件

windowEx 原始版本的 `main.cpp` 第 1 行就是：

```cpp
#include <windows.h>   // 引入整个 Win32 API 头文件树（数万行声明）
#include <dbt.h>       // 设备通知（Device Broadcast Types）
```

在 Android/Linux/macOS 上，`windows.h` 根本不存在。更深层的问题不只是头文件：

| 问题层次 | 举例 | 解决方向 |
|---------|------|---------|
| **类型不存在** | `HWND`, `HMENU`, `HDC`, `HBITMAP`, `HICON` | 用 `win32_dt.h` 提供 typedef |
| **常量不存在** | `WM_SIZE`, `WM_SYSCOMMAND`, `SC_MINIMIZE` | 用 `#ifndef _WIN32` 块补定义 |
| **API 不存在** | `::GetWindowRect()`, `::PostMessage()` | stub 化（空实现或替代逻辑） |
| **机制不存在** | `registerMessageReceiver`（Win32 消息循环） | 整体删除该依赖 |
| **硬件不存在** | 最大化/最小化按钮（移动端无标题栏概念） | stub 返回固定值 |

kirikiroid2 的解决方案**不是**"把 Win32 API 全部在 Android 上重新实现"，而是：
- **能用跨平台替代方案的** → 换一种方式实现（如 `getWindowRect` 改读 TJS 属性）
- **根本没有对应概念的** → stub 化（空函数体，或返回合理固定值）
- **整个机制无法移植的** → 完全删除（消息接收器）

## 2. 三层适配策略

### 策略一：常量补定义（#ifndef _WIN32 块）

原版用 `#include <windows.h>` 引入所有 Win32 常量。适配版的解决方案：**在 `#ifndef _WIN32` 块里手动补定义这些常量的数值**。

```cpp
// 来自 cpp/plugins/windowEx.cpp，第 25-313 行（节选关键部分）

#ifndef _WIN32
// WM_* 消息编号——这些都是 Win32 规范中固定的整数值
#define WM_NULL         0x0000
#define WM_CREATE       0x0001
#define WM_DESTROY      0x0002
#define WM_SIZE         0x0005   // 窗口大小改变
#define WM_SYSCOMMAND   0x0112   // 系统命令（最大化/最小化等）
#define WM_HOTKEY       0x0312   // 全局热键触发
#define WM_MOVING       0x0216   // 窗口正在被拖动
// ... 共约 150 个 WM_* 常量全部手动定义

// HT* 命中测试结果——表示鼠标在窗口哪个区域
#define HTCLIENT    1   // 客户区
#define HTCAPTION   2   // 标题栏
#define HTSYSMENU   3   // 系统菜单按钮
#define HTLEFT      10  // 左边框
// ... 其他 HTBORDER/HTBOTTOM 等
#endif
```

> **为什么要补定义这些常量？**
>
> 虽然 stub 化后的函数实际上不会真正处理消息，但 `onMessage` 的 switch/case 语句、`bitHooks` 位集合的初始化、以及 `_getNotificationVariant` 函数里的大型 WM 映射表——这些代码仍然需要编译通过。编译器不在乎这些 case 分支在运行时是否被触达，但它需要能解析 `WM_SIZE`、`HTCLIENT` 等标识符。

**数量统计**：适配版的 `#ifndef _WIN32` 块中定义了约 **165 个** WM_* 常量和 **14 个** HT* 常量。这些值与 Win32 规范完全一致（同一个整数），因此如果未来有平台真正需要处理这些消息，数值不会出错。

### 策略二：类型 Shim（win32_dt.h）

`win32_dt.h` 是适配层专门为非 Windows 平台提供的"类型替身"头文件：

```cpp
// 来自 cpp/plugins/win32_dt.h（完整内容）

#pragma once

#if !defined(_WIN32) && !defined(_WIN64)
// Win32 句柄类型——在非 Windows 平台上定义为 void*
// 这样代码里的 HWND hwnd = nullptr; 在 Android/Linux 上也能编译
typedef void *HWND;     // 窗口句柄
typedef void *HMENU;    // 菜单句柄
typedef void *HDC;      // 设备上下文句柄（Device Context）
typedef void *HBITMAP;  // 位图句柄
typedef void *HICON;    // 图标句柄
typedef void *HDEVNOTIFY;  // 设备通知注册句柄

// 简单数值类型——在 Linux/Android 标准库中没有这些别名
typedef unsigned long  DWORD;    // 32 位无符号整数
typedef unsigned short WORD;     // 16 位无符号整数
typedef unsigned char  BYTE;     // 8 位无符号整数
typedef long           LONG;
typedef int            BOOL;

// 矩形结构体（与 Win32 RECT 兼容的跨平台版本）
// 注意：KiriKiri 自有 tTVPRect，但 windowEx 直接用 RECT
struct RECT {
    LONG left, top, right, bottom;
};

#endif  // !defined(_WIN32)
```

> **Shim 的核心思路**：不是让 `HWND` 真正"能用"（你不能真的把 `void*` 传给 `SetWindowPos`），而是让代码**能编译**。在 stub 化后的函数体内，这些句柄永远是 `nullptr`，函数在验证句柄非空后直接返回。

### 策略三：方法 Stub 化

Stub 化有三种程度，从轻到重：

**程度 1：替代实现（功能替换）** — 用跨平台的方式实现相同语义

```cpp
// 原版：调用 Win32 GetWindowRect API
static tjs_error getWindowRect(..., iTJSDispatch2 *obj) {
    RECT rect;
    HWND hwnd = GetHWND(obj);
    if (hwnd != NULL && ::GetWindowRect(hwnd, &rect)) {
        // 返回包含 x/y/w/h 的字典
    }
}

// 适配版：改读 TJS Window 对象的属性（cpp/plugins/windowEx.cpp，第 398-425 行）
static tjs_error getWindowRect(..., iTJSDispatch2 *obj) {
    WindowEx::GetHWND(obj);  // 调用但不使用结果（维持副作用）
    if(r != nullptr) r->Clear();

    // 改用 TJS 属性读取——Window 对象本身记录了 left/top/width/height
    ncbPropAccessor ncbPropAcc{ obj };
    tjs_int left   = ncbPropAcc.getIntValue(TJS_W("left"));
    tjs_int top    = ncbPropAcc.getIntValue(TJS_W("top"));
    tjs_int width  = ncbPropAcc.getIntValue(TJS_W("width"));
    tjs_int height = ncbPropAcc.getIntValue(TJS_W("height"));

    // 构造 tTVPRect 并填入字典
    tTVPRect rect{ left, top, width + left, height + top };
    ncbDictionaryAccessor ncbDictAcc;
    ncbDictAcc.SetValue(TJS_W("x"), rect.left);
    ncbDictAcc.SetValue(TJS_W("y"), rect.top);
    ncbDictAcc.SetValue(TJS_W("w"), rect.right  - rect.left);
    ncbDictAcc.SetValue(TJS_W("h"), rect.bottom - rect.top);
    auto *dis = ncbDictAcc.GetDispatch();
    r->SetObject(dis, dis);
    return TJS_S_OK;
}
```

> **这个替代方案的合理性**：在 Android/移动端，游戏通常全屏运行，`Window` 对象的 `left/top/width/height` 属性正好反映了渲染区域的位置和尺寸，语义上与"窗口外边框矩形"相近。这不是精确的 Win32 语义，但对游戏脚本的实际用途（获取游戏区域坐标）而言已经足够。

**程度 2：固定值 stub** — 返回硬编码的"合理"值

```cpp
// 适配版 cpp/plugins/windowEx.cpp，第 464-476 行

// 最大化状态：移动端游戏几乎总是全屏=最大化，固定返回 true
static bool isMaximized(iTJSDispatch2 *obj) { return true; }

static tjs_error getMaximized(...) {
    if(r) *r = isMaximized(obj);  // 始终 true
    return TJS_S_OK;
}

// 最小化状态：移动端不能最小化，固定返回 false
static bool isMinimized(iTJSDispatch2 *obj) { return false; }

static tjs_error getMinimized(...) {
    if(r) *r = isMinimized(obj);  // 始终 false
    return TJS_S_OK;
}
```

**程度 3：空函数体 stub** — 完全不做任何事

```cpp
// 适配版 cpp/plugins/windowEx.cpp，第 363-367 行

// 最小化命令：移动端没有"最小化到任务栏"的概念，直接空实现
static tjs_error minimize(tTJSVariant *r, tjs_int n, tTJSVariant **p,
                          iTJSDispatch2 *obj) {
    return TJS_S_OK;   // 什么都不做，但返回成功
}

// 同样是空 stub：
static tjs_error maximize(...)     { return TJS_S_OK; }
static tjs_error showRestore(...)  { return TJS_S_OK; }
static tjs_error setWindowIcon(...){ return TJS_S_OK; }
static tjs_error resetWindowIcon(...)  { return TJS_S_OK; }
static tjs_error setWindowCornerPreference(...) { return TJS_S_OK; }
static tjs_error resetExSystemMenu(...)         { return TJS_S_OK; }
```

## 3. 消息接收器的完全弃用

这是最重大的架构变化。原版的 `regist()` 是整个插件的核心，适配版将其完全清空：

```cpp
// 原版（stuffs/wtnbgo_windowex_src/main.cpp，第 877-883 行）
void regist(bool en) {
    tTJSVariant mode(en ? wrmRegister : wrmUnregister);
    tTJSVariant func((tTVInteger)(&receiver));
    tTJSVariant data((tTVInteger)(self));
    tTJSVariant rslt, *params[] = { &mode, &func, &data };
    self->FuncCall(0, TJS_W("registerMessageReceiver"), 0, &rslt, 3, params, self);
}

// 适配版（cpp/plugins/windowEx.cpp，第 709 行）
void regist(bool en) {}   // 完全空函数体
```

### 为什么可以直接删除？

`registerMessageReceiver` 是 KiriKiri 引擎的 Win32 消息循环扩展点。在 Android/Linux/macOS 上：

1. **没有 Win32 消息循环**：引擎用 Cocos2d-x/SDL 的事件系统，不经过 Win32 消息泵
2. **没有 `tTVPWindowMessage`**：该结构体封装的是 Win32 `MSG`，跨平台版本中不存在
3. **所有依赖消息接收器的功能都已 stub**：`onMinimize`、`onMoving` 等事件不会被触发，`disableResize`/`disableMove` 不会真正工作

### 连带影响：构造函数简化

```cpp
// 原版构造函数（第 886-905 行）
WindowEx(iTJSDispatch2 *obj)
    : self(obj), menuex(0),
      sysMenuModified(0), sysMenuModMap(0),
      cachedHWND(0),          // Win32 句柄缓存
      sysMenu(0),             // Win32 菜单句柄
      externalIcon(0),        // Win32 图标句柄
      deviceNotify(0),        // Win32 设备通知句柄
      hasResizing(false),
      hasMoving(false),
      hasMove(false),
      disableResize(false),
      disableMove(false),
      enableNCMEvent(false),
      enableWinMsgHook(false),
      ovbmp(NULL)
{
    cachedHWND = GetHWND(self);   // 缓存 HWND
    regist(true);                 // 注册消息接收器
    setMessageHookAll(false);
}

// 适配版构造函数（第 712-718 行）
WindowEx(iTJSDispatch2 *obj) :
    self(obj), menuex(0), sysMenuModified(0), sysMenuModMap(0),
    disableResize(false), disableMove(false), enableNCMEvent(false),
    enableWinMsgHook(false) {
    // 注意：没有 cachedHWND（已从成员变量中删除）
    // 注意：没有 sysMenu/externalIcon/deviceNotify（也已删除）
    regist(true);           // 调用空 stub，无实际效果
    setMessageHookAll(false);
}

// 适配版析构函数（第 720-728 行）
~WindowEx() {
    if(menuex)          menuex         ->Release();  // TJS 引用计数管理保留
    if(sysMenuModified) sysMenuModified->Release();
    // 原版析构中的这些全部删除：
    // if (deviceNotify)    ::UnregisterDeviceNotification(deviceNotify);
    // if (externalIcon)    ::DestroyIcon(externalIcon);
    resetSystemMenu();
    deleteOverlayBitmap();
    regist(false);   // 调用空 stub
}
```

## 4. 六大改动类型对比表

```
╔══════════════════════════════════════════════════════════════════╗
║  改动类型            │ 原版行为        │ 适配版行为       │ 原因
╠══════════════════════════════════════════════════════════════════╣
║  1. 替代实现         │ Win32 API 调用  │ 读 TJS 属性      │ 跨平台替代
║  2. 固定值 stub      │ 读真实 API 状态 │ 返回硬编码值     │ 平台无此概念
║  3. 空函数 stub      │ 执行 Win32 操作 │ 什么也不做       │ 平台无此能力
║  4. 架构性删除       │ 消息接收器注册  │ 完全空实现       │ 机制不存在
║  5. 注释掉的方法     │ 暴露给 TJS      │ 注册代码被注释   │ 避免误导
║  6. OverlayBitmap   │ 完整 Win32 子窗 │ 骨架保留 setBitmap │ 代码存在但无效
╚══════════════════════════════════════════════════════════════════╝
```

### 注释掉的方法（类型 5）详解

适配版的 `NCB_ATTACH_CLASS_WITH_HOOK` 注册块中，有几个方法注册被用注释禁用：

```cpp
// cpp/plugins/windowEx.cpp，第 962-977 行

NCB_ATTACH_CLASS_WITH_HOOK(WindowEx, Window) {
    // ... 常量注册 ...

    RawCallback(TJS_W("minimize"), &Class::minimize, 0);
    RawCallback(TJS_W("maximize"), &Class::maximize, 0);

    // !! 下面这些被注释掉了 !!
    // RawCallback(TJS_W("maximizeBox"), &Class::getMaximizeBox,
    //             &Class::setMaximizeBox, 0);
    // RawCallback(TJS_W("minimizeBox"), &Class::getMinimizeBox,
    //             &Class::setMinimizeBox, 0);
    // RawCallback(TJS_W("maximized"),   &Class::getMaximized,
    //             &Class::setMaximized, 0);
    // RawCallback(TJS_W("minimized"),   &Class::getMinimized,
    //             &Class::setMinimized, 0);
    RawCallback(TJS_W("showRestore"), &Class::showRestore, 0);

    // ...
    // RawCallback(TJS_W("disableMove"), &Class::getDisableMove,
    //             &Class::setDisableMove, 0);
    // RawCallback(TJS_W("resetExSystemMenu"), &Class::resetExSystemMenu, 0);
}
```

注意：这些方法的 **C++ 实现仍然存在**，只是 **TJS 注册被注释掉了**。

这意味着：TJS 脚本调用 `window.maximizeBox` 会得到 `undefined`（属性不存在），而不是调用 stub 实现。这是有意为之的设计决策——宁愿让脚本报错，也不愿提供一个"看起来工作实际上什么都没做"的接口。

**为什么 `disableResize` 被保留但 `disableMove` 被注释？**

- `disableResize`：在原版中是通过 `WM_NCHITTEST` 实现的。适配版虽然不处理 Win32 消息，但这个标志的值仍然会被读写——某些平台相关代码可能在某种程度上响应它
- `disableMove`：在原版同样通过 `WM_NCHITTEST` + 菜单禁用实现。适配者认为这个功能在移动端完全无意义（移动端用户不会拖动窗口），不如彻底注释掉

## 5. OverlayBitmap 的骨架保留

OverlayBitmap 子类在适配版中采用了"骨架保留"策略——代码结构存在，但所有 Win32 操作都被替换为空实现：

```cpp
// 适配版 cpp/plugins/windowEx.cpp，第 900-916 行

struct OverlayBitmap {
    OverlayBitmap() : overlay(0), bitmap(0), bmpdc(0) {}

    bool setBitmap(iTJSDispatch2 *win, iTJSDispatch2 *lay) {
        // 验证 lay 是否为 Layer 对象（保留了类型检查逻辑）
        if(!lay || !lay->IsInstanceOf(0, 0, 0, TJS_W("Layer"), lay))
            return false;
        WindowEx::GetHWND(win);  // 调用但不使用 HWND
        return true;             // 直接返回 true（不创建任何窗口）
    }

private:
    HWND    overlay;  // 定义存在，但永远是 nullptr
    HBITMAP bitmap;
    HDC     bmpdc;
    tjs_int bmpx, bmpy, bmpw, bmph;
    // static ATOM WindowClass;  // 被注释掉了
} *ovbmp;
```

对比原版（`stuffs/wtnbgo_windowex_src/main.cpp`，第 1163-1195 行），原版的 `setBitmap` 共 30+ 行，涉及：
- `::EnumChildWindows` 寻找滚动框子窗口
- `::CreateCompatibleDC` + `CopyLayerToBitmap` 复制像素
- `::GetWindowRect` + `::SetParent` + `::SetWindowPos` 定位叠加窗口
- `::ShowWindow` + `::UpdateWindow` 显示

适配版完全跳过了所有这些，但保留了骨架结构，原因是：
1. `WindowEx` 的成员变量和析构函数引用了 `ovbmp`（`deleteOverlayBitmap`），骨架保留避免了大规模重构
2. 未来如果某个平台有类似能力（如 Android 的 SurfaceView 叠加），可以在这个骨架上继续实现

## 6. getWindowRect 的完整适配分析

`getWindowRect` 是适配质量最高的函数，值得深入分析。

### 原版实现

```cpp
// stuffs/wtnbgo_windowex_src/main.cpp，第 242-251 行
static tjs_error TJS_INTF_METHOD getWindowRect(
    tTJSVariant *r, tjs_int n, tTJSVariant **p, iTJSDispatch2 *obj)
{
    RECT rect;
    HWND hwnd = GetHWND(obj);  // 从 TJS 读取 HWND 属性（真实 Win32 句柄）
    if (r) r->Clear();
    if (hwnd != NULL && ::GetWindowRect(hwnd, &rect)) {
        // GetWindowRect 返回屏幕坐标系下的外边框矩形
        ncbDictionaryAccessor dict;
        if (SetRect(dict, &rect) && r) *r = tTJSVariant(dict, dict);
    }
    return TJS_S_OK;
}
```

### 适配版实现

```cpp
// cpp/plugins/windowEx.cpp，第 398-425 行
static tjs_error getWindowRect(
    tTJSVariant *r, tjs_int n, tTJSVariant **p, iTJSDispatch2 *obj)
{
    WindowEx::GetHWND(obj);   // 副作用调用：确保 HWND 读取路径不被优化掉
    if(r != nullptr) r->Clear();

    // 核心改变：不调用 Win32 API，改从 TJS Window 对象读属性
    ncbPropAccessor ncbPropAcc{ obj };
    tjs_int left   = ncbPropAcc.getIntValue(TJS_W("left"));   // Window.left
    tjs_int top    = ncbPropAcc.getIntValue(TJS_W("top"));    // Window.top
    tjs_int width  = ncbPropAcc.getIntValue(TJS_W("width"));  // Window.width
    tjs_int height = ncbPropAcc.getIntValue(TJS_W("height")); // Window.height

    // 用 tTVPRect 构造（KiriKiri 跨平台矩形类型，不是 Win32 RECT）
    tTVPRect rect{ left, top, width + left, height + top };

    // 填充返回字典：{x, y, w, h}（接口格式与原版一致）
    ncbDictionaryAccessor ncbDictAcc;
    ncbDictAcc.SetValue(TJS_W("x"), rect.left);
    ncbDictAcc.SetValue(TJS_W("y"), rect.top);
    auto w = rect.right  - rect.left;
    auto h = rect.bottom - rect.top;
    ncbDictAcc.SetValue(TJS_W("w"), w);
    ncbDictAcc.SetValue(TJS_W("h"), h);
    auto *dis = ncbDictAcc.GetDispatch();
    r->SetObject(dis, dis);
    return TJS_S_OK;
}
```

### 关键差异分析

```
原版坐标来源: HWND → GetWindowRect(hwnd) → Win32 系统的屏幕坐标
适配版坐标来源: iTJSDispatch2* → Window.left/top/width/height → KiriKiri 引擎维护的逻辑坐标
```

**语义差异**：
- Win32 `GetWindowRect` 返回的是包含标题栏和边框的"外边框"矩形
- TJS `Window.left/top` 是引擎逻辑坐标（不一定包含原生系统边框）
- 对移动端全屏游戏而言，两者通常相同（都是 `0, 0, 屏幕宽, 屏幕高`）

**接口兼容性**：返回的字典格式 `{x, y, w, h}` 与原版完全相同，TJS 脚本代码无需修改。

### getClientRect 和 getNormalRect 的极简 stub

注意适配版对这两个函数的处理：

```cpp
// cpp/plugins/windowEx.cpp，第 427-438 行

// getClientRect 直接复用 getWindowRect——移动端全屏时客户区=窗口区
static tjs_error getClientRect(tTJSVariant *r, tjs_int n, tTJSVariant **p,
                               iTJSDispatch2 *obj) {
    getWindowRect(r, n, p, obj);  // 直接委托给 getWindowRect
    return TJS_S_OK;
}

// getNormalRect 同样直接复用——移动端没有"非最大化时的正常位置"概念
static tjs_error getNormalRect(tTJSVariant *r, tjs_int n, tTJSVariant **p,
                               iTJSDispatch2 *obj) {
    getWindowRect(r, n, p, obj);  // 同样委托
    return TJS_S_OK;
}
```

原版的 `getNormalRect` 涉及 `GetWindowPlacement` + `GetMonitorInfoW`（多显示器修正），适配版直接用 `getWindowRect` 代替，语义上不完全等价，但对移动端够用。

## 8. win32_dt.h 与 windowEx.cpp 的协同机制

理解两个文件的依赖关系：

```
┌──────────────────────────────────────────────────────────────┐
│  cpp/plugins/windowEx.cpp                                    │
│                                                              │
│  #include "win32_dt.h"          ← 引入类型 shim             │
│  #include "ncbind.hpp"          ← TJS 绑定框架              │
│                                                              │
│  #if !defined(_WIN64) && !defined(_WIN32)                   │
│    typedef unsigned long ULONG_PTR;  ← 补充 win32_dt.h 未定义的 │
│  #else                                                       │
│    #include <windows.h>         ← 在真正的 Windows 上用真 API │
│    #include <WinDef.h>                                       │
│    #include <WinUser.h>                                      │
│  #endif                                                      │
│                                                              │
│  #ifndef _WIN32                                              │
│    #define WM_NULL 0x0000       ← 补充 windows.h 中的常量    │
│    #define WM_SIZE 0x0005       ←（共约 165 个）              │
│    // ...                                                    │
│  #endif                                                      │
└──────────────────────────────────────────────────────────────┘
```

在真正的 Windows 平台上（`#ifdef _WIN32` 为真）：
- `win32_dt.h` 的 typedef 块被 `#if !defined(_WIN32)` 跳过，不影响真实 HWND 类型
- `windowEx.cpp` 的 `#else` 分支引入真实 `windows.h`，一切按 Win32 正常工作
- 甚至原版的 Win32 功能可能在 Windows 平台上部分可用！（因为代码没有被删除，只是 stub 化了）

> **验证：在 Windows 上 windowEx.cpp 的行为**
>
> 因为 `#ifndef _WIN32` 块在 Windows 上完全不编译，`HWND` 就是真正的 `HWND`，`GetHWND` 能真正读到窗口句柄。但 `regist()` 是空函数，消息接收器不注册，所以 `onMinimize` 等事件仍然不工作。适配版在 Windows 上是"部分退化"的——`getWindowRect` 会返回 TJS 属性而不是真正的 Win32 矩形。

## 本节小结

- kirikiroid2 的 Win32 插件适配分三层：**常量补定义**（`#ifndef _WIN32` 块，165 个 WM_* 常量）、**类型 shim**（`win32_dt.h`，提供 HWND/HMENU/HDC 等类型）、**方法 stub 化**
- Stub 化有三种程度：替代实现（`getWindowRect` 读 TJS 属性）、固定值（`isMaximized` 返回 true）、空函数体（`minimize` 什么都不做）
- **消息接收器是最重大的删除**：`regist()` 变成空函数，`onMessage()` 完全删除，所有依赖 Win32 消息的扩展事件不工作
- **注释掉的方法**（maximizeBox/minimizeBox/disableMove 等）C++ 实现存在但不注册 TJS，让脚本调用时得到 undefined 而不是假装成功的 stub
- **OverlayBitmap** 采用骨架保留策略：结构存在，`setBitmap` 直接返回 true，不创建任何窗口
- 适配版在 Windows 平台上也会编译并运行（`#else` 分支引入真实 `windows.h`），但功能因消息接收器空 stub 而降级

## 练习题与答案

### 题目 1：类型 Shim 的局限性

`win32_dt.h` 把 `HWND` 定义为 `void*`，而适配版的 `GetHWND` 函数确实返回了一个非空指针（读取 TJS 的 HWND 属性）。那么在 Android 上，当代码做 `if (hwnd != nullptr)` 检查时，这个条件是 `true` 还是 `false`？这会不会导致意外地"执行"到本应 stub 化的代码分支？

<details>
<summary>查看答案</summary>

**答案：可能是 true，但不会导致问题，原因是双重保护。**

TJS `Window` 对象的 `HWND` 属性在 Android 平台上也确实存储了一个值（可能是 Cocos2d-x 的某个窗口描述符，或者仍然是 0），所以理论上 `GetHWND` 可能返回非 `nullptr`。

但这不会导致问题，因为**所有需要真正使用 HWND 的代码路径都被 stub 化了**：

```cpp
// 以 resetWindowIcon 为例
static tjs_error resetWindowIcon(tTJSVariant *r, tjs_int n, tTJSVariant **p,
                                 iTJSDispatch2 *obj) {
    WindowEx::GetHWND(obj);  // 调用但不使用返回值
    return TJS_S_OK;         // 直接返回，根本不做 if (hwnd != nullptr) 检查
}
```

真正会做 `if (hwnd != nullptr)` 检查的地方（如 `getWindowRect`），代码路径已经被完全替换为不依赖 `HWND` 的逻辑（读 TJS 属性）。

其他 stub 函数直接 `return TJS_S_OK`，根本不检查 `hwnd`。

**总结**：类型 shim 的目的仅是让代码**编译通过**，不是让代码在非 Windows 平台上"正确工作"。所有实际使用句柄的逻辑都已被 stub 替代。

</details>

### 题目 2：注释的语义

适配版中，`maximizeBox` 的 C++ 方法（`getMaximizeBox`/`setMaximizeBox`）存在于代码中，但注册 TJS 的 `RawCallback` 行被注释掉了。这与直接写一个空 stub 并注册有什么本质区别？哪种方式更好？

<details>
<summary>查看答案</summary>

**本质区别在于脚本层的可见性和错误处理：**

**方案 A：注释掉注册（当前适配版的做法）**
```cpp
// TJS 脚本访问 window.maximizeBox：
// → TJS 抛出"属性不存在"错误（或返回 void/undefined）
// → 游戏脚本的错误处理代码会被触发
// → 开发者能立刻知道"这个功能在当前平台不可用"
```

**方案 B：注册空 stub**
```cpp
// 如果注册了：
static tjs_error getMaximizeBox(...) { if(r) *r = 0; return TJS_S_OK; }
// TJS 脚本访问 window.maximizeBox：
// → 返回 0（false）
// → 游戏脚本认为"最大化按钮不可见"（可能进入错误逻辑分支）
// → 开发者看不出任何问题，但游戏行为不对
```

**哪种方式更好：取决于具体语义**

对于 `maximizeBox`（控制最大化按钮显示/隐藏）：
- 在移动端，最大化按钮根本不存在，返回 `0` 在逻辑上是"正确的"（没有最大化按钮），
  但游戏脚本可能因此认为不需要调整 UI 布局，实际上它应该做完全不同的全屏适配
- 注释掉注册让脚本**显式失败**，迫使开发者处理这个平台差异，是更诚实的做法

对于 `minimize()`（让窗口最小化）：
- 注册一个空 stub 是合理的——游戏调用 `window.minimize()`，啥都不发生，游戏继续运行
- 比"抛出属性不存在错误"更友好，不会让游戏崩溃

**原则**：如果 stub 的固定返回值会让调用者产生错误的行为预期，注释掉注册更好；如果空 stub 的效果是"什么都不发生，游戏继续"，注册 stub 更友好。

</details>

### 题目 3：实战编写适配函数

为 kirikiroid2 编写 `setClientRect` 的跨平台 stub 实现，要求：
1. 保留参数数量验证（`n < 1` 返回 `TJS_E_BADPARAMCOUNT`）
2. 保留参数类型验证（非 object 类型返回 `TJS_E_INVALIDPARAM`）
3. 尝试用 TJS 属性设置实现一个"尽力而为"的版本（设置 `Window.left/top/width/height`）

<details>
<summary>查看答案</summary>

```cpp
// 适配版 setClientRect 实现
// 策略：保留参数验证，用 TJS 属性代替 Win32 SetWindowPos

static tjs_error setClientRect(tTJSVariant *r, tjs_int n, tTJSVariant **p,
                               iTJSDispatch2 *obj) {
    // 参数数量检查（与原版一致）
    if (n < 1) return TJS_E_BADPARAMCOUNT;
    // 参数类型检查（与原版一致）
    if (p[0]->Type() != tvtObject) return TJS_E_INVALIDPARAM;

    if (r) r->Clear();

    // "尽力而为"实现：通过 TJS 属性设置窗口位置/尺寸
    // 注意：这不是 Win32 SetWindowPos 的精确等价，但对移动端全屏游戏够用
    ncbPropAccessor dict(*p[0]);
    ncbPropAccessor wnd(obj);

    // 读取输入字典的 x/y/w/h 值
    // （如果字典里没有某个键，getIntValue 会返回 0）
    tjs_int x = dict.getIntValue(TJS_W("x"), 0);
    tjs_int y = dict.getIntValue(TJS_W("y"), 0);
    tjs_int w = dict.getIntValue(TJS_W("w"), 0);
    tjs_int h = dict.getIntValue(TJS_W("h"), 0);

    // 通过 TJS PropSet 设置 Window 对象的属性
    tTJSVariant vx(x), vy(y), vw(w), vh(h);
    bool ok = true;
    ok &= TJS_SUCCEEDED(obj->PropSet(0, TJS_W("left"),   nullptr, &vx, obj));
    ok &= TJS_SUCCEEDED(obj->PropSet(0, TJS_W("top"),    nullptr, &vy, obj));
    ok &= TJS_SUCCEEDED(obj->PropSet(0, TJS_W("width"),  nullptr, &vw, obj));
    ok &= TJS_SUCCEEDED(obj->PropSet(0, TJS_W("height"), nullptr, &vh, obj));

    if (r) *r = ok;  // 返回是否全部设置成功
    return TJS_S_OK;
}

// 并在 NCB_ATTACH_CLASS_WITH_HOOK 块中添加注册：
// RawCallback(TJS_W("setClientRect"), &Class::setClientRect, 0);
```

**注意点**：
1. 使用 `ncbPropAccessor` 而不是 `ncbDictionaryAccessor`——前者用于读取任意 TJS 对象的属性，后者专门创建新字典。这里的 `*p[0]` 是脚本传来的现有字典对象，应该用前者
2. `getIntValue` 的第二个参数是默认值，当字典中没有该键时返回它
3. TJS `Window.width/height` 是否可写取决于引擎实现，可能触发窗口实际调整大小的逻辑

</details>
