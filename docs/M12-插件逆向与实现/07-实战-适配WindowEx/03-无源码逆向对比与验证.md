## 术语预览

本节将涉及以下术语，先有个印象，正文中会逐一详细讲解：

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| 导出表 | Export Table | DLL 对外公开的函数列表，外部程序通过函数名或序号调用这些函数 |
| 导入表 | Import Table | DLL 运行时依赖的其他 DLL 函数列表，揭示了插件使用了哪些系统 API |
| RTTI | Run-Time Type Information | C++ 运行时类型信息，编译器在二进制中留下的类名字符串，逆向的宝贵线索 |
| ncbind | NCBind | KiriKiri2 的插件绑定框架，用于把 C++ 类/函数注册到 TJS2 脚本引擎 |
| 伪代码 | Pseudocode | IDA 的 Hex-Rays 反编译器将汇编代码翻译成接近 C 语言的高级代码 |
| 交叉引用 | Cross Reference (XREF) | IDA 中标注哪些函数调用了某个函数，或哪些代码引用了某个字符串 |
| TJS2 | TJS2 Scripting Engine | KiriKiri 游戏引擎内置的脚本语言引擎，插件通过特定接口注册到它 |

## 1. 逆向分析的起点：导出表

当你拿到一个完全陌生的 DLL，第一步永远是查看**导出表（Export Table）**。导出表列出了这个 DLL"愿意"让外部程序调用的所有函数名和地址。

对于 KiriKiri2 插件系统，只需要两个导出函数就能工作：

```
V2Link   — 插件加载时调用，引擎把全局 TJS 对象指针传进来
V2Unlink — 插件卸载时调用，做清理工作
```

windowEx.dll 的导出表（来自 IDA 分析结果）：

```
## EXPORTS

  [1] 0x1001A580 V2Link
  [2] 0x1001A5A0 V2Unlink
  [3] 0x1001A580 _V2Link@4
  [4] 0x1001A5A0 _V2Unlink@0
  [268546971] 0x1001B39B DllEntryPoint
```

**关键判断**：
- `V2Link` 和 `V2Unlink` 同时存在 → 这是一个标准 KiriKiri2 插件，不是普通 DLL
- `_V2Link@4` 是 32 位 stdcall 调用约定的装饰名（带参数字节数后缀），与 `V2Link` 指向同一地址
- `DllEntryPoint` 序号 268546971（0x10040DBB）是标准 DLL 入口点序号

### 1.1 V2Link 的反汇编验证

```asm
; V2Link @ 0x1001A580
0x1001A580: mov     eax, [esp+arg_0]      ; 取第一个参数（iTJSDispatch2* 指针）
0x1001A584: mov     dword_10032A34, eax   ; 保存到全局变量（TJS 全局对象指针）
0x1001A589: call    sub_1001A520          ; 调用真正的初始化逻辑
0x1001A58E: mov     ecx, dword_10032A30   ; 读取引用计数
0x1001A594: mov     dword_10032A20, ecx   ; 记录初始引用计数
0x1001A59A: xor     eax, eax              ; 返回 0（成功）
0x1001A59C: retn    4                     ; stdcall：调用者参数由被调用者清除（4字节）
```

**逆向推断**：
- `[esp+arg_0]` = 第一个参数，即 `iTJSDispatch2*`（TJS 全局调度接口）
- `dword_10032A34` = 全局变量，保存引擎传来的指针（后续注册类时使用）
- `sub_1001A520` = 实际执行类注册的初始化函数
- `retn 4` 说明这是 `__stdcall` 调用约定，函数有 1 个 4 字节参数

与原始源码对照（`main.cpp`，ncbind 框架自动生成的 V2Link 代码）完全吻合。

**导入表（Import Table）**是逆向分析的第二个突破口。它告诉你这个 DLL 在运行时需要调用哪些系统函数，从而推断出插件的功能领域。

### 2.2 从导入表推断插件功能

即使没有任何注释，也可以通过分组归纳推断出完整功能：

|---|---|

| `EnumDisplayMonitors/GetMonitorInfoW` | **多显示器信息查询** |

### 2.3 特别注意：动态加载的 API

在导入表的字符串区域（`INTERESTING STRINGS`），发现了两个不在导入表里的 API 名字：

```
0x10025A08  "GetProcessWindowStation"
0x10025A50  "GetActiveWindow"
0x10029B78  "DwmSetWindowAttribute"   ← DWM（桌面窗口管理器）
```

这些字符串被用于**运行时动态加载**（`GetProcAddress` 加载），意味着：
- `DwmSetWindowAttribute`（Vista+ 的毛玻璃特效API）被条件性使用——如果系统不支持，优雅降级
- 这类 API 不能静态链接（某些 Windows 版本没有），所以用 `GetProcAddress` 动态获取

## 3. 函数大小分析：找到核心逻辑

函数大小（字节数）是另一个强力线索。**最大的非 CRT 函数通常就是核心业务逻辑。**

CRT（C 运行时库）函数指 `_memcpy`、`_realloc`、`__sbh_alloc_block` 等标准库函数，在逆向时可以跳过。

### 3.1 windowEx.dll 函数大小排名（前 15，非 CRT）

```
排名  地址          大小      推断功能
 1   0x10005970   33684字节  消息分发主函数 onMessage()
 2   0x10013AB0    3554字节  TJS 属性 setter/getter（字符串处理大量代码）
 3   0x1000E5E0    2602字节  枚举常量注册函数（注册 ncht* / WM_* 常量）
 4   0x10004690    1955字节  MenuItemEx 析构/属性处理
 5   0x100102E0    1708字节  OverlayBitmap 窗口消息处理
 6   0x1000F3D0    1334字节  EnumChildWindows 回调（窗口枚举）
 7   0x10010C20    1268字节  属性注册/类初始化
 8   0x100169E0    1049字节  系统菜单操作
```

**为何 `sub_10005970`（33684字节）几乎必然是 `onMessage()`？**

1. **大小悬殊**：第一名 33684 字节，第二名仅 3554 字节——相差 9 倍。Windows 消息分发函数（switch-case 处理数十种 WM_* 消息）天然会很大
2. **被调用关系**：被 `sub_10005550` 和 `sub_100056E0` 调用，这两个函数分别是 ncbind 框架的消息接收适配器（receiver 的 `regist` 和事件分发包装）
3. **反汇编开头**：
   ```asm
   0x10005970: push ebp
   0x10005971: mov  ebp, esp
   0x10005973: and  esp, 0FFFFFFF8h    ; 栈对齐
   0x10005976: push 0FFFFFFFFh          ; SEH（结构化异常处理）帧
   0x10005978: push offset SEH_10005970
   0x1000597D: mov  eax, large fs:0     ; 获取线程 SEH 链
   0x10005984: sub  esp, 3B4h           ; 分配 0x3B4=948 字节的局部变量空间
   ```
   948 字节的局部变量 + SEH 保护，说明是高复杂度的业务函数
4. **引用的字符串**：
   ```asm
   0x100059EA: push offset aWindow; "Window"
   ```
   函数一开始就访问名为 `"Window"` 的 TJS 对象——正是 onMessage 的宿主对象

### 3.2 `sub_1000E5E0`：枚举常量注册函数

分析反汇编可以看到这个函数做了什么——逐一调用 `sub_10017320`，传入常量名字符串和整数值：

```asm
0x1000E5EA: push offset aNchterror;      "nchtError"
0x1000E5F0: push 0FFFEh                  ; 值 = -2
0x1000E5F6: call sub_10017320            ; 注册枚举值

0x1000E5FB: push offset aNchttransparen; "nchtTransparent"
0x1000E601: push 0FFFFh                  ; 值 = -1
0x1000E607: call sub_10017320

0x1000E609: push offset aNchtnowhere;   "nchtNoWhere"
0x1000E60F: push 0                       ; 值 = 0
0x1000E615: call sub_10017320

0x1000E617: push offset aNchtclient;    "nchtClient"
0x1000E61D: push 1                       ; 值 = HTCLIENT
0x1000E623: call sub_10017320
```

一眼就能看出：这是在把 `HT*`（HitTest 结果枚举，如 `HTCLIENT=1`, `HTCAPTION=2`）注册为 TJS 全局常量，并加了 `ncht` 前缀（non-client hit-test）。

与原始源码 `main.cpp` 中的 ncbConst 块完全对应：

```cpp
// 原始 main.cpp 中的枚举注册
NCB_REGISTER_SUBCLASS(tWindowExBase) {
    NCB_CONST_INT(nchtError,       -2);
    NCB_CONST_INT(nchtTransparent, -1);
    NCB_CONST_INT(nchtNoWhere,      0);
    NCB_CONST_INT(nchtClient,       1);
    NCB_CONST_INT(nchtCaption,      2);
    // ...
}
```

## 4. RTTI 字符串：还原 C++ 类层次

**RTTI（运行时类型信息）**是 C++ 编译器在启用 RTTI 时（默认启用）在二进制中写入的类型描述信息。即使没有符号文件，RTTI 字符串也能告诉你完整的类名。

### 4.1 windowEx.dll 中发现的 RTTI 字符串

```
0x10030F2C  ".?AU?$ncbAttachTJS2ClassAutoRegister@UWindowEx@@@@"
0x10030F68  ".?AU?$ncbInstanceAdaptor@UWindowEx@@@@"
0x10030EB8  ".?AU?$ncbAttachTJS2ClassAutoRegister@UMenuItemEx@@@@"
0x10030EF8  ".?AU?$ncbInstanceAdaptor@UMenuItemEx@@@@"
0x10030B50  ".?AU?$ncbAttachTJS2ClassAutoRegister@UPadEx@@@@"
```

这些是 MSVC 的**修饰名（Mangled Name）**，格式为 `.?AU<template_name>@<class_name>@@`。

解码方式：
```
.?AU?$ncbAttachTJS2ClassAutoRegister@UWindowEx@@@@
      └─ 模板类                      └─ 模板参数类名
= ncbAttachTJS2ClassAutoRegister<WindowEx>
```

`U` 前缀表示这是一个 `struct`（在 MSVC 中，`struct` 和 `class` 的 RTTI 标记不同，`U` = struct，`V` = class）。

### 4.2 从 RTTI 恢复类信息表

| RTTI 字符串片段 | 解码结果 | 含义 |
|---|---|---|
| `ncbAttachTJS2ClassAutoRegister@UWindowEx` | `ncbAttachTJS2ClassAutoRegister<WindowEx>` | WindowEx 类注册器（全局静态对象） |
| `ncbInstanceAdaptor@UWindowEx` | `ncbInstanceAdaptor<WindowEx>` | WindowEx TJS 实例适配器 |
| `ncbAttachTJS2ClassAutoRegister@UMenuItemEx` | `ncbAttachTJS2ClassAutoRegister<MenuItemEx>` | MenuItemEx 类注册器 |
| `ncbInstanceAdaptor@UMenuItemEx` | `ncbInstanceAdaptor<MenuItemEx>` | MenuItemEx TJS 实例适配器 |
| `ncbAttachTJS2ClassAutoRegister@UPadEx` | `ncbAttachTJS2ClassAutoRegister<PadEx>` | PadEx 类注册器 |

**结论**：DLL 中存在三个 TJS 类：`WindowEx`、`MenuItemEx`、`PadEx`。与原始源码完全匹配。

### 4.3 ncbFunctionTag 字符串：全局函数注册

除类注册外，还发现了 `ncbNativeFunctionAutoRegisterTempl` 系列字符串：

```
ncbFunctionTag_System_getCPUType
ncbFunctionTag_System_getAboutString
ncbFunctionTag_System_clearGraphicCache
ncbFunctionTag_System_isBreathing
ncbFunctionTag_System_breathe
ncbFunctionTag_System_setDpiAwareness
ncbFunctionTag_System_getDoubleClickTime

ncbFunctionTag_System_setApplicationIcon
ncbFunctionTag_System_expandEnvString
ncbFunctionTag_System_readEnvValue
ncbFunctionTag_System_getSystemMetrics
ncbFunctionTag_System_setCursorPos
ncbFunctionTag_System_getCursorPos
ncbFunctionTag_System_getMonitorInfo
ncbFunctionTag_System_getDisplayMonitors
ncbFunctionTag_Debug_console_bringAfter
ncbFunctionTag_Debug_console_setPlacement
ncbFunctionTag_Debug_console_getPlacement
```

## 5. 特殊命名函数：`fnEnum`、`fn`、`EnumFunc`

函数列表中有三个名字特殊的函数，不是 `sub_XXXXXXXX` 格式——这说明它们的名字被某种方式保留了下来（可能是导出了，也可能是字符串中存在对应名字）。

```
0x10013280  306字节  fnEnum
0x10011780  288字节  fn
0x1000DD10  281字节  EnumFunc
```

### 5.1 `EnumFunc`（0x1000DD10，281字节）

这个名字出现在字符串常量区（`INTERESTING STRINGS` 中有 `EnumThreadWindows` 和 `EnumChildWindows`）并被 ncbind 注册。结合大小和位置，推断它是 `EnumChildWindows` 的回调函数（`WNDENUMPROC`），用于枚举子窗口：

```c
// 推断的原型
BOOL CALLBACK EnumFunc(HWND hwnd, LPARAM lParam);
```

与原始源码一致——`main.cpp` 中确实定义了一个 `EnumFunc` 作为 `EnumChildWindows` 的回调。

### 5.2 `fnEnum`（0x10013280，306字节）

`fnEnum` 在 ncbind 框架中通常是"带枚举参数的 TJS 回调函数"的命名模式——它被注册为一个可以接收枚举值并返回对应字符串的辅助函数。

### 5.3 `fn`（0x10011780，288字节）

`fn` 是 ncbind 中最简单的函数回调包装名，通常对应一个只有一两行代码的简单 TJS 方法封装。

## 6. 交叉引用分析：追踪调用链

IDA 的交叉引用（XREF）功能可以显示每个函数的调用者。这让我们可以从底层 Win32 调用反向追踪到顶层 TJS 暴露接口。

### 6.1 以 `IsZoomed` 为例追踪调用链

```
USER32::IsZoomed
    ↑ 被调用于 sub_1000F010 (373字节)
        ↑ 被调用于 sub_100102E0 (1708字节, OverlayBitmap 消息处理)
        ↑ 被调用于 sub_10010990 (544字节)
            ↑ 被调用于 sub_10010C20 (1268字节, 属性 getter 集合)
```

**推断**：
- `sub_1000F010`（373字节）= `isMaximized` 属性的 getter 实现
- `sub_10010C20` = 多个属性 getter 的批量注册或调度表

对照原始源码 `isMaximized` 实现：

```cpp
// main.cpp 中
bool isMaximized() {
    return IsZoomed(hwnd) != 0;
}
```

与逆向推断完全吻合。

### 6.2 以 `RegisterHotKey` 为例追踪调用链

```
USER32::RegisterHotKey
    ↑ 被调用于 sub_10003550 (538字节)
        ↑ 被调用于 sub_10005970 的某个分支 (onMessage 中的 WM_HOTKEY 处理)
```

这说明热键功能由 `sub_10003550` 封装，在 `onMessage` 的 `WM_HOTKEY` 分支被调用——与原始 `main.cpp` 的 `onHotKey()` 方法对应。

## 7. 有源码 vs 无源码逆向对比

通过上面的分析，我们可以系统地对比两种工作模式的效率差异：

| 分析维度 | 有源码 | 无源码 |
|---|---|---|
| **类名/函数名** | 直接可读 | RTTI 字符串还原（约 80% 准确） |
| **功能推断** | 直接阅读注释和代码 | 导入表 + 字符串 + 大小分析（约 70% 准确） |
| **参数类型** | 直接查看函数签名 | 需看汇编调用约定（耗时 3-5x） |
| **业务逻辑细节** | 完整可读 | 需要 Hex-Rays 反编译 + 大量猜测 |
| **TJS 接口清单** | 看 NCB_ATTACH_CLASS 宏 | 需从 ncbFunctionTag 字符串提取 |
| **适配工作量** | 低（直接搬代码 + 改 API） | 高（先逆向 + 再实现 + 验证） |
| **验证难度** | 低（源码即文档） | 高（需要测试用例验证行为） |

**关键结论**：
1. 对于**有源码插件**（如 windowEx），逆向 DLL 是**验证步骤**，确认适配版没有遗漏功能
2. 对于**无源码插件**（如 PackinOne/TextRender），逆向 DLL 是**唯一的信息来源**，需要花费数倍时间

## 8. 编写 IDAPython 脚本提取插件信息

以下展示如何编写专用分析脚本，自动从 IDA 数据库中提取 KiriKiri2 插件的关键信息。

### 8.1 脚本一：提取所有 ncbFunctionTag 字符串

```python
# extract_ncb_tags.py
# 用途：从 IDA 数据库提取所有 ncbFunctionTag 注册名，生成 TJS 接口清单
# 运行：idat64.exe -A -S"extract_ncb_tags.py" windowEx.dll.i64

import idc
import idautils
import idaapi

def find_ncb_tags():
    """
    扫描所有字符串，找到 ncbFunctionTag_ 开头的条目
    这些字符串对应插件注册到 TJS 引擎的函数名
    """
    tags = []
    # 遍历数据段中的所有字符串引用
    for seg in idautils.Segments():
        seg_name = idc.get_segm_name(seg)
        # 只扫描数据段（.rdata, .data）
        if seg_name not in ('.rdata', '.data', '.rodata'):
            continue
        ea = idc.get_segm_start(seg)
        end = idc.get_segm_end(seg)
        while ea < end:
            # 尝试读取以 ncbFunctionTag_ 开头的字符串
            s = idc.get_strlit_contents(ea, -1, idc.STRTYPE_C)
            if s and s.startswith(b'ncbFunctionTag_'):
                # 解析格式: ncbFunctionTag_<对象>_<方法>
                tag_str = s.decode('utf-8', errors='replace')
                parts = tag_str[len('ncbFunctionTag_'):].split('_', 1)
                obj = parts[0] if len(parts) > 0 else '?'
                method = parts[1] if len(parts) > 1 else '?'
                tags.append({
                    'addr': ea,
                    'tag': tag_str,
                    'object': obj,
                    'method': method
                })
            ea = idc.next_head(ea)
    return tags

def main():
    tags = find_ncb_tags()
    print(f"Found {len(tags)} ncbFunctionTag entries:\n")

    # 按对象名分组输出
    from collections import defaultdict
    by_obj = defaultdict(list)
    for t in tags:
        by_obj[t['object']].append(t['method'])

    for obj, methods in sorted(by_obj.items()):
        print(f"Object: {obj}")
        for m in sorted(methods):
            print(f"  .{m}()")
        print()

    idc.qexit(0)

main()
```

### 8.2 脚本二：提取 RTTI 类名

```python
# extract_rtti_classes.py
# 用途：提取 DLL 中所有 RTTI 类型描述符，还原 C++ 类层次
# 对于 ncbind 插件，重点找 ncbAttachTJS2ClassAutoRegister<X> 模式

import idc
import idautils
import re

def demangle_rtti(mangled):
    """
    简化的 MSVC RTTI 字符串解码
    格式: .?AU?$<template>@<arg>@@@@
    """
    # 匹配模板类模式
    # .?AU?$ncbAttachTJS2ClassAutoRegister@UWindowEx@@@@
    pat = r'\.\?AU\?\$(\w+)@U(\w+)@@@@'
    m = re.match(pat, mangled)
    if m:
        return f"{m.group(1)}<{m.group(2)}>"

    # 匹配普通类模式
    # .?AVtTJSDispatch@@
    pat2 = r'\.\?A[UV](\w+)@@'
    m2 = re.match(pat2, mangled)
    if m2:
        return m2.group(1)

    return mangled  # 无法解码，返回原始

def find_rtti_strings():
    """扫描所有段，查找 RTTI 类型描述符（以 .?A 开头）"""
    results = []
    for seg in idautils.Segments():
        seg_name = idc.get_segm_name(seg)
        if seg_name not in ('.rdata', '.data'):
            continue
        ea = idc.get_segm_start(seg)
        end = idc.get_segm_end(seg)
        while ea < end:
            s = idc.get_strlit_contents(ea, -1, idc.STRTYPE_C)
            if s and s.startswith(b'.?A'):
                raw = s.decode('utf-8', errors='replace')
                decoded = demangle_rtti(raw)
                results.append({
                    'addr': ea,
                    'raw': raw,
                    'decoded': decoded
                })
            ea = idc.next_head(ea)
    return results

def main():
    results = find_rtti_strings()
    print(f"Found {len(results)} RTTI entries:\n")

    # 优先显示 ncb 相关条目
    ncb_entries = [r for r in results if 'ncb' in r['decoded']]
    other_entries = [r for r in results if 'ncb' not in r['decoded']]

    print("=== ncbind 框架类 ===")
    for r in ncb_entries:
        print(f"  0x{r['addr']:08X}  {r['decoded']}")

    print("\n=== TJS 基础类 ===")
    for r in other_entries:
        if any(x in r['decoded'] for x in ['TJS', 'iTJS', 'tTJS']):
            print(f"  0x{r['addr']:08X}  {r['decoded']}")

    idc.qexit(0)

main()
```

### 8.3 脚本三：函数大小排名（定位核心逻辑）

```python
# rank_functions.py
# 用途：按函数大小降序排列，排除 CRT 函数，定位业务逻辑
# 对于首次分析陌生插件，这是最快速找到核心函数的方法

import idc
import idautils
import idaapi

# 已知的 CRT/标准库函数名前缀，用于过滤
CRT_PREFIXES = (
    '_', '?', '__', 'SEH_',
    'memcpy', 'memmove', 'memset', 'malloc', 'free', 'realloc',
    'printf', 'sprintf', 'vsprintf',
)

def is_crt_function(name):
    """判断是否是 CRT/标准库函数"""
    if not name:
        return False
    for prefix in CRT_PREFIXES:
        if name.startswith(prefix):
            return True
    return False

def get_function_sizes(min_size=100, max_count=50):
    """获取所有函数的大小，过滤 CRT，按大小降序返回"""
    funcs = []
    for ea in idautils.Functions():
        name = idc.get_func_name(ea)
        if is_crt_function(name):
            continue
        size = idc.get_func_attr(ea, idc.FUNCATTR_END) - ea
        if size >= min_size:
            funcs.append((ea, size, name))

    funcs.sort(key=lambda x: x[1], reverse=True)
    return funcs[:max_count]

def main():
    funcs = get_function_sizes(min_size=200, max_count=30)

    print(f"Top {len(funcs)} largest non-CRT functions:\n")
    print(f"{'排名':>4}  {'地址':>12}  {'大小':>8}  函数名")
    print("-" * 60)
    for i, (ea, size, name) in enumerate(funcs, 1):
        print(f"{i:>4}  0x{ea:08X}  {size:>8} bytes  {name}")

    print("\n分析建议：")
    if funcs:
        top = funcs[0]
        print(f"  最大函数 {top[2]} ({top[1]} bytes) 极可能是消息处理主函数")
        print(f"  建议用 Hex-Rays 反编译地址 0x{top[0]:08X}")

    idc.qexit(0)

main()
```

### 题目 1：RTTI 字符串解码

下面是一个从某 KiriKiri2 插件 DLL 中发现的 RTTI 字符串：

```
.?AU?$ncbAttachTJS2ClassAutoRegister@ULayerExDraw@@@@
```

请解码出：
1. 这是什么模板类的实例？
2. 模板参数（插件实现的 C++ 类名）是什么？
3. 对应 TJS 脚本中的哪个类名？

<details>
<summary>查看答案</summary>

1. **模板类**：`ncbAttachTJS2ClassAutoRegister`，这是 ncbind 框架的自动类注册器模板
2. **C++ 类名**：`LayerExDraw`（`U` 前缀表示 struct，`@@@@` 是结尾）
3. **TJS 类名**：通常与 C++ 类名相同，即在 TJS 脚本中可以使用 `new LayerExDraw()` 创建实例

解码过程：
```
.?AU?$ncbAttachTJS2ClassAutoRegister@ULayerExDraw@@@@
       └──── 模板名 ─────────────────┘└── 参数 ─┘
= ncbAttachTJS2ClassAutoRegister<LayerExDraw>
```

验证：可以在 `cpp/plugins/layerex_draw/` 目录下找到 `LayerExDraw` 类的实现。

</details>

### 题目 2：从导入表推断功能

假设你拿到一个未知插件 DLL，其导入表包含以下函数：

```
WINMM.DLL:
  waveOutOpen, waveOutWrite, waveOutClose
  waveOutGetDevCaps, waveOutSetVolume
  midiOutOpen, midiOutShortMsg, midiOutClose

KERNEL32.DLL:
  CreateThread, CreateEvent, SetEvent, WaitForSingleObject
```

请推断这个插件的功能，并说明推断依据。

<details>
<summary>查看答案</summary>

**推断功能**：这是一个**音频/MIDI 播放插件**，具体功能如下：

1. **PCM 音频播放**：`waveOutOpen/Write/Close` = 标准 Windows 波形音频输出 API，用于播放 PCM 格式的音频数据
2. **MIDI 音乐播放**：`midiOutOpen/ShortMsg/Close` = MIDI 合成器接口，用于播放 MIDI 音乐（游戏 BGM 常用）
3. **音量控制**：`waveOutSetVolume` = 设置波形输出音量
4. **设备枚举**：`waveOutGetDevCaps` = 查询音频设备能力（如采样率、位深支持）
5. **异步播放**：`CreateThread/Event/WaitForSingleObject` = 创建后台播放线程，主线程异步控制

在 KiriKiri2 的插件体系中，这对应 `WaveSoundBuffer` 或类似的音频插件接口，允许 TJS 脚本控制音频播放、暂停、停止、音量调节等。

注意：现代 kirikiroid2 适配版会把这些 WinMM API 替换为 OpenAL 或 FFmpeg 的音频接口。

</details>

### 题目 3：最大函数推断

在分析一个 KiriKiri2 插件时，发现以下函数大小排名（非CRT）：

```
sub_10008000   28432 bytes  (最大)
sub_1000C500    2156 bytes
sub_1000D800    1890 bytes
sub_1000E100     956 bytes
sub_1001A200     312 bytes  fnEnum
sub_1001A500     180 bytes  fn
```

并且 `sub_10008000` 被以下两个函数调用：
- `sub_10007800`（调用点：偏移 0x30 处，`mov ecx, [esi]; call [ecx+18h]`）
- `sub_10007A00`（调用点：偏移 0x58 处，`push [ebp+arg_0]; call sub_10008000`）

同时，`sub_10008000` 的开头有如下反汇编：

```asm
sub esp, 250h        ; 分配 592 字节局部变量
push offset aVisible ; "visible"
```

问：`sub_10008000` 最可能是什么函数？并解释你的推断依据。

<details>
<summary>查看答案</summary>

**推断**：`sub_10008000` 最可能是**消息/事件分发主函数**（相当于 `onMessage()` 或 Windows 窗口过程 `WndProc`）。

**推断依据**：

1. **大小悬殊**：28432 字节远超第二名（2156 字节），约 13 倍之差。只有包含大量 switch-case 分支（处理几十种消息）的函数才会如此大
2. **局部变量空间大**：`sub esp, 250h`（592字节）说明函数内部有大量临时变量，消息处理函数需要保存各种消息参数
3. **引用 "visible" 字符串**：这是 TJS `Window` 对象的 `visible` 属性名，消息处理函数需要读取窗口可见性状态（如处理 WM_SHOWWINDOW 消息时）
4. **被两个函数调用**：
   - `sub_10007800` 通过虚函数表调用（`call [ecx+18h]`）— 这是 ncbind receiver 通过接口指针调度
   - `sub_10007A00` 直接调用并传 `[ebp+arg_0]`（WPARAM 或消息参数）— 这是消息适配层
5. **`fnEnum` 和 `fn` 的存在**：这两个命名函数是 ncbind 的枚举回调和函数回调，与消息处理框架配套出现

综上，这个函数处理各种 TJS/Win32 消息，在 kirikiroid2 适配时会被 stub 化或替换为 Cocos2d-x 事件处理。

</details>
