## 一、KrKr2 UI 层整体架构

### 1.1 为什么有两套渲染

KrKr2 的渲染系统由两层构成：

- **游戏内容层（Visual Layer）**：由 `cpp/core/visual/` 管理，负责绘制游戏场景——背景图、立绘、文字框、CG 动画。这一层直接操作 OpenGL，绕过 Cocos2d-x 的场景图（Scene Graph）。
- **系统 UI 层（System UI Layer）**：由 `cpp/core/environ/ui/` 管理，负责绘制模拟器自身的 UI——游戏选择对话框、设置页、音量控制、存档管理等。这一层使用 Cocos2d-x 的 `ui::` 命名空间组件（按钮、滑块、文本框等），通过场景图渲染。

两层叠加的方式是：游戏内容层始终在下（Z-order 低），系统 UI 层在需要时弹出覆盖在上（Z-order 高）。这种设计的优点是分离关注点，缺点是 UI 实现与 Cocos2d-x 紧耦合，难以替换渲染引擎。

```
KrKr2 渲染层次（从下到上）：
┌──────────────────────────────────────┐
│  系统 UI 层  （Cocos2d-x ui::组件）   │  Z-order: 1000+
│  cpp/core/environ/ui/*.cpp           │
├──────────────────────────────────────┤
│  游戏内容层 （KrKr2 visual 直接 OGL）  │  Z-order: 0-999
│  cpp/core/visual/ + OpenGL           │
├──────────────────────────────────────┤
│  Cocos2d-x Director + OpenGL Context │
└──────────────────────────────────────┘
```

### 1.2 `environ/ui/` 目录一览

查看实际文件列表：

```bash
# 列出所有 UI 表单文件
ls cpp/core/environ/ui/
```

根据目录结构，KrKr2 的 UI 表单包括：

| 文件 | 对应表单 | 功能描述 |
|------|---------|---------|
| `ConfigDialog.*` | 游戏配置对话框 | 修改游戏运行时设置（分辨率、帧率、音量等） |
| `GameStartDialog.*` | 游戏启动对话框 | 首次启动时选择游戏目录 / 最近游戏列表 |
| `KeyMappingDialog.*` | 键位映射对话框 | 自定义手柄/键盘映射 |
| `MessageLayer.*` | 消息提示层 | 显示临时通知（如"正在加载..."） |
| `VideoSettingDialog.*` | 视频设置对话框 | 分辨率、全屏、抗锯齿、VSync 设置 |
| `AudioSettingDialog.*` | 音频设置对话框 | 主音量、BGM/SE/Voice 独立音量 |
| `SaveDataDialog.*` | 存档管理对话框 | 读取/覆盖/删除存档槽 |

每个表单通常由三部分组成：
1. **布局文件**（`.csb`）：存放在 `ui/cocos-studio/` 下，定义控件位置、大小、样式
2. **C++ 控制器**（`.cpp/.h`）：继承 `cocos2d::ui::Widget` 或 `cocos2d::Layer`，绑定事件处理逻辑
3. **Locale 文本**（`.xml`）：存放在 `ui/cocos-studio/` 下，实现多语言支持

### 1.3 一个完整表单的生命周期

以 `ConfigDialog` 为例，追踪从"弹出对话框"到"用户点击确定"的完整生命周期：

```
用户操作/代码触发
      ↓
ConfigDialog::show()
      ↓
CSLoader::createNode("ui/cocos-studio/ConfigDialog.csb")  ← 加载布局
      ↓
配置绑定 (bindWidgets())  ← 通过 getChildByName() 获取各控件引用
      ↓
事件注册 (registerCallbacks())  ← 按钮的 addClickEventListener()
      ↓
添加到场景 (Director::getRunningScene()->addChild(this, 1000))
      ↓
[ 用户交互 — 滑动、点击、输入 ]
      ↓
回调触发 → 修改 ConfigManager 中的配置值
      ↓
用户点击"确定"
      ↓
ConfigDialog::onConfirm()  → 保存配置 → removeFromParentAndCleanup()
```

## 二、深入 CSB 文件：Cocos Studio 二进制布局

### 2.1 CSB 文件的本质

CSB（Cocos Studio Binary）是 Cocos Studio 编辑器将 UI 布局序列化的格式。从技术角度看，它是一个 **Protocol Buffers（protobuf）编码的二进制文件**，存储了完整的节点树：每个节点的类型、位置、大小、颜色、子节点列表等。

```
CSB 文件内部结构（简化）：
┌─────────────────────────────┐
│  Magic Header: "csd "       │
│  Version: 3.x               │
├─────────────────────────────┤
│  Node Tree (protobuf)       │
│  ├── RootNode (Layout)      │
│  │   ├── Background (Sprite)│
│  │   ├── TitleLabel (Text)  │
│  │   ├── OkButton (Button)  │
│  │   │   ├── Normal Image   │
│  │   │   └── Pressed Image  │
│  │   └── VolumeSlider       │
│  └── ...                    │
└─────────────────────────────┘
```

### 2.2 CSLoader 如何加载 CSB

`CSLoader::createNode()` 是加载 CSB 文件的入口：

```cpp
// 示例：在 C++ 中加载一个 CSB 布局
#include "cocos2d.h"
#include "editor-support/cocostudio/CocoStudio.h"

// 方式一：同步加载（直接创建节点树）
cocos2d::Node* dialogNode =
    cocostudio::CSLoader::createNode("ui/cocos-studio/ConfigDialog.csb");

// 方式二：异步加载（不阻塞渲染线程）
cocostudio::CSLoader::createNodeAsync(
    "ui/cocos-studio/ConfigDialog.csb",
    [this](cocos2d::Node* node) {
        // 加载完成回调
        this->addChild(node);
        this->bindWidgets(node);  // 绑定控件引用
    },
    this
);
```

加载完成后，可以通过节点名称获取子控件：

```cpp
// 通过在 Cocos Studio 中设置的节点名称查找控件
// 注意：getChildByName 只查找直接子节点，深层节点需用 findChild
auto titleLabel = dialogNode->findChild<cocos2d::Label*>("TitleLabel");
auto okButton   = dialogNode->findChild<cocos2d::ui::Button*>("OkButton");
auto volumeSlider = dialogNode->findChild<cocos2d::ui::Slider*>("VolumeSlider");

if (!okButton) {
    // 如果找不到控件，通常是 CSB 文件中的节点名称拼写错误
    CCLOG("错误：找不到 OkButton，检查 CSB 文件中的节点名称");
    return;
}

// 注册点击回调
okButton->addClickEventListener([this](cocos2d::Ref* sender) {
    this->onConfirmClicked();
});
```

### 2.3 CSB vs CSD vs JSON

Cocos Studio 支持三种输出格式，了解它们的区别有助于理解为什么 KrKr2 选择 CSB：

| 格式 | 后缀 | 特点 | 适用场景 |
|------|------|------|---------|
| CSB | `.csb` | protobuf 二进制，加载速度快，不可读 | 生产环境，KrKr2 使用此格式 |
| CSD | `.csd` | XML 格式，人可读，加载慢 | 编辑器内部格式，版本控制友好 |
| JSON | `.json` | JSON 格式，与 CSD 类似 | 与 Web 端互操作 |

KrKr2 在 `ui/cocos-studio/` 中同时保留 `.csd`（便于编辑和版本控制）和 `.csb`（运行时加载），这是 Cocos2d-x 项目的标准做法。

## 三、Locale XML：国际化机制解析

### 3.1 文件结构

`ui/cocos-studio/` 目录下的 locale XML 文件存储多语言字符串，格式如下：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!-- ui/cocos-studio/locale/zh-CN.xml -->
<resources>
    <!-- 配置对话框 -->
    <string name="config_dialog_title">游戏设置</string>
    <string name="config_dialog_ok">确定</string>
    <string name="config_dialog_cancel">取消</string>
    <string name="config_video_resolution">分辨率</string>
    <string name="config_audio_master_volume">主音量</string>
    <string name="config_audio_bgm_volume">背景音乐音量</string>

    <!-- 游戏启动对话框 -->
    <string name="game_start_title">选择游戏</string>
    <string name="game_start_browse">浏览...</string>
    <string name="game_start_recent">最近游戏</string>
</resources>
```

### 3.2 C++ 中读取 Locale 字符串

KrKr2 通过 `ConfigManager` 的 `Locale` 子系统读取本地化字符串：

```cpp
// 在 C++ 中读取本地化文本
#include "environ/ConfigManager/LocaleManager.h"

// 获取本地化字符串（如果找不到键名，返回键名本身作为降级）
std::string title = LocaleManager::getInstance()->getString("config_dialog_title");
// 在简体中文环境下返回："游戏设置"

// 动态切换语言（重新加载对应的 XML 文件）
LocaleManager::getInstance()->setLocale("en-US");
// 此后 getString("config_dialog_title") 返回 "Game Settings"

// 在 Label 上设置本地化文字
auto titleLabel = dialogNode->findChild<cocos2d::Label*>("TitleLabel");
titleLabel->setString(LocaleManager::getInstance()->getString("config_dialog_title"));
```

### 3.3 国际化机制的局限性

当前实现存在几个设计问题，这也是替换 UI 框架的动机之一：

1. **不支持运行时热切换语言**：切换语言后需要重新创建所有 UI 节点才能生效
2. **字体不支持多脚本**：Cocos2d-x 的默认 BMP 字体不包含日文/韩文 CJK 字符，需要单独配置 TTF 字体
3. **布局不自适应文本长度**：CSB 中的控件尺寸是固定的，长文本会被截断
4. **无 RTL（从右到左书写）支持**：不支持阿拉伯语、希伯来语等从右向左的语言

Flutter/Compose 对这些问题都有成熟的解决方案，这也是迁移的价值所在。

## 五、对照项目源码

打开 Cocos2d-x 的 CSLoader 源码，了解 CSB 反序列化的内部实现：

```
Cocos2d-x（KrKr2 引用版本）中 CSLoader 的位置：
third_party/cocos2d-x/cocos/editor-support/cocostudio/CSLoader.cpp
```

关键函数：
- `CSLoader::createNode(const std::string& filename)` — 入口，判断 CSB vs CSD 格式
- `CSLoader::nodeWithFlatBuffersForSimulator()` — 反序列化 CSB（基于 FlatBuffers，不是 protobuf）
- `CSLoader::createNodeWithFlatBuffersFile()` — 加载 FlatBuffers 布局

> **注意**：Cocos Studio 3.x 之后将序列化格式从 protobuf 改为 **FlatBuffers**（一种零拷贝序列化库）。KrKr2 使用的 Cocos2d-x 版本对应的是 FlatBuffers 格式，因此 `.csb` 实际上是 FlatBuffers 编码，而非 protobuf。

相关文件：
- `cpp/core/environ/ui/*.cpp` — 所有 UI 表单控制器
- `ui/cocos-studio/` — CSB 布局文件 + locale XML
- `cpp/core/environ/cocos2d/AppDelegate.cpp` — 应用启动入口，初始化 UI 层

## 本节小结

- KrKr2 有两套独立渲染层：游戏内容层（直接 OpenGL）和系统 UI 层（Cocos2d-x ui::）
- 系统 UI 层的每个表单由 CSB 布局文件 + C++ 控制器 + locale XML 三部分组成
- CSB 文件是 FlatBuffers 编码的二进制布局，通过 `CSLoader::createNode()` 加载
- 通过 `findChild<T*>("NodeName")` 获取控件引用，通过 `addClickEventListener` 注册回调
- 当前 UI 层存在硬编码、不支持热切换语言、布局不自适应等缺陷，是迁移的主要动机

## 练习题与答案

### 题目 1：CSB 文件加载机制

`CSLoader::createNode()` 返回 `cocos2d::Node*`，但按钮控件的类型是 `cocos2d::ui::Button*`。请解释为什么可以用 `findChild<cocos2d::ui::Button*>("OkButton")` 获取按钮，而不需要手动 `dynamic_cast`。

<details>
<summary>查看答案</summary>

`findChild<T*>()` 是 Cocos2d-x 提供的模板辅助函数，内部使用 `dynamic_cast` 将 `Node*` 转换为 `T*`。调用 `findChild<cocos2d::ui::Button*>("OkButton")` 等价于：

```cpp
cocos2d::Node* raw = root->getChildByName("OkButton");
cocos2d::ui::Button* btn = dynamic_cast<cocos2d::ui::Button*>(raw);
// 如果 raw 实际上不是 Button 类型，dynamic_cast 返回 nullptr
```

`CSLoader` 在反序列化 CSB 时，根据节点的类型字段创建对应的 C++ 对象（例如类型字段为 "Button" 则创建 `cocos2d::ui::Button` 实例）。所以只要 CSB 中确实定义了一个 Button 节点，并且名称正确，`findChild<cocos2d::ui::Button*>` 就能成功返回非空指针。

如果名称拼写错误或节点类型不匹配，返回 `nullptr`，所以代码中必须检查返回值。

</details>

### 题目 2：Locale XML 的降级机制

现有代码中 `LocaleManager::getInstance()->getString("nonexistent_key")` 会返回什么？这种设计有什么优缺点？

<details>
<summary>查看答案</summary>

当 key 不存在时，返回 key 本身（即 `"nonexistent_key"`）。

**优点：**
- 不会因为缺少翻译而崩溃
- 开发者可以很容易地看出哪些字符串没有被翻译（界面上显示的是 key 名而非乱码）
- 降级行为可预测，便于调试

**缺点：**
- 生产环境中用户会看到 key 名而非可读文字，用户体验差
- 没有任何警告机制，开发者可能长时间不知道某个 key 缺失
- 更好的做法是同时记录日志警告，或者有一个 fallback locale（例如中文缺失时退回英文）

改进建议：
```cpp
std::string LocaleManager::getString(const std::string& key) {
    auto it = m_strings.find(key);
    if (it != m_strings.end()) {
        return it->second;
    }
    // 尝试 fallback locale（英文）
    auto fbIt = m_fallbackStrings.find(key);
    if (fbIt != m_fallbackStrings.end()) {
        CCLOG("警告：key '%s' 在当前 locale 中未找到，使用 fallback", key.c_str());
        return fbIt->second;
    }
    CCLOG("错误：locale key '%s' 完全缺失", key.c_str());
    return key;  // 最后降级返回 key 本身
}
```

</details>

### 题目 3：CSB 与 CSD 格式的选择

在一个需要频繁修改 UI 布局的项目中，你会把 CSB 还是 CSD 文件提交到 git？理由是什么？

<details>
<summary>查看答案</summary>

**提交 CSD（XML 格式），不提交 CSB（二进制格式）**。理由：

1. **版本控制可读性**：CSD 是 XML 文本格式，`git diff` 可以直观看出修改了哪些控件属性；CSB 是二进制，`git diff` 只能看到乱码，无法 code review
2. **合并冲突可解决**：两人同时修改 CSD 的不同控件时，可以手动解决 XML 合并冲突；CSB 二进制文件的冲突无法手动解决
3. **构建时生成 CSB**：在构建流程中加入一步"CSD → CSB"的转换（Cocos Studio 提供命令行工具），确保运行时使用高效的 CSB

```cmake
# CMakeLists.txt 中添加 CSD → CSB 的构建步骤（示意）
find_program(COCOS_STUDIO_PACKER CocosStuioPacker)
if(COCOS_STUDIO_PACKER)
    add_custom_command(
        OUTPUT ${CMAKE_BINARY_DIR}/ui/cocos-studio/ConfigDialog.csb
        COMMAND ${COCOS_STUDIO_PACKER}
                -p ${CMAKE_SOURCE_DIR}/ui/cocos-studio/ConfigDialog.csd
                -o ${CMAKE_BINARY_DIR}/ui/cocos-studio/
        DEPENDS ${CMAKE_SOURCE_DIR}/ui/cocos-studio/ConfigDialog.csd
        COMMENT "打包 ConfigDialog.csd → ConfigDialog.csb"
    )
endif()
```

总结：**源文件提交 CSD，构建产物（CSB）加入 .gitignore**。

</details>
