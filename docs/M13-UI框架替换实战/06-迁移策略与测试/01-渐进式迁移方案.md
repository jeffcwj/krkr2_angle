# 01 渐进式迁移方案

> **所属模块**：M13 UI 框架替换实战 › 第06章 迁移策略与测试  
> **前置知识**：M13/02 UI 抽象接口设计、M13/03-04 Flutter/Compose 嵌入实战、M13/05 渲染与桥接  
> **预计阅读时间**：60 分钟

---

## 本节目标

1. 理解"大爆炸式替换"的风险，掌握渐进式迁移的必要性
2. 掌握四阶段迁移路线图的具体实施步骤
3. 实现编译时与运行时两级 Feature Flag 开关
4. 学会逐步替换各对话框（从低风险到高风险）的策略
5. 设计可以在 5 分钟内完成回滚的应急方案

---

## 术语预览

| 术语 | 含义 |
|------|------|
| Feature Flag | 特性开关，通过配置控制新功能是否生效，无需重新部署代码 |
| 大爆炸式替换 | 在同一个版本中一次性替换所有旧代码，风险极高 |
| 渐进式迁移 | 分阶段、可回滚地逐步替换，每一步可独立验证 |
| 金丝雀发布 | 先对少量用户开启新功能，验证稳定后再全量推送 |
| 双写模式 | 新旧两套代码同时运行，通过 Flag 切换激活哪一套 |
| 桩代码（Stub） | 实现了接口但不做实际工作的占位实现，用于测试和过渡 |
| 技术债 | 为了快速交付而采用的临时方案，后续需要投入成本偿还 |
| 回滚策略 | 出现问题时快速恢复到上一个稳定版本的预定方案 |

## 1. 为什么不能"一步到位"

### 1.1 大爆炸替换的历史教训

软件工程史上有大量"大爆炸式重写"失败的案例。对于 KrKr2 这样已经稳定运行、有大量玩家在用的项目，一次性替换所有 UI 代码面临：

**技术风险**：
- Cocos2d-x UI 经过多年测试，隐含大量已知但未文档化的边界行为
- Flutter/Compose 的事件系统与 Cocos2d-x 的触摸模型存在细微差异
- 纹理共享、生命周期同步等桥接代码在极端场景（低内存、快速切换应用）下可能有未知 Bug

**工程风险**：
- 新旧代码同时存在时，依赖关系难以理清
- 一旦发现严重问题，很难快速定位是哪个组件引入的
- 回滚需要撤销整个版本，影响其他已稳定的改动

**用户风险**：
- 核心功能（游戏配置、存档管理）如果在新 UI 下工作异常，会直接影响用户体验
- 不同设备（老旧 Android 机、小内存设备）的兼容性无法在发布前完全测试

### 1.2 渐进式迁移的核心原则

```
原则 1：每次只替换一个对话框
原则 2：新旧代码同时存在，通过 Flag 切换
原则 3：每个阶段完成后必须有可验证的质量指标
原则 4：任何时候都能在 5 分钟内回滚到旧 UI
原则 5：用户感知到的功能不能减少
```

## 2. 四阶段迁移路线图

### 2.1 全局视图

```
Phase 1：建立抽象层（已完成 M13/02）
  └─ 定义 IDialog / IUIManager 接口
  └─ Cocos2d-x 实现完整实现这些接口
  └─ 所有调用方通过接口而非直接引用 Cocos2d-x

Phase 2：并行运行（本章重点）
  └─ 引入 Feature Flag 系统
  └─ 新 UI 实现与旧实现并存
  └─ 通过 Flag 切换：默认走旧路径

Phase 3：逐步切换对话框
  └─ ConfigDialog 替换（低风险，影响范围小）
  └─ GameSelectDialog 替换（中等风险）
  └─ FilePickerDialog 替换（高风险，与文件系统深度耦合）

Phase 4：完全切换，移除旧代码
  └─ 所有对话框验证通过
  └─ Feature Flag 全量开启
  └─ 删除 Cocos2d-x UI 相关代码
  └─ 清理 Flag 基础设施
```

### 2.2 迁移风险矩阵

在开始替换之前，对每个对话框进行风险评估：

| 对话框 | 功能复杂度 | 影响范围 | 文件系统依赖 | 迁移顺序 |
|--------|-----------|---------|-------------|---------|
| `ToastMessage` | 低（纯显示） | 全局 | 无 | 第1个 |
| `ProgressDialog` | 低（加载中） | 局部 | 无 | 第2个 |
| `ConfigDialog` | 中（表单交互） | 玩家设置 | 间接（配置文件） | 第3个 |
| `GameSelectDialog` | 高（列表+缩略图） | 游戏启动 | 是（扫描目录） | 第4个 |
| `FilePickerDialog` | 高（文件系统导航） | 存档/导入 | 是（直接操作） | 第5个 |

## 3. Feature Flag 系统设计

### 3.1 两级开关架构

Feature Flag 分为两个层次：

```
编译时开关（Compile-time Flag）
  └─ CMakeLists.txt 中定义
  └─ 用于完全排除某个 UI 后端的编译
  └─ 修改需要重新编译

运行时开关（Runtime Flag）
  └─ JSON 配置文件或环境变量
  └─ 无需重新编译即可切换
  └─ 支持热更新（重启 App 生效）
  └─ 支持远程下发（A/B 测试）
```

### 3.2 编译时 Flag（CMakeLists.txt）

```cmake
# CMakeLists.txt
# ──────────────────────────────────────────────────────────────
# UI 后端选择（编译时）
# 可选值：COCOS | FLUTTER | COMPOSE | AUTO
# AUTO = 根据平台自动选择
# ──────────────────────────────────────────────────────────────
option(KRKR2_UI_BACKEND "UI Backend" "AUTO")

if(KRKR2_UI_BACKEND STREQUAL "FLUTTER")
    add_compile_definitions(KRKR2_FLUTTER_UI=1)
    set(KRKR2_UI_SOURCES
        ${FLUTTER_UI_SOURCES}
        ${BRIDGE_SOURCES}
    )
elseif(KRKR2_UI_BACKEND STREQUAL "COMPOSE")
    add_compile_definitions(KRKR2_COMPOSE_UI=1)
    set(KRKR2_UI_SOURCES ${COMPOSE_UI_SOURCES})
else()
    # 默认：Cocos2d-x（向后兼容）
    add_compile_definitions(KRKR2_COCOS_UI=1)
    set(KRKR2_UI_SOURCES ${COCOS_UI_SOURCES})
endif()

# 允许并行运行两套 UI（过渡阶段专用）
option(KRKR2_UI_DUAL_MODE "Enable dual UI mode for migration" OFF)
if(KRKR2_UI_DUAL_MODE)
    add_compile_definitions(KRKR2_DUAL_UI=1)
    # 编译两套 UI 的源文件
    list(APPEND KRKR2_UI_SOURCES
        ${COCOS_UI_SOURCES}
        ${FLUTTER_UI_SOURCES}
        ${BRIDGE_SOURCES}
    )
endif()
```

### 3.3 运行时 Flag 管理器

```cpp
// cpp/core/environ/ui/FeatureFlags.h
#pragma once
#include <string>
#include <unordered_map>
#include <functional>

/// 运行时 Feature Flag 管理器（单例）
/// 从 JSON 配置文件加载，支持运行时查询
class FeatureFlags {
public:
    static FeatureFlags& instance();

    /// 从文件加载（App 启动时调用一次）
    bool loadFromFile(const std::string& path);

    /// 从 JSON 字符串加载（用于远程配置下发）
    bool loadFromJson(const std::string& json);

    /// 查询布尔 Flag（Flag 不存在时返回默认值）
    bool getBool(const std::string& key, bool defaultVal = false) const;

    /// 查询字符串 Flag
    std::string getString(const std::string& key,
                          const std::string& defaultVal = "") const;

    /// 注册变更监听器（Flag 更新时回调）
    void addListener(const std::string& key,
                     std::function<void(bool)> callback);

    /// 强制覆盖某个 Flag（测试用）
    void override(const std::string& key, bool value);
    void clearOverrides();

private:
    FeatureFlags() = default;
    std::unordered_map<std::string, std::string> flags_;
    std::unordered_map<std::string,
        std::vector<std::function<void(bool)>>> listeners_;
    std::unordered_map<std::string, bool> overrides_;
    mutable std::mutex mutex_;
};
```

```cpp
// cpp/core/environ/ui/FeatureFlags.cpp
#include "FeatureFlags.h"
#include <fstream>
#include <nlohmann/json.hpp>
using json = nlohmann::json;

FeatureFlags& FeatureFlags::instance() {
    static FeatureFlags inst;
    return inst;
}

bool FeatureFlags::loadFromFile(const std::string& path) {
    std::ifstream f(path);
    if (!f.is_open()) return false;
    std::string content((std::istreambuf_iterator<char>(f)),
                        std::istreambuf_iterator<char>());
    return loadFromJson(content);
}

bool FeatureFlags::loadFromJson(const std::string& jsonStr) {
    try {
        auto j = json::parse(jsonStr);
        std::lock_guard<std::mutex> lock(mutex_);
        for (auto& [key, val] : j.items()) {
            std::string oldVal = flags_[key];
            flags_[key] = val.dump();

            // 通知监听器
            bool newBool = val.is_boolean() ? val.get<bool>()
                          : (val.is_string()
                             ? (val.get<std::string>() == "true")
                             : false);
            if (listeners_.count(key)) {
                for (auto& cb : listeners_[key]) cb(newBool);
            }
        }
        return true;
    } catch (...) {
        return false;
    }
}

bool FeatureFlags::getBool(const std::string& key,
                           bool defaultVal) const {
    std::lock_guard<std::mutex> lock(mutex_);
    // 覆盖值优先
    if (overrides_.count(key)) return overrides_.at(key);
    auto it = flags_.find(key);
    if (it == flags_.end()) return defaultVal;
    // 解析 JSON 值
    try {
        auto v = json::parse(it->second);
        if (v.is_boolean()) return v.get<bool>();
        if (v.is_string()) return v.get<std::string>() == "true";
        return defaultVal;
    } catch (...) {
        return defaultVal;
    }
}

void FeatureFlags::override(const std::string& key, bool value) {
    std::lock_guard<std::mutex> lock(mutex_);
    overrides_[key] = value;
}

void FeatureFlags::clearOverrides() {
    std::lock_guard<std::mutex> lock(mutex_);
    overrides_.clear();
}
```

## 4. UIBackendFactory 的 Flag 驱动路由

### 4.1 工厂方法集成 Feature Flag

在 M13/02 中定义的 `UIBackendFactory` 需要改造为支持 Feature Flag 路由：

```cpp
// cpp/core/environ/ui/abstract/UIBackendFactory.h（更新版）
#pragma once
#include "IDialog.h"
#include "IUIManager.h"
#include "FeatureFlags.h"

/// 对话框类型枚举（用于 Flag 查询）
enum class DialogType {
    Toast,
    Progress,
    Config,
    GameSelect,
    FilePicker,
};

class UIBackendFactory {
public:
    static UIBackendFactory& instance();

    /// 初始化（App 启动时调用，传入 flags 文件路径）
    void initialize(const std::string& flagsPath);

    /// 根据当前 Flag 创建最合适的对话框实现
    std::unique_ptr<IToastMessage> createToast();
    std::unique_ptr<IProgressDialog> createProgress();
    std::unique_ptr<IConfigDialog> createConfigDialog();
    std::unique_ptr<IGameSelectDialog> createGameSelectDialog();
    std::unique_ptr<IFilePickerDialog> createFilePicker();

private:
    UIBackendFactory() = default;

    /// 查询某个对话框类型是否应使用新 UI
    bool shouldUseNewUI(DialogType type) const;

    FeatureFlags& flags_ = FeatureFlags::instance();
};
```

```cpp
// cpp/core/environ/ui/abstract/UIBackendFactory.cpp
#include "UIBackendFactory.h"

// 旧实现（Cocos2d-x）
#include "../cocos/CocosToast.h"
#include "../cocos/CocosConfigDialog.h"
#include "../cocos/CocosGameSelectDialog.h"
#include "../cocos/CocosFilePicker.h"

// 新实现（Flutter）—— 条件编译，仅 DUAL_UI 模式下包含
#ifdef KRKR2_DUAL_UI
#include "../flutter/FlutterToast.h"
#include "../flutter/FlutterConfigDialog.h"
#include "../flutter/FlutterGameSelectDialog.h"
#include "../flutter/FlutterFilePicker.h"
#endif

UIBackendFactory& UIBackendFactory::instance() {
    static UIBackendFactory inst;
    return inst;
}

void UIBackendFactory::initialize(const std::string& flagsPath) {
    flags_.loadFromFile(flagsPath);
}

bool UIBackendFactory::shouldUseNewUI(DialogType type) const {
#ifndef KRKR2_DUAL_UI
    return false;  // 未编译双模式，始终用旧 UI
#else
    switch (type) {
        case DialogType::Toast:
            return flags_.getBool("ui.use_flutter_toast", false);
        case DialogType::Config:
            return flags_.getBool("ui.use_flutter_config_dialog", false);
        case DialogType::GameSelect:
            return flags_.getBool("ui.use_flutter_game_select", false);
        case DialogType::FilePicker:
            return flags_.getBool("ui.use_flutter_file_picker", false);
        default:
            return false;
    }
#endif
}

std::unique_ptr<IConfigDialog> UIBackendFactory::createConfigDialog() {
#ifdef KRKR2_DUAL_UI
    if (shouldUseNewUI(DialogType::Config)) {
        return std::make_unique<FlutterConfigDialog>();
    }
#endif
    return std::make_unique<CocosConfigDialog>();
}

std::unique_ptr<IGameSelectDialog> UIBackendFactory::createGameSelectDialog() {
#ifdef KRKR2_DUAL_UI
    if (shouldUseNewUI(DialogType::GameSelect)) {
        return std::make_unique<FlutterGameSelectDialog>();
    }
#endif
    return std::make_unique<CocosGameSelectDialog>();
}
```

## 5. 各对话框迁移实战

### 5.1 Phase 3-A：Toast 通知替换（最低风险）

Toast 是最简单的迁移起点：无状态，不需要用户交互，失败影响极小。

```cpp
// cpp/core/environ/ui/flutter/FlutterToast.h
#pragma once
#include "../abstract/IUIManager.h"
#include "../flutter/ChannelRouter.h"

class FlutterToast : public IToastMessage {
public:
    explicit FlutterToast(ChannelRouter* router) : router_(router) {}

    void show(const std::string& message,
              ToastType type,
              int durationMs) override {

        nlohmann::json payload = {
            {"message", message},
            {"type",    toastTypeToString(type)},
            {"duration", durationMs}
        };
        router_->invokeMethod("toast/show", payload.dump(),
            [](const std::string& /*result*/) {
                // Toast 不需要回调处理
            });
    }

    void hide() override {
        router_->invokeMethod("toast/hide", "", nullptr);
    }

private:
    static std::string toastTypeToString(ToastType t) {
        switch (t) {
            case ToastType::Info:    return "info";
            case ToastType::Success: return "success";
            case ToastType::Warning: return "warning";
            case ToastType::Error:   return "error";
            default: return "info";
        }
    }
    ChannelRouter* router_;
};
```

**Flutter Dart 侧对应实现：**

```dart
// lib/channels/toast_channel.dart
import 'package:flutter/services.dart';
import 'package:flutter/material.dart';

class ToastChannel {
  static const _channel = MethodChannel('krkr2/toast');
  static OverlayEntry? _currentToast;

  static void initialize(BuildContext context) {
    _channel.setMethodCallHandler((call) async {
      switch (call.method) {
        case 'toast/show':
          final args = call.arguments as Map<String, dynamic>;
          _showToast(context,
              message: args['message'] as String,
              type: args['type'] as String,
              duration: args['duration'] as int);
          break;
        case 'toast/hide':
          _currentToast?.remove();
          _currentToast = null;
          break;
      }
    });
  }

  static void _showToast(BuildContext context, {
    required String message,
    required String type,
    required int duration,
  }) {
    _currentToast?.remove();
    _currentToast = OverlayEntry(
      builder: (_) => Positioned(
        bottom: 80,
        left: 16, right: 16,
        child: Material(
          elevation: 4,
          borderRadius: BorderRadius.circular(8),
          color: _colorForType(type),
          child: Padding(
            padding: const EdgeInsets.symmetric(
                horizontal: 16, vertical: 12),
            child: Text(message,
                style: const TextStyle(color: Colors.white)),
          ),
        ),
      ),
    );
    Overlay.of(context)?.insert(_currentToast!);
    Future.delayed(Duration(milliseconds: duration), () {
      _currentToast?.remove();
      _currentToast = null;
    });
  }

  static Color _colorForType(String type) {
    switch (type) {
      case 'success': return Colors.green.shade700;
      case 'warning': return Colors.orange.shade700;
      case 'error':   return Colors.red.shade700;
      default:        return Colors.blueGrey.shade700;
    }
  }
}
```

### 5.2 Phase 3-B：ConfigDialog 替换

ConfigDialog 是玩家最常使用的界面，替换时需要做完整的表单数据对等验证：

```cpp
// cpp/core/environ/ui/flutter/FlutterConfigDialog.h
#pragma once
#include "../abstract/IDialog.h"
#include "../flutter/ChannelRouter.h"
#include <functional>

class FlutterConfigDialog : public IConfigDialog {
public:
    explicit FlutterConfigDialog(ChannelRouter* router) : router_(router) {}

    void show() override {
        // 先读取当前配置，传给 Flutter 侧初始化表单
        auto config = ConfigManager::getInstance()->getAllValues();
        nlohmann::json payload = config.toJson();

        router_->invokeMethod("dialog/config/open", payload.dump(),
            [this](const std::string& result) {
                // Flutter 侧确认打开后，记录状态
                isVisible_ = true;
            });
    }

    void hide() override {
        router_->invokeMethod("dialog/config/close", "", nullptr);
        isVisible_ = false;
    }

    bool isVisible() const override { return isVisible_; }

    void onConfigChanged(std::function<void(const ConfigValues&)> cb) override {
        onChangedCb_ = std::move(cb);
        // 注册 Flutter → C++ 回调
        router_->registerHandler("dialog/config/apply",
            [this](const std::string& data) -> std::string {
                ConfigValues vals = ConfigValues::fromJson(
                    nlohmann::json::parse(data));
                if (onChangedCb_) onChangedCb_(vals);
                return "ok";
            });
    }

private:
    ChannelRouter* router_;
    bool isVisible_ = false;
    std::function<void(const ConfigValues&)> onChangedCb_;
};
```

## 6. 回滚策略

### 6.1 快速回滚路径

回滚的核心是 Feature Flag，不需要代码变更：

```bash
# 紧急回滚：通过配置更新关闭所有新 UI
# 1. 更新服务端配置（如果使用远程配置）
curl -X POST https://config.example.com/api/flags \
  -H "Authorization: Bearer $TOKEN" \
  -d '{
    "ui.use_flutter_toast": false,
    "ui.use_flutter_config_dialog": false,
    "ui.use_flutter_game_select": false,
    "ui.use_flutter_file_picker": false
  }'

# 2. 或者：覆盖本地配置文件（App 重启后生效）
adb push feature_flags_rollback.json \
  /data/data/org.krkr2/files/feature_flags.json
```

### 6.2 三级回滚机制

```
Level 1：Flag 回滚（5分钟内生效）
  └─ 关闭对应 Feature Flag
  └─ 重启 App 后生效
  └─ 无需重新打包

Level 2：热更新回滚（15分钟内生效）
  └─ 下发新的 flag 配置（通过远程配置系统）
  └─ App 下次启动时拉取新配置
  └─ 无需更新 App 版本

Level 3：版本回滚（1-2小时，极端情况）
  └─ 在应用商店/GitHub Release 发布回退版本
  └─ 用于 Level 1/2 无法解决的崩溃问题
```

### 6.3 回滚后的状态保障

回滚不能让用户丢失数据：

```cpp
// 在切换 UI 后端时，确保数据层不变
// 数据层（ConfigManager、SaveManager）是独立的，不依赖 UI 后端
// 无论使用哪个 UI，数据都写入相同的文件路径

// 验证数据兼容性的单元测试：
TEST(UIBackendSwitch, ConfigDataPreserved) {
    // 用旧 UI 写入配置
    auto oldDialog = std::make_unique<CocosConfigDialog>();
    oldDialog->setConfig("screen_mode", "fullscreen");
    oldDialog->applyAndSave();

    // 用新 UI 读取配置
    auto newDialog = std::make_unique<FlutterConfigDialog>(router);
    EXPECT_EQ(newDialog->getConfig("screen_mode"), "fullscreen");
}
```

## 练习题与答案

**练习 1**：在 `UIBackendFactory::createConfigDialog()` 中，如果 `FlutterConfigDialog` 构造失败（例如 Flutter Engine 未初始化），应该降级到旧实现还是直接崩溃？为什么？

<details>
<summary>答案</summary>

应该**降级到旧实现**（Graceful Degradation），而不是崩溃。原因：

1. 对话框功能是可选的 UI 增强，不是核心游戏逻辑，降级不影响游戏正常运行
2. 崩溃会导致玩家直接失去所有未保存进度，体验极差
3. Flutter Engine 初始化失败是可预期的边界情况（低内存设备、缺少 so 文件等）

实现方式：
```cpp
std::unique_ptr<IConfigDialog> UIBackendFactory::createConfigDialog() {
#ifdef KRKR2_DUAL_UI
    if (shouldUseNewUI(DialogType::Config)) {
        try {
            return std::make_unique<FlutterConfigDialog>(getRouter());
        } catch (const std::exception& e) {
            CCLOGWARN("FlutterConfigDialog init failed: %s, fallback to Cocos", e.what());
            // 自动降级
        }
    }
#endif
    return std::make_unique<CocosConfigDialog>();
}
```

</details>

**练习 2**：Feature Flag 的运行时配置文件存在安全风险：恶意用户可以修改设备上的 `feature_flags.json` 来开启未完成的实验性功能。在 KrKr2 这个开源游戏引擎的场景下，这个风险是否值得防护？如何防护？

<details>
<summary>答案</summary>

在 KrKr2 的场景下，**风险较低，但值得适度防护**。原因分析：

- KrKr2 是开源项目，源代码公开，"实验性功能"本身没有商业秘密价值
- 玩家修改配置只影响自己的设备，没有服务端数据被窃取的风险
- 但如果未完成的功能存在严重 Bug（崩溃、存档损坏），可能导致玩家数据丢失和差评

适度防护方案：
1. **只在 Debug 构建中读取本地 flags 文件**，Release 构建使用编译时写死的默认值
2. 或：对配置文件内容做简单校验（如 HMAC 签名），防止随意篡改
3. 彻底方案：Release 构建中使用编译时常量替换所有 Flag 查询，消除运行时可篡改性

```cpp
#ifdef NDEBUG
// Release 构建：硬编码所有 flag 值
bool FeatureFlags::getBool(const std::string& key, bool defaultVal) const {
    return defaultVal;  // 总是返回保守默认值
}
#endif
```

</details>

**练习 3**：假设迁移到第3阶段（ConfigDialog 已用 Flutter 实现）后，发现 Flutter ConfigDialog 在中文 IME（输入法）下存在文字输入延迟问题。按照三级回滚机制，应该选哪一级？执行哪些具体步骤？

<details>
<summary>答案</summary>

选择 **Level 1：Flag 回滚**（最快速，5分钟内生效）。

具体步骤：
1. 将 `feature_flags.json`（或远程配置）中 `"ui.use_flutter_config_dialog"` 改为 `false`
2. 在 GitHub Issues 中记录 Bug：IME 延迟、复现步骤、影响版本
3. 通知正在测试的用户重启 App（或下次启动自动生效）
4. 在开发环境中复现问题，排查 `FlutterEngineSendKeyEvent` 的实现是否正确处理了输入法上屏事件（`commitText`）
5. 修复后重新开启 Flag 验证

选择 Level 1 而非 Level 2 或 Level 3 的原因：问题是功能性 Bug（不是崩溃），不需要发版；只影响一个对话框，影响范围小；修改配置文件可以立即生效。

</details>
