## 术语预览

| 术语 | 含义 |
|------|------|
| A/B 测试 | 将用户随机分为两组（A=旧版，B=新版），对比关键指标的统计方法 |
| 回归测试 | 验证新改动没有破坏原有功能的测试类型 |
| 视觉回归 | UI 外观发生了意外变化（颜色、布局、字体等），与功能无关 |
| 截图对比（Snapshot Test） | 将 UI 渲染为图片，与基准图片做像素级对比，发现视觉变化 |
| 基准图片（Baseline） | 已知正确的截图，用于后续对比的参考标准 |
| 像素差异率 | 两张图片中像素值不同的比例，用于量化视觉差异 |
| `glReadPixels` | OpenGL ES API，将 GPU 帧缓冲区内容读回 CPU 内存 |
| 帧时间（Frame Time） | 渲染单帧所需时间（ms），帧时间 < 16.6ms 即 60fps |
| P50/P95/P99 | 第 50/95/99 百分位数，描述延迟分布的统计指标 |
| SSIM | 结构相似性指数，比像素差异更接近人眼感知的图像质量指标（范围 0~1，1=完全相同） |

## 1. A/B 测试框架设计

### 1.1 为什么 UI 迁移需要 A/B 测试

仅凭主观感受"新 UI 看起来更好"不足以做出发布决策。A/B 测试提供客观数据：

- **用户行为数据**：新 UI 下用户完成配置的成功率是否下降？
- **性能数据**：新 UI 在低端设备上的对话框打开耗时是否可接受？
- **稳定性数据**：新 UI 的崩溃率是否比旧版高？
- **用户满意度**：通过日志中的操作路径，判断用户是否能顺利找到所需功能

### 1.2 分组策略

KrKr2 的 A/B 分组基于设备 ID 的哈希值，确保同一设备始终在同一组：

```cpp
// cpp/core/environ/ui/ABTestManager.h
#pragma once
#include <string>
#include <cstdint>

/// A/B 测试分组管理器
/// 基于设备 ID 哈希，保证同一设备分组稳定
class ABTestManager {
public:
    static ABTestManager& instance();

    /// 初始化（传入设备唯一标识符）
    void initialize(const std::string& deviceId);

    /// 判断当前设备是否在实验组 B（新 UI）
    /// @param experimentName 实验名称（用于隔离不同实验）
    /// @param rolloutPercent  B 组比例（0-100）
    bool isInGroupB(const std::string& experimentName,
                    int rolloutPercent = 50) const;

    /// 记录事件（用于后续分析）
    void logEvent(const std::string& eventName,
                  const std::string& experimentName,
                  const std::string& group,
                  double value = 0.0);

private:
    ABTestManager() = default;

    /// 确定性哈希：相同输入始终产生相同输出
    static uint32_t deterministicHash(const std::string& input);

    std::string deviceId_;
};
```

```cpp
// cpp/core/environ/ui/ABTestManager.cpp
#include "ABTestManager.h"
#include <functional>
#include <fstream>

ABTestManager& ABTestManager::instance() {
    static ABTestManager inst;
    return inst;
}

void ABTestManager::initialize(const std::string& deviceId) {
    deviceId_ = deviceId;
}

uint32_t ABTestManager::deterministicHash(const std::string& input) {
    // FNV-1a 哈希（确定性，无随机种子）
    uint32_t hash = 2166136261u;
    for (unsigned char c : input) {
        hash ^= c;
        hash *= 16777619u;
    }
    return hash;
}

bool ABTestManager::isInGroupB(const std::string& experimentName,
                               int rolloutPercent) const {
    // 将设备 ID + 实验名拼接，确保不同实验的分组独立
    std::string key = deviceId_ + ":" + experimentName;
    uint32_t hash = deterministicHash(key);
    // 取 hash 的后 7 位（0~127），与 rolloutPercent 对应的阈值对比
    uint32_t bucket = hash % 100;  // 0~99
    return bucket < static_cast<uint32_t>(rolloutPercent);
}

void ABTestManager::logEvent(const std::string& eventName,
                             const std::string& experimentName,
                             const std::string& group,
                             double value) {
    // 追加到本地日志文件（实际项目中可上报到分析后台）
    std::ofstream log("ab_test_events.jsonl", std::ios::app);
    if (!log.is_open()) return;

    // JSON Lines 格式（每行一个 JSON 对象）
    log << "{"
        << "\"event\":\"" << eventName << "\","
        << "\"experiment\":\"" << experimentName << "\","
        << "\"group\":\"" << group << "\","
        << "\"value\":" << value << ","
        << "\"ts\":" << time(nullptr)
        << "}\n";
}
```

## 2. 截图对比测试（视觉回归）

### 2.1 截图捕获原理

KrKr2 使用 OpenGL ES 渲染，截图通过 `glReadPixels` 从帧缓冲区读取：

```cpp
// cpp/testing/ScreenshotCapture.h
#pragma once
#include <string>
#include <vector>
#include <cstdint>

struct Screenshot {
    int width;
    int height;
    std::vector<uint8_t> pixels;  // RGBA，每像素 4 字节，行序从上到下
};

class ScreenshotCapture {
public:
    /// 在 GL 线程上调用，捕获当前帧
    static Screenshot capture(int x, int y, int width, int height);

    /// 将截图保存为 PPM 文件（无依赖，便于调试）
    static bool saveAsPPM(const Screenshot& s, const std::string& path);

    /// 从 PPM 文件加载
    static Screenshot loadFromPPM(const std::string& path);
};
```

```cpp
// cpp/testing/ScreenshotCapture.cpp
#include "ScreenshotCapture.h"
#include <GLES2/gl2.h>
#include <fstream>
#include <algorithm>

Screenshot ScreenshotCapture::capture(int x, int y, int w, int h) {
    Screenshot s;
    s.width = w;
    s.height = h;
    s.pixels.resize(w * h * 4);

    // glReadPixels 从 GL 坐标系读取（Y 从底部向上）
    std::vector<uint8_t> raw(w * h * 4);
    glReadPixels(x, y, w, h, GL_RGBA, GL_UNSIGNED_BYTE, raw.data());

    // 翻转 Y 轴（GL 底部向上 → 图像顶部向下）
    for (int row = 0; row < h; ++row) {
        memcpy(
            s.pixels.data() + row * w * 4,
            raw.data() + (h - 1 - row) * w * 4,
            w * 4
        );
    }
    return s;
}

bool ScreenshotCapture::saveAsPPM(const Screenshot& s,
                                   const std::string& path) {
    std::ofstream f(path, std::ios::binary);
    if (!f.is_open()) return false;
    // PPM 头部
    f << "P6\n" << s.width << " " << s.height << "\n255\n";
    // 写入 RGB（跳过 A 通道）
    for (int i = 0; i < s.width * s.height; ++i) {
        f.write(reinterpret_cast<const char*>(&s.pixels[i * 4]), 3);
    }
    return true;
}
```

### 2.2 像素差异计算

```cpp
// cpp/testing/VisualDiff.h
#pragma once
#include "ScreenshotCapture.h"
#include <cmath>

struct DiffResult {
    double pixelDiffRatio;   // 0.0~1.0，差异像素比例
    double ssim;             // 0.0~1.0，结构相似性（1=完全相同）
    int diffPixelCount;      // 绝对差异像素数
    Screenshot diffImage;    // 差异可视化（红色=差异处）
};

class VisualDiff {
public:
    static constexpr double kDefaultThreshold = 0.02;  // 2% 差异视为失败

    /// 对比两张截图，返回差异报告
    static DiffResult compare(const Screenshot& baseline,
                              const Screenshot& current,
                              int tolerance = 8);  // 每通道容差

    /// 判断差异是否在可接受范围内
    static bool isAcceptable(const DiffResult& result,
                             double threshold = kDefaultThreshold);
};
```

```cpp
// cpp/testing/VisualDiff.cpp
#include "VisualDiff.h"
#include <algorithm>
#include <numeric>

DiffResult VisualDiff::compare(const Screenshot& baseline,
                               const Screenshot& current,
                               int tolerance) {
    DiffResult result;
    result.diffImage = baseline;  // 初始化为 baseline 大小

    if (baseline.width  != current.width ||
        baseline.height != current.height) {
        result.pixelDiffRatio = 1.0;
        result.ssim = 0.0;
        result.diffPixelCount = baseline.width * baseline.height;
        return result;
    }

    int total = baseline.width * baseline.height;
    result.diffPixelCount = 0;
    result.diffImage.pixels.resize(total * 4, 0);

    for (int i = 0; i < total; ++i) {
        int offset = i * 4;
        bool diff = false;
        for (int c = 0; c < 3; ++c) {  // 只比较 RGB
            int delta = std::abs(
                (int)baseline.pixels[offset + c] -
                (int)current.pixels[offset + c]
            );
            if (delta > tolerance) {
                diff = true;
                break;
            }
        }
        if (diff) {
            ++result.diffPixelCount;
            // 差异处标红
            result.diffImage.pixels[offset + 0] = 255;  // R
            result.diffImage.pixels[offset + 1] = 0;    // G
            result.diffImage.pixels[offset + 2] = 0;    // B
            result.diffImage.pixels[offset + 3] = 255;  // A
        } else {
            // 相同处保留原图（半透明显示）
            result.diffImage.pixels[offset + 0] =
                baseline.pixels[offset + 0] / 2;
            result.diffImage.pixels[offset + 1] =
                baseline.pixels[offset + 1] / 2;
            result.diffImage.pixels[offset + 2] =
                baseline.pixels[offset + 2] / 2;
            result.diffImage.pixels[offset + 3] = 128;
        }
    }

    result.pixelDiffRatio = (double)result.diffPixelCount / total;

    // 简化 SSIM（仅用于示意；完整实现需要滑动窗口方差计算）
    // 这里使用 1 - 差异率作为近似
    result.ssim = 1.0 - result.pixelDiffRatio;

    return result;
}

bool VisualDiff::isAcceptable(const DiffResult& result,
                              double threshold) {
    return result.pixelDiffRatio <= threshold;
}
```

### 2.3 视觉回归测试套件

```cpp
// cpp/testing/VisualRegressionTest.cpp
#include "VisualDiff.h"
#include "ScreenshotCapture.h"
#include <gtest/gtest.h>
#include <filesystem>

class VisualRegressionTest : public ::testing::Test {
protected:
    std::string baselineDir_ = "test_baselines/";
    std::string outputDir_   = "test_output/";

    void SetUp() override {
        std::filesystem::create_directories(outputDir_);
    }

    /// 截图并与基准对比。基准不存在时自动创建
    void assertVisualMatch(const std::string& name,
                           int x, int y, int w, int h,
                           double threshold = 0.02) {
        auto current = ScreenshotCapture::capture(x, y, w, h);
        std::string baselinePath = baselineDir_ + name + ".ppm";

        if (!std::filesystem::exists(baselinePath)) {
            // 首次运行：创建基准
            ScreenshotCapture::saveAsPPM(current, baselinePath);
            GTEST_SKIP() << "Baseline created: " << baselinePath;
            return;
        }

        auto baseline = ScreenshotCapture::loadFromPPM(baselinePath);
        auto diff = VisualDiff::compare(baseline, current);

        // 保存差异图（失败时方便排查）
        std::string diffPath = outputDir_ + name + "_diff.ppm";
        ScreenshotCapture::saveAsPPM(diff.diffImage, diffPath);

        EXPECT_TRUE(VisualDiff::isAcceptable(diff, threshold))
            << "Visual regression detected in: " << name
            << "\n  Diff pixels: " << diff.diffPixelCount
            << " (" << (diff.pixelDiffRatio * 100) << "%)"
            << "\n  SSIM: " << diff.ssim
            << "\n  Diff image: " << diffPath;
    }
};

// 测试用例：ConfigDialog 关闭状态（初始界面）
TEST_F(VisualRegressionTest, ConfigDialogClosed) {
    // 确保对话框已关闭
    UIManagerImpl::getInstance()->hideAllDialogs();
    // 等待一帧渲染完成
    renderOneFrame();
    // 截取界面右上角配置按钮区域
    assertVisualMatch("config_button_area", 1180, 10, 100, 40);
}

// 测试用例：ConfigDialog 打开状态
TEST_F(VisualRegressionTest, ConfigDialogOpen) {
    UIManagerImpl::getInstance()->showConfigDialog();
    renderOneFrame();
    // 截取整个对话框区域
    assertVisualMatch("config_dialog_open", 200, 100, 880, 600);
}

// 测试用例：Toast 显示（成功类型）
TEST_F(VisualRegressionTest, ToastSuccess) {
    UIManagerImpl::getInstance()->showToast(
        "设置已保存", ToastType::Success, 3000);
    renderOneFrame();
    assertVisualMatch("toast_success", 100, 620, 1080, 60, 0.05);
}
```

## 4. 性能对比测试

### 4.1 帧时间测量框架

```cpp
// cpp/testing/PerformanceProfiler.h
#pragma once
#include <chrono>
#include <vector>
#include <string>
#include <algorithm>
#include <numeric>

/// 单次操作的帧时间记录器
class PerformanceProfiler {
public:
    using Clock = std::chrono::high_resolution_clock;
    using Duration = std::chrono::duration<double, std::milli>;

    void start() {
        startTime_ = Clock::now();
    }

    void stop() {
        auto elapsed = Duration(Clock::now() - startTime_).count();
        samples_.push_back(elapsed);
    }

    /// 统计报告
    struct Report {
        double min, max, p50, p95, p99, mean;
        int sampleCount;

        std::string toString() const {
            char buf[256];
            snprintf(buf, sizeof(buf),
                "count=%d  mean=%.2fms  p50=%.2fms"
                "  p95=%.2fms  p99=%.2fms  [%.2f, %.2f]",
                sampleCount, mean, p50, p95, p99, min, max);
            return buf;
        }
    };

    Report report() const {
        if (samples_.empty()) return {};
        std::vector<double> sorted = samples_;
        std::sort(sorted.begin(), sorted.end());

        Report r;
        r.sampleCount = (int)sorted.size();
        r.min   = sorted.front();
        r.max   = sorted.back();
        r.mean  = std::accumulate(sorted.begin(), sorted.end(), 0.0)
                  / sorted.size();
        r.p50   = percentile(sorted, 0.50);
        r.p95   = percentile(sorted, 0.95);
        r.p99   = percentile(sorted, 0.99);
        return r;
    }

    void reset() { samples_.clear(); }

private:
    static double percentile(const std::vector<double>& sorted,
                             double p) {
        int idx = (int)(p * (sorted.size() - 1));
        return sorted[idx];
    }

    Clock::time_point startTime_;
    std::vector<double> samples_;
};
```

```cpp
// cpp/testing/UIPerformanceBenchmark.cpp
#include <gtest/gtest.h>
#include "PerformanceProfiler.h"
#include "UIBackendFactory.h"
#include "FeatureFlags.h"

/// 对比 Cocos2d-x / Flutter / Compose 三种后端的 ConfigDialog 打开耗时
class UIPerformanceBenchmark : public ::testing::Test {
protected:
    static constexpr int kWarmupRuns = 5;
    static constexpr int kMeasureRuns = 50;

    void benchmarkConfigDialogOpen(const std::string& backendName,
                                   UIBackend backend) {
        // 强制指定后端
        setBackend(backend);
        auto& factory = UIBackendFactory::instance();

        PerformanceProfiler profiler;

        // 预热（避免首次冷启动影响测量）
        for (int i = 0; i < kWarmupRuns; ++i) {
            auto dlg = factory.createConfigDialog();
            dlg->show();
            renderOneFrame();
            dlg->hide();
        }

        // 正式测量
        for (int i = 0; i < kMeasureRuns; ++i) {
            auto dlg = factory.createConfigDialog();
            profiler.start();
            dlg->show();
            renderOneFrame();
            profiler.stop();
            dlg->hide();
        }

        auto report = profiler.report();
        std::cout << "[" << backendName << "] ConfigDialog open: "
                  << report.toString() << "\n";

        // 断言性能在可接受范围内（P95 < 300ms）
        EXPECT_LT(report.p95, 300.0)
            << backendName << " P95 open time exceeds 300ms";
    }

    void setBackend(UIBackend backend) {
        auto& flags = FeatureFlags::instance();
        flags.clearOverrides();
        switch (backend) {
            case UIBackend::Flutter:
                flags.override("ui.use_flutter_config_dialog", true);
                break;
            default:
                break;
        }
    }
};
```

### 4.3 三框架性能对比矩阵（预期数据）

以下数据是在典型 Android 中端机（Snapdragon 720G, 6GB RAM）上的预期范围，实际测量结果因设备差异而异：

| 指标 | Cocos2d-x UI | Flutter UI | Compose UI |
|------|-------------|-----------|-----------|
| 对话框冷启动（首次） | ~50ms | ~200ms（引擎预热后降至 ~80ms） | ~150ms |
| 对话框打开 P50 | ~20ms | ~30ms | ~25ms |
| 对话框打开 P95 | ~40ms | ~80ms | ~60ms |
| 对话框打开 P99 | ~80ms | ~150ms | ~100ms |
| 帧时间（对话框渲染中）P50 | ~8ms | ~12ms | ~10ms |
| 首帧内存增量 | ~2MB | ~8MB（Flutter heap） | ~5MB |
| 低内存设备（1GB RAM）稳定性 | 高 | 中（需调整 Flutter 缓存上限） | 中高 |

**关键结论**：
1. Flutter 的**首次冷启动**比 Cocos2d-x 慢约 4 倍，但引擎预热后差距缩小到 1.5 倍
2. 可以通过在 App 启动后台预热 Flutter 引擎来消除冷启动延迟
3. Compose 的性能介于两者之间，在 Android 12+ 上受益于 Hardware Compositing 优化

### 4.4 Flutter 引擎预热策略

```cpp
// App 启动时在后台预热 Flutter 引擎，消除冷启动延迟
void UIManagerImpl::preWarmFlutterEngine() {
    // 在独立线程上初始化引擎（不阻塞主线程）
    std::thread([this] {
        try {
            flutterEngine_ = std::make_unique<FlutterEngineWrapper>();
            flutterEngine_->initialize(getProjectArgs());
            // 发送一个空的渲染请求，触发 Dart 代码预热
            flutterEngine_->sendEmptyFrame();
        } catch (const std::exception& e) {
            CCLOGERROR("Flutter engine prewarm failed: %s", e.what());
        }
    }).detach();
}
```

## 5. 测试数据分析

### 5.1 A/B 测试日志分析脚本

```python
#!/usr/bin/env python3
# scripts/analyze_ab_test.py
"""
分析 ab_test_events.jsonl 日志，生成对比报告
用法：python analyze_ab_test.py --input ab_test_events.jsonl --experiment config_dialog_flutter
"""

import json
import sys
import argparse
from collections import defaultdict
from statistics import mean, median, quantiles

def load_events(path):
    events = []
    with open(path) as f:
        for line in f:
            line = line.strip()
            if line:
                events.append(json.loads(line))
    return events

def analyze(events, experiment):
    groups = defaultdict(lambda: defaultdict(list))
    
    for e in events:
        if e.get('experiment') != experiment:
            continue
        group = e.get('group', 'unknown')
        event_name = e.get('event', '')
        value = e.get('value', 0)
        groups[group][event_name].append(value)
    
    print(f"\n=== A/B Test Report: {experiment} ===\n")
    
    for group_name in sorted(groups.keys()):
        data = groups[group_name]
        print(f"--- Group {group_name} ---")
        
        if 'config_dialog_open_ms' in data:
            times = data['config_dialog_open_ms']
            qs = quantiles(times, n=20)  # 5% intervals
            print(f"  Open time: n={len(times)}, mean={mean(times):.1f}ms, "
                  f"p50={median(times):.1f}ms, p95={qs[18]:.1f}ms")
        
        if 'config_dialog_applied' in data:
            applied = len(data['config_dialog_applied'])
            cancelled = len(data.get('config_dialog_cancelled', []))
            total = applied + cancelled
            if total > 0:
                conversion = applied / total * 100
                print(f"  Conversion rate: {conversion:.1f}% "
                      f"({applied}/{total} applied)")
        
        if 'dialog_create_failed' in data:
            fails = len(data['dialog_create_failed'])
            print(f"  ⚠ Creation failures: {fails}")
        
        print()

if __name__ == '__main__':
    parser = argparse.ArgumentParser()
    parser.add_argument('--input', default='ab_test_events.jsonl')
    parser.add_argument('--experiment', required=True)
    args = parser.parse_args()
    
    events = load_events(args.input)
    analyze(events, args.experiment)
```

### 5.2 决策标准

完成 A/B 测试后，依据以下标准决定是否全量推送新 UI：

```
全量推送（Green Light）条件（需全部满足）：
  ✅ 新 UI（B 组）对话框打开 P95 < 旧 UI（A 组）× 2.0（最多慢一倍）
  ✅ 新 UI 转化率（用户点击"应用"的比例）≥ 旧 UI × 0.95（不低于 5%）
  ✅ 新 UI 崩溃率 < 0.1%（每1000次对话框操作少于1次崩溃）
  ✅ 视觉回归测试 100% 通过（无 >2% 像素差异）
  ✅ 功能回归测试 100% 通过

暂停发布（Yellow Light）条件（出现任意一条则暂停并调查）：
  ⚠ 新 UI P95 比旧 UI 慢 2~5 倍
  ⚠ 转化率下降 5~15%
  ⚠ 出现视觉回归但 SSIM > 0.90（轻微差异）

回滚（Red Light）条件（出现任意一条则立即回滚）：
  🔴 任何 SSIM < 0.80 的视觉回归（明显 UI 破损）
  🔴 功能回归测试失败（如配置不被保存）
  🔴 崩溃率 > 0.5%
  🔴 新 UI P99 > 2000ms（超过2秒，用户明显感知卡顿）
```

## 练习题与答案

**练习 1**：`ABTestManager::deterministicHash` 使用 FNV-1a 算法而非 `std::hash`。为什么不能用 `std::hash`？

<details>
<summary>答案</summary>

`std::hash` 的实现在 C++ 标准中是**未规定的**（implementation-defined），不同编译器、不同版本的 STL 可能返回不同的哈希值。更严重的是，C++14 之后某些实现会在每次程序启动时引入随机盐（salt）以防止哈希表碰撞攻击，导致同一设备在不同次启动时得到不同的哈希值。

A/B 测试要求**同一设备在所有版本、所有启动中始终分配到同一组**。使用 FNV-1a（或 MurmurHash3 等确定性哈希）可以保证这一点：相同输入永远产生相同输出，跨平台、跨编译器一致。

</details>

---

**练习 2**：在视觉回归测试中，`tolerance = 8`（每通道容差 8/255 ≈ 3%）的设计意图是什么？设置为 0 会有什么问题？

<details>
<summary>答案</summary>

`tolerance = 8` 的设计意图是**容忍抗锯齿、子像素渲染和字体渲染引擎之间的微小差异**。不同平台（Android vs Windows）的字体渲染使用不同的 hinting 算法，即使渲染"完全相同"的文字，像素级别的颜色值也可能有 1~5/255 的差异。如果设置 `tolerance = 0`，这些微小差异会导致测试频繁误报（False Positive），使测试失去实际意义。

`tolerance = 8` 是经验值，足够过滤掉字体和抗锯齿引起的正常差异，同时足够严格以捕捉真正的视觉回归（如颜色变了、元素消失、布局偏移）。

</details>

---

**练习 3**：性能测试中使用了 `kWarmupRuns = 5` 的预热轮次。如果不做预热，测量数据会出现什么偏差？

<details>
<summary>答案</summary>

不预热时，前几次测量会包含以下一次性开销：

1. **JIT 编译**：Flutter 引擎的 Dart 代码在首次执行时会被 JIT 编译（Debug/Profile 模式下），第一次调用明显慢
2. **磁盘 I/O 缓存**：首次加载对话框资源（图片、字体、布局文件）会触发文件系统读取，后续调用从内存缓存中读取
3. **OpenGL 着色器编译**：首次渲染时 GPU 驱动需要编译 GLSL 着色器程序，后续复用编译结果
4. **内存分配器热身**：`malloc` 在首次大量分配时可能触发更多 `sbrk/mmap` 系统调用

这些因素会导致前几次测量的数字远高于"稳态"性能，使 P95/P99 数据虚高。预热后测量的是 App 在用户日常使用中（对话框被多次打开后）的真实性能。

</details>

---

**练习 4（综合）**：假设 A/B 测试结果显示：B 组（Flutter UI）的转化率（用户点击"应用"的比例）比 A 组低了 8%。这是否一定说明 Flutter UI 更难用？还有哪些可能的解释？如何进一步调查？

<details>
<summary>答案</summary>

不一定说明 Flutter UI 更难用。8% 的转化率下降可能有以下解释：

1. **样本偏差**：B 组设备可能碰巧集中了更多"只浏览不修改配置"的用户
2. **渲染延迟影响认知**：如果 Flutter UI 打开稍慢（例如 P95 = 200ms vs Cocos = 40ms），部分用户可能以为"加载失败"而直接关闭
3. **UI 差异导致的发现性问题**：Flutter UI 的"应用"按钮可能在视觉上不够突出，用户不知道需要点击（但这是 Bug，不是"难用"）
4. **Bug**：某些设备上"应用"按钮点击无响应（没有日志会忽略这种情况）

**调查方法**：
- 拆分分析：分别看"用户点开了对话框但没有修改任何设置就关闭"和"用户修改了设置但没有点应用"的比例，定位转化漏斗的哪一步出了问题
- 在 Flutter 侧的"应用"按钮增加点击日志，验证按钮点击是否被正确记录
- 进行用户体验录屏分析（如果合规），观察用户操作路径

</details>
