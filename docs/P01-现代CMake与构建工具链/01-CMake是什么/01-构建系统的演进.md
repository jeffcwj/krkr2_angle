## 手动编译的痛点

初学 C++ 时，一条命令就能编译：

```bash
g++ main.cpp -o main
```

但项目变大后，这种方式会迅速失控。

### 1）多文件编译命令膨胀

先看一个可运行的最小多文件示例：

```cpp
// main.cpp
#include "math_utils.h"
#include <iostream>

int main() {
    std::cout << "sum=" << add(20, 22) << std::endl;
    return 0;
}
```

```cpp
// math_utils.h
#pragma once
int add(int a, int b);
```

```cpp
// math_utils.cpp
#include "math_utils.h"
int add(int a, int b) {
    return a + b;
}
```

手动构建命令如下：

```bash
g++ -std=c++17 -Wall -Wextra -c main.cpp -o main.o
g++ -std=c++17 -Wall -Wextra -c math_utils.cpp -o math_utils.o
g++ main.o math_utils.o -o calc
./calc
```

当文件从 2 个增长到 200 个时，
你要维护的命令数量、顺序和参数会显著增加。

### 2）头文件依赖关系难以追踪

C++ 的编译单元以 `.cpp` 为边界。
头文件会在预处理阶段被展开。
如果一个公共头文件被多个源文件包含，
修改一次头文件就可能触发大规模重编译。
这就是为什么大型项目必须依赖自动化构建系统维护依赖图。

### 3）第三方库链接参数复杂

一个稍复杂的手动命令可能长这样：

```bash
g++ main.cpp renderer.cpp audio.cpp \
  -I/usr/local/include/sdl2 \
  -I/usr/local/include/freetype2 \
  -L/usr/local/lib -lSDL2 -lfreetype \
  -Wl,-rpath,/usr/local/lib \
  -o game
```

你需要自己维护 include 路径、库路径、链接顺序。

### 4）跨平台命令分裂

同一份源码，
不同平台命令并不统一：

```bash
# Windows (MSVC)
cl /EHsc /std:c++17 /W4 hello.cpp /Fe:hello.exe
```

```bash
# Linux (GCC)
g++ -std=c++17 -Wall -Wextra hello.cpp -o hello
```

```bash
# macOS (Apple Clang)
clang++ -std=c++17 -Wall -Wextra hello.cpp -o hello
```

```bash
# Android (NDK arm64-v8a)
$ANDROID_NDK/toolchains/llvm/prebuilt/linux-x86_64/bin/aarch64-linux-android21-clang++ \
  -std=c++17 -fPIC hello.cpp -o libhello.so
```

跨平台项目维护这类命令非常容易失控。

## Makefile 的诞生和语法

Make 通过规则表达“目标如何由依赖生成”。

基本语法：

```makefile
target: dep1 dep2
	command
```

下面是完整可运行示例：

```makefile
# Makefile
CXX := g++
CXXFLAGS := -std=c++17 -Wall -Wextra -O2 -MMD -MP
TARGET := calc
SRCS := main.cpp math_utils.cpp
OBJS := $(SRCS:.cpp=.o)
DEPS := $(OBJS:.o=.d)

.PHONY: all clean run
all: $(TARGET)

$(TARGET): $(OBJS)
	$(CXX) $(OBJS) -o $@

%.o: %.cpp
	$(CXX) $(CXXFLAGS) -c $< -o $@

run: $(TARGET)
	./$(TARGET)

clean:
	rm -f $(OBJS) $(DEPS) $(TARGET)

-include $(DEPS)
```

```bash
make
./calc
make clean
```

## Autotools 工作流程

Autotools 常见于传统 GNU/Unix 项目。
典型流程：

1. `configure` 检测环境并生成 Makefile。
2. `make` 执行构建。
3. `make install` 安装产物。

命令示例：

```bash
autoreconf -i
./configure --prefix=/usr/local
make -j8
make install
```

最小可运行配置：

```m4
# configure.ac
AC_INIT([demo-autotools], [1.0.0], [dev@example.com])
AM_INIT_AUTOMAKE([foreign])
AC_PROG_CXX
AC_CONFIG_FILES([Makefile])
AC_OUTPUT
```

```makefile
# Makefile.am
bin_PROGRAMS = demo-autotools
demo_autotools_SOURCES = main.cpp
```

```cpp
// main.cpp
#include <iostream>

int main() {
    std::cout << "hello from autotools" << std::endl;
    return 0;
}
```

## CMake 的革命性改进

### 1）跨平台统一描述

一份 `CMakeLists.txt` 可以生成多平台工程。

### 2）源码外构建（Out-of-Source Build）

```bash
cmake -S . -B out/build-debug -G Ninja -DCMAKE_BUILD_TYPE=Debug
cmake --build out/build-debug
./out/build-debug/calc
```

### 3）目标导向（Target-based）

```cmake
cmake_minimum_required(VERSION 3.28)
project(target_demo LANGUAGES CXX)

add_library(math_utils STATIC math_utils.cpp)
target_include_directories(math_utils PUBLIC ${CMAKE_CURRENT_SOURCE_DIR})
target_compile_features(math_utils PUBLIC cxx_std_17)

add_executable(calc main.cpp)
target_link_libraries(calc PRIVATE math_utils)
```

## 构建系统对比表

| 维度 | Make | Autotools | CMake | Meson | Bazel |
|---|---|---|---|---|---|
| 定位 | 构建执行器 | Unix 构建套件 | 元构建系统 | 元构建系统 | 构建与测试平台 |
| 跨平台能力 | 弱 | 中（偏 Unix） | 强 | 强 | 强 |
| 学习成本 | 中 | 高 | 中 | 中 | 高 |
| IDE 集成 | 弱 | 弱 | 强 | 中 | 中 |
| 典型场景 | 小型本地项目 | 传统 GNU 项目 | 现代 C/C++ 跨平台项目 | 中小型现代项目 | 超大规模仓库 |
| 适配 KrKr2 | 不推荐单独使用 | 不推荐 | 推荐 | 备选 | 暂不优先 |

## 为什么 KrKr2 选择 CMake

KrKr2 目标平台是 Windows、Linux、macOS、Android。
项目还包含核心、插件、平台层多个目标。

选择 CMake 的关键原因：
1. 一套配置覆盖多平台。
2. 与 vcpkg 和 Ninja 搭配成熟。
3. target-based 模型适合大型模块化项目。
4. CMake Presets 便于本地和 CI 统一。

## 现代 CMake 与传统 CMake

| 主题 | 传统 CMake（2.x 思路） | 现代 CMake（3.x 思路） |
|---|---|---|
| 选项设置 | `set(CMAKE_CXX_FLAGS "-std=c++11 -Wall")` | `target_compile_options()` |
| 头文件目录 | `include_directories()` | `target_include_directories()` |
| 宏定义 | `add_definitions()` | `target_compile_definitions()` |
| 依赖传播 | 人工维护 | `PUBLIC/PRIVATE/INTERFACE` |

传统写法示例：

```cmake
cmake_minimum_required(VERSION 2.8)
project(old_style)

set(CMAKE_CXX_FLAGS "${CMAKE_CXX_FLAGS} -std=c++11 -Wall")
include_directories(${CMAKE_SOURCE_DIR}/include)
add_definitions(-DUSE_LOG)

add_executable(app main.cpp util.cpp)
```

现代写法示例：

```cmake
cmake_minimum_required(VERSION 3.28)
project(new_style LANGUAGES CXX)

add_library(util STATIC util.cpp)
target_include_directories(util PUBLIC ${CMAKE_CURRENT_SOURCE_DIR}/include)
target_compile_features(util PUBLIC cxx_std_17)
target_compile_definitions(util PRIVATE USE_LOG)
target_compile_options(util PRIVATE -Wall -Wextra)

add_executable(app main.cpp)
target_link_libraries(app PRIVATE util)
```

## 本节小结

- 手工编译在大型项目中维护成本极高。
- Makefile 提供自动化和增量构建，但跨平台能力有限。
- Autotools 流程成熟，但学习曲线陡。
- CMake 的跨平台生成和目标导向是现代主流方案。

## 练习题与答案

### 题目 1：为什么头文件改动会导致大量重编译？

<details>
<summary>查看答案</summary>

因为头文件会被预处理展开到所有包含它的 `.cpp` 中。
头文件内容变化意味着多个编译单元都失效，
因此必须重编译这些目标文件。

</details>

### 题目 2：Makefile 中 target/dependencies/recipe 各表示什么？

<details>
<summary>查看答案</summary>

- target：要生成的目标文件或伪目标。
- dependencies：生成 target 依赖的输入。
- recipe：真正执行的命令。

</details>

### 题目 3：CMake 为什么叫“元构建系统”？

<details>
<summary>查看答案</summary>

因为它本身不直接编译源码，
而是根据 `CMakeLists.txt` 生成底层构建系统文件
（如 Ninja、Makefile、Visual Studio 工程），
再由这些系统调用编译器完成构建。

</details>
