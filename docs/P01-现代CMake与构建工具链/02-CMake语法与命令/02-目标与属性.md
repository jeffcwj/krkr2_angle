## 正文内容

### 1. 为什么要“以目标为中心”

旧式 CMake 常把 include、宏、链接信息写成全局命令。
小项目还能维持，但在多模块工程中容易失控。
Modern CMake 提倡：把配置挂到目标，谁需要谁声明。

```cmake
# 示例 1：目标化最小配置
add_executable(app_demo src/main.cpp)
target_include_directories(app_demo PRIVATE ${CMAKE_CURRENT_SOURCE_DIR}/include)
target_compile_definitions(app_demo PRIVATE APP_DEMO_USE_LOG=1)
target_compile_features(app_demo PRIVATE cxx_std_17)
```

这类写法的优势是依赖边界清晰、可维护性高。

### 2. `add_executable()` 完整语法

```cmake
add_executable(<name> [WIN32] [MACOSX_BUNDLE] [EXCLUDE_FROM_ALL] <sources>)
```

参数说明：
- `<name>`：目标名。
- `WIN32`：Windows 图形程序模式。
- `MACOSX_BUNDLE`：macOS 应用包模式。
- `EXCLUDE_FROM_ALL`：默认不参与 all 构建。

```cmake
# 示例 2：可执行程序
cmake_minimum_required(VERSION 3.28)
project(demo_exec LANGUAGES CXX)
add_executable(hello_app src/main.cpp)
target_compile_features(hello_app PRIVATE cxx_std_17)
```

```cpp
// src/main.cpp
#include <iostream>
int main() {
    std::cout << "hello_app 运行成功" << std::endl;
    return 0;
}
```

### 3. `add_library()` 完整语法与四种库类型

```cmake
add_library(<name> [STATIC | SHARED | MODULE | INTERFACE] [EXCLUDE_FROM_ALL] <sources>)
```

#### 3.1 STATIC

- 产物：`.lib` 或 `.a`
- 场景：内部复用、部署简单

```cmake
# 示例 3：静态库
add_library(math_static STATIC src/math.cpp)
target_include_directories(math_static PUBLIC ${CMAKE_CURRENT_SOURCE_DIR}/include)
```

#### 3.2 SHARED

- 产物：`.dll/.so/.dylib`
- 场景：动态替换实现、共享库

```cmake
# 示例 4：动态库
add_library(math_shared SHARED src/math.cpp)
target_include_directories(math_shared PUBLIC ${CMAKE_CURRENT_SOURCE_DIR}/include)
```

#### 3.3 MODULE

- 产物：模块库
- 场景：插件体系，运行时加载

```cmake
# 示例 5：模块库
add_library(my_plugin MODULE plugin/plugin_entry.cpp)
target_compile_features(my_plugin PRIVATE cxx_std_17)
```

#### 3.4 INTERFACE

- 无二进制产物
- 场景：头文件库、聚合依赖

```cmake
# 示例 6：接口库
add_library(str_utils INTERFACE)
target_include_directories(str_utils INTERFACE ${CMAKE_CURRENT_SOURCE_DIR}/include)
target_compile_features(str_utils INTERFACE cxx_std_17)
```

### 4. `target_include_directories()` 详解

```cmake
target_include_directories(<target> [SYSTEM] [AFTER|BEFORE]
    <INTERFACE|PUBLIC|PRIVATE> dirs
)
```

可见性规则：
- `PRIVATE`：只给当前目标
- `PUBLIC`：当前 + 下游
- `INTERFACE`：只给下游

```cmake
# 示例 7：include 传播
add_library(net_core STATIC src/net_core.cpp)
target_include_directories(net_core
    PRIVATE ${CMAKE_CURRENT_SOURCE_DIR}/internal
    PUBLIC  ${CMAKE_CURRENT_SOURCE_DIR}/include
)
add_executable(client src/main.cpp)
target_link_libraries(client PRIVATE net_core)
```

### 5. `target_compile_definitions()` 详解

```cmake
target_compile_definitions(<target>
    <INTERFACE|PUBLIC|PRIVATE> defs
)
```

常见用途：功能开关、版本号、平台宏。

```cmake
# 示例 8：宏传播
add_library(config_flags INTERFACE)
target_compile_definitions(config_flags INTERFACE APP_VERSION=3 APP_FEATURE_X=1)
add_executable(show_config src/main.cpp)
target_link_libraries(show_config PRIVATE config_flags)
```

```cpp
// src/main.cpp
#include <iostream>
int main() {
    std::cout << "APP_VERSION=" << APP_VERSION << std::endl;
#if APP_FEATURE_X
    std::cout << "功能X开启" << std::endl;
#endif
    return 0;
}
```

### 6. `target_compile_features()` 与 C++ 标准

推荐写法：

```cmake
target_compile_features(my_target PUBLIC cxx_std_17)
```

需要固定行为时可用属性：

```cmake
set_target_properties(my_target PROPERTIES
    CXX_STANDARD 20
    CXX_STANDARD_REQUIRED ON
    CXX_EXTENSIONS OFF
)
```

### 7. `target_compile_options()` 与告警策略

```cmake
# 示例 9：跨编译器告警
add_executable(warn_demo src/main.cpp)
if(MSVC)
    target_compile_options(warn_demo PRIVATE /W4 /permissive-)
else()
    target_compile_options(warn_demo PRIVATE -Wall -Wextra -Wpedantic)
endif()
```

### 8. PUBLIC / PRIVATE / INTERFACE 传播图解

依赖关系：

```text
[App] ---> [LibA] ---> [LibB]
```

情况 A：

```cmake
target_link_libraries(LibA PUBLIC LibB)
target_link_libraries(App PRIVATE LibA)
```

App 会继承 LibB 的使用需求。

情况 B：

```cmake
target_link_libraries(LibA PRIVATE LibB)
target_link_libraries(App PRIVATE LibA)
```

App 不会继承 LibB 的使用需求。

| 关键字 | 当前目标可见 | 依赖者可见 |
| --- | --- | --- |
| `PRIVATE` | 是 | 否 |
| `PUBLIC` | 是 | 是 |
| `INTERFACE` | 否 | 是 |

### 9. `get_target_property()` 与 `set_target_properties()`

```cmake
# 示例 10：属性读写
add_executable(prop_app src/main.cpp)
set_target_properties(prop_app PROPERTIES
    OUTPUT_NAME "prop_app_final"
    CXX_STANDARD 17
    CXX_STANDARD_REQUIRED ON
)
get_target_property(_std prop_app CXX_STANDARD)
message(STATUS "prop_app CXX_STANDARD = ${_std}")
```

### 10. KrKr2 项目实例说明

对照 `krkr2/CMakeLists.txt`：
- 第 47/51/53/79 行：按 Android、Linux、Windows、macOS 选择主目标形态。
- 第 86 行：`target_include_directories(${PROJECT_NAME} PUBLIC ${KRKR2CORE_PATH}/environ/cocos2d)`。
- 第 95-97 行：`target_link_libraries(${PROJECT_NAME} PUBLIC krkr2plugin krkr2core)`。

对照 `krkr2/cpp/core/CMakeLists.txt`：
- 第 6 行：`add_library(${PROJECT_NAME} INTERFACE)`。
- 第 8-16 与 18-28 行：引入并聚合 9 个核心子模块。

- 第 44-45 行：非 Apple 平台导出 OpenMP 配置。

这说明 KrKr2 已经采用目标化 + 接口聚合模式。

## 动手实践

目标关系：

```text
app_demo -> engine_api(INTERFACE) -> engine_impl(STATIC)
```

### 步骤 1：目录

```text
practice_target/
├─ CMakeLists.txt
├─ include/engine_api.hpp
└─ src/
   ├─ engine_impl.cpp
   └─ main.cpp
```

### 步骤 2：代码

```cpp
// include/engine_api.hpp
#pragma once
int engine_sum(int a, int b);
```

```cpp
// src/engine_impl.cpp
#include "engine_api.hpp"
int engine_sum(int a, int b) { return a + b; }
```

```cpp
// src/main.cpp
#include <iostream>
#include "engine_api.hpp"
int main() {
    std::cout << "engine_sum(10, 32) = " << engine_sum(10, 32) << std::endl;
    return 0;
}
```

### 步骤 3：CMakeLists

```cmake
cmake_minimum_required(VERSION 3.28)
project(practice_target LANGUAGES CXX)

add_library(engine_impl STATIC src/engine_impl.cpp)
target_include_directories(engine_impl PUBLIC ${CMAKE_CURRENT_SOURCE_DIR}/include)
target_compile_features(engine_impl PUBLIC cxx_std_17)

add_library(engine_api INTERFACE)
target_link_libraries(engine_api INTERFACE engine_impl)
target_compile_definitions(engine_api INTERFACE ENGINE_API_VERSION=1)

add_executable(app_demo src/main.cpp)
target_link_libraries(app_demo PRIVATE engine_api)
```

### 步骤 4：构建

Windows：

```bash
cmake -S . -B build -G Ninja
cmake --build build
./build/app_demo.exe
```

Linux/macOS：

```bash
cmake -S . -B build -G Ninja
cmake --build build
./build/app_demo
```

运行后应看到：`engine_sum(10, 32) = 42`。

## 本节小结

- Target 是 Modern CMake 的核心配置单位。
- 四类库对应不同构建与发布需求。
- `target_include_directories()` 与 `target_compile_definitions()` 决定传播边界。
- `target_compile_features()` 和属性配置共同控制 C++ 标准。
- `PUBLIC / PRIVATE / INTERFACE` 是依赖传播的核心。
