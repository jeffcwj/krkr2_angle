## 正文内容

在传统的 CMake 脚本中，开发者往往通过设置大量的全局变量（如 `set(INCLUDE_DIRS ...)`）来控制编译过程。这种方式在大型项目中会导致严重的依赖混乱。

**Modern CMake（现代 CMake）** 引入了“目标（Target）”的概念。它将每一个库或可执行文件看作一个对象，这个对象拥有自己的属性（头文件路径、链接依赖、宏定义等）。

### 1. 什么是目标（Target）

简单来说，你在 `add_executable` 或 `add_library` 中定义的名称就是一个目标。

```cmake
add_executable(my_app main.cpp) 
# "my_app" 就是一个目标
```

与其设置全局变量，不如直接对目标进行配置：
```cmake
# ❌ 旧方式
include_directories(include/)
link_libraries(m)

# ✅ 现代方式
target_include_directories(my_app PRIVATE include/)
target_link_libraries(my_app PRIVATE m)
```

### 2. 目标类型详解

在 CMake 中，库主要分为以下几种：

#### 静态库 (STATIC Library)
```cmake
add_library(my_static_lib STATIC source.cpp)
# Windows 下生成 .lib，Linux/Android/macOS 下生成 .a
```

#### 动态库 (SHARED Library)
```cmake
add_library(my_shared_lib SHARED source.cpp)
# Windows 下生成 .dll，Linux/Android 下生成 .so，macOS 下生成 .dylib
```

#### 接口库 (INTERFACE Library)
接口库不直接生成二进制文件。它通常用于：
- **纯头文件库**（Header-only library）
- **聚合目标**：将一堆子模块聚合到一个名称下。

在 KrKr2 项目的 `cpp/core/CMakeLists.txt` 中，就用了这种方式聚合了 9 个子模块：
```cmake
# KrKr2 项目源码参考
add_library(${PROJECT_NAME} INTERFACE)
add_subdirectory(tjs2)
add_subdirectory(base)
# ... (其他模块)
target_link_libraries(${PROJECT_NAME} INTERFACE tjs2 core_base_module)
```

#### 对象库 (OBJECT Library)
```cmake
add_library(my_objects OBJECT source.cpp)
# 仅编译出 .o/.obj，不链接成库，可以被其他目标引用
```

### 3. 目标属性三关键字：PUBLIC / PRIVATE / INTERFACE

这是 CMake 中最重要，也是最容易混淆的概念。它们决定了属性（如头文件、依赖等）如何**传播**。

| 关键字 | 对当前目标有效 | 会传递给依赖我的目标 |
| --- | --- | --- |
| **PRIVATE** | 是 | 否 |
| **INTERFACE** | 否 | 是 |
| **PUBLIC** | 是 | 是 |

#### 场景模拟
假设你有三个库：A, B, C。
- 如果 A `PRIVATE` 链接 B，那么 C 链接 A 时，C 不会看到 B。
- 如果 A `PUBLIC` 链接 B，那么 C 链接 A 时，C 也会自动链接 B。
- 如果 A `INTERFACE` 链接 B，那么 A 自己不链接 B，但依赖 A 的 C 会链接 B。

#### 示例
```cmake
# 仅我需要引用 internal.h
target_include_directories(my_lib PRIVATE internal_headers/)

# 使用我的用户也需要这些头文件
target_include_directories(my_lib PUBLIC public_headers/)

# 纯头文件库的头文件路径
target_include_directories(header_only_lib INTERFACE include/)
```

### 4. target_sources — 追加源文件

在早期的 CMake 中，我们常用 `set(SOURCES ...)` 定义列表。现代 CMake 推荐使用 `target_sources()` 直接向目标追加源文件。

在 KrKr2 插件系统 (`cpp/plugins/CMakeLists.txt`) 中，你会看到类似的用法：

```cmake
# 定义一个名为 krkr2plugin 的静态库目标
add_library(${PROJECT_NAME} STATIC)

# 向目标追加源文件
target_sources(${PROJECT_NAME} PUBLIC 
    scriptsEx.cpp 
    win32dialog.cpp 
    csvParser.cpp
)

# 向目标链接依赖
target_link_libraries(${PROJECT_NAME} PUBLIC krkr2core)
```

### 5. 链接传播 (Linking Propagation)

假设你有如下依赖关系：

```text
  [ App ] ----> [ LibA ] ----> [ LibB ]
```

如果你在 CMake 中配置如下：
```cmake
target_link_libraries(LibA PUBLIC LibB)
target_link_libraries(App PRIVATE LibA)
```
因为 LibA 是 `PUBLIC` 链接 LibB，所以 App 在链接 LibA 时，会自动把 LibB 也链接进来。

如果 LibA 是 `PRIVATE` 链接 LibB，App 就不会自动链接 LibB，这通常用于不希望用户（App）知道 LibA 内部依赖了哪些库的场景。

### 6. 对照项目源码

让我们看一段 KrKr2 项目中实际的配置：

**文件：cpp/core/CMakeLists.txt**
```cmake
# 这里将所有子模块聚合到一个接口库中
add_library(${PROJECT_NAME} INTERFACE)

# 递归进入子目录
add_subdirectory(tjs2)
add_subdirectory(base)
# ... (其他模块)

# 将各个子模块链接到主接口库
target_link_libraries(${PROJECT_NAME} INTERFACE 
    tjs2 
    core_base_module
)

# 向接口库添加全局宏定义
target_compile_definitions(${PROJECT_NAME} INTERFACE -DTJS_TEXT_OUT_CRLF)
```

## 本节小结

1. **Modern CMake** 以目标（Target）为中心，抛弃全局变量。
2. **目标类型** 包括：可执行文件、静态库、动态库、接口库（INTERFACE）和对象库。
3. **属性传播** 是核心：
   - `PRIVATE`: 仅限内部使用。
   - `INTERFACE`: 仅限外部用户使用。
   - `PUBLIC`: 内部使用 + 外部用户共享。
4. **`target_sources`** 和 **`target_link_libraries`** 是最常用的配置命令。

## 练习题与答案

### 题目 1：
假设你正在开发一个名为 `FastMath` 的库，它由一个 `FastMath.h` 和一个 `FastMath.cpp` 组成。
你希望生成一个静态库。同时，你需要向外暴露 `include/` 目录给用户使用。请写出对应的 CMake 代码。

<details><summary>查看答案</summary>
```cmake
add_library(FastMath STATIC FastMath.cpp)

# 设置包含目录，由于用户也需要包含头文件，所以使用 PUBLIC
target_include_directories(FastMath PUBLIC include/)
```
</details>

### 题目 2：
以下两种写法有什么区别？

写法 A: `add_library(my_lib STATIC a.cpp b.cpp)`
写法 B: `add_library(my_lib STATIC) \n target_sources(my_lib PRIVATE a.cpp b.cpp)`

<details><summary>查看答案</summary>
本质上效果相同。
但写法 B 更符合 Modern CMake 风格，它允许你在定义目标之后，根据不同平台或条件灵活地追加源文件（例如：在 Windows 下追加 `win_impl.cpp`，在 Linux 下追加 `linux_impl.cpp`）。
</details>

### 题目 3：
如果 `LibA` 以 `PRIVATE` 方式链接了 `LibB`，那么直接依赖 `LibA` 的 `App` 能否直接调用 `LibB` 中的函数？

<details><summary>查看答案</summary>
不能直接调用（App 在编译时不会自动获取 LibB 的头文件路径，链接时也不会自动链接 LibB）。
除非 App 也显式地 `target_link_libraries(App PRIVATE LibB)`。这就是 `PRIVATE` 的封装作用：隐藏内部实现细节。
</details>
