# 03-生成器表达式

> **所属模块：** P01-现代CMake与构建工具链

> **预计阅读时间：** 20 分钟

## 本节目标

读完本节后，你将能够：
1. 理解为什么需要生成器表达式以及它们与普通变量的区别
2. 掌握常用的条件判断、配置判断和平台判断表达式
3. 能够读懂并编写复杂的条件编译选项

## 正文内容

在 CMake 中，构建过程分为两个阶段：
1. **配置阶段 (Configure Stage)**：读取 `CMakeLists.txt`，处理变量、`if()` 判断等。
2. **生成阶段 (Generate Stage)**：根据配置阶段的结果，生成具体的构建文件（如 VS 工程、Ninja 文件等）。

有些信息在“配置阶段”是拿不到的。最典型的例子就是 **构建类型（Debug 或 Release）**。在 Visual Studio 这种多配置生成器中，你在配置时并不确定用户最后会选择哪种模式进行编译。

这时候，**生成器表达式 (Generator Expression)** 就派上用场了。

### 1. 基本语法

生成器表达式的格式为 `$<...>`。它们在生成阶段才会被求值。

### 2. 条件判断表达式

最常用的格式是 `$<condition:true_string>`。如果条件为真，则结果为 `true_string`；否则为空字符串。

#### 示例：根据构建类型添加宏
```cmake
add_compile_definitions(
    $<$<CONFIG:Debug>:DEBUG_MODE>
)
```
这段代码的意思是：如果当前是 Debug 配置，则添加 `DEBUG_MODE` 宏定义。

### 3. 常用表达式详解

#### 1) 配置判断：`$<CONFIG:config>`
检查当前的构建类型（Debug, Release 等）。

#### 2) 逻辑判断：`$<BOOL:value>`, `$<NOT:cond>`, `$<AND:conds>`, `$<OR:conds>`
用于组合复杂的逻辑。

#### 3) 条件分支：`$<IF:condition,true_val,false_val>`
类似于 C++ 的三目运算符 `? :`。

#### 4) 平台判断：`$<PLATFORM_ID:platform>`
检查操作系统。常用的 ID 有：`Windows`, `Linux`, `Darwin` (macOS), `Android`。

#### 5) 语言判断：`$<COMPILE_LANGUAGE:lang>`
检查当前正在编译的文件语言（如 `CXX` 代表 C++）。这在混合编译的项目中非常有用。

### 4. 跨平台实战：KrKr2 中的应用

在 KrKr2 的根目录 `CMakeLists.txt` 中，大量使用了生成器表达式来处理不同平台的编译器选项。

#### 场景 A：条件宏定义
```cmake
add_compile_definitions(
    # 如果是 Debug 模式，定义 DEBUG 和 _DEBUG
    $<$<CONFIG:Debug>:DEBUG>
    $<$<CONFIG:Debug>:_DEBUG>
    # 如果不是 Debug 模式，定义 NDEBUG
    $<$<NOT:$<CONFIG:Debug>>:NDEBUG>
)
```

#### 场景 B：MSVC 编译器特有选项
```cmake
# 仅当使用 C++ 编译且编译器是 MSVC 时，添加 /utf-8 等选项
add_compile_options(
    "$<$<AND:$<COMPILE_LANGUAGE:CXX>,$<CXX_COMPILER_ID:MSVC>>:/EHsc;/MP;/utf-8>"
)
```

#### 场景 C：Android 条件链接
```cmake
# 如果是 Android 平台，则链接特定的库
target_link_libraries(my_target PRIVATE
    $<$<BOOL:${ANDROID}>:$<LINK_LIBRARY:WHOLE_ARCHIVE,cocos2dx::cpp_android_spec>>
)

### 5. 何时使用：`if()` vs 生成器表达式

这是新手最容易混淆的地方。

- **使用 `if()` 的场景**：
  - 在配置阶段就能确定的逻辑（例如：检查 `set(MY_OPTION ON)`）。
  - 需要执行复杂逻辑（如 `if(...) else(...) endif`）。
  - 需要在配置时就判断某些目录是否存在。

- **使用生成器表达式的场景**：
  - 需要根据 **构建配置 (Debug/Release)** 进行判断。
  - 需要获取 **目标的最终输出路径** (`$<TARGET_FILE:target_name>`)。
  - 需要在 `target_...` 类命令中编写简洁的条件。

### 6. 动手实践：构建一个 Debug 下带调试信息的库

```cmake
cmake_minimum_required(VERSION 3.10)
project(GenExpDemo LANGUAGES CXX)

add_library(MathLib STATIC Math.cpp)

# 在 Debug 下开启调试详细日志宏
target_compile_definitions(MathLib PRIVATE 
    $<$<CONFIG:Debug>:DEBUG_VERBOSE_LOG>
)

# 仅在 Windows 下启用 /utf-8 选项
target_compile_options(MathLib PRIVATE 
    $<$<PLATFORM_ID:Windows>:/utf-8>
)

# 在 Linux 下链接线程库
target_link_libraries(MathLib PRIVATE
    $<$<PLATFORM_ID:Linux>:threads>
)
```

## 本节小结

1. **生成器表达式** 在生成阶段求值，是处理多配置（Debug/Release）的核心。
2. **`$<CONFIG:Debug>`** 和 **`$<PLATFORM_ID:Windows>`** 是最常用的判断方式。
3. **`$<$<condition>:value>`** 是最标准的用法。
4. **记住**：生成器表达式不能用于像 `set()` 或 `if()` 这种只在配置阶段运行的代码中。

## 练习题与答案

### 题目 1：
请使用生成器表达式，为目标 `MyTarget` 设置一个宏：仅当当前是 `Release` 配置且当前平台是 `Linux` 时，定义宏 `OPTIMIZED_FOR_LINUX`。

<details><summary>查看答案</summary>
```cmake
target_compile_definitions(MyTarget PRIVATE 
    $<$<AND:$<CONFIG:Release>,$<PLATFORM_ID:Linux>>:OPTIMIZED_FOR_LINUX>
)
```
</details>

### 题目 2：
以下写法在 Visual Studio 生成器下会有什么问题？为什么？

```cmake
if(CMAKE_BUILD_TYPE STREQUAL "Debug")
    add_definitions(-DDEBUG_MODE)
endif()
```

<details><summary>查看答案</summary>
在 Visual Studio（多配置生成器）下，`CMAKE_BUILD_TYPE` 变量在配置阶段通常是空的。
VS 的 Debug 或 Release 是在构建时通过 IDE 选定的，因此这个 `if` 判断会失效。
正确的做法是使用生成器表达式：
`add_compile_definitions($<$<CONFIG:Debug>:DEBUG_MODE>)`。
</details>

### 题目 3：
如何使用生成器表达式，使得 `MyLib` 库在 C++ 编译时添加 `-Wall` 选项，而在 C 编译时不添加？

<details><summary>查看答案</summary>
```cmake
target_compile_options(MyLib PRIVATE
    $<$<COMPILE_LANGUAGE:CXX>:-Wall>
)
```
</details>
