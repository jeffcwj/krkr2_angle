## 本节目标

读完本节后，你将能够：
1. 把单文件 CMake 工程演进为多模块结构。
2. 编写根 CMakeLists 与子模块 CMakeLists。
3. 用 `target_link_libraries` 管理模块依赖传递。
4. 落地 `public/private headers` 组织策略。
5. 使用 `install()` 让工程可发布、可复用。

## 从单文件到多模块的演进过程

单文件工程适合教学入门，但不适合长期维护。随着功能增长，常见问题是：

- 目标之间职责不清，任何改动都影响全局。
- include 路径变多后，依赖来源难追踪。
- 没有模块边界，无法单独复用功能。

推荐演进路线：

```mermaid
graph LR
    A[单目标 main] --> B[拆成 math render asset]
    B --> C[引入 INTERFACE 聚合层 engine]
    C --> D[接入 plugin 与 app]
    D --> E[补齐 install 导出]
```

这条路线与 KrKr2 的真实组织方式一致：根层调度，核心层聚合，子模块各司其职。

## 完整的多模块目录结构设计

```text
MiniEngineProject/
├─ CMakeLists.txt
├─ engine/
│  ├─ CMakeLists.txt
│  ├─ math/
│  │  ├─ CMakeLists.txt
│  │  ├─ include/math/math.h
│  │  └─ src/math.cpp
│  ├─ render/
│  │  ├─ CMakeLists.txt
│  │  ├─ include/render/render.h
│  │  └─ src/render.cpp
│  └─ asset/
│     ├─ CMakeLists.txt
│     ├─ include/asset/asset.h
│     └─ src/asset.cpp
├─ plugins/
│  ├─ CMakeLists.txt
│  ├─ include/plugins/plugin.h
│  └─ src/plugin.cpp
└─ app/
   ├─ CMakeLists.txt
   └─ main.cpp
```

## 示例代码 1：根 CMakeLists.txt

```cmake
cmake_minimum_required(VERSION 3.28)
project(MiniEngineProject LANGUAGES CXX)

set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

add_subdirectory(engine)
add_subdirectory(plugins)
add_subdirectory(app)

install(EXPORT MiniEngineTargets
    FILE MiniEngineTargets.cmake
    NAMESPACE MiniEngine::
    DESTINATION lib/cmake/MiniEngine
)
```

## 示例代码 2：聚合模块 engine

```cmake
# engine/CMakeLists.txt
add_library(engine INTERFACE)

add_subdirectory(math)
add_subdirectory(render)
add_subdirectory(asset)

target_link_libraries(engine INTERFACE
    engine_math
    engine_render
    engine_asset
)

install(TARGETS engine EXPORT MiniEngineTargets)
```

这里 `engine` 不编译源码，只做依赖传递，这与 KrKr2 的 `krkr2core` 角色一致。

## 示例代码 3：math 子模块

```cmake
# engine/math/CMakeLists.txt
add_library(engine_math STATIC src/math.cpp)

target_include_directories(engine_math
    PUBLIC
        $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
        $<INSTALL_INTERFACE:include>
)

install(TARGETS engine_math EXPORT MiniEngineTargets
    ARCHIVE DESTINATION lib
    LIBRARY DESTINATION lib
    RUNTIME DESTINATION bin
)

install(DIRECTORY include/ DESTINATION include)
```

```cpp
// engine/math/include/math/math.h
#pragma once
namespace math {
    int square(int x);
}
```

```cpp
// engine/math/src/math.cpp
#include "math/math.h"
namespace math {
    int square(int x) { return x * x; }
}
```

## 示例代码 4：render 子模块（public/private headers）

```cmake
# engine/render/CMakeLists.txt
add_library(engine_render STATIC src/render.cpp)

target_include_directories(engine_render
    PUBLIC
        $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
        $<INSTALL_INTERFACE:include>
    PRIVATE
        ${CMAKE_CURRENT_SOURCE_DIR}/src
)

target_link_libraries(engine_render PRIVATE engine_math)

install(TARGETS engine_render EXPORT MiniEngineTargets
    ARCHIVE DESTINATION lib
    LIBRARY DESTINATION lib
    RUNTIME DESTINATION bin
)

install(DIRECTORY include/ DESTINATION include)
```

```cpp
// engine/render/include/render/render.h
#pragma once
namespace render {
    void draw_frame();
}
```

```cpp
// engine/render/src/render.cpp
#include "render/render.h"
#include <iostream>
namespace render {
    void draw_frame() {
        std::cout << "[Render] draw" << std::endl;
    }
}
```

## 示例代码 5：asset + plugins + app

```cmake
# engine/asset/CMakeLists.txt
add_library(engine_asset STATIC src/asset.cpp)
target_include_directories(engine_asset PUBLIC
    $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
    $<INSTALL_INTERFACE:include>
)
install(TARGETS engine_asset EXPORT MiniEngineTargets
    ARCHIVE DESTINATION lib
    LIBRARY DESTINATION lib
    RUNTIME DESTINATION bin
)
install(DIRECTORY include/ DESTINATION include)
```

```cpp
// engine/asset/include/asset/asset.h
#pragma once
#include <string>
namespace asset {
    std::string resolve_asset_path(const std::string& name);
}
```

```cpp
// engine/asset/src/asset.cpp
#include "asset/asset.h"
namespace asset {
    std::string resolve_asset_path(const std::string& name) {
        return std::string("assets/") + name;
    }
}
```

```cmake
# plugins/CMakeLists.txt
add_library(mini_plugin STATIC src/plugin.cpp)
target_include_directories(mini_plugin PUBLIC
    $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
    $<INSTALL_INTERFACE:include>
)
target_link_libraries(mini_plugin PUBLIC engine)
install(TARGETS mini_plugin EXPORT MiniEngineTargets
    ARCHIVE DESTINATION lib
    LIBRARY DESTINATION lib
    RUNTIME DESTINATION bin
)
install(DIRECTORY include/ DESTINATION include)
```

```cpp
// plugins/src/plugin.cpp
#include "plugins/plugin.h"
#include "asset/asset.h"
#include "math/math.h"
#include <iostream>
namespace plugins {
    void run_plugin_logic() {
        std::cout << math::square(10) << std::endl;
        std::cout << asset::resolve_asset_path("hero.png") << std::endl;
    }
}
```

```cmake
# app/CMakeLists.txt
add_executable(main_app main.cpp)
target_link_libraries(main_app PRIVATE mini_plugin engine)
install(TARGETS main_app RUNTIME DESTINATION bin)
```

```cpp
// app/main.cpp
#include "plugins/plugin.h"
#include "render/render.h"
int main() {
    render::draw_frame();
    plugins::run_plugin_logic();
    return 0;
}
```

## 模块间依赖管理（target_link_libraries 传递）

依赖关系如下：

`main_app -> mini_plugin -> engine -> engine_math/engine_render/engine_asset`

最常用的三条规则：

- `PRIVATE`：仅当前目标实现需要。
- `PUBLIC`：当前目标与下游都需要。
- `INTERFACE`：仅做转发，不参与编译。

落地时可直接记忆：`render -> math` 用 `PRIVATE`，`plugin -> engine` 用 `PUBLIC`，`engine` 聚合子模块用 `INTERFACE`。

## 头文件组织策略

- `include/<module>/`：对外 API（Public）。
- `src/`：内部实现与私有头（Private）。

通过 `target_include_directories(目标 PUBLIC 公共目录 PRIVATE 私有目录)` 显式声明边界，可以减少误依赖并降低后续重构成本。

## 安装规则（install 命令）

多模块工程至少包含三类安装：目标安装、头文件安装、导出安装。对应命令分别是 `install(TARGETS main_app)`、`install(DIRECTORY include/ DESTINATION include)`、`install(EXPORT MiniEngineTargets FILE MiniEngineTargets.cmake)`。三者都缺一不可。

## 动手实践

### Step 1：创建目录与文件
按“目录结构设计”一次性建好目录与空文件。

### Step 2：先跑通最小链路
先实现 `engine_math + app`，确认根配置与 include 路径正确。

### Step 3：补齐其余模块
实现 `engine_render`、`engine_asset`、`mini_plugin`，并接入 `engine` 聚合层。

### Step 4：补齐安装规则
每个模块都添加 `install(TARGETS 目标名)` 与 `install(DIRECTORY include/ DESTINATION include)`，根层添加 `install(EXPORT MiniEngineTargets FILE MiniEngineTargets.cmake)`。

### Step 5：构建验证（四平台）

- Windows：`cmake -S . -B build -G "Visual Studio 17 2022" -A x64 && cmake --build build --config Release`
- Linux：`cmake -S . -B build -G Ninja && cmake --build build`
- macOS：`cmake -S . -B build -G Ninja && cmake --build build`
- Android：`cmake -S . -B build-android -DCMAKE_TOOLCHAIN_FILE=$ANDROID_NDK/build/cmake/android.toolchain.cmake -DANDROID_ABI=arm64-v8a -DANDROID_PLATFORM=android-21 && cmake --build build-android`

### Step 6：安装验证

```bash
cmake --install build --prefix ./install
```

检查 `install/bin/main_app`、`install/include/math/math.h`、`install/lib/cmake/MiniEngine/MiniEngineTargets.cmake` 是否存在。

## 对照项目源码（KrKr2 实际多模块结构）

### 根层组织方式
在 `krkr2/CMakeLists.txt` 中，使用 `KRKR2CORE_PATH`、`KRKR2PLUGIN_PATH` 管理模块路径，随后通过 `add_subdirectory` 引入 core/plugins，并在最终目标统一链接 `krkr2plugin` 与 `krkr2core`。
这和本节示例中的“根层调度 + 聚合目标链接”是一致范式。

### core 层聚合方式

在 `krkr2/cpp/core/CMakeLists.txt` 中：

- `add_library(krkr2core INTERFACE)`
- `add_subdirectory(tjs2/base/environ/extension/plugin/movie/sound/visual/utils)`
- `target_link_libraries(krkr2core INTERFACE tjs2 core_base_module core_visual_module)`

说明 `krkr2core` 是能力聚合层，而不是实现层。

### 平台差异分层

KrKr2 在根层按平台建立入口目标，在 core 层补充平台依赖（如 Android 系统库链接），可避免条件编译混乱。

## 本节小结

- 单体工程向多模块演进时，先拆职责，再做依赖聚合。
- `INTERFACE` 聚合目标可以显著简化上层链接关系。
- `PUBLIC/PRIVATE` 是 API 边界工具，不是语法装饰。
- KrKr2 的结构为多模块 CMake 提供了真实可借鉴模板。

## 练习题与答案

### 题目 1：依赖可见性判断

`plugin.h` 暴露了来自 `engine` 的类型，`mini_plugin` 链接 `engine` 时应使用 `PUBLIC` 还是 `PRIVATE`？

<details>
<summary>查看答案</summary>

应使用 `PUBLIC`。`plugin.h` 已暴露 engine 类型，消费者编译插件头文件时也需要该依赖；若用 `PRIVATE`，下游可能编译失败。

</details>

### 题目 2：新增 audio 子模块

请说明把 `engine/audio` 接入聚合层并可安装的最小步骤。

<details>
<summary>查看答案</summary>

1. 创建 `engine/audio/include/audio` 与 `engine/audio/src`。
2. 定义 `engine_audio` 并配置 `PUBLIC include`。
3. 添加 `install(TARGETS engine_audio)` 与 `install(DIRECTORY include/ DESTINATION include)`。
4. 在聚合层加入 `add_subdirectory(audio)` 和 `target_link_libraries(engine INTERFACE engine_audio)`。

</details>

### 题目 3：install(EXPORT) 的作用

为什么多模块库建议使用 `install(EXPORT MiniEngineTargets FILE MiniEngineTargets.cmake)`？

<details>
<summary>查看答案</summary>

它会导出可导入目标信息，让外部项目直接使用 `MiniEngine::xxx` 目标，不必手写 include 目录和链接顺序，从而降低集成复杂度并减少链接错误。

</details>
