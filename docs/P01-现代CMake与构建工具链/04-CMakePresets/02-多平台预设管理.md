# 02-多平台预设管理

在前一章中，你已经掌握了 `CMakePresets.json` 的基本语法。本章我们将以 KrKr2 项目真实的配置文件为例，深入探讨如何在大规模跨平台项目中组织和管理预设。

---

## KrKr2 的完整 CMakePresets.json 解读

KrKr2 模拟器支持 Windows、Linux、macOS 和 Android 四个平台。为了让构建过程在所有桌面上保持一致，我们编写了如下配置：

```json
{
  "version": 6,
  "cmakeMinimumRequired": { "major": 3, "minor": 28, "patch": 0 },
  "configurePresets": [
    {
      "name": "Windows Config",
      "generator": "Ninja",
      "hidden": true,
      "cacheVariables": {
        "WINDOWS": true,
        "VCPKG_TARGET_TRIPLET": "x64-windows-static-md",
        "CMAKE_EXPORT_COMPILE_COMMANDS": true
      },
      "condition": {
        "type": "equals",
        "lhs": "${hostSystemName}",
        "rhs": "Windows"
      },
      "architecture": { "value": "x64", "strategy": "external" },
      "vendor": {
        "microsoft.com/VisualStudioSettings/CMake/1.0": {
          "hostOS": ["Windows"]
        }
      }
    },
    {
      "name": "Windows Debug Config",
      "inherits": "Windows Config",
      "binaryDir": "${sourceDir}/out/windows/debug",
      "cacheVariables": { "CMAKE_BUILD_TYPE": "Debug" }
    },
    {
      "name": "Windows Release Config",
      "inherits": "Windows Config",
      "binaryDir": "${sourceDir}/out/windows/release",
      "cacheVariables": { "CMAKE_BUILD_TYPE": "Release" }
    },
    {
      "name": "Linux Config",
      "generator": "Ninja",
      "hidden": true,
      "cacheVariables": {
        "LINUX": true,
        "CMAKE_EXPORT_COMPILE_COMMANDS": true
      },
      "condition": {
        "type": "equals",
        "lhs": "${hostSystemName}",
        "rhs": "Linux"
      }
    },
    {
      "name": "Linux Debug Config",
      "inherits": "Linux Config",
      "binaryDir": "${sourceDir}/out/linux/debug",
      "cacheVariables": { "CMAKE_BUILD_TYPE": "Debug" }
    },
    {
      "name": "Linux Release Config",
      "inherits": "Linux Config",
      "binaryDir": "${sourceDir}/out/linux/release",
      "cacheVariables": { "CMAKE_BUILD_TYPE": "Release" }
    },
    {
      "name": "MacOS Config",
      "generator": "Ninja",
      "hidden": true,
      "cacheVariables": {
        "MACOS": true,
        "CMAKE_C_COMPILER": "clang",
        "CMAKE_CXX_COMPILER": "clang++",
        "CMAKE_EXPORT_COMPILE_COMMANDS": true
      },
      "condition": {
        "type": "equals",
        "lhs": "${hostSystemName}",
        "rhs": "Darwin"
      }
    },
    {
      "name": "MacOS Debug Config",
      "inherits": "MacOS Config",
      "binaryDir": "${sourceDir}/out/macos/debug",
      "cacheVariables": { "CMAKE_BUILD_TYPE": "Debug" }
    },
    {
      "name": "MacOS Release Config",
      "inherits": "MacOS Config",
      "binaryDir": "${sourceDir}/out/macos/release",
      "cacheVariables": { "CMAKE_BUILD_TYPE": "Release" }
    }
  ],
  "buildPresets": [
    { "name": "Windows Debug Build", "configurePreset": "Windows Debug Config" },
    { "name": "Windows Release Build", "configurePreset": "Windows Release Config" },
    { "name": "Linux Debug Build", "configurePreset": "Linux Debug Config" },
    { "name": "Linux Release Build", "configurePreset": "Linux Release Config" },
    { "name": "MacOS Debug Build", "configurePreset": "MacOS Debug Config" },
    { "name": "MacOS Release Build", "configurePreset": "MacOS Release Config" }
  ]
}
```

## 设计模式分析：隐藏基类 + 继承变体

在上面的 JSON 中，我们采用了一种非常经典的设计模式：**“1 个隐藏基类 + N 个公开变体”**。

1. **隐藏基类（如 `Windows Config`）**：
   - 设置 `hidden: true`，使其不会干扰用户的选择。
   - 包含该平台通用的设置：生成器（Ninja）、平台宏（`WINDOWS`）、包管理器参数（`VCPKG_TARGET_TRIPLET`）。
   - 使用 `condition` 确保该基类仅在正确的宿主系统上被激活。

2. **公开变体（如 `Windows Debug Config`）**：
   - 继承自基类。
   - 仅定义差异化参数：`CMAKE_BUILD_TYPE` 和 `binaryDir`（构建输出目录）。

这种模式的好处是：如果你需要修改 Windows 平台的通用参数，只需要改动一处，所有 Debug/Release 预设都会同步生效。

---

## 自定义平台变量的妙用

你可能注意到，我们在每个平台中定义了 `WINDOWS`、`LINUX` 或 `MACOS` 变量，而不是直接使用 CMake 标准的 `CMAKE_SYSTEM_NAME`。

**为什么这么做？**
1. **简化条件判断**：在 `CMakeLists.txt` 中，你可以直接写 `if(WINDOWS)`，这比写 `if(CMAKE_SYSTEM_NAME STREQUAL "Windows")` 要简洁得多。
2. **处理 Android 特例**：Android 构建通常由 Gradle 驱动，并通过 NDK 的工具链文件（如 `vcpkg_android.cmake`）注入参数。在复杂的混合构建环境中，标准的系统名称变量有时可能未按预期初始化，自定义变量能提供更可靠的判断依据。

---

## Android 为何不在 Presets 中？

这是一个非常关键的问题。既然 Presets 这么好用，为什么不把 Android 也放进去？

在 KrKr2 中，Android 的构建流程是：
**Gradle -> CMake -> vcpkg**

Android 的构建环境非常复杂，涉及到 NDK 路径、API Level、ABI（arm64-v8a, x86_64）等。这些参数通常由 Android Studio 的 Gradle 插件动态生成并传递给 CMake。

如果将 Android 强行写入 `CMakePresets.json`：
1. 难以同步 Gradle 中的路径配置。
2. 无法方便地处理 Android 特有的多 ABI 同时构建需求。
因此，我们维持 Android 的传统构建模式，而将桌面端（Win/Lin/Mac）交给 Presets 管理。

## 如何添加新平台预设

假设你现在要为 KrKr2 添加一个 **FreeBSD** 支持，你应该遵循以下步骤：

1. **添加隐藏基类**：
   ```json
   {
     "name": "FreeBSD Config",
     "generator": "Ninja",
     "hidden": true,
     "condition": {
       "type": "equals",
       "lhs": "${hostSystemName}",
       "rhs": "FreeBSD"
     },
     "cacheVariables": { "FREEBSD": true }
   }
   ```
2. **添加公开变体**：
   ```json
   {
     "name": "FreeBSD Debug Config",
     "inherits": "FreeBSD Config",
     "binaryDir": "${sourceDir}/out/freebsd/debug",
     "cacheVariables": { "CMAKE_BUILD_TYPE": "Debug" }
   }
   ```

## IDE 集成与命令查看

现代 IDE 对 Presets 提供了完美支持：
- **VS Code**：安装 CMake Tools 插件后，状态栏底部会自动出现预设选择器。
- **CLion**：自动检测 `CMakePresets.json` 并将其导入为 CMake Profiles。
- **Visual Studio**：直接打开项目文件夹，它会读取 `vendor` 字段并自动配置。

你也可以通过命令行查看当前可用的所有预设：
```bash
cmake --list-presets
```

## 练习题与答案

### 题目 1：理解继承冲突

如果子预设和父预设定义了同一个 `cacheVariable`（例如父预设定义 `A=1`，子预设定义 `A=2`），最终 CMake 会使用哪个值？

<details>
<summary>查看答案</summary>

子预设的值会覆盖父预设的值。在这个例子中，`A` 的最终值是 `2`。这正是我们利用继承来实现 Debug/Release 差异化的基础——父预设定义通用配置，子预设覆盖构建类型。

</details>

### 题目 2：分析 condition 变量

在 KrKr2 的 macOS 预设中，`condition` 里的 `rhs` 填的是 `"Darwin"` 而不是 `"MacOS"`，这是为什么？

<details>
<summary>查看答案</summary>

因为 `hostSystemName` 是 CMake 从底层操作系统获取的内核名称。macOS 的底层内核是 Darwin，所以 CMake 将其识别为 `Darwin`。这是一个常见的"坑"——如果你写成 `"MacOS"` 或 `"macOS"`，条件永远不会匹配。

</details>

### 题目 3：实战扩展

请为 Linux 平台增加一个名为 `Linux-Clang` 的预设，要求继承自现有的 `Linux Config`，但显式指定编译器为 `clang` 和 `clang++`。

<details>
<summary>查看答案</summary>

```json
{
  "name": "Linux-Clang",
  "inherits": "Linux Config",
  "binaryDir": "${sourceDir}/out/linux/clang",
  "cacheVariables": {
    "CMAKE_C_COMPILER": "clang",
    "CMAKE_CXX_COMPILER": "clang++",
    "CMAKE_BUILD_TYPE": "Debug"
  }
}
```

注意：通过 `inherits` 继承 `Linux Config`，自动获得了 `LINUX: true` 和 `CMAKE_EXPORT_COMPILE_COMMANDS: true` 等缓存变量，只需覆盖编译器即可。

</details>
