# 02-多平台预设管理

在前一章中，你已经掌握了 `CMakePresets.json` 的基本语法。本章我们将以 KrKr2 项目真实的配置文件为例，深入探讨如何在大规模跨平台项目中组织和管理预设。

> **所属模块：** P01-现代CMake与构建工具链  
> **前置知识：** [01-Presets配置详解](./01-Presets配置详解.md)、[02-目标与属性](../02-CMake语法与命令/02-目标与属性.md)  
> **预计阅读时间：** 35 分钟

## 本节目标

读完本节后，你将能够：

1. 按 `base -> platform -> variant` 三层继承组织多平台预设，避免配置重复。
2. 理解 Windows、Linux、macOS 在编译器、架构、运行时和生成器上的核心差异。
3. 为 Android 设计可维护的 CMake 预设方案，并正确处理 NDK Toolchain 与 ABI。
4. 在 CI/CD 中稳定复用 Presets，并通过 `CMakeUserPresets.json` 管理个人差异配置。
5. 读懂 KrKr2 当前每个 preset 的职责，知道哪些配置应保留、哪些配置可演进。

## 多平台预设设计策略：base -> platform -> variant

在中大型 C++ 项目里，建议采用三层继承，而不是复制粘贴：

- **base 层**：共享项（如 `generator`、`CMAKE_EXPORT_COMPILE_COMMANDS`）。
- **platform 层**：平台差异（如 Windows triplet、macOS 编译器、Linux 路径）。
- **variant 层**：构建差异（Debug/Release/ASan/LTO）。

KrKr2 当前已稳定使用“平台基类 + Debug/Release 变体”，后续可以自然扩展到三层。

## KrKr2 的完整 CMakePresets.json 解读

KrKr2 模拟器支持 Windows、Linux、macOS 和 Android 四个平台。为了让构建过程在所有桌面上保持一致，我们编写了如下配置：

```json
{
  "version": 6,
  "cmakeMinimumRequired": { "major": 3, "minor": 28, "patch": 0 },
  "configurePresets": [
    {
      "name": "Windows Config",
      "generator": "Ninja",
      "hidden": true,
      "cacheVariables": {
        "WINDOWS": true,
        "VCPKG_TARGET_TRIPLET": "x64-windows-static-md",
        "CMAKE_EXPORT_COMPILE_COMMANDS": true
      },
      "condition": {
        "type": "equals",
        "lhs": "${hostSystemName}",
        "rhs": "Windows"
      },
      "architecture": { "value": "x64", "strategy": "external" },
      "vendor": {
        "microsoft.com/VisualStudioSettings/CMake/1.0": {
          "hostOS": ["Windows"]
        }
      }
    },
    {
      "name": "Windows Debug Config",
      "inherits": "Windows Config",
      "binaryDir": "${sourceDir}/out/windows/debug",
      "cacheVariables": { "CMAKE_BUILD_TYPE": "Debug" }
    },
    {
      "name": "Windows Release Config",
      "inherits": "Windows Config",
      "binaryDir": "${sourceDir}/out/windows/release",
      "cacheVariables": { "CMAKE_BUILD_TYPE": "Release" }
    },
    {
      "name": "Linux Config",
      "generator": "Ninja",
      "hidden": true,
      "cacheVariables": {
        "LINUX": true,
        "CMAKE_EXPORT_COMPILE_COMMANDS": true
      },
      "condition": {
        "type": "equals",
        "lhs": "${hostSystemName}",
        "rhs": "Linux"
      }
    },
    {
      "name": "Linux Debug Config",
      "inherits": "Linux Config",
      "binaryDir": "${sourceDir}/out/linux/debug",
      "cacheVariables": { "CMAKE_BUILD_TYPE": "Debug" }
    },
    {
      "name": "Linux Release Config",
      "inherits": "Linux Config",
      "binaryDir": "${sourceDir}/out/linux/release",
      "cacheVariables": { "CMAKE_BUILD_TYPE": "Release" }
    },
    {
      "name": "MacOS Config",
      "generator": "Ninja",
      "hidden": true,
      "cacheVariables": {
        "MACOS": true,
        "CMAKE_C_COMPILER": "clang",
        "CMAKE_CXX_COMPILER": "clang++",
        "CMAKE_EXPORT_COMPILE_COMMANDS": true
      },
      "condition": {
        "type": "equals",
        "lhs": "${hostSystemName}",
        "rhs": "Darwin"
      }

    {
      "name": "MacOS Debug Config",
      "inherits": "MacOS Config",

      "cacheVariables": { "CMAKE_BUILD_TYPE": "Debug" }
    },
    {
      "name": "MacOS Release Config",
      "inherits": "MacOS Config",
      "binaryDir": "${sourceDir}/out/macos/release",
      "cacheVariables": { "CMAKE_BUILD_TYPE": "Release" }
    }
  ],
  "buildPresets": [
    { "name": "Windows Debug Build", "configurePreset": "Windows Debug Config" },
    { "name": "Windows Release Build", "configurePreset": "Windows Release Config" },
    { "name": "Linux Debug Build", "configurePreset": "Linux Debug Config" },
    { "name": "Linux Release Build", "configurePreset": "Linux Release Config" },
    { "name": "MacOS Debug Build", "configurePreset": "MacOS Debug Config" },
    { "name": "MacOS Release Build", "configurePreset": "MacOS Release Config" }
  ]
}
```

## 设计模式分析：隐藏基类 + 继承变体

在上面的 JSON 中，我们采用了一种非常经典的设计模式：**“1 个隐藏基类 + N 个公开变体”**。

1. **隐藏基类（如 `Windows Config`）**：
   - 设置 `hidden: true`，使其不会干扰用户的选择。
   - 包含该平台通用的设置：生成器（Ninja）、平台宏（`WINDOWS`）、包管理器参数（`VCPKG_TARGET_TRIPLET`）。
   - 使用 `condition` 确保该基类仅在正确的宿主系统上被激活。

2. **公开变体（如 `Windows Debug Config`）**：
   - 继承自基类。
   - 仅定义差异化参数：`CMAKE_BUILD_TYPE` 和 `binaryDir`（构建输出目录）。

这种模式的好处是：如果你需要修改 Windows 平台的通用参数，只需要改动一处，所有 Debug/Release 预设都会同步生效。

---

## 自定义平台变量的妙用

你可能注意到，我们在每个平台中定义了 `WINDOWS`、`LINUX` 或 `MACOS` 变量，而不是直接使用 CMake 标准的 `CMAKE_SYSTEM_NAME`。

**为什么这么做？**
1. **简化条件判断**：在 `CMakeLists.txt` 中，你可以直接写 `if(WINDOWS)`，这比写 `if(CMAKE_SYSTEM_NAME STREQUAL "Windows")` 要简洁得多。
2. **处理 Android 特例**：Android 构建通常由 Gradle 驱动，并通过 NDK 的工具链文件（如 `vcpkg_android.cmake`）注入参数。在复杂的混合构建环境中，标准的系统名称变量有时可能未按预期初始化，自定义变量能提供更可靠的判断依据。

## Windows 预设详解

KrKr2 的 Windows 主线是 `Ninja + MSVC`，核心是 `VCPKG_TARGET_TRIPLET=x64-windows-static-md`：

- `x64` 表示 64 位目标；如需 x86，通常另建 `x86-windows-*` 变体。
- `static` 表示依赖库倾向静态链接。
- `md` 表示 CRT 采用动态运行时（`/MD` 或 `/MDd`）。

因此它不是“全静态”，而是工程上常见的“依赖静态 + CRT 动态”折中方案。

---

## Linux 预设详解

KrKr2 Linux 预设默认跟随系统编译器（通常是 GCC），必要时可派生 Clang 变体。

- GCC 往往更贴近发行版默认生态。
- Clang 诊断可读性更好，建议在 CI 增加一条 Clang 检查线。
- 库路径尽量通过 `find_package()` 与 `CMAKE_PREFIX_PATH` 管理，避免硬编码 `/usr/*`。

---

## macOS 预设详解

KrKr2 macOS 使用 `Ninja + clang/clang++`，优点是与 Linux 命令行流程一致，CI 脚本更统一。

- **Xcode** 更偏 IDE 工程管理；**Ninja** 更偏自动化与可重复构建。
- 架构上要明确 arm64、x86_64 的交付范围。
- 若需 Universal Binary，可在 preset 注入 `CMAKE_OSX_ARCHITECTURES=arm64;x86_64`，但要接受体积与构建时长上升。

---

## Android 预设详解（建议方案）

KrKr2 当前 Android 由 Gradle 主导，这是一种合理分工。若要补 Android presets，最关键是固定三项：

- `CMAKE_TOOLCHAIN_FILE=$env{ANDROID_NDK}/build/cmake/android.toolchain.cmake`
- `ANDROID_ABI`（如 `arm64-v8a`、`x86_64`）
- `ANDROID_PLATFORM`（如 `android-24`）

建议保持“Gradle 为主、preset 为辅”，避免两套入口相互漂移。

---

## CI/CD 中使用预设的最佳实践

1. 只调用 preset 名称，不在 CI 里拼长参数。
2. 每个平台维护成对的 configure/build preset。
3. 固定构建目录与缓存目录，提升可复现性。
4. 将编译器版本固定在 Runner 镜像，减少“同 preset 不同结果”。
5. 在流水线开始先跑 `cmake --list-presets` 做快速自检。

---

## CMakeUserPresets.json：个人定制不污染仓库

团队常见矛盾是“每个人本机路径不同”。正确做法不是改 `CMakePresets.json`，而是使用 `CMakeUserPresets.json` 覆盖本地差异。

常见覆盖项包括 `VCPKG_ROOT`、`CMAKE_PREFIX_PATH`、实验编译器路径和临时 Sanitizer 开关。建议将 `CMakeUserPresets.json` 放入 `.gitignore`，仅提交公共模板说明。

## KrKr2 项目的多平台预设设计分析（逐个 preset）

下面基于仓库当前 `krkr2/CMakePresets.json` 做逐项解读：

- `Windows Config`：Windows 基类，隐藏，提供 Ninja、triplet、x64 架构、VS vendor 配置。
- `Windows Debug Config`：继承 Windows 基类，输出到 `out/windows/debug`，仅覆盖 `CMAKE_BUILD_TYPE=Debug`。
- `Windows Release Config`：继承 Windows 基类，输出到 `out/windows/release`，覆盖为 `Release`。
- `Windows MinGW Config`：Linux 主机交叉编译 Windows 的基类，显式指定 `i686-w64-mingw32-*` 工具链。
- `Linux Config`：Linux 基类，隐藏，设置 `LINUX=true` 与 `CMAKE_EXPORT_COMPILE_COMMANDS=true`。
- `Linux Debug Config`：Linux Debug 变体，输出 `out/linux/debug`。
- `Linux Release Config`：Linux Release 变体，输出 `out/linux/release`。
- `MacOS Config`：macOS 基类，隐藏，固定 `clang/clang++` 并导出编译数据库。
- `MacOS Debug Config`：macOS Debug 变体，输出 `out/macos/debug`。
- `MacOS Release Config`：macOS Release 变体，输出 `out/macos/release`。
- `Windows Debug Build` 到 `MacOS Release Build`：构建预设层，职责单一，只绑定 configurePreset。

从可维护性角度看，这套设计已经稳健。建议后续增加“工具链/特性变体层”（如 Clang、ASan、LTO），避免在现有 Debug/Release 上持续堆叠字段。

## Android 为何不在 Presets 中？

这是一个非常关键的问题。既然 Presets 这么好用，为什么不把 Android 也放进去？

在 KrKr2 中，Android 的构建流程是：
**Gradle -> CMake -> vcpkg**

Android 的构建环境非常复杂，涉及到 NDK 路径、API Level、ABI（arm64-v8a, x86_64）等。这些参数通常由 Android Studio 的 Gradle 插件动态生成并传递给 CMake。

如果将 Android 强行写入 `CMakePresets.json`：
1. 难以同步 Gradle 中的路径配置。
2. 无法方便地处理 Android 特有的多 ABI 同时构建需求。
因此，我们维持 Android 的传统构建模式，而将桌面端（Win/Lin/Mac）交给 Presets 管理。

## 如何添加新平台预设

假设你现在要为 KrKr2 添加一个 **FreeBSD** 支持，你应该遵循以下步骤：

1. **添加隐藏基类**：
   ```json
   {
     "name": "FreeBSD Config",
     "generator": "Ninja",
     "hidden": true,
     "condition": {
       "type": "equals",
       "lhs": "${hostSystemName}",
       "rhs": "FreeBSD"
     },
     "cacheVariables": { "FREEBSD": true }
   }
   ```
2. **添加公开变体**：
   ```json
   {
     "name": "FreeBSD Debug Config",
     "inherits": "FreeBSD Config",
     "binaryDir": "${sourceDir}/out/freebsd/debug",
     "cacheVariables": { "CMAKE_BUILD_TYPE": "Debug" }
   }
   ```

## IDE 集成与命令查看

现代 IDE 对 Presets 提供了完美支持：
- **VS Code**：安装 CMake Tools 插件后，状态栏底部会自动出现预设选择器。
- **CLion**：自动检测 `CMakePresets.json` 并将其导入为 CMake Profiles。
- **Visual Studio**：直接打开项目文件夹，它会读取 `vendor` 字段并自动配置。

你也可以通过命令行查看当前可用的所有预设：
```bash
cmake --list-presets
```

## 练习题与答案

### 题目 1：理解继承冲突

如果子预设和父预设定义了同一个 `cacheVariable`（例如父预设定义 `A=1`，子预设定义 `A=2`），最终 CMake 会使用哪个值？

<details>
<summary>查看答案</summary>

子预设的值会覆盖父预设的值。在这个例子中，`A` 的最终值是 `2`。这正是我们利用继承来实现 Debug/Release 差异化的基础——父预设定义通用配置，子预设覆盖构建类型。

</details>

### 题目 2：分析 condition 变量

在 KrKr2 的 macOS 预设中，`condition` 里的 `rhs` 填的是 `"Darwin"` 而不是 `"MacOS"`，这是为什么？

<details>
<summary>查看答案</summary>

因为 `hostSystemName` 是 CMake 从底层操作系统获取的内核名称。macOS 的底层内核是 Darwin，所以 CMake 将其识别为 `Darwin`。这是一个常见的"坑"——如果你写成 `"MacOS"` 或 `"macOS"`，条件永远不会匹配。

</details>

### 题目 3：实战扩展

请为 Linux 平台增加一个名为 `Linux-Clang` 的预设，要求继承自现有的 `Linux Config`，但显式指定编译器为 `clang` 和 `clang++`。

<details>
<summary>查看答案</summary>

```json
{
  "name": "Linux-Clang",
  "inherits": "Linux Config",
  "binaryDir": "${sourceDir}/out/linux/clang",
  "cacheVariables": {
    "CMAKE_C_COMPILER": "clang",
    "CMAKE_CXX_COMPILER": "clang++",
    "CMAKE_BUILD_TYPE": "Debug"
  }
}
```

注意：通过 `inherits` 继承 `Linux Config`，自动获得了 `LINUX: true` 和 `CMAKE_EXPORT_COMPILE_COMMANDS: true` 等缓存变量，只需覆盖编译器即可。

</details>

## 本节小结

- 多平台 preset 的核心是“分层继承”，而不是复制粘贴。
- KrKr2 当前采用隐藏平台基类 + Debug/Release 变体，结构清晰且易维护。
- Windows 需要特别关注 triplet、架构、CRT 策略；Linux 关注编译器与系统库路径；macOS 关注生成器与多架构产物；Android 关注 NDK Toolchain 与 ABI。
- CI/CD 应直接复用 preset 名称，避免脚本参数漂移。
- 个人机器差异应放入 `CMakeUserPresets.json`，不要污染团队公共配置。
