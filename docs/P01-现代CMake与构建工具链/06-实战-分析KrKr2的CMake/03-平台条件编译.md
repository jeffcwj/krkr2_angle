# 03-平台条件编译

在开发 KrKr2 这样的大型跨平台项目时，最头疼的问题莫过于如何让一套代码在 Windows、Linux、macOS 和 Android 上都能跑通。CMake 为我们提供了强大的条件编译支持。

本章我们将总结 KrKr2 是如何通过 CMake 优雅地处理跨平台差异的。

---

## 1. 自定义平台变量 vs CMAKE_SYSTEM_NAME

在 CMake 中，有一个内置变量叫 `CMAKE_SYSTEM_NAME`，它会告诉你当前是在什么系统上（比如 "Windows", "Linux", "Darwin"）。

但在 KrKr2 中，你经常会看到这种写法：
```cmake
if(ANDROID)
    # ...
elseif(LINUX)
    # ...
elseif(WINDOWS)
    # ...
elseif(MACOS)
    # ...
endif()
```

**为什么不直接用 `CMAKE_SYSTEM_NAME`？**
1. **可读性**：`ANDROID` 或 `WINDOWS` 显然比检查字符串 `CMAKE_SYSTEM_NAME STREQUAL "Darwin"` 要直观得多。
2. **Preset 设置**：KrKr2 在 `CMakePresets.json` 中预先定义了这些变量。对于桌面平台，我们在配置阶段就手动指定了 `WINDOWS` 或 `LINUX`。而 `ANDROID` 变量则是由 Android NDK 的工具链文件（toolchain file）自动注入的。
3. **灵活性**：有时我们需要区分具体的发行版或架构，自定义变量可以更精细地控制构建逻辑。

---

## 2. 贯穿全项目的条件编译模式

KrKr2 采用了一种“层层过滤”的条件编译策略：

1. **根目录级别**：决定入口点（Executable vs Shared Library）。
2. **核心库级别（core）**：决定是否链接 OpenMP，是否链接 Android 系统库。
3. **模块内部级别**：决定使用哪个 PAL（Platform Abstraction Layer，平台抽象层）实现。

例如在 `cpp/core/environ/` 目录下，子文件夹的组织结构如下：
- `win32/`
- `linux/`
- `android/`
- `apple/`

CMake 会根据当前的平台变量，选择性地进入 these 子目录。

---

## 3. 平台入口点差异

这是跨平台开发中最明显的区别。

### Android：SHARED 库 + JNI
Android 的应用入口在 Java 层（Activity）。C++ 代码必须被打包成 `.so` 文件。
```cmake
add_library(${PROJECT_NAME} SHARED ${CMAKE_CURRENT_SOURCE_DIR}/platforms/android/cpp/krkr2_android.cpp)
```

### Windows：add_executable + .rc 资源
Windows 桌面程序通常需要一个资源文件来定义图标。
```cmake
add_executable(${PROJECT_NAME}
    ${CMAKE_CURRENT_SOURCE_DIR}/platforms/windows/main.cpp
    ${CMAKE_CURRENT_SOURCE_DIR}/platforms/windows/game.rc
)
```

### Linux：纯净的 add_executable
Linux 版本最为简洁，通常只需要一个 `main.cpp`。

### macOS：Bundle 资源处理
macOS 上的可执行文件需要配套 `Info.plist` 才能被系统识别。
```cmake
set(APP_UI_RES
        ${CMAKE_CURRENT_SOURCE_DIR}/platforms/apple/macos/Icon.icns
        ${CMAKE_CURRENT_SOURCE_DIR}/platforms/apple/macos/Info.plist
)
add_executable(${PROJECT_NAME}
        ${CMAKE_CURRENT_SOURCE_DIR}/platforms/apple/macos/main.cpp
        ${APP_UI_RES}
)
```

## 4. 编译器特殊处理 (Compiler-Specific)

除了操作系统，不同平台的默认编译器也是巨大的挑战：
- **Windows**: MSVC (Microsoft Visual C++)
- **Linux**: GCC 或 Clang
- **Android/macOS**: Clang

在 KrKr2 中，我们使用这种模式进行适配：
```cmake
if(MSVC)
    add_compile_options("/utf-8" "/MP")
elseif(CMAKE_CXX_COMPILER_ID MATCHES "Clang" OR CMAKE_CXX_COMPILER_ID MATCHES "GNU")
    add_compile_options("-fPIC" "-Wall")
endif()
```
- `CMAKE_CXX_COMPILER_ID` 是 CMake 内置的编译器 ID 变量。
- 对于 Clang/GCC，我们需要显式开启 `-fPIC`（Position Independent Code），但在 MSVC 中则是通过 `CMAKE_POSITION_INDEPENDENT_CODE ON` 自动处理的。

## 5. 条件编译最佳实践

从 KrKr2 项目中，你可以学到以下三条黄金原则：

1. **宏定义隔离**：尽可能在 C++ 代码内部使用宏（如 `#ifdef _WIN32`），而只在 CMake 中处理决定性的编译/链接差异。
2. **生成器表达式**：对于简单的条件差异，优先使用生成器表达式 `$<$<BOOL:${ANDROID}>:...>`。它不仅语法简洁，而且在配置多配置（Multi-config）生成器（如 VS 或 Xcode）时更可靠。
3. **分层原则**：平台相关的 `add_subdirectory` 放在最顶层，核心库内尽量保持逻辑统一。

---

## 6. 如何添加新平台支持 (实战演练)

假设有一天，你需要支持 **FreeBSD** 系统。你应该怎么做？

1. **设置平台变量**：在 `CMakePresets.json` 中定义一个新的 Preset，并传入 `-DFREEBSD=ON`。
2. **根目录入口**：在根 `CMakeLists.txt` 中添加 `elseif(FREEBSD)` 分支，指向 FreeBSD 的入口文件。
3. **第三方库配置**：由于 FreeBSD 可能不支持某些特定库，你需要在 `find_package` 时做好降级处理。
4. **子模块适配**：在 `environ` 等模块中，可能需要为 FreeBSD 编写一套专属的 PAL 实现（通常可以复用 Linux 的部分代码）。

## 练习题与答案

### 题目 1：代码分析

分析以下代码段，并说明为什么 `Info.plist` 必须包含在 `add_executable` 的源文件列表中。
```cmake
if(MACOS)
    add_executable(${PROJECT_NAME} main.cpp Info.plist Icon.icns)
endif()
```

<details>
<summary>查看答案</summary>

在 macOS 上，`Info.plist` 并不是普通的资源文件，它是应用包（App Bundle）的核心配置文件。通过将其包含在 `add_executable` 的列表中，CMake 能够识别到这是一个 macOS 特有的构建任务，并自动根据该文件生成完整的 App Bundle 目录结构。

</details>

### 题目 2：编译器差异

如果我们在 MSVC 编译器下误用了 `-Wall` 选项，会发生什么？CMake 中应该如何避免这种情况？

<details>
<summary>查看答案</summary>

- **后果**：MSVC 编译器会报错，因为它不识别 `-Wall`。MSVC 对应的选项是 `/W4` 或 `/Wall`（但其 `/Wall` 的语义与 GCC 不同，通常不建议直接使用）。
- **避免方法**：使用 `if(MSVC)` 或 `if(CMAKE_CXX_COMPILER_ID STREQUAL "MSVC")` 进行编译器类型判断，确保不同的编译器接收到各自支持的选项参数。

</details>
