## 1. 平台条件编译的两层模型

KrKr2 的平台处理不是一个开关，而是两层协同：

- **配置层（CMake）**：决定构建目标、源文件、链接库。
- **编译层（预处理器）**：决定某段语句是否参与编译。

为什么要分两层：

1. 配置层适合解决“大分叉”，如 Android 要产出 `.so`。
2. 编译层适合解决“小分叉”，如同一文件内 API 调用差异。
3. 大部分平台问题前移到 CMake，会显著降低源码复杂度。

## 2. CMake 平台检测变量完整清单

| 变量 | 类型 | 示例值 | 作用 |
|---|---|---|---|
| `WIN32` | 内置布尔 | `TRUE/FALSE` | 目标系统是否 Windows |
| `APPLE` | 内置布尔 | `TRUE/FALSE` | 目标系统是否 Apple 平台 |
| `UNIX` | 内置布尔 | `TRUE/FALSE` | 目标系统是否 Unix-like |
| `ANDROID` | 内置布尔 | `TRUE/FALSE` | 是否使用 Android NDK 工具链 |
| `CMAKE_SYSTEM_NAME` | 字符串 | `Windows/Linux/Darwin/Android` | 目标系统名 |
| `CMAKE_SYSTEM_PROCESSOR` | 字符串 | `x86_64/arm64/aarch64` | 目标架构 |

实战注意：

1. Android 往往也满足 Unix-like 特征，`UNIX` 可能为真。
2. `CMAKE_SYSTEM_NAME` 表示目标，不是宿主机。
3. 架构细分建议使用 `CMAKE_SYSTEM_PROCESSOR` 或 `ANDROID_ABI`。

## 3. KrKr2 的平台变量策略

KrKr2 在 `CMakePresets.json` 中显式设置 `WINDOWS/LINUX/MACOS`。
Android 则主要依赖 NDK 工具链注入的 `ANDROID`。

这种策略的收益：

1. 语义清晰，`if(MACOS)` 比字符串比较更直观。
2. 跨平台预设统一管理，配置行为更稳定。
3. 后续新增平台时可以增量扩展变量体系。

## 4. `if/elseif/else/endif` 语法详解

### 4.1 基本结构

```cmake
if(<condition>)
    # A
elseif(<condition2>)
    # B
else()
    # fallback
endif()
```

### 4.2 组合条件

```cmake
if(WINDOWS OR APPLE)
    message(STATUS "desktop")
endif()

if(ENABLE_TESTS AND NOT (IOS OR ANDROID))
    enable_testing()
endif()
```

### 4.3 规则与陷阱

1. 布尔变量优先写 `if(VAR)`。
2. 字符串比较才用 `STREQUAL`。
3. 分支顺序从“最特化”到“最通用”。

错误示例：

```cmake
if(UNIX)
    target_link_libraries(app PRIVATE pthread)
elseif(ANDROID)
    target_link_libraries(app PRIVATE log android)
endif()
```

正确示例：

```cmake
if(ANDROID)
    target_link_libraries(app PRIVATE log android)
elseif(UNIX)
    target_link_libraries(app PRIVATE pthread)
endif()
```

## 5. C/C++ 预处理器宏与 CMake 条件配合

### 5.1 配合原则

- CMake 决定“编不编这个文件”。
- 预处理器决定“编这个文件时走哪段语句”。

### 5.2 KrKr2 的现实做法

检索可见，KrKr2 业务源码中少见大面积 `_WIN32/__APPLE__/__ANDROID__` 分支。
平台差异主要通过目录拆分与 CMake 装配完成。

### 5.3 教学示例

```cpp
#if defined(_WIN32)
const char* kPlatform = "_WIN32";
#elif defined(__APPLE__)
const char* kPlatform = "__APPLE__";
#elif defined(__ANDROID__)
const char* kPlatform = "__ANDROID__";
#elif defined(__linux__)
const char* kPlatform = "__linux__";
#else
const char* kPlatform = "unknown";
#endif
```

## 6. KrKr2 实例逐一分析

### 6.1 根入口：`krkr2/CMakeLists.txt`

核心分支（第 46-84 行）决定入口形态：

```cmake
if(ANDROID)
    add_library(${PROJECT_NAME} SHARED ${CMAKE_CURRENT_SOURCE_DIR}/platforms/android/cpp/krkr2_android.cpp)
elseif(LINUX)
    add_executable(${PROJECT_NAME} ${CMAKE_CURRENT_SOURCE_DIR}/platforms/linux/main.cpp)
elseif(WINDOWS)
    add_executable(${PROJECT_NAME}
        ${CMAKE_CURRENT_SOURCE_DIR}/platforms/windows/main.cpp
        ${CMAKE_CURRENT_SOURCE_DIR}/platforms/windows/game.rc
    )
elseif(MACOS)
    set(APP_UI_RES
            ${CMAKE_CURRENT_SOURCE_DIR}/platforms/apple/macos/Icon.icns
            ${CMAKE_CURRENT_SOURCE_DIR}/platforms/apple/macos/Info.plist
    )
    add_executable(${PROJECT_NAME}
            ${CMAKE_CURRENT_SOURCE_DIR}/platforms/apple/macos/main.cpp
            ${APP_UI_RES}
    )
endif()
```

分析：Android 为 JNI 场景产出 `SHARED`，Windows 入口附带 `.rc`，macOS 入口附带 `Info.plist` 与图标资源。

### 6.2 根层门禁：测试与工具

```cmake
if(ENABLE_TESTS AND NOT (IOS OR ANDROID))
    enable_testing()
    add_subdirectory(tests)
endif()

if(BUILD_TOOLS AND NOT (IOS OR ANDROID))
    add_subdirectory(tools)
endif()
```

这确保移动端构建不会误带桌面测试/工具目标。

### 6.3 core 层：`krkr2/cpp/core/CMakeLists.txt`

OpenMP 与 Android 系统库分支：

```cmake
if(NOT APPLE)
    find_package(OpenMP REQUIRED)
    target_link_libraries(${PROJECT_NAME} INTERFACE OpenMP::OpenMP_CXX)
endif ()

if(ANDROID)
    target_link_libraries(${PROJECT_NAME} INTERFACE
        log
        android
        EGL
        GLESv2
        GLESv1_CM
        OpenSLES
    )
endif()
```

生成器表达式补充：

```cmake
$<$<BOOL:${ANDROID}>:$<LINK_LIBRARY:WHOLE_ARCHIVE,cocos2dx::cpp_android_spec>>
```

### 6.4 environ 层：`krkr2/cpp/core/environ/CMakeLists.txt`

平台文件装配片段：

```cmake
$<$<BOOL:${ANDROID}>:${ENVIRON_PATH}/android/AndroidUtils.cpp>
$<$<BOOL:${ANDROID}>:${ENVIRON_PATH}/linux/Platform.cpp>
$<$<BOOL:${LINUX}>:${ENVIRON_PATH}/linux/Platform.cpp>
$<$<BOOL:${WINDOWS}>:${ENVIRON_PATH}/win32/Platform.cpp>
$<$<BOOL:${MACOS}>:${ENVIRON_PATH}/apple/macos/platform.mm>
```

这里展示了“按平台选实现文件”而非“在单文件堆宏分支”。

### 6.5 资源层：`krkr2/cmake/CocosBuildHelpers.cmake`

`setup_cocos_app_config` 的平台分支：

```cmake
if(APPLE)
    set_target_properties(${app_name} PROPERTIES MACOSX_BUNDLE 1)
elseif(MSVC)
    set_property(TARGET ${app_name} APPEND PROPERTY LINK_FLAGS "/SUBSYSTEM:WINDOWS")
endif()

if(LINUX OR WINDOWS)
    cocos_def_copy_resource_target(${app_name})
endif()
```

### 6.6 Android 工具链层：`krkr2/cmake/vcpkg_android.cmake`

ABI 映射与双 toolchain：

```cmake
if (ANDROID_ABI MATCHES "arm64-v8a")
    set(VCPKG_TARGET_TRIPLET "arm64-android" CACHE STRING "" FORCE)
elseif(ANDROID_ABI MATCHES "armeabi-v7a")
    set(VCPKG_TARGET_TRIPLET "arm-android" CACHE STRING "" FORCE)
elseif(ANDROID_ABI MATCHES "x86_64")
    set(VCPKG_TARGET_TRIPLET "x64-android" CACHE STRING "" FORCE)
elseif(ANDROID_ABI MATCHES "x86")
    set(VCPKG_TARGET_TRIPLET "x86-android" CACHE STRING "" FORCE)
endif()

set(VCPKG_CHAINLOAD_TOOLCHAIN_FILE $ENV{ANDROID_NDK_HOME}/build/cmake/android.toolchain.cmake)
set(CMAKE_TOOLCHAIN_FILE $ENV{VCPKG_ROOT}/scripts/buildsystems/vcpkg.cmake)
```

## 7. 跨平台抽象层设计模式

KrKr2 体现了三段式模式：

1. **目录抽象**：`win32/linux/android/apple` 分目录。
2. **构建抽象**：CMake 装配平台实现文件。
3. **链接抽象**：target 级平台库集中声明。

这种模式让平台差异点集中，新增平台可以增量扩展，业务层也更干净。

## 8. 平台特有编译选项和链接库

### 8.1 Windows

根 CMake 使用：

```cmake
if(MSVC)
    add_compile_options("$<$<COMPILE_LANGUAGE:CXX>:/EHsc>" "/MP" "/utf-8")
    add_link_options("/ignore:4099" "/INCREMENTAL" "/DEBUG:FASTLINK")
endif()
```

### 8.2 Linux / macOS / Android

- Linux：主要差异在入口与资源同步流程，core 层参与 OpenMP。
- macOS：确保 `Info.plist`、`MACOSX_BUNDLE`、`.mm` 编译链同时生效。
- Android：核心是 `log/android/EGL/GLESv2/GLESv1_CM/OpenSLES` 系统库集合与 ABI 对齐。

## 9. Android NDK 交叉编译特殊处理

### 9.1 根入口触发点

```cmake
if(ANDROID)
    set(VCPKG_TARGET_ANDROID ON)
    set(ENV{ANDROID_NDK_HOME} "$ENV{ANDROID_NDK}")
    include(cmake/vcpkg_android.cmake)
endif()
```

### 9.2 ABI 对应关系

| ANDROID_ABI | VCPKG_TARGET_TRIPLET |
|---|---|
| arm64-v8a | arm64-android |
| armeabi-v7a | arm-android |
| x86_64 | x64-android |
| x86 | x86-android |

### 9.3 常见失败点

1. `ANDROID_NDK_HOME` 未设置。
2. `ANDROID_ABI` 未传入或值不支持。
3. toolchain 组合顺序错误。
4. ABI 与 triplet 不一致。

## 10. 常见跨平台陷阱与解决方案

### 陷阱 1：Android 与 Unix 条件冲突

解决：Android 分支必须前置。

### 陷阱 2：混用 `WIN32` 与 `WINDOWS`

解决：统一团队主变量，保持一致命名。

### 陷阱 3：漏加平台实现文件

解决：对照 `ENVIRON_SOURCE_FILES` 与平台目录实现。

### 陷阱 4：macOS 资源未进入 Bundle

解决：确保 `Info.plist`、图标、`MACOSX_BUNDLE` 三者同时到位。

### 陷阱 5：Android 架构错配

解决：固定 ABI 输入，脚本映射 triplet，避免手工写死。

## 12. 动手实践

> 目标：搭建一个最小跨平台实验工程，验证条件分支是否命中预期。

### 步骤 1：准备目录

```text
platform-lab/
├── CMakeLists.txt
└── src/
    ├── main.cpp
    ├── platform_windows.cpp
    ├── platform_linux.cpp
    ├── platform_android.cpp
    └── platform_macos.mm
```

### 步骤 2：写入口与平台函数声明

`src/main.cpp`：

```cpp
#include <iostream>

const char* platform_name();

int main() {
    std::cout << "platform = " << platform_name() << std::endl;
    return 0;
}
```

### 步骤 3：写四个平台实现

```cpp
// src/platform_windows.cpp

```

```cpp
// src/platform_linux.cpp

```

```cpp
// src/platform_android.cpp
const char* platform_name() { return "Android"; }
```

```cpp
// src/platform_macos.mm
const char* platform_name() { return "macOS"; }
```

### 步骤 4：写 CMake 装配逻辑

```cmake

project(platform_lab LANGUAGES CXX)

if(APPLE)
    enable_language(OBJCXX)
endif()

add_executable(platform_lab
    src/main.cpp
    $<$<BOOL:${WINDOWS}>:${CMAKE_CURRENT_SOURCE_DIR}/src/platform_windows.cpp>
    $<$<BOOL:${LINUX}>:${CMAKE_CURRENT_SOURCE_DIR}/src/platform_linux.cpp>
    $<$<BOOL:${ANDROID}>:${CMAKE_CURRENT_SOURCE_DIR}/src/platform_android.cpp>
    $<$<BOOL:${MACOS}>:${CMAKE_CURRENT_SOURCE_DIR}/src/platform_macos.mm>
)
```

### 步骤 5：验证

1. 切换不同 Preset，观察输出平台名。
2. 检查编译日志，确认只编译一个平台实现文件。

## 13. 本节小结

- 平台条件编译要明确“配置期”与“编译期”分工。
- KrKr2 主要通过 CMake 分支和目录隔离解决平台差异。
- 六个关键变量要会用：`WIN32/APPLE/UNIX/ANDROID/CMAKE_SYSTEM_NAME/CMAKE_SYSTEM_PROCESSOR`。
- Android 交叉编译最关键的是 ABI、triplet、toolchain 三者一致。
- 目录抽象 + 构建抽象 + 链接抽象，是可维护跨平台工程的稳定模式。

### 题目 2：变量语义

说明 `CMAKE_SYSTEM_NAME` 与宿主系统名的区别，并给出 Android 交叉编译时的典型值。

<details>
<summary>查看答案</summary>

`CMAKE_SYSTEM_NAME` 描述目标系统，不描述宿主系统。
在 Linux 主机交叉编译 Android 时，宿主是 Linux，但 `CMAKE_SYSTEM_NAME` 应是 `Android`。
这决定了编译器、sysroot、系统库和 ABI 的选择。

</details>
