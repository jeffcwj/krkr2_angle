## vcpkg 是什么

vcpkg 是 Microsoft 维护的开源 C++ 包管理器。

它把依赖下载、构建、安装流程统一化。

在传统 C++ 项目中，手工接入依赖很常见。

手工流程会带来版本漂移和环境不一致。

vcpkg 的价值就是提供可复现依赖流程。

### 与 pip、npm、cargo 的对比

- `pip` 管理 Python 包。
- `npm` 管理 Node.js 包。
- `cargo` 管理 Rust crate。
- `vcpkg` 管理 C/C++ 第三方库。

共性：

1. 支持依赖声明。
2. 支持自动安装。
3. 强调可复现。

差异：

C/C++ 依赖涉及 ABI、编译器和工具链差异。

因此 vcpkg 还要处理 triplet 与交叉编译。

## 为什么需要包管理器

没有包管理器时，C++ 团队常见问题：

1. 本地依赖版本不一致。
2. CI 与本地结果不一致。
3. 依赖升级回滚困难。
4. 新成员环境搭建慢。

vcpkg 的改进路径是：

- 依赖写入 `vcpkg.json`。
- CMake 通过工具链统一接入。
- triplet 统一平台差异。

## 安装 vcpkg（四平台）

### Windows：git clone + bootstrap-vcpkg.bat

```powershell
cd C:\dev
git clone https://github.com/microsoft/vcpkg.git
cd .\vcpkg
.\bootstrap-vcpkg.bat
```

完成后应有 `vcpkg.exe`。

### Linux：git clone + bootstrap-vcpkg.sh

```bash
sudo apt-get update
sudo apt-get install -y build-essential git curl zip unzip tar pkg-config
cd $HOME
git clone https://github.com/microsoft/vcpkg.git
cd vcpkg
./bootstrap-vcpkg.sh
```

完成后应有 `vcpkg` 可执行文件。

### macOS：brew 或手动

```bash
xcode-select --install
brew install git cmake ninja
cd $HOME
git clone https://github.com/microsoft/vcpkg.git
cd vcpkg
./bootstrap-vcpkg.sh
```

### CI 环境

Linux Runner：

```bash
git clone https://github.com/microsoft/vcpkg.git "$RUNNER_TEMP/vcpkg"
"$RUNNER_TEMP/vcpkg/bootstrap-vcpkg.sh"
```

Windows Runner：

```powershell
git clone https://github.com/microsoft/vcpkg.git "$env:RUNNER_TEMP\vcpkg"
& "$env:RUNNER_TEMP\vcpkg\bootstrap-vcpkg.bat"
```

Android 依赖通常在宿主机交叉编译。

## 设置 VCPKG_ROOT 环境变量（四平台）

`VCPKG_ROOT` 指向 vcpkg 根目录。

工具链路径通常写作：

`$VCPKG_ROOT/scripts/buildsystems/vcpkg.cmake`

### Windows

```powershell
$env:VCPKG_ROOT = "C:\dev\vcpkg"
[Environment]::SetEnvironmentVariable("VCPKG_ROOT", "C:\dev\vcpkg", "User")
echo $env:VCPKG_ROOT
Test-Path "$env:VCPKG_ROOT\scripts\buildsystems\vcpkg.cmake"
```

### Linux

```bash
export VCPKG_ROOT=$HOME/vcpkg
export PATH=$VCPKG_ROOT:$PATH
source ~/.bashrc
echo $VCPKG_ROOT
test -f "$VCPKG_ROOT/scripts/buildsystems/vcpkg.cmake" && echo OK
```

### macOS

```bash
export VCPKG_ROOT=$HOME/vcpkg
export PATH=$VCPKG_ROOT:$PATH
source ~/.zshrc
echo $VCPKG_ROOT
test -f "$VCPKG_ROOT/scripts/buildsystems/vcpkg.cmake" && echo OK
```

### CI 与 Android

```yaml
env:
  VCPKG_ROOT: ${{ runner.temp }}/vcpkg
```

Android 常见附加变量：

- `ANDROID_NDK`
- `VCPKG_CHAINLOAD_TOOLCHAIN_FILE`

## vcpkg 目录结构说明

```text
vcpkg/
├─ ports/
├─ installed/
├─ buildtrees/
└─ downloads/
```

### `ports/`

存放包构建配方。

### `installed/`

存放最终安装产物。

示例：

- `installed/x64-windows/include`
- `installed/x64-linux/lib`
- `installed/arm64-android/lib`

### `buildtrees/`

存放构建中间文件与日志。

失败时优先查看。

### `downloads/`

存放下载缓存。

CI 缓存它可减少下载时间。

## Classic 模式 vs Manifest 模式概述

| 维度 | Classic | Manifest |
|---|---|---|
| 依赖声明 | 命令行临时安装 | `vcpkg.json` |
| 复现性 | 一般 | 强 |
| 协作性 | 一般 | 强 |
| CI 适配 | 一般 | 强 |

Classic：

```bash
vcpkg install zlib
```

Manifest：

```bash
vcpkg install --triplet x64-windows
```

本项目采用 Manifest。

## vcpkg integrate install（Visual Studio 集成）

```powershell
vcpkg integrate install
```

作用是启用用户级 VS 集成。

它不能替代 `vcpkg.json`。


PowerShell：

```powershell
Test-Path "$env:VCPKG_ROOT\scripts\buildsystems\vcpkg.cmake"
```

## 常见安装问题排查

### 问题 1：命令不可用

1. 未执行 bootstrap。
2. PATH 未包含 vcpkg。
3. 用绝对路径确认可执行文件。

### 问题 2：CMake 找不到包

1. 未传工具链参数。
2. triplet 不匹配。
3. 对应包未安装。

```bash
cmake -S . -B build -DCMAKE_TOOLCHAIN_FILE="$VCPKG_ROOT/scripts/buildsystems/vcpkg.cmake"
```

### 问题 3：下载失败或校验失败

1. 检查网络和代理。
2. 清理损坏缓存。
3. 检查 CI 缓存策略。

### 问题 4：Android 交叉编译失败

1. 检查 `ANDROID_NDK`。
2. 检查 `VCPKG_CHAINLOAD_TOOLCHAIN_FILE`。
3. 检查 `arm64-android` 等 triplet。

## 本节小结

- vcpkg 统一 C++ 依赖管理。
- 安装核心是 clone + bootstrap。
- `VCPKG_ROOT` 是关键配置。
- `buildtrees/` 与 `downloads/` 是常用排障目录。
- 团队项目优先 Manifest 模式。

## 练习题与答案

### 题目 1

说出 vcpkg 与 pip 的一个共同点和一个差异点。

<details>
<summary>查看答案</summary>

共同点：都可自动安装依赖。

差异点：vcpkg 管理 C/C++ 库并涉及本地编译，pip 主要管理 Python 包。

</details>

### 题目 2

包构建失败时为什么要先看 `buildtrees/`。

<details>
<summary>查看答案</summary>

因为 `buildtrees/` 保存 configure/build/install 日志，最容易定位失败根因。

</details>

### 题目 3

团队要保证本地和 CI 依赖一致，应选哪种模式。

<details>
<summary>查看答案</summary>

应选 Manifest 模式。

依赖记录在 `vcpkg.json`，便于版本化和可复现构建。

</details>

### 题目 4

`vcpkg integrate install` 能否替代 `vcpkg.json`。

<details>
<summary>查看答案</summary>

不能。

前者是用户级 VS 集成，后者是项目级依赖声明。

</details>
