## 二进制缓存核心概念

### 什么是 Cache Hit 与 Cache Miss

- **Cache Hit（命中）**：vcpkg 在缓存后端找到了当前构建指纹对应的包，直接下载并解压。
- **Cache Miss（未命中）**：缓存里没有可用包，vcpkg 执行完整编译并回写缓存。

### 二进制缓存工作流

```text
计算 ABI/配置指纹
    ↓
按 VCPKG_BINARY_SOURCES 顺序查询缓存源
    ↓
命中: 下载并解压到 installed
未命中: 编译 -> 打包 -> 写入可写缓存源
```

### 缓存键为什么稳定且安全

缓存键不是简单的“库名+版本”，而是与构建上下文强绑定。这样可以防止“错误二进制复用”。例如 Debug 与 Release、x64 与 arm64 必然分离。

## 支持的后端类型总览

vcpkg 常见可用后端如下（本节重点覆盖）：

| 后端类型 | 典型用途 | 是否适合团队共享 | 备注 |
|---|---|---|---|
| `default` | 本机默认缓存目录 | 否 | 零配置可用，适合个人开发 |
| `files` | 局域网/NAS 文件共享 | 是 | 内网团队常用，成本低 |
| `nuget` | CI/CD 远程缓存 | 是 | 与 GitHub Packages、Azure Artifacts 配合好 |
| `http` | 自建制品下载服务 | 是 | 常用于企业私有仓库 |
| `gcs` | Google Cloud Storage | 是 | GCP 体系常见 |
| `aws` | Amazon S3 | 是 | AWS 体系常见 |
| `cos` | 腾讯云 COS | 是 | 国内云场景常见 |

## 默认缓存路径（跨平台）

默认情况下，vcpkg 本地缓存开启。路径按宿主系统区分：

- **Windows：** `%LOCALAPPDATA%\vcpkg\archives`
- **Linux：** `~/.cache/vcpkg/archives`
- **macOS：** `~/.cache/vcpkg/archives`
- **Android：** Android 目标库通常在宿主机交叉编译，缓存仍落在宿主机路径上；如果 CI Runner 是 Linux，则仍为 `~/.cache/vcpkg/archives`

建议你把这个目录加入磁盘监控。大型项目缓存可能增长到数 GB 到数十 GB。

## 环境变量 VCPKG_BINARY_SOURCES 配置语法

### 基础格式

```bash
VCPKG_BINARY_SOURCES="<source1>;<source2>;<source3>"
```

规则说明：

1. 使用分号分隔多个源。
2. 查询顺序与书写顺序一致。
3. `clear` 一般放在最前，避免继承旧配置。
4. 每个源后面可以附读写策略，如 `read`、`write`、`readwrite`。

### 常见片段

```bash
clear
default,readwrite
files,\\\\nas\vcpkg-cache,readwrite
nuget,https://nuget.pkg.github.com/<owner>/index.json,readwrite
http,https://cache.example.com/vcpkg,read
```

### 推荐组合示例

#### 个人开发（仅本地）

```bash
set VCPKG_BINARY_SOURCES=clear;default,readwrite
```

#### 团队内网（局域网共享 + 本地兜底）

```bash
set VCPKG_BINARY_SOURCES=clear;files,\\nas\vcpkg-cache,readwrite;default,readwrite
```

#### CI（远程可写 + 本地可读）

```bash
VCPKG_BINARY_SOURCES="clear;nuget,https://nuget.pkg.github.com/org/index.json,readwrite;default,read"
```

## 在 vcpkg-configuration.json 中配置二进制缓存

当你希望“配置跟仓库走”而非“配置跟开发者机器走”时，应优先使用 `vcpkg-configuration.json`。

### 示例一：团队统一配置

```json
{
  "$schema": "https://raw.githubusercontent.com/microsoft/vcpkg-tool/main/docs/vcpkg-configuration.schema.json",
  "default-registry": {
    "kind": "builtin",
    "baseline": "aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa"
  },
  "binarySources": [
    "clear",
    "nuget,https://nuget.pkg.github.com/your-org/index.json,read",
    "files,//nas/vcpkg-cache,readwrite",
    "default,readwrite"
  ]
}
```

说明：

- `binarySources` 与环境变量语义一致，但可版本化管理。
- `nuget` 放前面时，命中远程后可直接复用。
- `files` 适合作为团队内部二级缓存。

### 示例二：按分支策略切换权限

主分支构建可设置可写，PR 构建设置只读，避免不稳定产物污染主缓存。该策略通常由 CI 变量动态注入实现。

## NuGet 后端配置详解（CI/CD 重点）

NuGet 后端是 vcpkg 在 CI 场景中最常见的远程缓存方案，原因有三点：

1. 认证体系成熟，支持 Token。
2. 在 GitHub/Azure 生态中接入成本低。
3. 对“包上传+下载”模型支持稳定。

### GitHub Actions 样例

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup NuGet + vcpkg cache source
        shell: bash
        run: |
          OWNER="${{ github.repository_owner }}"
          TOKEN="${{ secrets.GITHUB_TOKEN }}"
          dotnet nuget add source "https://nuget.pkg.github.com/${OWNER}/index.json" --name github-vcpkg --username "${OWNER}" --password "${TOKEN}" --store-password-in-clear-text
          echo "VCPKG_BINARY_SOURCES=clear;nuget,https://nuget.pkg.github.com/${OWNER}/index.json,readwrite;default,read" >> "$GITHUB_ENV"
      - run: vcpkg install --debug
```

如果你的安全策略要求 PR 不写缓存，可将 `readwrite` 改为 `read`。

## 文件系统后端（files）

`files` 后端通常部署在局域网共享目录（SMB、NFS、NAS）。

### 适用场景

- 公司内网开发机数量多。
- 外网带宽有限，拉取云仓库慢。
- 希望成本最低、部署最简单。

### 配置示例

Windows 开发机：

```powershell
$env:VCPKG_BINARY_SOURCES = "clear;files,\\nas\vcpkg-cache,readwrite;default,readwrite"
```

Linux/macOS 开发机：

```bash
export VCPKG_BINARY_SOURCES="clear;files,/mnt/nas/vcpkg-cache,readwrite;default,readwrite"
```

### 文件系统后端运维建议

1. 缓存目录单独分区，避免挤占业务盘。
2. 按月归档，保留最近活跃 triplet 的产物。
3. 对共享目录开启只读镜像权限给普通开发者，写权限留给 CI。

## HTTP 后端（http）

HTTP 后端适合已有制品系统的团队。常见模式是：

- CI 将二进制包上传到对象存储。
- 对外暴露只读 HTTP 下载地址。
- 开发机只读使用，统一来源。

### 配置示例

```bash
export VCPKG_BINARY_SOURCES="clear;http,https://artifacts.example.com/vcpkg,read;default,readwrite"
```

### HTTP 后端注意事项

1. 建议全站 HTTPS，避免中间人篡改。
2. 搭配签名或网关鉴权，防止未授权下载。
3. 设置合理缓存头，减轻源站压力。

## 云对象存储后端补充（gcs、aws、cos）

这三类后端主要用于云上团队：

- `gcs`：Google Cloud Storage
- `aws`：Amazon S3
- `cos`：Tencent Cloud Object Storage

实践建议：

1. 使用最小权限访问密钥。
2. 设置生命周期规则，自动清理陈旧缓存。
3. 统一区域，避免跨区流量成本。

## GitHub Actions 缓存集成策略

推荐将远程 NuGet 与 `actions/cache` 混合使用：远程负责跨 Runner 复用，本地负责同分支二次加速。

### 组合示例

```yaml
- name: Restore local vcpkg archive cache
  uses: actions/cache@v4
  with:
    path: ~/.cache/vcpkg/archives
    key: vcpkg-archives-${{ runner.os }}-${{ hashFiles('vcpkg.json') }}
    restore-keys: |
      vcpkg-archives-${{ runner.os }}-
```

注意：`actions/cache` 是工作流级缓存，NuGet 是制品级缓存，组合时命中率通常更高。

## 缓存键生成规则（理解未命中的前提）

vcpkg 的二进制缓存命中依赖 ABI 相关输入。可理解为“配置指纹”。常见组成如下：

1. **Triplet**（例如 `x64-windows`、`arm64-android`）。
2. **Portfile 与 patches 的哈希**。
3. **依赖图版本状态**（上游依赖变化会传导）。
4. **编译器与工具链信息**（如 MSVC、Clang 版本）。
5. **关键编译选项与环境变量**（影响 ABI 的项）。
6. **vcpkg 版本与部分内部策略**。

这就是为什么“只改了一点 triplet”也会触发缓存失效。严格失效是正确行为，不是故障。

## 缓存命中/未命中的调试方法

### 启用调试输出

```bash
vcpkg install zlib --debug
```

你应重点关注这些信息：

1. 当前生效的 `VCPKG_BINARY_SOURCES` 顺序。
2. 每个源是否可访问、是否有权限。
3. ABI 或二进制包 ID 是否变化。
4. 命中后是否成功解压，未命中后是否成功回写。

### 常见未命中原因与修复

| 现象 | 根因 | 修复建议 |
|---|---|---|
| 同仓库同分支频繁未命中 | Triplet 被多人修改 | 冻结 triplet，改动走评审 |
| 本地命中，CI 未命中 | CI 缺少认证或源顺序不同 | 对齐 `VCPKG_BINARY_SOURCES` 与凭据 |
| PR 构建命中率低 | PR 使用临时编译器镜像 | 固定 Runner 镜像版本 |
| 缓存可读不可写 | Token 权限不足 | 检查包写入权限范围 |

## CI/CD 中的最佳实践

1. **分层读取**：远程主缓存在前，本地默认缓存在后。
2. **权限分级**：主分支 `readwrite`，PR `read`。
3. **统一工具链**：固定 CMake、编译器、vcpkg 版本。
4. **可观测性**：保留 `--debug` 关键日志，便于故障回溯。
5. **生命周期治理**：给远程缓存设置保留周期和清理任务。
6. **容量预警**：对本地和远端缓存设置阈值告警。
7. **避免污染**：不要把实验性 triplet 写入共享生产缓存。

## 对照项目源码

- `.github/workflows/`：查看缓存源注入与凭据设置。
- `vcpkg.json`、`vcpkg/`：查看依赖和 triplet 变化是否影响命中。
- CI 构建日志：查看 `--debug` 下的 Binary Cache 输出。

## 动手实践

### 步骤 1：验证本地 default 缓存

1. 设置仅使用本地缓存：`clear;default,readwrite`。
2. 执行一次 `vcpkg install zlib`。
3. 删除 `installed` 目录后再次安装。
4. 观察第二次是否明显提速并出现缓存命中日志。

### 步骤 2：接入 files 局域网共享

1. 创建共享目录并授权 CI 可写。
2. 设置 `VCPKG_BINARY_SOURCES=clear;files,<共享路径>,readwrite;default,readwrite`。
3. 在两台机器上安装同一依赖集。
4. 验证第二台机器是否实现“首编译即命中下载”。

### 步骤 3：执行一次未命中排障

1. 故意修改一个 triplet 变量。
2. 执行 `vcpkg install --debug`。
3. 记录命中失败原因。
4. 回滚 triplet 后再次执行，确认命中恢复。

## 本节小结

- 二进制缓存的目标是减少重复编译，核心机制是“按构建指纹复用产物”。
- `VCPKG_BINARY_SOURCES` 决定了缓存源顺序、权限和回写行为。
- `default` 适合个人，`files` 适合内网，`nuget/http/gcs/aws/cos` 适合远程与云环境。
- CI 推荐采用“远程主缓存 + 本地辅助缓存”的分层策略。
- 通过 `--debug` 可快速定位命中失败的关键原因。

## 练习题与答案

### 题目 1：后端选择题
你所在团队有 30 台 Windows 开发机，全部在公司内网，不允许访问公网。最先落地的缓存后端应该选什么？请给出一条可用配置。

<details>
<summary>查看答案</summary>

优先选 `files`，因为部署简单且满足内网共享。
可用配置：

```powershell
$env:VCPKG_BINARY_SOURCES = "clear;files,\\nas\vcpkg-cache,readwrite;default,readwrite"
```

解释：`clear` 防止历史配置干扰，`files` 提供团队共享缓存，`default` 保留本机缓存提高二次安装速度。

</details>

### 题目 2：命中分析题
同事反馈“昨天 CI 能命中，今天全部未命中”。你检查到 `vcpkg.json` 未变化，但 `arm64-android` triplet 刚被修改。请判断是否合理，并说明原因。

<details>
<summary>查看答案</summary>

这是合理现象。triplet 是 ABI 指纹的重要输入。triplet 变化会直接影响缓存键，导致旧包不再匹配，触发未命中并重新编译。
建议对 triplet 变更做评审并记录影响范围，同时在变更窗口预热主缓存，减少开发者等待。

</details>

### 题目 3：配置纠错题
以下配置有什么问题，如何修正？

```text
VCPKG_BINARY_SOURCES="nuget,https://nuget.pkg.github.com/acme/index.json,readwrite;clear;default,read"
```

<details>
<summary>查看答案</summary>

问题是 `clear` 放在了中间。它会清空之前累积的源定义，导致前面的 `nuget` 无效。
修正为：

```text
VCPKG_BINARY_SOURCES="clear;nuget,https://nuget.pkg.github.com/acme/index.json,readwrite;default,read"
```

同时建议在 CI 中打印最终环境变量，避免源顺序错误并便于审计。

</details>
