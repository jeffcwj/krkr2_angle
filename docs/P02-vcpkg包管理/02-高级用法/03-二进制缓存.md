# 03-二进制缓存 (Binary Caching)

在大型项目中，依赖库的编译往往需要耗费数小时甚至更久。**二进制缓存 (Binary Caching)** 是 vcpkg 提供的一项极其强大的功能，它能让你只需在本地或 CI 服务器上编译一次，其他开发者或后续的 CI 构建就可以直接下载编译好的产物。

## 什么是二进制缓存

二进制缓存的原理非常直观：
1.  vcpkg 在开始编译库之前，会计算一个唯一的 **哈希值 (Hash)**。
2.  这个哈希值由 `vcpkg.json`、`portfile.cmake`、所选的 Triplet 以及编译器版本共同决定。
3.  vcpkg 会去缓存源（本地或远程）中搜索这个哈希值对应的压缩包。
4.  如果找到了（Cache Hit），就直接解压到 `installed` 目录，跳过编译。
5.  如果找不到（Cache Miss），就进行正常的编译流程，编译完成后将结果上传到缓存源。

## 本地缓存 (Default)

默认情况下，vcpkg 的本地二进制缓存是开启的。
-   **Windows**: `%LOCALAPPDATA%\vcpkg\archives`
-   **Linux/macOS**: `~/.cache/vcpkg/archives`

你可以通过查看这些目录，发现里面存满了 `.zip` (Windows) 或 `.tar.gz` (Unix) 文件。当你清理项目目录并重新执行 `vcpkg install` 时，这些缓存会瞬间完成部署。

## 远程缓存方案

为了让团队成员共享编译成果，或者加速 GitHub Actions 等 CI 流程，我们需要使用远程缓存。vcpkg 支持多种后端：

1.  **NuGet**: 它是 vcpkg 最推荐的 CI 缓存方式。因为 GitHub Packages 原生支持 NuGet，而且可以很好地处理大文件。
2.  **GitHub Packages**: 配合 NuGet 使用，这是目前大多数开源项目的首选。
3.  **HTTP/HTTPS**: 如果你有私有的静态文件服务器，可以使用这个模式。
4.  **Azure Blob Storage / AWS S3 / GCS**: 适合在云原生环境下使用。

## 环境变量 VCPKG_BINARY_SOURCES

二进制缓存的行为通过 `VCPKG_BINARY_SOURCES` 环境变量或 `vcpkg-configuration.json` 进行配置。

其语法格式为：`VCPKG_BINARY_SOURCES="<源1>;<源2>..."`

常见的配置示例：
-   `default`: 使用本地默认缓存。
-   `clear`: 清除之前所有的缓存源配置。
-   `nuget,<URL>,<User>,<Token>`: 配置一个 NuGet 源。

## 在 GitHub Actions 中配置 NuGet 二进制缓存

这是 KrKr2 项目目前采用的方案，能大幅缩短 CI 的等待时间。

### 第一步：获取 GitHub Token

GitHub Actions 自动提供一个 `${{ secrets.GITHUB_TOKEN }}`，权限足够。

### 第二步：配置 CI 脚本

在你的 `.github/workflows/ci.yml` 中添加如下步骤：

```yaml
- name: 配置 vcpkg 二进制缓存
  shell: bash
  run: |
    vcpkg_token="${{ secrets.GITHUB_TOKEN }}"
    vcpkg_owner="${{ github.repository_owner }}"
    
    # 启用 NuGet 缓存
    echo "VCPKG_BINARY_SOURCES=clear;nuget,https://nuget.pkg.github.com/$vcpkg_owner/index.json,readwrite" >> $GITHUB_ENV
    
    # 向 NuGet 客户端添加认证信息 (Linux/macOS)
    mono /usr/local/bin/nuget.exe sources add -Name "GitHub" \
      -Source "https://nuget.pkg.github.com/$vcpkg_owner/index.json" \
      -UserName "$vcpkg_owner" -Password "$vcpkg_token" \
      -StorePasswordInClearText
```

## 缓存失效与调试

如果你发现明明编译过却还是发生了 Cache Miss，可能是由以下原因导致的：

-   **环境变量不一致**：例如，一个开发者在编译时开启了 `LTO`（Link Time Optimization），而另一个开发者没有。这会改变生成的二进制文件的指纹。
-   **vcpkg 版本更新**：vcpkg 核心代码的升级（尤其是 Hash 算法的变更）会导致旧缓存失效。
-   **Triplet 的微小差异**：即使两个 Triplet 名字一样，如果其中一个 Triplet 文件的内容在这一周被修改了，之前的缓存也会失效。

你可以通过添加 `--debug` 标志来调试缓存匹配过程：

```bash
vcpkg install zlib --debug
```

在输出中寻找 `[DEBUG] Binary cache check...` 字样，它会告诉你 vcpkg 查找了哪些路径，以及为什么没匹配上。

## CI 环境中的最佳实践

1.  **分层缓存**：将 `VCPKG_BINARY_SOURCES` 设置为 `clear;nuget,...;default`。这表示先查 NuGet 远程缓存，没查到再查本地磁盘缓存。
2.  **清理老旧缓存**：定期清理 NuGet 上的历史包，以免产生不必要的费用。
3.  **只读缓存（对于非主分支）**：在 Pull Request 的 CI 中，可以设置缓存源为 `read` 模式，防止不合规的代码生成错误的缓存覆盖了官方缓存。

## 练习题与答案

### 题目 1：哈希值（Hash）的组成部分

以下哪些因素会影响 vcpkg 的二进制缓存哈希值？（多选）
A. 库的源码版本
B. `portfile.cmake` 中的编译选项
C. 当前计算机的 CPU 序列号
D. 目标 Triplet 的配置（如 x64-windows）

<details>
<summary>查看答案</summary>

答案是：A、B、D。
-   **A (源码版本)** 和 **B (编译脚本)** 直接决定了编译过程。
-   **D (Triplet)** 决定了目标平台。
-   **C (CPU 序列号)** 不会影响哈希值，只要指令集架构相同（如都是 x64），不同品牌的 CPU 生成的缓存通常是通用的。

</details>

### 题目 2：VCPKG_BINARY_SOURCES 的配置顺序

假设你配置了 `VCPKG_BINARY_SOURCES="clear;nuget,URL,user,pass;default"`。当 vcpkg 安装库时，它会如何执行缓存查找和存储？

<details>
<summary>查看答案</summary>

1.  首先清除所有默认源。
2.  **查找阶段**：先去远程 NuGet 源搜索。如果找到，则直接下载。如果没找到，再去本地 `default` 路径搜索。
3.  **存储阶段**：如果最终发生了编译，编译完成后会将产物同时推送到远程 NuGet 源和本地 `default` 路径（取决于权限设置）。

</details>

### 题目 3：缓存失效的场景

如果你在本地修改了 `vcpkg-triplets/arm64-android.cmake` 文件，将 `VCPKG_CMAKE_SYSTEM_VERSION` 从 28 改成了 30。此时执行 `vcpkg install`，之前的二进制缓存是否还能命中？为什么？

<details>
<summary>查看答案</summary>

**不能命中**。
因为 Triplet 文件的内容是哈希计算的重要组成部分。当你修改了 API 版本，编译出的库针对的 Android 系统底层调用可能已经发生了变化，因此哈希值会重新生成，导致之前的缓存全部失效。这保证了编译产物的正确性和安全性。

</details>
