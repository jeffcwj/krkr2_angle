# 跨平台基础：操作系统差异

> **所属模块：** P03-跨平台 C++ 开发
> **前置知识：** [P01-现代 CMake 与构建工具链](../../P01-现代CMake与构建工具链/)、C++ 基础（STL、面向对象）
> **预计阅读时间：** 35 分钟

## 本节目标

读完本节后，你将能够：
1. 列举 Windows、Linux、macOS 和 Android 四个平台在文件系统、线程、网络、字符编码方面的核心差异
2. 理解字节序（Endianness）对跨平台数据交换的影响，并编写安全的字节序转换代码
3. 知晓 C++ 基本数据类型在不同平台上的大小差异，并使用固定宽度整数类型避免陷阱
4. 理解 UTF-8 / UTF-16 / Shift-JIS 等编码在不同操作系统上的默认行为差异
5. 在后续阅读 KrKr2 源码时，能识别哪些代码是为了处理跨平台差异而存在的

## 为什么需要关心跨平台差异

如果你只写过 Windows 或只写过 Linux 程序，可能从未遇到过以下问题：

- 路径分隔符是 `\` 还是 `/`？
- `long` 类型是 4 字节还是 8 字节？
- 文件名是否区分大小写？
- 默认字符编码是 UTF-8 还是 GBK/Shift-JIS？
- 网络数据的字节序是大端还是小端？

这些看似细小的差异，在跨平台项目中会演变成**编译失败、运行崩溃、数据损坏**等严重问题。KrKr2 模拟器在四个平台上运行同一份 C++ 代码，因此它的代码库中充斥着处理这些差异的逻辑。理解这些差异，是阅读和修改项目代码的前提。

## 文件系统差异

文件系统是跨平台开发中差异最大、最容易踩坑的领域。

### 路径分隔符

| 平台 | 路径分隔符 | 示例路径 |
|------|-----------|----------|
| Windows | `\`（反斜杠） | `C:\Users\dev\project\src\main.cpp` |
| Linux | `/`（正斜杠） | `/home/dev/project/src/main.cpp` |
| macOS | `/`（正斜杠） | `/Users/dev/project/src/main.cpp` |
| Android | `/`（正斜杠） | `/data/data/com.example/files/main.cpp` |

Windows 实际上也支持正斜杠 `/`，大多数 Windows API 能正确处理。但某些旧版 API 和命令行工具（如 `cmd.exe`）可能不识别正斜杠。

**跨平台安全写法：**

```cpp
#include <filesystem>
#include <iostream>

int main() {
    // std::filesystem::path 自动处理路径分隔符
    std::filesystem::path base = "/home/dev/project"; // 在 Windows 上也能工作
    std::filesystem::path file = base / "src" / "main.cpp"; // 用 / 运算符拼接

    std::cout << "原生路径: " << file.string() << std::endl;
    // Windows 输出: \home\dev\project\src\main.cpp
    // Linux 输出:   /home/dev/project/src/main.cpp

    // 获取各部分
    std::cout << "文件名: " << file.filename() << std::endl;       // main.cpp
    std::cout << "扩展名: " << file.extension() << std::endl;      // .cpp
    std::cout << "父目录: " << file.parent_path() << std::endl;    // .../src

    return 0;
}
```

**KrKr2 中的实际做法：** 项目使用 `TVPGetLocallyAccessibleName()` 等函数将内部路径转换为平台原生路径。在 `cpp/core/base/` 的归档模块中，XP3 归档文件内部始终使用 `/` 作为路径分隔符，读取后按目标平台转换。

### 文件名大小写敏感性

| 平台 | 大小写敏感 | 说明 |
|------|-----------|------|
| Windows | ❌ 不敏感 | `FILE.txt` 和 `file.txt` 是同一个文件 |
| Linux | ✅ 敏感 | `FILE.txt` 和 `file.txt` 是两个不同的文件 |
| macOS | ❌ 默认不敏感 | HFS+/APFS 默认不区分大小写（可配置） |
| Android | ✅ 敏感 | 基于 Linux 内核，ext4 区分大小写 |

这是跨平台项目最常见的 Bug 来源之一：

```cpp
// 这段代码在 Windows/macOS 上能正常运行，但在 Linux/Android 上打开失败
std::ifstream file("Data/Scripts/startup.tjs"); // 实际文件名是 Startup.TJS
```

**KrKr2 的处理方式：** KiriKiri 引擎原版是 Windows 独占的，大量脚本和资源文件的引用不区分大小写。KrKr2 在非 Windows 平台上必须实现大小写不敏感的文件查找。通常做法是：

```cpp
// 伪代码：大小写不敏感文件查找
std::string findFileInsensitive(const std::string& dir, const std::string& name) {
    // 将目标文件名转为小写
    std::string lowerName = toLower(name);

    // 遍历目录中的所有文件
    for (const auto& entry : std::filesystem::directory_iterator(dir)) {
        std::string entryName = entry.path().filename().string();
        if (toLower(entryName) == lowerName) {
            return entry.path().string(); // 找到匹配文件
        }
    }
    return ""; // 未找到
}
```

### 最大路径长度

| 平台 | 默认最大路径 | 扩展方式 |
|------|-------------|----------|
| Windows | 260 字符（MAX_PATH） | 使用 `\\?\` 前缀可达 32,767 字符 |
| Linux | 4,096 字符（PATH_MAX） | 通常足够 |
| macOS | 1,024 字符（PATH_MAX） | 通常足够 |
| Android | 4,096 字符 | 同 Linux |

Windows 的 260 字符限制是一个臭名昭著的历史遗留问题。KrKr2 项目在 vcpkg 构建时，嵌套依赖路径很容易超过此限制：

```
C:\Users\Administrator\Downloads\code\KR2\krkr2\out\build\windows-debug\vcpkg_installed\x64-windows\share\opencv4\OpenCVModules-debug.cmake
```

上面这个路径已经有 140+ 字符，加上更深的子路径就可能超限。

**推荐做法：**
- Windows 上将项目放在短路径下（如 `C:\dev\kr2\`）
- 在 CMake 中设置较短的构建目录：`cmake --preset=... -B out`

### 文件锁定行为

| 平台 | 默认锁定行为 |
|------|-------------|
| Windows | 文件被打开时不能删除或重命名 |
| Linux/macOS/Android | 文件被打开后仍可删除（inode 引用计数机制） |

```cpp
// Windows 特有问题：
std::ofstream out("log.txt");
out << "Hello";
// 此时另一个进程无法删除或重命名 log.txt
// 在 Linux 上可以正常删除（文件在关闭后才真正释放空间）
```

## 线程模型差异

### 线程 API

| 平台 | 原生线程 API | C++ 标准线程 |
|------|-------------|-------------|
| Windows | Win32 线程 (`CreateThread`, `WaitForSingleObject`) | `std::thread`（C++11） |
| Linux | POSIX 线程 (`pthread_create`, `pthread_join`) | `std::thread`（C++11） |
| macOS | POSIX 线程 + GCD (Grand Central Dispatch) | `std::thread`（C++11） |
| Android | POSIX 线程 + Java Thread (JNI) | `std::thread`（C++11） |

C++11 引入的 `std::thread` 大幅简化了跨平台线程编程。但某些场景仍需使用平台特定 API：

```cpp
#include <thread>
#include <mutex>
#include <iostream>

// 跨平台线程：使用 C++ 标准库
std::mutex mtx;
int counter = 0;

void worker(int id) {
    for (int i = 0; i < 1000; ++i) {
        std::lock_guard<std::mutex> lock(mtx); // RAII 锁
        ++counter;
    }
    std::cout << "Worker " << id << " done\n";
}

int main() {
    std::thread t1(worker, 1);
    std::thread t2(worker, 2);

    t1.join();
    t2.join();

    std::cout << "Counter: " << counter << std::endl; // 2000
    return 0;
}
```

### 线程优先级

不同平台的线程优先级模型完全不同：

| 平台 | 优先级范围 | API |
|------|-----------|-----|
| Windows | 0-31（7 级名称常量） | `SetThreadPriority(handle, THREAD_PRIORITY_ABOVE_NORMAL)` |
| Linux | -20 到 19（nice 值）或 0-99（实时） | `pthread_setschedparam()` 或 `nice()` |
| macOS | 同 Linux（POSIX） | `pthread_setschedparam()` |
| Android | 同 Linux + Android 调度组 | `setpriority()` 或 `Process.setThreadPriority()` (Java) |

`std::thread` 没有提供设置优先级的跨平台方法，必须使用条件编译：

```cpp
#include <thread>

void setThreadPriority(std::thread& t, int priority) {
#ifdef _WIN32
    // Windows: 使用 Win32 API
    HANDLE handle = t.native_handle();
    SetThreadPriority(handle, THREAD_PRIORITY_ABOVE_NORMAL);
#elif defined(__linux__) || defined(__APPLE__)
    // POSIX: 使用 pthread API
    sched_param param;
    param.sched_priority = priority;
    pthread_setschedparam(t.native_handle(), SCHED_FIFO, &param);
#endif
}
```

### Android 的特殊线程限制

Android 上的 C++ 线程有特殊限制：

1. **JNI 线程附加**：C++ 创建的线程无法直接调用 Java API，必须先 `AttachCurrentThread()`
2. **UI 线程**：只有主线程（UI 线程）能操作 Android View 系统
3. **ANR 超时**：主线程阻塞超过 5 秒会触发 Application Not Responding 对话框

```cpp
// Android: C++ 线程调用 Java 必须先 Attach
#ifdef __ANDROID__
#include <jni.h>

extern JavaVM* g_jvm; // 全局 JavaVM 指针

void backgroundWork() {
    JNIEnv* env = nullptr;
    // 附加当前线程到 JVM
    g_jvm->AttachCurrentThread(&env, nullptr);

    // 现在可以通过 env 调用 Java 方法
    jclass cls = env->FindClass("com/example/MyClass");
    jmethodID method = env->GetStaticMethodID(cls, "doSomething", "()V");
    env->CallStaticVoidMethod(cls, method);

    // 用完必须 Detach
    g_jvm->DetachCurrentThread();
}
#endif
```

KrKr2 的 `platforms/android/cpp/krkr2_android.cpp` 中就包含了 JNI 线程管理的实际代码。

## 字节序（Endianness）

### 什么是字节序

多字节数据（如 `int32_t`）在内存中的存储顺序有两种：

| 字节序 | 存储方式 | 十六进制 0x12345678 在内存中 | 使用平台 |
|--------|---------|---------------------------|---------|
| **小端**（Little-Endian） | 低字节在低地址 | `78 56 34 12` | x86/x64、ARM（默认） |
| **大端**（Big-Endian） | 高字节在低地址 | `12 34 56 78` | 网络协议、MIPS、旧 PowerPC |

```
内存地址:    0x1000  0x1001  0x1002  0x1003
小端 (LE):   0x78    0x56    0x34    0x12     ← x86、ARM
大端 (BE):   0x12    0x34    0x56    0x78     ← 网络字节序
```

### 为什么要关心字节序

KrKr2 处理的二进制文件格式（XP3 归档、TLG 图像、PSB 动画等）都有固定的字节序。当这些文件在不同平台间传输时，如果读取代码不处理字节序，就会得到错误的数据。

```cpp
#include <cstdint>
#include <cstring>
#include <iostream>

// 错误的做法：直接强制转换，在大端平台上会得到错误结果
void readWrong(const uint8_t* data) {
    uint32_t value = *reinterpret_cast<const uint32_t*>(data); // 未定义行为 + 字节序问题
    std::cout << "value = " << value << std::endl;
}

// 正确的做法：显式按字节序读取
uint32_t readLE32(const uint8_t* data) {
    // 显式小端读取：无论平台字节序如何，结果都正确
    return static_cast<uint32_t>(data[0])
         | (static_cast<uint32_t>(data[1]) << 8)
         | (static_cast<uint32_t>(data[2]) << 16)
         | (static_cast<uint32_t>(data[3]) << 24);
}

uint32_t readBE32(const uint8_t* data) {
    // 显式大端读取
    return (static_cast<uint32_t>(data[0]) << 24)
         | (static_cast<uint32_t>(data[1]) << 16)
         | (static_cast<uint32_t>(data[2]) << 8)
         | static_cast<uint32_t>(data[3]);
}

int main() {
    // 模拟从文件读取的 4 字节数据（小端格式）
    uint8_t fileData[] = {0x78, 0x56, 0x34, 0x12};

    uint32_t le = readLE32(fileData); // 正确: 0x12345678
    uint32_t be = readBE32(fileData); // 也正确（只是解释不同）: 0x78563412

    std::cout << std::hex;
    std::cout << "小端解读: 0x" << le << std::endl; // 0x12345678
    std::cout << "大端解读: 0x" << be << std::endl; // 0x78563412

    return 0;
}
```

### 网络字节序转换

网络协议统一使用大端序（Big-Endian），因此跨平台网络编程中经常需要转换：

```cpp
#ifdef _WIN32
    #include <winsock2.h> // htonl, ntohl
#else
    #include <arpa/inet.h> // htonl, ntohl
#endif

// htonl: Host TO Network Long (32位，主机序→网络序/大端)
// ntohl: Network TO Host Long (32位，网络序→主机序)
// htons / ntohs: 16位版本

uint32_t hostValue = 0x12345678;
uint32_t netValue = htonl(hostValue);
// 在小端机器上: netValue = 0x78563412 (字节反转)
// 在大端机器上: netValue = 0x12345678 (不变)
```

### C++20 的 `std::endian`

C++20 引入了编译期检测字节序的标准方法：

```cpp
#include <bit>       // C++20
#include <iostream>

int main() {
    if constexpr (std::endian::native == std::endian::little) {
        std::cout << "当前平台: 小端序 (Little-Endian)" << std::endl;
    } else if constexpr (std::endian::native == std::endian::big) {
        std::cout << "当前平台: 大端序 (Big-Endian)" << std::endl;
    } else {
        std::cout << "当前平台: 混合字节序 (Mixed)" << std::endl;
    }
    return 0;
}
```

KrKr2 目标平台（x86/x64 和 ARM）全部是小端序，但代码中仍应使用显式字节序读写，以避免潜在移植问题。

## 数据类型大小差异

### 基本类型的陷阱

C++ 标准只规定了类型的**最小**大小，实际大小取决于平台和编译器：

| 类型 | Windows x64 (MSVC) | Linux x64 (GCC) | macOS arm64 (Clang) | Android arm64 (NDK) |
|------|-------------------|-----------------|--------------------|--------------------|
| `char` | 1 字节 | 1 字节 | 1 字节 | 1 字节 |
| `short` | 2 字节 | 2 字节 | 2 字节 | 2 字节 |
| `int` | 4 字节 | 4 字节 | 4 字节 | 4 字节 |
| **`long`** | **4 字节** | **8 字节** | **8 字节** | **8 字节** |
| `long long` | 8 字节 | 8 字节 | 8 字节 | 8 字节 |
| `size_t` | 8 字节 | 8 字节 | 8 字节 | 8 字节 |
| **`wchar_t`** | **2 字节 (UTF-16)** | **4 字节 (UTF-32)** | **4 字节 (UTF-32)** | **4 字节 (UTF-32)** |

**关键差异：**

1. **`long` 类型**：这是最臭名昭著的跨平台陷阱。Windows MSVC 上 `long` 是 4 字节（ILP32LL 模型），Linux/macOS 上是 8 字节（LP64 模型）。
2. **`wchar_t` 类型**：Windows 上是 2 字节（UTF-16），Unix-like 上是 4 字节（UTF-32）。这意味着 `std::wstring` 在不同平台上编码完全不同。

### 使用固定宽度整数类型

```cpp
#include <cstdint>
#include <iostream>

// ❌ 错误：使用 long，在不同平台上大小不同
long fileSize = 1234567890L;

// ✅ 正确：使用固定宽度类型，所有平台保证相同大小
int32_t fileSize32 = 1234567890;      // 始终 4 字节
int64_t fileSize64 = 1234567890LL;    // 始终 8 字节
uint8_t  byte = 0xFF;                 // 始终 1 字节无符号
uint16_t word = 0xFFFF;               // 始终 2 字节无符号
uint32_t dword = 0xFFFFFFFF;          // 始终 4 字节无符号

int main() {
    std::cout << "int32_t: " << sizeof(int32_t) << " bytes\n";  // 4
    std::cout << "int64_t: " << sizeof(int64_t) << " bytes\n";  // 8
    std::cout << "long:    " << sizeof(long) << " bytes\n";      // 4 或 8
    std::cout << "wchar_t: " << sizeof(wchar_t) << " bytes\n";   // 2 或 4
    return 0;
}
```

### 指针大小与对齐

| 平台 | 指针大小 | 默认对齐 |
|------|---------|---------|
| Windows x64 | 8 字节 | 16 字节 |
| Linux x64 | 8 字节 | 16 字节 |
| macOS arm64 | 8 字节 | 16 字节 |
| Android arm64 | 8 字节 | 16 字节 |
| Android arm32 | 4 字节 | 8 字节 |

KrKr2 目前支持 arm64 和 x86_64，不再支持 32 位平台，因此指针大小统一为 8 字节。但在读取二进制文件时，仍需注意结构体对齐问题：

```cpp
#include <cstdint>
#include <iostream>

// 文件头结构（需要精确控制内存布局）
#pragma pack(push, 1) // 关闭对齐填充
struct XP3Header {
    uint8_t  magic[11];   // "XP3\r\n\x1a\x0a\x8b\x67\x01"
    uint64_t indexOffset;  // 索引表偏移量
};
#pragma pack(pop)       // 恢复默认对齐

int main() {
    // 有 #pragma pack(1) 时: sizeof = 11 + 8 = 19 字节
    // 无 #pragma pack(1) 时: sizeof = 11 + 5(填充) + 8 = 24 字节
    std::cout << "XP3Header size: " << sizeof(XP3Header) << std::endl;
    return 0;
}
```

## 字符编码差异

### 各平台默认编码

| 平台 | 文件系统编码 | 控制台编码 | `std::string` 通常编码 |
|------|------------|-----------|----------------------|
| Windows | UTF-16 (NTFS) | 系统区域设置（中文=GBK，日文=Shift-JIS） | 取决于代码页（ANSI） |
| Linux | UTF-8 | UTF-8 | UTF-8 |
| macOS | UTF-8 (APFS) | UTF-8 | UTF-8 |
| Android | UTF-8 | UTF-8 | UTF-8 |

Windows 是唯一一个默认不使用 UTF-8 的主流平台。这对 KrKr2 尤为重要，因为原版 KiriKiri 引擎使用 Shift-JIS 编码（日文 Windows 的默认代码页），而 KrKr2 需要在所有平台上正确处理这些日文文本。

### Windows 的 Wide API vs Narrow API

Windows 系统 API 有两套版本：

```cpp
// Narrow API (A 后缀) — 使用当前代码页的 char*
CreateFileA("file.txt", ...);  // 受代码页影响，非 ASCII 字符可能乱码

// Wide API (W 后缀) — 使用 UTF-16 的 wchar_t*
CreateFileW(L"文件.txt", ...); // 推荐：支持所有 Unicode 字符
```

在 Windows 上，**始终使用 Wide (W) 版本 API** 是跨平台项目的最佳实践。但这意味着你需要在 UTF-8（项目内部编码）和 UTF-16（Windows API）之间转换：

```cpp
#include <string>

#ifdef _WIN32
#include <windows.h>

// UTF-8 → UTF-16 (Windows Wide)
std::wstring utf8ToWide(const std::string& utf8) {
    if (utf8.empty()) return {};
    int wlen = MultiByteToWideChar(CP_UTF8, 0, utf8.c_str(), -1, nullptr, 0);
    std::wstring wide(wlen - 1, L'\0'); // -1 排除 null 终止符
    MultiByteToWideChar(CP_UTF8, 0, utf8.c_str(), -1, wide.data(), wlen);
    return wide;
}

// UTF-16 → UTF-8
std::string wideToUtf8(const std::wstring& wide) {
    if (wide.empty()) return {};
    int len = WideCharToMultiByte(CP_UTF8, 0, wide.c_str(), -1, nullptr, 0, nullptr, nullptr);
    std::string utf8(len - 1, '\0');
    WideCharToMultiByte(CP_UTF8, 0, wide.c_str(), -1, utf8.data(), len, nullptr, nullptr);
    return utf8;
}
#endif
```

### KrKr2 中的编码处理

KrKr2 在 `vcpkg.json` 中依赖了 `icu` 库用于国际化和编码转换。项目编译时定义了 `USE_UNICODE_FSTRING` 宏，表示内部使用 Unicode 字符串。`cpp/core/tjs2/` 中的 TJS2 脚本引擎需要同时处理 Shift-JIS（原版脚本）和 UTF-8/UTF-16 编码。

## 常见错误及解决方案

### 错误 1：硬编码路径分隔符

```cpp
// ❌ 错误：硬编码反斜杠，Linux/macOS/Android 上路径无效
std::string path = "data\\scripts\\startup.tjs";

// ✅ 正确：使用 std::filesystem::path
std::filesystem::path path = std::filesystem::path("data") / "scripts" / "startup.tjs";
```

### 错误 2：用 `long` 存储文件大小

```cpp
// ❌ 错误：Windows 上 long 是 4 字节，文件超过 2GB 就溢出
long fileSize = getFileSize("big_archive.xp3"); // Windows: 最大 2GB

// ✅ 正确：使用 int64_t 或 std::uintmax_t
std::uintmax_t fileSize = std::filesystem::file_size("big_archive.xp3");
```

### 错误 3：假设 `wchar_t` 是 UTF-16

```cpp
// ❌ 错误：在 Linux 上 wchar_t 是 4 字节 UTF-32，不是 UTF-16
std::wstring wstr = L"你好"; // Windows: 4 字节  Linux: 8 字节
size_t bytes = wstr.size() * sizeof(wchar_t); // 结果不同！

// ✅ 正确：使用 char16_t（始终 2 字节 UTF-16）或 char32_t（始终 4 字节 UTF-32）
std::u16string u16str = u"你好"; // 所有平台: 4 字节 (2 × 2)
std::u32string u32str = U"你好"; // 所有平台: 8 字节 (2 × 4)
```

## 本节小结

- **文件系统**：路径分隔符（`\` vs `/`）、大小写敏感性、最大路径长度、文件锁定行为在各平台间差异巨大。使用 `std::filesystem` 是最安全的跨平台方案
- **线程模型**：`std::thread` 提供了跨平台基础，但线程优先级、Android JNI 线程绑定等仍需平台特定代码
- **字节序**：KrKr2 的四个目标平台都是小端序，但处理二进制文件格式时仍应使用显式字节序读写函数
- **数据类型**：`long` 和 `wchar_t` 是最危险的跨平台类型，应使用 `<cstdint>` 中的固定宽度类型
- **字符编码**：Windows 默认非 UTF-8，使用 Wide API 和 UTF-8/UTF-16 转换；Unix-like 平台默认 UTF-8

## 练习题与答案

### 题目 1：平台特征判断

以下代码在 Windows x64 (MSVC) 和 Linux x64 (GCC) 上编译运行，分别输出什么？

```cpp
#include <iostream>
int main() {
    std::cout << sizeof(long) << " " << sizeof(wchar_t) << std::endl;
    return 0;
}
```

<details>
<summary>查看答案</summary>

- **Windows x64 (MSVC)**：输出 `4 2`
  - `long` 在 MSVC 中始终是 4 字节（ILP32LL 数据模型）
  - `wchar_t` 在 Windows 上是 2 字节（UTF-16）
- **Linux x64 (GCC)**：输出 `8 4`
  - `long` 在 64 位 Linux 中是 8 字节（LP64 数据模型）
  - `wchar_t` 在 Linux 上是 4 字节（UTF-32）

**教训：** 永远不要用 `long` 来存储需要跨平台传输的数据，使用 `int32_t` 或 `int64_t`。

</details>

### 题目 2：字节序转换

编写一个函数 `readLE16`，从 `uint8_t*` 指针读取一个小端序的 16 位无符号整数。要求在任何字节序的平台上都能正确工作。

<details>
<summary>查看答案</summary>

```cpp
#include <cstdint>

uint16_t readLE16(const uint8_t* data) {
    // 低字节在低地址（小端），手动按位组装
    return static_cast<uint16_t>(data[0])
         | (static_cast<uint16_t>(data[1]) << 8);
}

// 测试
#include <iostream>
int main() {
    uint8_t data[] = {0x34, 0x12}; // 小端表示的 0x1234
    uint16_t value = readLE16(data);
    std::cout << std::hex << "0x" << value << std::endl; // 输出: 0x1234
    return 0;
}
```

**关键点：**
- 不使用 `reinterpret_cast`（避免未定义行为和对齐问题）
- 显式按字节组装，不依赖平台字节序
- 在大端和小端平台上都能得到正确结果

</details>

### 题目 3：跨平台路径处理

以下代码在 Linux 上运行时，有什么问题？如何修复？

```cpp
#include <fstream>
std::ifstream file("C:\\Data\\Scripts\\startup.tjs");
if (file.is_open()) {
    // 处理文件
}
```

<details>
<summary>查看答案</summary>

**问题：**
1. `C:\Data\Scripts\startup.tjs` 是 Windows 绝对路径，Linux 上不存在 `C:` 盘符
2. 即使路径存在，`\D`、`\S`、`\s` 在字符串中可能被解释为转义字符（虽然这几个不是有效转义，编译器会警告）
3. Linux 文件系统区分大小写，`startup.tjs` 和 `Startup.TJS` 是不同文件

**修复方案：**

```cpp
#include <filesystem>
#include <fstream>

namespace fs = std::filesystem;

// 1. 使用相对路径，避免平台绝对路径
fs::path scriptPath = fs::path("Data") / "Scripts" / "startup.tjs";

// 2. 如果文件名可能大小写不匹配，实现不敏感查找
std::ifstream openCaseInsensitive(const fs::path& dir, const std::string& filename) {
    std::string lowerTarget;
    for (char c : filename) lowerTarget += std::tolower(c);

    for (const auto& entry : fs::directory_iterator(dir)) {
        std::string entryName = entry.path().filename().string();
        std::string lowerEntry;
        for (char c : entryName) lowerEntry += std::tolower(c);

        if (lowerEntry == lowerTarget) {
            return std::ifstream(entry.path());
        }
    }
    return std::ifstream(); // 未找到，返回无效流
}

int main() {
    auto file = openCaseInsensitive("Data/Scripts", "startup.tjs");
    if (file.is_open()) {
        // 处理文件 — 即使实际文件名是 Startup.TJS 也能找到
    }
    return 0;
}
```

</details>
