## 为什么需要关心跨平台差异

### 一个真实的故事

假设你参与了一个游戏引擎项目。在 Windows 上开发一切正常——文件读取正常、脚本加载正常、多线程播放音频也没问题。然后你把代码推送到 CI，Linux 构建通过了，但运行测试时：

```
Test FAILED: Cannot open file "Data/Scripts/Startup.tjs"
   - File exists on disk as "data/scripts/startup.tjs"
```

怎么回事？文件明明在那里，为什么打不开？

答案是：**Linux 的文件名区分大小写**，而 Windows 不区分。你在代码里写的是 `Startup.tjs`，但实际文件名是 `startup.tjs`。在 Windows 上这完全没问题，但 Linux 认为这是两个不同的文件名。

这就是 KrKr2 项目面临的真实挑战。原版 KiriKiri 引擎只在 Windows 上运行，代码中大量使用不区分大小写的文件引用。当 KrKr2 要移植到 Linux、macOS 和 Android 时，这类问题遍布整个代码库。

**跨平台差异不是理论知识——它们是你在四平台项目中每天都会遇到的 Bug 来源。** 本节将逐一剖析这些差异，让你在遇到问题时能快速定位原因。

## 文件系统差异

文件系统是跨平台开发中差异最大、最容易踩坑的领域。我们从最常见的问题开始。

### 路径分隔符：`\` 还是 `/`？

**场景：** 你在 Windows 上硬编码了一个路径 `data\scripts\startup.tjs`，代码在 Windows 上完美运行。但在 Linux 上，这个路径被解释为一个名叫 `data\scripts\startup.tjs` 的单个文件名（因为 `\` 在 Linux 上不是路径分隔符），导致文件找不到。

| 平台 | 路径分隔符 | 示例路径 |
|------|-----------|----------|
| Windows | `\`（反斜杠） | `C:\Users\dev\project\src\main.cpp` |
| Linux | `/`（正斜杠） | `/home/dev/project/src/main.cpp` |
| macOS | `/`（正斜杠） | `/Users/dev/project/src/main.cpp` |
| Android | `/`（正斜杠） | `/data/data/com.example/files/main.cpp` |

好消息是：Windows 的大多数 API 也接受正斜杠 `/`。所以跨平台代码中**统一使用正斜杠**是最简单的方案。但更推荐的做法是使用 C++17 的 `std::filesystem::path`：

```cpp
#include <filesystem>
#include <iostream>

int main() {
    // std::filesystem::path 自动处理路径分隔符
    // 在任何平台上，这行代码都能正确工作
    std::filesystem::path base = "/home/dev/project";
    std::filesystem::path file = base / "src" / "main.cpp"; // 用 / 运算符拼接

    std::cout << "原生路径: " << file.string() << std::endl;
    // Windows 输出: \home\dev\project\src\main.cpp
    // Linux 输出:   /home/dev/project/src/main.cpp

    // 提取路径的各个部分
    std::cout << "文件名: " << file.filename() << std::endl;       // "main.cpp"
    std::cout << "扩展名: " << file.extension() << std::endl;      // ".cpp"
    std::cout << "父目录: " << file.parent_path() << std::endl;    // ".../src"
    std::cout << "无扩展: " << file.stem() << std::endl;           // "main"

    return 0;
}
```

**关键点：** `std::filesystem::path` 的 `/` 运算符不是数学除法——它是路径拼接。这个设计让路径操作代码读起来非常自然。

**KrKr2 中的实际做法：** 项目在 `cpp/core/base/` 的归档模块中，XP3 归档文件内部始终使用 `/` 作为路径分隔符。读取归档时按目标平台转换为原生路径。另外 `TVPGetLocallyAccessibleName()` 函数负责将引擎内部路径转换为平台原生路径。

### 文件名大小写敏感性

**场景：** KiriKiri 原版引擎是 Windows 独占的。脚本中引用资源文件时经常不注意大小写——`@image storage="BG01.png"` 可能引用的实际文件叫 `bg01.png`。在 Windows 上完全没问题，但移植到 Linux 后全部找不到。

| 平台 | 大小写敏感 | 说明 |
|------|-----------|------|
| Windows | ❌ 不敏感 | `FILE.txt` 和 `file.txt` 是同一个文件 |
| Linux | ✅ 敏感 | `FILE.txt` 和 `file.txt` 是两个不同的文件 |
| macOS | ❌ 默认不敏感 | HFS+/APFS 默认不区分（可配置为敏感） |
| Android | ✅ 敏感 | 基于 Linux 内核，ext4 文件系统区分大小写 |

这是跨平台项目中最常见的 Bug 来源之一。来看一个具体例子：

```cpp
#include <fstream>
#include <iostream>

int main() {
    // 实际文件名是 "startup.tjs"（全小写）
    // 下面这行在 Windows/macOS 上能正常打开，但在 Linux/Android 上失败
    std::ifstream file("Data/Scripts/Startup.tjs");

    if (file.is_open()) {
        std::cout << "文件打开成功" << std::endl;
    } else {
        std::cout << "文件打开失败！"  << std::endl;
        std::cout << "提示：检查文件名大小写是否完全匹配" << std::endl;
    }

    return 0;
}
```

**KrKr2 的解决方案：** 在非 Windows 平台上实现大小写不敏感的文件查找。核心思路是：将要查找的文件名转小写，然后遍历目录中所有文件，逐一比较小写化后的名称：

```cpp
#include <filesystem>
#include <algorithm>
#include <string>
#include <iostream>

// 将字符串转为小写（简化版，仅处理 ASCII）
std::string toLower(const std::string& s) {
    std::string result = s;
    std::transform(result.begin(), result.end(), result.begin(),
                   [](unsigned char c) { return std::tolower(c); });
    return result;
}

// 大小写不敏感文件查找
// 在 Linux/Android 上，文件系统区分大小写，
// 但 KiriKiri 脚本中经常不注意大小写
std::string findFileInsensitive(const std::string& dir,
                                 const std::string& name) {
    std::string lowerName = toLower(name);

    // 遍历目录中的所有条目
    for (const auto& entry : std::filesystem::directory_iterator(dir)) {
        std::string entryName = entry.path().filename().string();
        if (toLower(entryName) == lowerName) {
            return entry.path().string(); // 找到大小写不同但匹配的文件
        }
    }
    return ""; // 未找到
}

int main() {
    // 假设目录中有文件 "startup.tjs"
    // 但脚本引用的是 "Startup.TJS"
    std::string result = findFileInsensitive(".", "Startup.TJS");
    if (!result.empty()) {
        std::cout << "找到匹配文件: " << result << std::endl;
    } else {
        std::cout << "未找到匹配文件" << std::endl;
    }

    return 0;
}
```

> **性能提示：** 上面的 `directory_iterator` 遍历在每次查找时都要扫描整个目录，对于大目录（几千个文件）会很慢。KrKr2 的实际实现会在启动时把目录树缓存到一个 map 里（key 是小写文件名），之后的查找是 O(1) 的 hash lookup。

### 最大路径长度

**场景：** 你在 Windows 上用 vcpkg 安装依赖，构建路径嵌套得很深：

```
C:\Users\Administrator\Downloads\code\KR2\krkr2\out\build\windows-debug\vcpkg_installed\x64-windows\share\opencv4\OpenCVModules-debug.cmake
```

这个路径已经 140+ 字符了。Windows 默认的路径上限是 260 字符（`MAX_PATH` 常量），加上更深层的子路径就可能超限，导致构建失败。

| 平台 | 默认最大路径 | 扩展方式 |
|------|-------------|----------|
| Windows | 260 字符（MAX_PATH） | 使用 `\\?\` 前缀可达 32,767 字符 |
| Linux | 4,096 字符（PATH_MAX） | 通常足够 |
| macOS | 1,024 字符（PATH_MAX） | 通常足够 |
| Android | 4,096 字符 | 同 Linux |

```cpp
#include <cstdio>
#include <climits>

#ifdef _WIN32
#include <windows.h>
#endif

int main() {
    // 打印当前平台的路径长度限制
#ifdef _WIN32
    printf("Windows MAX_PATH = %d\n", MAX_PATH);           // 260
#else
    printf("PATH_MAX = %d\n", PATH_MAX);                    // Linux: 4096, macOS: 1024
#endif

    // Windows 上检查路径是否超限
    const char* longPath =
        "C:\\Users\\Administrator\\Downloads\\code\\KR2\\krkr2"
        "\\out\\build\\windows-debug\\vcpkg_installed"
        "\\x64-windows\\share\\opencv4\\OpenCVModules-debug.cmake";
    printf("示例路径长度: %zu 字符\n", strlen(longPath));

    return 0;
}
```

**推荐做法：**
- Windows 上将项目放在短路径下（如 `C:\dev\kr2\`），而不是 `C:\Users\Administrator\Downloads\code\KR2\krkr2\`
- 在 CMake 中设置较短的构建输出目录

### 文件锁定行为

**场景：** 你的程序打开了一个日志文件正在写入，用户想删除旧日志文件。在 Linux 上这完全没问题——文件被打开时仍然可以删除。但在 Windows 上，你会得到"文件正在使用，无法删除"的错误。

| 平台 | 默认锁定行为 |
|------|-------------|
| Windows | 文件被打开时默认不能删除或重命名 |
| Linux/macOS/Android | 文件被打开后仍可删除（inode 引用计数机制） |

这个差异源于两种文件系统的根本设计哲学不同：

```cpp
#include <fstream>
#include <filesystem>
#include <iostream>

int main() {
    // 创建一个测试文件
    {
        std::ofstream out("test_lock.txt");
        out << "Hello, World!" << std::endl;
    }

    // 打开文件保持句柄
    std::ifstream in("test_lock.txt");
    std::cout << "文件已打开" << std::endl;

    // 尝试删除
    std::error_code ec;
    std::filesystem::remove("test_lock.txt", ec);

    if (ec) {
        // Windows 上通常会进入这里
        std::cout << "删除失败: " << ec.message() << std::endl;
        std::cout << "（Windows 行为：文件被占用时不能删除）" << std::endl;
    } else {
        // Linux/macOS/Android 上通常会进入这里
        std::cout << "删除成功" << std::endl;
        std::cout << "（Unix 行为：文件的目录条目被删除，"
                  << "但已打开的文件句柄仍然有效，"
                  << "数据在最后一个句柄关闭后才真正释放）" << std::endl;
    }

    in.close();
    return 0;
}
```

**KrKr2 影响：** 引擎在热更新资源文件（如替换纹理、重新加载脚本）时，需要在 Windows 上先关闭所有文件句柄再替换文件，而在 Linux 上可以直接替换。这导致资源管理器的实现需要平台特定的逻辑。

## 线程模型差异

### 为什么线程也有跨平台问题？

**场景：** 你用 `std::thread` 写了一个后台解码线程，在 Windows 和 Linux 上运行良好。然后你在 Android 上测试，发现后台线程无法调用任何 Java API（比如读取 Android assets 目录中的文件）——程序直接崩溃。

原因是：Android 的 C++ 线程需要显式"附加"到 Java 虚拟机才能调用 JNI 方法。这种平台特有的限制，`std::thread` 的抽象层无法帮你处理。

### 线程创建 API 对比

| 平台 | 原生线程 API | C++ 标准线程 |
|------|-------------|-------------|
| Windows | Win32 线程 (`CreateThread`, `WaitForSingleObject`) | `std::thread`（C++11） |
| Linux | POSIX 线程 (`pthread_create`, `pthread_join`) | `std::thread`（C++11） |
| macOS | POSIX 线程 + GCD (Grand Central Dispatch) | `std::thread`（C++11） |
| Android | POSIX 线程 + Java Thread (JNI) | `std::thread`（C++11） |

C++11 的 `std::thread` 是跨平台线程编程的首选方案。它在底层封装了各平台的原生 API，但接口统一：

```cpp
#include <thread>
#include <mutex>
#include <iostream>
#include <vector>

// 跨平台线程示例：多线程计数器
// 这段代码在 Windows、Linux、macOS、Android 上行为完全一致

std::mutex mtx;         // 互斥锁：保护共享数据
int counter = 0;        // 共享计数器

void worker(int id) {
    for (int i = 0; i < 1000; ++i) {
        std::lock_guard<std::mutex> lock(mtx); // RAII 风格加锁
        ++counter;                              // 安全地修改共享数据
    }
    // lock_guard 在这里自动解锁（离开作用域时析构）
    std::cout << "Worker " << id << " 完成" << std::endl;
}

int main() {
    // 创建多个工作线程
    std::vector<std::thread> threads;
    for (int i = 0; i < 4; ++i) {
        threads.emplace_back(worker, i);  // 启动线程并传入参数
    }

    // 等待所有线程完成
    for (auto& t : threads) {
        t.join();  // join() 阻塞直到线程结束
    }

    std::cout << "最终计数: " << counter << std::endl; // 4000
    return 0;
}
```

### 线程优先级：每个平台都不一样

**场景：** 你的游戏引擎有一个音频解码线程，需要设置为高优先级以避免音频卡顿。但 `std::thread` 没有提供设置优先级的方法——因为各平台的优先级模型完全不同。

| 平台 | 优先级范围 | API |
|------|-----------|-----|
| Windows | 0-31（7 级名称常量） | `SetThreadPriority(handle, THREAD_PRIORITY_ABOVE_NORMAL)` |
| Linux | -20 到 19（nice 值）或 0-99（实时） | `pthread_setschedparam()` 或 `nice()` |
| macOS | 同 Linux（POSIX） | `pthread_setschedparam()` |
| Android | 同 Linux + Android 调度组 | `setpriority()` 或 `Process.setThreadPriority()` (Java) |

这就是需要条件编译的场景。`std::thread` 提供了 `native_handle()` 方法来获取底层平台句柄：

```cpp
#include <thread>
#include <iostream>

#ifdef _WIN32
#include <windows.h>
#elif defined(__linux__) || defined(__APPLE__)
#include <pthread.h>
#include <sched.h>
#endif

// 跨平台设置线程优先级
// std::thread 本身不提供优先级设置，需要用 native_handle() 获取平台句柄
void setHighPriority(std::thread& t) {
#ifdef _WIN32
    // Windows: 使用 Win32 API
    HANDLE handle = (HANDLE)t.native_handle();
    BOOL success = SetThreadPriority(handle, THREAD_PRIORITY_ABOVE_NORMAL);
    if (success) {
        std::cout << "[Windows] 线程优先级已设为 ABOVE_NORMAL" << std::endl;
    } else {
        std::cout << "[Windows] 设置线程优先级失败" << std::endl;
    }
#elif defined(__linux__) || defined(__APPLE__)
    // POSIX: 使用 pthread API
    sched_param param;
    param.sched_priority = 10; // 实时优先级 1-99
    int ret = pthread_setschedparam(t.native_handle(), SCHED_FIFO, &param);
    if (ret == 0) {
        std::cout << "[POSIX] 线程优先级已设为 SCHED_FIFO:10" << std::endl;
    } else {
        // 非 root 用户可能没有权限设置实时调度
        std::cout << "[POSIX] 设置失败（可能需要 root 权限），错误码: "
                  << ret << std::endl;
    }
#endif
}

void audioDecoder() {
    std::cout << "音频解码线程运行中..." << std::endl;
    // 模拟解码工作
    std::this_thread::sleep_for(std::chrono::milliseconds(100));
    std::cout << "音频解码完成" << std::endl;
}

int main() {
    std::thread decoder(audioDecoder);

    // 设置高优先级（音频线程不能被其他线程抢占）
    setHighPriority(decoder);

    decoder.join();
    return 0;
}
```

### Android 的特殊线程限制

Android 上的 C++ 线程有三个独特的限制，如果你不了解它们，移植代码到 Android 时会遇到非常困惑的崩溃：

**限制 1：JNI 线程附加。** C++ 创建的线程（`std::thread` 或 `pthread_create`）默认不在 JVM 中注册。如果在这种线程中调用任何 JNI 方法，会直接崩溃。

```cpp
#ifdef __ANDROID__
#include <jni.h>

extern JavaVM* g_jvm; // 全局 JavaVM 指针（在 JNI_OnLoad 中保存）

void backgroundWork() {
    JNIEnv* env = nullptr;

    // 必须先将线程附加到 JVM
    // 不附加就调用 env->FindClass() 等 JNI 方法会导致段错误
    jint result = g_jvm->AttachCurrentThread(&env, nullptr);
    if (result != JNI_OK) {
        // 附加失败，不能使用任何 JNI 功能
        return;
    }

    // 现在可以安全地调用 Java 方法
    jclass cls = env->FindClass("com/example/game/AssetManager");
    jmethodID method = env->GetStaticMethodID(cls, "loadAsset",
                                               "(Ljava/lang/String;)[B");
    // ... 调用方法 ...

    // 线程结束前必须解除附加
    // 忘记 Detach 会导致内存泄漏
    g_jvm->DetachCurrentThread();
}
#endif
```

**限制 2：UI 线程。** 只有 Android 主线程（UI 线程）能操作 View 系统。从 C++ 后台线程直接修改 UI 会崩溃。

**限制 3：ANR 超时。** 主线程阻塞超过 5 秒会触发 "Application Not Responding" 对话框，用户可以选择强制关闭应用。所以耗时操作（文件 I/O、网络、复杂计算）必须放到后台线程。

KrKr2 的 `platforms/android/cpp/krkr2_android.cpp` 中包含了完整的 JNI 线程管理代码。

## 常见错误与排查

### 错误 1：路径分隔符硬编码

```cpp
// ❌ 错误：硬编码反斜杠
// Linux 上 '\' 不是路径分隔符，整个字符串被当作一个文件名
std::string path = "data\\scripts\\startup.tjs";
FILE* f = fopen(path.c_str(), "r"); // Linux 上打开失败

// ✅ 方案 1：统一使用正斜杠（所有平台都接受）
std::string path = "data/scripts/startup.tjs";

// ✅ 方案 2：使用 std::filesystem（推荐，最安全）
auto path = std::filesystem::path("data") / "scripts" / "startup.tjs";
```

### 错误 2：假设文件名大小写不敏感

```cpp
// ❌ 错误：在 Windows 上测试通过，Linux 上文件找不到
std::ifstream script("Data/Scripts/CONFIG.TJS");
// 实际文件名是 "data/scripts/config.tjs"

// ✅ 正确：始终使用与文件系统中完全一致的大小写
std::ifstream script("data/scripts/config.tjs");
// 或者在文件查找层做大小写不敏感匹配（如 KrKr2 的做法）
```

### 错误 3：忘记 Android 线程的 JNI 附加

```cpp
// ❌ 错误：C++ 后台线程直接使用 JNIEnv，崩溃（段错误）
void decode() {
    JNIEnv* env; // 未附加的线程，env 是野指针
    env->FindClass("...");  // 崩溃！
}

// ✅ 正确：先 Attach，用完再 Detach
void decode() {
    JNIEnv* env = nullptr;
    g_jvm->AttachCurrentThread(&env, nullptr);
    // ... 使用 env ...
    g_jvm->DetachCurrentThread();
}
```

## 动手实践

### 实践 1：编写跨平台文件信息查看器

创建一个程序，显示指定文件/目录的平台相关信息：

```cpp
// file_info.cpp — 跨平台文件信息查看器
#include <filesystem>
#include <iostream>
#include <string>
#include <cstdio>

#ifdef _WIN32
#include <windows.h>
#endif

namespace fs = std::filesystem;

void showFileInfo(const fs::path& path) {
    std::cout << "=== 文件信息 ===" << std::endl;
    std::cout << "路径: " << path.string() << std::endl;
    std::cout << "原生路径: " << path.native() << std::endl;

    // 检查文件是否存在
    std::error_code ec;
    if (!fs::exists(path, ec)) {
        std::cout << "文件不存在!" << std::endl;
        return;
    }

    // 文件类型
    if (fs::is_regular_file(path)) {
        std::cout << "类型: 普通文件" << std::endl;
        std::cout << "大小: " << fs::file_size(path) << " 字节" << std::endl;
    } else if (fs::is_directory(path)) {
        std::cout << "类型: 目录" << std::endl;
        int count = 0;
        for (const auto& entry : fs::directory_iterator(path)) {
            count++;
        }
        std::cout << "包含 " << count << " 个条目" << std::endl;
    }

    // 最后修改时间
    auto ftime = fs::last_write_time(path);
    std::cout << "最后修改: (file_time_type 值)" << std::endl;

    // 权限信息
    auto perms = fs::status(path).permissions();
    std::cout << "权限: ";
    std::cout << ((perms & fs::perms::owner_read) != fs::perms::none ? "r" : "-");
    std::cout << ((perms & fs::perms::owner_write) != fs::perms::none ? "w" : "-");
    std::cout << ((perms & fs::perms::owner_exec) != fs::perms::none ? "x" : "-");
    std::cout << std::endl;
}

void showPlatformInfo() {
    std::cout << "\n=== 平台信息 ===" << std::endl;

#ifdef _WIN32
    std::cout << "平台: Windows" << std::endl;
    std::cout << "路径分隔符: \\" << std::endl;
    std::cout << "大小写敏感: 否" << std::endl;
    std::cout << "MAX_PATH: " << MAX_PATH << std::endl;
#elif defined(__linux__)
#ifdef __ANDROID__
    std::cout << "平台: Android" << std::endl;
#else
    std::cout << "平台: Linux" << std::endl;
#endif
    std::cout << "路径分隔符: /" << std::endl;
    std::cout << "大小写敏感: 是" << std::endl;
    std::cout << "PATH_MAX: " << PATH_MAX << std::endl;
#elif defined(__APPLE__)
    std::cout << "平台: macOS" << std::endl;
    std::cout << "路径分隔符: /" << std::endl;
    std::cout << "大小写敏感: 默认否（可配置）" << std::endl;
    std::cout << "PATH_MAX: " << PATH_MAX << std::endl;
#endif
}

int main() {
    showPlatformInfo();
    std::cout << std::endl;

    // 查看当前目录
    showFileInfo(fs::current_path());

    return 0;
}
```

### 实践 2：多线程计数器验证

编写一个程序验证 `std::mutex` 的正确性——先不加锁观察数据竞争，再加锁验证结果：

```cpp
// thread_race.cpp — 验证数据竞争与互斥锁
#include <thread>
#include <mutex>
#include <iostream>
#include <vector>

int counter_unsafe = 0;  // 不受保护的计数器
int counter_safe = 0;    // 受互斥锁保护的计数器
std::mutex mtx;

void unsafeIncrement(int iterations) {
    for (int i = 0; i < iterations; ++i) {
        counter_unsafe++;  // 数据竞争！多线程同时修改
    }
}

void safeIncrement(int iterations) {
    for (int i = 0; i < iterations; ++i) {
        std::lock_guard<std::mutex> lock(mtx);
        counter_safe++;    // 受互斥锁保护，线程安全
    }
}

int main() {
    const int numThreads = 8;
    const int iterations = 100000;

    // 测试 1：无锁（有数据竞争）
    {
        std::vector<std::thread> threads;
        for (int i = 0; i < numThreads; ++i) {
            threads.emplace_back(unsafeIncrement, iterations);
        }
        for (auto& t : threads) t.join();

        std::cout << "无锁结果:   " << counter_unsafe
                  << " (期望: " << numThreads * iterations << ")" << std::endl;
        // 结果通常小于期望值，因为存在数据竞争
    }

    // 测试 2：有锁（线程安全）
    {
        std::vector<std::thread> threads;
        for (int i = 0; i < numThreads; ++i) {
            threads.emplace_back(safeIncrement, iterations);
        }
        for (auto& t : threads) t.join();

        std::cout << "有锁结果:   " << counter_safe
                  << " (期望: " << numThreads * iterations << ")" << std::endl;
        // 结果总是等于期望值
    }

    return 0;
}
```

KrKr2 项目中处理文件系统和线程跨平台差异的代码分布在以下位置：

相关文件：
- `cpp/core/environ/` — 平台抽象层根目录，包含各平台的适配实现
- `cpp/core/environ/win32/Platform.cpp` — Windows 平台特有代码（文件路径转换、宽字符 API）
- `cpp/core/environ/android/AndroidUtils.cpp` — Android 平台特有代码（JNI 文件访问、assets 读取）
- `cpp/core/base/StorageIntf.cpp` — 跨平台文件存储接口，内部处理路径分隔符转换
- `cpp/core/utils/ThreadIntf.h` — 线程工具的跨平台封装
- `platforms/android/cpp/krkr2_android.cpp` — Android JNI 入口，包含 JavaVM 指针保存和线程附加逻辑

## 本节小结

- **路径分隔符**：Windows 用 `\`，其他平台用 `/`。跨平台代码统一使用 `/` 或 `std::filesystem::path`
- **文件名大小写**：Windows 和 macOS 默认不敏感，Linux 和 Android 敏感。KrKr2 在 Linux/Android 上实现大小写不敏感查找来兼容原版 KiriKiri 脚本
- **路径长度上限**：Windows 默认 260 字符（臭名昭著的 MAX_PATH），其他平台 1024-4096 字符。Windows 项目应放在短路径下
- **文件锁定**：Windows 上被打开的文件不能删除/重命名，Linux/macOS/Android 可以（inode 引用计数）
- **线程创建**：用 `std::thread` 可以跨平台。但优先级设置、Android JNI 附加等需要平台特定代码
- **Android 线程限制**：C++ 线程要调用 JNI 必须先 `AttachCurrentThread()`，主线程阻塞超过 5 秒会 ANR

### 题目 2：大小写不敏感查找

编写一个函数 `findInsensitive(const std::string& dir, const std::string& filename)`，在指定目录中查找与 `filename` 大小写不敏感匹配的文件。如果找到，返回实际文件名；否则返回空字符串。

<details>
<summary>查看答案</summary>

```cpp
#include <filesystem>
#include <algorithm>
#include <string>
#include <iostream>
#include <cctype>

namespace fs = std::filesystem;

std::string toLower(const std::string& s) {
    std::string result = s;
    std::transform(result.begin(), result.end(), result.begin(),
                   [](unsigned char c) { return std::tolower(c); });
    return result;
}

std::string findInsensitive(const std::string& dir,
                             const std::string& filename) {
    std::string target = toLower(filename);

    std::error_code ec;
    for (const auto& entry : fs::directory_iterator(dir, ec)) {
        if (entry.is_regular_file()) {
            std::string name = entry.path().filename().string();
            if (toLower(name) == target) {
                return name; // 返回文件系统中的实际名称
            }
        }
    }
    return "";
}

int main() {
    // 在当前目录查找 "README.MD"（不区分大小写）
    std::string result = findInsensitive(".", "README.MD");
    if (!result.empty()) {
        std::cout << "找到: " << result << std::endl;
    } else {
        std::cout << "未找到" << std::endl;
    }
    return 0;
}
```

### 题目 3：线程安全问题诊断

以下代码在单线程时输出正确，但多线程运行时结果不确定。找出问题并修复：

```cpp
#include <thread>
#include <vector>
#include <iostream>

int total = 0;

void addToTotal(int value) {
    for (int i = 0; i < value; ++i) {
        total = total + 1; // 问题在哪里？
    }
}

int main() {
    std::vector<std::thread> threads;
    for (int i = 0; i < 10; ++i) {
        threads.emplace_back(addToTotal, 10000);
    }
    for (auto& t : threads) t.join();
    std::cout << "Total: " << total << std::endl; // 期望 100000
    return 0;
}
```

<details>
<summary>查看答案</summary>

**问题：** `total = total + 1` 不是原子操作。多个线程同时读取 `total` 的值，各自加 1 后写回，导致部分增量丢失（数据竞争/race condition）。

**修复方案 1：使用互斥锁**

```cpp
#include <thread>
#include <mutex>
#include <vector>
#include <iostream>

int total = 0;
std::mutex mtx;

void addToTotal(int value) {
    for (int i = 0; i < value; ++i) {
        std::lock_guard<std::mutex> lock(mtx);
        total = total + 1;
    }
}

int main() {
    std::vector<std::thread> threads;
    for (int i = 0; i < 10; ++i) {
        threads.emplace_back(addToTotal, 10000);
    }
    for (auto& t : threads) t.join();
    std::cout << "Total: " << total << std::endl; // 始终输出 100000
    return 0;
}
```

**修复方案 2：使用原子变量（更高效）**

```cpp
#include <thread>
#include <atomic>
#include <vector>
#include <iostream>

std::atomic<int> total{0}; // 原子变量，线程安全

void addToTotal(int value) {
    for (int i = 0; i < value; ++i) {
        total.fetch_add(1, std::memory_order_relaxed);
    }
}

int main() {
    std::vector<std::thread> threads;
    for (int i = 0; i < 10; ++i) {
        threads.emplace_back(addToTotal, 10000);
    }
    for (auto& t : threads) t.join();
    std::cout << "Total: " << total.load() << std::endl; // 始终输出 100000
    return 0;
}
```

方案 2 更高效，因为 `std::atomic` 使用 CPU 的原子指令（如 x86 的 `lock xadd`），不需要操作系统级别的互斥锁。

</details>
