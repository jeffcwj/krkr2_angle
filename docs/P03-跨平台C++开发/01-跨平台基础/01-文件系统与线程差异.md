# 跨平台基础：操作系统差异

> **所属模块：** P03-跨平台 C++ 开发
> **前置知识：** [P01-现代 CMake 与构建工具链](../../P01-现代CMake与构建工具链/)、C++ 基础（STL、面向对象）
> **预计阅读时间：** 35 分钟

## 本节目标

读完本节后，你将能够：
1. 列举 Windows、Linux、macOS 和 Android 四个平台在文件系统、线程、网络、字符编码方面的核心差异
2. 理解字节序（Endianness）对跨平台数据交换的影响，并编写安全的字节序转换代码
3. 知晓 C++ 基本数据类型在不同平台上的大小差异，并使用固定宽度整数类型避免陷阱
4. 理解 UTF-8 / UTF-16 / Shift-JIS 等编码在不同操作系统上的默认行为差异
5. 在后续阅读 KrKr2 源码时，能识别哪些代码是为了处理跨平台差异而存在的

## 为什么需要关心跨平台差异

如果你只写过 Windows 或只写过 Linux 程序，可能从未遇到过以下问题：

- 路径分隔符是 `\` 还是 `/`？
- `long` 类型是 4 字节还是 8 字节？
- 文件名是否区分大小写？
- 默认字符编码是 UTF-8 还是 GBK/Shift-JIS？
- 网络数据的字节序是大端还是小端？

这些看似细小的差异，在跨平台项目中会演变成**编译失败、运行崩溃、数据损坏**等严重问题。KrKr2 模拟器在四个平台上运行同一份 C++ 代码，因此它的代码库中充斥着处理这些差异的逻辑。理解这些差异，是阅读和修改项目代码的前提。

## 文件系统差异

文件系统是跨平台开发中差异最大、最容易踩坑的领域。

### 路径分隔符

| 平台 | 路径分隔符 | 示例路径 |
|------|-----------|----------|
| Windows | `\`（反斜杠） | `C:\Users\dev\project\src\main.cpp` |
| Linux | `/`（正斜杠） | `/home/dev/project/src/main.cpp` |
| macOS | `/`（正斜杠） | `/Users/dev/project/src/main.cpp` |
| Android | `/`（正斜杠） | `/data/data/com.example/files/main.cpp` |

Windows 实际上也支持正斜杠 `/`，大多数 Windows API 能正确处理。但某些旧版 API 和命令行工具（如 `cmd.exe`）可能不识别正斜杠。

**跨平台安全写法：**

```cpp
#include <filesystem>
#include <iostream>

int main() {
    // std::filesystem::path 自动处理路径分隔符
    std::filesystem::path base = "/home/dev/project"; // 在 Windows 上也能工作
    std::filesystem::path file = base / "src" / "main.cpp"; // 用 / 运算符拼接

    std::cout << "原生路径: " << file.string() << std::endl;
    // Windows 输出: \home\dev\project\src\main.cpp
    // Linux 输出:   /home/dev/project/src/main.cpp

    // 获取各部分
    std::cout << "文件名: " << file.filename() << std::endl;       // main.cpp
    std::cout << "扩展名: " << file.extension() << std::endl;      // .cpp
    std::cout << "父目录: " << file.parent_path() << std::endl;    // .../src

    return 0;
}
```

**KrKr2 中的实际做法：** 项目使用 `TVPGetLocallyAccessibleName()` 等函数将内部路径转换为平台原生路径。在 `cpp/core/base/` 的归档模块中，XP3 归档文件内部始终使用 `/` 作为路径分隔符，读取后按目标平台转换。

### 文件名大小写敏感性

| 平台 | 大小写敏感 | 说明 |
|------|-----------|------|
| Windows | ❌ 不敏感 | `FILE.txt` 和 `file.txt` 是同一个文件 |
| Linux | ✅ 敏感 | `FILE.txt` 和 `file.txt` 是两个不同的文件 |
| macOS | ❌ 默认不敏感 | HFS+/APFS 默认不区分大小写（可配置） |
| Android | ✅ 敏感 | 基于 Linux 内核，ext4 区分大小写 |

这是跨平台项目最常见的 Bug 来源之一：

```cpp
// 这段代码在 Windows/macOS 上能正常运行，但在 Linux/Android 上打开失败
std::ifstream file("Data/Scripts/startup.tjs"); // 实际文件名是 Startup.TJS
```

**KrKr2 的处理方式：** KiriKiri 引擎原版是 Windows 独占的，大量脚本和资源文件的引用不区分大小写。KrKr2 在非 Windows 平台上必须实现大小写不敏感的文件查找。通常做法是：

```cpp
// 伪代码：大小写不敏感文件查找
std::string findFileInsensitive(const std::string& dir, const std::string& name) {
    // 将目标文件名转为小写
    std::string lowerName = toLower(name);

    // 遍历目录中的所有文件
    for (const auto& entry : std::filesystem::directory_iterator(dir)) {
        std::string entryName = entry.path().filename().string();
        if (toLower(entryName) == lowerName) {
            return entry.path().string(); // 找到匹配文件
        }
    }
    return ""; // 未找到
}
```

### 最大路径长度

| 平台 | 默认最大路径 | 扩展方式 |
|------|-------------|----------|
| Windows | 260 字符（MAX_PATH） | 使用 `\\?\` 前缀可达 32,767 字符 |
| Linux | 4,096 字符（PATH_MAX） | 通常足够 |
| macOS | 1,024 字符（PATH_MAX） | 通常足够 |
| Android | 4,096 字符 | 同 Linux |

Windows 的 260 字符限制是一个臭名昭著的历史遗留问题。KrKr2 项目在 vcpkg 构建时，嵌套依赖路径很容易超过此限制：

```
C:\Users\Administrator\Downloads\code\KR2\krkr2\out\build\windows-debug\vcpkg_installed\x64-windows\share\opencv4\OpenCVModules-debug.cmake
```

上面这个路径已经有 140+ 字符，加上更深的子路径就可能超限。

**推荐做法：**
- Windows 上将项目放在短路径下（如 `C:\dev\kr2\`）
- 在 CMake 中设置较短的构建目录：`cmake --preset=... -B out`

### 文件锁定行为

| 平台 | 默认锁定行为 |
|------|-------------|
| Windows | 文件被打开时不能删除或重命名 |
| Linux/macOS/Android | 文件被打开后仍可删除（inode 引用计数机制） |

```cpp
// Windows 特有问题：
std::ofstream out("log.txt");
out << "Hello";
// 此时另一个进程无法删除或重命名 log.txt
// 在 Linux 上可以正常删除（文件在关闭后才真正释放空间）
```

## 线程模型差异

### 线程 API

| 平台 | 原生线程 API | C++ 标准线程 |
|------|-------------|-------------|
| Windows | Win32 线程 (`CreateThread`, `WaitForSingleObject`) | `std::thread`（C++11） |
| Linux | POSIX 线程 (`pthread_create`, `pthread_join`) | `std::thread`（C++11） |
| macOS | POSIX 线程 + GCD (Grand Central Dispatch) | `std::thread`（C++11） |
| Android | POSIX 线程 + Java Thread (JNI) | `std::thread`（C++11） |

C++11 引入的 `std::thread` 大幅简化了跨平台线程编程。但某些场景仍需使用平台特定 API：

```cpp
#include <thread>
#include <mutex>
#include <iostream>

// 跨平台线程：使用 C++ 标准库
std::mutex mtx;
int counter = 0;

void worker(int id) {
    for (int i = 0; i < 1000; ++i) {
        std::lock_guard<std::mutex> lock(mtx); // RAII 锁
        ++counter;
    }
    std::cout << "Worker " << id << " done\n";
}

int main() {
    std::thread t1(worker, 1);
    std::thread t2(worker, 2);

    t1.join();
    t2.join();

    std::cout << "Counter: " << counter << std::endl; // 2000
    return 0;
}
```

### 线程优先级

不同平台的线程优先级模型完全不同：

| 平台 | 优先级范围 | API |
|------|-----------|-----|
| Windows | 0-31（7 级名称常量） | `SetThreadPriority(handle, THREAD_PRIORITY_ABOVE_NORMAL)` |
| Linux | -20 到 19（nice 值）或 0-99（实时） | `pthread_setschedparam()` 或 `nice()` |
| macOS | 同 Linux（POSIX） | `pthread_setschedparam()` |
| Android | 同 Linux + Android 调度组 | `setpriority()` 或 `Process.setThreadPriority()` (Java) |

`std::thread` 没有提供设置优先级的跨平台方法，必须使用条件编译：

```cpp
#include <thread>

void setThreadPriority(std::thread& t, int priority) {
#ifdef _WIN32
    // Windows: 使用 Win32 API
    HANDLE handle = t.native_handle();
    SetThreadPriority(handle, THREAD_PRIORITY_ABOVE_NORMAL);
#elif defined(__linux__) || defined(__APPLE__)
    // POSIX: 使用 pthread API
    sched_param param;
    param.sched_priority = priority;
    pthread_setschedparam(t.native_handle(), SCHED_FIFO, &param);
#endif
}
```

### Android 的特殊线程限制

Android 上的 C++ 线程有特殊限制：

1. **JNI 线程附加**：C++ 创建的线程无法直接调用 Java API，必须先 `AttachCurrentThread()`
2. **UI 线程**：只有主线程（UI 线程）能操作 Android View 系统
3. **ANR 超时**：主线程阻塞超过 5 秒会触发 Application Not Responding 对话框

```cpp
// Android: C++ 线程调用 Java 必须先 Attach
#ifdef __ANDROID__
#include <jni.h>

extern JavaVM* g_jvm; // 全局 JavaVM 指针

void backgroundWork() {
    JNIEnv* env = nullptr;
    // 附加当前线程到 JVM
    g_jvm->AttachCurrentThread(&env, nullptr);

    // 现在可以通过 env 调用 Java 方法
    jclass cls = env->FindClass("com/example/MyClass");
    jmethodID method = env->GetStaticMethodID(cls, "doSomething", "()V");
    env->CallStaticVoidMethod(cls, method);

    // 用完必须 Detach
    g_jvm->DetachCurrentThread();
}
#endif
```

KrKr2 的 `platforms/android/cpp/krkr2_android.cpp` 中就包含了 JNI 线程管理的实际代码。
