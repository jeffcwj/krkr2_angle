## 字节序（Endianness）

### 从一个 Bug 说起

假设你在开发一个游戏引擎，需要读取 XP3 归档文件。你在 Windows 上写了这样的代码：

```cpp
// 从文件读取一个 4 字节整数（文件中的索引偏移量）
uint32_t offset;
fread(&offset, sizeof(offset), 1, file);
printf("索引偏移量: %u\n", offset);  // Windows 上输出: 305419896 (正确)
```

在 Windows（x86/x64）上完美运行。但当你把同样的代码编译到一台大端 CPU 的机器上运行时：

```
索引偏移量: 2018915346  ← 完全错误的数字！
```

**为什么？** 因为文件里存储的是小端序的字节 `78 56 34 12`，x86 CPU 自然按小端序理解它（得到 0x12345678 = 305419896）。但大端 CPU 按大端序理解同样的字节（得到 0x78563412 = 2018915346）。

虽然 KrKr2 当前的目标平台（x86/x64 和 ARM）都是小端序，但理解字节序仍然重要——因为网络协议使用大端序，某些二进制格式可能混用字节序，并且养成好习惯能避免未来移植时的 Bug。

### 什么是字节序

当一个多字节的数据（比如 32 位整数 0x12345678）存储到内存中时，4 个字节的排列顺序有两种：

| 字节序 | 英文 | 排列方式 | 0x12345678 在内存中 | 使用者 |
|--------|------|---------|---------------------|--------|
| **小端** | Little-Endian | 低字节在低地址 | `78 56 34 12` | x86/x64、ARM（默认）、大多数现代 CPU |
| **大端** | Big-Endian | 高字节在低地址 | `12 34 56 78` | 网络协议、MIPS、旧 PowerPC、Java 字节码 |

用一个形象的比喻：想象你要把数字"一千二百三十四"写在纸上。

- **大端序**就像正常书写：从最高位开始写 `1 2 3 4`——先写"千"位
- **小端序**就像倒着写：从最低位开始写 `4 3 2 1`——先写"个"位

```
内存地址:     0x1000  0x1001  0x1002  0x1003
             ┌───┐   ┌───┐   ┌───┐   ┌───┐
小端 (LE):   │ 78│   │ 56│   │ 34│   │ 12│   ← x86、ARM
             └───┘   └───┘   └───┘   └───┘
             低字节          →        高字节
             ┌───┐   ┌───┐   ┌───┐   ┌───┐
大端 (BE):   │ 12│   │ 34│   │ 56│   │ 78│   ← 网络字节序
             └───┘   └───┘   └───┘   └───┘
             高字节          →        低字节
```

### 编写安全的字节序读写函数

**核心原则：** 永远不要用 `reinterpret_cast<uint32_t*>(data)` 直接读取二进制数据。这既有字节序问题，也违反了 C++ 的严格别名规则（Strict Aliasing），属于未定义行为。

正确做法是**逐字节组装**：

```cpp
#include <cstdint>
#include <cstdio>

// ===== 小端序读取函数 =====
// 这些函数无论在什么平台上运行，都能正确读取小端序数据

uint16_t readLE16(const uint8_t* data) {
    return static_cast<uint16_t>(data[0])
         | (static_cast<uint16_t>(data[1]) << 8);
}

uint32_t readLE32(const uint8_t* data) {
    return static_cast<uint32_t>(data[0])
         | (static_cast<uint32_t>(data[1]) << 8)
         | (static_cast<uint32_t>(data[2]) << 16)
         | (static_cast<uint32_t>(data[3]) << 24);
}

uint64_t readLE64(const uint8_t* data) {
    return static_cast<uint64_t>(data[0])
         | (static_cast<uint64_t>(data[1]) << 8)
         | (static_cast<uint64_t>(data[2]) << 16)
         | (static_cast<uint64_t>(data[3]) << 24)
         | (static_cast<uint64_t>(data[4]) << 32)
         | (static_cast<uint64_t>(data[5]) << 40)
         | (static_cast<uint64_t>(data[6]) << 48)
         | (static_cast<uint64_t>(data[7]) << 56);
}

// ===== 大端序读取函数 =====
// 用于网络协议或大端序格式的文件

uint16_t readBE16(const uint8_t* data) {
    return (static_cast<uint16_t>(data[0]) << 8)
         | static_cast<uint16_t>(data[1]);
}

uint32_t readBE32(const uint8_t* data) {
    return (static_cast<uint32_t>(data[0]) << 24)
         | (static_cast<uint32_t>(data[1]) << 16)
         | (static_cast<uint32_t>(data[2]) << 8)
         | static_cast<uint32_t>(data[3]);
}

// ===== 小端序写入函数 =====
// 将整数按小端序写入字节数组

void writeLE32(uint8_t* data, uint32_t value) {
    data[0] = static_cast<uint8_t>(value & 0xFF);
    data[1] = static_cast<uint8_t>((value >> 8) & 0xFF);
    data[2] = static_cast<uint8_t>((value >> 16) & 0xFF);
    data[3] = static_cast<uint8_t>((value >> 24) & 0xFF);
}

int main() {
    // 模拟从 XP3 文件读取的字节序列
    // 文件中存储的是小端序的 0x12345678
    uint8_t fileData[] = {0x78, 0x56, 0x34, 0x12};

    uint32_t correct = readLE32(fileData);
    uint32_t wrong   = readBE32(fileData);

    printf("小端序读取（正确）: 0x%08X = %u\n", correct, correct);
    // 输出: 0x12345678 = 305419896

    printf("大端序读取（错误）: 0x%08X = %u\n", wrong, wrong);
    // 输出: 0x78563412 = 2018915346

    // 验证写入：将 0xDEADBEEF 写成小端序
    uint8_t output[4];
    writeLE32(output, 0xDEADBEEF);
    printf("写入结果: %02X %02X %02X %02X\n",
           output[0], output[1], output[2], output[3]);
    // 输出: EF BE AD DE（小端序：低字节在前）

    return 0;
}
```

### 网络字节序转换

网络协议统一使用大端序（也叫"网络字节序"），这是早期互联网标准（RFC 1700）的规定。操作系统提供了一组转换函数：

```cpp
#include <cstdint>
#include <cstdio>

// 平台头文件差异——又一个需要条件编译的地方
#ifdef _WIN32
    #include <winsock2.h>  // htonl, ntohl, htons, ntohs
    #pragma comment(lib, "ws2_32.lib")
#else
    #include <arpa/inet.h> // htonl, ntohl, htons, ntohs
#endif

int main() {
    // htonl = Host TO Network Long（32位，主机序 → 网络序/大端）
    // ntohl = Network TO Host Long（32位，网络序 → 主机序）
    // htons / ntohs = 16位版本

    uint32_t hostValue = 0x12345678;
    uint32_t netValue = htonl(hostValue);

    printf("主机序: 0x%08X\n", hostValue);  // 0x12345678
    printf("网络序: 0x%08X\n", netValue);   // 0x78563412（小端机器上字节被反转）

    // 网络序转回主机序
    uint32_t backToHost = ntohl(netValue);
    printf("转回:   0x%08X\n", backToHost); // 0x12345678（又变回来了）

    // 16 位版本（常用于端口号）
    uint16_t port = 8080;
    uint16_t netPort = htons(port);
    printf("端口 %u → 网络序 0x%04X\n", port, netPort);

    return 0;
}
```

**函数名记忆法：**
- `h` = Host（主机）
- `to` = 转换方向
- `n` = Network（网络）
- `l` = Long（32位）/ `s` = Short（16位）

### C++20 的 `std::endian`

C++20 引入了编译期检测字节序的标准方法，终结了各种平台特定的宏定义判断：

```cpp
#include <cstdio>

// C++20 方式
#if __cplusplus >= 202002L
#include <bit>

void checkEndian_cpp20() {
    if constexpr (std::endian::native == std::endian::little) {
        printf("[C++20] 当前平台: 小端序 (Little-Endian)\n");
    } else if constexpr (std::endian::native == std::endian::big) {
        printf("[C++20] 当前平台: 大端序 (Big-Endian)\n");
    } else {
        printf("[C++20] 当前平台: 混合字节序 (Mixed-Endian)\n");
    }
}
#endif

// C++17 兼容方式（KrKr2 使用 C++17）
void checkEndian_portable() {
    // 运行时检测：将 1 存入 int，检查第一个字节
    uint32_t test = 1;
    uint8_t* bytes = reinterpret_cast<uint8_t*>(&test);

    if (bytes[0] == 1) {
        printf("[C++17] 当前平台: 小端序 (最低字节 0x01 在地址最低处)\n");
    } else {
        printf("[C++17] 当前平台: 大端序\n");
    }
}

int main() {
#if __cplusplus >= 202002L
    checkEndian_cpp20();
#endif
    checkEndian_portable();
    return 0;
}
```

> **KrKr2 注意：** 项目使用 C++17 标准，所以不能直接用 `std::endian`。但由于所有目标平台（x86/x64、ARM64）都是小端序，实际代码中并没有运行时字节序检测——而是在读取二进制文件时统一使用小端序读取函数。

## 数据类型大小差异

### 一个令人困惑的 Bug

你写了一个函数来处理大文件（>2GB）：

```cpp
long fileSize = getFileSize("big_archive.xp3");
printf("文件大小: %ld 字节\n", fileSize);
```

在 Linux 上，这个程序报告 5GB 的归档文件大小完全正确。但同样的代码在 Windows 上编译运行后，报告文件大小是一个负数！

**原因：** `long` 在 Linux 上是 8 字节（可以表示到 ±9.2×10¹⁸），但在 Windows 上只有 4 字节（最大 ±2.1×10⁹ ≈ 2GB）。5GB 的文件大小溢出了 4 字节的 `long`，产生了负数。

### 各平台类型大小对比

C++ 标准只规定了类型的**最小**大小，实际大小取决于平台和编译器的数据模型：

| 类型 | Windows x64 (MSVC) | Linux x64 (GCC) | macOS arm64 (Clang) | Android arm64 (NDK) |
|------|-------------------|-----------------|--------------------|--------------------|
| `char` | 1 字节 | 1 字节 | 1 字节 | 1 字节 |
| `short` | 2 字节 | 2 字节 | 2 字节 | 2 字节 |
| `int` | 4 字节 | 4 字节 | 4 字节 | 4 字节 |
| **`long`** | **4 字节** ⚠️ | **8 字节** | **8 字节** | **8 字节** |
| `long long` | 8 字节 | 8 字节 | 8 字节 | 8 字节 |
| `size_t` | 8 字节 | 8 字节 | 8 字节 | 8 字节 |
| **`wchar_t`** | **2 字节 (UTF-16)** ⚠️ | **4 字节 (UTF-32)** | **4 字节 (UTF-32)** | **4 字节 (UTF-32)** |
| 指针 | 8 字节 | 8 字节 | 8 字节 | 8 字节 |

两个最危险的差异：

1. **`long`**：Windows 使用 LLP64 数据模型（`long` = 4 字节），而 Linux/macOS/Android 使用 LP64 数据模型（`long` = 8 字节）。这是 C/C++ 跨平台开发中最臭名昭著的陷阱。

2. **`wchar_t`**：Windows 上是 2 字节（存储 UTF-16 编码单元），Unix-like 系统上是 4 字节（存储 UTF-32 码点）。这意味着 `std::wstring` 在不同平台上的内存布局完全不同。

### 验证类型大小

编写一个程序来验证当前平台的类型大小：

```cpp
#include <cstdint>
#include <cstdio>
#include <climits>

int main() {
    printf("===== 基本类型大小 =====\n");
    printf("char:        %zu 字节 (范围: %d ~ %d)\n",
           sizeof(char), CHAR_MIN, CHAR_MAX);
    printf("short:       %zu 字节 (范围: %d ~ %d)\n",
           sizeof(short), SHRT_MIN, SHRT_MAX);
    printf("int:         %zu 字节 (范围: %d ~ %d)\n",
           sizeof(int), INT_MIN, INT_MAX);
    printf("long:        %zu 字节 (范围: %ld ~ %ld)\n",
           sizeof(long), LONG_MIN, LONG_MAX);
    printf("long long:   %zu 字节\n", sizeof(long long));
    printf("size_t:      %zu 字节\n", sizeof(size_t));
    printf("wchar_t:     %zu 字节\n", sizeof(wchar_t));
    printf("void*:       %zu 字节\n", sizeof(void*));

    printf("\n===== 固定宽度类型 =====\n");
    printf("int8_t:      %zu 字节\n", sizeof(int8_t));
    printf("int16_t:     %zu 字节\n", sizeof(int16_t));
    printf("int32_t:     %zu 字节\n", sizeof(int32_t));
    printf("int64_t:     %zu 字节\n", sizeof(int64_t));

    printf("\n===== 数据模型 =====\n");
    if (sizeof(long) == 4 && sizeof(void*) == 8) {
        printf("数据模型: LLP64 (Windows)\n");
    } else if (sizeof(long) == 8 && sizeof(void*) == 8) {
        printf("数据模型: LP64 (Linux/macOS/Android)\n");
    }

    return 0;
}
```

**Windows 上的输出：**
```
long:        4 字节 (范围: -2147483648 ~ 2147483647)
数据模型: LLP64 (Windows)
```

**Linux 上的输出：**
```
long:        8 字节 (范围: -9223372036854775808 ~ 9223372036854775807)
数据模型: LP64 (Linux/macOS/Android)
```

### 固定宽度整数类型：跨平台的解药

`<cstdint>` 头文件提供了固定宽度的整数类型，在所有平台上保证大小一致：

```cpp
#include <cstdint>
#include <cstdio>

int main() {
    // ❌ 错误：使用 long，Windows 上 4 字节，Linux 上 8 字节
    long fileSize = 5000000000L;  // Windows 上溢出！
    printf("long fileSize = %ld\n", fileSize);

    // ✅ 正确：使用固定宽度类型，所有平台保证相同大小
    int64_t  fileSize64 = 5000000000LL; // 始终 8 字节，不会溢出
    uint32_t checksum   = 0xDEADBEEF;   // 始终 4 字节无符号
    int16_t  audioSample = -32768;       // 始终 2 字节有符号
    uint8_t  pixel       = 255;          // 始终 1 字节无符号

    printf("int64_t  fileSize = %lld (保证 8 字节)\n", (long long)fileSize64);
    printf("uint32_t checksum = 0x%08X (保证 4 字节)\n", checksum);
    printf("int16_t  sample   = %d (保证 2 字节)\n", audioSample);
    printf("uint8_t  pixel    = %u (保证 1 字节)\n", pixel);

    return 0;
}
```

**固定宽度类型速查表：**

| 类型 | 大小 | 范围 | 典型用途 |
|------|------|------|---------|
| `int8_t` / `uint8_t` | 1 字节 | -128~127 / 0~255 | 像素分量、字节数据 |
| `int16_t` / `uint16_t` | 2 字节 | ±32,767 / 0~65,535 | 音频采样、端口号 |
| `int32_t` / `uint32_t` | 4 字节 | ±21 亿 / 0~42 亿 | 一般整数、校验和 |
| `int64_t` / `uint64_t` | 8 字节 | 极大范围 | 文件大小、时间戳 |

**KrKr2 的做法：** 项目在 `cpp/core/tjs2/tjsTypes.h` 中定义了一套类型别名（如 `tjs_int`、`tjs_uint64`），底层全部映射到 `<cstdint>` 的固定宽度类型。这确保了脚本引擎中所有数据类型的跨平台一致性。

### 结构体对齐与 `#pragma pack`

**场景：** 你定义了一个结构体来映射 XP3 归档文件头，然后直接从文件 fread 到结构体。但发现读取的数据全部错位。

```cpp
#include <cstdint>
#include <cstdio>

// ❌ 问题：编译器会在成员之间插入填充字节来对齐
struct BadHeader {
    uint8_t  magic;       // 1 字节
    // ← 编译器可能在这里插入 3-7 字节的填充！
    uint32_t dataSize;    // 4 字节
    // ← 编译器可能在这里插入 2 字节的填充！
    uint16_t version;     // 2 字节
};

// ✅ 正确：使用 #pragma pack 关闭对齐填充
#pragma pack(push, 1)   // 保存当前对齐设置，设为 1 字节对齐
struct GoodHeader {
    uint8_t  magic;       // 1 字节（偏移 0）
    uint32_t dataSize;    // 4 字节（偏移 1，紧挨着 magic）
    uint16_t version;     // 2 字节（偏移 5，紧挨着 dataSize）
};
#pragma pack(pop)        // 恢复之前的对齐设置

int main() {
    printf("BadHeader  size: %zu 字节\n", sizeof(BadHeader));
    // 通常是 12 字节（有 5 字节填充）

    printf("GoodHeader size: %zu 字节\n", sizeof(GoodHeader));
    // 始终是 7 字节（1 + 4 + 2，无填充）

    // 验证成员偏移
    printf("\nBadHeader 成员偏移:\n");
    printf("  magic    = 偏移 %zu\n", offsetof(BadHeader, magic));     // 0
    printf("  dataSize = 偏移 %zu\n", offsetof(BadHeader, dataSize)); // 4 (有 3 字节填充!)
    printf("  version  = 偏移 %zu\n", offsetof(BadHeader, version));  // 8

    printf("\nGoodHeader 成员偏移:\n");
    printf("  magic    = 偏移 %zu\n", offsetof(GoodHeader, magic));     // 0
    printf("  dataSize = 偏移 %zu\n", offsetof(GoodHeader, dataSize)); // 1 (紧挨着)
    printf("  version  = 偏移 %zu\n", offsetof(GoodHeader, version));  // 5

    return 0;
}
```

**为什么编译器要插入填充？** 因为 CPU 访问对齐的数据更快。比如在 x86 上，读取一个 4 字节对齐的 `uint32_t` 只需要一次内存访问，但读取未对齐的可能需要两次。编译器默认会在结构体成员之间插入填充字节（padding）来保证对齐。

**但在读写二进制文件时，** 我们需要结构体的内存布局与文件中的字节序列完全匹配，这时就必须用 `#pragma pack(1)` 来关闭对齐填充。

**KrKr2 中的实际例子：**

```cpp
// XP3 归档文件头（cpp/core/base/XP3Archive.cpp 中的概念）
#pragma pack(push, 1)
struct XP3ArchiveHeader {
    uint8_t  magic[11];      // "XP3\r\n\x1a\x0a\x8b\x67\x01"
    uint64_t indexOffset;    // 索引表在文件中的偏移量
};
// sizeof(XP3ArchiveHeader) = 11 + 8 = 19 字节
// 如果不用 #pragma pack(1)，可能变成 24 字节（8字节对齐）
#pragma pack(pop)

// 正确读取文件头的方式
// FILE* fp = fopen("game.xp3", "rb");
// XP3ArchiveHeader header;
// fread(&header, sizeof(header), 1, fp);
// uint64_t indexOff = header.indexOffset; // 直接使用结构体成员
```

> **注意：** `#pragma pack` 不是 C++ 标准的一部分，但被所有主流编译器（MSVC、GCC、Clang）支持。它是跨平台二进制文件处理中的事实标准。

## 常见错误与排查

### 错误 1：用 `long` 存储文件大小

```cpp
// ❌ 错误：Windows 上 long 是 4 字节，>2GB 的文件大小会溢出
long fileSize = getFileSize("big_archive.xp3");

// ✅ 正确方案 1：使用 int64_t
int64_t fileSize = getFileSize("big_archive.xp3");

// ✅ 正确方案 2：使用 std::filesystem
#include <filesystem>
std::uintmax_t fileSize = std::filesystem::file_size("big_archive.xp3");
// std::uintmax_t 在所有 64 位平台上都是 8 字节
```

### 错误 2：直接 reinterpret_cast 读取二进制数据

```cpp
// ❌ 错误：未定义行为 + 字节序问题 + 对齐问题
const uint8_t* data = readFromFile();
uint32_t value = *reinterpret_cast<const uint32_t*>(data);

// ✅ 正确：逐字节组装
uint32_t value = readLE32(data);

// ✅ 或使用 memcpy（安全但仍有字节序问题）
uint32_t value;
memcpy(&value, data, sizeof(value));
// 注意：memcpy 不处理字节序，只在本机字节序 == 文件字节序时正确
```

### 错误 3：假设 `wchar_t` 的大小

```cpp
// ❌ 错误：假设 wchar_t 始终是 2 字节
wchar_t ch = L'你';
uint8_t buffer[2];  // Windows 上刚好，Linux 上少了 2 字节！
memcpy(buffer, &ch, sizeof(ch));

// ✅ 正确：使用 char16_t 或 char32_t
char16_t ch16 = u'你';  // 始终 2 字节
char32_t ch32 = U'你';  // 始终 4 字节
```

### 实践 2：编写跨平台二进制文件读写器

```cpp
// binary_rw.cpp — 跨平台二进制文件读写
#include <cstdint>
#include <cstdio>
#include <cstring>

// 小端序读写工具函数
uint32_t readLE32(const uint8_t* p) {
    return uint32_t(p[0]) | (uint32_t(p[1]) << 8)
         | (uint32_t(p[2]) << 16) | (uint32_t(p[3]) << 24);
}

void writeLE32(uint8_t* p, uint32_t v) {
    p[0] = uint8_t(v); p[1] = uint8_t(v >> 8);
    p[2] = uint8_t(v >> 16); p[3] = uint8_t(v >> 24);
}

uint16_t readLE16(const uint8_t* p) {
    return uint16_t(p[0]) | (uint16_t(p[1]) << 8);
}

void writeLE16(uint8_t* p, uint16_t v) {
    p[0] = uint8_t(v); p[1] = uint8_t(v >> 8);
}

// 简单的二进制记录格式
// 每条记录: [4字节ID][2字节类型][4字节数据]
struct Record {
    uint32_t id;
    uint16_t type;
    uint32_t data;
};

void writeRecords(const char* filename, const Record* records, int count) {
    FILE* fp = fopen(filename, "wb");
    if (!fp) { printf("写入失败\n"); return; }

    // 先写记录数量（4字节，小端序）
    uint8_t buf[4];
    writeLE32(buf, (uint32_t)count);
    fwrite(buf, 1, 4, fp);

    // 写每条记录
    for (int i = 0; i < count; ++i) {
        uint8_t rec[10]; // 4 + 2 + 4 = 10 字节
        writeLE32(rec + 0, records[i].id);
        writeLE16(rec + 4, records[i].type);
        writeLE32(rec + 6, records[i].data);
        fwrite(rec, 1, 10, fp);
    }

    fclose(fp);
    printf("写入 %d 条记录到 %s\n", count, filename);
}

void readRecords(const char* filename) {
    FILE* fp = fopen(filename, "rb");
    if (!fp) { printf("读取失败\n"); return; }

    // 读记录数量
    uint8_t buf[4];
    fread(buf, 1, 4, fp);
    uint32_t count = readLE32(buf);
    printf("共 %u 条记录:\n", count);

    // 读每条记录
    for (uint32_t i = 0; i < count; ++i) {
        uint8_t rec[10];
        fread(rec, 1, 10, fp);

        uint32_t id   = readLE32(rec + 0);
        uint16_t type = readLE16(rec + 4);
        uint32_t data = readLE32(rec + 6);

        printf("  [%u] id=%u, type=%u, data=%u\n", i, id, type, data);
    }

    fclose(fp);
}

int main() {
    Record records[] = {
        {1, 100, 0xDEADBEEF},
        {2, 200, 0xCAFEBABE},
        {3, 300, 0x12345678},
    };

    writeRecords("test.bin", records, 3);
    readRecords("test.bin");

    return 0;
}
```

## 对照项目源码

KrKr2 项目中处理字节序和数据类型差异的关键位置：

相关文件：
- `cpp/core/tjs2/tjsTypes.h` — 定义 `tjs_int`、`tjs_uint64`、`tjs_char` 等类型别名，底层映射到 `<cstdint>` 固定宽度类型
- `cpp/core/base/XP3Archive.cpp` — XP3 归档文件解析，使用小端序读取文件头和索引
- `cpp/core/visual/LoadTLG.cpp` — TLG 图像格式加载，需要按特定字节序读取像素数据
- `cpp/plugins/psbfile/` — PSB（E-mote）文件解析，使用 `#pragma pack` 定义二进制结构体
- `cpp/core/sound/WaveIntf.cpp` — WAV 文件解析，RIFF 格式使用小端序

## 本节小结

- **字节序**是多字节数据在内存中的存储顺序。小端序（低字节在低地址）用于 x86/ARM，大端序用于网络协议
- **永远不要用 `reinterpret_cast` 读取二进制数据**——使用逐字节组装的读写函数
- **`long` 类型是跨平台地雷**：Windows 4 字节，Linux/macOS/Android 8 字节。始终使用 `int32_t`/`int64_t`
- **`wchar_t` 大小也不一致**：Windows 2 字节（UTF-16），Unix 4 字节（UTF-32）。用 `char16_t`/`char32_t` 代替
- **结构体对齐**会在成员间插入填充字节。读写二进制文件时用 `#pragma pack(1)` 关闭填充
- 网络字节序是大端序，使用 `htonl`/`ntohl` 系列函数转换
- KrKr2 在 `tjsTypes.h` 中统一使用固定宽度类型，在归档/图像/音频解析中使用小端序读取函数

### 题目 1：手动字节序转换

给定小端序字节序列 `A4 01 00 00`，计算它表示的 `uint32_t` 值。然后编写 C++ 代码验证你的计算。

<details>
<summary>查看答案</summary>

```
小端序：低字节在前
  data[0] = 0xA4  → 最低字节
  data[1] = 0x01  → 次低字节
  data[2] = 0x00
  data[3] = 0x00

值 = 0xA4 | (0x01 << 8) | (0x00 << 16) | (0x00 << 24)
   = 0xA4 | 0x0100 | 0 | 0
   = 0x000001A4
   = 420
```

验证代码：

```cpp
#include <cstdint>
#include <cstdio>

uint32_t readLE32(const uint8_t* data) {
    return uint32_t(data[0]) | (uint32_t(data[1]) << 8)
         | (uint32_t(data[2]) << 16) | (uint32_t(data[3]) << 24);
}

int main() {
    uint8_t bytes[] = {0xA4, 0x01, 0x00, 0x00};
    uint32_t value = readLE32(bytes);
    printf("小端序 A4 01 00 00 = %u (0x%08X)\n", value, value);
    // 输出: 小端序 A4 01 00 00 = 420 (0x000001A4)
    return 0;
}
```

### 题目 2：为什么 KrKr2 不用 `long`

KrKr2 的 `tjsTypes.h` 中定义了 `typedef int64_t tjs_int64;` 而不是 `typedef long long tjs_int64;`。虽然 `long long` 在所有目标平台上都是 8 字节，为什么仍然推荐使用 `int64_t`？

<details>
<summary>查看答案</summary>

1. **语义明确性**：`int64_t` 的名字本身就声明了"恰好是 64 位"，而 `long long` 只保证"至少 64 位"。虽然实际上所有现代平台的 `long long` 都是 64 位，但语义上 `int64_t` 更精确。

2. **一致性**：如果代码中混用 `int64_t` 和 `long long`，在某些编译器上它们虽然大小相同但是**不同的类型**，可能导致函数重载、模板特化等问题。统一使用 `int64_t` 避免此问题。

3. **格式化字符串**：`printf` 中 `long long` 使用 `%lld`，但 `int64_t` 使用 `<inttypes.h>` 中定义的 `PRId64` 宏。后者在所有平台上都能正确工作：

```cpp
#include <cstdint>
#include <cinttypes>
#include <cstdio>

int main() {
    int64_t value = 9223372036854775807LL;

    // ✅ 跨平台正确的格式化方式
    printf("int64_t: %" PRId64 "\n", value);

    // ⚠️ long long 方式：在大多数平台可以，但不是 100% 可移植
    printf("long long: %lld\n", (long long)value);

    return 0;
}
```

</details>

### 题目 3：修复对齐问题

以下结构体用于读取二进制文件头。指出问题并修复：

```cpp
struct FileHeader {
    char     magic[3];    // "KR2"
    uint32_t version;     // 版本号
    uint16_t flags;       // 标志位
    uint64_t dataOffset;  // 数据偏移量
};
// 预期大小: 3 + 4 + 2 + 8 = 17 字节
// 实际大小: ? 字节
```

<details>
<summary>查看答案</summary>

**问题：** 编译器会在成员之间和结构体末尾插入填充字节来满足对齐要求。实际的内存布局是：

```
偏移 0: magic[3]     = 3 字节
偏移 3: (填充 1 字节) ← 为了让 version 4 字节对齐
偏移 4: version       = 4 字节
偏移 8: flags         = 2 字节
偏移 10: (填充 6 字节) ← 为了让 dataOffset 8 字节对齐
偏移 16: dataOffset    = 8 字节
总计: 24 字节（而不是预期的 17 字节！）
```

用 `fread(&header, sizeof(header), 1, file)` 读取 24 字节就会把文件数据读错位。

**修复方案：**

```cpp
#pragma pack(push, 1)
struct FileHeader {
    char     magic[3];    // 偏移 0, 3 字节
    uint32_t version;     // 偏移 3, 4 字节
    uint16_t flags;       // 偏移 7, 2 字节
    uint64_t dataOffset;  // 偏移 9, 8 字节
};
#pragma pack(pop)
// 现在 sizeof(FileHeader) = 17 字节，与文件格式完全匹配

#include <cstdio>
#include <cstdint>
#include <cstddef>

int main() {
    printf("FileHeader size: %zu\n", sizeof(FileHeader)); // 17
    printf("magic  offset: %zu\n", offsetof(FileHeader, magic));      // 0
    printf("version offset: %zu\n", offsetof(FileHeader, version));   // 3
    printf("flags  offset: %zu\n", offsetof(FileHeader, flags));      // 7
    printf("dataOffset: %zu\n", offsetof(FileHeader, dataOffset));    // 9
    return 0;
}
```

</details>
