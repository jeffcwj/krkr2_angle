## 预处理器基础回顾

C++ 编译过程的第一步是**预处理**（Preprocessing），由预处理器执行。预处理器处理所有以 `#` 开头的指令，在编译器看到代码之前就完成了文本替换和条件选择。

```
源代码 (.cpp) → [预处理器] → 预处理后代码 → [编译器] → 目标文件 (.o/.obj)
```

### 条件编译指令一览

| 指令 | 含义 | 示例 |
|------|------|------|
| `#define MACRO` | 定义宏（无值） | `#define _DEBUG` |
| `#define MACRO value` | 定义宏（有值） | `#define VERSION 3` |
| `#undef MACRO` | 取消宏定义 | `#undef NDEBUG` |
| `#ifdef MACRO` | 如果宏**已定义** | `#ifdef _WIN32` |
| `#ifndef MACRO` | 如果宏**未定义** | `#ifndef NDEBUG` |
| `#if expr` | 如果表达式为真 | `#if VERSION >= 3` |
| `#elif expr` | 否则如果 | `#elif defined(__linux__)` |
| `#else` | 否则 | `#else` |
| `#endif` | 结束条件块 | `#endif` |

### `#ifdef` vs `#if defined` 的区别

这两种写法功能相同，但 `#if defined` 更灵活：

```cpp
// 方式1: #ifdef — 只能检查单个宏
#ifdef _WIN32
    // Windows 代码
#endif

// 方式2: #if defined — 可以组合多个条件
#if defined(_WIN32)
    // 等价于 #ifdef _WIN32
#endif

// #if defined 的优势：支持逻辑运算
#if defined(_WIN32) && !defined(_WIN64)
    // 32位 Windows
#endif

#if defined(__linux__) || defined(__APPLE__)
    // Unix-like 系统（Linux 或 macOS）
#endif

#if defined(__ANDROID__) && __ANDROID_API__ >= 24
    // Android API 24+ (Android 7.0 Nougat)
#endif
```

**推荐做法：** 简单的单宏判断用 `#ifdef`，需要组合逻辑时用 `#if defined()`。

## 各平台预定义宏速查表

### 操作系统宏

编译器会自动定义这些宏，无需手动 `#define`：

| 宏名 | Windows (MSVC) | Linux (GCC/Clang) | macOS (Clang) | Android (NDK Clang) |
|------|---------------|-------------------|---------------|-------------------|
| `_WIN32` | ✅ | ❌ | ❌ | ❌ |
| `_WIN64` | ✅ (64位) | ❌ | ❌ | ❌ |
| `__linux__` | ❌ | ✅ | ❌ | ✅ |
| `__APPLE__` | ❌ | ❌ | ✅ | ❌ |
| `__MACH__` | ❌ | ❌ | ✅ | ❌ |
| `__ANDROID__` | ❌ | ❌ | ❌ | ✅ |
| `__ANDROID_API__` | ❌ | ❌ | ❌ | ✅ (数字，如 24) |
| `__unix__` | ❌ | ✅ | ✅ | ✅ |
| `__posix__` | ❌ | ✅ | ✅ | ✅ |

**注意：** Android 同时定义了 `__linux__` 和 `__ANDROID__`。因此判断 Android 时必须优先检查 `__ANDROID__`：

```cpp
// ❌ 错误：Linux 分支会匹配 Android
#ifdef __linux__
    // 这段代码在 Android 上也会执行！
#endif

// ✅ 正确：先检查 Android
#if defined(__ANDROID__)
    // Android 专用代码
#elif defined(__linux__)
    // 纯 Linux（非 Android）
#elif defined(__APPLE__)
    // macOS / iOS
#elif defined(_WIN32)
    // Windows
#endif
```

### 编译器宏

| 宏名 | 含义 | 示例值 |
|------|------|--------|
| `_MSC_VER` | MSVC 编译器版本 | `1940`（VS 2022 17.10） |
| `__GNUC__` | GCC 主版本号 | `13` |
| `__GNUC_MINOR__` | GCC 次版本号 | `2` |
| `__clang__` | Clang 编译器 | 已定义 |
| `__clang_major__` | Clang 主版本号 | `17` |

```cpp
// 按编译器选择不同的特性
#if defined(_MSC_VER)
    // MSVC 特有: __declspec, #pragma warning
    #pragma warning(disable: 4996) // 禁用 deprecated 警告
#elif defined(__clang__)
    // Clang 特有: __has_feature, __attribute__
    #pragma clang diagnostic ignored "-Wdeprecated"
#elif defined(__GNUC__)
    // GCC 特有: __attribute__, __builtin_*
    #pragma GCC diagnostic ignored "-Wdeprecated"
#endif
```

### 架构宏

| 宏名 | 含义 |
|------|------|
| `__x86_64__` 或 `_M_X64` | x86-64 架构 |
| `__i386__` 或 `_M_IX86` | x86 32位架构 |
| `__aarch64__` 或 `_M_ARM64` | ARM 64位架构 |
| `__arm__` 或 `_M_ARM` | ARM 32位架构 |

```cpp
// KrKr2 支持的架构
#if defined(__x86_64__) || defined(_M_X64)
    constexpr const char* ARCH_NAME = "x86_64";
#elif defined(__aarch64__) || defined(_M_ARM64)
    constexpr const char* ARCH_NAME = "arm64";
#else
    #error "Unsupported architecture"
#endif
```

### C++ 标准版本宏

| 宏值 | 对应标准 |
|------|---------|
| `__cplusplus >= 201703L` | C++17 |
| `__cplusplus >= 202002L` | C++20 |
| `__cplusplus >= 202302L` | C++23 |

```cpp
#if __cplusplus >= 202002L
    #include <bit>    // C++20: std::endian
    #include <span>   // C++20: std::span
#elif __cplusplus >= 201703L
    #include <optional>  // C++17
    #include <variant>   // C++17
#endif
```

**注意：** MSVC 默认不正确设置 `__cplusplus`（始终返回 `199711L`），需要添加编译选项 `/Zc:__cplusplus`。KrKr2 使用 CMake 的 `CMAKE_CXX_STANDARD 17`，CMake 会自动处理这个问题。

## CMake 向 C++ 传递平台信息

### KrKr2 的做法

KrKr2 项目在根 `CMakeLists.txt` 中定义了自定义平台变量，并通过 `target_compile_definitions` 传递给 C++ 代码：

```cmake
# krkr2/CMakeLists.txt（简化）

# 自定义平台检测变量
if(CMAKE_SYSTEM_NAME STREQUAL "Windows")
    set(WINDOWS TRUE)
elseif(CMAKE_SYSTEM_NAME STREQUAL "Linux")
    set(LINUX TRUE)
elseif(CMAKE_SYSTEM_NAME STREQUAL "Darwin")
    set(MACOS TRUE)
elseif(CMAKE_SYSTEM_NAME STREQUAL "Android")
    set(ANDROID TRUE)
endif()

# 将平台信息传递给 C++ 代码
target_compile_definitions(krkr2core INTERFACE
    $<$<BOOL:${WINDOWS}>:WINDOWS=1>
    $<$<BOOL:${LINUX}>:LINUX=1>
    $<$<BOOL:${MACOS}>:MACOS=1>
    $<$<BOOL:${ANDROID}>:ANDROID=1>
)
```

这样 C++ 代码中就可以使用 `WINDOWS`、`LINUX`、`MACOS`、`ANDROID` 宏：

```cpp
// 使用 KrKr2 自定义的平台宏（通过 CMake 传入）
#ifdef WINDOWS
    #include <windows.h>
    void showMessage(const char* msg) {
        MessageBoxA(nullptr, msg, "KrKr2", MB_OK);
    }
#elif defined(LINUX)
    #include <gtk/gtk.h>
    void showMessage(const char* msg) {
        GtkWidget* dialog = gtk_message_dialog_new(nullptr,
            GTK_DIALOG_MODAL, GTK_MESSAGE_INFO, GTK_BUTTONS_OK, "%s", msg);
        gtk_dialog_run(GTK_DIALOG(dialog));
        gtk_widget_destroy(dialog);
    }
#elif defined(ANDROID)
    #include <android/log.h>
    void showMessage(const char* msg) {
        __android_log_print(ANDROID_LOG_INFO, "KrKr2", "%s", msg);
    }
#elif defined(MACOS)
    // macOS 使用 Objective-C++ 的 NSAlert 或 Cocos2d UI
    void showMessage(const char* msg);
#endif
```

### `target_compile_definitions` 详解

```cmake
# 基本用法：为目标添加编译定义
target_compile_definitions(my_target PRIVATE
    MY_MACRO              # 定义 MY_MACRO（无值）
    VERSION=3             # 定义 VERSION 为 3
    NAME="hello"          # 定义 NAME 为字符串 "hello"
)

# 可见性关键字
# PRIVATE   — 仅当前目标可见
# PUBLIC    — 当前目标 + 链接此目标的所有目标
# INTERFACE — 仅链接此目标的其他目标（当前目标不可见）

# 生成器表达式：条件定义
target_compile_definitions(my_target PRIVATE
    # 仅 Debug 配置时定义
    $<$<CONFIG:Debug>:_DEBUG>

    # 仅 Windows 平台时定义
    $<$<PLATFORM_ID:Windows>:WIN32_LEAN_AND_MEAN>

    # 组合条件：Debug + Windows
    $<$<AND:$<CONFIG:Debug>,$<PLATFORM_ID:Windows>>:WIN_DEBUG_MODE>
)
```

### `configure_file` 方式

另一种常见做法是通过 `configure_file` 生成配置头文件：

```cmake
# CMakeLists.txt
set(PROJECT_VERSION_MAJOR 2)
set(PROJECT_VERSION_MINOR 0)
set(HAS_OPENGL TRUE)
set(HAS_VULKAN FALSE)

configure_file(
    ${CMAKE_SOURCE_DIR}/config.h.in
    ${CMAKE_BINARY_DIR}/config.h
)
```

```cpp
// config.h.in（模板文件）
#pragma once

#define PROJECT_VERSION_MAJOR @PROJECT_VERSION_MAJOR@
#define PROJECT_VERSION_MINOR @PROJECT_VERSION_MINOR@

#cmakedefine HAS_OPENGL
#cmakedefine HAS_VULKAN
// HAS_OPENGL 为 TRUE → 生成 #define HAS_OPENGL
// HAS_VULKAN 为 FALSE → 生成 /* #undef HAS_VULKAN */
```

```cpp
// 生成的 config.h
#pragma once

#define PROJECT_VERSION_MAJOR 2
#define PROJECT_VERSION_MINOR 0

#define HAS_OPENGL
/* #undef HAS_VULKAN */
```

```cpp
// 使用生成的配置
#include "config.h"

#ifdef HAS_OPENGL
    #include <GL/gl.h>
    // OpenGL 渲染代码
#endif

#ifdef HAS_VULKAN
    #include <vulkan/vulkan.h>
    // Vulkan 渲染代码（当前未启用）
#endif
```

## KrKr2 条件编译实例分析

### 实例 1：平台入口点

KrKr2 的每个平台有独立的入口文件，但它们都调用相同的核心逻辑：

```
platforms/
├── windows/main.cpp     ← #include <windows.h>, WinMain 入口
├── linux/main.cpp       ← #include <gtk/gtk.h>, int main() 入口
├── apple/macos/main.cpp ← #include <cocos2d.h>, int main() 入口
└── android/cpp/
    └── krkr2_android.cpp ← JNI_OnLoad, Java 桥接
```

这种方式称为**文件级条件编译**——不是用 `#ifdef` 在一个文件内区分平台，而是为每个平台提供独立的源文件，通过 CMake 选择编译哪些文件：

```cmake
# 根据平台选择入口文件
if(WINDOWS)
    add_executable(krkr2 WIN32 platforms/windows/main.cpp)
elseif(LINUX)
    add_executable(krkr2 platforms/linux/main.cpp)
elseif(MACOS)
    add_executable(krkr2 MACOSX_BUNDLE platforms/apple/macos/main.cpp)
elseif(ANDROID)
    add_library(krkr2 SHARED platforms/android/cpp/krkr2_android.cpp)
endif()
```

### 实例 2：编译定义传递

KrKr2 的 `cpp/core/CMakeLists.txt` 中为核心库添加了多个编译定义：

```cmake
# cpp/core/CMakeLists.txt（实际代码简化版）
target_compile_definitions(krkr2core INTERFACE
    TJS_TEXT_OUT_CRLF           # TJS2 脚本输出使用 CRLF 换行
    __STDC_CONSTANT_MACROS      # C99 常量宏（INT64_C 等）
    USE_UNICODE_FSTRING         # 使用 Unicode 字符串
)
```

这些定义使用 `INTERFACE` 可见性——`krkr2core` 是 INTERFACE 库，它本身不编译任何代码，但所有链接到它的目标（如 `krkr2` 可执行文件）都会继承这些定义。

### 实例 3：OpenMP 条件启用

```cmake
# 根 CMakeLists.txt
if(NOT APPLE)
    find_package(OpenMP)
    if(OpenMP_CXX_FOUND)
        target_link_libraries(krkr2core INTERFACE OpenMP::OpenMP_CXX)
    endif()
endif()
```

Apple 平台默认不支持 OpenMP（Apple Clang 未内置），因此 KrKr2 在 macOS 上跳过 OpenMP。对应的 C++ 代码中：

```cpp
// 使用 _OPENMP 宏判断 OpenMP 是否可用
#ifdef _OPENMP
    #include <omp.h>
    #pragma omp parallel for
    for (int i = 0; i < count; ++i) {
        processPixel(i);
    }
#else
    // 串行回退
    for (int i = 0; i < count; ++i) {
        processPixel(i);
    }
#endif
```

### 实例 4：environ 目录的条件编译

`cpp/core/environ/` 目录包含五个平台实现子目录：

```
environ/
├── cocos2d/       ← 所有平台共用的 Cocos2d 桥接
├── win32/         ← Windows 专用
├── linux/         ← Linux 专用
├── android/       ← Android 专用
├── apple/         ← macOS 专用
├── sdl/           ← SDL2 后端（Linux/macOS 可选）
└── ui/            ← Cocos2d UI 组件（所有平台）
```

CMake 根据平台变量选择编译哪些子目录中的源文件，而不是在单个文件中用大量 `#ifdef`。这是一种更清晰的跨平台代码组织方式。

## 条件编译最佳实践

### 原则 1：最小化条件编译范围

```cpp
// ❌ 差：整个函数都在条件编译中，代码重复严重
#ifdef _WIN32
void processFile(const std::string& path) {
    HANDLE hFile = CreateFileA(path.c_str(), GENERIC_READ, 0, nullptr,
                               OPEN_EXISTING, 0, nullptr);
    // ... 100行 Windows 特有代码 ...
    char buffer[4096];
    // ... 处理逻辑（与平台无关） ...
    CloseHandle(hFile);
}
#else
void processFile(const std::string& path) {
    int fd = open(path.c_str(), O_RDONLY);
    // ... 100行 POSIX 特有代码 ...
    char buffer[4096];
    // ... 处理逻辑（与平台无关，与上面重复） ...
    close(fd);
}
#endif

// ✅ 好：仅隔离平台差异部分，共享公共逻辑
class FileHandle {
public:
    explicit FileHandle(const std::string& path) {
#ifdef _WIN32
        handle_ = CreateFileA(path.c_str(), GENERIC_READ, 0, nullptr,
                              OPEN_EXISTING, 0, nullptr);
#else
        fd_ = open(path.c_str(), O_RDONLY);
#endif
    }

    size_t read(char* buffer, size_t size) {
#ifdef _WIN32
        DWORD bytesRead = 0;
        ReadFile(handle_, buffer, static_cast<DWORD>(size), &bytesRead, nullptr);
        return bytesRead;
#else
        return ::read(fd_, buffer, size);
#endif
    }

    ~FileHandle() {
#ifdef _WIN32
        if (handle_ != INVALID_HANDLE_VALUE) CloseHandle(handle_);
#else
        if (fd_ >= 0) close(fd_);
#endif
    }

private:
#ifdef _WIN32
    HANDLE handle_ = INVALID_HANDLE_VALUE;
#else
    int fd_ = -1;
#endif
};

// 平台无关的处理逻辑——不需要任何 #ifdef
void processFile(const std::string& path) {
    FileHandle file(path);
    char buffer[4096];
    size_t bytesRead = file.read(buffer, sizeof(buffer));
    // ... 处理逻辑 ...
}
```

### 原则 2：优先用文件分离替代 `#ifdef`

当平台差异较大时（如整个函数的实现完全不同），把不同平台的代码放在不同文件中：

```
src/
├── file_handle.h            ← 公共接口（纯声明）
├── file_handle_win32.cpp    ← Windows 实现
├── file_handle_posix.cpp    ← Linux/macOS 实现
└── file_handle_android.cpp  ← Android 实现
```

```cmake
# CMakeLists.txt
target_sources(mylib PRIVATE
    file_handle.h
    $<$<PLATFORM_ID:Windows>:file_handle_win32.cpp>
    $<$<PLATFORM_ID:Linux>:file_handle_posix.cpp>
    $<$<PLATFORM_ID:Darwin>:file_handle_posix.cpp>
    $<$<PLATFORM_ID:Android>:file_handle_android.cpp>
)
```

### 原则 3：使用 `#error` 捕获未处理的平台

```cpp
#if defined(_WIN32)
    // Windows
#elif defined(__ANDROID__)
    // Android
#elif defined(__linux__)
    // Linux
#elif defined(__APPLE__)
    // macOS
#else
    #error "Unsupported platform: please add support for this OS"
#endif
```

### 原则 4：避免嵌套条件编译

```cpp
// ❌ 差：三层嵌套，难以阅读和维护
#ifdef _WIN32
    #ifdef _DEBUG
        #if _MSC_VER >= 1920
            // ...
        #endif
    #endif
#endif

// ✅ 好：用 #if defined + 逻辑运算符压平嵌套
#if defined(_WIN32) && defined(_DEBUG) && (_MSC_VER >= 1920)
    // ...
#endif
```

## 常见错误及解决方案

### 错误 1：Android 被 `__linux__` 匹配

```cpp
// 问题：Android 上 __linux__ 也为 true
#ifdef __linux__
    #include <gtk/gtk.h>  // Android 上没有 GTK！编译失败
#endif

// 解决：Android 判断放在 Linux 前面
#if defined(__ANDROID__)
    #include <android/log.h>
#elif defined(__linux__)
    #include <gtk/gtk.h>
#endif
```

### 错误 2：MSVC 的 `__cplusplus` 不准确

```cpp
// 问题：MSVC 默认 __cplusplus == 199711L，即使用了 /std:c++17
#if __cplusplus >= 201703L
    // 这段代码在 MSVC 上不会编译！
    #include <optional>
#endif

// 解决方案 1：在 CMake 中添加 /Zc:__cplusplus
// target_compile_options(mytarget PRIVATE /Zc:__cplusplus)

// 解决方案 2：同时检查 _MSC_VER
#if __cplusplus >= 201703L || (defined(_MSC_VER) && _MSC_VER >= 1914)
    #include <optional>
#endif
```

### 错误 3：忘记 `#endif`

```cpp
#ifdef _WIN32
    void winFunc() { /* ... */ }
// 忘记 #endif → 后面所有代码都只在 Windows 编译

void commonFunc() { /* 这个函数在非 Windows 上消失了！ */ }
```

**预防措施：** 在 `#endif` 后面加注释标明对应的条件：

```cpp
#ifdef _WIN32
    void winFunc() { /* ... */ }
#endif // _WIN32
```

## 动手实践

### 练习：编写跨平台日志宏

创建一个 `cross_log.h` 头文件，实现跨平台的日志输出：

```cpp
// cross_log.h
#pragma once

#include <cstdio>

#if defined(__ANDROID__)
    #include <android/log.h>
    #define LOG_TAG "MyApp"
    #define LOG_INFO(fmt, ...)  __android_log_print(ANDROID_LOG_INFO, LOG_TAG, fmt, ##__VA_ARGS__)
    #define LOG_ERROR(fmt, ...) __android_log_print(ANDROID_LOG_ERROR, LOG_TAG, fmt, ##__VA_ARGS__)
    #define LOG_DEBUG(fmt, ...) __android_log_print(ANDROID_LOG_DEBUG, LOG_TAG, fmt, ##__VA_ARGS__)

#elif defined(_WIN32)
    #include <windows.h>
    // Windows: 输出到调试器的 Output 窗口
    #define LOG_INFO(fmt, ...)  do { char buf[1024]; snprintf(buf, sizeof(buf), "[INFO] " fmt "\n", ##__VA_ARGS__); OutputDebugStringA(buf); printf("[INFO] " fmt "\n", ##__VA_ARGS__); } while(0)
    #define LOG_ERROR(fmt, ...) do { char buf[1024]; snprintf(buf, sizeof(buf), "[ERROR] " fmt "\n", ##__VA_ARGS__); OutputDebugStringA(buf); fprintf(stderr, "[ERROR] " fmt "\n", ##__VA_ARGS__); } while(0)
    #define LOG_DEBUG(fmt, ...) do { char buf[1024]; snprintf(buf, sizeof(buf), "[DEBUG] " fmt "\n", ##__VA_ARGS__); OutputDebugStringA(buf); printf("[DEBUG] " fmt "\n", ##__VA_ARGS__); } while(0)

#else
    // Linux / macOS: 直接输出到 stdout/stderr
    #define LOG_INFO(fmt, ...)  printf("[INFO] " fmt "\n", ##__VA_ARGS__)
    #define LOG_ERROR(fmt, ...) fprintf(stderr, "[ERROR] " fmt "\n", ##__VA_ARGS__)
    #define LOG_DEBUG(fmt, ...) printf("[DEBUG] " fmt "\n", ##__VA_ARGS__)
#endif

// 仅 Debug 模式启用 DEBUG 日志
#ifdef NDEBUG
    #undef LOG_DEBUG
    #define LOG_DEBUG(fmt, ...) ((void)0) // Release 模式下不输出
#endif
```

**使用示例：**

```cpp
// main.cpp
#include "cross_log.h"

int main() {
    LOG_INFO("Application started, version %d.%d", 2, 0);
    LOG_DEBUG("Debug info: pointer = %p", (void*)main);
    LOG_ERROR("Failed to open file: %s", "data.bin");
    return 0;
}
```

**对应的 CMakeLists.txt：**

```cmake
cmake_minimum_required(VERSION 3.28)
project(cross_log_demo)
set(CMAKE_CXX_STANDARD 17)

add_executable(demo main.cpp)

# Debug 模式不定义 NDEBUG，LOG_DEBUG 会输出
# Release 模式定义 NDEBUG，LOG_DEBUG 被禁用
if(ANDROID)
    target_link_libraries(demo PRIVATE log) # Android 需要链接 log 库
endif()
```

## 本节小结

- **`#ifdef` vs `#if defined`**：前者简洁，后者支持逻辑组合（`&&`、`||`、`!`），推荐根据复杂度选择
- **平台宏优先级**：Android (`__ANDROID__`) 必须在 Linux (`__linux__`) 之前判断
- **CMake 传递定义**：使用 `target_compile_definitions` + 生成器表达式实现条件定义传递，比在源码中硬编码 `#define` 更灵活
- **最佳实践**：
  - 最小化 `#ifdef` 范围，隔离平台差异到小函数或独立文件
  - 文件级平台分离优于文件内大量 `#ifdef` 嵌套
  - 用 `#error` 在未处理平台上产生明确编译错误
- **KrKr2 的做法**：自定义 `WINDOWS`/`LINUX`/`MACOS`/`ANDROID` 宏（通过 CMake 传入），environ 目录用文件分离实现平台抽象

## 练习题与答案

### 题目 1：宏判断顺序

以下代码在 Android 设备上编译运行，会输出什么？为什么？

```cpp
#include <cstdio>
int main() {
#ifdef __linux__
    printf("Linux\n");
#elif defined(__ANDROID__)
    printf("Android\n");
#endif
    return 0;
}
```

<details>
<summary>查看答案</summary>

**输出：** `Linux`

**原因：** Android NDK 编译时同时定义了 `__linux__` 和 `__ANDROID__`。`#ifdef __linux__` 先被匹配到，`#elif` 分支被跳过。

**修复方法：** 把 `__ANDROID__` 的判断放在 `__linux__` 前面：

```cpp
#include <cstdio>
int main() {
#if defined(__ANDROID__)
    printf("Android\n");
#elif defined(__linux__)
    printf("Linux\n");
#endif
    return 0;
}
```

</details>

### 题目 2：CMake 编译定义

写出 CMake 代码，实现以下需求：
- 为目标 `myapp` 添加编译定义 `APP_VERSION=3`
- 仅在 Debug 配置时定义 `ENABLE_LOGGING`
- 仅在 Windows 平台时定义 `USE_WINSOCK`

<details>
<summary>查看答案</summary>

```cmake
target_compile_definitions(myapp PRIVATE
    APP_VERSION=3
    $<$<CONFIG:Debug>:ENABLE_LOGGING>
    $<$<PLATFORM_ID:Windows>:USE_WINSOCK>
)
```

**解释：**
- `APP_VERSION=3` — 无条件定义，所有平台所有配置都有
- `$<$<CONFIG:Debug>:ENABLE_LOGGING>` — 生成器表达式，仅 Debug 配置时展开为 `ENABLE_LOGGING`
- `$<$<PLATFORM_ID:Windows>:USE_WINSOCK>` — 仅 Windows 平台时展开为 `USE_WINSOCK`

</details>

### 题目 3：条件编译重构

以下代码使用了过多的 `#ifdef`，请将其重构为文件分离的方式，给出文件结构和 CMakeLists.txt：

```cpp
// network.cpp
#include "network.h"

void initNetwork() {
#ifdef _WIN32
    WSADATA wsaData;
    WSAStartup(MAKEWORD(2, 2), &wsaData);
#elif defined(__linux__) || defined(__APPLE__)
    // POSIX 不需要初始化
    signal(SIGPIPE, SIG_IGN); // 忽略 SIGPIPE
#elif defined(__ANDROID__)
    signal(SIGPIPE, SIG_IGN);
    // Android 可能还需要检查网络权限
#endif
}

void cleanupNetwork() {
#ifdef _WIN32
    WSACleanup();
#endif
}
```

<details>
<summary>查看答案</summary>

**文件结构：**

```
src/
├── network.h                  ← 公共接口声明
├── network_win32.cpp          ← Windows 实现
├── network_posix.cpp          ← Linux/macOS 实现
├── network_android.cpp        ← Android 实现
└── CMakeLists.txt
```

**network.h：**

```cpp
#pragma once

void initNetwork();
void cleanupNetwork();
```

**network_win32.cpp：**

```cpp
#include "network.h"
#include <winsock2.h>

void initNetwork() {
    WSADATA wsaData;
    WSAStartup(MAKEWORD(2, 2), &wsaData);
}

void cleanupNetwork() {
    WSACleanup();
}
```

**network_posix.cpp：**

```cpp
#include "network.h"
#include <csignal>

void initNetwork() {
    signal(SIGPIPE, SIG_IGN);
}

void cleanupNetwork() {
    // POSIX 不需要清理
}
```

**network_android.cpp：**

```cpp
#include "network.h"
#include <csignal>

void initNetwork() {
    signal(SIGPIPE, SIG_IGN);
    // Android 特有：可以在此检查网络权限或初始化 JNI 网络回调
}

void cleanupNetwork() {
    // Android 不需要清理
}
```

**CMakeLists.txt：**

```cmake
add_library(network STATIC
    network.h
    $<$<PLATFORM_ID:Windows>:network_win32.cpp>
    $<$<PLATFORM_ID:Linux>:network_posix.cpp>
    $<$<PLATFORM_ID:Darwin>:network_posix.cpp>
    $<$<PLATFORM_ID:Android>:network_android.cpp>
)

target_include_directories(network PUBLIC ${CMAKE_CURRENT_SOURCE_DIR})

if(WIN32)
    target_link_libraries(network PRIVATE ws2_32) # WinSock 库
endif()
```

**优势：**
- 每个文件专注一个平台，不需要 `#ifdef`
- 添加新平台只需新增一个 `.cpp` 文件
- 编译时只编译当前平台的文件，减少编译时间
- 代码更清晰、更易测试

</details>
