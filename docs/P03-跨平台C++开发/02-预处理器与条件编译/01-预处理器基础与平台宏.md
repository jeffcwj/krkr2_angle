## 为什么需要学习预处理器

### 一个跨平台项目中的真实场景

假设你正在为 KrKr2 引擎添加一个新功能：读取 XP3 归档文件的修改时间。你已经知道（上一章学的）不同平台的文件 API 不同，所以你需要在同一份 `.cpp` 文件中编写不同平台的实现。

不使用预处理器，你可能会想到这样的方案：

```cpp
// ❌ 方案 1：为每个平台维护一份独立的代码文件
// file_time_windows.cpp  — Windows 实现
// file_time_linux.cpp    — Linux 实现  
// file_time_macos.cpp    — macOS 实现
// file_time_android.cpp  — Android 实现
```

这意味着四份文件、四倍的维护成本。每次修改接口签名，都要同步修改四个文件。

使用预处理器的条件编译，你可以在**一份文件**中完成：

```cpp
// ✅ 方案 2：一份文件，预处理器负责选择正确的实现
#include <cstdint>
#include <string>

// 获取文件最后修改时间（返回 Unix 时间戳）
int64_t getFileModTime(const std::string& path) {
#if defined(_WIN32)
    // Windows: 使用 GetFileAttributesEx
    WIN32_FILE_ATTRIBUTE_DATA data;
    GetFileAttributesExA(path.c_str(), GetFileExInfoStandard, &data);
    // 将 FILETIME 转换为 Unix 时间戳...
    return convertFileTime(data.ftLastWriteTime);
#elif defined(__ANDROID__)
    // Android: 使用 POSIX stat（注意：先检查 Android 再检查 Linux）
    struct stat st;
    stat(path.c_str(), &st);
    return static_cast<int64_t>(st.st_mtime);
#elif defined(__linux__) || defined(__APPLE__)
    // Linux/macOS: 也使用 POSIX stat，但 macOS 支持纳秒精度
    struct stat st;
    stat(path.c_str(), &st);
    return static_cast<int64_t>(st.st_mtime);
#else
    #error "Unsupported platform"
#endif
}
```

编译器在 Windows 上编译时，只会看到 `#if defined(_WIN32)` 分支的代码；在 Linux 上编译时，只会看到 `__linux__` 分支。**其他分支的代码不存在**——连语法检查都不会做。

这就是预处理器的核心价值：**在编译前选择代码**，让一份源文件能在多个平台上编译。

## 预处理器基础

### 编译流程中的预处理阶段

C++ 的编译过程可以分为四个阶段。预处理是第一步，发生在编译器真正"理解"代码之前：

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  源代码   │ →  │  预处理器  │ →  │  编译器   │ →  │  链接器   │
│  (.cpp)  │    │ 文本替换   │    │ 生成.o   │    │ 生成exe  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
                     │
                     ├── #include → 插入头文件内容
                     ├── #define  → 宏替换
                     ├── #if/#ifdef → 条件选择代码段
                     └── #pragma  → 编译器特定指令
```

**关键理解**：预处理器是一个**纯文本处理工具**。它不理解 C++ 的语法、类型或作用域——它只做文本替换和条件选择。这意味着：

- `#define` 定义的宏不遵守 C++ 的作用域规则
- 预处理器不做类型检查
- 宏展开可能产生意想不到的文本组合

### 查看预处理结果

在学习预处理器时，一个非常有用的调试技巧是查看预处理后的代码。你可以让编译器只执行预处理步骤，输出结果：

```bash
# GCC/Clang: -E 选项只做预处理，输出到标准输出
g++ -E -P my_file.cpp > preprocessed.cpp
# -P 选项去掉行号标记，让输出更干净

# MSVC: /P 选项输出预处理结果到 .i 文件
cl /P /EP my_file.cpp
# 生成 my_file.i

# 只查看定义了哪些宏（不处理源文件）
g++ -dM -E - < /dev/null     # Linux/macOS: 列出所有预定义宏
cl /EP /Zc:preprocessor /PD nul  # MSVC: 列出预定义宏
```

让我们来实际试一下。创建一个简单的测试文件：

```cpp
// preprocess_test.cpp
#include <iostream>

#if defined(_WIN32)
    const char* PLATFORM = "Windows";
#elif defined(__linux__)
    const char* PLATFORM = "Linux";
#elif defined(__APPLE__)
    const char* PLATFORM = "macOS";
#endif

int main() {
    std::cout << "Platform: " << PLATFORM << std::endl;
    return 0;
}
```

在 Linux 上运行 `g++ -E -P preprocess_test.cpp | tail -20`，你会看到（省略了 `<iostream>` 展开的几千行标准库代码）：

```cpp
// 预处理后的代码（Linux 上）
    const char* PLATFORM = "Linux";   // 只剩下 Linux 分支

int main() {
    std::cout << "Platform: " << PLATFORM << std::endl;
    return 0;
}
```

Windows 和 macOS 的分支**完全消失了**——它们不会被编译器看到，也不会被语法检查。

## 条件编译指令详解

### 完整指令一览

| 指令 | 含义 | 使用场景 |
|------|------|----------|
| `#define MACRO` | 定义宏（无值） | 用作"开关"，如 `#define DEBUG_MODE` |
| `#define MACRO value` | 定义宏（有值） | 传递数字或字符串，如 `#define VERSION 3` |
| `#define MACRO(x)` | 定义函数式宏 | 简单的代码生成，如 `#define MAX(a,b) ((a)>(b)?(a):(b))` |
| `#undef MACRO` | 取消宏定义 | 在特定范围内关闭某个宏 |
| `#ifdef MACRO` | 如果宏**已定义** | 检查单个宏 |
| `#ifndef MACRO` | 如果宏**未定义** | 头文件保护、检查缺失宏 |
| `#if expr` | 如果表达式为真 | 复杂条件判断 |
| `#elif expr` | 否则如果 | 多分支条件 |
| `#else` | 否则 | 默认分支 |
| `#endif` | 结束条件块 | 必须与 `#if`/`#ifdef`/`#ifndef` 配对 |
| `#error "msg"` | 产生编译错误 | 不支持的平台/配置时终止编译 |
| `#warning "msg"` | 产生编译警告 | 提醒（MSVC 不支持，需用 `#pragma message`） |
| `#pragma` | 编译器特定指令 | 对齐、警告控制等 |

### `#ifdef` vs `#if defined()` —— 选哪个

这两种写法在检查单个宏时完全等价：

```cpp
// 以下两种写法完全等价：
#ifdef _WIN32
    // Windows 代码
#endif

#if defined(_WIN32)
    // Windows 代码
#endif
```

但 `#if defined()` 有一个巨大优势：**支持逻辑运算符**。

```cpp
// ✅ #if defined 可以组合多个条件
#if defined(_WIN32) && !defined(_WIN64)
    // 仅 32 位 Windows
#endif

#if defined(__linux__) || defined(__APPLE__)
    // Unix-like 系统（Linux 或 macOS）
#endif

#if defined(__ANDROID__) && __ANDROID_API__ >= 24
    // Android 7.0 (API 24) 及以上
#endif

// ❌ #ifdef 无法做到上面这些——它只能检查单个宏
// #ifdef _WIN32 && !_WIN64    ← 语法错误！
```

**推荐做法**：
- 简单的单宏判断 → `#ifdef` / `#ifndef`（更简洁）
- 需要 AND/OR/NOT 组合 → `#if defined()`（更灵活）
- KrKr2 项目中两种都有使用，但复杂条件一律用 `#if defined()`

### `#ifndef` 与头文件保护

`#ifndef`（"if not defined"）最常见的用途是**头文件保护**（Include Guard），防止头文件被重复包含：

```cpp
// my_header.h
#ifndef MY_HEADER_H    // 如果 MY_HEADER_H 尚未定义
#define MY_HEADER_H    // 定义它（下次包含时就跳过了）

// 头文件的实际内容...
class MyClass {
    // ...
};

#endif  // MY_HEADER_H
```

原理：第一次 `#include "my_header.h"` 时，`MY_HEADER_H` 未定义，所以进入 `#ifndef` 块，定义该宏并包含内容。第二次 `#include` 时，宏已存在，`#ifndef` 为假，整个文件被跳过。

现代替代方案是 `#pragma once`，效果相同但更简洁：

```cpp
// my_header.h
#pragma once  // 编译器保证此文件只被包含一次

class MyClass {
    // ...
};
```

KrKr2 项目中大量使用 `#pragma once`（因为所有目标编译器都支持）。

### `#error` 和 `#warning` —— 编译期消息

当代码遇到不支持的配置时，可以用 `#error` 直接终止编译并给出错误信息：

```cpp
#if !defined(_WIN32) && !defined(__linux__) && !defined(__APPLE__) && !defined(__ANDROID__)
    #error "KrKr2 不支持当前平台。支持的平台: Windows, Linux, macOS, Android"
#endif

// 检查 C++ 标准版本
#if __cplusplus < 201703L && (!defined(_MSC_VER) || _MSVC_LANG < 201703L)
    #error "KrKr2 需要 C++17 或更高版本。请在 CMake 中设置 CMAKE_CXX_STANDARD 17"
#endif
```

`#warning` 不终止编译，只是提醒开发者：

```cpp
#if defined(__APPLE__) && defined(__arm__)
    #warning "iOS ARM32 支持已弃用，将在下个版本移除"
#endif
```

> **注意**：MSVC 不支持 `#warning` 指令。要在 MSVC 中输出编译期警告，使用 `#pragma message("...")`：
>
> ```cpp
> #if defined(_MSC_VER)
>     #pragma message("注意: 使用 MSVC 编译")
> #else
>     #warning "注意: 使用非 MSVC 编译器"
> #endif
> ```

## 各平台预定义宏速查表

编译器会自动定义一系列宏，无需你手动 `#define`。记住这些宏是编写跨平台代码的基础。

### 操作系统宏

| 宏名 | Windows (MSVC) | Linux (GCC/Clang) | macOS (Apple Clang) | Android (NDK Clang) |
|------|:-:|:-:|:-:|:-:|
| `_WIN32` | ✅ | ❌ | ❌ | ❌ |
| `_WIN64` | ✅ (64位) | ❌ | ❌ | ❌ |
| `__linux__` | ❌ | ✅ | ❌ | ✅ ⚠️ |
| `__APPLE__` | ❌ | ❌ | ✅ | ❌ |
| `__MACH__` | ❌ | ❌ | ✅ | ❌ |
| `__ANDROID__` | ❌ | ❌ | ❌ | ✅ |
| `__ANDROID_API__` | ❌ | ❌ | ❌ | ✅ (数字) |
| `__unix__` | ❌ | ✅ | ✅ | ✅ |

> ⚠️ **陷阱：Android 也定义 `__linux__`**！因为 Android 的内核是 Linux。如果你用 `#ifdef __linux__` 来判断"是否是桌面 Linux"，Android 也会匹配。必须先检查 `__ANDROID__`。

#### 正确的平台判断顺序

这是一个关键模式，在跨平台项目中你会反复使用：

```cpp
// ✅ 正确的判断顺序：特殊平台在前，通用平台在后
#if defined(__ANDROID__)
    // Android 专用代码
    // 必须在 __linux__ 之前检查！
#elif defined(__linux__)
    // 桌面 Linux 专用代码
#elif defined(__APPLE__)
    #include <TargetConditionals.h>
    #if TARGET_OS_IPHONE
        // iOS 代码（KrKr2 暂未支持）
    #else
        // macOS 代码
    #endif
#elif defined(_WIN32)
    // Windows 代码（_WIN32 在 32 位和 64 位都定义）
#else
    #error "Unsupported platform"
#endif
```

```cpp
// ❌ 错误的判断顺序
#if defined(__linux__)
    // 这段代码在 Android 上也会执行！
    // 因为 Android 也定义了 __linux__
#elif defined(__ANDROID__)
    // 永远不会走到这里！
#endif
```

### 编译器宏

| 宏名 | 含义 | 示例值 | 说明 |
|------|------|--------|------|
| `_MSC_VER` | MSVC 编译器版本 | `1940` | VS 2022 17.10 |
| `_MSC_FULL_VER` | MSVC 完整版本 | `194000000` | 含补丁号 |
| `__GNUC__` | GCC 主版本号 | `13` | Clang 也定义此宏！ |
| `__GNUC_MINOR__` | GCC 次版本号 | `2` | |
| `__clang__` | 是否 Clang 编译器 | 已定义 | 包括 Apple Clang |
| `__clang_major__` | Clang 主版本号 | `17` | |

> ⚠️ **陷阱：Clang 也定义了 `__GNUC__`**！因为 Clang 兼容 GCC 的很多扩展。判断编译器时必须先检查 `__clang__`：

```cpp
// ✅ 正确的编译器判断顺序
#if defined(_MSC_VER)
    // MSVC 编译器
    #pragma warning(disable: 4996)  // 禁用 "unsafe" 函数警告
#elif defined(__clang__)
    // Clang 编译器（必须在 __GNUC__ 之前！）
    #pragma clang diagnostic ignored "-Wdeprecated-declarations"
#elif defined(__GNUC__)
    // GCC 编译器
    #pragma GCC diagnostic ignored "-Wdeprecated-declarations"
#endif
```

### 架构宏

| 宏名 (GCC/Clang) | 宏名 (MSVC) | 含义 |
|-------------------|-------------|------|
| `__x86_64__` | `_M_X64` | x86-64 架构 |
| `__i386__` | `_M_IX86` | x86 32位架构 |
| `__aarch64__` | `_M_ARM64` | ARM 64位架构 |
| `__arm__` | `_M_ARM` | ARM 32位架构 |

注意同一个架构在不同编译器中使用不同的宏名。跨编译器的正确写法：

```cpp
// 跨编译器的架构检测
#if defined(__x86_64__) || defined(_M_X64)
    constexpr const char* ARCH_NAME = "x86_64";
    // 可以使用 SSE/AVX 指令集
#elif defined(__aarch64__) || defined(_M_ARM64)
    constexpr const char* ARCH_NAME = "arm64";
    // 可以使用 NEON 指令集
#elif defined(__i386__) || defined(_M_IX86)
    constexpr const char* ARCH_NAME = "x86";
#elif defined(__arm__) || defined(_M_ARM)
    constexpr const char* ARCH_NAME = "arm32";
#else
    constexpr const char* ARCH_NAME = "unknown";
#endif
```

KrKr2 支持四种架构组合：
- Windows x64（`_M_X64`）
- Linux x64（`__x86_64__`）
- macOS ARM64（`__aarch64__`）
- Android ARM64 和 x86_64（`__aarch64__` 或 `__x86_64__`）

### C++ 标准版本宏

`__cplusplus` 宏的值表示当前使用的 C++ 标准版本：

| 宏值 | 对应标准 |
|------|---------|
| `199711L` | C++98/03（或 MSVC 默认值⚠️） |
| `201103L` | C++11 |
| `201402L` | C++14 |
| `201703L` | C++17 |
| `202002L` | C++20 |
| `202302L` | C++23 |

```cpp
// 根据 C++ 标准版本选择不同的实现
#if __cplusplus >= 202002L
    // C++20: 使用 std::endian 检测字节序
    #include <bit>
    constexpr bool is_little_endian = (std::endian::native == std::endian::little);
#elif __cplusplus >= 201703L
    // C++17: 使用运行时检测
    inline bool isLittleEndian() {
        uint32_t val = 1;
        return *reinterpret_cast<uint8_t*>(&val) == 1;
    }
#endif
```

> ⚠️ **MSVC 的 `__cplusplus` 陷阱**：默认情况下，MSVC 的 `__cplusplus` 始终返回 `199711L`（即 C++98），**无论你设置了什么标准版本**。这是一个历史遗留问题。
>
> 要让 MSVC 正确报告 `__cplusplus` 值，需要添加编译选项 `/Zc:__cplusplus`。
>
> 在 CMake 中可以这样处理：
>
> ```cmake
> if(MSVC)
>     # 让 MSVC 正确设置 __cplusplus 的值
>     target_compile_options(my_target PRIVATE /Zc:__cplusplus)
> endif()
> ```
>
> 如果你不想依赖这个选项，可以用 MSVC 特有的 `_MSVC_LANG` 宏代替：
>
> ```cpp
> // 兼容 MSVC 的 C++ 版本检测
> #if __cplusplus >= 202002L || (defined(_MSVC_LANG) && _MSVC_LANG >= 202002L)
>     // C++20 代码
> #endif
> ```

## 实用预处理器模式

### 模式 1：平台专用的头文件包含

不同平台需要包含不同的系统头文件。预处理器让你只包含当前平台需要的：

```cpp
// platform_headers.h — 统一的平台头文件包含
#pragma once

#if defined(_WIN32)
    #ifndef WIN32_LEAN_AND_MEAN
        #define WIN32_LEAN_AND_MEAN  // 减少 Windows.h 包含的内容
    #endif
    #ifndef NOMINMAX
        #define NOMINMAX  // 防止 Windows.h 定义 min/max 宏
    #endif
    #include <Windows.h>
    #include <direct.h>       // _mkdir
    #include <io.h>           // _access
#elif defined(__ANDROID__)
    #include <android/log.h>  // __android_log_print
    #include <jni.h>          // JNI 接口
    #include <unistd.h>       // POSIX API
    #include <sys/stat.h>
#elif defined(__linux__) || defined(__APPLE__)
    #include <unistd.h>
    #include <sys/stat.h>
    #include <dlfcn.h>        // dlopen/dlsym
    #include <pthread.h>
#endif
```

> **注意 `WIN32_LEAN_AND_MEAN` 和 `NOMINMAX`**：
> - `WIN32_LEAN_AND_MEAN` 阻止 `Windows.h` 包含一堆很少用到的子头文件（Winsock、Cryptography 等），显著加快编译速度
> - `NOMINMAX` 阻止 `Windows.h` 定义 `min` 和 `max` 宏，否则会与 `std::min`/`std::max` 冲突

### 模式 2：跨编译器的属性和修饰符

不同编译器使用不同的语法来表达同一个概念（如"不使用返回值时产生警告"）。预处理器可以统一它们：

```cpp
// compiler_compat.h — 编译器兼容宏
#pragma once

// 标记函数为"必须使用返回值"
#if defined(_MSC_VER)
    #define KR2_NODISCARD _Check_return_
#elif defined(__has_cpp_attribute)
    #if __has_cpp_attribute(nodiscard)
        #define KR2_NODISCARD [[nodiscard]]
    #else
        #define KR2_NODISCARD __attribute__((warn_unused_result))
    #endif
#else
    #define KR2_NODISCARD
#endif

// 标记函数为"已弃用"
#if __cplusplus >= 201402L
    #define KR2_DEPRECATED(msg) [[deprecated(msg)]]
#elif defined(_MSC_VER)
    #define KR2_DEPRECATED(msg) __declspec(deprecated(msg))
#elif defined(__GNUC__) || defined(__clang__)
    #define KR2_DEPRECATED(msg) __attribute__((deprecated(msg)))
#else
    #define KR2_DEPRECATED(msg)
#endif

// 标记函数为"不会返回"（如 exit、abort）
#if __cplusplus >= 201103L
    #define KR2_NORETURN [[noreturn]]
#elif defined(_MSC_VER)
    #define KR2_NORETURN __declspec(noreturn)
#elif defined(__GNUC__)
    #define KR2_NORETURN __attribute__((noreturn))
#else
    #define KR2_NORETURN
#endif

// 导出/导入符号（Windows DLL）
#if defined(_WIN32)
    #define KR2_EXPORT __declspec(dllexport)
    #define KR2_IMPORT __declspec(dllimport)
#elif defined(__GNUC__) || defined(__clang__)
    #define KR2_EXPORT __attribute__((visibility("default")))
    #define KR2_IMPORT
#else
    #define KR2_EXPORT
    #define KR2_IMPORT
#endif

// 使用示例：
KR2_NODISCARD int computeHash(const char* data);    // 必须使用返回值
KR2_DEPRECATED("Use newFunction() instead")
void oldFunction();                                   // 标记为弃用
```

### 模式 3：调试与发布的差异化

```cpp
// debug_utils.h
#pragma once
#include <cstdio>

// NDEBUG 在 Release 构建中由 CMake 自动定义
// _DEBUG 在 MSVC 的 Debug 构建中自动定义

#if !defined(NDEBUG) || defined(_DEBUG)
    // Debug 模式：启用断言和日志
    #define KR2_ASSERT(expr, msg)                        \
        do {                                              \
            if (!(expr)) {                                \
                std::fprintf(stderr,                      \
                    "ASSERT FAILED: %s\n"                 \
                    "  File: %s, Line: %d\n"              \
                    "  Message: %s\n",                    \
                    #expr, __FILE__, __LINE__, msg);      \
                std::abort();                             \
            }                                             \
        } while (0)

    #define KR2_LOG(fmt, ...)                             \
        std::fprintf(stderr, "[DEBUG] " fmt "\n",         \
                     ##__VA_ARGS__)
#else
    // Release 模式：断言和日志编译为空（零开销）
    #define KR2_ASSERT(expr, msg) ((void)0)
    #define KR2_LOG(fmt, ...) ((void)0)
#endif
```

**关键知识点**：
- `#expr` 是**字符串化操作符**，将宏参数转为字符串字面量。`KR2_ASSERT(x > 0, "must be positive")` 展开后，`#expr` 变成 `"x > 0"`
- `__FILE__` 和 `__LINE__` 是编译器预定义宏，展开为当前文件名和行号
- `do { ... } while(0)` 是宏定义中的标准技巧，让宏像普通语句一样使用（在 `if` 后面不会出问题）
- `##__VA_ARGS__` 是 GNU 扩展（GCC/Clang 支持），当可变参数为空时自动移除前面的逗号

## 常见错误与陷阱

### 错误 1：Android/Linux 判断顺序颠倒

```cpp
// ❌ 错误：Android 被错误地当作桌面 Linux 处理
#if defined(__linux__)
    #include <X11/Xlib.h>     // X11 窗口系统 — Android 上不存在！
#elif defined(__ANDROID__)
    #include <android/native_window.h>  // 永远走不到这里
#endif
```

```cpp
// ✅ 正确：Android 在前
#if defined(__ANDROID__)
    #include <android/native_window.h>
#elif defined(__linux__)
    #include <X11/Xlib.h>     // 纯桌面 Linux
#endif
```

### 错误 2：忘记 MSVC 的 `__cplusplus` 问题

```cpp
// ❌ 错误：MSVC 下 __cplusplus 默认是 199711L
#if __cplusplus >= 201703L
    // 在 MSVC 上永远不会进入这个分支（即使你用了 /std:c++17）
    #include <optional>
    #include <string_view>
#endif

// ✅ 正确：同时检查 _MSVC_LANG
#if __cplusplus >= 201703L || (defined(_MSVC_LANG) && _MSVC_LANG >= 201703L)
    #include <optional>
    #include <string_view>
#endif
```

### 错误 3：Clang 被当作 GCC

```cpp
// ❌ 错误：Clang 也定义 __GNUC__，导致使用了 GCC 专用的内建函数
#if defined(__GNUC__)
    // Clang 也会进入这里！
    int count = __builtin_popcount(value);  // 这个碰巧两者都支持
    // 但如果用了 GCC 独有的扩展就会出问题
#endif

// ✅ 正确：先排除 Clang
#if defined(__clang__)
    // Clang 特定处理
#elif defined(__GNUC__)
    // 纯 GCC
#endif
```

## 动手实践

### 练习：编写一个跨平台/跨编译器的宏定义头文件

创建 `platform_defs.h`，要求：
1. 定义 `KR2_PLATFORM_NAME` 宏，值为当前平台名称字符串
2. 定义 `KR2_COMPILER_NAME` 宏，值为当前编译器名称字符串
3. 定义 `KR2_ARCH_NAME` 宏，值为当前架构名称字符串
4. 如果平台/编译器/架构不支持，用 `#error` 终止编译

然后写一个 `main.cpp` 使用这些宏并打印结果。

```cpp
// platform_defs.h
#pragma once

// 平台检测（注意顺序！）
#if defined(__ANDROID__)
    #define KR2_PLATFORM_NAME "Android"
    #define KR2_PLATFORM_ANDROID 1
#elif defined(__linux__)
    #define KR2_PLATFORM_NAME "Linux"
    #define KR2_PLATFORM_LINUX 1
#elif defined(__APPLE__)
    #define KR2_PLATFORM_NAME "macOS"
    #define KR2_PLATFORM_MACOS 1
#elif defined(_WIN32)
    #define KR2_PLATFORM_NAME "Windows"
    #define KR2_PLATFORM_WINDOWS 1
#else
    #error "不支持的平台。KrKr2 支持 Windows/Linux/macOS/Android"
#endif

// 编译器检测（注意顺序！）
#if defined(_MSC_VER)
    #define KR2_COMPILER_NAME "MSVC"
    #define KR2_COMPILER_MSVC 1
#elif defined(__clang__)
    #define KR2_COMPILER_NAME "Clang"
    #define KR2_COMPILER_CLANG 1
#elif defined(__GNUC__)
    #define KR2_COMPILER_NAME "GCC"
    #define KR2_COMPILER_GCC 1
#else
    #error "不支持的编译器。需要 MSVC、GCC 或 Clang"
#endif

// 架构检测
#if defined(__x86_64__) || defined(_M_X64)
    #define KR2_ARCH_NAME "x86_64"
    #define KR2_ARCH_X64 1
#elif defined(__aarch64__) || defined(_M_ARM64)
    #define KR2_ARCH_NAME "arm64"
    #define KR2_ARCH_ARM64 1
#elif defined(__i386__) || defined(_M_IX86)
    #define KR2_ARCH_NAME "x86"
    #define KR2_ARCH_X86 1
#elif defined(__arm__) || defined(_M_ARM)
    #define KR2_ARCH_NAME "arm"
    #define KR2_ARCH_ARM 1
#else
    #error "不支持的架构"
#endif

// 简要信息宏
#define KR2_PLATFORM_INFO \
    KR2_PLATFORM_NAME " / " KR2_COMPILER_NAME " / " KR2_ARCH_NAME
```

```cpp
// main.cpp
#include "platform_defs.h"
#include <iostream>

int main() {
    std::cout << "KrKr2 平台信息: " << KR2_PLATFORM_INFO << std::endl;
    std::cout << "平台:   " << KR2_PLATFORM_NAME << std::endl;
    std::cout << "编译器: " << KR2_COMPILER_NAME << std::endl;
    std::cout << "架构:   " << KR2_ARCH_NAME << std::endl;

    // 演示平台专用代码
#if defined(KR2_PLATFORM_WINDOWS)
    std::cout << "Windows 特有: 使用 Win32 API" << std::endl;
#elif defined(KR2_PLATFORM_ANDROID)
    std::cout << "Android 特有: 使用 JNI" << std::endl;
#elif defined(KR2_PLATFORM_LINUX)
    std::cout << "Linux 特有: 使用 POSIX API" << std::endl;
#elif defined(KR2_PLATFORM_MACOS)
    std::cout << "macOS 特有: 使用 Cocoa/Metal" << std::endl;
#endif

    return 0;
}
```

编译并运行：

```bash
# Windows
cl /std:c++17 /EHsc main.cpp /Fe:main.exe && main.exe
# 输出: KrKr2 平台信息: Windows / MSVC / x86_64

# Linux
g++ -std=c++17 -o main main.cpp && ./main
# 输出: KrKr2 平台信息: Linux / GCC / x86_64

# macOS
clang++ -std=c++17 -o main main.cpp && ./main
# 输出: KrKr2 平台信息: macOS / Clang / arm64
```

krkr2/CMakeLists.txt
```

### 平台抽象层的条件编译

```
cpp/core/environ/          — 按平台分子目录
cpp/core/environ/win32/    — Windows 专用实现
cpp/core/environ/linux/    — Linux 专用实现  
cpp/core/environ/android/  — Android 专用实现
cpp/core/environ/apple/    — macOS 专用实现

KrKr2 使用"编译期文件分离"模式：不是在一个文件中用 `#ifdef` 分隔代码，而是为每个平台创建独立的 `.cpp` 文件，通过 CMake 选择编译哪些文件。这是第三章"平台抽象层设计模式"的核心内容。

项目中可以看到处理 MSVC 与 GCC/Clang 差异的预处理器代码，特别是在处理以下方面：
- `__declspec(dllexport)` vs `__attribute__((visibility("default")))`
- `#pragma warning` vs `#pragma GCC diagnostic`
- 各编译器对 C++ 标准特性的支持差异

## 本节小结

- **预处理器**是 C++ 编译的第一步，在编译器之前执行纯文本替换和条件选择
- `#ifdef` 适合单宏检查；`#if defined()` 支持逻辑组合（AND/OR/NOT），更灵活
- **操作系统宏判断顺序至关重要**：Android 必须在 Linux 之前检查，Clang 必须在 GCC 之前检查
- 每个平台/编译器/架构都有独特的预定义宏，编写跨平台代码时需要同时考虑
- MSVC 的 `__cplusplus` 默认不正确，需要 `/Zc:__cplusplus` 或检查 `_MSVC_LANG`
- 预处理器调试技巧：`g++ -E -P` 或 `cl /P /EP` 可以查看预处理后的实际代码
- `WIN32_LEAN_AND_MEAN` 和 `NOMINMAX` 是包含 `Windows.h` 时的必备防护宏

## 练习题与答案

### 题目 1：预定义宏判断

以下代码在 Android ARM64 设备上编译运行，会输出什么？

```cpp
#include <iostream>

int main() {
#if defined(__linux__)
    std::cout << "A ";
#endif
#if defined(__ANDROID__)
    std::cout << "B ";
#endif
#if defined(__aarch64__)
    std::cout << "C ";
#endif
#if defined(_WIN32)
    std::cout << "D ";
#endif
    std::cout << std::endl;
    return 0;
}
```

<details>
<summary>查看答案</summary>

输出：`A B C`

**解析**：
- `__linux__`：Android 的内核是 Linux，所以 `__linux__` 被定义 → 输出 A
- `__ANDROID__`：Android NDK 编译器定义此宏 → 输出 B
- `__aarch64__`：ARM64 架构 → 输出 C
- `_WIN32`：不是 Windows → D 不输出

注意这里用了四个独立的 `#if`（不是 `#elif`），所以每个条件都独立检查。如果用 `#elif`，A 匹配后 B 和 C 就被跳过了——这正是为什么平台判断时需要注意 `#if` 和 `#elif` 的区别。

</details>

### 题目 2：找出 Bug

以下代码的目标是在 MSVC 上禁用特定警告，在 GCC 上使用另一种方式。但在用 Clang 编译时产生了意外行为。找出问题。

```cpp
#if defined(__GNUC__)
    #pragma GCC diagnostic ignored "-Wunused-parameter"
#elif defined(_MSC_VER)
    #pragma warning(disable: 4100)
#elif defined(__clang__)
    #pragma clang diagnostic ignored "-Wunused-parameter"
#endif
```

<details>
<summary>查看答案</summary>

**问题**：Clang 也定义了 `__GNUC__`（因为它兼容 GCC），所以 Clang 编译时会进入第一个分支 `#if defined(__GNUC__)`，使用 `#pragma GCC diagnostic`，而永远不会走到 `#elif defined(__clang__)` 分支。

虽然 `#pragma GCC diagnostic` 碰巧在 Clang 上也能工作（因为 Clang 兼容这个 pragma），但这不是你的本意，而且如果使用了 GCC 独有的 pragma 选项就会报警告或错误。

**修复**：将 `__clang__` 的检查放在 `__GNUC__` 之前：

```cpp
#if defined(_MSC_VER)
    #pragma warning(disable: 4100)
#elif defined(__clang__)
    // 必须在 __GNUC__ 之前！
    #pragma clang diagnostic ignored "-Wunused-parameter"
#elif defined(__GNUC__)
    #pragma GCC diagnostic ignored "-Wunused-parameter"
#endif
```

**规则**：编译器判断顺序应该是 MSVC → Clang → GCC（从特殊到通用）。

</details>

### 题目 3：编写兼容 MSVC 的版本检查

编写一段预处理器代码，检测当前是否支持 C++17 的 `<optional>` 头文件。要求在 MSVC（未设置 `/Zc:__cplusplus`）、GCC 和 Clang 上都能正确工作。

<details>
<summary>查看答案</summary>

```cpp
// 兼容所有编译器的 C++17 检测
#if __cplusplus >= 201703L
    // GCC 和 Clang 正确设置 __cplusplus
    #define HAS_CPP17 1
#elif defined(_MSC_VER) && defined(_MSVC_LANG) && _MSVC_LANG >= 201703L
    // MSVC 使用 _MSVC_LANG 代替（即使 __cplusplus 仍是 199711L）
    #define HAS_CPP17 1
#else
    #define HAS_CPP17 0
#endif

#if HAS_CPP17
    #include <optional>
    #include <string_view>
    #include <variant>
    // 可以安全使用 C++17 特性
#else
    #error "This project requires C++17 or later"
#endif

// 测试
#include <iostream>
int main() {
    std::optional<int> value = 42;
    if (value.has_value()) {
        std::cout << "C++17 optional works: " << *value << std::endl;
    }
    return 0;
}
```

**关键点**：
1. 先检查标准的 `__cplusplus`（适用于 GCC/Clang）
2. 再用 `_MSC_VER` 确认是 MSVC，然后检查 `_MSVC_LANG`（MSVC 专用的正确版本宏）
3. 将结果统一为 `HAS_CPP17` 宏，后续代码只需检查一个宏

</details>
