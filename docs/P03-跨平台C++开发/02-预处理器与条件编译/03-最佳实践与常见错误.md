## 从一个代码审查说起

假设你是一名刚加入跨平台项目的开发者，前辈给你发来一份代码审查意见：

> "你这段条件编译写法不好。整个函数都在 `#ifdef` 里，四个平台重复了大量逻辑。
> 而且你忘记在 `#else` 末尾检查未知平台了——如果有人在 FreeBSD 上编译，
> 代码会静默跳过所有实现，链接时才报'未定义符号'错误，到那时候根本找不到原因。"

这段审查意见提到了跨平台 C++ 开发中条件编译的两个核心问题：

1. **代码重复** — 不同平台的 `#ifdef` 分支中包含大量相同的逻辑
2. **静默失败** — 没有处理"未知平台"的情况，导致编译通过但运行时出 Bug

本节将系统性地讲解如何写出**可维护、可扩展、不容易出错**的条件编译代码。我们会从 5 个最佳实践原则讲起，然后逐一拆解 5 种最常见的条件编译错误。

## 最佳实践 1：最小化条件编译范围

### 问题：整个函数都在 `#ifdef` 中

初学者最常犯的错误，就是把整个函数体都包裹在 `#ifdef ... #else ... #endif` 中：

```cpp
// ❌ 差：整个函数体都在条件编译中
// 四个平台 × 100 行 = 400 行重复代码
#ifdef _WIN32
void processFile(const std::string& path) {
    // 1) 打开文件 — Windows API
    HANDLE hFile = CreateFileA(path.c_str(), GENERIC_READ, 0,
                               nullptr, OPEN_EXISTING, 0, nullptr);
    if (hFile == INVALID_HANDLE_VALUE) {
        std::cerr << "Failed to open: " << path << std::endl;
        return;
    }

    // 2) 读取内容 — 这段逻辑其实和平台无关！
    char buffer[4096];
    DWORD bytesRead = 0;
    ReadFile(hFile, buffer, sizeof(buffer), &bytesRead, nullptr);

    // 3) 处理数据 — 与平台完全无关，但被迫写在 #ifdef 里
    for (DWORD i = 0; i < bytesRead; ++i) {
        buffer[i] = std::toupper(buffer[i]);
    }
    std::cout << "Processed " << bytesRead << " bytes" << std::endl;

    // 4) 关闭文件 — Windows API
    CloseHandle(hFile);
}
#else
void processFile(const std::string& path) {
    // 1) 打开文件 — POSIX API
    int fd = open(path.c_str(), O_RDONLY);
    if (fd < 0) {
        std::cerr << "Failed to open: " << path << std::endl;
        return;
    }

    // 2) 读取内容 — 注意：步骤 3 和 Windows 版完全一样！
    char buffer[4096];
    ssize_t bytesRead = ::read(fd, buffer, sizeof(buffer));

    // 3) 处理数据 — 复制粘贴的代码，维护噩梦
    for (ssize_t i = 0; i < bytesRead; ++i) {
        buffer[i] = std::toupper(buffer[i]);
    }
    std::cout << "Processed " << bytesRead << " bytes" << std::endl;

    // 4) 关闭文件 — POSIX API
    close(fd);
}
#endif
```

这段代码的问题一目了然：步骤 2（读取）和步骤 3（处理）的逻辑**完全相同**，却被迫在两个 `#ifdef` 分支中各写了一遍。如果将来要修改处理逻辑（比如改成小写转换），你必须同时修改两处——忘了一处就是 Bug。

### 解决方案：把平台差异封装成最小的类或函数

**核心思路**：只把"必须因平台而异的部分"放在 `#ifdef` 里，其他逻辑保持平台无关。

```cpp
#include <cstddef>  // size_t
#include <iostream>
#include <string>
#include <cctype>

#ifdef _WIN32
    #include <windows.h>
#else
    #include <fcntl.h>   // open
    #include <unistd.h>  // read, close
#endif

// 第一步：用 RAII 类封装平台差异（仅平台 API 在 #ifdef 中）
class FileHandle {
public:
    explicit FileHandle(const std::string& path) {
#ifdef _WIN32
        handle_ = CreateFileA(path.c_str(), GENERIC_READ, 0,
                              nullptr, OPEN_EXISTING, 0, nullptr);
#else
        fd_ = open(path.c_str(), O_RDONLY);
#endif
    }

    // 是否打开成功
    bool isValid() const {
#ifdef _WIN32
        return handle_ != INVALID_HANDLE_VALUE;
#else
        return fd_ >= 0;
#endif
    }

    // 读取数据
    size_t read(char* buffer, size_t size) {
#ifdef _WIN32
        DWORD bytesRead = 0;
        ReadFile(handle_, buffer, static_cast<DWORD>(size),
                 &bytesRead, nullptr);
        return static_cast<size_t>(bytesRead);
#else
        ssize_t result = ::read(fd_, buffer, size);
        return result > 0 ? static_cast<size_t>(result) : 0;
#endif
    }

    ~FileHandle() {
#ifdef _WIN32
        if (handle_ != INVALID_HANDLE_VALUE) CloseHandle(handle_);
#else
        if (fd_ >= 0) close(fd_);
#endif
    }

    // 禁止拷贝（文件句柄不可复制）
    FileHandle(const FileHandle&) = delete;
    FileHandle& operator=(const FileHandle&) = delete;

private:
#ifdef _WIN32
    HANDLE handle_ = INVALID_HANDLE_VALUE;
#else
    int fd_ = -1;
#endif
};

// 第二步：处理逻辑完全平台无关——没有任何 #ifdef！
void processFile(const std::string& path) {
    FileHandle file(path);
    if (!file.isValid()) {
        std::cerr << "Failed to open: " << path << std::endl;
        return;
    }

    char buffer[4096];
    size_t bytesRead = file.read(buffer, sizeof(buffer));

    // 数据处理逻辑只写一次，永远不会出现"忘改一处"的问题
    for (size_t i = 0; i < bytesRead; ++i) {
        buffer[i] = static_cast<char>(std::toupper(buffer[i]));
    }
    std::cout << "Processed " << bytesRead << " bytes" << std::endl;
    // FileHandle 析构函数自动关闭文件句柄
}
```

**效果对比**：

| 维度 | ❌ 整个函数 `#ifdef` | ✅ 封装后 |
|------|---------------------|----------|
| 平台无关代码重复 | 每个平台重复一次 | 只写一次 |
| 修改处理逻辑需改几处 | N 个平台改 N 次 | 只改 1 处 |
| 新增平台需要做什么 | 复制整个函数 | 只加 `#ifdef` 分支 |
| 代码行数（假设 4 平台） | ~400 行 | ~130 行 |

> **KrKr2 项目中的实践：** KrKr2 的文件系统层就使用了这种封装模式——`environ/` 目录下按平台分目录（`win32/`、`linux/`、`android/`、`apple/`），每个平台只实现差异部分，公共逻辑在 `cocos2d/` 中共享。

## 最佳实践 2：大块平台差异用文件分离

### 什么时候 `#ifdef` 不够用

上面的 `FileHandle` 类只有几十行平台差异代码，用 `#ifdef` 内联是合适的。但如果平台差异很大——比如一个平台的实现有 200 行，另一个平台有 300 行——继续用 `#ifdef` 就会让文件变得又长又难读。

**判断标准**：

| 平台差异规模 | 推荐方式 |
|-------------|---------|
| < 30 行差异 | `#ifdef` 内联 |
| 30-100 行差异 | 封装成函数/类，`#ifdef` 放在实现内部 |
| > 100 行差异 | **文件分离** |

### 文件分离的目录结构

```
src/
├── audio_player.h              ← 公共接口（纯声明，没有 #ifdef）
├── audio_player_common.cpp     ← 平台无关的公共实现
├── audio_player_win32.cpp      ← Windows: DirectSound / WASAPI
├── audio_player_posix.cpp      ← Linux/macOS: ALSA / CoreAudio
└── audio_player_android.cpp    ← Android: OpenSL ES / AAudio
```

头文件只声明接口，完全没有 `#ifdef`：

```cpp
// audio_player.h — 干净的接口，没有任何平台特定内容
#pragma once
#include <string>
#include <cstdint>

class AudioPlayer {
public:
    AudioPlayer();
    ~AudioPlayer();

    bool open(const std::string& filePath);
    void play();
    void pause();
    void stop();
    float getVolume() const;
    void setVolume(float vol);  // 0.0 ~ 1.0

private:
    struct Impl;           // PIMPL 模式，隐藏平台实现细节
    Impl* impl_ = nullptr;
};
```

然后在 CMake 中用 Generator Expressions 选择编译哪个文件：

```cmake
# CMakeLists.txt
add_library(audio_player
    audio_player.h
    audio_player_common.cpp         # 所有平台都编译
    # 以下只有对应平台才会编译：
    $<$<PLATFORM_ID:Windows>:audio_player_win32.cpp>
    $<$<PLATFORM_ID:Linux>:audio_player_posix.cpp>
    $<$<PLATFORM_ID:Darwin>:audio_player_posix.cpp>  # macOS 和 Linux 共用
    $<$<PLATFORM_ID:Android>:audio_player_android.cpp>
)
```

**文件分离的优势**：

1. **每个文件只包含一个平台的代码** — 打开 `audio_player_win32.cpp` 就知道这是 Windows 实现，不需要在 `#ifdef` 之间跳来跳去
2. **编译器只看到当前平台的代码** — 不会因为语法错误在非当前平台的 `#ifdef` 分支中而浪费排查时间
3. **代码审查更清晰** — PR 只修改了 `audio_player_android.cpp`，审查者立刻知道只影响 Android
4. **IDE 支持更好** — 代码补全、跳转定义只显示当前平台的实现

> **KrKr2 项目中的实践：** `cpp/core/environ/` 目录就是这种模式——`win32/`、`linux/`、`android/`、`apple/` 各自包含该平台的完整实现，CMake 根据目标平台选择编译哪个目录。

## 最佳实践 3：用 `#pragma once` 防止头文件重复包含

### 问题：头文件被包含多次

在大型项目中，一个头文件可能被多个 `.cpp` 文件包含，而这些头文件之间又互相包含。如果不做保护，同一个头文件的内容可能被编译器看到多次，导致"重复定义"错误：

```cpp
// math_types.h — 没有任何保护
struct Vector3 {
    float x, y, z;
};
```

```cpp
// physics.h
#include "math_types.h"  // 第 1 次包含 Vector3
// ...
```

```cpp
// renderer.h
#include "math_types.h"  // 第 2 次包含 Vector3
// ...
```

```cpp
// main.cpp
#include "physics.h"     // 间接包含了 math_types.h
#include "renderer.h"    // 又包含了 math_types.h → 重复定义！
```

编译 `main.cpp` 时，编译器会报错：

```
error: redefinition of 'struct Vector3'
```

### 两种解决方案

**方案 1：传统 Include Guard**

```cpp
// math_types.h — 使用 Include Guard
#ifndef MATH_TYPES_H    // 第一次包含时未定义 → 进入
#define MATH_TYPES_H    // 定义宏，下次再包含时跳过

struct Vector3 {
    float x, y, z;
};

#endif // MATH_TYPES_H  // 第二次包含时已定义 → 直接跳到这里
```

**方案 2：`#pragma once`（推荐）**

```cpp
// math_types.h — 使用 #pragma once
#pragma once            // 告诉编译器：这个文件只包含一次

struct Vector3 {
    float x, y, z;
};
```

### 两种方式的对比

| 维度 | Include Guard | `#pragma once` |
|------|---------------|----------------|
| 标准性 | C/C++ 标准正式支持 | 非标准，但所有主流编译器都支持 |
| 写法 | 3 行样板代码 | 1 行 |
| 容易出错吗 | 是（宏名冲突、拼写错误） | 否 |
| 编译速度 | 略慢（仍需打开文件检查宏） | 略快（编译器可直接跳过文件） |
| 符号链接/拷贝文件 | 正常工作 | 极少数情况可能出问题 |
| GCC 支持 | ✅ | ✅ |
| Clang 支持 | ✅ | ✅ |
| MSVC 支持 | ✅ | ✅ |
| ARM/Android NDK | ✅ | ✅ |

**实际选择**：绝大多数现代 C++ 项目都使用 `#pragma once`，因为它简洁、不容易出错。KrKr2 项目的所有头文件都使用 `#pragma once`。

> **Include Guard 的命名陷阱：** 如果两个不同的头文件不小心用了相同的宏名，其中一个会被静默跳过。比如你的项目有 `math/types.h` 和 `physics/types.h`，如果都用 `#ifndef TYPES_H`，第二个被包含时就会被跳过，导致莫名其妙的编译错误。

### Include Guard 的命名规范（如果你必须使用）

```cpp
// 规范：项目名_目录_文件名_H_
// 全大写，路径分隔符用下划线

// 文件: krkr2/core/visual/types.h
#ifndef KRKR2_CORE_VISUAL_TYPES_H_
#define KRKR2_CORE_VISUAL_TYPES_H_

// ...

#endif // KRKR2_CORE_VISUAL_TYPES_H_
```

## 最佳实践 4：用 `#error` 捕获未处理的平台

### 问题：静默编译导致延迟报错

这是一个非常隐蔽的问题。看下面的代码：

```cpp
// ❌ 危险：没有 #else 分支
int getProcessorCount() {
#if defined(_WIN32)
    SYSTEM_INFO info;
    GetSystemInfo(&info);
    return info.dwNumberOfProcessors;
#elif defined(__linux__) || defined(__ANDROID__)
    return sysconf(_SC_NPROCESSORS_ONLN);
#elif defined(__APPLE__)
    int count = 0;
    size_t size = sizeof(count);
    sysctlbyname("hw.ncpu", &count, &size, nullptr, 0);
    return count;
#endif
    // 如果在 FreeBSD 上编译？
    // 编译器不报错，但函数没有 return → 未定义行为！
}
```

如果有人在 FreeBSD 上编译这段代码：
- 编译器可能只给一个 warning（"control reaches end of non-void function"）
- 代码链接成功，运行时返回垃圾值
- 你花了三天才发现是这个函数的问题

### 解决方案：`#error` 主动报错

```cpp
// ✅ 安全：未知平台立即报编译错误
int getProcessorCount() {
#if defined(_WIN32)
    SYSTEM_INFO info;
    GetSystemInfo(&info);
    return info.dwNumberOfProcessors;
#elif defined(__ANDROID__)
    // 注意：Android 必须在 __linux__ 前面！
    return sysconf(_SC_NPROCESSORS_ONLN);
#elif defined(__linux__)
    return sysconf(_SC_NPROCESSORS_ONLN);
#elif defined(__APPLE__)
    int count = 0;
    size_t size = sizeof(count);
    sysctlbyname("hw.ncpu", &count, &size, nullptr, 0);
    return count;
#else
    #error "getProcessorCount() 不支持当前平台，请添加实现"
#endif
}
```

现在如果有人在 FreeBSD 上编译，**编译阶段就会立即报错**，错误信息明确告诉你哪个函数需要添加新平台支持。

### `#error` 的其他用法

不仅可以检查平台，还可以检查编译配置：

```cpp
// 检查 C++ 标准版本
#if defined(_MSC_VER)
    // MSVC 需要用 _MSVC_LANG 检查（上一节讲过的坑！）
    #if _MSVC_LANG < 201703L
        #error "本项目要求 C++17 或更高版本，请使用 /std:c++17"
    #endif
#else
    #if __cplusplus < 201703L
        #error "本项目要求 C++17 或更高版本，请使用 -std=c++17"
    #endif
#endif

// 检查必要的编译定义是否通过 CMake 传入
#ifndef PROJECT_VERSION
    #error "PROJECT_VERSION 未定义，请检查 CMakeLists.txt 中的 configure_file"
#endif
```

## 最佳实践 5：避免深层嵌套的条件编译

### 问题：三层嵌套变成"读代码地狱"

```cpp
// ❌ 差：三层嵌套，你能一眼看出这段代码在什么条件下编译吗？
#ifdef _WIN32
    #ifdef _DEBUG
        #if _MSC_VER >= 1920
            void enableMemoryLeakDetection() {
                _CrtSetDbgFlag(_CRTDBG_ALLOC_MEM_DF | _CRTDBG_LEAK_CHECK_DF);
            }
        #else
            void enableMemoryLeakDetection() {
                // 旧版 MSVC 使用不同的 API
                _CrtSetReportMode(_CRT_WARN, _CRTDBG_MODE_DEBUG);
            }
        #endif
    #else
        // Release 模式什么都不做
        void enableMemoryLeakDetection() {}
    #endif
#else
    // 非 Windows 什么都不做
    void enableMemoryLeakDetection() {}
#endif
```

这段代码有 3 层嵌套，要搞清楚"哪段代码在什么条件下编译"需要仔细数 `#ifdef` 和 `#endif` 的配对关系。

### 解决方案 1：用复合条件压平嵌套

```cpp
// ✅ 好：用 #if defined() + 逻辑运算符，一行说清条件
#if defined(_WIN32) && defined(_DEBUG) && (_MSC_VER >= 1920)
void enableMemoryLeakDetection() {
    _CrtSetDbgFlag(_CRTDBG_ALLOC_MEM_DF | _CRTDBG_LEAK_CHECK_DF);
}
#elif defined(_WIN32) && defined(_DEBUG)
void enableMemoryLeakDetection() {
    _CrtSetReportMode(_CRT_WARN, _CRTDBG_MODE_DEBUG);
}
#else
void enableMemoryLeakDetection() {
    // 非 Windows 或 Release 模式：不需要内存泄漏检测
}
#endif
```

### 解决方案 2：提取为配置宏

如果条件很复杂，可以先在文件顶部定义配置宏，然后在代码中使用简单的 `#ifdef`：

```cpp
// 在文件顶部集中处理复杂条件
#if defined(_WIN32) && defined(_DEBUG) && (_MSC_VER >= 1920)
    #define HAS_CRT_LEAK_DETECTION 1
    #define LEAK_DETECTION_VERSION 2  // 新版 API
#elif defined(_WIN32) && defined(_DEBUG)
    #define HAS_CRT_LEAK_DETECTION 1
    #define LEAK_DETECTION_VERSION 1  // 旧版 API
#endif

// 代码中使用简洁的宏名 — 不需要再想"什么条件"
#ifdef HAS_CRT_LEAK_DETECTION
void enableMemoryLeakDetection() {
    #if LEAK_DETECTION_VERSION >= 2
        _CrtSetDbgFlag(_CRTDBG_ALLOC_MEM_DF | _CRTDBG_LEAK_CHECK_DF);
    #else
        _CrtSetReportMode(_CRT_WARN, _CRTDBG_MODE_DEBUG);
    #endif
}
#else
void enableMemoryLeakDetection() {}
#endif
```

**核心思想**：把"判断条件"和"实现代码"分离。文件顶部负责"做判断"，下面负责"写实现"。

## 常见错误 1：Android 被 `__linux__` 匹配

### 为什么会出错

Android 系统底层基于 Linux 内核，因此 Android NDK 编译器会**同时定义 `__ANDROID__` 和 `__linux__`**。这意味着：

```cpp
// ❌ 这段代码在 Android 上也会编译！
#if defined(__linux__)
    #include <X11/Xlib.h>  // Android 上根本没有 X11 → 编译失败
    void createWindow() {
        Display* display = XOpenDisplay(nullptr);
        // ...
    }
#endif
```

### 正确做法：Android 检查必须在 Linux 前面

```cpp
// ✅ 正确顺序：先检查 Android，再检查 Linux
#if defined(__ANDROID__)
    // Android 专有代码（不含桌面 Linux 的库）
    #include <android/native_window.h>
    void createWindow() {
        // 使用 ANativeWindow API...
    }
#elif defined(__linux__)
    // 桌面 Linux 代码（可以使用 X11/Wayland）
    #include <X11/Xlib.h>
    void createWindow() {
        Display* display = XOpenDisplay(nullptr);
        // ...
    }
#endif
```

**记忆口诀：** "Android 是特殊的 Linux，特殊情况先判断。" 这和 C++ 中异常捕获的规则一样——派生类的 `catch` 必须在基类前面。

### 完整的平台检查顺序模板

```cpp
// 推荐的四平台检查顺序（从最具体到最通用）
#if defined(_WIN32)
    // Windows（包括 Win64，Win64 也定义 _WIN32）
#elif defined(__ANDROID__)
    // Android（必须在 __linux__ 前面！）
#elif defined(__linux__)
    // 桌面 Linux
#elif defined(__APPLE__)
    #include <TargetConditionals.h>
    #if TARGET_OS_IPHONE
        // iOS
    #else
        // macOS
    #endif
#else
    #error "Unsupported platform"
#endif
```

> **注意 `__APPLE__` 的特殊性：** macOS 和 iOS 都定义 `__APPLE__`。如果你需要区分，必须包含 `<TargetConditionals.h>` 然后检查 `TARGET_OS_IPHONE`。

## 常见错误 2：MSVC 的 `__cplusplus` 宏报告错误值

### 问题描述

C++ 标准要求编译器定义 `__cplusplus` 宏来报告当前使用的标准版本：

| C++ 版本 | `__cplusplus` 应有的值 |
|----------|----------------------|
| C++11 | `201103L` |
| C++14 | `201402L` |
| C++17 | `201703L` |
| C++20 | `202002L` |

但 **MSVC 有一个历史遗留问题**：默认情况下，不管你用的是 C++17 还是 C++20，`__cplusplus` 都报告 `199711L`（C++98 的值！）。

```cpp
#include <iostream>

int main() {
    std::cout << "__cplusplus = " << __cplusplus << std::endl;
    // GCC/Clang 用 -std=c++17 编译 → 输出 201703
    // MSVC 用 /std:c++17 编译 → 输出 199711 ← 不对！
}
```

### 这会导致什么问题

```cpp
// ❌ 在 MSVC 上，这段代码永远不会被编译到
#if __cplusplus >= 201703L
    #include <optional>
    #include <string_view>
    #include <filesystem>
#endif
```

即使你在 MSVC 中开启了 `/std:c++17`，`__cplusplus` 仍然是 `199711L`，导致 C++17 特性被跳过。

### 解决方案

**方案 1：在 CMake 中添加 `/Zc:__cplusplus` 编译选项**

```cmake
# CMakeLists.txt
if(MSVC)
    target_compile_options(my_target PRIVATE /Zc:__cplusplus)
endif()
```

添加后，MSVC 会像 GCC/Clang 一样正确报告 `__cplusplus` 的值。

**方案 2：同时检查 `_MSVC_LANG`（更保险）**

MSVC 提供了一个专有宏 `_MSVC_LANG`，它总是正确报告标准版本：

```cpp
// ✅ 兼容所有编译器的写法
// 定义一个统一的版本检查宏
#if defined(_MSVC_LANG)
    #define CPP_VERSION _MSVC_LANG    // MSVC: 用 _MSVC_LANG
#else
    #define CPP_VERSION __cplusplus   // GCC/Clang: 用 __cplusplus
#endif

#if CPP_VERSION >= 201703L
    #include <optional>
    #include <string_view>
    #include <filesystem>
    #define HAS_CPP17 1
#endif

#if CPP_VERSION >= 202002L
    #include <concepts>
    #include <ranges>
    #define HAS_CPP20 1
#endif
```

## 常见错误 3：忘记 `#endif` 导致代码静默消失

### 问题描述

```cpp
#ifdef _WIN32
void windowsOnlySetup() {
    // Windows 初始化代码...
}
// 忘记写 #endif！

// 下面的所有代码都变成了"仅 Windows"编译
// 但看代码的人根本不会意识到，因为下面的代码没有缩进

void commonFunction() {
    // 这个函数在非 Windows 上完全不存在
    // 链接时报"未定义符号"错误，但你不知道为什么
}

class SharedClass {
    // 整个类在非 Windows 上都消失了...
};
```

这个错误极其隐蔽：
- **编译器不报错**（语法上 `#ifdef` 不要求立即 `#endif`）
- **非 Windows 平台上**，后面所有代码都被跳过
- **错误表现为链接错误**（"undefined reference to commonFunction"），而不是编译错误
- 你会花大量时间检查 `commonFunction` 的实现，而问题根本不在那里

### 预防措施

**规则 1：`#endif` 后面写注释标明对应条件**

```cpp
#ifdef _WIN32
void windowsOnlySetup() {
    // ...
}
#endif // _WIN32   ← 注释说明这个 #endif 对应哪个 #ifdef
```

**规则 2：在写 `#ifdef` 的同时立即写 `#endif`**

养成习惯：先写框架，再填内容。

```cpp
// 先写框架
#ifdef _WIN32
#endif // _WIN32

// 再在中间填入代码
#ifdef _WIN32
void windowsOnlySetup() {
    // ...
}
#endif // _WIN32
```

**规则 3：使用编辑器的括号匹配功能**

大多数现代 IDE（VS Code、CLion、Visual Studio）都支持 `#ifdef` / `#endif` 配对高亮。如果你的 IDE 没有高亮，考虑安装相关插件。

## 常见错误 4：宏参数中的运算符优先级

### 问题描述

预处理器宏是纯文本替换，不理解 C++ 的运算符优先级。这意味着如果宏参数中有表达式，可能产生意想不到的结果：

```cpp
// ❌ 有 Bug 的宏
#define SQUARE(x) x * x

int a = SQUARE(3);      // 展开为 3 * 3 = 9  ✅ 正确
int b = SQUARE(1 + 2);  // 展开为 1 + 2 * 1 + 2 = 5  ❌ 应该是 9！
int c = SQUARE(a++);    // 展开为 a++ * a++ → 未定义行为！
```

`SQUARE(1 + 2)` 被展开为 `1 + 2 * 1 + 2`，由于 `*` 优先级高于 `+`，实际计算是 `1 + (2 * 1) + 2 = 5`，而不是 `(1 + 2) * (1 + 2) = 9`。

### 解决方案：给宏参数和整个表达式加括号

```cpp
// ✅ 安全的宏：参数和整体都加括号
#define SQUARE(x) ((x) * (x))

int a = SQUARE(3);      // ((3) * (3)) = 9  ✅
int b = SQUARE(1 + 2);  // ((1 + 2) * (1 + 2)) = 9  ✅
```

### 更好的方案：用 `constexpr` 函数代替宏

在 C++17 中，几乎所有宏函数都可以用 `constexpr` 函数代替：

```cpp
// ✅✅ 最佳方案：constexpr 函数
// 类型安全、不会有优先级问题、支持调试断点
template<typename T>
constexpr T square(T x) {
    return x * x;
}

int a = square(3);      // 9  ✅
int b = square(1 + 2);  // 9  ✅
// int c = square(a++);  // 编译器会警告——优于宏的静默错误
```

**经验法则**：

| 场景 | 用宏还是 constexpr |
|------|-------------------|
| 简单常量值 | `constexpr` 变量 |
| 计算函数 | `constexpr` 函数 |
| 条件编译（平台/配置） | 只能用宏 `#if` / `#ifdef` |
| 字符串化 `#`、拼接 `##` | 只能用宏 |
| 包含 `#include` 路径拼接 | 只能用宏 |

## 常见错误 5：`#ifdef` vs `#if defined()` 的陷阱

### 问题描述

`#ifdef X` 和 `#if defined(X)` 在检查单个宏时效果相同。但当你需要组合多个条件时，`#ifdef` 就无法使用了：

```cpp
// ❌ 语法错误：#ifdef 不能用逻辑运算符
#ifdef _WIN32 && _DEBUG    // 编译器报错！
    // ...
#endif

// ❌ 语法错误：#ifdef 不能用 || 
#ifdef _WIN32 || __linux__  // 编译器报错！
    // ...
#endif
```

### 正确做法

```cpp
// ✅ 组合条件必须用 #if defined()
#if defined(_WIN32) && defined(_DEBUG)
    // Windows Debug 模式
#endif

#if defined(_WIN32) || defined(__linux__)
    // Windows 或 Linux
#endif

// ✅ 否定条件
#if !defined(__ANDROID__)
    // 非 Android 平台
#endif

// ✅ 复杂组合
#if (defined(_WIN32) || defined(__linux__)) && !defined(__ANDROID__)
    // Windows 或桌面 Linux（排除 Android）
#endif
```

**经验建议**：为了一致性，**统一使用 `#if defined()` 形式**，不要混用 `#ifdef` 和 `#if defined()`。这样当你需要添加第二个条件时，不需要重构整行。

## 本节小结

### 五条最佳实践

| 原则 | 核心要点 | 何时使用 |
|------|---------|---------|
| 最小化条件编译范围 | 只把平台差异部分放在 `#ifdef` 中 | 差异 < 30 行时 |
| 文件分离 | 大块差异放在不同 `.cpp` 文件中 | 差异 > 100 行时 |
| `#pragma once` | 防止头文件重复包含 | 所有头文件 |
| `#error` 捕获未知平台 | 让未支持的平台在编译期报错 | 所有平台分支 |
| 避免深层嵌套 | 用复合条件或配置宏压平嵌套 | 嵌套 > 2 层时 |

### 五种常见错误速查

| 错误 | 症状 | 解决方案 |
|------|------|---------|
| Android 被 `__linux__` 匹配 | Android 编译失败（找不到桌面 Linux 的库） | `__ANDROID__` 检查放在 `__linux__` 前面 |
| MSVC `__cplusplus` 不准确 | C++17 特性在 MSVC 上被跳过 | 用 `_MSVC_LANG` 或添加 `/Zc:__cplusplus` |
| 忘记 `#endif` | 非目标平台代码静默消失，链接报错 | `#endif` 后加注释，先写框架再填内容 |
| 宏参数优先级 | 计算结果错误 | 参数加括号 `((x) * (x))`，或用 `constexpr` |
| `#ifdef` 不能组合条件 | 语法错误 | 统一用 `#if defined()` 形式 |

### 黄金规则

> **条件编译代码的衡量标准：打开任意一个源文件，5 秒内能否看懂"这段代码在什么条件下编译"。** 如果做不到，就需要重构。

## 练习题与答案

### 题目 1：代码审查

以下代码有 3 处条件编译的问题，请指出并修复：

```cpp
#ifdef _WIN32
    #include <windows.h>
#endif

#ifdef __linux__
    #include <sys/stat.h>
#endif

size_t getFileSize(const char* path) {
#ifdef _WIN32
    WIN32_FILE_ATTRIBUTE_DATA data;
    GetFileAttributesExA(path, GetFileExInfoStandard, &data);
    return data.nFileSizeLow;
#endif
#ifdef __linux__
    struct stat st;
    stat(path, &st);
    return st.st_size;
#endif
}
```

<details>
<summary>查看答案</summary>

**问题 1**：`__linux__` 也会在 Android 上定义，但 `sys/stat.h` 在 Android 上的行为可能不同。应该区分 Android 和桌面 Linux。

**问题 2**：没有 macOS 支持。macOS 上 `_WIN32` 和 `__linux__` 都不定义，函数没有 `return` → 未定义行为。

**问题 3**：没有 `#error` 捕获未知平台，也没有使用 `#elif`（两个独立 `#ifdef` 可能同时为真）。

修复后的代码：

```cpp
#if defined(_WIN32)
    #include <windows.h>
#elif defined(__ANDROID__) || defined(__linux__) || defined(__APPLE__)
    #include <sys/stat.h>   // POSIX 三平台都支持 stat
#else
    #error "Unsupported platform for getFileSize"
#endif

size_t getFileSize(const char* path) {
#if defined(_WIN32)
    WIN32_FILE_ATTRIBUTE_DATA data;
    if (!GetFileAttributesExA(path, GetFileExInfoStandard, &data)) {
        return 0;
    }
    return static_cast<size_t>(data.nFileSizeLow) |
           (static_cast<size_t>(data.nFileSizeHigh) << 32);
#elif defined(__ANDROID__) || defined(__linux__) || defined(__APPLE__)
    struct stat st;
    if (stat(path, &st) != 0) {
        return 0;
    }
    return static_cast<size_t>(st.st_size);
#else
    #error "Unsupported platform"
#endif
}
```

注意：修复后还修正了 Windows 版本只用 `nFileSizeLow` 的 Bug（大于 4GB 的文件会得到错误大小），并添加了错误处理。

</details>

### 题目 2：选择最佳策略

对于以下场景，你应该选择"内联 `#ifdef`"还是"文件分离"？请说明理由。

1. 一个 `getClipboardText()` 函数，Windows 用 Win32 API（约 40 行），Linux 用 X11（约 50 行），macOS 用 NSPasteboard（约 30 行），Android 用 JNI 调用 Java（约 80 行）
2. 一个 `sleep_ms(int ms)` 函数，Windows 用 `Sleep(ms)`，POSIX 用 `usleep(ms * 1000)`
3. 一个窗口管理模块，Windows 用 Win32 窗口（500 行），Linux 用 SDL2（400 行），macOS 用 Cocoa（600 行）

<details>
<summary>查看答案</summary>

1. **`getClipboardText()`**：推荐 **文件分离**。四个平台的实现差异大（30-80 行不等），用的 API 完全不同（Win32、X11、NSPasteboard、JNI），且 Android 的 JNI 部分很长。内联 `#ifdef` 会让文件超过 200 行，可读性差。

2. **`sleep_ms()`**：推荐 **内联 `#ifdef`**。平台差异只有一两行，封装成函数后总共不超过 10 行：
   ```cpp
   void sleep_ms(int ms) {
   #ifdef _WIN32
       Sleep(ms);
   #else
       usleep(ms * 1000);
   #endif
   }
   ```

3. **窗口管理模块**：**必须文件分离**。每个平台的实现都有 400-600 行，而且使用完全不同的 API 体系。应该定义公共接口（`window.h`），然后 `window_win32.cpp`、`window_sdl2.cpp`、`window_cocoa.mm` 各自实现。这也是 KrKr2 `environ/` 目录使用的模式。

</details>

### 题目 3：修复宏

以下宏有什么问题？请修复，并给出用 `constexpr` 函数替代的版本。

```cpp
#define CLAMP(val, lo, hi) val < lo ? lo : val > hi ? hi : val
```

<details>
<summary>查看答案</summary>

**问题**：宏参数没有加括号。

```cpp
int result = CLAMP(x + 1, 0, 100);
// 展开为：x + 1 < 0 ? 0 : x + 1 > 100 ? 100 : x + 1
// 运算符优先级问题：+ 高于 <，所以 x + 1 < 0 没问题
// 但如果 lo 或 hi 是表达式就会出错：
int result2 = CLAMP(x, a + b, c * d);
// 展开为：x < a + b ? a + b : x > c * d ? c * d : x
// 这里 a + b 和 c * d 被计算了两次（如果有副作用就是 Bug）
```

修复后的宏：

```cpp
#define CLAMP(val, lo, hi) \
    ((val) < (lo) ? (lo) : (val) > (hi) ? (hi) : (val))
```

`constexpr` 替代版本（推荐）：

```cpp
template<typename T>
constexpr T clamp(T val, T lo, T hi) {
    return val < lo ? lo : (val > hi ? hi : val);
}

// C++17 标准库已有 std::clamp：
#include <algorithm>
int result = std::clamp(x + 1, 0, 100);  // 最佳选择
```

注意：C++17 标准库已经提供了 `std::clamp`，实际项目中应直接使用标准库，不需要自己写宏或函数。

</details>
