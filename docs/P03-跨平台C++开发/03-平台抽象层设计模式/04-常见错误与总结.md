## PAL 开发中的五大常见错误

### 错误一：平台头文件泄漏到公共接口

**场景**：你在写一个跨平台文件操作模块，想在头文件里用 Windows 的 `HANDLE` 类型。

```cpp
// ❌ file_ops.h — 公共头文件泄漏了 Windows 头文件
#pragma once
#ifdef _WIN32
#include <windows.h>    // 问题！包含这个头文件的人都会被污染
#endif

class FileOps {
public:
    bool open(const char* path);
    void close();
private:
#ifdef _WIN32
    HANDLE hFile_;       // Windows 句柄
#else
    int fd_;             // POSIX 文件描述符
#endif
};
```

**为什么这是个问题？**

`<windows.h>` 定义了大量宏（`min`、`max`、`ERROR`、`DELETE`、`near`、`far` 等），这些宏会与标准库和其他代码冲突：

```cpp
// 某个毫不相关的数学模块
#include "file_ops.h"    // 间接引入了 <windows.h>
#include <algorithm>

int result = std::min(a, b);  // 编译错误！
// error: 'min' 不是 'std' 的成员
// 因为 windows.h 把 min 定义成了宏：#define min(a,b) ((a)<(b)?(a):(b))
```

**修复方案（三种）**：

```cpp
// ✅ 方案 A：使用 PIMPL 隐藏平台细节（推荐）
// file_ops.h
#pragma once
#include <memory>

class FileOps {
public:
    FileOps();
    ~FileOps();
    bool open(const char* path);
    void close();
private:
    struct Impl;                    // 前向声明，不暴露任何平台类型
    std::unique_ptr<Impl> impl_;
};

// ✅ 方案 B：如果必须在头文件中用 <windows.h>，至少限制宏污染
#ifdef _WIN32
#define WIN32_LEAN_AND_MEAN   // 排除大量不常用的 API
#define NOMINMAX              // 阻止 min/max 宏
#include <windows.h>
#endif

// ✅ 方案 C：使用文件分离，头文件只有接口声明
// file_ops.h — 完全无 #ifdef
#pragma once
class FileOps {
public:
    bool open(const char* path);
    void close();
};
// 每个平台一个 file_ops_xxx.cpp，各自 #include 自己需要的头文件
```

### 错误二：接口函数签名不一致

**场景**：团队的两个人分别实现 Windows 和 Linux 版本，结果函数签名不匹配。

```cpp
// ❌ Platform.h 声明的是 std::string 版本
bool TVPDeleteFile(const std::string& path);

// ❌ win32/Platform.cpp 实现了 std::wstring 版本（不匹配！）
bool TVPDeleteFile(const std::wstring& path) {   // 这是一个不同的函数
    return DeleteFileW(path.c_str());
}

// ❌ linux/Platform.cpp 实现了 const char* 版本（也不匹配！）
bool TVPDeleteFile(const char* path) {            // 又一个不同的函数
    return unlink(path) == 0;
}
```

**后果**：链接器报 `undefined reference to TVPDeleteFile(std::string const&)`，因为没有人实现了头文件声明的那个签名。

**修复**：

```cpp
// ✅ 头文件定义唯一的接口签名
// Platform.h
bool TVPDeleteFile(const std::string& path);  // UTF-8 路径

// ✅ 所有平台的 .cpp 必须实现完全一样的签名
// win32/Platform.cpp
bool TVPDeleteFile(const std::string& path) {
    // Windows 内部需要转为宽字符
    std::wstring wpath = Utf8ToWide(path);    // 内部转换
    return DeleteFileW(wpath.c_str()) != 0;
}

// linux/Platform.cpp
bool TVPDeleteFile(const std::string& path) {
    return unlink(path.c_str()) == 0;         // Linux 直接支持 UTF-8
}
```

**规则**：接口签名只出现在**一个头文件**中，用标准库类型（`std::string`、`int`、`bool`），平台特有类型的转换在实现文件内部完成。

### 错误三：添加新函数时遗漏某个平台

**场景**：你在 `Platform.h` 中添加了新函数声明，写了 Windows 和 Linux 的实现，但忘了 macOS 和 Android。

```
# macOS 编译时
Undefined symbols for architecture arm64:
  "TVPGetBatteryLevel()", referenced from:
      StatusBar::update() in StatusBar.cpp.o
ld: symbol(s) not found for architecture arm64
```

**修复方案**：

```cpp
// ✅ 方案 A：在 Platform.h 中提供一个默认实现（弱函数）
// Platform.h
#include <cstdio>

// 默认实现，未适配的平台至少不会链接失败
inline int TVPGetBatteryLevel() {
    fprintf(stderr, "WARNING: TVPGetBatteryLevel() not implemented "
                    "on this platform\n");
    return -1;  // -1 表示"不支持"
}

// ✅ 方案 B：用 static_assert 在编译期强制检查
// 在某个被所有平台编译的文件中
#if !defined(_WIN32) && !defined(__linux__) && \
    !defined(__APPLE__) && !defined(__ANDROID__)
static_assert(false, "New platform detected! "
    "Please implement TVPGetBatteryLevel() in your platform's Platform.cpp");
#endif

// ✅ 方案 C（推荐）：维护一个清单文件
// 每次添加新函数时，在 Platform.h 的注释中列出所有需要实现的平台：
//
// TVPGetBatteryLevel() — 实现清单：
//   [x] win32/Platform.cpp
//   [x] linux/Platform.cpp
//   [ ] apple/macos/platform.mm    ← TODO
//   [ ] android/AndroidUtils.cpp   ← TODO
```

### 错误四：类型兼容层中的大小不匹配

这个错误我们在上一节已经详细讨论过，这里做一个对照汇总：

```cpp
// ❌ 常见的类型映射错误
typedef unsigned long  DWORD;      // 64 位 Linux/macOS 上是 8 字节！
typedef unsigned int   WORD;       // 4 字节，但 WORD 应该是 2 字节
typedef unsigned int   HRESULT;    // 无符号，FAILED() 宏失效
typedef int            LARGE_INTEGER;  // 只有 4 字节，应该是 8 字节

// ✅ 正确的映射
typedef uint32_t  DWORD;           // 精确 4 字节
typedef uint16_t  WORD;            // 精确 2 字节
typedef int32_t   HRESULT;         // 有符号 4 字节
typedef int64_t   LARGE_INTEGER;   // 精确 8 字节（简化版）
```

| 类型 | 错误映射 | 错误原因 | 正确映射 |
|------|---------|---------|---------|
| `DWORD` | `unsigned long` | 64 位系统上可能 8 字节 | `uint32_t` |
| `WORD` | `unsigned int` | 4 字节而非 2 字节 | `uint16_t` |
| `HRESULT` | `uint32_t` | 无符号，`FAILED()` 失效 | `int32_t` |
| `BYTE` | `char` | `char` 的符号性未定义 | `uint8_t` |
| `BOOL` | `bool` | Windows `BOOL` 是 `int` (4 字节) | `int32_t` |

最后一条需要特别注意：Windows 的 `BOOL` 不是 C++ 的 `bool`！

```cpp
// ❌ 看似正确，实际有隐患
typedef bool BOOL;  // sizeof(bool) = 1，sizeof(BOOL) 在 Windows 上是 4

// 如果有 COM 接口使用 BOOL 参数，大小不匹配会导致栈损坏
HRESULT SomeComMethod(BOOL flag);  // 期望 4 字节参数
```

### 错误五：PIMPL 模式的生命周期问题

```cpp
// ❌ 析构函数在头文件中默认内联 — 编译错误
// widget.h
#pragma once
#include <memory>

class Widget {
public:
    Widget();
    // 没有声明析构函数！编译器生成的默认析构会内联
    // 但此时 Impl 是不完整类型，unique_ptr 无法析构它

    void draw();
private:
    struct Impl;
    std::unique_ptr<Impl> impl_;
};
```

**编译报错**：

```
error: invalid application of 'sizeof' to an incomplete type 'Widget::Impl'
    static_assert(sizeof(_Tp) > 0, "default_delete cannot delete incomplete type");
```

**修复**：

```cpp
// ✅ 析构函数必须在 .cpp 中定义（此时 Impl 已完整定义）
// widget.h
class Widget {
public:
    Widget();
    ~Widget();                        // 声明但不定义
    Widget(Widget&&) noexcept;        // 移动构造也需要
    Widget& operator=(Widget&&) noexcept;
    void draw();
private:
    struct Impl;
    std::unique_ptr<Impl> impl_;
};

// widget.cpp（或平台特定的 widget_win32.cpp）
#include "widget.h"

struct Widget::Impl {
    // ... 平台相关成员
};

Widget::Widget() : impl_(std::make_unique<Impl>()) {}
Widget::~Widget() = default;        // 在这里 Impl 是完整类型，ok
Widget::Widget(Widget&&) noexcept = default;
Widget& Widget::operator=(Widget&&) noexcept = default;
```

**记忆口诀**：用 PIMPL + `unique_ptr` 时，**析构函数、移动构造、移动赋值**这三个"大五"成员必须在 `.cpp` 中定义（即使写 `= default`）。

## 三种 PAL 模式选型指南

### 决策流程图

```
需要跨平台抽象？
    │
    ├── 平台在编译期确定？
    │       │
    │       ├── 是 → 头文件中有平台特定的成员变量？
    │       │       │
    │       │       ├── 否 → 【文件分离】✅ 最简单，零运行时开销
    │       │       │
    │       │       └── 是 → 需要 ABI 稳定性？
    │       │               │
    │       │               ├── 是 → 【PIMPL】✅ 隐藏内部细节
    │       │               │
    │       │               └── 否 → 【文件分离】也行，用条件编译成员
    │       │
    │       └── 否（运行时选择后端）→ 【运行时多态】✅ 接口 + 工厂
    │
    └── 已有大量旧代码使用 Windows 类型？
            │
            └── 是 → 【类型兼容层】✅ typedef 模拟，改动最小
```

### 对比表

| 维度 | 文件分离 | PIMPL | 运行时多态 | 类型兼容层 |
|------|---------|-------|-----------|-----------|
| **编译期 vs 运行时** | 编译期 | 编译期 | 运行时 | 编译期 |
| **运行时开销** | 零 | 一次指针间接访问 | 虚函数调用 | 零 |
| **头文件干净度** | 高 | 最高 | 高 | 中（有 typedef） |
| **添加新平台** | 新增 .cpp | 新增 .cpp | 新增子类 | 新增 #elif |
| **适用场景** | 大多数 PAL 函数 | 需要隐藏细节 | 运行时切换后端 | 移植旧代码 |
| **KrKr2 使用频率** | ⭐⭐⭐⭐⭐ 最多 | ⭐⭐ 少量 | ⭐⭐ 少量 | ⭐⭐⭐ typedefine.h 等 |
| **代码复杂度** | 低 | 中 | 高 | 低 |
| **典型代表** | `Platform.h` + `win32/Platform.cpp` | `Timer` 类 | `IWindowBackend` | `typedefine.h` |

### KrKr2 为什么混合使用多种模式？

KrKr2 以**文件分离**为主，但在特定场景用了其他模式：

- **文件分离**：`environ/win32/`、`environ/linux/`、`environ/android/`、`environ/apple/` 中的绝大多数文件
- **类型兼容层**：`typedefine.h`、`combase.h`、`vkdefine.h`——因为要移植大量使用 Windows 类型的旧代码
- **运行时多态**：`cocos2d/` 渲染后端——因为 Cocos2d-x 本身就是运行时选择 OpenGL/Metal 后端
- **PIMPL**：少量工具类——不是 KrKr2 的主要风格

这种混合使用是正常的。没有哪个真实项目只用一种 PAL 模式——**根据具体场景选择最合适的模式**才是正确的做法。

## 动手实践

### 综合练习：重构一个"意大利面"模块

以下是一个典型的充满 `#ifdef` 的跨平台通知模块。请使用本章学到的 PAL 模式重构它。

**原始代码（混乱版）：**

```cpp
// notification.h — 混乱的跨平台通知模块
#pragma once
#include <string>

#ifdef _WIN32
#include <windows.h>       // 头文件泄漏！
#include <shellapi.h>      // 更多泄漏
#elif defined(__APPLE__)
#include <objc/runtime.h>  // 泄漏
#endif

class NotificationManager {
public:
    void showNotification(const std::string& title,
                          const std::string& message) {
#ifdef _WIN32
        // Windows: 使用托盘图标通知
        NOTIFYICONDATAW nid = {};
        nid.cbSize = sizeof(nid);
        nid.uFlags = NIF_INFO;
        // ... 30 行 Windows 代码
        Shell_NotifyIconW(NIM_MODIFY, &nid);
#elif defined(__APPLE__)
        // macOS: 使用 NSUserNotification
        id notification = objc_msgSend(/* ... */);
        // ... 20 行 Objective-C 代码
#elif defined(__linux__)
        // Linux: 使用 libnotify
        // ... 15 行 Linux 代码
#elif defined(__ANDROID__)
        // Android: 使用 JNI 调用 Java NotificationManager
        // ... 25 行 JNI 代码
#endif
    }

    bool isSupported() {
#ifdef _WIN32
        return true;  // Windows Vista+
#elif defined(__APPLE__)
        return true;  // macOS 10.8+
#elif defined(__ANDROID__)
        return true;
#else
        return false;  // Linux: 取决于 libnotify
#endif
    }

    void dismiss() {
#ifdef _WIN32
        Shell_NotifyIconW(NIM_DELETE, &nid_);
#elif defined(__APPLE__)
        // ...
#endif
        // Linux 和 Android 没实现
    }

private:
#ifdef _WIN32
    NOTIFYICONDATAW nid_;   // Windows 特有成员
    HWND hwnd_;
#endif
};
```

**重构要求：**
1. 头文件中不能出现任何 `#ifdef` 和平台特有头文件
2. 每个平台的实现独立成文件
3. 所有平台的 `dismiss()` 都必须有实现（即使是空操作）

**期望结果的文件列表：**

```
notification/
├── notification.h              # 纯净接口
├── notification_win32.cpp      # Windows 实现
├── notification_linux.cpp      # Linux 实现
├── notification_macos.mm       # macOS 实现（Obj-C++）
├── notification_android.cpp    # Android 实现
└── CMakeLists.txt              # 按平台选择编译
```

<summary>查看完整重构答案</summary>

**notification.h — 纯净的公共接口**

```cpp
#pragma once
#include <string>

// 跨平台通知管理器
// 每个平台在独立的 .cpp 文件中实现
class NotificationManager {
public:
    // 显示系统通知
    // title: 通知标题（UTF-8）
    // message: 通知正文（UTF-8）
    void showNotification(const std::string& title,
                          const std::string& message);

    // 当前平台是否支持系统通知
    bool isSupported() const;

    // 关闭当前显示的通知
    // 如果平台不支持主动关闭，此函数为空操作
    void dismiss();
};
```

**notification_win32.cpp — Windows 实现**

```cpp
#include "notification.h"

#define WIN32_LEAN_AND_MEAN
#define NOMINMAX
#include <windows.h>
#include <shellapi.h>
#include <codecvt>
#include <locale>

// 辅助函数：UTF-8 → UTF-16
static std::wstring toWide(const std::string& utf8) {
    if (utf8.empty()) return {};
    int len = MultiByteToWideChar(CP_UTF8, 0,
        utf8.c_str(), (int)utf8.size(), nullptr, 0);
    std::wstring result(len, 0);
    MultiByteToWideChar(CP_UTF8, 0,
        utf8.c_str(), (int)utf8.size(), result.data(), len);
    return result;
}

void NotificationManager::showNotification(
        const std::string& title, const std::string& message) {
    NOTIFYICONDATAW nid = {};
    nid.cbSize = sizeof(nid);
    nid.uFlags = NIF_INFO;
    nid.dwInfoFlags = NIIF_INFO;

    std::wstring wTitle = toWide(title);
    std::wstring wMessage = toWide(message);

    wcsncpy(nid.szInfoTitle, wTitle.c_str(), 63);
    wcsncpy(nid.szInfo, wMessage.c_str(), 255);

    Shell_NotifyIconW(NIM_MODIFY, &nid);
}

bool NotificationManager::isSupported() const {
    return true;  // Windows Vista+
}

void NotificationManager::dismiss() {
    NOTIFYICONDATAW nid = {};
    nid.cbSize = sizeof(nid);
    Shell_NotifyIconW(NIM_DELETE, &nid);
}
```

**notification_linux.cpp — Linux 实现**

```cpp
#include "notification.h"
#include <cstdlib>
#include <cstdio>

void NotificationManager::showNotification(
        const std::string& title, const std::string& message) {
    // 使用 notify-send 命令行工具（大多数 Linux 发行版自带）
    char cmd[1024];
    snprintf(cmd, sizeof(cmd),
        "notify-send '%s' '%s' 2>/dev/null",
        title.c_str(), message.c_str());
    system(cmd);
}

bool NotificationManager::isSupported() const {
    // 检查 notify-send 是否存在
    return system("which notify-send > /dev/null 2>&1") == 0;
}

void NotificationManager::dismiss() {
    // Linux notify-send 不支持主动关闭通知
    // 空操作
}
```

**notification_macos.mm — macOS 实现（Objective-C++）**

```objc
#include "notification.h"
#import <Foundation/Foundation.h>
#import <AppKit/AppKit.h>

void NotificationManager::showNotification(
        const std::string& title, const std::string& message) {
    @autoreleasepool {
        NSUserNotification* notification =
            [[NSUserNotification alloc] init];
        notification.title =
            [NSString stringWithUTF8String:title.c_str()];
        notification.informativeText =
            [NSString stringWithUTF8String:message.c_str()];

        [[NSUserNotificationCenter defaultUserNotificationCenter]
            deliverNotification:notification];
    }
}

bool NotificationManager::isSupported() const {
    return true;  // macOS 10.8+
}

void NotificationManager::dismiss() {
    @autoreleasepool {
        [[NSUserNotificationCenter defaultUserNotificationCenter]
            removeAllDeliveredNotifications];
    }
}
```

**notification_android.cpp — Android 实现**

#include "notification.h"
#include <android/log.h>

void NotificationManager::showNotification(

    return true;
}

void NotificationManager::dismiss() {
    // 需要 JNI 调用 NotificationManager.cancel()
    // 简化版：空操作
}

**CMakeLists.txt — 按平台选择编译**

```cmake
add_library(notification STATIC notification.h)

if(WIN32)
    target_sources(notification PRIVATE notification_win32.cpp)
elseif(APPLE)
    target_sources(notification PRIVATE notification_macos.mm)
elseif(ANDROID)
    target_sources(notification PRIVATE notification_android.cpp)
else()  # Linux
    target_sources(notification PRIVATE notification_linux.cpp)
endif()

target_include_directories(notification PUBLIC ${CMAKE_CURRENT_SOURCE_DIR})
```

</details>

## 本节小结

- PAL 开发中最常见的五个错误：**头文件泄漏**（`<windows.h>` 宏污染）、**签名不一致**（各平台实现的函数参数类型不同）、**遗漏平台**（添加新函数忘记某个平台）、**类型不匹配**（`unsigned long` vs `uint32_t`）、**PIMPL 生命周期**（析构不完整类型）
- **文件分离**是最常用、最简单的 PAL 模式，适合大多数编译期确定平台的场景
- **PIMPL** 适合需要在头文件中隐藏平台细节的场景（特别是避免 `<windows.h>` 污染）
- **运行时多态**适合需要运行时选择实现的场景（如 Linux 上选择 X11 或 Wayland）
- **类型兼容层**适合移植大量使用平台特有类型的旧代码
- KrKr2 **混合使用**所有模式——以文件分离为主，辅以类型兼容层和少量运行时多态
- **选型原则**：能用文件分离就用文件分离，需要隐藏细节用 PIMPL，需要运行时切换用多态，移植旧代码用类型兼容层

### 题目 1：PAL 模式选择

你需要为一个跨平台游戏引擎设计窗口管理系统。已知需求：
- 支持 Windows（Win32 API）、Linux（X11/Wayland）、macOS（Cocoa）
- Linux 上可能需要运行时选择 X11 或 Wayland 后端
- 用户可以通过配置文件指定使用哪个后端

应该选择哪种 PAL 模式？为什么？

<details>
<summary>查看答案</summary>

应该选择**运行时多态（接口 + 工厂）**模式。

**原因分析：**
1. Linux 上需要在 X11 和 Wayland 之间**运行时选择**——文件分离和 PIMPL 都在编译期决定用哪个实现，无法运行时切换
2. 用户通过配置文件指定后端，意味着选择发生在程序启动时而非编译时
3. 即使 Windows 和 macOS 只有一种实现，统一的接口让上层代码不用关心

**设计：**

```cpp
// window_backend.h
#pragma once
#include <memory>
#include <string>

class IWindowBackend {
public:
    virtual ~IWindowBackend() = default;
    virtual bool createWindow(int w, int h, const char* title) = 0;
    virtual void pollEvents() = 0;
    virtual void swapBuffers() = 0;

    // 工厂方法
    static std::unique_ptr<IWindowBackend> create(
        const std::string& preferred);
};

// window_factory.cpp
std::unique_ptr<IWindowBackend> IWindowBackend::create(
        const std::string& preferred) {
#ifdef _WIN32
    return std::make_unique<Win32Window>();
#elif defined(__APPLE__)
    return std::make_unique<CocoaWindow>();
#elif defined(__linux__)
    if (preferred == "wayland") {
        return std::make_unique<WaylandWindow>();
    }
    return std::make_unique<X11Window>();  // 默认 X11
#endif
}
```

注意工厂方法内部仍然用了 `#ifdef`，但这是允许的——**一个文件中的 `#ifdef`** 比每处使用都 `#ifdef` 好得多。

</details>

### 题目 2：找出错误

以下 PIMPL 代码有两处会导致编译错误，找出并修正：

```cpp
// widget.h
#pragma once
#include <memory>

class Widget {
public:
    Widget() : impl_(std::make_unique<Impl>()) {}
    void draw();
private:
    struct Impl;
    std::unique_ptr<Impl> impl_;
};
```

<details>
<summary>查看答案</summary>

**错误 1：** 构造函数体内使用了 `std::make_unique<Impl>()`，但 `Impl` 此时是不完整类型（只有前向声明），`make_unique` 需要 `sizeof(Impl)` 来分配内存，编译失败。

**错误 2：** 没有声明析构函数。编译器生成的默认析构函数会内联在头文件中，此时 `unique_ptr<Impl>` 需要析构 `Impl`，但 `Impl` 是不完整类型，编译失败。

**修正：**

```cpp
// widget.h
#pragma once
#include <memory>

class Widget {
public:
    Widget();     // 只声明，不定义
    ~Widget();    // 必须声明
    void draw();
private:
    struct Impl;
    std::unique_ptr<Impl> impl_;
};

// widget.cpp
#include "widget.h"

struct Widget::Impl {
    int x = 0;
    int y = 0;
};

Widget::Widget() : impl_(std::make_unique<Impl>()) {}  // 这里 Impl 完整
Widget::~Widget() = default;  // 这里 Impl 完整
```

</details>

### 题目 3：类型兼容层设计

你在移植一个 DirectX 游戏到 OpenGL。原代码大量使用以下 DirectX 类型。为非 Windows 平台设计类型兼容层：

- `D3DCOLOR`（32 位 ARGB 颜色值，无符号）
- `D3DVECTOR`（三维向量，3 个 float）
- `D3DMATRIX`（4x4 矩阵，16 个 float）
- `LPDIRECT3DDEVICE9`（设备指针，不透明句柄）

<details>
<summary>查看答案</summary>

```cpp
// d3d_compat.h — DirectX 类型兼容层
#pragma once
#include <cstdint>

#if defined(_WIN32) && defined(USING_DIRECTX)
    #include <d3d9.h>    // Windows + DirectX: 用原生定义
#else
    // 非 Windows 或 OpenGL 模式: typedef 模拟

    // D3DCOLOR: 32 位 ARGB 颜色
    // 格式: 0xAARRGGBB
    typedef uint32_t D3DCOLOR;

    // 颜色构造宏（与 DirectX SDK 一致）
    #define D3DCOLOR_ARGB(a,r,g,b) \
        ((D3DCOLOR)((((a)&0xff)<<24)|(((r)&0xff)<<16)| \
                    (((g)&0xff)<<8)|((b)&0xff)))
    #define D3DCOLOR_XRGB(r,g,b) D3DCOLOR_ARGB(0xff,r,g,b)

    // D3DVECTOR: 三维向量
    typedef struct _D3DVECTOR {
        float x;
        float y;
        float z;
    } D3DVECTOR;

    // D3DMATRIX: 4x4 变换矩阵
    typedef struct _D3DMATRIX {
        union {
            struct {
                float _11, _12, _13, _14;
                float _21, _22, _23, _24;
                float _31, _32, _33, _34;
                float _41, _42, _43, _44;
            };
            float m[4][4];  // 数组访问方式
        };
    } D3DMATRIX;

    // LPDIRECT3DDEVICE9: 不透明设备指针
    // OpenGL 不需要设备对象，但旧代码中有参数传递
    typedef void* LPDIRECT3DDEVICE9;

#endif
```

**设计要点：**
- `D3DCOLOR` 用 `uint32_t`（精确 4 字节）
- `D3DVECTOR` 和 `D3DMATRIX` 保持与 DirectX SDK 相同的成员布局
- `D3DCOLOR_ARGB` 宏与 DirectX 一致，旧代码无需修改
- `LPDIRECT3DDEVICE9` 用 `void*`（不透明指针），只需编译通过

</details>
