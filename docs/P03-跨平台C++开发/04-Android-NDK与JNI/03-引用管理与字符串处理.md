## 为什么引用管理是 JNI 最难的部分？

### 场景：一个隐蔽的崩溃

假设你在 KrKr2 的 Android 端添加了一个功能——扫描用户设备上的所有游戏文件，返回文件路径列表。你写了这样的代码：

```cpp
// ❌ 有 Bug 的版本——你能看出问题吗？
JNIEXPORT jobjectArray JNICALL
Java_com_example_Scanner_scanFiles(JNIEnv *env, jclass cls,
                                    jstring rootPath) {
    const char *path = env->GetStringUTFChars(rootPath, nullptr);

    // 假设这个函数返回找到的文件路径
    std::vector<std::string> files = scanDirectory(path);

    // 创建 Java String 数组
    jclass strClass = env->FindClass("java/lang/String");
    jobjectArray result = env->NewObjectArray(
        files.size(), strClass, nullptr);

    for (int i = 0; i < files.size(); i++) {
        jstring jstr = env->NewStringUTF(files[i].c_str());
        env->SetObjectArrayElement(result, i, jstr);
        // 忘记了 DeleteLocalRef(jstr)!
    }

    env->ReleaseStringUTFChars(rootPath, path);
    return result;
}
```

这段代码在文件少的时候完全正常。但当用户设备上有 600+ 个文件时，应用突然崩溃，logcat 输出：

```
JNI ERROR (app bug): local reference table overflow (max=512)
```

**问题的根源**：JNI 的本地引用表有容量限制（默认 512 个）。循环中每次 `NewStringUTF` 创建一个本地引用，600 次循环就超出了限制。这就是为什么 **JNI 引用管理是最容易出 Bug 的部分**——它不像 Java 的 GC 那样自动管理，也不像 C++ 的智能指针那样有编译器帮你检查。

## JNI 的三种引用类型

JNI 有三种引用类型，分别对应不同的生命周期需求。可以用"图书借阅"来类比：

```
┌──────────────────────────────────────────────────────────┐
│                   JNI 引用类型比喻                        │
├──────────────────────────────────────────────────────────┤
│  本地引用 (Local Reference)                              │
│  → 类似"阅览室借阅"：只能在阅览室内看，离开就归还          │
│  → 生命周期：当前 native 方法调用期间                     │
│  → 创建方式：大部分 JNI 函数的返回值都是本地引用           │
│                                                          │
│  全局引用 (Global Reference)                             │
│  → 类似"长期借阅"：办了借书证，可以带回家                  │
│  → 生命周期：直到显式调用 DeleteGlobalRef                 │
│  → 创建方式：env->NewGlobalRef(localRef)                 │
│                                                          │
│  弱全局引用 (Weak Global Reference)                      │
│  → 类似"预约借阅"：可以长期持有预约，但书可能被别人借走     │
│  → 生命周期：持续到 DeleteWeakGlobalRef，但对象可被 GC 回收 │
│  → 创建方式：env->NewWeakGlobalRef(localRef)             │
└──────────────────────────────────────────────────────────┘
```

### 1. 本地引用（Local Reference）

本地引用是最常见的引用类型。**几乎所有返回 `jobject`（及其子类型如 `jstring`、`jclass`）的 JNI 函数都返回本地引用**。

```cpp
void Java_com_example_Demo_test(JNIEnv *env, jobject thiz) {
    // 以下每个调用都创建一个本地引用：
    jclass cls = env->FindClass("java/lang/String");    // 本地引用 1
    jstring str = env->NewStringUTF("hello");             // 本地引用 2
    jobject obj = env->AllocObject(cls);                  // 本地引用 3
    jclass cls2 = env->GetObjectClass(thiz);              // 本地引用 4

    // 函数返回时，所有 4 个本地引用自动释放
    // JVM/ART 会在 native 方法返回后清理本地引用表
}
```

**本地引用的关键规则：**

| 规则 | 说明 |
|------|------|
| 自动释放 | native 方法返回时自动释放所有本地引用 |
| 容量有限 | 默认最多 512 个（Android ART），超出则崩溃 |
| 不能跨线程 | 本地引用只在创建它的线程上有效 |
| 不能存储 | 不能保存到 static/全局变量中跨调用使用 |
| 可手动释放 | `env->DeleteLocalRef(ref)` 提前释放 |

**常见错误 1：循环中不释放本地引用**

```cpp
// ❌ 错误：循环创建大量本地引用不释放
void processItems(JNIEnv *env, jobjectArray items) {
    int count = env->GetArrayLength(items);
    for (int i = 0; i < count; ++i) {
        jobject item = env->GetObjectArrayElement(items, i);
        // 处理 item ...

        // 没有 DeleteLocalRef！
        // 如果 count > 512，崩溃！
    }
}

// ✅ 正确：循环中及时释放
void processItems(JNIEnv *env, jobjectArray items) {
    int count = env->GetArrayLength(items);
    for (int i = 0; i < count; ++i) {
        jobject item = env->GetObjectArrayElement(items, i);
        // 处理 item ...

        env->DeleteLocalRef(item);  // 用完立即释放
    }
}
```

**常见错误 2：把本地引用存储到全局变量**

```cpp
// ❌ 致命错误：把本地引用存为全局变量
static jclass g_MyClass = nullptr;

void Java_com_example_Demo_init(JNIEnv *env, jclass cls) {
    g_MyClass = env->FindClass("com/example/MyClass");
    // FindClass 返回的是本地引用
    // 当这个 native 方法返回后，g_MyClass 指向的引用被释放
    // 之后使用 g_MyClass 会导致"使用无效引用"崩溃！
}

void Java_com_example_Demo_doSomething(JNIEnv *env, jclass cls) {
    // ❌ 崩溃！g_MyClass 已经是无效引用
    jmethodID mid = env->GetMethodID(g_MyClass, "run", "()V");
}
```

### 2. 全局引用（Global Reference）

当你需要在多次 JNI 调用之间保持对 Java 对象的引用时，必须使用全局引用。

```cpp
// ✅ 正确：用全局引用缓存 jclass
static jclass g_MyClass = nullptr;      // 全局引用
static jmethodID g_RunMethod = nullptr; // jmethodID 不是引用，不需要特殊管理

// 在 JNI_OnLoad 中初始化
JNIEXPORT jint JNI_OnLoad(JavaVM *vm, void *reserved) {
    JNIEnv *env;
    vm->GetEnv(reinterpret_cast<void**>(&env), JNI_VERSION_1_6);

    // 第 1 步：获取本地引用
    jclass localCls = env->FindClass("com/example/MyClass");
    if (localCls == nullptr) {
        return JNI_ERR;  // 类不存在
    }

    // 第 2 步：转为全局引用
    g_MyClass = static_cast<jclass>(env->NewGlobalRef(localCls));

    // 第 3 步：释放本地引用（全局引用已经独立存在）
    env->DeleteLocalRef(localCls);

    // 缓存 methodID（methodID 是稳定值，不需要全局引用）
    g_RunMethod = env->GetMethodID(g_MyClass, "run", "()V");

    return JNI_VERSION_1_6;
}

// 在 JNI_OnUnload 中清理
void JNI_OnUnload(JavaVM *vm, void *reserved) {
    JNIEnv *env;
    vm->GetEnv(reinterpret_cast<void**>(&env), JNI_VERSION_1_6);

    if (g_MyClass) {
        env->DeleteGlobalRef(g_MyClass);
        g_MyClass = nullptr;
    }
}

// 之后可以在任何 native 方法中安全使用
void Java_com_example_Demo_doSomething(JNIEnv *env, jclass cls) {
    // ✅ 安全：g_MyClass 是全局引用，一直有效
    jobject obj = env->AllocObject(g_MyClass);
    env->CallVoidMethod(obj, g_RunMethod);
    env->DeleteLocalRef(obj);  // obj 是本地引用，用完释放
}
```

**全局引用 vs 本地引用对比表：**

| 特性 | 本地引用 | 全局引用 |
|------|----------|----------|
| 生命周期 | 当前 native 方法内 | 直到 `DeleteGlobalRef` |
| 容量限制 | 有限（默认 512） | 无硬性限制（但不能无限） |
| 跨线程 | ❌ 不可以 | ✅ 可以 |
| 跨 native 调用 | ❌ 不可以 | ✅ 可以 |
| 阻止 GC 回收 | ✅ 是 | ✅ 是 |
| 典型用途 | 临时对象操作 | 缓存 jclass、全局回调 |

### 3. 弱全局引用（Weak Global Reference）

弱全局引用是一种特殊的全局引用——它不会阻止 Java GC 回收对象。如果 Java 端没有其他强引用指向该对象，GC 可能随时回收它。

```cpp
static jweak g_WeakCallback = nullptr;

void Java_com_example_Demo_setCallback(JNIEnv *env, jclass cls,
                                        jobject callback) {
    // 清理旧的弱引用
    if (g_WeakCallback) {
        env->DeleteWeakGlobalRef(g_WeakCallback);
    }
    // 创建新的弱引用
    g_WeakCallback = env->NewWeakGlobalRef(callback);
}

void notifyCallback(JNIEnv *env) {
    if (g_WeakCallback == nullptr) {
        return;  // 从未设置过
    }

    // ⚠️ 使用前必须检查对象是否还活着
    jobject callback = env->NewLocalRef(g_WeakCallback);
    if (callback == nullptr) {
        // 对象已被 GC 回收
        env->DeleteWeakGlobalRef(g_WeakCallback);
        g_WeakCallback = nullptr;
        return;
    }

    // 对象还活着，可以安全使用
    jclass cls = env->GetObjectClass(callback);
    jmethodID mid = env->GetMethodID(cls, "onEvent", "()V");
    env->CallVoidMethod(callback, mid);

    // 释放本地引用
    env->DeleteLocalRef(callback);
    env->DeleteLocalRef(cls);
}
```

**三种引用类型的选择指南：**

```
需要保持引用吗？
├── 不需要（方法内临时使用） → 本地引用（默认）
├── 需要（跨调用/跨线程）
│   ├── 需要阻止 GC 回收  → 全局引用
│   └── 允许 GC 回收      → 弱全局引用
└── 只是缓存 methodID/fieldID → 直接存 jmethodID/jfieldID（不是引用！）
```

## KrKr2 中的引用管理实践

KrKr2 的 JNI 代码展示了在生产环境中如何正确管理引用。让我们分析几个关键案例：

### 案例 1：消息对话框——循环内的严格清理

```cpp
// cpp/core/environ/android/AndroidUtils.cpp 第 515-549 行
int TVPShowSimpleMessageBox(const char *pszText, const char *pszTitle,
                            unsigned int nButton, const char **btnText) {
    JniMethodInfo methodInfo;
    if (JniHelper::getStaticMethodInfo(
            methodInfo, "org/tvp/kirikiri2/KR2Activity", "ShowMessageBox",
            "(Ljava/lang/String;Ljava/lang/String;[Ljava/lang/String;)V")) {

        // 创建 Java 字符串（本地引用）
        jstring jstrTitle = methodInfo.env->NewStringUTF(pszTitle);  // 引用 1
        jstring jstrText = methodInfo.env->NewStringUTF(pszText);    // 引用 2

        // 创建 String 数组
        jclass strcls = methodInfo.env->FindClass("java/lang/String"); // 引用 3
        jobjectArray btns = methodInfo.env->NewObjectArray(
            nButton, strcls, nullptr);                                  // 引用 4

        // ⭐ 关键：循环内创建的引用立即释放
        for (unsigned int i = 0; i < nButton; ++i) {
            jstring jstrBtn = methodInfo.env->NewStringUTF(btnText[i]);
            methodInfo.env->SetObjectArrayElement(btns, i, jstrBtn);
            methodInfo.env->DeleteLocalRef(jstrBtn);  // ✅ 用完即释放
        }

        // 调用 Java 方法
        methodInfo.env->CallStaticVoidMethod(
            methodInfo.classID, methodInfo.methodID,
            jstrTitle, jstrText, btns);

        // ⭐ 逐个释放所有本地引用
        methodInfo.env->DeleteLocalRef(jstrTitle);
        methodInfo.env->DeleteLocalRef(jstrText);
        methodInfo.env->DeleteLocalRef(btns);
        methodInfo.env->DeleteLocalRef(methodInfo.classID);  // classID 也是本地引用！

        // ... 等待用户响应 ...
        return MsgBoxRet;
    }
    return -1;
}
```

**学习点**：
1. 循环内的 `jstrBtn` 用完立即 `DeleteLocalRef`——即使按钮最多 3-4 个不会溢出，但这是好习惯
2. `methodInfo.classID` 也需要释放——`getStaticMethodInfo` 内部调用了 `FindClass`，返回的也是本地引用
3. 方法最后逐个清理所有引用——不依赖函数返回时的自动清理

### 案例 2：获取存储路径——数组遍历中的引用管理

```cpp
// cpp/core/environ/android/AndroidUtils.cpp 第 363-372 行
static int InsertFilepathInto(JNIEnv *env, std::vector<std::string> &vec,
                              jobjectArray FileObjs) {
    int count = env->GetArrayLength(FileObjs);
    for (int i = 0; i < count; ++i) {
        jobject FileObj = env->GetObjectArrayElement(FileObjs, i);
        std::string path = File_getAbsolutePath(FileObj);
        if (!path.empty())
            vec.emplace_back(path);
        // ⚠️ 注意：这里没有 DeleteLocalRef(FileObj)！
        // 这是因为 count 通常很小（1-3 个存储路径），不会溢出
        // 但如果你维护这段代码，最好补上 DeleteLocalRef
    }
    return count;
}
```

**学习点**：
- 生产代码中有时会省略不必要的释放——当循环次数确定很小时
- 但这不是最佳实践。如果未来 `FileObjs` 数组变大，就会出问题
- 写新代码时，**始终释放循环中的本地引用**

### 案例 3：获取设备 ID——字符串转换与引用释放

```cpp
// cpp/core/environ/android/AndroidUtils.cpp 第 94-174 行
std::string TVPGetDeviceID() {
    std::string ret;
    {
        JniMethodInfo methodInfo;
        if (JniHelper::getStaticMethodInfo(methodInfo, KR2ActJavaPath,
                                           "getDeviceId",
                                           "()Ljava/lang/String;")) {
            // CallStaticObjectMethod 返回本地引用
            auto result = (jstring)methodInfo.env->CallStaticObjectMethod(
                methodInfo.classID, methodInfo.methodID);

            // jstring2string 内部会 GetStringUTFChars + 复制 + ReleaseStringUTFChars
            ret = JniHelper::jstring2string(result);

            // ✅ 释放 jstring 本地引用
            methodInfo.env->DeleteLocalRef(result);
            // ✅ 释放 classID 本地引用
            methodInfo.env->DeleteLocalRef(methodInfo.classID);

            // 后续处理：把 ':' 替换为 '='
            char *t = (char *)ret.c_str();
            while (*t) {
                if (*t == ':') { *t = '='; break; }
                t++;
            }
        }
    }
    return ret;
}
```

## JNI 字符串处理

### Java String 与 C++ string 的根本差异

Java 和 C++ 对字符串的处理方式完全不同，这是跨语言编程中最容易出错的地方之一：

| 特性 | Java String | C++ std::string |
|------|-------------|-----------------|
| 编码 | **UTF-16**（每个字符 2 或 4 字节） | **通常 UTF-8**（1-4 字节） |
| 可变性 | **不可变**（创建后不能修改） | **可变**（可以随时修改） |
| 内存管理 | GC 自动管理 | 对象析构时释放 |
| 空字符 | 可以包含 `\0` | `\0` 是字符串结束符 |
| "Hello" 的内存 | 10 字节（5 × 2字节 UTF-16） | 5 字节（ASCII = UTF-8） |
| "你好" 的内存 | 4 字节（2 × 2字节 UTF-16） | 6 字节（2 × 3字节 UTF-8） |

### 两套转换函数

JNI 提供了两套字符串转换函数，对应两种编码：

```
UTF-16 系列（Java 原生编码）：        Modified UTF-8 系列（更常用）：
├── GetStringChars()                 ├── GetStringUTFChars()
├── ReleaseStringChars()             ├── ReleaseStringUTFChars()
├── GetStringLength()                ├── GetStringUTFLength()
├── NewString()                      ├── NewStringUTF()
├── GetStringRegion()                ├── GetStringUTFRegion()
└── GetStringCritical()              └── (无对应)
```

**实际开发中 90% 的情况用 UTF-8 系列**，因为 C/C++ 标准库、Linux API、网络协议几乎都用 UTF-8。

### GetStringUTFChars / ReleaseStringUTFChars

这是最常用的字符串转换对。它把 Java String 转换为 Modified UTF-8 的 C 字符串。

```cpp
// 基本用法
JNIEXPORT void JNICALL
Java_com_example_Demo_printMessage(JNIEnv *env, jclass cls,
                                    jstring message) {
    // 第 1 步：获取 UTF-8 字符串
    // 第二个参数 isCopy 可以传 nullptr（不关心是否复制）
    // 或者传 jboolean* 来知道 JVM 是否做了复制
    jboolean isCopy;
    const char *utf8 = env->GetStringUTFChars(message, &isCopy);

    if (utf8 == nullptr) {
        // 内存分配失败（极罕见），不要继续
        return;
    }

    // 如果 isCopy == JNI_TRUE，说明 JVM 复制了一份（常见情况）
    // 如果 isCopy == JNI_FALSE，说明直接返回了内部指针（少见）
    printf("收到消息: %s (isCopy=%d)\n", utf8, isCopy);

    // 第 2 步：必须释放！
    // 如果 isCopy 为 true，释放复制的内存
    // 如果 isCopy 为 false，通知 JVM 不再需要固定该内存区域
    env->ReleaseStringUTFChars(message, utf8);

    // ❌ 此后不能再访问 utf8 指针！
    // printf("%s", utf8);  // 未定义行为！
}
```

### Modified UTF-8 vs 标准 UTF-8

JNI 返回的不是标准 UTF-8，而是 **Modified UTF-8**。它们有两个关键区别：

| 字符 | 标准 UTF-8 | Modified UTF-8 | 影响 |
|------|-----------|----------------|------|
| 空字符 `\0` | `0x00`（1 字节） | `0xC0 0x80`（2 字节） | Modified UTF-8 中字符串可以包含 `\0` |
| U+10000+ 补充字符 | 4 字节序列 | 两个 3 字节序列（代理对） | emoji 等字符的编码不同 |
| ASCII（U+0001-007F） | 相同 | 相同 | 无影响 |
| BMP（U+0080-FFFF） | 相同 | 相同 | 中日韩字符无影响 |

**对 KrKr2 的影响**：KrKr2 处理的是日文视觉小说游戏，文本主要是日文（BMP 范围内）和 ASCII。在这个范围内 Modified UTF-8 和标准 UTF-8 完全一致，所以 KrKr2 不需要额外的编码转换。

```cpp
// KrKr2 的字符串处理模式（krkr2_android.cpp 第 278-297 行）
JNIEXPORT void JNICALL
Java_org_tvp_kirikiri2_KR2Activity_nativeInsertText(
    JNIEnv *env, jclass cls, jstring text) {
    // GetStringUTFChars 返回 Modified UTF-8
    const char *pszText = env->GetStringUTFChars(text, nullptr);
    if (pszText && *pszText) {
        // 直接复制到 std::string——因为日文在 BMP 范围内，
        // Modified UTF-8 和标准 UTF-8 编码相同
        std::string str = pszText;

        // 通过事件队列推送到 Cocos2d 线程
        Android_PushEvents([str]() {
            cocos2d::IMEDispatcher::sharedDispatcher()
                ->dispatchInsertText(str.c_str(), str.length());
        });
    }
    // ⚠️ 注意：即使 pszText 为 nullptr，也要调用 Release
    // （但 KrKr2 这里在 if 外面调用了）
    env->ReleaseStringUTFChars(text, pszText);
}
```

### 常见错误：忘记 Release

```cpp
// ❌ 错误 1：忘记 ReleaseStringUTFChars
void processText(JNIEnv *env, jstring text) {
    const char *str = env->GetStringUTFChars(text, nullptr);
    std::string result(str);
    // 忘记 Release！内存泄漏，或阻止 GC 回收 Java String
    return;
}

// ❌ 错误 2：条件分支中遗漏 Release
void processText(JNIEnv *env, jstring text) {
    const char *str = env->GetStringUTFChars(text, nullptr);
    if (str == nullptr) {
        return;  // ✅ 这里不需要 Release（获取失败了）
    }
    if (strlen(str) > 1000) {
        return;  // ❌ 忘了 Release！
    }
    // ... 处理 ...
    env->ReleaseStringUTFChars(text, str);
}

// ✅ 正确：所有返回路径都有 Release
void processText(JNIEnv *env, jstring text) {
    const char *str = env->GetStringUTFChars(text, nullptr);
    if (str == nullptr) {
        return;
    }
    if (strlen(str) > 1000) {
        env->ReleaseStringUTFChars(text, str);  // ✅ 别忘了！
        return;
    }
    // ... 处理 ...
    env->ReleaseStringUTFChars(text, str);
}
```

## RAII 封装：自动管理 JNI 资源

手动管理 `GetStringUTFChars` / `ReleaseStringUTFChars` 配对很容易遗漏。C++ 的 RAII（Resource Acquisition Is Initialization，资源获取即初始化）模式可以完美解决这个问题。

### JniString 封装类

```cpp
// jni_utils.h
#pragma once
#include <jni.h>
#include <string>

// RAII 封装：自动管理 JNI 字符串生命周期
class JniString {
    JNIEnv *env_;
    jstring jstr_;
    const char *utf8_;

public:
    // 构造时获取 UTF-8 字符串
    JniString(JNIEnv *env, jstring jstr)
        : env_(env), jstr_(jstr),
          utf8_(jstr ? env->GetStringUTFChars(jstr, nullptr) : nullptr) {}

    // 析构时自动释放
    ~JniString() {
        if (utf8_ && jstr_) {
            env_->ReleaseStringUTFChars(jstr_, utf8_);
        }
    }

    // 禁止拷贝（防止 double release）
    JniString(const JniString &) = delete;
    JniString &operator=(const JniString &) = delete;

    // 允许移动
    JniString(JniString &&other) noexcept
        : env_(other.env_), jstr_(other.jstr_), utf8_(other.utf8_) {
        other.utf8_ = nullptr;
        other.jstr_ = nullptr;
    }

    // 访问接口
    const char *c_str() const { return utf8_ ? utf8_ : ""; }
    bool empty() const { return !utf8_ || !*utf8_; }
    bool isNull() const { return utf8_ == nullptr; }
    std::string str() const { return c_str(); }
    operator std::string() const { return str(); }
};
```

### 使用示例

```cpp
// 使用 RAII 封装后，再也不用担心忘记 Release
JNIEXPORT void JNICALL
Java_com_example_Demo_processInput(JNIEnv *env, jclass cls,
                                    jstring input) {
    JniString str(env, input);  // 获取 UTF-8 字符串

    if (str.empty()) {
        return;  // ✅ 析构函数自动调用 ReleaseStringUTFChars
    }

    if (strlen(str.c_str()) > 1000) {
        return;  // ✅ 无论从哪里返回，都会自动释放
    }

    printf("处理: %s\n", str.c_str());
    // ✅ 函数结束时，JniString 析构自动释放
}
```

### JniLocalRef 封装类

类似地，可以封装本地引用的自动管理：

```cpp
// RAII 封装：自动管理本地引用
template<typename T>
class JniLocalRef {
    JNIEnv *env_;
    T ref_;

public:
    JniLocalRef(JNIEnv *env, T ref) : env_(env), ref_(ref) {}

    ~JniLocalRef() {
        if (ref_) {
            env_->DeleteLocalRef(ref_);
        }
    }

    // 禁止拷贝
    JniLocalRef(const JniLocalRef &) = delete;
    JniLocalRef &operator=(const JniLocalRef &) = delete;

    // 允许移动
    JniLocalRef(JniLocalRef &&other) noexcept
        : env_(other.env_), ref_(other.ref_) {
        other.ref_ = nullptr;
    }

    T get() const { return ref_; }
    operator T() const { return ref_; }

    // 释放所有权（用于返回值）
    T release() {
        T tmp = ref_;
        ref_ = nullptr;
        return tmp;
    }
};

// 使用示例
void safeJniOperation(JNIEnv *env) {
    // 自动管理 jclass 本地引用
    JniLocalRef<jclass> cls(env, env->FindClass("java/lang/String"));
    if (!cls.get()) {
        return;  // 自动清理
    }

    // 自动管理 jstring 本地引用
    JniLocalRef<jstring> str(env, env->NewStringUTF("hello"));

    // 使用引用...
    jmethodID mid = env->GetMethodID(cls, "length", "()I");
    jint len = env->CallIntMethod(str, mid);

    printf("长度: %d\n", len);
    // 函数结束时，str 和 cls 自动 DeleteLocalRef
}
```

## 数组元素的获取与释放

除了字符串，JNI 数组元素的获取/释放也需要配对管理。

### Get/Release 模式

```cpp
JNIEXPORT jdouble JNICALL
Java_com_example_Stats_standardDeviation(JNIEnv *env, jclass cls,
                                          jdoubleArray data) {
    jsize length = env->GetArrayLength(data);
    if (length == 0) return 0.0;

    // 获取数组元素指针（可能是复制，也可能是直接指针）
    jdouble *elements = env->GetDoubleArrayElements(data, nullptr);
    if (elements == nullptr) return 0.0;

    // 计算平均值
    jdouble sum = 0.0;
    for (jsize i = 0; i < length; i++) {
        sum += elements[i];
    }
    jdouble mean = sum / length;

    // 计算方差
    jdouble variance = 0.0;
    for (jsize i = 0; i < length; i++) {
        jdouble diff = elements[i] - mean;
        variance += diff * diff;
    }
    variance /= length;

    // 释放数组元素
    // 第三个参数 mode 有三种选择：
    //   0         — 复制修改回原数组 + 释放缓冲区
    //   JNI_COMMIT — 复制修改回原数组 + 不释放缓冲区（之后还要用）
    //   JNI_ABORT  — 不复制修改 + 释放缓冲区（只读场景，最高效）
    env->ReleaseDoubleArrayElements(data, elements, JNI_ABORT);  // 只读，用 ABORT

    return sqrt(variance);
}
```

### KrKr2 中的数组处理

```cpp
// platforms/android/cpp/krkr2_android.cpp 第 125-170 行
// 触摸移动事件——处理 int[] 和 float[] 数组
JNIEXPORT void JNICALL
Java_org_tvp_kirikiri2_KR2Activity_nativeTouchesMove(
    JNIEnv *env, jclass thiz,
    jintArray ids, jfloatArray xs, jfloatArray ys) {
    int size = env->GetArrayLength(ids);

    // 单点触摸的优化路径：避免分配 vector
    if (size == 1) {
        jint id;
        jfloat x, y;
        // GetXxxArrayRegion 直接复制到栈变量，无需 Release
        env->GetIntArrayRegion(ids, 0, 1, &id);
        env->GetFloatArrayRegion(xs, 0, 1, &x);
        env->GetFloatArrayRegion(ys, 0, 1, &y);
        // ... 处理 ...
        return;
    }

    // 多点触摸：批量复制
    jint id[size];  // VLA（非标准 C++，但 NDK 支持）
    std::vector<jfloat> x(size), y(size);

    // GetXxxArrayRegion 批量复制，无需 Release
    env->GetIntArrayRegion(ids, 0, size, id);
    env->GetFloatArrayRegion(xs, 0, size, &x[0]);
    env->GetFloatArrayRegion(ys, 0, size, &y[0]);

    // ... 处理 ...
}
```

**学习点**：`GetXxxArrayRegion` 和 `GetXxxArrayElements` 的区别：

| 函数 | 行为 | 需要 Release | 适用场景 |
|------|------|--------------|----------|
| `GetIntArrayElements` | 返回指向数组的指针（可能复制） | ✅ 需要 | 需要操作整个数组 |
| `GetIntArrayRegion` | 复制到你提供的缓冲区 | ❌ 不需要 | 只需部分元素或小数组 |

### 练习 1：修复引用泄漏

以下代码有引用泄漏问题，请找出并修复：

```cpp
JNIEXPORT jobjectArray JNICALL
Java_com_example_Demo_getNames(JNIEnv *env, jclass cls) {
    std::vector<std::string> names = {"Alice", "Bob", "Charlie"};

    jclass strClass = env->FindClass("java/lang/String");
    jobjectArray result = env->NewObjectArray(
        names.size(), strClass, nullptr);

    for (int i = 0; i < names.size(); i++) {
        jstring jstr = env->NewStringUTF(names[i].c_str());
        env->SetObjectArrayElement(result, i, jstr);
    }

    return result;
}
```

<details>
<summary>查看答案</summary>

问题：
1. 循环中 `jstr` 没有释放——虽然 3 个元素不会溢出，但应该养成好习惯
2. `strClass` 没有释放——可能被后续代码依赖，但函数返回前应清理

修复后：

```cpp
JNIEXPORT jobjectArray JNICALL
Java_com_example_Demo_getNames(JNIEnv *env, jclass cls) {
    std::vector<std::string> names = {"Alice", "Bob", "Charlie"};

    jclass strClass = env->FindClass("java/lang/String");
    jobjectArray result = env->NewObjectArray(
        names.size(), strClass, nullptr);

    for (int i = 0; i < names.size(); i++) {
        jstring jstr = env->NewStringUTF(names[i].c_str());
        env->SetObjectArrayElement(result, i, jstr);
        env->DeleteLocalRef(jstr);  // ✅ 修复：释放循环中的引用
    }

    env->DeleteLocalRef(strClass);  // ✅ 修复：释放 strClass
    // result 不需要释放——它是返回值，JNI 会保持它有效
    return result;
}
```

注意：返回值 `result` 不需要 `DeleteLocalRef`，因为它作为返回值传回 Java，JNI 会自动保持这个引用有效。

</details>

### 练习 2：实现 RAII 字符串数组转换

写一个工具函数，把 Java `String[]` 转换为 `std::vector<std::string>`，要求：
1. 正确处理 null 数组和 null 元素
2. 释放所有中间引用
3. 使用前面介绍的 `JniString` RAII 类

<details>
<summary>查看答案</summary>

```cpp
std::vector<std::string> jstringArrayToVector(
    JNIEnv *env, jobjectArray jarray) {
    std::vector<std::string> result;

    // 处理 null 输入
    if (jarray == nullptr) {
        return result;
    }

    jsize length = env->GetArrayLength(jarray);
    result.reserve(length);

    for (jsize i = 0; i < length; i++) {
        // GetObjectArrayElement 返回本地引用
        jstring jstr = (jstring)env->GetObjectArrayElement(jarray, i);

        if (jstr != nullptr) {
            // 使用 RAII 封装自动管理 GetStringUTFChars
            JniString str(env, jstr);
            result.emplace_back(str.str());
            // JniString 析构时自动 ReleaseStringUTFChars
        } else {
            result.emplace_back("");  // null 元素用空字符串
        }

        // 释放 GetObjectArrayElement 返回的本地引用
        if (jstr) {
            env->DeleteLocalRef(jstr);
        }
    }

    return result;
}
```

</details>

## 本节小结

- JNI 有**三种引用类型**：本地引用（方法内有效）、全局引用（显式释放前有效）、弱全局引用（对象可被 GC 回收）
- **本地引用表容量有限**（默认 512），循环中必须及时 `DeleteLocalRef`
- **不能把本地引用存到全局变量**——方法返回后引用失效，之后使用会崩溃
- `jmethodID` 和 `jfieldID` **不是引用**，不需要管理（它们是稳定的标识符）
- JNI 字符串用 `GetStringUTFChars` / `ReleaseStringUTFChars` 配对——**必须配对！**
- JNI 返回的是 **Modified UTF-8**，对 ASCII 和 BMP 字符（含中日韩）与标准 UTF-8 相同
- 用 **RAII 封装**（如 `JniString` 类）可以彻底消除忘记释放的风险
- `GetXxxArrayRegion` 复制到用户缓冲区，不需要 Release；`GetXxxArrayElements` 需要 Release

### 题目 1：引用类型选择

你正在写一个 JNI 库，需要缓存 Java 类的 `jclass` 引用以便在多个 native 方法中使用。以下三种方案，哪个是正确的？为什么？

```cpp
// 方案 A
static jclass g_cls;
void init(JNIEnv *env) {
    g_cls = env->FindClass("com/example/Foo");
}

// 方案 B
static jclass g_cls;
void init(JNIEnv *env) {
    jclass local = env->FindClass("com/example/Foo");
    g_cls = (jclass)env->NewGlobalRef(local);
    env->DeleteLocalRef(local);
}

// 方案 C
static jclass g_cls;
void init(JNIEnv *env) {
    g_cls = (jclass)env->NewWeakGlobalRef(
        env->FindClass("com/example/Foo"));
}
```

<details>
<summary>查看答案</summary>

**方案 B 是正确的。**

- **方案 A 错误**：`FindClass` 返回本地引用，native 方法返回后失效。之后使用 `g_cls` 会导致崩溃（使用无效引用）。
- **方案 B 正确**：将本地引用转为全局引用，可以安全地跨多次 native 调用使用。最后记得在 `JNI_OnUnload` 中调用 `DeleteGlobalRef(g_cls)`。
- **方案 C 有风险**：弱全局引用不阻止 GC 回收。虽然 `jclass` 对象通常不会被回收（类加载器持有强引用），但规范不保证这一点。而且代码还有另一个 Bug：`FindClass` 返回的本地引用没有释放。

</details>

### 题目 2：分析 KrKr2 的引用管理

阅读 `AndroidUtils.cpp` 第 211-221 行的 `GetPackageName` 函数，分析它的引用管理是否有问题：

```cpp
static std::string GetPackageName() {
    JniMethodInfo methodInfo;
    jobject sInstance = GetKR2ActInstance();
    if (!JniHelper::getMethodInfo(methodInfo, "android/content/ContextWrapper",
                                 "getPackageName", "()Ljava/lang/String;")) {
        methodInfo.env->DeleteLocalRef(sInstance);
        return "";
    }
    return JniHelper::jstring2string((jstring)methodInfo.env->CallObjectMethod(
        sInstance, methodInfo.methodID));
}
```

<details>
<summary>查看答案</summary>

这段代码有**两个潜在的引用泄漏**：

1. **成功路径缺少释放 `sInstance`**：当 `getMethodInfo` 成功时，函数直接返回，没有释放 `sInstance`（`GetKR2ActInstance()` 返回的本地引用）
2. **`CallObjectMethod` 返回的 `jstring` 没有释放**：虽然 `jstring2string` 内部会提取 UTF-8 字符串，但 `CallObjectMethod` 返回的 `jstring` 本地引用没有显式释放

不过实际运行中，这两个引用会在 native 方法返回时自动释放。因为 `GetPackageName` 是 `static` 函数、被其他 native 方法内部调用，而且只调用一次（获取包名），所以不会造成引用表溢出。

严格修复后的版本：

```cpp
static std::string GetPackageName() {
    JniMethodInfo methodInfo;
    jobject sInstance = GetKR2ActInstance();
    if (!JniHelper::getMethodInfo(methodInfo, "android/content/ContextWrapper",
                                 "getPackageName", "()Ljava/lang/String;")) {
        methodInfo.env->DeleteLocalRef(sInstance);
        return "";
    }
    jstring result = (jstring)methodInfo.env->CallObjectMethod(
        sInstance, methodInfo.methodID);
    std::string ret = JniHelper::jstring2string(result);
    methodInfo.env->DeleteLocalRef(result);     // ✅ 释放 jstring
    methodInfo.env->DeleteLocalRef(sInstance);   // ✅ 释放 sInstance
    return ret;
}
```

</details>

### 题目 3：为什么 KrKr2 的触摸事件处理用 GetXxxArrayRegion 而不是 GetXxxArrayElements？

查看 `krkr2_android.cpp` 中 `nativeTouchesMove` 函数，解释为什么用 `GetIntArrayRegion` / `GetFloatArrayRegion` 而不是 `GetIntArrayElements` / `GetFloatArrayElements`。

<details>
<summary>查看答案</summary>

三个原因：

1. **不需要 Release 调用**：`GetXxxArrayRegion` 直接复制到用户提供的缓冲区，不需要配对 `ReleaseXxxArrayElements`。在复杂的事件处理代码中（有多个返回路径），减少 Release 配对可以降低泄漏风险。

2. **触摸数据是只读的**：C++ 端只需要读取触摸坐标，不需要修改 Java 数组。`GetXxxArrayRegion` 的语义就是"只读复制"，比 `GetXxxArrayElements` 更明确。

3. **数据量小**：触摸点通常只有 1-10 个（多点触摸最多 10 指），复制开销可忽略不计。`GetXxxArrayElements` 的优势（可能直接返回内部指针、避免复制）在小数组上没有意义。

总结：**对于小量、只读的数组数据，`GetXxxArrayRegion` 是更安全、更简洁的选择。**

</details>
