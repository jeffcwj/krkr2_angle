## 为什么需要这个实战？

**场景**：你刚学完 P03 前四章的跨平台知识——文件系统差异、条件编译、抽象层模式、JNI 桥梁。你觉得自己已经"会了"。然后老板让你给项目加一个跨平台的文件监控功能（File Watcher），你打开编辑器，发现一个字也写不出来：

- 目录结构怎么组织？公共头文件放哪？平台实现放哪？
- `#ifdef` 写在头文件里还是 CMake 里？
- Windows 用 `ReadDirectoryChangesW`，Linux 用 `inotify`，macOS 用 `FSEvents`——这三个签名完全不同，怎么统一？
- Android 根本没有原生的文件监控 API，怎么办？

**"知道"和"能做"之间有巨大鸿沟**。本章就是用来填这个鸿沟的。

我们将从零构建一个名叫 **Mini-PAL** 的小型平台抽象层库，包含三个最常用的跨平台模块：

| 模块 | 统一接口 | Win32 后端 | POSIX 后端 | Android 特殊处理 |
|------|---------|-----------|-----------|-----------------|
| **文件系统** | `FileExists`, `GetFileInfo`, `DeleteFile` | `GetFileAttributesW` | `stat()` | `AAssetManager` 只读资产 |
| **线程** | `Thread`, `Mutex` | `CreateThread` / `CRITICAL_SECTION` | `pthread_create` / `pthread_mutex` | 复用 POSIX（NDK 支持 pthread） |
| **日志** | `Log(level, fmt, ...)` | `OutputDebugStringA` | `fprintf(stderr, ...)` | `__android_log_print` |

这三个模块虽然简单，但覆盖了跨平台开发中 **90%** 的常见场景。掌握它们的设计思路后，无论是文件监控、网络、音频还是图形，都是同一套方法论。

### 对照 KrKr2 源码

Mini-PAL 的三个模块在 KrKr2 中都有对应物：

| Mini-PAL 模块 | KrKr2 对应文件 | 说明 |
|-------------|--------------|------|
| `platform.h`（平台检测） | `cpp/core/environ/typedefine.h` | 平台宏 + 类型别名 |
| `filesystem.h`（文件系统） | `cpp/core/environ/Platform.h` 第 40-70 行 | `TVP_stat`, `TVPDeleteFile`, `TVPRenameFile` |
| `thread.h`（线程） | `cpp/core/utils/ThreadIntf.h` | 线程、互斥锁、条件变量 |
| `log.h`（日志） | spdlog 集成 | `TVPPrintLog` + spdlog 后端 |

## 项目结构设计

### 为什么目录结构很重要？

**场景**：想象两个开发者同时维护一个跨平台项目。开发者 A 需要修改 Linux 下的文件操作 bug，开发者 B 需要给 Windows 添加新的线程池功能。如果所有平台代码都混在一个文件里，两人的修改必然冲突。但如果按平台拆分到不同目录，他们可以各自修改 `src/posix/filesystem.cpp` 和 `src/win32/thread.cpp`，互不干扰。

好的目录结构要满足三个原则：

1. **按平台隔离**：每个平台的代码在独立目录中，互不依赖
2. **接口与实现分离**：公共头文件在 `include/` 中，实现在 `src/` 中
3. **一目了然**：看一眼目录树就知道每个文件的职责

### Mini-PAL 目录结构

```
mini-pal/
├── CMakeLists.txt              # 顶层构建文件
├── include/                    # 公共头文件（所有平台共享）
│   └── pal/
│       ├── platform.h          # 平台检测宏 + 基础类型
│       ├── filesystem.h        # 文件系统统一接口
│       ├── thread.h            # 线程与同步原语接口
│       └── log.h               # 日志统一接口
├── src/                        # 平台特定实现
│   ├── win32/                  # Windows 后端
│   │   ├── filesystem.cpp      # GetFileAttributesW, DeleteFileW
│   │   ├── thread.cpp          # CreateThread, CRITICAL_SECTION
│   │   └── log.cpp             # OutputDebugStringA
│   ├── posix/                  # Linux + macOS 后端
│   │   ├── filesystem.cpp      # stat(), unlink()
│   │   ├── thread.cpp          # pthread_create, pthread_mutex
│   │   └── log.cpp             # fprintf(stderr, ...)
│   └── android/                # Android 特殊后端
│       ├── filesystem.cpp      # AAssetManager（只读资产）
│       └── log.cpp             # __android_log_print
├── tests/                      # 跨平台统一测试
│   ├── CMakeLists.txt
│   └── test_pal.cpp            # 一套测试代码，四个平台运行
└── README.md
```

### 设计决策解析

**为什么 Android 没有 `thread.cpp`？**

因为 Android NDK 完整支持 POSIX 线程（`pthread`）。`pthread_create`、`pthread_mutex_init` 等函数在 Android 上和 Linux 上行为一致。所以 Android 的线程实现直接复用 `posix/thread.cpp`，不需要单独实现。

但文件系统和日志就不同了：
- **文件系统**：Android 应用的资产文件打包在 APK 中，必须通过 `AAssetManager` 访问，普通的 `fopen` 无法读取
- **日志**：Android 的标准输出（`stdout`/`stderr`）不会显示在 Logcat 中，必须用 `__android_log_print`

**为什么头文件放在 `include/pal/` 而不是直接放在 `include/`？**

这是 C++ 库的通用约定。加一层 `pal/` 命名空间目录可以避免头文件名冲突。使用时：

```cpp
#include <pal/filesystem.h>   // ✅ 明确知道这是 PAL 库的头文件
#include <filesystem.h>       // ❌ 可能与 C++17 的 <filesystem> 冲突
```

### 对照 KrKr2 的目录结构

```
KrKr2 目录:                         Mini-PAL 对应:
cpp/core/environ/                   include/pal/
├── Platform.h                      ├── filesystem.h + log.h
├── typedefine.h                    └── platform.h
├── win32/
│   └── Platform.cpp                src/win32/filesystem.cpp + log.cpp
├── linux/
│   └── Platform.cpp                src/posix/filesystem.cpp + log.cpp
├── apple/macos/
│   └── platform.mm                 src/posix/（macOS 也是 POSIX）
└── android/
    └── AndroidUtils.cpp            src/android/filesystem.cpp + log.cpp
```

KrKr2 的设计理念和我们完全一致：**一个头文件声明接口，多个平台子目录提供实现**。区别在于 KrKr2 把所有平台函数声明在一个 `Platform.h` 中，我们按功能拆分为 `filesystem.h`、`thread.h`、`log.h`——拆分方式使每个文件更聚焦，适合教学。

## 第一步：平台检测头文件

平台检测是整个 PAL 的基础——其他所有模块都依赖它来决定编译哪些代码。

### 检测顺序的陷阱

**场景**：一个新手这样写平台检测：

```cpp
// ❌ 错误的检测顺序
#if defined(__linux__)
    #define PAL_PLATFORM_LINUX 1
#elif defined(__ANDROID__)
    #define PAL_PLATFORM_ANDROID 1    // 永远不会被执行！
#endif
```

**为什么 Android 分支永远不会执行？** 因为 Android NDK 的编译器**同时**定义了 `__linux__` 和 `__ANDROID__`。如果先检测 `__linux__`，Android 设备就会被错误地识别为 Linux。

正确的顺序是：**先检测更具体的平台，再检测更通用的**。

```
检测优先级：Android > macOS/iOS > Linux > Windows
原因：Android 包含 __linux__，macOS 包含 __APPLE__
     先检测特殊的，再检测通用的
```

### 完整的平台检测头文件

```cpp
// include/pal/platform.h — 平台检测与基础定义
#pragma once

// ============================================================
// 平台检测
// ============================================================
// 检测顺序至关重要：
// 1. Android 必须在 Linux 之前（因为 __ANDROID__ 和 __linux__ 同时存在）
// 2. macOS 必须在通用 Unix 之前（__APPLE__ 是 Apple 特有的）
// 3. Windows 放在最后（_WIN32 不会与其他平台冲突）

#if defined(__ANDROID__)
    // Android：NDK 编译环境
    // __ANDROID__ 由 Android NDK 的编译器定义
    // 注意：__linux__ 在 Android 上也是定义的！
    #define PAL_PLATFORM_ANDROID  1
    #define PAL_PLATFORM_NAME     "Android"

#elif defined(__APPLE__)
    // Apple 平台：需要进一步区分 macOS / iOS / tvOS
    #include <TargetConditionals.h>
    #if TARGET_OS_MAC && !TARGET_OS_IPHONE
        #define PAL_PLATFORM_MACOS 1
        #define PAL_PLATFORM_NAME  "macOS"
    #elif TARGET_OS_IPHONE
        #define PAL_PLATFORM_IOS 1
        #define PAL_PLATFORM_NAME "iOS"
        // 本项目暂不支持 iOS，但预留检测
    #else
        #error "未知的 Apple 平台"
    #endif

#elif defined(__linux__)
    // Linux：在 Android 检测之后，保证这里只匹配桌面 Linux
    #define PAL_PLATFORM_LINUX 1
    #define PAL_PLATFORM_NAME  "Linux"

#elif defined(_WIN32) || defined(_WIN64)
    // Windows：_WIN32 在 32 位和 64 位都定义
    // _WIN64 仅在 64 位定义
    #define PAL_PLATFORM_WINDOWS 1
    #define PAL_PLATFORM_NAME    "Windows"

#else
    #error "不支持的平台：请为目标平台添加检测宏"
#endif

// ============================================================
// 平台分组宏
// ============================================================

// POSIX 系平台（Linux + macOS + Android）共享大量 POSIX API
// 用途：pthread、stat()、unistd.h 等
#if defined(PAL_PLATFORM_LINUX) || defined(PAL_PLATFORM_MACOS) || \
    defined(PAL_PLATFORM_ANDROID)
    #define PAL_PLATFORM_POSIX 1
#endif

// 桌面平台（Windows + Linux + macOS）
// 用途：完整的文件系统访问、控制台 I/O
#if defined(PAL_PLATFORM_WINDOWS) || defined(PAL_PLATFORM_LINUX) || \
    defined(PAL_PLATFORM_MACOS)
    #define PAL_PLATFORM_DESKTOP 1
#endif

// ============================================================
// 符号导出宏
// ============================================================
// 作用：控制动态库中哪些函数可以被外部调用
//
// Windows 使用 __declspec(dllexport/dllimport)
// Linux/macOS 使用 __attribute__((visibility("default")))
//
// 为什么需要这个？
// 因为 Windows 的 DLL 默认不导出任何符号，必须显式标记
// 而 ELF（Linux/macOS）默认导出所有符号，但最佳实践是
// 编译时加 -fvisibility=hidden，只导出需要的

#if defined(PAL_PLATFORM_WINDOWS)
    #ifdef PAL_BUILD_DLL
        // 正在编译 PAL 库本身 → 导出符号
        #define PAL_API __declspec(dllexport)
    #elif defined(PAL_USE_DLL)
        // 使用 PAL 库的客户端代码 → 导入符号
        #define PAL_API __declspec(dllimport)
    #else
        // 静态链接 → 不需要导出标记
        #define PAL_API
    #endif
#else
    // Linux/macOS/Android：使用 GCC/Clang 的 visibility 属性
    #define PAL_API __attribute__((visibility("default")))
#endif

// ============================================================
// 架构检测（可选但实用）
// ============================================================
#if defined(__x86_64__) || defined(_M_X64)
    #define PAL_ARCH_X64 1
    #define PAL_ARCH_NAME "x86_64"
#elif defined(__aarch64__) || defined(_M_ARM64)
    #define PAL_ARCH_ARM64 1
    #define PAL_ARCH_NAME "ARM64"
#elif defined(__i386__) || defined(_M_IX86)
    #define PAL_ARCH_X86 1
    #define PAL_ARCH_NAME "x86"
#elif defined(__arm__)
    #define PAL_ARCH_ARM32 1
    #define PAL_ARCH_NAME "ARM32"
#else
    #define PAL_ARCH_NAME "Unknown"
#endif

// ============================================================
// 基础头文件
// ============================================================
#include <cstdint>
#include <string>
```

### 逐段解析

#### 平台检测宏的来源

每个编译器/平台组合会预定义特定的宏，我们利用这些宏来识别平台：

| 宏 | 定义者 | 在哪些平台上存在 |
|----|--------|---------------|
| `__ANDROID__` | Android NDK (Clang) | 仅 Android |
| `__APPLE__` | Apple Clang | macOS, iOS, tvOS, watchOS |
| `__linux__` | GCC/Clang | Linux, **Android**（注意！） |
| `_WIN32` | MSVC, MinGW | Windows 32位 和 64位 |
| `_WIN64` | MSVC, MinGW | 仅 Windows 64位 |
| `__x86_64__` | GCC/Clang | x86_64 架构 |
| `_M_X64` | MSVC | x86_64 架构 |
| `__aarch64__` | GCC/Clang | ARM64 架构 |
| `_M_ARM64` | MSVC | ARM64 架构 |

你可以用编译器打印所有预定义宏来验证：

```bash
# GCC/Clang — 打印所有预定义宏
echo | gcc -dM -E -

# MSVC — 打印预定义宏
cl /EP /Zc:preprocessor /PD empty.cpp 2>nul

# Android NDK 交叉编译时的宏
echo | $ANDROID_NDK/toolchains/llvm/prebuilt/linux-x86_64/bin/aarch64-linux-android21-clang -dM -E -
# 输出中会同时包含 __ANDROID__ 和 __linux__
```

#### 分组宏的用途

分组宏（`PAL_PLATFORM_POSIX`、`PAL_PLATFORM_DESKTOP`）让你可以在代码中表达"这段代码适用于所有 POSIX 系统"：

```cpp
// ✅ 用分组宏：简洁，不容易遗漏平台
#ifdef PAL_PLATFORM_POSIX
    #include <unistd.h>   // Linux + macOS + Android 都可用
    #include <sys/stat.h>
#endif

// ❌ 不用分组宏：冗长，容易忘记添加新平台
#if defined(PAL_PLATFORM_LINUX) || defined(PAL_PLATFORM_MACOS) || \
    defined(PAL_PLATFORM_ANDROID)
    #include <unistd.h>
    #include <sys/stat.h>
#endif
```

#### 导出宏详解

**为什么 Windows 和 Linux 的导出方式不同？**

这与两种可执行文件格式的设计哲学有关：

| 格式 | 平台 | 默认行为 | 导出方式 |
|------|------|---------|---------|
| **PE (Portable Executable)** | Windows | 不导出任何符号 | `__declspec(dllexport)` |
| **ELF (Executable and Linkable Format)** | Linux/macOS/Android | 导出所有符号 | `-fvisibility=hidden` + `__attribute__((visibility("default")))` |

Windows 的 PE 格式设计理念是"显式导出"，你必须主动标记要导出的函数。这样做的好处是 DLL 的导出表很小，加载速度快。

ELF 格式默认导出所有符号，但现代最佳实践是编译时加 `-fvisibility=hidden`，然后只对需要的函数标记 `visibility("default")`。效果和 Windows 一样。

在 CMake 中设置默认隐藏：

```cmake
# 设置默认符号可见性为隐藏
set(CMAKE_CXX_VISIBILITY_PRESET hidden)
set(CMAKE_VISIBILITY_INLINES_HIDDEN ON)
```

### 常见错误

**错误 1**：忘记 Android 的 `__linux__` 问题

```cpp
// ❌ 错误：Android 设备被识别为 Linux
#if defined(__linux__)
    #define PAL_PLATFORM_LINUX 1    // Android 也会进入这里！
#elif defined(__ANDROID__)
    #define PAL_PLATFORM_ANDROID 1  // 永远到不了
#endif

// ✅ 正确：Android 在 Linux 之前
#if defined(__ANDROID__)
    #define PAL_PLATFORM_ANDROID 1
#elif defined(__linux__)
    #define PAL_PLATFORM_LINUX 1
#endif
```

**错误 2**：macOS 上忘记包含 `<TargetConditionals.h>`

```cpp
// ❌ 错误：TARGET_OS_MAC 未定义，条件永远为 false
#if defined(__APPLE__) && TARGET_OS_MAC
    #define PAL_PLATFORM_MACOS 1
#endif

// ✅ 正确：先包含头文件
#if defined(__APPLE__)
    #include <TargetConditionals.h>
    #if TARGET_OS_MAC
        #define PAL_PLATFORM_MACOS 1
    #endif
#endif
```

**错误 3**：导出宏在编译和使用时搞混

```cpp
// 编译库时：cmake 定义 PAL_BUILD_DLL
// 使用库时：cmake 定义 PAL_USE_DLL（或什么都不定义用静态链接）

// ❌ 错误：编译库时用了 dllimport
// 原因：忘记在 CMakeLists.txt 中添加 target_compile_definitions
add_library(pal SHARED ${PAL_SOURCES})
// 没有定义 PAL_BUILD_DLL → PAL_API 变成 dllimport → 链接失败

// ✅ 正确：
add_library(pal SHARED ${PAL_SOURCES})
target_compile_definitions(pal PRIVATE PAL_BUILD_DLL)
# PRIVATE：只有编译 pal 库自身时才定义 PAL_BUILD_DLL
```

## 对照 KrKr2：typedefine.h 的设计

KrKr2 的 `typedefine.h`（`cpp/core/environ/typedefine.h`，115 行）采用了一种不同但同样有效的策略：

```cpp
// cpp/core/environ/typedefine.h 第 1-12 行
#pragma once
#include <stdint.h>

// 平台检测宏
#if defined(_WIN32) || defined(_WIN64)
#define TARGET_WINDOWS 1
#include <windows.h>        // Windows：直接包含全部 Win32 API
#else
#define TARGET_WINDOWS 0    // 非 Windows：用 0 而不是不定义
#include "tjsTypes.h"       // 自定义类型定义
#include <sys/types.h>
#endif
```

### KrKr2 vs Mini-PAL 设计对比

| 设计决策 | KrKr2 (`typedefine.h`) | Mini-PAL (`platform.h`) |
|---------|----------------------|----------------------|
| 平台宏值 | `TARGET_WINDOWS 1` / `0` | `PAL_PLATFORM_WINDOWS 1` / 不定义 |
| 检测方式 | 编译器宏 + CMake 变量 | 仅编译器宏 |
| Android 检测 | CMake 设置 `ANDROID` 变量 | `__ANDROID__` 编译器宏 |
| 未支持平台 | 静默回退到 POSIX | `#error` 编译失败 |
| 类型适配 | 在非 Windows 上定义 `DWORD`、`HRESULT` 等 Win32 类型 | 不做类型适配 |

KrKr2 的 `TARGET_WINDOWS` 用 `1`/`0` 而不是定义/未定义，这意味着你可以这样写：

```cpp
// KrKr2 风格：if 条件判断
#if TARGET_WINDOWS    // 值为 1 或 0，都是合法的
    #include <windows.h>
#endif

// Mini-PAL 风格：ifdef 存在性判断
#ifdef PAL_PLATFORM_WINDOWS    // 定义了就进入，没定义就跳过
    #include <windows.h>
#endif
```

两种风格各有优劣：

| 风格 | 优点 | 缺点 |
|------|------|------|
| `#if VALUE` | 可以用 `&&`、`\|\|` 组合条件 | 忘记定义时默认为 `0`（静默错误） |
| `#ifdef NAME` | 忘记定义时编译器报警 | 不能直接做算术组合 |

> **建议**：对于新项目，推荐 `#ifdef` 风格，因为它能更早发现"忘记包含平台头文件"的错误。

### KrKr2 的类型兼容层

KrKr2 的 `typedefine.h` 做了一件特别的事：在非 Windows 平台上定义 `DWORD`、`HRESULT`、`HBITMAP` 等 Win32 类型：

```cpp
// cpp/core/environ/typedefine.h 第 25-52 行（非 Windows 时生效）
#if !TARGET_WINDOWS

typedef int32_t  LONG;
typedef uint32_t ULONG;
typedef uint32_t DWORD;     // Windows 的 32 位无符号整数
typedef uint16_t WORD;
typedef uint8_t  BYTE;

typedef void *HBITMAP;      // 模拟 Windows 位图句柄
typedef void *HDC;          // 模拟 Windows 设备上下文
typedef void *HFONT;        // 模拟 Windows 字体句柄
typedef LONG HRESULT;       // 模拟 COM 返回值

typedef struct _FILETIME {
    DWORD dwLowDateTime;
    DWORD dwHighDateTime;
} FILETIME;

#endif // !TARGET_WINDOWS
```

**为什么要这样做？** 因为 KrKr2 移植自 Windows 原生的 KiriKiri 引擎，原始代码大量使用 Win32 类型（`DWORD`、`HRESULT` 等）。为了尽量少改动原始代码，KrKr2 在非 Windows 平台上用 `typedef` 模拟这些类型，而不是把所有 `DWORD` 都改成 `uint32_t`。

这是一种**务实的移植策略**——虽然不够"纯净"，但减少了 90% 的代码修改量。在我们的 Mini-PAL 中不需要这种兼容层，因为我们从零开始，直接使用标准 C++ 类型。

## 本节小结

- 好的项目目录结构要按平台隔离、接口与实现分离、一目了然
- 平台检测的顺序至关重要：Android → Apple → Linux → Windows（从特殊到通用）
- 分组宏（`PAL_PLATFORM_POSIX`、`PAL_PLATFORM_DESKTOP`）简化条件编译代码
- 符号导出宏在 Windows 上用 `__declspec`，在 ELF 平台上用 `visibility` 属性
- KrKr2 的 `typedefine.h` 采用 `TARGET_WINDOWS 1/0` 风格，并在非 Windows 上提供 Win32 类型兼容层
- 新项目推荐 `#ifdef` 风格检测，能更早发现遗漏错误

## 练习题与答案

### 题目 1：为什么 Android 必须在 Linux 之前检测？如果顺序颠倒会导致什么具体问题？

<details>
<summary>查看答案</summary>

因为 Android NDK 的编译器同时定义了 `__ANDROID__` 和 `__linux__` 两个宏。如果先检测 `__linux__`，Android 设备会被错误识别为桌面 Linux 系统。

具体问题示例：
```cpp
// 如果 Android 被误判为 Linux：
#ifdef PAL_PLATFORM_LINUX
    // 这段代码在 Android 上也会编译
    #include <X11/Xlib.h>  // ❌ Android 没有 X11，编译失败
    fprintf(stderr, "日志输出\n");  // ❌ stderr 在 Android 上不显示到 Logcat
#endif
```

而正确识别为 Android 后：
```cpp
#ifdef PAL_PLATFORM_ANDROID
    #include <android/log.h>
    __android_log_print(ANDROID_LOG_INFO, "PAL", "日志输出");  // ✅ 显示在 Logcat
#endif
```

</details>

### 题目 2：KrKr2 的 `typedefine.h` 为什么要在非 Windows 平台上定义 `DWORD`、`HRESULT` 等类型？这种做法有什么优点和风险？

<details>
<summary>查看答案</summary>

**原因**：KrKr2 移植自 Windows 原生的 KiriKiri 引擎。原始代码大量使用 Win32 类型，如 `DWORD`（无符号 32 位整数）、`HRESULT`（COM 返回值）、`HBITMAP`（位图句柄）等。通过在非 Windows 平台上用 `typedef` 定义这些类型，可以让原始代码几乎不做修改就能在 Linux/macOS/Android 上编译。

**优点**：
1. 大幅减少代码修改量（不需要把所有 `DWORD` 改成 `uint32_t`）
2. 保持与原始 Windows 代码的一致性，方便对照
3. 移植速度快——只需写一个兼容头文件

**风险**：
1. 类型语义丢失：`HBITMAP` 在 Windows 上是真正的位图句柄，在 Linux 上只是 `void*`，没有类型安全
2. 增加理解难度：新开发者看到 `DWORD` 会以为代码只能在 Windows 上运行
3. 不完全兼容：有些 Win32 类型的行为（如 `HRESULT` 的错误码含义）无法通过简单的 `typedef` 模拟
4. 命名空间污染：在全局作用域定义 `DWORD` 等常用名称，可能与其他库冲突

</details>

### 题目 3：编写一个 `PAL_STRINGIFY` 宏，能将平台检测宏的值转换为字符串。例如 `PAL_STRINGIFY(PAL_PLATFORM_WINDOWS)` 在 Windows 上输出 `"1"`，在其他平台上输出 `"undefined"`。

<details>
<summary>查看答案</summary>

```cpp
#include <cstdio>

// 二级宏展开技巧
#define PAL_STRINGIFY_IMPL(x) #x
#define PAL_STRINGIFY(x) PAL_STRINGIFY_IMPL(x)

// 检测宏是否定义的技巧
// 如果 PAL_PLATFORM_WINDOWS 已定义（值为 1），
// PAL_STRINGIFY(PAL_PLATFORM_WINDOWS) 展开为 "1"
// 如果未定义，需要另一种方式

// 更实用的版本：打印宏是否定义
#define PAL_CHECK_MACRO(name) \
    printf("  %-30s = %s\n", #name, \
           PAL_STRINGIFY(name)[0] == #name[0] ? "undefined" : PAL_STRINGIFY(name))

// 上面的技巧原理：
// 如果 X 未定义，PAL_STRINGIFY(X) 展开为 "X"（宏名本身）
// "X"[0] == #X 的 'X' 字符 → 相等 → 未定义
// 如果 X 定义为 1，PAL_STRINGIFY(X) 展开为 "1"
// "1"[0] == 'X' → 不等 → 已定义

int main() {
    printf("=== 宏定义检测 ===\n");
    PAL_CHECK_MACRO(PAL_PLATFORM_WINDOWS);
    PAL_CHECK_MACRO(PAL_PLATFORM_LINUX);
    PAL_CHECK_MACRO(PAL_PLATFORM_MACOS);
    PAL_CHECK_MACRO(PAL_PLATFORM_ANDROID);
    PAL_CHECK_MACRO(PAL_PLATFORM_POSIX);
    PAL_CHECK_MACRO(PAL_PLATFORM_DESKTOP);
    return 0;
}
```

注意：`PAL_CHECK_MACRO` 的实现依赖一个巧妙的特性——如果宏 `X` 未被定义，`PAL_STRINGIFY(X)` 会展开为字符串 `"X"`（宏名本身），此时字符串的首字符与宏名的首字符相同。如果宏被定义为数值（如 `1`），展开后的字符串首字符就不会与宏名匹配。

</details>
