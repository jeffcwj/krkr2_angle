## 为什么文件系统是第一个要实现的模块？

**场景**：你的游戏引擎需要在启动时检查存档文件是否存在。在 Windows 上你写了：

```cpp
#include <windows.h>
bool save_exists = (GetFileAttributesA("save.dat") != INVALID_FILE_ATTRIBUTES);
```

完美运行。然后你把代码移到 Linux 上编译——直接报错，因为 Linux 没有 `<windows.h>`。改成 `stat()` 后 Linux 能跑了，但 Android 上又出问题：APK 内打包的资产文件用 `stat()` 找不到，必须用 `AAssetManager_open`。

这就是为什么文件系统是跨平台差异最大的模块之一——**同一个"文件是否存在"的问题，四个平台有四种完全不同的答案**。

| 平台 | "检查文件存在"的 API | 特殊限制 |
|------|-------------------|---------|
| **Windows** | `GetFileAttributesA()` | 返回 `INVALID_FILE_ATTRIBUTES` 表示不存在 |
| **Linux** | `access(path, F_OK)` 或 `stat()` | 需要 `<unistd.h>` |
| **macOS** | 同 Linux（POSIX 兼容） | 文件系统默认大小写不敏感 |
| **Android** | `access()` + `AAssetManager_open()` | APK 内资产不在文件系统上 |

KrKr2 引擎面临完全相同的问题。它的 `Platform.h`（第 40-67 行）声明了 `TVP_stat`、`TVPDeleteFile`、`TVPRenameFile` 等统一接口，然后在 `win32/Platform.cpp`、`linux/Platform.cpp`、`android/AndroidUtils.cpp` 中分别实现。我们的 Mini-PAL 用同样的思路，只是接口更简化。

## 统一接口设计

### FileInfo 结构体

在设计接口之前，先想想调用者需要什么信息。一个"获取文件信息"的函数至少要返回：

- **文件大小**：用于分配缓冲区、显示进度条
- **最后修改时间**：用于缓存失效检查、增量构建
- **是否是目录**：用于递归遍历、路径验证

我们把这些信息打包到一个结构体中：

```cpp
// include/pal/filesystem.h — 文件系统统一接口
#pragma once
#include "platform.h"
#include <string>
#include <cstdint>

namespace pal {

// 文件信息结构体
// 对照 KrKr2：tTVP_stat（Platform.h 第 61-67 行）
// KrKr2 的 tTVP_stat 包含 st_mode、st_size、st_atime、st_mtime、st_ctime
// 我们简化为三个最常用的字段
struct FileInfo {
    uint64_t size;       // 文件大小（字节）
    uint64_t mtime;      // 最后修改时间（Unix 时间戳，秒）
    bool     is_dir;     // 是否为目录
};
```

**为什么用 `uint64_t` 而不是 `size_t`？**

`size_t` 在 32 位系统上是 32 位的，最大只能表示 4GB。现代游戏的单个文件（比如视频、贴图包）很容易超过 4GB。`uint64_t` 固定为 64 位，能表示到 16EB（1600 万 TB），永远不会溢出。

**为什么修改时间用 Unix 时间戳？**

Unix 时间戳（自 1970-01-01 00:00:00 UTC 起的秒数）是最通用的时间表示。Linux/macOS 的 `stat()` 直接返回 Unix 时间戳，Windows 需要从 `FILETIME` 转换，但转换公式是固定的。使用 Unix 时间戳作为统一格式，调用者不需要关心底层差异。

### 接口函数声明

```cpp
// ========== 文件系统操作 ==========

// 检查文件是否存在
// 对照 KrKr2：TVP_stat() 间接实现存在性检查
PAL_API bool FileExists(const std::string& path);

// 获取文件信息（大小、修改时间、是否目录）
// 对照 KrKr2：TVP_stat(const char* name, tTVP_stat& s)
PAL_API bool GetFileInfo(const std::string& path, FileInfo& info);

// 删除文件（不能删除目录）
// 对照 KrKr2：TVPDeleteFile(const std::string& filename)
PAL_API bool DeleteFile(const std::string& path);

// 重命名/移动文件
// 对照 KrKr2：TVPRenameFile(const std::string& from, const std::string& to)
PAL_API bool RenameFile(const std::string& from, const std::string& to);

// 获取应用数据目录（每个平台的惯例不同）
// 对照 KrKr2：TVPGetAppStoragePath()
PAL_API std::string GetAppDataDir();

} // namespace pal
```

**接口设计原则**：

1. **返回 `bool` 表示成功/失败**：简单直观，调用者一个 `if` 就能处理错误
2. **输出参数用引用**：`GetFileInfo` 的 `info` 参数是输出型引用，避免动态内存分配
3. **路径用 `const std::string&`**：兼顾 C 字符串（隐式转换）和 C++ 字符串
4. **不抛异常**：文件不存在是正常情况，不应该用异常来处理

> **注意**：完整的文件系统 PAL 还应该包含 `CreateDirectory`、`ReadFile`、`WriteFile`、`ListDirectory` 等操作。我们这里只实现最核心的五个函数，已经足够演示跨平台设计的全部要点。

## Windows 实现

Windows 文件操作使用 Win32 API，它的特点是：函数名都带 `A`（ANSI）或 `W`（Wide/Unicode）后缀，返回值是 `BOOL`（0 表示失败），出错时通过 `GetLastError()` 获取错误码。

```cpp
// src/win32/filesystem.cpp — Windows 文件系统实现
#include "pal/filesystem.h"

#ifndef WIN32_LEAN_AND_MEAN
#define WIN32_LEAN_AND_MEAN  // 排除不常用的 Windows 头文件，加速编译
#endif
#include <windows.h>
#include <shlobj.h>          // SHGetFolderPathA

namespace pal {

bool FileExists(const std::string& path) {
    // GetFileAttributesA：获取文件属性，如果文件不存在返回 INVALID_FILE_ATTRIBUTES
    // 这比 CreateFile + CloseHandle 的方式快得多（不需要打开文件）
    DWORD attrs = GetFileAttributesA(path.c_str());
    return (attrs != INVALID_FILE_ATTRIBUTES);
}
```

**为什么用 `GetFileAttributesA` 而不是 `_access` 或 `PathFileExistsA`？**

| 方法 | 优点 | 缺点 |
|------|------|------|
| `GetFileAttributesA` | 最快（一次系统调用）、同时能判断目录 | 返回值需要比较 `INVALID_FILE_ATTRIBUTES` |
| `_access(path, 0)` | POSIX 兼容 | 不是原生 Win32 API，底层也调用 `GetFileAttributes` |
| `PathFileExistsA` | 名字直观 | 需要链接 `shlwapi.lib`，多一个依赖 |

```cpp
bool GetFileInfo(const std::string& path, FileInfo& info) {
    WIN32_FILE_ATTRIBUTE_DATA data;
    // GetFileAttributesExA：一次调用获取所有文件属性
    // 比先 GetFileAttributesA + 再 GetFileSize 效率更高
    if (!GetFileAttributesExA(path.c_str(), GetFileExInfoStandard, &data)) {
        return false;  // 文件不存在或无权限
    }

    // Windows 的文件大小分高低 32 位存储
    // 必须合并为 64 位整数才能正确表示 >4GB 的文件
    info.size = (static_cast<uint64_t>(data.nFileSizeHigh) << 32)
                | data.nFileSizeLow;

    // ====== FILETIME → Unix 时间戳转换 ======
    // FILETIME 是从 1601-01-01 00:00:00 UTC 起的 100 纳秒间隔数
    // Unix 时间戳是从 1970-01-01 00:00:00 UTC 起的秒数
    //
    // 两个纪元之间的差值：
    // 从 1601-01-01 到 1970-01-01 共 369 年
    // = 134774 天 × 24 × 60 × 60 × 10,000,000（100ns 间隔/秒）
    // = 116,444,736,000,000,000
    //
    // 转换公式：unix_seconds = (filetime - 116444736000000000) / 10000000
    ULARGE_INTEGER ull;
    ull.LowPart  = data.ftLastWriteTime.dwLowDateTime;
    ull.HighPart = data.ftLastWriteTime.dwHighDateTime;
    info.mtime = (ull.QuadPart - 116444736000000000ULL) / 10000000ULL;

    // FILE_ATTRIBUTE_DIRECTORY 标志位判断是否为目录
    info.is_dir = (data.dwFileAttributes & FILE_ATTRIBUTE_DIRECTORY) != 0;
    return true;
}
```

**FILETIME 转换公式图解**：

```
Windows FILETIME 时间轴:
|-- 1601-01-01 --|-------------- 369 年 --------------|-- 1970-01-01 --|-- 现在 --|
                  <-- 116,444,736,000,000,000 个 100ns 间隔 -->

Unix 时间戳时间轴:
                                                       |-- 1970-01-01 --|-- 现在 --|
                                                       <------- 秒数 -------->

转换: unix_time = (filetime_value - 116444736000000000) / 10000000
```

```cpp
bool DeleteFile(const std::string& path) {
    // Windows API 的 DeleteFileA 不能删除目录（删目录用 RemoveDirectoryA）
    // 我们的接口也约定只删除文件，保持行为一致
    // 注意：前面加 :: 是为了调用全局命名空间的 Win32 API
    // 否则会递归调用我们自己的 pal::DeleteFile
    return ::DeleteFileA(path.c_str()) != 0;
}

bool RenameFile(const std::string& from, const std::string& to) {
    // MoveFileA 的行为：如果目标文件已存在，操作会失败
    // 如果需要覆盖，用 MoveFileExA + MOVEFILE_REPLACE_EXISTING 标志
    return ::MoveFileA(from.c_str(), to.c_str()) != 0;
}

std::string GetAppDataDir() {
    char path[MAX_PATH];
    // CSIDL_APPDATA：用户漫游数据目录
    // 典型路径：C:\Users\<用户名>\AppData\Roaming
    //
    // 其他常用 CSIDL 值：
    // CSIDL_LOCAL_APPDATA — 本地数据（不跟随漫游配置文件）
    // CSIDL_COMMON_APPDATA — 所有用户共享数据
    // CSIDL_PERSONAL — "我的文档"目录
    if (SUCCEEDED(SHGetFolderPathA(nullptr, CSIDL_APPDATA, nullptr, 0, path))) {
        return std::string(path) + "\\MiniPAL";
    }
    return ".";  // 回退到当前目录
}

} // namespace pal
```

> **对照 KrKr2 源码**：`win32/Platform.cpp` 使用完全相同的 Win32 API 模式。`TVPDeleteFile` 调用 `::DeleteFileA`，`TVPRenameFile` 调用 `::MoveFileA`。KrKr2 的 `TVPGetSelfUsedMemory()` 则展示了另一个 Win32 API —— `GetProcessMemoryInfo`（第 26-30 行）。

## POSIX 实现（Linux + macOS 共用）

Linux 和 macOS 都遵循 POSIX 标准，绝大多数文件操作 API 完全相同。只有一个例外：**应用数据目录的惯例不同**（Linux 遵循 XDG 规范，macOS 用 `~/Library/Application Support/`）。

```cpp
// src/posix/filesystem.cpp — POSIX 文件系统实现（Linux 和 macOS 共用）
#include "pal/filesystem.h"
#include <sys/stat.h>       // stat() — 获取文件元数据
#include <unistd.h>         // unlink() — 删除文件, access() — 检查权限
#include <cstdio>           // rename() — 重命名
#include <cstdlib>          // getenv() — 读取环境变量
#include <pwd.h>            // getpwuid() — 通过 UID 获取用户信息

namespace pal {

bool FileExists(const std::string& path) {
    // access() 检查文件是否存在
    // F_OK = 仅检查存在性（不检查读写权限）
    // 返回 0 表示存在，-1 表示不存在或出错
    return access(path.c_str(), F_OK) == 0;
}
```

**`access()` vs `stat()` 检查文件存在**：

| 函数 | 优点 | 缺点 |
|------|------|------|
| `access(path, F_OK)` | 语义明确、代码简洁 | 存在 TOCTOU 竞争条件 |
| `stat(path, &st)` | 能同时获取文件信息 | 需要声明 `struct stat` 变量 |

我们在 `FileExists` 中用 `access`（只需判断存在），在 `GetFileInfo` 中用 `stat`（需要详细信息）。

> **TOCTOU（Time-of-Check to Time-of-Use）**：在你检查文件存在之后、实际使用之前，另一个进程可能已经删除了该文件。这是所有"先检查后操作"模式的固有问题，不论用 `access` 还是 `stat` 都无法避免。正确的做法是直接尝试操作，然后处理失败情况。

```cpp
bool GetFileInfo(const std::string& path, FileInfo& info) {
    struct stat st;
    // stat() 获取文件元数据
    // 失败返回 -1，errno 被设置为错误码
    // 常见 errno：ENOENT（文件不存在）、EACCES（无权限）
    if (stat(path.c_str(), &st) != 0) {
        return false;
    }

    info.size  = static_cast<uint64_t>(st.st_size);
    info.mtime = static_cast<uint64_t>(st.st_mtime);  // 已经是 Unix 时间戳
    info.is_dir = S_ISDIR(st.st_mode);                 // S_ISDIR 宏判断目录
    return true;
}
```

**`struct stat` 的关键字段**：

| 字段 | 类型 | 含义 |
|------|------|------|
| `st_mode` | `mode_t` | 文件类型和权限（用 `S_ISDIR`、`S_ISREG` 等宏判断类型） |
| `st_size` | `off_t` | 文件大小（字节） |
| `st_atime` | `time_t` | 最后访问时间 |
| `st_mtime` | `time_t` | 最后修改时间（内容变化） |
| `st_ctime` | `time_t` | 最后状态改变时间（权限、所有者等元数据变化） |

注意 POSIX 的 `st_mtime` 已经是 Unix 时间戳（秒），不需要像 Windows 那样转换。

```cpp
bool DeleteFile(const std::string& path) {
    // unlink() 删除文件（实际上是删除目录项，文件的 link count 减 1）
    // 如果 link count 降为 0 且没有进程打开该文件，文件才真正被删除
    // 不能删除目录——删目录用 rmdir()
    return unlink(path.c_str()) == 0;
}

bool RenameFile(const std::string& from, const std::string& to) {
    // POSIX rename() 的行为与 Windows MoveFileA 不同！
    // ⚠ POSIX：如果目标已存在，会被静默覆盖
    // ⚠ Windows：如果目标已存在，操作会失败
    // 这是一个常被忽略的跨平台差异
    return std::rename(from.c_str(), to.c_str()) == 0;
}
```

⚠ **关键跨平台差异**：`rename` 目标文件已存在时的行为

```
POSIX (Linux/macOS):
  rename("a.txt", "b.txt")
  如果 b.txt 已存在 → b.txt 被静默覆盖 ✅ 操作成功

Windows (MoveFileA):
  MoveFileA("a.txt", "b.txt")
  如果 b.txt 已存在 → ❌ 操作失败，返回 0
```

如果你需要跨平台一致的"覆盖式重命名"，需要在 Windows 上使用 `MoveFileExA` 并指定 `MOVEFILE_REPLACE_EXISTING` 标志。

```cpp
std::string GetAppDataDir() {
#if defined(PAL_PLATFORM_MACOS)
    // macOS 惯例：~/Library/Application Support/<AppName>
    // 这是 Apple 官方推荐的应用数据目录
    const char* home = getenv("HOME");
    if (!home) {
        // HOME 未设置时通过 getpwuid 获取（罕见但可能）
        struct passwd* pw = getpwuid(getuid());
        home = pw ? pw->pw_dir : "/tmp";
    }
    return std::string(home) + "/Library/Application Support/MiniPAL";

#else  // Linux
    // Linux 惯例：遵循 XDG Base Directory 规范
    // 优先使用 $XDG_DATA_HOME，未设置则默认 ~/.local/share
    //
    // XDG 规范定义了多个标准目录：
    // $XDG_DATA_HOME   — 用户数据（默认 ~/.local/share）
    // $XDG_CONFIG_HOME  — 用户配置（默认 ~/.config）
    // $XDG_CACHE_HOME   — 用户缓存（默认 ~/.cache）
    const char* xdg = getenv("XDG_DATA_HOME");
    if (xdg && xdg[0] != '\0') {
        return std::string(xdg) + "/MiniPAL";
    }
    const char* home = getenv("HOME");
    if (!home) {
        struct passwd* pw = getpwuid(getuid());
        home = pw ? pw->pw_dir : "/tmp";
    }
    return std::string(home) + "/.local/share/MiniPAL";
#endif
}

} // namespace pal
```

**四个平台的应用数据目录对比**：

| 平台 | 典型路径 | 来源 |
|------|---------|------|
| **Windows** | `C:\Users\<用户>\AppData\Roaming\MiniPAL` | `SHGetFolderPathA(CSIDL_APPDATA)` |
| **Linux** | `~/.local/share/MiniPAL` | `$XDG_DATA_HOME` 或默认值 |
| **macOS** | `~/Library/Application Support/MiniPAL` | Apple 惯例 |
| **Android** | `/data/data/<包名>/files` | Java `Context.getFilesDir()` 传入 |

## Android 特殊实现

Android 的文件系统比桌面平台复杂，因为它有**两类存储**：

```
Android 存储模型:

┌─────────────────────────────────────────────┐
│                 APK 文件                      │
│  ┌───────────────────────────────────────┐   │
│  │  assets/                               │   │
│  │  ├── config.json    ← 只读              │   │
│  │  ├── level1.dat     ← 只读              │   │
│  │  └── textures/      ← 只读              │   │
│  │  必须用 AAssetManager 访问               │   │
│  └───────────────────────────────────────┘   │
└─────────────────────────────────────────────┘

┌─────────────────────────────────────────────┐
│             内部存储（可读写）                  │
│  /data/data/<包名>/files/                     │
│  ├── save.dat          ← 可读写              │
│  ├── settings.json     ← 可读写              │
│  可以用标准 POSIX API（fopen/stat/unlink）     │
└─────────────────────────────────────────────┘
```

**关键问题**：`assets/` 里的文件不是真正的"文件"——它们打包在 APK（实际上是 ZIP 文件）内部，没有独立的 inode，不能用 `stat()`、`fopen()` 等标准 API 访问。必须用 NDK 提供的 `AAssetManager` API。

```cpp
// src/android/filesystem.cpp — Android 文件系统实现
#include "pal/filesystem.h"
#include <sys/stat.h>
#include <unistd.h>
#include <cstdio>
#include <android/asset_manager.h>     // AAssetManager API
#include <android/asset_manager_jni.h> // AAssetManager_fromJava

// 全局 AAssetManager 指针
// 在 JNI_OnLoad 或 Activity 创建时由 Java 层传入
static AAssetManager* g_asset_manager = nullptr;

namespace pal {

// 初始化函数：由 Java 层在 Activity.onCreate 中调用
// 必须在使用任何 FileExists/GetFileInfo 之前调用
void SetAssetManager(AAssetManager* mgr) {
    g_asset_manager = mgr;
}
```

**为什么 `g_asset_manager` 是全局变量？**

`AAssetManager` 的生命周期由 Java 层的 `Activity` 管理。C++ 层不能自行创建它，只能接收 Java 层传递的指针。使用全局变量是 Android NDK 开发的常见模式（spdlog、SDL2 等库都这样做）。KrKr2 的 `AndroidUtils.cpp` 中也有类似的全局状态管理。

```cpp
bool FileExists(const std::string& path) {
    // 策略：先查常规文件系统，再查 APK 资产
    // 这样优先使用用户下载/创建的文件（比如存档），
    // 资产作为回退（比如默认配置）

    // 1. 检查常规文件系统（内部存储/外部存储）
    if (access(path.c_str(), F_OK) == 0) {
        return true;
    }

    // 2. 检查 APK 内的 assets/ 目录
    if (g_asset_manager) {
        AAsset* asset = AAssetManager_open(
            g_asset_manager,
            path.c_str(),
            AASSET_MODE_UNKNOWN  // 不指定访问模式（只是检查存在性）
        );
        if (asset) {
            AAsset_close(asset);  // 用完必须关闭
            return true;
        }
    }
    return false;
}

bool GetFileInfo(const std::string& path, FileInfo& info) {
    // 策略同上：先尝试常规 stat，再尝试 APK 资产
    struct stat st;
    if (stat(path.c_str(), &st) == 0) {
        info.size  = static_cast<uint64_t>(st.st_size);
        info.mtime = static_cast<uint64_t>(st.st_mtime);
        info.is_dir = S_ISDIR(st.st_mode);
        return true;
    }

    // APK 资产：只能获取大小，没有修改时间和目录信息
    if (g_asset_manager) {
        AAsset* asset = AAssetManager_open(
            g_asset_manager, path.c_str(), AASSET_MODE_UNKNOWN);
        if (asset) {
            info.size  = static_cast<uint64_t>(AAsset_getLength(asset));
            info.mtime = 0;        // APK 资产没有独立的修改时间
            info.is_dir = false;    // AAssetManager 不支持目录查询
            AAsset_close(asset);
            return true;
        }
    }
    return false;
}

bool DeleteFile(const std::string& path) {
    // 只能删除常规文件系统上的文件
    // APK 资产是只读的，不能删除
    return unlink(path.c_str()) == 0;
}

bool RenameFile(const std::string& from, const std::string& to) {
    // 同上：只能重命名常规文件
    return std::rename(from.c_str(), to.c_str()) == 0;
}

std::string GetAppDataDir() {
    // Android 的应用数据路径通常由 Java 层通过 JNI 传入
    // Java 代码：context.getFilesDir().getAbsolutePath()
    // 典型路径：/data/data/org.example.minipal/files
    //
    // 完整的实现应该在 JNI 初始化时保存这个路径
    // 这里硬编码一个默认值用于演示
    return "/data/data/org.example.minipal/files";
}

} // namespace pal
```

> **对照 KrKr2**：`AndroidUtils.cpp`（1002 行）中大量使用 `AAssetManager` 进行资产读取。KrKr2 通过 `JniHelper::callObjectMethod` 从 Java 层获取存储路径（调用 `Context.getFilesDir()`），这正是我们简化版 `GetAppDataDir()` 的完整形态。在生产代码中，应该像 KrKr2 那样从 Java 层传入路径，而不是硬编码。

### AAssetManager API 速查表

| 函数 | 功能 | 返回值 |
|------|------|--------|
| `AAssetManager_open(mgr, path, mode)` | 打开资产 | `AAsset*`（失败返回 `nullptr`） |
| `AAsset_getLength(asset)` | 获取资产大小 | `off_t` |
| `AAsset_read(asset, buf, count)` | 读取资产内容 | 实际读取的字节数 |
| `AAsset_seek(asset, offset, whence)` | 定位读取位置 | 新的位置（失败返回 -1） |
| `AAsset_close(asset)` | 关闭资产 | 无 |
| `AAssetManager_openDir(mgr, dirName)` | 列出目录 | `AAssetDir*` |

**访问模式**（`AAssetManager_open` 的第三个参数）：

| 模式 | 含义 | 使用场景 |
|------|------|---------|
| `AASSET_MODE_UNKNOWN` | 系统自行决定 | 只是检查存在或获取大小 |
| `AASSET_MODE_RANDOM` | 支持随机读取 | 需要 `AAsset_seek` 时 |
| `AASSET_MODE_STREAMING` | 顺序流式读取 | 大文件按块读取 |
| `AASSET_MODE_BUFFER` | 整个文件加载到内存 | 小文件一次性读取 |

## 常见错误与排查

### 错误 1：忘记 `::` 前缀导致递归调用

```cpp
// ❌ 错误：pal::DeleteFile 调用了自己
namespace pal {
bool DeleteFile(const std::string& path) {
    return DeleteFile(path.c_str());  // 无限递归！栈溢出崩溃
}
}

// ✅ 正确：用 :: 调用全局命名空间的 Win32 API
namespace pal {
bool DeleteFile(const std::string& path) {
    return ::DeleteFileA(path.c_str()) != 0;  // 调用 Windows API
}
}
```

**排查方法**：如果程序在调用 `DeleteFile` 时崩溃，且调用栈显示无限深度的 `pal::DeleteFile` 调用，就是忘了 `::` 前缀。

### 错误 2：Windows FILETIME 转换公式错误

```cpp
// ❌ 错误：忘记除以 10000000，得到的是 100ns 间隔数而不是秒数
info.mtime = ull.QuadPart - 116444736000000000ULL;
// 得到一个巨大的数字（十亿级别），不是预期的秒级时间戳

// ❌ 错误：魔数写错（少了一个 0）
info.mtime = (ull.QuadPart - 11644473600000000ULL) / 10000000ULL;
// 魔数应该是 116444736000000000（18 位数字）

// ✅ 正确：
info.mtime = (ull.QuadPart - 116444736000000000ULL) / 10000000ULL;
```

**排查方法**：如果时间戳看起来不合理（比如 2026 年的文件显示为 1600 年），先检查魔数是否正确。可以用一个已知修改时间的文件做验证：

```cpp
// 验证代码
FileInfo info;
GetFileInfo("known_file.txt", info);
printf("mtime = %llu\n", info.mtime);
// 2026-03-19 大约等于 1774070400
// 如果输出的数字远大于这个值，说明没有正确转换
```

### 错误 3：Android 忘记初始化 AAssetManager

```cpp
// ❌ 错误：没有调用 SetAssetManager，g_asset_manager 为 nullptr
// 所有 APK 资产文件的 FileExists 都返回 false
bool exists = pal::FileExists("config.json");  // 永远是 false

// ✅ 正确：在 JNI 初始化时传入 AssetManager
// Java 层:
// native void nativeInit(AssetManager assetManager);
//
// C++ 层:
extern "C" JNIEXPORT void JNICALL
Java_com_example_app_MainActivity_nativeInit(
    JNIEnv* env, jobject, jobject java_asset_mgr) {
    AAssetManager* mgr = AAssetManager_fromJava(env, java_asset_mgr);
    pal::SetAssetManager(mgr);
}
```

**排查方法**：在 `SetAssetManager` 中加一行日志确认它被调用了。如果 `g_asset_manager` 为空，说明 Java 层没有正确传入。

## 本节小结

- 文件系统是跨平台差异最大的模块之一：同一个操作在四个平台需要四种不同的 API
- 统一接口用 `bool` 返回值 + 输出参数引用，简洁高效
- Windows 的 `FILETIME`（1601 年基准、100ns 精度）需要转换为 Unix 时间戳（1970 年基准、秒精度）
- POSIX 的 `rename()` 会覆盖已存在的目标文件，Windows 的 `MoveFileA()` 不会——这是一个常见的跨平台 bug 来源
- Android 有两类存储：APK 内资产（只读，需要 `AAssetManager`）和内部存储（可读写，标准 POSIX API）
- 命名空间内调用同名的系统 API 时，必须加 `::` 前缀避免递归调用

## 练习题与答案

### 题目 1：为什么 Windows `FILETIME` 的纪元是 1601 年 1 月 1 日？这个看似奇怪的日期有什么历史原因？

<details>
<summary>查看答案</summary>

1601 年 1 月 1 日是格里高利历（公历）400 年周期的起始日。格里高利历的闰年规则以 400 年为一个完整周期（400 年中有 97 个闰年），1601 年正好是一个新周期的开始。

Windows NT 团队选择这个日期是为了简化日期计算：从 400 年周期的起始点开始计时，可以避免闰年计算中的边界情况。

相比之下，Unix 时间戳选择 1970 年 1 月 1 日（Unix 操作系统诞生的大致时间）纯粹是历史原因，没有数学上的特殊含义。

</details>

### 题目 2：修改 Android 的 `GetFileInfo` 实现，使其在 APK 资产的 `mtime` 字段中填入 APK 文件本身的修改时间（而非硬编码 0）。提示：APK 就是一个普通文件，可以对它调用 `stat()`。

<details>
<summary>查看答案</summary>

```cpp
// 需要一个函数获取 APK 路径
// 通常由 Java 层通过 JNI 传入
static std::string g_apk_path;

void SetApkPath(const std::string& path) {
    g_apk_path = path;
}

bool GetFileInfo(const std::string& path, FileInfo& info) {
    // 先尝试常规 stat
    struct stat st;
    if (stat(path.c_str(), &st) == 0) {
        info.size  = static_cast<uint64_t>(st.st_size);
        info.mtime = static_cast<uint64_t>(st.st_mtime);
        info.is_dir = S_ISDIR(st.st_mode);
        return true;
    }

    // APK 资产
    if (g_asset_manager) {
        AAsset* asset = AAssetManager_open(
            g_asset_manager, path.c_str(), AASSET_MODE_UNKNOWN);
        if (asset) {
            info.size  = static_cast<uint64_t>(AAsset_getLength(asset));
            info.is_dir = false;

            // 用 APK 文件本身的 mtime 作为近似值
            // APK 路径通常是 /data/app/<包名>/base.apk
            struct stat apk_st;
            if (!g_apk_path.empty() && stat(g_apk_path.c_str(), &apk_st) == 0) {
                info.mtime = static_cast<uint64_t>(apk_st.st_mtime);
            } else {
                info.mtime = 0;  // 无法获取时回退
            }

            AAsset_close(asset);
            return true;
        }
    }
    return false;
}
```

Java 层传入 APK 路径：
```java
// Activity.onCreate 中：
String apkPath = getApplicationInfo().sourceDir;
nativeSetApkPath(apkPath);  // 传给 C++ 层
```

这样 APK 资产的 `mtime` 就是 APK 安装/更新时间，虽然不是资产文件本身的修改时间，但在大多数场景下是一个合理的近似值。

</details>

### 题目 3：我们的 `FileExists` 实现存在 TOCTOU 竞争条件。请解释什么是 TOCTOU，并说明如何通过"直接操作"模式避免这个问题。

<details>
<summary>查看答案</summary>

**TOCTOU（Time-of-Check to Time-of-Use，检查时间与使用时间的竞争）**：

```
线程 A:                          线程 B:
  if (FileExists("data.txt")) {
    // 此时文件存在 ✅
                                   DeleteFile("data.txt");
                                   // 文件被删除 ❌
    ReadFile("data.txt");
    // 文件不存在了！崩溃 💥
  }
```

在 `FileExists` 返回 `true` 和实际使用文件之间，另一个线程或进程可能已经删除了文件。

**解决方案：直接操作模式**

不要"先检查再操作"，而是"直接操作，处理失败"：

```cpp
// ❌ TOCTOU 风险
if (pal::FileExists("save.dat")) {
    std::ifstream ifs("save.dat");  // 可能在这之间被删除
    // ...
}

// ✅ 直接操作，处理失败
std::ifstream ifs("save.dat");
if (ifs.is_open()) {
    // 文件确实打开了，安全使用
    // ...
} else {
    // 文件不存在或无权限，正常处理
}
```

直接操作模式将"检查"和"使用"合并为一步原子操作，消除了竞争窗口。在多线程或多进程环境中，这是唯一可靠的方式。

`FileExists` 仍然有其价值——用于 UI 显示（"存档是否存在"）、日志记录等不需要原子性的场景。但在任何涉及"检查→操作"的流程中，都应该优先使用直接操作模式。

</details>
