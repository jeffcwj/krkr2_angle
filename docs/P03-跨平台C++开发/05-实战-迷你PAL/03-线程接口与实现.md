## 为什么线程模块需要 PAL？

**场景**：你正在开发一个游戏引擎的资源加载器。为了不让加载过程卡住主线程，你决定在后台线程中加载纹理：

```cpp
// Windows 版本
#include <windows.h>
HANDLE thread = CreateThread(nullptr, 0, LoadTexture, &path, 0, nullptr);
WaitForSingleObject(thread, INFINITE);
CloseHandle(thread);
```

换到 Linux 上编译？直接报错——Linux 没有 `CreateThread`，得用 `pthread_create`：

```cpp
// Linux 版本
#include <pthread.h>
pthread_t thread;
pthread_create(&thread, nullptr, LoadTexture, &path);
pthread_join(thread, nullptr);
```

不仅 API 不同，**连函数签名都不兼容**：

| 平台 | 线程入口函数签名 | 返回值 |
|------|----------------|--------|
| Windows | `DWORD WINAPI func(LPVOID param)` | `DWORD`（32 位无符号整数） |
| POSIX | `void* func(void* param)` | `void*`（通用指针） |

而互斥锁的差异更大：

| 平台 | 互斥锁类型 | 运行层级 |
|------|-----------|---------|
| Windows | `CRITICAL_SECTION` | 用户态（极快，不进内核） |
| Windows | `HANDLE`（Mutex 内核对象） | 内核态（可跨进程） |
| POSIX | `pthread_mutex_t` | 通常用户态（futex 优化） |

> **"你可能会问：C++11 不是已经有 `std::thread` 和 `std::mutex` 了吗？"**
>
> 没错。现代 C++ 项目应该优先使用标准库。但有两个理由让我们还是要学习原生 API：
> 1. **理解底层**：`std::thread` 在 Windows 上就是封装了 `CreateThread`，在 Linux 上就是封装了 `pthread_create`。不理解底层，就无法排查诡异的线程 bug
> 2. **对照 KrKr2**：KrKr2 的部分代码直接使用原生线程 API（如 `ThreadIntf.h`），理解原生 API 才能读懂项目源码

## 统一接口设计

### 为什么用 PIMPL？

在文件系统模块中，我们的接口都是自由函数（`FileExists`、`GetFileInfo`），不需要在头文件中暴露平台类型。但线程模块需要**对象**——`Mutex` 对象持有锁状态，`Thread` 对象持有线程句柄。

如果直接在头文件中放平台类型：

```cpp
// ❌ 头文件暴露了 Windows 类型
// 所有包含这个头文件的代码都必须能编译 <windows.h>
#include <windows.h>
class Mutex {
    CRITICAL_SECTION cs_;  // Linux 上编译失败！
};
```

PIMPL（Pointer to Implementation）模式解决这个问题：**头文件只声明一个不完整类型的指针，具体定义放在 .cpp 文件中**。

```cpp
// ✅ 头文件中：只有前向声明，不包含任何平台头文件
class Mutex {
    struct Impl;     // 前向声明——编译器只知道"有这么个类型"
    Impl* impl_;     // 指针大小在所有平台上都是固定的（32 位或 64 位）
};

// win32/thread.cpp 中：
struct Mutex::Impl {
    CRITICAL_SECTION cs;  // Windows 实现细节
};

// posix/thread.cpp 中：
struct Mutex::Impl {
    pthread_mutex_t mtx;  // POSIX 实现细节
};
```

PIMPL 的本质是**用一个指针的间接层，把编译期依赖推迟到链接期**。

### 完整的线程接口

```cpp
// include/pal/thread.h — 线程与互斥锁接口
#pragma once
#include "platform.h"
#include <functional>
#include <string>

namespace pal {

// ========== 互斥锁 ==========
// 互斥锁（Mutual Exclusion）是最基本的同步原语
// 它保证同一时刻只有一个线程能执行被锁保护的代码段
//
// 类比：想象一间只有一把钥匙的房间。谁拿到钥匙谁进去，
// 出来时把钥匙放回门口。没有钥匙的人只能排队等。
//
// 对照 KrKr2：项目使用 std::mutex 和 std::lock_guard（C++11 标准）
// 这里演示 OS 原生 API，帮助理解底层原理
class PAL_API Mutex {
public:
    Mutex();
    ~Mutex();

    // 获取锁（如果已被其他线程持有，当前线程会阻塞等待）
    void Lock();
    // 释放锁（必须由持有锁的线程调用）
    void Unlock();

    // 禁止拷贝：互斥锁绑定了 OS 资源，不能复制
    Mutex(const Mutex&) = delete;
    Mutex& operator=(const Mutex&) = delete;

private:
    struct Impl;           // PIMPL：隐藏平台特定类型
    Impl* impl_;
};

// ========== RAII 锁守卫 ==========
// RAII（Resource Acquisition Is Initialization）：
// 构造时自动加锁，析构时自动解锁
// 类似 std::lock_guard<std::mutex>
//
// 为什么需要 RAII？因为手动 Lock/Unlock 太容易出错：
//   mutex.Lock();
//   if (error) return;   // ❌ 忘记 Unlock！死锁！
//   mutex.Unlock();
//
// 用 LockGuard：
//   LockGuard lock(mutex);
//   if (error) return;   // ✅ 析构自动 Unlock，安全
class PAL_API LockGuard {
public:
    explicit LockGuard(Mutex& m) : mutex_(m) { mutex_.Lock(); }
    ~LockGuard() { mutex_.Unlock(); }

    LockGuard(const LockGuard&) = delete;
    LockGuard& operator=(const LockGuard&) = delete;

private:
    Mutex& mutex_;
};

// ========== 线程 ==========
// 线程是操作系统调度的最小执行单元
// 一个进程可以有多个线程，它们共享进程的内存空间
//
// 对照 KrKr2：ThreadIntf.h 定义了 tTVPThreadPriority 和 TVPExecThreadTask
class PAL_API Thread {
public:
    // 任务函数类型：无参数无返回值的可调用对象
    // std::function 可以接受 lambda、函数指针、绑定表达式等
    using TaskFunc = std::function<void()>;

    Thread();
    ~Thread();                  // 析构时自动 Join

    // 启动线程执行 func
    // 返回 false 表示启动失败（如线程已在运行）
    bool Start(TaskFunc func);

    // 等待线程执行完毕（阻塞当前线程直到目标线程退出）
    void Join();

    // 线程是否正在运行
    bool IsRunning() const;

    Thread(const Thread&) = delete;
    Thread& operator=(const Thread&) = delete;

private:
    struct Impl;
    Impl* impl_;
};

// 让出当前线程的 CPU 时间片
// 告诉操作系统："我暂时不需要 CPU，让其他线程先跑"
// 对照 KrKr2：TVPRelinquishCPU()
PAL_API void YieldThread();

// 获取 CPU 核心数
// 用于决定线程池大小、并行度等
// 对照 KrKr2：TVPGetProcessorNum()
PAL_API int GetProcessorCount();

} // namespace pal
```

### 接口设计决策

**为什么 `Thread::Start` 接受 `std::function<void()>` 而不是函数指针？**

```cpp
// 函数指针只能传全局函数或静态函数，非常受限
typedef void (*ThreadFunc)(void*);

// std::function 可以接受任何可调用对象
Thread t;
int count = 0;
t.Start([&count]() {  // ✅ lambda，可以捕获局部变量
    count++;
});
```

**为什么析构函数自动 Join？**

如果线程还在运行时 `Thread` 对象被销毁，会发生什么？在标准 C++ 中，如果 `std::thread` 对象在 joinable 状态下被销毁，程序会调用 `std::terminate()` 直接崩溃。我们的实现选择自动 Join，更安全但可能导致析构时阻塞。

```cpp
// 如果不自动 Join，下面的代码会导致问题
void dangerous() {
    Thread t;
    t.Start([]() {
        std::this_thread::sleep_for(std::chrono::seconds(10));
    });
    // t 析构——如果不 Join，线程还在运行但 Impl 已被 delete
    // 线程回调访问已释放的内存 → 未定义行为 💥
}
```

## Windows 实现

### 互斥锁：CRITICAL_SECTION

Windows 有两种互斥机制：

| 类型 | `CRITICAL_SECTION` | `HANDLE` (Mutex 内核对象) |
|------|-------------------|--------------------------|
| 运行层级 | 用户态 | 内核态 |
| 性能 | 极快（无竞争时 ~20ns） | 较慢（~1000ns） |
| 跨进程 | ❌ 不支持 | ✅ 支持 |
| 使用场景 | 线程同步（95% 的情况） | 进程间同步 |

我们选择 `CRITICAL_SECTION`，因为 PAL 只需要线程级同步。

```cpp
// src/win32/thread.cpp — Windows 线程实现
#include "pal/thread.h"
#include <windows.h>

namespace pal {

// ===== Mutex::Impl =====
struct Mutex::Impl {
    CRITICAL_SECTION cs;   // Windows 临界区
};

Mutex::Mutex() : impl_(new Impl) {
    // 初始化临界区
    // 在 Windows Vista+ 上，InitializeCriticalSection 永远不会失败
    // 在更早的版本上可能抛出 STATUS_NO_MEMORY 异常
    InitializeCriticalSection(&impl_->cs);
}

Mutex::~Mutex() {
    // 删除临界区——释放其占用的系统资源
    // 如果此时有线程还在等待这个锁，行为是未定义的
    DeleteCriticalSection(&impl_->cs);
    delete impl_;
}

void Mutex::Lock()   { EnterCriticalSection(&impl_->cs); }
void Mutex::Unlock() { LeaveCriticalSection(&impl_->cs); }
```

### 线程：CreateThread

```cpp
// ===== Thread::Impl =====
struct Thread::Impl {
    HANDLE handle = nullptr;          // 线程句柄
    Thread::TaskFunc func;            // 要执行的函数
    bool running = false;             // 运行状态标志

    // Windows 线程入口函数
    // 签名必须是 DWORD WINAPI (*)(LPVOID)，不能改变
    static DWORD WINAPI ThreadProc(LPVOID param) {
        auto* self = static_cast<Thread::Impl*>(param);
        self->func();                 // 执行用户任务
        self->running = false;        // 标记完成
        return 0;
    }
};

Thread::Thread() : impl_(new Impl) {}

Thread::~Thread() {
    Join();                           // 析构时确保线程已结束
    if (impl_->handle) {
        CloseHandle(impl_->handle);   // 释放线程句柄（OS 资源）
    }
    delete impl_;
}

bool Thread::Start(TaskFunc func) {
    if (impl_->running) return false; // 防止重复启动

    impl_->func = std::move(func);   // 移动语义避免拷贝
    impl_->running = true;

    impl_->handle = CreateThread(
        nullptr,                      // 默认安全描述符
        0,                            // 默认栈大小（通常 1MB）
        Impl::ThreadProc,             // 线程入口函数
        impl_,                        // 传递 Impl 指针作为参数
        0,                            // 立即运行（0 = 立即；CREATE_SUSPENDED = 挂起）
        nullptr                       // 不需要线程 ID
    );
    return impl_->handle != nullptr;
}

void Thread::Join() {
    if (impl_->handle && impl_->running) {
        // INFINITE = 永远等待，直到线程退出
        // 也可以指定超时毫秒数（如 5000 = 5 秒）
        WaitForSingleObject(impl_->handle, INFINITE);
    }
}

bool Thread::IsRunning() const { return impl_->running; }
```

### 辅助函数

```cpp
void YieldThread() {
    // SwitchToThread()：让出时间片给同优先级或更高优先级的就绪线程
    // 如果没有符合条件的线程，立即返回（不会让出）
    //
    // 对比 Sleep(0)：让出给所有优先级的就绪线程
    // 对比 Sleep(1)：至少等待 1ms，可能导致不必要的延迟
    SwitchToThread();
}

int GetProcessorCount() {
    SYSTEM_INFO si;
    GetSystemInfo(&si);
    return static_cast<int>(si.dwNumberOfProcessors);
}

} // namespace pal
```

## POSIX 实现（Linux + macOS + Android 共用）

Linux、macOS 和 Android 都支持 POSIX 线程（pthread）。这意味着我们只需要一份代码就能覆盖三个平台——这就是 POSIX 标准的价值。

```cpp
// src/posix/thread.cpp — POSIX 线程实现
#include "pal/thread.h"
#include <pthread.h>         // POSIX 线程 API
#include <sched.h>           // sched_yield()
#include <unistd.h>          // sysconf()

namespace pal {

// ===== Mutex::Impl =====
struct Mutex::Impl {
    pthread_mutex_t mtx;     // POSIX 互斥锁
};

Mutex::Mutex() : impl_(new Impl) {
    // pthread_mutex_init 动态初始化互斥锁
    // 第二个参数 nullptr = 使用默认属性
    //
    // 注意：PTHREAD_MUTEX_INITIALIZER 只能用于静态（全局）初始化：
    //   static pthread_mutex_t mtx = PTHREAD_MUTEX_INITIALIZER;
    // 动态分配的 mutex 必须用 pthread_mutex_init
    pthread_mutex_init(&impl_->mtx, nullptr);
}

Mutex::~Mutex() {
    // 销毁互斥锁——如果此时锁是锁定状态，行为未定义
    pthread_mutex_destroy(&impl_->mtx);
    delete impl_;
}

void Mutex::Lock()   { pthread_mutex_lock(&impl_->mtx); }
void Mutex::Unlock() { pthread_mutex_unlock(&impl_->mtx); }
```

### POSIX 线程

```cpp
// ===== Thread::Impl =====
struct Thread::Impl {
    pthread_t tid = 0;               // 线程 ID
    Thread::TaskFunc func;
    bool running = false;

    // POSIX 线程入口函数
    // 签名必须是 void* (*)(void*)，与 Windows 的 DWORD WINAPI 不同
    static void* ThreadProc(void* param) {
        auto* self = static_cast<Thread::Impl*>(param);
        self->func();
        self->running = false;
        return nullptr;               // 线程返回值（Join 时可获取）
    }
};

Thread::Thread() : impl_(new Impl) {}

Thread::~Thread() {
    Join();
    delete impl_;
    // 注意：POSIX 不需要像 Windows 那样手动关闭线程句柄
    // pthread_join 之后线程资源自动释放
}

bool Thread::Start(TaskFunc func) {
    if (impl_->running) return false;

    impl_->func = std::move(func);
    impl_->running = true;

    // pthread_create 参数：
    //   &tid       — 输出：线程 ID（用于后续 join）
    //   nullptr    — 线程属性（默认：可 join、默认栈大小）
    //   ThreadProc — 线程入口
    //   impl_      — 传递给入口函数的参数
    int ret = pthread_create(
        &impl_->tid,
        nullptr,
        Impl::ThreadProc,
        impl_
    );
    return ret == 0;  // 0 = 成功，非 0 = 错误码（如 EAGAIN = 资源不足）
}

void Thread::Join() {
    if (impl_->tid && impl_->running) {
        // pthread_join：阻塞直到目标线程退出
        // 第二个参数可以接收线程的返回值（我们不需要）
        pthread_join(impl_->tid, nullptr);
    }
}

bool Thread::IsRunning() const { return impl_->running; }
```

### POSIX 辅助函数

```cpp
void YieldThread() {
    // sched_yield()：让出 CPU 时间片
    // 对照 KrKr2：linux/Platform.cpp 第 56 行 sched_yield()
    sched_yield();
}

int GetProcessorCount() {
    // _SC_NPROCESSORS_ONLN = 当前在线（可用）的 CPU 核心数
    // 另一个选项 _SC_NPROCESSORS_CONF = 系统配置的总核心数（可能有离线的）
    return static_cast<int>(sysconf(_SC_NPROCESSORS_ONLN));
}

} // namespace pal
```

## 关键平台差异对比

| 特性 | Windows | POSIX (Linux/macOS/Android) |
|------|---------|---------------------------|
| 线程入口签名 | `DWORD WINAPI f(LPVOID)` | `void* f(void*)` |
| 创建线程 | `CreateThread()` | `pthread_create()` |
| 等待线程结束 | `WaitForSingleObject(h, INFINITE)` | `pthread_join(tid, nullptr)` |
| 释放线程资源 | `CloseHandle(h)` — 必须手动调用 | `pthread_join()` 后自动释放 |
| 互斥锁类型 | `CRITICAL_SECTION`（用户态快锁） | `pthread_mutex_t`（默认也是用户态） |
| 初始化互斥锁 | `InitializeCriticalSection()` | `pthread_mutex_init()` |
| 销毁互斥锁 | `DeleteCriticalSection()` | `pthread_mutex_destroy()` |
| 让出时间片 | `SwitchToThread()` | `sched_yield()` |
| 获取核心数 | `GetSystemInfo()` | `sysconf(_SC_NPROCESSORS_ONLN)` |
| 默认栈大小 | 1MB | 8MB (Linux) / 512KB (macOS) |

**Android 特别说明**：Android NDK 完整实现了 POSIX 线程，`pthread_create` 等函数与桌面 Linux 行为完全一致。所以 Android 直接复用 `posix/thread.cpp`，不需要单独的 `android/thread.cpp`。这也是为什么 Mini-PAL 的目录结构中 `src/android/` 下没有 `thread.cpp`。

## 常见错误与排查

### 错误 1：忘记 Join 导致资源泄漏或崩溃

```cpp
// ❌ 错误：Thread 析构时线程还在运行
void bad_example() {
    Thread t;
    t.Start([]() {
        // 执行 10 秒的任务...
        for (int i = 0; i < 10; i++) {
            // 模拟耗时操作
        }
    });
    // 函数返回，t 被销毁
    // 如果析构函数不自动 Join，线程仍在运行
    // 但 Impl 已经被 delete → 访问已释放内存 → 崩溃！
}

// ✅ 正确：显式 Join 或依赖析构函数自动 Join
void good_example() {
    Thread t;
    t.Start([]() {
        // 执行任务...
    });
    t.Join();  // 显式等待完成
    // 或者什么都不做，让 ~Thread() 自动 Join
}
```

**排查方法**：如果程序在退出时偶尔崩溃，调用栈显示访问了无效内存，检查是否有 `Thread` 对象在线程未完成时被销毁。

### 错误 2：`running` 标志的数据竞争

```cpp
// ⚠ 我们的实现有一个潜在问题
struct Thread::Impl {
    bool running = false;  // 主线程和工作线程都会读写这个变量

    static DWORD WINAPI ThreadProc(LPVOID param) {
        auto* self = static_cast<Thread::Impl*>(param);
        self->func();
        self->running = false;  // 工作线程写入
        return 0;
    }
};

bool Thread::IsRunning() const {
    return impl_->running;       // 主线程读取
}
```

严格来说，两个线程同时读写同一个 `bool` 变量是**数据竞争（Data Race）**，属于未定义行为。在实际生产代码中应该使用 `std::atomic<bool>`：

```cpp
// ✅ 生产级实现：用 atomic 消除数据竞争
#include <atomic>
struct Thread::Impl {
    std::atomic<bool> running{false};  // 原子变量，线程安全
    // ...
};
```

我们的教学版本省略了 `atomic`，是为了保持代码简洁聚焦于跨平台 API 差异。但你需要知道这个问题存在。

### 错误 3：线程入口函数签名不匹配

```cpp
// ❌ Windows 上：忘记 WINAPI 调用约定
static DWORD ThreadProc(LPVOID param) {   // 缺少 WINAPI
    // 在 x86 上会导致栈不平衡（调用约定 __cdecl vs __stdcall）
    // 在 x64 上没问题（x64 只有一种调用约定）
    // 但代码不可移植到 32 位
}

// ❌ POSIX 上：返回类型写成 int
static int ThreadProc(void* param) {  // 应该返回 void*
    // 编译器可能警告但不报错
    // 某些平台上可能导致未定义行为
}

// ✅ 正确的签名
// Windows: static DWORD WINAPI ThreadProc(LPVOID param);
// POSIX:   static void* ThreadProc(void* param);
```

**排查方法**：如果线程启动后程序立即崩溃（尤其是 32 位 Windows），检查线程入口函数是否加了 `WINAPI` 修饰符。

### 实践 1：线程基本操作测试

```cpp
// test_thread.cpp
#include "pal/thread.h"
#include "pal/log.h"       // 用日志输出测试结果
#include <cstdio>
#include <atomic>

int main() {
    printf("===== 线程 PAL 测试 =====\n\n");

    // --- 测试 1：获取 CPU 核心数 ---
    int cores = pal::GetProcessorCount();
    printf("[测试1] CPU 核心数: %d\n\n", cores);

    // --- 测试 2：基本线程启动与 Join ---
    printf("[测试2] 启动线程...\n");
    std::atomic<int> counter{0};

    pal::Thread t;
    bool started = t.Start([&counter]() {
        for (int i = 0; i < 100; i++) {
            counter++;
        }
        printf("  [工作线程] 计数完成: %d\n", counter.load());
    });

    printf("  启动结果: %s\n", started ? "成功 ✅" : "失败 ❌");
    printf("  线程运行中: %s\n", t.IsRunning() ? "是" : "否");

    t.Join();
    printf("  Join 后计数值: %d（期望 100）%s\n",
           counter.load(),
           counter.load() == 100 ? " ✅" : " ❌");
    printf("  Join 后运行中: %s\n\n",
           t.IsRunning() ? "是 ❌" : "否 ✅");

    // --- 测试 3：互斥锁保护共享数据 ---
    printf("[测试3] 互斥锁测试...\n");
    pal::Mutex mutex;
    int shared_counter = 0;
    const int iterations = 10000;

    pal::Thread t1, t2;

    t1.Start([&mutex, &shared_counter, iterations]() {
        for (int i = 0; i < iterations; i++) {
            pal::LockGuard lock(mutex);
            shared_counter++;
        }
    });

    t2.Start([&mutex, &shared_counter, iterations]() {
        for (int i = 0; i < iterations; i++) {
            pal::LockGuard lock(mutex);
            shared_counter++;
        }
    });

    t1.Join();
    t2.Join();

    int expected = iterations * 2;
    printf("  共享计数器: %d（期望 %d）%s\n",
           shared_counter, expected,
           shared_counter == expected ? " ✅" : " ❌ 数据竞争！");

    // --- 测试 4：不加锁的对比（演示数据竞争） ---
    printf("\n[测试4] 不加锁的对比...\n");
    int unsafe_counter = 0;

    pal::Thread t3, t4;
    t3.Start([&unsafe_counter, iterations]() {
        for (int i = 0; i < iterations; i++) {
            unsafe_counter++;  // ⚠ 无锁保护！
        }
    });
    t4.Start([&unsafe_counter, iterations]() {
        for (int i = 0; i < iterations; i++) {
            unsafe_counter++;  // ⚠ 无锁保护！
        }
    });
    t3.Join();
    t4.Join();

    printf("  无锁计数器: %d（期望 %d）\n", unsafe_counter, expected);
    printf("  如果值小于期望值，就是数据竞争导致的丢失更新\n");
    printf("  （多核 CPU 上几乎必然出现；单核可能碰巧正确）\n");

    // --- 测试 5：YieldThread ---
    printf("\n[测试5] YieldThread 调用...\n");
    pal::YieldThread();
    printf("  YieldThread 调用成功 ✅\n");

    printf("\n===== 测试完成 =====\n");
    return 0;
}
```

### 实践 2：用 Mutex 实现线程安全计数器类

```cpp
// safe_counter.cpp — 完整可编译示例
#include "pal/thread.h"
#include <cstdio>

// 线程安全计数器——封装 Mutex + 共享数据
class SafeCounter {
public:
    void Increment() {
        pal::LockGuard lock(mutex_);  // 构造时加锁
        value_++;
        // 离开作用域 → ~LockGuard() 自动解锁
    }

    void Decrement() {
        pal::LockGuard lock(mutex_);
        value_--;
    }

    int Get() const {
        // ⚠ 读取也需要加锁！
        // 否则可能读到中间状态（另一个线程正在写）
        // 但我们的 Mutex 没有 const 版本的 Lock
        // 生产代码应使用 mutable Mutex
        return value_;  // 简化版：int 的读取在大多数平台上是原子的
    }

private:
    mutable pal::Mutex mutex_;  // mutable 允许 const 方法中加锁
    int value_ = 0;
};

int main() {
    SafeCounter counter;
    const int per_thread = 50000;

    pal::Thread threads[4];
    for (auto& t : threads) {
        t.Start([&counter, per_thread]() {
            for (int i = 0; i < per_thread; i++) {
                counter.Increment();
            }
        });
    }

    for (auto& t : threads) {
        t.Join();
    }

    int expected = per_thread * 4;
    printf("计数器值: %d（期望 %d）%s\n",
           counter.Get(), expected,
           counter.Get() == expected ? " ✅" : " ❌");
    return 0;
}
```

## 本节小结

- 线程模块需要 PIMPL 模式来隐藏平台特定的类型（`CRITICAL_SECTION` / `pthread_mutex_t`）
- Windows 用 `CRITICAL_SECTION`（用户态快锁），POSIX 用 `pthread_mutex_t`——性能相当，API 不同
- Windows 线程入口签名是 `DWORD WINAPI f(LPVOID)`，POSIX 是 `void* f(void*)`——不能互换
- RAII 的 `LockGuard` 确保异常安全——手动 Lock/Unlock 在有 `return`/异常/`goto` 的代码中极易出错
- Android 完整支持 POSIX 线程，可以直接复用 `posix/thread.cpp`
- `running` 标志的读写在严格意义上是数据竞争，生产代码应使用 `std::atomic<bool>`
- Windows 需要手动 `CloseHandle` 释放线程句柄，POSIX 在 `pthread_join` 后自动释放

## 练习题与答案

### 题目 1：如果把 `Thread::~Thread()` 中的 `Join()` 调用去掉，会发生什么？请从 Windows 和 POSIX 两个平台分别分析。

<details>
<summary>查看答案</summary>

**Windows**：
1. `~Thread()` 中 `delete impl_` 释放了 `Impl` 结构体
2. 工作线程的 `ThreadProc` 仍在运行，它通过 `self` 指针访问 `impl_` 的成员（`func`、`running`）
3. 访问已释放的内存 → 未定义行为（可能崩溃、可能数据损坏）
4. `CloseHandle(impl_->handle)` 关闭了线程句柄，但线程本身仍在运行（`CloseHandle` 不会终止线程）

**POSIX**：
1. 同样的 `delete impl_` 释放了内存
2. 工作线程访问已释放的内存 → 未定义行为
3. 由于没有调用 `pthread_join`，线程资源（栈、内核数据结构）不会被回收
4. 如果线程是 joinable 状态（默认），线程退出后会变成"僵尸线程"，其资源直到进程退出才释放

**总结**：不管哪个平台，核心问题是一样的：**被引用的资源在仍被使用时被释放**。这就是为什么析构函数必须 Join。

</details>

### 题目 2：解释为什么 Windows 的 `CRITICAL_SECTION` 比 Mutex 内核对象快，并说明什么场景下必须使用内核 Mutex。

<details>
<summary>查看答案</summary>

**为什么 CRITICAL_SECTION 更快？**

`CRITICAL_SECTION` 在无竞争时完全在用户态运行，使用 `InterlockedCompareExchange`（原子 CAS 操作）来获取/释放锁，不需要进入内核。只有在发生竞争（多个线程同时争抢锁）时，才会退化到内核等待。

Mutex 内核对象每次 Lock/Unlock 都要进入内核态（系统调用），即使没有竞争也是如此。用户态 → 内核态的上下文切换开销在 ~1000ns 级别，而 `CRITICAL_SECTION` 无竞争时只需要 ~20ns。

**何时必须使用内核 Mutex？**

1. **跨进程同步**：两个不同的进程需要协调对共享资源（如共享内存、文件）的访问。`CRITICAL_SECTION` 只在进程内有效，内核 Mutex 可以通过名字在进程间共享
2. **需要超时等待**：`WaitForSingleObject(mutex, timeout)` 支持超时，而 `EnterCriticalSection` 会无限等待（虽然 `TryEnterCriticalSection` 可以非阻塞尝试，但没有超时参数）
3. **需要跨线程释放**：内核 Mutex 有所有权概念，但在某些高级场景下可以跨线程操作；`CRITICAL_SECTION` 必须在同一线程中获取和释放

```cpp
// 跨进程同步示例
HANDLE mutex = CreateMutexA(nullptr, FALSE, "MyApp_SharedMutex");
// 另一个进程用 OpenMutexA("MyApp_SharedMutex") 获取同一个 mutex
WaitForSingleObject(mutex, 5000);  // 等待最多 5 秒
// ... 访问共享资源 ...
ReleaseMutex(mutex);
```

</details>

### 题目 3：我们的 `Thread` 类不支持"分离"（detach）模式。请解释什么是分离线程，以及为什么我们的 PAL 设计不包含它。

<details>
<summary>查看答案</summary>

**什么是分离线程？**

分离（detach）后的线程变成"后台线程"——它独立运行，不需要任何人调用 Join。线程结束后，资源自动回收。

```cpp
// std::thread 的 detach 用法
std::thread t([]() { /* 后台任务 */ });
t.detach();  // 从此 t 对象与线程脱离关系
// t 析构不会崩溃，但你也无法知道线程何时结束
```

对应的原生 API：
- Windows：创建线程后立即 `CloseHandle`，线程继续运行
- POSIX：调用 `pthread_detach(tid)`，线程退出后自动清理

**为什么 Mini-PAL 不支持 detach？**

1. **生命周期管理困难**：分离线程无法被 Join，你不知道它何时结束。如果分离线程引用了局部变量或堆对象，这些对象可能在线程结束前被销毁，导致悬空指针
2. **资源泄漏风险**：分离线程中的资源（文件句柄、内存、锁）没有确定的释放时机
3. **退出时的竞争**：进程退出时（`main` 返回或 `exit` 被调用），分离线程可能还在运行。C++ 标准不保证分离线程会被优雅终止——它们可能在执行到一半时被强制杀死
4. **教学简化**：Mini-PAL 的目标是演示跨平台模式，不需要覆盖所有线程功能

在实际项目中，线程池（Thread Pool）通常比分离线程更安全、更可控。KrKr2 也使用了 `TVPExecThreadTask` 风格的任务分发，而不是创建大量分离线程。

</details>
