## 为什么要学 OpenGL？

**场景**：你正在研究 KrKr2 模拟器的源码，发现游戏画面有时候会出现"闪烁"——某些精灵在特定过渡效果中会短暂消失。你打开 Cocos2d-x 的渲染代码，看到一堆 `glBindTexture`、`glDrawElements`、`GL_SRC_ALPHA` 之类的调用，完全不知道它们是什么意思。

这就是为什么你需要学 OpenGL。KrKr2 的渲染路径可以简化为：

```
TJS2 脚本指令（"显示立绘 A，透明度 50%"）
    ↓
KrKr2 visual 层（图层管理、过渡效果计算）
    ↓
Cocos2d-x 渲染引擎（组织 draw call、管理纹理）
    ↓
OpenGL / OpenGL ES API（GPU 指令）
    ↓
GPU 硬件（实际绘制像素）
```

虽然 KrKr2 不直接调用 OpenGL（而是通过 Cocos2d-x），但理解 OpenGL 是理解 Cocos2d-x 行为的基础。具体来说，当你遇到以下问题时，OpenGL 知识是必需的：

- **纹理显示异常**——需要理解纹理格式、采样器参数、Mipmap
- **渲染性能问题**——需要理解 draw call、批量渲染、状态切换开销
- **移动端适配**——需要理解 OpenGL ES 的功能限制和精度差异
- **添加后处理效果**——需要理解帧缓冲、离屏渲染、多 Pass 管线

## GPU 架构基础

### CPU vs GPU：为什么需要专用硬件？

在你的电脑里，CPU 和 GPU 是两种设计哲学完全不同的芯片。理解它们的差异是理解图形编程的第一步。

**类比**：把计算任务想象成搬砖。CPU 就像一个力量型搬运工——他很聪明，能规划路线、处理意外情况（"这块砖是歪的怎么办？"），但一次只能搬几块砖。GPU 就像一万个只会直线走路的机器人——每个都不太聪明，但所有人同时搬砖，总搬运量远超那个聪明人。

| 特性 | CPU | GPU |
|------|-----|-----|
| 核心数 | 4-32 个强核心 | 数千个弱核心 |
| 单核性能 | 极高（复杂分支、乱序执行） | 较低（简单 ALU） |
| 并行能力 | 有限（线程切换开销大） | 极强（SIMT 架构，天然并行） |
| 缓存 | 大（L1/L2/L3 共数十 MB） | 小（侧重带宽而非延迟） |
| 内存带宽 | 中等（50-100 GB/s） | 极高（200-1000 GB/s） |
| 适合任务 | 串行复杂逻辑 | 大规模并行计算 |
| 典型代表 | Intel Core i9, AMD Ryzen 9 | NVIDIA RTX 4090, Apple M2 GPU |

### 为什么图形渲染天然适合 GPU？

图形渲染的核心特征是**对大量像素做相同的计算**。一个 1920×1080 的屏幕有 207 万个像素，每个像素都需要执行相同的流程：采样纹理 → 计算光照 → 混合颜色。这种"数据并行"（Data Parallelism）正是 GPU 的设计目标。

```cpp
// CPU 方式（串行处理 207 万像素）：
for (int y = 0; y < 1080; ++y)
    for (int x = 0; x < 1920; ++x)
        framebuffer[y][x] = shade(x, y);  // 一个一个来，慢！
// 假设 shade() 耗时 100ns，总计: 2073600 × 100ns ≈ 207ms（约 5 FPS）

// GPU 方式（并行处理 207 万像素）：
// 每个 GPU 核心独立处理一个像素
// 假设 4096 个核心同时工作：2073600 / 4096 ≈ 506 个批次
// 506 × 100ns ≈ 0.05ms（轻松达到 60 FPS）
void fragment_shader(int x, int y) {
    output = shade(x, y);  // 所有像素"同时"执行
}
```

> **术语解释**：**SIMT**（Single Instruction, Multiple Threads）是 GPU 的执行模型，意思是"一条指令，多个线程同时执行"。GPU 把数千个线程分成若干组（NVIDIA 叫 Warp，每组 32 个线程），同一组内的线程执行完全相同的指令，只是操作不同的数据。这就是为什么 GPU 不擅长处理 if/else 分支——如果组内一半线程走 if，一半走 else，GPU 必须两条路都走，浪费一半算力。

### GPU 的基本组成

现代 GPU 通常包含以下五大核心组件：

```
┌─────────────────────────────────────────┐
│                  GPU                     │
│                                          │
│  ┌─────────────────────────────────┐    │
│  │  ① 命令处理器（Command Proc.）   │    │
│  │   解析 CPU 提交的 GL 命令队列     │    │
│  │   类似 CPU 的指令译码器          │    │
│  └──────────┬──────────────────────┘    │
│             ↓                            │
│  ┌──────────────────────────────────┐   │
│  │  ② 顶点处理单元（Vertex Shader）  │   │
│  │  × N 个 Shader Core 并行执行      │   │
│  │  负责坐标变换（模型→屏幕空间）     │   │
│  └──────────┬───────────────────────┘   │
│             ↓                            │
│  ┌──────────────────────────────────┐   │
│  │  ③ 光栅化器（Rasterizer）         │   │
│  │  三角形 → 像素片段（固定功能硬件）  │   │
│  │  不可编程，速度极快               │   │
│  └──────────┬───────────────────────┘   │
│             ↓                            │
│  ┌──────────────────────────────────┐   │
│  │  ④ 片段处理单元（Fragment Shader） │   │
│  │  × N 个 Shader Core 并行执行      │   │
│  │  计算每个像素的最终颜色           │   │
│  └──────────┬───────────────────────┘   │
│             ↓                            │
│  ┌──────────────────────────────────┐   │
│  │  ⑤ 输出合并（Output Merger）      │   │
│  │  深度测试 / Alpha 混合 / 写入帧缓冲│   │
│  │  决定像素最终是否显示             │   │
│  └──────────────────────────────────┘   │
│                                          │
│  ┌──────────────────────────────────┐   │
│  │         显存（VRAM）              │   │
│  │  存储：纹理、顶点缓冲、帧缓冲     │   │
│  │  带宽远高于系统内存               │   │
│  └──────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

**每个组件的作用**：

1. **命令处理器**：CPU 通过 OpenGL API 提交的命令（`glDrawElements`、`glBindTexture` 等）会进入一个命令队列。命令处理器从队列中取出命令，分发给对应的硬件单元执行。这就是为什么 OpenGL 调用是"异步"的——CPU 提交命令后不等 GPU 执行完就返回了。

2. **顶点处理单元**：运行你编写的顶点着色器（Vertex Shader），将顶点从模型坐标变换到屏幕坐标。对于 2D 游戏，主要工作是用投影矩阵把像素坐标映射到 GPU 的标准化坐标。

3. **光栅化器**：固定功能硬件，不可编程。将三角形"铺满"到像素网格上——确定三角形覆盖了哪些像素位置。这个过程速度极快，因为是专门的硬件电路在做。

4. **片段处理单元**：运行你编写的片段着色器（Fragment Shader），决定每个像素的最终颜色。在 2D 游戏中，最常见的操作是"从纹理中采样颜色"。

5. **输出合并**：执行深度测试（哪个物体在前面）、Alpha 混合（半透明叠加）等操作，最终把颜色写入帧缓冲。

### 显存与系统内存的关系

GPU 有自己的专用内存（VRAM，Video RAM）。纹理、顶点数据等渲染所需的数据必须先从系统内存**上传**到显存，GPU 才能使用：

```
┌──────────┐         PCI-E 总线          ┌──────────┐
│   CPU    │  ─────────────────────────→  │   GPU    │
│          │   glTexImage2D：上传纹理      │          │
│  系统内存  │   glBufferData：上传顶点     │   显存    │
│  (RAM)   │                              │  (VRAM)  │
│  16-64GB │   ←─────────────────────────  │  4-24GB  │
│          │   glReadPixels：回读像素      │          │
└──────────┘     （很慢！尽量避免）         └──────────┘
```

> **性能关键**：CPU→GPU 的数据传输（特别是纹理上传）是一个常见的性能瓶颈。好的做法是在加载阶段一次性上传所有纹理，而不是每帧上传。KrKr2 的图像加载器（`LoadPNG.cpp`、`LoadJPEG.cpp` 等）解码图片后通过 Cocos2d-x 上传到 GPU 纹理，之后渲染时直接使用显存中的纹理，不再涉及 CPU→GPU 传输。

## OpenGL 是什么？

**OpenGL**（Open Graphics Library）是一个**跨平台的图形 API 规范**。注意关键词"规范"——OpenGL 本身不是一个库或软件，而是一份文档，定义了几百个函数的行为。具体的实现由 GPU 驱动厂商提供：

| 平台 | OpenGL 实现者 |
|------|--------------|
| Windows + NVIDIA GPU | NVIDIA 驱动 |
| Windows + AMD GPU | AMD 驱动 |
| Windows + Intel GPU | Intel 驱动 |
| macOS | Apple（已弃用 OpenGL，推荐 Metal） |
| Linux | Mesa 3D（开源实现）或厂商驱动 |
| Android | GPU 厂商驱动（Qualcomm Adreno, ARM Mali, etc.） |

这意味着**同一段 OpenGL 代码在不同 GPU 和驱动上可能行为略有不同**。这是跨平台图形编程的一大挑战——你必须在多种设备上测试。

### OpenGL vs 其他图形 API

| API | 优势 | 劣势 | KrKr2 使用 |
|-----|------|------|-----------|
| **OpenGL / ES** | 跨平台、成熟、文档丰富 | 状态机模型效率低、Apple 已弃用 | ✅ 通过 Cocos2d-x |
| **Vulkan** | 极致性能、显式控制 | 学习曲线陡峭（几千行代码画三角形） | ❌ |
| **DirectX 11/12** | Windows 平台最优支持 | 仅 Windows/Xbox | ❌ |
| **Metal** | Apple 平台最优支持 | 仅 Apple 设备 | ❌ |

对于 KrKr2 这种 2D 视觉小说引擎，OpenGL / ES 的性能完全够用。Vulkan 的显式控制对 2D 渲染来说大炮打蚊子。

## OpenGL 的状态机模型

OpenGL 最让初学者困惑的设计是它的**状态机模型**。现代 API（如 Vulkan）使用"命令缓冲区 + 显式对象"，而 OpenGL 维护一个巨大的全局状态，所有操作都隐式作用于"当前状态"。

### 什么是状态机？

**类比**：想象你在操作一个古老的卡拉 OK 点歌系统。这个系统有以下状态：

- 当前选中的歌曲（`glBindTexture` → 当前绑定的纹理）
- 当前音量（`glBlendFunc` → 当前混合模式）
- 当前播放模式（`glUseProgram` → 当前使用的着色器）

你不能说"播放歌曲 A 并用模式 B 的音量 C"（这是 Vulkan 的做法）。你只能分步操作：

1. 选择歌曲 A（`glBindTexture(GL_TEXTURE_2D, texA)`）
2. 设置音量 C（`glBlendFunc(GL_SRC_ALPHA, GL_ONE_MINUS_SRC_ALPHA)`）
3. 设置播放模式 B（`glUseProgram(shaderB)`）
4. 按下播放（`glDrawElements(GL_TRIANGLES, ...)`）

```cpp
// OpenGL 的"设置状态-执行绘制"模式
// ---- 第一步：设置状态 ----
glBindVertexArray(vao);                 // 绑定顶点数组对象
glUseProgram(shaderProgram);            // 使用着色器程序
glBindTexture(GL_TEXTURE_2D, tex);      // 绑定纹理
glUniformMatrix4fv(loc, 1, GL_FALSE, &proj[0][0]); // 设置投影矩阵

// ---- 第二步：使用当前状态执行绘制 ----
glDrawElements(GL_TRIANGLES, 6, GL_UNSIGNED_INT, 0);

// ---- 如果要画另一个物体，需要切换相关状态 ----
glBindTexture(GL_TEXTURE_2D, anotherTex);  // 切换纹理
glUniformMatrix4fv(loc, 1, GL_FALSE, &proj2[0][0]); // 切换变换矩阵
glDrawElements(GL_TRIANGLES, 6, GL_UNSIGNED_INT, 0);  // 再次绘制
```

### 状态机的陷阱

```cpp
// ❌ 常见 Bug：忘记状态是全局持久的
void DrawBackground() {
    glBindTexture(GL_TEXTURE_2D, bgTexture);
    glDrawElements(...);
    // bgTexture 仍然处于绑定状态！不会自动"解绑"
}

void DrawCharacter() {
    // 忘记绑定角色纹理——结果用了背景纹理画角色！
    glDrawElements(...);  // BUG
}

// ✅ 正确做法：每次绘制前显式设置所有需要的状态
void DrawCharacter() {
    glBindTexture(GL_TEXTURE_2D, charTexture);  // 显式绑定
    glDrawElements(...);
}
```

> **性能注意**：每次 `glBind*` / `glUseProgram` 都有 CPU 开销（驱动需要验证和记录状态变化），频繁切换纹理和着色器是渲染性能的主要杀手。这就是为什么 Cocos2d-x（以及所有游戏引擎）都实现了**批量渲染**（Batching）——把使用相同状态的绘制调用合并，减少状态切换次数。

## 本节小结

- **GPU 是专为数据并行计算设计的硬件**，拥有数千个简单核心，适合同时处理数百万像素
- **CPU 擅长串行复杂逻辑，GPU 擅长并行简单计算**——两者互补，不可替代
- **GPU 的五大核心组件**：命令处理器、顶点处理单元、光栅化器、片段处理单元、输出合并
- **OpenGL 是图形 API 规范**，由 GPU 驱动厂商实现，不同平台/驱动可能行为有差异
- **OpenGL 使用状态机模型**：先设置状态（绑定纹理、VAO、着色器），再执行绘制
- **状态是全局持久的**——忘记重新绑定是最常见的渲染 Bug 来源
- **KrKr2 通过 Cocos2d-x 间接使用 OpenGL**，不需要直接管理 GL 上下文和状态

## 练习题与答案

### 题目 1：为什么 GPU 有数千个核心但单个核心性能比 CPU 弱？这种设计取舍对图形渲染有什么好处？

<details>
<summary>查看答案</summary>

GPU 的设计目标是**吞吐量**（Throughput）而非**延迟**（Latency）。

CPU 核心为了降低单任务延迟，配备了深流水线、分支预测、乱序执行、大容量缓存等复杂机制，这些都需要大量晶体管。因此在相同芯片面积下，CPU 只能放置少量核心。

GPU 核心放弃了这些复杂机制，只保留基本的算术逻辑单元（ALU），因此同样面积可以放置数千个核心。虽然单个核心处理单个任务更慢，但数千个核心同时工作，**总吞吐量**远超 CPU。

对图形渲染的好处：屏幕上有数百万个像素，每个像素的着色计算是**独立的**（不依赖其他像素的结果）。这种"尴尬并行"（Embarrassingly Parallel）的特点完美匹配了 GPU 的多核架构。CPU 串行处理 200 万像素需要几十毫秒，GPU 并行处理只需要几毫秒。

</details>

### 题目 2：OpenGL 的状态机模型有什么缺点？为什么现代图形 API（Vulkan）放弃了这种设计？

<details>
<summary>查看答案</summary>

OpenGL 状态机模型的主要缺点：

1. **隐式依赖**：绘制结果取决于"当前绑定了什么"，但代码中看不出这种依赖关系。一个函数修改了状态，另一个函数可能受到意外影响（全局状态污染）。

2. **难以多线程**：状态机是线程局部的（每线程一个 GL 上下文），无法在多个线程中并行构建绘制命令。而现代 CPU 都是多核的，单线程提交 GL 命令成为 CPU 瓶颈。

3. **驱动开销大**：每次 `glBind*` 调用，驱动都需要验证状态组合是否合法、记录状态变化、可能进行延迟编译。这些隐式检查导致 CPU 开销不可控。

4. **冗余状态切换**：OpenGL 不知道你"其实没改变什么"——即使连续两次绑定同一个纹理，驱动也可能做一次完整的验证。

Vulkan 的解决方案：**显式控制一切**。命令缓冲区可以在多个线程中并行录制；管线状态对象（PSO）将所有状态打包为一个不可变对象，创建时验证一次，使用时零开销；资源绑定通过描述符集显式管理，不存在"当前绑定"的概念。

代价是 API 复杂度大增——画一个三角形可能需要 1000+ 行代码（OpenGL 只需 50 行左右）。

</details>

### 题目 3：假设你要为 KrKr2 添加一个自定义后处理效果（比如屏幕模糊），需要用到 GPU 渲染管线的哪些组件？简述数据流。

<details>

后处理效果的数据流：

1. **正常渲染**：先把整个游戏画面渲染到一个**离屏帧缓冲**（FBO, Framebuffer Object）的颜色纹理上，而不是直接渲染到屏幕。
   - 涉及组件：完整管线（顶点→光栅化→片段→输出合并），输出目标是 FBO 而非默认帧缓冲

2. **后处理 Pass**：把 FBO 的颜色纹理作为输入，绘制一个覆盖全屏的矩形（2 个三角形），在片段着色器中对纹理进行模糊采样：
   - **顶点处理**：4 个顶点覆盖全屏（不需要坐标变换）
   - **光栅化**：生成覆盖整个屏幕的片段
   - **片段着色器**：对每个像素，采样周围多个像素的颜色取平均（高斯模糊）
   - **输出合并**：写入默认帧缓冲（屏幕）

```
第 1 Pass：正常渲染 → FBO 纹理
第 2 Pass：全屏矩形 + 模糊着色器 → 屏幕
```

关键组件：FBO（离屏渲染目标）、片段着色器（实现模糊算法）、纹理采样器（读取 FBO 纹理）。后续章节会详细讲解帧缓冲和后处理。
