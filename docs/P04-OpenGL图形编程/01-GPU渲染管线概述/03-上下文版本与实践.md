## 场景导入：为什么我的 GL 函数调用全部失败了？

想象这个场景：你刚写好一个 OpenGL 程序，编译通过了，但运行时所有 GL 函数都返回错误。`glGenTextures` 返回 0，`glGetError` 报告 `GL_INVALID_OPERATION`，窗口一片黑。

你花了两个小时排查着色器、顶点数据、纹理加载——全都没问题。最后你发现，是因为你在**创建 GL 上下文之前**就调用了 GL 函数。或者更隐蔽的情况：你在一个**工作线程**中调用了 GL 函数，但那个线程根本没有绑定 GL 上下文。

这就是为什么理解 GL 上下文至关重要——它是整个 OpenGL 运行的前提条件。没有上下文，所有 GL 调用都是未定义行为。

---

## 什么是 GL 上下文？

**OpenGL 上下文**（GL Context）是 OpenGL 状态机的一个**实例**。上一节讲过，OpenGL 是状态机——所有 GL 函数调用都隐式操作"当前状态"。而这个"当前状态"就存储在上下文对象中。

**类比**：如果 OpenGL 是一个画板，GL 上下文就是画板本身——包括画板上的颜料、画笔、画布尺寸等所有状态。你不能凭空画画，必须先有一个画板。而且你同一时间只能操作一个画板（一个线程绑定一个上下文）。

一个 GL 上下文包含：

```
GL 上下文内容：
├── 所有绑定状态（当前 VBO、VAO、纹理、着色器程序……）
├── 所有 GL 对象（已创建的纹理 ID、缓冲区 ID、着色器 ID……）
├── 渲染状态（混合模式、深度测试开关、视口大小……）
├── 错误状态（最近的 GL 错误码）
└── 版本信息（支持的 GL 版本、扩展列表……）
```

### GL 上下文不是 OpenGL 的一部分

创建 GL 上下文**不是 OpenGL 标准定义的功能**——它由**窗口系统**（Window System）的 API 负责。这是一个很容易让初学者困惑的设计：OpenGL 定义了几百个渲染函数，但"获得调用这些函数的权利"这件事，不在 OpenGL 的管辖范围内。

| 平台 | 窗口系统 | 上下文创建 API | 头文件 |
|------|---------|--------------|--------|
| Windows | Win32 | WGL (`wglCreateContext`) | `<GL/wgl.h>` |
| Linux | X11 | GLX (`glXCreateContext`) | `<GL/glx.h>` |
| Linux | Wayland | EGL (`eglCreateContext`) | `<EGL/egl.h>` |
| macOS | Cocoa | CGL/NSOpenGLContext | `<OpenGL/OpenGL.h>` |
| Android | EGL | EGL (`eglCreateContext`) | `<EGL/egl.h>` |
| iOS | EGL | EAGLContext | `<OpenGLES/ES3/gl.h>` |

这就是为什么几乎所有跨平台 OpenGL 程序都使用**封装库**（如 SDL2、GLFW、Qt）来创建 GL 上下文——自己写 4 套平台代码太痛苦了。

## 四个平台的上下文创建

### Windows（WGL）

```cpp
// Windows 上创建 GL 上下文的简化流程
#include <windows.h>
#include <GL/gl.h>
#include <GL/wgl.h>

// 1. 创建窗口（Win32 API）
HWND hwnd = CreateWindowEx(0, L"OpenGLClass", L"GL Window",
    WS_OVERLAPPEDWINDOW, CW_USEDEFAULT, CW_USEDEFAULT,
    800, 600, nullptr, nullptr, hInstance, nullptr);
HDC hdc = GetDC(hwnd);  // 获取设备上下文

// 2. 设置像素格式（指定颜色深度、深度缓冲、双缓冲等）
PIXELFORMATDESCRIPTOR pfd = {};
pfd.nSize = sizeof(pfd);
pfd.nVersion = 1;
pfd.dwFlags = PFD_DRAW_TO_WINDOW | PFD_SUPPORT_OPENGL | PFD_DOUBLEBUFFER;
pfd.iPixelType = PFD_TYPE_RGBA;
pfd.cColorBits = 32;     // 32 位颜色（RGBA 各 8 位）
pfd.cDepthBits = 24;     // 24 位深度缓冲
pfd.cStencilBits = 8;    // 8 位模板缓冲

int format = ChoosePixelFormat(hdc, &pfd);
SetPixelFormat(hdc, format, &pfd);

// 3. 创建 GL 上下文并绑定到当前线程
HGLRC hglrc = wglCreateContext(hdc);
wglMakeCurrent(hdc, hglrc);  // 关键：绑定到当前线程

// 现在可以调用 GL 函数了！
glClearColor(0.2f, 0.3f, 0.3f, 1.0f);
glClear(GL_COLOR_BUFFER_BIT);
SwapBuffers(hdc);  // 双缓冲交换——将后缓冲内容显示到窗口

// 4. 清理
wglMakeCurrent(nullptr, nullptr);  // 解绑
wglDeleteContext(hglrc);            // 删除上下文
ReleaseDC(hwnd, hdc);
```

### Linux（GLX / EGL）

```cpp
// Linux X11 + GLX 创建 GL 上下文
#include <X11/Xlib.h>
#include <GL/glx.h>

Display* display = XOpenDisplay(nullptr);
int screen = DefaultScreen(display);

// 选择视觉配置（类似 Windows 的像素格式）
int attribs[] = {
    GLX_RGBA, GLX_DEPTH_SIZE, 24,
    GLX_DOUBLEBUFFER, None
};
XVisualInfo* vi = glXChooseVisual(display, screen, attribs);

// 创建窗口和 GL 上下文
Window root = RootWindow(display, screen);
XSetWindowAttributes swa;
swa.colormap = XCreateColormap(display, root, vi->visual, AllocNone);
Window win = XCreateWindow(display, root, 0, 0, 800, 600,
    0, vi->depth, InputOutput, vi->visual, CWColormap, &swa);
XMapWindow(display, win);

GLXContext ctx = glXCreateContext(display, vi, nullptr, GL_TRUE);
glXMakeCurrent(display, win, ctx);  // 绑定

// 渲染循环
glClearColor(0.2f, 0.3f, 0.3f, 1.0f);
glClear(GL_COLOR_BUFFER_BIT);
glXSwapBuffers(display, win);  // 双缓冲交换
```

### Android（EGL）

Android 使用 **EGL**（Embedded-System Graphics Library）——一个专为嵌入式/移动设备设计的 GL 上下文管理 API。EGL 也可用于 Linux Wayland 环境。

```cpp
// Android EGL 创建 GL ES 上下文
#include <EGL/egl.h>
#include <GLES3/gl3.h>

// 1. 获取 EGL 显示连接
EGLDisplay eglDisplay = eglGetDisplay(EGL_DEFAULT_DISPLAY);
eglInitialize(eglDisplay, nullptr, nullptr);

// 2. 选择配置
EGLint configAttribs[] = {
    EGL_RENDERABLE_TYPE, EGL_OPENGL_ES3_BIT,  // 请求 ES 3.0
    EGL_SURFACE_TYPE, EGL_WINDOW_BIT,
    EGL_BLUE_SIZE, 8,
    EGL_GREEN_SIZE, 8,
    EGL_RED_SIZE, 8,
    EGL_DEPTH_SIZE, 24,
    EGL_NONE
};
EGLConfig config;
EGLint numConfigs;
eglChooseConfig(eglDisplay, configAttribs, &config, 1, &numConfigs);

// 3. 创建窗口表面（需要原生窗口句柄，Android 通过 ANativeWindow 获取）
EGLSurface surface = eglCreateWindowSurface(
    eglDisplay, config, nativeWindow, nullptr);

// 4. 创建上下文
EGLint contextAttribs[] = {
    EGL_CONTEXT_CLIENT_VERSION, 3,  // 请求 OpenGL ES 3.0
    EGL_NONE
};
EGLContext context = eglCreateContext(
    eglDisplay, config, EGL_NO_CONTEXT, contextAttribs);

// 5. 绑定到当前线程
eglMakeCurrent(eglDisplay, surface, surface, context);

// 现在可以调用 GL ES 函数了
glClearColor(0.2f, 0.3f, 0.3f, 1.0f);
glClear(GL_COLOR_BUFFER_BIT);
eglSwapBuffers(eglDisplay, surface);  // 双缓冲交换
```

### macOS（NSOpenGLContext / CGL）

```objc
// macOS Cocoa 创建 GL 上下文（Objective-C）
#import <Cocoa/Cocoa.h>
#import <OpenGL/OpenGL.h>
#import <OpenGL/gl3.h>

NSOpenGLPixelFormatAttribute attrs[] = {
    NSOpenGLPFAOpenGLProfile, NSOpenGLProfileVersion3_2Core,  // 请求 3.2 Core
    NSOpenGLPFADoubleBuffer,
    NSOpenGLPFAColorSize, 24,
    NSOpenGLPFADepthSize, 24,
    0
};
NSOpenGLPixelFormat* pixelFormat =
    [[NSOpenGLPixelFormat alloc] initWithAttributes:attrs];
NSOpenGLContext* context =
    [[NSOpenGLContext alloc] initWithFormat:pixelFormat shareContext:nil];
[context makeCurrentContext];  // 绑定到当前线程

// 注意：Apple 已在 macOS 10.14 中弃用 OpenGL，推荐使用 Metal
// 但现有 OpenGL 代码仍然可以运行
```

> **KrKr2 的做法**：KrKr2 不直接调用上述任何平台 API——Cocos2d-x 的 `GLView` 类封装了所有平台的窗口创建和 GL 上下文管理。`AppDelegate.cpp` 中调用 `Director::getInstance()` 时，Cocos2d-x 已经完成了上下文创建。如果你需要追踪 GL 上下文的创建，要到 Cocos2d-x 的源码中查找。

## GL 上下文与线程

### 核心规则

**一个 GL 上下文在同一时刻只能绑定到一个线程。** 这是最重要的线程安全规则。

```
线程 A: MakeCurrent(context) → glDraw*()  ✅ 正常工作
线程 B: glDraw*()                          ❌ 没有绑定上下文，崩溃

// 转移上下文到另一个线程：
线程 A: MakeCurrent(nullptr)     // 先解绑
线程 B: MakeCurrent(context)     // 再在新线程绑定
线程 B: glDraw*()                // ✅ 现在可以了
```

### 多上下文共享

有时你需要在工作线程中加载纹理（避免阻塞渲染线程）。解决方案是创建**共享上下文**——两个上下文可以共享纹理、缓冲区等 GL 对象，但各自维护独立的绑定状态：

```cpp
// 创建共享上下文的示例（WGL）
HGLRC mainContext = wglCreateContext(hdc);    // 主渲染上下文
HGLRC loadContext = wglCreateContext(hdc);    // 加载线程上下文
wglShareLists(mainContext, loadContext);       // 共享 GL 对象

// 主线程：渲染
wglMakeCurrent(hdc, mainContext);
// ... 使用纹理渲染

// 工作线程：异步加载纹理
std::thread loader([&]() {
    wglMakeCurrent(hdc, loadContext);          // 绑定共享上下文
    GLuint tex;
    glGenTextures(1, &tex);                    // 在共享上下文中创建
    glBindTexture(GL_TEXTURE_2D, tex);
    glTexImage2D(GL_TEXTURE_2D, 0, GL_RGBA,
        width, height, 0, GL_RGBA, GL_UNSIGNED_BYTE, data);
    glFinish();                                // 确保 GPU 完成上传
    // tex 可以在主线程中使用（因为上下文共享）
});
```

> **Cocos2d-x 的方案**：Cocos2d-x 选择了更简单的策略——所有 GL 调用都在主线程（渲染线程）执行。其他线程通过 `Director::getScheduler()->performFunctionInCocosThread()` 提交回调，在渲染线程中执行 GL 操作。

## OpenGL 版本简史

### 桌面 OpenGL 版本

| 版本 | 年份 | 关键特性 | 实际意义 |
|------|------|---------|---------|
| 1.0 | 1992 | 固定管线（`glBegin`/`glEnd`） | 已废弃，不要学习 |
| 2.0 | 2004 | GLSL 着色器 1.10 | 可编程管线的开始 |
| 3.0 | 2008 | 废弃固定管线 API | 过渡版本 |
| **3.3** | **2010** | **Core Profile, VAO 必须, GLSL 3.30** | **现代 GL 基线，推荐学习** |
| 4.0 | 2010 | 曲面细分着色器 | 3D 游戏用 |
| 4.3 | 2012 | 计算着色器 | GPU 通用计算 |
| 4.6 | 2017 | SPIR-V 支持 | 最终版本 |

### OpenGL ES 版本

| 版本 | 年份 | 关键特性 | 设备支持 |
|------|------|---------|---------|
| ES 1.0/1.1 | 2003 | 固定管线（移动） | 已过时 |
| **ES 2.0** | **2007** | 可编程管线 | 99% 的 Android 设备 |
| **ES 3.0** | **2012** | 纹理格式统一, 3D 纹理, 实例化 | 90%+ 的现代设备 |
| ES 3.1 | 2014 | 计算着色器 | 高端设备 |
| ES 3.2 | 2015 | 几何/曲面着色器 | 最新设备 |

### Core Profile vs Compatibility Profile

OpenGL 3.0 引入了**废弃机制**（Deprecation），3.1 正式移除了大量旧 API。从此 OpenGL 分裂为两个 Profile：

```
┌─────────────────────────────────────────────────────┐
│            Compatibility Profile（兼容模式）          │
│                                                      │
│  包含所有 OpenGL 功能（1.0 ~ 4.6）                    │
│  ├── 现代 API（VAO, 着色器, FBO...）    ✅           │
│  └── 旧 API（glBegin/glEnd, 显示列表...）  ✅       │
│                                                      │
│  适用于：维护旧代码、快速原型                          │
│  问题：旧 API 性能差，macOS 不支持兼容模式             │
├──────────────────────────────────────────────────────┤
│              Core Profile（核心模式）                  │
│                                                      │
│  只包含现代 API                                       │
│  ├── 现代 API（VAO, 着色器, FBO...）    ✅           │
│  └── 旧 API（glBegin/glEnd...）         ❌ 已移除    │
│                                                      │
│  适用于：新项目、跨平台开发（macOS 唯一选项）           │
│  优势：API 精简，强制使用最佳实践                      │
└──────────────────────────────────────────────────────┘
```

```cpp
// 请求 Core Profile 3.3（推荐）
// GLFW 示例：
glfwWindowHint(GLFW_CONTEXT_VERSION_MAJOR, 3);
glfwWindowHint(GLFW_CONTEXT_VERSION_MINOR, 3);
glfwWindowHint(GLFW_OPENGL_PROFILE, GLFW_OPENGL_CORE_PROFILE);
// macOS 需要额外加这一行：
glfwWindowHint(GLFW_OPENGL_FORWARD_COMPAT, GL_TRUE);

// SDL2 示例：
SDL_GL_SetAttribute(SDL_GL_CONTEXT_MAJOR_VERSION, 3);
SDL_GL_SetAttribute(SDL_GL_CONTEXT_MINOR_VERSION, 3);
SDL_GL_SetAttribute(SDL_GL_CONTEXT_PROFILE_MASK, SDL_GL_CONTEXT_PROFILE_CORE);
```

> **KrKr2 的选择**：通过 Cocos2d-x 间接使用 OpenGL，桌面端默认使用 OpenGL 2.1+（兼容模式），移动端使用 OpenGL ES 2.0+。Cocos2d-x 的渲染器自动处理版本差异和 API 可用性。

## 查询 GL 上下文信息

在调试渲染问题时，了解当前运行的 GL 环境非常重要。以下代码可以打印所有关键信息：

```cpp
#include <cstdio>
// 在 GL 上下文创建并绑定后调用
void PrintGLInfo() {
    // 基本信息
    printf("GL Vendor:   %s\n", glGetString(GL_VENDOR));    // GPU 厂商
    printf("GL Renderer: %s\n", glGetString(GL_RENDERER));  // GPU 型号
    printf("GL Version:  %s\n", glGetString(GL_VERSION));   // GL 版本
    printf("GLSL Version: %s\n",
        glGetString(GL_SHADING_LANGUAGE_VERSION));           // GLSL 版本

    // 能力限制（对渲染优化很重要）
    GLint maxTexSize, maxTexUnits, maxAttribs;
    glGetIntegerv(GL_MAX_TEXTURE_SIZE, &maxTexSize);         // 最大纹理尺寸
    glGetIntegerv(GL_MAX_TEXTURE_IMAGE_UNITS, &maxTexUnits); // 片段着色器纹理单元数
    glGetIntegerv(GL_MAX_VERTEX_ATTRIBS, &maxAttribs);       // 最大顶点属性数

    printf("Max Texture Size: %d\n", maxTexSize);     // 通常 4096~16384
    printf("Max Texture Units: %d\n", maxTexUnits);   // 通常 16~32
    printf("Max Vertex Attribs: %d\n", maxAttribs);   // 通常 16

    // 查询支持的扩展
    GLint numExtensions;
    glGetIntegerv(GL_NUM_EXTENSIONS, &numExtensions);
    printf("Extensions: %d\n", numExtensions);
    for (GLint i = 0; i < numExtensions; ++i) {
        printf("  %s\n", glGetStringi(GL_EXTENSIONS, i));
    }
}
```

**不同设备的输出差异**：

| 设备 | GL_RENDERER | GL_VERSION | Max Texture |
|------|-------------|-----------|-------------|
| PC (NVIDIA RTX 3060) | GeForce RTX 3060/PCIe | 4.6.0 NVIDIA 535.154.05 | 32768 |
| PC (Intel HD 630) | Intel(R) HD Graphics 630 | 4.5.0 | 16384 |
| Android (Snapdragon 8 Gen 2) | Adreno (TM) 740 | OpenGL ES 3.2 | 16384 |
| Android (低端 Mali) | Mali-G52 | OpenGL ES 3.2 | 4096 |
| macOS (M2) | Apple M2 | 4.1 Metal | 16384 |

> **实用技巧**：在 KrKr2 启动时调用 `PrintGLInfo()` 并记入日志，可以帮助诊断用户报告的渲染问题——很多 Bug 只在特定 GPU/驱动组合上出现。

## 常见错误与排查

### 错误 1：GL 函数调用返回全黑/什么都不显示

**排查清单**：

```cpp
// 1. 是否创建了 GL 上下文？
//    → 检查 MakeCurrent 返回值
if (!wglMakeCurrent(hdc, hglrc)) {
    printf("MakeCurrent failed: %d\n", GetLastError());
}

// 2. 着色器是否编译成功？
GLint success;
glGetShaderiv(shader, GL_COMPILE_STATUS, &success);
if (!success) {
    char log[512];
    glGetShaderInfoLog(shader, 512, nullptr, log);
    printf("着色器编译失败: %s\n", log);
}

// 3. 着色器程序是否链接成功？
glGetProgramiv(program, GL_LINK_STATUS, &success);
if (!success) {
    char log[512];
    glGetProgramInfoLog(program, 512, nullptr, log);
    printf("程序链接失败: %s\n", log);
}

// 4. 顶点坐标是否在 NDC 范围内？
//    → 临时将顶点设为 (-0.5, -0.5) 到 (0.5, 0.5) 测试
```

### 错误 2：`glGetError` 返回 `GL_INVALID_OPERATION`

**最常见原因**：在没有 GL 上下文的线程调用 GL 函数。

```cpp
// ❌ 在工作线程直接调用 GL
std::thread worker([]() {
    GLuint tex;
    glGenTextures(1, &tex);  // GL_INVALID_OPERATION！
    // 这个线程没有绑定 GL 上下文
});

// ✅ 在渲染线程执行 GL 操作
void OnRenderThread() {
    GLuint tex;
    glGenTextures(1, &tex);  // 渲染线程有 GL 上下文，正常工作
}
```

### 错误 3：移动端显示异常但桌面正常

**根本原因**：OpenGL ES 不支持某些桌面 GL 特性。

| 特性 | 桌面 GL 3.3 | ES 2.0 | ES 3.0 |
|------|------------|--------|--------|
| `GL_QUADS` 图元 | ✅ | ❌ 必须用三角形 | ❌ |
| 整数顶点属性 `glVertexAttribI*` | ✅ | ❌ | ✅ |
| `glTexImage2D` 内部格式 `GL_RGBA` | ✅（宽松） | ❌ 必须精确匹配 | ✅ |
| 片段着色器 `highp float` | 总是可用 | **可能不支持** | ✅ |
| 浮点纹理 | ✅ | ❌ 需扩展 | 部分支持 |
| MRT（多渲染目标） | ✅ | ❌ | ✅ |
| 3D 纹理 | ✅ | ❌ | ✅ |

```glsl
// ❌ 桌面端正常但移动端可能失败
#version 300 es
precision mediump float;  // 没有声明默认精度
out vec4 FragColor;
void main() {
    float val = 1.0 / 3.0;  // mediump 精度可能不够
    FragColor = vec4(val, val, val, 1.0);
}

// ✅ 兼容写法
#version 300 es
precision highp float;  // 声明高精度（ES 3.0 保证支持）
out vec4 FragColor;
void main() {
    float val = 1.0 / 3.0;
    FragColor = vec4(val, val, val, 1.0);
}
```

## 动手实践

### 实践 1：查询你的 GPU 信息

在你的开发环境中（无论使用 GLFW、SDL2 还是其他库），创建一个最小的 GL 程序，调用上面的 `PrintGLInfo()` 函数。记录：

1. 你的 GPU 型号（`GL_RENDERER`）
2. 支持的 GL 版本（`GL_VERSION`）
3. 最大纹理尺寸
4. 是否支持 `GL_EXT_texture_compression_s3tc` 扩展（桌面端纹理压缩）

### 实践 2：对比 Core Profile 与 Compatibility Profile

如果你使用 GLFW，分别创建两种 Profile 的上下文，尝试调用 `glBegin(GL_TRIANGLES)`：

```cpp
// Core Profile 下调用 glBegin 会怎样？
glfwWindowHint(GLFW_OPENGL_PROFILE, GLFW_OPENGL_CORE_PROFILE);
// ...
glBegin(GL_TRIANGLES);  // 会发生什么？（提示：GL_INVALID_OPERATION）
```

## 本节小结

- **GL 上下文是 OpenGL 状态机的实例**——没有上下文，所有 GL 调用都是未定义行为
- **上下文创建不属于 OpenGL 标准**——由各平台的窗口系统 API（WGL/GLX/EGL/CGL）负责
- **一个上下文同一时刻只能绑定到一个线程**——多线程 GL 需要共享上下文或消息队列
- **OpenGL 3.3 Core Profile 是推荐的学习基线**——现代 API、跨平台、macOS 唯一选项
- **OpenGL ES 是桌面 GL 的子集**——ES 2.0 缺少许多功能，编写跨平台着色器时需注意精度和格式限制
- **KrKr2 通过 Cocos2d-x 封装了所有上下文管理**——开发者不需要直接接触 WGL/EGL/CGL

## 练习题与答案

### 题目 1：为什么在工作线程中调用 `glGenTextures` 会失败？如何正确地在工作线程中加载纹理？

<details>
<summary>查看答案</summary>

**为什么失败**：`glGenTextures` 是一个 GL 函数，它隐式操作"当前线程绑定的 GL 上下文"。如果工作线程没有调用 `MakeCurrent` 绑定上下文，GL 函数无法知道应该操作哪个状态机实例，行为是未定义的（通常返回错误或直接崩溃）。

**正确方案有两种**：

**方案 A：共享上下文**

创建一个与主上下文共享的辅助上下文，绑定到工作线程：

```cpp
// 主线程创建主上下文
HGLRC mainCtx = wglCreateContext(hdc);
// 创建共享上下文
HGLRC loadCtx = wglCreateContext(hdc);
wglShareLists(mainCtx, loadCtx);

// 工作线程
std::thread([&]() {
    wglMakeCurrent(hdc, loadCtx);  // 绑定共享上下文
    GLuint tex;
    glGenTextures(1, &tex);        // ✅ 成功
    // 上传纹理数据...
    glFinish();                    // 确保 GPU 完成
    // tex 可以在主线程使用
}).detach();
```

**方案 B：回调到渲染线程**（Cocos2d-x 的方案）

工作线程只做 CPU 侧的解码工作，GL 操作提交到渲染线程执行：

```cpp
std::thread([&]() {
    // CPU 侧：解码图片（不需要 GL 上下文）
    auto imageData = LoadAndDecodeImage("character.png");

    // 提交到渲染线程执行 GL 操作
    Director::getInstance()->getScheduler()
        ->performFunctionInCocosThread([imageData]() {
            GLuint tex;
            glGenTextures(1, &tex);
            glBindTexture(GL_TEXTURE_2D, tex);
            glTexImage2D(..., imageData);  // ✅ 在渲染线程执行
        });
}).detach();
```

</details>

### 题目 2：macOS 上创建 OpenGL Compatibility Profile 会怎样？为什么 Apple 只支持 Core Profile？

<details>
<summary>查看答案</summary>

**macOS 上创建 Compatibility Profile 会失败**——Apple 从 macOS 10.7 开始只支持 OpenGL Core Profile（最高 4.1），不提供 Compatibility Profile。

尝试请求 Compatibility Profile 会导致：
- 上下文创建失败，或
- 降级到 OpenGL 2.1（固定管线版本）

**为什么 Apple 只支持 Core Profile**：

1. **Metal 战略**：Apple 在 2014 年推出 Metal API，2018 年正式弃用 OpenGL。只支持 Core Profile 是逐步淘汰 OpenGL 的过渡策略——Core Profile 体积更小、易于维护。

2. **驱动简化**：Compatibility Profile 要求驱动同时支持 20+ 年的旧 API（`glBegin`/`glEnd`、显示列表、固定管线光照……）。维护这些旧代码需要大量工程资源，Apple 选择不投入。

3. **性能一致性**：Core Profile 的 API 更接近底层硬件模型，驱动可以更高效地翻译 GL 调用为 Metal 命令（Apple 的 GL 实现实际上是基于 Metal 的翻译层）。

**对开发者的影响**：编写跨平台 GL 代码时，**必须使用 Core Profile**。任何依赖固定管线 API 的代码在 macOS 上都无法运行。

</details>

### 题目 3：如果一个 Android 设备的 `GL_MAX_TEXTURE_SIZE` 返回 4096，而你的背景图是 4500×3000 像素，应该怎么办？

<details>
<summary>查看答案</summary>

`GL_MAX_TEXTURE_SIZE` 表示 GPU 支持的最大纹理尺寸（宽和高的最大值）。4500 > 4096，这张纹理无法直接上传。

**解决方案**：

1. **缩放纹理**（最常用）：将图片缩小到 4096×2731（等比缩放），或者 4096×3000（只缩宽度）。对于视觉小说背景，4096 像素的分辨率在手机屏幕上通常已经足够。

2. **分块纹理**：将大图切割为多个小纹理（如 4 个 2250×1500），分别上传，绘制时拼接。这种方案更复杂但保留了原始分辨率。

3. **运行时检查并降级**：

```cpp
GLint maxTexSize;
glGetIntegerv(GL_MAX_TEXTURE_SIZE, &maxTexSize);

if (imageWidth > maxTexSize || imageHeight > maxTexSize) {
    float scale = (float)maxTexSize /
        std::max(imageWidth, imageHeight);
    // 缩放图像数据
    int newWidth = (int)(imageWidth * scale);
    int newHeight = (int)(imageHeight * scale);
    ResizeImage(imageData, newWidth, newHeight);
}
```

**KrKr2 的做法**：KrKr2 的原始游戏资源通常是为 PC 屏幕设计的（1920×1080 或更大），Cocos2d-x 在加载纹理时会自动检查 `GL_MAX_TEXTURE_SIZE` 限制。超出限制的纹理会被缩放。

</details>
