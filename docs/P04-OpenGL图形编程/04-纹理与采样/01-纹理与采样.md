## 一、纹理基础概念与 UV 坐标系
纹理（Texture）是 GPU 内存中的像素数据对象。
采样（Sampling）是着色器根据 UV 坐标读取纹理数据。
纹理文件只是输入来源。
最终参与渲染的是“上传后”的 GPU 纹理对象。

从工程角度看，纹理链路通常分四步：
1. CPU 解码图片。
2. 创建并绑定 GL 纹理对象。
3. 上传像素到 GPU。
4. Shader 根据 UV 采样。

### 1.1 UV 坐标与图片坐标原点差异

OpenGL 纹理坐标常见约定：
- `(0,0)` 在左下角。
- `(1,1)` 在右上角。
- U 向右增加。
- V 向上增加。

很多图片解码库的像素顺序是“第一行=顶部”。
因此图片坐标常被理解为左上角原点。
这会直接导致“纹理上下翻转”。

常见修复策略：
1. 上传前在 CPU 侧翻转像素行。
2. 顶点 UV 写成 `v = 1.0 - v`。
3. 片元中采样前翻转 V。

建议统一在资源层做翻转。
这样 shader 与顶点数据更干净。

### 1.2 示例1：最小 UV 采样示例

```cpp
#include <GLES2/gl2.h>

struct Vertex {
    float x, y, z;
    float u, v;
};

static Vertex kQuad[4] = {
    { -1.0f, -1.0f, 0.0f, 0.0f, 0.0f }, // 左下角 -> UV(0,0)
    {  1.0f, -1.0f, 0.0f, 1.0f, 0.0f }, // 右下角 -> UV(1,0)
    { -1.0f,  1.0f, 0.0f, 0.0f, 1.0f }, // 左上角 -> UV(0,1)
    {  1.0f,  1.0f, 0.0f, 1.0f, 1.0f }, // 右上角 -> UV(1,1)
};

// 片元着色器核心：
// uniform sampler2D uTex0;
// varying vec2 vUV;
// void main() {
//     gl_FragColor = texture2D(uTex0, vUV); // 根据UV采样纹理
// }
```

## 二、纹理对象生命周期

纹理对象生命周期是最关键的稳定性基础。
标准流程如下：
1. `glGenTextures`
2. `glBindTexture`
3. `glTexImage2D`
4. （可选）`glTexSubImage2D`
5. `glDeleteTextures`

### 2.1 每一步意义

`glGenTextures`：生成纹理对象 ID。
`glBindTexture`：指定后续命令作用对象。
`glTexImage2D`：定义尺寸、格式并上传初始像素。
`glTexSubImage2D`：更新已存在纹理的子区域。
`glDeleteTextures`：释放显存资源。

### 2.2 示例2：完整生命周期代码

```cpp
#include <GLES2/gl2.h>

GLuint CreateTextureRGBA(const unsigned char* pixels, int w, int h) {
    GLuint tex = 0;
    glGenTextures(1, &tex); // 生成纹理对象
    glBindTexture(GL_TEXTURE_2D, tex); // 绑定纹理对象

    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_LINEAR); // 缩小过滤
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_LINEAR); // 放大过滤
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, GL_CLAMP_TO_EDGE); // U环绕
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, GL_CLAMP_TO_EDGE); // V环绕

    glTexImage2D(GL_TEXTURE_2D, 0, GL_RGBA, w, h, 0,
                 GL_RGBA, GL_UNSIGNED_BYTE, pixels); // 上传像素
    return tex;
}

void DestroyTexture(GLuint& tex) {
    if (tex != 0) {
        glDeleteTextures(1, &tex); // 释放纹理资源
        tex = 0;
    }
}
```

### 2.3 对照 KrKr2 的实现思路

在 `krkr2/cpp/core/visual/ogl/RenderManager_ogl.cpp` 中：
- `tTVPOGLTexture2D` 构造里创建纹理并设置参数。
- `InternalInit` 里执行 `glTexImage2D` 上传。
- `InternalUpdate` 里执行 `glTexSubImage2D` 更新。
- 析构里释放纹理对象。

这就是工程中可复用的生命周期模板。

## 三、纹理参数详解：Wrap 与 Filter

Wrap 与 Filter 决定采样行为。
它们直接影响画面效果和兼容性。

### 3.1 环绕参数

`GL_TEXTURE_WRAP_S`：U 方向。
`GL_TEXTURE_WRAP_T`：V 方向。

可选值：
- `GL_REPEAT`
- `GL_CLAMP_TO_EDGE`
- `GL_MIRRORED_REPEAT`

`GL_REPEAT` 适合平铺地面与墙体。
`GL_CLAMP_TO_EDGE` 适合 UI 图集与字体图集。
`GL_MIRRORED_REPEAT` 在边界过渡上更平滑。

### 3.2 过滤参数

`GL_TEXTURE_MIN_FILTER` 控制缩小时采样。
`GL_TEXTURE_MAG_FILTER` 控制放大时采样。

常见值：
- `GL_NEAREST`
- `GL_LINEAR`

注意：
`MAG_FILTER` 只能是 `NEAREST` 或 `LINEAR`。
`MIN_FILTER` 才能使用 mipmap 相关枚举。

### 3.3 示例3：Wrap/Filter 一次性配置

```cpp
void ConfigureSampler(GLuint tex, bool repeat, bool pixelStyle) {
    glBindTexture(GL_TEXTURE_2D, tex); // 绑定纹理

    GLint wrap = repeat ? GL_REPEAT : GL_CLAMP_TO_EDGE;
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, wrap); // U方向环绕
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, wrap); // V方向环绕

    GLint filter = pixelStyle ? GL_NEAREST : GL_LINEAR;
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, filter); // 缩小过滤
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, filter); // 放大过滤
}
```

## 四、Mipmap：多级纹理与缩小采样

Mipmap 是原纹理的多级降采样链。
每级尺寸通常是上一级的一半。

为什么需要它：
1. 降低远处纹理闪烁。
2. 减少高频噪声与摩尔纹。
3. 提升缩小时的采样稳定性。

### 4.1 生成 Mipmap

最常用方式：
`glGenerateMipmap(GL_TEXTURE_2D)`。

调用顺序通常是：
1. `glBindTexture`
2. `glTexImage2D`
3. `glGenerateMipmap`

### 4.2 Mipmap 过滤模式

`MIN_FILTER` 常见选项：
- `GL_NEAREST_MIPMAP_NEAREST`
- `GL_LINEAR_MIPMAP_NEAREST`
- `GL_NEAREST_MIPMAP_LINEAR`
- `GL_LINEAR_MIPMAP_LINEAR`

`GL_LINEAR_MIPMAP_LINEAR` 即三线性过滤。
质量最好，代价也更高。

### 4.3 示例4：启用 Mipmap 的标准代码

```cpp
void UploadWithMipmap(GLuint tex, const void* pixels, int w, int h) {
    glBindTexture(GL_TEXTURE_2D, tex); // 绑定目标纹理

    glTexImage2D(GL_TEXTURE_2D, 0, GL_RGBA, w, h, 0,
                 GL_RGBA, GL_UNSIGNED_BYTE, pixels); // 上传0级纹理

    glGenerateMipmap(GL_TEXTURE_2D); // 自动生成mipmap链

    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER,
                    GL_LINEAR_MIPMAP_LINEAR); // 缩小时三线性
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER,
                    GL_LINEAR);                // 放大时线性
}
```

如果你设置了 mipmap 类型的 `MIN_FILTER`，
但没有提供 mipmap 层，
纹理可能出现黑屏或采样异常。

## 五、纹理单元（Texture Unit）与多纹理采样

纹理单元是 sampler 与纹理对象之间的映射层。
sampler uniform 存的是“单元索引”。
它不是纹理对象 ID。

### 5.1 单元绑定规则

1. `glActiveTexture(GL_TEXTURE0 + i)`
2. `glBindTexture(GL_TEXTURE_2D, tex)`
3. `glUniform1i(locSampler, i)`

### 5.2 示例5：三纹理绑定（0/1/2）

```cpp
void BindThreeTextures(GLuint program, GLuint t0, GLuint t1, GLuint t2) {
    glUseProgram(program); // 使用目标着色器程序

    GLint loc0 = glGetUniformLocation(program, "uTex0"); // 采样器0
    GLint loc1 = glGetUniformLocation(program, "uTex1"); // 采样器1
    GLint loc2 = glGetUniformLocation(program, "uTex2"); // 采样器2

    glActiveTexture(GL_TEXTURE0); // 激活单元0
    glBindTexture(GL_TEXTURE_2D, t0); // 绑定纹理0
    glUniform1i(loc0, 0); // 告诉采样器读单元0

    glActiveTexture(GL_TEXTURE1); // 激活单元1
    glBindTexture(GL_TEXTURE_2D, t1); // 绑定纹理1
    glUniform1i(loc1, 1); // 告诉采样器读单元1

    glActiveTexture(GL_TEXTURE2); // 激活单元2
    glBindTexture(GL_TEXTURE_2D, t2); // 绑定纹理2
    glUniform1i(loc2, 2); // 告诉采样器读单元2
}
```

### 5.3 项目对应：YUVSprite 三单元采样

`krkr2/cpp/core/environ/cocos2d/YUVSprite.cpp` 中：
1. 绑定 0 号单元的 Y 纹理。
2. 绑定 1 号单元的 U 纹理。
3. 绑定 2 号单元的 V 纹理。

片元着色器读取 `CC_Texture0/1/2`。
然后完成 YUV 到 RGB 的转换。

## 六、纹理格式：内部格式与像素格式

`glTexImage2D` 里有两个“格式”参数。
很多新手会把它们混淆。

1. 内部格式（internal format）
2. 像素格式（format）

内部格式描述 GPU 存储目标。
像素格式描述 CPU 输入布局。

### 6.1 常见格式含义

- `GL_RGBA`：四通道。
- `GL_RGB`：三通道。
- `GL_LUMINANCE`：单通道亮度（ES2 常见）。
- `GL_ALPHA`：单通道 alpha（旧路径常见）。

在 ES2 场景中，
`GL_LUMINANCE` 与 `GL_ALPHA` 仍较常见。
在现代桌面核心 OpenGL 里，
通常需要更现代的单通道格式替代。

### 6.2 示例6：按源数据选择格式

```cpp
enum class SrcImageFmt {
    Gray8,
    RGB24,
    RGBA32
};

void UploadByFormat(GLuint tex, SrcImageFmt sf, int w, int h, const void* data) {
    GLenum internalFmt = GL_RGBA;
    GLenum pixelFmt = GL_RGBA;

    switch (sf) {
        case SrcImageFmt::Gray8:
            internalFmt = GL_LUMINANCE; // GPU按亮度存
            pixelFmt = GL_LUMINANCE;    // 输入也是亮度
            break;
        case SrcImageFmt::RGB24:
            internalFmt = GL_RGB;       // GPU按RGB存
            pixelFmt = GL_RGB;          // 输入RGB排列
            break;
        case SrcImageFmt::RGBA32:
            internalFmt = GL_RGBA;      // GPU按RGBA存
            pixelFmt = GL_RGBA;         // 输入RGBA排列
            break;
    }

    glBindTexture(GL_TEXTURE_2D, tex); // 绑定纹理对象
    glTexImage2D(GL_TEXTURE_2D, 0, internalFmt, w, h, 0,
                 pixelFmt, GL_UNSIGNED_BYTE, data); // 上传像素
}
```

### 6.3 对照 KrKr2 的格式映射

`RenderManager_ogl.cpp` 中 `InternalInit` 与 `InternalUpdate`：
1. Gray 映射 `GL_LUMINANCE`。
2. RGB 映射 `GL_RGB`。
3. RGBA 映射 `GL_RGBA`。

这种做法清晰且可维护。

## 七、glPixelStorei：GL_UNPACK_ALIGNMENT 与 GL_UNPACK_ROW_LENGTH

图像上传时，
除了指针，还要描述行内存布局。

### 7.1 `GL_UNPACK_ALIGNMENT`

意义：每行起始地址的对齐要求。
可用值：1、2、4、8。

默认通常是 4。
如果行字节数不是 4 的倍数，
可能出现错行、花屏、颜色错位。

### 7.2 `GL_UNPACK_ROW_LENGTH`

意义：源图每行像素数。
非常适合“有 pitch 的子区域上传”。

跨平台差异：
1. Desktop OpenGL 通常可用。
2. ES 2.0 常依赖 `GL_EXT_unpack_subimage`。

KrKr2 通过 `GL_CHECK_unpack_subimage` 检测。
支持就直接设置 `GL_UNPACK_ROW_LENGTH`。
不支持就重排数据再上传。

### 7.3 示例7：扩展可用与 fallback 两条路径

```cpp
#include <vector>
#include <cstring>

void UploadSubRectRGBA(
    GLuint tex,
    const unsigned char* src,
    int x, int y, int w, int h,
    int pitchBytes,
    bool supportUnpackSubimage
) {
    glBindTexture(GL_TEXTURE_2D, tex); // 绑定目标纹理
    glPixelStorei(GL_UNPACK_ALIGNMENT, 1); // 先设置安全对齐

    if (supportUnpackSubimage) {
        glPixelStorei(GL_UNPACK_ROW_LENGTH, pitchBytes / 4); // RGBA每像素4字节
        glTexSubImage2D(GL_TEXTURE_2D, 0, x, y, w, h,
                        GL_RGBA, GL_UNSIGNED_BYTE, src); // 直接上传
        glPixelStorei(GL_UNPACK_ROW_LENGTH, 0); // 恢复默认状态
        return;
    }

    int rowBytes = w * 4;
    std::vector<unsigned char> tmp(rowBytes * h);
    for (int i = 0; i < h; ++i) {
        memcpy(tmp.data() + i * rowBytes, src + i * pitchBytes, rowBytes); // 行重排
    }
    glTexSubImage2D(GL_TEXTURE_2D, 0, x, y, w, h,
                    GL_RGBA, GL_UNSIGNED_BYTE, tmp.data()); // 上传重排结果
}
```

记住“恢复状态”很关键。
OpenGL 是状态机。
你不恢复，后续上传会被污染。

## 八、glTexSubImage2D：局部更新

`glTexSubImage2D` 用于更新已存在纹理的子区域。
它不会重新分配整张纹理。

适用场景：
1. 字体图集增量更新。
2. 视频帧分块刷新。
3. UI 动态小块更新。

### 8.1 使用要点

1. 先保证纹理已由 `glTexImage2D` 分配。
2. 子区域不能越界。
3. 格式与类型要匹配。
4. 对齐状态要正确。

### 8.2 示例8：先分配再局部更新

```cpp
void InitEmptyRGBA(GLuint tex, int w, int h) {
    glBindTexture(GL_TEXTURE_2D, tex); // 绑定纹理
    glTexImage2D(GL_TEXTURE_2D, 0, GL_RGBA, w, h, 0,
                 GL_RGBA, GL_UNSIGNED_BYTE, nullptr); // 只分配不上传
}

void UpdatePatchRGBA(GLuint tex, int x, int y, int w, int h, const void* pixels) {
    glBindTexture(GL_TEXTURE_2D, tex); // 绑定纹理
    glPixelStorei(GL_UNPACK_ALIGNMENT, 1); // 设置对齐
    glTexSubImage2D(GL_TEXTURE_2D, 0, x, y, w, h,
                    GL_RGBA, GL_UNSIGNED_BYTE, pixels); // 更新局部区域
}
```

KrKr2 的 `InternalUpdate` 正是这一模式的工程实现。

## 九、跨平台差异：NPOT 与 unpack 扩展

NPOT（Non-Power-Of-Two）即非 2 的幂纹理。
Desktop OpenGL 现代实现通常支持较完整。
ES 2.0 老设备可能限制 `REPEAT` 或 mipmap。

推荐兼容策略：
1. 兼容优先时用 POT 纹理。
2. 失败机型降级为 `CLAMP_TO_EDGE + GL_LINEAR`。
3. 再不稳就关闭 mipmap。

`GL_UNPACK_ROW_LENGTH` 在 Desktop 常可直接用。
在 ES 2.0 常需要 `GL_EXT_unpack_subimage`。
KrKr2 用 `GL_CHECK_unpack_subimage` 做检测并回退。

## 十、常见错误与解决方案

1. **纹理黑屏**：mipmap 过滤已开但未生成 mip 链。解决：生成 mipmap 或改 `MIN_FILTER=GL_LINEAR`。
2. **纹理翻转**：UV 原点与图片原点不一致。解决：统一翻转策略（资源层/顶点层/shader层）。
3. **花屏错行**：`GL_UNPACK_ALIGNMENT` 或 `GL_UNPACK_ROW_LENGTH` 设置错误。解决：先用 alignment=1 验证，再按 pitch 精确配置。
4. **ES2 的 NPOT 异常**：REPEAT 或 mipmap 组合受限。解决：改 POT、改 CLAMP、关 mipmap。

## 本节小结

- 纹理是 GPU 对象，采样是 shader 读取流程。
- Wrap/Filter/Mipmap 共同决定采样结果。
- 多纹理核心在“纹理单元索引映射”，不是纹理 ID。
- `glPixelStorei` 与 `glTexSubImage2D` 决定动态更新稳定性。
- 跨平台必须预设 NPOT 与扩展回退路径。

<details><summary>答案</summary>sampler 绑定的是纹理单元编号。真正对象由 `glBindTexture` 绑定到该单元。</details>

<details><summary>答案</summary>会出现错行、花屏、颜色错位。因为 GL 以错误步长读取每行起始地址。</details>
