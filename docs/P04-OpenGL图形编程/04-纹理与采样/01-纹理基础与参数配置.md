## 场景引入：为什么一张图片不能直接画到屏幕上？

想象你在做一个视觉小说引擎——角色立绘、背景图、UI 按钮，全都是"图片"。你用 PNG 解码库把图片读到了内存里，得到一个 `unsigned char*` 像素数组。然后你想在 OpenGL 里显示它。

直觉做法是"把像素数组直接传给 GPU 画出来"。但 GPU 不接受"裸像素数组"——GPU 需要的是一个**纹理对象（Texture Object）**。这个对象有自己的 ID、尺寸、格式、采样参数，由 OpenGL 驱动管理。

整个流程是这样的：

```
┌──────────┐      ┌──────────────┐      ┌──────────────┐      ┌───────────┐
│ 图片文件 │ ──→  │ CPU 解码像素 │ ──→  │ 上传到 GPU   │ ──→  │ Shader    │
│ (.png)   │      │ (内存数组)   │      │ (纹理对象)   │      │ 根据 UV   │
│          │      │              │      │              │      │ 采样纹理  │
└──────────┘      └──────────────┘      └──────────────┘      └───────────┘
```

这就引出了三个核心概念：
- **纹理（Texture）**：GPU 显存中的像素数据对象，有自己的 ID 和参数
- **UV 坐标**：告诉 GPU "这个顶点对应纹理的哪个位置"
- **采样（Sampling）**：片元着色器根据 UV 坐标从纹理中读取颜色值

本节将逐步讲清这三者的关系，以及如何正确配置纹理参数。

## 一、UV 坐标系：纹理空间与图片空间的差异

### 1.1 什么是 UV 坐标

UV 坐标是纹理空间的二维坐标系。U 对应水平方向（类似 X），V 对应垂直方向（类似 Y）。无论纹理实际像素是 64×64 还是 4096×4096，UV 坐标都被归一化到 `[0.0, 1.0]` 范围：

```
V
1.0 ┌─────────────────┐
    │                 │
    │    纹理图片     │
    │                 │
0.0 └─────────────────┘
   0.0               1.0  U
```

OpenGL 的约定是：
- **原点 `(0, 0)` 在左下角**
- **`(1, 1)` 在右上角**
- U 向右递增，V **向上**递增

### 1.2 图片解码库的"上下颠倒"陷阱

问题来了：绝大多数图片解码库（stb_image、libpng、Android Bitmap）输出的像素顺序是"**第一行 = 图片顶部**"。这意味着图片像素的原点在**左上角**，而 OpenGL 纹理坐标的原点在**左下角**。

如果你直接把解码出来的像素上传到纹理，然后用标准的 UV `(0,0)→(1,1)` 去采样，画面就是上下翻转的。

```
图片解码输出（左上角原点）：        OpenGL UV 期望（左下角原点）：

(0,0)──────→ (W,0)                  (0,1)──────→ (1,1)
  │  第0行（顶部）                    │  图片顶部应在这里
  │  第1行                            │
  ↓                                   ↓
(0,H)──────→ (W,H)                  (0,0)──────→ (1,0)
```

### 1.3 三种翻转修复策略

| 策略 | 方式 | 优点 | 缺点 |
|------|------|------|------|
| **CPU 翻转像素** | 上传前逐行 swap | Shader/顶点数据干净 | 每次加载多一次内存拷贝 |
| **顶点 UV 翻转** | 将 `v` 写成 `1.0 - v` | 零运行时开销 | 所有顶点数据都需要记住翻转 |
| **Shader 翻转** | `texture2D(tex, vec2(uv.x, 1.0-uv.y))` | 修改集中 | 每个片元多一次减法 |

**工程建议**：在资源加载层统一做 CPU 翻转。这样 Shader 和顶点数据都不需要特殊处理，代码维护最简单。

stb_image 提供了开关：

```cpp
#define STB_IMAGE_IMPLEMENTATION
#include "stb_image.h"

// 在加载前设置翻转标志
stbi_set_flip_vertically_on_load(true); // 自动翻转Y轴

int w, h, channels;
unsigned char* pixels = stbi_load("character.png", &w, &h, &channels, 4);
// 现在 pixels 已经是左下角原点顺序，可以直接上传给 OpenGL
```

### 1.4 示例1：最小 UV 采样——显示一张全屏贴图

```cpp
#include <GLES2/gl2.h>

// 顶点结构：位置(x,y,z) + 纹理坐标(u,v)
struct Vertex {
    float x, y, z;   // 顶点位置（NDC坐标）
    float u, v;       // 纹理坐标
};

// 全屏四边形（两个三角形组成）
// 注意：NDC 的 (-1,-1) 对应左下角，UV 的 (0,0) 也对应左下角
static Vertex kQuad[4] = {
    //  位置x    位置y   位置z   U     V
    { -1.0f, -1.0f, 0.0f,  0.0f, 0.0f }, // 左下角
    {  1.0f, -1.0f, 0.0f,  1.0f, 0.0f }, // 右下角
    { -1.0f,  1.0f, 0.0f,  0.0f, 1.0f }, // 左上角
    {  1.0f,  1.0f, 0.0f,  1.0f, 1.0f }, // 右上角
};

// 顶点着色器
static const char* kVertSrc = R"(
    attribute vec4 aPos;       // 顶点位置
    attribute vec2 aTexCoord;  // 纹理坐标
    varying vec2 vUV;          // 传给片元的UV

    void main() {
        gl_Position = aPos;    // 直接使用NDC坐标
        vUV = aTexCoord;       // 透传UV
    }
)";

// 片元着色器
static const char* kFragSrc = R"(
    #ifdef GL_ES
    precision mediump float;   // ES设备必须指定精度
    #endif
    uniform sampler2D uTex0;   // 纹理采样器
    varying vec2 vUV;          // 从顶点着色器接收的UV

    void main() {
        gl_FragColor = texture2D(uTex0, vUV); // 根据UV从纹理采样颜色
    }
)";
```

**关键理解**：`texture2D(uTex0, vUV)` 就是"采样"操作。GPU 根据 UV 坐标在纹理中找到对应位置的颜色值返回。UV `(0.5, 0.5)` 就是纹理正中心的颜色。

## 二、纹理对象生命周期

纹理对象的管理遵循 OpenGL 资源的标准模式：**创建 → 绑定 → 配置 → 使用 → 销毁**。记住一个类比：纹理对象就像银行账户——你先开户（Gen），然后把户头选为当前操作对象（Bind），往里存钱（Upload），最后销户（Delete）。

### 2.1 五步标准流程

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│ glGenTextures │ ──→ │ glBindTexture│ ──→ │ glTexParameteri │
│  (生成ID)    │     │  (绑定目标)  │     │  (配置参数)  │
└──────────────┘     └──────────────┘     └──────┬───────┘
                                                  │
     ┌───────────────┐     ┌──────────────┐       │
     │glDeleteTextures│ ←── │ glTexImage2D │ ←─────┘
     │  (释放资源)   │     │  (上传像素)  │
     └───────────────┘     └──────────────┘
```

| 步骤 | 函数 | 说明 |
|------|------|------|
| 1. 生成 | `glGenTextures(n, &id)` | 返回 `n` 个未使用的纹理 ID（整数） |
| 2. 绑定 | `glBindTexture(target, id)` | 将 ID 绑定到目标（通常 `GL_TEXTURE_2D`），后续操作都针对它 |
| 3. 配置 | `glTexParameteri(...)` | 设置 Wrap、Filter 等采样参数 |
| 4. 上传 | `glTexImage2D(...)` | 定义纹理尺寸、格式，并上传像素数据 |
| 5. 销毁 | `glDeleteTextures(n, &id)` | 释放 GPU 显存，ID 不再有效 |

### 2.2 示例2：完整的纹理创建与销毁

```cpp
#include <GLES2/gl2.h>

// 创建 RGBA 纹理：传入解码后的像素数据
GLuint CreateTextureRGBA(const unsigned char* pixels, int w, int h) {
    GLuint tex = 0;

    // 第1步：生成纹理对象，获取ID
    glGenTextures(1, &tex);

    // 第2步：绑定到 GL_TEXTURE_2D 目标
    // 后续所有 glTexParameter / glTexImage2D 调用都作用于这个纹理
    glBindTexture(GL_TEXTURE_2D, tex);

    // 第3步：配置采样参数（先配置再上传是好习惯）
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_LINEAR);      // 缩小时线性插值
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_LINEAR);      // 放大时线性插值
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, GL_CLAMP_TO_EDGE);   // U方向钳制
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, GL_CLAMP_TO_EDGE);   // V方向钳制

    // 第4步：上传像素数据到 GPU
    // 参数说明：目标、mip层级、内部格式、宽、高、border(必须0)、像素格式、像素类型、数据指针
    glTexImage2D(GL_TEXTURE_2D,    // 目标
                 0,                 // mipmap 层级 0（原始尺寸）
                 GL_RGBA,           // GPU内部存储格式
                 w, h,              // 宽高
                 0,                 // border（OpenGL ES中必须为0）
                 GL_RGBA,           // CPU输入像素排列格式
                 GL_UNSIGNED_BYTE,  // 每个分量的数据类型（8位无符号）
                 pixels);           // 像素数据指针

    return tex;
}

// 销毁纹理：释放GPU显存
void DestroyTexture(GLuint& tex) {
    if (tex != 0) {
        glDeleteTextures(1, &tex); // 释放纹理对象
        tex = 0;                    // 将ID置零防止野引用
    }
}
```

### 2.3 glTexImage2D 参数深度解析

`glTexImage2D` 是纹理系统中最复杂的函数。它有 9 个参数，初学者经常搞混：

```cpp
void glTexImage2D(
    GLenum  target,         // 纹理目标：GL_TEXTURE_2D
    GLint   level,          // Mipmap层级：0=原始尺寸
    GLint   internalformat, // GPU内部存储格式（GPU怎么存）
    GLsizei width,          // 纹理宽度（像素）
    GLsizei height,         // 纹理高度（像素）
    GLint   border,         // 边框宽度（必须为0）
    GLenum  format,         // CPU输入像素格式（CPU数据怎么排的）
    GLenum  type,           // 像素分量数据类型
    const void* data        // 像素数据指针（可为nullptr，仅分配不上传）
);
```

**最易混淆**的是 `internalformat`（第3个参数）和 `format`（第7个参数）：
- `internalformat`：告诉 GPU "你在显存里怎么存这个纹理"
- `format`：告诉 GPU "我传给你的 CPU 内存数据是什么排列方式"

在 OpenGL ES 2.0 中，两者通常相同（都是 `GL_RGBA`、`GL_RGB` 等）。在桌面 OpenGL 3.3+ 中，`internalformat` 可以写更精确的类型（如 `GL_RGBA8`、`GL_R8`）。

### 2.4 对照 KrKr2 的纹理管理

KrKr2 项目中的纹理管理集中在以下文件：

**`cpp/core/visual/ogl/RenderManager_ogl.cpp`**（4950 行）：
- 纹理对象的创建、参数配置、像素上传都在此文件中
- 使用 `GL_CHECK_unpack_subimage` 标志检测 `GL_EXT_unpack_subimage` 扩展（第 107 行）
- 在 Windows 上主动禁用此扩展以避免驱动兼容问题（第 140 行）
- 支持 ETC/PVRTC/ASTC 等压缩纹理格式的检测与上传（第 194-208 行）

**`cpp/core/environ/cocos2d/YUVSprite.cpp`**（264 行）：
- 视频播放时使用 3 个纹理分别存储 Y/U/V 分量

这就是标准的"生命周期 + 格式选择"工程模式。后续在你自己的项目中，也应该将纹理管理封装成类似的辅助函数。

## 三、Wrap 模式：纹理边界的采样行为

### 3.1 什么是 Wrap 模式

当 UV 坐标超出 `[0.0, 1.0]` 范围时，GPU 需要决定如何处理。这就是 Wrap（环绕）模式的作用。

想象纹理是一块瓷砖。当你的地面需要 5×5 块瓷砖的面积时，UV 坐标会达到 `(5.0, 5.0)`。Wrap 模式决定了那些超出一块瓷砖范围的区域怎么填充。

### 3.2 三种 Wrap 模式对比

```
UV 范围 [0, 3]：

GL_REPEAT（重复平铺）：          GL_CLAMP_TO_EDGE（钳制到边缘）：
┌───┬───┬───┐                   ┌───┬───────────┐
│ A │ A │ A │                   │ A │ 边缘像素  │
│ B │ B │ B │                   │ B │ 拉伸重复  │
│ C │ C │ C │                   │ C │ ......... │
└───┴───┴───┘                   └───┴───────────┘

GL_MIRRORED_REPEAT（镜像重复）：
┌───┬───┬───┐
│ A │ A̅ │ A │   A̅ = A的镜像翻转
│ B │ B̅ │ B │
│ C │ C̅ │ C │
└───┴───┴───┘
```

| 模式 | 枚举值 | 行为 | 典型用途 |
|------|--------|------|----------|
| 重复 | `GL_REPEAT` | 超出部分重复整张纹理 | 地面、墙壁、天空盒等平铺场景 |
| 钳制 | `GL_CLAMP_TO_EDGE` | 超出部分取边缘像素颜色 | UI 元素、精灵图、字体图集 |
| 镜像 | `GL_MIRRORED_REPEAT` | 每次重复交替翻转 | 水面倒影、对称图案 |

### 3.3 Wrap 参数设置

```cpp
// S 方向 = U 方向（水平），T 方向 = V 方向（垂直）
// 两个方向可以独立设置不同的模式
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, GL_REPEAT);          // U方向重复
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, GL_CLAMP_TO_EDGE);   // V方向钳制
```

**为什么叫 S 和 T 而不是 U 和 V？** 这是 OpenGL 的历史命名。S/T/R 对应纹理的三个维度（3D 纹理时用 R），与 U/V/W 完全等价。

### 3.4 示例3：可切换 Wrap 模式的配置函数

```cpp
#include <GLES2/gl2.h>

// Wrap 模式枚举
enum class WrapMode {
    Repeat,     // 适用于地面、墙壁等需要平铺的场景
    Clamp,      // 适用于UI、精灵图等不需要平铺的场景
    Mirror      // 适用于需要对称效果的场景
};

// 根据场景设置 Wrap 模式
void SetTextureWrap(GLuint tex, WrapMode mode) {
    glBindTexture(GL_TEXTURE_2D, tex); // 必须先绑定目标纹理

    GLint glMode;
    switch (mode) {
        case WrapMode::Repeat:
            glMode = GL_REPEAT;           // 超出部分重复
            break;
        case WrapMode::Clamp:
            glMode = GL_CLAMP_TO_EDGE;    // 超出部分取边缘色
            break;
        case WrapMode::Mirror:
            glMode = GL_MIRRORED_REPEAT;  // 超出部分镜像翻转
            break;
    }

    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, glMode); // 水平方向
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, glMode); // 垂直方向
}
```

### 3.5 跨平台 Wrap 注意事项

| 平台 | GL_REPEAT | GL_CLAMP_TO_EDGE | GL_MIRRORED_REPEAT | 注意事项 |
|------|-----------|-------------------|--------------------|----------|
| Desktop GL 3.3+ | ✅ | ✅ | ✅ | 无限制 |
| OpenGL ES 3.0 | ✅ | ✅ | ✅ | 无限制 |
| OpenGL ES 2.0 | ⚠️ 仅 POT | ✅ | ⚠️ 仅 POT | **NPOT 纹理只能用 CLAMP_TO_EDGE** |
| WebGL 1.0 | ⚠️ 仅 POT | ✅ | ⚠️ 仅 POT | 同 ES 2.0 限制 |

> **POT** = Power Of Two（2的幂），即宽高都是 2、4、8、16、32、64、128、256、512、1024、2048、4096 等。
> **NPOT** = Non-Power Of Two（非2的幂），如 300×200、1920×1080 等。

**关键陷阱**：在 ES 2.0 上，如果纹理尺寸是 NPOT，你**只能**用 `GL_CLAMP_TO_EDGE`。使用 `GL_REPEAT` 不会报错，但画面可能是全黑或花屏。

## 四、Filter 模式：纹理放大与缩小

### 4.1 为什么需要过滤

纹理是离散的像素网格，但采样时 UV 坐标是连续的浮点数。当 UV 坐标落在两个纹理像素之间时，GPU 需要决定返回什么颜色。这就是过滤（Filter）的工作。

两种场景：
- **放大（Magnification）**：纹理比屏幕区域小，一个纹理像素被拉伸到多个屏幕像素
- **缩小（Minification）**：纹理比屏幕区域大，多个纹理像素被压缩到一个屏幕像素

### 4.2 两种基础过滤模式

```
GL_NEAREST（最近邻）：                GL_LINEAR（双线性）：
取最近的一个纹素                      取周围4个纹素加权平均

  ┌─┬─┬─┐                              ┌─┬─┬─┐
  │ │ │ │                              │ │▓│ │
  ├─┼─┼─┤    采样点 ●                  ├─┼─┼─┤    采样点 ●
  │ │█│ │    → 返回 █ 的颜色           │ │▓│▓│    → 返回 4个▓ 的加权混合
  ├─┼─┼─┤                              ├─┼─┼─┤
  │ │ │ │                              │ │ │ │
  └─┴─┴─┘                              └─┴─┴─┘
```

| 模式 | 效果 | 性能 | 适用场景 |
|------|------|------|----------|
| `GL_NEAREST` | 锐利但有锯齿，像素块状感 | 最快 | 像素风格游戏、体素渲染 |
| `GL_LINEAR` | 平滑但可能模糊 | 稍慢 | 照片、UI、3D 场景 |

### 4.3 MAG 和 MIN 过滤器的区别

OpenGL 把放大和缩小的过滤分开设置：

```cpp
// 放大过滤（MAG_FILTER）——只能选 NEAREST 或 LINEAR
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_LINEAR);

// 缩小过滤（MIN_FILTER）——可以选 NEAREST、LINEAR，或带 Mipmap 的组合
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_LINEAR);
```

**重要规则**：
- `GL_TEXTURE_MAG_FILTER`：**只能**是 `GL_NEAREST` 或 `GL_LINEAR`
- `GL_TEXTURE_MIN_FILTER`：可以是 `GL_NEAREST`、`GL_LINEAR`，或 4 种 Mipmap 组合

### 4.4 示例4：像素风格 vs 平滑风格的过滤配置

```cpp
#include <GLES2/gl2.h>

// 像素风格：适用于像素游戏、字体点阵
void ConfigurePixelStyle(GLuint tex) {
    glBindTexture(GL_TEXTURE_2D, tex);

    // 放大时保持像素锐利的方块边缘
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_NEAREST);
    // 缩小时也保持锐利
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_NEAREST);
    // 像素游戏通常不需要平铺
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, GL_CLAMP_TO_EDGE);
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, GL_CLAMP_TO_EDGE);
}

// 平滑风格：适用于照片、UI、3D场景
void ConfigureSmoothStyle(GLuint tex) {
    glBindTexture(GL_TEXTURE_2D, tex);

    // 放大时平滑插值，避免锯齿
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_LINEAR);
    // 缩小时也平滑插值
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_LINEAR);
    // UI纹理通常钳制边缘
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, GL_CLAMP_TO_EDGE);
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, GL_CLAMP_TO_EDGE);
}
```

## 五、Mipmap：解决远处纹理闪烁的利器

### 5.1 问题：远处的纹理为什么会闪烁

想象一个棋盘格纹理贴在一个远处的地面上。当相机很远时，整个纹理被缩小到很小的屏幕区域。此时一个屏幕像素可能覆盖了纹理的数十个像素。

如果只用 `GL_LINEAR` 或 `GL_NEAREST`，GPU 只会采样 1~4 个纹素来决定颜色。但实际上这个屏幕像素"应该代表"数十个纹素的混合。结果就是：
- 相机稍微移动，采样到的纹素就变了
- 画面上出现闪烁、摩尔纹、噪点

### 5.2 Mipmap 的原理

Mipmap（多级渐远纹理）的解决方案很直觉：**预先准备好缩小版的纹理**。

```
Level 0: 256×256  （原始尺寸）
Level 1: 128×128  （1/2）
Level 2: 64×64    （1/4）
Level 3: 32×32    （1/8）
Level 4: 16×16    （1/16）
Level 5: 8×8      （1/32）
Level 6: 4×4      （1/64）
Level 7: 2×2      （1/128）
Level 8: 1×1      （1/256）
```

GPU 根据纹理在屏幕上的投影大小，自动选择最接近的 Mipmap 层级。远处用小的层级，近处用大的层级。

**额外显存开销**：Mipmap 链的总显存是原始纹理的 **1/3**（等比数列求和：1/4 + 1/16 + 1/64 + ... ≈ 1/3）。这在现代 GPU 上完全可以接受。

### 5.3 生成 Mipmap

最常用的方式是调用 `glGenerateMipmap`，让 GPU 驱动自动生成所有 Mipmap 层级：

```cpp
glBindTexture(GL_TEXTURE_2D, tex);                     // 绑定目标纹理
glTexImage2D(GL_TEXTURE_2D, 0, GL_RGBA, w, h, 0,       // 先上传0级（原始尺寸）
             GL_RGBA, GL_UNSIGNED_BYTE, pixels);
glGenerateMipmap(GL_TEXTURE_2D);                        // GPU自动生成1级到最后一级
```

**调用顺序很重要**：必须先 `glTexImage2D`（上传原始数据），再 `glGenerateMipmap`（基于原始数据生成缩小版）。

### 5.4 Mipmap 过滤模式详解

当 `MIN_FILTER` 使用 Mipmap 时，有 4 种组合：

| 枚举值 | 纹素插值 | Mip层级选择 | 质量 | 性能 |
|--------|----------|-------------|------|------|
| `GL_NEAREST_MIPMAP_NEAREST` | 最近邻 | 最近层 | 最低 | 最快 |
| `GL_LINEAR_MIPMAP_NEAREST` | 双线性 | 最近层 | 中等 | 较快 |
| `GL_NEAREST_MIPMAP_LINEAR` | 最近邻 | 两层混合 | 中等 | 较快 |
| `GL_LINEAR_MIPMAP_LINEAR` | 双线性 | 两层混合 | **最高** | 最慢 |

命名规则是 `GL_[纹素方式]_MIPMAP_[层选方式]`：
- 前半部分：在选定的 Mipmap 层内怎么采样（NEAREST 还是 LINEAR）
- 后半部分：怎么选 Mipmap 层（取最近的一层 NEAREST，还是取最近两层混合 LINEAR）

`GL_LINEAR_MIPMAP_LINEAR` = **三线性过滤**（Trilinear Filtering），是最常用的高质量方案。

### 5.5 示例5：启用 Mipmap 的完整纹理创建

```cpp
#include <GLES2/gl2.h>

// 创建带 Mipmap 的高质量纹理
GLuint CreateTextureWithMipmap(const unsigned char* pixels, int w, int h) {
    GLuint tex = 0;
    glGenTextures(1, &tex);               // 生成纹理ID
    glBindTexture(GL_TEXTURE_2D, tex);     // 绑定纹理

    // 上传0级纹理数据
    glTexImage2D(GL_TEXTURE_2D, 0, GL_RGBA, w, h, 0,
                 GL_RGBA, GL_UNSIGNED_BYTE, pixels);

    // 自动生成所有Mipmap层级
    glGenerateMipmap(GL_TEXTURE_2D);

    // 缩小时使用三线性过滤（最高质量）
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER,
                    GL_LINEAR_MIPMAP_LINEAR);
    // 放大时使用双线性过滤
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_LINEAR);

    // 钳制边缘（UI/精灵图常用）
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, GL_CLAMP_TO_EDGE);
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, GL_CLAMP_TO_EDGE);

    return tex;
}

// 对比：不带 Mipmap 的简单纹理
GLuint CreateTextureNoMipmap(const unsigned char* pixels, int w, int h) {
    GLuint tex = 0;
    glGenTextures(1, &tex);
    glBindTexture(GL_TEXTURE_2D, tex);

    glTexImage2D(GL_TEXTURE_2D, 0, GL_RGBA, w, h, 0,
                 GL_RGBA, GL_UNSIGNED_BYTE, pixels);

    // 不生成 Mipmap → MIN_FILTER 不能用 *_MIPMAP_* 系列
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_LINEAR);
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_LINEAR);

    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, GL_CLAMP_TO_EDGE);
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, GL_CLAMP_TO_EDGE);

    return tex;
}
```

### 5.6 Mipmap 跨平台差异

| 平台 | glGenerateMipmap | NPOT + Mipmap | 注意事项 |
|------|------------------|---------------|----------|
| Desktop GL 3.3+ | ✅ | ✅ | 无限制 |
| OpenGL ES 3.0 | ✅ | ✅ | 无限制 |
| OpenGL ES 2.0 | ✅ | ❌ **不支持** | NPOT 纹理不能用 Mipmap，否则黑屏 |
| macOS Core Profile | ✅ | ✅ | 需要 3.2+ Core Profile |

**ES 2.0 关键限制**：如果纹理是 NPOT（如 300×200），**不能**使用 Mipmap。调用 `glGenerateMipmap` 不会报错，但 `MIN_FILTER` 设成 Mipmap 模式后纹理会变成**全黑**。

## 六、综合示例：完整的纹理加载与渲染管线

把前面的知识串起来，实现一个完整的纹理加载 + 渲染示例：

```cpp
#include <GLES2/gl2.h>
#include <cstdio>
#include <cstdlib>
#include <cstring>

// ========== 纹理管理工具 ==========

// 判断是否为2的幂
static bool IsPowerOfTwo(int x) {
    return (x > 0) && ((x & (x - 1)) == 0); // 位运算技巧
}

// 安全的纹理创建：自动处理POT/NPOT差异
GLuint CreateTextureSafe(const unsigned char* pixels, int w, int h,
                         bool wantMipmap, bool wantRepeat) {
    GLuint tex = 0;
    glGenTextures(1, &tex);
    glBindTexture(GL_TEXTURE_2D, tex);

    // 上传像素
    glTexImage2D(GL_TEXTURE_2D, 0, GL_RGBA, w, h, 0,
                 GL_RGBA, GL_UNSIGNED_BYTE, pixels);

    bool isPOT = IsPowerOfTwo(w) && IsPowerOfTwo(h);

    // Mipmap 处理：NPOT 纹理在 ES 2.0 上不能用 Mipmap
    if (wantMipmap && isPOT) {
        glGenerateMipmap(GL_TEXTURE_2D);
        glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER,
                        GL_LINEAR_MIPMAP_LINEAR); // 三线性
    } else {
        // 无 Mipmap 或 NPOT 降级
        glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_LINEAR);
    }

    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_LINEAR);

    // Wrap 处理：NPOT 纹理在 ES 2.0 上只能用 CLAMP_TO_EDGE
    if (wantRepeat && isPOT) {
        glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, GL_REPEAT);
        glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, GL_REPEAT);
    } else {
        glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, GL_CLAMP_TO_EDGE);
        glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, GL_CLAMP_TO_EDGE);
    }

    return tex;
}

// ========== 着色器工具 ==========

// 编译单个着色器
static GLuint CompileShader(GLenum type, const char* src) {
    GLuint shader = glCreateShader(type);      // 创建着色器对象
    glShaderSource(shader, 1, &src, nullptr);   // 设置源码
    glCompileShader(shader);                    // 编译

    GLint ok = 0;
    glGetShaderiv(shader, GL_COMPILE_STATUS, &ok);
    if (!ok) {
        char log[512];
        glGetShaderInfoLog(shader, 512, nullptr, log);
        printf("Shader compile error: %s\n", log); // 打印编译错误
        glDeleteShader(shader);
        return 0;
    }
    return shader;
}

// 链接着色器程序
static GLuint LinkProgram(const char* vertSrc, const char* fragSrc) {
    GLuint vs = CompileShader(GL_VERTEX_SHADER, vertSrc);
    GLuint fs = CompileShader(GL_FRAGMENT_SHADER, fragSrc);
    if (!vs || !fs) return 0;

    GLuint prog = glCreateProgram();
    glAttachShader(prog, vs);     // 挂载顶点着色器
    glAttachShader(prog, fs);     // 挂载片元着色器
    glLinkProgram(prog);          // 链接

    GLint ok = 0;
    glGetProgramiv(prog, GL_LINK_STATUS, &ok);
    if (!ok) {
        char log[512];
        glGetProgramInfoLog(prog, 512, nullptr, log);
        printf("Program link error: %s\n", log);
        glDeleteProgram(prog);
        prog = 0;
    }

    glDeleteShader(vs);  // 链接后可以释放着色器对象
    glDeleteShader(fs);
    return prog;
}

// ========== 渲染 ==========

// 顶点着色器
static const char* kVert = R"(
    attribute vec4 aPos;
    attribute vec2 aUV;
    varying vec2 vUV;
    void main() {
        gl_Position = aPos;
        vUV = aUV;
    }
)";

// 片元着色器
static const char* kFrag = R"(
    #ifdef GL_ES
    precision mediump float;
    #endif
    uniform sampler2D uTex;
    varying vec2 vUV;
    void main() {
        gl_FragColor = texture2D(uTex, vUV);
    }
)";

// 渲染一帧（假设已有 OpenGL 上下文）
void RenderFrame(GLuint program, GLuint texture) {
    float vertices[] = {
        // pos(x,y)    uv(u,v)
        -1.0f, -1.0f,  0.0f, 0.0f,  // 左下
         1.0f, -1.0f,  1.0f, 0.0f,  // 右下
        -1.0f,  1.0f,  0.0f, 1.0f,  // 左上
         1.0f,  1.0f,  1.0f, 1.0f,  // 右上
    };

    glUseProgram(program);

    // 绑定纹理到纹理单元0
    glActiveTexture(GL_TEXTURE0);              // 激活纹理单元0
    glBindTexture(GL_TEXTURE_2D, texture);     // 将纹理绑定到单元0
    glUniform1i(glGetUniformLocation(program, "uTex"), 0); // sampler指向单元0

    // 设置顶点属性
    GLint posLoc = glGetAttribLocation(program, "aPos");
    GLint uvLoc  = glGetAttribLocation(program, "aUV");

    glEnableVertexAttribArray(posLoc);
    glVertexAttribPointer(posLoc, 2, GL_FLOAT, GL_FALSE,
                          4 * sizeof(float), vertices);

    glEnableVertexAttribArray(uvLoc);
    glVertexAttribPointer(uvLoc, 2, GL_FLOAT, GL_FALSE,
                          4 * sizeof(float), vertices + 2);

    glDrawArrays(GL_TRIANGLE_STRIP, 0, 4);    // 绘制四边形

    glDisableVertexAttribArray(posLoc);
    glDisableVertexAttribArray(uvLoc);
}
```

```cpp
// ✅ 修复方案B：手动逐行翻转
void FlipImageVertically(unsigned char* pixels, int w, int h, int channels) {
    int rowBytes = w * channels;                    // 每行字节数
    unsigned char* tmpRow = new unsigned char[rowBytes];
    for (int y = 0; y < h / 2; y++) {
        unsigned char* top = pixels + y * rowBytes;           // 顶部行
        unsigned char* bot = pixels + (h - 1 - y) * rowBytes; // 底部行
        memcpy(tmpRow, top, rowBytes);   // 暂存顶部
        memcpy(top, bot, rowBytes);      // 底部→顶部
        memcpy(bot, tmpRow, rowBytes);   // 暂存→底部
    }
    delete[] tmpRow;
}
```

## 本节小结

- **纹理不是图片**：纹理是 GPU 显存中的对象，图片只是纹理的数据来源
- **UV 坐标归一化**：无论纹理实际尺寸多大，UV 坐标都在 `[0.0, 1.0]` 范围内
- **注意翻转**：图片解码通常是左上角原点，OpenGL 纹理坐标是左下角原点，必须统一
- **生命周期五步**：Gen → Bind → Configure → Upload → Delete
- **Wrap 模式**：REPEAT 平铺、CLAMP_TO_EDGE 钳制、MIRRORED_REPEAT 镜像
- **Filter 模式**：NEAREST 锐利但有锯齿、LINEAR 平滑但略模糊
- **Mipmap**：预生成的缩小版纹理链，解决远处闪烁问题，额外 1/3 显存开销
- **NPOT 陷阱**：ES 2.0 上 NPOT 纹理只能用 CLAMP_TO_EDGE，不能用 Mipmap

## 练习题与答案

### 题目1：为什么 `glTexImage2D` 有两个"格式"参数（internalformat 和 format）？它们分别控制什么？

<details>
<summary>查看答案</summary>

`internalformat`（第 3 个参数）告诉 GPU "在显存中以什么格式存储这个纹理"，例如 `GL_RGBA8` 表示每通道 8 位共 32 位。

`format`（第 7 个参数）告诉 GPU "我传给你的 CPU 内存数据是什么排列方式"，例如 `GL_RGBA` 表示数据按 R-G-B-A 顺序排列。

在 OpenGL ES 2.0 中，两者通常相同（如都写 `GL_RGBA`）。在桌面 OpenGL 3.3+ 中，`internalformat` 可以指定更精确的类型（如 `GL_RGBA8`、`GL_R8`、`GL_RG16F`），而 `format` 只描述通道顺序。

GPU 驱动会根据这两个参数自动进行格式转换。比如 CPU 数据是 `GL_RGB`，`internalformat` 是 `GL_RGBA`，驱动会自动补上 alpha=255。

</details>

### 题目2：在 OpenGL ES 2.0 设备上，有一张 1920×1080 的背景图需要显示。你应该如何配置纹理参数？为什么不能用 `GL_REPEAT` 和 Mipmap？

<details>
<summary>查看答案</summary>

1920×1080 是 NPOT（非 2 的幂），在 ES 2.0 上受到严格限制：

```cpp
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, GL_CLAMP_TO_EDGE); // 必须用 CLAMP
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, GL_CLAMP_TO_EDGE);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_LINEAR);    // 不能用 Mipmap
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_LINEAR);
```

不能用 `GL_REPEAT` 的原因：ES 2.0 规范明确规定 NPOT 纹理只支持 `GL_CLAMP_TO_EDGE`。使用 `GL_REPEAT` 可能导致黑屏或渲染异常（具体表现因 GPU 厂商而异）。

不能用 Mipmap 的原因：ES 2.0 不支持为 NPOT 纹理生成 Mipmap 链。即使调用 `glGenerateMipmap` 不报错，设置 `MIN_FILTER` 为 Mipmap 模式后纹理会变成全黑。

如果确实需要 REPEAT 或 Mipmap，应该将纹理尺寸调整为 POT（如 2048×1024），在 CPU 侧缩放后再上传。

</details>

### 题目3：下面的代码有什么问题？运行时会出现什么现象？

```cpp
glGenTextures(1, &tex);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_LINEAR);
glTexImage2D(GL_TEXTURE_2D, 0, GL_RGBA, w, h, 0, GL_RGBA, GL_UNSIGNED_BYTE, pixels);
```

<details>
<summary>查看答案</summary>

问题：**缺少 `glBindTexture` 调用**。

`glGenTextures` 只是生成了一个纹理 ID，并没有将它绑定为当前操作对象。后续的 `glTexParameteri` 和 `glTexImage2D` 作用的是**之前绑定的纹理**（可能是纹理 0 或其他纹理），不是新生成的 `tex`。

运行时现象：
- 如果之前没有绑定任何纹理，`glTexParameteri` 和 `glTexImage2D` 作用于默认纹理对象 0（非法操作，可能产生 `GL_INVALID_OPERATION` 错误）
- 如果之前绑定了其他纹理，这些调用会意外修改那个纹理的参数和数据
- 新创建的 `tex` 永远是空的，使用时显示全黑

正确写法：

```cpp
glGenTextures(1, &tex);
glBindTexture(GL_TEXTURE_2D, tex);  // ← 必须先绑定
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_LINEAR);
glTexImage2D(GL_TEXTURE_2D, 0, GL_RGBA, w, h, 0, GL_RGBA, GL_UNSIGNED_BYTE, pixels);
```
