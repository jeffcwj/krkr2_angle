## 五、纹理单元（Texture Unit）与多纹理采样

纹理单元是 sampler 与纹理对象之间的映射层。
sampler uniform 存的是“单元索引”。
它不是纹理对象 ID。

### 5.1 单元绑定规则

1. `glActiveTexture(GL_TEXTURE0 + i)`
2. `glBindTexture(GL_TEXTURE_2D, tex)`
3. `glUniform1i(locSampler, i)`

### 5.2 示例5：三纹理绑定（0/1/2）

```cpp
void BindThreeTextures(GLuint program, GLuint t0, GLuint t1, GLuint t2) {
    glUseProgram(program); // 使用目标着色器程序

    GLint loc0 = glGetUniformLocation(program, "uTex0"); // 采样器0
    GLint loc1 = glGetUniformLocation(program, "uTex1"); // 采样器1
    GLint loc2 = glGetUniformLocation(program, "uTex2"); // 采样器2

    glActiveTexture(GL_TEXTURE0); // 激活单元0
    glBindTexture(GL_TEXTURE_2D, t0); // 绑定纹理0
    glUniform1i(loc0, 0); // 告诉采样器读单元0

    glActiveTexture(GL_TEXTURE1); // 激活单元1
    glBindTexture(GL_TEXTURE_2D, t1); // 绑定纹理1
    glUniform1i(loc1, 1); // 告诉采样器读单元1

    glActiveTexture(GL_TEXTURE2); // 激活单元2
    glBindTexture(GL_TEXTURE_2D, t2); // 绑定纹理2
    glUniform1i(loc2, 2); // 告诉采样器读单元2
}
```

### 5.3 项目对应：YUVSprite 三单元采样

`krkr2/cpp/core/environ/cocos2d/YUVSprite.cpp` 中：
1. 绑定 0 号单元的 Y 纹理。
2. 绑定 1 号单元的 U 纹理。
3. 绑定 2 号单元的 V 纹理。

片元着色器读取 `CC_Texture0/1/2`。
然后完成 YUV 到 RGB 的转换。

## 六、纹理格式：内部格式与像素格式

`glTexImage2D` 里有两个“格式”参数。
很多新手会把它们混淆。

1. 内部格式（internal format）
2. 像素格式（format）

内部格式描述 GPU 存储目标。
像素格式描述 CPU 输入布局。

### 6.1 常见格式含义

- `GL_RGBA`：四通道。
- `GL_RGB`：三通道。
- `GL_LUMINANCE`：单通道亮度（ES2 常见）。
- `GL_ALPHA`：单通道 alpha（旧路径常见）。

在 ES2 场景中，
`GL_LUMINANCE` 与 `GL_ALPHA` 仍较常见。
在现代桌面核心 OpenGL 里，
通常需要更现代的单通道格式替代。

### 6.2 示例6：按源数据选择格式

```cpp
enum class SrcImageFmt {
    Gray8,
    RGB24,
    RGBA32
};

void UploadByFormat(GLuint tex, SrcImageFmt sf, int w, int h, const void* data) {
    GLenum internalFmt = GL_RGBA;
    GLenum pixelFmt = GL_RGBA;

    switch (sf) {
        case SrcImageFmt::Gray8:
            internalFmt = GL_LUMINANCE; // GPU按亮度存
            pixelFmt = GL_LUMINANCE;    // 输入也是亮度
            break;
        case SrcImageFmt::RGB24:
            internalFmt = GL_RGB;       // GPU按RGB存
            pixelFmt = GL_RGB;          // 输入RGB排列
            break;
        case SrcImageFmt::RGBA32:
            internalFmt = GL_RGBA;      // GPU按RGBA存
            pixelFmt = GL_RGBA;         // 输入RGBA排列
            break;
    }

    glBindTexture(GL_TEXTURE_2D, tex); // 绑定纹理对象
    glTexImage2D(GL_TEXTURE_2D, 0, internalFmt, w, h, 0,
                 pixelFmt, GL_UNSIGNED_BYTE, data); // 上传像素
}
```

### 6.3 对照 KrKr2 的格式映射

`RenderManager_ogl.cpp` 中 `InternalInit` 与 `InternalUpdate`：
1. Gray 映射 `GL_LUMINANCE`。
2. RGB 映射 `GL_RGB`。
3. RGBA 映射 `GL_RGBA`。

这种做法清晰且可维护。

## 七、glPixelStorei：GL_UNPACK_ALIGNMENT 与 GL_UNPACK_ROW_LENGTH

图像上传时，
除了指针，还要描述行内存布局。

### 7.1 `GL_UNPACK_ALIGNMENT`

意义：每行起始地址的对齐要求。
可用值：1、2、4、8。

默认通常是 4。
如果行字节数不是 4 的倍数，
可能出现错行、花屏、颜色错位。

### 7.2 `GL_UNPACK_ROW_LENGTH`

意义：源图每行像素数。
非常适合“有 pitch 的子区域上传”。

跨平台差异：
1. Desktop OpenGL 通常可用。
2. ES 2.0 常依赖 `GL_EXT_unpack_subimage`。

KrKr2 通过 `GL_CHECK_unpack_subimage` 检测。
支持就直接设置 `GL_UNPACK_ROW_LENGTH`。
不支持就重排数据再上传。

### 7.3 示例7：扩展可用与 fallback 两条路径

```cpp
#include <vector>
#include <cstring>

void UploadSubRectRGBA(
    GLuint tex,
    const unsigned char* src,
    int x, int y, int w, int h,
    int pitchBytes,
    bool supportUnpackSubimage
) {
    glBindTexture(GL_TEXTURE_2D, tex); // 绑定目标纹理
    glPixelStorei(GL_UNPACK_ALIGNMENT, 1); // 先设置安全对齐

    if (supportUnpackSubimage) {
        glPixelStorei(GL_UNPACK_ROW_LENGTH, pitchBytes / 4); // RGBA每像素4字节
        glTexSubImage2D(GL_TEXTURE_2D, 0, x, y, w, h,
                        GL_RGBA, GL_UNSIGNED_BYTE, src); // 直接上传
        glPixelStorei(GL_UNPACK_ROW_LENGTH, 0); // 恢复默认状态
        return;
    }

    int rowBytes = w * 4;
    std::vector<unsigned char> tmp(rowBytes * h);
    for (int i = 0; i < h; ++i) {
        memcpy(tmp.data() + i * rowBytes, src + i * pitchBytes, rowBytes); // 行重排
    }
    glTexSubImage2D(GL_TEXTURE_2D, 0, x, y, w, h,
                    GL_RGBA, GL_UNSIGNED_BYTE, tmp.data()); // 上传重排结果
}
```

记住“恢复状态”很关键。
OpenGL 是状态机。
你不恢复，后续上传会被污染。

## 八、glTexSubImage2D：局部更新

`glTexSubImage2D` 用于更新已存在纹理的子区域。
它不会重新分配整张纹理。

适用场景：
1. 字体图集增量更新。
2. 视频帧分块刷新。
3. UI 动态小块更新。

### 8.1 使用要点

1. 先保证纹理已由 `glTexImage2D` 分配。
2. 子区域不能越界。
3. 格式与类型要匹配。
4. 对齐状态要正确。

### 8.2 示例8：先分配再局部更新

```cpp
void InitEmptyRGBA(GLuint tex, int w, int h) {
    glBindTexture(GL_TEXTURE_2D, tex); // 绑定纹理
    glTexImage2D(GL_TEXTURE_2D, 0, GL_RGBA, w, h, 0,
                 GL_RGBA, GL_UNSIGNED_BYTE, nullptr); // 只分配不上传
}

void UpdatePatchRGBA(GLuint tex, int x, int y, int w, int h, const void* pixels) {
    glBindTexture(GL_TEXTURE_2D, tex); // 绑定纹理
    glPixelStorei(GL_UNPACK_ALIGNMENT, 1); // 设置对齐
    glTexSubImage2D(GL_TEXTURE_2D, 0, x, y, w, h,
                    GL_RGBA, GL_UNSIGNED_BYTE, pixels); // 更新局部区域
}
```

KrKr2 的 `InternalUpdate` 正是这一模式的工程实现。

## 九、跨平台差异：NPOT 与 unpack 扩展

NPOT（Non-Power-Of-Two）即非 2 的幂纹理。
Desktop OpenGL 现代实现通常支持较完整。
ES 2.0 老设备可能限制 `REPEAT` 或 mipmap。

推荐兼容策略：
1. 兼容优先时用 POT 纹理。
2. 失败机型降级为 `CLAMP_TO_EDGE + GL_LINEAR`。
3. 再不稳就关闭 mipmap。

`GL_UNPACK_ROW_LENGTH` 在 Desktop 常可直接用。
在 ES 2.0 常需要 `GL_EXT_unpack_subimage`。
KrKr2 用 `GL_CHECK_unpack_subimage` 做检测并回退。

## 十、常见错误与解决方案

1. **纹理黑屏**：mipmap 过滤已开但未生成 mip 链。解决：生成 mipmap 或改 `MIN_FILTER=GL_LINEAR`。
2. **纹理翻转**：UV 原点与图片原点不一致。解决：统一翻转策略（资源层/顶点层/shader层）。
3. **花屏错行**：`GL_UNPACK_ALIGNMENT` 或 `GL_UNPACK_ROW_LENGTH` 设置错误。解决：先用 alignment=1 验证，再按 pitch 精确配置。
4. **ES2 的 NPOT 异常**：REPEAT 或 mipmap 组合受限。解决：改 POT、改 CLAMP、关 mipmap。

## 动手实践

## 本节小结

- 纹理是 GPU 对象，采样是 shader 读取流程。
- Wrap/Filter/Mipmap 共同决定采样结果。
- 多纹理核心在“纹理单元索引映射”，不是纹理 ID。
- `glPixelStorei` 与 `glTexSubImage2D` 决定动态更新稳定性。
- 跨平台必须预设 NPOT 与扩展回退路径。
