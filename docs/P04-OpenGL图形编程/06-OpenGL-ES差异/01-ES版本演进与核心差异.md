## 为什么需要了解 OpenGL ES？

**场景**：你在桌面上开发了一个 OpenGL 渲染器，一切运行正常。然后把代码移植到 Android 手机上——着色器编译失败、纹理显示异常、FBO 报不完整。这些问题都源于一个事实：**OpenGL ES 不是桌面 OpenGL 的"缩小版"，它是一套独立的标准**。

OpenGL ES（Embedded Systems，嵌入式系统）是 Khronos Group（OpenGL 标准的制定组织）专门为手机、平板、嵌入式设备设计的图形 API。它和桌面 OpenGL 共享很多概念，但在细节上有大量差异——而正是这些差异会让你的跨平台代码在移动端崩溃。

KrKr2 项目需要同时在 Windows（桌面 OpenGL）和 Android（OpenGL ES）上运行，所以理解这些差异是必须的。

## 一、版本演进：ES 1.1 → ES 2.0 → ES 3.0 → ES 3.2

### 1.1 ES 1.1——固定功能时代

OpenGL ES 1.1 发布于 2004 年，对应桌面 OpenGL 1.5 时代。它只提供**固定功能管线**（Fixed-Function Pipeline）——开发者不能编写着色器，只能通过一组预定义的 GL 函数来配置光照、变换、纹理混合等效果。

```
固定功能管线的工作方式：

开发者 → 调用 glLightfv() 设光源参数
       → 调用 glMaterialfv() 设材质参数
       → 调用 glMatrixMode() + glLoadMatrixf() 设变换矩阵
       → GPU 按固定的光照公式 + 纹理混合规则渲染

你无法修改 GPU 的计算逻辑，只能调参数
```

优点：简单直接，不需要写着色器代码。缺点：灵活性极低，无法实现自定义光照模型或特效。

> **现在还需要关心 ES 1.1 吗？** 基本不需要。Android 4.3+（2013 年）的设备都至少支持 ES 2.0。KrKr2 项目也不使用 ES 1.1 API。但了解它有助于理解"为什么 ES 2.0 是个重大转折"。

### 1.2 ES 2.0——可编程管线的转折点

OpenGL ES 2.0 发布于 2007 年，是移动图形的分水岭。它**移除了几乎所有固定功能接口**，要求开发者必须编写顶点着色器和片段着色器（Fragment Shader，也叫片元着色器——两者含义完全相同，本教程统一用"片段着色器"）。

ES 2.0 移除的桌面 OpenGL 功能：
- `glBegin` / `glEnd`（立即模式绘制，Immediate Mode——每个顶点一次函数调用，效率极低）
- `glMatrixMode` / `glLoadIdentity` / `glTranslatef`（固定矩阵栈——GPU 内置的模型/视图/投影矩阵管理）
- `glLightfv` / `glMaterialfv`（固定光照——GPU 内置的 Phong 光照模型）
- `glColor3f` / `glNormal3f`（逐顶点立即属性——每调一次 GL 函数设置一个顶点的颜色/法线）

这意味着：**如果你不写着色器，ES 2.0 什么都画不出来**。

ES 2.0 使用 GLSL ES 1.00（着色器语言版本），语法和桌面 GLSL 有显著差异（后面详细讲）。

### 1.3 ES 3.0——工程化能力提升

OpenGL ES 3.0 发布于 2012 年，使用 GLSL ES 3.00。它没有改变"可编程管线"的基本模型，但补齐了大量桌面 OpenGL 早已拥有的实用功能：

- **UBO**（Uniform Buffer Object，统一缓冲对象）：把多个 uniform 变量打包到一个缓冲区中，一次绑定即可，避免逐个调用 `glUniform*` 的开销
- **MRT**（Multiple Render Targets，多渲染目标）：一次渲染可以同时输出到多张纹理，是延迟渲染（Deferred Rendering，先把几何信息画到多张纹理上，再统一计算光照，适合大量光源场景）的基础
- **Instancing**（实例化渲染）：一次 draw call（绘制调用——CPU 发给 GPU 的一次绘制命令）绘制同一个模型的多个副本（如草地上的 1000 棵草），每个副本可以有不同的位置/颜色/大小
- **Transform Feedback**（变换反馈）：捕获顶点着色器的输出并写回缓冲区，可以在 GPU 上做粒子模拟等迭代计算
- **VAO**（Vertex Array Object，顶点数组对象）：ES 2.0 的 VAO 是可选扩展，ES 3.0 将其升级为核心功能

### 1.4 ES 3.2——高级特性标准化

OpenGL ES 3.2 发布于 2015 年，使用 GLSL ES 3.20。它的主要工作是**把之前以扩展形式存在的高级特性纳入核心规范**：

- 几何着色器（Geometry Shader，在顶点着色器之后、片段着色器之前执行，可以增删顶点，常用于粒子特效和线框渲染）
- 曲面细分着色器（Tessellation Shader，把粗糙的几何体在 GPU 上细分为更精细的网格）
- 计算着色器（Compute Shader，通用 GPU 计算，不直接参与渲染管线）

> KrKr2 作为 2D 视觉小说引擎，不需要几何着色器或曲面细分。它的 Android 版主要使用 ES 2.0 API（兼容最广泛的设备），仅在可用时启用部分 ES 3.0 特性。

### 1.5 版本对照总表

| 版本 | 发布年份 | 着色器语言 | 管线模型 | 关键新增能力 |
|------|----------|-----------|----------|-------------|
| ES 1.1 | 2004 | 无 GLSL | 固定功能 | 固定变换、固定光照、多纹理混合 |
| ES 2.0 | 2007 | GLSL ES 1.00 | 可编程 | 顶点/片段着色器、FBO（扩展）、VBO 必须 |
| ES 3.0 | 2012 | GLSL ES 3.00 | 可编程增强 | UBO、MRT、Instancing、VAO 核心化、`in`/`out` 语法 |
| ES 3.2 | 2015 | GLSL ES 3.20 | 高级可编程 | 几何/曲面细分/计算着色器进入核心 |

## 二、ES 2.0 vs 桌面 OpenGL 3.x Core：关键差异

很多开发者的第一反应是"ES 2.0 应该和桌面 OpenGL 差不多吧"。实际上，差异之多可能超出你的预期。以下按类别逐一拆解。

### 2.1 GLSL 语法差异

这是跨平台开发中**最高频的坑**。同一个着色器，桌面能编译，ES 2.0 就编译失败。

| 语法元素 | 桌面 GLSL 330 core | GLSL ES 1.00 (ES 2.0) | GLSL ES 3.00 (ES 3.0) |
|----------|--------------------|-----------------------|-----------------------|
| 版本声明 | `#version 330 core` | `#version 100` | `#version 300 es` |
| 顶点输入 | `in vec3 aPos;` | `attribute vec3 aPos;` | `in vec3 aPos;` |
| 跨阶段传值 | `out`(顶点) / `in`(片段) | `varying` | `out` / `in` |
| 片段输出 | `out vec4 FragColor;` | 写入 `gl_FragColor` | `out vec4 FragColor;` |
| 纹理采样 | `texture(tex, uv)` | `texture2D(tex, uv)` | `texture(tex, uv)` |
| 精度声明 | 可选（通常不写） | **片段着色器必须**声明 `precision` | **片段着色器必须**声明 `precision` |

完整示例——同一个"纹理采样"着色器的三个版本：

```glsl
// ===== 桌面 OpenGL 3.3 =====
#version 330 core
in vec2 vUV;
uniform sampler2D uTex;
out vec4 FragColor;
void main() {
    FragColor = texture(uTex, vUV);
}

// ===== OpenGL ES 2.0 =====
#version 100
precision mediump float;   // 必须！不写就编译失败
varying vec2 vUV;          // varying 而不是 in
uniform sampler2D uTex;
void main() {
    gl_FragColor = texture2D(uTex, vUV); // texture2D 而不是 texture
}

// ===== OpenGL ES 3.0 =====
#version 300 es
precision mediump float;   // 仍然必须
in vec2 vUV;               // 和桌面一样用 in
uniform sampler2D uTex;
out vec4 FragColor;        // 和桌面一样用 out
void main() {
    FragColor = texture(uTex, vUV); // texture 和桌面一样
}
```

> **`precision mediump float;`** 是 ES 的精度限定符（Precision Qualifier）声明。它告诉 GPU 这个着色器中所有没有单独指定精度的 `float` 变量都使用中精度（16 位浮点）。桌面 GPU 的片段着色器默认使用高精度（32 位），所以不需要声明。下一节会详细讲精度限定符。

### 2.2 NPOT 纹理限制

NPOT 是 Non-Power-Of-Two 的缩写，意思是**纹理宽高不是 2 的幂**。例如 800×600 是 NPOT，而 1024×512 是 POT（Power-Of-Two）。

桌面 OpenGL 2.0+ 完全支持 NPOT 纹理——你可以用任意宽高、任意过滤模式、任意环绕模式。但 **ES 2.0 对 NPOT 纹理有严格限制**：

| 操作 | POT 纹理 (如 512×512) | NPOT 纹理 (如 800×600) |
|------|------------------------|-------------------------|
| `GL_REPEAT` 环绕 | ✅ | ❌ 必须用 `GL_CLAMP_TO_EDGE` |
| Mipmap（一组逐级缩小的纹理副本，GPU 根据物体远近自动选择合适的级别，避免远处纹理闪烁） | ✅ | ❌ 不支持 |
| `GL_LINEAR_MIPMAP_LINEAR` 过滤 | ✅ | ❌ 只能用 `GL_LINEAR` 或 `GL_NEAREST` |

如果你在 ES 2.0 上对 NPOT 纹理设了 `GL_REPEAT` 或 `MIPMAP` 过滤，**纹理会变成全黑或产生未定义行为**。

KrKr2 的做法——保守安全的 NPOT 兼容策略：

```cpp
// KrKr2 中对所有纹理使用安全的默认参数
// 文件：cpp/core/visual/ogl/RenderManager_ogl.cpp

void SetupTextureParams(GLuint tex, bool isPOT) {
    glBindTexture(GL_TEXTURE_2D, tex);
    // 环绕模式统一用 CLAMP_TO_EDGE（POT 和 NPOT 都安全）
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, GL_CLAMP_TO_EDGE);
    // glTexParameteri：设置纹理参数
    //   GL_TEXTURE_WRAP_S：水平方向（S 轴）的环绕模式
    //   GL_CLAMP_TO_EDGE：超出 [0,1] 范围的坐标取边缘像素颜色
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, GL_CLAMP_TO_EDGE);
    //   GL_TEXTURE_WRAP_T：垂直方向（T 轴）的环绕模式
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_LINEAR);
    //   GL_TEXTURE_MIN_FILTER：缩小过滤模式
    //   GL_LINEAR：双线性过滤，取周围 4 个像素的平均值
    glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_LINEAR);
    //   GL_TEXTURE_MAG_FILTER：放大过滤模式
}
```

> **为什么 KrKr2 不用 POT 尺寸？** 视觉小说引擎处理的图像（立绘、背景、UI）尺寸各不相同，强制 POT 会浪费大量显存（如 800×600 的图片要扩展到 1024×1024）。所以 KrKr2 选择使用 NPOT 纹理 + 保守的采样参数。

### 2.3 FBO 差异

上一章已经详细讲过 FBO，这里总结 ES 2.0 和桌面 OpenGL 的关键区别：

| 特性 | 桌面 OpenGL 3.3 | ES 2.0 |
|------|-----------------|--------|
| FBO 是否核心 | ✅ 核心功能 | ⚠️ 通过 `OES_framebuffer_object` 扩展提供（几乎所有 ES 2.0 设备都支持） |
| 深度+模板合并 | ✅ `GL_DEPTH24_STENCIL8` | ❌ 必须分开创建 |
| MRT | ✅ 最多 8 个颜色附件 | ❌ 只有 1 个颜色附件 |
| `glBlitFramebuffer` | ✅ 核心 | ❌ 不支持（需 `NV_framebuffer_blit` 扩展） |
| 附件尺寸不一致 | ✅ 取最小值 | ❌ 所有附件必须相同尺寸 |

### 2.4 扩展命名体系

OpenGL/ES 的很多功能最初以**扩展**（Extension）形式提供，经过验证后才可能纳入下一版核心规范。扩展名的前缀告诉你它的"标准化程度"：

| 前缀 | 含义 | 示例 | 可靠性 |
|------|------|------|--------|
| `GL_KHR_*` | Khronos 官方批准的跨厂商扩展 | `GL_KHR_debug` | ⭐⭐⭐ 最高 |
| `GL_EXT_*` | 多厂商支持的扩展 | `GL_EXT_shader_framebuffer_fetch` | ⭐⭐ 较高 |
| `GL_OES_*` | OpenGL ES 专用扩展 | `GL_OES_texture_float` | ⭐⭐ 较高 |
| `GL_ARB_*` | OpenGL 架构审查委员会扩展（桌面） | `GL_ARB_copy_image` | ⭐⭐ 较高 |
| `GL_NV_*` / `GL_AMD_*` / `GL_ARM_*` / `GL_QCOM_*` | 单厂商专有扩展 | `GL_NV_shader_framebuffer_fetch` | ⭐ 仅限该厂商 |

KrKr2 检查的扩展列表（`RenderManager_ogl.cpp` 第 84-101 行）：

```cpp
// KrKr2 关心的 GL 扩展
struct _UsedGLExtInfo {
    DEFEXT(EXT_unpack_subimage);         // 纹理上传时支持行间距
    DEFEXT(EXT_shader_framebuffer_fetch); // 标准版：片段着色器直接读帧缓冲
    DEFEXT(ARM_shader_framebuffer_fetch); // ARM Mali GPU 专用版
    DEFEXT(NV_shader_framebuffer_fetch);  // NVIDIA Tegra 专用版
    DEFEXT(EXT_copy_image);              // 标准版：GPU 间纹理拷贝
    DEFEXT(OES_copy_image);              // ES 版：GPU 间纹理拷贝
    DEFEXT(ARB_copy_image);              // 桌面版：GPU 间纹理拷贝
    DEFEXT(NV_copy_image);               // NVIDIA 专用版
    DEFEXT(EXT_clear_texture);           // 标准版：快速清除纹理
    DEFEXT(ARB_clear_texture);           // 桌面版：快速清除纹理
    DEFEXT(QCOM_alpha_test);             // 高通 Adreno：硬件 Alpha 测试
};
```

注意 `shader_framebuffer_fetch` 同一个功能有 3 个扩展名——这就是为什么需要逐一检查。不同 GPU 厂商实现同一功能时使用不同的扩展名。

### 2.5 上下文创建差异

在桌面平台，创建 OpenGL 上下文使用平台专用 API：
- **Windows**：WGL（Windows GL，需要先创建一个"哑"上下文加载函数指针，再创建真正的上下文——前面章节讲过）
- **Linux**：GLX（X11 GL）或 EGL
- **macOS**：CGL（Core GL）或 NSGL

在移动平台（Android/iOS），创建 OpenGL ES 上下文使用 **EGL**（Embedded-System Graphics Library，嵌入式图形库）。EGL 是 Khronos 标准化的、平台无关的上下文管理 API。

```
桌面上下文创建流程（以 Windows WGL 为例）：
  CreateWindow → GetDC → ChoosePixelFormat → SetPixelFormat
  → wglCreateContext → wglMakeCurrent

移动上下文创建流程（EGL）：
  eglGetDisplay → eglInitialize → eglChooseConfig
  → eglCreateWindowSurface → eglCreateContext → eglMakeCurrent
```

KrKr2 通过 Cocos2d-x 框架隐藏了这些差异——框架在每个平台上使用对应的 API 创建上下文，引擎代码不需要直接调用 WGL 或 EGL。

## 三、对照项目源码

KrKr2 的 GL 扩展处理代码展示了一个真实项目如何应对跨平台差异。

### 扩展检查机制（第 104-147 行）

```cpp
// 文件：cpp/core/visual/ogl/RenderManager_ogl.cpp

// 存储当前设备支持的所有 GL 扩展名
static std::unordered_set<std::string> sTVPGLExtensions;
// 常用扩展的快速检查标记
static bool GL_CHECK_unpack_subimage;
static bool GL_CHECK_shader_framebuffer_fetch;

// 检查某个扩展是否可用
bool TVPCheckGLExtension(const std::string &extname) {
    return sTVPGLExtensions.find(extname) != sTVPGLExtensions.end();
}

static void TVPInitGLExtensionInfo() {
    // 1. 从 GPU 驱动获取所有支持的扩展
    std::string gl_extensions = (const char*)glGetString(GL_EXTENSIONS);
    // glGetString(GL_EXTENSIONS)：返回一个空格分隔的扩展名列表
    //   例如 "GL_EXT_unpack_subimage GL_OES_texture_float ..."
    //   注意：OpenGL 3.0+ 推荐用 glGetStringi 逐个查询

    // 2. 按空格拆分成集合
    // ... (解析代码省略，把每个扩展名加入 sTVPGLExtensions)

    // 3. 允许通过配置禁用特定扩展（用于调试或规避驱动 bug）
    IndividualConfigManager *cfgMgr = IndividualConfigManager::GetInstance();
    for (const char* const* name = ...; *name; ++name) {
        if (!cfgMgr->GetValue<int>(*name, 1)) {
            sTVPGLExtensions.erase(*name);
        }
    }

    // 4. Windows 上强制禁用 unpack_subimage（该扩展是 ES 专用的）
    #ifdef WIN32
    sTVPGLExtensions.erase("GL_EXT_unpack_subimage");
    #endif
}
```

**设计要点**：
1. **运行时检测**：不假设任何扩展一定存在，用 `unordered_set` 做 O(1) 查询
2. **配置开关**：通过 `IndividualConfigManager` 允许用户/开发者禁用特定扩展（规避某些 GPU 驱动的 bug）
3. **平台条件编译**：Windows 桌面 OpenGL 不需要 `EXT_unpack_subimage`（这是 ES 扩展），直接移除

### 函数指针动态加载（第 149-187 行）

在桌面 OpenGL 上，很多新版本的 GL 函数需要在运行时动态加载：

```cpp
// 不同平台获取 GL 函数指针的方式不同
#ifdef _MSC_VER
    typedef PROC(WINAPI fGetProcAddress)(LPCSTR);      // Windows: wglGetProcAddress
#elif defined(__ANDROID__)
    typedef void*(EGLAPIENTRY fGetProcAddress)(const char*); // Android: eglGetProcAddress
#elif defined(LINUX)
    typedef void*(GLAPIENTRY fGetProcAddress)(const char*);  // Linux: glXGetProcAddress
#endif

// 动态加载的函数指针
static fCopyImageSubData* glCopyImageSubData;   // GPU 间纹理拷贝
static fClearTexImage* glClearTexImage;           // 快速清除纹理
```

**为什么需要动态加载？** 因为这些函数是通过扩展提供的，不一定在所有设备上可用。在使用前必须先通过 `GetProcAddress` 获取函数指针，如果返回 `nullptr` 说明当前设备不支持，就使用回退方案（Fallback）。

## 四、动手实践

### 实践：编写跨 ES/Desktop 的着色器加载器

下面是一个实用的着色器工具函数，能自动根据平台生成正确的着色器源码：

```cpp
#include <string>
#include <cstdio>

// 检测当前 GL 上下文是 ES 还是桌面
bool isOpenGLES() {
    const char* version = (const char*)glGetString(GL_VERSION);
    // glGetString(GL_VERSION)：返回 GL 版本字符串
    //   桌面：如 "4.6.0 NVIDIA 537.58"
    //   ES：如 "OpenGL ES 3.0 V@..."
    return (version && strstr(version, "OpenGL ES") != nullptr);
}

// 根据平台生成兼容的片段着色器头部
std::string getFragmentShaderHeader() {
    if (isOpenGLES()) {
        return "#version 100\n"
               "precision mediump float;\n"
               "#define TEXTURE2D texture2D\n"
               "#define FRAG_COLOR gl_FragColor\n"
               "#define IN varying\n";
    } else {
        return "#version 330 core\n"
               "#define TEXTURE2D texture\n"
               "#define FRAG_COLOR FragColor\n"
               "#define IN in\n"
               "out vec4 FragColor;\n";
    }
}

// 使用示例：着色器源码用宏写，编译时自动适配平台
const char* fragBody = R"(
IN vec2 vUV;
uniform sampler2D uTex;
void main() {
    FRAG_COLOR = TEXTURE2D(uTex, vUV);
}
)";

// 拼接完整着色器源码
std::string fullShaderSrc = getFragmentShaderHeader() + fragBody;
```

这个方案的好处是着色器逻辑只写一份，通过 `#define` 宏适配不同平台的语法差异。

### 实践：运行时查询 GL 能力

```cpp
#include <cstdio>

void printGLCapabilities() {
    // 获取基本信息
    printf("GL 厂商：%s\n", glGetString(GL_VENDOR));
    // glGetString(GL_VENDOR)：GPU 厂商名（如 "NVIDIA Corporation"）
    printf("GL 渲染器：%s\n", glGetString(GL_RENDERER));
    // glGetString(GL_RENDERER)：GPU 型号（如 "GeForce RTX 3080"）
    printf("GL 版本：%s\n", glGetString(GL_VERSION));
    printf("GLSL 版本：%s\n", glGetString(GL_SHADING_LANGUAGE_VERSION));
    // glGetString(GL_SHADING_LANGUAGE_VERSION)：GLSL 版本

    // 查询关键限制
    GLint maxTexSize, maxTexUnits, maxVaryings, maxVertAttribs;
    glGetIntegerv(GL_MAX_TEXTURE_SIZE, &maxTexSize);
    // glGetIntegerv：查询 GL 整数状态值
    //   GL_MAX_TEXTURE_SIZE：支持的最大纹理尺寸（通常 4096-16384）
    glGetIntegerv(GL_MAX_TEXTURE_IMAGE_UNITS, &maxTexUnits);
    //   GL_MAX_TEXTURE_IMAGE_UNITS：片段着色器最多可用的纹理单元数
    glGetIntegerv(GL_MAX_VARYING_VECTORS, &maxVaryings);
    //   GL_MAX_VARYING_VECTORS：顶点→片段最多可传的 varying/out 向量数
    glGetIntegerv(GL_MAX_VERTEX_ATTRIBS, &maxVertAttribs);
    //   GL_MAX_VERTEX_ATTRIBS：顶点着色器最多可用的属性数

    printf("最大纹理尺寸：%d\n", maxTexSize);
    printf("最大纹理单元数：%d\n", maxTexUnits);
    printf("最大 varying 向量数：%d\n", maxVaryings);
    printf("最大顶点属性数：%d\n", maxVertAttribs);
}
```

典型输出对比：

| 参数 | 桌面 GPU (RTX 3080) | 移动 GPU (Adreno 640) |
|------|---------------------|------------------------|
| `GL_MAX_TEXTURE_SIZE` | 32768 | 16384 |
| `GL_MAX_TEXTURE_IMAGE_UNITS` | 32 | 16 |
| `GL_MAX_VARYING_VECTORS` | 32 | 15 |
| `GL_MAX_VERTEX_ATTRIBS` | 16 | 16 |

> **重要**：不要假设这些值。始终在运行时查询。尤其是 `GL_MAX_VARYING_VECTORS`——如果你的着色器用了 16 个 varying 但设备只支持 15 个，着色器会链接失败。

## 常见错误及解决方案

### 错误一：ES 2.0 片段着色器编译失败——缺少 precision

```
ERROR: 0:1: '' : No precision specified for (float)
```

**原因**：GLSL ES 的片段着色器必须有默认精度声明，桌面 GLSL 不要求。

**修复**：在片段着色器开头加 `precision mediump float;`。

### 错误二：NPOT 纹理在 ES 2.0 上显示全黑

**原因**：NPOT 纹理使用了 `GL_REPEAT` 环绕或 `MIPMAP` 过滤。

**修复**：所有 NPOT 纹理使用 `GL_CLAMP_TO_EDGE` + `GL_LINEAR`。

### 错误三：`texture()` 函数在 ES 2.0 上编译失败

**原因**：ES 2.0 的纹理采样函数叫 `texture2D`，不叫 `texture`。

**修复**：用 `#define` 宏或条件编译。

## 本节小结

- OpenGL ES 不是桌面 OpenGL 的子集，它是一套独立标准，面向低功耗嵌入式设备
- ES 1.1（固定功能）→ ES 2.0（可编程管线转折）→ ES 3.0（工程能力补齐）→ ES 3.2（高级特性标准化）
- ES 2.0 和桌面 OpenGL 3.x 的最常见差异：精度声明、GLSL 语法（`attribute`/`varying` vs `in`/`out`）、NPOT 限制、FBO 限制
- NPOT 纹理在 ES 2.0 上必须用 `GL_CLAMP_TO_EDGE` + `GL_LINEAR`，不能用 `GL_REPEAT` 和 mipmap
- GL 扩展名前缀代表标准化程度：`KHR` > `EXT`/`OES`/`ARB` > 厂商专有
- KrKr2 通过运行时检测扩展 + 配置开关 + 条件编译来处理跨平台差异

## 练习题与答案

### 题目 1：版本识别

以下着色器代码是为哪个 GL 版本编写的？说出理由。

```glsl
precision highp float;
attribute vec3 aPosition;
varying vec2 vTexCoord;
uniform mat4 uMVP;
void main() {
    gl_Position = uMVP * vec4(aPosition, 1.0);
}
```

<details>
<summary>查看答案</summary>

**OpenGL ES 2.0**（GLSL ES 1.00）。

理由：
1. 使用 `attribute` 而不是 `in`——ES 2.0 / GLSL ES 1.00 专用关键字
2. 使用 `varying` 而不是 `out`——同上
3. 有 `precision highp float;` 声明——ES 风格

注意：虽然没有写 `#version 100`，但 ES 2.0 的 GLSL 默认就是 version 100。不过良好实践是始终显式写出版本声明。

</details>

### 题目 2：NPOT 纹理修复

以下代码在桌面 OpenGL 上正常显示一个 800×600 的纹理，但在 Android ES 2.0 设备上纹理全黑。找出问题并修复。

```cpp
glBindTexture(GL_TEXTURE_2D, texture);
glTexImage2D(GL_TEXTURE_2D, 0, GL_RGBA, 800, 600, 0,
             GL_RGBA, GL_UNSIGNED_BYTE, pixelData);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, GL_REPEAT);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, GL_REPEAT);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER,
                GL_LINEAR_MIPMAP_LINEAR);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_LINEAR);
glGenerateMipmap(GL_TEXTURE_2D);
```

<details>
<summary>查看答案</summary>

**问题**：800×600 是 NPOT 尺寸，在 ES 2.0 上有三个违规：
1. `GL_REPEAT` 环绕——NPOT 纹理不支持
2. `GL_LINEAR_MIPMAP_LINEAR` 过滤——NPOT 纹理不支持 mipmap
3. `glGenerateMipmap`——NPOT 纹理调用此函数无效或报错

**修复**：

```cpp
glBindTexture(GL_TEXTURE_2D, texture);
glTexImage2D(GL_TEXTURE_2D, 0, GL_RGBA, 800, 600, 0,
             GL_RGBA, GL_UNSIGNED_BYTE, pixelData);
// 修复 1：NPOT 必须用 CLAMP_TO_EDGE
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, GL_CLAMP_TO_EDGE);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, GL_CLAMP_TO_EDGE);
// 修复 2：NPOT 不能用 MIPMAP 过滤
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_LINEAR);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_LINEAR);
// 修复 3：移除 glGenerateMipmap 调用
```

</details>

### 题目 3：扩展检查

KrKr2 为同一个功能（`shader_framebuffer_fetch`）检查了 3 个不同的扩展名。为什么？如果你要使用这个功能，应该怎样编写检查代码？

<details>
<summary>查看答案</summary>

**为什么 3 个扩展名？** 因为不同 GPU 厂商以不同的扩展名提供同一功能：
- `GL_EXT_shader_framebuffer_fetch`：标准扩展名（多数现代 GPU）
- `GL_ARM_shader_framebuffer_fetch`：ARM Mali GPU 的早期实现
- `GL_NV_shader_framebuffer_fetch`：NVIDIA Tegra GPU 的实现

这是 OpenGL 扩展体系的一个常见问题——同一功能在标准化之前，各厂商各自实现，用不同的扩展名。

**检查代码**：

```cpp
bool hasFramebufferFetch() {
    // 按优先级检查：标准扩展 > 厂商专有
    return TVPCheckGLExtension("GL_EXT_shader_framebuffer_fetch")
        || TVPCheckGLExtension("GL_ARM_shader_framebuffer_fetch")
        || TVPCheckGLExtension("GL_NV_shader_framebuffer_fetch");
}
```

注意：虽然功能相同，但不同扩展可能在着色器中的语法略有不同（如 `inout` 变量的声明方式），需要分别处理。

</details>
