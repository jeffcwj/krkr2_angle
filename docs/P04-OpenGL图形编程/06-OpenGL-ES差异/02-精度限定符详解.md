## 一、什么是精度限定符（Precision Qualifier）？

### 1.1 问题引入

上一节提到 ES 2.0 片段着色器必须写 `precision mediump float;`，否则编译失败。你可能会问：为什么桌面 OpenGL 不需要这个声明？这个 `mediump` 到底是什么意思？为什么不直接用最高精度？

答案要从 GPU 硬件说起。

**桌面 GPU**（如 NVIDIA GeForce、AMD Radeon）主要目标是高性能和高画质，电力来自墙上插座，芯片面积和功耗不是首要限制。所以桌面 GPU 的着色器单元通常只支持一种浮点精度——32 位（IEEE 754 单精度）。不管你在着色器里怎么写，计算都是 32 位精度，声不声明精度没有区别。

**移动 GPU**（如 ARM Mali、Qualcomm Adreno、Apple GPU、Imagination PowerVR）靠电池供电，必须严格控制功耗。GPU 的功耗和每次浮点运算的位数直接相关——32 位运算的功耗大约是 16 位的 4 倍。所以移动 GPU 的着色器单元支持多种精度：32 位（高精度）、16 位（中精度）、甚至 10 位（低精度）。开发者通过精度限定符告诉 GPU："这个变量用 16 位就够了"，GPU 就能用更省电的运算单元来处理。

### 1.2 精度限定符的定义

精度限定符（Precision Qualifier）是 GLSL ES 引入的一种语法修饰符，放在变量类型前面，告诉 GPU 这个变量应该用多少位精度来存储和计算。GLSL ES 定义了三个精度级别：

```glsl
highp float position;    // 高精度：至少 32 位浮点
mediump float color;     // 中精度：至少 16 位浮点
lowp float flag;         // 低精度：至少 10 位浮点（通常 8-10 位）
```

注意"至少"两个字——规范只定义了最低位数要求，实际实现可以更高。比如有些移动 GPU 的 `mediump` 实际上也是 32 位（和 `highp` 一样），但你不能依赖这个行为。

### 1.3 默认精度声明

除了逐变量指定精度，还可以用 `precision` 关键字设置整个着色器的默认精度：

```glsl
precision mediump float;  // 本着色器中所有没有单独指定精度的 float 都用 mediump
```

这条声明的作用范围是**整个着色器文件**。声明之后，你写 `float x = 1.0;` 等价于 `mediump float x = 1.0;`。

**关键规则**：
- 在 GLSL ES 中，**片段着色器（Fragment Shader）的 `float` 类型没有默认精度**。如果你不声明 `precision mediump float;`，又使用了没有精度修饰的 `float` 变量，编译器会报错
- 顶点着色器（Vertex Shader）的 `float` 和 `int` 默认是 `highp`——因为顶点位置计算需要高精度
- 所有着色器中，`int` 默认精度是 `highp`（ES 3.0+）或 `mediump`（ES 2.0）

## 二、三级精度的具体含义

### 2.1 位数、范围与精度对照表

GLSL ES 规范定义了每个精度级别的**最低保证**：

| 精度级别 | 浮点位数（最低） | 浮点范围（最低） | 浮点精度（最低） | 整数位数（最低） |
|----------|------------------|------------------|------------------|------------------|
| `highp` | 32 位（IEEE 754） | ±2^62 ≈ ±4.6×10^18 | 2^-16 ≈ 1.5×10^-5 | 32 位（含符号） |
| `mediump` | 16 位 | ±2^14 = ±16384 | 2^-10 ≈ 0.001 | 16 位（含符号） |
| `lowp` | 10 位 | ±2.0 | 2^-8 ≈ 0.004 | 9 位（含符号） |

这些数字意味着什么？我们用具体例子说明。

### 2.2 highp——高精度（32 位浮点）

```
highp 的精度：约 7 位十进制有效数字
范围：约 ±3.4×10^38

能精确表示的值示例：
  1.0000001（第 7 位小数有效）
  123456.7（大数也能保持精度）
  0.000001（极小值也能表示）
```

**适用场景**：顶点位置、矩阵变换、深度值、UV 坐标（大型纹理或需要亚像素精度时）。

**代价**：在支持真正 16 位运算的移动 GPU 上，`highp` 的功耗是 `mediump` 的约 4 倍，ALU（Arithmetic Logic Unit，算术逻辑单元——GPU 中执行加减乘除等数学运算的硬件模块）吞吐量减半。

### 2.3 mediump——中精度（16 位浮点）

```
mediump 的精度：约 3 位十进制有效数字
范围：±16384

能精确表示的值示例：
  1.001（3 位小数精度足够）
  255.0（8 位颜色值完全精确）
  0.01（日常颜色/UV 计算够用）

无法精确表示的值示例：
  65536.0（超出范围，会变成 inf）
  1.0001（第 4 位小数丢失）
```

**适用场景**：颜色计算（RGBA 分量通常在 0.0-1.0 范围）、法线向量（分量在 -1.0 到 1.0 之间）、小范围的 UV 坐标。

**这是移动端最常用的精度级别**——在大多数情况下，颜色和纹理坐标用 `mediump` 就完全够了，而功耗和速度比 `highp` 好得多。

### 2.4 lowp——低精度（8-10 位浮点）

```
lowp 的精度：约 2 位十进制有效数字
范围：±2.0

能精确表示的值示例：
  0.5（颜色半值）
  1.0（满值）
  -1.0（法线分量）

无法精确表示的值示例：
  3.0（超出范围！）
  0.001（精度不够，会变成 0）
```

**适用场景**：布尔标志（0.0 或 1.0）、归一化的颜色分量（严格在 0.0-1.0 范围内）、简单的混合权重。

**注意**：`lowp` 在现代移动 GPU 上经常被实现为和 `mediump` 相同精度——因为硬件上 10 位和 16 位的运算单元成本差别不大。但你不能依赖这个行为。

### 2.5 精度选择决策图

```
                    这个变量需要多大范围和精度？
                    ┌──────────────┴──────────────┐
                    │                             │
              范围 > 16384                   范围 ≤ 16384
              或精度 > 0.001                     │
                    │                      ┌─────┴──────┐
                    ▼                      │            │
                 highp               范围 ≤ 2.0     范围 > 2.0
                                    精度 ≈ 0.01    精度 ≈ 0.001
                                    就够用          才够用
                                         │            │
                                         ▼            ▼
                                       lowp       mediump

典型变量→精度映射：
  顶点位置 (aPosition)      → highp   （世界坐标可能很大）
  MVP 矩阵 (uMVP)          → highp   （矩阵乘法需要高精度）
  深度值 (gl_FragCoord.z)   → highp   （深度缓冲精度敏感）
  纹理坐标 (vTexCoord)      → mediump （通常 0.0-1.0 范围）
  颜色值 (vColor)           → mediump （0.0-1.0 范围）
  法线向量 (vNormal)        → mediump （-1.0 到 1.0 范围）
  布尔开关 (uEnabled)       → lowp    （只用 0.0 或 1.0）
```

## 三、平台差异：桌面 vs 移动

### 3.1 桌面 OpenGL 的精度行为

桌面 GLSL（如 `#version 330 core`）有以下特点：

1. **精度限定符是可选的**——你可以写 `highp`/`mediump`/`lowp`，编译器接受但完全忽略。所有运算都是 32 位
2. **不需要默认精度声明**——`precision mediump float;` 在桌面上是合法但无效的语法
3. **`GL_ES` 宏不被定义**——桌面 GLSL 中 `#ifdef GL_ES` 判断为假

这意味着：在桌面上，精度限定符纯粹是"装饰"。不管你写 `lowp float` 还是 `highp float`，GPU 都用 32 位计算。

### 3.2 OpenGL ES 的精度行为

GLSL ES（`#version 100` 或 `#version 300 es`）的规则完全不同：

1. **片段着色器的 `float` 没有默认精度**——必须显式声明 `precision mediump float;` 或逐变量指定精度
2. **精度限定符影响真实的运算位数**——`mediump` 变量真的用 16 位计算（在支持的 GPU 上）
3. **`GL_ES` 宏被自动定义**——可以用于条件编译

### 3.3 为什么桌面不强制精度声明？

因为桌面 GPU 的设计目标是"最高画质"，不需要在精度和功耗之间做权衡。所有 ALU 都是 32 位的，声明 `lowp` 也不会省电或加速。强制开发者写精度声明只会增加代码冗余。

### 3.4 各平台 GPU 精度实现对比

| GPU 厂商/系列 | `highp` 实现 | `mediump` 实现 | `lowp` 实现 | 备注 |
|---------------|-------------|---------------|------------|------|
| NVIDIA GeForce (桌面) | 32 位 | 32 位 | 32 位 | 全部按 32 位处理 |
| AMD Radeon (桌面) | 32 位 | 32 位 | 32 位 | 全部按 32 位处理 |
| ARM Mali (G71+) | 32 位 | 16 位 | 16 位 | `mediump` 真的用 16 位 |
| ARM Mali (旧款) | 32 位 | 16 位 | 10-16 位 | `lowp` 可能真的更低 |
| Qualcomm Adreno 5xx+ | 32 位 | 16 位 | 16 位 | `lowp` = `mediump` |
| Apple GPU (A11+) | 32 位 | 16 位 | 16 位 | Metal 也有对应精度 |
| Imagination PowerVR | 32 位 | 16 位 | 10 位 | 少数真正区分三级精度 |

> **重要发现**：大多数现代移动 GPU 的 `lowp` 和 `mediump` 实现相同。但 ARM Mali 的旧款 GPU 和 Imagination PowerVR 可能真正区分。如果你的应用需要兼容 2015 年以前的低端 Android 设备，精度选择需要更谨慎。

## 四、精度对性能的影响

### 4.1 为什么精度影响性能？

移动 GPU 的着色器核心（Shader Core）通常有两种 ALU：

```
着色器核心内部结构（简化）：

┌─────────────────────────────────┐
│         着色器核心               │
│  ┌──────────────┐               │
│  │  FP32 ALU    │ ← 32 位浮点运算单元       │
│  │  (highp)     │   每周期处理 N 个运算       │
│  └──────────────┘               │
│  ┌──────────────┐               │
│  │  FP16 ALU    │ ← 16 位浮点运算单元       │
│  │  (mediump)   │   每周期处理 2N 个运算      │
│  └──────────────┘   （同样的硬件面积，吞吐量翻倍）│
└─────────────────────────────────┘
```

FP16 ALU 的优势：
- **吞吐量翻倍**：同样的芯片面积可以放两倍数量的 16 位运算单元
- **带宽减半**：16 位数据读写的带宽是 32 位的一半
- **功耗降低**：16 位乘法器的功耗约为 32 位的 1/4

### 4.2 性能对比实测数据

以下是典型移动 GPU 上的性能对比（数据来源：ARM Mali 和 Qualcomm Adreno 的公开基准测试）：

| 操作 | highp (FP32) | mediump (FP16) | 加速比 |
|------|-------------|---------------|--------|
| 向量乘法 (vec4 × vec4) | 1x | 2x | 2 倍 |
| 矩阵乘法 (mat4 × vec4) | 1x | 1.5-2x | 最高 2 倍 |
| 纹理采样 + 混合 | 1x | 1.2-1.5x | 最高 1.5 倍 |
| 寄存器压力 | 高 | 低 | 更少溢出 |

> **"寄存器压力"是什么？** GPU 着色器核心有固定数量的寄存器（Register，临时存储运算中间结果的超高速存储单元）。32 位变量占用的寄存器空间是 16 位的两倍。如果着色器用了太多 highp 变量，寄存器可能不够用，GPU 就不得不把中间结果暂存到显存再读回来——这叫"寄存器溢出"（Register Spilling），会严重拖慢速度。

### 4.3 代码示例 1：精度对颜色计算的影响

```glsl
// 文件：precision_color_demo.frag
// 演示：用 mediump 做颜色计算完全够用

#ifdef GL_ES
precision mediump float;  // ES：设置默认精度为 mediump
#endif

// mediump 的精度约为 0.001
// RGBA 颜色分量范围 0.0-1.0，最终映射到 8 位（0-255）
// 即使 mediump 有微小误差，量化到 8 位后完全看不出来

uniform sampler2D uTexture;  // 纹理采样器
varying vec2 vTexCoord;       // 纹理坐标（从顶点着色器传入）

void main() {
    // texture2D：ES 2.0 的纹理采样函数
    //   参数 1：纹理采样器
    //   参数 2：纹理坐标 (s, t)
    //   返回值：该坐标处的颜色 (r, g, b, a)
    vec4 texColor = texture2D(uTexture, vTexCoord);

    // 亮度调整：乘以 1.2 让颜色稍微变亮
    // mediump 下 1.2 * 0.8 = 0.959961（精确值 0.96）
    // 误差 0.000039，量化到 0-255 范围后完全不可见
    vec4 brightened = texColor * 1.2;

    // 确保不超过 1.0
    gl_FragColor = clamp(brightened, 0.0, 1.0);
    // clamp(x, min, max)：将 x 限制在 [min, max] 范围内
    //   小于 min 返回 min，大于 max 返回 max，否则返回 x
}
```

### 4.4 代码示例 2：精度不足导致的 bug

```glsl
// 文件：precision_bug_demo.frag
// 演示：mediump 在某些场景下精度不足

#ifdef GL_ES
precision mediump float;
#endif

uniform float uTime;  // 从 CPU 传入的时间值（秒）
varying vec2 vTexCoord;

void main() {
    // 问题场景：用时间值做动画
    // uTime 在程序运行 10 秒后 = 10.0
    // 运行 1 分钟后 = 60.0
    // 运行 1 小时后 = 3600.0

    // mediump 的精度约 0.001
    // 当 uTime = 3600.0 时，mediump 能表示的最小步长约为 3600 * 0.001 = 3.6
    // 这意味着 sin(uTime) 的输入跳跃式变化，动画会变成"卡顿"的

    float wave = sin(uTime * 3.14159);
    // ❌ 运行 1 小时后，wave 值的变化不再平滑！

    // 正确做法：对 uTime 取模或用 fract，保持值在小范围内
    float safeTime = mod(uTime, 6.28318);
    // mod(x, y)：返回 x 对 y 取模的结果
    //   让时间值始终在 0 ~ 2π 范围内循环
    float smoothWave = sin(safeTime);
    // ✅ safeTime 始终在 0-6.28 范围，mediump 精度完全够用

    gl_FragColor = vec4(vec3(smoothWave * 0.5 + 0.5), 1.0);
}
```

这个例子说明了一个重要原则：**不是所有变量都适合 `mediump`，关键是看变量的值域范围**。

### 4.5 代码示例 3：运行时精度信息查询

```cpp
// 文件：query_precision.cpp
// 查询当前 GPU 着色器精度信息

#include <cstdio>

void queryShaderPrecision() {
    // glGetShaderPrecisionFormat：查询着色器中指定类型的精度信息
    //   参数 1：着色器类型（GL_VERTEX_SHADER 或 GL_FRAGMENT_SHADER）
    //   参数 2：精度类型（GL_LOW_FLOAT, GL_MEDIUM_FLOAT, GL_HIGH_FLOAT,
    //           GL_LOW_INT, GL_MEDIUM_INT, GL_HIGH_INT）
    //   参数 3 (range)：输出，表示该精度能表示的范围 [2^range[0], 2^range[1])
    //   参数 4 (precision)：输出，表示精度位数

    GLint range[2], precision;

    // 查询片段着色器的 mediump float
    glGetShaderPrecisionFormat(GL_FRAGMENT_SHADER, GL_MEDIUM_FLOAT,
                               range, &precision);
    printf("片段着色器 mediump float:\n");
    printf("  范围: 2^%d 到 2^%d\n", range[0], range[1]);
    printf("  精度位数: %d\n", precision);

    // 查询片段着色器的 highp float
    glGetShaderPrecisionFormat(GL_FRAGMENT_SHADER, GL_HIGH_FLOAT,
                               range, &precision);
    printf("片段着色器 highp float:\n");
    printf("  范围: 2^%d 到 2^%d\n", range[0], range[1]);
    printf("  精度位数: %d\n", precision);

    // 查询片段着色器的 lowp float
    glGetShaderPrecisionFormat(GL_FRAGMENT_SHADER, GL_LOW_FLOAT,
                               range, &precision);
    printf("片段着色器 lowp float:\n");
    printf("  范围: 2^%d 到 2^%d\n", range[0], range[1]);
    printf("  精度位数: %d\n", precision);
}

// 典型输出（ARM Mali G72）：
// 片段着色器 mediump float:
//   范围: 2^14 到 2^14
//   精度位数: 10
// 片段着色器 highp float:
//   范围: 2^127 到 2^127
//   精度位数: 23
// 片段着色器 lowp float:
//   范围: 2^14 到 2^14     ← lowp 和 mediump 范围相同（实现为同精度）
//   精度位数: 10
```

### 4.6 代码示例 4：精度测试着色器

```glsl
// 文件：precision_test.frag
// 测试 mediump 的实际精度——在不同设备上运行观察颜色

#ifdef GL_ES
precision mediump float;
#endif

void main() {
    // 测试 1：mediump 能否区分 1.0 和 1.001？
    mediump float a = 1.0;
    mediump float b = 1.001;
    // 如果 mediump 真的是 16 位，a 和 b 应该能区分
    // 如果 mediump 被实现为 32 位（桌面），当然能区分

    // 测试 2：mediump 能否区分 1000.0 和 1001.0？
    mediump float c = 1000.0;
    mediump float d = 1001.0;
    // 16 位浮点在 1000 附近的精度约为 1.0
    // 所以 c 和 d 在真正的 16 位 GPU 上可能无法区分！

    // 输出：如果 a==b，红色；如果 c==d，绿色；否则蓝色
    if (a == b) {
        gl_FragColor = vec4(1.0, 0.0, 0.0, 1.0); // 红——精度太低
    } else if (c == d) {
        gl_FragColor = vec4(0.0, 1.0, 0.0, 1.0); // 绿——中等精度（真 16 位）
    } else {
        gl_FragColor = vec4(0.0, 0.0, 1.0, 1.0); // 蓝——高精度（32 位伪装）
    }
}
```

在不同设备上运行这个着色器：
- 桌面 GPU → 蓝色（32 位，能区分两组数据）
- 真 FP16 移动 GPU → 绿色（能区分 1.0/1.001，但不能区分 1000/1001）

## 常见错误及解决方案

### 错误一：ES 2.0 片段着色器缺少默认精度声明

**错误信息**：
```
ERROR: 0:1: '' : No precision specified for (float)
```

**原因**：GLSL ES 片段着色器的 `float` 类型没有内置默认精度。如果你写了 `float x = 1.0;` 但没有声明 `precision mediump float;`（或者没有给 x 加精度修饰），编译器无法确定该用多少位来存储 x。

**修复**：在片段着色器开头加上默认精度声明：

```glsl
#ifdef GL_ES
precision mediump float;  // 推荐 mediump 作为默认
#endif
```

### 错误二：时间动画在移动端运行一段时间后卡顿

**现象**：基于 `uTime`（从程序启动开始累加的秒数）的动画，在桌面上始终流畅，但在移动端运行几分钟后变得不平滑。

**原因**：`uTime` 作为 `mediump float` 传入着色器。当 `uTime` 超过约 1024 时，16 位浮点的精度步长变成 1.0 左右，帧间差异被"量化"掉了。

**修复**：在 CPU 端对时间值取模后再传给着色器：

```cpp
// CPU 端：传入着色器的时间值保持在小范围
float fmodTime = fmod(elapsedSeconds, 2.0f * 3.14159f);
glUniform1f(uTimeLoc, fmodTime);
// glUniform1f(location, value)：设置一个 float 类型的 uniform 变量
//   location：uniform 变量位置（通过 glGetUniformLocation 获取）
//   value：要设置的值
```

### 错误三：`highp` 在某些 ES 2.0 设备的片段着色器中不可用

**现象**：在低端移动设备上，着色器中使用 `highp float` 导致编译失败。

**原因**：GLSL ES 2.0 规范不要求片段着色器支持 `highp`。部分旧款 GPU（尤其是 Mali-400 系列）的片段着色器确实不支持 highp。

**检测方法**：

```glsl
// 在着色器中检测 highp 支持
#ifdef GL_FRAGMENT_PRECISION_HIGH
    // 支持 highp
    precision highp float;
#else
    // 不支持 highp，降级到 mediump
    precision mediump float;
#endif
```

`GL_FRAGMENT_PRECISION_HIGH` 是 GLSL ES 内置宏——如果当前 GPU 的片段着色器支持 `highp`，这个宏就会被定义。

---

## 本节小结

- **精度限定符**（`highp`/`mediump`/`lowp`）是 GLSL ES 引入的语法，告诉 GPU 用多少位精度处理浮点数。桌面 GLSL 接受但忽略这些修饰符
- **三级精度**：`highp` ≥ 32 位（位置/矩阵），`mediump` ≥ 16 位（颜色/UV），`lowp` ≥ 10 位（标志/权重）
- ES 片段着色器的 `float` **没有默认精度**，必须显式声明 `precision mediump float;`
- 在移动 GPU 上，使用 `mediump` 比 `highp` **吞吐量提升最高 2 倍**，功耗大幅降低
- 精度不足的典型 bug：大数值丢失精度导致动画卡顿、颜色断层
- 用 `glGetShaderPrecisionFormat` 可以在运行时查询当前 GPU 的实际精度能力
- 部分旧款 ES 2.0 GPU 的片段着色器不支持 `highp`，需要用 `GL_FRAGMENT_PRECISION_HIGH` 宏检测

### 题目 1：精度选择

以下着色器变量应该用什么精度？说出理由。

```glsl
??? vec4 gl_Position;          // (a) 顶点位置
??? float brightness;          // (b) 亮度值（0.0-1.0）
??? mat4 uModelViewProjection; // (c) MVP 矩阵
??? float uAnimSpeed;          // (d) 动画速度（0.0-10.0）
??? vec4 vColor;               // (e) 颜色值（0.0-1.0）
```

<details>
<summary>查看答案</summary>

```glsl
highp   vec4 gl_Position;          // (a) highp——顶点位置在世界坐标中可能很大，
                                   //     且位置精度直接影响画面抖动
mediump float brightness;          // (b) mediump——0.0-1.0 范围，最终映射到
                                   //     8 位颜色通道，mediump 精度 0.001 足够
highp   mat4 uModelViewProjection; // (c) highp——矩阵乘法会放大误差，
                                   //     低精度会导致远处物体位置抖动
mediump float uAnimSpeed;          // (d) mediump——0.0-10.0 范围在 mediump
                                   //     的 ±16384 范围内，精度 0.01 足够
mediump vec4 vColor;               // (e) mediump——颜色值 0.0-1.0 范围，
                                   //     mediump 精度远超 8 位颜色通道的分辨率
```

注意：`gl_Position` 是 GLSL 内置变量，顶点着色器中默认就是 `highp`，不需要手动声明。这里只是用来说明精度选择的逻辑。

</details>

### 题目 2：精度 bug 诊断

一个 2D 游戏在移动设备上运行时，背景滚动动画在运行约 30 秒后开始出现明显的"跳跃感"（不是卡顿，帧率正常，但滚动不平滑）。着色器代码如下：

```glsl
precision mediump float;
uniform float uScrollX;  // 从 CPU 传入的水平滚动偏移，持续增长
varying vec2 vUV;
uniform sampler2D uBackground;
void main() {
    vec2 scrolledUV = vUV + vec2(uScrollX, 0.0);
    gl_FragColor = texture2D(uBackground, scrolledUV);
}
```

问题出在哪里？如何修复？

<details>
<summary>查看答案</summary>

**问题**：`uScrollX` 持续增长。30 秒后，假设滚动速度为每秒 100 像素、纹理宽度 512 像素，则 `uScrollX` ≈ 100×30/512 ≈ 5.86（归一化 UV 坐标）。

`mediump` 的精度约 0.001。当 `uScrollX` 增长到 5.86 时，精度步长约 0.006——对于 512 像素宽的纹理，这意味着每帧的 UV 偏移至少跳 0.006 × 512 ≈ 3 像素。这就是"跳跃感"的来源。

随着时间增长，精度会越来越差。运行 5 分钟后，`uScrollX` 可能达到约 60，精度步长约 0.06，每帧跳 30+ 像素——完全不可接受。

**修复方案**：在 CPU 端对 `uScrollX` 取模，让它始终在 0.0-1.0 范围内循环：

```cpp
// CPU 端
float scrollX = fmod(totalScrollX, 1.0f);
// fmod(x, y)：C 标准库浮点取模函数
//   保证返回值在 [0, 1.0) 范围内（当 x > 0, y > 0 时）
glUniform1f(uScrollXLoc, scrollX);
```

着色器不需要修改——因为纹理坐标超出 [0,1] 范围时，`GL_REPEAT` 环绕模式会自动取模（但要注意 ES 2.0 的 NPOT 纹理不能用 `GL_REPEAT`，需要手动取 `fract`）。

</details>

### 题目 3：highp 降级检测

编写一段 GLSL 代码，在片段着色器中安全地使用 `highp`——如果当前 GPU 不支持 `highp`，自动降级到 `mediump`。

<details>
<summary>查看答案</summary>

```glsl
// 使用 GL_FRAGMENT_PRECISION_HIGH 宏检测
#ifdef GL_FRAGMENT_PRECISION_HIGH
    precision highp float;    // GPU 支持 highp，使用高精度
    #define PRECISION_HIGH highp
#else
    precision mediump float;  // GPU 不支持 highp，降级到 mediump
    #define PRECISION_HIGH mediump
#endif

// 需要高精度的变量用 PRECISION_HIGH 修饰
uniform PRECISION_HIGH float uTime;
// 不需要高精度的变量直接使用默认精度
uniform sampler2D uTex;
varying vec2 vUV;

void main() {
    // uTime 使用可用的最高精度，避免大值精度丢失
    float t = mod(uTime, 6.28318);
    vec4 color = texture2D(uTex, vUV);
    color.rgb *= 0.5 + 0.5 * sin(t);
    gl_FragColor = color;
}
```

</details>
