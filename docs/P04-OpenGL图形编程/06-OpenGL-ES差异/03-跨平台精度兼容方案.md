## 一、核心矛盾

上一节介绍了精度限定符的含义和性能影响。但当你尝试在桌面和移动端共用同一份着色器代码时，会遇到一个根本性矛盾：

- **桌面 GLSL** 不认识 `precision` 关键字（`#version 330 core` 会报警告或直接忽略）
- **ES GLSL** 必须有 `precision mediump float;`，否则编译失败
- 你想用**一份着色器代码**同时在桌面和 ES 上运行

这就是跨平台精度兼容编码要解决的核心问题。下面介绍三种从简到繁的解决方案。

---

## 二、方案一：`#ifdef GL_ES` 条件编译

这是最常见、最简单的做法。利用 GLSL 编译器自动定义（或不定义）的 `GL_ES` 宏来区分平台：

```glsl
// 文件：cross_platform_shader.frag
// 方案一：#ifdef GL_ES 条件编译

#ifdef GL_ES
    // GL_ES 宏在 GLSL ES 编译器中自动定义
    // 桌面 GLSL 编译器不定义此宏
    precision mediump float;
#endif

// 下面的代码桌面和 ES 都能编译
varying vec2 vUV;           // ES 2.0 用 varying（从顶点着色器传入的插值变量）
uniform sampler2D uTex;     // uniform：从 CPU 传入的只读变量

void main() {
    gl_FragColor = texture2D(uTex, vUV);
    // texture2D(sampler, coord)：ES 2.0 纹理采样函数
    //   sampler：纹理采样器（绑定了一张纹理）
    //   coord：纹理坐标 (s, t)，范围通常 [0, 1]
    //   返回值：该坐标处的颜色 vec4(r, g, b, a)
}
```

**优点**：简单直接，一看就懂。
**缺点**：如果着色器同时需要兼容 ES 2.0 和 ES 3.0 以及桌面，`varying` / `in`、`texture2D` / `texture`、`gl_FragColor` / `out` 的差异需要更多条件编译，代码会变得很乱。

### 何时使用方案一

- 项目只需要兼容**桌面 + ES 2.0**（不需要 ES 3.0）
- 着色器数量少（< 10 个）
- 不需要对不同变量使用不同精度级别

## 三、方案二：精度宏头文件

定义一组宏，在桌面上把精度限定符替换为空字符串：

```glsl
// 文件：precision_compat.glsl
// 方案二：精度兼容宏（在所有着色器开头 #include 此文件）

#ifdef GL_ES
    precision mediump float;    // ES 默认中精度
    #define HP highp            // 高精度缩写宏
    #define MP mediump          // 中精度缩写宏
    #define LP lowp             // 低精度缩写宏
#else
    #define HP                  // 桌面上替换为空——没有精度效果
    #define MP
    #define LP
#endif
```

使用时：

```glsl
// 文件：effect.frag
// 假设 precision_compat.glsl 的内容已通过 CPU 拼接到此文件开头

HP vec2 uScrollOffset;     // 高精度——可能是大范围滚动偏移
MP vec4 vColor;            // 中精度——颜色值
LP float uAlphaThreshold;  // 低精度——只用于 0/1 判断

uniform sampler2D uTex;
varying vec2 vUV;

void main() {
    MP vec4 texColor = texture2D(uTex, vUV + uScrollOffset);
    if (texColor.a < uAlphaThreshold) {
        discard;  // 丢弃透明像素
        // discard：GLSL 内置关键字，在片段着色器中丢弃当前像素
        //   该像素不会写入帧缓冲（Framebuffer，存储最终渲染结果的显存区域），
        //   相当于"这个像素不画了"
    }
    gl_FragColor = texColor * vColor;
}
```

**优点**：可以对不同变量使用不同精度，代码可读性好。
**缺点**：GLSL ES 1.00 不支持 `#include`，所以不能真的用 `#include "precision_compat.glsl"`，需要在 CPU 端手动拼接。

## 四、方案三：CPU 端着色器源码拼接

这是大型引擎（包括 KrKr2/Cocos2d-x）使用的方案。不依赖 GLSL 的 `#include`（因为 GLSL ES 1.00 不支持），而是在 CPU 端用 C++ 字符串拼接着色器源码：

```cpp
// 文件：ShaderBuilder.cpp
// KrKr2/Cocos2d-x 的着色器拼接策略

#include <string>
#include <cstring>

// 着色器头部——平台自适应
const char* getShaderPreamble() {
    // 运行时检测当前 GL 是否是 ES
    const char* version = (const char*)glGetString(GL_VERSION);
    // glGetString(GL_VERSION)：返回当前 GL 实现的版本字符串
    //   参数：GL_VERSION（要查询的字符串类型）
    //   返回值：以 null 结尾的 ASCII 字符串
    //   桌面返回如 "4.6.0 NVIDIA 537.58"
    //   ES 返回如 "OpenGL ES 3.0 V@..."
    //   注意：必须在 GL 上下文创建后才能调用
    bool isES = (version && strstr(version, "OpenGL ES") != nullptr);

    if (isES) {
        return "#ifdef GL_ES\n"
               "precision mediump float;\n"
               "#endif\n";
    } else {
        return "";  // 桌面不需要精度声明
    }
}
```

### KrKr2 实际代码——YUVSprite.cpp 的着色器字符串

```cpp
// 文件：cpp/core/environ/cocos2d/YUVSprite.cpp（第 7-38 行）
// KrKr2 的 YUV 视频渲染着色器——C++ 字符串拼接方式

static const char* yuvFragShader =
    "#ifdef GL_ES\n"
    "precision lowp float;\n"     // 注意：用了 lowp 而不是常见的 mediump
    "#endif\n"
    "varying vec4 v_fragmentColor;\n"
    "varying vec2 v_texCoord;\n"
    "void main() {\n"
    "    vec3 yuv;\n"
    "    yuv.x = texture2D(CC_Texture0, v_texCoord).r - 0.0625;\n"
    "    yuv.y = texture2D(CC_Texture1, v_texCoord).r - 0.5;\n"
    "    yuv.z = texture2D(CC_Texture2, v_texCoord).r - 0.5;\n"
    "    vec3 rgb = mat3(1.164, 1.164, 1.164,\n"
    "        0.0, -0.392, 2.017,\n"
    "        1.596, -0.813, 0.0) * yuv;\n"
    "    gl_FragColor = vec4(rgb, 1.0) * v_fragmentColor;\n"
    "}\n";
```

> **为什么用 `lowp`？** YUV 到 RGB 的转换涉及的值域：YUV 分量 0.0-1.0（纹理采样得到）、转换矩阵系数最大值 2.017（在 `lowp` 的 ±2.0 范围极限附近）、输出 RGB 0.0-1.0。严格来说，矩阵系数 2.017 超过了 `lowp` 规范保证的 ±2.0 范围。但实际移动 GPU 上 `lowp` 通常实现为 `mediump`（16 位），所以不会有问题。这是一个"实用但不严格安全"的做法——如果要严格遵守规范，应该用 `mediump`。

### 代码示例：着色器前缀生成器

将前面三种方案的思想整合成一个实用工具函数：

```cpp
// 文件：ShaderPreamble.h
// 封装着色器前缀生成器——根据平台自动生成 #version、precision、宏定义

#pragma once
#include <string>

inline std::string buildShaderPreamble(bool isES, int esVersion = 100) {
    std::string result;

    // 版本声明
    if (isES) {
        if (esVersion >= 300) {
            result = "#version 300 es\n";
            // ES 3.0 着色器必须以 #version 300 es 开头
        } else {
            result = "#version 100\n";
            // ES 2.0 着色器使用 #version 100
        }
        result += "precision mediump float;\n";
        // ES 必须声明默认精度
        result += "#define HP highp\n";
        result += "#define MP mediump\n";
        result += "#define LP lowp\n";
    } else {
        result = "#version 330 core\n";
        // 桌面使用 OpenGL 3.3 Core Profile
        result += "#define HP\n";   // 桌面上宏替换为空
        result += "#define MP\n";
        result += "#define LP\n";
    }

    // 统一纹理采样函数名
    // ES 2.0 用 texture2D()，ES 3.0+ 和桌面 3.3+ 用 texture()
    if (isES && esVersion < 300) {
        result += "#define SAMPLE texture2D\n";
    } else {
        result += "#define SAMPLE texture\n";
    }

    // 统一输入/输出关键字
    // ES 2.0 用 attribute/varying，ES 3.0+ 和桌面用 in/out
    if (isES && esVersion < 300) {
        result += "#define IN_ATTR attribute\n";
        // attribute：顶点着色器的输入（每个顶点的属性数据）
        result += "#define IN_VARY varying\n";
        // varying（片段着色器中）：从顶点着色器插值传来的数据
        result += "#define OUT_VARY varying\n";
        // varying（顶点着色器中）：输出到片段着色器的数据
    } else {
        result += "#define IN_ATTR in\n";
        result += "#define IN_VARY in\n";
        result += "#define OUT_VARY out\n";
    }

    return result;
}

// 使用示例：
// GLCapabilities caps = probeGLCapabilities();
// std::string preamble = buildShaderPreamble(caps.isGLES,
//     caps.majorVersion * 100 + caps.minorVersion * 10);
// std::string fullSrc = preamble + shaderBody;
// const char* src = fullSrc.c_str();
// glShaderSource(shader, 1, &src, nullptr);
// glShaderSource(shader, count, strings, lengths)：设置着色器源码
//   shader：着色器对象 ID（由 glCreateShader 创建）
//   count：源码字符串数组的元素数
//   strings：源码字符串数组（指针的数组）
//   lengths：每个字符串长度（nullptr 表示以 null 结尾自动计算）
```

使用 `buildShaderPreamble` 后，着色器代码可以完全跨平台：

```glsl
// 文件：universal_shader.vert
// 假设前缀已通过 buildShaderPreamble() 拼接到开头

IN_ATTR HP vec4 aPosition;    // 顶点位置（高精度）
IN_ATTR MP vec2 aTexCoord;    // 纹理坐标（中精度）
OUT_VARY MP vec2 vTexCoord;   // 传给片段着色器

uniform HP mat4 uMVP;         // 模型-视图-投影矩阵
// MVP（Model-View-Projection）：三个变换矩阵的乘积
//   Model（模型矩阵）：物体从自身坐标系变换到世界坐标系
//   View（视图矩阵）：世界坐标系变换到摄像机坐标系
//   Projection（投影矩阵）：摄像机坐标系变换到屏幕坐标系

void main() {
    gl_Position = uMVP * aPosition;
    // gl_Position：GLSL 内置输出变量，顶点着色器必须写入
    //   表示该顶点在裁剪空间中的坐标 (x, y, z, w)
    vTexCoord = aTexCoord;
}
```

```glsl
// 文件：universal_shader.frag
// 假设前缀已通过 buildShaderPreamble() 拼接到开头

IN_VARY MP vec2 vTexCoord;
uniform sampler2D uTexture;

void main() {
    // SAMPLE 在 ES 2.0 上替换为 texture2D，在桌面/ES 3.0 上替换为 texture
    gl_FragColor = SAMPLE(uTexture, vTexCoord);
}
```

## 五、三种方案对比

| 方案 | 复杂度 | 灵活性 | 运行时开销 | 适用场景 |
|------|--------|--------|------------|----------|
| `#ifdef GL_ES` 条件编译 | 低 | 低 | 无 | 简单项目、只需兼容 ES 2.0 和桌面 |
| 精度宏头文件 | 中 | 中 | 无 | 需要对不同变量使用不同精度 |
| CPU 端字符串拼接 | 高 | 最高 | 微小（字符串拼接） | 大型引擎、需要运行时选择着色器变体 |

决策建议：
- **着色器 < 10 个**：方案一足够
- **着色器 10-50 个**，需要精度控制：方案二
- **着色器 > 50 个**，或需要动态变体：方案三（KrKr2/Cocos2d-x 的选择）

### 实践 3：运行时精度信息查询

```cpp
// 文件：query_precision.cpp
// 查询当前 GPU 着色器精度信息——需要有效的 GL 上下文

#include <cstdio>

void queryShaderPrecision() {
    // glGetShaderPrecisionFormat(shaderType, precisionType, range, precision)
    //   查询着色器中指定类型的精度信息
    //   参数 1 (shaderType)：GL_VERTEX_SHADER 或 GL_FRAGMENT_SHADER
    //   参数 2 (precisionType)：
    //     GL_LOW_FLOAT, GL_MEDIUM_FLOAT, GL_HIGH_FLOAT
    //     GL_LOW_INT, GL_MEDIUM_INT, GL_HIGH_INT
    //   参数 3 (range)：输出数组[2]，[2^range[0], 2^range[1]] 范围
    //   参数 4 (precision)：输出，尾数位数
    //   注意：此函数仅在 ES 上有实际意义，桌面也支持但返回固定值

    GLint range[2], precision;

    // 查询片段着色器三个精度级别
    const char* names[] = {"lowp", "mediump", "highp"};
    GLenum types[] = {GL_LOW_FLOAT, GL_MEDIUM_FLOAT, GL_HIGH_FLOAT};

    for (int i = 0; i < 3; i++) {
        glGetShaderPrecisionFormat(GL_FRAGMENT_SHADER, types[i],
                                   range, &precision);
        printf("片段着色器 %s float:\n", names[i]);
        printf("  范围: 2^%d 到 2^%d\n", range[0], range[1]);
        printf("  精度位数: %d\n", precision);
    }
}

// 典型输出（ARM Mali G72）：
// 片段着色器 lowp float:
//   范围: 2^14 到 2^14     ← lowp 和 mediump 相同
//   精度位数: 10
// 片段着色器 mediump float:
//   范围: 2^14 到 2^14
//   精度位数: 10
// 片段着色器 highp float:
//   范围: 2^127 到 2^127
//   精度位数: 23

// 典型输出（NVIDIA GeForce，桌面）：
// 全部三级都返回 range=127, precision=23（即 FP32）
```

## 常见错误及解决方案

### 错误一：`#version` 放在 `#ifdef` 后面导致编译失败

**错误代码**：
```glsl
#ifdef GL_ES
    #version 100
    precision mediump float;
#else
    #version 330 core
#endif
```

**原因**：GLSL 规范要求 `#version` 必须是着色器源码的**第一条非空白、非注释语句**。`#ifdef` 出现在 `#version` 前面违反了这个规则，某些 GLSL 编译器会报错。

**修复**：在 CPU 端通过字符串拼接处理版本声明，不要在 GLSL 内部用 `#ifdef` 包裹 `#version`。使用 `buildShaderPreamble()` 函数可以自动处理这个问题。

### 错误二：忘记处理 `texture2D` / `texture` 差异

**现象**：使用 `#ifdef GL_ES` 处理了精度声明，但着色器中仍然使用 `texture()` 函数，导致 ES 2.0 编译失败。

**错误信息**：
```
ERROR: 0:X: 'texture' : no matching overloaded function found
```

**原因**：ES 2.0（GLSL ES 1.00）使用 `texture2D()` 进行 2D 纹理采样。`texture()` 是 ES 3.0+ 和桌面 GL 3.3+ 的统一采样函数。

**修复**：使用宏统一函数名（参见 `buildShaderPreamble` 中的 `SAMPLE` 宏定义）。

### 错误三：ES 2.0 的 `varying`/`attribute` 和 ES 3.0+ 的 `in`/`out` 混用

**现象**：着色器在 ES 2.0 设备上正常，但在 ES 3.0 设备上报语法错误。

**原因**：ES 3.0（`#version 300 es`）废弃了 `attribute` 和 `varying` 关键字，必须用 `in`/`out` 替代。如果你写了 `#version 300 es` 但仍使用 `varying`，编译器会报错。

**修复**：使用 `IN_ATTR`/`IN_VARY`/`OUT_VARY` 宏（参见 `buildShaderPreamble`），根据版本自动选择正确的关键字。

---

## 本节小结

- 跨平台精度兼容的**核心矛盾**：ES 必须有 `precision` 声明，桌面不需要
- **三种解决方案**：
  - `#ifdef GL_ES` 条件编译——最简单，适合小项目
  - 精度宏头文件（HP/MP/LP）——可以精细控制每个变量的精度
  - CPU 端字符串拼接——最灵活，大型引擎首选（KrKr2 使用此方案）
- `buildShaderPreamble()` 工具函数可以自动处理版本声明、精度声明、关键字差异、采样函数差异
- KrKr2 使用 `#ifdef GL_ES` + `precision lowp float;` 处理 YUV 着色器的精度兼容
- `#version` 必须是着色器的第一条语句，不能放在 `#ifdef` 内部——需要用 CPU 端拼接处理
- 用 `glGetShaderPrecisionFormat` 可以在运行时查询当前 GPU 的实际精度能力

### 题目 1：跨平台着色器修复

以下着色器在桌面 OpenGL 上正常工作。请列出它在 Android ES 2.0 上编译时会遇到的所有问题，并给出修复后的跨平台版本。

```glsl
#version 330 core
in vec2 vUV;
in vec4 vColor;
uniform sampler2D uTex;
out vec4 FragColor;
void main() {
    vec4 tex = texture(uTex, vUV);
    FragColor = tex * vColor;
}
```

<details>
<summary>查看答案</summary>

**问题列表**（6 个）：
1. `#version 330 core` → ES 2.0 不支持，应为 `#version 100`
2. 缺少 `precision mediump float;`
3. `in vec2 vUV` → ES 2.0 用 `varying vec2 vUV`
4. `in vec4 vColor` → ES 2.0 用 `varying vec4 vColor`
5. `out vec4 FragColor` → ES 2.0 用内置 `gl_FragColor`
6. `texture()` → ES 2.0 用 `texture2D()`

**修复方案——方案一（手动条件编译）**：

```glsl
#ifdef GL_ES
precision mediump float;
#endif

varying vec2 vUV;
varying vec4 vColor;
uniform sampler2D uTex;

void main() {
    vec4 tex = texture2D(uTex, vUV);
    gl_FragColor = tex * vColor;
}
```

注意：这个方案放弃了 `#version` 声明（ES 2.0 默认 `#version 100`，省略也可以编译）。如果要同时兼容桌面 GL 3.3 的 Core Profile（需要 `#version 330 core`），则必须用 CPU 端拼接处理版本声明。

**修复方案——方案三（CPU 端拼接）**：

```cpp
// CPU 端
std::string preamble = buildShaderPreamble(caps.isGLES,
    caps.majorVersion * 100 + caps.minorVersion * 10);
const char* fragBody =
    "IN_VARY MP vec2 vUV;\n"
    "IN_VARY MP vec4 vColor;\n"
    "uniform sampler2D uTex;\n"
    "void main() {\n"
    "    MP vec4 tex = SAMPLE(uTex, vUV);\n"
    "    gl_FragColor = tex * vColor;\n"
    "}\n";
std::string fullSrc = preamble + fragBody;
```

</details>

### 题目 2：方案选型

你正在开发一个 2D 游戏引擎，有以下特征：
- 需要支持 Windows (桌面 GL) + Android (ES 2.0/3.0) + iOS (ES 3.0)
- 约 30 个不同的着色器
- 部分着色器需要根据设备能力动态选择特性（如是否启用软阴影）

应该选择哪种跨平台方案？说明理由。

<details>
<summary>查看答案</summary>

应该选择**方案三：CPU 端字符串拼接**，理由如下：

1. **三平台三版本**：需要同时兼容桌面 GL 3.3、ES 2.0、ES 3.0 三种 GLSL 版本。`#version` 声明不同，关键字不同（varying/in），采样函数不同（texture2D/texture）——用方案一需要大量 `#ifdef` 嵌套
2. **30 个着色器**：数量中等，手动维护 `#ifdef` 分支容易出错且难以一致修改
3. **动态特性选择**：方案一和方案二的宏在 GLSL 编译时就固定了。方案三可以在运行时根据设备能力拼接不同的代码路径（如：`if (caps.hasShadow) preamble += "#define HAS_SHADOW 1\n";`）
4. **统一维护点**：`buildShaderPreamble()` 集中管理所有平台差异，新增平台支持只需修改一处

如果不需要动态特性选择，30 个着色器也可以用方案二（精度宏头文件）——但需要额外处理 `#version` 和关键字差异。

</details>

### 题目 3：精度宏设计

设计一个精度宏系统，除了 HP/MP/LP 三个精度宏外，再增加一个 `PRECISION_HIGH_OR_MEDIUM` 宏——在支持 `highp` 的 GPU 上使用 `highp`，在不支持的 GPU 上自动降级到 `mediump`。

<details>
<summary>查看答案</summary>

```glsl
// 文件：precision_compat_v2.glsl

#ifdef GL_ES
    precision mediump float;
    #define HP highp
    #define MP mediump
    #define LP lowp

    // 利用 GL_FRAGMENT_PRECISION_HIGH 宏检测 highp 支持
    #ifdef GL_FRAGMENT_PRECISION_HIGH
        #define PRECISION_HIGH_OR_MEDIUM highp
    #else
        #define PRECISION_HIGH_OR_MEDIUM mediump
    #endif
#else
    // 桌面上所有精度宏替换为空
    #define HP
    #define MP
    #define LP
    #define PRECISION_HIGH_OR_MEDIUM
#endif
```

使用场景：

```glsl
// 需要尽可能高精度但要兼容低端设备的变量
uniform PRECISION_HIGH_OR_MEDIUM float uTime;
// 在支持 highp 的 GPU 上：highp float uTime
// 在不支持的 GPU 上：mediump float uTime
// 在桌面上：float uTime（32 位，精度宏为空）
```

`GL_FRAGMENT_PRECISION_HIGH` 是 GLSL ES 内置宏——如果当前 GPU 的片段着色器支持 `highp`，此宏会被自动定义。大多数现代移动 GPU（2014 年后）都支持 `highp`，但部分旧款 Mali-400 系列不支持。

</details>
