## 一、为什么需要动态加载函数指针

上一节的能力结构体告诉我们"GPU 能做什么"。但光知道能力还不够——你还需要**拿到对应功能的函数入口地址**才能调用它。

**问题**：GL 核心函数（如 `glBindTexture`、`glDrawArrays`）在 GL 头文件中有声明，链接器能直接找到它们。但**扩展函数**（如 `glCopyImageSubDataEXT`、`glClearTexImageEXT`）不在标准头文件中——它们是驱动动态提供的。

**类比**：想象你去一家餐厅。菜单上的菜（核心函数）直接点就行。但有些隐藏菜单上的特色菜（扩展函数），你需要先问服务员"今天有没有这道菜？有的话怎么点？"——这就是 `GetProcAddress` 做的事情。

```
核心函数：glBindTexture(...)    ← 头文件声明，链接器直接找到
扩展函数：glCopyImageSubData... ← 头文件没有，需要运行时查询地址
```

**解决方案**：
1. 先声明函数指针类型（函数的签名）
2. 用平台特定的 `GetProcAddress` 函数查询地址
3. 如果查到了（非 `nullptr`），就通过函数指针调用

## 二、平台特定的 GetProcAddress

每个平台查询 GL 函数地址的方式不同：

| 平台 | 函数名 | 头文件 | 调用约定 |
|------|--------|--------|----------|
| Windows | `wglGetProcAddress` | `<GL/gl.h>` + `<wingdi.h>` | `WINAPI` |
| Android | `eglGetProcAddress` | `<EGL/egl.h>` | `EGLAPIENTRY` |
| Linux | `glXGetProcAddress` | `<GL/glx.h>` | `GLAPIENTRY` |
| macOS | `dlsym(RTLD_DEFAULT, ...)` | `<dlfcn.h>` | 标准 C |

### 代码示例 1：跨平台函数指针加载封装

```cpp
// 文件：GLLoader.h — 跨平台 GL 函数指针加载封装
#pragma once

// 函数指针类型——定义要加载的函数签名
// 以 glCopyImageSubData 为例：把一张纹理的指定区域拷贝到另一张，不经 CPU
typedef void (*PFN_glCopyImageSubData)(
    unsigned int srcName,  // 源纹理 ID
    unsigned int srcTarget,// 源纹理类型（GL_TEXTURE_2D 等）
    int srcLevel,          // 源 Mipmap 级别
    int srcX, int srcY, int srcZ,  // 源区域起点
    unsigned int dstName,  // 目标纹理 ID
    unsigned int dstTarget, int dstLevel,
    int dstX, int dstY, int dstZ,  // 目标区域起点
    int width, int height, int depth // 拷贝区域大小
);

static PFN_glCopyImageSubData pfn_glCopyImageSubData = nullptr;

// 跨平台加载单个函数指针
void* loadGLFunction(const char* name) {
#ifdef _MSC_VER
    // Windows：wglGetProcAddress 只能查扩展函数
    return (void*)wglGetProcAddress(name);
#elif defined(__ANDROID__)
    // Android：eglGetProcAddress 可查核心和扩展
    return (void*)eglGetProcAddress(name);
#elif defined(__linux__)
    return (void*)glXGetProcAddress((const unsigned char*)name);
#elif defined(__APPLE__)
    // macOS：dlsym 从动态链接库查询
    return dlsym(RTLD_DEFAULT, name);
#else
    return nullptr;
#endif
}
```

> **实际项目推荐**使用 **glad** 或 **GLEW** 自动处理所有平台差异。手动加载主要用于理解原理和处理特殊扩展。

## 三、多后缀回退加载策略

同一功能的函数名有多个后缀，因为不同厂商给同一功能起不同名字：

| 扩展来源 | 函数名 |
|----------|--------|
| EXT（跨厂商扩展） | `glCopyImageSubDataEXT` |
| OES（ES 官方扩展） | `glCopyImageSubDataOES` |
| NV（NVIDIA 专有） | `glCopyImageSubDataNV` |
| 核心（ARB 标准化后） | `glCopyImageSubData` |

**策略**：按优先级逐一尝试，第一个成功的就使用：

### 代码示例 2：多后缀回退加载

```cpp
// 文件：GLLoader.cpp — 多后缀回退加载
struct FuncSuffix {
    const char* extName;   // 对应的扩展名
    const char* funcName;  // 函数名（带后缀）
};

// 按优先级排列：EXT（最通用）→ OES → NV → 核心
static const FuncSuffix copyImageSuffixes[] = {
    {"GL_EXT_copy_image", "glCopyImageSubDataEXT"},
    {"GL_OES_copy_image", "glCopyImageSubDataOES"},
    {"GL_NV_copy_image",  "glCopyImageSubDataNV"},
    {"GL_ARB_copy_image", "glCopyImageSubData"},  // 标准化后无后缀
    {nullptr, nullptr}  // 哨兵值
};

// 通用回退加载——先检查扩展再加载函数指针
void* loadWithFallback(const FuncSuffix* suffixes) {
    for (int i = 0; suffixes[i].funcName; i++) {
        if (suffixes[i].extName && !checkExtension(suffixes[i].extName))
            continue;  // 扩展不支持，跳过
        void* ptr = loadGLFunction(suffixes[i].funcName);
        if (ptr) return ptr;  // 加载成功
    }
    return nullptr;  // 所有后缀都失败
}

// 使用：
// pfn_glCopyImageSubData = (PFN_glCopyImageSubData)
//     loadWithFallback(copyImageSuffixes);
// if (pfn_glCopyImageSubData) { /* GPU 纹理拷贝 */ }
// else { /* 降级：glReadPixels + glTexSubImage2D 经 CPU 拷贝 */ }
```

## 四、对照 KrKr2 源码——三层架构完整实现

KrKr2 的 `RenderManager_ogl.cpp` 完整实现了上一节介绍的三层架构。以下逐层分析。

### 第一层：函数指针声明与 GetProcAddress（第 149-191 行）

```cpp
// 文件：cpp/core/visual/ogl/RenderManager_ogl.cpp 第 149-191 行
namespace GL {
    // 平台特定的 GetProcAddress 类型定义
#ifdef _MSC_VER
    typedef PROC(WINAPI fGetProcAddress)(LPCSTR);
    // Windows：PROC 是返回类型，WINAPI 是调用约定，LPCSTR 是参数
#elif defined(__ANDROID__)
    typedef void *(EGLAPIENTRY fGetProcAddress)(const char *);
    // Android：EGLAPIENTRY 是 EGL 调用约定
#elif defined(LINUX)
    typedef void *(GLAPIENTRY fGetProcAddress)(const char *);
    // Linux：GLAPIENTRY 是 GL 标准调用约定
#endif
    static fGetProcAddress *glGetProcAddress = nullptr;

    // 扩展函数指针声明——每个函数都是一个 typedef + static 变量
    typedef void(GLAPIENTRY fCopyImageSubData)(
        GLuint srcName, GLenum srcTarget, GLint srcLevel,
        GLint srcX, GLint srcY, GLint srcZ,
        GLuint dstName, GLenum dstTarget, GLint dstLevel,
        GLint dstX, GLint dstY, GLint dstZ,
        GLsizei width, GLsizei height, GLsizei depth);
    static fCopyImageSubData *glCopyImageSubData;

    typedef void(GLAPIENTRY fClearTexImage)(...);
    static fClearTexImage *glClearTexImage;
    typedef void(GLAPIENTRY fClearTexSubImage)(...);
    static fClearTexSubImage *glClearTexSubImage;
}
```

**关键设计**：所有扩展函数指针放在 `GL` 命名空间中，避免与全局 GL 函数名冲突。`#ifdef` 只出现在 `GetProcAddress` 的类型定义——这是三层架构中**唯一允许 `#ifdef` 的地方**。

### 第二层：扩展信息收集（第 84-147 行）

```cpp
// 文件：RenderManager_ogl.cpp 第 84-103 行
// 用宏批量声明扩展名字符串
struct _UsedGLExtInfo {
    const char *NameBegin = nullptr;
#define DEFEXT(name) const char *GLEXT_##name = #name
    DEFEXT(GL_EXT_unpack_subimage);
    DEFEXT(GL_EXT_shader_framebuffer_fetch);
    DEFEXT(GL_ARM_shader_framebuffer_fetch);
    DEFEXT(GL_NV_shader_framebuffer_fetch);
    DEFEXT(GL_EXT_copy_image);
    DEFEXT(GL_OES_copy_image);
    DEFEXT(GL_ARB_copy_image);
    DEFEXT(GL_NV_copy_image);
    DEFEXT(GL_EXT_clear_texture);
    DEFEXT(GL_ARB_clear_texture);
    DEFEXT(GL_QCOM_alpha_test);
#undef DEFEXT
    const char *NameEnd = nullptr;
};
```

> **DEFEXT 宏的巧妙之处**：`DEFEXT(GL_EXT_copy_image)` 展开为 `const char *GLEXT_GL_EXT_copy_image = "GL_EXT_copy_image"`。`NameBegin` 和 `NameEnd` 是哨兵——利用结构体成员在内存中连续排列的特性，可以从 `NameBegin+1` 遍历到 `NameEnd` 来迭代所有扩展名。

```cpp
// 第 114-147 行：扩展信息初始化
static void TVPInitGLExtensionInfo() {
    // 1. 解析 glGetString(GL_EXTENSIONS) 到 unordered_set
    std::string gl_extensions = (const char*)glGetString(GL_EXTENSIONS);
    // 按空格分割，逐个插入 set
    // ...

    // 2. 配置文件可禁用特定扩展（用于调试/规避驱动 bug）
    IndividualConfigManager *cfgMgr = IndividualConfigManager::GetInstance();
    for (const char *const *name = (&UsedGLExtInfo.NameBegin) + 1;
         *name; ++name) {
        if (!cfgMgr->GetValue<int>(*name, 1)) {
            sTVPGLExtensions.erase(*name);  // 配置为 0 则移除
        }
    }

    // 3. Windows 强制禁用 GL_EXT_unpack_subimage
    // 桌面 GL 内置 GL_UNPACK_ROW_LENGTH，不需要此 ES 扩展
#ifdef WIN32
    sTVPGLExtensions.erase("GL_EXT_unpack_subimage");
#endif
}
```

### 第一层续：函数指针加载（第 210-275 行）

```cpp
// 第 210-275 行：多后缀回退加载
static void TVPInitGLExtensionFunc() {
#ifdef _MSC_VER
    GL::glGetProcAddress = wglGetProcAddress;   // Windows
#elif defined(EGLAPI)
    GL::glGetProcAddress = (GL::fGetProcAddress*)eglGetProcAddress; // Android
#endif

    // glCopyImageSubData 的四后缀回退：
    if (!GL::glCopyImageSubData &&
        TVPCheckGLExtension("GL_EXT_copy_image"))
        GL::glCopyImageSubData = (GL::fCopyImageSubData*)
            GL::glGetProcAddress("glCopyImageSubDataEXT");
    if (!GL::glCopyImageSubData &&
        TVPCheckGLExtension("GL_OES_copy_image"))
        GL::glCopyImageSubData = (GL::fCopyImageSubData*)
            GL::glGetProcAddress("glCopyImageSubDataOES");
    if (!GL::glCopyImageSubData &&
        TVPCheckGLExtension("GL_NV_copy_image"))
        GL::glCopyImageSubData = (GL::fCopyImageSubData*)
            GL::glGetProcAddress("glCopyImageSubDataNV");
    if (!GL::glCopyImageSubData &&
        TVPCheckGLExtension("GL_ARB_copy_image"))
        GL::glCopyImageSubData = (GL::fCopyImageSubData*)
            GL::glGetProcAddress("glCopyImageSubData");  // ARB 无后缀

    // glClearTexImage 同理——EXT → ARB 两后缀
    // glAlphaFunc——核心 GL 有，ES 需要 QCOM 扩展
}
```

> **对比教程中的 `loadWithFallback`**：KrKr2 没有封装通用的回退函数，而是对每个扩展函数手动写 if 链。两种风格各有优劣：手动写更直观但重复代码多，封装后更简洁但需要额外的数据结构。

### KrKr2 扩展函数加载总表

| 函数名 | 后缀变体 | 用途 |
|--------|----------|------|
| `glCopyImageSubData` | EXT/OES/NV/ARB（4 种） | GPU 间纹理拷贝 |
| `glClearTexImage` | EXT/ARB（2 种） | 快速纹理清零 |
| `glClearTexSubImage` | EXT/ARB（2 种） | 快速纹理子区域清零 |
| `glGetTextureImage` | 仅 Windows | 读取纹理像素到 CPU |
| `glAlphaFunc` | 核心/QCOM（2 种） | Alpha 测试（ES 无核心支持） |

## 动手实践

### 实践：实现完整的扩展加载器

```cpp
// 文件：practice_loader.cpp
// 完整的扩展函数加载器——结合能力探测和函数指针加载
#include <cstdio>
#include <cstring>

// 模拟 GL 类型（实际项目中来自 GL 头文件）
typedef unsigned int GLuint;
typedef unsigned int GLenum;
typedef int GLint;
typedef int GLsizei;

// 函数指针类型
typedef void (*PFN_CopyImageSubData)(
    GLuint, GLenum, GLint, GLint, GLint, GLint,
    GLuint, GLenum, GLint, GLint, GLint, GLint,
    GLsizei, GLsizei, GLsizei);

// 回退表
struct FuncEntry {
    const char* ext;
    const char* func;
};

static const FuncEntry kCopyImageEntries[] = {
    {"GL_EXT_copy_image", "glCopyImageSubDataEXT"},
    {"GL_OES_copy_image", "glCopyImageSubDataOES"},
    {"GL_NV_copy_image",  "glCopyImageSubDataNV"},
    {"GL_ARB_copy_image", "glCopyImageSubData"},
    {nullptr, nullptr}
};

class GLExtensionLoader {
    PFN_CopyImageSubData mCopyImageSubData = nullptr;
    bool mHasCopyImage = false;
public:
    void init() {
        for (int i = 0; kCopyImageEntries[i].func; i++) {
            if (kCopyImageEntries[i].ext
                && !checkExtension(kCopyImageEntries[i].ext))
                continue;
            mCopyImageSubData = (PFN_CopyImageSubData)
                loadGLFunction(kCopyImageEntries[i].func);
            if (mCopyImageSubData) {
                mHasCopyImage = true;
                printf("已加载 %s (来自 %s)\n",
                       kCopyImageEntries[i].func,
                       kCopyImageEntries[i].ext);
                break;
            }
        }
        if (!mHasCopyImage)
            printf("glCopyImageSubData 不可用，将使用 CPU 拷贝\n");
    }
    bool hasCopyImage() const { return mHasCopyImage; }
};
```

## 常见错误及解决方案

### 错误一：不检查 nullptr 直接调用函数指针

**现象**：程序在某些 GPU 上崩溃（空指针调用）。**原因**：扩展不支持时函数指针为 `nullptr`。**修复**——调用前必须检查：

```cpp
// 错误——直接调用
GL::glCopyImageSubData(src, GL_TEXTURE_2D, ...);  // 崩溃！

// 正确——先检查
if (GL::glCopyImageSubData) {
    GL::glCopyImageSubData(src, GL_TEXTURE_2D, ...);
} else {
    fallbackCopyViaReadPixels(src, dst, ...);
}
```

### 错误二：函数指针与扩展名不匹配

**现象**：加载到错误的函数，参数个数或类型不对，导致内存损坏。**原因**：用 `GL_EXT_copy_image` 的函数名加载了 `GL_OES_copy_image` 的入口地址。虽然大部分实现中这些是同一个函数，但规范上不保证。**修复**——严格配对扩展名和函数名：

```cpp
// 错误——扩展和函数名不配对
if (checkExtension("GL_OES_copy_image"))
    ptr = loadGLFunction("glCopyImageSubDataEXT");  // 应该用 OES 后缀！

// 正确——配对
if (checkExtension("GL_OES_copy_image"))
    ptr = loadGLFunction("glCopyImageSubDataOES");
```

### 错误三：在上下文切换后使用旧函数指针

**现象**：多窗口应用中，窗口 B 的 GL 调用用了窗口 A 加载的函数指针，导致崩溃或渲染错误。**原因**：不同 GL 上下文的函数指针地址可能不同（特别是 Windows 上的 `wglGetProcAddress`，返回的是当前上下文的函数地址）。**修复**——每个上下文加载自己的函数指针，或验证上下文共享。

## 本节小结

- GL 扩展函数不在标准头文件中，需要通过 **GetProcAddress** 在运行时动态获取函数指针
- 每个平台使用不同的 GetProcAddress：Windows (`wgl`)、Android (`egl`)、Linux (`glX`)、macOS (`dlsym`)
- **多后缀回退**：同一功能有 EXT/OES/NV/ARB 等多个函数名，按优先级逐一尝试加载
- KrKr2 在 `RenderManager_ogl.cpp` 完整实现了三层架构：
  - 第一层（149-275 行）：`GL` 命名空间声明函数指针 + `TVPInitGLExtensionFunc` 加载
  - 第二层（84-147 行）：`_UsedGLExtInfo` + `TVPInitGLExtensionInfo` 收集扩展信息
  - 第三层（300+ 行）：渲染代码通过 `TVPCheckGLExtension()` 读取布尔结果
- 调用扩展函数前**必须检查函数指针非空**，否则在不支持的 GPU 上会崩溃

### 题目 1：设计一个通用的扩展函数加载模板

C++ 模板可以避免每个函数都手写类型转换。请用模板实现一个通用的 `loadGLFunctionWithFallback` 函数。

<summary>查看答案</summary>

```cpp
template<typename FuncType>
FuncType loadGLFunctionWithFallback(
    const FuncEntry* entries)
{
    for (int i = 0; entries[i].func; i++) {
        if (entries[i].ext && !checkExtension(entries[i].ext))
            continue;
        auto ptr = (FuncType)loadGLFunction(entries[i].func);
        if (ptr) return ptr;
    }
    return nullptr;
}

// 使用——类型安全，无需手动转换
PFN_CopyImageSubData pfn = loadGLFunctionWithFallback<PFN_CopyImageSubData>(
    kCopyImageEntries);
```

### 题目 2：KrKr2 为什么在 Windows 上强制禁用 `GL_EXT_unpack_subimage`？

<details>
<summary>查看答案</summary>

`GL_EXT_unpack_subimage` 提供的功能是 `GL_UNPACK_ROW_LENGTH`（设置纹理上传时的行间距），允许从大图中上传一个子区域。

桌面 OpenGL 从 1.0 起就**内置支持** `GL_UNPACK_ROW_LENGTH`，不需要通过扩展获取。这个扩展是为 **OpenGL ES** 设计的（ES 2.0 没有此功能）。

如果 Windows 上某些驱动也报告支持此扩展，可能导致代码错误地走 ES 兼容路径，使用扩展版本而非核心版本的函数。强制移除此扩展可以确保 Windows 代码走正确的桌面 GL 路径。

参考源码：`RenderManager_ogl.cpp` 第 140 行。

</details>
