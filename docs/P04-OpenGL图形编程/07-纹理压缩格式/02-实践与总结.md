## 动手实践

实践目标：
做一个跨平台压缩纹理加载模块。

步骤：
1. 离线生成 ASTC/ETC2/BC/PVRTC 四套资源。
2. 引擎启动时查询压缩格式支持集。
3. 按平台与 Alpha 需求选择格式。
4. 用 `glCompressedTexImage2D` 上传全部 mip 层。
5. 上传失败时回退 RGBA8。
6. 输出格式选择、错误码、回退路径日志。
7. 在 Windows/Linux/macOS/Android/iOS 验证。

验收标准：
- 透明 UI 无黑边。
- 远景采样无闪烁。
- 显存占用明显低于纯 RGBA8。

## 本节小结

1. 文件压缩与 GPU 压缩是两条不同链路。
2. 块压缩的本质是固定比特率与随机访问友好。
3. 桌面优先 BC，移动优先 ASTC/ETC2，特定场景兼容 PVRTC。
4. `imageSize` 计算错误是 `glCompressedTexImage2D` 的首要坑点。
5. 运行时检测与回退策略是跨平台稳定性的保障。
6. KrKr2 的 `ogl` 目录是非常好的实战参考。

## 练习题与答案

### 题目 1：为什么“全用 PNG”仍可能导致运行时显存爆炸？

<details>
<summary>查看答案</summary>

PNG 只压缩文件。上传 GPU 前会解码成像素数据。若最终驻留 RGBA8，显存和带宽压力不会下降。要降低运行时成本，必须使用 GPU 压缩格式（ASTC/ETC/BC/PVRTC）。

</details>

### 题目 2：ETC1 与 ETC2 在透明纹理处理上有什么本质差异？

<details>
<summary>查看答案</summary>

ETC1 不支持完整 Alpha，透明纹理不适用。ETC2（ES3+）支持 RGBA/EAC 变体，可直接用于透明纹理。

</details>

### 题目 3：1024×512 的 ASTC 8×8 单层数据大小是多少？

<details>
<summary>查看答案</summary>

`ceil(1024/8)=128`，`ceil(512/8)=64`，每块 16 字节。总大小 `128×64×16=131072` 字节，约 128KB。

</details>

### 题目 4：压缩纹理上传报 `GL_INVALID_VALUE`，优先排查哪三项？

<details>
<summary>查看答案</summary>

1. `imageSize` 是否按块计算。  
2. mip 层尺寸和数据长度是否匹配。  
3. NPOT 边缘块是否补齐。

</details>
