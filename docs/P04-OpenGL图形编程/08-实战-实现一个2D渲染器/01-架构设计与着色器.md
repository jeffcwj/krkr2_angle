## 什么是精灵渲染器

在 2D 游戏和视觉小说引擎中，屏幕上显示的每一张图片（角色立绘、背景、UI 按钮等）都叫做**精灵（Sprite）**。精灵渲染器就是负责把这些图片正确绘制到屏幕上的模块。它需要完成三件事：

1. **确定图片在屏幕上的位置和大小** — 需要坐标变换
2. **把图片的像素读出来** — 需要纹理采样（Texture Sampling，从纹理图片中读取颜色的操作）
3. **处理透明和叠加** — 需要混合（Blending，将新像素和屏幕上已有像素按规则混合的操作）

一个精灵渲染器的典型架构如下：

```
┌──────────────────────────────────────────────────────┐
│                2D 精灵渲染器总体架构                    │
│                                                       │
│  ┌────────────┐  ┌────────────┐  ┌────────────────┐  │
│  │  着色器管理  │  │  纹理管理   │  │  精灵批处理器   │  │
│  │  (Shader)   │  │  (Texture)  │  │  (Batcher)    │  │
│  └─────┬──────┘  └──────┬─────┘  └───────┬────────┘  │
│        │                │                │            │
│        └────────────────┼────────────────┘            │
│                         │                             │
│                  ┌──────┴───────┐                     │
│                  │   渲染上下文   │                     │
│                  │  (VAO/VBO)   │                     │
│                  └──────────────┘                     │
└──────────────────────────────────────────────────────┘
```

各组件的职责：

| 组件 | 职责 | 对应的 GL 概念 |
|------|------|---------------|
| 着色器管理 | 编译、链接、切换着色器程序 | `glCreateShader`、`glCreateProgram` |
| 纹理管理 | 加载图片、绑定纹理单元 | `glGenTextures`、`glBindTexture` |
| 精灵批处理器 | 合并多个精灵的顶点数据，减少绘制调用 | `glBufferData`、`glDrawElements` |
| 渲染上下文 | 管理 VAO/VBO/EBO 等 GL 对象的生命周期 | `glGenVertexArrays`、`glGenBuffers` |

### 关键设计决策

在开始写代码之前，需要做几个设计选择。不同的选择会影响后续所有代码的写法：

| 决策项 | 选择 | 原因 |
|--------|------|------|
| 坐标系 | 屏幕坐标（左上角原点，Y 轴向下） | 2D 游戏/UI 的通用惯例，与窗口系统一致 |
| 投影方式 | 正交投影（Orthographic） | 2D 渲染不需要透视效果（近大远小） |
| 顶点布局 | 交错式（位置 + UV + 颜色放在一起） | 对 GPU 缓存更友好，减少内存跳转 |
| 混合模式 | `SRC_ALPHA, ONE_MINUS_SRC_ALPHA` | 标准的 Alpha 半透明混合 |

> **交错式布局（Interleaved Layout）** 是指将同一个顶点的所有属性（位置、纹理坐标、颜色）紧挨着存放在内存中，形如 `[pos0,uv0,color0, pos1,uv1,color1, ...]`。与之相对的是分离式布局（Separate Layout），把位置、UV、颜色分别放在三个不同的数组里。交错式的好处是 GPU 读取一个顶点的所有数据时只需要一次缓存行（Cache Line）加载。

## 第一步：着色器

着色器是渲染器的核心——GPU 上运行的小程序，决定了每个像素最终显示什么颜色。2D 精灵渲染器只需要两个着色器：**顶点着色器**和**片段着色器**。

### 顶点着色器

顶点着色器（Vertex Shader）的工作是：接收每个顶点的屏幕坐标（像素单位），通过**正交投影矩阵**变换到 NDC（Normalized Device Coordinates，归一化设备坐标，范围 `[-1, +1]`），然后传递纹理坐标和颜色给片段着色器。

```cpp
// 2D 精灵渲染器 — 顶点着色器（Desktop OpenGL 3.3 版本）
const char* vertexShaderSrc = R"(
#version 330 core

// ===== 输入属性（从 VBO 中读取） =====
layout(location = 0) in vec2 aPos;       // 屏幕坐标，单位：像素
layout(location = 1) in vec2 aTexCoord;  // 纹理坐标，范围 [0,1]
layout(location = 2) in vec4 aColor;     // 顶点颜色（RGBA，含透明度）

// ===== Uniform（CPU 传入的全局参数） =====
uniform mat4 uProjection;  // 正交投影矩阵（将像素坐标映射到 NDC）

// ===== 输出（传递给片段着色器） =====
out vec2 vTexCoord;
out vec4 vColor;

void main() {
    // 将 2D 像素坐标扩展为 4D 齐次坐标，再乘以投影矩阵
    // vec4(aPos, 0.0, 1.0) 中：
    //   aPos.x, aPos.y = 屏幕像素坐标
    //   0.0 = Z 轴坐标（2D 渲染中固定为 0）
    //   1.0 = 齐次坐标的 w 分量（必须为 1，否则投影结果会被缩放）
    gl_Position = uProjection * vec4(aPos, 0.0, 1.0);

    // 纹理坐标和颜色直接传递，不做变换
    vTexCoord = aTexCoord;
    vColor = aColor;
}
)";
```

> **`layout(location = N)`** 是 OpenGL 3.3 引入的语法，用来指定顶点属性的编号。编号必须和 `glVertexAttribPointer` 中传入的第一个参数对应。如果不写 `layout`，就需要用 `glGetAttribLocation` 在运行时查询编号。

### GL 函数详解：着色器编译

编译一个着色器需要调用以下 GL 函数。这里给出完整的参数讲解：

```cpp
// ========== glCreateShader ==========
// 创建一个空的着色器对象
// 参数 shaderType：着色器类型
//   GL_VERTEX_SHADER   — 顶点着色器
//   GL_FRAGMENT_SHADER — 片段着色器
// 返回值：着色器对象的 ID（GLuint），0 表示创建失败
GLuint shader = glCreateShader(GL_VERTEX_SHADER);

// ========== glShaderSource ==========
// 向着色器对象传入源代码文本
// 参数1 shader：着色器 ID
// 参数2 count：源码字符串的数量（可以把多段代码拼接编译）
// 参数3 string：源码字符串数组的指针
// 参数4 length：每段源码的长度数组，传 nullptr 表示以 '\0' 结尾
const char* src = vertexShaderSrc;
glShaderSource(shader, 1, &src, nullptr);

// ========== glCompileShader ==========
// 编译着色器，将 GLSL 文本编译为 GPU 可执行的指令
// 编译结果需要用 glGetShaderiv 查询
glCompileShader(shader);

// ========== glGetShaderiv ==========
// 查询着色器的编译状态
// 参数 pname：
//   GL_COMPILE_STATUS  — 编译是否成功（GL_TRUE / GL_FALSE）
//   GL_INFO_LOG_LENGTH — 错误信息的字符长度
GLint success;
glGetShaderiv(shader, GL_COMPILE_STATUS, &success);
if (!success) {
    char infoLog[512];
    // ========== glGetShaderInfoLog ==========
    // 获取编译错误的详细信息（文本形式）
    // 参数2 maxLength：缓冲区最大长度
    // 参数3 length：实际写入的长度（可传 nullptr）
    // 参数4 infoLog：错误信息输出缓冲区
    glGetShaderInfoLog(shader, 512, nullptr, infoLog);
    printf("着色器编译失败: %s\n", infoLog);
}
```

KrKr2 中的着色器编译函数与上述流程完全一致。在 `RenderManager_ogl.cpp` 第 1871-1891 行：

```cpp

// 文件：cpp/core/visual/ogl/RenderManager_ogl.cpp  第 1871-1891 行

    GLenum shader = glCreateShader(shadertype);     // 1. 创建着色器
    const char *pszSource = src.c_str();

    glCompileShader(shader);                         // 3. 编译

    GLint status;

        GLint length;
        glGetShaderiv(shader, GL_INFO_LOG_LENGTH, &length);
        char *info = new char[length + 1];
        info[length] = 0;
        glGetShaderInfoLog(shader, length, nullptr, info); // 5. 获取错误信息
        // ... 抛出异常
    }
    return shader;
}
```

### 片段着色器

片段着色器（Fragment Shader）的工作是：对每个像素，从纹理中采样颜色，再与顶点颜色相乘，输出最终颜色。

```cpp
// 2D 精灵渲染器 — 片段着色器（Desktop OpenGL 3.3 版本）
const char* fragmentShaderSrc = R"(
#version 330 core

// ===== 从顶点着色器接收的插值数据 =====
in vec2 vTexCoord;   // 当前像素的纹理坐标（GPU 自动插值）
in vec4 vColor;      // 当前像素的颜色（GPU 自动插值）

// ===== Uniform =====
uniform sampler2D uTexture;  // 纹理采样器（绑定到纹理单元 0）

// ===== 输出 =====
out vec4 FragColor;  // 最终输出的像素颜色

void main() {
    // texture() 函数从纹理中采样颜色
    // 参数1：采样器（哪张纹理）
    // 参数2：纹理坐标（在纹理上的哪个位置）
    vec4 texColor = texture(uTexture, vTexCoord);

    // 纹理颜色 × 顶点颜色 = 最终颜色
    FragColor = texColor * vColor;
}
)";
```

**为什么要乘顶点颜色？** 这个乘法是 2D 渲染器最常用的技巧之一：

| 顶点颜色设置 | 效果 | 应用场景 |
|-------------|------|---------|
| `vec4(1,1,1,1)` 全白 | 纹理颜色不变 | 正常显示 |
| `vec4(1,1,1,0.5)` | 纹理半透明 | 淡入淡出动画 |
| `vec4(1,0,0,1)` 红色 | 纹理被染红 | 角色受击闪烁 |
| `vec4(0.5,0.5,0.5,1)` | 纹理变暗 | 按钮被按下时的视觉反馈 |

这种做法叫做**颜色调制（Color Modulation）**，即用乘法把一个颜色"叠加"到另一个颜色上。KrKr2 的着色器也用同样的技巧。

### 着色器程序的链接

编译好顶点着色器和片段着色器之后，需要将它们链接（Link）成一个**着色器程序（Shader Program）**。链接过程会检查两个着色器之间的接口是否匹配（比如顶点着色器输出的 `vTexCoord` 和片段着色器输入的 `vTexCoord` 类型是否一致）。

```cpp
// ========== glCreateProgram ==========
// 创建一个空的着色器程序对象
// 返回值：程序 ID（GLuint）
GLuint program = glCreateProgram();

// ========== glAttachShader ==========
// 将编译好的着色器附加到程序上
// 一个程序至少需要一个顶点着色器和一个片段着色器
glAttachShader(program, vertShader);
glAttachShader(program, fragShader);

// ========== glLinkProgram ==========
// 链接程序：检查接口匹配、分配 uniform 位置、生成可执行代码
glLinkProgram(program);

// 检查链接结果（与编译检查类似）
GLint linkSuccess;
glGetProgramiv(program, GL_LINK_STATUS, &linkSuccess);
if (!linkSuccess) {
    char infoLog[512];
    // ========== glGetProgramInfoLog ==========
    // 获取链接错误信息
    glGetProgramInfoLog(program, 512, nullptr, infoLog);
    printf("着色器链接失败: %s\n", infoLog);
}

// ========== glUseProgram ==========
// 激活着色器程序——后续的绘制调用将使用这个程序
// 传 0 表示取消激活（不使用任何着色器程序）
glUseProgram(program);
```


### Uniform 变量的设置

着色器中的 Uniform（全局变量）需要在 CPU 端通过 GL 函数设置。常用的 Uniform 设置函数：

```cpp
// ========== glGetUniformLocation ==========
// 根据名称查询 uniform 变量的位置（ID）
// 参数1 program：着色器程序 ID
// 参数2 name：uniform 变量名（必须与着色器源码中的名称完全一致）
// 返回值：位置 ID（-1 表示未找到，可能是被编译器优化掉了）
GLint projLoc = glGetUniformLocation(program, "uProjection");
GLint texLoc  = glGetUniformLocation(program, "uTexture");

// ========== glUniformMatrix4fv ==========
// 设置一个 4×4 矩阵 uniform
// 参数1 location：uniform 位置
// 参数2 count：矩阵个数（通常为 1）
// 参数3 transpose：是否转置（GL_FALSE = 列主序，OpenGL 默认）
// 参数4 value：浮点数组指针（16 个 float）
float projMatrix[16];
createOrthoProjection(projMatrix, screenW, screenH);
glUniformMatrix4fv(projLoc, 1, GL_FALSE, projMatrix);

// ========== glUniform1i ==========
// 设置一个整数 uniform（常用于绑定纹理采样器到纹理单元）
// 参数2 value：纹理单元编号（0 = GL_TEXTURE0）
glUniform1i(texLoc, 0);  // 纹理采样器绑定到纹理单元 0

// ========== glUniform1f / glUniform2f / glUniform4f ==========
// 设置浮点数 uniform（KrKr2 中大量使用）
// glUniform1f(loc, 0.5f);                    // 单个 float
// glUniform2f(loc, 1.0f, 2.0f);              // vec2
// glUniform4f(loc, r, g, b, a);              // vec4（颜色）

// KrKr2 中的用法（第 2039-2040 行）：
// int loc = glGetUniformLocation(program, "tex0");
// glUniform1i(loc, 0);  // tex0 -> GL_TEXTURE0
```

## OpenGL ES 2.0 兼容版本

KrKr2 需要在 Android 设备上运行，因此着色器必须兼容 OpenGL ES 2.0。两个版本的主要语法差异：

| 特性 | Desktop OpenGL 3.3 | OpenGL ES 2.0 |
|------|-------------------|---------------|
| 版本声明 | `#version 330 core` | `#version 100` |
| 输入变量关键字 | `in` | `attribute`（顶点着色器）/ `varying`（片段着色器） |
| 输出变量关键字 | `out` | `varying`（顶点着色器）/ 不需要声明（用内置 `gl_FragColor`） |
| 纹理采样函数 | `texture()` | `texture2D()` |
| 精度声明 | 不需要 | 片段着色器**必须**声明 `precision mediump float;` |
| 输出目标 | 自定义 `out vec4 FragColor` | 内置变量 `gl_FragColor` |
| 属性位置指定 | `layout(location = N)` | 只能用 `glGetAttribLocation` 查询 |

ES 2.0 版本的着色器：

```cpp
// ES 2.0 顶点着色器
const char* vertexShaderSrc_ES2 = R"(
#version 100
// ES 2.0 用 attribute 代替 in（表示从 VBO 读取的顶点属性）
attribute vec2 aPos;
attribute vec2 aTexCoord;
attribute vec4 aColor;

uniform mat4 uProjection;

// ES 2.0 用 varying 代替 out（表示传递给片段着色器的插值数据）
varying vec2 vTexCoord;
varying vec4 vColor;

void main() {
    gl_Position = uProjection * vec4(aPos, 0.0, 1.0);
    vTexCoord = aTexCoord;
    vColor = aColor;
}
)";

// ES 2.0 片段着色器
const char* fragmentShaderSrc_ES2 = R"(
#version 100
// 精度声明：ES 2.0 的片段着色器中，float 没有默认精度，必须手动声明
// mediump = 中等精度（16位浮点），对 2D 渲染完全够用
// 不声明会直接编译报错！
precision mediump float;

varying vec2 vTexCoord;
varying vec4 vColor;

uniform sampler2D uTexture;

void main() {
    // ES 2.0 用 texture2D() 而不是 texture()
    // 输出到内置变量 gl_FragColor，不需要声明 out
    gl_FragColor = texture2D(uTexture, vTexCoord) * vColor;
}
)";
```

KrKr2 使用条件编译来兼容两个版本。在 `RenderManager_ogl.cpp` 第 1931-1933 行：

```cpp
// KrKr2 的做法：在着色器源码开头加入条件编译
str << "#ifdef GL_ES\n"              // 如果运行在 ES 环境
       "precision mediump float;\n"  // 声明 float 精度
       "#endif\n";                   // Desktop GL 忽略这段
```

`GL_ES` 是 OpenGL ES 着色器编译器自动定义的宏。Desktop OpenGL 不会定义它，所以 `#ifdef GL_ES` 块内的代码只在移动端生效。这样一套着色器源码就能同时兼容桌面和移动平台。

## 常见错误及解决方案

### 错误 1：ES 2.0 片段着色器编译失败——缺少精度声明

```
ERROR: 0:1: '' : No precision specified for (float)
```

**原因**：OpenGL ES 2.0 的片段着色器中，`float` 类型没有默认精度，必须显式声明。  
**解决**：在片段着色器最开头添加 `precision mediump float;`。KrKr2 用 `#ifdef GL_ES` 条件编译实现跨平台兼容。

### 错误 2：着色器链接失败——varying 名称不匹配

```
ERROR: Varying vTexCoord has mismatched type between vertex and fragment shader
```

**原因**：顶点着色器输出的变量名和片段着色器输入的变量名不一致，或类型不一致。  
**解决**：确保两个着色器中 `out`/`in`（或 `varying`）变量的名称和类型完全相同。

### 错误 3：Uniform 位置返回 -1

```cpp
GLint loc = glGetUniformLocation(program, "uProjection");
// loc == -1
```

**原因**：可能是 uniform 名字写错了，也可能是着色器编译器检测到这个 uniform 从未被使用，自动优化掉了。  
**解决**：检查名称拼写。如果确实未使用，编译器会静默移除，这是正常行为。

## 本节小结

- 2D 精灵渲染器由四个核心组件组成：着色器管理、纹理管理、批处理器、渲染上下文
- 顶点着色器通过正交投影矩阵将屏幕像素坐标转换为 NDC
- 片段着色器从纹理采样后与顶点颜色相乘，实现颜色调制（Color Modulation）
- 着色器编译流程：`glCreateShader` → `glShaderSource` → `glCompileShader` → 检查 `GL_COMPILE_STATUS`
- 着色器链接流程：`glCreateProgram` → `glAttachShader` → `glLinkProgram` → 检查 `GL_LINK_STATUS`
- Desktop GL 3.3 和 ES 2.0 有 6 处主要语法差异，可用 `#ifdef GL_ES` 条件编译统一
- KrKr2 用 `CompileShader` + `CombineProgram` 完成着色器编译链接，并支持运行时动态生成

### 题目 1：手写一个着色器编译+链接的完整函数

编写一个 C++ 函数 `createShaderProgram`，接收顶点着色器和片段着色器的源码字符串，返回链接好的程序 ID。要求：编译失败时打印错误信息并返回 0。

<details>
<summary>查看答案</summary>

```cpp
#include <cstdio>
#include <GL/glew.h>  // 或其他 GL 加载库

// 编译单个着色器的辅助函数
static GLuint compileShader(GLenum type, const char* source) {
    GLuint shader = glCreateShader(type);
    glShaderSource(shader, 1, &source, nullptr);
    glCompileShader(shader);

    GLint success;
    glGetShaderiv(shader, GL_COMPILE_STATUS, &success);
    if (!success) {
        char infoLog[1024];
        glGetShaderInfoLog(shader, sizeof(infoLog), nullptr, infoLog);
        printf("[着色器编译错误] 类型=%s\n%s\n",
               type == GL_VERTEX_SHADER ? "顶点" : "片段", infoLog);
        glDeleteShader(shader);
        return 0;
    }
    return shader;
}

// 创建着色器程序：编译+链接
GLuint createShaderProgram(const char* vertSrc, const char* fragSrc) {
    GLuint vert = compileShader(GL_VERTEX_SHADER, vertSrc);
    if (!vert) return 0;

    GLuint frag = compileShader(GL_FRAGMENT_SHADER, fragSrc);
    if (!frag) {
        glDeleteShader(vert);
        return 0;
    }

    GLuint program = glCreateProgram();
    glAttachShader(program, vert);
    glAttachShader(program, frag);
    glLinkProgram(program);

    // 链接后着色器对象可以删除（已被程序引用）
    glDeleteShader(vert);
    glDeleteShader(frag);

    GLint linkSuccess;
    glGetProgramiv(program, GL_LINK_STATUS, &linkSuccess);
    if (!linkSuccess) {
        char infoLog[1024];
        glGetProgramInfoLog(program, sizeof(infoLog), nullptr, infoLog);
        printf("[着色器链接错误]\n%s\n", infoLog);
        glDeleteProgram(program);
        return 0;
    }
    return program;
}
```

注意 `glDeleteShader` 在链接成功后可以安全调用——着色器对象会在程序被删除时才真正释放。

</details>

### 题目 2：写一个跨平台着色器源码生成器

参考 KrKr2 的 `#ifdef GL_ES` 做法，编写一个 C++ 函数，接收片段着色器的 body 部分（不含版本声明和精度声明），自动生成兼容 Desktop GL 和 ES 2.0 的完整片段着色器源码。

<details>
<summary>查看答案</summary>

```cpp
#include <string>
#include <sstream>

// 生成兼容 Desktop GL / ES 2.0 的片段着色器
// body 参数示例："gl_FragColor = texture2D(tex0, vTexCoord) * vColor;"
// nTextures：使用的纹理数量
std::string generateFragmentShader(const std::string& body, int nTextures) {
    std::ostringstream ss;

    // 条件编译：ES 2.0 需要精度声明
    ss << "#ifdef GL_ES\n"
       << "precision mediump float;\n"
       << "#endif\n\n";

    // 声明 varying（从顶点着色器接收）
    ss << "varying vec2 vTexCoord;\n"
       << "varying vec4 vColor;\n\n";

    // 声明纹理采样器
    for (int i = 0; i < nTextures; ++i) {
        ss << "uniform sampler2D tex" << i << ";\n";
    }
    ss << "\n";

    // 主函数
    ss << "void main() {\n"
       << "    " << body << "\n"
       << "}\n";

    return ss.str();
}

// 使用示例：
// auto src = generateFragmentShader(
//     "gl_FragColor = texture2D(tex0, vTexCoord) * vColor;", 1);
```

注意使用 `varying` 而非 `in`，以及 `texture2D` 而非 `texture`。Desktop GL 在兼容模式下也接受这些旧语法，因此一套代码可以在两个平台运行。

</details>

### 题目 3：解释 KrKr2 着色器代码中的 `GL_ES` 条件编译

阅读以下 KrKr2 源码片段，解释每一行的作用，并说明为什么 Desktop GL 上运行也不会出错。

```cpp
str << "#ifdef GL_ES\n"
       "precision mediump float;\n"
       "#endif\n";
```

<details>
<summary>查看答案</summary>

逐行解释：

1. `#ifdef GL_ES` — GLSL 预处理指令。`GL_ES` 是 OpenGL ES 着色器编译器**自动定义**的宏，Desktop GL 编译器**不会定义**这个宏。因此这行相当于"如果当前运行在 ES 环境"。

2. `precision mediump float;` — 设置片段着色器中 `float` 类型的默认精度为 `mediump`（中等精度，16 位浮点）。ES 2.0 的片段着色器**必须**声明精度，否则编译报错。Desktop GL 的片段着色器中 `float` 默认为 `highp`（高精度，32 位），不需要也不推荐声明。

3. `#endif` — 结束条件编译块。

**为什么 Desktop GL 上不会出错？** 因为 Desktop GL 编译器不定义 `GL_ES` 宏，所以 `#ifdef GL_ES` 和 `#endif` 之间的代码会被预处理器完全跳过，就像不存在一样。实际上 Desktop GL 即使遇到 `precision mediump float;` 也不会报错（它会被当作合法但无效的声明忽略），但用 `#ifdef` 包裹是更规范的做法。

</details>
