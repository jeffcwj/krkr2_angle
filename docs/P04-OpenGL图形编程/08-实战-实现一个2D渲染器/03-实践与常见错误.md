## 对照项目源码

### KrKr2 的纹理管理（`tTVPOGLTexture2D` 类）

**文件**：`cpp/core/visual/ogl/RenderManager_ogl.cpp` 第 862-961 行

KrKr2 的纹理类比我们的简化版本复杂得多：

```cpp
// KrKr2 纹理创建（简化摘录）
class tTVPOGLTexture2D : public iTVPTexture2D {
    GLuint texture = 0;         // GL 纹理 ID
    TVPTextureFormat::e Format; // RGBA/RGB/Gray
    unsigned int internalW, internalH; // 实际分配尺寸

    // 构造时即创建 GL 纹理对象
    tTVPOGLTexture2D(unsigned int w, unsigned int h,
                     TVPTextureFormat::e format,
                     GLint mode = GL_LINEAR) {
        glGenTextures(1, &texture);
        cocos2d::GL::bindTexture2D(texture);
        glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, mode);
        glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, mode);
        glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, GL_CLAMP_TO_EDGE);
        glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, GL_CLAMP_TO_EDGE);
    }
};
```

**与我们的实现对比**：
- KrKr2 使用 `cocos2d::GL::bindTexture2D` 而非直接 `glBindTexture`，因为 Cocos2d-x 有纹理绑定缓存，避免重复绑定
- KrKr2 支持 Gray（单通道）、RGB、RGBA 三种格式，我们的 `loadTexture` 同样通过 `channels` 判断
- KrKr2 跟踪 `_totalVMemSize` 来监控显存使用量

### KrKr2 的 FBO 渲染目标切换

**文件**：`cpp/core/visual/ogl/RenderManager_ogl.cpp` 第 469-518 行

KrKr2 的 `TVPSetRenderTarget` 函数（详见 Ch5 分析）使用单个共享 FBO + 动态纹理附件切换，这是一种比为每个纹理创建独立 FBO 更高效的设计。

### KrKr2 的 YUV 精灵渲染

**文件**：`cpp/core/environ/cocos2d/YUVSprite.cpp` 第 1-264 行

YUVSprite 展示了一个完整的自定义精灵渲染流程：
- 使用 3 个纹理单元（Y/U/V 分别绑到 `GL_TEXTURE0/1/2`）
- 自定义 GLSL 着色器做 YUV→RGB 色彩空间转换
- 手动管理 VAO/VBO（在支持的平台上）
- 这与我们的渲染器结构非常相似：着色器 + 纹理 + VAO/VBO + Draw Call

### 像素对齐设置

**文件**：`cpp/core/visual/ogl/RenderManager_ogl.cpp` 第 934-950 行

```cpp
// KrKr2 根据 pitch（行字节数）动态设置对齐
switch (pitch & 7) {
    case 0: glPixelStorei(GL_UNPACK_ALIGNMENT, 8); break;
    case 2: glPixelStorei(GL_UNPACK_ALIGNMENT, 2); break;
    case 4: glPixelStorei(GL_UNPACK_ALIGNMENT, 4); break;
    case 6: glPixelStorei(GL_UNPACK_ALIGNMENT, 2); break;
    default: glPixelStorei(GL_UNPACK_ALIGNMENT, 1); break;
}
```

这比我们简化版的 `if (channels == 3) alignment = 1` 更精确：KrKr2 根据实际数据行的字节对齐情况选择最优的 `GL_UNPACK_ALIGNMENT`，减少 GPU 数据传输时的对齐填充。

## 本节小结

- 2D 精灵渲染器由四个核心组件构成：着色器、纹理管理、批处理器、投影矩阵
- 正交投影矩阵将像素坐标映射到 NDC，实现屏幕坐标系下的 2D 渲染
- 批处理是性能优化的关键：将相同纹理的精灵合并，减少 Draw Call
- 索引缓冲（EBO）的模式固定（每精灵 6 个索引），只需初始化一次
- 精灵变换（平移/缩放/旋转）在 CPU 端完成，直接修改顶点坐标
- Alpha 混合需要启用 `GL_BLEND` 并设置混合函数
- KrKr2 使用 Cocos2d-x 封装层管理纹理和渲染状态，底层仍是标准 OpenGL 调用
- 生产级渲染器还需要处理：显存监控、大纹理分片、压缩纹理、帧缓冲管理等

## 常见错误及解决方案

### 错误一：精灵显示为黑色矩形

**原因**：纹理加载失败或未正确绑定。

**排查步骤**：
1. 检查 `loadTexture` 返回值是否为 0
2. 确认图片文件路径正确（相对于工作目录）
3. 确认 `stbi_set_flip_vertically_on_load(true)` 已调用
4. 确认着色器中的 sampler uniform 已设置（`glUniform1i(loc, 0)`）

### 错误二：Alpha 透明不生效

**原因**：未启用 GL 混合或混合函数设置错误。

**解决**：
```cpp
glEnable(GL_BLEND);
glBlendFunc(GL_SRC_ALPHA, GL_ONE_MINUS_SRC_ALPHA);
// 确保在绘制所有半透明物体之前调用
```

**注意**：2D 渲染中一般不启用深度测试（`GL_DEPTH_TEST`），否则半透明精灵的渲染顺序会出问题。如果需要深度测试，必须从后往前排序（画家算法）。

### 错误三：批处理时精灵纹理混乱

**原因**：纹理切换时未 flush 之前批次。

**排查**：确保 `drawSprite` 在检测到纹理切换时调用 `flush()`。我们上面的实现已经包含了这个检查。如果手动操作，务必在绑定新纹理前刷新旧批次。

### 题目 1：分析批处理效率

假设一个场景包含以下精灵（按绘制顺序）：

| 序号 | 精灵 | 纹理 |
|------|------|------|
| 1 | 背景 | tex_A |
| 2 | 立绘1 | tex_B |
| 3 | 立绘2 | tex_B |
| 4 | 文本框 | tex_C |
| 5 | 按钮1 | tex_C |
| 6 | 按钮2 | tex_C |
| 7 | 光标 | tex_A |

问：使用我们的 SpriteBatcher，需要几次 Draw Call？如何通过调整绘制顺序减少 Draw Call？

<details>
<summary>查看答案</summary>

**原始顺序的 Draw Call 分析**：
1. 精灵 1 (tex_A) → 批次开始
2. 精灵 2 (tex_B) → 纹理切换，flush → **Draw Call 1** (tex_A, 1个精灵)
3. 精灵 3 (tex_B) → 同纹理，合入批次
4. 精灵 4 (tex_C) → 纹理切换，flush → **Draw Call 2** (tex_B, 2个精灵)
5. 精灵 5,6 (tex_C) → 同纹理，合入批次
6. 精灵 7 (tex_A) → 纹理切换，flush → **Draw Call 3** (tex_C, 3个精灵)
7. 渲染循环结束，最终 flush → **Draw Call 4** (tex_A, 1个精灵)

**总计 4 次 Draw Call**。

**优化方案**：按纹理分组绘制：
- 先画 tex_A（背景 + 光标）：1 次 Draw Call
- 再画 tex_B（立绘1 + 立绘2）：1 次 Draw Call
- 最后画 tex_C（文本框 + 按钮1 + 按钮2）：1 次 Draw Call

**优化后 3 次 Draw Call**。但要注意：重新排序可能影响遮挡关系（光标应该在最上层）。实际中通常使用**纹理图集（Texture Atlas）** 将所有 UI 元素打包到一张大纹理中，实现真正的单次 Draw Call。

</details>

### 题目 2：实现精灵闪烁效果

使用顶点颜色实现一个精灵的"受击闪烁"效果：精灵在正常颜色和红色之间快速切换。写出关键代码。

<details>
<summary>查看答案</summary>

```cpp
// 在渲染循环中
float time = (float)glfwGetTime();
// 每 0.1 秒切换一次，持续 0.5 秒
bool isHit = (time - hitStartTime) < 0.5f;
float r = 1.0f, g = 1.0f, b = 1.0f;

if (isHit) {
    // 用 sin 函数产生 10Hz 的闪烁
    float flash = sinf((time - hitStartTime) * 10.0f * 3.14159f);
    if (flash > 0.0f) {
        // 闪烁帧：红色调制
        g = 0.3f;
        b = 0.3f;
    }
    // flash <= 0 时保持原色
}

batcher.drawSprite(charTex,
    posX, posY, charW, charH,
    0.0f, 0.0f, 1.0f, 1.0f,
    r, g, b, 1.0f);  // 颜色调制
```

原理：当 `vColor = (1, 0.3, 0.3, 1)` 时，片段着色器中 `texture * vColor` 会让绿色和蓝色通道变暗，红色通道保持不变，视觉上呈现红色闪烁效果。

</details>

### 题目 3：支持精灵图集（Sprite Sheet）

假设有一张 512×512 的精灵图集，包含 4×4 = 16 个均匀分布的 128×128 子精灵。编写一个函数，给定精灵索引（0-15），计算其 UV 坐标并调用 `drawSprite`。

<details>
<summary>查看答案</summary>

```cpp
// 精灵图集参数
const int ATLAS_SIZE = 512;    // 图集总尺寸
const int SPRITE_SIZE = 128;   // 单个精灵尺寸
const int COLS = 4;            // 每行精灵数
const int ROWS = 4;            // 行数

void drawSpriteFromAtlas(SpriteBatcher& batcher,
                         GLuint atlasTex,
                         int spriteIndex,
                         float screenX, float screenY,
                         float drawW, float drawH) {
    // 计算精灵在图集中的行列位置
    int col = spriteIndex % COLS;  // 列（0-3）
    int row = spriteIndex / COLS;  // 行（0-3）

    // 计算 UV 坐标（归一化到 0-1）
    float u0 = (float)(col * SPRITE_SIZE) / ATLAS_SIZE;
    float v0 = (float)(row * SPRITE_SIZE) / ATLAS_SIZE;
    float u1 = (float)((col + 1) * SPRITE_SIZE) / ATLAS_SIZE;
    float v1 = (float)((row + 1) * SPRITE_SIZE) / ATLAS_SIZE;

    // 例如 spriteIndex=5:
    //   col=1, row=1
    //   u0=0.25, v0=0.25, u1=0.5, v1=0.5

    batcher.drawSprite(atlasTex,
        screenX, screenY, drawW, drawH,
        u0, v0, u1, v1,
        1.0f, 1.0f, 1.0f, 1.0f);
}

// 使用示例：绘制行走动画（帧 0-3 循环）
int frame = ((int)(glfwGetTime() * 8.0f)) % 4; // 8 FPS 动画
drawSpriteFromAtlas(batcher, walkAtlas, frame,
                    playerX, playerY, 128, 128);
```

关键点：
- UV 坐标通过 `(像素位置 / 图集总尺寸)` 归一化到 `[0, 1]`
- 精灵图集 + 批处理可以实现单次 Draw Call 绘制所有 UI 元素
- 动画通过按时间切换 `spriteIndex` 实现

</details>
