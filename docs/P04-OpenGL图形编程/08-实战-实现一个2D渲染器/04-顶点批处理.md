## 什么是批处理

### Draw Call 的开销

每调用一次 `glDrawArrays` 或 `glDrawElements`，GPU 驱动都需要执行一系列固定开销的操作：验证当前 OpenGL 状态是否合法、将命令打包并提交到 GPU 命令队列、等待 GPU 调度执行。这些操作本身与你绘制了多少个三角形无关——画 2 个三角形和画 2000 个三角形的驱动开销几乎相同。这一次完整的"驱动验证→命令提交→GPU执行"流程叫做一次 **Draw Call**（绘制调用）。

在视觉小说引擎中，一个典型场景包含：1 个背景层、2-3 个角色立绘、1 个文本框、若干按钮和特效粒子，加起来可能有 30-50 个独立精灵（Sprite，即一个带纹理的矩形）。如果每个精灵各调用一次 `glDrawArrays`，就产生 30-50 次 Draw Call。在移动设备上，100 次以上的 Draw Call 就会造成明显卡顿。

### 批处理的核心思想

**把多个精灵的顶点数据合并到同一个 VBO 中，用一次 Draw Call 绘制全部**。前提是这些精灵使用相同的着色器和纹理——因为切换着色器或纹理本身就需要新的 Draw Call。

| 方案 | Draw Call 数 | 性能 |
|------|-------------|------|
| 逐精灵绘制 | N（精灵数量） | 差（N=50 时移动端卡顿） |
| 按纹理分组批处理 | ≤ 纹理种类数 | 好（通常 5-10 次） |
| 纹理图集 + 批处理 | 1 | 最优 |

> **纹理图集**（Texture Atlas）：把多张小图片合并到一张大图上，通过 UV 坐标选取子区域。这样所有精灵共用同一张纹理，可以放进同一个批次。本节先实现按纹理分组的批处理，纹理图集在练习题中讨论。

## 精灵顶点结构

每个精灵是一个四边形（Quad），由 4 个顶点和 6 个索引组成。我们为每个顶点定义如下结构体：

```cpp
// 单个顶点的数据布局
struct SpriteVertex {
    float x, y;       // 屏幕坐标（像素），由正交投影矩阵转换到 NDC
    float u, v;       // 纹理坐标（0.0~1.0），用于从纹理中采样颜色
    float r, g, b, a; // 顶点颜色（含透明度），与纹理颜色相乘实现色调调制
};
// 总大小 = (2+2+4) × 4 字节 = 32 字节/顶点
```

四个顶点的排列和索引关系：

```
v0 ---- v1        索引: {0, 1, 2,  2, 3, 0}
|  \     |        三角形1: v0 → v1 → v2
|    \   |        三角形2: v2 → v3 → v0
v3 ---- v2
```

GPU 只能绘制三角形，一个四边形需要拆成 2 个三角形。使用索引缓冲（EBO）可以共享 v0 和 v2，比用 6 个独立顶点省 33% 的顶点数据。

## 关键 GL 函数详解

### glBufferData — 分配缓冲区

```cpp
void glBufferData(GLenum target,       // 缓冲区类型：GL_ARRAY_BUFFER 或 GL_ELEMENT_ARRAY_BUFFER
                  GLsizeiptr size,      // 要分配的字节数
                  const void* data,     // 初始数据指针，传 nullptr 表示只分配不填充
                  GLenum usage);        // 使用提示，告诉驱动如何优化存储
```

`usage` 参数是对驱动的**优化提示**（不是硬性约束），常用值：

| 值 | 含义 | 适用场景 |
|----|------|----------|
| `GL_STATIC_DRAW` | 数据设置一次，多次绘制 | 固定不变的网格、索引数据 |
| `GL_DYNAMIC_DRAW` | 数据频繁更新，多次绘制 | 批处理器的顶点缓冲（每帧更新） |
| `GL_STREAM_DRAW` | 数据每帧全部替换 | 粒子系统、即时模式模拟 |

**重要**：`glBufferData` 会重新分配整块显存。如果只需要更新部分数据，用 `glBufferSubData` 更高效。

### glBufferSubData — 局部更新缓冲区

```cpp
void glBufferSubData(GLenum target,     // GL_ARRAY_BUFFER 等
                     GLintptr offset,    // 起始偏移（字节），从缓冲区头部算起
                     GLsizeiptr size,    // 要更新的字节数
                     const void* data);  // CPU 端数据指针
```

**为什么用它而不是 `glBufferData`？** 因为 `glBufferData` 每次调用都会重新分配显存，而 `glBufferSubData` 只把数据写入已有的缓冲区，不触发重新分配。在批处理器中，我们在初始化时用 `glBufferData` 分配一块足够大的 VBO，之后每帧只用 `glBufferSubData` 更新实际用到的部分：

```cpp
// 初始化时：分配最大容量，data 传 nullptr
glBufferData(GL_ARRAY_BUFFER,
             sizeof(SpriteVertex) * MAX_VERTICES,
             nullptr, GL_DYNAMIC_DRAW);

// 每帧：只更新实际用到的部分
glBufferSubData(GL_ARRAY_BUFFER, 0,
                sizeof(SpriteVertex) * actualVertexCount,
                vertices);
```

### glDrawElements — 索引绘制

```cpp
void glDrawElements(GLenum mode,        // 图元类型：GL_TRIANGLES, GL_LINES 等
                    GLsizei count,       // 要绘制的索引数量（不是三角形数）
                    GLenum type,         // 索引数据类型：GL_UNSIGNED_INT 或 GL_UNSIGNED_SHORT
                    const void* indices); // EBO 绑定时传偏移量（通常为 0）
```

与 `glDrawArrays` 的区别：`glDrawArrays` 按顺序读取顶点，`glDrawElements` 按索引数组指定的顺序读取。对于四边形批处理，索引模式是固定的（`{0,1,2, 2,3,0, 4,5,6, 6,7,4, ...}`），所以索引缓冲只需初始化一次。

```cpp
// 绘制 spriteCount 个精灵，每个精灵 6 个索引
glDrawElements(GL_TRIANGLES,
               spriteCount * 6,      // 总索引数
               GL_UNSIGNED_INT,      // 索引类型
               0);                   // EBO 偏移量
```

> **GL_UNSIGNED_SHORT vs GL_UNSIGNED_INT**：`GL_UNSIGNED_SHORT`（2字节）最多索引 65535 个顶点（约 16383 个精灵），对大多数 2D 场景足够且更省显存。`GL_UNSIGNED_INT`（4字节）无此限制但占用翻倍。本教程用 `GL_UNSIGNED_INT` 简化代码，生产环境建议用 `GL_UNSIGNED_SHORT`。

### glActiveTexture — 激活纹理单元

```cpp
void glActiveTexture(GLenum texture);  // GL_TEXTURE0, GL_TEXTURE1, ..., GL_TEXTURE31
```

OpenGL 支持多个纹理单元（Texture Unit），每个单元可以绑定一张纹理。`glActiveTexture` 切换"当前活跃"的纹理单元，之后的 `glBindTexture` 会绑定到这个单元上。在片段着色器中，`sampler2D` uniform 的值就是纹理单元编号（0 对应 `GL_TEXTURE0`）。

```cpp
// 单纹理渲染（最常见的情况）
glActiveTexture(GL_TEXTURE0);                // 激活 0 号单元
glBindTexture(GL_TEXTURE_2D, myTexture);     // 绑定纹理到 0 号单元
glUniform1i(texUniformLoc, 0);               // 告诉着色器：uTexture 采样 0 号单元

// 多纹理渲染（如 YUV→RGB 转换）
glActiveTexture(GL_TEXTURE0);
glBindTexture(GL_TEXTURE_2D, yTexture);      // Y 分量
glActiveTexture(GL_TEXTURE1);
glBindTexture(GL_TEXTURE_2D, uTexture);      // U 分量
glActiveTexture(GL_TEXTURE2);
glBindTexture(GL_TEXTURE_2D, vTexture);      // V 分量
```

> KrKr2 的 `YUVSprite.cpp` 正是用 3 个纹理单元分别绑定 Y/U/V 数据，在着色器中完成色彩空间转换。

## 实现 SpriteBatcher

现在把以上函数组合起来，实现一个完整的批处理器：

```cpp
#include <cstring>   // memcpy
#include <iostream>

// 最大批处理精灵数（可根据需求调整）
const int MAX_SPRITES  = 1000;
const int MAX_VERTICES = MAX_SPRITES * 4;  // 每精灵 4 顶点
const int MAX_INDICES  = MAX_SPRITES * 6;  // 每精灵 6 索引

class SpriteBatcher {
private:
    GLuint vao, vbo, ebo;                  // OpenGL 对象
    SpriteVertex vertices[MAX_VERTICES];   // CPU 端顶点缓冲
    int spriteCount;                       // 当前批次已提交的精灵数
    GLuint currentTexture;                 // 当前批次绑定的纹理

public:
    SpriteBatcher() : spriteCount(0), currentTexture(0) {
        // ---- 创建 VAO ----
        glGenVertexArrays(1, &vao);
        glBindVertexArray(vao);

        // ---- 创建 VBO（动态更新）----
        glGenBuffers(1, &vbo);
        glBindBuffer(GL_ARRAY_BUFFER, vbo);
        // 预分配最大容量，data=nullptr 表示只分配不填充
        glBufferData(GL_ARRAY_BUFFER,
                     sizeof(SpriteVertex) * MAX_VERTICES,
                     nullptr, GL_DYNAMIC_DRAW);

        // ---- 设置顶点属性布局 ----
        // location=0: 位置 (vec2, 偏移 0 字节)
        glEnableVertexAttribArray(0);
        glVertexAttribPointer(0, 2, GL_FLOAT, GL_FALSE,
                              sizeof(SpriteVertex),
                              (void*)offsetof(SpriteVertex, x));
        // location=1: 纹理坐标 (vec2, 偏移 8 字节)
        glEnableVertexAttribArray(1);
        glVertexAttribPointer(1, 2, GL_FLOAT, GL_FALSE,
                              sizeof(SpriteVertex),
                              (void*)offsetof(SpriteVertex, u));
        // location=2: 颜色 (vec4, 偏移 16 字节)
        glEnableVertexAttribArray(2);
        glVertexAttribPointer(2, 4, GL_FLOAT, GL_FALSE,
                              sizeof(SpriteVertex),
                              (void*)offsetof(SpriteVertex, r));

        // ---- 创建 EBO（索引模式固定，只初始化一次）----
        glGenBuffers(1, &ebo);
        GLuint indices[MAX_INDICES];
        for (int i = 0; i < MAX_SPRITES; i++) {
            int vi = i * 4;  // 第 i 个精灵的顶点起始索引
            int ii = i * 6;  // 第 i 个精灵的索引起始位置
            indices[ii + 0] = vi + 0;  // 三角形 1: v0→v1→v2
            indices[ii + 1] = vi + 1;
            indices[ii + 2] = vi + 2;
            indices[ii + 3] = vi + 2;  // 三角形 2: v2→v3→v0
            indices[ii + 4] = vi + 3;
            indices[ii + 5] = vi + 0;
        }
        glBindBuffer(GL_ELEMENT_ARRAY_BUFFER, ebo);
        // 索引数据不会变化，用 GL_STATIC_DRAW
        glBufferData(GL_ELEMENT_ARRAY_BUFFER,
                     sizeof(indices), indices, GL_STATIC_DRAW);

        glBindVertexArray(0);
    }

    ~SpriteBatcher() {
        glDeleteVertexArrays(1, &vao);
        glDeleteBuffers(1, &vbo);
        glDeleteBuffers(1, &ebo);
    }

    // 提交一个精灵到当前批次
    void drawSprite(GLuint texture,
                    float x, float y, float w, float h,
                    float u0, float v0, float u1, float v1,
                    float r, float g, float b, float a) {
        // 纹理切换 → 先刷新当前批次再开始新批次
        if (texture != currentTexture && spriteCount > 0) {
            flush();
        }
        // 批次满了 → 强制刷新
        if (spriteCount >= MAX_SPRITES) {
            flush();
        }
        currentTexture = texture;

        // 填充 4 个顶点（CPU 端数组，还没上传到 GPU）
        int base = spriteCount * 4;
        vertices[base + 0] = {x,     y,     u0, v0, r, g, b, a};  // 左上
        vertices[base + 1] = {x + w, y,     u1, v0, r, g, b, a};  // 右上
        vertices[base + 2] = {x + w, y + h, u1, v1, r, g, b, a};  // 右下
        vertices[base + 3] = {x,     y + h, u0, v1, r, g, b, a};  // 左下

        spriteCount++;
    }

    // 刷新批次：把 CPU 端数据上传到 GPU 并绘制
    void flush() {
        if (spriteCount == 0) return;

        glBindVertexArray(vao);

        // 用 glBufferSubData 只上传本批次实际用到的顶点
        glBindBuffer(GL_ARRAY_BUFFER, vbo);
        glBufferSubData(GL_ARRAY_BUFFER, 0,
                        sizeof(SpriteVertex) * spriteCount * 4,
                        vertices);

        // 绑定当前批次的纹理到 0 号纹理单元
        glActiveTexture(GL_TEXTURE0);
        glBindTexture(GL_TEXTURE_2D, currentTexture);

        // 索引绘制：每个精灵 6 个索引
        glDrawElements(GL_TRIANGLES,
                       spriteCount * 6,
                       GL_UNSIGNED_INT, 0);

        spriteCount = 0;  // 重置，准备下一批次
        glBindVertexArray(0);
    }
};
```

### 批处理工作流程图

```
应用层调用 drawSprite()
        │
        ▼
┌─ 纹理与上一次相同？──┐
│  是                    │  否
│  ▼                    ▼
│  追加到 vertices[]    先 flush() 当前批次
│                       再追加到 vertices[]
│                              │
└──────────┬───────────────────┘
           ▼
   spriteCount >= MAX_SPRITES?
     是 → flush() 强制刷新
     否 → 等待更多精灵
           │
           ▼
   帧结束时调用 flush()
   ── glBufferSubData 上传数据
   ── glDrawElements 绘制
   ── spriteCount 归零
```

## 对照项目源码：KrKr2 的渲染方式

**文件**：`cpp/core/visual/ogl/RenderManager_ogl.cpp` 第 2345-2353 行

KrKr2 使用 `glDrawArrays(GL_TRIANGLES, 0, 6)` 直接绘制 6 个顶点（2 个三角形组成一个矩形），**没有使用索引缓冲（EBO）和批处理**：

```cpp
// KrKr2 的绘制方式（简化）
glVertexAttribPointer(method->GetPosAttr(), 2, GL_FLOAT, GL_FALSE, 0,
                      vertices);            // 直接传顶点数组指针
glVertexAttribPointer(method->GetTexCoordAttr(0), 2, GL_FLOAT, GL_FALSE,
                      0, &vtx.vtx.front()); // 直接传纹理坐标数组指针
glDrawArrays(GL_TRIANGLES, 0, 6);           // 画 6 个顶点 = 2 个三角形
```

**为什么 KrKr2 不做批处理？** 因为 KrKr2 的渲染系统由 Cocos2d-x 管理。Cocos2d-x 在更上层做了渲染命令排序和批处理（`Renderer::render()` → `GroupCommand` → `BatchCommand`），KrKr2 只负责每个矩形的绘制逻辑。这是"引擎做批处理 vs 应用做批处理"的典型分工。

## 常见错误及解决方案

### 错误一：`glBufferSubData` 导致崩溃或 GL 错误

**原因**：写入的字节数超过了 `glBufferData` 分配的容量。

**解决**：确保 `spriteCount * 4 * sizeof(SpriteVertex)` 不超过初始化时分配的 `MAX_VERTICES * sizeof(SpriteVertex)`。在 `drawSprite` 中检查 `spriteCount >= MAX_SPRITES` 时及时 `flush()`。

```cpp
// 防御性检查：永远在 flush 之前验证
if (spriteCount >= MAX_SPRITES) {
    flush();  // 如果没有这行，下面的顶点写入会越界
}
```

### 错误二：批处理时部分精灵纹理错乱

**原因**：纹理切换时没有先刷新之前的批次。

**解决**：确保 `drawSprite` 在检测到 `texture != currentTexture` 时立即调用 `flush()`。漏掉这个检查会导致用新纹理绘制旧批次的顶点——所有旧精灵突然变成新纹理的外观。

### 错误三：EBO 索引数据不正确导致三角形交叉

**原因**：索引生成循环中 `vi` 和 `ii` 的偏移量计算错误。

**解决**：仔细检查每个精灵的顶点基索引 `vi = i * 4` 和索引基位置 `ii = i * 6`。一个常见错误是写成 `vi = i * 6`（把索引偏移当成顶点偏移）。如果出现三角形交叉或闪烁，先打印前几个精灵的索引值验证模式是否正确。

## 本节小结

- **Draw Call** 的开销与三角形数无关，与调用次数相关——减少 Draw Call 是 2D 渲染优化的关键
- **批处理器**将同纹理的精灵合并到一个 VBO，用一次 `glDrawElements` 绘制
- `glBufferData` 分配缓冲区（一次），`glBufferSubData` 局部更新（每帧）——避免重新分配
- **索引缓冲**（EBO）的模式固定（每精灵 `{0,1,2, 2,3,0}`），只需初始化一次
- KrKr2 不自己做批处理，而是依赖 Cocos2d-x 的渲染管线统一管理

### 题目 1：分析 Draw Call 数量

假设一个场景按以下顺序绘制精灵：

| 序号 | 精灵 | 纹理 |
|------|------|------|
| 1 | 背景 | tex_A |
| 2 | 立绘1 | tex_B |
| 3 | 立绘2 | tex_B |
| 4 | 文本框 | tex_C |
| 5 | 按钮1 | tex_C |
| 6 | 按钮2 | tex_C |
| 7 | 光标 | tex_A |

问：(1) 使用 SpriteBatcher 需要几次 Draw Call？(2) 如何优化？

<details>
<summary>查看答案</summary>

**(1) 4 次 Draw Call**：
1. 精灵 1 (tex_A) → 批次开始
2. 精灵 2 (tex_B) → 纹理切换，flush → **DC 1** (tex_A, 1 精灵)
3. 精灵 3 (tex_B) → 同纹理，合并
4. 精灵 4 (tex_C) → 纹理切换，flush → **DC 2** (tex_B, 2 精灵)
5. 精灵 5,6 (tex_C) → 同纹理，合并
6. 精灵 7 (tex_A) → 纹理切换，flush → **DC 3** (tex_C, 3 精灵)
7. 帧结束 flush → **DC 4** (tex_A, 1 精灵)

**(2) 优化方案**：
- **按纹理排序**：先画 tex_A（背景+光标 → 1 DC），再 tex_B（2 DC），再 tex_C（3 DC），总计 3 次。但需注意遮挡顺序——光标必须在最上层
- **纹理图集**：把所有 UI 元素打包到一张大纹理，UV 坐标选取子区域，实现 1 次 Draw Call
- 实际中两者结合：使用纹理图集 + Z 排序的深度缓冲

</details>

### 题目 2：实现 `GL_UNSIGNED_SHORT` 索引

将 SpriteBatcher 的索引类型从 `GL_UNSIGNED_INT` 改为 `GL_UNSIGNED_SHORT`，写出需要修改的代码。最多支持多少个精灵？

<details>
<summary>查看答案</summary>

```cpp
// 修改 1：索引数组类型从 GLuint 改为 GLushort
GLushort indices[MAX_INDICES];
for (int i = 0; i < MAX_SPRITES; i++) {
    int vi = i * 4;
    int ii = i * 6;
    indices[ii + 0] = (GLushort)(vi + 0);
    indices[ii + 1] = (GLushort)(vi + 1);
    indices[ii + 2] = (GLushort)(vi + 2);
    indices[ii + 3] = (GLushort)(vi + 2);
    indices[ii + 4] = (GLushort)(vi + 3);
    indices[ii + 5] = (GLushort)(vi + 0);
}

// 修改 2：glDrawElements 的索引类型参数
glDrawElements(GL_TRIANGLES,
               spriteCount * 6,
               GL_UNSIGNED_SHORT,  // 原来是 GL_UNSIGNED_INT
               0);
```

**最大精灵数**：`GLushort` 最大值 = 65535。每精灵 4 个顶点，最大顶点索引 = `MAX_SPRITES * 4 - 1 ≤ 65535`，所以 `MAX_SPRITES ≤ 16383`。

**优势**：索引缓冲大小减半（每个索引 2 字节 vs 4 字节），在移动设备上更友好。16383 个精灵对 2D 场景绰绰有余。

</details>

### 题目 3：双缓冲 VBO 优化

当前实现中，`flush()` 调用 `glBufferSubData` 更新 VBO 后立即 `glDrawElements`。如果 GPU 还在使用上一帧的 VBO 数据，`glBufferSubData` 会导致 GPU 等待（称为"管线阻塞"）。请设计一个双缓冲 VBO 方案来避免这个问题。

<details>
<summary>查看答案</summary>

```cpp
class SpriteBatcher {
private:
    GLuint vao[2], vbo[2];  // 两套 VBO
    GLuint ebo;              // EBO 共享（索引不变）
    int currentBuffer = 0;   // 当前使用的缓冲区索引（0 或 1）
    // ... 其余成员不变

public:
    SpriteBatcher() {
        // 创建两套 VAO/VBO
        glGenVertexArrays(2, vao);
        glGenBuffers(2, vbo);
        for (int i = 0; i < 2; i++) {
            glBindVertexArray(vao[i]);
            glBindBuffer(GL_ARRAY_BUFFER, vbo[i]);
            glBufferData(GL_ARRAY_BUFFER,
                         sizeof(SpriteVertex) * MAX_VERTICES,
                         nullptr, GL_DYNAMIC_DRAW);
            // 设置顶点属性（两套 VAO 都要设置）
            // ... 与原来相同
        }
        // EBO 只需一份
        // ...
    }

    void flush() {
        if (spriteCount == 0) return;
        // 使用当前缓冲区
        glBindVertexArray(vao[currentBuffer]);
        glBindBuffer(GL_ARRAY_BUFFER, vbo[currentBuffer]);
        glBufferSubData(GL_ARRAY_BUFFER, 0,
                        sizeof(SpriteVertex) * spriteCount * 4,
                        vertices);
        // ... 绑定纹理、绘制
        glDrawElements(GL_TRIANGLES, spriteCount * 6,
                       GL_UNSIGNED_INT, 0);
        spriteCount = 0;
        // 切换到另一个缓冲区，下一次 flush 用另一套 VBO
        currentBuffer = 1 - currentBuffer;
    }
};
```

**原理**：帧 N 写入 VBO[0] 并绘制，帧 N+1 写入 VBO[1] 并绘制。GPU 可以一边读 VBO[0]（上一帧），CPU 一边写 VBO[1]（当前帧），避免管线阻塞。
