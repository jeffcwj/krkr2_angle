## 1. 像素的本质：数字化的颜色

### 1.1 从光到数字

计算机屏幕上的每一个像素（Pixel，Picture Element）由红（R）、绿（G）、蓝（B）三个颜色通道混合而成。每个通道通常用 8 位无符号整数（0~255）表示亮度，再加上一个 Alpha（A）通道表示透明度，共 32 位（4 字节）。这就是最常见的 **32 位 ARGB 像素格式**。

```
一个像素 = 4 字节 = 32 位
┌────────┬────────┬────────┬────────┐
│  A(8)  │  R(8)  │  G(8)  │  B(8)  │
│ 透明度  │   红   │   绿    │   蓝   │
└────────┴────────┴────────┴────────┘
高位 ←────────────────────────────→ 低位
```

### 1.2 为什么是 32 位？

- **对齐友好**：32 位（4 字节）恰好是现代 CPU 的最小高效处理单元。一次 `uint32_t` 读写即可操作完整一个像素
- **SIMD 友好**：SSE2 的 128 位寄存器可一次装入 4 个像素（128 ÷ 32 = 4），AVX2 的 256 位寄存器可装 8 个
- **直接映射**：操作系统的帧缓冲（Framebuffer）通常直接以 32 位格式存储，无需格式转换即可显示

## 2. 常见像素格式与内存布局

### 2.1 格式命名规则

像素格式的名称描述**从高位到低位**的通道排列顺序。但需注意：在**小端序（Little Endian）**机器上（x86/ARM 默认），内存中的字节顺序与寄存器中的位顺序相反。

| 格式名称 | 32位值（高→低） | 内存字节序（低地址→高） | 使用场景 |
|----------|-----------------|------------------------|----------|
| **ARGB** | `AARRGGBB` | B, G, R, A | KrKr2/Windows GDI/DirectX |
| **RGBA** | `RRGGBBAA` | A, B, G, R | OpenGL/PNG 文件 |
| **BGRA** | `BBGGRRAA` | A, R, G, B | Windows Bitmap/Vulkan |
| **ABGR** | `AABBGGRR` | R, G, B, A | OpenGL ES（某些实现） |

> **关键点**：KrKr2 全程使用 **ARGB** 格式，像素类型为 `tjs_uint32`（即 `uint32_t`）。

### 2.2 C++ 中的像素表示

```cpp
#include <cstdint>

// KrKr2 风格：直接使用 uint32_t
using Pixel = uint32_t;

// 从 32 位像素中提取各通道（ARGB 格式）
inline uint8_t getAlpha(Pixel p) { return (p >> 24) & 0xFF; }  // 高 8 位
inline uint8_t getRed(Pixel p)   { return (p >> 16) & 0xFF; }  // 次高 8 位
inline uint8_t getGreen(Pixel p) { return (p >> 8)  & 0xFF; }  // 次低 8 位
inline uint8_t getBlue(Pixel p)  { return  p        & 0xFF; }  // 低 8 位

// 将四个通道合成为一个像素
inline Pixel makePixel(uint8_t a, uint8_t r, uint8_t g, uint8_t b) {
    return (static_cast<uint32_t>(a) << 24) |
           (static_cast<uint32_t>(r) << 16) |
           (static_cast<uint32_t>(g) << 8)  |
            static_cast<uint32_t>(b);
}

int main() {
    // 半透明红色：Alpha=128, R=255, G=0, B=0
    Pixel red = makePixel(128, 255, 0, 0);  // 值为 0x80FF0000
    
    printf("Alpha=%u, R=%u, G=%u, B=%u\n",
           getAlpha(red), getRed(red), getGreen(red), getBlue(red));
    // 输出：Alpha=128, R=255, G=0, B=0
    return 0;
}
```

### 2.3 KrKr2 的像素操作风格

KrKr2 的 `tvpgl.h` 中大量使用**位运算技巧**同时处理多个通道，避免逐通道提取的开销：

```cpp
// 来源：krkr2/cpp/core/visual/tvpgl.h 第 73-80 行
// 饱和加法：两个像素的对应通道相加，超过 255 则饱和为 255
static tjs_uint32 TVPSaturatedAdd(tjs_uint32 a, tjs_uint32 b) {
    // 利用并行位运算同时处理 4 个通道
    tjs_uint32 tmp = ((a & b) + (((a ^ b) >> 1) & 0x7f7f7f7f)) & 0x80808080;
    tmp = (tmp << 1) - (tmp >> 7);  // 将溢出位扩展为 0xFF 掩码
    return (a + b - tmp) | tmp;     // 溢出通道设为 0xFF
}
```

**解析**：这段代码的核心思想是"不拆分通道，用位运算并行判断每个字节是否溢出"：
1. `(a & b) + (((a ^ b) >> 1) & 0x7f7f7f7f)` — 计算每字节的平均值（近似），检测高位是否为 1
2. `& 0x80808080` — 只保留每字节的最高位（溢出标志）
3. `(tmp << 1) - (tmp >> 7)` — 将 `0x80` 扩展为 `0xFF`（溢出掩码）
4. `(a + b - tmp) | tmp` — 未溢出的通道正常相加，溢出的通道设为 `0xFF`

### 2.4 像素数组与图像缓冲区

一张 W×H 的图像在内存中通常以**行优先（Row-Major）**方式连续存储：

```cpp
// 图像缓冲区的基本结构
struct ImageBuffer {
    uint32_t* pixels;   // 像素数组（ARGB 格式）
    int width;          // 宽度（像素）
    int height;         // 高度（像素）
    int pitch;          // 行跨度（字节），通常 >= width * 4
    
    // 获取第 (x, y) 个像素
    uint32_t& at(int x, int y) {
        // pitch 以字节为单位，需要转换为像素偏移
        return *(reinterpret_cast<uint32_t*>(
            reinterpret_cast<uint8_t*>(pixels) + y * pitch) + x);
    }
};
```

> **为什么有 pitch？** 因为 GPU/OS 可能要求每行对齐到 16/32/64 字节边界。如果图像宽度为 100 像素（400 字节），pitch 可能是 416（对齐到 32 字节）。**永远用 pitch 而不是 `width * 4` 来计算行偏移。**

## 3. 预乘 Alpha（Premultiplied Alpha）

### 3.1 标准 Alpha vs 预乘 Alpha

**标准 Alpha（Straight Alpha）**存储原始颜色值：

```
像素 = (A, R, G, B) = (128, 255, 0, 0)
含义：50% 透明的纯红色
```

**预乘 Alpha（Premultiplied Alpha）**将颜色通道预先乘以 Alpha：

```
像素 = (A, R*A/255, G*A/255, B*A/255) = (128, 128, 0, 0)
含义：颜色值已经包含透明度信息
```

### 3.2 为什么预乘 Alpha 更好？

**标准 Alpha 混合公式（Porter-Duff Over）：**

```
Out.R = Src.R * Src.A / 255 + Dst.R * (255 - Src.A) / 255
Out.G = Src.G * Src.A / 255 + Dst.G * (255 - Src.A) / 255
Out.B = Src.B * Src.A / 255 + Dst.B * (255 - Src.A) / 255
```

每个通道需要 **2 次乘法 + 1 次加法 + 2 次除法**。

**预乘 Alpha 混合公式：**

```
Out.R = Src.R + Dst.R * (255 - Src.A) / 255
Out.G = Src.G + Dst.G * (255 - Src.A) / 255
Out.B = Src.B + Dst.B * (255 - Src.A) / 255
```

每个通道只需 **1 次乘法 + 1 次加法 + 1 次除法**——节省了 `Src.R * Src.A / 255` 这一步（因为已经预乘了）。

### 3.3 预乘转换代码

```cpp
#include <cstdint>

// 标准 Alpha → 预乘 Alpha
inline uint32_t toPremultiplied(uint32_t pixel) {
    uint32_t a = (pixel >> 24) & 0xFF;
    if (a == 255) return pixel;  // 完全不透明，无需处理
    if (a == 0)   return 0;     // 完全透明
    
    uint32_t r = ((pixel >> 16) & 0xFF) * a / 255;
    uint32_t g = ((pixel >> 8)  & 0xFF) * a / 255;
    uint32_t b = ( pixel        & 0xFF) * a / 255;
    
    return (a << 24) | (r << 16) | (g << 8) | b;
}

// 预乘 Alpha → 标准 Alpha（需要查表避免除零）
inline uint32_t toStraight(uint32_t pixel) {
    uint32_t a = (pixel >> 24) & 0xFF;
    if (a == 0)   return 0;
    if (a == 255) return pixel;
    
    // KrKr2 使用 TVPDivTable 查找表加速这个除法
    // TVPDivTable[a * 256 + c] ≈ c * 255 / a
    uint32_t r = ((pixel >> 16) & 0xFF) * 255 / a;
    uint32_t g = ((pixel >> 8)  & 0xFF) * 255 / a;
    uint32_t b = ( pixel        & 0xFF) * 255 / a;
    
    // 防止溢出（预乘后再反转可能略超 255）
    if (r > 255) r = 255;
    if (g > 255) g = 255;
    if (b > 255) b = 255;
    
    return (a << 24) | (r << 16) | (g << 8) | b;
}
```

### 3.4 KrKr2 的两套 Alpha 体系

KrKr2 同时支持两种 Alpha 模式，函数命名体现了这一设计：

```
TVPAlphaBlend_d    — d = dest has alpha（目标有标准 Alpha）
TVPAlphaBlend_a    — a = dest has additive-alpha（目标有预乘 Alpha）
TVPAlphaBlend_o    — o = with opacity（额外的全局透明度参数）
TVPAlphaBlend_HDA  — HDA = Hold Dest Alpha（保持目标 Alpha 不变）
```

来源：`krkr2/cpp/core/visual/tvpgl.h` 第 17-21 行的注释：
```
key to blending suffix:
d : destination has alpha
a : destination has additive-alpha
o : blend with opacity
```

## 4. 颜色空间：sRGB 与线性

### 4.1 人眼的非线性感知

人眼对暗部的亮度变化比亮部更敏感。如果用线性编码（数值翻倍=亮度翻倍），暗部的精度会严重不足（0~50 之间的微小变化人眼可以分辨，但 200~255 之间几乎分辨不出）。

**sRGB 编码**通过一条 Gamma 曲线（约 γ=2.2）将更多编码空间分配给暗部：

```
sRGB → 线性：
  if (sRGB <= 0.04045)
      线性 = sRGB / 12.92
  else
      线性 = ((sRGB + 0.055) / 1.055) ^ 2.4

线性 → sRGB：
  if (线性 <= 0.0031308)
      sRGB = 线性 * 12.92
  else
      sRGB = 1.055 * 线性^(1/2.4) - 0.055
```

### 4.2 Gamma 校正对混合的影响

在 sRGB 空间直接做 Alpha 混合会导致**中间色调偏暗**（因为非线性编码空间中的线性插值不等于真实亮度的线性插值）。正确流程是：

```
1. sRGB → 线性空间（去 Gamma）
2. 在线性空间做混合运算
3. 线性空间 → sRGB（加 Gamma）
```

但这样每次混合都需要两次 Gamma 转换，开销很大。实际工程中（包括 KrKr2），**通常直接在 sRGB 空间混合**，接受轻微的颜色偏差以换取性能。

### 4.3 KrKr2 的 Gamma 校正系统

KrKr2 提供了可配置的 Gamma 校正结构体（来源：`tvpgl.h` 第 31-44 行）：

```cpp
// Gamma 校正参数（每个通道独立调节）
typedef struct {
    float RGamma;     // R 通道 Gamma 值（0.10 ~ 9.99，默认 1.0）
    tjs_int RFloor;   // R 输出最小值（0 ~ 255）
    tjs_int RCeil;    // R 输出最大值（0 ~ 255）
    float GGamma;     // G 通道
    tjs_int GFloor;
    tjs_int GCeil;
    float BGamma;     // B 通道
    tjs_int BFloor;
    tjs_int BCeil;
} tTVPGLGammaAdjustData;

// 默认值：不做任何 Gamma 校正
// 来源：LayerBitmapIntf.cpp 第 43 行
tTVPGLGammaAdjustData TVPIntactGammaAdjustData = {
    1.0, 0, 255,   // R: gamma=1.0, floor=0, ceil=255
    1.0, 0, 255,   // G
    1.0, 0, 255    // B
};
```

### 4.4 完整 Gamma 转换实现

```cpp
#include <cmath>
#include <cstdint>

// sRGB 通道值（0~255）→ 线性浮点值（0.0~1.0）
inline float srgbToLinear(uint8_t srgb) {
    float s = srgb / 255.0f;
    if (s <= 0.04045f)
        return s / 12.92f;
    else
        return std::pow((s + 0.055f) / 1.055f, 2.4f);
}

// 线性浮点值（0.0~1.0）→ sRGB 通道值（0~255）
inline uint8_t linearToSrgb(float linear) {
    float s;
    if (linear <= 0.0031308f)
        s = linear * 12.92f;
    else
        s = 1.055f * std::pow(linear, 1.0f / 2.4f) - 0.055f;
    
    // 钳制到 [0, 255] 并四舍五入
    int result = static_cast<int>(s * 255.0f + 0.5f);
    if (result < 0) return 0;
    if (result > 255) return 255;
    return static_cast<uint8_t>(result);
}

// 在线性空间做正确的 Alpha 混合
uint32_t blendLinear(uint32_t src, uint32_t dst) {
    float sa = ((src >> 24) & 0xFF) / 255.0f;  // 源 Alpha
    
    // 提取并转换到线性空间
    float sr = srgbToLinear((src >> 16) & 0xFF);
    float sg = srgbToLinear((src >> 8)  & 0xFF);
    float sb = srgbToLinear( src        & 0xFF);
    
    float dr = srgbToLinear((dst >> 16) & 0xFF);
    float dg = srgbToLinear((dst >> 8)  & 0xFF);
    float db = srgbToLinear( dst        & 0xFF);
    
    // 线性空间混合
    float inv_sa = 1.0f - sa;
    float or_ = sr * sa + dr * inv_sa;
    float og  = sg * sa + dg * inv_sa;
    float ob  = sb * sa + db * inv_sa;
    
    // 转回 sRGB
    return (0xFF << 24) |
           (linearToSrgb(or_) << 16) |
           (linearToSrgb(og) << 8) |
            linearToSrgb(ob);
}
```

## 5. 查找表（LUT）优化

### 5.1 为什么需要查找表？

`c * 255 / a` 这样的整数除法在 CPU 上非常昂贵（比乘法慢 20~40 倍）。KrKr2 使用预计算查找表避免运行时除法：

```cpp
// 来源：krkr2/cpp/core/visual/tvpgl.cpp 第 29-45 行

// TVPDivTable[a * 256 + b] = b * 255 / a（256KB 查找表）
unsigned char TVPDivTable[256 * 256];

// TVPOpacityOnOpacityTable：合成两个半透明层时的结果 Alpha
unsigned char TVPOpacityOnOpacityTable[256 * 256];

// TVPNegativeMulTable[a * 256 + b] = 255 - (255 - a) * (255 - b) / 255
// 用于 Screen 混合模式
unsigned char TVPNegativeMulTable[256 * 256];

// 倒数表：TVPRecipTable256[i] = 65536 / i（用乘法+移位替代除法）
tjs_uint32 TVPRecipTable256[256];
```

### 5.2 查找表初始化代码

```cpp
// 来源：krkr2/cpp/core/visual/tvpgl.cpp 第 224-295 行（简化）
static void TVPCreateTable() {
    for (int a = 0; a < 256; a++) {
        for (int b = 0; b < 256; b++) {
            int addr = b * 256 + a;
            
            // 计算 TVPOpacityOnOpacityTable：
            // 两个独立半透明层叠加后的等效透明度
            if (a) {
                float at = a / 255.0f, bt = b / 255.0f;
                float c = bt / at;
                c /= (1.0f - bt + c);
                TVPOpacityOnOpacityTable[addr] = (unsigned char)(c * 255);
            } else {
                TVPOpacityOnOpacityTable[addr] = 255;
            }
            
            // 计算 Screen 混合：255 - (255-a)(255-b)/255
            TVPNegativeMulTable[addr] =
                (unsigned char)(255 - (255 - a) * (255 - b) / 255);
        }
    }
    
    // 除法查找表
    for (int b = 0; b < 256; b++) {
        TVPDivTable[(0 << 8) + b] = 0;  // 除以 0 特殊处理
        for (int a = 1; a < 256; a++) {
            int tmp = b * 255 / a;
            if (tmp > 255) tmp = 255;
            TVPDivTable[(a << 8) + b] = (uint8_t)tmp;
        }
    }
    
    // 倒数表（用移位乘法替代除法）
    TVPRecipTable256[0] = 65536;
    for (int i = 1; i < 256; i++) {
        TVPRecipTable256[i] = 65536 / i;
        // 使用时：x / i ≈ x * TVPRecipTable256[i] >> 16
    }
}
```

## 6. 跨平台像素格式差异

不同平台和图形 API 使用不同的默认像素格式：

| 平台/API | 默认格式 | 字节序 | 备注 |
|----------|----------|--------|------|
| Windows GDI/DirectX | BGRA（`DWORD`） | 小端 | 内存中为 B,G,R,A |
| macOS Core Graphics | ARGB 预乘 | 大端兼容 | `kCGImageAlphaPremultipliedFirst` |
| Linux X11/Wayland | BGRA 或 ARGB | 取决于视觉类型 | 需查询 Visual 配置 |
| Android (Java `Bitmap`) | ARGB_8888 | 大端语义 | Java 层为 `0xAARRGGBB` |
| OpenGL | RGBA | — | `glTexImage2D(GL_RGBA, GL_UNSIGNED_BYTE, ...)` |
| OpenGL ES | RGBA | — | 同 OpenGL |
| Vulkan | 可配置 | — | `VK_FORMAT_B8G8R8A8_UNORM` 等 |

KrKr2 统一使用 ARGB (`0xAARRGGBB`) 作为内部格式，在输出到不同平台时进行转换。转换代码的核心是 **R 和 B 通道互换**：

```cpp
// 来源：krkr2/cpp/core/visual/tvpgl.cpp 第 23-24 行
#define TVP_REVRGB(v) \
    ((v & 0xFF00FF00) | ((v >> 16) & 0xFF) | ((v & 0xFF) << 16))

// 示例：ARGB 0x80FF0000（半透明红）→ ABGR 0x800000FF
// 保持 A 和 G 不变（0xFF00FF00 掩码），交换 R 和 B
```

## 动手实践

创建一个完整的像素操作工具库：

```cpp
// pixel_utils.h
#pragma once
#include <cstdint>
#include <vector>
#include <fstream>

struct Image {
    std::vector<uint32_t> pixels;  // ARGB 格式
    int width = 0;
    int height = 0;
    
    Image(int w, int h, uint32_t fill = 0xFF000000)
        : pixels(w * h, fill), width(w), height(h) {}
    
    uint32_t& at(int x, int y) { return pixels[y * width + x]; }
    
    // 导出为 PPM 格式（便于查看）
    void savePPM(const char* path) const {
        std::ofstream f(path, std::ios::binary);
        f << "P6\n" << width << " " << height << "\n255\n";
        for (auto p : pixels) {
            f.put((p >> 16) & 0xFF);  // R
            f.put((p >> 8)  & 0xFF);  // G
            f.put( p        & 0xFF);  // B
        }
    }
};

// 创建棋盘格图案（用于测试 Alpha 混合）
Image createCheckerboard(int w, int h, int cellSize = 16) {
    Image img(w, h);
    for (int y = 0; y < h; y++) {
        for (int x = 0; x < w; x++) {
            bool isWhite = ((x / cellSize) + (y / cellSize)) % 2 == 0;
            img.at(x, y) = isWhite ? 0xFFFFFFFF : 0xFFC0C0C0;
        }
    }
    return img;
}
```

## 常见错误及解决方案

### 错误 1：忘记 pitch 导致图像错位

```cpp
// ❌ 错误：假设每行紧密排列
uint32_t* pixel = buffer + y * width + x;

// ✅ 正确：使用 pitch（以字节为单位）
uint8_t* row = reinterpret_cast<uint8_t*>(buffer) + y * pitch;
uint32_t* pixel = reinterpret_cast<uint32_t*>(row) + x;
```

### 错误 2：预乘 Alpha 反转时除零

```cpp
// ❌ 错误：Alpha=0 时除以零
uint8_t r = (pixel >> 16 & 0xFF) * 255 / (pixel >> 24 & 0xFF);

// ✅ 正确：先判断 Alpha
uint8_t a = (pixel >> 24) & 0xFF;
if (a == 0) return 0;  // 全透明像素直接返回 0
uint8_t r = std::min(255u, ((pixel >> 16) & 0xFF) * 255u / a);
```

### 错误 3：混淆 sRGB 和线性空间做混合

```cpp
// ❌ 错误：在 sRGB 空间取平均值（中间色调偏暗）
uint8_t avg = (srgb_a + srgb_b) / 2;

// ✅ 正确：转换到线性空间再取平均
float linear_a = srgbToLinear(srgb_a);
float linear_b = srgbToLinear(srgb_b);
uint8_t avg = linearToSrgb((linear_a + linear_b) * 0.5f);
```

## 本节小结

- **像素格式**：KrKr2 使用 ARGB 格式（`tjs_uint32`），`0xAARRGGBB`
- **位运算技巧**：通过掩码和移位同时处理多个通道，避免逐通道提取的分支和开销
- **预乘 Alpha**：减少混合运算量（3 次乘法→2 次），是工业标准做法
- **颜色空间**：sRGB 是非线性编码，正确的混合应在线性空间进行，但工程中常在 sRGB 直接混合以换取性能
- **查找表**：KrKr2 使用 256×256 查找表替代整数除法，空间换时间（每张表 64KB）
- **跨平台差异**：不同平台使用不同像素格式，`TVP_REVRGB` 宏用于 R/B 通道互换

## 练习题与答案

### 题目 1：像素通道提取

给定一个 ARGB 像素值 `0xC0336699`，请手动计算其 Alpha、Red、Green、Blue 通道的十进制值。如果将它转换为预乘 Alpha 格式，各通道的值是多少？

<details>
<summary>查看答案</summary>

**通道提取：**
- Alpha = `0xC0` = 192
- Red = `0x33` = 51
- Green = `0x66` = 102
- Blue = `0x99` = 153

**预乘 Alpha 转换：**
- A' = 192（不变）
- R' = 51 × 192 / 255 = 9792 / 255 ≈ 38
- G' = 102 × 192 / 255 = 19584 / 255 ≈ 76
- B' = 153 × 192 / 255 = 29376 / 255 ≈ 115

预乘后的像素值：`0xC0264C73`

</details>

### 题目 2：实现 TVP_REVRGB 的逆操作

`TVP_REVRGB` 将 ARGB 转为 ABGR。请证明：对同一个像素值连续调用两次 `TVP_REVRGB`，结果等于原值（即该操作是自反的）。

<details>
<summary>查看答案</summary>

```cpp
#define TVP_REVRGB(v) \
    ((v & 0xFF00FF00) | ((v >> 16) & 0xFF) | ((v & 0xFF) << 16))
```

设原值 `v = 0xAARRGGBB`：
1. 第一次调用：
   - `v & 0xFF00FF00` = `0xAA00GG00`（保留 A 和 G）
   - `(v >> 16) & 0xFF` = `0x000000RR`（R 移到 B 位置）
   - `(v & 0xFF) << 16` = `0x00BB0000`（B 移到 R 位置）
   - 结果：`0xAABBGGRR`

2. 对 `0xAABBGGRR` 再次调用：
   - `v & 0xFF00FF00` = `0xAA00GG00`
   - `(v >> 16) & 0xFF` = `0x000000BB`
   - `(v & 0xFF) << 16` = `0x00RR0000`
   - 结果：`0xAARRGGBB` — 恢复原值 ✅

该操作本质上是交换第 0 字节和第 2 字节，执行两次即恢复。

</details>

### 题目 3：查找表空间分析

KrKr2 使用了 `TVPDivTable[256*256]`、`TVPOpacityOnOpacityTable[256*256]`、`TVPNegativeMulTable[256*256]` 三张查找表。请计算它们总共占用多少内存？如果要增加 16 位精度版本（`uint16_t` 数组，索引范围扩大到 65536×256），需要多少内存？这种方案是否可行？

<details>
<summary>查看答案</summary>

**当前 8 位查找表：**
- 每张表：256 × 256 × 1 字节 = 65,536 字节 = 64 KB
- 三张表合计：64 × 3 = **192 KB**

这在现代系统上非常小（L1 缓存通常为 32-64 KB，L2 为 256-512 KB）。

**16 位精度版本：**
- 每张表：65536 × 256 × 2 字节 = 33,554,432 字节 = **32 MB**
- 三张表合计：32 × 3 = **96 MB**

**结论**：96 MB 的查找表完全不可行：
1. 远超 L1/L2/L3 缓存容量，导致大量缓存未命中
2. 桌面平台勉强可以，但 Android 等移动设备内存紧张
3. 更好的方案是使用定点数乘法 + 移位来近似除法

```cpp
// 用定点数替代查找表：x / a ≈ x * (65536 / a) >> 16
uint16_t recipTable[256]; // 只需 512 字节
recipTable[a] = 65536 / a;

uint8_t result = (uint8_t)((value * recipTable[alpha]) >> 16);
```

</details>
