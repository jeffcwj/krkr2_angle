## 1. 像素的本质：数字化的颜色

### 1.1 从光到数字

计算机屏幕上的每一个像素（Pixel，Picture Element 的缩写，意为"图像元素"）由红（R）、绿（G）、蓝（B）三个颜色通道混合而成。这三种颜色被称为"加色三原色"——因为光的混合是"越加越亮"的。每个通道通常用 8 位无符号整数（0~255）表示亮度，0 代表该颜色完全不发光，255 代表最大亮度。

除了 RGB 三个颜色通道，现代图形系统还会为每个像素增加一个 **Alpha（A）通道**，用于表示该像素的不透明度。Alpha = 0 表示完全透明（像素不可见），Alpha = 255 表示完全不透明。这样一个像素总共由 4 个通道组成，每个 8 位，共 32 位（4 字节），这就是最常见的 **32 位像素格式**。

```
一个像素 = 4 字节 = 32 位
┌────────┬────────┬────────┬────────┐
│  A(8)  │  R(8)  │  G(8)  │  B(8)  │
│ 透明度  │   红   │   绿    │   蓝   │
└────────┴────────┴────────┴────────┘
高位 ←────────────────────────────→ 低位
```

### 1.2 为什么是 32 位？

32 位像素格式之所以成为事实标准，有三个重要原因：

- **对齐友好**：32 位（4 字节）恰好是现代 CPU 的最小高效处理单元。一次 `uint32_t` 读写即可操作完整一个像素，不需要跨字节拼接
- **SIMD 友好**：SSE2（一种 CPU 指令集扩展，第 4 章详讲）的 128 位寄存器可一次装入 4 个像素（128 ÷ 32 = 4），AVX2 的 256 位寄存器可装 8 个，实现批量并行处理
- **直接映射**：操作系统的帧缓冲（Framebuffer，即显存中存储屏幕画面的内存区域）通常直接以 32 位格式存储，无需格式转换即可显示

### 1.3 其他位深度

虽然 32 位是主流，但还有其他像素格式存在：

| 位深度 | 格式示例 | 说明 | 使用场景 |
|--------|----------|------|----------|
| 8 位 | 灰度、调色板索引 | 每像素 1 字节 | 复古游戏、图标 |
| 16 位 | RGB565、RGBA4444 | 省内存但色彩精度低 | 移动端纹理压缩 |
| 24 位 | RGB888 | 无 Alpha，不利于对齐 | JPEG 文件存储 |
| **32 位** | **ARGB8888** | **带 Alpha，对齐友好** | **绝大多数引擎（包括 KrKr2）** |
| 64 位 | RGBA16161616 | 每通道 16 位浮点 | HDR 渲染、专业图像处理 |

KrKr2 全程使用 32 位 ARGB 格式，像素类型为 `tjs_uint32`（即 `uint32_t`）。

## 2. 常见像素格式与内存布局

### 2.1 格式命名规则

像素格式的名称描述**从高位到低位**的通道排列顺序。例如 "ARGB" 表示：最高 8 位是 Alpha，接下来是 Red，然后 Green，最低 8 位是 Blue。

但需注意一个重要的计算机基础概念——**字节序（Byte Order）**。在**小端序（Little Endian）**机器上（x86 和 ARM 默认都是小端序），多字节数据在内存中是"低字节在低地址"的。这意味着一个 `uint32_t` 值 `0xAARRGGBB` 在内存中的实际字节顺序是 `BB, GG, RR, AA`。

```
寄存器值: 0xAARRGGBB
               ↕ 小端序存储
内存布局: [BB] [GG] [RR] [AA]
地址:      低 ────────────→ 高
```

这会导致一个容易混淆的问题：你用 `unsigned char*` 指针逐字节读取一个 ARGB 像素时，读到的顺序是 B, G, R, A——和格式名称完全相反！

### 2.2 主流格式对照表

| 格式名称 | 32位值（高→低） | 内存字节序（低地址→高，小端） | 使用场景 |
|----------|-----------------|-------------------------------|----------|
| **ARGB** | `AARRGGBB` | B, G, R, A | **KrKr2**/Windows GDI/DirectX |
| **RGBA** | `RRGGBBAA` | A, B, G, R | OpenGL（一种跨平台图形 API）/PNG 文件 |
| **BGRA** | `BBGGRRAA` | A, R, G, B | Windows Bitmap/Vulkan（新一代图形 API） |
| **ABGR** | `AABBGGRR` | R, G, B, A | OpenGL ES 某些实现 |

> **关键点**：KrKr2 全程使用 **ARGB** 格式，像素类型为 `tjs_uint32`（即 `uint32_t`）。在输出到不同平台的图形 API 时，引擎会自动做格式转换（见本章第 3 节）。

### 2.3 格式转换的本质：通道交换

不同格式之间转换的核心操作是**交换特定的字节位置**。以最常见的 ARGB ↔ ABGR 转换为例，只需交换 R 和 B 通道：

```cpp
// 来源：krkr2/cpp/core/visual/tvpgl.cpp 第 23-24 行
#define TVP_REVRGB(v) \
    ((v & 0xFF00FF00) | ((v >> 16) & 0xFF) | ((v & 0xFF) << 16))

// 解析：
// v & 0xFF00FF00  → 保留 A 和 G 通道（它们的位置不变）
// (v >> 16) & 0xFF → R 通道右移到 B 的位置
// (v & 0xFF) << 16 → B 通道左移到 R 的位置
// 示例：ARGB 0x80FF0000（半透明红）→ ABGR 0x800000FF
```

## 3. C++ 中的像素操作

### 3.1 通道提取与合成

```cpp
#include <cstdint>
#include <cstdio>

// KrKr2 风格：直接使用 uint32_t（项目中用 tjs_uint32 别名）
using Pixel = uint32_t;

// 从 32 位像素中提取各通道（ARGB 格式）
// 通过右移 + 掩码，将目标通道移到最低 8 位并截取
inline uint8_t getAlpha(Pixel p) { return (p >> 24) & 0xFF; }  // 高 8 位
inline uint8_t getRed(Pixel p)   { return (p >> 16) & 0xFF; }  // 次高 8 位
inline uint8_t getGreen(Pixel p) { return (p >> 8)  & 0xFF; }  // 次低 8 位
inline uint8_t getBlue(Pixel p)  { return  p        & 0xFF; }  // 低 8 位

// 将四个通道合成为一个像素
// 每个通道左移到对应位置，用按位或拼合
inline Pixel makePixel(uint8_t a, uint8_t r, uint8_t g, uint8_t b) {
    return (static_cast<uint32_t>(a) << 24) |
           (static_cast<uint32_t>(r) << 16) |
           (static_cast<uint32_t>(g) << 8)  |
            static_cast<uint32_t>(b);
}

int main() {
    // 半透明红色：Alpha=128, R=255, G=0, B=0
    Pixel red = makePixel(128, 255, 0, 0);  // 值为 0x80FF0000
    printf("Pixel value: 0x%08X\n", red);
    printf("Alpha=%u, R=%u, G=%u, B=%u\n",
           getAlpha(red), getRed(red), getGreen(red), getBlue(red));
    // 输出：
    // Pixel value: 0x80FF0000
    // Alpha=128, R=255, G=0, B=0

    // 完全不透明的白色
    Pixel white = makePixel(255, 255, 255, 255);  // 0xFFFFFFFF
    printf("White: 0x%08X\n", white);

    // 25% 透明的蓝色
    Pixel blue = makePixel(64, 0, 0, 255);  // 0x400000FF
    printf("Blue:  0x%08X  Alpha=%u\n", blue, getAlpha(blue));

    return 0;
}
```

### 3.2 KrKr2 的位运算技巧：并行通道处理

KrKr2 的 `tvpgl.h` 中大量使用**位运算技巧**同时处理多个通道，避免逐通道提取的开销。这是软件渲染性能优化的核心手段。

最经典的例子是**饱和加法（Saturated Addition）**——两个像素的对应通道相加，如果超过 255 则钳制为 255（而不是溢出回绕）：

```cpp
// 来源：krkr2/cpp/core/visual/tvpgl.h 第 73-80 行
static tjs_uint32 TVPSaturatedAdd(tjs_uint32 a, tjs_uint32 b) {
    // 利用并行位运算同时处理 4 个通道
    tjs_uint32 tmp = ((a & b) + (((a ^ b) >> 1) & 0x7f7f7f7f)) & 0x80808080;
    tmp = (tmp << 1) - (tmp >> 7);  // 将溢出位扩展为 0xFF 掩码
    return (a + b - tmp) | tmp;     // 溢出通道设为 0xFF
}
```

这段代码看起来很神秘，让我们逐步拆解：

1. **`(a & b) + (((a ^ b) >> 1) & 0x7f7f7f7f)`** — 这实际上是在计算每个字节的近似平均值。`a & b` 是进位部分，`(a ^ b) >> 1` 是不进位部分。合起来等价于 `(a + b) >> 1`，但避免了中间结果溢出
2. **`& 0x80808080`** — 只保留每字节的最高位。如果两个字节相加会超过 255，这一位就为 1
3. **`(tmp << 1) - (tmp >> 7)`** — 将 `0x80` 扩展为 `0xFF`。例如 `0x80 << 1 = 0x100`，`0x80 >> 7 = 0x01`，`0x100 - 0x01 = 0xFF`
4. **`(a + b - tmp) | tmp`** — 未溢出的通道正常相加，溢出的通道被 `| 0xFF` 设为最大值

### 3.3 颜色乘法

另一个常用操作是**颜色乘法（Color Multiply）**——将像素的每个通道乘以一个因子，用于调节亮度或实现预乘 Alpha（Premultiplied Alpha，下一节详讲）：

```cpp
// 来源：krkr2/cpp/core/visual/tvpgl.h 第 161-166 行（简化）
// 将像素每个通道乘以 ratio（0~255），结果右移 8 位近似除以 256
static tjs_uint32 TVPMulColor(tjs_uint32 color, tjs_uint32 ratio) {
    // 同时处理 R+B 两个通道（它们被 G 通道隔开，不会互相干扰）
    tjs_uint32 rb = ((color & 0x00ff00ff) * ratio >> 8) & 0x00ff00ff;
    // 单独处理 G 通道
    tjs_uint32 g  = ((color & 0x0000ff00) * ratio >> 8) & 0x0000ff00;
    return rb | g | (color & 0xff000000);  // 保留原始 Alpha
}
```

这里使用了和饱和加法类似的"双通道并行"技巧：R 在 bit 16-23、B 在 bit 0-7，中间隔着 G 的 8 位，所以 `(R * ratio)` 和 `(B * ratio)` 的结果不会相互溢出覆盖（前提是 ratio ≤ 255）。

## 4. 像素数组与图像缓冲区

### 4.1 行优先存储

一张 W×H 的图像在内存中通常以**行优先（Row-Major）**方式连续存储。第一行的 W 个像素排在最前面，紧接着是第二行，依此类推：

```
内存布局（W=4, H=3 的图像）：
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│(0,0) │(1,0) │(2,0) │(3,0) │(0,1) │(1,1) │(2,1) │(3,1) │(0,2) │(1,2) │(2,2) │(3,2) │
│ 第0行                      │ 第1行                      │ 第2行                      │
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
低地址 ──────────────────────────────────────────────────────────────────→ 高地址
```

### 4.2 pitch（行跨度）

在实际系统中，每行像素的末尾可能会有**填充字节（padding）**，使下一行的起始地址对齐到特定边界（通常是 16、32 或 64 字节）。这种对齐可以显著提升 CPU 和 GPU 访问内存的效率。

**pitch**（也叫 stride）是相邻两行像素起始地址之间的字节距离。它通常 ≥ `width × 4`：

```
实际内存布局（W=5 像素 = 20 字节，但 pitch=32 字节，对齐到 32）：

行0: [P00][P01][P02][P03][P04][PAD][PAD][PAD]   ← 32 字节
行1: [P10][P11][P12][P13][P14][PAD][PAD][PAD]   ← 32 字节
行2: [P20][P21][P22][P23][P24][PAD][PAD][PAD]   ← 32 字节

每个 [Pxy] = 4 字节，[PAD] = 4 字节填充
pitch = 32 字节，而不是 width * 4 = 20 字节
```

```cpp
// 图像缓冲区的基本结构
struct ImageBuffer {
    uint32_t* pixels;   // 像素数组首地址（ARGB 格式）
    int width;          // 宽度（像素数）
    int height;         // 高度（像素数）
    int pitch;          // 行跨度（字节），通常 >= width * 4

    // 获取第 (x, y) 个像素——必须用 pitch 计算偏移！
    uint32_t& at(int x, int y) {
        // pitch 以字节为单位，需要先转为 uint8_t* 加偏移，再转回 uint32_t*
        return *(reinterpret_cast<uint32_t*>(
            reinterpret_cast<uint8_t*>(pixels) + y * pitch) + x);
    }
};

// 遍历所有像素的正确方式
void fillRed(ImageBuffer& img) {
    for (int y = 0; y < img.height; y++) {
        // 计算当前行的起始地址
        uint8_t* rowPtr = reinterpret_cast<uint8_t*>(img.pixels) + y * img.pitch;
        uint32_t* row = reinterpret_cast<uint32_t*>(rowPtr);
        for (int x = 0; x < img.width; x++) {
            row[x] = 0xFFFF0000;  // 不透明红色
        }
        // 注意：不要访问 row[width] ~ row[pitch/4 - 1]，那是填充区域
    }
}
```

> **关键规则**：**永远用 pitch 而不是 `width * 4` 来计算行偏移。** 忘记这一点是像素操作中最常见的 Bug，会导致图像出现斜线条纹或错位。

### 4.3 KrKr2 中的图像缓冲区

KrKr2 的位图类 `tTVPBaseBitmap`（定义在 `cpp/core/visual/LayerBitmapIntf.h`）内部使用了类似的结构：

```cpp
// 简化自 KrKr2 源码
class tTVPBaseBitmap {
    tjs_uint32* Bits;       // 像素数据指针
    tjs_uint BitsWidth;     // 图像宽度
    tjs_uint BitsHeight;    // 图像高度
    tjs_int PitchBytes;     // 行跨度（字节）——注意可以是负数！

    // 获取指定行的像素指针
    tjs_uint32* GetScanLine(tjs_uint y) {
        return reinterpret_cast<tjs_uint32*>(
            reinterpret_cast<tjs_uint8*>(Bits) + y * PitchBytes);
    }
};
```

注意 `PitchBytes` 类型是 `tjs_int`（有符号整数）。当 pitch 为负值时，表示图像是**上下颠倒存储的（bottom-up）**——第一行在内存的最高地址，最后一行在最低地址。Windows 的 DIB（Device Independent Bitmap，设备无关位图）就常用这种方式。

## 动手实践

创建一个完整的像素操作工具库，生成棋盘格图案并导出为 PPM 文件：

```cpp
// main.cpp
#include <cstdint>
#include <cstdio>
#include <vector>
#include <fstream>

using Pixel = uint32_t;

// --- 通道操作函数 ---
inline uint8_t getAlpha(Pixel p) { return (p >> 24) & 0xFF; }
inline uint8_t getRed(Pixel p)   { return (p >> 16) & 0xFF; }
inline uint8_t getGreen(Pixel p) { return (p >> 8)  & 0xFF; }
inline uint8_t getBlue(Pixel p)  { return  p        & 0xFF; }

inline Pixel makePixel(uint8_t a, uint8_t r, uint8_t g, uint8_t b) {
    return (uint32_t(a) << 24) | (uint32_t(r) << 16) |
           (uint32_t(g) << 8) | uint32_t(b);
}

// --- 格式转换 ---
// ARGB → ABGR（交换 R 和 B）
inline Pixel argbToAbgr(Pixel v) {
    return (v & 0xFF00FF00) | ((v >> 16) & 0xFF) | ((v & 0xFF) << 16);
}

// --- 简易图像类 ---
struct Image {
    std::vector<Pixel> pixels;
    int width = 0, height = 0;

    Image(int w, int h, Pixel fill = 0xFF000000)
        : pixels(w * h, fill), width(w), height(h) {}

    Pixel& at(int x, int y) { return pixels[y * width + x]; }

    // 导出为 PPM 格式（P6 二进制，几乎所有图片查看器都支持）
    void savePPM(const char* path) const {
        std::ofstream f(path, std::ios::binary);
        // PPM 文件头：P6 表示二进制 RGB 格式
        f << "P6\n" << width << " " << height << "\n255\n";
        for (auto p : pixels) {
            f.put(getRed(p));    // R
            f.put(getGreen(p));  // G
            f.put(getBlue(p));   // B
            // 注意：PPM 不支持 Alpha 通道
        }
    }
};

// --- 生成棋盘格（用于后续测试 Alpha 混合） ---
Image createCheckerboard(int w, int h, int cellSize = 16) {
    Image img(w, h);
    for (int y = 0; y < h; y++) {
        for (int x = 0; x < w; x++) {
            bool isWhite = ((x / cellSize) + (y / cellSize)) % 2 == 0;
            img.at(x, y) = isWhite ? 0xFFFFFFFF : 0xFFC0C0C0;
        }
    }
    return img;
}

// --- 生成渐变图 ---
Image createGradient(int w, int h) {
    Image img(w, h);
    for (int y = 0; y < h; y++) {
        for (int x = 0; x < w; x++) {
            uint8_t r = static_cast<uint8_t>(x * 255 / (w - 1));
            uint8_t g = static_cast<uint8_t>(y * 255 / (h - 1));
            uint8_t b = 128;
            img.at(x, y) = makePixel(255, r, g, b);
        }
    }
    return img;
}

int main() {
    // 测试通道提取
    Pixel test = 0xC0336699;
    printf("Pixel: 0x%08X\n", test);
    printf("  A=%u  R=%u  G=%u  B=%u\n",
           getAlpha(test), getRed(test), getGreen(test), getBlue(test));

    // 测试格式转换
    Pixel converted = argbToAbgr(test);
    printf("ARGB→ABGR: 0x%08X → 0x%08X\n", test, converted);
    // 验证自反性：转换两次应该回到原值
    printf("双重转换: 0x%08X（应等于原值）\n", argbToAbgr(converted));

    // 生成并保存图片
    auto checker = createCheckerboard(256, 256);
    checker.savePPM("checkerboard.ppm");
    printf("已保存 checkerboard.ppm (256x256)\n");

    auto gradient = createGradient(256, 256);
    gradient.savePPM("gradient.ppm");
    printf("已保存 gradient.ppm (256x256)\n");

    return 0;
}
```

## 常见错误及解决方案

### 错误 1：忘记 pitch 导致图像错位

```cpp
// ❌ 错误：假设每行紧密排列
uint32_t* pixel = buffer + y * width + x;

// ✅ 正确：使用 pitch（以字节为单位）
uint8_t* row = reinterpret_cast<uint8_t*>(buffer) + y * pitch;
uint32_t* pixel = reinterpret_cast<uint32_t*>(row) + x;
```

**现象**：图像出现斜线条纹、每行向右偏移几个像素。
**原因**：`width * 4` 与 `pitch` 不相等时（有填充字节），按 width 计算地址会逐行累积偏差。

### 错误 2：混淆字节序与格式名称

```cpp
// ❌ 错误：以为 ARGB 在内存中就是 A,R,G,B 顺序
uint8_t* bytes = reinterpret_cast<uint8_t*>(&pixel);
uint8_t alpha = bytes[0];  // 在小端序上，这读到的是 B 而不是 A！

// ✅ 正确：用位移操作提取通道（与字节序无关）
uint8_t alpha = (pixel >> 24) & 0xFF;  // 始终正确
```

**现象**：颜色通道错乱，红色变蓝色。
**根因**：小端序（x86/ARM）上，`uint32_t` 的最低字节在最低地址。`0xAARRGGBB` 在内存中是 `BB GG RR AA`。

### 错误 3：通道合成时忘记类型提升

```cpp
// ❌ 错误：uint8_t 左移超过 8 位会溢出
uint8_t a = 128, r = 255;
uint32_t pixel = (a << 24) | (r << 16);  // 未定义行为！

// ✅ 正确：先转为 uint32_t 再移位
uint32_t pixel = (static_cast<uint32_t>(a) << 24) |
                 (static_cast<uint32_t>(r) << 16);
```

**现象**：编译器警告或者高位通道值错误。
**根因**：`uint8_t << 24` 在 C++ 中会先将 `uint8_t` 提升为 `int`（32位有符号），但 `128 << 24` 的结果超出了 `int` 的正数范围，是未定义行为。

## 本节小结

- **像素** = 4 字节 = 32 位，包含 A（透明度）、R（红）、G（绿）、B（蓝）四个 8 位通道
- **格式命名**描述高位到低位的通道顺序。KrKr2 使用 **ARGB** 格式（`0xAARRGGBB`）
- **小端序**机器上，内存字节顺序与格式名称相反。用**位移操作**提取通道可避免字节序问题
- **pitch**（行跨度）是图像缓冲区中相邻两行起始地址的字节距离，可能大于 `width × 4`。**必须用 pitch 计算行偏移**
- KrKr2 用**双通道并行位运算**技巧（利用 R 和 B 通道被 G 通道隔开的特性）同时处理两个颜色通道
- **`TVPSaturatedAdd`** 用纯位运算实现四通道并行的饱和加法，无任何分支

## 练习题与答案

### 题目 1：手动计算像素通道值

给定一个 ARGB 像素值 `0xC0336699`，请手动计算其 Alpha、Red、Green、Blue 通道的十进制值。然后写出该像素在小端序机器上的内存字节排列（低地址到高地址）。

<details>
<summary>查看答案</summary>

**通道提取（位移法）**：
- Alpha = `(0xC0336699 >> 24) & 0xFF` = `0xC0` = **192**
- Red   = `(0xC0336699 >> 16) & 0xFF` = `0x33` = **51**
- Green = `(0xC0336699 >> 8)  & 0xFF` = `0x66` = **102**
- Blue  = `(0xC0336699)       & 0xFF` = `0x99` = **153**

**小端序内存布局**（低地址在左）：
```
地址:   N    N+1   N+2   N+3
内容: [99]  [66]  [33]  [C0]
      (B)   (G)   (R)   (A)
```

即 `0x99, 0x66, 0x33, 0xC0`——与 ARGB 的字面顺序完全相反。

</details>

### 题目 2：实现安全的像素行遍历

编写一个函数 `invertColors(ImageBuffer& img)`，将图像中每个像素的 RGB 通道取反（255 - 值），但保持 Alpha 不变。要求正确处理 pitch。

<details>
<summary>查看答案</summary>

```cpp
#include <cstdint>

struct ImageBuffer {
    uint32_t* pixels;
    int width, height, pitch;  // pitch 以字节为单位
};

void invertColors(ImageBuffer& img) {
    for (int y = 0; y < img.height; y++) {
        // 用 pitch 计算当前行起始地址
        uint8_t* rowBytes = reinterpret_cast<uint8_t*>(img.pixels)
                          + y * img.pitch;
        uint32_t* row = reinterpret_cast<uint32_t*>(rowBytes);

        for (int x = 0; x < img.width; x++) {
            uint32_t p = row[x];
            // 取反 RGB，保留 Alpha
            // 0x00FFFFFF 掩码覆盖 R、G、B 三个通道
            // 异或 0x00FFFFFF 等价于对这三个通道取 (255 - 值)
            row[x] = (p & 0xFF000000) | (~p & 0x00FFFFFF);
        }
    }
}
```

**解析**：
- `~p` 对所有 32 位取反，包括 Alpha
- `~p & 0x00FFFFFF` 只保留取反后的 RGB，丢弃被取反的 Alpha
- `p & 0xFF000000` 保留原始 Alpha
- 两者按位或拼合

</details>

### 题目 3：分析 TVPSaturatedAdd 的极端情况

对于 `TVPSaturatedAdd(0xFF808080, 0xFF808080)`（两个半灰色像素相加），请手动推算每个通道的结果值，验证是否正确饱和到 255。

<details>
<summary>查看答案</summary>

每个通道：`0x80 + 0x80 = 128 + 128 = 256`，超过 255，应该饱和到 **255（0xFF）**。

手动推算 `TVPSaturatedAdd` 的步骤：
```
a = 0xFF808080, b = 0xFF808080

步骤 1: a & b = 0xFF808080
步骤 2: a ^ b = 0x00000000
步骤 3: ((a ^ b) >> 1) & 0x7f7f7f7f = 0x00000000
步骤 4: 步骤1 + 步骤3 = 0xFF808080
步骤 5: & 0x80808080 = 0x80808080  （每个通道最高位都是 1，表示都会溢出）
步骤 6: tmp = (0x80808080 << 1) - (0x80808080 >> 7)
       = 0x01010100 - 0x01010101
       上面有个问题——让我重新计算：
       0x80808080 << 1 = 0x01010100（各字节 0x80<<1 溢出到高字节）

实际上应按字节理解：
  每字节 0x80 << 1 在字节内 = 0x00（溢出），进位到高字节
  但这里是整体 32 位移位，所以：
  0x80808080 << 1 = 0x01010100
  0x80808080 >> 7 = 0x01010101

  tmp = 0x01010100 - 0x01010101 会出现下溢...
```

实际上更好的理解方式是：对于输入 `a=b=0xFF808080`：
- `a + b = 0xFF808080 + 0xFF808080 = 0xFF010100`（32位运算会丢失最高位进位）
- 每个通道本应是 0x80+0x80=0x100，但字节溢出到了上一个通道

KrKr2 的饱和加法正确处理了这种情况。最终结果为 **`0xFFFFFFFF`**——所有通道饱和到 255。

验证代码：
```cpp
#include <cstdint>
#include <cstdio>

uint32_t TVPSaturatedAdd(uint32_t a, uint32_t b) {
    uint32_t tmp = ((a & b) + (((a ^ b) >> 1) & 0x7f7f7f7f)) & 0x80808080;
    tmp = (tmp << 1) - (tmp >> 7);
    return (a + b - tmp) | tmp;
}

int main() {
    uint32_t result = TVPSaturatedAdd(0xFF808080, 0xFF808080);
    printf("Result: 0x%08X\n", result);
    // 输出：Result: 0xFFFFFFFF  ✅ 所有通道饱和到 255
    return 0;
}
```

</details>
