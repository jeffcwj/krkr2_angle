## 1. 各平台的默认像素格式

### 1.1 为什么存在格式差异？

不同的操作系统和图形 API 在历史上各自选择了不同的像素通道排列方式。这些选择往往受到硬件设计、API 设计年代和向后兼容性的影响。对于跨平台引擎来说，必须在内部统一使用一种格式，然后在输出到各平台时进行转换。

### 1.2 四平台格式对照表

| 平台/API | 默认格式 | 32位整数值 | 内存字节序（小端） | 备注 |
|----------|----------|-----------|-------------------|------|
| **Windows GDI/DirectX** | BGRA | `0xAARRGGBB`* | B, G, R, A | `COLORREF` 宏使用 `0x00BBGGRR`（无 Alpha） |
| **macOS Core Graphics** | ARGB 预乘 | `0xAARRGGBB` | B, G, R, A | `kCGImageAlphaPremultipliedFirst` |
| **Linux X11/Wayland** | 取决于 Visual | 通常 BGRA | B, G, R, A | 需运行时查询 `XVisualInfo.blue_mask` |
| **Android Bitmap (Java)** | ARGB_8888 | `0xAARRGGBB` | B, G, R, A | Java 层用大端语义描述，底层小端存储 |
| **OpenGL** | RGBA | `0xRRGGBBAA` | R, G, B, A | `glTexImage2D(GL_RGBA, GL_UNSIGNED_BYTE, ...)` |
| **Vulkan** | 可配置 | — | — | 常用 `VK_FORMAT_B8G8R8A8_UNORM` |

> **注意**：Windows 的 `COLORREF` 格式（`RGB()` 宏返回值）是 `0x00BBGGRR`，没有 Alpha 通道，且 R/B 位置与 ARGB 相反。这是一个非常容易踩的坑。

*严格说 Windows `DWORD` 存储的是 BGRA 字节序（内存中 B,G,R,A），但由于小端序，读出来的 `uint32_t` 值是 `0xAARRGGBB`——和 ARGB 的整数值一样。

### 1.3 Windows 平台细节

Windows 的位图（Bitmap）系统有一个独特特性——**DIB（Device Independent Bitmap，设备无关位图）支持 bottom-up 存储**。这意味着图像的第一行（y=0）可以在内存的最高地址处，最后一行在最低地址处。此时 `biHeight` 为正数，pitch 为**负值**：

```cpp
// Windows BITMAPINFOHEADER 结构（简化）
typedef struct {
    LONG  biWidth;       // 图像宽度（像素）
    LONG  biHeight;      // 正值 = bottom-up，负值 = top-down
    WORD  biBitCount;    // 每像素位数（32）
    // ...
} BITMAPINFOHEADER;

// biHeight > 0 时：
//   内存最低地址 → 最后一行（y = height-1）
//   内存最高地址 → 第一行（y = 0）
//   pitch = -(width * 4)  或更小的负值（含对齐填充）

// biHeight < 0 时（top-down）：
//   内存最低地址 → 第一行（y = 0）
//   pitch = width * 4  或更大的正值
```

KrKr2 的 `tTVPBaseBitmap` 中 `PitchBytes` 类型是 `tjs_int`（有符号整数），正是为了支持负 pitch：

```cpp
// 来源：cpp/core/visual/LayerBitmapIntf.h（简化）
class tTVPBaseBitmap {
    tjs_uint32* Bits;       // 像素数据指针
    tjs_int PitchBytes;     // 行跨度，可以是负数！

    tjs_uint32* GetScanLine(tjs_uint y) {
        // pitch 为负时，y 增加 → 地址减小（bottom-up）
        return reinterpret_cast<tjs_uint32*>(
            reinterpret_cast<tjs_uint8*>(Bits) + y * PitchBytes);
    }
};
```

### 1.4 macOS 平台细节

macOS 的 Core Graphics 框架**强制要求**预乘 Alpha 格式。创建位图上下文（CGBitmapContext）时必须指定 `kCGImageAlphaPremultipliedFirst` 或 `kCGImageAlphaPremultipliedLast`：

```cpp
// macOS 创建位图上下文
CGContextRef ctx = CGBitmapContextCreate(
    buffer,                              // 像素缓冲区
    width, height,                       // 尺寸
    8,                                   // 每通道 8 位
    width * 4,                           // 每行字节数
    colorSpace,
    kCGImageAlphaPremultipliedFirst      // ARGB 预乘格式
    // 也可用 kCGImageAlphaPremultipliedLast → RGBA 预乘
);
// 注意：Core Graphics 不支持非预乘（straight）Alpha！
// 传入 kCGImageAlphaNone 会被拒绝（对32位图像）
```

这意味着 KrKr2 在 macOS 上输出图像时，如果内部使用 straight Alpha，还需要在格式转换的同时做一次预乘转换。

### 1.5 Linux 平台细节

Linux 上的 X11 系统使用 **Visual**（视觉类型）来描述像素格式。不同的 X 服务器可能支持不同的格式，程序需要在运行时查询：

```cpp
// X11 查询当前 Visual 的像素格式
#include <X11/Xlib.h>

void queryPixelFormat(Display* dpy, int screen) {
    XVisualInfo vinfo;
    // 请求 TrueColor Visual，32 位深度
    if (XMatchVisualInfo(dpy, screen, 32, TrueColor, &vinfo)) {
        printf("Red   mask: 0x%lX\n", vinfo.red_mask);
        printf("Green mask: 0x%lX\n", vinfo.green_mask);
        printf("Blue  mask: 0x%lX\n", vinfo.blue_mask);
        // 典型输出（BGRA Visual）：
        //   Red   mask: 0xFF0000
        //   Green mask: 0x00FF00
        //   Blue  mask: 0x0000FF
    }
}
```

现代 Linux 桌面（使用 Wayland 合成器）通常默认 ARGB32 格式（`WL_SHM_FORMAT_ARGB8888`），与 KrKr2 的内部格式一致，无需转换。

### 1.6 Android 平台细节

Android 的 Java `Bitmap` 类使用 `ARGB_8888` 格式。Java 是大端语义的，所以 Java 代码中 `Color.argb(a, r, g, b)` 返回的整数值为 `0xAARRGGBB`。但在底层 C++ 通过 JNI（Java Native Interface，Java 和 C++ 之间的调用桥梁）访问位图像素时，由于 ARM 处理器是小端序，内存中的字节顺序是 B, G, R, A——与 Windows 相同。

```cpp
// Android JNI 中访问 Bitmap 像素
#include <android/bitmap.h>

void processPixels(JNIEnv* env, jobject bitmap) {
    AndroidBitmapInfo info;
    AndroidBitmap_getInfo(env, bitmap, &info);
    // info.format == ANDROID_BITMAP_FORMAT_RGBA_8888
    // 注意：尽管名叫 RGBA_8888，实际 uint32_t 读出来是 0xAABBGGRR
    // 这与 KrKr2 的 ARGB (0xAARRGGBB) 不同，需要 R↔B 交换！

    void* pixels = nullptr;
    AndroidBitmap_lockPixels(env, bitmap, &pixels);
    uint32_t* px = static_cast<uint32_t*>(pixels);

    for (uint32_t i = 0; i < info.width * info.height; i++) {
        // KrKr2 ARGB → Android ABGR：交换 R 和 B
        px[i] = (px[i] & 0xFF00FF00) |
                ((px[i] >> 16) & 0xFF) |
                ((px[i] & 0xFF) << 16);
    }

    AndroidBitmap_unlockPixels(env, bitmap);
}
```

## 2. KrKr2 的跨平台格式统一策略

### 2.1 内部格式选择

KrKr2 选择 **ARGB（`0xAARRGGBB`）** 作为内部统一格式。所有像素操作——混合、缩放、滤波——都在 ARGB 格式下进行。只在**输出到屏幕**或**从外部加载图片**时才做格式转换。

这种设计的好处：

1. **渲染核心代码无需关心平台**：所有混合函数（`blend_functor_c.h`）只处理 ARGB
2. **转换集中在边界**：格式转换代码只在加载器（Load*.cpp）和输出模块中
3. **性能开销极小**：R↔B 交换是纯位运算，每像素只需 3 条指令

### 2.2 TVP_REVRGB 宏详解

格式转换的核心是 `TVP_REVRGB` 宏，它交换一个像素中的 R 和 B 通道：

```cpp
// 来源：krkr2/cpp/core/visual/tvpgl.cpp 第 23-24 行
#define TVP_REVRGB(v) \
    ((v & 0xFF00FF00) | ((v >> 16) & 0xFF) | ((v & 0xFF) << 16))
```

逐步分析这个宏的三个部分：

```
原始像素 v = 0xAARRGGBB

第1部分: v & 0xFF00FF00 = 0xAA00GG00
  → 保留 Alpha 和 Green（它们在 ARGB↔ABGR 转换中位置不变）

第2部分: (v >> 16) & 0xFF = 0x000000RR
  → 将 R 通道从 bit 16-23 移到 bit 0-7（B 的位置）

第3部分: (v & 0xFF) << 16 = 0x00BB0000
  → 将 B 通道从 bit 0-7 移到 bit 16-23（R 的位置）

结果 = 0xAABBGGRR（ABGR 格式）
```

### 2.3 批量转换函数

实际使用中，引擎需要对整行甚至整帧像素做批量转换。KrKr2 提供了行级别的转换函数：

```cpp
// 将整行像素从 ARGB 转换为 ABGR（或反过来，因为是自反操作）
void TVPReverseRGB(tjs_uint32* dest, const tjs_uint32* src, tjs_int len) {
    for (tjs_int i = 0; i < len; i++) {
        dest[i] = TVP_REVRGB(src[i]);
    }
}

// 在 SIMD 优化版本中（第 4 章详讲），使用 SSE2 一次处理 4 个像素：
// __m128i pixels = _mm_loadu_si128(...);
// 通过 _mm_shufflelo_epi16 + _mm_shufflehi_epi16 做字节重排
// 比逐像素处理快 4 倍
```

### 2.4 各平台的转换需求

| 平台 | 内部格式 | 输出格式 | 需要的转换 |
|------|---------|---------|-----------|
| Windows (GDI/DX) | ARGB (`0xAARRGGBB`) | BGRA（但 `uint32_t` 值相同） | **无需转换** |
| macOS (Core Graphics) | ARGB | ARGB 预乘 | 需要预乘 Alpha |
| Linux (X11 TrueColor) | ARGB | 通常 BGRA | 需查询 Visual，通常**无需转换** |
| Linux (Wayland) | ARGB | ARGB | **无需转换** |
| Android (ABGR) | ARGB | ABGR | **需要 TVP_REVRGB** |
| OpenGL (RGBA) | ARGB | RGBA | 需要字节重排 |

可以看到，KrKr2 选择 ARGB 作为内部格式是非常明智的——在最主要的 Windows 平台上完全不需要格式转换，大大减少了性能开销。

## 3. 底层内存布局深入分析

### 3.1 Bottom-Up 与 Top-Down 存储

前面提到 Windows DIB 支持 bottom-up 存储。这里用图示详细说明两种存储方式的内存布局差异：

```
=== Top-Down 存储（pitch > 0）===
内存地址增长方向 →

低地址                                           高地址
┌───────────┬───────────┬───────────┬───────────┐
│ 行0(y=0)  │ 行1(y=1)  │ 行2(y=2)  │ 行3(y=3)  │
│ 图像顶部   │           │           │ 图像底部   │
└───────────┴───────────┴───────────┴───────────┘
Bits 指针 ↑


=== Bottom-Up 存储（pitch < 0）===
内存地址增长方向 →

低地址                                           高地址
┌───────────┬───────────┬───────────┬───────────┐
│ 行3(y=3)  │ 行2(y=2)  │ 行1(y=1)  │ 行0(y=0)  │
│ 图像底部   │           │           │ 图像顶部   │
└───────────┴───────────┴───────────┴───────────┘
                                     Bits 指针 ↑
```

在 bottom-up 模式下，`Bits` 指针指向内存中图像第一行（y=0）的位置（最高地址处），`PitchBytes` 为负数。这样 `Bits + y * PitchBytes` 的计算仍然正确：y 增大时地址减小，指向更低内存地址的行。

### 3.2 为什么要支持负 pitch？

1. **Windows 兼容**：Windows `CreateDIBSection` 默认创建 bottom-up DIB
2. **显示硬件**：某些显示控制器（Display Controller）期望 bottom-up 的扫描顺序
3. **历史原因**：CRT（阴极射线管）显示器从屏幕底部开始扫描（电子束从下往上）

### 3.3 处理负 pitch 的代码模式

```cpp
#include <cstdint>
#include <cstdio>
#include <cstdlib>
#include <cstring>

// 支持 top-down 和 bottom-up 的图像缓冲区
struct ImageBuffer {
    uint8_t* data;      // 分配的内存基地址
    uint32_t* bits;     // 逻辑行0的起始地址（可能指向 data 的中间位置）
    int width;
    int height;
    int pitchBytes;     // 正 = top-down, 负 = bottom-up

    // 创建 top-down 图像
    static ImageBuffer createTopDown(int w, int h) {
        ImageBuffer img;
        img.width = w;
        img.height = h;
        img.pitchBytes = w * 4;  // 正 pitch，无填充
        img.data = new uint8_t[h * img.pitchBytes];
        img.bits = reinterpret_cast<uint32_t*>(img.data);  // 行0在最低地址
        return img;
    }

    // 创建 bottom-up 图像（模拟 Windows DIB）
    static ImageBuffer createBottomUp(int w, int h) {
        ImageBuffer img;
        img.width = w;
        img.height = h;
        int rowBytes = w * 4;
        img.pitchBytes = -rowBytes;  // 负 pitch！
        img.data = new uint8_t[h * rowBytes];
        // 行0 指向最后一行的内存位置
        img.bits = reinterpret_cast<uint32_t*>(
            img.data + (h - 1) * rowBytes);
        return img;
    }

    // 获取指定行——与 pitch 正负无关，统一公式
    uint32_t* getScanLine(int y) {
        return reinterpret_cast<uint32_t*>(
            reinterpret_cast<uint8_t*>(bits) + y * pitchBytes);
    }

    void release() { delete[] data; }
};

int main() {
    // 创建两种存储方式的图像
    auto topDown = ImageBuffer::createTopDown(4, 3);
    auto bottomUp = ImageBuffer::createBottomUp(4, 3);

    // 在行0写入红色，行2写入蓝色
    for (int x = 0; x < 4; x++) {
        topDown.getScanLine(0)[x] = 0xFFFF0000;   // 红
        topDown.getScanLine(2)[x] = 0xFF0000FF;   // 蓝
        bottomUp.getScanLine(0)[x] = 0xFFFF0000;  // 红
        bottomUp.getScanLine(2)[x] = 0xFF0000FF;  // 蓝
    }

    // 验证：两种方式的逻辑结果相同
    printf("TopDown  行0第一个像素: 0x%08X\n", topDown.getScanLine(0)[0]);
    printf("BottomUp 行0第一个像素: 0x%08X\n", bottomUp.getScanLine(0)[0]);

    // 但内存布局不同
    printf("\nTopDown  内存首4字节: 0x%08X（行0 = 红色）\n",
           *reinterpret_cast<uint32_t*>(topDown.data));
    printf("BottomUp 内存首4字节: 0x%08X（行2 = 蓝色）\n",
           *reinterpret_cast<uint32_t*>(bottomUp.data));

    topDown.release();
    bottomUp.release();
    return 0;
}
```

预期输出：

```
TopDown  行0第一个像素: 0xFFFF0000
BottomUp 行0第一个像素: 0xFFFF0000

TopDown  内存首4字节: 0xFFFF0000（行0 = 红色）
BottomUp 内存首4字节: 0xFF0000FF（行2 = 蓝色）
```

## 4. 完整跨平台像素工具库

将前两节学到的所有知识整合为一个实用的像素操作工具库：

```cpp
// pixel_toolkit.h — 跨平台像素操作工具库
#pragma once
#include <cstdint>
#include <vector>
#include <fstream>
#include <algorithm>
#include <cstring>

using Pixel = uint32_t;  // KrKr2 中对应 tjs_uint32

// ===== 通道操作 =====
inline uint8_t getA(Pixel p) { return (p >> 24) & 0xFF; }
inline uint8_t getR(Pixel p) { return (p >> 16) & 0xFF; }
inline uint8_t getG(Pixel p) { return (p >> 8)  & 0xFF; }
inline uint8_t getB(Pixel p) { return  p        & 0xFF; }

inline Pixel makePixel(uint8_t a, uint8_t r, uint8_t g, uint8_t b) {
    return (uint32_t(a) << 24) | (uint32_t(r) << 16) |
           (uint32_t(g) << 8) | uint32_t(b);
}

// ===== 格式转换 =====
// ARGB ↔ ABGR（交换 R 和 B，自反操作）
#define TVP_REVRGB(v) \
    ((v & 0xFF00FF00) | ((v >> 16) & 0xFF) | ((v & 0xFF) << 16))

// ARGB → RGBA（将 Alpha 从高位移到低位）
inline Pixel argbToRgba(Pixel v) {
    return (v << 8) | (v >> 24);
}

// RGBA → ARGB
inline Pixel rgbaToArgb(Pixel v) {
    return (v >> 8) | (v << 24);
}

// ===== 预乘 Alpha 转换 =====
inline Pixel toPremultiplied(Pixel p) {
    uint32_t a = getA(p);
    if (a == 255) return p;
    if (a == 0)   return 0;
    uint32_t r = getR(p) * a / 255;
    uint32_t g = getG(p) * a / 255;
    uint32_t b = getB(p) * a / 255;
    return makePixel(a, r, g, b);
}

inline Pixel toStraight(Pixel p) {
    uint32_t a = getA(p);
    if (a == 0)   return 0;
    if (a == 255) return p;
    return makePixel(a,
        std::min(255u, (uint32_t)getR(p) * 255 / a),
        std::min(255u, (uint32_t)getG(p) * 255 / a),
        std::min(255u, (uint32_t)getB(p) * 255 / a));
}

// ===== 图像类 =====
struct Image {
    std::vector<Pixel> pixels;
    int width = 0, height = 0;

    Image() = default;
    Image(int w, int h, Pixel fill = 0xFF000000)
        : pixels(w * h, fill), width(w), height(h) {}

    Pixel& at(int x, int y) { return pixels[y * width + x]; }
    const Pixel& at(int x, int y) const { return pixels[y * width + x]; }

    // 导出 PPM（P6 二进制 RGB，几乎所有图片查看器都支持）
    bool savePPM(const char* path) const {
        std::ofstream f(path, std::ios::binary);
        if (!f) return false;
        f << "P6\n" << width << " " << height << "\n255\n";
        for (auto p : pixels) {
            f.put(getR(p));
            f.put(getG(p));
            f.put(getB(p));
        }
        return f.good();
    }

    // 导出 BMP（Windows 位图格式，支持 Alpha 通道）
    bool saveBMP(const char* path) const {
        std::ofstream f(path, std::ios::binary);
        if (!f) return false;

        // BMP 文件头（14 字节）
        uint32_t dataSize = width * height * 4;
        uint32_t fileSize = 14 + 40 + dataSize;
        uint8_t fileHeader[14] = {
            'B', 'M',                                    // 签名
            uint8_t(fileSize), uint8_t(fileSize >> 8),   // 文件大小（低字节在前）
            uint8_t(fileSize >> 16), uint8_t(fileSize >> 24),
            0, 0, 0, 0,                                  // 保留字段
            54, 0, 0, 0                                  // 像素数据偏移量
        };
        f.write(reinterpret_cast<char*>(fileHeader), 14);

        // BITMAPINFOHEADER（40 字节）
        uint8_t infoHeader[40] = {};
        *reinterpret_cast<uint32_t*>(infoHeader + 0)  = 40;      // 头大小
        *reinterpret_cast<int32_t*>(infoHeader + 4)   = width;   // 宽度
        *reinterpret_cast<int32_t*>(infoHeader + 8)   = -height; // 负值 = top-down
        *reinterpret_cast<uint16_t*>(infoHeader + 12) = 1;       // 颜色平面数
        *reinterpret_cast<uint16_t*>(infoHeader + 14) = 32;      // 位深度
        f.write(reinterpret_cast<char*>(infoHeader), 40);

        // 像素数据（BGRA 顺序——BMP 的标准格式）
        for (auto p : pixels) {
            uint8_t bgra[4] = { getB(p), getG(p), getR(p), getA(p) };
            f.write(reinterpret_cast<char*>(bgra), 4);
        }
        return f.good();
    }
};
```

## 动手实践

使用上面的工具库，编写一个完整的跨平台像素格式转换演示程序：

```cpp
// main.cpp — 跨平台像素格式转换演示
#include <cstdio>
#include <cstdint>
#include <vector>
#include <fstream>
#include <algorithm>

using Pixel = uint32_t;

// --- 通道操作 ---
inline uint8_t getA(Pixel p) { return (p >> 24) & 0xFF; }
inline uint8_t getR(Pixel p) { return (p >> 16) & 0xFF; }
inline uint8_t getG(Pixel p) { return (p >> 8)  & 0xFF; }
inline uint8_t getB(Pixel p) { return  p        & 0xFF; }
inline Pixel makePixel(uint8_t a, uint8_t r, uint8_t g, uint8_t b) {
    return (uint32_t(a) << 24) | (uint32_t(r) << 16) |
           (uint32_t(g) << 8) | uint32_t(b);
}

// --- KrKr2 风格的格式转换 ---
#define TVP_REVRGB(v) \
    ((v & 0xFF00FF00) | ((v >> 16) & 0xFF) | ((v & 0xFF) << 16))

inline Pixel argbToRgba(Pixel v) { return (v << 8) | (v >> 24); }

// --- 简易图像类（含 PPM 导出）---
struct Image {
    std::vector<Pixel> pixels;
    int width = 0, height = 0;
    Image(int w, int h, Pixel fill = 0xFF000000)
        : pixels(w * h, fill), width(w), height(h) {}
    Pixel& at(int x, int y) { return pixels[y * width + x]; }
    void savePPM(const char* path) const {
        std::ofstream f(path, std::ios::binary);
        f << "P6\n" << width << " " << height << "\n255\n";
        for (auto p : pixels) {
            f.put(getR(p)); f.put(getG(p)); f.put(getB(p));
        }
    }
};

void printPixelFormats(Pixel p) {
    printf("=== 像素格式转换演示 ===\n");
    printf("原始 ARGB:  0x%08X  (A=%u R=%u G=%u B=%u)\n",
           p, getA(p), getR(p), getG(p), getB(p));

    Pixel abgr = TVP_REVRGB(p);
    printf("ABGR:       0x%08X  (A=%u B=%u G=%u R=%u)\n",
           abgr, getA(abgr), getR(abgr), getG(abgr), getB(abgr));

    Pixel rgba = argbToRgba(p);
    printf("RGBA:       0x%08X\n", rgba);

    // 验证 TVP_REVRGB 的自反性
    Pixel back = TVP_REVRGB(abgr);
    printf("ABGR→ARGB:  0x%08X  %s\n", back,
           (back == p) ? "✅ 自反验证通过" : "❌ 错误！");
}

// 生成彩色条纹图（展示格式转换效果）
Image createColorBars(int w, int h) {
    Image img(w, h);
    // 8 色标准色条：白、黄、青、绿、品红、红、蓝、黑
    Pixel colors[] = {
        0xFFFFFFFF, 0xFFFFFF00, 0xFF00FFFF, 0xFF00FF00,
        0xFFFF00FF, 0xFFFF0000, 0xFF0000FF, 0xFF000000
    };
    int barWidth = w / 8;
    for (int y = 0; y < h; y++) {
        for (int x = 0; x < w; x++) {
            int bar = std::min(7, x / barWidth);
            img.at(x, y) = colors[bar];
        }
    }
    return img;
}

int main() {
    // 1. 格式转换演示
    printPixelFormats(0x80FF8040);  // 半透明的 R=255 G=128 B=64
    printf("\n");
    printPixelFormats(0xC0336699);  // A=192, R=51, G=102, B=153

    // 2. 批量转换性能测试
    printf("\n=== 批量 R↔B 交换 ===\n");
    const int COUNT = 1000000;
    std::vector<Pixel> buffer(COUNT);
    for (int i = 0; i < COUNT; i++) {
        buffer[i] = makePixel(255, i & 0xFF, (i >> 8) & 0xFF, (i >> 16) & 0xFF);
    }

    // ARGB → ABGR（模拟输出到 Android）
    for (int i = 0; i < COUNT; i++) {
        buffer[i] = TVP_REVRGB(buffer[i]);
    }
    printf("已转换 %d 个像素（ARGB→ABGR）\n", COUNT);

    // 验证：再转回来应该与原值一致
    for (int i = 0; i < COUNT; i++) {
        buffer[i] = TVP_REVRGB(buffer[i]);
    }
    // 抽检
    Pixel expected = makePixel(255, 42, 0, 0);
    printf("抽检 buffer[42]: 0x%08X，期望: 0x%08X  %s\n",
           buffer[42], expected,
           (buffer[42] == expected) ? "✅" : "❌");

    // 3. 生成色条图并保存
    auto bars = createColorBars(256, 64);
    bars.savePPM("color_bars.ppm");
    printf("\n已保存 color_bars.ppm (256x64)\n");

    return 0;
}
```

## 常见错误及解决方案

### 错误 1：忘记 pitch 导致图像错位

```cpp
// ❌ 错误：假设每行紧密排列
uint32_t* pixel = buffer + y * width + x;

// ✅ 正确：使用 pitch（以字节为单位）计算行偏移
uint8_t* row = reinterpret_cast<uint8_t*>(buffer) + y * pitchBytes;
uint32_t* pixel = reinterpret_cast<uint32_t*>(row) + x;
```

**现象**：图像出现斜线条纹，每行向右偏移几个像素。
**原因**：`width * 4` 与 `pitchBytes` 不相等时（有填充字节），按 width 计算地址会逐行累积偏差。

### 错误 2：混淆 Windows COLORREF 与 ARGB

```cpp
// ❌ 错误：把 COLORREF 当 ARGB 用
COLORREF winColor = RGB(255, 0, 0);  // 值为 0x000000FF（不是 0x00FF0000！）
uint32_t argb = 0xFF000000 | winColor;  // 结果是蓝色而非红色！

// ✅ 正确：手动转换 COLORREF → ARGB
uint32_t argb = 0xFF000000 |
    ((winColor & 0xFF) << 16) |       // R
    (winColor & 0xFF00) |              // G
    ((winColor >> 16) & 0xFF);         // B
```

**现象**：Windows 平台上颜色红蓝互换。
**根因**：`COLORREF` 是 `0x00BBGGRR`，R 在最低字节；而 ARGB 是 `0xAARRGGBB`，R 在次高字节。

### 错误 3：Android 上忽略 ARGB↔ABGR 转换

```cpp
// ❌ 错误：直接把 KrKr2 ARGB 像素写入 Android Bitmap
AndroidBitmap_lockPixels(env, bitmap, &pixels);
memcpy(pixels, krkr2_buffer, size);  // 红蓝反转！

// ✅ 正确：逐像素做 R↔B 交换
uint32_t* dst = static_cast<uint32_t*>(pixels);
for (int i = 0; i < pixelCount; i++) {
    dst[i] = TVP_REVRGB(krkr2_buffer[i]);
}
```

**现象**：Android 上画面颜色异常，红色变蓝色、蓝色变红色。

## 本节小结

- **各平台像素格式不同**：Windows 使用 BGRA（但 `uint32_t` 值与 ARGB 相同），macOS 要求预乘 ARGB，Linux 需运行时查询 Visual，Android 使用 ABGR（`RGBA_8888`）
- **KrKr2 统一使用 ARGB** 作为内部格式，仅在平台输出边界做转换，最大限度减少转换开销
- **`TVP_REVRGB` 宏**：3 条位运算交换 R↔B 通道，是 ARGB↔ABGR 转换的核心，且是自反操作（调用两次回到原值）
- **负 pitch（Bottom-Up DIB）**：Windows 的位图默认是 bottom-up 存储，`PitchBytes` 为负值。KrKr2 用有符号整数类型的 `PitchBytes` 统一处理两种存储方向
- **Windows `COLORREF`**：格式是 `0x00BBGGRR`（无 Alpha），与 ARGB 的 R/B 位置相反，是最容易踩的跨平台陷阱之一

### 题目 1：手写全格式转换函数

编写以下三个格式转换函数，并写一个 `main()` 验证它们对 `0xC0336699` 的转换结果：
- `argbToAbgr()` — ARGB → ABGR
- `argbToRgba()` — ARGB → RGBA
- `argbToBgra()` — ARGB → BGRA

<details>
<summary>查看答案</summary>

```cpp
#include <cstdint>
#include <cstdio>

using Pixel = uint32_t;

// ARGB → ABGR：交换 R 和 B，保持 A 和 G
Pixel argbToAbgr(Pixel v) {
    return (v & 0xFF00FF00) | ((v >> 16) & 0xFF) | ((v & 0xFF) << 16);
}

// ARGB → RGBA：Alpha 从最高字节移到最低字节
Pixel argbToRgba(Pixel v) {
    return (v << 8) | (v >> 24);
}

// ARGB → BGRA：交换 A↔B 和 R↔G... 不对！
// ARGB = 0xAARRGGBB  →  BGRA = 0xBBGGRRAA
// 实际上是整个字节序反转（对小端机器来说就是 bswap）
Pixel argbToBgra(Pixel v) {
    return ((v & 0xFF000000) >> 24) |  // A → 最低字节
           ((v & 0x00FF0000) >> 8)  |  // R → 次低字节
           ((v & 0x0000FF00) << 8)  |  // G → 次高字节
           ((v & 0x000000FF) << 24);   // B → 最高字节
}

int main() {
    Pixel p = 0xC0336699;
    printf("原始 ARGB: 0x%08X\n", p);
    printf("ABGR:      0x%08X  (期望: 0xC0996633)\n", argbToAbgr(p));
    printf("RGBA:      0x%08X  (期望: 0x336699C0)\n", argbToRgba(p));
    printf("BGRA:      0x%08X  (期望: 0x996633C0)\n", argbToBgra(p));
    return 0;
}
```

```
RGBA:      0x336699C0  (期望: 0x336699C0)
```

### 题目 2：处理负 pitch 的图像翻转

编写一个函数 `flipVertical(ImageBuffer& img)`，将 top-down 图像翻转为 bottom-up（或反过来），要求就地翻转（不分配新内存）。提示：交换第 i 行和第 (height-1-i) 行的像素。

<details>
<summary>查看答案</summary>

```cpp
#include <cstdint>
#include <cstring>
#include <vector>
#include <cstdio>

struct ImageBuffer {
    uint32_t* pixels;
    int width, height, pitchBytes;  // pitchBytes 正值，无填充
};

void flipVertical(ImageBuffer& img) {
    // 临时缓冲区存放一行像素
    std::vector<uint8_t> tempRow(std::abs(img.pitchBytes));
    int rowBytes = std::abs(img.pitchBytes);

    for (int i = 0; i < img.height / 2; i++) {
        int j = img.height - 1 - i;
        // 计算两行的内存地址
        uint8_t* rowA = reinterpret_cast<uint8_t*>(img.pixels) + i * rowBytes;
        uint8_t* rowB = reinterpret_cast<uint8_t*>(img.pixels) + j * rowBytes;
        // 三步交换
        std::memcpy(tempRow.data(), rowA, rowBytes);
        std::memcpy(rowA, rowB, rowBytes);
        std::memcpy(rowB, tempRow.data(), rowBytes);
    }

    // 翻转后取反 pitch 表示存储方向改变
    img.pitchBytes = -img.pitchBytes;
}

int main() {
    const int W = 4, H = 3;
    uint32_t data[W * H];
    // 行0=红，行1=绿，行2=蓝
    for (int x = 0; x < W; x++) {
        data[0 * W + x] = 0xFFFF0000;  // 红
        data[1 * W + x] = 0xFF00FF00;  // 绿
        data[2 * W + x] = 0xFF0000FF;  // 蓝
    }

    ImageBuffer img = { data, W, H, W * 4 };
    printf("翻转前: 行0=0x%08X 行2=0x%08X\n", data[0], data[2 * W]);

    flipVertical(img);
    printf("翻转后: 行0=0x%08X 行2=0x%08X\n", data[0], data[2 * W]);
    printf("Pitch: %d (负值表示 bottom-up)\n", img.pitchBytes);

    return 0;
}
```

输出：
```
翻转前: 行0=0xFFFF0000 行2=0xFF0000FF
翻转后: 行0=0xFF0000FF 行2=0xFFFF0000
Pitch: -16 (负值表示 bottom-up)
```

### 题目 3：跨平台格式检测工具

编写一个程序，接收一个 `uint32_t` 像素值和它在内存中的 4 个字节（按地址从低到高），自动判断该平台是大端序还是小端序，并输出像素在 ARGB/BGRA/RGBA 三种格式下的解释。

<details>
<summary>查看答案</summary>

```cpp
#include <cstdint>
#include <cstdio>

void detectAndPrint(uint32_t intValue) {
    // 通过检查整数值的第一个字节来判断字节序
    uint8_t* bytes = reinterpret_cast<uint8_t*>(&intValue);

    // 如果最低地址的字节等于整数值的最低 8 位 → 小端序
    bool isLittleEndian = (bytes[0] == (intValue & 0xFF));
    printf("字节序: %s\n", isLittleEndian ? "小端 (Little Endian)" :
                                            "大端 (Big Endian)");

    printf("整数值: 0x%08X\n", intValue);
    printf("内存字节 [低→高]: %02X %02X %02X %02X\n",
           bytes[0], bytes[1], bytes[2], bytes[3]);

    printf("\n解释为不同格式:\n");
    // ARGB 解释（高位到低位 = A, R, G, B）
    printf("  ARGB: A=%u R=%u G=%u B=%u\n",
           (intValue >> 24) & 0xFF, (intValue >> 16) & 0xFF,
           (intValue >> 8) & 0xFF, intValue & 0xFF);
    // BGRA 解释（高位到低位 = B, G, R, A）
    printf("  BGRA: B=%u G=%u R=%u A=%u\n",
           (intValue >> 24) & 0xFF, (intValue >> 16) & 0xFF,
           (intValue >> 8) & 0xFF, intValue & 0xFF);
    // RGBA 解释（高位到低位 = R, G, B, A）
    printf("  RGBA: R=%u G=%u B=%u A=%u\n",
           (intValue >> 24) & 0xFF, (intValue >> 16) & 0xFF,
           (intValue >> 8) & 0xFF, intValue & 0xFF);
}

int main() {
    printf("=== 测试 1: 0xC0336699 ===\n");
    detectAndPrint(0xC0336699);
    printf("\n=== 测试 2: 0xFFFF0000 ===\n");
    detectAndPrint(0xFFFF0000);
    return 0;
}
```
