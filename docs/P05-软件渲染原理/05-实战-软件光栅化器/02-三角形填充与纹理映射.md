## 一、扫描线三角形填充

### 1.1 为什么需要扫描线法

上一节我们用 Bresenham 画了三角形的三条边（线框模式）。但真正的渲染需要**填充**三角形内部——每个在三角形边界内的像素都要着色。这正是光栅化（Rasterization，把几何图形转化为像素的过程）的核心任务。

扫描线法（Scanline Algorithm）的思路很直观：
1. 沿 y 轴从上到下逐行扫描
2. 对每一行 y，计算三角形左右两条边在该 y 值处的 x 坐标（交点）
3. 填充 `[xLeft, xRight]` 区间内的所有像素

这种方法之所以成为经典，是因为它天然适合帧缓冲的行优先内存布局（Frame Buffer 按行存储，连续填充一行像素的缓存命中率（Cache Hit Rate，CPU 缓存找到数据的成功比例）最高）。

### 1.2 平底/平顶三角形分解

并非所有三角形都能直接一步扫描完成。扫描线法的标准做法是：

1. 将三个顶点按 y 值排序：`v0.y ≤ v1.y ≤ v2.y`
2. 判断是否已经是特殊形状：
   - `v1.y == v2.y` → 平底三角形（Flat-Bottom），一个顶点在上，两个顶点在下且 y 值相同
   - `v0.y == v1.y` → 平顶三角形（Flat-Top），两个顶点在上且 y 值相同，一个顶点在下
3. 否则，在 `y = v1.y` 处沿长边做水平切割，拆成一个平底 + 一个平顶

```text
           v0                    v0          切分点 d
          /  \                  / \            /|
         /    \      拆成 →    /   \          / |
        /------\ ← y=v1.y    /-----d   v1---/  |
       /  |    | \           平底△          平顶△ |
      v1--+----+--v2                         \  |
       \  |    | /                            \ |
        \ |    |/                              \|
         v2                                    v2
```

切分点 `d` 的 x 坐标通过插值得到：

```text
t = (v1.y - v0.y) / (v2.y - v0.y)
d.x = v0.x + t * (v2.x - v0.x)
d.y = v1.y
```

### 1.3 填充平底三角形

平底三角形：顶点 v0 在上，v1 和 v2 在下且 `v1.y == v2.y`。从 v0 向下扫描到 v1.y：

```cpp
// 文件：main.cpp（三角形填充模块）
#include <algorithm>  // std::min, std::max, std::swap
#include <cmath>      // std::ceil, std::floor

struct Vec2f { float x, y; };

// 填充平底三角形：v0 在顶部，v1/v2 在底部（v1.y == v2.y）
static void fillFlatBottom(FrameBuffer& fb,
                           Vec2f v0, Vec2f v1, Vec2f v2,
                           uint32_t color)
{
    // invslope = 每向下走 1 像素行，x 方向移动多少
    // 即边的斜率的倒数（inverse slope，Δx/Δy）
    float invslope1 = (v1.x - v0.x) / (v1.y - v0.y);
    float invslope2 = (v2.x - v0.x) / (v2.y - v0.y);

    // 当前扫描线左右端点的 x 坐标（从 v0 出发）
    float curx1 = v0.x;
    float curx2 = v0.x;

    // 像素坐标取整：y 向上取整避免越界
    int yStart = std::max(0, (int)std::ceil(v0.y));
    int yEnd   = std::min(fb.height - 1, (int)std::floor(v1.y));

    // 补偿：v0.y 不在整数像素上时，先推进到第一个整数行
    curx1 += (yStart - v0.y) * invslope1;
    curx2 += (yStart - v0.y) * invslope2;

    for (int y = yStart; y <= yEnd; ++y) {
        // 当前行的左右 x 边界
        int xL = (int)std::ceil(std::min(curx1, curx2));
        int xR = (int)std::floor(std::max(curx1, curx2));
        // 裁剪到帧缓冲范围
        xL = std::max(0, xL);
        xR = std::min(fb.width - 1, xR);

        for (int x = xL; x <= xR; ++x)
            fb.putPixelRaw(x, y, color);

        // 下一行：两条边各推进一个 invslope
        curx1 += invslope1;
        curx2 += invslope2;
    }
}
```

**关键细节：**
- `std::ceil` 对左边界取上整、`std::floor` 对右边界取下整——这保证只填充真正在三角形**内部**的像素
- 裁剪（Clipping，将超出画面的部分丢弃）到 `[0, width-1]` × `[0, height-1]`，避免越界写入

### 1.4 填充平顶三角形

平顶三角形：v0 和 v1 在上（`v0.y == v1.y`），v2 在下。从顶部向下扫描到 v2.y：

```cpp
// 填充平顶三角形：v0/v1 在顶部（v0.y == v1.y），v2 在底部
static void fillFlatTop(FrameBuffer& fb,
                        Vec2f v0, Vec2f v1, Vec2f v2,
                        uint32_t color)
{
    // 两条边分别从 v0→v2 和 v1→v2
    float invslope1 = (v2.x - v0.x) / (v2.y - v0.y);
    float invslope2 = (v2.x - v1.x) / (v2.y - v1.y);

    int yStart = std::max(0, (int)std::ceil(v0.y));
    int yEnd   = std::min(fb.height - 1, (int)std::floor(v2.y));

    // 分别从两个顶点出发，补偿到第一个整数行
    float curx1 = v0.x + (yStart - v0.y) * invslope1;
    float curx2 = v1.x + (yStart - v1.y) * invslope2;

    for (int y = yStart; y <= yEnd; ++y) {
        int xL = (int)std::ceil(std::min(curx1, curx2));
        int xR = (int)std::floor(std::max(curx1, curx2));
        xL = std::max(0, xL);
        xR = std::min(fb.width - 1, xR);

        for (int x = xL; x <= xR; ++x)
            fb.putPixelRaw(x, y, color);

        curx1 += invslope1;
        curx2 += invslope2;
    }
}
```

### 1.5 通用三角形填充入口

将排序、判断、拆分逻辑整合成一个统一入口：

```cpp
// 通用三角形填充：任意三角形 → 排序 → 判断/拆分 → 填充
static void fillTriangleScanline(FrameBuffer& fb,
                                 Vec2f a, Vec2f b, Vec2f c,
                                 uint32_t color)
{
    // 步骤 1：按 y 排序（冒泡排三次即可）
    if (a.y > b.y) std::swap(a, b);
    if (a.y > c.y) std::swap(a, c);
    if (b.y > c.y) std::swap(b, c);
    // 此时 a.y <= b.y <= c.y

    // 步骤 2：判断特殊情况
    if ((int)std::round(b.y) == (int)std::round(c.y)) {
        // 平底三角形
        fillFlatBottom(fb, a, b, c, color);
    } else if ((int)std::round(a.y) == (int)std::round(b.y)) {
        // 平顶三角形
        fillFlatTop(fb, a, b, c, color);
    } else {
        // 步骤 3：拆成平底 + 平顶
        float t = (b.y - a.y) / (c.y - a.y);
        Vec2f d { a.x + t * (c.x - a.x), b.y };  // 切分点
        fillFlatBottom(fb, a, b, d, color);        // 上半部分
        fillFlatTop(fb, b, d, c, color);           // 下半部分
    }
}
```

**为什么用 `std::round` 比较而不是直接 `==`？** 浮点数直接比较容易因精度误差（Floating-Point Precision Error）失败。取整到最近整数后再比较，容差约 0.5 像素，对光栅化足够。

## 二、纹理映射基础

### 2.1 什么是纹理映射

纯色填充只是第一步。要让三角形呈现出图片效果，我们需要把一张 2D 图像（纹理，Texture）"贴"到三角形表面。具体做法是：给每个顶点分配一对 **UV 坐标**（也叫纹理坐标，Texture Coordinates），范围通常为 `[0, 1]`，其中 `(0,0)` 是纹理左上角，`(1,1)` 是纹理右下角。光栅化时，对三角形内部每个像素，通过插值算出该像素对应的 UV 值，然后去纹理图上查颜色——这个查颜色的过程叫**采样**（Sampling）。

### 2.2 纹理数据结构

```cpp
// 文件：main.cpp（纹理模块）
#include <vector>
#include <cstdint>
#include <cmath>
#include <algorithm>

struct Texture {
    int w = 0, h = 0;                  // 纹理宽高（像素）
    std::vector<uint32_t> data;        // ARGB 像素数据

    // 安全取像素：越界时 clamp（钳制）到边缘
    uint32_t get(int x, int y) const {
        x = std::clamp(x, 0, w - 1);  // clamp：限制值在 [min, max] 范围内
        y = std::clamp(y, 0, h - 1);
        return data[size_t(y) * size_t(w) + size_t(x)];
    }
};
```

`std::clamp(x, 0, w-1)` 实现了边缘重复（Clamp-to-Edge）的纹理环绕模式——当 UV 超出 `[0,1]` 范围导致像素坐标越界时，取边缘像素而不是访问非法内存。这是最常用的环绕模式，OpenGL 中对应 `GL_CLAMP_TO_EDGE`。

### 2.3 最近邻采样

最近邻采样（Nearest-Neighbor Sampling）是最简单的方法：把 UV 坐标乘以纹理尺寸，四舍五入到最近的整数像素坐标，直接取该像素的颜色：

```cpp
// 最近邻采样：UV → 最近整数像素坐标 → 取色
uint32_t sampleNearest(const Texture& tex, float u, float v) {
    // u,v 在 [0,1] 范围，乘以 (w-1) 得到像素坐标
    int x = (int)std::round(u * (tex.w - 1));
    int y = (int)std::round(v * (tex.h - 1));
    return tex.get(x, y);
}
```

**优点：** 速度最快，一次采样只读一个像素。
**缺点：** 放大时出现明显的"马赛克"方块感（像素锯齿），缩小时出现闪烁（走样，Aliasing）。

### 2.4 双线性插值采样

双线性插值（Bilinear Interpolation）读取目标点周围的 4 个像素，按距离加权混合，产生平滑的结果：

```text
UV 坐标 (fx, fy) 落在四个整数像素之间：

    (x0,y0)-----(x1,y0)
       |    ·P    |        P 的颜色 = 四个角按 (tx, ty) 加权混合
       |          |        tx = fx - x0（水平偏移）
    (x0,y1)-----(x1,y1)   ty = fy - y0（垂直偏移）
```

```cpp
// 双线性插值采样：读 4 个邻近像素，按距离加权混合
uint32_t sampleBilinear(const Texture& tex, float u, float v) {
    float fx = u * (tex.w - 1);  // 连续坐标
    float fy = v * (tex.h - 1);
    int x0 = (int)std::floor(fx);  // 左上角整数坐标
    int y0 = (int)std::floor(fy);
    int x1 = x0 + 1;               // 右下角
    int y1 = y0 + 1;
    float tx = fx - x0;            // 水平插值权重 [0, 1)
    float ty = fy - y0;            // 垂直插值权重 [0, 1)

    // 辅助函数：单通道线性插值
    auto lerp8 = [](int a, int b, float t) -> int {
        return (int)std::round(a + (b - a) * t);
    };

    // 取四个角的像素
    auto c00 = tex.get(x0, y0);  // 左上
    auto c10 = tex.get(x1, y0);  // 右上
    auto c01 = tex.get(x0, y1);  // 左下
    auto c11 = tex.get(x1, y1);  // 右下

    // 对 R/G/B/A 四个通道分别做两次水平插值 + 一次垂直插值
    int r0 = lerp8((c00 >> 16) & 255, (c10 >> 16) & 255, tx);  // 上边水平
    int g0 = lerp8((c00 >>  8) & 255, (c10 >>  8) & 255, tx);
    int b0 = lerp8( c00        & 255,  c10        & 255, tx);
    int a0 = lerp8((c00 >> 24) & 255, (c10 >> 24) & 255, tx);

    int r1 = lerp8((c01 >> 16) & 255, (c11 >> 16) & 255, tx);  // 下边水平
    int g1 = lerp8((c01 >>  8) & 255, (c11 >>  8) & 255, tx);
    int b1 = lerp8( c01        & 255,  c11        & 255, tx);
    int a1 = lerp8((c01 >> 24) & 255, (c11 >> 24) & 255, tx);

    // 最后一次垂直插值
    int r = lerp8(r0, r1, ty);
    int g = lerp8(g0, g1, ty);
    int b = lerp8(b0, b1, ty);
    int a = lerp8(a0, a1, ty);

    return (uint32_t(a) << 24) | (uint32_t(r) << 16) |
           (uint32_t(g) << 8)  |  uint32_t(b);
}
```

| 采样方式 | 每像素读取次数 | 视觉质量 | 速度 | 适用场景 |
|----------|---------------|----------|------|----------|
| 最近邻 | 1 次 | 马赛克感 | 最快 | 像素风格、1:1 不缩放 |
| 双线性 | 4 次 | 平滑过渡 | 较慢（约 4×） | 通用纹理映射 |

## 三、透视校正插值

### 3.1 为什么屏幕空间插值会失真

在 3D 场景中，近大远小——一个正方形地板在透视投影下，近处占更多像素、远处被压缩。如果我们直接在屏幕空间对 UV 坐标做线性插值，纹理会被"拉花"（近处和远处的纹理密度不正确），因为屏幕空间的线性关系不等于 3D 空间的线性关系。

```text
错误（屏幕空间线性插值）：          正确（透视校正插值）：

  /‾‾‾‾‾‾‾‾‾\                      /‾‾‾‾‾‾‾‾‾\
 /  ||||||||| \                    /  ||| | |  | \
/   ||||||||| |\                  /   ||  |  |  |  \
纹理均匀分布 → 近远密度一样        纹理近密远疏 → 符合透视
```

### 3.2 透视校正公式

解决方法是利用齐次坐标（Homogeneous Coordinates，用 4 维 `(x,y,z,w)` 表示 3D 点的数学表示法）中的 `w` 分量。关键推导：

```text
在透视投影后，顶点的齐次坐标为 (x, y, z, w)
透视除法后得到 NDC：x' = x/w, y' = y/w, z' = z/w

属性（如 UV）在 3D 空间中是线性的，但在屏幕空间中不是。
然而，u/w 和 v/w 在屏幕空间中是线性的！

所以正确的做法是：
  1. 对每个顶点，预计算 u/w、v/w、1/w
  2. 在屏幕空间对 u/w、v/w、1/w 做线性插值
  3. 在每个像素处，恢复真实的 u = (u/w) / (1/w)、v = (v/w) / (1/w)
```

### 3.3 实现透视校正的扫描线

```cpp
// 带透视校正的顶点属性
struct VertexAttr {
    float u, v;     // 纹理坐标
    float w;        // 齐次 w（来自投影矩阵输出）
};

// 在扫描线中透视校正插值 UV
// 假设已在扫描线左右端点插值好了 u/w、v/w、1/w
static void scanlineTextured(FrameBuffer& fb, const Texture& tex,
                             int y, int xL, int xR,
                             float uOverW_L, float vOverW_L, float oneOverW_L,
                             float uOverW_R, float vOverW_R, float oneOverW_R)
{
    if (xL > xR) return;
    float invSpan = 1.0f / (xR - xL);  // 扫描线宽度的倒数

    for (int x = xL; x <= xR; ++x) {
        // t: 当前像素在扫描线上的插值比例 [0, 1]
        float t = (xR == xL) ? 0.0f : (float)(x - xL) * invSpan;

        // 线性插值 u/w、v/w、1/w（这三个量在屏幕空间是线性的）
        float uOverW = uOverW_L + (uOverW_R - uOverW_L) * t;
        float vOverW = vOverW_L + (vOverW_R - vOverW_L) * t;
        float oneOverW = oneOverW_L + (oneOverW_R - oneOverW_L) * t;

        // 恢复真实 UV：除以 1/w 即乘以 w
        float u = uOverW / oneOverW;
        float v = vOverW / oneOverW;

        // 采样纹理
        uint32_t texColor = sampleBilinear(tex, u, v);
        fb.putPixelRaw(x, y, texColor);
    }
}
```

**性能提示：** 每像素一次浮点除法（`uOverW / oneOverW`）是透视校正的主要开销。生产级光栅器通常用"每 N 像素做一次精确除法 + 中间像素线性逼近"的方式分摊成本（被称为 affine span，仿射段优化）。

## 四、Alpha 混合集成（Porter-Duff Over）

### 4.1 为什么需要混合

纹理通常包含透明区域（Alpha 通道不是 255）。比如游戏中的角色立绘，人物周围是透明的。如果直接覆盖帧缓冲，透明区域会显示为黑色或错误颜色。我们需要把纹理颜色和帧缓冲中已有的背景颜色**混合**。

标准混合公式是 Porter-Duff "Source Over"（详见 [P05/Ch2](../02-Alpha混合与合成/01-Alpha通道与标准混合.md)）：

```text
Out = Src Over Dst

Ao = As + Ad × (1 - As)
Co = (Cs × As + Cd × Ad × (1 - As)) / Ao

预乘格式下更简单：
Co_premul = Cs_premul + Cd_premul × (1 - As)
```

### 4.2 非预乘实现

我们的教学光栅器使用非预乘（Straight Alpha，颜色通道未乘以 Alpha 值的格式）输入：

```cpp
// Porter-Duff Source Over（非预乘输入）
// dst: 帧缓冲中已有的像素（背景）
// src: 新绘制的像素（前景）
static inline uint32_t alphaOver(uint32_t dst, uint32_t src) {
    int sa = (src >> 24) & 255;  // 源 Alpha
    int sr = (src >> 16) & 255;  // 源 Red
    int sg = (src >>  8) & 255;  // 源 Green
    int sb =  src        & 255;  // 源 Blue

    int da = (dst >> 24) & 255;  // 目标 Alpha
    int dr = (dst >> 16) & 255;
    int dg = (dst >>  8) & 255;
    int db =  dst        & 255;

    int invSa = 255 - sa;        // 1 - As（8 位整数版）

    // 标准 Porter-Duff Over 公式（整数近似，+127 用于四舍五入）
    int outA = sa + ((da * invSa + 127) / 255);
    int outR = (sr * sa + dr * invSa + 127) / 255;
    int outG = (sg * sa + dg * invSa + 127) / 255;
    int outB = (sb * sa + db * invSa + 127) / 255;

    return (uint32_t(outA) << 24) | (uint32_t(outR) << 16) |
           (uint32_t(outG) <<  8) |  uint32_t(outB);
}
```

这与 KrKr2 中 `tvpgl.h` / `tvpgl.cpp` 的实现思路一致：
- `TVPAlphaBlend_*` 系列函数负责不同目标 Alpha 语义的混合
- `TVPOpacityOnOpacityTable`、`TVPNegativeMulTable` 用预计算查找表（LUT，Look-Up Table）加速除法
- `TVPSaturatedAdd` 用位运算实现通道饱和加法（Saturated Addition，加法结果超过 255 时钳制为 255），避免逐通道分支判断

## 五、SSE2 加速扫描线混合

### 5.1 热点分析

扫描线填充时，逐像素调用 `alphaOver` 是性能热点（Hot Spot，程序中消耗最多 CPU 时间的代码段）。一帧画面 640×360 = 230,400 像素，每像素 4 次乘法 + 4 次除法——在 CPU 上这很慢。

SSE2（Streaming SIMD Extensions 2，Intel 提供的 128 位向量指令集）可以一次处理 4 个 32 位像素（128 位 ÷ 32 位 = 4 个），吞吐量理论上提升 4 倍。

### 5.2 SSE2 向量化混合

```cpp
// 文件：main.cpp（SIMD 混合模块）
#include <emmintrin.h>  // SSE2 内建函数头文件

// SSE2 加速的批量 Alpha 混合
// dst: 目标像素数组（帧缓冲行指针）
// src: 源像素数组（纹理行数据）
// count: 像素个数
static void blendSpanSSE2(uint32_t* dst,
                          const uint32_t* src,
                          int count)
{
    int i = 0;
    const __m128i zero = _mm_setzero_si128();  // 全零向量，用于解包

    // 主循环：每次处理 4 个像素
    for (; i + 3 < count; i += 4) {
        // 加载 4 个源像素和 4 个目标像素（128 位 = 4×32 位）
        // _mm_loadu_si128: 不要求 16 字节对齐的加载
        __m128i s = _mm_loadu_si128(
            reinterpret_cast<const __m128i*>(src + i));
        __m128i d = _mm_loadu_si128(
            reinterpret_cast<const __m128i*>(dst + i));

        // 解包 8 位 → 16 位（为乘法留出空间，避免溢出）
        // _mm_unpacklo_epi8: 交错取低 64 位的字节，与 zero 交错得到 16 位值
        __m128i sLo = _mm_unpacklo_epi8(s, zero);  // 前 2 像素
        __m128i sHi = _mm_unpackhi_epi8(s, zero);  // 后 2 像素
        __m128i dLo = _mm_unpacklo_epi8(d, zero);
        __m128i dHi = _mm_unpackhi_epi8(d, zero);

        // 提取源 Alpha 并广播到同一像素的所有通道
        // _mm_shufflelo_epi16(..., _MM_SHUFFLE(3,3,3,3)):
        //   把低 64 位中第 3 个 16 位值（Alpha）复制到 4 个位置
        __m128i aLo = _mm_shufflelo_epi16(sLo, _MM_SHUFFLE(3,3,3,3));
        aLo = _mm_shufflehi_epi16(aLo, _MM_SHUFFLE(3,3,3,3));
        __m128i aHi = _mm_shufflelo_epi16(sHi, _MM_SHUFFLE(3,3,3,3));
        aHi = _mm_shufflehi_epi16(aHi, _MM_SHUFFLE(3,3,3,3));

        // invAlpha = 255 - alpha
        __m128i invALo = _mm_sub_epi16(_mm_set1_epi16(255), aLo);
        __m128i invAHi = _mm_sub_epi16(_mm_set1_epi16(255), aHi);

        // out = src * alpha + dst * (255 - alpha)
        __m128i outLo = _mm_add_epi16(
            _mm_mullo_epi16(sLo, aLo),
            _mm_mullo_epi16(dLo, invALo));
        __m128i outHi = _mm_add_epi16(
            _mm_mullo_epi16(sHi, aHi),
            _mm_mullo_epi16(dHi, invAHi));

        // 除以 255 的近似：(x + 127) >> 8
        outLo = _mm_srli_epi16(
            _mm_add_epi16(outLo, _mm_set1_epi16(127)), 8);
        outHi = _mm_srli_epi16(
            _mm_add_epi16(outHi, _mm_set1_epi16(127)), 8);

        // 打包回 8 位并存储
        // _mm_packus_epi16: 把 16 位有符号数饱和压缩到 8 位无符号数
        __m128i out = _mm_packus_epi16(outLo, outHi);
        _mm_storeu_si128(reinterpret_cast<__m128i*>(dst + i), out);
    }

    // 尾部处理：不足 4 个像素的部分用标量版
    for (; i < count; ++i)
        dst[i] = alphaOver(dst[i], src[i]);
}
```

**教学 vs 生产的差异：**

| 方面 | 本节教学版 | KrKr2 生产版 |
|------|-----------|-------------|
| 输入格式 | 非预乘 ARGB | 预乘 ARGB（`_a` 后缀函数） |
| 除法方式 | `(x + 127) >> 8` | `TVPDivTable[256*256]` 查表 |
| 尾部处理 | 标量回退 | 独立的标量循环或 SSSE3 优化 |
| Alpha 语义 | 统一 Over | 30+ 混合模式分发 |

## 六、完整渲染管线（MVP 到屏幕）

### 6.1 3D 到 2D 的流程

要把 3D 世界中的三角形显示到 2D 屏幕上，需要经过一系列坐标变换，统称 MVP 管线：

```text
模型空间 ──Model矩阵──→ 世界空间 ──View矩阵──→ 观察空间 ──Projection矩阵──→ 裁剪空间
    │
    └→ 透视除法 → NDC 空间 → 屏幕映射 → 屏幕像素坐标
```

- **Model 矩阵（模型矩阵）**：把物体从局部坐标系移到世界坐标系（包含平移、旋转、缩放）
- **View 矩阵（视图矩阵）**：把世界坐标系变换到以摄像机为中心的坐标系
- **Projection 矩阵（投影矩阵）**：把 3D 空间压缩到一个标准化的裁剪体积内
- **透视除法（Perspective Division）**：`(x,y,z,w) → (x/w, y/w, z/w)` 得到 NDC（Normalized Device Coordinates，标准化设备坐标，范围 `[-1, 1]`）

### 6.2 数学工具

```cpp
// 文件：main.cpp（数学工具模块）
struct Vec3 { float x, y, z; };
struct Vec4 { float x, y, z, w; };

struct Mat4 {
    float m[16];  // 列主序存储（Column-Major，和 OpenGL 一致）

    // 单位矩阵：对角线为 1，其余为 0
    static Mat4 identity() {
        Mat4 r{};
        r.m[0] = r.m[5] = r.m[10] = r.m[15] = 1.0f;
        return r;
    }
};

// 矩阵-向量乘法（列主序）
// 列主序意味着 m[0..3] 是第一列，m[4..7] 是第二列，以此类推
static Vec4 mul(const Mat4& a, const Vec4& v) {
    Vec4 r{};
    r.x = a.m[0]*v.x + a.m[4]*v.y + a.m[8] *v.z + a.m[12]*v.w;
    r.y = a.m[1]*v.x + a.m[5]*v.y + a.m[9] *v.z + a.m[13]*v.w;
    r.z = a.m[2]*v.x + a.m[6]*v.y + a.m[10]*v.z + a.m[14]*v.w;
    r.w = a.m[3]*v.x + a.m[7]*v.y + a.m[11]*v.z + a.m[15]*v.w;
    return r;
}
```

### 6.3 NDC 到屏幕的映射

```cpp
// NDC [-1, 1] → 屏幕像素 [0, w-1] × [0, h-1]
// 注意 y 轴翻转：NDC 的 y=+1 对应屏幕顶部（y=0）
static Vec3 ndcToScreen(const Vec3& n, int w, int h) {
    float sx = (n.x * 0.5f + 0.5f) * (float)(w - 1);
    // y 翻转：图像内存从上到下存储，y=0 是第一行
    float sy = (1.0f - (n.y * 0.5f + 0.5f)) * (float)(h - 1);
    return {sx, sy, n.z};
}
```

### 6.4 三个最容易被忽略的陷阱

1. **齐次裁剪必须在裁剪空间做**——在 `-w ≤ x,y,z ≤ w` 的范围内判断，而不是在 NDC 空间（因为透视除法后 `w < 0` 的顶点会映射到错误位置）
2. **透视除法后才进入 NDC**——先裁剪，再除 `w`
3. **屏幕映射时 y 轴必须翻转**——帧缓冲内存按行从上到下存储，y=0 对应图像顶部，而 NDC 的 y=+1 是顶部

## 常见错误及解决方案

### 错误 1：相邻三角形共享边出现"裂缝"

**现象：** 两个三角形共享一条边时，边上出现亮线或黑线（像素被填充了两次或零次）。

**原因：** 左右边取整规则不一致——一个三角形用 `floor`，另一个用 `ceil`，导致共享边的像素归属不确定。

**解决：** 统一采用 **Top-Left 规则**（Top-Left Fill Convention）：
- 水平边只填充顶部的那条
- 非水平边只填充左侧的那条
- 具体实现：统一使用 `[ceil(xL), floor(xR)]` 且保证共享边只归属一侧

### 错误 2：纹理在透视角度下被"拉花"

**现象：** 近大远小的面上，纹理线条扭曲、不自然。

**原因：** 直接在屏幕空间线性插值 `u,v`，未做透视校正。

**解决：** 改为插值 `u/w`、`v/w`、`1/w`，每像素恢复 `u = (u/w) / (1/w)`。详见本节第三章。

### 错误 3：Alpha 混合后整体偏暗

**现象：** 半透明图层叠加后颜色偏暗、发灰。

**原因：** 把预乘 Alpha（Premultiplied Alpha，`R = R_original × A`）和非预乘 Alpha（Straight Alpha）路径混用。

**解决：** 明确输入格式。非预乘用 `src × sa + dst × (1-sa)`；预乘用 `src_premul + dst_premul × (1-sa)`。两者公式不同，不可混用。

## 本节小结

- **扫描线法**通过平底/平顶分解，把任意三角形的填充化归为逐行左右端点计算 + 水平填充，天然契合帧缓冲行存储
- **纹理映射**通过 UV 坐标把 2D 图像贴到三角形上，最近邻采样快但有马赛克，双线性插值平滑但读 4 个像素
- **透视校正**利用齐次 `w` 分量——对 `u/w`、`v/w`、`1/w` 做屏幕空间线性插值后除回，解决透视失真
- **Porter-Duff Over** 实现半透明混合，KrKr2 的 `TVPAlphaBlend_*` 是同一思路的工业强化版
- **SSE2** 一次处理 4 像素的混合热点，是"解包→乘→移位→打包"的标准流程
- **MVP 管线**经过模型→视图→投影→裁剪→透视除法→屏幕映射六步，将 3D 顶点投影到 2D 像素

### 题目 2：若纹理数据已经是预乘 Alpha 格式，`alphaOver` 函数应如何修改？

<details>
<summary>查看答案</summary>

预乘格式下，颜色通道已经乘过 Alpha，所以混合公式变为：

```cpp
static inline uint32_t alphaOverPremul(uint32_t dst, uint32_t src) {
    int sa = (src >> 24) & 255;
    int invSa = 255 - sa;

    // 预乘格式：颜色通道已含 Alpha，不再乘 sa
    // out = src + dst * (1 - sa)
    int outR = ((src >> 16) & 255) + ((((dst >> 16) & 255) * invSa + 127) / 255);
    int outG = ((src >>  8) & 255) + ((((dst >>  8) & 255) * invSa + 127) / 255);
    int outB = ( src        & 255) + ((( dst        & 255) * invSa + 127) / 255);
    int outA = sa + ((((dst >> 24) & 255) * invSa + 127) / 255);

    return (uint32_t(std::min(outA, 255)) << 24) |
           (uint32_t(std::min(outR, 255)) << 16) |
           (uint32_t(std::min(outG, 255)) <<  8) |
            uint32_t(std::min(outB, 255));
}
```

关键区别：颜色通道不再乘 `sa`（因为已经预乘过了），直接加上 `dst * (1-sa)`。

</details>

### 题目 3：为什么 KrKr2 要在初始化时建立 `TVPDivTable[256×256]` 这样的查找表？

<details>
<summary>查看答案</summary>

因为整数除法 `/255` 在像素循环中非常昂贵且难以向量化：
1. **CPU 整数除法指令**比乘法慢 5-20 倍（取决于架构）
2. **SIMD 指令集**（SSE2/NEON）不直接提供整数除法指令
3. 查表把除法转化为**一次内存读取**（`TVPDivTable[a * 256 + b]`），延迟约 4 个周期
4. 64KB 的表完全装入 L1 数据缓存（Data Cache，CPU 最快的缓存层，通常 32-64KB），访问延迟稳定

运行时变成一条 `mov` 指令，吞吐比除法高一个数量级。这是经典的"空间换时间"优化。

</details>
