## 一、基准测试原理

### 1.1 为什么需要基准测试

"感觉快了"不算数——性能优化必须有数字支撑。基准测试（Benchmark）是用可重复的实验量化性能差异的唯一正确方法。

一个有效的基准测试需要满足三个条件：
1. **同输入**：标量版和 SIMD 版使用完全相同的输入数据
2. **同输出**：两者产生的结果必须一致（误差在可接受范围内）
3. **足够轮次**：重复执行足够多次（通常 100-1000 次），让统计结果稳定

### 1.2 常见陷阱

| 陷阱 | 现象 | 解决方案 |
|------|------|----------|
| 编译器优化掉被测代码 | 两种实现耗时相同且接近 0 | 使用计算结果（如打印或累加），阻止死代码消除（DCE，Dead Code Elimination） |
| 缓存未预热 | 第一轮耗时远大于后续轮次 | 先跑 1-2 轮"热身"，不计入统计 |
| 数据量太小 | SIMD 版反而更慢 | 确保数据量 ≥ 数千像素，让向量化循环的启动开销（Pipeline Fill Overhead）被分摊 |
| Debug 模式编译 | `-O0` 下 SIMD 内建函数不会被优化，性能数据无参考价值 | 始终在 `-O2` 或 `/O2` 下测量 |
| 测量包含 I/O | 文件写入耗时混入测量 | 只测核心计算循环，I/O 在测量外执行 |

### 1.3 计时工具

C++17 提供了 `<chrono>` 标准库，可以获取高精度时间戳（High-Resolution Clock）：

```cpp
#include <chrono>

// 获取当前时间点
auto t0 = std::chrono::high_resolution_clock::now();

// ... 执行被测代码 ...

auto t1 = std::chrono::high_resolution_clock::now();

// 计算耗时（毫秒）
double ms = std::chrono::duration<double, std::milli>(t1 - t0).count();
```

`high_resolution_clock` 在不同平台的精度：

| 平台 | 典型精度 | 底层实现 |
|------|----------|----------|
| Windows | ~100 纳秒 | `QueryPerformanceCounter`（硬件计数器 API） |
| Linux | ~1 纳秒 | `clock_gettime(CLOCK_MONOTONIC)` |
| macOS | ~1 纳秒 | `mach_absolute_time()` |
| Android | ~1 纳秒 | 同 Linux |

## 二、标量 vs SIMD 基准测试

### 2.1 标量版基准

```cpp
// 文件：main.cpp（基准测试模块）
#include <chrono>
#include <vector>
#include <cstdint>
#include <iostream>

// alphaOver 函数定义见上一节

// 标量混合基准：逐像素调用 alphaOver
static double benchmarkBlendScalar(
    std::vector<uint32_t>& dst,
    const std::vector<uint32_t>& src,
    int rounds)
{
    auto t0 = std::chrono::high_resolution_clock::now();

    for (int r = 0; r < rounds; ++r) {
        for (size_t i = 0; i < dst.size(); ++i) {
            dst[i] = alphaOver(dst[i], src[i]);
        }
    }

    auto t1 = std::chrono::high_resolution_clock::now();
    return std::chrono::duration<double, std::milli>(t1 - t0).count();
}
```

### 2.2 SIMD 版基准

```cpp
// blendSpanSSE2 函数定义见上一节

// SIMD 混合基准：批量调用 blendSpanSSE2
static double benchmarkBlendSIMD(
    std::vector<uint32_t>& dst,
    const std::vector<uint32_t>& src,
    int rounds)
{
    auto t0 = std::chrono::high_resolution_clock::now();

    for (int r = 0; r < rounds; ++r) {
        blendSpanSSE2(dst.data(), src.data(), (int)dst.size());
    }

    auto t1 = std::chrono::high_resolution_clock::now();
    return std::chrono::duration<double, std::milli>(t1 - t0).count();
}
```

### 2.3 运行与解读

```cpp
// 在 main() 中调用
int pixelCount = 640 * 360;  // 230,400 像素
int rounds = 200;

std::vector<uint32_t> src(pixelCount, 0x803399FF);  // 半透明蓝紫色
std::vector<uint32_t> dstA(pixelCount, 0xFF101018); // 深色背景（标量）
std::vector<uint32_t> dstB = dstA;                   // 深色背景（SIMD）

double scalarMs = benchmarkBlendScalar(dstA, src, rounds);
double simdMs   = benchmarkBlendSIMD(dstB, src, rounds);

std::cout << "标量(ms): " << scalarMs << "\n";
std::cout << "SIMD(ms): " << simdMs   << "\n";
std::cout << "加速比:   " << scalarMs / simdMs << "x\n";
```

典型结果（Release 编译，i7-12700K）：

```text
标量(ms): 312.5
SIMD(ms): 89.2
加速比:   3.50x
```

加速比通常在 2.5×-4.0× 之间，取决于：
- CPU 微架构（Microarchitecture，CPU 内部的执行单元设计）
- 内存带宽（Memory Bandwidth，内存数据传输速率）是否成为瓶颈
- 编译器是否自动向量化了标量版

## 三、常见错误及解决方案

### 错误 1：三角形边缘出现"裂缝"或重复填充

**现象：** 相邻三角形共享边时出现亮线或黑线。

**原因：** 左右边取整规则不一致（一个三角形用 `floor`，另一个用 `ceil`），导致共享边上的像素被两个三角形都填充（亮线）或都跳过（黑线）。

**解决：** 统一采用 Top-Left 填充规则，或始终使用 `[ceil(xL), floor(xR)]` 并保证共享边只归属一侧。GPU 硬件光栅器内部也使用类似规则（通常是 Top-Left Rule + 子像素精度）。

### 错误 2：纹理在透视角度下被"拉花"

**现象：** 近大远小时，纹理线条扭曲，地板格子不是正方形。

**原因：** 直接线性插值 `u,v`，没有做透视校正。屏幕空间的线性关系不等于 3D 空间的线性关系。

**解决：** 改为插值 `u/w`、`v/w`、`1/w`，每像素恢复真实 UV。详见上一节第三章。

### 错误 3：Alpha 混合后整体偏暗或发灰

**现象：** 半透明图层叠加后颜色不对，明显偏暗。

**原因：** 把预乘 Alpha（Premultiplied Alpha，颜色通道 = 原始颜色 × Alpha）与非预乘 Alpha（Straight Alpha，颜色通道 = 原始颜色）路径混用。预乘数据用非预乘公式混合会多乘一次 Alpha，导致暗化。

**解决：** 统一输入格式。非预乘：`out = src × sa + dst × (1-sa)`；预乘：`out = src_premul + dst_premul × (1-sa)`。

### 错误 4：SSE2 路径比标量还慢

**现象：** 开启 SIMD 后耗时不降反升。

**原因（及对应解决）：**
- **数据量太小**（<100 像素）：SIMD 有启动开销（寄存器初始化、尾部标量处理），需要足够的数据量分摊 → 只在扫描线长度 ≥ 16 像素时启用 SIMD
- **内存不连续**：`_mm_loadu_si128` 对非连续内存无法发挥 SIMD 优势 → 确保像素数据是紧凑数组
- **反复装箱拆箱**：循环体内频繁在整数/浮点/SIMD 类型间转换 → 保持数据在 `__m128i` 寄存器中，减少类型转换
- **Debug 编译**：`-O0` 下 intrinsic 不内联 → 始终用 Release 编译测量

### 4.3 main.cpp（完整版）

```cpp
// 文件：main.cpp — 软件光栅化器完整教学示例
// 功能：帧缓冲 → 线框 → 填充 → 纹理 → Alpha 混合 → SIMD → MVP 管线
// 编译：cmake -B build && cmake --build build --config Release
// 运行：./build/software_rasterizer（生成 out_rasterizer.ppm）

#include <algorithm>
#include <chrono>
#include <cmath>
#include <cstdint>
#include <fstream>
#include <iostream>
#include <vector>
#include <emmintrin.h>  // SSE2

// ===== 帧缓冲 =====
struct FrameBuffer {
    int width, height;
    std::vector<uint32_t> pixels;  // ARGB 格式

    FrameBuffer(int w, int h)
        : width(w), height(h),
          pixels((size_t)w * h, 0xFF101018u) {}  // 深色背景

    bool inBounds(int x, int y) const {
        return x >= 0 && x < width && y >= 0 && y < height;
    }

    uint32_t getPixel(int x, int y) const {
        return pixels[(size_t)y * width + x];
    }

    void putPixelRaw(int x, int y, uint32_t c) {
        if (inBounds(x, y))
            pixels[(size_t)y * width + x] = c;
    }

    // 导出为 PPM 格式（无需第三方库）
    bool exportPPM(const char* filename) const {
        std::ofstream out(filename, std::ios::binary);
        if (!out) return false;
        out << "P6\n" << width << " " << height << "\n255\n";
        for (int y = 0; y < height; ++y) {
            for (int x = 0; x < width; ++x) {
                auto v = getPixel(x, y);
                unsigned char rgb[3] = {
                    (unsigned char)(v >> 16),  // R
                    (unsigned char)(v >> 8),   // G
                    (unsigned char)(v)         // B
                };
                out.write((char*)rgb, 3);
            }
        }
        return true;
    }
};

// ===== Bresenham 画线 =====
static void drawLine(FrameBuffer& fb,
                     int x0, int y0, int x1, int y1,
                     uint32_t color)
{
    int dx = std::abs(x1 - x0), sx = x0 < x1 ? 1 : -1;
    int dy = -std::abs(y1 - y0), sy = y0 < y1 ? 1 : -1;
    int err = dx + dy;
    for (;;) {
        fb.putPixelRaw(x0, y0, color);
        if (x0 == x1 && y0 == y1) break;
        int e2 = err << 1;
        if (e2 >= dy) { err += dy; x0 += sx; }
        if (e2 <= dx) { err += dx; y0 += sy; }
    }
}

// ===== 线框三角形 =====
static void drawTriangleWireframe(FrameBuffer& fb,
    int x0, int y0, int x1, int y1, int x2, int y2,
    uint32_t color)
{
    drawLine(fb, x0, y0, x1, y1, color);
    drawLine(fb, x1, y1, x2, y2, color);
    drawLine(fb, x2, y2, x0, y0, color);
}

// ===== Alpha 混合（Porter-Duff Over，非预乘） =====
static inline uint32_t alphaOver(uint32_t dst, uint32_t src) {
    int sa = (src >> 24) & 255;
    int sr = (src >> 16) & 255;
    int sg = (src >> 8) & 255;
    int sb = src & 255;

    int dr = (dst >> 16) & 255;
    int dg = (dst >> 8) & 255;
    int db = dst & 255;
    int isa = 255 - sa;

    int a = sa + (((dst >> 24) & 255) * isa + 127) / 255;
    int r = (sr * sa + dr * isa + 127) / 255;
    int g = (sg * sa + dg * isa + 127) / 255;
    int b = (sb * sa + db * isa + 127) / 255;

    return (uint32_t(a) << 24) | (uint32_t(r) << 16) |
           (uint32_t(g) << 8) | uint32_t(b);
}

// ===== SSE2 批量混合 =====
static void blendSpanSSE2(uint32_t* dst, const uint32_t* src,
                          int count)
{
    int i = 0;
    const auto zero = _mm_setzero_si128();

    for (; i + 3 < count; i += 4) {
        auto S = _mm_loadu_si128((const __m128i*)(src + i));
        auto D = _mm_loadu_si128((const __m128i*)(dst + i));

        auto Sl = _mm_unpacklo_epi8(S, zero);
        auto Sh = _mm_unpackhi_epi8(S, zero);
        auto Dl = _mm_unpacklo_epi8(D, zero);
        auto Dh = _mm_unpackhi_epi8(D, zero);

        auto al = _mm_shufflehi_epi16(
            _mm_shufflelo_epi16(Sl, _MM_SHUFFLE(3,3,3,3)),
            _MM_SHUFFLE(3,3,3,3));
        auto ah = _mm_shufflehi_epi16(
            _mm_shufflelo_epi16(Sh, _MM_SHUFFLE(3,3,3,3)),
            _MM_SHUFFLE(3,3,3,3));

        auto ial = _mm_sub_epi16(_mm_set1_epi16(255), al);
        auto iah = _mm_sub_epi16(_mm_set1_epi16(255), ah);

        auto ol = _mm_srli_epi16(_mm_add_epi16(
            _mm_add_epi16(_mm_mullo_epi16(Sl, al),
                          _mm_mullo_epi16(Dl, ial)),
            _mm_set1_epi16(127)), 8);
        auto oh = _mm_srli_epi16(_mm_add_epi16(
            _mm_add_epi16(_mm_mullo_epi16(Sh, ah),
                          _mm_mullo_epi16(Dh, iah)),
            _mm_set1_epi16(127)), 8);

        _mm_storeu_si128((__m128i*)(dst + i),
                         _mm_packus_epi16(ol, oh));
    }
    for (; i < count; ++i)
        dst[i] = alphaOver(dst[i], src[i]);
}

// ===== 主函数 =====
int main() {
    // 创建帧缓冲
    FrameBuffer fb(640, 360);

    // 画一个线框三角形
    drawTriangleWireframe(fb,
        80, 60, 520, 80, 300, 300,
        0xFFFFCC33u);  // 黄色

    // 准备测试数据
    int n = fb.width * fb.height;
    std::vector<uint32_t> src(n, 0x803399FFu);  // 半透明蓝紫
    auto dstScalar = fb.pixels;  // 标量版的输入副本
    auto dstSimd   = fb.pixels;  // SIMD 版的输入副本

    int rounds = 200;

    // 标量基准测试
    auto t0 = std::chrono::high_resolution_clock::now();
    for (int r = 0; r < rounds; ++r)
        for (int i = 0; i < n; ++i)
            dstScalar[i] = alphaOver(dstScalar[i], src[i]);
    auto t1 = std::chrono::high_resolution_clock::now();

    // SIMD 基准测试
    for (int r = 0; r < rounds; ++r)
        blendSpanSSE2(dstSimd.data(), src.data(), n);
    auto t2 = std::chrono::high_resolution_clock::now();

    // 输出结果
    fb.pixels = dstSimd;  // 使用 SIMD 结果输出
    fb.exportPPM("out_rasterizer.ppm");

    double scalarMs = std::chrono::duration<double, std::milli>(
        t1 - t0).count();
    double simdMs = std::chrono::duration<double, std::milli>(
        t2 - t1).count();

    std::cout << "标量(ms): " << scalarMs << "\n"
              << "SIMD(ms): " << simdMs   << "\n"
              << "加速比:   " << scalarMs / simdMs << "x\n";

    return 0;
}
```

## 本节小结

- **基准测试三原则**：同输入、同输出、足够轮次——缺任何一个，测量结果都不可信
- **标量 vs SIMD** 在 640×360 图像上典型加速比 2.5×-4.0×，受限于内存带宽和编译器自动向量化
- 本章从帧缓冲出发，完整走通了 **线框 → 填充 → 纹理 → 混合 → SIMD → MVP 管线** 的全流程
- 扫描线与 Alpha 混合是 CPU 光栅器最核心的两个热点——前者决定正确性，后者决定性能
- KrKr2 的 `LayerBitmapIntf + tvpgl + blend_functor_c` 可以看作本章方案的**工业强化版**：更多混合模式、更完整的边界处理、更重视查表与向量化

## 练习题与答案

### 题目 1：如果你用 `-O0`（无优化）编译基准测试，结果会怎样？为什么？

<details>
<summary>查看答案</summary>

`-O0` 下两种实现的差距会大幅缩小甚至消失，原因有三：

1. **SSE2 intrinsic 不会被内联（Inline，编译器将函数调用替换为函数体的优化）**——每个 `_mm_*` 调用都是一次真正的函数调用，开销巨大
2. **标量版也不会被优化**——循环不展开（Loop Unrolling，编译器复制循环体减少跳转次数的优化）、不自动向量化
3. **内存访问模式不被优化**——编译器不会重排指令以利用 CPU 流水线

结论：**性能测量必须在 Release 编译（`-O2` 或更高）下进行**，Debug 模式的结果没有参考价值。

</details>

```cpp
        float v = (float)y / 511.0f;
        fb.putPixelRaw(x, y, sampleBilinear(tex, u, v));
    }
fb.exportPPM("bilinear.ppm");
```

对比：最近邻输出有明显方块台阶，双线性的边缘过渡平滑。也可程序化验证：对同一个非整数 UV 采样，最近邻返回某个角的精确颜色，双线性返回四角的加权平均值。

</details>

### 题目 3：为什么 KrKr2 要做 `TVPOpacityOnOpacityTable` 这类查表而不直接计算？

<details>
<summary>查看答案</summary>

三个原因：

1. **CPU 整数除法指令慢**——在 x86 上 `div` 指令的延迟（Latency，从输入到输出的 CPU 周期数）为 20-40 个周期，而乘法只需 3-4 个周期
2. **SIMD 没有整数除法**——SSE2 / NEON 均不提供向量整数除法指令（`_mm_div_epi16` 不存在），必须用乘法+移位近似或标量回退
3. **查表一次内存读取**——`TVPOpacityOnOpacityTable[opa1 * 256 + opa2]` 只需一条 `mov` 指令（约 4 周期），且 64KB 的表完全装入 L1 数据缓存

这是经典的**空间换时间**（Space-Time Tradeoff）优化：牺牲少量内存（64KB），换取数量级的性能提升。

</details>
