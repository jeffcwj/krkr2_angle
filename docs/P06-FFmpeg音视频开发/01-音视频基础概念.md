## 一、为什么需要学音视频基础？

如果你想理解 KrKr2 模拟器中的视频播放（`cpp/core/movie/ffmpeg/`）和音频解码（`cpp/core/sound/`），那么音视频基础概念是不可跳过的前置知识。

视觉小说引擎需要处理：
- **OP/ED 动画**：游戏的片头、片尾动画通常是 MPEG/H.264 视频
- **背景音乐**：BGM 通常是 Ogg Vorbis 或 WAV 格式
- **语音**：角色语音通常是压缩音频格式
- **音效**：环境音效、UI 音效

这些功能的底层都涉及相同的核心概念：数据如何被压缩存储（编码），又如何被还原播放（解码）。

> **类比**：如果把音视频比作快递包裹——容器格式是包装盒（决定里面能装什么），编码格式是包装方式（决定物品如何被压缩），而 PTS/DTS 是配送时间表（决定何时送达）。

## 二、容器格式 vs 编码格式

这是音视频领域最容易混淆的概念对。

### 2.1 容器格式（Container Format）

容器格式定义了数据的**存储结构** —— 它规定了视频流、音频流、字幕流等如何被组织在一个文件中。容器本身不关心数据是什么编码，它只负责"装箱"。

常见容器格式：

| 容器格式 | 扩展名 | 特点 | 常见用途 |
|----------|--------|------|---------|
| MP4 (MPEG-4 Part 14) | `.mp4`, `.m4a`, `.m4v` | 最通用，支持 H.264/H.265/AAC | 网络视频、移动端 |
| MKV (Matroska) | `.mkv`, `.mka` | 开放格式，支持几乎所有编码 | 高清视频收藏 |
| AVI | `.avi` | 微软早期格式，结构简单 | 旧版视频 |
| OGG | `.ogg`, `.ogv` | 开源容器，常搭配 Vorbis/Opus | 游戏音频（KrKr2 广泛使用） |
| WAV | `.wav` | 未压缩音频容器 | 音效、中间处理 |
| FLV | `.flv` | Flash 视频容器 | 直播推流（RTMP） |
| MPEG-TS | `.ts`, `.m2ts` | 传输流，抗丢包 | 电视广播、蓝光 |

### 2.2 编码格式（Codec）

编码格式定义了数据的**压缩算法** —— 它规定了原始像素/采样数据如何被压缩成更小的体积，以及如何被还原。

常见视频编码：

| 编码 | 全称 | 压缩率 | 复杂度 | 用途 |
|------|------|--------|--------|------|
| H.264 (AVC) | Advanced Video Coding | 高 | 中 | 最广泛使用的视频编码 |
| H.265 (HEVC) | High Efficiency Video Coding | 极高 | 高 | 4K/8K 视频 |
| VP9 | — | 高 | 中 | YouTube、WebM |
| AV1 | AOMedia Video 1 | 极高 | 极高 | 新一代开源编码 |
| MPEG-2 | — | 低 | 低 | DVD、旧版广播 |
| MPEG-1 | — | 低 | 低 | VCD、旧版游戏 OP（KiriKiri 常用） |

常见音频编码：

| 编码 | 全称 | 压缩率 | 用途 |
|------|------|--------|------|
| AAC | Advanced Audio Coding | 高 | MP4 默认音频，流媒体 |
| Vorbis | — | 高 | OGG 容器，**KrKr2 BGM 主力格式** |
| Opus | — | 极高 | 语音通话、低延迟音频 |
| MP3 | MPEG-1 Audio Layer 3 | 中 | 老牌压缩音频 |
| FLAC | Free Lossless Audio Codec | 无损 | 高品质音乐收藏 |
| PCM | Pulse Code Modulation | 无压缩 | WAV 内部数据、原始音频 |

### 2.3 容器与编码的组合关系

一个容器可以装不同编码的数据，一种编码也可以装在不同容器里：

```
┌─────────────────────────────────────────────────┐
│  MP4 容器                                        │
│  ┌──────────────────┐  ┌──────────────────────┐ │
│  │ 视频流: H.264     │  │ 音频流: AAC          │ │
│  │ 1920x1080, 24fps  │  │ 44100Hz, 立体声      │ │
│  └──────────────────┘  └──────────────────────┘ │
│  ┌──────────────────┐                            │
│  │ 字幕流: SRT       │                           │
│  └──────────────────┘                            │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│  OGG 容器                                        │
│  ┌──────────────────────────────────────────┐   │
│  │ 音频流: Vorbis                            │   │
│  │ 44100Hz, 立体声, 192kbps                  │   │
│  └──────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
```

> **KrKr2 中的实际应用**：KiriKiri 引擎的 OP 动画通常使用 MPEG-1 视频编码（存放在 `.mpg` 或 `.avi` 容器中），BGM 使用 Ogg Vorbis（`.ogg` 容器），语音也多为 Ogg Vorbis。KrKr2 项目在 `cpp/core/sound/VorbisWaveDecoder.cpp` 中实现了 Vorbis 解码器，在 `cpp/core/movie/ffmpeg/` 中实现了基于 FFmpeg 的通用视频播放器。

### 2.4 如何判断文件的容器和编码？

使用 FFmpeg 的 `ffprobe` 命令行工具可以查看任意媒体文件的详细信息：

```bash
# 查看文件信息
ffprobe -v quiet -show_format -show_streams input.mp4

# 简化输出
ffprobe -v error -show_entries format=format_name \
        -show_entries stream=codec_type,codec_name input.mp4
```

输出示例：

```
[FORMAT]
format_name=mov,mp4,m4a,3gp,3g2,mj2    ← 容器格式
[/FORMAT]
[STREAM]
codec_type=video                         ← 流类型：视频
codec_name=h264                          ← 编码格式：H.264
[/STREAM]
[STREAM]
codec_type=audio                         ← 流类型：音频
codec_name=aac                           ← 编码格式：AAC
[/STREAM]
```

## 三、采样率、比特率、帧率

### 3.1 视频帧率（Frame Rate / FPS）

帧率是视频每秒显示的画面帧数，单位是 fps（frames per second）。

| 帧率 | 用途 | 说明 |
|------|------|------|
| 24 fps | 电影 | 人眼觉得流畅的最低帧率 |
| 25 fps | PAL 制式电视 | 欧洲/中国广播标准 |
| 29.97 fps | NTSC 制式电视 | 美国/日本广播标准 |
| 30 fps | 网络视频 | 常见的折中选择 |
| 60 fps | 游戏/高帧率视频 | 明显更流畅 |

> **KrKr2 中的帧率**：视觉小说的 OP 动画通常是 24fps 或 29.97fps。KrKr2 的 `BasePlayer`（`VideoPlayer.cpp`）通过 `CDVDClock` 时钟控制帧的显示节奏。

### 3.2 音频采样率（Sample Rate）

模拟音频是连续波形，数字化需要将其"切片"成离散的采样点。**采样率**是每秒采集的样本数，单位是 Hz。

```
模拟波形:  ~~~~^~~~~v~~~~^~~~~v~~~~     ← 连续信号

采样（8kHz）: •    •    •    •    •     ← 每秒 8000 个点

采样（44.1kHz）: ••••••••••••••••••••   ← 每秒 44100 个点（更精确）
```

**奈奎斯特定理（Nyquist Theorem）**：要完整还原频率为 f 的信号，采样率必须 ≥ 2f。人耳能听到的最高频率约 20kHz，所以 CD 音质采用 44.1kHz（44100Hz）。

常见采样率：

| 采样率 | 用途 | 频率上限 |
|--------|------|---------|
| 8000 Hz | 电话语音 | 4 kHz |
| 22050 Hz | FM 广播质量 | ~11 kHz |
| 44100 Hz | CD 音质 | ~20 kHz |
| 48000 Hz | DVD/蓝光、专业音频 | ~22 kHz |
| 96000 Hz | 高清音频（Hi-Res） | ~44 kHz |

### 3.3 音频位深（Bit Depth / Sample Size）

每个采样点用多少位来表示幅度。位深越高，动态范围越大（安静和响亮之间的差距）。

| 位深 | 动态范围 | 用途 |
|------|---------|------|
| 8 bit | 48 dB | 早期游戏音效 |
| 16 bit | 96 dB | CD 音质（**KrKr2 内部默认**） |
| 24 bit | 144 dB | 专业录音 |
| 32 bit float | ~1528 dB（理论） | 音频处理中间格式 |

> **KrKr2 源码对照**：`cpp/core/sound/WaveIntf.h` 中的 `tTVPWaveFormat` 结构体定义了 PCM 格式——包含 `SamplesPerSec`（采样率）、`Channels`（声道数）、`BitsPerSample`（位深）、`IsFloat`（是否浮点格式）。

```cpp
// KrKr2 中的 PCM 格式定义（简化）
// 来源：cpp/core/sound/WaveIntf.h
struct tTVPWaveFormat {
    int SamplesPerSec;   // 采样率，如 44100
    int Channels;        // 声道数，1=单声道，2=立体声
    int BitsPerSample;   // 位深，如 16
    bool IsFloat;        // 是否浮点格式
};
```

### 3.4 比特率（Bitrate）

比特率是媒体数据每秒占用的位数，单位是 bps（bits per second）。比特率越高，质量越好，文件越大。

**视频比特率**参考值：

| 质量 | 分辨率 | H.264 比特率 | 每分钟文件大小 |
|------|--------|-------------|---------------|
| 低 | 720p | 2-4 Mbps | ~15-30 MB |
| 中 | 1080p | 5-10 Mbps | ~38-75 MB |
| 高 | 1080p | 15-20 Mbps | ~113-150 MB |
| 极高 | 4K | 35-60 Mbps | ~263-450 MB |

**音频比特率**参考值：

| 质量 | 编码 | 比特率 |
|------|------|--------|
| 语音 | Opus | 16-32 kbps |
| 普通音乐 | Vorbis/AAC | 128-192 kbps |
| 高品质音乐 | Vorbis/AAC | 256-320 kbps |
| 无损 | FLAC | ~800-1400 kbps |
| 未压缩 | PCM 16bit/44.1kHz/立体声 | 1411 kbps |

**比特率的计算公式**：

```
未压缩视频比特率 = 宽 × 高 × 帧率 × 每像素位数
例：1920 × 1080 × 24fps × 24bit = 1,194,393,600 bps ≈ 1.19 Gbps

未压缩音频比特率 = 采样率 × 声道数 × 位深
例：44100 × 2 × 16 = 1,411,200 bps = 1411.2 kbps（即 CD 音质）
```

由此可见，**压缩是绝对必要的** —— 未压缩的 1080p 视频每分钟约 8.9 GB！H.264 编码可以将其压缩到几十 MB，压缩比高达 100-200 倍。

## 四、PTS 与 DTS —— 时间戳系统

PTS 和 DTS 是音视频同步的基石，理解它们对于读懂 KrKr2 的 `CDVDClock` 和 `BasePlayer` 代码至关重要。

### 4.1 PTS（Presentation Time Stamp）

**PTS = 显示时间戳**。它告诉播放器："这一帧应该在什么时候**显示给用户**"。

```
PTS: 0ms    33ms    67ms    100ms    133ms
      │      │       │       │        │
帧:   [F0]   [F1]    [F2]    [F3]     [F4]
```

PTS 通常以**时间基（time_base）**为单位。例如，time_base = 1/90000 表示 1 个 PTS 单位 = 1/90000 秒。如果一帧的 PTS = 9000，那么它应该在 9000/90000 = 0.1 秒时显示。

### 4.2 DTS（Decoding Time Stamp）

**DTS = 解码时间戳**。它告诉解码器："这一帧应该在什么时候被**解码**"。

对于简单编码（只有 I 帧和 P 帧），DTS 和 PTS 是相同的。但当引入 B 帧后，DTS 和 PTS 会出现差异。

### 4.3 为什么 DTS ≠ PTS？ —— B 帧的影响

B 帧（Bidirectional Predicted Frame，双向预测帧）需要参考前后两帧才能解码。这意味着解码顺序和显示顺序不同：

```
显示顺序（PTS 顺序）：I0   B1   B2   P3   B4   B5   P6
                      ↓    ↓    ↓    ↓    ↓    ↓    ↓
解码顺序（DTS 顺序）：I0   P3   B1   B2   P6   B4   B5
```

为什么？因为 B1 和 B2 需要参考 I0（前向）和 P3（后向），所以 P3 必须先于 B1、B2 被解码。

```
解码顺序：I0 → P3 → B1 → B2 → P6 → B4 → B5
          ↑    ↑              ↑
          │    │              │
          │    └── B1,B2 的后向参考
          └─────── B1,B2 的前向参考

显示顺序：I0 → B1 → B2 → P3 → B4 → B5 → P6
```

| 帧 | DTS（解码时间） | PTS（显示时间） |
|----|----------------|----------------|
| I0 | 0 ms | 0 ms |
| P3 | 33 ms | 100 ms |
| B1 | 67 ms | 33 ms |
| B2 | 100 ms | 67 ms |
| P6 | 133 ms | 200 ms |
| B4 | 167 ms | 133 ms |
| B5 | 200 ms | 167 ms |

> **KrKr2 源码对照**：`cpp/core/movie/ffmpeg/Clock.h` 中的 `CDVDClock` 类维护全局时钟，`VideoPlayerVideo.cpp` 根据 PTS 决定帧的显示时机，`VideoPlayerAudio.cpp` 根据音频 PTS 调整音频输出节奏。

### 4.4 time_base 与时间转换

FFmpeg 中的时间戳都基于**时间基（time_base）**，它是一个有理数分数（`AVRational`），表示一个时间单位等于多少秒：

```cpp
// FFmpeg 的时间基定义
typedef struct AVRational {
    int num; // 分子
    int den; // 分母
} AVRational;

// 常见时间基
// 容器层：1/90000（MPEG-TS 标准）
// 编码层：1/帧率，如 1/24、1/30
// 内部计算：AV_TIME_BASE = 1000000（微秒）

// 时间戳转秒
double seconds = pts * av_q2d(time_base);
// 等价于：
double seconds = (double)pts * time_base.num / time_base.den;
```

## 五、关键帧、预测帧与 GOP

### 5.1 三种帧类型

视频压缩的核心思想是**去除冗余** —— 相邻帧之间大部分内容是相同的，只编码差异即可大幅压缩体积。

| 帧类型 | 全称 | 说明 | 大小 |
|--------|------|------|------|
| **I 帧** | Intra Frame（帧内编码帧） | 完整独立的一帧，不依赖其他帧 | 最大 |
| **P 帧** | Predicted Frame（前向预测帧） | 只存储与前一个 I/P 帧的差异 | 中等 |
| **B 帧** | Bidirectional Frame（双向预测帧） | 存储与前后帧的差异 | 最小 |

```
I 帧（完整画面）：
┌─────────────┐
│ 🌳🏠☀️    │  ← 存储完整像素数据（如 100KB）
│ 🚗  🚶    │
└─────────────┘

P 帧（差异）：
┌─────────────┐
│ -------- -  │  ← 只存储变化部分：车移动了
│  → 🚗🚶    │     （如 20KB）
└─────────────┘

B 帧（前后差异）：
┌─────────────┐
│ ----------- │  ← 参考前后帧，存储极少差异
│  →🚗- -     │     （如 8KB）
└─────────────┘
```

### 5.2 GOP（Group of Pictures）

GOP 是一组从 I 帧开始的帧序列。**I 帧是 Seek（跳转）的锚点** —— 你只能跳转到 I 帧的位置，因为只有 I 帧不依赖其他帧。

```
GOP 1                    GOP 2                    GOP 3
┌──────────────────┐    ┌──────────────────┐    ┌──────
I  B  B  P  B  B  P    I  B  B  P  B  B  P    I ...
│← GOP 长度 = 7 →│    │← GOP 长度 = 7 →│
```

GOP 长度的影响：

| GOP 长度 | 压缩率 | Seek 精度 | 适用场景 |
|----------|--------|-----------|---------|
| 短（1-2） | 低 | 高（几乎逐帧定位） | 编辑/剪辑 |
| 中（12-15） | 中 | 中 | 通用视频 |
| 长（30-60+） | 高 | 低（跳转不精确） | 流媒体 |

> **为什么 Seek 慢？**：当用户拖动进度条到某个位置时，解码器必须先找到最近的 I 帧，然后从 I 帧开始逐帧解码到目标位置。如果 GOP 很长（如 60 帧），最坏情况要解码 59 帧才能到达目标位置。KrKr2 的 `DemuxFFmpeg.cpp` 中的 `SeekTime()` 方法就是处理这个问题的。

### 5.3 关键帧在 FFmpeg API 中的表示

```cpp
// 在 FFmpeg 中，通过 AVPacket 的 flags 判断关键帧
AVPacket *pkt = av_packet_alloc();
av_read_frame(fmt_ctx, pkt);

if(pkt->flags & AV_PKT_FLAG_KEY) {
    // 这是一个关键帧（I 帧）
    printf("关键帧, PTS=%lld\n", pkt->pts);
}

// AVFrame 解码后也有标记
AVFrame *frame = av_frame_alloc();
// ... 解码 ...
if(frame->key_frame) {
    printf("解码后确认: 这是关键帧\n");
}
printf("帧类型: %c\n", av_get_picture_type_char(frame->pict_type));
// 输出: 'I', 'P', 或 'B'
```

## 六、像素格式与色彩空间

### 6.1 RGB 与 YUV

计算机显示器使用 **RGB**（红绿蓝）色彩模型，但视频编码几乎全部使用 **YUV** 色彩模型：

- **Y**（Luminance / 亮度）：画面的明暗信息
- **U**（Cb / 蓝色色度）：蓝色偏移
- **V**（Cr / 红色色度）：红色偏移

为什么用 YUV 而不是 RGB？

1. **人眼对亮度敏感，对色度不敏感** —— 可以降低 U/V 的分辨率而不影响感知质量
2. **历史兼容** —— YUV 与黑白电视兼容（Y 分量就是黑白画面）
3. **压缩效率更高** —— Y/U/V 分量的信息冗余度不同，可以针对性压缩

### 6.2 YUV 采样格式

YUV 格式的关键差异在于 U/V 相对于 Y 的采样比例：

```
YUV 4:4:4（无降采样）：
Y:  □□□□    U:  □□□□    V:  □□□□
    □□□□        □□□□        □□□□
每像素: Y+U+V = 3字节 → 24 bit/pixel

YUV 4:2:2（水平半采样）：
Y:  □□□□    U:  □ □     V:  □ □
    □□□□        □ □         □ □
每像素平均: 2字节 → 16 bit/pixel

YUV 4:2:0（最常用 — 水平+垂直半采样）：
Y:  □□□□    U:  □ □     V:  □ □
    □□□□
每像素平均: 1.5字节 → 12 bit/pixel
```

**YUV 4:2:0 是视频编码的事实标准** —— H.264、H.265、VP9 等编码默认输出 YUV420P。

### 6.3 FFmpeg 中的像素格式

FFmpeg 使用 `AVPixelFormat` 枚举表示像素格式：

```cpp
// 常见像素格式
enum AVPixelFormat {
    AV_PIX_FMT_YUV420P,   // YUV 4:2:0 planar（最常见）
    AV_PIX_FMT_YUV422P,   // YUV 4:2:2 planar
    AV_PIX_FMT_YUV444P,   // YUV 4:4:4 planar
    AV_PIX_FMT_NV12,      // YUV 4:2:0 semi-planar（Y平面+UV交错）
    AV_PIX_FMT_RGB24,     // RGB 打包格式
    AV_PIX_FMT_RGBA,      // RGBA 打包格式（带透明通道）
    // ...
};
```

> **Planar vs Packed**：
> - **Planar（平面格式）**：Y、U、V 数据分别存储在三个连续内存区域 → `YYYYUUVV`
> - **Packed（打包格式）**：Y、U、V 数据交错存储 → `YUVYUVYUV`
> - **Semi-planar（半平面）**：Y 单独一个平面，UV 交错存储 → `YYYY` + `UVUV`（NV12）

```cpp
// 解码后访问 YUV420P 帧数据
AVFrame *frame; // 已解码的视频帧

// Y 平面
uint8_t *y_data = frame->data[0];     // Y 数据指针
int y_linesize = frame->linesize[0];  // Y 每行字节数

// U 平面（宽高各是 Y 的一半）
uint8_t *u_data = frame->data[1];
int u_linesize = frame->linesize[1];

// V 平面
uint8_t *v_data = frame->data[2];
int v_linesize = frame->linesize[2];

// 访问坐标 (x, y) 的亮度值
uint8_t luminance = y_data[y * y_linesize + x];

// 访问坐标 (x, y) 的色度值（注意除以 2）
uint8_t cb = u_data[(y/2) * u_linesize + (x/2)];
uint8_t cr = v_data[(y/2) * v_linesize + (x/2)];
```

> **KrKr2 源码对照**：`cpp/core/movie/ffmpeg/RenderFormats.h` 定义了 `ERenderFormat` 枚举，包括 `RENDER_FMT_YUV420P`、`RENDER_FMT_NV12` 等。`BaseRenderer` 根据解码器输出的像素格式选择对应的渲染路径。

### 6.4 YUV → RGB 转换

最终显示到屏幕时，需要将 YUV 转换为 RGB：

```cpp
// BT.601 标准（SD 视频）
R = Y + 1.402 * (V - 128)
G = Y - 0.344 * (U - 128) - 0.714 * (V - 128)
B = Y + 1.772 * (U - 128)

// BT.709 标准（HD 视频）
R = Y + 1.5748 * (V - 128)
G = Y - 0.1873 * (U - 128) - 0.4681 * (V - 128)
B = Y + 1.8556 * (U - 128)
```

FFmpeg 提供 `libswscale` 库来高效完成这个转换：

```cpp
#include <libswscale/swscale.h>

// 创建转换上下文：YUV420P → RGB24
struct SwsContext *sws_ctx = sws_getContext(
    width, height, AV_PIX_FMT_YUV420P,  // 源格式
    width, height, AV_PIX_FMT_RGB24,    // 目标格式
    SWS_BILINEAR,    // 缩放算法
    NULL, NULL, NULL  // 滤镜（不需要）
);

// 执行转换
uint8_t *dst_data[1] = { rgb_buffer };  // 输出 RGB 缓冲区
int dst_linesize[1] = { width * 3 };    // 每行 RGB 字节数

sws_scale(
    sws_ctx,
    frame->data, frame->linesize,   // 源 YUV 数据
    0, height,                       // 起始行, 行数
    dst_data, dst_linesize           // 目标 RGB 数据
);

// 用完释放
sws_freeContext(sws_ctx);
```

## 七、音频采样格式

### 7.1 整数格式 vs 浮点格式

| 格式 | FFmpeg 枚举 | 范围 | 用途 |
|------|-------------|------|------|
| 8-bit unsigned | `AV_SAMPLE_FMT_U8` | [0, 255] | 早期游戏音效 |
| 16-bit signed | `AV_SAMPLE_FMT_S16` | [-32768, 32767] | CD 音质、最常见 |
| 32-bit signed | `AV_SAMPLE_FMT_S32` | [-2^31, 2^31-1] | 专业音频 |
| 32-bit float | `AV_SAMPLE_FMT_FLT` | [-1.0, 1.0] | 音频处理中间格式 |
| 64-bit float | `AV_SAMPLE_FMT_DBL` | [-1.0, 1.0] | 高精度处理 |

### 7.2 Planar vs Interleaved（交错）

音频同样有平面和交错两种内存布局：

```
立体声（2声道）交错格式（Interleaved）：
L R L R L R L R L R ...    ← 左右声道交替存储

立体声（2声道）平面格式（Planar）：
L L L L L L L L L L ...    ← 左声道连续存储
R R R R R R R R R R ...    ← 右声道连续存储
```

FFmpeg 中，Planar 格式名称带 `P` 后缀：

```cpp
AV_SAMPLE_FMT_S16   // 16-bit 交错格式
AV_SAMPLE_FMT_S16P  // 16-bit 平面格式
AV_SAMPLE_FMT_FLT   // 32-bit float 交错格式
AV_SAMPLE_FMT_FLTP  // 32-bit float 平面格式
```

> **KrKr2 源码对照**：`cpp/core/sound/WaveIntf.cpp` 中的 `TVPConvertPCMTo16bits` 和 `TVPConvertPCMToFloat` 函数负责在不同 PCM 格式之间转换。`FFWaveDecoder.cpp` 解码后会将 FFmpeg 输出的采样格式统一转换为 KrKr2 内部使用的 16-bit S16 交错格式。

### 7.3 使用 libswresample 进行音频格式转换

```cpp
#include <libswresample/swresample.h>

// 创建重采样上下文
SwrContext *swr_ctx = swr_alloc_set_opts(
    NULL,
    AV_CH_LAYOUT_STEREO,     // 输出：立体声
    AV_SAMPLE_FMT_S16,       // 输出：16-bit 交错
    44100,                    // 输出采样率
    av_get_default_channel_layout(
        codec_ctx->channels), // 输入声道布局
    codec_ctx->sample_fmt,    // 输入格式（可能是 FLTP）
    codec_ctx->sample_rate,   // 输入采样率
    0, NULL
);
swr_init(swr_ctx);

// 转换
uint8_t *out_buf;
int out_samples = swr_convert(
    swr_ctx,
    &out_buf, max_out_samples,        // 输出
    (const uint8_t **)frame->data,    // 输入
    frame->nb_samples                  // 输入样本数
);

// 用完释放
swr_free(&swr_ctx);
```

## 常见错误及解决方案

### 错误 1：混淆容器格式和编码格式

**症状**：认为 `.mp4` 文件一定是 H.264 编码，或者认为 H.264 只能放在 MP4 容器里。

**正确理解**：容器和编码是独立的两个维度。MP4 可以装 H.264 也可以装 H.265；H.264 可以放在 MP4、MKV、FLV 等任何容器中。

**验证方法**：用 `ffprobe` 实际查看文件，不要根据扩展名猜测。

### 错误 2：忽略 PTS/DTS 的区别

**症状**：只使用 DTS 来控制显示时间，导致含 B 帧的视频画面顺序错乱。

**正确做法**：
- **显示帧**时使用 PTS
- **解码排序**时使用 DTS
- 对于纯 I+P 帧的视频（无 B 帧），两者相同，不会出问题
- 对于含 B 帧的视频，必须严格区分

### 错误 3：采样率转换时未正确处理

**症状**：将 48kHz 音频直接按 44.1kHz 播放，导致语速变慢、音调偏低。

**正确做法**：使用 `libswresample`（`swr_convert`）进行采样率转换，它会自动处理插值和抗混叠滤波。

## 本节小结

- **容器格式**（MP4/MKV/OGG）定义数据的存储结构；**编码格式**（H.264/Vorbis/AAC）定义数据的压缩算法。两者独立但需配合使用
- **采样率**决定音频频率范围（44.1kHz = CD 音质），**位深**决定动态范围（16bit = 96dB），**帧率**决定视频流畅度（24fps = 电影标准）
- **PTS**（显示时间戳）控制帧的显示时机，**DTS**（解码时间戳）控制帧的解码顺序；当存在 B 帧时两者不同
- **I/P/B 帧**构成 GOP 结构 —— I 帧是完整帧（Seek 锚点），P 帧存储前向差异，B 帧存储前后差异（最小）
- **YUV 4:2:0** 是视频编码的事实标准 —— 利用人眼对亮度敏感、对色度不敏感的特点，将色度分辨率降为亮度的 1/4
- 音频有**交错格式**（LRLRLR）和**平面格式**（LLLL...RRRR...）两种内存布局

## 练习题与答案

### 题目 1：容器与编码辨析

以下哪些是容器格式，哪些是编码格式？请分类：

`MP4`、`H.264`、`OGG`、`Vorbis`、`MKV`、`AAC`、`WAV`、`FLAC`、`AVI`、`Opus`

<details>
<summary>查看答案</summary>

**容器格式**：MP4、OGG、MKV、WAV、AVI

**编码格式**：H.264、Vorbis、AAC、FLAC、Opus

**易混淆点**：
- **WAV** 是容器，不是编码。WAV 通常包含 PCM 未压缩数据，但也可以包含其他编码（如 ADPCM、GSM）
- **FLAC** 是编码格式（无损压缩算法），虽然 `.flac` 文件有自己的容器，但 FLAC 也可以封装在 OGG 或 MKV 容器中
- **OGG** 是容器，**Vorbis/Opus** 是编码。"Ogg Vorbis" 是指使用 Vorbis 编码的 OGG 容器文件

</details>

### 题目 2：PTS/DTS 排列

给定以下 GOP 结构（显示顺序）：

```
I0  B1  B2  P3  B4  P5
```

请写出：
1. 解码顺序（DTS 顺序）
2. 每一帧的 DTS 和 PTS 值（假设帧间隔为 33ms）

<details>
<summary>查看答案</summary>

**1. 解码顺序**：

```
I0 → P3 → B1 → B2 → P5 → B4
```

原因：B1、B2 需要参考 I0（前向）和 P3（后向），所以 P3 必须在 B1、B2 之前解码。同理 B4 需要参考 P3 和 P5。

**2. DTS 和 PTS 值**：

| 解码顺序 | 帧 | DTS | PTS |
|---------|-----|-----|-----|
| 1 | I0 | 0 ms | 0 ms |
| 2 | P3 | 33 ms | 99 ms |
| 3 | B1 | 66 ms | 33 ms |
| 4 | B2 | 99 ms | 66 ms |
| 5 | P5 | 132 ms | 165 ms |
| 6 | B4 | 165 ms | 132 ms |

**验证**：按 PTS 排序后恢复显示顺序：I0(0) → B1(33) → B2(66) → P3(99) → B4(132) → P5(165) ✓

</details>

### 题目 3：比特率计算

计算以下参数：

1. 未压缩 720p (1280×720) 视频，YUV420 格式，30fps，每秒数据量是多少 MBps？
2. 一首 3 分钟的歌曲，以 Vorbis 编码、192kbps 比特率存储，文件大约多大？
3. 同一首歌以未压缩 PCM（44.1kHz, 16bit, 立体声）存储，文件大约多大？压缩比是多少？

<details>
<summary>查看答案</summary>

**1. 720p YUV420 视频每秒数据量**：

```
每帧字节数 = 1280 × 720 × 12 / 8 = 1,382,400 字节 ≈ 1.32 MB
（YUV420 每像素 12 bit = 1.5 字节）

每秒数据量 = 1.32 MB × 30fps = 39.6 MB/s ≈ 316.8 Mbps
```

**2. 3 分钟 Vorbis 192kbps 音频**：

```
数据量 = 192 kbps × 180 秒 = 34,560 kbit = 4,320 KB ≈ 4.22 MB
```

**3. 3 分钟未压缩 PCM 音频**：

```
数据量 = 44100 × 2声道 × 2字节 × 180秒 = 31,752,000 字节 ≈ 30.3 MB

压缩比 = 30.3 / 4.22 ≈ 7.2:1
```

Vorbis 在 192kbps 下将 CD 音质音频压缩了约 7 倍，同时保持人耳几乎无法区分的音质（属于有损压缩的高品质档）。

</details>
