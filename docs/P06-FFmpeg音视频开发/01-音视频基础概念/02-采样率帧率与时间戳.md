## 一、视频帧率（Frame Rate / FPS）

帧率是视频每秒显示的画面帧数（Frame，一帧就是一张静止图片），单位是 **fps**（frames per second，每秒帧数）。人眼看到快速切换的静止画面时会产生"运动"的错觉——这就是视频的基本原理。

| 帧率 | 用途 | 说明 |
|------|------|------|
| 24 fps | 电影 | 人眼觉得流畅的最低帧率，电影工业标准 |
| 25 fps | PAL 制式电视 | 欧洲/中国广播标准 |
| 29.97 fps | NTSC 制式电视 | 美国/日本广播标准（因彩色兼容性取近似值） |
| 30 fps | 网络视频 | 常见的折中选择 |
| 60 fps | 游戏/高帧率视频 | 明显更流畅，动作场景效果好 |

> **为什么是 29.97 而不是 30？** 早期美国黑白电视用 30fps。引入彩色信号后，为避免色度信号（Chrominance，画面的颜色信息）与声音载波之间的干扰，将帧率微调为 30×1000/1001 ≈ 29.97fps。这个历史遗留值沿用至今。

> **KrKr2 中的帧率**：视觉小说的 OP 动画通常是 24fps 或 29.97fps。KrKr2 的 `VideoPlayer.cpp` 中的 `BasePlayer` 通过 `CDVDClock`（一个全局时钟管理类，详见第 5 章）控制帧的显示节奏。

### 在 FFmpeg 中获取帧率

```cpp
#include <libavformat/avformat.h>

void print_fps(AVFormatContext *fmt_ctx) {
    for (unsigned i = 0; i < fmt_ctx->nb_streams; i++) {
        AVStream *st = fmt_ctx->streams[i];
        if (st->codecpar->codec_type == AVMEDIA_TYPE_VIDEO) {
            // avg_frame_rate：平均帧率（AVRational 有理数）
            // r_frame_rate：真实基础帧率（可能更准确）
            // av_q2d() 将 AVRational 转为 double
            printf("平均帧率: %.2f fps\n",
                   av_q2d(st->avg_frame_rate));
            printf("基础帧率: %.2f fps\n",
                   av_q2d(st->r_frame_rate));

            // 29.97fps 的 AVRational 表示：{30000, 1001}
            printf("帧率分数: %d/%d\n",
                   st->avg_frame_rate.num,
                   st->avg_frame_rate.den);
        }
    }
}
```

## 二、音频采样率与位深

### 2.1 采样率（Sample Rate）

模拟音频是连续波形，要把它变成数字信号就需要**采样**——每隔固定时间间隔测量一次波形的幅度值。**采样率**就是每秒采集的样本数，单位是 Hz。

```
模拟波形:  ~~~~^~~~~v~~~~^~~~~v~~~~     ← 连续信号（现实中的声音）

采样（8kHz）:   •    •    •    •    •   ← 每秒 8,000 个点（粗糙）

采样（44.1kHz）: •••••••••••••••••••••  ← 每秒 44,100 个点（精细）
```

**奈奎斯特定理（Nyquist Theorem）**：要完整还原最高频率为 f 的信号，采样率必须 ≥ 2f。人耳能听到的最高频率约 20kHz，所以 CD 音质采用 44.1kHz（44100Hz），能还原最高约 22kHz 的声音。如果采样率不够，高频信号会发生**混叠（Aliasing）**——本来是高音的信号被错误还原为低音。

常见采样率：

| 采样率 | 用途 | 可还原最高频率 |
|--------|------|--------------|
| 8,000 Hz | 电话语音 | 4 kHz |
| 22,050 Hz | FM 广播质量 | ~11 kHz |
| 44,100 Hz | **CD 音质**（KrKr2 默认） | ~20 kHz |
| 48,000 Hz | DVD/蓝光、专业音频 | ~22 kHz |
| 96,000 Hz | 高清音频（Hi-Res Audio） | ~44 kHz |

### 2.2 位深（Bit Depth / Sample Size）

每个采样点用多少位（bit）来表示幅度。位深越高，能表示的幅度级别越多，**动态范围**（Dynamic Range，最安静和最响亮之间的差距）越大。

| 位深 | 幅度级别数 | 动态范围 | 用途 |
|------|-----------|---------|------|
| 8 bit | 256 级 | 48 dB | 早期游戏音效 |
| 16 bit | 65,536 级 | 96 dB | **CD 音质**（KrKr2 内部默认） |
| 24 bit | 16,777,216 级 | 144 dB | 专业录音 |
| 32 bit float | 浮点精度 | ~1528 dB（理论） | 音频处理中间格式 |

> **动态范围 48 dB 是什么概念？** 大约是安静房间（~30dB）到吸尘器（~78dB）的差距。96 dB（16 bit）则覆盖从几乎无声到摇滚演唱会的全部范围，对消费级音频绰绰有余。

> **KrKr2 源码对照**：`cpp/core/sound/WaveIntf.h` 中的 `tTVPWaveFormat` 结构体定义了 KrKr2 内部使用的 PCM（Pulse Code Modulation，脉冲编码调制——最基础的音频数字化方式，直接存储每个采样点的幅度值）音频格式：

```cpp
// KrKr2 的 PCM 格式定义
// 来源：cpp/core/sound/WaveIntf.h
struct tTVPWaveFormat {
    int SamplesPerSec;   // 采样率，如 44100
    int Channels;        // 声道数，1=单声道，2=立体声
    int BitsPerSample;   // 位深，如 16
    bool IsFloat;        // 是否使用浮点格式（false=整数，true=浮点）
};
// KrKr2 内部默认：44100Hz / 16bit / 立体声 / 整数格式
```

### 2.3 在 FFmpeg 中获取音频参数

```cpp
void print_audio_info(AVFormatContext *fmt_ctx) {
    for (unsigned i = 0; i < fmt_ctx->nb_streams; i++) {
        AVStream *st = fmt_ctx->streams[i];
        AVCodecParameters *par = st->codecpar;

            // av_get_bytes_per_sample() 返回每样本字节数

                (enum AVSampleFormat)par->format);
            printf("每样本字节数: %d\n", bytes);
        }
    }
}
```

## 三、比特率（Bitrate）

比特率是媒体数据每秒占用的位数，单位是 **bps**（bits per second）。比特率越高，数据量越大，通常质量也越好。

### 3.1 视频比特率参考

| 画质 | 分辨率 | H.264 典型比特率 | 每分钟文件大小 |
|------|--------|-----------------|---------------|
| 低 | 720p (1280×720) | 2-4 Mbps | ~15-30 MB |
| 中 | 1080p (1920×1080) | 5-10 Mbps | ~38-75 MB |
| 高 | 1080p | 15-20 Mbps | ~113-150 MB |
| 极高 | 4K (3840×2160) | 35-60 Mbps | ~263-450 MB |

### 3.2 音频比特率参考

| 质量 | 编码 | 比特率 |
|------|------|--------|
| 语音 | Opus | 16-32 kbps |
| 普通音乐 | Vorbis/AAC | 128-192 kbps |
| 高品质音乐 | Vorbis/AAC | 256-320 kbps |
| 无损 | FLAC | ~800-1400 kbps |
| 未压缩 | PCM 16bit/44.1kHz/立体声 | 1411 kbps |

### 3.3 比特率计算公式

```
未压缩视频比特率 = 宽 × 高 × 帧率 × 每像素位数
例：1920 × 1080 × 24fps × 12bit(YUV420) = 597,196,800 bps ≈ 597 Mbps
    1920 × 1080 × 24fps × 24bit(RGB)    = 1,194,393,600 bps ≈ 1.19 Gbps

未压缩音频比特率 = 采样率 × 声道数 × 位深
例：44100 × 2声道 × 16bit = 1,411,200 bps = 1411.2 kbps（CD 音质）
```

H.264 编码可将 1080p/24fps 视频从 ~597 Mbps 压缩到 5-20 Mbps，**压缩比 30-120 倍**。由此可见，没有编码压缩，数字视频根本不可能存储和传输。

### 3.4 用 C++ 计算比特率

```cpp
#include <cstdio>
#include <cstdint>

// 计算未压缩视频每秒数据量
void calc_video_bitrate(int width, int height, double fps,
                        int bits_per_pixel) {
    double bps = (double)width * height * fps * bits_per_pixel;
    printf("未压缩视频: %dx%d @ %.2ffps, %dbpp\n",
           width, height, fps, bits_per_pixel);
    printf("  比特率: %.2f Mbps\n", bps / 1e6);
    printf("  每秒数据量: %.2f MB/s\n", bps / 8.0 / 1e6);
    printf("  每分钟数据量: %.2f GB/min\n", bps / 8.0 / 1e9 * 60);
}

// 计算未压缩音频每秒数据量
void calc_audio_bitrate(int sample_rate, int channels,
                        int bits_per_sample) {
    double bps = (double)sample_rate * channels * bits_per_sample;
    printf("未压缩音频: %dHz, %dch, %dbit\n",
           sample_rate, channels, bits_per_sample);
    printf("  比特率: %.2f kbps\n", bps / 1e3);
    printf("  每分钟数据量: %.2f MB/min\n", bps / 8.0 / 1e6 * 60);
}

int main() {
    // 1080p YUV420 视频
    calc_video_bitrate(1920, 1080, 24.0, 12);
    printf("\n");
    // CD 音质音频
    calc_audio_bitrate(44100, 2, 16);
    return 0;
}
```

输出：

```
未压缩视频: 1920x1080 @ 24.00fps, 12bpp
  比特率: 597.20 Mbps
  每秒数据量: 74.65 MB/s
  每分钟数据量: 4.48 GB/min

未压缩音频: 44100Hz, 2ch, 16bit
  比特率: 1411.20 kbps
  每分钟数据量: 10.09 MB/min
```

## 四、PTS 与 DTS —— 时间戳系统

PTS（Presentation Time Stamp，显示时间戳）和 DTS（Decoding Time Stamp，解码时间戳）是音视频同步的基石。理解它们对于读懂 KrKr2 的 `CDVDClock` 和 `BasePlayer` 代码至关重要。

### 4.1 PTS —— 什么时候显示

**PTS = 显示时间戳**。它告诉播放器："这一帧应该在什么时候**显示给用户**"。

```
时间轴:  0ms    33ms    67ms    100ms    133ms
          │      │       │       │        │
帧显示:  [F0]   [F1]    [F2]    [F3]     [F4]
          ↑      ↑       ↑       ↑        ↑
          PTS=0  PTS=33  PTS=67  PTS=100  PTS=133
```

PTS 的值通常不是以毫秒为单位，而是以**时间基（time_base）**为单位。例如，time_base = 1/90000 表示 1 个时间戳单位 = 1/90000 秒。如果一帧的 PTS = 9000，那么它应该在 9000 / 90000 = 0.1 秒 = 100 毫秒时显示。

### 4.2 DTS —— 什么时候解码

**DTS = 解码时间戳**。它告诉解码器："这一帧应该在什么时候被**送入解码器解码**"。

对于只有 I 帧（Intra Frame，帧内编码帧，不依赖其他帧的完整画面）和 P 帧（Predicted Frame，前向预测帧，只存储与前一帧的差异）的视频，DTS 和 PTS 是**相同**的——解码顺序就是显示顺序。但当存在 B 帧时，两者会出现差异。

### 4.3 为什么 DTS ≠ PTS？—— B 帧的影响

B 帧（Bidirectional Predicted Frame，双向预测帧）需要参考**前后两帧**才能解码。这就意味着：某些帧必须在它"应该显示"之前就被解码，好让 B 帧有参考依据。

```
显示顺序（按 PTS 排列）：I0   B1   B2   P3   B4   B5   P6
                         ↓    ↓    ↓    ↓    ↓    ↓    ↓
解码顺序（按 DTS 排列）：I0   P3   B1   B2   P6   B4   B5
```

为什么 P3 要在 B1、B2 之前解码？因为 B1 和 B2 需要参考 I0（前向参考）和 P3（后向参考）。P3 必须先被解码好，B1、B2 才有后向参考帧可用。

```
参考关系示意：

I0 ──────────────────────── P3 ──────────────────────── P6
 ↑ 前向参考    ↑ 后向参考    ↑ 前向参考    ↑ 后向参考
 └──── B1 ─────┘             └──── B4 ─────┘
 └──── B2 ─────┘             └──── B5 ─────┘
```

具体的 DTS/PTS 时间表（假设帧间隔 33ms，即 ~30fps）：

| 帧 | 类型 | DTS（解码时间） | PTS（显示时间） | 说明 |
|----|------|----------------|----------------|------|
| I0 | I | 0 ms | 0 ms | 解码和显示同时 |
| P3 | P | 33 ms | 100 ms | 提前解码，延后显示 |
| B1 | B | 67 ms | 33 ms | 在 I0+P3 都解码后才能解码 |
| B2 | B | 100 ms | 67 ms | 同上 |
| P6 | P | 133 ms | 200 ms | 提前解码 |
| B4 | B | 167 ms | 133 ms | 需要 P3+P6 作为参考 |
| B5 | B | 200 ms | 167 ms | 同上 |

> **KrKr2 源码对照**：`cpp/core/movie/ffmpeg/VideoPlayerVideo.cpp` 根据 PTS 决定帧的显示时机；`VideoPlayerAudio.cpp` 根据音频 PTS 调整输出节奏；`Clock.h` 中的 `CDVDClock` 类维护全局播放时钟，协调音视频的时间对齐。

### 4.4 time_base 与时间转换

FFmpeg 中所有时间戳都基于**时间基（time_base）**——一个有理数分数（`AVRational` 结构体），表示一个时间戳单位等于多少秒。

```cpp
// AVRational 定义（libavutil/rational.h）
typedef struct AVRational {
    int num; // 分子（numerator）
    int den; // 分母（denominator）
} AVRational;
// 含义：1 个时间戳单位 = num/den 秒

// 常见的 time_base 值：
// 容器层（MPEG-TS）：  {1, 90000}   → 1 单位 = 1/90000 秒
// 编码层（24fps 视频）：{1, 24}     → 1 单位 = 1/24 秒
// FFmpeg 内部：        AV_TIME_BASE = 1000000（微秒）

// 将时间戳转换为秒
double pts_to_seconds(int64_t pts, AVRational time_base) {
    // av_q2d() 将 AVRational 转为 double：num / den
    return (double)pts * av_q2d(time_base);
    // 等价于：(double)pts * time_base.num / time_base.den
}

// 在不同 time_base 之间转换时间戳
int64_t convert_ts(int64_t ts,
                   AVRational src_tb, AVRational dst_tb) {
    // av_rescale_q() 精确转换，避免浮点误差
    return av_rescale_q(ts, src_tb, dst_tb);
}

// 使用示例
void demo_time_conversion() {
    AVRational tb_90k = {1, 90000};  // MPEG-TS 时间基
    AVRational tb_ms  = {1, 1000};   // 毫秒时间基

    int64_t pts = 450000;  // MPEG-TS 时间戳
    double sec = pts_to_seconds(pts, tb_90k);
    printf("PTS=%lld → %.3f 秒\n", pts, sec);
    // 输出: PTS=450000 → 5.000 秒

    int64_t ms = convert_ts(pts, tb_90k, tb_ms);
    printf("PTS=%lld (90kHz) → %lld ms\n", pts, ms);
    // 输出: PTS=450000 (90kHz) → 5000 ms
}
```

## 五、关键帧、预测帧与 GOP

### 5.1 三种帧类型

视频压缩的核心思想是**去除冗余**——相邻帧之间大部分内容相同（背景不变、人物只移动了几个像素），只编码差异即可大幅缩小数据。

| 帧类型 | 全称 | 说明 | 相对大小 |
|--------|------|------|---------|
| **I 帧** | Intra Frame（帧内编码帧） | 完整独立的一帧画面，不依赖任何其他帧 | 最大（100%） |
| **P 帧** | Predicted Frame（前向预测帧） | 只存储与前一个 I 帧或 P 帧的差异数据 | 中等（~20%） |
| **B 帧** | Bidirectional Frame（双向预测帧） | 存储与前后参考帧的差异，压缩率最高 | 最小（~8%） |

```
I 帧（完整画面）：              P 帧（差异数据）：
┌─────────────┐                ┌─────────────┐
│ 🌳  🏠  ☀️ │ ← 100KB        │ ─── ── ──── │ ← ~20KB
│ 🚗    🚶   │                │   →🚗 🚶   │ （只有车移动了）
└─────────────┘                └─────────────┘

B 帧（前后差异）：
┌─────────────┐
│ ─── ── ──── │ ← ~8KB
│  →🚗─ ──   │ （参考前后帧计算差异，数据极少）
└─────────────┘
```

### 5.2 GOP（Group of Pictures）

GOP（Group of Pictures，画面组）是一组从 I 帧开始到下一个 I 帧之前的帧序列。**I 帧是视频跳转（Seek）的锚点**——播放器只能精确跳转到 I 帧的位置，因为只有 I 帧不依赖其他帧就能解码显示。

```
GOP 1                    GOP 2                    GOP 3
┌──────────────────┐    ┌──────────────────┐    ┌──────
I  B  B  P  B  B  P    I  B  B  P  B  B  P    I ...
│←  GOP 长度 = 7  →│    │←  GOP 长度 = 7  →│
```

GOP 长度对播放体验的影响：

| GOP 长度 | 压缩率 | Seek 精度 | 适用场景 |
|----------|--------|-----------|---------|
| 短（1-2） | 低 | 高（几乎逐帧定位） | 视频编辑/剪辑 |
| 中（12-15） | 中等 | 中等 | 通用视频播放 |
| 长（30-60+） | 高 | 低（跳转不精确） | 流媒体、直播 |

> **为什么拖动进度条有时很慢？** 当用户拖动进度条到某个位置时，播放器必须先找到最近的 I 帧（因为只有 I 帧能独立解码），然后从该 I 帧开始逐帧解码直到目标位置。如果 GOP 长度为 60 帧，最坏情况需要额外解码 59 帧才能到达目标。KrKr2 的 `DemuxFFmpeg.cpp` 中的 `SeekTime()` 方法就是处理这个 Seek 逻辑的。

### 5.3 在 FFmpeg 中判断帧类型

```cpp
#include <libavformat/avformat.h>
#include <libavcodec/avcodec.h>

// 分析视频文件的帧类型分布
void analyze_frame_types(const char *filename) {
    AVFormatContext *fmt_ctx = NULL;
    avformat_open_input(&fmt_ctx, filename, NULL, NULL);
    avformat_find_stream_info(fmt_ctx, NULL);

    // 找到视频流索引
    int video_idx = -1;
    for (unsigned i = 0; i < fmt_ctx->nb_streams; i++) {
        if (fmt_ctx->streams[i]->codecpar->codec_type
            == AVMEDIA_TYPE_VIDEO) {
            video_idx = i;
            break;
        }
    }

    // 统计帧类型（通过 AVPacket 的 flags 判断关键帧）
    AVPacket *pkt = av_packet_alloc(); // 分配数据包结构
    int i_count = 0, total = 0;

    // av_read_frame() 从容器中读取下一个数据包
    while (av_read_frame(fmt_ctx, pkt) >= 0) {
        if (pkt->stream_index == video_idx) {
            total++;
            // AV_PKT_FLAG_KEY 标志表示这是一个关键帧（I 帧）
            if (pkt->flags & AV_PKT_FLAG_KEY) {
                i_count++;
                printf("关键帧 #%d: PTS=%lld, 大小=%d bytes\n",
                       i_count, pkt->pts, pkt->size);
            }
        }
        // 释放 pkt 内部引用的数据，为读取下一个包做准备
        av_packet_unref(pkt);
    }

    printf("\n总帧数: %d, 关键帧: %d, GOP 平均长度: %.1f\n",
           total, i_count,
           i_count > 0 ? (double)total / i_count : 0.0);

    av_packet_free(&pkt);       // 释放 pkt 结构本身
    avformat_close_input(&fmt_ctx);
}

int main() {
    analyze_frame_types("test_output.mp4");
    return 0;
}
```

## 常见错误及解决方案

### 错误 1：用 DTS 而非 PTS 控制显示时间

**症状**：含 B 帧的视频画面顺序错乱，画面"跳跃"或"闪烁"。

**原因**：DTS 是解码顺序，PTS 是显示顺序。含 B 帧时两者不同。

**正确做法**：显示帧时永远使用 PTS；DTS 只用于解码排序。对于纯 I+P 帧视频（无 B 帧），两者相同。

### 错误 2：不同 time_base 之间直接比较时间戳

**症状**：音视频不同步——视频流的 PTS 值可能是几万（time_base=1/90000），音频流可能是几千（time_base=1/44100），直接比较毫无意义。

**正确做法**：使用 `av_rescale_q()` 将时间戳统一到同一个 time_base 后再比较，或者都转换为秒（浮点数）。

### 错误 3：Seek 后立即显示帧导致花屏

**症状**：拖动进度条后画面出现花屏/绿屏/马赛克。

**原因**：Seek 跳到 I 帧附近，但解码器拿到的第一个包可能不是 I 帧，缺少参考帧导致无法正确解码。

**正确做法**：Seek 后调用 `avcodec_flush_buffers()` 清空解码器缓存，然后从最近的 I 帧开始重新解码。KrKr2 的 `DemuxFFmpeg::SeekTime()` 使用 `AVSEEK_FLAG_BACKWARD` 标志确保向前 Seek 到最近的关键帧。

## 本节小结

- **帧率**决定视频流畅度（24fps=电影，60fps=游戏级流畅）
- **采样率**决定音频频率范围（44.1kHz=CD 音质，能还原 20kHz 以下的声音）
- **位深**决定音频动态范围（16bit=96dB，覆盖人耳全部范围）
- **比特率**反映数据的每秒数据量——未压缩 1080p 视频约 600 Mbps，H.264 编码可压缩到 5-20 Mbps
- **PTS** 控制帧的显示时机，**DTS** 控制帧的解码顺序——含 B 帧时两者不同
- **I/P/B 帧**构成 **GOP** 结构：I 帧是独立完整帧（Seek 锚点），P 帧存前向差异，B 帧存前后差异
- FFmpeg 中时间戳基于 **time_base**（有理数分数），用 `av_q2d()` 转秒、用 `av_rescale_q()` 跨 time_base 转换

## 练习题与答案

### 题目 1：PTS/DTS 排列

给定以下 GOP 结构（显示顺序）：

```
I0  B1  B2  P3  B4  P5
```

请写出：
1. 解码顺序（DTS 顺序）
2. 每帧的 DTS 和 PTS 值（假设帧间隔 33ms）

<details>
<summary>查看答案</summary>

**1. 解码顺序**：I0 → P3 → B1 → B2 → P5 → B4

B1、B2 需要参考 I0（前向）和 P3（后向），所以 P3 必须先于 B1、B2 解码。同理 B4 需要 P3 和 P5。

**2. DTS/PTS 值**：

| 解码位置 | 帧 | DTS | PTS |
|---------|-----|------|------|
| 1 | I0 | 0 ms | 0 ms |
| 2 | P3 | 33 ms | 99 ms |
| 3 | B1 | 66 ms | 33 ms |
| 4 | B2 | 99 ms | 66 ms |
| 5 | P5 | 132 ms | 165 ms |
| 6 | B4 | 165 ms | 132 ms |

验证：按 PTS 排序 → I0(0) → B1(33) → B2(66) → P3(99) → B4(132) → P5(165) ✓ 恢复原始显示顺序。

</details>

### 题目 2：比特率计算

1. 计算未压缩 4K（3840×2160）视频，YUV 4:2:0 格式，60fps 的每秒数据量（MB/s）
2. 一首 4 分钟的歌曲以 Vorbis 编码、256kbps 存储，文件大约多大（MB）？
3. 同一首歌以未压缩 PCM（48kHz, 16bit, 立体声）存储，文件多大？压缩比多少？

<details>
<summary>查看答案</summary>

**1. 4K YUV420 60fps**：

```
每帧 = 3840 × 2160 × 12 bit = 99,532,800 bit = 12,441,600 bytes ≈ 11.87 MB
每秒 = 11.87 × 60 = 712.2 MB/s ≈ 5,697.6 Mbps
```

一分钟未压缩约 41.7 GB！所以 4K 视频必须使用 H.265 等高压缩比编码。

**2. 4 分钟 Vorbis 256kbps**：

```
256 kbps × 240 秒 = 61,440 kbit = 7,680 KB ≈ 7.5 MB
```

**3. 4 分钟 PCM 48kHz/16bit/立体声**：

```
48000 × 2 × 2 × 240 = 46,080,000 bytes ≈ 43.9 MB
压缩比 = 43.9 / 7.5 ≈ 5.9:1
```

</details>

### 题目 3：time_base 换算

一个视频流的 time_base 为 {1, 90000}，某帧的 PTS 为 270000。
1. 这帧应该在第几秒显示？
2. 如果要转换到 time_base={1, 1000}（毫秒），PTS 值变为多少？
3. 写出用 `av_rescale_q()` 进行转换的 C++ 代码。

<details>
<summary>查看答案</summary>

**1.** 270000 × (1/90000) = **3.0 秒**

**2.** 3.0 秒 × 1000 = **3000**（毫秒单位的 PTS）

**3.** 代码：

```cpp
#include <libavutil/mathematics.h>  // av_rescale_q
#include <libavutil/rational.h>     // AVRational

int main() {
    AVRational src_tb = {1, 90000}; // 源时间基
    AVRational dst_tb = {1, 1000};  // 目标时间基（毫秒）
    int64_t src_pts = 270000;

    // av_rescale_q 精确转换，内部用 128 位整数避免溢出
    int64_t dst_pts = av_rescale_q(src_pts, src_tb, dst_tb);
    printf("PTS: %lld (1/90000) → %lld (1/1000)\n",
           src_pts, dst_pts);
    // 输出: PTS: 270000 (1/90000) → 3000 (1/1000)

    // 也可以直接转秒
    double seconds = (double)src_pts * av_q2d(src_tb);
    printf("= %.3f 秒\n", seconds);
    // 输出: = 3.000 秒
    return 0;
}
```

</details>
