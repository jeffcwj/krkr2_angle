## 一、RGB 与 YUV 色彩模型

### 1.1 RGB —— 显示器使用的色彩模型

计算机显示器的每个像素由**红（R）、绿（G）、蓝（B）**三个颜色通道混合而成。每个通道通常是 8 位（0-255），三个通道共 24 位（约 1677 万种颜色）。加上透明度通道（Alpha，控制像素的透明程度），就是 32 位 RGBA。

```
RGB 像素（24-bit，无透明通道）：
┌──────┬──────┬──────┐
│  R   │  G   │  B   │  每个通道 8 bit = 1 字节
│ 0-255│ 0-255│ 0-255│  共 3 字节/像素
└──────┴──────┴──────┘

RGBA 像素（32-bit，含透明通道）：
┌──────┬──────┬──────┬──────┐
│  R   │  G   │  B   │  A   │  共 4 字节/像素
│ 0-255│ 0-255│ 0-255│ 0-255│
└──────┴──────┴──────┴──────┘
```

> **KrKr2 源码对照**：KrKr2 内部使用 **ARGB** 格式（`tjs_uint32` = `uint32_t`），每像素 4 字节，排列为 `0xAARRGGBB`。在 `cpp/core/visual/` 的 blend 函数中你会大量看到这种格式。

### 1.2 YUV —— 视频编码使用的色彩模型

视频编码几乎全部使用 **YUV** 色彩模型：

- **Y**（Luminance / 亮度）：画面的明暗信息，人眼最敏感的分量
- **U**（Cb / 蓝色色度差）：蓝色与亮度的偏差值
- **V**（Cr / 红色色度差）：红色与亮度的偏差值

为什么视频编码要用 YUV 而不是 RGB？三个原因：

| 原因 | 解释 |
|------|------|
| **人眼对亮度敏感，对色度不敏感** | 可以降低 U/V 分辨率而人眼几乎察觉不到差异，从而大幅压缩数据 |
| **历史兼容** | YUV 与黑白电视兼容——Y 分量就是黑白画面，彩色电视只是在 Y 基础上加了 U/V |
| **压缩效率更高** | Y/U/V 三个分量的信息特性不同，可以分别优化压缩策略 |

```
YUV 分量可视化：

原始彩色画面      Y（亮度）        U（蓝色差）      V（红色差）
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 🌈 彩色   │ →  │ ⬛⬜ 灰度 │ +  │ 蓝偏移   │ +  │ 红偏移   │
│ 画面      │    │ 画面     │    │ 信息     │    │ 信息     │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
```

### 1.3 简单的 YUV↔RGB 转换

数学上，YUV 和 RGB 之间可以互相转换。有两套主要标准：

**BT.601 标准**（Standard Definition，标清视频，分辨率 ≤720×576）：

```
// RGB → YUV
Y  =  0.299 * R + 0.587 * G + 0.114 * B
U  = -0.169 * R - 0.331 * G + 0.500 * B + 128
V  =  0.500 * R - 0.419 * G - 0.081 * B + 128

// YUV → RGB
R = Y + 1.402 * (V - 128)
G = Y - 0.344 * (U - 128) - 0.714 * (V - 128)
B = Y + 1.772 * (U - 128)
```

**BT.709 标准**（High Definition，高清视频，分辨率 ≥1280×720）：

```
// YUV → RGB（BT.709）
R = Y + 1.5748 * (V - 128)
G = Y - 0.1873 * (U - 128) - 0.4681 * (V - 128)
B = Y + 1.8556 * (U - 128)
```

> **何时用哪个标准？** SD（标清，Standard Definition）视频用 BT.601，HD（高清，High Definition）及以上用 BT.709。FFmpeg 的 `sws_getContext()` 会根据分辨率自动选择，但也可以通过 `SWS_CS_*` 标志手动指定。

## 二、YUV 采样格式

YUV 的关键变量是 U/V 相对于 Y 的**采样比例**——不同比例决定了不同的数据量和画质。

### 2.1 三种常见采样格式

```
YUV 4:4:4（无降采样）—— 每个像素都有完整的 Y、U、V 数据
Y:  □□□□    U:  □□□□    V:  □□□□
    □□□□        □□□□        □□□□
每像素: Y + U + V = 3 字节 → 24 bit/pixel

YUV 4:2:2（水平半采样）—— U/V 水平分辨率减半
Y:  □□□□    U:  □ □     V:  □ □
    □□□□        □ □         □ □
每像素平均: 2 字节 → 16 bit/pixel

YUV 4:2:0（水平 + 垂直半采样）—— 最常用，U/V 宽高各减半
Y:  □□□□    U:  □ □     V:  □ □
    □□□□        (与上行共享)    (与上行共享)
每像素平均: 1.5 字节 → 12 bit/pixel
```

**YUV 4:2:0 是视频编码的事实标准**——H.264、H.265、VP9、AV1 等编码默认输出 YUV420P。利用人眼对色度不敏感的特点，将色度数据量降为亮度的 1/4，节省 50% 的存储空间（相比 4:4:4）。

### 2.2 "4:2:0" 这些数字是什么意思？

这三个数字 `J:a:b` 表示在一个 `J×2` 像素的参考块中：

- **J**：参考块的宽度（通常是 4）
- **a**：第一行中 U/V 的采样数
- **b**：第二行中 U/V 的采样数

| 标记 | 第一行采样数 | 第二行采样数 | U/V 总数据比 |
|------|------------|------------|-------------|
| 4:4:4 | 4 | 4 | 100%（与 Y 相同） |
| 4:2:2 | 2 | 2 | 50%（水平减半） |
| 4:2:0 | 2 | 0 | 25%（宽高各减半） |
| 4:1:1 | 1 | 1 | 25%（仅水平 1/4） |

## 三、内存布局：Planar / Packed / Semi-planar

同样是 YUV 4:2:0 数据，在内存中可以有不同的排列方式。

### 3.1 Planar（平面格式）

Y、U、V 数据分别存储在**三个连续的内存区域**。FFmpeg 中带 `P` 后缀的格式都是 Planar。

```
内存布局（4×2 像素的 YUV420P）：
offset 0:   Y0 Y1 Y2 Y3 Y4 Y5 Y6 Y7    ← Y 平面（全部 Y 数据连续）
offset 8:   U0 U1                         ← U 平面（宽高各半）
offset 10:  V0 V1                         ← V 平面（宽高各半）
```

### 3.2 Packed（打包格式）

Y、U、V 数据交错存储。常见于 RGB 格式（RGB24、RGBA 等）。

```
RGB24 打包格式：
R0 G0 B0  R1 G1 B1  R2 G2 B2  R3 G3 B3  ...
│  像素0  │  像素1  │  像素2  │  像素3  │
```

### 3.3 Semi-planar（半平面格式）

Y 单独一个平面，U 和 V 交错存储在第二个平面。**NV12** 是最常见的 Semi-planar 格式（Android 硬件解码器经常输出 NV12）。

```
NV12 半平面格式（4×2 像素）：
平面 0:  Y0 Y1 Y2 Y3 Y4 Y5 Y6 Y7    ← Y 平面
平面 1:  U0 V0 U1 V1                  ← UV 交错平面
```

### 3.4 在 FFmpeg 中表示像素格式

```cpp
#include <libavutil/pixfmt.h>   // AVPixelFormat 定义
#include <libavutil/pixdesc.h>  // 像素格式描述函数

// 常用像素格式枚举
enum AVPixelFormat {
    AV_PIX_FMT_YUV420P,   // YUV 4:2:0 planar —— 最常见的视频解码输出
    AV_PIX_FMT_YUV422P,   // YUV 4:2:2 planar
    AV_PIX_FMT_YUV444P,   // YUV 4:4:4 planar
    AV_PIX_FMT_NV12,      // YUV 4:2:0 semi-planar（Y + UV交错）
    AV_PIX_FMT_NV21,      // YUV 4:2:0 semi-planar（Y + VU交错）
    AV_PIX_FMT_RGB24,     // RGB 打包格式（每像素 3 字节）
    AV_PIX_FMT_RGBA,      // RGBA 打包格式（每像素 4 字节）
    AV_PIX_FMT_BGRA,      // BGRA 打包格式
    // ...
};

// 查询像素格式信息
void print_pix_fmt_info(enum AVPixelFormat fmt) {
    const AVPixFmtDescriptor *desc = av_pix_fmt_desc_get(fmt);
    if (!desc) return;

    printf("格式名称: %s\n", desc->name);
    printf("每像素位数: %d\n",
           av_get_bits_per_pixel(desc));
    printf("平面数: %d\n", desc->nb_components);
    // 判断是否是 planar 格式
    printf("是否 Planar: %s\n",
           (desc->flags & AV_PIX_FMT_FLAG_PLANAR)
               ? "是" : "否");
}
```

> **KrKr2 源码对照**：`cpp/core/movie/ffmpeg/RenderFormats.h` 定义了 `ERenderFormat` 枚举，包括 `RENDER_FMT_YUV420P`、`RENDER_FMT_NV12`、`RENDER_FMT_YUV420P10` 等。`BaseRenderer` 根据解码器输出的像素格式选择对应的渲染路径（OpenGL 纹理格式/着色器）。

## 四、使用 libswscale 进行像素格式转换

解码器输出通常是 YUV420P 格式，但显示到屏幕需要 RGB/RGBA。FFmpeg 的 **libswscale** 库负责高效的像素格式转换和图像缩放。

### 4.1 基本转换流程

```cpp
#include <libswscale/swscale.h>  // 像素格式转换库
#include <libavutil/imgutils.h>  // 图像工具函数

// 将 YUV420P 帧转换为 RGB24
void convert_yuv_to_rgb(AVFrame *yuv_frame, int width, int height) {
    // 1. 创建转换上下文
    // 参数：源宽高+格式，目标宽高+格式，缩放算法
    struct SwsContext *sws_ctx = sws_getContext(
        width, height, AV_PIX_FMT_YUV420P,   // 源：YUV420P
        width, height, AV_PIX_FMT_RGB24,      // 目标：RGB24
        SWS_BILINEAR,    // 缩放算法（双线性插值，速度与质量平衡）
        NULL, NULL, NULL  // 源滤镜、目标滤镜、滤镜参数（不需要）
    );
    if (!sws_ctx) {
        printf("无法创建转换上下文\n");
        return;
    }

    // 2. 分配 RGB 输出缓冲区
    int rgb_linesize = width * 3;  // RGB24 每行字节数
    uint8_t *rgb_buffer = (uint8_t *)malloc(rgb_linesize * height);

    uint8_t *dst_data[1] = { rgb_buffer };   // 输出数据指针数组
    int dst_linesize[1] = { rgb_linesize };   // 输出每行字节数数组

    // 3. 执行转换
    // 参数：上下文，源数据+行大小，起始行，行数，目标数据+行大小
    sws_scale(
        sws_ctx,
        yuv_frame->data,       // 源 YUV 数据（3 个平面的指针数组）
        yuv_frame->linesize,   // 源每行字节数数组
        0,                     // 起始行（从第 0 行开始）
        height,                // 处理的行数
        dst_data,              // 目标 RGB 数据
        dst_linesize           // 目标每行字节数
    );

    printf("转换完成: YUV420P → RGB24, %d×%d\n", width, height);
    printf("RGB 缓冲区大小: %d bytes\n", rgb_linesize * height);

    // 4. 使用 rgb_buffer 进行显示或保存...

    // 5. 清理
    free(rgb_buffer);
    sws_freeContext(sws_ctx);
}
```

### 4.2 缩放算法选择

`sws_getContext()` 的缩放算法参数影响转换质量和速度：

| 算法 | 常量名 | 速度 | 质量 | 适用场景 |
|------|--------|------|------|---------|
| 最近邻 | `SWS_POINT` | 极快 | 低（锯齿） | 像素风格、调试 |
| 双线性 | `SWS_BILINEAR` | 快 | 中 | 实时视频播放（推荐） |
| 双三次 | `SWS_BICUBIC` | 中 | 高 | 静态图像/截图 |
| Lanczos | `SWS_LANCZOS` | 慢 | 极高 | 高质量缩放 |

### 4.3 访问 YUV420P 帧的原始数据

```cpp
// 理解 AVFrame 中 YUV420P 数据的内存布局
void access_yuv_data(AVFrame *frame) {
    int width = frame->width;
    int height = frame->height;

    // Y 平面 —— 与原始分辨率相同
    uint8_t *y_data = frame->data[0];       // Y 数据起始指针
    int y_linesize = frame->linesize[0];    // Y 每行字节数（可能 > width，因对齐）

    // U 平面 —— 宽高各是 Y 的一半（YUV420）
    uint8_t *u_data = frame->data[1];
    int u_linesize = frame->linesize[1];    // ≈ width / 2

    // V 平面
    uint8_t *v_data = frame->data[2];
    int v_linesize = frame->linesize[2];    // ≈ width / 2

    // 读取坐标 (x, y) 处的 YUV 值
    int x = 100, y = 50;
    uint8_t Y_val = y_data[y * y_linesize + x];
    // 注意：U/V 的坐标要除以 2（因为分辨率减半）
    uint8_t U_val = u_data[(y / 2) * u_linesize + (x / 2)];
    uint8_t V_val = v_data[(y / 2) * v_linesize + (x / 2)];

    printf("像素 (%d, %d): Y=%d, U=%d, V=%d\n",
           x, y, Y_val, U_val, V_val);

    // 注意 linesize vs width 的区别：
    // linesize 是内存中每行的实际字节数（含对齐填充）
    // width 是图像的实际像素宽度
    // linesize >= width（可能因 CPU 对齐需要多出几个字节）
    printf("Y 平面: width=%d, linesize=%d (差=%d 字节对齐填充)\n",
           width, y_linesize, y_linesize - width);
}
```

## 五、音频采样格式

### 5.1 整数格式 vs 浮点格式

音频数据的每个采样点可以用整数或浮点数表示：

| 格式 | FFmpeg 枚举 | 取值范围 | 用途 |
|------|-------------|---------|------|
| 8-bit 无符号 | `AV_SAMPLE_FMT_U8` | [0, 255]，128 为静音 | 早期游戏音效 |
| 16-bit 有符号 | `AV_SAMPLE_FMT_S16` | [-32768, 32767]，0 为静音 | **CD 音质，最常见** |
| 32-bit 有符号 | `AV_SAMPLE_FMT_S32` | [-2³¹, 2³¹-1] | 专业音频 |
| 32-bit 浮点 | `AV_SAMPLE_FMT_FLT` | [-1.0, 1.0]，0.0 为静音 | 音频处理中间格式 |
| 64-bit 浮点 | `AV_SAMPLE_FMT_DBL` | [-1.0, 1.0] | 高精度处理 |

> **KrKr2 内部格式**：KrKr2 音频系统默认使用 **16-bit S16 交错格式**（`BitsPerSample=16, IsFloat=false`）。`FFWaveDecoder.cpp` 在解码后将 FFmpeg 输出的任何采样格式统一转换为 S16 交错格式。

### 5.2 Planar vs Interleaved（交错）

音频同样有两种内存布局。这与视频像素格式中的 Planar/Packed 概念完全对应：

```
立体声（2 声道）交错格式（Interleaved）：
L0 R0 L1 R1 L2 R2 L3 R3 ...    ← 左右声道样本交替存储
│ 帧0 │ 帧1 │ 帧2 │ 帧3 │        （一个"帧"= 所有声道的一组样本）

立体声（2 声道）平面格式（Planar）：
L0 L1 L2 L3 L4 L5 L6 L7 ...    ← 左声道连续存储（data[0]）
R0 R1 R2 R3 R4 R5 R6 R7 ...    ← 右声道连续存储（data[1]）
```

FFmpeg 中，Planar 格式名称带 **P** 后缀：

```cpp
AV_SAMPLE_FMT_S16   // 16-bit 有符号整数，交错格式
AV_SAMPLE_FMT_S16P  // 16-bit 有符号整数，平面格式
AV_SAMPLE_FMT_FLT   // 32-bit 浮点，交错格式
AV_SAMPLE_FMT_FLTP  // 32-bit 浮点，平面格式（现代解码器常用）
```

> **为什么解码器常输出 Planar？** 大多数现代音频解码器（如 AAC、Vorbis 解码器）内部处理是分声道进行的，直接输出 Planar 格式更高效。但播放设备（声卡/音频 API）通常期望 Interleaved 格式，所以需要转换。

### 5.3 在 FFmpeg 中访问音频数据

```cpp
// 访问解码后的音频帧数据
void access_audio_data(AVFrame *frame) {
    enum AVSampleFormat fmt =
        (enum AVSampleFormat)frame->format;

    printf("采样格式: %s\n", av_get_sample_fmt_name(fmt));
    printf("样本数: %d\n", frame->nb_samples);
    printf("声道数: %d\n", frame->ch_layout.nb_channels);

    int channels = frame->ch_layout.nb_channels;

    // 判断是否为平面格式
    if (av_sample_fmt_is_planar(fmt)) {
        // Planar：每个声道的数据在不同的 data[] 指针中
        printf("布局: Planar（每声道独立缓冲区）\n");
        for (int ch = 0; ch < channels; ch++) {
            float *samples = (float *)frame->data[ch];
            printf("  声道 %d 前 3 个样本: %.4f, %.4f, %.4f\n",
                   ch, samples[0], samples[1], samples[2]);
        }
    } else {
        // Interleaved：所有声道数据交错在 data[0] 中
        printf("布局: Interleaved（交错存储）\n");
        int16_t *samples = (int16_t *)frame->data[0];
        for (int i = 0; i < 3; i++) {
            printf("  帧 %d: L=%d, R=%d\n",
                   i,
                   samples[i * channels],
                   samples[i * channels + 1]);
        }
    }
}
```

## 六、使用 libswresample 进行音频格式转换

FFmpeg 的 **libswresample** 库负责音频重采样（改变采样率）和格式转换（改变采样格式、声道布局）。

### 6.1 典型转换场景

```
解码器输出                     播放设备期望
┌──────────────┐              ┌──────────────┐
│ FLTP (planar) │  ─ swr ─→  │ S16 (interleaved) │
│ 48000 Hz      │              │ 44100 Hz      │
│ 5.1 声道      │              │ 立体声        │
└──────────────┘              └──────────────┘
```

### 6.2 完整转换示例

```cpp
#include <libswresample/swresample.h>  // 音频重采样库
#include <libavutil/channel_layout.h>  // 声道布局定义
#include <libavutil/samplefmt.h>       // 采样格式定义
#include <libavutil/opt.h>             // 选项设置

// 将解码后的音频帧转换为 S16 交错格式
int convert_audio(AVFrame *input_frame) {
    SwrContext *swr_ctx = NULL;

    // 1. 分配重采样上下文
    // swr_alloc() 创建一个空的重采样上下文
    swr_ctx = swr_alloc();
    if (!swr_ctx) {
        printf("无法分配重采样上下文\n");
        return -1;
    }

    // 2. 设置输入/输出参数
    // 输出：立体声 / S16 交错 / 44100Hz
    AVChannelLayout out_ch_layout = AV_CHANNEL_LAYOUT_STEREO;
    av_opt_set_chlayout(swr_ctx, "out_chlayout",
                        &out_ch_layout, 0);
    av_opt_set_int(swr_ctx, "out_sample_rate", 44100, 0);
    av_opt_set_sample_fmt(swr_ctx, "out_sample_fmt",
                          AV_SAMPLE_FMT_S16, 0);

    // 输入：从 AVFrame 获取参数
    av_opt_set_chlayout(swr_ctx, "in_chlayout",
                        &input_frame->ch_layout, 0);
    av_opt_set_int(swr_ctx, "in_sample_rate",
                   input_frame->sample_rate, 0);
    av_opt_set_sample_fmt(swr_ctx, "in_sample_fmt",
                          (enum AVSampleFormat)input_frame->format,
                          0);

    // 3. 初始化（验证参数有效性）
    if (swr_init(swr_ctx) < 0) {
        printf("初始化重采样上下文失败\n");
        swr_free(&swr_ctx);
        return -1;
    }

    // 4. 计算输出样本数
    // swr_get_delay() 返回内部缓存的延迟样本数
    int64_t out_samples = av_rescale_rnd(
        swr_get_delay(swr_ctx, input_frame->sample_rate)
            + input_frame->nb_samples,
        44100,                      // 目标采样率
        input_frame->sample_rate,   // 源采样率
        AV_ROUND_UP                 // 向上取整
    );

    // 5. 分配输出缓冲区
    uint8_t *out_buf = NULL;
    int out_linesize;
    av_samples_alloc(&out_buf, &out_linesize,
                     2,              // 输出声道数
                     (int)out_samples,
                     AV_SAMPLE_FMT_S16,
                     0);             // 对齐

    // 6. 执行转换
    int converted = swr_convert(
        swr_ctx,
        &out_buf, (int)out_samples,                     // 输出
        (const uint8_t **)input_frame->data,            // 输入
        input_frame->nb_samples                         // 输入样本数
    );

    printf("转换完成: %d → %d 样本\n",
           input_frame->nb_samples, converted);

    // 7. out_buf 中就是 S16 交错格式的 PCM 数据
    // 可以直接写入声卡或文件

    // 8. 清理
    av_freep(&out_buf);
    swr_free(&swr_ctx);
    return converted;
}
```

> **KrKr2 源码对照**：`cpp/core/sound/FFWaveDecoder.cpp` 中就使用了类似的流程——解码 FFmpeg 音频帧后，通过 `swr_convert()` 将输出统一转换为 KrKr2 内部的 16-bit S16 交错格式。`WaveIntf.cpp` 中的 `TVPConvertPCMTo16bits()` 和 `TVPConvertPCMToFloat()` 则是 KrKr2 自己实现的轻量级格式转换函数。

## 常见错误及解决方案

### 错误 1：混淆 linesize 和 width

**症状**：将 YUV 帧数据按 `width` 逐行复制，结果画面出现斜纹或错位。

**原因**：`AVFrame.linesize[i]` 可能大于实际宽度——FFmpeg 出于 CPU 缓存对齐（通常 16 或 32 字节对齐）会在每行末尾添加填充字节。

**正确做法**：复制数据时使用 `linesize` 作为行步长，而不是 `width`：

```cpp
// 错误：直接用 width
memcpy(dst, frame->data[0], width * height);  // ✗

// 正确：逐行复制，使用 linesize
for (int y = 0; y < height; y++) {
    memcpy(dst + y * width,
           frame->data[0] + y * frame->linesize[0],
           width);  // 每行只复制 width 个有效像素
}
```

### 错误 2：解码器输出 Planar 音频却按 Interleaved 处理

**症状**：播放音频时听到噪音、杂音或间断声。

**原因**：现代解码器（如 AAC、Vorbis）通常输出 `FLTP`（32-bit float planar），但代码按照 `S16`（16-bit interleaved）处理，导致数据解读完全错误。

**正确做法**：检查 `AVFrame.format` 判断实际格式，使用 `swr_convert()` 统一转换：

```cpp
if (frame->format != AV_SAMPLE_FMT_S16) {
    // 使用 libswresample 转换为 S16 交错格式
    // ...（见上方完整示例）
}
```

### 错误 3：忘记初始化 SwsContext/SwrContext 就调用转换函数

**症状**：程序崩溃（段错误）或输出全黑/全静音。

**正确做法**：`sws_getContext()` 或 `swr_alloc()` + `swr_init()` 必须在第一次调用 `sws_scale()` / `swr_convert()` 之前完成。转换上下文可以复用——如果输入输出参数不变，不需要每帧重新创建。

## 本节小结

- **RGB** 是显示器色彩模型（R+G+B 三通道），**YUV** 是视频编码色彩模型（Y 亮度 + U/V 色度）
- 视频用 YUV 因为人眼对亮度敏感、对色度不敏感——**YUV 4:2:0** 将色度分辨率降为亮度的 1/4，节省 50% 数据
- 内存布局有三种：**Planar**（YYY...UUU...VVV...）、**Packed**（RGBRGBRGB...）、**Semi-planar**（YYY... + UVUV...）
- FFmpeg 用 `libswscale` 做像素格式转换（YUV→RGB），用 `libswresample` 做音频格式转换（FLTP→S16）
- 音频采样格式分整数/浮点、交错/平面四种组合——KrKr2 内部统一使用 **S16 交错格式**
- 操作 `AVFrame` 数据时注意 `linesize` ≠ `width`（有对齐填充），Planar 格式的不同声道/平面在不同的 `data[]` 指针中

### 题目 3：编写格式转换代码

写出将 `AV_SAMPLE_FMT_FLTP`（32-bit float planar）、48000Hz、立体声 音频帧转换为 `AV_SAMPLE_FMT_S16`（16-bit signed interleaved）、44100Hz、立体声 的 SwrContext 初始化代码。

<details>
<summary>查看答案</summary>

```cpp
#include <libswresample/swresample.h>
#include <libavutil/channel_layout.h>
#include <libavutil/opt.h>

SwrContext *create_audio_converter() {
    SwrContext *swr = swr_alloc();

    // 设置输出参数
    AVChannelLayout stereo = AV_CHANNEL_LAYOUT_STEREO;
    av_opt_set_chlayout(swr, "out_chlayout", &stereo, 0);
    av_opt_set_int(swr, "out_sample_rate", 44100, 0);
    av_opt_set_sample_fmt(swr, "out_sample_fmt",
                          AV_SAMPLE_FMT_S16, 0);

    // 设置输入参数
    av_opt_set_chlayout(swr, "in_chlayout", &stereo, 0);
    av_opt_set_int(swr, "in_sample_rate", 48000, 0);
    av_opt_set_sample_fmt(swr, "in_sample_fmt",
                          AV_SAMPLE_FMT_FLTP, 0);

    // 初始化
    if (swr_init(swr) < 0) {
        printf("初始化失败\n");
        swr_free(&swr);
        return NULL;
    }

    printf("转换器创建成功: FLTP/48kHz → S16/44.1kHz\n");
    return swr;
    // 用完后记得 swr_free(&swr);
}
```

关键点：
- 输入 `FLTP` 是 planar 格式，输出 `S16` 是 interleaved 格式——`swr_convert()` 会自动处理布局转换
- 采样率从 48000→44100 需要重采样——`swr_convert()` 内部使用高质量的插值算法
- 声道布局相同（都是立体声），不需要混音处理

</details>
