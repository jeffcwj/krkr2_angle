# 六、像素格式与色彩空间
## 六、像素格式与色彩空间

### 6.1 RGB 与 YUV

计算机显示器使用 **RGB**（红绿蓝）色彩模型，但视频编码几乎全部使用 **YUV** 色彩模型：

- **Y**（Luminance / 亮度）：画面的明暗信息
- **U**（Cb / 蓝色色度）：蓝色偏移
- **V**（Cr / 红色色度）：红色偏移

为什么用 YUV 而不是 RGB？

1. **人眼对亮度敏感，对色度不敏感** —— 可以降低 U/V 的分辨率而不影响感知质量
2. **历史兼容** —— YUV 与黑白电视兼容（Y 分量就是黑白画面）
3. **压缩效率更高** —— Y/U/V 分量的信息冗余度不同，可以针对性压缩

### 6.2 YUV 采样格式

YUV 格式的关键差异在于 U/V 相对于 Y 的采样比例：

```
YUV 4:4:4（无降采样）：
Y:  □□□□    U:  □□□□    V:  □□□□
    □□□□        □□□□        □□□□
每像素: Y+U+V = 3字节 → 24 bit/pixel

YUV 4:2:2（水平半采样）：
Y:  □□□□    U:  □ □     V:  □ □
    □□□□        □ □         □ □
每像素平均: 2字节 → 16 bit/pixel

YUV 4:2:0（最常用 — 水平+垂直半采样）：
Y:  □□□□    U:  □ □     V:  □ □
    □□□□
每像素平均: 1.5字节 → 12 bit/pixel
```

**YUV 4:2:0 是视频编码的事实标准** —— H.264、H.265、VP9 等编码默认输出 YUV420P。

### 6.3 FFmpeg 中的像素格式

FFmpeg 使用 `AVPixelFormat` 枚举表示像素格式：

```cpp
// 常见像素格式
enum AVPixelFormat {
    AV_PIX_FMT_YUV420P,   // YUV 4:2:0 planar（最常见）
    AV_PIX_FMT_YUV422P,   // YUV 4:2:2 planar
    AV_PIX_FMT_YUV444P,   // YUV 4:4:4 planar
    AV_PIX_FMT_NV12,      // YUV 4:2:0 semi-planar（Y平面+UV交错）
    AV_PIX_FMT_RGB24,     // RGB 打包格式
    AV_PIX_FMT_RGBA,      // RGBA 打包格式（带透明通道）
    // ...
};
```

> **Planar vs Packed**：
> - **Planar（平面格式）**：Y、U、V 数据分别存储在三个连续内存区域 → `YYYYUUVV`
> - **Packed（打包格式）**：Y、U、V 数据交错存储 → `YUVYUVYUV`
> - **Semi-planar（半平面）**：Y 单独一个平面，UV 交错存储 → `YYYY` + `UVUV`（NV12）

```cpp
// 解码后访问 YUV420P 帧数据
AVFrame *frame; // 已解码的视频帧

// Y 平面
uint8_t *y_data = frame->data[0];     // Y 数据指针
int y_linesize = frame->linesize[0];  // Y 每行字节数

// U 平面（宽高各是 Y 的一半）
uint8_t *u_data = frame->data[1];
int u_linesize = frame->linesize[1];

// V 平面
uint8_t *v_data = frame->data[2];
int v_linesize = frame->linesize[2];

// 访问坐标 (x, y) 的亮度值
uint8_t luminance = y_data[y * y_linesize + x];

// 访问坐标 (x, y) 的色度值（注意除以 2）
uint8_t cb = u_data[(y/2) * u_linesize + (x/2)];
uint8_t cr = v_data[(y/2) * v_linesize + (x/2)];
```

> **KrKr2 源码对照**：`cpp/core/movie/ffmpeg/RenderFormats.h` 定义了 `ERenderFormat` 枚举，包括 `RENDER_FMT_YUV420P`、`RENDER_FMT_NV12` 等。`BaseRenderer` 根据解码器输出的像素格式选择对应的渲染路径。

### 6.4 YUV → RGB 转换

最终显示到屏幕时，需要将 YUV 转换为 RGB：

```cpp
// BT.601 标准（SD 视频）
R = Y + 1.402 * (V - 128)
G = Y - 0.344 * (U - 128) - 0.714 * (V - 128)
B = Y + 1.772 * (U - 128)

// BT.709 标准（HD 视频）
R = Y + 1.5748 * (V - 128)
G = Y - 0.1873 * (U - 128) - 0.4681 * (V - 128)
B = Y + 1.8556 * (U - 128)
```

FFmpeg 提供 `libswscale` 库来高效完成这个转换：

```cpp
#include <libswscale/swscale.h>

// 创建转换上下文：YUV420P → RGB24
struct SwsContext *sws_ctx = sws_getContext(
    width, height, AV_PIX_FMT_YUV420P,  // 源格式
    width, height, AV_PIX_FMT_RGB24,    // 目标格式
    SWS_BILINEAR,    // 缩放算法
    NULL, NULL, NULL  // 滤镜（不需要）
);

// 执行转换
uint8_t *dst_data[1] = { rgb_buffer };  // 输出 RGB 缓冲区
int dst_linesize[1] = { width * 3 };    // 每行 RGB 字节数

sws_scale(
    sws_ctx,
    frame->data, frame->linesize,   // 源 YUV 数据
    0, height,                       // 起始行, 行数
    dst_data, dst_linesize           // 目标 RGB 数据
);

// 用完释放
sws_freeContext(sws_ctx);
```

## 七、音频采样格式

### 7.1 整数格式 vs 浮点格式

| 格式 | FFmpeg 枚举 | 范围 | 用途 |
|------|-------------|------|------|
| 8-bit unsigned | `AV_SAMPLE_FMT_U8` | [0, 255] | 早期游戏音效 |
| 16-bit signed | `AV_SAMPLE_FMT_S16` | [-32768, 32767] | CD 音质、最常见 |
| 32-bit signed | `AV_SAMPLE_FMT_S32` | [-2^31, 2^31-1] | 专业音频 |
| 32-bit float | `AV_SAMPLE_FMT_FLT` | [-1.0, 1.0] | 音频处理中间格式 |
| 64-bit float | `AV_SAMPLE_FMT_DBL` | [-1.0, 1.0] | 高精度处理 |

### 7.2 Planar vs Interleaved（交错）

音频同样有平面和交错两种内存布局：

```
立体声（2声道）交错格式（Interleaved）：
L R L R L R L R L R ...    ← 左右声道交替存储

立体声（2声道）平面格式（Planar）：
L L L L L L L L L L ...    ← 左声道连续存储
R R R R R R R R R R ...    ← 右声道连续存储
```

FFmpeg 中，Planar 格式名称带 `P` 后缀：

```cpp
AV_SAMPLE_FMT_S16   // 16-bit 交错格式
AV_SAMPLE_FMT_S16P  // 16-bit 平面格式
AV_SAMPLE_FMT_FLT   // 32-bit float 交错格式
AV_SAMPLE_FMT_FLTP  // 32-bit float 平面格式
```

> **KrKr2 源码对照**：`cpp/core/sound/WaveIntf.cpp` 中的 `TVPConvertPCMTo16bits` 和 `TVPConvertPCMToFloat` 函数负责在不同 PCM 格式之间转换。`FFWaveDecoder.cpp` 解码后会将 FFmpeg 输出的采样格式统一转换为 KrKr2 内部使用的 16-bit S16 交错格式。

### 7.3 使用 libswresample 进行音频格式转换

```cpp
#include <libswresample/swresample.h>

// 创建重采样上下文
SwrContext *swr_ctx = swr_alloc_set_opts(
    NULL,
    AV_CH_LAYOUT_STEREO,     // 输出：立体声
    AV_SAMPLE_FMT_S16,       // 输出：16-bit 交错
    44100,                    // 输出采样率
    av_get_default_channel_layout(
        codec_ctx->channels), // 输入声道布局
    codec_ctx->sample_fmt,    // 输入格式（可能是 FLTP）
    codec_ctx->sample_rate,   // 输入采样率
    0, NULL
);
swr_init(swr_ctx);

// 转换
uint8_t *out_buf;
int out_samples = swr_convert(
    swr_ctx,
    &out_buf, max_out_samples,        // 输出
    (const uint8_t **)frame->data,    // 输入
    frame->nb_samples                  // 输入样本数
);

// 用完释放
swr_free(&swr_ctx);
```

## 常见错误及解决方案

### 错误 1：混淆容器格式和编码格式

**症状**：认为 `.mp4` 文件一定是 H.264 编码，或者认为 H.264 只能放在 MP4 容器里。

**正确理解**：容器和编码是独立的两个维度。MP4 可以装 H.264 也可以装 H.265；H.264 可以放在 MP4、MKV、FLV 等任何容器中。

**验证方法**：用 `ffprobe` 实际查看文件，不要根据扩展名猜测。

### 错误 2：忽略 PTS/DTS 的区别

**症状**：只使用 DTS 来控制显示时间，导致含 B 帧的视频画面顺序错乱。

**正确做法**：
- **显示帧**时使用 PTS
- **解码排序**时使用 DTS
- 对于纯 I+P 帧的视频（无 B 帧），两者相同，不会出问题
- 对于含 B 帧的视频，必须严格区分

### 错误 3：采样率转换时未正确处理

**症状**：将 48kHz 音频直接按 44.1kHz 播放，导致语速变慢、音调偏低。

**正确做法**：使用 `libswresample`（`swr_convert`）进行采样率转换，它会自动处理插值和抗混叠滤波。

## 动手实践

### 实践 1：使用 ffprobe 分析媒体文件

准备任意一个视频文件（没有的话可以用 FFmpeg 生成一个测试文件）：

```bash
# 生成 5 秒测试视频（H.264 + AAC，640x480，30fps）
ffmpeg -f lavfi -i testsrc=duration=5:size=640x480:rate=30 \
       -f lavfi -i sine=frequency=440:duration=5:sample_rate=44100 \
       -c:v libx264 -c:a aac -pix_fmt yuv420p \
       test_output.mp4

# 分析容器和编码信息
ffprobe -v error -show_format -show_streams test_output.mp4

# 只看关键帧位置
ffprobe -v error -select_streams v -show_frames \
        -show_entries frame=pict_type,pts_time \
        test_output.mp4 | head -60
```

观察输出中：
- `format_name` 是什么容器？
- `codec_name` 是什么编码？
- 有几个流（stream）？各是什么类型？
- 关键帧（I）出现的频率是多少？

### 实践 2：计算比特率

手动计算以下参数：

1. **未压缩 1080p 视频**：1920×1080 像素，24fps，YUV420（12bit/pixel），每秒数据量是多少 MB/s？
2. **CD 音质音频**：44100Hz，16bit，立体声，每分钟数据量是多少 MB？
3. 如果 H.264 编码后视频比特率为 5Mbps，压缩比是多少？

## 本节小结

- **容器格式**（MP4/MKV/OGG）定义数据的存储结构；**编码格式**（H.264/Vorbis/AAC）定义数据的压缩算法。两者独立但需配合使用
- **采样率**决定音频频率范围（44.1kHz = CD 音质），**位深**决定动态范围（16bit = 96dB），**帧率**决定视频流畅度（24fps = 电影标准）
- **PTS**（显示时间戳）控制帧的显示时机，**DTS**（解码时间戳）控制帧的解码顺序；当存在 B 帧时两者不同
- **I/P/B 帧**构成 GOP 结构 —— I 帧是完整帧（Seek 锚点），P 帧存储前向差异，B 帧存储前后差异（最小）
- **YUV 4:2:0** 是视频编码的事实标准 —— 利用人眼对亮度敏感、对色度不敏感的特点，将色度分辨率降为亮度的 1/4
- 音频有**交错格式**（LRLRLR）和**平面格式**（LLLL...RRRR...）两种内存布局

## 练习题与答案

### 题目 1：容器与编码辨析

以下哪些是容器格式，哪些是编码格式？请分类：

`MP4`、`H.264`、`OGG`、`Vorbis`、`MKV`、`AAC`、`WAV`、`FLAC`、`AVI`、`Opus`

<details>
<summary>查看答案</summary>

**容器格式**：MP4、OGG、MKV、WAV、AVI

**编码格式**：H.264、Vorbis、AAC、FLAC、Opus

**易混淆点**：
- **WAV** 是容器，不是编码。WAV 通常包含 PCM 未压缩数据，但也可以包含其他编码（如 ADPCM、GSM）
- **FLAC** 是编码格式（无损压缩算法），虽然 `.flac` 文件有自己的容器，但 FLAC 也可以封装在 OGG 或 MKV 容器中
- **OGG** 是容器，**Vorbis/Opus** 是编码。"Ogg Vorbis" 是指使用 Vorbis 编码的 OGG 容器文件

</details>

### 题目 2：PTS/DTS 排列

给定以下 GOP 结构（显示顺序）：

```
I0  B1  B2  P3  B4  P5
```

请写出：
1. 解码顺序（DTS 顺序）
2. 每一帧的 DTS 和 PTS 值（假设帧间隔为 33ms）

<details>
<summary>查看答案</summary>

**1. 解码顺序**：

```
I0 → P3 → B1 → B2 → P5 → B4
```

原因：B1、B2 需要参考 I0（前向）和 P3（后向），所以 P3 必须在 B1、B2 之前解码。同理 B4 需要参考 P3 和 P5。

**2. DTS 和 PTS 值**：

| 解码顺序 | 帧 | DTS | PTS |
|---------|-----|-----|-----|
| 1 | I0 | 0 ms | 0 ms |
| 2 | P3 | 33 ms | 99 ms |
| 3 | B1 | 66 ms | 33 ms |
| 4 | B2 | 99 ms | 66 ms |
| 5 | P5 | 132 ms | 165 ms |
| 6 | B4 | 165 ms | 132 ms |

**验证**：按 PTS 排序后恢复显示顺序：I0(0) → B1(33) → B2(66) → P3(99) → B4(132) → P5(165) ✓

</details>

### 题目 3：比特率计算

计算以下参数：

1. 未压缩 720p (1280×720) 视频，YUV420 格式，30fps，每秒数据量是多少 MBps？
2. 一首 3 分钟的歌曲，以 Vorbis 编码、192kbps 比特率存储，文件大约多大？
3. 同一首歌以未压缩 PCM（44.1kHz, 16bit, 立体声）存储，文件大约多大？压缩比是多少？

<details>
<summary>查看答案</summary>

**1. 720p YUV420 视频每秒数据量**：

```
每帧字节数 = 1280 × 720 × 12 / 8 = 1,382,400 字节 ≈ 1.32 MB
（YUV420 每像素 12 bit = 1.5 字节）

每秒数据量 = 1.32 MB × 30fps = 39.6 MB/s ≈ 316.8 Mbps
```

**2. 3 分钟 Vorbis 192kbps 音频**：

```
数据量 = 192 kbps × 180 秒 = 34,560 kbit = 4,320 KB ≈ 4.22 MB
```

**3. 3 分钟未压缩 PCM 音频**：

```
数据量 = 44100 × 2声道 × 2字节 × 180秒 = 31,752,000 字节 ≈ 30.3 MB

压缩比 = 30.3 / 4.22 ≈ 7.2:1
```

Vorbis 在 192kbps 下将 CD 音质音频压缩了约 7 倍，同时保持人耳几乎无法区分的音质（属于有损压缩的高品质档）。

</details>
