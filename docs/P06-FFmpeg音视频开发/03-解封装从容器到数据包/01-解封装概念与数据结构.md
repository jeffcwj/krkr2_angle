## 3.1 什么是解封装（Demuxing）

### 3.1.1 容器与编码——两个独立维度

在第 1 章我们学过，一个视频文件由**容器格式**和**编码格式**共同决定。现在我们再明确一下这两个概念：

- **容器格式**（Container Format，又称封装格式、多媒体容器）：决定"数据怎么打包在一个文件里"。常见的容器格式有 MP4、MKV（Matroska）、AVI、FLV 等。容器本身不关心视频画面是什么样的，它只负责把多条数据流按照一定规则组织在一起，并附上时间戳（Timestamp，指示每段数据应该在什么时间播放）和索引表（Index，允许播放器快速跳转到指定时间位置）。
- **编码格式**（Codec，即 Coder-Decoder 的缩写，编解码器）：决定"原始画面/声音怎么压缩成体积小的数据"。常见的视频编码有 H.264（AVC）、H.265（HEVC）、VP9、AV1；常见的音频编码有 AAC、MP3、Opus、Vorbis。编码格式不关心数据怎么存储到文件，它只负责压缩和解压缩。

这两个维度是**正交**（互不依赖）的。同一种编码可以放在不同的容器里（H.264 视频可以放在 MP4 里，也可以放在 MKV 里），同一种容器也可以包含不同编码的流（一个 MKV 文件里可以同时有 H.264 视频流和 AAC 音频流）。

```
┌─────────────────────────────────────────────────────┐
│                  MP4 容器文件                         │
│                                                      │
│  ┌─────────────┐  ┌─────────────┐  ┌──────────────┐│
│  │ 视频流 #0    │  │ 音频流 #1    │  │ 字幕流 #2    ││
│  │ H.264 编码   │  │ AAC 编码     │  │ SRT 文本     ││
│  │              │  │              │  │              ││
│  │ [pkt][pkt]   │  │ [pkt][pkt]   │  │ [pkt][pkt]  ││
│  │ [pkt][pkt]   │  │ [pkt][pkt]   │  │             ││
│  └─────────────┘  └─────────────┘  └──────────────┘│
│                                                      │
│  文件头：流信息、时间信息、元数据                       │
│  索引表：快速定位到任意时间位置                         │
└─────────────────────────────────────────────────────┘
```

### 3.1.2 解封装的定义

**解封装**（Demuxing，全称 Demultiplexing，反多路复用）就是从容器文件中把各条流的压缩数据包（Packet，一小段压缩数据）**分离提取**出来的过程。如果把容器文件想象成一个包裹，里面混装了视频帧、音频帧、字幕帧，那么解封装就是"拆包裹并按类型分拣"。

```
MP4 文件 ──解封装──→ 视频 AVPacket 序列（H.264 压缩数据）
                   → 音频 AVPacket 序列（AAC 压缩数据）
                   → 字幕 AVPacket 序列（文本数据）
```

注意：解封装的输出仍然是**压缩**数据。要把压缩数据还原为原始的像素（Pixel）和采样（Sample）数据，需要下一步——**解码**（Decoding，第 4 章详讲）。

### 3.1.3 为什么需要解封装？

你可能会问：为什么不直接把原始视频数据和音频数据各存一个文件？原因有五：

1. **多流复用**：一个文件同时包含音频、视频、字幕等多条流，用户只需打开一个文件
2. **时间戳同步**：容器记录每个数据包的 PTS（Presentation Timestamp，显示时间戳——指示这个包应该在什么时间呈现给用户）和 DTS（Decoding Timestamp，解码时间戳——指示解码器应该在什么时间解码这个包），播放器据此精确同步音视频
3. **随机访问**（Seeking，跳转播放位置）：容器的索引表允许播放器快速跳转到任意时间位置，而不必从头读取
4. **元数据**（Metadata，描述性附加信息）：标题、作者、专辑封面、章节标记等信息可以存储在容器头部
5. **流选择**：多音轨（如中文/英文配音）、多字幕轨（如中文/英文字幕），用户可以动态切换

## 3.2 常见容器格式对比

不同容器格式各有特点，选择时需要考虑兼容性、功能需求和使用场景：

| 容器格式 | 常见扩展名 | 视频编码支持 | 音频编码支持 | 特点与使用场景 |
|----------|-----------|-------------|-------------|---------------|
| **MP4** | .mp4, .m4v, .m4a | H.264, H.265, VP9, AV1 | AAC, MP3, Opus, AC3 | 最通用的容器格式，浏览器/移动端首选。索引表（moov atom）在文件头部或尾部 |
| **MKV** | .mkv, .mka, .webm | 几乎所有编码 | 几乎所有编码 | 功能最全面的容器：支持无限数量的音频/字幕轨、章节标记、附件（字体文件等）。开源标准 |
| **AVI** | .avi | 几乎所有编码 | 几乎所有编码 | 微软上世纪 90 年代设计的古老格式，索引表在文件尾部（意味着文件损坏时无法 Seek），不支持可变帧率（VFR），但兼容性极广 |
| **FLV** | .flv | H.264, VP6 | AAC, MP3 | 早期网络流媒体（Flash Video）常用，结构简单。现已基本被 MP4/WebM 取代 |
| **TS** | .ts, .m2ts | H.264, H.265, MPEG-2 | AAC, AC3, MP2, DTS | 广播电视/直播标准（MPEG Transport Stream）。特点是可以边录边播、抗数据丢失——即使中间一段数据损坏，后面的数据仍然可以正常播放 |
| **OGG** | .ogg, .ogv, .oga | Theora, VP8 | Vorbis, Opus, FLAC | 完全开源的容器格式（Xiph.Org 基金会维护），常用于游戏和开源项目 |
| **WebM** | .webm | VP8, VP9, AV1 | Vorbis, Opus | MKV 的子集，专为 Web 设计。所有现代浏览器原生支持 |

> **KrKr2 中常见的视频格式**：KiriKiri 系列 Galgame 的 OP/ED（Opening/Ending 动画）通常使用 **AVI（MPEG-4/WMV 编码）** 或 **MP4（H.264 编码）** 格式。KrKr2 的 `CDVDDemuxFFmpeg`（位于 `cpp/core/movie/ffmpeg/DemuxFFmpeg.h`）通过 FFmpeg 的 `libavformat` 库自动探测容器格式，无需硬编码格式支持列表。

## 3.3 解封装核心数据结构

FFmpeg 的 `libavformat`（负责封装/解封装的库，参见第 2 章六大核心库详解）围绕四个核心结构体设计其解封装 API。理解这四个结构体及其关系，是掌握 FFmpeg 解封装的关键。

### 3.3.1 AVFormatContext——格式上下文（整个文件的"总管"）

`AVFormatContext` 是 `libavformat` 的核心结构体，代表一个已打开的媒体文件（或网络流）。你可以把它想象成一张"文件信息卡"——它记录了这个文件的格式是什么、包含哪些流、总时长多少、码率多少等全部信息。

```c
// AVFormatContext 关键字段（简化版，完整定义在 libavformat/avformat.h）
typedef struct AVFormatContext {
    // ── 格式信息 ──
    const AVInputFormat  *iformat;     // 输入格式描述符，包含格式名称和解析器
                                       // 例如 MP4 文件对应 "mov,mp4,m4a,3gp"
    const AVOutputFormat *oformat;     // 输出格式描述符（封装/输出时使用，解封装时为 NULL）

    // ── I/O 层 ──
    AVIOContext  *pb;                  // 底层 I/O 上下文，控制数据从哪里读取
                                       // 默认从文件系统读取，可替换为自定义回调
                                       //（从内存、网络或归档文件中读取）

    // ── 流信息 ──
    unsigned int  nb_streams;          // 文件中流的数量（视频+音频+字幕+...）
    AVStream    **streams;             // 流数组指针，通过 streams[i] 访问第 i 条流
                                       // 下标 i 即为 stream_index（流索引号）

    // ── 时间与码率 ──
    int64_t       duration;            // 文件总时长，单位是 AV_TIME_BASE（微秒，即 1/1000000 秒）
                                       // 例如 duration=90000000 表示 90 秒
    int64_t       bit_rate;            // 总码率（bits/second），例如 2500000 表示 2.5 Mbps
    int64_t       start_time;          // 第一帧的时间戳（AV_TIME_BASE 单位）

    // ── 元数据 ──
    AVDictionary *metadata;            // 文件级元数据键值对（标题、作者、专辑封面路径等）
                                       // 通过 av_dict_get() 读取
} AVFormatContext;
```

**生命周期**——`AVFormatContext` 从创建到销毁的完整过程：

```
avformat_alloc_context()          ← 手动分配（可选，avformat_open_input 会自动分配）
       ↓
avformat_open_input(&fmt_ctx, filename, ...)   ← 打开文件，填充格式信息
       ↓
avformat_find_stream_info(fmt_ctx, ...)        ← 探测流详细参数
       ↓
使用：遍历 streams[]、av_read_frame() 读包...
       ↓
avformat_close_input(&fmt_ctx)                 ← 关闭文件，释放所有内存
                                                  fmt_ctx 被置为 NULL
```

> **重要**：`avformat_close_input()` 不仅关闭文件 I/O，还会释放 `AVFormatContext` 自身以及它持有的所有 `AVStream`。调用后 `fmt_ctx` 指针变为 `NULL`，不可再使用。

### 3.3.2 AVStream——流信息（文件中的一路数据流）

一个媒体文件通常包含多条流（Stream）：至少一条视频流和一条音频流，可能还有字幕流、章节流等。每条流在 FFmpeg 中用一个 `AVStream` 结构体表示。

```c
// AVStream 关键字段（简化版，完整定义在 libavformat/avformat.h）
typedef struct AVStream {
    int            index;              // 流索引号（即在 AVFormatContext::streams[] 数组中的下标）
                                       // 这个值非常重要——av_read_frame() 读出的每个 AVPacket
                                       // 都通过 stream_index 字段标明自己属于哪条流

    AVCodecParameters *codecpar;       // 编解码参数（详见下文 3.3.3）
                                       // 包含编码类型(视频/音频)、编码器 ID、分辨率、采样率等

    AVRational     time_base;          // 时间基准——该流的时间戳单位
                                       // 例如 {1, 90000} 表示该流每个时间单位 = 1/90000 秒
                                       // pts=90000 就表示 1 秒处

    int64_t        duration;           // 该流的时长（以 time_base 为单位，不是秒！）
                                       // 转换为秒：duration * av_q2d(time_base)

    int64_t        nb_frames;          // 该流的帧数（可能为 0，表示容器未记录帧数）

    AVRational     avg_frame_rate;     // 平均帧率（仅视频流有意义）
                                       // 用 av_q2d(avg_frame_rate) 转为 double

    AVDictionary  *metadata;           // 流级元数据（如音频流的"语言"、字幕流的"标题"等）
} AVStream;
```

> **`AVRational` 是什么？** `AVRational` 是 FFmpeg 表示有理数（分数）的结构体，定义非常简单：
> ```c
> typedef struct AVRational {
>     int num;  // 分子（numerator）
>     int den;  // 分母（denominator）
> } AVRational;
> ```
> 用 `av_q2d(AVRational a)` 可以把它转成 `double`（即 `a.num / (double)a.den`）。FFmpeg 中大量使用 `AVRational` 来精确表示帧率、时间基准等值，避免浮点数精度丢失。

### 3.3.3 AVCodecParameters——编解码参数（流的编码属性）

每条 `AVStream` 都有一个 `codecpar` 字段，指向 `AVCodecParameters` 结构体。它描述了这条流使用什么编码、视频分辨率多大、音频采样率多高等信息。解码器需要这些参数才能正确初始化。

```c
// AVCodecParameters 关键字段（简化版）
typedef struct AVCodecParameters {
    // ── 通用字段 ──
    enum AVMediaType codec_type;       // 媒体类型：AVMEDIA_TYPE_VIDEO / AVMEDIA_TYPE_AUDIO
                                       //           AVMEDIA_TYPE_SUBTITLE / AVMEDIA_TYPE_DATA
    enum AVCodecID   codec_id;         // 编码器 ID：AV_CODEC_ID_H264 / AV_CODEC_ID_AAC / ...
                                       // 用 avcodec_get_name(codec_id) 可获取可读名称（如 "h264"）

    // ── 视频专用字段 ──
    int              width, height;    // 视频分辨率（像素），如 1920×1080
    enum AVPixelFormat format;         // 像素格式（如 AV_PIX_FMT_YUV420P，
                                       //  即 YUV 4:2:0 平面格式——第 1 章已介绍 YUV 概念）

    // ── 音频专用字段 ──
    int              sample_rate;      // 音频采样率（Hz），如 44100 表示每秒 44100 个采样点
    int              channels;         // 音频声道数，如 2 = 立体声
    uint64_t         channel_layout;   // 声道布局位掩码（bitmask），描述每个声道的空间位置
                                       // 如 AV_CH_LAYOUT_STEREO = 左声道 + 右声道
    enum AVSampleFormat format;        // 音频采样格式（如 AV_SAMPLE_FMT_FLTP = 32位浮点平面格式）
                                       // 注意：视频和音频共用 format 字段名，但枚举类型不同

    // ── 解码器初始化数据 ──
    uint8_t         *extradata;        // 解码器初始化数据（如 H.264 的 SPS/PPS 参数集——
                                       // 描述视频的编码配置，解码器必须先读取这些数据才能正常工作）
    int              extradata_size;   // extradata 的字节数
} AVCodecParameters;
```

**如何使用**：解封装完成后，我们通过 `codecpar` 判断流的类型和参数，然后传给解码器：

```c
// 示例：检查第 i 条流是不是视频流，并打印其分辨率
AVStream *stream = fmt_ctx->streams[i];
AVCodecParameters *par = stream->codecpar;

if (par->codec_type == AVMEDIA_TYPE_VIDEO) {
    printf("视频流 #%d: 编码=%s, 分辨率=%dx%d, 像素格式=%s\n",
           i,
           avcodec_get_name(par->codec_id),  // 如 "h264"
           par->width, par->height,           // 如 1920, 1080
           av_get_pix_fmt_name(par->format)); // 如 "yuv420p"
}

if (par->codec_type == AVMEDIA_TYPE_AUDIO) {
    printf("音频流 #%d: 编码=%s, 采样率=%d Hz, 声道=%d\n",
           i,
           avcodec_get_name(par->codec_id),  // 如 "aac"
           par->sample_rate,                  // 如 44100
           par->channels);                    // 如 2
}
```

### 3.3.4 AVPacket——压缩数据包（解封装的输出单元）

`AVPacket` 是解封装输出的基本单元。每次调用 `av_read_frame()` 都会返回一个 `AVPacket`，它包含一个流的一段压缩数据（对视频来说通常是一帧的压缩数据，对音频来说可能是若干帧的压缩数据）。

```c
// AVPacket 关键字段（简化版，完整定义在 libavcodec/packet.h）
typedef struct AVPacket {
    // ── 时间信息 ──
    int64_t  pts;              // PTS（Presentation Timestamp，显示时间戳）
                               // 表示这个包的数据应该在什么时间显示/播放
                               // 单位是所属流的 time_base
    int64_t  dts;              // DTS（Decoding Timestamp，解码时间戳）
                               // 表示解码器应该在什么时间解码这个包
                               // 通常 DTS ≤ PTS
    int64_t  duration;         // 这个包的显示时长（所属流的 time_base 单位）

    // ── 数据 ──
    uint8_t *data;             // 指向压缩数据的指针
    int      size;             // 压缩数据大小（字节）

    // ── 归属与标志 ──
    int      stream_index;     // 所属流的索引号（对应 AVFormatContext::streams[] 的下标）
    int      flags;            // 标志位，最重要的是 AV_PKT_FLAG_KEY（= 关键帧标志）
                               // 用法：if (pkt.flags & AV_PKT_FLAG_KEY) { /* 这是关键帧 */ }
    int64_t  pos;              // 这个包在文件中的字节偏移位置（调试用）
} AVPacket;
```

**PTS 与 DTS 的区别**：

- **PTS**（Presentation Timestamp）：帧应该被**显示**的时间
- **DTS**（Decoding Timestamp）：帧应该被**解码**的时间
- 对于没有 B 帧（Bidirectional Frame，双向预测帧——需要参考前后两帧才能解码的帧类型）的视频（如大多数游戏视频），PTS 总是等于 DTS
- 有 B 帧时，解码顺序和显示顺序不同。例如：显示顺序是 I-B-B-P，但解码顺序必须是 I-P-B-B（因为 B 帧需要先知道后面的 P 帧才能解码）。此时 DTS < PTS

```
显示顺序（PTS）：  I(0)  B(1)  B(2)  P(3)  B(4)  B(5)  P(6)
解码顺序（DTS）：  I(0)  P(3)  B(1)  B(2)  P(6)  B(4)  B(5)
                   ↑     ↑     ↑     ↑
                   先解码I  再解码P  然后B帧才能参考I和P来解码
```

**AVPacket 的生命周期与引用计数**：

`AVPacket` 使用**引用计数**（Reference Counting——通过计数器追踪有多少个变量正在使用同一块数据，计数器归零时自动释放数据内存）来管理其 `data` 指向的内存：

```
┌─────────────────────────────────────────────────────────┐
│         AVPacket 生命周期                                │
│                                                          │
│  av_read_frame(&pkt)                                     │
│      │  ← pkt 持有数据引用（引用计数 = 1）              │
│      ↓                                                   │
│  使用 pkt.data / pkt.size / pkt.pts                      │
│      │  ← 可以送给解码器（avcodec_send_packet）          │
│      ↓                                                   │
│  av_packet_unref(&pkt)                                   │
│      │  ← 释放引用（引用计数 - 1）                      │
│      │     如果计数归零，底层内存自动释放                 │
│      ↓                                                   │
│  下一轮 av_read_frame()                                  │
│                                                          │
│  ⚠️ 如果需要保存包的数据副本（例如放入缓冲队列）：       │
│  ● av_packet_clone(&pkt)   → 创建新的 AVPacket 副本     │
│  ● av_packet_ref(dst, src) → 让 dst 也引用同一块数据    │
│    └── 引用计数 + 1，数据不会因为 src 的 unref 而丢失   │
└─────────────────────────────────────────────────────────┘
```

> **常见内存泄漏**：忘记调用 `av_packet_unref()`。每次 `av_read_frame()` 成功后，都**必须**配对一个 `av_packet_unref()`——否则每读一个包就泄漏一段内存，长时间播放后内存耗尽。
>
> KrKr2 中 `CDVDDemuxFFmpeg::Read()` 读取包后会将所有权转移给 `CDVDDemuxerPacket`（一个包装类，添加了与 KrKr2 播放器匹配的接口），由消息队列负责最终释放。

## 3.4 四个结构体的关系

理解了单个结构体后，把它们的关系串起来：

```
AVFormatContext（一个打开的媒体文件）
│
├── iformat         → 格式描述（如 "mov,mp4,m4a,3gp"）
├── pb              → AVIOContext（I/O 层，可自定义）
├── nb_streams = 3  → 这个文件有 3 条流
│
├── streams[0]: AVStream（视频流, index=0）
│   ├── codecpar: AVCodecParameters
│   │   ├── codec_type = AVMEDIA_TYPE_VIDEO
│   │   ├── codec_id = AV_CODEC_ID_H264
│   │   ├── width = 1920, height = 1080
│   │   └── format = AV_PIX_FMT_YUV420P
│   ├── time_base = {1, 90000}
│   └── avg_frame_rate = {30000, 1001}  → ≈29.97 fps
│
├── streams[1]: AVStream（音频流, index=1）
│   ├── codecpar: AVCodecParameters
│   │   ├── codec_type = AVMEDIA_TYPE_AUDIO
│   │   ├── codec_id = AV_CODEC_ID_AAC
│   │   ├── sample_rate = 44100
│   │   └── channels = 2
│   └── time_base = {1, 44100}
│
├── streams[2]: AVStream（字幕流, index=2）
│   └── codecpar: AVCodecParameters
│       └── codec_type = AVMEDIA_TYPE_SUBTITLE
│
└── av_read_frame() → 每次返回一个 AVPacket
    ├── stream_index = 0 → 这个包是视频数据
    ├── stream_index = 1 → 这个包是音频数据
    └── stream_index = 2 → 这个包是字幕数据
```

**用一句话概括**：`AVFormatContext` 持有多个 `AVStream`，每个 `AVStream` 持有一个 `AVCodecParameters` 描述编码属性。调用 `av_read_frame()` 从 `AVFormatContext` 读出 `AVPacket`，通过 `AVPacket::stream_index` 判断它属于哪条 `AVStream`。

## 3.5 动手实践：打印数据结构关系

以下完整程序演示如何遍历一个媒体文件的所有流和数据结构：

```c
// print_structures.c —— 打印 AVFormatContext/AVStream/AVCodecParameters 关系
#include <libavformat/avformat.h>
#include <libavcodec/avcodec.h>
#include <libavutil/avutil.h>
#include <stdio.h>
#include <inttypes.h>

int main(int argc, char *argv[]) {
    if (argc < 2) {
        fprintf(stderr, "用法: %s <媒体文件路径>\n", argv[0]);
        return 1;
    }
    const char *filename = argv[1];

    // 打开文件 → 获得 AVFormatContext
    AVFormatContext *fmt_ctx = NULL;
    int ret = avformat_open_input(&fmt_ctx, filename, NULL, NULL);
    if (ret < 0) {
        char errbuf[256];
        av_strerror(ret, errbuf, sizeof(errbuf));  // 错误码转可读字符串
        fprintf(stderr, "无法打开文件: %s\n", errbuf);
        return 1;
    }

    // 探测流信息 → 填充 streams[] 和 codecpar
    ret = avformat_find_stream_info(fmt_ctx, NULL);
    if (ret < 0) {
        fprintf(stderr, "无法获取流信息\n");
        avformat_close_input(&fmt_ctx);
        return 1;
    }

    // ── 打印 AVFormatContext 信息 ──
    printf("=== AVFormatContext ===\n");
    printf("格式名称: %s\n", fmt_ctx->iformat->name);
    printf("格式全称: %s\n", fmt_ctx->iformat->long_name);
    printf("流数量  : %u\n", fmt_ctx->nb_streams);
    printf("总时长  : %.2f 秒\n", fmt_ctx->duration / (double)AV_TIME_BASE);
    printf("总码率  : %"PRId64" kbps\n", fmt_ctx->bit_rate / 1000);
    printf("\n");

    // ── 遍历每条流，打印 AVStream 和 AVCodecParameters ──
    for (unsigned int i = 0; i < fmt_ctx->nb_streams; i++) {
        AVStream *stream = fmt_ctx->streams[i];
        AVCodecParameters *par = stream->codecpar;

        printf("--- AVStream #%u ---\n", i);
        printf("  index        : %d\n", stream->index);
        printf("  time_base    : %d/%d\n", stream->time_base.num, stream->time_base.den);
        printf("  duration     : %"PRId64" (≈%.2f 秒)\n",
               stream->duration,
               stream->duration > 0 ? stream->duration * av_q2d(stream->time_base) : 0.0);
        printf("  nb_frames    : %"PRId64"\n", stream->nb_frames);

        printf("  --- AVCodecParameters ---\n");
        printf("    codec_type : ");
        switch (par->codec_type) {
        case AVMEDIA_TYPE_VIDEO:    printf("VIDEO\n"); break;
        case AVMEDIA_TYPE_AUDIO:    printf("AUDIO\n"); break;
        case AVMEDIA_TYPE_SUBTITLE: printf("SUBTITLE\n"); break;
        default:                    printf("OTHER (%d)\n", par->codec_type); break;
        }
        printf("    codec_id   : %s (id=%d)\n", avcodec_get_name(par->codec_id), par->codec_id);

        if (par->codec_type == AVMEDIA_TYPE_VIDEO) {
            printf("    width×height: %d×%d\n", par->width, par->height);
            printf("    pixel_format: %s\n", av_get_pix_fmt_name(par->format));
            printf("    frame_rate  : %.2f fps\n", av_q2d(stream->avg_frame_rate));
        } else if (par->codec_type == AVMEDIA_TYPE_AUDIO) {
            printf("    sample_rate : %d Hz\n", par->sample_rate);
            printf("    channels    : %d\n", par->channels);
            printf("    sample_fmt  : %s\n", av_get_sample_fmt_name(par->format));
        }

        if (par->extradata_size > 0) {
            printf("    extradata   : %d 字节\n", par->extradata_size);
        }
        printf("\n");
    }

    // ── 读取前 10 个 AVPacket，展示 stream_index 分发 ──
    printf("=== 前 10 个 AVPacket ===\n");
    AVPacket pkt;
    int count = 0;
    while (av_read_frame(fmt_ctx, &pkt) >= 0 && count < 10) {
        AVStream *s = fmt_ctx->streams[pkt.stream_index];
        const char *type = "???";
        switch (s->codecpar->codec_type) {
        case AVMEDIA_TYPE_VIDEO:    type = "视频"; break;
        case AVMEDIA_TYPE_AUDIO:    type = "音频"; break;
        case AVMEDIA_TYPE_SUBTITLE: type = "字幕"; break;
        default: break;
        }
        printf("  包 #%02d: stream=%d [%s], size=%5d, pts=%8"PRId64", dts=%8"PRId64", key=%s\n",
               count, pkt.stream_index, type, pkt.size,
               pkt.pts, pkt.dts,
               (pkt.flags & AV_PKT_FLAG_KEY) ? "Y" : "N");
        av_packet_unref(&pkt);  // 必须释放！
        count++;
    }

    avformat_close_input(&fmt_ctx);  // 关闭并释放所有资源
    printf("\n清理完成。\n");
    return 0;
}
```

## 常见错误及解决方案

### 错误 1：`avformat_open_input` 返回负数但不知道含义

**现象**：调用 `avformat_open_input()` 失败，返回 `-2`（或其他负数），不知道什么意思

**原因**：FFmpeg 使用负数错误码，基于 POSIX errno 取反（`-ENOENT = -2` 表示文件不存在）

**解决**：始终用 `av_strerror()` 将错误码转为可读字符串：

```c
int ret = avformat_open_input(&fmt_ctx, filename, NULL, NULL);
if (ret < 0) {
    char errbuf[AV_ERROR_MAX_STRING_SIZE];     // 推荐用 AV_ERROR_MAX_STRING_SIZE（64 字节）
    av_strerror(ret, errbuf, sizeof(errbuf));  // 转为 "No such file or directory" 等
    fprintf(stderr, "打开失败 (错误码 %d): %s\n", ret, errbuf);
}
```

### 错误 2：访问 `codecpar->width` 得到 0

**现象**：打开文件后直接读取 `streams[0]->codecpar->width`，得到 0

**原因**：没有调用 `avformat_find_stream_info()`。`avformat_open_input()` 只读取容器头部的基本信息，部分容器（如 MPEG-TS）不在头部存储分辨率等详细参数

**解决**：打开文件后必须调用 `avformat_find_stream_info()` 再访问 `codecpar`：

```c
avformat_open_input(&fmt_ctx, filename, NULL, NULL);
avformat_find_stream_info(fmt_ctx, NULL);  // 先探测！
// 现在 codecpar->width, height, sample_rate 等字段已正确填充
printf("分辨率: %dx%d\n", fmt_ctx->streams[0]->codecpar->width,
                           fmt_ctx->streams[0]->codecpar->height);
```

### 错误 3：循环读包后忘记 `av_packet_unref` 导致内存持续增长

**现象**：程序运行一段时间后内存占用越来越高，最终 OOM（Out Of Memory）崩溃

**原因**：`av_read_frame()` 每次成功都会分配（或引用）一块数据缓冲区。如果不调用 `av_packet_unref()` 释放引用，这些缓冲区就永远不会被回收

**解决**：在每次循环末尾或不再需要包数据时，必须调用 `av_packet_unref()`：

```c
AVPacket pkt;
while (av_read_frame(fmt_ctx, &pkt) >= 0) {
    // 处理 pkt ...
    av_packet_unref(&pkt);  // ← 必须！每次循环都要调用
}
```

## 本节小结

- **解封装**（Demuxing）是从容器文件中分离提取各流的压缩数据包的过程
- **容器格式**（MP4/MKV/AVI 等）和**编码格式**（H.264/AAC 等）是两个独立维度，互不依赖
- FFmpeg 解封装围绕四个核心结构体：
  - `AVFormatContext`：整个文件的"总管"，持有所有流信息
  - `AVStream`：文件中的一路数据流，持有时间基准和编码参数
  - `AVCodecParameters`：一条流的编解码参数（编码类型、分辨率、采样率等）
  - `AVPacket`：解封装输出的基本单元，包含一段压缩数据及其时间戳
- `AVPacket` 使用引用计数管理内存，`av_read_frame()` 与 `av_packet_unref()` 必须配对
- PTS（显示时间戳）和 DTS（解码时间戳）在有 B 帧时不同，无 B 帧时相等

## 练习题与答案

### 题目 1：容器格式和编码格式的区别是什么？为什么说它们是"正交"的？

<details>
<summary>查看答案</summary>

**容器格式**决定"数据怎么组织在一个文件里"——负责将多条流（视频、音频、字幕）打包到一起，附加时间戳和索引表。常见的有 MP4、MKV、AVI。

**编码格式**决定"原始数据怎么压缩成更小的体积"——负责将原始像素/采样数据通过数学算法压缩，以及将压缩数据还原。常见的有 H.264、AAC、VP9。

说它们"正交"是因为它们可以自由组合：
- H.264 视频 + AAC 音频可以放在 MP4 里，也可以放在 MKV 里
- 同一个 MKV 容器可以包含 H.264 视频，也可以包含 VP9 视频
- 容器不关心流的具体编码，编码不关心最终存在哪种容器里

在代码中的体现：`AVFormatContext::iformat` 描述容器格式，`AVCodecParameters::codec_id` 描述编码格式，两者独立。

</details>

### 题目 2：写一段代码，打开一个媒体文件并找出码率最高的流，打印其信息

<details>
<summary>查看答案</summary>

```c
#include <libavformat/avformat.h>
#include <libavcodec/avcodec.h>
#include <stdio.h>
#include <inttypes.h>

int main(int argc, char *argv[]) {
    if (argc < 2) {
        fprintf(stderr, "用法: %s <媒体文件>\n", argv[0]);
        return 1;
    }

    AVFormatContext *fmt_ctx = NULL;
    if (avformat_open_input(&fmt_ctx, argv[1], NULL, NULL) < 0) {
        fprintf(stderr, "无法打开文件\n");
        return 1;
    }
    avformat_find_stream_info(fmt_ctx, NULL);

    // 找码率最高的流
    int best_idx = -1;
    int64_t max_bitrate = 0;

    for (unsigned int i = 0; i < fmt_ctx->nb_streams; i++) {
        int64_t br = fmt_ctx->streams[i]->codecpar->bit_rate;
        if (br > max_bitrate) {
            max_bitrate = br;
            best_idx = (int)i;
        }
    }

    if (best_idx >= 0) {
        AVStream *s = fmt_ctx->streams[best_idx];
        AVCodecParameters *par = s->codecpar;
        const char *type_name = "未知";
        if (par->codec_type == AVMEDIA_TYPE_VIDEO) type_name = "视频";
        if (par->codec_type == AVMEDIA_TYPE_AUDIO) type_name = "音频";

        printf("码率最高的流: #%d [%s]\n", best_idx, type_name);
        printf("  编码: %s\n", avcodec_get_name(par->codec_id));
        printf("  码率: %"PRId64" kbps\n", max_bitrate / 1000);
        printf("  time_base: %d/%d\n", s->time_base.num, s->time_base.den);

        if (par->codec_type == AVMEDIA_TYPE_VIDEO) {
            printf("  分辨率: %dx%d\n", par->width, par->height);
        } else if (par->codec_type == AVMEDIA_TYPE_AUDIO) {
            printf("  采样率: %d Hz, 声道: %d\n", par->sample_rate, par->channels);
        }
    } else {
        printf("没有找到有码率信息的流\n");
    }

    avformat_close_input(&fmt_ctx);
    return 0;
}
```

</details>

### 题目 3：解释 AVPacket 的 PTS 和 DTS 在什么情况下会不同？KrKr2 的游戏视频通常会遇到这种情况吗？

<details>
<summary>查看答案</summary>

**PTS（Presentation Timestamp）** 和 **DTS（Decoding Timestamp）** 在视频流使用 **B 帧**（Bidirectional Frame，双向预测帧）时会不同。

B 帧的解码需要参考它前面和后面的 I 帧或 P 帧。因此，解码顺序（DTS）和显示顺序（PTS）不一致：

```
显示顺序：I(0) B(1) B(2) P(3) B(4) B(5) P(6)
解码顺序：I(0) P(3) B(1) B(2) P(6) B(4) B(5)
```

在这种情况下，P(3) 的 DTS < PTS（DTS 排在前面让它先解码，但 PTS 排在后面让它后显示）。

**KrKr2 的游戏视频通常不会遇到**。原因：
1. KiriKiri 系列游戏的 OP/ED 视频通常使用较简单的编码配置
2. 很多游戏视频使用不含 B 帧的 Baseline Profile H.264 或 MPEG-4 Simple Profile
3. 对于没有 B 帧的视频，PTS 始终等于 DTS

不过 KrKr2 的播放器代码（`CDVDDemuxFFmpeg`）仍然正确处理了 PTS ≠ DTS 的情况，以保证对所有视频格式的兼容性。

</details>
