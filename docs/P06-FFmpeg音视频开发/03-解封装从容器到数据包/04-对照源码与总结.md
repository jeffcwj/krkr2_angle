## CDVDDemuxFFmpeg 类总览

KrKr2 的解封装逻辑完全封装在 `CDVDDemuxFFmpeg` 类中。这个类继承自 `IDemux` 接口（IDemux 是 KrKr2 定义的解封装器抽象基类，规定了 `Read()`、`SeekTime()` 等纯虚方法），位于：

```
cpp/core/movie/ffmpeg/DemuxFFmpeg.h    — 类声明（157 行）
cpp/core/movie/ffmpeg/DemuxFFmpeg.cpp  — 类实现（1656 行）
```

> **架构背景：** KrKr2 的媒体播放器架构移植自 XBMC/Kodi（一个知名的开源媒体播放器）。
> 如果你在代码中看到 `DVD`、`CDVDxxx` 等前缀，不要误以为是光盘相关——这只是 Kodi 的历史命名习惯
>（Kodi 最初是 Xbox Media Center，DVD 播放是其核心功能之一）。

### 类继承关系

```
IDemux（纯虚接口：Read / SeekTime / GetStream / Flush ...）
  └── CDVDDemuxFFmpeg（FFmpeg 实现）
        ├── 持有 AVFormatContext*    — FFmpeg 的解封装上下文
        ├── 持有 AVIOContext*        — 自定义 I/O 上下文
        ├── 持有 InputStream*        — KrKr2 的输入流抽象
        └── 管理 CDemuxStream* map   — 流信息映射表
```

### 完整公开接口

下面是 `DemuxFFmpeg.h` 第 50-86 行的公开接口，每个方法都标注了对应的 FFmpeg API：

```cpp
// 文件: cpp/core/movie/ffmpeg/DemuxFFmpeg.h 第 50-86 行
class CDVDDemuxFFmpeg : public IDemux {
public:
    CDVDDemuxFFmpeg();
    ~CDVDDemuxFFmpeg() override;

    // 打开媒体源：内部调用 avformat_open_input + avformat_find_stream_info
    // streaminfo=true 时会探测流信息（默认行为）
    // fileinfo=true 时只读文件头不探测流（用于快速元数据读取）
    bool Open(InputStream *pInput, bool streaminfo = true,
              bool fileinfo = false);

    // 释放所有资源：内部调用 avformat_close_input + av_free
    void Dispose();

    // 重置解封装器：Dispose() 后重新 Open()，用于格式变更
    void Reset() override;

    // 刷新缓冲区：内部调用 avio_flush + avformat_flush
    void Flush() override;

    // 中止读取：设置超时标记，interrupt_cb 回调会返回 1
    void Abort() override;

    // 设置播放速度：高倍速时丢弃非关键帧（AVDISCARD_NONKEY）
    void SetSpeed(int iSpeed) override;

    // 读取下一个包：内部调用 av_read_frame，返回 DemuxPacket*
    DemuxPacket *Read() override;

    // 按时间跳转（毫秒）：内部调用 av_seek_frame
    bool SeekTime(int time, bool backwords = false,
                  double *startpts = nullptr) override;

    // 按字节位置跳转
    bool SeekByte(int64_t pos);

    // 获取总时长（毫秒）：读取 m_pFormatContext->duration
    int GetStreamLength() override;

    // 获取指定流的信息
    CDemuxStream *GetStream(int iStreamId) const override;

    // 获取所有流
    std::vector<CDemuxStream *> GetStreams() const override;

    // 获取流数量
    int GetNrOfStreams() const override;

    // 获取流的编解码器名称
    std::string GetStreamCodecName(int iStreamId) override;

    // 检查是否已中止
    bool Aborted();

    // 公开成员——供自定义 I/O 回调访问
    AVFormatContext *m_pFormatContext;  // FFmpeg 格式上下文
    InputStream *m_pInput = nullptr;   // 输入流指针
};
```

> **设计观察：** `m_pFormatContext` 和 `m_pInput` 是 `public` 的，这在 C++ 中并不常见。
> 原因是自定义 I/O 的 C 回调函数（`dvd_file_read`、`dvd_file_seek`）需要通过 `void*`
> 指针访问这些成员——C 回调无法调用 C++ 的 private getter。

## 自定义 I/O 架构——InputStream 适配层

前面章节我们学过，`avformat_open_input()` 默认通过文件路径打开媒体。但 KrKr2 的媒体数据可能来自多种来源——XP3 归档包、内存缓冲区、网络流等。这些数据源无法用简单的文件路径表示，因此 KrKr2 使用了**自定义 I/O**（Custom I/O）架构。

### 回调函数实现

KrKr2 在 `DemuxFFmpeg.cpp` 第 53-78 行定义了三个 C 函数，作为 `AVIOContext` 的回调：

```c
// 文件: cpp/core/movie/ffmpeg/DemuxFFmpeg.cpp 第 53-78 行

// 中断回调：检查是否应取消读取
// 返回 1 表示取消，0 表示继续
static int interrupt_cb(void *ctx) {
    auto *demuxer = static_cast<CDVDDemuxFFmpeg *>(ctx);
    if (demuxer && demuxer->Aborted())
        return 1;   // 超时或用户取消 → 中断 FFmpeg 阻塞操作
    return 0;
}

// 读取回调：从 KrKr2 的 InputStream 读取数据
// FFmpeg 每次需要数据时调用此函数，而不是直接读文件
static int dvd_file_read(void *h, uint8_t *buf, int size) {
    if (interrupt_cb(h))       // 先检查是否应中断
        return AVERROR_EXIT;   // AVERROR_EXIT 告诉 FFmpeg 立即停止

    // 将 void* 还原为 CDVDDemuxFFmpeg*，再取出 InputStream*
    InputStream *pInputStream =
        static_cast<CDVDDemuxFFmpeg *>(h)->m_pInput;
    return pInputStream->Read(buf, size);  // 委托给 KrKr2 的流接口
}

// 跳转回调：在数据源中定位
static int64_t dvd_file_seek(void *h, int64_t pos, int whence) {
    if (interrupt_cb(h))
        return AVERROR_EXIT;

    InputStream *pInputStream =
        static_cast<CDVDDemuxFFmpeg *>(h)->m_pInput;

    if (whence == AVSEEK_SIZE) {
        // AVSEEK_SIZE 是 FFmpeg 特有的 whence 值
        // 含义：不跳转，只返回数据源总长度
        return pInputStream->GetLength();
    } else {
        // 普通 seek：SEEK_SET / SEEK_CUR / SEEK_END
        // AVSEEK_FORCE 标志需要去掉（KrKr2 的 Seek 不认识它）
        return pInputStream->Seek(pos, whence & ~AVSEEK_FORCE);
    }
}
```

### AVIOContext 的创建

在 `Open()` 方法中（第 170-173 行），KrKr2 用上述回调创建自定义 I/O 上下文：

```cpp
// 文件: cpp/core/movie/ffmpeg/DemuxFFmpeg.cpp 第 170-173 行

// 分配 32KB 的读取缓冲区
auto *buffer = (unsigned char *)av_malloc(FFMPEG_FILE_BUFFER_SIZE);

// 创建自定义 I/O 上下文
m_ioContext = avio_alloc_context(
    buffer,                    // 缓冲区指针
    FFMPEG_FILE_BUFFER_SIZE,   // 缓冲区大小（32768 字节）
    0,                         // write_flag=0，只读模式
    this,                      // opaque：传入 CDVDDemuxFFmpeg* 自身
    dvd_file_read,             // 读取回调
    nullptr,                   // 写入回调（不需要）
    dvd_file_seek              // 跳转回调
);
```

> **关键点：** `avio_alloc_context()` 的第 4 个参数 `opaque` 传入了 `this`
>（即 `CDVDDemuxFFmpeg*`），这就是为什么回调函数能通过 `static_cast<CDVDDemuxFFmpeg*>(h)`
> 访问到解封装器实例和它的 `m_pInput` 成员。

## Read() 方法详解——线程安全的逐包读取

`CDVDDemuxFFmpeg::Read()` 方法（第 726-940 行）是整个解封装器的核心循环。与我们简单的 `while (av_read_frame() >= 0)` 不同，KrKr2 的实现考虑了多线程环境和异常处理：

```cpp
// 文件: DemuxFFmpeg.cpp 第 726-756 行（简化展示）
DemuxPacket *CDVDDemuxFFmpeg::Read() {
    DemuxPacket *pPacket = nullptr;
    bool bReturnEmpty = false;

    {
        // 关键：加锁保护，防止其他线程同时操作 m_pFormatContext
        CSingleLock lock(m_critSection);

        if (m_pFormatContext) {
            // 重置 EOF 标记——让 FFmpeg 继续尝试读取
            if (m_pFormatContext->pb)
                m_pFormatContext->pb->eof_reached = 0;

            // 检查是否有上次缓存的包（用于处理节目变更）
            if (m_pkt.result < 0) {
                m_pkt.pkt.size = 0;
                m_pkt.pkt.data = nullptr;

                // 设置 20 秒读取超时
                m_timeout.Set(20000);
                // 核心调用——与我们的 av_read_frame 完全相同
                m_pkt.result = av_read_frame(m_pFormatContext, &m_pkt.pkt);
                m_timeout.SetInfinite();
            }

            // 处理返回值
            if (m_pkt.result == AVERROR(EINTR) ||
                m_pkt.result == AVERROR(EAGAIN)) {
                // 超时或暂时不可用 → 返回空包，主循环会重试
                bReturnEmpty = true;
            } else if (m_pkt.result < 0) {
                // 真正的错误 → 刷新缓冲区
                Flush();
            }
            // ... 后续处理：包大小校验、流匹配、DemuxPacket 封装
        }
    }
    // ... 返回 pPacket
}
```

**对比要点：**

| 特性 | 我们的教程代码 | KrKr2 的 Read() |
|------|---------------|-----------------|
| 线程安全 | 无（单线程） | `CSingleLock` 互斥锁 |
| 超时控制 | 无 | 20 秒读取超时 + `interrupt_cb` |
| EAGAIN 处理 | 简单重试 | 返回空包，由上层主循环重试 |
| 包缓存 | 无 | `m_pkt` 结构缓存上一次的包 |
| 返回类型 | 直接使用 `AVPacket` | 封装为 `DemuxPacket*`（引用计数） |

## SeekTime() 方法详解——毫秒到时间戳的转换

`CDVDDemuxFFmpeg::SeekTime()` 方法（第 988-1070 行）封装了 `av_seek_frame()`：

```cpp
// 文件: DemuxFFmpeg.cpp 第 988-1052 行（简化展示）
bool CDVDDemuxFFmpeg::SeekTime(int time, bool backwords, double *startpts) {
    if (!m_pInput) return false;

    // 负时间 → 跳到开头
    if (time < 0) { time = 0; }

    // 清空缓存的包
    m_pkt.result = -1;
    av_packet_unref(&m_pkt.pkt);

    // 关键转换：毫秒 → AV_TIME_BASE 时间戳
    // AV_TIME_BASE = 1000000（微秒）
    // time 单位是毫秒，所以乘以 (1000000 / 1000) = 1000
    int64_t seek_pts = (int64_t)time * (AV_TIME_BASE / 1000);

    int ret;
    {
        CSingleLock lock(m_critSection);  // 加锁

        // 核心调用——与我们的 av_seek_frame 相同
        // stream_index = -1 表示使用默认流
        // backwords 参数控制是否向后搜索关键帧
        ret = av_seek_frame(m_pFormatContext, -1, seek_pts,
                            backwords ? AVSEEK_FLAG_BACKWARD : 0);

        if (ret >= 0)
            UpdateCurrentPTS();  // 更新内部时间戳记录
    }

    // 设置输出参数：Seek 后的起始 PTS
    if (startpts)
        *startpts = DVD_MSEC_TO_TIME(time);

    return (ret >= 0);
}
```

**关键观察：**
1. KrKr2 的 `SeekTime()` 接受**毫秒**参数（`int time`），而 `av_seek_frame()` 需要
   `AV_TIME_BASE` 单位的时间戳——所以乘以 `AV_TIME_BASE / 1000`
2. Seek 后没有调用 `avcodec_flush_buffers()`——因为解码器刷新在更上层的
   `VideoPlayer` 中处理（解封装器不负责解码器状态）

## Dispose() 方法详解——安全的资源释放

`CDVDDemuxFFmpeg::Dispose()` 方法（第 448-480 行）展示了正确的资源释放顺序：

```cpp
// 文件: DemuxFFmpeg.cpp 第 448-480 行
void CDVDDemuxFFmpeg::Dispose() {
    // 1. 清空包缓存
    m_pkt.result = -1;
    av_packet_unref(&m_pkt.pkt);

    if (m_pFormatContext) {
        // 2. 关闭所有流的解码器上下文
        for (unsigned int i = 0; i < m_pFormatContext->nb_streams; i++) {
            avcodec_close(m_pFormatContext->streams[i]->codec);
        }

        // 3. 检查 I/O 上下文是否被 FFmpeg 偷偷替换
        // （某些解封装器会修改 pb 指针，导致内存泄漏）
        if (m_ioContext && m_pFormatContext->pb &&
            m_pFormatContext->pb != m_ioContext) {
            m_ioContext = m_pFormatContext->pb;  // 使用新指针
        }

        // 4. 关闭格式上下文——对应 avformat_open_input
        avformat_close_input(&m_pFormatContext);
    }

    // 5. 释放自定义 I/O 的缓冲区和上下文
    if (m_ioContext) {
        av_free(m_ioContext->buffer);  // 释放 32KB 缓冲区
        av_free(m_ioContext);          // 释放 AVIOContext 本身
    }

    m_ioContext = nullptr;
    m_pFormatContext = nullptr;

    // 6. 释放流映射表
    DisposeStreams();

    // 7. 释放输入流（引用计数 -1）
    SAFE_RELEASE(m_pInput);
}
```

**对比我们的代码：** 我们只需 `avformat_close_input(&fmt_ctx)` 一行。KrKr2 的 Dispose()
还要额外处理：自定义 I/O 缓冲区释放、解码器关闭、I/O 上下文被替换的防御性检查、引用计数管理。

```cpp
class CDemuxStreamAudioFFmpeg : public CDemuxStreamAudio {
```

```cpp
// mini_demuxer.h — 简化版解封装器类

extern "C" {
#include <libavformat/avformat.h>
}

#include <string>
#include <vector>

// 简化版 DemuxPacket
struct MiniPacket {
    uint8_t *data;      // 压缩数据指针
    int size;           // 数据大小（字节）
    int stream_index;   // 流索引
    int64_t pts;        // 显示时间戳
    int64_t dts;        // 解码时间戳
    bool is_key;        // 是否关键帧

    // 从 AVPacket 构造
    static MiniPacket *FromAVPacket(const AVPacket *pkt) {
        auto *mp = new MiniPacket();
        mp->size = pkt->size;
        mp->data = new uint8_t[pkt->size];
        memcpy(mp->data, pkt->data, pkt->size);
        mp->stream_index = pkt->stream_index;
        mp->pts = pkt->pts;
        mp->dts = pkt->dts;
        mp->is_key = (pkt->flags & AV_PKT_FLAG_KEY) != 0;
        return mp;
    }

    // 释放
    static void Free(MiniPacket *pkt) {
        if (pkt) {
            delete[] pkt->data;
            delete pkt;
        }
    }
};

// 流信息
struct StreamInfo {
    int index;
    enum AVMediaType type;   // AVMEDIA_TYPE_VIDEO / AUDIO
    std::string codec_name;  // 编解码器名称
    AVRational time_base;    // 时间基
};

class MiniDemuxer {
public:
    MiniDemuxer() = default;
    ~MiniDemuxer() { Dispose(); }

    // 打开媒体文件
    bool Open(const std::string &filename) {
        // 打开输入文件
        if (avformat_open_input(&m_fmtCtx, filename.c_str(),
                                nullptr, nullptr) < 0) {
            return false;
        }
        // 探测流信息
        if (avformat_find_stream_info(m_fmtCtx, nullptr) < 0) {
            Dispose();
            return false;
        }
        // 收集流信息
        for (unsigned i = 0; i < m_fmtCtx->nb_streams; i++) {
            AVStream *st = m_fmtCtx->streams[i];
            AVCodec *codec = avcodec_find_decoder(
                st->codecpar->codec_id);
            StreamInfo info;
            info.index = i;
            info.type = st->codecpar->codec_type;
            info.codec_name = codec ? codec->name : "unknown";
            info.time_base = st->time_base;
            m_streams.push_back(info);
        }
        return true;
    }

    // 读取下一个包
    MiniPacket *Read() {
        AVPacket pkt;
        av_init_packet(&pkt);
        pkt.data = nullptr;
        pkt.size = 0;

        int ret = av_read_frame(m_fmtCtx, &pkt);
        if (ret < 0) {
            av_packet_unref(&pkt);
            return nullptr;  // EOF 或错误
        }

        MiniPacket *mp = MiniPacket::FromAVPacket(&pkt);
        av_packet_unref(&pkt);
        return mp;
    }

    // 按毫秒跳转
    bool SeekTime(int time_ms, bool backward = true) {
        int64_t seek_pts = (int64_t)time_ms * (AV_TIME_BASE / 1000);
        int flags = backward ? AVSEEK_FLAG_BACKWARD : 0;
        return av_seek_frame(m_fmtCtx, -1, seek_pts, flags) >= 0;
    }

    // 释放资源
    void Dispose() {
        if (m_fmtCtx) {
            avformat_close_input(&m_fmtCtx);
        }
        m_fmtCtx = nullptr;
        m_streams.clear();
    }

    // 获取流信息
    int GetNrOfStreams() const {
        return static_cast<int>(m_streams.size());
    }

    const StreamInfo &GetStream(int idx) const {
        return m_streams[idx];
    }

    // 获取总时长（毫秒）
    int GetDurationMs() const {
        if (!m_fmtCtx || m_fmtCtx->duration == AV_NOPTS_VALUE)
            return 0;
        return static_cast<int>(m_fmtCtx->duration / (AV_TIME_BASE / 1000));
    }

private:
    AVFormatContext *m_fmtCtx = nullptr;
    std::vector<StreamInfo> m_streams;
};
```

## 常见错误及解决方案

### 错误 1：av_read_frame 返回 AVERROR(EAGAIN)

**现象：** 循环中 `av_read_frame()` 返回 `-11`（EAGAIN），程序看起来卡住了。

**原因：** 非阻塞 I/O 模式（`AVFMT_FLAG_NONBLOCK`，KrKr2 在 DemuxFFmpeg.cpp 第 401 行设置了此标志）下，数据尚未就绪。常见于网络流或自定义 I/O 源。

**KrKr2 的处理方式：** 返回一个空包（0 字节），让上层的 `VideoPlayer` 主循环稍后重试。

**我们的处理方式：**

```c
while (1) {
    ret = av_read_frame(fmt_ctx, &pkt);

    if (ret == AVERROR(EAGAIN)) {
        // 数据暂时不可用——等待 10ms 后重试
        // EAGAIN（Error AGAIN）表示"再试一次"
        SDL_Delay(10);  // 或 usleep(10000)
        continue;
    }
    if (ret == AVERROR_EOF || ret < 0) {
        break;  // 真正的 EOF 或不可恢复错误
    }

    // 正常处理 pkt ...
    process_packet(&pkt);
    av_packet_unref(&pkt);
}
```

> **区分：** EAGAIN 不是错误，是"暂时没数据，请稍后再来"。AVERROR_EOF 才是文件结束。

### 错误 2：avformat_find_stream_info 超时

**现象：** 调用 `avformat_find_stream_info()` 阻塞数十秒甚至更久。

**原因：** 网络流不稳定、文件损坏或容器格式复杂（如 MPEG-TS）导致探测时间过长。

**KrKr2 的处理方式：** 在 Open() 中设置 30 秒全局超时（`m_timeout.Set(30000)`），配合 `interrupt_cb` 在超时后中断 FFmpeg 操作。对 MJPEG 和 MPEG-TS 还额外限制了分析时长为 500ms。

**我们的处理方式：**

```c
AVDictionary *opts = NULL;

// 限制探测时长为 5 秒（单位：微秒）
av_dict_set(&opts, "analyzeduration", "5000000", 0);

// 限制探测数据量为 5MB（单位：字节）
av_dict_set(&opts, "probesize", "5000000", 0);

int ret = avformat_find_stream_info(fmt_ctx, &opts);
av_dict_free(&opts);

if (ret < 0) {
    // 即使返回错误，nb_streams 可能已有部分结果
    // 某些流的 codec 参数可能不完整，但基本可用
    fprintf(stderr, "警告: 流探测未完全完成 (ret=%d)\n", ret);

    if (fmt_ctx->nb_streams == 0) {
        // 一个流都没找到 → 文件可能真的有问题
        fprintf(stderr, "错误: 未找到任何流\n");
        avformat_close_input(&fmt_ctx);
        return -1;
    }
    // 有部分流信息 → 可以继续尝试
}
```

### 错误 3：Seek 后画面花屏或绿屏

**现象：** `av_seek_frame()` 之后解码出的前几帧有马赛克、绿色块或花屏。

**原因：** Seek 只能跳到关键帧（I 帧，也叫 IDR 帧——Instantaneous Decoder Refresh），
但解码器内部缓冲区中可能还保留着 Seek 之前的参考帧。P 帧和 B 帧
（P 帧依赖前一帧做预测，B 帧依赖前后两帧做双向预测）用旧的参考帧解码就会出错。

**解决方案：**

```c
// Seek 操作
av_seek_frame(fmt_ctx, -1, target_ts, AVSEEK_FLAG_BACKWARD);

// 关键：Seek 后必须刷新所有解码器缓冲区
// 丢弃解码器中缓存的旧参考帧
avcodec_flush_buffers(video_codec_ctx);
avcodec_flush_buffers(audio_codec_ctx);

// 然后从新位置开始解码
// 注意：Seek 后的第一帧一定是关键帧
// 但显示出来可能不是 Seek 目标时间的帧——精确 Seek 需要丢弃若干帧
```

> **KrKr2 的处理：** `SeekTime()` 本身不刷新解码器（因为解封装器不持有解码器上下文）。
> 解码器刷新在上层的 `VideoPlayerVideo` 和 `VideoPlayerAudio` 中完成。

## 本节小结

本章（第 3 章）完整学习了 FFmpeg 解封装流程。回顾全部知识点：

### 核心概念

| 概念 | 说明 |
|------|------|
| 解封装（Demuxing） | 从容器文件中提取各个流的压缩数据包 |
| AVFormatContext | FFmpeg 的解封装上下文，持有所有文件和流的元信息 |
| AVStream | 单个流的信息（编码参数、时间基、帧率等） |
| AVPacket | 一个压缩数据包（一帧视频或一段音频的压缩数据） |
| AVIOContext | 自定义 I/O 上下文，让 FFmpeg 从任意数据源读取 |
| time_base | 时间基——时间戳的"刻度单位"，如 {1, 90000} 表示 1/90000 秒 |

### 核心 API 速查表

| API | 功能 | 必须配对的释放 API |
|-----|------|-------------------|
| `avformat_open_input()` | 打开文件/流 | `avformat_close_input()` |
| `avformat_find_stream_info()` | 探测流信息 | 无 |
| `av_find_best_stream()` | 找到最佳音/视频流 | 无 |
| `av_read_frame()` | 读取下一个包 | `av_packet_unref()` |
| `av_seek_frame()` | 按时间/字节跳转 | 无 |
| `avio_alloc_context()` | 创建自定义 I/O | `av_free()` |
| `av_rescale_q()` | 时间戳转换 | 无 |

### KrKr2 对照要点

1. `CDVDDemuxFFmpeg` 类封装了本章所有 FFmpeg API
2. 自定义 I/O 通过 `InputStream` 适配层实现，支持 XP3 归档等多种数据源
3. 线程安全通过 `CSingleLock` 互斥锁保证
4. 超时控制通过 `interrupt_cb` 中断回调实现
5. `DemuxPacket` 是 `AVPacket` 的 C++ 封装，添加了引用计数

## 练习题与答案

### 题目 1：解封装和解码有什么区别？为什么需要分成两步？

<details>
<summary>查看答案</summary>

**解封装（Demuxing）：**
- 从容器文件（MP4/MKV/AVI）中提取各个流的压缩数据包（AVPacket）
- 不涉及数据的解压缩——输出仍然是压缩数据（H.264 NALU、AAC 帧等）
- 使用 `libavformat` 库
- 类比：打开一个快递包裹，把里面的东西分开（但不拆开每样东西的塑封）

**解码（Decoding）：**
- 将压缩数据包还原为原始像素/采样数据（AVFrame）
- 涉及复杂的数学运算（DCT 逆变换、运动补偿、熵解码等）
- 使用 `libavcodec` 库
- 类比：把每样东西的塑封拆开，看到实际内容

**为什么分开？**

1. **解耦合**：同一种编码可以放在不同容器中——H.264 可以在 MP4、MKV、TS 中。
   如果把解封装和解码写在一起，M 种容器 × N 种编码需要 M×N 份代码。分开后只需 M + N 份。
2. **流水线并行**：解封装在前，解码在后，可以用不同线程并行处理（KrKr2 就是这么做的）
3. **灵活性**：可以只解封装不解码（如 `ffmpeg -c copy` 转封装），或直接解码裸流（H.264 Annex B 格式）
4. **KrKr2 的实际体现**：`CDVDDemuxFFmpeg` 只负责解封装，`VideoCodecFFmpeg` 负责解码，两个类完全独立

</details>

### 题目 2：为什么 KrKr2 使用自定义 I/O（AVIOContext）而不是直接传文件路径？请结合源码说明。

<details>
<summary>查看答案</summary>

**原因：KrKr2 的媒体数据来源多样化。**

1. **XP3 归档包**：KiriKiri 引擎的游戏资源打包在 XP3 归档中。
   文件路径是 `archive.xp3/video/opening.avi`，这不是一个真实的文件系统路径，
   FFmpeg 的标准文件 I/O 无法打开它。

2. **跨平台抽象**：Android 上的 assets 目录、内存缓冲区等数据源
   都没有标准文件路径。

3. **统一接口**：KrKr2 定义了 `InputStream` 抽象基类，提供 `Read()`、`Seek()`、
   `GetLength()` 方法。所有数据源（文件、XP3、内存）都实现这个接口。

**源码实现（DemuxFFmpeg.cpp）：**

```cpp
// 第 60-66 行：读取回调
static int dvd_file_read(void *h, uint8_t *buf, int size) {
    if (interrupt_cb(h)) return AVERROR_EXIT;
    InputStream *pInputStream =
        static_cast<CDVDDemuxFFmpeg *>(h)->m_pInput;
    return pInputStream->Read(buf, size);  // 委托给 InputStream
}

// 第 170-173 行：创建 AVIOContext
m_ioContext = avio_alloc_context(
    buffer, FFMPEG_FILE_BUFFER_SIZE, 0,
    this,            // opaque = CDVDDemuxFFmpeg*
    dvd_file_read,   // 读取回调
    nullptr,         // 不需要写入
    dvd_file_seek    // 跳转回调
);
```

这样 FFmpeg 内部每次需要读取数据时，都会通过 `dvd_file_read()` → `InputStream::Read()`
的链路，从而支持任意数据源。

</details>

### 题目 3：time_base 为 {1, 90000} 的流中，pts=450000 对应多少秒？如果要转换到 time_base 为 {1, 1000} 的流，新的 pts 是多少？请写出计算过程和 FFmpeg API 调用。

<details>
<summary>查看答案</summary>

**计算秒数：**

```
秒 = pts × time_base
   = 450000 × (1/90000)
   = 5.0 秒
```

**转换到 {1, 1000}：**

```
新 pts = 原 pts × (原 time_base / 新 time_base)
       = 450000 × (1/90000) / (1/1000)
       = 450000 × 1000 / 90000
       = 5000
```

**用 FFmpeg API：**

```c
#include <libavutil/mathematics.h>

AVRational src_tb = {1, 90000};  // 源时间基
AVRational dst_tb = {1, 1000};   // 目标时间基
int64_t src_pts = 450000;

// av_rescale_q 安全地进行时间基转换（内部避免整数溢出）
int64_t dst_pts = av_rescale_q(src_pts, src_tb, dst_tb);
// dst_pts = 5000

// 转换为秒数（浮点）
double seconds = src_pts * av_q2d(src_tb);
// seconds = 5.0
```

**验证：**
- `5000 × (1/1000) = 5.0 秒` ✅
- 两个时间基表示的物理时间相同

</details>

### 题目 4：阅读 KrKr2 的 DemuxFFmpeg.cpp，找出 Read() 方法中处理 EAGAIN 的代码，并解释为什么返回空包而不是直接重试。

<details>
<summary>查看答案</summary>

**代码位置：** DemuxFFmpeg.cpp 第 752-756 行

```cpp
if (m_pkt.result == AVERROR(EINTR) ||
    m_pkt.result == AVERROR(EAGAIN)) {
    // timeout, probably no real error, return empty packet
    bReturnEmpty = true;
}
```

**为什么返回空包而不是 `continue` 重试？**

1. **避免阻塞主循环**：`Read()` 被 `VideoPlayer` 的主线程调用。如果在 `Read()` 内部
   死循环重试，会阻塞整个播放器——用户无法暂停、调整音量、切换窗口。

2. **控制权交还上层**：返回空包后，`VideoPlayer` 的主循环可以：
   - 检查是否有用户操作（暂停、停止）
   - 处理其他消息（窗口事件）
   - 决定是否重试读取

3. **超时控制**：KrKr2 设置了 20 秒超时（第 747 行 `m_timeout.Set(20000)`）。
   如果在 `Read()` 内部重试，无法利用这个超时机制——因为每次 `av_read_frame()`
   调用都会重置超时计时器。

4. **设计模式**：这是一个**非阻塞 I/O + 事件循环**模式。与我们教程中简单的阻塞循环
   相比，更适合 GUI 应用程序（需要保持界面响应）。

</details>
