## 硬件加速解码基础

### 什么是硬件加速解码

硬件加速解码（Hardware Accelerated Decoding）是指利用 GPU（Graphics Processing Unit，图形处理单元）或专用视频处理芯片来完成视频解码工作，而不是完全依赖 CPU。现代显卡和 SoC（System on Chip，片上系统——将 CPU、GPU、内存控制器等集成在一块芯片上的设计）都内置了专门的视频解码引擎，能够以极低的功耗完成 H.264、H.265、VP9 等编码格式的解码。

**软件解码 vs 硬件加速解码的对比：**

| 特性 | 软件解码（CPU） | 硬件加速解码（GPU/专用芯片） |
|------|----------------|---------------------------|
| 执行单元 | CPU 通用计算核心 | GPU 视频解码引擎（固定功能单元） |
| CPU 占用率 | 高（4K H.265 可能占满多核） | 极低（<5%） |
| 功耗 | 高 | 低（移动设备尤为重要） |
| 灵活性 | 支持所有格式（只要有软件解码器） | 仅支持芯片预设的格式 |
| 延迟 | 稍高（需等 CPU 调度） | 通常更低（专用管线） |
| 画质 | 标准参考质量 | 部分实现可能有微小差异 |
| 兼容性 | 跨平台一致 | 每个平台 API 不同 |

### 各平台的硬件加速 API

每个操作系统都有自己的硬件加速解码 API，FFmpeg 通过 `AVHWDeviceType` 和特殊的像素格式来支持这些 API：

| 平台 | 硬件加速 API | FFmpeg 像素格式常量 | 支持的编码格式（典型） |
|------|-------------|---------------------|---------------------|
| Windows | DXVA2（DirectX Video Acceleration 2） | `AV_PIX_FMT_DXVA2_VLD` | H.264, VC-1, MPEG-2, H.265（Win10+） |
| Windows 10+ | D3D11VA（Direct3D 11 Video Acceleration） | `AV_PIX_FMT_D3D11` | H.264, H.265, VP9, AV1 |
| Linux（Intel/AMD） | VAAPI（Video Acceleration API） | `AV_PIX_FMT_VAAPI` | H.264, H.265, VP9（取决于驱动） |
| Linux（NVIDIA） | VDPAU（Video Decode and Presentation API for Unix） | `AV_PIX_FMT_VDPAU` | H.264, MPEG-2, VC-1 |
| macOS / iOS | VideoToolbox | `AV_PIX_FMT_VIDEOTOOLBOX` | H.264, H.265, ProRes |
| Android | MediaCodec | `AV_PIX_FMT_MEDIACODEC` | H.264, H.265, VP9（取决于设备） |
| 树莓派 | MMAL（Multimedia Abstraction Layer） | 通过自定义机制拦截 | H.264 |

**工作流程概述：**

```
┌─────────────┐     ┌──────────────────────┐     ┌─────────────┐
│ 压缩数据包   │────→│ FFmpeg 解码器框架       │────→│ 解码后的帧   │
│ (AVPacket)  │     │                      │     │ (AVFrame)   │
└─────────────┘     │  ┌─ CPU 软件解码 ──┐  │     └─────────────┘
                    │  │ libavcodec 内部  │  │
                    │  │ 纯 C 实现       │  │
                    │  └────────────────┘  │
                    │        OR            │
                    │  ┌─ 硬件加速解码 ──┐  │
                    │  │ 将数据发送给 GPU │  │
                    │  │ GPU 返回解码结果 │  │
                    │  └────────────────┘  │
                    └──────────────────────┘
```

对于应用层代码来说，无论使用软件解码还是硬件加速解码，API 调用方式几乎完全相同。差别主要在于：
1. 打开解码器时需要指定硬件设备
2. 解码后的帧数据可能存储在 GPU 显存中，需要额外步骤拷贝到系统内存

### FFmpeg 中配置硬件加速的两种方式

**方式一：通过 `avcodec_get_hw_config` 查询（推荐）**

```cpp
#include <cstdio>
extern "C" {
#include <libavcodec/avcodec.h>
#include <libavutil/hwcontext.h>
}

// 查询解码器支持的硬件加速配置
// avcodec_get_hw_config: 获取解码器的第 N 个硬件加速配置
// 参数1: AVCodec 指针（解码器）
// 参数2: 配置索引（从 0 开始）
// 返回: AVCodecHWConfig 指针，NULL 表示没有更多配置
void listHWConfigs(const AVCodec *codec) {
    printf("解码器 '%s' 支持的硬件加速:\n", codec->name);
    for (int i = 0;; i++) {
        const AVCodecHWConfig *config = avcodec_get_hw_config(codec, i);
        if (!config) break;  // 没有更多配置

        // AV_CODEC_HW_CONFIG_METHOD_HW_DEVICE_CTX 表示
        // 通过 AVHWDeviceContext 激活硬件加速
        if (config->methods & AV_CODEC_HW_CONFIG_METHOD_HW_DEVICE_CTX) {
            // av_hwdevice_get_type_name: 将设备类型枚举转为名称字符串
            // 参数: AVHWDeviceType 枚举值
            // 返回: 名称字符串（如 "cuda"、"vaapi"、"dxva2"）
            printf("  [%d] 设备类型: %s, 像素格式: %s\n",
                   i,
                   av_hwdevice_get_type_name(config->device_type),
                   av_get_pix_fmt_name(config->pix_fmt));
        }
    }
}

int main() {
    // 查询 H.264 解码器
    const AVCodec *codec = avcodec_find_decoder(AV_CODEC_ID_H264);
    if (codec) {
        listHWConfigs(codec);
    }

    // 查询 H.265/HEVC 解码器
    codec = avcodec_find_decoder(AV_CODEC_ID_HEVC);
    if (codec) {
        listHWConfigs(codec);
    }
    return 0;
}
```

**方式二：通过 `get_format` 回调协商（KrKr2 采用的方式）**

这种方式更灵活，也是 KrKr2 项目使用的方式。解码器在需要选择像素格式时，会回调应用层提供的函数，应用层可以在回调中尝试初始化各种硬件加速方案：

```cpp
// get_format 回调的基本模式
static enum AVPixelFormat getFormatCallback(
    struct AVCodecContext *ctx,
    const enum AVPixelFormat *pixFmts)  // FFmpeg 提供的候选格式列表
{
    // 遍历候选格式
    for (const enum AVPixelFormat *p = pixFmts; *p != AV_PIX_FMT_NONE; p++) {
        // 检查是否为硬件加速格式
        if (*p == AV_PIX_FMT_DXVA2_VLD ||
            *p == AV_PIX_FMT_VAAPI ||
            *p == AV_PIX_FMT_VDPAU ||
            *p == AV_PIX_FMT_VIDEOTOOLBOX) {

            // 尝试初始化对应的硬件加速上下文
            int ret = tryInitHWAccel(ctx, *p);
            if (ret == 0) {
                printf("硬件加速初始化成功: %s\n",
                       av_get_pix_fmt_name(*p));
                return *p;  // 返回硬件格式
            }
            printf("硬件加速初始化失败: %s, 继续尝试...\n",
                   av_get_pix_fmt_name(*p));
        }
    }

    // 所有硬件方案都失败，回退到默认软件解码格式
    printf("回退到软件解码\n");
    // avcodec_default_get_format: FFmpeg 默认的格式选择函数
    // 参数同 get_format 回调
    // 返回: 从候选列表中选择第一个纯软件格式
    return avcodec_default_get_format(ctx, pixFmts);
}

// 在打开解码器前注册回调
void setupDecoder(AVCodecContext *ctx) {
    ctx->get_format = getFormatCallback;
    // ctx->opaque 可以存放自定义数据指针
    // 以便回调函数访问应用层的状态
}
```

### 硬件帧到系统内存的拷贝

当使用硬件加速解码时，解码后的帧数据通常存储在 GPU 显存中，不能直接被 CPU 访问。如果需要在 CPU 端处理这些数据（例如保存为图片、进行像素格式转换），需要将数据从显存拷贝到系统内存：

```cpp
#include <cstdio>
extern "C" {
#include <libavcodec/avcodec.h>
#include <libavutil/hwcontext.h>
}

// 将硬件帧的数据拷贝到系统内存
// hw_frame: 存储在 GPU 显存中的帧
// sw_frame: 目标帧（系统内存）
int transferHWFrame(AVFrame *hw_frame, AVFrame *sw_frame) {
    // 检查帧是否确实在硬件设备上
    if (hw_frame->format == AV_PIX_FMT_DXVA2_VLD ||
        hw_frame->format == AV_PIX_FMT_VAAPI ||
        hw_frame->format == AV_PIX_FMT_VDPAU ||
        hw_frame->format == AV_PIX_FMT_VIDEOTOOLBOX) {

        // av_hwframe_transfer_data: 将帧数据从硬件设备拷贝到系统内存
        // 参数1: 目标帧（系统内存，会自动分配缓冲区）
        // 参数2: 源帧（硬件设备上的帧）
        // 参数3: 标志位（通常传 0）
        // 返回: 0 成功，< 0 失败
        int ret = av_hwframe_transfer_data(sw_frame, hw_frame, 0);
        if (ret < 0) {
            fprintf(stderr, "av_hwframe_transfer_data 失败\n");
            return ret;
        }
        // 拷贝帧的元数据（pts、时间基等）
        sw_frame->pts = hw_frame->pts;
        sw_frame->pkt_dts = hw_frame->pkt_dts;
        printf("硬件帧已拷贝到系统内存, 格式: %s\n",
               av_get_pix_fmt_name((enum AVPixelFormat)sw_frame->format));
        return 0;
    }

    // 不是硬件帧，直接引用即可
    av_frame_ref(sw_frame, hw_frame);
    return 0;
}
```

## 多线程软件解码

### 为什么需要多线程

当硬件加速不可用时（比如旧显卡不支持某种编码格式，或者驱动有问题），FFmpeg 会回退到纯 CPU 软件解码。单线程软解在处理高分辨率视频时性能不足，因此 FFmpeg 支持两种多线程解码模式：

| 多线程模式 | 英文名 | 工作方式 | 适用场景 |
|-----------|--------|---------|---------|
| 帧级别并行 | Frame Threading | 不同线程同时解码不同帧 | H.264、H.265 等大多数编码格式 |
| 切片级别并行 | Slice Threading | 同一帧的不同切片由不同线程解码 | 开启了切片编码的 H.264 |

### 配置多线程解码

```cpp
#include <cstdio>
extern "C" {
#include <libavcodec/avcodec.h>
}

void configureThreading(AVCodecContext *ctx, const AVCodec *codec) {
    // 获取系统 CPU 核心数
    int cpuCores = 4;  // 实际应用中可用 std::thread::hardware_concurrency()

    // 方法 1：让 FFmpeg 自动决定线程数
    // 设置 thread_count = 0，FFmpeg 会根据 CPU 核心数自动选择
    ctx->thread_count = 0;  // 自动

    // 方法 2：手动指定线程数
    // KrKr2 使用 CPU 核心数的 1.5 倍，上限 16
    int num_threads = cpuCores * 3 / 2;
    num_threads = (num_threads < 1) ? 1 : (num_threads > 16) ? 16 : num_threads;
    ctx->thread_count = num_threads;

    // 设置线程类型（可选，FFmpeg 通常自动选择最合适的）
    // FF_THREAD_FRAME: 帧级别并行（推荐，性能最好）
    // FF_THREAD_SLICE: 切片级别并行（需要编码端支持）
    if (codec->capabilities & AV_CODEC_CAP_FRAME_THREADS) {
        ctx->thread_type = FF_THREAD_FRAME;
    } else if (codec->capabilities & AV_CODEC_CAP_SLICE_THREADS) {
        ctx->thread_type = FF_THREAD_SLICE;
    }

    // thread_safe_callbacks: 告诉 FFmpeg 回调函数是否线程安全
    // 1 = 线程安全（推荐设置为 1）
    // 0 = 不安全（FFmpeg 会加锁保护，降低并行度）
    ctx->thread_safe_callbacks = 1;

    printf("线程配置: %d 线程, 类型=%s\n",
           ctx->thread_count,
           (ctx->thread_type == FF_THREAD_FRAME) ? "帧级别" :
           (ctx->thread_type == FF_THREAD_SLICE) ? "切片级别" : "自动");
}
```

### 多线程解码的注意事项

使用多线程解码时需要注意以下几点：

1. **帧顺序**：帧级别并行可能导致输出帧的顺序与输入不同。需要依靠 `AVFrame::pts` 来重新排序。
2. **回调线程安全**：`get_buffer2`、`get_format` 等回调可能在工作线程中调用。如果设置 `thread_safe_callbacks = 0`，FFmpeg 会在调用回调前加锁。
3. **内存占用**：每个线程都需要独立的工作缓冲区。N 个线程 ≈ N 倍的内存占用。对于 4K 视频，每帧约 12 MB（YUV420P），16 线程需要约 192 MB。
4. **lowres 降级**：某些解码器支持在解码阶段直接降低分辨率（`ctx->lowres`），但这与多线程不兼容。

```cpp
// 示例：带完整错误处理的多线程解码器打开
int openDecoderWithThreading(AVCodecContext *ctx, const AVCodec *codec) {
    // 配置多线程
    ctx->thread_count = 4;
    ctx->thread_safe_callbacks = 1;

    // 尝试打开解码器
    int ret = avcodec_open2(ctx, codec, nullptr);
    if (ret < 0) {
        char errbuf[256];
        av_strerror(ret, errbuf, sizeof(errbuf));
        fprintf(stderr, "多线程打开失败 (%s), 尝试单线程...\n", errbuf);

        // 回退到单线程
        ctx->thread_count = 1;
        ret = avcodec_open2(ctx, codec, nullptr);
        if (ret < 0) {
            av_strerror(ret, errbuf, sizeof(errbuf));
            fprintf(stderr, "单线程也失败: %s\n", errbuf);
            return ret;
        }
        printf("已回退到单线程解码\n");
    } else {
        printf("多线程解码器已打开: %d 线程\n", ctx->thread_count);
    }
    return 0;
}
```

## KrKr2 硬件加速架构分析

### CDVDVideoCodecFFmpeg 的硬件加速协商

KrKr2 的 `CDVDVideoCodecFFmpeg`（位于 `krkr2/cpp/core/movie/ffmpeg/VideoCodecFFmpeg.cpp`，1113 行）通过 `GetFormat` 静态回调实现硬件加速协商。这个回调在 FFmpeg 需要选择像素格式时自动触发。

**完整的协商流程：**

```
avcodec_decode_video2() 内部需要选择输出像素格式
        │
        ▼
调用 ctx->get_format 回调（即 CDVDVideoCodecFFmpeg::GetFormat）
        │
        ├─ 检查 m_decoderState
        │   ├─ 不是 STATE_HW_SINGLE → 跳过硬件尝试
        │   │   └─ 调用 avcodec_default_get_format() → 返回软解格式
        │   └─ 是 STATE_HW_SINGLE → 继续硬件协商
        │
        ├─ 清理旧的硬件解码器（如有）
        │   └─ SetHardware(nullptr)
        │   └─ 恢复 avcodec_default_get_buffer2
        │
        ├─ 遍历 FFmpeg 提供的候选像素格式列表 (*fmt)
        │   ├─ 遇到 DXVA2 格式 (HAS_DX 宏启用时)
        │   │   └─ new DXVA::CDecoder → dec->Open() 尝试初始化
        │   │       ├─ 成功 → SetHardware(dec) → return DXVA2 格式
        │   │       └─ 失败 → dec->Release() → 继续下一个
        │   │
        │   ├─ 遇到 VDPAU 格式 (HAVE_LIBVDPAU 启用时)
        │   │   └─ 同上流程
        │   │
        │   ├─ 遇到 VAAPI 格式 (HAVE_LIBVA 启用时)
        │   │   └─ 同上流程
        │   │
        │   ├─ 遇到 VideoToolbox 格式 (TARGET_DARWIN 启用时)
        │   │   └─ 同上流程
        │   │
        │   └─ 遇到 MMAL 格式 (HAS_MMAL 启用时)
        │       └─ 同上流程
        │
        └─ 所有候选格式遍历完毕，无硬件解码器成功
            └─ m_decoderState = STATE_HW_FAILED
            └─ return avcodec_default_get_format()（回退软解）
```

**对应的源码实现**（`VideoCodecFFmpeg.cpp` 第 104-230 行）：

```cpp
// 来源：krkr2/cpp/core/movie/ffmpeg/VideoCodecFFmpeg.cpp 第 104-230 行（关键部分）
enum AVPixelFormat CDVDVideoCodecFFmpeg::GetFormat(
    struct AVCodecContext *avctx, const AVPixelFormat *fmt)
{
    // 通过 opaque 指针恢复 this（因为这是一个 C 风格的静态回调）
    CDVDVideoCodecFFmpeg *ctx = (CDVDVideoCodecFFmpeg *)avctx->opaque;

    // 如果当前不是"尝试硬件单线程解码"状态，直接走默认路径
    if (ctx->m_decoderState != STATE_HW_SINGLE) {
        return avcodec_default_get_format(avctx, fmt);
    }

    // 清理旧的硬件解码器
    if (ctx->GetHardware()) {
        ctx->SetHardware(nullptr);
        avctx->get_buffer2 = avcodec_default_get_buffer2;
    }

    // 遍历候选像素格式列表
    const AVPixelFormat *cur = fmt;
    while (*cur != AV_PIX_FMT_NONE) {
#ifdef HAS_DX  // Windows 平台
        if (DXVA::CDecoder::Supports(*cur) &&
            CSettings::GetInstance().GetBool(
                CSettings::SETTING_VIDEOPLAYER_USEDXVA2)) {
            // 创建 DXVA2 硬件解码器并尝试打开
            DXVA::CDecoder *dec = new DXVA::CDecoder(ctx->m_processInfo);
            if (dec->Open(avctx, ctx->m_pCodecContext, *cur,
                          ctx->m_uSurfacesCount)) {
                ctx->SetHardware(dec);   // 成功！绑定硬件解码器
                return *cur;             // 返回 DXVA2 像素格式
            } else {
                dec->Release();          // 失败，释放并继续
            }
        }
#endif
#ifdef HAVE_LIBVDPAU  // Linux NVIDIA
        if (VDPAU::CDecoder::Supports(*cur)) {
            VDPAU::CDecoder *dec = new VDPAU::CDecoder(ctx->m_processInfo);
            if (dec->Open(avctx, ctx->m_pCodecContext, *cur)) {
                ctx->SetHardware(dec);
                return *cur;
            } else {
                dec->Release();
            }
        }
#endif
        // ... 类似处理 VAAPI、VideoToolbox、MMAL
        cur++;
    }

    // 所有硬件方案都失败
    ctx->m_decoderState = STATE_HW_FAILED;
    return avcodec_default_get_format(avctx, fmt);  // 回退软解
}
```

**关键设计要点：**

1. **条件编译**：每个硬件 API 都被对应的宏（`HAS_DX`、`HAVE_LIBVDPAU`、`HAVE_LIBVA`、`TARGET_DARWIN`、`HAS_MMAL`）包裹，确保只在支持该 API 的平台上编译相关代码。
2. **opaque 指针模式**：`avctx->opaque` 存放了 `CDVDVideoCodecFFmpeg` 实例的指针，让 C 风格的静态回调能够访问 C++ 对象的成员变量和方法。
3. **优雅回退**：硬件解码器 `Open()` 失败后立即 `Release()`，继续尝试下一个候选格式。全部失败后回退到软件解码，而不是报错中断。

### CDVDVideoCodecFFmpeg 的 Open 方法

`Open` 方法（第 266-435 行）负责完整的解码器初始化，包括硬件/软件模式选择：

```cpp
// 来源：krkr2/cpp/core/movie/ffmpeg/VideoCodecFFmpeg.cpp 第 266-348 行（简化注释版）
bool CDVDVideoCodecFFmpeg::Open(CDVDStreamInfo &hints,
                                 CDVDCodecOptions &options) {
    // 1. 查找解码器
    AVCodec *pCodec = avcodec_find_decoder(hints.codec);
    if (!pCodec) return false;

    // 2. 分配解码器上下文
    m_pCodecContext = avcodec_alloc_context3(pCodec);
    m_pCodecContext->opaque = (void *)this;    // 存放 this 指针给回调用
    m_pCodecContext->get_format = GetFormat;    // 注册格式协商回调

    // 3. 配置线程模型
    if (!hints.software) {
        // 检查各平台是否启用了硬件加速设置
        bool tryhw = false;
#ifdef HAS_DX
        if (CSettings::GetInstance().GetBool(SETTING_VIDEOPLAYER_USEDXVA2))
            tryhw = true;
#endif
#ifdef HAVE_LIBVDPAU
```

### CDropControl — 自适应丢帧控制器

`CDropControl`（第 61-102 行）是 KrKr2 从 Kodi 继承的一个智能丢帧控制器。在播放器来不及解码所有帧时，它通过分析 PTS（Presentation Time Stamp，显示时间戳）间隔来智能决定哪些帧可以安全丢弃：

```
状态机：

  ┌───────────┐                   ┌───────────┐
  │   INIT    │ ──收集>10帧且──→  │   VALID   │
  │ （学习中） │   平均间隔>0     │ （可判断） │
  └───────────┘                   └───────────┘
        ▲                              │
        │                              │
        └──── 连续>5次偏差>20% ────────┘
               （帧率发生变化）
```

- **INIT 阶段**：收集相邻帧的 PTS 差值，累加到 `m_diffPTS`，帧计数器 `m_count` 递增。当收集超过 10 帧后，计算平均帧间隔。
- **VALID 阶段**：每帧检查实际 PTS 差值是否偏离平均值超过 20%。如果连续偏离 5 次以上，说明视频帧率发生了实质性变化（例如从 30fps 切到 24fps），重置到 INIT 重新学习。

**源码对应**（第 73-102 行）：

```cpp
// 来源：krkr2/cpp/core/movie/ffmpeg/VideoCodecFFmpeg.cpp 第 73-102 行
void CDVDVideoCodecFFmpeg::CDropControl::Process(int64_t pts, bool drop) {
    if (m_state == INIT) {
        // 学习阶段：累积帧间 PTS 差值
        if (pts != AV_NOPTS_VALUE && m_lastPTS != AV_NOPTS_VALUE) {
            m_diffPTS += pts - m_lastPTS;  // 累加差值
            m_count++;                     // 增加计数
        }
        if (m_count > 10) {
            m_diffPTS = m_diffPTS / m_count;  // 计算平均帧间隔
            if (m_diffPTS > 0) {
                m_state = VALID;  // 切换到有效状态
                m_count = 0;
            }
        }
    } else if (m_state == VALID && !drop) {
        // 有效阶段：检测帧率是否发生变化
        if (std::abs(pts - m_lastPTS - m_diffPTS) > m_diffPTS * 0.2) {
            m_count++;          // 偏差计数增加
            if (m_count > 5) {
                Reset(true);    // 连续偏差过多，重新学习
            }
        } else {
            m_count = 0;        // 正常帧，重置偏差计数
        }
    }
    m_lastPTS = pts;
}
```

## FFWaveDecoder 桥接架构

### 为什么有两套音频解码

KrKr2 中存在两套独立的音频解码实现，服务于不同的场景：

| 特性 | CDVDAudioCodecFFmpeg（movie 模块） | FFWaveDecoder（sound 模块） |
|------|-----------------------------------|---------------------------|
| 用途 | 视频播放时的音频轨道解码 | BGM（背景音乐）、SE（音效）等纯音频文件 |
| 来源 | 移植自 Kodi/XBMC | KrKr2 自研 |
| API 版本 | 旧版 `avcodec_decode_audio4` | 新版 `avcodec_send_packet`/`avcodec_receive_frame` |
| 格式转换 | 不转换，直传下游 | 手动平面→交错转换 |
| I/O 方式 | 通过 movie 模块的 demuxer 获取数据 | 通过 KiriKiri 虚拟文件系统（`tTJSBinaryStream`） |
| 解码器接口 | `CDVDAudioCodec`（Kodi 接口） | `tTVPWaveDecoder`（KiriKiri 接口） |

### FFWaveDecoder 的自定义 I/O

`FFWaveDecoder` 最有趣的部分是它如何通过自定义 I/O 回调将 FFmpeg 与 KiriKiri 的虚拟文件系统连接起来：

```cpp
// 来源：krkr2/cpp/core/sound/FFWaveDecoder.cpp（概念性简化）
// KiriKiri 引擎有自己的虚拟文件系统，文件可能来自 XP3 压缩包、
// 内存缓冲区等，不是标准的磁盘文件路径

// 自定义读取回调：FFmpeg 调用它来读取数据
static int ffmpegReadCallback(void *opaque, uint8_t *buf, int buf_size) {
    // opaque 指向 tTJSBinaryStream（KiriKiri 的流对象）
    tTJSBinaryStream *stream = (tTJSBinaryStream *)opaque;
    // 从 KiriKiri 流中读取数据到 FFmpeg 的缓冲区
    tjs_uint read = stream->Read(buf, buf_size);
    if (read == 0) return AVERROR_EOF;
    return (int)read;
}

// 自定义定位回调：FFmpeg 调用它来定位（seek）
static int64_t ffmpegSeekCallback(void *opaque, int64_t offset, int whence) {
    tTJSBinaryStream *stream = (tTJSBinaryStream *)opaque;
    switch (whence) {
        case SEEK_SET: stream->SetPosition(offset); break;
        case SEEK_CUR: stream->SetPosition(stream->GetPosition() + offset); break;
        case SEEK_END: stream->SetPosition(stream->GetSize() + offset); break;
        case AVSEEK_SIZE: return stream->GetSize();  // FFmpeg 特有的查询文件大小
    }
    return stream->GetPosition();
}

// 在 SetStream() 中创建自定义 I/O 上下文
void FFWaveDecoder::SetStream(tTJSBinaryStream *stream) {
    // avio_alloc_context: 创建自定义 I/O 上下文
    // 参数1: 缓冲区指针
    // 参数2: 缓冲区大小（通常 4096~32768）
    // 参数3: 0=只读，1=可写
    // 参数4: opaque 指针（传给回调函数）
    // 参数5: 读回调
    // 参数6: 写回调（只读时传 NULL）
    // 参数7: seek 回调
    AVIOContext *ioCtx = avio_alloc_context(
        buffer, bufferSize, 0,
        (void *)stream,          // opaque = KiriKiri 流对象
        ffmpegReadCallback,      // 读数据
        nullptr,                 // 不需要写
        ffmpegSeekCallback       // 定位
    );

    // 将自定义 I/O 绑定到 AVFormatContext
    m_pFormatContext->pb = ioCtx;
    // 然后正常调用 avformat_open_input 即可
    avformat_open_input(&m_pFormatContext, nullptr, nullptr, nullptr);
}
```

这种自定义 I/O 机制让 FFmpeg 能够解码 KiriKiri 引擎虚拟文件系统中的文件，包括 XP3 压缩包中的音频资源——而 FFmpeg 本身对 XP3 格式一无所知。

## 动手实践

### 实践：列出系统支持的硬件加速

编写一个程序，遍历所有 FFmpeg 支持的硬件设备类型，并尝试初始化每一个，报告当前系统上哪些硬件加速可用：

```cpp
// hw_check.cpp — 检测系统支持的硬件加速
#include <cstdio>
extern "C" {
#include <libavutil/hwcontext.h>
#include <libavcodec/avcodec.h>
}

int main() {
    printf("=== 系统硬件加速支持检测 ===\n\n");

    // 遍历所有硬件设备类型
    // av_hwdevice_iterate_types: 迭代所有已注册的硬件设备类型
    // 参数: 上一次调用的返回值（第一次传 AV_HWDEVICE_TYPE_NONE）
    // 返回: 下一个设备类型，AV_HWDEVICE_TYPE_NONE 表示结束
    enum AVHWDeviceType type = AV_HWDEVICE_TYPE_NONE;
    while ((type = av_hwdevice_iterate_types(type)) != AV_HWDEVICE_TYPE_NONE) {
        const char *name = av_hwdevice_get_type_name(type);

        // 尝试创建硬件设备上下文
        // av_hwdevice_ctx_create: 创建指定类型的硬件设备上下文
        // 参数1: 输出的 AVBufferRef 指针
        // 参数2: 设备类型
        // 参数3: 设备路径（NULL = 默认设备）
        // 参数4: 额外选项（NULL = 默认）
        // 参数5: 标志位（通常传 0）
        // 返回: 0 成功，< 0 失败
        AVBufferRef *hwDeviceCtx = nullptr;
        int ret = av_hwdevice_ctx_create(&hwDeviceCtx, type, nullptr, nullptr, 0);

        if (ret == 0) {
            printf("  ✅ %s — 可用\n", name);
            av_buffer_unref(&hwDeviceCtx);
        } else {
            char errbuf[256];
            av_strerror(ret, errbuf, sizeof(errbuf));
            printf("  ❌ %s — 不可用 (%s)\n", name, errbuf);
        }
    }

    // 查询 H.264 解码器的硬件加速支持
    printf("\n=== H.264 解码器硬件加速 ===\n");
    const AVCodec *codec = avcodec_find_decoder(AV_CODEC_ID_H264);
    if (codec) {
        for (int i = 0;; i++) {
            const AVCodecHWConfig *config = avcodec_get_hw_config(codec, i);
            if (!config) break;
            if (config->methods & AV_CODEC_HW_CONFIG_METHOD_HW_DEVICE_CTX) {
                printf("  %s → 像素格式 %s\n",
                       av_hwdevice_get_type_name(config->device_type),
                       av_get_pix_fmt_name(config->pix_fmt));
            }
        }
    }

    return 0;
}
```

```bash
#   ✅ dxva2 — 可用
#   ✅ d3d11va — 可用
#   ❌ vaapi — 不可用 (...)
#   ❌ vdpau — 不可用 (...)
```

## 常见错误及解决方案

### 错误 1：硬件加速初始化失败但没有正确回退

**症状**：程序崩溃或黑屏，日志显示硬件解码器创建失败但没有切换到软解。

**原因**：没有在硬件初始化失败后正确回退到软件解码。

**解决方案**：参考 KrKr2 的做法——在 `get_format` 回调中，当所有硬件方案都失败时，设置状态标志（如 `STATE_HW_FAILED`）并返回 `avcodec_default_get_format()`。解码入口检测到此状态后请求重新打开解码器。

```cpp
// 关键模式：永远有回退路径
if (allHardwareFailed) {
    m_decoderState = STATE_HW_FAILED;
    return avcodec_default_get_format(avctx, fmt);  // 软解回退
}
```

### 错误 2：多线程解码导致画面花屏或崩溃

**症状**：使用多线程解码时偶尔出现画面撕裂、花屏，或随机崩溃。

**原因**：回调函数不是线程安全的，或者在多线程解码的 `get_buffer2` 回调中使用了共享资源但没有加锁。

**解决方案**：

```cpp
// 确保回调线程安全
ctx->thread_safe_callbacks = 1;

// 或者使用 FFmpeg 自带的锁保护
ctx->thread_safe_callbacks = 0;  // FFmpeg 会在调用回调前加锁

// 如果使用自定义 get_buffer2，确保它是可重入的
```

### 错误 3：硬件解码后无法读取帧数据

**症状**：`avcodec_receive_frame` 成功，但 `frame->data[0]` 为 NULL 或读取时段错误（segfault）。

**原因**：硬件解码的帧数据存储在 GPU 显存中，`frame->data` 指针指向的是设备内存，不能直接用 CPU 访问。

**解决方案**：使用 `av_hwframe_transfer_data` 将数据从 GPU 拷贝到系统内存：

```cpp
if (frame->format == AV_PIX_FMT_DXVA2_VLD ||
    frame->format == AV_PIX_FMT_VAAPI) {
    AVFrame *sw_frame = av_frame_alloc();
    av_hwframe_transfer_data(sw_frame, frame, 0);
    // 现在 sw_frame->data 可以安全访问
    // 使用完毕后 av_frame_free(&sw_frame)
}
```

## 本节小结

- **硬件加速解码**利用 GPU/专用芯片完成视频解码，CPU 占用极低，但仅支持特定编码格式
- **各平台 API 不同**：Windows 用 DXVA2/D3D11VA，Linux 用 VAAPI/VDPAU，macOS 用 VideoToolbox，Android 用 MediaCodec
- **FFmpeg 通过 `get_format` 回调**协商硬件加速，应用层在回调中遍历候选格式并尝试初始化硬件解码器
- **多线程软解**是硬件加速的回退方案，KrKr2 使用 `CPU核心数 × 1.5`（上限 16）个线程
- **KrKr2 的 `CDVDVideoCodecFFmpeg`** 实现了完整的硬件/软件解码切换：`GetFormat` 回调尝试硬件 → 失败则设置 `STATE_HW_FAILED` → `Decode` 方法检测到后请求重新打开
- **`CDropControl`** 是一个两状态的自适应丢帧控制器，通过分析 PTS 间隔来智能丢帧，并能检测帧率变化
- **`CDVDAudioCodecFFmpeg`** 使用旧 API 解码音频，不做格式转换，直接映射 AV→AE 格式传给下游
- **`FFWaveDecoder`** 通过自定义 I/O 回调桥接 KiriKiri 虚拟文件系统与 FFmpeg，手动实现平面→交错转换

## 练习题与答案

### 题目 1：硬件加速回退流程

描述当一个 Windows 用户播放 H.264 视频时，KrKr2 的解码器初始化经历的完整步骤。假设该用户的显卡 DXVA2 驱动有 bug，初始化会失败。

<details>
<summary>查看答案</summary>

1. `CDVDVideoCodecFFmpeg::Open()` 被调用
2. `avcodec_find_decoder(AV_CODEC_ID_H264)` 找到 H.264 解码器
3. 分配 `AVCodecContext`，设置 `get_format = GetFormat`
4. 检查 `HAS_DX` 宏已定义，检查设置中 DXVA2 已启用 → `tryhw = true`
5. `m_decoderState = STATE_HW_SINGLE`（标记为尝试硬件解码）
6. `avcodec_open2()` 打开解码器
7. 第一次解码时，FFmpeg 内部调用 `GetFormat` 回调
8. `GetFormat` 检查 `m_decoderState == STATE_HW_SINGLE` → 进入硬件协商
9. 遍历候选像素格式，遇到 `AV_PIX_FMT_DXVA2_VLD`
10. `new DXVA::CDecoder()` → `dec->Open()` → **失败**（驱动 bug）
11. `dec->Release()` 释放失败的解码器
12. 继续遍历候选格式，没有更多硬件格式
13. `m_decoderState = STATE_HW_FAILED`
14. 返回 `avcodec_default_get_format()`（选择 YUV420P 软解格式）
15. 回到 `Decode()` 方法，检测到 `STATE_HW_FAILED` → 返回 `VC_REOPEN`
16. 上层 `VideoPlayer` 收到 `VC_REOPEN`，重新打开解码器
17. 这次 `Open()` 中，因为 `m_decoderState` 不再是 `STATE_NONE`，走软解路径
18. 计算线程数 = `CPU核心数 × 1.5`，设置多线程软解
19. 解码器以多线程软解模式正常工作

</details>

### 题目 2：比较两套音频解码器

KrKr2 有两套音频解码实现（`CDVDAudioCodecFFmpeg` 和 `FFWaveDecoder`），请回答：
1. 它们分别使用哪个版本的 FFmpeg 解码 API？
2. 在格式转换策略上有什么不同？为什么？
3. 如果要给 KrKr2 添加一个新的音频格式支持（比如 FLAC），需要修改哪些地方？

<details>
<summary>查看答案</summary>

1. **API 版本**：
   - `CDVDAudioCodecFFmpeg` 使用旧版 API `avcodec_decode_audio4()`
   - `FFWaveDecoder` 使用新版 API `avcodec_send_packet()`/`avcodec_receive_frame()`（`audio_decode_frame()` 内部）
   - 两者都能正常工作，因为 FFmpeg 3.x 同时支持新旧两套 API

2. **格式转换策略**：
   - `CDVDAudioCodecFFmpeg` **不做格式转换**——通过 `GetDataFormat()` 将 AV 格式映射为 AE 格式，原样传给下游的音频引擎（`CAudioSinkAE`），由音频引擎决定是否需要重采样
   - `FFWaveDecoder` **手动做格式转换**——使用 `CopySmaples` 模板函数将平面格式转为交错格式，因为 KiriKiri 的 `tTVPWaveDecoder` 接口要求交错格式的 PCM 输出
   - 原因：两者服务的下游接口不同。Kodi 的音频引擎自带格式转换能力，KiriKiri 的接口则期望直接可用的交错 PCM

3. **添加 FLAC 支持需要修改的地方**：
   - **可能不需要修改任何代码**——FFmpeg 自带 FLAC 解码器，`avcodec_find_decoder(AV_CODEC_ID_FLAC)` 会自动找到它
   - 需要确认 KrKr2 的 vcpkg FFmpeg overlay port（`vcpkg/ports/ffmpeg/`）在编译时启用了 FLAC 解码支持
   - 如果 KiriKiri 的文件类型判断逻辑不识别 `.flac` 扩展名，需要在音频加载器的格式检测代码中添加 FLAC 支持

</details>

### 题目 3：设计一个简易的硬件加速管理器

基于 KrKr2 的 `GetFormat` 模式，设计一个简化版的硬件加速管理器类，要求：
1. 支持 DXVA2 和 VAAPI 两种硬件加速
2. 优先尝试硬件加速，失败后回退到多线程软解
3. 提供 `isHardwareAccelerated()` 查询方法

<details>
<summary>查看答案</summary>

```cpp
#include <cstdio>
#include <algorithm>
extern "C" {
#include <libavcodec/avcodec.h>
}

class HWAccelManager {
public:
    enum State { NONE, TRY_HW, HW_ACTIVE, HW_FAILED, SW_MULTI };

    HWAccelManager() : m_state(NONE), m_hwPixFmt(AV_PIX_FMT_NONE) {}

    // 配置解码器上下文
    void configure(AVCodecContext *ctx, int cpuCores) {
        ctx->opaque = this;
        ctx->get_format = formatCallback;
        m_state = TRY_HW;
        m_cpuCores = cpuCores;
    }

    bool isHardwareAccelerated() const { return m_state == HW_ACTIVE; }

    // 如果硬件失败，配置软解
    void fallbackToSoftware(AVCodecContext *ctx) {
        int threads = std::max(1, std::min(m_cpuCores * 3 / 2, 16));
        ctx->thread_count = threads;
        ctx->thread_safe_callbacks = 1;
        m_state = SW_MULTI;
        printf("回退到 %d 线程软解\n", threads);
    }

private:
    State m_state;
    AVPixelFormat m_hwPixFmt;
    int m_cpuCores = 4;

    static enum AVPixelFormat formatCallback(
        AVCodecContext *ctx, const enum AVPixelFormat *fmts)
    {
        auto *self = (HWAccelManager *)ctx->opaque;
        if (self->m_state != TRY_HW) {
            return avcodec_default_get_format(ctx, fmts);
        }

        for (const auto *p = fmts; *p != AV_PIX_FMT_NONE; p++) {
            if (*p == AV_PIX_FMT_DXVA2_VLD || *p == AV_PIX_FMT_VAAPI) {
                // 实际项目中这里要初始化硬件设备
                self->m_hwPixFmt = *p;
                self->m_state = HW_ACTIVE;
                printf("硬件加速激活: %s\n", av_get_pix_fmt_name(*p));
                return *p;
            }
        }

        self->m_state = HW_FAILED;
        printf("硬件加速不可用\n");
        return avcodec_default_get_format(ctx, fmts);
    }
};
```

</details>
