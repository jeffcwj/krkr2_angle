## 5.1 为什么需要音视频同步

### 5.1.1 同步问题的本质

在一个视频播放器中，音频和视频是**两条独立的数据流**，各自由独立的线程解码和渲染：

```
┌─────────────────┐        ┌──────────────────┐
│   音频线程       │        │   视频线程        │
│                 │        │                  │
│  解码音频包 ──> │        │ <── 解码视频包    │
│  送入声卡 ──>   │        │ <── 渲染到屏幕    │
│                 │        │                  │
│  速率：声卡驱动  │        │ 速率：帧率/VSync  │
└─────────────────┘        └──────────────────┘
```

问题在于：
- **音频输出速率**由声卡硬件时钟驱动，每秒精确消耗 48000 个采样（48kHz 时）
- **视频输出速率**由帧率和 VSync 决定，通常每秒 24/30/60 帧
- 两者的时钟源不同，长时间运行后**必然产生偏差**

即使各自完美按照 PTS 时间戳播放，一小时后音视频也可能偏差几百毫秒甚至几秒。

### 5.1.2 人耳对同步偏差的感知

| 偏差范围 | 感知效果 |
|----------|----------|
| ±20ms 以内 | 完全感知不到，理想状态 |
| ±40ms | 敏感用户可能察觉 |
| ±80ms | 多数用户可感知"嘴型不对" |
| ±120ms | 明显不同步，观看体验严重受损 |
| ±200ms+ | 完全无法接受 |

> **关键洞察**：人类对"音频领先于视频"比"视频领先于音频"更敏感。音频提前 20ms 就可能被察觉，而视频提前 40ms 才会被注意到。因此同步算法通常对音频领先的容忍度更低。

### 5.1.3 同步问题的常见症状

| 症状 | 原因 | 解决方向 |
|------|------|---------|
| 嘴型不对 | 音视频偏差超过 80ms | 检查主时钟选择、同步策略 |
| 视频卡顿但音频流畅 | 视频解码太慢跟不上音频时钟 | 丢帧、降低分辨率、硬件加速 |
| 音频断续 | 音频缓冲区欠载（underrun） | 增大缓冲区、优先级调整 |
| 播放久了越来越不同步 | 时钟漂移（drift）未修正 | 启用持续的漂移补偿 |
| seek 后音画不同步 | seek 后未正确重置同步状态 | flush 后重新建立同步点 |

## 5.2 PTS、DTS 与时间基（time_base）

### 5.2.1 PTS 和 DTS 复习

在第一节中我们介绍了 PTS 和 DTS 的概念，这里深入讲解它们在同步中的具体应用：

- **PTS**（Presentation Time Stamp）：帧应该被**显示/播放**的时间
- **DTS**（Decode Time Stamp）：帧应该被**解码**的时间

对于**没有 B 帧**的编码（如多数音频编码、MJPEG 等），PTS == DTS。
对于**有 B 帧**的编码（如 H.264、H.265），DTS ≤ PTS，因为 B 帧的解码顺序和显示顺序不同。

```
编码顺序 (DTS):  I0  P3  B1  B2  P6  B4  B5
显示顺序 (PTS):  I0  B1  B2  P3  B4  B5  P6

DTS:  0   1   2   3   4   5   6
PTS:  0   2   3   1   5   6   4
```

### 5.2.2 时间基（time_base）详解

FFmpeg 中的时间戳不是绝对时间，而是以**时间基**为单位的整数。时间基是一个有理数（分数），表示一个时间刻度代表多少秒：

```cpp
// AVRational 表示一个分数
typedef struct AVRational {
    int num;  // 分子
    int den;  // 分母
} AVRational;

// 常见时间基
AVStream->time_base  = {1, 90000}   // MPEG-TS 常用，1 tick = 1/90000 秒
AVStream->time_base  = {1, 1000}    // Matroska 常用，1 tick = 1 毫秒
AVStream->time_base  = {1, 48000}   // 音频流，1 tick = 1 个采样

// 将 PTS 转换为秒
double pts_seconds = frame->pts * av_q2d(stream->time_base);
// av_q2d 将 AVRational 转为 double: num / den

// 不同时间基之间的转换
int64_t target_pts = av_rescale_q(
    source_pts,
    source_time_base,   // 源时间基
    target_time_base    // 目标时间基
);
```

### 5.2.3 时间基转换实战

```cpp
#include <cstdio>
extern "C" {
#include <libavformat/avformat.h>
#include <libavutil/mathematics.h>
}

void demonstrate_timebase() {
    // 场景：视频流 time_base={1,90000}，帧 PTS=4500000
    AVRational video_tb = {1, 90000};
    int64_t video_pts = 4500000;

    // 转换为秒
    double seconds = video_pts * av_q2d(video_tb);
    printf("视频帧时间: %.3f 秒\n", seconds);  // 输出: 50.000 秒

    // 转换为毫秒时间基
    AVRational ms_tb = {1, 1000};
    int64_t ms_pts = av_rescale_q(video_pts, video_tb, ms_tb);
    printf("毫秒时间: %lld ms\n", (long long)ms_pts);  // 输出: 50000 ms

    // 转换为 DVD_TIME_BASE（微秒，KrKr2 使用的时间基）
    // DVD_TIME_BASE = 1000000
    AVRational dvd_tb = {1, 1000000};
    int64_t dvd_pts = av_rescale_q(video_pts, video_tb, dvd_tb);
    printf("DVD 时间: %lld μs\n", (long long)dvd_pts);  // 输出: 50000000 μs
}
```

### 5.2.4 KrKr2 的时间戳转换

KrKr2 在 `Clock.h` 中定义了自己的时间系统（继承自 Kodi）：

```cpp
// 来源：krkr2/cpp/core/movie/ffmpeg/Clock.h 第 8-16 行
#define DVD_TIME_BASE    1000000               // 基础单位：微秒
#define DVD_NOPTS_VALUE  0xFFF0000000000000     // 无效时间戳哨兵值

#define DVD_TIME_TO_MSEC(x)  ((int)((double)(x) * 1000 / DVD_TIME_BASE))
#define DVD_SEC_TO_TIME(x)   ((double)(x) * DVD_TIME_BASE)
#define DVD_MSEC_TO_TIME(x)  ((double)(x) * DVD_TIME_BASE / 1000)

#define DVD_PLAYSPEED_PAUSE   0     // 暂停
#define DVD_PLAYSPEED_NORMAL  1000  // 正常速度 (1.0x)
```

在解码器中，FFmpeg 的 `AV_TIME_BASE`（也是微秒）和 `DVD_TIME_BASE` 之间的转换：

```cpp
// 来源：VideoCodecFFmpeg.cpp 第 599-602 行
avpkt.dts = (dts == DVD_NOPTS_VALUE) ? AV_NOPTS_VALUE
                                     : dts / DVD_TIME_BASE * AV_TIME_BASE;
avpkt.pts = (pts == DVD_NOPTS_VALUE) ? AV_NOPTS_VALUE
                                     : pts / DVD_TIME_BASE * AV_TIME_BASE;
```

由于 `DVD_TIME_BASE == AV_TIME_BASE == 1000000`（两者恰好相等），这里实际上是直接赋值。但代码保留了转换以保持通用性。

## 5.3 三种主时钟策略

### 5.3.1 概述

音视频同步的核心问题是：**以谁为基准**？三种策略各有优劣：

```
策略 1：音频主时钟           策略 2：视频主时钟          策略 3：外部主时钟
┌──────────┐                ┌──────────┐               ┌──────────┐
│ 音频时钟  │  ← 基准       │ 视频时钟  │  ← 基准      │ 系统时钟  │  ← 基准
│ (声卡驱动) │               │ (帧率驱动) │              │ (墙钟)    │
└────┬─────┘                └────┬─────┘               └────┬─────┘
     │ 视频跟随音频               │ 音频跟随视频              │ 音/视频都跟随
     ▼                           ▼                          ▼
┌──────────┐                ┌──────────┐               ┌──────────┐
│ 视频调整  │                │ 音频调整  │               │ 音频调整  │
│ 丢帧/重复 │                │ 重采样    │               │ 视频调整  │
└──────────┘                └──────────┘               └──────────┘
```

### 5.3.2 音频主时钟（Audio Master Clock）— 最常用

**原理**：以声卡的硬件时钟为基准。音频数据按固定速率输出到声卡，视频根据音频时钟调整显示节奏。

**优点**：
- 声卡硬件时钟非常稳定精确
- 人耳对音频中断极其敏感，保证音频连续性最重要
- 实现相对简单

**缺点**：
- 视频可能需要丢帧或重复帧来追赶/等待音频
- 丢帧会导致画面跳跃

**实现算法**：

```cpp
// 伪代码：音频主时钟的视频同步
void render_video_frame(AVFrame *frame) {
    double video_pts = frame->pts * av_q2d(video_stream->time_base);
    double audio_clock = get_audio_clock();  // 从声卡获取当前播放位置

    double diff = video_pts - audio_clock;   // 正值=视频超前，负值=视频落后

    double sync_threshold = 0.04;  // 40ms 容差

    if (diff > sync_threshold) {
        // 视频超前：等待（延迟显示）
        double delay = diff;
        usleep((int)(delay * 1000000));
        display_frame(frame);
    } else if (diff < -sync_threshold) {
        // 视频落后：丢弃当前帧，尝试追赶
        // 不显示这帧，直接跳到下一帧
        skip_frame();
    } else {
        // 在容差范围内：正常显示
        display_frame(frame);
    }
}
```

### 5.3.3 视频主时钟（Video Master Clock）

**原理**：以视频帧率或 VSync 刷新率为基准。视频按固定帧率显示，音频通过重采样（resample）调整播放速率来匹配。

**优点**：
- 视频播放绝对流畅，无丢帧
- 适合对画面连续性要求极高的场景

**缺点**：
- 音频重采样会引入微小的音质损失
- 如果偏差较大，变速播放可能被听出来

### 5.3.4 外部主时钟（External/System Clock）

**原理**：使用系统墙钟（wall clock）作为基准，音频和视频都独立调整自己的播放速度来匹配。

**优点**：
- 适合直播、视频会议等需要与真实时间对齐的场景
- 两条流都不是"特权"方

**缺点**：
- 实现复杂度最高
- 音频和视频都需要调整，出问题的概率更大

### 5.3.5 KrKr2 的同步策略选择

KrKr2 支持两种同步模式，在 `CVideoPlayerAudio` 中定义：

```cpp
// 来源：推断自 VideoPlayerAudio.cpp 第 498-522 行
enum SyncType {
    SYNC_DISCON  = 0,  // 不连续校正（discontinuity correction）
    SYNC_RESAMPLE = 1  // 重采样校正
};
```

- **SYNC_DISCON**（默认）：音频主时钟模式。检测到同步误差时通过 `CDVDClock::ErrorAdjust` 跳变时钟（discontinuity），视频线程根据时钟调整。
- **SYNC_RESAMPLE**：视频主时钟模式。音频通过重采样调整速率来匹配视频帧率。适合实时流（`hints.realtime == true`）。

```cpp
// 来源：VideoPlayerAudio.cpp 第 109-116 行
m_synctype = SYNC_DISCON;
m_setsynctype = SYNC_DISCON;
// 实时流使用重采样同步
if (hints.realtime)
    m_setsynctype = SYNC_RESAMPLE;
```

## 5.4 CDVDClock 时钟系统详解

### 5.4.1 时钟架构

`CDVDClock` 是 KrKr2 播放器的核心时钟组件，维护一个**可暂停、可变速、可跳变**的播放时钟：

```
┌──────────────────────────────────────────────┐
│                  CDVDClock                     │
│                                               │
│  CVideoReferenceClock (系统时钟源)             │
│      │                                        │
│      ▼ GetTime()                              │
│  m_systemOffset ──┐                           │
│                   ▼                           │
│  SystemToAbsolute(system)                     │
│      = (system - m_systemOffset)              │
│        / m_systemFrequency * DVD_TIME_BASE    │
│                   │                           │
│                   ▼                           │
│  SystemToPlaying(system)                      │
│      = (current - m_startClock + m_systemAdjust)│
│        / m_systemUsed * DVD_TIME_BASE + m_iDisc│
│                                               │
│  关键变量：                                    │
│  ├── m_startClock    起始系统时间              │
│  ├── m_systemUsed    有效频率（受速度影响）     │
│  ├── m_iDisc         不连续跳变量              │
│  ├── m_systemAdjust  累计微调量                │
│  ├── m_speedAdjust   速度微调系数              │
│  ├── m_pauseClock    暂停时冻结的时间          │
│  └── m_vSyncAdjust   VSync 校正量             │
└──────────────────────────────────────────────┘
```

### 5.4.2 三层时间模型

`CDVDClock` 维护了三层时间概念：

| 层次 | 方法 | 含义 |
|------|------|------|
| 系统时间 | `m_videoRefClock->GetTime()` | 单调递增的系统时钟（纳秒/微秒级） |
| 绝对时间 | `SystemToAbsolute()` | 从系统时间换算为 `DVD_TIME_BASE` 单位 |
| 播放时间 | `SystemToPlaying()` | 考虑暂停、变速、跳变后的实际播放位置 |

```cpp
// 来源：Clock.cpp 第 234-237 行
// 系统时间 → 绝对时间
double CDVDClock::SystemToAbsolute(int64_t system) {
    return DVD_TIME_BASE * (double)(system - m_systemOffset) /
        m_systemFrequency;
}

// 来源：Clock.cpp 第 244-267 行
// 系统时间 → 播放时间
double CDVDClock::SystemToPlaying(int64_t system) {
    // 首次调用时重置
    if (m_bReset) {
        m_startClock = system;
        m_systemUsed = m_systemFrequency;
        m_iDisc = 0;
        m_systemAdjust = 0;
        m_speedAdjust = 0;
        m_bReset = false;
    }

    int64_t current;
    if (m_pauseClock)
        current = m_pauseClock;   // 暂停时使用冻结时间
    else
        current = system;         // 正常播放使用当前时间

    // 核心公式：播放时间 = 经过的系统时间 / 有效频率 * 基准 + 不连续量
    return DVD_TIME_BASE * (double)(current - m_startClock + m_systemAdjust) /
        m_systemUsed + m_iDisc;
}
```

### 5.4.3 速度控制

播放速度通过调整 `m_systemUsed`（有效频率）实现：

```cpp
// 来源：Clock.cpp 第 91-119 行
void CDVDClock::SetSpeed(int iSpeed) {
    // iSpeed: DVD_PLAYSPEED_NORMAL=1000 代表 1.0x
    //         500 代表 0.5x，2000 代表 2.0x
    //         DVD_PLAYSPEED_PAUSE=0 代表暂停

    if (iSpeed == DVD_PLAYSPEED_PAUSE) {
        if (!m_pauseClock)
            m_pauseClock = m_videoRefClock->GetTime();  // 冻结当前时间
        return;
    }

    // 计算新的有效频率
    int64_t newfreq = m_systemFrequency * DVD_PLAYSPEED_NORMAL / iSpeed;
    // 例：2x 速度 → newfreq = freq * 1000 / 2000 = freq / 2
    // 分母减半，同样的系统时间流逝，播放时间增长 2 倍

    current = m_videoRefClock->GetTime();

    // 从暂停恢复时，补偿暂停期间的时间
    if (m_pauseClock) {
        m_startClock += current - m_pauseClock;
        m_pauseClock = 0;
    }

    // 平滑切换速度：重新计算 startClock 使得时间连续
    m_startClock = current -
        (int64_t)((double)(current - m_startClock) * newfreq / m_systemUsed);
    m_systemUsed = newfreq;
}
```

### 5.4.4 误差校正（ErrorAdjust）

这是 SYNC_DISCON 模式的核心——当检测到音视频偏差时，通过跳变时钟来纠正：

```cpp
// 来源：Clock.cpp 第 134-169 行
double CDVDClock::ErrorAdjust(double error, const char *log) {
    double clock, absolute, adjustment;
    clock = GetClock(absolute);

    // 当 speedAdjust 活跃且误差较小（<100ms）时跳过微调
    // 避免与速度调整冲突
    if (m_speedAdjust != 0 && error < DVD_MSEC_TO_TIME(100))
        return 0;

    adjustment = error;

    if (m_vSyncAdjust != 0) {
        // VSync 模式下使用帧时间为单位进行校正
        // 音频超前 >20ms → 向前调整一帧时间
        // 音频落后 >27ms → 向后调整一帧时间
        // 20ms/27ms 的不对称阈值正是因为人耳对音频领先更敏感
        if (error > 0.02 * DVD_TIME_BASE)
            adjustment = m_frameTime;       // +1 帧
        else if (error < -0.027 * DVD_TIME_BASE)
            adjustment = -m_frameTime;      // -1 帧
        else
            adjustment = 0;                 // 在容差内，不调整
    }

    if (adjustment == 0)
        return 0;

    // 执行不连续跳变
    Discontinuity(clock + adjustment, absolute);
    return adjustment;
}
```

### 5.4.5 不连续跳变（Discontinuity）

`Discontinuity` 方法直接重置时钟到新的播放位置，通常在 seek、流切换或误差校正时调用：

```cpp
// 来源：Clock.cpp 第 171-180 行
void CDVDClock::Discontinuity(double clock, double absolute) {
    m_startClock = AbsoluteToSystem(absolute);  // 重置起始点
    if (m_pauseClock)
        m_pauseClock = m_startClock;
    m_iDisc = clock;              // 设置新的基准偏移
    m_bReset = false;
    m_systemAdjust = 0;           // 清除累计微调
    m_speedAdjust = 0;            // 清除速度微调
}
```

## 5.5 音频同步状态机

### 5.5.1 CVideoPlayerAudio 的同步状态

`CVideoPlayerAudio` 维护了一个三状态的同步状态机：

```
                 首帧输出
SYNC_STARTING ──────────> SYNC_WAITSYNC
    │                         │
    │ flush/reset             │ 收到 RESYNC 消息
    │                         ▼
    └──────────────────── SYNC_INSYNC
                              │
                              │ flush/reset
                              ▼
                         SYNC_STARTING
```

| 状态 | 含义 | 行为 |
|------|------|------|
| `SYNC_STARTING` | 刚开始或重置后 | 解码并填充音频缓冲区，等待足够数据 |
| `SYNC_WAITSYNC` | 音频就绪，等待视频同步 | 向主循环发送 `PLAYER_STARTED` 消息 |
| `SYNC_INSYNC` | 正常同步播放 | 持续解码+输出，检测同步误差 |

### 5.5.2 音频处理主循环

`CVideoPlayerAudio::Process` 是音频线程的主循环（第 194-496 行），采用消息驱动架构：

```
┌─────────────────────────────────────────────────┐
│            CVideoPlayerAudio::Process            │
│                                                  │
│  while (!m_bStop) {                              │
│      msg = m_messageQueue.Get(timeout, priority) │
│      │                                           │
│      ├── GENERAL_SYNCHRONIZE → 等待同步信号       │
│      ├── GENERAL_RESYNC → 设置 audioClock + INSYNC│
```

### 5.5.3 同步误差检测与修正

在 `OutputPacket` 中，音频线程检测并修正同步误差：

```cpp
// 来源：VideoPlayerAudio.cpp 第 525-538 行
bool CVideoPlayerAudio::OutputPacket(DVDAudioFrame &audioframe) {
    // 获取音频输出与时钟之间的同步误差
    double syncerror = m_dvdAudio.GetSyncError();

    // SYNC_DISCON 模式：误差超过 10ms 时通过时钟跳变修正
    if (m_synctype == SYNC_DISCON && fabs(syncerror) > DVD_MSEC_TO_TIME(10)) {
        double correction = m_pClock->ErrorAdjust(
            syncerror, "CVideoPlayerAudio::OutputPacket");
        if (correction != 0) {
            m_dvdAudio.SetSyncErrorCorrection(-correction);
        }
    }

    // 将音频帧送入音频设备
    m_dvdAudio.AddPackets(audioframe);
    return true;
}
```

整个同步修正流程：

```
音频设备报告 syncerror (当前播放位置 vs 期望位置)
    │
    ▼
|syncerror| > 10ms ?
    │
    ├── 否 → 误差在容忍范围内，不调整
    │
    └── 是 → CDVDClock::ErrorAdjust(syncerror)
              │
              ├── VSync 模式：按帧时间步进调整（±1帧）
              │     20ms < error  → +1 帧
              │     error < -27ms → -1 帧
              │
              └── 非 VSync：直接跳变 error 量
                    │
                    ▼
              Discontinuity(clock + adjustment)
              → 时钟瞬间跳变，视频线程跟随
```

### 5.5.4 音频时钟的维护

音频播放时钟 `m_audioClock` 通过两种方式更新：

```cpp
// 方式 1：从解码帧的 PTS 更新（精确）
// 来源：VideoPlayerAudio.cpp 第 360-366 行
if (audioframe.pts == DVD_NOPTS_VALUE) {
    audioframe.pts = m_audioClock;       // 无 PTS 时用预测值
    audioframe.hasTimestamp = false;
} else {
    m_audioClock = audioframe.pts;       // 有 PTS 时直接采用
}

// 方式 2：根据帧持续时间递增（预测下一帧）
// 来源：VideoPlayerAudio.cpp 第 473 行

// duration = nb_samples / sample_rate * DVD_TIME_BASE
```

### 实践 1：实现音频主时钟同步

以下是一个简化的音视频同步框架，展示音频主时钟策略的核心逻辑：

```cpp
// av_sync_demo.cpp — 音视频同步原理演示
#include <cstdio>
#include <cmath>
#include <chrono>
#include <thread>

// 模拟时钟（简化版 CDVDClock）
class SimpleClock {
public:
    void start() {
        m_startTime = now();
        m_paused = false;
    }

    void pause(bool p) {
        if (p && !m_paused) {
            m_pauseTime = now();
            m_paused = true;
        } else if (!p && m_paused) {
            m_startTime += (now() - m_pauseTime);
            m_paused = false;
        }
    }

    // 获取当前播放时间（秒）
    double getClock() const {
        if (m_paused)
            return (m_pauseTime - m_startTime) / 1e6;
        return (now() - m_startTime) / 1e6;
    }

    // 跳变修正
    void adjust(double seconds) {
        m_startTime -= (int64_t)(seconds * 1e6);
    }

private:
    static int64_t now() {
        auto t = std::chrono::steady_clock::now().time_since_epoch();
        return std::chrono::duration_cast<std::chrono::microseconds>(t).count();
    }

    int64_t m_startTime = 0;
    int64_t m_pauseTime = 0;
    bool m_paused = false;
};

// 模拟视频帧
struct VideoFrame {
    double pts;        // 显示时间（秒）
    int frameNumber;
    bool isKeyFrame;
};

// 音频主时钟同步逻辑
class AudioMasterSync {
public:
    AudioMasterSync() : m_syncThreshold(0.04) {}  // 40ms 容差

    enum Action { DISPLAY, SKIP, WAIT };

    // 决定视频帧的处理方式
    Action decide(const VideoFrame &frame, double audioClock) {
        double diff = frame.pts - audioClock;

        if (diff < -m_syncThreshold) {
            // 视频落后超过阈值 → 丢帧
            m_droppedFrames++;
            return SKIP;
        } else if (diff > m_syncThreshold) {
            // 视频超前超过阈值 → 等待
            m_waitMs = (int)(diff * 1000);
            return WAIT;
        } else {
            // 在同步范围内 → 正常显示
            return DISPLAY;
        }
    }

    int getWaitMs() const { return m_waitMs; }
    int getDroppedFrames() const { return m_droppedFrames; }

private:
    double m_syncThreshold;
    int m_waitMs = 0;
    int m_droppedFrames = 0;
};

int main() {
    SimpleClock clock;
    AudioMasterSync sync;

    // 模拟 30fps 视频帧序列
    const int totalFrames = 150;  // 5 秒
    const double fps = 30.0;

    clock.start();

    for (int i = 0; i < totalFrames; i++) {
        VideoFrame frame;
        frame.pts = i / fps;
        frame.frameNumber = i;
        frame.isKeyFrame = (i % 30 == 0);

        // 模拟音频时钟（假设音频稳定播放，加入小随机抖动）
        double audioClock = clock.getClock();

        auto action = sync.decide(frame, audioClock);

        switch (action) {
            case AudioMasterSync::DISPLAY:
                printf("帧 #%03d [%.3fs]: 显示 (diff=%.1fms)\n",
                       i, frame.pts, (frame.pts - audioClock) * 1000);
                // 模拟显示耗时
                std::this_thread::sleep_for(std::chrono::milliseconds(33));
                break;

            case AudioMasterSync::SKIP:
                printf("帧 #%03d [%.3fs]: 丢弃 (落后 %.1fms)\n",
                       i, frame.pts, (audioClock - frame.pts) * 1000);
                break;

            case AudioMasterSync::WAIT: {
                int waitMs = sync.getWaitMs();
                printf("帧 #%03d [%.3fs]: 等待 %dms\n",
                       i, frame.pts, waitMs);
                std::this_thread::sleep_for(std::chrono::milliseconds(waitMs));
                printf("帧 #%03d [%.3fs]: 显示（等待后）\n", i, frame.pts);
                break;
            }
        }
    }

    printf("\n=== 同步统计 ===\n");
    printf("总帧数: %d\n", totalFrames);
    printf("丢弃帧: %d\n", sync.getDroppedFrames());
    printf("实际时长: %.3f 秒（期望 %.3f 秒）\n",
           clock.getClock(), totalFrames / fps);

    return 0;
}
```

### 实践 2：PTS 时间基转换工具

```cpp
// timebase_converter.cpp — 时间基转换实用工具
#include <cstdio>
#include <cstdlib>

extern "C" {
#include <libavformat/avformat.h>
#include <libavutil/mathematics.h>
}

int main(int argc, char *argv[]) {
    if (argc < 2) {
        fprintf(stderr, "用法: %s <媒体文件>\n", argv[0]);
        return 1;
    }

    AVFormatContext *fmtCtx = nullptr;
    avformat_open_input(&fmtCtx, argv[1], nullptr, nullptr);
    avformat_find_stream_info(fmtCtx, nullptr);

    printf("=== 流时间基信息 ===\n");
    for (unsigned i = 0; i < fmtCtx->nb_streams; i++) {
        AVStream *st = fmtCtx->streams[i];
        const char *type = av_get_media_type_string(st->codecpar->codec_type);
        printf("\n流 #%u (%s):\n", i, type ? type : "未知");
        printf("  time_base: %d/%d (1 tick = %.6f 秒)\n",
               st->time_base.num, st->time_base.den,
               av_q2d(st->time_base));
        printf("  duration:  %lld ticks = %.3f 秒\n",
               (long long)st->duration,
               st->duration * av_q2d(st->time_base));
        printf("  start_time: %lld ticks = %.3f 秒\n",
               (long long)st->start_time,
               st->start_time != AV_NOPTS_VALUE
                   ? st->start_time * av_q2d(st->time_base) : 0.0);
    }

    // 读取前 20 个包，展示 PTS/DTS 转换
    printf("\n=== 前 20 个包的时间戳 ===\n");
    printf("%-6s %-8s %-12s %-12s %-12s %-12s\n",
           "流", "类型", "PTS(tick)", "PTS(秒)", "DTS(tick)", "DTS(秒)");

    AVPacket *pkt = av_packet_alloc();
    int count = 0;
    while (av_read_frame(fmtCtx, pkt) >= 0 && count < 20) {
        AVStream *st = fmtCtx->streams[pkt->stream_index];
        const char *type = av_get_media_type_string(st->codecpar->codec_type);

        double ptsSec = (pkt->pts != AV_NOPTS_VALUE)
                            ? pkt->pts * av_q2d(st->time_base) : -1.0;
        double dtsSec = (pkt->dts != AV_NOPTS_VALUE)
                            ? pkt->dts * av_q2d(st->time_base) : -1.0;

        printf("%-6u %-8s %-12lld %-12.3f %-12lld %-12.3f\n",
               pkt->stream_index, type ? type : "?",
               (long long)pkt->pts, ptsSec,
               (long long)pkt->dts, dtsSec);

        av_packet_unref(pkt);
        count++;
    }

    av_packet_free(&pkt);
    avformat_close_input(&fmtCtx);
    return 0;
}
```

### 常见错误与解决方案

**错误 1：seek 后音画不同步**

```
原因：seek 后没有正确 flush 解码器和重置同步状态
解决：
1. avcodec_flush_buffers() 清空解码器内部缓冲
2. 清空音频输出缓冲区
3. 重置同步状态为 SYNC_STARTING
4. 用 seek 目标位置重新设置 audioClock
（KrKr2 通过 GENERAL_FLUSH 消息完成以上所有步骤）
```

**错误 2：播放一段时间后逐渐不同步**

```
原因：时钟漂移（drift）—— 声卡实际采样率与标称值有微小偏差
      例如标称 48000Hz 的声卡实际可能是 47998Hz
解决：
1. 持续监测同步误差
2. 使用微量的速度调整（speedAdjust）而非大幅跳变
3. KrKr2 通过 CDVDClock::SetSpeedAdjust() 实现微调
```

**错误 3：av_rescale_q 溢出**

```
原因：大的 PTS 值在不同时间基转换时可能溢出 int64
解决：使用 av_rescale_q_rnd() 指定舍入模式
      或用 double 中间值：pts * av_q2d(src_tb) / av_q2d(dst_tb)
```

## 5.7 本节小结

- **音视频同步**的核心问题是：音频和视频由不同硬件时钟驱动，长时间运行后会产生漂移
- **PTS/DTS/time_base** 三者共同定义了帧的时间位置，跨流/跨模块传递时必须正确转换时间基
- 三种主时钟策略：**音频主时钟**（最常用，保证音频连续性）、**视频主时钟**（重采样调整音频）、**外部主时钟**（直播场景）
- KrKr2 使用 **SYNC_DISCON**（音频主时钟 + 不连续跳变修正）为默认策略
- **CDVDClock** 维护三层时间模型（系统时间→绝对时间→播放时间），支持暂停、变速、不连续跳变
- **ErrorAdjust** 是同步修正的核心：VSync 模式下按帧步进（±1帧），非 VSync 模式下直接跳变
- **CVideoPlayerAudio** 的三状态同步状态机（STARTING→WAITSYNC→INSYNC）确保启动时的平滑同步
- 音频时钟通过 PTS 精确更新 + duration 预测递增的组合方式维护

## 5.8 练习题与答案

### 题目 1：时间基转换

一个 MPEG-TS 视频流的 `time_base = {1, 90000}`，某帧的 PTS 为 `8100000`。请计算：
1. 该帧应该在第几秒显示？
2. 如果要转换到 `time_base = {1, 1000}`（毫秒），新的 PTS 值是多少？
3. 如果要转换到 KrKr2 的 `DVD_TIME_BASE`（1000000），新的 PTS 值是多少？

<details>
<summary>查看答案</summary>

1. **显示时间**：`8100000 * (1/90000) = 90.0 秒`

2. **转为毫秒时间基**：
   ```
   new_pts = 8100000 * (1/90000) / (1/1000) = 8100000 / 90 = 90000
   ```
   或使用 `av_rescale_q(8100000, {1,90000}, {1,1000}) = 90000`
   
   PTS = 90000（即 90000 毫秒 = 90 秒）

3. **转为 DVD_TIME_BASE**：
   ```
   new_pts = 8100000 * (1/90000) / (1/1000000) = 8100000 * 1000000 / 90000 = 90000000
   ```
   或 `av_rescale_q(8100000, {1,90000}, {1,1000000}) = 90000000`
   
   PTS = 90000000（即 90000000 微秒 = 90 秒）

</details>

### 题目 2：同步策略分析

阅读以下场景，判断应该使用哪种同步策略，并说明理由：

- 场景 A：用户在电脑上观看一部 2 小时的电影
- 场景 B：直播推流，需要与真实时间对齐
- 场景 C：视觉小说游戏中播放过场动画（KrKr2 的典型场景）

<details>
<summary>查看答案</summary>

- **场景 A：音频主时钟**
  - 理由：2 小时长时间播放，音频连续性最重要（中断音频比丢视频帧更令人不适）。声卡硬件时钟稳定，适合作为长期基准。视频偶尔丢帧用户几乎感知不到。
  
- **场景 B：外部主时钟**
  - 理由：直播需要与真实时间（墙钟）对齐，不能让任何一条流"自主"决定时间。音频和视频都需要根据墙钟独立调整速率。

- **场景 C：音频主时钟（SYNC_DISCON）**
  - 理由：视觉小说的过场动画通常很短（几十秒到几分钟），用户对画面和音频的配合非常敏感（对话配音对口型）。音频主时钟保证语音连续流畅。KrKr2 默认使用 SYNC_DISCON 正是因为这个原因。短时间播放不太会遇到严重的时钟漂移问题。

</details>

### 题目 3：CDVDClock::SystemToPlaying 公式推导

`CDVDClock::SystemToPlaying` 的核心公式是：

```
playing = DVD_TIME_BASE * (current - m_startClock + m_systemAdjust) / m_systemUsed + m_iDisc
```

请回答：
1. 当播放速度从 1.0x 变为 2.0x 时，`m_systemUsed` 如何变化？为什么这能实现 2 倍速？
2. `m_iDisc` 的作用是什么？什么情况下它会被修改？
3. 为什么暂停时使用 `m_pauseClock` 而不是简单地让 `m_systemUsed = 0`？

<details>
<summary>查看答案</summary>

1. **m_systemUsed 变化**：
   - 1.0x 速度时：`m_systemUsed = m_systemFrequency * 1000 / 1000 = m_systemFrequency`
   - 2.0x 速度时：`m_systemUsed = m_systemFrequency * 1000 / 2000 = m_systemFrequency / 2`
   
   当 `m_systemUsed` 减半时，同样的系统时间差 `(current - m_startClock)` 除以 `m_systemUsed` 后得到的播放时间变为 2 倍。即系统过了 1 秒，播放时间走了 2 秒。

2. **m_iDisc（discontinuity）的作用**：
   - 它是时钟跳变的基准偏移量。当调用 `Discontinuity(clock, absolute)` 时，`m_iDisc = clock`，同时 `m_startClock` 被重置为当前绝对时间。
   - 这使得 `SystemToPlaying` 在跳变后立即返回 `m_iDisc`（因为 `current - m_startClock ≈ 0`），从而实现时钟的瞬间跳转。
   - 修改场景：seek、同步误差修正（`ErrorAdjust`）、流切换、首次播放。

3. **为什么不用 m_systemUsed = 0**：
   - 如果 `m_systemUsed = 0`，公式中会出现除以零（division by zero），导致未定义行为。
   - 使用 `m_pauseClock` 是更优雅的方案：冻结 `current` 值为暂停时刻的系统时间，公式的分子 `(m_pauseClock - m_startClock)` 变为常量，播放时间自然停止增长。
   - 恢复播放时，通过 `m_startClock += current - m_pauseClock` 补偿暂停期间的系统时间流逝，使播放时间从暂停点无缝继续。

</details>
