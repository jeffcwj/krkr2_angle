## 为什么需要音视频同步

### 同步问题的本质

在一个视频播放器中，音频和视频是**两条完全独立的数据流**，各自由独立的线程解码和渲染：

```
┌─────────────────────────────────────────────────────────────┐
│                    播放器内部架构                              │
├──────────────────────┬──────────────────────────────────────┤
│     音频线程          │          视频线程                     │
│                      │                                      │
│  解码音频包 ────>     │     <──── 解码视频包                  │
│  送入声卡缓冲 ──>     │     <──── 渲染到屏幕                  │
│                      │                                      │
│  驱动：声卡硬件时钟    │     驱动：帧率 / VSync 信号           │
│  精度：采样级别       │     精度：帧级别                      │
│  (1/48000 = 20.8μs)  │     (1/30 = 33.3ms)                  │
└──────────────────────┴──────────────────────────────────────┘
```

问题在于：
- **音频输出速率**由声卡硬件时钟驱动。当采样率为 48000Hz 时，声卡每秒精确消耗 48000 个采样——这个速率由声卡晶振（Crystal Oscillator，一种高精度振荡电路）决定
- **视频输出速率**由帧率和 VSync（Vertical Synchronization，垂直同步信号——显示器每刷新一帧发出的信号）决定，通常每秒 24/30/60 帧
- 两者的时钟源是**完全不同的物理硬件**，即使各自完美按照 PTS 时间戳播放，长时间运行后**必然产生偏差**

举个具体例子：假设声卡标称采样率 48000Hz，实际晶振频率是 47998Hz（偏差仅 0.004%）。播放一小时后：
```
理论输出采样数: 48000 × 3600 = 172,800,000
实际输出采样数: 47998 × 3600 = 172,792,800
差值: 7,200 采样 = 0.15 秒 = 150 毫秒
```
仅仅 0.004% 的频率偏差，一小时后就积累了 150ms 的音视频偏差！

### 人耳对同步偏差的感知

ITU-R BT.1359-1（国际电信联盟的广播技术标准）对音视频同步偏差给出了如下感知阈值：

| 偏差范围 | 感知效果 | 技术要求 |
|----------|----------|----------|
| ±20ms 以内 | 完全感知不到 | 理想同步状态 |
| ±40ms | 敏感用户可能察觉嘴型对不上 | 专业播放器的目标上限 |
| ±80ms | 多数用户明显感知"声画不同步" | 必须触发同步修正 |
| ±120ms | 严重不同步，观看体验受损 | 必须立即修正或跳变 |
| ±200ms+ | 完全无法接受 | 视为同步失败 |

> **关键洞察：不对称感知**。人类对"音频领先于视频"比"视频领先于音频"更敏感。具体来说：
> - 音频提前 20ms → 可能被察觉（嘴巴还没动就听到声音）
> - 视频提前 40ms → 才开始被注意到（嘴巴先动后才听到声音）
> 
> 因此同步算法通常对音频领先的容忍度更低。KrKr2 的 `CDVDClock::ErrorAdjust` 中就体现了这一点：音频超前阈值 20ms vs 音频落后阈值 27ms（不对称设计）。

### 同步问题的常见症状与排查方向

| 症状 | 可能原因 | 排查方向 |
|------|---------|---------|
| 嘴型不对 | 音视频偏差超过 80ms | 检查主时钟选择、同步策略是否生效 |
| 视频卡顿但音频流畅 | 视频解码太慢跟不上音频时钟 | 检查丢帧逻辑、考虑硬件加速或降低分辨率 |
| 音频断续（噼啪声） | 音频缓冲区欠载（Underrun，即声卡需要数据时缓冲区已空） | 增大缓冲区、提高解码线程优先级 |
| 播放越久越不同步 | 时钟漂移（Drift，声卡实际频率与标称值的微小偏差） | 启用持续的漂移补偿（如 `CDVDClock::SetSpeedAdjust`） |
| Seek 后音画不同步 | Seek 后未正确重置同步状态 | Flush 解码器 + 重置同步状态机 + 用 Seek 目标重新设置时钟 |

## PTS、DTS 与时间基（time_base）

### PTS 和 DTS 在同步中的应用

在第一章中我们介绍了 PTS 和 DTS 的基本概念，这里深入讲解它们在同步中的具体作用：

- **PTS**（Presentation Time Stamp，显示时间戳）：帧应该被**显示/播放**的时间。同步算法的核心——决定"什么时候给用户看/听到这一帧"
- **DTS**（Decode Time Stamp，解码时间戳）：帧应该被**解码**的时间。主要用于解码器的输入调度，同步算法一般不直接使用

对于**没有 B 帧**的编码格式（如多数音频编码、MJPEG、无 B 帧的 H.264 Baseline Profile 等），PTS 总是等于 DTS。

对于**有 B 帧**的编码格式（如 H.264 Main/High Profile、H.265/HEVC），DTS ≤ PTS，因为 B 帧（Bidirectional Frame，双向预测帧——同时参考前后帧进行预测的帧类型）需要在参考帧之后解码，但在参考帧之间显示：

```
编码/解码顺序 (按 DTS 排列):  I0   P3   B1   B2   P6   B4   B5
显示顺序 (按 PTS 排列):       I0   B1   B2   P3   B4   B5   P6

帧号:  0    1    2    3    4    5    6
DTS:   0    1    2    3    4    5    6
PTS:   0    2    3    1    5    6    4
```

可以看到：P3 帧的 DTS=1（第二个解码），但 PTS=3（第四个显示）。这是因为 B1 和 B2 需要引用 P3 作为前向参考帧，所以 P3 必须先于 B1、B2 解码。

### 时间基（time_base）详解

FFmpeg 中的时间戳不是绝对时间（秒或毫秒），而是以**时间基**为单位的整数值。时间基是一个有理数（分数），定义了"一个时间刻度代表多少秒"：

```cpp
// AVRational 是 FFmpeg 定义的有理数类型，表示一个分数
typedef struct AVRational {
    int num;  // 分子 (numerator)
    int den;  // 分母 (denominator)
} AVRational;

// 不同容器格式使用不同的时间基
// MPEG-TS（数字电视传输流）：1 tick = 1/90000 秒 ≈ 11.1μs
AVStream->time_base = {1, 90000};

// Matroska/WebM（.mkv/.webm）：1 tick = 1/1000 秒 = 1ms
AVStream->time_base = {1, 1000};

// 某些音频流：1 tick = 1 个采样
AVStream->time_base = {1, 48000};  // 48kHz 音频

// MP4 视频流（常见）：1 tick = 1/帧率的整数倍
AVStream->time_base = {1, 12800};  // 25fps 的 512 倍
```

### 时间基转换函数

FFmpeg 提供了一组工具函数进行时间基转换：

```cpp
// av_q2d：将 AVRational 转为 double
// 计算 num / den，返回每个 tick 对应的秒数
double av_q2d(AVRational a);
// 例：av_q2d({1, 90000}) = 0.0000111...

// 将 PTS 转换为秒
double pts_seconds = frame->pts * av_q2d(stream->time_base);
// 例：PTS=4500000, time_base={1,90000}
//     → 4500000 × (1/90000) = 50.0 秒

// av_rescale_q：在两个时间基之间转换
// 等价于 a * bq / cq，但使用 128-bit 中间结果避免溢出
int64_t av_rescale_q(int64_t a, AVRational bq, AVRational cq);
// 例：将 MPEG-TS 的 PTS 转为毫秒时间基
// av_rescale_q(4500000, {1,90000}, {1,1000}) = 50000 (ms)

// av_rescale_q_rnd：带舍入模式的时间基转换
int64_t av_rescale_q_rnd(int64_t a, AVRational bq, AVRational cq,
                         enum AVRounding rnd);
// 舍入模式：AV_ROUND_NEAR_INF（四舍五入）、
//          AV_ROUND_DOWN（向下取整）、AV_ROUND_UP（向上取整）
```

### 实践：时间基转换演示程序

```cpp
// timebase_demo.cpp — 演示不同时间基之间的转换
#include <cstdio>

extern "C" {
#include <libavutil/rational.h>      // AVRational, av_q2d
#include <libavutil/mathematics.h>   // av_rescale_q
}

int main() {
    printf("===== 时间基转换演示 =====\n\n");

    // 场景 1：MPEG-TS 视频流，time_base = {1, 90000}
    AVRational mpeg_ts_tb = {1, 90000};
    int64_t video_pts = 4500000;

    double seconds = video_pts * av_q2d(mpeg_ts_tb);
    printf("场景 1：MPEG-TS 视频帧\n");
    printf("  time_base = {%d, %d}  (1 tick = %.6f 秒)\n",
           mpeg_ts_tb.num, mpeg_ts_tb.den, av_q2d(mpeg_ts_tb));
    printf("  PTS = %lld → %.3f 秒\n\n", (long long)video_pts, seconds);

    // 场景 2：转换到毫秒时间基
    AVRational ms_tb = {1, 1000};
    int64_t ms_pts = av_rescale_q(video_pts, mpeg_ts_tb, ms_tb);
    printf("场景 2：转换到毫秒时间基 {1, 1000}\n");
    printf("  PTS = %lld → %lld ms\n\n", (long long)video_pts,
           (long long)ms_pts);

    // 场景 3：转换到 DVD_TIME_BASE（微秒，KrKr2 使用的时间基）
    AVRational dvd_tb = {1, 1000000};  // DVD_TIME_BASE = 1000000
    int64_t dvd_pts = av_rescale_q(video_pts, mpeg_ts_tb, dvd_tb);
    printf("场景 3：转换到 DVD_TIME_BASE {1, 1000000}\n");
    printf("  PTS = %lld → %lld μs\n\n", (long long)video_pts,
           (long long)dvd_pts);

    // 场景 4：MKV 到 MPEG-TS 的转换
    AVRational mkv_tb = {1, 1000};
    int64_t mkv_pts = 50000;  // 50秒
    int64_t ts_pts = av_rescale_q(mkv_pts, mkv_tb, mpeg_ts_tb);
    printf("场景 4：MKV → MPEG-TS\n");
    printf("  MKV PTS = %lld (time_base {1,1000}) → ",
           (long long)mkv_pts);
    printf("MPEG-TS PTS = %lld (time_base {1,90000})\n\n",
           (long long)ts_pts);

    // 场景 5：AV_TIME_BASE（FFmpeg 内部时间基，微秒）
    // AV_TIME_BASE = 1000000, AV_TIME_BASE_Q = {1, 1000000}
    printf("场景 5：AV_TIME_BASE 和 DVD_TIME_BASE 的关系\n");
    printf("  AV_TIME_BASE = %d (FFmpeg 内部，微秒)\n", AV_TIME_BASE);
    printf("  DVD_TIME_BASE = 1000000 (KrKr2，微秒)\n");
    printf("  两者相等！所以 KrKr2 中 FFmpeg↔DVD 时间戳可以直接赋值\n\n");

    // 场景 6：计算帧持续时间
    double fps = 23.976;  // 电影常见帧率
    double frame_duration_sec = 1.0 / fps;
    int64_t frame_duration_us = (int64_t)(frame_duration_sec * 1000000);
    printf("场景 6：帧持续时间计算\n");
    printf("  帧率: %.3f fps\n", fps);
    printf("  帧持续时间: %.4f 秒 = %lld 微秒\n",
           frame_duration_sec, (long long)frame_duration_us);

    return 0;
}
```

### KrKr2 的时间戳转换

KrKr2 在 `Clock.h`（`cpp/core/movie/ffmpeg/Clock.h` 第 8-16 行）中定义了自己的时间系统，继承自 Kodi 的设计：

```cpp
// DVD_TIME_BASE 是 KrKr2 内部的时间单位：1 = 1微秒
#define DVD_TIME_BASE    1000000

// 无效时间戳哨兵值（类似 FFmpeg 的 AV_NOPTS_VALUE）
// 用于表示"此帧没有有效的时间戳"
#define DVD_NOPTS_VALUE  0xFFF0000000000000

// 时间单位转换宏
#define DVD_TIME_TO_MSEC(x)  ((int)((double)(x) * 1000 / DVD_TIME_BASE))
#define DVD_SEC_TO_TIME(x)   ((double)(x) * DVD_TIME_BASE)
#define DVD_MSEC_TO_TIME(x)  ((double)(x) * DVD_TIME_BASE / 1000)

// 播放速度定义
#define DVD_PLAYSPEED_PAUSE   0     // 暂停
#define DVD_PLAYSPEED_NORMAL  1000  // 正常速度 (1.0x)
// 500 = 0.5x 慢速，2000 = 2.0x 倍速
```

在解码器中，FFmpeg 时间戳到 KrKr2 时间戳的转换：

```cpp
// 来源：VideoCodecFFmpeg.cpp 第 599-602 行
// 将 KrKr2 内部时间戳（DVD_TIME_BASE 单位）转为 FFmpeg 时间戳
avpkt.dts = (dts == DVD_NOPTS_VALUE)
    ? AV_NOPTS_VALUE
    : dts / DVD_TIME_BASE * AV_TIME_BASE;  // 微秒 → 微秒
avpkt.pts = (pts == DVD_NOPTS_VALUE)
    ? AV_NOPTS_VALUE
    : pts / DVD_TIME_BASE * AV_TIME_BASE;  // 微秒 → 微秒
```

由于 `DVD_TIME_BASE == AV_TIME_BASE == 1000000`（两者恰好相等），这里实际上是一个恒等变换。但代码保留了除法和乘法以保持通用性——如果未来任一方修改了时间基，代码仍能正确工作。

## 三种主时钟策略

### 概述

音视频同步的核心决策是：**以谁为基准？** 有三种策略：

```
策略 1：音频主时钟              策略 2：视频主时钟           策略 3：外部主时钟
┌───────────────┐              ┌───────────────┐           ┌───────────────┐
│  声卡硬件时钟  │ ← 基准      │  视频帧率      │ ← 基准   │  系统墙钟      │ ← 基准
│  (晶振驱动)   │              │  (VSync 驱动)  │           │  (OS 时钟)    │
└───────┬───────┘              └───────┬───────┘           └───────┬───────┘
        │ 视频跟随                     │ 音频跟随                  │ 两者都跟随
        ▼                             ▼                          ▼
┌───────────────┐              ┌───────────────┐           ┌───────────────┐
│  视频线程调整  │              │  音频线程调整  │           │  两条线程各自   │
│  丢帧 / 重复帧 │              │  重采样变速    │           │  独立调整       │
└───────────────┘              └───────────────┘           └───────────────┘
```

### 音频主时钟（Audio Master Clock）—— 最常用

**原理**：以声卡的硬件时钟为基准。音频数据按固定速率输出到声卡（不做任何变速），视频根据当前音频播放位置调整自己的显示时机。

**优点**：
- 声卡硬件时钟非常稳定精确（晶振精度通常在 ±50ppm 以内）
- 人耳对音频中断/变速极其敏感——保证音频连续性和原始速率最重要
- 实现相对简单：只需要调整视频帧的显示时机

**缺点**：
- 视频可能需要丢帧（视频落后时）或重复帧（视频超前时）
- 丢帧会导致画面跳跃，虽然通常不明显但对专业视频编辑可能不可接受

**核心算法（伪代码）**：

```cpp
// 音频主时钟的视频同步——决定每帧如何处理
void render_video_frame(AVFrame *frame, double audio_clock) {
    // 计算视频帧的显示时间
    double video_pts = frame->pts * av_q2d(video_stream->time_base);

    // diff > 0 表示视频超前于音频，diff < 0 表示视频落后于音频
    double diff = video_pts - audio_clock;

    // 同步容差（40ms）——在此范围内认为已同步
    double sync_threshold = 0.04;

    if (diff > sync_threshold) {
        // 视频超前太多：延迟显示，等音频追上来
        double delay = diff;
        usleep((int)(delay * 1000000));
        display_frame(frame);
    } else if (diff < -sync_threshold) {
        // 视频落后太多：丢弃当前帧，尝试追赶音频
        // 不显示这帧，直接解码下一帧
        drop_frame(frame);
    } else {
        // 在容差范围内：正常显示
        display_frame(frame);
    }
}
```

### 视频主时钟（Video Master Clock）

**原理**：以视频帧率或 VSync 刷新率为基准。视频按固定帧率/刷新率严格显示，音频通过重采样（Resample，改变音频的采样率从而改变播放速度）来调整播放速率以匹配视频。

**优点**：
- 视频播放绝对流畅，没有丢帧或重复帧
- 适合对画面连续性要求极高的场景（如动画、游戏过场）

**缺点**：
- 音频重采样会引入微小的音质损失
- 如果偏差较大，变速播放可能被听出来（音调会变高或变低）
- 实现更复杂：需要实时重采样音频

**音频速率调整示例**：

```cpp
// 视频主时钟模式下，音频根据偏差调整播放速率
double calculate_audio_speed(double video_clock, double audio_clock) {
    double diff = audio_clock - video_clock;

    // 比例调整：偏差越大，调整幅度越大
    // 但限制在 ±5% 以内，避免音频变调太明显
    double speed_adjust = 1.0 - diff * 0.1;  // P 控制器
    if (speed_adjust < 0.95) speed_adjust = 0.95;  // 最慢 0.95x
    if (speed_adjust > 1.05) speed_adjust = 1.05;  // 最快 1.05x

    return speed_adjust;
}
```

### 外部主时钟（External/System Clock）

**原理**：使用系统墙钟（Wall Clock，即操作系统提供的单调递增时钟，如 Linux 的 `CLOCK_MONOTONIC`、Windows 的 `QueryPerformanceCounter`）作为基准。音频和视频都独立调整自己的播放速度来匹配系统时间。

**优点**：
- 适合直播、视频会议等需要与真实时间对齐的场景
- 两条流都不是"特权"方，不会因为一条流出问题而影响另一条

**缺点**：
- 实现复杂度最高：需要同时调整音频和视频
- 音频和视频都需要微量变速，两者都可能有质量损失

### 三种策略对比

| 维度 | 音频主时钟 | 视频主时钟 | 外部主时钟 |
|------|-----------|-----------|-----------|
| 基准 | 声卡硬件时钟 | 视频帧率/VSync | 系统墙钟 |
| 音频质量 | ★★★★★ 最佳 | ★★★★ 有微量变速 | ★★★★ 有微量变速 |
| 视频流畅度 | ★★★★ 可能丢帧 | ★★★★★ 最佳 | ★★★★ 可能丢帧 |
| 实现复杂度 | ★★ 简单 | ★★★ 中等 | ★★★★ 复杂 |
| 适用场景 | 电影/视频播放 | 动画/游戏 | 直播/视频会议 |
| 长时间稳定性 | ★★★★★ 最佳 | ★★★★ 好 | ★★★★ 好 |

### KrKr2 的同步策略选择

KrKr2 支持两种同步模式，在 `CVideoPlayerAudio` 中定义：

```cpp
// 来源：推断自 VideoPlayerAudio.cpp
enum SyncType {
    SYNC_DISCON   = 0,  // 不连续校正（Discontinuity Correction）
    SYNC_RESAMPLE = 1   // 重采样校正
};
```

| 模式 | 策略 | 描述 | 使用场景 |
|------|------|------|---------|
| `SYNC_DISCON` | 音频主时钟 | 检测到同步误差时通过时钟跳变修正。视频线程跟随跳变后的时钟调整显示。**默认模式** | 本地视频播放（KrKr2 的主要用例） |
| `SYNC_RESAMPLE` | 视频主时钟 | 音频通过重采样调整速率来匹配视频帧率 | 实时流（`hints.realtime == true`） |

```cpp
// 来源：VideoPlayerAudio.cpp 第 109-116 行
m_synctype = SYNC_DISCON;
m_setsynctype = SYNC_DISCON;

// 实时流使用重采样同步
if (hints.realtime)
    m_setsynctype = SYNC_RESAMPLE;
```

KrKr2 默认使用 `SYNC_DISCON`（音频主时钟 + 不连续跳变修正），这是因为视觉小说的过场动画通常较短（几十秒到几分钟），对语音配音与口型的精确同步要求很高。音频主时钟保证语音连续流畅不变速，而短时间播放也不太会遇到严重的时钟漂移问题。

## 动手实践：音频主时钟同步演示

### 目标

编写一个简化版的音视频同步演示程序（不需要 FFmpeg），模拟音频主时钟策略的核心决策逻辑：对每帧视频，根据音频时钟决定显示、丢弃还是等待。

### 完整代码

```cpp
// av_sync_demo.cpp — 音视频同步原理演示（纯 C++，无需 FFmpeg）
// 演示音频主时钟策略的核心决策逻辑
#include <cstdio>
#include <cmath>
#include <chrono>
#include <thread>
#include <random>

// 简化版时钟（模拟 CDVDClock 的核心功能）
class SimpleClock {
public:
    // 启动时钟
    void start() {
        m_startTime = now();
        m_pauseTime = 0;
        m_paused = false;
    }

    // 暂停/恢复
    void pause(bool p) {
        if (p && !m_paused) {
            m_pauseTime = now();  // 冻结当前时间
            m_paused = true;
        } else if (!p && m_paused) {
            // 恢复时，补偿暂停期间的时间流逝
            m_startTime += (now() - m_pauseTime);
            m_paused = false;
        }
    }

    // 获取当前播放时间（秒）
    double getClock() const {
        int64_t current = m_paused ? m_pauseTime : now();
        return (current - m_startTime) / 1e6;  // 微秒 → 秒
    }

    // 不连续跳变修正（模拟 CDVDClock::Discontinuity）
    void adjust(double seconds) {
        m_startTime -= (int64_t)(seconds * 1e6);
    }

private:
    static int64_t now() {
        auto t = std::chrono::steady_clock::now().time_since_epoch();
        return std::chrono::duration_cast<std::chrono::microseconds>(t).count();
    }

    int64_t m_startTime = 0;
    int64_t m_pauseTime = 0;
    bool m_paused = false;
};

// 模拟视频帧
struct VideoFrame {
    double pts;        // 显示时间（秒）
    int frameNumber;   // 帧序号
    bool isKeyFrame;   // 是否关键帧
};

// 音频主时钟同步决策器
class AudioMasterSync {
public:
    AudioMasterSync(double threshold = 0.04)  // 默认 40ms 容差
        : m_syncThreshold(threshold) {}

    enum Action { DISPLAY, SKIP, WAIT };

    // 决定视频帧的处理方式
    Action decide(const VideoFrame &frame, double audioClock) {
        double diff = frame.pts - audioClock;

        if (diff < -m_syncThreshold) {
            m_droppedFrames++;
            return SKIP;        // 视频落后 → 丢帧
        } else if (diff > m_syncThreshold) {
            m_waitMs = (int)(diff * 1000);
            return WAIT;        // 视频超前 → 等待
        } else {
            m_displayedFrames++;
            return DISPLAY;     // 在范围内 → 正常显示
        }
    }

    int getWaitMs() const { return m_waitMs; }
    int getDroppedFrames() const { return m_droppedFrames; }
    int getDisplayedFrames() const { return m_displayedFrames; }

private:
    double m_syncThreshold;
    int m_waitMs = 0;
    int m_droppedFrames = 0;
    int m_displayedFrames = 0;
};

int main() {
    SimpleClock clock;
    AudioMasterSync sync;
    std::mt19937 rng(42);
    std::uniform_int_distribution<int> jitter(-5, 5);  // ±5ms 抖动

    const int totalFrames = 150;  // 5 秒 @30fps
    const double fps = 30.0;

    printf("===== 音频主时钟同步演示 =====\n");
    printf("帧率: %.0f fps, 总帧数: %d, 同步容差: 40ms\n\n", fps, totalFrames);

    clock.start();

    for (int i = 0; i < totalFrames; i++) {
        VideoFrame frame;
        frame.pts = i / fps;
        frame.frameNumber = i;
        frame.isKeyFrame = (i % 30 == 0);  // 每 30 帧一个关键帧

        // 模拟音频时钟（加入随机抖动模拟真实场景）
        double audioClock = clock.getClock();

        auto action = sync.decide(frame, audioClock);

        switch (action) {
            case AudioMasterSync::DISPLAY:
                if (i % 30 == 0) {  // 每秒打印一次
                    printf("帧 #%03d [%.3fs]: 显示 (偏差=%.1fms)\n",
                           i, frame.pts, (frame.pts - audioClock) * 1000);
                }
                // 模拟渲染耗时（33ms ± 随机抖动）
                std::this_thread::sleep_for(
                    std::chrono::milliseconds(33 + jitter(rng)));
                break;

            case AudioMasterSync::SKIP:
                printf("帧 #%03d [%.3fs]: ★丢弃★ (落后 %.1fms)\n",
                       i, frame.pts, (audioClock - frame.pts) * 1000);
                break;

            case AudioMasterSync::WAIT: {
                int waitMs = sync.getWaitMs();
                if (waitMs > 50) {
                    printf("帧 #%03d [%.3fs]: 等待 %dms 后显示\n",
                           i, frame.pts, waitMs);
                }
                std::this_thread::sleep_for(std::chrono::milliseconds(waitMs));
                break;
            }
        }
    }

    printf("\n===== 同步统计 =====\n");
    printf("总帧数:   %d\n", totalFrames);
    printf("已显示:   %d\n", sync.getDisplayedFrames());
    printf("已丢弃:   %d\n", sync.getDroppedFrames());
    printf("实际时长: %.3f 秒 (期望 %.3f 秒)\n",
           clock.getClock(), totalFrames / fps);
    printf("偏差:     %.1f ms\n",
           (clock.getClock() - totalFrames / fps) * 1000);

    return 0;
}
```

## 实践二：时间基转换工具

### 完整代码

```cpp
// timebase_tool.cpp — 分析媒体文件中各流的时间基和 PTS 分布
#include <cstdio>
#include <cstdlib>

extern "C" {
#include <libavformat/avformat.h>
#include <libavutil/mathematics.h>
}

int main(int argc, char *argv[]) {
    if (argc < 2) {
        fprintf(stderr, "用法: %s <媒体文件>\n", argv[0]);
        fprintf(stderr, "示例: %s test.mp4\n", argv[0]);
        return 1;
    }

    AVFormatContext *fmtCtx = nullptr;
    int ret = avformat_open_input(&fmtCtx, argv[1], nullptr, nullptr);
    if (ret < 0) {
        char buf[256];
        av_strerror(ret, buf, sizeof(buf));
        fprintf(stderr, "无法打开文件: %s\n", buf);
        return 1;
    }
    avformat_find_stream_info(fmtCtx, nullptr);

    // 打印每条流的时间基信息
    printf("===== 流时间基信息 =====\n");
    for (unsigned i = 0; i < fmtCtx->nb_streams; i++) {
        AVStream *st = fmtCtx->streams[i];
        const char *type = av_get_media_type_string(st->codecpar->codec_type);
        printf("\n流 #%u (%s):\n", i, type ? type : "未知");
        printf("  time_base:  {%d, %d}  → 1 tick = %.6f 秒\n",
               st->time_base.num, st->time_base.den,
               av_q2d(st->time_base));
        if (st->duration != AV_NOPTS_VALUE) {
            printf("  duration:   %lld ticks = %.3f 秒\n",
                   (long long)st->duration,
                   st->duration * av_q2d(st->time_base));
        }
        if (st->start_time != AV_NOPTS_VALUE) {
            printf("  start_time: %lld ticks = %.3f 秒\n",
                   (long long)st->start_time,
                   st->start_time * av_q2d(st->time_base));
        }

        // 转换到 DVD_TIME_BASE 演示
        if (st->duration != AV_NOPTS_VALUE) {
            AVRational dvd_tb = {1, 1000000};
            int64_t dvd_dur = av_rescale_q(st->duration, st->time_base, dvd_tb);
            printf("  DVD_TIME:   %lld μs\n", (long long)dvd_dur);
        }
    }

    // 读取前 20 个包，展示 PTS/DTS 分布
    printf("\n===== 前 20 个包的时间戳 =====\n");
    printf("%-4s %-6s %12s %10s %12s %10s %8s\n",
           "#", "类型", "PTS(tick)", "PTS(秒)", "DTS(tick)", "DTS(秒)", "大小");

    AVPacket *pkt = av_packet_alloc();
    int count = 0;
    while (av_read_frame(fmtCtx, pkt) >= 0 && count < 20) {
        AVStream *st = fmtCtx->streams[pkt->stream_index];
        const char *type = av_get_media_type_string(st->codecpar->codec_type);

        double ptsSec = (pkt->pts != AV_NOPTS_VALUE)
            ? pkt->pts * av_q2d(st->time_base) : -1.0;
        double dtsSec = (pkt->dts != AV_NOPTS_VALUE)
            ? pkt->dts * av_q2d(st->time_base) : -1.0;

        printf("%-4d %-6s %12lld %10.3f %12lld %10.3f %8d\n",
               count, type ? type : "?",
               (long long)pkt->pts, ptsSec,
               (long long)pkt->dts, dtsSec,
               pkt->size);

        av_packet_unref(pkt);
        count++;
    }

    av_packet_free(&pkt);
    avformat_close_input(&fmtCtx);
    return 0;
}
```

## 常见错误及解决方案

### 错误 1：时间基转换精度丢失

**现象**：经过多次时间基转换后，PTS 产生舍入误差，累积后导致同步偏差。

**原因**：`av_rescale_q` 默认使用向零舍入，某些情况下会丢失精度。

```cpp
// ❌ 错误：反复转换可能累积误差
int64_t pts_ms = av_rescale_q(pts, src_tb, {1, 1000});    // 90000 → 1000
int64_t pts_back = av_rescale_q(pts_ms, {1, 1000}, src_tb); // 1000 → 90000
// pts_back 可能 != pts！

// ✅ 正确：尽量减少转换次数，或用 av_rescale_q_rnd 指定舍入模式
int64_t pts_ms = av_rescale_q_rnd(pts, src_tb, {1, 1000},
    AV_ROUND_NEAR_INF | AV_ROUND_PASS_MINMAX);
```

### 错误 2：忽略 AV_NOPTS_VALUE

**现象**：某些帧的 PTS 为 `AV_NOPTS_VALUE`（表示无效），但程序直接用于计算，得到巨大的错误时间值。

```cpp
// ❌ 错误：不检查就使用
double pts_sec = frame->pts * av_q2d(stream->time_base);
// 如果 frame->pts == AV_NOPTS_VALUE (0x8000000000000000)
// 结果将是一个巨大的负数！

// ✅ 正确：先检查有效性
if (frame->pts != AV_NOPTS_VALUE) {
    double pts_sec = frame->pts * av_q2d(stream->time_base);
    // 使用 pts_sec ...
} else {
    // 使用上一帧 PTS + 帧持续时间来估算
    double pts_sec = last_pts + frame_duration;
}
```

### 错误 3：混淆流时间基和解码器时间基

**现象**：从 `AVCodecContext.time_base` 获取时间基用于 PTS 转换，得到错误结果。

```cpp
// ❌ 错误：使用解码器时间基
double pts_sec = frame->pts * av_q2d(codec_ctx->time_base);
// codec_ctx->time_base 通常是 {1, fps}，不是包/帧的时间基！

// ✅ 正确：使用流时间基
double pts_sec = frame->pts * av_q2d(stream->time_base);
// stream->time_base 才是 av_read_frame 返回的 PTS 使用的时间基
```

## 本节小结

- **音视频同步**的核心问题：音频和视频由不同的物理硬件时钟驱动（声卡晶振 vs 显示器刷新），长时间运行后必然产生漂移
- **人类感知阈值**：±40ms 以内才算可接受；音频领先比音频落后更容易被察觉（不对称感知）
- **PTS/DTS**：PTS 决定帧的显示/播放时间，DTS 决定帧的解码时间。有 B 帧时两者不同
- **时间基（time_base）**：FFmpeg 中时间戳是以时间基为单位的整数；不同容器格式使用不同时间基。用 `av_rescale_q` 在不同时间基之间转换
- **KrKr2 时间系统**：`DVD_TIME_BASE = 1000000`（微秒），恰好等于 FFmpeg 的 `AV_TIME_BASE`
- **三种主时钟策略**：音频主时钟（最常用，保证音频连续）、视频主时钟（保证画面流畅）、外部主时钟（直播场景）
- **KrKr2 默认使用 SYNC_DISCON**（音频主时钟 + 不连续跳变修正），适合视觉小说过场动画的短时间播放场景

## 练习题与答案

### 题目 1：时间基转换计算

一个 MPEG-TS 视频流的 `time_base = {1, 90000}`，某帧的 PTS 为 `8100000`。请计算：
1. 该帧应该在第几秒显示？
2. 如果要转换到 `time_base = {1, 1000}`（毫秒），新的 PTS 值是多少？
3. 如果要转换到 KrKr2 的 `DVD_TIME_BASE`（微秒，`{1, 1000000}`），新的 PTS 值是多少？

<details>
<summary>查看答案</summary>

1. **显示时间**：`8100000 × (1/90000) = 90.0 秒`

2. **转为毫秒时间基**：
   ```
   new_pts = av_rescale_q(8100000, {1,90000}, {1,1000})
           = 8100000 × 1000 / 90000
           = 90000
   ```
   PTS = 90000（即 90000 毫秒 = 90 秒）✓

3. **转为 DVD_TIME_BASE**：
   ```
   new_pts = av_rescale_q(8100000, {1,90000}, {1,1000000})
           = 8100000 × 1000000 / 90000
           = 90000000
   ```
   PTS = 90000000（即 90000000 微秒 = 90 秒）✓

</details>

### 题目 2：同步策略选择

判断以下场景应使用哪种主时钟策略，并说明理由：

- 场景 A：用户在电脑上观看一部 2 小时的电影
- 场景 B：直播推流，需要与真实时间对齐
- 场景 C：视觉小说游戏中播放过场动画（KrKr2 的典型场景）
- 场景 D：一个视频编辑软件的预览窗口

<details>
<summary>查看答案</summary>

- **场景 A：音频主时钟**
  - 理由：2 小时长时间播放，音频连续性最重要（中断或变速音频比丢视频帧更令人不适）。声卡硬件时钟稳定，适合作为长期基准。偶尔丢一帧视频几乎感知不到。

- **场景 B：外部主时钟**
  - 理由：直播需要与真实时间（墙钟）对齐——观众看到的直播画面必须是"现在"发生的事。音频和视频都需要独立根据系统时钟调整速率。

- **场景 C：音频主时钟（SYNC_DISCON）**
  - 理由：过场动画通常较短（几十秒到几分钟），用户对语音配音与口型的精确匹配非常敏感。音频主时钟保证语音不变速不中断。KrKr2 默认使用 `SYNC_DISCON` 正是这个原因。

- **场景 D：视频主时钟**
  - 理由：视频编辑软件的预览需要逐帧精确显示，画面连续性是最高优先级。音频可以容忍微量变速（编辑时通常频繁 Seek，音频质量本就不是首要关注点）。

</details>

### 题目 3：时钟漂移问题

某播放器使用音频主时钟策略。声卡标称采样率 44100Hz，实际频率 44102Hz。播放一个 2 小时的视频后：
1. 音频会比视频快还是慢？快/慢多少毫秒？
2. 如果不做漂移补偿，用户会有什么感受？
3. 提出一种修正方案。

<details>
<summary>查看答案</summary>

1. **音频更快**。计算过程：
   ```
   2 小时 = 7200 秒
   标称消耗采样数: 44100 × 7200 = 317,520,000
   实际消耗采样数: 44102 × 7200 = 317,534,400
   多消耗: 14,400 采样
   对应时间: 14400 / 44100 ≈ 0.3265 秒 ≈ 326.5 毫秒
   ```
   音频比视频快约 326.5 毫秒。

2. 用户感受：由于使用音频主时钟，视频会跟随音频加速。但 326ms 的累积偏差会导致视频需要频繁丢帧来追赶，用户可能在电影后半段感觉到偶尔的画面跳跃。如果同步算法没有持续的漂移补偿，偏差会持续增长。

3. **修正方案**：持续检测音频时钟与系统墙钟之间的漂移，用微量的 `speedAdjust`（速度微调）来补偿。KrKr2 中通过 `CDVDClock::SetSpeedAdjust()` 实现：它不是直接跳变时钟，而是让 `m_systemAdjust` 按 `m_speedAdjust × 时间增量` 缓慢累积，使播放时间"渐进"靠拢目标。这种微调幅度极小（通常 < 0.1%），人耳完全感知不到。

</details>
