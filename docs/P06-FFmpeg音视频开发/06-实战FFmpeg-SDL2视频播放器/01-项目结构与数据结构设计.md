## 核心数据结构：PacketQueue

在写播放器逻辑之前，先设计好数据结构。好的数据结构是程序成功的一半。

### 为什么需要 PacketQueue

播放器有两个独立的工作单元：
- **解封装线程**：不断从文件读取 `AVPacket`（生产者）
- **解码端**（音频回调 / 视频渲染）：从队列取出 `AVPacket` 进行解码（消费者）

这是经典的**生产者-消费者模型**（Producer-Consumer Pattern，一种多线程协作模式：生产者生成数据放入共享缓冲区，消费者从缓冲区取出数据处理）。需要一个线程安全的队列来连接它们。

```
解封装线程                PacketQueue              SDL音频回调
(生产者)                 (共享缓冲区)              (消费者)
   │                   ┌─────────────┐               │
   │── put(pkt) ──────▶│  pkt1 → pkt2 │──── get() ──▶│
   │                   │  → pkt3 → ...│               │
   │                   └─────────────┘               │
   │                   mutex + cond 保护              │
```

### PacketQueue 实现

```c
// ============================================================
// PacketQueue — 线程安全的 AVPacket FIFO 队列
// ============================================================

// 链表节点：每个节点包含一个 AVPacket 和指向下一个节点的指针
typedef struct PacketList {
    AVPacket         pkt;    // 存储的数据包（浅拷贝，拥有 data 所有权）
    struct PacketList *next;  // 下一个节点
} PacketList;

// 队列结构：包含头尾指针、统计信息、同步原语
typedef struct PacketQueue {
    PacketList  *first;        // 队列头（最早入队的包）
    PacketList  *last;         // 队列尾（最晚入队的包）
    int          nb_packets;   // 队列中包的数量
    int          size;         // 队列中所有包的总字节数（用于内存限制）
    SDL_mutex   *mutex;        // 互斥锁（保护并发访问）
    SDL_cond    *cond;         // 条件变量（实现阻塞等待）
    int          abort_request; // 退出标志（1=中止，唤醒所有等待者）
} PacketQueue;

// 初始化队列
static void packet_queue_init(PacketQueue *q) {
    memset(q, 0, sizeof(PacketQueue));  // 全部清零
    q->mutex = SDL_CreateMutex();       // 创建互斥锁
    q->cond  = SDL_CreateCond();        // 创建条件变量
    // nb_packets, size, abort_request 均为 0
    // first, last 均为 NULL（空队列）
}

// 入队 —— 生产者调用（解封装线程）
// 返回 0 成功，-1 失败
static int packet_queue_put(PacketQueue *q, AVPacket *pkt) {
    // 分配链表节点
    PacketList *node = (PacketList *)av_malloc(sizeof(PacketList));
    if (!node) return -1;  // 内存分配失败

    node->pkt  = *pkt;    // 浅拷贝：转移 pkt->data 的所有权给队列
    node->next = NULL;

    SDL_LockMutex(q->mutex);  // 加锁

    // 追加到队列尾部
    if (!q->last) {
        q->first = node;      // 空队列：first 和 last 都指向新节点
    } else {
        q->last->next = node;  // 非空：追加到尾部
    }
    q->last = node;
    q->nb_packets++;
    q->size += pkt->size;      // 累加字节数（用于内存限制判断）

    SDL_CondSignal(q->cond);   // 通知等待的消费者："有新数据了"
    SDL_UnlockMutex(q->mutex); // 解锁
    return 0;
}

// 出队 —— 消费者调用（音频回调 / 视频渲染）
// block=1 时阻塞等待直到有数据；block=0 时立即返回
// 返回 1 成功取到包，0 队列为空（非阻塞），-1 已中止
static int packet_queue_get(PacketQueue *q, AVPacket *pkt, int block) {
    PacketList *node;
    int ret = -1;

    SDL_LockMutex(q->mutex);
    for (;;) {
        // 检查中止标志（退出时需要唤醒所有等待者）
        if (q->abort_request) {
            ret = -1;
            break;
        }

        // 尝试取出队列头部的节点
        node = q->first;
        if (node) {
            q->first = node->next;
            if (!q->first) q->last = NULL;  // 队列变空
            q->nb_packets--;
            q->size -= node->pkt.size;
            *pkt = node->pkt;    // 所有权转移给调用者
            av_free(node);       // 释放链表节点（不释放 pkt->data）
            ret = 1;
            break;
        } else if (!block) {
            ret = 0;             // 非阻塞模式：队列为空，直接返回
            break;
        } else {
            // 阻塞模式：队列为空，等待生产者放入新数据
            // SDL_CondWait 会原子地释放 mutex 并阻塞
            // 被唤醒后重新获取 mutex，回到 for 循环顶部
            SDL_CondWait(q->cond, q->mutex);
        }
    }
    SDL_UnlockMutex(q->mutex);
    return ret;
}

// 清空队列 —— 释放所有未消费的包
static void packet_queue_flush(PacketQueue *q) {
    PacketList *node, *next;
    SDL_LockMutex(q->mutex);
    for (node = q->first; node; node = next) {
        next = node->next;
        av_packet_unref(&node->pkt);  // 释放包数据
        av_free(node);                 // 释放节点
    }
    q->first = q->last = NULL;
    q->nb_packets = 0;
    q->size = 0;
    SDL_UnlockMutex(q->mutex);
}

// 销毁队列 —— 释放所有资源
static void packet_queue_destroy(PacketQueue *q) {
    packet_queue_flush(q);          // 先清空所有包
    SDL_DestroyMutex(q->mutex);     // 销毁互斥锁
    SDL_DestroyCond(q->cond);       // 销毁条件变量
}

// 中止队列 —— 退出时调用，唤醒阻塞的消费者
static void packet_queue_abort(PacketQueue *q) {
    SDL_LockMutex(q->mutex);
    q->abort_request = 1;
    SDL_CondSignal(q->cond);   // 唤醒可能正在 SDL_CondWait 的消费者
    SDL_UnlockMutex(q->mutex);
}
```

### 设计要点解析

**为什么用链表而不是环形缓冲区？**

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 链表 | 无大小限制、入队O(1) | 频繁 malloc/free | 包大小不均匀、队列长度不可预测 |
| 环形缓冲区 | 无动态分配、缓存友好 | 固定大小，需要预估 | 固定大小的元素（如 FrameQueue） |

视频包大小差异巨大（I帧（Intra Frame，关键帧，不依赖其他帧可独立解码）可达几百 KB，P帧（Predicted Frame，预测帧，依赖前一帧解码）可能只有几 KB），用链表更灵活。KrKr2 的 `CDVDMessageQueue` 也使用链表实现。

**为什么用 SDL 的同步原语而不是 C++ 的 std::mutex？**

因为我们的代码是 C 语言。即使用 C++，SDL 的同步原语在跨平台上更可靠——SDL 已经为每个平台做了最优实现（Windows 用 CRITICAL_SECTION，Linux 用 pthread_mutex）。

**`av_packet_unref` 只在 `flush` 中调用的原因？**

正常出队时，`*pkt = node->pkt` 将所有权转移给了调用者。调用者负责在使用完后调用 `av_packet_unref`。如果在 `get` 中就 unref，调用者拿到的就是已释放的数据——这是一个常见的所有权错误。

## 播放器全局状态：PlayerState

### 结构设计

```c
// ============================================================
// PlayerState — 播放器全局状态
// ============================================================

typedef struct PlayerState {
    // ── 格式上下文 ──
    AVFormatContext    *fmt_ctx;         // FFmpeg 的格式/容器上下文

    // ── 音频相关 ──
    int                 audio_stream_idx; // 音频流在 fmt_ctx->streams 中的索引
    AVCodecContext     *audio_codec_ctx;  // 音频解码器上下文
    PacketQueue         audio_queue;      // 音频包队列
    uint8_t             audio_buf[MAX_AUDIO_FRAME_SIZE]; // 解码后的 PCM 缓冲区
    unsigned int        audio_buf_size;   // 缓冲区中有效数据的字节数
    unsigned int        audio_buf_index;  // 当前消费到的位置
    AVFrame            *audio_frame;      // 复用的音频解码帧
    SwrContext         *swr_ctx;          // 音频重采样上下文

    // ── 视频相关 ──
    int                 video_stream_idx; // 视频流索引
    AVCodecContext     *video_codec_ctx;  // 视频解码器上下文
    PacketQueue         video_queue;      // 视频包队列
    struct SwsContext   *sws_ctx;         // 像素格式转换上下文（备用）

    // ── 同步时钟 ──
    double              audio_clock;      // 当前音频播放时间（秒）
                                          // 由音频回调更新，视频渲染读取

    // ── SDL 资源 ──
    SDL_Window         *window;           // 窗口
    SDL_Renderer       *renderer;         // 渲染器（GPU 加速）
    SDL_Texture        *texture;          // YUV 纹理（每帧更新）
    int                 video_width;      // 视频宽度（像素）
    int                 video_height;     // 视频高度（像素）

    // ── 控制标志 ──
    int                 quit;             // 退出标志（1=退出）
    int                 paused;           // 暂停标志（1=暂停）
    SDL_Thread         *demux_tid;        // 解封装线程句柄
} PlayerState;

// 全局播放器实例
// 为什么用全局变量？SDL 的音频回调函数签名是
//   void callback(void *userdata, Uint8 *stream, int len)
// 虽然有 userdata 参数（我们会传入 PlayerState*），但为了代码简洁
// 我们同时保留一个全局指针。KrKr2 使用类成员变量（this 指针）替代。
static PlayerState *global_ps = NULL;
```

### 成员分组解析

PlayerState 的成员按功能分为 5 组：

```
PlayerState
├── 格式层（1 个成员）
│   └── fmt_ctx: 文件级别的信息（流列表、时长、格式）
│
├── 音频层（7 个成员）
│   ├── audio_stream_idx: 哪个流是音频
│   ├── audio_codec_ctx: 音频解码器
│   ├── audio_queue: 待解码的音频包
│   ├── audio_buf/size/index: 解码后的 PCM 缓冲区及读写位置
│   ├── audio_frame: 复用的解码帧（避免反复分配）
│   └── swr_ctx: 重采样器（将解码格式转为 SDL 需要的 S16）
│
├── 视频层（4 个成员）
│   ├── video_stream_idx: 哪个流是视频
│   ├── video_codec_ctx: 视频解码器
│   ├── video_queue: 待解码的视频包
│   └── sws_ctx: 像素格式转换器（非 YUV420P 时使用）
│
├── 同步层（1 个成员）
│   └── audio_clock: 音频播放进度（秒），作为 A/V 同步的主时钟
│
└── 表示层（6 个成员）
    ├── window/renderer/texture: SDL 的显示资源
    ├── video_width/height: 视频尺寸
    └── quit/paused/demux_tid: 控制逻辑
```

### 与 KrKr2 的对照

| 我们的 PlayerState 成员 | KrKr2 对应 | 差异说明 |
|------------------------|------------|---------|
| `fmt_ctx` | `CDVDDemuxFFmpeg::m_pFormatContext` | 相同，封装在 Demux 类中 |

### 实践 2：测试 PacketQueue 线程安全

```c
// queue_test.c — 测试 PacketQueue 的线程安全性
#include <stdio.h>
#include <string.h>
#include <SDL.h>
#include <libavformat/avformat.h>
#include <libavcodec/avcodec.h>

// [这里粘贴上面的 PacketQueue 实现]

// 生产者线程：写入 100 个模拟包
static int producer_thread(void *arg) {
    PacketQueue *q = (PacketQueue *)arg;
    for (int i = 0; i < 100; i++) {
        AVPacket pkt;
        av_init_packet(&pkt);
        pkt.data = NULL;
        pkt.size = i;  // 用 size 字段存储序号
        packet_queue_put(q, &pkt);
        SDL_Delay(1);  // 模拟解封装耗时
    }
    printf("生产者: 100 个包写入完毕\n");
    return 0;
}

// 消费者线程：读取所有包
static int consumer_thread(void *arg) {
    PacketQueue *q = (PacketQueue *)arg;
    int count = 0;
    AVPacket pkt;
    while (count < 100) {
        int ret = packet_queue_get(q, &pkt, 1);  // 阻塞等待
        if (ret < 0) break;  // 队列中止
        if (ret == 1) {
            count++;
            av_packet_unref(&pkt);
        }
    }
    printf("消费者: 收到 %d 个包\n", count);
    return 0;
}

int main() {
    SDL_Init(0);
    PacketQueue q;
    packet_queue_init(&q);

    SDL_Thread *prod = SDL_CreateThread(producer_thread, "prod", &q);
    SDL_Thread *cons = SDL_CreateThread(consumer_thread, "cons", &q);

    SDL_WaitThread(prod, NULL);
    SDL_WaitThread(cons, NULL);

    packet_queue_destroy(&q);
    SDL_Quit();
    printf("线程安全测试通过!\n");
    return 0;
}
```


```
原因：未正确链接 FFmpeg 库

排查步骤：
1. 确认 pkg-config 能找到库：
   pkg-config --libs libavformat  # 应输出 -lavformat
```

### 错误 3：AVPacket 所有权混乱

```
现象：播放一段时间后崩溃（double free / use after free）
原因：PacketQueue 的所有权语义没有正确理解

规则：
  packet_queue_put() → 接管 pkt->data 的所有权
    调用者不应再操作该 pkt
  packet_queue_get() → 将所有权转交给调用者
    调用者必须在用完后调用 av_packet_unref()
  packet_queue_flush() → 释放队列中所有未消费的包
    对每个包调用 av_packet_unref()
```

## 本节小结

- 一个完整的视频播放器由**解封装线程**、**音频解码回调**、**视频渲染主线程**三个并发单元组成
- **PacketQueue** 是连接生产者（解封装）和消费者（解码）的核心数据结构，使用 SDL 互斥锁和条件变量保证线程安全
- 所有权语义：`put` 接管、`get` 转交、`flush` 释放——理解所有权是避免内存错误的关键
- **PlayerState** 集中管理所有播放器状态，按格式层/音频层/视频层/同步层/表示层组织
- CMake 配置支持 pkg-config 和 vcpkg 两种方式，覆盖 Linux/macOS/Windows 三个桌面平台

## 练习题与答案

### 题目 1：PacketQueue 死锁分析

以下代码是否会导致死锁？为什么？

```c
// 线程 A
SDL_LockMutex(audio_queue.mutex);
packet_queue_get(&video_queue, &pkt, 1);  // 阻塞等待
SDL_UnlockMutex(audio_queue.mutex);

// 线程 B
SDL_LockMutex(video_queue.mutex);
packet_queue_get(&audio_queue, &pkt, 1);  // 阻塞等待
SDL_UnlockMutex(video_queue.mutex);
```

<details>
<summary>查看答案</summary>

**会死锁**。这是经典的**锁顺序死锁**（Lock Ordering Deadlock）：

- 线程 A 持有 `audio_queue.mutex`，等待获取 `video_queue.mutex`（在 `packet_queue_get` 内部）
- 线程 B 持有 `video_queue.mutex`，等待获取 `audio_queue.mutex`（在 `packet_queue_get` 内部）

两个线程互相等待对方释放锁，形成循环等待。

**解决方案**：永远不要在持有一个队列锁的情况下操作另一个队列。我们的播放器设计中没有这种情况——每个消费者只操作自己的队列（音频回调只操作 `audio_queue`，视频渲染只操作 `video_queue`）。

</details>

### 题目 2：为什么 audio_clock 不需要锁保护？

`audio_clock` 被音频回调线程写入、被主线程（视频渲染）读取。为什么代码中没有用锁保护？

<details>
<summary>查看答案</summary>

严格来说，`audio_clock` 是一个 `double`（8 字节），在 x86/x64 平台上对齐的 8 字节读写是原子的，不会出现"撕裂读"（读到半新半旧的值）。但这依赖于平台特性，不具有可移植性。

在我们的简单播放器中，不加锁是可以接受的，原因：
1. **单写者**：只有音频回调线程写入 `audio_clock`
2. **容忍瞬时不一致**：即使主线程读到稍旧的值，差异不超过一帧时间（~33ms），在 ±50ms 的同步阈值内
3. **简化设计**：加锁会增加音频回调的延迟，可能导致音频欠载

KrKr2 的 `CDVDClock` 使用了更安全的设计——通过 `m_systemAdjust` 和原子操作避免竞态条件。生产级代码应使用 `std::atomic<double>`（C++）或平台原子操作。

</details>

### 题目 3：扩展 PacketQueue 支持大小限制

修改 `packet_queue_put` 函数，当队列总大小超过 `max_size` 字节时阻塞生产者，直到消费者消费后空间释放。

<details>
<summary>查看答案</summary>

```c
// 修改 PacketQueue 结构，新增 max_size 字段
typedef struct PacketQueue {
    // ... 原有字段 ...
    int max_size;  // 最大字节数限制（0 = 无限制）
} PacketQueue;

// 修改 packet_queue_put
static int packet_queue_put(PacketQueue *q, AVPacket *pkt) {
    PacketList *node = (PacketList *)av_malloc(sizeof(PacketList));
    if (!node) return -1;

    node->pkt  = *pkt;
    node->next = NULL;

    SDL_LockMutex(q->mutex);

    // 新增：当队列超过大小限制时，等待消费者消费
    while (q->max_size > 0 && q->size >= q->max_size
           && !q->abort_request) {
        SDL_CondWait(q->cond, q->mutex);
    }

    if (q->abort_request) {
        av_free(node);
        av_packet_unref(pkt);
        SDL_UnlockMutex(q->mutex);
        return -1;
    }

    if (!q->last) {
        q->first = node;
    } else {
        q->last->next = node;
    }
    q->last = node;
    q->nb_packets++;
    q->size += pkt->size;

    SDL_CondSignal(q->cond);
    SDL_UnlockMutex(q->mutex);
    return 0;
}

// 修改 packet_queue_get，出队后也要通知生产者
static int packet_queue_get(PacketQueue *q, AVPacket *pkt, int block) {
    // ... 原有逻辑 ...
    // 在成功出队后，新增：
    if (ret == 1) {
        SDL_CondSignal(q->cond);  // 通知可能阻塞的生产者
    }
    // ...
}
```

这种双向等待模式（生产者满时等、消费者空时等）是生产者-消费者模型的完整实现。KrKr2 的解封装线程 `BasePlayer::Process()` 在 `VideoPlayer.cpp:581-585` 使用了类似策略——当队列满时 `Sleep(10)`。

</details>
