## 音频解码函数 audio_decode_frame

### 为什么需要单独的音频解码函数

在上一节中，我们设计了 `PacketQueue`（数据包队列）作为解封装线程和解码端之间的桥梁。音频解码函数 `audio_decode_frame` 是消费者端的核心——它从音频队列中取出压缩数据包（AVPacket，第3章讲过的 FFmpeg 压缩数据容器），送入解码器得到原始 PCM 帧（AVFrame，解码后的原始音频数据），再通过重采样转换为 SDL 音频设备期望的格式。

SDL2 的音频系统采用 **pull 模式**（拉取模式）：SDL 在音频设备需要数据时主动调用我们注册的回调函数，而不是我们主动向设备推送数据。这意味着我们必须提前准备好解码数据，让 SDL 随时能拉取。

### 完整实现

```c
// 最大音频帧大小（192000 字节 ≈ 1秒 48kHz 立体声 16bit 数据）
#define MAX_AUDIO_FRAME_SIZE 192000

// ============================================================
// audio_decode_frame —— 从音频队列取包、解码、重采样
// 返回值：重采样后的字节数，失败返回 -1
// ============================================================
static int audio_decode_frame(PlayerState *ps) {
    AVPacket pkt;         // 存放从队列取出的压缩数据包
    int data_size = 0;    // 最终输出的字节数

    for (;;) {
        // 第1步：从包队列取一个音频包（阻塞模式，block=1）
        // packet_queue_get 内部使用 SDL_CondWait 等待，
        // 当队列为空时线程会休眠，不浪费 CPU
        if (packet_queue_get(&ps->audio_queue, &pkt, 1) < 0) {
            return -1;  // 队列已中止（quit=1），通知调用方停止
        }

        // 第2步：将压缩包发送到解码器
        // avcodec_send_packet 将 AVPacket 送入解码器的输入缓冲区
        // 参数：
        //   ps->audio_codec_ctx —— 音频解码器上下文（第4章详细讲过）
        //   &pkt                —— 要解码的压缩数据包
        // 返回值：0 表示成功，负值表示错误
        int ret = avcodec_send_packet(ps->audio_codec_ctx, &pkt);
        av_packet_unref(&pkt);  // 送入解码器后立即释放包的引用
        if (ret < 0) {
            fprintf(stderr, "音频 avcodec_send_packet 错误: %d\n", ret);
            continue;  // 跳过这个坏包，尝试下一个
        }

        // 第3步：从解码器接收解码后的原始帧
        // avcodec_receive_frame 从解码器的输出缓冲区取出一帧 PCM 数据
        // 参数：
        //   ps->audio_codec_ctx —— 同一个解码器上下文
        //   ps->audio_frame     —— 用于接收解码结果的 AVFrame
        // 返回值：
        //   0           —— 成功取到一帧
        //   AVERROR(EAGAIN) —— 解码器需要更多输入数据
        //   其他负值    —— 解码错误
        ret = avcodec_receive_frame(ps->audio_codec_ctx, ps->audio_frame);
        if (ret == AVERROR(EAGAIN)) {
            continue;  // 解码器缓冲区内数据不够，需要再送一个包
        } else if (ret < 0) {
            fprintf(stderr, "音频 avcodec_receive_frame 错误: %d\n", ret);
            continue;
        }

        // 第4步：重采样 —— 将解码器输出格式转换为 SDL 需要的格式
        // 解码器输出的格式可能是：
        //   - AV_SAMPLE_FMT_FLTP（32位浮点平面格式，常见于 AAC）
        //   - AV_SAMPLE_FMT_S16P（16位整数平面格式，常见于 MP3）
        // SDL 需要的格式是：
        //   - AV_SAMPLE_FMT_S16（16位整数交错格式）
        //
        // swr_convert 执行格式转换和采样率转换
        // 参数：
        //   ps->swr_ctx   —— SwrContext*，重采样上下文（在 player_init 中初始化）
        //   &ps->audio_buf —— 输出缓冲区指针（uint8_t*）
        //   MAX_AUDIO_FRAME_SIZE / 4 —— 输出缓冲区最大样本数
        //   ps->audio_frame->data    —— 输入数据（解码器输出）
        //   ps->audio_frame->nb_samples —— 输入样本数
        // 返回值：实际输出的样本数，负值表示错误
        int out_samples = swr_convert(
            ps->swr_ctx,
            &ps->audio_buf,
            MAX_AUDIO_FRAME_SIZE / 4,
            (const uint8_t **)ps->audio_frame->data,
            ps->audio_frame->nb_samples
        );

        if (out_samples < 0) {
            fprintf(stderr, "swr_convert 失败\n");
            return -1;
        }

        // 第5步：计算输出字节数
        // 公式：字节数 = 样本数 × 声道数 × 每样本字节数
        // 我们输出双声道（2）、16位（2字节/样本），所以 × 2 × 2 = × 4
        data_size = out_samples * 2 * sizeof(int16_t);

        // 第6步：更新音频时钟（A/V 同步的基准）
        // audio_clock 记录"当前音频播放到了第几秒"
        if (ps->audio_frame->pts != AV_NOPTS_VALUE) {
            // pts 是以 time_base 为单位的整数时间戳
            // av_q2d(tb) 将 AVRational 转换为 double（例如 {1, 44100} → 0.0000227）
            // pts × time_base = 秒数
            AVRational tb = ps->fmt_ctx->streams[ps->audio_stream_idx]->time_base;
            ps->audio_clock = av_q2d(tb) * ps->audio_frame->pts;
        }
        // 加上本帧的播放时长，确保时钟在帧内也能递增
        // 时长 = 样本数 ÷ 采样率（例如 1024 ÷ 44100 ≈ 0.023秒）
        ps->audio_clock += (double)out_samples /
                           ps->audio_codec_ctx->sample_rate;

        av_frame_unref(ps->audio_frame);  // 释放帧数据，下次循环复用
        return data_size;
    }
}
```

### 关键设计要点

**1. 为什么用无限循环？**

一个 AVPacket 送入解码器后，不一定立即能输出 AVFrame。某些编解码器（如 AAC）需要积累多个包才能输出第一帧（编码器内部有"引导帧"概念）。因此我们用 `for(;;)` 循环不断送包，直到成功收到一帧。

**2. 交错格式 vs 平面格式**

```
交错格式（Interleaved，如 AV_SAMPLE_FMT_S16）：
  [L₁ R₁ L₂ R₂ L₃ R₃ ...]  ← SDL 需要这种格式

平面格式（Planar，如 AV_SAMPLE_FMT_FLTP）：
  data[0]: [L₁ L₂ L₃ ...]   ← 左声道单独存放
  data[1]: [R₁ R₂ R₃ ...]   ← 右声道单独存放
```

大多数解码器输出平面格式（处理效率更高），但音频设备需要交错格式。`swr_convert` 自动完成这个转换。

**3. 音频时钟更新**

```
时钟更新示意（44100Hz 采样率，每帧 1024 样本）：

  帧 1: pts=0     → clock = 0.000 + 1024/44100 = 0.023s
  帧 2: pts=1024  → clock = 0.023 + 1024/44100 = 0.046s
  帧 3: pts=2048  → clock = 0.046 + 1024/44100 = 0.069s
  ...
```

### KrKr2 对照

KrKr2 中的音频解码在 `CVideoPlayerAudio::Process()` 中实现（`cpp/core/movie/ffmpeg/VideoPlayerAudio.cpp`）。主要差异：

| 特性 | 简单播放器 | KrKr2 |
|------|-----------|-------|
| 解码 API | `send_packet`/`receive_frame`（新 API） | `avcodec_decode_audio4`（旧 API） |
| 时钟 | `double audio_clock` 直接赋值 | `CDVDClock` 三层时钟系统 |
| 重采样 | `swr_convert` 固定输出 S16/44100/stereo | 根据硬件能力动态选择输出格式 |
| 线程 | SDL 回调线程 | 独立的 `CThread` 线程 |

## SDL 音频回调 sdl_audio_callback

### pull 模式的工作原理

SDL2 音频采用 **pull 模式**（拉取模式）——当音频设备的硬件缓冲区需要新数据时，SDL 会自动调用我们注册的回调函数。这与 **push 模式**（我们主动提交数据，如 OpenAL）形成对比。

```
Pull 模式数据流：

  音频硬件 ──"我需要数据"──→ SDL ──调用──→ sdl_audio_callback()
                                              │
                                              ├─ 缓冲区有数据？ → memcpy 到 stream
                                              └─ 缓冲区空了？   → audio_decode_frame()
                                                                    │
                                                                    └─→ packet_queue_get()
                                                                          （阻塞等待解封装线程）
```

**关键特性**：回调函数运行在 SDL 创建的**独立音频线程**中，不在主线程。因此对 `PlayerState` 共享数据的访问需要注意线程安全。

### 完整实现

```c
// ============================================================
// sdl_audio_callback —— SDL 音频设备回调
// 参数：
//   userdata —— 用户数据指针，我们传入 PlayerState*
//   stream   —— SDL 提供的输出缓冲区，我们需要填满它
//   len      —— SDL 期望我们填充的字节数
// ============================================================
static void sdl_audio_callback(void *userdata, Uint8 *stream, int len) {
    PlayerState *ps = (PlayerState *)userdata;
    int len1, audio_size;

    // SDL 要求我们填满整个 stream 缓冲区（len 字节）
    // 如果我们的解码缓冲区不够，需要多次调用 audio_decode_frame
    while (len > 0) {
        // 检查当前解码缓冲区是否还有未消费的数据
        if (ps->audio_buf_index >= ps->audio_buf_size) {
            // 缓冲区已用完，解码新的一帧
            audio_size = audio_decode_frame(ps);
            if (audio_size < 0) {
                // 解码失败或队列已中止 → 填充静音数据
                // 静音 = 全零（对于有符号 16 位格式，0 就是静音）
                memset(stream, 0, len);
                break;
            }
            ps->audio_buf_size = audio_size;  // 记录新数据的大小
            ps->audio_buf_index = 0;          // 重置消费位置
        }

        // 计算本次能复制多少字节
        len1 = ps->audio_buf_size - ps->audio_buf_index;
        if (len1 > len) len1 = len;  // 不超过 SDL 需要的量

        // 将解码数据复制到 SDL 的输出缓冲区
        memcpy(stream, ps->audio_buf + ps->audio_buf_index, len1);

        // 更新指针和剩余量
        len -= len1;
        stream += len1;
        ps->audio_buf_index += len1;
    }
}
```

### 缓冲区消费示意

```
假设 SDL 每次请求 4096 字节，audio_decode_frame 每次返回 8192 字节：

  第1次回调（len=4096）：
    audio_buf: [████████████████]  buf_size=8192, buf_index=0
    复制 4096 字节 → buf_index=4096

  第2次回调（len=4096）：
    audio_buf: [        ████████]  buf_size=8192, buf_index=4096
    复制 4096 字节 → buf_index=8192

  第3次回调（len=4096）：
    audio_buf: [                ]  buf_index >= buf_size → 调用 audio_decode_frame
    解码新帧 → buf_size=8192, buf_index=0
    复制 4096 字节 → buf_index=4096
```

### 常见错误与排查

| 现象 | 原因 | 解决方案 |
|------|------|----------|
| 播放有杂音/爆音 | 重采样参数不匹配 | 检查 `swr_alloc_set_opts` 的输入输出格式是否正确 |
| 只有一个声道有声 | 声道布局（channel layout）错误 | 确认输出为 `AV_CH_LAYOUT_STEREO` |
| 播放速度异常 | 采样率不匹配 | SDL 打开的采样率必须与 `swr_convert` 输出采样率一致 |
| 开头有短暂杂音 | 缓冲区未初始化 | 分配 `audio_buf` 后用 `memset(0)` 清零 |
| 音频断断续续 | 解码速度跟不上消费速度 | 增大包队列上限，或降低音频质量 |
| 静音无声 | `SDL_PauseAudio(0)` 未调用 | 打开音频设备后必须调用此函数开始播放 |

## 视频解码与渲染 video_refresh

### 定时器驱动的视频渲染

与音频不同，视频渲染在主线程的事件循环中以定时器驱动。每次定时器触发时，`video_refresh` 从视频队列取包、解码、执行 A/V 同步判断、渲染到 SDL 窗口。

**为什么视频在主线程？** SDL2 规定所有渲染 API（`SDL_RenderCopy`、`SDL_RenderPresent` 等）必须在创建窗口的线程中调用。这是因为 GPU 驱动通常将 OpenGL/DirectX 上下文绑定到特定线程。

### 完整实现

```c
// ============================================================
// video_refresh —— 解码一帧视频并渲染到 SDL 纹理
// 在主线程的事件循环中由定时器事件触发
// ============================================================
static void video_refresh(PlayerState *ps) {
    AVPacket pkt;
    AVFrame *frame = av_frame_alloc();      // 分配解码输出帧
    AVFrame *frame_rgb = av_frame_alloc();  // 备用 RGB 帧（非 YUV420P 时使用）
    int got_frame = 0;

    if (!frame || !frame_rgb) return;

    // 从视频队列取包并解码（非阻塞模式，block=0）
    // 非阻塞意味着队列为空时立即返回，不会卡住主线程
    while (packet_queue_get(&ps->video_queue, &pkt, 0) > 0) {
        // 将压缩包送入视频解码器
        int ret = avcodec_send_packet(ps->video_codec_ctx, &pkt);
        av_packet_unref(&pkt);
        if (ret < 0) continue;

        // 接收解码后的视频帧
        ret = avcodec_receive_frame(ps->video_codec_ctx, frame);
        if (ret == AVERROR(EAGAIN)) continue;  // 需要更多输入
        if (ret < 0) break;                     // 解码错误

        got_frame = 1;

        // ────── A/V 同步判断 ──────
        // 计算当前视频帧的显示时间（秒）
        double pts = 0.0;
        if (frame->pts != AV_NOPTS_VALUE) {
            AVRational tb =
                ps->fmt_ctx->streams[ps->video_stream_idx]->time_base;
            pts = av_q2d(tb) * frame->pts;
        }

        // 计算视频与音频的时间差
        // diff > 0：视频超前于音频（视频帧应该在未来显示）
        // diff < 0：视频落后于音频（视频帧应该早就显示了）
        double diff = pts - ps->audio_clock;

        if (diff > 0.05) {
            // 视频超前音频 50ms 以上 → 跳过本次渲染
            // 等下次定时器触发时再检查
            av_frame_unref(frame);
            got_frame = 0;
            continue;
        } else if (diff < -0.05) {
            // 视频落后音频 50ms 以上 → 丢弃这一帧
            // 继续取下一帧，尝试追赶音频
            av_frame_unref(frame);
            got_frame = 0;
            continue;
        }

        // diff 在 [-50ms, +50ms] 范围内 → 正常渲染
        break;
    }

    if (!got_frame) {
        av_frame_free(&frame);
        av_frame_free(&frame_rgb);
        return;  // 没有可渲染的帧
    }

    // ────── YUV 纹理更新（零拷贝渲染）──────
    // SDL_UpdateYUVTexture 直接将 YUV420P 数据上传到 GPU 纹理
    // GPU 着色器在显示时自动完成 YUV→RGB 转换
    // 参数：
    //   ps->texture     —— SDL 纹理（创建时指定了 SDL_PIXELFORMAT_IYUV）
    //   NULL            —— 更新整个纹理（也可传 SDL_Rect* 更新局部）
    //   frame->data[0]  —— Y 平面（亮度，每像素 1 字节）
    //   frame->linesize[0] —— Y 平面每行字节数（可能 > width，因为对齐）
    //   frame->data[1]  —— U 平面（色度蓝差，宽高各为 Y 的 1/2）
    //   frame->linesize[1] —— U 平面每行字节数
    //   frame->data[2]  —— V 平面（色度红差，宽高各为 Y 的 1/2）
    //   frame->linesize[2] —— V 平面每行字节数
    SDL_UpdateYUVTexture(
        ps->texture, NULL,
        frame->data[0], frame->linesize[0],
        frame->data[1], frame->linesize[1],
        frame->data[2], frame->linesize[2]
    );

    // 渲染流程：清除 → 复制纹理 → 呈现
    SDL_RenderClear(ps->renderer);     // 清除上一帧
    SDL_RenderCopy(ps->renderer, ps->texture, NULL, NULL);  // 纹理 → 渲染目标
    SDL_RenderPresent(ps->renderer);   // 提交给显示器

    // 释放帧资源
    av_frame_free(&frame);
    av_frame_free(&frame_rgb);
}
```

### A/V 同步策略详解

我们采用**音频主时钟**策略（与 KrKr2 项目一致）——以音频播放位置为基准，调节视频的显示节奏：

```
┌───────────────────────────────────────────────────────┐
│              A/V 同步决策流程                           │
│                                                        │
│  video_pts   = 当前视频帧的 PTS（秒）                   │
│  audio_clock = 当前音频播放位置（秒）                    │
│  diff = video_pts - audio_clock                        │
│                                                        │
│  ┌─ diff > +0.05 ─→ 视频超前 → 延迟渲染（等下一次）     │
│  ├─ diff < -0.05 ─→ 视频落后 → 丢帧追赶               │
│  └─ |diff| ≤ 0.05 ─→ 同步范围 → 正常渲染              │
└───────────────────────────────────────────────────────┘
```

**为什么选择 50ms 阈值？**

- 人眼对音画不同步的感知阈值约为 **80-100ms**
- 留 30-50ms 余量确保用户几乎感觉不到延迟
- 阈值太小（如 10ms）会导致频繁丢帧/延迟，画面卡顿
- 阈值太大（如 200ms）则同步效果差，用户能感知音画不一致

### 为什么用 SDL_UpdateYUVTexture 而不是 sws_scale

```
方案 A（CPU 转换，慢）：
  YUV帧 ─→ sws_scale(YUV→RGB) ─→ SDL_UpdateTexture(RGB)
           ↑ CPU 消耗大            ↑ 数据量是 YUV 的 2 倍

方案 B（GPU 转换，快，我们采用）：
  YUV帧 ─→ SDL_UpdateYUVTexture(直传 YUV) ─→ GPU 着色器自动转 RGB
           ↑ 零格式转换开销                    ↑ GPU 并行计算，极快
```

方案 B 比方案 A **快 3-5 倍**。KrKr2 项目中的 `TVPYUVSprite`（见 `KRMoviePlayer.cpp`）也采用了相同策略——直接将 YUV 数据传递给 GPU。

### 常见视频渲染问题

| 现象 | 原因 | 解决方案 |
|------|------|----------|
| 画面全绿 | YUV 数据错误或 linesize 不匹配 | 确认 `frame->linesize` 与视频宽度的关系 |
| 画面倒置 | 某些编码器输出底部向上 | 检查 `frame->linesize[0]` 是否为负值 |
| 画面拉伸变形 | SDL 窗口与视频宽高比不一致 | 用 `SDL_RenderSetLogicalSize` 设定逻辑尺寸 |
| 画面撕裂（上下不同步） | 未启用 VSync（垂直同步） | 创建 renderer 时加 `SDL_RENDERER_PRESENTVSYNC` |
| 画面卡顿 | CPU 解码速度不够 | 降低视频分辨率，或使用硬件加速解码 |

## 解封装线程 demux_thread

### 线程的角色：数据泵

解封装线程是播放器的"数据泵"（data pump），它持续从媒体文件中读取压缩数据包，并按流类型分发到对应的队列。这个线程是生产者-消费者模型中的**生产者**。

```
┌──────────┐    av_read_frame     ┌──────────────┐   packet_queue_put   ┌──────────────┐
│ 媒体文件  │ ──────────────────→ │  解封装线程    │ ──────────────────→ │  audio_queue  │
│ (MP4/MKV) │                     │ (demux_thread)│ ──────────────────→ │  video_queue  │
└──────────┘                      └──────────────┘                      └──────────────┘
```

### 完整实现

```c
// ============================================================
// demux_thread —— 解封装线程函数
// 从容器文件读取数据包，按流索引分发到音视频队列
// 参数：arg —— PlayerState* 的 void* 形式
// 返回值：0 表示正常退出
// ============================================================
static int demux_thread(void *arg) {
    PlayerState *ps = (PlayerState *)arg;
    AVPacket pkt;  // 读取到的数据包

    while (!ps->quit) {
        // ── 暂停处理 ──
        // 暂停时不读取新数据，但线程不退出
        if (ps->paused) {
            SDL_Delay(10);  // 休眠 10ms 避免 CPU 空转
            continue;
        }

        // ── 队列容量保护（防止内存爆炸）──
        // 当音视频队列总大小超过 15MB 时，暂停读取
        // 等待消费端（解码器）消化数据
        if (ps->audio_queue.size + ps->video_queue.size
            > 15 * 1024 * 1024) {
            SDL_Delay(10);
            continue;
        }

        // ── 读取一个数据包 ──
        // av_read_frame 从容器中读取下一个数据包
        // 返回值：0 表示成功，负值表示错误或 EOF
        int ret = av_read_frame(ps->fmt_ctx, &pkt);
        if (ret < 0) {
            if (ret == AVERROR_EOF || avio_feof(ps->fmt_ctx->pb)) {
                // 文件结束 → 发送 flush 包通知解码器输出缓存帧
                // flush 包 = data 为 NULL、size 为 0 的空包
                // FFmpeg 收到 NULL 包后进入 "drain mode"（排空模式），
                // 会输出编码器内部缓存的最后几帧
                if (ps->video_stream_idx >= 0) {
                    AVPacket flush_pkt;
                    av_init_packet(&flush_pkt);
                    flush_pkt.data = NULL;
                    flush_pkt.size = 0;
                    packet_queue_put(&ps->video_queue, &flush_pkt);
                }
                if (ps->audio_stream_idx >= 0) {
                    AVPacket flush_pkt;
                    av_init_packet(&flush_pkt);
                    flush_pkt.data = NULL;
                    flush_pkt.size = 0;
                    packet_queue_put(&ps->audio_queue, &flush_pkt);
                }
                break;  // 文件读完，退出线程
            }
            // 其他读取错误（如磁盘 I/O 错误），等待后重试
            SDL_Delay(10);
            continue;
        }

        // ── 按流索引分发数据包 ──
        if (pkt.stream_index == ps->audio_stream_idx) {
            packet_queue_put(&ps->audio_queue, &pkt);
        } else if (pkt.stream_index == ps->video_stream_idx) {
            packet_queue_put(&ps->video_queue, &pkt);
        } else {
            // 其他流（字幕、附件等），直接释放
            av_packet_unref(&pkt);
        }
    }

    return 0;
}
```

### 设计要点详解

**1. 队列容量限制（15MB）**

为什么是 15MB？这是一个经验值，平衡了以下因素：

```
太小（如 1MB）：
  → 高码率视频（20Mbps）只能缓存 0.4秒
  → 解码端稍有波动就会饥饿（buffer underrun）

太大（如 100MB）：
  → 内存占用过高
  → seek 时需要丢弃大量缓存数据，响应慢

15MB（经验值）：
  → 常见码率（5-10Mbps）可缓存 10-25秒
  → 足够应对解码端的短暂波动
  → 内存开销可接受
```

KrKr2 中 `BasePlayer::Process()`（`VideoPlayer.cpp` 第 581-585 行）使用类似策略——当音视频队列都满时 `Sleep(10)`。

**2. EOF 与 flush 包**

文件读完不代表播放结束。解码器内部可能还缓存着最后几帧（B 帧（双向预测帧，需要参考前后帧才能解码）需要等后续帧到达才能输出）。发送 NULL 数据包触发 FFmpeg 的 **drain mode**，让解码器输出所有缓存帧。

**3. 暂停处理的简化**

我们用 `SDL_Delay(10)` 轮询（polling）等待暂停结束。生产级代码应使用**条件变量**（condition variable，一种线程同步机制，让线程在条件不满足时休眠，满足时被唤醒）阻塞等待，避免 CPU 空转：

```c
// 更好的暂停处理（使用条件变量）
SDL_LockMutex(ps->pause_mutex);
while (ps->paused && !ps->quit) {
    SDL_CondWait(ps->pause_cond, ps->pause_mutex);
}
SDL_UnlockMutex(ps->pause_mutex);
```

### 线程安全分析

```
┌─────────────────────────────────────────────────────┐
│                    线程模型                           │
│                                                      │
│  解封装线程 ──put──→ [audio_queue] ──get──→ SDL回调线程│
│            ──put──→ [video_queue] ──get──→ 主线程     │
│                                                      │
│  共享状态访问：                                        │
│    ps->quit     → int 原子读写（单写多读）             │
│    ps->paused   → int 原子读写（单写多读）             │
│    ps->audio_clock → 仅 SDL 回调线程写，主线程读       │
│                     （单写者模式，可容忍瞬时不一致）    │
│                                                      │
│  线程安全保证：                                        │
│    PacketQueue  → SDL_mutex + SDL_cond 完全保护       │
│    SDL 渲染API  → 仅主线程调用（SDL 硬性要求）         │
│    FFmpeg 解码  → 每个 codec_ctx 仅被一个线程使用      │
└─────────────────────────────────────────────────────┘
```

**关于 `ps->quit` 的线程安全**：在 x86 架构上，`int` 的读写是原子的（单条 MOV 指令），因此不需要额外同步。但严格来说，跨平台安全的做法应使用 `SDL_AtomicInt` 或 C11 的 `_Atomic int`。

### KrKr2 的解封装线程

KrKr2 的 `BasePlayer::Process()` 比我们的 `demux_thread` 复杂得多，核心差异：

```
我们的解封装线程：
  while (!quit) {
      av_read_frame → 按 stream_index 分发到 audio/video 队列
  }

KrKr2 的 BasePlayer::Process()：
  while (!m_bStop) {
      HandleMessages()      ← 处理控制消息（Seek、Flush、变速等）
      CheckContinuity()     ← 检查时间戳连续性
      ReadPacket()           ← 读取数据包
      RoutePacket()          ← 分发到 CDVDMessageQueue（不是简单的包队列）
      UpdateCacheStatus()   ← 更新 4 级缓存状态机
  }
```

KrKr2 使用 `CDVDMessageQueue`（消息队列）而非简单的包队列，这让它可以在数据流中嵌入控制命令（如 FLUSH、RESYNC、SETSPEED），而不需要额外的控制通道。

## 本节小结

- `audio_decode_frame`：从队列取包 → 解码 → 重采样 → 更新音频时钟，是音频数据流的核心处理函数
- `sdl_audio_callback`：SDL 的 pull 模式回调，在独立线程中运行，按需从 `audio_decode_frame` 拉取数据
- `video_refresh`：在主线程的定时器事件中运行，执行解码、A/V 同步判断、YUV 纹理渲染三步操作
- `demux_thread`：独立线程中的数据泵，负责从文件读取数据包并分发到音视频队列
- 音频主时钟策略：视频帧根据与 `audio_clock` 的差值决定渲染（±50ms 内）、丢帧（落后 50ms+）或延迟（超前 50ms+）

## 常见错误

| 错误 | 原因 | 解决方案 |
|------|------|----------|
| 播放一开始就卡住 | `packet_queue_get` 阻塞但解封装线程未启动 | 确保 `demux_thread` 在 `sdl_audio_callback` 注册之前启动 |
| 播放结束时程序崩溃 | 清理顺序错误，解码器被释放后回调仍在访问 | 先 `SDL_CloseAudio()` 停止回调，再释放 codec_ctx |
| 音视频越来越不同步 | 音频时钟未正确更新 | 检查 `audio_clock` 的 PTS 计算和帧时长累加 |
| CPU 占用率 100% | 解封装线程没有 sleep | 队列满/暂停/EOF 时必须 `SDL_Delay` |
| 视频只显示第一帧 | 定时器事件未注册 | 检查 `SDL_AddTimer` 调用是否成功 |

## 练习题与答案

### 题目 1：为什么 `audio_decode_frame` 中 `av_packet_unref` 在 `avcodec_send_packet` 之后立即调用，而不是等到函数结束？

<details>
<summary>查看答案</summary>

`avcodec_send_packet` 会在内部**复制** AVPacket 的数据（或增加引用计数）。调用完成后，解码器已经持有了数据的独立副本，我们传入的 `pkt` 不再被需要。

立即释放的好处：
1. **减少内存占用**：不持有不必要的数据引用
2. **避免内存泄漏**：如果后续代码 `continue` 跳过了释放步骤，就会泄漏
3. **编程习惯**：获取资源后尽快释放是 C 语言内存管理的最佳实践

如果不调用 `av_packet_unref`，每个 AVPacket 的 `buf` 引用计数不会递减，底层的 `AVBufferRef` 永远不会被释放，最终导致内存持续增长直到进程被系统杀死。

</details>

### 题目 2：如果将 A/V 同步的阈值从 50ms 改为 500ms，用户体验会怎样？改为 5ms 呢？

<details>
<summary>查看答案</summary>

**阈值 = 500ms 时：**

```c
if (diff > 0.5)      // 视频超前 500ms 才延迟
if (diff < -0.5)     // 视频落后 500ms 才丢帧
```

效果：
- 几乎不会触发同步调整（大多数差值在 ±100ms 内）
- 相当于关闭了同步功能
- 用户会明显感知到音画不同步，尤其是对话场景中嘴型与声音不匹配
- 人眼感知阈值约 80-100ms，500ms 的偏差非常明显

**阈值 = 5ms 时：**

```c
if (diff > 0.005)    // 视频超前 5ms 就延迟
if (diff < -0.005)   // 视频落后 5ms 就丢帧
```

效果：
- 过于敏感，频繁触发丢帧或延迟
- 由于系统调度、解码耗时等因素，5ms 级别的偏差是常态
- 表现为画面严重卡顿（大量帧被丢弃或延迟）
- 实际播放可能只有几 fps（本来 30fps 的视频可能只剩 5-10fps）

**最佳实践：** 30-80ms 是常用范围。ffplay 使用自适应阈值（根据帧率和历史偏差动态调整），KrKr2 的 `CRenderManager::PrepareNextRender()` 也使用了类似的动态阈值策略。

</details>

### 题目 3：修改 `sdl_audio_callback`，添加音量控制功能

<details>
<summary>查看答案</summary>

```c
// 在 PlayerState 结构体中添加：
typedef struct PlayerState {
    // ... 其他字段 ...
    float volume;  // 音量，范围 0.0（静音）~ 1.0（最大）
} PlayerState;

// 初始化时设置默认音量：
ps->volume = 1.0f;

// 修改后的回调函数：
static void sdl_audio_callback(void *userdata, Uint8 *stream, int len) {
    PlayerState *ps = (PlayerState *)userdata;
    int len1, audio_size;

    // 先将输出缓冲区清零（静音）
    // 这样即使我们只填充了部分数据，剩余部分也是静音而非随机噪声
    SDL_memset(stream, 0, len);

    while (len > 0) {
        if (ps->audio_buf_index >= ps->audio_buf_size) {
            audio_size = audio_decode_frame(ps);
            if (audio_size < 0) {
                break;  // stream 已经是静音（全零）
            }
            ps->audio_buf_size = audio_size;
            ps->audio_buf_index = 0;
        }

        len1 = ps->audio_buf_size - ps->audio_buf_index;
        if (len1 > len) len1 = len;

        // 使用 SDL_MixAudioFormat 混合数据，支持音量控制
        // 参数：
        //   stream   —— 目标缓冲区（已清零）
        //   source   —— 源数据
        //   format   —— 音频格式（AUDIO_S16SYS）
        //   len1     —— 数据长度（字节）
        //   volume   —— 音量（0 ~ SDL_MIX_MAXVOLUME，即 0 ~ 128）
        SDL_MixAudioFormat(
            stream,
            ps->audio_buf + ps->audio_buf_index,
            AUDIO_S16SYS,
            len1,
            (int)(ps->volume * SDL_MIX_MAXVOLUME)
        );

        len -= len1;
        stream += len1;
        ps->audio_buf_index += len1;
    }
}

// 在事件循环中添加按键控制：
case SDLK_UP:
    ps->volume = fminf(1.0f, ps->volume + 0.1f);
    printf("音量: %.0f%%\n", ps->volume * 100);
    break;
case SDLK_DOWN:
    ps->volume = fmaxf(0.0f, ps->volume - 0.1f);
    printf("音量: %.0f%%\n", ps->volume * 100);
    break;
```

关键改动说明：
1. 用 `SDL_memset(stream, 0, len)` 预先清零，避免随机噪声
2. 用 `SDL_MixAudioFormat` 替代 `memcpy`，该函数在混合时自动应用音量系数
3. `SDL_MIX_MAXVOLUME` 的值为 128，乘以 `volume`（0.0~1.0）得到实际音量

</details>
