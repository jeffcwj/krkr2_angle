## 1. 日志系统与调试基础设施

KrKr2 使用 **spdlog**（一个高性能 C++ 日志库，支持格式化输出和多级日志）作为全局日志框架。媒体播放器模块使用 `"core"` logger（日志记录器，即 spdlog 中用于分类输出的命名实例）。调试播放器的第一步是理解如何高效地添加和使用日志。

### 1.1 spdlog 日志级别与使用

spdlog 提供六个日志级别，从低到高：

| 级别 | 宏 | 用途 | 播放器中的典型场景 |
|------|-----|------|-------------------|
| trace | `SPDLOG_TRACE(...)` | 最细粒度，逐帧数据 | 每帧 PTS（Presentation Time Stamp，显示时间戳）值 |
| debug | `SPDLOG_DEBUG(...)` | 开发调试信息 | 丢帧决策、缓冲区状态变化 |
| info | `SPDLOG_INFO(...)` | 关键状态变化 | 播放开始/停止、seek 完成 |
| warn | `SPDLOG_WARN(...)` | 潜在问题 | 解码超时、缓冲区接近满 |
| error | `SPDLOG_ERROR(...)` | 可恢复错误 | 解码失败但可跳帧 |
| critical | `SPDLOG_CRITICAL(...)` | 致命错误 | 解封装器（Demuxer）打开失败 |

在 Debug 构建（CMake 中 `CMAKE_BUILD_TYPE=Debug`）中，`SPDLOG_ACTIVE_LEVEL` 默认为 `SPDLOG_LEVEL_DEBUG`，所有 `SPDLOG_DEBUG` 及以上级别的日志都会输出。Release 构建中只输出 `info` 及以上。

### 1.2 在播放器关键路径添加日志

以下是一个完整的日志插桩（Instrumentation，在代码中添加监测点以收集运行时数据）示例，覆盖播放器的四个关键路径：

```cpp
// === 文件：VideoPlayer.cpp — BasePlayer::Process() 主循环 ===
// 在 ReadPacket() 调用前后添加计时日志

#include <chrono>  // C++ 标准库的时间测量头文件

void CVideoPlayer::Process() {
    // ... 省略初始化代码 ...
    while (!m_bAbortRequest) {  // 主循环，m_bAbortRequest 为停止标志
        // 1. 处理消息（seek、暂停等控制命令）
        HandleMessages();

        // 2. 读取数据包并计时
        auto t0 = std::chrono::steady_clock::now();  // 记录开始时间
        CDVDMsg* pMsg = nullptr;
        int result = ReadPacket(pMsg);  // 从解封装器读取一个数据包
        auto t1 = std::chrono::steady_clock::now();

        // 计算耗时（微秒）
        auto readUs = std::chrono::duration_cast<
            std::chrono::microseconds>(t1 - t0).count();

        // 超过 5ms 才记录，避免日志洪水
        if (readUs > 5000) {
            SPDLOG_WARN("ReadPacket slow: {}us, result={}",
                        readUs, result);
        }

        // 3. 分发数据包到音频/视频子线程
        if (pMsg) {
            ProcessPacket(pMsg);
            pMsg->Release();  // 释放引用计数
        }
    }
}
```

## 2. 问题诊断：视频播放卡顿

视频卡顿是最常见的播放器问题。在 KrKr2 中，卡顿通常由三个环节之一引起：**解码慢**、**渲染阻塞**、**丢帧策略过激**。

### 2.1 症状识别

卡顿有几种不同表现，对应不同原因：

| 症状 | 特征 | 可能原因 |
|------|------|----------|
| 均匀卡顿 | 每隔固定间隔顿一下 | Cocos2d-x 调度频率低于视频帧率 |
| 随机卡顿 | 不规律地突然卡住 | I 帧（关键帧，可独立解码的完整帧）解码耗时过长 |
| 渐进卡顿 | 越播越卡 | 内存泄漏或缓冲区堆积 |
| 开头流畅后变卡 | 播放几十秒后开始卡 | 解码速度略慢于帧率，延迟累积 |

### 2.2 丢帧统计监测

KrKr2 的 `CalcDropRequirement()` 函数（位于 `VideoPlayerVideo.cpp` 第 1013-1090 行）是丢帧决策的核心。通过监测它的输出可以量化卡顿程度：

```cpp
// === 文件：VideoPlayerVideo.cpp — CalcDropRequirement() ===
// 源码位置：第 1013-1090 行
// 功能：根据当前帧的迟到程度决定是否丢帧

int CVideoPlayerVideo::CalcDropRequirement(double pts) {
    double iClock = m_pClock->GetClock();  // 获取主时钟当前值
    // iLateness = 帧应显示时间 - 当前时钟
    // 负值表示帧已迟到，正值表示帧提前
    double iLateness = pts - iClock;

    // ===== 添加调试日志 =====
    SPDLOG_DEBUG(
        "[DropCalc] pts={:.3f}s clock={:.3f}s lateness={:.1f}ms "
        "lateframes={} frametime={:.1f}ms",
        pts / DVD_TIME_BASE,
        iClock / DVD_TIME_BASE,
        iLateness / 1000.0,           // 微秒转毫秒
        m_droppingStats.m_lateframes, // 连续迟到帧计数
        m_fFrameTime / 1000.0         // 帧间隔（微秒转毫秒）
    );

    // 丢帧策略：连续迟到帧越多，容忍度越低
    // lateframes ≤ 6：grace = 0.98 × frametime（几乎不丢）
    // lateframes 7-10：grace 线性下降
    // lateframes > 10：grace = 0（零容忍，立即丢弃）
    double grace = 0;
    if (m_droppingStats.m_lateframes <= 6) {
        grace = m_fFrameTime * 0.98;
    } else if (m_droppingStats.m_lateframes <= 10) {
        // 线性插值：从 0.98 降到 0
        double ratio = (10.0 - m_droppingStats.m_lateframes) / 4.0;
        grace = m_fFrameTime * 0.98 * ratio;
    }
    // lateframes > 10: grace 保持 0

    if (-iLateness > grace) {
        m_droppingStats.m_lateframes++;
        m_droppingStats.m_dropcount++;
        SPDLOG_WARN("[DropCalc] DROPPING frame! total_drops={}",
                    m_droppingStats.m_dropcount);
        return 1;  // 1 = 需要丢弃此帧
    }

    // 帧没有迟到，重置连续计数
    if (iLateness >= 0) {
        m_droppingStats.m_lateframes = 0;
    }
    return 0;  // 0 = 正常显示此帧
}
```

### 2.3 渲染队列状态监测

`CRenderManager`（渲染管理器，负责管理帧缓冲队列的组件）维护三个队列：`m_free`（空闲）、`m_queued`（等待显示）、`m_discard`（待回收）。队列状态直接反映播放健康度：

```
正常状态：  free=3-4  queued=1-2  discard=0-1  （解码和渲染速度匹配）
解码过慢：  free=5    queued=0    discard=0    （渲染空等解码）
渲染过慢：  free=0    queued=4-5  discard=0    （解码快但渲染消费不了）
泄漏状态：  free=0    queued=0    discard=5    （帧没有被正确回收到 free）
```

添加周期性状态输出：

```cpp
// === 文件：VideoRenderer.cpp — CRenderManager::FrameMove() ===
// 源码位置：第 302-355 行
// 在每次帧推进时记录队列状态

void CRenderManager::FrameMove() {
    // ... 原有的状态机逻辑 ...

    // ===== 每 120 帧输出一次统计（约 2 秒 @60fps）=====
    static int s_frameCounter = 0;
    if (++s_frameCounter % 120 == 0) {
        SPDLOG_INFO(
            "[RenderMgr] free={} queued={} discard={} "
            "state={} presentpts={:.3f}s",
            m_free.size(),
            m_queued.size(),
            m_discard.size(),
            static_cast<int>(m_renderState),  // IDLE=0, READY=1, FLIP=2
            m_presentpts / DVD_TIME_BASE
        );
    }
}
```

### 2.4 解决方案矩阵

| 原因 | 诊断特征 | 解决方案 |
|------|----------|----------|
| I 帧解码慢 | `ReadPacket slow` 日志 + 大 GOP（Group of Pictures，一组从 I 帧到下一个 I 帧的连续帧） | 降低视频分辨率或启用硬件解码 |
| 环形缓冲区太小 | `WaitForBuffer` 频繁超时 | 增大 `MAX_BUFFER_COUNT`（默认 4→8） |
| 调度频率不足 | 均匀卡顿 + `queued` 持续增长 | 提高 Cocos2d-x `Director::setAnimationInterval()` |
| 丢帧过多 | `DROPPING frame` 日志密集 | 检查 CPU 占用，优化解码或降低分辨率 |

## 3. 问题诊断：音视频不同步

音视频不同步（A/V Desync）表现为画面和声音对不上。KrKr2 的同步系统基于 CDVDClock（主时钟），音频作为时钟源（Master Clock），视频向音频对齐。

### 3.1 同步架构回顾

```
┌─────────────────────────────────────────────────┐
│                  CDVDClock（主时钟）               │
│  m_startClock + m_systemAdjust + speedAdjust    │
│         ↑ 音频线程校准          ↓ 视频线程查询     │
├────────────────────┬────────────────────────────┤
│  AudioPlayer       │  VideoPlayer               │
│  解码音频 →        │  解码视频 →                  │
│  播放到音频设备 →   │  OutputPicture() →          │
│  用实际播放位置     │  查询 GetClock() →          │
│  校准 CDVDClock    │  比较 PTS vs Clock →        │
│                    │  决定显示/等待/丢弃          │
└────────────────────┴────────────────────────────┘
```

### 3.2 时钟漂移监测

时钟漂移（Clock Drift）是指 CDVDClock 的值与实际音频播放位置之间逐渐累积的偏差。监测方法：

```cpp
// === 文件：Clock.cpp — CDVDClock::GetClock() ===
// 源码位置：第 120-150 行
// 添加漂移监测日志

double CDVDClock::GetClock(bool interpolated) {
    std::lock_guard<std::mutex> lock(m_critSection);

    double clock = m_startClock;
    // 累积速度调整量
    auto now = std::chrono::steady_clock::now();
    double elapsed = std::chrono::duration<double>(
        now - m_startTime).count();

    clock += elapsed * m_systemFrequency;  // 基于系统频率推进
    clock += m_systemAdjust;               // 加上累积调整量

    // ===== 漂移监测 =====
    // m_iDisc 是最近一次 Discontinuity() 设定的基准时间戳
    // 如果 clock 与 m_iDisc 偏差过大，说明时钟漂移
    static double s_lastLogTime = 0;
    if (elapsed - s_lastLogTime > 5.0) {  // 每 5 秒记录一次
        s_lastLogTime = elapsed;
        SPDLOG_DEBUG(
            "[Clock] value={:.3f}s adjust={:.3f}ms "
            "speedAdj={:.6f} freq={:.0f}",
            clock / DVD_TIME_BASE,
            m_systemAdjust / 1000.0,
            m_speedAdjust,
            m_systemFrequency
        );
    }

    return clock;
}
```

### 3.3 同步启动流程诊断

A/V 不同步经常在播放刚开始时就出现，原因是同步启动流程出错。正常启动流程（参见 `VideoPlayer.cpp` 第 1991-2094 行）：

```
1. 音频/视频子线程各自解码足够数据
2. 各自发送 PLAYER_STARTED 消息（syncState: SYNC_STARTING → SYNC_WAITSYNC）
3. BasePlayer::HandlePlaySpeed() 检测到两个流都到达 SYNC_WAITSYNC
4. 调用 m_clock.Discontinuity(clock) 重置时钟基准
5. 发送 GENERAL_RESYNC 给所有子线程
6. 同步状态变为 SYNC_INSYNC，正式开始播放
```

诊断启动同步问题的日志：

```cpp
// === 文件：VideoPlayer.cpp — HandlePlaySpeed() ===
// 源码位置：第 1991-2094 行

// 在 SYNC_WAITSYNC 检测点添加日志
if (m_CurrentVideo.syncState == SYNC_WAITSYNC &&
    m_CurrentAudio.syncState == SYNC_WAITSYNC) {

    SPDLOG_INFO("[Sync] Both streams ready! "
                "videoPts={:.3f}s audioPts={:.3f}s",
                m_CurrentVideo.startpts / DVD_TIME_BASE,
                m_CurrentAudio.startpts / DVD_TIME_BASE);

    // Discontinuity 会重置时钟到指定值
    double clock = std::min(m_CurrentVideo.startpts,
                            m_CurrentAudio.startpts);
    m_clock.Discontinuity(clock);

    SPDLOG_INFO("[Sync] Clock reset to {:.3f}s, entering INSYNC",
                clock / DVD_TIME_BASE);

    // 发送 GENERAL_RESYNC
    SendMessage(CDVDMsg::GENERAL_RESYNC, clock);
    m_syncState = SYNC_INSYNC;
}
```

### 3.4 常见不同步原因与修复

| 原因 | 诊断方法 | 修复策略 |
|------|----------|----------|
| 音频设备延迟 | 音频 PTS 与实际播放位置差 > 50ms | 调整音频缓冲区大小 |
| 视频解码过慢 | `drift` 持续为负且绝对值增大 | 增加丢帧积极性或降低分辨率 |
| 时钟频率偏差 | `speedAdj` 值偏离 1.0 太多 | 检查 `ErrorAdjust()` 阈值 |
| 启动不同步 | 只有一个流到达 SYNC_WAITSYNC | 检查对应解码线程是否正常启动 |
| seek 后不同步 | seek 后 `drift` 突变 | 确认 `Discontinuity()` 被正确调用 |

## 4. 问题诊断：Seek 后黑屏与花屏

Seek（跳转播放位置）是媒体播放器中最容易出问题的操作，因为它涉及多个线程的状态重置和协调。

### 4.1 正常 Seek 流程

```
用户点击进度条
    │
    ▼
BasePlayer::HandleMessages()
    │ 收到 PLAYER_SEEK 消息
    ▼
FlushBuffers()  ← 三阶段清洗
    │
    ├── Phase 1: RESET
    │   └── 通知子线程停止处理
    ├── Phase 2: FLUSH
    │   ├── 发送 GENERAL_FLUSH 给 VideoPlayer
    │   ├── 发送 GENERAL_FLUSH 给 AudioPlayer
    │   └── 子线程收到后清空各自的消息队列和解码缓冲
    └── Phase 3: SYNCHRONIZE
        └── 等待所有子线程确认清洗完成
    │
    ▼
CDVDDemuxFFmpeg::SeekTime()
    │ 调用 av_seek_frame()（FFmpeg 的跳转函数）
    │ 跳转到目标时间戳附近的 I 帧位置
    ▼
继续 ReadPacket() → ProcessPacket()
    │ 从新位置开始读取数据包
    │ 第一个可解码的帧必须是 I 帧
    ▼
视频恢复正常显示
```

### 4.2 黑屏问题

**症状：** seek 后画面全黑，声音正常播放。

**原因分析与诊断：**

```cpp
// === 诊断步骤 1：确认 FLUSH 消息到达视频线程 ===
// 文件：VideoPlayerVideo.cpp — Process()
// 源码位置：第 209-473 行

void CVideoPlayerVideo::Process() {
    while (!m_bStop) {
        CDVDMsg* pMsg = nullptr;
        // timeout=1000 表示等待最多 1 秒
        int ret = m_messageQueue.Get(&pMsg, 1000);

        if (ret == MSGQ_OK) {
            switch (pMsg->GetType()) {
            case CDVDMsg::GENERAL_FLUSH:
                SPDLOG_INFO("[VideoThread] Received FLUSH, "
                            "resetting decoder");
                // 重置解码器状态
                m_pVideoCodec->Reset();
                // 清空本地帧缓冲
                m_droppingStats = {};
                break;

            case CDVDMsg::GENERAL_RESYNC: {
                double pts = static_cast<CDVDMsgDouble*>(pMsg)
                    ->m_value;
                SPDLOG_INFO("[VideoThread] Received RESYNC, "
                            "newPts={:.3f}s",
                            pts / DVD_TIME_BASE);
                break;
            }
            // ... 其他消息类型 ...
            }
            pMsg->Release();
        }
    }
}

// === 诊断步骤 2：确认解码器被正确重置 ===
// 文件：VideoCodecFFmpeg.cpp — Reset()
// 如果 avcodec_flush_buffers() 没有被调用，
// 解码器内部残留旧帧的参考帧数据，导致花屏

void CDVDVideoCodecFFmpeg::Reset() {
    if (m_pCodecContext) {
        avcodec_flush_buffers(m_pCodecContext);
        // avcodec_flush_buffers() 清除解码器内部缓存：
        // - 参考帧缓冲（B帧解码依赖的前后参考帧）
        // - 半解码的帧数据
        // - 内部状态计数器
        SPDLOG_DEBUG("[Codec] Decoder flushed");
    }
    m_iLastKeyframe = 0;  // 重置关键帧计数
}
```

**常见黑屏原因清单：**

| 原因 | 诊断 | 修复 |
|------|------|------|
| FLUSH 消息被 Abort 吞掉 | 日志中无 "Received FLUSH" | 检查 `m_messageQueue.Abort()` 的调用时机 |
| 解码器没有重置 | 花屏而非纯黑 | 确认 `Reset()` 调用了 `avcodec_flush_buffers()` |
| seek 后读到 B/P 帧 | 花屏（马赛克） | `av_seek_frame()` 需要 `AVSEEK_FLAG_BACKWARD` 标志 |
| 环形缓冲区残留旧数据 | 闪烁显示旧画面 | FLUSH 时清空 `m_buffers[]` |
| RESYNC 时间戳错误 | 音视频时间不匹配 | 检查 `Discontinuity()` 的参数 |

### 4.3 花屏问题

花屏（画面出现马赛克或色块）的根本原因是解码器使用了**错误的参考帧**。视频压缩中，P 帧（Predicted Frame，预测帧，依赖前一帧数据的帧）和 B 帧（Bidirectional Frame，双向预测帧，依赖前后两帧数据的帧）不能独立解码，它们需要 I 帧作为参考。

```cpp
// === 花屏修复完整方案 ===
// 确保 seek 后的第一个解码帧是 I 帧

// 步骤 1：在 DemuxFFmpeg.cpp 中确保 seek 到 I 帧边界
bool CDVDDemuxFFmpeg::SeekTime(double time) {
    int64_t seekTarget = static_cast<int64_t>(
        time * AV_TIME_BASE / DVD_TIME_BASE);

    // AVSEEK_FLAG_BACKWARD：向后搜索到最近的 I 帧
    // 这确保 seek 后的第一个数据包是 I 帧
    int ret = av_seek_frame(
        m_pFormatContext,  // 格式上下文
        -1,                // -1 表示让 FFmpeg 自动选择流
        seekTarget,        // 目标时间戳
        AVSEEK_FLAG_BACKWARD  // 向后搜索标志
    );

    if (ret < 0) {
        char errBuf[AV_ERROR_MAX_STRING_SIZE];
        av_strerror(ret, errBuf, sizeof(errBuf));
        SPDLOG_ERROR("[Demux] seek failed: {} (target={:.3f}s)",
                     errBuf, time / DVD_TIME_BASE);
        return false;
    }

    SPDLOG_INFO("[Demux] seek to {:.3f}s succeeded", 
                time / DVD_TIME_BASE);
    return true;
}

// 步骤 2：在 VideoCodecFFmpeg 中等待 I 帧
bool CDVDVideoCodecFFmpeg::Decode(uint8_t* pData, int iSize,
                                  double dts, double pts) {
    // 如果刚 flush 过，跳过非关键帧
    if (m_iLastKeyframe == 0 && !isKeyFrame(pData, iSize)) {
        SPDLOG_DEBUG("[Codec] Skipping non-keyframe after flush");
        return true;  // 告诉调用者"成功处理"（实际丢弃了）
    }
    // ... 正常解码逻辑 ...
    return true;
}
```

## 5. 内存问题诊断

媒体播放器的内存问题主要有两类：**引用计数泄漏**（CDVDMsg）和**缓冲区泄漏**（YUV 数据）。

### 5.1 CDVDMsg 引用计数泄漏

CDVDMsg 使用手动引用计数（Reference Counting，一种内存管理技术，每个对象维护一个计数器，计数归零时自动释放）。每个消息创建时 `refcount=1`，`Get()` 取出后消费者必须调用 `Release()`。

```cpp
// ❌ 泄漏模式 1：提前返回忘记 Release
void ProcessMessages() {
    CDVDMsg* pMsg = nullptr;
    m_messageQueue.Get(&pMsg, 1000);  // refcount 仍为 1

    if (pMsg->GetType() == CDVDMsg::DEMUXER_PACKET) {
        auto* pkt = static_cast<CDVDMsgDemuxerPacket*>(pMsg);
        if (pkt->GetPacket()->stream_index != m_streamIndex) {
            return;  // ❌ 忘记 Release！每次走这条路径泄漏一个消息
        }
        // 处理数据包...
    }
    pMsg->Release();  // ✅ 只有不走 return 的路径才到这里
}

// ❌ 泄漏模式 2：异常路径
void DecodePacket(CDVDMsg* pMsg) {
    auto* pkt = static_cast<CDVDMsgDemuxerPacket*>(pMsg);
    int ret = m_pCodec->Decode(pkt->GetData(), pkt->GetSize());
    if (ret < 0) {
        throw std::runtime_error("decode failed");
        // ❌ 异常抛出后 Release 永远执行不到
    }
    pMsg->Release();
}

// ✅ 正确做法：RAII 包装器
class CDVDMsgGuard {
    CDVDMsg* m_pMsg;
public:
    explicit CDVDMsgGuard(CDVDMsg* msg) : m_pMsg(msg) {}
    ~CDVDMsgGuard() {
        if (m_pMsg) m_pMsg->Release();  // 析构时自动释放
    }
    CDVDMsg* get() { return m_pMsg; }
    // 禁止拷贝
    CDVDMsgGuard(const CDVDMsgGuard&) = delete;
    CDVDMsgGuard& operator=(const CDVDMsgGuard&) = delete;
};

// 使用 RAII 包装器的安全代码
void SafeProcessMessages() {
    CDVDMsg* pMsg = nullptr;
    m_messageQueue.Get(&pMsg, 1000);
    CDVDMsgGuard guard(pMsg);  // 无论怎么返回都会 Release

    if (pMsg->GetType() == CDVDMsg::DEMUXER_PACKET) {
        auto* pkt = static_cast<CDVDMsgDemuxerPacket*>(pMsg);
        if (pkt->GetPacket()->stream_index != m_streamIndex) {
            return;  // ✅ guard 析构函数会自动调用 Release
        }
    }
    // guard 在这里也会自动析构
}
```

### 5.2 YUV 缓冲区泄漏

TVPMoviePlayer 使用 `TJSAlignedAlloc()` 分配 YUV 缓冲区。如果播放器销毁时没有正确释放这些缓冲区，会导致内存泄漏：

```cpp

bool TVPMoviePlayer::AddVideoPicture(DVDVideoPicture& pic) {
    if (pic.format != RENDER_FMT_YUV420P)
        return false;

    int idx = m_curPicture % MAX_BUFFER_COUNT;
    YUVPicture& buf = m_pictures[idx];

    int ySize = pic.iWidth * pic.iHeight;
    int uvSize = (pic.iWidth / 2) * (pic.iHeight / 2);

    if (!buf.data[0]) {

        // 注意：这里对齐是 4 字节，不是 SSE 要求的 16 或 AVX 的 32
        buf.data[0] = (uint8_t*)TJSAlignedAlloc(ySize, 4);    // Y 平面
        buf.data[1] = (uint8_t*)TJSAlignedAlloc(uvSize, 4);   // U 平面
        buf.data[2] = (uint8_t*)TJSAlignedAlloc(uvSize, 4);   // V 平面
    }

    // 拷贝 YUV 数据到环形缓冲区
    // ... 逐行拷贝（处理 stride 差异）...
    m_curPicture++;
    return true;
}

// ✅ 析构函数中必须释放所有缓冲区
TVPMoviePlayer::~TVPMoviePlayer() {
    for (int i = 0; i < MAX_BUFFER_COUNT; i++) {
        for (int j = 0; j < 3; j++) {

## 6. 性能分析与瓶颈定位

播放器性能问题通常集中在三个环节：**解封装（I/O）**、**解码（CPU/GPU）**、**渲染（GPU + 主线程调度）**。定位瓶颈需要量化每个环节的耗时。

### 6.1 逐环节计时框架

```cpp

// 用于测量代码块的执行时间，支持累积统计

#include <chrono>
#include <string>
#include <spdlog/spdlog.h>

class ScopedTimer {
    std::string m_name;
    std::chrono::steady_clock::time_point m_start;
    double m_warnThresholdMs;  // 超过此阈值输出警告

public:
    // name: 计时器名称（用于日志输出）
    // warnMs: 超时警告阈值（毫秒），默认 10ms
    ScopedTimer(const char* name, double warnMs = 10.0)
        : m_name(name)
        , m_start(std::chrono::steady_clock::now())
        , m_warnThresholdMs(warnMs) {}

    ~ScopedTimer() {
        auto elapsed = std::chrono::steady_clock::now() - m_start;
        double ms = std::chrono::duration<double, std::milli>(
            elapsed).count();

        if (ms > m_warnThresholdMs) {
            // 超时才输出，避免日志洪水
            SPDLOG_WARN("[Timer] {} took {:.2f}ms (threshold={:.1f}ms)",
                        m_name, ms, m_warnThresholdMs);
        }
    }
};

            ScopedTimer t("ReadPacket", 10.0);

        {
            ScopedTimer t("ProcessPacket", 5.0);

        }

### 6.2 解码性能基准

不同分辨率和编码格式的解码性能差异巨大。以下是典型参考值（CPU 软解码，单核）：

| 视频规格 | H.264 解码 | H.265 解码 | 说明 |
|----------|-----------|-----------|------|
| 720p@30fps | ~5ms/帧 | ~8ms/帧 | 轻松实时 |
| 1080p@30fps | ~12ms/帧 | ~20ms/帧 | 刚好实时（33ms 余量） |
| 1080p@60fps | ~12ms/帧 | ~20ms/帧 | H.265 可能来不及（16ms 余量） |
| 4K@30fps | ~45ms/帧 | ~80ms/帧 | 软解码无法实时 |

> **注意：** KrKr2 使用 FFmpeg 3.3.9 的旧 API（`avcodec_decode_video2`），不支持多线程解码的某些优化。实际性能可能低于上表。

### 6.3 缓存状态机性能影响

BasePlayer 的缓存状态机（Cache State Machine，`HandlePlaySpeed()` 中第 1865-2183 行）有五个状态：

```
CACHESTATE_DONE    ← 正常播放状态
    │ 缓冲不足
    ▼
CACHESTATE_FLUSH   ← 清空现有缓冲
    │
    ▼
CACHESTATE_FULL    ← 等待缓冲填满
    │ 缓冲达到阈值
    ▼
CACHESTATE_INIT    ← 初始化同步
    │
    ▼
CACHESTATE_PLAY    ← 开始播放
    │ 稳定后
    ▼
CACHESTATE_DONE    ← 回到正常状态

## 动手实践

### 实践 1：构建播放器健康度仪表盘

在 `CRenderManager::FrameMove()` 中添加一个结构化的健康度报告，每 2 秒输出一次。这个报告应该让你一眼就能判断播放器是否健康。

**完整实现：**

```cpp
// === 文件：VideoRenderer.cpp ===
// 在 CRenderManager 类中添加统计结构体

struct PlaybackHealthStats {
    int totalFrames = 0;       // 总帧数
    int droppedFrames = 0;     // 丢弃帧数
    double maxDriftMs = 0.0;   // 最大时钟漂移（毫秒）
    double avgDriftMs = 0.0;   // 平均时钟漂移
    double driftSum = 0.0;     // 漂移累加器
    int driftSamples = 0;      // 漂移采样数

    void recordFrame(double driftMs, bool dropped) {
        totalFrames++;
        if (dropped) droppedFrames++;
        driftSum += std::abs(driftMs);
        driftSamples++;
        avgDriftMs = driftSum / driftSamples;
        if (std::abs(driftMs) > std::abs(maxDriftMs)) {
            maxDriftMs = driftMs;
        }
    }

    // 健康度评分：0-100
    int healthScore() const {
        if (totalFrames == 0) return 100;
        double dropRate = static_cast<double>(droppedFrames)
                          / totalFrames;
        int score = 100;
        score -= static_cast<int>(dropRate * 200);   // 丢帧率扣分
        score -= static_cast<int>(avgDriftMs / 5.0); // 平均漂移扣分
        return std::max(0, std::min(100, score));
    }

    void reset() { *this = PlaybackHealthStats{}; }
};

// 在 FrameMove() 中使用：
void CRenderManager::FrameMove() {
    // ... 原有逻辑 ...

    // 每 120 帧（约 2 秒）输出健康度报告
    static PlaybackHealthStats stats;
    static int reportCounter = 0;

    // 记录当前帧的漂移
    double currentDrift = 0.0;
    if (m_presentpts != DVD_NOPTS_VALUE) {
        double clock = m_pClock->GetClock();
        currentDrift = (m_presentpts - clock) / 1000.0; // 毫秒
    }
    stats.recordFrame(currentDrift, m_lateframes > 0);

    if (++reportCounter % 120 == 0) {
        // 输出健康度报告
        const char* status = "HEALTHY";
        if (stats.healthScore() < 50) status = "CRITICAL";
        else if (stats.healthScore() < 80) status = "WARNING";

        SPDLOG_INFO(
            "\n"
            "╔══════════════════════════════════════╗\n"
            "║     Playback Health Report           ║\n"
            "╠══════════════════════════════════════╣\n"
            "║ Status:     {:<10s}               ║\n"
            "║ Score:      {}/100                  ║\n"
            "║ Frames:     {} total, {} dropped    ║\n"
            "║ Drop Rate:  {:.1f}%                  ║\n"
            "║ Avg Drift:  {:.1f}ms                 ║\n"
            "║ Max Drift:  {:.1f}ms                 ║\n"
            "║ Queued:     {} | Free: {} | Disc: {} ║\n"
            "╚══════════════════════════════════════╝",
            status,
            stats.healthScore(),
            stats.totalFrames, stats.droppedFrames,
            stats.totalFrames > 0
                ? 100.0 * stats.droppedFrames / stats.totalFrames
                : 0.0,
            stats.avgDriftMs,
            stats.maxDriftMs,
            m_queued.size(), m_free.size(), m_discard.size()
        );

        stats.reset();  // 重置统计，开始下一个周期
    }
}
```

### 实践 3：缓冲区大小调优实验

修改缓冲区参数，测量对播放性能的影响：

```cpp
// === 实验 A：修改 TVPMoviePlayer 环形缓冲区大小 ===
// 文件：KRMoviePlayer.h
// 原值：MAX_BUFFER_COUNT = 4

// 实验值及预期影响：
// MAX_BUFFER_COUNT = 2  → 内存减半，但 WaitForBuffer 更频繁阻塞
// MAX_BUFFER_COUNT = 4  → 默认值，一般场景足够
// MAX_BUFFER_COUNT = 8  → 解码线程有更多缓冲余量

// Y 平面: 1920 × 1080 = 2,073,600 字节（约 2MB）
// U 平面: 960 × 540 = 518,400 字节（约 0.5MB）
// V 平面: 960 × 540 = 518,400 字节（约 0.5MB）
// 每帧合计: 约 3MB
// MAX_BUFFER_COUNT=4: 约 12MB
// MAX_BUFFER_COUNT=8: 约 24MB

// === 实验 B：修改 CRenderManager 帧缓冲数量 ===
// 文件：VideoRenderer.cpp
// 原值：NUM_BUFFERS = 6

## 本节小结

- **日志是第一调试工具**：在关键路径（ReadPacket、OutputPicture、FrameMove、FlushBuffers）添加结构化日志，用时间戳和量化指标替代猜测
- **卡顿三步排查法**：① 查 `CalcDropRequirement()` 的丢帧统计 → ② 查 `FrameMove()` 的队列状态 → ③ 用 ScopedTimer 定位慢环节
- **A/V 不同步的核心是 CDVDClock**：监测 `GetClock()` 的漂移值和 `speedAdjust` 参数，检查 `Discontinuity()` 是否在 seek/循环点被正确调用
- **Seek 后花屏必查 `avcodec_flush_buffers()`**：这是解码器重置的关键函数，遗漏会导致参考帧残留
- **内存泄漏两大来源**：CDVDMsg 忘记 `Release()`（用 RAII 包装器解决）和 YUV 缓冲区析构时漏释放
- **DVD_NOPTS_VALUE = 0**：这是 KrKr2 与标准 FFmpeg 的关键差异，比较时必须用 KrKr2 的定义
- **跨线程安全**：解码线程不能直接操作 Cocos2d-x 对象，必须通过环形缓冲区传递到主线程
- **性能分析量化优先**：用 ScopedTimer 测量各环节耗时，用 PlaybackHealthStats 计算健康度评分，避免凭感觉调试

### 题目 1：消息优先级与 Seek 流程分析

假设视频正在播放中（PTS 约 5.0 秒），用户点击进度条跳转到 30.0 秒。此时 `CVideoPlayerVideo` 的消息队列中有以下消息：

```
队列内容（入队顺序）：
1. DEMUXER_PACKET  (priority=0, PTS=5.000s)
2. DEMUXER_PACKET  (priority=0, PTS=5.033s)
3. DEMUXER_PACKET  (priority=0, PTS=5.067s)
4. GENERAL_FLUSH   (priority=1)              ← seek 触发
5. GENERAL_RESYNC  (priority=1, PTS=30.0s)   ← seek 目标
```

问题：
1. 这 5 条消息的实际**取出**顺序是什么？
2. 最终哪些帧会被显示？
3. 如果 `GENERAL_FLUSH` 的 priority 改为 0（与数据包同优先级），会发生什么？

<details>
<summary>查看答案</summary>

**1. 实际取出顺序：**

`CDVDMessageQueue::Get()` 优先返回 `priority > 0` 的消息，不论其在队列中的位置：

```
取出 1: GENERAL_FLUSH  (priority=1)   ← 最先取出
取出 2: GENERAL_RESYNC (priority=1)   ← 第二个取出
取出 3: DEMUXER_PACKET (PTS=5.000s)   ← 按入队顺序
取出 4: DEMUXER_PACKET (PTS=5.033s)
取出 5: DEMUXER_PACKET (PTS=5.067s)
```

**2. 最终显示的帧：**

- 处理 `GENERAL_FLUSH` 时，视频线程清空解码器状态和内部缓冲
- 处理 `GENERAL_RESYNC` 时，设置新的同步基准 PTS=30.0s
- 取出 3-5 的旧数据包虽然被取出，但它们的 PTS（5.0s 附近）远小于同步基准（30.0s），会被丢弃
- 最终只有 **seek 后从新位置读取的帧**（PTS ≥ 30.0s 的 I 帧开始）才会被显示

**3. 如果 FLUSH priority=0：**

取出顺序变为严格入队顺序：1→2→3→4→5

后果：
- 数据包 1-3（PTS=5.0~5.067s）会被先解码和显示
- 用户会看到 seek 前的旧画面闪过（约 100ms）
- 然后 FLUSH 才到达，清空解码器
- 最终 RESYNC 设置新基准
- 用户体验：seek 后画面先闪回旧位置，再跳到新位置（不可接受的 UX 问题）

这就是为什么控制消息**必须**用高优先级——它们需要"插队"到所有数据包前面。

</details>

### 题目 2：帧缓冲队列状态诊断

CRenderManager 的 `NUM_BUFFERS=6`，在某一时刻你观察到以下日志输出持续 10 秒：

```
[RenderMgr] free=0 queued=5 discard=0 state=1 presentpts=12.345s
[RenderMgr] free=0 queued=5 discard=0 state=1 presentpts=12.345s
[RenderMgr] free=0 queued=5 discard=0 state=1 presentpts=12.345s
```

问题：
1. `free=0 queued=5` 说明什么问题？
2. `presentpts` 不变说明什么？
3. `state=1`（READY）持续不变说明什么？
4. 你会如何修复这个问题？

<details>
<summary>查看答案</summary>

**1. `free=0 queued=5`：**

总共 6 个缓冲槽位：0 个空闲 + 5 个等待显示 + 0 个待回收 = 5 个在用 + 1 个正在 present。解码线程在 `WaitForBuffer()` 中**阻塞**，因为没有空闲槽位可以写入新帧。

**2. `presentpts` 不变：**

当前正在显示的帧（present）没有被更新。正常情况下 `presentpts` 应该每 16-33ms 更新一次（对应 30-60fps）。不变意味着 `PrepareNextRender()` 没有执行——即**渲染主循环停滞了**。

**3. `state=1`（READY）持续：**

READY 状态意味着"有新帧准备好，等待 Flip（显示切换）"。持续 READY 说明 `FrameMove()` 虽然在被调用（因为日志还在输出），但 Flip 操作没有完成。可能原因：
- Cocos2d-x 的 `Director::drawScene()` 没有调用 `Flip()`
- 主线程被其他操作阻塞（如 TJS2 脚本执行过慢）

**4. 修复方案：**

```cpp
// 检查 Cocos2d-x 调度器是否正常调用 PresentPicture()
// 在 VideoPresentOverlay（继承自 cocos2d::Node）的 visit() 中添加日志：
void VideoPresentOverlay::visit(Renderer* renderer,
                                const Mat4& parentTransform,
                                uint32_t parentFlags) {
    SPDLOG_DEBUG("[Overlay] visit() called, visible={}",
                 isVisible());
    Node::visit(renderer, parentTransform, parentFlags);
}

// 如果 visit() 正常调用但帧不更新，检查 PresentPicture()：
// 1. m_curpts 是否在累积？
// 2. delta time 是否为 0？
// 3. 帧选择逻辑是否总是选中同一帧？

// 常见根因：Director::setAnimationInterval(1.0/60.0) 没有被调用，
// 或者 Director 处于暂停状态（Director::pause()）
```

### 题目 3：设计一个 CDVDMsg 泄漏检测器

CDVDMsg 使用手动引用计数。请实现一个简单的泄漏检测器，能够在程序退出时报告未释放的消息数量和类型。

要求：
1. 不修改 CDVDMsg 的基类接口（只用继承或静态统计）
2. 支持多线程安全
3. Debug 构建启用，Release 构建编译为空操作

<details>
<summary>查看答案</summary>

```cpp
// === CDVDMsg 泄漏检测器 ===
// 使用全局原子计数器跟踪活跃消息数

#include <atomic>
#include <mutex>
#include <unordered_map>
#include <string>
#include <spdlog/spdlog.h>

#ifdef KRKR2_MEDIA_DEBUG

// 线程安全的泄漏跟踪器
class CDVDMsgLeakTracker {
    // 单例模式
    static CDVDMsgLeakTracker& instance() {
        static CDVDMsgLeakTracker tracker;
        return tracker;
    }

    std::atomic<int> m_totalAlive{0};       // 当前活跃消息总数
    std::atomic<int> m_totalCreated{0};     // 历史创建总数
    std::mutex m_typeMutex;                 // 保护类型统计的互斥锁
    std::unordered_map<int, int> m_typeCount; // 按类型统计活跃数

public:
    static void onCreate(int msgType) {
        auto& t = instance();
        t.m_totalAlive.fetch_add(1);       // 原子自增
        t.m_totalCreated.fetch_add(1);
        {
            std::lock_guard<std::mutex> lock(t.m_typeMutex);
            t.m_typeCount[msgType]++;
        }
    }

    static void onDestroy(int msgType) {
        auto& t = instance();
        t.m_totalAlive.fetch_sub(1);       // 原子自减
        {
            std::lock_guard<std::mutex> lock(t.m_typeMutex);
            t.m_typeCount[msgType]--;
        }
    }

    static void report() {
        auto& t = instance();
        int alive = t.m_totalAlive.load();
        int created = t.m_totalCreated.load();

        if (alive == 0) {
            SPDLOG_INFO("[LeakTracker] No leaks! {} messages "
                        "created and released.", created);
            return;
        }

        SPDLOG_ERROR("[LeakTracker] LEAK DETECTED! {} messages "
                     "still alive (of {} created):",
                     alive, created);

        std::lock_guard<std::mutex> lock(t.m_typeMutex);
        for (auto& [type, count] : t.m_typeCount) {
            if (count > 0) {
                SPDLOG_ERROR("  Type {}: {} leaked", type, count);
            }
        }
    }
};

// 在 CDVDMsg 构造函数和 Release() 中插入跟踪调用：
// CDVDMsg::CDVDMsg(int type) : m_type(type), m_refs(1) {
//     CDVDMsgLeakTracker::onCreate(type);
// }
// void CDVDMsg::Release() {
//     if (--m_refs == 0) {
//         CDVDMsgLeakTracker::onDestroy(m_type);
//         delete this;
//     }
// }

// 在播放器销毁时调用：
// CDVDMsgLeakTracker::report();

#else  // Release 构建：编译为空操作

class CDVDMsgLeakTracker {
public:
    static void onCreate(int) {}    // 空函数，编译器会优化掉
    static void onDestroy(int) {}
    static void report() {}
};

#endif
```

</details>
