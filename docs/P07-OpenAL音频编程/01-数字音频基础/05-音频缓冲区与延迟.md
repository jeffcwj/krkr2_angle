## 什么是音频延迟

### 延迟的定义

音频延迟（Audio Latency）是指从"应用程序决定播放某段声音"到"声音从扬声器/耳机发出"之间的时间差。对于游戏和交互式应用，延迟直接影响用户体验：

| 延迟范围 | 用户感知 | 典型应用 |
|----------|----------|----------|
| < 10 ms | 几乎无感知 | 专业音乐制作（ASIO/JACK） |
| 10-30 ms | 可接受 | 游戏音效、实时语音 |
| 30-100 ms | 明显延迟 | 视觉小说配音（可接受） |
| 100-300 ms | 严重延迟 | 音乐播放（可接受） |
| > 300 ms | 不可接受 | 任何场景都能感知到明显脱节 |

> **视觉小说场景：** KrKr2 主要处理视觉小说游戏，音频延迟在 50-100 ms 是完全可接受的——用户不会注意到对话音频比文字显示晚了 0.05 秒。

### 延迟的三个来源

```
应用程序                    操作系统                    硬件
┌──────────┐            ┌──────────┐            ┌──────────┐
│ 应用缓冲区 │ ─写入──→ │ 系统混音器 │ ─送出──→ │ DAC 缓冲区 │ → 🔊
│ (App      │            │ (OS Mixer │            │ (Hardware │
│  Buffer)  │            │  Buffer)  │            │  Buffer)  │
└──────────┘            └──────────┘            └──────────┘
   T1: 应用延迟            T2: 系统延迟            T3: 硬件延迟

总延迟 = T1 + T2 + T3
```

**T1 — 应用缓冲区延迟：** 应用程序准备的 PCM 数据量。例如一个 4096 样本（44100 Hz 下约 93 ms）的缓冲区意味着至少 93 ms 延迟。这是开发者**可以直接控制**的部分。

**T2 — 系统混音器延迟：** 操作系统将多个应用的音频混合后输出。不同平台差异巨大：

| 平台 | 音频后端 | 典型系统延迟 |
|------|----------|-------------|
| Windows (WASAPI Shared) | Windows Audio Session API | 10-30 ms |
| Windows (WASAPI Exclusive) | 独占模式 | 3-10 ms |
| Linux (PulseAudio) | PulseAudio | 20-50 ms |
| Linux (JACK) | JACK Audio | 2-10 ms |
| macOS (CoreAudio) | Core Audio | 5-15 ms |
| Android (AAudio/Oboe) | AAudio | 10-50 ms |
| Android (OpenSL ES) | OpenSL ES | 20-100 ms |

**T3 — 硬件延迟：** DAC（数模转换器）和放大器的处理时间，通常 < 5 ms，对软件开发者透明。

## 缓冲区大小的计算

### 基础公式

```
缓冲区延迟(ms) = 缓冲区样本数 / 采样率 × 1000

等价公式（从字节计算）：
缓冲区延迟(ms) = 缓冲区字节数 / (采样率 × 声道数 × 每样本字节数) × 1000
```

### 常见缓冲区大小对照

以 44100 Hz 立体声 16 位为例（每帧 4 字节）：

| 缓冲区帧数 | 字节数 | 延迟 | 适用场景 |
|-----------|--------|------|----------|
| 128 | 512 B | 2.9 ms | ASIO 专业音频 |
| 256 | 1024 B | 5.8 ms | 低延迟游戏 |
| 512 | 2048 B | 11.6 ms | 一般游戏 |
| 1024 | 4096 B | 23.2 ms | 普通应用 |
| 2048 | 8192 B | 46.4 ms | 音乐播放 |
| 4096 | 16384 B | 92.9 ms | 流媒体/视觉小说 |

### 缓冲区大小的权衡

```
小缓冲区                              大缓冲区
┌─────────────────────────────────────────────────┐
│  ← 低延迟              高延迟 →                   │
│  ← 高 CPU 开销         低 CPU 开销 →              │
│  ← 容易欠载（卡顿）     不易欠载 →                  │
│  ← 实时交互             后台播放 →                  │
└─────────────────────────────────────────────────┘
```

**欠载（Buffer Underrun / Xrun）：** 当应用程序来不及填充缓冲区时，音频设备无数据可播，产生爆音或静音。缓冲区越小，留给应用的处理时间越短，欠载风险越高。

```cpp
// 计算缓冲区延迟的工具函数
double buffer_latency_ms(uint32_t frames, uint32_t sample_rate) {
    return static_cast<double>(frames) / sample_rate * 1000.0;
}

// 示例
std::cout << "512 帧 @ 44100 Hz = "
          << buffer_latency_ms(512, 44100) << " ms" << std::endl;
// 输出: 512 帧 @ 44100 Hz = 11.6099 ms

std::cout << "256 帧 @ 48000 Hz = "
          << buffer_latency_ms(256, 48000) << " ms" << std::endl;
// 输出: 256 帧 @ 48000 Hz = 5.33333 ms
```

## 双缓冲与三缓冲

### 为什么需要多缓冲

单缓冲的致命问题：**读写冲突**。当音频设备正在读取缓冲区播放时，应用程序不能同时往同一块内存写入新数据。

```
单缓冲（有冲突）：
  时间 →  [==写入==][==播放==][==写入==][==播放==]
                                           ↑ 必须等播放完才能写

双缓冲（交替使用）：
  Buffer A: [==播放==]          [==播放==]
  Buffer B:          [==播放==]          [==播放==]
  应用写入: [写B]     [写A]     [写B]     [写A]
            ↑ 设备播 A 时，应用写 B
```

### 双缓冲（Double Buffering）

使用两块缓冲区交替进行读写，是最常见的方案：

```cpp
#include <AL/al.h>
#include <array>

// 双缓冲 OpenAL 播放示例框架
constexpr int BUFFER_COUNT = 2;
constexpr int BUFFER_FRAMES = 4096;  // 每缓冲区 4096 帧
constexpr int SAMPLE_RATE = 44100;

ALuint source;
std::array<ALuint, BUFFER_COUNT> buffers;

void init_double_buffer() {
    alGenSources(1, &source);
    alGenBuffers(BUFFER_COUNT, buffers.data());

    // 初始填充两个缓冲区
    for (int i = 0; i < BUFFER_COUNT; ++i) {
        std::vector<int16_t> pcm(BUFFER_FRAMES);
        fill_audio_data(pcm.data(), BUFFER_FRAMES);  // 用户提供的填充函数
        alBufferData(buffers[i], AL_FORMAT_MONO16,
                     pcm.data(), BUFFER_FRAMES * sizeof(int16_t),
                     SAMPLE_RATE);
        alSourceQueueBuffers(source, 1, &buffers[i]);
    }
    alSourcePlay(source);
}

void update_stream() {
    ALint processed = 0;
    alGetSourcei(source, AL_BUFFERS_PROCESSED, &processed);

    while (processed-- > 0) {
        ALuint buf;
        alSourceUnqueueBuffers(source, 1, &buf);  // 回收已播完的缓冲区

        std::vector<int16_t> pcm(BUFFER_FRAMES);
        fill_audio_data(pcm.data(), BUFFER_FRAMES);
        alBufferData(buf, AL_FORMAT_MONO16,
                     pcm.data(), BUFFER_FRAMES * sizeof(int16_t),
                     SAMPLE_RATE);
        alSourceQueueBuffers(source, 1, &buf);     // 重新入队
    }

    // 防止播放因缓冲区耗尽而停止
    ALint state;
    alGetSourcei(source, AL_SOURCE_STATE, &state);
    if (state != AL_PLAYING) {
        alSourcePlay(source);  // 重启播放
    }
}
```

**双缓冲延迟 = 2 × 单缓冲区延迟**。上例中：2 × 4096/44100 × 1000 ≈ 186 ms。

### 三缓冲（Triple Buffering）

增加第三个缓冲区，提供更大的容错空间：

```
三缓冲：
  Buffer A: [==播放==]                    [==播放==]
  Buffer B:          [==播放==]
  Buffer C:                    [==播放==]
  应用写入: [写B][写C] [写A]    [写B]     [写C]
            ↑ 有一个额外缓冲区做"缓冲"
```

| 对比 | 双缓冲 | 三缓冲 |
|------|--------|--------|
| 缓冲区数 | 2 | 3 |
| 延迟 | 2 × 单缓冲 | 3 × 单缓冲 |
| 欠载容忍度 | 低（一次慢就卡） | 高（允许偶尔一次慢） |
| 内存占用 | 较小 | 多 50% |
| 适用场景 | 实时交互、低延迟 | 后台解码、网络流 |

> **选择建议：** 游戏音效用双缓冲 + 小缓冲区（512-1024 帧）；音乐/视觉小说用三缓冲 + 大缓冲区（2048-4096 帧）。

### 常见错误：缓冲区数量 vs 缓冲区大小

```cpp
// ❌ 误解：用很多小缓冲区来降低延迟
constexpr int BUFFER_COUNT = 8;    // 8 个缓冲区
constexpr int BUFFER_FRAMES = 256; // 每个 256 帧
// 实际延迟 = 8 × 256/44100 × 1000 = 46.4 ms（并没有变低！）
// 还增加了 alSourceQueueBuffers 的调用开销

// ✅ 正确理解：减少每个缓冲区的大小，保持合理数量
constexpr int BUFFER_COUNT = 2;     // 双缓冲
constexpr int BUFFER_FRAMES = 512;  // 每个 512 帧
// 实际延迟 = 2 × 512/44100 × 1000 = 23.2 ms（真正降低了）
```

### 常见错误：没有处理播放停止

```cpp
// ❌ 只检查 processed，不检查播放状态
void bad_update() {
    ALint processed;
    alGetSourcei(source, AL_BUFFERS_PROCESSED, &processed);
    // ... 填充缓冲区 ...
    // 问题：如果所有缓冲区都播完了，source 会自动停止
    // 之后 processed 永远为 0，再也不会恢复！
}

// ✅ 必须检查并重启
void good_update() {
    // ... 填充缓冲区 ...
    ALint state;
    alGetSourcei(source, AL_SOURCE_STATE, &state);
    if (state != AL_PLAYING) {
        alSourcePlay(source);  // 重启
        std::cerr << "警告: 缓冲区欠载，重启播放" << std::endl;
    }
}
```

## 动手实践

### 缓冲区延迟计算器

```cpp
// latency_calc.cpp — 交互式缓冲区延迟计算器
#include <iostream>
#include <iomanip>
#include <cmath>

int main() {
    uint32_t sample_rate, channels, bits, buf_frames, buf_count;

    std::cout << "=== 音频缓冲区延迟计算器 ===" << std::endl;
    std::cout << "采样率 (Hz): ";     std::cin >> sample_rate;
    std::cout << "声道数: ";          std::cin >> channels;
    std::cout << "位深: ";            std::cin >> bits;
    std::cout << "每缓冲区帧数: ";     std::cin >> buf_frames;
    std::cout << "缓冲区数量: ";       std::cin >> buf_count;

    double bytes_per_frame = channels * (bits / 8.0);
    double buf_bytes = buf_frames * bytes_per_frame;
    double single_latency = static_cast<double>(buf_frames)
                            / sample_rate * 1000.0;
    double total_latency = single_latency * buf_count;

    std::cout << std::fixed << std::setprecision(2);
    std::cout << "\n=== 计算结果 ===" << std::endl;
    std::cout << "每帧字节数:       " << bytes_per_frame << " B" << std::endl;
    std::cout << "单缓冲区大小:     " << buf_bytes << " B ("
              << buf_bytes / 1024.0 << " KB)" << std::endl;
    std::cout << "单缓冲区延迟:     " << single_latency << " ms" << std::endl;
    std::cout << "总缓冲延迟:       " << total_latency << " ms" << std::endl;
    std::cout << "总缓冲区内存:     " << buf_bytes * buf_count / 1024.0
              << " KB" << std::endl;

    // 评级
    if (total_latency < 15)
        std::cout << "评级: ★★★ 专业低延迟" << std::endl;
    else if (total_latency < 50)
        std::cout << "评级: ★★ 游戏/交互适用" << std::endl;
    else if (total_latency < 150)
        std::cout << "评级: ★ 音乐/视觉小说适用" << std::endl;
    else
        std::cout << "评级: ⚠ 延迟较高，检查配置" << std::endl;

    return 0;
}
```

3. 解码线程和播放线程异步运行，需要足够的缓冲区做"蓄水池"

## 本节小结

- 音频延迟 = 应用缓冲延迟(T1) + 系统混音延迟(T2) + 硬件延迟(T3)，T1 是开发者可控的
- 缓冲区延迟(ms) = 帧数 / 采样率 × 1000，总延迟 = 单缓冲区延迟 × 缓冲区数量
- 小缓冲区 = 低延迟但高 CPU 开销和欠载风险；大缓冲区 = 高延迟但稳定
- 双缓冲用于实时交互（游戏音效），三缓冲用于流式播放（音乐、视觉小说）
- KrKr2 使用三缓冲 + 48000 Hz，适合视觉小说的低交互性需求
- 必须检查 `AL_SOURCE_STATE` 并在欠载时重启播放，否则流式播放会永久停止

## 练习题与答案

### 题目 1：延迟计算

一个 48000 Hz、立体声、16 位的音频流，使用 3 个缓冲区，每个 2048 帧。计算：(a) 单缓冲区的字节大小和延迟，(b) 总缓冲延迟，(c) 每秒需要调用 `update_stream()` 多少次才能不欠载。

<details>
<summary>查看答案</summary>

```
(a) 单缓冲区：
    字节 = 2048 帧 × 2 声道 × 2 字节 = 8192 B = 8 KB
    延迟 = 2048 / 48000 × 1000 = 42.67 ms

(b) 总缓冲延迟 = 42.67 × 3 = 128.0 ms

(c) 每秒消耗帧数 = 48000
    每个缓冲区 = 2048 帧
    每秒消耗缓冲区数 = 48000 / 2048 ≈ 23.4 个
    因此 update_stream() 至少每秒调用 24 次（约每 42 ms 一次）
    但为了安全，建议以更高频率调用（如每 20 ms 一次）
```

</details>

### 题目 2：欠载诊断

你的流式播放器出现了周期性的爆音（每隔约 1 秒一次），但 CPU 使用率很低。可能的原因和修复方案？

<details>
<summary>查看答案</summary>

最可能的原因是 **update_stream() 的调用频率不够**，而非 CPU 不足。

**诊断步骤：**
1. 在 `update_stream()` 中添加日志，打印 `processed` 值和调用间隔
2. 如果 `processed` 经常 ≥ 2，说明缓冲区来不及补充

**常见原因及修复：**

```cpp
// 原因 1：update 频率太低（如放在 60fps 渲染循环中但帧率不稳定）
// 修复：使用独立的音频线程，或提高调用频率
std::thread audio_thread([]() {
    while (running) {
        update_stream();
        std::this_thread::sleep_for(std::chrono::milliseconds(10));
    }
});

// 原因 2：缓冲区数量不足（只用了双缓冲）
// 修复：改为三缓冲，增加容错空间
constexpr int BUFFER_COUNT = 3;  // 2 → 3

// 原因 3：单缓冲区太小
// 修复：增大缓冲区帧数
constexpr int BUFFER_FRAMES = 4096;  // 1024 → 4096
```

</details>

### 题目 3：为不同应用设计缓冲区参数

为以下三种应用设计合理的缓冲区参数（采样率、缓冲区帧数、缓冲区数量），并说明理由：
(a) 音乐节奏游戏（需要精确的节拍反馈）
(b) 有声小说播放器（后台长时间播放）
(c) 实时语音聊天

<details>
<summary>查看答案</summary>

```
(a) 音乐节奏游戏：
    采样率: 48000 Hz（现代标准）
    缓冲区帧数: 256（延迟 = 256/48000×1000 = 5.3 ms）
    缓冲区数量: 2（双缓冲，总延迟 ≈ 10.7 ms）
    理由: 节奏游戏需要 <20ms 延迟来确保打击感；
          双缓冲减少延迟；需要高优先级音频线程防止欠载

(b) 有声小说播放器：
    采样率: 44100 Hz（文件原始采样率，避免重采样）
    缓冲区帧数: 4096（延迟 = 92.9 ms）
    缓冲区数量: 3（三缓冲，总延迟 ≈ 278.7 ms）
    理由: 播放延迟不敏感；大缓冲区降低 CPU 唤醒频率，省电；
          三缓冲容忍后台任务抢占

(c) 实时语音聊天：
    采样率: 48000 Hz
    缓冲区帧数: 480（延迟 = 10 ms，恰好是 Opus 编码的一帧）
    缓冲区数量: 2（双缓冲，总延迟 ≈ 20 ms）
    理由: 语音通话需要 <30ms 延迟；480 帧 = 10ms 对齐 Opus 编码；
          加上网络延迟，端到端约 50-100 ms
```

</details>
