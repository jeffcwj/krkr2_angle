## 为什么要学编译原理

你可能会问：我只是想读懂 KrKr2 的 TJS2 脚本引擎，为什么需要学编译原理？

答案很直接——TJS2 **就是**一个完整的编译器 + 虚拟机系统。它把 `.tjs` 脚本文件编译成字节码，然后在虚拟机中执行。如果你不理解编译器的工作原理，那么 `tjsLex.cpp`（词法器）、`bison/tjs.y`（语法文件）、`tjsInterCodeGen.cpp`（代码生成器）、`tjsInterCodeExec.cpp`（虚拟机）这些文件对你来说就只是天书。

编译原理不只是"学院派知识"。调试脚本执行问题、扩展 TJS2 语法、优化脚本性能、实现插件系统——这些实际工作都需要你理解编译器各阶段的职责。所以，这不是"可选的理论课"，而是读懂 KrKr2 核心代码的必修课。

## 编译器的五个经典阶段

一个编译器（或解释器）将源代码转化为可执行形式，这个过程可以分解为五个经典阶段。我们用一个 ASCII 图来展示完整的流水线：

```
源代码 ──▶ [1.词法分析] ──Token流──▶ [2.语法分析] ──AST──▶ [3.语义分析]
                                                              │
                                                        标注AST/符号表
                                                              │
                                                              ▼
                                                       [4.中间代码生成]
                                                              │
                                                          字节码/IR
                                                              │
                                                              ▼
                                                       [5.目标代码/执行]
```

下面逐一讲解每个阶段。

### 阶段 1：词法分析（Lexical Analysis）

**输入：** 源代码字符流（一串字符）
**输出：** Token 序列（词素流）

词法分析器（Lexer，也叫 Scanner 或 Tokenizer）的工作是把源代码的字符流分割成有意义的"词"——称为 **Token**（词法单元）。这类似于你阅读英文时，把连续的字母序列识别为单词。

例如，对于这行 TJS2 代码：

```javascript
var x = 42 + y;
```

词法器会产生以下 Token 序列：

| Token 类型 | 词素（Lexeme） | 含义 |
|-----------|---------------|------|
| `T_VAR` | `var` | 变量声明关键字 |
| `T_SYMBOL` | `x` | 标识符 |
| `T_EQUAL` | `=` | 赋值运算符 |
| `T_CONSTVAL` | `42` | 整数字面量 |
| `T_PLUS` | `+` | 加法运算符 |
| `T_SYMBOL` | `y` | 标识符 |
| `T_SEMICOLON` | `;` | 语句结束符 |

词法器需要处理的事情比看起来要多得多：跳过空白和注释、识别关键字（`var` 是关键字而不是普通标识符）、解析字符串中的转义字符（`\n` 表示换行）、处理数字的各种格式（十进制、十六进制 `0xFF`、二进制 `0b1010`）等。

**示例 1：最简词法器**

我们用 C++ 实现一个能识别数字和加号的极简词法器，体验 Token 化的核心逻辑：

```cpp
#include <iostream>
#include <string>
#include <vector>

// Token 类型枚举
enum class TokenType {
    Number,     // 数字字面量
    Plus,       // +
    Minus,      // -
    EndOfFile,  // 输入结束
    Unknown     // 无法识别
};

// Token 结构体：类型 + 词素文本
struct Token {
    TokenType type;
    std::string lexeme;
};

// 最简词法器：逐字符扫描
std::vector<Token> tokenize(const std::string& source) {
    std::vector<Token> tokens;
    size_t pos = 0;

    while (pos < source.size()) {
        char ch = source[pos];

        // 跳过空白字符
        if (ch == ' ' || ch == '\t' || ch == '\n') {
            pos++;
            continue;
        }

        // 识别数字：连续读取所有数字字符
        if (ch >= '0' && ch <= '9') {
            std::string num;
            while (pos < source.size() && source[pos] >= '0'
                   && source[pos] <= '9') {
                num += source[pos++];
            }
            tokens.push_back({TokenType::Number, num});
            continue;
        }

        // 识别运算符
        if (ch == '+') {
            tokens.push_back({TokenType::Plus, "+"});
            pos++;
            continue;
        }
        if (ch == '-') {
            tokens.push_back({TokenType::Minus, "-"});
            pos++;
            continue;
        }

        // 无法识别的字符
        tokens.push_back({TokenType::Unknown,
                          std::string(1, ch)});
        pos++;
    }

    tokens.push_back({TokenType::EndOfFile, ""});
    return tokens;
}

int main() {
    std::string input = "12 + 34 - 5";
    auto tokens = tokenize(input);

    // 打印所有 Token
    for (const auto& tok : tokens) {
        const char* name = "?";
        switch (tok.type) {
            case TokenType::Number:  name = "Number";  break;
            case TokenType::Plus:    name = "Plus";    break;
            case TokenType::Minus:   name = "Minus";   break;
            case TokenType::EndOfFile: name = "EOF";   break;
            case TokenType::Unknown: name = "Unknown"; break;
        }
        std::cout << "[" << name << "] \"" << tok.lexeme
                  << "\"\n";
    }
    return 0;
}
```

编译运行（保存为 `lexer_demo.cpp`）：

```bash
g++ -std=c++17 -o lexer_demo lexer_demo.cpp && ./lexer_demo  # Linux/macOS
cl /std:c++17 /EHsc lexer_demo.cpp && lexer_demo.exe          # Windows
```

输出：

```
[Number] "12"
[Plus] "+"
[Number] "34"
[Minus] "-"
[Number] "5"
[EOF] ""
```

这就是词法分析的本质：**逐字符扫描，按规则分组，输出 Token 序列**。

### 阶段 2：语法分析（Syntax Analysis / Parsing）

**输入：** Token 序列
**输出：** 抽象语法树（Abstract Syntax Tree，AST）

语法分析器（Parser）的工作是检查 Token 序列是否符合语言的语法规则，并构建一棵表示程序结构的树。如果说词法分析像"识别单词"，语法分析就像"理解句子结构"。

对于 `var x = 42 + y;`，语法分析器会构建如下 AST：

```
        VarDecl
       /       \
     "x"      BinOp(+)
              /       \
          Literal(42)  Ident("y")
```

AST 丢弃了源代码中的空白、括号等"语法糖"，只保留程序的逻辑结构。例如表达式 `(1 + 2) * 3` 和 `1 + 2 * 3` 会生成不同的 AST，因为运算符优先级不同。

语法分析有两大流派：
- **自顶向下**（Top-Down）：从根节点开始，尝试推导出 Token 序列。代表方法：递归下降（Recursive Descent）
- **自底向上**（Bottom-Up）：从 Token 序列开始，逐步归约为根节点。代表方法：LALR(1)，Bison 生成的就是此类

**常见错误 1：语法错误不等于逻辑错误**

> 语法分析只检查结构是否合法，不检查语义。`var x = "hello" + 42;` 语法完全正确（赋值语句中有加法表达式），但在某些强类型语言中会在语义分析阶段报错。TJS2 是动态类型语言，这个表达式在 TJS2 中完全合法——字符串和数字相加会自动转换。

### 阶段 3：语义分析（Semantic Analysis）

**输入：** AST
**输出：** 标注后的 AST + 符号表

语义分析器遍历 AST，检查程序的"含义"是否正确。它的主要任务包括：

1. **符号解析**（Name Resolution）：变量 `y` 是否已声明？在哪个作用域？
2. **类型检查**（Type Checking）：`42 + "hello"` 在静态类型语言中是类型错误
3. **作用域管理**（Scope Management）：内层函数能否访问外层变量？
4. **常量折叠**（Constant Folding）：`3 + 4` 可以在编译时计算为 `7`

对于动态类型语言（如 TJS2、JavaScript、Python），类型检查大多推迟到运行时，但符号解析和作用域管理仍在编译期完成。

语义分析的核心数据结构是**符号表**（Symbol Table）——它记录每个变量的名称、所在作用域和类型信息。想象一个"嵌套的字典"：

```
全局作用域: { x: declared }
  └─ 函数作用域: { y: declared }
```

查找变量时**从内向外逐层搜索**：在函数内查找 `x`，先查函数作用域（没找到），再查全局作用域（找到了）。退出函数作用域后，`y` 就不再可见。

> 完整的符号表实现（含 C++ 代码）将在 [第4章：语义分析](../04-语义分析/01-符号表与作用域.md) 中深入讲解。

### 阶段 4：中间代码生成（Intermediate Code Generation）

**输入：** 标注后的 AST
**输出：** 中间表示（Intermediate Representation，IR）

编译器不一定直接生成机器码。大多数现代编译器先生成一种**中间表示**——它比源代码更低级（更接近机器），但比机器码更高级（不依赖具体硬件）。

常见的 IR 形式包括：
- **三地址码**（Three-Address Code）：每条指令最多三个操作数，形如 `t1 = a + b`
- **SSA 形式**（Static Single Assignment）：每个变量只赋值一次，便于优化（LLVM 使用此形式）
- **字节码**（Bytecode）：紧凑的二进制指令序列，由虚拟机解释执行（Java/.NET/TJS2 使用此形式）

对于 `42 + y`，TJS2 生成的字节码类似于：

```
CONST  %r2, 42     // 将常量 42 加载到寄存器 r2
CP     %r3, %y     // 将变量 y 的值复制到 r3
ADD    %r2, %r3    // r2 = r2 + r3
```

> 注意：TJS2 的字节码使用**基于寄存器**的设计（类似 Lua 5），而非 Java 那样的纯栈式设计。寄存器实际上是虚拟寄存器，对应栈帧中的槽位。

**常见错误 2：混淆字节码与机器码**

> 字节码（Bytecode）是给虚拟机执行的指令，可以跨平台；机器码（Machine Code）是给 CPU 直接执行的指令，特定于架构（x86、ARM 等）。TJS2 生成的是字节码，不是机器码。

### 阶段 5：目标代码生成 / 执行

**输入：** 中间表示
**输出：** 可执行结果

最后一步取决于语言实现方式：
- **编译到机器码**：C/C++/Rust 的编译器生成本地机器指令，由 CPU 直接执行
- **解释执行字节码**：Java/Python/TJS2 的虚拟机逐条读取字节码并执行
- **JIT 编译**：V8（JavaScript）、HotSpot（Java）在运行时将热点字节码编译为机器码

TJS2 属于第二类——虚拟机解释执行。`tjsInterCodeExec.cpp`（3109 行）中有一个巨大的 switch-case 分派循环，逐条解释字节码指令。

**示例 2：最小表达式求值器（从 Token 到执行）**

这个示例把词法分析、语法分析（递归下降）和直接解释执行串起来，形成一个完整的微型解释器。它能计算 `12 + 34 - 5` 这样的加减表达式：

```cpp
#include <iostream>
#include <string>
#include <vector>
#include <stdexcept>

// ========== 第一阶段：词法分析 ==========
enum class TokenType { Number, Plus, Minus, Eof };
struct Token { TokenType type; int value; }; // value 仅 Number 使用

std::vector<Token> tokenize(const std::string& src) {
    std::vector<Token> tokens;
    for (size_t i = 0; i < src.size(); ) {
        if (src[i] == ' ') { i++; continue; }
        if (src[i] >= '0' && src[i] <= '9') {
            int val = 0;
            while (i < src.size() && src[i] >= '0'
                   && src[i] <= '9')
                val = val * 10 + (src[i++] - '0');
            tokens.push_back({TokenType::Number, val});
        } else if (src[i] == '+') {
            tokens.push_back({TokenType::Plus, 0}); i++;
        } else if (src[i] == '-') {
            tokens.push_back({TokenType::Minus, 0}); i++;
        } else {
            throw std::runtime_error(
                std::string("Unexpected: ") + src[i]);
        }
    }
    tokens.push_back({TokenType::Eof, 0});
    return tokens;
}

// ========== 第二阶段：语法分析 + AST ==========
// 文法：expr = number (('+' | '-') number)*
struct ASTNode {
    enum Kind { Num, Add, Sub } kind;
    int value;                     // Num 时使用
    ASTNode* left  = nullptr;      // Add/Sub 时使用
    ASTNode* right = nullptr;
};

class Parser {
    const std::vector<Token>& tokens_;
    size_t pos_ = 0;
    Token current() { return tokens_[pos_]; }
    Token consume(TokenType expected) {
        if (current().type != expected)
            throw std::runtime_error("Parse error");
        return tokens_[pos_++];
    }
public:
    explicit Parser(const std::vector<Token>& t)
        : tokens_(t) {}

    ASTNode* parseExpr() {
        auto* node = new ASTNode{
            ASTNode::Num,
            consume(TokenType::Number).value};
        while (current().type == TokenType::Plus
               || current().type == TokenType::Minus) {
            auto op = current().type;
            pos_++;
            auto* right = new ASTNode{
                ASTNode::Num,
                consume(TokenType::Number).value};
            node = new ASTNode{
                op == TokenType::Plus
                    ? ASTNode::Add : ASTNode::Sub,
                0, node, right};
        }
        return node;
    }
};

// ========== 第三/四/五阶段：直接解释执行 ==========
int evaluate(const ASTNode* node) {
    if (node->kind == ASTNode::Num) return node->value;
    int l = evaluate(node->left), r = evaluate(node->right);
    return node->kind == ASTNode::Add ? l + r : l - r;
}

void freeAST(ASTNode* n) {      // 递归释放
    if (!n) return;
    freeAST(n->left); freeAST(n->right); delete n;
}

int main() {
    std::string input = "12 + 34 - 5";
    auto tokens = tokenize(input);     // 阶段1：词法分析
    Parser parser(tokens);
    ASTNode* ast = parser.parseExpr(); // 阶段2：语法分析
    std::cout << input << " = "
              << evaluate(ast) << "\n"; // 阶段3-5：执行
    freeAST(ast);
    return 0;
}
```

编译运行（保存为 `mini_eval.cpp`）：

```bash
g++ -std=c++17 -o mini_eval mini_eval.cpp && ./mini_eval  # Linux/macOS
cl /std:c++17 /EHsc mini_eval.cpp && mini_eval.exe          # Windows
```

输出：

```
12 + 34 - 5 = 41
```

这个 ~90 行的程序浓缩了编译器五阶段中的三个核心阶段（词法→语法→执行）。

## 本节小结

- 编译器将源代码转化为可执行形式，经典流程分为**五个阶段**：词法→语法→语义→中间代码→执行
- **词法分析**把字符流切成 Token 序列，**语法分析**组织成 AST，**语义分析**检查含义正确性
- **中间代码生成**产出与平台无关的 IR（TJS2 使用寄存器式字节码），**执行阶段**负责最终产出
- TJS2 完整链路：`tjsLex.cpp` → `tjs.y` → `tjsInterCodeGen` → `tjsInterCodeExec`

### 题目 1：阶段匹配

下列操作分别在编译器的哪个阶段完成？

- (a) 把 `var` 识别为关键字而非普通标识符
- (b) 发现 `x + ;` 缺少右操作数
- (c) 发现变量 `y` 未声明就使用
- (d) 把 `a + b` 翻译为 `ADD %r1, %r2`
- (e) 执行 `ADD` 指令并得到结果

<summary>查看答案</summary>

- (a) **词法分析**：词法器查关键字表，发现 `var` 是保留字，产出 `T_VAR` Token 而非 `T_SYMBOL`

- (d) **中间代码生成**：将 AST 中的 `BinOp(+, a, b)` 节点翻译为字节码指令
- (e) **目标代码执行**（虚拟机）：VM 读取 `ADD` 操作码，执行寄存器加法

### 题目 2：动手改代码

修改本节"示例 2"的 `evaluate()` 函数，让它在计算时打印每一步的中间过程，例如对 `12 + 34 - 5` 输出：

```
EVAL: 12
EVAL: 34
ADD: 12 + 34 = 46
EVAL: 5
SUB: 46 - 5 = 41
```

<details>
<summary>查看答案</summary>

```cpp
int evaluate(const ASTNode* node) {
    switch (node->kind) {
        case ASTNode::Num:
            std::cout << "EVAL: " << node->value << "\n";
            return node->value;
        case ASTNode::Add: case ASTNode::Sub: {
            int l = evaluate(node->left);
            int r = evaluate(node->right);
            bool isAdd = (node->kind == ASTNode::Add);
            int result = isAdd ? l + r : l - r;
            std::cout << (isAdd ? "ADD: " : "SUB: ")
                      << l << (isAdd ? " + " : " - ") << r
                      << " = " << result << "\n";
            return result;
        }
    }
    return 0;
}
```

关键改动：`Num` 分支打印叶节点值，`Add`/`Sub` 分支先递归求值左右子树再打印运算过程。递归自然顺序保证输出与计算顺序一致。

</details>
