## 前端与后端的划分

编译器的五个阶段（见上一节）可以分为两大部分：

```
          前端（Front-End）                    后端（Back-End）
 ┌─────────────────────────────┐    ┌─────────────────────────────┐
 │ 词法分析 → 语法分析 → 语义分析 │    │ 中间代码生成 → 目标代码生成   │
 │                             │    │                             │
 │  与源语言相关                 │    │  与目标平台相关              │
 │  与目标平台无关               │    │  与源语言无关               │
 └─────────────────────────────┘    └─────────────────────────────┘
              │                              ▲
              └──── 中间表示（IR）────────────┘
```

**前端**负责理解源代码——它与具体的编程语言紧密绑定，但完全不关心目标平台。**后端**负责生成可执行代码——它只看 IR，不关心源代码是什么语言写的。

### 为什么要划分前端和后端？

这种划分的核心价值是**复用**：

| 场景 | 做法 | 复用部分 |
|------|------|----------|
| 一种语言 → 多个平台 | 一个前端 + 多个后端 | 前端 |
| 多种语言 → 一个平台 | 多个前端 + 一个后端 | 后端 |
| M 种语言 × N 个平台 | M 个前端 + N 个后端 | 通过 IR 解耦 |

**示例 1：LLVM 的前端/后端架构**

经典例子是 LLVM。它的 IR（LLVM IR）让不同语言的前端共享同一套优化器和代码生成后端：

```
前端                     中间层                 后端
┌──────┐              ┌──────────┐          ┌─────────┐
│ Clang │─┐            │          │    ┌────▶│ x86-64  │
│(C/C++)│ │            │  LLVM IR │    │     └─────────┘
└──────┘ │  LLVM IR   │    +     │    │     ┌─────────┐
┌──────┐ ├──────────▶ │  优化器   │────┼────▶│  ARM64  │
│ rustc│ │            │          │    │     └─────────┘
│(Rust) │ │            └──────────┘    │     ┌─────────┐
└──────┘ │                             └────▶│  WASM   │
┌──────┐ │                                   └─────────┘
│ swiftc│─┘
│(Swift)│
└──────┘
```

3 个前端 × 3 个后端 = 通过 IR 只需 **6 个组件**。如果没有 IR，需要 3 × 3 = **9 个独立编译器**。语言和平台越多，差距越大：10 种语言 × 5 个平台只需 15 个组件（而非 50 个）。

### TJS2 的前端和后端

TJS2 虽然不像 LLVM 那样支持多语言多平台，但它同样遵循前端/后端划分：

```
前端                              后端
┌──────────┐  Token  ┌────────┐  AST  ┌──────────────┐  字节码  ┌──────────────┐
│ tjsLex   │───────▶│ tjs.y  │──────▶│tjsInterCodeGen│───────▶│tjsInterCodeExec│
│ 词法分析  │        │ 语法分析│       │语义+代码生成   │        │ 虚拟机执行     │
└──────────┘        └────────┘       └──────────────┘        └──────────────┘
```

| 部分 | 源文件 | 职责 |
|------|--------|------|
| 前端-词法 | `cpp/core/tjs2/tjsLex.cpp` (~1874行) | 字符流 → Token 序列 |
| 前端-语法 | `cpp/core/tjs2/bison/tjs.y` (862行) | Token → AST |
| 前端-语义+IR | `cpp/core/tjs2/tjsInterCodeGen.cpp` | AST → 字节码（含符号解析） |
| 后端-执行 | `cpp/core/tjs2/tjsInterCodeExec.cpp` (3109行) | 字节码 → 运行结果 |

> **注意**：TJS2 的"语义分析"和"代码生成"合并在 `tjsInterCodeGen.cpp` 中完成——这在小型编译器中很常见，不需要像教科书那样严格分层。

### 划分的边界在哪里？

实际编译器中，前端和后端的边界并不总是泾渭分明：

| 情况 | 说明 |
|------|------|
| 教科书划分 | 前端 = 词法+语法+语义，后端 = IR优化+目标代码 |
| 实际编译器 | 语义分析经常和代码生成混在一起（如 TJS2） |
| LLVM 风格 | 前端只负责到 LLVM IR，优化和代码生成全部属于"中间层+后端" |
| 脚本语言 | 整个"后端"就是一个 VM 解释器，没有传统意义上的代码生成 |

**关键判断标准**：前端处理**语言特性**（语法规则、关键字、类型系统），后端处理**执行特性**（指令格式、寄存器分配、运行时调度）。

## 完整追踪：一行代码的编译之旅

让我们追踪 `var x = 3 + 4;` 从文本到执行的每一步：

```
┌─ 源代码 ─────────────────────────────────────────────────────┐
│  var x = 3 + 4;                                              │
└──────────────────────────────────────────────────────────────┘
                            │ 阶段1：词法分析
                            ▼
┌─ Token 序列 ─────────────────────────────────────────────────┐
│  [VAR] [SYMBOL:"x"] [=] [INT:3] [+] [INT:4] [;]            │
└──────────────────────────────────────────────────────────────┘
                            │ 阶段2：语法分析
                            ▼
┌─ AST ────────────────────────────────────────────────────────┐
│        VarDecl                                               │
│       /       \                                              │
│     "x"      BinOp(+)                                       │
│              /       \                                       │
│          Lit(3)    Lit(4)                                     │
└──────────────────────────────────────────────────────────────┘
                            │ 阶段3：语义分析
                            ▼
┌─ 标注 AST ───────────────────────────────────────────────────┐
│  同上 AST + 符号表条目 { x: local, slot=2 }                  │
│  常量折叠: BinOp(+, 3, 4) → Lit(7)   ← 可选优化             │
└──────────────────────────────────────────────────────────────┘
                            │ 阶段4：中间代码生成
                            ▼
┌─ 字节码 ─────────────────────────────────────────────────────┐
│  CONST  %r3, 3       // 加载常量 3 到寄存器 r3               │
│  CONST  %r4, 4       // 加载常量 4 到寄存器 r4               │
│  ADD    %r3, %r4     // r3 = r3 + r4 = 7                    │
│  CP     %r2, %r3     // 将结果复制到变量 x 的槽位 r2         │
└──────────────────────────────────────────────────────────────┘
                            │ 阶段5：虚拟机执行
                            ▼
┌─ 执行结果 ───────────────────────────────────────────────────┐
│  寄存器状态: r2 = 7, r3 = 7, r4 = 4                         │
│  变量 x 的值为整数 7                                         │
└──────────────────────────────────────────────────────────────┘
```

让我们逐阶段详细分析：

### 阶段 1 详解：词法器如何处理这行代码

词法器（`tjsLex.cpp` 的 `GetToken()` 方法）从左到右扫描字符：

1. 读到 `v`、`a`、`r` → 字母序列 → 查关键字哈希表 → 命中 `var` → 产出 `T_VAR`
2. 跳过空格
3. 读到 `x` → 字母序列 → 查关键字表 → 未命中 → 产出 `T_SYMBOL("x")`
4. 跳过空格
5. 读到 `=` → 单字符运算符 → 产出 `T_EQUAL`（注意要检查下一个字符是否为 `=`，避免与 `==` 混淆）
6. 读到 `3` → 数字 → 调用数字解析 → 产出 `T_CONSTVAL(3)`
7. 读到 `+` → 产出 `T_PLUS`
8. 读到 `4` → 产出 `T_CONSTVAL(4)`
9. 读到 `;` → 产出 `T_SEMICOLON`

### 阶段 2 详解：语法分析器如何构建 AST

Bison 生成的 LALR(1) 解析器按照 `tjs.y` 中的文法规则：

1. 看到 `T_VAR` → 进入 `var_def` 规则
2. 看到 `T_SYMBOL("x")` → 记录变量名
3. 看到 `T_EQUAL` → 期待一个初始化表达式
4. 看到 `T_CONSTVAL(3)` → 创建 `Literal(3)` 节点
5. 看到 `T_PLUS` → 创建 `BinOp(+)` 节点
6. 看到 `T_CONSTVAL(4)` → 创建 `Literal(4)` 节点
7. 归约：`BinOp(+)` 的左右子节点分别是 `Literal(3)` 和 `Literal(4)`
8. 归约：`VarDecl("x")` 的初始值是 `BinOp(+)` 节点

### 阶段 3-4 详解：语义分析 + 代码生成

`tjsInterCodeGen.cpp` 遍历 AST，同时完成符号注册和字节码生成：

1. 遇到 `VarDecl("x")` → 在当前作用域注册变量 `x`，分配寄存器槽位 `%r2`
2. 遇到 `Literal(3)` → 将 3 加入常量池，生成 `CONST %r3, 3`
3. 遇到 `Literal(4)` → 将 4 加入常量池，生成 `CONST %r4, 4`
4. 遇到 `BinOp(+)` → 生成 `ADD %r3, %r4`（结果存回 `%r3`）
5. 赋值给 `x` → 生成 `CP %r2, %r3`

### 阶段 5 详解：虚拟机执行

`tjsInterCodeExec.cpp` 的 switch-dispatch 循环：

```cpp
// 伪代码，简化自 tjsInterCodeExec.cpp
while (true) {
    tjs_uint32 code = *code_ptr++;      // 取指令
    switch (code & 0xFF) {              // 解码操作码
        case VM_CONST:
            reg[ra] = const_pool[rb];   // 常量加载
            break;
        case VM_ADD:
            reg[ra] += reg[rb];         // 寄存器加法
            break;
        case VM_CP:
            reg[ra] = reg[rb];          // 寄存器复制
            break;
        // ... 更多操作码
    }
}
```

> **注意**：如果编译器实现了常量折叠优化，阶段 3 就能把 `3 + 4` 直接计算为 `7`，这样阶段 4 只需生成 `CONST %r2, 7` 一条指令。TJS2 确实实现了部分常量折叠。

**常见错误 1：混淆编译时和运行时**

> 初学者常把"编译器做的事"和"运行时做的事"搞混。简单判断方法：如果一个操作只需要看源代码文本就能完成（如把 `3 + 4` 算出 `7`），它就**可以**在编译时做。如果一个操作需要知道变量的实际值（如 `x + y`，x 和 y 的值在运行时才确定），它就**必须**在运行时做。

### 示例 2：追踪一个带函数调用的表达式

追踪更复杂的代码 `var result = Math.sqrt(16);`，关注与简单表达式的差异：

| 阶段 | 处理 | 差异点 |
|------|------|--------|
| 词法 | `[VAR][SYMBOL:result][=][SYMBOL:Math][.][SYMBOL:sqrt][(][INT:16][)][;]` | 多了 `.` 和 `()`，词法器需区分成员访问和函数调用 |
| 语法 | `VarDecl("result", CallExpr(MemberAccess("Math","sqrt"), [Lit(16)]))` | AST 更深：嵌套了成员访问和函数调用节点 |
| 语义 | 查找 `Math` 是否已声明 → 全局对象。`sqrt` 是其属性 → 无需编译时解析 | 动态类型语言中属性访问推迟到运行时 |
| 字节码 | `GPD %r3, %Math, "sqrt"` → `CALL %r3, %r3, [%r4]` → `CP %r2, %r3` | 需要属性访问指令 `GPD` 和函数调用指令 `CALL` |
| 执行 | VM 查找 Math 对象 → 获取 sqrt 属性 → 调用函数 → 返回 4.0 | 运行时需要对象系统和函数调度 |

**常见错误 2：以为属性访问在编译时解析**

> 在静态类型语言（如 C++）中，`obj.method()` 的方法地址在编译时就确定了（除虚函数外）。但在动态类型语言（如 TJS2、JavaScript）中，对象的属性集合可以在运行时改变，所以 `Math.sqrt` 必须在运行时动态查找。这就是为什么 TJS2 字节码中需要 `GPD`（Get Property Direct）指令。

| 编译阶段 | 源文件 | 行数 | 关键入口 |
|----------|--------|------|----------|
| 词法分析 | `cpp/core/tjs2/tjsLex.cpp` | ~1874 | `tTJSLexicalAnalyzer::GetToken()` (L1255) |
| 语法分析 | `cpp/core/tjs2/bison/tjs.y` | 862 | Bison 生成的 LALR(1) 解析器 |

相关文件：
- `cpp/core/tjs2/tjsLex.h` 第 49-140 行 — `tTJSLexicalAnalyzer` 类声明，包含位置追踪、Token 缓存等成员

## 本节小结

- 编译器分为**前端**（语言相关：词法+语法+语义）和**后端**（平台相关：IR优化+代码生成/执行），通过 **IR** 解耦
- 前端/后端划分的核心价值是**复用**：M 种语言 × N 个平台只需 M+N 个组件（而非 M×N）
- TJS2 的前端是 `tjsLex` + `tjs.y` + `tjsInterCodeGen`，后端是 `tjsInterCodeExec`
- **编译时**处理源码文本可确定的信息，**运行时**处理需要实际值的操作
- 动态类型语言（如 TJS2）将大量检查推迟到运行时，这是其灵活性的代价

## 练习题与答案

### 题目 1：前端与后端

假设你要设计一个能同时编译 TJS2 和 Lua 脚本的系统，并且需要同时支持 x86 和 ARM 两个平台。

- (a) 你需要实现几个前端？几个后端？
- (b) 总共需要多少个独立组件（前端 + 后端 + IR 规范）？
- (c) 如果不使用 IR 而是每种语言直接生成每种平台的机器码，需要多少个组件？

<details>
<summary>查看答案</summary>

- (a) **2 个前端**（TJS2 前端 + Lua 前端），**2 个后端**（x86 后端 + ARM 后端）
- (b) **5 个组件**：2 前端 + 2 后端 + 1 IR 规范。所有前端输出同一种 IR，所有后端接受同一种 IR
- (c) **4 个组件**：TJS2→x86、TJS2→ARM、Lua→x86、Lua→ARM。语言数 × 平台数 = 2 × 2 = 4。这就是没有 IR 解耦的代价——组件数是**乘法增长**而非**加法增长**

</details>

### 题目 2：编译时 vs 运行时判断

下列操作在编译时还是运行时完成？请说明理由。

- (a) `var x = 3 + 4;` 中的 `3 + 4` 求值
- (b) `var y = a + b;` 中的 `a + b` 求值
- (c) `var z = "he" + "llo";` 中的字符串拼接
- (d) `obj.method()` 中查找 `method` 属性

<details>
<summary>查看答案</summary>

- (a) **可以在编译时**完成（常量折叠）。`3` 和 `4` 都是字面量，编译器可以直接算出 `7`
- (b) **必须在运行时**完成。`a` 和 `b` 的值在编译时未知，必须等到实际执行时才能求值
- (c) **可以在编译时**完成。两个字符串都是字面量，编译器可以直接拼接为 `"hello"`
- (d) **在 TJS2 中必须在运行时**完成。TJS2 是动态类型语言，对象的属性集合可以在运行时改变，所以需要动态查找。在 C++ 等静态语言中，非虚方法调用可以在编译时确定地址

</details>

### 题目 3：阶段匹配

下列操作分别在编译器的哪个阶段完成？

- (a) 把 `var` 识别为关键字而非普通标识符
- (b) 发现 `x + ;` 缺少右操作数
- (c) 发现变量 `y` 未声明就使用
- (d) 把 `a + b` 翻译为 `ADD %r1, %r2`
- (e) 执行 `ADD` 指令并得到结果

<details>
<summary>查看答案</summary>

- (a) **词法分析**：词法器查关键字表，发现 `var` 是保留字，产出 `T_VAR` Token 而非 `T_SYMBOL`
- (b) **语法分析**：语法规则要求 `+` 后面必须跟一个表达式，但遇到了 `;`，报语法错误
- (c) **语义分析**：查符号表发现 `y` 不在任何作用域中，报"未声明的标识符"错误
- (d) **中间代码生成**：将 AST 中的 `BinOp(+, a, b)` 节点翻译为字节码指令
- (e) **目标代码执行**（虚拟机）：VM 读取 `ADD` 操作码，执行寄存器加法

</details>
