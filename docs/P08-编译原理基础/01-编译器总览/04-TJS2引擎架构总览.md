## TJS2 在 KrKr2 中的定位

### 什么是 TJS2

TJS2（**T**JS **2**nd generation）是 KiriKiri 引擎的嵌入式脚本语言，语法高度类似 JavaScript/ECMAScript，但增加了游戏引擎专用的扩展特性。它是一个**字节码编译型脚本引擎**——处于上一节光谱中"字节码 VM"的位置。

**类比理解**：如果把 KrKr2 引擎比作一个操作系统，TJS2 就是它的"Shell + 系统调用接口"。游戏逻辑不直接用 C++ 编写，而是用 TJS2 脚本调用引擎提供的 API（图形、音频、文件等）。

### 为什么需要脚本引擎

| 方案 | 优点 | 缺点 |
|------|------|------|
| 游戏逻辑全用 C++ | 运行最快 | 修改需重新编译、热更新困难、开发效率低 |
| 游戏逻辑用脚本 | 热加载、快速迭代、非程序员可编写 | 运行速度稍慢（但对文字冒险足够） |
| 混合：C++ 底层 + 脚本上层 | 兼顾性能和灵活性 | 需要维护两层代码和绑定 |

KrKr2 选择第三种方案：C++ 实现核心引擎（渲染、音频、文件IO），TJS2 实现游戏逻辑和 UI。这在游戏引擎中是主流做法（Unity 用 C#、Unreal 用 Blueprint/C++、Godot 用 GDScript）。

### TJS2 vs JavaScript：相似与差异

TJS2 的语法和 JavaScript 非常相似，但有几个关键扩展：

```javascript
// ====== JavaScript 风格的基础语法 ======
var x = 42;
var name = "TJS2";
var arr = [1, 2, 3];
function add(a, b) { return a + b; }

// ====== TJS2 独有扩展 ======
// 1. 字典字面量用 %[ ] 而非 { }
var dict = %["name" => "Alice", "age" => 20];

// 2. property 语法（getter/setter 是语言内置的）
class Player {
    var _hp;
    property hp {
        getter { return _hp; }
        setter(v) { _hp = v > 0 ? v : 0; }
    }
}

// 3. incontextof 运算符（改变 this 上下文）
var func = function() { return this.x; };
var obj = %["x" => 99];
var result = func incontextof obj;  // result == 99

// 4. invalidate / isvalid（对象生命周期管理）
var obj = new Player();
invalidate obj;       // 手动标记对象无效
// obj.hp;           // 此后访问会抛异常

// 5. 嵌入式表达式字符串
var msg = @"HP is ${player.hp}, name is &player.name;";

// 6. 八进制字面量（octet）
var data = <% 48 65 6C 6C 6F %>;  // 二进制数据
```

> **常见错误 1：把 TJS2 当 JavaScript 写**
>
> TJS2 不是 JavaScript。它没有原型链（prototype chain），没有 `undefined`（只有 `void`），没有 `===`（用 `===` 会报语法错误，应该用 `==`），`class` 和 `property` 是关键字而非语法糖。用 JavaScript 的心智模型写 TJS2 会踩很多坑。

### TJS2 与 KAG 的关系

KrKr2 引擎有两层脚本：

```
KAG 脚本 (.ks) ← 游戏剧本作者   [tag param=value] 语法
    ↓ 调用
TJS2 脚本 (.tjs) ← 游戏程序员   class/function/property/var
    ↓ 调用
C++ 引擎层 ← 引擎开发者         渲染、音频、文件IO、平台抽象
```

**KAG 是 TJS2 之上的 DSL**——KAG 解析器本身用 TJS2 编写（项目中 `cpp/core/base/KAGParser.cpp` 是 C++ 高性能版本），每个 KAG 标签最终调用 TJS2 函数。

## 编译执行全链路

下面用一段具体的 TJS2 代码追踪它从源码到执行的完整过程：

```javascript
// example.tjs
var x = 10 + 20;
System.inform(x);
```

### 阶段 1：词法分析（Lexer）

词法分析器（`tjsLex.cpp` 中的 `tTJSLexicalAnalyzer`）将源码文本切分为 token 序列：

```
源码文本: "var x = 10 + 20;"

Token 序列:
  [T_VAR]  [T_SYMBOL:"x"]  [T_EQUAL]  [T_CONSTVAL:10]
  [T_PLUS]  [T_CONSTVAL:20]  [T_SEMICOLON]
```

TJS2 词法分析器的特点：**手写实现**（非自动生成），`GetToken()` 是巨大的 switch 语句；**支持嵌入式表达式**，对 `@"text ${expr}"` 维护状态栈；**保留字用哈希表**映射关键字到 token ID；**数字支持多进制**（十/十六/八/二进制/浮点）。

```
关键调用链：
  tTJSScriptBlock::Compile()
    → tTJSLexicalAnalyzer 构造
      → 逐字符扫描: GetToken() → GetNext()
        → TJSParseNumber()     // 数字
        → TJSInternalParseString()  // 字符串
        → TJSSkipComment()    // 注释
```

### 阶段 2：语法分析（Parser）

语法分析器（由 Bison 从 `tjs.y` 生成）将 token 序列组织成 AST（抽象语法树）：

```
Token 序列 → AST:

    VarDecl
    ├── name: "x"
    └── init: BinaryExpr(+)
              ├── left:  NumLiteral(10)
              └── right: NumLiteral(20)
```

TJS2 语法分析器的特点：**Bison 生成，C++ 模式**（`%language "C++"`）；**LALR(1) 文法**（`%expect 1`，容忍一个移入/归约冲突）；**AST 用 `tTJSExprNode` 表示**，通过 `CreateExprNode()` 工厂方法创建；**标准表达式优先级**，文法层层嵌套实现——`expr → assign → cond → or → ... → factor`。

```
关键调用链：
  tTJSScriptBlock::Compile()
    → yyparse(ptr)              // Bison 生成的解析入口
      → yylex() → lx->GetNext()   // 从词法器获取 token
      → cn(T_PLUS, $1, $3)     // 创建 AST 节点（宏展开为 CreateExprNode）
```

### 阶段 3：字节码生成（Code Generator）

字节码生成器（`tjsInterCodeGen.cpp` 中的 `tTJSInterCodeGenerator`）遍历 AST，生成寄存器式字节码：

```
AST → 字节码:

  CONST  %r1, 10     // 将常量 10 加载到寄存器 r1
  CONST  %r2, 20     // 将常量 20 加载到寄存器 r2
  ADD    %r1, %r2    // r1 = r1 + r2（结果 30）
  // ... System.inform(r1) 的调用序列
```

**寄存器式 vs 栈式**——这是 TJS2 和 CPython/Lua 5.0 的最大区别：

| 设计 | 栈式（CPython/JVM） | 寄存器式（TJS2/Lua 5/Dalvik） |
|------|---------------------|-------------------------------|
| 操作数位置 | 隐式（栈顶） | 显式（寄存器编号） |
| `a + b` 指令 | `LOAD a` / `LOAD b` / `ADD`（3条） | `ADD r1, r2`（1条） |
| 指令数 | 多（栈上搬运） | 少（直接引用寄存器） |
| 实际性能 | 较慢（更多分派） | 较快（更少分派） |

> **常见错误 2："栈式一定比寄存器式慢"**
>
> 这取决于实现质量。优化良好的栈式 VM（如 HotSpot 的解释器）可以很快。但在**相同实现质量**下，寄存器式通常更快，因为它需要更少的指令分派——而分派是解释器最大的开销。TJS2 选择寄存器式是正确的工程决策。

### 阶段 4：VM 执行

VM（`tjsInterCodeExec.cpp`）用 switch-dispatch 循环执行字节码：

```cpp
// 伪代码简化版——实际代码在 tjsInterCodeExec.cpp（3109行）
while (true) {
    tjs_int32 code = *code_ptr++;  // 取指令
    switch (code & 0xFF) {         // 操作码在低 8 位
        case VM_CONST:
            TJS_GET_VM_REG(ra, code >> 8) = consts[*code_ptr++];
            break;
        case VM_ADD:
            TJS_GET_VM_REG(ra, code >> 8) +=
                TJS_GET_VM_REG(ra, *code_ptr++);
            break;
        case VM_CALL:
            // 调用函数：准备参数、切换上下文、递归执行
            break;
        case VM_RET:
            return;
        // ... 约 80 种操作码
    }
}
```

**`TJS_GET_VM_REG(base, x)` 宏**是寄存器访问的核心——它将寄存器编号映射到一个 `tTJSVariant` 数组中的元素。每个函数调用帧有自己的寄存器数组。

### 完整流水线图

```
源码 (.tjs)
  │  var x = 10 + 20; System.inform(x);
  ▼
① 词法分析 ─── tjsLex.cpp ──── GetToken() switch 扫描 ──→ Token 序列
  ▼
② 语法分析 ─── bison/tjs.y ─── LALR(1) 解析 ──→ AST
  ▼
③ 字节码生成 ─ tjsInterCodeGen.cpp ─ AST遍历 ──→ tjs_int32[] 寄存器式字节码
  ▼
④ VM 执行 ─── tjsInterCodeExec.cpp ─ switch-dispatch ──→ 程序运行结果

可选：③ 输出可通过 tjsByteCodeLoader.cpp 序列化到 .tjs.bin，跳过 ①②③ 直接加载执行
```

## TJS2 上下文类型

TJS2 定义了 8 种上下文类型（`tjsInterCodeGen.h` 第 166-175 行），比 JavaScript 的"函数/全局"丰富得多：

```cpp
enum tTJSContextType {
    ctTopLevel = 0,       // 顶层脚本          ctPropertySetter = 4, // property setter
    ctFunction = 1,       // 普通函数          ctPropertyGetter = 5, // property getter
    ctExprFunction = 2,   // 匿名函数表达式    ctClass = 6,          // class 体
    ctProperty = 3,       // property 声明体   ctSuperClassGetter = 7 // 父类获取器
};
```

每种上下文编译时生成独立的字节码段和寄存器分配方案。一个 `.tjs` 文件编译后不是一个字节码块，而是**多个字节码块的集合**——每个函数、property getter/setter、class 都有独立字节码。`tTJSInterCodeContext` 对象持有一个上下文的全部编译结果（字节码、常量表、寄存器数量等）。

```
example.tjs 编译后的结构：
tTJSScriptBlock
├── TopLevel 上下文（文件级代码）
├── Function "add" 上下文
├── Class "Player" 上下文（类初始化）
│   ├── Property "hp" getter 上下文
│   └── Property "hp" setter 上下文
└── ExprFunction（匿名函数）
```

> **常见错误 3：以为 TJS2 是"解释执行"的**
>
> 严格来说，TJS2 是"编译 + 解释"的——它先将源码**编译**为字节码（这一步是真正的编译），然后用 VM **解释**执行字节码。说"TJS2 解释执行 .tjs 文件"虽然不完全错，但更准确的说法是"TJS2 将 .tjs 编译为字节码后解释执行"。

## 示例：追踪一个函数的编译过程

用 C++ 代码模拟 TJS2 从 AST 到寄存器式字节码的简化过程：

```cpp
#include <iostream>
#include <vector>
#include <string>
#include <unordered_map>

enum NodeType { ND_VARDECL, ND_ADD, ND_NUM, ND_IDENT };
struct ASTNode {
    NodeType type;  std::string name;  int value;
    ASTNode* left = nullptr;  ASTNode* right = nullptr;
};

enum VMOp : uint8_t {
    VM_CONST = 0x01,  // VM_CONST reg, constIdx
    VM_ADD   = 0x02,  // VM_ADD   dst, src
    VM_STORE = 0x03,  // VM_STORE varIdx, reg
    VM_HALT  = 0xFF
};

class MiniCompiler {
    std::vector<uint8_t> code_;
    std::vector<int> consts_;
    std::unordered_map<std::string, int> vars_;
    int nextReg_ = 0;

    int allocReg() { return nextReg_++; }
    int addConst(int val) { consts_.push_back(val); return (int)consts_.size()-1; }

    int compileExpr(ASTNode* n) { // 递归编译，返回结果寄存器
        if (n->type == ND_NUM) {
            int r = allocReg(), ci = addConst(n->value);
            code_.insert(code_.end(), {VM_CONST, (uint8_t)r, (uint8_t)ci});
            return r;
        }
        if (n->type == ND_ADD) {
            int l = compileExpr(n->left), r = compileExpr(n->right);
            code_.insert(code_.end(), {VM_ADD, (uint8_t)l, (uint8_t)r});
            return l; // 结果存回左寄存器
        }
        return -1;
    }
public:
    void compile(ASTNode* vardecl) {
        int reg = compileExpr(vardecl->right);
        int vi = (int)vars_.size();
        vars_[vardecl->name] = vi;
        code_.insert(code_.end(), {VM_STORE, (uint8_t)vi, (uint8_t)reg});
        code_.push_back(VM_HALT);
    }
    void dump() const {
        std::cout << "=== Constants ===\n";
        for (size_t i = 0; i < consts_.size(); ++i)
            std::cout << "  [" << i << "] = " << consts_[i] << "\n";
        std::cout << "=== Bytecode (" << code_.size() << " bytes) ===\n";
        for (size_t ip = 0; ip < code_.size();) {
            switch (code_[ip]) {
            case VM_CONST: std::cout << "  CONST r" << (int)code_[ip+1]
                << ", #" << (int)code_[ip+2] << "\n"; ip += 3; break;
            case VM_ADD: std::cout << "  ADD   r" << (int)code_[ip+1]
                << ", r" << (int)code_[ip+2] << "\n"; ip += 3; break;
            case VM_STORE: std::cout << "  STORE v" << (int)code_[ip+1]
                << ", r" << (int)code_[ip+2] << "\n"; ip += 3; break;
            case VM_HALT: std::cout << "  HALT\n"; ip++; break;
            default: ip++; break;
            }
        }
    }
};

int main() {
    // 手动构建 AST：var x = 10 + 20;
    ASTNode num10{ND_NUM, "", 10}, num20{ND_NUM, "", 20};
    ASTNode add{ND_ADD, "", 0, &num10, &num20};
    ASTNode vardecl{ND_VARDECL, "x", 0, nullptr, &add};
    MiniCompiler compiler;
    compiler.compile(&vardecl);
    compiler.dump();
}
```

编译运行：

```bash
g++ -std=c++17 -o mini_compile mini_compile.cpp && ./mini_compile  # Linux/macOS
cl /std:c++17 /EHsc mini_compile.cpp && mini_compile.exe           # Windows
```

输出：

```
=== Constants ===
  [0] = 10
  [1] = 20
=== Bytecode (10 bytes) ===
  CONST r0, #0
  CONST r1, #1
  ADD   r0, r1
  STORE v0, r0
  HALT
```

`CONST r0, #0` 对应 TJS2 的 `VM_CONST`（从常量表加载到寄存器），`ADD r0, r1` 对应 `VM_ADD`。真正的 TJS2 指令更复杂——操作码和寄存器编号编码在一个 32 位整数中——但核心思路完全一致。

## 本节小结

- TJS2 是 KrKr2 的嵌入式脚本引擎，语法类似 JavaScript，但有 `property`、`incontextof`、`invalidate` 等独有特性
- TJS2 与 KAG 是两层关系：KAG 是面向剧本作者的 DSL，底层由 TJS2 实现
- TJS2 的编译执行分四阶段：词法分析（手写）→ 语法分析（Bison LALR(1)）→ 字节码生成 → VM 执行
- TJS2 采用**寄存器式字节码**，每条指令显式指定操作数寄存器，比栈式需要更少的指令分派
- 一个 `.tjs` 文件编译后产生多个字节码上下文（函数、property、class 各有独立字节码）
- 字节码可序列化为 `.tjs.bin`，实现预编译加载，跳过编译阶段

## 练习题与答案

### 题目 1：架构理解

TJS2 的编译执行链路中，哪些阶段是"编译时"完成的，哪些是"运行时"完成的？如果使用预编译的 `.tjs.bin` 文件，哪些阶段可以跳过？

<details>
<summary>查看答案</summary>

**编译时**：词法分析 → 语法分析 → 字节码生成（将源码转化为字节码，完成后源码不再需要）。
**运行时**：VM 执行（每次运行都需 VM 解释字节码）。
**使用 `.tjs.bin`**：跳过词法/语法/代码生成三阶段，由 `tjsByteCodeLoader.cpp` 直接加载字节码到内存交给 VM 执行，显著加快脚本加载速度。

</details>

### 题目 2：寄存器式 vs 栈式

用伪指令写出 `a = b + c * d` 的字节码，分别用栈式和寄存器式表示。假设变量 b/c/d 已在内存中。

<details>
<summary>查看答案</summary>

**栈式**（6 条指令）：`LOAD b` → `LOAD c` → `LOAD d` → `MUL` → `ADD` → `STORE a`

**寄存器式**（5 条指令）：`LOAD r1,b` → `LOAD r2,c` → `MUL r2,d` → `ADD r1,r2` → `STORE a,r1`

寄存器式指令分派次数更少（5 vs 6），这是其主要性能优势。差距在复杂表达式中更明显。

</details>

### 题目 3：上下文类型

以下 TJS2 代码会产生几个编译上下文？分别是什么类型？

```javascript
class Enemy {
    var _hp = 100;
    property hp {
        getter { return _hp; }
        setter(v) { _hp = v; }
    }
    function attack(target) {
        target.hp -= 10;
    }
}
var boss = new Enemy();
```

<details>
<summary>查看答案</summary>

**5 个**上下文：① **ctTopLevel**（`var boss = new Enemy();`）② **ctClass**（`class Enemy{...}` 类初始化，含 `_hp=100`）③ **ctPropertyGetter**（hp getter）④ **ctPropertySetter**（hp setter）⑤ **ctFunction**（`attack` 函数）。每个上下文有独立的字节码段、常量表和寄存器分配方案。

</details>
