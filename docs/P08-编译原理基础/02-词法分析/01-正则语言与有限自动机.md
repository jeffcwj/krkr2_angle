## 本节目标

读完本节后，你将能够：

## 为什么词法分析需要理论

上一章我们看到，TJS2 的词法分析器将源码文本切分成 token 序列。表面上这只是"字符串匹配"——遇到数字就扫描数字，遇到字母就扫描标识符。但实际上，词法分析背后有一套严谨的数学理论：**正则语言（Regular Language）** 和 **有限自动机（Finite Automaton）**。

理解这套理论的实际价值：

| 场景 | 没有理论 | 有理论 |
|------|----------|--------|
| 设计新 token | 凭直觉写 if/else | 先写正则表达式，证明无歧义 |
| 处理冲突 | `>=` vs `>` 调试半天 | 用最长匹配规则（DFA 保证） |
| 性能分析 | "应该够快吧" | O(n) 保证（DFA 单遍扫描） |
| 生成工具 | 不会用 Flex/re2c | 输入正则→自动生成 C 代码 |

## 正则表达式：描述 Token 模式

### 基本语法

正则表达式（Regular Expression, regex）用于描述字符串的模式。编译原理中使用的是**理论正则表达式**，比日常编程中的正则更简洁，只有三种基本运算：

| 运算 | 符号 | 含义 | 示例 |
|------|------|------|------|
| 连接 | AB | A 后面跟 B | `ab` 匹配 "ab" |
| 选择 | A\|B | A 或者 B | `a\|b` 匹配 "a" 或 "b" |
| 闭包 | A* | 零次或多次 A | `a*` 匹配 ""、"a"、"aa"、... |

加上两个辅助记号：`A+`（一次或多次，即 `AA*`）和 `A?`（零次或一次，即 `A|ε`）。

### 用正则表达式描述编程语言 Token

下面是编程语言中常见 token 的正则表达式定义：

```
标识符（Identifier）:
  letter   = [a-zA-Z_]
  digit    = [0-9]
  ident    = letter (letter | digit)*
  匹配: x, myVar, _count, Player2

整数字面量（Integer Literal）:
  decimal  = [1-9] digit* | 0
  hex      = 0[xX] [0-9a-fA-F]+
  binary   = 0[bB] [01]+
  integer  = decimal | hex | binary
  匹配: 42, 0xFF, 0b1010, 0

浮点数（Float Literal）:
  float    = digit+ '.' digit+ ([eE] [+-]? digit+)?
  匹配: 3.14, 1.0e10, 2.5E-3

字符串（String Literal）:
  string   = '"' ([^"\\] | '\\' .)* '"'
  匹配: "hello", "line\n", "say \"hi\""

单行注释:
  comment  = '//' [^\n]* '\n'

运算符（按长度排列以实现最长匹配）:
  op3      = '>>>' | '<<='
  op2      = '==' | '!=' | '>=' | '<=' | '&&' | '||' | '+=' | '-='
  op1      = '+' | '-' | '*' | '/' | '=' | '<' | '>' | '!'
```

**示例 1：验证正则表达式匹配**

```cpp
#include <iostream>
#include <regex>
#include <string>
#include <vector>

int main() {
    std::regex identRe("[a-zA-Z_][a-zA-Z0-9_]*");       // 标识符
    std::regex intRe("0[xX][0-9a-fA-F]+|0[bB][01]+|0|[1-9][0-9]*"); // 整数
    std::regex floatRe("[0-9]+\\.[0-9]+([eE][+-]?[0-9]+)?");         // 浮点数

    std::vector<std::string> tests = {
        "myVar", "_count", "42", "0xFF", "0b1010",
        "3.14", "1.0e10", "2.5E-3", "123abc"
    };
    for (auto& s : tests) {
        std::cout << "\"" << s << "\": ";
        if (std::regex_match(s, identRe))      std::cout << "IDENT";
        else if (std::regex_match(s, intRe))    std::cout << "INT";
        else if (std::regex_match(s, floatRe))  std::cout << "FLOAT";
        else                                     std::cout << "UNKNOWN";
        std::cout << "\n";
    }
}
```

```bash
g++ -std=c++17 -o regex_test regex_test.cpp && ./regex_test  # Linux/macOS
cl /std:c++17 /EHsc regex_test.cpp && regex_test.exe          # Windows
```

输出：`"myVar": IDENT`, `"_count": IDENT`, `"42": INT`, `"0xFF": INT`, `"0b1010": INT`, `"3.14": FLOAT`, `"1.0e10": FLOAT`, `"2.5E-3": FLOAT`, `"123abc": UNKNOWN`

## NFA：非确定有限自动机

正则表达式描述了"要匹配什么"，而有限自动机描述了"怎么匹配"。第一步是将正则表达式转换为 **NFA（Nondeterministic Finite Automaton）**。

### NFA 的定义

NFA 是一个五元组 (Q, Σ, δ, q₀, F)：

| 符号 | 含义 | 示例 |
|------|------|------|
| Q | 状态的有限集合 | {S0, S1, S2, S3} |
| Σ | 输入字母表 | {a, b, 0-9, ...} |
| δ | 转移函数 Q × (Σ∪{ε}) → P(Q) | δ(S0, 'a') = {S1, S2} |
| q₀ | 初始状态 | S0 |
| F | 接受状态集合 | {S3} |

关键区别于 DFA：NFA 允许**同一个输入到达多个状态**，也允许**不消耗输入的 ε 转移**。

### Thompson 构造法：正则 → NFA

Thompson 构造法用三条规则递归构建 NFA：

**规则 1：单字符 `a`**

```
  (S0) ──a──▶ ((S1))      圆括号表示接受状态
```

**规则 2：连接 `AB`**

```
  A 的 NFA ──ε──▶ B 的 NFA
  （A 的接受状态通过 ε 连接到 B 的起始状态）
```

**规则 3：选择 `A|B`**

```
         ┌──ε──▶ A 的 NFA ──ε──┐
  (S0) ──┤                      ├──▶ ((Sf))
         └──ε──▶ B 的 NFA ──ε──┘
```

**规则 4：闭包 `A*`**

```
              ┌──────ε──────┐
              ▼             │
  (S0) ──ε──▶ A 的 NFA ──ε─┘──ε──▶ ((Sf))
    │                                  ▲
    └──────────────ε───────────────────┘
```

### 示例 2：将 `(a|b)*c` 转换为 NFA

一步一步应用 Thompson 构造法：

```
正则: (a|b)*c

第1步: a 的 NFA    (S0)──a──▶(S1)
第2步: b 的 NFA    (S2)──b──▶(S3)
第3步: a|b 的 NFA（选择规则）
           ┌─ε─▶(S0)─a─▶(S1)─ε─┐
    (S4)───┤                     ├──▶(S5)
           └─ε─▶(S2)─b─▶(S3)─ε─┘
第4步: (a|b)* 的 NFA（闭包规则）
              ┌────────ε────────┐
              ▼                 │
    (S6)─ε─▶(S4)─[a|b NFA]─▶(S5)─ε─▶(S7)
      │                                ▲
      └────────────ε────────────────────┘
第5步: (a|b)*c 的 NFA（连接规则）
    上面的 NFA ─ε─▶(S8)─c─▶((S9))

最终有 10 个状态，接受状态 S9
```

**示例 3：NFA 模拟器**

```cpp
#include <iostream>
#include <vector>
#include <set>
#include <map>
#include <string>

// NFA 状态转移：(状态, 字符) → {目标状态集合}，字符 '\0' 表示 ε 转移
struct NFA {
    int numStates;
    int start;
    std::set<int> accept;
    std::map<std::pair<int,char>, std::set<int>> trans;

    void addTrans(int from, char ch, int to) {
        trans[{from, ch}].insert(to);
    }
    // 计算 ε 闭包
    std::set<int> epsilonClosure(const std::set<int>& states) const {
        std::set<int> closure = states;
        std::vector<int> stack(states.begin(), states.end());
        while (!stack.empty()) {
            int s = stack.back(); stack.pop_back();
            auto it = trans.find({s, '\0'}); // '\0' 代表 ε
            if (it == trans.end()) continue;
            for (int t : it->second)
                if (closure.insert(t).second) stack.push_back(t);
        }
        return closure;
    }
    // 模拟 NFA：追踪所有可能状态
    bool matches(const std::string& input) const {
        auto current = epsilonClosure({start});
        for (char ch : input) {
            std::set<int> next;
            for (int s : current) {
                auto it = trans.find({s, ch});
                if (it != trans.end())
                    next.insert(it->second.begin(), it->second.end());
            }
            current = epsilonClosure(next);
        }
        for (int s : current) if (accept.count(s)) return true;
        return false;
    }
};

int main() {
    // 手动构建 (a|b)*c 的 NFA
    NFA nfa;
    nfa.numStates = 10;  nfa.start = 6;  nfa.accept = {9};
    // a|b 内部
    nfa.addTrans(0, 'a', 1);  nfa.addTrans(2, 'b', 3);
    // 选择: S4→S0, S4→S2, S1→S5, S3→S5
    nfa.addTrans(4, '\0', 0);  nfa.addTrans(4, '\0', 2);
    nfa.addTrans(1, '\0', 5);  nfa.addTrans(3, '\0', 5);
    // 闭包: S6→S4, S5→S4, S6→S7, S5→S7
    nfa.addTrans(6, '\0', 4);  nfa.addTrans(5, '\0', 4);
    nfa.addTrans(6, '\0', 7);  nfa.addTrans(5, '\0', 7);
    // 连接: S7→S8, S8─c→S9
    nfa.addTrans(7, '\0', 8);  nfa.addTrans(8, 'c', 9);

    // 测试
    std::vector<std::string> tests = {
        "c", "ac", "bc", "abc", "aabbc", "bac",
        "a", "ab", "", "cc"
    };
    for (auto& s : tests) {
        std::cout << "\"" << s << "\": "
                  << (nfa.matches(s) ? "MATCH" : "NO") << "\n";
    }
}
```

```bash
g++ -std=c++17 -o nfa_sim nfa_sim.cpp && ./nfa_sim  # Linux/macOS
cl /std:c++17 /EHsc nfa_sim.cpp && nfa_sim.exe      # Windows
```

输出：`"c": MATCH`, `"ac": MATCH`, `"bc": MATCH`, `"abc": MATCH`, `"aabbc": MATCH`, `"bac": MATCH`, `"a": NO`, `"ab": NO`, `"": NO`, `"cc": NO`

## DFA：确定有限自动机

NFA 虽然容易构造，但执行时需要同时追踪多个状态——效率不高。**DFA（Deterministic Finite Automaton）** 保证在任何状态下，对任何输入字符，最多只有一个目标状态。

### 子集构造法：NFA → DFA

核心思想：DFA 的每个状态对应 NFA 的一个**状态子集**。

算法步骤：
1. DFA 初始状态 = NFA 初始状态的 ε 闭包
2. 对 DFA 的每个新状态（NFA 状态子集），对每个输入字符 c：
   - 计算 `move(S, c)`：S 中所有状态沿 c 转移能到达的状态集合
   - 计算 `ε-closure(move(S, c))`：这就是 DFA 的目标状态
3. 如果目标状态是新的，加入工作队列
4. DFA 接受状态 = 包含 NFA 任一接受状态的子集

**示例 4：对 `(a|b)*c` 的 NFA 做子集构造**

```
NFA 初始: ε-closure({S6}) = {S6, S4, S0, S2, S7, S8}  → 命名为 D0

D0 遇到 'a': move({S0},a)={S1} → ε-closure={S1,S5,S4,S0,S2,S7,S8} → D1
D0 遇到 'b': move({S2},b)={S3} → ε-closure={S3,S5,S4,S0,S2,S7,S8} → D2
D0 遇到 'c': move({S8},c)={S9} → ε-closure={S9}                    → D3(接受)

D1 遇到 'a': → D1（同上）
D1 遇到 'b': → D2
D1 遇到 'c': → D3(接受)

D2 遇到 'a': → D1
D2 遇到 'b': → D2
D2 遇到 'c': → D3(接受)
```

最终 DFA 只有 4 个状态（而 NFA 有 10 个）：

```
       a      b      c
D0 ──▶ D1   D0──▶D2  D0──▶((D3))
D1 ──▶ D1   D1──▶D2  D1──▶((D3))
D2 ──▶ D1   D2──▶D2  D2──▶((D3))
```

> **常见错误 1：以为 NFA 比 DFA 慢**
>
> NFA 的状态数更少但执行时需要追踪多个当前状态；DFA 状态数可能指数级增长但执行只需跟踪一个状态。在词法分析器中，DFA 的 O(n) 单遍扫描是最终选择——对每个输入字符只做一次查表。

**示例 5：DFA 表驱动词法器**

```cpp
#include <iostream>
#include <string>
#include <vector>

// (a|b)*c 的 DFA：状态 0=D0, 1=D1, 2=D2, 3=D3(接受), -1=死状态
int dfaTable[4][128];

void initDFA() {
    for (auto& row : dfaTable)
        for (auto& cell : row) cell = -1;
    // D0, D1, D2 对 a/b/c 的转移完全一致
    for (int s = 0; s <= 2; ++s) {
        dfaTable[s]['a'] = 1;  dfaTable[s]['b'] = 2;  dfaTable[s]['c'] = 3;
    }
}

bool dfaMatch(const std::string& input) {
    int state = 0;
    for (char ch : input) {
        if (ch < 0 || ch >= 128) return false;
        state = dfaTable[state][ch];
        if (state == -1) return false;
    }
    return state == 3; // 接受状态
}

int main() {
    initDFA();
    std::vector<std::string> tests = {
        "c", "ac", "bc", "abc", "aabbc", "bac",
        "a", "ab", "", "cc"
    };
    for (auto& s : tests)
        std::cout << "\"" << s << "\": "
                  << (dfaMatch(s) ? "MATCH" : "NO") << "\n";
}
```

```bash
g++ -std=c++17 -o dfa_test dfa_test.cpp && ./dfa_test  # Linux/macOS
cl /std:c++17 /EHsc dfa_test.cpp && dfa_test.exe       # Windows
```

输出与 NFA 版本完全一致，但每个字符只做一次查表——确定性执行。

## 词法分析器为什么是 DFA

所有词法分析器（无论手写还是自动生成）的核心都是 DFA：

```
正则表达式 ──Thompson──▶ NFA ──子集构造──▶ DFA ──最小化──▶ 最小 DFA
                                                         ↓
                                                    转移表/switch代码
```

**自动生成工具**（Flex/Lex/re2c）接收正则表达式，自动执行上述转换，生成 C/C++ 代码——本质是一个大型 switch 或查表的 DFA。

**手写词法器**（如 TJS2 的 `tjsLex.cpp`）本质也是 DFA——程序员直接用 switch/if-else 编码了状态转移，而非从正则表达式自动生成。

> **常见错误 2：手写词法器不需要理论**
>
> 即使不使用 Flex 等工具，理解 DFA 理论仍然关键：它帮助你正确处理最长匹配（贪婪匹配）、token 优先级、错误恢复。TJS2 的 `GetToken()` 虽然是手写的，但其逻辑严格遵循 DFA 的状态转移模式。

### TJS2 为什么选择手写而非自动生成

| 因素 | 手写（TJS2 的选择） | 自动生成（Flex/re2c） |
|------|---------------------|----------------------|
| 灵活性 | ★★★★★ 完全控制 | ★★★ 受限于工具能力 |
| 嵌入式表达式 | 容易实现状态栈 | 很难（需要 hack） |
| 错误消息 | 精确到字符级 | 通常只能报"非法字符" |
| 构建依赖 | 无额外工具 | 需要安装 Flex |
| 维护性 | ★★★ 需要理解代码 | ★★★★ 修改正则即可 |
| 性能 | ★★★★ 手动优化 | ★★★★★ 工具深度优化 |

TJS2 选择手写的主要原因是**嵌入式表达式字符串**（`@"text ${expr}"`）——这种特性需要词法器在扫描字符串时临时切换到表达式模式，用自动生成工具很难优雅实现。

> **常见错误 3：自动生成一定比手写好**
>
> 不一定。对于简单的 token 模式，自动生成确实省心。但当语言有复杂的词法特性（嵌入式表达式、上下文敏感的缩进如 Python、字符串插值）时，手写往往更灵活。V8（JavaScript）、GCC（C/C++）、Clang 都使用手写词法器。

## 本节小结

- 正则表达式描述"匹配什么"，有限自动机描述"怎么匹配"
- **Thompson 构造法**将正则表达式转为 NFA（简单但不确定）
- **子集构造法**将 NFA 转为 DFA（确定且高效，O(n) 单遍扫描）
- 所有词法分析器本质都是 DFA——手写是人肉编码状态转移，自动生成是工具代劳
- TJS2 选择手写词法器是因为嵌入式表达式等复杂特性不适合自动生成

### 题目 2：NFA 状态数

对于正则表达式 `a*b*c`，用 Thompson 构造法构建 NFA 后，一共有多少个状态？画出 NFA 状态图。

<details>
<summary>查看答案</summary>

分解：`a*` 产生 4 个状态（S0-S3），`b*` 产生 4 个状态（S4-S7），`c` 产生 2 个状态（S8-S9），连接需要 ε 转移。总计 **10 个状态**。

```
a* 部分:
      ┌──ε──┐
      ▼     │
S0─ε→S1─a→S2─ε→S3
│                ▲
└──────ε─────────┘

b* 部分（S3─ε→S4 连接）:
      ┌──ε──┐
      ▼     │
S4─ε→S5─b→S6─ε→S7
│                ▲
└──────ε─────────┘

c 部分（S7─ε→S8 连接）:
S8─c→((S9))
```

这个 NFA 转为 DFA 后只需 4 个状态，因为 `a*b*c` 的匹配模式很简单。

</details>

### 题目 3：DFA vs NFA 性能

假设一个 NFA 有 N 个状态，转换为 DFA 后最坏情况下有多少个状态？为什么实际中很少遇到最坏情况？

<details>
<summary>查看答案</summary>

最坏情况下 DFA 有 **2^N** 个状态（因为每个 DFA 状态是 NFA 状态的一个子集，N 个状态的子集最多 2^N 个）。

实际中很少遇到最坏情况，原因：
1. 大多数 NFA 状态子集在实际输入中不可达（子集构造法只构建可达状态）
2. 编程语言的 token 模式通常比较规则，不会触发指数爆炸
3. 对于常见的 `(a|b)*` 类模式，DFA 状态数通常是线性的

经典的最坏情况构造：正则 `(a|b)*a(a|b)^{n-1}`（匹配倒数第 n 个字符为 a 的字符串）确实需要 2^n 个 DFA 状态。但这种模式在编程语言的词法规则中几乎不会出现。

</details>
