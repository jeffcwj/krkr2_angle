## 整体架构

TJS2 词法器是**纯手写**的单遍扫描器——没有使用 Flex 等生成工具。整个实现集中在两个文件里：

| 文件 | 行数 | 职责 |
|------|------|------|
| `tjsLex.h` | 145 | 类声明、枚举、嵌入式表达式状态机 |
| `tjsLex.cpp` | 1895 | 全部扫描逻辑：注释、字符串、数字、运算符、标识符/关键字、预处理器 |

调用链非常清晰：

```
Bison 语法分析器
    └─ GetNext(value)          ← 唯一公开入口
         ├─ PreProcess()       ← 首次调用：换行统一 + 条件编译
         ├─ GetToken(n)        ← 核心 switch-dispatch
         │    ├─ TJSSkipComment()
         │    ├─ TJSParseString() / TJSParseNumber()
         │    ├─ TJSParseRegExp() / TJSParseOctet()
         │    └─ 标识符 → ReservedWordHash 查表
         └─ 嵌入式表达式状态机  ← @"..." 模板字符串
```

## tTJSLexicalAnalyzer 类剖析

### 构造函数（第 992-1036 行）

```cpp
tTJSLexicalAnalyzer::tTJSLexicalAnalyzer(
    tTJSScriptBlock *block, const tjs_char *script,
    bool exprmode, bool resneeded) {
    TJSInitReservedWordsHashTable();       // 懒初始化关键字哈希表
    Block = block;  ExprMode = exprmode;   ResultNeeded = resneeded;
    auto len = (tjs_int)TJS_strlen(script);
    Script = new tjs_char[len + 2];        // 复制到私有缓冲区（会原地修改！）
    TJS_strcpy(Script, script);
    if(ExprMode) { Script[len] = ';'; Script[len+1] = 0; }     // eval 追加分号
    else if(Script[0]=='#' && Script[1]=='!') { Script[0]='/'; Script[1]='/'; } // shebang→注释
    Current = Script;  First = true;       // 首次 GetNext 触发预处理
}
```

**三个设计决策：** ①复制源码到私有缓冲区——预处理会原地修改（`\r` → `TJS_SKIP_CODE`），不能污染外部数据。②表达式模式追加 `;`——`eval("1+2")` 需要完整语句才能正确归约。③`#!` 转行注释——兼容 Unix shebang。

`RetValDeque` 是最值得注意的成员——嵌入式表达式 `@"${name}"` 需要一次性推入 `(`、`string`、`(` 等多个合成 Token，队列让词法器能在一次扫描中产出**多个 Token**。

## 预处理阶段：PreProcess()

`GetNext()` 首次调用时执行预处理。核心是**换行符统一**：

```cpp
// tjsLex.cpp 第 1610-1630 行
void tTJSLexicalAnalyzer::PreProcess() {
    tjs_char *p = Script;
    while(*p) {
        if(*p == '\r' && *(p + 1) == '\n') *p = TJS_SKIP_CODE; // \r\n → SKIP + \n
        else if(*p == '\r') *p = '\n';                           // 孤立 \r → \n
        p++;
    }
}
```

`TJS_SKIP_CODE`（值为 `~((tjs_char)0)`，全 1 位）不是合法 Unicode 字符。`TJSNext()` 推进指针时自动跳过它。**为什么不直接删除 `\r`？** 删除会改变后续所有字符的偏移量，导致错误报告中的行号/列号不准确。SKIP_CODE 占位保持了偏移一致性。

### 条件编译（@if/@set/@endif）

TJS2 在词法阶段支持预处理器指令，例如 `@if(LANG == "ja") ... @endif`。处理逻辑在 `ProcessPPStatement()`（第 1123-1233 行）：`@set(expr)` 求值表达式，`@if(cond)` 条件为假时调用 `SkipUntil_endif()` 跳过代码块。`IfLevel` 变量追踪嵌套深度，确保 `@if` 内再嵌套 `@if` 时能正确匹配对应的 `@endif`。

## 核心扫描：GetToken()

`GetToken()`（第 1255-1575 行）是 320 行的巨型 switch-case，按首字符分发到不同的扫描子程序。

### 运算符扫描：最长匹配宏

TJS2 用 4 个宏实现最长匹配：`TJS_MATCH_S(word, code)` 匹配符号串、`TJS_MATCH_W(word, code)` 匹配关键字（检查边界）、带 `_V` 后缀的版本附带值。以 `>` 开头的运算符为例：

```cpp
case TJS_W('>'):                                     // tjsLex.cpp 第 1283 行
    TJS_MATCH_S(">>>=", T_RBITSHIFTEQUAL);           // 4字符最先匹配
    TJS_MATCH_S(">>>",  T_RBITSHIFT);                // 3字符
    TJS_MATCH_S(">>=",  T_RARITHSHIFTEQUAL);         // 3字符
    TJS_MATCH_S(">>",   T_RARITHSHIFT);  TJS_MATCH_S(">=", T_GTOREQUAL); // 2字符
    TJS_1CHAR(T_GT);                                 // 1字符兜底
```

**从长到短排列**，第一个匹配成功就返回。`TJSStringMatch()` 逐字符比较，失败时恢复指针（保存 `save` 回退点）。`isword` 参数为 true 时额外检查后续字符不是字母/下划线——防止 `instanceof` 前缀匹配到 `in`。

### 注释处理：嵌套块注释

```cpp
// tjsLex.cpp 第 162-212 行
static tTJSSkipCommentResult TJSSkipComment(const tjs_char **ptr) {
    if((*ptr)[0] != '/') return scrNotComment;

    if((*ptr)[1] == '/') {
        // 行注释：跳到换行
        while(*(*ptr) != '\n') if(!TJSNext(&(*ptr))) break;
        // ...
        return scrContinue;
    } else if((*ptr)[1] == '*') {
        // 块注释：支持嵌套！
        (*ptr) += 2;
        tjs_int level = 0;
        for(;;) {
            if((*ptr)[0] == '/' && (*ptr)[1] == '*') level++;  // 嵌套加深
            if((*ptr)[0] == '*' && (*ptr)[1] == '/') {
                if(level == 0) { (*ptr) += 2; break; }        // 匹配结束
                level--;                                        // 嵌套回退
            }
            if(!TJSNext(&(*ptr))) TJS_eTJSError(TJSUnclosedComment);
        }
        return scrContinue;
    }
    return scrNotComment;
}
```

嵌套块注释在 C/C++ 中不被支持（`/* /* */ */` 是非法的），但 TJS2 允许。这在注释大段包含注释的代码时非常实用。

> **常见错误：** 注释源码中提到 "we cannot avoid comment processing when the nested comment is in string literals"——字符串中的 `/*` 也会被误识别为注释开始。这是已知缺陷，实际项目中极少触发。

### 数字扫描：五进制全支持

数字解析由 `TJSParseNumber()`（第 700-727 行）+ `TJSParseNumber2()`（第 611-698 行）完成：

```
入口：TJSParseNumber
  ├─ 处理正负号
  └─ TJSParseNumber2
       ├─ 匹配 "true"/"false"/"NaN"/"Infinity"（是的，这些在数字解析器里！）
       ├─ 0x... → TJSParseNonDecimalNumber(TJSHexNum, 4)
       ├─ 0b... → TJSParseNonDecimalNumber(TJSBinNum, 1)
       ├─ 0...  → TJSParseNonDecimalNumber(TJSOctNum, 3)  // 八进制
       └─ 其他  → 十进制整数或浮点
```

每种进制共用 `TJSParseNonDecimalNumber()`，仅传入不同的数字判定函数和位移量：

```cpp
// TJSHexNum: 'a'-'f'/'A'-'F'/'0'-'9' → 0-15, 其他 → -1
// TJSOctNum: '0'-'7' → 0-7, 其他 → -1
// TJSBinNum: '0' → 0, '1' → 1, 其他 → -1
// TJSDecNum: '0'-'9' → 0-9, 其他 → -1
```

**设计亮点：** 用函数指针参数化数字解析，一套逻辑处理四种进制。这正是上一节中我们讨论的"通用策略"在工业代码中的体现。

### 字符串扫描：三种模式

TJS2 的字符串处理分三层：

| 函数 | 用途 | 调用点 |
|------|------|--------|
| `TJSParseString()` | 普通字符串 `'...'` / `"..."` | GetToken case `'`/`"` |
| `TJSInternalParseString()` | 内部实现，支持嵌入式表达式断点 | TJSParseString + 嵌入式状态机 |
| `TJSParseRegExp()` | 正则表达式 `/pattern/flags` | GetToken 被标记为正则模式时 |

`TJSInternalParseString()` 的返回值是一个**四值枚举**：

```cpp
enum tTJSInternalParseStringResult {
    psrNone,        // 错误（未找到结束符）
    psrDelimiter,   // 正常结束（遇到匹配的引号）
    psrAmpersand,   // 遇到 & （@"...&expr;..." 中断）
    psrDollar       // 遇到 ${ （@"...${expr}..." 中断）
};
```

当返回 `psrAmpersand` 或 `psrDollar` 时，词法器暂停字符串解析，切换到**嵌入式表达式模式**——这就是下一节的重点。

### 转义字符

`TJSUnescapeBackSlash()`（第 135-156 行）处理标准 C 风格转义：`\a`(0x07)、`\b`(0x08)、`\f`(0x0c)、`\n`(0x0a)、`\r`(0x0d)、`\t`(0x09)、`\v`(0x0b)，其他字符原样返回（`\\` → `\`）。此外支持 `\xHH`（十六进制）和 `\0OOO`（八进制）Unicode 码点。

## 嵌入式表达式：@"..." 状态机

这是 TJS2 词法器最复杂的部分。模板字符串 `@"name is ${name}, age is &age;"` 被词法器拆解为一系列 Token：

```
@"name is ${name}, age is &age;"
↓ 词法器产出的 Token 序列：
( "name is " + string( name ) + ", age is " + string( age ) )
```

### 状态机设计

```cpp
// tjsLex.h 第 84-96 行
enum tEmbeddableExpressionState {
    evsStart,                  // 初始：推入 '('
    evsNextIsStringLiteral,    // 下一段是字符串字面量
    evsNextIsExpression        // 下一段是表达式
};

struct tEmbeddableExpressionData {
    tEmbeddableExpressionState State;
    tjs_int WaitingNestLevel;  // 表达式结束时的目标嵌套层级
    tjs_int WaitingToken;      // 等待的结束 Token（; 或 }）
    tjs_char Delimiter;        // 字符串定界符（' 或 "）
    bool NeedPlus;             // 是否需要 + 连接符
};
```

状态转换流程（以 `@"Hello ${name}!"` 为例）：

```
1. 遇到 @" → 压栈 {evsStart, NestLevel, '"'}
2. evsStart → 推入 '(' → 切换到 evsNextIsStringLiteral
3. evsNextIsStringLiteral:
   - 解析 "Hello " → 推入 CONSTVAL("Hello ")
   - 遇到 ${ → 推入 + string ( → 切换到 evsNextIsExpression
4. evsNextIsExpression:
   - 正常扫描 name → 返回 T_SYMBOL
   - 遇到 } 且 NestLevel 匹配 → 推入 ) → 切回 evsNextIsStringLiteral
5. evsNextIsStringLiteral:
   - 解析 "!" → 推入 + CONSTVAL("!")
   - 遇到 " (结束) → 推入 ) → 弹栈
```

最终产出的 Token 等价于表达式 `("Hello " + string(name) + "!")`。

### 两种嵌入语法

| 语法 | 终止条件 | 适用场景 |
|------|----------|----------|
| `&variable;` | 遇到 `;` | 简单变量引用 |
| `${expression}` | 遇到匹配的 `}` | 任意表达式 |

`${}` 语法通过 `NestLevel` 计数来正确处理嵌套的花括号：

```cpp
// tjsLex.cpp 第 1809-1811 行
if(res == psrDollar)
    data.WaitingToken = T_RBRACE,
    NestLevel++;  // 需要等到 NestLevel 回落才算结束
```

## 标识符与关键字查找

标识符的扫描在 GetToken 的 switch 末尾（第 1497-1574 行）：

```cpp
// 1. 验证首字符：字母、下划线、或 >0x0100 的 Unicode
if(!TJS_iswalpha(*Current) && *Current != '_')
    TJS_eTJSError(TJSInvalidChar);

// 2. 贪婪扫描：字母/数字/下划线/高位Unicode/SKIP_CODE
const tjs_char *ptr = Current;
tjs_int nch = 0;
while(TJS_iswdigit(*ptr) || TJS_iswalpha(*ptr) ||
      *ptr == '_' || *ptr > 0x0100 || *ptr == TJS_SKIP_CODE)
    ptr++, nch++;

// 3. 清除 SKIP_CODE（预处理残留）
ttstr str(Current, nch);
tjs_char *s, *d;
s = d = str.Independ();
while(*s) {
    if(*s == TJS_SKIP_CODE) { s++; continue; }
    *d = *s; d++, s++;
}
```

**Unicode 支持：** `TJS_iswalpha()` 对 `ch > 0xFF` 一律返回 `true`——这意味着中文、日文等字符都可以作为标识符。TJS2 脚本完全可以写 `var 名前 = "太郎";`。

### 关键字哈希表

扫描到标识符后，查询关键字哈希表判断是标识符还是保留字：

```cpp
// tjsLex.cpp 第 1534-1545 行
tjs_int retnum;
if(BareWord)
    retnum = -1;                    // . 后面的标识符不查表
else
    retnum = TJSReservedWordHash->GetValueInteger(
        str.c_str(), str.GetHint());

if(retnum == -1) {
    n = PutValue(str);
    return T_SYMBOL;                // 普通标识符
}
```

哈希表在构造函数中懒初始化，注册了约 50 个关键字（`break`、`continue`、`const`、`class`、`delete`、`extends`、`function`、`for`、`if`、`else`、`return`、`switch`、`this`、`try`、`typeof`、`var`、`while`、`void`、`null`、`true`、`false`、`NaN`、`Infinity`……）。其中 `incontextof`、`invalidate`、`isvalid`、`property`、`getter`/`setter`、`octet`、`global` 是 TJS2 独有的关键字，在 JavaScript 中不存在。

**`BareWord` 机制：** 当语法分析器遇到 `.` 运算符时，会调用 `SetNextIsBareWord()` 通知词法器——下一个标识符即使是保留字也当普通标识符处理。这样 `obj.class`、`obj.return` 都是合法的属性访问。

## 正则表达式歧义消解

`/` 既是除法运算符又是正则的定界符。TJS2 的解决方案是**由语法分析器回调通知**：

```cpp
// 语法分析器在适当时机调用：
void SetStartOfRegExp() { RegularExpression = true; }

// GetToken 检查标志：
if(RegularExpression) {
    RegularExpression = false;
    Current = Script + PrevPos;   // 回退到 '/' 位置
    TJSNext(&Current);
    TJSParseRegExp(pat, &Current);
    return T_REGEXP;
}
```

这要求语法分析器能在语法层面判断 `/` 的语义——例如 `if(/regex/.test(x))` vs `a / b`。

## DicFunc Quick-Hack

`GetNext()` 中有一段被标记为 "dicfunc quick-hack" 的代码（第 1702-1747 行）。它将字典/数组字面量包裹成立即执行函数：

```javascript
// 原始代码              → 词法器改写后等价代码
%[ key: value ]         → function { return %[ key: value ]; }()
[ 1, 2, 3 ]            → function { return [ 1, 2, 3 ]; }()
```

**为什么需要这个 hack？** 原始解析器在处理大型字典/数组字面量时会导致内存溢出。将字面量包裹成函数调用，限制了解析器的递归深度。

这个 hack 通过全局变量 `tjsEnableDicFuncQuickHack` 控制，默认由源码中一个巧妙的字符串标记决定：

```cpp
tjs_nchar tjsEnableDicFuncQuickHack_mark[] =
    " - 0 <- Put '1' to enable dicfunc quick-hack - ";
bool tjsEnableDicFuncQuickHack =
    tjsEnableDicFuncQuickHack_mark[3] != '0';
```

开发者可以在二进制文件中直接修改这个字节来启用/禁用——无需重新编译。这是游戏引擎中常见的"运行时补丁"技巧。

## 本节小结

- TJS2 词法器是 1895 行的**纯手写单遍扫描器**，核心是一个 320 行的 switch-case (`GetToken`)
- 用 `TJS_SKIP_CODE` 占位替代删除字符，**保持偏移一致性**以支持精确的错误定位
- **嵌套块注释**通过 level 计数实现，是对 C/C++ 注释规则的改进
- **嵌入式表达式**用三状态状态机 + Token 缓冲队列实现，将 `@"..${expr}.."` 转换为 `(".."+string(expr)+"..")`
- **关键字查找**使用 TJS2 自身的对象系统（`tTJSCustomObject`）作为哈希表——"吃自己的狗粮"
- 正则表达式的 `/` 歧义由**语法分析器回调**消解，词法器本身不做猜测
- DicFunc Quick-Hack 展示了游戏引擎中"可热修改"的工程技巧

预处理后缓冲区内容（用 S 代表 SKIP_CODE）：`v a r S \n a   =   1 ;`

`\r` 被替换为 SKIP_CODE，`\n` 保留。`TJSNext()` 遍历时自动跳过 SKIP_CODE，所以看到的序列是：`v a r \n a   =   1 ;`——与 Unix 换行完全一致。

关键代码（`tjsLex.cpp` 第 1624-1625 行）：
```cpp
if(*p == '\r' && *(p + 1) == '\n')
    *p = TJS_SKIP_CODE;  // \r 变 SKIP，\n 不动
```

偏移量保持不变：`a` 仍然在偏移 5（而非偏移 4），错误信息中的列号与源文件一致。

</details>

### 题目 2：嵌入式表达式嵌套

下面的 TJS2 代码合法吗？如果合法，词法器会产出什么 Token 序列？

```javascript
var s = @"result is ${a > b ? @'yes ${x}' : 'no'}!";
```

<details>
<summary>查看答案</summary>

合法。`EmbeddableExpressionDataStack` 是一个 `vector`（栈），支持嵌套。

外层 `@"..."` 压栈一个状态，内层 `@'...'` 再压栈一个。Token 序列（简化）：

```
T_VAR  T_SYMBOL("s")  T_EQUAL
(                              ← 外层 @" 开始
  "result is "
  + string(
    a > b ? (                  ← 内层 @' 开始
      "yes " + string( x ) 
    )                          ← 内层 @' 结束
    : "no"
  )
  + "!"
)                              ← 外层 @" 结束
T_SEMICOLON
```

内层 `@'...'` 遇到 `'` 结束时弹出内层栈帧，外层继续解析直到遇到 `"`。`NestLevel` 的追踪确保 `${...}` 中的 `{` `}` 不会被误判为表达式结束。

</details>

### 题目 3：BareWord 机制

解释为什么下面两行代码的行为不同，并指出 `BareWord` 标志在哪里被设置：

```javascript
var class = 1;       // 编译错误
obj.class = 1;       // 正常运行
```

<details>
<summary>查看答案</summary>

第一行：`class` 被关键字哈希表匹配为 `T_CLASS`，语法分析器在 `var` 后期望 `T_SYMBOL` 而非 `T_CLASS`，报错。

第二行：语法分析器在处理 `.` 运算符时调用 `SetNextIsBareWord()`（`tjsLex.cpp` 第 1886-1891 行），将 `BareWord` 置为 `true`。词法器在标识符查找时检查该标志：

```cpp
// tjsLex.cpp 第 1534-1536 行
if(BareWord)
    retnum = -1;   // 跳过关键字哈希表查找
```

`class` 被当作普通标识符 `T_SYMBOL("class")` 返回，属性访问正常工作。`BareWord` 在使用后立即重置为 `false`（第 1540 行），只影响紧接的下一个标识符。

</details>
