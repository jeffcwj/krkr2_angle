## 为什么需要符号表

语法分析器只关心代码的**结构**——括号配对、运算符优先级。但代码的**含义**需要更多信息：

```javascript
var x = 10;
function foo() {
    var y = x + 1;    // x 指的是哪个 x？是局部变量还是全局变量？
    var x = 20;        // 这个 x 和外面的 x 是同一个吗？
}
```

要回答这些问题，编译器需要一张**符号表（Symbol Table）**——记录每个名字（变量、函数、类）的属性：

| 属性 | 说明 | 示例 |
|------|------|------|
| 名称（Name） | 标识符字符串 | `"x"`、`"foo"` |
| 种类（Kind） | 变量、函数、参数、类 | `Variable`、`Function` |
| 类型（Type） | 值的类型 | `int`、`string`、`any` |
| 作用域层级（Level） | 在第几层嵌套中声明 | 0（全局）、1（函数体）、2（内部块） |
| 存储位置（Location） | 运行时如何访问 | 寄存器编号、栈偏移、全局表索引 |

> **C++ 程序员类比：** 符号表就像 C++ 编译器内部维护的名字查找表。当你写 `std::cout` 时，编译器需要查表确认 `cout` 是 `std` 命名空间中的一个 `ostream` 对象。

## 符号表的数据结构

两种常见实现方案：

```cpp
// === 方案 1：线性列表 ===
struct Symbol { std::string name; int kind; int location; };

class LinearSymbolTable {
    std::vector<Symbol> symbols;
public:
    void add(const std::string& name, int kind, int loc) {
        symbols.push_back({name, kind, loc});
    }
    int find(const std::string& name) {  // O(n) 线性扫描
        for (int i = (int)symbols.size() - 1; i >= 0; i--)
            if (symbols[i].name == name) return symbols[i].location;
        return -1;
    }
};

// === 方案 2：哈希表 ===
class HashSymbolTable {
    std::unordered_map<std::string, Symbol> table;
public:
    void add(const std::string& name, int kind, int loc) {
        table[name] = {name, kind, loc};
    }
    int find(const std::string& name) {  // O(1) 平均
        auto it = table.find(name);
        return (it != table.end()) ? it->second.location : -1;
    }
};
```

| 方案 | 查找 | 优点 | 缺点 |
|------|------|------|------|
| 线性列表 | O(n) | 实现简单，缓存友好 | 变量多时慢 |
| 哈希表 | O(1) | 大规模快速查找 | 内存开销大 |

**实际选择：** 大多数生产级编译器（GCC、Clang、V8）使用哈希表。TJS2 则使用**线性列表**——因为局部变量数量通常很少（几十个），线性查找的常数因子比哈希更低。

## 作用域与嵌套

### 词法作用域 vs 动态作用域

```javascript
var x = "global";
function outer() {
    var x = "outer";
    function inner() {
        return x;   // 词法作用域："outer"  动态作用域：取决于调用者
    }
    return inner;
}
var f = outer();
f();  // 词法作用域返回 "outer"
```

| 特性 | 词法作用域（Lexical） | 动态作用域（Dynamic） |
|------|----------------------|----------------------|
| 变量查找依据 | 代码的**文本位置** | 函数的**调用链** |
| 确定时机 | **编译期**可确定 | 只有**运行时**才知道 |
| 代表语言 | C/C++, Java, JS, TJS2 | Bash, Emacs Lisp |
| 可预测性 | 高（看代码就知道） | 低（取决于谁调用） |

TJS2 使用**词法作用域**——变量的查找在编译期就可以确定，不依赖运行时调用链。

### 作用域链

嵌套的花括号创建嵌套的作用域。查找变量时，从内向外逐层搜索（作用域链）：

```javascript
var a = 1;                  // 全局作用域
function foo() {
    var b = 2;              // foo 的作用域
    {
        var c = 3;          // 块作用域
        // 查找 a：块→foo→全局 ✓ | 查找 d：块→foo→全局→未找到 ✗
    }
    // c 在这里已不可见
}
```

```
┌──────────┐    查找方向
│ 块: c=3  │ ──→ foo: b=2 ──→ 全局: a=1, foo
└──────────┘    （从内到外）
```

### 遮蔽（Shadowing）

内层作用域可以声明与外层同名的变量，**遮蔽**外层的定义：

```javascript
var x = 10;
function foo() {
    var x = 20;     // 遮蔽全局的 x——创建全新的局部变量，与全局 x 无关
    return x;       // 返回 20，不是 10。函数返回后全局 x 仍为 10
}
```

## 实现嵌套作用域的符号表

核心思想：**栈式管理**——进入新作用域时压入一层，离开时弹出。

```cpp
#include <iostream>
#include <string>
#include <vector>
#include <unordered_map>

// 一层作用域
struct Scope {
    std::unordered_map<std::string, int> symbols;  // name → 局部变量索引
};

class ScopeManager {
    std::vector<Scope> stack;
    int nextIndex = 0;
    std::vector<int> savedIndex;   // 每层进入时的索引快照
public:
    void enterScope()  { savedIndex.push_back(nextIndex); stack.push_back(Scope{}); }
    void exitScope()   { nextIndex = savedIndex.back(); savedIndex.pop_back(); stack.pop_back(); }

    int declare(const std::string& name) {
        int idx = nextIndex++;
        stack.back().symbols[name] = idx;
        return idx;
    }
    int lookup(const std::string& name) {  // 从内向外查找
        for (int i = (int)stack.size() - 1; i >= 0; i--) {
            auto it = stack[i].symbols.find(name);
            if (it != stack[i].symbols.end()) return it->second;
        }
        return -1;
    }
    int depth() const { return (int)stack.size(); }
};

int main() {
    ScopeManager mgr;
    mgr.enterScope();                                  // 全局作用域
    int a = mgr.declare("a");                          // a → 0
    mgr.declare("foo");                                // foo → 1
    std::cout << "a 的索引: " << a << "\n";            // 输出：0

    mgr.enterScope();                                  // 函数作用域
    mgr.declare("b");                                  // b → 2
    mgr.declare("x");                                  // x → 3
    std::cout << "查找 a: " << mgr.lookup("a") << "\n";  // 0（全局）
    std::cout << "查找 x: " << mgr.lookup("x") << "\n";  // 3（局部）

    mgr.enterScope();                                  // 块作用域
    mgr.declare("x");                                  // x → 4（遮蔽）
    std::cout << "查找 x: " << mgr.lookup("x") << "\n";  // 4
    mgr.exitScope();
    std::cout << "查找 x: " << mgr.lookup("x") << "\n";  // 3（恢复）
    mgr.exitScope();
    mgr.exitScope();
    return 0;
}
```

编译运行：

```bash
g++ -std=c++17 -o scope scope.cpp && ./scope
# 输出：
# a 的索引: 0
# 查找 a: 0
# 查找 x: 3
# 查找 x: 4
# 查找 x: 3
```

> **设计要点：** `savedIndex` 记录每层进入时的 `nextIndex` 快照。弹出作用域时恢复它，这样局部变量的索引号就能被下一个并列作用域**复用**——和 TJS2 的 `MaxCount` 机制殊途同归。

## 对照项目源码——TJS2 的作用域管理

TJS2 的作用域管理集中在两个文件中：

- `cpp/core/tjs2/tjsNamespace.h`（第 20-152 行）——类声明
- `cpp/core/tjs2/tjsNamespace.cpp`（第 20-233 行）——实现

### 核心数据结构

TJS2 使用三级结构：

```
tTJSLocalNamespace                    ← 整个编译单元的命名空间
  └── vector<tTJSLocalSymbolList*> Levels    ← 作用域栈（每层一个列表）
        └── vector<tTJSLocalSymbol*> List    ← 一个作用域内的符号列表
              └── tTJSLocalSymbol { tjs_char* Name }  ← 单个符号
```

对应源码（`tjsNamespace.h`）：

```cpp
// 单个符号——只存名字，位置由 vector 下标隐含
struct tTJSLocalSymbol {            // 第 26-28 行
    tjs_char *Name;
};

// 一层作用域——线性列表
class tTJSLocalSymbolList {         // 第 31-65 行
    std::vector<tTJSLocalSymbol*> List;   // 符号数组
    tjs_int LocalCountStart;              // 本层起始索引（相对于整个函数的局部变量区）
public:
    void Add(const tjs_char* name);       // 添加符号到本层
    tjs_int Find(const tjs_char* name);   // 在本层查找，返回 LocalCountStart + 偏移
    void Remove(const tjs_char* name);    // "删除"——实际置 nullptr（墓碑）
};

// 命名空间——作用域栈
class tTJSLocalNamespace {          // 第 70-104 行
    std::vector<tTJSLocalSymbolList*> Levels;   // 栈：Levels[0] 是最外层
    tjs_int MaxCount;               // 局部变量峰值计数
public:
    void Push();                    // 进入新作用域
    void Pop();                     // 离开当前作用域
    tjs_int Find(const tjs_char* name);   // 从内向外搜索
    void Add(const tjs_char* name);       // 在最内层添加符号
    tjs_int GetMaxCount();                // 返回峰值——VM 用它分配栈帧大小
};
```

### 关键操作详解

Push/Pop/Find 三个核心方法的简化源码：

```cpp
// === Push：进入作用域 === (tjsNamespace.cpp 第 132-145 行)
void tTJSLocalNamespace::Push() {
    tTJSLocalSymbolList* list = new tTJSLocalSymbolList();
    if (Levels.size() > 0) {
        // 新层的起始索引 = 上一层起始 + 上一层符号数
        list->SetLocalCountStart(
            Levels.back()->GetLocalCountStart() + Levels.back()->GetCount());
    } else {
        list->SetLocalCountStart(0);
    }
    Levels.push_back(list);
}

// === Pop：离开作用域 === (第 147-154 行)
void tTJSLocalNamespace::Pop() {
    Levels.back()->Commit(MaxCount);  // 比较 Start+Count 与 MaxCount，取较大值
    delete Levels.back();             // 销毁该层所有符号
    Levels.pop_back();
}
// MaxCount 记录整个函数中同时存活的局部变量峰值——VM 据此分配栈帧大小

// === Find：作用域链查找 === (第 156-171 行)
tjs_int tTJSLocalNamespace::Find(const tjs_char* name) {
    for (int i = (int)Levels.size() - 1; i >= 0; i--) {  // 从内向外
        tjs_int idx = Levels[i]->Find(name);               // 层内线性扫描 O(n)
        if (idx != -1) return idx;
    }
    return -1;
}

// === Remove：墓碑删除 === (第 89-102 行，在 tTJSLocalSymbolList 中)
void tTJSLocalSymbolList::Remove(const tjs_char* name) {
    for (auto& sym : List) {
        if (sym && !TJS_strcmp(sym->Name, name)) {
            delete[] sym->Name; delete sym;
            sym = nullptr;   // 不缩容——保持索引稳定，后续 Add 可复用空槽
            return;
        }
    }
}
```

> **墓碑删除的原因：** 已编译的字节码通过 `LocalCountStart + 偏移` 引用变量。如果 erase 中间元素，后面所有变量的索引都会偏移，字节码引用全部失效。

### RAII 辅助与编译器集成

TJS2 提供 RAII 辅助类（`tjsNamespace.h` 第 109-119 行），构造时 `Push()`、析构时 `Pop()`，确保即使异常也能正确弹出作用域：

```cpp
class tTJSLocalNamespaceAutoPushPop {
    tTJSLocalNamespace& Namespace;
public:
    tTJSLocalNamespaceAutoPushPop(tTJSLocalNamespace& ns) : Namespace(ns)
        { Namespace.Push(); }
    ~tTJSLocalNamespaceAutoPushPop()
        { Namespace.Pop(); }
};
```

编译器（`tjsInterCodeGen.h` 第 404 行）持有 `tTJSLocalNamespace Namespace` 成员。解析器遇到 `{ ... }` 块时，语义动作调用 `EnterBlock()` → 编译代码 → `ExitBlock()`，通过作用域栈自动管理变量可见性。

### 与我们的实现对比

| 特性 | 我们的 ScopeManager | TJS2 tTJSLocalNamespace |
|------|---------------------|------------------------|
| 作用域栈 | `vector<Scope>` | `vector<tTJSLocalSymbolList*>` |
| 单层数据结构 | `unordered_map` | `vector<tTJSLocalSymbol*>` |
| 查找复杂度 | O(层数) × O(1) | O(层数) × O(n) |
| 索引复用 | exitScope 恢复 nextIndex | Pop 时 delete，Commit 更新 MaxCount |
| 删除支持 | 不支持 | 墓碑删除（置 nullptr） |
| RAII 辅助 | 无 | `tTJSLocalNamespaceAutoPushPop` |

TJS2 选择线性列表而非哈希表，是因为脚本语言函数的局部变量极少（通常 < 20 个），线性查找的缓存命中率反而比哈希更高。

## 常见错误与解决方案

### 错误 1：弹出作用域后仍然访问已销毁的符号

```cpp
ScopeManager mgr;
mgr.enterScope();
int idx = mgr.declare("temp");
mgr.exitScope();
int result = mgr.lookup("temp");  // 返回 -1！用它做数组索引会越界
```

**解决：** `lookup` 返回 -1 时必须报告"未定义变量"错误。TJS2 在 `Find` 返回 -1 后会尝试全局对象查找，若仍失败则报编译错误。

### 错误 2：忘记区分"声明"和"赋值"导致意外遮蔽

```javascript
var x = 10;
function foo() {
    x = 20;        // 想修改外层 x
    var x = 30;    // 但 var 声明会被"提升"到函数开头
}
```

TJS2 和 JavaScript 一样有**变量提升**（hoisting）——编译器先扫描所有 `var` 声明注册到符号表，再生成代码。因此 `x = 20` 修改的是尚未初始化的局部 `x`，不是全局 `x`。

**解决：** 想修改外层变量就不要在同一函数中 `var` 同名变量。现代 JS 的 `let`/`const` 用"暂时性死区"（TDZ）捕获这类错误。

### 错误 3：作用域栈 Push/Pop 不配对

```cpp
// ❌ 错误——if 分支只 Push 不 Pop
void compileIfStatement(ASTNode* node) {
    namespace.Push();
    compileBlock(node->thenBranch);
    // 忘了 Pop！直接去编译 else 了
}
```

**症状：** 作用域栈持续增长，`MaxCount` 计算错误导致栈帧分配不足，VM 访问越界。

**解决：** 使用 RAII 辅助类（如 `tTJSLocalNamespaceAutoPushPop`），C++ 析构函数保证 Pop 一定执行。

## 本节小结

- **符号表**是编译器记忆名字含义的核心数据结构，每个条目包含名称、种类、类型、层级、存储位置
- **线性列表** O(n) 查找但实现简单，**哈希表** O(1) 查找但内存开销大；TJS2 选择线性列表
- **词法作用域**在编译期根据代码文本位置确定变量归属；TJS2 使用词法作用域
- **作用域链**从内向外逐层查找，内层同名变量**遮蔽**外层
- **栈式管理**：进入作用域 Push、离开 Pop，RAII 保证配对
- TJS2 的 `tTJSLocalNamespace` 用 `vector<tTJSLocalSymbolList*>` 实现作用域栈，`MaxCount` 记录局部变量峰值供 VM 分配栈帧
- **墓碑删除**：Remove 置 nullptr 而非缩容，因为已编译的字节码依赖索引位置

```javascript
var x = 1;
function test() {
    var y = 2;
    {
        var x = 3;
        var z = x + y;  // ← 编译到此行
    }
}
```

<details>
<summary>查看答案</summary>

作用域栈（从底到顶）：

```
Level 0（全局）:  x → 索引 0,  test → 索引 1
Level 1（test 函数）:  y → 索引 0   (LocalCountStart = 0)
Level 2（块）:  x → 索引 2,  z → 索引 3   (LocalCountStart = 2)
```

查找 `x`：从 Level 2 开始 → 找到 `x` 索引 2（遮蔽了全局的 `x`）。
查找 `y`：Level 2 没有 → Level 1 找到 `y` 索引 0。

注意：函数内的局部变量从 0 重新编号，全局变量走不同的查找路径（全局对象属性），所以不会混淆。

</details>

### 题目 2：修复 Bug——实现 MaxCount 机制

下面的 ScopeManager 缺少 MaxCount 追踪。请在 `exitScope` 中补全一行代码，使 `getMaxCount()` 返回整个生命周期中同时存活的局部变量最大数量：

```cpp
class ScopeManager {
    std::vector<Scope> stack;
    int nextIndex = 0;
    std::vector<int> savedIndex;
    int maxCount = 0;              // ← 新增
public:
    void enterScope() { savedIndex.push_back(nextIndex); stack.push_back(Scope{}); }
    void exitScope() {
        // TODO: 在恢复 nextIndex 之前更新 maxCount
        nextIndex = savedIndex.back();
        savedIndex.pop_back();
        stack.pop_back();
    }
    int declare(const std::string& name) {
        int idx = nextIndex++;
        stack.back().symbols[name] = idx;
        return idx;
    }
    int getMaxCount() const { return maxCount; }
};
```

<details>
<summary>查看答案</summary>

在恢复 `nextIndex` **之前**加一行峰值检查：

```cpp
void exitScope() {
    if (nextIndex > maxCount) maxCount = nextIndex;  // ← 关键一行
    nextIndex = savedIndex.back();
    savedIndex.pop_back();
    stack.pop_back();
}
```

这与 TJS2 的 `Commit(MaxCount)` 完全对应——Pop 前检查水位线。VM 最终根据 `maxCount` 分配栈帧空间。例如：声明 a(0), 进块声明 b(1) c(2), 退块, 进新块声明 d(1), 退块——`maxCount = 3`（峰值在第一个块退出时记录）。

</details>

### 题目 3：对比分析

阅读 TJS2 源码 `tjsNamespace.cpp` 中 `tTJSLocalSymbolList::Remove` 的实现，回答：

1. 为什么 Remove 将指针置为 `nullptr` 而不是从 vector 中 `erase`？
2. 如果改用 `erase`，会导致什么具体问题？
3. TJS2 中哪种语句会触发 Remove？（提示：考虑 `with` 语句的别名）

<details>
<summary>查看答案</summary>

1. **为什么用墓碑：** 已编译的字节码通过 `LocalCountStart + 偏移` 引用变量。erase 会使后续元素索引偏移，字节码引用全部失效。

2. **erase 的后果：** List = `[a, b, c]`（索引 0,1,2），字节码有 `LOAD_LOCAL 2` 引用 c。erase b 后 List = `[a, c]`，`LOAD_LOCAL 2` 越界；`LOAD_LOCAL 1` 本该是 b 却变成了 c。

3. **触发场景：** TJS2 中 Remove 调用极少，大多数变量靠 Pop 整层销毁。可能的触发场景是编译器内部优化释放不再使用的临时变量符号名。

</details>
