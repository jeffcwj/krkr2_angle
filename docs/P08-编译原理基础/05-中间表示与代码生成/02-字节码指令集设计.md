> **所属模块：** P08-编译原理基础

> **预计阅读时间：** 30 分钟
> **关键术语：** 字节码（Bytecode）、操作码（Opcode）、栈式虚拟机（Stack-based VM）、寄存器式虚拟机（Register-based VM）

## 本节目标

读完本节后，你将能够：
1. 理解字节码与机器码的区别——字节码是虚拟机的机器码
2. 掌握栈式和寄存器式两种字节码架构的核心差异
3. 能将简单表达式手工翻译为栈式和寄存器式字节码

5. 理解 TJS2 采用寄存器式字节码的设计考量

## 字节码 vs 机器码

| 维度 | 机器码 | 字节码 |
|------|--------|--------|
| 执行者 | CPU 硬件 | 虚拟机（软件） |
| 指令宽度 | 固定（如 ARM 32 位）或可变（如 x86） | 通常 1-4 字节操作码 |
| 操作数 | 寄存器、内存地址 | 虚拟寄存器、常量池索引 |
| 类型系统 | 无（操作原始比特） | 可以有（如 JVM 区分 iadd/fadd） |
| 可移植性 | 特定 CPU 架构 | 跨平台（任何有 VM 的地方都能跑） |
| 性能 | 最快 | 比机器码慢 10-100x，但 JIT 可接近原生 |

字节码本质是**为虚拟机设计的指令集**。Java、Python、Lua、TJS2 都使用字节码。

## 栈式 vs 寄存器式字节码

这是字节码设计最核心的架构选择。

### 栈式字节码（Stack-based）

所有运算在**操作数栈**上进行。操作数从栈中弹出，结果压回栈中：

```
// 计算 a + b * c
// 假设变量 a=寄存器0, b=寄存器1, c=寄存器2

LOAD 1          // 压入 b        栈: [b]
LOAD 2          // 压入 c        栈: [b, c]
MUL             // 弹出 b,c 压入 b*c  栈: [b*c]
LOAD 0          // 压入 a        栈: [b*c, a]
ADD             // 弹出并相加    栈: [a+b*c]
STORE 3         // 弹出存入结果  栈: []
```

**特点：** 指令短（不需要编码操作数位置），编译简单，但指令数多。

### 寄存器式字节码（Register-based）

操作数在**虚拟寄存器**（实际是数组槽位）中，指令直接指定操作数位置：

```
// 计算 a + b * c
// a=R0, b=R1, c=R2, 临时=R3, 结果=R4

MUL R3, R1, R2    // R3 = R1 * R2  (b * c)
ADD R4, R0, R3    // R4 = R0 + R3  (a + b*c)
```

**特点：** 指令宽（需要编码寄存器号），但指令数少，执行更快。

### 详细对比

| 维度 | 栈式 | 寄存器式 |
|------|------|---------|
| 代表 | JVM, Python, CLR, Lua 5.0 | TJS2, Lua 5.1+, Dalvik |
| 指令宽度 | 1 字节操作码 | 操作码 + 2-3 个操作数 |
| 指令数量 | 多（频繁 push/pop） | 少（一条指令做更多事） |
| 编译难度 | 简单（后序遍历 AST 即可） | 较复杂（需要寄存器分配） |
| 执行效率 | 较慢（栈操作开销） | 较快（减少内存访问） |
| 代码体积 | 小（每条指令短） | 大（每条指令编码操作数） |

### 动手实践——同一表达式的两种编码

表达式：`x = (a + b) * (c - d)`

**栈式字节码：**

```
LOAD a      // [a]
LOAD b      // [a, b]
ADD         // [a+b]
LOAD c      // [a+b, c]
LOAD d      // [a+b, c, d]
SUB         // [a+b, c-d]
MUL         // [(a+b)*(c-d)]
STORE x     // []
```

8 条指令。

**寄存器式字节码：**

```
ADD  R4, R0, R1    // R4 = a + b
SUB  R5, R2, R3    // R5 = c - d
MUL  R6, R4, R5    // R6 = R4 * R5
MOV  x, R6         // x = R6
```

4 条指令。寄存器式只需要一半的指令数。

## 设计迷你字节码指令集

下面我们设计一个完整的寄存器式字节码指令集，后面的实战章节会实现它的虚拟机。

### 指令编码格式

```
每条指令 = 1 字节操作码 + 0-3 个 1 字节操作数

格式 A: [opcode]                    // 无操作数（如 HALT, RET）
格式 B: [opcode] [dst]              // 单操作数（如 PRINT R0）
格式 C: [opcode] [dst] [src]        // 双操作数（如 MOV R0, R1）
格式 D: [opcode] [dst] [src1] [src2] // 三操作数（如 ADD R0, R1, R2）
```

### 指令集定义

```cpp
enum class OpCode : uint8_t {
    // 数据移动
    CONST,      // CONST Rd, #idx    — 从常量池加载到 Rd
    MOV,        // MOV Rd, Rs        — Rd = Rs
    
    // 算术运算
    ADD,        // ADD Rd, Rs1, Rs2  — Rd = Rs1 + Rs2
    SUB,        // SUB Rd, Rs1, Rs2  — Rd = Rs1 - Rs2
    MUL,        // MUL Rd, Rs1, Rs2  — Rd = Rs1 * Rs2
    DIV,        // DIV Rd, Rs1, Rs2  — Rd = Rs1 / Rs2
    MOD,        // MOD Rd, Rs1, Rs2  — Rd = Rs1 % Rs2
    NEG,        // NEG Rd, Rs        — Rd = -Rs
    
    // 比较运算（结果为 0 或 1）
    EQ,         // EQ  Rd, Rs1, Rs2  — Rd = (Rs1 == Rs2)
    LT,         // LT  Rd, Rs1, Rs2  — Rd = (Rs1 < Rs2)
    LE,         // LE  Rd, Rs1, Rs2  — Rd = (Rs1 <= Rs2)
    
    // 逻辑运算
    NOT,        // NOT Rd, Rs        — Rd = !Rs
    
    // 控制流
    JMP,        // JMP offset        — PC += offset
    JF,         // JF  Rs, offset    — if !Rs then PC += offset
    CALL,       // CALL Rd, func, argc — Rd = func(args...)
    RET,        // RET Rs            — return Rs
    
    // 其他
    PRINT,      // PRINT Rs          — 打印 Rs（调试用）
    HALT,       // HALT              — 停止执行
};
```

共 18 条指令，足以支持算术、比较、分支、函数调用。

### 编码为字节流

```cpp
struct Instruction {
    OpCode op;
    uint8_t a = 0;  // 操作数 1（通常是目标寄存器）
    uint8_t b = 0;  // 操作数 2
    uint8_t c = 0;  // 操作数 3
};

// 常量池
struct BytecodeChunk {
    std::vector<Instruction> code;     // 指令序列
    std::vector<double> constants;     // 数值常量
    std::vector<std::string> strings;  // 字符串常量
    int registerCount = 0;             // 需要的寄存器数
};
```

### 手工编码示例

将 `var x = 3 + 4 * 2;` 编码为字节码：

```cpp
BytecodeChunk chunk;
// 常量池
chunk.constants = {3.0, 4.0, 2.0};  // idx: 0, 1, 2
chunk.registerCount = 4;  // R0-R3

chunk.code = {
    {OpCode::CONST, 0, 0},        // R0 = 3.0    (constants[0])
    {OpCode::CONST, 1, 1},        // R1 = 4.0    (constants[1])
    {OpCode::CONST, 2, 2},        // R2 = 2.0    (constants[2])
    {OpCode::MUL,   3, 1, 2},     // R3 = R1 * R2 = 8.0
    {OpCode::ADD,   0, 0, 3},     // R0 = R0 + R3 = 11.0
    {OpCode::PRINT, 0},           // print R0 → 输出 11
    {OpCode::HALT},               // 停止
};
```

## 动手实践

1. **扩展指令集**：在本节的 `OpCode` 枚举中添加 `MOD`（取模）和 `NEG`（取负）指令。修改 `VM::execute` 的 switch-case 实现这两条新指令，编写测试用例验证 `10 % 3 = 1` 和 `-5` 的计算结果。

2. **实现栈式 VM**：仿照本节的寄存器式 VM，实现一个栈式 VM。定义 `PUSH`、`POP`、`ADD`、`MUL`、`PRINT`、`HALT` 六条指令，用 `std::stack<double>` 作为操作数栈。将 `3 + 4 * 2` 编码为栈式字节码并执行，验证输出 11。

3. **反汇编器**：编写一个 `disassemble(const BytecodeChunk& chunk)` 函数，将字节码翻译回可读的文本格式（如 `0000: CONST R0, #0 (3.000)`）。对本节的完整示例程序运行反汇编，确认输出与手写的指令序列一致。

4. **性能对比实验**：分别用栈式和寄存器式字节码实现同一段计算密集的循环（如求 1+2+...+1000000），用 `<chrono>` 测量两种 VM 的执行时间。记录指令条数和执行耗时，分析寄存器式指令"数量少但每条更复杂"的权衡。

## 对照项目源码——TJS2 的字节码设计

TJS2 在 `tjsInterCodeGen.h`（第 44-131 行）定义了完整的操作码枚举 `tTJSVMCodes`：

### TJS2 指令分类

| 类别 | 指令 | 说明 |
|------|------|------|
| 数据移动 | `VM_CONST`, `VM_CP`, `VM_CL`, `VM_CCL` | 加载常量、拷贝寄存器、清零 |
| 算术 | `VM_ADD`, `VM_SUB`, `VM_MUL`, `VM_DIV`, `VM_IDIV`, `VM_MOD` | 每个有 4 种变体（普通 + 3 种属性访问） |
| 位运算 | `VM_BAND`, `VM_BOR`, `VM_BXOR`, `VM_SAR`, `VM_SAL`, `VM_SR` | 与/或/异或/算术右移/左移/逻辑右移 |
| 比较 | `VM_CEQ`, `VM_CDEQ`, `VM_CLT`, `VM_CGT` | 等于/严格等于/小于/大于 |
| 逻辑 | `VM_LNOT`, `VM_LOR`, `VM_LAND` | 逻辑非/或/与 |
| 跳转 | `VM_JMP`, `VM_JF`, `VM_JNF` | 无条件跳转/假跳/非假跳 |
| 调用 | `VM_CALL`, `VM_CALLD`, `VM_CALLI`, `VM_NEW` | 调用/直接调用/间接调用/创建对象 |
| 属性 | `VM_GPD`, `VM_SPD`, `VM_GPI`, `VM_SPI` + 变体 | 通过名字/索引 获取/设置 属性 |
| 异常 | `VM_ENTRY`, `VM_EXTRY`, `VM_THROW` | try/catch 进入/退出/抛出 |
| 其他 | `VM_SRV`, `VM_RET`, `VM_GLOBAL`, `VM_TYPEOF` | 设返回值/返回/全局对象/类型检查 |

### TJS2 的属性访问变体宏

```cpp
#define TJS_NORMAL_AND_PROPERTY_ACCESSER(x) x, x##PD, x##PI, x##P
```

每个算术/逻辑指令都展开为 4 个操作码：
- `VM_ADD` — 普通寄存器加法
- `VM_ADDPD` — 属性直接（Property Direct）加法
- `VM_ADDPI` — 属性间接（Property Indirect）加法
- `VM_ADDP` — 属性（Property）加法

这是因为 TJS2 支持 `obj.prop += 1` 这样的操作——需要先读属性、执行运算、再写回属性。用专门指令一步完成，避免多条指令的开销。

### TJS2 指令编码

TJS2 使用 `tjs_uint32`（32 位无符号整数）数组存储字节码。每条指令占 1 个或多个 uint32 槽位：

```
[操作码] [操作数1] [操作数2] ...

操作码：32 位中低位部分为 tTJSVMCodes 枚举值
操作数：32 位有符号整数，表示寄存器偏移量（以 sizeof(tTJSVariant) 为单位）
```

注意操作数不是简单的寄存器编号，而是**字节偏移量**——乘以 `sizeof(tTJSVariant)` 后直接作为内存偏移使用，省去了运行时的乘法计算。

## 常见错误及解决方案

### 错误 1：混淆寄存器索引和常量池索引

```cpp
// CONST R0, 42   ← 42 是常量池索引，不是立即数！
// 如果常量池只有 3 个元素，42 就越界了
```

**解决：** 在指令定义中明确每个操作数的语义。CONST 指令的第二个操作数是常量池索引，ADD 指令的操作数是寄存器编号。

### 错误 2：忘记对齐跳转偏移量

`JMP offset` 的 offset 需要与指令计数保持一致。如果有的指令占 1 个槽位，有的占 4 个，偏移量计算很容易出错。

**解决：** 统一指令大小（如我们设计的固定 4 字节/条），或使用标签占位符，在编码完成后回填。

### 错误 3：栈式 VM 中栈不平衡

```
LOAD a    // 栈: [a]
LOAD b    // 栈: [a, b]
ADD       // 栈: [a+b]
ADD       // ❌ 栈只有1个元素，弹不出2个！
```

**解决：** 在编译阶段模拟栈深度变化，确保每条指令执行时栈有足够元素。JVM 的字节码验证器就做这件事。

## 本节小结

- 字节码是虚拟机的指令集，比机器码更高级，可跨平台执行
- 栈式字节码指令短但数量多，编译简单；寄存器式指令长但数量少，执行快
- 我们设计了 18 条指令的迷你寄存器式字节码，包含数据移动、算术、比较、控制流
- TJS2 使用寄存器式字节码，用宏展开为 4 种变体来优化属性访问
- TJS2 操作数用字节偏移量编码，直接用于内存寻址，避免运行时乘法

## 练习题与答案

### 题目 1：栈式编码

将 `result = a > b ? a : b`（取最大值）翻译为栈式字节码。提示：需要使用条件跳转。

<details>
<summary>查看答案</summary>

```
LOAD a          // [a]
LOAD b          // [a, b]
GT              // [a>b]       比较结果 0 或 1
JF L1           // 如果假(a<=b) 跳到 L1
LOAD a          // [a]         a > b 的情况
JMP L2
L1: LOAD b      // [b]         a <= b 的情况
L2: STORE result // []
```

共 8 条指令。栈式 VM 中三目运算符需要跳转来实现。

</details>

### 题目 2：寄存器式编码

用我们设计的迷你指令集，编码以下逻辑：

```
if (x < 10) { y = x + 1; } else { y = x - 1; }
```

假设 x 在 R0，y 在 R1，常量 10 在 constants[0]，常量 1 在 constants[1]。

<details>
<summary>查看答案</summary>

```
CONST R2, #0       // R2 = 10    (constants[0])
CONST R3, #1       // R3 = 1     (constants[1])
LT    R4, R0, R2   // R4 = (x < 10)
JF    R4, +3       // 如果 R4==0 跳过 3 条指令（到 else 分支）
ADD   R1, R0, R3   // y = x + 1  (then 分支)
JMP   +2           // 跳过 else 分支（2 条指令）
SUB   R1, R0, R3   // y = x - 1  (else 分支)
HALT               // 结束
```

关键是跳转偏移量的计算：JF 跳过 then 分支（ADD + JMP = 2 条），JMP 跳过 else 分支（SUB = 1 条）。实际偏移量需根据指令编号精确计算。

</details>

### 题目 3：分析 TJS2 操作码

TJS2 的 `VM_CDEQ` 和 `VM_CEQ` 有什么区别？为什么需要两个等于比较指令？（提示：类比 JavaScript 的 `==` 和 `===`）

<details>
<summary>查看答案</summary>

- `VM_CEQ` 对应 TJS2 的 `==` 运算符，执行**类型转换后比较**（如 `"123" == 123` 为真）
- `VM_CDEQ` 对应 TJS2 的 `===` 运算符，执行**严格比较**（类型和值都必须相同，`"123" === 123` 为假）

这与 JavaScript 的 `==`/`===` 完全类似。TJS2 作为动态类型脚本语言，需要同时支持宽松比较（方便脚本编写）和严格比较（避免隐式转换的陷阱）。大多数静态类型语言只需要一种等于运算符。

</details>
