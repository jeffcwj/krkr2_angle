## TJS2 编译器架构

TJS2 采用单遍编译——Bison 语法分析器在归约时调用 `tTJSInterCodeGenerator` 的方法，直接生成字节码。不存在独立的"编译器"阶段。

```
源码 → [词法器 tjsLex] → tokens → [语法器 tjs.y + Bison] → 归约动作
                                                                ↓
                                                    tTJSInterCodeGenerator
                                                    ├─ 管理符号表(Namespace)
                                                    ├─ 分配虚拟寄存器
                                                    ├─ 生成字节码到 CodeArea
                                                    └─ 存储常量到 DataArea
```

### 核心类：`tTJSInterCodeContext`

`tTJSInterCodeContext`（继承自 `tTJSInterCodeGenerator`）是 TJS2 中每个**编译单元**的核心。源文件 `tjsInterCodeGen.h` 第 390-450 行的关键成员：

```cpp
class tTJSInterCodeContext : public tTJSInterCodeGenerator {
    // 代码区 — 存储字节码指令
    tjs_int32 *CodeArea;       // 字节码数组（每个元素 32 位）
    tjs_int CodeAreaSize;       // 当前已用大小
    tjs_int CodeAreaCapa;       // 已分配容量

    // 数据区 — 存储常量（字符串、数字、对象引用等）
    tTJSVariant *DataArea;      // 常量池（每个元素是 tTJSVariant）
    tjs_int DataAreaSize;

    // 符号表（作用域栈）
    tTJSLocalNamespace Namespace;

    // 帧信息
    tjs_int FrameBase;          // 当前帧的寄存器基址
    tjs_int MaxFrameCount;      // 最大帧深度（函数嵌套）
    tjs_int MaxVariableCount;   // 最大局部变量数

    // 上下文元数据
    tTJSContextType ContextType; // ctTopLevel / ctFunction / ctClass 等
    tTJSScriptBlock *Block;      // 所属脚本块
    tTJSInterCodeContext *Parent; // 父上下文（闭包链）
};
```

### 上下文类型

每个函数、类、属性、顶层代码都有自己的 `tTJSInterCodeContext`：

```cpp
enum tTJSContextType {
    ctTopLevel,       // 顶层代码
    ctFunction,       // 普通函数
    ctExprFunction,   // 函数表达式
    ctProperty,       // 属性 getter/setter
    ctPropertySetter,
    ctPropertyGetter,
    ctClass,          // 类定义
    ctSuperClassGetter
};
```

一个包含函数和类的 TJS2 脚本会生成多个上下文，形成树形结构：

```
TopLevel Context
├── Function "main" Context
│   └── Function "helper" Context (闭包)
└── Class "Player" Context
    ├── PropertyGetter "name" Context
    └── Function "attack" Context
```

## 字节码存储格式

### 代码区（Code Area）

TJS2 字节码用 `tjs_int32`（32 位有符号整数）数组存储。每条指令由操作码和零个或多个操作数组成：

```
索引  [0]       [1]        [2]       [3]       [4]      [5]     ...
内容  VM_CONST  reg_offset data_idx  VM_ADD    dst_off  src1_off src2_off
      └── 指令1 ──────────┘         └── 指令2 ─────────────────┘
```

操作数是**字节偏移量**，不是寄存器编号。转换关系：

```cpp
#define TJS_TO_VM_REG_ADDR(x) ((x) * (tjs_int)sizeof(tTJSVariant))
#define TJS_FROM_VM_REG_ADDR(x) ((tjs_int)(x) / (tjs_int)sizeof(tTJSVariant))
```

在 64 位系统上 `sizeof(tTJSVariant)` 约为 20-24 字节，所以寄存器 0 的偏移量是 0，寄存器 1 的偏移量是 20（或 24），以此类推。

### 数据区（Data Area）

常量池存储编译时已知的值——数字字面量、字符串、正则表达式等：

```cpp
tTJSVariant *DataArea;   // 数组，每个元素是一个 tTJSVariant
tjs_int DataAreaSize;     // 元素数量
```

`VM_CONST` 指令的第二个操作数是数据区索引。例如 `VM_CONST R0, #3` 表示将 `DataArea[3]` 的值加载到寄存器 R0。

## 寄存器分配

TJS2 的"寄存器"实际是 `tTJSVariant` 数组的槽位。分配策略非常简单：

### 局部变量 = 固定寄存器

每个 `var` 声明通过 `Namespace.Add()` 分配一个寄存器槽位。`Add()` 返回的整数就是该变量在寄存器数组中的索引。在整个作用域内，变量始终占据同一个槽位。

### 临时值 = 动态分配

表达式求值需要临时寄存器。编译器维护一个"帧指针" `FrameBase`，临时寄存器在局部变量之后分配：

```
寄存器数组:
[0]    [1]    [2]    [3]    [4]    [5]   ...
 ↑      ↑      ↑      ↑      ↑      ↑
 this   局部x  局部y  临时t0  临时t1  临时t2
 └── 局部变量区 ──┘   └── 临时区 ────────┘
```

`MaxVariableCount` 记录局部变量数，`MaxFrameCount` 记录最大寄存器总数（含临时）。VM 在执行前按 `MaxFrameCount` 分配寄存器数组。

## 编译过程追踪

让我们追踪一段简单代码的编译全过程：

```javascript
// TJS2 代码
var x = 10;
var y = x + 20;
```

### 第一步：`var x = 10;`

1. 词法器产出 tokens: `T_VAR`, `T_SYMBOL("x")`, `=`, `T_CONSTVAL(10)`, `;`
2. Bison 匹配 `var_decl` 规则，触发语义动作
3. 语义动作调用 `Namespace.Add("x")`，分配寄存器 R1（R0 通常保留给 `this`）
4. 调用 `PutCode(VM_CONST)`，操作数为 R1 的偏移量和常量 10 的数据区索引
5. 常量 10 被添加到 DataArea：`DataArea[0] = tTJSVariant(10)`

生成字节码：`VM_CONST R1, #0`（将 DataArea[0] 即 10 加载到 R1）

### 第二步：`var y = x + 20;`

1. Bison 匹配 `var_decl`，其中初始化表达式是 `x + 20`
2. `Namespace.Add("y")` 分配 R2
3. 编译 `x + 20`：
   - `x` 通过 `Namespace.Find("x")` 解析为 R1
   - `20` 添加到 DataArea：`DataArea[1] = tTJSVariant(20)`
   - 分配临时寄存器 R3
   - 生成 `VM_CONST R3, #1`（加载 20 到临时寄存器）
   - 生成 `VM_ADD R2, R1, R3`（R2 = R1 + R3）

### 完整字节码

```
地址   指令
0000:  VM_CONST  R1, #0       // R1 = 10
0003:  VM_CONST  R3, #1       // R3 = 20   (临时寄存器)
0006:  VM_ADD    R2, R1, R3   // R2 = R1 + R3 = 30
```

数据区：`[0] = 10, [1] = 20`


## 跳转与回填

控制流指令（if、while、for）生成跳转字节码时，目标地址尚未确定。TJS2 使用**回填**（Backpatching）技术：

### 原理

```
1. 生成 JF 指令，跳转目标暂填 0：  JF R0, 0
2. 记录这条指令的地址到"待回填列表"
3. 继续编译后续代码...
4. 编译到目标位置时，用当前地址回填之前记录的位置
```

### TJS2 的实现

`tjsInterCodeGen.h` 中的 `NestVector` 和 `FixList` 用于管理回填：

```cpp
struct tNestData {
    tjs_int Type;       // 嵌套类型（if/while/for/switch）
    tjs_int Patch1;     // 需要回填的地址 1（如 if 的跳转）
    tjs_int Patch2;     // 需要回填的地址 2（如 else 的跳转）
    // ...
};
std::vector<tNestData> NestVector;  // 嵌套栈

struct tFixData {
    tjs_int StartIP;
    tjs_int Size;
    tjs_int NewAddress;
};
std::list<tFixData> FixList;        // 全局回填列表
```

当编译 `if (cond) { ... } else { ... }` 时：
1. 编译 `cond`，生成 `VM_JF R_cond, ???`，记录 Patch1
2. 编译 then 分支
3. 生成 `VM_JMP ???`，记录 Patch2
4. 回填 Patch1 = 当前地址（else 分支起点）
5. 编译 else 分支
6. 回填 Patch2 = 当前地址（if-else 之后的代码）

## 对照项目源码

TJS2 编译器的核心实现位于以下文件：

- `cpp/core/tjs2/tjsInterCodeGen.cpp` — 字节码生成器主体（`tTJSInterCodeGenerator` 类），包含所有 Bison 语义动作对应的代码生成方法
- `cpp/core/tjs2/tjsInterCodeGen.h` 第 44-131 行 — 操作码枚举 `tTJSVMCodes` 和寻址宏定义
- `cpp/core/tjs2/tjs.y` — Bison 语法文件，语义动作中调用 `cb->...` 方法生成字节码
- `cpp/core/tjs2/tjsInterCodeExec.cpp` — 字节码执行器，与生成器配对，可对照理解指令语义

重点关注 `CodeArea`（`tjs_int32` 数组）和 `DataArea`（`tTJSVariant` 数组）的交互方式——编译器向这两个数组写入数据，执行器从中读取执行。

## 常见错误及解决方案

### 错误 1：回填时地址偏移计算错误

忘记 TJS2 的代码区用 `tjs_int32` 数组索引，而操作数用字节偏移量，导致跳转到错误位置。

**解决：** 使用 `TJS_TO_VM_CODE_ADDR` / `TJS_FROM_VM_CODE_ADDR` 宏进行转换，不要手工计算。

### 错误 2：临时寄存器泄漏

编译复杂表达式后忘记释放临时寄存器，导致 `MaxFrameCount` 不断增长。

**解决：** 使用 RAII 或配对的 alloc/free 调用。TJS2 在表达式编译结束时重置帧指针，回收所有临时寄存器。

### 错误 3：常量池重复

同一个常量（如数字 0）被多次添加到数据区。

**解决：** 在添加常量前先搜索数据区是否已有相同值。TJS2 的实现中会检查已有常量避免重复。

## 本节小结

- TJS2 编译器 `tTJSInterCodeGenerator` 在 Bison 归约时直接生成字节码（单遍编译）
- 字节码存储在 `CodeArea`（uint32 数组），常量存储在 `DataArea`（Variant 数组）
- 每个函数/类/属性有独立的 `tTJSInterCodeContext`，形成树形上下文结构
- 寄存器分配：局部变量占固定槽位，临时值动态分配在其后
- 操作数用字节偏移量编码，直接用于内存寻址
- 跳转指令使用回填技术处理前向引用

## 练习题与答案

### 题目 1：手工编译

将以下 TJS2 代码手工编译为字节码（写出指令序列和数据区内容）：

```javascript
var a = 5;
var b = a * 3 + 1;
```

<details>
<summary>查看答案</summary>

假设 R0 = this, R1 = a, R2 = b, R3 = 临时

数据区：`[0] = 5, [1] = 3, [2] = 1`

```
0: VM_CONST  R1, #0       // R1 = 5       (a = 5)
3: VM_CONST  R3, #1       // R3 = 3
6: VM_MUL    R3, R1, R3   // R3 = R1 * R3 = 15   (a * 3)
9: VM_CONST  R4, #2       // R4 = 1
12: VM_ADD   R2, R3, R4   // R2 = R3 + R4 = 16   (a*3 + 1)
```

注意编译器可能复用临时寄存器 R3，先用于存 3，乘法后存结果 15，这样可以少用一个寄存器。

</details>

### 题目 2：回填分析

以下 TJS2 代码编译时，`VM_JF` 指令的跳转目标应该在第几条指令？

```javascript
if (x > 0) {
    y = 1;
} else {
    y = 2;
}
```

<details>
<summary>查看答案</summary>

```
0: VM_CGT    R_tmp, R_x, R_zero  // 比较 x > 0
3: VM_JF     R_tmp, → 9          // 假则跳到 else (地址 9)
5: VM_CONST  R_y, #idx_1         // y = 1
7: VM_JMP    → 11                // 跳过 else
9: VM_CONST  R_y, #idx_2         // y = 2  ← JF 跳转目标
11: ...                           // ← JMP 跳转目标
```

VM_JF 在地址 3，跳转到地址 9（else 分支的第一条指令）。回填发生在编译完 then 分支并生成 JMP 之后。

</details>

### 题目 3：寄存器数量分析

以下代码至少需要多少个寄存器（不含 this）？

```javascript
var a = 1;
var b = 2;
var c = (a + b) * (a - b);
```

<details>
<summary>查看答案</summary>

- R1 = a（局部变量）
- R2 = b（局部变量）
- R3 = c（局部变量）
- R4 = 临时（a + b 的结果）
- R5 = 临时（a - b 的结果）

最少需要 **5 个寄存器**（加上 R0=this 共 6 个）。编译器的 `MaxFrameCount` 会记录为 6。如果编译器优化复用临时寄存器，在 `a + b` 计算完成后、`a - b` 计算时可以复用 R4 之后的位置，但由于乘法需要两个操作数同时存在，至少需要 2 个临时寄存器。

</details>
