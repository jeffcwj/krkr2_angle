## 什么是字节码解释循环

编译器前端将源代码翻译为字节码，而**字节码解释器**（Bytecode Interpreter）逐条解读并执行这些字节码。其核心是一个循环：

```
  取指 (Fetch)  →  译码 (Decode)  →  执行 (Execute)  →  推进 IP  →  回到取指
```

这个 **Fetch-Decode-Execute Cycle** 持续运行，直到遇到 `HALT` 或 `RET` 指令终止。

### 基于栈 vs 基于寄存器的执行差异

| 特性 | 基于栈 | 基于寄存器 |
|------|--------|-----------|
| 操作数位置 | 隐式（栈顶） | 显式（寄存器编号） |
| 取操作数 | `pop()` | `regs[code[n]]` |
| 写结果 | `push()` | `regs[code[n]] = ...` |
| 指令条数 | 多（需 LOAD/STORE） | 少（直接引用寄存器） |

TJS2 采用**基于寄存器**的方案——每条指令通过寄存器偏移量直接读写操作数。

## 三种 Dispatch 技术

### 1. Switch-Dispatch（switch 分发）

最直观——用 `switch` 语句对操作码分支：

```cpp
// switch-dispatch 解释循环
void interpret(BytecodeChunk& chunk) {
    uint8_t* ip = chunk.code.data();  // 指令指针 (Instruction Pointer)
    while (true) {
        uint8_t opcode = *ip++;       // 取指
        switch (opcode) {             // 译码 + 执行
            case OP_ADD: {
                uint8_t dst = *ip++, a = *ip++, b = *ip++;
                regs[dst] = regs[a] + regs[b];
                break;
            }
            case OP_HALT: return;
        }
    }
}
```

**优点：** 可移植性好、调试方便。**缺点：** 循环中有一次间接跳转，CPU 分支预测不友好。

### 2. Computed Goto（计算跳转）

GCC/Clang 扩展支持 `&&label` 将标签地址存入数组，消除循环回跳：

```cpp
// GCC/Clang 专用（MSVC 不支持 &&label 语法）
static void* dispatch[] = { &&op_add, &&op_halt, /* ... */ };
#define DISPATCH() goto *dispatch[*ip++]
DISPATCH();
op_add: { /* ... */ DISPATCH(); }  // 每条指令末尾直接跳到下一条
op_halt: return;
```

**优点：** 比 switch 快 20-50%，CPU 可更精确地预测每个分发点的跳转目标。
**缺点：** 依赖 GCC 扩展，MSVC 不支持。

### 3. 尾调用分发（Tail Call Dispatch）

每个操作码用独立函数实现，函数末尾调用下一个操作码的处理函数。依赖编译器尾调用优化（TCO），否则栈溢出，实际较少使用。

### 性能对比

| 技术 | 相对性能 | 可移植性 | 使用案例 |
|------|---------|---------|---------|
| switch-dispatch | 基准 1.0× | ✅ 全平台 | CPython、TJS2、Lua 5.1 |
| computed goto | ~1.2-1.5× | ❌ GCC/Clang | Lua 5.2+、Ruby YARV |
| JIT 编译 | 5-50× | ❌ 平台相关 | V8、LuaJIT |

## 从零实现：寄存器式解释器

结合上一节的迷你指令集，实现完整解释器。

### 解释器核心

```cpp
#include <iostream>
#include <vector>
#include <cstdint>
#include <cmath>
#include <stdexcept>

enum class OpCode : uint8_t {
    CONST, MOV, ADD, SUB, MUL, DIV, MOD, NEG,
    EQ, LT, LE, NOT, JMP, JF, CALL, RET, PRINT, HALT
};
struct Instruction { OpCode op; uint8_t a, b, c; };
struct BytecodeChunk {
    std::vector<Instruction> code;
    std::vector<double>      constants;
    int registerCount = 16;
};

class VM {
    std::vector<double> regs;
public:
    void execute(const BytecodeChunk& chunk) {
        regs.assign(chunk.registerCount, 0.0);
        int ip = 0;
        const auto& code = chunk.code;

        while (ip < static_cast<int>(code.size())) {
            const auto& ins = code[ip];
            switch (ins.op) {
                case OpCode::CONST: regs[ins.a] = chunk.constants[ins.b]; ip++; break;
                case OpCode::MOV:   regs[ins.a] = regs[ins.b];           ip++; break;
                case OpCode::ADD:   regs[ins.a] = regs[ins.b] + regs[ins.c]; ip++; break;
                case OpCode::SUB:   regs[ins.a] = regs[ins.b] - regs[ins.c]; ip++; break;
                case OpCode::MUL:   regs[ins.a] = regs[ins.b] * regs[ins.c]; ip++; break;
                case OpCode::DIV:
                    if (regs[ins.c] == 0.0) throw std::runtime_error("division by zero");
                    regs[ins.a] = regs[ins.b] / regs[ins.c]; ip++; break;
                case OpCode::MOD:   regs[ins.a] = std::fmod(regs[ins.b], regs[ins.c]); ip++; break;
                case OpCode::NEG:   regs[ins.a] = -regs[ins.a]; ip++; break;
                case OpCode::EQ:
                    regs[ins.a] = (regs[ins.b] == regs[ins.c]) ? 1.0 : 0.0; ip++; break;
                case OpCode::LT:
                    regs[ins.a] = (regs[ins.b] < regs[ins.c]) ? 1.0 : 0.0; ip++; break;
                case OpCode::LE:
                    regs[ins.a] = (regs[ins.b] <= regs[ins.c]) ? 1.0 : 0.0; ip++; break;
                case OpCode::NOT:
                    regs[ins.a] = (regs[ins.a] == 0.0) ? 1.0 : 0.0; ip++; break;
                case OpCode::JMP:   ip += static_cast<int8_t>(ins.a); break;
                case OpCode::JF:
                    ip += (regs[ins.a] == 0.0) ? static_cast<int8_t>(ins.b) : 1; break;
                case OpCode::PRINT: std::cout << regs[ins.a] << "\n"; ip++; break;
                case OpCode::HALT:  return;
                default: throw std::runtime_error("unknown opcode");
            }
        }
    }
};
```

```cpp
    chunk.constants = {3.0, 4.0, 2.0};

        {OpCode::ADD,   0, 0, 3},  // r0 = r0 + r3 = 11.0
        {OpCode::PRINT, 0, 0, 0},  // print(r0) → "11"
        {OpCode::HALT,  0, 0, 0},
    };

    VM vm;
    vm.execute(chunk);  // 输出: 11
}
```

## 动手实践

1. **添加函数调用指令**：在本节的迷你 VM 中添加 `CALL` 和 `RET` 指令。`CALL offset` 将当前 IP 压入调用栈并跳转到 `offset`，`RET` 从调用栈弹出地址并跳回。编写一段字节码程序，定义一个"加倍"函数并调用两次，验证返回值正确。

2. **实现断点调试器**：为 VM 的 `execute` 方法添加一个 `std::set<int> breakpoints` 参数。当 IP 命中断点时暂停执行，打印当前所有寄存器的值，等待用户按回车继续。用这个调试器单步跟踪本节的倒计时程序。

3. **性能计数器**：在 `execute` 循环中添加一个 `std::unordered_map<OpCode, int> profile` 计数器，统计每种指令的执行次数。运行求和程序 `1+2+...+100` 后打印 profile，分析哪条指令执行最频繁、循环是否正确展开。

4. **对照 TJS2 的 switch 效率**：阅读 `tjsInterCodeExec.cpp` 第 853-871 行的 switch-case 结构，注意 TJS2 使用 `while(true) switch(*code)` 而非函数指针表或计算 goto。查阅你使用的编译器（GCC/Clang/MSVC）的文档，了解 switch-case 在 case 数量超过 50 时是否会自动优化为跳转表。

## 对照 TJS2 源码：ExecuteCode 解析

TJS2 的核心解释循环位于 `tjsInterCodeExec.cpp` 第 853-1303 行的 `ExecuteCode` 方法。

### 寄存器布局

```
文件：cpp/core/tjs2/tjsInterCodeExec.cpp 第 640-648 行

 regs[] 数组（VariantArrayStack 分配）
 ┌──────────────┬────────────┬───┬──────────────────┐
 │ 局部变量区    │ 保留区(2)  │RA │ 帧区(临时寄存器)  │
 │ MaxVariable  │ VarReserve │   │ MaxFrame         │
 └──────────────┴────────────┴─↑─┴──────────────────┘
                               ra（寄存器基地址指针）
 ra[-2] = objthis-proxy   ra[-1] = objthis
 ra[0]  = 结果返回槽       ra[-3], ra[-4]... = 函数参数（逆序）
```

```cpp

tjs_int num_alloc = MaxVariableCount + VariableReserveCount + 1 + MaxFrameCount;
```

### 解释循环主体

```cpp
// 第 853-871 行（简化）
tjs_int32 *code = CodeArea + startip;  // 指令指针
tTJSVariant *da = DataArea;            // 常量区
bool flag = false;                     // 条件标志位

while (true) {
    switch (*code) {
        case VM_NOP:   code++; break;
        case VM_CONST: // ra[code[1]] = da[code[2]]
            TJS_GET_VM_REG(ra, code[1]).CopyRef(TJS_GET_VM_REG(da, code[2]));
            code += 3; break;
        case VM_ADD:   // ra[code[1]] += ra[code[2]]
            TJS_GET_VM_REG(ra, code[1]).operator+=(TJS_GET_VM_REG(ra, code[2]));
            code += 3; break;
        case VM_RET:   return code + 1 - CodeArea;
        // ... 共约 80+ 个 case
    }
}
```

### 操作数寻址宏——字节偏移而非寄存器编号

```cpp
// cpp/core/tjs2/tjsInterCodeGen.h 第 34-41 行
#define TJS_TO_VM_REG_ADDR(x)  ((x) * (tjs_int)sizeof(tTJSVariant))
#define TJS_GET_VM_REG_ADDR(base, x) \
    ((tTJSVariant *)((char *)(base) + (tjs_int)(x)))
#define TJS_GET_VM_REG(base, x)  (*(TJS_GET_VM_REG_ADDR(base, x)))
```

`code[1]` 存的是 `n * sizeof(tTJSVariant)` 的字节偏移量，而非寄存器编号 `n`。这避免了运行时乘法（`tTJSVariant` 约 20 字节，非 2 的幂，无法用移位优化），寻址变为简单的指针加法。

### 条件标志与跳转

TJS2 用独立 `bool flag` 变量（类似 CPU FLAGS 寄存器）存储比较结果：

```cpp
case VM_CEQ:  // flag = (ra[code[1]] == ra[code[2]])
    flag = TJS_GET_VM_REG(ra, code[1]).NormalCompare(TJS_GET_VM_REG(ra, code[2]));
    code += 3; break;
case VM_JF:   // if (flag) jump
    if (flag) TJS_ADD_VM_CODE_ADDR(code, code[1]);
    else      code += 2;
    break;
```

### 算术操作的宏展开

每个算术操作有 4 个变体，用宏批量生成 case 分支：

```cpp
// 第 1010-1043 行
#define TJS_DEF_VM_P(vmcode, rope)              \
    case VM_##vmcode:                           \
        TJS_GET_VM_REG(ra,code[1]).rope(        \
            TJS_GET_VM_REG(ra,code[2]));        \
        code += 3; break;                       \
    case VM_##vmcode##PD: /* 属性直接 */         \
        OperatePropertyDirect(...); code+=5;break;\
    case VM_##vmcode##PI: /* 属性间接 */         \
        OperatePropertyIndirect(...);code+=5;break;\
    case VM_##vmcode##P:  /* 属性一般 */         \
        OperateProperty(...); code+=4; break

TJS_DEF_VM_P(ADD, operator+=);  // → VM_ADD, VM_ADDPD, VM_ADDPI, VM_ADDP
TJS_DEF_VM_P(SUB, operator-=);  // 同理 14 个运算符 × 4 = 56 个 case
```

### 异常处理（try-catch 映射）

TJS 脚本的 `try-catch` 映射为 C++ 的 `try-catch`——`VM_ENTRY` 递归调用 `ExecuteCode`：

```cpp
case VM_ENTRY:  // 进入 try 块
    code = CodeArea + ExecuteCodeInTryBlock(
        ra, code-CodeArea+3, args, numargs, result,
        catchip,    // catch 块起始 ip
        exobjreg);  // 异常对象存入的寄存器
    break;
case VM_EXTRY:  // 正常离开 try 块（等同 RET）
    return code + 1 - CodeArea;
```

## 常见错误与解决方案

### 错误 1：指令指针越界

**症状：** crash 或执行到错误操作码。
**原因：** JMP 偏移计算错误或忘记在某个 case 末尾推进 ip。
**解决：** `code += N` 的 N 必须精确匹配该指令的操作数个数。Debug 时加断言检查 ip 范围。

### 错误 2：寄存器越界

**症状：** 访问未初始化内存。
**原因：** 编译器分配的寄存器编号超过 `registerCount`。
**解决：** Debug 模式在每次寄存器访问前检查边界。

### 错误 3：除零未保护

**症状：** 程序崩溃或产生 `inf`/`nan`。
**解决：** 在 DIV/MOD 的 case 中显式检查除数为零，抛出有意义的错误。

## 本节小结

- 字节码解释器的核心是 **Fetch-Decode-Execute** 循环
- switch-dispatch 是最通用的方式，TJS2、CPython 等都采用此方案
- computed goto 可提升 20-50% 性能，但依赖 GCC/Clang 扩展
- TJS2 的 `ExecuteCode` 使用 `int32[]` 作为代码流，操作数为字节偏移量
- TJS2 用独立 `bool flag` 存储比较结果，与 CPU FLAGS 寄存器类似
- 算术操作通过宏展开为 4 个变体，这是 TJS2 操作码数量达 80+ 的主要原因
- try-catch 通过 C++ 原生异常机制实现，`VM_ENTRY` 递归调用 `ExecuteCode`

### 题目 2：解释 TJS2 为什么用字节偏移而非寄存器编号

TJS2 的 `code[1]` 存的是 `n * sizeof(tTJSVariant)` 而不是 `n`。解释性能好处和缺点。

<details>
<summary>查看答案</summary>

**好处：** 消除运行时乘法。`tTJSVariant` 约 20 字节（非 2 的幂，无法用移位优化），存字节偏移后寻址变为 `(char*)base + offset` 一次指针加法，x86 的 `[base+disp]` 寻址模式直接对应。

**缺点：**
1. 操作数从 1 字节膨胀为 4 字节，字节码体积增大
2. 反汇编可读性差，需除以 `sizeof(tTJSVariant)` 才能还原寄存器编号
3. 如果 `tTJSVariant` 大小变化，所有已编译字节码失效

</details>
