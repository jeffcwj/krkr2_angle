## 调用栈基础

### 什么是调用栈

当函数 A 调用函数 B，B 又调用函数 C 时，CPU 需要记录"每个函数执行到哪了、局部变量是什么"。这些信息存放在**调用栈**（Call Stack）上，每次调用压入一个**栈帧**（Stack Frame / Activation Record）：

```
┌─────────────────────┐ ← 栈顶 (Top)
│  帧 C               │
│  - 局部变量          │
│  - 返回地址 → B:L42  │
├─────────────────────┤
│  帧 B               │
│  - 局部变量          │
│  - 返回地址 → A:L17  │
├─────────────────────┤
│  帧 A               │
│  - 局部变量          │
│  - 返回地址 → main   │
└─────────────────────┘ ← 栈底 (Bottom)
```

函数返回时弹出栈顶帧，恢复调用者的执行状态——这就是"调用栈"名称的由来。

### 栈帧的组成

一个完整的栈帧通常包含以下字段：

| 字段 | 说明 | 示例 |
|------|------|------|
| **返回地址** (Return Address) | 调用完成后跳回的指令位置 | `ip = 42` |
| **函数引用** (Function Ref) | 当前正在执行的函数/字节码块 | 指向 `BytecodeChunk*` |
| **局部寄存器** (Local Registers) | 本函数的寄存器数组 | `regs[0..N]` |
| **参数区域** (Arguments) | 调用者传入的参数 | `arg0, arg1, ...` |
| **返回值槽** (Result Slot) | 存放函数返回值的位置 | `regs[dest]` |
| **调用者帧指针** (Caller FP) | 指向上一级帧，用于帧链还原 | `prev_frame` |

对于**基于寄存器的 VM**，局部变量和参数都存放在寄存器数组中，不需要额外的"局部变量区"——寄存器本身就是局部变量区。

## 实战：为迷你 VM 添加函数调用

### 回顾上一节的 VM 结构

上一节我们实现了一个支持 18 条指令的寄存器 VM，但 `CALL` 和 `RET` 还只是占位符。现在让我们补全它们。

### 第一步：定义栈帧结构

在上一节的 `BytecodeChunk` 基础上新增 `paramCount` 字段，并定义调用帧：

```cpp
// 在上一节的 BytecodeChunk 中追加一个字段
struct BytecodeChunk {
    // ... name, code, constants, registerCount（与上一节相同）
    int paramCount;    // 形参个数（新增）
};

// 调用帧——每次函数调用压入一个
struct CallFrame {
    const BytecodeChunk* chunk;     // 当前执行的字节码块
    int                  ip;        // 返回时恢复的指令指针
    int                  base;      // 本帧寄存器在全局 regs 中的起始下标
    int                  resultReg; // 调用者期望写入返回值的寄存器（绝对下标）
};
```

关键设计：所有函数共享一个大的寄存器数组 `regs`，每个帧占据 `regs[base .. base+registerCount-1]`。调用时在栈顶追加新帧的寄存器空间，返回时回收：

```
regs[] 全局数组:
┌──────────────┬──────────────┬──────────────┐
│ 帧A [0..7]   │ 帧B [8..15]  │ 帧C [16..23] │
└──────────────┴──────────────┴──────────────┘
 base=0          base=8         base=16
```

### 第二步：在 VM 中实现 CALL 和 RET

上一节的 `VM::execute` 的 switch 中，`CALL` 和 `RET` 是占位。现在补全（其余 case 不变，只展示新增部分）：

```cpp
// ========== 核心：函数调用与返回 ==========
case OpCode::CALL: {
    // ins.a = 目标寄存器（写入返回值）
    // ins.b = 被调函数在 chunks 中的索引
    // ins.c = 实参个数
    const auto& callee = chunks[ins.b];
    int newBase = base + chunk->registerCount; // 新帧紧接当前帧

    // 复制实参：调用者 regs[base+1..] → 被调者 regs[newBase+0..]
    for (int i = 0; i < ins.c; ++i)
        regs[newBase + i] = reg(base, 1 + i);

    // 保存返回信息并压入新帧
    callStack.back().ip = ip + 1;
    callStack.push_back({&callee, 0, newBase, base + ins.a});

    // 切换到被调函数
    chunk = &callee;
    base  = newBase;
    ip    = 0;
    break;
}
case OpCode::RET: {
    double retVal = reg(base, ins.a);  // 取返回值
    CallFrame finished = callStack.back();
    callStack.pop_back();

    regs[finished.resultReg] = retVal; // 写到调用者指定寄存器

    // 恢复调用者上下文
    const auto& caller = callStack.back();
    chunk = caller.chunk;
    base  = caller.base;
    ip    = caller.ip;
    break;
}
```

核心逻辑：**`CALL`** 在栈顶开辟新寄存器空间、复制参数、压帧、跳转。**`RET`** 写返回值、弹帧、恢复。

### 第四步：测试递归——阶乘函数

```cpp
// 构造字节码：factorial(n) = n <= 1 ? 1 : n * factorial(n-1)
// chunk[0] = main:  调用 factorial(5)，打印结果
// chunk[1] = factorial: 递归实现

int main() {
    VM vm;
    std::vector<BytecodeChunk> chunks(2);

    // -- chunk[0]: main --
    // regs: r0=result, r1=arg(5)
    auto& m = chunks[0];
    m.name = "main";
    m.registerCount = 4;
    m.paramCount = 0;
    m.constants = {5.0, 1.0};
    m.code = {
        {OpCode::CONST, 1, 0, 0},  // r1 = 5.0      (传给factorial的参数)
        {OpCode::CALL,  0, 1, 1},  // r0 = factorial(r1)  chunk[1], 1个参数
        {OpCode::PRINT, 0, 0, 0},  // print r0       → 应输出 120
        {OpCode::HALT,  0, 0, 0},
    };

    // -- chunk[1]: factorial(n) --
    // regs: r0=n(参数), r1=temp, r2=temp2
    auto& f = chunks[1];
    f.name = "factorial";
    f.registerCount = 4;
    f.paramCount = 1;
    f.constants = {1.0};
    f.code = {
        // if (n <= 1) return 1
        {OpCode::CONST, 1, 0, 0},  // r1 = 1.0
        {OpCode::LE,    0, 1, 0},  // flag = (r0 <= r1)
        {OpCode::JF,    4, 0, 0},  // if (!flag) jump +4 → 到递归分支
        {OpCode::RET,   1, 0, 0},  // return r1 (= 1.0)
        // else: return n * factorial(n - 1)
        {OpCode::SUB,   1, 0, 1},  // r1 = r0 - r1   (n - 1)，注意此时 r1 仍为 1.0
        {OpCode::MOV,   2, 0, 0},  // r2 = r0         (保存 n，因为 CALL 会覆盖 r1)
        // 准备调用：r1 = n-1 已经是参数
        {OpCode::CALL,  3, 1, 1},  // r3 = factorial(r1)  传 1 个参数
        {OpCode::MUL,   0, 2, 3},  // r0 = r2 * r3    (n * factorial(n-1))
        {OpCode::RET,   0, 0, 0},  // return r0
    };

    vm.execute(chunks);  // 输出: 120
    return 0;
}
```

运行后输出 `120`，证明递归调用栈正确工作。调用栈最深时有 6 帧（main + factorial(5) 到 factorial(1)），每帧占 4 个寄存器，共使用 `regs[0..23]`。

## 参数传递策略对比

| 策略 | 原理 | 代表 VM | 优点 | 缺点 |
|------|------|---------|------|------|
| **寄存器窗口** | 调用者/被调者寄存器区域重叠 | Lua 5.x | 零拷贝传参 | 编译器需精确布局 |
| **栈传参** | 参数压栈，被调者从栈读取 | JVM, CPython | 实现简单 | push/pop 开销 |
| **数组拷贝** | 分配新数组，逐个复制参数 | TJS2, 迷你VM | 帧完全隔离 | 拷贝开销 |

Lua 的寄存器窗口方案将调用者末尾寄存器与被调者开头寄存器重叠，实现零拷贝。性能最优，但对编译器要求高——必须保证参数刚好在正确位置。

### TJS2 的数组拷贝方案

TJS2 采用数组拷贝方式：每次调用从 `VariantArrayStack` 分配一块新的寄存器数组，再把参数逐个复制进去。虽然有拷贝开销，但帧之间完全隔离，不易出 bug：

```cpp
// tjsInterCodeExec.cpp 第 703-728 行（简化）
if (numargs >= FuncDeclArgCount) {
    tTJSVariant *r = ra - 3;       // 参数从 ra[-3] 开始，向下增长
    tTJSVariant **a = args;         // args 是调用者传入的参数数组
    tjs_int n = FuncDeclArgCount;
    while (true) {
        *r = **(a++);               // 逐个复制
        if (--n == 0) break;
        r--;                        // 向低地址方向
    }
}
```

注意参数存放方向是**向低地址增长**：第一个参数在 `ra[-3]`，第二个在 `ra[-4]`，以此类推。

## 对照项目源码：TJS2 函数调用全流程

TJS2 的函数调用涉及三个关键函数，形成清晰的调用链：

```
VM_CALL 指令
  → CallFunction()           // 准备参数、提取闭包
    → clo.FuncCall()         // 通过闭包接口调用
      → ExecuteAsFunction()  // 分配帧、复制参数、执行
        → ExecuteCode()      // 进入被调函数的字节码循环
```

### 第一层：`CallFunction`（参数准备）

```cpp
// tjsInterCodeExec.cpp 第 2100-2143 行
tjs_int CallFunction(tTJSVariant *ra, const tjs_int32 *code,
                     tTJSVariant **args, tjs_int numargs) {
    TJS_BEGIN_FUNC_CALL_ARGS(code + 3)  // 宏：从字节码读取参数列表

    // 从寄存器中取出要调用的函数对象（闭包）
    tTJSVariantClosure clo = TJS_GET_VM_REG(ra, code[2]).AsObjectClosure();

    if (code[0] == VM_CALL) {
        // 普通函数调用
        hr = clo.FuncCall(0, nullptr, nullptr,
            code[1] ? TJS_GET_VM_REG_ADDR(ra, code[1]) : nullptr,  // 返回值目标
            pass_args_count, pass_args,                              // 参数
            clo.ObjThis ? clo.ObjThis : ra[-1].AsObjectNoAddRef()); // this 对象
    } else {
        // VM_NEW: 创建新对象
        iTJSDispatch2 *dsp;
        hr = clo.CreateNew(0, nullptr, nullptr, &dsp,
            pass_args_count, pass_args,
            clo.ObjThis ? clo.ObjThis : ra[-1].AsObjectNoAddRef());
        // ... 包装为 tTJSVariant 写入目标寄存器 ...
    }
    clo.Release();  // 释放闭包引用
    return code_size + 3;
}
```

关键点：`code[1]` 是结果寄存器，`code[2]` 是函数对象寄存器，`code[3..]` 编码参数列表。

### 第二层：`ExecuteAsFunction`（帧分配）

```cpp
// tjsInterCodeExec.cpp 第 635-777 行（骨架）
void ExecuteAsFunction(iTJSDispatch2 *objthis,
                       tTJSVariant **args, tjs_int numargs,
                       tTJSVariant *result, tjs_int start_ip) {
    // 1. 计算帧大小并分配寄存器数组
    tjs_int num_alloc = MaxVariableCount + VariableReserveCount + 1 + MaxFrameCount;
    tTJSVariant *regs = TJSVariantArrayStack->Allocate(num_alloc);
    tTJSVariant *ra = regs + MaxVariableCount + VariableReserveCount;

    // 2. 设置 objthis 代理（ra[-2]）和 objthis（ra[-1]）
    tTJSObjectProxy proxy;
    proxy.SetObjects(objthis, Block->GetTJS()->GetGlobalNoAddRef());
    ra[-2] = &proxy;   // 双分派代理
    ra[-1].SetObject(objthis, objthis);
    ra[0].Clear();      // 返回值槽

    // 3. 复制参数到寄存器（从 ra[-3] 向下）
    if (numargs >= FuncDeclArgCount) {
        tTJSVariant *r = ra - 3;
        for (int i = 0; i < FuncDeclArgCount; i++)
            *(r--) = *args[i];
    }

    // 4. 可变参数折叠（如有）
    if (FuncDeclCollapseBase >= 0) {
        // 创建数组对象，将多余参数收入其中
    }

    // 5. 进入字节码执行循环
    ExecuteCode(ra, start_ip, args, numargs, result);

    // 6. 清理：释放寄存器数组
    ra[-2].Clear();
    TJSVariantArrayStack->Deallocate(num_alloc, regs);
}
```

## 闭包：函数 + 环境

### 什么是闭包

**闭包**（Closure）= 函数代码 + 它捕获的外部变量。当一个内部函数引用了外层函数的局部变量时，即使外层函数已经返回，这些变量仍需保持存活——闭包正是实现这一点的机制。

```
function makeCounter() {
    var count = 0;                // 局部变量
    return function() {           // 内部函数 → 闭包
        count++;                  // 捕获外层的 count
        return count;
    };
}
var c = makeCounter();
c();  // → 1
c();  // → 2  （count 依然存活，没有随 makeCounter 返回而销毁）
```

### 闭包的实现策略

| 策略 | 原理 | 代表语言 |
|------|------|----------|
| **Upvalue/引用捕获** | 被捕获变量升级到堆上，闭包持有指针 | Lua, Python |
| **对象即闭包** | 闭包用对象表示，捕获的变量是对象成员 | TJS2, JavaScript |
| **拷贝捕获** | 创建闭包时复制变量值（不共享后续修改） | C++ lambda（默认） |

### TJS2 的闭包实现

TJS2 中函数就是对象（`tTJSInterCodeContext` 继承自 `iTJSDispatch2`），闭包的"捕获环境"通过 **objthis** 实现——调用时 `VM_CHGTHIS` 指令把当前上下文绑定为函数的 `this`：

```cpp
// tjsInterCodeExec.cpp 第 1272-1276 行
case VM_CHGTHIS:
    TJS_GET_VM_REG(ra, code[1])
        .ChangeClosureObjThis(
            TJS_GET_VM_REG(ra, code[2]).AsObjectNoAddRef());
    code += 3;
    break;
```

`ChangeClosureObjThis` 修改变量内部的 `ObjThis` 指针——这样当闭包被调用时，它访问的 `this` 就指向创建它的那个上下文对象，从而能访问外层变量。

TJS2 的 8 种上下文类型（`tTJSContextType`）中，`ctFunction` 和 `ctExprFunction` 都可以产生闭包。`tTJSObjectProxy` 的双分派机制确保了：先在 `objthis`（闭包绑定的对象）中查找成员，找不到再回退到全局对象。

## 常见错误与解决方案

### 错误 1：调用栈溢出（Stack Overflow）

```
错误现象：递归调用时程序崩溃，或报 "stack overflow"
原因：递归过深，callStack 无限增长
解决：添加栈深度检查
```

```cpp
case OpCode::CALL: {
    if (callStack.size() >= 512) {
        std::cerr << "Stack overflow: max call depth exceeded\n";
        return;
    }
    // ... 正常调用逻辑 ...
}
```

### 错误 2：返回值寄存器地址错误

```
错误现象：函数返回后，调用者读到的值不对（是 0 或旧值）
原因：resultReg 计算为相对偏移而非绝对下标
解决：确保 resultReg = base + ins.a（绝对地址）
```

### 错误 3：参数个数不匹配未处理

```
错误现象：传入参数少于形参个数时，被调者读到未初始化的寄存器
原因：只复制了 numargs 个参数，剩余参数位置仍是上次调用的残留值
解决：像 TJS2 一样，不足的参数清零
```

```cpp
for (int i = 0; i < ins.c && i < callee.paramCount; ++i)
    regs[newBase + i] = reg(base, 1 + i);
for (int i = ins.c; i < callee.paramCount; ++i)
    regs[newBase + i] = 0.0;  // 未传入的参数默认为 0
```

## 本节小结

- 调用栈是函数调用的核心数据结构，每次调用压入帧，返回时弹出
- 栈帧记录返回地址、寄存器基址、返回值目标等信息
- 参数传递有三种主流方案：寄存器窗口（零拷贝）、栈传参、数组拷贝
- TJS2 采用数组拷贝方案，参数从 `ra[-3]` 向低地址方向存放
- 闭包 = 函数代码 + 捕获的环境引用；TJS2 通过 `objthis` 绑定实现闭包语义

```cpp
    {OpCode::RET,  0, 0, 0},  // 永远不会执行到
};
```

运行后应输出 `Error: stack overflow (depth=256)` 并正常退出。

</details>

### 题目 2：解释 TJS2 为什么用 `ra[-3]` 而不是 `ra[1]` 存放第一个参数

<details>
<summary>查看答案</summary>

TJS2 的寄存器布局中，`ra[0]` 是返回值槽，`ra[-1]` 是 `objthis`，`ra[-2]` 是 `objthis-proxy`。这三个位置被占用，所以参数只能从 `ra[-3]` 开始。向低地址方向增长的设计使得参数区域不会与帧区域（`ra[1..]`）冲突——帧区域用于子函数调用的临时寄存器，向高地址方向增长。两个方向互不干扰。

</details>

### 题目 3：闭包与普通函数调用的区别

请说明在 TJS2 中，调用一个闭包函数和调用一个普通全局函数，在 `CallFunction` → `FuncCall` 这条路径上有什么不同。

<details>
<summary>查看答案</summary>

关键区别在于 `ObjThis` 的来源：

```cpp
clo.FuncCall(0, nullptr, nullptr, result, pass_args_count, pass_args,
    clo.ObjThis ? clo.ObjThis : ra[-1].AsObjectNoAddRef());
```

- **闭包函数**：`clo.ObjThis` 不为空——它指向闭包创建时通过 `VM_CHGTHIS` 绑定的上下文对象。被调者的 `ra[-1]` 将是这个对象，从而可以访问外层函数的局部变量
- **普通全局函数**：`clo.ObjThis` 为空，回退到 `ra[-1]`（当前帧的 `objthis`，通常是全局对象）。被调者只能访问全局作用域

</details>
