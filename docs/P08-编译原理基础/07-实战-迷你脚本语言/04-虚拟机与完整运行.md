## 虚拟机实现

### VM 数据结构

回顾 Ch06 的 CallFrame 和 VM 设计，在此整合为完整实现：

```cpp
#include <cstdint>
#include <stdexcept>
#include <string>
#include <vector>
#include <iostream>
#include <cmath>

// OpCode、Instruction、BytecodeChunk 定义复用前节（见 03-字节码编译器.md）

struct CallFrame {
    const BytecodeChunk* chunk;  // 当前执行的字节码块
    int ip;                      // 指令指针 (Instruction Pointer)
    int base;                    // 寄存器窗口基址
    int resultReg;               // 调用方期望的返回值寄存器
};

class VM {
    std::vector<double> regs_;           // 寄存器数组
    std::vector<CallFrame> callStack_;   // 调用栈
    static constexpr int MAX_REGS = 1024;
    static constexpr int MAX_CALL_DEPTH = 256;

    // 获取当前帧
    CallFrame& frame() { return callStack_.back(); }

    // 读取当前指令并推进 IP
    Instruction fetch() {
        auto& f = frame();
        return f.chunk->code[f.ip++];
    }

    // 绝对寄存器索引 = 帧基址 + 相对偏移
    double& reg(uint8_t r) { return regs_[frame().base + r]; }

public:
    void execute(const std::vector<BytecodeChunk>& chunks) {
        regs_.resize(MAX_REGS, 0.0);

        // 从 <main>（chunks[0]）开始执行
        callStack_.push_back({&chunks[0], 0, 0, 0});

        while (!callStack_.empty()) {
            Instruction instr = fetch();
            switch (instr.op) {

            case OpCode::CONST:  // CONST dst, constIdx
                reg(instr.a) = frame().chunk->constants[instr.b];
                break;

            case OpCode::MOV:    // MOV dst, src
                reg(instr.a) = reg(instr.b);
                break;

            case OpCode::ADD: reg(instr.a) = reg(instr.b) + reg(instr.c); break;
            case OpCode::SUB: reg(instr.a) = reg(instr.b) - reg(instr.c); break;
            case OpCode::MUL: reg(instr.a) = reg(instr.b) * reg(instr.c); break;
            case OpCode::DIV:
                if (reg(instr.c) == 0.0) throw std::runtime_error("除以零");
                reg(instr.a) = reg(instr.b) / reg(instr.c);
                break;
            case OpCode::MOD:
                reg(instr.a) = std::fmod(reg(instr.b), reg(instr.c));
                break;
            case OpCode::NEG: reg(instr.a) = -reg(instr.b); break;

            case OpCode::EQ:  // EQ dst, a, b → dst = (a == b) ? 1.0 : 0.0
                reg(instr.a) = (reg(instr.b) == reg(instr.c)) ? 1.0 : 0.0;
                break;
            case OpCode::LT:
                reg(instr.a) = (reg(instr.b) < reg(instr.c)) ? 1.0 : 0.0;
                break;
            case OpCode::LE:
                reg(instr.a) = (reg(instr.b) <= reg(instr.c)) ? 1.0 : 0.0;
                break;
            case OpCode::NOT:
                reg(instr.a) = (reg(instr.b) == 0.0) ? 1.0 : 0.0;
                break;

            case OpCode::JMP: {  // JMP addr（地址编码在 a|b）
                size_t target = static_cast<size_t>(instr.a)
                              | (static_cast<size_t>(instr.b) << 8);
                frame().ip = static_cast<int>(target);
                break;
            }
            case OpCode::JF: {   // JF addr, condReg
                size_t target = static_cast<size_t>(instr.a)
                              | (static_cast<size_t>(instr.b) << 8);
                if (reg(instr.c) == 0.0)  // 条件为假 → 跳转
                    frame().ip = static_cast<int>(target);
                break;
            }

            case OpCode::CALL: { // CALL funcIdx, baseReg, argc
                int funcIdx = instr.a;
                int baseReg = instr.b;
                int argc    = instr.c;
                if (funcIdx >= static_cast<int>(chunks.size()))
                    throw std::runtime_error("函数索引越界");
                if (callStack_.size() >= MAX_CALL_DEPTH)
                    throw std::runtime_error("调用栈溢出（递归过深）");
                // 新帧的寄存器窗口从 base + baseReg + 1 开始
                int newBase = frame().base + baseReg + 1;
                callStack_.push_back({&chunks[funcIdx], 0, newBase, frame().base + baseReg});
                break;
            }

            case OpCode::RET: {  // RET valReg, hasValue
                double retVal = (instr.b == 1) ? reg(instr.a) : 0.0;
                int resultReg = frame().resultReg;
                callStack_.pop_back();
                if (callStack_.empty()) return;  // main 返回，结束
                regs_[resultReg] = retVal;       // 写入调用方的结果寄存器
                break;
            }

            case OpCode::PRINT:
                std::cout << reg(instr.a) << "\n";
                break;

            case OpCode::HALT:
                return;
            }
        }
    }
};
```

**关键实现细节：**

| 机制 | 实现方式 |
|------|----------|
| 寄存器窗口 | `frame().base + r` 将相对寄存器号映射到全局数组 |
| 函数调用 | `CALL` 压入新 `CallFrame`，参数通过寄存器窗口传递 |
| 函数返回 | `RET` 弹出帧，将返回值写入调用方的 `resultReg` |
| 跳转编码 | 16 位地址拆分到 `a`（低 8 位）和 `b`（高 8 位） |
| 错误处理 | 除以零、调用栈溢出、函数索引越界均抛异常 |

## 端到端管线集成

现在将 Lexer、Parser、Compiler、VM 连接成完整管线：

```cpp
// Lexer、Parser、Compiler 类定义见前三节
// 此处展示集成代码

void run(const std::string& source) {
    // 阶段 1: 词法分析
    Lexer lexer(source);
    std::vector<Token> tokens = lexer.scanAll();

    // 阶段 2: 语法分析
    Parser parser(tokens);
    std::vector<StmtPtr> ast = parser.parseProgram();

    // 阶段 3: 编译
    Compiler compiler;
    std::vector<BytecodeChunk> chunks = compiler.compile(ast);

    // 阶段 4: 执行
    VM vm;
    vm.execute(chunks);
}

int main() {
    std::string program = R"(
        var x = 10;
        var y = 20;
        print x + y;
    )";
    run(program);  // 输出: 30
    return 0;
}
```

管线数据流：

```
源代码字符串  →  Lexer   →  Token 流
Token 流     →  Parser  →  AST (语句列表)
AST          →  Compiler →  BytecodeChunk 列表
BytecodeChunk →  VM      →  程序输出
```

## 完整测试程序

上面的简单加法只验证了基础管线。下面用三个逐渐复杂的程序，覆盖函数调用、递归、条件分支和嵌套作用域。

### 测试 1：Fibonacci 数列

```cpp
// fibonacci.cpp — 用 MiniScript 计算第 10 个 Fibonacci 数
int main() {
    std::string source = R"(
        fn fib(n) {
            if (n <= 1) {
                return n;
            }
            return fib(n - 1) + fib(n - 2);
        }
        print fib(10);
    )";
    run(source);  // 期望输出: 55
    return 0;
}
```

**执行过程追踪（以 `fib(4)` 为例）：**

```
fib(4) → fib(3) → fib(2) → fib(1) ret 1    帧 0: base=0  (main)
                           → fib(0) ret 0    帧 1: base=4  (fib, n=4)
                  ← ret 1                    帧 2: base=12 (fib, n=3)
         → fib(1) ret 1                      帧 3: base=20 (fib, n=2)
← ret 2+1 = 3                                帧 4: base=28 (fib, n=1)
```

每次 `CALL` 产生新帧，`base` 偏移 = 上层 `base + baseReg + 1`。`RET` 弹帧后将返回值写回调用方的 `resultReg`。MAX_REGS=1024 可支持约 128 层递归深度（每帧 ~8 寄存器）。

### 测试 2：阶乘（带错误检查）

```cpp
int main() {
    std::string source = R"(
        fn factorial(n) {
            if (n <= 1) {
                return 1;
            }
            return n * factorial(n - 1);
        }

        print factorial(1);
        print factorial(5);
        print factorial(10);
    )";
    run(source);
    // 期望输出:
    // 1
    // 120
    // 3628800
    return 0;
}
```

**编译产物分析（`factorial` 函数字节码）：**

```
chunk[1]: "factorial" (paramCount=1, registerCount=6)
  0: CONST r1, 1       ; 常量 1.0
  1: LE    r2, r0, r1  ; r2 = (n <= 1)
  2: JF    5|0, r2     ; 若 false 跳到 5
  3: CONST r3, 1       4: RET r3, 1        ; return 1 (基线)
  5: CONST r4, 1       6: SUB r5, r0, r4   ; r5 = n-1
  7: CALL  1, r3, 1    ; r3 = factorial(n-1)  ← funcIdx=1 即自身
  8: MUL   r2, r0, r3  9: RET r2, 1        ; return n * factorial(n-1)
```

第 7 条 `CALL 1, r3, 1` 中 `funcIdx=1` 就是 `factorial` 在 `chunks` 数组中的索引（`chunks[0]` 是 `<main>`），这就是递归的字节码表达。

### 测试 3：嵌套作用域与变量遮蔽

```cpp
int main() {
    std::string source = R"(
        var x = 10;
        print x;
        {
            var x = 20;
            print x;
            {
                var x = 30;
                print x;
            }
            print x;
        }
        print x;
    )";
    run(source);
    // 期望输出:
    // 10
    // 20
    // 30
    // 20
    // 10
    return 0;
}
```

**作用域机制解析：**

编译器的 `ScopeChain` 为每层 `{}` 块分配独立的寄存器槽。进入块时 `pushScope()`，退出时 `popScope()` 释放该层的寄存器。三个 `x` 变量在编译后映射到不同寄存器：

```
作用域深度 0:  x → r0   (值 10)
作用域深度 1:  x → r1   (值 20)  — 遮蔽外层 r0
作用域深度 2:  x → r2   (值 30)  — 遮蔽外层 r1
```

退出内层块后，编译器的名称查找回到外层，`print x` 自动绑定到外层寄存器。这与 TJS2 的作用域链机制（`tjs2/tjsInterCodeGen.cpp` 中的 `PushContextState` / `PopContextState`）原理一致。

## MiniScript 与 TJS2 管线对照

MiniScript 的四阶段管线直接映射到 TJS2 的实现。下面逐阶段对照：

| 管线阶段 | MiniScript | TJS2 | 关键差异 |
|----------|-----------|------|----------|
| 词法分析 | `Lexer` — 手写单遍扫描，35 种 Token | `tjsLex.cpp` — 手写词法器，支持正则表达式字面量、八进制转义 | TJS2 词法器复杂 10 倍（~1900 行 vs ~200 行） |
| 语法分析 | `Parser` — 手写递归下降 | `bison/tjs.y` — Bison 生成的 LALR(1) 解析器 | MiniScript 手写更直观；TJS2 用生成器处理复杂语法 |
| 编译 | `Compiler` — AST 直接生成字节码 | `tjsInterCodeGen` — AST→中间码，含常量折叠、寄存器分配 | TJS2 有优化 pass，MiniScript 无优化 |
| 执行 | `VM::execute()` — 18 条指令 | `tjsInterCodeExec.cpp` — ~90 条指令（含对象操作、属性访问、正则匹配） | TJS2 是完整语言 VM；MiniScript 是教学子集 |

### TJS2 管线流程

```
.tjs 文件 → tjsLex(词法) → tjs.y(Bison语法) → tjsInterCodeGen(中间码)
         → CodeArea(字节码) + DataArea(常量池) → tjsInterCodeExec(VM执行) → 输出
```

**TJS2 的额外复杂度：** (1) **对象系统**——操作 `tTJSVariant`（变体类型）而非 `double`，每条算术指令内部做类型判断转换；(2) **属性访问**——`obj.prop` 和 `obj["prop"]` 需要 `VM_SET`/`VM_GET` 操作码族；(3) **异常处理**——内置 `try-catch-finally`，维护异常帧栈；(4) **闭包**——函数捕获外层变量，VM 维护闭包环境链。MiniScript 刻意省略这些特性，让你专注理解编译→执行的核心流程。

## 常见错误与解决方案

### 错误 1：调用栈溢出（无限递归）

```cpp
// 错误代码：忘记基线条件
std::string bad_code = R"(
    fn forever(n) {
        return forever(n + 1);  // 永远不会停止
    }
    print forever(0);
)";
```

**症状：** `runtime_error: 调用栈溢出（递归过深）`

**原因：** 没有基线条件（base case），`CALL` 不断压入新帧，超过 `MAX_CALL_DEPTH=256`。

**解决方案：** 确保每个递归函数都有终止条件：

```cpp
fn forever(n) {
    if (n >= 100) { return n; }  // 添加基线条件
    return forever(n + 1);
}
```

**TJS2 中的对应行为：** TJS2 同样有调用栈深度限制（默认 ~1024 帧），超出后抛出 `eTJSScriptError`。

### 错误 2：寄存器窗口基址计算错误

```cpp
// 错误实现：CALL 时 newBase 计算不对
int newBase = baseReg + 1;  // ❌ 忘记加 frame().base
// 正确：
int newBase = frame().base + baseReg + 1;  // ✓ 绝对地址
```

**症状：** 函数参数读到错误值，多层递归时数值越来越不对。

**诊断方法：** 在 `CALL` 和 `RET` 处添加 `std::cerr` 输出 `funcIdx`、`oldBase`、`newBase`、`callStack_.size()`，追踪每帧的寄存器窗口是否重叠。

### 错误 3：缺少 HALT 指令导致越界

```cpp
// 编译器 bug：main chunk 末尾没有 HALT
// VM 执行完最后一条有效指令后，fetch() 继续读取，ip 超出 code 范围
```

**症状：** 程序正常逻辑执行完后崩溃，`fetch()` 中数组越界。

**解决方案：** 编译器的 `compile()` 末尾必须追加 `emit({OpCode::HALT, 0, 0, 0})`。更健壮的做法是在 `fetch()` 中也加边界检查：

```cpp
Instruction fetch() {
    auto& f = frame();
    if (f.ip >= static_cast<int>(f.chunk->code.size()))
        throw std::runtime_error("IP 越界：可能缺少 HALT 指令");
    return f.chunk->code[f.ip++];
}
```

```cpp
void disassemble(const BytecodeChunk& chunk) {
    std::cout << "=== " << chunk.name << " (regs="
              << chunk.registerCount << " params="
              << chunk.paramCount << ") ===\n";
    for (size_t i = 0; i < chunk.code.size(); ++i) {
        auto& ins = chunk.code[i];
        std::cout << "  " << i << ": ";
        // 按 OpCode 格式化：CONST r{a}, #{b}; ADD r{a}, r{b}, r{c} 等
        // 完整 switch-case 覆盖所有 18 条指令（此处略去重复分支）
        switch (ins.op) {
            case OpCode::CONST: std::cout << "CONST r" << (int)ins.a
                << ", #" << (int)ins.b; break;
            case OpCode::ADD:   std::cout << "ADD   r" << (int)ins.a
                << ", r" << (int)ins.b << ", r" << (int)ins.c; break;
            case OpCode::CALL:  std::cout << "CALL  " << (int)ins.a
                << ", r" << (int)ins.b << ", " << (int)ins.c; break;
            case OpCode::RET:   std::cout << "RET   r" << (int)ins.a
                << ", " << (int)ins.b; break;
            case OpCode::HALT:  std::cout << "HALT"; break;
            default: std::cout << "OP" << (int)ins.op << " "
                << (int)ins.a << " " << (int)ins.b << " " << (int)ins.c;
        }
        std::cout << "\n";
    }
}
// 在 run() 中执行前插入：
for (auto& chunk : chunks) disassemble(chunk);
```

## 对照项目源码

| `VM::execute()` | `cpp/core/tjs2/tjsInterCodeExec.cpp` | ~3109 | 巨型 switch-case 解释循环 |
| `BytecodeChunk` | `tTJSInterCodeContext` | — | TJS2 的编译单元，含 CodeArea 和 DataArea |
| `CallFrame` | 执行栈帧 | — | TJS2 用 C++ 调用栈而非显式帧栈 |

**值得深入阅读的关键代码段：**

- **`tjsInterCodeExec.cpp` 第 200-400 行** — VM 主循环入口和指令分派宏定义
- **`tjsInterCodeGen.h` 第 100-200 行** — `tTJSInterCodeContext` 类定义，对照 `BytecodeChunk`
- **`tjsLex.cpp` 第 600-800 行** — 数值和字符串字面量解析，对照 `Lexer::scanToken()`

## 本节小结

- **端到端管线**：Lexer → Parser → Compiler → VM 四步将源代码变为可执行程序，与 TJS2、Lua、Python 架构一致
- **寄存器式 VM**：`CallFrame` + 寄存器窗口实现函数调用，每帧通过 `base` 偏移映射到全局数组
- **递归执行**：`CALL` 压帧、`RET` 弹帧、`resultReg` 传回返回值。Fibonacci 和 factorial 验证了递归正确性
- **作用域隔离**：编译器 `ScopeChain` 在编译期完成变量绑定，VM 执行时无需名称查找
- **TJS2 对照**：MiniScript 18 条指令是 TJS2 ~90 条指令的核心子集
- **调试技巧**：反汇编器、`CALL`/`RET` 日志、边界检查——三招定位大多数 VM bug

### 题目 2：追踪 VM 执行状态

### 题目 3：扩展 VM 支持新指令

为 MiniScript VM 添加 `GT`（greater than）和 `GE`（greater than or equal）两条比较指令。写出：(a) OpCode 枚举扩展、(b) VM switch-case 分支、(c) 一个使用新指令的测试程序。

<details>
<summary>查看答案</summary>

**(a) OpCode 枚举扩展：**

```cpp
enum class OpCode : uint8_t {
    CONST, MOV, ADD, SUB, MUL, DIV, MOD, NEG,
    EQ, LT, LE, GT, GE, NOT,  // 新增 GT, GE
    JMP, JF, CALL, RET, PRINT, HALT
};
```

**(b) VM 分支：**

```cpp
case OpCode::GT:
    reg(instr.a) = (reg(instr.b) > reg(instr.c)) ? 1.0 : 0.0;
    break;
case OpCode::GE:
    reg(instr.a) = (reg(instr.b) >= reg(instr.c)) ? 1.0 : 0.0;
    break;
```

**(c) 测试程序：**

```cpp
int main() {
    std::string source = R"(
        fn max(a, b) {
            if (a > b) { return a; }
            return b;
        }
        print max(7, 3);
        print max(2, 9);
    )";
    run(source);
    // 输出:
    // 7
    // 9
    return 0;
}
```

注意：添加 `GT`/`GE` 后还需更新编译器 `visitBinaryExpr`，让 `>`/`>=` 生成新指令而非用 `LT`/`LE` + 操作数交换模拟。

</details>
