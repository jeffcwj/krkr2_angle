## 逆向工程的定义

### 正向工程 vs 逆向工程

**正向工程（Forward Engineering）** 是开发者日常工作的过程：

```
需求分析 → 系统设计 → 编码实现 → 编译链接 → 可执行文件
  (人类)     (人类)     (人类)     (编译器)    (机器码)
```

**逆向工程（Reverse Engineering）** 则是这个过程的反向——从最终产物（二进制可执行文件）出发，推导出设计和实现：

```
可执行文件 → 反汇编/反编译 → 算法理解 → 行为还原 → 源码重建
  (机器码)     (工具辅助)     (人类分析)   (人类)     (人类)
```

更精确的定义：**软件逆向工程是在不获取源代码的前提下，通过分析软件的二进制表示（机器码、字节码、中间代码等），理解其内部结构、功能、行为和设计决策的过程。**

### 逆向 ≠ 破解

一个常见的误解是把逆向工程等同于软件破解（Cracking）。两者有本质区别：

| 维度 | 逆向工程 | 软件破解 |
|------|---------|---------|
| 目的 | 理解软件工作原理 | 绕过版权保护/授权 |
| 方法 | 分析、记录、重建 | 修改、绕过、注入 |
| 合法性 | 多数场景合法 | 大多数场景违法 |
| 产出 | 技术文档、兼容实现 | 破解补丁、密钥生成器 |
| 示例 | Wine（Linux运行Windows程序）| 游戏破解 |

KrKr2 项目中的逆向工程属于**互操作性逆向**（Interoperability Reverse Engineering）——为了让开源模拟器能运行原版游戏，需要理解原版引擎的插件接口和数据格式。这在多数法律体系下是受保护的合法行为。

### 信息损失与重建

P08 中我们学过编译流水线：源代码经过词法分析、语法分析、语义分析、代码生成后变成机器码。这个过程中存在**不可逆的信息损失**：

```
源码中存在但二进制中丢失的信息：
├── 变量名、函数名 → 变成地址偏移（除非保留调试符号）
├── 注释          → 完全消失
├── 类型信息      → 部分丢失（结构体变成偏移计算）
├── 高级控制结构  → 变成 jmp/jcc 指令（if/while 消失）
├── 宏定义        → 预处理时展开，无法还原
└── 编码风格      → 优化器重排指令，原始风格消失
```

这意味着逆向工程**不可能 100% 还原原始源码**。逆向的目标是重建**功能等价**（Functionally Equivalent）的代码——行为一致即可，不要求源码完全相同。

### 保留的信息

虽然很多信息丢失了，但二进制中仍然保留了大量可分析的内容：

```
二进制中仍然存在的信息：
├── 函数边界    → 通过序言/尾声模式识别（push rbp; mov rbp, rsp）
├── 导出符号    → DLL 导出表中的函数名（有时是 C++ 修饰名）
├── 字符串常量  → 硬编码的错误信息、日志文本、格式化字符串
├── 调用关系    → CALL 指令建立的函数调用图
├── 数据结构    → 通过访问模式推断（偏移 +0x10 → 结构体第三个成员）
├── 控制流      → 基本块和跳转关系（可重建 CFG）
└── 导入函数    → 调用了哪些系统 API 和外部库函数
```

## 应用场景

### 场景 1：兼容性与互操作性

**典型案例：** Wine 项目（Linux 上运行 Windows 程序）

Wine 不依赖微软源码，而是通过逆向分析 Windows API 的行为来创建兼容的开源实现。KrKr2 项目中也有类似的需求——逆向 KiriKiri 原版引擎的插件，创建功能兼容的开源实现。

```cpp
// KiriKiri 原版插件的导出接口（逆向分析得到）
// 每个 KiriKiri 插件 DLL 必须导出这两个函数：
extern "C" __declspec(dllexport)
HRESULT _stdcall V2Link(iTVPFunctionExporter* exporter) {
    // 插件初始化：注册 TJS2 类和方法
    TVPInitImportStub(exporter);
    // ... 注册插件功能 ...
    return S_OK;
}

extern "C" __declspec(dllexport)
HRESULT _stdcall V2Unlink() {
    // 插件清理
    return S_OK;
}
```

KrKr2 项目中 `doc/krkr2_plugins.md` 记录了需要逆向的插件列表：

| 插件名 | 源码状态 | 逆向优先级 | 功能 |
|--------|---------|-----------|------|
| PackinOne | 无源码 | 高 | 多文件打包压缩 |
| TextRender | 无源码 | 高 | 高级文字渲染 |
| layerExAlpha | 无源码 | 中 | 图层 Alpha 扩展操作 |
| lzfs | 无源码 | 中 | LZ 压缩文件系统 |

### 场景 2：安全审计与漏洞分析

安全研究人员通过逆向分析发现软件漏洞：

```
目标二进制
  │
  ▼ 反编译
伪代码重建
  │
  ▼ 人工审计
发现缓冲区溢出 → strcpy(局部缓冲区, 用户输入)  ← 无长度检查
  │
  ▼ PoC 构造
验证漏洞可利用性
  │
  ▼ 报告
提交 CVE 漏洞报告 → 厂商修复
```

### 场景 3：恶意软件分析

反病毒厂商逆向分析恶意软件以提取特征码：

- **静态分析：** 提取字符串、导入函数、代码模式
- **动态分析：** 在沙箱中运行，监控文件/注册表/网络行为
- **产出：** 检测签名、行为特征、C&C 通信协议

### 场景 4：遗留系统维护

原始开发者已离职或公司已倒闭，源码丢失，但系统仍在运行。逆向是恢复系统可维护性的唯一途径。KiriKiri/KiriKiriZ 引擎的部分组件就属于这种情况。

### 场景 5：竞品分析与学习

分析优秀软件的实现技术：

- 了解游戏引擎的渲染技巧
- 研究高性能网络库的实现细节
- 学习算法的工业级优化手法

## 法律与伦理边界

### 各法域的法律框架

| 法域 | 关键法律 | 逆向工程条款 |
|------|---------|-------------|
| 美国 | DMCA (数字千年版权法) | §1201(f) 允许为互操作性目的的逆向工程 |
| 欧盟 | 计算机程序指令 91/250/EEC | 第 6 条允许为互操作性目的的反编译 |
| 日本 | 著作権法 | 第 30 条之四允许为分析目的使用著作物 |
| 中国 | 计算机软件保护条例 | 第 16 条允许为学习研究目的的反编译 |

**共同原则：** 大多数法域允许为以下目的进行逆向工程：

1. **互操作性**（Interoperability）— 让自己的软件与他人的软件协同工作
2. **安全研究**（Security Research）— 发现并报告安全漏洞
3. **学习与教育**（Education）— 理解软件工作原理

**KrKr2 项目的法律定位：** 为开源游戏引擎模拟器实现插件兼容性，属于典型的**互操作性逆向**，在主要法域中均受法律保护。

### 伦理准则

即使法律允许，逆向工程也应遵循伦理准则：

```
✅ 应该做的：
├── 明确目的是互操作性或安全研究
├── 只分析实现插件兼容性所必需的部分
├── 公开发布分析结果，让社区受益
├── 尊重原始开发者的劳动成果
└── 如果发现安全漏洞，负责任地报告

❌ 不应该做的：
├── 逆向付费软件以避免购买
├── 复制竞品的全部代码并声称为自己的作品
├── 绕过版权保护或 DRM
├── 分发未授权的修改版本
└── 忽视终端用户许可协议（EULA）中的合理限制
```

## 逆向工程的心理模型

### "侦探工作"类比

逆向工程就像犯罪现场调查：

```
犯罪现场（二进制文件）
  │
  ├── 收集证据（字符串、导出表、导入表）
  ├── 分析线索（交叉引用、调用图、数据流）
  ├── 提出假设（这个函数可能是做 XXX 的）
  ├── 验证假设（动态调试、输入输出对比）
  └── 重建事件（写出等价的源代码）
```

### 逆向分析的层次模型

```
层次 5: 系统行为     "这个程序做什么"    ← 最高层，最先了解
层次 4: 组件交互     "模块间如何通信"
层次 3: 算法逻辑     "每个函数的算法"    ← 核心工作量
层次 2: 指令序列     "每条指令做什么"
层次 1: 二进制编码   "0x55 0x48 0x89 ..."  ← 最底层，工具处理
```

逆向分析通常**自顶向下**：先从导出表和字符串获得高层线索，再逐步深入关键函数的具体实现。工具（IDA/Ghidra）负责层次 1-2 的自动翻译，人类主要工作在层次 3-5。

### 从 P08 到 P09 的过渡

在 P08 中你学习了正向编译流程：

```
源码 ──→ Lexer ──→ Parser ──→ Compiler ──→ VM/机器码
         │          │           │             │
         Token      AST      字节码/IR      执行
```

P09 就是这个过程的反向——但不是简单的逆转。由于信息损失，逆向过程需要**推理和猜测**：

```
二进制 ──→ 反汇编器 ──→ 反编译器 ──→ 人工分析 ──→ 重建源码
  │           │            │            │            │
机器码     汇编文本    伪C代码     算法理解      可编译代码
(确定)    (确定)     (近似)     (需推理)     (功能等价)
```

## 动手实践

1. **查看自己的程序信息损失**：编写一个简单的 C++ 程序（包含函数名、注释、宏定义），用 `g++ -O2 -o test test.cpp` 编译为二进制，再用 `objdump -d test`（Linux）或 `dumpbin /disasm test.exe`（Windows）查看反汇编输出。对比源码和反汇编结果，列出丢失了哪些信息。

2. **比较调试版和发布版**：将同一段代码分别用 `g++ -g -O0` 和 `g++ -O2 -s` 编译，用 `file` 命令查看两个二进制文件的元数据差异，用 `nm` 命令对比符号表的有无。记录调试信息对逆向分析的帮助程度。

3. **搜索 KrKr2 插件清单**：打开 `doc/krkr2_plugins.md`，找出所有标记为 `nan`（无源码）的插件。选一个插件，搜索网上是否有该插件的公开文档或接口说明，记录你的发现。

4. **法律条款速查**：查阅你所在地区关于逆向工程的法律条款（如中国《著作权法》第 24 条、美国 DMCA §1201(f)、欧盟软件指令第 6 条），用一段话总结"为互操作性目的的逆向"在该法域下的合法性边界。

## 常见错误与解决方案

### 错误 1：试图 100% 还原源码

初学者期望逆向结果与原始源码一模一样。实际上由于信息损失，变量名、注释、宏定义等无法恢复，目标应当是**功能等价**的重建。

### 错误 2：忽略法律边界

在逆向分析前没有确认行为的合法性。不同国家的法律保护范围不同，且"合法逆向"与"非法使用结果"是两个独立判断。务必先查阅相关法律条款。

### 错误 3：跳过信息收集直接反汇编

拿到二进制文件后立刻打开 IDA/Ghidra 看汇编代码，而不先用 `file`、`strings`、`dumpbin` 等工具收集高层信息。正确流程是**自顶向下**：先了解文件类型、导出函数、字符串线索，再深入代码细节。

## 本节小结

- **逆向工程**是从二进制可执行文件推导出软件设计和实现的过程，是正向编译的反向操作
- **信息损失**使得逆向不可能 100% 还原源码，目标是**功能等价**的重建
- 逆向的主要应用场景包括互操作性、安全审计、恶意软件分析、遗留系统维护
- KrKr2 项目的逆向需求属于**互操作性逆向**，在主要法域中受法律保护
- 逆向分析是**自顶向下**的过程：从系统行为到组件交互再到算法细节
- 工具处理底层翻译（机器码→汇编→伪代码），人类负责高层理解和源码重建

## 练习题与答案

### 题目 1：信息损失分析

给定以下 C++ 源码，列出编译后的二进制文件中哪些信息会丢失、哪些会保留：

```cpp
// utils.cpp — 计算文件 CRC32 校验和
#include <cstdint>
#define POLY 0xEDB88320  // CRC-32 多项式

uint32_t crc32(const uint8_t* data, size_t length) {
    uint32_t crc = 0xFFFFFFFF;
    for (size_t i = 0; i < length; ++i) {
        crc ^= data[i];
        for (int j = 0; j < 8; ++j) {
            crc = (crc >> 1) ^ (POLY & -(crc & 1));
        }
    }
    return ~crc;  // 取反
}
```

<details>
<summary>查看答案</summary>

**丢失的信息：**
- 变量名 `crc`、`data`、`length`、`i`、`j` → 变成寄存器（rax, rcx, rdx 等）
- 函数名 `crc32` → 如果不导出则变成地址偏移（如果导出则保留在导出表中）
- 注释 `// 计算文件 CRC32 校验和`、`// CRC-32 多项式`、`// 取反` → 完全消失
- 宏 `POLY` → 预处理展开为 `0xEDB88320`，宏名消失
- 文件名 `utils.cpp` → 除非保留调试信息，否则消失
- `for` 循环结构 → 变成 `cmp`/`jl`/`jmp` 指令序列
- 类型 `uint32_t`、`uint8_t`、`size_t` → 变成操作宽度（32位操作 vs 8位操作）

**保留的信息：**
- 常量 `0xFFFFFFFF` 和 `0xEDB88320` → 出现在指令操作数中
- 函数的输入输出行为 → 给定相同输入，输出相同结果
- 循环计数 8 → `cmp` 指令的立即数
- XOR、右移、AND 等操作 → 对应的汇编指令
- 取反操作 `~crc` → NOT 指令
- 调用约定 → 参数通过寄存器/栈传递的模式

**逆向可重建的部分：** 通过识别 `0xEDB88320` 常量和 XOR-shift 模式，有经验的逆向工程师能直接认出这是 CRC-32 算法，甚至不需要逐行分析。

</details>

### 题目 2：逆向合法性判断

判断以下场景中的逆向行为是否合法，并说明理由：

1. 小明买了一款 Windows 游戏，逆向其 DRM 保护以在 Linux 上运行
2. 安全公司分析一个可疑的 .exe 文件以确认是否为恶意软件
3. KrKr2 开发者逆向 KiriKiri 的 TextRender.dll 以实现兼容的开源版本
4. 某公司逆向竞品的核心算法并直接复制到自己的产品中

<details>
<summary>查看答案</summary>

1. **灰色地带**：为互操作性（在 Linux 上运行）目的的逆向通常受保护，但"绕过 DRM"在 DMCA 下可能违法。关键在于是否仅为互操作性目的且未分发破解工具。DMCA §1201(f) 可能提供豁免，但需要具体分析。

2. **合法**：安全研究和恶意软件分析在几乎所有法域中都被视为合法行为。多数反病毒公司的日常工作就是逆向分析恶意软件。

3. **合法**：这是典型的互操作性逆向工程。为了让开源引擎兼容原版插件而分析插件接口，符合 DMCA §1201(f)、欧盟指令第 6 条等条款的保护范围。

4. **违法**：这不是互操作性逆向，而是商业间谍行为。直接复制竞品算法违反著作权法和可能的商业秘密法。即使逆向本身合法，**复制结果**用于竞争可能违法。

</details>

### 题目 3：描述逆向分析路径

假设你拿到一个未知的 KiriKiri 插件 DLL（`mystery.dll`，约 50KB），请设计一个从零开始的分析计划，列出你会使用哪些工具、采取什么步骤、期望获得什么信息。

<details>
<summary>查看答案</summary>

**分析计划：**

```
第 1 步：基础信息收集
  工具：file、dumpbin /headers（或 PE-bear）
  获取：文件类型、目标架构（x86/x64）、编译时间戳、节表信息
  期望：确认这是一个 32 位 Windows DLL

第 2 步：导出表分析
  工具：dumpbin /exports mystery.dll
  获取：导出函数列表（预期找到 V2Link、V2Unlink）
  期望：确认这是 KiriKiri 插件（V2Link/V2Unlink 签名）

第 3 步：字符串提取
  工具：strings mystery.dll 或 IDA 的字符串视图
  获取：硬编码的字符串（错误信息、类名、方法名）
  期望：从字符串推断插件功能（如"TextRender"、"AlphaBlend"）

第 4 步：静态分析
  工具：IDA Free 或 Ghidra
  步骤：
    (a) 加载 DLL，定位 V2Link 函数
    (b) 分析 V2Link 中的 TJS2 类注册调用
    (c) 追踪注册的方法，反编译每个方法
    (d) 识别算法模式、数据结构、外部依赖
  期望：得到插件注册了哪些 TJS2 类和方法，每个方法的伪代码
```
