## 2. 反汇编器（Disassembler）

反汇编器将机器码翻译回汇编指令，是逆向分析的基础工具。

### 2.1 IDA Pro / IDA Free

```
IDA 产品线：

└─────────────┴────────────────┴──────────────────────────┘
```

**核心特性：**

```c
// IDA 的自动分析能力示例
// 原始机器码：
// 55 8B EC 83 EC 10 C7 45 F0 01 00 00 00

// IDA 自动识别出函数序言：
.text:00401000  push    ebp
.text:00401001  mov     ebp, esp
.text:00401003  sub     esp, 10h
.text:00401006  mov     [ebp+var_10], 1

// IDA 自动完成的分析：
// 1. 识别函数边界（push ebp 序言）
// 2. 创建栈帧变量 var_10
// 3. 标注立即数类型
// 4. 建立交叉引用（xref）
```

**IDAPython 脚本示例：**

```python
# 遍历当前函数的所有基本块
import idautils
import idc

func_ea = idc.here()  # 当前光标地址
func = idaapi.get_func(func_ea)

if func:
    # 获取函数流程图
    flowchart = idaapi.FlowChart(func)
    for block in flowchart:
        print(f"基本块: {hex(block.start_ea)} - {hex(block.end_ea)}")
        # 遍历块内的每条指令
        ea = block.start_ea
        while ea < block.end_ea:
            disasm = idc.GetDisasm(ea)
            print(f"  {hex(ea)}: {disasm}")
            ea = idc.next_head(ea, block.end_ea)
```

## 3. 反编译器（Decompiler）

反编译器将汇编代码还原为类 C 伪代码，大幅降低分析难度。

### 3.1 反编译原理

```
机器码 → 汇编 → 中间表示(IR) → 优化 → 伪C代码

具体流程：
┌────────┐    ┌────────┐    ┌─────────┐    ┌────────┐
│ 指令解码 │───→│ 提升IR  │───→│ 数据流   │───→│ 类型推断│
│ Decode  │    │ Lift   │    │ 分析     │    │ Typing │
└────────┘    └────────┘    └─────────┘    └────────┘
                                                │
┌────────┐    ┌────────┐    ┌─────────┐         │
│ 输出C码 │←───│ 控制流  │←───│ 变量恢复 │←────────┘
│ Emit   │    │ 重建    │    │ Recovery│
└────────┘    └────────┘    └─────────┘
```

### 3.2 各反编译器对比

```c
// 同一函数在不同反编译器的输出对比

// === Hex-Rays (IDA Pro) ===
// 最接近原始代码，变量名推断准确
int __fastcall calculate_hash(const char *str, int seed)
{
    int hash = seed;
    while (*str) {
        hash = hash * 31 + *str;
        ++str;
    }
    return hash;
}

// === Ghidra ===
// 功能正确，但变量命名较生硬
int calculate_hash(char *param_1, int param_2)
{
    int iVar1 = param_2;
    while (*param_1 != '\0') {
        iVar1 = iVar1 * 0x1f + (int)*param_1;
        param_1 = param_1 + 1;
    }
    return iVar1;
}

// === RetDec (开源) ===
// 可用但输出较冗长
int32_t calculate_hash(char * str, int32_t seed) {
    int32_t v1 = seed;  // ebx
    char * v2 = str;    // esi
    if (*str != 0) {
        do {
            v1 = v1 * 31 + (int32_t)*v2;
            v2++;
        } while (*v2 != 0);
    }
    return v1;
}
```

### 3.3 反编译器的局限性

```c
// 反编译器无法完美还原的情况：

// 1. 虚函数调用 —— 无法静态确定目标
// 反编译输出：
(**(code **)(*(int *)obj + 0x10))(obj, param);
// 实际源码：
obj->OnEvent(param);  // 虚函数调用

// 2. 内联函数 —— 边界消失
// 反编译输出：一大段平铺代码
// 实际源码：多个小函数调用

// 3. 优化后的循环
// 反编译输出：
for (i = 0; i < n; i += 4) {
    // 展开的4次迭代...
}
// 实际源码：
for (i = 0; i < n; i++) {
    // 简单循环体
}

// 4. 模板实例化 —— 类型信息丢失
// 反编译输出：多个相似函数
void func_int(int* p, int n) { ... }
void func_float(float* p, int n) { ... }
// 实际源码：
template<typename T> void func(T* p, int n) { ... }
```

## 動手実践

用 `dumpbin` 分析一个 KiriKiri2 插件 DLL 的导出信息：

```batch
:: 步骤1: 打开 Visual Studio 开发者命令提示符
:: 或设置 PATH 包含 MSVC 工具目录

:: 步骤2: 查看 wuvorbis.dll 导出表
dumpbin /exports wuvorbis.dll

:: 预期输出类似：
:: ordinal hint RVA      name
::       1    0 00001000 V2Link
::       2    1 00001050 V2Unlink

:: 步骤3: 查看导入依赖
dumpbin /dependents wuvorbis.dll

:: 步骤4: 用 Python 脚本批量分析所有插件
```

```python
import subprocess
import os

plugin_dir = r"C:\game\plugin"
for fname in os.listdir(plugin_dir):
    if fname.lower().endswith(".dll"):
        path = os.path.join(plugin_dir, fname)
        result = subprocess.run(
            ["dumpbin", "/exports", path],
            capture_output=True, text=True
        )
        # 检查是否有 V2Link（KiriKiri2 插件标志）
        if "V2Link" in result.stdout:
            print(f"[KR2插件] {fname}")
            # 提取导出函数名
            for line in result.stdout.splitlines():
                line = line.strip()
                if line and not line.startswith(("Microsoft", "Dump", "ordinal",
                                                  "Section", "File")):
                    if any(c.isalpha() for c in line):
                        print(f"  {line}")
        else:
            print(f"[非KR2]   {fname}")
```

KiriKiri2 项目中，插件加载逻辑位于以下文件：

逆向分析 DLL 时，`tp_stub.h` 定义的接口是关键参照：

```cpp
// tp_stub.h 中的插件入口签名
extern "C" __declspec(dllexport)
HRESULT _stdcall V2Link(iTJSDispatch2 *tjs);

extern "C" __declspec(dllexport)
HRESULT _stdcall V2Unlink();

// 逆向时在 DLL 导出表中寻找这两个函数
// 就能确认该 DLL 是 KiriKiri2 插件
```

## 常见错误与解决

### 错误1：dumpbin 提示"不是内部或外部命令"

```
原因：未在 VS 开发者命令提示符中运行
解决：
  方法A：从开始菜单打开 "Developer Command Prompt for VS 2022"
  方法B：手动设置环境
    call "C:\Program Files\Microsoft Visual Studio\2022\Community\VC\Auxiliary\Build\vcvars64.bat"
```

### 错误2：Ghidra 打开大文件卡死

```
原因：默认 JVM 内存不足
解决：编辑 ghidraRun.bat，增加内存：
  修改 MAXMEM 参数：
    set MAXMEM=4G   (默认可能是 1G)
  或编辑 support/launch.properties：
    MAXMEM=4G
```

### 错误3：x64dbg 无法附加进程

```
原因：目标进程有反调试保护，或权限不足
解决：
  1. 以管理员身份运行 x64dbg
  2. 使用 ScyllaHide 插件绕过反调试
  3. 改用 WinDbg 内核调试（绕过用户态反调试）
```

## 本節小結

| 工具类别 | 代表工具 | 核心用途 | 推荐场景 |
|---------|---------|---------|---------|
| 反汇编器 | IDA / Ghidra / r2 | 机器码→汇编 | 所有静态分析 |
| 反编译器 | Hex-Rays / Ghidra | 汇编→伪C代码 | 理解函数逻辑 |
| 调试器 | x64dbg / WinDbg / GDB | 运行时分析 | 动态行为确认 |
| 辅助工具 | 010 Editor / ProcMon | 数据提取/监控 | 补充信息收集 |

关键认知：
- 没有万能工具，实际分析需要**多工具配合**
- 免费方案（Ghidra + x64dbg + ProcMon）已经能完成绝大多数任务
- 工具只是手段，**分析思维**才是逆向的核心

### 练习1：为 KiriKiri2 插件逆向设计工具链

### 练习2：对比 Ghidra 和 IDA Free 的适用场景

**题目**：在以下三种场景中，你会优先选择 Ghidra 还是 IDA Free？说明理由。
- (A) 快速查看一个小型 DLL 的导出函数反汇编
- (B) 需要反编译一个复杂的 C++ 类层次结构
- (C) 需要编写脚本批量分析 100 个 DLL

<details>
<summary>参考答案</summary>

```
(A) 快速查看小型 DLL → IDA Free
  理由: IDA 启动快、自动分析速度快、界面直观
        对于简单查看反汇编，IDA Free 完全够用

(B) 反编译复杂 C++ → Ghidra
  理由: IDA Free 不含反编译器（Hex-Rays 需付费）
        Ghidra 内置免费反编译器，支持类型恢复
        Ghidra 的 "Retype Variable" 功能帮助还原 C++ 类

(C) 批量脚本分析 → Ghidra
  理由: Ghidra 支持 Headless 模式（无GUI命令行分析）
        ghidra_headless.bat 可以批量处理
        脚本用 Java 或 Python 编写
        IDA Free 不支持批处理模式

  Ghidra Headless 批量分析命令：
  analyzeHeadless /tmp/project MyProject \
    -import *.dll \
    -postScript ExportAnalysis.java \
    -deleteProject
```

</details>
