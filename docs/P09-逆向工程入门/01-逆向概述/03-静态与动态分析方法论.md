```python
# Python 脚本识别文件类型
def identify_binary(filepath):
    """识别二进制文件类型"""
    with open(filepath, 'rb') as f:
        magic = f.read(4)

    signatures = {
        b'MZ':       'PE (Windows EXE/DLL)',
        b'\x7fELF':  'ELF (Linux/Unix)',
        b'\xfe\xed\xfa\xce': 'Mach-O 32-bit (macOS)',
        b'\xfe\xed\xfa\xcf': 'Mach-O 64-bit (macOS)',
        b'\xca\xfe\xba\xbe': 'Java Class / Mach-O Fat',
        b'PK':       'ZIP/APK/JAR',
    }

    for sig, desc in signatures.items():
        if magic[:len(sig)] == sig:
            return desc
    return f'Unknown (magic: {magic.hex()})'

# 使用
print(identify_binary('krmovie.dll'))
# 输出: PE (Windows EXE/DLL)
```

### 2.4 步骤3：字符串提取

```python
# 提取二进制文件中的可读字符串
import re

def extract_strings(filepath, min_length=4):
    """提取 ASCII 和 Unicode 字符串"""
    with open(filepath, 'rb') as f:
        data = f.read()

    # ASCII 字符串
    ascii_pattern = rb'[\x20-\x7e]{%d,}' % min_length
    ascii_strings = [m.group().decode('ascii')
                     for m in re.finditer(ascii_pattern, data)]

    # Unicode (UTF-16LE) 字符串
    unicode_pattern = rb'(?:[\x20-\x7e]\x00){%d,}' % min_length
    unicode_strings = [m.group().decode('utf-16-le')
                       for m in re.finditer(unicode_pattern, data)]

    return ascii_strings, unicode_strings

ascii_strs, unicode_strs = extract_strings('plugin.dll')

# 过滤有意义的字符串
keywords = ['V2Link', 'TJS', 'Plugin', 'Error', 'class', 'http']
for s in ascii_strs + unicode_strs:
    if any(kw.lower() in s.lower() for kw in keywords):
        print(f"  [关键] {s}")
```

### 2.6 步骤5：逻辑还原

```c
// 反编译伪代码还原过程示例

// === 反编译器原始输出 ===
int __cdecl sub_10001200(int param_1, int param_2)
{
    int iVar1;
    int local_8;

    local_8 = 0;
    iVar1 = *(int *)(param_1 + 4);
    while (local_8 < iVar1) {
        if (*(int *)(*(int *)(param_1 + 8) + local_8 * 4) == param_2) {
            return local_8;
        }
        local_8 = local_8 + 1;
    }
    return -1;
}

// === 分析过程 ===
// 1. param_1+4 是某种 count/size → 重命名为 container
// 2. param_1+8 是数据指针 → 重命名为 elements
// 3. 循环遍历 + 比较 + 返回索引 → 这是 indexOf/Find 函数
// 4. param_2 是要查找的值 → 重命名为 target

// === 还原后的可读代码 ===
int Container_Find(Container *self, int target)
{
    for (int i = 0; i < self->count; i++) {
        if (self->elements[i] == target) {
            return i;
        }
    }
    return -1;  // 未找到
}
```

## 3. 动态分析详解

### 3.1 动态分析工作流程

```
环境准备 → 初始断点 → 执行跟踪 → 数据捕获 → 行为分析
   │          │          │          │          │
   ▼          ▼          ▼          ▼          ▼
  VM/沙箱   入口点     单步/条件   内存快照   调用序列
  快照      API 断点   跟踪日志    参数记录   状态机
```

### 3.2 环境准备

```
动态分析环境安全准则：
┌──────────────────────────────────────────────────────┐
│ ⚠ 永远在隔离环境中进行动态分析                          │
├──────────────────────────────────────────────────────┤
│ 1. 使用虚拟机（VMware/VirtualBox/Hyper-V）            │
│ 2. 禁用网络或使用 Host-Only 网络                       │
│ 3. 在分析前创建快照，分析后可还原                       │
│ 4. 不要在宿主机上直接运行可疑程序                       │
│ 5. 已知安全的目标（如开源项目）可放宽限制                │
└──────────────────────────────────────────────────────┘

KiriKiri2 分析特殊情况：
  KiriKiri2 是开源项目，插件通常安全
  但逆向未知来源的 DLL 时仍需要隔离环境
```

### 3.3 断点策略

```c
// x64dbg 断点类型与应用场景

// 1. 软件断点（INT3）—— 最常用
// 在指定地址插入 0xCC 指令
// 场景：已知函数入口、关键代码位置
bp 0x10001000    // 在 V2Link 入口设断点

// 2. 硬件断点 —— 不修改代码
// 使用 CPU 调试寄存器（DR0-DR3，最多4个）
// 场景：代码段有完整性校验（自校验）
bph 0x10001000   // 硬件执行断点

// 3. 内存断点 —— 监控内存访问
// 修改内存页属性触发异常
// 场景：追踪数据读写
bpm 0x10020000 r   // 读取时中断
bpm 0x10020000 w   // 写入时中断

// 4. 条件断点 —— 过滤无关触发
// 仅在条件满足时中断
// 场景：循环中只关注特定迭代
bp 0x10001050 condition: eax==0x42
```

### 3.4 执行跟踪

```
x64dbg 跟踪模式：

1. 手动单步（F7/F8）
   适用：短函数、关键路径
   缺点：大量循环时效率低

2. 运行到返回（Ctrl+F9）
   适用：跳过已分析的函数
   技巧：在调用目标函数前设断点

3. 条件跟踪（Trace Into/Over）
   x64dbg → 菜单 → Trace → Trace Into
   设置记录条件和停止条件：
     Record condition: 1       (记录所有)
     Stop condition:   eip==0x10001100  (到达目标时停止)
     Log text:         {p:eip} {i:eip}  (记录地址和指令)

4. 日志断点（Log Breakpoint）
   在断点属性中设置：
     Log: "V2Link called, param1={d:[esp+4]}"
     Break: Never  (不中断，只记录)
```

## 4. 混合分析策略

实际逆向中，纯静态或纯动态分析很少能独立完成任务。混合分析是主流方法。

### 4.1 混合分析工作流

```
┌─────────┐     ┌─────────┐     ┌─────────┐
│ 静态分析  │────→│ 假设建立  │────→│ 动态验证  │
│ (全局)   │     │         │     │ (局部)   │
└─────────┘     └─────────┘     └────┬────┘
     ▲                                │
     │          ┌─────────┐           │
     └──────────│ 修正理解  │←──────────┘
                │ 补充分析  │
                └─────────┘

循环直到完全理解目标代码
```

### 4.2 实例：分析 KiriKiri2 插件的初始化流程

```
Step 1: 静态分析（Ghidra）
═══════════════════════════
  → 打开 plugin.dll
  → 导航到 V2Link 函数
  → 反编译查看伪代码：

    int V2Link(int param_1) {
        FUN_10001100();          // 未知函数
        FUN_10001200(param_1);   // 使用了 TJS 参数
        return 0;
    }

  → 假设：FUN_10001100 是初始化，FUN_10001200 是注册

Step 2: 动态验证（x64dbg）
═══════════════════════════
  → 在 V2Link 设断点，运行 kirikiri2.exe
  → 单步进入 FUN_10001100：
    → 发现调用了 CoInitialize → COM 初始化
    → 假设修正：这是 COM 组件初始化

  → 单步进入 FUN_10001200：
    → 参数 param_1 确实是 iTJSDispatch2*
    → 内部调用了类似 CreateObject 的函数
    → 确认：注册 TJS2 类到引擎

Step 3: 回到静态（补充分析）
═══════════════════════════
  → 在 Ghidra 中重命名：
    FUN_10001100 → InitializeCOM
    FUN_10001200 → RegisterTJSClasses
  → 深入 RegisterTJSClasses，分析注册了哪些类
  → 发现内部创建了 3 个 TJS2 类

Step 4: 再次动态验证
═══════════════════════════
  → 在注册函数内设断点
  → 记录每次注册的类名字符串
  → 确认类名：VideoOverlay, WMVDecoder, ScreenTransition
```

## 5. 反分析技术与对抗

### 5.1 常见反分析手段

```c
// 1. 反调试检测
// Windows API 检测调试器
BOOL isDebugged = IsDebuggerPresent();

// PEB 标志检测
PPEB pPeb = (PPEB)__readfsdword(0x30);
if (pPeb->BeingDebugged) { /* 检测到调试器 */ }

// 时间检测（调试时执行变慢）
DWORD t1 = GetTickCount();
// ... 一些代码 ...
DWORD t2 = GetTickCount();
if (t2 - t1 > 1000) { /* 可能被调试 */ }

// 2. 代码混淆
// 控制流平坦化（将 if/while 转为 switch-case 状态机）
// 不透明谓词（永远为真/假但难以静态判断的条件）
// 虚拟化保护（将代码转为自定义字节码）

// 3. 反反汇编
// 垃圾字节插入（干扰线性反汇编）
__asm {
    jmp next
    db 0xE8       // 假的 CALL 指令前缀，干扰反汇编器
next:
    nop
}
```

### 5.2 对抗策略

```
反分析检测 → 对抗方法：
┌────────────────────┬─────────────────────────────┐
│ 检测手段             │ 对抗方法                      │
├────────────────────┼─────────────────────────────┤
│ IsDebuggerPresent  │ 在 API 入口 patch 返回 0      │
│ PEB.BeingDebugged  │ 修改 PEB 标志位               │
│ NtQueryInformation │ Hook 该 API 返回假值          │
│ 时间检测            │ Hook GetTickCount 返回固定差值 │
│ 硬件断点检测         │ 使用软件断点或 Hook 方式       │
│ CRC/校验和检测       │ 修复校验值或 patch 检测代码    │
├────────────────────┼─────────────────────────────┤
│ 通用方案             │ ScyllaHide (x64dbg 插件)     │
│                    │ 自动绕过常见反调试              │
└────────────────────┴─────────────────────────────┘
```

**注意**：KiriKiri2 及其插件通常不含反调试保护，但商业加密壳（如 Themida）可能被第三方添加。

## 動手実践

使用混合分析法分析一个简单的 C 程序：

```c
// target.c — 编译后用于练习逆向
// 编译: cl /Od /Zi target.c (MSVC, 禁优化)
// 或:   gcc -O0 -g -o target target.c (GCC)
#include <stdio.h>
#include <string.h>

int encode(const char *input, char *output) {
    int len = (int)strlen(input);
    for (int i = 0; i < len; i++) {
        output[i] = input[i] ^ 0x42;  // 简单 XOR 编码
    }
    output[len] = '\0';
    return len;
}

int verify(const char *encoded, const char *expected) {
    return strcmp(encoded, expected) == 0;
}

int main() {
    char buf[64];
    const char *secret = "Hello KiriKiri2";
    // 编码后的预期值（secret 每字节 XOR 0x42）
    const char expected[] = {
        0x0a, 0x27, 0x2e, 0x2e, 0x2d, 0x62, 0x09, 0x21,
        0x30, 0x21, 0x09, 0x21, 0x30, 0x21, 0x72, 0x00
    };

    encode(secret, buf);
    if (verify(buf, expected)) {
        printf("Verification passed!\n");
    }
    return 0;
}
```

**练习步骤：**

```
1. 编译上述程序（禁用优化）
2. 静态分析：用 Ghidra 打开编译后的 .exe
   → 找到 main 函数
   → 找到 encode 函数，分析 XOR 密钥
   → 找到 expected 数组的值

3. 动态验证：用 x64dbg 打开
   → 在 encode 函数设断点
   → 单步观察 XOR 过程
   → 在 verify 调用前查看 buf 内容

4. 解密：用 Python 还原明文
```

```python
# 从静态分析中提取的加密数据
expected = bytes([
    0x0a, 0x27, 0x2e, 0x2e, 0x2d, 0x62, 0x09, 0x21,
    0x30, 0x21, 0x09, 0x21, 0x30, 0x21, 0x72
])
key = 0x42  # 从反编译 encode 函数中获取

# 解密
plaintext = bytes(b ^ key for b in expected)
print(f"解密结果: {plaintext.decode('ascii')}")
# 输出: 解密结果: Hello KiriKiri2
```

## 常见错误与解决

### 错误1：静态分析中函数边界识别错误

```
症状：IDA/Ghidra 把两个函数合并成一个，或把一个函数拆成多个
原因：编译器优化（尾调用优化、函数内联、跳转表）
解决：
  1. 在 IDA 中：Edit → Functions → Create Function 手动定义
  2. 在 Ghidra 中：右键地址 → Create Function
  3. 参考调用约定（函数序言 push ebp; mov ebp, esp）手动判断
```

### 错误2：动态分析时断点不生效

```
症状：设置了断点但程序不中断
可能原因：
  1. DLL 尚未加载 → 使用 "DLL 加载时断点" (sxe ld:xxx.dll)
  2. 地址错误（ASLR 重定位）→ 使用模块名+偏移: bp plugin+0x1000
  3. 代码未被执行 → 确认触发条件（如：需要播放视频才触发 krmovie）
  4. 反调试干扰 → 安装 ScyllaHide 插件
```

### 错误3：反编译输出与源码差异巨大

```
症状：反编译的伪 C 代码难以理解，与预期源码差异大
原因：编译器优化重组了代码结构
解决：
  1. 使用 Debug 版（/Od 或 -O0）编译的目标练习
  2. 先掌握无优化版本的模式，再分析优化版本
  3. 在 Ghidra 中手动设置变量类型、重命名变量
  4. 参考 FLIRT 签名识别标准库函数（减少分析量）
```

## 本節小結

| 分析方法 | 核心思想 | 优势 | 局限 |
|---------|---------|------|-----|
| 静态分析 | 不执行，看代码 | 全局覆盖、安全 | 受混淆影响、缺运行时信息 |
| 动态分析 | 执行时观察 | 能看到真实数据 | 只覆盖执行路径、有风险 |
| 混合分析 | 两者结合迭代 | 互补验证 | 需要两类工具经验 |

关键原则：
- **静态优先**：先建立全局理解，再用动态验证
- **假设驱动**：每次分析都带着明确的问题/假设
- **记录一切**：分析过程文档化，方便回溯和协作
- **迭代深入**：不要试图一次看懂所有代码

## 練習題与答案

### 练习1：选择分析策略

**题目**：以下场景应使用哪种分析方法？说明理由。
- (A) 确定一个加密函数使用的算法
- (B) 找出程序运行时连接的服务器地址
- (C) 分析一个 DLL 的完整接口定义

<details>
<summary>参考答案</summary>

```
(A) 确定加密算法 → 静态分析为主
  理由：加密算法是确定性的数学运算
        静态分析可以看到完整代码逻辑
        可以搜索已知常量（如 AES 的 S-Box、MD5 的初始值）
        动态分析只能看到特定输入的运算过程

(B) 服务器地址 → 动态分析为主
  理由：地址可能是加密存储或运行时计算的
        直接在网络 API（connect/send）设断点最高效
        配合 Wireshark 抓包可以一步到位
        静态分析可能需要先逆向解密逻辑

(C) DLL 接口定义 → 静态分析为主
  理由：接口定义是结构性信息
        导出表 + 反编译可以获取完整接口
        交叉引用可以找到所有方法和参数
        动态分析只能看到被调用的接口子集
```

</details>

### 练习2：设计断点方案

**题目**：你需要分析一个 KiriKiri2 图像解码插件。该插件在打开 .tlg 图片时被调用。请设计 x64dbg 断点方案。

<details>
<summary>参考答案</summary>

```
Step 1: 初始断点
  bp LoadLibraryW          // 监控 DLL 加载
  → 等待目标插件 DLL 被加载

Step 2: 插件入口断点
  bp plugin!V2Link         // 捕获插件注册
  → 单步分析注册了哪些类

Step 3: 文件操作断点
  bp CreateFileW           // 条件: 文件名包含 .tlg
  → 找到图片打开的入口

Step 4: 解码函数断点（根据 Step 2 的分析结果）
  bp plugin+0xXXXX         // 在图片解码函数入口
  → 分析解码参数（宽度、高度、像素格式）

Step 5: 内存断点
  bpm [output_buffer] w    // 在输出缓冲区设写断点
  → 追踪解码后的像素数据写入

Step 6: 条件日志
  在关键函数设日志断点（不中断）：
  Log: "Decode({d:ecx}x{d:edx}, format={d:[esp+8]})"
  → 自动记录所有解码调用的参数
```

</details>

### 练习3：修复反编译错误

**题目**：以下 Ghidra 反编译输出有哪些可以改进的地方？

```c
void FUN_10002000(int param_1, undefined4 param_2, int param_3)
{
    if (param_3 != 0) {
        *(undefined4 *)(param_1 + 8) = param_2;
        *(int *)(param_1 + 0xc) = *(int *)(param_1 + 0xc) + 1;
        FUN_10002100(param_1);
    }
    return;
}
```

<details>
<summary>参考答案</summary>

```c
// 分析过程：
// 1. param_1 被当作结构体指针使用（多次偏移访问）
// 2. param_1+8 存储数据 → 可能是 value/data 字段
// 3. param_1+0xc 自增 → 是计数器（count/size）
// 4. param_3 作为布尔判断 → 可能是 enable/valid 标志
// 5. FUN_10002100 在添加数据后调用 → 可能是 notify/update

// 改进后：
typedef struct {
    void *vtable;    // +0x00
    int capacity;    // +0x04 (推测)
    int value;       // +0x08
    int count;       // +0x0C
} Container;

void Container_Add(Container *self, int value, BOOL valid)
{
    if (valid) {
        self->value = value;
        self->count++;
        Container_Notify(self);  // 通知更新
    }
}

// 改进要点：
// 1. 创建结构体类型（Ghidra: Data Type Manager → New Structure）
// 2. 重新类型化 param_1 为 Container*
// 3. 重命名函数和参数
// 4. undefined4 → int（根据使用方式推断）
```

</details>
