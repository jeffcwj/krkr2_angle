## 1. PE 格式总览

PE 格式是 Windows 平台上 .exe、.dll、.sys、.ocx 等可执行文件的标准格式。

```
PE 文件整体结构：
┌──────────────────────────────┐  偏移 0x0000
│       DOS Header (64字节)     │
│  e_magic = "MZ" (0x5A4D)    │
│  e_lfanew = PE头偏移          │
├──────────────────────────────┤
│       DOS Stub               │
│  (可选的DOS程序, 通常很短)    │
├──────────────────────────────┤  偏移 = e_lfanew
│       NT Headers             │
│  ├── Signature "PE\0\0"     │
│  ├── FILE_HEADER (20字节)    │
│  └── OPTIONAL_HEADER         │
│       ├── 标准字段            │
│       └── DataDirectory[16]  │
├──────────────────────────────┤
│     Section Headers (节表)    │
│  ├── .text   (代码段)        │
│  ├── .rdata  (只读数据)      │
│  ├── .data   (可读写数据)    │
│  ├── .rsrc   (资源)          │
│  └── .reloc  (重定位)        │
├──────────────────────────────┤
│       .text 节数据            │
│     (机器码指令)              │
├──────────────────────────────┤
│       .rdata 节数据           │
│  (导入表/导出表/字符串常量)   │
├──────────────────────────────┤
│       .data 节数据            │
│     (全局变量)                │
├──────────────────────────────┤
│       其他节数据...           │
└──────────────────────────────┘
```

### 文件偏移 vs 虚拟地址

```
PE 文件存在两套地址系统：

磁盘文件偏移 (File Offset / Raw Offset)
  → 文件中的字节位置
  → 节头中的 PointerToRawData 字段

内存虚拟地址 (Virtual Address / VA)
  → 加载到内存后的地址
  → VA = ImageBase + RVA
  → 节头中的 VirtualAddress 字段

相对虚拟地址 (Relative Virtual Address / RVA)
  → 相对于 ImageBase 的偏移
  → RVA = VA - ImageBase

转换公式：
  FileOffset = RVA - Section.VirtualAddress + Section.PointerToRawData

示例：
  ImageBase = 0x10000000
  .text: VirtualAddress=0x1000, PointerToRawData=0x400
  RVA = 0x1234
  VA  = 0x10000000 + 0x1234 = 0x10001234
  FileOffset = 0x1234 - 0x1000 + 0x400 = 0x634
```

## 2. DOS Header

DOS Header 是 PE 文件的入口，保留了 DOS 兼容性。

```c
// winnt.h 中的定义
typedef struct _IMAGE_DOS_HEADER {
    WORD   e_magic;      // 0x00: Magic number = 0x5A4D ("MZ")
    WORD   e_cblp;       // 0x02: 最后一页的字节数
    WORD   e_cp;         // 0x04: 页数
    WORD   e_crlc;       // 0x06: 重定位项数
    WORD   e_cparhdr;    // 0x08: 头部段落数
    WORD   e_minalloc;   // 0x0A: 最小额外段落
    WORD   e_maxalloc;   // 0x0C: 最大额外段落
    WORD   e_ss;         // 0x0E: 初始 SS
    WORD   e_sp;         // 0x10: 初始 SP
    WORD   e_csum;       // 0x12: 校验和
    WORD   e_ip;         // 0x14: 初始 IP
    WORD   e_cs;         // 0x16: 初始 CS
    WORD   e_lfarlc;     // 0x18: 重定位表偏移
    WORD   e_ovno;       // 0x1A: 覆盖号
    WORD   e_res[4];     // 0x1C: 保留
    WORD   e_oemid;      // 0x24: OEM 标识
    WORD   e_oeminfo;    // 0x26: OEM 信息
    WORD   e_res2[10];   // 0x28: 保留
    LONG   e_lfanew;     // 0x3C: ★ PE 头偏移（最重要的字段）
} IMAGE_DOS_HEADER;      // 共 64 字节
```

**关键字段：**

```
e_magic (偏移 0x00):
  必须是 0x5A4D（ASCII "MZ"，取自 Mark Zbikowski 的首字母）
  这是识别 PE 文件的第一个标志

e_lfanew (偏移 0x3C):
  指向 NT Headers 的文件偏移
  这是从 DOS Header 跳转到真正 PE 头的关键
  逆向分析时，读取此值即可定位 PE 签名
```

## 3. NT Headers

NT Headers 是 PE 格式的核心，包含文件头和可选头。

```c
typedef struct _IMAGE_NT_HEADERS {
    DWORD                   Signature;      // "PE\0\0" = 0x00004550
    IMAGE_FILE_HEADER       FileHeader;     // 20 字节
    IMAGE_OPTIONAL_HEADER32 OptionalHeader; // 32位: 224 字节
} IMAGE_NT_HEADERS32;
```

### 3.1 FILE_HEADER（文件头）

```c
typedef struct _IMAGE_FILE_HEADER {
    WORD  Machine;              // CPU 架构
    WORD  NumberOfSections;     // 节的数量
    DWORD TimeDateStamp;        // 编译时间戳（UTC Unix时间）
    DWORD PointerToSymbolTable; // 符号表偏移（通常为0）
    DWORD NumberOfSymbols;      // 符号数量（通常为0）
    WORD  SizeOfOptionalHeader; // OptionalHeader 大小
    WORD  Characteristics;      // 文件属性标志
} IMAGE_FILE_HEADER;

// Machine 常见值：
// 0x014C = IMAGE_FILE_MACHINE_I386   (x86 32位)
// 0x8664 = IMAGE_FILE_MACHINE_AMD64  (x64 64位)
// 0x01C4 = IMAGE_FILE_MACHINE_ARMNT  (ARM)

// Characteristics 常见标志：
// 0x0002 = IMAGE_FILE_EXECUTABLE_IMAGE  (可执行)
// 0x0020 = IMAGE_FILE_LARGE_ADDRESS_AWARE (支持>2GB地址)
// 0x0100 = IMAGE_FILE_32BIT_MACHINE     (32位)
// 0x2000 = IMAGE_FILE_DLL               (★ DLL文件标志)
```

**KiriKiri2 插件 DLL 的典型 FILE_HEADER：**

```
Machine:          0x014C (i386, 32位)
NumberOfSections: 5 (.text/.rdata/.data/.rsrc/.reloc)
TimeDateStamp:    0x5F3E2A00 (2020-08-20)
Characteristics:  0x2102 (DLL | EXECUTABLE | 32BIT)
                  ← 0x2000(DLL) | 0x0100(32BIT) | 0x0002(EXECUTABLE)
```

### 3.2 OPTIONAL_HEADER（可选头）

```c
typedef struct _IMAGE_OPTIONAL_HEADER {
    // === 标准字段 ===
    WORD    Magic;                  // 0x10B=PE32, 0x20B=PE32+
    BYTE    MajorLinkerVersion;
    BYTE    MinorLinkerVersion;
    DWORD   SizeOfCode;             // 代码段总大小
    DWORD   SizeOfInitializedData;
    DWORD   SizeOfUninitializedData;
    DWORD   AddressOfEntryPoint;    // ★ 入口点 RVA
    DWORD   BaseOfCode;             // 代码段起始 RVA

    // === PE32 特有 ===
    DWORD   BaseOfData;             // 数据段起始 RVA

    // === Windows 特有字段 ===
    DWORD   ImageBase;              // ★ 首选加载基址
    DWORD   SectionAlignment;       // 内存中的节对齐
    DWORD   FileAlignment;          // 文件中的节对齐
    WORD    MajorOperatingSystemVersion;
    WORD    MinorOperatingSystemVersion;
    WORD    MajorImageVersion;
    WORD    MinorImageVersion;
    WORD    MajorSubsystemVersion;
    WORD    MinorSubsystemVersion;
    DWORD   Win32VersionValue;
    DWORD   SizeOfImage;            // 映像总大小
    DWORD   SizeOfHeaders;          // 头部总大小
    DWORD   CheckSum;               // 校验和
    WORD    Subsystem;              // 子系统类型
    WORD    DllCharacteristics;     // DLL 属性（ASLR等）
    DWORD   SizeOfStackReserve;
    DWORD   SizeOfStackCommit;
    DWORD   SizeOfHeapReserve;
    DWORD   SizeOfHeapCommit;
    DWORD   LoaderFlags;
    DWORD   NumberOfRvaAndSizes;    // 数据目录项数

    // === 数据目录 ===
    IMAGE_DATA_DIRECTORY DataDirectory[16];
} IMAGE_OPTIONAL_HEADER32;
```

### 3.3 数据目录（DataDirectory）

```c
typedef struct _IMAGE_DATA_DIRECTORY {
    DWORD VirtualAddress;  // 数据的 RVA
    DWORD Size;            // 数据大小
} IMAGE_DATA_DIRECTORY;

// 16 个数据目录项：
// [0]  EXPORT        导出表        ★ DLL 分析关键
// [1]  IMPORT        导入表        ★ 依赖分析关键
// [2]  RESOURCE      资源表
// [3]  EXCEPTION     异常表
// [4]  SECURITY      安全目录（数字签名）
// [5]  BASERELOC     重定位表      ★ ASLR 相关
// [6]  DEBUG         调试信息（PDB路径）
// [7]  ARCHITECTURE  (保留)
// [8]  GLOBALPTR     全局指针
// [9]  TLS           线程局部存储
// [10] LOAD_CONFIG   加载配置
// [11] BOUND_IMPORT  绑定导入
// [12] IAT           导入地址表    ★ 运行时 API 调用
// [13] DELAY_IMPORT  延迟导入
// [14] COM_DESCRIPTOR .NET 元数据
// [15] (保留)
```

## 4. 节表（Section Headers）

```c
typedef struct _IMAGE_SECTION_HEADER {
    BYTE  Name[8];               // 节名（如 ".text\0\0\0"）
    union {
        DWORD PhysicalAddress;
        DWORD VirtualSize;       // 内存中的实际大小
    } Misc;
    DWORD VirtualAddress;        // 内存中的起始 RVA
    DWORD SizeOfRawData;         // 文件中的数据大小
    DWORD PointerToRawData;      // 文件中的数据偏移
    DWORD PointerToRelocations;  // (OBJ 文件用)
    DWORD PointerToLinenumbers;  // (已废弃)
    WORD  NumberOfRelocations;
    WORD  NumberOfLinenumbers;
    DWORD Characteristics;       // ★ 节属性标志
} IMAGE_SECTION_HEADER;

// Characteristics 常见标志：
// 0x00000020 = IMAGE_SCN_CNT_CODE            代码
// 0x00000040 = IMAGE_SCN_CNT_INITIALIZED_DATA 已初始化数据
// 0x20000000 = IMAGE_SCN_MEM_EXECUTE         可执行
// 0x40000000 = IMAGE_SCN_MEM_READ            可读
// 0x80000000 = IMAGE_SCN_MEM_WRITE           可写
//
// 组合示例：
// .text:  0x60000020 = CODE | EXECUTE | READ
// .rdata: 0x40000040 = INITIALIZED_DATA | READ
// .data:  0xC0000040 = INITIALIZED_DATA | READ | WRITE
```

**典型 PE 节布局：**

```
┌───────────┬───────────────┬──────────────────────────────────┐
│ 节名       │ 属性           │ 内容                              │
├───────────┼───────────────┼──────────────────────────────────┤
│ .text     │ 代码,可读,可执行│ 编译后的机器码指令                 │
│ .rdata    │ 数据,只读      │ 常量字符串、导入/导出表、vtable    │
│ .data     │ 数据,可读写    │ 全局变量、静态变量                 │
│ .bss      │ 数据,可读写    │ 未初始化的全局变量（文件中不占空间）│
│ .rsrc     │ 数据,只读      │ 资源（图标、对话框、版本信息）     │
│ .reloc    │ 数据,只读      │ 基址重定位表（ASLR 用）           │
│ .tls      │ 数据,可读写    │ 线程局部存储初始数据               │
│ .pdata    │ 数据,只读      │ 异常处理信息（x64）               │
└───────────┴───────────────┴──────────────────────────────────┘
```

## 5. 用 Python 解析 PE 文件

### 5.1 手动解析（理解原理）

```python
import struct

def parse_pe_manual(filepath):
    """手动解析 PE 文件核心结构"""
    with open(filepath, 'rb') as f:
        data = f.read()

    # === DOS Header ===
    e_magic = struct.unpack_from('<H', data, 0)[0]
    assert e_magic == 0x5A4D, f"不是 PE 文件: magic={hex(e_magic)}"
    e_lfanew = struct.unpack_from('<I', data, 0x3C)[0]
    print(f"DOS Header:")
    print(f"  e_magic:  {hex(e_magic)} ('MZ')")
    print(f"  e_lfanew: {hex(e_lfanew)}")

    # === NT Headers ===
    pe_sig = struct.unpack_from('<I', data, e_lfanew)[0]
    assert pe_sig == 0x00004550, f"无效 PE 签名: {hex(pe_sig)}"

    # FILE_HEADER (紧跟 PE 签名之后)
    fh_offset = e_lfanew + 4
    machine, num_sections, timestamp = struct.unpack_from(
        '<HHI', data, fh_offset
    )
    size_opt_hdr, characteristics = struct.unpack_from(
        '<HH', data, fh_offset + 16
    )

    machine_names = {0x014C: 'i386', 0x8664: 'AMD64'}
    print(f"\nFILE_HEADER:")
    print(f"  Machine:       {hex(machine)} ({machine_names.get(machine, '?')})")
    print(f"  Sections:      {num_sections}")
    print(f"  Timestamp:     {timestamp}")
    print(f"  Characteristics: {hex(characteristics)}")
    is_dll = bool(characteristics & 0x2000)
    print(f"  Is DLL:        {is_dll}")

    # OPTIONAL_HEADER
    oh_offset = fh_offset + 20
    magic = struct.unpack_from('<H', data, oh_offset)[0]
    is_pe32plus = (magic == 0x20B)
    print(f"\nOPTIONAL_HEADER:")
    print(f"  Magic:         {hex(magic)} ({'PE32+' if is_pe32plus else 'PE32'})")

    if not is_pe32plus:
        entry_rva = struct.unpack_from('<I', data, oh_offset + 16)[0]
        image_base = struct.unpack_from('<I', data, oh_offset + 28)[0]
        section_align = struct.unpack_from('<I', data, oh_offset + 32)[0]
        file_align = struct.unpack_from('<I', data, oh_offset + 36)[0]
    else:
        entry_rva = struct.unpack_from('<I', data, oh_offset + 16)[0]
        image_base = struct.unpack_from('<Q', data, oh_offset + 24)[0]
        section_align = struct.unpack_from('<I', data, oh_offset + 32)[0]
        file_align = struct.unpack_from('<I', data, oh_offset + 36)[0]

    print(f"  EntryPoint:    {hex(entry_rva)}")
    print(f"  ImageBase:     {hex(image_base)}")
    print(f"  SectionAlign:  {hex(section_align)}")
    print(f"  FileAlign:     {hex(file_align)}")

    # === Section Headers ===
    section_offset = oh_offset + size_opt_hdr
    print(f"\nSection Headers:")
    print(f"  {'Name':8s} {'VirtAddr':>10s} {'VirtSize':>10s} "
          f"{'RawAddr':>10s} {'RawSize':>10s} {'Flags':>10s}")
    print(f"  {'-'*58}")

    for i in range(num_sections):
        off = section_offset + i * 40
        name = data[off:off+8].rstrip(b'\x00').decode('ascii', errors='replace')
        vsize, vaddr, raw_size, raw_addr = struct.unpack_from(
            '<IIII', data, off + 8
        )
        flags = struct.unpack_from('<I', data, off + 36)[0]
        print(f"  {name:8s} {hex(vaddr):>10s} {hex(vsize):>10s} "
              f"{hex(raw_addr):>10s} {hex(raw_size):>10s} {hex(flags):>10s}")

# 使用示例
# parse_pe_manual('plugin.dll')
```

### 5.2 使用 pefile 库（实际工作推荐）

```python
# pip install pefile
import pefile
import time

def analyze_pe(filepath):
    """使用 pefile 库全面分析 PE 文件"""
    pe = pefile.PE(filepath)

    # 基本信息
    print(f"=== {filepath} ===")
    print(f"类型: {'DLL' if pe.is_dll() else 'EXE'}")
    print(f"架构: {'x64' if pe.FILE_HEADER.Machine == 0x8664 else 'x86'}")

    # 编译时间
    ts = pe.FILE_HEADER.TimeDateStamp
    print(f"编译时间: {time.ctime(ts)} (UTC)")

    # 入口点
    ep = pe.OPTIONAL_HEADER.AddressOfEntryPoint
    print(f"入口点 RVA: {hex(ep)}")

    # 导出函数（DLL）
    if hasattr(pe, 'DIRECTORY_ENTRY_EXPORT'):
        print(f"\n导出函数 ({pe.DIRECTORY_ENTRY_EXPORT.struct.NumberOfFunctions}):")
        for exp in pe.DIRECTORY_ENTRY_EXPORT.symbols:
            name = exp.name.decode() if exp.name else f"Ordinal_{exp.ordinal}"
            print(f"  [{exp.ordinal:3d}] {name:30s} RVA={hex(exp.address)}")

    # 导入依赖
    if hasattr(pe, 'DIRECTORY_ENTRY_IMPORT'):
        print(f"\n导入依赖:")
        for entry in pe.DIRECTORY_ENTRY_IMPORT:
            dll_name = entry.dll.decode()
            funcs = [imp.name.decode() if imp.name else f"Ord_{imp.ordinal}"
                     for imp in entry.imports]
            print(f"  {dll_name}: {len(funcs)} 个函数")
            for fn in funcs[:5]:  # 只显示前5个
                print(f"    - {fn}")
            if len(funcs) > 5:
                print(f"    ... 还有 {len(funcs)-5} 个")

    # 节信息
    print(f"\n节表:")
    for sec in pe.sections:
        name = sec.Name.decode('ascii', errors='replace').rstrip('\x00')
        entropy = sec.get_entropy()
        print(f"  {name:8s}  VA={hex(sec.VirtualAddress):>10s}  "
              f"Size={sec.Misc_VirtualSize:>8d}  "
              f"Entropy={entropy:.2f}  "
              f"{'⚠高熵' if entropy > 7.0 else ''}")

    pe.close()

# analyze_pe('wuvorbis.dll')
```

## 動手実践

编写一个 PE 文件快速检查工具：

```python
import struct
import sys
import os

def pe_quick_check(filepath):
    """PE 文件快速安全检查"""
    size = os.path.getsize(filepath)
    print(f"文件: {filepath}")
    print(f"大小: {size:,} bytes ({size/1024:.1f} KB)")

    with open(filepath, 'rb') as f:
        # 检查 MZ 签名
        magic = f.read(2)
        if magic != b'MZ':
            print("❌ 不是 PE 文件（缺少 MZ 签名）")
            return

        # 读取 e_lfanew
        f.seek(0x3C)
        e_lfanew = struct.unpack('<I', f.read(4))[0]
        if e_lfanew > size:
            print("❌ e_lfanew 超出文件大小，文件可能损坏")
            return

        # 检查 PE 签名
        f.seek(e_lfanew)
        pe_sig = f.read(4)
        if pe_sig != b'PE\x00\x00':
            print("❌ 无效 PE 签名")
            return

        # 读取 Machine 和 Characteristics
        machine = struct.unpack('<H', f.read(2))[0]
        num_sec = struct.unpack('<H', f.read(2))[0]
        timestamp = struct.unpack('<I', f.read(4))[0]

        f.seek(e_lfanew + 4 + 16)
        _, chars = struct.unpack('<HH', f.read(4))

        arch = {0x014C: 'x86', 0x8664: 'x64', 0x01C4: 'ARM'}.get(
            machine, f'Unknown({hex(machine)})')
        is_dll = bool(chars & 0x2000)

        print(f"\n✅ 有效 PE 文件")
        print(f"  架构:     {arch}")
        print(f"  类型:     {'DLL' if is_dll else 'EXE'}")
        print(f"  节数:     {num_sec}")
        print(f"  编译时间: {timestamp} (Unix)")

        # 安全检查
        print(f"\n安全标志:")
        f.seek(e_lfanew + 4 + 20)  # OPTIONAL_HEADER 起始
        opt_magic = struct.unpack('<H', f.read(2))[0]

        if opt_magic == 0x10B:  # PE32
            f.seek(e_lfanew + 4 + 20 + 46)  # DllCharacteristics
        else:  # PE32+
            f.seek(e_lfanew + 4 + 20 + 70)

        dll_chars = struct.unpack('<H', f.read(2))[0]
        aslr = bool(dll_chars & 0x0040)
        dep = bool(dll_chars & 0x0100)
        seh = not bool(dll_chars & 0x0400)
        cfg = bool(dll_chars & 0x4000)
        print(f"  ASLR:     {'✅ 启用' if aslr else '❌ 未启用'}")
        print(f"  DEP/NX:   {'✅ 启用' if dep else '❌ 未启用'}")
        print(f"  SEH保护:  {'✅ 启用' if seh else '❌ 未启用'}")
        print(f"  CFG:      {'✅ 启用' if cfg else '❌ 未启用'}")

        # KiriKiri2 插件检查
        if is_dll:
            f.seek(0)
            content = f.read()
            has_v2link = b'V2Link' in content
            has_v2unlink = b'V2Unlink' in content
            if has_v2link and has_v2unlink:
                print(f"\n🎮 检测到 KiriKiri2 插件标志 (V2Link + V2Unlink)")

if __name__ == '__main__':
    if len(sys.argv) < 2:
        print("用法: python pe_check.py <文件路径>")
    else:
        pe_quick_check(sys.argv[1])
```

```
编译后的 KiriKiri2 PE 特征：
  Machine:    0x014C (x86 32-bit)
  ImageBase:  0x00400000 (EXE) 或 0x10000000 (DLL)
  Subsystem:  0x0002 (IMAGE_SUBSYSTEM_WINDOWS_GUI)
```

## 常见错误与解决

### 错误1：pefile 解析报错 "Invalid PE file"

```
原因：文件不是 PE 格式，或文件头被破坏
解决：
  1. 先用十六进制编辑器检查文件头
     - 偏移 0x00 应为 "MZ" (4D 5A)
     - 偏移 e_lfanew 处应为 "PE\0\0" (50 45 00 00)
  2. 如果头部被加密壳修改，需要先脱壳
  3. 确认文件不是 .NET 程序集（需要不同的分析工具）
```

### 错误2：RVA 到文件偏移转换错误

```
原因：直接用 RVA 当作文件偏移读取数据
解决：必须经过节表转换
  正确公式: FileOffset = RVA - VirtualAddress + PointerToRawData

  常见陷阱：
  - RVA 在头部范围内时，FileOffset = RVA（头部不需要转换）
  - SizeOfRawData 可能小于 VirtualSize（差值部分是 BSS）
  - 有些打包工具会故意修改节表，使转换失败
```

### 错误3：ASLR 导致每次加载地址不同

```
原因：DllCharacteristics 设置了 IMAGE_DLLCHARACTERISTICS_DYNAMIC_BASE
解决：
  1. 逆向分析时使用 RVA 而不是绝对 VA
  2. 在调试器中用 "模块名+偏移" 设断点: bp plugin+0x1234
  3. 或临时关闭 ASLR（修改 PE 头的 DllCharacteristics 标志位）
     CFF Explorer → Optional Header → DllCharacteristics → 去掉 0x0040
```

## 本節小結

| PE 组件 | 关键字段 | 逆向用途 |
|--------|---------|---------|
| DOS Header | e_lfanew | 定位 PE 头 |
| FILE_HEADER | Machine, Characteristics | 判断架构、是否 DLL |
| OPTIONAL_HEADER | EntryPoint, ImageBase | 入口点、基址 |
| DataDirectory | Export/Import/Reloc | 导入导出、重定位分析 |
| Section Headers | Name, VA, Characteristics | 代码/数据区域定位 |

核心认知：
- PE 格式是 Windows 逆向的基础，几乎所有工具都基于 PE 解析
- **RVA → FileOffset 转换**是手动分析的必备技能
- 数据目录的 EXPORT 和 IMPORT 项是 DLL 逆向的入口
- 了解节属性（可读/可写/可执行）有助于识别代码区和数据区

## 練習題与答案

### 练习1：判断 PE 文件类型

**题目**：给定以下 FILE_HEADER.Characteristics 值，判断文件类型和属性：
- (A) 0x0102
- (B) 0x2022
- (C) 0x210E

<details>
<summary>参考答案</summary>

```
标志位定义：
  0x0002 = EXECUTABLE_IMAGE
  0x0020 = LARGE_ADDRESS_AWARE
  0x0100 = 32BIT_MACHINE
  0x0200 = DEBUG_STRIPPED
  0x2000 = DLL
  0x0004 = LINE_NUMS_STRIPPED
  0x0008 = LOCAL_SYMS_STRIPPED

(A) 0x0102 = 0x0100 | 0x0002
    → 32位可执行文件 (EXE)

(B) 0x2022 = 0x2000 | 0x0020 | 0x0002
    → DLL文件，支持大地址（>2GB），可执行

(C) 0x210E = 0x2000 | 0x0100 | 0x0008 | 0x0004 | 0x0002
    → 32位 DLL，已剥离行号和符号信息
    （Release 编译的 DLL 通常有此特征）
```

</details>

### 练习2：RVA 转换计算

**题目**：PE 文件的节表信息如下：

```
.text:  VirtualAddress=0x1000, PointerToRawData=0x400,  VirtualSize=0xA000
.rdata: VirtualAddress=0xB000, PointerToRawData=0xA200, VirtualSize=0x3000
```

计算以下 RVA 对应的文件偏移：
- (A) RVA = 0x1234
- (B) RVA = 0xB500
- (C) RVA = 0x200

<details>
<summary>参考答案</summary>

```
公式: FileOffset = RVA - Section.VirtualAddress + Section.PointerToRawData

(A) RVA = 0x1234
    属于 .text 段 (0x1000 <= 0x1234 < 0x1000+0xA000)
    FileOffset = 0x1234 - 0x1000 + 0x400 = 0x634

(B) RVA = 0xB500
    属于 .rdata 段 (0xB000 <= 0xB500 < 0xB000+0x3000)
    FileOffset = 0xB500 - 0xB000 + 0xA200 = 0xA700

(C) RVA = 0x200
    不属于任何节（小于 .text 的 VirtualAddress）
    此 RVA 位于 PE 头部区域
    FileOffset = 0x200（头部的 RVA 等于文件偏移）
```

</details>
