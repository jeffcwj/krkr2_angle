## 1. PE 导出表（Export Table）

导出表是 DLL 对外公开函数的目录，是逆向 DLL 的第一步。

### 1.1 导出表数据结构

```c
typedef struct _IMAGE_EXPORT_DIRECTORY {
    DWORD Characteristics;       // 保留，通常为0
    DWORD TimeDateStamp;         // 导出表创建时间
    WORD  MajorVersion;          // 版本号
    WORD  MinorVersion;
    DWORD Name;                  // ★ DLL 名称的 RVA
    DWORD Base;                  // ★ 导出序号基值
    DWORD NumberOfFunctions;     // ★ 导出函数总数
    DWORD NumberOfNames;         // ★ 按名称导出的函数数
    DWORD AddressOfFunctions;    // ★ 函数地址表（EAT）的 RVA
    DWORD AddressOfNames;        // ★ 函数名称表的 RVA
    DWORD AddressOfNameOrdinals; // ★ 名称序号对照表的 RVA
} IMAGE_EXPORT_DIRECTORY;
```

### 1.2 导出表三表关联

```
导出表由三个并行数组构成：

AddressOfNames[i]     →  函数名 RVA      → "V2Link"
AddressOfNameOrdinals[i] → 序号           → 0
AddressOfFunctions[序号] → 函数地址 RVA   → 0x1000

查找流程（按名称导出）：
┌───────────────┐    ┌──────────────┐    ┌──────────────────┐
│ Names Table    │    │ Ordinals     │    │ Functions Table   │
│ (RVA 数组)     │    │ (WORD 数组)  │    │ (RVA 数组)        │
├───────────────┤    ├──────────────┤    ├──────────────────┤
│[0] → "V2Link" │───→│[0] → 0       │───→│[0] → 0x00001000  │
│[1] → "V2Unlink"│──→│[1] → 1       │───→│[1] → 0x00001050  │
└───────────────┘    └──────────────┘    └──────────────────┘

按名称查找 "V2Link":
  1. 在 Names Table 中二分查找 "V2Link" → 找到索引 0
  2. 用索引 0 查 Ordinals Table → 得到序号 0
  3. 用序号 0 查 Functions Table → 得到 RVA 0x1000
  4. 实际地址 = ImageBase + 0x1000
```

### 1.3 用 Python 解析导出表

```python
import pefile

def parse_exports(filepath):
    """解析 PE 导出表"""
    pe = pefile.PE(filepath)

    if not hasattr(pe, 'DIRECTORY_ENTRY_EXPORT'):
        print("该文件没有导出表（不是 DLL 或无导出）")
        return

    exp = pe.DIRECTORY_ENTRY_EXPORT
    print(f"DLL 名称: {exp.struct.Name}")
    print(f"导出基值: {exp.struct.Base}")
    print(f"函数总数: {exp.struct.NumberOfFunctions}")
    print(f"命名函数: {exp.struct.NumberOfNames}")
    print()

    print(f"{'Ordinal':>8s}  {'RVA':>10s}  {'Name'}")
    print(f"{'─'*8}  {'─'*10}  {'─'*30}")

    for symbol in exp.symbols:
        name = symbol.name.decode() if symbol.name else "(unnamed)"
        ordinal = symbol.ordinal
        rva = symbol.address

        # 检查是否是转发导出（Forwarded Export）
        # 如果 RVA 指向导出表所在区域，则是转发
        export_dir = pe.OPTIONAL_HEADER.DATA_DIRECTORY[0]
        is_forwarded = (export_dir.VirtualAddress <= rva <
                       export_dir.VirtualAddress + export_dir.Size)

        if is_forwarded:
            # 读取转发字符串
            fwd_str = pe.get_string_at_rva(rva).decode()
            print(f"{ordinal:8d}  {'FORWARD':>10s}  {name} → {fwd_str}")
        else:
            print(f"{ordinal:8d}  {hex(rva):>10s}  {name}")

    pe.close()

# parse_exports('wuvorbis.dll')
# 预期输出：
#  Ordinal         RVA  Name
#  ────────  ──────────  ──────────────────────────────
#        1     0x1000    V2Link
#        2     0x1050    V2Unlink
```

### 1.4 转发导出（Forwarded Export）

```
转发导出是一种特殊机制：DLL A 的导出函数实际由 DLL B 实现

示例：kernel32.dll 的部分函数转发到 ntdll.dll
  kernel32!HeapAlloc → NTDLL.RtlAllocateHeap
  kernel32!HeapFree  → NTDLL.RtlFreeHeap

在导出表中的表现：
  Functions Table 中的 RVA 指向导出目录区域内的一个字符串
  字符串格式: "DllName.FunctionName"
  例如: "NTDLL.RtlAllocateHeap"

逆向意义：
  追踪调用链时，需要注意转发导出
  否则会误以为函数在当前 DLL 中实现
```

## 2. PE 导入表（Import Table）

导入表记录了程序依赖的外部 DLL 和函数。

### 2.1 导入表数据结构

```c
// 导入描述符（每个导入的 DLL 一个）
typedef struct _IMAGE_IMPORT_DESCRIPTOR {
    union {
        DWORD Characteristics;
        DWORD OriginalFirstThunk;  // ★ INT (Import Name Table) 的 RVA
    };
    DWORD TimeDateStamp;           // 绑定时间戳
    DWORD ForwarderChain;          // 转发链
    DWORD Name;                    // ★ DLL 名称的 RVA
    DWORD FirstThunk;              // ★ IAT (Import Address Table) 的 RVA
} IMAGE_IMPORT_DESCRIPTOR;

// INT/IAT 条目（32位）
typedef struct _IMAGE_THUNK_DATA32 {
    union {
        DWORD ForwarderString;
        DWORD Function;           // 运行时填入函数地址
        DWORD Ordinal;            // 按序号导入
        DWORD AddressOfData;      // IMAGE_IMPORT_BY_NAME 的 RVA
    };
} IMAGE_THUNK_DATA32;

// 按名称导入的结构
typedef struct _IMAGE_IMPORT_BY_NAME {
    WORD  Hint;    // 导出序号提示（加速查找）
    CHAR  Name[1]; // 函数名（变长字符串）
} IMAGE_IMPORT_BY_NAME;
```

### 2.2 导入解析流程

```
加载前（文件中）：
┌─────────────────┐     ┌─────────────────┐
│  INT (OrigFirst │     │  IAT (First     │
│  Thunk)         │     │  Thunk)         │
├─────────────────┤     ├─────────────────┤
│→ {Hint=0,       │     │→ {Hint=0,       │  ← 两表内容相同
│   "MessageBoxW"}│     │   "MessageBoxW"}│
│→ {Hint=0,       │     │→ {Hint=0,       │
│   "ExitProcess"}│     │   "ExitProcess"}│
│→ NULL           │     │→ NULL           │
└─────────────────┘     └─────────────────┘

加载后（内存中）：
┌─────────────────┐     ┌─────────────────┐
│  INT            │     │  IAT            │
├─────────────────┤     ├─────────────────┤
│→ "MessageBoxW"  │     │→ 0x77D507EA     │  ← 被替换为真实地址！
│→ "ExitProcess"  │     │→ 0x7C81CAA2     │
│→ NULL           │     │→ NULL           │
└─────────────────┘     └─────────────────┘

代码中的调用：
  call dword ptr [0x00402000]  ← 这是 IAT 中的条目地址
  // 运行时 IAT[0] = MessageBoxW 的真实地址
  // 所以实际调用了 MessageBoxW
```

### 2.3 用 Python 解析导入表

```python
import pefile

def parse_imports(filepath):
    """解析 PE 导入表"""
    pe = pefile.PE(filepath)

    if not hasattr(pe, 'DIRECTORY_ENTRY_IMPORT'):
        print("该文件没有导入表")
        return

    total_funcs = 0
    for entry in pe.DIRECTORY_ENTRY_IMPORT:
        dll_name = entry.dll.decode()
        funcs = []
        for imp in entry.imports:
            if imp.name:
                funcs.append(f"{imp.name.decode()} (hint={imp.hint})")
            else:
                funcs.append(f"Ordinal #{imp.ordinal}")

        total_funcs += len(funcs)
        print(f"\n{dll_name} ({len(funcs)} 个函数):")
        print(f"  INT RVA: {hex(entry.struct.OriginalFirstThunk)}")
        print(f"  IAT RVA: {hex(entry.struct.FirstThunk)}")
        for fn in funcs:
            print(f"    - {fn}")

    print(f"\n共导入 {len(pe.DIRECTORY_ENTRY_IMPORT)} 个 DLL, "
          f"{total_funcs} 个函数")
    pe.close()

# parse_imports('kirikiri2.exe')
```

## 3. ELF 动态链接机制

### 3.1 GOT / PLT 机制

```
ELF 的动态链接通过 GOT 和 PLT 两个表配合实现：

GOT (Global Offset Table) — 全局偏移表
  存储外部符号的运行时地址
  位于 .got / .got.plt 节
  可写（运行时被动态链接器填充）

PLT (Procedure Linkage Table) — 过程链接表
  外部函数调用的跳板代码
  位于 .plt 节
  只读（固定的跳板指令）
```

### 3.2 PLT 调用流程（延迟绑定）

```
首次调用 printf:
┌──────────┐     ┌──────────────┐     ┌──────────────┐
│ call code │     │ printf@plt    │     │ .got.plt[n]  │
│           │────→│ jmp *GOT[n]  │────→│ (初始值:      │
│           │     │              │     │  push_n_addr) │
│           │     │ push n       │←────│              │
│           │     │ jmp PLT[0]   │     └──────────────┘
└──────────┘     │              │
                  │ PLT[0]:      │     ┌──────────────┐
                  │ push GOT[1]  │────→│ _dl_runtime   │
                  │ jmp  GOT[2]  │     │ _resolve      │
                  └──────────────┘     │              │
                                       │ 1.解析printf  │
                                       │ 2.填GOT[n]   │
                                       │ 3.跳到printf  │
                                       └──────────────┘

后续调用 printf:
┌──────────┐     ┌──────────────┐     ┌──────────────┐
│ call code │────→│ printf@plt    │────→│ .got.plt[n]  │
│           │     │ jmp *GOT[n]  │     │ → 0x7f...    │──→ printf 实际代码
└──────────┘     └──────────────┘     │ (已解析)      │
                                       └──────────────┘
```

### 3.3 PLT 条目的汇编代码

```asm
; PLT[0] — 公共解析器入口
; 所有未解析的 PLT 条目最终跳到这里
.plt:
    push   QWORD PTR [rip+GOT[1]]   ; 推入 link_map 指针
    jmp    QWORD PTR [rip+GOT[2]]   ; 跳到 _dl_runtime_resolve

; PLT[n] — printf 的 PLT 条目
printf@plt:
    jmp    QWORD PTR [rip+GOT_printf]  ; 跳到 GOT 中存储的地址
    push   0x0                          ; relocation index
    jmp    .plt                         ; 跳到公共解析器

; 首次调用时 GOT_printf 指向 "push 0x0" 这条指令
; 解析后 GOT_printf 被修改为 printf 的真实地址
```

### 3.4 .dynamic 段

```bash
# .dynamic 段包含动态链接器需要的所有信息
readelf -d target

# 关键标签：
#  NEEDED   — 依赖的共享库名
#  SONAME   — 自身的 SO 名称
#  SYMTAB   — 动态符号表地址
#  STRTAB   — 动态字符串表地址
#  PLTGOT   — GOT 表地址
#  JMPREL   — PLT 重定位表地址
#  REL/RELA — 其他重定位表地址
#  INIT     — 初始化函数地址
#  FINI     — 终止函数地址
```

## 4. 延迟加载（Delay Import）

### 4.1 PE 延迟导入

```c
// 延迟导入描述符
typedef struct ImgDelayDescr {
    DWORD grAttrs;         // 属性
    DWORD rvaDLLName;      // DLL 名称 RVA
    DWORD rvaHmod;         // 模块句柄 RVA
    DWORD rvaIAT;          // 延迟 IAT 的 RVA
    DWORD rvaINT;          // 延迟 INT 的 RVA
    DWORD rvaBoundIAT;     // 绑定延迟 IAT
    DWORD rvaUnloadIAT;    // 卸载延迟 IAT
    DWORD dwTimeStamp;     // 时间戳
} ImgDelayDescr;
```

```
延迟导入的工作原理：

普通导入：程序启动时加载所有 DLL → 启动慢但调用快
延迟导入：首次调用时才加载 DLL → 启动快但首次调用慢

流程：
  1. 编译时：call [delay_IAT_entry]
  2. 首次调用：delay_IAT 指向延迟加载辅助函数
     → LoadLibrary(dll_name)
     → GetProcAddress(hModule, func_name)
     → 将真实地址写入 delay_IAT
     → 跳转到真实函数
  3. 后续调用：delay_IAT 已是真实地址，直接调用

KiriKiri2 中的应用：
  部分不常用的 Windows API（如打印、网络）可能使用延迟导入
  避免在所有环境下都要求这些 DLL 存在
```

### 4.2 ELF 的延迟加载

```c
// ELF 使用 dlopen/dlsym 实现按需加载
#include <dlfcn.h>

// 运行时加载共享库（类似 LoadLibrary）
void *handle = dlopen("libplugin.so", RTLD_LAZY);
if (!handle) {
    fprintf(stderr, "dlopen failed: %s\n", dlerror());
    return -1;
}

// 获取函数指针（类似 GetProcAddress）
typedef int (*V2LinkFunc)(void *);
V2LinkFunc v2link = (V2LinkFunc)dlsym(handle, "V2Link");
if (!v2link) {
    fprintf(stderr, "dlsym failed: %s\n", dlerror());
    dlclose(handle);
    return -1;
}

// 调用插件入口
int result = v2link(tjs_dispatch);

// 卸载（类似 FreeLibrary）
dlclose(handle);
```

## 5. 跨平台对比总结

```
┌────────────────┬─────────────────────┬──────────────────────┐
│ 功能            │ PE (Windows)         │ ELF (Linux)           │
├────────────────┼─────────────────────┼──────────────────────┤
│ 导出函数列表    │ Export Table (EAT)  │ .dynsym + .dynstr    │
│ 导入函数列表    │ Import Table + IAT  │ .dynsym + .rela.plt  │
│ 运行时地址表    │ IAT                  │ GOT / .got.plt       │
│ 调用跳板        │ (直接 call [IAT])   │ PLT                  │
│ 绑定时机        │ 加载时（eager）      │ 首次调用（lazy）      │
│ 显式加载库      │ LoadLibrary         │ dlopen               │
│ 显式获取函数    │ GetProcAddress      │ dlsym                │
│ 卸载库          │ FreeLibrary         │ dlclose              │
│ 延迟导入        │ Delay Import Table  │ RTLD_LAZY (默认)     │
│ 查看导出        │ dumpbin /exports    │ nm -D / readelf -s   │
│ 查看导入        │ dumpbin /imports    │ readelf -d / objdump │
│ 转发导出        │ "DLL.Function"      │ 符号别名(alias)      │
└────────────────┴─────────────────────┴──────────────────────┘
```

## 動手実践

编写一个导入/导出表完整分析工具：

```python
import pefile
import sys

def full_analysis(filepath):
    """PE 导入导出表完整分析"""
    pe = pefile.PE(filepath)
    is_dll = pe.is_dll()

    print(f"{'='*60}")
    print(f"文件: {filepath}")
    print(f"类型: {'DLL' if is_dll else 'EXE'}")
    print(f"{'='*60}")

    # 导出表分析
    if hasattr(pe, 'DIRECTORY_ENTRY_EXPORT'):
        exp = pe.DIRECTORY_ENTRY_EXPORT
        print(f"\n[导出表] ({exp.struct.NumberOfFunctions} 个函数)")
        print(f"  DLL名称:    {pe.get_string_at_rva(exp.struct.Name).decode()}")
        print(f"  基序号:     {exp.struct.Base}")

        kr2_markers = []
        for sym in exp.symbols:
            name = sym.name.decode() if sym.name else None
            if name in ('V2Link', 'V2Unlink'):
                kr2_markers.append(name)
            rva_str = hex(sym.address) if sym.address else 'N/A'
            print(f"  [{sym.ordinal:3d}] {name or '(ordinal only)':30s} "
                  f"RVA={rva_str}")

        if len(kr2_markers) == 2:
            print(f"\n  🎮 确认: KiriKiri2 插件 (导出 V2Link + V2Unlink)")
    else:
        print(f"\n[导出表] 无")

    # 导入表分析
    if hasattr(pe, 'DIRECTORY_ENTRY_IMPORT'):
        print(f"\n[导入表]")
        api_categories = {
            'File I/O': ['CreateFile', 'ReadFile', 'WriteFile', 'CloseHandle'],
            'Memory': ['VirtualAlloc', 'VirtualFree', 'HeapAlloc'],
            'DLL Loading': ['LoadLibrary', 'GetProcAddress', 'FreeLibrary'],
            'Thread': ['CreateThread', 'WaitForSingleObject'],
            'Registry': ['RegOpenKey', 'RegQueryValue'],
            'Network': ['socket', 'connect', 'send', 'recv'],
            'COM': ['CoInitialize', 'CoCreateInstance'],
        }

        detected_apis = {}
        for entry in pe.DIRECTORY_ENTRY_IMPORT:
            dll = entry.dll.decode()
            func_names = []
            for imp in entry.imports:
                name = imp.name.decode() if imp.name else f"Ord#{imp.ordinal}"
                func_names.append(name)
                # 分类 API
                for cat, patterns in api_categories.items():
                    if any(p in name for p in patterns):
                        detected_apis.setdefault(cat, []).append(
                            f"{dll}:{name}")

            print(f"\n  {dll} ({len(func_names)} 函数)")
            for fn in func_names[:8]:
                print(f"    {fn}")
            if len(func_names) > 8:
                print(f"    ... +{len(func_names)-8} more")

        if detected_apis:
            print(f"\n[API 行为分析]")
            for cat, apis in detected_apis.items():
                print(f"  {cat}: {', '.join(apis[:3])}")
    else:
        print(f"\n[导入表] 无")

    pe.close()

if __name__ == '__main__':
    if len(sys.argv) < 2:
        print("用法: python pe_analysis.py <PE文件>")
    else:
        full_analysis(sys.argv[1])
```

## 対照項目源码

KiriKiri2 的插件加载代码直接体现了导入/导出机制的使用：

```cpp
// 插件加载流程（简化版，参考 cpp/core/plugin/ 目录）

// 1. 使用 LoadLibrary 加载插件 DLL
HMODULE hPlugin = LoadLibraryW(L"plugin\\wuvorbis.dll");
// → 此时 Windows loader 解析 DLL 的导入表
// → DLL 的 IAT 被填充为真实函数地址

// 2. 使用 GetProcAddress 获取导出函数
typedef HRESULT (__stdcall *V2LinkFunc)(iTJSDispatch2*);
V2LinkFunc pV2Link = (V2LinkFunc)GetProcAddress(hPlugin, "V2Link");
// → 查找 DLL 导出表中名为 "V2Link" 的条目
// → 返回其 RVA + 加载基址 = 实际内存地址

// 3. 调用插件入口
HRESULT hr = pV2Link(pTJSGlobal);
// → 插件内部注册 TJS2 类到引擎

// 4. 卸载时调用 V2Unlink
V2UnlinkFunc pV2Unlink = (V2UnlinkFunc)GetProcAddress(hPlugin, "V2Unlink");
pV2Unlink();
FreeLibrary(hPlugin);
```

## 常见错误与解决

### 错误1：导入表中看不到 GetProcAddress 调用的函数

```
原因：GetProcAddress 是运行时动态解析，不在导入表中
解决：
  导入表只记录编译时确定的依赖
  GetProcAddress 动态获取的函数需要通过：
  1. 搜索字符串表 → 找函数名字符串 → 追踪交叉引用
  2. 动态分析 → 在 GetProcAddress 设断点 → 查看参数
  3. KiriKiri2 插件的 V2Link 就是通过 GetProcAddress 获取的
```

### 错误2：IAT 地址在调试器中与 IDA 中不同

```
原因：ASLR 导致 DLL 加载基址变化，IAT 内容也随之变化
解决：
  1. IAT 中存储的是运行时绝对地址，每次加载都可能不同
  2. 静态分析时关注 IAT 的 RVA（文件中的位置不变）
  3. IDA 中看到的 IAT 是文件中的初始值（函数名/序号）
  4. 调试器中看到的是加载器填充后的真实地址
```

### 错误3：ELF .so 的 nm 输出全是 "U" 未定义符号

```
原因：使用了 nm 而不是 nm -D，或库已被 strip
解决：
  nm        → 查看 .symtab（可能被 strip 删除）
  nm -D     → 查看 .dynsym（动态符号表，不会被 strip）
  readelf -s --dyn-syms  → 同样查看动态符号表
```

---

## 本節小結

| 概念 | PE 机制 | ELF 机制 | 逆向分析要点 |
|------|---------|---------|------------|
| 导出 | Export Table + EAT | .dynsym | DLL 逆向第一步：查看导出 |
| 导入 | Import Table + IAT | GOT + PLT | 了解程序依赖和调用的 API |
| 运行时解析 | LoadLibrary + GetProcAddress | dlopen + dlsym | 字符串搜索找动态加载目标 |
| 绑定时机 | 加载时 / 延迟导入 | 延迟绑定 / LD_BIND_NOW | 影响断点策略 |

核心认知：
- 导出表是逆向 DLL 的**入口**——从导出函数开始分析
- 导入表揭示程序的**能力**——知道它调用了哪些 API
- **IAT Hook** 是常见的拦截技术：修改 IAT 条目指向自己的函数
- KiriKiri2 的插件系统完整体现了动态链接的实际应用

### 练习1：IAT Hook 原理

**题目**：解释 IAT Hook 的原理，并说明如何用它拦截 KiriKiri2 的 CreateFileW 调用。

<details>
<summary>参考答案</summary>

```
IAT Hook 原理：
  1. 程序调用 CreateFileW 时，实际执行 call [IAT_CreateFileW]
  2. IAT_CreateFileW 存储的是 CreateFileW 的真实地址
  3. IAT Hook 将 IAT_CreateFileW 修改为我们的 Hook 函数地址
  4. 之后所有 CreateFileW 调用都会先经过 Hook 函数

实现步骤：
  // 1. 找到 IAT 中 CreateFileW 的条目
  PIMAGE_IMPORT_DESCRIPTOR pImport = ...;  // 遍历导入表
  // 找到 KERNEL32.dll 的导入描述符
  // 遍历其 FirstThunk (IAT) 找到 CreateFileW

  // 2. 修改内存保护（IAT 通常是只读的）
  DWORD oldProtect;
  VirtualProtect(pIATEntry, sizeof(DWORD), PAGE_READWRITE, &oldProtect);

  // 3. 替换函数指针
  FARPROC originalFunc = *pIATEntry;
  *pIATEntry = (FARPROC)MyCreateFileW;

  // 4. 恢复内存保护
  VirtualProtect(pIATEntry, sizeof(DWORD), oldProtect, &oldProtect);

  // 5. Hook 函数（记录并转发）
  HANDLE WINAPI MyCreateFileW(LPCWSTR lpFileName, ...) {
      printf("CreateFileW: %ws\n", lpFileName);  // 记录访问
      return originalFunc(lpFileName, ...);  // 调用原函数
  }

应用：监控 KiriKiri2 打开了哪些游戏资源文件
```

</details>

### 练习2：比较 PE 和 ELF 的符号可见性控制

**题目**：PE 用 `__declspec(dllexport)` 导出函数，ELF 用什么机制？两者的默认行为有何不同？

<details>
<summary>参考答案</summary>

```
PE 符号可见性：
  默认：函数不导出（私有）
  导出：必须显式标记 __declspec(dllexport) 或使用 .def 文件
  → 导出表只包含显式导出的函数

ELF 符号可见性：
  默认：函数全部导出（公有）! ← 与 PE 完全相反
  控制方式：
    1. __attribute__((visibility("hidden")))  — 隐藏特定函数
    2. -fvisibility=hidden 编译选项 — 默认隐藏所有
       + __attribute__((visibility("default"))) — 显式导出
    3. 版本脚本（linker version script）— 精确控制

关键差异：
  PE:  opt-in 导出（需要显式标记）
  ELF: opt-out 导出（需要显式隐藏）

  这意味着：
  - 未优化的 ELF .so 可能导出大量内部函数 → 逆向更容易
  - 使用 -fvisibility=hidden 的库更难分析（类似 PE）
  - strip 可以删除 .symtab 但不影响 .dynsym（已导出的动态符号）
```

</details>
