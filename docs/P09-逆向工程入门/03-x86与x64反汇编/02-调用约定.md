# 调用约定

## 本节目标

- 理解调用约定（Calling Convention）的核心要素：参数传递、栈清理、寄存器保存
- 掌握 x86 下 cdecl、stdcall、thiscall、fastcall 四种调用约定的差异
- 掌握 x86-64 下 Microsoft x64 和 System V AMD64 两种调用约定
- 能从反汇编中识别函数使用的调用约定
- 结合 KiriKiri2 实际代码分析 thiscall 和 stdcall 的混合使用

---

## 1. 调用约定概述

调用约定定义了函数调用时的三个关键规则：

```
┌─────────────────────────────────────────────────────────┐
│                    调用约定三要素                         │
├─────────────────┬───────────────────────────────────────┤
│ ① 参数传递方式  │ 通过栈/寄存器，顺序（左→右 or 右→左）│
│ ② 栈清理责任   │ 调用者(caller) or 被调者(callee) 清理 │
│ ③ 寄存器保存   │ 哪些寄存器由被调者负责保存/恢复       │
├─────────────────┴───────────────────────────────────────┤
│ 影响范围: 函数原型推断、参数个数识别、栈帧分析          │
└─────────────────────────────────────────────────────────┘
```

> **为什么逆向工程师必须掌握调用约定？**
> 反汇编中没有函数签名——你必须从 `push`/`pop`/`add esp` 模式推断出参数个数和类型。错判调用约定会导致参数个数完全错误。

## 2. x86（32 位）调用约定

### 2.1 cdecl —— C 语言默认

cdecl 是 C 语言的默认调用约定，也是最基本的约定：

```
特征:
- 参数: 右→左 入栈
- 栈清理: 调用者 (caller)
- 返回值: EAX (整数/指针), ST(0) (浮点)
- 修饰名: _function (前缀下划线)
```

```nasm
; C 源码: int result = printf("%d + %d = %d\n", a, b, a+b);
; cdecl 调用示例:
push    eax              ; 参数 4: a+b (最右边的参数最先入栈)
push    [ebp+var_b]      ; 参数 3: b
push    [ebp+var_a]      ; 参数 2: a
push    offset fmt_str   ; 参数 1: "%d + %d = %d\n"
call    _printf          ; cdecl 函数调用
add     esp, 10h         ; ← 调用者清栈: 4 个参数 × 4 字节 = 16 = 0x10
mov     [ebp+result], eax ; 返回值在 EAX
```

**关键识别特征**：`call` 之后紧跟 `add esp, N` → 几乎一定是 cdecl。

```nasm
; ⚠ 编译器优化：延迟栈清理
push    eax
call    _func1           ; cdecl
push    ebx
call    _func2           ; cdecl
add     esp, 8           ; 一次性清理两次调用的参数！
; 优化器将多次 add esp 合并，新手容易误判参数个数
```

### 2.2 stdcall —— Windows API 标准

stdcall 是 Windows API 的标准约定：

```
特征:
- 参数: 右→左 入栈
- 栈清理: 被调者 (callee)
- 返回值: EAX
- 修饰名: _function@N (N = 参数字节数)
```

```nasm
; C 源码: HANDLE h = CreateFileW(path, GENERIC_READ, 0, NULL, OPEN_EXISTING, 0, NULL);
; stdcall — 7 个参数:
push    0                ; hTemplateFile = NULL
push    0                ; dwFlagsAndAttributes = 0
push    3                ; dwCreationDisposition = OPEN_EXISTING (3)
push    0                ; lpSecurityAttributes = NULL
push    0                ; dwShareMode = 0
push    80000000h        ; dwDesiredAccess = GENERIC_READ
push    [ebp+path]       ; lpFileName
call    ds:CreateFileW   ; stdcall — 被调者清栈
; ← 没有 add esp！这是 stdcall 的标志
mov     [ebp+handle], eax
```

**关键识别特征**：`call` 之后**没有** `add esp` → 可能是 stdcall（或 thiscall）。

```nasm
; stdcall 函数内部的 RET 指令:
ret     1Ch              ; ret N — 弹出 7×4=28=0x1C 字节参数
; RET N 说明这是 stdcall，N/4 就是参数个数
```

### 2.3 thiscall —— C++ 成员函数（MSVC）

thiscall 是 MSVC 编译器用于 C++ 非静态成员函数的约定：

```
特征:
- this 指针: ECX
- 其余参数: 右→左 入栈
- 栈清理: 被调者 (callee)（与 stdcall 一致）
- 修饰名: ?method@Class@@QAEHXZ (C++ name mangling)
```

```nasm
; C++: obj->SetPosition(100, 200);
; thiscall 调用:
push    200              ; 参数 2: y (右→左)
push    100              ; 参数 1: x
mov     ecx, [ebp+obj]  ; this → ECX ← thiscall 的标志
call    ?SetPosition@CObject@@QAEHKK@Z
; 没有 add esp（被调者清栈）
```

**KiriKiri2 中的 thiscall 实例**：

```nasm
; C++: dispatch->FuncCall(0, L"onCreate", NULL, NULL, 0, NULL, dispatch);
; iTJSDispatch2::FuncCall 是虚函数 + thiscall
mov     ecx, [ebp+dispatch]     ; this → ECX
mov     eax, [ecx]              ; eax = vtable 指针
push    ecx                     ; 参数 7: objthis
push    0                       ; 参数 6: numparams
push    0                       ; 参数 5: param
push    0                       ; 参数 4: result
push    offset aOncreate        ; 参数 3: membername = L"onCreate"
push    0                       ; 参数 2: hint (*)
push    0                       ; 参数 1: flag
call    dword ptr [eax+44h]     ; vtable[0x44/4] = FuncCall
; 虚函数调用 = thiscall + 间接调用
```

### 2.4 fastcall —— 寄存器传参

fastcall 使用寄存器传递前两个参数以提升性能：

```
特征:
- 前两个参数: ECX, EDX (整数/指针)
- 其余参数: 右→左 入栈
- 栈清理: 被调者 (callee)
- 修饰名: @function@N
```

```nasm
; C: __fastcall void process(int a, int b, int c, int d);
; fastcall 调用:
push    [ebp+d]          ; 参数 4: d (栈)
push    [ebp+c]          ; 参数 3: c (栈)
mov     edx, [ebp+b]     ; 参数 2: b → EDX
mov     ecx, [ebp+a]     ; 参数 1: a → ECX
call    @process@16      ; fastcall，16 = 4个参数 × 4字节
; 没有 add esp
```

> **易混淆点**：thiscall 只用 ECX 传 this，fastcall 同时用 ECX 和 EDX。如果看到 `call` 前同时设置了 ECX 和 EDX 且不是内存操作数准备，考虑 fastcall。

### 2.5 四种约定对比总结

```
┌────────────┬──────────┬──────────┬──────────┬──────────┐
│            │  cdecl   │ stdcall  │ thiscall │ fastcall │
├────────────┼──────────┼──────────┼──────────┼──────────┤
│ 参数传递    │ 全部栈   │ 全部栈   │ ECX=this │ ECX,EDX  │
│            │          │          │ 余下入栈  │ 余下入栈 │
│ 参数顺序    │ 右→左   │ 右→左   │ 右→左    │ 右→左   │
│ 栈清理     │ 调用者   │ 被调者   │ 被调者   │ 被调者   │
│ 可变参数    │ ✓       │ ✗       │ ✗       │ ✗       │
│ RET 指令   │ ret      │ ret N   │ ret N    │ ret N   │
│ call 之后   │ add esp  │ (无)    │ (无)     │ (无)    │
│ 修饰名前缀  │ _func   │ _func@N │ ?mangled │ @func@N │
│ 典型使用场景│ C 库函数 │ WinAPI  │ C++ 方法 │ 性能热点 │
└────────────┴──────────┴──────────┴──────────┴──────────┘
```

## 3. x86-64（64 位）调用约定

64 位模式下调用约定简化为两种主要标准。

### 3.1 Microsoft x64 调用约定

Windows 64 位程序统一使用此约定（不再区分 cdecl/stdcall/thiscall）：

```
特征:
- 前 4 个整数/指针参数: RCX, RDX, R8, R9
- 前 4 个浮点参数: XMM0, XMM1, XMM2, XMM3
- 第 5+ 个参数: 栈传递
- 栈清理: 调用者
- 影子空间 (Shadow Space): 调用者预留 32 字节 (4×8)
- 返回值: RAX (整数), XMM0 (浮点)
```

```nasm
; C++: result = CreateFileW(path, GENERIC_READ, 0, NULL, OPEN_EXISTING, 0, NULL);
; Microsoft x64:
sub     rsp, 58h         ; 预留栈空间 (影子空间 + 参数 + 对齐)
mov     qword ptr [rsp+30h], 0   ; 参数 7: hTemplateFile (栈)
mov     dword ptr [rsp+28h], 0   ; 参数 6: dwFlagsAndAttributes (栈)
mov     dword ptr [rsp+20h], 3   ; 参数 5: dwCreationDisposition (栈)
xor     r9d, r9d                 ; 参数 4: lpSecurityAttributes → R9 = NULL
xor     r8d, r8d                 ; 参数 3: dwShareMode → R8 = 0
mov     edx, 80000000h           ; 参数 2: dwDesiredAccess → RDX
mov     rcx, [path]              ; 参数 1: lpFileName → RCX
call    CreateFileW
; RAX = 返回值
add     rsp, 58h                 ; 调用者清栈
```

**影子空间（Shadow Space）详解**：

```
调用者栈帧布局 (Microsoft x64):
┌──────────────────────────┐ ← RSP 调用前
│ 返回地址 (8 字节)         │ ← call 指令自动压入
├──────────────────────────┤
│ Shadow[0] for RCX (8B)   │ [RSP+8]  ← 被调者可在此保存 RCX
│ Shadow[1] for RDX (8B)   │ [RSP+16]
│ Shadow[2] for R8  (8B)   │ [RSP+24]
│ Shadow[3] for R9  (8B)   │ [RSP+32]
├──────────────────────────┤
│ 参数 5 (8B)              │ [RSP+40]
│ 参数 6 (8B)              │ [RSP+48]
│ ...                      │
└──────────────────────────┘

; 被调者（被调函数）可以将寄存器参数保存到影子空间:
; 函数入口处常见:
mov     [rsp+8],  rcx    ; 保存参数 1 到影子空间
mov     [rsp+10h], rdx   ; 保存参数 2
mov     [rsp+18h], r8    ; 保存参数 3
mov     [rsp+20h], r9    ; 保存参数 4
```

### 3.2 System V AMD64 调用约定（Linux/macOS）

```
特征:
- 前 6 个整数/指针参数: RDI, RSI, RDX, RCX, R8, R9
- 前 8 个浮点参数: XMM0–XMM7
- 第 7+ 个参数: 栈传递
- 栈清理: 调用者
- 无影子空间
- Red Zone: RSP 下方 128 字节可直接使用（叶子函数优化）
- 返回值: RAX (整数), XMM0 (浮点)
```

```nasm
; C: printf("%d + %d = %d\n", a, b, sum);
; System V AMD64:
mov     ecx, [rbp+sum]   ; 参数 4 → RCX (注意：与 MS x64 的 R9 不同！)
mov     edx, [rbp+b]     ; 参数 3 → RDX
mov     esi, [rbp+a]     ; 参数 2 → RSI
lea     rdi, [fmt_str]   ; 参数 1 → RDI
xor     eax, eax         ; AL = 0 (可变参数函数：XMM 寄存器使用数)
call    printf
```

### 3.3 两种 x64 约定对比

```
┌────────────────┬──────────────────┬──────────────────┐
│                │ Microsoft x64    │ System V AMD64   │
├────────────────┼──────────────────┼──────────────────┤
│ 参数 1         │ RCX / XMM0      │ RDI              │
│ 参数 2         │ RDX / XMM1      │ RSI              │
│ 参数 3         │ R8  / XMM2      │ RDX              │
│ 参数 4         │ R9  / XMM3      │ RCX              │
│ 参数 5         │ [RSP+28h]       │ R8               │
│ 参数 6         │ [RSP+30h]       │ R9               │
│ 参数 7+        │ 栈              │ 栈               │
│ 影子空间       │ 32 字节 (必须)   │ 无               │
│ Red Zone       │ 无              │ 128 字节          │
│ 寄存器参数上限  │ 4 个            │ 6 个             │
│ 被调者保存     │ RBX,RBP,RDI,    │ RBX,RBP,         │
│                │ RSI,R12-R15     │ R12-R15          │
│ 使用平台       │ Windows         │ Linux, macOS     │
└────────────────┴──────────────────┴──────────────────┘
```

> **KiriKiri2 上下文**：KiriKiri2 是 Windows 专用引擎，所有 64 位构建使用 Microsoft x64 约定。但如果你在 Linux 上交叉分析移植版或插件，需要注意 SysV 差异。

## 4. 寄存器保存规则

### 4.1 Caller-saved vs Callee-saved

```
┌─────────────────────────────────────────────────────────┐
│        x86 寄存器保存规则                                │
├──────────────────┬──────────────────────────────────────┤
│ Caller-saved     │ EAX, ECX, EDX                       │
│ (易失性/volatile)│ 调用函数后值可能改变                  │
│                  │ 调用者需自行保存                      │
├──────────────────┼──────────────────────────────────────┤
│ Callee-saved     │ EBX, ESI, EDI, EBP, ESP             │
│ (非易失/preserve)│ 被调者如果使用必须先保存再恢复        │
│                  │ 跨函数调用后值不变                    │
└──────────────────┴──────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│        x64 (Microsoft) 寄存器保存规则                    │
├──────────────────┬──────────────────────────────────────┤
│ Caller-saved     │ RAX, RCX, RDX, R8-R11               │
│ (volatile)       │ XMM0-XMM5                            │
├──────────────────┼──────────────────────────────────────┤
│ Callee-saved     │ RBX, RBP, RDI, RSI, R12-R15         │
│ (non-volatile)   │ XMM6-XMM15                           │
└──────────────────┴──────────────────────────────────────┘
```

**逆向中的实际意义**：

```nasm
; 看到函数开头保存 ESI, EDI → 它们是 callee-saved
push    esi
push    edi
; ... 函数体中使用 ESI, EDI ...
pop     edi              ; 恢复
pop     esi
ret

; 如果函数体中用 ECX 存了重要值，但中间有 call 指令
; → ECX 在 call 后已被破坏（caller-saved）
; → 编译器会在 call 前把 ECX 保存到栈或 callee-saved 寄存器
mov     esi, ecx         ; 保存到 callee-saved 寄存器
call    SomeFunction     ; ECX 可能被覆盖
mov     ecx, esi         ; 恢复
```

## 5. 调用约定的识别技巧

### 5.1 快速识别流程

```
                 函数调用识别流程
                       │
            call 之后有 add esp?
              ╱                ╲
            是                  否
            │                   │
         cdecl             检查 ECX 设置
            │               ╱        ╲
     (可变参数)        有 ECX         无 ECX
                       设置 this      设置
                        │              │
                    thiscall       检查 RET N
                                   ╱      ╲
                                有 N      无 N
                                 │         │
                            stdcall    可能是间接
                            或fastcall  调用/优化
```

### 5.2 实战识别示例

```nasm
; 示例 1: 识别 cdecl
push    offset str_hello
push    ecx
call    sub_401000
add     esp, 8           ; ← caller 清栈 → cdecl
; 结论: sub_401000 是 cdecl，2 个参数

; 示例 2: 识别 stdcall
push    0
push    offset str_title
push    offset str_msg
push    [ebp+hwnd]
call    ds:MessageBoxW   ; Windows API
; 没有 add esp → stdcall
; 结论: MessageBoxW(hwnd, msg, title, 0) — 4 参数 stdcall

; 示例 3: 识别 thiscall
lea     ecx, [ebp+var_obj]  ; ← this → ECX
push    42                   ; 参数 1
call    sub_402000
; 没有 add esp → 被调者清栈
; ECX 设置了 this → thiscall
; 结论: obj.Method(42) — thiscall, 1 个显式参数

; 示例 4: 识别 fastcall
mov     edx, [ebp+param2]   ; ← EDX = 参数 2
mov     ecx, [ebp+param1]   ; ← ECX = 参数 1
push    [ebp+param3]         ; 参数 3 入栈
call    sub_403000
; ECX + EDX 同时设置 → fastcall
; 结论: func(p1, p2, p3) — fastcall, 3 个参数
```

### 5.3 IDA Pro 中的调用约定标注

```c
// IDA 反编译器 (Hex-Rays) 自动识别调用约定:
int __cdecl parse_line(const char *line, int len);

int __fastcall calc_hash(int key, int seed, int salt);

// 如果 IDA 判断错误，可以手动修改:
// Edit → Functions → Set function type (Y 快捷键)
// 输入: int __thiscall func(int this, int a, int b)
```

## 6. KiriKiri2 中的调用约定实战

### 6.1 混合调用约定分析

KiriKiri2 代码中同时存在多种调用约定：

```cpp
// tp_stub.h 中的接口定义
class iTJSDispatch2 {
public:
    // 虚函数 → thiscall (MSVC)
    virtual tjs_error TJS_INTF_METHOD
        FuncCall(tjs_uint32 flag,
                 const tjs_char *membername,
                 tjs_uint32 *hint,
                 tTJSVariant *result,
                 tjs_int numparams,
                 tTJSVariant **param,
                 iTJSDispatch2 *objthis) = 0;
};

// TJS_INTF_METHOD 在 Windows 上定义为 __cdecl
// 但通过虚函数调用时，实际行为取决于编译器实现
```

对应反汇编分析：

```nasm
; iTJSDispatch2::FuncCall 的调用
; thiscall: ECX = this
mov     ecx, [ebp+dispatch]
mov     eax, [ecx]              ; vtable

; 参数从右到左入栈 (7 个参数)
push    ecx                     ; objthis (参数 7)
push    0                       ; numparams (参数 6)
; ... 省略中间参数 ...
push    0                       ; flag (参数 1)

; 虚函数调用
call    dword ptr [eax+44h]     ; FuncCall 在 vtable 中的偏移

; 检查返回值 (tjs_error)
test    eax, eax
js      loc_Error               ; 负数 = 错误 (TJS_E_xxx)
```

### 6.2 插件导出函数的约定

KiriKiri2 插件的标准导出函数使用特定约定：

```cpp
// 插件标准入口 — 使用 __stdcall
extern "C" __declspec(dllexport) HRESULT __stdcall
    V2Link(iTVPFunctionExporter *exporter);

extern "C" __declspec(dllexport) HRESULT __stdcall
    V2Unlink();
```

```nasm
; V2Link 反汇编
; __stdcall: 参数通过栈传递，被调者清栈
_V2Link@4 proc near         ; 修饰名: _V2Link@4 (1个参数×4字节)
    arg_exporter = dword ptr 8

    push    ebp
    mov     ebp, esp
    mov     eax, [ebp+arg_exporter]  ; iTVPFunctionExporter*
    ; ... 初始化逻辑 ...
    xor     eax, eax         ; return S_OK (0)
    pop     ebp
    ret     4                ; ← stdcall: 清理 4 字节参数
_V2Link@4 endp
```

## 7. 动手实践

### 练习 1：调用约定识别

分析以下四段反汇编，判断每段使用的调用约定：

```nasm
; 片段 A
push    0
push    offset aTitle
push    offset aMessage
push    0
call    dword ptr [ebx+10h]
; (没有 add esp)

; 片段 B
push    200
push    100
call    sub_401500
add     esp, 8

; 片段 C
mov     edx, 64h
mov     ecx, [ebp+obj]
push    0Ah
call    sub_402000
; (没有 add esp)

; 片段 D
lea     ecx, [ebp-20h]
call    sub_403000
; (没有 add esp，无 push)
```

<details>
<summary>参考答案</summary>

| 片段 | 调用约定 | 理由 |
|------|---------|------|
| A | stdcall 或 thiscall | 无 `add esp`，间接调用（可能是虚函数→thiscall）|
| B | cdecl | `add esp, 8` → 调用者清栈 |
| C | fastcall | ECX + EDX 同时设置，无 `add esp` |
| D | thiscall | 仅 ECX 设置（this），无额外参数→`Method()` |

片段 A 的进一步判断：`call dword ptr [ebx+10h]` 是间接调用。如果 EBX 是对象指针，`[ebx]` 是 vtable，`[ebx+10h]` 是虚函数 → thiscall。但需要看 ECX 是否在之前被设置。

</details>

### 练习 2：参数个数推断

```nasm
; 函数 sub_405000 的入口:
sub_405000 proc near
    push    ebp
    mov     ebp, esp
    push    esi
    push    edi
    mov     esi, [ebp+8]      ; 参数 1
    mov     edi, [ebp+0Ch]    ; 参数 2
    mov     ecx, [ebp+10h]    ; 参数 3
    ; ... 函数体 ...
    pop     edi
    pop     esi
    pop     ebp
    ret     0Ch               ; ← 弹出 12 字节
sub_405000 endp
```

<details>
<summary>参考答案</summary>

- `ret 0Ch` = `ret 12` → 被调者清理 12 字节 = 3 个 dword 参数
- 调用约定：stdcall（非 thiscall，因为参数从 `[ebp+8]` 开始，不是 ECX）
- 函数原型：`int __stdcall sub_405000(int param1, int param2, int param3);`
- 保存了 ESI 和 EDI（callee-saved 寄存器），说明函数体中使用了它们

</details>

### 练习 3：x64 调用约定参数映射

将以下 C 函数调用翻译为 Microsoft x64 反汇编：

```c
int result = calculate(10, 20, 30, 40, 50, 60);
```

<details>
<summary>参考答案</summary>

```nasm
; Microsoft x64 调用约定
sub     rsp, 48h             ; 影子空间(32) + 参数5-6(16) + 对齐
mov     dword ptr [rsp+28h], 60  ; 参数 6 → 栈 [RSP+40]
mov     dword ptr [rsp+20h], 50  ; 参数 5 → 栈 [RSP+32]
mov     r9d, 40              ; 参数 4 → R9
mov     r8d, 30              ; 参数 3 → R8
mov     edx, 20              ; 参数 2 → RDX
mov     ecx, 10              ; 参数 1 → RCX
call    calculate
; RAX = result
add     rsp, 48h             ; 调用者清栈
```

注意：
- 影子空间 32 字节必须预留，即使只有 ≤4 个参数
- 参数 5、6 在 `[RSP+20h]`、`[RSP+28h]`（影子空间之后）
- 总栈分配需要 16 字节对齐

</details>

### 练习 4：用 C++ 编译验证

编写测试代码，编译后用 `dumpbin` 观察不同调用约定的反汇编差异：

```cpp
// calling_conv_test.cpp
#include <cstdio>

// 显式指定不同调用约定
int __cdecl    func_cdecl(int a, int b, int c)    { return a + b + c; }
int __stdcall  func_stdcall(int a, int b, int c)   { return a + b + c; }
int __fastcall func_fastcall(int a, int b, int c)  { return a + b + c; }

class MyClass {
public:
    int value;
    int __thiscall GetSum(int a, int b) { return value + a + b; }
};

int main() {
    MyClass obj;
    obj.value = 10;

    int r1 = func_cdecl(1, 2, 3);
    int r2 = func_stdcall(1, 2, 3);
    int r3 = func_fastcall(1, 2, 3);
    int r4 = obj.GetSum(1, 2);

    printf("%d %d %d %d\n", r1, r2, r3, r4);
    return 0;
}
```

```bash
# 编译为 32 位 (禁用优化以保留调用约定细节)
cl /Od /Fa calling_conv_test.cpp /link /MACHINE:X86

# 查看生成的汇编
type calling_conv_test.asm

# 或直接反汇编
dumpbin /disasm calling_conv_test.obj
```

<details>
<summary>关键输出差异</summary>

| 函数 | 修饰名 | 参数传递 | RET 指令 |
|------|--------|---------|---------|
| func_cdecl | `_func_cdecl` | 全部栈 | `ret` |
| func_stdcall | `_func_stdcall@12` | 全部栈 | `ret 0Ch` |
| func_fastcall | `@func_fastcall@12` | ECX,EDX+栈 | `ret 4` |
| GetSum | `?GetSum@MyClass@@QAEHKK@Z` | ECX=this,栈 | `ret 8` |

注意 fastcall 的 `ret 4`：只清理第 3 个参数（栈上 1 个），前 2 个通过寄存器传递不占栈空间。

</details>

## 8. 常见错误与解决方案

### 错误 1：误判 cdecl 为 stdcall

```nasm
; 编译器优化：延迟栈清理
push    arg1
call    func_cdecl_A
; ← 这里没有 add esp！
push    arg2
call    func_cdecl_B
add     esp, 8           ; 一次性清理两次调用的参数

; 新手误判: func_cdecl_A 是 stdcall（因为后面没有 add esp）
; 实际: 优化器合并了栈清理

; 解决方案: 检查函数内部的 RET 指令
; stdcall: ret N (N > 0)
; cdecl:   ret   (无参数)
```

### 错误 2：thiscall 中遗漏 this 参数

```nasm
mov     ecx, [ebp+obj]   ; this → ECX (隐含参数!)
push    3                 ; 显式参数 2
push    1                 ; 显式参数 1
call    sub_401000

; 错误: "这个函数有 2 个参数"
; 正确: "这个函数有 3 个参数: this(ECX) + 2个栈参数"
; 原型: int __thiscall sub_401000(void* this, int a, int b)
```

### 错误 3：x64 影子空间计算失误

```nasm
; 常见的 x64 栈帧大小误算
sub     rsp, 28h         ; 0x28 = 40 字节

; 分解:
; 影子空间: 32 字节 (0x20) — 用于 4 个寄存器参数保存
; 对齐填充: 8 字节 — 确保 RSP 16 字节对齐
; (call 压入 8 字节返回地址 + 8 字节对齐 = 16 字节对齐)

; ⚠ 不要把影子空间当作"参数"计算在内
; 0x28 字节栈分配 ≠ 5 个参数！
```

## 本节小结

| 概念 | 要点 |
|------|------|
| cdecl | 调用者清栈，`add esp, N`，支持可变参数 |
| stdcall | 被调者清栈，`ret N`，Windows API 标准 |
| thiscall | ECX = this，其余同 stdcall |
| fastcall | ECX + EDX 传前两个参数 |
| MS x64 | RCX/RDX/R8/R9 + 32 字节影子空间 |
| SysV x64 | RDI/RSI/RDX/RCX/R8/R9，无影子空间 |
| 识别技巧 | 看 `add esp` → cdecl；看 `ret N` → stdcall/fastcall |
| KiriKiri2 | thiscall (C++方法) + stdcall (插件导出) 混合 |

## 练习题与答案

### 题目 1

KiriKiri2 插件的 `V2Link` 函数使用什么调用约定？如何从反汇编中确认？

<details>
<summary>答案</summary>

`V2Link` 使用 `__stdcall`。确认方法：
1. 导出名修饰：`_V2Link@4`（`@4` 表示 1 个 4 字节参数）
2. 函数尾部：`ret 4`（被调者弹出 4 字节）
3. 调用处无 `add esp`

</details>

### 题目 2

为什么 C 标准库函数（如 `printf`）不能使用 stdcall？

<details>
<summary>答案</summary>

`printf` 是可变参数函数（variadic function）。stdcall 要求被调者清栈，但被调者在编译时不知道调用者传了多少个参数，无法确定 `ret N` 中的 N。只有 cdecl（调用者清栈）支持可变参数，因为调用者知道自己压了多少个参数。

</details>

### 题目 3

在 x64 (Microsoft) 下，以下函数调用中参数分别在哪里？

```cpp
void func(int a, double b, int c, double d, int e);
func(1, 2.0, 3, 4.0, 5);
```

<details>
<summary>答案</summary>

Microsoft x64 中，参数位置由**参数序号**决定，不是由类型决定：

| 参数 | 类型 | 位置 | 原因 |
|------|------|------|------|
| a=1 | int | RCX | 第 1 个参数 → 整数用 RCX |
| b=2.0 | double | XMM1 | 第 2 个参数 → 浮点用 XMM1 |
| c=3 | int | R8 | 第 3 个参数 → 整数用 R8 |
| d=4.0 | double | XMM3 | 第 4 个参数 → 浮点用 XMM3 |
| e=5 | int | [RSP+20h] | 第 5 个参数 → 栈 |

关键：整数和浮点参数**共享**位置编号。第 2 个参数是 double → XMM1（不是 XMM0！因为 XMM0 对应位置 1，已被跳过）。

</details>
