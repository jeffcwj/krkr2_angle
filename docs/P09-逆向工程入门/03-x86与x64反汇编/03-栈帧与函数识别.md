# 栈帧与函数识别

## 本节目标

- 理解 x86 和 x64 函数的栈帧结构（序言/尾声）
- 掌握局部变量和参数在栈帧中的布局方式
- 理解栈帧指针省略（FPO）及其对逆向分析的影响
- 能够从反汇编中识别函数边界、参数个数、局部变量
- 结合 KiriKiri2 实际代码练习完整的函数还原

## 1. 栈帧基础

### 1.1 什么是栈帧

每次函数调用都会在栈上创建一个「栈帧」（Stack Frame），包含：

```
高地址 (栈底方向)
┌──────────────────────────────┐
│ 调用者的栈帧                  │
├──────────────────────────────┤
│ 参数 N                       │ [EBP+8+4*(N-1)]
│ ...                          │
│ 参数 2                       │ [EBP+0Ch]
│ 参数 1                       │ [EBP+08h]
├──────────────────────────────┤
│ 返回地址 (4/8 字节)           │ [EBP+04h]  ← CALL 指令压入
├──────────────────────────────┤
│ 旧 EBP (保存的帧指针)         │ [EBP]      ← 当前 EBP 指向这里
├──────────────────────────────┤
│ 局部变量 1                    │ [EBP-04h]
│ 局部变量 2                    │ [EBP-08h]
│ ...                          │
│ 保存的寄存器 (ESI, EDI 等)     │
├──────────────────────────────┤
│ (对齐填充)                    │
├──────────────────────────────┤ ← ESP (栈顶)
│ 传给子函数的参数               │
└──────────────────────────────┘
低地址 (栈顶方向)
```

### 1.2 栈增长方向

```
x86/x64 栈向低地址增长:
- PUSH: ESP -= 4(或8), 然后写入 [ESP]
- POP:  读取 [ESP], 然后 ESP += 4(或8)
- 局部变量: EBP - offset (负偏移)
- 函数参数: EBP + offset (正偏移)
```

## 2. 函数序言与尾声

### 2.1 标准序言（Prologue）

几乎所有非优化编译的函数都以相同的三条指令开头：

```nasm
; 标准序言 — x86
push    ebp              ; ① 保存调用者的帧指针
mov     ebp, esp         ; ② 建立新帧指针
sub     esp, 20h         ; ③ 分配局部变量空间 (0x20 = 32 字节)

; 可选：保存 callee-saved 寄存器
push    esi
push    edi
push    ebx
```

**序言执行过程图解**：

```
执行前 (调用者视角):          执行 CALL 后:           序言完成后:
┌────────────┐               ┌────────────┐          ┌────────────┐
│ 参数 2     │               │ 参数 2     │          │ 参数 2     │
│ 参数 1     │               │ 参数 1     │          │ 参数 1     │
│            │ ← ESP         │ 返回地址   │          │ 返回地址   │
└────────────┘               │            │ ← ESP    │ 旧 EBP     │ ← EBP
                             └────────────┘          │ 局部变量   │
                                                     │ ...        │ ← ESP
                                                     └────────────┘
```

### 2.2 标准尾声（Epilogue）

```nasm
; 标准尾声 — x86
; 方式 1: 显式恢复
pop     ebx              ; 恢复 callee-saved 寄存器
pop     edi
pop     esi
mov     esp, ebp         ; 释放局部变量空间
pop     ebp              ; 恢复调用者帧指针
ret                      ; 返回 (cdecl)
ret     8                ; 返回并清理参数 (stdcall, 2个参数)

; 方式 2: 使用 LEAVE 指令
leave                    ; 等效 mov esp, ebp; pop ebp
ret
```

### 2.3 x64 序言与尾声

x64 下栈帧结构有所不同（通常不使用 RBP 作为帧指针）：

```nasm
; x64 标准序言 (Microsoft)
sub     rsp, 48h         ; 分配栈空间 (局部变量 + 影子空间 + 对齐)
mov     [rsp+48h+var_s0], rcx  ; 保存参数到影子空间
mov     [rsp+48h+var_s1], rdx

; 如果使用 RBP:
push    rbp
lea     rbp, [rsp+20h]   ; RBP 不一定指向栈帧顶部！
sub     rsp, 40h

; x64 标准尾声
add     rsp, 48h          ; 或 lea rsp, [rbp-20h]
ret
```

**关键差异 x86 vs x64**:

```
┌──────────────────┬─────────────────┬──────────────────┐
│                  │ x86             │ x64              │
├──────────────────┼─────────────────┼──────────────────┤
│ 帧指针          │ EBP (常用)       │ RSP (通常省略RBP)│
│ 参数位置        │ [EBP+8],[EBP+C] │ 寄存器RCX,RDX..  │
│ 局部变量定位    │ [EBP-N]          │ [RSP+N]          │
│ 栈对齐          │ 4 字节           │ 16 字节          │
│ 序言固定模式    │ push ebp;mov..   │ sub rsp, N       │
└──────────────────┴─────────────────┴──────────────────┘
```

## 3. 局部变量识别

### 3.1 通过 EBP 偏移识别变量

在使用标准帧指针的函数中，变量通过 EBP 偏移直接定位：

```nasm
; int calculate(int x, int y) {
;     int sum = x + y;
;     int product = x * y;
;     return sum + product;
; }

push    ebp
mov     ebp, esp
sub     esp, 8              ; 2 个局部变量 × 4 字节

; 参数:
;   x = [ebp+8]     (参数 1)
;   y = [ebp+0Ch]   (参数 2)
; 局部变量:
;   sum     = [ebp-4]
;   product = [ebp-8]

mov     eax, [ebp+8]        ; eax = x
add     eax, [ebp+0Ch]      ; eax = x + y
mov     [ebp-4], eax         ; sum = x + y

mov     eax, [ebp+8]        ; eax = x
imul    eax, [ebp+0Ch]      ; eax = x * y
mov     [ebp-8], eax         ; product = x * y

mov     eax, [ebp-4]        ; eax = sum
add     eax, [ebp-8]        ; eax = sum + product
; return eax

mov     esp, ebp
pop     ebp
ret
```

**IDA 的自动命名**：

```nasm
; IDA 自动为栈变量命名:
var_8   = dword ptr -8       ; product (局部变量)
var_4   = dword ptr -4       ; sum (局部变量)
arg_0   = dword ptr  8       ; x (参数 1)
arg_4   = dword ptr  0Ch     ; y (参数 2)

; 使用命名后:
mov     eax, [ebp+arg_0]    ; 更易读
add     eax, [ebp+arg_4]
mov     [ebp+var_4], eax
```

### 3.2 变量类型推断

通过访问模式推断变量的大小和类型：

```nasm
; 1. 通过操作数大小推断
mov     byte ptr [ebp-1], 0      ; char (1 字节)
mov     word ptr [ebp-4], 0      ; short (2 字节)
mov     dword ptr [ebp-8], 0     ; int (4 字节)
mov     qword ptr [ebp-10h], 0   ; __int64 (8 字节, x64)

; 2. 通过指令推断
fld     qword ptr [ebp-10h]      ; double (8 字节浮点)
fld     dword ptr [ebp-14h]      ; float (4 字节浮点)
movss   xmm0, [ebp-14h]         ; float (SSE)
movsd   xmm0, [ebp-10h]         ; double (SSE2)

; 3. 通过使用上下文推断
lea     ecx, [ebp-20h]          ; 取地址 → 可能是结构体/数组
push    ecx                      ; 传递地址给函数
call    sub_401000               ; 可能是 init_struct(&local)

; 4. 通过 MOVZX/MOVSX 推断
movzx   eax, byte ptr [ebp-1]   ; unsigned char
movsx   eax, byte ptr [ebp-1]   ; signed char
movzx   eax, word ptr [ebp-4]   ; unsigned short
```

### 3.3 结构体与数组识别

```nasm
; 结构体访问模式:
; 基址 + 不同的固定偏移 → 结构体成员
mov     eax, [esi]         ; member_0 (可能是 vtable)
mov     ecx, [esi+4]       ; member_4
mov     edx, [esi+8]       ; member_8
mov     ebx, [esi+0Ch]     ; member_C

; 推断结构体:
; struct S {
;     void* vtable;     // +0
;     int   field1;     // +4
;     int   field2;     // +8
;     int   field3;     // +C
; };

; 数组访问模式:
; 基址 + 变量索引 × 元素大小 → 数组元素
mov     eax, [esi+ecx*4]       ; int array[n] → arr[ecx]
movzx   eax, word ptr [esi+ecx*2]  ; short array[n] → arr[ecx]
mov     al, [esi+ecx]         ; char array[n] → arr[ecx]

; 二维数组 / 结构体数组:
; arr[i].field 其中 sizeof(struct) = 0x14
imul    ecx, [ebp+index], 14h  ; ecx = index * 0x14
mov     eax, [esi+ecx+8]      ; arr[index].field_at_8
```

## 4. 栈帧指针省略（FPO）

### 4.1 什么是 FPO

FPO（Frame Pointer Omission）是一种编译器优化，不使用 EBP 作为帧指针，而是直接用 ESP 定位栈上数据：

```nasm
; 标准栈帧（有 EBP）:
push    ebp
mov     ebp, esp
sub     esp, 10h
mov     eax, [ebp+8]     ; 参数通过 EBP 访问
mov     [ebp-4], eax      ; 局部变量通过 EBP 访问

; FPO 栈帧（无 EBP，EBP 当普通寄存器用）:
sub     esp, 10h
mov     eax, [esp+14h]   ; 参数通过 ESP 访问
mov     [esp+0Ch], eax    ; 局部变量通过 ESP 访问
; EBP 可以当作普通寄存器使用，多一个可用寄存器
```

### 4.2 FPO 对逆向分析的影响

```
┌──────────────────────────────────────────────────────────┐
│                   FPO 的逆向影响                          │
├────────────────┬─────────────────────────────────────────┤
│ 正常帧         │ FPO 帧                                  │
├────────────────┼─────────────────────────────────────────┤
│ 参数: [EBP+8]  │ 参数: [ESP+N] (N 随 push/pop 变化!)    │
│ 变量: [EBP-4]  │ 变量: [ESP+M] (M 也会变化!)            │
│ 偏移固定不变   │ 每次 PUSH/POP 后偏移改变               │
│ 易于分析       │ 需要追踪 ESP 变化历史                    │
│ 回溯简单       │ 需要 .pdb 或 FPO 数据才能回溯           │
└────────────────┴─────────────────────────────────────────┘
```

**FPO 函数中 ESP 偏移的变化追踪**：

```nasm
; FPO 函数示例
sub     esp, 10h              ; ESP -= 0x10
; 此时 参数1 在 [ESP+14h] (0x10 + 返回地址4)

push    esi                   ; ESP -= 4
; 此时 参数1 在 [ESP+18h] (多了 4)

push    edi                   ; ESP -= 4
; 此时 参数1 在 [ESP+1Ch] (又多了 4)

mov     eax, [esp+1Ch]        ; 参数1

call    SomeFunc              ; CALL 压入返回地址
; 在 SomeFunc 内部，回溯困难——因为不知道调用者分配了多少栈空间
```

> **IDA 的处理**：IDA 会自动追踪 ESP 变化并调整偏移显示。在 IDA 中按 `Alt+K` 可查看每条指令处的 ESP delta。

### 4.3 MSVC 编译选项

```bash
# 启用 FPO（Release 默认）
cl /Oy /O2 source.c

# 禁用 FPO（Debug 默认，推荐逆向练习时使用）
cl /Oy- /Od source.c

# 生成 FPO 调试信息
cl /Oy /O2 /Zi source.c
# 链接时包含 FPO 数据到 PDB:
link /DEBUG source.obj
```

## 5. 函数边界识别

### 5.1 函数开始的识别标志

```nasm
; 标志 1: 标准序言
push    ebp
mov     ebp, esp

; 标志 2: FPO 序言（看到大的 sub esp）
sub     esp, 28h          ; 函数开始

; 标志 3: 对齐填充后的函数起始
; 编译器在函数之间插入 NOP 或 INT 3 作为对齐填充:
db 90h                    ; NOP
db 90h                    ; NOP
db 0CCh                   ; INT 3 (MSVC 填充)
; ← 下一个函数从这里开始

; 标志 4: 导出名或调试符号标注的函数起始
; IDA: sub_401000 proc near
; Ghidra: FUN_00401000
```

### 5.2 函数结束的识别标志

```nasm
; 标志 1: RET 指令
ret                       ; cdecl
ret     0Ch               ; stdcall (清理 3 个参数)

; 标志 2: 多个出口
; 同一函数可能有多个 RET（多个 return 路径）
test    eax, eax
jz      loc_EarlyReturn
; ... 正常路径 ...
ret
loc_EarlyReturn:
xor     eax, eax
ret                       ; 第二个 RET

; 标志 3: 尾调用优化 (Tail Call)
; 函数末尾直接 JMP 到另一个函数而不是 CALL + RET
mov     esp, ebp
pop     ebp
jmp     sub_402000        ; ← 尾调用，不是函数内跳转
; 等效于: return sub_402000(...);
```

### 5.3 编译器生成的辅助代码

```nasm
; SEH (结构化异常处理) 序言 — MSVC x86
push    ebp
mov     ebp, esp
push    -1                 ; ← SEH 标志 (-1 = 初始状态)
push    offset seh_handler ; 异常处理函数
mov     eax, fs:[0]        ; 获取当前 SEH 链头
push    eax                ; 保存旧链头
mov     fs:[0], esp        ; 注册新 SEH 处理器
sub     esp, 20h           ; 局部变量

; SEH 尾声
mov     ecx, [ebp-0Ch]     ; 恢复旧 SEH 链头
mov     fs:[0], ecx
pop     ecx
pop     ecx
pop     ecx
mov     esp, ebp
pop     ebp
ret
```

> **识别 SEH 的关键**：看到 `push -1; push offset; mov eax, fs:[0]` 组合即可确认使用了 SEH。这在 KiriKiri2 中非常普遍，因为 TJS2 解释器使用异常处理做流程控制。

## 6. 完整函数还原实战

### 6.1 从反汇编到 C 代码 —— 逐步方法

```
还原步骤:
┌──────────────────────────────────────────────────────┐
│ 1. 识别函数边界（序言→尾声）                          │
│ 2. 确定调用约定（看 RET、add esp、ECX 使用）          │
│ 3. 统计参数（EBP+正偏移 或 寄存器参数）              │
│ 4. 统计局部变量（EBP-负偏移 或 ESP+偏移）            │
│ 5. 推断变量类型（操作数大小、使用模式）               │
│ 6. 识别控制流（if-else、循环、switch）               │
│ 7. 识别函数调用（CALL + 参数）                       │
│ 8. 组装成 C/C++ 代码                                │
└──────────────────────────────────────────────────────┘
```

### 6.2 实战示例：字符串长度计算

```nasm
; sub_401200 — 猜测功能是什么？
sub_401200 proc near
    arg_0 = dword ptr  8

    push    ebp                ; ① 序言
    mov     ebp, esp
    mov     eax, [ebp+arg_0]  ; ② eax = 参数1 (某指针)
    test    eax, eax           ; ③ if (eax == NULL)
    jz      short loc_ret_neg1

    xor     ecx, ecx           ; ④ ecx = 0 (计数器)

loc_loop:
    movzx   edx, word ptr [eax] ; ⑤ edx = *(wchar_t*)eax
    test    edx, edx            ; ⑥ if (*eax == 0)
    jz      short loc_done
    inc     ecx                 ; ⑦ count++
    add     eax, 2             ; ⑧ eax += 2 (下一个 wchar_t)
    jmp     short loc_loop

loc_done:
    mov     eax, ecx           ; ⑨ return count
    pop     ebp
    ret                        ; ⑩ cdecl

loc_ret_neg1:
    or      eax, 0FFFFFFFFh   ; ⑪ eax = -1
    pop     ebp
    ret
sub_401200 endp
```

**还原过程**:

```
步骤 1: 函数边界 — push ebp ... ret
步骤 2: 调用约定 — ret (无参数) + 无 add esp → cdecl
步骤 3: 参数 — [ebp+8] 一个参数
步骤 4: 局部变量 — 无 sub esp → 无局部变量（用寄存器）
步骤 5: 类型推断:
  - 参数: 指针（test eax, eax 检查 NULL）
  - word ptr [eax] + add eax, 2 → wchar_t*
步骤 6: 控制流 — NULL 检查 + while 循环
步骤 7: 无子函数调用
```

```c
// 还原结果:
int wcslen_custom(const wchar_t* str) {
    if (str == NULL)
        return -1;
    int len = 0;
    while (*str != 0) {
        len++;
        str++;
    }
    return len;
}
// 这是一个自定义的 wcslen 实现，带 NULL 检查
```

### 6.3 实战示例：KiriKiri2 风格的引用计数

```nasm
; 模拟 KiriKiri2 中 tTJSVariant 的 AddRef/Release 模式
sub_401400 proc near
    ; thiscall: ECX = this

    mov     eax, [ecx+4]       ; eax = this->refCount
    dec     eax                ; refCount--
    mov     [ecx+4], eax       ; 写回
    test    eax, eax           ; if (refCount == 0)
    jnz     short loc_ret

    ; refCount 降到 0，释放对象
    push    ecx                ; 保存 this (ECX 是 caller-saved)
    push    1                  ; 参数: 1 (标志)
    mov     eax, [ecx]        ; vtable
    call    dword ptr [eax+8]  ; this->Destroy(1) — 虚函数调用
    pop     ecx                ; 恢复 this
    push    ecx
    call    operator_delete    ; ::operator delete(this)
    add     esp, 4             ; cdecl 清栈

loc_ret:
    ret                        ; thiscall, 无显式参数
sub_401400 endp
```

<details>
<summary>还原结果</summary>

```cpp
// thiscall, 无显式参数 → 成员函数
void SomeRefCounted::Release() {
    if (--this->refCount == 0) {
        this->Destroy(1);       // 虚函数调用
        ::operator delete(this); // 释放内存
    }
}
```

**分析要点**：
- `[ecx+4]` → refCount 在对象偏移 4 处（偏移 0 是 vtable）
- `[ecx] → [eax+8]` → 虚函数表第 2 个函数（索引 2，偏移 8/4=2）
- `operator_delete` 使用 cdecl（有 `add esp, 4`）
- 这是典型的 COM 风格引用计数模式，KiriKiri2 的 `iTJSDispatch2` 接口使用类似机制

</details>

## 7. 动手实践

### 练习 1：栈帧布局绘制

给出以下函数的完整栈帧布局图：

```nasm
sub_405000 proc near
    push    ebp
    mov     ebp, esp
    sub     esp, 18h         ; 24 字节局部变量
    push    esi
    push    edi
    mov     esi, [ebp+8]     ; 参数 1
    mov     edi, [ebp+0Ch]   ; 参数 2
    mov     ecx, [ebp+10h]   ; 参数 3
    ; ... 函数体 ...
    pop     edi
    pop     esi
    mov     esp, ebp
    pop     ebp
    ret     0Ch              ; stdcall, 3 个参数
sub_405000 endp
```

<details>
<summary>参考答案</summary>

```
高地址
┌──────────────────┐
│ 参数 3 (arg_8)   │ [EBP+10h]
│ 参数 2 (arg_4)   │ [EBP+0Ch]
│ 参数 1 (arg_0)   │ [EBP+08h]
│ 返回地址          │ [EBP+04h]
│ 旧 EBP           │ [EBP+00h]  ← EBP
│ 局部变量[0]       │ [EBP-04h]
│ 局部变量[1]       │ [EBP-08h]
│ 局部变量[2]       │ [EBP-0Ch]
│ 局部变量[3]       │ [EBP-10h]
│ 局部变量[4]       │ [EBP-14h]
│ 局部变量[5]       │ [EBP-18h]   sub esp, 18h 分配 6 个 dword
│ 保存的 ESI       │             push esi
│ 保存的 EDI       │ ← ESP      push edi
低地址
```

- 调用约定：stdcall（`ret 0Ch`，3 个参数×4=12=0xC）
- 保存了 2 个 callee-saved 寄存器：ESI、EDI
- 24 字节局部空间 ÷ 4 = 最多 6 个 int 局部变量

</details>

### 练习 2：FPO 函数分析

分析以下 FPO（无帧指针）函数，追踪 ESP 变化：

```nasm
sub_406000 proc near
    sub     esp, 8           ; 分配 8 字节局部空间
    ; ESP delta = -8
    mov     [esp+4], ecx     ; 保存 this (thiscall)
    mov     eax, [esp+0Ch]   ; ← 这访问的是什么？

    push    esi              ; ESP delta = -12
    mov     esi, [esp+10h]   ; ← 这访问的是什么？

    ; ... 函数体 ...
    pop     esi              ; ESP delta = -8
    add     esp, 8           ; ESP delta = 0
    ret     4                ; 弹出 4 字节 (1 个栈参数)
sub_406000 endp
```

<details>
<summary>参考答案</summary>

ESP 追踪表：

| 指令后 | ESP 偏移 | [ESP+0Ch] 实际 | [ESP+10h] 实际 |
|--------|---------|---------------|---------------|
| call (进入函数) | 0 | — | — |
| sub esp, 8 | -8 | +4 处=参数1 | +8=参数2 |
| push esi | -12 | +0=参数1 | +4=? |

详细分析：
- 进入函数时：`[ESP]` = 返回地址，`[ESP+4]` = 参数 1
- `sub esp, 8` 后：`[ESP+0Ch]` = 返回地址上方 = 参数 1（8+4=0xC）
- `push esi` 后：`[ESP+10h]` = 参数 1（8+4+4=0x10）

所以 `[esp+0Ch]` 和 `[esp+10h]` 在各自的 ESP 上下文中都指向参数 1。

函数原型：`int __thiscall func(void* this, int param1)`
- this → ECX，保存到 `[esp+4]`
- param1 → 栈
- `ret 4` → 清理 1 个栈参数

</details>

### 练习 3：完整函数还原

将以下反汇编完整还原为 C++ 代码：

```nasm
; 假设这是 KiriKiri2 的某个方法
sub_407000 proc near
    push    ebp
    mov     ebp, esp
    sub     esp, 10h
    push    esi
    push    edi

    mov     esi, ecx              ; this
    mov     edi, [ebp+8]          ; 参数1: name (const wchar_t*)
    mov     [ebp-4], 0            ; result = 0

    ; 获取成员数量
    mov     eax, [esi+8]          ; count = this->count
    test    eax, eax
    jle     short loc_NotFound

    xor     ecx, ecx              ; i = 0
    mov     [ebp-8], ecx          ; 保存 i

loc_Loop:
    ; 获取第 i 个成员名
    mov     edx, [esi+0Ch]        ; names_array
    mov     eax, [ebp-8]          ; i
    mov     ecx, [edx+eax*4]     ; names[i]

    ; 比较名称
    push    edi                   ; 参数2: name
    push    ecx                   ; 参数1: names[i]
    call    _wcscmp
    add     esp, 8
    test    eax, eax
    jz      short loc_Found

    ; 递增
    mov     eax, [ebp-8]
    inc     eax
    mov     [ebp-8], eax
    cmp     eax, [esi+8]         ; i < count
    jl      short loc_Loop

loc_NotFound:
    mov     [ebp-4], 0FFFFFFFFh  ; result = -1
    jmp     short loc_Return

loc_Found:
    mov     eax, [ebp-8]
    mov     [ebp-4], eax          ; result = i

loc_Return:
    mov     eax, [ebp-4]          ; return result
    pop     edi
    pop     esi
    mov     esp, ebp
    pop     ebp
    ret     4                     ; thiscall, 1 个栈参数
sub_407000 endp
```

<details>
<summary>参考答案</summary>

```cpp
// thiscall + ret 4 = 成员函数，1个显式参数
// this (ECX) + name (栈)
int MyClass::FindMember(const wchar_t* name) {
    int result = 0;             // [ebp-4]

    // this->count 在 [esi+8]
    if (this->count <= 0)
        goto not_found;

    for (int i = 0; i < this->count; i++) {  // [ebp-8]
        // this->names 在 [esi+0Ch]，是 wchar_t** 数组
        if (wcscmp(this->names[i], name) == 0) {
            return i;           // 找到，返回索引
        }
    }

not_found:
    return -1;                  // 未找到
}

// 推断的类结构:
// struct MyClass {
//     void*          vtable;   // +0
//     ???            field_4;  // +4
//     int            count;    // +8
//     const wchar_t** names;   // +C
// };
```

</details>

## 8. 常见错误与解决方案

### 错误 1：混淆 EBP 正负偏移

```nasm
mov     eax, [ebp+8]      ; ← 正偏移 = 参数（返回地址之上）
mov     eax, [ebp-4]      ; ← 负偏移 = 局部变量

; 新手常犯: 把 [ebp-4] 当作参数
; 记忆: EBP 上方(+) = 调用者给的 = 参数
;        EBP 下方(-) = 自己分配的 = 局部变量
```

### 错误 2：FPO 函数中 ESP 偏移计算错误

```nasm
; 每条 PUSH/POP/SUB ESP/ADD ESP 都改变后续偏移
sub     esp, 8
; [esp+0Ch] → 参数 1
push    eax               ; ESP -= 4
; [esp+10h] → 同一个参数 1 (偏移 +4)
push    ebx               ; ESP -= 4
; [esp+14h] → 还是同一个参数 1 (再 +4)

; 解决方案: 画表格追踪每条指令后的 ESP delta
; IDA 的 Options → General → Stack pointer (开启 ESP 追踪显示)
```

### 错误 3：尾调用误判为函数内跳转

```nasm
; 看到 JMP 不一定是函数内跳转！
mov     esp, ebp
pop     ebp
jmp     sub_402000        ; ← 这是尾调用 (tail call)
; 不是跳转到函数内的某个标签！

; 判断方法:
; - JMP 目标是函数开头（有序言）→ 尾调用
; - JMP 前有尾声（pop ebp 等）→ 尾调用
; - JMP 目标在当前函数范围内 → 函数内跳转
```

## 本节小结

| 概念 | 要点 |
|------|------|
| 序言 | `push ebp; mov ebp, esp; sub esp, N` |
| 尾声 | `mov esp, ebp; pop ebp; ret` 或 `leave; ret` |
| 参数 | `[EBP+8]` 起，正偏移 |
| 局部变量 | `[EBP-4]` 起，负偏移 |
| FPO | 省略 EBP，用 ESP+变化偏移访问 |
| 变量类型 | 通过操作数大小和使用模式推断 |
| 函数边界 | 序言/NOP 填充/RET 指令 |
| SEH | `push -1; push handler; mov eax, fs:[0]` |
| 尾调用 | 尾声后 JMP 而非 CALL |

## 练习题与答案

### 题目 1

以下函数分配了多少字节的局部变量空间？有几个参数？什么调用约定？

```nasm
push    ebp
mov     ebp, esp
sub     esp, 24h
; ... 函数体 ...
mov     esp, ebp
pop     ebp
ret     10h
```

<details>
<summary>答案</summary>

- 局部变量空间：`0x24` = 36 字节
- 参数：`ret 10h` = `ret 16`，16 ÷ 4 = 4 个参数
- 调用约定：stdcall（被调者通过 `ret N` 清栈）

如果函数入口前有 `mov ecx, xxx`，可能是 thiscall（this + 3 个栈参数 = 4 个逻辑参数，但栈上只有 3 个，ret 应为 0Ch）。这里 `ret 10h` 清理 4 个栈参数，说明不是 thiscall。

</details>

### 题目 2

为什么 x64 函数常常没有 `push rbp; mov rbp, rsp` 序言？

<details>
<summary>答案</summary>

1. x64 有 16 个通用寄存器（比 x86 多 8 个），不需要释放 RBP 当普通寄存器
2. x64 的 `.pdata` 和 `.xdata` 节包含所有函数的展开信息（Unwind Info），调试器和异常处理不依赖帧指针链
3. x64 使用基于表的异常处理（Table-Based SEH），不需要 `fs:[0]` 链
4. 编译器默认使用 RSP 相对寻址，因为 x64 的 `sub rsp` 在函数入口只执行一次，之后 RSP 不变（不像 x86 需要频繁 PUSH 参数）

但部分函数仍可能使用 RBP（如 alloca、VLA 等动态栈分配场景）。

</details>

### 题目 3

画出以下 x64 函数的栈帧布局：

```nasm
push    rbp
push    rsi
push    rdi
sub     rsp, 30h
lea     rbp, [rsp+20h]
; RCX = param1, RDX = param2
mov     [rbp+10h], rcx    ; 保存 param1
mov     [rbp+18h], rdx    ; 保存 param2
```

<details>
<summary>答案</summary>

```
栈帧布局 (高地址在上):
┌──────────────────────┐
│ param2 影子空间       │ [RSP_entry + 10h] = [RBP+18h]
│ param1 影子空间       │ [RSP_entry + 08h] = [RBP+10h]
│ 返回地址             │ [RSP_entry]
│ 保存的 RBP           │ [RSP_entry - 08h]
│ 保存的 RSI           │ [RSP_entry - 10h]
│ 保存的 RDI           │ [RSP_entry - 18h]
│ 局部变量区 (0x30)     │ sub rsp, 30h
│ ...                  │
│                      │ ← RSP (最终)
│        +20h          │
│                      │ ← RBP = RSP + 0x20
└──────────────────────┘

RBP 指向 RSP+0x20 位置，
所以 [RBP+10h] = [RSP+30h] = push 区上方 = 影子空间中的 param1
[RBP+18h] = [RSP+38h] = 影子空间中的 param2
```

</details>
