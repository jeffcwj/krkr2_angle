## 4.3.1 x64dbg 概述与安装

### 什么是 x64dbg

x64dbg 是一款开源的 Windows 用户态调试器，支持 x86（32 位）和 x64（64 位）程序。它是 OllyDbg 的精神继承者，但提供了更现代的界面和更强的功能：

| 特性 | OllyDbg | x64dbg |
|------|---------|--------|
| 架构支持 | 仅 x86 | x86 + x64 |
| 开源 | 否 | 是（GPLv3） |
| 插件系统 | 有（旧式 SDK） | 有（现代 SDK + Qt） |
| 反汇编引擎 | 自研 | Zydis（高质量） |
| 脚本支持 | 自有脚本 | 命令 + 脚本 + Python |
| 维护状态 | 已停止 | 活跃更新 |
| 符号支持 | 有限 | PDB + DWARF |

> **为什么选择 x64dbg**：KiriKiri2 是 32 位程序，用 x32dbg（x64dbg 的 32 位版本）调试。
> x64dbg 免费、开源、社区活跃，是逆向工程入门的最佳动态调试工具。

### 安装与配置

从 [GitHub Releases](https://github.com/x64dbg/x64dbg/releases) 下载最新快照版：

```

├── x32\
│   ├── x32dbg.exe        # 32 位调试器
│   ├── x32bridge.dll

├── x64\
```

## 4.3.3 断点系统详解

断点是动态调试的核心。x64dbg 支持四种断点类型，各有适用场景：

### 软件断点（INT3）

**原理**：将目标地址的第一个字节替换为 `0xCC`（INT 3 指令），CPU 执行到此触发异常，调试器捕获：

```
原始代码:  55           push ebp         ; 原始字节 0x55
设断点后:  CC           int 3            ; 替换为 0xCC
触发断点:  调试器捕获 → 恢复 0x55 → 暂停
```

**设置方式**：
```
# 方式 1：在反汇编窗口按 F2
# 方式 2：命令窗口
bp 00401000              # 在指定地址设断点
bp CreateFileW           # 在 API 入口设断点
bp kernel32.CreateFileW  # 带模块名的 API 断点

# 删除断点
bc 00401000              # 删除指定断点
bpc                      # 删除所有断点
```

**优点**：数量不限、不影响性能
**缺点**：修改了代码字节，可被反调试检测

### 硬件断点

**原理**：使用 CPU 的调试寄存器（DR0-DR3），最多同时设 4 个：

```
DR0-DR3: 断点地址（最多 4 个）
DR6:     调试状态寄存器
DR7:     调试控制寄存器
         ├─ 每个断点的类型: 执行/写入/读写
         └─ 每个断点的长度: 1/2/4 字节
```

**设置方式**：
```
# 命令窗口
bph 00401000, x          # 执行断点
bph 0045A3B0, w, 4       # 写入断点（4 字节）
bph 0045A3B0, r, 4       # 读写断点（4 字节）

# 或在反汇编窗口右键 → Breakpoint → Set Hardware on Execution
```

**应用场景**：
```nasm
; 场景：追踪某个全局变量何时被修改
; 假设 KiriKiri2 的 TVPMainWindow 指针在 0x004A2340
; 设置硬件写入断点：
bph 004A2340, w, 4
; 当程序写入该地址时，调试器暂停
; 查看调用堆栈即可找到修改者
```

**优点**：不修改代码，不可被常规反调试检测
**缺点**：最多 4 个

### 内存断点

**原理**：修改内存页的保护属性，触发访问异常：

```
; 将目标页设为 PAGE_NOACCESS 或 PAGE_GUARD
; 任何访问该页的操作都触发异常
; 调试器捕获异常 → 检查是否命中断点地址

# 设置方式
bpm 004A0000, 0, r       # 读取断点
bpm 004A0000, 0, w       # 写入断点
bpm 004A0000, 0, x       # 执行断点
```

**优点**：可以监控大范围内存
**缺点**：以页（4KB）为粒度，可能频繁触发

### 条件断点

条件断点在满足特定表达式时才暂停，是高级调试的关键：

```
# 基本条件断点
bp CreateFileW
# 右键断点 → Edit → Break condition:
# 当第一个参数包含 ".xp3" 时才断下

# x64dbg 条件表达式语法
# 寄存器: eax, ecx, edx, ...
# 内存:   [address] 或 dword:[address]
# 栈参数: [esp+4] (第1个参数), [esp+8] (第2个参数)

# 示例：CreateFileW 的第一个参数是文件名（wchar_t*）
# 条件：当打开的文件名包含 "xp3" 时断下
bp CreateFileW
# 设置条件（在断点属性中）:
# Break condition: utf16(arg.get(0)) == "data.xp3"
```

**实用条件断点示例**：

```
# 场景 1：只在特定调用次数时断下
bp 00401000
# Hit count condition: 断点命中第 100 次时暂停

# 场景 2：寄存器值满足条件
bp 00401000
# Break condition: eax > 0x1000 && ecx != 0

# 场景 3：日志断点（不暂停，只记录）
bp CreateFileW
# Log text: "CreateFileW({utf16@[esp+4]})"
# Break condition: 0    ← 永不断下，只记录
```

## 4.3.4 调试操作详解

### 执行控制

| 操作 | 快捷键 | 说明 |
|------|--------|------|
| 运行 | `F9` | 继续执行直到下一个断点 |
| 单步步入 | `F7` | 执行一条指令，遇 CALL 进入 |
| 单步步过 | `F8` | 执行一条指令，遇 CALL 跳过 |
| 运行到返回 | `Ctrl+F9` | 执行到当前函数的 RET |
| 运行到光标 | `F4` | 执行到光标所在行 |
| 运行到用户代码 | `Alt+F9` | 跳过系统 DLL，回到用户模块 |
| 暂停 | `F12` | 暂停正在运行的程序 |

### 单步调试实战流程

```nasm
; 假设我们在 KiriKiri2 的插件加载函数中断下
; 函数原型: void LoadPlugin(const wchar_t* path)

004523A0  push ebp              ; ← 当前位置（EIP）
004523A1  mov ebp, esp
004523A3  sub esp, 0x20
004523A6  push esi
004523A7  push edi
004523A8  mov esi, [ebp+8]      ; esi = path 参数
004523AB  push esi
004523AC  call LoadLibraryW     ; 加载 DLL
004523B1  test eax, eax         ; 检查返回值
004523B3  jz short 004523E0     ; 失败则跳转
004523B5  mov edi, eax          ; edi = DLL 句柄
004523B7  push offset "V2Link"  ; 函数名
004523BC  push edi              ; DLL 句柄
004523BD  call GetProcAddress   ; 获取 V2Link 地址
004523C2  test eax, eax
004523C4  jz short 004523D0     ; 没找到 V2Link
004523C6  push edi              ; 传入 DLL 句柄
004523C7  call eax              ; 调用 V2Link！
```

**调试步骤**：

1. **F8**（步过）到 `004523A8`，查看 ESI 寄存器 → 得到 path 字符串
2. **F8** 到 `004523AC`，此时 `[ESP]` = path → 即 LoadLibraryW 的参数
3. **F8** 步过 `call LoadLibraryW` → 查看 EAX = DLL 基址（或 0 表示失败）
4. **F8** 到 `004523BD`，查看 `[ESP]` = DLL 句柄，`[ESP+4]` = "V2Link"
5. **F8** 步过 `call GetProcAddress` → EAX = V2Link 函数地址
6. **F7**（步入）`call eax` → 进入插件的 V2Link 函数！

### 追踪调用链

```
# 使用 "运行到返回"（Ctrl+F9）快速跳出深层调用
# 假设当前在 ntdll!RtlAllocateHeap 内部

Ctrl+F9  →  返回到 kernel32!HeapAlloc
Ctrl+F9  →  返回到 msvcrt!malloc
Ctrl+F9  →  返回到 KiriKiri!operator_new
Ctrl+F9  →  返回到 KiriKiri!TVPCreatePlugin  ← 目标层
```

## 4.3.5 内存分析技术

### 内存映射（Alt+M）

```
┌──────────┬──────────┬──────────┬─────────────┬──────────────────┐
│ 基址     │ 大小     │ 保护属性  │ 类型        │ 模块             │
├──────────┼──────────┼──────────┼─────────────┼──────────────────┤
│ 00400000 │ 00001000 │ R        │ IMG         │ krkr.exe         │
│ 00401000 │ 00234000 │ R E      │ IMG         │ krkr.exe (.text) │
│ 00635000 │ 00089000 │ R        │ IMG         │ krkr.exe (.rdata)│
│ 006BE000 │ 0001A000 │ RW       │ IMG         │ krkr.exe (.data) │
│ 10000000 │ 00001000 │ R        │ IMG         │ extrans.dll      │
│ 10001000 │ 00045000 │ R E      │ IMG         │ extrans.dll      │
│ 7C800000 │ 00001000 │ R        │ IMG         │ kernel32.dll     │
└──────────┴──────────┴──────────┴─────────────┴──────────────────┘
```

**保护属性含义**：
- `R` = 可读，`W` = 可写，`E` = 可执行
- `R E` = 代码段（.text），`RW` = 数据段（.data/.bss）
- `R` = 只读数据段（.rdata，包含字符串常量、虚函数表）

### 内存搜索

```
# 搜索 ASCII 字符串
# 右键内存转储 → Search for → Pattern
# 或命令：
findall 00400000, "KiriKiri"    # 在模块中搜索字符串

# 搜索 Unicode 字符串（KiriKiri2 大量使用宽字符）
findall 00400000, #4B00690072006900#  # "Kiri" 的 UTF-16LE

# 搜索字节模式（特征码）
findall 00401000, "55 8B EC 83 EC ?? 53 56 57"
# 匹配: push ebp; mov ebp,esp; sub esp,??; push ebx; push esi; push edi
```

### 内存修改

```
# 在内存转储窗口中直接编辑
# 选中字节 → Ctrl+E → 输入新值

# 命令方式
mov [004A2340], 1              # 写入 DWORD
fill 004A2340, 10, 00          # 用 0x00 填充 16 字节
```

## 4.3.6 API 断点追踪程序行为

API 断点是动态分析最强大的技术之一。通过在 Windows API 入口设断点，可以监控程序的所有系统交互：

### 文件操作追踪

```
# 监控文件打开
bp CreateFileW
# 断下时查看参数：
#   [esp+4]  = lpFileName (wchar_t*)
#   [esp+8]  = dwDesiredAccess
#   [esp+C]  = dwShareMode
#   [esp+14] = dwCreationDisposition

# 监控文件读取
bp ReadFile
# 断下时查看：
#   [esp+4]  = hFile (句柄)
#   [esp+8]  = lpBuffer (读取目标地址)
#   [esp+C]  = nNumberOfBytesToRead

# 实际案例：追踪 KiriKiri2 读取 XP3 存档
bp CreateFileW
# 日志断点设置：
# Log text: "Open: {utf16@[esp+4]}"
# Break condition: 0
# 运行后查看日志输出：
#   Open: C:\game\data.xp3
#   Open: C:\game\patch.xp3
#   Open: C:\game\system\Config.tjs
```

### 注册表操作追踪

```
# KiriKiri2 使用注册表存储配置
bp RegOpenKeyExW
bp RegQueryValueExW
bp RegSetValueExW

# 典型输出：
#   RegOpenKeyExW(HKCU, "Software\krkr2\game_title")
#   RegQueryValueExW(..., "screenWidth", ...)
```

### 内存分配追踪

```
# 追踪大块内存分配（可能是解密/解压缩缓冲区）
bp VirtualAlloc
# 条件：只在分配大于 1MB 时断下
# Break condition: [esp+8] > 0x100000

# 断下时记录：
#   EAX = 分配的内存地址
#   [esp+8] = 大小
#   [esp+C] = 保护属性
```

### DLL 加载追踪

```
# 追踪 KiriKiri2 的插件加载
bp LoadLibraryW
# Log text: "LoadLibrary: {utf16@[esp+4]}"

# 追踪插件函数解析
bp GetProcAddress
# Log text: "GetProcAddress({s:utf8@[esp+8]}) from {modname([esp+4])}"

# 典型输出（KiriKiri2 加载插件）：
#   LoadLibrary: C:\game\plugin\extrans.dll
```

## 4.3.7 x64dbg 脚本与自动化

### 内置命令脚本

x64dbg 支持脚本文件（`.txt`），实现自动化调试：

```
// script: trace_xp3_load.txt
// 功能：自动记录 KiriKiri2 的 XP3 文件加载

// 设置日志断点
bp CreateFileW
SetBreakpointLog CreateFileW, "Open: {utf16@[esp+4]}"
SetBreakpointCondition CreateFileW, 0  // 不暂停

bp ReadFile
SetBreakpointLog ReadFile, "Read: handle={[esp+4]}, size={[esp+C]}"
SetBreakpointCondition ReadFile, 0

// 设置 LoadLibrary 断点（暂停）
bp LoadLibraryW

// 运行
run

// 执行：File → Script → Run Script
```

### 条件日志脚本

```
// script: monitor_plugin_init.txt
// 监控所有插件的 V2Link 调用

// 在 GetProcAddress 设断点
bp GetProcAddress

:loop
run

// 检查是否在找 "V2Link"
cmp [esp+8], "V2Link"
jne loop

// 找到 V2Link！记录信息
log "Found V2Link in module at {[esp+4]}"

// 步过 GetProcAddress 获取返回值
sto
log "V2Link address = {eax}"

// 在 V2Link 入口设断点
bp eax
jmp loop
```

### Python 脚本（通过 x64dbgpy 插件）

```python
# x64dbgpy 脚本：导出 KiriKiri2 的虚函数表
import x64dbg

def dump_vtable(addr, count=20):
    """导出指定地址的虚函数表"""
    result = []
    for i in range(count):
        func_addr = x64dbg.ReadDword(addr + i * 4)
        if func_addr == 0:
            break
        # 尝试获取符号名
        sym = x64dbg.GetSymbolAt(func_addr)
        name = sym if sym else f"sub_{func_addr:08X}"
        result.append(f"  vtable[{i:2d}] = 0x{func_addr:08X}  ; {name}")
    return "\n".join(result)

# 使用示例：导出 iTJSDispatch2 虚函数表
# 首先找到一个 iTJSDispatch2 对象
# 对象的第一个 DWORD 就是 vtable 指针
obj_addr = 0x0045A3B0  # 假设这是一个 TJS 对象
vtable_ptr = x64dbg.ReadDword(obj_addr)
print(f"Object at 0x{obj_addr:08X}, vtable at 0x{vtable_ptr:08X}")
print(dump_vtable(vtable_ptr))
```

## 4.3.8 KiriKiri2 调试实战

### 调试 KiriKiri2 的完整流程

```
Step 1: 启动调试
  File → Open → 选择 krkr.exe
  命令行参数: 不填（让它在当前目录找 data.xp3）

Step 2: 设置初始断点
  bp LoadLibraryW        # 监控 DLL 加载
  bp GetProcAddress      # 监控函数解析
  bp CreateFileW         # 监控文件访问

Step 3: 运行到 OEP（Original Entry Point）
  F9 运行
  # 如果有 TLS 回调，会先在 TLS 断下
  # 继续 F9 直到到达 krkr.exe 的代码段

Step 4: 找到 WinMain
  # 方法 1：在 Symbols 标签中搜索 "WinMain" 或 "main"
  # 方法 2：在 OEP 附近 F8 步过，找到调用 WinMain 的 CALL
  # 方法 3：对 GetCommandLineW 设断点，断下后查看调用栈
```

### 追踪插件加载

```
# 在 LoadLibraryW 断下后的分析流程

1. 查看 [ESP+4] 获取 DLL 路径
   "C:\game\plugin\extrans.dll"

2. F9 继续 → 在 GetProcAddress 断下
   [ESP+4] = extrans.dll 的基址
   [ESP+8] → "V2Link"

3. 步过(F8) GetProcAddress → EAX = V2Link 地址
   EAX = 0x10001234

4. 在 V2Link 入口设断点
   bp 10001234
   F9 → 断在 V2Link 入口

5. 分析 V2Link 内部
   F7 步入，观察它如何注册 TJS 类/函数
```

### 追踪 TJS 脚本执行

```
# TJS2 引擎的核心执行函数
# 通过字符串搜索找到关键函数

# 搜索 "TJS2" 相关字符串
findall krkr.exe, "TJS"

# 常见的关键函数（通过 IDA/Ghidra 分析确定地址）

bp 00423456    # tTJS::ExecScript

# 在 ExecScript 断下时：
# [ESP+4] 或 ECX = tTJS* this
```


## 4.3.10 常见错误与解决方案

### 错误 1：x32dbg 无法附加到正在运行的进程

```
症状: File → Attach 列表中找不到目标进程，或附加失败
原因: 权限不足或进程架构不匹配

解决方案:
1. 以管理员身份运行 x32dbg
2. 确认目标进程是 32 位（用 Task Manager → Details 查看）
3. 32 位进程用 x32dbg，64 位进程用 x64dbg
4. 检查是否有其他调试器已附加（一个进程只能有一个调试器）
```

### 错误 2：断点设在系统 DLL 中不生效

```
症状: 对 CreateFileW 设断点，程序运行后没有断下
原因: ASLR 导致 DLL 基址变化，或断点地址不正确

解决方案:
1. 使用符号名设断点: bp CreateFileW（而不是硬编码地址）
2. 确认模块已加载: 查看 Alt+E 模块列表
3. 如果模块延迟加载，先在 LoadLibraryW 处断下
4. 检查断点是否被禁用: Alt+B 查看断点列表
```

### 错误 3：条件断点表达式报错

```
症状: 设置条件断点后，弹出 "Invalid expression" 错误
原因: 条件表达式语法错误

解决方案:
# 错误写法:
eax == 0 and ecx != 0          # 'and' 不被识别

# 正确写法:
eax == 0 && ecx != 0           # 使用 C 风格运算符

# 错误写法:
[esp+4] == "hello"             # 不能直接比较字符串

# 正确写法:
dword:[esp+4] == 0x6C6C6568    # 比较内存中的 DWORD（"hell" 的 ASCII）

# 常用表达式:
eax > 0x1000                   # 寄存器比较
dword:[esp+4] != 0             # 栈参数非零
byte:[eax] == 0x55             # 检查字节值
```

## 練習題与答案

### 题目 1：断点类型选择

**问题**：你需要监控一个全局变量 `g_pluginCount`（地址 0x004A5678）何时被修改，应该使用哪种断点？为什么？

<details>
<summary>查看答案</summary>

使用**硬件写入断点**：

```
bph 004A5678, w, 4
```

原因：
1. 软件断点只能在**代码执行**时触发，不能监控数据写入
2. 内存断点可以监控写入，但粒度是 4KB 页，可能频繁误触发
3. 硬件写入断点精确到字节级别，只在目标地址被写入时触发
4. 触发后查看 EIP 和调用栈，即可找到修改该变量的代码

注意硬件断点最多 4 个，用完要及时清理。

</details>

### 题目 2：API 参数分析

**问题**：在 `CreateFileW` 断下后，如何判断程序是在读取文件还是创建新文件？

<details>
<summary>查看答案</summary>

查看两个关键参数：

```
dwDesiredAccess    = [ESP+8]
dwCreationDisposition = [ESP+14]
```

**dwDesiredAccess**（访问模式）：
```
GENERIC_READ    = 0x80000000   → 读取
GENERIC_WRITE   = 0x40000000   → 写入
GENERIC_READ | GENERIC_WRITE = 0xC0000000  → 读写
```

**dwCreationDisposition**（创建方式）：
```
CREATE_NEW        = 1  → 创建新文件（已存在则失败）
CREATE_ALWAYS     = 2  → 总是创建（覆盖已有）
OPEN_EXISTING     = 3  → 打开已有文件（不存在则失败）
OPEN_ALWAYS       = 4  → 打开或创建
TRUNCATE_EXISTING = 5  → 清空已有文件
```

组合判断：
- `GENERIC_READ + OPEN_EXISTING` = 读取已有文件
- `GENERIC_WRITE + CREATE_ALWAYS` = 创建/覆盖文件
- `GENERIC_READ | GENERIC_WRITE + OPEN_ALWAYS` = 读写打开或创建

在 x64dbg 中查看方法：
```
在断点处查看 ESP：
[ESP+4]  = 文件名指针 → 在内存窗口查看 Unicode 字符串
[ESP+8]  = 0x80000000 → GENERIC_READ
[ESP+14] = 3          → OPEN_EXISTING
结论：程序在读取已有文件
```

</details>

### 题目 3：编写日志断点脚本

**问题**：编写一个 x64dbg 脚本，自动记录程序调用 `MessageBoxW` 时的标题和内容文本。

<details>
<summary>查看答案</summary>

```
// script: log_messagebox.txt
// MessageBoxW 原型:
// int MessageBoxW(HWND hWnd, LPCWSTR lpText, LPCWSTR lpCaption, UINT uType)
// 参数位置: [esp+4]=hWnd, [esp+8]=lpText, [esp+C]=lpCaption, [esp+10]=uType

bp MessageBoxW
SetBreakpointLog MessageBoxW, "MessageBox: Title=\"{utf16@[esp+C]}\", Text=\"{utf16@[esp+8]}\", Type={[esp+10]}"
SetBreakpointCondition MessageBoxW, 0

// 运行程序
run
```

如果需要在特定条件下暂停（例如标题包含 "Error"），可以使用条件断点：

```
bp MessageBoxW
// 用 Python 脚本实现字符串匹配条件
// 或手动设置：当 uType 包含 MB_ICONERROR (0x10) 时断下
SetBreakpointCondition MessageBoxW, "[esp+10] & 0x10"
```

</details>
