## 5.1.1 DLL 导出表原理

### 什么是导出表

DLL（Dynamic Link Library）通过**导出表**（Export Table）向外部公开可调用的函数和数据。导出表是 PE 文件的一个数据目录项，记录了所有导出符号的名称、地址和序号。

```
┌─────────────── DLL 文件 ───────────────┐
│                                        │
│  PE Header                             │
│  ├─ Data Directory[0] = Export Table ──┼──┐
│  └─ ...                                │  │
│                                        │  │
│  .text (代码段)                         │  │
│  ├─ FuncA 实现 @ RVA 0x1000           │  │
│  ├─ FuncB 实现 @ RVA 0x1200           │  │
│  └─ FuncC 实现 @ RVA 0x1500           │  │
│                                        │  │
│  .rdata (只读数据段)                    │  │
│  ├─ Export Directory Table ◄───────────┼──┘
│  │  ├─ Name: "example.dll"            │
│  │  ├─ NumberOfFunctions: 3           │
│  │  ├─ NumberOfNames: 3              │
│  │  ├─ AddressOfFunctions ──┐        │
│  │  ├─ AddressOfNames ──────┼──┐     │
│  │  └─ AddressOfNameOrdinals┼──┼──┐  │
│  │                          │  │  │  │
│  │  Export Address Table ◄──┘  │  │  │
│  │  [0] 0x00001000 (FuncA)    │  │  │
│  │  [1] 0x00001200 (FuncB)    │  │  │
│  │  [2] 0x00001500 (FuncC)    │  │  │
│  │                             │  │  │
│  │  Export Name Table ◄────────┘  │  │
│  │  [0] → "FuncA"                │  │
│  │  [1] → "FuncB"                │  │
│  │  [2] → "FuncC"                │  │
│  │                                │  │
│  │  Ordinal Table ◄──────────────┘  │
│  │  [0] 0  (FuncA → ordinal 0)     │
│  │  [1] 1  (FuncB → ordinal 1)     │
│  │  [2] 2  (FuncC → ordinal 2)     │
│  └──────────────────────────────────│
└────────────────────────────────────────┘
```

### IMAGE_EXPORT_DIRECTORY 结构

```cpp
// PE 文件中导出目录的核心结构
typedef struct _IMAGE_EXPORT_DIRECTORY {
    DWORD Characteristics;          // 保留，通常为 0
    DWORD TimeDateStamp;            // 导出表创建时间戳
    WORD  MajorVersion;             // 主版本号
    WORD  MinorVersion;             // 次版本号
    DWORD Name;                     // DLL 名称字符串的 RVA
    DWORD Base;                     // 序号基值（通常为 1）
    DWORD NumberOfFunctions;        // 导出函数总数（通过序号导出）
    DWORD NumberOfNames;            // 通过名称导出的函数数量
    DWORD AddressOfFunctions;       // 导出地址表（EAT）的 RVA
    DWORD AddressOfNames;           // 导出名称表的 RVA
    DWORD AddressOfNameOrdinals;    // 序号表的 RVA
} IMAGE_EXPORT_DIRECTORY;
```

### 名称查找过程

当程序调用 `GetProcAddress(hModule, "FuncB")` 时，系统执行以下查找：

```
1. 在 Export Name Table 中二分查找 "FuncB"
   → 找到索引 i = 1

2. 用索引 i 查 Ordinal Table
   → ordinal = OrdinalTable[1] = 1

3. 用 ordinal 查 Export Address Table
   → address = EAT[1] = 0x00001200

4. 将 RVA 加上 DLL 基址得到实际地址
   → actual = DllBase + 0x00001200
```

```
┌───────────────┐     ┌──────────────┐     ┌───────────────────┐
│ Name Table    │     │ Ordinal Table│     │ Address Table     │
├───────────────┤     ├──────────────┤     ├───────────────────┤
│ [0] "FuncA"   │────→│ [0]  0       │────→│ [0] 0x1000        │
│ [1] "FuncB" ★ │────→│ [1]  1     ★ │────→│ [1] 0x1200      ★ │
│ [2] "FuncC"   │────→│ [2]  2       │────→│ [2] 0x1500        │
└───────────────┘     └──────────────┘     └───────────────────┘
```

### 转发导出（Forwarder）

导出地址表中的条目可以指向另一个 DLL 的函数，称为**转发导出**：

```cpp
// 如果 EAT 中的地址落在导出表的 RVA 范围内
// 则该地址不是函数入口，而是转发字符串
// 例如: "NTDLL.RtlAllocateHeap"

// kernel32.dll 中的典型转发：
//   HeapAlloc → NTDLL.RtlAllocateHeap
//   HeapFree  → NTDLL.RtlFreeHeap

// 判断方法（伪代码）：
DWORD exportDirRVA  = dataDir[0].VirtualAddress;
DWORD exportDirSize = dataDir[0].Size;
DWORD funcRVA = EAT[ordinal];

if (funcRVA >= exportDirRVA && funcRVA < exportDirRVA + exportDirSize) {
    // 转发导出：funcRVA 指向字符串 "DLL.Function"
    const char* forwarder = (const char*)(imageBase + funcRVA);
    printf("Forwarded to: %s\n", forwarder);
} else {
    // 正常导出：funcRVA 是函数入口
    void* funcAddr = (void*)(imageBase + funcRVA);
}
```

## 5.1.2 dumpbin 提取导出表

### dumpbin 基本用法

`dumpbin` 是 MSVC 自带的 PE 文件分析工具，通过 Developer Command Prompt 使用：

```cmd
:: 打开 Visual Studio Developer Command Prompt
:: 或运行 vcvarsall.bat

:: 查看导出表
dumpbin /exports extrans.dll

:: 输出示例：
Microsoft (R) COFF/PE Dumper Version 14.38.33135.0
Copyright (C) Microsoft Corporation.  All rights reserved.

Dump of file extrans.dll

File Type: DLL

  Section contains the following exports for extrans.dll

    00000000 characteristics
    FFFFFFFF time date stamp
           0.00 version
           1 ordinal base
           3 number of functions
           3 number of names

    ordinal hint RVA      name

          1    0 00001234 V2Link
          2    1 00001567 V2Unlink
          3    2 00003456 _wcsicmp

  Summary

        1000 .data
        5000 .rdata
        A000 .text
```

### 字段含义详解

```
ordinal  hint  RVA       name
───────  ────  ────────  ────────────
  1      0     00001234  V2Link
  │      │     │         │
  │      │     │         └─ 导出名称
  │      │     └─ 函数在 DLL 中的相对虚拟地址
  │      └─ 名称提示值（用于加速查找）
  └─ 导出序号（可用于按序号导入）
```

### 常用 dumpbin 选项

```cmd
:: 查看导出表
dumpbin /exports example.dll

:: 查看导入表（DLL 依赖了哪些其他 DLL）
dumpbin /imports example.dll

:: 查看所有头信息
dumpbin /headers example.dll

:: 查看段（section）信息
dumpbin /sections example.dll

:: 反汇编代码段
dumpbin /disasm example.dll

:: 查看重定位表
dumpbin /relocations example.dll

:: 同时查看导入和导出
dumpbin /imports /exports example.dll > analysis.txt
```

### 批量分析 KiriKiri2 插件

```cmd
:: 分析 KiriKiri2 游戏目录下的所有插件
:: 批处理脚本: analyze_plugins.bat

@echo off
setlocal
set GAME_DIR=C:\game\plugin
set OUTPUT=plugin_exports.txt

echo KiriKiri2 Plugin Export Analysis > %OUTPUT%
echo ================================= >> %OUTPUT%

for %%f in (%GAME_DIR%\*.dll) do (
    echo. >> %OUTPUT%
    echo === %%~nxf === >> %OUTPUT%
    dumpbin /exports "%%f" | findstr /R "^  " >> %OUTPUT%
)

echo Done. Results in %OUTPUT%
```

**典型输出**：

```
=== extrans.dll ===
          1    0 00001234 V2Link
          2    1 00001567 V2Unlink

=== wuvorbis.dll ===
          1    0 00002345 V2Link
          2    1 00002678 V2Unlink

          1    0 00001111 V2Link
```

### Python 脚本提取

使用 `pefile` 库进行自动化分析（适合批量处理）：

```python
# pip install pefile
import pefile
import sys
import os

def analyze_exports(dll_path):
    """分析 DLL 导出表"""
    pe = pefile.PE(dll_path)
    
    print(f"\n{'='*60}")
    print(f"File: {os.path.basename(dll_path)}")
    print(f"{'='*60}")
    
    # 检查是否有导出表
    if not hasattr(pe, 'DIRECTORY_ENTRY_EXPORT'):
        print("  No exports found!")
        return
    
    export_dir = pe.DIRECTORY_ENTRY_EXPORT
    print(f"  DLL Name: {export_dir.name.decode()}")
    print(f"  Number of exports: {len(export_dir.symbols)}")
    print()
    
    # 分类统计
    named_exports = []
    ordinal_only = []
    forwarded = []
    
    for exp in export_dir.symbols:
        if exp.forwarder:
            forwarded.append(exp)
        elif exp.name:
            named_exports.append(exp)
        else:
            ordinal_only.append(exp)
    
    # 打印命名导出
    if named_exports:
        print(f"  Named exports ({len(named_exports)}):")
        print(f"  {'Ordinal':<10} {'RVA':<12} {'Name':<40}")
        print(f"  {'-'*10} {'-'*12} {'-'*40}")
        for exp in sorted(named_exports, key=lambda e: e.name):
            name = exp.name.decode() if exp.name else ""
            print(f"  {exp.ordinal:<10} 0x{exp.address:08X}   {name}")
    
    # 打印仅序号导出
    if ordinal_only:
        print(f"\n  Ordinal-only exports ({len(ordinal_only)}):")
        for exp in ordinal_only:
            print(f"  #{exp.ordinal:<8} 0x{exp.address:08X}")
    
    # 打印转发导出
    if forwarded:
        print(f"\n  Forwarded exports ({len(forwarded)}):")
        for exp in forwarded:
            name = exp.name.decode() if exp.name else f"#{exp.ordinal}"
            target = exp.forwarder.decode()
            print(f"  {name} → {target}")
    
    pe.close()

# 分析单个 DLL
analyze_exports("extrans.dll")

# 批量分析目录下所有 DLL
def batch_analyze(directory):
    """批量分析目录下所有 DLL"""
    for filename in sorted(os.listdir(directory)):
        if filename.lower().endswith('.dll'):
            filepath = os.path.join(directory, filename)
            try:
                analyze_exports(filepath)
            except Exception as e:
                print(f"\nError analyzing {filename}: {e}")

batch_analyze(r"C:\game\plugin")
```

## 5.1.4 导出类型识别

通过导出名称可以判断导出类型：

### C 风格导出

```
V2Link              ← 标准 C 函数（KiriKiri2 插件入口）
V2Unlink            ← 标准 C 函数（KiriKiri2 插件卸载）
DllMain             ← DLL 入口点
sqlite3_open        ← C 库函数
```

特征：名称简洁，无特殊前缀/后缀

### C++ 修饰名（Decorated Names）

```
?CreateInstance@CMyClass@@SAPAV1@XZ     ← MSVC C++ 修饰名
_ZN8CMyClass14CreateInstanceEv          ← GCC/Clang C++ 修饰名
```

特征：以 `?` 开头（MSVC）或 `_Z` 开头（GCC/Clang）。详见下一节。

### stdcall 修饰名

```
_V2Link@4           ← stdcall 修饰：_函数名@参数字节数
_DllRegisterServer@0
```

特征：以 `_` 开头，`@数字` 结尾。数字表示参数总字节数。

### 仅序号导出

```
# 有些 DLL 不导出函数名，只导出序号
# 这常见于有意隐藏接口的 DLL

dumpbin /exports hidden.dll
  ordinal hint RVA      name
        1         00001000 [NONAME]
        2         00001200 [NONAME]
        3         00001500 [NONAME]
```

**分析仅序号导出的策略**：


### 对比分析多个插件

```python
import pefile
import os

def compare_plugin_exports(plugin_dir):
    """对比分析多个 KiriKiri2 插件"""
    plugins = {}
    
    for filename in sorted(os.listdir(plugin_dir)):
        if not filename.lower().endswith('.dll'):
            continue
        filepath = os.path.join(plugin_dir, filename)
        try:
            pe = pefile.PE(filepath)
            exports = []
            if hasattr(pe, 'DIRECTORY_ENTRY_EXPORT'):
                for exp in pe.DIRECTORY_ENTRY_EXPORT.symbols:
                    name = exp.name.decode() if exp.name else f"#{exp.ordinal}"
                    exports.append(name)
            
            imports = []
            if hasattr(pe, 'DIRECTORY_ENTRY_IMPORT'):
                for entry in pe.DIRECTORY_ENTRY_IMPORT:
                    imports.append(entry.dll.decode())
            
            plugins[filename] = {
                'exports': exports,
                'imports': imports,
                'is_krkr_plugin': 'V2Link' in exports and 'V2Unlink' in exports
            }
            pe.close()
        except Exception as e:
            print(f"Error: {filename}: {e}")
    
    # 输出对比结果
    print(f"{'Plugin':<25} {'KrKr?':<8} {'Exports':<30} {'Key Imports'}")
    print("-" * 100)
    for name, info in plugins.items():
        is_krkr = "✓" if info['is_krkr_plugin'] else "✗"
        exports = ", ".join(info['exports'][:5])
        key_imports = ", ".join(info['imports'][:4])
        print(f"{name:<25} {is_krkr:<8} {exports:<30} {key_imports}")

compare_plugin_exports(r"C:\game\plugin")
```

## 4.3.10 常见错误与解决方案

### 错误 1：dumpbin 报 "not a valid PE file"

```
症状: dumpbin /exports example.dll
      LINK : fatal error LNK1136: invalid or corrupt file

原因:
1. 文件损坏
2. 文件不是 PE 格式（可能是 .NET 程序集需要 ildasm）
3. 文件被压缩/加壳

解决方案:
1. 检查文件完整性: certutil -hashfile example.dll MD5
2. 用十六进制编辑器检查文件头:
   - 前两字节应为 "MZ"（0x4D5A）
   - 如果不是，文件可能损坏或不是 PE 文件
3. 如果是 .NET 程序集: 用 ildasm 或 dnSpy 代替
4. 如果是加壳 DLL: 先用对应脱壳工具处理
```

### 错误 2：pefile 库解析异常

```python
# 症状: pefile.PE("example.dll") 抛出 PEFormatError
# 原因: PE 文件格式不标准或被修改

# 解决方案: 使用 fast_load 模式跳过部分检查
try:
    pe = pefile.PE("example.dll", fast_load=True)
    # 手动加载需要的目录
    pe.parse_data_directories(directories=[
        pefile.DIRECTORY_ENTRY['IMAGE_DIRECTORY_ENTRY_EXPORT']
    ])
except pefile.PEFormatError as e:
    print(f"PE format error: {e}")
    # 使用十六进制方式手动解析
```

### 错误 3：看不到导出函数名称

```
症状: dumpbin 输出中函数名显示为 [NONAME]
原因: DLL 使用纯序号导出（DEF 文件中指定了 NONAME）

; 对应的 DEF 文件可能是这样的:
EXPORTS
    V2Link    @1 NONAME
    V2Unlink  @2 NONAME

解决方案:
1. 查看调用方的导入表，看是否有名称信息
2. 在 IDA/Ghidra 中反汇编该函数，推断功能
3. 如果有 .lib 文件，lib 中可能保留了名称
4. 搜索同一 DLL 的其他版本，可能有保留名称的版本
```

## 本節小結

1. **导出表结构**：由名称表、地址表、序号表三部分组成，支持按名称和按序号两种查找方式
2. **dumpbin 是首选工具**：MSVC 自带，命令行操作方便批量处理
3. **GUI 工具辅助**：CFF Explorer 和 PE-bear 适合可视化浏览和编辑
4. **Python 自动化**：pefile 库可以编程提取和分析导出信息
5. **KiriKiri2 特征**：标准插件只导出 V2Link 和 V2Unlink，所有功能通过 TJS 注册

## 練習題与答案

### 题目 1：导出表查找过程

**问题**：描述 `GetProcAddress(hDll, "V2Link")` 在 PE 导出表中的查找步骤。如果该函数是转发导出，会发生什么？

<details>
<summary>查看答案</summary>

查找步骤：

1. 从 PE Optional Header 的 Data Directory[0] 获取导出表 RVA
2. 定位到 `IMAGE_EXPORT_DIRECTORY` 结构
3. 在 `AddressOfNames` 指向的名称表中**二分查找** "V2Link"
4. 找到名称在数组中的索引 `i`
5. 用索引 `i` 查询 `AddressOfNameOrdinals` 数组，得到 `ordinal`
6. 用 `ordinal` 查询 `AddressOfFunctions` 数组，得到函数 RVA
7. 判断 RVA 是否在导出目录范围内：
   - **不在范围内**：正常导出，返回 `DllBase + RVA` 作为函数地址
   - **在范围内**：转发导出，RVA 指向的是形如 `"OTHER.DllFunc"` 的字符串

如果是转发导出：
1. 系统解析转发字符串，提取目标 DLL 名和函数名
2. 如果目标 DLL 未加载，先 `LoadLibrary` 加载
3. 递归调用 `GetProcAddress` 在目标 DLL 中查找
4. 最终返回目标 DLL 中的实际函数地址

</details>

### 题目 2：判断导出类型

**问题**：以下导出名称各属于什么类型？

```
(a) _DllRegisterServer@0
(b) ?GetInstance@CFactory@@SAPAV1@XZ
(c) V2Link
(d) #5 (无名称，仅序号)
(e) KERNEL32.HeapAlloc
```

<details>
<summary>查看答案</summary>

```
(a) _DllRegisterServer@0
    类型: stdcall 修饰名
    含义: __stdcall DllRegisterServer()，参数总大小 0 字节（无参数）
    
(b) ?GetInstance@CFactory@@SAPAV1@XZ
    类型: MSVC C++ 修饰名
    含义: public: static class CFactory* CFactory::GetInstance(void)
    可用 undname 工具反修饰
    
(c) V2Link
    类型: C 风格导出（extern "C"）
    含义: 标准 C 函数名，无修饰
    KiriKiri2 插件的标准入口函数
    
(d) #5
    类型: 仅序号导出（NONAME）
    含义: 只能通过序号 5 访问，名称被故意隐藏
    需要通过反汇编或动态调试推断功能
    
(e) KERNEL32.HeapAlloc
    类型: 转发导出（Forwarder）
    含义: 不是实际函数，而是转发到 KERNEL32.dll 的 HeapAlloc
    调用此函数会被重定向到 kernel32!HeapAlloc
```

</details>
