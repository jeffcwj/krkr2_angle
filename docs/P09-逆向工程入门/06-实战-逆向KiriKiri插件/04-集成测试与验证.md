## 6.4.1 编译还原的源码

### 项目目录结构

将上一节还原的 `imageEx.cpp` 组织为独立插件项目：

```
imageEx_plugin/
├── CMakeLists.txt          # 构建配置
├── imageEx.cpp             # 还原的插件源码（上一节完成）
├── tp_stub.h               # 从 cpp/core/plugin/ 复制
├── tp_stub.cpp             # 从 cpp/core/plugin/ 复制
├── ncbind.hpp              # 从 cpp/core/plugin/ 复制
├── ncbind.cpp              # 从 cpp/core/plugin/ 复制
└── test/
    └── test_imageEx.tjs    # TJS 测试脚本
```

### CMakeLists.txt 编写

```cmake
cmake_minimum_required(VERSION 3.15)
project(imageEx VERSION 1.0 LANGUAGES CXX)
set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

# KiriKiri2 插件必须是 SHARED 库（DLL）
add_library(imageEx SHARED imageEx.cpp tp_stub.cpp ncbind.cpp)

if(WIN32)
    set_target_properties(imageEx PROPERTIES PREFIX "")  # 去掉 lib 前缀
    target_compile_definitions(imageEx PRIVATE _WIN32 WIN32 _WINDOWS UNICODE _UNICODE)
    if(MSVC)
        target_compile_options(imageEx PRIVATE /W3 /wd4996 /utf-8)
    endif()
endif()

if(UNIX)
    set_target_properties(imageEx PROPERTIES PREFIX "" SUFFIX ".so")
    target_compile_definitions(imageEx PRIVATE __GNUC__)
endif()

target_include_directories(imageEx PRIVATE ${CMAKE_CURRENT_SOURCE_DIR})
```

### 编译步骤（四平台）

**Windows (MSVC)**:

```bash
# 在 imageEx_plugin/ 目录下
mkdir build && cd build
cmake .. -G "Visual Studio 17 2022" -A Win32    # 32 位！KiriKiri2 是 x86
cmake --build . --config Release

# 生成: build/Release/imageEx.dll
```

> **关键：KiriKiri2 原版是 32 位（x86）**。编译插件时必须指定 `-A Win32`，否则 64 位 DLL 无法被 32 位 krkr.exe 加载。

**Windows (MinGW)**:

```bash
mkdir build && cd build
cmake .. -G "MinGW Makefiles" -DCMAKE_CXX_FLAGS="-m32"
cmake --build .

# 生成: build/imageEx.dll
```

**Linux (GCC)**:

```bash
mkdir build && cd build
cmake .. -DCMAKE_CXX_FLAGS="-m32"   # 32 位模式
cmake --build .

# 生成: build/imageEx.so
# 注意: Linux 下运行 KiriKiri2 需要 Wine 或本项目的跨平台模拟器
```

**macOS (Clang)**:

```bash
mkdir build && cd build
cmake ..    # macOS 不需要 -m32（本项目模拟器为 64 位）
cmake --build .

# 生成: build/imageEx.so
```

### 常见编译错误处理

```
错误 1: fatal error: 'windows.h' not found
平台: Linux/macOS
原因: tp_stub.h 包含了 Windows 头文件
解决: 在 tp_stub.h 中用 #ifdef _WIN32 包裹 Windows 特定代码，
      或在编译时添加 -D_WIN32（仅用于类型定义，不调用 Win API）

错误 2: error LNK2019: unresolved external symbol _V2Link@8
原因: V2Link/V2Unlink 未正确导出
解决: 确认 ncbind.cpp 已加入编译，ncbind 宏会自动定义这两个函数。
      如需手动导出，添加 .def 文件:
      EXPORTS
          V2Link  @1
          V2Unlink @2

错误 3: error C2664: 'void free(void *)': cannot convert argument
原因: malloc/free 的参数类型与 tjs_uint32* 不匹配
解决: 添加显式类型转换:
      pixel_buffer_ = static_cast<tjs_uint32*>(malloc(size));
```

## 6.4.2 DLL 加载验证

### KiriKiri2 插件加载机制

```
KiriKiri2 启动流程（插件相关部分）:

krkr.exe 启动
  ↓
扫描 plugin/ 目录下所有 *.dll
  ↓
对每个 DLL 执行 LoadLibrary()
  ↓
GetProcAddress("V2Link")
  ↓
调用 V2Link(iTVPFunctionExporter*)
  ↓
V2Link 内部: 初始化 tp_stub 函数表
  ↓
ncbind: 注册所有 NCB_REGISTER_CLASS 定义的类
  ↓
TJS 脚本可以使用 new NativeImageFilter()
```

### 部署测试

```
游戏目录结构:
game_root/
├── krkr.exe              # KiriKiri2 引擎
├── data.xp3              # 游戏数据包
├── plugin/               # 插件目录
│   ├── imageEx.dll       # ← 将编译好的 DLL 复制到这里
│   └── (其他插件...)
└── startup.tjs           # 启动脚本
```

将编译好的 `imageEx.dll` 复制到游戏的 `plugin/` 目录后，启动 krkr.exe。如果插件加载成功，不会有错误弹窗。

### 验证 V2Link 是否被调用

在 `imageEx.cpp` 中添加调试日志（编译 Debug 版本时）：

```cpp
// 在 ncbind 注册之前添加（文件顶部，ncbind.hpp 之后）
#ifdef _DEBUG
#include <cstdio>
#define PLUGIN_LOG(fmt, ...) fprintf(stderr, "[imageEx] " fmt "\n", ##__VA_ARGS__)
#else
#define PLUGIN_LOG(fmt, ...)
#endif

// 在 NCB_REGISTER_CLASS 块之后添加:
NCB_PRE_REGIST_CALLBACK(pre_regist) {
    PLUGIN_LOG("V2Link called, registering NativeImageFilter");
}
NCB_POST_UNREGIST_CALLBACK(post_unregist) {
    PLUGIN_LOG("V2Unlink called, unregistering NativeImageFilter");
}
```

### 加载失败排查

```
症状 1: DLL 无法加载（弹窗 "プラグインを読み込めません"）
检查:
  (a) 用 dumpbin /headers imageEx.dll 确认是 x86（不是 x86_64）
  (b) 用 dumpbin /dependents imageEx.dll 确认无缺失依赖
  (c) 用 Dependency Walker / Dependencies 检查导入表

症状 2: V2Link 返回失败
检查:
  (a) 确认 tp_stub.cpp 已编译且链接（提供函数指针表初始化）
  (b) 用调试器在 V2Link 下断点，观察 exporter 参数是否为 NULL
  (c) 确认编译时未定义 TVP_COMPILING_BASE_MODULE（这是引擎内部标志）

症状 3: DLL 加载后 TJS 找不到类
检查:
  (a) 确认 NCB_REGISTER_CLASS 中的类名拼写正确
  (b) 确认类名与原始 DLL 中逆向出的名称完全一致（区分大小写）
```

    System.inform("测试3 通过: radius 设置/读取正常");

## 6.4.4 二进制行为对比

### 导出表对比

还原的 DLL 应该具有与原始 DLL 相同的导出函数：

```bash
# 原始 DLL
dumpbin /exports original_imageEx.dll > orig_exports.txt

# 还原 DLL
dumpbin /exports imageEx.dll > new_exports.txt

# 对比（应只有 V2Link 和 V2Unlink 两个导出）
diff orig_exports.txt new_exports.txt
```

预期输出：

```
ordinal hint RVA      name
      1    0 00001000 V2Link
      2    1 00001050 V2Unlink
```

> 除 V2Link/V2Unlink 外不应有其他导出。如原始 DLL 有额外导出（如 DllMain），需在 `.def` 文件中补充。

### 文件大小对比

```
还原 DLL 与原始 DLL 的大小差异参考:
- 差异 < 20%  → 还原质量优秀（结构高度匹配）
- 差异 20-50% → 还原质量良好（可能有未还原的内部函数或优化差异）
- 差异 > 50%  → 需要检查是否遗漏了功能或包含了多余代码
```

### 功能等价性测试

通过 TJS 脚本对比原始 DLL 和还原 DLL 的行为：

```javascript
// compare_behavior.tjs — 行为对比脚本
// 先用原始 DLL 运行一次记录结果，再用还原 DLL 运行对比

var filter = new NativeImageFilter();

// 测试用例矩阵
var test_cases = [
    %["method" => "adjustBrightness", "arg" => 0.0],
    %["method" => "adjustBrightness", "arg" => 0.5],
    %["method" => "adjustBrightness", "arg" => -0.5],
    %["method" => "adjustContrast",   "arg" => 0.0],

for (var i = 0; i < test_cases.count; i++) {
    var tc = test_cases[i];
    System.inform("测试 " + tc.method + "(" + tc.arg + ")");
}
invalidate filter;
```

### CRC32 快速校验

```cpp
// 在 C++ 测试工具中实现像素缓冲区的 CRC32 校验
// 对同样的输入执行同样的操作，对比 CRC32

#include <cstdint>

uint32_t crc32_buffer(const uint32_t* data, size_t count) {
    uint32_t crc = 0xFFFFFFFF;
    const uint8_t* bytes = reinterpret_cast<const uint8_t*>(data);
    size_t len = count * sizeof(uint32_t);
    for (size_t i = 0; i < len; i++) {
        crc ^= bytes[i];
        for (int j = 0; j < 8; j++) {
            crc = (crc >> 1) ^ (0xEDB88320 & -(crc & 1));
        }
    }
    return crc ^ 0xFFFFFFFF;
}

// 使用:
// 1. 用原始 DLL 处理 → 得到 CRC_orig
// 2. 用还原 DLL 处理相同输入 → 得到 CRC_new
// 3. CRC_orig == CRC_new → 功能等价 ✓
```

## 本節小結

1. **编译配置是第一关**——必须确保 32 位目标、正确的宏定义、tp_stub + ncbind 参与编译
2. **DLL 加载验证需逐层检查**——架构匹配 → 依赖完整 → V2Link 成功 → 类注册成功
3. **TJS 测试脚本是最直接的功能验证手段**——逐一测试属性、方法、边界条件
4. **二进制对比是终极验证**——导出表一致 + 同输入同输出的 CRC32 匹配证明功能等价
5. **还原质量的最终标准**：原始 DLL 和还原 DLL 对使用者来说行为完全相同（可互换）

### 题目 3：分析加载失败原因

你编译了一个插件 DLL，但 KiriKiri2 启动时弹窗 "プラグインを読み込めません"。`dumpbin /headers` 显示 `machine (x86)`，`dumpbin /exports` 显示 V2Link 和 V2Unlink 都存在。可能的原因是什么？如何排查？

<details>
<summary>查看答案</summary>

```
DLL 架构正确（x86）且导出正常，但仍无法加载，可能原因:

1. 缺少运行时依赖
   排查: dumpbin /dependents myPlugin.dll
   常见缺失: MSVCP140.dll, VCRUNTIME140.dll（需安装 VC++ Redistributable）
   或者静态链接: cmake 中设置 /MT 而非 /MD

2. DLL 初始化失败（DllMain 返回 FALSE）
   排查: 用 x32dbg 对 DllMain 下断点
   常见原因: 全局对象构造函数中的异常

3. 依赖的 DLL 自身加载失败（传递性依赖）
   排查: 使用 Dependencies (lucasg/Dependencies) 工具查看完整依赖树
   着重查看所有依赖项是否在系统 PATH 或游戏目录中

4. DEP/ASLR 兼容性问题
   排查: dumpbin /headers 检查 DLL characteristics
   解决: 链接时添加 /DYNAMICBASE /NXCOMPAT

5. CRT 版本不匹配
   排查: 确认 DLL 和 krkr.exe 使用相同的 CRT 版本
   解决: 统一使用 /MD（动态链接 CRT）或 /MT（静态链接）
```

</details>
