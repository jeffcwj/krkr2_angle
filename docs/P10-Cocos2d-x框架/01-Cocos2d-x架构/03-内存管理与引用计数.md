## Ref：引用计数基类

### 为什么需要引用计数

C++ 没有垃圾回收器（GC）。手动 `new/delete` 容易出错——忘记 `delete` 导致内存泄漏，过早 `delete` 导致悬垂指针。Cocos2d-x 采用了类似 Objective-C 的**引用计数**（Reference Counting）机制来半自动管理内存。

核心思想：每个对象维护一个"引用计数器"，记录有多少地方在使用它。计数降为 0 时自动销毁。

```
创建对象 → refcount = 1
retain() → refcount++
release() → refcount--
refcount == 0 → delete this
```

### Ref 类的实现

```cpp
// cocos2d-x/cocos/base/CCRef.h（简化）
class CC_DLL Ref {
public:
    // 增加引用计数
    void retain() {
        CCASSERT(_referenceCount > 0,
            "reference count should be greater than 0");
        ++_referenceCount;
    }

    // 减少引用计数，到 0 则删除
    void release() {
        CCASSERT(_referenceCount > 0,
            "reference count should be greater than 0");
        --_referenceCount;
        if (_referenceCount == 0) {
            delete this; // 自动销毁
        }
    }

    // 延迟释放：将对象加入 autorelease 池
    Ref* autorelease() {
        PoolManager::getInstance()
            ->getCurrentPool()
            ->addObject(this);
        return this;
    }

    // 获取当前引用计数
    unsigned int getReferenceCount() const {
        return _referenceCount;
    }

protected:
    Ref() : _referenceCount(1) {} // 构造时 refcount=1
    virtual ~Ref();

    unsigned int _referenceCount; // 引用计数器
};
```

### retain / release 基本用法

```cpp
#include "cocos2d.h"
USING_NS_CC;

void demonstrateRefCount() {
    // 1. 创建对象，refcount = 1
    auto sprite = new Sprite();
    CCLOG("创建后: refcount = %u",
          sprite->getReferenceCount()); // 1

    // 2. retain 增加引用
    sprite->retain();
    CCLOG("retain后: refcount = %u",
          sprite->getReferenceCount()); // 2

    // 3. release 减少引用
    sprite->release();
    CCLOG("release后: refcount = %u",
          sprite->getReferenceCount()); // 1

    // 4. 再次 release → refcount=0 → 自动 delete
    sprite->release();
    // sprite 已被销毁！不可再访问
    // sprite->getPosition(); // 未定义行为！
}
```

## create() 工厂模式

### 为什么要用 create() 而非 new

Cocos2d-x 的所有节点类都提供静态 `create()` 方法，内部调用 `new` + `init()` + `autorelease()`：

```cpp
// Sprite::create() 的典型实现
Sprite* Sprite::create(const std::string& filename) {
    auto sprite = new(std::nothrow) Sprite();
    if (sprite && sprite->initWithFile(filename)) {
        sprite->autorelease(); // 关键：标记为自动释放
        return sprite;
    }
    CC_SAFE_DELETE(sprite);
    return nullptr;
}
```

这个模式确保了：

| 步骤 | 作用 |
|------|------|
| `new(std::nothrow)` | 分配内存，失败返回 nullptr 而非抛异常 |
| `init()` | 初始化对象状态（可能失败） |
| `autorelease()` | 将对象放入自动释放池 |
| `CC_SAFE_DELETE` | 初始化失败时安全删除 |

### 使用 create() 的正确姿势

```cpp
// 正确：使用 create()，对象在帧末自动释放（除非被 addChild）
auto sprite = Sprite::create("player.png");
// refcount=1，且被标记为 autorelease

// 将 sprite 添加到场景 → 场景 retain 它 → refcount=2
scene->addChild(sprite);
// 帧末 autorelease 池清理 → release 一次 → refcount=1
// sprite 仍然存活（被场景持有）

// 错误：用 new 但忘记 release
auto sprite2 = new Sprite();
sprite2->initWithFile("enemy.png");
// 忘记 release → 内存泄漏！
```

### autorelease 的生命周期

autorelease 对象在**当前帧末尾**被自动 release 一次。这意味着：

```cpp
auto sprite = Sprite::create("icon.png");
// refcount=1，标记为 autorelease

// 在同一帧内，sprite 是安全的 —— autorelease 还没执行
sprite->setPosition(Vec2(100, 100));  // 安全
sprite->setScale(2.0f);              // 安全

// 如果不 addChild，帧末 autorelease → release → refcount=0 → 销毁
// 下一帧再访问 sprite → 悬垂指针！
```

```
帧 N 执行流程：

  [ 用户代码 ] ← sprite = Sprite::create() // refcount=1
       |
       |  scene->addChild(sprite) // 如果执行了 → refcount=2
       |
  [ drawScene() ] ← 渲染
       |
  [ autorelease 清理 ] ← sprite->release() // refcount 减 1
       |
  如果被 addChild: refcount=1，存活
  如果没 addChild: refcount=0，销毁 ← 危险！
```

## AutoreleasePool 详解

### 池的工作机制

AutoreleasePool 是一个容器，收集所有调用了 `autorelease()` 的对象，在池被清理（clear）时对每个对象调用一次 `release()`。

```cpp
// cocos2d-x/cocos/base/CCAutoreleasePool.h（简化）
class CC_DLL AutoreleasePool {
public:
    // 添加一个自动释放对象
    void addObject(Ref* object) {
        _managedObjectArray.push_back(object);
    }

    // 清理池中所有对象（对每个调用 release）
    void clear() {
        // 使用临时数组，防止 release 过程中对象
        // 的析构函数再次操作池
        std::vector<Ref*> releasings;
        releasings.swap(_managedObjectArray);

        for (auto obj : releasings) {
            obj->release();
        }
    }

private:
    std::vector<Ref*> _managedObjectArray;
};
```

### PoolManager：池的嵌套管理

```cpp
// PoolManager 管理一个池栈
class CC_DLL PoolManager {
public:
    static PoolManager* getInstance();

    // 获取当前活动池（栈顶）
    AutoreleasePool* getCurrentPool() const {
        return _releasePoolStack.back();
    }

    // 主循环中清理当前池
    // Director::mainLoop() 调用
    friend class Director;
};
```

池栈的用途——在短生命周期内创建大量临时对象时，手动创建嵌套池可以**提前释放**，避免大量对象堆积到帧末：

```cpp
void loadHundredTextures() {
    // 创建局部 autorelease 池
    // 构造时自动 push 到 PoolManager 栈
    AutoreleasePool pool;

    for (int i = 0; i < 100; i++) {
        auto tex = Texture2D::create(); // autorelease 到局部池
        // ... 使用 tex 做一些处理
    }

    // pool 析构时自动 clear → 100 个临时纹理被释放
    // 不必等到帧末
}
```

## 与 C++ 智能指针的对比

| 特性 | Cocos2d-x Ref | std::shared_ptr | std::unique_ptr |
|------|---------------|-----------------|-----------------|
| 引用计数 | 侵入式（对象内部） | 非侵入式（控制块） | 无（独占所有权） |
| 自动释放 | autorelease（延迟到帧末） | 析构时立即释放 | 析构时立即释放 |
| 拷贝语义 | 手动 retain/release | 自动 ref++ / ref-- | 不可拷贝 |
| 循环引用 | 需手动打断 | 用 weak_ptr 打断 | 不存在 |
| 线程安全 | 不安全 | ref 操作原子性 | N/A |

### 为什么 Cocos2d-x 不用 shared_ptr？

1. **历史原因**：Cocos2d-x 最初从 Objective-C 移植而来，沿用了 ObjC 的引用计数惯例
2. **与场景图集成**：`addChild()` 内部自动 `retain()`，`removeChild()` 内部自动 `release()`，与智能指针的 RAII 语义冲突
3. **autorelease 模式**：允许在 `create()` 中返回裸指针，调用方不必关心所有权——如果 `addChild()` 了就由场景图管理，否则帧末自动清理
4. **性能**：侵入式引用计数没有额外的控制块开销

```cpp
// Cocos2d-x 风格（返回裸指针，所有权由场景图管理）
auto sprite = Sprite::create("icon.png"); // autorelease
scene->addChild(sprite); // 场景 retain → 场景管理生命周期

// 如果用 shared_ptr 会很别扭：
// auto sprite = std::make_shared<Sprite>("icon.png");
// scene->addChild(sprite.get()); // 场景图和 shared_ptr 都持有引用
// 谁负责最终释放？所有权语义混乱
```

## 常见内存管理模式

### 模式 1：创建即添加（最常见）

```cpp
bool MyScene::init() {
    if (!Scene::init()) return false;

    // create() 返回 autorelease 对象
    // addChild() 内部 retain → 由场景管理生命周期
    auto bg = Sprite::create("bg.png");
    addChild(bg);

    auto label = Label::createWithSystemFont("Hello", "Arial", 24);
    addChild(label);

    // 无需手动 release —— 场景销毁时自动 release 所有子节点
    return true;
}
```

### 模式 2：成员变量持有（需要后续访问）

```cpp
class GameScene : public Scene {
    Sprite* _player = nullptr; // 成员变量

    bool init() override {
        if (!Scene::init()) return false;

        _player = Sprite::create("player.png");
        addChild(_player);
        // _player 由场景持有，无需额外 retain

        return true;
    }

    void update(float dt) override {
        // 安全访问 —— _player 仍被场景持有
        _player->setPositionX(
            _player->getPositionX() + 100 * dt);
    }
};
```

### 模式 3：延迟添加（需要手动 retain）

```cpp
class GameScene : public Scene {
    Sprite* _pendingSprite = nullptr;

    bool init() override {
        if (!Scene::init()) return false;

        // 创建但暂时不添加到场景
        _pendingSprite = Sprite::create("item.png");
        _pendingSprite->retain(); // 必须 retain！否则帧末被释放

        // 3 秒后再添加
        scheduleOnce([this](float) {
            addChild(_pendingSprite);
            _pendingSprite->release(); // 平衡之前的 retain
        }, 3.0f, "add_later");

        return true;
    }
};
```

### 模式 4：使用 Vector<T> 和 Map<K,V> 容器

Cocos2d-x 提供了自动管理引用计数的容器：

```cpp
#include "cocos2d.h"
USING_NS_CC;

// cocos2d::Vector<T> —— 类似 std::vector 但自动 retain/release
Vector<Sprite*> enemies;

auto e1 = Sprite::create("e1.png");
auto e2 = Sprite::create("e2.png");

enemies.pushBack(e1); // 内部调用 e1->retain()
enemies.pushBack(e2); // 内部调用 e2->retain()

CCLOG("e1 refcount = %u", e1->getReferenceCount()); // 2（autorelease+vector）

enemies.eraseObject(e1); // 内部调用 e1->release()
// e1 refcount 回到 1（autorelease）

enemies.clear(); // 对所有元素 release

// cocos2d::Map<K, V> —— 类似 std::unordered_map 但自动管理 V 的引用
Map<std::string, Sprite*> namedSprites;
namedSprites.insert("hero", Sprite::create("hero.png")); // retain
namedSprites.erase("hero"); // release
```

> **重要规则：** 永远不要把 `Ref*` 对象放进 `std::vector` 或 `std::map`——它们不会自动 `retain/release`，容易导致悬垂指针。使用 Cocos2d-x 的 `Vector` 和 `Map`。

## 循环引用与防范

引用计数的经典陷阱是**循环引用**（Circular Reference）：

```
A --retain--> B --retain--> A
refcount:  A=2, B=2

释放 A 的外部引用 → A refcount=1（B 还持有）
释放 B 的外部引用 → B refcount=1（A 还持有）
→ A 和 B 永远不会到 0 → 内存泄漏！
```

### 防范方法

```cpp
// 方法 1：弱引用（不 retain）
class Child : public Node {
    Node* _parent; // 裸指针，不 retain（父节点已经 retain 了子节点）
    // 不要写成 _parent->retain()！会导致循环引用
};

// 方法 2：回调使用 lambda 捕获弱引用
class MyNode : public Node {
    bool init() {
        if (!Node::init()) return false;

        auto btn = ui::Button::create("btn.png");
        // 不要捕获 this 的强引用 —— 按钮已经是子节点
        btn->addClickEventListener([this](Ref* sender) {
            // this 在这里是安全的，因为 btn 是 this 的子节点
            // 如果 this 被销毁，btn 也会被销毁，lambda 不会执行
            this->onButtonClick();
        });
        addChild(btn);

        return true;
    }
};

// 方法 3：手动打断循环
class A : public Ref {
public:
    B* partner = nullptr;

    void breakCycle() {
        if (partner) {
            partner->release(); // 手动打断
            partner = nullptr;
        }
    }
};
```

```cpp
#include "cocos2d.h"
USING_NS_CC;

class RefCountDemo : public Scene {
public:
    static RefCountDemo* create() {
        auto s = new(std::nothrow) RefCountDemo();
        if (s && s->init()) { s->autorelease(); return s; }
        CC_SAFE_DELETE(s);
        return nullptr;
    }

    bool init() override {
        if (!Scene::init()) return false;

        // 步骤 1：create 后观察
        auto sprite = Sprite::create("icon.png");
        CCLOG("[1] create后: refcount=%u",
              sprite->getReferenceCount()); // 1

        // 步骤 2：addChild 后观察
        addChild(sprite);
        CCLOG("[2] addChild后: refcount=%u",
              sprite->getReferenceCount()); // 2

        // 步骤 3：存入 Vector
        Vector<Sprite*> vec;
        vec.pushBack(sprite);
        CCLOG("[3] pushBack后: refcount=%u",
              sprite->getReferenceCount()); // 3

        // 步骤 4：从 Vector 移除
        vec.eraseObject(sprite);
        CCLOG("[4] erase后: refcount=%u",
              sprite->getReferenceCount()); // 2

        // 步骤 5：帧末 autorelease 清理后
        scheduleOnce([sprite](float) {
            CCLOG("[5] 下一帧: refcount=%u",
                  sprite->getReferenceCount()); // 1
        }, 0.0f, "check_next_frame");

        return true;
    }
};
```

```
[2] addChild后: refcount=2
[3] pushBack后: refcount=3
[4] erase后: refcount=2

```

## 常见错误及解决方案

### 错误 1：create() 后忘记 addChild 导致悬垂指针

```cpp
Sprite* _cached = nullptr;

void MyScene::init() {
    _cached = Sprite::create("icon.png");
    // 忘记 addChild 或 retain
    // 帧末 autorelease → refcount=0 → 销毁
}

void MyScene::update(float dt) {
    _cached->setPosition(...); // 崩溃！悬垂指针
}
```

**解决方案：** 如果不立即 addChild，必须手动 retain：

```cpp
void MyScene::init() {
    _cached = Sprite::create("icon.png");
    _cached->retain(); // 保住引用
}

void MyScene::onExit() {
    CC_SAFE_RELEASE_NULL(_cached); // 清理时 release
    Scene::onExit();
}
```

### 错误 2：retain/release 不配对

```cpp
auto obj = Sprite::create("a.png");
obj->retain();  // refcount=2
obj->retain();  // refcount=3
obj->release(); // refcount=2
// 忘了 release 两次 → 内存泄漏
```

**解决方案：** 每个 `retain()` 必须对应一个 `release()`。使用 `CC_SAFE_RELEASE_NULL` 宏可以同时 release 并置空指针：

```cpp
#define CC_SAFE_RELEASE_NULL(p) do { if(p) { (p)->release(); (p) = nullptr; } } while(0)
```

### 错误 3：在析构函数中访问已释放的子节点

```cpp
MyScene::~MyScene() {
    // 错误：子节点可能已经在 cleanup() 中被释放
    _player->getPosition(); // 未定义行为
}
```

**解决方案：** 不要在析构函数中访问子节点。清理逻辑放在 `onExit()` 或 `cleanup()` 中。

## 本节小结

- Cocos2d-x 使用**侵入式引用计数**（`Ref` 基类）管理对象生命周期
- `create()` 工厂方法 = `new` + `init()` + `autorelease()`，是创建对象的标准方式
- `autorelease()` 将对象的一次 `release()` 延迟到**帧末**执行
- `addChild()` 内部调用 `retain()`，`removeChild()` 内部调用 `release()`
- 如果 `create()` 后不 `addChild()`，必须手动 `retain()` 保住对象
- 使用 `cocos2d::Vector<T>` 和 `Map<K,V>` 代替 STL 容器管理 `Ref*` 对象
- 避免循环引用：子节点不要 `retain` 父节点

```cpp
auto parent = Node::create();                 // (2)
```

<summary>查看答案</summary>

```
步骤  | sprite refcount | 说明
(1)   | 1              | create() 内部 new→refcount=1，autorelease 标记但不立即执行
```

### 题目 2：修复内存泄漏

以下代码存在内存泄漏，找出问题并修复：

```cpp
class BulletPool : public Node {
    std::vector<Sprite*> _pool; // 问题！

    void createBullets() {
        for (int i = 0; i < 50; i++) {
            auto bullet = Sprite::create("bullet.png");
            bullet->retain();
            _pool.push_back(bullet);
        }
    }

    // 从未调用 release...
};
```

<details>
<summary>查看答案</summary>

问题：
1. 使用 `std::vector` 存储 `Ref*`，不会自动管理引用计数
2. 手动 `retain()` 了 50 次但从未 `release()`

修复方案：

```cpp
class BulletPool : public Node {
    cocos2d::Vector<Sprite*> _pool; // 使用 Cocos2d 容器

    void createBullets() {
        for (int i = 0; i < 50; i++) {
            auto bullet = Sprite::create("bullet.png");
            _pool.pushBack(bullet); // 自动 retain
        }
    }

    void cleanup() override {
        _pool.clear(); // 自动 release 所有元素
        Node::cleanup();
    }

    // 如果需要从池中移除单个子弹：
    void removeBullet(Sprite* bullet) {
        _pool.eraseObject(bullet); // 自动 release
    }
};
```

使用 `cocos2d::Vector` 后：
- `pushBack()` 自动 `retain()`
- `eraseObject()` 自动 `release()`
- `clear()` 对所有元素 `release()`
- 析构时自动 `clear()`

无需手动管理 `retain/release`，杜绝泄漏。

</details>

### 题目 3：autorelease 时机判断

以下代码在运行时会崩溃吗？为什么？

```cpp
void MyScene::init() {
    auto label = Label::createWithSystemFont("Hi", "Arial", 24);
    // 不 addChild，不 retain

    scheduleOnce([label](float dt) {
        label->setString("Bye"); // 这里安全吗？
    }, 0.0f, "delayed");
}
```

<details>
<summary>查看答案</summary>

**会崩溃（未定义行为）。**

分析执行时序：
1. `init()` 中 `create()` → label refcount=1，标记 autorelease
2. lambda 按值捕获了 label 指针（拷贝了指针值，但不增加 refcount）
3. `scheduleOnce` 延迟 0 秒 = 下一帧执行
4. 本帧末：autorelease 池清理 → `label->release()` → refcount=0 → 销毁
5. 下一帧：lambda 执行 → `label->setString("Bye")` → 访问已释放内存 → 崩溃

修复：在 lambda 执行前保住 label 的引用。

```cpp
void MyScene::init() {
    auto label = Label::createWithSystemFont("Hi", "Arial", 24);
    label->retain(); // 保住引用

    scheduleOnce([this, label](float dt) {
        label->setString("Bye"); // 安全
        addChild(label);         // 添加到场景
        label->release();        // 平衡 retain
    }, 0.0f, "delayed");
}
```

</details>
