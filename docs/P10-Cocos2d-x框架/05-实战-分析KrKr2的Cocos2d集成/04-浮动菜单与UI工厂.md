## TVPGameMainMenu —— 浮动游戏菜单

`TVPGameMainMenu` 是 KrKr2 在游戏运行时显示的一个可拖拽浮动菜单条。它不继承 `iTVPBaseForm`，而是直接继承 `cocos2d::Node`，因为它不走表单的 Header/Body/Footer 三段式布局——它只是一个悬浮在游戏画面底部的工具栏。

### 类结构概览

```
TVPGameMainMenu : cocos2d::Node
├── _root          ─ CSBReader 加载的 GameMenu.csb 根节点
│   ├── handler    ─ 拖拽手柄区域（可水平拖动、竖直拖动整个菜单）
│   ├── btn_gamemenu  ─ 打开 TJS2 脚本层菜单
│   ├── btn_window    ─ 显示窗口管理覆盖层
│   ├── btn_mousemode ─ 切换虚拟鼠标/触摸模式
│   ├── btn_keyboard  ─ 弹出软键盘（跨平台差异）
│   └── btn_exit      ─ 关闭当前窗口
├── _handler       ─ 指向 handler 节点的引用
├── _icon_touch    ─ 触摸模式图标
└── _icon_mouse    ─ 鼠标模式图标
```

### 初始化流程：CSB 加载与自适应缩放

源码位于 `cpp/core/environ/ui/GameMainMenu.cpp` 第 40-133 行：

```cpp
// GameMainMenu.cpp — init() 核心片段
bool TVPGameMainMenu::init() {
    bool ret = Node::init();

    // 1. 加载 CSB 资源
    CSBReader reader;
    _root = reader.Load("ui/GameMenu.csb");

    // 2. 根据屏幕方向计算缩放基准
    cocos2d::Size size = TVPMainScene::GetInstance()->getGameNodeSize();
    float scale;
    if(size.width > size.height) {
        scale = size.height / 720;  // 横屏：以高度 720 为基准
    } else {
        scale = size.width / 720;   // 竖屏：以宽度 720 为基准
    }
    setScale(scale);

    // 3. 调整根节点宽度填满屏幕（高度保持 CSB 原始值）
    size.width /= scale;
    size.height = _root->getContentSize().height;
    _root->setContentSize(size);
    ui::Helper::doLayout(_root);  // 触发子节点重新布局
    addChild(_root);

    // 4. 获取 handler 节点并注册触摸监听器
    _handler = reader.findController("handler");
    // ... (触摸监听器注册，见下文)

    return ret;
}
```

关键设计点：

| 步骤 | 说明 |
|------|------|
| 缩放基准 720px | 无论屏幕分辨率多大，菜单条总是以 720px 为参考进行等比缩放 |
| `setScale(scale)` | 对整个 `TVPGameMainMenu` 节点设缩放，子节点自动继承 |
| `size.width /= scale` | 反算出缩放后的逻辑宽度，让 `_root` 在逻辑坐标系下填满屏幕 |
| `Helper::doLayout` | Cocos2d-x UI 的重排接口，让 HORIZONTAL/VERTICAL 布局生效 |

### 双触摸监听器：handler 与 background

`TVPGameMainMenu` 注册了 **两个独立的** `EventListenerTouchOneByOne`，分别绑定到不同节点：

```cpp
// 监听器 1：绑定到 _handler（拖拽手柄）
_eventDispatcher->removeEventListenersForTarget(_handler);
EventListenerTouchOneByOne *listener = EventListenerTouchOneByOne::create();
listener->setSwallowTouches(true);  // 吞没触摸，不传递给下层
listener->onTouchBegan =
    CC_CALLBACK_2(TVPGameMainMenu::onHandlerTouchBegan, this);
listener->onTouchMoved =
    CC_CALLBACK_2(TVPGameMainMenu::onHandlerTouchMoved, this);
listener->onTouchEnded =
    CC_CALLBACK_2(TVPGameMainMenu::onHandlerTouchEnded, this);
listener->onTouchCancelled =
    CC_CALLBACK_2(TVPGameMainMenu::onHandlerTouchCancelled, this);
_eventDispatcher->addEventListenerWithSceneGraphPriority(listener, _handler);

// 监听器 2：绑定到 _root（整个菜单背景）
listener = EventListenerTouchOneByOne::create();
listener->setSwallowTouches(true);
listener->onTouchBegan =
    CC_CALLBACK_2(TVPGameMainMenu::onBackgroundTouchBegan, this);
// ... Moved/Ended/Cancelled 回调类似
_eventDispatcher->addEventListenerWithSceneGraphPriority(listener, _root);
```

两个监听器的分工：

```
┌─────────────────────────────────────────────────────┐
│  _root（背景监听器）                                  │
│  ┌───────────────────────────────────────────┐      │
│  │  各种按钮 (btn_gamemenu, btn_window, ...) │      │
│  └───────────────────────────────────────────┘      │
│  ┌──────────┐                                       │
│  │ _handler │ ← handler 监听器（拖拽 + 点击判定）    │
│  └──────────┘                                       │
└─────────────────────────────────────────────────────┘
  触摸 handler → 拖拽/展开/收起
  触摸 background（菜单展开时）→ 收起菜单
  触摸 background（菜单收起时）→ 不响应（return false）
```

background 监听器的逻辑极其简单：

```cpp
// GameMainMenu.cpp 第 269-276 行
bool TVPGameMainMenu::onBackgroundTouchBegan(cocos2d::Touch *touch,
                                             cocos2d::Event *unusedEvent) {
    if(!_shrinked) {
        shrink();       // 菜单展开时，点击背景 → 收起
        return true;
    }
    return false;       // 菜单已收起，不吞没触摸（让游戏接收）
}
```

### 拖拽交互：水平移动 handler + 竖直移动菜单

handler 的触摸交互是本节最复杂的部分。核心在于 `onHandlerTouchMoved`：

```cpp
// GameMainMenu.cpp 第 162-196 行
void TVPGameMainMenu::onHandlerTouchMoved(cocos2d::Touch *touch,
                                          cocos2d::Event *unusedEvent) {
    Vec2 nsp = _handler->convertToNodeSpace(touch->getLocation());
    Vec2 movDist = nsp - _touchBeganPosition;

    // 1. 标记是否发生了拖拽（阈值 1px）
    if(!_draggingY && std::abs(movDist.y) > 1) _draggingY = true;
    if(!_draggingX && std::abs(movDist.x) > 1) _draggingX = true;

    // 2. 水平方向：移动 handler 自身（带边界限制）
    _handler->setPositionX(_handler->getPositionX() + movDist.x);
    // ... 边界检测：handler 左边不超过 0，右边不超过 root 宽度

    // 3. 竖直方向：移动整个 _root（带橡皮筋效果）
    float newPosY = _root->getPositionY() + movDist.y;
    if(newPosY < -rootsize.height)
        newPosY = -rootsize.height;     // 下限：完全隐藏
    else if(newPosY > 0) {
        // 橡皮筋公式：越拉越难拉
        float t = rootsize.height / 2;
        newPosY = (1 - 1 / (newPosY / t + 1)) * t;
    }
    _root->setPositionY(newPosY);
}
```

**橡皮筋回弹公式详解**：

当用户向上拖拽（`newPosY > 0`，即试图把菜单拖到屏幕上方），采用非线性阻尼：

```
实际位移 = t × (1 - 1/(x/t + 1))

其中 t = rootsize.height / 2，x = 原始位移
```

这是一个经典的饱和函数——输入趋于无穷时，输出趋于 `t`。效果是：

| 原始位移 x | 实际位移 | 阻尼比 |
|-----------|---------|--------|
| 0 | 0 | 0% |
| t/2 | t/3 ≈ 33% | 67% 阻尼 |
| t | t/2 = 50% | 50% 阻尼 |
| 2t | 2t/3 ≈ 67% | 33% 阻尼 |
| ∞ | t = 上限 | 100% 阻尼 |

### 点击判定与展开/收起决策

`onHandlerTouchEnded` 需要区分"点击"和"拖拽"：

```cpp
// GameMainMenu.cpp 第 198-219 行
void TVPGameMainMenu::onHandlerTouchEnded(cocos2d::Touch *touch,
                                          cocos2d::Event *unusedEvent) {
    _hitted = false;
    bool isClick = TVPGetTickCount() - _touchBeganTime < 500;  // 500ms 内
    bool doExpand = false;

    if(isClick && !_draggingY) {
        // 快速触摸且无竖直拖拽 → 视为点击
        doExpand = _shrinked ^ _draggingX;
        // 收起状态 + 无水平拖拽 → 展开
        // 收起状态 + 有水平拖拽 → 不展开（只是在调整 handler 位置）
        // 展开状态 + 无水平拖拽 → 收起
        // 展开状态 + 有水平拖拽 → 展开（保持）
    } else {
        // 拖拽模式：根据当前位置判断
        const cocos2d::Size &rootsize = _root->getContentSize();
        if(_shrinked) {
            doExpand = _root->getPositionY() > -rootsize.height * 0.75f;
            // 已收起状态，拖到超过 25% 可见 → 展开
        } else {
            doExpand = _root->getPositionY() > -rootsize.height * 0.25f;
            // 已展开状态，拖到低于 75% 可见 → 收起
        }
    }
    if(doExpand) expand(); else shrink();
}
```

判定逻辑流程：`onTouchBegan` 记录时间/位置 → `onTouchMoved` 更新 `_draggingX`/`_draggingY` → `onTouchEnded` 判断：若 < 500ms 且无 Y 拖拽 → 点击路径（XOR 逻辑），否则 → 拖拽路径（75%/25% 位置阈值） → 最终调用 `expand()` 或 `shrink()`。

### shrink/expand 动画：匀速滑动模型

收起和展开都使用 **恒定速度** 而非恒定时长，这样无论菜单当前位置如何，视觉速度都一致：

```cpp
// GameMainMenu.cpp 第 226-241 行
const float _shrinkSpd = 700; // px/sec — 收起速度
const float _expandSpd = 700; // px/sec — 展开速度

void TVPGameMainMenu::shrink() {
    if(_hitted) return;          // 正在触摸中，不响应
    _shrinked = true;

    float h = _root->getContentSize().height * _root->getScaleY();
    float topBoundary = _root->getPosition().y + h;
    float duration = topBoundary / _shrinkSpd;  // 距离 / 速度 = 时间

    _root->stopAllActions();
    _root->runAction(MoveTo::create(duration, Vec2(0, -h)));

    // handler 延迟 2 秒后淡出到不活跃透明度
    _handler->runAction(Sequence::createWithTwoActions(
        DelayTime::create(2),
        FadeTo::create(0.35f, _handler_inactive_opacity)));
}

void TVPGameMainMenu::expand() {
    if(_hitted) return;
    _shrinked = false;

    float bottomBoundary = _root->getPosition().y;
    float duration = std::abs(bottomBoundary) / _expandSpd;

    _root->stopAllActions();
    _root->runAction(MoveTo::create(duration, Vec2(0, 0)));
    _handler->runAction(FadeIn::create(0.15f));  // 快速淡入
}
```

**动画时间线对比**：

| 动画 | _root 行为 | _handler 行为 |
|------|-----------|--------------|
| shrink | `MoveTo(duration, (0, -h))` 匀速下滑 | 延迟 2s → `FadeTo(0.35s, 不活跃透明度)` |
| expand | `MoveTo(duration, (0, 0))` 匀速上滑 | `FadeIn(0.15s)` 快速淡入 |
| shrinkWithTime | 额外 `DelayTime(dur)` + 同 shrink | 额外延迟 `2+dur` 秒后淡出 |

`shrinkWithTime` 用于启动时的入场动画——先展示菜单几秒，再自动收起。与 `shrink()` 相同，只是在 `MoveTo` 前插入 `DelayTime(dur)` 延迟，handler 淡出也额外延迟 `dur` 秒。

## CsdUIFactory —— 纯代码 UI 构建

`CsdUIFactory` 位于 `cpp/core/environ/ui/csd/CsdUIFactory.h`（1097 行），是一个 `namespace Csd` 下的静态函数集合。每个函数签名统一为 `NodeBuilderFn`：

### 为什么需要纯代码构建？

前一节介绍的 `iTVPBaseForm::initFromFile` 接受 `NodeBuilderFn` 参数来构建 Header/Body/Footer 三个区域。有两种方式提供内容：

1. **CSB 加载**：通过 lambda 包装 `CSBReader::Load`
2. **纯代码构建**：直接调用 `Csd::createXxx` 系列函数

```
initFromFile(naviBuilder, bodyBuilder, footerBuilder)
                │              │             │
     ┌──────────┘              │             └──────────┐
     ▼                         ▼                        ▼
  CSB 方式:              纯代码方式:               跳过区域:
  [](Size &s, float) {   Csd::createNaviBarWithMenu  Csd::createEmpty
    CSBReader r;                                     (返回 nullptr)
    return r.Load(...)
  }
```

纯代码构建的优势：

- **不依赖 .csb 文件**：无需 Cocos Studio 设计资源，降低外部工具链依赖
- **动态尺寸适配**：接收 `size` 和 `scale` 参数，可精确计算每个子控件的尺寸
- **编译期检查**：类型错误在编译时暴露，CSB 加载错误只在运行时发现

### 深入分析：createFileItem

`createFileItem`（第 161-274 行）是最复杂的工厂函数之一。它构建文件列表的单个单元格，节点层次如下：

```

├── filename (Text)        ← setSwallowTouches(false)，触摸穿透
├── underline (Layout)     ← 渐变分割线 (229,229,229) → (42,42,42)
```

关键设计：`highlight` 和 `filename` 都设置 `setSwallowTouches(false)`，使触摸事件可穿透到父节点 `TTouchEventRouter`（下文详述）。`highlight` 最后添加确保在 Z 序最上层，点击时产生半透明视觉反馈。

### 共通模式：setName 与 NodeMap 兼容

所有工厂函数创建的子节点都调用了 `setName()`，且 name 与 CSB 文件中的节点名保持一致，使得 `CSBReader::findController` / `NodeMap::findController` 在 CSB 加载和纯代码构建两种模式下都能正常工作。这是**接口等价性**的核心保证。

## TTouchEventRouter —— 触摸事件路由

### 问题场景

在 `TableView` 中，每个 `TableViewCell` 包含复杂的子控件（按钮、复选框等）。Cocos2d-x 的默认行为是：子控件的触摸事件不会通知到父 Widget。但 `FileSelectorForm` 需要在子控件被点击时通知到 Cell 层——比如点击文件名时要触发"打开文件"逻辑。

### TTouchEventRouter 实现

`TTouchEventRouter` 位于 `BaseForm.h` 第 152-177 行，只有 26 行代码：

```cpp
// BaseForm.h 第 152-177 行
class TTouchEventRouter : public cocos2d::ui::Widget {
public:
    typedef std::function<void(
        cocos2d::ui::Widget::TouchEventType event,
        cocos2d::ui::Widget *sender,
        cocos2d::Touch *touch)> EventFunc;

    static TTouchEventRouter *create() {
        auto *ret = new TTouchEventRouter;
        ret->init();
        ret->autorelease();
        return ret;
    }

    void setEventFunc(const EventFunc &func) { _func = func; }

    // 核心：重写 interceptTouchEvent
    void interceptTouchEvent(
        cocos2d::ui::Widget::TouchEventType event,
        cocos2d::ui::Widget *sender,
        cocos2d::Touch *touch) override {
        if(_func) _func(event, sender, touch);
    }

private:
    EventFunc _func;
};
```

**工作原理**：Cocos2d-x 的 `Widget` 类有一个 `interceptTouchEvent` 虚函数。当子 Widget 被触摸时，Cocos2d-x 会沿着节点树向上遍历，在每个父 Widget 上调用 `interceptTouchEvent`。`TTouchEventRouter` 重写此方法，将事件转发给注册的回调函数。

事件传播流程：

```
用户触摸 highlight 按钮
    │
    ▼
Widget::onTouchBegan(highlight)
    │
    ▼  沿父节点链向上查找 interceptTouchEvent
    │
FileItemCellImpl (继承自 TTouchEventRouter)
    │  interceptTouchEvent() 被调用
    ▼
_func(BEGAN, highlight, touch)
    │
    ▼
FileItemCell::onClicked() / onLongPress()
```

### 实际使用：FileItemCellImpl

`FileItemCellImpl`（`FileSelectorForm.h` 第 107-139 行）直接继承 `TTouchEventRouter`，接收 `Csd::NodeBuilderFn`（通常是 `Csd::createFileItem`）来构建内部 UI。因为它继承了 `TTouchEventRouter`，所有子控件（filename、highlight、checkBox）的触摸事件都会自动路由到 `interceptTouchEvent`，无需额外配置。

### TCommonTableCell 中的路由器模式

对于不直接继承 `TTouchEventRouter` 的 Cell，可以组合使用。`TCommonTableCell`（`BaseForm.h` 第 179-205 行）持有一个 `_router` 成员，通过手动 `retain()`/`release()` 管理其生命周期（因为路由器可能被 detach/attach，不能依赖 autorelease）。它还在 `setContentSize` 中同步路由器尺寸，确保触摸区域正确。

## 常见错误与解决方案

### 错误 1：忘记 removeEventListenersForTarget 导致重复监听

```cpp
// ❌ 错误：CSB 加载的节点可能已有默认监听器
_handler = reader.findController("handler");
EventListenerTouchOneByOne *listener = EventListenerTouchOneByOne::create();
listener->onTouchBegan = CC_CALLBACK_2(...);
_eventDispatcher->addEventListenerWithSceneGraphPriority(listener, _handler);
// 结果：handler 上有两个监听器，触摸行为不可预期

// ✅ 正确：先清除再注册
_handler = reader.findController("handler");
_eventDispatcher->removeEventListenersForTarget(_handler);  // 清除原有监听器
EventListenerTouchOneByOne *listener = EventListenerTouchOneByOne::create();
// ... 注册新监听器
```

### 错误 2：匀速动画中传入负数 duration

```cpp
// ❌ 危险：如果菜单已在目标位置，duration 可能为 0 或负数
float duration = topBoundary / _shrinkSpd;
// 当 topBoundary <= 0 时，duration <= 0

// ✅ KrKr2 的做法：shrink() 开头有 if(_hitted) return; 保护
// 同时 MoveTo::create 对 duration=0 的处理是立即完成（安全）
// 但自己实现时建议加 std::max(duration, 0.0f) 防护
```

### 错误 3：setSwallowTouches 配置错误导致触摸穿透

```cpp
// ❌ 错误：highlight 按钮吞没触摸，TTouchEventRouter 收不到事件
highlight->setSwallowTouches(true);   // 触摸被 highlight 吞掉
// FileItemCellImpl::interceptTouchEvent 永远不会被调用

// ✅ 正确：子控件不吞触摸，让事件可以穿透到 Router
highlight->setSwallowTouches(false);  // 触摸穿透
// 但 EventListenerTouchOneByOne 的 setSwallowTouches(true) 是可以的
// 因为那是 handler 级别的监听器，用于防止触摸传到游戏层
```

## 动手实践

### 实践 1：实现匀速动画的浮动工具栏

核心要点：使用**距离/速度**计算 duration，而非固定时长：

```cpp
// 匀速收起/展开——与 TVPGameMainMenu 相同的动画模型
class SimpleBar : public cocos2d::Node {
    cocos2d::ui::Layout *_bar;    // 工具栏本体
    bool _collapsed = false;
    static constexpr float SPEED = 500.0f;  // px/sec

    void collapse() {
        _collapsed = true;
        float h = _bar->getContentSize().height;
        float visible = _bar->getPositionY() + h;  // 当前可见高度
        float dur = visible / SPEED;                // 距离 / 速度
        _bar->stopAllActions();
        _bar->runAction(cocos2d::MoveTo::create(dur, cocos2d::Vec2(0, -h)));
    }

    void expand() {
        _collapsed = false;
        float dist = std::abs(_bar->getPositionY());
        float dur = dist / SPEED;
        _bar->stopAllActions();
        _bar->runAction(cocos2d::MoveTo::create(dur, cocos2d::Vec2(0, 0)));
    }
};
// 关键：任何中间位置触发收起/展开，速度恒定 500px/s
// 距终点越近 duration 越短，视觉速度不变
```

### 题目 1：橡皮筋阻尼计算

假设 `rootsize.height = 100`，用户向上拖拽了 150px（`newPosY` 原始值为 150）。计算实际 `newPosY` 的值，以及阻尼比（实际位移/原始位移）。

<details>
<summary>查看答案</summary>

已知：
- `t = rootsize.height / 2 = 50`
- `x = 150`（原始位移）

代入公式：

```
newPosY = t × (1 - 1/(x/t + 1))
        = 50 × (1 - 1/(150/50 + 1))
        = 50 × (1 - 1/(3 + 1))
        = 50 × (1 - 0.25)
        = 50 × 0.75
        = 37.5
```

实际位移 = 37.5px，阻尼比 = 37.5 / 150 = **25%**（即 75% 的位移被"吃掉"了）。

验证代码：

```cpp
#include <cstdio>

int main() {
    float rootHeight = 100.0f;
    float t = rootHeight / 2;  // 50
    float x = 150.0f;          // 原始位移

    float actual = t * (1 - 1 / (x / t + 1));
    float ratio = actual / x;

    printf("t = %.1f\n", t);              // 50.0
    printf("原始位移 = %.1f\n", x);        // 150.0
    printf("实际位移 = %.1f\n", actual);    // 37.5
    printf("阻尼比 = %.1f%%\n", ratio * 100); // 25.0%
    return 0;
}
```

### 题目 2：XOR 点击判定逻辑

在 `onHandlerTouchEnded` 中，点击判定使用了 `doExpand = _shrinked ^ _draggingX`。请列出所有 4 种组合的结果，并解释每种情况的物理含义。

<details>
<summary>查看答案</summary>

| `_shrinked` | `_draggingX` | `_shrinked ^ _draggingX` | 物理含义 |
|-------------|-------------|--------------------------|----------|
| `false`（展开） | `false`（未水平拖拽） | `false` → 收起 | 展开状态下点击 handler → 收起菜单 |
| `false`（展开） | `true`（有水平拖拽） | `true` → 展开 | 展开状态下水平滑动 handler → 保持展开（只是调整位置） |
| `true`（收起） | `false`（未水平拖拽） | `true` → 展开 | 收起状态下点击 handler → 展开菜单 |
| `true`（收起） | `true`（有水平拖拽） | `false` → 收起 | 收起状态下水平滑动 handler → 保持收起（只是调整位置） |

核心逻辑：**纯点击（无拖拽）时翻转状态，有水平拖拽时保持当前状态**。XOR 运算巧妙地用一行代码表达了这个语义——`_shrinked` 为真表示"想展开"，`_draggingX` 为真表示"取反"（因为在拖拽时不应该改变展开/收起状态）。

验证：将上述四组值代入 `shrinked ^ draggingX`，对照输出即可确认。

</details>
