## 术语预览

本节将涉及以下术语，先有个印象，正文中会逐一详细讲解：

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| 引用计数 | Reference Counting | 一种资源管理技术：每次"使用"时计数加 1，每次"释放"时减 1，减到 0 才真正释放资源 |

## 子系统依赖关系

上一节我们学习了 SDL2 的 9 个子系统各自的功能。但这些子系统并不是完全独立的——有些子系统在初始化时会**自动拉起**它所依赖的其他子系统。理解这些依赖关系，是写出正确初始化代码的前提。

### 依赖关系图

下面这张 ASCII 图展示了 SDL2 子系统之间的依赖关系。箭头表示"A 依赖 B"，即初始化 A 时会自动初始化 B：

```
┌─────────────────────────────────────────────────────────┐
│                  SDL2 子系统依赖关系图                     │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  GAMECONTROLLER ──→ JOYSTICK ──→ EVENTS                │
│                                                         │
│  VIDEO ──→ EVENTS                                      │
│                                                         │
│  JOYSTICK ──→ EVENTS                                   │
│                                                         │
│  ─────────────────────────────────                      │
│  以下子系统完全独立，不依赖任何其他子系统：                 │
│                                                         │
│  TIMER      （独立）                                    │
│  AUDIO      （独立）                                    │
│  HAPTIC     （独立）                                    │
│  SENSOR     （独立）                                    │
│  EVENTS     （被依赖，但自身不依赖任何子系统）              │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 依赖规则详解

让我们逐条解释每个依赖关系的原因：

**1. GAMECONTROLLER → JOYSTICK → EVENTS**

GAMECONTROLLER（游戏控制器，提供标准化手柄映射的高级接口）建立在 JOYSTICK（游戏摇杆，提供原始手柄输入的低级接口）之上。GAMECONTROLLER 本质上是对 JOYSTICK 数据的重新映射——它读取 JOYSTICK 的原始轴值和按钮状态，然后按照预定义的映射表转换为标准的 Xbox 风格布局。因此，没有 JOYSTICK 就不可能有 GAMECONTROLLER。

而 JOYSTICK 子系统需要通过事件队列向应用程序报告手柄的连接、断开、按键、摇杆等事件。这些事件通过 EVENTS 子系统的事件队列分发。因此 JOYSTICK 依赖 EVENTS。

这形成了一个三级依赖链：

```cpp
// 当你只写了这一行：
SDL_Init(SDL_INIT_GAMECONTROLLER);

// SDL2 内部实际执行的等效操作：
// 1. 初始化 EVENTS      (JOYSTICK 的依赖)
// 2. 初始化 JOYSTICK     (GAMECONTROLLER 的依赖)
// 3. 初始化 GAMECONTROLLER
```

**2. VIDEO → EVENTS**

VIDEO（视频子系统，负责创建窗口和管理显示设备）需要 EVENTS 来分发窗口事件——窗口移动、大小调整、获得/失去焦点、关闭请求等都是通过事件队列传递给应用程序的。没有 EVENTS 子系统，窗口就无法向应用程序报告任何状态变化。

```cpp
// 当你只写了这一行：
SDL_Init(SDL_INIT_VIDEO);

// SDL2 内部实际执行：
// 1. 初始化 EVENTS  (VIDEO 的依赖)
// 2. 初始化 VIDEO
```

**3. 独立子系统**

TIMER（定时器）、AUDIO（音频）、HAPTIC（触觉反馈）、SENSOR（传感器）四个子系统完全独立，不依赖任何其他子系统。它们各自管理自己的硬件资源，不需要事件队列来工作：

- **TIMER** 通过操作系统的定时器 API 直接工作，不产生 SDL 事件
- **AUDIO** 通过回调函数直接接收/发送音频数据，不走事件队列
- **HAPTIC** 直接控制振动马达硬件，不需要事件分发
- **SENSOR** 通过轮询 API（`SDL_SensorGetData`）读取数据，不依赖事件

### 依赖规则总结表

| 初始化标志 | 自动初始化的依赖 | 实际初始化的子系统数量 |
|-----------|-----------------|---------------------|
| `SDL_INIT_TIMER` | 无 | 1 |
| `SDL_INIT_AUDIO` | 无 | 1 |
| `SDL_INIT_VIDEO` | EVENTS | 2 |
| `SDL_INIT_JOYSTICK` | EVENTS | 2 |
| `SDL_INIT_HAPTIC` | 无 | 1 |
| `SDL_INIT_GAMECONTROLLER` | JOYSTICK, EVENTS | 3 |
| `SDL_INIT_EVENTS` | 无 | 1 |
| `SDL_INIT_SENSOR` | 无 | 1 |
| `SDL_INIT_EVERYTHING` | — | 全部 8 个 |

> **注意：** `SDL_INIT_EVERYTHING` 是一个便捷标志，等价于把所有子系统标志做位或（OR）运算。它在开发阶段很方便，但在发布版本中不推荐使用——因为它会初始化程序并不需要的子系统（比如你的程序不用手柄，却初始化了 JOYSTICK 和 GAMECONTROLLER），浪费系统资源并可能触发不必要的权限请求（Android 上尤其明显）。

### 验证依赖关系的代码

下面这个程序可以验证上述依赖关系——它只初始化 VIDEO，然后检查 EVENTS 是否被自动初始化了：

```cpp
#include <SDL.h>
#include <cstdio>

int main(int argc, char* argv[]) {
    // 只显式初始化 VIDEO
    if (SDL_Init(SDL_INIT_VIDEO) < 0) {
        printf("初始化失败: %s\n", SDL_GetError());
        return 1;
    }

    // 检查各子系统的初始化状态
    // SDL_WasInit() 返回指定子系统中已初始化的标志位
    printf("=== 依赖关系验证 ===\n\n");
    printf("显式初始化: SDL_INIT_VIDEO\n\n");

    // VIDEO 应该已初始化（我们显式请求的）
    printf("VIDEO:          %s\n",
           SDL_WasInit(SDL_INIT_VIDEO)          ? "已初始化 ✓" : "未初始化 ✗");
    // EVENTS 应该已初始化（VIDEO 的依赖）
    printf("EVENTS:         %s\n",
           SDL_WasInit(SDL_INIT_EVENTS)         ? "已初始化 ✓ (自动)" : "未初始化 ✗");
    // AUDIO 应该未初始化（独立子系统）
    printf("AUDIO:          %s\n",
           SDL_WasInit(SDL_INIT_AUDIO)          ? "已初始化 ✓" : "未初始化 ✗ (独立)");
    // TIMER 应该未初始化（独立子系统）
    printf("TIMER:          %s\n",
           SDL_WasInit(SDL_INIT_TIMER)          ? "已初始化 ✓" : "未初始化 ✗ (独立)");
    // JOYSTICK 应该未初始化（没有被 VIDEO 拉起）
    printf("JOYSTICK:       %s\n",
           SDL_WasInit(SDL_INIT_JOYSTICK)       ? "已初始化 ✓" : "未初始化 ✗ (独立)");
    // GAMECONTROLLER 应该未初始化
    printf("GAMECONTROLLER: %s\n",
           SDL_WasInit(SDL_INIT_GAMECONTROLLER) ? "已初始化 ✓" : "未初始化 ✗ (独立)");
    // HAPTIC 应该未初始化（独立子系统）
    printf("HAPTIC:         %s\n",
           SDL_WasInit(SDL_INIT_HAPTIC)         ? "已初始化 ✓" : "未初始化 ✗ (独立)");
    // SENSOR 应该未初始化（独立子系统）
    printf("SENSOR:         %s\n",
           SDL_WasInit(SDL_INIT_SENSOR)         ? "已初始化 ✓" : "未初始化 ✗ (独立)");

    SDL_Quit();
    return 0;
}
```

## 引用计数机制

SDL2 的子系统初始化使用了**引用计数**（Reference Counting）机制。这个机制解决了一个现实问题：当多个模块都需要同一个子系统时，谁来负责初始化？谁来负责清理？

### 什么是引用计数？

引用计数是一种资源管理技术。每个子系统维护一个内部计数器：

- 每次调用 `SDL_Init()` 或 `SDL_InitSubSystem()` 初始化该子系统时，计数器 **+1**
- 每次调用 `SDL_QuitSubSystem()` 时，计数器 **-1**
- 只有当计数器减到 **0** 时，子系统才会**真正关闭**
- `SDL_Quit()` 无视引用计数，**强制关闭所有子系统**

这个机制类似于 C++ 中的 `std::shared_ptr`——多个持有者共享同一个资源，最后一个释放者负责真正销毁资源。

### 引用计数的行为演示

下面这个程序详细展示了引用计数的工作过程：

```cpp
#include <SDL.h>
#include <cstdio>

// 辅助函数：打印 EVENTS 子系统的状态
void printEventsStatus(const char* step) {
    bool inited = SDL_WasInit(SDL_INIT_EVENTS) != 0;
    printf("[%s] EVENTS 子系统: %s\n", step, inited ? "运行中" : "已关闭");
}

int main(int argc, char* argv[]) {
    printf("=== SDL2 引用计数演示 ===\n\n");

    // 步骤 1：初始化 VIDEO（自动拉起 EVENTS，EVENTS 引用计数 = 1）
    SDL_Init(SDL_INIT_VIDEO);
    printEventsStatus("步骤1: Init VIDEO");
    // EVENTS 引用计数：1（被 VIDEO 自动初始化）

    // 步骤 2：显式再初始化 EVENTS（EVENTS 引用计数 = 2）
    SDL_InitSubSystem(SDL_INIT_EVENTS);
    printEventsStatus("步骤2: Init EVENTS");
    // EVENTS 引用计数：2（VIDEO 拉起的 1 次 + 我们显式的 1 次）

    // 步骤 3：再次初始化 EVENTS（EVENTS 引用计数 = 3）
    SDL_InitSubSystem(SDL_INIT_EVENTS);
    printEventsStatus("步骤3: Init EVENTS 再次");
    // EVENTS 引用计数：3

    // 步骤 4：退出 EVENTS 一次（引用计数 3 → 2，不会真正关闭）
    SDL_QuitSubSystem(SDL_INIT_EVENTS);
    printEventsStatus("步骤4: Quit EVENTS (3→2)");
    // EVENTS 引用计数：2，仍在运行

    // 步骤 5：再退出一次（引用计数 2 → 1，仍不会关闭）
    SDL_QuitSubSystem(SDL_INIT_EVENTS);
    printEventsStatus("步骤5: Quit EVENTS (2→1)");
    // EVENTS 引用计数：1，仍在运行

    // 步骤 6：最后一次退出（引用计数 1 → 0，真正关闭）
    SDL_QuitSubSystem(SDL_INIT_EVENTS);
    printEventsStatus("步骤6: Quit EVENTS (1→0)");
    // EVENTS 引用计数：0，真正关闭！

    // 步骤 7：VIDEO 此时还在吗？
    bool videoAlive = SDL_WasInit(SDL_INIT_VIDEO) != 0;
    printf("[步骤7] VIDEO 子系统: %s\n", videoAlive ? "运行中" : "已关闭");
    // VIDEO 仍在运行——它的引用计数是独立的

    printf("\n注意：EVENTS 被关闭后，VIDEO 仍在运行，\n"
           "但 VIDEO 此时无法正常分发窗口事件！\n"
           "这是一个典型的引用计数误用场景。\n");

    // 清理：SDL_Quit() 强制关闭一切
    SDL_Quit();
    printf("\n[最终] SDL_Quit() 后一切已清理。\n");

    return 0;
}
```

运行输出：

```
=== SDL2 引用计数演示 ===

[步骤1: Init VIDEO] EVENTS 子系统: 运行中
[步骤2: Init EVENTS] EVENTS 子系统: 运行中
[步骤3: Init EVENTS 再次] EVENTS 子系统: 运行中
[步骤4: Quit EVENTS (3→2)] EVENTS 子系统: 运行中
[步骤5: Quit EVENTS (2→1)] EVENTS 子系统: 运行中
[步骤6: Quit EVENTS (1→0)] EVENTS 子系统: 已关闭
[步骤7] VIDEO 子系统: 运行中

注意：EVENTS 被关闭后，VIDEO 仍在运行，
但 VIDEO 此时无法正常分发窗口事件！
这是一个典型的引用计数误用场景。
```

### 引用计数的陷阱

上面的演示暴露了一个重要陷阱：**引用计数只管"谁还持有"，不管"谁还需要"**。步骤 6 中 EVENTS 被关闭了，但 VIDEO 还在运行。VIDEO 依赖 EVENTS 来分发窗口事件，所以 EVENTS 关闭后 VIDEO 虽然"活着"但已经"残废"——窗口无法响应任何操作系统事件。

**正确的做法：** Init 和 Quit 必须成对出现。如果你用 `SDL_InitSubSystem()` 初始化了 N 次，就必须用 `SDL_QuitSubSystem()` 退出 N 次。或者，直接用 `SDL_Quit()` 一次性清理（适用于程序退出时）。

```cpp
// 正确：Init 和 Quit 成对
SDL_InitSubSystem(SDL_INIT_EVENTS);
// ... 使用 EVENTS ...
SDL_QuitSubSystem(SDL_INIT_EVENTS);  // 只减少引用计数

// 错误：多 Init 少 Quit → 资源泄漏（虽然 SDL_Quit 最终会清理）
SDL_InitSubSystem(SDL_INIT_EVENTS);
SDL_InitSubSystem(SDL_INIT_EVENTS);
SDL_QuitSubSystem(SDL_INIT_EVENTS);
// 还有 1 次引用没有释放！
```

## SDL_GetError 错误处理

SDL2 采用了类似 C 语言 `errno` 的错误报告机制：当 SDL 函数失败时，它会在内部设置一个**错误字符串**（Error String，即线程局部存储的文本消息），你可以通过 `SDL_GetError()` 获取这个字符串来了解失败原因。

### 错误处理基础

SDL2 函数的返回值约定：

| 返回类型 | 成功 | 失败 |
|---------|------|------|
| `int` | `0` | 负数（通常是 `-1`） |
| 指针 | 非 `NULL` | `NULL` |
| `SDL_bool` | `SDL_TRUE` | `SDL_FALSE` |

```cpp
#include <SDL.h>
#include <cstdio>

int main(int argc, char* argv[]) {
    // 示例 1：int 返回值的错误检查
    if (SDL_Init(SDL_INIT_VIDEO) < 0) {
        // SDL_GetError() 返回最近一次错误的描述字符串
        // 这个字符串是线程局部的，每个线程有自己的错误信息
        printf("SDL 初始化失败: %s\n", SDL_GetError());
        return 1;
    }

    // 示例 2：指针返回值的错误检查
    SDL_Window* window = SDL_CreateWindow(
        "错误处理示例",           // 窗口标题
        SDL_WINDOWPOS_CENTERED,  // X 坐标（居中）
        SDL_WINDOWPOS_CENTERED,  // Y 坐标（居中）
        800, 600,                // 宽度、高度
        SDL_WINDOW_SHOWN         // 窗口标志
    );
    if (window == NULL) {
        printf("创建窗口失败: %s\n", SDL_GetError());
        SDL_Quit();
        return 1;
    }

    printf("窗口创建成功！\n");

    // 示例 3：SDL_ClearError() 清除错误字符串
    // 在连续操作之间调用，避免读到旧的错误信息
    SDL_ClearError();

    // 尝试一个可能失败的操作
    SDL_Surface* surface = SDL_LoadBMP("不存在的文件.bmp");
    if (surface == NULL) {
        printf("加载 BMP 失败: %s\n", SDL_GetError());
        // 这里不需要退出——加载图片失败不是致命错误
    }

    SDL_DestroyWindow(window);
    SDL_Quit();
    return 0;
}
```

### SDL_GetError 的重要特性

`SDL_GetError()` 有几个容易被忽略的特性：

**1. 线程局部存储**

每个线程有自己独立的错误字符串。在线程 A 中调用 `SDL_GetError()` 不会返回线程 B 中设置的错误。这意味着在多线程程序中，你必须在发生错误的**同一线程**中立即调用 `SDL_GetError()`。

**2. 不会自动清除**

错误字符串在下一次 SDL 错误发生之前会一直保留。这意味着如果你在操作成功后调用 `SDL_GetError()`，可能会得到**之前某次失败**遗留的错误信息。所以正确的做法是：**先检查返回值，确认失败后再调用 `SDL_GetError()`**。

```cpp
// 错误做法：不检查返回值就读错误信息
SDL_Init(SDL_INIT_VIDEO);
printf("错误: %s\n", SDL_GetError());  // 可能打印旧的错误信息！

// 正确做法：先检查返回值
if (SDL_Init(SDL_INIT_VIDEO) < 0) {
    printf("错误: %s\n", SDL_GetError());  // 确实是本次失败的信息
}
```

**3. 返回空字符串而非 NULL**

当没有错误时，`SDL_GetError()` 返回空字符串 `""`，而不是 `NULL`。所以用 `strlen(SDL_GetError()) > 0` 来检查是否有错误是安全的，但不推荐——应该始终通过函数返回值判断成功与否。

### 安全初始化封装函数

在实际项目中，我们通常会封装 SDL 的初始化调用，让错误处理更一致：

```cpp
#include <SDL.h>
#include <cstdio>
#include <cstdlib>

// 安全初始化 SDL 主系统
// 参数 flags：子系统标志的组合（如 SDL_INIT_VIDEO | SDL_INIT_AUDIO）
// 失败时打印错误信息并退出程序
void initSDL(Uint32 flags) {
    if (SDL_Init(flags) < 0) {
        fprintf(stderr, "[致命错误] SDL 初始化失败: %s\n", SDL_GetError());
        exit(EXIT_FAILURE);  // 初始化失败是不可恢复的
    }
}

// 安全初始化单个子系统
// 参数 flag：单个子系统标志（如 SDL_INIT_HAPTIC）
// 参数 name：子系统名称（用于日志输出）
// 返回值：true 表示成功，false 表示失败
// 与 initSDL 不同，这个函数允许失败（可选子系统）
bool initSubSystem(Uint32 flag, const char* name) {
    if (SDL_InitSubSystem(flag) < 0) {
        fprintf(stderr, "[警告] %s 子系统初始化失败: %s\n",
                name, SDL_GetError());
        SDL_ClearError();  // 清除错误，避免影响后续检查
        return false;
    }
    printf("[信息] %s 子系统初始化成功\n", name);
    return true;
}

// 安全创建窗口的封装
// 失败时返回 NULL 并打印详细错误信息
SDL_Window* createWindow(const char* title, int w, int h, Uint32 flags) {
    SDL_Window* window = SDL_CreateWindow(
        title,
        SDL_WINDOWPOS_CENTERED,
        SDL_WINDOWPOS_CENTERED,
        w, h, flags
    );
    if (window == NULL) {
        fprintf(stderr, "[错误] 创建窗口 '%s' (%dx%d) 失败: %s\n",
                title, w, h, SDL_GetError());
    }
    return window;
}

int main(int argc, char* argv[]) {
    // 第一步：核心初始化（失败则退出）
    initSDL(SDL_INIT_VIDEO | SDL_INIT_AUDIO);
    printf("核心子系统初始化完成\n\n");

    // 第二步：可选子系统（失败则降级）
    bool hasHaptic = initSubSystem(SDL_INIT_HAPTIC, "HAPTIC");
    bool hasController = initSubSystem(
        SDL_INIT_GAMECONTROLLER, "GAMECONTROLLER"
    );
    bool hasSensor = initSubSystem(SDL_INIT_SENSOR, "SENSOR");

    printf("\n=== 功能可用性报告 ===\n");
    printf("振动反馈: %s\n", hasHaptic ? "可用" : "不可用");
    printf("手柄支持: %s\n", hasController ? "可用" : "不可用");
    printf("传感器:   %s\n", hasSensor ? "可用" : "不可用");

    // 第三步：创建窗口
    SDL_Window* window = createWindow("安全初始化示例", 640, 480,
                                       SDL_WINDOW_SHOWN);
    if (window == NULL) {
        SDL_Quit();
        return 1;
    }

    // 主循环（简化版）
    printf("\n窗口已创建。按关闭按钮退出。\n");
    bool running = true;
    SDL_Event event;
    while (running) {
        while (SDL_PollEvent(&event)) {
            if (event.type == SDL_QUIT) {
                running = false;
            }
        }
        SDL_Delay(16);  // 约 60 FPS
    }

    // 清理
    SDL_DestroyWindow(window);
    SDL_Quit();
    return 0;
}
```

## 本节小结

- SDL2 子系统之间存在依赖关系：GAMECONTROLLER → JOYSTICK → EVENTS，VIDEO → EVENTS，其他子系统（TIMER、AUDIO、HAPTIC、SENSOR）完全独立
- SDL2 的子系统初始化使用引用计数机制：每次 Init 计数 +1，每次 QuitSubSystem 计数 -1，减到 0 才真正关闭；`SDL_Quit()` 无视引用计数强制关闭一切
- `SDL_GetError()` 是线程局部的，不会自动清除，必须在检查返回值确认失败后才调用
- 封装 `initSDL()` 和 `initSubSystem()` 函数可以让错误处理更一致，区分致命错误（退出）和可恢复错误（降级）

## 练习题与答案

### 题目 1：依赖关系分析

假设你执行以下代码：

```cpp
SDL_Init(SDL_INIT_GAMECONTROLLER | SDL_INIT_AUDIO);
```

请回答：
1. 执行完毕后，哪些子系统处于已初始化状态？
2. EVENTS 子系统的引用计数是多少？
3. 如果之后执行 `SDL_QuitSubSystem(SDL_INIT_JOYSTICK)`，JOYSTICK 会被关闭吗？

<details>
<summary>查看答案</summary>

**1.** 已初始化的子系统有 5 个：
- GAMECONTROLLER（显式请求）
- JOYSTICK（GAMECONTROLLER 的直接依赖，自动初始化）
- EVENTS（JOYSTICK 的依赖，自动初始化）
- AUDIO（显式请求）
- 注意：TIMER、VIDEO、HAPTIC、SENSOR 均未初始化

**2.** EVENTS 的引用计数是 **1**。虽然 GAMECONTROLLER → JOYSTICK → EVENTS 形成了依赖链，但 SDL2 内部在处理依赖时，EVENTS 只被初始化一次（JOYSTICK 初始化时拉起 EVENTS，计数 +1）。GAMECONTROLLER 依赖 JOYSTICK 而不是直接依赖 EVENTS，所以 EVENTS 不会被多次计数。

**3.** 取决于 JOYSTICK 的引用计数。JOYSTICK 被 GAMECONTROLLER 自动初始化，引用计数为 1。调用 `SDL_QuitSubSystem(SDL_INIT_JOYSTICK)` 会将计数减为 0，所以 **JOYSTICK 会被关闭**。这会导致 GAMECONTROLLER 无法正常工作（它的底层依赖被移除了）。这是一个典型的误用场景——你不应该单独关闭被其他活跃子系统依赖的子系统。

</details>

### 题目 2：引用计数推理

阅读以下代码，预测每行注释处 EVENTS 子系统的引用计数和状态：

```cpp
SDL_Init(SDL_INIT_VIDEO);           // A: EVENTS 引用计数 = ?
SDL_InitSubSystem(SDL_INIT_EVENTS); // B: EVENTS 引用计数 = ?
SDL_Init(SDL_INIT_JOYSTICK);       // C: EVENTS 引用计数 = ?
SDL_QuitSubSystem(SDL_INIT_VIDEO);  // D: EVENTS 引用计数 = ?
SDL_QuitSubSystem(SDL_INIT_EVENTS); // E: EVENTS 引用计数 = ?
SDL_QuitSubSystem(SDL_INIT_EVENTS); // F: EVENTS 引用计数 = ?
```

<details>
<summary>查看答案</summary>

- **A:** 引用计数 = **1**。`SDL_Init(SDL_INIT_VIDEO)` 初始化 VIDEO 时自动拉起 EVENTS，EVENTS 计数从 0 变为 1。
- **B:** 引用计数 = **2**。显式初始化 EVENTS，计数 +1。
- **C:** 引用计数 = **3**。`SDL_Init(SDL_INIT_JOYSTICK)` 初始化 JOYSTICK 时自动拉起 EVENTS，计数 +1。
- **D:** 引用计数 = **2**。`SDL_QuitSubSystem(SDL_INIT_VIDEO)` 在关闭 VIDEO 时**会**递减 EVENTS 的引用计数（因为 VIDEO 初始化时增加了 EVENTS 的引用计数）。所以 EVENTS 引用计数从 3 减为 2。
- **E:** 引用计数 = **1**。显式 Quit 一次，计数 -1。EVENTS 仍在运行。
- **F:** 引用计数 = **0**。计数减到 0，EVENTS **真正关闭**。此时 JOYSTICK 虽然还"活着"，但已无法正常分发事件。

</details>

### 题目 3：错误处理实战

以下代码有三处错误处理问题，找出它们并解释为什么有问题：

```cpp
SDL_Init(SDL_INIT_VIDEO);
printf("错误信息: %s\n", SDL_GetError());

SDL_Window* win = SDL_CreateWindow("Test",
    SDL_WINDOWPOS_CENTERED, SDL_WINDOWPOS_CENTERED,
    800, 600, SDL_WINDOW_SHOWN);

SDL_Surface* bmp = SDL_LoadBMP("image.bmp");
if (bmp == NULL) {
    printf("加载图片失败: %s\n", SDL_GetError());
}
if (win == NULL) {
    printf("创建窗口失败: %s\n", SDL_GetError());
}
```

<details>
<summary>查看答案</summary>

**问题 1（第 1-2 行）：** 没有检查 `SDL_Init()` 的返回值就调用 `SDL_GetError()`。如果 `SDL_Init()` 成功了，`SDL_GetError()` 可能返回之前某次失败遗留的旧错误信息，造成误判。正确做法是先检查返回值：

```cpp
if (SDL_Init(SDL_INIT_VIDEO) < 0) {
    printf("错误信息: %s\n", SDL_GetError());
    return 1;
}
```

**问题 2（第 8-10 行 vs 第 11-13 行）：** 错误检查顺序不对。`SDL_CreateWindow()` 在 `SDL_LoadBMP()` 之前调用，但错误检查把 `win == NULL` 放在了 `bmp == NULL` 之后。如果 `bmp` 加载失败，`SDL_GetError()` 会被设置为 BMP 加载的错误信息。之后再检查 `win`，即使 `win` 确实为 NULL，此时 `SDL_GetError()` 返回的已经是 BMP 的错误信息了，而不是窗口创建的错误信息。

正确做法：**每次可能失败的操作之后立即检查**，不要延后：

```cpp
SDL_Window* win = SDL_CreateWindow(...);
if (win == NULL) {
    printf("创建窗口失败: %s\n", SDL_GetError());
    // 处理错误
}

SDL_Surface* bmp = SDL_LoadBMP("image.bmp");
if (bmp == NULL) {
    printf("加载图片失败: %s\n", SDL_GetError());
}
```

**问题 3：** `SDL_Init()` 如果成功了但 `SDL_CreateWindow()` 失败，代码没有调用 `SDL_Quit()` 就继续执行了（在 bmp 检查之后）。应该在窗口创建失败时清理 SDL 并返回。

</details>
