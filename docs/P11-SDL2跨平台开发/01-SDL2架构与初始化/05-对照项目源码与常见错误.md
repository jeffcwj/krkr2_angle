## 对照项目源码

让我们看看 KrKr2 项目中 SDL2 初始化是怎么做的。答案可能出乎意料——KrKr2 几乎不直接操作 SDL2。

### tvpsdl.cpp — 引擎侧的 SDL2 使用

文件路径：`cpp/core/environ/sdl/tvpsdl.cpp`

这个文件是 KrKr2 引擎中唯一与 SDL2 相关的源文件，但它的内容出人意料地简单——**没有 `#include <SDL.h>`**。整个文件只有约 50 行，主要做的是引擎模块自身的初始化（注册内部回调、设置引擎状态），完全不涉及 SDL2 子系统的初始化。

为什么？因为 KrKr2 采用了职责分离设计：

```
┌─────────────────────────────────────────┐
│  应用层：KrKr2 引擎                      │
│  职责：TJS2 脚本执行、图层管理、           │
│       音视频解码、插件系统                 │
│  不关心：窗口创建、事件循环、GL 上下文     │
├─────────────────────────────────────────┤
│  框架层：Cocos2d-x                       │
│  职责：窗口创建、渲染循环、触摸/键盘事件   │
│  管理：SDL2 初始化与清理                  │
├─────────────────────────────────────────┤
│  底层：SDL2                              │
│  职责：跨平台硬件抽象                     │
└─────────────────────────────────────────┘
```

KrKr2 引擎代码只和 Cocos2d-x 的 API 交互（如 `cocos2d::Director`、`cocos2d::Scene`），而 Cocos2d-x 内部负责调用 SDL2 的 `SDL_Init()`、`SDL_CreateWindow()`、`SDL_GL_CreateContext()` 等底层函数。

### AppDelegate.cpp — Cocos2d-x 的生命周期

文件路径：`cpp/core/environ/cocos2d/AppDelegate.cpp`

这是 Cocos2d-x 应用的入口点。`TVPAppDelegate` 继承自 `cocos2d::Application`（Cocos2d-x 提供的应用基类，管理 SDL2 窗口和主循环）。它的 `applicationDidFinishLaunching()` 方法是应用启动后第一个被调用的用户代码。但即使在这里，你也看不到 `SDL_Init()` 的直接调用——因为 Cocos2d-x 的 `Application::run()` 方法在调用 `applicationDidFinishLaunching()` 之前，已经完成了 SDL2 的初始化。

启动流程：

```
platforms/*/main.cpp
  └→ TVPAppDelegate::run()          // 继承自 cocos2d::Application
       └→ Cocos2d-x 内部:
            ├→ SDL_Init(SDL_INIT_VIDEO | SDL_INIT_AUDIO | ...)
            ├→ SDL_CreateWindow(...)
            ├→ SDL_GL_CreateContext(...)
            └→ applicationDidFinishLaunching()  // 这里才是 KrKr2 代码
                 └→ TJS2 引擎初始化
                 └→ 脚本加载
                 └→ 游戏开始
```

### 职责分离的好处

这种设计对你自己的项目有什么启发？

1. **引擎代码不依赖特定底层库**：如果将来把 SDL2 换成其他库（如 GLFW），KrKr2 引擎代码几乎不需要改动
2. **初始化逻辑集中**：SDL2 的初始化、错误处理、清理都由 Cocos2d-x 一个地方管理，不会散落在各处
3. **跨平台差异被封装**：不同平台的 SDL2 行为差异（如 Android 需要通过 JNI 桥接）由 Cocos2d-x 的平台层处理，引擎层看不到这些差异

在你自己的 SDL2 项目中，即使不使用 Cocos2d-x，也建议将 SDL2 初始化封装在一个独立的模块中（比如前面章节的 `initSDL()` 或 `initSDLLayered()` 函数），而不是散落在 `main()` 里。

### 跨平台启动入口对比

KrKr2 的四个平台入口文件都遵循相同的模式——构造 `TVPAppDelegate` 然后调用 `run()`：

| 平台 | 入口文件 | 入口函数 | 特殊处理 |
|------|----------|----------|----------|
| Windows | `platforms/windows/main.cpp` | `WinMain()` | Windows 资源文件(.rc)、DPI 感知 |
| Linux | `platforms/linux/main.cpp` | `main()` | 无特殊处理 |
| macOS | `platforms/apple/macos/main.cpp` | `main()` | Objective-C 自动释放池 |
| Android | `platforms/android/cpp/krkr2_android.cpp` | JNI 入口 | SDL2 通过 JNI 桥接 Java 层 |

所有平台最终都执行：

```cpp
TVPAppDelegate app;
return app.run();  // Cocos2d-x 内部初始化 SDL2 → 调用用户代码
```

这就是职责分离的威力——四个平台的差异被压缩到几行入口代码里，引擎核心代码完全一致。

## 常见错误及解决方案

### 错误 1：散落的初始化导致引用计数混乱

**症状：** 程序退出时某些子系统没有被正确清理，或者在运行中途调用 `SDL_QuitSubSystem()` 后某个功能突然失效。

**原因：** 多个模块各自调用 `SDL_Init()` 或 `SDL_InitSubSystem()`，没有统一管理，导致引用计数（上一节讲过的 SDL2 内部机制：每次 Init 计数 +1，每次 QuitSubSystem 计数 -1）不可预测。

```cpp
// ❌ 错误示范：三个不同的 .cpp 文件各自初始化

// renderer.cpp
void initRenderer() {
    SDL_Init(SDL_INIT_VIDEO);  // VIDEO 引用计数 +1
}

// audio.cpp
void initAudio() {
    SDL_Init(SDL_INIT_VIDEO | SDL_INIT_AUDIO);  // VIDEO 引用计数再 +1
}

// input.cpp
void initInput() {
    SDL_InitSubSystem(SDL_INIT_EVENTS);  // 但 VIDEO 已经拉起了 EVENTS...
}
```

**解决方案：** 统一入口，只在一个地方初始化：

```cpp
// ✅ 正确做法：一个函数管所有初始化
bool initAll(SDLCapabilities& caps) {
    return initSDLLayered(caps);  // 使用上一节的分层初始化函数
}

// 各模块只检查能力，不自行初始化
void initRenderer(const SDLCapabilities& caps) {
    if (!caps.hasVideo) {
        printf("VIDEO 不可用，无法初始化渲染器\n");
        return;
    }
    // 直接使用 VIDEO 子系统，不需要再调用 SDL_Init
}
```

### 错误 2：在子系统初始化前调用 API

**症状：** `SDL_CreateWindow()` 返回 NULL，`SDL_GetError()` 报 "Video subsystem has not been initialized"。

**原因：** 忘记调用 `SDL_Init()` 就直接调用了需要子系统的函数。这在复杂项目中容易发生——初始化逻辑在一个文件，使用 API 在另一个文件，两者的执行顺序出了问题。

```cpp
// ❌ 错误：没有初始化就创建窗口
int main(int argc, char* argv[]) {
    // 忘了 SDL_Init(SDL_INIT_VIDEO);
    SDL_Window* window = SDL_CreateWindow(  // 返回 NULL
        "Test", 0, 0, 800, 600, SDL_WINDOW_SHOWN);
    // SDL_GetError() -> "Video subsystem has not been initialized"
}
```

**解决方案：** 在程序最开头就调用初始化函数，并检查返回值：

```cpp
// ✅ 正确：先初始化，检查成功后再调用 API
int main(int argc, char* argv[]) {
    if (SDL_Init(SDL_INIT_VIDEO) < 0) {
        printf("初始化失败: %s\n", SDL_GetError());
        return 1;
    }
    // 现在可以安全地创建窗口了
    SDL_Window* window = SDL_CreateWindow(
        "Test",
        SDL_WINDOWPOS_CENTERED, SDL_WINDOWPOS_CENTERED,
        800, 600, SDL_WINDOW_SHOWN);
    if (window == NULL) {
        printf("窗口创建失败: %s\n", SDL_GetError());
        SDL_Quit();
        return 1;
    }
    // ...
    SDL_DestroyWindow(window);
    SDL_Quit();
    return 0;
}
```

**调试技巧：** 如果你不确定某个子系统是否已初始化，可以用 `SDL_WasInit()` 检查：

```cpp
// SDL_WasInit 返回已初始化的子系统标志位
// 传入 0 返回所有已初始化的子系统
// 传入具体标志检查特定子系统
if (SDL_WasInit(SDL_INIT_VIDEO) == 0) {
    printf("VIDEO 子系统尚未初始化！\n");
}
```

### 错误 3：Android 平台权限问题

**症状：** HAPTIC 或 SENSOR 在 Android 上初始化失败，但同样的代码在桌面平台（Windows/Linux/macOS）正常。

**原因：** Android 系统要求某些硬件功能必须在 `AndroidManifest.xml`（Android 应用的声明文件，告诉系统这个应用需要什么权限和硬件）中声明权限。如果没有声明，SDL2 在尝试访问硬件时会被系统拒绝。

**AndroidManifest.xml 权限配置：**

```xml
<manifest xmlns:android="http://schemas.android.com/apk/res/android">
    <!-- 振动权限——HAPTIC 子系统需要 -->
    <uses-permission android:name="android.permission.VIBRATE"/>

    <!-- 声明使用传感器硬件——SENSOR 子系统需要 -->
    <!-- required="false" 表示没有传感器的设备也能安装此应用 -->
    <uses-feature android:name="android.hardware.sensor.accelerometer"
                  android:required="false"/>
    <uses-feature android:name="android.hardware.sensor.gyroscope"
                  android:required="false"/>

    <!-- 如果需要手柄支持 -->
    <uses-feature android:name="android.hardware.gamepad"
                  android:required="false"/>
</manifest>
```

**关键：** `android:required="false"` 很重要。如果设为 `true`（默认值），Google Play 会阻止没有该硬件的设备安装你的应用。设为 `false` 表示"有这个硬件最好，没有也行"——配合分层初始化的降级运行策略。

**KrKr2 项目中的权限配置：**

KrKr2 的 Android 清单文件位于 `platforms/android/app/src/main/AndroidManifest.xml`。由于 KrKr2 通过 Cocos2d-x 管理 SDL2，Cocos2d-x 的 Android 模块已经声明了必要的权限（如 `INTERNET`、`WRITE_EXTERNAL_STORAGE`），SDL2 相关的权限也由 Cocos2d-x 的 Gradle 依赖自动合并。

## 本节小结

- **KrKr2 项目采用职责分离设计**：Cocos2d-x 管理 SDL2 的初始化/清理/事件循环，KrKr2 引擎只在 Cocos2d-x 之上处理游戏逻辑，`tvpsdl.cpp` 甚至没有 `#include <SDL.h>`
- **四个平台的入口文件**都遵循相同的模式：构造 `TVPAppDelegate` → 调用 `run()` → Cocos2d-x 内部初始化 SDL2 → 回调 `applicationDidFinishLaunching()`
- **统一初始化入口**是避免引用计数混乱的最佳实践——所有 `SDL_Init()` 调用集中在一个函数/模块中，其他模块只查询能力而不自行初始化
- **`SDL_WasInit()`** 可以在运行时检查特定子系统是否已初始化，帮助调试初始化顺序问题
- **Android 平台**需要在 `AndroidManifest.xml` 中声明 `VIBRATE` 权限（给 HAPTIC）和 `uses-feature`（给 SENSOR），且建议设置 `required="false"` 配合降级运行

### 题目 1：引用计数与统一初始化

以下代码存在初始化管理问题，请指出问题并给出修复方案：

```cpp
// module_a.cpp
void moduleA_init() {
    SDL_Init(SDL_INIT_VIDEO | SDL_INIT_AUDIO);
}

// module_b.cpp
void moduleB_init() {
    SDL_Init(SDL_INIT_VIDEO | SDL_INIT_JOYSTICK);
}

// module_a.cpp
void moduleA_shutdown() {
    SDL_QuitSubSystem(SDL_INIT_VIDEO | SDL_INIT_AUDIO);
}

// module_b.cpp
void moduleB_shutdown() {
    SDL_Quit();  // 强制关闭一切
}
```

如果执行顺序是 `moduleA_init() → moduleB_init() → moduleA_shutdown() → moduleB_shutdown()`，会发生什么？

<details>
<summary>查看答案</summary>

**问题分析：**

1. `moduleA_init()`：初始化 VIDEO（引用计数=1）、AUDIO（引用计数=1），VIDEO 自动拉起 EVENTS（引用计数=1）
2. `moduleB_init()`：VIDEO 引用计数变为 2，初始化 JOYSTICK（引用计数=1），JOYSTICK 自动拉起 EVENTS（引用计数=2）
3. `moduleA_shutdown()`：`SDL_QuitSubSystem(VIDEO)` 使 VIDEO 引用计数从 2 减为 1（仍可用）。`SDL_QuitSubSystem(AUDIO)` 使 AUDIO 引用计数从 1 减为 0（被关闭）
4. `moduleB_shutdown()`：`SDL_Quit()` 无视所有引用计数，**强制关闭一切**。即使 VIDEO 引用计数还是 1，也直接清零

**核心问题：**
- 两个模块各自初始化 SDL2，导致引用计数不可预测
- `moduleB_shutdown()` 使用 `SDL_Quit()` 强制清理，无视 `moduleA` 可能还需要的子系统
- 如果 `moduleA` 在 `moduleB_shutdown()` 之后还有代码试图使用 VIDEO，会直接崩溃

**修复方案：统一初始化入口**

```cpp
// sdl_manager.cpp — 唯一管理 SDL2 生命周期的模块
static SDLCapabilities g_caps;

bool sdlInit() {
    return initSDLLayered(g_caps);  // 上一节的分层初始化
}

void sdlShutdown() {
    SDL_Quit();  // 只在程序退出时调用一次
}

const SDLCapabilities& sdlGetCaps() {
    return g_caps;  // 其他模块只读取能力，不自行初始化
}

// module_a.cpp — 不再自行初始化
void moduleA_init() {
    if (!sdlGetCaps().hasAudio) return;
    // 使用 AUDIO...
}

// module_b.cpp — 不再自行初始化
void moduleB_init() {
    if (!sdlGetCaps().hasVideo) return;
    // 使用 VIDEO...
}
```

</details>

### 题目 2：RAII 封装 SDL2 初始化

设计一个 `SDLManager` 类，使用 RAII（Resource Acquisition Is Initialization，资源获取即初始化——构造时获取资源、析构时释放资源）模式管理 SDL2 的生命周期。要求：
1. 构造函数执行分层初始化
2. 析构函数调用 `SDL_Quit()`
3. 提供 `isReady()` 方法判断核心子系统是否就绪
4. 提供 `getCaps()` 方法获取能力信息
5. 禁止拷贝（SDL2 只能初始化一次）

<details>
<summary>查看答案</summary>

```cpp
#include <SDL.h>
#include <cstdio>
#include <cstring>

struct SDLCapabilities {
    bool hasVideo;
    bool hasAudio;
    bool hasGameController;
    bool hasHaptic;
    bool hasSensor;
    bool hasTimer;
};

class SDLManager {
public:
    // 构造时执行分层初始化
    SDLManager() : m_ready(false) {
        memset(&m_caps, 0, sizeof(m_caps));
        m_ready = initLayered();
    }

    // 析构时清理——RAII 的核心：离开作用域自动清理
    ~SDLManager() {
        if (m_ready) {
            SDL_Quit();
            printf("[SDLManager] SDL2 已清理\n");
        }
    }

    // 禁止拷贝——SDL2 全局状态不能有两份
    SDLManager(const SDLManager&) = delete;
    SDLManager& operator=(const SDLManager&) = delete;

    // 允许移动（转移所有权的场景）
    SDLManager(SDLManager&& other) noexcept
        : m_caps(other.m_caps), m_ready(other.m_ready) {
        other.m_ready = false;  // 原对象不再负责清理
    }

    // 核心子系统是否就绪
    bool isReady() const { return m_ready; }

    // 获取能力信息（只读引用）
    const SDLCapabilities& getCaps() const { return m_caps; }

private:
    SDLCapabilities m_caps;
    bool m_ready;

    // 与上一节的 initSDLLayered 相同的逻辑
    bool initLayered() {
        // 核心层
        if (SDL_Init(SDL_INIT_VIDEO | SDL_INIT_TIMER) < 0) {
            printf("[FATAL] 核心子系统初始化失败: %s\n",
                   SDL_GetError());
            return false;
        }
        m_caps.hasVideo = true;
        m_caps.hasTimer = true;

        // 可选层
        m_caps.hasAudio =
            (SDL_InitSubSystem(SDL_INIT_AUDIO) == 0);
        m_caps.hasGameController =
            (SDL_InitSubSystem(SDL_INIT_GAMECONTROLLER) == 0);
        m_caps.hasHaptic =
            (SDL_InitSubSystem(SDL_INIT_HAPTIC) == 0);
        m_caps.hasSensor =
            (SDL_InitSubSystem(SDL_INIT_SENSOR) == 0);

        return true;
    }
};

// 使用示例
int main(int argc, char* argv[]) {
    SDLManager sdl;  // 构造时自动初始化

    if (!sdl.isReady()) {
        printf("SDL2 初始化失败，程序退出。\n");
        return 1;
    }

    printf("音频可用: %s\n",
           sdl.getCaps().hasAudio ? "是" : "否");
    printf("手柄可用: %s\n",
           sdl.getCaps().hasGameController ? "是" : "否");

    // 正常运行...
    // 创建窗口、进入事件循环等

    return 0;
    // SDLManager 析构时自动调用 SDL_Quit()
    // 即使中途 return 或抛出异常，析构函数也会执行
}
```

**RAII 的优势：**
- **不会忘记清理**：`SDL_Quit()` 在析构函数中自动调用，无论程序如何退出
- **异常安全**：如果中间代码抛出 C++ 异常，栈展开（stack unwinding，C++ 在异常传播时自动析构局部变量的机制）会触发析构函数
- **作用域控制**：在花括号 `{}` 内创建 `SDLManager`，离开花括号时自动清理
- **禁止拷贝**：通过 `= delete` 阻止意外拷贝，避免重复初始化/清理

</details>

### 题目 3：降级运行实现

编写一个完整的 `initWithFallback()` 函数，在 VIDEO 子系统初始化失败时，尝试使用 `SDL_VIDEODRIVER` 环境变量切换到 "dummy" 驱动（一个不实际显示窗口的虚拟驱动，用于测试或无头环境）。如果 dummy 驱动也失败，才返回错误。

<details>
<summary>查看答案</summary>

```cpp
#include <SDL.h>
#include <cstdio>
#include <cstdlib>

// 尝试使用指定驱动初始化 VIDEO
// driverName 为 NULL 表示使用默认驱动
bool tryVideoDriver(const char* driverName) {
    if (driverName) {
        // SDL_setenv 在所有平台上可用（SDL 封装了 setenv/putenv 的差异）
        // 设置 SDL_VIDEODRIVER 环境变量后，SDL_Init 会优先使用该驱动
        SDL_setenv("SDL_VIDEODRIVER", driverName, 1);
        printf("[INFO] 尝试视频驱动: %s\n", driverName);
    } else {
        printf("[INFO] 尝试默认视频驱动\n");
    }

    if (SDL_Init(SDL_INIT_VIDEO) < 0) {
        printf("[WARN] 视频驱动初始化失败: %s\n", SDL_GetError());
        return false;
    }
    printf("[OK]   视频驱动 '%s' 初始化成功\n",
           SDL_GetCurrentVideoDriver());
    return true;
}

bool initWithFallback() {
    // 第一次尝试：默认驱动（通常是 x11/wayland/windows/cocoa）
    if (tryVideoDriver(NULL)) {
        return true;
    }

    // 清理失败状态
    SDL_Quit();

    // 第二次尝试：dummy 驱动（无头模式，不显示窗口）
    // dummy 驱动在所有平台上都可用，因为它不依赖任何图形硬件
    if (tryVideoDriver("dummy")) {
        printf("[WARN] 运行在 dummy 视频模式，无实际图形输出\n");
        return true;
    }

    SDL_Quit();
    printf("[FATAL] 所有视频驱动均不可用\n");
    return false;
}

int main(int argc, char* argv[]) {
    if (!initWithFallback()) {
        return 1;
    }

    printf("当前视频驱动: %s\n", SDL_GetCurrentVideoDriver());

    // 即使使用 dummy 驱动，也可以创建"窗口"和进行渲染（只是不显示）
    // 这对于自动化测试（CI/CD）很有用
    SDL_Window* window = SDL_CreateWindow(
        "Fallback Test",
        SDL_WINDOWPOS_CENTERED, SDL_WINDOWPOS_CENTERED,
        800, 600, SDL_WINDOW_SHOWN);

    if (window) {
        printf("窗口创建成功（dummy 模式下不会实际显示）\n");
        SDL_DestroyWindow(window);
    }

    SDL_Quit();
    return 0;
}
```

**各平台的驱动回退表：**

| 平台 | 默认驱动 | 可能的回退驱动 |
|------|----------|----------------|
| Linux | x11 或 wayland | dummy, offscreen |
| Windows | windows | dummy |
| macOS | cocoa | dummy |
| Android | android | dummy |

**说明：**
- `SDL_VIDEODRIVER` 环境变量可以强制 SDL2 使用指定的视频驱动
- `dummy` 驱动不需要图形硬件，适合 CI/CD 环境或无头服务器
- 这种回退策略让程序在不同环境下都能运行，只是输出方式不同
