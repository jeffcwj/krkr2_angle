## 术语预览

本节将涉及以下术语，先有个印象，正文中会逐一详细讲解：

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| 前缓冲 | Front Buffer | 显示器正在读取并显示的那块显存 |
| 后缓冲 | Back Buffer | GPU 正在绘制新画面的那块显存，画完后与前缓冲交换 |
| 画面撕裂 | Screen Tearing | GPU 刷新和显示器刷新不同步，导致屏幕上半截显示旧帧、下半截显示新帧 |
| 帧率 | FPS (Frames Per Second) | 每秒渲染并显示的画面数量，越高越流畅 |
| 自适应 VSync | Adaptive VSync | 帧率够高时启用 VSync 防撕裂，帧率不够时关闭 VSync 避免卡顿 |
| 固定时间步长 | Fixed Timestep | 每帧逻辑更新使用固定的时间间隔（如 1/60 秒），不受实际帧率影响 |
| Delta Time | Delta Time | 上一帧到当前帧的实际经过时间，用于可变时间步长下的平滑更新 |

## 双缓冲与 VSync

### 为什么需要双缓冲

如果只有一块显存（单缓冲），GPU 画图和显示器读取图像会争抢同一块内存：
- GPU 画到一半时显示器读取 → 上半截是新画面、下半截是旧画面 → **画面撕裂**

双缓冲的解决方案：
- **前缓冲（Front Buffer）**：显示器正在显示的内容
- **后缓冲（Back Buffer）**：GPU 正在绘制的内容
- 每帧结束时，两个缓冲区交换（Swap）——GPU 的画完了就亮出来，旧的交给 GPU 继续画

```
单缓冲（撕裂风险）：      双缓冲（安全）：
┌────────────────┐      前缓冲：显示器读  后缓冲：GPU 画
│ GPU 边画       │      互不干扰，画完交换
│ 显示器边读     │
└────────────────┘
```

### VSync 的三种模式

`SDL_GL_SetSwapInterval()` 是 SDL2 用于控制缓冲区交换时机的函数。它接受一个整数参数，决定帧交换是否等待显示器的垂直刷新信号（VBlank，即显示器从屏幕底部扫描完毕后短暂的空白期）。返回值为 0 表示成功，-1 表示该模式不支持。

| 模式 | 值 | 行为 | 适用场景 |
|------|-----|------|----------|
| 关闭 | 0 | 画完就换，不等显示器 | 性能测试、需要最低延迟 |
| 标准 | 1 | 等显示器刷新信号再换 | 日常使用，画面最稳定 |
| 自适应 | -1 | 来得及就等，来不及就不等 | 性能时好时差的场景 |

```cpp
// 示例 1：VSync 自适应策略——从最优到最差逐级降级
void setupVSync() {
    // 优先尝试自适应 VSync：帧率够时启用同步，不够时自动关闭
    if (SDL_GL_SetSwapInterval(-1) == 0) {
        printf("自适应 VSync 已启用\n");
        return;
    }
    printf("自适应 VSync 不支持: %s\n", SDL_GetError());

    // 回退到标准 VSync：强制每帧等待 VBlank 信号
    if (SDL_GL_SetSwapInterval(1) == 0) {
        printf("标准 VSync 已启用\n");
        return;
    }
    printf("标准 VSync 也不支持: %s\n", SDL_GetError());

    // 最终回退：无 VSync
    SDL_GL_SetSwapInterval(0);
    printf("VSync 已关闭（无硬件支持）\n");
}
```

### 三种 VSync 模式的帧率表现（60Hz 显示器）

| GPU 能力 | VSync 关(0) | VSync 开(1) | 自适应(-1) |
|----------|-------------|-------------|------------|
| 能跑 120帧 | 显示 120帧⚠️撕裂 | 锁定 60帧✅ | 锁定 60帧✅ |
| 能跑 45帧 | 显示 45帧⚠️撕裂 | 锁定 30帧❌腰斩 | 显示 45帧⚠️偶有撕裂 |

标准 VSync 在帧率低于 60 时直接降到 30——它只能在 VBlank（显示器扫描完一帧后的短暂空白期）时交换，错过就等下一个（60Hz 下等 2 个周期 = 30帧）。自适应 VSync 此时跳过同步直接交换。

## 完整渲染循环

以下是一个完整的 SDL2 + OpenGL 渲染循环示例，使用系统自带的 GL 头文件（不依赖 GLEW/GLAD 等 GL 加载库）：

```cpp
#include <SDL.h>

#if defined(__ANDROID__)
    #include <GLES3/gl3.h>      // Android: OpenGL ES 3.0
#elif defined(__APPLE__)
    #include <OpenGL/gl3.h>     // macOS: 系统 GL 头文件
#else
    #include <GL/gl.h>          // Windows/Linux: 基础 GL（实际项目建议用 GLAD）
#endif

#include <cstdio>
#include <cmath>

// 示例 2：完整的 SDL2 + OpenGL 渲染循环（含帧率统计）
int main(int argc, char* argv[]) {
    // --- 初始化 ---
    if (SDL_Init(SDL_INIT_VIDEO) < 0) {
        printf("SDL 初始化失败: %s\n", SDL_GetError());
        return 1;
    }

    // 设置 GL 属性（必须在创建窗口之前）
    SDL_GL_SetAttribute(SDL_GL_CONTEXT_MAJOR_VERSION, 3);
    SDL_GL_SetAttribute(SDL_GL_CONTEXT_MINOR_VERSION, 3);
    SDL_GL_SetAttribute(SDL_GL_CONTEXT_PROFILE_MASK,
                        SDL_GL_CONTEXT_PROFILE_CORE);
    SDL_GL_SetAttribute(SDL_GL_DOUBLEBUFFER, 1);  // 启用双缓冲
    SDL_GL_SetAttribute(SDL_GL_DEPTH_SIZE, 24);   // 24位深度缓冲
    SDL_GL_SetAttribute(SDL_GL_STENCIL_SIZE, 8);  // 8位模板缓冲

    // 创建窗口（必须带 SDL_WINDOW_OPENGL 标志）
    SDL_Window* window = SDL_CreateWindow(
        "OpenGL 渲染循环",
        SDL_WINDOWPOS_CENTERED, SDL_WINDOWPOS_CENTERED,
        800, 600,
        SDL_WINDOW_OPENGL | SDL_WINDOW_RESIZABLE
    );
    if (!window) {
        printf("窗口创建失败: %s\n", SDL_GetError());
        SDL_Quit();
        return 1;
    }

    // 创建 GL 上下文
    SDL_GLContext glContext = SDL_GL_CreateContext(window);
    if (!glContext) {
        printf("GL 上下文创建失败: %s\n", SDL_GetError());
        SDL_DestroyWindow(window);
        SDL_Quit();
        return 1;
    }

    // 设置 VSync
    SDL_GL_SetSwapInterval(1);  // 标准 VSync

    // --- 渲染循环 ---
    bool running = true;          // 控制主循环
    Uint32 frameCount = 0;        // 帧计数器
    Uint32 lastTick = SDL_GetTicks();  // 上一次计时（毫秒）

    while (running) {
        // 步骤1：处理事件——必须在每帧开头调用
        SDL_Event event;
        while (SDL_PollEvent(&event)) {
            switch (event.type) {
                case SDL_QUIT:
                    // 用户关闭窗口（点叉或 Alt+F4）
                    running = false;
                    break;

                case SDL_WINDOWEVENT:
                    if (event.window.event == SDL_WINDOWEVENT_RESIZED) {
                        // 窗口大小改变时，更新 GL 视口以匹配新尺寸
                        int w = event.window.data1;
                        int h = event.window.data2;
                        glViewport(0, 0, w, h);
                        printf("视口调整为 %dx%d\n", w, h);
                    }
                    break;

                case SDL_KEYDOWN:
                    if (event.key.keysym.sym == SDLK_ESCAPE) {
                        running = false;  // ESC 退出
                    }
                    break;
            }
        }

        // 步骤2：清屏——使用随时间变化的颜色来验证渲染正常工作
        float time = SDL_GetTicks() / 1000.0f;  // 转换为秒
        float r = (sinf(time * 1.0f) + 1.0f) / 2.0f;  // 0~1 之间平滑变化
        float g = (sinf(time * 0.7f) + 1.0f) / 2.0f;
        float b = (sinf(time * 1.3f) + 1.0f) / 2.0f;
        glClearColor(r, g, b, 1.0f);  // 设置清屏颜色（RGBA）
        glClear(GL_COLOR_BUFFER_BIT | GL_DEPTH_BUFFER_BIT);  // 清除颜色和深度

        // 步骤3：在这里绘制你的 3D/2D 内容
        // （本示例只做清屏，实际项目会在这里调用 glDraw*）

        // 步骤4：交换缓冲区——把后缓冲的内容显示到屏幕上
        SDL_GL_SwapWindow(window);

        // 步骤5：帧率统计
        frameCount++;
        Uint32 currentTick = SDL_GetTicks();
        if (currentTick - lastTick >= 1000) {
            // 每秒打印一次 FPS
            printf("FPS: %u\n", frameCount);
            frameCount = 0;
            lastTick = currentTick;
        }
    }

    // --- 清理（顺序：上下文 → 窗口 → SDL）---
    SDL_GL_DeleteContext(glContext);
    SDL_DestroyWindow(window);
    SDL_Quit();
    printf("程序正常退出\n");
    return 0;
}
```

### 渲染循环的五个步骤

每一帧都按固定顺序执行以下五步。顺序不可颠倒——先处理事件再渲染，渲染完毕后再交换缓冲区：

```
┌──────────────────────────────────────────────┐
│  ① SDL_PollEvent()                           │
│  处理所有待处理事件（关闭、键盘、鼠标、窗口调整）  │
├──────────────────────────────────────────────┤
│  ② glClearColor() + glClear()                │
│  清除上一帧的内容，设置背景色                     │
├──────────────────────────────────────────────┤
│  ③ glDraw*() / 绘制调用                       │
│  执行实际的渲染命令（顶点、纹理、着色器）          │
├──────────────────────────────────────────────┤
│  ④ SDL_GL_SwapWindow()                       │
│  前后缓冲区交换，新画面显示到屏幕                  │
├──────────────────────────────────────────────┤
│  ⑤ 帧率统计 / 定时逻辑                        │
│  可选：计算 FPS、调节帧间隔                      │
└──────────────────────────────────────────────┘
```

### 关键 GL 函数说明

渲染循环中用到的 GL 函数，这里汇总说明每个函数的参数含义和使用场景：

| 函数 | 参数 | 作用 |
|------|------|------|
| `glViewport(x, y, w, h)` | 视口左下角坐标和宽高（像素） | 设置 GL 渲染区域，窗口大小变化时必须调用 |
| `glClearColor(r, g, b, a)` | RGBA 颜色分量，范围 0.0~1.0 | 设置清屏时使用的背景颜色 |
| `glClear(mask)` | 位掩码，如 `GL_COLOR_BUFFER_BIT \| GL_DEPTH_BUFFER_BIT` | 清除指定缓冲区（颜色/深度/模板） |

`glViewport` 的坐标系原点在左下角（不是左上角）。在高 DPI 显示器上（如 Retina），窗口的"点"尺寸和"像素"尺寸可能不同，应使用 `SDL_GL_GetDrawableSize()` 获取实际像素尺寸：

```cpp
// 示例 3：高 DPI 显示器正确获取视口尺寸
int pixelW, pixelH;
SDL_GL_GetDrawableSize(window, &pixelW, &pixelH);
glViewport(0, 0, pixelW, pixelH);
// 在 Retina 显示器上：pixelW 可能是窗口宽度的 2 倍
```

## 帧率控制策略

渲染循环不仅需要画面不撕裂，还需要控制游戏逻辑的更新速度。有两种主流策略：

### 可变时间步长（Variable Timestep）

每帧根据实际经过的时间（Delta Time，即上一帧到当前帧的时间差）来更新逻辑。物体移动距离 = 速度 × deltaTime，无论帧率高低，单位时间内移动量一致。

```cpp
// 示例 4：可变时间步长——物体速度不受帧率影响
Uint64 lastCounter = SDL_GetPerformanceCounter();  // 高精度计时器
Uint64 frequency = SDL_GetPerformanceFrequency();  // 每秒多少个计数

float playerX = 0.0f;
const float SPEED = 200.0f;  // 像素/秒

while (running) {
    // 计算 deltaTime（秒）
    Uint64 currentCounter = SDL_GetPerformanceCounter();
    float deltaTime = (float)(currentCounter - lastCounter) / frequency;
    lastCounter = currentCounter;

    // 防止 deltaTime 过大（比如断点调试后恢复，deltaTime 可能是几十秒）
    if (deltaTime > 0.1f) {
        deltaTime = 0.1f;  // 最大 100ms
    }

    // 处理事件
    SDL_Event event;
    while (SDL_PollEvent(&event)) {
        if (event.type == SDL_QUIT) running = false;
    }

    // 检查按键状态并更新位置
    const Uint8* keys = SDL_GetKeyboardState(NULL);
    if (keys[SDL_SCANCODE_RIGHT]) {
        playerX += SPEED * deltaTime;  // 无论帧率如何，每秒移动 200 像素
    }
    if (keys[SDL_SCANCODE_LEFT]) {
        playerX -= SPEED * deltaTime;
    }

    // 渲染...
    glClear(GL_COLOR_BUFFER_BIT);
    // drawPlayer(playerX);
    SDL_GL_SwapWindow(window);
}
```

`SDL_GetPerformanceCounter()` 返回高精度计时器值，`SDL_GetPerformanceFrequency()` 返回每秒计数次数，两者相除得到精确秒数。比 `SDL_GetTicks()`（毫秒精度）更适合帧率计算。

### 固定时间步长（Fixed Timestep）

物理模拟和网络同步场景更推荐固定时间步长：每次逻辑更新使用固定间隔（如 1/60 秒），不受渲染帧率影响。渲染快于逻辑时用插值（Interpolation，按时间比例计算中间值）平滑显示；渲染慢于逻辑时一帧内追赶多次逻辑更新。

```cpp
// 示例 5：固定时间步长——逻辑更新与渲染帧率解耦
const float FIXED_DT = 1.0f / 60.0f;  // 逻辑更新固定 60 次/秒
float accumulator = 0.0f;
Uint64 lastCounter = SDL_GetPerformanceCounter();
Uint64 frequency = SDL_GetPerformanceFrequency();
float logicPosX = 0.0f, prevPosX = 0.0f, renderPosX = 0.0f;

while (running) {
    Uint64 currentCounter = SDL_GetPerformanceCounter();
    float frameTime = (float)(currentCounter - lastCounter) / frequency;
    lastCounter = currentCounter;
    if (frameTime > 0.25f) frameTime = 0.25f;  // 防暂停后跳帧
    accumulator += frameTime;

    SDL_Event event;
    while (SDL_PollEvent(&event)) {
        if (event.type == SDL_QUIT) running = false;
    }

    // 逻辑更新：消耗累积时间，每次步进 FIXED_DT
    while (accumulator >= FIXED_DT) {
        prevPosX = logicPosX;
        const Uint8* keys = SDL_GetKeyboardState(NULL);
        if (keys[SDL_SCANCODE_RIGHT]) logicPosX += 200.0f * FIXED_DT;
        accumulator -= FIXED_DT;
    }

    // 插值：alpha 是当前帧在两次逻辑更新之间的进度（0~1）
    float alpha = accumulator / FIXED_DT;
    renderPosX = prevPosX + alpha * (logicPosX - prevPosX);

    glClear(GL_COLOR_BUFFER_BIT);
    // drawPlayer(renderPosX);  // 使用插值位置，画面更平滑
    SDL_GL_SwapWindow(window);
}
```

### 两种策略对比

| 特性 | 可变时间步长 | 固定时间步长 |
|------|-------------|-------------|
| 实现复杂度 | ⭐ 简单 | ⭐⭐⭐ 复杂（需要累加器 + 插值） |
| 物理模拟稳定性 | ❌ 帧率波动会导致物理结果不同 | ✅ 每次步长固定，结果可重现 |
| 网络同步 | ❌ 不同机器帧率不同→状态发散 | ✅ 相同输入→相同结果 |
| 画面平滑度 | ✅ 直接使用实际帧率 | ✅ 需要插值才能平滑 |
| 适用场景 | 简单 2D 游戏、UI 动画 | 物理游戏、格斗游戏、网络对战 |

KrKr2 作为视觉小说引擎，场景更新频率低，使用的是 Cocos2d-x 的 `Director::setAnimationInterval()` 设定帧率间隔（本质上是可变时间步长的变体）。

## 本节小结

- **双缓冲**用两块显存交替显示，避免画面撕裂；通过 `SDL_GL_SetAttribute(SDL_GL_DOUBLEBUFFER, 1)` 启用
- **VSync** 通过 `SDL_GL_SetSwapInterval()` 控制，推荐先尝试自适应（-1）再回退到标准（1）
- 渲染循环五步骤：处理事件 → 清屏 → 绘制 → 交换缓冲区 → 帧率统计
- **可变时间步长**简单但物理不稳定，**固定时间步长**复杂但结果可重现
- 高 DPI 显示器需要用 `SDL_GL_GetDrawableSize()` 获取实际像素尺寸
- `SDL_GetPerformanceCounter()` 比 `SDL_GetTicks()` 精度更高，适合帧率计算

### 题目 1：解释以下代码为什么会导致画面撕裂

```cpp
SDL_GL_SetAttribute(SDL_GL_DOUBLEBUFFER, 1);
// ... 创建窗口和上下文 ...
SDL_GL_SetSwapInterval(0);  // 关闭 VSync

while (running) {
    glClear(GL_COLOR_BUFFER_BIT);
    drawScene();
    SDL_GL_SwapWindow(window);
}
```

<details>
<summary>查看答案</summary>

虽然启用了双缓冲（`SDL_GL_DOUBLEBUFFER, 1`），但 VSync 被关闭了（`SetSwapInterval(0)`）。

双缓冲 + VSync 关闭的情况下：
1. GPU 画完一帧后立即调用 `SDL_GL_SwapWindow` 交换缓冲区
2. 但此时显示器可能正在刷新上一帧（比如正刷到屏幕中间位置）
3. 交换后，显示器上半部分显示旧帧，下半部分显示新帧 → **撕裂**

**双缓冲只保证 GPU 不会画到正在显示的缓冲区上，但不保证交换时机与显示器同步。** VSync 的作用正是让 `SDL_GL_SwapWindow` 等到显示器刷完一整帧后再交换。

**修复方案：** 将 `SDL_GL_SetSwapInterval(0)` 改为 `SDL_GL_SetSwapInterval(1)` 或 `-1`（自适应）。

</details>

### 题目 2：实现一个帧率计数器类

编写一个 `FPSCounter` 类，要求：
- 每秒计算并返回一次 FPS
- 使用 `SDL_GetPerformanceCounter()` 实现高精度计时
- 提供 `tick()` 方法每帧调用，`getFPS()` 方法获取当前 FPS

<details>
<summary>查看答案</summary>

```cpp
#include <SDL.h>
#include <cstdio>

// 帧率计数器类——每秒更新一次 FPS 值
class FPSCounter {
public:
    FPSCounter()
        : frequency_(SDL_GetPerformanceFrequency())  // 获取计时器频率
        , lastTime_(SDL_GetPerformanceCounter())      // 记录起始时间
        , frameCount_(0)
        , currentFPS_(0.0f) {}

    // 每帧调用一次
    void tick() {
        frameCount_++;
        Uint64 now = SDL_GetPerformanceCounter();
        // 计算距上次更新的时间（秒）
        float elapsed = (float)(now - lastTime_) / frequency_;

        if (elapsed >= 1.0f) {
            // 每秒更新一次 FPS
            currentFPS_ = frameCount_ / elapsed;
            frameCount_ = 0;
            lastTime_ = now;
        }
    }

    // 获取当前 FPS
    float getFPS() const { return currentFPS_; }

    // 打印 FPS 到控制台
    void print() const {
        printf("FPS: %.1f\n", currentFPS_);
    }

private:
    Uint64 frequency_;   // 计时器频率（每秒计数次数）
    Uint64 lastTime_;    // 上一次更新 FPS 的时间
    int frameCount_;     // 距上次更新的帧数
    float currentFPS_;   // 当前帧率
};

// 使用方式：
//   FPSCounter fps;
//   while (running) {
//       fps.tick();
//       printf("FPS: %.1f\n", fps.getFPS());
//       // ... 渲染循环 ...
//   }
```

### 题目 3：可变时间步长 vs 固定时间步长

假设有一个赛车游戏，玩家以 100 像素/秒的速度行驶。使用可变时间步长时，在以下两种情况下分别行驶了多少像素？

- 情况 A：稳定 60 FPS，运行 1 秒
- 情况 B：前 0.5 秒 120 FPS，后 0.5 秒 30 FPS，共运行 1 秒

然后回答：如果这是一个网络对战游戏，两台机器分别处于情况 A 和 B，会有什么问题？固定时间步长如何解决？

<details>
<summary>查看答案</summary>

**理论上两种情况都是 100 像素（速度 × 时间 = 100 × 1 = 100）**，可变时间步长的公式 `position += speed * deltaTime` 累加后结果应该相同。

但实际上由于浮点精度误差：
- **情况 A**：60 帧 × (100 × 1/60) ≈ 99.999... 像素
- **情况 B**：60 帧 × (100 × 1/120) + 15 帧 × (100 × 1/30) ≈ 50.0 + 50.0 = 100.0 像素

理论差异微小，但在复杂物理模拟中（碰撞检测、弹簧系统等），帧率差异导致非线性累积误差，两台机器可能出现位置偏差。

**固定时间步长的解决方案：**
- 两台机器都以 1/60 秒为逻辑步长更新
- 无论实际帧率是 30 还是 120，每次逻辑更新都步进相同的时间
- 相同输入 → 相同计算 → 相同结果 → 完美同步
