## 术语预览

本节将涉及以下术语，先有个印象，正文中会逐一详细讲解：

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| 声明式 UI | Declarative UI | 描述"界面应该长什么样"，框架自动计算变化并更新 |
| 命令式 UI | Imperative UI | 手动发出"创建按钮→设置文本→添加到容器"等一步步指令 |
| Widget | Widget | UI 框架中描述界面元素的基本单位，可以是按钮、文本、布局容器等 |
| 即时模式 GUI | Immediate Mode GUI | 每帧重新提交所有 UI 元素的绘制请求，无状态保持——Dear ImGui 的工作方式 |
| 保留模式 GUI | Retained Mode GUI | 创建 UI 元素后保留在内存中，只在变化时更新——Flutter/Qt/Compose 的工作方式 |
| AOT 编译 | Ahead-of-Time Compilation | 在程序发布前就编译成机器码，运行时无需解释——Flutter Release 模式使用 |
| JIT 编译 | Just-in-Time Compilation | 程序运行时动态编译成机器码——Flutter Debug 模式的热重载依赖此技术 |
| 渲染后端 | Rendering Backend | UI 框架底层实际执行图形绘制的引擎（如 Skia、OpenGL、Vulkan） |

## 为什么要了解现代跨平台 UI 框架？

KrKr2 模拟器当前使用 Cocos2d-x 构建用户界面。Cocos2d-x 是一个优秀的 2D 游戏引擎，但作为 **应用级 UI 框架**，它存在明显不足：

| 问题 | 具体表现 |
|------|----------|
| 控件库贫乏 | 只有 Button、Label、ListView 等基础控件，没有下拉菜单、日期选择器、富文本输入框等 |
| 样式系统原始 | 没有 CSS 或主题系统，每个控件的外观都需要手动设置纹理和参数 |
| 无响应式布局 | 不支持自适应不同屏幕尺寸，需要手动计算坐标 |
| 开发效率低 | C++ 编写 UI 代码冗长，修改一个按钮位置需要重新编译 |
| 无障碍支持缺失 | 不支持屏幕阅读器、键盘导航等无障碍功能 |
| 社区萎缩 | Cocos2d-x 已停止活跃维护，UI 部分几乎没有新功能 |

因此，项目的中期目标之一是 **用现代 UI 框架替换 Cocos2d-x 的 UI 层**（保留 Cocos2d-x 的渲染能力用于游戏画面显示）。要做出正确的技术选型，我们需要全面了解当前可用的方案。

## 四大主流方案概览

我们重点考察四种方案，它们代表了跨平台 UI 的四种不同哲学：

```
┌─────────────────────────────────────────────────────────┐
│                   跨平台 UI 方案光谱                       │
│                                                         │
│  轻量/嵌入式 ◄─────────────────────────────► 全功能/独立  │
│                                                         │
│  Dear ImGui        Qt          Compose MP    Flutter    │
│  (即时模式)      (原生控件)    (Kotlin声明式)  (自绘声明式) │
│  C++ 原生        C++ 原生      Kotlin/Native   Dart      │
│  无状态/每帧重绘   有状态/事件驱动  有状态/组合式    有状态/组合式  │
└─────────────────────────────────────────────────────────┘
```

### 方案 A：Flutter

**Flutter** 是 Google 于 2017 年发布的开源 UI 框架。它使用 Dart 语言编写界面代码，通过自带的 Skia 渲染引擎（一个由 Google 维护的高性能 2D 图形库，也被 Chrome 浏览器和 Android 系统使用）在每个平台上自绘所有像素，不依赖平台原生控件。

**核心特征：**

- **语言：** Dart（Google 设计的面向对象语言，语法类似 Java/JavaScript，支持 AOT 编译为原生机器码和 JIT 编译实现热重载）
- **渲染方式：** 完全自绘。Flutter 有自己的渲染引擎（Impeller，正在替代 Skia），所有控件的每一个像素都由 Flutter 自己绘制，不使用平台原生的按钮、文本框等
- **平台覆盖：** Android、iOS、Windows、macOS、Linux、Web
- **声明式 UI 范式：** 开发者描述"界面应该长什么样"，框架负责计算差异并高效更新

**Flutter 的 Hello World：**

```dart
// main.dart — Flutter 的最小应用
import 'package:flutter/material.dart';

void main() {
  runApp(const MyApp()); // 启动 Flutter 应用
}

class MyApp extends StatelessWidget {
  const MyApp({super.key});

  @override
  Widget build(BuildContext context) {
    // build 方法描述"界面长什么样"
    // 每次状态变化时，Flutter 会重新调用 build，计算差异后更新界面
    return MaterialApp(
      home: Scaffold(
        appBar: AppBar(title: const Text('KrKr2 设置')),
        body: Center(
          child: ElevatedButton(
            onPressed: () {
              print('按钮被点击'); // 事件处理
            },
            child: const Text('打开游戏'),
          ),
        ),
      ),
    );
  }
}
```

**关键优势：**
- 热重载（Hot Reload）：修改代码后 1 秒内看到效果，开发效率极高
- 控件库极其丰富：Material Design 和 Cupertino（iOS 风格）两套完整控件
- 性能优秀：AOT 编译后运行速度接近原生
- 社区活跃：pub.dev 上有超过 40000 个第三方包

**关键劣势：**
- 需要学习 Dart 语言（C++ 开发者额外学习成本）
- 应用体积较大（即使空应用也需要 ~10MB 的引擎运行时）
- 与 C++ 互操作需要通过 FFI（Foreign Function Interface）或 Platform Channel，有一定复杂度

### 方案 B：Compose Multiplatform

**Compose Multiplatform** 是 JetBrains 基于 Google 的 Jetpack Compose（Android 官方推荐的声明式 UI 框架）扩展到桌面端和 iOS 的跨平台 UI 方案。它使用 Kotlin 语言编写界面代码，底层通过 Skia 引擎自绘 UI——和 Flutter 一样不依赖平台原生控件。

**核心特征：**

- **语言：** Kotlin（JetBrains 设计的现代语言，Android 官方开发语言，语法比 Java 简洁，支持协程、扩展函数、类型推导等现代特性）
- **渲染方式：** 自绘。桌面端和 iOS 端通过 Skia 渲染（与 Flutter 使用同一个底层绘图引擎）；Android 端直接使用 Android 系统内置的 Compose 支持
- **平台覆盖：** Android、iOS（Beta）、Windows、macOS、Linux、Web（实验性）
- **声明式 UI 范式：** 与 Flutter 类似，开发者描述"界面应该长什么样"，框架自动追踪状态变化并重组（Recomposition，指当数据变化时框架重新执行相关的 UI 描述函数，只更新变化的部分）

**Compose Multiplatform 的 Hello World：**

```kotlin
// Main.kt — Compose Multiplatform 桌面端最小应用
import androidx.compose.foundation.layout.*
import androidx.compose.material3.*
import androidx.compose.runtime.*
import androidx.compose.ui.Alignment
import androidx.compose.ui.Modifier
import androidx.compose.ui.unit.dp
import androidx.compose.ui.window.Window
import androidx.compose.ui.window.application

fun main() = application {
    // application {} 是桌面端的入口函数
    Window(
        onCloseRequest = ::exitApplication,
        title = "KrKr2 设置"
    ) {
        // @Composable 函数描述"界面长什么样"
        MaterialTheme {
            // Column 是垂直布局容器（类似 Flutter 的 Column）
            Column(
                modifier = Modifier.fillMaxSize().padding(16.dp),
                horizontalAlignment = Alignment.CenterHorizontally,
                verticalArrangement = Arrangement.Center
            ) {
                // 用 remember + mutableStateOf 管理状态
                // 当 clickCount 变化时，Compose 自动重组这部分 UI
                var clickCount by remember { mutableStateOf(0) }

                Text(
                    text = "已点击 $clickCount 次",
                    style = MaterialTheme.typography.headlineMedium
                )

                Spacer(modifier = Modifier.height(16.dp))

                Button(onClick = {
                    clickCount++ // 状态变化 → 自动更新 UI
                    println("按钮被点击") // 事件处理
                }) {
                    Text("打开游戏")
                }
            }
        }
    }
}
```

**关键优势：**
- 与 Android 生态无缝衔接：Android 端直接使用 Jetpack Compose，零额外成本
- Kotlin 语言优秀：空安全（Null Safety，编译期阻止空指针异常）、协程（Coroutine，轻量级并发）、与 Java 100% 互操作
- IDE 支持卓越：IntelliJ IDEA / Android Studio 提供实时预览、代码补全、调试工具
- 通过 Kotlin/Native 可编译为原生代码，能与 C 函数互操作（通过 cinterop 机制）

**关键劣势：**
- iOS 支持仍在 Beta 阶段，稳定性和性能尚有不足
- 控件库不如 Flutter 丰富（第三方库生态正在成长中）
- 与 C++ 互操作需要经过 Kotlin/Native → C 的桥接层，比 Flutter FFI 更复杂一层
- 桌面端社区相对较小，遇到问题时可参考的资料较少

### 方案 C：Qt

**Qt**（发音同"cute"）是一个拥有 30 年历史的 C++ 跨平台应用框架，由 The Qt Company 维护。Qt 提供了完整的应用开发能力——不仅包括 UI 控件，还涵盖网络、数据库、多媒体、3D 渲染等。Qt 有两套 UI 系统：传统的 Qt Widgets（命令式 C++ API，使用平台原生渲染后端）和较新的 Qt Quick/QML（声明式标记语言，使用 Scene Graph 自绘渲染）。

**核心特征：**

- **语言：** C++（核心框架）+ QML（Qt 自研的声明式 UI 标记语言，语法类似 JSON + JavaScript）
- **渲染方式：** Qt Widgets 使用平台原生绘图 API（Windows 上用 GDI/Direct2D，macOS 上用 Core Graphics，Linux 上用 X11/Wayland）；Qt Quick 通过 Scene Graph 自绘（底层使用 OpenGL/Vulkan/Metal）
- **平台覆盖：** Windows、macOS、Linux、Android、iOS、嵌入式 Linux、WebAssembly
- **两种范式并存：** Qt Widgets 是命令式的（创建控件对象、设置属性、连接信号槽），Qt Quick 是声明式的（用 QML 描述界面结构）

**Qt Widgets 的 Hello World（命令式 C++ API）：**

```cpp
// main.cpp — Qt Widgets 的最小应用
#include <QApplication>   // 应用程序类
#include <QMainWindow>    // 主窗口类
#include <QPushButton>    // 按钮控件
#include <QVBoxLayout>    // 垂直布局
#include <QLabel>         // 文本标签
#include <QWidget>        // 基础控件类
#include <QDebug>         // 调试输出

int main(int argc, char *argv[]) {
    QApplication app(argc, argv);  // 创建应用实例

    QMainWindow window;
    window.setWindowTitle("KrKr2 设置");  // 设置窗口标题

    // 创建中心部件和布局
    auto *centralWidget = new QWidget(&window);
    auto *layout = new QVBoxLayout(centralWidget);

    // 创建标签 — 命令式地一步步构建 UI
    auto *label = new QLabel("欢迎使用 KrKr2 模拟器");
    label->setAlignment(Qt::AlignCenter);  // 设置对齐方式
    layout->addWidget(label);              // 添加到布局

    // 创建按钮
    auto *button = new QPushButton("打开游戏");
    layout->addWidget(button);

    // 连接信号（Signal）与槽（Slot）— Qt 的核心事件机制
    // 当按钮被点击时（clicked 信号），执行 lambda 函数（槽）
    QObject::connect(button, &QPushButton::clicked, [label]() {
        qDebug() << "按钮被点击";          // 控制台输出
        label->setText("游戏加载中...");    // 更新界面文本
    });

    window.setCentralWidget(centralWidget);
    window.resize(400, 300);   // 设置窗口大小
    window.show();             // 显示窗口

    return app.exec();  // 进入事件循环
}
```

**对应的 CMakeLists.txt：**

```cmake
cmake_minimum_required(VERSION 3.16)
project(krkr2_qt_demo LANGUAGES CXX)

set(CMAKE_CXX_STANDARD 17)
set(CMAKE_AUTOMOC ON)  # 自动运行 MOC（Qt 的元对象编译器，处理信号槽）

find_package(Qt6 REQUIRED COMPONENTS Widgets)

add_executable(krkr2_qt_demo main.cpp)
target_link_libraries(krkr2_qt_demo PRIVATE Qt6::Widgets)
```

**关键优势：**
- C++ 原生框架：无需跨语言桥接，与 KrKr2 的 C++ 代码库天然兼容
- 控件库极其完善：数百个开箱即用的控件，涵盖表格、树形视图、富文本编辑器等复杂组件
- 信号槽（Signal-Slot）机制：类型安全的事件处理系统，比回调函数更易维护
- 成熟的文档和社区：30 年积累，几乎任何问题都能找到答案

**关键劣势：**
- **许可证复杂：** Qt 有 GPLv3（开源免费但要求衍生作品也开源）和商业许可（每年数千美元）两种，LGPL 许可仅覆盖部分模块，对闭源项目有潜在法律风险
- 框架体积大：完整的 Qt 运行时有 50-100MB，远超其他方案
- QML 学习成本：要用好 Qt Quick 需要同时掌握 C++ 和 QML 两种语言
- 移动端体验不如 Flutter：Qt 在 Android/iOS 上的控件外观和交互体验不够"原生"

### 方案 D：Dear ImGui

**Dear ImGui**（常简称 ImGui）是一个轻量级的 C++ GUI 库，采用"即时模式"（Immediate Mode）设计理念。与前三种框架截然不同，ImGui 不保留 UI 元素的持久状态——每一帧都重新提交所有 UI 的绘制请求，由库立即处理并产生绘制命令。这种方式特别适合嵌入到已有渲染循环的应用中（如游戏引擎、3D 编辑器）。

**即时模式 vs 保留模式详解：**

```
┌────────────────────────────────────────────────┐
│          保留模式 GUI（Flutter/Qt/Compose）       │
│                                                │
│  创建按钮对象 → 存储在内存 → 事件回调 → 更新状态  │
│  框架维护一棵 UI 树，只在变化时更新              │
│                                                │
│  优点：性能好（只更新变化部分）                   │
│  缺点：状态同步复杂（UI 状态 ≠ 数据状态时出 bug） │
├────────────────────────────────────────────────┤
│          即时模式 GUI（Dear ImGui）              │
│                                                │
│  每帧：if(按钮("点我")) { 处理点击; }            │
│  无 UI 树，无持久对象，无状态同步问题            │
│                                                │
│  优点：代码简单直接，无状态 bug                   │
│  缺点：每帧重新提交所有 UI（但 ImGui 优化极好）   │
└────────────────────────────────────────────────┘
```

**核心特征：**

- **语言：** C++（原生 C++ 头文件库，也有 C 绑定和几十种语言的第三方绑定）
- **渲染方式：** 即时模式自绘。ImGui 生成顶点缓冲（Vertex Buffer）和绘制命令列表，由宿主程序的渲染后端（OpenGL/Vulkan/DirectX/Metal）执行实际绘制
- **平台覆盖：** 理论上支持任何有图形 API 的平台（通过后端适配）。官方提供 OpenGL2/3、Vulkan、DirectX9/10/11/12、Metal、WebGPU 后端
- **无框架约束：** ImGui 不管理窗口创建和事件循环，这些由宿主程序（如 SDL2、GLFW）负责

**Dear ImGui 的 Hello World（使用 SDL2 + OpenGL3 后端）：**

```cpp
// main.cpp — Dear ImGui + SDL2 + OpenGL3 最小应用
#include "imgui.h"                        // ImGui 核心头文件
#include "imgui_impl_sdl2.h"              // SDL2 平台后端
#include "imgui_impl_opengl3.h"           // OpenGL3 渲染后端
#include <SDL2/SDL.h>                     // SDL2 窗口和事件
#include <SDL2/SDL_opengl.h>              // SDL2 OpenGL 支持
#include <cstdio>

int main(int argc, char *argv[]) {
    // 初始化 SDL2 和 OpenGL
    SDL_Init(SDL_INIT_VIDEO);
    SDL_Window *window = SDL_CreateWindow(
        "KrKr2 设置",                     // 窗口标题
        SDL_WINDOWPOS_CENTERED, SDL_WINDOWPOS_CENTERED,
        640, 480,                          // 窗口尺寸
        SDL_WINDOW_OPENGL | SDL_WINDOW_RESIZABLE
    );
    SDL_GLContext gl_ctx = SDL_GL_CreateContext(window);

    // 初始化 ImGui
    IMGUI_CHECKVERSION();                  // 检查版本兼容性
    ImGui::CreateContext();                // 创建 ImGui 上下文
    ImGui_ImplSDL2_InitForOpenGL(window, gl_ctx);  // 初始化平台后端
    ImGui_ImplOpenGL3_Init("#version 130");         // 初始化渲染后端

    bool running = true;
    int click_count = 0;

    while (running) {
        // 事件处理
        SDL_Event event;
        while (SDL_PollEvent(&event)) {
            ImGui_ImplSDL2_ProcessEvent(&event);  // 传递事件给 ImGui
            if (event.type == SDL_QUIT) running = false;
        }

        // 新帧开始
        ImGui_ImplOpenGL3_NewFrame();
        ImGui_ImplSDL2_NewFrame();
        ImGui::NewFrame();

        // 即时模式 UI — 每帧重新描述所有 UI
        ImGui::Begin("KrKr2 控制面板");           // 开始一个窗口
        ImGui::Text("已点击 %d 次", click_count); // 显示文本
        if (ImGui::Button("打开游戏")) {           // 按钮：返回 true 表示被点击
            click_count++;
            printf("按钮被点击\n");
        }
        ImGui::End();                              // 结束窗口

        // 渲染
        ImGui::Render();
        glClearColor(0.1f, 0.1f, 0.1f, 1.0f);
        glClear(GL_COLOR_BUFFER_BIT);
        ImGui_ImplOpenGL3_RenderDrawData(ImGui::GetDrawData());
        SDL_GL_SwapWindow(window);
    }

    // 清理
    ImGui_ImplOpenGL3_Shutdown();
    ImGui_ImplSDL2_Shutdown();
    ImGui::DestroyContext();
    SDL_GL_DeleteContext(gl_ctx);
    SDL_DestroyWindow(window);
    SDL_Quit();
    return 0;
}
```

**关键优势：**
- 极度轻量：整个库只有几个 .cpp/.h 文件，约 50KB 编译产物，零外部依赖
- C++ 原生：与 KrKr2 代码库零摩擦集成
- 完美嵌入渲染循环：可以直接在 Cocos2d-x 的渲染循环中插入 ImGui 绘制
- 代码即 UI：不需要额外的标记语言或设计工具，C++ 代码就是界面定义
- 广泛应用：Unity、Unreal、Godot 等主流游戏引擎的调试界面都用 ImGui

**关键劣势：**
- 外观简朴：默认主题偏"程序员风格"，不适合面向终端用户的应用
- 缺少高级控件：没有富文本编辑器、日期选择器、复杂表单等应用级组件
- 无障碍支持为零：不支持屏幕阅读器、键盘导航标准
- 不适合内容密集型 UI：如果 UI 很复杂（几百个控件），即时模式的每帧重建开销会变大
- 无官方移动端支持：Android/iOS 需要社区维护的后端，成熟度不如桌面端

## 综合对比表

将四种方案的核心维度汇总到一张表中，方便快速比较：

| 维度 | Flutter | Compose MP | Qt | Dear ImGui |
|------|---------|------------|-----|------------|
| **开发语言** | Dart | Kotlin | C++ / QML | C++ |
| **UI 范式** | 声明式（自绘） | 声明式（自绘） | 命令式 + 声明式 | 即时模式（自绘） |
| **渲染引擎** | Impeller / Skia | Skia | 平台原生 / Scene Graph | 宿主渲染后端 |

| **移动端成熟度** | ★★★★★ | ★★★☆☆ | ★★★☆☆ | ★☆☆☆☆ |

### 维度说明

- **C++ 互操作**：指从 C++ 代码调用框架功能或从框架调用 C++ 代码的难易程度。"原生 C++"表示框架本身就是 C++ 写的，无需跨语言桥接
- **嵌入 C++ 引擎**：指将 UI 框架嵌入到一个已有 C++ 渲染引擎（如 KrKr2 + Cocos2d-x）中的技术复杂度
- **应用体积增量**：指集成该框架后应用包增加的大小
- **热重载（Hot Reload）**：修改 UI 代码后无需重新编译就能看到效果，大幅提升开发效率

## 各方案在 KrKr2 场景下的适用性分析

我们关注的不是"哪个框架最好"，而是"哪个框架最适合嵌入到 KrKr2 这个 C++ 模拟器项目中"。分析三个关键约束：

### 约束 1：必须跨四个平台

KrKr2 需要在 Windows、Linux、macOS、Android 四个平台运行：

| 框架 | Windows | Linux | macOS | Android | 四平台齐全？ |
|------|---------|-------|-------|---------|-------------|
| Flutter | ✅ 稳定 | ✅ 稳定 | ✅ 稳定 | ✅ 稳定 | ✅ |
| Compose MP | ✅ 稳定 | ✅ 稳定 | ✅ 稳定 | ✅ 稳定 | ✅ |
| Qt | ✅ 稳定 | ✅ 稳定 | ✅ 稳定 | ⚠️ 可用但体验一般 | ⚠️ |
| Dear ImGui | ✅ 稳定 | ✅ 稳定 | ✅ 稳定 | ⚠️ 社区后端 | ⚠️ |

### 约束 2：必须能嵌入 Cocos2d-x 渲染循环

KrKr2 的游戏画面由 Cocos2d-x 渲染，UI 框架需要和 Cocos2d-x 共存，最好能在同一个 OpenGL/Vulkan 上下文中渲染：

```
┌──────────────────────────────┐
│     KrKr2 应用主循环          │
│                              │
│  ┌────────────────────────┐  │
│  │  Cocos2d-x 渲染循环     │  │
│  │  ├─ 游戏画面渲染        │  │
│  │  └─ UI 框架渲染 ←─需要 │  │
│  └────────────────────────┘  │
│                              │
│  两者共享同一个 GL 上下文     │
└──────────────────────────────┘
```

- **Dear ImGui**：最容易嵌入，因为它就是设计给嵌入渲染循环用的
- **Qt**：可以通过 `QOpenGLWidget` 共享 GL 上下文，但架构较重
- **Flutter**：通过 Flutter Engine 嵌入 API + Texture Widget 实现纹理共享，复杂但可行
- **Compose MP**：需要 Kotlin/Native + Skia 共享，跨语言桥接最复杂

### 约束 3：许可证兼容性

KrKr2 目前不要求开源（非 GPL 项目），因此：

| 框架 | 许可证 | 兼容性 |
|------|--------|--------|
| Flutter | BSD-3-Clause | ✅ 完全兼容 |
| Compose MP | Apache-2.0 | ✅ 完全兼容 |
| Qt | GPLv3 / LGPL / 商业 | ⚠️ 需谨慎处理动态链接 |
| Dear ImGui | MIT | ✅ 完全兼容 |

## 对照项目源码

KrKr2 项目当前使用 Cocos2d-x 构建 UI，了解现有 UI 代码的结构有助于评估替换方案的工作量。

相关文件：

| 文件路径 | 作用 | 与本节的关联 |
|----------|------|-------------|
| `cpp/core/environ/ui/` | Cocos2d 实现的 UI 表单（设置、文件浏览等） | 这些就是未来需要用新框架替换的 UI 代码 |
| `ui/cocos-studio/` | Cocos Studio 导出的 UI 资源（.csb 二进制布局文件、locale XML 文件） | .csb 文件定义了当前 UI 的布局和控件层次结构 |
| `cpp/core/environ/cocos2d/AppDelegate.cpp` | 应用启动入口，初始化 Cocos2d 引擎 | 新框架需要在这里或附近初始化 |
| `cpp/core/environ/ConfigManager/` | 配置管理器（全局、个人、语言设置） | UI 框架需要与配置系统交互 |

### 当前 UI 的问题实例

打开 `ui/cocos-studio/` 目录，你会看到大量 `.csb` 文件。这些文件是 Cocos Studio（一个已停止维护的可视化 UI 编辑器）导出的二进制格式——无法用文本编辑器查看或修改，也无法版本控制（diff 无意义）。这是 Cocos2d UI 方案的一个实际痛点：UI 布局被锁在二进制工具中。

相比之下，Flutter 的 UI 定义就是 Dart 源代码，Compose 的 UI 定义就是 Kotlin 源代码，都可以用普通文本编辑器编写、用 Git 版本控制、用代码审查工具 Review。

## 常见错误与排查

### 错误 1：Flutter 安装后 `flutter doctor` 报告缺少 Android SDK

**现象：**

```
[✗] Android toolchain - develop for Android devices
    ✗ Unable to locate Android SDK
```

**原因：** Flutter 的 `flutter doctor` 默认检查所有平台支持。即使你只想在桌面端运行，它也会检查 Android SDK。

**解决方案：**

```bash
# 如果你只需要桌面开发，可以忽略 Android 相关警告
# 用 flutter config 禁用不需要的平台
flutter config --no-enable-android   # 禁用 Android 支持检查

# 如果你需要 Android 开发，安装 Android SDK
# 推荐通过 Android Studio 安装（会自动配置环境变量）
```

### 错误 2：Dear ImGui 示例编译时找不到 `imgui_impl_sdl2.h`

**现象：**

```
fatal error: imgui_impl_sdl2.h: No such file or directory
```

**原因：** vcpkg 的 imgui 包默认不包含平台后端（backend）文件，需要在安装时指定 feature。

**解决方案：**

在 `vcpkg.json` 中明确指定需要的后端 feature：

```json
{
    "name": "imgui",
    "features": ["sdl2-binding", "opengl3-binding"]
}
```

或者通过命令行安装：

```bash
vcpkg install imgui[sdl2-binding,opengl3-binding]
```

### 错误 3：Qt 项目编译报 `undefined reference to vtable for ...`

**现象：**

```
undefined reference to `vtable for MyWidget'
```

**原因：** Qt 的信号槽机制依赖 MOC（Meta-Object Compiler，元对象编译器）预处理。如果使用了 `Q_OBJECT` 宏的类没有被 MOC 处理，就会出现 vtable 链接错误。

**解决方案：**

在 CMakeLists.txt 中启用自动 MOC：

```cmake
set(CMAKE_AUTOMOC ON)   # 自动对包含 Q_OBJECT 的头文件运行 MOC
set(CMAKE_AUTOUIC ON)   # 自动处理 .ui 文件（可选）
set(CMAKE_AUTORCC ON)   # 自动处理 .qrc 资源文件（可选）
```

## 本节小结

- **Flutter** 是最全面的跨平台方案，拥有丰富的控件库、出色的开发体验（热重载）和活跃社区，但需要学习 Dart 语言，与 C++ 互操作需要 FFI/Platform Channel
- **Compose Multiplatform** 是 Android 生态的最佳选择，Kotlin 语言现代高效，但 iOS/桌面端仍在追赶 Flutter，与 C++ 互操作需要 Kotlin/Native cinterop
- **Qt** 是 C++ 原生框架中的"全家桶"，控件最丰富、文档最完善，但许可证复杂、体积庞大、移动端体验不如 Flutter
- **Dear ImGui** 是最轻量的选项，完美嵌入渲染循环，但外观简朴、缺少高级控件、无移动端官方支持
- 在 KrKr2 的场景下，**Flutter** 和 **Dear ImGui** 是最值得深入研究的两个方案——Flutter 适合面向用户的正式 UI，ImGui 适合开发者工具和调试界面
- 技术选型不是"最好的"框架，而是"最适合约束条件的"框架——需要综合考虑平台覆盖、嵌入难度、许可证、团队技能等多个维度
- 声明式 UI 是现代 UI 框架的主流趋势，理解声明式 vs 命令式 vs 即时模式的区别有助于学习任何新框架

## 练习题与答案

### 题目 1：声明式 UI 与即时模式 GUI 的核心区别是什么？它们各自适合什么场景？

<details>
<summary>查看答案</summary>

**声明式 UI**（如 Flutter、Compose）：
- 开发者描述"界面应该长什么样"，框架负责计算差异并高效更新
- 框架维护一棵 UI 元素树（Widget Tree），状态变化时只重建变化的部分
- 适合复杂的、状态丰富的应用级 UI（如设置页面、表单、导航流程）
- 优点：状态管理有系统化方案，性能优化由框架处理
- 缺点：框架较重，需要理解状态管理和重建机制

**即时模式 GUI**（如 Dear ImGui）：
- 每帧重新提交所有 UI 元素的绘制请求，不保留 UI 树
- 代码结构：`if (Button("点我")) { 处理点击; }` —— UI 描述和事件处理在同一行
- 适合嵌入式工具 UI（如游戏引擎编辑器、调试面板、性能监控器）
- 优点：代码极其简洁，无状态同步 bug，极易嵌入已有渲染循环
- 缺点：不适合大规模复杂 UI，缺少高级控件，无无障碍支持

**场景对比：**
- KrKr2 的用户设置界面（分辨率选择、按键映射、存档管理）→ 用声明式 UI（Flutter）更合适
- KrKr2 的开发者调试面板（FPS 显示、内存占用、TJS2 变量查看）→ 用即时模式（ImGui）更合适

</details>

### 题目 2：为什么 Qt 的 GPLv3 许可证对 KrKr2 项目构成潜在风险？如果非要用 Qt，有什么规避方法？

<details>
<summary>查看答案</summary>

**风险分析：**

GPLv3（GNU General Public License v3）要求任何使用 GPL 代码的软件也必须以 GPL 许可证发布，并公开完整源代码。如果 KrKr2 未来需要发布闭源版本（例如商业授权），静态链接 Qt 库会使整个项目被"传染"为 GPL。

**规避方法：**

1. **使用 LGPL 模块：** Qt 的核心模块（Qt Core、Qt GUI、Qt Widgets）使用 LGPLv3 许可，允许闭源项目通过动态链接使用。但 Qt Charts、Qt Data Visualization 等附加模块是 GPLv3，不能用
2. **动态链接：** LGPL 允许闭源项目动态链接 LGPL 库（.dll/.so/.dylib），但用户必须能够替换这些库——意味着不能将 Qt 库打包进单一可执行文件
3. **购买商业许可：** Qt 商业许可允许静态链接和闭源分发，但费用不菲（约 $300-500/月/开发者）
4. **使用替代方案：** 选择 Flutter（BSD-3）或 Dear ImGui（MIT）等宽松许可证的框架，从根本上避免许可证问题

对于 KrKr2 这样的开源社区项目，如果始终保持开源，GPL 不是问题；但考虑到可能有用户想二次分发修改版本，宽松许可证更友好。

</details>

### 题目 3：假设你需要为 KrKr2 添加一个"游戏存档管理器"UI 页面（包含存档列表、缩略图预览、删除确认对话框），你会选择哪个框架？为什么？

<details>
<summary>查看答案</summary>

**推荐方案：Flutter**

理由分析：

1. **控件需求复杂：** 存档管理器需要列表视图（ListView）、图片展示（Image）、对话框（AlertDialog）、滑动删除等高级交互——Flutter 的 Material Design 控件库全部原生支持

2. **跨平台一致性：** 存档管理器需要在四个平台上提供一致的用户体验。Flutter 自绘所有像素，保证四平台 UI 完全相同

3. **开发效率：** 使用热重载可以快速迭代 UI 布局和交互。同样的功能，Flutter 的代码量远少于 C++ + Qt Widgets

4. **许可证安全：** BSD-3 许可证无任何限制

**Flutter 实现草案：**

```dart

            onPressed: () => Navigator.pop(ctx),
            child: const Text('取消'),
          ),
          TextButton(
            onPressed: () {
              setState(() => saves.removeAt(index));
              Navigator.pop(ctx);
            },
            child: const Text('删除', style: TextStyle(color: Colors.red)),
          ),
        ],
      ),
    );
  }

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      appBar: AppBar(title: const Text('存档管理')),
      body: ListView.builder(
        itemCount: saves.length,

          final save = saves[index];
          return ListTile(
            leading: const Icon(Icons.save, size: 40), // 缩略图占位
            title: Text(save['name']!),
            subtitle: Text(save['date']!),
            trailing: IconButton(
              icon: const Icon(Icons.delete),
              onPressed: () => _deleteSave(index),
            ),
          );
        },
      ),
    );
  }
}
```

**为什么不选其他方案？**
- **Dear ImGui**：缺少 ListView 滚动惯性、图片缩略图管理、动画对话框等功能，UI 外观过于简朴
- **Qt**：功能上完全可行，但许可证和体积是顾虑
- **Compose MP**：可行，但桌面端生态不如 Flutter 成熟
