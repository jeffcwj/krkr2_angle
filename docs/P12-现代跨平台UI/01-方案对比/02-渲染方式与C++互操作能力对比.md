## 术语预览

本节将涉及以下术语，先有个印象，正文中会逐一详细讲解：

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| 自绘渲染 | Custom Rendering | UI 框架自己绘制每个像素，不使用操作系统的原生控件 |
| 原生控件 | Native Widgets | 操作系统提供的标准 UI 元素（如 Windows 的按钮、macOS 的菜单），外观和行为由 OS 决定 |
| 渲染管线 | Rendering Pipeline | 从 UI 描述（代码/标记）到屏幕上可见像素的完整处理流程 |
| FFI | Foreign Function Interface | 外部函数接口，让一种编程语言直接调用另一种语言编译出的函数 |
| Platform Channel | Platform Channel | Flutter 的跨语言异步通信通道，支持方法调用和事件流 |
| cinterop | C Interop | Kotlin/Native 的 C 语言互操作机制，通过 .def 文件生成 Kotlin 绑定 |
| 信号槽 | Signal-Slot | Qt 的事件通信机制，信号（事件发生）连接到槽（处理函数），类型安全的观察者模式 |
| GL 上下文 | OpenGL Context | OpenGL 的运行环境，包含所有渲染状态、纹理、着色器等资源 |
| 纹理共享 | Texture Sharing | 让两个不同的渲染系统共享同一块 GPU 纹理内存，避免数据拷贝 |

## 两种渲染策略：自绘 vs 原生控件

UI 框架在屏幕上显示控件有两种根本不同的策略。理解这个区别是选型的关键。

### 策略 1：原生控件渲染

**原理：** 框架调用操作系统提供的 UI API 来创建和显示控件。例如，在 Windows 上创建一个按钮，框架调用 Win32 API 的 `CreateWindowEx("BUTTON", ...)`；在 macOS 上创建同一个按钮，调用 AppKit 的 `NSButton`。

```
┌────────────────────────────────────────────────────┐
│              原生控件渲染流程                         │
│                                                    │
│  应用代码                                           │
│    ↓ "创建一个按钮"                                  │
│  UI 框架                                           │
│    ↓ 调用平台 API                                   │
│  ┌──────────────┐  ┌──────────────┐                │
│  │ Windows:     │  │ macOS:       │                │
│  │ CreateWindow  │  │ NSButton     │                │
│  │ + GDI 绘制   │  │ + Core       │                │
│  │              │  │   Graphics   │                │
│  └──────────────┘  └──────────────┘                │
│    ↓                  ↓                             │
│  Windows 风格按钮    macOS 风格按钮                  │
│  （两个平台上长得不一样）                             │
└────────────────────────────────────────────────────┘
```

**代表框架：** Qt Widgets（部分）、Electron（通过 Chromium 渲染网页控件）

**优点：**
- 控件外观和行为与操作系统一致，用户感觉"原生"
- 自动获得 OS 更新带来的新外观（如 Windows 11 圆角按钮）
- 无障碍支持自动继承自 OS

**缺点：**
- 各平台控件差异大，难以实现跨平台一致的 UI
- 受限于 OS 控件能力——OS 没有的控件你也没有
- 调试困难——UI 行为由 OS 控制，框架无法完全管理

### 策略 2：自绘渲染

**原理：** 框架完全忽略操作系统的原生控件，通过底层图形 API（OpenGL、Vulkan、Metal、Skia 等）自己绘制每一个像素。一个"按钮"不是 OS 按钮，而是框架画出来的一个矩形+文字+点击区域。

```
┌────────────────────────────────────────────────────┐
│              自绘渲染流程                             │
│                                                    │
│  应用代码                                           │
│    ↓ "创建一个按钮"                                  │
│  UI 框架                                           │
│    ↓ 计算布局、生成绘制命令                          │
│  渲染引擎（Skia / Impeller / Scene Graph）          │
│    ↓ 翻译为图形 API 调用                            │
│  ┌──────────────────────────────────────────┐      │
│  │ OpenGL / Vulkan / Metal / DirectX        │      │
│  └──────────────────────────────────────────┘      │
│    ↓                                               │
│  所有平台上看起来完全一样的按钮                      │
└────────────────────────────────────────────────────┘
```

**代表框架：** Flutter、Compose Multiplatform、Dear ImGui、Qt Quick

**优点：**
- 跨平台 UI 完全一致——像素级相同
- 框架有完全的控制权，可以实现任何 UI 效果（自定义动画、非标准控件等）
- 不受 OS 控件限制

**缺点：**
- 不会自动跟随 OS 外观变化（如暗色模式需要自己实现）
- 需要自己实现无障碍支持（而不是继承自 OS）
- 渲染引擎本身有一定的性能开销和体积

### 四种框架的渲染策略对比

| 框架 | 渲染策略 | 渲染引擎 | 底层图形 API |
|------|---------|---------|-------------|
| Flutter | 自绘 | Impeller（新）/ Skia（旧） | Vulkan / Metal / OpenGL ES / DirectX |
| Compose MP | 自绘 | Skia | OpenGL / Metal / Direct3D |
| Qt Widgets | 原生 + 自绘混合 | 平台原生 API + QPainter | GDI / Core Graphics / X11 |
| Qt Quick | 自绘 | Scene Graph | OpenGL / Vulkan / Metal / Direct3D |
| Dear ImGui | 自绘 | 无内置引擎（依赖宿主） | 宿主提供（OpenGL/Vulkan/DX/Metal） |

**对 KrKr2 的影响：** KrKr2 使用 Cocos2d-x + OpenGL 进行游戏渲染。如果选择自绘渲染的 UI 框架，两个渲染系统需要共享 GL 上下文（OpenGL Context，GPU 的渲染环境，包含纹理、着色器等所有资源）或通过纹理共享传递画面。

## 四种框架的渲染管线详解

### Flutter 渲染管线

Flutter 的渲染流程分为三个阶段，每一帧都经过完整的管线：

```
┌─────────────────────────────────────────────────────┐
│                Flutter 渲染管线                       │
│                                                     │
│  1. Widget 树构建（Dart 侧）                         │
│     ├─ StatelessWidget.build() 返回 Widget 描述      │
│     └─ StatefulWidget.build() 根据 State 返回描述    │
│                    ↓                                 │
│  2. Element 树 + RenderObject 树（框架内部）          │
│     ├─ Element 树：Widget 的实例化版本，管理生命周期   │
│     ├─ RenderObject 树：负责布局（Layout）和绘制      │
│     └─ 布局算法：单遍（One-pass）从上往下确定尺寸     │
│                    ↓                                 │
│  3. Skia/Impeller 渲染（C++ 侧）                     │
│     ├─ RenderObject 生成 Layer 树                    │
│     ├─ Layer 树序列化为 Scene                         │
│     └─ Skia/Impeller 将 Scene 翻译为 GPU 绘制命令    │
│                    ↓                                 │
│  屏幕像素                                            │
└─────────────────────────────────────────────────────┘
```

**关键点：** Flutter 引擎本身是 C++ 编写的。Dart 代码只负责 UI 描述（步骤 1），实际渲染（步骤 3）由 C++ 的 Skia/Impeller 执行。这意味着：
- 渲染性能由 C++ 引擎决定，Dart 代码的性能影响仅限于 UI 描述阶段
- Dart 和 C++ 之间的通信是 Flutter 架构的核心瓶颈点

**Flutter 渲染管线中的性能关键代码：**

```dart
// 示例：理解 Flutter 的三棵树
import 'package:flutter/material.dart';

// Widget 树 — 描述 UI 的不可变配置（轻量对象，频繁重建）
class GameSettingsWidget extends StatefulWidget {
  const GameSettingsWidget({super.key});

  @override
  State<GameSettingsWidget> createState() => _GameSettingsState();
}

class _GameSettingsState extends State<GameSettingsWidget> {
  double _volume = 0.5; // 状态：音量

  @override
  Widget build(BuildContext context) {
    // 每次 setState 时，这个 build 方法都会重新执行
    // 但 Flutter 不会重建整个界面——Element 树会 diff，只更新变化的部分
    return Column(
      children: [
        // Text Widget 只是一个描述："显示文本，内容是 '音量: 50%'"
        // Flutter 内部的 RenderParagraph 负责实际的文本布局和绘制
        Text('音量: ${(_volume * 100).toInt()}%'),

        // Slider Widget 描述一个滑块
        // 内部对应多个 RenderObject：轨道、滑块、活动区域等
        Slider(
          value: _volume,
          onChanged: (v) => setState(() => _volume = v),
        ),
      ],
    );
  }
}
```

### Compose Multiplatform 渲染管线

Compose 的渲染管线与 Flutter 非常相似，也采用声明式描述 + 自动 diff + GPU 渲染：

```
┌─────────────────────────────────────────────────────┐
│           Compose Multiplatform 渲染管线              │
│                                                     │
│  1. Composable 函数执行（Kotlin 侧）                 │
│     ├─ @Composable 函数描述 UI 结构                  │
│     └─ Compose 编译器插件注入 slot table 管理代码     │
│                    ↓                                 │
│  2. Composition + LayoutNode 树（框架内部）           │
│     ├─ Composition：追踪状态依赖，仅重组变化部分      │
│     ├─ LayoutNode 树：负责测量（Measure）和放置       │
│     └─ 绘制阶段：生成 Canvas 绘制命令                │
│                    ↓                                 │
│  3. Skia 渲染（桌面端/iOS，通过 Skiko 绑定）          │
│     ├─ Canvas 命令翻译为 Skia 操作                   │
│     └─ Skia 输出到 OpenGL/Metal/Direct3D            │
│                    ↓                                 │
│  屏幕像素                                            │
└─────────────────────────────────────────────────────┘
```

**关键区别于 Flutter：**
- Android 端直接使用 Android 原生 Compose 渲染（不经过 Skia，性能最佳）
- 桌面端和 iOS 使用 Skiko（Skia for Kotlin，JetBrains 维护的 Skia Kotlin 绑定）
- Compose 编译器插件在编译期自动注入状态追踪代码，不需要手动调用 `setState`

```kotlin
// 示例：理解 Compose 的重组（Recomposition）机制
import androidx.compose.foundation.layout.*
import androidx.compose.material3.*
import androidx.compose.runtime.*

@Composable
fun GameSettings() {
    // mutableStateOf 创建可观察状态
    // Compose 编译器自动追踪：哪些 Composable 读取了这个状态
    var volume by remember { mutableStateOf(0.5f) }

    Column {
        // 只有读取了 volume 的部分会在 volume 变化时重组
        Text("音量: ${(volume * 100).toInt()}%")

        Slider(
            value = volume,
            onValueChange = { volume = it }
            // 当用户拖动滑块时：
            // 1. volume 变化
            // 2. Compose 检测到 Text 和 Slider 读取了 volume
            // 3. 只重组这两个 Composable，不影响其他 UI
        )
    }
}
```

### Dear ImGui 渲染管线

Dear ImGui 的渲染管线是四种框架中最简单直接的——因为它没有自己的渲染引擎，而是生成绘制命令让宿主程序的渲染后端执行：

```
┌─────────────────────────────────────────────────────┐
│            Dear ImGui 渲染管线                        │
│                                                     │
│  1. 即时模式 UI 描述（每帧执行）                      │
│     ├─ ImGui::Begin("窗口")                          │
│     ├─ ImGui::Button("按钮") → 返回是否被点击        │
│     └─ ImGui::End()                                  │
│                    ↓                                 │
│  2. 内部缓冲区填充                                    │
│     ├─ ImGui 将 UI 元素转换为三角形顶点               │
│     ├─ 文本通过内置字体光栅化器生成纹理               │
│     └─ 生成 ImDrawList（绘制命令列表）                │
│                    ↓                                 │
│  3. 宿主程序的渲染后端执行绘制                        │
│     ├─ 读取 ImDrawData（顶点缓冲 + 索引缓冲 + 命令） │
│     ├─ 上传顶点数据到 GPU                            │
│     └─ 按命令逐个绘制（每个命令 = 一次 draw call）    │
│                    ↓                                 │
│  屏幕像素（叠加在宿主程序画面之上）                    │
└─────────────────────────────────────────────────────┘
```

**关键点：** ImGui 不拥有 GL 上下文，不管理窗口，不创建渲染表面。它只生成绘制数据（顶点 + 命令），由宿主程序选择何时、何地执行这些绘制。这使得 ImGui 可以完美插入到任何已有的渲染循环中——包括 Cocos2d-x。

```cpp
// 示例：ImGui 在 Cocos2d-x 渲染循环中的嵌入点
// （概念伪代码，展示集成思路）

// Cocos2d-x 的每帧渲染回调
void AppDelegate::mainLoop() {
    // 1. Cocos2d-x 渲染游戏画面
    director->drawScene();   // 绘制所有 Cocos2d 节点

    // 2. 在 Cocos2d 渲染之后、SwapBuffers 之前，插入 ImGui 渲染
    ImGui_ImplOpenGL3_NewFrame();     // 准备新帧
    ImGui_ImplSDL2_NewFrame();
    ImGui::NewFrame();

    // 3. 定义 UI（即时模式——每帧都执行）
    if (show_debug_panel) {
        ImGui::Begin("调试面板");
        ImGui::Text("FPS: %.1f", ImGui::GetIO().Framerate);
        ImGui::Text("Draw Calls: %d", director->getDrawCallCount());
        ImGui::End();
    }

    // 4. 渲染 ImGui（叠加在 Cocos2d 画面上）
    ImGui::Render();
    ImGui_ImplOpenGL3_RenderDrawData(ImGui::GetDrawData());

    // 5. 最后 SwapBuffers 交换前后缓冲区
    SDL_GL_SwapWindow(window);
}
```

### Qt 的混合渲染管线

Qt 比较特殊——它有两套独立的 UI 系统，渲染方式完全不同：

| 子系统 | 渲染方式 | 适用场景 |
|--------|---------|---------|
| Qt Widgets | QPainter → 平台原生 API | 传统桌面应用 |
| Qt Quick (QML) | Scene Graph → OpenGL/Vulkan | 现代动画丰富的应用 |

```cpp

    QGuiApplication app(argc, argv);

```

## C++ 互操作能力深度对比

对于 KrKr2 项目，C++ 互操作能力是最关键的选型因素。UI 框架必须能够：
1. 被 C++ 代码启动和管理
2. 从 C++ 发送数据到 UI（如游戏状态、配置信息）
3. 从 UI 接收用户操作回传到 C++（如用户修改设置、点击按钮）
4. 与 C++ 渲染引擎共享纹理（显示游戏画面）

### Flutter ↔ C++ 互操作

Flutter 提供两条与 C++ 通信的路径：

**路径 1：Dart FFI（Foreign Function Interface）— 直接函数调用**

Dart FFI 允许 Dart 代码直接调用 C 函数（不经过任何中间层），性能接近原生调用。适合高频数据传输。

```dart
// Dart 侧：通过 FFI 调用 C++ 编译的共享库
import 'dart:ffi';          // Dart FFI 库
import 'dart:io' show Platform;

// 定义 C 函数的类型签名
// C 函数原型：int get_volume(void);
typedef GetVolumeC = Int32 Function();     // C 侧类型
typedef GetVolumeDart = int Function();    // Dart 侧类型

// C 函数原型：void set_volume(int volume);
typedef SetVolumeC = Void Function(Int32);
typedef SetVolumeDart = void Function(int);

void main() {
    // 加载 KrKr2 的共享库
    final String libPath;
    if (Platform.isWindows) {
        libPath = 'krkr2.dll';         // Windows 动态链接库
    } else if (Platform.isLinux) {
        libPath = 'libkrkr2.so';       // Linux 共享对象
    } else if (Platform.isMacOS) {
        libPath = 'libkrkr2.dylib';    // macOS 动态库
    } else {
        throw UnsupportedError('不支持的平台');
    }

    final dylib = DynamicLibrary.open(libPath);

    // 查找并绑定 C 函数
    final getVolume = dylib
        .lookupFunction<GetVolumeC, GetVolumeDart>('get_volume');
    final setVolume = dylib
        .lookupFunction<SetVolumeC, SetVolumeDart>('set_volume');

    // 直接调用 — 就像调用 Dart 函数一样
    print('当前音量: ${getVolume()}');   // 调用 C++ 的 get_volume()
    setVolume(80);                      // 调用 C++ 的 set_volume(80)
    print('新音量: ${getVolume()}');
}
```

对应的 C++ 侧代码：

```cpp
// krkr2_ffi.cpp — 暴露给 Dart FFI 的 C 接口
#include <cstdint>

static int32_t g_volume = 50;  // 模拟器音量

// extern "C" 确保函数名不被 C++ 名称修饰（mangling）
// 这样 Dart FFI 才能通过函数名找到它
extern "C" {

// 导出符号（Windows 需要 __declspec(dllexport)）
#ifdef _WIN32
#define EXPORT __declspec(dllexport)
#else
#define EXPORT __attribute__((visibility("default")))
#endif

EXPORT int32_t get_volume() {
    return g_volume;
}

EXPORT void set_volume(int32_t volume) {
    g_volume = (volume < 0) ? 0 : (volume > 100) ? 100 : volume;
}

}  // extern "C"
```

**路径 2：Platform Channel — 异步消息传递**

Platform Channel 是 Flutter 的官方跨语言通信机制，通过异步消息在 Dart 和平台代码（C++/Java/Swift）之间传递结构化数据。适合不频繁的命令式交互（如"打开文件"、"保存设置"）。

```dart
// Dart 侧：通过 MethodChannel 调用平台方法
import 'package:flutter/services.dart';

class KrKr2Bridge {
  // 创建命名通道 — 名称必须与 C++ 侧一致
  static const channel = MethodChannel('com.krkr2/settings');

  // 异步调用平台方法
  static Future<int> getVolume() async {
    final volume = await channel.invokeMethod<int>('getVolume');
    return volume ?? 50;  // 默认值
  }

  static Future<void> setVolume(int volume) async {
    await channel.invokeMethod('setVolume', {'volume': volume});
  }

  // 监听来自 C++ 的事件（如游戏状态变化）
  static void listenForEvents() {
    const eventChannel = EventChannel('com.krkr2/events');
    eventChannel.receiveBroadcastStream().listen((event) {
      print('收到 C++ 事件: $event');
    });
  }
}
```

### Dear ImGui ↔ C++ 互操作

Dear ImGui 本身就是 C++ 库——没有"互操作"的概念，因为根本没有语言边界：

```cpp
// Dear ImGui 与 KrKr2 C++ 代码的交互：直接调用
#include "imgui.h"
#include "ConfigManager.h"   // KrKr2 的配置管理器

void drawSettingsUI(ConfigManager &config) {
    ImGui::Begin("KrKr2 设置");

    // 直接读写 C++ 对象 — 零开销
    int volume = config.getVolume();
    if (ImGui::SliderInt("音量", &volume, 0, 100)) {
        config.setVolume(volume);  // 直接调用 C++ 方法
    }

    // 直接访问 C++ 容器
    auto &recentGames = config.getRecentGames();  // std::vector<string>
    for (const auto &game : recentGames) {
        if (ImGui::Selectable(game.c_str())) {
            config.loadGame(game);  // 直接调用
        }
    }

    ImGui::End();
}
```

**这是 ImGui 最大的优势：零互操作开销。** 不需要序列化、不需要异步通道、不需要跨语言桥接。C++ 代码直接操作 ImGui API，ImGui 直接读写 C++ 数据结构。

### Compose MP ↔ C++ 互操作

Compose Multiplatform 的 C++ 互操作最为复杂，需要经过两层桥接：

```
C++ 函数 → C 接口（extern "C"） → cinterop .def 文件 → Kotlin/Native 绑定 → Kotlin 代码
```

```kotlin
// 步骤 1：编写 .def 文件定义 C 函数签名
// krkr2.def
// headers = krkr2_ffi.h
// libraryPaths = /path/to/libs
// linkerOpts = -lkrkr2
```

```kotlin
// 步骤 2：Kotlin 调用生成的绑定
import krkr2.*  // cinterop 自动生成的包

fun getVolumeFromEngine(): Int {
    return get_volume()  // 调用 C 函数（通过 cinterop 绑定）
}

fun setVolumeInEngine(volume: Int) {
    set_volume(volume)   // 调用 C 函数
}
```

### 互操作复杂度对比表

| 维度 | Flutter | Dear ImGui | Qt | Compose MP |
|------|---------|-----------|-----|------------|
| 语言边界 | Dart ↔ C | 无 | 无（或 QML ↔ C++） | Kotlin ↔ C |
| 调用方式 | FFI（同步）/ Channel（异步） | 直接调用 | 直接调用 / 信号槽 | cinterop（同步） |
| 数据传输开销 | FFI: 低 / Channel: 中 | 零 | 零 / QML: 低 | 低 |
| 回调支持 | FFI 回调 / Channel 事件流 | 直接函数指针 | 信号槽 / QML 绑定 | C 函数指针回调 |
| 内存管理 | FFI 需手动管理指针 | C++ RAII | C++ RAII / Qt 父子所有权 | Kotlin GC + 手动释放 |
| 调试难度 | 高（跨 Dart/C++ 断点） | 低（纯 C++） | 低 / 中（QML 调试器） | 高（Kotlin/Native 调试器不成熟） |

### 为什么不选 Qt？

虽然 Qt 在评分矩阵中得分很高，但有两个"一票否决"因素：

1. **许可证风险：** KrKr2 可能有二次分发需求，GPLv3 传染性是硬伤。LGPL 动态链接的合规要求也增加了分发复杂度
2. **体积问题：** Qt 运行时 50-100MB 对桌面应用可接受，但对移动端不友好。KrKr2 的 Android 版需要控制包体大小

### 为什么不选 Compose MP？

1. **C++ 互操作最复杂：** 需要 Kotlin/Native + cinterop 双层桥接
2. **桌面端/iOS 不够成熟：** 虽然在快速进步，但目前生产案例远少于 Flutter
3. **团队技能匹配度低：** KrKr2 是 C++ 项目，引入 Kotlin 工具链增加了构建和维护复杂度

## 动手实践

### 练习 1：测量 Dart FFI 的调用开销

编写一个简单的 Dart 程序，通过 FFI 调用一个空的 C 函数 100 万次，测量总耗时：

```cpp
// ffi_bench.cpp — 被测的空函数
extern "C" {
#ifdef _WIN32
#define EXPORT __declspec(dllexport)
#else
#define EXPORT __attribute__((visibility("default")))
#endif

EXPORT int noop(int x) {
    return x + 1;  // 最简操作，消除优化
}
}
```

```cmake
# CMakeLists.txt — 编译为共享库
cmake_minimum_required(VERSION 3.16)
project(ffi_bench LANGUAGES CXX)
add_library(ffi_bench SHARED ffi_bench.cpp)
```

```dart
// ffi_bench.dart — Dart 侧测量
import 'dart:ffi';
import 'dart:io' show Platform;

typedef NoopC = Int32 Function(Int32);
typedef NoopDart = int Function(int);

void main() {
    final lib = DynamicLibrary.open(
        Platform.isWindows ? 'ffi_bench.dll'
      : Platform.isMacOS   ? 'libffi_bench.dylib'
      :                       'libffi_bench.so'
    );

    final noop = lib.lookupFunction<NoopC, NoopDart>('noop');

    const iterations = 1000000;
    final sw = Stopwatch()..start();

    int result = 0;
    for (int i = 0; i < iterations; i++) {
        result = noop(result);  // FFI 调用
    }

    sw.stop();
    print('$iterations 次 FFI 调用耗时: ${sw.elapsedMilliseconds} ms');
    print('平均每次: ${sw.elapsedMicroseconds / iterations} µs');
    print('结果: $result');  // 防止编译器优化掉循环
}
```

预期结果：在现代硬件上，100 万次 FFI 调用约 10-30 毫秒（每次约 10-30 纳秒）。这证明 FFI 开销非常低，适合高频数据传输。

## 对照项目源码

KrKr2 当前 UI 代码与新框架的对应关系：

| 当前文件 | 功能 | 替换方案 |
|----------|------|---------|
| `cpp/core/environ/ui/` | Cocos2d UI 表单 | Flutter Widget / ImGui 面板 |
| `ui/cocos-studio/*.csb` | 二进制 UI 布局 | Flutter Dart 代码 / ImGui C++ 代码 |
| `cpp/core/environ/ConfigManager/` | 配置管理 | 保留，通过 FFI/直接调用访问 |
| `cpp/core/environ/cocos2d/AppDelegate.cpp` | 渲染循环入口 | ImGui 在此嵌入；Flutter 通过 Engine API 嵌入 |

**关键观察：** 当前 `ui/cocos-studio/` 下的 `.csb` 文件存储了所有 UI 布局信息。这些文件是 Cocos Studio 的专有二进制格式，意味着：
- 无法用文本编辑器修改
- 无法有效进行 Git diff
- Cocos Studio 已停止维护，无法编辑这些文件

换用 Flutter 或 ImGui 后，UI 布局变成纯代码（Dart 或 C++），完全可读、可编辑、可版本控制。

## 常见错误与排查

### 错误 1：Dart FFI 调用 C++ 函数时崩溃（SIGSEGV）

**现象：** 调用 `DynamicLibrary.open()` 或 `lookupFunction()` 后程序崩溃。

**常见原因和解决方案：**

1. **函数名不匹配：** C++ 会对函数名做名称修饰（Name Mangling）。必须用 `extern "C"` 包裹导出函数
2. **参数类型不匹配：** Dart 的 `Int32` 对应 C 的 `int32_t`，不是 `int`（`int` 在不同平台可能是 16/32/64 位）
3. **共享库路径错误：** 确保 `.dll`/`.so`/`.dylib` 文件在程序的搜索路径中

```dart
// 错误：类型不匹配
typedef WrongC = Int64 Function(Int32);  // C 函数返回 int32，这里写成 Int64

// 正确：类型必须精确匹配
typedef CorrectC = Int32 Function(Int32);
```

### 错误 2：ImGui 文本显示乱码（中文显示为方块）

**现象：** ImGui 窗口中的中文字符显示为 `□□□` 或空白。

**原因：** ImGui 默认只加载 ASCII 字体范围（0x0020-0x00FF），不包含中文字符。

**解决方案：** 加载支持中文的字体并设置中文 Unicode 范围：

```cpp
// 初始化 ImGui 后，加载中文字体
ImGuiIO &io = ImGui::GetIO();

// 设置中文字体（需要提供 .ttf 字体文件）
ImFontConfig font_config;
font_config.MergeMode = false;

// 方法 1：加载系统字体
#ifdef _WIN32
io.Fonts->AddFontFromFileTTF(
    "C:\\Windows\\Fonts\\msyh.ttc",   // 微软雅黑
    16.0f, &font_config,
    io.Fonts->GetGlyphRangesChineseFull()  // 完整中文字符范围
);
#elif __APPLE__
io.Fonts->AddFontFromFileTTF(
    "/System/Library/Fonts/PingFang.ttc",
    16.0f, &font_config,
    io.Fonts->GetGlyphRangesChineseFull()
);
#else  // Linux
io.Fonts->AddFontFromFileTTF(
    "/usr/share/fonts/truetype/droid/DroidSansFallbackFull.ttf",
    16.0f, &font_config,
    io.Fonts->GetGlyphRangesChineseFull()
);
#endif
```

### 错误 3：Flutter Engine 嵌入后 GL 上下文冲突

**现象：** 同时运行 Cocos2d-x 和 Flutter 时，画面闪烁或一个引擎的画面消失。

**原因：** 两个引擎抢占同一个 GL 上下文。OpenGL 是单线程状态机，一个时刻只有一个上下文可以"激活"（通过 `eglMakeCurrent` / `wglMakeCurrent`）。

**解决方案（下一节详细讲解）：**
1. 使用独立 GL 上下文 + 共享纹理（推荐）
2. 在同一 GL 上下文中交替渲染（需要保存/恢复 GL 状态）
3. Flutter 渲染到离屏纹理，Cocos2d-x 将纹理贴到 Sprite 上显示

## 本节小结

- **自绘渲染**（Flutter、Compose、ImGui）控制所有像素，跨平台 UI 完全一致；**原生控件渲染**（Qt Widgets）使用 OS 控件，外观与系统风格一致
- Flutter 的渲染管线分三棵树（Widget → Element → RenderObject），实际绘制由 C++ 的 Skia/Impeller 引擎完成
- Dear ImGui 只生成顶点和绘制命令，由宿主程序的渲染后端执行——因此可以完美嵌入任何渲染循环
- C++ 互操作方面：ImGui 和 Qt 是原生 C++（零开销），Flutter 通过 FFI 直接调用（低开销），Compose 需要 cinterop 桥接（较复杂）
- **KrKr2 推荐分层策略：** ImGui 用于开发工具（立即可做），Flutter 用于面向用户的 UI（中期目标），底层设计抽象接口隔离具体框架
- 技术选型不应只看"分数最高"，还要考虑许可证、体积、团队技能等"一票否决"因素

## 练习题与答案

### 题目 1：解释 Flutter 渲染管线中"三棵树"各自的职责，并说明为什么需要三棵树而不是一棵

<details>
<summary>查看答案</summary>

**三棵树：**

1. **Widget 树：** 不可变的 UI 配置描述。每次 `setState()` 或状态变化时，Flutter 重新调用 `build()` 方法生成新的 Widget 树。Widget 是轻量级对象，创建成本极低

2. **Element 树：** Widget 的实例化版本，管理 Widget 的生命周期。Element 在两次 build 之间持续存在，负责比较新旧 Widget 是否有变化。如果新 Widget 的类型和 key 相同，Element 会复用而不是重建

3. **RenderObject 树：** 负责实际的布局计算（确定每个元素的大小和位置）和绘制命令生成。RenderObject 是真正的"重量级"对象，持有 GPU 资源引用

**为什么需要三棵树？**

- **性能优化：** Widget 树频繁重建（因为是不可变对象），但 Element 树通过 diff 算法判断哪些 RenderObject 需要更新。如果只有一棵树，每次状态变化都要重建布局和 GPU 资源——开销巨大
- **关注点分离：** Widget 关注"描述"（是什么），Element 关注"生命周期"（何时创建/更新/销毁），RenderObject 关注"实现"（怎么画）。三者职责清晰，代码可维护性高
- **类比：** Widget = HTML，Element = Virtual DOM，RenderObject = Real DOM。与 React 的 Virtual DOM diff 算法原理类似

</details>

### 题目 2：在 KrKr2 中集成 Dear ImGui 时，应该在 Cocos2d-x 渲染循环的哪个位置插入 ImGui 渲染调用？为什么？

<details>
<summary>查看答案</summary>

**应该在 Cocos2d-x 的 `drawScene()` 之后、`SwapBuffers` 之前插入。**

具体位置和原因：

```cpp
void AppDelegate::mainLoop() {
    // ① Cocos2d-x 渲染所有游戏场景
    director->drawScene();

    // ② ★ 在这里插入 ImGui 渲染 ★
    ImGui_ImplOpenGL3_NewFrame();
    ImGui_ImplSDL2_NewFrame();
    ImGui::NewFrame();
    // ... ImGui UI 定义 ...
    ImGui::Render();
    ImGui_ImplOpenGL3_RenderDrawData(ImGui::GetDrawData());

    // ③ 交换前后缓冲区，显示最终画面
    SDL_GL_SwapWindow(window);
}
```

**为什么在 drawScene() 之后：**
- ImGui 的绘制会叠加（overlay）在已有画面之上。如果在 drawScene() 之前渲染 ImGui，游戏画面会覆盖掉 ImGui 的 UI
- ImGui 使用 alpha 混合（Alpha Blending），其窗口背景是半透明的，需要底层有游戏画面作为"底色"

**为什么在 SwapBuffers 之前：**
- `SwapBuffers` 将后缓冲区的内容显示到屏幕上。所有绘制操作必须在这之前完成
- 在 `SwapBuffers` 之后绘制的内容会出现在下一帧，导致 ImGui 界面延迟一帧显示

**注意事项：**
- ImGui 渲染会修改 OpenGL 状态（混合模式、裁剪矩形等）。如果之后还有其他渲染操作，需要恢复 GL 状态
- 在 Android 上，SDL2 的 `SDL_GL_SwapWindow` 对应 `eglSwapBuffers`，原理相同

</details>

### 题目 3：Dart FFI 和 Platform Channel 各适合什么场景？如果要实现"将 C++ 渲染的每一帧画面传给 Flutter 显示"，应该用哪种方式？

<details>
<summary>查看答案</summary>

**各自适合的场景：**

**Dart FFI：**
- 高频数据传输（每帧调用、大量数据）
- 同步调用（立即返回结果）
- 数值类型为主的数据（int、float、指针）
- 性能敏感场景

**Platform Channel：**
- 低频命令式交互（打开文件、保存设置、获取系统信息）
- 异步调用（不阻塞 UI 线程）
- 结构化数据（Map、List、String）
- 需要双向通信（Dart ↔ C++ 都可以主动发起）

**传递每帧画面应该用：Texture Widget + GPU 纹理共享（都不是！）**

既不用 FFI 也不用 Platform Channel——直接通过 Flutter 的 Texture Widget 机制共享 GPU 纹理：

1. C++ 侧将游戏画面渲染到一个 OpenGL 纹理（texture ID）
2. 通过 Flutter Embedder API 注册这个纹理为外部纹理
3. Dart 侧使用 `Texture(textureId: id)` Widget 显示这个纹理

这避免了任何 CPU 侧的数据拷贝——画面始终在 GPU 内存中流转。FFI 只用于传递纹理 ID（一个整数），不传递像素数据。Platform Channel 的异步特性也不适合每帧传输。

下一章第 02 节将详细讲解这个 Texture Widget 机制。

</details>
