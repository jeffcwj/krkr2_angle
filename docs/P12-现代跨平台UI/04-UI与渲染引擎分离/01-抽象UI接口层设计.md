## 1. 为什么需要 UI 与渲染引擎分离？

### 1.1 问题背景

KrKr2 项目当前使用 Cocos2d-x 同时承担两个角色：

1. **游戏渲染引擎**：绘制背景、立绘、特效、文字等游戏画面
2. **UI 框架**：实现菜单、对话框、设置面板等用户界面

这种"一个框架做两件事"的架构存在以下问题：

```
当前架构（紧耦合）：

┌──────────────────────────────┐
│        KrKr2 应用             │
├──────────────────────────────┤
│  游戏逻辑 (TJS2)             │
├──────────────────────────────┤
│  ┌──────────┐  ┌──────────┐ │
│  │ 游戏渲染 │  │ UI 渲染  │ │  ← 两者都直接依赖 Cocos2d-x
│  └────┬─────┘  └────┬─────┘ │
│       │              │       │
│  ┌────┴──────────────┴────┐  │
│  │    Cocos2d-x 框架      │  │  ← 单一框架，不可替换
│  └────────────────────────┘  │
│            OpenGL             │
└──────────────────────────────┘
```

**问题 1：UI 升级受限**。Cocos2d-x 是游戏引擎而非 UI 框架，缺少现代 UI 能力（动画过渡、响应式布局、无障碍访问、国际化等）。但由于 UI 直接写在 Cocos2d-x 上，不可能单独替换 UI 部分。

**问题 2：平台不一致**。Cocos2d-x 的 UI 在不同平台表现一致（因为全部自绘），但这也意味着无法使用原生 UI 特性（如 iOS 的滑动手势、Android 的 Material Design 组件）。

**问题 3：维护成本**。Cocos2d-x 社区活跃度下降，Bug 修复和新平台适配只能靠项目自身。如果 UI 层可以切换到活跃维护的框架（Flutter/Compose），维护成本大幅降低。

### 1.2 目标架构

分离后的架构使用抽象接口层隔离 UI 和渲染引擎：

```
目标架构（松耦合）：

┌──────────────────────────────────────────┐
│              KrKr2 应用                    │
├──────────────────────────────────────────┤
│  游戏逻辑 (TJS2)                          │
├──────────────────────────────────────────┤
│  ┌──────────┐  ┌───────────────────────┐ │
│  │ 游戏渲染 │  │   UI 抽象接口层       │ │
│  │ (Cocos)  │  │   (IUIBackend)       │ │
│  └────┬─────┘  └───────────┬───────────┘ │
│       │                    │              │
│       │         ┌──────────┼──────────┐  │
│       │         │          │          │  │
│  ┌────┴────┐  ┌─┴──────┐ ┌┴───────┐ ┌┴────────┐
│  │Cocos2d-x│  │Cocos2d │ │Flutter │ │Compose  │
│  │ 渲染    │  │ UI实现 │ │UI实现  │ │UI实现   │
│  └─────────┘  └────────┘ └────────┘ └─────────┘
│     OpenGL      自绘       Skia       Skiko
└──────────────────────────────────────────┘
```

**核心思想**：游戏引擎的核心逻辑只依赖抽象接口（`IUIBackend`），不知道也不关心具体使用哪个 UI 框架。通过在运行时注入不同的实现（策略模式），可以自由切换 UI 后端。

## 2. 设计模式基础

### 2.1 策略模式（Strategy Pattern）

策略模式是实现 UI 可插拔的核心设计模式。它的思想很简单：**把可变的部分（UI 实现）封装到独立的类中，通过统一的接口调用**。

```cpp
// 示例 1：策略模式基础 — UI 后端切换
// 文件：src/ui/strategy_demo.cpp

#include <memory>
#include <string>
#include <cstdio>

// ---- 抽象策略接口 ----
// 定义"显示消息框"的统一行为
class IMessageBoxStrategy {
public:
    virtual ~IMessageBoxStrategy() = default;

    // 显示消息框，返回用户选择的按钮索引
    virtual int show(
        const std::string& title,     // 标题
        const std::string& message,   // 消息内容
        const std::string& buttons    // 按钮列表（用 | 分隔）
    ) = 0;

    // 获取策略名称（用于日志）
    virtual const char* getName() const = 0;
};

// ---- 具体策略 A：Cocos2d 实现 ----
class Cocos2dMessageBox : public IMessageBoxStrategy {
public:
    int show(const std::string& title, const std::string& message,
             const std::string& buttons) override {
        printf("[Cocos2d] 显示消息框: %s - %s\n", title.c_str(), message.c_str());
        // 实际实现会调用 Cocos2d-x 的 UI 层
        // cocos2d::ui::Text / cocos2d::ui::Button 等
        return 0;  // 返回第一个按钮
    }

    const char* getName() const override { return "Cocos2d"; }
};

// ---- 具体策略 B：Flutter 实现 ----
class FlutterMessageBox : public IMessageBoxStrategy {
public:
    int show(const std::string& title, const std::string& message,
             const std::string& buttons) override {
        printf("[Flutter] 通过 Platform Channel 显示对话框: %s\n", title.c_str());
        // 实际实现会通过 FlutterEngine 的 MethodChannel 发送消息
        // Flutter 侧用 showDialog() 显示 AlertDialog
        return 0;
    }

    const char* getName() const override { return "Flutter"; }
};

// ---- 具体策略 C：Compose 实现 ----
class ComposeMessageBox : public IMessageBoxStrategy {
public:
    int show(const std::string& title, const std::string& message,
             const std::string& buttons) override {
        printf("[Compose] 通过 JNI/cinterop 显示对话框: %s\n", title.c_str());
        // 实际实现会通过 cinterop 调用 Kotlin Compose 的 AlertDialog
        return 0;
    }

    const char* getName() const override { return "Compose"; }
};

// ---- 上下文（使用策略的客户端） ----
class GameUI {
    std::unique_ptr<IMessageBoxStrategy> messageBoxStrategy_;

public:
    // 注入策略（构造时或运行时）
    void setMessageBoxStrategy(std::unique_ptr<IMessageBoxStrategy> strategy) {
        messageBoxStrategy_ = std::move(strategy);
        printf("UI 后端切换为: %s\n", messageBoxStrategy_->getName());
    }

    // 使用策略 — 不关心具体实现
    int showMessage(const std::string& title, const std::string& message) {
        if (!messageBoxStrategy_) {
            printf("错误：未设置消息框策略\n");
            return -1;
        }
        return messageBoxStrategy_->show(title, message, "确定|取消");
    }
};

int main() {
    GameUI ui;

    // 场景 1：使用 Cocos2d 后端
    ui.setMessageBoxStrategy(std::make_unique<Cocos2dMessageBox>());
    ui.showMessage("提示", "是否保存进度？");

    // 场景 2：运行时切换到 Flutter
    ui.setMessageBoxStrategy(std::make_unique<FlutterMessageBox>());
    ui.showMessage("提示", "是否保存进度？");

    // 场景 3：切换到 Compose
    ui.setMessageBoxStrategy(std::make_unique<ComposeMessageBox>());
    ui.showMessage("提示", "是否保存进度？");

    return 0;
}
```

**运行输出**：
```
UI 后端切换为: Cocos2d
[Cocos2d] 显示消息框: 提示 - 是否保存进度？
UI 后端切换为: Flutter
[Flutter] 通过 Platform Channel 显示对话框: 提示
UI 后端切换为: Compose
[Compose] 通过 JNI/cinterop 显示对话框: 提示
```

### 2.2 抽象工厂模式（Abstract Factory）

单个消息框的策略模式不够——实际项目需要一整套 UI 组件（消息框、菜单、设置面板、文件选择器等）。抽象工厂模式解决"创建一族相关对象"的问题：

```cpp
// 示例 2：抽象工厂 — 创建一整套 UI 组件
// 文件：src/ui/abstract_factory_demo.cpp

#include <memory>
#include <string>
#include <vector>
#include <cstdio>

// ---- 抽象产品接口 ----

// 消息框
class IMessageBox {
public:
    virtual ~IMessageBox() = default;
    virtual int show(const std::string& title, const std::string& msg) = 0;
};

// 菜单
class IMenu {
public:
    virtual ~IMenu() = default;
    virtual void addItem(const std::string& label, int id) = 0;
    virtual int show() = 0;  // 返回选中项 ID
};

// 文件选择器
class IFileDialog {
public:
    virtual ~IFileDialog() = default;
    virtual std::string openFile(const std::string& filter) = 0;
    virtual std::string saveFile(const std::string& defaultName) = 0;
};

// 进度条
class IProgressBar {
public:
    virtual ~IProgressBar() = default;
    virtual void show(const std::string& title) = 0;
    virtual void setProgress(float value) = 0;  // 0.0 ~ 1.0
    virtual void hide() = 0;
};

// ---- 抽象工厂 ----
class IUIFactory {
public:
    virtual ~IUIFactory() = default;

    virtual std::unique_ptr<IMessageBox> createMessageBox() = 0;
    virtual std::unique_ptr<IMenu> createMenu() = 0;
    virtual std::unique_ptr<IFileDialog> createFileDialog() = 0;
    virtual std::unique_ptr<IProgressBar> createProgressBar() = 0;

    virtual const char* getBackendName() const = 0;
};

// ---- Cocos2d 具体工厂 ----
class Cocos2dMessageBox : public IMessageBox {
public:
    int show(const std::string& title, const std::string& msg) override {
        printf("[Cocos2d] MessageBox: %s\n", title.c_str());
        return 0;
    }
};

class Cocos2dMenu : public IMenu {
    struct Item { std::string label; int id; };
    std::vector<Item> items_;
public:
    void addItem(const std::string& label, int id) override {
        items_.push_back({label, id});
    }
    int show() override {
        printf("[Cocos2d] Menu with %zu items\n", items_.size());
        return items_.empty() ? -1 : items_[0].id;
    }
};

class Cocos2dFileDialog : public IFileDialog {
public:
    std::string openFile(const std::string& filter) override {
        printf("[Cocos2d] 文件选择对话框 (filter: %s)\n", filter.c_str());
        return "/path/to/file.krkr";
    }
    std::string saveFile(const std::string& defaultName) override {
        return defaultName;
    }
};

class Cocos2dProgressBar : public IProgressBar {
public:
    void show(const std::string& title) override {
        printf("[Cocos2d] 进度条: %s\n", title.c_str());
    }
    void setProgress(float value) override {
        printf("[Cocos2d] 进度: %.0f%%\n", value * 100);
    }
    void hide() override {
        printf("[Cocos2d] 隐藏进度条\n");
    }
};

class Cocos2dUIFactory : public IUIFactory {
public:
    std::unique_ptr<IMessageBox> createMessageBox() override {
        return std::make_unique<Cocos2dMessageBox>();
    }
    std::unique_ptr<IMenu> createMenu() override {
        return std::make_unique<Cocos2dMenu>();
    }
    std::unique_ptr<IFileDialog> createFileDialog() override {
        return std::make_unique<Cocos2dFileDialog>();
    }
    std::unique_ptr<IProgressBar> createProgressBar() override {
        return std::make_unique<Cocos2dProgressBar>();
    }
    const char* getBackendName() const override { return "Cocos2d-x"; }
};

// ---- 使用工厂（客户端代码） ----
class GameApp {
    std::unique_ptr<IUIFactory> uiFactory_;

public:
    void setUIFactory(std::unique_ptr<IUIFactory> factory) {
        uiFactory_ = std::move(factory);
        printf("UI 后端: %s\n", uiFactory_->getBackendName());
    }

    void showSaveDialog() {
        auto msgBox = uiFactory_->createMessageBox();
        int result = msgBox->show("保存", "是否保存当前进度？");
        if (result == 0) {
            auto progress = uiFactory_->createProgressBar();
            progress->show("保存中...");
            progress->setProgress(0.5f);
            progress->setProgress(1.0f);
            progress->hide();
        }
    }
};

int main() {
    GameApp app;
    app.setUIFactory(std::make_unique<Cocos2dUIFactory>());
    app.showSaveDialog();
    return 0;
}
```

### 2.3 依赖倒置原则（DIP）

**依赖倒置原则**（Dependency Inversion Principle）是 UI 抽象层设计的理论基础：

> **高层模块不应依赖低层模块，两者都应依赖抽象。**
> **抽象不应依赖具体，具体应依赖抽象。**

在 KrKr2 项目中的应用：

```
违反 DIP（当前状况）：
  GameLogic ──依赖──► Cocos2dUI
  （高层）              （低层）
  如果换掉 Cocos2dUI，GameLogic 必须改

遵循 DIP（目标）：
  GameLogic ──依赖──► IUIBackend（抽象接口）
                          ▲
                          │
              ┌───────────┼───────────┐
              │           │           │
         Cocos2dUI   FlutterUI   ComposeUI
         （具体实现，依赖接口，不被 GameLogic 直接引用）
```

## 3. 完整 UI 抽象接口设计

### 3.1 接口层次结构

一个实用的 UI 抽象层需要覆盖多个层次，从底层的窗口管理到高层的业务组件：

```
层次 4：业务组件接口
┌──────────────────────────────────────────┐
│ ISaveDialog │ ISettingsPanel │ ILogView  │  ← 特定于 KrKr2 的业务 UI
└──────────────────────┬───────────────────┘
                       │ 组合
层次 3：通用 UI 组件接口
┌──────────────────────┴───────────────────┐
│ IMessageBox │ IMenu │ IFileDialog │ ...  │  ← 通用可复用组件
└──────────────────────┬───────────────────┘
                       │ 依赖
层次 2：UI 后端接口
┌──────────────────────┴───────────────────┐
│ IUIBackend (init/shutdown/createXxx)     │  ← 工厂 + 生命周期管理
└──────────────────────┬───────────────────┘
                       │ 依赖
层次 1：渲染集成接口
┌──────────────────────┴───────────────────┐
│ IRenderBridge (纹理共享/同步/尺寸)       │  ← 与游戏渲染引擎的交互
└──────────────────────────────────────────┘
```

### 3.2 核心接口定义

```cpp
// 示例 3：完整的 UI 抽象接口层定义
// 文件：include/ui/IUIBackend.h

#pragma once

#include <cstdint>
#include <functional>
#include <memory>
#include <string>
#include <vector>

namespace krkr2 {
namespace ui {

// ============================
// 前向声明
// ============================
class IMessageBox;
class IMenu;
class IFileDialog;
class IProgressBar;
class ITextInput;
class IGameView;
class IRenderBridge;

// ============================
// 基础类型
// ============================

// UI 后端类型枚举
enum class BackendType {
    Cocos2d,    // 当前实现
    Flutter,    // Flutter 嵌入
    Compose,    // Compose Multiplatform
    ImGui,      // Dear ImGui（调试用）
    Native      // 各平台原生 UI
};

// 矩形区域
struct Rect {
    float x, y, width, height;
};

// 颜色（RGBA，0.0-1.0）
struct Color {
    float r, g, b, a;
    static Color white() { return {1, 1, 1, 1}; }
    static Color black() { return {0, 0, 0, 1}; }
    static Color transparent() { return {0, 0, 0, 0}; }
};

// UI 事件类型
enum class UIEventType {
    ButtonClicked,   // 按钮被点击
    MenuSelected,    // 菜单项被选中
    TextChanged,     // 文本内容改变
    DialogClosed,    // 对话框关闭
    SliderChanged,   // 滑块值改变
    CheckboxToggled  // 复选框切换
};

// UI 事件数据
struct UIEvent {
    UIEventType type;
    int id;                // 组件 ID
    int intValue;          // 整数值（如选中项索引、按钮索引）
    float floatValue;      // 浮点值（如滑块值）
    std::string stringValue;  // 字符串值（如输入文本）
};

// UI 事件回调
using UIEventCallback = std::function<void(const UIEvent&)>;

// ============================
// 核心接口：IUIBackend
// ============================

/**
 * UI 后端抽象接口
 *
 * 这是 UI 抽象层的核心。游戏引擎通过此接口创建和管理所有 UI 组件，
 * 而不需要知道底层使用哪个 UI 框架。
 *
 * 生命周期：
 *   构造 → init() → [使用] → shutdown() → 析构
 */
class IUIBackend {
public:
    virtual ~IUIBackend() = default;

    // ---- 生命周期管理 ----

    /**
     * 初始化 UI 后端
     * @param windowHandle 平台窗口句柄（可选，某些后端自行创建窗口）
     * @return true 成功
     */
    virtual bool init(void* windowHandle = nullptr) = 0;

    /** 关闭 UI 后端，释放所有资源 */
    virtual void shutdown() = 0;

    /** 获取后端类型 */
    virtual BackendType getType() const = 0;

    /** 获取后端名称（用于日志） */
    virtual const char* getName() const = 0;

    // ---- 组件工厂方法 ----

    virtual std::unique_ptr<IMessageBox> createMessageBox() = 0;
    virtual std::unique_ptr<IMenu> createMenu() = 0;
    virtual std::unique_ptr<IFileDialog> createFileDialog() = 0;
    virtual std::unique_ptr<IProgressBar> createProgressBar() = 0;
    virtual std::unique_ptr<ITextInput> createTextInput() = 0;
    virtual std::unique_ptr<IGameView> createGameView() = 0;

    // ---- 渲染桥接 ----

    /** 获取渲染桥接器（用于与游戏引擎共享纹理） */
    virtual IRenderBridge* getRenderBridge() = 0;

    // ---- 帧更新 ----

    /**
     * 每帧调用，让 UI 后端处理内部事件和动画
     * @param deltaTime 距上一帧的时间（秒）
     */
    virtual void update(float deltaTime) = 0;

    // ---- 全局事件 ----

    /** 设置全局事件回调 */
    virtual void setEventCallback(UIEventCallback callback) = 0;

    // ---- 能力查询 ----

    /** 查询后端是否支持某项能力 */
    virtual bool hasCapability(const char* capabilityName) const = 0;
};

// ============================
// 组件接口：IMessageBox
// ============================

class IMessageBox {
public:
    virtual ~IMessageBox() = default;

    enum class Type {
        Info,      // 信息提示
        Warning,   // 警告
        Error,     // 错误
        Question   // 询问（需要用户选择）
    };

    virtual void setType(Type type) = 0;
    virtual void setTitle(const std::string& title) = 0;
    virtual void setMessage(const std::string& message) = 0;
    virtual void setButtons(const std::vector<std::string>& buttons) = 0;

    /**
     * 显示消息框（阻塞或异步取决于实现）
     * @param callback 用户选择回调，参数为按钮索引
     */
    virtual void show(std::function<void(int)> callback) = 0;
};

// ============================
// 组件接口：IMenu
// ============================

class IMenu {
public:
    virtual ~IMenu() = default;

    struct MenuItem {
        int id;
        std::string label;
        bool enabled = true;
        bool checked = false;
        std::vector<MenuItem> children;  // 子菜单
    };

    virtual void setItems(const std::vector<MenuItem>& items) = 0;

    /**
```

## 4. 生命周期管理

### 4.1 UI 组件的生命周期状态机

UI 组件（窗口、对话框、面板）都有明确的生命周期。设计一个统一的状态机可以避免内存泄漏和非法状态访问：

```
┌─────────┐    init()     ┌──────────┐
│ Created │──────────────►│ Ready    │
│ (构造)   │              │ (就绪)    │
└─────────┘              └────┬─────┘
                              │ show()
                              ▼
                         ┌──────────┐
                    ┌───►│ Visible  │◄───┐
                    │    │ (可见)    │    │
                    │    └────┬─────┘    │
                    │         │          │
              show()│    hide()│    show()│
                    │         ▼          │
                    │    ┌──────────┐    │
                    └────│ Hidden   │────┘
                         │ (隐藏)    │
                         └────┬─────┘
                              │ destroy()
                              ▼
                         ┌──────────┐
                         │Destroyed │
                         │ (已销毁)  │
                         └──────────┘
```

```cpp
// 示例 4：生命周期管理器
// 文件：include/ui/UILifecycle.h

#pragma once
#include <cstdio>
#include <stdexcept>
#include <string>

namespace krkr2 {
namespace ui {

// 组件状态枚举
enum class ComponentState {
    Created,    // 已构造，未初始化
    Ready,      // 已初始化，可以显示
    Visible,    // 正在显示
    Hidden,     // 已隐藏（仍在内存中）
    Destroyed   // 已销毁（不可再使用）
};

// 状态名称（用于日志输出）
inline const char* stateToString(ComponentState state) {
    switch (state) {
        case ComponentState::Created:   return "Created";
        case ComponentState::Ready:     return "Ready";
        case ComponentState::Visible:   return "Visible";
        case ComponentState::Hidden:    return "Hidden";
        case ComponentState::Destroyed: return "Destroyed";
        default: return "Unknown";
    }
}

/**
 * UI 组件基类 — 提供统一的生命周期管理
 *
 * 所有 UI 组件都继承此基类，获得状态管理和合法性检查。
 * 派生类实现 onXxx() 虚函数来响应生命周期事件。
 */
class UIComponent {
    ComponentState state_ = ComponentState::Created;
    std::string name_;    // 组件名称（调试用）
    int id_;              // 组件唯一 ID

    static int nextId_;   // 全局 ID 计数器

protected:
    // 状态检查辅助方法
    void requireState(ComponentState expected, const char* action) const {
        if (state_ != expected) {
            char msg[256];
            snprintf(msg, sizeof(msg),
                "组件 '%s' (id=%d): 无法执行 %s，当前状态 %s，期望 %s",
                name_.c_str(), id_, action,
                stateToString(state_), stateToString(expected));
            throw std::logic_error(msg);
        }
    }

    void requireNotDestroyed(const char* action) const {
        if (state_ == ComponentState::Destroyed) {
            char msg[256];
            snprintf(msg, sizeof(msg),
                "组件 '%s' (id=%d): 无法执行 %s，组件已销毁",
                name_.c_str(), id_, action);
            throw std::logic_error(msg);
        }
    }

    // ---- 派生类实现的生命周期回调 ----
    virtual bool onInit() { return true; }
    virtual void onShow() {}
    virtual void onHide() {}
    virtual void onDestroy() {}
    virtual void onUpdate(float dt) {}

public:
    explicit UIComponent(const std::string& name)
        : name_(name), id_(nextId_++) {}

    virtual ~UIComponent() {
        if (state_ != ComponentState::Destroyed && state_ != ComponentState::Created) {
            printf("警告：组件 '%s' 未正确销毁（当前状态: %s）\n",
                   name_.c_str(), stateToString(state_));
        }
    }

    // ---- 公开的生命周期方法 ----

    bool init() {
        requireState(ComponentState::Created, "init");
        if (onInit()) {
            state_ = ComponentState::Ready;
            printf("[Lifecycle] %s: Created → Ready\n", name_.c_str());
            return true;
        }
        return false;
    }

    void show() {
        requireNotDestroyed("show");
        if (state_ == ComponentState::Visible) return;  // 已经可见
        if (state_ == ComponentState::Created) {
            init();  // 自动初始化
        }
        onShow();
        state_ = ComponentState::Visible;
        printf("[Lifecycle] %s: %s → Visible\n", name_.c_str(),
               stateToString(state_));
    }

    void hide() {
        if (state_ != ComponentState::Visible) return;  // 已经隐藏
        onHide();
        state_ = ComponentState::Hidden;
        printf("[Lifecycle] %s: Visible → Hidden\n", name_.c_str());
    }

    void destroy() {
        if (state_ == ComponentState::Destroyed) return;
        if (state_ == ComponentState::Visible) {
            hide();  // 先隐藏
        }
        onDestroy();
        state_ = ComponentState::Destroyed;
        printf("[Lifecycle] %s: → Destroyed\n", name_.c_str());
    }

    void update(float dt) {
        if (state_ == ComponentState::Visible) {
            onUpdate(dt);
        }
    }

    // ---- 查询方法 ----
    ComponentState getState() const { return state_; }
    const std::string& getName() const { return name_; }
    int getId() const { return id_; }
    bool isVisible() const { return state_ == ComponentState::Visible; }
};

// 静态成员初始化（在 .cpp 文件中）
// int UIComponent::nextId_ = 1;

} // namespace ui
} // namespace krkr2
```

## 5. 事件分发机制

### 5.1 事件总线设计

UI 层需要将用户操作（点击、输入、菜单选择）传递给游戏引擎，同时游戏引擎也需要向 UI 发送通知（进度更新、状态变化）。**事件总线**（Event Bus）是一种解耦的事件分发方案：

```
┌─────────────┐     publish      ┌─────────────┐     deliver     ┌──────────────┐
│ UI 组件      │────────────────►│  EventBus   │───────────────►│ 游戏引擎      │
│ (生产者)     │                 │  (中转站)    │                │ (消费者)      │
└─────────────┘                 └──────┬──────┘                └──────────────┘
                                       │
                                       │ deliver
                                       ▼
                                ┌──────────────┐
                                │ 其他订阅者    │
                                │ (日志/统计)   │
                                └──────────────┘
```

```cpp
// 示例 5：类型安全的事件总线
// 文件：include/ui/EventBus.h

#pragma once

#include <cstdio>
#include <functional>
#include <mutex>
#include <string>
#include <typeindex>
#include <unordered_map>
#include <vector>

namespace krkr2 {
namespace ui {

// ---- 事件基类 ----
struct Event {
    virtual ~Event() = default;
    virtual const char* getName() const = 0;
};

// ---- 具体事件类型 ----

// UI 按钮点击事件
struct ButtonClickEvent : Event {
    int buttonId;
    std::string buttonLabel;
    const char* getName() const override { return "ButtonClick"; }
};

// 菜单选择事件
struct MenuSelectEvent : Event {
    int menuItemId;
    std::string menuItemLabel;
    const char* getName() const override { return "MenuSelect"; }
};

// 文本输入事件
struct TextInputEvent : Event {
    std::string text;
    bool isComplete;  // true = 用户按了回车
    const char* getName() const override { return "TextInput"; }
};

// 游戏引擎状态变化事件（引擎→UI）
struct EngineStateEvent : Event {
    enum class State { Loading, Running, Paused, Stopped };
    State state;
    float progress;  // 加载进度（0.0-1.0）
    const char* getName() const override { return "EngineState"; }
};

// 触摸/鼠标事件（UI→引擎）
struct TouchEvent : Event {
    enum class Action { Down, Up, Move, Cancel };
    Action action;
    float x, y;       // 坐标
    int pointerId;     // 触摸点 ID
    const char* getName() const override { return "Touch"; }
};

// ---- 事件总线实现 ----

class EventBus {
    // 订阅记录：事件类型 → 回调函数列表
    using HandlerList = std::vector<std::function<void(const Event&)>>;
    std::unordered_map<std::type_index, HandlerList> handlers_;
    std::mutex mutex_;

    // 订阅 ID 管理
    int nextSubId_ = 1;
    struct Subscription {
        std::type_index eventType;
        size_t handlerIndex;
    };
    std::unordered_map<int, Subscription> subscriptions_;

public:
    /**
     * 订阅事件
     * @tparam T 事件类型（必须继承 Event）
     * @param handler 事件处理函数
     * @return 订阅 ID，用于取消订阅
     */
    template<typename T>
    int subscribe(std::function<void(const T&)> handler) {
        static_assert(std::is_base_of_v<Event, T>,
                      "T 必须继承 Event 基类");

        std::lock_guard<std::mutex> lock(mutex_);

        auto typeIdx = std::type_index(typeid(T));
        auto& list = handlers_[typeIdx];

        // 包装为基类回调
        list.push_back([handler](const Event& e) {
            handler(static_cast<const T&>(e));
        });

        int subId = nextSubId_++;
        subscriptions_[subId] = {typeIdx, list.size() - 1};

        return subId;
    }

    /**
     * 发布事件 — 同步分发给所有订阅者
     * @param event 事件对象
     */
    template<typename T>
    void publish(const T& event) {
        static_assert(std::is_base_of_v<Event, T>,
                      "T 必须继承 Event 基类");

        std::lock_guard<std::mutex> lock(mutex_);

        auto it = handlers_.find(std::type_index(typeid(T)));
        if (it != handlers_.end()) {
            for (auto& handler : it->second) {
                if (handler) {
                    handler(event);
                }
            }
        }
    }

    /**
     * 取消订阅
     * @param subscriptionId subscribe() 返回的 ID
     */
    void unsubscribe(int subscriptionId) {
        std::lock_guard<std::mutex> lock(mutex_);
        auto it = subscriptions_.find(subscriptionId);
        if (it != subscriptions_.end()) {
            auto& list = handlers_[it->second.eventType];
            if (it->second.handlerIndex < list.size()) {
                list[it->second.handlerIndex] = nullptr;  // 置空而非删除（保持索引稳定）
            }
            subscriptions_.erase(it);
        }
    }
};

    });

    TouchEvent touch;
    touch.action = TouchEvent::Action::Down;
    touch.x = 100.0f;
    touch.y = 200.0f;
```

### 5.2 线程安全考虑

在实际应用中，UI 事件和游戏引擎通常运行在不同线程。上面的 `EventBus` 使用 `std::mutex` 提供基本的线程安全，但同步分发可能导致死锁（如果回调函数中再次发布事件）。更稳健的方案是使用**事件队列**：

```cpp
// 线程安全事件队列（异步分发）
// 生产者线程将事件入队，消费者线程在自己的循环中处理

#include <queue>
#include <mutex>
#include <condition_variable>
#include <memory>

class EventQueue {
    std::queue<std::unique_ptr<Event>> queue_;
    std::mutex mutex_;
    std::condition_variable cv_;
    bool stopped_ = false;

public:
    // 生产者调用：将事件入队
    void enqueue(std::unique_ptr<Event> event) {
        {
            std::lock_guard<std::mutex> lock(mutex_);
            queue_.push(std::move(event));
        }
        cv_.notify_one();  // 通知消费者
    }

    // 消费者调用：取出一个事件（阻塞等待）
    std::unique_ptr<Event> dequeue() {
        std::unique_lock<std::mutex> lock(mutex_);
        cv_.wait(lock, [this] { return !queue_.empty() || stopped_; });
        if (stopped_ && queue_.empty()) return nullptr;
        auto event = std::move(queue_.front());
        queue_.pop();
        return event;
    }

    // 消费者调用：取出所有事件（非阻塞）
    std::vector<std::unique_ptr<Event>> drainAll() {
        std::lock_guard<std::mutex> lock(mutex_);
        std::vector<std::unique_ptr<Event>> events;
        while (!queue_.empty()) {
            events.push_back(std::move(queue_.front()));
            queue_.pop();
        }
        return events;
    }

    void stop() {
        {
            std::lock_guard<std::mutex> lock(mutex_);
            stopped_ = true;
        }
        cv_.notify_all();
    }
};
```

**使用模式**：

- **UI 线程**：用户点击按钮 → `eventQueue.enqueue(make_unique<ButtonClickEvent>(...))` → 立即返回
- **引擎主循环**：每帧调用 `auto events = eventQueue.drainAll()` → 处理所有积累的事件 → 继续渲染

## 6. 接口版本控制

### 6.1 为什么需要版本控制？

当 UI 接口发布后，可能有多个实现（Cocos2d、Flutter、Compose）同时存在。如果修改接口（添加新方法、修改参数），所有实现都必须同步更新——这在实际项目中不现实。**接口版本控制**允许新旧实现共存：

```cpp
// 版本控制方案：能力查询 + 可选接口
// 文件：include/ui/IUIBackendVersioned.h

#pragma once
#include <cstdint>

namespace krkr2 {
namespace ui {

// 接口版本号
constexpr uint32_t UI_API_VERSION_1_0 = 0x0100;  // v1.0 — 初始版本
constexpr uint32_t UI_API_VERSION_1_1 = 0x0101;  // v1.1 — 添加动画支持
constexpr uint32_t UI_API_VERSION_2_0 = 0x0200;  // v2.0 — 添加无障碍访问

// 当前 API 版本
constexpr uint32_t UI_API_CURRENT_VERSION = UI_API_VERSION_1_1;

class IUIBackendVersioned {
public:
    virtual ~IUIBackendVersioned() = default;

    /** 获取此实现支持的 API 版本 */
    virtual uint32_t getApiVersion() const = 0;

    /**
     * 查询是否支持某项能力
     *
     * 能力字符串列表：
     * - "animation"       动画过渡效果
     * - "accessibility"   无障碍访问（屏幕阅读器等）
     * - "i18n"            国际化 / 多语言
     * - "gpu_texture"     GPU 纹理共享
     * - "pixel_copy"      像素拷贝模式
     * - "native_dialog"   使用系统原生对话框
     */
    virtual bool hasCapability(const char* capability) const = 0;

    /**
     * 获取扩展接口（用于可选功能）
     *
     * @param interfaceId 接口标识符（如 "IAnimationSupport"）
     * @return 接口指针（需要转型），nullptr 表示不支持
     *
     * 示例：
     *   auto* anim = static_cast<IAnimationSupport*>(
     *       backend->getExtension("IAnimationSupport"));
     *   if (anim) { anim->fadeIn(component, 0.3f); }
     */
    virtual void* getExtension(const char* interfaceId) = 0;
};

// 可选扩展接口示例：动画支持
class IAnimationSupport {
public:
    virtual ~IAnimationSupport() = default;
    virtual void fadeIn(int componentId, float duration) = 0;
    virtual void fadeOut(int componentId, float duration) = 0;
    virtual void slideTo(int componentId, float x, float y, float duration) = 0;
};

} // namespace ui
} // namespace krkr2
```

### 6.2 版本协商流程

```
应用启动时的版本协商：

1. 应用加载 UI 后端插件
2. 调用 backend->getApiVersion()
   ├─ 返回 >= UI_API_CURRENT_VERSION → 完整功能
   ├─ 返回 >= UI_API_VERSION_1_0     → 基础功能
   └─ 返回 < UI_API_VERSION_1_0      → 拒绝加载，版本过旧
3. 对于可选功能，使用 hasCapability() 动态检测
   ├─ hasCapability("animation") → true  → 使用动画过渡
   └─ hasCapability("animation") → false → 使用简单的显示/隐藏
```

## 7. 完整示例：UI 管理器

将以上所有概念整合为一个完整的 UI 管理器：

```cpp
// 示例 6：完整的 UIManager
// 文件：src/ui/UIManager.cpp

#include <cstdio>
#include <memory>
#include <string>
#include <unordered_map>
#include <vector>

// （此处省略前面定义的接口头文件 include）

namespace krkr2 {
namespace ui {

/**
 * UI 管理器 — 管理 UI 后端和所有 UI 组件的中央控制器
 *
 * 职责：
 * 1. 管理 UI 后端的生命周期
 * 2. 维护所有活跃 UI 组件的注册表
 * 3. 分发帧更新给所有可见组件
 * 4. 管理事件总线
 */
class UIManager {
    std::unique_ptr<IUIBackend> backend_;
    EventBus eventBus_;
    std::unordered_map<int, std::unique_ptr<UIComponent>> components_;

    // 后端注册表（支持运行时切换）
    using BackendFactory = std::function<std::unique_ptr<IUIBackend>()>;
    std::unordered_map<BackendType, BackendFactory> backendFactories_;

public:
    // 注册后端工厂
    void registerBackend(BackendType type, BackendFactory factory) {
        backendFactories_[type] = std::move(factory);
        printf("[UIManager] 注册后端: %d\n", (int)type);
    }

    // 初始化指定后端
    bool initBackend(BackendType type, void* windowHandle = nullptr) {
        // 如果已有后端，先关闭
        if (backend_) {
            shutdownBackend();
        }

        auto it = backendFactories_.find(type);
        if (it == backendFactories_.end()) {
            printf("[UIManager] 错误：未注册后端 %d\n", (int)type);
            return false;
        }

        backend_ = it->second();
        if (!backend_->init(windowHandle)) {
            printf("[UIManager] 错误：后端初始化失败\n");
            backend_.reset();
            return false;
        }

        printf("[UIManager] 后端 '%s' 初始化成功\n", backend_->getName());
        return true;
    }

    // 关闭当前后端
    void shutdownBackend() {
        if (!backend_) return;

        // 销毁所有组件
        for (auto& [id, comp] : components_) {
            comp->destroy();
        }
        components_.clear();

        backend_->shutdown();
        printf("[UIManager] 后端 '%s' 已关闭\n", backend_->getName());
        backend_.reset();
    }

    // 每帧更新
    void update(float deltaTime) {
        if (!backend_) return;

        // 更新后端
        backend_->update(deltaTime);

        // 更新所有活跃组件
        for (auto& [id, comp] : components_) {
            comp->update(deltaTime);
        }
    }

    // 获取事件总线
    EventBus& getEventBus() { return eventBus_; }

    // 获取当前后端
    IUIBackend* getBackend() { return backend_.get(); }

    // 显示消息框（便捷方法）
    void showMessage(
        const std::string& title,
        const std::string& message,
        std::function<void(int)> callback = nullptr
    ) {
        if (!backend_) return;

        auto msgBox = backend_->createMessageBox();
        msgBox->setTitle(title);
        msgBox->setMessage(message);
        msgBox->setButtons({"确定", "取消"});
        msgBox->show(callback ? callback : [](int) {});
    }
};

} // namespace ui
} // namespace krkr2
```

## 本节小结

1. **UI 与渲染引擎分离**的核心动机是让 UI 框架可以独立于游戏渲染引擎升级和替换
2. **策略模式**将可变的 UI 实现封装到独立类中，通过统一接口调用，实现运行时切换
3. **抽象工厂**模式创建一整套相关 UI 组件，确保组件之间的一致性
4. **依赖倒置原则**（DIP）要求高层模块（游戏逻辑）依赖抽象接口而非具体实现
5. UI 组件需要严格的**生命周期管理**：Created → Ready → Visible → Hidden → Destroyed
6. **事件总线**解耦了事件的生产者（UI 组件）和消费者（游戏引擎），支持灵活的订阅/发布
7. **渲染桥接器**（IRenderBridge）是 UI 层与游戏渲染引擎共享纹理的标准接口
8. **接口版本控制**通过版本号 + 能力查询 + 扩展接口，允许新旧实现共存
9. 完整的 **UIManager** 整合了后端管理、组件注册、帧更新和事件分发

## 练习题与答案

### 题目 1：策略模式 vs 桥接模式

问题：在 UI 抽象层设计中，策略模式和桥接模式（Bridge Pattern）都可以用于实现"接口与实现分离"。请分析两者的区别，并说明 KrKr2 的 UI 抽象层应该使用哪种模式（或两者结合）。

<details>
<summary>查看答案</summary>

**策略模式**：
- 关注点：**行为的可替换性**——一个对象的某个行为可以在运行时切换
- 粒度：单个行为/算法（如"显示消息框的方式"）
- 关系：组合关系（Context 持有 Strategy 的引用）
- 切换时机：运行时随时可以切换

**桥接模式**：
- 关注点：**抽象与实现的分离**——将"抽象维度"和"实现维度"解耦
- 粒度：整个类层次结构（如"UI 组件层次"和"渲染后端层次"）
- 关系：聚合关系（Abstraction 持有 Implementor 的引用）
- 切换时机：通常在构造时确定

**KrKr2 的选择**：两者结合。

1. **桥接模式用于 IUIBackend**：`IUIBackend` 是"抽象维度"（定义 UI 能做什么），`Cocos2dUIBackend` / `FlutterUIBackend` 是"实现维度"（具体怎么做）。这是典型的桥接模式——抽象的 UI 组件层次和具体的渲染后端层次可以独立扩展。

2. **策略模式用于具体行为**：在同一个后端内部，某些行为可能有多种实现。例如 `FlutterUIBackend` 可能对"文件选择器"使用系统原生实现或 Flutter 自绘实现——这个选择用策略模式处理。

```cpp
// 桥接模式：UI 抽象 + 后端实现
class IUIBackend { ... };           // 抽象
class FlutterUIBackend : IUIBackend { ... };  // 实现

// 策略模式：后端内部的行为切换
class FlutterUIBackend : IUIBackend {
    std::unique_ptr<IFileDialogStrategy> fileDialogStrategy_;
    // 可以是 FlutterFileDialog 或 NativeFileDialog
};
```

</details>

### 题目 2：事件总线死锁分析

问题：以下代码在什么情况下会死锁？请说明原因并给出解决方案。

```cpp
EventBus bus;

bus.subscribe<ButtonClickEvent>([&bus](const ButtonClickEvent& e) {
    // 按钮点击后，发布一个菜单选择事件
    MenuSelectEvent menuEvent;
    menuEvent.menuItemId = e.buttonId;
    bus.publish(menuEvent);  // 在回调中再次发布
});
```

<details>
<summary>查看答案</summary>

**死锁原因**：`EventBus::publish()` 内部持有 `mutex_` 锁。当 `ButtonClickEvent` 的回调被调用时，`publish()` 仍持有锁。回调中再次调用 `bus.publish(menuEvent)`，需要再次获取同一个 `mutex_`。如果 `mutex_` 是 `std::mutex`（不可重入），第二次加锁会导致死锁。

**解决方案**（三选一）：

**方案 1**：使用 `std::recursive_mutex`（最简单但性能较差）
```cpp
class EventBus {
    std::recursive_mutex mutex_;  // 允许同一线程多次加锁
    // ...
};
```

**方案 2**：在回调中使用异步队列（推荐）
```cpp
bus.subscribe<ButtonClickEvent>([&eventQueue](const ButtonClickEvent& e) {
    auto menuEvent = std::make_unique<MenuSelectEvent>();
    menuEvent->menuItemId = e.buttonId;
    eventQueue.enqueue(std::move(menuEvent));  // 入队，不直接 publish
});
```

**方案 3**：在 `publish()` 中拷贝 handler 列表后释放锁再调用
```cpp
template<typename T>
void publish(const T& event) {
    HandlerList handlersCopy;
    {
        std::lock_guard<std::mutex> lock(mutex_);
        auto it = handlers_.find(std::type_index(typeid(T)));
        if (it != handlers_.end()) {
            handlersCopy = it->second;  // 拷贝
        }
    }
    // 锁已释放，安全调用回调
    for (auto& handler : handlersCopy) {
        if (handler) handler(event);
    }
}
```

**推荐方案 2**，因为它同时解决了死锁和线程安全问题。

</details>

```cpp
class CapabilityQuery {
    std::unordered_set<std::string> capabilities_;

public:
    void addCapability(const char* cap) {
        capabilities_.insert(cap);
    }

    // 精确查询
    bool has(const char* cap) const {
        return capabilities_.count(cap) > 0;
    }

    // 层级查询：has("basic") 返回 true 如果任何 "basic.*" 存在
    bool hasAny(const char* prefix) const {
        std::string prefixStr(prefix);
        for (const auto& cap : capabilities_) {
            if (cap.find(prefixStr) == 0) return true;
        }
        return false;
    }

    // 列出某类别下所有能力
    std::vector<std::string> list(const char* prefix) const {
        std::vector<std::string> result;
        std::string prefixStr(prefix);
        for (const auto& cap : capabilities_) {
            if (cap.find(prefixStr) == 0) {
                result.push_back(cap);
            }
        }
        return result;
    }
};
```
