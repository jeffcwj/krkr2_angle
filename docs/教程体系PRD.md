## 一、项目背景与架构总结

### 1.1 项目简介

KrKr2 Emulator 是一个跨平台 C++17 模拟器，用于运行基于 KiriKiri/吉里吉里（T Visual Presenter）引擎的视觉小说游戏。项目目标是在 Android、Windows、Linux、macOS 上完整还原原版 KiriKiri2 引擎行为。

### 1.2 系统架构全景

```
┌─────────────────────────────────────────────────────────┐
│                    平台入口层 (platforms/)                │
│  Windows(WinMain) │ Linux(main) │ macOS(main) │ Android(JNI) │
└──────────────────────────┬──────────────────────────────┘
                           │ 构造 TVPAppDelegate → run()
┌──────────────────────────▼──────────────────────────────┐
│                 引擎核心层 (cpp/core/)                    │
│  ┌─────────┐ ┌─────────┐ ┌──────────┐ ┌─────────────┐  │
│  │  tjs2   │ │  base   │ │ environ  │ │   visual    │  │
│  │ 脚本引擎 │ │ I/O归档 │ │ 平台抽象  │ │ 渲染/图像   │  │
│  └─────────┘ └─────────┘ └──────────┘ └─────────────┘  │
│  ┌─────────┐ ┌─────────┐ ┌──────────┐ ┌─────────────┐  │
│  │  sound  │ │  movie  │ │  plugin  │ │   utils     │  │
│  │ 音频系统 │ │ 视频播放 │ │ 插件加载  │ │ 工具/线程   │  │
│  └─────────┘ └─────────┘ └──────────┘ └─────────────┘  │
│  ┌─────────────────────────────────────────────────┐    │
│  │              extension (扩展API)                 │    │
│  └─────────────────────────────────────────────────┘    │
└──────────────────────────┬──────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────┐
│                  插件实现层 (cpp/plugins/)                │
│  psdfile │ psbfile │ motionplayer │ layerex_draw │ ...  │
└──────────────────────────┬──────────────────────────────┘
                           │
```

## 二、目标读者与学习目标

### 2.1 目标读者画像

- 具备 C++ 基础（语法、STL、面向对象）
- **没有**跨平台多模块项目经验
- **没有**CMake/vcpkg/Ninja 等现代构建工具链经验
- **没有**图形渲染（OpenGL）、音视频编解码（FFmpeg）、脚本引擎等领域经验
- **没有**逆向工程经验
- 有意愿从零学习上述所有技术栈，最终能独立参与本项目开发

### 2.2 终极学习目标（读完全部教程后能做到）

1. **理解项目全貌** — 能画出项目架构图，说清楚每个模块的职责和交互
2. **独立构建项目** — 在任意目标平台上从零配置环境并成功编译运行
3. **修复 Bug** — 定位、分析、修复项目中的问题
4. **添加功能** — 新增图像格式支持、新音频解码器、新插件等
5. **理解渲染系统** — 掌握 OpenGL/软件渲染原理，能修改渲染管线
6. **理解音视频** — 掌握 FFmpeg 解封装/解码流程、A/V 同步、OpenAL 输出
7. **理解脚本引擎** — 掌握 TJS2 词法/语法/字节码/VM 全链路
8. **替换 UI 框架** — 能用 Flutter 或 Compose Multiplatform 替换 Cocos2d-x UI 层
9. **逆向插件** — 掌握二进制逆向基础，能还原无源码的 KiriKiri 插件
10. **独立开发模块** — 能自主设计并实现新的引擎子系统

## 七、质量标准

### 7.1 文档审查清单

每篇文档发布前必须通过以下检查：

| 检查项 | 标准 |
|--------|------|
| **准确性** | 所有技术描述经过验证，代码示例可编译运行 |
| **完整性** | 覆盖大纲中列出的所有内容要点 |
| **可读性** | 面向 C++ 基础读者，无未解释的术语跳跃 |
| **实践性** | 每章至少包含一个可动手的代码示例或练习 |
| **源码对照** | 项目模块（M系列）必须包含"对照项目源码"环节，标注具体文件和行号 |
| **交叉引用** | 引用其他章节时使用正确的相对路径链接 |
| **格式规范** | 符合第五章定义的模板结构 |
| **长度控制** | 单文件 200-500 行，超出则拆分 |

### 7.2 代码示例要求

- **可编译：** 所有 C++ 示例必须能用 CMake + 指定工具链编译通过
- **自包含：** 示例尽量自包含，避免隐含的外部依赖
- **渐进式：** 同一章内的示例逐步增加复杂度
- **有注释：** 关键行必须有中文注释
- **有输出：** 示例应有可观察的输出（控制台打印/渲染结果/文件生成）
- **对照项目：** 项目模块的示例应尽量使用项目实际代码片段，标注来源

### 7.3 持续维护

- **代码同步：** 项目代码变更后，相关教程中的行号引用和代码片段需同步更新
- **版本标注：** PRD 和各模块 README 标注对应的项目 commit/tag
- **反馈收集：** 教程使用者的问题和建议记录在 GitHub Issues 中，定期回顾改进
- **覆盖率追踪：** 维护一个教程覆盖率矩阵，追踪哪些源文件已被教程覆盖
